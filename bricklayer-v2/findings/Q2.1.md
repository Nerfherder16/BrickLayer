# Finding: Q2.1 — Frontier mode program validation

**Question**: Does the Frontier mode program (`modes/frontier.md`) produce a complete, actionable loop? What questions would Wave 1 look like for the "Uncreated App" project?
**Agent**: regulatory-researcher
**Verdict**: HEALTHY
**Severity**: Info
**Mode**: validate
**Target**: `bricklayer-v2/modes/frontier.md`

## Summary

The Frontier mode program is complete and actionable. The loop is well-defined: 4-6 questions per wave, speculative framing, PROMISING/WEAK/BLOCKED verdict vocabulary, and a clear stop condition (human signal). The `ideas.md` dual-output pattern (findings + living registry) is the correct design. One gap: the hypothesis generator prompt is advisory ("should be prompted") rather than a required preamble. Wave 1 example questions for the Uncreated App demonstrate the loop produces grounded, structured output.

### Gap: hypothesis generator prompt is advisory

The current text says "Hypothesis generator should be prompted: ..." — this is guidance, not enforcement. When the campaign loop's hypothesis generator fires, it won't automatically use this framing unless the mode context is injected (via Q1.1's mode dispatch mechanism). Once Q1.1 is implemented, the `modes/frontier.md` content becomes the injected system prompt, making this a non-issue.

### Gap: no evidence sourcing for non-web contexts

The program says "Evidence is gathered via: Web search (Exa, Firecrawl) for analogues, prior art, market signals." In air-gapped or no-internet contexts, this fails gracefully if agent reasoning is allowed as fallback — but the program should explicitly state: "If web search unavailable, agent reasoning from known facts is sufficient for PROMISING/WEAK verdicts; only BLOCKED claims about real-world prerequisites require external verification."

### Wave 1 example: Uncreated App

Using Frontier mode logic, Wave 1 would look like:

| ID | Question | Expected Verdict |
|----|----------|-----------------|
| F1.1 | Given Tim's Recall memory infrastructure and Claude Code access, what is the most valuable new product that uses real-time memory as a core feature? | PROMISING |
| F1.2 | What would a developer-facing memory API look like that makes Recall commercially licensable? | PROMISING |
| F1.3 | What analogue exists for personal AI memory systems in other domains (medical, legal, enterprise) that Tim could adapt? | PROMISING or WEAK |
| F1.4 | What prerequisite gap in Tim's current stack prevents him from building a voice-first product today? | BLOCKED |
| F1.5 | If Tim's Recall system were productized for teams (not individuals), what is the core architecture change? | PROMISING |

These are well-formed Frontier questions: each asks "what could" not "what is", each has a discoverable answer, and each naturally produces a PROMISING/WEAK/BLOCKED verdict with reasoning.

### SUBJECTIVE verdict gap

Frontier mode doesn't include SUBJECTIVE. This is correct — ideation doesn't require human aesthetic input. However, the cross-mode handoff table (`program.md`) shows `Frontier PROMISING → Research`. If a PROMISING idea is highly speculative and requires human approval before Research-mode resources are spent, a `PROMISING_SPECULATIVE` pseudo-verdict could signal the handoff requires human review. This is a minor enhancement, not a gap.

## Verdict Rationale

HEALTHY: The Frontier mode loop is complete, evidence-gathering approach is sound, verdict vocabulary is appropriate, and the session-end Top 3 pattern creates a concrete handoff. Wave 1 example demonstrates the loop produces real output.
