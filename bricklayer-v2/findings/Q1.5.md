# Finding: Q1.5 — PENDING_EXTERNAL + resume_after: mechanism

**Question**: How should PENDING_EXTERNAL + `resume_after:` work in the campaign loop? Does it require changes to `bl/campaign.py` or only to `bl/questions.py`?
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: Medium
**Mode**: diagnose
**Target**: `bl/questions.py`, `bl/findings.py`, `bl/campaign.py`

## Summary

PENDING_EXTERNAL requires changes in three places: (1) a new `resume_after:` field parsed by `questions.py`; (2) `findings.py` writing `"PENDING_EXTERNAL"` as question status instead of `"INCONCLUSIVE"` when verdict is `PENDING_EXTERNAL`; (3) `campaign.py` checking at startup whether any PENDING_EXTERNAL questions have elapsed their `resume_after` date and re-activating them. The mechanism is lighter than DIAGNOSIS_COMPLETE reactivation — no git diff required, just a date comparison.

## Evidence

### Why PENDING_EXTERNAL exists

From bricklayer-meta Q4.3: 15.4% of Recall questions were INCONCLUSIVE due to external timing dependencies (cron windows, GC eligibility dates, deployment windows). These questions were re-checked repeatedly, accumulating INCONCLUSIVE chains 4+ waves long before escalating to FAILURE. PENDING_EXTERNAL stops the re-check cycle: once a question is blocked on a known external event, park it with a `resume_after:` date.

Examples from Recall:
- Q29.1: GC eligibility 2026-03-21T19:12 UTC — 6 days 8 hours away
- Q29.3: Week 12 dates (Mar 17-21) not yet reached
- Q16.3a/Q16.4a: Deferred until Tier 1 fixes deployed (no date — requires human trigger)

### New field: `resume_after:`

Add to question format:
```markdown
## Q34.8 [agent] Does hygiene cron fire correctly after restart?
**Operational Mode**: diagnose
**Status**: PENDING
**Hypothesis**: ...
**resume_after**: 2026-03-21T19:12:00Z
```

Parsed by `questions.py`:
```python
# In field_pattern, add "resume_after" to the alternation
# In dict construction:
"resume_after": fields.get("resume_after", "").strip(),
```

### Mechanism in `findings.py`

When verdict is `PENDING_EXTERNAL`, `_mark_question_done()` writes `"PENDING_EXTERNAL"` status to questions.md:
```python
new_status = (
    "PENDING_EXTERNAL" if verdict == "PENDING_EXTERNAL"
    else "DIAGNOSIS_COMPLETE" if verdict == "DIAGNOSIS_COMPLETE"
    else "DONE" if verdict in _TERMINAL_VERDICTS
    else "INCONCLUSIVE"
)
```

The question can also receive a `resume_after:` field from the finding result dict. If the result contains `"resume_after": "2026-03-21T19:12:00Z"`, write it to the questions.md block. This lets an agent running the question set the park date based on what it observes.

### Mechanism in `campaign.py`

At the start of `run_campaign()`, scan for PENDING_EXTERNAL questions with elapsed dates:

```python
def _reactivate_pending_external(questions: list[dict]) -> list[str]:
    """Re-activate PENDING_EXTERNAL questions whose resume_after has elapsed."""
    from datetime import datetime, timezone
    now = datetime.now(timezone.utc)
    reactivated = []

    for q in questions:
        if q["status"] != "PENDING_EXTERNAL":
            continue
        resume_after = q.get("resume_after", "")
        if not resume_after:
            continue  # no date — requires manual re-activation
        try:
            target = datetime.fromisoformat(resume_after.replace("Z", "+00:00"))
            if now >= target:
                _reactivate_question(q["id"])  # set status PENDING in questions.md
                reactivated.append(q["id"])
                print(f"[campaign] Re-activated {q['id']} (resume_after elapsed: {resume_after})", file=sys.stderr)
        except ValueError:
            pass  # malformed date — skip
    return reactivated
```

Call at `run_campaign()` start:
```python
all_questions = parse_questions()
reactivated = _reactivate_pending_external(all_questions)
if reactivated:
    all_questions = parse_questions()  # re-parse after mutations
```

### What requires campaign.py vs only questions.py

`questions.py` alone cannot handle re-activation because:
1. It has no awareness of "now" — it's a pure parser
2. It doesn't modify questions.md on its own (that's the responsibility of `_mark_question_done()` in findings.py)

`campaign.py` is the right place for re-activation because it's the entry point with temporal awareness.

### Manual-trigger PENDING_EXTERNAL

Some PENDING_EXTERNAL questions have no `resume_after` date — they're blocked on a human action (deployment, approval). These should NOT be auto-reactivated. The questions.md shows them as `**Status**: PENDING_EXTERNAL` and a human re-sets them to `PENDING` manually (or via the dashboard) when the blocker is cleared.

### Relationship to INCONCLUSIVE

INCONCLUSIVE means "we ran the question but couldn't get a result." PENDING_EXTERNAL means "we know why we can't run it yet and when to try again." They're different:

| Status | Re-run? | When? | Hypothesis generator? |
|--------|---------|-------|----------------------|
| INCONCLUSIVE | Yes | Next wave | May generate follow-ups |
| PENDING_EXTERNAL | Yes | After resume_after | No follow-ups generated |
| DIAGNOSIS_COMPLETE | Only on code change | When target file changes | Never generates re-checks |
