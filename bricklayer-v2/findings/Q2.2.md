# Finding: Q2.2 — Fix mode scope-creep prevention

**Question**: Does the Fix mode program (`modes/fix.md`) correctly prevent scope creep? Is the pre-flight checklist sufficient to catch an underspecified DIAGNOSIS_COMPLETE finding?
**Agent**: regulatory-researcher
**Verdict**: WARNING
**Severity**: Medium
**Mode**: validate
**Target**: `bricklayer-v2/modes/fix.md`

## Summary

Fix mode's anti-patterns section correctly names scope creep as forbidden. However, the pre-flight checklist has a structural gap: it validates that a Fix Specification section *exists* but does not validate that it is *specific enough to execute without interpretation*. An underspecified finding ("fix the decay logic") passes the pre-flight check and allows the agent to interpret scope during implementation — the exact failure mode the check is meant to prevent.

## Evidence

### Anti-patterns section — STRONG

The anti-patterns list is precisely correct and complete:
- "Do not expand scope beyond the specified fix" ✓
- "Do not refactor surrounding code" ✓
- "Do not add features or improvements" ✓
- "Do not attempt a second fix approach if the first fails" ✓ (critical — prevents diagnosis-in-fix)
- "Do not skip the pre-flight check" ✓

This list directly addresses every failure pattern observed in real-world agentic fix attempts.

### Pre-flight checklist — STRUCTURAL GAP

Current pre-flight (step 2):
```
2. Confirm the Fix Specification section is complete:
   - File path ✓
   - Line or change description ✓
   - Verification method ✓
```

**The gap**: "Line or change description ✓" is binary existence check, not specificity check. A Fix Specification that says:

```
**File**: decay.py
**Change**: Fix the user_id handling logic
**Verification**: Run decay tests
```

...passes the pre-flight check but is underspecified. An agent executing this will invent the implementation, which IS scope creep hidden inside the specified scope.

**What "specific enough" looks like**:
```
**File**: qdrant.py:1207
**Change**: Change `if uid is not None:` to `if uid is not None and uid > 0:`
**Verification**: python -m pytest tests/test_decay.py -k test_user_id_filter
```

### Pre-flight should include a specificity gate

Add to the pre-flight:
```
2b. Verify Fix Specification specificity:
    - File path includes line number or function name ✓
    - Change description is a concrete edit (not "fix X" or "improve Y") ✓
    - Verification is a runnable command with expected output ✓
    - If any of these are vague — STOP. Write a note to the Diagnose session:
      "Fix Specification for [ID] requires refinement before Fix mode can proceed."
      Do NOT interpret vague specs. Return to Diagnose.
```

### Verdict assignment gap

Current verdict assignment:
```
FIX_FAILED — change implemented but verification fails OR tests regress
```

Missing case: **"change implemented as specified, but spec was wrong/insufficient"**. This happens when the Fix Specification was correctly followed but identified the wrong root cause. Current flow sends this back to Diagnose — which is correct — but the finding should explicitly encode: "DIAGNOSIS_COMPLETE finding for [original_id] was incorrect; new root cause is [X]." The current FIX_FAILED format doesn't capture this clearly enough.

Suggested addition to FIX_FAILED finding format:
```
## Root Cause Update
[If FIX_FAILED because spec was wrong, not because implementation failed,
state the corrected hypothesis here. This becomes the input to the next Diagnose session.]
```

### Session end artifact naming

`findings/{original_finding_id}_fix.md` — this is correct. However, the instruction also says "Update the original finding's status from DIAGNOSIS_COMPLETE to FIXED or FIX_FAILED." This requires writing to an existing finding file — should specify that the update is an append (`## Fix Attempt` section) rather than a rewrite of the original finding.

## Verdict Rationale

WARNING: The anti-patterns list is strong and correct. The pre-flight exists but has a specificity gap that allows underspecified Fix Specifications to pass. This is a real risk: a poorly-written DIAGNOSIS_COMPLETE finding leads to an agent that interprets scope, which is exactly the failure Fix mode exists to prevent. Pre-flight must validate specificity, not just existence.
