# Finding: Q2.5 — Cross-mode handoff table completeness

**Question**: Does the cross-mode handoff table in `program.md` cover all meaningful transitions? Are there handoffs that are missing or incorrect?
**Verdict**: WARNING
**Severity**: Medium
**Mode**: validate
**Target**: `bricklayer-v2/program.md`

## Summary

The cross-mode handoff table covers 9 transitions and correctly models the primary lifecycle flows. Three gaps were identified: (1) Benchmark → Evolve is missing (Benchmark CALIBRATED is the natural trigger for Evolve mode); (2) Research HEALTHY → Validate is missing (confirmed assumptions should progress to design review); (3) Fix FIXED → Monitor is missing (after a fix is verified, the repaired metric should be added to Monitor targets). One incorrect entry: Validate FAILURE → Diagnose is too narrow — a Validate FAILURE in a non-existent system should go to Frontier, not Diagnose.

## Evidence

### Current handoff table

| Source | Trigger | Target |
|--------|---------|--------|
| Frontier PROMISING | Human approves | Research |
| Research FAILURE | Critical assumption refuted | Frontier |
| Validate FAILURE | Design flaw found | Diagnose |
| Diagnose DIAGNOSIS_COMPLETE | Root cause + fix spec | Fix |
| Fix FIXED | Verified | Benchmark |
| Audit NON_COMPLIANT | Gap found | Diagnose |
| Monitor ALERT | Threshold crossed | Diagnose |
| Predict IMMINENT | Cascade identified | Fix |
| Evolve IMPROVEMENT | Baseline updated | Benchmark |

### Gap 1: Benchmark → Evolve (MISSING)

The natural progression after Benchmark is Evolve (if you have baselines, you can improve against them). Currently:
- `Fix FIXED → Benchmark` is present
- But `Benchmark CALIBRATED → Evolve` is missing

Without this, there's no defined path from "we measured things" to "we improved things." Add:
```
| Benchmark CALIBRATED | Baselines established | Evolve — improve against baseline |
```

### Gap 2: Research HEALTHY → Validate (MISSING)

Research HEALTHY means a critical assumption holds. After validating key assumptions, the natural next step is to design against confirmed ground truth. Add:
```
| Research HEALTHY | Core assumptions confirmed | Validate — design review |
```

### Gap 3: Fix FIXED → Monitor (MISSING)

After a fix is verified, the metric that was failing should be added to `monitor-targets.md` to watch for regression. Currently the flow goes `Fix FIXED → Benchmark` but doesn't explicitly seed Monitor. Add:
```
| Fix FIXED | Repaired metric identified | Monitor — add metric to monitor-targets |
```

### Incorrect entry: Validate FAILURE → Diagnose

Current: "Validate FAILURE → Diagnose — find root cause in existing system"

This is correct when a system already exists (validation found a design flaw in working code). But Validate mode can also run on a system that doesn't exist yet (validating the design before building). In that case, Validate FAILURE → Diagnose has no target (there's no existing system to diagnose).

Correction:
```
| Validate FAILURE | Design flaw in existing system | Diagnose |
| Validate FAILURE | Design flaw in planned system | Frontier or Research — revisit assumptions |
```

### Correct entries — confirmed

- `Frontier PROMISING → Research` ✓ (ideas need stress-testing before resources committed)
- `Research FAILURE → Frontier` ✓ (refuted assumption → rethink concept, not just diagnose)
- `Diagnose DIAGNOSIS_COMPLETE → Fix` ✓ (core lifecycle backbone)
- `Audit NON_COMPLIANT → Diagnose` ✓ (compliance gaps have code-level root causes)
- `Monitor ALERT → Diagnose` ✓ (Monitor is Diagnose's early warning system)
- `Predict IMMINENT → Fix` ✓ (prediction should directly seed the fix queue)

### Missing: Evolve REGRESSION → Diagnose

When Evolve produces a REGRESSION (an attempted improvement regressed a different metric), the revert is immediate — but the root cause of why the optimization caused regression should be investigated by Diagnose. Add:
```
| Evolve REGRESSION | Optimization caused regression | Diagnose — find why optimization regressed |
```

## Verdict Rationale

WARNING: The existing handoffs are correct. Three gaps (Benchmark→Evolve, Research HEALTHY→Validate, Fix FIXED→Monitor) and one incorrect entry (Validate FAILURE assumes existing system) are meaningful omissions that would leave practitioners without a defined path forward in real use. The additions are small and well-defined.
