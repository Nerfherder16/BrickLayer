# E1.1: Should `modes/monitor.md` add a DEGRADED_TRENDING verdict?

**Status**: IMPROVEMENT
**Date**: 2026-03-24
**Mode**: evolve
**Agent**: evolve-optimizer

## Hypothesis

Improving monitor.md's verdict set to include `DEGRADED_TRENDING` should enable proactive handoff
to Predict mode before a threshold is crossed, reducing the ALERT→fire-drill pattern that occurs
when DEGRADED is only detected after the WARNING threshold is breached.

## Before State

- `modes/monitor.md` verdict set: `OK | ALERT | DEGRADED | UNKNOWN` (4 verdicts)
- `program.md` mode dispatch: `OK, DEGRADED, ALERT, UNKNOWN`
- Cross-mode table: only `Monitor DEGRADED (sustained)` → Predict (triggers AFTER threshold crossing)
- Gap: A metric at 90% of WARNING threshold with consistent upward trend fires no alert until
  it crosses. The signal is invisible until too late.

## Change Applied

1. **`modes/monitor.md`** — Added `DEGRADED_TRENDING` verdict with explicit criteria:
   - ≥3 consecutive runs showing monotonic movement toward threshold
   - Projected to cross within ≤5 more runs at current rate
   - Trend projection formula: `runs_to_cross = (threshold - current) / avg_delta`
   - Added "Trend projection" section with concrete calculation procedure
   - Updated session-end summary to include DEGRADED_TRENDING count

2. **`program.md`** — Updated mode dispatch table verdict list
   - Added `DEGRADED_TRENDING` row to cross-mode handoff table:
     `Monitor DEGRADED_TRENDING → Predict` (early, before threshold)
   - Existing `Monitor DEGRADED (sustained) → Predict` preserved (post-threshold path)

## Delta

- **Metric**: Monitor mode verdict coverage (% of degradation scenarios detectable)
- **Baseline**: 3 scenarios covered (OK, at-threshold, above-threshold)
- **After**: 4 scenarios covered (OK, trending-toward, at-threshold, above-threshold)
- **Improvement**: +33% scenario coverage, early warning path now exists
- **Regression check**: All existing 4 verdicts preserved with identical semantics. program.md
  cross-mode table: 0 rows removed, 1 row added. No existing handoff paths changed.
- **Method**: Read before/after state of modes/monitor.md + program.md

## What Would Change This Verdict

Would become HEALTHY (no improvement needed) if: the existing `Monitor DEGRADED (sustained)`
cross-mode trigger was already meant to capture pre-threshold trends — but the current definition
of DEGRADED explicitly says "crossed WARNING threshold", so sustained-DEGRADED means already
at or above threshold, not approaching it.
