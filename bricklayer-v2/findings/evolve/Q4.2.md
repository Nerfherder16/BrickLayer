# Finding: Q4.2 — New project bootstrap sequence with modes

**Question**: What should the new project bootstrap sequence look like? Currently: copy template, edit project-brief, run question-designer. With modes: what's the right first question to ask?
**Agent**: research-analyst
**Verdict**: HEALTHY
**Severity**: Info
**Mode**: evolve
**Target**: `autosearch/CLAUDE.md` bootstrap sequence

## Summary

The BL 2.0 bootstrap sequence adds one decision step: "what mode are you in?" before generating the question bank. The first question to ask is always: **"Where are you in the lifecycle?"** — the answer determines the starting mode. Everything else in the bootstrap is unchanged. The question-designer agent gains one new input: the starting mode, which shapes which types of questions it generates.

## Evidence

### Current bootstrap (BL 1.x)

```
1. cp -r template/ {project}/
2. Edit project-brief.md
3. Drop specs into docs/
4. Edit constants.py
5. Edit simulate.py
6. python simulate.py → verify HEALTHY
7. Generate questions (question-designer agent)
8. git init && git commit
9. Start loop: claude --dangerously-skip-permissions "Read program.md..."
```

### New bootstrap (BL 2.0) — the one new step

```
1. cp -r template/ {project}/
2. Edit project-brief.md
3. Drop specs into docs/
4. Edit constants.py (if using simulation/benchmark mode)
5. Edit simulate.py (if using benchmark/evolve mode)
6. ** NEW: Determine starting mode **
7. Generate questions for that mode (question-designer, mode-aware)
8. git init && git commit
9. Start loop with DISABLE_OMC=1
```

Step 6 is the only new decision. Everything else is identical.

### The "what mode are you in?" decision tree

```
Is the project idea clearly defined?
  NO → Frontier mode (ideation first)
  YES ↓
Is the idea validated (key assumptions stress-tested)?
  NO → Research mode
  YES ↓
Is there a design to review before building?
  YES → Validate mode
Does working code exist?
  NO → skip to Research or Validate
  YES ↓
Is something broken that you need to find and diagnose?
  YES → Diagnose mode (default)
  NO ↓
Do you have a DIAGNOSIS_COMPLETE finding to implement?
  YES → Fix mode
  NO ↓
Do you have benchmarks and want to improve the system?
  YES → Evolve mode
Is this a compliance review?
  YES → Audit mode
Do you want to project what breaks next?
  YES → Predict mode
Is this ongoing health monitoring?
  YES → Monitor mode
```

For new projects, the path almost always starts at Frontier (concept unknown) or Research (concept known but untested).

### Question-designer agent update

The question-designer agent should receive the starting mode as context. Current invocation:
```
Act as the question-designer agent. Read project-brief.md, all files in docs/,
constants.py, and simulate.py. Generate the initial question bank in questions.md.
```

New invocation:
```
Act as the question-designer agent. Read program.md, project-brief.md, all files in docs/,
constants.py, and simulate.py. Generate the initial question bank in questions.md.
Starting mode: {frontier|research|validate|diagnose}.
Questions should be in BL 2.0 format with **Operational Mode**: {mode} field.
Question IDs should use the mode prefix: F{wave}.{n} for frontier, R{wave}.{n} for research, etc.
```

### First question for each starting mode

| Starting Mode | First question pattern |
|--------------|----------------------|
| Frontier | "What is the most compelling version of [concept]?" |
| Research | "Does [core assumption] hold under [stress condition]?" |
| Validate | "Does [design decision] produce the claimed behavior?" |
| Diagnose | "What is the root cause of [observed failure]?" |
| Fix | (requires DIAGNOSIS_COMPLETE input — not a first question) |

### No change to QUICKSTART.md structure

The existing QUICKSTART.md sequence is preserved with one addition between "Generate questions" and "Init git": a note that the question-designer should be told the starting mode.

## Verdict Rationale

HEALTHY: The BL 2.0 bootstrap adds exactly one decision step (mode determination) and one input to the question-designer agent (starting mode). All existing steps remain unchanged. The decision tree for mode selection is simple enough to include inline in CLAUDE.md without requiring a separate document.
