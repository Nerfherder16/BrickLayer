## Root Cause of Post-Removal Variance

### Non-JSON output from the model (score=0.00 root cause)

`eval_agent.py:_score_example()` immediately returns `0.0` when JSON parse fails:
```python
try:
    predicted = json.loads(predicted_raw)
except (json.JSONDecodeError, ValueError):
    return 0.0, {}  ← score=0.00 on any non-JSON output
```

The WARNING record (Q6.5: "mypy errors 0→0 (0 changes committed)") triggers this in 2/3 runs:
- Run 2, position 2: **0.00** (model produced prose instead of JSON)
- Run 4, position 2: **0.00** (same failure)

This is a stochastic JSON compliance failure — the question about mypy errors sometimes triggers
the model to reason in prose rather than follow the JSON instruction.

### High variance from 5-record sample

With 5 records and one problematic record failing stochastically, the variance is ±2 failures:
- Best case: 5/5 = 1.00 (no stochastic failures)
- Mid case: 3/5 = 0.60 (one stochastic failure)
- Worst case: 2/5 = 0.40 (two stochastic failures)

The pre-removal 6-record dataset had ONE consistently failing record, which paradoxically gave
more stable measurements (5 passing records rarely had stochastic failures → reliably 5/6).

## Recommendation

synthesizer-bl2 needs **10+ records** before a reliable baseline is possible. With 5 records:
- Sample variance: ±2 failures = ±40% score swings
- One stochastic failure skews average by 0.20

**Actionable fixes**:

   different verdict types) — prioritize non-WARNING records to balance the set
