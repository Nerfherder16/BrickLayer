# E15.1 — Fix improve_agent.py UnicodeDecodeError (encoding='utf-8' in subprocess reader)

**Question**: Fix improve_agent.py UnicodeDecodeError — add encoding='utf-8' to subprocess reader thread
**Verdict**: FIXED
**Confidence**: 0.92
**Date**: 2026-03-24
**Agent**: fix-implementer

---

## Summary

Applied `encoding="utf-8"` to the `subprocess.run` call in `masonry/scripts/eval_agent.py`. This is the same fix already applied to `optimize_with_claude.py` in E14.2. The crash in E14.8 (`'charmap' codec can't decode byte 0x8f`) was caused by `text=True` using Windows' default cp1252 encoding when reading Claude's subprocess stdout. With `encoding="utf-8"` added, the subprocess reader will correctly decode UTF-8 output from Claude.

---

## Evidence

### Root cause

In `eval_agent.py`, the `subprocess.run` call at line 297-309 used:
```python
text=True,
```
Without specifying `encoding`. On Windows, `text=True` defaults to the system locale encoding (cp1252 / charmap), which fails on byte 0x8F (a character outside cp1252's range). Claude's JSON output can include UTF-8 encoded characters (e.g., Unicode symbols in evidence text, special quotes, etc.).

### Fix applied

`masonry/scripts/eval_agent.py`, line 308-309:

**Before:**
```python
text=True,
```

**After:**
```python
text=True,
encoding="utf-8",
```

### Pattern consistency

- `optimize_with_claude.py` line 362: `encoding="utf-8"` already present (E14.2 fix)
- `eval_agent_live.py` line 191: `encoding="utf-8"` already present
- `eval_agent.py` line 309: NOW fixed — `encoding="utf-8"` added

All three subprocess callers in the eval/optimize pipeline now use explicit UTF-8 encoding. The pattern is consistent.
