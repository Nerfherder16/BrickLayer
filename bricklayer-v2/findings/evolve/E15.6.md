# E15.6 — Live Eval for quantitative-analyst — Authoritative Baseline

**Question**: Live eval for quantitative-analyst — establish authoritative baseline
**Verdict**: INCONCLUSIVE
**Confidence**: 0.85
**Date**: 2026-03-24
**Agent**: evolve-optimizer

## Summary

quantitative-analyst live eval cannot be run with current infrastructure. The eval_agent_live.py loads records from `scored_all.jsonl`, which only contains karen and git-nerd records. quantitative-analyst's 45 training records are in `scored_findings.jsonl` with a different input schema (`question_text` key instead of `question`). A data migration or harness update is required before live eval can run.

---

## Evidence

### Record distribution across training data files

| File | Agents present | quantitative-analyst records |
|------|----------------|------------------------------|
| scored_all.jsonl | karen (577), git-nerd (136) | 0 |
| scored_findings.jsonl | quantitative-analyst (45), regulatory-researcher (7), competitive-analyst (1) | 45 |
| scored_all_wave13.jsonl | quantitative-analyst (35), competitive-analyst (6), synthesizer-bl2 (6) | 35 |

**scored_all.jsonl** is the file that both `eval_agent.py` and `eval_agent_live.py` use. It does not contain quantitative-analyst records.

### Schema mismatch

`scored_findings.jsonl` uses `input.question_text` as the question field.
`eval_agent_live.py` constructs the prompt using `record.get("input", {}).get("question", "")` — returns empty string for `question_text`-schema records.

The E14.6 static eval of quantitative-analyst (score 0.45) was run against `scored_findings.jsonl` records, likely via a version of eval_agent.py that was patched to use that file, or the records were temporarily merged.

### What would resolve this

Option A — Migrate records: Add all `scored_findings.jsonl` quantitative-analyst records to `scored_all.jsonl`, normalizing the `question_text` key to `question`. This is a one-time migration.

Option B — Update eval harness: Add a `--data-file` flag to `eval_agent_live.py` so it can read from `scored_findings.jsonl`. Also add fallback: `record.get("input", {}).get("question", record.get("input", {}).get("question_text", ""))`.

Option C — Generate live records: Use `eval_agent_live.py` to generate new quantitative-analyst records (similar to E12.1 for research-analyst) from fresh questions, bypassing the schema issue entirely.
