# E2.1: Karen eval 0.55→0.30 — regression root cause diagnosis

**Status**: WARNING
**Date**: 2026-03-24
**Mode**: evolve
**Agent**: evolve-optimizer

## Hypothesis

The Wave 1 prompt fix (commit-type-first rule) should have raised karen's score from 0.55 to
~0.90. Instead the score dropped to 0.30. This question diagnoses whether this is a true
regression (fix made things worse) or an artifact of the eval methodology.

## Root Cause Analysis

### Cause 1: Eval uses last-N records (not random) — sampling shift

`eval_agent.py:run_eval()` uses `held_out = records[-eval_size:]` — the LAST 20 karen records.
Between Wave 1 eval (score=0.55) and Wave 2 eval (score=0.30), the training data file
`scored_all.jsonl` gained new records. The last 20 changed:

**Wave 1 eval (0.55)**: Mixed set — included records with expected="skipped"/"reverted" where
karen predicting "skipped" scores 0.75+ → some passes.

**Wave 2 eval (0.30)**: The last 20 now consist of:
- Records 0-14: feat/docs/refactor commits with `files_modified = ["ROADMAP.md"]`,
  expected="updated" — karen predicts "skipped", scores 0.25-0.45 (fail)
- Records 15-16: revert commits, claude failed to produce output, score=0.00
- Records 17-19: revert commits, expected="reverted", karen predicts "skipped",
  scores 0.85/0.85/0.60 (pass by partial action match + quality match for qs=0.0 → 0.5)

6/20 passed = 0.30. Confirmed by manual score reconstruction for all 20 examples.

**Conclusion**: The 0.55 → 0.30 drop is primarily a sampling artifact. The Wave 1 eval
happened to select records with expected="skipped" (easy wins). Wave 2 eval selected records
where every expected="updated" case has `files_modified = ["ROADMAP.md"]`.

### Cause 2: Prompt fix ineffective — model ignores commit-type-first rule

Despite the rule: "A `feat:`, `fix:`, `refactor:`, `docs:`, `perf:`, or `test:` commit ALWAYS
warrants action='updated' regardless of which files were modified", karen continues to output
`action="skipped"` for ALL feat/docs commits in the last 20 records.

Evidence from examples 0-14:
- `commit_subject = "feat(bricklayer): C-07/C-08 http and subprocess runner format"`
- `files_modified = ["ROADMAP.md"]`
- predicted `action = "skipped"`
- `changelog_entry = "No changes detected — session produced no file modifications"`

The model reasons: "ROADMAP.md is in files_modified, which means documentation was already
written for this commit. There's nothing for me to add. Skip." This is CORRECT reasoning
given the training data context (these records capture karen's own output commits), but
produces WRONG results in the eval context.

**The rule "files_modified is scope context, NOT a signal to skip" is insufficient** because
the model has an even stronger signal: "ROADMAP.md IS my output file — if it appears in
files_modified, I've already run."

### Root Cause (structural): Training data self-reference

`score_ops_agents._score_karen()` finds commits that TOUCHED karen's output files and records
them as training examples:

```
input.commit_subject = subject of the commit that wrote ROADMAP.md  ← karen's output commit
input.files_modified = ["ROADMAP.md"]                                ← karen's own writes
output.doc_files_written = 1                                         ← karen wrote 1 doc
```

Karen is then evaluated on these records. When it sees a feat commit with files_modified=
["ROADMAP.md"], it correctly infers "ROADMAP.md was already updated in this commit" → skip.
But the expected label says "updated".

The eval is asking karen to do the WRONG thing: "should you document a commit that already
contains your documentation output?" The correct answer is "the docs are already written,
nothing to do" but the label says "updated".

**The Wave 1 prompt fix cannot work** because it fights against the model's correct reasoning.
The training data must be restructured so that:
- `input.files_modified` = source code files from the TRIGGERING commit (parent)
- `input.commit_subject` = the triggering commit's subject
- `output.doc_files_written` = whether karen then wrote docs (current record's output)
