## Evidence

### Root cause

`eval_agent_live.py._load_agent_records()` only reads `scored_all.jsonl`. That file contains karen (577) and git-nerd (136) records; zero quantitative-analyst records. quantitative-analyst's 45 training records live in `scored_findings.jsonl` with a different input schema (`input.question_text` vs `input.question`).

The `question_text` vs `question` fallback was already present at the prompt-construction site (lines 179-180 of the original file), but it was unreachable because no records were being loaded for the agent in the first place.

### Fix applied

File: `masonry/scripts/eval_agent_live.py`

1. Added `_DATA_FILE_FINDINGS` constant pointing to `masonry/training_data/scored_findings.jsonl`.
2. Refactored `_load_agent_records()` to use an internal `_read_from_file()` helper.
3. After reading `scored_all.jsonl`, if zero records are found for the agent, the function reads `scored_findings.jsonl` with `normalize_question_key=True` — this converts `input.question_text` → `input.question` in-memory before appending the record.
4. The `id_prefix` filter uses `question_id` from either the top-level field or `input.question_id`.
