## Evidence

### What the model actually produces

When given the generic instruction `"Respond ONLY with a valid JSON object..."`, the
research-analyst ignores it and does actual agentic research:

```
Based on my investigation of the recall-arch-frontier project, I need to stress-test
the claim that "test-writer will add fast-mode tests..."

## Critical Infrastructure Gaps
**1. No test-writer agent exists**
...
```

This is correct agent behavior — the research-analyst is designed to do real research using tools
(Read, Glob, Grep, WebFetch, WebSearch). The eval instruction is insufficient to override this.

### Why the metric is misleading

The 1/5 "pass" scored 0.60:
- Expected verdict: HEALTHY
- Predicted verdict: WARNING (WRONG)
- Score: 0 (verdict) + 0.4 (good evidence) + 0.2 (confidence) = **0.60 PASSES**

A verdict mismatch can still "pass" because `build_metric()` weights evidence quality (0.4) and
confidence calibration (0.2) equally to verdict match (0.4). A good evidence response with wrong
verdict passes. This is a metric calibration flaw.

---

## Root Cause Analysis

### 1. Eval instruction insufficient for agentic agents

The research-analyst was built to use tools and produce nuanced findings — not to classify inputs
as JSON. The generic eval instruction doesn't override 17K chars of agentic system prompt.

**Fix**: Add a `_RESEARCH_JSON_INSTRUCTION` to `eval_agent.py` with explicit field specification
matching the `FindingPayload` schema.

### 2. Training data too small and homogeneous

5 records, all HEALTHY verdicts, from a single campaign phase. A meaningful eval needs:
- Minimum 20 records
- Mixed verdicts (HEALTHY, WARNING, FAILURE, INCONCLUSIVE)
- From diverse question types and projects

**Fix**: Run more research-analyst sessions and capture findings to `scored_all.jsonl`.

### 3. Fundamental eval design mismatch

The research-analyst's job is to DO research (read files, search web, analyze evidence) and
return a structured finding. Evaluating it by replaying historical inputs as classification tasks
doesn't test what it's actually supposed to do. The eval should instead:
- Give it a fresh research question it hasn't seen before
- Let it run with tools enabled
- Score the result on evidence quality, verdict calibration, and summary accuracy

## Recommendation

Before investing in optimizing research-analyst accuracy, fix the two blockers:

1. **Eval harness instruction fix** (1 hour): Add `_RESEARCH_JSON_INSTRUCTION` with explicit
   output schema. This should improve the JSON compliance for examples that don't need deep research.

2. **Generate more training data** (ongoing): After running more research-analyst sessions, build
   up the record set from 5 to 20+ with diverse verdicts.

Until then, the 0.20 score is meaningless as a performance indicator. The `--system-prompt-file`
fix is confirmed working (eval now runs without the "command line too long" error).
