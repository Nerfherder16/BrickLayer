## Root Cause: Structural Mismatch

The research-analyst's production quality depends on tool-augmented reasoning:
- In production: reads files, searches code, produces evidence-backed verdicts
- In eval: runs `--no-session-persistence`, no tools, answers from knowledge alone

The same question produces different reasonable verdicts depending on how much context
the agent has. Without tool access, all three of FAILURE/WARNING/INCONCLUSIVE are
defensible for "Is this system design sound?" questions.

**Stochastic failure pattern**: 4+ records consistently flip between verdicts across
runs, causing ±15% variance in any single-run eval:

| Record | Observations | Stochastic? |
|--------|-------------|-------------|
| E9.1-rec-1 (FAILURE) | WARNING or FAILURE | Yes — both defensible |
| E9.1-rec-2 (WARNING) | WARNING, HEALTHY, INCONCLUSIVE | Yes — all defensible |
| E9.3-rec-3 (WARNING) | WARNING or prose | Yes — stochastic JSON compliance |
| E9.4b-rec-1 (FAILURE) | WARNING or FAILURE | Yes — severity judgment |
| E9.4b-rec-2 (WARNING) | WARNING or INCONCLUSIVE | Yes — evidence threshold |

## Stopping Criterion Reached

After 9 waves of evolve-mode optimization on research-analyst:
- **Practical ceiling**: 0.61 average (single-run best so far)
- **Minimum floor**: 0.44 (worst run)
- **Required for target**: 0.85 (15+/18 consistently)

The gap from 0.61 to 0.85 requires either:
1. **Live eval** (Path B, E6.3): Eval with tools enabled; eliminates tool-access mismatch
2. **Much larger dataset**: 50+ records so stochastic variance averages out
3. **Deterministic questions**: Questions with logically-forced correct verdicts only

None of these can be resolved within evolve mode without architectural changes to the
eval harness. Accept 0.61 as the practical ceiling for tool-free eval.
