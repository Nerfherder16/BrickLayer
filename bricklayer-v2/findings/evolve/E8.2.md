## Why Variance Persists

The 18 records split into three scoring tiers:

**Tier 1 (stable high, 0.80-0.99)**: ~2-3 records per run
- Reasoning questions where the model produces clean JSON + correct verdict match
- These records earn verdict_match (0.4) + evidence_quality (0.4) + confidence (0.2) = 0.97-0.99
- Score is deterministic because the correct verdict is unambiguous and the model reliably outputs it

**Tier 2 (borderline, 0.45-0.60)**: ~8-12 records per run
- Either prose output scoring 0.40 (evidence quality only, 2-stage eval)
- Or clean JSON with correct verdict but lower evidence quality: 0.4 + 0.2 + 0.2 = 0.80
- Or clean JSON with wrong verdict but good evidence: 0.0 + 0.4 + 0.2 = 0.60
- Borderline records flip above/below 0.50 threshold between runs

**Tier 3 (stable low, 0.40)**: ~3-5 records per run
- Code-inspection questions producing prose → 2-stage eval → 0.40 floor

The high variance comes from Tier 2 swinging across the 0.50 threshold. In Run 1, more Tier 2
records happened to score above 0.50, giving 12/18. In Runs 2-3, fewer did, giving 6-7/18.

## Root Cause of Borderline Scoring

The borderline scoring pattern reveals a structural issue: the eval metric rewards both
verdict correctness AND evidence quality, but these two components can compensate for each
other:

- Wrong verdict + good evidence = 0.0 + 0.4 + 0.2 = 0.60 (passes)
- Right verdict + weak evidence = 0.4 + 0.2 + 0.2 = 0.80 (passes)
- Wrong verdict + weak evidence = 0.0 + 0.2 + 0.2 = 0.40 (fails)

This means records with correct verdicts but stochastic evidence quality flip above/below
the threshold. Records with incorrect verdicts but consistent evidence quality reliably score
0.60 (always passing). The E8.2-rec-5 WARNING record (build_metric bias) may be
scoring 0.60 when the model says HEALTHY instead of WARNING (wrong verdict, good evidence).

**Implication**: To reliably reach AT TARGET (0.85), the training set needs records where:
1. The verdict is unambiguous (the model should agree ~100% of time)
2. The evidence is specific enough to always score 0.4 (not 0.2)

Records with ambiguous verdicts should be excluded from the eval set even if their expected
output is theoretically correct.
