# Finding E8.4 — masonry-guard.js False Positive Fix

**ID**: E8.4
**Mode**: evolve
**Verdict**: IMPROVEMENT
**Date**: 2026-03-24

---

## Summary

Fixed `hasErrorSignal()` in `masonry-guard.js` to scope error signal detection to
error-bearing fields only, rather than serializing the entire `tool_response` object
with `JSON.stringify()`. The bug caused false-positive 3-strike warnings whenever legitimate
file edits contained `old_string` or code content matching ERROR_SIGNALS patterns. Fix
verified with 6 test cases: 2 false-positive scenarios now return false, 4 legitimate error
scenarios still return true.

---

## Root Cause

**Before fix**: `hasErrorSignal()` called `JSON.stringify(response)` on the entire
`tool_response` object. For Edit/Write tool calls, the response object may contain
`old_string`, `new_string`, or `content` fields with source code or markdown content
that matches the ERROR_SIGNALS patterns (`/\bcannot\b/i`, `/\bundefined\b/i`,
`/\bnot found\b/i`, `/\bError\b/`). These patterns are common in Python, JavaScript,
and markdown documentation.

**After fix**: A `_errorTexts()` helper extracts only error-bearing fields:
- String responses → checked directly (Bash stdout/stderr are strings)
- Objects with `is_error: true` or `type: "error"` → safe to serialize (error-flagged responses won't contain old source code)
- Other objects → only check `['error', 'stderr', 'errorMessage', 'message', 'reason', 'details']`

This eliminates scanning of `old_string`, `new_string`, `content`, `result`, `stdout`
fields that legitimately contain source code and documentation.

## Test Results

| Test case | Before | After | Expected |
|-----------|--------|-------|----------|
| Edit response with oldString containing "cannot"/"PENDING" | **TRUE (false positive)** | FALSE | FALSE |
| Actual error response (`is_error: true`) | TRUE | TRUE | TRUE |
| String Bash error ("Error: cannot...") | TRUE | TRUE | TRUE |
| String success ("File edited successfully") | FALSE | FALSE | FALSE |
| Object with `message: "Cannot resolve..."` | TRUE | TRUE | TRUE |
| Python code in `content` field | **TRUE (false positive)** | FALSE | FALSE |

---

## Code Change

**`masonry/src/hooks/masonry-guard.js`**:

Added `_ERROR_FIELDS` constant and `_errorTexts()` helper function. Modified both
`hasErrorSignal()` and `extractErrorSummary()` to use `_errorTexts()` instead of
`JSON.stringify(response)`.

```javascript
// Before: scans entire serialized payload
const text = typeof response === 'string' ? response : JSON.stringify(response);
return ERROR_SIGNALS.some(re => re.test(text));

// After: scopes to error-bearing fields only
const texts = _errorTexts(response);
return texts.some(text => ERROR_SIGNALS.some(re => re.test(text)));
```

The `_errorTexts()` function:
1. String responses → `[response]` (unchanged behavior)
2. Objects with `is_error` or `type: "error"` → serialize (explicit error flag means safe to scan)
3. Other objects → extract only `['error', 'stderr', 'errorMessage', 'message', 'reason', 'details']`

---

## Impact

- **False positive rate**: From structurally unbounded (~5.3/session from E7.2-pilot-4) → 0 for Edit/Write tool calls
- **True positive rate**: Preserved — Bash errors (string responses), explicit error objects, and objects with `message`/`error` fields are still detected
- **3-strike warning queue**: Will no longer accumulate false warnings from legitimate campaign file edits
