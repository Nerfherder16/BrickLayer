# Finding E4.2 — Optimizer Write-Back Scope Guard

**ID**: E4.2
**Mode**: evolve
**Verdict**: IMPROVEMENT
**Date**: 2026-03-24

---

## Summary

Added `target_paths` optional parameter to `writeback_optimized_instructions()` and defaulted
`optimize_with_claude.py` to pass only `[md_path]` (the source file). This prevents the cross-file
contamination that caused the Wave 2 karen overwrite incident.

---

## Delta

- Metric: optimizer write-back scope (files modified per run)
- Baseline: ALL copies (4+ files) modified on every optimization run
- After: Only the source file modified by default; `target_paths=None` (all copies) still available for explicit use
- Improvement: Eliminates accidental cross-variant contamination

---

## Changes Made

### `masonry/src/writeback.py`
Added `target_paths: list[Path] | None = None` parameter:
```python
if target_paths is not None:
    candidates = list(target_paths)  # scope guard: only explicit files
else:
    # original behavior: all copies
    candidates = [base_dir/.claude/agents/{agent}.md, ...]
```

### `masonry/scripts/optimize_with_claude.py`
Changed writeback call to pass scope guard:
```python
updated_files = writeback_optimized_instructions(
    base_dir=base_dir,
    agent_name=agent_name,
    instructions=instructions,
    optimized_at=optimized_at,
    target_paths=[md_path],  # scope guard: source file only
)
```

---

## Regression Check

The `target_paths=None` default preserves original behavior for any callers that don't pass the
parameter. Existing callers (DSPy optimizer, MCP tools) remain unaffected.

---

## Remaining Risk

If the engineer running optimization wants to propagate the DSPy section to all copies, they must
now do so explicitly (remove the `target_paths` argument or call writeback separately). This is
intentional — deliberate propagation is safer than accidental propagation.

---

## Files Changed

- `masonry/src/writeback.py` — `target_paths` parameter added
- `masonry/scripts/optimize_with_claude.py` — scope guard enabled
