## Summary

Designed a training data generation plan for research-analyst that can produce 20+ diverse records
with varied verdicts (WARNING, FAILURE, INCONCLUSIVE, PROMISING, HEALTHY). The root cause of the
current 5-record/all-HEALTHY dataset is question framing bias — all 5 questions were
hypothesis-verification tasks on a single project that happened to verify cleanly. The fix is
verdict-targeted question design using question templates that reliably trigger each verdict type,
run across multiple source projects.

---

## Current State Analysis

### Why 5 records / all HEALTHY

The 5 existing records come from a single Recall project session. All questions follow the pattern:
> "agent X will do Y to component Z"

These are predictive hypothesis checks — when the agent verifies the code is correct, verdict=HEALTHY.
There is no mechanism driving the agent toward WARNING/FAILURE unless it finds actual issues.

Additionally:
- `question` field is empty in all records (only `question_text` populated) — minor schema issue
- All have `score=60` (old 0-100 scoring) — borderline, pre-PROMISING-fix scoring
- From only one project (Recall), one phase (Wave 4 hypothesis testing)

### Eval Design Mismatch (carries forward from E3.3)

The current eval harness replays historical inputs to the agent without tools. research-analyst is
an agentic researcher — 17K chars of system prompt drive it to use Read/Grep/WebSearch tools and
produce full research outputs. Without tools in eval, it either ignores JSON instruction (agentic
behavior) or returns a low-quality JSON without evidence.

This mismatch cannot be fully resolved without a live-eval approach. But generating better training
data with diverse verdicts will improve the measurement signal even under the current eval.

### Question Templates by Verdict

#### WARNING questions (aim: find partial/degraded state)
These questions target areas likely to have *partial* coverage or *minor* issues:

```
1. Does [component]'s test suite cover error paths, or only the happy path?
2. Is [module]'s documentation accurate relative to the current implementation?
3. Are there any deprecated APIs or patterns in [codebase] that could break in future upgrades?
4. Does [feature] have any edge cases that are tested but not handled in production?
5. Are log messages in [service] sufficiently structured for debugging production incidents?
```

For bricklayer-v2 specifically:
- "Does `masonry/scripts/eval_agent.py` have test coverage for its metric functions?"
- "Are there any agent .md files in the fleet with outdated tool listings or wrong file paths?"
- "Does the four-layer routing system log sufficient context to diagnose incorrect routing decisions?"
- "Are there question IDs in bricklayer-v2/questions.md that are missing corresponding finding files?"

#### FAILURE questions (aim: find critical gaps)
Target known weak areas:

```
1. Is there any authentication/authorization in [service API endpoint X]?
2. Does [component] handle concurrent access correctly, or is there a race condition?
3. Are there any hardcoded secrets or credentials in [codebase]?
```

For bricklayer-v2 specifically:
- "Is the `scored_all.jsonl` training file protected against accidental truncation during append operations?"
- "Does the optimizer write-back mechanism have any remaining cross-file contamination vectors after E4.2?"
- "Are there any agent files in the registry that reference non-existent tool names?"

#### INCONCLUSIVE questions (aim: need external data to conclude)
Target questions that require unavailable information:

```
1. What is the production error rate for [service] over the past 30 days?
2. How does [system]'s performance compare to industry benchmarks?
3. What percentage of users encounter [specific error] in production?
```

For bricklayer-v2 specifically:
- "What is the false-positive rate of the masonry-guard.js hook in production across Tim's sessions?"
- "How often does the semantic routing layer actually improve routing quality over the deterministic layer?"
- "What fraction of BrickLayer campaigns successfully reach synthesis.md without human intervention?"

#### PROMISING questions (aim: find improvement opportunities)
Target areas with clear optimization potential:

```
1. Could [component X] be significantly simplified without losing functionality?
2. Is there a high-ROI caching or memoization opportunity in [hot code path]?
3. Could [agent prompt] be restructured to improve response quality while reducing token usage?
```

For bricklayer-v2 specifically:
- "Could the `scored_all.jsonl` record extraction be automated from git history instead of manual curation?"
- "Would adding verdict-level scoring weights to `build_metric()` produce a more meaningful eval score?"

#### HEALTHY questions (aim: confirm things that are actually fine)
Target well-implemented areas:

```
1. Is [well-tested component] implemented correctly per its specification?
2. Does [mature feature] handle the documented use cases correctly?
```

For bricklayer-v2 specifically:
- "Is the karen eval metric (`build_karen_metric`) correctly scoring the action/quality_score dimensions?"
- "Does the `optimize_with_claude.py` script correctly scope write-back to target files only?"

## Eval Approach Recommendation

The current eval (replay historical inputs without tools) cannot accurately measure research-analyst
performance. Two paths:

### Path A — Better training data only (low effort)
Generate 25 diverse records with the question templates above. This won't fix the fundamental
eval-design-mismatch, but it will:
- Surface the true HEALTHY/WARNING/FAILURE distribution (not all-HEALTHY bias)
- Give the eval harness verdict-balanced examples to test against
- Measure whether the research-analyst can correctly classify under the JSON instruction

**Cost**: ~2 hours of running research-analyst on targeted questions + reviewing outputs.

### Path B — Live eval harness (higher effort, more accurate)
Build a separate eval mode in `eval_agent.py` that:
- Runs research-analyst with tools enabled (no `--no-tools` flag)
- Uses a "gold standard" question bank with known expected verdicts
- Scores the live output on evidence quality + verdict calibration
- Suitable for running monthly (not per-commit, since it requires 30-60 min)

**Cost**: ~4 hours dev + ~1 hour setup per eval run.

**Recommendation**: Start with Path A. If after 25 records the eval score remains below 0.50,
implement Path B.

---

## Implementation Steps (Path A)

1. Write 25 questions following the verdict templates above across 3+ projects
2. Run research-analyst on each question using `masonry_run_question` or direct agent invocation
3. Review each output — correct verdict if agent mis-labeled edge cases
4. Append to `masonry/training_data/scored_all.jsonl` with proper schema:
   - `agent: "research-analyst"`
   - `input.question_text`: the question asked
   - `input.question`: same (fix empty field issue)
   - `output.verdict`: actual verdict produced
   - `output.evidence`: research output
   - `output.confidence`: 0.0-1.0
   - `score`: 0-100 converted to new scoring
5. Re-run eval: `python masonry/scripts/eval_agent.py research-analyst`
6. Target: eval score ≥ 0.70 with 25 records (not 0.85 — lower target given eval-design-mismatch)
