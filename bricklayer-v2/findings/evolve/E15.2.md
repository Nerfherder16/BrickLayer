| E9.4-rec-1 | INCONCLUSIVE | WARNING | [timeout — question type: eval coverage gap assessment] |

### Root cause

These questions ask about production/runtime soundness or comprehensive coverage — properties that cannot be confirmed by reading static code. The routing pipeline code exists and is readable; whether it's "sound for production usage" requires running it in production. The old Rule 4 said "Never return INCONCLUSIVE for questions answerable by reading a file" — but reading the routing code shows it's *implemented*, not that it's *production-sound*. The agent correctly reads the code, finds an implementation, and then misfires WARNING on the gap.

### Fix applied

Rule 4 in `bricklayer-v2/.claude/agents/research-analyst.md` and `template/.claude/agents/research-analyst.md`:

**Added INCONCLUSIVE trigger (d)**: "OR (d) the question asks about production/real-world behavior that cannot be confirmed from code alone."

**Added INCONCLUSIVE trigger guidance block**: Explicit examples of production/soundness questions that should produce INCONCLUSIVE rather than WARNING.

**Added cosmetic pattern example** (fixes E12.1-live-15): "A mismatched print message (e.g., `print('>120s')` when the actual timeout is 180s) is documentation-only and does not affect behavior. This is HEALTHY, not WARNING."
