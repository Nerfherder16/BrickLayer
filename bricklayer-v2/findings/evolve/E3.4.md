# Finding E3.4 — Optimizer Write-Back Mechanism Audit

**ID**: E3.4
**Mode**: evolve
**Verdict**: WARNING
**Date**: 2026-03-24

---

## Summary

The `writeback_optimized_instructions()` function in `masonry/src/writeback.py` writes optimized
instructions to ALL copies of an agent .md file simultaneously, with no scope guard. This is the
mechanism that caused the Wave 2 incident where `karen.md` was overwritten with wrong content
across 4 files. The write-back itself is correct in scope but lacks pre-flight validation and
rollback capability.

---

## What writeback_optimized_instructions() Does

`optimize_with_claude.py` calls writeback via this flow:

1. `_find_agent_md(agent_name, base_dir)` — finds the PRIMARY agent .md (project-level first, then global)
2. `_read_current_instructions(md_path)` — reads and STRIPS existing DSPy section → sends bare instructions to Claude
3. Claude generates `improved_instructions` JSON
4. `writeback_optimized_instructions(base_dir, agent_name, instructions, ...)` — writes to ALL copies:
   - `base_dir/.claude/agents/{agent}.md`
   - `base_dir/{project}/.claude/agents/{agent}.md` (all subdirs one level deep)
   - `~/.claude/agents/{agent}.md`

**Critical**: The write-back searches ALL subdirectories one level deep and writes to every found copy.
For the BrickLayer repo with `template/`, `template-frontier/`, `bl-audit/`, and `bricklayer-v2/`
subdirs, this means 4+ copies are modified simultaneously.

---

## Wave 2 Incident Root Cause

Wave 2 established that `karen.md` was overwritten by commit `801cf17`. Based on this audit:

1. The optimizer was likely run with `base_dir = C:/Users/trg16/Dev/Bricklayer2.0`
2. `_find_agent_md` found the project-level `.claude/agents/karen.md` as the PRIMARY source
3. If that file was already the project-org version at the time, it was read as source
4. `writeback_optimized_instructions` then wrote the generated instructions to ALL 4 copies:
   - `.claude/agents/karen.md`
   - `template/.claude/agents/karen.md`
   - `template-frontier/.claude/agents/karen.md`
   - `~/.claude/agents/karen.md`

The result: all copies became the project-org variant (whatever the project-level source was).

---

## Guards Present

| Guard | Present? |
|-------|---------|
| Only modifies DSPy section (not full file) | YES — regex section replacement |
| `strip_optimized_instructions()` for rollback | YES — but not called automatically |
| Pre-flight verification (will the output be correct?) | NO |
| Backup/git stash before writing | NO |
| Scope limit (only write to the file that was READ) | NO |
| Diff preview before applying | NO |
| Confirmation prompt | NO |

---

## Risk Assessment

**Current risk level**: MEDIUM-HIGH

The missing scope limit is the most dangerous gap. If the optimizer reads source A and writes to
copies B, C, D (which may be correct versions of a different variant), all copies get contaminated.

This risk exists even though the writeback only modifies the DSPy section — because the DSPy
section carries the entire optimized instruction set which overrides the agent's core behavior.

---

## Proposed Fix

**Minimal fix (scope guard)**:
Limit writeback to the same file that was read for optimization, unless `--all-copies` is explicitly
passed:

```python
# In optimize_with_claude.py after building `md_path`:
updated_files = writeback_optimized_instructions(
    base_dir=base_dir,
    agent_name=agent_name,
    instructions=instructions,
    optimized_at=optimized_at,
    target_paths=[md_path],  # ADD: limit to source file only
)
```

**Additional hardening**:
- Auto-stash `git stash` before writeback, unstash on error
- Log all modified paths to console before writing (always, not just verbose mode)
- Add `--dry-run` to `optimize_with_claude.py` that shows the diff, not just the prompt

## Files to Fix
