# Finding: Q4.1 — Template evolution for BrickLayer 2.0 mode system

**Question**: What changes are required to `template/` to support BrickLayer 2.0's mode system? Should `template/` become `template-diagnose/` with sibling templates for each domain?
**Verdict**: HEALTHY
**Severity**: Info
**Mode**: evolve
**Target**: `autosearch/template/`

## Summary

`template/` should NOT be renamed to `template-diagnose/`. A single template directory serves all modes because the mode is a runtime parameter, not a structural variant. The required changes to `template/` are additive: add `modes/` directory with the 9 mode programs, add `monitor-targets.md` and `ideas.md` stubs, and update `program.md` to the BL 2.0 master dispatcher. `simulate.py` stays as-is — it's used by the Benchmark runner when quantitative measurement is needed, not by all modes. No per-mode template directories are needed.

## Evidence

### Why NOT separate templates per mode

A single `template/` works because:
1. All modes share the same project structure (`findings/`, `results.tsv`, `synthesis.md`, `project-brief.md`)
2. Mode selection is a runtime decision by the user, not a structural one at project creation
3. Creating `template-frontier/`, `template-diagnose/`, etc. would duplicate `findings/`, `constants.py`, `project-brief.md`, and `program.md` across 9 directories — pure waste
4. Any project can switch modes mid-lifecycle (Recall started in Diagnose, will move to Monitor)

The right model: one template, with all mode programs included, and mode selection at session start.

### Required additions to `template/`

**1. Add `modes/` directory** — copy all 9 mode programs from `bricklayer-v2/modes/`:
```
template/
  modes/
    frontier.md
    research.md
    validate.md
    benchmark.md
    diagnose.md
    fix.md
    audit.md
    evolve.md
    predict.md
    monitor.md
```

**2. Update `program.md`** — replace current BL 1.x program.md with the BL 2.0 master dispatcher (the session start protocol, mode dispatch table, cross-mode handoffs).

**3. Add stub files for mode-specific outputs**:
```
template/
  ideas.md          ← Frontier mode output (empty stub)
  monitor-targets.md ← Monitor mode: metrics + thresholds (empty stub with format example)
  audit-checklist.md ← Audit mode (empty stub)
  failure-cascade-map.md ← Predict mode (empty stub)
  benchmarks.json   ← Benchmark mode (empty stub: {})
```

**4. Update `questions.md` stub** — add example questions in BL 2.0 format with `**Operational Mode**:` field.

**5. No change to `simulate.py`** — it's used specifically by the Benchmark and performance runners. Modes that don't need simulation (frontier, research, fix, monitor) simply don't call it. It remains in template/ as an optional tool.

**6. No change to `constants.py` template** — still the right pattern for threshold definition.

**7. No change to `.claude/agents/`** — existing agents (quantitative-analyst, regulatory-researcher, competitive-analyst, etc.) are still valid. New agents (if needed for specific modes) are added per-project.

### What does NOT change

- `project-brief.md` stub — unchanged, highest-authority document
- `findings/` directory structure — unchanged, all modes write here
- `results.tsv` format — unchanged
- The bootstrap sequence in `CLAUDE.md` — nearly unchanged (step 7 adds: "copy modes/ from template")
- `simulate.py` — stays as-is; Benchmark mode uses it; other modes ignore it

## Verdict Rationale

HEALTHY: The template structure evolves cleanly by addition, not restructuring. A single template directory serves all 9 modes by making mode selection a runtime decision rather than a structural one. The changes are minimal and backward-compatible — existing projects copied from the old template continue to work.
