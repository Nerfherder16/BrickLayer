# Finding: Q1.4 — DIAGNOSIS_COMPLETE suppression mechanism in bl/questions.py

**Question**: How should DIAGNOSIS_COMPLETE suppression work in `bl/questions.py`? What is the mechanism to park a question and re-activate it when code changes?
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: High
**Mode**: diagnose
**Target**: `bl/questions.py`, `bl/findings.py`, `bl/campaign.py`

## Summary

DIAGNOSIS_COMPLETE suppression requires three interlocking mechanisms: (1) status tracking — questions with this verdict get status `"DIAGNOSIS_COMPLETE"` in questions.md, excluded from `get_next_pending()`; (2) hypothesis generator prohibition — the followup generator in `campaign.py` must never create re-check questions for DIAGNOSIS_COMPLETE findings; (3) re-activation — a `check_diagnosis_complete_reactivations()` function runs at campaign start and re-activates questions whose code-file targets have changed in git since the finding was written. This directly fixes the 18-wave re-check loop observed in the Recall campaign.

## Evidence

### Why this matters

bricklayer-meta Wave 1 (Q4.1) established that the double-decay finding was re-checked 18 consecutive waves — the root cause was identified at code level in Q33.2b, but no suppression mechanism existed to stop the hypothesis generator from producing more re-checks. At ~7 questions/wave, 18 waves of re-checks consumed ~126 question slots for a known-FAILURE finding with a known fix. DIAGNOSIS_COMPLETE suppression would have stopped this after 1 wave.

### Mechanism 1: Status tracking (questions.py)

`_mark_question_done()` in findings.py maps DIAGNOSIS_COMPLETE → status `"DIAGNOSIS_COMPLETE"` (not "DONE" — it needs its own status to distinguish from a resolved question).

`get_next_pending()` in questions.py currently returns any question with `status == "PENDING"`. Add exclusion:
```python
def get_next_pending(questions: list[dict]) -> dict | None:
    for q in questions:
        if q["status"] == "PENDING":
            return q
    return None
# No change needed — DIAGNOSIS_COMPLETE status != PENDING, so it's already excluded
```

`parse_questions()` reads status from `get_question_status()`, which reads results.tsv. Since `_mark_question_done()` writes `"DIAGNOSIS_COMPLETE"` status to questions.md, the questions.md file has the suppressed status, and `parse_questions()` returns it correctly.

### Mechanism 2: Hypothesis generator prohibition (campaign.py)

The `generate_followup()` call at campaign.py:60 generates follow-up questions for FAILURE/WARNING verdicts. Add DIAGNOSIS_COMPLETE to the check conditions:

Currently:
```python
if result.get("verdict") in ("FAILURE", "WARNING"):
    followup_ids = generate_followup(question, result, cfg.questions_md)
```

No change needed here — DIAGNOSIS_COMPLETE is not in `("FAILURE", "WARNING")`, so `generate_followup` is never called for DIAGNOSIS_COMPLETE verdicts.

The hypothesis generator in `bl/hypothesis.py` is the other generation path. It must receive a list of DIAGNOSIS_COMPLETE findings to exclude from re-check generation. Add a parameter:

```python
def generate_hypotheses(questions_md: Path, results_tsv: Path,
                        suppress_ids: set[str] | None = None) -> None:
    """suppress_ids: question IDs with DIAGNOSIS_COMPLETE — never generate re-checks for these"""
```

In `campaign.py:run_campaign()`, collect suppressed IDs before calling `generate_hypotheses`:
```python
dc_ids = {q["id"] for q in parse_questions() if q["status"] == "DIAGNOSIS_COMPLETE"}
generate_hypotheses(cfg.questions_md, cfg.results_tsv, suppress_ids=dc_ids)
```

### Mechanism 3: Re-activation on code change (new function)

This is the key innovation. When the files mentioned in a DIAGNOSIS_COMPLETE finding's Fix Specification are changed, the finding is no longer a valid terminal state — the fix may have been attempted, and the question should re-run to verify.

```python
def check_diagnosis_complete_reactivations(project_root: Path) -> list[str]:
    """
    For each DIAGNOSIS_COMPLETE finding, check if any file in its Fix Specification
    has changed in git since the finding was written.
    Returns list of question IDs re-activated to PENDING.
    """
    import subprocess
    reactivated = []

    for finding_file in sorted(cfg.findings_dir.glob("*.md")):
        qid = finding_file.stem
        content = finding_file.read_text(encoding="utf-8")

        # Only process DIAGNOSIS_COMPLETE findings
        if "**Verdict**: DIAGNOSIS_COMPLETE" not in content:
            continue

        # Extract files mentioned in Fix Specification section
        fix_spec = _extract_fix_spec_files(content)
        if not fix_spec:
            continue

        # Check if any listed file has changed since finding was written
        finding_mtime = finding_file.stat().st_mtime
        for filepath in fix_spec:
            full_path = project_root.parent / filepath  # relative to repo root
            if full_path.exists() and full_path.stat().st_mtime > finding_mtime:
                # File changed — re-activate this question
                _reactivate_question(qid)
                reactivated.append(qid)
                break

    return reactivated
```

`_extract_fix_spec_files()` parses the `## Fix Specification` section in each finding for file paths (`.py`, `.ts`, etc.).

Call `check_diagnosis_complete_reactivations()` at the start of `run_campaign()`, before the pending loop.

### Alternative re-activation trigger

A simpler alternative: run `git diff --name-only HEAD~1` at campaign start. If any changed file matches files mentioned in a DIAGNOSIS_COMPLETE finding, re-activate that question. This is faster than mtime comparison and works correctly in git-based workflows.

```python
def _get_recently_changed_files() -> set[str]:
    result = subprocess.run(
        ["git", "diff", "--name-only", "HEAD~1"],
        capture_output=True, text=True, cwd=cfg.project_root.parent
    )
    return set(result.stdout.splitlines())
```
