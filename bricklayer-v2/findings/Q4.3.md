# Finding: Q4.3 — simulate.py rename to evaluate.py

**Question**: Should `simulate.py` be renamed `evaluate.py` in the new template, or kept as-is for backward compatibility? What does the new template's evaluate.py stub look like?
**Agent**: research-analyst
**Verdict**: HEALTHY
**Severity**: Info
**Mode**: evolve
**Target**: `autosearch/template/simulate.py`

## Summary

Keep `simulate.py` in the template — do not rename. The name accurately describes its function (Monte Carlo / parametric simulation), and renaming would break all existing projects (recall/, adbp/, bricklayer-meta/) which reference `simulate.py` in their program.md files and Claude invocations. However, add an optional `evaluate.py` stub alongside `simulate.py` for modes that need evidence-based evaluation rather than parametric simulation. These are two distinct tools serving different modes.

## Evidence

### Why NOT rename simulate.py

1. **All active projects reference it**: `recall/program.md`, `adbp/program.md`, `bricklayer-meta/program.md` all reference `simulate.py`. Renaming the template breaks the convention these projects established.

2. **Name is accurate for its purpose**: `simulate.py` runs parametric scenarios through a model to find failure boundaries. That's simulation. The name is not misleading.

3. **BL 2.0 Benchmark mode needs simulation**: The Benchmark runner calls `simulate.py` to establish parametric baselines. If renamed, the Benchmark mode program (`modes/benchmark.md`) references would need updating.

4. **Backward compatibility**: Any future project template copies should include the same `simulate.py` convention to remain compatible with existing documentation and Claude invocations.

### Why ADD evaluate.py (optional)

Some BL 2.0 modes don't need simulation — they need evidence collection and structured assessment:
- **Research mode**: "Does this assumption hold?" — needs evidence gathering, not simulation
- **Audit mode**: "Is this system compliant?" — needs a checklist runner, not simulation
- **Diagnose mode**: "What is broken?" — needs code analysis and test execution, not simulation

For these modes, a minimal `evaluate.py` stub provides a structured runner that collects evidence from the real system (API calls, file reads, test runs) rather than parametric model runs.
