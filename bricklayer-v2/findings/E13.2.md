**Verdict**: IMPROVEMENT

## Investigation

### Root cause of E12.1-live-14 stochasticity

The original record asked the agent to compare `bricklayer-v2/findings/synthesis.md` against two separate finding files (E11.1.md and E11.2.md). This multi-file comparison requires the agent to:
1. Read a 400+ line synthesis.md
2. Navigate to a specific table section ("Updated Cumulative Agent Eval Scores")
3. Read two additional finding files
4. Cross-reference three documents

When the task complexity is high, the agent sometimes produces a prose answer rather than following the JSON format in `_LIVE_PREAMBLE`. Prose responses score ~0.40 regardless of verdict correctness (evidence quality partial credit, verdict match = 0).

The record scored 90 when the agent produced JSON (correct HEALTHY verdict), ~0.40 when it produced prose. This stochasticity is not a verdict problem — it's a format compliance problem triggered by complexity.

### Replacement record design

Replaced E12.1-live-14 with `E13.2-live-replacement`:

**Question**: "How many training records exist in `masonry/training_data/scored_all.jsonl` for the `research-analyst` agent? Count them and check if the count meets the minimum 35-record threshold. Return HEALTHY if count >= 35, WARNING if 20-34, FAILURE if fewer than 20."

**Why this is cleaner**:
- Single file to read (scored_all.jsonl)
- Numeric count — unambiguous answer (53 records)
- Clear verdict criteria embedded in the question
- No cross-file comparison required
- Expected verdict: HEALTHY (53 >= 35)

### Does it meet ≥0.90 consistently?

**Partially**: 2 of 3 runs exceed 0.90 (0.95 and 0.96). Run 2 (seed 99) scored 0.75 due to evidence quality variation (the agent produced a shorter evidence string in that run), not verdict misclassification.

The key improvement over E12.1-live-14: **zero prose production** across 3 runs. The original scored ~0.40 in some runs because the verdict field was blank (agent wrote a prose paragraph). The replacement record always produces valid JSON with correct HEALTHY verdict.

The 0.75 run reflects evidence quality scoring (partial evidence vs. comprehensive), not stochasticity of format compliance.

## Residual risk

The 0.75 score in run 2 is evidence quality variation: the agent counted correctly but cited less evidence. The `build_metric()` scoring penalizes brief evidence strings. This is acceptable stochasticity (format correct, verdict correct) vs. the previous failure mode (format incorrect, verdict blank).

If ≥0.90 on all 3 runs is required, a further tightening of the question (e.g., "return the exact count") could improve evidence completeness. Current replacement is a significant improvement from the ~0.40 floor.
