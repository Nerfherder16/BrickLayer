# Finding: Q5.2 — Visual BrickLayer dashboard single-screen view

**Question**: What would a visual BrickLayer dashboard look like that shows all projects, their current mode, open findings, and the lifecycle stage progress? What is the most useful single-screen view?
**Agent**: quantitative-analyst
**Verdict**: PROMISING
**Severity**: Info
**Mode**: frontier
**Target**: BrickLayer 2.0 dashboard concept — not yet implemented

## Summary

The most useful single-screen view is a **Project Command Center**: a left sidebar of projects (6-8 rows) with the current mode as a colored badge, and a main panel split into three zones — lifecycle stage progress (top), open findings by severity (middle), and a recent activity feed (bottom). The key insight: the dashboard's primary job is answering "what needs my attention right now?" — not showing history. This maps directly to the existing `dashboard/` codebase and could be implemented as a new `/bricklayer` route in the existing FastAPI + React stack.

### Why this layout, not others

**What was rejected:**

| Alternative | Problem |
|-------------|---------|
| Grid of project cards | Wastes space on metadata; can't see findings without clicking |
| Gantt-style timeline | Projects don't have fixed durations; modes aren't sequential |
| Kanban by mode | Projects with mixed modes confuse the columns; wrong mental model |
| Per-project lifecycle bar (horizontal) | Fine for one project, illegible at 6+ projects |

**What makes the left sidebar + detail panel work:**

1. The project list is compact — 6-8 rows, mode badge visible at a glance
2. The detail panel prioritizes FAILURE/WARNING findings by default — the most urgent items surface immediately
3. The lifecycle progress bar is the orientation layer: it answers "where are we in this project's life?"
4. The activity feed answers "what happened since I last looked?" — critical for long-running campaigns

### Information hierarchy (what the dashboard must answer, in order)

```
Priority 1: What's on fire?
  → FAILURE findings, IMMINENT predictions, BLOCKED projects

Priority 2: What should I do next?
  → Current mode + next pending question ID + estimated effort

Priority 3: Where are we?
  → Lifecycle stage, wave number, questions answered/pending

Priority 4: What changed?
  → Activity feed, last 24h
```

### Mode badges (color system)

| Mode | Badge Color | Status Meaning |
|------|------------|----------------|
| FRONTIER | violet | Ideation phase |
| RESEARCH | cyan | Assumption testing |
| VALIDATE | blue | Design review |
| BENCHMARK | amber | Baselining |
| DIAGNOSE | orange | Investigating failure |
| FIX | red | Active remediation |
| AUDIT | indigo | Compliance check |
| EVOLVE | emerald | Performance improvement |
| PREDICT | rose | Failure forecasting |
| MONITOR | gray | Steady-state watch |
| BLOCKED | dark red | Needs external action |

### Lifecycle progress bar semantics

```
Frontier → Research → Validate → [Benchmark] → Diagnose → Fix → [Audit] → Evolve → Predict → Monitor
   ✓           ✓          ✓           –              ◉          ○          –          ○         ○         ○
```

- `✓` = mode completed (has a HEALTHY/FIXED/COMPLIANT/CALIBRATED finding)
- `◉` = current active mode
- `○` = not yet reached
- `–` = not applicable (skipped or N/A)
- `⚠` = mode completed with WARNING (passed but degraded)

Modes in brackets (Benchmark, Audit) are optional — shown dimmed if not used.

### Finding summary panel — sort order

```
1. FAILURE (bright red)
2. IMMINENT (rose, blinking dot if < 3 days)
3. WARNING + PROBABLE (orange)
4. PENDING_EXTERNAL with approaching deadline (amber)
5. WARNING + POSSIBLE (amber, dimmed)
6. DIAGNOSIS_COMPLETE awaiting Fix (blue)
7. HEALTHY / OK / COMPLIANT / CALIBRATED (gray, collapsed by default)
```

HEALTHY findings are collapsed by default — they're resolved, not actionable.

### What NOT to show on the main screen

- Individual question text (too long — link to finding detail)
- Full `simulate.py` / `evaluate.py` output (belongs in finding detail)
- Git diff view (belongs in fix workflow, not status dashboard)
- All historical verdicts (synthesis.md covers this — link out)

## Verdict Rationale

PROMISING: The layout is derivable from existing data files without new storage, and the backend endpoints are additive to the existing `dashboard/` codebase (3-4 new routes, 1 new React page). The information hierarchy (fire → action → orientation → history) is the right prioritization for a research tool. The mode badge color system and lifecycle progress bar are the two novel UI concepts; both are straightforward to implement with existing Tailwind patterns.
