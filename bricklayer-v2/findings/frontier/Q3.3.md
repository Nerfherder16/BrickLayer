### F1.1 — What is the most valuable product Tim could build that uses Recall as its core differentiator?

**Idea**: **Recall-as-a-Service (RaaS)** — License the Recall memory stack to other Claude Code users who want cross-session memory without running their own infrastructure.

**Core mechanism**:
- Tim's existing Recall stack (FastAPI + Qdrant + Neo4j + Redis + Ollama) already works
- The only missing piece: multi-tenancy (user isolation, billing, onboarding)
- Distribution: Claude Code hooks (already the delivery mechanism — just pointed at Tim's server instead of localhost)
- Revenue model: $10-20/month/user; 1,000 users = $10-20K MRR

**Prerequisites**:
- Auth layer (per-user Qdrant collections or namespace isolation)
- Billing integration (Stripe)
- HTTPS endpoint (currently LAN-only)
- ~2-4 weeks of work on existing codebase

**Failure modes**:
- Anthropic builds this natively (non-zero probability given their memory product trajectory)
- Recall requires RTX 3090 for embeddings — per-user cost is high for <10K users

**Verdict**: PROMISING — mechanism is clear, Tim has the infrastructure, single-user→multi-user is a known engineering path, and there's no equivalent self-hosted alternative available today.

### F1.2 — What would a homelab management product look like that Tim couldn't find on the market?

**Idea**: **Homelab Copilot** — Natural-language interface to CasaOS/Proxmox/Docker that understands context ("why is plex slow?" → checks container resources, runs diagnostics, proposes fix).

**Core mechanism**:
- Proxmox MCP + CasaOS API already exist (Tim has them configured)
- Missing: a product that persists homelab context (which containers fail, when, why) and answers contextual questions
- This is Recall applied to infrastructure: instead of Claude Code sessions, store homelab events
- Users: other self-hosters who are comfortable with Claude but not with terminal

**Prerequisites**:
- Homelab-specific memory schema (container health events, config changes, incident history)
- Dashboard (Tim already builds these)
- Dockerized deployment (runs on any homelab)

**Failure modes**:
- Niche market (self-hosters who also use Claude Code is a small overlap)
- Proxmox/CasaOS API surface is large and inconsistent — high maintenance burden

**Verdict**: PROMISING — solves a real gap (Tim actively struggles with this himself), infrastructure is ready, but market is narrow. Seed question for Research: "What is the realistic addressable market for AI-assisted homelab management?"

### F1.3 — What prerequisite gap currently prevents Tim from shipping anything?

**Idea analysis**: The question is about constraint removal, not a new product. What constraint, if removed, unlocks the most?

**Identified prerequisites**:
1. **Multi-tenancy** — Every project Tim builds is single-user. Adding user isolation is the unlock for RaaS, Homelab Copilot, or FamilyHub licensing.
2. **Public endpoint** — All projects are LAN-only. Without HTTPS+DNS, nothing is shippable.
3. **Payment processing** — No Stripe integration in any project. Required for any subscription product.

**The minimum viable unlock**: Cloudflare Tunnel (free, already in homelab ecosystem) + Stripe checkout. These two additions make any existing project externally accessible and monetizable without changing the core product.

**Verdict**: BLOCKED — not a product idea, but a constraint analysis. Blocked on: (a) exposing a public endpoint safely and (b) Stripe integration. Once unblocked → F1.1 (RaaS) becomes immediately executable.
