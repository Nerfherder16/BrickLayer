# Finding: Q1.2 — questions.md field naming: `mode:` conflict and rename strategy

**Question**: How should `questions.md` represent the new `mode:` field (operational mode) without breaking the existing `mode:` field (which currently means runner type)? What rename is required?
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: Medium
**Mode**: diagnose
**Target**: `bl/questions.py`, `questions.md` format

## Summary

The `mode:` field in BL 1.x questions is overloaded: it means runner type (agent/http/subprocess/correctness) AND appears in the header tag `[agent]`. For BL 2.0, the operational mode (diagnose/frontier/research/etc.) is a different concept. The minimal resolution: keep `[TAG]` header as runner type, add `**Operational Mode**: diagnose` as a new field in the question body, and update `parse_questions()` to extract it. No rename of existing `mode` key required — the question dict gains a new `operational_mode` key.

## Evidence

### Current format analysis

Questions.md header pattern: `## Q1.1 [agent] Question title`

The `[agent]` tag in brackets is parsed at `questions.py:24-26`:
```python
block_pattern = re.compile(r"^## (Q[\w.-]+) \[(\w+)\] (.+?)$", ...)
tag_raw = m.group(2)
mode_raw = tag_raw.lower()  # becomes question["mode"]
```

Inside the block, `**Mode**: agent` is parsed by `field_pattern` but stored in `fields["mode"]` — however this is ignored in favor of `mode_raw` from the header. The `**Mode**:` body field is currently dead.

### The conflict

BL 2.0 needs questions to carry two distinct concepts:
1. **Runner type** (how to execute): `agent | http | subprocess | correctness | performance`
2. **Operational mode** (what kind of work): `diagnose | frontier | research | validate | benchmark | fix | audit | evolve | predict | monitor`

These are orthogonal: a `research` question can use the `agent` runner; a `monitor` question can use the `http` runner.

### Resolution: no rename, new field

The cleanest path adds a new `**Operational Mode**: {mode}` field to the question body. The header tag continues to represent runner type. The `questions.py` parser gets one new extraction:

```python
# In field_pattern, add "Operational Mode" to the alternation:
field_pattern = re.compile(
    r"^\*\*(Mode|Target|Hypothesis|Test|Verdict threshold|Agent|Finding|Source|Operational Mode)\*\*:\s*(.+?)(?=\n\*\*|\Z)",
    re.MULTILINE | re.DOTALL,
)
# In the dict construction:
"operational_mode": fields.get("operational_mode", fields.get("mode", "")).strip(),
```

Note: `fields.get("operational_mode")` uses key `"operational_mode"` because the field parser lowercases and replaces spaces with underscores (`"Operational Mode"` → `"operational_mode"`).

### Question ID as mode signal

The BL 2.0 question ID format already encodes operational mode:
- `F{wave}.{n}` → frontier
- `R{wave}.{n}` → research
- `V{wave}.{n}` → validate
- `Q{wave}.{n}` → diagnose (existing convention)
- `FX{n}` → fix
- etc.

This means `operational_mode` can also be inferred from the ID prefix if the field is absent — a useful fallback for backward compatibility.

### Backward compatibility

Existing questions without `**Operational Mode**:` produce `operational_mode=""`, which causes `_load_mode_context("")` to return `""` (empty string, no injection). All existing campaigns continue to work without modification.

### New question format example

```markdown
## Q1.1 [agent] Does the double-decay fix reduce floor-clamped count?
**Operational Mode**: diagnose
**Target**: decay.py
**Hypothesis**: ...
**Test**: ...
**Status**: PENDING
**Verdict threshold**: ...
```
