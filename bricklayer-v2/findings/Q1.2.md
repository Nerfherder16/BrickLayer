# Q1.2 — `mode:` field naming collision and required rename

**Question**: How should `questions.md` represent the new `mode:` field (operational mode) without breaking the existing `mode:` field (which currently means runner type)? What rename is required?

**Verdict**: DIAGNOSIS_COMPLETE

**Summary**: BL 2.0 has already repurposed `mode:` to mean operational mode in new projects (bl-audit). The old runner-type meaning survives in `questions.py` and `runners/__init__.py`. Fix requires: add `**Runner**` field to question blocks; update `run_question()` to dispatch by `runner` first, `mode` as fallback. 3-line code change + docs update.

---

## Investigation

### 1. What `mode:` currently means (two conflicting meanings)

#### BL 1.x meaning — runner type
In BL 1.x, the bracket tag `## ID [agent] title` and `**Mode**: agent` body field specified the **runner type** — which registered runner function to dispatch to. The `bl/runners/__init__.py:run_question()` uses this value directly:

```python
# bl/runners/__init__.py:276
mode = question["mode"]
runner = get(mode)  # looks up runner by mode string
```

Registered runner modes: `agent`, `subprocess`, `http`, `correctness`, `performance`, `static`, `browser`, `document`, `baseline_check`, `benchmark`, `contract`, `quality`, `simulate`, `swarm`.

#### BL 2.0 meaning — operational mode
In BL 2.0 (bl-audit project), the bracket tag `## D1.1 [audit] title` and `**Mode**: audit` body field now specify the **operational mode** — which lifecycle stage the question belongs to. This is verified in `bl-audit/questions.md`:

```markdown
## D1.1 [audit] Are masonry-agent-onboard.js and masonry-recall-check.js dead code?

**Status**: DONE
**Mode**: audit
**Agent**: compliance-auditor
```

"audit" is NOT a registered runner. When `run_question()` tries `get("audit")`, it returns None and the result is `INCONCLUSIVE: Unknown mode 'audit'`.

### 2. The collision point

`bl/runners/__init__.py:run_question()` dispatches exclusively by `question["mode"]`:

```python
def run_question(question: dict) -> dict:
    mode = question["mode"]       # ← uses mode for runner lookup
    runner = get(mode)             # ← "audit", "diagnose", "research" have no runner
    if runner is not None:
        result = runner(question)
    else:
        result = {
            "verdict": "INCONCLUSIVE",
            "summary": f"Unknown mode '{mode}' — no runner registered",
            ...
        }
```

`bl/questions.py` already has `operational_mode` as a separate field (defaults to "diagnose"):
```python
"operational_mode": fields.get("operational_mode", "diagnose"),
```
But there is **no `runner` field** parsed from question bodies.

### 3. The required rename

**`**Mode**` → `**Runner**` for runner specification.** In BL 2.0 question blocks:

| Field | Meaning | Example |
|-------|---------|---------|
| Bracket tag `[audit]` | Operational mode | `[diagnose]`, `[audit]`, `[research]` |
| `**Mode**: audit` | Operational mode (body override) | Already used this way in bl-audit |
| `**Runner**: agent` | **NEW** — runner type for dispatch | `agent`, `subprocess`, `http` |

Old BL 1.x questions (without `**Runner**` field) fall back to `question["mode"]` for dispatch — full backward compatibility.

### 4. Fix specification

**File 1: `bl/questions.py`** — add `runner` to the field_pattern and the returned dict:

```python
# In field_pattern regex — add "Runner" to the capture group:
r"^\*\*(Mode|Runner|Target|Hypothesis|Test|Verdict threshold|Agent|Finding|Source|Operational Mode|Resume After)\*\*:\s*(.+?)(?=\n\*\*|\Z)",

# In questions.append() dict:
"runner": fields.get("runner", "").strip(),   # NEW — runner type; empty = use mode as fallback
```

**File 2: `bl/runners/__init__.py:run_question()`** — dispatch by `runner` first, `mode` as fallback:

```python
def run_question(question: dict) -> dict:
    runner_key = question.get("runner") or question["mode"]   # backward compat
    mode = question["mode"]   # kept for logging/output only
    runner = get(runner_key)
    ...
    result["mode"] = mode       # output still shows operational mode
    result["runner"] = runner_key  # NEW — shows which runner was used
    return result
```

**Schema documentation** (`project-brief.md` / `template/` readme) — update question block template:

```markdown
## {ID} [{operational_mode}] {title}

**Status**: PENDING
**Mode**: {operational_mode}   ← diagnose | fix | research | audit | etc.
**Runner**: agent              ← agent | subprocess | http | correctness | etc.
**Priority**: HIGH
**Hypothesis**: ...
**Agent**: {agent-name}
**Success criterion**: ...
```

### 5. Backward compatibility analysis

| Project | Bracket format | `**Mode**` field | `**Runner**` field | Dispatch result after fix |
|---------|---------------|-------------------|--------------------|--------------------------|
| ADBP (BL 1.x) | `[agent]` | `agent` | absent | `runner_key = "agent"` → dispatches to agent runner |
| bl-audit (BL 2.0) | `[audit]` | `audit` | absent | `runner_key = "audit"` → no runner, INCONCLUSIVE |
| bl-audit with fix | `[audit]` | `audit` | `agent` | `runner_key = "agent"` → dispatches to agent runner |
| Any future BL 2.0 | `[diagnose]` | `diagnose` | `agent` | `runner_key = "agent"` → dispatches correctly |

The bl-audit project needs `**Runner**: agent` added to existing question blocks to restore correct dispatch.

### 6. The questions.md table format (this project)

`bricklayer-v2/questions.md` uses a simplified table format for human tracking — not the block format parsed by `bl/questions.py`. The `| Mode |` column in this table refers to the **operational mode** (diagnose, validate, research, etc.). This table is never fed to `run_question()` — it exists only for manual/parallel worker campaign management via `claim.py`. No collision here.
