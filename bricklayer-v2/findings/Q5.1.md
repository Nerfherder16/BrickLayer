# Finding: Q5.1 — Multi-agent BrickLayer: parallel modes on the same project

**Question**: If BrickLayer 2.0 had a multi-agent variant — where multiple modes run in parallel on the same project — what would the coordination mechanism look like? What prevents mode conflicts?
**Agent**: quantitative-analyst
**Verdict**: PROMISING
**Severity**: Info
**Mode**: frontier
**Target**: BrickLayer 3.0 concept — not yet implemented

## Summary

A multi-agent BrickLayer where Diagnose + Monitor + Predict run in parallel on the same project is architecturally feasible today. The coordination mechanism is simple: modes are read-only against shared project state (`findings/`, `synthesis.md`, `questions.md`) and write to separate namespaced output files. Mode conflicts are prevented by the single write rule: only one mode writes to `findings/{ID}.md` at a time, enforced by question ID namespacing (`Q` for Diagnose, `M` for Monitor, `P` for Predict). The `results.tsv` append pattern is safe for parallel writers. The primary open question is the `synthesis.md` merge — two modes updating it simultaneously creates a race condition.

## Evidence

### Why parallel modes are architecturally feasible

The existing BrickLayer file structure is almost naturally parallel-safe:

| File | Parallel-safe? | Mechanism |
|------|---------------|-----------|
| `findings/{ID}.md` | ✓ | Each question has a unique ID; two modes never write the same ID |
| `results.tsv` | Mostly ✓ | Append-only; race condition on same-ID upsert (different modes don't share IDs) |
| `questions.md` | ✗ | Both modes parse and mark status; concurrent writes corrupt the file |
| `synthesis.md` | ✗ | Both modes update it at session end; needs merge resolution |
| `monitor-log.tsv` | ✓ | Monitor only; append-only |
| `ideas.md` | ✓ | Frontier only |

Two of seven shared files are not parallel-safe. This is solvable.

### Coordination mechanism

**Option 1: Naive parallel (good enough for 80% of cases)**

Run modes as separate `claude` processes:
```bash
# Terminal 1
DISABLE_OMC=1 claude "Read modes/diagnose.md and questions.md. Run Diagnose campaign."
# Terminal 2
DISABLE_OMC=1 claude "Read modes/monitor.md and monitor-targets.md. Run Monitor pass."
```

This works because Diagnose writes to `findings/Q*.md` and Monitor writes to `monitor-log.tsv` — they never touch the same files. The conflict only arises at session end when both try to update `synthesis.md`.

**Option 2: Lock file coordination**

A simple `{project}/.mode-lock` file prevents two modes from updating `synthesis.md` simultaneously:
```
.mode-lock contents: {mode: "diagnose", pid: 12345, started: "2026-03-16T14:00:00Z"}
```

Before updating synthesis.md:
1. Check if `.mode-lock` exists
2. If yes, wait (or write to `findings/synthesis-{mode}-patch.md` for human merge)
3. If no, acquire lock, update synthesis.md, release lock

**Option 3: Mode-namespaced synthesis**

Instead of one `synthesis.md`, each mode maintains `synthesis-{mode}.md`:
- `synthesis-diagnose.md` — accumulated Diagnose findings
- `synthesis-monitor.md` — Monitor trend analysis
- `synthesis-predict.md` — Failure cascade maps

A `synthesis.md` master is generated on-demand by merging the mode syntheses. This is cleanest architecturally.

### What prevents mode conflicts

The key insight: **mode conflicts only occur when two modes make mutually exclusive claims**. In practice:
- Diagnose and Monitor are compatible (different question types, different files)
- Diagnose and Fix are NOT parallelizable (Fix consumes DIAGNOSIS_COMPLETE — if Diagnose is still running, Fix's input may not be final)
- Frontier and Research are compatible (different question IDs, different output files)
- Monitor and Predict are compatible (Monitor feeds Predict via ALERT findings)

**Safe parallel pairs**: (Diagnose + Monitor), (Frontier + Research), (Monitor + Predict)
**Unsafe parallel pairs**: (Diagnose + Fix), (any two modes writing the same question IDs)

### Conflict prevention rules for multi-agent BrickLayer

```
MULTI_AGENT_RULES:
1. Question IDs must be mode-namespaced (Q, F, R, V, B, FX, A, E, P, M — no overlap)
2. synthesis.md update is serialized (lock or mode-namespaced files)
3. questions.md status updates use file locking (or optimistic retry)
4. Fix mode cannot run parallel to Diagnose on the same project
5. Each parallel mode needs a dedicated terminal/process — no shared state in memory
```

### Implementation estimate

- Phase 1 (naive parallel, 2 modes): no code changes needed — just run two claude processes
- Phase 2 (lock file coordination): ~50 lines in `campaign.py`
- Phase 3 (mode-namespaced synthesis): template change + synthesizer.py update, ~100 lines

## Verdict Rationale

PROMISING: The architecture is sound and the coordination mechanism is simple. The question ID namespacing (which BL 2.0 already establishes) is the primary conflict prevention mechanism. The `synthesis.md` race condition is the only non-trivial problem, and three solutions exist with varying complexity. Research mode question to seed: "What is the simplest coordination mechanism that handles the synthesis.md merge without human intervention?"
