# Finding: Q2.3 — Monitor vs Diagnose operational distinction

**Question**: Does the Monitor mode program (`modes/monitor.md`) correctly distinguish itself from a scheduled Diagnose run? What is the exact operational difference?
**Verdict**: HEALTHY
**Severity**: Info
**Mode**: validate
**Target**: `bricklayer-v2/modes/monitor.md`

## Summary

Monitor mode is correctly and clearly distinguished from Diagnose mode. The distinction is structural, not operational: Monitor watches **known metrics against known thresholds** (what we already watch), while Diagnose investigates **unknown failures** (what we don't yet know). The program correctly states "Monitor never replaces Diagnose — it can't find NEW unknown failures." One gap: the `monitor-targets.md` format is well-defined but the program doesn't specify what happens when Monitor finds a metric *degrading toward* a threshold (early warning before ALERT).

## Evidence

### The fundamental distinction — correctly captured

| Dimension | Monitor | Diagnose |
|-----------|---------|----------|
| Question type | "Is metric X within threshold Y?" | "Why is the system broken?" |
| Input | monitor-targets.md (known metrics) | Any observable symptom |
| Output | OK/DEGRADED/ALERT per metric | Root cause at code level |
| Hypothesis generator | None — questions come from targets | Wave-by-wave generation |
| Findings written | Only on ALERT | Every question |
| Cadence | Scheduled (cron) | Campaign (human-initiated) |
| Terminal condition | None — runs forever | DIAGNOSIS_COMPLETE |

This is a clean, non-overlapping partition. A scheduled Diagnose would still generate hypotheses, still write findings for every question, and still have a campaign loop structure. Monitor has none of these — it's pure measurement and alerting.

### Relationship to Diagnose — correctly modeled

"When Monitor fires an ALERT, the alert finding seeds a new Diagnose question" — this is the correct architecture. Monitor is Diagnose's early-warning sensor network. The alert finding becomes a new PENDING question in the project's questions.md, and the next Diagnose session picks it up.

### Gap: DEGRADED trajectory tracking

Current Monitor outputs:
- `OK` — within expected range
- `DEGRADED` — crossed WARNING threshold
- `ALERT` — crossed FAILURE threshold

But the program doesn't specify what to do with a metric that is:
- Currently OK, but trending toward WARNING at a measurable rate

Example: floor-clamped count at 750 (OK, below 800 WARNING threshold), but growing at +40/day → will cross threshold in 1.25 days.

This is a meaningful signal that should be captured. Suggested addition to Monitor program:

```
### Trend analysis (optional but recommended)

For each metric with a numeric value, if prior runs in monitor-log.tsv exist:
- Compute delta from last run and rate of change
- If rate of change would cross WARNING threshold within 3 runs — mark DEGRADED_TRENDING
  and include in the summary even if current value is OK
```

This prevents Monitor from looking healthy on the day before a threshold crossing.

### Alert format — complete

The alert format block is well-specified:
```
🚨 ALERT: {metric_name}
Current: {value}
Threshold: {failure_threshold}
Baseline: {benchmark_value} ({delta:+.1f}%)
Finding reference: {original_finding_id if applicable}
```

The `Finding reference` field is excellent — it ties Monitor alerts back to the Diagnose findings that established the threshold. This creates the full lifecycle loop.

### monitor-log.tsv vs findings/ separation — correct

Not writing to findings/ for OK verdicts is the right call. Monitor runs could generate hundreds of OK entries per day, flooding the findings directory. The findings/ directory is for actionable knowledge; monitor-log.tsv is for operational telemetry.

## Verdict Rationale

HEALTHY: Monitor is correctly scoped and correctly distinguished from Diagnose. The program is complete for its purpose. The missing trend-analysis extension is a useful enhancement but not a gap in the core loop.
