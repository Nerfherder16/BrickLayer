# Finding: D3.2 — _reactivate_pending_external() location mismatch is a spec/doc gap, not a functional bug

**Question**: Does the `_reactivate_pending_external()` location mismatch (campaign.py vs spec's questions.py) cause a functional race where the dual `resume_after` UTC evaluations produce different results?
**Verdict**: HEALTHY
**Severity**: Info
**Mode**: diagnose
**Motivated by**: D4 FAILURE (location mismatch)

## Evidence

`run_campaign()` in `bl/campaign.py` lines 544-549:
```python
questions = parse_questions()
_reactivate_pending_external(questions)   # drops PENDING_EXTERNAL rows from results.tsv
questions = parse_questions()             # re-parses — status now read from results.tsv
pending = [q for q in questions if q["status"] == "PENDING"]
```

`parse_questions()` in `bl/questions.py` line 60:
```python
status = get_question_status(qid)  # reads from results.tsv, NOT questions.md
```

**Flow after reactivation**:
1. `_reactivate_pending_external()` drops the PENDING_EXTERNAL row from results.tsv
2. The second `parse_questions()` calls `get_question_status(qid)` → no row found → returns "PENDING"
3. The question correctly appears in the `pending` list with `status = "PENDING"`

**`get_next_pending()` resume_after check**: `get_next_pending()` also evaluates `resume_after`, but it's only called inside `run_campaign()`'s loop when refreshing the pending list (line 564-565). By that point, `_reactivate_pending_external()` has already run and the question is visible as "PENDING". The `resume_after` in `get_next_pending()` serves as a secondary gate for questions that enter PENDING status via other paths (not PENDING_EXTERNAL reactivation).

## Conclusion

The location mismatch is a spec/doc issue only. The actual implementation in campaign.py works correctly: the re-parse at line 548 correctly picks up the reactivated question as "PENDING" because `parse_questions()` reads status from results.tsv (not questions.md). No functional race exists.

The spec should be updated to reflect that `_reactivate_pending_external()` lives in campaign.py, not questions.py.
