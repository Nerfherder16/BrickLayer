# Finding: D16.1.F1 — HEAL_EXHAUSTED absent from _TERMINAL_VERDICTS but re-triggering does not occur

**Question**: Does the absence of `HEAL_EXHAUSTED` from `_TERMINAL_VERDICTS` in `questions.py` and `_PARKED_STATUSES` cause exhausted heal loops to re-trigger indefinitely on the next campaign run?
**Verdict**: HEALTHY
**Severity**: Low

## Evidence

**Re-triggering concern (HEALTHY)**:

The hypothesis assumes HEAL_EXHAUSTED absence causes the question to be re-queued as PENDING on the next campaign run. Tracing the actual flow:

1. `run_and_record()` line 87 calls `update_results_tsv(qid, "FAILURE", ...)` **before** the heal loop starts. This calls `_mark_question_done(qid, "FAILURE")` which writes `**Status**: FAILURE` to questions.md. FAILURE is in `_PRESERVE_AS_IS` (findings.py line 511).

2. The heal loop then runs and exhausts without writing HEAL_EXHAUSTED (the D16.1 bug). The question's status in questions.md remains FAILURE.

3. On the next campaign run: `pending = [q for q in questions if q["status"] == "PENDING"]` — FAILURE ≠ PENDING → question NOT added to pending list → heal loop NOT re-triggered.

**The re-triggering concern is not realized.** The FAILURE status in questions.md prevents re-queueing even without HEAL_EXHAUSTED being added to `_PARKED_STATUSES`.

**Secondary gap (Low severity)**:

After D16.1 is fixed and `update_results_tsv(original_qid, "HEAL_EXHAUSTED", ...)` is added to healloop.py:

- results.tsv gets two rows for original_qid: FAILURE (written first in run_and_record) then HEAL_EXHAUSTED (written after loop exhaustion)
- `sync_status_from_results()` reads FAILURE row → updates status to FAILURE in done_ids; reads HEAL_EXHAUSTED row → NOT in `_TERMINAL_VERDICTS` → skipped
- questions.md status stays FAILURE — correct behavior for campaign re-queueing, but the human-visible status doesn't reflect HEAL_EXHAUSTED

**Frozenset audit**:
- `_PARKED_STATUSES` (questions.py lines 99-112): HEAL_EXHAUSTED absent — but this is used by `get_next_pending()`, not `run_campaign()`. `run_campaign()` filters by `q["status"] == "PENDING"` directly.
- `_TERMINAL_VERDICTS` (questions.py lines 161-193): HEAL_EXHAUSTED absent — causes `sync_status_from_results()` to ignore HEAL_EXHAUSTED rows, leaving status at FAILURE.
- `_PRESERVE_AS_IS` (findings.py lines 502-516): HEAL_EXHAUSTED absent — after D16.1 fix, `_mark_question_done(qid, "HEAL_EXHAUSTED")` would map to "DONE" (not HEAL_EXHAUSTED). This IS a minor signal loss.
- `agent_db.py` frozensets: HEAL_EXHAUSTED absent from all three (`_SUCCESS_VERDICTS`, `_PARTIAL_VERDICTS`, `_FAILURE_VERDICTS`) — would be scored as 0.0 (unrecognized, defaults to _FAILURE_VERDICTS behavior in scoring).

## Mitigation Recommendation

The re-triggering concern is not a real risk given the current campaign flow. However, once D16.1 is fixed, add `HEAL_EXHAUSTED` to:
1. `_TERMINAL_VERDICTS` in `questions.py` — so `sync_status_from_results()` updates the status correctly
2. `_PRESERVE_AS_IS` in `findings.py` — so `_mark_question_done()` preserves "HEAL_EXHAUSTED" as the visible status (not "DONE")
3. `agent_db.py _PARTIAL_VERDICTS` — score it as half-credit (agent ran correctly, system exhausted, human needed)
4. `_PARKED_STATUSES` in `questions.py` — defensive completeness so `get_next_pending()` never returns a HEAL_EXHAUSTED question

These are cleanup items for after D16.1 is fixed — not blocking, Low severity.
