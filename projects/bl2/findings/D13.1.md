# Finding: D13.1 — followup.py agent_name default is dead code; empty-string agent silently corrupts sub-question prompts

**Question**: followup.py sub-question prompt uses wrong default agent for BL 2.0 code_audit questions
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: Medium
**Mode**: code_audit
**Operational Mode**: diagnose

## Evidence

### questions.py — parsing of **Agent** field

`parse_questions()` at line 75 of `C:/Users/trg16/Dev/autosearch/bl/questions.py`:

```python
"agent_name": fields.get("agent", "").strip(),
```

The key `agent_name` is **always inserted** into the returned dict, even when the `**Agent**:` field is absent from the question body. When absent, `agent_name` is set to `""` (empty string). The key is never missing from the dict.

The field_pattern at line 27 correctly captures `**Agent**:` when present:

```python
field_pattern = re.compile(
    r"^\*\*(Mode|Target|Hypothesis|Test|Verdict threshold|Agent|Finding|Source|Operational Mode|Resume After)\*\*:\s*(.+?)(?=\n\*\*|\Z)",
    re.MULTILINE | re.DOTALL,
)
```

Confirmed in `C:/Users/trg16/Dev/autosearch/projects/bl2/questions.md`: all BL 2.0 questions do have `**Agent**:` populated (e.g., `**Agent**: diagnose-analyst`, `**Agent**: compliance-auditor`).

### followup.py — agent default fallback

`_build_followup_prompt()` at line 102 of `C:/Users/trg16/Dev/autosearch/bl/followup.py`:

```python
agent = question.get("agent_name", "quantitative-analyst")
```

`dict.get(key, default)` only uses `default` when the key is **absent from the dict**. Since `questions.py` always inserts `agent_name` (even as `""`), the `"quantitative-analyst"` default can never be reached. Verified:

```python
>>> {'agent_name': ''}.get('agent_name', 'quantitative-analyst')
''           # default NOT used — key is present with empty-string value
>>> {}.get('agent_name', 'quantitative-analyst')
'quantitative-analyst'  # default used only when key absent
```

### Effect on generated sub-question prompts

When `agent_name == ""` (question without `**Agent**:` field), line 129 of followup.py produces:

```
**Agent**:
```

This is a blank Agent field in the injected sub-question. The LLM receiving this prompt may propagate a blank `**Agent**:` into the generated block, or hallucinate any agent name, producing sub-questions that the BrickLayer loop cannot route correctly.

For the current BL 2.0 campaign, `**Agent**:` IS populated on all questions (confirmed by grep). So BL 2.0 sub-questions for D*, F*, A*, V*, M* questions correctly inherit the parent's agent (e.g., `diagnose-analyst`). The original hypothesis that BL 2.0 code_audit questions fail is **incorrect for current questions** — they have the field.

### Where the real bug lives

The `"quantitative-analyst"` default at followup.py line 102 is **dead code** — it can never activate due to the always-present key from questions.py. The actual failure mode (blank `**Agent**:` in sub-questions) affects any question added without an `**Agent**:` field, which the code silently tolerates.

## Impact

- **For current BL 2.0 questions**: No runtime impact. All questions have `**Agent**:` populated, so sub-questions inherit the correct agent.
- **For future questions or imported questions without `**Agent**:`**: Sub-questions get `**Agent**: ` (blank). The LLM generating sub-questions sees a blank agent field and may hallucinate any agent name or propagate the blank, causing routing failures.
- **For the fallback intent**: The developer intended `"quantitative-analyst"` as the fallback agent for questions missing the field. This intent is silently defeated — the fallback is unreachable.

## Fix Specification

**Root Cause**: `parse_questions()` in `bl/questions.py` always inserts `agent_name` into the question dict (even as `""`), so `question.get("agent_name", "quantitative-analyst")` in `followup.py` can never fall back to `"quantitative-analyst"`. The correct fix is to use `or` to handle the empty-string case.

**File**: `C:/Users/trg16/Dev/autosearch/bl/followup.py`

**Location**: Line 102

**Change**: Replace

```python
agent = question.get("agent_name", "quantitative-analyst")
```

with:

```python
agent = question.get("agent_name") or "quantitative-analyst"
```

`or` evaluates the right-hand side for any falsy value (`""`, `None`, `False`), correctly activating the fallback when `agent_name` is an empty string.

**Verification**:

```python
# Run from autosearch root
python3 -c "
from bl.questions import parse_questions
from bl.config import cfg
# Simulate a question with no Agent field
q_empty = {'id': 'TEST.1', 'agent_name': ''}
agent = q_empty.get('agent_name') or 'quantitative-analyst'
assert agent == 'quantitative-analyst', f'Expected fallback, got: {agent!r}'
q_present = {'agent_name': 'diagnose-analyst'}
agent2 = q_present.get('agent_name') or 'quantitative-analyst'
assert agent2 == 'diagnose-analyst', f'Expected diagnose-analyst, got: {agent2!r}'
print('PASS: both cases correct')
"
```

**Risk**: Minimal. The change only affects questions where `agent_name` is `""` (currently none in BL 2.0). Questions with a valid agent name are unaffected. No adjacent tests reference this line directly; any test for `_build_followup_prompt` output should be checked to confirm agent field value expectations.
