# Finding: F12.3 — Fixed C-04 adaptive follow-up to trigger on NON_COMPLIANT verdicts

**Question**: Fix A12.1: C-04 follow-up guard in campaign.py and followup.py excludes NON_COMPLIANT
**Verdict**: FIXED
**Severity**: Medium
**Mode**: code_audit
**Source finding**: A12.1 (NON_COMPLIANT)

## Changes Made

### bl/campaign.py — C-04 follow-up trigger (change 1 of 2)

Before:
```python
if result.get("verdict") in ("FAILURE", "WARNING"):
```

After:
```python
if result.get("verdict") in ("FAILURE", "WARNING", "NON_COMPLIANT"):
```

Line 106 — the C-04 branch that invokes `generate_followup()` was gated on only two verdict
strings. NON_COMPLIANT results silently bypassed drill-down entirely.

### bl/followup.py — generate_followup() guard (change 2 of 2)

Before:
```python
if verdict not in ("FAILURE", "WARNING"):
    return []
```

After:
```python
if verdict not in ("FAILURE", "WARNING", "NON_COMPLIANT"):
    return []
```

Line 251 — even if the campaign.py gate were fixed alone, `generate_followup()` contained a
second identical guard that would have short-circuited and returned `[]` for NON_COMPLIANT.
Both layers required the same one-token addition.

## Verification

Trace for a NON_COMPLIANT result:

1. `campaign.py` line 106: `result.get("verdict") in ("FAILURE", "WARNING", "NON_COMPLIANT")`
   → evaluates True → enters C-04 branch → calls `generate_followup(question, result, cfg.questions_md)`
2. `followup.py` line 251: `verdict not in ("FAILURE", "WARNING", "NON_COMPLIANT")`
   → evaluates False → does NOT return early → continues to drill-down logic
3. Sub-questions are generated and appended to `result["followup_questions"]` as expected.

## Backward Compatibility

FAILURE and WARNING verdict paths are unchanged — both strings remain in both tuples.
BL 1.x behavior is fully preserved; the only new behavior is NON_COMPLIANT now entering
the same drill-down path those verdicts already used.
