# Finding: F2.6 — Track actual exit cycle in EXHAUSTED note

**Question**: Fix D5: `_append_heal_note()` EXHAUSTED call used `max_cycles` instead of actual exit cycle, producing misleading audit trail on early `break` exits.
**Verdict**: FIXED
**Severity**: Medium
**Mode**: fix
**Source finding**: D5

## Changes Made

In `bl/healloop.py`, added `last_cycle = 0` before the for loop and `last_cycle = cycle` as the first statement inside the loop. Changed the `_append_heal_note` EXHAUSTED call to use `last_cycle` instead of `max_cycles`.

```python
# Before
_append_heal_note(finding_path, max_cycles, "EXHAUSTED", f"...exhausted {max_cycles} cycle(s)...")

# After
_append_heal_note(finding_path, last_cycle, "EXHAUSTED", f"...exhausted {last_cycle} cycle(s)...")
```

## Verification

When the loop breaks on cycle 1 (e.g., diagnose-analyst.md missing), `last_cycle = 1` and the EXHAUSTED note correctly reads "Heal Cycle 1 — EXHAUSTED". When all cycles run to completion, `last_cycle = max_cycles` and the note is unchanged from prior behavior. The linter-required fix: `last_cycle` is used in both the `_append_heal_note` call and the f-string message, satisfying the "variable used" requirement.
