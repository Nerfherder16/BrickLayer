# Finding: A10.1 — synthesizer.py _build_findings_corpus() truncation drops audit/diagnose findings

**Question**: Does _build_findings_corpus() systematically exclude high-severity findings under budget pressure?
**Verdict**: NON_COMPLIANT
**Severity**: Medium
**Mode**: code_audit
**Operational Mode**: audit

## Audit

### Truncation strategy

```python
# Findings files sorted by name (alphabetical)
for finding_file in sorted(findings_dir.glob("*.md")):
    ...

# Truncate oldest findings first (drop from front of list)
while finding_sections and budget < sum(len(s) for s in finding_sections):
    finding_sections.pop(0)
```

The comment says "oldest findings first" but the sort is **alphabetical**, not chronological. In BL 2.0, alphabetical filename order is:

```
A7.1.md, A8.1.md, A9.1.md, A10.1.md,
D10.1.md, D10.2.md, D7.1.md, D7.2.md, D8.1.md, D8.2.md, D9.1.md, D9.2.md,
F10.1.md, F10.2.md, F7.1.md, F7.2.md, F8.1.md, F8.2.md, F9.1.md, F9.2.md,
V7.1.md, V8.1.md, V9.1.md, V10.1.md,
synthesis.md
```

`pop(0)` drops from the front: A*.md first, then D*.md, then F*.md, then V*.md.

### Budget calculation for current bl2 campaign

- results.tsv: ~70 rows × ~100 chars = ~7000 chars TSV section
- Budget remaining: 12000 − 7000 = 5000 chars
- Average finding size: ~1200 chars
- Total finding chars (25 findings): ~30000 chars
- Findings that must be dropped: 30000 − 5000 = 25000 chars ÷ 1200 = ~21 findings dropped
- Findings remaining: ~4 (the last 4 alphabetically: V9.1.md, V10.1.md, synthesis.md + possibly one F*.md)

**In practice:** The synthesizer sees only validate-mode findings (COMPLIANT verdicts) — all audit (A*.md) and diagnose (D*.md) root cause analyses are excluded. The LLM receives no qualitative context about WHY failures occurred, only the TSV tallies.

### What the LLM misses

| Excluded type | Content | Severity |
|---------------|---------|---------|
| A*.md | Audit findings — COMPLIANT/NON_COMPLIANT with reasoning | High value |
| D*.md | Diagnose findings — root causes, FAILURE details | Highest value |
| F*.md | Fix findings — what was changed | Medium value |

The TSV partially compensates (verdict counts visible) but the LLM cannot explain WHY failures occurred or what was fixed — synthesizing only "there were 9 FAILUREs" without context.

### Severity

This is a quality defect, not a correctness crash. The synthesizer still functions but produces lower-quality synthesis reports under budget pressure. As campaigns grow (Wave 10+), the bias worsens because the TSV grows larger (reducing budget) while findings accumulate.

## Verdict: NON_COMPLIANT

`pop(0)` on an alphabetically sorted list systematically drops audit and diagnose findings before fix and validate findings. Under realistic budget pressure (Wave 10: 25 findings, 7KB TSV), the corpus contains only the last 3-4 validate findings. Root-cause context is lost. Minor fix: sort findings by wave number descending (keep newest), or reserve a minimum budget for FAILURE/NON_COMPLIANT findings.
