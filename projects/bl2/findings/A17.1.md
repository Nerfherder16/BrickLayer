# Finding: A17.1 — parse_questions() called inside per-finding loop in _inject_override_questions() — O(N) parse overhead

**Question**: Does the D16.2.F1 fix call parse_questions() inside the per-finding loop, performing N full parses of questions.md per _inject_override_questions() invocation?
**Verdict**: NON_COMPLIANT
**Severity**: Low

## Evidence

**`bl/campaign.py` lines 405-411** (inside `for finding_file in sorted(cfg.findings_dir.glob("*.md")):`):
```python
qid = finding_file.stem
reexam_marker = f"Re-examine {qid}"
if reexam_marker in questions_text:
    continue

# D16.2.F1: skip code_audit findings — their test fields are prose, not runnable
original_question = get_question_by_id(parse_questions(), qid)  ← called per iteration
```

`parse_questions()` reads and parses the entire `questions.md` file on every call. With 50 finding files in `cfg.findings_dir`, `_inject_override_questions()` performs **50 full parses** of questions.md per invocation.

**Current campaign scale**: ~60 questions, ~50 findings, questions.md ~1800 lines. At this scale, each parse is a full file read + regex-based block splitting. The overhead is measurable but not catastrophic.

**Staleness risk analysis**: `question["mode"]` is immutable — set at question creation time via `write_question_block()` and never modified by any downstream code path (only `status` and `verdict` fields are updated). Hoisting `parse_questions()` above the loop introduces **zero staleness risk**.

## Fix Specification

**Target file**: `bl/campaign.py`
**Target location**: `_inject_override_questions()` — before the `for finding_file in` loop
**Concrete edit**: Hoist `parse_questions()` call above the loop and pass cached result to `get_question_by_id()`:
```python
def _inject_override_questions() -> None:
    ...
    questions_text = cfg.questions_md.read_text(encoding="utf-8")
    all_questions = parse_questions()  # A17.1: hoist above loop — one parse per invocation

    injected = 0
    for finding_file in sorted(cfg.findings_dir.glob("*.md")):
        ...
        # D16.2.F1: skip code_audit findings
        original_question = get_question_by_id(all_questions, qid)  # use cached result
```
**Verification command**: `grep -n "parse_questions" bl/campaign.py` — confirms parse_questions() is called once before the for loop, not inside it.

## Fix Applied

Hoisted `all_questions = parse_questions()` above the `for finding_file in` loop. Changed `get_question_by_id(parse_questions(), qid)` to `get_question_by_id(all_questions, qid)` inside the loop. N full parses per invocation reduced to 1.
