# Finding: F4.4 — Fixed write_finding() Mode header and _synthetic_question() question_type

**Question**: Fix D4.2 and D4.3: write_finding() used wrong field for **Mode** header; _synthetic_question() inherited code_audit question_type causing C-30 confidence cap on synthetic heal findings.
**Verdict**: FIXED
**Severity**: Medium
**Mode**: fix
**Source finding**: D4.2 (FAILURE), D4.3 (FAILURE)

## Changes Made

### Fix 1: bl/findings.py — write_finding() **Mode** header (D4.2)

```python
# Before:
**Mode**: {question["mode"]}

# After:
**Mode**: {question.get("operational_mode", question["mode"])}
```

Synthetic heal findings now show `**Mode**: diagnose` or `**Mode**: fix` (from `operational_mode`). Primary BL 2.0 findings show their actual operational mode. Falls back to `question["mode"]` for BL 1.x compatibility.

### Fix 2: bl/healloop.py — _synthetic_question() question_type (D4.3)

```python
# Before:
q = dict(original_question)  # inherited question_type="code_audit"
...

# After:
q = dict(original_question)
q["question_type"] = "behavioral"  # F4.4: reset — synthetic heal questions are live agent runs, not code_audit
...
```

Synthetic diagnose/fix questions now always have `question_type="behavioral"`. The C-30 confidence cap (code_audit only) no longer applies to synthetic heal findings. Diagnose-analyst results with `confidence="high"` are written faithfully.

## Verification

```
grep -n "operational_mode" bl/findings.py | head -3
→ **Mode**: {question.get("operational_mode", question["mode"])}

grep -n "question_type" bl/healloop.py
→ q["question_type"] = "behavioral"  # F4.4
```

Both changes confirmed applied.

## Backward Compatibility

- `question.get("operational_mode", question["mode"])` falls back for BL 1.x questions with no `operational_mode` — no regression
- `question_type = "behavioral"` only affects synthetic questions; primary campaign questions get `question_type` from `parse_questions()` logic — unaffected
