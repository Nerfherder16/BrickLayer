# Finding: D16.3 — runners/agent.py _parse_text_output() has no fallback for BL 2.0 agents

**Question**: Does `_parse_text_output()` in `bl/runners/agent.py` return `{}` for BL 2.0 agents (diagnose-analyst, fix-implementer, etc.), causing plain-text output to always produce INCONCLUSIVE regardless of the verdict text present?
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: High

## Evidence

**`_parse_text_output()` lines 138-198**: The function initializes `out: dict = {}`, extracts commit hashes (unconditional), then dispatches on `agent_name` with four `if/elif` branches:

```python
if agent_name == "security-hardener":    ...
elif agent_name == "test-writer":        ...
elif agent_name == "type-strictener":    ...
elif agent_name == "perf-optimizer":     ...
# NO else clause
return out
```

For any agent name not in these four — including all BL 2.0 agents (`diagnose-analyst`, `fix-implementer`, `compliance-auditor`, `design-reviewer`, `compliance-auditor`) — the function returns `out` which equals `{}` (unless commit hashes were found in the text, which adds `changes_committed`).

**`_verdict_from_agent_output()` `else` branch (lines 122-129)** handles BL 2.0 agents when the output dict is properly formed. But when called with `{}` from the text-fallback path:

```python
else:
    self_verdict_early = output.get("verdict", "").upper()  # "" — not in _ALL_VERDICTS
    if self_verdict_early in _ALL_VERDICTS:
        return self_verdict_early   # skipped
    if output.get("changes_committed", 0) > 0:
        return "HEALTHY"            # skipped (no commits in plain-text fallback)
```

Falls through to:
```python
self_verdict = output.get("verdict", "").upper()  # "" again
if self_verdict in _ALL_VERDICTS:
    return self_verdict   # skipped
return "INCONCLUSIVE"     # ← always reached for BL 2.0 agents via text-fallback
```

**Concrete failure scenario**: A `diagnose-analyst` agent produces plain text containing:
```
verdict: DIAGNOSIS_COMPLETE
summary: update_results_tsv not called on exhausted path
```
JSON block extraction fails (no `{...}` block). `_parse_text_output("diagnose-analyst", text)` returns `{}`. `_verdict_from_agent_output("diagnose-analyst", {})` returns `"INCONCLUSIVE"`. The DIAGNOSIS_COMPLETE verdict in the plain text is silently discarded.

**Scope**: This is the text-fallback path only — triggered when the JSON block extraction at `run_agent()` lines 391-396 fails. The primary path (JSON block present) is unaffected. However, any transient formatting failure by a BL 2.0 agent under any condition — prompt compliance failure, claude CLI version change, context window truncation — will silently downgrade a correct DIAGNOSIS_COMPLETE to INCONCLUSIVE.

## Fix Specification

**Target file**: `bl/runners/agent.py`
**Target location**: Lines 188-198 — after the `elif agent_name == "perf-optimizer":` block and before `return out`
**Concrete edit**: Add an `else` clause that extracts `verdict` and `summary` from plain text:
```python
else:
    # BL 2.0 universal fallback: extract verdict and summary from plain text
    m = re.search(r"^verdict:\s*(\w+)", text, re.IGNORECASE | re.MULTILINE)
    if m:
        out["verdict"] = m.group(1).upper()
    m = re.search(r"^summary:\s*(.+)", text, re.IGNORECASE | re.MULTILINE)
    if m:
        out["summary"] = m.group(1).strip()
```
**Verification command**: `grep -n "else:" bl/runners/agent.py | grep -A3 "perf-optimizer"` — confirm an `else` clause follows the `perf-optimizer` branch and extracts `verdict` and `summary` from plain text.

## Mitigation Recommendation

1. Add the `else` clause to `_parse_text_output()` to extract `verdict` and `summary` from plain text using regex on `^verdict:` and `^summary:` lines.
2. All BL 2.0 agents that produce structured output should include `verdict:` on its own line in plain-text responses as a fallback signal (already common practice in their prompts).
3. Consider adding a log warning when the text-fallback path is triggered for a BL 2.0 agent, so transient formatting failures are visible in campaign output.
