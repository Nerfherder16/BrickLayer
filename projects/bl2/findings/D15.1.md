# Finding: D15.1 — questions.py sync_status_from_results() uses "\n## Q" sentinel — misses BL 2.0 block boundaries

**Question**: Does sync_status_from_results() correctly find block boundaries for BL 2.0 questions?
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: Medium
**Mode**: code_audit
**Operational Mode**: diagnose

## Root Cause

**`questions.py` line 225**:
```python
next_block = text.find("\n## Q", block_start + 1)
```

The sentinel `"\n## Q"` is a BL 1.x artifact. BL 2.0 blocks use `D`, `F`, `A`, `V`, `M` prefixes — none of which match `"\n## Q"`. For every BL 2.0 question, `next_block` returns `-1` and `block_end` falls back to `len(text)` — the block slice spans the entire remainder of the file.

## Why It Doesn't Actively Corrupt Today

Two guards contain the damage:
1. `replace("**Status**: PENDING", ..., 1)` — the `1` argument replaces only the first occurrence, which is always the correct field since `block_start` is anchored to the target question's header line
2. `"**Status**: PENDING" in block` check — already-updated questions are skipped entirely

## Latent Failure Path

If any question's hypothesis, test, or body text contains the literal string `**Status**: PENDING` before the actual `**Status**:` field, the replace hits the wrong occurrence. This does not occur in current question data but is a real trap as the question bank grows.

## Fix Specification

**File**: `C:\Users\trg16\Dev\autosearch\bl\questions.py`
**Line**: 225

Change from:
```python
next_block = text.find("\n## Q", block_start + 1)
```
Change to:
```python
next_block = text.find("\n## ", block_start + 1)
```

**Risk**: Low. `"\n## "` is the same generic boundary already used by `parse_questions()`. Only regression: a question body containing a bare `## ` heading line causes early truncation — pre-existing risk, not introduced by this fix.
