# D13.2: followup.py sub-question format missing **Operational Mode** field

**Question**: followup.py sub-question format missing **Operational Mode** field — all sub-questions default to diagnose
**Verdict**: FAILURE
**Severity**: High
**Mode**: code_audit
**Operational Mode**: diagnose

## Evidence

### 1. `_build_followup_prompt()` template — `bl/followup.py` lines 125-138

The EXACT FORMAT block in the prompt template is:

```
EXACT FORMAT (use this precisely):
---
## {parent_id}.N [DOMAIN] Short drill-down title
**Mode**: {mode}
**Agent**: {agent}
**Status**: PENDING
**Hypothesis**: Specific follow-on hypothesis targeting the observed failure.
**Test**: Concrete test command or agent instruction targeting the specific failure.
**Verdict threshold**:
- FAILURE: specific measurable condition
- WARNING: specific measurable condition
- HEALTHY: specific measurable condition
**Derived from**: {parent_id} ({verdict})
---
```

`**Operational Mode**:` is entirely absent from this template. The LLM is given no instruction to emit it, and the format string has no `{op_mode}` interpolation.

The `mode` variable on line 101 is `question.get("mode", "agent")` — this is the runner-dispatch mode tag (e.g., "agent", "code_audit"), NOT the operational mode (e.g., "audit", "diagnose", "fix").

### 2. `parse_questions()` default — `bl/questions.py` line 78

```python
"operational_mode": fields.get("operational_mode", "diagnose"),
```

When `**Operational Mode**:` is absent from a question block, `operational_mode` defaults to `"diagnose"`. Because the sub-question template never emits this field, every generated sub-question silently defaults to `operational_mode="diagnose"`.

### 3. Concrete failure trace: Parent A5.1 (audit) → sub-question

`A5.1` has `**Operational Mode**: audit` (confirmed in questions.md). If A5.1 returns NON_COMPLIANT and `generate_followup()` is triggered:

- `_build_followup_prompt()` is called with `question = A5.1_dict`
- Template emits `**Mode**: {mode}` (runner mode, e.g., "agent") — not operational mode
- Template emits no `**Operational Mode**:` field at all
- LLM output block contains no `**Operational Mode**:` line
- `_parse_followup_blocks()` validates only `**Status**: PENDING` and `**Derived from**:` — no operational_mode check
- Sub-question block appended to questions.md without `**Operational Mode**:` field
- `parse_questions()` reads the sub-question: `fields.get("operational_mode", "diagnose")` → `"diagnose"`
- Sub-question A5.1.1 now has `operational_mode="diagnose"` instead of `"audit"`

### 4. Downstream impact — `bl/campaign.py` lines 52-57

```python
op_mode = question.get("operational_mode", "")
if op_mode:
    mode_ctx = _load_mode_context(op_mode)
    if mode_ctx:
        question["mode_context"] = mode_ctx
```

`_load_mode_context("diagnose")` loads `modes/diagnose.md` and injects it into the agent prompt. For an audit follow-up sub-question, the agent receives the *diagnose-analyst* role definition instead of the *audit-analyst* role definition. The agent runs under the wrong operational identity — diagnose mode instructs the agent to trace root causes and produce Fix Specifications, while audit mode instructs it to verify compliance against a checklist. These are fundamentally different operational roles producing structurally different outputs.

### 5. Session context insight line — `bl/campaign.py` line 143

```python
insight_line = f"[{qid}] {result['verdict']} [{question.get('operational_mode', question['mode'])}]: ..."
```

The session context records the wrong operational mode label for the finding, further polluting the session context that downstream questions read.

## Impact

- All follow-up sub-questions generated from audit (`operational_mode="audit"`) or validate/fix/monitor parent questions are silently reclassified as `operational_mode="diagnose"`.
- The wrong `modes/{operational_mode}.md` context file is injected into the agent prompt — agents run under the wrong operational identity.
- Audit drill-downs run as diagnose agents: they produce Fix Specifications instead of compliance checklists, producing structurally invalid findings for audit questions.
- Fix drill-downs run as diagnose agents: they attempt to diagnose already-identified root causes rather than implementing fixes.
- The session context insight line records `[DIAGNOSE]` for all follow-up sub-questions regardless of parent operational mode.
- This failure is silent: no error is raised, no warning is logged. The campaign proceeds normally with operationally misconfigured sub-questions.

## Fix Specification

**Root Cause**: `_build_followup_prompt()` in `bl/followup.py` does not extract `operational_mode` from the parent question dict and does not include `**Operational Mode**: {op_mode}` in the sub-question format template. The LLM therefore never emits this field, and `parse_questions()` defaults all generated sub-questions to `operational_mode="diagnose"`.

**File**: `bl/followup.py`

**Location**: Lines 94-140, specifically the template string at lines 125-138

**Change**:

1. After line 103 (`test = question.get("test", "")`), add:
   ```python
   op_mode = question.get("operational_mode", "diagnose")
   ```

2. In the EXACT FORMAT template block (lines 125-138), add `**Operational Mode**: {op_mode}` immediately after `**Mode**: {mode}`:
   ```
   ## {parent_id}.N [DOMAIN] Short drill-down title
   **Mode**: {mode}
   **Operational Mode**: {op_mode}
   **Agent**: {agent}
   **Status**: PENDING
   ...
   ```

**Risk**: Low regression surface. The change only adds a new field to the LLM prompt template and a new variable extraction. It does not alter parsing logic, runner dispatch, or any other sub-question field. Sub-questions that previously defaulted to "diagnose" will now correctly inherit the parent's operational mode. No existing tests exercise sub-question field content.

## Verdict: FAILURE

`**Operational Mode**:` is confirmed absent from `_build_followup_prompt()` template (lines 125-138). `parse_questions()` line 78 confirms the default is `"diagnose"`. All follow-up sub-questions generated from non-diagnose parent questions are silently misconfigured, causing agents to run under the wrong operational identity with wrong mode context injected.
