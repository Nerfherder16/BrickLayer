# Finding: D12.1 — _REGRESSIONS set contains only BL 1.x verdict pairs; BL 2.0 regressions invisible

**Question**: `_REGRESSIONS` set in history.py contains only BL 1.x verdict pairs — BL 2.0 regressions invisible
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: High
**Mode**: code_audit
**Operational Mode**: diagnose

## Evidence

`C:\Users\trg16\Dev\autosearch\bl\history.py` lines 130–134:

```python
_REGRESSIONS: set[tuple[str, str]] = {
    ("HEALTHY", "FAILURE"),
    ("HEALTHY", "WARNING"),
    ("WARNING", "FAILURE"),
}
```

That is the complete set. No other pairs exist in this literal anywhere in `history.py`.

BL 2.0 introduces a significantly expanded verdict vocabulary, confirmed across multiple files:

- `bl/findings.py` lines 28–34, 269–270, 362–366, 384–385: `DIAGNOSIS_COMPLETE`, `PENDING_EXTERNAL`, `FIXED`, `COMPLIANT`, `NON_COMPLIANT`, `PARTIAL`, `NOT_APPLICABLE`
- `bl/agent_db.py` lines 27–72: full taxonomy — `FIXED`, `COMPLIANT`, `DIAGNOSIS_COMPLETE`, `NON_COMPLIANT`, `ALERT`, `REGRESSION`, `OK`, `DEGRADED`, `DEGRADED_TRENDING`
- `bl/runners/agent.py` lines 29–60: agent runner accepts and parses all of these verdicts
- `bl/questions.py` lines 101–208: question parser treats `DIAGNOSIS_COMPLETE`, `PENDING_EXTERNAL`, `FIXED`, `COMPLIANT`, `NON_COMPLIANT` as terminal statuses

The `detect_regression()` function at lines 137–159 checks only the three pairs in `_REGRESSIONS`. Trace for a `COMPLIANT → NON_COMPLIANT` scenario:

```python
# history[1].verdict = "COMPLIANT"  (prior run)
# new_verdict        = "NON_COMPLIANT" (current run)

if ("COMPLIANT", "NON_COMPLIANT") in _REGRESSIONS:   # False — not in the set
    ...                                                # skipped
return None                                            # regression missed
```

The same `return None` occurs for every BL 2.0 regression pair:
- `("COMPLIANT", "NON_COMPLIANT")` — compliance audit regressed
- `("FIXED", "FAILURE")` — a previously fixed question broke again
- `("FIXED", "NON_COMPLIANT")` — a fix introduced a compliance violation
- `("HEALTHY", "NON_COMPLIANT")` — a healthy question violated a compliance rule
- `("DIAGNOSIS_COMPLETE", "FAILURE")` — a diagnosed question failed during re-run
- `("OK", "DEGRADED")` — monitor saw a downgrade
- `("OK", "ALERT")` — monitor escalated to alert from clean state
- `("DEGRADED", "ALERT")` — monitor escalated

`detect_regression()` is actively called in `bl/campaign.py` line 96 — it is not dead code. The result at line 97–103 prints a `[REGRESSION]` warning and attaches the regression dict to the result. Because the function always returns `None` for BL 2.0 verdicts, none of these warnings ever fire.

`get_regressions()` (lines 162–197) uses the same `_REGRESSIONS` set at line 187. Both the per-question call path and the bulk scan path are blind to BL 2.0 regressions.

The docstring at line 11–13 of `history.py` even acknowledges only the BL 1.x verdict set: `"FAILURE|WARNING|HEALTHY|INCONCLUSIVE"`.

## Impact

1. **Silent regressions go undetected.** A question that was `COMPLIANT` last wave and is now `NON_COMPLIANT` produces no `[REGRESSION]` banner in the campaign output, no `result["regression"]` dict, and no entry from `get_regressions()`. The campaign operator has no automated signal that the quality bar dropped.

2. **FIXED → FAILURE regressions are completely invisible.** This is the most dangerous case: the heal loop marks a question `FIXED`, but if a subsequent run shows `FAILURE` again, `detect_regression()` returns `None`. The fix was silently reverted with no alert.

3. **`regression_report()` (line 205) will always return `""` for BL 2.0 campaigns** unless a verdict happens to transit through the three legacy pairs. The report is meaningless for the BL 2.0 self-audit campaign.

4. **`get_regressions()` used by the dashboard API (if wired)** will report zero regressions even when the history ledger contains multiple clear degradations.

## Fix Specification

**Root Cause**: `_REGRESSIONS` at `bl/history.py` lines 130–134 was written for BL 1.x and was never updated when BL 2.0 expanded the verdict taxonomy. All BL 2.0 verdict transitions that represent a quality degradation are absent from the set.

**File**: `bl/history.py`

**Location**: Lines 130–134

**Change**: Replace the three-pair set with the full BL 2.0-aware regression set. The principle is: any transition from a "pass" or "positive" state to a "failure" or "degraded" state is a regression.

```python
# A regression is any transition from a passing/positive state to a failing/degraded state.
# BL 1.x pairs
# BL 2.0 pairs added: COMPLIANT→NON_COMPLIANT, FIXED→FAILURE, FIXED→NON_COMPLIANT,
#   HEALTHY→NON_COMPLIANT, DIAGNOSIS_COMPLETE→FAILURE,
#   OK→DEGRADED, OK→ALERT, DEGRADED→ALERT
_REGRESSIONS: set[tuple[str, str]] = {
    # BL 1.x legacy
    ("HEALTHY", "FAILURE"),
    ("HEALTHY", "WARNING"),
    ("WARNING", "FAILURE"),
    # BL 2.0 — compliance audit
    ("COMPLIANT", "NON_COMPLIANT"),
    ("COMPLIANT", "FAILURE"),
    # BL 2.0 — fix state
    ("FIXED", "FAILURE"),
    ("FIXED", "NON_COMPLIANT"),
    ("FIXED", "WARNING"),
    # BL 2.0 — cross-mode
    ("HEALTHY", "NON_COMPLIANT"),
    ("HEALTHY", "ALERT"),
    ("DIAGNOSIS_COMPLETE", "FAILURE"),
    # BL 2.0 — monitor mode
    ("OK", "DEGRADED"),
    ("OK", "ALERT"),
    ("DEGRADED", "ALERT"),
}
```

**Risk**: The expanded set introduces more regression alerts. If verdicts bounce legitimately between states (e.g., `FIXED` followed by a re-audit that produces `NON_COMPLIANT` due to a stricter agent), operators may see more `[REGRESSION]` banners. This is the desired behavior — these were previously invisible. No false negatives introduced; false positive risk is low because all added pairs represent genuine quality reversals.
