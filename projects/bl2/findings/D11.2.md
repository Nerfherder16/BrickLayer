# Finding: D11.2 — questions.py sync_status_from_results() maps FAILURE/NON_COMPLIANT to DONE

**Question**: Does sync_status_from_results() preserve failure verdicts or map them to DONE?
**Verdict**: FAILURE
**Severity**: Medium
**Mode**: code_audit
**Operational Mode**: diagnose

## Root Cause

### sync_status_from_results() lines 196-208

```python
done_ids[parts[0]] = (
    verdict
    if verdict
    in (
        "INCONCLUSIVE",
        "DIAGNOSIS_COMPLETE",
        "PENDING_EXTERNAL",
        "FIXED",
        "FIX_FAILED",
        "BLOCKED",
    )
    else "DONE"
)
```

Verdicts NOT in this preserve set → status = "DONE". This includes:
- **FAILURE** → "DONE" (D-questions that failed)
- **NON_COMPLIANT** → "DONE" (A-questions that are non-compliant)
- **WARNING** → "DONE"
- **REGRESSION** → "DONE"
- **ALERT** → "DONE"

### Comparison to F8.2 fix in findings.py

F8.2 extended `_PRESERVE_AS_IS` in findings.py to include:
```python
"FAILURE", "NON_COMPLIANT", "WARNING", "REGRESSION", "ALERT",
```

But `sync_status_from_results()` is a separate code path that was NOT updated by F8.2. The two paths diverge:
- `_mark_question_done()` in findings.py: FAILURE → "FAILURE" in questions.md ✓ (F8.2 fixed)
- `sync_status_from_results()` in questions.py: FAILURE → "DONE" in questions.md ✗ (not fixed)

### When sync_status_from_results() is called

`sync_status_from_results()` is called from `campaign.py` during the reconciliation phase (typically at campaign startup or after batch runs). It reads results.tsv and updates questions.md status fields. If it overwrites "FAILURE" with "DONE", subsequent calls to `get_question_status()` would return "DONE" (via results.tsv lookup which returns the correct value) — actually wait:

`get_question_status()` reads from results.tsv directly (line 94: `return parts[1].strip()`), NOT from questions.md. So the actual runtime status is correct (FAILURE from TSV). The visible **Status** field in questions.md would show "DONE" — incorrect for human review.

### Concrete impact

For this BL 2.0 campaign, if `sync_status_from_results()` were called after Wave 10:
- D10.1 (**Status**: FAILURE in questions.md as written) → would be overwritten to "DONE"
- D10.2 (**Status**: FAILURE in questions.md as written) → would be overwritten to "DONE"
- A10.1 (**Status**: NON_COMPLIANT in questions.md as written) → would be overwritten to "DONE"

Questions.md would show all FAILURE questions as "DONE" — misleading to human reviewers and dashboards that read questions.md directly.

## Verdict: FAILURE

sync_status_from_results() has a preserve list that excludes FAILURE, NON_COMPLIANT, WARNING, REGRESSION, ALERT — mapping them all to "DONE". This is the same class of bug fixed by F8.2 in findings.py but left unfixed in this code path. Fix target: F11.2.
