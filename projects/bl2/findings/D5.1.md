# Finding: D5.1 — parse_questions() uses bracket tag for runner dispatch, discarding **Mode** body field

**Question**: Does `parse_questions()` use the bracket tag (e.g., "correctness", "fix") as `question["mode"]` for runner dispatch, ignoring the `**Mode**:` body field?
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: High
**Mode**: diagnose
**Operational Mode**: diagnose

## Root Cause

In `bl/questions.py` `parse_questions()`, line 65:

```python
questions.append(
    {
        "id": qid,
        "mode": mode_raw,          # BUG: always uses bracket tag string
        "title": title,
        ...
        "operational_mode": fields.get("operational_mode", "diagnose"),  # body field read correctly
    }
)
```

`mode_raw = tag_raw.lower()` is the bracket tag (e.g., "correctness" from `## D1 [CORRECTNESS]`). The `fields` dict is populated from the question body — including `**Mode**:` lines — via `field_pattern`, which explicitly matches "Mode" as a parseable field (line 27). However, `fields.get("mode")` is never used; `mode_raw` is hardcoded.

## Evidence

```
# field_pattern line 27 — "Mode" IS parsed from body:
r"^\*\*(Mode|Target|Hypothesis|Test|...)

# line 37: mode_raw = bracket tag
mode_raw = tag_raw.lower()

# line 65: body Mode field silently discarded
"mode": mode_raw,
```

`run_question()` dispatches on `question["mode"]`. Registered runners: "agent", "correctness", "http", "performance", "quality", "static", "subprocess". BL 2.0 bracket tags not registered: "fix", "compliance", "monitor", "diagnose", "validate", "frontier", "predict", "research", "evolve", "audit". All produce INCONCLUSIVE via the fallback path.

**Example broken flow**: A BL 2.0 question with `## D1 [CORRECTNESS]` and body `**Mode**: agent` → `question["mode"] = "correctness"` → dispatches to `run_correctness` (a pytest subprocess runner, not a live agent). The body `**Mode**: agent` is discarded.

## Fix Specification

- **File**: `bl/questions.py`, `parse_questions()`
- **Line**: 65
- **Change**: `"mode": mode_raw` → `"mode": fields.get("mode", mode_raw)`
- **Semantics**: Body `**Mode**:` field takes priority; bracket tag is fallback for BL 1.x questions without a body `**Mode**` field (backward compatible)
- **Verification**: For `## D1 [CORRECTNESS]` question with body `**Mode**: agent`, `question["mode"]` must equal `"agent"` after parse

## Impact

All BL 2.0 questions with non-"correctness" bracket tags that rely on `**Mode**: agent` in the body will produce INCONCLUSIVE instead of invoking the correct runner. This is the primary reason BL 2.0 campaigns could not execute any non-correctness questions correctly until this is fixed.
