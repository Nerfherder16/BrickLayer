# Finding: D4 — _reactivate_pending_external() location mismatch with spec

**Question**: The project-brief states `_reactivate_pending_external()` lives in `bl/questions.py`, but the actual implementation is in `bl/campaign.py` (lines 497-539).
**Verdict**: FAILURE
**Severity**: Medium
**Mode**: agent

## Evidence

`project-brief.md` section `bl/questions.py — three additions`, item 4:
> **`_reactivate_pending_external()`** — removes PENDING_EXTERNAL rows from results.tsv when `resume_after` has passed

Actual location: `bl/campaign.py` lines 497-539. `bl/questions.py` has no such function.

```bash
grep -rn "_reactivate_pending_external" bl/
# Output:
# bl/campaign.py:497:def _reactivate_pending_external(questions: list[dict]) -> int:
# bl/campaign.py:545:    _reactivate_pending_external(questions)
```

`bl/questions.py` has no `_reactivate_pending_external` implementation at all.

**Functional impact**: The function itself works correctly in campaign.py. However:
1. The spec contract is violated — callers expecting `from bl.questions import _reactivate_pending_external` will get an ImportError.
2. The function cannot be unit-tested in isolation from campaign.py (it's a private internal function of campaign, not the questions API).
3. The duplicate `resume_after` evaluation exists in theory: `_reactivate_pending_external()` checks `resume_after` to decide whether to remove the PENDING_EXTERNAL row, and `get_next_pending()` in questions.py also checks `resume_after` on PENDING questions. At the exact moment when `resume_after` time boundary passes between the two evaluations, they could produce different results — but in practice both run within milliseconds.

## Mitigation Recommendation

Move `_reactivate_pending_external()` to `bl/questions.py` to match the spec. Update campaign.py to import it:
```python
from bl.questions import _reactivate_pending_external
```

This makes the function testable and aligns implementation with spec.
