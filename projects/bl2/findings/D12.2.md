# Finding: D12.2 — classify_failure_type_local() guard excludes BL 2.0 verdicts NON_COMPLIANT and REGRESSION

**Question**: `classify_failure_type_local()` in `bl/local_inference.py` line 55: `if verdict not in ("FAILURE", "INCONCLUSIVE"): return None`. BL 2.0 failure verdicts `NON_COMPLIANT`, `REGRESSION`, and `ALERT` are not in this tuple — the function returns `None` for them, so `failure_type` is never classified for compliance audit failures. Additionally, `_SYSTEM_PROMPT` (lines 15-22) only describes BL 1.x verdicts (HEALTHY/WARNING/FAILURE/INCONCLUSIVE), so even if the guard were fixed, the local model would have no context for BL 2.0 verdict semantics.
**Verdict**: FAILURE
**Severity**: Low
**Mode**: code_audit
**Operational Mode**: diagnose

```python
def classify_failure_type(result: dict, mode: str) -> str | None:
    verdict = result.get("verdict", "")
    if verdict in _NON_FAILURE_VERDICTS:   # line 65: ALERT stops here correctly
        return None

    local = classify_failure_type_local(result, mode)   # line 69: NON_COMPLIANT / REGRESSION reach this
    if local:
        return local

    # Heuristic fallback — runs for NON_COMPLIANT and REGRESSION when local returns None
    details = (result.get("details", "") or "").lower()
    summary = (result.get("summary", "") or "").lower()
    combined = details + " " + summary

    if any(s in combined for s in ("timeout", "timed out", ...)):
        return "timeout"
    if any(s in combined for s in ("connection refused", "importerror", ...)):
        return "tool_failure"
    if any(s in combined for s in ("syntaxerror", "indentationerror", ...)):
        return "syntax"
    if mode in ("correctness", "performance"):
        return "logic"
    if mode in ("agent", "quality", "static"):
        if any(s in combined for s in ("no evidence", "assumed", ...)):
            return "hallucination"
    return "unknown"
```

For `NON_COMPLIANT` and `REGRESSION` verdicts:
- They are NOT in `_NON_FAILURE_VERDICTS`, so they reach line 69
- `classify_failure_type_local()` returns `None` for them (line 55 guard)
- The heuristic fallback then runs and produces a classification

## Impact

The actual impact is narrower than the hypothesis claimed:

1. **ALERT** — No bug. `ALERT` is correctly in `_NON_FAILURE_VERDICTS` in `findings.py` and is properly excluded from failure classification at line 65, before `classify_failure_type_local` is ever called.

2. **NON_COMPLIANT and REGRESSION** — Confirmed partial bug. For these two verdicts, `classify_failure_type_local()` returns `None` due to the line-55 guard. The local Ollama model is never consulted. However, the heuristic keyword fallback in `findings.py` runs and produces a classification (`timeout`, `tool_failure`, `syntax`, `hallucination`, `logic`, or `unknown`). This means `failure_type` is NOT permanently missing — it is classified by the less-accurate heuristic path.

3. **Secondary issue — stale _SYSTEM_PROMPT** — Even if line 55 were widened, the Ollama model would receive no definition of what `NON_COMPLIANT` or `REGRESSION` mean when reasoning about failure type. This would likely cause the model to produce lower-quality classifications for those verdicts.

4. **Functional severity is Low**: The heuristic fallback compensates for the missing local model call. `failure_type` is populated in all downstream consumers. The defect degrades classification accuracy for `NON_COMPLIANT`/`REGRESSION` results but does not cause null values or crashes.

## Fix Specification

**Root Cause**: Line 55 in `bl/local_inference.py` uses a hardcoded two-item tuple `("FAILURE", "INCONCLUSIVE")` that only covers BL 1.x failure verdicts. BL 2.0 introduced `NON_COMPLIANT` and `REGRESSION` as additional failure-class verdicts, but the guard was never updated. A secondary defect is that `_SYSTEM_PROMPT` (lines 15–22) still only describes BL 1.x verdict semantics, providing no context to the Ollama model for BL 2.0 verdicts even if the guard were widened.

**File**: `bl/local_inference.py`

**Location**:
- Primary: line 55 (guard tuple)
- Secondary: lines 15–22 (`_SYSTEM_PROMPT`)

**Change**:

Primary fix — widen the guard to include BL 2.0 failure verdicts:
```python
# line 55: change from:
if verdict not in ("FAILURE", "INCONCLUSIVE"):
# to:
if verdict not in ("FAILURE", "INCONCLUSIVE", "NON_COMPLIANT", "REGRESSION"):
```

Secondary fix — extend `_SYSTEM_PROMPT` to include BL 2.0 verdict semantics. After "INCONCLUSIVE=could not determine." add:
```
"NON_COMPLIANT=audit found a rule violation or constraint breach, "
"REGRESSION=previously passing condition has deteriorated or broken. "
```
