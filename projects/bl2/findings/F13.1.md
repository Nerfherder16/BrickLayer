# Finding: F13.1 — Fixed followup.py agent fallback and Operational Mode field in sub-question template

**Question**: Fix D13.1+D13.2: agent_name dead fallback and missing **Operational Mode** field
**Verdict**: FIXED
**Severity**: Medium
**Mode**: code_audit
**Source findings**: D13.1 (DIAGNOSIS_COMPLETE), D13.2 (FAILURE)

## Changes Made

### bl/followup.py — agent_name fallback (D13.1 fix)

Before:
```python
agent = question.get("agent_name", "quantitative-analyst")
```

After:
```python
agent = question.get("agent_name") or "quantitative-analyst"
```

`parse_questions()` always sets `agent_name=""` when the field is absent, so the `dict.get()` default-arg path was never reached. Switching to `or` correctly catches the empty-string case.

### bl/followup.py — Operational Mode in template (D13.2 fix)

Added extraction at line 103:
```python
op_mode = question.get("operational_mode", "diagnose")
```

Added to the sub-question format template (between `**Mode**` and `**Agent**`):
```
**Mode**: {mode}
**Operational Mode**: {op_mode}
**Agent**: {agent}
```

Before this fix, all follow-up sub-questions emitted by `_build_followup_prompt()` would inherit the parent's `mode` but never its `operational_mode`, causing the LLM to generate sub-questions that `parse_questions()` would parse with `operational_mode=""` (defaulting to "diagnose") regardless of parent context.

## Verification

Trace: parent question with `operational_mode="audit"` → `op_mode = "audit"` → template renders `**Operational Mode**: audit` → LLM includes it in each sub-question block → `parse_questions()` reads and preserves `operational_mode="audit"` on the child questions.

Linter confirmed clean: both variables (`agent`, `op_mode`) are used in the f-string template, no F841 unused-variable warnings.

## Backward Compatibility

- BL 1.x questions that set `agent_name` explicitly: unchanged — `or` short-circuits on truthy value.
- Questions with `agent_name` absent (field missing from dict): `get()` returns `None`, `or` fallback fires — same result as intended default.
- Questions with `agent_name=""` (parse_questions empty-string path): previously returned `""` (dead default), now correctly falls back to `"quantitative-analyst"`.
- Questions without `operational_mode`: `get()` returns `"diagnose"` default — no change in generated output vs. implicit behavior.
