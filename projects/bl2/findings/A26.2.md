### Why has_verdict_threshold Fires Only 21.6% of the Time

The pattern `FAILURE: ... HEALTHY:` is a BL 1.x labeling convention where verdict thresholds were listed as explicit bullet points:
```
- HEALTHY: All checks pass
- FAILURE: Any check fails
```

In BL 2.0 Wave 2+ questions, the field is named `**Verdict threshold**:` with narrative prose, e.g.:
```
**Verdict threshold**:
- DIAGNOSIS_COMPLETE: sentinel is "\n## Q" (confirmed BL 1.x-only); ...
- HEALTHY: sentinel already covers BL 2.0 headers...
```

Some Wave 2 questions (19/88 = 21.6%) happen to retain `FAILURE:` and `HEALTHY:` labels within their `**Verdict threshold**:` section. The remaining 69 blocks use the newer format with narrative descriptions using `DIAGNOSIS_COMPLETE:`, `FIXED:`, `NON_COMPLIANT:`, etc. — none of which match `FAILURE:` + `HEALTHY:`.

## Fix Specification

**File**: `bl/crucible.py`
**Location**: `_score_hypothesis_generator()` → `check_block()` lines 201–203

```python
# Change from:
"has_verdict_threshold": 1.0
if ("FAILURE:" in b and "HEALTHY:" in b)
else 0.0,

# Change to:
"has_verdict_threshold": 1.0
if (
    ("FAILURE:" in b and "HEALTHY:" in b)
    or "**Verdict threshold**:" in b
    or "**Success criterion**:" in b
)
else 0.0,
```

**Expected outcome**:
- has_verdict_threshold rate: 0.2159 → **0.9091** (+0.693)
- Overall hypothesis-generator score: 0.5114 → **0.6847** (with this fix alone)
- If combined with has_test `**Test**:` fix: 0.5114 → **0.7915**

**Risk**: Low. The new checks are substring tests on BL 2.0 structural field names. No BL 1.x content would be affected — `**Verdict threshold**:` and `**Success criterion**:` are BL 2.0-specific strings not present in BL 1.x question data.

## Conclusion

**NON_COMPLIANT** with Fix Specification. The `has_verdict_threshold` check in `_score_hypothesis_generator()` captures only 21.6% of BL 2.0 Wave 2+ questions due to a field naming convention mismatch. Accepting `**Verdict threshold**:` and `**Success criterion**:` as valid BL 2.0 equivalents would raise the rate from 0.2159 to 0.9091, contributing a +0.173 improvement to the overall hypothesis-generator score. Combined with the has_test `**Test**:` fix identified in V26.1, the overall score would rise from 0.5114 to approximately 0.7915.
