# Finding: D10 — get_next_pending() double-gates on _PARKED_STATUSES and status != PENDING

**Question**: In `bl/questions.py` `get_next_pending()`, the function first checks `if q["status"] in _PARKED_STATUSES: continue` then checks `if q["status"] != "PENDING": continue`. Does `get_question_status()` return raw verdicts from results.tsv or mapped statuses from questions.md? Could there be a verdict/status combination where a completed question is re-run?
**Verdict**: HEALTHY
**Severity**: Info
**Mode**: agent

## Evidence

`bl/questions.py` `get_question_status()` reads from results.tsv and returns the raw verdict (e.g., "HEALTHY", "WARNING", "FAILURE") OR "PENDING" if not found.

`get_next_pending()` applies two guards:
1. `if q["status"] in _PARKED_STATUSES: continue` — catches DIAGNOSIS_COMPLETE, PENDING_EXTERNAL, DONE, INCONCLUSIVE, FIXED, FIX_FAILED, COMPLIANT, NON_COMPLIANT, CALIBRATED, BLOCKED
2. `if q["status"] != "PENDING": continue` — catches any remaining non-PENDING status (HEALTHY, WARNING, FAILURE, etc.)

The double-gate correctly handles all cases:
- DONE questions: caught by guard 1 (DONE is in `_PARKED_STATUSES`)
- HEALTHY/WARNING/FAILURE questions: caught by guard 2
- INCONCLUSIVE, FIXED, etc.: caught by guard 1

No scenario exists where a completed question (with any non-PENDING status in results.tsv) passes both guards. The redundancy is harmless — guard 1 filters terminal-mapped verdicts, guard 2 is a catch-all for any non-PENDING raw verdict.

`_mark_question_done()` in findings.py updates questions.md Status field, but `get_question_status()` reads from results.tsv (not questions.md), so questions.md Status is irrelevant to `get_next_pending()` logic.

## Mitigation Recommendation

No action required. The double-gate is redundant but correct. The code is slightly defensive — guard 2 would be sufficient alone, but guard 1 makes the intent clearer for terminal mapped verdicts.

Optional refactor: replace guard 1 + guard 2 with a single positive check:
```python
if q["status"] != "PENDING":
    continue
```
This is functionally equivalent and simpler.
