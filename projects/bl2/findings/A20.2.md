# Finding: A20.2 — _score_fix_implementer fix_rows lacks F-prefix guard; D-mid.4 historical FIXED row inflates fixed_rate; structural guard warranted

**Question**: Does `_score_fix_implementer` require a structural question_id prefix guard given that D-prefix FIXED rows are a confirmed historical edge case?
**Verdict**: NON_COMPLIANT
**Severity**: Low

## Evidence

**`_score_fix_implementer` fix_rows filter** (`bl/crucible.py` line 447):
```python
fix_rows = [ln for ln in lines if re.search(r"\t(FIXED|FIX_FAILED)\t", ln)]
```
No question_id prefix filter. Matches any row with FIXED or FIX_FAILED verdict.

**project-brief.md line 18** — Fix mode verdict assignment:
```
| Fix | FIXED / FIX_FAILED / INCONCLUSIVE | Implement a DIAGNOSIS_COMPLETE spec |
```
FIXED and FIX_FAILED are designated exclusively to Fix operational mode (F-prefix questions). The brief explicitly scopes these verdicts to Fix-mode agents.

**D-mid.4 confirmed edge case**: `D-mid.4` is a D-prefix question that produced FIXED in results.tsv (a diagnose question that self-fixed in-session). This row is present in the current results.tsv and enters `fix_rows`. With 40 fix_rows total, this one D-prefix row represents a 2.5% contamination of the fix_rows corpus.

**Current impact**: D-mid.4 has verdict FIXED (not FIX_FAILED), so it adds 1 to the FIXED count and 1 to total fix_rows. The fixed_rate computation: `fixed_rate = count(FIXED) / len(fix_rows)` includes this row in both numerator and denominator, yielding net-zero rate distortion for this specific case. However:
- If D-mid.4 had verdict FIX_FAILED, it would drag fixed_rate downward without being a genuine fix-implementer failure.
- Future campaigns where diagnose questions self-fix more frequently would compound this distortion.

**Structural gap**: The project-brief designates FIXED as Fix-mode-only. No code enforces this invariant in the scorer. The D-mid.4 occurrence proves the gap is not merely theoretical.

## Fix Specification

**Target file**: `bl/crucible.py`
**Target location**: `_score_fix_implementer()` line 447
**Concrete edit**:
```python
# Before (verdict-only filter, no prefix guard):
fix_rows = [ln for ln in lines if re.search(r"\t(FIXED|FIX_FAILED)\t", ln)]

# After (BL 2.0 F-prefix guard — new-format rows where col[1] starts with "F"):
def _is_fix_row(ln: str) -> bool:
    parts = ln.split("\t")
    # New format: N/A | qid | verdict | ...
    if len(parts) >= 3 and parts[0] == "N/A":
        return parts[1].startswith("F") and bool(
            re.search(r"^(FIXED|FIX_FAILED)$", parts[2])
        )
    # Old format (BL 1.x): qid | verdict | ...
    return bool(re.search(r"\t(FIXED|FIX_FAILED)\t", ln))
fix_rows = [ln for ln in lines if _is_fix_row(ln)]
```

**Verification command**: Count fix_rows before and after — D-mid.4 should be excluded, reducing total from ~40 to ~39 for new-format rows.
