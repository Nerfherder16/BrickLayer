# Finding: D5 — Heal loop EXHAUSTED note uses max_cycles instead of actual cycle count

**Question**: In `bl/healloop.py` lines 323-329, the EXHAUSTED `_append_heal_note()` call always uses `max_cycles` as the cycle number, even when the loop exited early via `break` (e.g., missing agent on cycle 1, or diagnose returned non-DIAGNOSIS_COMPLETE on cycle 2).
**Verdict**: FAILURE
**Severity**: Medium
**Mode**: agent

## Evidence

`bl/healloop.py` lines 323-329 (after the for loop):
```python
_append_heal_note(
    finding_path,
    max_cycles,        # ← always uses max_cycles, not the actual exit cycle
    "EXHAUSTED",
    f"Self-healing exhausted {max_cycles} cycle(s) — human intervention required. "
    f"Review the heal cycle notes above for root cause history.",
)
```

The `for cycle in range(1, max_cycles + 1):` loop exits via `break` in 3 cases:
1. Line 180: `break` — diagnose-analyst.md missing (could be cycle 1)
2. Line 215: `break` — diagnose-analyst returned non-DIAGNOSIS_COMPLETE (any cycle)
3. Line 321: `break` — unexpected verdict in the `else` branch (any cycle)

In all three cases, the `cycle` variable holds the actual exit cycle (e.g., 1 if it broke on the first iteration), but `_append_heal_note` is called with `max_cycles` (e.g., 3).

**Concrete example**: If `diagnose-analyst.md` is missing, the loop breaks on cycle 1. The EXHAUSTED note will say "Heal Cycle 3 — EXHAUSTED" when the loop only ran 1 cycle. The audit trail incorrectly suggests 3 full cycles were attempted.

Note: when the loop runs to natural exhaustion (all max_cycles complete without FIXED), `cycle` would equal `max_cycles` at end — so in THAT case the note is correct. The bug manifests only on early `break` exits.

## Mitigation Recommendation

Track the actual exit cycle explicitly:
```python
last_cycle = 0
for cycle in range(1, max_cycles + 1):
    last_cycle = cycle
    ...  # existing logic with break paths

_append_heal_note(
    finding_path,
    last_cycle,      # ← use actual exit cycle
    "EXHAUSTED",
    ...
)
```
