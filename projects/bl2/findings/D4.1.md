# Finding: D4.1 — parse_questions() Q-prefix regex excludes all BL 2.0 question IDs

**Question**: Does `parse_questions()` use a regex that only matches Q-prefixed IDs (BL 1.x), making all BL 2.0 questions invisible to the campaign engine?
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: Critical
**Mode**: diagnose
**Type**: CODE-AUDIT

## Root Cause

`bl/questions.py` `parse_questions()` defines:

```python
block_pattern = re.compile(
    r"^## (Q[\w.-]+) \[(\w+)\] (.+?)$",
    re.MULTILINE,
)
```

The group `(Q[\w.-]+)` requires question IDs to start with the literal character `Q`. This was correct for BL 1.x where all questions used `Q1`, `Q2`, `Q1.1` IDs.

BL 2.0 uses mode-prefixed IDs: `D1`, `F2.1`, `A4`, `M3.1`, `R2`, `V3.1`, `P1`, `E1`, `C1`, `N1`. None start with `Q`.

## Verified Impact

```
python -c "
import re
p = re.compile(r'^## (Q[\w.-]+) \[(\w+)\] (.+?)$', re.MULTILINE)
text = open('projects/bl2/questions.md').read()
print(len(list(p.finditer(text))))  # → 0 (zero BL 2.0 questions matched)
"
```

In any BL 2.0 campaign:
1. `parse_questions()` → returns `[]` (empty list)
2. `get_next_pending([])` → returns `None`
3. Campaign loop exits immediately — **zero questions ever executed**

This makes `bl/campaign.py` non-functional for any BL 2.0-format question bank.

## Fix Specification

**File**: `bl/questions.py`, `parse_questions()`, `block_pattern` line

**Change**: Replace `Q[\w.-]+` with `[\w][\w.-]*` to match any alphanumeric-prefixed question ID.

**Before**:
```python
block_pattern = re.compile(
    r"^## (Q[\w.-]+) \[(\w+)\] (.+?)$",
    re.MULTILINE,
)
```

**After**:
```python
block_pattern = re.compile(
    r"^## ([\w][\w.-]*) \[(\w+)\] (.+?)$",
    re.MULTILINE,
)
```

**Verification**:
```
python -c "
import re
p = re.compile(r'^## ([\w][\w.-]*) \[(\w+)\] (.+?)$', re.MULTILINE)
text = open('projects/bl2/questions.md').read()
matches = list(p.finditer(text))
print(len(matches))  # → 38 (all BL 2.0 questions matched)
for m in matches[:3]: print(m.group(1))  # D1, D2, D3
"
```
Expected: 38 questions matched, IDs: D1, D2, D3 ...

**Secondary**: Also update `_mark_question_done()` in `bl/findings.py` which uses `text.find("\n## Q", ...)` for next-block detection. Replacing with `text.find("\n## ", ...)` would correctly bound BL 2.0 blocks. (Currently harmless — see D4.2 analysis — but confusing.)

## Residual

`_mark_question_done()` at `bl/findings.py:519` uses `text.find("\n## Q", block_start + 1)` which never finds the next BL 2.0 block. However, `block_end = len(text)` and `replace(..., 1)` with count=1 still correctly updates only the target question's `**Status**: PENDING` because it replaces the first occurrence (which is always the target question's status field). This is correct but confusing and should be cleaned up in the same fix.
