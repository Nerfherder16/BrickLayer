# Finding: A13.1 — followup.py bracket tag injection produces [CODE_AUDIT] — invalid BL 2.0 operational mode tag

**Question**: followup.py bracket tag injection uses mode.upper() — produces [CODE_AUDIT] instead of operational mode bracket
**Verdict**: NON_COMPLIANT
**Severity**: Medium
**Mode**: code_audit
**Operational Mode**: audit

## Audit Check

Verified whether the bracket tag injection in `bl/followup.py` lines 206-220 correctly produces valid BL 2.0 operational mode bracket tags when processing sub-questions whose `**Mode**` field is `code_audit`.

Checked:
1. The exact code path for `mode_tag` construction (line 212)
2. The bracket tag convention used in questions.md (all `[AUDIT]`, `[FIX]`, `[DIAGNOSE]` etc.)
3. Whether `[CODE_AUDIT]` is a registered/valid bracket tag in BL 2.0

## Evidence

**Code — `bl/followup.py` lines 206-220:**

```python
# Ensure [MODE] tag is present in the header — LLMs sometimes omit it.
# Extract mode from **Mode**: field and inject it if missing.
header_match = re.match(r"^## \S+(.*)$", seg, re.MULTILINE)
if header_match and "[" not in header_match.group(1):
    mode_match = re.search(r"\*\*Mode\*\*:\s*(\w+)", seg)
    if mode_match:
        mode_tag = f"[{mode_match.group(1).upper()}]"   # line 212 — key line
        seg = re.sub(
            r"^(## \S+) ",
            rf"\1 {mode_tag} ",
            seg,
            count=1,
            flags=re.MULTILINE,
        )
```

**Trace for a `**Mode**: code_audit` sub-question:**
- `mode_match.group(1)` = `"code_audit"`
- `.upper()` = `"CODE_AUDIT"`
- `mode_tag` = `"[CODE_AUDIT]"`
- Injected header becomes: `## A13.2 [CODE_AUDIT] Some sub-question title`

**Valid BL 2.0 bracket tags observed in questions.md:**

| Tag | Usage |
|-----|-------|
| `[AUDIT]` | All A-series questions (A1–A13) |
| `[FIX]` | All F-series questions |
| `[DIAGNOSE]` | All D-series questions |
| `[VERIFY]` | All V-series questions |

`[CODE_AUDIT]` does not appear anywhere in questions.md. It is not a registered or recognised BL 2.0 bracket tag.

The operational mode for Wave 4+ questions is `code_audit` (the runner alias registered as equivalent to `agent`). The *bracket tag* for audit-type questions is `[AUDIT]`, matching the question series letter (`A`). These are two distinct namespaces — runner mode vs. question series bracket — and `mode.upper()` conflates them.

## Impact

When followup.py generates sub-questions for any question with `**Mode**: code_audit` and the LLM omits the bracket tag, the injected header will be:

```
## A13.2 [CODE_AUDIT] <title>
```

instead of the correct:

```
## A13.2 [AUDIT] <title>
```

Downstream effects:
1. **Parser mismatch**: Any parser or downstream tool that validates bracket tags against the known set (`AUDIT`, `FIX`, `DIAGNOSE`, `VERIFY`) will reject or warn on `[CODE_AUDIT]`.
2. **Inconsistent question bank**: Sub-questions injected under Wave 4+ audit entries will carry a non-standard tag, breaking visual consistency and any automated question-series detection.
3. **Scope**: All Wave 4+ sub-questions generated by followup.py for `code_audit`-mode questions are affected — this covers the majority of the bl2 campaign (A13, D13, F13, V13 and any future waves sharing the `code_audit` mode).

The correct fix is to map the runner mode `code_audit` to its canonical bracket tag `AUDIT` rather than blindly uppercasing the mode string. A lookup table (e.g., `{"code_audit": "AUDIT", "agent": "DIAGNOSE"}`) or deriving the tag from the question ID prefix (first letter of the question ID) would both be valid approaches.

## Verdict: NON_COMPLIANT

`mode.upper()` on `"code_audit"` produces `[CODE_AUDIT]`, which is not a valid BL 2.0 operational mode bracket tag. Valid tags are `[AUDIT]`, `[FIX]`, `[DIAGNOSE]`, and `[VERIFY]`. All sub-questions generated by followup.py for Wave 4+ `code_audit`-mode questions will carry an invalid bracket tag.
