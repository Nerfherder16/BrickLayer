# Finding: D4.2 — write_finding() uses question["mode"] (runner tag) not operational_mode for **Mode** header

**Question**: Does `write_finding()` write `**Mode**: agent` (the runner type) in finding headers instead of the actual operational mode (diagnose, fix, monitor, etc.)?
**Verdict**: FAILURE
**Severity**: Medium
**Mode**: diagnose
**Type**: CODE-AUDIT

## Root Cause

`bl/findings.py` `write_finding()` writes the finding markdown header as:

```python
content = f"""# Finding: {qid} — {question["title"]}

**Question**: {question["hypothesis"]}
**Verdict**: {verdict}
**Severity**: {severity}{failure_type_line}
**Mode**: {question["mode"]}
**Type**: {type_label}
```

`question["mode"]` comes from two sources:

1. **Primary BL 2.0 questions** (parsed by `parse_questions()`): `mode_raw = tag_raw.lower()` where `tag_raw = m.group(2)` — the bracket tag. For `## D1 [CORRECTNESS]`, mode = "correctness". For `## F2.1 [FIX]`, mode = "fix". These are not operational modes.

2. **Synthetic heal questions** (`_synthetic_question()`): `q["mode"] = "agent"` (hardcoded, line 79). All synthetic diagnose and fix questions have mode = "agent".

The correct field is `question["operational_mode"]` which holds "diagnose", "fix", "monitor", etc.

## Impact

- All synthetic heal findings (diag_q, fix_q) written with `**Mode**: agent` instead of `**Mode**: diagnose` or `**Mode**: fix`
- Primary BL 2.0 findings written with `**Mode**: correctness`, `**Mode**: fix`, etc. (lowercase bracket tag, not operational mode)
- Human readers see misleading mode labels in finding files
- Any tooling that parses `**Mode**:` from findings gets wrong data

## Fix Specification

**File**: `bl/findings.py`, `write_finding()`, the content f-string

**Change**: Replace `{question["mode"]}` with `{question.get("operational_mode", question["mode"])}` in the `**Mode**:` line.

**Before**:
```python
**Mode**: {question["mode"]}
```

**After**:
```python
**Mode**: {question.get("operational_mode", question["mode"])}
```

**Verification**:
- Synthetic diag finding shows `**Mode**: diagnose`
- Synthetic fix finding shows `**Mode**: fix`
- Primary BL 2.0 question with `operational_mode: diagnose` shows `**Mode**: diagnose`
- `grep -n "operational_mode" bl/findings.py` returns the updated line

**Backward compatibility**: For BL 1.x questions that have no `operational_mode` field, `question.get("operational_mode", question["mode"])` falls back to `question["mode"]` — no regression.
