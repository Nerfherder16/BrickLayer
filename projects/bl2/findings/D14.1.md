# Finding: D14.1 — goal.py wave-index collision — QG-only scan misses BL 2.0 numeric wave IDs

**Question**: Does `_get_next_wave_index()` return the correct wave index for BL 2.0 questions.md?
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: High
**Mode**: code_audit
**Operational Mode**: diagnose

## Root Cause

Two confirmed defects in `bl/goal.py`:

### Bug 1 — `_get_next_wave_index()` line 220

```python
matches = re.findall(r'## QG(\d+)\.\d+', questions_text)
```

The regex matches only `## QG`-prefixed headers. BL 2.0 headers (`## D1.1`, `## F2.3`, `## A14.3`, `## V5.1`) contain no `QG` substring. In any `questions.md` with only BL 2.0 questions, `matches` is always empty and the function returns `1` unconditionally.

**Effect**: Every goal campaign against a BL 2.0 file stamps output `QG1.x`, causing wave collision — all goal-generated questions land in wave 1 regardless of how many waves already exist.

### Bug 2 — `_parse_goal_questions()` line 242

```python
if "## QG" not in block: continue
```

Hard gate that silently drops any LLM output block whose header does not contain the literal `"## QG"`. The LLM prompt (lines 155-184) instructs it to emit `QG` headers, so the gate is internally consistent — but the wave-index being wrong means even well-formed `QG1.x` output collides.

### Dead Code — line 33

`_QUESTION_BLOCK_HEADER` regex is defined but never referenced in the file — dead code with a QG-only pattern.

## Fix Specification

**File**: `C:\Users\trg16\Dev\autosearch\bl\goal.py`
**Line**: 220

Change from:
```python
matches = re.findall(r'## QG(\d+)\.\d+', questions_text)
if not matches:

return max(int(m) for m in matches) + 1
```

Change to:
```python
qg_matches = [int(m) for m in re.findall(r'## QG(\d+)\.\d+', questions_text)]
bl2_matches = [int(m) for m in re.findall(r'## [DFAV](\d+)\.\d+', questions_text)]
all_waves = qg_matches + bl2_matches
if not all_waves:
    return 1
return max(all_waves) + 1
```
