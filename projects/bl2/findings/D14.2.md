# Finding: D14.2 — goal.py focus default uses BL 1.x domain codes D1-D6, forcing wrong bracket tags from LLM

**Question**: Does the goal.py prompt template use BL 2.0 operational mode bracket tags or BL 1.x domain codes?
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: High
**Mode**: code_audit
**Operational Mode**: diagnose

## Root Cause

Two bugs in `bl/goal.py` combine to corrupt every goal-generated question:

### Bug 1 — Line 128, focus default string

```python
focus_str = ", ".join(goal["focus"]) if goal["focus"] else "all domains (D1–D6)"
```

When `goal.md` omits a `Focus:` field, the prompt injects `"all domains (D1–D6)"` — BL 1.x terminology — priming the LLM to think in domain-code terms.

### Bug 2 — Lines 158 and 170, format examples

```
## QG1.1 [D1] Short descriptive title
...
## QG1.2 [D4] Another question title
```

The only two concrete bracket-tag examples the LLM sees are `[D1]` and `[D4]`. By imitation it generates `[D1]`–`[D6]` on every question.

## What parse_questions() does with them

In `questions.py` lines 37, 53-58:
- `mode_raw = tag_raw.lower()` → `"d1"`, `"d4"` — values no BL 2.0 dispatcher recognises
- Neither `_CODE_AUDIT_TAGS` nor `_BEHAVIORAL_TAGS` contains `"d1"`–`"d6"`, so `question_type` silently defaults to `"behavioral"` for all goal-generated questions
- The prompt template also omits `**Operational Mode**` from example blocks, so every question gets `operational_mode = "diagnose"` (hardcoded default at line 78 of questions.py)

**The failure is silent** — no parse error raised; questions are enqueued with corrupted `mode` values.

## Fix Specification

**File**: `C:\Users\trg16\Dev\autosearch\bl\goal.py`

**Line 128**: Replace:
```python
focus_str = ", ".join(goal["focus"]) if goal["focus"] else "all domains (D1–D6)"
```
With:
```python
focus_str = ", ".join(goal["focus"]) if goal["focus"] else "all operational modes (DIAGNOSE, FIX, AUDIT, VALIDATE)"
```

**Lines 158/170**: Replace `[D1]`/`[D4]` examples with `[DIAGNOSE]`/`[AUDIT]` and add `**Operational Mode**:` field to each example block:

```
## QG1.1 [DIAGNOSE] Short descriptive title
**Operational Mode**: diagnose
...
## QG1.2 [AUDIT] Another question title
**Operational Mode**: audit
```

**Verification**:
```python
python -c "from bl.goal import _build_prompt; p = _build_prompt({'goal':'test','target':'','focus':[],'max_questions':2,'context':''}, ''); assert '[D1]' not in p and '[DIAGNOSE]' in p and 'Operational Mode' in p, 'FAIL'; print('PASS')"
```

**Risk**: `goal.md` files that explicitly set `Focus: D1, D3` will still inject BL 1.x strings via user input (unaffected by this fix). Any test asserting the prompt contains `[D1]` or `[D4]` must be updated.
