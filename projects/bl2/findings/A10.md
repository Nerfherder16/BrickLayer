# Finding: A10 — Heal intermediate finding IDs follow pattern (checklist item A10)

**Question**: Synthetic heal finding IDs must follow `{qid}_heal{n}_{type}` pattern (where type is `diag` or `fix`). The pattern must not collide with real question IDs.
**Verdict**: COMPLIANT
**Severity**: Info
**Mode**: agent

## Evidence

`bl/healloop.py` line 218:
```python
diag_q["id"] = f"{original_qid}_heal{cycle}_diag"
```

`bl/healloop.py` line 275:
```python
fix_q["id"] = f"{original_qid}_heal{cycle}_fix"
```

Both follow the `{qid}_heal{n}_{type}` pattern where type is `diag` or `fix`.

**Collision risk analysis**: Real question IDs in BL 2.0 campaigns use prefixes like `Q`, `D`, `A`, `R`, `M`, `F`, `E`, `V`, `P` (from the BL 2.0 question-designer convention). Synthetic IDs are derived from the original qid with `_heal{n}_` infix. For a collision to occur, a real question ID would need to contain `_heal` followed by a digit — which is not a valid BL 2.0 question ID format. No collision risk exists.

**Note on D3**: The `_synthetic_question()` function at line 75 temporarily generates `{qid}_heal{cycle}_diagnose` (not `_diag`) for the agent's context, which is then overwritten to `_diag` at line 218. The FINAL ID written to disk is always the correct `_diag` pattern (see D3 finding for details on the temporary mismatch).

## Mitigation Recommendation

No action required for the ID pattern itself. See D3 for the temporary mismatch during agent execution.
