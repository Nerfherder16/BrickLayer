# Finding: V20.1 — _is_bl2_diag_row() edge-case handling is correct for all results.tsv read paths

**Question**: Does `_is_bl2_diag_row()` in crucible.py correctly handle edge cases — empty lines, lines with fewer than 3 tab-separated columns, and rows where parts[2] contains trailing whitespace?
**Verdict**: COMPLIANT
**Severity**: Info

## Evidence

**`_is_bl2_diag_row()` implementation** (`bl/crucible.py` lines 383-391):
```python
def _is_bl2_diag_row(ln: str) -> bool:
    parts = ln.split("\t")
    if len(parts) >= 3 and parts[0] == "N/A":
        return parts[1].startswith("D") and bool(
            re.search(
                r"^(DIAGNOSIS_COMPLETE|HEALTHY|FAILURE|INCONCLUSIVE)$", parts[2]
            )
        )
    return False
```

**Edge case analysis:**

1. **Empty lines**: `splitlines()` at line 395 is the only read path. `"".split("\t")` → `[""]` → `len(parts) == 1 < 3` → guard fails → returns `False`. ✅

2. **Short rows (< 3 cols)**: `len(parts) >= 3` guard at line 385 handles all short/malformed rows explicitly. ✅

3. **Trailing whitespace in parts[2]**: `splitlines()` strips `\n` from each line. A row with trailing tab (e.g., `"N/A\tD1\tDIAGNOSIS_COMPLETE\t..."`) produces `parts[2] == "DIAGNOSIS_COMPLETE"` — the regex `^(...)$` anchors match this correctly. If parts[2] had inline spaces (impossible for valid verdict tokens), the anchored regex would reject it cleanly. ✅

4. **No other call site**: `_is_bl2_diag_row()` is a nested function — not accessible outside `_score_diagnose_analyst()`. All callers pass lines from `splitlines()`. ✅

5. **BOM at file start**: On Windows, `read_text(encoding="utf-8", errors="replace")` reads raw UTF-8. If a BOM is present, the first line's `parts[0]` would be `"\ufeffN/A"`, not `"N/A"` — the equality check would return `False`, silently excluding the first row. **This is a theoretical risk** but not present in the current results.tsv (confirmed: file written by `update_results_tsv()` which uses `open(..., "a", encoding="utf-8")` without BOM). ✅

## Summary

All edge cases handled correctly for the current and expected future results.tsv format. The one theoretical BOM risk is academic — the file is written by `update_results_tsv()` which never emits a BOM. No silent exclusions of valid BL 2.0 D-prefix rows occur.
