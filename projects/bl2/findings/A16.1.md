# Finding: A16.1 — campaign.py pending-list refresh inside the loop is dead code for execution

**Question**: Is the pending-list refresh inside `run_campaign()`'s for-loop (lines 582-584) dead code for execution purposes — with injected questions silently skipped in the current run and the progress display denominator potentially diverging?
**Verdict**: NON_COMPLIANT
**Severity**: Medium

## Evidence

**`run_campaign()` loop structure (lines 563-601)**:

```python
pending = [q for q in questions if q["status"] == "PENDING"]   # line 568 — pre-loop list
pending = _preflight_mode_check(pending)                         # line 569
...
questions_done = 0

for i, question in enumerate(pending, 1):    # line 579 — iterator bound to original list
    check_sentinels()

    if questions_done > 0:
        refreshed = parse_questions()
        pending = [q for q in refreshed if q["status"] == "PENDING"]   # line 584 — rebind

    print(f"\n[{i}/{len(pending)}] {question['id']} ...", ...)    # line 587 — uses rebound pending
    run_and_record(question)
    questions_done += 1
```

**Key behavior confirmed**:

1. **Iterator is bound to the original list**: `enumerate(pending, 1)` creates an iterator over the list object that `pending` pointed to at line 579. In Python, `enumerate` iterates the original object — a subsequent rebind of the name `pending` to a new list has no effect on the in-progress iterator. `question` always comes from the original pre-loop list. ✓ (NON_COMPLIANT confirmed)

2. **Injected questions are not executed**: `check_sentinels()` at line 580 calls `_inject_override_questions()` which may append `{qid}.R` re-exam blocks to questions.md. These are written to disk but are never in the pre-loop `pending` list. They will not be visited by the for-loop iterator in the current campaign invocation. ✓

3. **Progress display denominator diverges after injection**: `len(pending)` on line 587 uses the REBOUND list (post-refresh), so after any question injection, the denominator reflects the new `questions.md` count while the numerator `i` tracks position in the ORIGINAL list. If injection adds 3 questions, the display might show `[5/8]` when only 7 questions were in the original pending list — creating a false impression of remaining work. ✓

4. **The refresh is misleading dead code for execution**: The comment `# re-parse after reactivation` appears elsewhere (line 567), suggesting the refresh was intentional in some prior context. For execution purposes in the current campaign run, it is purely cosmetic (affects only the progress display denominator).

**Intentionality assessment**: The non-execution of injected questions appears to be an artifact of the BL 1.x→2.0 migration. The original BL 1.x loop may have used a while-loop or re-fetched pending at each iteration. The current for-loop with an enum rebind is the likely result of a refactor that preserved the refresh call without preserving its semantic effect. There is no comment or documentation indicating that skipping injected questions in the current run is intentional design.

**Practical impact**: Medium — injected questions ARE picked up on the next `python -m bl.campaign` invocation. The bug doesn't permanently lose questions, it only delays them by one campaign run. However, the misleading `[{i}/{len(pending)}]` display may cause operators to misread remaining work.

## Mitigation Recommendation

**Option A (minimal fix)**: Remove the rebind dead code (lines 582-584) and capture `total_pending` before the loop for a stable denominator:
```python
total_pending = len(pending)
for i, question in enumerate(pending, 1):
    ...
    print(f"\n[{i}/{total_pending}] ...")
```

**Option B (dynamic pickup)**: Replace the for-loop with a while-loop that re-fetches pending at each iteration, enabling injected questions to be picked up in the same run. Higher complexity, higher correctness.

Option A is recommended — simpler, eliminates the misleading rebind, and the one-run delay for injected questions is acceptable behavior.
