# Finding: D16.1 — healloop.py exhausted-loop leaves results.tsv with stale FAILURE verdict

**Question**: Does `run_heal_loop()` in `bl/healloop.py` write back to results.tsv for the original question when the heal loop exhausts all cycles without a FIXED verdict?
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: High

## Evidence

Traced `bl/healloop.py` lines 288-343 (the exhausted-loop path):

**FIXED path (lines 288-306)**: When `fix_verdict == "FIXED"`, `update_results_tsv(original_qid, "FIXED", ...)` is called at lines 296-300. ✓

**Exhausted path (lines 332-343)**:
```python
_append_heal_note(
    finding_path,
    last_cycle,
    "EXHAUSTED",
    f"Self-healing exhausted {last_cycle} cycle(s) — human intervention required. ...",
)
print(f"[heal-loop] {original_qid} exhausted {last_cycle} cycle(s) — still unresolved", ...)
return current_result
```

No `update_results_tsv(original_qid, ...)` call exists anywhere in the exhausted path. The function returns `current_result` without writing the final HEAL_EXHAUSTED state.

**campaign.py caller (lines 128-140)**: After `run_heal_loop` returns, the caller only does an identity check (`healed_result is not result`) and propagates `result = healed_result`. It does NOT call `update_results_tsv` for the original `qid` after receiving an exhausted result.

**Consequence**: The original question (e.g., D16.x) retains its pre-heal verdict (FAILURE or DIAGNOSIS_COMPLETE) in results.tsv permanently, even though the heal loop ran multiple diagnose/fix sub-cycles and appended an EXHAUSTED note to the finding file. The synthesizer, crucible, and agent_db all treat results.tsv as the authoritative verdict store — they will read the original FAILURE and score the question as a clean failure, potentially triggering redundant followup questions and incorrect agent performance penalties.

## Fix Specification

**Target file**: `bl/healloop.py`
**Target location**: Lines 332-343 — after `_append_heal_note(..., "EXHAUSTED", ...)` and before `return current_result`
**Concrete edit**: Add `update_results_tsv(original_qid, "HEAL_EXHAUSTED", f"Self-healing exhausted {last_cycle} cycle(s) — human intervention required.", None)` immediately after the `_append_heal_note` call in the exhausted path.
**Verification command**: Inspect `bl/healloop.py` lines 332-345; confirm `update_results_tsv(original_qid, "HEAL_EXHAUSTED", ...)` is present after `_append_heal_note`. Also confirm `"HEAL_EXHAUSTED"` is added to the appropriate frozenset in `bl/agent_db.py` (likely `_TERMINAL_VERDICTS` or `_PARKED_STATUSES` equivalent).

## Mitigation Recommendation

1. Add `update_results_tsv(original_qid, "HEAL_EXHAUSTED", ...)` at the exhausted-loop exit point in `healloop.py`.
2. Add `"HEAL_EXHAUSTED"` to the terminal verdict frozensets in `agent_db.py` so it doesn't re-trigger the heal loop on next campaign run.
3. Add `"HEAL_EXHAUSTED"` to `_PRESERVE_AS_IS` in `findings.py` and to `sync_status_from_results()` in `questions.py` so the status is preserved across syncs.
