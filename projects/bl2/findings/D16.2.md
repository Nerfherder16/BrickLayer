# Finding: D16.2 — campaign.py peer-reviewer spawned unconditionally for code_audit questions

**Question**: Does `run_campaign()` in `bl/campaign.py` spawn peer-reviewer for all question modes including `code_audit`, producing vacuous verdicts that inject spurious re-exam questions?
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: High

## Evidence

**campaign.py lines 593-601**: `_spawn_agent_background("peer-reviewer", ...)` is unconditionally inside the `for i, question in enumerate(pending, 1)` loop with NO guard on `question["mode"]`:

```python
_spawn_agent_background(
    "peer-reviewer",
    f"primary_finding={cfg.findings_dir / (question['id'] + '.md')}\n"
    f"target_git={cfg.project_root.parent}\n"
    f"agents_dir={cfg.agents_dir}\n\n"
    f"Re-run the original test for {question['id']}, review the fix code, "
    f"and append a ## Peer Review section with verdict "
    f"CONFIRMED | CONCERNS | OVERRIDE to the finding file.",
)
```

This runs for every question including `code_audit` mode (all Wave 16+ questions: D16.1, D16.2, D16.3, V16.1, A16.1).

**peer-reviewer.md Step 2**: "Extract the `**Test**:` command from the corresponding question in `questions.md`. Run it against the current state of target_git (post-fix)."

For `code_audit` questions, the `**Test**:` field contains prose such as:
> "Read `bl/healloop.py` lines 288-343. Trace the code path when `fix_verdict != 'FIXED'`..."

This is NOT an executable command. The peer-reviewer will fail to run it and likely emit CONCERNS or OVERRIDE as a safe default.

**peer-reviewer.md trigger/when-to-run fields**: "fix wave completes and at least one agent finding was written" and "Invoked after every fix wave (mode = agent findings)." The agent is explicitly designed for `mode = agent` questions, not `code_audit`.

**`_inject_override_questions()` lines 398-437**: Scans ALL `*.md` findings (no mode filter) for `## Peer Review.*?\*\*Verdict\*\*:\s*OVERRIDE` pattern. Any OVERRIDE from a code_audit question appends a `{qid}.R` re-exam block with `**Mode**: agent` and `**Test**: Re-run the original test command from {qid}` — which for code_audit questions is still a prose instruction, not an executable command. The re-exam question is also unresolvable.

**Net effect**: Every code_audit question in Wave 16+ (and all future waves) spawns a peer-reviewer that cannot execute its test and will emit CONCERNS/OVERRIDE, injecting an unresolvable `.R` re-exam question into questions.md, growing the question bank indefinitely.

## Fix Specification

**Target file**: `bl/campaign.py`
**Target location**: Lines 593-601 — `_spawn_agent_background("peer-reviewer", ...)` call inside the for-loop
**Concrete edit**: Wrap the `_spawn_agent_background("peer-reviewer", ...)` call with a mode guard:
```python
if question.get("mode") not in ("code_audit",):
    _spawn_agent_background(
        "peer-reviewer",
        ...
    )
```
**Verification command**: `grep -n "peer-reviewer\|code_audit" bl/campaign.py` — confirm the peer-reviewer spawn is guarded by `if question.get("mode") not in ("code_audit",)`.

## Mitigation Recommendation

1. Add mode guard to the peer-reviewer spawn in `run_campaign()` — skip for `code_audit`, `simulate`, and any other non-executable modes.
2. Optionally update `peer-reviewer.md` trigger conditions to explicitly exclude `code_audit` so the agent itself also checks its mode context.
3. Existing `.R` re-exam questions already injected for code_audit questions should be reviewed and removed from questions.md if they reference prose-only test commands.

## Suggested Follow-ups
- Are there other background agent spawns in `run_campaign()` that are also mode-unconditional and should be guarded similarly (e.g., forge-check, hypothesis-generator)?
- Should `_inject_override_questions()` also filter out OVERRIDE verdicts from code_audit mode questions to prevent the secondary injection path from triggering?
