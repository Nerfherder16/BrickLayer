# Finding: D10.1 — hypothesis.py _parse_question_blocks() rejects all BL 2.0 LLM output

**Question**: Does hypothesis.py _parse_question_blocks() silently reject BL 2.0 format question blocks from LLM output?
**Verdict**: FAILURE
**Severity**: High
**Mode**: code_audit
**Operational Mode**: diagnose

## Root Cause

### _parse_question_blocks() line 167

```python
if not re.search(rf"## Q{next_wave}\.\d+", block):
    continue
```

For wave 10, this becomes `## Q10\.\d+`. Any block produced by the LLM with a BL 2.0 format header such as:
- `## D10.1 [CORRECTNESS] title`
- `## F10.1 [FIX] title`
- `## A10.1 [COMPLIANCE] title`
- `## V10.1 [VALIDATE] title`

will NOT match this pattern and will be silently skipped. `_parse_question_blocks()` returns `[]`.

### Prompt template (line 109, 115)

```python
5. Question IDs must start at Q{next_wave}.1 and increment
...
## Q{next_wave}.1 [CATEGORY] Short descriptive title
```

The LLM is explicitly instructed to produce Q-prefix IDs. Even if we fix `_parse_question_blocks()`, the LLM will keep generating Q-format IDs (e.g., `## Q10.1`) when called via `_ollama_hypothesize()`.

### Impact

`generate_hypotheses()` calls `_parse_question_blocks()` on the LLM response. If all blocks are rejected, it returns 0 questions and no new hypotheses are appended to questions.md. This makes hypothesis generation completely non-functional for any BL 2.0 campaign (where BL 2.0 format IDs are expected).

Note: BL 1.x campaigns are unaffected — the LLM produces Q-format IDs which match.

### Interaction with F9.1

F9.1 fixed `_QUESTION_BLOCK_HEADER` so that BL 2.0 IDs are correctly read FROM questions.md. But the complementary path — generating NEW BL 2.0 format questions via Ollama — remains broken at two points:
1. Prompt template instructs Q-format IDs
2. Validation filter enforces Q-format IDs

## Verdict: FAILURE

Both `_parse_question_blocks()` (line 167) and the LLM prompt template (line 109, 115) hardcode Q-prefix IDs, making hypothesis generation silently return zero questions for BL 2.0 campaigns. Fix target: F10.1.
