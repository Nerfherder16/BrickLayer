# Finding: A12 — BL 1.x backwards compatibility — FIX_LOOP vs HEAL_LOOP independence (checklist item A12)

**Question**: `BRICKLAYER_FIX_LOOP` (BL 1.x) and `BRICKLAYER_HEAL_LOOP` (BL 2.0) must be independent environment variables. Enabling one must not disable or override the other.
**Verdict**: COMPLIANT
**Severity**: Info
**Mode**: agent

## Evidence

`bl/campaign.py` lines 113-138:
```python
# C-06: fix loop — attempt to repair FAILURE automatically (opt-in, BL 1.x)
if (
    result.get("verdict") == "FAILURE"
    and os.environ.get("BRICKLAYER_FIX_LOOP") == "1"
):
    from bl.fixloop import run_fix_loop
    ...
    result = fixed_result

# BL 2.0: self-healing loop
if (
    result.get("verdict") in ("FAILURE", "DIAGNOSIS_COMPLETE")
    and os.environ.get("BRICKLAYER_HEAL_LOOP") == "1"
):
    from bl.healloop import run_heal_loop
    ...
```

Both are independent `if` blocks (not `if/elif`). Setting `BRICKLAYER_FIX_LOOP=1` does not disable `BRICKLAYER_HEAL_LOOP`, and vice versa. They can theoretically both be enabled simultaneously (though unusual — if FIX_LOOP converts FAILURE → HEALTHY, the HEAL_LOOP block won't trigger since it requires FAILURE or DIAGNOSIS_COMPLETE).

One subtle interaction: if BL 1.x FIX_LOOP converts a FAILURE to HEALTHY (line 122-126), then the subsequent HEAL_LOOP check (line 129-130) would find `result.get("verdict") in ("FAILURE", "DIAGNOSIS_COMPLETE")` to be False, so HEAL_LOOP wouldn't run. This is correct behavior — if BL 1.x already fixed it, BL 2.0 heal loop should not re-run.

## Mitigation Recommendation

No action required. The independence is correctly implemented.
