## Evidence

I have all the information needed. Here is the full analysis:

**Write order**: Qdrant first → Neo4j second (both paths)

**Rollback logic (single store, lines 260–267)**:
```python
await qdrant.store(memory, embedding)        # step 1 — Qdrant
try:
    await neo4j.create_memory_node(memory)   # step 2 — Neo4j
except Exception as neo4j_err:
    logger.error(...)
    await qdrant.delete(memory.id)           # compensating rollback ← UNGUARDED
    raise
```

**Rollback logic (batch store, lines 1246–1263)**:  
Same pattern — compensating `qdrant.delete(memory.id)` is called bare, no `try/except`.

**Key findings**:
1. **Rollback EXISTS** — compensating Qdrant delete fires when Neo4j write fails. The mechanism is architecturally correct.
2. **Compensating delete is unguarded** — if `qdrant.delete()` itself throws (Qdrant transiently down, network blip), the original `neo4j_err` is masked by the new exception, AND the Qdrant vector is left orphaned with no recovery path.
3. **Hypothesis inversion** — the hypothesis says "orphaned Neo4j nodes due to Qdrant failure." The actual risk is the reverse: orphaned **Qdrant vectors** when the compensating delete fails. Neo4j is written second so it can't be orphaned by a Qdrant failure.
4. **Batch path** has identical gap — `qdrant.delete(memory.id)` at line 1253 is also unguarded.

**Verdict**: WARNING — rollback logic is present and correctly structured for the primary failure path, but the compensating operation itself has no error handling, making full recovery conditional on Qdrant availability at rollback time.
