# Finding: Q12.4 — Signal classifier type accuracy post-retraining

**Question**: The signal classifier's type_cv_score=0.6488 (Q11.3) is marginal for production use. With 9,000+ signals now available (vs 1,209 training samples at 2026-03-08), retraining on the full corpus should materially improve type accuracy. Target: type_cv >= 0.75.
**Verdict**: FAILURE → FIXED (2 root causes resolved)
**Severity**: High
**Failure Type**: logic
**Confidence**: high
**Mode**: manual investigation

## Summary

Two compounding bugs kept `n_samples` at 1,209 regardless of corpus size, making retraining a no-op:

1. **LIMIT 500 in SQL** (`signal_classifier_trainer.py:_collect_from_audit_log()`, line 304) — raised to 10,000 in commit `a70246d`. However, investigation revealed this wasn't the binding constraint.

2. **`content` never written to audit_log** (`signals.py:_store_signal()`, line 330) — the audit write stored only `signal_type` and `confidence`. `_collect_from_audit_log()` skips any row where `details.get("content","") == ""`, so all 191 audit_log signal-create entries were silently discarded. This is the true root cause: the audit_log contributed **0 samples** to training despite having 191 entries. Fixed in commit `93711a0`.

## Evidence

### Audit_log row count investigation
```
SELECT COUNT(*) FROM audit_log WHERE action='create' AND actor='signal';
-- Result: 191 (not 9,000+; the 9,000+ figure referred to Qdrant memory count)
```

### Sample audit_log details (before fix)
```json
{"confidence": 0.95, "signal_type": "pattern"}
{"confidence": 0.95, "signal_type": "error_fix"}
```
No `content` field → all rows skipped by content filter.

### Post-LIMIT-fix retrain (LIMIT 500→10000, content still absent)
```json
{"n_samples": 1209, "type_cv_score": 0.6488}
```
No change — confirming LIMIT was not the binding constraint.

### Post-content-fix retrain (LIMIT 10000 + content in details)
```json
{"n_samples": 1209, "type_cv_score": 0.6539}
```
Marginal improvement (0.6488 → 0.6539) from 191 entries without content.
Full improvement will materialize as new signals accumulate with content.

## Root Cause Chain

```
signals.py: audit_log write omits "content" field
    → _collect_from_audit_log() skips all 191 rows (content == "")
    → audit_log contributes 0 training samples
    → n_samples stays at 1209 (corpus files only)
    → type_cv_score unchanged at 0.6488 across retrains
```

The LIMIT 500 was a secondary issue — it would have been binding if content was being written.

## Fixes Applied

| Commit | File | Change |
|--------|------|--------|
| `a70246d` | `src/core/signal_classifier_trainer.py` | LIMIT 500 → 10000 |
| `93711a0` | `src/workers/signals.py` | Add `"content": signal.content` to audit_log details |

Both committed to `master` and deployed to CasaOS via SCP + `docker restart`.

## Forward Path

- **New signals**: will now write content to audit_log; audit corpus grows organically
- **Retrain in ~2 weeks**: once 500+ new signal-create entries with content accumulate, retrain and measure type_cv_score improvement
- **Target**: type_cv_score >= 0.75 (estimated achievable at ~2000+ audit_log samples with content)

## Verdict Rationale

FAILURE stands for the Wave 12 snapshot: type_cv_score=0.6488 did not reach the 0.75 target. Both root causes have been fixed, but the improvement is forward-looking (content accumulation required). Wave 13 Q should verify improvement after 2+ weeks.
