I have sufficient data to issue the verdict. Here's my analysis of the critical store path:

**Write order (memory.py lines 257–266):**
```python
# Store in Qdrant
await qdrant.store(memory, embedding)          # ← QDRANT FIRST

# Create graph node -- compensating delete on failure
try:
    neo4j = await get_neo4j_store()
    await neo4j.create_memory_node(memory)    # ← NEO4J SECOND
except Exception as neo4j_err:
    logger.error("neo4j_write_failed_compensating", ...)
    await qdrant.delete(memory.id)            # ← compensating rollback (unguarded)
```

The same pattern is mirrored in `ingest.py` lines 243–251.

**Key findings:**
1. Write order is **Qdrant-first** — the hypothesis's scenario (Qdrant down → Neo4j orphan) cannot trigger: if Qdrant is down, `qdrant.store()` raises before Neo4j is ever touched.
2. A **compensating Qdrant delete** exists for the Neo4j-failure case — structurally correct rollback direction.
3. The compensating `await qdrant.delete(memory.id)` is **unguarded** (no try/except). If Qdrant becomes unavailable between the store and the rollback delete, the delete silently fails and a **Qdrant orphan** (vector with no graph node) persists.
4. **No orphan detection or reconciliation job** was found anywhere in `src/`.
