# Finding: Q11.3 — ML pipeline observability gap

**Question**: Is ML pipeline health (reranker, signal classifier) visible in the health dashboard?
**Verdict**: WARNING
**Severity**: Medium
**Confidence**: high
**Mode**: manual
**Target**: GET /admin/health/dashboard + /admin/ml/reranker-status + /admin/ml/signal-classifier-status

## Summary

Two dedicated ML health endpoints exist and return rich status data. However, the main health dashboard (`GET /admin/health/dashboard`) has no ML section — an operator monitoring the dashboard would be blind to ML degradation (stale model, low CV score, training data skew). The reranker was silently broken for 6 days before Q10.5 detected it; a dashboard ML panel would have surfaced this immediately.

## Issues Identified

### Issue 1: No ML panel in health dashboard (CRITICAL OBSERVATION)
The dashboard is the primary ops monitoring surface. ML health is only accessible via two separate endpoints that aren't linked from the dashboard. An operator doing a routine health check would miss:
- Stale model (last trained >7 days ago)
- Low CV score degradation
- Class imbalance shifts

### Issue 2: Signal classifier is stale
`type_cv_score=0.6488` is marginal (below 0.70 threshold for production confidence). Trained 2026-03-08 — 6 days without retraining while 7,893 new signals have been processed. Binary classification is healthy (0.943) but type classification accuracy is borderline.

### Issue 3: Retraining frequency unclear
No evidence of scheduled signal classifier retraining (unlike reranker which has a cron trigger). The classifier may depend on manual triggers only.

## Recommended Fix

Add an `ml` section to `GET /admin/health/dashboard` response aggregating both model statuses:

```json
"ml": {
  "reranker": {
    "status": "trained",
    "cv_score": 0.9541,
    "trained_at": "2026-03-14T20:12:04Z",
    "n_samples": 9102,
    "staleness_hours": 4.5,
    "health": "HEALTHY"
  },
  "signal_classifier": {
    "status": "trained",
    "binary_cv_score": 0.943,
    "type_cv_score": 0.6488,
    "trained_at": "2026-03-08T04:00:35Z",
    "n_samples": 1209,
    "staleness_hours": 160,
    "health": "WARNING"
  }
}
```

Thresholds:
- HEALTHY: cv_score >= 0.85, staleness < 168h (7 days)
- WARNING: cv_score 0.70–0.84 OR staleness 168–336h
- FAILURE: cv_score < 0.70 OR staleness > 336h (14 days)

## Verdict Rationale

The dedicated endpoints exist and are functional — this is not a data gap, it's a surfacing gap. But the 6-day reranker silent failure (Q10.5) proves that "data exists somewhere" is insufficient. Operators need ML health in the primary monitoring surface. WARNING.
