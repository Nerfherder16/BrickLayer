## Verdict Threshold

- FAILURE: mean importance < 0.35 for any memory type
- WARNING: p25 < 0.30 for any type OR >30% of memories at exactly same value
- HEALTHY: mean >= 0.40 across all types AND p25 >= 0.30

## Root Cause (2026-03-14)

`SIGNAL_DETECTION_SYSTEM` in `src/core/prompts.py` had no importance rubric — just `"importance": 1-10` with no guidance. `OBSERVER_SYSTEM` (file edits) already had a 5-tier rubric. The LLM (qwen3:14b) defaulted to low scores (1-3) for technical domain signals when unconstrained.

## Fix Applied (commit d6eff71)

Added structured 5-tier importance rubric to `SIGNAL_DETECTION_SYSTEM` matching `OBSERVER_SYSTEM`, with explicit floor: *"Technical facts (domain=documentation/api/frontend/database/devops/infrastructure) default to 5+ unless they are trivial details."*

### Live validation post-fix
```
domain=api,      importance=5  (was 1-3 range) — FastAPI endpoint schema fact
domain=database, importance=7  (was 1-3 range) — PostgreSQL index decision
```

### Test coverage
7 unit tests added in `tests/core/test_signal_detection_rubric.py` — all passing.

### Impact on corpus
New memories stored after this fix will score 0.5+ for technical content. The existing 5,695-memory corpus is unaffected — distribution shift requires either a backfill pass or natural decay/rehab cycle over time.

## Resolved

- Q10.3.1: `/search/browse` datetime bug fixed (6 patches across retrieval.py, reranker.py, search.py) — no longer contributing to this WARNING.
- Q10.3 root cause: `SIGNAL_DETECTION_SYSTEM` rubric added — commit d6eff71; forward-fix deployed 2026-03-14.
