# Finding: Q10.3.1 — /search/browse datetime fix verification

**Question**: Is the /search/browse datetime bug in a different module than decay.py, and is it causing silent search result degradation?
**Verdict**: HEALTHY (fixed)
**Severity**: High (was causing 500 on all browse calls)
**Confidence**: high
**Mode**: agent
**Target**: POST /search/browse

## Summary

The `/search/browse` endpoint was crashing for ALL requests due to naive/aware datetime mismatches in 4 locations across 3 files. Root cause: pre-migration records were stored without timezone info in `last_accessed` and `created_at` fields. All fixed; endpoint now returns 200 with results.

## Root Cause Chain

The bug was the same class as `ac0454b` (decay.py fix) but affected 4 code paths:

| Location | Field | Fix commit |
|----------|-------|------------|
| `src/api/routes/search.py:258` | `valid_at` | `67a8e2c` |
| `src/core/retrieval.py:1166` | `last_accessed` (recency boost) | `927cd67` |
| `src/core/retrieval.py:1526` | `last_accessed` (access reinforcement) | `f15246e` |
| `src/core/reranker.py:54-55` | `last_accessed` + `created_at` (feature extraction) | `bb181cb` |
| `src/core/reranker.py:147` | `created_at` (recency boost) | `bb181cb` |

Secondary fix: `src/api/routes/search.py:_uid()` — admin users were filtered to `user_id=2` but all hook-stored memories have `user_id=0` (system), yielding 0 results. Fixed: admins pass `user_id=None` → no filter → see all memories. Commit: `e6f9bb8`.

## Fix Pattern

All occurrences follow the same normalization idiom:
```python
field_tz = field if field.tzinfo else field.replace(tzinfo=UTC)
delta = datetime.now(UTC) - field_tz
```
