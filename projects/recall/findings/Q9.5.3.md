## Evidence

**Summary of findings:**

1. **`hygiene.py`** — Qdrant+Postgres only. Archives stale memories by importance/age. No Neo4j orphan detection or cleanup.

2. **`workers/main.py` cron schedule** — No cron job for orphan detection. Scheduled jobs: consolidation, decay, hygiene, pattern extraction, dream consolidation, ML retrain, profile drift, auto-tuner, cognitive distiller, dark memory, contradiction backfill. None target Neo4j orphan nodes.

4. **`health.py` orphan hub detection** — Reports high-centrality low-importance nodes as `orphan_hub` conflicts. Read-only diagnostics, no cleanup.

## Verdict Threshold

- FAILURE: No orphan detection or cleanup mechanism
- WARNING: Inadequate or infrequent orphan detection
- HEALTHY: Regular and effective orphan detection and cleanup process

## Mitigation Applied

Two gaps identified by original WARNING were both addressed in commit `ee1203b`:

1. **No scheduled cleanup** → `src/workers/reconcile.py` created with `run_reconcile()` function. Wired into `WorkerSettings.functions` and `WorkerSettings.cron_jobs` (weekly Sunday 5:30am). Confirmed in `workers/main.py` lines 26, 344, 424.

2. **Neo4j orphan deletion missing in repair mode** → `reconcile.py` explicitly calls `neo4j.delete_memory(memory_id)` for each `neo4j_orphan` (in Neo4j but not Qdrant). The original `/reconcile` endpoint only created missing Neo4j nodes; the worker now also deletes stale ones.

Additional coverage: importance mismatches and `superseded_by` drift are also synced Qdrant→Neo4j in the same worker run.
