# Finding: Q5.1 — Recall tag namespacing is consistent across all three agents; planner's broad query is the only minor gap

**Question**: Are Mortar's Recall checkpoint stores consistent with how agent-auditor and planner query them?
**Verdict**: HEALTHY
**Severity**: Low

## Evidence

### Tag convention audit

| Agent | Store tags | Query tags |
|-------|-----------|------------|
| Mortar (store) | `["bricklayer", "agent:mortar", "type:checkpoint"]` | — |
| Mortar (query) | — | `tags=["agent:mortar"]` |
| agent-auditor (store) | `["bricklayer", "agent:agent-auditor", "type:audit-report"]` | — |
| agent-auditor (query) | — | `tags=["agent:agent-auditor"]` |
| planner (store) | `["bricklayer", "agent:planner", "type:campaign-plan"]` | — |
| planner (query) | — | no tag filter (broad query) |
| peer-reviewer (OVERRIDE store) | `["bricklayer", "agent:peer-reviewer", "type:override"]` | — |
| peer-reviewer (CONFIRMED store) | `["bricklayer", "agent:peer-reviewer", "type:confirmed"]` | — |

### Cross-agent query compatibility

**Mortar at campaign start**: queries `tags=["agent:mortar"]` → returns ONLY Mortar's own checkpoints. ✓ Correct — Mortar retrieves its own checkpoint to resume from the last known wave state.

**agent-auditor at audit start**: queries `tags=["agent:agent-auditor"]` → returns ONLY agent-auditor's prior audit reports. ✓ Correct — used to detect score trajectory changes over time.

**planner at campaign init**: queries with NO tag filter:
```
recall_search(query="bricklayer campaign failure critical finding", domain="{project}-bricklayer")
recall_search(query="bricklayer override inconclusive warning", domain="{project}-bricklayer")
```
Returns ALL stores in the project domain. This includes Mortar checkpoints, agent-auditor reports, peer-reviewer OVERRIDE records, and prior planner plans. Most of this is relevant context for a campaign planner (it wants to know about failures and OVERRIDEs to list as "known landmines").

**Not a gap**: planner's broad query is intentional — it specifically wants cross-agent context. However, it may also surface Mortar wave-progress checkpoints ("wave 3: 15 done, 5 pending") as if they were failure findings. Semantic search should rank failure-related content higher than wave-progress content, but it's not guaranteed.

### Domain naming consistency

All agents use the same domain naming convention: `{project}-bricklayer`. Consistent substitution of the project name is required. If any agent uses a different project name suffix (e.g., "bricklayer" vs "{project}-bricklayer"), cross-agent recall queries fail. This is a documentation gap — no agent specifies where the project name literal comes from or how to canonicalize it. If the project directory is named "agent-meta", the domain should be "agent-meta-bricklayer" — but there's no enforcement.

### The hypothesis claim — tested

Hypothesis: "The tag namespacing may cause cross-agent queries to miss relevant stores."

Test:
- Mortar checkpoint is stored with `tags=["bricklayer", "agent:mortar", "type:checkpoint"]`
- Mortar query uses `tags=["agent:mortar"]`
- Does "agent:mortar" appear in the stored tags? **YES** → query returns the checkpoint ✓

- agent-auditor stores with `tags=["bricklayer", "agent:agent-auditor", "type:audit-report"]`
- Query uses `tags=["agent:agent-auditor"]`
- **YES** → ✓

The hypothesis is **not confirmed**. Tag namespacing works correctly.

## Summary

The tag namespacing convention is well-structured and consistent. Each agent's self-queries correctly retrieve their own stores. Cross-agent queries (planner's broad search) are intentional and appropriate. The only gaps are:

1. **No enforcement of consistent project name substitution** — if any agent uses wrong project name, queries return empty
2. **planner's broad query may surface wave-progress records** — minor semantic retrieval noise, not a correctness issue

## Verdict rationale

HEALTHY: the primary risk (cross-agent queries missing relevant stores due to tag format mismatch) does not exist. The system works as designed. The two noted gaps are low-severity documentation and retrieval quality concerns.
