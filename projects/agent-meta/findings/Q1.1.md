# Finding: Q1.1 ā€” Mortar routing table missing 4 Mode values used in campaigns

**Question**: Does Mortar's routing table cover all Mode values that appear in template questions.md examples?
**Verdict**: WARNING
**Severity**: High

## Evidence

Mortar's routing table (`mortar.md`) defines 7 mode mappings:

| Mode | Agent |
|------|-------|
| `simulate` | quantitative-analyst |
| `diagnose` | diagnose-analyst |
| `fix` | fix-implementer |
| `audit` | compliance-auditor |
| `research` | regulatory-researcher or competitive-analyst |
| `benchmark` | benchmark-engineer |
| `agent` | use the `**Agent**:` field directly |

**Fallback**: unrecognized mode ā†’ `quantitative-analyst` + logged warning.

Cross-searching all questions.md files across the repo (`bricklayer-v2`, `bricklayer-meta`, `template-frontier`, `recall-arch-frontier`, `projects/bl2`) reveals the following Mode values in actual campaigns that are NOT in the routing table:

| Undocumented Mode | Source | Mortar Fallback |
|-------------------|--------|-----------------|
| `code_audit` | Multiple question files (line 6 repeated, meaning multiple questions) | `quantitative-analyst` (wrong) |
| `correctness` | `bricklayer-v2/questions.md` line 149 | `quantitative-analyst` (wrong) |
| `quality` | `bricklayer-v2/questions.md` line 161 | `quantitative-analyst` (wrong) |
| `subprocess` | `template-frontier/questions.md` line 23 | `quantitative-analyst` (wrong) |
| `correctness \| quality \| performance \| agent` | `bricklayer-v2/questions.md` line 75 | pipe-separated value ā€” Mortar's exact-match lookup fails entirely |

The note is: The **template questions.md** itself does not use the Mode field at all (it's a plain table format). So the template is not the source of the mismatch ā€” actual campaign questions.md files generated by `question-designer-bl2` / `question-designer` are.

### Impact

1. Every question with `Mode: code_audit` gets routed to `quantitative-analyst`, which runs a financial simulation instead of a code audit ā€” producing garbage findings that look valid.
2. The pipe-separated mode `correctness | quality | performance | agent` is unrecognized as a whole string, falls back to `quantitative-analyst`.
3. No error is raised ā€” Mortar only logs a warning. The question is marked DONE, the finding is written, and the campaign advances with silently incorrect analysis.

## Mitigation Recommendation

1. **Add routing entries** in `mortar.md` for the four missing modes:
   - `code_audit` ā†’ `code-reviewer`
   - `correctness` ā†’ `diagnose-analyst` (correctness analysis is diagnosis)
   - `quality` ā†’ `diagnose-analyst` or `code-reviewer`
   - `subprocess` ā†’ `diagnose-analyst`
2. **Handle pipe-separated modes** ā€” either forbid multi-mode in question format (document it) or parse the first value as the routing key.
3. **Upgrade the fallback** ā€” instead of silently routing to `quantitative-analyst`, fail loudly: write an error finding and mark the question INCONCLUSIVE rather than DONE.
4. **Audit `question-designer-bl2`** ā€” it is generating undocumented Mode values. Add a Mode enum constraint to its output spec.

## Suggested Follow-ups
- Does `question-designer-bl2` or `question-designer` have a documented list of valid Mode values, and does it match Mortar's routing table?
- If Mortar silently misfires on `code_audit` questions, how many findings in existing campaign archives were produced by the wrong agent?
