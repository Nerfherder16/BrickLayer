# Finding: Q1.1-WM3 — Double field name mismatch: question-designer-bl2 uses `**Method**:` but Mortar reads `**Agent**:`; plus references non-existent `frontier-analyst`

**Question**: Does question-designer-bl2's agent-mode question format (the `**Agent**:` field) match what Mortar expects to parse when Mode is `agent`?
**Verdict**: WARNING
**Severity**: High

## Evidence

### Field name mismatch #2 (compound failure)

question-designer-bl2 question format (from question-designer-bl2.md, line 86):
```
**Method**: {which agent runs this: diagnose-analyst | fix-implementer | research-analyst | compliance-auditor | design-reviewer | evolve-optimizer | health-monitor | cascade-analyst | frontier-analyst}
```

Mortar's `agent` mode routing (from mortar.md, line 36):
```
| `agent` | use the `**Agent**:` field in the question directly |
```

**These are different field names**: `**Method**:` vs `**Agent**:`. This is the third field name mismatch found in this investigation (previous: `**Operational Mode**:` vs `**Mode**:`).

### Compound failure path

When a question-designer-bl2 question with `**Operational Mode**: Diagnose` and `**Method**: diagnose-analyst` reaches Mortar:
1. Mortar looks for `**Mode**:` → not found → falls back to `quantitative-analyst` (wrong)
2. Even if Mortar somehow resolved the mode as `agent`, it would look for `**Agent**:` → not found → calls undefined agent or silently fails

The compound effect means ALL question-designer-bl2 questions fail Mortar routing at two layers.

### Non-existent agent reference

question-designer-bl2 lists `frontier-analyst` as a valid `**Method**:` value:
```
**Method**: ... | frontier-analyst
```

There is no `frontier-analyst.md` in the template agents directory (confirmed in Q1.1-FU3 — only `research-analyst.md` covers frontier territory). If a `**Method**: frontier-analyst` question somehow reached Mortar, the agent invocation would reference a non-existent file.

### Summary of all interface mismatches found in D1 investigation

| question-designer-bl2 field | Mortar expects | Status |
|-----------------------------|----------------|--------|
| `**Operational Mode**:` | `**Mode**:` | MISMATCH |
| Mode values CamelCase (Diagnose) | lowercase (diagnose) | MISMATCH |
| `**Method**:` | `**Agent**:` | MISMATCH |
| `frontier-analyst` agent reference | No such agent exists | PHANTOM |

These are not edge cases — they affect 100% of questions generated by question-designer-bl2.

## Mitigation Recommendation

1. **Choose a canonical question format** and make ALL agents use it. Either:
   - Update question-designer-bl2 to output `**Mode**:` (lowercase) and `**Agent**:` — matching Mortar's expectations
   - OR update Mortar to read `**Operational Mode**:` and `**Method**:` — matching question-designer-bl2's output
2. Add `frontier-analyst.md` to the template agents directory (thin wrapper around research-analyst with Frontier-specific instructions).
3. The `**Method**:` field in question-designer-bl2 should also enumerate valid agent names that actually exist — add validation to the output contract.

## Suggested Follow-ups
- Is there a "master question schema" document that defines the canonical question format for BL 2.0? If not, this is the root cause of all three field-name mismatches.
