# Finding: Q1.1-WM4 — No master question schema document exists; three field-name mismatches have a common root cause

**Question**: Is there a master question schema document that defines the canonical BL 2.0 question format? If not, is this the root cause of all three field-name mismatches found?
**Verdict**: WARNING
**Severity**: High

## Evidence

Searching across the BrickLayer 2.0 repository for any master question schema document:
- Root `.md` files: ARCHITECTURE.md, FRAMEWORK.md, QUICKSTART.md, README.md, ROADMAP.md — none contain a canonical question field specification
- Template directory: `template/.claude/agents/` — contains individual agent definitions but no schema file
- No `QUESTION_SCHEMA.md`, `QUESTION_FORMAT.md`, `INTERFACE_CONTRACT.md`, or equivalent found anywhere

The only places where question format is defined are inside individual agent definition files:
- `question-designer.md` — defines BL 1.x format with `**Status**:`, no Mode field
- `question-designer-bl2.md` — defines BL 2.0 format with `**Operational Mode**:`, `**Method**:`, `**Priority**:`
- `mortar.md` — defines what it reads: `**Mode**:`, `**Agent**:`
- None of these cross-reference each other

### The three mismatches and their common root

| Mismatch | Producer | Consumer | Impact |
|----------|----------|----------|--------|
| `**Operational Mode**:` vs `**Mode**:` | question-designer-bl2 | mortar | 100% routing failure |
| CamelCase modes vs lowercase modes | question-designer-bl2 | mortar | Case-sensitive match failure |
| `**Method**:` vs `**Agent**:` | question-designer-bl2 | mortar | agent-mode invocation failure |

All three mismatches are between the same producer (question-designer-bl2) and the same consumer (mortar). This strongly suggests they were developed **independently and never cross-tested** against each other. The absence of a shared schema document means there was no mechanism for the two agents to agree on interface.

### Comparison to BL 1.x

BL 1.x avoids this problem by design: question-designer (BL 1.x) generates questions with no Mode field, Mortar routes entirely by domain (D1-D6). The routing is implicit and embedded in the domain convention, not in a field. No field-name contract was needed.

BL 2.0 introduced an explicit routing field, which requires a shared contract. That contract was never written as a standalone document.

### ARCHITECTURE.md note

ARCHITECTURE.md does reference "Operational Mode injection" (F13.1 fix in the bl2 Python runner), but this refers to the Python-based campaign runner's question parsing, not to the Mortar agent. The two systems evolved in parallel without a shared interface specification.

## Mitigation Recommendation

1. **Create `template/QUESTION_SCHEMA.md`** — the single authoritative document defining:
   - Canonical field names for BL 2.0 questions
   - Valid values for each field (Mode enum, Priority enum, Status enum)
   - Which fields are required vs optional
   - Examples of correct question format for each Mode
2. **Update question-designer-bl2.md** to reference QUESTION_SCHEMA.md as its output spec
3. **Update mortar.md** to reference QUESTION_SCHEMA.md as its input spec
4. **Add a CI check** or validator script that parses questions.md and validates against QUESTION_SCHEMA.md before the loop starts
