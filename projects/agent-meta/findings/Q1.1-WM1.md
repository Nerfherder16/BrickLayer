# Finding: Q1.1-WM1 — Mortar has no campaign-startup Mode validation; routing failures discovered only at runtime

**Question**: Does Mortar have a campaign-startup validation step to detect unrecognized Mode values before the first question runs?
**Verdict**: WARNING
**Severity**: Medium

## Evidence

Reviewing mortar.md for any pre-flight validation logic:

```
# Mortar's core loop (from mortar.md):
while PENDING questions exist:
    q = next PENDING question (by priority, then order)
    route(q) → specialist agent
    receive finding → validate structure
    ...
```

There is no pre-flight scan. Mode validation is:
1. **At routing time**: "If mode is missing or unrecognized, default to `quantitative-analyst` and log a warning" — this is per-question, at dispatch time
2. **Post-dispatch**: "validate structure" — validates the finding *after* the agent runs, not the mode before

### Impact

In a 30-question campaign where questions 5–30 have `Validate`, `Evolve`, `Monitor`, or `Predict` modes:
- The campaign starts normally, no warning is visible
- Question 5 routes to `quantitative-analyst` instead of `design-reviewer`
- Mortar "logs a warning" (unclear where — stdout? findings?) but continues
- All 26 remaining questions with those modes produce garbage findings
- The campaign ends "successfully" with 30 findings, all stamped DONE

The warning logging is also undocumented: mortar.md says "log a warning" but specifies no output channel — it may be a console print that disappears, not a persistent flag.

### Note on scope

This gap compounds the routing table issues found in Q1.1 and Q1.1-FU1. If either of those were fixed (routing table expanded, field name unified), this gap would be less dangerous. But as a defense-in-depth measure, startup validation would catch mismatches introduced by:
- New campaign with custom Mode values
- Manual question additions by humans using non-standard Mode names
- Version skew (question-designer updated, mortar not yet updated)

## Mitigation Recommendation

1. Add a "Campaign Pre-flight" section to mortar.md that runs before the main loop:
   ```
   # On campaign start:
   Scan all PENDING questions — extract all Mode values
   For each unique Mode: check against routing table
   If any Mode is unrecognized: WARN loudly, list them, prompt:
     "N questions have unrecognized modes: [list]. They will fall back to quantitative-analyst.
      Continue? [yes/no — default no]"
   ```
2. Log the warning to a persistent file (`campaign-warnings.log`) not just stdout.
3. Consider making startup validation a hard block (not just a warning) for Mode values that are clearly wrong (e.g., `code_audit` is a bl2 Python type, not a routing mode).
