# Finding: Q3.2 — NEEDS_REVISION vs BLOCKED criteria are clearly defined; three soft enforcement gaps exist but are low-severity

**Question**: Does code-reviewer's NEEDS_REVISION vs BLOCKED distinction have clear enough criteria to be applied consistently?
**Verdict**: WARNING
**Severity**: Low

## Evidence

### Verdict criteria table (from code-reviewer.md)

| Verdict | Criteria |
|---------|---------|
| APPROVED | Diff matches spec, no lint errors, no regression risk, verification passes |
| NEEDS_REVISION | Lint warnings, incomplete coverage, style violations |
| BLOCKED | Lint errors, verification failure, regression introduced, diff contradicts spec |

The core NEEDS_REVISION vs BLOCKED distinction is **lint warnings vs lint errors** and **verification pass vs fail**. Both are binary and deterministic.

### Gap 1 — "Incomplete coverage" is listed as NEEDS_REVISION but no coverage tool is available

code-reviewer runs: `flake8`, `mypy`, `eslint`, `tsc`. None of these report code coverage. Coverage requires `pytest-cov`, `jest --coverage`, or `nyc`. Without these tools, coverage completeness cannot be checked — but it remains listed as a NEEDS_REVISION trigger. In practice, code-reviewer must skip this criterion for every project that doesn't run a coverage reporter in the verification command.

**Effect**: Coverage issues are silently ignored; code-reviewer can only issue NEEDS_REVISION for lint warnings and style violations.

### Gap 2 — Regression risk is LLM judgment, not test execution

Step 4 instructs code-reviewer to "read the files adjacent to the changed code" and flag call sites and interface changes. This is static code reading — no test runner is invoked. "Regression introduced" as a BLOCKED trigger is subjective: if code-reviewer misjudges a safe change as risky, it incorrectly BLOCKs. If it misses a real regression, it falsely APPROVEs.

Tests are not run as part of code-reviewer's protocol. A regression in the test suite would only be caught by the verification command (if it runs tests) or by the peer-reviewer's independent re-run.

**Effect**: Regression detection reliability depends entirely on LLM code-reading quality, not deterministic test execution.

### Gap 3 — 2-cycle revision limit is advisory, not tracked by code-reviewer

code-reviewer's NEEDS_REVISION description says "Max 2 revision cycles before escalating to BLOCKED." But code-reviewer does not track cycle count. The `revision_cycle: 0` field in its output contract is never incremented across calls — each invocation starts fresh. code-reviewer cannot know if it's reviewing a 1st or 5th revision.

The cycle limit enforcement depends on fix-implementer reading the note and counting its own revision attempts. If fix-implementer doesn't track this, the 2-cycle limit is never enforced.

**Effect**: A fix that can't converge in 2 cycles could loop indefinitely between NEEDS_REVISION and revision attempts.

### Linter availability and APPROVED

When no linter is available (Step 3: "skip if not found"), code-reviewer records "Linter not available." If the diff matches spec and verification passes, it correctly returns APPROVED. This is expected behavior, not a gap — you cannot check what you cannot run.

### Summary

| Criterion | Clarity | Enforceability |
|-----------|---------|----------------|
| Lint errors → BLOCKED | High — binary | High — deterministic |
| Lint warnings → NEEDS_REVISION | High — binary | High — deterministic |
| Verification failure → BLOCKED | High — binary | High — deterministic |
| Coverage incomplete → NEEDS_REVISION | Clear intent | Low — no coverage tool |
| Regression introduced → BLOCKED | Clear intent | Low — LLM static reading |
| Diff contradicts spec → BLOCKED | High | Medium — LLM judgment |
| 2-cycle limit → escalate to BLOCKED | Stated | Not tracked |

The primary hypothesis concern ("an agent may always return APPROVED when no linter is available") is not a serious risk — APPROVED without lint is correct behavior when lint is unavailable. The real gaps are coverage checking and cycle tracking.

## Mitigation Recommendation

1. **Remove "incomplete coverage" from NEEDS_REVISION unless verification command runs coverage**: Only cite coverage if the verification command includes a coverage reporter.

2. **Run tests before issuing APPROVED**: Add Step 4.5 — if a test runner exists, run it. Failing tests → BLOCKED.

3. **Track revision cycle in the finding file**: Append a `<!-- revision-cycle: N -->` comment to the finding on each review pass, so code-reviewer can read its own prior verdicts and count cycles.
