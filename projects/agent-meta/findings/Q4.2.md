# Finding: Q4.2 — Mortar's OVERRIDE re-queuing relies on filename-to-question-ID convention; three edge cases cause silent failure

**Question**: Does Mortar's OVERRIDE re-queuing logic correctly identify which question to re-queue when a finding file is overridden?
**Verdict**: WARNING
**Severity**: Medium

## Evidence

### Mortar's OVERRIDE re-queuing protocol

```bash
grep -l "Verdict.*OVERRIDE" findings/*.md 2>/dev/null
```

For each match:
1. Read the finding — extract the original question ID
2. Set that question back to `PENDING` in questions.md
3. Append note: `[RE-QUEUED by Mortar after OVERRIDE by peer-reviewer — {date}]`

### How question ID is extracted

Mortar's protocol says "read the finding — extract the original question ID" but does not specify HOW. Two possible mechanisms:

1. **From filename**: `findings/Q3.2.md` → strip `findings/` and `.md` → `Q3.2`
2. **From file content**: Read the `## Peer Review` section's or finding header's reference to the original question

The grep returns file paths like `findings/Q3.2.md`. The most natural (and only reliable) extraction is from the filename. File content parsing would require a consistent header format across all finding types.

### Edge case 1 — Fix findings with `_fix` suffix

fix-implementer writes to `findings/{id}_fix.md`. If peer-reviewer reviews a fix finding and issues OVERRIDE, the grep returns `findings/Q3.2_fix.md`. Mortar extracts `Q3.2_fix` — a non-existent question ID. The re-queue attempt in questions.md silently fails:

```python
# Looking for: **Question ID**: Q3.2_fix (doesn't exist)
# Result: no PENDING status update, question stays DONE
```

**Impact**: OVERRIDE on a fix finding is silently ignored. The bad fix remains committed.

### Edge case 2 — No disambiguation between original finding and appended Peer Review section

After peer-reviewer appends to a finding, the file `findings/Q3.2.md` contains BOTH the original verdict AND the peer review verdict. The grep pattern `"Verdict.*OVERRIDE"` matches the peer review's `**Verdict**: OVERRIDE`. It ALSO matches if the original finding somehow used OVERRIDE (e.g., if a specialist misfired and wrote OVERRIDE as the primary verdict).

If a specialist wrote `**Verdict**: OVERRIDE` as its primary verdict (non-standard but possible via LLM error), Mortar would re-queue it as if peer-reviewer issued the OVERRIDE — even with no peer review present.

**Impact**: Spurious re-queueing on specialist verdict format errors.

### Edge case 3 — OVERRIDE not cleared after handling (double re-queue — already documented in Q2.2)

As noted in Q2.2: after re-queuing, the OVERRIDE verdict remains in the finding. On the next wave-start check, the same `findings/Q3.2.md` matches the grep again, and Q3.2 is re-queued a second time. Mortar appends another `[RE-QUEUED...]` note but effectively doubles the question in the queue.

This is a cascade from the gap documented in Q2.2 — root cause: no handled-marker written after re-queuing.

### Edge case 4 — Non-standard question IDs from hypothesis-generator

hypothesis-generator can produce questions with IDs like `Q1.3-FU1`, `Q1.1-WM3`, `Q2.1-FU1`, etc. These IDs contain hyphens and suffixes. The filename convention follows: `findings/Q1.3-FU1.md`. Extracting `Q1.3-FU1` from the filename is correct — hyphens in filenames map directly to hyphens in question IDs.

**This edge case is NOT a problem.** Hyphen-based IDs survive the filename stripping intact.

### What "read the finding to extract the question ID" means if not from filename

If Mortar reads the finding file's content to find the question ID, it would look for the `**Question**:` or the question block header (e.g., `# Finding: Q3.2 — ...`). All findings written by this campaign follow the convention `# Finding: {question_id} — {title}`. Extracting the ID from the `# Finding:` header is reliable for standard findings but requires parsing and would still fail for `findings/Q3.2_fix.md` (whose header might say `# Fix Finding: Q3.2_fix`).

### Summary

| Scenario | ID extraction | Re-queue outcome |
|----------|---------------|-----------------|
| Standard finding `findings/Q3.2.md` | Q3.2 from filename | ✓ Correct |
| Fix finding `findings/Q3.2_fix.md` | Q3.2_fix (wrong) | ✗ Silent failure |
| Follow-up IDs `findings/Q1.1-FU1.md` | Q1.1-FU1 from filename | ✓ Correct |
| Wave-mid IDs `findings/Q1.1-WM3.md` | Q1.1-WM3 from filename | ✓ Correct |
| Specialist misfires OVERRIDE as primary | False OVERRIDE match | ✗ Spurious re-queue |
| Same file matched on next wave-start | Second re-queue | ✗ Duplicate queue (Q2.2) |

## Mitigation Recommendation

1. **Strip `_fix` suffix when extracting question ID**: `re.sub(r'_fix$', '', stem)` before looking up in questions.md.

2. **Check peer review section exists before re-queuing**: Only re-queue if the OVERRIDE appears inside a `## Peer Review` section, not if it's the primary verdict.

3. **Write handled-marker after re-queuing (Q2.2 root fix)**: Append `<!-- OVERRIDE-HANDLED: {date} -->` to the finding after re-queuing so the grep skips it on next check.
