# Finding: Q2.1 — fix-implementer and code-reviewer have mismatched Fix Specification field names; LLM-parseable but not a strict contract

**Question**: Does fix-implementer's Fix Specification format match what code-reviewer expects to read?
**Verdict**: WARNING
**Severity**: Low-Medium

## Evidence

### fix-implementer.md — what it writes

fix-implementer requires a Fix Specification to pass its pre-flight "specificity gate". The required fields are:

```
**Target file**: exact file path
**Target location**: line number OR function/method name
**Concrete edit**: diff-level description of the change
**Verification command**: runnable command to prove the fix
```

fix-implementer also writes a finding to `findings/{original_finding_id}_fix.md` with sections:
- `## Fix Specification` (summary of what was specified)
- `## Actions Taken` (what was actually changed)
- `## Verification` (result of running the verification command)

### code-reviewer.md — what it reads

code-reviewer's Step 1 is: "Read the finding file at finding_path. Extract the Fix Specification: File, Line/Location, Change, Verification."

The field names code-reviewer expects:
- `File`
- `Line/Location`
- `Change`
- `Verification`

### Field name comparison

| fix-implementer writes | code-reviewer reads |
|------------------------|---------------------|
| `Target file` | `File` |
| `Target location` | `Line/Location` |
| `Concrete edit` | `Change` |
| `Verification command` | `Verification` |

All four fields are present in both but with different names. The semantic meaning is identical; the string labels differ.

### Impact assessment

**The mismatch is real but low-severity** because both agents are LLM-based. An LLM reading a section with "Target file: foo.py" and looking for "File" will correctly identify "Target file" as that field. This is unlike a strict string-key parser that would fail on an unrecognized header.

However, the contract is informal. The practical risks are:

1. **Ambiguity on multi-target fixes**: If fix-implementer writes multiple "Target file" entries (one per changed file), code-reviewer looking for a singular "File" may handle the plural case inconsistently.

2. **Missing Line/Location**: fix-implementer's `Target location` allows either a line number OR a function name. code-reviewer reads `Line/Location`. If the finding only has a function name (not a line), code-reviewer's diff-check step ("verify the change appears at the expected location") may attempt line-number matching and find nothing.

3. **Verification vs Verification command**: Functionally identical. The field is named "Verification command" in fix-implementer but "Verification" in code-reviewer — no real risk here.

4. **`Concrete edit` vs `Change`**: If the edit description is long and narrative (as fix-implementer allows), code-reviewer's extraction of "Change" may truncate or lose nuance needed to evaluate the diff.

### Secondary gap: code-reviewer reads `finding_path` directly

code-reviewer's protocol reads the **original finding file** (e.g., `findings/Q3.1.md`), not the fix finding file (`findings/Q3.1_fix.md`). This means code-reviewer evaluates the **pre-fix state** of the problem, not the actual fix implementation. Unless fix-implementer has appended a `## Fix Implementation` section to the original finding (which it does not — it writes a separate file), code-reviewer may be reviewing a stale finding rather than the fix artifact.

This is a more significant gap than the field-name mismatch: **code-reviewer and fix-implementer write to/read from different files**.

## Summary

| Issue | Severity |
|-------|----------|
| Field name mismatch (Target file vs File, Concrete edit vs Change) | Low — semantically equivalent, LLM-parseable |
| fix-implementer writes `_fix.md`, code-reviewer reads original `.md` | Medium — may produce reviews of stale findings |
| Target location (function name) vs Line/Location (line number) | Low — ambiguous but workable |

## Mitigation Recommendation

1. **Standardize field names**: Update fix-implementer.md and code-reviewer.md to use identical field labels. Canonical: `## Fix Specification` → `**File**:`, `**Location**:`, `**Change**:`, `**Verification**:`.

2. **Clarify file routing**: Add explicit instruction in code-reviewer.md: "If `{finding_id}_fix.md` exists, read it instead of `{finding_id}.md` — this is the fix artifact to review."

3. **Or reverse**: Update fix-implementer to append to the original finding rather than writing a new file, so code-reviewer always has the complete picture in one document.
