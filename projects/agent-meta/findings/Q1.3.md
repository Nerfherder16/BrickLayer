# Finding: Q1.3 — Planner writes CAMPAIGN_PLAN.md but neither question-designer nor Mortar reads it; the interface is severed

**Question**: Does the planner agent's CAMPAIGN_PLAN.md format provide enough context for question-designer to prioritize domains correctly?
**Verdict**: WARNING
**Severity**: High

## Evidence

### The intended interface

planner.md documents that it writes `CAMPAIGN_PLAN.md` with:
- Domain risk rankings (priority scores D1–D6)
- Targeting brief ("generate 3-5 questions in this area")
- Known landmines (settled questions to skip)
- An "Instruction Block for Question-Designer" section

### The actual interface — severed

Searching both question-designer.md and question-designer-bl2.md for any reference to:
- `CAMPAIGN_PLAN.md` → no matches
- `planner` → no matches
- `campaign_plan` → no matches
- `campaign-plan` → no matches

**Neither question-designer variant reads CAMPAIGN_PLAN.md.** The file is written by planner but never consumed.

Searching mortar.md and program.md for CAMPAIGN_PLAN or planner invocation in the main loop:
- No references found

### Impact

When the planner is invoked at campaign start:
1. Planner reads all project files, scores domains, identifies landmines, writes a prioritized targeting brief ✓
2. CAMPAIGN_PLAN.md sits on disk with no consumer
3. question-designer reads `project-brief.md` + `docs/` + `constants.py` + `simulate.py` independently
4. question-designer prioritizes domains based on its own reading, NOT the planner's risk analysis
5. The planner's domain priority scores, known landmines, and mode selection guidance are ignored

### Additional gap: BL 2.0 mode selection not covered

Even if question-designer-bl2 DID read CAMPAIGN_PLAN.md, the plan's domain model (D1–D6 from BL 1.x) doesn't map to BL 2.0 operational modes (Diagnose, Validate, Frontier, etc.). Planner would need to produce BL 2.0 mode guidance, not just D-domain priorities, to be useful for question-designer-bl2.

### BL 1.x vs BL 2.0 mismatch

Planner.md is written for the BL 1.x domain model (D1–D6 numerical domains). question-designer-bl2 has its own 9-mode system and doesn't reference the D1–D6 taxonomy at all. The CAMPAIGN_PLAN.md output format from planner is incompatible with the BL 2.0 question-designer's input expectations.

## Mitigation Recommendation

1. **Add CAMPAIGN_PLAN.md to question-designer-bl2's pre-flight reading**:
   ```bash
   cat CAMPAIGN_PLAN.md 2>/dev/null && echo "Campaign plan found — use for domain priority"
   ```
   Add a section in question-designer-bl2.md: "If CAMPAIGN_PLAN.md exists, read it first. Use domain priorities to weight mode selection (high-priority D5 domains → more Diagnose questions)."

2. **Update planner.md for BL 2.0**: Add a second output section translating domain priorities to BL 2.0 mode counts:
   ```
   ## BL 2.0 Mode Allocation
   | Mode | Recommended questions | Rationale |
   ```

3. **Add a planner invocation check to Mortar**: Before starting the loop, check if CAMPAIGN_PLAN.md exists. If not, suggest running planner first (non-blocking).

4. **Add CAMPAIGN_PLAN.md to question-designer.md (BL 1.x)**: Same fix — read it if present.
