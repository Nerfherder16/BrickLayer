# Finding: Q1.1-FU2 — code_audit Mode values in bl2 are Python runner type flags, not Mortar routing modes; no archive corruption found

**Question**: If Mortar silently misfires on code_audit questions (routing to quantitative-analyst), how many findings in existing campaign archives were produced by the wrong agent?
**Verdict**: HEALTHY
**Severity**: Medium

## Evidence

The `code_audit` Mode values found in `projects/bl2/questions.md` are used by a Python-based campaign runner, not by the Mortar agent.

### bl2 project infrastructure

The bl2 campaign uses `simulate.py` + `evaluate.py` + `analyze.py` — a Python-based loop, not the Mortar agent orchestration system. The `**Mode**: code_audit` in bl2's questions.md is a type flag consumed by the Python runner's `parse_questions()` function, which maps `[CORRECTNESS]` tags in question titles to `question_type="code_audit"`. This triggers the C-30 confidence-cap logic in `bl/findings.py` — it is **not** a routing directive to Mortar.

Evidence from bl2's questions.md:
- Line 535: "D4.3 [CORRECTNESS] _synthetic_question() inherits question_type from original, triggering C-30 on heal findings" — confirms code_audit is from `[CORRECTNESS]` tags
- Findings in bl2 show proper BL 2.0 verdicts (NON_COMPLIANT, DIAGNOSIS_COMPLETE, FIXED) — the correct agents ran

### No Mortar-based campaigns with code_audit Mode found

Searching across all questions.md files, campaigns that use the structured `**Mode**:` field with `code_audit`, `correctness`, `quality`, `subprocess` values all belong to bl2 (Python runner) or are draft/research projects (bricklayer-v2, template-frontier). None appear to have been run through the template Mortar agent.

The bricklayer-v2 and template-frontier questions.md files have these Mode values but their findings/ directories either don't exist or are empty, meaning these campaigns either haven't run or ran with a custom runner.

### Why no archive corruption

The silently-misrouted scenario from Q1.1 (Mortar → quantitative-analyst for code_audit questions) hasn't manifested in practice because:
1. bl2 project uses the Python runner, which handles code_audit correctly
2. Template Mortar campaigns (the ones using mortar.md) all use Mode values from the original 7-entry routing table (simulate, diagnose, fix, audit, research, benchmark, agent)
3. The routing gap is a **future risk** — if someone runs a BL 2.0 campaign with question-designer-bl2 questions through template Mortar, all 5 unrouted modes would misfire

### The actual risk assessment

The routing gaps identified in Q1.1 and Q1.1-FU1 are real design defects but have not yet caused finding corruption. The risk is latent: it activates when a BL 2.0 campaign is first run end-to-end using the template Mortar agent.

## Mitigation Recommendation

1. The Q1.1 and Q1.1-FU1 mitigations remain valid — fix the routing table and field name before any BL 2.0 campaign goes live with Mortar.
2. Add a campaign-startup validation step in Mortar: scan all PENDING questions and warn if any Mode values are unrecognized. Surface the warning before the first question is dispatched.
3. Document in mortar.md that `code_audit` is a bl2 Python runner concept, not a Mortar routing Mode, to prevent future confusion.
