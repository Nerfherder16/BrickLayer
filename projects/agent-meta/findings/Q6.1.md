# Finding: Q6.1 — Mortar has no finding validation step; empty or malformed findings create ghost DONE entries in results.tsv

**Question**: What happens to the campaign if Mortar calls a specialist agent that produces an empty or malformed finding file?
**Verdict**: WARNING
**Severity**: High

## Evidence

### Mortar's finding-handling protocol (from mortar.md)

After invoking a specialist:
```
receive finding → validate structure
spawn peer-reviewer in background
update questions.md status
```

The protocol says "validate structure" — but no validation procedure is specified. Mortar's instructions describe what to DO after a finding (update questions.md, log to results.tsv) but not what to CHECK before doing so.

### What "validate structure" means in practice

Mortar's only structural instructions after receiving a finding are:
1. Read the finding to extract the verdict
2. Update questions.md: `PENDING → DONE`, add `**Finding**: findings/{id}.md`, `**Completed**: {date}`
3. Append to results.tsv: `{id}\t{verdict}\t{agent}\t{date}\t{summary}`
4. Spawn peer-reviewer

None of these steps include:
- Checking that `findings/{id}.md` exists (it might not have been written if the agent crashed)
- Checking that the file is non-empty
- Checking that a `**Verdict**:` field is present
- Checking that the verdict is in VALID_VERDICTS
- Checking that an `## Evidence` section exists

### Failure scenarios

**Scenario A: finding file not written**

If the specialist agent fails mid-execution (context limit, tool error), `findings/{id}.md` may not be created. Mortar receives a response (the agent's failure message) and may attempt to update questions.md. If Mortar treats any non-empty agent response as a completed finding, it sets the question to DONE with no actual finding file. The `**Finding**: findings/{id}.md` reference would point to a non-existent file.

**Scenario B: empty finding file**

If the agent writes `findings/{id}.md` with 0 bytes or only whitespace, Mortar:
1. Reads the file → finds no `**Verdict**:`
2. Extracts verdict → undefined (empty string or None)
3. Appends to results.tsv → `{id}\t\t{agent}\t{date}\t` (blank verdict and summary)
4. Marks question DONE → appears completed but is hollow

peer-reviewer then reads the empty finding and either fails (no evidence to re-run) or returns INCONCLUSIVE. If it returns INCONCLUSIVE, Mortar doesn't re-queue the question — INCONCLUSIVE is not OVERRIDE.

**Scenario C: malformed finding (no Verdict field)**

An agent writes a finding with narrative content but omits `**Verdict**:`. Mortar reads the file, fails to extract a verdict, defaults to empty string, logs a blank verdict. The question is marked DONE with a blank verdict row in results.tsv.

simulate.py's `score_agent` checks for VALID_VERDICTS in the agent definition — not in results.tsv. So blank verdicts in results.tsv don't affect the fleet score. They are silent corruption.

**Scenario D: verdict not in VALID_VERDICTS**

An agent writes `**Verdict**: UNCERTAIN` (not in VALID_VERDICTS). Mortar extracts the verdict string and logs it. No validation occurs. results.tsv contains a non-standard verdict. agent-auditor's definitive rate calculation would count this as non-definitive (since "UNCERTAIN" doesn't match any entry in the definitive whitelist). This would incorrectly lower the agent's definitive rate.

### What peer-reviewer does in the malformed case

peer-reviewer runs "re-execute the exact test or command described in the finding's Evidence section." If there is no Evidence section (empty file), peer-reviewer:
- Cannot find the evidence → records INCONCLUSIVE
- Writes to the finding file (creating or appending to it)
- INCONCLUSIVE does not trigger re-queuing

So peer-reviewer partially mitigates empty findings by producing an INCONCLUSIVE peer review. But this creates a ghost entry where the question appears DONE with no real finding — just a peer review of nothing.

### Quantified risk

In a 30-question campaign:
- If 1 agent fails to write its finding (probability: low but non-zero with long contexts or tool errors)
- Mortar marks it DONE with no file
- The ghost entry appears in results.tsv and questions.md
- Campaign completes with 30/30 DONE but 1 finding missing
- synthesizer reads all findings — finds no file for one question, may produce a synthesis with a gap

## Mitigation Recommendation

1. **Add explicit finding validation step**: After the specialist returns, check:
   ```python
   assert os.path.exists(finding_path), f"Finding not written: {finding_path}"
   assert os.path.getsize(finding_path) > 0, f"Finding is empty: {finding_path}"
   assert "**Verdict**:" in open(finding_path).read(), f"No verdict in {finding_path}"
   ```

2. **On validation failure: mark as INCONCLUSIVE, not DONE**: If validation fails, write a stub finding with `**Verdict**: INCONCLUSIVE` and note "Finding not produced by specialist agent — validation failure."

3. **Validate verdict against VALID_VERDICTS**: If extracted verdict is not in VALID_VERDICTS, log a warning and default to INCONCLUSIVE.

4. **Count hollow questions in results.tsv**: Add a fleet-level check that counts blank-verdict rows. Any campaign with > 0 blank verdicts should trigger a WARNING.
