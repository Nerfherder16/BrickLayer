# Finding: Q6.2 — No retry limit on OVERRIDE re-queuing; a persistently overridden question can loop indefinitely

**Question**: Is there a maximum recursion risk if peer-reviewer issues OVERRIDE on a re-queued question, and Mortar re-queues it again?
**Verdict**: WARNING
**Severity**: Medium

## Evidence

### The loop path

1. Mortar dispatches question Q to specialist agent S
2. S produces finding `findings/Q.md` with verdict V
3. peer-reviewer re-runs the test independently, finds contradiction → appends `**Verdict**: OVERRIDE` to `findings/Q.md`
4. At next wave-start: Mortar greps findings, finds Q.md has OVERRIDE, sets Q back to PENDING
5. Mortar dispatches Q again → same specialist S (or another)
6. S produces a new finding (overwrites or appends to Q.md — unclear which)
7. peer-reviewer re-runs again → if the underlying condition hasn't changed, peer-reviewer issues OVERRIDE again
8. GOTO 4

### Maximum retry limit check

**mortar.md**: "For each OVERRIDE found: 1. Read the finding — extract the original question ID 2. Set that question back to PENDING..." — **no max-retry count**.

**peer-reviewer.md**: "OVERRIDE is a serious signal. Use it only when your independent result directly contradicts the primary finding..." — **no retry awareness**.

Neither agent tracks or checks how many times a question has been re-queued. There is no `**Retry count**:` field in the question block. There is no `MAX_OVERRIDES` constant in constants.py.

### Conditions for infinite loop

For the loop to persist indefinitely:
1. The specialist consistently produces the same (wrong) finding — same LLM prompt, same context, same output
2. peer-reviewer consistently contradicts it — same re-run, same contradicting output
3. No human intervention

Condition 1 is plausible if the question's context is stable and the LLM is deterministic (or near-deterministic). Condition 2 is plausible for empirical findings where the file/code state hasn't changed.

**Realistic infinite loop scenario**: Question Q asks "Does function X in file Y have a race condition?" Specialist answers DIAGNOSIS_COMPLETE (yes, race condition). peer-reviewer re-reads the code, doesn't see it, issues OVERRIDE. Specialist re-answers DIAGNOSIS_COMPLETE again. Loop.

### Stopping conditions that don't exist

| Mechanism | Present? |
|-----------|---------|
| Max-retry counter per question | ✗ Not tracked |
| Max-OVERRIDE counter in results.tsv | ✗ Not checked |
| Human escalation after N overrides | ✗ Not defined |
| Mortar stops re-queuing after 2nd OVERRIDE | ✗ Not defined |
| findings/Q.md OVERRIDE-HANDLED marker prevents second match | ✗ Not cleared (Q2.2) |

### Amplifier: OVERRIDE not cleared after handling (Q2.2)

As documented in Q2.2, once an OVERRIDE is written to `findings/Q.md`, it is never cleared. So even if the re-queued question is answered a second time (new finding written), if the new finding is in a separate section appended to the same file, the original OVERRIDE line persists. The next wave-start grep matches it again and re-queues the question a third time.

This means even a single OVERRIDE from a since-resolved disagreement creates an ongoing re-queue trigger unless the finding file is overwritten (not appended).

### Campaign-level impact

In a 30-question campaign with one looping question:
- Questions 1–29 complete normally
- Question Q loops: PENDING → DONE → OVERRIDE → PENDING → DONE → OVERRIDE → ...
- Campaign never reaches 0 PENDING questions
- Mortar never calls synthesizer
- `NEVER STOP` instruction means Mortar continues the loop indefinitely

This is the highest-severity concrete failure scenario in the campaign: **the NEVER STOP instruction combined with no retry limit creates a genuinely infinite loop.**

### What halts it in practice

- Human interruption (Ctrl+C / kill process)
- Claude context limit (session ends, loop can be resumed — and loop resumes from the same stuck question)
- The specialist produces a different answer on the re-run (LLM non-determinism)
- peer-reviewer produces CONFIRMED on a subsequent re-run (LLM non-determinism)

The system relies on LLM non-determinism as its de facto retry-limit mechanism, which is unreliable.

## Mitigation Recommendation

1. **Add `**Override count**:` field to question blocks**: When re-queuing, increment the count. If count ≥ 3, don't re-queue — instead write a finding `findings/Q_unresolvable.md` with verdict `INCONCLUSIVE` and note the disagreement.

2. **Add `MAX_OVERRIDES = 3` to constants.py**: Mortar checks this constant before re-queuing.

3. **Clear OVERRIDE marker after handling (Q2.2 fix)**: Append `<!-- OVERRIDE-HANDLED -->` to the finding to prevent the grep from matching it on subsequent wave-start checks.

4. **Escalate to human after 2 OVERRIDEs**: Add to Mortar's escalation protocol: "If a question has been re-queued 2+ times due to OVERRIDE, output: `[MORTAR] ESCALATION: Q{id} has been overridden {N} times — human review required.` and mark it PENDING_HUMAN."
