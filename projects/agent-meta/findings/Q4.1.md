# Finding: Q4.1 — Mortar's session-local question counter resets on resume; sentinel fire timing is unreliable across sessions

**Question**: Can Mortar's 5-question forge-check and 10-question agent-auditor sentinels fire correctly if questions complete out of order?
**Verdict**: WARNING
**Severity**: Medium

## Evidence

### Mortar's sentinel protocol

```
Track question count processed this session.

| Every N questions | Action |
|-------------------|--------|
| Every 5 | Spawn forge-check |
| Every 10 | Spawn agent-auditor |
```

Key phrase: **"this session"** — the counter tracks questions processed in the current Claude process context, not questions completed since the last sentinel run.

### Gap 1 — Counter resets on campaign resume

When Mortar is stopped (Claude process ends) and resumed later:
1. The session question counter resets to 0
2. Mortar reads the first PENDING question and begins routing
3. At question N=5 of the resumed session, it fires forge-check
4. At question N=10, it fires agent-auditor

If a 30-question campaign is resumed after question 28 was answered:
- Questions 29 and 30 are PENDING
- Mortar's counter starts at 0 and never reaches 5 (only 2 questions remain)
- **Both sentinels never fire** for the final wave

If resumed after question 4:
- Questions 5 → 25 remain
- N=5 → forge-check fires (OK)
- N=10 → agent-auditor fires (OK)
- Correct behavior here

If resumed mid-wave at question N=7 (3 done, 7 remain):
- N=5 of resumed session → forge-check fires at global question 12 (should have fired at 10, 15)
- The 5-question forge-check actually ran at 5 and now again at 12 (7 questions later, not 5)

**The sentinel interval drifts from the intended cadence whenever a campaign resumes.**

### Gap 2 — Out-of-order completion doesn't affect sentinel trigger, but affects content quality

Mortar's counter is based on questions dispatched in sequence. If question N=3 completes before N=2 (background agents returning findings asynchronously), the counter still increments linearly. Order doesn't affect the trigger — only the session restart issue matters.

### Gap 3 — Recall checkpoint check at campaign start does not include sentinel last-run timestamps

Mortar's Recall checkpoint stores:
```
"Mortar checkpoint [{project}] wave {N}: {completed} done, {pending} pending,
last question: {id}. Overrides: {N}. Background agents: forge-check {last_run}, agent-auditor {last_run}."
```

The `forge-check {last_run}` and `agent-auditor {last_run}` fields ARE stored in the checkpoint. However, Mortar's campaign-start Recall query is:
```
recall_search(query="campaign wave progress questions completed", domain="...", tags=["agent:mortar"])
```

This query doesn't explicitly request `background agents` status. Whether the checkpoint is surfaced with sentinel timing depends on Recall's semantic retrieval. If it's retrieved, Mortar can calculate "forge-check ran at Q=25, we're now at Q=27, next trigger at Q=30." If it's not retrieved, sentinel timing resets.

**The sentinel timing correction on resume depends on Recall retrieval quality — not a deterministic check.**

### Concrete failure scenario

1. Campaign starts, 20 questions answered. forge-check fired at Q5, Q10, Q15, Q20. agent-auditor at Q10, Q20. All correct.
2. Session ends. New session starts.
3. Mortar queries Recall, does NOT retrieve the checkpoint (e.g., Recall server unavailable, or query misses the checkpoint).
4. Counter resets to 0. 20 more PENDING questions queued.
5. forge-check fires at Q=5 (global Q=25). agent-auditor fires at Q=10 (global Q=30). Both intervals are halved — sentinels fire more frequently than intended but not incorrectly.

Actually, frequent sentinel firing is not a correctness bug — it's excess background load. The real failure is when sentinels DON'T fire (campaign with < 5 remaining questions on resume, as described above).

## Summary

| Scenario | Sentinel behavior | Severity |
|----------|-----------------|----------|
| Normal single-session campaign | Correct — fires every 5/10 | ✓ OK |
| Resume with many questions remaining | Drifts — fires at wrong intervals | Low — sentinels fire, just at wrong cadence |
| Resume with < 5 questions remaining | Sentinels never fire | Medium — final wave not audited |
| Out-of-order question completion | No effect on trigger | ✓ OK |

## Mitigation Recommendation

1. **Store last sentinel question number in the questions.md file or a .mortar-state file**: At session start, read the last-run question number and adjust the counter accordingly.

2. **Trigger sentinels at campaign end** regardless of counter state: Before calling synthesizer, always run forge-check and agent-auditor if they haven't run in the final wave.

3. **Use global question count, not session count**: Count from `results.tsv` row count (global completed), not session counter. `wc -l results.tsv` gives the total questions answered. Fire at multiples of 5/10 of global count.
