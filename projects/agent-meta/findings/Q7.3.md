# Finding: Q7.3 — End-to-end symptom of question-designer-bl2 + Mortar: silent misrouting to quantitative-analyst with no error; user sees plausibly correct but wrong findings

**Question**: What observable symptom does a user see when running a question-designer-bl2 campaign through Mortar, given the three field-name mismatches?
**Verdict**: WARNING
**Severity**: High

## Evidence

### The three mismatches (from Q1.1-WM3, Q1.1-FU1)

| Layer | question-designer-bl2 writes | Mortar reads | Mismatch? |
|-------|------------------------------|--------------|-----------|
| Mode field name | `**Operational Mode**: Diagnose` | `**Mode**:` | ✓ YES — different key |
| Mode value case | `Diagnose` (CamelCase) | `diagnose` (lowercase) | ✓ YES — even if found |
| Agent field | `**Method**: diagnose-analyst` | `**Agent**:` | ✓ YES — different key |

### Mortar's behavior on missing/unrecognized Mode

```
If mode is missing or unrecognized, default to `quantitative-analyst` and log a warning.
```

Step-by-step execution on a question-designer-bl2 question:

1. Mortar reads the question block, looks for `**Mode**:` → not found (field is `**Operational Mode**:`)
2. Mode = empty/missing → **default to quantitative-analyst + log warning**
3. Mortar dispatches the question to quantitative-analyst (even if the question is a Diagnose/code analysis question)
4. quantitative-analyst runs a simulation-style analysis on a question it wasn't designed for
5. Finds produce:
   - A verdict in the quantitative-analyst taxonomy (DIAGNOSIS_COMPLETE, HEALTHY, WARNING)
   - Evidence based on simulate.py scenario parameters, not code reading
   - A finding that addresses the question format superficially but with the wrong methodology

### What the user observes

**No errors**: Mortar logs "WARNING: unrecognized Mode, defaulting to quantitative-analyst" — but this is an LLM output line, not a system exception. The campaign continues.

**Plausibly formatted findings**: All findings have the correct structure (## Evidence, ## Verdict, etc.) because quantitative-analyst knows the finding format.

**Wrong methodology for the question type**: A Diagnose question ("Does module X have a race condition?") answered by quantitative-analyst would be interpreted as a simulation parameter variation ("What happens when race condition probability = 0.1?") — entirely different from code inspection.

**Verdict is unreliable**: A question asking for a code analysis verdict gets a simulation verdict. If quantitative-analyst produces HEALTHY for "Does X have a race condition?" it likely means "at current parameters, no threshold is exceeded" — not "I reviewed the code and found no race condition."

**100% silent**: No agent produces an error log visible to the user. results.tsv shows agent=quantitative-analyst for every question. If the user doesn't check the agent field in results.tsv, they may not notice.

### Observable signal that could alert a careful user

1. **results.tsv agent column**: Every row shows `quantitative-analyst` even for Diagnose/Validate/Frontier questions
2. **Finding evidence section**: References simulate.py parameter variations instead of code files, grep outputs, or file path analysis
3. **Mortar's warning log**: "WARNING: unrecognized Mode, defaulting to quantitative-analyst" — appears in Mortar's output stream but is not written to any file

### Scope of damage

If a user generates 20 questions with question-designer-bl2 and starts Mortar:
- 100% of questions (assuming all have `**Operational Mode**:`) are misrouted to quantitative-analyst
- 0 questions reach their intended agent (diagnose-analyst, fix-implementer, research-analyst, etc.)
- Campaign produces 20 findings from the wrong agent
- Synthesis treats all findings as valid — it reads verdict and evidence without checking which agent produced them
- User receives a PDF report with 20 "valid" findings, all produced by the wrong methodology

### No self-correction mechanism

Neither Mortar, question-designer-bl2, nor any other agent detects the systematic misrouting. The Q1.1-WM1 finding confirmed: Mortar has no startup Mode validation, so the misrouting is only discoverable question-by-question from the warning log.

## Mitigation Recommendation

(Already documented in Q1.1-WM3 and Q1.1-WM4 — this finding documents the end-to-end observable impact to motivate priority escalation)

The root fix remains: create a master question schema and align question-designer-bl2 with Mortar's field expectations. Until fixed, **the BL 2.0 campaign workflow is non-functional for any user who follows the documented init sequence with question-designer-bl2.**
