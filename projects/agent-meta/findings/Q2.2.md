# Finding: Q2.2 — peer-reviewer's OVERRIDE format matches Mortar's grep; but two structural gaps make re-queuing unreliable

**Question**: Does peer-reviewer's OVERRIDE escalation actually reach Mortar's wave-start OVERRIDE check?
**Verdict**: WARNING
**Severity**: Medium

## Evidence

### peer-reviewer.md — what it writes on OVERRIDE

peer-reviewer appends a `## Peer Review` section to the finding file with this exact format:

```markdown
---

## Peer Review

**Reviewer**: peer-reviewer
**Date**: {ISO-8601}
**Verdict**: OVERRIDE
```

### mortar.md — grep pattern

```bash
grep -l "Verdict.*OVERRIDE" findings/*.md 2>/dev/null
```

### Pattern match test

| Pattern | Text | Match? |
|---------|------|--------|
| `Verdict.*OVERRIDE` | `**Verdict**: OVERRIDE` | ✓ YES — `Verdict` literal + `.*` covers `**: ` + `OVERRIDE` |

The grep pattern DOES match. The hypothesis that the pattern might not match the format peer-reviewer uses is **not confirmed** — the match works correctly.

### Gap 1 — OVERRIDE is checked only at wave-start (every 10 questions), not after every question

Mortar's protocol: "At the start of each wave (every 10 questions), check for OVERRIDE verdicts."

peer-reviewer runs in the background **after every question**. If peer-reviewer issues OVERRIDE on question N=6 of a 10-question wave, Mortar won't catch it until wave-start at N=10. Questions N=7, 8, 9, 10 are answered without awareness of the OVERRIDE. If those questions build on the overridden finding, 4 additional findings may be contaminated.

This is a timing gap, not a format gap.

### Gap 2 — Re-queuing logic reads finding by filename, but OVERRIDE not cleared after re-queuing

Mortar's OVERRIDE handling:
1. `grep -l "Verdict.*OVERRIDE" findings/*.md` — finds all matching files
2. For each file: extract original question ID, set back to PENDING in questions.md
3. Log to results.tsv

**Problem**: Mortar does not clear or mark the OVERRIDE verdict in the finding after handling it. On the NEXT wave-start check (10 questions later), the same finding still contains `**Verdict**: OVERRIDE`. Mortar will match it again, re-queue the same question a second time, and create a loop.

There is no mechanism to mark an OVERRIDE as "already re-queued".

### Gap 3 — Question ID extraction relies on filename convention

Mortar's step 1 for each OVERRIDE finding: "extract the original question ID." The finding filename is `findings/{question_id}.md` (e.g., `findings/Q3.1.md`). Mortar extracts the ID from the filename by stripping `findings/` and `.md`.

This works for standard question IDs. It breaks for:
- Fix findings: `findings/Q3.1_fix.md` — would extract `Q3.1_fix`, not `Q3.1`
- Wave-mid follow-ups: `findings/Q1.1-WM3.md` — works (hyphen is fine in grep)
- Non-standard IDs introduced by hypothesis-generator: untested

If peer-reviewer reviews a `_fix.md` finding and issues OVERRIDE, Mortar would try to set `Q3.1_fix` back to PENDING — a non-existent question ID. The re-queuing would silently fail.

## Summary

| Issue | Severity |
|-------|----------|
| Grep pattern DOES match peer-reviewer's format | ✓ No issue |
| OVERRIDE checked every 10 questions, not every question | Medium — up to 9 contaminated downstream findings possible |
| OVERRIDE not cleared after handling → infinite re-queue loop | High — structural bug |
| Question ID extraction breaks for `_fix.md` findings | Low — rare but silent failure |

## Mitigation Recommendation

1. **Clear OVERRIDE after re-queuing**: After re-queuing a question, rename the finding file (e.g., append `.overridden`) or add a marker line `[OVERRIDE-HANDLED]` so the next grep skips it.

2. **More frequent OVERRIDE checks**: Check at every wave-start is already defined as every 10 questions. Mortar could also check after each 5-question forge-check interval.

3. **Fix ID extraction**: When grep returns `findings/Q3.1_fix.md`, extract base ID by stripping `_fix` and any other suffixes before `.md`.

4. **Document the timing gap**: Add a note in mortar.md that OVERRIDE re-queuing may delay up to 10 questions — acceptable if the campaign has many questions; unacceptable for short campaigns.
