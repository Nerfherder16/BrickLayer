# Q5.5 — results.tsv race condition on concurrent writes

**Verdict**: WARNING
**Timestamp**: 2026-03-13T00:00:00Z

## Finding

`update_results_tsv()` in `recall/simulate.py` (line 1510) appends to `results.tsv` with no file lock. `grep` for `lock`, `fcntl`, `threading`, or `msvcrt` near the write path returns no results.

The `--campaign` mode (line 1716) appears to run questions sequentially (no parallel worker invocation found in grep output). This makes the race condition theoretical under current usage.

## Evidence

```
$ grep -n "update_results_tsv\|lock\|fcntl\|parallel\|threading" recall/simulate.py
1510: def update_results_tsv(...)
1579: update_results_tsv(...)
(no lock/fcntl/threading matches near write path)
```

## Risk

Low. `--campaign` mode is sequential. The race only triggers if two separate `recall/simulate.py` processes run against the same project simultaneously (e.g., two terminals, or a future parallel campaign mode). The consequence is interleaved TSV rows or lost writes — not a crash.

## Recommended Action

If parallel campaign execution is ever added, wrap the TSV append in a file lock:
```python
import fcntl  # Linux/Mac
# or msvcrt.locking on Windows
```
For now, document the single-process assumption in a comment near `update_results_tsv`.
