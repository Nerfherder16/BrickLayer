# Q3.2 — Wildcard CORS on mutation endpoints

**Verdict**: FAILURE
**Timestamp**: 2026-03-13T00:00:00Z

## Finding

`dashboard/backend/main.py` lines 15–17:

```python
CORSMiddleware,
allow_origins=["*"],
allow_methods=["*"],
```

Combined with server binding to `0.0.0.0` (confirmed in `dashboard/start.sh`), any website can issue cross-origin POST requests to:
- `POST /api/questions` — adds questions to `questions.md`
- `POST /api/findings/{id}/correct` — appends corrections to finding files

No authentication layer exists on any route.

## Evidence

```
$ grep -n "allow_origins\|allow_methods" dashboard/backend/main.py
16:    allow_origins=["*"],
17:    allow_methods=["*"],

$ grep "host 0.0.0.0" dashboard/start.sh
uvicorn main:app --host 0.0.0.0 --port 8100 --reload

$ grep -n "@app.post" dashboard/backend/main.py
198: @app.post("/api/questions")
256: @app.post("/api/findings/{finding_id}/correct")
```

## Risk

High. Any browser tab on the LAN that visits a malicious page can silently:
1. Inject questions into the campaign via POST /api/questions
2. Corrupt findings via POST /api/findings/{id}/correct
3. Combine with path traversal (Q3.1) to write to arbitrary files

The combination of wildcard CORS + 0.0.0.0 binding + no auth + path traversal creates a full write primitive reachable from any LAN browser session.

## Recommended Action

Q4.1 dispatches security-hardener to restrict `allow_origins` to `["http://localhost:3100", "http://127.0.0.1:3100"]` and change the uvicorn binding to `127.0.0.1` in `start.sh` (or add a `--lan` flag for explicit LAN exposure).
