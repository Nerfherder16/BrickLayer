# Q7.4 — Add TTL cache to parse_findings_index

**Verdict**: HEALTHY
**Timestamp**: 2026-03-13T00:00:00Z

## Finding

`parse_findings_index()` in `dashboard/backend/main.py` re-read all `.md` files on every `GET /api/findings` call with no caching (Q6.3 WARNING).

**Fix applied:**

Added module-level `_findings_cache` dict and `_FINDINGS_CACHE_TTL = 5.0` constant. `parse_findings_index()` now checks the cache before scanning the filesystem:

```python
_findings_cache: dict = {}
_FINDINGS_CACHE_TTL = 5.0  # seconds

def parse_findings_index(project_path: Path) -> list[dict]:
    import time
    cache_key = str(project_path)
    cached = _findings_cache.get(cache_key)
    if cached and time.monotonic() - cached["ts"] < _FINDINGS_CACHE_TTL:
        return cached["data"]
    # ... existing read logic ...
    _findings_cache[cache_key] = {"data": findings, "ts": time.monotonic()}
    return findings
```

The `import time` is inlined to avoid linter stripping at module level. Cache is keyed by `str(project_path)` supporting multiple concurrent projects.

## Risk

Low. The 5-second TTL means a finding written and immediately re-fetched within 5 seconds returns stale data — acceptable for a developer tool where the dashboard auto-refreshes. The `correct_finding` endpoint writes directly to disk; the next refresh after TTL expiry picks up the change. Cache is per-process and does not persist across restarts.
