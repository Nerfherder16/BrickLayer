**Verdict**: WARNING
**Timestamp**: 2026-03-13T00:00:00Z

## Finding

`parse_findings_index()` in `dashboard/backend/main.py` line 144 reads every `.md` file in `findings/` on each call:


`GET /api/findings` (line 306) calls `parse_findings_index()` directly with no caching. Every page load in the dashboard re-reads all finding files from disk.

## Risk

Low for small campaigns (<50 findings). Medium for large campaigns (100+ findings with detailed content). The dashboard refreshes frequently during active campaigns, so disk I/O accumulates. No in-process cache or ETag header is used.

## Recommended Action

Add a simple module-level TTL cache (e.g., `functools.lru_cache` with `maxsize=None` and periodic invalidation, or just cache the result for 5 seconds using a timestamp comparison). For the current campaign scale this is low priority.
