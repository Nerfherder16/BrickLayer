# Q1.3 — Dashboard parse_questions handles wave-based format

**Verdict**: FAILURE
**Timestamp**: 2026-03-13T00:00:00Z

## Finding

`parse_questions()` in `dashboard/backend/main.py` only matches the old pipe-table format:

```python
row_match = re.match(r"^\|\s*([^\|]+?)\s*\|\s*([^\|]+?)\s*\|\s*(.+?)\s*\|$", line)
```

BrickLayer's current `questions.md` uses the block format:

```
## Q1.1 [CORRECTNESS] Title
**Mode**: correctness
**Target**: ...
```

The dashboard API returns 0 questions for any current BrickLayer project. Additionally, `add_question()` writes new questions in the old table format (`| {new_id} | PENDING | {title} |`) into a block-format file — a double format mismatch that corrupts the file on any dashboard write.

## Evidence

- `dashboard/backend/main.py` lines 44–76: pure pipe-table regex, no block-format branch
- `dashboard/backend/main.py` line 217: `new_row = f"| {new_id} | PENDING | {title} |"`
- `projects/bricklayer/questions.md`: 14 questions in `## Q{n}.{m}` block format
- `/api/questions?project=...` would return 0 questions against any current project

## Risk

High. The dashboard is the primary human interface for the campaign. If it shows 0 questions, operators cannot monitor progress, add questions, or flag findings. Any question added via the dashboard is written in the wrong format and invisible to the simulate.py runner.

## Recommended Action

Update `parse_questions()` in `dashboard/backend/main.py` to support the block format. Update `add_question()` to write block-format entries. This is a breaking change that must be coordinated — both the runner's parser and the dashboard parser need to agree on one canonical format.
