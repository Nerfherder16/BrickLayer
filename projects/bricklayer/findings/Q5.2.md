# Q5.2 — recall/simulate.py stdout wrapper not guarded by __main__

**Verdict**: FAILURE
**Timestamp**: 2026-03-13T00:00:00Z

## Finding

`recall/simulate.py` line 31 runs at module import time:

```python
# Line 31 — module level, no guard
sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding="utf-8")
```

There is no `if __name__ == "__main__":` guard. Any module that imports `simulate.py` (including the test suite) permanently replaces `sys.stdout` with a new `TextIOWrapper`. This broke pytest's output capture during Q4.3 test writing — the test-writer agent had to implement a workaround using a `BytesIO` stub.

## Evidence

```
$ python -c "
lines = open('recall/simulate.py').readlines()
for i, l in enumerate(lines[28:36], 29):
    print(f'{i}: {l}', end='')
print('In __main__ block:', any('if __name__' in lines[j] for j in range(28,36)))
"
31: sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding="utf-8")
In __main__ block: False
```

The Q4.3 test-writer's `_import_simulate()` helper confirmed this: without the workaround, pytest teardown would crash when trying to reset a replaced `sys.stdout`.

## Risk

Medium. The module is not designed to be imported — it's a CLI runner. But as test coverage grows (Q4.3 created 19 tests), more tests may need to import the module. The workaround in `tests/test_core.py` is fragile and must be maintained forever.

## Recommended Action

Move the `sys.stdout` reassignment inside `if __name__ == "__main__":`:
```python
if __name__ == "__main__":
    sys.stdout = io.TextIOWrapper(sys.stdout.buffer, encoding="utf-8")
```
This is a one-line move with no functional change when running as a script.
