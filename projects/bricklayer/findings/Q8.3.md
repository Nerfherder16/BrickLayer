# Q8.3 — correct_finding does not invalidate _findings_cache

**Verdict**: WARNING → FIXED (Q8.4)
**Timestamp**: 2026-03-13T00:00:00Z

## Finding

`correct_finding()` appended a Human Correction block to a finding file but never cleared `_findings_cache`. A `GET /api/findings` within 5 seconds of a correction returned stale data with `has_correction: false`.

This is a logical inconsistency: the write endpoint modified disk state while the read endpoint served cached data from before the write.

## Fix (Q8.4)

One-line fix added to `correct_finding()` after the file write:

```python
_findings_cache.pop(str(project_path), None)
```

Now `GET /api/findings` immediately after a correction returns fresh data.

## Risk

Negligible after fix. The `_findings_cache` is process-local so the pop is safe and instantaneous.
