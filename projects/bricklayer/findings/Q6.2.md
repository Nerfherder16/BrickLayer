# Q6.2 — Unbounded recursive globs in detect_stack

**Verdict**: FAILURE
**Timestamp**: 2026-03-13T00:00:00Z

## Finding

`detect_stack()` in `onboard.py` uses fully unbounded `rglob()` calls on the repo root:

```python
# line 262
if any(repo_path.rglob("build.gradle.kts")) or any(repo_path.rglob("*.kt")):
# line 264
    if any(repo_path.rglob("AndroidManifest.xml")):
# line 268
if any(repo_path.glob("*.xcodeproj")) or any(repo_path.rglob("*.swift")):
# line 276
if any(repo_path.rglob("*.tf")):
```

On a large repo (e.g., a monorepo with node_modules, .git, or deeply nested vendor directories), each `rglob("*.kt")` walks the entire filesystem tree. With 4 separate unbounded rglob calls plus the package.json/Cargo.toml reads, `detect_stack()` can take tens of seconds on a large codebase.

The `any()` short-circuit helps once a match is found, but if no match exists (the common case for most checks), the full tree is traversed.

## Evidence

```python
# onboard.py lines 262-276
if any(repo_path.rglob("build.gradle.kts")) or any(repo_path.rglob("*.kt")):  # unbounded
if any(repo_path.rglob("AndroidManifest.xml")):  # unbounded
if any(repo_path.glob("*.xcodeproj")) or any(repo_path.rglob("*.swift")):  # unbounded
if any(repo_path.rglob("*.tf")):  # unbounded
```

## Risk

Medium-High. Onboarding a large repo (e.g., a full Android project with thousands of .kt files) would be fast due to early exit, but onboarding a Python repo and checking for .kt would walk the entire tree unnecessarily. The `.git` directory is not excluded from the walk.

## Recommended Action

Q6.6 dispatches perf-optimizer to:
1. Limit search depth: `repo_path.rglob("*.kt")` → check only top 3 levels
2. Exclude `.git`, `node_modules`, `vendor`, `.venv` directories
3. Or use a single combined walk with early exit across all language checks
