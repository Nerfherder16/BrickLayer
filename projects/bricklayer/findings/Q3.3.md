# Q3.3 — Question ID collision in add_question

**Verdict**: FAILURE
**Timestamp**: 2026-03-13T00:00:00Z

## Finding

`add_question()` in `dashboard/backend/main.py` line 214:

```python
existing_ids = [q["id"] for q in questions if q["domain"] == body.domain]
next_num = len(existing_ids) + 1
new_id = f"{domain_num}.{next_num}x"
```

Two independent failure modes:

**1. ID collision after deletion** — If domain D1 has questions [1.1, 1.2, 1.3] and 1.2 is deleted, `len(existing_ids)` = 2, generating `1.3x` which collides with existing `1.3`. The results.tsv log and findings/ both become ambiguous.

**2. Format mismatch on write** — The new question is written as:
```python
new_row = f"| {new_id} | PENDING | {title} |"
```
This is pipe-table format, but current `questions.md` uses block format (`## Q{n}.{m}`). The added question is invisible to `simulate.py`'s block-format parser and corrupts the file's structural integrity.

## Evidence


## Risk

High (format mismatch) + Medium (ID collision). The format mismatch means the dashboard write path is completely broken for current projects — questions added via the UI are silently lost. The ID collision compounds Q1.3's format problem: even if fixed, the ID generator needs to scan for max existing ID, not count.

## Recommended Action

Fix ID generation to use `max(existing_numeric_ids) + 1` instead of `len()`. Update `new_row` to write block format aligned with `simulate.py`'s parser. Both fixes are prerequisite to Q1.3's parse_questions update.
