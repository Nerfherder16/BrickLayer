# Q3.1 — Path traversal via finding_id in correct_finding endpoint

**Verdict**: FAILURE
**Timestamp**: 2026-03-13T00:00:00Z

## Finding

Both `get_finding()` and `correct_finding()` in `dashboard/backend/main.py` use `finding_id` directly in a filesystem path with no sanitization:

```python
# line 250
fpath = project_path / "findings" / f"{finding_id}.md"

# line 261
fpath = project_path / "findings" / f"{finding_id}.md"
```

No `Path.resolve()` or `Path.is_relative_to()` guard exists. A caller can supply `finding_id=../../onboard` to read or append to arbitrary files outside `findings/`.

**The server binds to `0.0.0.0`** (confirmed in `dashboard/start.sh` line 16):
```bash
uvicorn main:app --host 0.0.0.0 --port 8100 --reload
```

This makes the endpoint reachable from LAN, not just localhost.

## Evidence

```
$ grep -n "finding_id" dashboard/backend/main.py
247: def get_finding(finding_id: str, project: Optional[str] = Query(None)):
250:     fpath = project_path / "findings" / f"{finding_id}.md"
256: @app.post("/api/findings/{finding_id}/correct")
261:     fpath = project_path / "findings" / f"{finding_id}.md"

$ grep "host 0.0.0.0" dashboard/start.sh
uvicorn main:app --host 0.0.0.0 --port 8100 --reload
```

No `resolve()` or `is_relative_to()` found anywhere near these paths.

## Risk

High. Any LAN host can read arbitrary files by requesting `/api/findings/../../onboard` and append arbitrary content via `/api/findings/../../onboard/correct`. Combined with wildcard CORS (Q3.2), a malicious web page can trigger this from any browser on the LAN.

## Recommended Action

Q4.1 dispatches security-hardener to add:
```python
fpath = (project_path / "findings" / f"{finding_id}.md").resolve()
if not fpath.is_relative_to((project_path / "findings").resolve()):
    raise HTTPException(status_code=400, detail="Invalid finding ID")
```
to both endpoints before any file operation.
