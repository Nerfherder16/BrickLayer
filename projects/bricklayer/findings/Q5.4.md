# Q5.4 — get_project_path containment guard bypassable via symlinks

**Verdict**: WARNING
**Timestamp**: 2026-03-13T00:00:00Z

## Finding

The Q4.1 security fix added a `resolve()`-based containment guard to `get_project_path()`:

```python
resolved = Path(path_str).resolve()
base = (Path(__file__).parent.parent.parent / "projects").resolve()
if not str(resolved).startswith(str(base)):
    raise HTTPException(status_code=400, detail="Project path outside allowed base")
return resolved
```

**Symlink handling**: `resolve()` follows symlinks before comparison — a symlink `projects/evil -> /etc` resolves to `/etc`, which does NOT start with the base, so the guard catches it correctly. Symlinks are handled safely.

**`startswith` string comparison bug**: The guard uses `str(resolved).startswith(str(base))`. If `base` is `C:\foo\autosearch\projects`, a path `C:\foo\autosearch\projects_evil\` passes the check because the string starts with the same prefix. This requires a crafted directory name and is unlikely in practice.

**Server binding**: `start.sh` still runs `--host 0.0.0.0 --port 8100`. CORS is restricted to localhost (Q4.1 fix), but direct non-browser requests from LAN (curl) can still reach all endpoints.

## Risk

Low-medium. The `startswith` bug requires a specifically crafted sibling directory. Symlinks are handled correctly. The server binding + CORS combination still exposes the API to direct LAN requests (CORS only protects browsers).

## Recommended Action

Replace `startswith` with `is_relative_to()` (Python 3.9+):
```python
if not resolved.is_relative_to(base):
    raise HTTPException(status_code=400, detail="Project path outside allowed base")
```
This eliminates the string prefix false-positive.
