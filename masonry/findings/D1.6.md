# Finding: D1.6 — `shlex.quote` Produces Unsafe Output on Windows cmd.exe

**Question ID**: D1.6
**Status**: DONE
**Verdict**: FAILURE
**Agent**: diagnose-analyst
**Severity**: High
**Confidence**: 1.0

## Summary

`shlex.quote` produces POSIX-style single-quoted strings, which are not recognized as quoting by Windows cmd.exe when `shell=True`. Any prompt containing shell metacharacters (`&`, `|`, `>`, `<`, `^`) will be interpreted by cmd.exe as command operators, not literal characters. A routing request containing `it's` or `echo foo | bar` in its text could cause arbitrary command execution or silent truncation of the prompt passed to Claude Haiku.

## Evidence

### The Problematic Code (llm_router.py lines 46-48)

```python
_is_windows = platform.system() == "Windows"
if _is_windows:
    import shlex
    cmd_str = f"claude --model {_LLM_MODEL} --print -p {shlex.quote(full_prompt)}"
```

Then called with `shell=True` (line 54).

### Why `shlex.quote` Fails on Windows

`shlex.quote()` follows POSIX quoting rules:
- Wraps the string in single quotes: `'text here'`
- Escapes embedded single quotes as: `'\''`

**Windows cmd.exe behavior:**
- cmd.exe does NOT treat single-quoted strings as quoted. Single quotes are literal characters.
- The `'` characters appear in the command string as-is.
- cmd.exe tokenizes on spaces and honors double-quoted strings, not single-quoted.

**Concrete failure case:**

Prompt text: `route this request: run build & deploy`

`shlex.quote` produces: `'route this request: run build & deploy'`

cmd.exe interprets: `claude --model ... --print -p 'route this request: run build ` then executes `deploy'` as a separate command. The `&` operator in cmd.exe means "run next command" regardless of surrounding single quotes.

**Single-quote-in-prompt failure case:**

Prompt text: `it's a routing question`

`shlex.quote` produces: `'it'\''s a routing question'` (POSIX escape)

cmd.exe interprets: three tokens: `'it'\` + `'s` + `a routing question'`. The `\` is not a line continuation in cmd.exe. The command breaks completely.

### Affected Inputs

The `full_prompt` is assembled as:
```python
full_prompt = f"{system_prompt}\n\nUser request: {request_text}"
```

The `system_prompt` itself contains:
```
"You are a routing agent... Respond with ONLY a JSON object: {\"target_agent\": ...}"
```

The curly braces `{` `}` in the system prompt are NOT shell metacharacters in cmd.exe, but the double quotes embedded after `shlex.quote` wrapping may cause issues. More critically, the `agent_list` in the prompt is assembled from agent names and descriptions read from `agent_registry.yml` — these could contain `&`, `|`, or `>` if agent descriptions are not controlled.

### Correct Fix for Windows

Use `subprocess.list2cmdline()` for Windows:
```python
if _is_windows:
    import subprocess as _sp
    # list2cmdline uses Windows-native double-quote escaping
    cmd_str = _sp.list2cmdline(["claude", "--model", _LLM_MODEL, "--print", "-p", full_prompt])
```

Or better: avoid `shell=True` entirely by finding `claude.cmd` explicitly:
```python
if _is_windows:
    import shutil
    claude_cmd = shutil.which("claude") or "claude.cmd"
    # Use list form (no shell=True)
    result = subprocess.run(
        [claude_cmd, "--model", _LLM_MODEL, "--print", "-p", full_prompt],
        capture_output=True, text=True, timeout=_LLM_TIMEOUT,
        shell=False,
    )
```

On Windows, `subprocess.run(["claude", ...], shell=False)` will fail because `claude` is a `.cmd` file (batch script). The correct approach is to use `shell=True` with proper cmd.exe quoting, or to explicitly invoke `cmd.exe /c claude.cmd`.

### Practical Impact on This System

This system runs on Windows (`platform: win32`, per the environment notes). The LLM router IS the Windows code path. Every Layer 3 routing call on this machine uses the vulnerable `shlex.quote` path. Prompts containing apostrophes (common in natural language like "it's", "don't", "user's") will corrupt the command, causing:
1. cmd.exe error or unrecognized token
2. `result.returncode != 0` → function returns `None`
3. Layer 4 fallback fires — routing fails silently

This means Layer 3 is **effectively broken on Windows** for any natural-language request containing apostrophes or shell metacharacters. The fallback to Layer 4 masks the failure.

## Verification

Test: `shlex.quote("it's broken")`
Output: `"'it'\\''s broken'"`

In cmd.exe: `claude ... -p 'it'\''s broken'` → parse error or broken prompt.
