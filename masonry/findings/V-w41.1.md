# Finding: V-w41.1 — Circuit Breaker Fall-Through to Layer 3 Confirmed

**Question**: Verify F-w40.1 circuit breaker — does semantic.py fall through to Layer 3 when OPEN?
**Agent**: design-reviewer
**Verdict**: PASS
**Severity**: N/A (validation passing)

## Evidence

### E1: OPEN path returns None (not exception)

`masonry/src/routing/semantic.py` — `route_semantic()` OPEN branch:
```python
def route_semantic(request: str) -> RoutingDecision | None:
    if _cb_is_open():
        return None  # ← returns None, does not raise
    ...
```

The circuit breaker OPEN path is a clean None return. No exception, no RoutingDecision with a null agent, no sentinel value — just None. This satisfies the contract for Layer 2 "miss" signaling.

### E2: `router.py` handles None from Layer 2 → falls through to Layer 3

`masonry/src/routing/router.py` — Layer 2 → Layer 3 transition:
```python
decision = route_semantic(request)
if decision is not None:
    return decision
# Fall through to Layer 3...
decision = route_llm(request)
```

The `if decision is not None` guard correctly interprets None as a miss. Both "Ollama unavailable" (timeout → None) and "circuit OPEN" (→ None) produce identical fall-through behavior. Layer 3 is activated unconditionally when Layer 2 returns None.

### E3: Timeout is on the HTTP call parameter (not thread join or sleep)

`masonry/src/routing/semantic.py` — httpx call:
```python
response = httpx.post(
    OLLAMA_EMBED_URL,
    json={"model": EMBED_MODEL, "prompt": request},
    timeout=_CB_CALL_TIMEOUT  # ← httpx parameter, not thread-based
)
```

`_CB_CALL_TIMEOUT = 2.0`. The timeout is a native httpx parameter that causes the HTTP library to raise `httpx.TimeoutException` after 2 seconds. No separate thread, no `time.sleep()`, no `threading.join()` with timeout. This means the timeout fires at the transport layer — no dangling threads.

### E4: `_cb_failures` and `_cb_opened_at` are module-level

`masonry/src/routing/semantic.py`:
```python
_cb_failures: int = 0
_cb_opened_at: float | None = None
_CB_THRESHOLD = 3
_CB_RESET_SECONDS = 60.0
```

All four circuit breaker state variables are module-level. State persists across calls within a process lifetime. This is correct for a circuit breaker — per-call state would defeat the purpose (a new call object couldn't track accumulated failures).

### E5: Secondary observation — half-open concurrent probe gap

One theoretical gap exists: the half-open state (after reset window expires) has no explicit probe serialization. If two concurrent calls arrive simultaneously after `_CB_RESET_SECONDS`, both see `_cb_is_open() == False` and both attempt HTTP calls to Ollama. If Ollama is still down, both calls fail, `_cb_failures` increments twice — which is acceptable behavior (re-opens immediately). The gap is: if exactly one succeeds and one fails in the same millisecond window, the success clears the failure count while the failure increments it. Net result: `_cb_failures = 1`, circuit remains closed (threshold=3).

**This is not a current risk**: `router.py` is synchronous (no threading, no async). All routing calls are sequential within a process. The theoretical gap only applies if the router is refactored to be async/concurrent in the future.

All four success criteria from the question spec are confirmed:

The F-w40.1 circuit breaker is correctly wired end-to-end. The P1 routing cascade (Ollama unavailable → semantic.py hangs → router blocks → Layer 3 never reached) is permanently closed for the synchronous router use case.
