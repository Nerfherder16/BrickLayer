# Finding: R8.1 — `masonry-observe.js` Correctly Stores Findings to Recall with One Gap: Silent Timeout on Recall Unavailability

**Status**: DONE
**Verdict**: HEALTHY
**Agent**: research-analyst
**Severity**: Info
**Confidence**: 0.90

## Summary

`masonry-observe.js` correctly detects finding files written to `findings/*.md`, reads the file content, extracts verdict/severity/qid, and calls `storeMemory()` to Recall with correct tags (`masonry:finding`, `qid:X`, `verdict:X`, `severity:X`). Recall at `http://100.70.195.84:8200` is currently reachable (HTTP 200). The one gap is that `storeMemory()` has a 3-second timeout and silently returns `null` on failure — if Recall is unreachable, findings are lost without any log. This does not affect the research loop (findings are written to disk regardless) but does affect recall-based continuity across sessions.

## Evidence

### 1. Finding detection regex

```javascript
const findingsRe = /findings[/\\]([^/\\]+\.md)$/i;
const match = filePath.match(findingsRe);
```

This regex matches `findings/{filename}.md` — single-level findings directory only. For the masonry self-research campaign, all findings are in `findings/` directly (not in subdirectories). The regex correctly captures `D7.1`, `synthesis_wave7`, etc.

**Gap**: `findings/wave1/R1.1.md` would NOT match because `[^/\\]+` doesn't allow path separators. Wave subdirectories are not detected. This is not a problem for the current campaign but would silently drop findings in projects using nested wave directories.

### 2. `storeMemory()` call structure

```javascript
await storeMemory({
    content: snippet,           // fileContent.slice(0, 500)
    domain: `${project}-autoresearch`,
    tags: [
        'masonry',
        `project:${project}`,

        `verdict:${verdict}`,   // e.g. "verdict:FAILURE"
        `severity:${severity}`, // e.g. "severity:Medium"
        'masonry:finding',
    ],
    importance,                 // from SEVERITY_IMPORTANCE map
});
```

Tag structure is correct and searchable. `importance` is tiered by severity:
```javascript
const SEVERITY_IMPORTANCE = {
    Critical: 0.95, High: 0.85, Medium: 0.65, Low: 0.45, Info: 0.3
};
```

### 3. `storeMemory()` timeout and silent failure

```javascript
async function storeMemory({ content, domain, tags, importance }) {
    try {
        const controller = new AbortController();
        const timeout = setTimeout(() => controller.abort(), 3000);
        const res = await fetch(`${baseUrl}/store`, { ... signal: controller.signal });
        clearTimeout(timeout);
        if (!res.ok) return null;
        return await res.json();
    } catch (_err) {
        return null;   // ← silent: no log on timeout/network error
    }
}
```

If Recall is unreachable, the hook:
1. Waits up to 3 seconds
2. Returns `null` silently
3. Continues to state update
4. Exits 0

Findings are NOT stored to Recall on network failure. The research loop is unaffected (findings written to disk), but session-to-session recall continuity via Recall is interrupted silently.

### 5. 500-char content truncation

```javascript

```

The first 500 characters of a typical finding captures:
- Title line (40-80 chars)
- Metadata fields: Question ID, Status, Verdict, Severity, Confidence (100-150 chars)
- Start of Summary section (200-300 chars)

This is sufficient for search-based retrieval — the key metadata is always in the header. The Evidence section is typically not captured, but it is stored on disk.
