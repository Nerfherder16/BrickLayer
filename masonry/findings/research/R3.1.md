The similarity distribution for this registry/model combination spans 0.32–0.85. Most agent-to-query similarities cluster in the 0.50–0.70 band. The 0.70 threshold captures only the clearest queries (campaign commands, security audits with OWASP keywords, optimize requests). The remaining 85% fall through to Layer 3 (dead) → Layer 4 (fallback).

### Margin Threshold Analysis

8/20 queries (40%) have margin >= 0.05. Of these 8:
- 7 route to plausible or correct agents
- 1 routes to debatable agent (cascade-analyst for DSPy failure analysis)

The 0.05 margin threshold is well-calibrated for quality — when the margin is >= 0.05, the routing decision is usually correct. The problem is coverage: only 40% of queries qualify.

### Recommended Parameter Adjustment

| Parameter | Current | Recommended | Effect |
|-----------|---------|-------------|--------|
| `_DEFAULT_THRESHOLD` | 0.70 | **0.60** | Increases queries reaching margin check from ~15% to ~65% |
| `_MARGIN_THRESHOLD` | N/A (not implemented) | **0.05** | Filters out thin-margin cases within those 65% |
| Combined acceptance | ~15% | ~40% | Still passes 40%, defers 60% to L3 |

**Why 0.60 and not lower**: Below 0.55, too many random non-matching agents score above threshold (e.g., spreadsheet-wizard at 0.32 would not qualify, but at 0.50 there are ~15 agents above threshold per query). The margin check handles this, but wider candidate sets increase false positives. 0.60 + 0.05 margin is the empirically-supported combination.

## Implementation

Two changes in `src/routing/semantic.py`:

```python
_DEFAULT_THRESHOLD = 0.60   # was 0.70
_MARGIN_THRESHOLD = 0.05    # new field

# In route_semantic():
sims = sorted(sims, key=lambda x: x[1], reverse=True)
if sims[0][1] >= _DEFAULT_THRESHOLD:
    if len(sims) == 1 or (sims[0][1] - sims[1][1]) >= _MARGIN_THRESHOLD:
        return RoutingDecision(target_agent=sims[0][0].name, layer="semantic", ...)
# margin too thin — fall through to Layer 3
return None
```
