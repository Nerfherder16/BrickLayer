## Summary

`run_vigil.py` flags `diagnose-analyst` as a Thorn with 0% pass rate over 8 findings. This is a false signal: the 8 findings come from `C:\Users\trg16\Dev\Bricklayer2.0\findings\` (the BL root findings directory), not from masonry self-research. These BL1.x-style diagnosis reports have `**Agent**: diagnose-analyst` but lack a `**Confidence**:` field. Vigil defaults missing confidence to 0.5, which falls below `CONFIDENCE_THRESHOLD=0.70`, causing all 8 to fail the quality gate. The diagnose-analyst agent's actual performance cannot be assessed from this data set.

### OVERCONFIDENT_PASS_RATE not the issue

Diagnose-analyst would be a Thorn via `pass_rate >= OVERCONFIDENT_PASS_RATE (0.95)` if masonry findings (conf=0.87–0.97) were being read. But since masonry/findings/ entries have no `**Agent**:` field (confirmed: all 104 masonry findings have `agent='unknown'` per the vigil regex), the OVERCONFIDENT path never triggers for diagnose-analyst. It's strictly the missing-confidence path.

### Root cause chain

1. `BLroot/findings/` contains BL1.x diagnosis reports (pre-masonry format)
2. These reports have `**Agent**: diagnose-analyst` but no `**Confidence**:` field
3. Vigil defaults missing confidence to 0.5
4. 0.5 < CONFIDENCE_THRESHOLD (0.70) → 100% failure rate over 8 findings
5. pass_rate=0% < THORN_PASS_RATE (0.50) → Thorn
6. `generate_proposals()` reads `metrics.get("diagnose-analyst", {})` → correct count/pass_rate from step 5

## Integration with Rubrics

`run_vigil.py` imports rubrics for the scored_all path:

```python
try:
    from masonry.src.scoring.rubrics import max_score as _rubric_max_score
    _HAS_RUBRICS = True
except ImportError:
    _HAS_RUBRICS = False
    def _rubric_max_score(agent_name: str) -> int:
        return 100
```

The `_HAS_RUBRICS` flag is `True` when `masonry.src.scoring.rubrics` is importable (masonry package installed). The `max_score()` function from rubrics.py returns the per-agent maximum score for normalizing the scored_all.jsonl scores. This path is only used for the **scored_all augmentation** (fleet agents not in findings) — it does not affect the diagnose-analyst calculation, which goes through the `parse_findings_dir` path.

## Why diagnose-analyst Does Not Appear in scored_all.jsonl

`scored_all.jsonl` is produced by `score_all_agents.py` which runs `score_findings.py`. The `discover_findings()` function in score_findings.py explicitly excludes `masonry/` from its scan. Diagnose-analyst findings in masonry/ (e.g., D13.1.md, D12.1.md) are never scored into scored_all.jsonl. The BLroot/findings/ D-prefix files also lack `**Agent**:` fields in the format expected by `score_findings.py`'s `backfill_agent_fields.py`. So diagnose-analyst contributes zero records to scored_all.jsonl, and vigil's scored_all augmentation loop skips it.

## Verdict

**WARNING** — The diagnose-analyst Thorn classification in run_vigil.py is a false signal caused by a format mismatch between BL1.x diagnosis artifacts (no confidence field) and the BL2.0 confidence-threshold scoring assumption. The actual diagnose-analyst performance cannot be assessed from this data. Two remediation options exist:

1. **Short fix**: Add `**Confidence**: 0.85` (or appropriate value) to the 8 BLroot diagnosis findings so vigil can evaluate them properly.
2. **Structural fix**: Vigil should separate the confidence-based path (BL2.0 findings with explicit confidence) from artifact-based diagnosis reports that lack confidence scores, or apply a neutral score (skip) rather than defaulting to 0.5 for absent confidence fields.
