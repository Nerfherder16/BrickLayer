# Finding: R33.3 — Training Corpus Adequacy: 57 Records Is Below the Binding Threshold

**Question ID**: R33.3
**Status**: DONE
**Verdict**: WARNING
**Severity**: Medium
**Confidence**: 0.87
**Agent**: research-analyst

## Summary

57 research-analyst records is insufficient to push MIPROv2 scores above the projected 70-80% ceiling. The binding constraint is **effective training set size after valset extraction**, not `project_context` emptiness. With `--valset-size 50` (R33.1 recommendation), only 7 examples remain for bootstrap training — 23% of the DSPy-documented 30-example threshold for "substantial value." Adding 30 more records (to 87 total) would raise the effective bootstrap set to 37 and likely move optimization scores above the 75% WARNING boundary. `project_context` emptiness is a secondary constraint that degrades demo quality but does not create the score ceiling.

## Hypothesis Under Test

The assumption being stress-tested: 57 research-analyst records in `scored_all.jsonl` is adequate for MIPROv2 optimization and the projected 70-80% score ceiling is caused by factors other than corpus size (e.g., `project_context` emptiness or metric ceiling).

## Evidence

### Supporting evidence — corpus size is not catastrophically small

- **[`run_optimization.py`, direct code inspection]**: The hard minimum for optimization is 3 records; a warning triggers below 10. At 57, the corpus passes all hard gates.
- **[`masonry/training_data/scored_all.jsonl`, direct measurement]**: 57 research-analyst records have `question_text` present (mean length 446 chars, median 500 chars) and verdicts are well-distributed: HEALTHY 22 (39%), WARNING 21 (37%), FAILURE 11 (19%), non-standard 3 (5%). Label balance is healthy for the 0.4-weight verdict component of the metric.
- **[DSPy docs, context7 query `/websites/dspy_ai`]**: "You can often get substantial value out of 30 examples." At 57, the raw corpus is 90% above this lower threshold.
- **[Evidence quality measurement]**: 53/57 records (93%) have evidence > 300 chars AND contain numbers/threshold language — meeting the full 0.4-weight evidence quality component of `build_metric()`. Only 4 records receive partial credit on the evidence component.

### Contradicting evidence — corpus size IS the binding constraint after valset split

- **[DSPy docs, context7 `/websites/dspy_ai`]**: "Aim for at least 300 examples." 57 is 19% of the target. For prompt optimizers, the recommended data split is 20% train / 80% validation — applying this to 57 records gives 11 training examples and 46 validation examples.
- **[`run_optimization.py` line 50 + `optimizer.py` line 233-239, direct code]**: The actual data split used is `total_corpus − valset_size = training_examples`. With `--valset-size 50` (R33.1 recommendation): 57 − 50 = **7 effective training examples**. With the default `--valset-size 100`: MIPROv2 clips to `min(100, 57) = 57` for validation, leaving **0 examples** for bootstrap training.
- **[`optimizer.py` line 234-235]**: `max_bootstrapped_demos=3, max_labeled_demos=3`. With only 7 bootstrap examples, MIPROv2 can select at most 3 bootstrapped demos but has minimal diversity to choose from. The few-shot quality ceiling is set by bootstrap diversity, not instruction quality.
- **[DSPy docs, context7 `/websites/dspy_ai`]**: "Prompt-based optimizers often overfit to small training sets." With 7 bootstrap examples, overfitting risk on the training partition is high even when the valset is adequate.
- **[`masonry/training_data/scored_all.jsonl`, direct measurement]**: All 57 research-analyst records have `project_context = ""` (empty string). `ResearchAgentSig` defines `project_context` as a named `InputField` — bootstrapped demos will always show the agent operating with no project context, meaning the optimized instructions cannot be validated against context-enriched reasoning patterns.

### Incremental gain analysis: 30 more records vs. `project_context` enrichment

**Option A — Add 30 more records (87 total):**
- With `--valset-size 50`: 87 − 50 = 37 effective bootstrap examples (123% of the 30-example DSPy threshold)
- Bootstrap demo pool increases from 7 to 37: 5.3x more diversity for the 3 selected demos
- Verdict distribution maintained (already balanced)
- Expected score improvement: moderate-to-substantial (moving from below-threshold to above-threshold bootstrap pool)
- Estimated score: potentially 73-82% range (above WARNING threshold of 75%)

**Option B — Enrich `project_context` in existing 57 records:**
- The metric (`build_metric()`) scores `verdict`, `evidence`, and `confidence` — it does NOT score `project_context` quality directly
- Enriching `project_context` improves demo quality (bootstrapped examples would show richer context-aware reasoning) but does not directly improve the metric score during optimization
- Primary effect: better few-shot demo realism; secondary effect: potentially higher verdict accuracy on context-dependent questions
- Expected score improvement: marginal (metric is context-blind)
- Constraint: Requires re-running the training data pipeline (`build_dataset()` already loads `project-brief.md` per finding — the issue is that `load_training_data_from_scored_all()` hardcodes `project_context: ""` rather than calling `_load_project_brief()`)

**Which constraint is more binding:** Corpus size (Option A) is more binding than `project_context` emptiness (Option B). The metric does not penalize empty `project_context`, so Option B cannot improve the optimization score. Option A directly increases bootstrap diversity, which is the principal lever for MIPROv2 few-shot quality.

## Threshold Analysis

- **DSPy documented threshold**: "substantial value out of 30 examples" (training examples, not total corpus)
- **Effective training examples with `--valset-size 50`**: 7
- **Gap**: 7/30 = 23% of threshold → **WARNING** (well below threshold)
- **Effective training examples with `--valset-size 100` (default)**: 0 (valset clips to all 57)
- **Gap at default**: 0/30 = 0% of threshold → **FAILURE** condition if default is used
- **DSPy target**: 300 total examples; 57 is 19% of target

The projected 70-80% score ceiling cited in the question aligns with the metric structure ceiling analysis:
- Metric max = 1.0 (0.4 verdict + 0.4 evidence + 0.2 confidence)
- `quantitative-analyst.json` baseline score: 66.07 (on 0-100 scale = 0.66 normalized), established with a larger and richer corpus
- With 7 bootstrap examples, the instruction optimizer has limited demo quality to work from; the ceiling is not the metric formula but the bootstrap diversity

## Verdict

**WARNING.** The 57-record corpus is not adequate when valset requirements are accounted for. The effective bootstrap training set after applying `--valset-size 50` is 7 examples — 23% of the DSPy-documented 30-example threshold. This is the primary ceiling constraint, not `project_context` emptiness (which is secondary: it degrades demo realism but the optimization metric does not score `project_context`). Adding 30 records would raise the effective training pool to 37 (above the 30-example threshold) and is the highest-leverage intervention. `project_context` enrichment should be addressed in the `load_training_data_from_scored_all()` function (which hardcodes `project_context: ""`), but this is a lower-priority fix than raw record count.

The projected 70-80% ceiling is plausible for the current corpus configuration, but is not a hard metric ceiling — it reflects the bootstrap diversity constraint. Reaching above 75% requires either expanding the corpus or reducing `--valset-size` further (e.g., `--valset-size 30`), which trades evaluation stability for bootstrap richness.

## What Would Change This Verdict

- **HEALTHY**: Evidence that MIPROv2 achieves scores consistently above 75% with 7 bootstrap examples on a comparable agent (the `quantitative-analyst.json` score of 66.07 with its corpus does not demonstrate this). Alternatively, if the actual E33.1 run produces a score > 75% with `--valset-size 50`, this verdict should be revised to HEALTHY.
- **FAILURE**: If the default `--valset-size 100` is used (not the R33.1-recommended 50), the effective training set is 0 — this is a hard failure condition.
- **INCONCLUSIVE remains excluded**: All calculations are deterministic from code and DSPy documentation; the actual E33.1 run score is not required to assess corpus adequacy.
