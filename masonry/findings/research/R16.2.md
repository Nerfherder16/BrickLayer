# Finding: R16.2 — "Agents" Column Inconsistency Is Accidental: score_findings Lacks `agents_covered` Key

**Question ID**: R16.2
**Status**: DONE
**Verdict**: WARNING
**Severity**: Low
**Confidence**: 0.91
**Agent**: research-analyst

## Summary

The "Agents" column inconsistency in `score_all_agents.py` summary is accidental, not intentional. `score_findings.run()` returns `agents_with_10_plus` dict (not `agents_covered`), so `score_all_agents.py` uses `agents_with_10_plus` keys as a proxy for `agents_covered`. Other scorers (score_code_agents, score_ops_agents) return `agents_covered` directly. This causes the same "Agents" column to mean different things per row: `agents_with_10_plus` count for score_findings (1 = quantitative-analyst only), total agents covered for others (2, 2). The TOTAL row is separately correct (9 unique agents from merged). Fix is minimal: either add `agents_covered` key to `score_findings.run()`, or document the inconsistency explicitly in the column header. Both options are low-effort and low-risk.

## Evidence

### Summary table (current output from score_all_agents.py)

```
--- Score All Agents Summary ---
Scorer                        Records     Agents   Agents w/10+
------------------------------------------------------------------
score_findings                     61          1              1
score_code_agents                   4          2              0
score_ops_agents                  171          2              0
score_routing                       0          0              0
------------------------------------------------------------------
TOTAL                             236          9              2
```

"Agents" column per-row represents different populations:
- `score_findings` row: 1 = `len(agents_with_10_plus)` = agents with ≥10 records only
- `score_code_agents` row: 2 = `len(agents_covered)` = all agents with any records
- `score_ops_agents` row: 2 = `len(agents_covered)` = all agents with any records
- `score_routing` row: 0 = hardcoded empty list

"Agents w/10+" column:
- `score_findings` row: 1 = same as "Agents" (both are agents_with_10_plus count) → meaningless redundancy
- Other rows: 0 = hardcoded

TOTAL row: "Agents" = 9 from `len(agent_record_counts)` (unique agents in merged scored_all.jsonl) — **correct**

### Root cause: score_findings.run() return schema differs

`score_findings.run()` returns:
```python
{
    "scanned": 688,
    "training_ready": 61,
    "agents_with_10_plus": {"quantitative-analyst": 36},  # ← no "agents_covered" key
    "output_path": "...",
}
```

`score_code_agents.run()` and `score_ops_agents.run()` return:
```python
{
    "training_ready": N,
    "agents_covered": ["karen", "git-nerd"],  # ← has "agents_covered" key
    ...
}
```

`score_all_agents.py` line 236:
```python
# score_findings:
"agents_covered": list(sf_summary.get("agents_with_10_plus", {}).keys()),  # ← uses 10+ keys as proxy
"agents_10plus": len(sf_summary.get("agents_with_10_plus", {})),            # ← same count again

# score_code_agents:
"agents_covered": sc_summary.get("agents_covered", []),   # ← uses real agents_covered
"agents_10plus": 0,  # code agents rarely hit 10+ in a single project  ← hardcoded
```

The inconsistency follows directly from `score_findings` not returning `agents_covered`. It was not designed to show "Agents w/10+" for findings — it happened because `agents_with_10_plus` was the only available count-like field.

### Is it intentional?

No evidence of intentional design:
- No comment explaining "we use 10+ keys here for a reason"
- The column header "Agents" without qualification implies total agents covered
- The duplicate `agents_covered` / `agents_10plus` = same value for score_findings is redundant, suggesting the author didn't realize both columns were showing the same thing
- The code comment for code agents: `"agents_10plus": 0,  # code agents rarely hit 10+ in a single project` — if the author intended score_findings to show 10+ count specially, they would have used the same pattern for code/ops agents

### Impact

- Misleading: score_findings "Agents: 1" looks like only 1 agent contributes, but 5 actually do (quantitative-analyst=36, synthesizer-bl2=9, competitive-analyst=6, regulatory-researcher=5, research-analyst=5)
- TOTAL "Agents: 9" is correct and unaffected
- Downstream: no effect on actual training data (scored_findings.jsonl, scored_all.jsonl) — display-only issue

### Correct unified metric

For the "Agents" column, all scorers should show `total agents with any records`:
- score_findings: 5 (not 1)
- score_code_agents: 2 (unchanged)
- score_ops_agents: 2 (unchanged)
- score_routing: 0 (unchanged)
- TOTAL: 9 (unchanged)

Minimum fix — add `agents_covered` key to `score_findings.run()`:
```python
# In score_findings.run() return dict:
return {
    "scanned": len(paths),
    "training_ready": len(training_records),
    "agents_covered": list(agent_counts.keys()),        # ← ADD THIS
    "agents_with_10_plus": agents_with_10_plus,
    "output_path": str(output_path),
}
```

Then `score_all_agents.py` line 236:
```python
"agents_covered": list(sf_summary.get("agents_covered", sf_summary.get("agents_with_10_plus", {}).keys())),
```

## Verdict

**WARNING** — The "Agents" column inconsistency is accidental, caused by `score_findings.run()` not returning an `agents_covered` key. It misleads users into thinking only 1 agent contributes to findings training data (actual: 5). Low-effort fix: add `agents_covered` to `score_findings.run()` return dict. Low priority since this is display-only and doesn't affect training data quality.
