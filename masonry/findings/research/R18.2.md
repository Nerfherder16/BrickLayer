## Summary

The quantitative-analyst agent has 36 training records — numerically above the DSPy MIPROv2 practical minimum of 20-50 — and the optimizer is fully implemented functional code (not a stub). However, three blockers prevent an immediate clean optimization run: (1) `ANTHROPIC_API_KEY` is not set in the environment, which will cause `configure_dspy()` to fail at the `dspy.LM()` call; (2) the 36 training records in `scored_findings.jsonl` all carry `score: 60` and `confidence: null`, indicating the scorer applied a minimum floor rather than genuine quality differentiation; (3) the training records carry only `question_text` in their `input` dict and lack `project_context` and `constraints`, requiring `build_dataset()` (not the pre-built JSONL) to be the actual data source for a properly shaped trainset.

## Evidence

### Optimizer code status

`masonry/src/dspy_pipeline/optimizer.py` is fully implemented. It:
- Defines a heuristic `build_metric()` (no LLM judge needed — uses verdict match 0.4, evidence length 0.4, confidence calibration 0.2)
- Calls `dspy.MIPROv2(metric=metric, num_threads=1)`
- Compiles with `max_bootstrapped_demos=3, max_labeled_demos=3`
- Falls back gracefully if MIPROv2 raises (writes minimal JSON rather than crashing)
- Uses `ResearchAgentSig` (not `QuantitativeAnalystSig`) as confirmed in `optimize_all()` code comment at line 196

DSPy version installed: **3.1.3** (confirmed via `import dspy; dspy.__version__`). No import errors.

### Training data quality — BLOCKER 1: uniform scores

All 36 QA records carry `score: 60` — the minimum passing threshold. No record scores above 60. This means `score_example()` in `training_extractor.py` would classify all 36 as "silver" (score 0.5-0.8 → weight 0.7), with zero "gold" examples (score > 0.8 → weight 1.0). MIPROv2 will treat all demonstrations as equal quality. The optimizer can still run, but the bootstrapped demos will not be quality-stratified.

### Training data schema — BLOCKER 2: field mismatch in pre-built JSONL

The `scored_findings.jsonl` records have this input structure:
```json
{"question_id": "1.1", "question_text": "..."}
```
`project_context` is absent in 36/36 records. `constraints` is absent in 36/36 records.

`ResearchAgentSig` requires `question_text`, `project_context`, and `constraints` as input fields. Running `optimize_agent()` directly from the JSONL data would produce trainset examples with empty `project_context` and `constraints`. This degrades optimization quality but does not cause a hard crash (both fields have `default=""` in the signature).

`build_dataset()` in `training_extractor.py` correctly shapes records to `ResearchAgentSig` fields (lines 262-272), loading `project_context` from `project-brief.md` at training time. The optimization entry point must use `build_dataset()` — not the raw JSONL — to get properly populated examples.

### Confidence null values — BLOCKER 3 (minor)

All 36 QA training records have `"confidence": null`. The metric's `confidence_calibration` component will fail to parse `null` as a float and fall through to `score += 0.0` for that component (0.2 weight). This means the metric is effectively a 0.4 + 0.4 two-component metric during optimization, not the intended three-component one. The metric still functions; it simply cannot tune toward calibrated confidence outputs.

### Anthropic API key — BLOCKER 4 (hard blocker)

```
ANTHROPIC_API_KEY set: False (length 0)
```

`configure_dspy()` calls `dspy.LM("anthropic/claude-sonnet-4-6")`, which will raise an authentication error immediately. The optimizer will not run at all without this key in the environment.

Ollama is reachable (`192.168.50.62:11434`) with models `qwen2.5:7b`, `qwen3:14b`, and others available. DSPy supports Ollama via `dspy.LM("ollama/qwen2.5:7b", api_base="http://192.168.50.62:11434")`. This is a viable alternative to Anthropic for the optimization run itself.

### Signature mismatch (non-blocking observation)

The auto-generated `QuantitativeAnalystSig` in `generated/quantitative-analyst.py` has different fields from `ResearchAgentSig` (uses `question`/`context` inputs vs. `question_text`/`project_context`/`constraints`). However, `optimize_all()` explicitly uses `ResearchAgentSig` for all agents (documented in the code comment at line 196 referencing R5.2), so the generated signature file is not used in the optimization path. This is consistent and not a blocker.

- If all four blockers are resolved, verdict flips to HEALTHY — the quantity and code readiness are sufficient

One is a hard blocker today: `ANTHROPIC_API_KEY` is absent. If the Ollama alternative is used (`qwen3:14b` is available at the target endpoint), the optimization trial can run without an Anthropic key. The training corpus is large enough, the optimizer is production-grade code, and the metric will produce a meaningful signal. This is a WARNING, not a FAILURE — the path to a successful trial is clear and requires no structural changes.
