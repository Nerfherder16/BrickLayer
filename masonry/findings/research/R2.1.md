# Finding: R2.1 — Threshold 0.70 Matches Model's Training-Data Positive-Pair Cutoff; Not Discriminative for Intra-Domain Registry

**Question ID**: R2.1
**Status**: DONE
**Verdict**: FAILURE
**Agent**: research-analyst
**Severity**: High
**Confidence**: 0.75

## Summary

The `_DEFAULT_THRESHOLD = 0.70` in `semantic.py` sits at the exact lower bound of `qwen3-embedding:0.6b`'s positive training distribution — the Qwen3 paper explicitly states training data was filtered by retaining pairs with cosine similarity > 0.70. This means the threshold is the edge of where the model considers things "similar," not the midpoint of a wide discriminative gap.

The Masonry registry contains at least three tight semantic clusters (research: 4 agents; audit/review: 5 agents; code/implementation: 4 agents) where descriptions are near-synonyms. For queries targeting any cluster, 2–4 agents simultaneously score above 0.70. The routing code uses `max()` with no margin check, so a 0.001 score difference deterministically routes to an agent with no uncertainty signal. Community reports confirm the model's score floor for short English text is ~0.50, compressing the entire ambiguous zone into 0.50–0.70 rather than distributing it below 0.30 as the model card examples suggest.

## Evidence

### 1. Training-Data Cutoff Equals Threshold (Primary Source)

From the Qwen3 Embedding technical paper (arXiv 2506.05176):
> "We employ a simple cosine similarity calculation to select data pairs, retaining those with a cosine similarity greater than 0.70 from randomly sampled data."

The 0.70 threshold in `semantic.py` is identical to the model's training-data positive-pair selection boundary. Routing above this threshold is equivalent to asking "is this a training-positive pair?" — the answer will be yes for the correct agent, but also yes for near-synonym agents in the same semantic cluster, because the model was trained to score similar-domain pairs near this boundary.

### 2. Score Floor ~0.50 on Short English Text

Multiple independent community reports (r/LLMDevs, July 2025) confirm:
- "None of my tests had it drop below 50%" for short English text comparisons
- "Bunches together scores" — the distribution is compressed into a narrow band
- "Matches on irrelevant things" — false positives above 0.70 for topically adjacent content

This compresses the entire ambiguous region into 0.50–0.70, meaning unrelated agents in the same domain (AI software tools) will regularly score in the 0.60–0.75 band rather than the 0.10–0.40 band shown in the model card's idealized examples.

Manual inspection of `agent_registry.yml` (47 agents) reveals near-synonym clusters:

For a query like "research how other teams have solved this problem," all four research-cluster agents likely score above 0.70 simultaneously.

### 4. Routing Code Has No Margin Check

From `semantic.py` (lines 118-128):
```python
agent_embs = [(a, _embedding_cache[_agent_corpus_key(a)]) for a in registry ...]
sims = [(a, cosine_sim(query_emb, emb)) for a, emb in agent_embs]
best, score = max(sims, key=lambda x: x[1])
if score >= _DEFAULT_THRESHOLD:
    return RoutingDecision(target_agent=best.name, layer="semantic", confidence=score, ...)
return None
```

The code takes `max()` and returns if score ≥ 0.70. No check of `best_score - second_best_score`. A margin of 0.001 routes with the same confidence as a margin of 0.20. For near-synonym agents, the margin is expected to be < 0.05.

### 5. MTEB Benchmark Context

`qwen3-embedding:0.6b` scores 70.70 on MTEB English v2 (strong for its size class). However, MTEB aggregates over diverse tasks including clearly dissimilar-domain retrieval. The intra-domain short-text disambiguation problem (47 functionally similar AI tool agents) is precisely the hardest case for embedding models and is not representative of MTEB averages.

## Threshold Analysis

| Zone | Expected Similarity Range |
|------|--------------------------|
| Different domain (truly unrelated) | 0.50–0.60 |
| Same domain, different task | 0.60–0.73 |
| Near-synonym cluster member | 0.68–0.80 |
| Correct agent | 0.70–0.85 |
| Perfect paraphrase | 0.85+ |

The threshold 0.70 sits inside the overlap zone between "near-synonym cluster member" and "correct agent." The max() selector can resolve within the overlap most of the time, but with margins so thin that small phrasing variations flip results.

## What Would Change This Verdict

- **FAILURE → WARNING**: Live measurement shows mean best-vs-second-best margin > 0.08 across 20+ representative queries against the actual registry
- **WARNING → CALIBRATED**: Mean margin > 0.12 AND fewer than 5% of queries have competing agents scoring above 0.70

A 20-line calibration script calling the Ollama endpoint with sample user prompts against the live registry would resolve this definitively.

## Recommended Fix

1. **Add margin check**: Route via semantic layer only if `best_score - second_best_score >= 0.05`. Queries with insufficient margin fall through to Layer 3 (LLM router). This prevents confident misroutes on near-synonym ambiguity.

2. **Raise threshold to 0.75–0.80** for this registry: Pushes below the positive training distribution boundary, reducing false-positive same-cluster matches. Combine with live calibration.

3. **Mode pre-filter**: Use the agent `modes` field (already in registry) as a deterministic pre-filter before semantic scoring. A query with `**Mode**: diagnose` eliminates all non-diagnose agents from the cosine comparison, tightening the candidate set from 47 to ~8 agents.

4. **Run calibration**: Execute the 20-query benchmark before committing to a new threshold. The measurement is the R2.2 complement for semantic rather than LLM layer.
