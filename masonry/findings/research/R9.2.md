# Finding: R9.2 — Deterministic Layer Rule 5 (Mode Field) Never Fires for Campaign Questions; R1.2 Coverage Estimate Stands With Downward Revision

**Question ID**: R9.2
**Status**: DONE
**Verdict**: WARNING
**Agent**: research-analyst
**Severity**: Medium
**Confidence**: 0.92

## Summary

R1.2 (Wave 1) already established the deterministic routing 60%+ claim is UNCALIBRATED — actual mixed-session coverage is 15-25%. This investigation adds one new finding: **Rule 5 (`**Mode**:` field) never fires for standard campaign question routing** because the regex `r"\*\*Mode\*\*:\s*(\w+)"` does not match `**Operational Mode**:` (the format used in `questions.md`). The Wave 1 estimate of "5-10% from Rule 5" is therefore 0% in practice for campaign-routed questions, lowering the mixed-session floor to ~10-15%.

## Evidence

### Rule 5 Regex vs. Actual Field Format

From `deterministic.py` line 36:
```python
_MODE_FIELD_RE = re.compile(r"\*\*Mode\*\*:\s*(\w+)", re.IGNORECASE)
```

This pattern requires the field to be named exactly `**Mode**:` (case-insensitive).

From `masonry/questions.md` (standard BL 2.0 question format):
```markdown
**Status**: PENDING
**Operational Mode**: research
**Priority**: HIGH
**Method**: research-analyst
```

The field is `**Operational Mode**:` — the regex does not match because `Operational Mode` ≠ `Mode`. Rule 5 returns no match for every standard BL 2.0 campaign question.

### Implication for R1.2 Estimate

R1.2 estimated Rule 5 contributes "5-10% of mixed-session coverage." That estimate assumed campaign questions were routed through the deterministic layer with a `**Mode**:` field. Since they use `**Operational Mode**:` instead, Rule 5 contributes ~0%, not 5-10%.

Revised estimates:

| Session Type | R1.2 Estimate | Revised Estimate |
|-------------|---------------|-----------------|
| Campaign active (masonry-state.json) | ~95% | ~95% (unchanged — Rule 3 fires) |
| Autopilot build active | ~90%+ | ~90%+ (unchanged — Rule 2 fires) |
| Mixed session (no active state) | 15-25% | 10-15% |
| Pure conversation | 0% | 0% |

### Campaign State Path (Rule 3) Verification

`masonry-state.json` IS written by multiple hooks (masonry-observe.js, masonry-subagent-tracker.js). During an active BrickLayer campaign, `mode=campaign` and `active_agent` fields are set. Rule 3 fires correctly during campaigns. The R1.2 estimate of ~95% for active campaigns is confirmed.

### Missing Slash Commands (R1.4 cross-reference)

As noted in R1.2, the `_SLASH_COMMANDS` table covers only 6 of the ~14 Masonry skills. This gap was documented in R1.4. Still unresolved.

## Root Cause

The `**Mode**:` regex was written to match a simplified field format. The actual questions.md format evolved to use `**Operational Mode**:` for clarity. The two were never reconciled. This is a documentation/implementation drift bug.

## Mitigation

Fix the regex to handle the actual field name:
```python
_MODE_FIELD_RE = re.compile(r"\*\*(?:Operational\s+)?Mode\*\*:\s*(\w+)", re.IGNORECASE)
```

Or alternatively match `**Method**:` (the agent attribution field also present in questions):
```python
_MODE_FIELD_RE = re.compile(r"\*\*(?:Operational\s+)?Mode\*\*:\s*(\w+)", re.IGNORECASE)
_METHOD_FIELD_RE = re.compile(r"\*\*Method\*\*:\s*(\S+)", re.IGNORECASE)
```

Either fix would enable Rule 5 to route campaign questions deterministically without needing `masonry-state.json`.

## Verdict

**WARNING** — Deterministic layer Rule 5 is silently broken for the standard BL 2.0 question format. Coverage for mixed sessions is ~10-15%, not the claimed 60%+. Coverage during active campaign/build sessions remains high (~90-95%) via Rules 2-3. The 60%+ claim is context-dependent and overstated for the general case. Fix: extend `_MODE_FIELD_RE` to match `**Operational Mode**:`.
