# Finding: R1.2 — Deterministic Coverage Claim of 60%+ Is Overstated; Actual Coverage ~25-40% for Mixed Sessions

**Question ID**: R1.2
**Status**: DONE
**Verdict**: UNCALIBRATED
**Agent**: research-analyst
**Severity**: Medium
**Confidence**: 0.85

## Summary

The 60%+ deterministic coverage claim appears in the `deterministic.py` docstring but was not measured — it is an assertion. Code analysis shows the deterministic layer only handles 6 slash commands, 3 autopilot state values, 1 campaign state condition, 2 UI state values, and Mode field matching. For mixed-use development sessions (the dominant use case), actual coverage is estimated at 25-40%. For pure BrickLayer campaign sessions, coverage may reach 60%+ but only while an autopilot or campaign state file is active.

## Evidence

### Deterministic Layer Rules Inventory

From `deterministic.py`, the complete rule set:

| Rule | Trigger | Target |
|------|---------|--------|
| 1. Slash commands | `/plan`, `/build`, `/fix`, `/verify`, `/bl-run`, `/masonry-run` | Various workflows |
| 2. Autopilot state | `.autopilot/mode` = `build`, `fix`, or `verify` | Build/fix/verify workflows |
| 3. Campaign state | `masonry-state.json` mode=campaign + active_agent | Active campaign agent |
| 4. UI state | `.ui/mode` = `compose` or `review` | UI workflows |
| 5. Mode field | `**Mode**:` field in request text | Registry agent for mode |

**Coverage analysis by session type:**

### Session Type A: Pure BrickLayer Campaign (Active State)

During an active BrickLayer campaign, `masonry-state.json` has `mode=campaign` and `active_agent` set. Rule 3 fires for every request during the campaign. Coverage: **~95%** (nearly all requests during a campaign are routed by Rule 3).

### Session Type B: Autopilot Build Active

During `/build`, `.autopilot/mode` = "build". Rule 2 fires for all requests. Coverage: **~90%+** (build sessions are dominated by build-related requests).

### Session Type C: Mixed Development Session (Common Case)

In a typical development session (no active campaign, no active build):
- No `.autopilot/mode`, no `masonry-state.json`, no `.ui/mode`
- Rules 2, 3, 4 produce zero matches
- Only Rule 1 (slash commands) and Rule 5 (Mode field) apply

Rule 1 coverage estimate: Slash command requests are a minority of all requests. In a typical development session with 50 requests: ~5-8 might be slash commands (`/build`, `/plan`, etc.). That's 10-16%.

Rule 5 coverage estimate: BrickLayer questions with `**Mode**:` fields are also a minority outside campaign mode. In a mixed session, these might represent 5-10% of requests.

**Total for mixed session: ~15-25% deterministic coverage.**

### Session Type D: Interactive Conversations

Questions like "what's wrong with this code?", "explain X", "how do I Y" — these match none of the 5 deterministic rules. 0% deterministic coverage.

### Gap Analysis: The 60% Claim

The 60% claim is only achievable during active campaign/autopilot state (Sessions A and B). For the broad category of mixed development sessions (Session C) and conversational interactions (Session D), actual coverage is 15-25%.

The claim would be valid if Masonry is primarily used in campaign and build contexts — which is plausible for a dedicated BrickLayer research environment. But for a general-purpose session (the majority of Claude Code usage), the coverage is substantially below 60%.

### Additional Gap: Slash Command Coverage

The 6 hardcoded slash commands in `_SLASH_COMMANDS` are a subset of the full Masonry skill set. From the `skills/` directory, the following are NOT in the deterministic table:
- `/masonry-init` (masonry-init.md)
- `/masonry-status` (masonry-status.md)
- `/masonry-fleet` (masonry-fleet.md)
- `/masonry-ultrawork` (masonry-ultrawork.md) — actually `/ultrawork` in CLAUDE.md
- `/masonry-pipeline` (masonry-pipeline.md)
- `/masonry-team` (masonry-team.md)
- `/masonry-code-review` (masonry-code-review.md)
- `/masonry-security-review` (masonry-security-review.md)

Each of these 8 missed slash commands falls through to Layer 2/3 instead of getting instant deterministic routing. This is covered in detail in R1.4.

## Mitigation

1. **Recalibrate the claim**: Change docstring from "60%+" to "60%+ during active campaign/build states; 15-25% for mixed sessions."
2. **Add missing slash commands** (see R1.4): Extend `_SLASH_COMMANDS` to cover all registered Masonry skills.
3. **Session-aware coverage reporting**: Log which rule matched (or that no rule matched) for each routing call to build empirical coverage data over time.
