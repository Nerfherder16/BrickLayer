## Hypothesis Under Test

Claude Code's native session credentials (OAuth token) can be forwarded to child Python processes to unblock MIPROv2 optimization runs that currently fail with ANTHROPIC_API_KEY not set.

---

## Evidence

### Supporting evidence

**E1 — ANTHROPIC_AUTH_TOKEN is a recognized LiteLLM alternative to ANTHROPIC_API_KEY**
Source: litellm 1.82.5 source (litellm/utils.py, lines 6175 and 6417), direct inspection.
Both `ANTHROPIC_API_KEY` and `ANTHROPIC_AUTH_TOKEN` satisfy LiteLLM's key-check logic. Setting either env var marks `keys_in_environment = True` and suppresses the missing-key error. This is a first-class supported path, not a workaround.

**E2 — dspy.LM() accepts api_key as a kwargs pass-through to litellm.completion()**
Source: dspy source inspection (dspy.LM.__init__, dspy.LM.forward), direct inspection.
`dspy.LM(model, **kwargs)` stores kwargs at `self.kwargs` and merges them into every `litellm.completion()` call via `request=dict(model=..., messages=..., **kwargs)`. `litellm.completion` has `api_key` as a named parameter. This means `dspy.LM("anthropic/claude-sonnet-4-6", api_key="sk-ant-...")` is a valid, direct path — no env var needed.

**E3 — litellm.anthropic_key module attribute is settable at runtime**
Source: litellm source inspection, direct inspection.
`litellm.anthropic_key` defaults to `None` but is read by `get_api_key()` before falling back to `get_secret("ANTHROPIC_API_KEY")`. Setting `litellm.anthropic_key = "sk-ant-..."` before calling `configure_dspy()` would inject the key without touching env vars.

**E4 — configure_dspy() does NOT pass api_key — relies entirely on env var**
Source: `masonry/src/dspy_pipeline/optimizer.py`, `configure_dspy()` function, lines 149-168.
Current call: `dspy.LM("anthropic/claude-sonnet-4-6")` — no api_key argument. This is the root of the BLOCKED state in F30.2, F29.3, F28.2.

**E5 — No ANTHROPIC_* env vars are present in bash subprocesses**
Source: direct bash subprocess test: `python -c "import os; keys = [k for k in os.environ if 'ANTHROPIC' in k ...]"` → `[]`
Claude Code does not inject ANTHROPIC_API_KEY or ANTHROPIC_AUTH_TOKEN into child process environments.

**E6 — Claude Code OAuth token IS present in AppData but is encrypted (Chromium DPAPI / v10)**
Source: `C:\Users\trg16\AppData\Roaming\Claude\config.json` key `oauth:tokenCache`, direct inspection.
The config file contains a 1980-char `oauth:tokenCache` value. Base64-decoding reveals a `v10` prefix — this is Chromium Electron's `safeStorage` encryption using Windows DPAPI. The plaintext token cannot be read by a Python subprocess without calling the Win32 `CryptUnprotectData` API within the same user session and application context. While technically possible via `win32crypt`, this is fragile, undocumented, and depends on Electron's key derivation. This path is NOT recommended.

**E7 — CLAUDECODE=1 env var is present**
Source: direct bash subprocess test.
Only `CLAUDECODE`, `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE`, `CLAUDE_CODE_ENTRYPOINT`, `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`, `CLAUDE_CODE_SSE_PORT` are injected. None carry API credentials.

**E8 — Ollama backend (fallback path) is currently unreachable**
Source: direct connectivity tests.
`http://192.168.50.62:11434` (configured Ollama host): timed out. `http://127.0.0.1:11434` and `http://localhost:11434`: connection refused. No local `ollama` binary found on PATH. The `--backend ollama` flag in `run_optimization.py` is non-functional in the current environment.

### Contradicting evidence

**C1 — ANTHROPIC_AUTH_TOKEN is not the same credential type as an API key**
LiteLLM accepts it in key-existence checks (lines 6175, 6417) but `get_api_key()` reads only `ANTHROPIC_API_KEY` (not `ANTHROPIC_AUTH_TOKEN`) when resolving the actual key value for the HTTP Authorization header. The auth token path is conditionally supported depending on LiteLLM version. For 1.82.5, direct testing would be needed to confirm.

### Analogues

- LiteLLM's `litellm.api_key` and provider-specific attributes (e.g. `litellm.anthropic_key`) are the standard pattern for injecting keys programmatically in Python applications that cannot use env vars — used by LangChain, LlamaIndex, and other frameworks as their primary credential injection mechanism.

## Threshold Analysis

No `constants.py` thresholds directly apply to this infrastructure question. The relevant test is binary: does a concrete, low-friction path exist to unblock optimization runs?

- **Path 1 (env var export, lowest friction)**: Export `ANTHROPIC_API_KEY` before launching Kiln or before invoking `run_optimization.py`. Requires manual one-time action. This is the minimum-friction path that works today with zero code changes.
- **Path 2 (api_key kwarg, zero friction at runtime)**: Modify `configure_dspy()` in `optimizer.py` to accept an `api_key` parameter and pass it to `dspy.LM()`. Kiln's optimization trigger can source the key from a `.env` file, environment, or a secure credential store. One code change, no operational burden.
- **Path 3 (litellm.anthropic_key module attribute)**: Set `litellm.anthropic_key` before calling `configure_dspy()`. Works but is less explicit than Path 2.
- **Path 4 (OAuth DPAPI decrypt)**: Technically possible via `win32crypt.CryptUnprotectData` in Python, but fragile and dependent on undocumented Electron internals. Not recommended.
- **Path 5 (Ollama backend)**: Requires the Ollama server at 192.168.50.62 to be reachable. Currently offline. Not viable without network fix.

Verdict is WARNING (not FAILURE) because a concrete unblock path (Path 1 or Path 2) exists and is low-cost to implement. The assumption that "credentials can be sourced from Claude Code's native session" is FALSE, but the problem is solvable without that.

### Durable fix (one code change in optimizer.py):

Modify `configure_dspy()` to accept an optional `api_key` parameter:

```python
def configure_dspy(model=None, backend="anthropic", api_key=None):
    if backend == "ollama":
        lm = dspy.LM(f"ollama_chat/{model or 'qwen3:14b'}", api_base="http://192.168.50.62:11434", max_tokens=4096)
    else:
        kwargs = {}
        if api_key:
            kwargs["api_key"] = api_key
        lm = dspy.LM(f"anthropic/{model or 'claude-sonnet-4-6'}", **kwargs)
    dspy.configure(lm=lm)
```

Then Kiln's optimize trigger reads the key from env and passes it through. This is the recommended path — it keeps credentials out of the file and makes the injection explicit.
