# Finding: R3.2 ‚Äî `build_dataset()` Produces Zero Training Examples from All 35 Findings Due to Missing Agent Attribution Field

**Question ID**: R3.2
**Status**: DONE
**Verdict**: FAILURE
**Agent**: research-analyst
**Severity**: High
**Confidence**: 1.0

## Summary

`build_dataset()` in `src/dspy_pipeline/training_extractor.py` is **completely non-functional** against the current finding format. All 35 findings (Waves 1‚Äì3) are excluded from training data because:

1. `extract_finding()` does not extract an `"agent"` field from finding files
2. `score_example()` requires `finding.get("agent")` ‚Äî returns `None` for all findings
3. `score_example()` returns `0.0` for any finding without `"agent"` field
4. `build_dataset()` skips any finding with weight `== 0.0`

**Result**: `build_dataset()` returns `{}` (empty dict) for all 35 findings. The DSPy training pipeline has produced zero valid training examples despite accumulating 35 findings across three campaign waves.

## Evidence

### The Missing Agent Field in `extract_finding()`

`extract_finding()` (lines 41-81) extracts these fields from finding files:
- `question_id` ‚Äî from `# Finding: X.Y` header
- `verdict` ‚Äî from `**Verdict**: ...`
- `severity` ‚Äî from `**Severity**: ...`
- `evidence` ‚Äî from `## Evidence` section
- `mitigation` ‚Äî from `## Mitigation` section (with None-filter)
- `source_file` ‚Äî the file path

**Missing**: No code extracts an `"agent"` field. There is no regex for `**Agent**: ...` or any attempt to infer the agent from finding metadata.

### `score_example()` Agent Requirement

```python
def score_example(finding: dict[str, Any], agent_db: dict[str, Any]) -> float:
    agent_name = finding.get("agent")  # ‚Üê always None ‚Äî field doesn't exist in dict
    if not agent_name:
        return 0.0  # ‚Üê ALL findings excluded here
    ...
```

### `build_dataset()` Exclusion Logic

```python
for finding in findings:
    weight = score_example(finding, agent_db)
    if weight == 0.0:  # ‚Üê ALL findings skipped
```

## Root Cause

Two independent design gaps:

1. **Finding template gap**: The BL 2.0 finding format does not include an `**Agent**:` field in finding files. Agent attribution exists in `questions.md` but not in findings.

2. **Extractor gap**: `extract_finding()` does not attempt to extract agent attribution, and `score_example()` requires it, creating a silent all-exclusion condition.

## Fix Specification

**Option A (minimal)**: Add agent attribution lookup from questions.md to `extract_training_data()`:

```python
def _build_qid_to_agent_map(questions_md_path: Path) -> dict[str, str]:
    """Extract question_id ‚Üí agent mapping from questions.md."""
    text = questions_md_path.read_text(encoding="utf-8")
    pattern = re.compile(
        r"###\s+(\w+\d+\.\d+):.*?\n.*?\n\*\*(?:Agent|Method)\*\*:\s*(\S+)",
        re.DOTALL
    )
    return {m.group(1): m.group(2) for m in pattern.finditer(text)}
```

Then in `extract_finding()`, optionally accept an `agent_map` parameter and populate the `"agent"` key.

**Option B (structural)**: Add `**Agent**: {agent-name}` to the finding file template and update the campaign loop to write it. This is the long-term fix ‚Äî self-contained, no cross-file lookup.

**Option C (bypass score_example)**: Remove the `score_example()` requirement and use all findings. This loses quality-weighting but produces non-empty training data immediately.

Option C is the fastest path to a functional pipeline; Option B is the cleanest long-term solution.

## Verdict

**FAILURE** ‚Äî The DSPy training extractor is structurally broken for the current finding format. All 35 accumulated findings are excluded, making the optimizer non-functional. This issue is independent of the D2.3 taxonomy fixes (F3.3) ‚Äî even with correct verdict scoring, no findings reach the training set.
