# Finding: R10.3 — `masonry_optimize_agent` Will Invoke Anthropic API During `MIPROv2.compile()`; One Minor Issue Found (`_weight` Noise, `best_score` Missing)

**Question ID**: R10.3
**Status**: DONE
**Verdict**: WARNING
**Agent**: research-analyst
**Severity**: Low
**Confidence**: 0.95

## Summary

Static code analysis of `optimize_agent()` in `optimizer.py` confirms no blocking failure modes for a `diagnose-analyst` run. `MIPROv2.compile()` will invoke the Anthropic API (expected — it generates bootstrapped demonstrations using the LM). `ChainOfThought.save()` and `.load()` exist in dspy 3.1.3. Two minor issues: (1) `_weight` is a masonry-internal field that gets passed into `dspy.Example` as noise (harmless); (2) `MIPROv2` does not have a `best_score` attribute, so `result["score"]` will always be 0.0 — the optimizer's quality score is not captured.

## Evidence

### `configure_dspy()` — Anthropic API required

```python
def configure_dspy(model: str = "claude-sonnet-4-6") -> None:
    lm = dspy.LM(f"anthropic/{model}")
    dspy.configure(lm=lm)
```

This configures DSPy to use the Anthropic API. During `MIPROv2.compile()`, DSPy will:
1. Generate candidate instruction variants (LM calls)
2. Bootstrap few-shot demonstrations for each candidate (LM calls)
3. Evaluate candidates on the training set (LM calls)

For 21 examples with `max_bootstrapped_demos=3, max_labeled_demos=3`, the number of API calls is proportional to `num_trials` (default in MIPROv2 3.1.3 is typically 10-30). **This is expected behavior for DSPy optimization — not a bug.** Cost should be moderate for Sonnet calls.

### `MIPROv2.compile()` signature verified

```python
inspect.signature(dspy.MIPROv2.compile).parameters:
['self', 'student', 'trainset', 'teacher', 'valset', 'num_trials',
 'max_bootstrapped_demos', 'max_labeled_demos', 'seed', 'minibatch',
```

The `compile()` call in `optimizer.py`:

```python
optimized = optimizer.compile(
```

All parameters match. No `*` or keyword-only constraint violations.

### `_weight` in training examples — harmless noise

`build_dataset()` includes `"_weight": 1.0` in each example dict. When passed to `dspy.Example(**ex)`, `_weight` becomes a field in the Example object:

```python
ex = dspy.Example(question_text='test', project_context='ctx', _weight=1.0)
ex.keys()  # ['question_text', 'project_context', '_weight']
```

After `.with_inputs('question_text', 'project_context', 'constraints')`, `_weight` becomes an output field by default. DSPy MIPROv2 will attempt to predict `_weight` as part of the output, but it's ignored by the metric function (which only reads `verdict`, `evidence`, `confidence`). Noise, not a bug.

**Recommendation**: Filter `_weight` before building trainset:
```python
trainset = [dspy.Example(**{k: v for k, v in ex.items() if not k.startswith("_")}).with_inputs(*input_keys) for ex in dataset]
```

### `best_score` attribute does not exist on `MIPROv2`

```python
hasattr(dspy.MIPROv2, 'best_score')  # False
```

The `optimizer.py` code:
```python
if hasattr(optimizer, "best_score"):
    best_score = float(optimizer.best_score)
```

The guard works — `best_score` remains 0.0 in the returned dict. The MCP response will show `"score": 0.0` regardless of actual optimization quality. This is a monitoring gap, not a correctness bug. Kiln shows the score as 0.0 for every optimized agent.

### `ChainOfThought.save()` confirmed functional

```python
cot = dspy.ChainOfThought(...)
hasattr(cot, 'save')  # True
hasattr(cot, 'load')  # True
```

The save/load API exists. The saved JSON will contain the optimized module state and can be loaded via `cot.load(str(path))`. Kiln reads this file for the "LOAD OPTIMIZED" button — the format is compatible.

## Verdict

**WARNING** — No blocking failure modes. The optimization will run and produce a valid `.json` file loadable by Kiln. Two minor issues:
1. `_weight` field creates noise in training examples (harmless, but cleaner to strip)
2. `best_score` always returns 0.0 — Kiln cannot show actual optimization quality score

The Anthropic API invocation during `compile()` is expected behavior, not an issue.
