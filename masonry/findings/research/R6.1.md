# Finding: R6.1 — `masonry-lint-check.js` `runBackground()`: Paths With Spaces Break Silently on Windows

**Question ID**: R6.1
**Status**: DONE
**Verdict**: FAILURE
**Agent**: research-analyst
**Severity**: Low
**Confidence**: 0.85

## Summary

`runBackground()` in `masonry-lint-check.js` uses `spawn(cmd, args, { shell: process.platform === "win32" })`. On Windows with `shell: true`, Node.js joins the args array with spaces and passes the result through `cmd.exe /c`. File paths containing spaces (e.g., `C:\Users\trg16\My Documents\project\file.py`) are not individually quoted, causing the path to be split at the space — breaking the ruff, prettier, and eslint invocations. All three background formatters run silently (`stdio: "ignore"`, `proc.unref()`) so failures leave no trace.

## Evidence

### `runBackground()` implementation

```javascript
function runBackground(cmd, args, cwd) {
  try {
    const proc = spawn(cmd, args, {
      detached: true,
      stdio: "ignore",
      cwd,
      shell: process.platform === "win32",  // ← shell: true on Windows
    });
    proc.unref();
  } catch {
    // ignore spawn errors — formatting is best-effort
  }
}
```

### Node.js shell: true behavior on Windows

When `shell: true`, Node.js spawns: `cmd.exe /c <cmd> <arg1> <arg2> ...`

The args array is joined into a single string with spaces **without quoting individual elements**. From the Node.js docs (child_process.spawn):
> "If shell is true, runs command inside of a shell. On Windows, this is `cmd.exe /c` with the command and arguments joined by a space."

If `winPath = "C:\\Users\\trg16\\My Documents\\file.py"`, Node.js constructs:
```
cmd.exe /c ruff format C:\Users\trg16\My Documents\file.py
```

cmd.exe interprets this as: `ruff format C:\Users\trg16\My` with `Documents\file.py` as an unrecognized extra argument → `ruff` reports an error or processes a non-existent path.

### Three affected invocations

| Call site | Command | Path argument | Fails on spaces? |
|---|---|---|---|
| Line 179 | `runBackground(ruff, ["format", winPath], ...)` | `winPath` | YES |
| Line 191 | `runBackground("npx", ["prettier", "--write", filePath], ...)` | `filePath` | YES |
| Line 212 | `runBackground("npx", ["eslint", "--fix", winPath], ...)` | `winPath` | YES |

### Comparison: synchronous `run()` call (correct)

```javascript
const check = run(`"${ruff}" check "${winPath}"`, cwdOpts);
```

The synchronous lint check (line 182) manually quotes both the ruff path and winPath with `"` — correctly handling spaces in both. The background calls do not apply the same quoting.

### Why `shell: true` was chosen

Comment in the code: "background formatters" need `shell: true` on Windows because `npx` is a `.cmd` file (not a `.exe`) that `spawn()` cannot execute directly without cmd.exe. This is the same Windows .cmd resolution issue fixed in `llm_router.py` (F4.2), which used `["cmd", "/c", "claude", ...]` with `shell: false` to avoid this.

### Blast radius

- All Windows users with spaces in their project path (common: `My Documents`, `Program Files`, project folders with spaces in names)
- Failures are completely silent — `stdio: "ignore"` discards all output, `proc.unref()` makes the process detached
- Impact: files are not auto-formatted on edit. This is inconvenient but not a correctness or security issue

### Why severity is Low

Background formatters are best-effort quality-of-life features. Failing silently degrades the developer experience but does not break functionality. The synchronous ruff lint check (exit 0 / warn) still runs correctly. Code correctness is unaffected.

## Fix Options

**Option A**: Quote paths manually in args array when `shell: true`:
```javascript
const quotedPath = process.platform === "win32" ? `"${winPath}"` : winPath;
runBackground(ruff, ["format", quotedPath], projectRoot);
```

**Option B**: Replace `shell: true` with explicit cmd.exe wrapper (same pattern as F4.2):
```javascript
function runBackground(cmd, args, cwd) {
  const finalCmd = process.platform === "win32" ? "cmd" : cmd;
  const finalArgs = process.platform === "win32" ? ["/c", cmd, ...args] : args;
  const proc = spawn(finalCmd, finalArgs, { detached: true, stdio: "ignore", cwd });
  proc.unref();
}
```

Option B is more principled — it eliminates `shell: true` entirely, matching the F4.2 approach.

## Verdict

**FAILURE** — File paths with spaces break all three background formatter invocations on Windows when `shell: true` joins args without quoting. Severity is Low (best-effort formatters, silent failure) but the bug is confirmed and affects real Windows paths. Fix is straightforward (either quote paths or replace shell: true with explicit cmd.exe wrapper).
