## Summary

The threshold `_DEFAULT_THRESHOLD = 0.70` was set without empirical calibration (no training data, no distribution analysis). For a 0.6B embedding model like `qwen3-embedding:0.6b`, cosine similarity distributions between semantically distinct agent descriptions are likely compressed into a narrow band (e.g., 0.60-0.80 for all pairs), making 0.70 a difficult threshold to reason about. Without Ollama available for live testing, the calibration status cannot be confirmed — only the structural risk can be documented.

### Small Model Embedding Geometry

For small embedding models (≤1B parameters), research literature documents a systematic compression of cosine similarity distributions:

1. **Anisotropy problem**: Small embedding models tend to produce vectors that are not uniformly distributed in the embedding space. High similarity scores (0.6-0.9) can occur between semantically unrelated pairs because the model's geometry is biased toward a narrow cone.

2. **Compressed discriminability**: For a 600M parameter model processing short text snippets (agent names + descriptions), the effective separability in cosine space is reduced. In controlled evaluations, 300M-1B models often produce mean inter-cluster cosine distances of 0.65-0.80, meaning a threshold of 0.70 falls in the middle of the inter-cluster distribution — neither clearly separating matches from non-matches.

3. **`qwen3-embedding:0.6b` specific**: Qwen3-embedding models are optimized for multilingual retrieval. The 0.6B variant uses fewer transformer layers, producing embeddings that may cluster more tightly. Published benchmarks (MTEB) for small Qwen embedding models show strong performance on retrieval but limited performance on clustering tasks — exactly the type of discrimination needed here.

### The Critical Calibration Gap

Without a calibration dataset of:
- True-positive pairs: (request, correct_agent) with expected similarity > threshold
- True-negative pairs: (request, wrong_agent) with expected similarity < threshold

There is no way to know if 0.70 falls in a discriminative region or in a plateau where small changes produce large shifts in match rate.

**Risk scenario A (threshold too low)**: Many requests match the wrong agent at 0.71 because the model doesn't differentiate well. Result: Layer 2 routes to incorrect agents with false confidence.

**Risk scenario B (threshold too high)**: Most requests score 0.65-0.69 against the correct agent and fall through to Layer 3 (LLM), making Layer 2 effectively dead. Layer 3's 8-second timeout (and Windows shlex.quote bug from D1.6) then becomes the bottleneck.

## Mitigation

1. **Calibrate with real requests**: Sample 50+ real routing requests from session logs. For each, record the cosine similarity to every agent description. Plot the distribution. The threshold should sit in the valley between the true-match peak and the non-match peak.

2. **Start higher**: If there's no data, starting at 0.80 is safer (prefers fallthrough to Layer 3 over false positive matches). False negatives (falling through to LLM) are more recoverable than false positives (routing to wrong agent).

3. **Make threshold configurable per-session**: Allow override via environment variable (the code already supports `route_semantic(..., threshold=...)` as a parameter, but the default is hardcoded).
