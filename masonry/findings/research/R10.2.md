# Finding: R10.2 — `masonry_drift_check` Produces Output With Real Verdicts; Two Secondary Defects Found

**Question ID**: R10.2
**Status**: DONE
**Verdict**: WARNING
**Agent**: research-analyst
**Severity**: Medium
**Confidence**: 0.93

## Summary

`masonry_drift_check` runs end-to-end and produces per-agent drift reports now that `agent_db.json["verdicts"]` is populated (D9.1). The core mechanism is functional. However, two secondary defects are found: (1) `registry_loader` skips ~14 agents due to `extra_forbidden` validation errors (extra YAML fields not in `AgentRegistryEntry` schema), causing `fix-implementer` and others to be excluded from drift checking; (2) the verdict distribution includes FAILURE verdicts from multi-project scans, producing "critical drift" warnings for all agents even when the Masonry self-research performance is healthy.

## Evidence

### Live drift check run

```python
from masonry.src.dspy_pipeline.drift_detector import run_drift_check
from masonry.src.schemas.registry_loader import load_registry

registry = load_registry(Path('masonry/agent_registry.yml'))  # 30 valid agents
reports = run_drift_check(Path('agent_db.json'), registry)    # 5 reports
```

Output:
```
Drift reports (5 agents):
  benchmark-engineer:  baseline=0.85 current=0.00 drift=100%  alert=critical
  research-analyst:    baseline=0.85 current=0.34 drift=59.8% alert=critical
  diagnose-analyst:    baseline=0.85 current=0.33 drift=60.8% alert=critical
  compliance-auditor:  baseline=0.85 current=0.33 drift=60.8% alert=critical
  design-reviewer:     baseline=0.85 current=0.32 drift=62.6% alert=critical
```

The output structure is correct (DriftReport per agent, with alert_level, recommendation). But all 5 show "critical" drift.

### Root cause: FAILURE verdicts from multi-project scan

The verdict distribution for `research-analyst` (19 verdicts):
```
FAILURE: 11    (0 points each)
HEALTHY:  4    (1.0 each)
COMPLETE: 1    (1.0 — in _OK_VERDICTS)
UNCALIBRATED: 2 (0.5 each)
WARNING:  1    (0.5)
```

Score = (4×1.0 + 1×1.0 + 2×0.5 + 1×0.5) / 19 = 6.5/19 ≈ 0.34 → matches output.

`sync_verdicts_to_agent_db.py` uses `extract_training_data()` which scans ALL projects under the BL2.0 root. Other projects (ADBP, adbp2, bl2, etc.) have research-analyst questions that ended in FAILURE. These contaminate the masonry-project verdict history. The current_score of 0.34 reflects cross-project performance, not Masonry self-research quality.

### Defect 1: registry_loader rejects ~14 agents

`AgentRegistryEntry` uses `model_config = ConfigDict(extra="forbid")`. The `agent_registry.yml` has extra fields (`dspy_status`, `drift_status`, `last_score`, `runs_since_optimization`, `registrySource`) that were added during agent onboarding (masonry-agent-onboard.js). These cause 14 agents including `fix-implementer` (25 verdicts) to be skipped by `registry_loader`.

As a result:
- `fix-implementer` (25 verdicts) → **not checked** despite being the most-data agent
- `code-reviewer`, `git-nerd`, and others → **not checked**
- Only 30/44 agents pass validation → drift check coverage is ~68%

**Threshold calibration assessment**: The thresholds (10%=ok, 10-25%=warning, >25%=critical) are reasonable in principle. But with cross-project FAILURE verdicts contaminating the current_score, they produce false critical alerts for agents that are actually performing well in the Masonry campaign context.

### Defect 2: baseline_score=0.85 is a uniform default

All agents in `agent_db.json` have `score=0.85` (set during onboarding). This is not a measured baseline — it's a placeholder. Drift detection compares measured `current_score` against this arbitrary baseline. The high baseline (0.85) amplifies apparent drift even for minor deviations.

## Mitigations

1. **Fix registry_loader** (next wave): Change `model_config = ConfigDict(extra="forbid")` to `extra="ignore"` in `AgentRegistryEntry`, or strip unknown fields before validation. This unblocks drift checking for all 44 agents.

2. **Scope verdict sync to single project** (next wave): Pass an explicit `questions_md_path` when syncing masonry verdicts. `sync_verdicts_to_agent_db.py --questions-md masonry/questions.md` limits attribution to Masonry questions only, excluding cross-project FAILURE contamination.

3. **Calibrate baselines from real data**: After fix #2, compute `baseline_score` from the actual historical verdict distribution for each agent using the masonry-only questions.

## Verdict

**WARNING** — `masonry_drift_check` is functional and produces structured output. The core mechanism (verdict scoring, alert thresholds, DriftReport) is correct. Two defects prevent accurate results: registry validation rejects 14 agents (extra fields), and cross-project FAILURE verdicts inflate apparent drift. Both are fixable. The output format is actionable once these are resolved.
