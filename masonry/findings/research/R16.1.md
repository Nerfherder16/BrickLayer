### Blocking Issue 2: Agent Name Truncation in extractMarkdownField

`masonry-observe.js` line 30:
```javascript
function extractMarkdownField(content, label) {
  const re = new RegExp(`\*{0,2}${label}\*{0,2}:\\s*([\\w]+)`, 'i');
  //                                               ^^^^ [\w]+ stops at hyphens!
  const m = content.match(re);
  return m ? m[1].trim() : null;
}
```

The character class `[\w]` matches `[A-Za-z0-9_]` — it does NOT match hyphens. So:

| **Agent**: field in finding | Extracted | In AGENT_CATEGORIES | correct_pts |
|------------------------------|-----------|---------------------|-------------|
| `fix-implementer` | `"fix"` | No | 0 |
| `research-analyst` | `"research"` | No | 0 |
| `synthesizer-bl2` | `"synthesizer"` | No | 0 |
| `quantitative-analyst` | `"quantitative"` | No | 0 |
| `regulatory-researcher` | `"regulatory"` | No | 0 |
| `mortar` | `"mortar"` | **Yes** | 70 |
| `trowel` | `"trowel"` | **Yes** | 70 |

All hyphenated specialist agents (which is the majority of the Masonry fleet) extract incorrectly. Only single-word agents (mortar, trowel, developer, unknown) would score `correct_agent_dispatched > 0`.

Even if Issue 1 were resolved and session_ids matched, most specialist-dispatched sessions would score:
- `correct_agent_dispatched = 0` (truncated name not in AGENT_CATEGORIES)
- `downstream_success = 30` (if finding exists)
- Total: 30 < `min_training_score(65)` → not included as training record

### "Sessions matched: 1" Is Misleading

```python

```

This computes `min(4, 1) = 1` regardless of whether any events actually pair. It is NOT a count of matched pairs. Should be `sessions_matched = len(records)` (actual training records) or a count of actual session_id matches.

The four-layer routing architecture and training pipeline architecture are correct. The two bugs are in upstream data generation (CWD guard — fixed in F16.1) and data extraction (agent name regex — not yet fixed).
