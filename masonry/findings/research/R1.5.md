# Finding: R1.5 — Ollama Unavailability Adds 15-Second Blocking Delay Per Non-Deterministic Request; No Circuit Breaker

**Question ID**: R1.5
**Status**: DONE
**Verdict**: FAILURE
**Agent**: research-analyst
**Severity**: High
**Confidence**: 1.0

## Summary

When Ollama at `192.168.50.62:11434` is unreachable, `route_semantic` blocks for the full 15-second timeout (`_TIMEOUT = 15.0`) before returning `None`. No circuit-breaker, no fast-fail after repeated failures, no cached failure state. Every non-deterministic request during an Ollama outage blocks for 15 seconds before proceeding to Layer 3. With Ollama on a LAN at `192.168.50.62`, outages during home network issues are expected.

## Evidence

### The Exact Timeout Path (semantic.py lines 89-126)

```python
try:
    with httpx.Client(timeout=_TIMEOUT) as client:
        uncached = [a for a in registry if _agent_corpus_key(a) not in _embedding_cache]
        if uncached:
            texts = [_agent_corpus_key(a) for a in uncached]
            try:
                resp = client.post(
                    f"{ollama_url}/api/embed",
                    json={"model": model, "input": texts},
                    timeout=_TIMEOUT,   # ← 15-second blocking wait
                )
                resp.raise_for_status()
                ...
            except Exception as exc:
                print(f"[semantic] Ollama batch error: {exc}", file=sys.stderr)
                return None   # ← only reached after full 15s timeout
```

**Key fact**: `httpx.Client(timeout=_TIMEOUT)` sets both connect timeout AND read timeout to 15.0 seconds. When Ollama is unreachable (no TCP response at all), the connect attempt waits the full 15 seconds before timing out.

For LAN hosts that are powered off (not just network issues), TCP RST is returned immediately — this would fail fast. But for network partitions (router issue, VLAN misconfiguration, WiFi dropout), the TCP SYN waits for the full timeout.

### No Circuit Breaker or Failure Cache

After 15 seconds, `route_semantic` returns `None` and the module forgets the failure immediately. The next invocation will wait another 15 seconds. There is no:
- Failure counter (track N consecutive failures → skip Ollama for T seconds)
- Last-failure timestamp (backoff after recent failure)
- Module-level circuit-breaker state
- Environment variable to disable semantic routing

### Cumulative Impact on Routing Latency

For a session with N requests that fail deterministic routing:
- Each request blocks 15s at Layer 2 before proceeding to Layer 3
- Layer 3 adds 8s (or instant failure via D1.6 on Windows)
- Layer 4 returns immediately

**Per-request overhead during Ollama outage (Windows): 15s + 8s = 23 seconds**

For a BrickLayer campaign session processing 50 requests, with 30 falling through deterministic routing: **11.5 minutes of dead time** that would be 0 seconds with a working circuit breaker.

### Fast-Fail Case (Agent Embeddings Already Cached)

If agent embeddings are cached from a previous successful call (session-level `_embedding_cache`), the batch call is skipped. However:
1. The request embedding still needs a single Ollama call (lines 112-114)
2. This call also blocks for 15 seconds on failure

So even with a warm cache, each request blocks 15s for the request embedding attempt.

### The 15-Second Value Is Justified for Normal Operation

For a legitimate batch request embedding 40+ agents, a 15-second timeout is reasonable — the Ollama API under normal conditions responds in 200-2000ms for this size. The timeout is well-chosen for the happy path.

The problem is the missing circuit breaker, not the timeout value itself.

## Mitigation

**Minimal fix — circuit breaker using module-level state:**

```python
_ollama_circuit_open: bool = False
_ollama_circuit_open_until: float = 0.0
_CIRCUIT_OPEN_DURATION: float = 60.0  # seconds before retry

def route_semantic(request_text, registry, ...):
    global _ollama_circuit_open, _ollama_circuit_open_until

    if _ollama_circuit_open:
        if time.monotonic() < _ollama_circuit_open_until:
            return None  # fast-fail
        else:
            _ollama_circuit_open = False  # reset for retry

    try:
        # ... existing code ...
    except Exception:
        _ollama_circuit_open = True
        _ollama_circuit_open_until = time.monotonic() + _CIRCUIT_OPEN_DURATION
        return None
```

This converts 15-second blocks into instant returns after the first failure, with a 60-second retry window. First failure still blocks 15s (unavoidable), but subsequent failures within 60 seconds are instant.

**Also useful**: Set a shorter connect timeout (e.g., 2s) vs. read timeout (15s):
```python
timeout = httpx.Timeout(connect=2.0, read=15.0, write=5.0, pool=2.0)
```
This fast-fails connection errors while allowing slow responses to complete.
