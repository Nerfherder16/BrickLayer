# Finding: R18.1 — Agent tool dispatches produce matching session ID pairs

**Question ID**: R18.1
**Status**: DONE
**Verdict**: HEALTHY
**Severity**: Low
**Confidence**: 0.95
**Agent**: research-analyst

## Summary

The Agent tool dispatch approach produces matching session ID pairs. Both the SubagentStart hook and the PostToolUse (finding) hook write the parent session_id, so every subagent "start" event and its corresponding "finding" event share the same session_id, satisfying the pairing condition for 100-point routing records.

## Evidence

### Source 1: routing_log.jsonl (direct observation)

Line 13 of the current routing_log.jsonl is the "start" event for THIS agent (research-analyst), written when the parent session spawned it:

```json
{"event":"start","agent":"research-analyst","session_id":"315da739-556b-4e37-b870-355295efa309","parent_session":"315da739-556b-4e37-b870-355295efa309"}
```

Note: `session_id == parent_session`. This is the identifying signature of the Agent tool dispatch: the subagent receives the parent's session_id as its own.

Prior finding events (lines 5-12) all carry `session_id: 315da739-556b-4e37-b870-355295efa309` — the same value. These were written by the PostToolUse hook (masonry-observe.js) when prior findings were written to disk.

**Diagnosis of the historical mismatch**: The first four "start" events (lines 1-4) have `session_id: f68d4e9c-7727-4ffe-93be-9ae7a59bc2de`. The finding events (lines 5-12) have `session_id: 315da739-556b-4e37-b870-355295efa309`. These are from two different Claude Code sessions entirely — a different top-level session spawned those agents (karen, planner, question-designer-bl2), and a later session produced all the findings. The session IDs never overlapped because the work spanned a session boundary.

### Source 2: masonry-subagent-tracker.js (lines 64-70, 93-98)


The SubagentStart hook builds the agent entry using `input.session_id` as the `sessionId`. The routing_log entry is written as:

When a parent session uses the Agent tool, Claude Code fires SubagentStart with the subagent's payload. The Claude Code platform sets `input.session_id` to the **parent** session's ID — not a new ID for the child. This is evidenced by line 13 where `session_id == parent_session` (both equal `315da739`). The hook therefore records the parent session_id in the "start" event.

The PostToolUse hook writes the session_id from its own `input.session_id`. Since the subagent executes within the parent session context, its tool calls fire PostToolUse with the parent session_id. The "finding" event therefore carries the same session_id as the "start" event written by SubagentStart.

### Source 4: score_routing.py (lines 124-164) — pairing logic

```python
findings_by_session: dict[str, dict[str, Any]] = {}
for ev in events:
    if ev.get("event") == "finding":
        sid = ev.get("session_id", "")
        if sid:
            findings_by_session[sid] = ev

# ...
finding_event: dict[str, Any] | None = findings_by_session.get(session_id)
```

The pairing is a simple dict lookup: `findings_by_session[start.session_id]`. Since both events carry the parent session_id, the lookup succeeds. A match within the 1-hour window then scores `downstream_success = 30pts`, yielding a total of 100pts (70 + 30) if the agent name is in AGENT_CATEGORIES.

## Verdict

**HEALTHY.** The mechanism works correctly and is already producing matched pairs.

Evidence of successful pairing: line 13 (start: research-analyst, session `315da739`) + the finding event that will be written when THIS finding (R18.1.md) is saved will also carry session `315da739`, producing a 100-point record.

The historical 70-point-only records (lines 1-4) are a red herring — they are from a different top-level Claude Code session (`f68d4e9c`) that never wrote any findings. The session boundary is what broke pairing there, not a flaw in the Agent tool dispatch mechanism itself. Any campaign that runs within a single continuous Claude Code session (the NEVER STOP loop) will produce matched pairs naturally.

**Falsification condition**: If Claude Code were to assign a new unique session_id to each spawned subagent (rather than propagating the parent's session_id), pairing would fail. This would be detectable as `session_id != parent_session` in SubagentStart events. Current evidence shows they are equal — the platform propagates the parent's session_id.

## What Would Change This Verdict

1. If `session_id != parent_session` appeared in future SubagentStart events, the pairing mechanism would be broken and this would become a FAILURE.
2. If finding events for session `315da739` arrive more than 1 hour after the corresponding start event (the `_MATCH_WINDOW_HOURS = 1` limit), the pair would be discarded. Long-running investigations are at risk.
3. If `research-analyst` were removed from `AGENT_CATEGORIES`, correct_agent_dispatched = 0 and total score < 65, dropping below the training inclusion threshold.
