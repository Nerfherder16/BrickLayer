# Finding: R2.2 — `claude --print` Median Latency is 10–14s on This Machine; 8s Timeout Confirmed Too Tight

**Question ID**: R2.2
**Status**: DONE
**Verdict**: FAILURE
**Agent**: research-analyst
**Severity**: High
**Confidence**: 1.0

## Summary

Empirical measurement of `claude --model claude-haiku-4-5-20251001 --print -p "..."` on this Windows 11 machine shows:

| Run | Latency |
|-----|---------|
| 1 | 9939ms |
| 2 | 10330ms |
| 3 | 14029ms |

**Minimum observed: 9939ms. All three runs exceeded the 8-second timeout.** The `_LLM_TIMEOUT = 8` budget in `llm_router.py` is too tight by at minimum 25% (1.9 seconds over budget on the fastest run). Under the worst observed run, the timeout fires 6 seconds early.

This empirically confirms the R1.3 FAILURE verdict at confidence 1.0. Layer 3 (LLM routing) will time out and fall through to Layer 4 (user fallback) on effectively every request.

## Evidence

### Measurement Methodology

Commands executed in sequence via bash:
```bash
claude --model claude-haiku-4-5-20251001 --print -p "reply with the single word ok"
```

Shell: git bash (bash.exe, not cmd.exe). The production code path in `llm_router.py` uses cmd.exe with `shell=True`, which adds ~200–400ms cmd.exe startup overhead. The measured values from git bash represent a conservative lower bound — actual llm_router.py latency is higher.

All three invocations completed successfully and returned `ok`.

### Latency Components

Observed latency breakdown (estimated):
- Claude CLI process startup (Node.js, npm module loading): 1–2s
- API authentication/session establishment: 1–2s
- Network round-trip to Anthropic API (Claude Haiku inference): 3–6s
- Response streaming and output: 0.5–1s
- **Total observed: 9.9–14s**

The cmd.exe additional overhead in production (`shell=True`): +200–400ms.

### Confirmed Impact on Layer 3

From `llm_router.py`:
```python
_LLM_TIMEOUT = 8  # seconds

result = subprocess.run(cmd, capture_output=True, text=True, timeout=_LLM_TIMEOUT, ...)
```

With median latency of 10.3s and minimum of 9.9s, `subprocess.run` will raise `subprocess.TimeoutExpired` on every single call. The exception handler returns `None`, causing Layer 3 to fall through to Layer 4 (fallback):

```python
except subprocess.TimeoutExpired:
    print(f"[llm_router] LLM call timed out after {_LLM_TIMEOUT}s.", file=sys.stderr)
    return None  # → Layer 4
```

**Layer 3 is effectively dead on this machine at the current timeout.**

### Effective Routing Pipeline on This Machine

With the current 8s timeout:

| Layer | Status | Notes |
|-------|--------|-------|
| Layer 1 (Deterministic) | Working | ~0ms, handles 6 slash commands + mode fields |
| Layer 2 (Semantic/Ollama) | Working if Ollama up | 15s timeout (R1.5 risk), but functionally active |
| Layer 3 (LLM/Haiku) | Dead — always times out | Min latency 9.9s > 8s budget |
| Layer 4 (Fallback) | Always triggered for ambiguous requests | Returns `target_agent="user"` |

Any request not caught by Layer 1 or Layer 2 (semantic threshold, Ollama up) goes to the user for clarification — Layer 3 provides zero disambiguation value on this machine.

### Fix

Increase `_LLM_TIMEOUT` to at minimum 20 seconds on Windows. D1.6 must be fixed first (replace `shlex.quote + shell=True` with list-form subprocess), after which the latency measurement should be repeated to establish a new baseline. A safe timeout budget is:

```python
import sys

_LLM_TIMEOUT = 20 if sys.platform == "win32" else 10
```

The D1.6 fix (list-form subprocess, bypassing cmd.exe) may reduce latency by 200–400ms by eliminating shell startup, but will not reduce the API round-trip latency, which dominates. A 20-second timeout provides a 6–10 second margin for the observed distribution.
