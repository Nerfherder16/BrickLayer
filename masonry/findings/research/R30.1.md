# Finding: R30.1 — One-Slot-Per-Type Collision Risk in masonry-preagent-tracker.js

**Question ID**: R30.1
**Status**: DONE
**Verdict**: WARNING
**Severity**: LOW-MEDIUM (training signal quality degradation, not functional failure)
**Confidence**: HIGH
**Date**: 2026-03-23
**Agent**: research-analyst

---

## Hypothesis Under Test

The one-slot-per-type strategy in `masonry-preagent-tracker.js` — which writes agent prompts to
`pending_agent_prompts/{subagentType}_latest.json` using an overwrite (no append, no concurrency guard) —
causes prompt collisions when two agents of the same type are spawned concurrently within the 10-second TTL
window. The result is either one agent getting the other's prompt label or one agent getting an empty
`request_text` in `routing_log.jsonl`.

---

## Evidence

### Slot write logic (masonry-preagent-tracker.js)

File: `masonry/src/hooks/masonry-preagent-tracker.js`, line 73–76.

The write is a plain synchronous `fs.writeFileSync` with no concurrency guard:

```
const slotPath = path.join(pendingDir, `${subagentType}_latest.json`);
fs.writeFileSync(slotPath, JSON.stringify(record), "utf8");  // blind overwrite
```

The code comment at line 13–15 acknowledges the risk explicitly:
> "Two-slot strategy: uses subagent_type as the key (not a UUID) since Claude hooks do not expose a stable
> event correlation ID. Race condition risk is low because concurrent spawns of the same specialist type
> are rare."

This is an informed design choice, not an oversight. The author accepted the race condition as low-risk.

### Slot read/consume logic (masonry-subagent-tracker.js)

File: `masonry/src/hooks/masonry-subagent-tracker.js`, lines 194–206.

The SubagentStart hook reads the slot if it was written within the last 10 seconds, then **deletes the slot
on first consume** (`fs.unlinkSync(slotPath)`). This is delete-on-consume with a 10-second TTL.

**Collision mechanics confirmed:**
1. PreToolUse fires for spawn A: writes `fix-implementer_latest.json` with prompt A.
2. PreToolUse fires for spawn B (within 10s): overwrites `fix-implementer_latest.json` with prompt B.
3. SubagentStart fires for spawn A: reads slot — gets prompt B (wrong). Deletes slot.
4. SubagentStart fires for spawn B: slot is gone — gets empty `request_text`.

Net result per collision: one routing_log entry gets the wrong prompt (cross-contaminated label), one
gets empty string. Both are corrupted training signals.

### Routing log analysis (routing_log.jsonl)

Analyzed 78 start events across the full routing log history.

**Confirmed collision windows (same agent type within 10-second TTL):**

| Agent type | Gap (seconds) | Era | Prompt A | Prompt B |
|---|---|---|---|---|
| fix-implementer | 5.1s | PRE-F28.3 (2026-03-22) | empty | empty |
| fix-implementer | 9.6s | PRE-F28.3 (2026-03-22) | empty | empty |
| diagnose-analyst | 6.8s | PRE-F28.3 (2026-03-22) | empty | empty |
| diagnose-analyst | 6.3s | PRE-F28.3 (2026-03-22) | empty | empty |

4 confirmed collision events out of 78 total starts = **5.1% collision rate** across all spawns.

**Important: all 4 collision events are from PRE-F28.3 era** (2026-03-22), before `masonry-preagent-tracker.js`
was active. During that period, `request_text` was universally empty because the hook did not exist yet.
This means the slots were never written, so the collision mechanics could not have produced corrupted labels —
the emptiness was already the baseline behavior.

**Post-F28.3 era (today, 2026-03-23):**
- 14 starts recorded.
- No concurrent same-type spawns within 10 seconds.
- 5/5 fix-implementer spawns have populated, distinct prompts (F29.1–F30.1).
- 3/3 research-analyst spawns have populated prompts.
- The closest research-analyst pair (V30.1 and R30.1) was 13.5 seconds apart — just outside the 10-second
  TTL window, so no collision occurred. Each got its own correct prompt.

**Current prompt capture rate (post-F28.3):** 10/14 starts have populated `request_text` = **71.4%**.
The 4 that are empty are from earlier in the session before the hook was active for that session type
(spec-writer, general-purpose, early hypothesis-generator-bl2).

### Theoretical collision scenario

The collision is **mechanistically confirmed** from code inspection — the overwrite + delete-on-consume
pattern guarantees incorrect behavior when two spawns of the same type fall within the 10-second TTL. But
**no confirmed post-F28.3 collision has occurred** in the routing log. The risk is real but unrealized so far.

The highest-risk scenario is the campaign loop dispatching multiple research-analyst or fix-implementer
agents in parallel (e.g., F29.1 and F29.2 dispatched simultaneously). Today's data shows F29.1 and F29.2
were 10.3 seconds apart — just outside the window. V30.1 and R30.1 were 13.5 seconds apart. The campaign
loop appears to introduce enough latency between sequential spawns that collisions are avoided in practice.

## Threshold Analysis

No explicit constant in `constants.py` governs this metric. Applying internal training signal quality
reasoning:

- **Acceptable miss rate for DSPy training data**: any individual entry being empty is tolerable (the
  optimizer skips low-confidence examples); a *wrong label* (cross-contaminated prompt) is a more severe
  quality issue because it trains incorrect agent-to-prompt associations.
- **Collision rate observed**: 5.1% of spawns fall in collision windows, but **0% of post-F28.3 spawns**
  have resulted in confirmed collisions.
- **Training signal capture rate post-F28.3**: 71.4% (10/14), trending toward 100% as more agents go
  through the preagent-tracker path.

The risk is a WARNING rather than FAILURE because:
1. No post-activation collision has occurred.
2. The impact is DSPy training label quality, not functional agent behavior.
3. Agents still receive their correct prompts from the orchestrator — only the *logging* of those prompts
   is at risk.

---

## Risk Assessment

**Impact when collision occurs:**
- One routing_log entry gets the wrong agent's prompt as `request_text` — a mislabeled training example.
- One routing_log entry gets empty `request_text` — a dropped training example.
- The agents themselves are unaffected — they receive their prompts through the standard Claude subagent
  mechanism, not through this slot file.
- DSPy optimizer will see a corrupted `(prompt, agent_type)` pair — this can degrade optimizer accuracy
  for the affected agent type over many training cycles, but a single bad example has minimal impact.

**Functional correctness:** NOT affected. The slot is only used for logging.

**Training signal quality:** Potentially degraded. The extent depends on collision frequency.

**Future risk escalation:** If the campaign loop starts dispatching multiple agents of the same type
genuinely in parallel (not sequentially), collision frequency will increase. The diagnose-analyst triple
collision (07:12:13, 07:12:20, 07:12:26 — three spawns in 13 seconds) is the highest observed density
and would cause 2 of 3 labels to be corrupted if the preagent tracker had been active.

---

## Fix Options (No Implementation)

**(a) Spawn-ID-keyed filename**
Use a unique ID in the slot filename instead of agent type:
`pending_agent_prompts/{subagentType}_{uuid}_latest.json`
Requires the PreToolUse hook to generate an ID that correlates with the SubagentStart event — but the
SubagentStart hook does not currently receive a stable correlation ID from Claude's hook infrastructure.
This is the root cause the original author cited for choosing type-keyed slots. Feasible only if Claude
exposes a stable `tool_call_id` or similar in SubagentStart.

**(b) Queue file (append-based)**
Instead of overwriting, append each pending record to a JSONL queue file
(`pending_agent_prompts/{subagentType}_queue.jsonl`). SubagentStart reads the oldest entry in the queue
and deletes it (FIFO). This handles any number of concurrent same-type spawns correctly, but requires
careful file locking or atomic operations to avoid queue corruption on concurrent reads.

**(c) Accept the current miss rate**
Given that: (1) no post-F28.3 collision has occurred, (2) functional behavior is unaffected, and (3)
DSPy training is robust to occasional mislabeled examples, the current design may be acceptable. Monitor
routing_log.jsonl for cross-contaminated labels (same `request_text` appearing under two different agents
in the same session). If collision rate increases as the loop parallelizes more, escalate to option (b).

**(d) Timestamp-ordered slot with read-back verification**
PreToolUse writes slot with a monotonic sequence counter. SubagentStart reads the slot and verifies the
sequence matches its own spawn ordering. Requires additional state tracking but no external ID correlation.

**Recommended priority**: (c) for now, (b) if parallelism increases. Option (a) is blocked by infrastructure.

---

## What Would Change This Verdict

- **FAILURE**: Evidence of a post-F28.3 cross-contaminated `request_text` in routing_log.jsonl — i.e., one
  agent type's prompt appearing under a different agent type's log entry, confirmed by session analysis.
- **FAILURE**: Campaign loop refactored to dispatch multiple same-type agents genuinely in parallel,
  causing collision frequency > 20% of same-type spawns.
- **HEALTHY**: A spawn-ID correlation mechanism added to Claude's hook infrastructure (SubagentStart
  receives the `tool_call_id` from the PreToolUse:Agent event), enabling option (a) to eliminate the
  race condition entirely.

---

## Summary

The one-slot-per-type collision scenario is **mechanistically confirmed** from code inspection: a blind
overwrite + delete-on-consume with a 10-second TTL guarantees mislabeled training data when two same-type
agents spawn within the window. However, routing_log.jsonl analysis finds **zero confirmed collisions in
the post-F28.3 era** (today's session). The 4 collision windows observed are all from the pre-F28.3 era
when the hook was not yet active, making them moot. Current collision rate under active hook conditions is
0%. The risk is real and latent but currently unrealized. Impact is limited to DSPy training signal quality;
functional agent behavior is not affected.

**Verdict: WARNING** — Theoretical collision risk confirmed by code, not yet materialized in practice.
Monitor; no immediate fix required unless the campaign loop begins dispatching same-type agents in parallel.
