# Finding: R1.6 — Layer 4 Fallback Is Indistinguishable Between Correct and Infrastructure Failure

**Question ID**: R1.6
**Status**: DONE
**Verdict**: FAILURE
**Agent**: research-analyst
**Severity**: Medium
**Confidence**: 1.0

## Summary

Layer 4 returns the same `RoutingDecision(target_agent="user", layer="fallback", confidence=0.0)` regardless of whether the fallback was caused by a genuinely ambiguous request or by infrastructure failure (Ollama down, LLM timeout, empty registry). A caller receives identical output in both cases with no way to distinguish them. This prevents programmatic recovery strategies (retry on infrastructure failure, prompt user on true ambiguity) and masks systemic problems.

## Evidence

### The Fallback Object (router.py lines 18-23)

```python
_FALLBACK = RoutingDecision(
    target_agent="user",
    layer="fallback",
    confidence=0.0,
    reason="Ambiguous request -- asking user for clarification",
)
```

This singleton is returned for ALL Layer 4 cases. The `reason` field always reads "Ambiguous request -- asking user for clarification" even when the true reason is Ollama unavailability or LLM timeout.

### Independent Failure Modes That Cause Fallback

Enumerating every path that reaches Layer 4:

**Infrastructure failures (should NOT prompt user for clarification):**

1. **D1.3 — Empty registry**: `_load_registry` returns `[]` → Layer 2 returns `None` immediately → Layer 3 routes LLM with empty agent list → likely returns hallucinated name or fails → Layer 4
2. **R1.5 — Ollama unreachable**: `route_semantic` catches `httpx.TimeoutException` after 15s → returns `None` → Layer 3
3. **R1.3 — LLM timeout**: `subprocess.TimeoutExpired` after 8s → returns `None` → Layer 4
4. **D1.6 — Windows shlex.quote failure**: cmd.exe command broken → non-zero returncode → `route_llm` returns `None` → Layer 4
5. **PyYAML not installed**: `registry_loader.py` line 29 returns `[]` → same as D1.3 path

**Legitimate ambiguity (correct to prompt user):**
6. Request is genuinely unclear (no slash command, no Mode field, Ollama running but no good match, LLM can't determine intent)

**Hybrid failures (infrastructure + ambiguity):**
7. Registry populated but agent descriptions are too generic → low similarity scores → Layer 2 falls through even with working Ollama

### RoutingDecision Schema Cannot Express Failure Type

```python
class RoutingDecision(BaseModel):
    model_config = ConfigDict(extra="forbid")
    target_agent: str
    layer: Literal["deterministic", "semantic", "llm", "fallback"]
    confidence: float
    reason: str = Field(max_length=100)
    fallback_agents: list[str] = Field(default_factory=list)
```

The `reason` field is 100 characters and is a human-readable string. A caller cannot programmatically check if it was an infrastructure failure vs. genuine ambiguity. `extra="forbid"` prevents adding structured metadata without a schema change.

The `layer="fallback"` field is the only machine-parseable signal, but it covers all fallback causes equally.

### Practical Consequences

1. **Misleading user prompt**: When Ollama is down, the user is asked to "clarify" a request that was actually well-formed. The correct response would be "Ollama is unreachable, routing degraded — proceeding with manual routing."

2. **No automated recovery**: A retry loop cannot distinguish "retry Ollama in 60 seconds" from "ask user for different wording." All fallbacks are treated as user-clarification needed.

3. **Silent infrastructure masking**: High fallback rates due to Ollama outages look identical to high ambiguity rates in any monitoring or logging system that only sees `RoutingDecision` outputs.

### Proposed Taxonomy

| Fallback Type | `target_agent` | `layer` | Distinguishing field |
|---|---|---|---|
| True ambiguity | "user" | "fallback" | `reason` contains "ambiguous" |
| Empty registry | "user" | "fallback" | No observable difference currently |
| Ollama failure | "user" | "fallback" | No observable difference currently |
| LLM timeout | "user" | "fallback" | No observable difference currently |

## Mitigation

Add a `fallback_reason` classification to the router output:

```python
# In router.py, distinct fallback objects:
_FALLBACK_AMBIGUOUS = RoutingDecision(
    target_agent="user", layer="fallback", confidence=0.0,
    reason="Ambiguous request -- asking user for clarification"
)
_FALLBACK_NO_REGISTRY = RoutingDecision(
    target_agent="user", layer="fallback", confidence=0.0,
    reason="Registry empty -- routing degraded"
)
_FALLBACK_SEMANTIC_TIMEOUT = RoutingDecision(
    target_agent="user", layer="fallback", confidence=0.0,
    reason="Ollama unavailable -- semantic routing skipped"
)
_FALLBACK_LLM_TIMEOUT = RoutingDecision(
    target_agent="user", layer="fallback", confidence=0.0,
    reason="LLM routing timed out -- fallback to user"
)
```

The `reason` field (max 100 chars) can carry sufficient distinction for both human-readable display and programmatic prefix-matching.
