# Finding: R35.1 — optimizer.py:optimize_agent() has no independent loading path

**Question ID**: R35.1
**Status**: DONE
**Verdict**: HEALTHY
**Severity**: Low
**Confidence**: 0.97
**Agent**: research-analyst
**Date**: 2026-03-23

## Summary

`optimizer.py:optimize_agent()` contains zero file I/O. All three callers supply `dataset` explicitly; F34.1's `project_context` enrichment is the sole injection point for that field.

## Hypothesis Under Test

`optimizer.py:optimize_agent()` takes a `dataset: list[dict]` parameter that is always provided by the caller. It does not load from disk. The `project_context` enrichment in F34.1 flows correctly from `run_optimization.py:load_training_data_from_scored_all()` through the `dataset` argument. No second hardcoding point exists in `optimizer.py`.

## Evidence

### Supporting evidence

**1. optimizer.py full read (lines 187–283)** — `optimize_agent()` signature:
```python
def optimize_agent(
    agent_name: str,
    signature_cls: type,
    dataset: list[dict],        # no default value
    output_dir: Path,
    backend: str = "anthropic",
    num_trials: int = 10,
    valset_size: int = 100,
    metric_fn: Any = None,
) -> dict[str, Any]:
```
The `dataset` parameter has **no default value** — it is a required positional argument. The function body contains no `open(`, `Path(` read calls, no `.read_text`, no `.jsonl` references, and no reference to `scored_all`. The only I/O is `output_dir.mkdir(...)` and `output_file.write_text(...)` for saving the result JSON. (Evidence: direct code inspection, `optimizer.py` lines 187–283.)

**2. `optimize_all()` also receives datasets externally (lines 289–338)** — `optimize_all()` accepts `datasets: dict[str, list[dict]]` and iterates over it. It contains no file reads. It calls `optimize_agent()` at line 335, passing `agent_dataset` drawn from the caller-supplied dict.

**3. Call chain in run_optimization.py:run() (lines 254–366)** — The canonical path is:
```
run(base_dir=base_dir)
  → load_training_data_from_scored_all(scored_all_path, agent_name, base_dir=base_dir)  # populates project_context from project-brief.md (F34.1 fix)
  → optimize_agent(agent_name=..., signature_cls=..., dataset=examples, ...)            # examples already enriched
```
`project_context` is set at line 62 of `run_optimization.py` via `_read_project_brief(base_dir)` and stamped into every example dict at line 81 before `optimize_agent` is called.

**4. server.py caller (_tool_masonry_optimize_agent, lines 431–476)** — Uses the legacy `build_dataset()` path from `training_extractor.py`. Inspection of `training_extractor.py` lines 273–291 shows `build_dataset()` also populates `project_context` by calling `_load_project_brief(source_file)` per finding. This path independently carries `project_context`; it does not bypass F34.1 because F34.1 only targeted the `scored_all` path, and this path has its own equivalent mechanism.

**5. run_r23_1.py caller (lines 36–61, 117–123)** — A one-off research script that builds examples via `extract_finding()` + `_load_project_brief()` and sets `ex["project_context"]` explicitly before calling `optimize_agent()`. This caller correctly enriches `project_context`.

### Contradicting evidence

None identified. All callers supply the dataset, and none rely on a default that would silently produce an empty `project_context`.

### Analogues

The pattern of separating data loading from optimization logic is standard in DSPy-based pipelines. MIPROv2 itself only receives `trainset` (a list of `dspy.Example` objects) — there is no mechanism inside `optimizer.py` through which it could perform independent file I/O without the caller knowing.
