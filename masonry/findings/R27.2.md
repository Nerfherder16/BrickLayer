# Finding: R27.2 — SubagentStart event does not expose `prompt`; PreToolUse:Agent hook is the correct fix point

**Question ID**: R27.2
**Status**: DONE
**Verdict**: WARNING
**Severity**: Low
**Confidence**: 0.91
**Agent**: research-analyst

## Summary

`masonry-subagent-tracker.js` reads `input.prompt || input.description` from the SubagentStart hook event, but both fields are empty for all 66 start events in `routing_log.jsonl`. The SubagentStart event exposes only: `session_id`, `subagent_type`, `agent_name`, and `parent_session` — it does not include the Agent tool's `prompt` parameter. A 1-3 line change to `masonry-subagent-tracker.js` **cannot** fix this: `input.prompt` is architecturally absent from SubagentStart events. The correct fix is a new **PreToolUse:Agent** hook that fires in the parent session context, where the full Agent tool parameters (including `prompt`) are available.

## Evidence

### Field Analysis — 66 Start Events in routing_log.jsonl

| Field | Non-empty count | Notes |
|---|---|---|
| `agent` | 66/66 | Populated from SubagentStart event |
| `event` | 66/66 | Always "start" |
| `parent_session` | 66/66 | Populated from `input.session_id` |
| `session_id` | 66/66 | Subagent session |
| `timestamp` | 66/66 | Always set |
| `request_text` | **0/66** | `input.prompt || input.description` — always empty |

### What the SubagentStart Event Exposes

Proven available (used by Mortar gate, which functions correctly):
- `input.subagent_type` ✓
- `input.agent_name` / `input.agent_type` ✓
- `input.session_id` ✓

Not available in SubagentStart:
- `input.prompt` ✗ — the task description passed to the Agent tool
- `input.description` ✗ — the 3-5 word summary passed to the Agent tool

### Current Hook Coverage

```json
PreToolUse matchers: "Write|Edit|Bash", "ExitPlanMode"
SubagentStart: masonry-subagent-tracker.js
```

There is **no PreToolUse hook for the Agent tool**. The Agent tool call fires in the parent session context and is not currently intercepted.

### Root Cause

The SubagentStart event fires in the **subagent's** process after initialization. By this point, the parent's Agent tool call parameters are no longer accessible. The `prompt` parameter was used to initialize the subagent's system prompt, but it isn't re-exposed in the SubagentStart payload.

## Fix Architecture

A PreToolUse hook on the Agent tool would intercept the call before the subagent spawns, with full access to `tool_input.prompt` and `tool_input.description`.

**Required changes:**

1. **New hook script** (`masonry/src/hooks/masonry-agent-preuse.js`):
   - Read `tool_input.prompt` and `tool_input.description` from PreToolUse event
   - Extract `subagent_type` from `tool_input.subagent_type`
   - Append a `{event: "pre_spawn", agent: subagent_type, request_text: prompt[:500]}` entry to `routing_log.jsonl`

2. **settings.json registration**:
   ```json
   PreToolUse: [{ "matcher": "Agent", "hooks": [{ "type": "command", "command": "node masonry-agent-preuse.js" }] }]
   ```

3. **Correlation**: The pre_spawn entry and subsequent start entry can be correlated by `subagent_type` + timestamp proximity (≤ 2s).

**Effort**: Moderate (new 50-100 line hook + settings change). Not a 1-3 line change to existing code.

## Impact Estimate

If a PreToolUse:Agent hook is implemented:
- All future research loop Agent spawns would produce `request_text`-populated entries
- The 66 existing empty entries would remain unlabelled (retrospective labelling remains infeasible per D26.1)
- Starting from the next campaign wave, routing training data quality would improve significantly

## Verdict

**WARNING** — A simple 1-3 line change to `masonry-subagent-tracker.js` cannot fix the empty `request_text` issue: the SubagentStart event does not expose the Agent tool's `prompt` parameter. The fix requires a new PreToolUse:Agent hook (moderate effort). The current fix path is architecturally clear but not trivial.
