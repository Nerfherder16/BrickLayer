# Finding: D33.1 — MCP code path calls configure_dspy() without api_key, creating a secondary BLOCKED vector

## Summary

F32.2 fixes the CLI path but the MCP tool `masonry_optimize_agent` calls `configure_dspy()` without forwarding `api_key`, leaving the Kiln-triggered optimization path permanently BLOCKED without env var.

## Evidence

### 1. CLI path (run_optimization.py) -- FIXED by F32.2

File: `masonry/scripts/run_optimization.py`

- Line 427-430: `--api-key` CLI arg with `default=os.environ.get("ANTHROPIC_API_KEY")` -- accepts explicit key or env var fallback.
- Line 304: `configure_dspy(backend=backend, api_key=api_key)` -- threads `api_key` through to `configure_dspy()`.
- Line 233 in `run()`: `api_key` parameter is accepted and forwarded.

This path is fully wired. F32.2 works here.

### 2. MCP tool path (server.py) -- NOT FIXED

File: `masonry/mcp_server/server.py`

- Line 466: `configure_dspy(model=model, backend=backend)` -- **no `api_key` parameter**
- The `_tool_masonry_optimize_agent` function (lines 431-475) has no `api_key` in its input schema (lines 741-771) and does not extract or forward any API key.
- When Kiln triggers optimization via the MCP `masonry_optimize_agent` tool, there is no mechanism to pass an API key. The call relies entirely on `ANTHROPIC_API_KEY` being set in the MCP server process environment.

### 3. DSPy -> LiteLLM chain -- api_key kwarg works correctly

Traced the full call chain:
- `dspy.LM.__init__(**kwargs)` stores `api_key` in `self.kwargs` (line ~76 of dspy/clients/lm.py)
- `dspy.LM.forward()` merges `self.kwargs` into the request dict
- `litellm_completion()` spreads `**request` into `litellm.completion()`
- `litellm.completion(api_key=...)` uses the explicit key when provided, falling back to `litellm.api_key` or env var only when `api_key` is None

This confirms F32.2's approach is sound -- when `api_key` reaches `dspy.LM()`, LiteLLM uses it exclusively.

### 4. No other hard-coded env reads

Grep of `os.environ.*ANTHROPIC` across all Python files under `masonry/` found only:
- `masonry/scripts/run_optimization.py:429` -- the CLI default (expected, correct behavior)
- Test files (expected)
- Documentation strings (not executable)

No secondary hard-reference to `os.environ["ANTHROPIC_API_KEY"]` exists elsewhere in the optimization pipeline code.

### 5. MCP tool input schema lacks api_key field

The `masonry_optimize_agent` tool schema (server.py lines 741-771) accepts: `agent_name`, `projects_dir`, `agent_db_path`, `questions_md_path`, `output_dir`, `model`. There is no `api_key` or `backend` property in the schema, though `backend` is read from args on line 447 with a default of `"anthropic"`.

## Root Cause

F32.2 added `api_key` plumbing to the CLI entry point (`run_optimization.py`) and the core function (`configure_dspy()`), but missed the MCP server entry point (`_tool_masonry_optimize_agent` in `server.py`). The MCP path -- which is the path triggered by the Kiln "OPTIMIZE" button -- calls `configure_dspy(model=model, backend=backend)` without `api_key`, so it always falls back to `os.environ["ANTHROPIC_API_KEY"]`.

This means:
- **CLI runs** (`python masonry/scripts/run_optimization.py agent --api-key sk-...`) -- UNBLOCKED by F32.2
- **Kiln UI "OPTIMIZE" button** (MCP tool) -- still BLOCKED without env var set in the MCP server process

Since the Kiln UI is the primary trigger for optimization (per CLAUDE.md: "Trigger from Kiln UI 'OPTIMIZE' button or via `masonry_optimize_agent` MCP tool"), the recurring BLOCKED state is only partially resolved.

## Fix Specification

- **File**: `masonry/mcp_server/server.py`
- **Line**: 466
- **Change**: Add `api_key` to the MCP tool input schema (alongside existing properties) and thread it through to `configure_dspy()`:
  1. Add `"api_key": {"type": "string", "description": "Anthropic API key. Falls back to ANTHROPIC_API_KEY env var."}` to the `masonry_optimize_agent` inputSchema properties (around line 766).
  2. In `_tool_masonry_optimize_agent()`, extract it: `api_key = args.get("api_key")` (after line 447).
  3. Change line 466 from `configure_dspy(model=model, backend=backend)` to `configure_dspy(model=model, backend=backend, api_key=api_key)`.
- **Verification**: `python -c "from masonry.mcp_server.server import TOOLS; schema = TOOLS['masonry_optimize_agent']['inputSchema']; assert 'api_key' in schema['properties'], 'api_key not in schema'; print('PASS')"` and inspect that `_tool_masonry_optimize_agent` calls `configure_dspy` with `api_key=`.
- **Risk**: Low. The change is additive -- when `api_key` is not provided via MCP, behavior is unchanged (falls back to env var). No existing tests should break.

## Verdict

However, the MCP tool entry point (`masonry_optimize_agent` in `server.py`) was not updated and still hard-depends on the `ANTHROPIC_API_KEY` environment variable. The recurring BLOCKED state across waves (F28.2, F29.3, F29.4, F30.2) will persist for Kiln-triggered runs until the MCP path is also wired.
