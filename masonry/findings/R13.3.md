# Finding: R13.3 — `backfill_agent_fields.py` Excludes Masonry Findings; Prefix Map Missing F/V Prefixes

**Question ID**: R13.3
**Status**: DONE
**Verdict**: WARNING
**Severity**: Medium
**Confidence**: 0.96

## Summary

`backfill_agent_fields.py` runs without errors and reports 0 files patched (620 already attributed, 32 skipped). However, masonry's 97 findings are NOT processed because the script uses the same `masonry/` directory exclusion as `score_findings.py`. If the exclusion were removed, the script would miss `F*` (fix-implementer) and `V*` (validate) prefixed findings — they're absent from `DEFAULT_PREFIX_MAP`. Masonry's `build_dataset()` path does not require `**Agent**:` fields, so this gap has no current impact.

## Dry Run Output

```
[DRY RUN] Scanned: 652 findings
[DRY RUN] Patched: 0
[DRY RUN] Skipped (no prefix map): 32
[DRY RUN] Already attributed: 620
```

## Root Cause: masonry/ Exclusion

`backfill_agent_fields.py:122` uses the identical exclusion:

```python
if child.is_dir() and child.name not in ("findings", "masonry", ".git"):
    _collect(child / "findings")
```

Masonry findings in `masonry/findings/` are never scanned.

## Impact Assessment

No current impact. The masonry optimization pipeline uses `build_dataset()` + `training_extractor.py`, which attributes findings via `**Question ID**:` prefix parsing (not `**Agent**:`). The `scored_all.jsonl` path (which uses `backfill_agent_fields.py`) is explicitly designed to exclude masonry findings.

## Verdict

**WARNING** — `backfill_agent_fields.py` correctly backfills ADBP project findings (620 already correct). The masonry exclusion is intentional. The missing F and V prefixes in `DEFAULT_PREFIX_MAP` represent a latent gap: if masonry were ever added to the scoring pipeline, fix-implementer and validate-type findings would be unattributed. No action required for current functionality.
