## Summary

`project-brief.md` (116 lines, domain-specific) is sufficient as static `project_context` for DSPy MIPROv2 training — it covers domain architecture, research domains, key code locations, and invariants that give the optimizer meaningful signal. However, `build_dataset()` still passes `"project_context": ""` after the F8.1 fix. The brief is not being read or injected. The WARNING is that usable context exists but is unreachable — the optimizer currently runs with an empty context field despite a 116-line project brief being available on disk.

## Evidence

### 1. `project-brief.md` content analysis

The brief contains:
- **Architecture section** (15 lines): What Masonry is, three-tier model, five implementation components
- **Research domains A and B** (45 lines): 8 specific routing questions + 8 hook safety questions + exact code file locations
- **Key invariants** (10 lines): No source modifications, findings-only output, Ollama availability caveat
- **Known uncertainties** (8 rows): Specific thresholds, file locations, and failure modes
- **Past misunderstandings** (5 items): Prevents the LM from common domain misconceptions

This is domain-specific, non-generic content. It directly addresses the research questions asked in the campaign.

### 2. What MIPROv2 does with `project_context`

MIPROv2 generates optimized prompt instructions by analyzing patterns across training examples. The `project_context` field in `ResearchAgentSig`:

```python
project_context: str = dspy.InputField(
    desc="Project brief summary, prior findings, and key constraints"
)
```

MIPROv2 uses the input distribution to generate instructions like "When investigating [context type], look for [pattern]." With the brief as static context, MIPROv2 learns: "When the context mentions Masonry hooks and routing layers, pay attention to [patterns from findings]."

### 3. Static context is sufficient for initial optimization

All training examples having the same `project_context` value is not a disqualifier for MIPROv2. The optimizer uses example inputs to generate prompt candidates and evaluates them. With identical context but varied `question_text` (now fixed in F8.1), MIPROv2 can still:
- Learn verdict patterns from question types
- Generate instructions optimized for the masonry domain
- Calibrate confidence guidance

The brief is domain-specific enough to produce meaningful optimizations — unlike an empty string.

### 5. Fix specification: inject project-brief.md

The minimal fix is to read `project-brief.md` from the project directory:

```python
# In build_dataset(), before the findings loop:
project_brief_text = ""
for findings_dir in projects_dir.rglob("findings"):
    brief_path = findings_dir.parent / "project-brief.md"
    if brief_path.exists():
        try:
            project_brief_text = brief_path.read_text(encoding="utf-8")
        except OSError:
            pass
        break  # use first found

# In example construction:
example = {
    "question_text": finding.get("question_text") or finding.get("question_id", ""),
    "project_context": project_brief_text[:2000],   # cap for token budget
    "constraints": "",
    ...
}
```

The 2000-char cap (approximately 500 tokens) balances context richness against DSPy token budget for optimizer runs.
