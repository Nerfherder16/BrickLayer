### 2. CasaOS Docker Compose architecture

CasaOS (running at `192.168.50.19`) manages apps via individual Docker Compose files stored under `/DATA/AppData/{app-name}/`. Each app has a `docker-compose.yml` at that path. CasaOS's app management UI writes compose files to this standard location.

For Ollama on CasaOS, the standard paths are:

- **Compose file**: `/DATA/AppData/ollama/docker-compose.yml`
- **Model data volume**: `/DATA/AppData/ollama/data/` (or a named Docker volume `ollama_data`)
- **Container name**: `ollama` (default in the official `ollama/ollama` CasaOS app entry)

**Confidence**: MEDIUM. CasaOS path convention is documented in CasaOS community sources and the official CasaOS app store definitions. The exact compose file path may vary slightly if Tim installed Ollama manually rather than via the CasaOS app store.

### 3. Model data persistence across container restarts

Docker named volumes and bind-mount volumes persist independently of container lifecycle. The Ollama container stores model weights in `/root/.ollama/models` inside the container, which is mounted to the host volume path. A `docker compose restart` or `docker compose up -d` does NOT delete volumes unless `docker compose down -v` is explicitly invoked.

Conclusion: `qwen3:14b` model data persists across container stop/start/restart operations. It would only be lost if:
- `docker compose down -v` was run (removes named volumes)
- The host volume path was manually deleted
- Disk failure on CasaOS storage

**Confidence**: HIGH. Standard Docker volume behavior — well-established and applies universally to all Docker installations.

## Diagnostic: Why the Outage May Have Occurred

Common causes in CasaOS homelab context (ranked by likelihood):

1. **CasaOS host reboot without Docker auto-restart** — Ollama container `restart: unless-stopped` policy handles this normally, but if the policy was not set or was overridden, the container stays stopped after host reboot. Check: `docker inspect ollama | grep RestartPolicy`.

2. **Docker daemon restart cleared container** — Less common; `restart: unless-stopped` should survive daemon restarts.

3. **OOM kill** — If CasaOS host ran low on RAM, the kernel OOM killer may have terminated the Ollama process inside the container. Container status would show `Exited (137)`. Fix: restart container; consider adding `--memory` limit to compose file.

4. **GPU driver / CUDA issue** — If Ollama uses GPU passthrough and a driver update broke CUDA, the container may crash on startup. Check: `docker logs ollama` for CUDA errors.

5. **Host unreachable (separate from container)** — Exit code 28 is also produced if the CasaOS host itself is down. Ping test: `ping 192.168.50.19`.
