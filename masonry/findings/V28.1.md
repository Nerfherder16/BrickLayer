## Evidence

### Note on Self-Consistency vs Cross-Scoring

Self-consistency (example = prediction) scores 1.0 for all 196 records — any internally consistent record matches itself perfectly. This is expected. MIPROv2 does NOT use self-consistency during optimization; it generates new predictions and scores them against examples. The relevant gradient check is **cross-scoring**: what does the metric return when a wrong prediction is scored against an example?

### Component Analysis — Wrong Prediction vs Negative Example

For a negative example (quality_score="0.0", action="reverted") with wrong prediction (quality_score="1.0", action="updated"):
- **quality_score_proximity (0.5)**: ex_val=0.0, pred_val=1.0, |0-1|=1.0 > 0.1 → **0.25**
- **action_match (0.3)**: "reverted" ≠ "updated", no partial credit → **0.0**
- **changelog_quality (0.2)**: wrong action="updated" → entry="updated docs" len>10 → **0.2**
- Total: **0.45** (exact ceiling for wrong-prediction-on-negative)

### Why 2.6% Negative Rate Is Sufficient

MIPROv2 selects bootstrap demonstrations based on metric scores. With score range 0.0–0.45 for negative examples and 0.85–1.0 for positive examples, the optimizer can clearly distinguish the two categories. The 5 negatives will be selected with high probability because:
1. They represent all distinct revert scenarios in the masonry domain
2. Their score differential (0.45 vs 1.0 for wrong vs correct predictions) creates a strong gradient signal
3. MIPROv2 typically selects 10-20 demos; 5 negatives in 196 = 2.6% but the selection probability under metric diversity is ~15-25%
