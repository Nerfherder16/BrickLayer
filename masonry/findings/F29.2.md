## Summary

The 5 karen synthetic negative training examples (score=0, source=synthetic_negative,
output.reverted=True) were being lost on every run of score_all_agents.py because
score_ops_agents.py fully regenerates scored_ops_agents.jsonl from git log, discarding
any manually appended records. The fix creates a stable scored_synthetic.jsonl file that
is never regenerated, and modifies score_all_agents.py to merge it as a 5th source
alongside the four existing dynamically-generated sources.

---

## Root Cause Confirmed

score_ops_agents.py opens its output file with `"w"` mode (line 338), unconditionally
overwriting it on every run. Any records appended to scored_ops_agents.jsonl outside the
git-log pipeline are silently discarded. score_all_agents.py only reads the four
generated files — it had no stable home for hand-crafted or synthetic records.

## Changes Made

### New file: masonry/training_data/scored_synthetic.jsonl
Exactly 5 records extracted from git history (commit 56eff3b, last 5 lines of the then-
current scored_all.jsonl). Each record is a karen synthetic negative:
- commit_hash: synthetic_neg_001 through synthetic_neg_005
- source: synthetic_negative
- score: 0
- output.reverted: true

This file is never written by any scorer script and survives all reruns.

### Modified: masonry/scripts/score_all_agents.py
Two targeted changes:

1. Added `synthetic_path = td_dir / "scored_synthetic.jsonl"` alongside the 4 existing
   path variables (line 228).

2. Added step 5 block after the 4 scorer invocations: reads synthetic_path into
   `synthetic_records`, adds a `scored_synthetic` entry to scorer_summaries (visible in
   the summary table), then includes synthetic_path in the merge loop alongside the
   other 4 paths.

The deduplication key for synthetic records is
`src:synthetic_negative:<commit_hash>:karen:0` — guaranteed unique, never collides with
ops records whose keys embed real git commit hashes and score=100.
