# Finding: V1.2 — `_MODE_FIELD_RE` Case-Sensitive; Misses Lowercase `**mode**:` Variants

**Question ID**: V1.2
**Status**: DONE
**Verdict**: FAILURE
**Agent**: design-reviewer
**Severity**: Low
**Confidence**: 1.0

## Summary

The Mode field regex `r"\*\*Mode\*\*:\s*(\w+)"` in `deterministic.py` is case-sensitive. It matches `**Mode**: diagnose` but fails to match `**mode**: diagnose` (lowercase M) or `**MODE**: diagnose` (all-caps). BrickLayer's question banks use `**Mode**:` with capital M consistently, but agent outputs and programmatic generators may produce lowercase variants. Any lowercase Mode field falls through deterministic routing to Layer 2/3.

## Evidence

### The Regex (deterministic.py line 36)

```python
_MODE_FIELD_RE = re.compile(r"\*\*Mode\*\*:\s*(\w+)")
```

`re.compile()` without `re.IGNORECASE` is case-sensitive by default. The pattern `\*\*Mode\*\*` matches exactly those characters.

### Test Cases

| Input | Match? | Reason |
|-------|--------|--------|
| `**Mode**: diagnose` | ✓ Yes | Exact match |
| `**mode**: diagnose` | ✗ No | `m` doesn't match `M` |
| `**MODE**: diagnose` | ✗ No | All-caps doesn't match |
| `**Mode** : diagnose` | ✗ No | Extra space before colon (not in pattern) |
| `**Mode**:diagnose` | ✓ Yes | `\s*` matches zero spaces |
| `**Mode**: Diagnose` | ✓ Yes (but wrong case) | `(\w+)` captures `Diagnose` with capital D; `get_agents_for_mode` checks `mode in a.modes` — if modes list is lowercase, this fails too |

### The Lowercase Mode Value Problem (Compound Issue)

Even if the Mode field header matches, the captured mode value is passed to `get_agents_for_mode(registry, mode_value)`:

```python
m = _MODE_FIELD_RE.search(request_text)
if m:
    mode_value = m.group(1).strip()
    agents = get_agents_for_mode(registry, mode_value)
```

And in `registry_loader.py`:
```python
def get_agents_for_mode(registry, mode: str) -> list[AgentRegistryEntry]:
    return [a for a in registry if mode in a.modes]
```

The `modes` list in `agent_registry.yml` uses lowercase values (e.g., `modes: [diagnose, research, validate]`). If a request contains `**Mode**: Diagnose` (capital D), `mode_value = "Diagnose"` is NOT in `["diagnose", "research", "validate"]` → `agents = []` → no deterministic match.

**Two case-sensitivity bugs compound:**
1. Header matching: `**Mode**` (capital M) required
2. Value matching: `mode_value` must match lowercase registry entries

### Actual BL 2.0 Question Bank Format

From `questions.md` in this project:
```
**Mode**: diagnose
**Mode**: research
**Mode**: validate
```

These all use capital M with lowercase mode value — the exact format the regex handles correctly. So in practice with this project's question bank, the regex is fine.

### Risk in Other Contexts

The risk materializes when:
1. An LLM generates a question with `**mode**: diagnose` (lowercase M) — common when generating BL questions programmatically
2. A user types a question with lowercase `**mode**:` field
3. An agent outputs a question payload where the header case varies

In these cases, Mode field routing silently falls through to Layer 2/3 instead of routing instantly at Layer 1.

## Mitigation

Add `re.IGNORECASE` flag:
```python
_MODE_FIELD_RE = re.compile(r"\*\*Mode\*\*:\s*(\w+)", re.IGNORECASE)
```

Also normalize the captured value to lowercase before registry lookup:
```python
mode_value = m.group(1).strip().lower()
agents = get_agents_for_mode(registry, mode_value)
```

These two changes make Mode field matching robust to case variations in both the header and the value.
