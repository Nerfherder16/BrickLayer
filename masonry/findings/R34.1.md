### Criterion 4 — score_findings.py qualification criteria

`score_findings.py` requires `score >= 60` (TRAINING_THRESHOLD). Score is the sum of three dimensions:
- **Confidence calibration** (0-40 pts): +10 for any `**Confidence**:` float, +15 for value in 0.5-1.0, +15 for calibration match to verdict severity.

- **Verdict clarity** (0-20 pts): +10 for non-empty `**Verdict**:` field, +10 for verdict in `VALID_VERDICTS` set.

Agent identity is extracted from `**Agent**:` field; if the field spans multiple lines (due to backtick rendering in the H1 title), the entire corrupted string becomes the agent value.

## Root Cause Analysis

Three distinct blockers prevent corpus growth to 87:

1. **Section name mismatch** (10 records blocked): The BL 2.0 finding template for research-analyst does not mandate `## Evidence` as the section heading. Files from waves 11-13 used custom section names (`Analysis`, `Verification Results`, `Code Trace`, `Run Output`). `score_findings.py` only matches the exact string `"Evidence"` in `_extract_section()`. These files have qualifying content but it is invisible to the scorer.

2. **Backtick agent field corruption** (2 records blocked): Files R14.1 and R17.2 have H1 titles containing backtick-wrapped expressions (e.g., `` # Finding: R14.1 — `diagnose-analyst`... ``). The `_RE_AGENT` regex matches the first `**Agent**:` line, but the H1 backtick text is captured instead of the intended agent value. This is a regex ordering defect: the agent regex fires on what appears to be a legitimate match location in the markdown.

3. **Insufficient findings volume** (5 records gap even after fixes): The finding pool simply needs 5 more research-analyst records to hit 87. At the current production rate of ~3-6 R-series findings per campaign wave, one additional wave would close this gap.
