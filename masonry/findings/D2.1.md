# Finding: D2.1 — Stop Hooks Run in Parallel, Not Sequential; D1.5 Hypothesis Was Incorrect

**Question ID**: D2.1
**Status**: DONE
**Verdict**: HEALTHY (D1.5 hypothesis refuted)
**Agent**: diagnose-analyst
**Severity**: Low
**Confidence**: 0.85

## Summary

Claude Code executes all registered Stop hooks **in parallel**, not sequentially. Exit code 2 from one Stop hook does not prevent other Stop hooks from running. When multiple hooks block (exit 2), all their stderr outputs are collected and displayed together. This means the D1.5 concern — that masonry-stop-guard would mask masonry-build-guard's block reason — was based on an incorrect sequential-short-circuit assumption. Both guards fire concurrently, both block reasons reach the user.

The double-fire problem from D1.1 applies to Stop hooks too: each Stop guard fires twice per Stop event (once from plugin, once from global settings). Under parallel execution, the user sees duplicate block messages, but this is cosmetic — the blocking behavior is correct.

## Evidence

### Claude Code Hook Execution Policy (Official Docs)

Per Claude Code documentation:
> **All matching hooks run in parallel** within each event.

For Stop hooks with exit code 2:
- All hooks in the Stop array run concurrently
- If any hook returns exit code 2, Claude cannot stop — the blocking decision is the **union** of all results
- stderr from all hooks is collected and presented to the user together
- A single exit-code-2 from any hook is sufficient to block; ALL hooks still execute

### Stop Hook Registration Order

From `~/.claude/settings.json`, Stop hooks in registration order:
1. `masonry-stop-guard.js` — blocks on uncommitted session files
2. `masonry-session-summary.js` — writes session summary (async, exit 0)
3. `masonry-build-guard.js` — blocks on pending autopilot tasks
4. `masonry-ui-compose-guard.js` — blocks on pending UI compose tasks

Under parallel execution, all four run simultaneously. The user sees the combined stderr from any that exit code 2.

### D1.5 Hypothesis Refuted

D1.5 hypothesized: "If Claude Code processes Stop hooks sequentially and the first hook exits with code 2, the second hook may not execute." This was incorrect. Sequential short-circuit does not occur.

**What D1.5 correctly identified (still valid):**
- stop-guard and build-guard can both produce exit-code-2 simultaneously — user sees both messages
- Both messages appear in the same stop-block notification — this is actually the CORRECT behavior
- The user gets full visibility into all blocking conditions

**What D1.5 incorrectly assumed:**
- Sequential execution model
- That the first exit-code-2 would mask later hooks

D1.5's FAILURE verdict should be revised to HEALTHY: the behavior works as desired — both block reasons are always shown when both conditions are true.

### Double-Fire Impact on Stop Hooks

From the D1.1/F2.1 analysis, Stop hooks are registered in both the plugin `hooks/hooks.json` AND `~/.claude/settings.json`. Under parallel execution with double-registration, all four Stop guards fire twice = 8 concurrent processes per Stop event.

Impact:
- masonry-stop-guard fires twice → user sees "Stop blocked — N uncommitted..." twice in the message
- masonry-build-guard fires twice → "Masonry build in progress!" appears twice
- masonry-session-summary fires twice → session summary written twice (benign)
- masonry-ui-compose-guard fires twice → similar duplication

The duplicate output is cosmetically annoying but functionally correct. Applying the F2.1 fix (empty the plugin hooks object) eliminates the duplication.

### Registration Within Hooks Array Structure

The Stop hooks are registered in a single `"hooks": [ ... ]` array within one Stop event object:
```json
"Stop": [
  {
    "hooks": [
      { "command": "masonry-stop-guard.js", "timeout": 10 },
      { "command": "masonry-session-summary.js", "timeout": 8 },
      { "command": "masonry-build-guard.js", "timeout": 10 },
      { "command": "masonry-ui-compose-guard.js", "timeout": 10 }
    ]
  }
]
```

All four items run in parallel. The 10-second timeouts are per-hook, not cumulative — all hooks have their full timeout budget regardless of other hooks' status.

### Confidence Caveat

Confidence is 0.85 (not 1.0) because the documentation summary is sourced from the claude-code-guide agent's fetch of official docs, not a direct primary source read. The parallel execution model is well-established in Claude Code's hook design, and the `stop_hook_active` field in hook input (which prevents recursive invocation) is consistent with a parallel model. An empirical test with two distinct exit-code-2 Stop hooks confirming both stderr messages appear simultaneously would raise confidence to 1.0.

## Revised Verdict for D1.5

D1.5 (masonry-stop-guard masking masonry-build-guard) should be revised from FAILURE to HEALTHY. The behavior is correct — both block reasons surface simultaneously. The only actionable item remains the cosmetic duplicate output caused by the D1.1 double-fire, which F2.1 resolves.
