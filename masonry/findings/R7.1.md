# Finding: R7.1 — `build_dataset()` Produces Degenerate Training Examples: `question_text = question_id`, `project_context = ""`

## Summary

`build_dataset()` in `training_extractor.py` constructs DSPy training examples with three structurally degenerate inputs:

1. `question_text` is set to the question ID (e.g., `"R5.1"`) — not the actual question text
2. `project_context` is always `""` (empty string)
3. `constraints` is always `""` (empty string, acceptable — has default)

`ResearchAgentSig` defines `question_text` as "The full research question to investigate" and `project_context` as "Project brief summary, prior findings, and key constraints." These are the primary semantic inputs. With all inputs degenerate, MIPROv2 cannot learn input-conditional prompt patterns — it can only learn generic output formatting. The resulting "optimized" prompts provide near-zero improvement over the default.

### Why MIPROv2 cannot learn from these examples

MIPROv2 works by examining `(input, output)` pairs and generating prompt instructions that maximize the likelihood of producing the correct output for a given input. The generated instruction is injected into the system prompt.

If inputs are:
- `question_text = "R5.1"` (opaque 4-character ID)
- `project_context = ""` (no context)
- `constraints = ""` (no constraints)

Then MIPROv2 sees a distribution of inputs that are all effectively identical (short IDs, no context). It cannot learn:
- "When the question asks about a race condition, use these diagnostic patterns"
- "When the question is about Windows-specific behavior, consider shell semantics"
- "When project context mentions DSPy, prioritize signature field alignment"

The optimizer can only learn generic output formatting — "always include an evidence section", "verdicts should be one of HEALTHY/FAILURE/etc." These are already encoded in the signature description. The optimization provides near-zero signal beyond the baseline.

### `project_context` gap analysis

The finding file format does NOT contain a `## Context` or `## Project Background` section — it contains `## Summary`, `## Evidence`, `## Verdict`, etc. The project context (project-brief.md, prior findings) is never incorporated into individual finding files. Even if `extract_finding()` tried to extract `project_context`, there is no source section in findings to extract it from.

This means populating `project_context` requires an architectural change: reading `project-brief.md` at dataset build time and injecting it into each example.

## Fix Specification

### Step 1: Extend `_build_qid_to_agent_map()` to return question text

```python
def _build_qid_to_question_map(questions_md_path: Path) -> dict[str, dict[str, str]]:
    """Extract question_id → {agent, question_text} mapping from questions.md."""
    ...
    for block in blocks:
        qid_m = re.search(r"###\s+(\w+\d+\.\d+):\s*(.+)", block)  # capture text after ":"
        agent_m = re.search(r"\*\*(?:Agent|Method)\*\*:\s*(\S+)", block)
        if qid_m:
            result[qid_m.group(1)] = {
                "agent": agent_m.group(1) if agent_m else None,
                "question_text": qid_m.group(2).strip(),  # actual question text
            }
    return result
```

### Step 2: Populate `project_context` from project-brief.md

In `extract_training_data()`, read `project_brief_path = findings_dir.parent / "project-brief.md"` and inject the first 500 characters as `project_context` for all examples from that project.
