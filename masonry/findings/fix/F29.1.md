## Summary

`score_findings.py` was extracting `question_text` solely from finding file headers (subtitle regex or `**Question**:` field), producing a median of ~88 chars for research-analyst records. The F26.1 hypothesis enrichment — which builds 500-char hypothesis-block text from `questions.md` — existed only in `training_extractor.py` and was never called from `score_findings.py`. This fix wires `_build_qid_to_agent_map()` into the scoring pipeline so `scored_all.jsonl` receives the enriched question_text for all questions that have hypothesis blocks in `questions.md`.

## Evidence

**Before fix** (sampled from `scored_all.jsonl`):
- research-analyst records: 56
- question_text median: 88 chars
- Min: 43 chars, Max: 187 chars
- Sample: `"Semantic Threshold 0.70 Lacks Calibration..."` (96 chars)

**After fix** (rerun of `score_all_agents.py`):
- research-analyst records: 56 (unchanged — no records lost)
- question_text median: **500 chars** (was 88 — 5.7x improvement)
- Min: 93 chars (questions without hypothesis blocks), Max: 500 chars
- Total records in scored_all.jsonl: 601 (was 593 — slight increase from new findings)
- Records with empty question_text: 0

**Sample enriched values:**
- `[500 chars] Is the semantic routing threshold of 0.70 well-calibrated for qwen3-embedding:0.6b... -- The 0.6B model may produce compressed similarity distributions where many dissimilar queries score above 0.70...`
- `[500 chars] What percentage of real Masonry requests are routed deterministically... -- Deterministic routing handles slash commands and mode-field dispatches...`

## Change Implemented

**File**: `C:/Users/trg16/Dev/Bricklayer2.0/masonry/scripts/score_findings.py`

Three targeted changes:

1. **Import block** (lines 17-21): Added guarded import of `_build_qid_to_agent_map` from `masonry.src.dspy_pipeline.training_extractor`, with fallback to `None` if the module is unavailable.

2. **`extract_finding_fields()` signature** (line 63): Added optional `qid_map: dict[str, dict[str, str]] | None = None` parameter. After deriving `question_text` from the file header, if `qid_map` has an entry for the `question_id` and the enriched text is longer than the header-derived text, the enriched text (capped at 500 chars) replaces it.

3. **`run()` function** (lines 362-378): Before the per-file scoring loop, builds the qid_map once by calling `_build_qid_to_agent_map()` on the first `questions.md` found at `base_dir/masonry/questions.md` or `base_dir/questions.md`. Passes `qid_map` to each `score_finding()` call.

The `score_finding()` function received a matching `qid_map` parameter to thread it through to `extract_finding_fields()`.
