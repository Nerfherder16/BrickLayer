## Summary

Three targeted changes to `masonry/scripts/score_findings.py` enable scoring of masonry self-research findings: removing `"masonry"` from the discovery exclusion list, fixing the `_extract_section` regex to not stop at `###` subsections, and adding `FIX_APPLIED` and `COMPLETE` to `VALID_VERDICTS`. Before the fixes, 0 masonry findings were discovered when scanning from the repo root (the `masonry/` directory was explicitly excluded). After the fixes, 140 masonry findings are discovered and 77 pass the 60-point training threshold (55%), adding 210 net new training-ready records (from 61 to 271 total).

### Fix Applied

**Change 1 — Line 282: Remove `"masonry"` from exclusion list**

```python
# Before (line 282):
if child.is_dir() and child.name not in ("findings", "masonry", ".git"):

# After (line 282):
if child.is_dir() and child.name not in ("findings", ".git"):
```

**Change 2 — Line 133: Fix `_extract_section` regex lookahead**

```python
# Before (line 133):
rf"^##\s+{re.escape(section_name)}\s*\n(.*?)(?=^##|\Z)",

# After (line 133):
rf"^##\s+{re.escape(section_name)}\s*\n(.*?)(?=^## [^#]|\Z)",
```

The original `(?=^##|\Z)` pattern with `re.MULTILINE` matches `###` because `##` is a prefix of `###`. The fix requires `## ` (double hash + space) followed by a non-`#` character, which `### SubsectionName` does not satisfy. This allows `###` subsection headers to remain inside the captured body.

**Change 3 — Lines 23-24: Add `FIX_APPLIED` and `COMPLETE` to `VALID_VERDICTS`**

```python
# Before:
"FIXED", "FIX_FAILED", "COMPLIANT", "NON_COMPLIANT", "PARTIAL",

# After:
"FIXED", "FIX_FAILED", "FIX_APPLIED", "COMPLETE",
"COMPLIANT", "NON_COMPLIANT", "PARTIAL",
```

`FIX_APPLIED` is used by fix-implementer findings (38 of 140 masonry findings). `COMPLETE` is used by 1 masonry finding. Both were previously treated as unrecognized verdicts, costing 10 points on verdict_clarity per finding.
