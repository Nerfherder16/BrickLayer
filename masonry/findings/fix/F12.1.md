## Changes Applied

### `masonry/scripts/sync_verdicts_to_agent_db.py`

Added confidence extraction alongside verdict extraction. When `--questions-md` is specified (scoped sync), all agents' `verdicts` and `confidences` lists are cleared before writing (F12.2 scope-clear, applied in same change):

```python
# Extract confidence floats alongside verdicts
confidences_by_agent: dict[str, list[float]] = defaultdict(list)
for finding in findings:
    ...
    try:
        conf = float(finding.get("confidence", 0.75) or 0.75)
        confidences_by_agent[agent].append(conf)
    except (TypeError, ValueError):
        confidences_by_agent[agent].append(0.75)

# Scope-clear: when --questions-md specified, zero all verdicts/confidences first
if questions_md_path is not None:
    for entry in agent_db.values():
        if isinstance(entry, dict):
            entry["verdicts"] = []
            entry["confidences"] = []

# Write confidences alongside verdicts
agent_db[agent_name]["confidences"] = confidences_by_agent.get(agent_name, [])
```

### `masonry/src/dspy_pipeline/drift_detector.py`

1. `detect_drift()` signature updated to accept `list[str] | list[float]`. When floats are passed, mean is computed directly (no `_score_verdict` conversion).
2. `run_drift_check()` now prefers `confidences` when present in agent_db, falls back to verdict scoring.

```python
# In run_drift_check():

```

## Why Confidence Is the Correct Signal

- A research agent finding `FAILURE` means the system under investigation has a bug. This is *correct agent behavior*.
- A research agent finding `FAILURE` with low confidence (0.5) means the agent is uncertain — this is the actual quality concern.
- Mean confidence across findings is a direct measure of agent certainty, independent of finding polarity.
- All masonry agents consistently produce high-confidence findings (0.88–1.00 mean), confirming they are well-calibrated.
