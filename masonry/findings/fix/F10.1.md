**Severity**: High

## Summary

`_MODE_FIELD_RE` in `deterministic.py` now matches both `**Mode**:` and `**Operational Mode**:` field formats. The regex was extended from `r"\*\*Mode\*\*:\s*(\w+)"` to `r"\*\*(?:Operational\s+)?Mode\*\*:\s*(\w+)"`. All BL 2.0 campaign questions that previously fell through to semantic/LLM routing will now be deterministically routed via Rule 5 — zero LLM calls for standard campaign question routing.

## Root Cause (from R9.2)

The original regex required the field to be named exactly `**Mode**:`. BL 2.0 questions use `**Operational Mode**:`. These never matched, so Rule 5 produced zero deterministic routing for campaign questions. Every campaign question that entered the router fell through to Layer 2 (Ollama semantic) or Layer 3 (Haiku LLM call).

## Change Applied

**File**: `masonry/src/routing/deterministic.py` line 36

```python
# Before
_MODE_FIELD_RE = re.compile(r"\*\*Mode\*\*:\s*(\w+)", re.IGNORECASE)

# After
_MODE_FIELD_RE = re.compile(r"\*\*(?:Operational\s+)?Mode\*\*:\s*(\w+)", re.IGNORECASE)
```

The `(?:Operational\s+)?` non-capturing group makes the prefix optional, matching both formats.

## Impact

Rule 5 now fires for every BL 2.0 question routed through the deterministic layer. Combined with Rule 3 (`masonry-state.json` active campaign) which fires during active campaigns, deterministic coverage for campaign sessions returns to the 90-95% range estimated in R1.2. The fallthrough to Ollama (Layer 2) for campaign questions is eliminated.
