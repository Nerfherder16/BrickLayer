## Summary

Both write sites in `scripts/onboard_agent.py` (`append_to_registry()` line 258 and `upsert_registry_entry()` line 328) now use the atomic rename pattern: write to `.yml.tmp.{pid}`, then `Path.replace()`. A mid-write process kill can no longer truncate `agent_registry.yml` to zero/partial YAML, so all 46 registered agents are protected from silent loss.

**After:**

## Why This Fix Is Correct

`Path.replace()` calls `os.rename()` internally — atomic on POSIX (guaranteed by POSIX spec) and atomic on Windows when both paths are on the same drive/filesystem (same as F5.1 and F5.3 which used `renameSync` in Node.js).

The PID suffix (`os.getpid()`) prevents collision between concurrent `onboard_agent.py` instances. If two instances run simultaneously (e.g., two hooks firing in parallel), each writes its own `.tmp.{pid}` file and the last `replace()` wins — the registry is always fully written.

The exception handler calls `tmp_path.unlink(missing_ok=True)` to clean up the temp file on failure, then re-raises so the caller gets the error.

The remaining `write_text` at `generate_dspy_signature_stub()` (line 408) writes a generated `.py` stub, not the registry — that write is for a new file that doesn't need atomic protection.
