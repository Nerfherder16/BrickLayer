## Summary

The `agents.json` write in `masonry-subagent-tracker.js` is now atomic. A new `atomicWrite()` helper replaces the direct `safeWrite()` call for the agents state file. `safeWrite()` is retained for `masonry-state.json` (non-critical metadata, deferred).

## Changes Applied

### New helper: `atomicWrite(p, obj)`

```javascript
function atomicWrite(p, obj) {
  const tmp = `${p}.tmp.${process.pid}`;
  try {
    fs.writeFileSync(tmp, JSON.stringify(obj, null, 2), "utf8");
    fs.renameSync(tmp, p);
  } catch (_) {
    try { fs.unlinkSync(tmp); } catch (_2) {}
  }
}
```

Uses `process.pid` suffix to distinguish concurrent instances' temp files. Cleanup on failure prevents orphaned `.tmp.*` files.

### `main()` — `agents.json` write

```javascript
// Before:
safeWrite(stateFile, state);

// After:
atomicWrite(stateFile, state);
```

## What Remains Non-Atomic (Acceptable)

| File | Write type | Rationale |
|------|-----------|-----------|
| `agents.json` | **Atomic rename** (fixed) | Read-modify-write, shared state, concurrent risk |
| `masonry-state.json` | `safeWrite` (direct) | Non-critical status field (`active_agent`), staleness acceptable |
| `routing_log.jsonl` | `appendFileSync` | Append-only, OS-serialized, no TOCTOU |

The `masonry-state.json` write at lines 103-105 sets `active_agent` and `active_agent_count` — a last-write-wins race here is acceptable since both concurrent writers would set valid (if potentially stale) values. This is monitoring/statusline data only.
