## Summary

`masonry-observe.js` (PostToolUse Write/Edit hook) now appends a `"finding"` event to `routing_log.jsonl` whenever a finding file is written with a non-UNKNOWN verdict. This pairs with `"start"` events from `masonry-subagent-tracker.js`, enabling `score_routing.py`'s `downstream_success` dimension (30pts) to detect that a dispatched agent produced a result. Before this fix, all routing training records scored at most 70/100. After this fix, sessions with confirmed findings can score up to 100/100.

## Change Applied

**File**: `masonry/src/hooks/masonry-observe.js`

Added after `writeState()` call in the finding detection branch:

```javascript
// Emit "finding" event to routing_log.jsonl for DSPy routing training signal (F14.2)
// Pairs with "start" events from masonry-subagent-tracker.js to score downstream_success.
if (verdict !== 'UNKNOWN') {
  const agentField = extractMarkdownField(fileContent, 'Agent') || 'unknown';
  const routingLogPath = path.join(cwd, 'masonry', 'routing_log.jsonl');
  const findingEntry = JSON.stringify({
    timestamp: new Date().toISOString(),
    event: 'finding',
    agent: agentField,
    session_id: sessionId,
    verdict,
    qid,
  });
  try {
    const masonryDir = path.join(cwd, 'masonry');
    if (fs.existsSync(masonryDir)) {
      fs.appendFileSync(routingLogPath, findingEntry + '\n', 'utf8');
    }
  } catch (_err) { /* non-fatal */ }
}
```

## How `score_routing.py` Uses This

`score_routing.py` pairs "start" events (mortar dispatched agent X) with "finding" events (agent X produced verdict Y). The pairing uses `session_id`. A "finding" event in the same session = `downstream_success=30pts`. A session with both "start" and "finding" events now scores 100/100 (70 correct_dispatch + 30 downstream_success), exceeding `min_training_score=65`.
