# Finding: F2.2 — V1.5 Path Blocklist Has Two Bypasses: Relative `src/` and `docs/` Paths; Bash Vector Unblocked

**Question ID**: F2.2
**Status**: DONE
**Verdict**: FAILURE
**Agent**: fix-implementer
**Severity**: Medium
**Confidence**: 1.0

## Summary

The V1.5 fix specification's path blocklist has two structural weaknesses:
1. **Relative path bypass**: Patterns `/[/\\]src[/\\]/` and `/[/\\]docs[/\\]/` require a path separator BEFORE `src` or `docs`. A `file_path` value of `src/hooks/foo.js` (no leading slash) bypasses both patterns. Developer agents commonly emit relative paths.
2. **Bash tool vector unblocked**: The fix applies `isTier1Tier2(filePath)` where `filePath = toolInput.file_path || toolInput.path || ''`. For Bash tool calls, both fields are undefined, producing `filePath = ''` → `isTier1Tier2('')` returns `false` → all Bash commands are auto-approved. A developer agent running `echo x >> project-brief.md` or `cp template project-brief.md` bypasses the blocklist entirely.

The suffix-based patterns (1–4) correctly handle path traversal (`../../masonry/project-brief.md` is caught). Only the directory patterns (5–6) have the bypass.

## Evidence

### Pattern-by-Pattern Analysis

**V1.5 proposed patterns:**
```javascript
const TIER1_TIER2_PATTERNS = [
  /project-brief\.md$/i,        // 1 — suffix match
  /agent_registry\.yml$/i,      // 2 — suffix match
  /hooks\.json$/i,               // 3 — suffix match
  /constants\.py$/i,             // 4 — suffix match
  /[/\\]src[/\\]/,               // 5 — directory match
  /[/\\]docs[/\\]/,              // 6 — directory match
];
```

#### Patterns 1–4 (Suffix-Based): Path Traversal BLOCKED ✓

| Input `file_path` | Pattern 1 match? | Result |
|-------------------|-----------------|--------|
| `project-brief.md` | Yes (`$` matches) | BLOCKED ✓ |
| `../../masonry/project-brief.md` | Yes (ends with `project-brief.md`) | BLOCKED ✓ |
| `C:\Users\foo\project-brief.md` | Yes | BLOCKED ✓ |
| `/home/user/masonry/project-brief.md` | Yes | BLOCKED ✓ |
| `myproject-brief.md` | No (prefixed) | allowed — correct |

Suffix patterns are path-traversal-safe because they test the terminal component of the path regardless of prefix. No bypass exists for these.

#### Pattern 5 `/[/\\]src[/\\]/`: Relative Path BYPASS ✗

| Input `file_path` | Match? | Result |
|-------------------|--------|--------|
| `C:\Users\trg16\Dev\masonry\src\hooks\foo.js` | Yes (`\src\`) | BLOCKED ✓ |
| `/home/user/masonry/src/hooks/foo.js` | Yes (`/src/`) | BLOCKED ✓ |
| `../../masonry/src/hooks/foo.js` | Yes (`/src/`) | BLOCKED ✓ |
| `./src/hooks/foo.js` | Yes (`/src/`) | BLOCKED ✓ |
| **`src/hooks/foo.js`** | **No** (no separator before `src`) | **BYPASS ✗** |
| **`src/constants.py`** | **No** | **BYPASS ✗** |

**The bypass**: When `file_path` starts with `src/` (no `./` prefix, no absolute prefix), the pattern fails. This is a realistic path format — developer agents routinely emit `src/hooks/masonry-guard.js` as a relative path.

#### Pattern 6 `/[/\\]docs[/\\]/`: Same Bypass ✗

| Input `file_path` | Match? | Result |
|-------------------|--------|--------|
| `/masonry/docs/architecture.md` | Yes | BLOCKED ✓ |
| `./docs/architecture.md` | Yes (`/docs/`) | BLOCKED ✓ |
| **`docs/architecture.md`** | **No** | **BYPASS ✗** |

### Bash Tool Vector (V1.5 Fix Is Incomplete)

The current approver code extracts `file_path` as:
```javascript
const toolInput = parsed.tool_input || {};
const filePath = toolInput.file_path || toolInput.path || '';
```

For a Bash tool call, `tool_input` looks like:
```json
{
  "command": "echo 'overwritten' > project-brief.md",
  "restart": false
}
```
Neither `file_path` nor `path` is present. `filePath = ''`. `isTier1Tier2('')` → `false`. The write is auto-approved.

`getCandidateDirs` does parse paths from the command string (via regex at lines 101–115), but those are used only for `.autopilot/mode` directory traversal — they are not passed to any blocklist check.

Attack vectors through the Bash gap:
- `echo "x" >> project-brief.md`
- `cp .autopilot/spec.md project-brief.md`
- `python -c "open('agent_registry.yml','w').write('...')"`
- `sed -i 's/old/new/' constants.py`

All of these would be auto-approved during an active build because the V1.5 blocklist is never reached for Bash calls.

### Corrected Fix Specification

**Fix for patterns 5–6 (relative path bypass):**
```javascript
const TIER1_TIER2_PATTERNS = [
  /project-brief\.md$/i,
  /agent_registry\.yml$/i,
  /hooks\.json$/i,
  /constants\.py$/i,
  /(?:^|[/\\])src[/\\]/,        // fixed: allow start-of-string
  /(?:^|[/\\])docs[/\\]/,       // fixed: allow start-of-string
];
```

**Fix for Bash tool vector (complement to V1.5 blocklist):**
```javascript
function isTier1Tier2Bash(command) {
  if (!command) return false;
  // Extract file targets from common write commands: >, >>, cp dest, mv dest
  const writeTargets = [];
  // Redirect target: > filename or >> filename
  const redirectMatches = command.match(/>{1,2}\s*([^\s;&|]+)/g) || [];
  for (const m of redirectMatches) {
    writeTargets.push(m.replace(/^>{1,2}\s*/, ''));
  }
  // cp/mv destination (last argument)
  const cpMatch = command.match(/\bcp\b.*\s+([^\s]+)$/);
  if (cpMatch) writeTargets.push(cpMatch[1]);
  const mvMatch = command.match(/\bmv\b.*\s+([^\s]+)$/);
  if (mvMatch) writeTargets.push(mvMatch[1]);
  // sed -i in-place edit (file argument)
  const sedMatch = command.match(/\bsed\b.*-i\b.*\s+([^\s]+)$/);
  if (sedMatch) writeTargets.push(sedMatch[1]);

  return writeTargets.some(t => isTier1Tier2(t));
}

// In main(), updated check:
const toolInput = parsed.tool_input || {};
const toolName = parsed.tool_name || '';
const filePath = toolInput.file_path || toolInput.path || '';
const bashCommand = toolInput.command || '';

const isSensitiveWrite = isTier1Tier2(filePath) ||
  (toolName === 'Bash' && isTier1Tier2Bash(bashCommand));

if (approve && isSensitiveWrite) {
  process.exit(0);
}
```

Note: The Bash regex parser is intentionally conservative — it catches the most common patterns (`>`, `cp`, `mv`, `sed -i`) but cannot catch all possible write paths (e.g., `python -c "..."`, `node -e "..."`). For full coverage, the Bash vector cannot be reliably blocked without executing the command. The pragmatic recommendation is to **not auto-approve Bash tool calls at all** during build mode, requiring user confirmation for any Bash execution. This is a more conservative but safer approach:

```javascript
// Safest Bash policy: never auto-approve Bash during build
if (approve && toolName === 'Bash') {
  process.exit(0);  // fall through to user prompt
}
```

This preserves auto-approval for Write/Edit (the main build workflow) while requiring user confirmation for all Bash calls. Developer agents that need to run tests, install packages, or execute scripts will prompt the user, which is acceptable since these are lower-frequency and higher-risk operations.

## Verdict

**FAILURE** — The V1.5 fix specification has two weaknesses that must be corrected before implementation:
1. Patterns 5 and 6 need `(?:^|[/\\])` prefix to catch relative paths
2. The Bash tool vector requires an additional check (either targeted command parsing or blanket Bash exclusion from auto-approval)

The fix for issue 1 is a two-character change per pattern. The recommended fix for issue 2 is to exclude Bash from auto-approval entirely. Both corrections are required for the fix to be implementation-ready.
