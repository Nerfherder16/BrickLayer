## Summary

The non-atomic `fs.writeFileSync()` write on the strike counter file in `masonry-guard.js` has been replaced with a write-to-tmp + `fs.renameSync()` pattern. Concurrent PostToolUse executions can no longer produce torn or lost counter updates.

## Change Applied

### `src/hooks/masonry-guard.js` â€” strike counter write

**Before:**
```javascript
// Persist updated counts
try {
  fs.writeFileSync(guardCountFile, JSON.stringify(counts), 'utf8');
} catch (_err) { /* non-fatal */ }
```

**After:**
```javascript
// Persist updated counts â€” atomic rename to prevent torn writes under concurrent execution
const guardCountTmp = `${guardCountFile}.tmp.${process.pid}`;
try {
  fs.writeFileSync(guardCountTmp, JSON.stringify(counts), 'utf8');
  fs.renameSync(guardCountTmp, guardCountFile);
} catch (_err) {
  try { fs.unlinkSync(guardCountTmp); } catch (_) { /* cleanup best-effort */ }
}
```

## Why This Fix

`fs.renameSync()` on POSIX systems is an atomic operation â€” the destination file is replaced atomically. On Windows, `fs.renameSync()` is best-effort atomic (uses `MoveFileExW` which is documented as atomic on the same volume). Either way, the guard counter is never partially-written; a reader will see either the old complete JSON or the new complete JSON.

The `process.pid` suffix ensures uniqueness: if two guard instances run concurrently, their temp files are distinct (`masonry-guard-<session>.json.tmp.<pid1>` vs `.tmp.<pid2>`). The final rename is a last-write-wins race â€” acceptable for a strike counter where approximate counts are sufficient.

The cleanup `unlinkSync` in the catch branch prevents orphaned `.tmp.*` files in tmpdir if the rename fails for any reason.
