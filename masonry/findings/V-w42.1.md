# Finding: V-w42.1 — Train/Eval Split Chain Correctly Wired, Two Latent Risks

**Question**: Verify F-w41.1 train/eval split — does _tier_examples() actually exclude held_out_ids in practice?
**Agent**: design-reviewer
**Verdict**: WARNING
**Severity**: Medium

## Evidence

### E1: _record_id() hash output is byte-for-byte identical between files

`eval_agent.py _record_id(record, index)`: `sort_keys=True`, `ensure_ascii=False`, `[:16]` prefix.
`optimize_with_claude.py _record_id(record)`: same parameters, same prefix.
Hash contract is consistent — the `index` parameter on the eval side is accepted but never used in the body. Output is identical for the same record.

### E2: held_out_ids chain fully wired

- `eval_agent.py` lines 242-244: `held_out_ids = [_record_id(rec, i) for i, rec in enumerate(held_out)]`
- `eval_agent.py` line 363: `"held_out_ids": held_out_ids` in result dict
- `improve_agent.py` lines 126, 133: `set(before_result.get("held_out_ids", []))` → passed as `excluded_ids`
- `optimize_with_claude.py` lines 103-108: `if excluded_ids: records = [r for r in records if _record_id(r) not in excluded_ids]`

Chain executes correctly when called through `improve_agent.py`.

### E3: _tier_examples() filter executes before sort and tier split

Exclusion at line 103-104 fires before `sorted_records` assignment (line 106) and before tier selection (lines 107-108). No held-out record can survive into either tier.

### E4: _MIN_TIER_RECORDS guard is correct

Lines 300-317: hard bail on combined `< 5`, warn-and-continue on per-tier `< 5`. The hard bail protects against degenerate prompts; the per-tier warning surfaces signal quality issues without aborting.

### E5: Dead `index` parameter on eval-side _record_id()

`eval_agent._record_id(record, index)` accepts `index: int` but never reads it. Call site passes `i` from `enumerate(held_out)`. The optimize-side `_record_id(record)` has no such parameter. Any future direct call to the eval-side function without the second argument raises `TypeError`. This is interface drift.

### E6: CLI bypass — optimize_with_claude.py has no --excluded-ids argument

`optimize_with_claude._main()` (lines 462-497): no `--excluded-ids` argument. Direct CLI invocation passes `excluded_ids=None` to `run()`. All records enter the training pool — the convergence trap F-w41.1 was designed to close is re-opened for any caller that bypasses `improve_agent.py`. No warning is printed.

### E7: F-w41.1 E4 documentation inaccuracy

F-w41.1 described the filter as: `available = [r for r in sorted_records if _record_id(r) not in (excluded_ids or set())]`
Actual code: `if excluded_ids: records = [r for r in records if _record_id(r) not in excluded_ids]`
Functionally equivalent but textually different. The finding file contains a minor inaccuracy.

## Analysis

The chain works correctly through the designed entry point (`improve_agent.py`). The two latent risks are:

1. **Dead `index` parameter**: causes `TypeError` for any future caller that invokes `eval_agent._record_id()` with one argument. Since the optimize-side version takes one argument, any caller written to match the optimize-side signature will fail on the eval side. Easy fix: remove the `index` parameter.

2. **CLI bypass**: `optimize_with_claude.py` CLI users silently skip the train/eval split. The CLAUDE.md docs show `improve_agent.py` as entry point, but any automation script or Kiln integration that calls `optimize_with_claude.py` directly re-opens the convergence trap without any warning.

## Verdict Reasoning

WARNING: Chain is correctly wired for the designed entry point. _record_id() hash contract is consistent. _MIN_TIER_RECORDS guard fires correctly. Two latent risks: (1) dead `index` parameter causes TypeError on direct single-argument calls — requires removing unused parameter; (2) optimize_with_claude.py CLI silently bypasses train/eval split — requires either adding --excluded-ids argument or a warning when excluded_ids=None. Neither causes a current failure but both represent traps for future callers.
