# Finding: D22.1 — confidence_calibration in build_metric() systematically penalizes correct high-confidence findings

**Question ID**: D22.1
**Status**: DONE

## Summary

The `_score_confidence_calibration()` function in `score_findings.py` uses a hard band of `[0.5, 0.95]` — any confidence value above 0.95 receives only 10 out of 40 possible points (presence credit only, no in-band or severity-match credit). This causes 77 findings with confidence > 0.95 to lose 30 points each. Of those, 40 findings that would otherwise meet the training threshold of 60 are suppressed entirely from training data. This is a 14.4% reduction in training volume. A separate but related issue exists in `optimizer.py` where `build_metric()` uses a center-biased formula (`1 - |conf - 0.75|`) that penalizes confidence=1.0 by 25% relative to the ideal 0.75.

## Evidence

### File 1: `masonry/scripts/score_findings.py`, lines 163-190

```python
def _score_confidence_calibration(confidence, verdict, severity):
    points = 0
    if confidence is None:
        return 0
    points += 10                          # +10: presence
    if 0.5 <= confidence <= 0.95:         # LINE 178: hard upper bound at 0.95
        points += 15                      # +15: in-band
        # +15: severity match (nested, only reachable if in-band)
        is_critical = verdict in _CRITICAL_VERDICTS or severity in _CRITICAL_SEVERITIES
        if is_critical and confidence >= 0.70:
            points += 15
        elif not is_critical and confidence <= 0.85:
            points += 15
    return points
```

The severity-match bonus (+15) is nested inside the band check. A finding with confidence=0.97 and a FAILURE verdict gets 10/40. The same finding at confidence=0.95 gets 40/40.

### File 2: `masonry/src/dspy_pipeline/optimizer.py`, lines 56-63

```python
# Confidence calibration (0.2 weight)
raw = str(getattr(prediction, "confidence", "0.75") or "0.75")
pred_conf = float(raw)
calibration = 1.0 - abs(pred_conf - 0.75)
score += 0.2 * calibration
```

This formula centers on 0.75 as the "ideal" confidence. At conf=1.0, the contribution is `0.2 * 0.75 = 0.15` instead of the maximum `0.20`. The penalty is softer here (a gradient, not a cliff) but still systematically under-rewards high-confidence predictions.

## Root Cause

The upper bound of the confidence band at line 178 of `score_findings.py` (`confidence <= 0.95`) treats any value above 0.95 as "overconfident" and drops 30 points. The intent was to discourage agents from always outputting 0.99, but the implementation cannot distinguish between a legitimately certain diagnose-analyst finding (code-level root cause confirmed, confidence=1.0 is appropriate) and a lazy agent defaulting to 1.0. The band exclusion is a cliff, not a gradient — 0.95 scores 40/40 while 0.96 scores 10/40.

In `optimizer.py`, the center-bias formula at line 60 (`1 - |pred_conf - 0.75|`) assumes 0.75 is the ideal confidence for all findings regardless of context, which penalizes any confident prediction.

## Fix Specification

**Option A (recommended): Widen the band to [0.5, 1.0] and move the severity-match check outside the band gate.**

This preserves the original scoring structure while removing the cliff. The severity-match check already handles the "overconfidence for low-stakes findings" concern — a non-critical finding with confidence > 0.85 still loses the 15-point severity-match bonus, which is the correct incentive.
