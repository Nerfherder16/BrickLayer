# Finding: R5.2 — DSPy Pipeline Dry Run: Field Mismatch for `diagnose-analyst` Blocks Full Optimization

**Question ID**: R5.2
**Status**: DONE
**Verdict**: FAILURE
**Agent**: research-analyst
**Severity**: Medium
**Confidence**: 0.95

## Summary

The DSPy optimization pipeline has a field mismatch for `diagnose-analyst`: `build_dataset()` always shapes examples to `ResearchAgentSig` fields (`question_text`, `project_context`, etc.), but `optimize_all()` selects `DiagnoseAgentSig` for agents with `input_schema: DiagnosePayload`. The `DiagnoseAgentSig` expects `symptoms`, `affected_files`, `prior_attempts` — none of which exist in the shaped example dicts. This mismatch would cause a DSPy runtime error when optimizing `diagnose-analyst` (the largest training set: 13 examples). The other four agents (`research-analyst`: 9, `design-reviewer`: 7, `fix-implementer`: 8) use `ResearchAgentSig` with perfect field alignment and would succeed.

## Analysis

### 1. LLM Backend Configuration

`configure_dspy()` at `optimizer.py:73-76`:
```python
def configure_dspy(model: str = "claude-sonnet-4-6") -> None:
    lm = dspy.LM(f"anthropic/{model}")
    dspy.configure(lm=lm)
```

- Uses `anthropic/claude-sonnet-4-6` — ANTHROPIC_API_KEY required
- In Claude Code sessions, ANTHROPIC_API_KEY is available → backend is correctly configured
- **Note**: `configure_dspy()` is NOT called inside `optimize_agent()` or `optimize_all()` — the caller must invoke it first. The MCP tool `masonry_optimize_agent` presumably does this. If called standalone without pre-configuring DSPy, the first call to `module.forward()` would fail with "No LM configured."

### 2. ResearchAgentSig Field Alignment (PASS for research/design/fix agents)

`build_dataset()` shapes all examples (`optimizer.py:213-223`):
```python
example = {
    "question_text": finding.get("question_id", ""),   # input field
    "project_context": "",                              # input field
    "constraints": "",                                  # input field
    "verdict": finding.get("verdict", ""),              # output field
    "severity": finding.get("severity", ""),            # output field
    "evidence": finding.get("evidence", ""),            # output field
    "mitigation": finding.get("mitigation", "") or "",  # output field
    "confidence": "0.75",                               # output field
    "_weight": weight,                                  # extra (DSPy ignores)
}
```

`ResearchAgentSig` fields (`signatures.py:12-41`):
- Input: `question_text`, `project_context`, `constraints`
- Output: `verdict`, `severity`, `evidence`, `mitigation`, `confidence`

**All 8 required fields present. Alignment: PERFECT.**

`_weight` is an extra key — `dspy.Example(**ex).with_inputs(*input_keys)` stores it as an attribute but `.with_inputs()` only marks the requested fields as "inputs". DSPy ignores non-signature fields during training.

### 3. DiagnoseAgentSig Field Mismatch (FAIL for diagnose-analyst)

`optimize_all()` signature selection logic (`optimizer.py:196`):
```python
sig = DiagnoseAgentSig if agent.input_schema == "DiagnosePayload" else ResearchAgentSig
```

`diagnose-analyst` registry entry (`agent_registry.yml:277`):
```yaml
input_schema: DiagnosePayload
```

So `optimize_all()` selects `DiagnoseAgentSig` for `diagnose-analyst`. But `DiagnoseAgentSig` fields are:
- Input: `symptoms`, `affected_files`, `prior_attempts`
- Output: `root_cause`, `fix_strategy`, `affected_scope`, `confidence`

The example dicts from `build_dataset()` have NONE of these fields — they have `question_text`, `project_context`, `verdict`, etc.

**Field mismatch:**
| DiagnoseAgentSig field | In example dict? |
|---|---|
| `symptoms` (input) | NO |
| `affected_files` (input) | NO |
| `prior_attempts` (input) | NO |
| `root_cause` (output) | NO |
| `fix_strategy` (output) | NO |
| `affected_scope` (output) | NO |

During `optimize_agent()`:
```python
input_keys = list(DiagnoseAgentSig.input_fields.keys())
# → ["symptoms", "affected_files", "prior_attempts"]
trainset = [dspy.Example(**ex).with_inputs(*input_keys) for ex in dataset]
```

`dspy.Example(**ex).with_inputs("symptoms", "affected_files", "prior_attempts")` creates an Example with `question_text`, `project_context`, etc. attributes, but marks the non-existent `symptoms/affected_files/prior_attempts` as "inputs". The resulting `Example` objects would have no input values for the signature's expected fields.

When `optimizer.compile()` tries to bootstrap demonstrations using this trainset, it would either:
- (a) Produce garbage demonstrations with empty input fields, or
- (b) Raise a `KeyError` or `AttributeError` on the missing `symptoms` field during the first forward pass

The `try/except` in `optimize_agent()` would catch the exception and fall back to the unoptimized module, but the output JSON would contain `score: 0.0` — a silent failure, not a complete crash.

## Root Cause

`build_dataset()` hardcodes the `ResearchAgentSig` output shape for all agents regardless of the agent's `input_schema`. The comment on line 212 confirms this intent: "# Shape the example to match ResearchAgentSig fields". But `optimize_all()` branches on `input_schema == "DiagnosePayload"` to select `DiagnoseAgentSig` — creating a contract mismatch for `diagnose-analyst`.

**Fix (not applied here — deferred to next Fix wave)**:

Option A: Shape `diagnose-analyst` examples to `DiagnoseAgentSig` fields in `build_dataset()` using a branch on `agent_name` or `input_schema`. The `evidence` section of a finding maps well to `symptoms`, the `question_id` to `affected_files`, and `verdict`/`severity` don't have `DiagnoseAgentSig` output equivalents.

Option B: Simplify `optimize_all()` to always use `ResearchAgentSig` for all agents (removing the `DiagnoseAgentSig` branch). This is simpler and loses only the specialized diagnose signature.

Option B is the lightest fix.
