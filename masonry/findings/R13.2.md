# Finding: R13.2 — `score_routing.py` Produces Valid Output But Low Signal Quality

**Question ID**: R13.2
**Status**: DONE
**Verdict**: WARNING
**Severity**: Medium
**Confidence**: 0.95

## Summary

`score_routing.py` runs without errors. It finds 4 events in `routing_log.jsonl`, matches 0 complete sessions (no "finding" events paired with "start" events), yet produces 3 training records for mortar — all scored 70/100 (correct_agent_dispatched=70, downstream_success=0). All 3 pass `min_training_score=65`. However, the training signal is thin: `downstream_success=0` for all records because no "finding" events exist in the log. Mortar's routing quality cannot be measured without evidence that the dispatched agent produced a result.

All three dispatched agents (karen, planner, question-designer-bl2) are in `AGENT_CATEGORIES` → `correct_agent_dispatched=70`. No "finding" events in routing_log.jsonl → `downstream_success=0`.

## Log Format Verification

The routing_log.jsonl format written by masonry-subagent-tracker.js (phase-16) matches what `score_routing.py` expects:

```json
{"timestamp":"...", "event":"start", "agent":"karen", "session_id":"...", "parent_session":"..."}
```

`score_routing.py` pairs "start" events with "finding" events by `session_id`. "Finding" events would be written when a research agent produces a verdict. Since the 3 "start" events are for setup agents (karen, planner, question-designer-bl2), no findings are expected. This is correct behavior.

## Root Cause of Zero Downstream Success

The `routing_log.jsonl` only captures SubagentStart events. A "finding" event would be emitted by a separate mechanism when a BL research agent writes a finding. Looking at the subagent tracker hook (masonry-subagent-tracker.js), it only logs "start" events — there is no "finding" event emission. The downstream_success dimension (30pts) can never be populated with the current hook implementation.

## Verdict

**WARNING** — `score_routing.py` runs correctly and produces valid training records. The log format is compatible. However, the `downstream_success` dimension is structurally blocked at 0 because masonry-subagent-tracker.js never writes "finding" events. All mortar routing training records will be capped at 70/100. This is sufficient for `min_training_score=65` threshold but represents only 70% of the available training signal. A future fix would emit "finding" events from masonry-observe.js or a dedicated post-finding hook.
