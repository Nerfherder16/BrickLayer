# Finding: V1.3 — Stale Embedding Cache Wastes Memory but Cannot Cause Misrouting

**Question ID**: V1.3
**Status**: DONE
**Verdict**: WARNING
**Agent**: design-reviewer
**Severity**: Low
**Confidence**: 1.0

## Summary

The `_embedding_cache` in `semantic.py` is keyed by agent description + capabilities concatenation. When agents are removed from the registry, their embeddings remain in the cache indefinitely (process-lifetime). Stale entries cannot cause misrouting because similarity is only computed against embeddings for agents currently in the registry — stale entries for deleted agents are never accessed. The issue is memory waste, not routing correctness.

## Evidence

### Cache Keying and Usage (semantic.py lines 23, 70-72)

```python
_embedding_cache: dict[str, list[float]] = {}

def _agent_corpus_key(agent: AgentRegistryEntry) -> str:
    """Build the corpus string for an agent and use it as cache key."""
    return agent.description + " " + ", ".join(agent.capabilities)
```

The cache maps `corpus_key → embedding_vector`.

### How Stale Entries Arise

1. Registry at time T has agents `[A, B, C]` with embeddings cached
2. Agent B is removed from registry (registry reload at time T+1)
3. Cache still contains `B_corpus_key → B_embedding`
4. B's embedding is never accessed again because B is not in the current registry

### Why Stale Entries Cannot Cause Misrouting

The routing code (lines 118-121):
```python
agent_embs: list[tuple[AgentRegistryEntry, list[float]]] = [
    (a, _embedding_cache[_agent_corpus_key(a)])
    for a in registry                     # ← iterates CURRENT registry
    if _agent_corpus_key(a) in _embedding_cache
]
```

The list comprehension iterates `registry` (current agents only). Stale cache entries for deleted agents are never `a` in this loop — they're orphaned in the cache dict but never looked up. The similarity comparison only includes embeddings for currently registered agents.

**Correctness is preserved**: You cannot be routed to a deleted agent because its key is never accessed in the routing loop.

### Memory Impact Assessment

A `qwen3-embedding:0.6b` model produces 1024-dimensional float vectors (based on Qwen3's documented output dimensions). Each embedding = 1024 × 4 bytes (float32) = 4KB.

For a session with 40 agents, maximum cache size = 40 × 4KB = 160KB. If agents are frequently added and removed (e.g., during testing), the cache could grow to 2× or 3× this, reaching ~500KB. For a long-running MCP server process, this is negligible.

However, if `_embedding_cache` grows unbounded due to schema migrations that change agent descriptions (each migration creates a new cache key for the same agent), the cache could grow larger. But this is an edge case.

### Registry Reload Frequency

The cache is module-level (process-lifetime). `route()` calls `_load_registry()` on every invocation, but `route_semantic()` uses the module-level `_embedding_cache`. If an agent's description changes:

1. Old key: `"old description old_cap"` → cached embedding
2. New registry entry: `"new description new_cap"` → not in cache → triggers new Ollama call
3. Old key remains in cache as dead weight

This is the memory waste scenario described in the hypothesis — confirmed but benign.

### One Correctness Edge Case (Not Currently Exploitable)

If two different agents happened to have identical `description + capabilities` strings (which would be a registry data quality issue), they would share a cache entry. The second agent's embedding would be looked up from the first agent's cached value, potentially routing the second agent using the first agent's embedding. In practice, agent descriptions should be unique, making this theoretical.

## Conclusion

**Verdict: WARNING** (not FAILURE). The stale cache:
- ✓ Does NOT cause misrouting to non-existent agents
- ✓ Does NOT cause routing to wrong existing agents
- ✗ Does grow monotonically during long sessions with frequent registry changes
- ✗ Could cause slight inflation of cache size in testing scenarios

No immediate action required. If long-lived MCP server processes become a concern, a cache size limit (e.g., `if len(_embedding_cache) > 200: _embedding_cache.clear()`) would prevent unbounded growth.
