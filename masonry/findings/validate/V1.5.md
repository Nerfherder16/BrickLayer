# Finding: V1.5 — masonry-approver Auto-Approves All Writes During Build Mode Regardless of Target File Path

**Question ID**: V1.5
**Status**: DONE
**Verdict**: FAILURE
**Agent**: design-reviewer
**Severity**: High
**Confidence**: 1.0

## Summary

`masonry-approver.js` grants `permissionDecision: "allow"` for ALL Write/Edit/Bash tool calls when `.autopilot/mode` is "build" or "fix", with no filtering by target file path. During an active build, writes to `project-brief.md`, `agent_registry.yml`, `constants.py`, or any Tier 1/2 authority file are auto-approved. This violates the project's source authority hierarchy, which designates Tier 1/2 files as human-only.

## Evidence

### The Approval Logic (masonry-approver.js lines 146-163)

```javascript
const approve =
  (autopilotMode === "build" || autopilotMode === "fix") ||
  (uiMode === "compose" || uiMode === "fix");

if (approve) {
  const reason = autopilotMode
    ? `Masonry build mode "${autopilotMode}" active`
    : `UI workflow mode "${uiMode}" active`;
  process.stdout.write(
    JSON.stringify({
      hookSpecificOutput: {
        hookEventName: "PreToolUse",
        permissionDecision: "allow",
        permissionDecisionReason: reason,
      },
    }),
  );
}
```

The `approve` decision is binary: if any active mode is found, **all** writes are approved. There is no code path that checks the file path being written.

### Path of Approval Decision

1. `getCandidateDirs(parsed)` — extracts candidate directories from `file_path`, `command`, or `cwd`
2. `findAutopilotMode(dir)` — walks up from each candidate looking for `.autopilot/mode`
3. `findUiMode(dir)` — similar for `.ui/mode`
4. `approve = (autopilotMode === "build" || ...)` — binary decision, no path filter

Steps 1-3 determine *whether a build is active*. Step 4 approves all writes unconditionally if a build is active.

### Tier 1/2 Files That Are Auto-Approved During Build

Per project-brief.md source authority hierarchy:
- **Tier 1** (human-only): `project-brief.md`, `docs/*`
- **Tier 2** (read-only for agents): All `src/` code, `hooks.json`, `agent_registry.yml`

During an autopilot build, if the spec instructs Claude to modify any of these files (incorrectly or maliciously), masonry-approver would auto-approve without prompting the user. Example scenarios:

1. A bad spec file tells the developer agent to "update project-brief.md with findings" → auto-approved
2. An agent in the build pipeline modifies `agent_registry.yml` to add itself as trusted → auto-approved
3. A corrupted build task writes to `hooks.json` → auto-approved

### Freshness Guard Does Not Help

The code includes a freshness guard for stale mode files:
```javascript
if (mode === "build" || mode === "fix") {
  const progressFile = join(dir, ".autopilot", "progress.json");
  if (!isFresh(progressFile)) return null;
}
```

This guard correctly rejects auto-approval from stale/crashed builds (good). But it only applies temporally — during an active, fresh build, all writes are approved regardless of path.

### Risk Assessment

This is not a theoretical concern. During an active `/build` session:
- The developer worker agent has Claude Code write arbitrary files
- masonry-approver sees every Write/Edit call and checks mode
- A misguided developer agent writing to `agent_registry.yml` would be silently approved

The original design intent is correct — auto-approve writes during builds to avoid interrupting the automated process. The implementation is over-broad.

## Mitigation

Add a path-based blocklist for Tier 1/2 files:

```javascript
const TIER1_TIER2_PATTERNS = [
  /project-brief\.md$/i,
  /agent_registry\.yml$/i,
  /hooks\.json$/i,
  /constants\.py$/i,
  /[/\\]src[/\\]/,     // all src/ files
  /[/\\]docs[/\\]/,    // all docs/ files
];

function isTier1Tier2(filePath) {
  if (!filePath) return false;
  return TIER1_TIER2_PATTERNS.some(re => re.test(filePath));
}

// In main(), before writing approval:
const toolInput = parsed.tool_input || {};
const filePath = toolInput.file_path || toolInput.path || '';
if (approve && isTier1Tier2(filePath)) {
  // Do not auto-approve — let the permission system prompt the user
  process.exit(0);
}
```

This preserves auto-approval for the vast majority of build writes while requiring explicit user confirmation for authority-sensitive files.
