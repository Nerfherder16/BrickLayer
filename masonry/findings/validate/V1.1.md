## Summary

`llm_router.py` constructs `RoutingDecision` using only named arguments (`target_agent`, `layer`, `confidence`, `reason`), never by unpacking the parsed JSON dict. The `extra="forbid"` configuration on `RoutingDecision` is safe in this context. The LLM router cannot trigger a Pydantic validation failure from extra JSON fields because it explicitly extracts only `target_agent` and `reason` from the parsed response and discards all other fields.

## Evidence

### The Exact Construction (llm_router.py lines 93-104)

```python
target = parsed.get("target_agent")
if not target:
    print("[llm_router] No target_agent in LLM response.", file=sys.stderr)
    return None

reason = parsed.get("reason", "LLM routing")[:100]

return RoutingDecision(
    target_agent=target,
    layer="llm",
    confidence=_LLM_CONFIDENCE,
    reason=reason,
)
```

The constructor receives exactly 4 keyword arguments:
- `target_agent`: extracted from parsed JSON with `get("target_agent")`
- `layer`: hardcoded string `"llm"`
- `confidence`: constant `_LLM_CONFIDENCE = 0.6`
- `reason`: extracted from parsed JSON with `get("reason", "LLM routing")[:100]`

The `parsed` dict is never unpacked (`**parsed` is not used). Any extra fields in the LLM JSON response (e.g., `{"target_agent": "foo", "reason": "bar", "explanation": "baz", "metadata": {...}}`) are silently discarded by the explicit `get()` calls. Only two fields are extracted; all others are ignored.

### RoutingDecision Schema Fields

```python
class RoutingDecision(BaseModel):
    model_config = ConfigDict(extra="forbid")
```

The `extra="forbid"` would only trigger if an unexpected kwarg were passed to the constructor. Since only `target_agent`, `layer`, `confidence`, `reason` are passed, and `fallback_agents` has a `default_factory=list`, the construction is valid.

### One Remaining Concern: `layer` Literal Validation

The `layer` field is `Literal["deterministic", "semantic", "llm", "fallback"]`. In `llm_router.py`, the hardcoded value `"llm"` matches the Literal. This is safe.

However, if a future refactor changes the layer name in the Literal without updating `llm_router.py`, a `ValidationError` would occur. This is a minor maintenance concern, not a current bug.
