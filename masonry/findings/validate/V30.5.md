## Hypothesis Under Test

CLAUDE.md claims: "Mortar injects optimized prompts on specialist invocation automatically."
MIPROv2 optimization produces JSON files in `masonry/optimized_prompts/{agent}.json`.
The hypothesis is that some code path reads these files at spawn time and prepends the
`instructions` field to the agent's system prompt before execution.

### optimizer.py — what the JSON file actually contains

`dspy.module.save()` stores the DSPy program state: demos, traces, and a
`signature.instructions` field (the optimized system prompt text). The `optimize_agent`
function then merges `agent`, `score`, and `optimized_at` into this file.

The `instructions` field IS present in the saved JSON (confirmed via
`masonry/optimized_prompts/quantitative-analyst.json`, line 40):

```json
"signature": {
  "instructions": "Analyze the research question within the project context, applying
    domain-specific reasoning and numerical modeling...",
  ...
}
```

But nothing reads this field when spawning an agent.

## Verdict Analysis

Three components are required for injection to work:

| Component | Status |
|-----------|--------|
| Optimization produces JSON with `instructions` field | PRESENT — `quantitative-analyst.json` confirmed |
| Code reads `instructions` from JSON at spawn time | ABSENT — no such code exists anywhere |
| Code prepends `instructions` to agent system prompt | ABSENT — no such code exists anywhere |

The injection path is broken at step 2. The optimized JSON is written and is
structurally correct, but nothing reads it when spawning a specialist agent.

**CLAUDE.md claim is partially true**: "Mortar injects optimized prompts on specialist invocation automatically." The code-enforced injection path is absent (no hook, runner, or router reads these files). However, **peer review OVERRIDE found that `mortar.md` line 277 contains**: "If `masonry/optimized_prompts/{agent_name}.json` exists, read it and inject the optimized instructions into the specialist invocation." This is an LLM-honor-based injection directive — Mortar (as an LLM orchestrator) is instructed to read and apply optimized prompts. This is weaker than code enforcement but is a non-zero injection path. The original finding missed this line.

---

## Threshold Analysis

No `constants.py` threshold applies directly. The question is binary:

- HEALTHY = complete injection path exists in code (hook/runner/router)
- WARNING = injection only documented in Mortar instructions (LLM-honor-based)
- FAILURE = no injection path exists anywhere

Evidence places this at **WARNING**: `mortar.md` line 277 provides an LLM-honor-based injection directive. Code-enforced injection via hooks, runner, or router remains absent. The original finding's FAILURE verdict was based on a miss of mortar.md line 277; the peer review correctly upgraded this to WARNING.

## Confidence

Evidence quality: **HIGH**

Reasoning: Direct code inspection of every file that references `optimized_prompts`.
The absence of injection code is definitive — this is not a configuration or runtime
issue. The JSON schema, the DSPy optimizer, and the MCP status tool are all confirmed
by reading primary source code. The one existing optimized JSON (`quantitative-analyst.json`)
confirms the file format is correct, making the absence of a reader unambiguous.

## What Would Change This Verdict

Evidence that would flip FAILURE to WARNING:
- Mortar's agent instructions (`mortar.md`) contained a section instructing it to
  read `optimized_prompts/{agent_name}.json` and prepend the `instructions` field.
  (This would be LLM-honor-based, not code-enforced, but would be non-zero effect.)

Evidence that would flip FAILURE to HEALTHY:
- A hook (PreToolUse or SubagentStart) that intercepts agent spawns and injects
  the `instructions` field from the corresponding JSON file.
- Or: `bl/runners/agent.py` loading `optimized_prompts/{agent}.json` and prepending
  it to the system prompt passed to `claude -p`.

Neither exists.

### Assessment

The primary finding states: "The Mortar agent instructions define a 'Constructing the Context Block' section... There is no instruction to read `optimized_prompts/{agent}.json` or inject an `instructions` field. The injection described in CLAUDE.md is not implemented in Mortar's own instructions." This is factually incorrect. `mortar.md` line 277 contains exactly such a directive: "If `masonry/optimized_prompts/{agent_name}.json` exists, read it and inject the optimized instructions into the specialist invocation." The primary finding's own threshold analysis states this evidence would flip the verdict from FAILURE to WARNING ("Mortar's agent instructions contained a section instructing it to read `optimized_prompts/{agent_name}.json` and prepend the `instructions` field... This would be LLM-honor-based, not code-enforced, but would be non-zero effect"). Since that condition is now confirmed present, the correct verdict is WARNING, not FAILURE. The code-level injection path (hooks, runner, router) is correctly identified as absent, but the LLM-honor-based path in mortar.md exists and was missed.
