## Summary [≤200 chars]

All 191 karen training records use an ops-domain schema (commit_subject/doc_files_written), not ResearchAgentSig fields. Optimization via ResearchAgentSig would produce a silent 0-example trainset.

## Architecture Check

### What karen actually produces

Karen's output contract is defined in `karen.md` (Output Contract section, lines 211–218). It returns:

```json
{
  "verdict": "DOCS_UPDATED | AUDIT_COMPLETE | SUMMARY_COMPLETE | DOCS_INIT_COMPLETE",
  "task": "init-docs | update-changelog | audit-folder | summarize-progress",
  "files_modified": ["path/to/file1.md"],
  "summary": "one-line description (max 120 chars)"
}
```

These are **documentation-management verdicts**, not research-finding verdicts. The `verdict` values (`DOCS_UPDATED`, `AUDIT_COMPLETE`, etc.) are a closed enum for ops tasks — they do not overlap with `ResearchAgentSig`'s expected values (`HEALTHY`, `WARNING`, `FAILURE`, `INCONCLUSIVE`).

### What the 191 training records actually contain

Verified by sampling `masonry/training_data/scored_all.jsonl` (all records where `agent == "karen"`):

- Source: `git_log` (ops scorer path, not the findings extractor path)
- Input fields present: `commit_subject`, `files_modified`
- Output fields present: `doc_files_written`, `reverted`
- Records with `verdict` (top-level or nested): **0 / 191**
- Records with `severity`: **0 / 191**
- Records with `evidence`: **0 / 191**
- Records with `confidence`: **0 / 191**

No karen record in `scored_all.jsonl` has any `ResearchAgentSig` output field populated.

### run_optimization.py field mapping (lines 47–58)

`load_training_data_from_scored_all()` reads karen records and maps:
- `inp.get("question_text", inp.get("question_id", ""))` — both keys are absent in karen's `input` dict (which only has `commit_subject` and `files_modified`). Result: **empty string**.
- `out.get("verdict", "")` — karen's `output` dict has no `verdict` key (only `doc_files_written` and `reverted`). Result: **empty string**.
- `out.get("evidence", "")` — absent. Result: **empty string**.
- `out.get("confidence")` — absent. Result: **0.75 (hardcoded fallback)**.

All 191 karen examples would be loaded with empty `question_text` and empty `verdict`. The MIPROv2 metric (`build_metric` in `optimizer.py`, lines 34–68) scores on `verdict_match` (0.4 weight) and `evidence_quality` (0.4 weight). With both empty, every example scores 0.0–0.2 (only the confidence calibration partial credit). The optimizer would converge on a degenerate prompt that produces no meaningful verdict or evidence output.

### generated/ stub status

`masonry/src/dspy_pipeline/generated/` contains only `economizer.py`. There is no `karen.py` signature stub. This is consistent with karen having been flagged as `tier: candidate` in `agent_registry.yml` but never having had an optimization run attempted.

## Training Data Quality Assessment

The 191 karen records are high-quality for their actual domain — they are git-log ops records scoring 100 points (60 operation_succeeded + 40 human_accepted). However, this quality is irrelevant to `ResearchAgentSig` compatibility:

- All 191 records would produce empty `verdict` and empty `evidence` when passed through `load_training_data_from_scored_all()`.
- The optimizer metric would see 191 examples all scoring near 0.0.
- This does not constitute a viable trainset for `ResearchAgentSig` — it would optimize toward empty outputs.

To constitute a viable trainset for a `KarenSig`, the 191 records are potentially usable because they have well-structured `commit_subject` → `doc_files_written`/`reverted` mappings. But this requires defining `KarenSig` first.

## Recommendation

**Do not attempt MIPROv2 optimization of karen via `ResearchAgentSig`.** The schema mismatch is structural, not cosmetic.

The following work is required before karen can be optimized:

1. **Define `KarenSig` in `masonry/src/dspy_pipeline/signatures.py`** with inputs matching karen's actual invocation contract (`task`, `project_context`) and outputs matching karen's actual output contract (`ops_verdict`, `files_written_count`, `summary`). The ops_verdict enum should cover `DOCS_UPDATED | AUDIT_COMPLETE | SUMMARY_COMPLETE | DOCS_INIT_COMPLETE`.

2. **Add a `KarenSig`-specific metric** in `optimizer.py` or a new `build_karen_metric()` function that scores on: ops_verdict match (0.5 weight), summary quality/length (0.3 weight), files_written_count reasonableness (0.2 weight).

3. **Add a `load_karen_training_data()` path** in `run_optimization.py` that reads karen's `input.commit_subject` and `output.doc_files_written`/`output.reverted` fields — or refactor `load_training_data_from_scored_all()` to be schema-aware.

4. **Generate `masonry/src/dspy_pipeline/generated/karen.py`** via `onboard_agent.py` using the `KarenSig` definition.

5. **Route karen to `KarenSig`** in `optimize_all()` (currently hardcoded to `ResearchAgentSig` for all agents, `optimizer.py` line 232).

The 191 training records are a genuine asset — they become usable once the correct signature is defined. Karen's corpus is the second largest in the fleet after quantitative-analyst (125), making it a high-priority optimization candidate once the schema work is complete.

**Viable next candidate for MIPROv2 via ResearchAgentSig**: the next agent after quantitative-analyst that produces research-finding outputs (verdict/severity/evidence fields) should be identified from `scored_all.jsonl` — karen does not qualify.
