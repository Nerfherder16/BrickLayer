# Finding: V1.4 — LLM Router Does Not Validate `target_agent` Against Registry; Hallucinated Names Propagate Unchecked

**Question ID**: V1.4
**Status**: DONE
**Verdict**: FAILURE
**Agent**: design-reviewer
**Severity**: Medium
**Confidence**: 1.0

## Summary

`llm_router.py` extracts `target_agent` from the LLM's JSON response and passes it directly to `RoutingDecision` without checking whether the name exists in the registry. Claude Haiku may hallucinate an agent name not in the registry. The invalid name propagates to the caller (`router.py`), which also performs no validation. The caller dispatches on a `RoutingDecision.target_agent` that refers to a non-existent agent, causing a downstream dispatch failure.

## Evidence

### The Validation Gap (llm_router.py lines 93-104)

```python
target = parsed.get("target_agent")
if not target:
    print("[llm_router] No target_agent in LLM response.", file=sys.stderr)
    return None

reason = parsed.get("reason", "LLM routing")[:100]

return RoutingDecision(
    target_agent=target,    # ← no registry membership check
    layer="llm",
    confidence=_LLM_CONFIDENCE,
    reason=reason,
)
```

The only check is `if not target` (None or empty string). A non-empty string that doesn't correspond to any registry agent passes through.

### Caller (`router.py`) Also Does Not Validate

```python
decision = route_llm(request_text, registry)
if decision is not None:
    print(
        f"[ROUTER] Layer llm resolved: {decision.target_agent} "
        f"(confidence {decision.confidence})",
        file=sys.stderr,
    )
    return decision
```

`router.py` logs and returns the decision without checking `decision.target_agent` against `registry`.

### Hallucination Scenarios

The LLM router builds the system prompt with:
```python
agent_list = "\n".join(
    f"- {a.name}: {a.description or ', '.join(a.capabilities)}"
    for a in registry
)
```

Claude Haiku is instructed to "select the best agent" from this list. However:
1. LLMs regularly produce agent names that resemble but don't exactly match the listed names (e.g., `"research-analyst"` vs `"research_analyst"` — hyphen vs underscore)
2. For small registries (<5 agents), Haiku may hallucinate a category name that wasn't listed
3. If `agent_list` is empty (D1.3 empty registry), Haiku receives no valid options and may produce any response

### Impact of Unvalidated Dispatch

What happens when the caller receives `RoutingDecision(target_agent="non-existent-agent", layer="llm", confidence=0.6)`?

This depends on the MCP tool layer that calls `route()`. If the MCP tool tries to spawn an agent by name from the returned `target_agent`, it would fail with "agent not found" or silently skip. The `RoutingDecision` provides no "check me before acting" signal.

### RoutingDecision Schema Cannot Enforce Validity

`RoutingDecision.target_agent: str` has no validator that checks against a runtime registry. Pydantic validates the type (`str`) but not semantic validity. Registry membership is a runtime constraint that cannot be expressed in the schema without passing the registry to the validator.

### Recommended Fix

Add registry membership validation in `route_llm` before returning:

```python
# In route_llm, after extracting target:
if target not in {a.name for a in registry}:
    print(
        f"[llm_router] LLM returned unknown agent '{target}'. "
        f"Valid agents: {[a.name for a in registry]}",
        file=sys.stderr,
    )
    return None  # fall through to Layer 4

return RoutingDecision(...)
```

This converts an invalid decision into a Layer 4 fallback, which is safer than propagating an invalid agent name.

Alternatively, a fuzzy match (e.g., difflib.get_close_matches) could recover from minor name variations:
```python
names = [a.name for a in registry]
close = difflib.get_close_matches(target, names, n=1, cutoff=0.8)
if close:
    target = close[0]  # correct the hallucinated name
```
