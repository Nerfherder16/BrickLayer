# Finding: V6.1 — `_tool_masonry_route` Error Fallback Is Not Schema-Compatible with `RoutingDecision`

**Question ID**: V6.1
**Status**: DONE
**Verdict**: FAILURE
**Agent**: design-reviewer
**Severity**: Medium
**Confidence**: 0.95

## Summary

`_tool_masonry_route()` in `mcp_server/server.py` returns a plain dict on import failure that is structurally incompatible with the dict returned by the success path (`decision.model_dump()`). The error fallback has three mismatches against `RoutingDecision`:

1. `"layer": 4` (int) — schema expects `Literal["deterministic", "semantic", "llm", "fallback"]` (str)
2. `"reason"` is absent — required field (`str`, no default) in `RoutingDecision`
3. `"error"` is an extra field — rejected by `ConfigDict(extra="forbid")` if deserialized

A caller that accesses `result["reason"]` after an error path receives a `KeyError`. A caller that checks `result["layer"] == "fallback"` receives `False` (int 4 ≠ str "fallback"). Any caller attempting `RoutingDecision(**result_dict)` gets a `ValidationError` on all three issues simultaneously.

## Evidence

### `_tool_masonry_route()` — success vs. error return shapes

```python

decision = route(request_text, project_dir)
return decision.model_dump()
# Returns:

#   "fallback_reason": "ambiguous"

# Error path (line 302):

# Returns:
# {
#   "error": "No module named 'masonry'",  # EXTRA (extra="forbid")
#   "target_agent": "user",

#   "confidence": 0.0,
#   # "reason" MISSING — required field
#   # "fallback_agents" MISSING — has default [], OK
#   # "fallback_reason" MISSING — Optional[...] = None, OK
```

### `RoutingDecision` schema — key fields

```python
class RoutingDecision(BaseModel):
    model_config = ConfigDict(extra="forbid")

    target_agent: str
    layer: Literal["deterministic", "semantic", "llm", "fallback"]  # STR, not int
    confidence: float = Field(ge=0.0, le=1.0)
    reason: str = Field(max_length=100)                              # REQUIRED, no default
    fallback_agents: list[str] = Field(default_factory=list)
    fallback_reason: Optional[Literal[
        "ambiguous", "ollama_timeout", "llm_timeout", "registry_empty", "multi_failure"
    ]] = None
```

Source: `src/schemas/payloads.py:127–139`

### Correct fallback construction for reference

The `_make_fallback()` function in `router.py:18–26` shows the correct pattern:

```python
def _make_fallback(fallback_reason: str = "ambiguous") -> RoutingDecision:
    return RoutingDecision(
        target_agent="user",
        layer="fallback",                                         # str
        confidence=0.0,
        reason="Ambiguous request -- asking user for clarification",  # present
        fallback_reason=fallback_reason,
    )
```

The error fallback in `server.py` was written by hand and diverged from this pattern — using integer `4` (the Layer 4 ordinal) instead of the string `"fallback"`, and omitting `reason`.

### Mismatch table

| Field | Success path | Error path | Schema | Compat? |
|---|---|---|---|---|
| `target_agent` | `"user"` | `"user"` | `str` | ✓ |
| `layer` | `"fallback"` | `4` (int) | `Literal["deterministic","semantic","llm","fallback"]` | ✗ WRONG TYPE |
| `confidence` | `0.0` | `0.0` | `float` | ✓ |
| `reason` | `"Ambiguous..."` | absent | `str` (required) | ✗ MISSING |
| `fallback_agents` | `[]` | absent | `list[str]` (default=[]) | ~ OK (missing but defaultable) |
| `fallback_reason` | `"ambiguous"` | absent | `Optional[Literal[...]] = None` | ✓ OK (None default) |
| `error` | absent | `"No module..."` | not in schema | ✗ EXTRA FIELD |

### Trigger condition

This error path fires when `from masonry.src.routing.router import route` fails — i.e., when the `masonry` package is not importable (sys.path issue, missing dependency, import-time error in `router.py`). This is not the normal runtime path but is the most likely failure mode during:

- Initial deployment before package is installed
- CI environments without `masonry` on sys.path
- When a dependency of `router.py` (e.g., `yaml`, `numpy` for DSPy embeddings) is missing

### Downstream impact

| Caller behavior | Result |
|---|---|
| `result.get("layer")` check for `"fallback"` | Returns `4` — string comparison fails |
| `result["reason"]` | `KeyError` — crashes caller |
| `RoutingDecision(**result)` | `ValidationError`: wrong layer type, missing reason, extra field |
| Kiln "layer" display | Shows `4` instead of `"fallback"` — visual mismatch |

## Fix Specification

Replace the error fallback dict with a proper `RoutingDecision`-shaped response:

```python
except Exception as exc:
    return {
        "error": str(exc),
        "target_agent": "user",
        "layer": "fallback",    # str, not int
        "confidence": 0.0,
        "reason": f"Router import failed: {str(exc)[:80]}",  # required field, max 100 chars
        "fallback_agents": [],
        "fallback_reason": "multi_failure",
    }
```

This matches the shape of `decision.model_dump()` exactly. The `error` key is extra and would still fail `RoutingDecision(**result)`, but since the MCP protocol passes it as raw JSON and callers read the dict directly (not via `RoutingDecision` deserialization), the `error` key is acceptable for diagnostic purposes. If strict schema compliance is required, omit the `error` key and surface the error in `reason`.

## Verdict

**FAILURE** — The `_tool_masonry_route` error fallback dict is not schema-compatible with `RoutingDecision.model_dump()`. Three structural mismatches: `layer` is `4` (int) instead of `"fallback"` (str), `reason` is absent (required field), and `error` is an extra field. Any caller reading `result["reason"]` after an import failure gets a `KeyError`. The `layer` int causes silent logic failures in callers that check `result["layer"] == "fallback"`.

**Severity**: Medium — only triggered when `masonry` package is unimportable, but that failure mode is precisely when callers most need a clean, readable error response.
