# Finding: R15.2 — `masonry-subagent-tracker.js` Lacks DISABLE_OMC; CWD Guard Provides Implicit Suppression

**Question ID**: R15.2
**Status**: DONE
**Verdict**: HEALTHY
**Severity**: Low
**Confidence**: 0.94

## Summary

`masonry-subagent-tracker.js` does NOT contain the DISABLE_OMC=1 kill switch. The commit 7b4472f that added the kill switch to "all Masonry hooks" covered exactly 7 specific hooks (masonry-build-guard, masonry-context-safety, masonry-lint-check, masonry-register, masonry-session-start, masonry-stop-guard, masonry-tdd-enforcer) — all hooks that actively interfered with BL subprocess operation. masonry-subagent-tracker.js was excluded because its routing_log.jsonl write is already gated by `if (fs.existsSync(path.join(cwd, 'masonry')))`, which implicitly suppresses the hook in BL subprocess contexts. The "start" event format is correct and matches score_routing.py's expected schema.

### Why commit 7b4472f excluded masonry-subagent-tracker.js

The commit message states:
> "The kill switch was documented in CLAUDE.md for BrickLayer subprocess mode but never implemented in the hooks themselves. This caused: masonry-register.js to inject Mortar routing into every BL research loop prompt... masonry-stop-guard.js to block the subprocess from stopping cleanly..."

The 7 hooks were selected because they **actively interfered** with BL subprocess operation:
1. Route injection (masonry-register.js) — hijacked agent loop prompts
2. Stop blocking (masonry-stop-guard.js, masonry-build-guard.js) — prevented clean exits
3. Context safety gate (masonry-context-safety.js) — blocked research loop
4. Lint/TDD checks (masonry-lint-check.js, masonry-tdd-enforcer.js) — unnecessary in BL mode

masonry-subagent-tracker.js is **passive** — it only tracks agents and writes routing data. In BL subprocess mode:
- Its routing_log.jsonl write is gated by `if (fs.existsSync(path.join(cwd, 'masonry')))`
- BL subprocess CWD = `{project}/` (e.g., `adbp/`) where `masonry/` is not a subdirectory
- Gate → false → routing_log.jsonl write is skipped → no interference

### "start" event format matches score_routing.py schema

**masonry-subagent-tracker.js** writes:
```json
{
  "timestamp": "ISO-8601",
  "event": "start",
  "agent": "<agent_name>",
  "session_id": "<session_id_or_generated_id>",
  "parent_session": "<input.session_id>"
}
```

**score_routing.py** expects (from `_match_events`):
```python
session_id = ev.get("session_id", "")    # used for pairing
parent_session = ev.get("parent_session", "")   # used for trowel attribution
agent = ev.get("agent", "unknown")        # used for correct_agent_dispatched
timestamp = ev.get("timestamp", "")      # used for time-window check
```

All 4 fields present. Format matches.

### agent_name extraction

```javascript
name: input.agent_name || input.agent_type || "agent",
```

SubagentStart hook receives `agent_name` from Claude Code. For named agent invocations (e.g., `subagent_type: "research-analyst"`), `input.agent_name = "research-analyst"`. This correctly maps to `AGENT_CATEGORIES["research-analyst"] = "findings"` → `correct_agent_dispatched=70pts`.

For anonymous agent invocations (`subagent_type: "general-purpose"`), `input.agent_name = "general-purpose"`. This is NOT in `AGENT_CATEGORIES` → `correct_pts=0`. This is correct behavior — general-purpose agents are not specialist dispatches.

### Session_id fallback risk

```javascript
session_id: agentEntry.sessionId || agentEntry.id,
```

If `input.session_id` is absent (shouldn't happen with Claude Code SubagentStart hooks), the fallback `agentEntry.id = input.agent_id || "agent-{timestamp}"` would produce a non-matching session_id. However: Claude Code always provides `session_id` in hook payloads. The 4 existing routing_log.jsonl "start" events all have `session_id = f68d4e9c...` (the parent session ID), confirming `input.session_id` is populated.

## Verdict

**HEALTHY** — masonry-subagent-tracker.js correctly excludes DISABLE_OMC because its CWD guard provides equivalent subprocess suppression passively. The "start" event format matches score_routing.py's schema. Agent name extraction and session_id pairing are correct. The only known gap (from R15.1) is that both hooks silently skip when CWD doesn't have a `masonry/` subdirectory — which correctly suppresses in BL subprocess mode but also suppresses in masonry self-research sessions (unintended side effect of the CWD guard design).
