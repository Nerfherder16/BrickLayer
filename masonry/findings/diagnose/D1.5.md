# Finding: D1.5 — Stop Hook Ordering Creates Block-Reason Masking

**Question ID**: D1.5
**Status**: DONE
**Verdict**: FAILURE
**Agent**: diagnose-analyst
**Severity**: Medium
**Confidence**: 0.8

## Summary

Both masonry-stop-guard and masonry-build-guard are synchronous Stop hooks registered in the same `hooks` array in global `settings.json`. Claude Code processes synchronous hooks in the order they are listed; when the first hook exits with code 2 (block), the stop is blocked and the user sees only that hook's message. The second hook's block reason is never shown. masonry-stop-guard (uncommitted files) comes before masonry-build-guard (pending tasks) — meaning a user with both conditions active only sees the git message, never the build status.

## Evidence

### Global settings.json Stop Hook Order (lines 156-196)

The global `settings.json` Stop configuration registers hooks in a single array in this order:
1. `recall/session-save.js` (sync, 10s) — Recall session persistence
2. `recall/recall-session-summary.js` (sync, 10s) — summary write
3. `masonry-stop-guard.js` (sync, 10s) — git uncommitted files check
4. `masonry-session-summary.js` (sync, 8s) — session summary
5. `masonry-handoff.js` (async, 8s) — handoff note (async, won't block)
6. `masonry-build-guard.js` (sync, 10s) — pending autopilot tasks check
7. `masonry-ui-compose-guard.js` (sync, 10s) — pending UI compose tasks check

### Plugin hooks/hooks.json Stop Hook Order

The plugin (`hooks/hooks.json`) registers:
1. `masonry-stop-guard.js` (sync, 10s)
2. `masonry-build-guard.js` (sync, 10s)
3. `masonry-ui-compose-guard.js` (sync, 10s)

Both sources list masonry-stop-guard before masonry-build-guard.

### Claude Code Hook Short-Circuit Behavior

Based on Claude Code's documented hook architecture: when a synchronous hook exits with code 2, the Stop operation is blocked and the process halts with the hook's stderr output displayed. Subsequent hooks in the sequence are not executed because the stop event has already been rejected.

**Scenario where masking occurs:**
- User has both uncommitted files AND a pending build in `.autopilot/`
- masonry-stop-guard fires first, exits 2 with: `"Stop blocked — 3 uncommitted session files..."`
- masonry-build-guard never runs
- User commits files, tries to stop again
- Now masonry-build-guard runs and shows the autopilot warning

The user must make two stop attempts to see both block reasons. In the worst case, they might not know the build is pending.

### Stop Hook Registration Duplication

Similar to D1.1, both the plugin (`hooks/hooks.json`) and global `settings.json` register masonry-stop-guard, masonry-build-guard, and masonry-ui-compose-guard for the Stop event. For sessions with the plugin active:

- Plugin registers: stop-guard, build-guard, ui-compose-guard (no matcher — always fires)
- Global registers: stop-guard, build-guard, ui-compose-guard (no matcher — always fires)

This means on Stop, each of these three hooks fires **twice**. Two separate git status checks, two separate autopilot dir checks. This is redundant but not harmful (idempotent checks) — unlike D1.1 where double-writes cause state corruption.

### Exit Code Behavior Verification

masonry-stop-guard.js line 166: `process.exit(2)` when uncommitted files found.
masonry-build-guard.js line 86: `process.exit(2)` when pending tasks found.

Both use exit code 2 per Claude Code hook contract for blocking Stop.

## Impact Assessment

The masking is a UX issue rather than a data integrity bug. The system is not unsafe — both conditions will eventually be caught on subsequent stop attempts. However, in an automated context (e.g., CI/CD), a masked block reason could cause confusion about why the stop was rejected.

## Mitigation

1. **Reorder hooks**: Move masonry-build-guard before masonry-stop-guard so the more actionable block (pending build tasks) appears first.
2. **Combined guard**: Merge stop-guard and build-guard into a single hook that checks all conditions and reports all block reasons in a single message.
3. **Eliminate Stop hook duplication**: Remove Stop hooks from plugin `hooks/hooks.json` since they're already in global settings.json — or vice versa.
