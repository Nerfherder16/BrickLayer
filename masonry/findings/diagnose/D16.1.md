# Finding: D16.1 — 99.5% of Below-Threshold Findings Fail on `confidence_calibration`: Missing `**Confidence**:` Field Is the Primary Gate

**Confidence**: 0.97
**Agent**: diagnose-analyst

## Summary

Of 627 below-threshold findings (score < 60), 624 (99.5%) score 0 on `confidence_calibration`, 572 (91.2%) score 0 on `evidence_quality`, and only 74 (11.8%) score 0 on `verdict_clarity`. The root cause is BL1.x format: findings from `recall/`, `recall-arch-frontier/` (494 of 627 below-threshold) lack the `**Confidence**:` field introduced in BL2.0. Without confidence → `confidence_calibration = 0` (max 40 pts), capping max possible score at 40+20=60. Since these findings also typically lack a `## Evidence` section (evidence_quality = 0), the effective score ceiling is 20 (verdict_clarity only). Score distribution confirms: 208 score 10-19 (verdict only), 342 score 20-29 (verdict + minimal evidence).

### Why confidence_calibration = 0 for BL1.x findings

`_score_confidence_calibration()` returns immediately if `confidence is None`:
```python
if confidence is None:
    return 0   # ← 0 of possible 40 points
```

BL1.x findings (recall, recall-arch-frontier) were written before the `**Confidence**:` field was standardized in BL2.0. They have `**Status**:`, `**Verdict**:`, `**Severity**:` but NOT `**Confidence**:` → `_RE_CONFIDENCE` match fails → confidence=None → 0 pts.

### Max score without confidence field

```
confidence_calibration: 0   (confidence is None → 0/40)
evidence_quality:       40  (max: evidence present + 50+ chars + 2+ numbers)
verdict_clarity:        20  (max: valid verdict present)
Total max:              60
```

A BL1.x finding CAN reach the 60-point threshold, but ONLY with a full evidence section (647+ chars, ≥2 numbers) AND a valid BL2.0-format verdict. Findings with `evidence_len < 50` score at most 20 (verdict only) → permanently below threshold.

### The 61 above-threshold findings all have BL2.0 format

Of the 61 training-ready findings:
- All come from BL2.0 projects: adbp (99 findings scored, many pass), bricklayer-meta, bricklayer-v2, root-level BL2.0 findings
- All have `**Confidence**:` fields (BL2.0 standard)
- All have `## Evidence` sections with sufficient content

### evidence_quality failure analysis

Of 572 with evidence_quality=0:
- `evidence = ""` → `_extract_section()` found no `## Evidence` heading
- BL1.x format used `## Observations`, `## Results`, `## Analysis` — NOT `## Evidence`
- `_score_evidence_quality()` requires `len(evidence) >= 50` for +15; most BL1.x findings with evidence sections use non-standard headings and score 0

### verdict_clarity failures (74 findings)

These 74 findings have missing or non-standard verdicts:
- BL1.x FAILURE/HEALTHY verdicts in wrong format → `_RE_VERDICT` match fails
- `verdict = ""` → verdict_clarity = 0 (no points at all)
- These are the oldest findings (observer-tail-latency.md etc.)

## Root Cause

The BL2.0 training threshold is calibrated for BL2.0 format findings. BL1.x findings fail on two structural dimensions:
1. **No `**Confidence**:` field** (99.5% of failures) — confidence_calibration = 0/40
2. **Non-standard evidence heading** (`## Observations` not `## Evidence`) (91.2% of failures) — evidence_quality = 0/40

These are expected failures. The 614/675 (88%) rejection rate is by design — the scorer filters for BL2.0-quality training data only. BL1.x findings were from a different toolchain and are not appropriate training examples for BL2.0 agents.

## Verdict

**DIAGNOSIS_COMPLETE** — Missing `**Confidence**:` field is the primary gate: 99.5% of below-threshold findings score 0/40 on confidence_calibration. Secondary gate: non-standard evidence section headings in BL1.x format (91.2% score 0/40 on evidence_quality). Together these prevent BL1.x findings from reaching the 60-point threshold except in rare cases (2 numbers + 647+ char evidence section). This is expected behavior — the scorer correctly rejects BL1.x findings as training data for BL2.0-format agents. No fix is warranted; the high rejection rate is a signal of format quality, not a bug.
