# Finding: D24.2 — Routing training records lack ground-truth target_agent, making all 70pt scores circular

**Question ID**: D24.2
**Status**: DONE
**Verdict**: DIAGNOSIS_COMPLETE
**Severity**: High
**Confidence**: 0.95
**Agent**: diagnose-analyst
**Date**: 2026-03-22

## Summary

`score_routing.py` awards 70pts by checking if the dispatched agent is a known agent, not whether it was the *correct* agent. No ground-truth `target_agent` field exists anywhere in the pipeline.

## Root Cause

The scoring pipeline has a structural gap across three layers:

1. **Data source gap** (`masonry/src/hooks/masonry-subagent-tracker.js` lines 190-206): The hook that writes `routing_log.jsonl` only captures `event: "start"`, `agent` (who was dispatched), `session_id`, and `parent_session`. It does not capture the original user request text or any routing intent label. Without the request text, there is no basis for determining what agent *should* have been dispatched.

2. **Scoring logic gap** (`masonry/scripts/score_routing.py` line 112): The `correct_agent_dispatched` score is awarded based on:
   ```python
   if agent and agent != "unknown" and agent in AGENT_CATEGORIES:
       correct_pts = 70
   ```
   The variable `agent` is the normalized `dispatched_agent` -- the agent that *was* dispatched. This check means "is this a recognized agent name?" which is trivially true for all dispatched agents. It does not compare dispatched vs. intended.

3. **Output schema gap** (`masonry/scripts/score_routing.py` lines 199-213): The output record writes `dispatched_agent` and `agent: "mortar"` but never writes a `target_agent` field. There is no field in the output schema for ground-truth routing intent.

**Causal chain**: Hook captures only dispatch events (no request text) -> Scorer has no ground truth to compare against -> Scorer falls back to "is agent name valid?" as a proxy for correctness -> All 17 records get 70pts trivially -> Records provide zero training signal for routing optimization.

## Evidence

### 1. routing_log.jsonl entries (source data)

Each `start` event contains only:
```json
{"timestamp":"...","event":"start","agent":"karen","session_id":"f68d4e9c-...","parent_session":"f68d4e9c-..."}
```
No `request_text`, `question_text`, or `routing_intent` field. The hook at `masonry-subagent-tracker.js:190-206` constructs this object from `agentEntry.name` and `input.session_id` only.

### 2. score_routing.py scoring logic (line 112)

```python
if agent and agent != "unknown" and agent in AGENT_CATEGORIES:
    correct_pts = 70
else:
    correct_pts = 0
```
`agent` is derived from `start_event.get("agent", "unknown")` at line 107-108 -- this is the dispatched agent, not a ground-truth label.

### 3. scored_routing.jsonl output records

All 17 records follow this pattern:
```json
{"agent": "mortar", "dispatched_agent": "karen", "score": 70, "score_breakdown": {"correct_agent_dispatched": 70, "downstream_success": 0}, "input": {"agent_dispatched": "karen"}, "output": {"finding_verdict": null}}
```
No `target_agent` field exists. The `RoutingDecision` schema in `payloads.py:132` has `target_agent: str` but this is a runtime routing output -- it is never persisted to the routing log or propagated to the scoring pipeline.

### 4. Missing data path

The `RoutingDecision.target_agent` field (payloads.py:132) is populated at runtime by the four-layer router, but the `masonry-subagent-tracker.js` hook does not have access to the `RoutingDecision` object. The hook fires on `SubagentStart` events from Claude Code, which only provide `agent_name`/`agent_type`/`session_id` -- not the routing decision metadata.

## Fix Specification

This is a two-part fix. Part A adds the request context to the routing log so ground truth can eventually be derived. Part B changes the scorer to flag records without ground-truth labels rather than silently awarding points.

### Part A: Capture request context in routing log

- **File**: `masonry/src/hooks/masonry-subagent-tracker.js`
- **Line**: 190-196
- **Change**: Add `request_text` (from `input.prompt` or `input.task_prompt`, truncated to 500 chars) and `subagent_type` to the routing log entry. This provides the request text needed for retrospective labelling:
  ```javascript
  const routingEntry = JSON.stringify({
    timestamp: new Date().toISOString(),
    event: 'start',
    agent: agentEntry.name,
    session_id: agentEntry.sessionId || agentEntry.id,
    parent_session: input.session_id || '',
    request_text: (input.prompt || input.task_prompt || '').slice(0, 500),
    subagent_type: (input.subagent_type || '').trim(),
  });
  ```
- **Verification**: `node -e "const fs=require('fs'); const h=require('./masonry/src/hooks/masonry-subagent-tracker.js'); console.log('syntax OK')" 2>&1 || echo "Check manually: spawn an agent and verify routing_log.jsonl entries contain request_text field"`
- **Risk**: Low. Adds two optional fields to an append-only log. No existing consumers read `request_text` yet. Truncation to 500 chars prevents log bloat. Privacy consideration: request text may contain sensitive content -- acceptable for local training data.

### Part B: Replace circular scoring with ground-truth-aware logic

- **File**: `masonry/scripts/score_routing.py`
- **Line**: 112-115
- **Change**: Replace the "is agent recognized?" check with a `target_agent` comparison when available. When `target_agent` is absent (all existing records), emit the record with `correct_agent_dispatched: 0` and a `"needs_label": true` flag instead of silently awarding 70 points:
  ```python
  # If ground-truth target_agent is available (future: from labelled data),
  # compare dispatched vs target. Otherwise, mark as unlabelled.
  target = start_event.get("target_agent")
  if target and agent == target:
      correct_pts = 70
  elif target and agent != target:
      correct_pts = 0
  elif agent and agent != "unknown" and agent in AGENT_CATEGORIES:
      # No ground truth available -- award partial credit but flag for labelling
      correct_pts = 35  # half credit: agent is valid but correctness unverified
  else:
      correct_pts = 0
  ```
  Also add `"needs_label": target is None` to the output record at line 199.
- **Verification**: `python masonry/scripts/score_routing.py --base-dir C:/Users/trg16/Dev/Bricklayer2.0 && python -c "import json; recs=[json.loads(l) for l in open('masonry/training_data/scored_routing.jsonl')]; labelled=[r for r in recs if not r.get('needs_label',True)]; unlabelled=[r for r in recs if r.get('needs_label',True)]; print(f'Labelled: {len(labelled)}, Unlabelled: {len(unlabelled)}'); assert all(r['score']<=35 for r in unlabelled if r['score_breakdown']['downstream_success']==0), 'Unlabelled records should not get 70pts'; print('PASS')"`
- **Risk**: Medium. Changes the score values for all existing 17 records (70->35 for those without downstream success). Any consumer that filters on `score >= 65` will exclude these unlabelled records, which is the desired behavior -- they should not be used for training until labelled. Adjacent impact: `score_all_agents.py` aggregates from `scored_routing.jsonl` and may show lower routing scores in fleet view.

## Mitigation

Until the fix is applied:
- The 17 existing routing records provide no useful training signal for routing optimization
- Records with `downstream_success: 30` (score 100) are partially useful -- they confirm the dispatch led to a successful outcome, even without ground-truth intent
- Manual labelling of the 17 records' `target_agent` field would immediately make them usable
