# Finding: D2.2 — masonry-subagent-tracker Has Read-Modify-Write Race on agents.json; Double-Fire (D1.1) Ensures Race on Every Spawn

**Question ID**: D2.2
**Status**: DONE
**Verdict**: FAILURE
**Agent**: diagnose-analyst
**Severity**: Medium
**Confidence**: 1.0

## Summary

`masonry-subagent-tracker.js` writes to three shared files per invocation: `~/.masonry/state/agents.json` (global), `{cwd}/masonry/routing_log.jsonl`, and `{cwd}/masonry-state.json`. The `agents.json` and `masonry-state.json` writes use non-atomic read-modify-write (`readFileSync` → mutate → `writeFileSync`). Under concurrent SubagentStart events, the last writer wins, silently dropping earlier agents from the active list.

Critically, the D1.1 double-fire means masonry-subagent-tracker runs **twice per SubagentStart event** (registered in both plugin hooks.json and global settings.json). This converts the race from a theoretical multi-agent scenario into a guaranteed race on every single agent spawn — two tracker instances always race on the same agent entry.

`routing_log.jsonl` uses `fs.appendFileSync` (O_APPEND semantics), which is append-safe under concurrent writes. No data loss for the JSONL log.

## Evidence

### agents.json Write Pattern (Non-Atomic)

```javascript
// Line 63-80: Full read-modify-write cycle
const stateFile = path.join(stateDir, "agents.json");  // ~/.masonry/state/agents.json
let state = tryJSON(stateFile) || { active: [], history: [] };  // READ

// Mutate in memory:
state.active = (state.active || []).filter(a => now - (a.startedAt || 0) < STALE_MS);
state.active.push(agentEntry);                  // MODIFY (add new entry)
state.history.push({ ...agentEntry, event: "start" });
state.updatedAt = now;

safeWrite(stateFile, state);  // WRITE → fs.writeFileSync (truncate-then-write, non-atomic)
```

`safeWrite` (lines 39-41): `fs.writeFileSync(p, JSON.stringify(obj, null, 2), "utf8")` — truncates then writes. No rename-based atomicity. Same pattern as D1.2 guard counter bug.

### Race Triggered by D1.1 Double-Fire (Guaranteed on Every Spawn)

From F2.1: SubagentStart is registered in both plugin hooks.json (no matcher) and global settings.json (no matcher) → two instances fire per SubagentStart event.

**Race timeline for a single agent spawn:**

```
T=0ms   SubagentStart fires → Instance A starts, Instance B starts (parallel)
T=5ms   Instance A: tryJSON("agents.json") → { active: [], history: [] }
T=5ms   Instance B: tryJSON("agents.json") → { active: [], history: [] }  (same read)
T=8ms   Instance A: push agentEntry → { active: [entry], history: [entry] }
T=9ms   Instance B: push agentEntry → { active: [entry], history: [entry] }
T=10ms  Instance A: writeFileSync → writes { active: [entry×1], history: [entry×1] }
T=11ms  Instance B: writeFileSync → writes { active: [entry×1], history: [entry×1] }
         ↑ overwrites A's write, but result is same (both wrote the same entry)
```

For a single-agent spawn, the double-fire produces a **duplicate entry** in `history` if Instance B reads A's write, or identical state if both read before either writes. The active list ends up with the correct single entry in most cases (last-writer wins with same data).

However, under `/ultrawork` or parallel agent dispatch (multiple SubagentStart events in rapid succession):

```
T=0ms   Agent1 SubagentStart → Instance A1, A2
T=1ms   Agent2 SubagentStart → Instance B1, B2

T=5ms   A1 reads:  { active: [],       history: [] }
T=5ms   A2 reads:  { active: [],       history: [] }
T=5ms   B1 reads:  { active: [],       history: [] }
T=5ms   B2 reads:  { active: [],       history: [] }

T=10ms  A1 writes: { active: [agent1], history: [agent1] }
T=10ms  B1 writes: { active: [agent2], history: [agent2] }  ← OVERWRITES agent1
T=11ms  A2 writes: { active: [agent1], history: [agent1] }  ← OVERWRITES agent2
T=11ms  B2 writes: { active: [agent2], history: [agent2] }  ← OVERWRITES agent1

Final:  { active: [agent2], history: [agent2] }
```

Result: `agents.json` shows only one of the agents. The statusline reports wrong active agent count. History is incomplete.

### masonry-state.json Write Pattern (Same Race)

```javascript
// Lines 100-106
const masonryState = tryJSON(masonryStateFile);
if (masonryState) {
  masonryState.active_agent = agentEntry.name;           // READ-MODIFY
  masonryState.active_agent_count = state.active.length; // uses agents.json count
  safeWrite(masonryStateFile, masonryState);             // WRITE (non-atomic)
}
```

Same pattern. Additionally, `active_agent_count` is computed from `state.active.length` which is the JUST-MODIFIED local in-memory `state` object, not re-read from disk. If the agents.json write was overwritten by another instance, `masonry-state.json` shows a count that is inconsistent with what's actually on disk.

### routing_log.jsonl Write (SAFE)

```javascript
// Line 95
fs.appendFileSync(routingLogPath, routingEntry + '\n', 'utf8');
```

`appendFileSync` uses O_APPEND flag, which is atomic at the OS level for writes smaller than PIPE_BUF (~4KB on Linux, larger on Windows). Each JSONL entry is ~150 bytes, well within the atomic threshold. Concurrent appends will not corrupt each other — lines arrive in non-deterministic order but no data is lost.

**Verdict: routing_log.jsonl is SAFE under concurrency.**

### Severity Assessment

The race primarily affects the active agent count displayed in the statusline and masonry-state.json's `active_agent` field. These are monitoring/display fields, not routing or finding integrity. The practical impact:
- **Statusline shows wrong active agent count** (too low)
- **masonry-state.json.active_agent** reflects the last-write-wins agent, not all active agents
- **agents.json history is incomplete** (some agent start events not recorded)
- **DSPy training signal**: routing_log.jsonl is safe (append-safe), but masonry-state.json's `active_agent` field may be used by drift detection

The primary fix is also addressed by F2.1: removing the SubagentStart entry from the plugin hooks.json eliminates the double-fire, reducing concurrent writes from 2n to n (where n is actual parallel agent spawns). For sequential single-agent sessions, the race disappears entirely.

## Recommended Fix (Beyond F2.1)

For parallel agent spawns (`.ultrawork`), even with F2.1 applied, the race can occur when multiple agents start within the same milliseconds. The correct fix is to use a file-lock pattern:

```javascript
const lockFile = stateFile + '.lock';
// Use cross-platform lock: create lock file with O_EXCL flag
// If lock exists and is <100ms old, retry once with 50ms delay
// Use rename-based atomic write after acquiring lock
```

Or, simpler and sufficient for the monitoring use case: switch agents.json to append-only JSONL format (like routing_log.jsonl), and compute the active set at read time by filtering entries.
