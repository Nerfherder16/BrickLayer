# Finding: D3.1 — masonry-register.js Double-Fire: Duplicate Mortar Directives Injected Per Prompt; Duplicate Recall Queries on Session Start

**Question ID**: D3.1
**Status**: DONE
**Verdict**: FAILURE
**Agent**: diagnose-analyst
**Severity**: Medium
**Confidence**: 1.0

## Summary

`masonry-register.js` (UserPromptSubmit hook) unconditionally writes the Mortar routing directive to `stdout` on every invocation, before any first-call detection. Because the hook is registered in both the plugin `hooks/hooks.json` and global `~/.claude/settings.json`, two instances fire per UserPromptSubmit event. The result: **Claude receives two copies of the Mortar routing directive per user prompt**, injected into its context by the hook framework. On session start (first call), both instances also call Recall (`searchMemory`) — generating two duplicate network requests to the Recall API.

The session state file write (`masonry-{sessionId}.json`) is NOT a race condition — both instances write identical data (write-once, no read-modify-write cycle) — but the stdout contamination is a correctness defect in the context injection model.

## Evidence

### Hook Registration Duplication

From `hooks/hooks.json` (plugin registration):
```json
{
  "hooks": {
    "UserPromptSubmit": [
      { "hooks": [{ "type": "command", "command": "node .../masonry-register.js" }] }
    ]
  }
}
```

From `~/.claude/settings.json` (global registration) — confirmed via prior F2.1 analysis:
```json
{
  "hooks": {
    "UserPromptSubmit": [
      { "hooks": [{ "type": "command", "command": "node .../masonry-register.js" }] }
    ]
  }
}
```

Both registrations reference the same script. Claude Code fires both → two instances per UserPromptSubmit.

### Unconditional Stdout Write (Line 117)

```javascript
// masonry-register.js — runs BEFORE any first-call detection
const directive = buildMasonryDirective(project, sessionContext);
process.stdout.write(directive + '\n');  // ← fires on EVERY invocation
```

The routing directive injected into Claude's context is approximately:
```
[MASONRY] Route this prompt through Mortar. Project: masonry. Session: {id}.
Use the masonry_route MCP tool with the user's message as the query.
Active hooks: masonry-approver, masonry-observe, masonry-guard, ...
```

Under double-fire, Claude receives this directive **twice** for every user message. Claude Code concatenates all hook stdout as a prefix to the user prompt (or injects it as a system-level injection, depending on event type). The net effect: the routing instruction appears twice, potentially causing Mortar to attempt routing twice or confusing the parsing of the directive.

### First-Call Detection Logic (Line 80-115)

```javascript
const tempFile = path.join(os.tmpdir(), `masonry-${sessionId}.json`);

let sessionState = null;
if (fs.existsSync(tempFile)) {
  sessionState = JSON.parse(fs.readFileSync(tempFile, 'utf8'));
}

if (sessionState && sessionState.firstCall === false) {
  // Subsequent call: flush guard warnings, return early
  flushGuardWarnings(sessionId);
  return;
}

// First call: write session marker
fs.writeFileSync(tempFile, JSON.stringify({
  project,
  sessionId,
  firstCall: false
}), 'utf8');

// Also call Recall for hydration
await searchMemory(sessionId, project);
```

**Double-fire race on first call:**
1. Instance A: `fs.existsSync(tempFile)` → `false` (file doesn't exist yet)
2. Instance B: `fs.existsSync(tempFile)` → `false` (file still doesn't exist — A hasn't written yet)
3. Instance A: enters first-call branch, calls `searchMemory` (Recall query #1)
4. Instance B: enters first-call branch, calls `searchMemory` (Recall query #2 — duplicate)
5. Instance A: `writeFileSync(tempFile, ...)` — writes marker
6. Instance B: `writeFileSync(tempFile, ...)` — overwrites with identical data (no corruption)

**Data integrity verdict**: The `writeFileSync` is write-once (both instances write the same JSON object), so the final file state is correct regardless of write order. No data corruption.

**Correctness verdict**: Two Recall API calls are made for every session start. Two Mortar directives are written to stdout on every prompt.

### Subsequent-Call Behavior Under Double-Fire

After the first prompt (tempFile exists):
1. Instance A: `fs.existsSync(tempFile)` → `true` → reads `{ firstCall: false }` → enters "subsequent call" branch → flushes guard warnings → returns early (no stdout write)
2. Instance B: same path as A

Wait — Instance A returns early before writing to stdout. Instance B does the same. **But the unconditional `process.stdout.write(directive)` executes BEFORE the first-call check.**

Re-examining the code flow:

```javascript
// Line 60-70: Build directive (unconditional)
const directive = buildMasonryDirective(project, sessionContext);

// Line 117: Write directive to stdout (unconditional)
process.stdout.write(directive + '\n');

// Line 120+: First-call detection
const tempFile = path.join(os.tmpdir(), `masonry-${sessionId}.json`);
if (fs.existsSync(tempFile)) {
  sessionState = ...
}
if (sessionState && sessionState.firstCall === false) {
  flushGuardWarnings(sessionId);
  return;  // ← EARLY RETURN — but stdout already written
}
```

The early return prevents the Recall call and guard flush from running twice on subsequent calls, but the stdout directive is already written before the early-return check. **Both instances write the directive to stdout on every prompt — first call AND subsequent calls.**

## Fix

Two changes needed:

**Fix 1: Move stdout write after first-call check**
```javascript
const tempFile = path.join(os.tmpdir(), `masonry-${sessionId}.json`);
let sessionState = null;
if (fs.existsSync(tempFile)) {
  sessionState = JSON.parse(fs.readFileSync(tempFile, 'utf8'));
}

if (sessionState && sessionState.firstCall === false) {
  flushGuardWarnings(sessionId);
  return;  // ← exit before writing directive
}

// Only reaches here for first call — write directive once
const directive = buildMasonryDirective(project, sessionContext);
process.stdout.write(directive + '\n');
```

**Fix 2 (root cause)**: Apply F2.1 (empty plugin hooks.json). The double-fire would be eliminated entirely, making this fix unnecessary.

Fix 1 is a standalone correctness improvement. Fix 2 (F2.1) is the root-cause resolution.

## Relationship to F2.1

F2.1 (empty plugin hooks.json) eliminates the double-fire for all 10 hook registrations simultaneously. D3.1 is one of the behavioral consequences of that double-fire. After F2.1 is applied, this finding becomes moot.

Until F2.1 is applied: Fix 1 (move stdout write after first-call check) eliminates the duplicate directive injection with a 3-line refactor.
