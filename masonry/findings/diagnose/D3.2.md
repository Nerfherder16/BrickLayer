# Finding: D3.2 — masonry-session-start.js Double-Fire: Auto-Resume Directive Produces Two JSON Objects to Stdout; Session Snapshot Write Is Safe

**Question ID**: D3.2
**Status**: DONE
**Verdict**: WARNING
**Agent**: diagnose-analyst
**Severity**: Medium
**Confidence**: 1.0

## Summary

`masonry-session-start.js` fires twice per session open (registered in both plugin `hooks/hooks.json` and global `~/.claude/settings.json`). The hook has two output paths: a fast path for interrupted-build detection (lines 71-77) that writes JSON to stdout and exits, and a normal path that writes status text to stderr. Under double-fire:

1. **Interrupted build path** (HIGH concern): Both instances detect the interrupted build state and both call `process.stdout.write(JSON.stringify({hookSpecificOutput: {...}}))`. Two JSON objects written to stdout produce malformed output (the hook framework expects a single JSON value). The auto-resume directive will either fail silently (parse error) or corrupt Claude's context. This means the interrupted-build auto-resume feature is **silently broken** under the current double-fire configuration.

2. **Normal path** (LOW concern): Both instances push the same status lines to their respective local `lines` arrays, then both write to `process.stderr`. Claude sees duplicated status messages (doubled `[Masonry] Autopilot BUILD mode active`, `[BL] masonry: N pending`, etc.). Cosmetic issue only.

3. **Session snapshot** (SAFE): Both instances write to the same `masonry-snap-{sessionId}.json` file via `fs.writeFileSync`. Both compute the same JSON value (same `git status` output, same sessionId). Write-once pattern, no data corruption.

The hook is fully idempotent and safe in BL research projects — it exits at line 45 (`isResearchProject(process.cwd())`) before any output or writes.

## Evidence

### BL Research Project Early Exit (Line 45)

```javascript
async function main() {
  if (isResearchProject(process.cwd())) process.exit(0);  // ← silent in BL projects
```

`isResearchProject` checks for `program.md` + `questions.md` in cwd. In the masonry project directory (which has both files for the self-research campaign), the hook exits silently. Double-fire is invisible when running BL campaigns from the masonry directory.

**This means the double-fire side effects only manifest in normal dev sessions** (not during BL campaign runs). The findings in this document apply to sessions opened from non-BL project directories.

### Interrupted Build Detection — Double JSON Write (Lines 56-78)

```javascript
const autopilotMode = tryRead(path.join(cwd, ".autopilot", "mode"));
if (autopilotMode && ["build", "fix", "verify"].includes(autopilotMode)) {
  const progress = tryJSON(path.join(cwd, ".autopilot", "progress.json"));
  if (progress) {
    const pending = (progress.tasks || []).filter(t => t.status !== "DONE" && t.status !== "BLOCKED");
    const done = (progress.tasks || []).filter(t => t.status === "DONE").length;
    const total = (progress.tasks || []).length;

    if (pending.length > 0 && (autopilotMode === "build" || autopilotMode === "fix")) {
      // FAST PATH: Both instances reach here simultaneously
      const resumeMsg = [...].join("\n");
      process.stdout.write(JSON.stringify({         // ← Instance A writes JSON #1
        hookSpecificOutput: {
          hookEventName: "SessionStart",
          content: resumeMsg + "\n\n...",
        },
      }));
      process.exit(0);  // ← Instance A exits
      // Instance B writes JSON #2 concurrently or immediately after
    }
```

**Under double-fire, stdout receives:**
```
{"hookSpecificOutput":{"hookEventName":"SessionStart","content":"[Masonry] Interrupted build..."}}
{"hookSpecificOutput":{"hookEventName":"SessionStart","content":"[Masonry] Interrupted build..."}}
```

This is two concatenated JSON values — not valid JSON. The Claude Code hook framework reads the SessionStart hook output and attempts to parse it. Behavior depends on the parser:
- If `JSON.parse(stdout)`: throws SyntaxError → hookSpecificOutput is silently ignored → auto-resume doesn't trigger
- If a JSON-lines parser: reads both lines → may inject the resume directive twice

Either outcome is incorrect. The auto-resume feature is broken under the current double-fire configuration for any session opened with an in-progress build.

### Normal Path — Duplicate stderr Output (Lines 80-160)

For sessions without an interrupted build:
- Both instances traverse all state checks (autopilot, UI compose, campaign, BL project detection, subagent state)
- Both push identical strings to their local `lines` array
- Both call `process.stderr.write("\n" + lines.join("\n") + "\n\n")` — writing the same output twice

The stderr output appears in the terminal (not in Claude's context). The user sees doubled status messages at the terminal prompt when Claude Code opens a session. This is cosmetic but can be confusing when troubleshooting hook behavior.

### Session Snapshot — Safe Write (Lines 165-181)

```javascript
const sessionId = input.session_id || input.sessionId || null;
if (sessionId) {
  const status = execSync("git status --porcelain", { encoding: "utf8", timeout: 5000, cwd }).trim();
  const preExisting = status ? status.split("\n").filter(Boolean).map(l => l.slice(3).trim()) : [];
  const snapPath = path.join(os.tmpdir(), `masonry-snap-${sessionId}.json`);
  fs.writeFileSync(snapPath, JSON.stringify({ sessionId, cwd, preExisting }), "utf8");
}
```

Both instances:
- Receive the same `input.session_id` (same event payload)
- Run `git status --porcelain` — identical output at session open (no files are modified between hook instances)
- Compute `snapPath` to the same file
- Write identical JSON to the same file

**Not a race condition.** The final file content is correct regardless of write order. The stop-guard reads this snapshot correctly.

### Contrast With D3.1 (masonry-register.js)

| Property | D3.2 (session-start) | D3.1 (register) |
|----------|---------------------|-----------------|
| Fires on | SessionStart (once per session) | UserPromptSubmit (every prompt) |
| Critical failure mode | Interrupted-build auto-resume produces invalid JSON output | Duplicate Mortar directive per prompt |
| Data corruption | None | None |
| Session snapshot | Safe (write-once identical) | Safe (write-once identical) |
| Frequency | Once per session | Every user message |

The D3.1 failure (duplicate Mortar directives) is higher-frequency and directly affects routing on every prompt. D3.2's interrupted-build JSON corruption is lower-frequency but completely disables a user-facing feature.

## Severity Assessment

- **Interrupted build auto-resume broken**: Medium severity. Feature is disabled (silently) when a session opens with a pending build. User will not get the auto-resume injection and must manually invoke `/build`. This is a silent regression — the feature was designed to reduce friction but currently never works under double-fire.
- **Duplicate stderr output**: Low severity. Terminal cosmetic issue.
- **Session snapshot**: No issue.

Overall verdict: **WARNING** (not FAILURE because there is no data corruption and the workaround is a manual `/build` invocation — the system remains functional).

## Fix

**Root cause fix**: Apply F3.1 (empty plugin hooks.json). All double-fire eliminated. Both D3.1 and D3.2 become moot.

**Standalone fix for interrupted-build path**: Add a deduplication guard before the stdout write:
```javascript
// Use a temp file to ensure only one instance writes hookSpecificOutput per session
const resumeLock = path.join(os.tmpdir(), `masonry-resume-${sessionId}.lock`);
if (!fs.existsSync(resumeLock)) {
  fs.writeFileSync(resumeLock, '1', 'utf8');  // lightweight lock — write-once
  process.stdout.write(JSON.stringify({hookSpecificOutput: {...}}));
}
process.exit(0);
```

This is a 4-line addition. However, F3.1 (emptying hooks.json) is the correct fix and eliminates the need for this workaround.
