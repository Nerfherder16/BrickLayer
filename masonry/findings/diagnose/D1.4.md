# Finding: D1.4 — observe/guard Do Not Share State; Double-Fire Creates Same-Hook Race

**Question ID**: D1.4
**Status**: DONE
**Verdict**: FAILURE
**Agent**: diagnose-analyst
**Severity**: Medium
**Confidence**: 0.95

## Summary

masonry-observe and masonry-guard write to entirely different files — there is no cross-hook shared state race between them. However, the double-fire from D1.1 means two instances of *each* hook fire concurrently on Write/Edit operations. This creates a same-hook race on masonry-state.json (two observe instances) and the guard strike counter (two guard instances, per D1.2). The hypothesis about cross-hook conflict is NOT confirmed; the intra-hook race from D1.1 is the real issue.

## Evidence

### File I/O Comparison

**masonry-observe.js writes to:**
1. `os.tmpdir()/masonry-activity-{sessionId}.ndjson` — activity log (append)
2. Recall API (HTTP POST to external server)
3. `masonry-state.json` in CWD — via `writeState(cwd, {...})` (read-modify-write)

**masonry-guard.js writes to:**
1. `os.tmpdir()/masonry-guard-{sessionId}.json` — strike counter (direct write)
2. `os.tmpdir()/masonry-guard-{sessionId}.ndjson` — guard queue (append)

**No shared files.** The two hooks operate on completely disjoint file paths. A cross-hook race would require both hooks to read/write the same path — they do not.

### Same-Hook Race: masonry-observe × 2 on masonry-state.json

From D1.1: two instances of masonry-observe fire for every Write/Edit call. Both instances:
1. Read current state: `readState(cwd)` → `masonry-state.json`
2. Compute updated verdicts
3. Write updated state: `writeState(cwd, {...})`

The `writeState` call in masonry-observe.js:
```javascript
writeState(cwd, {
  last_qid: qid,
  last_verdict: verdict,
  ...(Object.keys(verdictUpdate).length ? { verdicts: verdictUpdate } : {}),
});
```

If two instances run concurrently:
- Both read `{ verdicts: { FAILURE: 2 } }`
- Both compute `verdicts: { FAILURE: 3 }` (incrementing by 1 each)
- Both write `{ verdicts: { FAILURE: 3 } }`
- Result: count is 3 instead of 4. One verdict increment is lost.

This causes the campaign verdict summary to undercount findings. Over a campaign with 20 findings, this could misreport the FAILURE count by up to 20 (one per finding write).

### Same-Hook Race: masonry-guard × 2 on Strike Counter

Covered in detail in D1.2. The race is deterministic for Write/Edit calls: the counter advances by 1 instead of 2 per double-fired event, meaning a 3-strike pattern requires 6 actual errors instead of 3.

### Why Cross-Hook Race Is Not a Concern

masonry-observe exits at line 231 (`process.exit(0)`) for non-findings files before it ever touches masonry-state.json. masonry-guard exits at line 73 (`process.exit(0)`) for non-error responses. Their respective exit paths don't touch shared state even when both fire on the same event.

The only theoretical interaction: if masonry-observe reads masonry-state.json at the same time another hook (not guard) writes it. But guard does not write masonry-state.json.

## Root Cause

The cross-hook race hypothesis is incorrect. The actual bugs are:
1. D1.1 double-fire causing intra-observe and intra-guard races
2. Non-atomic writes in both hooks (D1.2)

## Mitigation

Fix D1.1 (eliminate double-fire) to resolve both races. The `writeState` function should also use atomic write (tmp-rename) to protect against concurrent writes from any source.
