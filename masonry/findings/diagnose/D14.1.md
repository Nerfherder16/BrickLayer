# Finding: D14.1 — 241→64 Collapse Caused by Ops Dedup Key Collision (Commit Hash Not Used)

**Question ID**: D14.1
**Status**: DONE
**Verdict**: DIAGNOSIS_COMPLETE
**Agent**: diagnose-analyst
**Severity**: Medium
**Confidence**: 0.97

## Summary

`score_all_agents.py` processes 241 records across four scorers, but only 64 survive into `scored_all.jsonl`. The collapse is not caused by `min_training_score` filtering — all 241 records already passed their respective thresholds. The reduction is caused by `_dedup_records()` collapsing 173 ops records (git-nerd: 3, karen: 170) into exactly **2** records because all ops records share identical dedup keys. Ops records have `source="git_log"` but lack `question_id`, `session_id`, and `branch` fields, causing every git-nerd commit to map to `"src:git_log::git-nerd:100"` and every karen commit to `"src:git_log::karen:100"`. The last-write-wins rule leaves a single record per agent. Git-nerd and karen are effectively data-starved in `scored_all.jsonl` despite having 3 and 170 qualifying operations respectively.

### Ops record schema (from score_ops_agents.py)

```json
{
  "agent": "karen",
  "source": "git_log",
  "commit_hash": "7b4472f4a3bc",
  "score": 100,
  "score_breakdown": {...},
  "input": {"commit_subject": "...", "files_modified": [...]},
  "output": {"doc_files_written": 1, "reverted": false}

```

Ops records have:
- `question_id`: absent → qid=""
- `session_id`: absent → session=""
- `branch`: absent → branch=""
- `source`: "git_log"
- `score`: 100 (all ops records score 100 = 60+40 when not reverted)

Dedup key for every karen commit: `"src:git_log::karen:100"` — all 170 collide.
Dedup key for every git-nerd commit: `"src:git_log::git-nerd:100"` — all 3 collide.

Last entry wins: only 1 karen record + 1 git-nerd record survive.

### Why the summary table is misleading

`score_all_agents.py` reports ops via `so_summary.get("total_records", 0)` but findings/code/routing via `training_ready`. The `total_records` for ops = records written to scored_ops_agents.jsonl (all post-threshold). The TOTAL row (241) is a mixed count: three scorers report post-threshold, one reports post-threshold but uses a different key name. The table is not wrong, but it obscures that 173 of the 241 rows will collapse in dedup.

### Remaining 177 → 64 arithmetic

```
241 pre-dedup records
- 171 ops records eliminated (173 → 2 survivors)
= 70 expected
- 6 additional cross-scorer duplicates eliminated
= 64 scored_all.jsonl records
```

The 6 additional deduplicates are likely findings records appearing in both scored_findings.jsonl and another path, or routing records with the same session_id as a findings record.

## Fix Specification

**Root cause**: Ops records lack a uniqueness identifier in their dedup-eligible fields. `commit_hash` is present but not included in the dedup key formula.

**Fix**: In `_dedup_records()`, extend the else-branch to use `commit_hash` when present:

```python
# Proposed change to score_all_agents.py:138-143
if qid:
    key = f"qid:{qid}:{agent}"
elif session:
    key = f"session:{session}:{agent}"
else:
    commit_hash = rec.get("commit_hash", "")
    key = f"src:{source}:{commit_hash or branch}:{agent}:{rec.get('score', 0)}"
```

With `commit_hash` as the distinguishing factor, each git-nerd commit produces a unique key like `"src:git_log:7b4472f4a3bc:git-nerd:100"`. Karen's 170 commits would produce 170 unique keys → 170 survivors. Git-nerd's 3 commits → 3 survivors. Scored_all.jsonl would grow from 64 to ~235 records, with karen having 170 training examples instead of 1.

## Verdict

**DIAGNOSIS_COMPLETE** — The 241→64 collapse is caused by missing `commit_hash` in the dedup key for ops records. All 173 ops records are already above threshold; none are filtered on quality grounds. The dedup formula reduces 173 diverse training signals to 2 identical ones. Git-nerd and karen each appear exactly once in scored_all.jsonl despite being the most operationally active agents in the repository. This is a structural training data gap that would suppress DSPy optimization quality for ops agents if MIPROv2 were run today. The fix is a one-line change to the else-branch of `_dedup_records()`.
