## Summary

`sync_verdicts_to_agent_db.py` is now implemented and correctly populates the `verdicts` field in `agent_db.json` from campaign findings. The script uses `extract_training_data()` for attribution, groups verdicts by agent, and writes back atomically. A `^[a-z][a-z0-9-]*$` regex filter eliminates garbage agent names that arise from multi-project scan contamination. Live run writes 81 verdicts across 6 agents, unblocking `masonry_drift_check`.

## Root Cause

`agent_db.json` had empty `verdicts: []` arrays for all agents. `masonry_drift_check` uses these arrays to detect verdict distribution changes, but with no data it always returned a trivially clean result. The gap existed because no mechanism ever populated the field from campaign findings (D7.1).

## Changes Applied

### 1. Script: `masonry/scripts/sync_verdicts_to_agent_db.py`

New script with:

**`sync_verdicts()` function:**
```python
def sync_verdicts(
    projects_dir: Path,
    agent_db_path: Path,
    questions_md_path: Path | None = None,
) -> dict[str, int]:
```

- Calls `extract_training_data(projects_dir, questions_md_path=questions_md_path)`
- Groups `finding["verdict"]` by `finding["agent"]` into `verdicts_by_agent`
- Validates agent names with `^[a-z][a-z0-9-]*$` (blocks 'Check', 'Edit', 'For', 'Read', 'Run' from cross-project contamination)
- Cleans garbage keys from existing `agent_db` before writing
- Writes only agents that pass validation AND exist in `agent_db`
- Atomic write via `tmp.{pid} + Path.replace()`

**Garbage name filtering:**
```python
import re as _re
_valid_agent_name = _re.compile(r"^[a-z][a-z0-9-]*$")

# Remove any garbage keys written by previous runs
for k in [k for k in list(agent_db.keys()) if not _valid_agent_name.match(k)]:
    del agent_db[k]

# Write only valid known agents
for agent_name, verdicts in verdicts_by_agent.items():
    if _valid_agent_name.match(agent_name) and agent_name in agent_db:
        agent_db[agent_name]["verdicts"] = verdicts
        written[agent_name] = len(verdicts)
```

The garbage names ('Check', 'Edit', 'For', 'Read', 'Run') originated from other BL projects' `questions.md` files containing `**Method**: Run`, `**Agent**: Check`, etc. — these are verbs in question metadata, not agent names. The lowercase-start constraint blocks all of them since real agent names are always lowercase-hyphenated.

### 2. Invocation

```bash

# With explicit questions.md override
python -m masonry.scripts.sync_verdicts_to_agent_db \
    --questions-md masonry/questions.md
```
