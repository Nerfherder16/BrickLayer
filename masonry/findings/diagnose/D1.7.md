# Finding: D1.7 — masonry-agent-onboard Registry Race Is Real But Hook Is Inactive

**Question ID**: D1.7
**Status**: DONE
**Verdict**: FAILURE
**Agent**: diagnose-analyst
**Severity**: Low
**Confidence**: 0.95

## Summary

The `append_to_registry` function in `onboard_agent.py` uses an unprotected read-modify-write pattern on `agent_registry.yml` with no file locking. Concurrent invocations (e.g., two agent files written in rapid succession) would cause the second write to overwrite the first, silently losing one agent registration. However, masonry-agent-onboard.js is currently NOT in the active hook configuration — the hook exists but is registered nowhere that Claude Code loads. The race condition is a real implementation flaw in code that is not currently executing.

## Evidence

### The Race-Prone Pattern in `onboard_agent.py` (lines 239-257)

```python
def append_to_registry(entry, registry_path):
    data = {"version": 1, "agents": []}
    if registry_path.exists():
        try:
            raw = registry_path.read_text(encoding="utf-8")
            loaded = yaml.safe_load(raw) or {}
            ...
            data = loaded
        except Exception:
            pass

    entry_dict = entry.model_dump(exclude_none=True)
    data["agents"].append(entry_dict)        # ← append in memory
    registry_path.write_text(yaml.dump(data), encoding="utf-8")  # ← non-atomic write
```

The docstring even acknowledges the issue: _"Does NOT check for duplicates — use upsert_registry_entry for idempotent writes."_

**Race scenario for two concurrent processes (A and B):**
1. A reads registry: `agents: [existing1]`
2. B reads registry: `agents: [existing1]` (same snapshot)
3. A appends `new_agent_1` → writes: `agents: [existing1, new_agent_1]`
4. B appends `new_agent_2` → writes: `agents: [existing1, new_agent_2]` (overwrites A's write)
5. `new_agent_1` is lost from the registry without any error

### Why the Hook Is Not Active

The `masonry-agent-onboard.js` hook triggers the above Python script. Investigation of active configurations:

| Configuration File | masonry-agent-onboard registered? |
|---|---|
| `~/.claude/settings.json` (global) | **No** |
| `masonry/hooks/hooks.json` (active plugin config) | **No** |
| `masonry/hooks.json` (project root, NOT loaded) | **Yes** |
| `masonry/docs/hooks_config.json` (documentation) | **Yes** |

The plugin loads `hooks/hooks.json` (per `plugin.json`: `"hooks": "../hooks/hooks.json"`). That file contains only masonry-observe and masonry-guard for PostToolUse — NOT masonry-agent-onboard. The project-root `hooks.json` (which does list agent-onboard) is not referenced by any active configuration.

### The `upsert_registry_entry` Function Exists But Is Not Used by the Hook

The `upsert_registry_entry` function (lines 263-316) is idempotent and handles the "already exists" case correctly. The hook spawns `onboard_agent.py` which calls... let's check what it actually calls in the main() function.

The script's `main()` determines which function to call based on CLI args. When called by the hook with a single file path argument, it likely calls `upsert_registry_entry` (idempotent). But in the concurrent scenario where two detached processes start within milliseconds of each other, even `upsert_registry_entry` would have the same read-modify-write race.

### Risk Assessment When/If Hook Is Activated

If masonry-agent-onboard is added to the active hook configuration:
- Concurrent agent writes (e.g., batch `/plan` spawning multiple spec files) trigger concurrent detached Python processes
- The race window is the I/O time to read and write the YAML file (typically 10-50ms)
- On fast SSDs, this is unlikely but possible with 2+ concurrent writes
- Result: one or more new agents silently missing from the registry until next full scan

## Mitigation

1. **Use file locking**: `filelock` library or OS-level advisory locks before read-modify-write.
2. **Atomic write**: write to temp file, rename atomically (prevents partial-write corruption, but doesn't prevent read-modify-write race without locking).
3. **Activate the correct hook**: If agent onboarding is desired, add masonry-agent-onboard to `hooks/hooks.json`, but fix the race first.
4. **Current state is accidentally safe**: The hook being inactive means the race cannot currently manifest. No immediate action required, but the hook should not be activated without addressing the locking issue.
