# Finding: D24.1 — `_build_qid_to_agent_map()` Drops 914 of 1082 Training Records From Table-Format Projects

## Summary

`_build_qid_to_agent_map()` only parses `### QID:` block format. Table-format projects (adbp, bricklayer-v2, recall, etc.) return 0 mappings, dropping 914 of 1082 findings from agent attribution. 603 of those are recoverable via `**Agent**:` in the finding file itself.

## Root Cause

`_build_qid_to_agent_map()` (line 25-48 of `training_extractor.py`) splits on `\n---\n` and searches each block for `### (\w+\d+\.\d+):\s*(.+)`. This regex matches the block-header format used by `masonry/questions.md` but cannot match table rows like `| 1.1 | DONE | question text |`.

Separately, `extract_finding()` (line 68-116) already reads the finding markdown file but does **not** extract the `**Agent**:` field, even though 791 of ~1197 finding files contain it. This is a second missed opportunity for agent attribution.

The combination means:
- **Block-format projects** (masonry): 134/135 findings get agent attribution (99.3%)
- **Table-format projects** (adbp, bricklayer-v2, recall, etc.): 0 findings get agent attribution (0%)

In `build_dataset()`, `score_example()` returns 0.0 for findings without an agent, so all 914 un-attributed findings are **excluded from DSPy training entirely**.

## Fix Specification

The minimum fix is to add `**Agent**:` extraction directly in `extract_finding()`. This is simpler and more robust than adding table-format parsing to `_build_qid_to_agent_map()` because:
1. It recovers 603 records immediately with no questions.md format dependency
2. The `**Agent**:` field is the authoritative source (it's in the finding itself)
3. It works regardless of questions.md format

### Fix 1 (Primary): Add Agent extraction to `extract_finding()`

- **File**: `masonry/src/dspy_pipeline/training_extractor.py`
- **Line**: 22 (add regex) and 87 (add extraction)
- **Change**:
  1. Add `_AGENT_RE = re.compile(r"^\*\*Agent\*\*:\s*(\S+)", re.MULTILINE)` after line 22
  2. After line 87 (question_id extraction), add:
     ```python
     # Extract agent
     agent_m = _AGENT_RE.search(text)
     agent = agent_m.group(1).strip() if agent_m else ""
     ```
  3. In the return dict (line 108), add `"agent": agent,` field
- **Verification**: `cd C:/Users/trg16/Dev/Bricklayer2.0/masonry && python -c "from pathlib import Path; import sys; sys.path.insert(0,'src/dspy_pipeline'); from training_extractor import extract_finding; f=extract_finding(Path('C:/Users/trg16/Dev/Bricklayer2.0/adbp/findings/1.1.md')); print('PASS' if f and f.get('agent') else 'FAIL')"`
- **Risk**: Low. The `_add()` helper (line 149) already checks `not finding.get("agent")` before overwriting from qid_map, so this change is additive. The finding-file agent takes precedence, qid_map becomes the fallback. No existing tests should break since the field was previously absent.

### Fix 2 (Optional enhancement): Add table-format parsing to `_build_qid_to_agent_map()`

- **File**: `masonry/src/dspy_pipeline/training_extractor.py`
- **Line**: 39-48
- **Change**: After the block-format loop, add a fallback regex scan for table rows: `re.findall(r"^\|\s*(\S+)\s*\|.*?\|\s*(DONE|PENDING|IN_PROGRESS)\s*\|\s*(.+?)\s*\|", text, re.MULTILINE)`. However, table-format questions.md files do NOT contain an Agent column, so this would only recover question_text, not agent attribution. This fix alone is insufficient.
- **Risk**: Minimal, but provides less value than Fix 1.

## Mitigation

Apply Fix 1 (primary). This recovers 603 of the 914 dropped records (66%) with a 3-line change. The remaining 311 records lack `**Agent**:` in both the finding file and questions.md and cannot be automatically attributed without manual annotation.
