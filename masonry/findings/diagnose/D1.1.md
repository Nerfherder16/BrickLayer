# Finding: D1.1 — Hook Double-Fire on Write/Edit Operations

**Question ID**: D1.1
**Status**: DONE
**Verdict**: FAILURE
**Agent**: diagnose-analyst
**Severity**: High
**Confidence**: 0.95

## Summary

masonry-observe and masonry-guard double-fire on Write/Edit operations due to registration in both the plugin hooks (`hooks/hooks.json`) and global `settings.json`. masonry-agent-onboard is NOT in the active plugin configuration — the hypothesis about triple-firing was based on a misidentification of the active hooks.json.

## Evidence

### Active Hook Sources

Two independent hook sources load for the masonry project:

**Source 1 — Plugin (`plugin.json` → `hooks/hooks.json`)**:
- PostToolUse (no matcher, fires on ALL tools):
  - `masonry-observe` (async, 5s timeout)
  - `masonry-guard` (async, 3s timeout)

**Source 2 — Global `~/.claude/settings.json`**:
- PostToolUse (matcher: `Write|Edit`):
  - `masonry-observe` (async, 5s timeout)
  - `masonry-guard` (async, 3s timeout)
  - (plus masonry-lint-check, design-token-enforcer, context-monitor, tdd-enforcer)

**Result for Write/Edit tool calls**: BOTH sources match. masonry-observe fires 2×, masonry-guard fires 2×.

**Result for other tools (Bash, Read, Glob, etc.)**: Only plugin source fires. 1× each.

### Consequences of Double-Fire

1. **masonry-observe (2×)**: Two instances simultaneously:
   - Append duplicate entries to `masonry-activity-{sessionId}.ndjson`
   - Store duplicate findings to Recall API (double `storeMemory` calls per finding write)
   - Both call `writeState(cwd, ...)` to update `masonry-state.json` — concurrent non-atomic writes

2. **masonry-guard (2×)**: Two instances simultaneously:
   - Both read `masonry-guard-{sessionId}.json` (the strike counter)
   - Both increment `counts[fp]` based on the same read
   - Race condition: one write wins, the other is lost — effectively a missed strike increment

### masonry-agent-onboard Status

The hypothesis stated masonry-agent-onboard is registered in both `hooks.json` (project-scoped) and `settings.json`. This is incorrect:

- `masonry/hooks.json` at project root DOES list masonry-agent-onboard — but this file is **not loaded by Claude Code**. It is not referenced by any configuration; it is documentation/legacy.
- The active plugin hooks (`hooks/hooks.json`, as referenced by `plugin.json`) do **not** include masonry-agent-onboard.
- Global `settings.json` does **not** include masonry-agent-onboard.
- **masonry-agent-onboard is currently inactive** in all live configurations.

### Files Involved

- `masonry/.claude-plugin/plugin.json` → `"hooks": "../hooks/hooks.json"` (active plugin config)
- `masonry/hooks/hooks.json` — actual plugin hook registrations (observe + guard only)
- `masonry/hooks.json` — project root file, NOT loaded (documentation artifact)
- `~/.claude/settings.json` — global hooks (observe + guard with Write|Edit matcher)

## Mitigation

1. Remove masonry-observe and masonry-guard from the plugin's `hooks/hooks.json` — let global settings.json remain the single source of truth for those hooks.
2. OR: Remove them from global settings.json and add full coverage via plugin hooks (but then non-masonry projects lose observe/guard).
3. The double-fire for Write/Edit is the correct fix target. The plugin hooks firing on ALL tools (no matcher) may also be over-broad — masonry-observe exits immediately for non-watched tools, so this is a silent cost but not a correctness issue.
