# Finding: D1.2 — masonry-guard Strike Counter Vulnerable to Non-Atomic Write

**Question ID**: D1.2
**Status**: DONE
**Verdict**: FAILURE
**Agent**: diagnose-analyst
**Severity**: High
**Confidence**: 0.9

## Summary

masonry-guard uses `fs.writeFileSync` (non-atomic direct write) for the strike counter file. A timeout-killed partial write leaves an empty or corrupt JSON file; the read-side catch block resets the counter to `{}` (zero strikes), silently losing accumulated strike data. Additionally, the D1.1 double-fire bug causes two concurrent guard instances to race on the same counter file for every Write/Edit tool call.

## Evidence

### Non-Atomic Write (masonry-guard.js lines 108-110)

```javascript
// Persist updated counts
try {
  fs.writeFileSync(guardCountFile, JSON.stringify(counts), 'utf8');
} catch (_err) { /* non-fatal */ }
```

`fs.writeFileSync` is NOT an atomic rename-based write. The sequence is:
1. Open file (truncates to zero length)
2. Write data
3. Close file

If the process is killed between steps 1 and 2, the file is empty. If killed between 2 and 3, the file may be partially written (broken JSON).

### Read-Side Recovery Resets to Zero

```javascript
let counts = {};
try {
  if (fs.existsSync(guardCountFile)) {
    counts = JSON.parse(fs.readFileSync(guardCountFile, 'utf8'));
  }
} catch (_err) { /* fresh */ }  // ← resets to {} on ANY parse error
```

`JSON.parse('')` (empty file) throws `SyntaxError`. The catch block silently sets `counts = {}`. Result: **all accumulated strikes are lost**. The guard effectively resets, allowing the 3-strike pattern to restart from zero without the user ever being warned.

### 3-Second Timeout Exposure Window

The hook timeout is 3 seconds (`hooks/hooks.json` line 30). The guard script:
1. Reads stdin (async, up to MAX_STDIN=2MB) — ~10ms
2. Parses JSON, checks for errors — ~1ms
3. Reads counter file — ~1ms
4. Modifies counts in memory — <1ms
5. **Writes counter file** — ~1ms
6. Exits

The timeout kill window for the write is approximately the last 2ms of a 3-second budget. The probability per individual call is very low. However, the 3-second timeout is process-level — if stdin read is slow (large tool response), the write may never execute.

### More Significant: Concurrent Write Race (D1.1 Interaction)

The double-fire from D1.1 is a higher-probability corruption path. For every Write/Edit tool call:

1. Instance A reads counter: `counts = { "fp123": 2 }`
2. Instance B reads counter: `counts = { "fp123": 2 }` (same read, concurrent)
3. Instance A increments: `counts = { "fp123": 3 }` → detects threshold → resets to 0 → writes `{}`
4. Instance B increments: `counts = { "fp123": 3 }` → detects threshold → resets to 0 → writes `{}`

Both instances see the same count. A 3rd-strike detection happens twice (harmless). But for non-threshold increments:
1. Instance A reads: `{ "fp123": 1 }` → writes `{ "fp123": 2 }`
2. Instance B reads: `{ "fp123": 1 }` → writes `{ "fp123": 2 }` (overwrites A's write)
3. Result: count is 2 instead of 3. One strike is lost per Write/Edit double-fire event.

This is a **deterministic** strike loss on every double-fired Write/Edit call, not a probabilistic timeout race.

## Mitigation

1. **Immediate (fix D1.1 first)**: Eliminate the double-fire from D1.1. This removes the deterministic race condition.
2. **Make writes atomic**: Use a write-to-temp-then-rename pattern:
   ```javascript
   const tmpFile = guardCountFile + '.tmp';
   fs.writeFileSync(tmpFile, JSON.stringify(counts), 'utf8');
   fs.renameSync(tmpFile, guardCountFile);
   ```
3. **File locking**: Use a lockfile or advisory lock before read-modify-write operations.

## Relationship to D1.1

D1.1's double-fire makes D1.2's race condition occur on **every Write/Edit tool call** that generates an error signal, not just on rare timeout kill events. D1.1 is the primary trigger; the non-atomic write is a secondary compounding factor.
