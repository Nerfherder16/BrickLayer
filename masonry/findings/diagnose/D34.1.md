## Summary

`build_metric()` uses exact string match for verdict scoring (line 43 of `optimizer.py`). "CONCERNS" is absent from both the canonical `VALID_VERDICTS` set and all training data files. The hypothesized scoring gap does not currently exist because zero records in `scored_all.jsonl` or `scored_findings.jsonl` contain `verdict="CONCERNS"`. The latent risk is real but dormant.

## Evidence

### 1. build_metric() verdict scoring mechanism

File: `masonry/src/dspy_pipeline/optimizer.py`, lines 39-46

```python
ex_verdict = str(getattr(example, "verdict", "") or "").strip()
pred_verdict = str(getattr(prediction, "verdict", "") or "").strip()
if ex_verdict and pred_verdict and ex_verdict == pred_verdict:
    score += 0.4
```

This is a pure exact string match. No normalization, no alias mapping, no set membership check. The verdict component (weight 0.4) scores either 0.4 (exact match) or 0.0 (any mismatch).

### 5. Score penalty calculation (hypothetical)

- As percentage of 606 total examples: 3/606 = 0.50% of records affected

## Analysis

The question's hypothesis (from R33.3) was that `scored_all.jsonl` contains records with `verdict="CONCERNS"`. This is **not the case**. The training data pipeline correctly extracts top-level finding verdicts, not peer review verdicts. The `_VERDICT_RE` regex captures the first `**Verdict**:` line in each file, which is the finding-level verdict — never "CONCERNS" (a peer-review-only value).

The exact-match scoring in `build_metric()` is technically fragile (any synonym mismatch scores 0), but the actual training corpus is clean. The "CONCERNS" value exists only in:
1. Peer review sections (not extracted by the training pipeline)
2. Agent prompt documentation (not in training data)
3. The `program.md` peer review instructions (not in training data)

The latent risk identified by R33.2 (prompt inconsistency between "CONCERNS" and "WARNING") remains valid as a prompt hygiene issue, but it does not cause a scoring gap in the metric because no CONCERNS-verdicted records reach the training dataset.

## Verdict

HEALTHY. The exact-match metric in `build_metric()` does not currently suffer from a CONCERNS/WARNING scoring gap because zero training records contain `verdict="CONCERNS"`. The training extractor correctly captures finding-level verdicts (which use the canonical set) rather than peer-review verdicts (where CONCERNS appears). The hypothesized 3-record contamination from R33.3 is not present in the actual data.

## What Would Change This Verdict

- **WARNING**: If the training extractor regex were changed to also capture peer review verdicts, or if an agent began writing "CONCERNS" as a top-level finding verdict. The `_VERDICT_RE` regex would capture it, and `build_metric()` would silently penalize those records.
- **FAILURE**: If a significant fraction (>5%) of new training records entered with non-canonical verdicts due to agent prompt drift, the exact-match metric would systematically undercount agent performance.

## Recommend

No fix required. The system is functioning correctly. The prompt-level inconsistency (CONCERNS vs WARNING in agent prompts) is tracked by R33.2 and is a separate concern from metric scoring.
