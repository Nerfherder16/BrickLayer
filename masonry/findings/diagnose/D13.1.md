## Summary

Existing `scored_all.jsonl` entries all score 60/100 because ADBP findings have `"confidence": null` → `confidence_calibration=0`. Evidence quality (40) + verdict clarity (20) = 60 (the minimum training threshold). Masonry findings would score 70/100 — better, but not 100 — because their confidence values (0.97–0.99) exceed the rubric's "non-extreme" ceiling of 0.95. The rubric treats confidence > 0.95 as overconfident, awarding only 10 of 40 possible calibration points.

## Confidence Calibration Scoring Formula

```python
def _score_confidence_calibration(confidence, verdict, severity) -> int:
    if confidence is None:
        return 0                  # ADBP: 0/40

    points = 10                  # +10: field present

    if 0.5 <= confidence <= 0.95:    # "non-extreme range"
        points += 15             # +15: not overconfident

        is_critical = verdict in _CRITICAL_VERDICTS or severity in _CRITICAL_SEVERITIES
        if is_critical and confidence >= 0.70:
            points += 15         # +15: high confidence on high-stakes finding
        elif not is_critical and confidence <= 0.85:
            points += 15         # +15: moderate confidence on low-stakes finding

    return points
```

## Design Tension

The rubric's confidence ceiling (0.5–0.95 = "well-calibrated") penalizes masonry findings that express 0.97–0.99 confidence. For code-verification research (binary outcomes like "does this import work?"), 0.97+ confidence is accurate, not overconfident. The rubric was designed for scenario analysis findings where overconfidence is a real quality problem.

**Consequence**: If masonry findings were included in `score_findings.py` (they're excluded by the masonry/ directory exclusion per R13.1), they would score 70 rather than 100. They would still pass `min_training_score=60`. But they'd receive less weight in DSPy training than their actual quality warrants.
