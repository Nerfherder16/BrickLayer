# Finding: D7.1 — Drift Detection Non-Functional: `agent_db.json` `verdicts` Is Never Populated

**Question ID**: D7.1
**Status**: DONE
**Verdict**: FAILURE
**Agent**: diagnose-analyst
**Severity**: Medium
**Confidence**: 1.0

## Summary

`run_drift_check()` always returns an empty list because `agent_db.json` has `"verdicts": []` for every agent — and no component in the system writes to this field. The drift detector's skip condition `if not verdicts: continue` (line 155) applies to all 50+ agents. `masonry_drift_check` MCP tool invariably returns `{"reports": [], "count": 0}`. The drift detection infrastructure is built but the data pipeline feeding it is unimplemented: `masonry-handoff.js` explicitly notes this as a Phase 2 placeholder.


### `masonry_drift_check` always returns empty

Calling `masonry_drift_check` with any valid `agent_db_path` returns:
```json
{"reports": [], "count": 0}
```

This is not an error — it's technically correct given the empty input. The tool works; the data doesn't exist.

### What `masonry-observe.js` writes vs. what drift detector reads

`masonry-observe.js` tracks verdicts in `masonry-state.json` under `state.verdicts` (a per-session campaign-level dict: `{HEALTHY: N, FAILURE: M, ...}`). This is a counts-by-type summary, not a per-agent list. The drift detector needs a per-agent list of individual verdict strings.

The data structures are incompatible — even if `masonry-observe.js` data were piped to `agent_db.json`, it would need transformation.

### `detect_drift()` design is sound, data layer is missing

The drift computation in `detect_drift()` is well-designed:
- `baseline_score`: from `entry.get("score", 0.0)` — currently `0.85` (stub)
- `current_score`: computed from `recent_verdicts` list using `_score_verdict()`
- Thresholds: < 10% drift = ok, 10–25% = warning, > 25% = critical

This would work correctly if `verdicts` were populated. The algorithm is not the bug.

## Root Cause

`agent_db.json` was created as a structural placeholder during Phase 15 development. The actual scoring and verdict attribution pipeline (which would read findings, match them to agents via questions.md, and write verdict lists back to `agent_db.json`) was deferred to Phase 2 / Phase 16 and is not yet implemented. The drift detection system is architecturally complete but has no data.

## Fix Specification

A `sync_verdicts_to_agent_db.py` script needs to be created that:

1. Reads all finding files from `findings/`
2. Looks up each finding's question ID in `questions.md` to get the agent name
3. Extracts the `verdict` from each finding
4. Groups verdicts by agent name
5. Writes the list to `agent_db[agent_name]["verdicts"]`

Alternatively, modify `score_all_agents.py` to call `build_dataset()` and derive `verdicts` from the grouped examples before writing the final `agent_db.json`. This reuses the existing attribution logic.
