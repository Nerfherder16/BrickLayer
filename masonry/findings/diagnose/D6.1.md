# Finding: D6.1 — `upsert_registry_entry()` Non-Atomic Write Can Truncate All 46 Registry Entries

**Question ID**: D6.1
**Status**: DONE
**Verdict**: FAILURE
**Agent**: diagnose-analyst
**Severity**: Medium
**Confidence**: 0.9

## Summary

`upsert_registry_entry()` (and `append_to_registry()`) in `onboard_agent.py` write the updated YAML via `registry_path.write_text(yaml.dump(data, ...), encoding="utf-8")` — a direct overwrite with no atomic rename pattern. If the process is killed mid-write (hook timeout, OOM, OS scheduler), the `agent_registry.yml` file is left in a partially-written or empty state. `load_registry()` has YAML parse-error recovery and would silently return `[]`, effectively making all 46 registered agents disappear from the router's view. Recovery requires a full re-scan and re-onboarding run.

This is the same class of bug as:
- masonry-guard.js strike counter (fixed: F5.1)
- masonry-subagent-tracker.js agents.json (fixed: F5.3)

## Evidence

### Write path in `onboard_agent.py`

`upsert_registry_entry()` lines 317-318:
```python
registry_path.parent.mkdir(parents=True, exist_ok=True)
registry_path.write_text(yaml.dump(data, sort_keys=False), encoding="utf-8")
```

`Path.write_text()` opens the file in write mode (`'w'`), which truncates the file to zero bytes before writing any content. If the process is killed after truncation but before the full YAML is written:

1. **Zero-byte file**: `yaml.safe_load("")` returns `None` → `data = {}` → `data.get("agents") = []` → `load_registry()` returns `[]`
2. **Partial YAML** (rare): `yaml.safe_load(partial)` raises `yaml.YAMLError` → caught by `load_registry()`'s `except yaml.YAMLError` → returns `[]` with stderr warning

In both cases, all 46 registered agents vanish from the router's perspective. The YAML file on disk has lost its content.

### Load-side recovery in `registry_loader.py`

`load_registry()` lines 33-42:
```python
try:
    raw = path.read_text(encoding="utf-8")
except (OSError, FileNotFoundError):
    return []

try:
    data: dict[str, Any] = yaml.safe_load(raw) or {}
except yaml.YAMLError as exc:
    print(f"[registry_loader] Failed to parse YAML: {exc}", file=sys.stderr)
    return []
```

This is defensive — it never crashes on a corrupt file. But it also silently returns an empty list, so callers (including `route()`) cannot distinguish between "registry file missing" and "registry file corrupted by partial write". The empty list causes `route()` to fall back to Layer 4 for ALL requests.

### Blast radius vs. other hooks

| Component | Write type | Kill-mid-write consequence | Recovery |
|---|---|---|---|
| masonry-guard.js strike counter | Now atomic (F5.1) | None | N/A |
| masonry-subagent-tracker.js agents.json | Now atomic (F5.3) | None | N/A |
| `agent_registry.yml` (onboard_agent.py) | **Non-atomic** | Lose all 46 entries | Re-scan + re-onboard |

### Trigger frequency

The hook fires when a new `.md` file is written to an `agents/` directory — this is rare (agents don't change often). But it also fires from the MCP `masonry_onboard` tool, which could be called programmatically. Low frequency, but the blast radius (all 46 agents) is larger than the other fixed races.

## Fix Specification

Replace `registry_path.write_text(...)` with an atomic write in both `upsert_registry_entry()` and `append_to_registry()`:

```python
import os

# Atomic write: write to temp file, rename into place
tmp_path = registry_path.with_suffix(f".yml.tmp.{os.getpid()}")
try:
    tmp_path.write_text(yaml.dump(data, sort_keys=False), encoding="utf-8")
    tmp_path.replace(registry_path)  # atomic on POSIX and Windows (same drive)
except Exception:
    try:
        tmp_path.unlink(missing_ok=True)
    except Exception:
        pass
    raise
```

`Path.replace()` is equivalent to `os.rename()` — atomic on POSIX and atomic on Windows when both paths are on the same drive/filesystem. The PID suffix prevents collision between concurrent instances.

## Verdict

**FAILURE** — `agent_registry.yml` writes are non-atomic. A mid-write kill would truncate all 46 registry entries to zero/partial YAML, silently emptying the router's agent view. Fix: apply the same atomic write pattern already used in masonry-guard.js (F5.1) and masonry-subagent-tracker.js (F5.3).

**Priority**: P2 (medium) — low trigger frequency (rare agent onboarding events) but high blast radius (entire registry).
