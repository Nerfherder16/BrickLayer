# Finding: R15.1 — Session_ID Match Verified BUT CWD Guard Silently Drops Finding Events When Running From masonry/ Dir

**Question ID**: R15.1
**Status**: DONE
**Verdict**: WARNING
**Severity**: Medium
**Confidence**: 0.92

## Summary

Session_id values DO match between "start" and "finding" events when both hooks run in the same Claude Code session — both use `input.session_id` from their respective hook payloads, which is the parent session's ID. However, both `masonry-subagent-tracker.js` and `masonry-observe.js` share a CWD guard that silently drops events when Claude Code runs from the masonry/ project directory (rather than the BL root). The check `if (fs.existsSync(path.join(cwd, 'masonry')))` only passes when `masonry/` exists as a subdirectory of the CWD. When CWD = masonry directory itself (as in this self-research session), the guard fails and no routing_log.jsonl events are written. This explains why no "finding" events appear in routing_log.jsonl despite F14.2 being active.

## Evidence

### Session_id extraction — both hooks

**masonry-subagent-tracker.js** (SubagentStart):
```javascript
const agentEntry = {
  id: input.agent_id || `agent-${now}`,
  sessionId: input.session_id || "",       // ← parent session_id
  ...
};
// "start" event:
session_id: agentEntry.sessionId || agentEntry.id,
parent_session: input.session_id || '',
```

**masonry-observe.js** (PostToolUse):
```javascript
const { ..., session_id: sessionId = 'unknown', ... } = input;
// "finding" event:
session_id: sessionId,   // ← same parent session_id
```

Both use `input.session_id`. Claude Code passes the same session_id to both hook types for a given session → **session_ids match**.

### Observed routing_log.jsonl

```
4 events, all "start", session_id=f68d4e9c (from earlier BL root session)
0 "finding" events despite writing 5+ findings in current session
```

Confirmed: F14.2 "finding" emission is not reaching routing_log.jsonl in the current session.

### Root cause: CWD guard fails for masonry-scoped sessions

**masonry-observe.js** (line 304):
```javascript
const masonryDir = path.join(cwd, 'masonry');
if (fs.existsSync(masonryDir)) {
  fs.appendFileSync(routingLogPath, findingEntry + '\n', 'utf8');
}
```

**masonry-subagent-tracker.js** (line 103):
```javascript
const masonryDir = path.join(cwd, 'masonry');
if (fs.existsSync(masonryDir)) {
  fs.appendFileSync(routingLogPath, routingEntry + '\n', 'utf8');
}
```

When `cwd = C:\Users\trg16\Dev\Bricklayer2.0\masonry`:
- `masonryDir = C:\Users\trg16\Dev\Bricklayer2.0\masonry\masonry`
- `fs.existsSync(masonryDir)` → **false** → events silently dropped

When `cwd = C:\Users\trg16\Dev\Bricklayer2.0` (BL root):
- `masonryDir = C:\Users\trg16\Dev\Bricklayer2.0\masonry`
- `fs.existsSync(masonryDir)` → **true** → events written

The 4 existing "start" events in routing_log.jsonl were written when a BL root session spawned subagents (session `f68d4e9c`). All current self-research sessions run from the masonry/ directory → zero routing events.

### Routing_log pairing will work when CWD is correct

`score_routing.py` uses `findings_by_session.get(session_id)` to pair "start" events with "finding" events. If both are written with the same `session_id`, pairing succeeds. The logic is correct. The downstream_success gap is purely the CWD guard blocking writes, not a schema or session_id mismatch.

### Fix Options

**Option A — Resolve routing_log.jsonl path differently** (preferred):
Instead of constructing path relative to `cwd`, resolve it from a known anchor:
```javascript
// Find masonry/ dir by walking up from cwd
function findMasonryDir(startDir) {
  let dir = startDir;
  for (let i = 0; i < 5; i++) {
    const candidate = path.join(dir, 'masonry');
    if (fs.existsSync(candidate)) return candidate;
    const parent = path.dirname(dir);
    if (parent === dir) break;
    dir = parent;
  }
  // Also check if startDir IS masonry
  if (path.basename(startDir) === 'masonry' && fs.existsSync(startDir)) return startDir;
  return null;
}
```

**Option B — Write routing_log.jsonl to masonry/ directly, not masonry/masonry/**:
In masonry-observe.js, after finding the masonryDir fails at `path.join(cwd, 'masonry')`, fall back to checking if `cwd` itself ends with `masonry`:
```javascript
const masonryDir = path.join(cwd, 'masonry');
const resolvedMasonryDir = fs.existsSync(masonryDir) ? masonryDir
  : (path.basename(cwd) === 'masonry' ? cwd : null);
if (resolvedMasonryDir) {
  const routingLogPath = path.join(resolvedMasonryDir, 'routing_log.jsonl');
  fs.appendFileSync(routingLogPath, findingEntry + '\n', 'utf8');
}
```

## Verdict

**WARNING** — Session_id matching is correct; downstream_success scoring logic is sound. The blocking issue is the CWD guard: when Claude Code runs from the masonry/ directory, both hooks silently skip routing_log.jsonl writes. This prevents any routing training signal from masonry self-research sessions. The fix is to make the routing_log path resolution CWD-agnostic (check if cwd is already a masonry dir, or walk up to find one).
