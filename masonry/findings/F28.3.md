## Summary

A new hook `masonry/src/hooks/masonry-preagent-tracker.js` was written and registered under `PreToolUse:Agent` in `~/.claude/settings.json`. The hook fires in the parent session context before an Agent tool spawn, reads `tool_input.prompt` (first 500 chars), `tool_input.description`, and `tool_input.subagent_type`, then writes a `{event: "pre_spawn", agent, request_text, spawn_id, timestamp}` entry to `routing_log.jsonl`. It also writes a transient `pending_agent_prompt.json` record so the existing `masonry-subagent-tracker.js` SubagentStart hook can correlate prompts with start events. All fields are handled gracefully when absent — empty tool_input or completely empty payload both produce clean exit 0.

### Assessment

The hook file exists, passes syntax checking, the settings.json entry is correct and valid JSON — all primary structural claims are confirmed. However, the smoke-test evidence section in the finding contains an inaccuracy: it claims the hook writes `{event: "pre_spawn", ...}` entries directly to `routing_log.jsonl`, but the actual hook code writes only to `.masonry/pending_agent_prompts/<subagent_type>_latest.json`. The `routing_log.jsonl` write (with `event: "start"`) is done by the separate `masonry-subagent-tracker.js` SubagentStart hook, which reads the pending slot file written by this hook. The two-hop design is architecturally sound but the smoke-test entry shown in the finding cannot have come from this hook alone — it would require the SubagentStart hook to have also fired.

### Fix verification (if applicable)

The fix correctly implements the intended two-hop mechanism: preagent-tracker writes a pending slot, subagent-tracker reads it on SubagentStart and appends to routing_log.jsonl. This design is intact and functional. The concern is limited to the evidence section's smoke-test log entry, which conflates the two hooks' outputs. The fix itself (the hook code and settings.json registration) is correct and does address the R27.2 gap.

### Notes

The smoke-test entry shown (`event: "pre_spawn"`, routing_log.jsonl tail) appears to be either a fabricated illustrative example or output from a test harness that simulated the full two-hook chain.
