# Finding: R-w42.1 — Research-Analyst Dry-Run: Corpus Cannot Support eval_size=50

**Question**: Research-analyst dry-run — does improve_agent.py produce a stable before_score post-Wave-41 changes?
**Agent**: research-analyst
**Verdict**: WARNING
**Severity**: High

## Evidence

### E1: eval_size CLI/function default mismatch

`improve_agent.py` line 228 (CLI parser): `default=50` — Wave-41 change confirmed.
`improve_agent.py` line 69 (`run_loop()` function signature): `eval_size: int = 20` — internal default NOT updated. Programmatic callers (Kiln, tests) get eval_size=20, not 50.

### E2: eval_latest.json — before_score=0.5333 at eval_size=30 (not 50)

`masonry/agent_snapshots/research-analyst/eval_latest.json`:
- `"score": 0.5333`

- `"evaluated_at": "2026-03-25T00:34:48Z"`
- `"passed": 16, "failed": 14`

The last run used eval_size=30 (capped from the 36-record corpus), not the new CLI default of 50.

### E3: Corpus size — 36 research-analyst records

`scored_all.jsonl` contains exactly 36 research-analyst records (exact agent key-value match). This is unchanged from R-next.1.

### E4: Corpus math with eval_size=50 hard-blocks optimization

`eval_agent.py` line 239: `held_out = records[-eval_size:] if len(records) >= eval_size else records`

With 36 records and eval_size=50: `36 < 50` → all 36 records become held_out. Zero records remain for the training pool.

`optimize_with_claude.py` line 301: hard bail when `_available < _MIN_TIER_RECORDS=5`. With 0 training records, exit code 1 fires — optimization aborts.

The eval completes (scoring all 36 records), but the loop cannot proceed to the optimize step.

### E5: held_out_ids chain correctly wired

`eval_agent.py` lines 242-244, 363: `held_out_ids` assembled and returned in result dict.
`improve_agent.py` lines 126, 133: extracted with safe `.get()` default, passed as `excluded_ids`.
Chain is correctly wired — the issue is corpus capacity, not wiring.

## Analysis

The before_score (0.5333) clears the 0.50 floor. However, running `improve_agent.py research-analyst --loops 3` with the current CLI default (eval_size=50) will abort at the optimize step because 36 records < 50 → all records held-out → 0 training records → `_MIN_TIER_RECORDS` hard block.

Safe operating eval_size for the current corpus: `eval_size ≤ 31` (36 records − 5 minimum training = 31 maximum held-out).

The CLI/function default mismatch means: CLI invocation with no flags uses eval_size=50 (fails), while programmatic calls use eval_size=20 (works). This is a fragile divergence introduced by F-w41.2.

## Verdict Reasoning

WARNING: before_score=0.5333 clears the floor. But corpus (36 records) cannot support eval_size=50 — all records held-out, `_MIN_TIER_RECORDS` hard block fires. N=50 path was never exercised post-Wave-41. Two fixes needed: (1) update `run_loop()` default to match CLI (or add corpus-size check that auto-caps eval_size), (2) ensure organic record collection to grow corpus to ≥65 before eval_size=50 is safe.
