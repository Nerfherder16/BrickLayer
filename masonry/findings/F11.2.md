## Summary

`sync_verdicts_to_agent_db.py --questions-md masonry/questions.md` correctly limits attribution to masonry campaign findings. `fix-implementer` now shows 100% FIX_APPLIED verdicts (current_score=1.0, no drift). However, `research-analyst` and `diagnose-analyst` still show "critical drift" because they legitimately produced many FAILURE verdicts — that is the purpose of a research campaign. The drift metric conflates "agent found a failure" with "agent performed badly." This is a design-level issue that requires separating `finding_verdict` (what was discovered) from `agent_quality` (how well the agent performed).

```
  → current_score = 1.0 → alert=ok ✓ (correctly no drift)

research-analyst: 26 verdicts: FAILURE(11), WARNING(6), HEALTHY(4), COMPLETE(1), COMPLIANT(2), UNCALIBRATED(2)
  → current_score = 0.38 → alert=critical ✗ (false positive)

diagnose-analyst: 32 verdicts: FAILURE(20), DIAGNOSIS_COMPLETE(3), FIX_APPLIED(2), HEALTHY(4), FIXED(1), WARNING(2)
  → current_score = 0.31 → alert=critical ✗ (false positive)
```

Cross-project contamination is resolved for `fix-implementer`. But `research-analyst` and `diagnose-analyst` still show false critical alerts because their FAILURE verdicts are legitimate research findings.

### Root cause: semantic mismatch

The `_score_verdict()` function scores FAILURE=0.0 (bad), WARNING=0.5 (partial), HEALTHY=1.0 (good). This is correct for measuring **output quality** — did the finding have the right structure, confidence, and evidence?

But it is incorrect for measuring **agent quality** via **finding verdicts**. A `research-analyst` finding `FAILURE` means the system being investigated has a problem. This is exactly what the agent is supposed to do. The agent "succeeded" by finding a real failure.

The intended usage of `agent_db["verdicts"]` as described in D7.1 and the drift detector's comment:
> "Compares recent agent verdicts against baseline scores to detect **performance degradation**"

Performance degradation for a research agent should be measured by:
- Confidence scores (is the agent confident in its findings?)
- Evidence quality (is the agent producing well-evidenced findings?)
- Not by whether findings are HEALTHY vs FAILURE

### Cross-project contamination resolution

Before masonry-only sync: `compliance-auditor` appeared with 3 verdicts (from other projects). After: `compliance-auditor` correctly disappears (no masonry questions attributed to it). `fix-implementer` count: 25 → 38 (more findings now attributed correctly).

### What the fix should be

**Short-term fix applied** (this finding): The `--questions-md masonry/questions.md` flag is documented and cross-project contamination is resolved. `fix-implementer` drift check is now accurate.

**Deferred deeper fix** (Wave 12): The drift check metric needs to use agent confidence (float) rather than finding verdict (categorical) as the quality signal. `confidence` is a direct measure of agent certainty and is consistently populated (all findings have `**Confidence**:` field). Using mean confidence across recent findings would correctly distinguish:
- research-analyst with high-confidence FAILURE findings (good agent, broken system)
- research-analyst with low-confidence FAILURE findings (uncertain agent, needs optimization)
