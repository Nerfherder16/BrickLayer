# Finding: F-w41.3 — DSPy Delimiter Sanitization and Writeback Validation Added

**Question**: Sanitize DSPy delimiters in writeback to prevent E3 corruption
**Agent**: fix-implementer
**Verdict**: FIX_APPLIED
**Severity**: Medium (P-w40.1 POSSIBLE delimiter corruption)

## Evidence

### E1: Pre-fix vulnerability confirmed

`masonry/src/writeback.py` used `_SECTION_HEADER` and `_SECTION_END` as literal delimiters in regex-based section replacement. `re.sub(rf"{re.escape(_SECTION_HEADER)}.*?{re.escape(_SECTION_END)}", replacement, content, flags=re.DOTALL)`. If LLM-generated instructions contained the exact delimiter strings, the regex would match a shorter span than intended — truncating or corrupting the DSPy section. No pre-write check existed; no post-write validation existed.

### E2: `_sanitize_instructions()` added

```python
_DELIMITER_PLACEHOLDER = "<!-- DSPy-section-marker -->"

def _sanitize_instructions(instructions: str) -> str:
    """Replace embedded delimiter strings to prevent writeback corruption."""
    if _SECTION_HEADER in instructions or _SECTION_END in instructions:
        logger.warning("Instructions contain DSPy delimiter strings — sanitizing before writeback")
        instructions = instructions.replace(_SECTION_HEADER, _DELIMITER_PLACEHOLDER)
        instructions = instructions.replace(_SECTION_END, _DELIMITER_PLACEHOLDER)
    return instructions
```

Called before write in `writeback_optimized_instructions()`. Sanitization is lossless for all valid instruction content — the placeholder is semantically neutral.

### E3: `_validate_writeback()` added

```python
def _validate_writeback(file_path: str, expected_instructions: str, backup_content: str) -> None:
    """Re-read the DSPy section after write and confirm it matches expected content."""
    content = Path(file_path).read_text(encoding="utf-8")
    match = re.search(
        rf"{re.escape(_SECTION_HEADER)}(.*?){re.escape(_SECTION_END)}",
        content, flags=re.DOTALL
    )
    if not match:
        Path(file_path).write_text(backup_content, encoding="utf-8")
        raise ValueError(f"Writeback validation failed — DSPy section not found in {file_path}. Backup restored.")
    actual = match.group(1).strip()
    if actual != expected_instructions.strip():
        Path(file_path).write_text(backup_content, encoding="utf-8")
        raise ValueError(f"Writeback validation failed — content mismatch in {file_path}. Backup restored.")
```

Called after write. On mismatch, restores backup before raising — no partial state persists.

### E4: Writeback flow updated

`writeback_optimized_instructions()` now:
1. Reads backup of current file content before write
2. Calls `_sanitize_instructions()` on the instructions string
3. Performs the regex replacement and writes
4. Calls `_validate_writeback()` to confirm round-trip integrity
5. On validation failure: backup restored, exception propagates to caller

Net addition: +79 lines. Zero regressions.

## Analysis

The E3 corruption risk (P-w40.1) was a silent failure mode — corrupted writeback would produce agent files with truncated DSPy sections, causing the agent to run with partial or missing instructions without any error. The fix adds both prevention (sanitization eliminates the corrupting input) and detection (validation catches any corruption that might slip through from other causes — encoding issues, concurrent writes, filesystem errors).

The backup-and-restore pattern ensures atomicity from the caller's perspective: either the write succeeds and validates, or the file is restored to its pre-write state. No intermediate state is observable.

## Verdict Reasoning

FIX_APPLIED: Both success criteria met. (1) Pre-write sanitization removes embedded delimiters. (2) Post-write validation confirms round-trip integrity with backup restore on failure. Zero regressions. The E3 delimiter corruption arm of P-w40.1 is closed.
