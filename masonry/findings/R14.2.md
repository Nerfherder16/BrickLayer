### How vigil uses max_score()

```python
# run_vigil.py:353-354  (scored_all augmentation path only)
avg_score = sum(scores) / len(scores)
max_pts = _rubric_max_score(agent)   # always 100
pct = avg_score / max_pts            # equivalent to avg_score / 100
```

The `pct` is then compared to `ROSE_PASS_RATE=0.80` and `THORN_PASS_RATE=0.50`. Since `max_pts=100` always, `pct = avg_score/100`. An agent scoring avg 80/100 in scored_all.jsonl becomes a Rose. This normalization is correct — the rubric max scores confirm 100 is the right denominator.

### min_training_score() is not used by vigil

`rubrics.py` exports `min_training_score()` (findings=60, code=70, ops=60, routing=65). This function is used by `score_all_agents.py` to filter scored_all.jsonl, not by `run_vigil.py`. Vigil only calls `max_score()`. The per-category min thresholds are not surfaced in vigil's classification.

### get_category() fallback behavior

```python
def get_category(agent_name: str) -> str:
    return AGENT_CATEGORIES.get(agent_name.lower().strip(), "findings")
```

Unknown agents default to "findings" category → max_score=100. This is safe since all categories return 100, but is semantically imprecise — an unknown agent claiming findings-category scores would be classified as Rose/Bud/Thorn on the same scale as research agents.
