# Finding: F-w42.1 — P4 Slot Collision Fixed: UUID-Keyed Slots Implemented

**Question**: Fix P4 slot collision — replace single-slot-per-type with UUID-keyed slots
**Agent**: fix-implementer
**Verdict**: FIXED
**Severity**: Low (R-w40.2 WARNING/Low)

## Evidence

### E1: Write side — masonry-preagent-tracker.js

`const crypto = require('crypto')` added. Slot filename changed from `${subagentType}_latest.json` to `${subagentType}-${crypto.randomUUID()}.json`. Each concurrent spawn of the same agent type writes to its own unique file — overwrite collision is impossible.

### E2: Read side — masonry-subagent-tracker.js

Single `existsSync`+`readFileSync` lookup replaced with three-phase approach:
1. `readdirSync` pending dir, filter files matching `${subagentType}-*.json` (UUID format), stat each for mtime, sort ascending (oldest-first), iterate until a fresh slot (age < TTL_MS) is found and consumed via `unlinkSync`. Stale slots deleted during scan.
2. Fallback to legacy `${subagentType}_latest.json` format for backwards compatibility — consume if fresh, delete if stale.
3. All I/O wrapped in try/catch — missing directory or unreadable file is non-fatal.

### E3: Node.js verification results

- Two concurrent same-type writes produce two distinct UUID-named files ✓
- First SubagentStart consumes oldest prompt, second consumes remaining — no collision ✓
- Legacy `_latest.json` file readable and consumable without crash ✓
- Zero-slot case completes without error ✓

### E4: Test results

56 passing / 20 pre-existing failures — no regressions.

## Analysis

R-w40.2 identified that slot collisions corrupted `routing_log.jsonl` `request_text` for ~1 in 6 routing records. The UUID-keyed approach eliminates the collision by design: no two writes can target the same filename. The oldest-first read strategy ensures requests are matched in dispatch order when multiple same-type agents are pending.

Backwards compatibility is preserved: the legacy `_latest.json` fallback means stale slots from before this fix expire and are consumed or cleaned up normally. No migration required.

The 5 stale slot files in `~/.masonry/pending_agent_prompts/` (explore, overseer, peer-reviewer, agent-auditor, forge-check identified in R-w40.2 E5) will be consumed or expire via the TTL check on next SubagentStart fire — no manual cleanup needed.

## Verdict Reasoning

FIXED: All four success criteria met. (1) Write side uses `crypto.randomUUID()` suffix. (2) Read side globs for `${subagentType}-*.json` and picks oldest within TTL. (3) Legacy `_latest.json` format handled via fallback — backwards compatible. (4) Zero-slot case handled without crash. P4 slot collision permanently closed. `routing_log.jsonl` `request_text` corruption rate drops from ~1/6 to effectively zero for all new spawns.
