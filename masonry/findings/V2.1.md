# Finding: V2.1 — masonry-approver Bash Tool Vector Is Completely Unblocked; All Bash Commands Auto-Approved During Build

**Question ID**: V2.1
**Status**: DONE
**Verdict**: FAILURE
**Agent**: design-reviewer
**Severity**: High
**Confidence**: 1.0

## Summary

The `getCandidateDirs` function in `masonry-approver.js` extracts paths from Bash `command` strings for the sole purpose of locating `.autopilot/mode` (to determine if a build is active). It does NOT pass these paths to any security check. The V1.5 fix specification's proposed `isTier1Tier2(filePath)` check uses `toolInput.file_path || toolInput.path || ''` — both fields are `undefined` for Bash tool calls. The effective check is `isTier1Tier2('')` → `false` → all Bash commands are auto-approved during build mode regardless of what files they write, copy, or delete.

This is a **complete bypass** of the V1.5 fix for any Bash-mediated write. Developer agents routinely use Bash for file writes during builds (`echo >`, `cp`, `python -c`, `sed -i`, etc.). The V1.5 fix specification is incomplete and cannot be implemented as written without also addressing this vector.

## Evidence

### getCandidateDirs Function (masonry-approver.js lines 92-123)

```javascript
function getCandidateDirs(parsed) {
  const dirs = [];
  const toolInput = parsed.tool_input || {};

  // Case 1: Write/Edit tool — file_path is present
  if (toolInput.file_path) {
    dirs.push(dirname(toolInput.file_path));
  }

  // Case 2: Bash tool — command string contains paths
  if (toolInput.command) {
    const winPaths = toolInput.command.match(/[A-Za-z]:[/\\][^\s"';&|)]+/g);
    if (winPaths) {
      for (const p of winPaths) {
        const clean = p.replace(/[/\\]$/, "");
        dirs.push(clean);                // ← pushed to dirs (for mode-finding)
        dirs.push(dirname(clean));       // ← NOT checked against Tier 1/2 patterns
      }
    }
    const unixPaths = toolInput.command.match(/(?<=\s|^|"|')\/[a-zA-Z][^\s"';&|)]+/g);
    if (unixPaths) {
      for (const p of unixPaths) {
        dirs.push(p);        // ← pushed to dirs (for mode-finding)
        dirs.push(dirname(p)); // ← NOT checked against Tier 1/2 patterns
      }
    }
  }

  if (parsed.cwd) {
    dirs.push(parsed.cwd);  // ← CWD also for mode-finding only
  }

  return dirs;
}
```

**Purpose**: `getCandidateDirs` returns directories to walk upward looking for `.autopilot/mode`. It is exclusively for mode detection.

**NOT used for security**: The returned `dirs` array feeds into `findAutopilotMode(dir)` and `findUiMode(dir)`. It does not flow into any Tier 1/2 path check. Paths extracted from Bash commands are never tested against `TIER1_TIER2_PATTERNS`.

### The V1.5 Fix Specification Gap

The V1.5 proposed check:
```javascript
const toolInput = parsed.tool_input || {};
const filePath = toolInput.file_path || toolInput.path || '';
if (approve && isTier1Tier2(filePath)) {
  process.exit(0);  // block approval
}
```

For a Bash tool call with `tool_input = { "command": "echo x >> project-brief.md" }`:
- `toolInput.file_path` → `undefined`
- `toolInput.path` → `undefined`
- `filePath = ''`
- `isTier1Tier2('')` → `false` (empty string doesn't match any pattern)
- **Result: auto-approved**

### Bash Write Attack Vectors During Build Mode

All of the following would be auto-approved by the current approver (and the incomplete V1.5 fix):

```bash
# Redirect-based writes
echo "overwritten" > project-brief.md
echo "overwritten" >> project-brief.md
cat new-content.txt > docs/architecture.md

# Copy-based writes
cp .autopilot/spec.md project-brief.md
cp modified-registry.yml agent_registry.yml

# In-place edits
sed -i 's/old/new/g' src/routing/router.py
sed -i 's/threshold/0.99/g' src/routing/semantic.py

# Python/script writes
python -c "open('project-brief.md','w').write('modified')"
python scripts/patch_registry.py

# Delete + recreate
rm agent_registry.yml && cp /tmp/fake-registry.yml agent_registry.yml
```

All of these set `tool_name = "Bash"` and `tool_input.command = "..."`. The V1.5 fix (as specified) passes `filePath = ''` and approves all of them.


`parsed.tool_name` is available but never checked in the current code. The fix can use this field.

### Corrected Fix Specification for Bash Vector

**Option A: Blanket Bash exclusion from auto-approval (recommended)**
```javascript
const toolName = (parsed.tool_name || '').toLowerCase();

// Never auto-approve Bash — command content is too hard to analyze reliably
if (approve && toolName === 'bash') {
  process.exit(0);  // fall through to user prompt
}
```

This is the safest approach. Developer agents that need to run tests, `npm install`, `python analyze.py`, etc. will still prompt the user. The vast majority of build automation uses Write/Edit (auto-approved) not Bash.

**Option B: Command-string pattern matching (defense-in-depth supplement)**
```javascript
const TIER1_TIER2_NAMES = [
  'project-brief.md', 'agent_registry.yml', 'hooks.json', 'constants.py'
];
const TIER1_TIER2_DIRS = ['src/', 'src\\', 'docs/', 'docs\\'];

function bashWritesToTier12(command) {
  if (!command) return false;
  const lower = command.toLowerCase();
  return (
    TIER1_TIER2_NAMES.some(n => lower.includes(n)) ||
    TIER1_TIER2_DIRS.some(d => lower.includes(d))
  );
}

if (approve && toolName === 'bash' && !bashWritesToTier12(bashCommand)) {
  // approve only Bash commands that don't touch Tier 1/2 files
  // ... write approval
}
// else: fall through to user prompt
```

Option B is more permissive but complex — command parsing is unreliable against obfuscated or indirect writes. Option A is simpler and more secure.

### Complete Fix Specification (V1.5 + V2.1 Combined)

```javascript
// In main(), before writing approval:
const toolInput = parsed.tool_input || {};
const toolName = (parsed.tool_name || '').toLowerCase();
const filePath = toolInput.file_path || toolInput.path || '';
const bashCommand = toolInput.command || '';

// Pattern 1: Block Tier 1/2 file writes for Write/Edit tools
if (approve && isTier1Tier2(filePath)) {
  process.exit(0);
}

// Pattern 2: Never auto-approve Bash during build
if (approve && toolName === 'bash') {
  process.exit(0);
}
```

With `isTier1Tier2` patterns corrected per F2.2 (adding `(?:^|[/\\])` prefix to patterns 5 and 6), this constitutes a complete, implementation-ready fix for both V1.5 and V2.1.
