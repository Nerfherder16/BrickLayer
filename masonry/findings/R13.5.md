# Finding: R13.5 — `masonry_nl_generate` Produces Trowel-Compatible Questions With Minor Format Differences

## Summary

`masonry_nl_generate` (via `bl.nl_entry.generate_from_description`) produces 4 questions per description with correct `mode` (diagnose/research/validate) and `**Status**: PENDING`. Trowel's deterministic router CAN route these questions because `_MODE_FIELD_RE` accepts both `**Mode**:` and `**Operational Mode**:`. However, the format deviates from BL 2.0 standard in two ways: (1) `**Test**:` instead of `**Method**:`, and (2) `**Verdict threshold**:` instead of `**Success criterion**:`. Routing works; agent-side context is slightly degraded.

**Test**: Read the relevant code and verify the behavior described above.
**Verdict threshold**:
- HEALTHY: No failure mode found; behavior is correct under the described conditions
- FAILURE: A concrete failure mode, race, or correctness gap is found

## Trowel Routing Compatibility

```python
# deterministic.py:36
_MODE_FIELD_RE = re.compile(r"\*\*(?:Operational\s+)?Mode\*\*:\s*(\w+)", re.IGNORECASE)
# → Matches BOTH "**Mode**: diagnose" AND "**Operational Mode**: research"
```

Trowel's deterministic routing reads `**Mode**:` via `_MODE_FIELD_RE` and dispatches the first registered agent for that mode. NL-generated questions with `**Mode**: diagnose` will correctly route to `diagnose-analyst`. Routing is NOT broken.

## Impact of Field Differences

- **`**Test**:` vs `**Method**:`**: The `**Method**:` field in standard BL 2.0 specifies which specialist agent to use (e.g., `research-analyst`). NL questions use `**Test**:` which describes the investigation approach instead. Agents receiving NL questions won't see a specialist name — they'll use the Mode routing fallback only.
- **`**Verdict threshold**:` vs `**Success criterion**:`**: Agents look for `**Success criterion**:` to determine completion. NL questions provide generic HEALTHY/FAILURE thresholds instead of specific success criteria. Finding quality may be lower.

## Verdict

**WARNING** — `masonry_nl_generate` produces valid, Trowel-routable questions. Routing works correctly. The `**Method**:` and `**Success criterion**:` deviations reduce the specificity of generated questions vs. hand-authored BL 2.0 questions, but do not break the research loop. Questions generated via nl_entry should be treated as "draft" questions that benefit from manual refinement before running a campaign.
