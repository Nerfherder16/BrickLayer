## Summary

The plugin `hooks/hooks.json` and global `~/.claude/settings.json` overlap on **every event type** the plugin registers, not just PostToolUse (observe+guard) as Wave 1 found. The global settings.json is a strict superset of the plugin's hook coverage — it includes all plugin hooks plus additional ones (masonry-approver, masonry-lint-check, masonry-design-token-enforcer, masonry-context-monitor, masonry-tdd-enforcer, masonry-session-summary). Removing all hooks from the plugin config (or the `"hooks"` key from `plugin.json`) eliminates all double-fires with zero coverage loss. The minimum safe fix is to strip the `PostToolUse` entries from `hooks/hooks.json` (the two entries causing D1.1); removing all plugin hook entries is the correct full fix.

### Double-Fire Scope Is Wider Than D1.1 Found

Wave 1 (D1.1) identified the PostToolUse double-fire for observe+guard. The full double-fire scope:

- **SessionStart**: masonry-session-start fires twice per session open
- **UserPromptSubmit**: masonry-register fires twice per user prompt
- **PreToolUse ExitPlanMode**: masonry-context-safety fires twice per plan-mode exit
- **PostToolUse Write/Edit**: masonry-observe fires twice, masonry-guard fires twice (confirmed D1.1)
- **PostToolUse other tools**: masonry-observe fires ONCE from plugin only (global has Write|Edit matcher); masonry-guard same — both have internal WATCHED_TOOLS filters so this extra invocation is harmless but wasteful
- **PostToolUseFailure**: masonry-tool-failure fires twice per tool failure
- **SubagentStart**: masonry-subagent-tracker fires twice per agent spawn (feeds D2.2's race condition)
- **Stop**: masonry-stop-guard, masonry-build-guard, masonry-ui-compose-guard each fire twice

### Why Coverage Is Fully Preserved After Removing Plugin Entries

The critical question: does the global settings.json matcher `Write|Edit` cover the same events as the plugin's no-matcher registration for observe and guard?

Yes, because:
1. `masonry-observe.js` has an internal `WATCHED_TOOLS` filter:
   ```javascript
   const WATCHED_TOOLS = ["Edit", "Write", "MultiEdit", "NotebookEdit"];
   if (!WATCHED_TOOLS.includes(toolName)) process.exit(0);
   ```
   When the plugin fires observe for non-Write/Edit tools (Read, Bash, Agent, etc.), observe exits immediately. The plugin's no-matcher registration adds zero effective coverage beyond Write/Edit.

2. `masonry-guard.js` has the same WATCHED_TOOLS guard (confirmed in D1.2 research).

3. All other duplicated hooks (session-start, register, context-safety, tool-failure, subagent-tracker, stop guards) have identical registrations in both plugin and global configs with no difference in matcher or event scope.

### Minimum Safe Fix (Targeted)

To fix only the D1.1 double-fire for observe and guard, remove the two PostToolUse entries from `hooks/hooks.json`:

```json
// Remove from hooks/hooks.json PostToolUse section:
{
  "hooks": [
    { "type": "command", "command": "...masonry-observe.js", "timeout": 5, "async": true }
  ]
},
{
  "hooks": [
    { "type": "command", "command": "...masonry-guard.js", "timeout": 3, "async": true }
  ]
}
```

### Recommended Full Fix (Zero-Loss, No Regression Risk)

Remove all hooks from `hooks/hooks.json` entirely and keep only the structural plugin config entries (SessionStart is fine to keep in plugin if desired, but global already covers it):

```json
{
  "description": "Masonry research-loop hooks",
  "hooks": {}
}
```

Or, remove the `"hooks"` key from `plugin.json` entirely, keeping only the `"mcp"` registration.

This is zero-regression because:
- Global settings.json covers every event type the plugin covers
- Global settings.json additionally provides masonry-approver, masonry-lint-check, masonry-design-token-enforcer, masonry-context-monitor, masonry-tdd-enforcer, masonry-session-summary — all without plugin support

### Residual Risk: SubagentStart Double-Fire Amplifies D2.2

The SubagentStart double-fire (masonry-subagent-tracker fires twice per spawn) was not identified in Wave 1 and feeds directly into D2.2's race condition hypothesis. Under the double-fire, two tracker invocations race on `agents.json` per SubagentStart event. The full fix (empty plugin hooks) resolves this simultaneously.
