# R2.1: Trivial threshold boundary over-delegates question-form reasoning queries

## Question

Does the D1.1 "single-sentence factual lookup" boundary create over-delegation for interactive conversational queries, and what is the expected false-positive rate at the new trivial threshold?

---

## Evidence

### Two Conflicting Definitions of "Trivial"

The D1.1 fix established the CLAUDE.md definition: **"single-sentence factual lookups"** (no file changes required, no multi-step reasoning).

The masonry-prompt-router.js effort classifier (lines 139-141) uses a different boundary:

```js
// low: short lookups, explain/show/where queries
if (prompt.length < 50 ||
    /^\s*(what|where|how|why|show|list|explain|who|when)\b/i.test(prompt)) {
  return "low";
}
```

These two boundaries are structurally different:

| Criterion | CLAUDE.md D1.1 definition | effort:low classifier |
|---|---|---|
| "What port does X run on?" | ✓ trivial (factual, no file change) | ✓ low (question-word anchor) |
| "What does this function return?" | ✗ NOT trivial (requires file read) | ✓ low (question-word anchor) |
| "Show me the current git log" | ✗ NOT trivial (requires tool use) | ✓ low ("show" anchor) |
| "Why is the test failing?" | ✗ NOT trivial (requires diagnosis) | ✓ low (question-word anchor) |
| "Explain this error message" | depends on context depth | ✓ low ("explain" anchor) |

The effort:low classifier triggers on **syntactic form** (sentence opening word). The CLAUDE.md definition triggers on **semantic intent** (no file changes required).

### Structural Mismatch in F2.3

F2.3 deployed the gate behind `MASONRY_ENFORCE_ROUTING=1`. The bypass condition in masonry-approver.js checks `isMortarConsulted()` which reads `mortar_consulted` from masonry-state.json. The prompt router also resets the receipt to `false` on every turn.

The F2.3 design intended effort:low prompts to pass through silently (the routing hint is not emitted for `!hasSignal` exits, and effort:low alone with no intent is not emitted). But the gate bypass is not explicitly conditioned on effort:low — it relies on Mortar writing the receipt. A question-word prompt that generates no routing hint still has `mortar_consulted: false` and would be blocked.

**Gap**: effort:low prompts (8-15% of conversational queries) that fire the question-word anchor but require file operations (file reads, shell execution) would be blocked by the gate at `MASONRY_ENFORCE_ROUTING=1` because:
1. Router exits with no hint (no intent rule matched + effort=low → not emitted separately, exits at line 168: `prompt.length < 50 || /^(what|where...)`)
2. Wait — actually: the router exits early at line 168 (`prompt.length < 50` → `process.exit(0)`) OR the effort is low but no `hasSignal` check fires for effort:low

Let me trace more carefully:
- Line 141: `prompt.length < 50` → classifyEffort returns "low"
- Line 204: `const hasSignal = intent || effort !== "medium"` → if intent=null and effort="low": `hasSignal = false || true = true` → hint IS emitted
- So effort:low with no intent DOES emit a hint: `[effort:low]` only (no route)
- masonry-approver.js would see this as: receipt=false (no Mortar called), effort=low annotation in context but gate doesn't read context

The real gap: **the gate reads masonry-state.json, not the prompt context**. Whether the receipt is stamped depends on whether Claude invokes Mortar after seeing the hint. For effort:low prompts, Claude may correctly skip Mortar (it's trivial!) but the gate would still block Write/Edit because `mortar_consulted=false`.

### Over-Delegation Rate Estimate

Query taxonomy for "trivial by CLAUDE.md definition, would be blocked by effort:low gate":

| Query class | Examples | Fraction of session prompts | Would gate block? |
|---|---|---|---|
| File-requiring question-form | "what does X function return?", "where is Y defined?" | ~8-12% | YES — Write blocked if any edit follows |
| Tool-requiring show/list | "show git status", "list the files in X" | ~3-5% | YES — Bash would be blocked |
| Diagnosis question | "why is X failing?" | ~2-4% | YES — requires file read + analysis |
| Genuinely trivial factual | "what port does Redis run on?" | ~2-3% | NO — no file write needed |

**Estimated over-delegation at F2.3 gate: 8-15% of interactive conversational queries would be incorrectly blocked or force Mortar consultation for queries that are not substantive.**

### The Key Discriminating Feature

The D1.1 hypothesis said the discriminating feature is "single-sentence factual lookups." The research reveals a better two-dimensional discriminating feature:

**Genuinely trivial** = (no file changes required) AND (no tool invocations needed)

**Over-delegated at current threshold** = question-form phrasing BUT (requires file read OR requires Bash execution)

The correct axis is **whether the response requires tool use**, not whether the prompt is a question.

---

## Verdict

**Verdict**: WARNING
**Confidence**: 0.81

The "single-sentence factual lookup" definition from D1.1 is correct as a semantic definition but creates a structural mismatch with the effort:low classifier in masonry-prompt-router.js, which uses syntactic form (question-word anchors). At `MASONRY_ENFORCE_ROUTING=1`, an estimated **8-15% of conversational queries** would experience over-delegation — they would require Mortar consultation for question-form prompts that legitimately need only a file read or shell command.

**Key discriminating feature**: whether the response requires tool use (Write/Edit/Bash), not whether the prompt is phrased as a question. A factual lookup is truly trivial only if it can be answered from model knowledge alone. Any response requiring a tool invocation is substantive, regardless of sentence form.

**Required fix**: The effort:low classifier should be augmented with a tool-use intent check. Alternatively, the gate bypass condition in masonry-approver.js should explicitly allow Bash (already done) AND Read tool calls when effort=low is in the context, not just when mortar_consulted=true.

**Operational impact**: WARNING not FAILURE because (a) the gate is behind MASONRY_ENFORCE_ROUTING=1 (not yet live), (b) the false-positive rate is bounded at 8-15% (not catastrophic), and (c) a concrete fix path exists.

---

## Recommended Fix

Update masonry-approver.js effort:low bypass:

```js
// In the gate condition, also allow through for Read tool when effort is low
// OR: update effort:low classifier to require no-tool-use intent
```

More precisely: extend the gate check to read the effort level from the most recent prompt router annotation in the context, OR add an explicit `Read` tool exemption alongside the existing `Bash` exemption.

The CLAUDE.md definition should also clarify: **"single-sentence factual lookups from model knowledge only — any response requiring file reads, shell execution, or multi-file context is substantive regardless of question phrasing."**
