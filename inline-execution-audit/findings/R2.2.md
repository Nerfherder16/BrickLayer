### Pattern Proposals for Spec+Build Coverage

**Pattern A: Spec/PRD verbs** (high confidence, collision-free)
```js
/\b(write.?a.?spec|create.?a.?spec|spec.?out|product.?requirements|prd|requirements.?doc)\b/i
```
- Tested against all 10 existing rules: zero overlap
- Coverage: formal spec/PRD creation requests
- False-positive risk: NONE — "spec" in isolation would be risky but context words prevent collision

**Pattern B: Plan-out verbs** (high confidence, collision-free at position 6)
```js
/\b(plan.?out|plan.?this|blueprint.?the|blueprint.?a|design.?the.?architecture|write.?a.?plan.?for)\b/i
```
- Risk: `plan` alone overlaps with architecture rule ("architecture" in L50 pattern)
- Mitigation: compound phrases (plan-out, plan-this) avoid collision
- `blueprint` has no match in any existing rule
- Placement: position 6 in INTENT_RULES array (after debugging, before build) prevents false-match

**Pattern C: Build pipeline verbs** (medium confidence, collision-free)
```js
/\b(spec.?to.?build|plan.?to.?implement|feature.?spec|feature.?plan|roadmap.?for.?building)\b/i
```
- These compound phrases uniquely identify spec+build pipeline requests
- No overlap with any existing rule

**Pattern D: architect+spec compound** (HARD COLLISION — DO NOT ADD)
```js
/\b(architect.{0,20}spec|design.{0,20}spec|architect.{0,20}plan)\b/i
```
- Hard collision with rule L50: `\b(architect|architecture|system.?design)\b/i`
- L50 fires FIRST (position 2 in array) and routes to "architect + design-reviewer"
- Cannot be resolved without positional change OR co-routing modification
- Recommendation: OMIT — accept 25% coverage gap here

### Maintenance Verb Expansion (Build Rule Fix)

Current build rule (line 76):
```js
/\b(build|implement|create|add.?feature|write.?code|develop|scaffold)\b.{0,40}\b(app|component|endpoint|api|service|function|class|module|page|route)\b/i
```

**Missing maintenance verbs** identified in D5.1: `fix`, `update`, `make`, `change`, `set`, `configure`, `apply`, `enable`, `disable`, `modify`

**Expanded verb set** (collision-safe):
```js
/\b(build|implement|create|add.?feature|write.?code|develop|scaffold|fix|update|make|change|set|configure|apply|enable|disable|modify)\b.{0,40}\b(app|component|endpoint|api|service|function|class|module|page|route)\b/i
```

Collision analysis:
- `fix` alone: no collision (debugging rule uses `debug|broken|error|exception` not `fix`)
- `update`: no collision (no existing rule uses update as standalone)
- `change`: no collision
- `set`: borderline — "set up" could be ambiguous, but the `.{0,40}\b(app|...)` suffix prevents false matches on non-code requests
- `configure`: no collision

**All 5 new patterns safe to add without collision.**
