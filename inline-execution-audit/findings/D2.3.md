# D2.3: Where exactly in Mortar's agent instructions should the receipt writer live, and what is the complete write specification to prevent the Mortar-before-file-write deadlock?

**Status**: DIAGNOSIS_COMPLETE
**Date**: 2026-03-29
**Agent**: diagnose-analyst

## Hypothesis Under Test

That a receipt writer can be added to Mortar's agent instructions to write `mortar_consulted: true` + `mortar_session_id` to masonry-state.json, and that this can be done without causing a deadlock when the PreToolUse gate (masonry-approver.js) fires on Mortar's own Write/Edit calls.

## Evidence

### 1. Mortar's tool list prevents Mortar from writing the receipt

Both copies of mortar.md (global at `C:/Users/trg16/.claude/agents/mortar.md` and project at `C:/Users/trg16/Dev/Bricklayer2.0/.claude/agents/mortar.md`) declare identical frontmatter:

```yaml
tools:
  - Read
  - Glob
  - Grep
  - Agent
  - mcp__recall__recall_search
  - mcp__recall__recall_store
```

**Mortar has no Write, Edit, or Bash tool.** It cannot write to any file. The D2.2 finding's recommendation ("Mortar's agent instructions must be updated to write `mortar_consulted: true`") is structurally impossible under the current tool grant. Adding a Write instruction to mortar.md would be ignored because Mortar's agent context does not have the Write tool available.

This is the root cause of the deadlock design problem: the receipt writer cannot live in Mortar's agent instructions because Mortar lacks write tools. This is not a bug --- it is an intentional design choice (Mortar is a read-and-route agent, not a write agent).

### 2. The two masonry-state.json files --- split brain

The approver reads from a hardcoded path:
```js

const MASONRY_STATE_PATH = "C:/Users/trg16/Dev/Bricklayer2.0/masonry/masonry-state.json";
```

The `writeState()` utility in `masonry/src/core/state.js` writes to `{cwd}/masonry-state.json` (cwd-relative). The observe hook uses `writeState(cwd, ...)` which writes to the root-level file `C:/Users/trg16/Dev/Bricklayer2.0/masonry-state.json`.

These are **different files**:
- Approver reads: `masonry/masonry-state.json` (contains `mortar_consulted: true` and `mortar_session_id: "2026-03-22T12:00:00.000Z"`)
- Observe writes: `masonry-state.json` (root-level, contains verdict counts but no `mortar_consulted` field)

The approver's hardcoded path points to a file that was presumably hand-edited once (7+ days ago) and never automatically updated. The observe hook's state writes go to a completely different file. **Even if the receipt writer existed, it would need to write to the correct path** (`masonry/masonry-state.json`), not via `writeState()`.

### 3. The subagent exemption already solves the Mortar deadlock --- but not the receipt problem

`isSubagentContext()` at masonry-approver.js lines 140-144:
```js
function isSubagentContext() {
  return (
    process.env.CLAUDE_SUBAGENT === "1" ||
    process.env.MASONRY_SUBAGENT === "1"
  );
}
```

The `mortarGateApplicable` flag at line 275:
```js
const mortarGateApplicable =
  targetToolIsActionable && !isSubagentContext();
```

Agents spawned by Mortar (via the Agent tool) inherit the `CLAUDE_SUBAGENT=1` environment variable. This means the gate **never fires on Mortar-spawned subagents**. There is no deadlock risk for agents that Mortar dispatches --- they are always exempt.

However, the gate DOES fire on the **top-level Claude context** (the orchestrator). If Mortar itself is running as a subagent (spawned by the user typing a prompt that triggers Agent tool), then Mortar's own context has `CLAUDE_SUBAGENT=1` and is exempt. But Mortar cannot write, so the receipt never gets written, and the top-level context (which has Write/Edit) will see the gate fire.

### 4. The correct receipt writer location: masonry-prompt-router.js (UserPromptSubmit hook)

The masonry-prompt-router.js hook fires on every user prompt submission (line 148-215). It already:
- Detects intent (line 190)
- Injects routing hints into Claude's context (line 206-210)
- Has access to `fs.writeFileSync` (line 162)
- Fires before Claude processes the prompt, before any tool calls

This is the only hook that fires at the correct timing: **after the user submits a prompt, before Claude invokes any tools**. The sequence is strictly ordered:

1. UserPromptSubmit fires (masonry-prompt-router.js)
2. Claude processes the prompt and routing hint
3. Claude decides to invoke Mortar (Agent tool)
4. Mortar reads, routes, and returns
5. Claude processes Mortar's response
6. Claude invokes Write/Edit tools
7. PreToolUse fires (masonry-approver.js checks receipt)

Writing the receipt at step 1 means it is always available at step 7. There is no race.

### 5. The deadlock scenario under F2.3 hardened gate

If the gate is hardened (F2.3: deny instead of advisory), the deadlock occurs as follows:

**Scenario A --- No receipt writer, hardened gate:**
- User submits prompt
- Claude decides to Write a file (no Mortar consultation)
- PreToolUse fires, `isMortarConsulted()` returns false (stale/missing receipt)
- Gate denies the Write
- Claude receives denial, cannot complete the task
- No recovery path --- Claude has no way to write the receipt because it needs Write permission to do so
- **This is a true deadlock**: the Write is blocked because no receipt exists, and the receipt cannot be written because Write is blocked

**Scenario B --- Receipt writer in masonry-prompt-router.js, hardened gate:**
- User submits prompt
- masonry-prompt-router.js writes `mortar_consulted: true` + fresh `mortar_session_id` (hook write, not gated)
- Claude decides to Write a file
- PreToolUse fires, `isMortarConsulted()` returns true
- Gate allows the Write
- **No deadlock**: the hook bypasses the gate because hooks are Node processes writing directly to the filesystem, not Claude tool calls

Wait --- this defeats the purpose. If the prompt router always writes the receipt, then Mortar is never actually required. The receipt is supposed to confirm that Mortar was consulted, not that a prompt was submitted.

### 6. The correct two-phase design

The receipt must be written in two phases:

**Phase 1 (Reset)**: masonry-prompt-router.js sets `mortar_consulted: false` at every UserPromptSubmit. This invalidates any stale receipt from a prior turn.

**Phase 2 (Stamp)**: Something that fires after Mortar completes routing must set `mortar_consulted: true`. The candidates:
- (a) Mortar's agent instructions --- impossible, Mortar has no Write tool
- (b) A PostToolUse hook watching for Agent tool completions where `subagent_type === "mortar"` --- the masonry-subagent-tracker.js hook fires on SubagentStart, but there is no SubagentComplete hook in Claude Code
- (c) The top-level orchestrator writes the receipt after Mortar returns --- but this requires the orchestrator to have Write access, which is exactly what the gate blocks
- (d) **Add Bash to Mortar's tool list** and instruct it to `echo '...' > masonry-state.json` --- Bash is exempt from the gate (line 294: `if (toolName !== "bash")`)
- (e) **A new PostToolUse hook** that detects Mortar's routing output (e.g., the `[MORTAR]` log prefix in Agent tool output) and writes the receipt

Option (d) is the simplest and most reliable. Bash is already unconditionally exempt from the Mortar gate. Adding `Bash` to Mortar's tool list and adding a single instruction line causes Mortar to write the receipt through the one tool that is never gated.

Option (e) is cleaner architecturally but requires a new hook and a fragile heuristic (parsing Agent tool output for `[MORTAR]` prefix).

### 7. Per-session vs global singleton analysis

The current `MASONRY_STATE_PATH` is hardcoded to one absolute path. Tim runs multiple Claude sessions simultaneously ("casaclaude", "proxyclaude"). Two concurrent sessions would:
- Both reset `mortar_consulted: false` on prompt submit
- Both write `mortar_consulted: true` after Mortar routes
- Race on the same file, with the last writer winning

The `writeState()` function in state.js uses atomic write (write to `.tmp.${pid}` then rename), which prevents corruption but not logical races.

The 4-hour `MORTAR_SESSION_FRESHNESS_MS` window is appropriate for a per-session model but overly broad for per-turn. A per-turn model should use a shorter window (e.g., 5 minutes) or a nonce-based approach.

**Recommendation**: Use a per-session temp file at `os.tmpdir()/masonry-receipt-${sessionId}.json` instead of the global singleton. The session ID is available in the UserPromptSubmit input payload (used at masonry-session-start.js line 88). This eliminates cross-session races entirely and matches the existing pattern used for session snapshots (`masonry-snap-${sessionId}.json`).

## Analysis

### Root cause chain

1. D2.2 identified that a receipt writer is needed but said it should live "in Mortar's agent instructions or a dedicated hook"
2. Mortar's tool list (`Read, Glob, Grep, Agent, recall_search, recall_store`) does not include Write, Edit, or Bash
3. Therefore Mortar **cannot** write the receipt from its agent instructions --- the recommendation is structurally impossible
4. The only hook-based location that fires at the correct timing is masonry-prompt-router.js (UserPromptSubmit)
5. But a simple receipt write in the prompt router would make Mortar consultation meaningless (always-true receipt)
6. The correct design is a two-phase pattern: prompt router resets, Mortar stamps via Bash (the one gate-exempt tool)

### The complete write specification

**mortar.md changes** (both copies):
- Add `Bash` to the `tools:` frontmatter list
- Add the following instruction block after the "Session Mode Detection" section and before any dispatch:

```markdown
## Routing Receipt

After detecting mode and before dispatching to any agent, write the routing receipt:

\`\`\`bash
node -e "
const fs = require('fs');
const p = 'C:/Users/trg16/Dev/Bricklayer2.0/masonry/masonry-state.json';
const s = JSON.parse(fs.readFileSync(p, 'utf8'));
s.mortar_consulted = true;
s.mortar_session_id = new Date().toISOString();
fs.writeFileSync(p, JSON.stringify(s, null, 2));
"
\`\`\`

This confirms routing was performed. The PreToolUse gate checks this receipt.
Do this ONCE per invocation, immediately after mode detection.
```

**masonry-prompt-router.js changes**:
- After line 187 (after the active-campaign skip), add a receipt reset:

```js
// Reset Mortar receipt — ensures stale receipts from prior turns don't carry over
try {
  const receiptPath = "C:/Users/trg16/Dev/Bricklayer2.0/masonry/masonry-state.json";
  const state = JSON.parse(fs.readFileSync(receiptPath, "utf8"));
  state.mortar_consulted = false;
  fs.writeFileSync(receiptPath, JSON.stringify(state, null, 2), "utf8");
} catch {}
```

### Why Bash solves the deadlock

The masonry-approver.js gate at line 294 checks: `if (toolName !== "bash")`. Bash is unconditionally exempt from both the advisory warning and any future hard deny. When Mortar uses Bash to write the receipt:
- The Bash tool fires PreToolUse
- masonry-approver.js sees `toolName === "bash"` and skips the gate entirely (line 294)
- The receipt is written
- Subsequent Write/Edit calls from the orchestrator (after Mortar returns) see a fresh receipt
- No deadlock

### Why 4 hours is wrong for per-turn

The `MORTAR_SESSION_FRESHNESS_MS` of 4 hours was designed for a per-session model. In a per-turn model where the prompt router resets `mortar_consulted: false` on every prompt, the freshness window only needs to cover the time between Mortar's stamp and the first Write/Edit call within the same turn --- typically under 30 seconds. However, since the prompt router reset already invalidates stale receipts, the 4-hour window becomes a backstop that only matters if the prompt router fails. Leaving it at 4 hours is safe but unnecessary.

## Verdict

**DIAGNOSIS_COMPLETE**

The receipt writer cannot live in Mortar's agent instructions as D2.2 proposed because Mortar lacks Write/Edit/Bash tools. The correct architecture is: (1) add Bash to Mortar's tool list, (2) instruct Mortar to write the receipt via a Bash one-liner (Bash is gate-exempt), (3) reset the receipt in masonry-prompt-router.js on every UserPromptSubmit. This prevents the deadlock because the Bash tool is unconditionally exempt from the PreToolUse gate at masonry-approver.js line 294.
