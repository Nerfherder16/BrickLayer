## Evidence

### Current state of routing_keywords

**Registry**: 114 agents in `masonry/agent_registry.yml`. Only 8 have `routing_keywords` populated in the registry. 106 have empty/absent `routing_keywords`.

**Agent .md files**: 20 of 105 `.md` files in `~/.claude/agents/` already have `routing_keywords` in their YAML frontmatter. The registry only reflects 8 because those 20 agents were onboarded before the keywords were added to their frontmatter, and `onboard_agent.py` only processes NEW agents (not existing ones).

**File quality breakdown** (105 agent .md files):
- 81 have parseable YAML frontmatter
- 21 are empty stubs (0 bytes) -- no-frontmatter agents like `docker-specialist.md`, `postgres-specialist.md`
- 1 has malformed frontmatter (`developer.md` -- unescaped colon in description)
- Of the 81 parseable: 20 already have `routing_keywords`, 61 have descriptions but no keywords

### How routing_keywords are used today

**Deterministic layer (Layer 1)**: `masonry/src/routing/deterministic.py` lines 294-310 -- `_route_from_registry_keywords()` OR-joins an agent's keywords into a word-boundary regex and matches against request text. This runs BEFORE the 30+ hardcoded regex patterns (line 326 runs before line 331+). This is live, working infrastructure.

**Semantic layer (Layer 2)**: `masonry/src/routing/semantic.py` line 120 -- `_agent_corpus_key()` builds corpus from `agent.description + " " + ", ".join(agent.capabilities)`. Does NOT use `routing_keywords`. Keywords would not affect semantic routing.

### Extraction feasibility from existing content

Of the 61 agents with parseable frontmatter but no `routing_keywords`:
- 61 (100%) have a `description` field with 10+ chars
- 11 (18%) use "Activate when..." phrasing with explicit trigger phrases in description
- 20 (33%) have body sections mentioning "When to invoke", "Trigger", or "Activate when"
- All 61 have at minimum a `description` field sufficient for keyword extraction

**Description quality examples** (agents without keywords):
- `cascade-analyst`: "Activate when the user wants to know what breaks next -- 'if X fails, what does it take down?', 'map the failure cascade', 'what's the propagation risk?'"
- `benchmark-engineer`: "Activate when real performance measurements are needed -- 'benchmark this', 'measure latency/throughput'"
- `evolve-optimizer`: "Activate when the system is working and the user wants to make it better -- 'optimize this', 'improve performance'"

These descriptions contain **quoted trigger phrases** that are directly extractable as routing keywords.

### Sync gap: frontmatter vs registry

The `onboard_agent.py` script (line 166-170) correctly reads `routing_keywords` from frontmatter. But it only runs for NEW agents via `detect_new_agents()` (line 43-79), which skips agents already in the registry. The 20 agents that had keywords added to their frontmatter after initial onboarding were never re-synced.

The single-file hook path (line 552-566) DOES use `upsert_registry_entry` which would sync keywords if triggered. But it only fires on Write/Edit events to agent files -- it does not run as a batch backfill.

## Analysis

### Root causes (three distinct issues)

**Root cause 1 -- Sync gap**: 20 agents have `routing_keywords` in their `.md` frontmatter but the registry has stale entries without them. The `onboard_agent.py` script's `detect_new_agents()` function skips agents already registered. Fix: run a backfill pass using the single-file upsert path for all existing agents.

**Root cause 2 -- No auto-extraction**: 61 agents have rich descriptions but no `routing_keywords` in frontmatter. The `onboard_agent.py` script explicitly says "no keyword inference from body text" (line 3, line 144). There is no extraction mechanism. Fix: add a keyword extraction function that parses quoted phrases from descriptions and "When to invoke" body sections.

**Root cause 3 -- Empty stubs**: 21 agents are 0-byte files with no content to extract from. These are genuinely dark -- no description, no frontmatter, nothing. Fix: these need manual annotation or generation (outside scope of auto-extraction).

### Extraction mechanism design

For the 61 agents with descriptions, a three-strategy extraction covers 80%+:

1. **Quoted phrase extraction** (covers ~18% -- agents with "Activate when" + quoted examples): Regex `r"['\"]([^'\"]{5,50})['\"]"` on the description field yields trigger phrases like "benchmark this", "map the failure cascade".

2. **Description noun-phrase extraction** (covers ~73% -- all agents with descriptions): Extract action-oriented phrases from the description: "security audit", "vulnerability assessment", "code review". Use the existing `capabilities` list as a supplement -- these are already semi-structured keywords.

3. **Body section extraction** (covers ~33%): Parse "When to invoke", "Use for", "Trigger" sections in the markdown body for additional phrases.

Combined, strategies 1+2 cover all 61 agents with parseable frontmatter. Strategy 3 adds supplementary keywords for a third of them.

### False-match risk assessment

**Current baseline**: 86% dark fleet (98 agents unreachable). Any false-positive routing to a wrong agent is strictly better than no routing at all, because the current state is 0% reachability for those agents.

**Keyword-based false matches**: The deterministic layer uses word-boundary regex (`\b`), so "benchmark" won't match "benchmarking" substring issues. The main risk is keyword collision -- two agents sharing the same keyword. Mitigation: the deterministic layer returns the FIRST match (line 303-309 iterates registry order), so more specific agents should be listed before general ones.

**Semantic layer impact**: Zero. `routing_keywords` are not used by the semantic layer (which uses `description + capabilities`).

**Estimated false-match rate**: For auto-extracted keywords from descriptions, approximately 5-10% of routes may hit a suboptimal agent. This is acceptable against the 86% dark-fleet baseline. Manual review of the top 20 most-used keywords post-extraction would reduce this to <2%.
