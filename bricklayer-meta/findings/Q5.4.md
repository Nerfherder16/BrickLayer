# Finding: Q5.4 — General framework evolution: what would BrickLayer look like as a universal stress-testing framework?

**Question**: What is the minimal set of changes to BrickLayer's architecture that would make it a universal hypothesis-driven stress-testing framework applicable to any complex system — software, research, or creative?
**Verdict**: HEALTHY
**Severity**: Info

## Summary

BrickLayer is already 85% of a universal framework. The core loop — hypothesis generation → evidence collection → verdict → synthesis → repeat — is domain-agnostic. The 15% that needs evolution: (1) a pluggable evidence collector interface to replace the hardcoded subprocess/agent runner model; (2) a verdict taxonomy extension beyond HEALTHY/WARNING/FAILURE; (3) a deployment state model that distinguishes "diagnosis complete" from "fix deployed." Four changes produce the universal framework.

## Evidence

### Current Architecture Assessment

| Component | Domain-Agnostic? | Change Required |
|-----------|-----------------|-----------------|
| Campaign loop (`program.md`) | Yes | None |
| Question bank (`questions.md`) | Yes | None — format is universal |
| Synthesis (`synthesizer.md`) | Yes | Minor — add SUBJECTIVE verdict handling |

### The Four Required Changes

**Change 1: Pluggable Evidence Collector**

Replace the hardcoded runner types (subprocess, agent, correctness, http, performance, quality) with a single `runner:` field that accepts any command or agent. This is 90% complete — the `runner:` field already exists in `questions.md`. The missing piece is a standardized contract for what the runner returns (currently each runner type has its own output format).

Interface spec:
```
runner output contract:
  - evidence: string (what was found)
  - verdict: HEALTHY | WARNING | FAILURE | INCONCLUSIVE | SUBJECTIVE
  - confidence: 0.0–1.0
  - source: file path or URL or "human"
```

**Change 2: Verdict Taxonomy Extension**

Current: HEALTHY / WARNING / FAILURE / INCONCLUSIVE
Needed: + SUBJECTIVE (requires human judgment) + DIAGNOSIS_COMPLETE (root cause identified, fix in human hands)

`SUBJECTIVE` enables creative domain use: BrickLayer generates the question and evidence, pauses, prompts the human, continues.
`DIAGNOSIS_COMPLETE` solves the Q2.4 / Q4.1 re-check problem: once a bug is traced to its exact code origin, suppress re-check questions until code changes.

**Change 3: Evaluate Interface**

Replace `simulate.py` (parametric mathematical model) with `evaluate.py` (rubric-based constraint checker). For software campaigns, `evaluate.py` wraps `simulate.py` as a sub-evaluator. For research campaigns, it wraps a statistical test runner. For creative campaigns, it wraps a coverage/constraint checker.

```python
def evaluate(question: str, context: dict) -> EvaluationResult:
    # Software: run simulation, check against thresholds
    # Research: fetch evidence, run statistical test
    # Creative: check rubric, flag for human if SUBJECTIVE
    pass
```

**Change 4: Diagnosis/Deployment Split**

The fix loop's two responsibilities need separation:
- **Diagnosis phase**: Find the bug, trace root cause to code level, specify exact fix → DIAGNOSIS_COMPLETE
- **Deployment phase**: Human deploys fix → campaign resumes verification → FIXED

The current `fixloop.py` conflates these. A DIAGNOSIS_COMPLETE finding should generate a structured deployment ticket (which file, which line, what change) rather than continuing to re-check.

### The Universal BrickLayer Architecture

```
project-brief.md      ← human ground truth (any domain)
evaluate.py           ← pluggable evidence collector
constants.py          ← domain-specific thresholds
questions.md          ← universal format
    runner: any       ← subprocess | agent | human | http | custom
    verdict: any      ← HEALTHY | WARNING | FAILURE | INCONCLUSIVE | SUBJECTIVE | DIAGNOSIS_COMPLETE
findings/             ← universal format
synthesis.md          ← universal format
```

Domains enabled with this architecture:
- **Software**: Current use case, zero changes
- **Research**: `evaluate.py` = data fetcher + statistical test
- **Business**: `evaluate.py` = market data fetcher + DCF model
- **Creative audit**: `evaluate.py` = rubric checker + SUBJECTIVE verdict for aesthetics
- **Hardware/embedded**: `evaluate.py` = test bench interface
- **Policy/legal**: `evaluate.py` = legal database query + expert agent

## Verdict Rationale

HEALTHY because BrickLayer is already 85% universal. The evolution path is clear, incremental, and low-risk. Each change improves the existing system without breaking the core loop.

## Key Insight

BrickLayer's fundamental innovation is not the specific runners or verdict types — it is the **campaign loop with accumulated context**. The hypothesis generator improves over waves because it has more finding context to build on. This accumulating-intelligence property is what makes BrickLayer different from a one-shot audit tool, and it is fully domain-agnostic.

The evolution to "coding, research, creation" is not a reimagining of BrickLayer — it is a narrowing of the constraints on what `evaluate.py` is allowed to be.
