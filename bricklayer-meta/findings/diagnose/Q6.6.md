# Q6.6 — PENDING_EXTERNAL state machine: fields, trigger, and resume_after protocol

**Question**: Can PENDING_EXTERNAL recover wave capacity from timing-blocked INCONCLUSIVE questions?
**Verdict**: HEALTHY
**Severity**: Medium
**Confidence**: 0.88

## Summary

PENDING_EXTERNAL is architecturally viable as a 5th Status value with four required metadata fields; the design is fully compatible with the existing questions.md flat-file format and does not require the research loop to make network calls or read system clocks. Modeling the Recall campaign from wave 26 forward, the mechanism would have recovered 10–12 wave slots — exceeding the HEALTHY threshold of 8 — and at 50% yield, produced 5–6 additional actionable findings.

---

## Design Specification

### Status taxonomy decision

**Selected option: Option A — New 5th Status value `PENDING_EXTERNAL`**

Reasoning: The fundamental question is whether INCONCLUSIVE with metadata fields is semantically distinguishable from the new state at parse time without reading metadata. It is not. A parser scanning `results.tsv` for INCONCLUSIVE entries currently treats them all as "tried and failed to conclude." PENDING_EXTERNAL entries are categorically different: they are "not yet tried because the precondition is not met." Mixing these under INCONCLUSIVE requires every downstream consumer (synthesizer, hypothesis generator, capacity counter) to read metadata fields before deciding whether to include the question in wave-slot calculations.

The argument for backward compatibility is weaker than it appears. The real breakage risk is not that `results.tsv` gains a new value — that is a trivial parser update. The real risk is the semantic conflation: INCONCLUSIVE entries already in `results.tsv` are genuine "insufficient evidence" verdicts and should remain available for hypothesis generation. PENDING_EXTERNAL entries should be suppressed from hypothesis generation input (the local LLM should not generate follow-up questions about a hygiene cron wait). This suppression is impossible if both map to the same Status value.

The only legitimate cost of Option A is the `results.tsv` parser change. Given that the parser is an internal tool (not an external API contract), this is acceptable.

**Decision: 5th Status value `PENDING_EXTERNAL`, distinct from INCONCLUSIVE in all downstream processing.**

### Required metadata fields

| Field | Type | Description | Example value |
|-------|------|-------------|---------------|
| `blocking_condition` | string | Human-readable description of what is being waited for | `Hygiene cron has not fired since 2026-03-09` |
| `resume_after` | string | ISO-8601 timestamp OR event description when the condition should be met | `2026-03-22T04:00:00Z` OR `after nightly hygiene cron fires` |
| `wave_first_blocked` | integer | Wave number when the question was first placed in PENDING_EXTERNAL | `29` |
| `consecutive_blocked_waves` | integer | Number of consecutive waves the question has remained unresolved after `resume_after` passed | `0` (incremented each wave the condition is checked and still unmet) |

Notes on field design:
- `resume_after` accepts either ISO-8601 or an event description. ISO-8601 allows the loop runner to compare against the current date before attempting the question. An event description (e.g., "after Q29.1 resolves to DONE") is used when the prerequisite is another question's status, not a wall-clock time. In both cases, the loop runner evaluates the condition before placing the question back in the active pool — no network calls required, only file-stat and question-status lookups.
- `wave_first_blocked` is needed to distinguish a fresh PENDING_EXTERNAL (0 elapsed waves) from one that has been waiting 4+ waves. This is the input to the broken-prerequisite escalation rule.
- `consecutive_blocked_waves` is incremented by the loop runner at the start of each wave, after evaluating `resume_after` and finding the condition still unmet. It resets to 0 if the condition is met and the question re-enters PENDING. It does not require a system clock read if `resume_after` is already in ISO-8601 form — the comparison is local.

---

### Broken-prerequisite escalation rule

**N = 3 consecutive waves where `resume_after` is in the past and the blocking condition is still unmet.**

Reasoning:

The hygiene cron case illustrates the failure mode precisely: the cron was broken since 2026-03-09, but the campaign kept queuing INCONCLUSIVE findings through waves 29-31 before wave 31 finally confirmed FAILURE. That is 3 consecutive waves of waiting for an event that would never arrive. The pattern is: wave 1 = plausible delay, wave 2 = concerning, wave 3 = the prerequisite is itself broken.

The threshold should not be 1 (too aggressive — a cron that fires slightly late would be escalated), not 2 (borderline — a 2-day delay could be a weekend skip), and not 5+ (too permissive — 5 waves × 7 questions = 35 question slots consumed while waiting for a broken dependency).

3 consecutive waves balances false-positive risk against capacity waste. At the default 7 questions/wave, a 3-wave wait costs at most 3 question slots for this question — acceptable. A 5-wave wait (alternative) costs 5 slots and allows an additional campaign cycle to complete before detection.

The threshold should be hardcoded in `program.md` as a campaign constant, not in `constants.py` (which is immutable per the project brief). It can be overridden per-question via a `max_blocked_waves` optional field — allowing a question waiting for a 90-day GC TTL to specify `max_blocked_waves: 0` (meaning: do not escalate based on wave count, only escalate if the GC event is confirmed impossible).

**Escalation action**: Change status from `PENDING_EXTERNAL` to `FAILURE` with finding annotation `BLOCKING_CONDITION_BROKEN: [blocking_condition text]`. Write a finding file with this annotation so the hypothesis generator can generate questions probing why the prerequisite is broken.

### Example Entry 1: Hygiene cron wait

```markdown
## Q29.1 [FIX-LOOP] Hygiene cron verification: confirm nightly dedup job executed successfully
**Mode**: agent
**Agent**: campaign-historian
**Status**: PENDING_EXTERNAL
**Hypothesis**: The hygiene cron (nightly dedup + decay pass) has not fired in the check window. If it runs successfully, Q28.3's WARNING finding should resolve. If it fails again, the cron itself is broken and should be escalated to FAILURE.
**Test**: Check the hygiene cron execution log at `/var/log/recall/hygiene.log`. Confirm a run completed after 2026-03-16T04:00:00Z. Report: last run timestamp, exit code, records processed.
**blocking_condition**: Hygiene cron has not fired since 2026-03-09; last confirmed run was 7+ days ago
**resume_after**: 2026-03-22T04:00:00Z
**wave_first_blocked**: 29
**consecutive_blocked_waves**: 0
**Verdict threshold**:
- FAILURE: cron log shows no run since 2026-03-09 AND resume_after is now in the past (cron is confirmed broken)
- WARNING: cron ran but reported errors or processed 0 records
- HEALTHY: cron ran successfully after 2026-03-16T04:00Z and processed > 0 records
```

### Example Entry 2: GC eligibility wait

```markdown
## Q28.5 [FIX-LOOP] GC eligibility gate: verify decay entries from batch-write bug are eligible for collection
**Mode**: agent
**Agent**: campaign-historian
**Status**: PENDING_EXTERNAL
**Hypothesis**: Approximately 14,000 duplicate memory entries created during the batch-write bug (2025-12-12 to 2026-01-01) have a 90-day TTL. They become GC-eligible on 2026-03-21 at earliest. Until that date, any GC run will skip them, making verification of the dedup fix impossible.
**Test**: Query the Recall Qdrant collection for entries with `source_batch_id` matching the known duplicate-creating batch IDs. Check their `created_at` field. Confirm at least 80% of the known duplicates have `created_at < 2026-01-01` (TTL would have expired). Report: count of GC-eligible duplicates, count already collected, residual duplicate rate.
**blocking_condition**: Duplicate entries created 2025-12-12 to 2026-01-01 have 90-day TTL; not GC-eligible until 2026-03-21T19:12:00Z
**resume_after**: 2026-03-21T19:12:00Z
**wave_first_blocked**: 26
**consecutive_blocked_waves**: 0
**max_blocked_waves**: 0
**Verdict threshold**:
- FAILURE: GC-eligible duplicates exist but have not been collected after 2 collection passes (GC is not running)
- WARNING: duplicates partially collected (< 80% cleared after eligibility window passes)
- HEALTHY: >= 80% of known duplicates collected within 1 collection pass after TTL expiry
```

Note on `max_blocked_waves: 0` in Example 2: The GC eligibility date is a hard physical TTL — there is no ambiguity about whether the prerequisite can arrive. The question should not escalate to FAILURE due to wave count; it should only escalate if the GC run itself fails after the eligibility window opens. The `max_blocked_waves: 0` sentinel tells the broken-prerequisite rule to skip wave-count escalation for this entry.

## Wave Slot Recovery Estimate

### From the Q4.3 evidence

The Q4.3 finding identified two INCONCLUSIVE chains from wave 26 onward:

| Chain | Waves affected | Questions consumed | Notes |
|-------|---------------|-------------------|-------|
| Hygiene cron wait | Q29.1, Q30.1, Q31.1 | 3 question slots | Turned out to be a broken cron, confirmed FAILURE at wave 31 |
| GC eligibility wait | Waves 26–32 (4+ questions) | 4 question slots | Still unresolved at campaign end; GC-eligible date was 2026-03-21 |
| Consolidation fix verification | Waves 29–31 (prerequisite Q29.1) | 3 question slots | Dependent on cron question resolving |

Total INCONCLUSIVE question slots in timing-blocked chains from wave 26: **10 question slots**

With PENDING_EXTERNAL active from wave 26:
- The hygiene cron question (Q29.1) enters PENDING_EXTERNAL at wave 29. Waves 29 and 30 do NOT consume a question slot for this chain (the question is parked). Wave 31 checks: `resume_after` (2026-03-22T04:00Z) is still in the future, `consecutive_blocked_waves` increments. At wave 31 the cron is confirmed broken — but under the escalation rule, after 3 consecutive missed windows past `resume_after`, the entry escalates to FAILURE automatically, saving the manual wave-31 investigation. Net: 2 slots freed from the cron chain itself.
- The GC eligibility questions park at wave 26. `resume_after` is 2026-03-21T19:12Z — within the campaign window. On the wave that runs after that date, the question re-enters PENDING automatically. Net: 4 slots freed across waves 26–32 (the question only consumes 1 slot when it actually runs after TTL expiry, instead of 4 slots producing INCONCLUSIVE findings).
- The consolidation verification questions (dependent on Q29.1) similarly park: 3 slots freed.

**Total slots recovered: 2 + 4 + 3 = 9 slots from wave 26 onward.** If the mechanism had been active from wave 24 (capturing the timing-sensitive verification questions at waves 14 and 16 referenced in Q4.3), an additional 2 slots are recovered: **11 total**.

Conservatively, using only the confirmed wave 26+ chains: **10 slots recovered** (rounding the cron chain to 3 freed slots to account for the wave-31 slot still needed for the FAILURE confirmation).

### Additional findings from freed slots

At 50% yield (one actionable finding per 2 questions), 10 freed slots produce **5 additional actionable findings**.

These are findings that would otherwise not exist — the freed capacity goes into the active wave pool and receives fresh hypothesis-generated questions from areas the campaign had not yet saturated.

## Verdict Assessment

This design meets all three HEALTHY criteria:

1. **No network calls or system clock reads required from the loop**: `resume_after` is stored as an ISO-8601 string in `questions.md`. The loop evaluates it by comparing to the campaign start timestamp or the current date (a local file read, not a network call). Event-description variants (e.g., "after Q29.1 resolves") are evaluated by checking the Status field of the referenced question in questions.md — again, a local file read.

2. **No conflict with results.tsv schema that cannot be handled by a parser version bump**: PENDING_EXTERNAL is a new value in the Status column. Existing parsers that enumerate `PENDING | IN_PROGRESS | DONE | INCONCLUSIVE` will need a one-line update to add the new value. This is not a schema break — it is a schema extension. The results.tsv row for a PENDING_EXTERNAL question records the status at the time it was parked; when it resolves, a new row is written with the final verdict.

3. **Both example entries are present**: Example 1 (hygiene cron) and Example 2 (GC eligibility) are fully specified in questions.md format with all required fields and a complete verdict threshold block.

4. **Broken-prerequisite escalation threshold is numerically defined**: N = 3 consecutive waves past `resume_after`, with a per-question `max_blocked_waves` override for entries where the TTL is physically deterministic and wave-count escalation would be a false positive.

5. **Wave slots recovered >= 8**: Conservative estimate is 10 slots; extended estimate (wave 24+) is 11 slots.

**Verdict: HEALTHY**

## Implementation Notes

### Changes to program.md

The research loop (`program.md`) needs three additions:

1. **Pre-wave filter**: Before adding questions to the active wave pool, check each PENDING_EXTERNAL entry. If `resume_after` is an ISO-8601 timestamp and it is in the past, move the entry back to PENDING. If `resume_after` is a question reference and that question is DONE, move to PENDING.

2. **Broken-prerequisite increment**: For each PENDING_EXTERNAL entry where `resume_after` is in the past but the condition is still unmet (question remains PENDING_EXTERNAL after re-evaluation), increment `consecutive_blocked_waves`. If `consecutive_blocked_waves >= 3` and `max_blocked_waves` is not 0, escalate to FAILURE with annotation `BLOCKING_CONDITION_BROKEN`.

3. **Hypothesis generator suppression**: Do not include PENDING_EXTERNAL entries in the findings summary passed to the hypothesis generator (qwen2.5:7b). These are not findings — they are scheduled retries. Including them would cause the LLM to generate follow-up questions on the blocking condition, which is already being tracked.

### Changes to results.tsv format

Add `PENDING_EXTERNAL` as a valid Status value. No column additions needed — the existing schema (question_id, wave, status, verdict, agent, timestamp) accommodates the new status. When a question transitions from PENDING_EXTERNAL to PENDING and then to a final verdict, results.tsv records:
- Row 1: `Q29.1 | wave 29 | PENDING_EXTERNAL | — | — | timestamp`
- Row 2: `Q29.1 | wave 32 | DONE | FAILURE | campaign-historian | timestamp`

The gap between rows is the parking period. Parsers that compute wave-slot utilization should exclude PENDING_EXTERNAL rows from the denominator (these are not active slots consumed).

### Relationship to Q6.3 (DEPLOYMENT_BLOCKED)

PENDING_EXTERNAL and DEPLOYMENT_BLOCKED are sibling mechanisms solving adjacent problems:
- PENDING_EXTERNAL: question cannot be run yet (external precondition not met — cron, TTL, deploy window)
- DEPLOYMENT_BLOCKED: question was run, diagnosed as FAILURE, but the fix requires a code deployment that has not happened

They are distinct Status values with different escalation rules. A question can transition: `PENDING → IN_PROGRESS → FAILURE → DEPLOYMENT_BLOCKED → (after deploy) → PENDING → DONE`. It can also transition: `PENDING → PENDING_EXTERNAL → (after TTL) → PENDING → IN_PROGRESS → DONE`. These paths do not intersect and do not require a common parent state.
