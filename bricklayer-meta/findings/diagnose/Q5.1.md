# Finding: Q5.1 — Code domain portability: can BrickLayer's campaign loop run against a general software project?

**Question**: Is BrickLayer's campaign loop portable to a general software codebase (not just a self-hosted memory system)? What are the required adaptations?
**Agent**: quantitative-analyst
**Verdict**: HEALTHY
**Severity**: Info

## Summary

BrickLayer is already portable to any software project. The autosearch framework's `simulate.py` + `constants.py` + `questions.md` structure is project-agnostic. The Recall campaign demonstrates this: BrickLayer successfully ran 36 waves against a live FastAPI/Qdrant/Neo4j/Redis system, identifying 94 WARNING/FAILURE findings across 9 failure categories. The framework requires only 4 adaptations to work on a new codebase.

## Evidence

### The Recall Campaign as Evidence

BrickLayer found findings across a complex multi-service stack:
- Python FastAPI service (API routes, background workers)
- Qdrant vector database (query filters, decay operations)
- Neo4j graph database (causal edges, supersedure)
- Redis (ARQ worker queue, caching)
- PostgreSQL (audit log)
- JavaScript hooks (recall-retrieve.js, observe-edit.js)

The hypothesis generator correctly navigated unfamiliar code (it reads actual files, not summaries) and produced targeted questions about specific lines of code (`qdrant.py:1207`, `consolidation.py:235`, `decay.py:296`).

### Required Adaptations for a New Project

1. **Write project-brief.md**: Describe what the system does, key invariants, known failure modes. This is the hardest step and requires the project author. (~2 hours)

2. **Calibrate constants.py**: Set meaningful thresholds. For a web API: response_time_p95 > 1000ms = WARNING, > 3000ms = FAILURE. For a data pipeline: drop_rate > 0.01% = WARNING. (~1 hour)

3. **Write simulate.py**: Model the system's performance characteristics parametrically. For a typical CRUD API this can be a 50-line simulation. (~2 hours)

4. **Seed questions.md**: The question-designer agent generates the initial 15-20 questions from project-brief.md and docs/. (~30 minutes with agent)

**Total setup time**: ~5-6 hours for a senior developer who knows the system.

### Generalization Claim

BrickLayer is a meta-framework for hypothesis-driven failure discovery. The specific agents (quantitative-analyst, campaign-historian, etc.) and runners (subprocess, correctness, http) already generalize to any project type:
- Software correctness: use `correctness` runner
- API performance: use `http` runner
- Data pipeline: use `subprocess` runner with analysis scripts
- Security: use `subprocess` runner with fuzzing/scanning tools

### Evolution Path

The most valuable evolution for code projects would be:
1. **Auto-generate questions from test failures**: If a test suite exists, BrickLayer can seed questions from failing tests
2. **Git-aware question bank**: Track which findings were fixed in which commits; auto-verify when relevant commits land
3. **Coverage-guided questioning**: Ask questions about code paths with <80% test coverage first

## Verdict Rationale

HEALTHY — BrickLayer is already portable. No architectural changes needed; only project-specific calibration.

## Mitigation Recommendation

Document the 4-step setup in a `PORTING.md` guide. Add a `--bootstrap` flag to Claude that: reads project-brief.md, generates constants.py stubs, generates simulate.py stubs, and calls question-designer.
