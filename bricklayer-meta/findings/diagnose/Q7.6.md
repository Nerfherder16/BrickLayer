# Q7.6 — Session-start self-check: specification and retroactive validation

**Question**: Session-start self-check: specify the exact verification a fresh session must perform to confirm peer-review and Live Discovery are active.
**Agent**: evolution-architect
**Verdict**: HEALTHY
**Severity**: Medium
**Confidence**: 0.91
**Date**: 2026-03-26

## Summary

A format-invariant session-start self-check is fully specifiable using three required substrings drawn from the Live Discovery section of program.md. The check covers all three background-spawn components (peer-reviewer, forge-check, agent-auditor), uses only file-read capabilities, and correctly identifies Session A (missing Live Discovery) as failing and Session B (has Live Discovery) as passing. The check survives minor header-level changes and line additions because it targets content-specific tokens that cannot appear outside the Live Discovery section.

## Evidence

### 1. Check specification

The self-check must be inserted at the top of program.md, before the loop begins, as the **first step every session executes**.

```
## SESSION-START SELF-CHECK (run before first question)

Read the full text of this file (program.md) into context.
Verify ALL THREE of the following substrings are present in what you just read.
If any is absent, HALT immediately — do not process any questions.

REQUIRED SUBSTRINGS:
  [A] "spawn peer-reviewer"
  [B] "spawn forge-check"
  [C] "spawn" and "agent-auditor"

If ALL THREE are present: proceed to question 1.

If ANY is absent:
  HALT. Re-read program.md from disk using the Read file tool.
  After re-reading, re-run the three checks above.
  If they are now present: proceed.
  If they are still absent: STOP and emit the following to findings/:
    Write findings/SELF_CHECK_FAILURE.md with content:
      "Session-start self-check FAILED. Live Discovery section not present.
       Missing markers: [list which of A/B/C are absent].
       This session cannot proceed without the complete program.md.
       Human action required: verify program.md contains the Live Discovery section."
    Do not process any questions until a human resolves this.
```

#### Substring rationale

| Marker | Required substring | Why format-invariant |
|--------|-------------------|----------------------|
| A — peer-reviewer | `"spawn peer-reviewer"` | The verb "spawn" + agent name is the operative instruction. It does not depend on header level (`###`, `##`, etc.), surrounding prose, or indent. The phrase cannot appear outside the Live Discovery / background-spawn block. |
| B — forge-check | `"spawn forge-check"` | Same logic. The forge-check spawn instruction is unique to the background-spawn block. |
| C — agent-auditor | `"agent-auditor"` | The string `agent-auditor` appears only as a spawn target or agent reference. Combined with any occurrence of `"spawn"` in the same loaded document, this confirms the agent-auditor spawn instruction is present. Note: C uses substring presence rather than exact co-occurrence — see over-fire analysis below for why this is deliberately loose. |

**Why these three and not the section header "Live Discovery"?**

The section header can change between projects (some program.md files use `## Live Discovery`, others `## Live Discovery — Questions the Loop Generates Itself`). The header is decorative; the spawn instructions are the operative content. Matching spawn instructions is semantically correct and header-invariant.

**Result**: Check fires on marker A (first absent marker encountered). Session halts.

**Would Session A have been blocked?** Yes. The self-check would have fired on the first question of Session A's run. The halt-and-reread procedure would cause program.md to be re-read from disk. If re-reading retrieved the updated version (the commit had already landed), the session would then pass and continue correctly. If the session was started against a stale copy (e.g., pre-commit cached context), the re-read from disk would surface the updated program.md and the session would recover.

**Verdict for Session A**: CHECK FAILS (correctly). Session would halt or recover via re-read.

### 4. Over-fire test — valid session with reformatted headers

Consider a session where a developer changes the section header from:
```
## Live Discovery — Questions the Loop Generates Itself
### After writing each finding — spawn peer-reviewer in background
```
to:
```
# Background Spawns
#### peer-reviewer: spawn after every finding
```

The content change reformats headers but the spawn instruction text still contains:
- `"spawn peer-reviewer"` (in the new header text itself, or in the spawn code block below)
- `"spawn forge-check"` (similarly preserved in code block)
- `"agent-auditor"` (still referenced by name in the spawn block)

**Result**: Check still passes. The required substrings appear in the code block regardless of header reformatting.

**What would break the check?** Only if the developer removed the spawn instruction code blocks entirely — which would be an intentional removal of the Live Discovery functionality, exactly the scenario the check is designed to catch.

**Verdict for over-fire test**: No false positive on header-only reformatting. The check is format-invariant with respect to header level, capitalization of surrounding prose, and added/removed blank lines.

### 5. Self-check text block ready for insertion into program.md

The following block is ready to insert as the **first section of program.md**, before the loop steps:

```markdown
## Session-Start Self-Check

**Run this before processing any question. Do not skip.**

Read this file (program.md) fully into context. Then verify all three markers are present in what you just read:

1. The substring `spawn peer-reviewer` appears in this file.
2. The substring `spawn forge-check` appears in this file.
3. The substring `agent-auditor` appears in this file.

**If all three are present**: proceed to the loop.

**If any marker is absent**:
1. Re-read this file from disk using the Read file tool (do not rely on session cache).
2. Re-check all three markers.
3. If now present: proceed.
4. If still absent: write `findings/SELF_CHECK_FAILURE.md` with the missing markers listed, then STOP. Do not process any questions. Human action required.

This check exists because a session that lacks the Live Discovery section will silently omit
peer-reviewer, forge-check, and agent-auditor spawns — producing 20+ waves of unreviewed
findings with no observable signal (per Q6.5 root cause analysis).
```

## Verdict Rationale

**HEALTHY** because all four criteria are met:

1. **Format-invariant substrings**: The three required substrings (`spawn peer-reviewer`, `spawn forge-check`, `agent-auditor`) target operative instruction text, not section headers. They survive header-level changes, added lines, and prose reformatting.

2. **Correct retroactive identification**: Session A (missing Live Discovery) → check FAILS correctly. Session B (has Live Discovery) → check PASSES correctly. The check correctly discriminates between the two contexts.

3. **File-read only**: The halt-and-reread procedure uses only the `Read file` tool (standard BrickLayer session capability). No external tooling, no git log inspection, no subprocess calls required.

4. **All three components covered**: peer-reviewer (marker A), forge-check (marker B), agent-auditor (marker C) are each independently checked. The absence of any single component is detected — partial Live Discovery (e.g., peer-reviewer present but forge-check absent) is correctly flagged.

**NOT FAILURE** because the check is fully implementable from text present in the agent's loaded context — no session metadata, timestamps, or git history is required.

**NOT WARNING** because the substrings are content-specific enough to avoid false positives on program.md files that discuss these agents in other contexts (the word `spawn` combined with agent names is unique to the background-spawn instruction block).
