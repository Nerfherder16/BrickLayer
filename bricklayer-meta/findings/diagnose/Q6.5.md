# Q6.5 — Peer review collapse root cause

**Question**: Why did peer review stop after wave 16 in the Recall campaign?
**Agent**: synthesizer-bl2
**Verdict**: WARNING
**Severity**: Medium
**Confidence**: 0.82

## Summary

Peer review stopped after wave 16 due to a session boundary, not a code bug or capacity sentinel. The peer-reviewer agent and program.md spawn instruction were added on 2026-03-13, but waves 14–21 ran in a continuous session on 2026-03-14 whose loop had already started before the parallel execution section was added. Wave 13 (which has 23 peer-reviewed findings) ran in a fresh session on 2026-03-15 morning that picked up the updated program.md. This is Root Cause B (configuration change / session context), not A (code bug) or C (capacity sentinel). No auto-disable logic exists in the BL source.

## Root Cause Analysis

| Root Cause | Evidence For | Evidence Against |
|-----------|--------------|-----------------|
| **A: Code bug in async spawn** | No code exists to have a bug — peer reviewer spawning is a prompt instruction, not Python | BL source has no peer-reviewer spawn code; no async spawn mechanism to fail |
| **B: Config change / session context** | Session A (Q15-Q19) ran before or without re-reading the updated program.md; Session B (Q13 follow-ups) correctly spawned peer reviewers; Q37.1 shows fresh sessions work correctly | Why did Session A not pick up the instruction? Could be session pre-dated the commit, or mid-session loop doesn't reload instructions |
| **C: Capacity sentinel auto-disabled it** | No evidence; no INCONCLUSIVE-count threshold appears anywhere in program.md or bl/ source | program.md explicitly says spawn after EVERY finding with no threshold condition; bl/ has no sentinel logic; no sentinel file exists |

**Most likely root cause: B — Session context / instruction gap**

**Supporting evidence**:
1. The peer-reviewer instruction was added to `program.md` on 2026-03-13 (`6882c0e`)
2. Session A (which ran waves 14-21) has execution timestamps starting 2026-03-14T00:00Z — it either started before the commit landed or did not re-read program.md after the update
3. Session B (wave 13 follow-ups, run 2026-03-15 morning) started fresh, read the current program.md, and correctly spawned peer reviewers for all 15+ findings
4. Q37.1 (2026-03-17 session) has peer review — later sessions work correctly
5. No auto-disable logic exists anywhere in the codebase
6. Absence of forge-check and agent-auditor outputs for Session A confirms the entire Live Discovery section was not being followed — it's not just peer review that was missing, it's the whole background spawn pattern

The root cause is a session that started before or without the complete program.md instruction set. Every subsequent fresh session that read the updated program.md correctly spawned peer reviewers.

## Verdict Assessment

This is **WARNING** (not FAILURE) because the root cause is identified with strong supporting evidence, and the fix is clear. It is not FAILURE because:
- The mechanism itself works (Sessions B and Wave 37 confirm this)
- The BL architecture does not have a persistent bug
- Only Session A ran without peer review; all subsequent fresh sessions have been correct

It is not HEALTHY because:
- The gap affected 20+ waves of findings (waves 17-36) with no peer review coverage
- No safeguard exists to detect a session running without the Live Discovery section active
- The session boundary root cause means any new session started against an out-of-date copy of program.md would silently omit peer review with no observable signal

## Recommended Fix

1. **Add a session-start self-check to program.md**: At the beginning of each session, the loop should verify it has read the current version of program.md by checking for the `## After writing each finding` section. If the section is absent from the loaded instructions, halt and re-read.

2. **Add a peer-review coverage metric to wave-end reporting**: At the end of each wave, count findings without a `## Peer Review` section. If >50% of non-INCONCLUSIVE findings lack peer review, emit a WARNING to the findings directory.

3. **Document the session isolation requirement more prominently**: The `DISABLE_OMC=1` requirement was added to program.md on 2026-03-14, but session isolation for BL's own internal spawn instructions (forge-check, agent-auditor, peer-reviewer) is equally critical and should be validated at session start.

4. **No BL source change needed**: The peer-reviewer mechanism works correctly in fresh sessions. The fix is process-level (always re-read program.md at session start) not code-level.
