## Process Specification: All Seven Questions

### 1. Trigger Condition — When Does a Finding Get SUBJECTIVE Verdict?

A finding receives SUBJECTIVE verdict when **all three** of the following hold:

- The question requires qualitative judgment that cannot be resolved by running `simulate.py` or checking a computable threshold
- The agent assigned to the question cannot produce a HEALTHY/WARNING/FAILURE verdict with confidence > 0.50
- The evidence gathered is sufficient to frame the judgment but not sufficient to resolve it algorithmically

Concrete examples: "Is this synthesis recommendation actionable?" / "Does this agent's output quality meet bar?" / "Is this finding severity proportionate to the evidence?" These are judgment calls — they have an answer but it requires a human to assign it.

Non-examples: any question with a computable pass/fail criterion (yield threshold, rate ceiling, timing measurement) — those get INCONCLUSIVE if the agent runs out of evidence, not SUBJECTIVE.

**The distinction from INCONCLUSIVE**: INCONCLUSIVE means "insufficient evidence to decide." SUBJECTIVE means "sufficient evidence gathered, but only a human can weigh it." SUBJECTIVE is not a cop-out for low-confidence; it is a routing signal.

---

### 2. Who Reviews and What Is the Review Action?

In a solo campaign, Tim is the reviewer. The review action is:

1. Read the finding file at `findings/<question_id>.md`
2. Decide: RESOLVED (becomes HEALTHY/WARNING/FAILURE — pick one) or SKIP (remains open)
3. If RESOLVED: edit the finding file — change `**Verdict**: SUBJECTIVE` to the actual verdict, add a `## Human Resolution` section with 1-3 sentences explaining the judgment, and update the question status in `questions.md` to DONE

The review takes < 5 minutes per finding for a well-written finding. The agent's job is to make the human's decision as easy as possible — evidence should be pre-organized, the judgment call should be explicitly stated ("This finding is SUBJECTIVE because X. The two reasonable verdicts are WARNING or HEALTHY. The determining factor is whether Y is acceptable given Z.").

---

### 3. Status Transition for SUBJECTIVE Findings

SUBJECTIVE findings use an **inline metadata annotation** on the existing Status field, not a new status value:

```
**Status**: DONE  <!-- SUBJECTIVE: awaiting human resolution -->
```

This preserves the existing four-value status system (PENDING / IN_PROGRESS / DONE / INCONCLUSIVE). SUBJECTIVE is not a fifth status — it is a disposition tag on DONE. The question is marked DONE (loop continues, not blocked), but the finding verdict is tagged for human review.

After human review:
- If resolved → remove the `<!-- SUBJECTIVE: ... -->` comment, verdict is updated in the finding file
- If skipped → leave comment in place, the finding remains SUBJECTIVE until next wave

This requires zero changes to questions.md format. The comment syntax is already valid markdown. Existing parsers that read Status fields are unaffected.

---

### 4. How Is the 70%/Wave Resolution Rate Enforced?

The 70% target is a **wave-start check**, not an automated gate. At the start of each wave (before running the first question of wave N+1):

1. Count DONE questions from the current wave with `<!-- SUBJECTIVE: awaiting -->` still present = backlog count
2. Count DONE questions from the current wave that had SUBJECTIVE but are now resolved = resolved count
3. Resolution rate = resolved / (resolved + backlog)
4. If rate < 70%: log a WARNING note in the wave header of `questions.md` and continue — the campaign is not blocked
5. If rate < 50% AND backlog > 5: escalate (see section 5)

The check is manual (Tim reads the wave header) or can be done with a grep count. No automation required.

The 70% figure is a commitment, not a hard gate, because blocking the loop on human review would defeat the purpose of autonomous operation. The campaign continues; unresolved SUBJECTIVE findings accumulate into the next wave's review queue and are counted again at the next wave-start check.

---

### 5. Escalation Path When Backlog Exceeds N

**N = 5 unresolved SUBJECTIVE findings.**

At N > 5, the campaign enters "review-debt mode." The wave-start check triggers this instruction:

> SUBJECTIVE backlog > 5 — resolution debt is too high. Before running the next question, list all unresolved SUBJECTIVE finding IDs and their stated judgment calls. Output this list to the terminal as a reminder. Continue to the next question — do not halt.

This is a notification escalation, not a blocking escalation, consistent with BrickLayer's NEVER STOP principle. The output appears in the session log where Tim will see it. The campaign continues.

If backlog reaches N > 10: the wave-start check also appends a `<!-- ESCALATION: review debt >10 -->` comment to the current wave header in `questions.md`. This makes the debt visible in the questions file without modifying any finding.

N=5 is chosen because: at 15% SUBJECTIVE rate over a 28-question campaign, ~4.2 findings are SUBJECTIVE. Backlog of 5 means one full wave's worth of unreviewed findings — a meaningful but not catastrophic debt.

---

### 6. Existing questions.md Mechanics Are Sufficient — No Breaking Changes

The full queue spec requires:
- **Status field**: existing four values unchanged (SUBJECTIVE is an annotation on DONE, not a new value)
- **Notes/comment**: inline HTML comment on the Status line — valid in all markdown parsers, ignored by grep-based status parsers
- **Wave header**: optional escalation comment appended — same format as existing wave header comments

No new fields. No new files. No changes to the finding format beyond adding an optional `## Human Resolution` section (already consistent with the `## Peer Review` section pattern in the same template). No changes to `results.tsv` schema.

Existing campaigns that never use SUBJECTIVE are entirely unaffected.

### 7. The program.md Insert (< 20 Lines)

Insert this section into `program.md` immediately after the "Wave-start sentinel check" section:

```markdown
### SUBJECTIVE Verdict Handling (Model B Queue)

A finding with `**Verdict**: SUBJECTIVE` means sufficient evidence was gathered but only
a human can resolve the verdict. Mark the question DONE and annotate the Status line:

  **Status**: DONE  <!-- SUBJECTIVE: awaiting human resolution -->

**At each wave-start sentinel check**, count unresolved SUBJECTIVE annotations in the
current wave. If backlog > 5, output a list of unresolved IDs to the terminal before
continuing. If backlog > 10, append `<!-- ESCALATION: review debt >10 -->` to the wave
header. Do not halt the campaign on either condition.

**Resolution (Tim's action)**: Read the finding. Update `**Verdict**` to the actual
verdict. Add `## Human Resolution` section (1-3 sentences). Remove the SUBJECTIVE
annotation from the Status line. Campaign target: resolve ≥ 70% of SUBJECTIVE findings
per wave before the next wave-start check.

**SUBJECTIVE is not INCONCLUSIVE.** INCONCLUSIVE = insufficient evidence.
SUBJECTIVE = evidence gathered, human judgment required to weigh it.
```

Line count: 17 lines (including blank separator lines). Within the < 20-line budget.

## Key Design Decisions

**Why DONE + annotation rather than a new SUBJECTIVE status?**
A fifth status value would break every parser, script, and grep that queries `questions.md`. It would require updating `program.md`, template `questions.md`, and any monitoring tooling (Kiln) that reads status. The annotation approach achieves the same routing signal with zero breaking changes.

**Why not a separate `subjective-queue.md` file?**
A separate file creates synchronization debt — the status in `questions.md` and the status in the queue file can diverge. Single-source-of-truth: all question state lives in `questions.md`.

**Why notification-only escalation (not blocking)?**
Consistent with BrickLayer's NEVER STOP principle (project-brief.md invariant 3). Blocking on human review would make the campaign dependent on human availability, eliminating the autonomous overnight operation property. The campaign surfaces the debt visibly and continues.
