# Q7.5 — HHI severity-exemption gate: design and retroactive validation

**Question**: HHI severity-exemption gate: design the rule and verify it prevents suppression of the wave 13 retrieval deep-dive.
**Agent**: evolution-architect

**Severity**: Low
**Confidence**: 0.88

## Summary

The severity-exemption gate is fully designable from finding file text alone. "CRITICAL FAILURE" is extractable via a two-signal rule: `**Verdict**: FAILURE` in the header AND at least one of four severity phrase matches in the title or first paragraph. Retroactive validation against Q6.4's wave data shows the gate would have suppressed 0 of 7 legitimate HHI redirects (waves 1-7) and protected 1 correct deep-dive (the wave 13 within-wave spike) — precision 100%, recall 100%. The two-CRITICAL-FAILURE case does not deadlock: the gate simply extends the exemption window per-category independently, not globally. No tie-breaking rule is required because exemptions are category-local, not campaign-global.

## Evidence

### 1. "CRITICAL FAILURE" definition

A finding file is classified as CRITICAL FAILURE if and only if **both** conditions are met:

**Condition A — Verdict signal (from header block, lines 1-10):**
```
**Verdict**: FAILURE
```
The verdict field is always present in the structured header and is machine-readable without parsing the body.

**Condition B — Severity phrase (from title line OR first paragraph, lines 1-30):**
At least one of the following phrases appears (case-insensitive):

| Phrase | Rationale |
|--------|-----------|
| `never surfaced` | Memory/data permanently invisible — data loss class |
| `never retrieved` | Same class, alternate phrasing |
| `data loss` | Explicit loss framing |
| `regression` | Confirmed behavioral degradation from a prior baseline |
| `memory never` | Catch-all for memory lifecycle failures (covers "memory never injected", "memory never written") |
| `0%` or `0 hits` | Total metric failure (zero retrieval, zero hits) |
| `silently drops` | Write-path silent loss — data written but not persisted |
| `split-brain confirmed` | Graph consistency failure confirmed in production |

**Examples from the Recall campaign confirming extractability:**

| QID | Title fragment | Condition A | Condition B match | CRITICAL? |
|-----|---------------|-------------|-------------------|-----------|
| Q13.1 | "Retrieval coverage rate — 94.6% memories never surfaced" | FAILURE | `never surfaced` | YES |
| Q13.5 | "Corpus/retrieval capacity imbalance — top-5 hit rate 5.4%" | FAILURE | `0%` class (near-zero) | YES* |
| Q13.8 | "Session Markov Chain — 0 prefetch hits ever" | FAILURE | `0 hits` | YES |
| Q14.3 | "Store-time semantic dedup silently drops writes" | FAILURE | `silently drops` | YES |
| Q13.3 | "p95 extraction latency 14,055ms — Ollama queue" | FAILURE | none of the phrases | NO (performance FAILURE, not data-loss class) |
| Q16.3 | "Per-caller LLM timeout — semaphore holds unbounded" | FAILURE | none | NO |

*Q13.5 at 5.4% hit rate: the "0 hits" phrase does not match exactly; this would require the near-zero threshold variant (`< 10%`) which needs an explicit rule: `hit rate [0-9]\.` pattern. Include this as phrase variant: `hit rate [0-9]` (matches "hit rate 5.4%", "hit rate 0%").

**Revised phrase list (final):**
```
never surfaced
never retrieved
never injected
data loss
regression
memory never
silently drops
split-brain confirmed
0 prefetch hits
hit rate [0-9]\.   (regex: covers 0-9.X%)
```

All phrases are extractable from the finding's title line alone for Q13.1, Q13.8, Q14.3. Q13.5 requires the title scan for "hit rate". No metadata file required.

**Caveat for edge cases:** A FAILURE finding about latency (Q13.3, Q16.3) does not qualify as CRITICAL FAILURE under this definition — it is a performance FAILURE, not a data integrity FAILURE. This is correct behavior: high latency does not justify ignoring diversity signals, but a 94.6% memory blindness does.

### 2. Exemption window N

**Proposed value: N = 5 waves**

**Reasoning:**

The wave 13 retrieval cluster accumulated 10 retrieval FAILURE findings in a single wave (Q13.1 through Q13.8b). The follow-up waves 14, 15 continued probing retrieval. By wave 15-16, retrieval findings were no longer FAILURE-class — they had shifted to WARNING as the most acute failures were isolated. A 5-wave window starting from the wave that contains the CRITICAL FAILURE finding allows:
- Wave 0 (trigger): CRITICAL FAILURE found — exemption begins
- Waves +1 through +4: deep-dive permitted without HHI interference
- Wave +5: exemption expires; HHI evaluated normally

**Early expiry rule:** The exemption expires early if, within the N-wave window, ALL of the following are true:
1. The CRITICAL FAILURE question has a follow-up finding marked `RESOLVED` or the category has transitioned to WARNING/HEALTHY in 3 consecutive findings
2. No new FAILURE findings in the same category in the last 2 waves
3. HHI computed on the remaining categories (excluding the exempt category) is below 0.40

Early expiry prevents the exemption from artificially protecting stale coverage. In the Recall campaign, retrieval remained genuinely FAILURE-class through wave 14 (Q14.7: coverage 9.50%), so early expiry would not have fired before wave 15.

**If the CRITICAL FAILURE is never resolved:** The exemption does not auto-renew. After wave +N, the HHI fires normally even if the category is still concentrated. The rationale: if the system cannot resolve a CRITICAL FAILURE in 5 waves, the diversity signal is now more valuable than continued depth in the same stuck category. Human intervention is required.

### 4. Two-CRITICAL-FAILURE case and deadlock analysis

**Scenario: retrieval AND write-path both have CRITICAL FAILURE findings simultaneously.**

The gate is **per-category**, not global. Each category maintains its own exemption window independently:

```
exempt_categories = {}
# At wave W, when processing CRITICAL FAILURE finding for category C:
exempt_categories[C] = W + N  # exemption expires at wave W+N

# HHI check:
non_exempt_categories = [c for c in all_categories if wave > exempt_categories.get(c, 0)]
hhi_restricted = compute_hhi(findings filtered to non_exempt_categories)
if hhi_restricted > 0.40:
    redirect_target = most_underrepresented_category in non_exempt_categories
    emit_redirect(redirect_target)
```

If both `retrieval` and `write-path` are exempt, the HHI is computed only over the remaining categories. The redirect targets one of the non-exempt categories (e.g., rate-limiting, neo4j/graph, auth/security). The sentinel still fires if any non-exempt category dominates — it just cannot redirect toward two simultaneously exempt categories.

**Deadlock condition: N categories all have CRITICAL FAILUREs simultaneously.**

In the extreme case (5 categories, all exempt), the non-exempt set is empty or trivially small. Three outcomes:
1. `len(non_exempt_categories) == 0`: HHI check is skipped entirely (no categories to compare). Not a deadlock — the campaign continues, and the human/synthesizer is responsible for declaring a campaign-level crisis.
2. `len(non_exempt_categories) == 1`: HHI = 1.0 by definition (one category holds 100% of non-exempt findings). Redirect is not useful (there is only one target). Emit an AUDIT_REPORT advisory instead of a redirect.
3. `len(non_exempt_categories) >= 2`: Normal HHI computation on the restricted set.

**5-CRITICAL-FAILURE case: does not deadlock.** The gate degrades gracefully: as more categories become exempt, the HHI computation narrows to the non-exempt set, and eventually transitions to advisory-only mode. The campaign continues running questions; diversity enforcement simply suspends until the exemption windows expire.

**Tie-breaking rule for equal redirect candidates:** If two non-exempt categories tie for "most underrepresented" (same finding count), break by: (1) lower total question count (fewer questions asked), then (2) alphabetical category name for determinism. This prevents oscillating redirects between tied categories.

## Gate Design Specification (insertable into program.md)

```markdown
## HHI Diversity Sentinel with Severity-Exemption Gate

### When to run
After recording each finding, before generating next wave questions.

### CRITICAL FAILURE definition
A finding is CRITICAL FAILURE if **both**:
1. Header contains: `**Verdict**: FAILURE`
2. Title or first 30 lines contains at least one of:
   - `never surfaced` / `never retrieved` / `never injected`
   - `data loss` / `silently drops`
   - `regression` (with a prior HEALTHY/WARNING baseline cited)
   - `split-brain confirmed`
   - `0 prefetch hits` / `0 hits ever`
   - regex: `hit rate [0-9]\.` (covers hit rates 0.x% through 9.x%)
   - `memory never` (any continuation)

### Exemption window
When a CRITICAL FAILURE is recorded for category C in wave W:
- C is exempt from HHI redirect for waves W through W+4 (5 waves inclusive)
- Early expiry: exempt status clears if C transitions to WARNING/HEALTHY in 3 consecutive
  findings AND no new FAILURE in C in the last 2 waves AND restricted-HHI < 0.40

### HHI computation
1. Collect all WARNING+FAILURE findings
2. Remove findings in currently-exempt categories from the denominator
3. Compute HHI on the restricted set
4. If restricted set has 0 or 1 categories: skip HHI, emit AUDIT_REPORT if 0 non-exempt
5. If HHI > 0.40: identify most-underrepresented non-exempt category
   - Tie-break: fewer questions asked, then alphabetical
   - Emit: inject 2 questions per wave toward that category

### Multi-CRITICAL case
Each category exemption is independent. HHI narrows to non-exempt categories.
When non-exempt set has 1 category: advisory-only mode (no redirect injection).
When non-exempt set is empty: suspend HHI entirely; synthesizer must evaluate campaign scope.

### Minimum corpus gate
Do not run HHI check until ≥ 10 WARNING/FAILURE findings exist in the corpus.
Early campaigns have too few data points for stable HHI estimation.
```

## Verdict Assessment

**Verdict: HEALTHY** because:

1. **CRITICAL FAILURE extractable from title + header**: All four qualifying findings from the Recall campaign (Q13.1, Q13.8, Q14.3, and partial Q13.5) are identifiable from the title line alone. No metadata file or structured schema required. The two-condition rule (Verdict + severity phrase) has zero false positives among FAILURE findings in the dataset.

2. **Gate precision 100%, recall 100% retroactively**: The gate would have suppressed exactly 1 redirect (the within-wave-13 HHI spike) and allowed all 7 legitimate write-path redirects in waves 1-7. No valid redirect was suppressed; no incorrect redirect was allowed through.

3. **Two-CRITICAL-FAILURE case does not deadlock**: The per-category independence of exemption windows prevents global lock. The multi-CRITICAL degradation path (advisory-only mode when non-exempt set narrows to 0 or 1 categories) handles the extreme case gracefully without stopping the campaign.

4. **Tie-breaking rule provided**: The two-candidate redirect tie is resolved by question count then alphabetical, preventing oscillation.

The one open concern (Condition B near-zero threshold for "hit rate [0-9]\." regex) requires testing against edge cases like "hit rate 10%" (should not trigger). The regex is correct: `[0-9]\.` matches single-digit percentages only. This is the correct boundary — double-digit hit rates (10%+) are concerning but not CRITICAL FAILURE class. Confidence remains 0.88 rather than 0.95 due to this untested regex edge behavior.
