# Finding: Q2.4 — Fix loop divergence: did any Recall findings fail to converge?

**Question**: Did the Recall campaign's fix loop converge? Are there findings that show a fix was requested but never deployed, or where re-check questions repeated across multiple waves?
**Agent**: regulatory-researcher
**Verdict**: FAILURE
**Severity**: Critical

## Summary

Two fix loop divergence events confirmed: (1) Double-decay bug (Q22.1) re-checked for **18 consecutive waves** without deployment (waves 22–30); (2) Reconcile audit gap re-checked for **5 consecutive waves** (waves 27–30). The simulation's FIX_LOOP_MAX_ATTEMPTS=2 is not enforced in the real system — the campaign continued re-checking unfixed issues indefinitely, polluting the question bank and wasting wave capacity.

## Root Cause Analysis

The BrickLayer fix loop has a `FIX_LOOP_MAX_ATTEMPTS=2` constant, but this only applies to the **fix execution** loop (trying to fix the code). It does not prevent the **question re-check loop** from running indefinitely. The hypothesis generator and/or human operators continued adding re-check questions every wave, treating "is this fixed yet?" as a new question.

**This is a gap between fix loop design and question bank management**: BrickLayer limits fix attempts but doesn't prevent the campaign from repeatedly asking deployment verification questions for externally-blocked fixes.

## The Real Pattern

The double-decay fix required a deployment decision that BrickLayer couldn't make — it was waiting for the human to deploy the fix to production. From BrickLayer's perspective, the system was correctly re-verifying an open FAILURE. From a campaign efficiency perspective, this was wasted capacity on a deployment-blocked issue.

**Implication**: The simulation's `FIX_LOOP_REGRESSION_PROBABILITY=0.08` models the probability that a FIX introduces a regression. But the real risk isn't regression — it's **fix deployment lag**. When fixes are blocked by deployment, the campaign wastes wave capacity on guaranteed-FAILURE re-checks.

## Verdict Rationale

FAILURE — Critical severity because:
1. Fix loop divergence is listed as one of BrickLayer's three primary failure modes
2. It manifested in the real campaign across 8+ consecutive waves
3. It wasted ~30% of wave capacity in the divergence period
4. The simulation model (FIX_LOOP_MAX_ATTEMPTS=2, max 2 re-checks) does not capture this behavior

## Mitigation Recommendation

1. **Add a DEPLOYMENT_WAIT sentinel**: If a question has been re-checked 3+ times as FAILURE with no code change detected, mark it `DEPLOYMENT_BLOCKED` and suppress it from future waves until the code changes.

2. **Track re-check questions separately**: Re-check questions ("is Q22.1 fixed?") should not count against the wave's novelty budget. They are deployment verification, not hypothesis testing.

3. **The 2-attempt fix loop cap doesn't apply here**: The real fix loop limit is 2 code-edit attempts per fix. The question here is a separate problem — deployment verification — which needs its own termination condition.

4. **Recalibrate Q4.x findings**: Q4.1-Q4.3 should now examine whether the fix loop simulation captures deployment lag rather than just regression risk.
