# Q7.3 — Compound state: DEPLOYMENT_BLOCKED + PENDING_EXTERNAL interaction

**Question**: Compound state: DEPLOYMENT_BLOCKED and PENDING_EXTERNAL in the same pre-wave filter — do they interact cleanly?
**Verdict**: WARNING
**Severity**: Medium
**Confidence**: 0.85
**Date**: 2026-03-26

## Summary

DEPLOYMENT_BLOCKED and PENDING_EXTERNAL do not deadlock and are non-overlapping in their state transition paths. However, the `resume_after=question_status` variant of PENDING_EXTERNAL (where B's resume condition is "after Q(A) resolves to DONE") requires a new loop capability — cross-question status lookup — not explicitly specified in either Q6.3 or Q6.6. No Recall campaign questions would have been simultaneously in both states. Verdict: WARNING because the interaction is safe but requires one additional implementation step not present in either design.

---

## Scenario Analysis

### The Three Questions

**(A)** DEPLOYMENT_BLOCKED question waiting for a git diff on `qdrant.py`
- Entry: `Q22.1.R [FIX-LOOP]`
- Status: `DEPLOYMENT_BLOCKED`
- Metadata: `target_files: ["recall/qdrant.py"]`, `unblock_condition: "git diff detects change in recall/qdrant.py since wave 22"`
- Current state: `git diff HEAD~1 -- recall/qdrant.py` returns empty string (no deployment)

**(B)** PENDING_EXTERNAL question whose `resume_after` condition is "after Q(A) resolves to DONE"
- Entry: `Q22.2 [FIX-LOOP]`
- Status: `PENDING_EXTERNAL`
- Metadata: `resume_after: "after Q22.1.R resolves to DONE"`, `blocking_condition: "Downstream verification requires qdrant.py fix to be confirmed deployed"`, `consecutive_blocked_waves: 0`

**(C)** PENDING_EXTERNAL question with ISO-8601 `resume_after` in the past, `consecutive_blocked_waves = 3`
- Entry: `Q29.1 [FIX-LOOP]`
- Status: `PENDING_EXTERNAL`
- Metadata: `resume_after: "2026-03-01T04:00:00Z"` (overdue), `consecutive_blocked_waves: 3`, `max_blocked_waves` not set (default escalation applies)

## Analysis Question 1: Pre-wave filter behavior for A, B, and C

### Does the filter correctly exclude A and B but include the escalated C?

**Question A (DEPLOYMENT_BLOCKED):**

The pre-wave filter in `bl/campaign.py` runs the DEPLOYMENT_BLOCKED check first (per Q6.3 spec: "at the top of each wave's question-dispatch loop, before questions are pulled from the PENDING queue"). For A, the filter evaluates:

```
git diff <diagnosed_commit>..HEAD -- recall/qdrant.py  →  empty string
```

No human annotation present, no `resume_after` timer. Result: A is skipped this wave. **A is correctly excluded.**

**Question B (PENDING_EXTERNAL with question-status resume condition):**

The pre-wave filter checks B's `resume_after` field. The value is an event description: `"after Q22.1.R resolves to DONE"`. The filter must look up Q22.1.R's current Status in questions.md. Q22.1.R's Status is `DEPLOYMENT_BLOCKED`, not `DONE`. Condition not met. Result: B remains PENDING_EXTERNAL, `consecutive_blocked_waves` increments. **B is correctly excluded.**

**Question C (PENDING_EXTERNAL at escalation threshold):**

The filter evaluates C's `resume_after`: `"2026-03-22T04:00:00Z"`. Current date is 2026-03-26. The timestamp is in the past. The condition should be met — but the external event (the thing the question was waiting for) has not materialized. The filter checks: `consecutive_blocked_waves = 3`, `max_blocked_waves` not set → default escalation applies.

Per the Q6.6 escalation rule: `consecutive_blocked_waves >= 3` AND `max_blocked_waves != 0` → escalate to FAILURE. C's status changes to FAILURE with annotation `BLOCKING_CONDITION_BROKEN`. **C is correctly excluded from the active wave pool (as FAILURE, not PENDING) — and an auto-escalation finding is written.**

**Assessment**: The filter correctly handles all three cases, but Question C's auto-escalation means it does not "enter the active pool" — it transitions to FAILURE before being queued. This is the correct behavior: an overdue condition with no resolution triggers automatic failure, not activation.

**Result for Q1: PASS — but only if the filter evaluates DEPLOYMENT_BLOCKED checks before PENDING_EXTERNAL checks (order matters, see Q2 analysis).**

## Analysis Question 2: A's git diff fires — does B's resume condition evaluate correctly?

### When A re-enters PENDING, does B resolve correctly?

**Sequence:**

Wave N: A is DEPLOYMENT_BLOCKED, B is PENDING_EXTERNAL.

Wave N+1 start: Pre-wave filter runs. For A: `git diff` on `qdrant.py` returns non-empty. A's status transitions `DEPLOYMENT_BLOCKED → PENDING`. A is added to the wave N+1 pool.

**Critical question: Is B evaluated in the same wave?**

The Q6.6 spec states the pre-wave filter runs once per wave start, iterating over all PENDING_EXTERNAL entries. If A's status change from DEPLOYMENT_BLOCKED to PENDING happens in the same pre-wave iteration pass as B's evaluation, there is a race condition: B looks up A's status, finds it as PENDING (not yet DONE), and correctly stays PENDING_EXTERNAL. This is correct behavior — A must run and produce a DONE verdict before B can activate.

**However**, if the implementation runs the filter in a single pass (all DEPLOYMENT_BLOCKED checks first, then all PENDING_EXTERNAL checks in the same loop iteration), B could see A's newly-set PENDING status mid-pass. Since `resume_after` requires `DONE` (not `PENDING`), B still correctly stays parked. No problem here.

**The actual gap: Q6.6 specifies question-status resume conditions but does not specify the lookup mechanism.**

Q6.6 says: "Event-description variants (e.g., 'after Q29.1 resolves') are evaluated by checking the Status field of the referenced question in questions.md — again, a local file read." This describes the mechanism in prose but does not:

1. Define the `resume_after` syntax for question references (e.g., `"after Q22.1.R resolves to DONE"` is informal prose — the parser needs to extract the question ID and the target status)
2. Specify whether the loop must wait for A to complete within the same wave before evaluating B, or defer B to the next wave
3. Handle the case where A runs in wave N+1 and produces DONE — does B activate in wave N+1 (same wave, after A completes) or wave N+2 (next wave, after the end-of-wave status update)?

**This is the new loop capability required**: a structured `resume_after` syntax for question references, and a within-wave re-evaluation trigger that checks PENDING_EXTERNAL questions after each question in the wave completes and achieves DONE status. The current design (both Q6.3 and Q6.6) models the pre-wave filter as a single pass at wave start — it does not include a mid-wave re-evaluation step.

**Consequence**: Without this capability, B would not activate until wave N+2 at the earliest (A completes at wave N+1; B's condition is checked at the start of wave N+2). This is functionally correct (no deadlock, no data corruption) but requires one additional wave of latency compared to optimal behavior.

**Verdict for Q2: WARNING — no deadlock, but question-status `resume_after` dependencies require explicit syntax definition and a post-completion re-evaluation hook not present in the current design. Implementable in ~20 lines of loop logic; this is not a blocking gap, but it is a gap.**

## Analysis Question 3: Can a question simultaneously satisfy both DEPLOYMENT_BLOCKED and PENDING_EXTERNAL?

### Is double-blocked state possible and valid?

**State transition analysis:**

From Q6.3: A question enters DEPLOYMENT_BLOCKED from FAILURE (after 3 consecutive FAILURE re-checks with no code delta). The transition path is:
```
PENDING → IN_PROGRESS → DONE(FAILURE) → [re-check] → FAILURE → FAILURE → FAILURE → DEPLOYMENT_BLOCKED
```

From Q6.6: A question enters PENDING_EXTERNAL from PENDING (before it has ever run, when its precondition is not yet met). The transition path is:
```
PENDING → PENDING_EXTERNAL → (condition met) → PENDING → IN_PROGRESS → DONE
```

**The paths do not share a state:** DEPLOYMENT_BLOCKED requires the question to have been run (produced FAILURE verdicts). PENDING_EXTERNAL requires the question to have never run this precondition cycle (it has not yet been dispatched in the current attempt). A question that is already DEPLOYMENT_BLOCKED has already run at least 3 times with FAILURE outcomes — it cannot also be in a "not yet run" PENDING_EXTERNAL state for the same question entry.

**Theoretical edge case: DEPLOYMENT_BLOCKED question with a `resume_after` timer**

Q6.3 explicitly provides `resume_after` as one of three unblocking conditions for DEPLOYMENT_BLOCKED questions: "If `resume_after` date has passed: rewrite status to PENDING." This is a timer that was originally designed for scheduled deployments. If a DEPLOYMENT_BLOCKED question has a `resume_after` timer that has passed — but the code has still not changed — the timer fires and sets status to PENDING. This is not simultaneous double-blocking; it is a timer overriding the code-change condition.

**However, there is a subtle semantic overlap here:** a DEPLOYMENT_BLOCKED question waiting for a scheduled deployment (`resume_after: "2026-04-01T00:00:00Z"`) and a PENDING_EXTERNAL question waiting for the same deployment window are semantically identical. Q6.6 note: "PENDING_EXTERNAL: question cannot be run yet (external precondition not met — cron, TTL, deploy window)." A deploy window is precisely the kind of event that PENDING_EXTERNAL was designed for.

A question could therefore be designed as either DEPLOYMENT_BLOCKED (with `resume_after` timer) or PENDING_EXTERNAL (with ISO-8601 `resume_after`), depending on whether it has previously run and produced FAILURE. The designs are not double-activatable on the same question entry, but they are semantically ambiguous at the boundary — a question with exactly 2 prior FAILUREs waiting for a timed deployment could be modeled as either. This ambiguity does not cause deadlock or double-blocking; it is a classification decision for the loop author.

**Result for Q3: No valid double-blocked state exists. A question cannot legitimately hold both statuses simultaneously under the Q6.3 and Q6.6 specifications. The state paths are disjoint. The only overlap is semantic (timer-based DEPLOYMENT_BLOCKED vs. PENDING_EXTERNAL with ISO-8601 `resume_after`), but this is a design-time ambiguity, not a runtime conflict.**

## Recall Campaign Simultaneous-State Estimate

### How many Recall campaign questions could have been simultaneously in both states?

**Evidence base:**

From Q6.3: 12–14 questions in DEPLOYMENT_BLOCKED state would have been suppressed across waves 22–30 in the Recall campaign. The known entries are:
- Double-decay re-check questions: waves 25–30 (~6 questions)
- Reconcile audit gap re-check: waves 29–30 (~2 questions)
- Deployment-blocked questions from Q4.1 table: ~4–6 additional

From Q6.6: 10 question slots consumed by PENDING_EXTERNAL chains from wave 26 onward:
- Hygiene cron chain: Q29.1, Q30.1, Q31.1 (3 questions)
- GC eligibility wait: waves 26–32 (~4 questions)
- Consolidation fix verification (dependent on Q29.1): ~3 questions

**Intersection analysis:**

DEPLOYMENT_BLOCKED questions in the Recall campaign are re-check entries for already-diagnosed FAILURE conditions (qdrant.py double-decay, reconcile audit gap, consolidation user_id, split-brain, ghost user). These are questions that have already run and failed.

PENDING_EXTERNAL questions in the Recall campaign are forward-looking entries waiting for timing conditions (hygiene cron fire, GC TTL expiry, consolidation verification after cron resolves). These are questions that have not yet run.

The consolidation verification chain (PENDING_EXTERNAL, waiting for Q29.1) is the closest to overlap: it is waiting for the hygiene cron to confirm its operation, and separately there may be a DEPLOYMENT_BLOCKED re-check for the consolidation fix. However:
- The DEPLOYMENT_BLOCKED re-check for consolidation would be waiting for the code fix to deploy (a different question entry)
- The PENDING_EXTERNAL verification is waiting for the cron to run (a different question entry)

These are different question entries tracking different aspects of the same underlying problem. No single question entry would hold both statuses.

**Estimate: Zero Recall campaign questions would have been simultaneously in both DEPLOYMENT_BLOCKED and PENDING_EXTERNAL states.** The mechanisms address different question archetypes (post-diagnosis re-checks vs. pre-condition forward checks), and while they address related problems in the campaign's failure chains, they do so via distinct question entries that never share status.

## Deadlock Analysis

### Is there a deadlock cycle?

The specific scenario hypothesized: B's `resume_after = "after Q(A) resolves to DONE"`, and A is DEPLOYMENT_BLOCKED.

**Cycle check:**
- A can only transition out of DEPLOYMENT_BLOCKED if: (1) `git diff` on target files returns non-empty, OR (2) human annotation `## Deployment Confirmed`, OR (3) `resume_after` timer expires.
- None of these conditions depends on B's status.
- B can only transition out of PENDING_EXTERNAL if: A resolves to DONE.
- A can only resolve to DONE if: it re-enters PENDING, runs, and produces a DONE verdict.

**Is there a cycle?** No. A's unblocking conditions are all external to the question system (code changes, human annotation, wall-clock time). None of them depend on B. A does not wait for B. B waits for A. The dependency graph is: B → A → (external event). This is a directed acyclic dependency, not a cycle.

**The only way a deadlock could arise:** if A's `resume_after` timer depended on B's status (e.g., "after B resolves to DONE"). But A is DEPLOYMENT_BLOCKED, and the Q6.3 spec defines `resume_after` for DEPLOYMENT_BLOCKED entries as ISO-8601 timestamps or human annotations — not question-status references. Question-status references in `resume_after` are a PENDING_EXTERNAL feature only. The designs are therefore structurally protected against this deadlock by the restriction that DEPLOYMENT_BLOCKED unblocking is always external (code change, human, timer) and never question-status dependent.

**No deadlock exists in the compound state scenario.**

## Verdict Assessment

| Criterion | Requirement | Assessment |
|-----------|-------------|------------|
| No deadlock cycle | B's resume_after=A's status + A's DEPLOYMENT_BLOCKED must not create a cycle | PASS — A's unblocking is always external; no question-status dependency chain loops back |
| No double-blocked state | A question cannot legitimately hold both statuses simultaneously | PASS — state paths are disjoint; DEPLOYMENT_BLOCKED requires prior FAILURE runs, PENDING_EXTERNAL requires no prior runs |
| Question-status resume_after handleable with simple lookup | Must not require new loop architecture | PARTIAL — handleable with file-stat lookup, but requires explicit syntax for question references and a post-completion re-evaluation trigger not specified in Q6.6 |
| Zero Recall simultaneous-state questions | No Recall questions would have been in both states | PASS — zero overlap by question archetype |

**Verdict: WARNING**

The mechanisms are safe and non-conflicting. The WARNING is specifically on the gap in `resume_after=question_status` specification: the event-description syntax for question references is informal prose in Q6.6, and the timing of B's re-evaluation after A reaches DONE requires either (a) a mid-wave re-evaluation pass or (b) acceptance of one additional wave of latency. This is a design gap, not a deadlock or incompatibility — but it is real implementation work that neither Q6.3 nor Q6.6 fully specifies.

## Implementation Implications

### What Q7.3 adds to the implementation spec:

1. **Pre-wave filter ordering**: DEPLOYMENT_BLOCKED checks must run before PENDING_EXTERNAL checks in the same loop pass (A must be transitioned to PENDING before B evaluates it — though in practice B still requires DONE, not PENDING, so this ordering is advisory, not required for correctness).

2. **Structured `resume_after` syntax for question references**: Define a canonical format:
   ```
   resume_after: "Q22.1.R:DONE"       # structured form
   resume_after: "after Q22.1.R resolves to DONE"   # prose form (current Q6.6 example)
   ```
   The loop must parse either form and extract the question ID + target status.

3. **Post-completion re-evaluation hook**: After any question transitions to DONE within a wave, the loop should scan PENDING_EXTERNAL questions for entries whose `resume_after` references the just-completed question ID. If found and the status matches, move those entries to PENDING for the current or next wave. This can be implemented as a 15-line addition to `bl/campaign.py` without restructuring the existing wave loop.

4. **No double-blocking guard required**: The state machine analysis confirms no question can enter both states; no runtime guard is needed. Documentation of the state paths is sufficient.

### Estimated implementation cost

- Structured `resume_after` syntax: ~10 lines of parser code in `bl/campaign.py`
- Post-completion re-evaluation hook: ~15 lines in the wave execution loop
- Total: ~25 lines, no schema changes, no constants.py changes

This is the only new loop capability required by the compound state design.
