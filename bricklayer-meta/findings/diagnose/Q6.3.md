# Q6.3 — DIAGNOSIS_COMPLETE design: suppress deployment-blocked re-check questions

**Question**: What is the exact mechanism to suppress deployment-blocked re-check questions in BrickLayer's questions.md without breaking existing format?
**Verdict**: HEALTHY
**Severity**: Medium
**Confidence**: 0.88

## Summary

The DEPLOYMENT_BLOCKED mechanism is feasible and fully compatible with the existing questions.md flat-file format. It requires no constants.py changes, uses `git diff` (already available to any BrickLayer campaign), and saves an estimated 12–14 question slots in the Recall campaign's waves 22–30.

---

## Design Specification

### Sub-question 1: Trigger Condition

**Trigger**: After **3 consecutive FAILURE re-checks** with no code change detected in the target file(s) identified in the finding, the question enters `DEPLOYMENT_BLOCKED`.

**Rationale**:
- **Not 2**: Two consecutive FAILUREs could be a flaky environment (cron timing, network latency). The Q4.3 finding showed 15.4% INCONCLUSIVE rate from timing dependencies — two failures in a row may be noise.
- **3 is the right threshold**: By the third consecutive re-check with no code delta, the pattern is structural. The Q2.4 finding shows the double-decay bug was first identified at wave ~22 and was still undeployed at wave 30 — a 3-wave trigger would have suppressed it starting at wave 25, saving 5–6 slots.
- **DIAGNOSIS_COMPLETE verdict as alternative trigger**: If the finding already contains a `DIAGNOSIS_COMPLETE` verdict annotation (root cause traced to a specific file and line), a single FAILURE re-check suffices. The Q4.1 recommendation supports this — when a finding reaches specific-file-and-line diagnosis, it's a deployment blocker by definition.

**Code change detection via `git diff`**:
`git diff HEAD~1 -- <target_file>` is available in any git-backed BrickLayer project. The target file(s) are identifiable from the finding body (Q2.4 identified `qdrant.py:114`, Q4.1's table lists specific files for each blocker). No external tooling is needed. The campaign loop already shells out to `git` for commit operations, so this is zero-dependency addition.

**Formal trigger rule**:
```
IF question.re_check_count >= 3
   AND all(verdict == "FAILURE" for last 3 re-checks)
   AND git_diff(question.target_files, since=question.diagnosed_wave) == ""
THEN status = "DEPLOYMENT_BLOCKED"

OR IF finding contains verdict == "DIAGNOSIS_COMPLETE"
   AND most_recent_recheck_verdict == "FAILURE"
THEN status = "DEPLOYMENT_BLOCKED"
```

### Sub-question 2: questions.md Entry Format

A DEPLOYMENT_BLOCKED entry reuses the existing question header structure and adds four metadata fields as a fenced block at the end of the entry. This preserves flat-file parseability — existing parsers that read Status, Mode, Hypothesis, etc. continue to function; they simply skip the unfamiliar `Deployment` block.

**Concrete example entry**:

```markdown
## Q22.1.R [FIX-LOOP] Is the double-decay bug fix deployed? (re-check)
**Mode**: subprocess
**Agent**: quantitative-analyst
**Status**: DEPLOYMENT_BLOCKED
**Hypothesis**: The double-decay root cause (qdrant.py:114 `_user_conditions(None)`) was diagnosed at wave 22. This re-check verifies deployment.
**Test**:
```
python C:/Users/trg16/Dev/autosearch/recall/simulate.py --check-only
```
expect_stdout: HEALTHY
timeout: 30
**Verdict threshold**:
- FAILURE: target fix not deployed — qdrant.py:114 still returns []
- HEALTHY: fix confirmed deployed

**Deployment**:
- `blocking_reason`: "qdrant.py:114 `_user_conditions(None)` returns `[]`; fix identified, requires human deployment to production"
- `diagnosed_wave`: 22
- `blocked_since_wave`: 25
- `re_check_count`: 3
- `target_files`: ["recall/qdrant.py"]
- `unblock_condition`: "git diff detects change in recall/qdrant.py since wave 22"
```

**Field definitions**:

| Field | Type | Purpose |
|-------|------|---------|
| `blocking_reason` | string | Human-readable description of what fix was diagnosed, including file/line if known |
| `diagnosed_wave` | int | Wave number when DIAGNOSIS_COMPLETE or initial FAILURE was recorded |
| `blocked_since_wave` | int | Wave number when suppression began (3rd consecutive FAILURE re-check) |
| `re_check_count` | int | Total number of re-checks attempted (for audit trail) |
| `target_files` | list[str] | File paths to monitor for changes via git diff |
| `unblock_condition` | string | Human-readable description of what must happen to resume |

**Parser compatibility**: Existing parsers read Status as the 4th line after `##`. `DEPLOYMENT_BLOCKED` is a new string value but the field position is unchanged. The `**Deployment**:` block is a new named section — parsers that iterate over known sections will skip it; only a parser explicitly checking for it will see it. This is additive, not breaking.

### Sub-question 3: Unblocking Condition

**Selected: Option D — combination, whichever comes first.**

| Option | Assessment |
|--------|-----------|
| A: `git diff` detects target file change | Feasible in BrickLayer's subprocess loop. `git diff <diagnosed_commit>..HEAD -- <target_file>` produces non-empty output when the file changes. This is the primary automated path. |
| B: Human annotates with `## Deployment Confirmed` | Feasible — BrickLayer already respects human annotation blocks (`## Human Correction` in the base CLAUDE.md). Adding a `## Deployment Confirmed` block is natural. This covers cases where the deployment is to infrastructure (DB migration, config flag) rather than a tracked code file. |
| C: `resume_after: <ISO-8601>` timer | Feasible but fragile. Timers require the campaign loop to check wall-clock time, which is available but produces false positives (timer fires but deployment still not done). Use only as a tertiary fallback for external blockers with known ETAs. |
| D: Combination | Whichever arrives first: git change detected, OR human annotation added, OR timer expires. The git path handles most code fixes; the annotation path handles infrastructure/config fixes; the timer path handles scheduled deployments with known ETAs. |

**Unblocking sequence**:
```
AT START OF EACH WAVE, FOR EACH DEPLOYMENT_BLOCKED QUESTION:
  IF git_diff(target_files, since=diagnosed_wave) != "":
      status = "PENDING"     # resume — code changed
      log("Unblocked by git delta: <files>")
  ELIF "## Deployment Confirmed" in question_body:
      status = "PENDING"     # resume — human confirmed
      log("Unblocked by human annotation")
  ELIF resume_after exists AND now() >= resume_after:
      status = "PENDING"     # resume — timer expired
      log("Unblocked by timer")
  ELSE:
      skip question this wave
```

This check runs in `bl/campaign.py` at the top of each wave's question-dispatch loop, before questions are pulled from the PENDING queue.

### Sub-question 4: Status Taxonomy Decision

**Recommendation: Add DEPLOYMENT_BLOCKED as a 5th Status value.** Do not model it as an INCONCLUSIVE sub-state.

**Arguments for 5th value (winner)**:

1. **Semantic clarity**: INCONCLUSIVE means "we cannot assess this question given available tooling or the system is in an ambiguous state." DEPLOYMENT_BLOCKED means "we assessed this, diagnosed it fully, and are waiting for an external action." These are categorically different epistemic states. Conflating them obscures the distinction between "unknown" and "known but blocked."

2. **results.tsv filterability**: DEPLOYMENT_BLOCKED in results.tsv allows direct querying: `grep DEPLOYMENT_BLOCKED results.tsv | wc -l` counts open deployment blockers. With INCONCLUSIVE sub-states, you'd need to parse the finding body.

3. **Hypothesis generator behavior**: The local LLM (qwen2.5:7b) reads questions.md to generate Wave N+1 questions. INCONCLUSIVE questions generate follow-up questions (the generator correctly infers there's unresolved uncertainty). DEPLOYMENT_BLOCKED questions should generate no follow-ups — suppression must be visible to the hypothesis generator's prompt. A distinct status value makes the suppression rule simple: "do not generate follow-up questions for DEPLOYMENT_BLOCKED entries."

4. **Parser migration is trivial**: results.tsv has a `verdict` column. The column already contains 4 string values. Adding a 5th string value (`DEPLOYMENT_BLOCKED`) is additive — parsers that check `if verdict == "FAILURE"` continue to work; parsers that enumerate all valid values need a one-line addition. The Q2.4 finding already identified that results.tsv has a formatting bug in waves 25–30 (numeric wave IDs in the verdict column). Any migration that fixes that bug can add the 5th value simultaneously at zero additional cost.

**Arguments for INCONCLUSIVE sub-state (rejected)**:

- Backward compatible with zero parser changes — but at the cost of ambiguous semantics that would confuse the hypothesis generator
- INCONCLUSIVE already carries the meaning "cannot assess" — using it for "diagnosed but undeployed" is a semantic overload that would make synthesis harder to interpret

**Verdict taxonomy after this change**:

```
PENDING      — not yet run
IN_PROGRESS  — actively running
DONE         — completed (sub-verdict: HEALTHY / WARNING / FAILURE / INCONCLUSIVE)
DEPLOYMENT_BLOCKED — diagnosis complete, suppressed until target code changes
```

Note: The finding's verdict field (HEALTHY/WARNING/FAILURE/INCONCLUSIVE) is separate from the question's Status field. DEPLOYMENT_BLOCKED is a Status value, not a verdict. The underlying verdict remains FAILURE.

### Sub-question 5: Estimated Capacity Saving

**Estimate: 12–14 question slots freed in the Recall campaign.**

**Evidence base**:

From Q2.4 (confirmed findings):
- Double-decay bug: 18 consecutive re-check questions across waves 22–30. With a 3-consecutive-FAILURE trigger, suppression activates at wave 25 (waves 22, 23, 24 = 3 consecutive). Waves 25–30 = 6 waves × 1 question/wave = **6 slots freed**.
- Reconcile audit gap: 5 consecutive re-check questions across waves ~26–30. With a 3-consecutive trigger, suppression activates at wave 29 (waves 26, 27, 28 = 3). Waves 29–30 = 2 waves × 1 slot = **2 slots freed**.

From Q4.1 (23+ open FAILURE findings at wave 36):
- The table in Q4.1 lists 4 findings with "root cause identified, fix not deployed" (double-decay, consolidation user_id, split-brain, ghost user). If each generated 1 re-check question in the final 3 waves on average, that's an additional 4 × 1 = **4 slots** in an extended campaign.
- Conservative estimate for the 36-wave Recall campaign: the above two confirmed cases provide a floor of 8 slots. Adding probable re-checks from the other 4 deployment-blocked findings: +4–6 slots.

**Total estimate: 12–14 question slots freed out of ~208 total questions run (5.8–6.7% recovery).**

The Q2.4 finding stated ~30% capacity waste in waves 25–30. The absolute count (12–14 slots) is consistent: 6 waves × ~7 questions/wave = 42 questions in that window; 12–14 / 42 = 28–33%, matching the ~30% figure.

This exceeds the HEALTHY threshold of >= 10 slots.

This does not rise to WARNING because:
1. The Q2.4 finding already flagged that results.tsv has a pre-existing formatting bug in waves 25–30
2. The migration required (add one enum value) is bundled with the existing bug fix at zero additional cost
3. No data is lost or misclassified

## Implementation Notes

**Files to change**:

1. **`bl/campaign.py`** — Add DEPLOYMENT_BLOCKED check at the top of each wave's question-dispatch loop (before PENDING questions are pulled). Logic:
   - For each question with `Status: DEPLOYMENT_BLOCKED`, run `git diff <diagnosed_commit>..HEAD -- <target_files>`
   - If non-empty output: rewrite status to PENDING, log unblocking event
   - If `## Deployment Confirmed` annotation present: rewrite status to PENDING
   - If `resume_after` date has passed: rewrite status to PENDING
   - Otherwise: skip the question (do not add to this wave's queue)

2. **`bl/results.py` (or equivalent TSV writer)** — Add `DEPLOYMENT_BLOCKED` to the valid verdict/status enum. Fix the existing numeric-ID corruption in waves 25–30 in the same pass.

3. **`questions.md` spec (program.md or equivalent)** — Append one section documenting the `**Deployment**:` block format and the 5 field definitions. This is a documentation change, not a behavior change.

4. **Hypothesis generator prompt in `bl/hypothesis.py`** — Add one instruction: "Do not generate follow-up questions for questions with Status: DEPLOYMENT_BLOCKED. These are suppressed pending external deployment."

**No changes to**: `constants.py`, `simulate.py`, `agents/*.md`, findings format, `results.tsv` column structure (only the set of valid values in the verdict column expands).

**Rollout**: The mechanism is retroactively applicable — existing campaigns can annotate known deployment-blocked questions with the `**Deployment**:` block and update their Status, causing suppression to take effect on the next wave run without any data migration.
