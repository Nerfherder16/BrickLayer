# Finding: Q4.3 — Fix loop timeout behavior: do investigations get stuck or time out?

**Question**: Did any investigation threads in the Recall campaign get permanently blocked or time out — failing to reach a conclusion within the campaign's available waves?
**Verdict**: WARNING
**Severity**: Medium

## Summary

Two investigation threads showed timeout-like behavior: (1) INCONCLUSIVE findings blocked by external timing constraints (cron windows, GC eligibility gates) appear repeatedly across multiple waves; (2) Wave 17 is entirely absent from results.tsv — either a true campaign pause or a data gap. The double-decay investigation was NOT blocked — it converged to a code-level root cause after 11 waves. The real timeout risk is **external timing dependency**, not algorithmic depth.

## Evidence

### INCONCLUSIVE Findings by Category

| Category | Count | Waves | Blocking Condition |
|----------|-------|-------|-------------------|
| Hygiene cron (waiting for scheduled fire) | 5+ | 27-31 | Target time: 2026-03-16T04:00 UTC (~17h away) |
| GC eligibility | 4+ | 26-32 | Target: 2026-03-21T19:12 UTC (6+ days away) |
| Consolidation fix verification | 3+ | 29-31 | Prerequisite Q29.1 (cron) unmet |
| Timing-sensitive verification | 2 | 14, 16 | Required same-day measurement windows |
| **Total INCONCLUSIVE** | **32** | Various | |

32 of 208 findings (15.4%) are INCONCLUSIVE — all due to external timing dependencies, not algorithmic failure.

### The Hygiene Cron Chain

Q29.1, Q30.1, Q31.1 are all INCONCLUSIVE because the hygiene cron had not fired at the time of checking. Each wave added a question waiting for the same cron event. By wave 31 (Q31.1 FAILURE), it was confirmed the cron had been broken since 2026-03-09 — the wait was for an event that would never arrive.

This is a compounded timeout: (a) The campaign waited for a cron window that (b) the cron would never reach because it was broken. BrickLayer correctly marked these INCONCLUSIVE rather than FAILURE until sufficient evidence accumulated (9th consecutive miss → FAILURE).

### The Missing Wave 17 Gap

Waves 8-12 are absent from results.tsv. Wave 17 is also missing. The absence of waves 8-12 represents a significant gap (if they were run but not recorded, or a true campaign pause). Wave 17 was explicitly flagged in the synthesis: "Wave 17 questions (Q16.3a, Q16.4a): PENDING — deferred until Tier 1 fixes deployed."

This is a voluntary pause — not a timeout but a deliberate hold. The campaign correctly recognized that some questions could only be answered after fixes were deployed.

### Double-Decay Investigation: No Timeout

The double-decay thread (Q22.1 through Q33.2b) ran for 11 waves and DID converge to a complete root cause:
- Wave 22: `_user_conditions(None)` returns `[]` — initial finding
- Wave 31: IS NULL fix deployed but doubling persists
- Wave 31: NOT admin API, NOT duplicate cron (Q31.2a)
- Wave 33: `get_distinct_user_ids()` returns `[0]` — exact root cause (Q33.2a, Q33.2b)

This is a SUCCESS case: a complex investigation spanning 11 waves converged to a 2-line fix. The timeout risk was real but did not materialize.

### Fix Loop Timeout Model vs Reality

| Scenario | Simulation Model | Empirical |
|----------|-----------------|-----------|
| Investigation depth needed | Up to 2 fix attempts | Up to 11 investigation waves |
| Blocking condition | Fix regression | External timing dependency |
| Timeout trigger | MAX_ATTEMPTS exceeded | Never (campaign continued) |
| Resolution rate | 100% within 2 attempts | 0% (deployment not in scope) |

The simulation models timeout as "too many fix attempts → give up." The real timeout risk is "external dependency (cron, GC gate, deploy window) → INCONCLUSIVE × N waves."

## Verdict Rationale

WARNING because:
1. 15.4% INCONCLUSIVE rate is above the campaign's expected noise floor
2. Multiple INCONCLUSIVE chains consuming wave capacity (cron + GC = 9+ questions across waves 26-32)
3. The hygiene cron wait (which turned out to be a broken cron) cost 4+ waves before FAILURE was declared
4. The simulation doesn't model this INCONCLUSIVE-accumulation pattern at all

## Mitigation Recommendation

1. **Add PENDING_EXTERNAL state**: Questions that are blocked by a specific external event (cron window, GC eligibility date, deploy window) should be parked as PENDING_EXTERNAL with a `resume_after` timestamp. Do not add them to the wave's question count until the unblocking condition is met.

2. **Add a broken-prerequisite detector**: If a question has been INCONCLUSIVE for 3+ consecutive waves with the same blocking condition, escalate to FAILURE with "blocking condition is itself broken."

3. **The simulation's `FIX_LOOP_TIMEOUT` (600s) is irrelevant**: Real timeouts are days-long external dependencies, not 10-minute subprocess limits.
