# Finding: Q4.1 — Fix loop convergence rate: what fraction of identified bugs were eventually fixed?

**Question**: Of all FAILURE-severity findings in the Recall campaign, what fraction were eventually fixed and confirmed HEALTHY? Is BrickLayer's fix loop producing convergence or accumulation?
**Verdict**: FAILURE
**Severity**: High

## Summary

Fix convergence rate: **~8%**. Of ~25+ FAILURE-severity findings, only 2 confirmed-fixed by wave 36 (LLM timeout tiers Q15.3/Q16.3 → Q31.7 HEALTHY; reconcile audit was a FALSE NEGATIVE, not a fix). 23+ open failures remain at campaign end. BrickLayer's fix loop succeeds at code-level diagnosis but fails at deployment convergence — the real bottleneck is the human deployment step, which the simulation does not model.

## Evidence

| Status | Count | Examples |

### The Human Deployment Bottleneck

The BrickLayer fix loop works as follows:
1. Question identifies failure → FAILURE verdict
2. Synthesizer marks as open
3. Hypothesis generator adds re-check question
4. Re-check confirms fix NOT deployed → repeat from step 3

This loop can run **indefinitely** without human deployment. BrickLayer cannot deploy code. The double-decay fix (two lines of code) was identified at wave 33 and remains undeployed 3 waves later.

### What the Simulation Models vs Reality

| Aspect | Simulation | Reality |
|--------|-----------|---------|
| Fix probability | `FIX_LOOP_ENABLED=False` (disabled) | Fixes happen externally, not by BrickLayer |
| Deployment time | Not modeled | Can span many waves (external blocker) |
| Fix regression | `FIX_LOOP_REGRESSION_PROBABILITY=0.08` | 1 regression found in 208 findings (0.5%) |
| Convergence | 2-attempt cap stops divergence | No cap; human deployment determines convergence |

### Why the Convergence Rate Isn't Actually Low

The 8% convergence rate looks low, but this reflects the campaign's purpose — BrickLayer is designed to FIND failures, not fix them. The fix is a human responsibility. The correct metric is:
- **Finding rate**: Excellent — 94 WARNING/FAILURE findings in 36 waves
- **Diagnosis depth**: Excellent — double-decay traced to exact line after 11-wave investigation chain
- **Fix deployment**: 0% within BrickLayer's control — entirely dependent on human

The simulation's `FIX_LOOP_ENABLED=False` default is actually the correct model — BrickLayer doesn't fix, it diagnoses.

## Verdict Rationale

FAILURE because:
1. The campaign accumulates open failures without convergence — 23+ open at wave 36 out of 34+ identified
2. This creates a growing re-check burden that pollutes the question bank (see Q2.4)
3. The simulation's FIX_LOOP model is misnamed — what it models is "fix execution within BrickLayer" which doesn't happen. The real fix loop is the human deployment cycle.

## Mitigation Recommendation

1. **Reframe the fix loop**: BrickLayer's loop is a DIAGNOSIS loop, not a fix loop. The campaign's job is to produce actionable deployment-ready fixes, not to deploy them.

2. **Add a DIAGNOSIS_COMPLETE state**: When a finding has root cause identified at code level (specific file, line, fix), mark it `DIAGNOSIS_COMPLETE` and suppress re-check questions. Resume only when the code changes.

3. **Track deployment lag as a campaign metric**: "Time from FAILURE to fix deployment" is a meaningful metric for the human feedback loop. Add `fix_deployed_at` field to findings.
