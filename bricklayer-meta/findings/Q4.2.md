# Finding: Q4.2 — Fix regressions: did any deployed fixes introduce new failures?

**Question**: Of the fixes that were deployed during the Recall campaign, did any introduce regressions — new failures caused by the fix itself?
**Verdict**: HEALTHY
**Severity**: Low

## Summary

Zero confirmed fix-introduced regressions across 2 deployed fixes and 1 FALSE NEGATIVE correction. The simulation's `FIX_LOOP_REGRESSION_PROBABILITY=0.08` is too pessimistic — empirical regression rate is 0/2 = 0%. The references to "regression" in 10 findings are pre-existing bugs or investigation notes, not fix-caused regressions.

## Evidence

### Deployed Fixes and Regression Check

| Fix | Wave Deployed | Regression? | Evidence |
|-----|--------------|-------------|---------|
| LLM timeout tiers (Q15.3/Q16.3) | ~31 | **No** | Q31.7 confirms 17/17 callsites correct; no new failures in latency or correctness cluster post-Q31.7 |
| Reconcile audit (Q24.8) | Pre-campaign | **No** | Q31.4 reversed 7 FALSE NEGATIVE FAILUREs — the "fix" was always deployed; audit IS working |
| IS NULL partial fix for double-decay | ~31 (partial) | **No regression, but incomplete** | Q31.2 FAILURE: fix reduced system pass scope but second doubling persists due to Q33.2b root cause; this is a "fix that didn't fix" not a regression |

None of these represent a deployed fix introducing a new failure.

### The IS NULL Partial Fix — Not a Regression

The IS NULL fix deployed around wave 31 changed `_user_conditions(None)` to use `IsNullCondition`. This correctly scoped the system pass. However, the double-decay persisted because `get_distinct_user_ids()` returning `[0]` causes the per-user loop to run a second pass with `user_id=0` → `IsNullCondition` (same effect). This is the original root cause (Q33.2b), NOT a regression introduced by the IS NULL fix. The fix was correct but addressed the wrong part of the code path.

### Comparison with Simulation

The simulation uses `FIX_LOOP_REGRESSION_PROBABILITY=0.08` (8% chance each fix introduces a regression). Empirically:
- 2 fixes deployed, 0 regressions: rate = 0%
- Sample is too small (n=2) to reject 8% statistically
- But the empirical evidence is consistent with the hypothesis that careful, targeted fixes have near-zero regression risk in a well-tested codebase

## Verdict Rationale

HEALTHY — no fix-introduced regressions observed. The simulation's 8% regression rate may be appropriate for rapid/untested fixes but appears conservative for carefully diagnosed, code-level targeted fixes with test coverage (Q2.1 confirmed the codebase has passing tests).

## Mitigation Recommendation

Keep `FIX_LOOP_REGRESSION_PROBABILITY=0.08` as a conservative default for new campaigns. For campaigns with ≥80% test coverage and careful code review, consider lowering to 0.03-0.05.
