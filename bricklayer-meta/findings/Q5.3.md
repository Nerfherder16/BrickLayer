# Finding: Q5.3 — Creation domain portability: can BrickLayer stress-test a creative work or design system?

**Question**: Can BrickLayer be adapted to evaluate a creative work — a design system, a novel, a game, a visual language? What are the structural analogies and required changes?
**Verdict**: WARNING
**Severity**: Medium

## Summary

Partial portability. BrickLayer's loop structure (hypothesis → evidence → verdict → synthesis) maps to creative review workflows, but two components require significant redesign: (1) `simulate.py` has no clear analog for creative work evaluation (can't simulate whether a novel is coherent), and (2) the HEALTHY/WARNING/FAILURE verdict system assumes objective criteria, while creative evaluation is inherently subjective or multi-stakeholder. BrickLayer can work as a creative AUDIT tool but not as a creative SIMULATION tool.

## Evidence

### Structural Analogy Table

| BrickLayer (Software) | BrickLayer (Creative) | Notes |
|----------------------|----------------------|-------|
| `simulate.py` | Evaluation rubric script | Can check consistency, coverage, completeness — not aesthetic quality |
| `questions.md` | Review questions | "Does the design system cover all 12 breakpoints?" "Are all character arcs resolved?" |
| FAILURE verdict | Objective quality gap | Missing component, logical inconsistency, broken constraint |
| WARNING verdict | Subjective weakness | Under-developed section, inconsistent tone |
| HEALTHY verdict | Criterion met | Component exists and meets spec |
| Hypothesis generator | Creative critic | Generates follow-up questions from prior review findings |
| Fix loop | Revision loop | Revise the work, re-run the question |

### Where It Works: Constraint-Based Creative Systems

BrickLayer works best for creative domains with **explicit constraints** that can be objectively evaluated:

1. **Design systems** (like `figma-designer-guide.md`): Has rules for Auto Layout, variable naming, component naming, contrast ratios. BrickLayer can check compliance programmatically (e.g., "do all components in the Figma file use Auto Layout?").

2. **Technical writing and documentation**: Has coverage requirements, cross-reference accuracy, code example correctness. BrickLayer can verify "does the API reference cover all endpoints?" or "do all code examples run without error?"

3. **Game balance**: Has numerical constraints (win rates, damage ranges, resource curves). Simulate.py can model a game's balance mechanics and find failure cases.

4. **Architectural design review**: Has structural requirements (accessibility compliance, responsive breakpoints). BrickLayer can run automated checks.

### Where It Breaks: Aesthetic Quality

BrickLayer cannot evaluate:
- Whether a novel's prose is beautiful
- Whether a design is visually striking
- Whether a piece of music is emotionally resonant
- Whether a game is fun

These require subjective evaluation — the verdict system needs a **SUBJECTIVE** category with human-graded evidence rather than automated checking.

### The Simulate.py Problem

For software, `simulate.py` is a mathematical model of the system. For creative work, no equivalent model exists. The closest analog is:
- A coverage matrix (does the work cover all required elements?)
- A constraint checker (does the work violate any explicit rules?)
- A consistency analyzer (are there internal contradictions?)

These are useful but only capture the structural layer of creative work, not the quality layer.

### Required Adaptations

1. **Replace `simulate.py` with `evaluate.py`**: A rubric-based evaluator that checks objective criteria (coverage, constraint compliance, consistency). Accept that aesthetic quality is not automatable.

2. **Add SUBJECTIVE verdict**: For questions that require human judgment, the system should generate the question + evidence, then ask the human for the verdict. The campaign continues after human response.

3. **`constants.py` thresholds become rubric weights**: Coverage threshold = 0.90 (90% of required elements present); consistency threshold = 0.95 (5% contradiction rate is WARNING); constraint compliance = 1.00 (any violation is FAILURE).

4. **Creative specialist agents**: Replace `quantitative-analyst` with domain-specific agents: `design-critic`, `fiction-editor`, `game-designer`, `architect-reviewer`. These agents evaluate based on craft knowledge, not code analysis.

### Example: BrickLayer as a Design System Auditor

Applied to the UI design system in `figma-designer-guide.md`:
- FAILURE: A Figma component uses absolute positioning when Auto Layout was required
- WARNING: Contrast ratio 3.8:1 (between 3:1 WCAG AA large-text minimum and 4.5:1 normal-text minimum)
- HEALTHY: All variables use the required slash-separated naming convention
- INCONCLUSIVE: Whether the component naming is "close enough" to the code library (requires human judgment)

This is a genuine use case where BrickLayer adds value.

## Verdict Rationale

WARNING because:
1. Portability is real but requires new components not present in the current template
2. The verdict system needs extension (SUBJECTIVE category)
3. `simulate.py` has no creative analog — the structural simulation approach doesn't translate
4. The strongest use case (constraint-based design system auditing) is a narrow slice of the creative domain

## Mitigation Recommendation

1. Create a `creative-audit-template/` with: `evaluate.py` (rubric checker), `SUBJECTIVE` verdict support, and agent definitions for design/writing/game domains.

2. Document the constraint: BrickLayer evaluates creative CORRECTNESS (constraint compliance, coverage, consistency), not creative QUALITY. Set expectations accordingly.

3. The HUMAN OVERRIDE mechanism already exists for subjective verdicts — document it as the intended path for aesthetic evaluation.
