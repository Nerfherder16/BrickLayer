# Finding: Q5.2 — Research domain portability: can BrickLayer run against an academic or business research question?

**Question**: Can BrickLayer's campaign loop be adapted to stress-test a research hypothesis or business assumption, rather than a software system? What changes are required?
**Verdict**: HEALTHY
**Severity**: Info

## Summary

BrickLayer is directly applicable to research and business hypothesis testing. The campaign loop — hypothesis generation → evidence collection → verdict → synthesis — maps cleanly to the scientific method. The bricklayer-meta project you are currently running IS this use case: BrickLayer testing BrickLayer. The only required change is replacing `simulate.py`'s code execution with evidence-based evaluation.

## Evidence

### Structural Mapping

| BrickLayer (Software) | BrickLayer (Research) | Example |
|----------------------|----------------------|---------|
| `simulate.py` | Literature/data analysis script | Run a Monte Carlo model, fetch a dataset, run a regression |
| `questions.md` | Research hypotheses to test | "Does X correlate with Y?" "Is assumption Z empirically supported?" |
| FAILURE verdict | Hypothesis refuted | Evidence contradicts the assumption |
| HEALTHY verdict | Hypothesis confirmed | Evidence supports the assumption |
| WARNING verdict | Mixed evidence | Partially supported with caveats |
| `campaign-historian` agent | Literature reviewer | Reads papers, data, prior findings |
| `hypothesis-generator` | Research gap identifier | Generates follow-up questions from prior findings |
| Fix loop | Research revision loop | Update hypothesis, re-test with new data |

### This Campaign as Evidence

The bricklayer-meta campaign is BrickLayer running against itself as a research subject. The simulation is the "system under test." The findings (Q1.2: peer review over-corrects at high novelty; Q3.1: novelty curve is J-shaped, not monotonic decay) are research findings about BrickLayer's behavior — not software bugs.

The research application requires:
1. Replace `simulate.py` with a data analysis or model evaluation script
2. Replace `constants.py` thresholds with research acceptance criteria (e.g., p<0.05, effect_size>0.3)
3. Replace domain-specific agents with research-domain agents (economist-analyst, biostatistician, etc.)
4. Replace `correctness` runner with `data-analysis` runner that can fetch datasets

### Business Model Stress-Testing

BrickLayer originated as a business model stress-tester (the autosearch QUICKSTART.md describes it for "business model stress-testing"). The framework already:
- Supports a `regulatory-researcher` agent for legal/compliance questions
- Supports a `competitive-analyst` agent for market analysis
- Has a `project-brief.md` structure that maps to a business model canvas

For business hypothesis testing:
- FAILURE = assumption has been empirically refuted by market data or regulatory finding
- WARNING = assumption is risky but not definitively wrong
- HEALTHY = assumption is supported by available evidence

### Required Adaptations

1. **`simulate.py` → `analyze.py`**: Replace scenario simulation with evidence collection. For business research, this might be a script that fetches market data, runs a DCF model, or queries a knowledge base.

2. **`constants.py` thresholds**: Define what constitutes FAILURE/WARNING in research terms (p-values, effect sizes, confidence intervals, market size minimums).

3. **Research agents**: The question-designer should be domain-specific (e.g., prompt it as an economist, market researcher, or regulatory analyst). The existing agent slots already support this.

4. **`results.tsv` interpretation**: In research context, a FAILURE means "this assumption doesn't hold" — a positive finding, not a negative one.

### Limitation: Evidence Collection Automation

The hardest part is automating evidence collection. For software, BrickLayer reads actual code. For research, the equivalent is:
- Web search results (already available via Exa MCP)
- Dataset analysis (Python REPL)
- Literature review (context7, Firecrawl)
- Expert consultation (human override)

BrickLayer can run all of these in research mode with the existing runner infrastructure.

## Verdict Rationale

HEALTHY — research domain portability is inherent in the design. No architectural changes needed.

## Mitigation Recommendation

Create a `research-template/` alongside the existing `template/` directory with:
- A `analyze.py` stub that fetches and evaluates evidence rather than running a simulation
- A `project-brief.md` template written for research hypotheses
- Pre-configured `constants.py` with statistical research thresholds
- A `research-analyst` agent definition
