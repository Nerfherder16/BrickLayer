# D15.4: Does `_compute_score()` silently treat unclassified verdicts as failures?

**Status**: HEALTHY
**Date**: 2026-03-16
**Agent**: diagnose-analyst

## Evidence

Read `C:\Users\trg16\Dev\autosearch\bl\agent_db.py` in full. The score model (lines 113-121):

```python
def _compute_score(record: dict) -> float:
    total = record.get("runs", 0)
    if total == 0:
        return 1.0
    verdicts = record.get("verdicts", {})
    success = sum(verdicts.get(v, 0) for v in _SUCCESS_VERDICTS)
    partial = sum(verdicts.get(v, 0) for v in _PARTIAL_VERDICTS)
    return (success + partial * 0.5) / total
```

The formula sums only verdicts that appear in `_SUCCESS_VERDICTS` or `_PARTIAL_VERDICTS`. Any verdict not in either set contributes 0 credit. Since the denominator is `total` (all runs, including unclassified), an unclassified verdict silently depresses the score with no log, no exception, no warning. The hypothesis is structurally correct about the mechanism.

**Unclassified = 0.** Every verdict currently emitted by BL 2.0 appears in exactly one of the three frozensets.

One phantom entry exists: `DONE` is in `_SUCCESS_VERDICTS` but no runner or agent emits it. This is benign — `verdicts.get("DONE", 0)` returns 0 and has no effect.

## Analysis

The hypothesis describes a real latent vulnerability in the design: there is no guard — no assertion, no set-difference check at module load, no logging in `record_run` — that would catch a newly introduced verdict falling through the cracks. If a future agent mode adds a verdict (e.g., `"DEFERRED"` or `"SKIPPED"`) without updating all three frozensets, it will silently count as 0.0 credit with no diagnostic signal.

However, as of the current codebase snapshot, that condition does not exist. All 30 active verdicts are classified. The failure mode is theoretical, not realized.

**Verdict: HEALTHY** — the bug class exists structurally but zero verdicts are currently affected.
