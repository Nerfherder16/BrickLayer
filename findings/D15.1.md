# D15.1: sync_status_from_results() sentinel string mismatch for BL 2.0 question IDs

**Status**: DIAGNOSIS_COMPLETE
**Date**: 2026-03-16
**Agent**: diagnose-analyst

## Evidence

File: `C:/Users/trg16/Dev/autosearch/bl/questions.py`

Line 225 (the sentinel lookup inside `sync_status_from_results()`):
```python
next_block = text.find("\n## Q", block_start + 1)
```

The sentinel `"\n## Q"` is hardcoded. BL 2.0 question IDs use prefixes `D`, `F`, `A`, `V`, `M` — none of which begin with `Q`.

The block-locating logic (lines 220-226) works as follows:
1. `block_start` is found correctly via `text.find(f"## {qid} [")` — this uses the full ID, so it works for any prefix (line 220).
2. `next_block` is then found with `text.find("\n## Q", block_start + 1)` — this searches for the next BL 1.x-style header only.
3. `block_end = next_block if next_block != -1 else len(text)` — if `next_block == -1` (which is the case for every BL 2.0 question block since none use `Q`-prefix), `block_end` is set to `len(text)`.

Effect: for every BL 2.0 question, `block` spans from `block_start` to the end of the entire file. The `replace("**Status**: PENDING", ...)` on line 229-231 will still replace only the first occurrence of `**Status**: PENDING` within that over-large block, so it will find and update the correct question's Status field. However, if a later question in the file also has `**Status**: PENDING`, that later block is within the over-large slice too — but `replace(..., 1)` (the `1` argument) limits replacement to the first match, so only the target question's Status field is replaced.

The replace call is therefore functionally correct in isolation: the first `**Status**: PENDING` within the (too-wide) block slice is always the correct one for the target question ID, because `block_start` lands on that question's header.

There is one genuine corruption scenario: if the replacement produces a text string longer than the original (which it always does — `"DONE"` is longer than `"PENDING"`, `"DIAGNOSIS_COMPLETE"` is much longer), and a second question is processed in the same loop pass, `block_start` for the second question was computed on the original `text` before the first replacement extended it. Because `text` is mutated in-place inside the loop (line 232: `text = text[:block_start] + new_block + text[block_end:]`), subsequent iterations recompute `text.find(f"## {qid} [")` on the already-mutated string — so offsets do stay correct for later iterations. No corruption occurs from this path either.

However, the silent expansion to `len(text)` is a correctness hazard for a different edge case: if the target question's block does NOT contain `**Status**: PENDING` but a later question's block does, the replace call will update the wrong question's status. This would happen if a BL 2.0 question has already had its status updated (no longer PENDING) but the subsequent question in the file still has PENDING — and both are in `done_ids`. For the already-updated question, `"**Status**: PENDING" in block` (line 228) evaluates to False (since the first occurrence is already non-PENDING), so the replace is skipped. That guard prevents the wrong-question corruption in practice.

Net assessment: the sentinel bug does not corrupt status in the current data patterns because of two mitigating guards:
1. `replace(..., 1)` limits to the first match, which is always the correct field.
2. The `"**Status**: PENDING" in block` guard skips already-updated questions, preventing cross-contamination.

The bug is real but latent. It becomes an active failure only if a future question block contains `**Status**: PENDING` text in its body content (e.g., in a hypothesis or test field) before the actual `**Status**:` field — which would cause the wrong occurrence to be replaced.

## Analysis

Root cause: line 225 uses the literal sentinel `"\n## Q"` inherited from BL 1.x where all question IDs had the `Q` prefix. BL 2.0 introduced multi-prefix IDs (`D`, `F`, `A`, `V`, `M`). The sentinel never matches for BL 2.0 questions, causing `block_end` to always be `len(text)`.

The function does not corrupt status in the current test data because of the `replace(..., 1)` constraint and the PENDING guard. The bug is a latent correctness hazard, not an active failure.

The correct fix is to replace `"\n## Q"` with `"\n## "` (generic section boundary), matching the same generic pattern that `parse_questions()` already uses via its `block_pattern` regex on line 22-25.
