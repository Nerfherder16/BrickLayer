# D14.3: synthesize() writes to wrong path — findings/synthesis.md never created

**Status**: DIAGNOSIS_COMPLETE
**Date**: 2026-03-16
**Agent**: diagnose-analyst

## Evidence

### synthesizer.py — write path (lines 226–227)

```python
synthesis_path = project_dir / "synthesis.md"
synthesis_path.write_text(output, encoding="utf-8")
```

`project_dir` is the value passed into `synthesize()`. At the call site in `campaign.py` line 712:

```python
synthesis_result = synthesize(cfg.project_root, wave=None)
```

So the synthesis is always written to:

```
{project_root}/synthesis.md        ← actual write destination
```

### campaign.py — skill-forge consumer reads a different path (lines 731–732)

```python
synthesis_path = cfg.project_root / "findings" / "synthesis.md"
_spawn_agent_background(
    "skill-forge",
    f"synthesis_md={synthesis_path}\n"
    ...
)
```

The `skill-forge` background agent is handed:

```
{project_root}/findings/synthesis.md   ← path passed to skill-forge
```

### _build_findings_corpus() — what the corpus builder reads (synthesizer.py lines 18, 33–37)

```python
findings_dir = project_dir / "findings"
...
for finding_file in sorted(findings_dir.glob("*.md")):
```

The corpus builder reads all `*.md` files under `{project_root}/findings/`. If `synthesis.md` were placed at the root it would never be included in its own corpus (irrelevant here), but the key consumer — `skill-forge` — is explicitly pointed at `findings/synthesis.md` which is never written.

### cfg.findings_dir (config.py lines 88, 102)

```python
cfg.findings_dir = project_dir / "findings"
cfg.findings_dir.mkdir(exist_ok=True)
```

`findings_dir` is always `{project_root}/findings/`, and the directory is guaranteed to exist. Individual findings (e.g., `D14.1.md`) land there. But `synthesize()` writes to `{project_root}/synthesis.md`, one directory level up.

## Analysis

Causal chain:

1. `synthesize()` computes `synthesis_path = project_dir / "synthesis.md"` (line 226 of `synthesizer.py`). `project_dir` = `cfg.project_root`.
2. Output is written to `{project_root}/synthesis.md`.
3. `campaign.py` line 712 calls `synthesize(cfg.project_root)` and receives the path back.
4. Lines 714–715 correctly read from the returned path — that part works because it uses the returned `Path` object directly.
5. However, lines 731–732 construct `synthesis_path = cfg.project_root / "findings" / "synthesis.md"` independently and pass it to `skill-forge`. This path is never written by `synthesize()`.
6. Result: `skill-forge` always receives a path pointing to a file that does not exist.

The two paths in play:
- **Written**: `{project_root}/synthesis.md`
- **Read by skill-forge**: `{project_root}/findings/synthesis.md`

These never resolve to the same file. `skill-forge` operates on a nonexistent path every wave.

## Fix Specification

- **File**: `C:/Users/trg16/Dev/autosearch/bl/synthesizer.py`
- **Line**: 226
- **Change**: Change `synthesis_path = project_dir / "synthesis.md"` to `synthesis_path = project_dir / "findings" / "synthesis.md"`. This places the output inside `findings/` where `skill-forge` (and any other downstream consumer) expects it. The `findings/` directory is guaranteed to exist at this point because `cfg.findings_dir.mkdir(exist_ok=True)` is called during `init_project()`.
- **Verification**: After the fix, run `python -m bl run --project bl --single D14.3` (or any question that completes a wave), then assert `ls {project_root}/findings/synthesis.md` exists and `ls {project_root}/synthesis.md` does not.
  Alternatively: `python -c "from pathlib import Path; from bl.synthesizer import synthesize; p = synthesize(Path('projects/bl'), dry_run=False); print(p)"` and confirm the returned path ends with `findings/synthesis.md`.
- **Risk**: The `campaign.py` lines 714–715 read synthesis back via the returned `Path` object, so they will continue to work correctly once the written path is corrected. No other callers of `synthesize()` in `campaign.py` hardcode the root-level path. The only regression surface is if any external script explicitly reads `{project_root}/synthesis.md` by convention — that stale root-level file will stop being produced, which is the intended correction.
