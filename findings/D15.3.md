## Evidence

Read `C:/Users/trg16/Dev/autosearch/bl/crucible.py` in full. Both bugs are visible at exact lines.

**Bug 1 — `_score_synthesizer()`, line 300:**
```python
"has_finding_refs": (1.0 if re.search(r"Q\d+\.\d+", text) else 0.0, 0.30),
```
The regex `Q\d+\.\d+` matches only `Q`-prefixed IDs (e.g. `Q14.1`). BL 2.0 synthesis.md files reference findings as `D14.1`, `F12.3`, `A13.1`, `V11.2`, etc. No match — check scores 0.0. This check carries weight 0.30, so the synthesizer's maximum achievable score on a BL 2.0 synthesis is 0.70.

**Bug 2 — `_score_quantitative_analyst()`, line 328:**
```python
for fpath in sorted(findings_dir.glob("Q*.md")):
```
The glob `Q*.md` matches only files whose names begin with `Q`. The confirmed BL 2.0 findings directory (`C:/Users/trg16/Dev/autosearch/findings/`) contains: `D14.1.md`, `D14.2.md`, `D14.3.md`, `D15.1.md`, `D15.2.md` — zero Q-prefixed files. `perf_files` will always be empty, triggering the early-return path at line 336-339 with score 0.0.

## Analysis

Causal chain:

1. BL 2.0 adopted domain-prefixed question/finding IDs (`D`, `F`, `A`, `V`) rather than the legacy `Q` prefix.
2. `_score_synthesizer()` checks for `Q\d+\.\d+` — a hardcoded BL 1.x pattern — to verify the synthesizer cited findings. Any BL 2.0 synthesis.md that correctly cites `D14.1` etc. fails this check, losing 0.30 weight unconditionally.
3. `_score_quantitative_analyst()` uses `glob("Q*.md")` to discover finding files. In any BL 2.0 project the glob returns an empty list, the scorer hits the early return, and records score 0.0 regardless of how many valid D/F/A-prefixed finding files exist and how well-formed they are.

Both failures are structural: the scorers will never pass these checks on BL 2.0 campaigns without a code change.

## Fix Specification

**Fix 1 — `_score_synthesizer()` regex**
- File: `C:/Users/trg16/Dev/autosearch/bl/crucible.py`
- Line: 300
- Change: Replace `r"Q\d+\.\d+"` with `r"\b[A-Z]\d+\.\d+"` so any single-letter-prefixed ID (Q, D, F, A, V, etc.) matches.
  ```python
  # Before
  "has_finding_refs": (1.0 if re.search(r"Q\d+\.\d+", text) else 0.0, 0.30),
  # After
  "has_finding_refs": (1.0 if re.search(r"\b[A-Z]\d+\.\d+", text) else 0.0, 0.30),
  ```
- Verification: `python -c "import re; text='See D14.1 and F12.3'; print(bool(re.search(r'\\b[A-Z]\\d+\\.\\d+', text)))"` — should print `True`.
- Risk: The looser pattern could match non-finding references that happen to look like `X##.##`. In practice, synthesis.md bodies do not contain other content matching that pattern. No adjacent rubric checks are affected since this is an isolated key in `checks_raw`.

**Fix 2 — `_score_quantitative_analyst()` glob**
- File: `C:/Users/trg16/Dev/autosearch/bl/crucible.py`
- Line: 328
- Change: Replace `findings_dir.glob("Q*.md")` with `findings_dir.glob("*.md")` (with `synthesis.md` excluded) so all finding files regardless of prefix are scanned.
  ```python

  # After
  for fpath in sorted(f for f in findings_dir.glob("*.md") if f.name != "synthesis.md"):
  ```
- Verification: Point the scorer at `C:/Users/trg16/Dev/autosearch/findings/` which contains only D-prefixed files and confirm score > 0.0. Quick check: `python -c "from pathlib import Path; p=Path('C:/Users/trg16/Dev/autosearch/findings'); print([f.name for f in p.glob('*.md') if f.name != 'synthesis.md'])"` — should list all five D-prefixed files.
- Risk: The scorer will now also inspect findings from non-performance domains (regulatory, competitive). However, the content filter on line 333 (`re.search(r"\b(performance|D1|D5|PERFORMANCE)\b", content)`) already narrows `perf_files` to relevant files, so non-performance findings will be read but not scored. No regression on BL 1.x campaigns because `Q*.md` files will also be included under `*.md`.
