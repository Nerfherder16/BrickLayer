## Analysis

### What the check looks for

The check scans `wave1_text` for six fixed code strings (`"D1"` through `"D6"`) plus BL 1.x semantic keywords:
- D1: `"D1"`, `"performance"`, `"load"`, `"latency"`
- D2: `"D2"`, `"legal"`, `"compliance"`, `"regulatory"`
- D3: `"D3"`, `"market"`, `"competitive"`, `"analogue"`
- D4: `"D4"`, `"correctness"`, `"accuracy"`, `"quality"`
- D5: `"D5"`, `"benchmark"`, `"ablation"`, `"model"`
- D6: `"D6"`, `"security"`, `"auth"`, `"vulnerability"`

### Would BL 2.0 question IDs trigger it?

A BL 2.0 `questions.md` contains blocks headed `## D14.1`, `## F14.1`, `## A14.1`, `## V14.2`, etc.

Evaluation of each keyword against BL 2.0 content:

| Code | Code string present in BL 2.0 IDs? | Semantic keywords present? |
|------|--------------------------------------|----------------------------|
| D1 | **Yes** — `"D1"` appears in `"D14.1"`, `"D15.2"`, etc. | Possibly — depends on question text |
| D2 | **Yes** — `"D2"` appears in `"D2x.y"` pattern IDs | Possibly |
| D3 | **Yes** — same pattern | Possibly |
| D4 | **Yes** — same pattern | Possibly |
| D5 | **Yes** — same pattern | Possibly |
| D6 | **Yes** — same pattern | Possibly |

This is a **false-positive partial match**. The string `"D1"` is a substring of `"D14"`, `"D15"`, etc. Python's `in` operator on strings performs substring matching, so `"D1" in "## D14.1 — Does the diagnose-analyst..."` evaluates to `True`.

However, this also means the `"D2"` code string is a substring of `"D2x"` patterns. Any campaign with wave numbers >= 2 will have question IDs like `D2x.y`, `F2x.y`, which contain `"D2"` only if the prefix happens to be `D`. A BL 2.0 campaign with prefixes `D`, `F`, `A`, `V`, `M` will:

- Match `"D1"` → `D1` registered (via D1x.y IDs)
- Match `"D2"` → `D2` registered (via D2x.y IDs, if any D-prefix questions exist at wave >= 20)
- Miss `"D3"` through `"D6"` unless the semantic keywords (`"market"`, `"competitive"`, etc.) or the literal strings `"D3"` ... `"D6"` appear in question text

### The actual scoring behavior in BL 2.0

The `wave1_text` variable is scoped to Wave 1 content (lines before the first Wave 2+ question ID — line 230-231). In BL 2.0 Wave 14+, Wave 1 questions carry IDs `D1.x`, `F1.x`, `A1.x`, `V1.x`, `M1.x`. The text `"D1"` appears in every `D1.x` question header, so `D1` will always register as a hit. The strings `"D2"` through `"D6"` will NOT appear in Wave 1 IDs in a BL 2.0 campaign (Wave 1 IDs have wave number = 1).

Therefore, in a BL 2.0 campaign where Wave 1 questions are `D1.x`, `F1.x`, `A1.x`, `V1.x`, `M1.x`:
- `"D1"` → hit (present in `D1.x` headers)
- `"D2"` → **miss** (no `D2.x` in wave 1)
- `"D3"` → **miss**
- `"D4"` → **miss**
- `"D5"` → **miss**
- `"D6"` → **miss**

Unless the question *text* happens to contain words like `"performance"`, `"legal"`, `"market"`, etc., the semantic keywords are also largely a miss unless the question designer happened to use those exact BL 1.x vocabulary terms.

Typical outcome: `domain_hits = {"D1"}`, `domains_score = 1/6 = 0.167`.

The maximum recoverable score from the other three checks (`has_thresholds` 0.30, `has_status` 0.15, `has_hypothesis` 0.20, total weight 0.65) is 0.65. With a perfect score on those, the overall maximum for a BL 2.0 campaign is:

```
max_score = (1/6 * 0.35) + (1.0 * 0.65) = 0.058 + 0.65 = 0.708
```

The promote threshold is 0.80 (line 26). A question-designer producing perfect BL 2.0 output cannot be promoted — it is capped at 0.708. A question-designer with typical (not perfect) scores on the other checks will score in the 0.40-0.55 range and be **flagged or retired**.

### Root cause

Line 247-259: `domain_keywords` is hardwired to BL 1.x domain codes D1-D6. The denominator on line 260 is hardcoded to `6.0`. BL 2.0 uses operational mode prefixes (D=Diagnose, F=Fix, A=Audit, V=Validate, M=Monitor). There is no mapping from BL 2.0 prefixes to the rubric, and the denominator assumes exactly 6 domains exist.
