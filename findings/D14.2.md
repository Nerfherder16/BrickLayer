## Evidence

### goal.py — focus default (line 128)

```python
focus_str = ", ".join(goal["focus"]) if goal["focus"] else "all domains (D1–D6)"
```

When no `Focus:` field is supplied in `goal.md`, the prompt injects the literal string `"all domains (D1–D6)"` under the `DOMAIN FOCUS:` heading. This primes the LLM to think in BL 1.x D1–D6 terminology before it ever sees the format examples.

### goal.py — prompt template bracket tag examples (lines 158, 170)

```python
## QG1.1 [D1] Short descriptive title
...
## QG1.2 [D4] Another question title
```

The two format examples shown to the LLM use `[D1]` and `[D4]` — BL 1.x domain codes — as the bracket tag in the question header. These are the only concrete examples the LLM sees; it will reproduce this pattern for every question it generates.

### questions.py — what parse_questions() does with the bracket tag (lines 22–58)

```python
block_pattern = re.compile(
    r"^## ([\w][\w.-]*) \[(\w+)\] (.+?)$",
    re.MULTILINE,

...
tag_raw = m.group(2)          # captures "D1", "D4", etc.
mode_raw = tag_raw.lower()    # becomes "d1", "d4"

# C-30: derive question_type from header tag or explicit **Type** field
```

The `mode` field stored in the question dict is set from the body `**Mode**` field (F5.1 override), falling back to `mode_raw` (which would be `"d1"`, `"d4"`, etc.). The `question_type` derivation uses `_CODE_AUDIT_TAGS` and `_BEHAVIORAL_TAGS` — neither contains any `"d1"`–`"d6"` value, so all LLM-generated questions silently fall through to `question_type = "behavioral"`.

### The operational routing impact

`parse_questions()` stores `operational_mode` from the body `**Operational Mode**` field, defaulting to `"diagnose"` (line 78). The prompt template (lines 155–184 of goal.py) does **not** include an `**Operational Mode**` field in the example blocks — only `**Mode**: agent`. The LLM therefore emits no `**Operational Mode**` field, so every generated question gets `operational_mode = "diagnose"` regardless of what was intended.

The bracket tag `[D1]`/`[D4]` contributes to `mode_raw` as `"d1"`/`"d4"`. If no body `**Mode**` override is present, the question dict gets `mode = "d1"` — a value that no BL 2.0 dispatcher recognises as a valid operational mode.

## Analysis

Causal chain:

1. `goal.md` omits `Focus:` → `_build_prompt()` sets `focus_str = "all domains (D1–D6)"` (line 128).
2. Prompt shows LLM two header examples: `[D1]` and `[D4]` (lines 158, 170).
3. LLM generates questions with `[D1]`–`[D6]` bracket tags by example imitation.
4. `parse_questions()` captures the bracket tag as `mode_raw = "d1"` (or `"d4"`, etc.).
5. No body `**Mode**` field override is emitted by the LLM (prompt template omits `**Operational Mode**`).
6. Result: every goal-generated question enters the campaign with `mode = "d1"` — an unrecognised BL 2.0 operational mode string — and `operational_mode = "diagnose"` (silent default).
7. The `question_type` always falls through to `"behavioral"` regardless of the intended domain, losing the `code_audit` distinction entirely for goal-generated questions.

The failure is silent: `parse_questions()` does not reject or warn on unknown bracket tags, so questions are enqueued but carry a corrupted `mode` value that dispatchers cannot route to any BL 2.0 operational mode.
