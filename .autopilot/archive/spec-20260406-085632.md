# Spec: BrickLayer 2.0 Partial JS Engine Migration

**Created**: 2026-04-06
**Branch**: autopilot/js-engine-wiring

## Goal

Wire the existing-but-unused JS engine modules in `masonry/src/engine/` into the MCP server hot paths. Three high-frequency MCP tools (`masonry_route`, `masonry_status`, `masonry_registry_list`) currently run inline Python logic — replace with calls to Node.js CLI wrappers that use the JS engine directly. Wire JS healloop into `masonry_run_question`. Extend `masonry-hook-watch.js` to emit a language-choice reminder when new hook/engine files are created. Update `docs/reviews/hook-map.md` to reflect all changes.

## Architecture

### Current State
- **MCP server** (`masonry/mcp_server/server.py`): Python FastMCP, inline tool implementations
- **JS engine** (`masonry/src/engine/*.js`): 55+ Node.js ports of `bl/*.py` — complete, tested, NOT wired
- **Hooks** (`masonry/src/hooks/*.js`): All Node.js, run in Claude Code hook system
- **Python engine** (`bl/`): Campaign runner, DSPy, scoring, crucible — stays Python

### Target State
- New `masonry/src/engine/cli/` directory with 4 JSON-outputting CLI wrappers
- Python MCP tools `masonry_route`, `masonry_status`, `masonry_registry_list` call Node.js CLI via `subprocess.run()`
- `masonry_run_question` calls JS healloop CLI after FAILURE/DIAGNOSIS_COMPLETE verdict
- `masonry-hook-watch.js` emits language-choice reminder when new `.js`/`.py` files appear in `hooks/` or `engine/`
- `docs/reviews/hook-map.md` updated to reflect new wiring

### Wiring Pattern
```
Claude Code → MCP server (Python/FastMCP)
                ↓ subprocess.run(["node", "cli/wrapper.js", "--args"], timeout=15)
              Node.js CLI wrapper (masonry/src/engine/cli/*.js)
                ↓ require()
              JS engine module (masonry/src/engine/*.js)
                ↓ reads
              YAML / JSON / MD files on disk
```

Each Python MCP handler keeps a fallback: if Node.js subprocess fails, existing inline Python logic runs with a warning log.

## Constraints — DO NOT CHANGE

- `bl/runners/`, `bl/tmux/`, `bl/crucible.py`, DSPy pipeline — stays Python, zero changes
- MCP server host stays Python FastMCP (`uv run python masonry/mcp_server/server.py`)
- All 71/71 existing hook benchmark tests must continue to pass
- All existing `masonry/tests/engine-*.test.js` tests must continue to pass

### Task 8 — Wire healloop into masonry_run_question post-verdict
**Mode**: python
**Files**: `masonry/mcp_server/server.py`

**What to build**: After `masonry_run_question` completes and produces a verdict, call `node masonry/src/engine/cli/healloop.js` if verdict is FAILURE or DIAGNOSIS_COMPLETE. Append heal loop result to the MCP tool response. Never block on failure.

```python
# After existing verdict is determined
if verdict in ("FAILURE", "DIAGNOSIS_COMPLETE"):
    heal_result = _call_js_engine("cli/healloop.js",
        ["--question-id", qid, "--verdict", verdict, "--project-dir", project_dir],
        timeout=300)  # heal loop can take minutes
    if heal_result:
        response["heal_loop"] = heal_result
```

**Acceptance criteria**:
- [ ] FAILURE verdict triggers healloop subprocess call
- [ ] DIAGNOSIS_COMPLETE verdict triggers healloop subprocess call
- [ ] HEALTHY/WARNING verdicts do NOT trigger healloop call
- [ ] Healloop failure does not break the main response
- [ ] Heal loop results appended to response under `"heal_loop"` key

### Task 9 — Extend masonry-hook-watch.js with language-choice reminder

**What to build**: Extend `shouldTrigger()` to also detect when a NEW hook file or engine module file is created (tool_name is Write, not Edit, and the file doesn't yet exist). **Reminder text** (emit to stderr):
```

─────────────────────────────────────────────────────
Language choice guide: docs/hook-creation-guide.md
Quick rule:
  Node.js  → hot path (fires every tool use), MCP tool integration, I/O-bound
  Python   → DSPy/ML, tmux agent spawn, campaign state mutation
─────────────────────────────────────────────────────
```

Detect "new file" by checking if `tool_name === "Write"` AND file did not exist before (check `fs.existsSync` at hook invocation time is unreliable since file is written before hook fires — use the `tool_name` field: Write = new or overwrite, Edit = modify existing).

Actually, the reliable signal is: `tool_name === "Write"` for new files (Write creates; Edit modifies). Use that signal.
