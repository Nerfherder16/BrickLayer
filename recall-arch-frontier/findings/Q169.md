# Q169 — injection_log and metrics_snapshot Retention Gap

**Question**: Do the injection_log and metrics_snapshot tables in PostgreSQL have the same retention gap as audit_log — does any scheduled worker, migration, or trigger bound their row counts, and if not, what are their current sizes and growth rates?

**Verdict**: CONFIRMED_ABSENCE

---

## DDL Analysis

Both tables are defined in `src/storage/postgres_store.py` within the single `_CREATE_TABLES` SQL constant (lines 64–85). Neither carries any TTL, partition clause, or expiry column in the DDL.

### injection_log (lines 72–85)

```sql
CREATE TABLE IF NOT EXISTS injection_log (
    id              BIGSERIAL PRIMARY KEY,
    timestamp       TIMESTAMPTZ NOT NULL DEFAULT now(),
    query           TEXT NOT NULL,
    domain          TEXT,
    injected_ids    TEXT[],
    similarities    FLOAT[],
    prompt_length   INTEGER,
    result_count    INTEGER,
    query_type      TEXT,
    is_first_prompt BOOLEAN DEFAULT false
);
CREATE INDEX IF NOT EXISTS idx_injection_log_timestamp ON injection_log(timestamp);
CREATE INDEX IF NOT EXISTS idx_injection_log_domain ON injection_log(domain);
```

Subsequent migrations (lines 225–278) add four more columns (`max_raw_similarity`, `injected_types`, `skipped`, `dedup_count`, `cross_session_hits`) — none of which introduce any expiry or row-limit constraint. No PARTITION BY clause exists. No CHECK or trigger for row count is present.

### metrics_snapshot (lines 64–70)

```sql
CREATE TABLE IF NOT EXISTS metrics_snapshot (
    id          BIGSERIAL PRIMARY KEY,
    timestamp   TIMESTAMPTZ NOT NULL DEFAULT now(),
    counters    JSONB NOT NULL,
    gauges      JSONB NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_metrics_snapshot_timestamp ON metrics_snapshot(timestamp);
```

The schema is minimal — just a timestamped JSONB pair. No partition, no TTL column, no row limit.

### Additional unbounded tables identified (same pattern)

During the search, four more tables in the same `_CREATE_TABLES` block were found with identical structural characteristics and confirmed zero retention:

- `prompt_metrics` — per-LLM-call metrics log, written by `src/core/prompts.py`
- `tuning_ledger` — auto-tuner parameter change log, written every 6h by `run_auto_tuner`
- `profile_proposals` — user profile drift proposals, written by weekly `run_profile_drift`
- `membench_runs` / `membench_healing_runs` — benchmark run history

None of these have DELETE/TRUNCATE anywhere in the codebase.

## Growth Rate Projection

### injection_log

**Trigger**: The `recall-retrieve.js` hook fires `POST /search/track-injection` on **every UserPromptSubmit event** — both when results are injected (line 808) and when retrieval returns zero results (line 764). One row per hook invocation, unconditionally.

This means:
- Every Claude Code prompt Tim types fires exactly one injection_log row.
- At Tim's stated usage rate (~100–300 prompts/day across multiple machines):

| Scenario | Rows/day | Rows/90 days | Rows/year |
|---|---|---|---|
| Low (100 prompts/day) | 100 | 9,000 | 36,500 |
| Mid (200 prompts/day) | 200 | 18,000 | 73,000 |
| High (300 prompts/day) | 300 | 27,000 | 109,500 |

Row size estimate: `id` (8B) + `timestamp` (8B) + `query` TEXT (up to 500 chars = ~500B) + `domain` (~20B) + `injected_ids` TEXT[] (~200B for 5 UUIDs) + `similarities` FLOAT[] (~40B) + scalar columns (~50B) + array overhead = **~850–1100 bytes/row average**.

Storage at 200 rows/day:
- 90 days: 18,000 rows × 1KB ≈ **18 MB** (table data only)
- 1 year: 73,000 rows × 1KB ≈ **73 MB**
- 3 years: 219,000 rows × 1KB ≈ **219 MB**

Index overhead (B-tree on timestamp, B-tree on domain): roughly 40–60% of table size additional.

### metrics_snapshot

**Trigger**: `save_metrics_snapshot` runs hourly at :30 (cron schedule line 363–368). One row per hour, unconditionally.

Row count is strictly deterministic: **24 rows/day**.

Row size: `id` (8B) + `timestamp` (8B) + `counters` JSONB + `gauges` JSONB. The metrics object contains all Prometheus-style counters and gauges. Based on `src/core/metrics.py` (which tracks ~30–50 named metrics), the JSONB payload is approximately 2–4 KB per snapshot.

| Period | Rows | Storage estimate |
|---|---|---|
| 90 days | 2,160 | ~6–9 MB |
| 1 year | 8,760 | ~26–35 MB |
| 3 years | 26,280 | ~79–105 MB |

**Absolute volume note**: metrics_snapshot grows more slowly than injection_log (24 vs 100–300 rows/day) but is not bounded. Historical metrics beyond 90 days have zero analytical value for operational dashboards; they only answer "what were the counter values 18 months ago," which no current dashboard feature queries.

## Comparison to audit_log Gap

The structural gap is **identical** across all three tables:

| Attribute | audit_log | injection_log | metrics_snapshot |
|---|---|---|---|
| DDL TTL/partition | None | None | None |
| DELETE anywhere | 0 matches | 0 matches | 0 matches |
| TRUNCATE anywhere | 0 matches | 0 matches | 0 matches |
| pg_partman | None | None | None |
| Cron cleanup | None | None | None |
| Natural size bound | No | No | No |
| Monotonic timestamp index | B-tree (standard) | B-tree (standard) | B-tree (standard) |
| Growth rate (rows/day) | ~18x memory creation rate (~600+) | 100–300 | 24 |

All three tables:
- Use BIGSERIAL primary keys (no natural ceiling)
- Have standard B-tree indexes on `timestamp` (BRIN would be appropriate for all)
- Write continuously with no compensating delete path
- Have no foreign key relationships that would cascade-limit their size

**One structural difference**: audit_log writes at ~18x the memory creation rate because every memory operation generates multiple audit events (store, retrieve, gc_delete, hygiene_run, etc.). injection_log writes at ~1x the retrieval rate. metrics_snapshot writes at exactly 24/day. The gap is identical in kind but audit_log is currently the fastest-growing (~600+ rows/day vs 100–300 for injection_log).

**Unified remediation feasibility**: Because all three tables follow the same `timestamp`-based monotonic append pattern, a single retention worker can handle all three with minimal code. The Q161 finding's proposed pattern (`DELETE FROM audit_log WHERE timestamp < NOW() - INTERVAL '90 days'`) applies verbatim to injection_log and metrics_snapshot by table name substitution. A single `run_log_retention` cron job with a configurable per-table retention window is the minimal viable fix.

## Key Findings

1. **injection_log has zero retention** — confirmed by exhaustive grep (0 DELETE matches, 0 TRUNCATE matches) across all Python, SQL, shell, and config files in the Recall codebase.

2. **metrics_snapshot has zero retention** — same confirmation. The only worker that touches metrics_snapshot is `save_metrics_snapshot`, which only INSERTs.

3. **injection_log grows at 100–300 rows/day** — triggered once per UserPromptSubmit hook fire, which is once per Claude Code prompt. At 200 prompts/day, this is 18,000 rows/90 days and 73,000 rows/year at ~73 MB.

4. **metrics_snapshot grows at exactly 24 rows/day** — cron-driven hourly snapshots. Slower growth but no ceiling.

5. **The gap is structural and identical to audit_log** — all three tables use BIGSERIAL + timestamp-indexed unbounded appends with no compensating delete path anywhere in the codebase or infrastructure.

6. **Four additional tables are in the same state** — `prompt_metrics`, `tuning_ledger`, `profile_proposals`, and `membench_runs` also have zero retention. The total unmanaged PostgreSQL log surface is at minimum 7 tables.

7. **A unified 3-table (or 7-table) retention worker is a single engineering effort** — all tables share the same `timestamp < NOW() - INTERVAL 'N days'` predicate pattern. One ARQ cron job with a table-retention config dict covers the entire gap.

8. **BRIN indexes would reduce index overhead ~1000x on all three tables** — identical to the Q161 recommendation for audit_log. injection_log and metrics_snapshot are monotonic append tables with no random-access write pattern, making BRIN optimal.

## Ideas

### Idea: unified-log-retention-worker
- **Novelty**: 0.71
- **Evidence**: 0.95 — exhaustive grep confirms zero retention on all target tables; cron schedule confirms no existing worker covers this
- **Feasibility**: 0.92 — trivial to implement: one new ARQ cron function with a config dict mapping table name to retention interval; can be added to `src/workers/hygiene.py` or as `src/workers/log_retention.py`
- **Description**: Add a single `run_log_retention` cron job (daily at 6:00am, staggered from existing jobs) that iterates a configurable dict `{table: interval}` and issues `DELETE FROM {table} WHERE timestamp < NOW() - INTERVAL '{interval}'`. Initial config: `audit_log: 90 days`, `injection_log: 90 days`, `metrics_snapshot: 180 days`, `prompt_metrics: 90 days`, `tuning_ledger: 365 days`. Emit per-table deleted-row counts to `audit_log` (action `log_retention_run`) and to the metrics system. Total implementation: ~60 lines in a new worker file plus 8 lines in `workers/main.py` cron config. Eliminates unbounded growth across the entire PostgreSQL log tier in one commit.

### Idea: brin-index-migration-for-log-tables
- **Novelty**: 0.67
- **Evidence**: 0.95 — Q161 already validated BRIN for audit_log; injection_log and metrics_snapshot have identical monotonic-append access patterns
- **Feasibility**: 0.88 — one `_MIGRATE_BRIN_LOG_INDEXES` migration constant added to `postgres_store.py`; BRIN index creation is non-blocking (`CREATE INDEX CONCURRENTLY`) and takes seconds on current row counts
- **Description**: Add a migration that replaces the B-tree `idx_injection_log_timestamp` and `idx_metrics_snapshot_timestamp` indexes with BRIN equivalents. BRIN is correct here because both tables only append in timestamp order — no out-of-order writes exist. Reduces index storage from O(N log N) to O(pages/128), a ~1000x reduction. Migration constant pattern already established in `postgres_store.py` for injection_log column additions (lines 225–278); BRIN migration follows the same DO $$ pattern.

### Idea: metrics-snapshot-downsample-instead-of-delete
- **Novelty**: 0.70
- **Evidence**: 0.82 — growth rate of 24 rows/day is deterministic; historical metrics value decreases rapidly beyond 7–30 days for operational dashboards
- **Feasibility**: 0.75 — requires a downsample step instead of pure DELETE; implementation is a two-phase SQL: aggregate older snapshots into hourly → daily granularity, then delete the per-hour originals
- **Description**: Rather than deleting metrics_snapshot rows older than 90 days, downsample them: collapse 24 hourly rows into 1 daily row (JSONB average of numeric counters, JSONB last-value for gauges) using a `INSERT ... SELECT AVG ... GROUP BY date_trunc('day')` into a `metrics_snapshot_daily` table, then DELETE the originals. This preserves long-term trend data (useful for capacity planning, regression detection after deployments) while capping storage at ~365 daily rows/year vs 8,760 hourly rows/year. Novelty comes from the fact that the current system has no concept of metric data lifecycle tiers — it treats all historical snapshots as equally current.
