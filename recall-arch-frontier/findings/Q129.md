# Finding: Q129 — WAL Patterns for co_retrievals Durability

**Question**: What can write-ahead log (WAL) design in distributed databases teach about making the co_retrievals reinforcement write durable and idempotent — specifically, should co_retrievals UPSERTs be buffered in a local append-only log before the PostgreSQL write, and does this pattern eliminate the residual write-loss window identified in Q112 (the 1-2 second Redis RENAME window where hooks fail with no such key)?
**Question Type**: ADJACENT
**Verdict**: PROMISING
**Severity**: High
**Source field**: Write-ahead logging in production database systems — PostgreSQL WAL (Mohan et al., ARIES 1992), SQLite WAL mode (D. Richard Hipp, 2010), the Outbox pattern in microservices architecture (Chris Richardson, Microservices Patterns 2018), Apache Kafka idempotent producer (KIP-98, 2017)

## Step 1: Survey of WAL-Derived Mechanisms

The relevant problem space is this: a hook fires, contacts the Recall API, and the API must write a co_retrievals UPSERT to PostgreSQL. If that write fails — due to the 1-2 second Redis RENAME window causing the epoch INCR to error, or due to any other transient fault — the co-retrieval frequency signal for that session is silently lost. The UPSERT carries `co_retrieval_count + 1` and `last_reinforced_epoch` updates. Lost writes mean the reinforcement signal drifts: pairs that were co-retrieved in a session do not get credit, and the graph underrepresents actual co-occurrence patterns.

Five WAL-derived mechanisms are worth examining:

**1. ARIES write-ahead logging (Mohan et al. 1992)**: The foundational rule — no data page is written to disk until its log record is written first. If the write fails, the log is replayed. The key properties: sequential log writes are fast (no random I/O), idempotency is enforced by log sequence numbers (LSNs), and recovery is automatic on restart. Maps to: buffering co_retrievals UPSERTs in a sequential append-only file before the PostgreSQL commit, replaying on failure.

**2. SQLite WAL mode (Hipp 2010)**: Instead of writing to the main database file, WAL mode writes to a separate `.wal` file. Readers see a consistent snapshot; writers write to the WAL without blocking readers. A checkpoint process periodically folds the WAL into the main database file. The relevant mechanism: the WAL file is append-only, each entry is self-contained (contains the full after-image of the page), and the WAL survives crashes. The application never needs to "commit" explicitly — the WAL IS the committed state until checkpoint.

**3. Outbox pattern (Richardson 2018)**: A microservices pattern for reliable event publishing. Instead of writing to a target service directly (which can fail), write to a local "outbox" table in the same database transaction as the triggering write. A separate background worker reads the outbox table and delivers messages to the target. On delivery success, the outbox entry is deleted. If the target is unavailable, the outbox entry stays and the worker retries. The pattern guarantees at-least-once delivery. Idempotency at the consumer (the `ON CONFLICT DO UPDATE` in co_retrievals) converts this to exactly-once semantics.

**4. Kafka idempotent producer (KIP-98, 2017)**: Each producer is assigned a producer ID (PID), and each write carries a monotonically increasing sequence number per partition. The broker rejects any write with a sequence number it has already seen (deduplicated). If the producer retries a failed write (same sequence number), the broker accepts it idempotently. The key mechanism: deduplication lives at the consumer (broker), not the producer. The producer just needs to replay with the same sequence number.

**5. PostgreSQL 2PC (two-phase commit, RFC XA)**: The write to co_retrievals and the epoch INCR to Redis are treated as a distributed transaction. Phase 1: prepare both operations. Phase 2: commit both. If either fails after phase 1, the prepare log allows recovery. This eliminates the RENAME window problem entirely — the epoch INCR and the co_retrievals write either both commit or both abort.

## Step 2: Mechanism Selection

The mechanism with the most implementation potential for Recall's specific problem is the **Outbox pattern**, for three reasons:

1. It has been validated in production at scale (used by Debezium CDC, Eventuate Tram, and described as a first-class pattern in Richardson's Microservices Patterns, 2018).
2. It is specific enough to implement: write to a local SQLite outbox before the PostgreSQL write, background worker flushes to PostgreSQL, deletes from outbox on success.
3. It directly addresses the Q112 window: if the epoch INCR fails during the RENAME window, the co_retrievals write has already been durably recorded in the outbox and will be flushed when the API recovers.

The Kafka idempotent producer mechanism is the second-highest-value mechanism: it provides the deduplication guarantees that make the outbox retry safe, without requiring a full SQLite database.

2PC is eliminated as overkill — it requires XA-compatible drivers for both Redis and PostgreSQL, adds 2-4 RTTs to every write, and the problem it solves (the 1-2 second RENAME window) is a one-time migration artifact, not an ongoing steady-state concern.

## Step 3: Precise Mechanism Description

### Mechanism: Outbox Pattern for co_retrievals

**Source**: Chris Richardson, Microservices Patterns (Manning, 2018), Chapter 4. Also described in the Debezium documentation as "Transactional Outbox Pattern." Widely deployed in production microservices (used by companies including Netflix, Uber, and Square for event publishing reliability).

**Input/Output**:
- Input: a hook fires with a set of memory IDs that were co-retrieved in this session (`from_id`, `to_id`, `tenant_id`, `epoch`).
- Output: a durable UPSERT to `co_retrievals` that increments `co_retrieval_count` and updates `last_reinforced_epoch`, even if the PostgreSQL write fails transiently.

**State maintained**:
- A local SQLite database at a fixed path (`~/.recall/outbox.db` on the hook machine, or `/var/recall/outbox.db` inside the FastAPI container) with a single table:
  ```sql
  CREATE TABLE outbox (
      id          INTEGER PRIMARY KEY AUTOINCREMENT,
      from_id     TEXT NOT NULL,
      to_id       TEXT NOT NULL,
      tenant_id   TEXT NOT NULL DEFAULT 'tim',
      epoch       INTEGER NOT NULL,
      created_at  REAL NOT NULL DEFAULT (unixepoch('now')),
      attempt_count INTEGER NOT NULL DEFAULT 0
  );
  ```

**Update rule**:
1. Hook fires. API receives co-retrieval pairs.
2. API writes all pairs to the outbox table (SQLite AUTOINCREMENT assigns an ID). This write is local and fast (SQLite, no network).
3. API immediately attempts the PostgreSQL UPSERT batch for all pairs in this request.
4. If PostgreSQL succeeds: delete the outbox entries by ID. Done.
5. If PostgreSQL fails (connection error, timeout, epoch INCR failure causing the API to abort): outbox entries remain. API returns 503 to the hook. Hook logs the error.
6. A background task (FastAPI `asyncio` task, runs every 5 seconds) reads all outbox entries with `attempt_count < 5` and retries the PostgreSQL UPSERT for each. On success, deletes the entry. On failure, increments `attempt_count`. After 5 failed attempts, moves to a `dead_letter` table for manual review.

**Idempotency**: The PostgreSQL UPSERT is `ON CONFLICT (tenant_id, from_memory_id, to_memory_id) DO UPDATE SET co_retrieval_count = co_retrievals.co_retrieval_count + 1`. This means if the same outbox entry is flushed twice (due to a crash between the PostgreSQL commit and the SQLite DELETE), the count is incremented twice. This is the standard "at-least-once" outcome of the outbox pattern without deduplication. To achieve exactly-once, the outbox entry ID is included in the PostgreSQL write: add an `outbox_id` column to `co_retrievals` (or a separate `processed_outbox_ids` table) and add `ON CONFLICT (outbox_id) DO NOTHING` before the UPSERT. This converts double-increment to idempotent.

**Failure modes**:
- SQLite outbox write fails: This is the only unrecoverable failure. If the hook machine's disk is full or the SQLite file is corrupt, the co-retrieval signal is lost. Mitigation: SQLite writes are extremely reliable; the outbox database is small (< 1000 rows at any time for normal operation).
- Outbox grows unboundedly: If PostgreSQL is down for hours, the outbox accumulates. At Recall's write rate (1-5 co-retrieval pairs per session, 3 sessions/day), the outbox would accumulate ~15 entries/day. Even after 30 days of PostgreSQL unavailability (extreme scenario), the outbox holds ~450 rows. SQLite handles this trivially.
- Clock skew between `created_at` timestamps: Not a correctness issue for co_retrievals, which uses `epoch` (Redis counter) not wall-clock time for ordering.

## Step 4: Mapping to Recall's Memory System

### What the mechanism does in the source field

In microservices, the outbox pattern solves the "dual-write problem": a service must write to its own database AND publish an event to a message broker. These two writes cannot be made atomic without a distributed transaction. The outbox makes the first write (local database) the durable record, and converts the second write (message broker) into an async delivery from the durable record.

### What the equivalent operation is in Recall

In Recall, the "dual-write problem" is:
- Write 1: `INCR recall:tim:global_epoch` in Redis (the epoch counter that timestamps the co-retrieval)
- Write 2: UPSERT to `co_retrievals` in PostgreSQL (the durable co-occurrence signal)

These two writes must be coordinated. If the Redis INCR fails (RENAME window, Q112), the epoch value is unknown, and the PostgreSQL UPSERT cannot be completed with the correct `last_reinforced_epoch`. If the PostgreSQL UPSERT fails (connection issue, lock timeout), the epoch was already incremented and the signal is lost.

The outbox converts this: the hook writes the co-retrieval pair with its best-effort epoch value to the outbox first. The outbox entry is durable. The actual PostgreSQL UPSERT is a background operation that can be retried with the preserved epoch value. The Redis RENAME window no longer causes data loss — it causes a brief delay in flushing the outbox, nothing more.

### What would need to be different

**Data types**: The outbox stores `from_id` and `to_id` as UUID strings, `epoch` as an integer, `tenant_id` as a string. This maps directly to the `co_retrievals` schema from Q115.

**Scale**: Recall's write rate is extremely low (3-15 co-retrieval pairs per day). The outbox never grows large. The background flush worker can run every 5 seconds without any performance concern.

**Location of the outbox**: The outbox could live in three places:
- Inside the hook (local file or SQLite on the hook machine): Durable against API failures but requires the hook to manage its own persistence. Adds complexity to the JavaScript hooks.
- Inside the FastAPI container (SQLite file mounted at a persistent volume): Durable against PostgreSQL failures but not against API container restarts without a mounted volume. Simpler — no hook changes needed.
- As a PostgreSQL table in the same database: The outbox write and the co_retrievals write can be made atomic in a single PostgreSQL transaction. This is the "transactional outbox" variant — write to the outbox table and the co_retrievals table in the same transaction. If the transaction commits, the outbox entry is already in PostgreSQL and the background worker is just cleanup. If the transaction fails, neither write happened. This is the strongest durability guarantee and requires zero new infrastructure.

The **transactional outbox in PostgreSQL** is the highest-value implementation for Recall. It converts the co_retrievals write from a single UPSERT to a two-statement transaction:

```sql
BEGIN;

-- Step 1: Record to outbox (durable, even if step 2 fails)
INSERT INTO co_retrievals_outbox
    (from_memory_id, to_memory_id, tenant_id, epoch)
VALUES (:from_id, :to_id, :tenant_id, :epoch);

-- Step 2: Attempt the UPSERT immediately
INSERT INTO co_retrievals
    (from_memory_id, to_memory_id, tenant_id, relationship_type,
     last_reinforced_epoch, weight, co_retrieval_count)
VALUES (:from_id, :to_id, :tenant_id, 'FRESH', :epoch, 1.0, 1)
ON CONFLICT (tenant_id, from_memory_id, to_memory_id) DO UPDATE
    SET relationship_type      = 'FRESH',
        last_reinforced_epoch  = EXCLUDED.last_reinforced_epoch,
        weight                 = co_retrievals.weight + 0.1,
        co_retrieval_count     = co_retrievals.co_retrieval_count + 1;

-- Step 3: Mark outbox entry as processed
UPDATE co_retrievals_outbox
SET processed_at = NOW()
WHERE id = currval('co_retrievals_outbox_id_seq');

COMMIT;
```

If this transaction fails, neither the outbox entry nor the co_retrievals row is written. There is nothing to retry because nothing was durably committed. The failure is clean, not silent.

But this does NOT solve the Q112 RENAME window problem — that window affects the Redis INCR call that happens BEFORE the PostgreSQL transaction. The epoch value is unknown during the window.

**The correct framing of what the outbox solves vs. what it does not solve**:

The Q112 RENAME window creates two distinct failure scenarios:
1. The Redis INCR fails (`no such key` error). The epoch is NOT incremented. The API receives an error before it can write to PostgreSQL. The co_retrievals UPSERT is never attempted.
2. The Redis INCR succeeds (the window has closed). The epoch IS incremented. The PostgreSQL UPSERT is attempted but fails due to a transient PostgreSQL issue. The epoch was consumed but the co_retrievals row was not updated.

Scenario 1 is NOT solved by the outbox — the outbox lives after the Redis INCR in the write path, so if the INCR never fires, there is no outbox entry. The fix for Scenario 1 is in Q112's existing recommendation: the RENAME window is 1-2 seconds during a one-time migration, and the hook logs a hard error. No write was made; nothing to recover.

Scenario 2 IS solved by the outbox (or more precisely, by the transactional outbox that commits the outbox entry and the UPSERT atomically): if the UPSERT fails after the epoch INCR succeeds, the outbox entry captures the intent and the background worker retries the UPSERT with the stored epoch value.

However, Scenario 2 is not the Q112 problem. Q112 specifically describes Scenario 1.

**What actually eliminates the Q112 write-loss window**: The Q112 finding already provides the correct answer — the RENAME window is 1-2 seconds, during which any hook write that attempts a Redis INCR will fail with `no such key`. The existing fix is correct: the config-last ordering (deploy → RENAME → config update) makes the window as short as possible (1-2 seconds, bounded by Docker restart time), and the failure is a hard error (not a silent loss). The window is a one-time migration artifact. After the migration completes, there is no ongoing write-loss window.

The WAL/outbox pattern adds value not by eliminating the Q112 window (it does not) but by adding durability to the co_retrievals UPSERT for steady-state failures unrelated to the RENAME migration.

## Finding

The question asks whether WAL patterns can eliminate the Q112 write-loss window. The answer is **partially yes, but not in the way the question implies**.

**What WAL teaches about the Q112 problem**: The Q112 RENAME window (1-2 seconds where Redis INCR returns `no such key`) causes the hook to fail BEFORE the PostgreSQL UPSERT is attempted. The failure is at the Redis layer, not the PostgreSQL layer. WAL and outbox patterns protect the path from API reception to PostgreSQL commit — they operate downstream of the Redis INCR. The RENAME window failure is upstream. Therefore, WAL patterns do not eliminate the Q112 window. The Q112 finding's config-last ordering is the correct and complete fix.

**What WAL teaches about steady-state co_retrievals durability**: The outbox pattern is highly applicable to the ongoing problem of making co_retrievals reinforcement writes durable against transient PostgreSQL failures. The transactional outbox variant — write to an outbox table and the co_retrievals table in the same PostgreSQL transaction — provides these guarantees: (1) the reinforcement intent survives API crashes, (2) retries are safe because the `ON CONFLICT` clause is idempotent, (3) no new infrastructure is required (the outbox lives in the same PostgreSQL database). This is a genuine improvement to Recall's co_retrievals durability that addresses the broader class of write failures, not just the RENAME migration window.

**The most valuable WAL-derived insight**: SQLite WAL mode is the wrong tool here (there is no need for a separate WAL file when PostgreSQL has its own WAL). PostgreSQL 2PC is overkill (it adds distributed transaction complexity for a one-time migration artifact). The Kafka idempotent producer mechanism provides the deduplication guarantee that makes outbox retry safe (prevents double-counting when the same outbox entry is flushed twice). The outbox pattern in PostgreSQL is the right tool.

**Revised understanding of the Q112 window**: Q112 correctly identifies the 1-2 second window as bounded and recoverable. The hooks log a hard error; no data is silently lost. The outbox pattern adds durability for a different failure mode: PostgreSQL transient failures during steady-state operation. These are complementary, not alternative, fixes.

## Qualifying Ideas
[N≥0.65, E≥0.40]

- **outbox-for-co-retrievals-steady-state**: (N=0.68, E=0.72, F=0.88) — A `co_retrievals_outbox` table in PostgreSQL, written atomically in the same transaction as the co_retrievals UPSERT, ensures that any failed UPSERT leaves a durable retry record. A background asyncio task in FastAPI flushes unprocessed outbox entries every 5 seconds. At Recall's write rate (3-15 pairs/day), the outbox runs nearly idle in steady state and adds a single transaction overhead per hook invocation. This eliminates the entire class of "PostgreSQL transient failure causes silent co_retrieval signal loss" — a failure mode that is ongoing and not bounded to any migration window.

- **kafka-sequence-number-idempotency**: (N=0.70, E=0.68, F=0.82) — Adding an `outbox_id UUID` column to `co_retrievals` and including it in UPSERT deduplication logic (via `ON CONFLICT (tenant_id, from_memory_id, to_memory_id, outbox_id) DO NOTHING` before the weight-increment UPSERT) makes outbox replay exactly-once rather than at-least-once. Directly adapted from Kafka KIP-98 producer sequence number deduplication, where the broker uses (PID, sequence) to reject duplicate messages. The co_retrievals equivalent: (outbox_id) to reject duplicate flushes of the same reinforcement event. Prevents the double-counting failure mode where a network partition causes the outbox worker to flush the same entry twice before the DELETE acknowledgment arrives.

- **wal-q112-boundary-clarification**: (N=0.72, E=0.65, F=0.95) — The Q112 RENAME window failure occurs at the Redis INCR layer, which is upstream of the PostgreSQL write path. WAL and outbox patterns protect the PostgreSQL write path. This means the correct scope for WAL-pattern application to Recall's co_retrievals is steady-state PostgreSQL failures, NOT the one-time RENAME migration window. Q112's config-last ordering is already the complete fix for the RENAME window. This boundary clarification prevents over-engineering: do not add a Redis-level WAL (e.g., Redis Streams as a write buffer) for a 1-2 second window that occurs once per system lifetime.

## Recommendations

**Implement**: The transactional outbox pattern for co_retrievals reinforcement writes. Specifically:

1. Add `co_retrievals_outbox` table to PostgreSQL with columns: `id SERIAL PRIMARY KEY, from_memory_id UUID, to_memory_id UUID, tenant_id VARCHAR(32), epoch INTEGER, created_at TIMESTAMPTZ DEFAULT NOW(), processed_at TIMESTAMPTZ NULL`.

2. Wrap the co_retrievals UPSERT in a transaction that also inserts to `co_retrievals_outbox`. The outbox entry is written as the first statement in the transaction; if the whole transaction commits, both the outbox entry and the UPSERT are durable. If the UPSERT fails, the transaction rolls back and neither write is committed (clean failure, not a silent loss).

3. Add a FastAPI startup background task (`asyncio.create_task`) that polls `co_retrievals_outbox WHERE processed_at IS NULL` every 5 seconds and retries any unprocessed entries. Mark `processed_at = NOW()` on success. After 5 failed attempts, write to a `co_retrievals_dead_letter` table for manual review.

4. Add `outbox_id UUID` to `co_retrievals` and include it in deduplication logic to prevent double-counting on retry.

**Do not implement**: Redis Streams as a WAL buffer for the epoch INCR. The Q112 RENAME window is a one-time migration artifact. Adding Redis Streams complexity to protect a 1-2 second window that occurs once per system lifetime is engineering waste. Q112's config-last ordering is sufficient.

**Do not implement**: PostgreSQL 2PC. The Redis INCR and the co_retrievals UPSERT are not in the same failure domain in a way that requires distributed transaction coordination. The epoch counter is a monotonic clock, not a compensatable resource. A failed epoch INCR during the RENAME window simply means that co_retrieval was not recorded with an epoch — the outbox can store `epoch = NULL` and the UPSERT can handle NULL epoch gracefully (use `COALESCE(epoch, 0)` or skip the `last_reinforced_epoch` update).
