## Evidence

### Mechanism 1: Lexical Scope Chain (Environment Model)

**Origin**: Abelson & Sussman, "Structure and Interpretation of Computer Programs" (MIT Press, 1985/1996), Chapter 3 — The Environment Model of Evaluation. This is the canonical formal description. Implemented in every Scheme/Lisp interpreter, JavaScript engines (V8, SpiderMonkey), Python (CPython LEGB), and every statically-scoped language since Algol 60.

**Precise data structure:**

An **environment** is a pair: `(frame, enclosing_environment)`. A frame is a dictionary mapping names to values. The chain of enclosing environments from innermost to global forms the **scope chain**.

```
Environment E3 (session scope):
  frame: { "redis_status": "BROKEN in this session",
           "current_task": "debug Redis" }
  enclosing: -> E2

Environment E2 (project scope):
  frame: { "redis_status": "healthy",
           "project": "System-Recall",
           "stack": ["FastAPI", "Qdrant", "Redis"] }
  enclosing: -> E1

Environment E1 (global/user scope):
  frame: { "redis_host": "192.168.50.19",
           "redis_port": 6379,
           "user": "Tim" }
  enclosing: -> null (the global environment)
```

**Name resolution algorithm** (lookup procedure):

```
function lookup(name, environment):
    if name in environment.frame:
        return environment.frame[name]
    elif environment.enclosing is null:
        raise NameError(name)
    else:
        return lookup(name, environment.enclosing)
```

This is O(d) where d is the depth of the scope chain — typically 3-5 levels. Name resolution walks inward-to-outward: inner scope always wins.

**Key formal property**: lookup stops at the first frame that contains the name. The outer definition is **completely shadowed** — it is never consulted once the inner binding is found. This is not a merge or weighted average: inner binding completely hides outer binding.

**define vs. set! semantics (SICP Ch. 3)**:

- `define` creates a new binding in the **current** (innermost) frame. If the name exists in an outer frame, it is unaffected — a new local binding shadows it.
- `set!` modifies the binding in the **first frame where the name is found** (walking outward). If the name is only in an outer frame, `set!` modifies the outer frame.

This distinction is critical: `define` implements "override at this scope without touching outer state"; `set!` implements "mutate wherever this name lives." The distinction maps directly to "session override" vs. "update global truth."

### Mechanism 2: Symbol Shadowing — Precise Semantics

**Origin**: Same source as above, plus ECMAScript 2023 specification §9.1 (Environment Records) and Python language reference §4.2.2 (Execution model, naming and binding).

**Shadowing is not deletion.** In every correctly implemented scoped language:

1. The outer binding continues to exist in its frame.
2. It is accessible through explicit scope-override syntax:
   - Python: `global x` declaration allows access to module-level `x` from a nested scope.
   - JavaScript: `window.x` accesses the global `x` even when a local `x` shadows it.
   - C++: `::x` accesses global scope.
   - Python `nonlocal x`: accesses the enclosing (not global) scope.

3. The shadowed binding becomes visible again when the inner scope exits.

**What this means**: shadowing is **temporary and reversible**. When the inner scope is destroyed (the function returns, the block exits), the outer binding is exposed again automatically. The runtime cost of un-shadowing is zero — it is simply that the inner frame no longer exists on the scope chain.

**Shadowing without conflict**: two bindings with the same name in different scopes coexist without interference. There is no merging, no averaging, no conflict resolution heuristic. The chain determines which one is visible at any given resolution point.

### Mechanism 3: Dynamic Scope (Lisp, Bash, Perl special variables)

**Origin**: Original Lisp (McCarthy 1958-1960) used dynamic scope by accident — it was a bug in the original APPLY implementation that became a "feature." Common Lisp's `defvar`/`defparameter` use dynamic scope. Bash uses dynamic scope for all variables by default. Perl's `local` keyword.

**Precise difference from lexical scope:**

- **Lexical scope**: name resolution depends on where the code is *written* (the source text nesting). The scope chain is fixed at compile time.
- **Dynamic scope**: name resolution depends on the call stack at *runtime*. The scope chain is the chain of active call frames, not the syntactic nesting.

**Example (Bash — dynamic scope by default):**

```bash
x="global"
inner() { echo $x; }          # uses dynamic scope
outer() { local x="outer"; inner; }
outer  # prints "outer", NOT "global"
```

In lexical scope, `inner()` would always see `x="global"` because that's where `inner` is defined. In dynamic scope, `inner()` sees `x="outer"` because `outer()` is on the call stack when `inner()` runs.

**When dynamic scope is superior to lexical scope:**

Dynamic scope is better when the "context" for a computation is determined by the *calling context*, not the *definition context*. Classic use cases:
- Current logging verbosity (set by whoever called this function)
- Current database transaction context
- Current user context (set by the request handler, visible to all functions it calls)
- **Current session context** (set by the session initiator, visible to all retrieval calls within the session)

This is exactly Recall's problem: a session's override of "redis_status" should be visible to all retrieval calls *within that session*, regardless of which code path triggered them.

**The critical insight**: for Recall's use case, **dynamic scope is the correct model**, not lexical scope. Session context is established by the session start event and should propagate dynamically to all retrieval calls that happen during the session — even retrieval calls made through different code paths, different domains, or different tag filters. The session scope is the "caller" and it should override inner retrieval operations.

### Mechanism 4: Activation Records and Frame Allocation

**Origin**: Alfred V. Aho, Monica S. Lam, Ravi Sethi, Jeffrey D. Ullman, "Compilers: Principles, Techniques, and Tools" (Dragon Book, 2nd ed. 2006), Chapter 7. Every production compiler implements this.

**Precise data structure:**

An **activation record** (also: stack frame) contains:
- Local variable bindings (the "frame" in environment model terms)
- A pointer to the **static link** (lexical parent frame — for lexical scope)
- A pointer to the **dynamic link** (caller's frame — for dynamic scope)
- Return address, saved registers, etc.

For scope chain traversal, the static link chain is the scope chain. When a function needs a variable not in its local frame, it follows the static link to the parent frame, then the parent's static link, etc.

**Key optimization: display arrays**

Instead of following a chain of pointers (O(d) per lookup), compilers maintain a **display**: an array `D[0..max_depth]` where `D[i]` points to the frame at depth `i` in the current scope chain. Lookup at depth k is `D[k][variable_offset]` — O(1) regardless of chain depth. The display is updated on each call (O(d) update, but amortized over many lookups).

**Time complexity summary:**
- Naive scope chain walk: O(d) per lookup, O(1) update
- Display array: O(1) per lookup, O(d) per call
- For Recall (d ≤ 4 levels: session → machine → project → global), the difference is negligible

### Mechanism 5: Python LEGB Rule and Module Scoping

**Origin**: Python Language Reference, §4.2.2. Python defines four scope levels: **L**ocal, **E**nclosing (closure), **G**lobal (module), **B**uilt-in. Name resolution walks these in order. This is a specific four-level instantiation of the general scope chain.

**Module-level scope:**

Python's module system gives each module its own global frame. When you `import foo`, you get access to `foo`'s global frame via the module object. You can read `foo.x` (accessing `foo`'s global `x`), but `x = 5` in your module doesn't affect `foo.x` — you're binding in your own global frame.

**The `__all__` export list** controls which names from a module's frame are publicly visible when another module does `from foo import *`. This is a **namespace filtering** mechanism: not all bindings in a frame are exported.

**Mapping to Recall**: modules are analogous to domains. `from system-recall import *` is analogous to querying the `system-recall` domain. The `__all__` filtering is analogous to tag filtering within a domain. But Python's module scoping adds the key feature that modules can be **imported into** a scope — you can compose contexts explicitly.

### Track record in source field

**Lexical scope chains** are the foundation of every production programming language designed after 1960. Scheme (1975, Steele & Sussman), JavaScript (1995), Python (1991), Java (1995), Rust (2010) all use lexical scope chains as their primary name resolution mechanism. The mechanism is formally proven correct (Church-Rosser theorem applies to lambda calculus with lexical scope). CPython resolves ~billions of name lookups per second using LEGB.

**Symbol shadowing** is validated at extreme scale: JavaScript closures create new scopes billions of times per second in browser engines. V8 uses hidden classes to optimize scope chain traversal to near-zero overhead. The shadowing mechanism itself is O(1) — adding a name to a frame is hash table insertion.

**Dynamic scope** is used in production in: Bash (default for all variables), Common Lisp (`defvar`), Emacs Lisp (historically), R (lexical by default but `parent.frame()` enables dynamic lookup), and implicitly in continuation-passing style coroutines.

**The environment model** described in SICP has been implemented in thousands of Scheme interpreters, teaching languages, and embedded scripting engines. It is probably the best-documented data structure in computer science education.

### Why AI memory systems don't do this yet

Three reasons:

1. **The problem is framed as search, not name resolution.** AI memory retrieval is built on vector search — it returns the top-k semantically similar memories. This is a fundamentally different model from scope chain lookup, which returns *exactly one* binding per name. The scope chain model requires knowing which names (topics, facts) you are looking up, whereas vector search works on unstructured queries. The mental model gap is large.

2. **Domains are treated as labels, not frames.** Recall's domain field is a string label ("system-recall", "personal") — metadata for filtering, not a structural container that can nest, inherit, or shadow. The key structural innovation of scope chains — frames form a chain where inner frames can shadow outer frames — is entirely absent. There is no code in Recall (or any AI memory system I'm aware of) that says "look in session scope first, then project scope, then global scope."

3. **Session context is ephemeral and unstructured.** In current systems, the concept of "this session's context" is implicit — it exists in the LLM's context window but is not represented as a structured scope in the memory system. There is no API to say "for this session, define that redis_status = BROKEN, and have that shadow the global fact that redis_status = healthy." The session/machine/project/global hierarchy is conceptually present but structurally absent from the data model.

## Ideas Extracted

### Idea: scope-chain-memory-retrieval

- **Novelty**: 0.80 — No production AI memory system implements hierarchical scope-based retrieval where inner scopes shadow outer scopes. Mem0, Zep, MemGPT, LangChain Memory all use flat namespace retrieval with metadata filters. The specific mechanism of scope chain walk (innermost-first, stopping at first binding found) applied to memory retrieval is not present anywhere. MemGPT has a paging/compression model, not a scoping model. Zep has entity graphs but no shadowing semantics.
- **Evidence**: 0.95 — Lexical scope chains are the most rigorously validated mechanism in all of computer science. SICP Chapter 3 formalization is 40+ years old. Every production language runtime implements this. The correctness properties (deterministic name resolution, shadowing behavior) are formally proven.
- **Feasibility**: 0.70 — Recall's existing data model needs two changes: (1) domains become a hierarchy, not a flat string (e.g., `session:abc123 → machine:casaclaude → project:system-recall → global`); (2) retrieval walks the hierarchy innermost-to-outermost and can stop or shadow. Qdrant supports metadata filtering with nested JSON; the hierarchy can be encoded in the payload. The retrieval path in FastAPI needs to be refactored from "filter by domain string" to "walk scope chain until k results found or chain exhausted." PostgreSQL already stores memory metadata — adding a `scope_chain` array column is trivial. Implementation: 3-4 weeks.
- **Description**: Restructure Recall's domain model as a four-level scope chain: `[session_id → machine_id → project → global]`. When retrieving memories, walk the scope chain from the innermost level outward. A memory found at the session scope level shadows a contradicting memory at the global scope level — the inner binding wins. The retrieval API gains a `context` parameter that defines the current scope chain (e.g., `{"session": "abc123", "machine": "casaclaude", "project": "system-recall"}`). Tim's casaclaude sessions and proxyclaude sessions become separate machine-level scopes — a memory about "current Redis debug session" stored in `session:abc123 on casaclaude` never appears in proxyclaude session retrieval. When the session ends, that scope is removed from the chain and the project-level and global memories are exposed again automatically.

### Idea: define-vs-set-memory-mutation

- **Novelty**: 0.85 — The distinction between "create a local override that shadows the global" (`define`) and "update the global binding" (`set!`) is not present in any AI memory system's API. All current systems have a single "upsert" semantic that updates the most-similar existing memory or creates a new one. The scoped-write distinction does not exist.
- **Evidence**: 0.90 — The define/set! distinction is a core design choice in Scheme (Steele & Sussman 1975, R7RS standard). It is present in Python (local assignment creates new binding vs. `global`/`nonlocal` declarations to modify outer bindings), JavaScript (`let`/`const` create local bindings vs. direct assignment to undeclared variables falls through to outer scope). The distinction prevents a class of bugs ("action at a distance" where a local computation accidentally mutates global state) that are precisely analogous to memory contamination between sessions.
- **Feasibility**: 0.65 — Requires API changes to Recall's store endpoint: a `scope` parameter and a `shadow_only` flag. `shadow_only=True` stores the memory only in the current innermost scope (session/machine level) and marks it as a local override. `shadow_only=False` (default) propagates to the appropriate persistent scope (project or global) based on domain. The retrieval change from `scope-chain-memory-retrieval` is a prerequisite. Additional complexity: deciding which scope a `shadow_only=False` store should go to (inference problem). Implementation: 2 weeks on top of scope-chain-memory-retrieval.
- **Description**: Recall's `recall_store` API gains a `shadow` parameter. `recall_store(content="redis is BROKEN", domain="session", shadow=True)` stores an override that is visible only in the current session scope and expires when the session ends — it never touches the project-level or global memories about Redis health. `recall_store(content="redis is healthy on this host", domain="infrastructure", shadow=False)` stores a persistent global fact. The session-scope shadow is automatically un-shadowed when the session ends (session scope removed from chain), making the project-level Redis-healthy memory visible again. This exactly mirrors Python's `global` keyword: you opt into global mutation explicitly; the default is local.

### Idea: dynamic-scope-session-context

- **Novelty**: 0.75 — Dynamic scope for AI memory context is not present in production systems. However, the general idea of "session-level context injection" is more commonly discussed in the AI community than the specific mechanism of dynamic scope chains. The novelty is in the precise mechanism (dynamic scope walk at retrieval time, not static filtering) rather than the general concept.
- **Evidence**: 0.80 — Dynamic scope is a validated mechanism in Bash (all variables), Common Lisp (`defvar`/`defparameter`), and is used in production threading models (Python `threading.local()`, Java `ThreadLocal<T>`). The `ThreadLocal` pattern is a direct dynamic-scope analogue in concurrent systems: each thread (analogous to session) has its own binding for a name that shadows the shared binding. Java's `ThreadLocal` is in production in essentially every Java web application framework (request-scoped context, transaction context, user context propagation). This is the exact Recall use case — session-scoped memory context propagated through all retrieval calls in the session.
- **Feasibility**: 0.75 — Dynamic scope for Recall means: when a retrieval call is made, the current scope chain is determined by the *session context* (who is calling, from which machine, in which project), not by the *query content*. Recall already has a session concept (the `session_id` parameter). Promoting session_id from a metadata filter to a scope-chain pointer is the key implementation step. The FastAPI server would maintain a session registry (in Redis: `session:{id} → scope_chain_json`) that each retrieval call consults to determine which scope levels to walk. Session registration happens at session-start hook; session cleanup happens at session-end hook. Both hooks already exist in Recall. Implementation: 2-3 weeks.
- **Description**: When Claude Code starts a session on casaclaude, the `recall-session-start` hook registers the session with a scope chain: `["session:abc123", "machine:casaclaude", "project:system-recall", "global"]`. Every `recall_retrieve` call within that session automatically walks this chain without the caller specifying scope parameters — the session context propagates dynamically, exactly as `defvar` in Common Lisp propagates through all functions called within a dynamic binding form. A proxyclaude session registers `["session:xyz789", "machine:proxyclaude", "project:system-recall", "global"]`. The machine-level scope isolation prevents cross-contamination. Within each session, memories stored with `shadow=True` are visible only to that session's scope chain. This requires zero changes to the calling code (hook scripts) — the scoping is transparent.

### Idea: namespace-export-filtering

- **Novelty**: 0.70 — The concept of a domain controlling which of its memories are "exported" to outer scopes (analogous to Python's `__all__`) is not present in AI memory systems. However, tag-based filtering exists everywhere, reducing novelty. The novel part is the directionality: currently any retrieval can see any domain's memories given the right filter. The export-list model inverts this — the domain controls what escapes upward.
- **Evidence**: 0.75 — Python's `__all__`, Java's `public`/`private`/`package-private` access modifiers, Rust's `pub`/`pub(crate)`/`pub(super)` visibility rules. These are widely deployed mechanisms. The specific concept of a scope defining what names propagate upward is a solved problem in every module system.
- **Feasibility**: 0.60 — Adding an `export_to_outer_scope` boolean flag to memories is straightforward in PostgreSQL. The scope chain retrieval logic would respect this flag: when walking from session scope to project scope, only memories marked `export_to_outer_scope=True` at the session level are visible at the project level. This enables memories like "current debugging state" to be session-scoped (never exported) while "discovered bug in module X" can be marked for export to the project scope where it persists after the session ends. Implementation: 1-2 weeks on top of scope-chain-memory-retrieval.
- **Description**: Each memory gains an `export` field (default: `False` for session-scope memories, `True` for project/global scope memories). When a session-scope memory is created with `export=True`, it is visible to outer scopes during the session AND promoted to the appropriate outer scope when the session ends — this is the mechanism for session-discovered facts to propagate into the long-term project memory. Session-scope memories with `export=False` are local scratchpad and expire with the session. This enables the pattern: "I discovered during this debug session that Redis requires `maxmemory-policy allkeys-lru`" — `export=True`, which after the session ends becomes a project-level memory. "Current debugging focus is auth token expiry" — `export=False`, session-local, never pollutes global memory.

### Idea: display-array-scope-indexing

- **Novelty**: 0.90 — The display array optimization (precomputed index into each scope level, enabling O(1) scope access) has no analogue in any AI memory system. This is a pure performance optimization that maps to Recall's multi-scope retrieval latency problem.
- **Evidence**: 0.80 — Display arrays are described in the Dragon Book (Aho et al., 2006) and implemented in production compilers including GCC and LLVM for languages with deep lexical nesting (Ada, Pascal). The O(1) lookup property is formally proven.
- **Feasibility**: 0.55 — For Recall's 4-level scope chain, a display array would be a precomputed set of Qdrant collection pointers or filter parameters for each scope level. On session start, compute and cache the scope-level query parameters (collection IDs, filter payloads) rather than recomputing them on each retrieval call. This is a micro-optimization compared to the structural changes required by scope-chain-memory-retrieval. The feasibility reduction is because Recall's retrieval latency is dominated by network RTT and embedding generation — scope chain walk overhead at 4 levels is negligible. This idea is INCREMENTAL until the other scope-chain ideas are implemented.
- **Description**: Cache the scope chain as a precomputed list of Qdrant filter objects on session start. Each scope level becomes a pre-built Qdrant `Filter(must=[FieldCondition(key="scope_level", match=MatchValue(value="session:abc123"))])`. The session registry stores these precomputed filters, not the raw scope chain strings. On each retrieval call, iterate through the precomputed filter list rather than building filters from the scope chain on each call. Reduces per-query overhead from O(d * filter_build_time) to O(d * filter_lookup_time). At Recall's current scale (d=4, retrieval dominated by Ollama embedding latency), this saves ~1ms per retrieval. At scale with thousands of sessions, it matters more.
