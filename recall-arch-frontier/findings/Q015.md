# Finding: Q015 — Taboo Architecture: Automatic Importance Scoring from First Principles

**Question**: Ignoring all existing tools, design an automatic importance scoring system that updates scores based on how often a memory is accessed, how recently it was accessed, and whether it was acted upon. What data structure would you use?
**Agent**: quantitative-analyst
**Question Type**: TABOO
**Verdict**: BREAKTHROUGH
**Severity**: High

---

## First Principles Derivation

### 1. What Is "Important"? (Formal Definition)

Start from decision theory, not intuition. A memory M is important to the degree that its presence improves a decision maker's task performance relative to its absence.

Formally: let P(task | M) be the probability of successful task completion when memory M is available, and P(task | ¬M) be the probability without it. The **decision-theoretic importance** of M is:

```
I(M) = P(task | M) − P(task | ¬M)
```

This is a counterfactual difference — importance is not a property of the memory itself but a property of the memory's relationship to tasks. Two consequences follow immediately:

**Consequence 1 — Importance is empirical, not inferrable at write time.** At write time, I(M) is entirely unknown. The probability P(task | M) can only be estimated from observed retrievals that led to actions. Any importance score assigned at write time is a prior with no evidence behind it. The current Recall system assigns this prior and never updates it — it treats the prior as the posterior, which violates basic evidential reasoning.

**Consequence 2 — Importance is relative to a query distribution.** A memory about Docker networking is unimportant if no Docker-related queries ever arrive. The same memory becomes critical if 40% of queries involve Docker. Therefore importance must be estimated jointly with the query distribution, not in isolation.

A computationally tractable proxy for I(M):

Given that we cannot run counterfactual experiments, we can bound I(M) from below using observed signals:

```
Ĩ(M) = α · freq(M) · decay(Δt) + β · action_count(M)
```

where:
- `freq(M)` is the number of times M has been retrieved
- `decay(Δt)` is a weighting function applied to each access based on its age Δt
- `action_count(M)` is the number of times M was retrieved and subsequently acted upon
- α, β are weights (derived below)

This proxy has the correct monotonicity: more accesses → higher importance, acted-upon memories → higher importance. It can be computed without LLM calls.

### 2. The Correct Update Rule (derived from properties)

We need an update rule for Ĩ(M) that has the following properties, derived from the definition:

**Property A — Recency weighting**: A retrieval 5 minutes ago is stronger evidence of current relevance than a retrieval 6 months ago. The update rule must weight recent accesses more than distant ones.

**Property B — Monotone in frequency**: More accesses means more evidence of importance. The score must be non-decreasing in access count (holding recency constant).

**Property C — Bounded above**: Scores must remain in a finite range. An uncapped accumulator grows without bound, making threshold comparisons unstable.

**Property D — Forgetting without evidence**: If a memory is never accessed, its importance should eventually approach a baseline. A score that can only go up violates this.

**Property E — Computability from a fixed-size state**: We cannot store an unbounded access log for millions of memories. The update rule must maintain O(1) state per memory, not O(accesses) state.

What function has all five properties? Consider the following update applied on each access event at time t:

```
score_new = score_old · λ^(Δt / τ) + δ
```

where:
- `λ` is a decay factor in (0, 1)
- `Δt` is the elapsed time since the last update
- `τ` is a time constant (e.g., 7 days in seconds)
- `δ` is an increment per access (e.g., 0.1)

This rule has all five properties by construction:

- **A**: The `λ^(Δt/τ)` term scales old score down before adding the new event, weighting the new event relatively higher the longer since last access. Equivalently, each access event's contribution decays exponentially with age.
- **B**: Adding δ > 0 on each access is strictly increasing.
- **C**: The fixed point of `s = s·λ + δ` is `s* = δ/(1−λ)`, providing a natural upper bound.
- **D**: Without any access, `score(t) = score(t₀) · λ^((t−t₀)/τ)`, which decays toward zero.
- **E**: Only three numbers per memory needed: `score`, `last_update_time`, and `access_count`. O(1) state.

The update can be applied lazily: when a memory is read, multiply the stored score by `λ^(Δt/τ)` before adding δ and writing back. No background sweep is needed.

The action signal gets its own increment `δ_action > δ_retrieve` (derive values from the ratio of how much more valuable an acted-upon memory is; empirically, a reasonable starting point is δ_action = 3 × δ_retrieve).

This derivation produces exactly the recency-weighted, frequency-driven, bounded, self-maintaining update rule that the problem requires — without naming any algorithm family.

### 3. The Correct Decay Function (derived from first principles)

When a memory is not accessed, importance should decay. What shape should the decay take?

Three candidate shapes: linear, power-law, and geometric (multiplicative-per-unit-time).

**Linear decay** (`I(t) = I₀ − k·t`): This implies that importance decreases by a constant absolute amount per unit time regardless of the current score. A memory with score 0.9 and a memory with score 0.1 both lose k per day. This is wrong for importance: a highly-accessed memory should be more resistant to forgetting than a rarely-accessed one (it has more evidence behind it). Linear decay treats all memories identically regardless of their history.

**Power-law decay** (`I(t) = I₀ · t^(−α)`): Derived from the spacing effect in human memory research (Ebbinghaus, 1885). Forgetting in biological memory is approximately power-law in retention probability. This would be appropriate if we were modeling human-like forgetting. But we are modeling **relevance to a task**, not biological forgetting. A technical memory about a Docker misconfiguration is either still relevant (the infrastructure hasn't changed) or suddenly irrelevant (the infrastructure changed), not gradually less relevant in a power-law curve.

**Geometric decay per unit time** (`I(t) = I₀ · λ^(t/τ)`): This is the correct shape for the following physical reason.

Consider the question: "Given that this memory was relevant k days ago, what is the probability it is still relevant today?" If the environment is stationary (facts don't change), the answer is 1.0 regardless of k. If the environment changes stochastically, the correct model is a two-state Markov chain: at each time step, a relevant fact becomes irrelevant with probability p and stays relevant with probability 1−p. The probability of remaining relevant after k time steps is:

```
P(still relevant | was relevant k steps ago) = (1 − p)^k
```

This is exactly geometric decay with λ = (1−p). The **correct decay shape is geometric because the underlying generative model of relevance-lifetime is memoryless** (Markov property). The system has no information about whether a fact became stale yesterday or will become stale tomorrow — each time period has equal marginal probability of rendering the fact stale.

Therefore τ (the time constant, where λ = e^(−1/τ) for continuous-time) should be set to the expected half-life of facts in the domain. For Recall, a reasonable starting estimate is τ = 14 days: technical facts about a project stay relevant for roughly two weeks before the project evolves or the context shifts.

### 4. The Correct Data Structure (derived from access patterns)

The operations required are:

1. **O(1) lookup by memory ID**: "Given memory ID M, return its current importance score." This is needed in the retrieve path where we multiply similarity by importance.

2. **O(1) update by memory ID**: "Given memory ID M at time t, apply the recency-weighted increment." This is needed after every retrieve event.

3. **O(log N) ordered retrieval**: "Return the top-k memories by importance score." This is needed for the threshold filter ("only surface memories above importance 0.5").

4. **Range deletion**: "Delete all memories below importance θ after time T." This is needed for the eventual forgetting/eviction policy.

**What data structure has these properties?**

Properties 1 and 2 require O(1) keyed access — a hash table (dictionary). Properties 3 and 4 require ordered traversal — a sorted structure (balanced binary search tree, or a skip list, or a sorted array that is rebuilt periodically).

The combination is a **two-index data structure**: a hash table mapping memory_id → (score, last_update_time, access_count), plus a separately-maintained ordered index on score for range queries.

The hash table gives O(1) for operations 1 and 2. The ordered index gives O(log N) for operations 3 and 4. Updates to the ordered index require O(log N) when a score changes — but score changes only happen on access events, which are infrequent relative to total memory count. The ordered index can be an in-memory B-tree (each node holds a score value and a set of memory IDs with that score) or a sorted skip list.

**The catch: lazy evaluation.** Under geometric decay, every memory's effective score is different at every point in time, even without an access event. Rebuilding the full ordered index continuously is O(N) — unacceptable. The solution is **lazy score materialization**:

- Store in the hash table: `(raw_score_at_last_update, last_update_time)`.
- The ordered index is built on `raw_score_at_last_update`, NOT on the time-decayed score.
- When querying the top-k by importance, retrieve the top-2k candidates by raw score (O(log N)), then materialize the actual current score for each by applying `score_effective = raw_score · λ^((now − last_update) / τ)`, sort the k materialized values, and return top-k.

This gives O(log N + k log k) for ordered retrieval and O(1) amortized for updates, with the ordered index only rebuilt when scores are explicitly updated (on access events), not on every time tick.

**Concretely for Recall**: this is a thin table in the relational store (PostgreSQL) with columns `(memory_id, raw_score, last_update_epoch, access_count, action_count)` plus a B-tree index on `raw_score`. The O(1) lookup is a primary-key lookup; the O(log N) range query is an index scan. No new database technology is needed — this uses the existing PostgreSQL instance already running for metadata.

### 5. Action-Signal Detection Without LLM Calls

The action signal (whether a retrieved memory was "acted upon") is the most valuable signal but the hardest to detect. The naive approach would be to ask the LLM "did you use memory M?" — but this adds latency, cost, and a round-trip.

A lightweight algorithmic approach can be derived from what "acted upon" physically means:

**Operational definition**: A memory M retrieved in query Q was acted upon if and only if the response to Q references information that appears only in M (and not in the conversation history or other retrieved memories). The response is the evidence; we need to find M's fingerprint in it.

**Algorithm: Term Overlap Fingerprinting**

Step 1 — At retrieval time, extract the **distinctive n-gram set** of memory M: terms that appear in M but not in the other (K−1) retrieved memories for this query. Call this `fingerprint(M)`. Time: O(|M| × K) per retrieve, done once per query.

Step 2 — At response time (PostToolUse hook fires when observe-edit.js detects a file edit, or at session end when recall-session-summary.js fires), compute the **term overlap** between the LLM's response text and `fingerprint(M)` for each M that was retrieved in this session.

Step 3 — If `|response ∩ fingerprint(M)| / |fingerprint(M)| > θ_overlap`, classify as "acted upon". A threshold of θ_overlap = 0.3 means 30% of M's distinctive terms appear in the response.

**Why this works**: If the LLM incorporated M into its reasoning, M's distinctive vocabulary will appear in the response. If the LLM ignored M, its distinctive terms will not appear. This is a set-membership test on token bags, not a semantic similarity computation — O(|response| + |fingerprint(M)|) per memory, using only a hash table.

**Edge cases handled**:
- If M's distinctive n-gram set is empty (M shares all content with other retrieved memories), no action signal is attributed. Conservative but correct.
- The overlap threshold θ is tunable. Start at 0.3; log misses and adjust.

**Fingerprint storage**: The fingerprint is computed at retrieval time and needs to survive until the response is available (same session). The session-scoped state stored in Redis (which Recall already maintains) is the natural place. `session:{id}:fingerprints` → a JSON map of `{memory_id: [distinctive_terms]}`. TTL = session length. No schema changes to PostgreSQL needed for the fingerprint itself.

### 6. What Static Importance Misses

The invariant violated by a static importance score is the **evidence accumulation principle**: the posterior probability of a hypothesis must update when new evidence arrives.

Formally, a static score treats the initial importance estimate (set at write time by whoever stored the memory) as the posterior despite all subsequent retrieval evidence. Every retrieval event is a data point that should shift the posterior toward the empirical access frequency. Every session that passes without accessing a memory is evidence that the memory may be less relevant than initially estimated.

By never updating, the current system violates a property that every rational inference system must satisfy: **the score should converge to the true importance as evidence accumulates**.

The concrete failure mode: a memory stored with importance 0.9 (because the author thought it was critical) that has never been retrieved in 50 subsequent sessions will continue to rank at the top of results, crowding out more recently-useful memories with lower initial scores. The system confuses **claimed importance** (at write time) with **demonstrated importance** (through use). These are fundamentally different quantities, and only one of them improves retrieval quality.

A second violated invariant: **temporal relevance coherence**. If a memory was important two years ago, static scoring implies it is equally important today. This is false in any dynamic environment. Facts about infrastructure configurations, project states, and team preferences all have finite relevance lifetimes. A static score assigns infinite relevance lifetime to every memory.

## Ideas Extracted (using real Recall stack components)

### Idea: lazy-decayed-importance-in-postgres

- **Novelty**: 0.65 — Combining lazy score materialization (computing effective score only at query time from raw stored values) with geometric decay and PostgreSQL B-tree index ordering is not a widely described pattern. Most production systems either (a) run a background job to update all scores periodically (eager), or (b) store no decay at all. Lazy materialization eliminates the background sweep entirely.
- **Evidence**: 0.75 — Geometric decay (the mathematical basis) is well-established in caching theory (e.g., LRU-K algorithms, LIRS, ARC). The lazy evaluation pattern is standard in functional programming and database query optimization. Their combination for importance scoring is engineering best practice applied to a novel surface.
- **Feasibility**: 0.90 — 1-2 week implementation. Add columns `raw_score FLOAT`, `last_update_epoch BIGINT`, `access_count INT`, `action_count INT` to the memories table in PostgreSQL. On every retrieve event that returns memory M, run `UPDATE memories SET raw_score = raw_score * 0.99^((now - last_update_epoch) / 86400) + 0.1, last_update_epoch = now, access_count = access_count + 1 WHERE id = M`. The retrieve ranking query uses `raw_score * 0.99^((now - last_update_epoch) / 86400) AS effective_score` in the ORDER BY. Existing B-tree index on `raw_score` supports range filtering.
- **Description**: Extend the PostgreSQL memories table with four columns: `raw_score`, `last_update_epoch`, `access_count`, `action_count`. Each retrieve event fires a single `UPDATE` on the returned memory IDs. The retrieve ranking query materializes effective scores via an arithmetic expression in the `SELECT`. No background workers, no new services, no schema migration beyond adding four columns. The recency-weighted score naturally pushes frequently-accessed memories to the top and lets stale high-initial-score memories decay into the background. Estimated implementation time: 3-5 days including migration and testing.

### Idea: redis-session-fingerprint-action-signal

- **Novelty**: 0.80 — Using term-overlap fingerprinting of retrieved memories against LLM response text to detect "acted upon" without an LLM call or semantic embedding is not a described pattern in any known AI memory system. The closest analog is document fingerprinting for plagiarism detection (Rabin-Karp rolling hash, shingling), applied here to cross-reference retrieval context with response text.
- **Evidence**: 0.55 — Term overlap as a measure of text relatedness is established (BM25 term frequency, Jaccard coefficient on bag-of-words). Application to the specific problem of detecting LLM citation of retrieved context is not peer-reviewed but is logically sound. Requires empirical validation to calibrate the overlap threshold.
- **Feasibility**: 0.65 — 2-4 week implementation. The session-scoped fingerprint map in Redis is straightforward (existing Redis instance, small payloads). The PostToolUse hook (observe-edit.js) already fires when the LLM writes to files — the response text captured at this hook point can be diffed against the fingerprint set. The challenge is that the LLM's response in Claude Code is not always cleanly captured as a text blob; it may be distributed across tool calls. Requires testing the hook capture to confirm response text is available.
- **Description**: At retrieve time, compute `fingerprint(M)` for each returned memory M as the set of tokens appearing in M but not in other co-retrieved memories. Store `session:{id}:fingerprints → {memory_id: [tokens]}` in Redis with TTL matching session duration. When observe-edit.js fires (LLM is editing a file, indicating an action was taken), extract the diff text and compute overlap against each fingerprint. For each memory M where overlap exceeds threshold, fire `POST /api/memories/{id}/action-signal` to increment `action_count` and apply a boosted importance update (`δ_action = 0.3` vs `δ_retrieve = 0.1`). This closes the feedback loop between retrieval and use.

### Idea: co-retrieval-graph-edge-importance

- **Novelty**: 0.70 — Memories that are consistently co-retrieved (appear together in the top-k results for multiple queries) should have their importance scores correlated: if one is important, the others likely are too. This creates a social-network-like "importance propagation" where high-access memories lift the scores of their co-retrieval neighbors. This is not described in any known AI memory system but is directly analogous to PageRank (a web page is important if important pages link to it) and to citation graph analysis.
- **Evidence**: 0.60 — Graph-based importance propagation (PageRank, eigenvector centrality, HITS) is well-validated in information retrieval. The direct application to a memory system using the graph layer (Neo4j) already deployed in Recall has not been validated but is logically sound. The co-retrieval edge is weaker than a hyperlink — it's probabilistic and query-distribution-dependent — so the propagation coefficient must be small to avoid runaway amplification.
- **Feasibility**: 0.45 — 1-3 months. Requires: (1) instrumenting the retrieve endpoint to log which memory IDs were returned together for each query; (2) storing co-retrieval edges in Neo4j (which is already deployed but underutilized); (3) implementing a periodic importance propagation step that updates raw scores based on co-retrieval graph centrality. The Neo4j graph layer is already in the stack but currently unused in the retrieval path. This gives it a concrete function.
- **Description**: When the retrieve endpoint returns K=5 memories for a query, write directed co-retrieval edges in Neo4j: `(M_i)-[:CO_RETRIEVED {weight: 1}]->(M_j)` for all pairs. Periodically (e.g., once per hour), run a graph traversal to compute degree-weighted importance: each memory M receives a co-retrieval bonus proportional to the sum of importance scores of its neighbors. `importance_bonus(M) = γ · Σ importance(M') for all M' co-retrieved with M`. Add this bonus (small γ = 0.05) to raw_score in PostgreSQL. This turns the underutilized Neo4j instance into an importance amplifier: memories that cluster with frequently-accessed, high-importance memories receive a gentle lift.

### Idea: write-time-prior-calibration

- **Novelty**: 0.55 — The write-time importance score is currently set by the caller (the Recall hooks). If we know the base rate of importance for memories in each domain (e.g., "homelab config memories have a 40% chance of being retrieved in the next month, project-summary memories have 80%"), the initial prior should reflect that base rate rather than a uniform 0.5 or whatever the caller passes. This is a prior calibration step — using historical retrieval data to set better initial scores for new memories.
- **Evidence**: 0.65 — Prior calibration from empirical base rates is standard Bayesian practice. In information retrieval, learning-to-rank (LTR) systems estimate prior relevance from feature distributions. The specific application to Recall's domain-based prior is straightforward given that domain metadata is already stored.
- **Feasibility**: 0.85 — 1 week implementation. After deploying the lazy-decayed-importance system (idea 1) and accumulating retrieval data, compute per-domain median effective importance scores. Store these as domain priors in PostgreSQL: `INSERT INTO domain_priors (domain, mean_importance, stddev) VALUES (...)`. When storing a new memory, set `raw_score = domain_priors.mean_importance` instead of the caller-provided importance value (or blend: `raw_score = 0.7 * domain_prior + 0.3 * caller_score`). This narrows the gap between claimed and demonstrated importance from the first access.
- **Description**: Add a background task (runs nightly or weekly) that computes per-domain importance statistics from the `raw_score` and `access_count` columns in PostgreSQL: `SELECT domain, AVG(effective_score), STDDEV(effective_score) FROM memories WHERE access_count > 0 GROUP BY domain`. Store results in a `domain_importance_priors` table. The store endpoint looks up this table to set initial raw_score for new memories, replacing the static default. Effect: new memories in frequently-retrieved domains start with a higher prior; new memories in rarely-accessed domains start lower. The dynamic system then corrects from this better starting point.

### Idea: importance-threshold-adaptive-retrieval

- **Novelty**: 0.60 — The retrieve path currently applies `final_score = similarity × importance` and returns top-k. If importance scores are now time-decayed and evidence-based, the threshold can be used adaptively: in a session where high-importance memories are available, raise the threshold to return only the most relevant ones. In a cold session with little context, lower the threshold to return more candidates. This is query-distribution-aware retrieval, not fixed-k.
- **Evidence**: 0.50 — Adaptive threshold selection has some basis in information retrieval (dynamic cutoff in learned sparse retrieval), but application to a personal AI memory system is not peer-reviewed. Logically sound: the optimal k is not a fixed number but a function of the score distribution of the current result set.
- **Feasibility**: 0.75 — 1-2 week implementation. After computing effective scores for candidates, inspect the score distribution: if the gap between rank-k and rank-(k+1) is large (a natural "elbow"), cut there instead of at a fixed k. If the top-5 scores cluster tightly around 0.8-0.9, return fewer (the memories are all similarly important). If scores spread from 0.9 to 0.1, return only those above 0.6. This is a sort-and-threshold operation on a small candidate set (O(k log k)) — negligible compute cost added to the retrieve path.
- **Description**: In the retrieve endpoint, after computing `effective_score = raw_score * λ^(Δt/τ) * similarity` for each candidate, sort the top-2k by effective_score. Find the "knee" of the sorted score curve: the index i where `score[i] - score[i+1] > 0.15` (a natural gap). Return memories `0..i` instead of a fixed k. If no natural gap exists, fall back to fixed k=5. This ensures the context window receives only memories that are meaningfully more relevant than the threshold, reducing noise in low-signal sessions and increasing recall in high-signal sessions.
