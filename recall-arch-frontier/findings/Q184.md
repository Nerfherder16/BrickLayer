**S4 — Epoch integrity (Redis/PG coherence)**

**S5 — Graph traversal epoch compliance**

## Ideas Extracted

### Idea: retrieval-quality-health-plane-is-fully-net-new
- **Novelty**: 0.90 — The complete absence of S1-S6 in a system with 22,423 memories and a running health dashboard is a structural finding. The research loop has been designing signals for an operational layer that does not exist. No AI memory system in reviewed literature has any of these retrieval quality signals.
- **Evidence**: 1.00 — Exhaustive source code audit with zero grep hits per signal. No ambiguity.
- **Feasibility**: 0.45 — S2 is the most feasible (add level-logging to retrieval.py, no new infrastructure). S3/S4/S5 require architectural substrates that represent weeks of work each. S6 is medium (feedback_summary table + cron).
- **Description**: S1-S6 are all net-new implementation work. Implementation priority order based on feasibility: S2 first (add MBC level logging to retrieval.py, ~1 day), then S6 (feedback_summary table + retention worker + S6 signal, ~3 days), then S1 (requires is_exploration + citation hook + propensity infrastructure, ~1-2 weeks), then S4 (epoch clock architecture, ~1 week), then S5 (Neo4j epoch filter + traversal predicate, ~1 week), then S3 (write-tier partition, ~2 weeks).

### Idea: co-retrieval-gravity-wrong-edge-type-live-in-production
- **Novelty**: 0.80 — The Q166 finding hypothesized this bug from absence evidence. This audit confirms the exact line (health.py:123) with the wrong filter (`== "RELATED_TO"` instead of `== "CO_RETRIEVED"`). It is live in production and has been computing the wrong metric since the health dashboard was built.
- **Evidence**: 1.00 — Direct code inspection. Line 123: `if r.get("rel_type", "").upper() == "RELATED_TO"`. CO_RETRIEVED edges do not exist (Q166 absence confirmed). The metric returns a value based on RELATED_TO edges, silently mislabeled as `co_retrieval_gravity`.
- **Feasibility**: 0.99 — 1-line fix: change filter to `"CO_RETRIEVED"`. However, this fix requires the CO_RETRIEVED flush worker (Q167) to exist first — otherwise the metric will always return 0.0, which is at least semantically correct.
- **Description**: The `co_retrieval_gravity` force in `health.py:compute_forces()` filters `rel_type == "RELATED_TO"` and returns that sum as co_retrieval_gravity. This produces a plausible-looking non-zero number that is entirely unrelated to co-retrieval behavior. The health dashboard displays this as a meaningful metric. The fix is one line after Q167 (flush worker) is deployed. Until then, the metric is noise.

### Idea: s2-mbc-level-logging-is-1-day-quick-win
- **Novelty**: 0.75 — Adding instrumentation to an existing retrieval pipeline to capture which routing tier succeeded is a well-understood pattern (Prometheus labels, structured logging). The novelty is in applying it to the specific MBC Level 1/2/3/fallback tier structure.
- **Evidence**: 0.90 — The retrieval pipeline in `src/core/retrieval.py` exists and handles the full MBC routing. Adding a level field to `injection_log` (or a new `retrieval_event_log` table) is a minor schema addition. Q071 specified exactly what to log.
- **Feasibility**: 0.92 — 1 day: (1) add `mbc_level` column to `injection_log` or a new table, (2) set the field at the level-decision point in retrieval.py, (3) add `get_mbc_level_1_fraction()` query to PostgreSQL store, (4) expose via `/health/retrieval-quality`. No architectural changes needed. This is the first S-signal that can be built without prerequisite infrastructure.
- **Description**: S2 is buildable in isolation by adding `mbc_level: 1|2|3|fallback` to the injection log at the moment the retrieval tier decision is made in retrieval.py. The `/health/retrieval-quality` endpoint already calls `pg.get_retrieval_quality_stats()` — extending that query to return `level_1_fraction` is straightforward. S2 immediately provides the vocabulary health signal (Q066's V≤22 target can be monitored in production for the first time).
