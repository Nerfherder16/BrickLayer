## Finding

### OBJECTIVE

Define the complete spec to: (a) add structured probe output to the worker's `run_reconcile`, (b) cache the probe result in Redis, (c) surface `s5_orphan_fraction` as a typed field in `/health`, and (d) confirm the weekly ARQ cron is already the correct scheduling mechanism.

### DATA

1. **Scheduling is already done.** `run_reconcile` is an ARQ cron job firing Sunday 5:30am. No new scheduler needed. The gap is that the result is only audit-logged, never written to a probe cache for `/health` to read.

2. **Read-only probe mode already works.** The ops.py endpoint uses `repair=False` as default. The worker (`workers/reconcile.py`) always runs with repair=True. For the scheduled probe, repair should be decoupled: the worker continues to repair, but separately a probe run writes a read-only snapshot. OR: the worker writes orphan metrics to Redis after its repair run (simpler — single source of truth).

3. **Health endpoint is opaque strings.** `checks["neo4j"] = "ok (22134 nodes)"`. S5_new requires a structured numeric field, not a parseable string. The `/health` response must be extended with a top-level `s5` section OR a `checks_structured` section to remain backward compatible.

4. **HealthComputer.compute_dashboard()** is the right integration point for the probe data. Adding `_orphan_sync()` as a parallel gather task follows the existing pattern exactly. However, a 64ms full-scan inside the dashboard 5-minute cache loop is acceptable (64ms vs 300s TTL = 0.04% overhead).

5. **Redis cache key pattern** for health data: `recall:health:{name}` is consistent across the codebase (`recall:health:dashboard`, `recall:health:retrieval_quality`, `recall:health:memory_quality`). Probe key should be `recall:health:orphan_probe`.

6. **`last_clean_ts`**: The probe result must record the timestamp of the last run where `orphan_fraction == 0`. This requires the worker to conditionally write to a separate Redis key only when clean, or always write the run timestamp and separately track the last-clean timestamp.

### FINDING

The upgrade requires **four discrete changes** with no structural risk:

1. **Worker (`workers/reconcile.py`)**: Compute `orphan_fraction`, write structured probe result to `recall:health:orphan_probe` in Redis after each run.
2. **HealthComputer (`core/health.py`)**: Add `_orphan_sync()` method that reads `recall:health:orphan_probe` from Redis (zero cost — just a Redis GET).
3. **`compute_dashboard()`**: Add `orphan_sync` to the `asyncio.gather()` call and include it in the return dict.
4. **`/health` endpoint (`api/main.py`)**: Add a structured `s5` section alongside the existing `checks` dict.

No new scheduler. No APScheduler integration. No changes to the ops.py endpoint (it can remain as a manual repair trigger).

### Change 1: Worker writes probe result to Redis

**File**: `src/workers/reconcile.py`

After the existing `logger.info("reconcile_complete", **stats)` block, add:

```python
import json
from datetime import UTC, datetime

# Compute S5_new signal
qdrant_total = len(qdrant_ids)
orphan_count = len(qdrant_orphans)
orphan_fraction = orphan_count / qdrant_total if qdrant_total > 0 else 0.0

probe_ts = datetime.now(UTC).isoformat()
last_clean_ts = None

# Read existing probe to preserve last_clean_ts
try:
    redis = await get_redis_store()
    existing_raw = await redis.client.get("recall:health:orphan_probe")
    if existing_raw:
        existing = json.loads(existing_raw)
        last_clean_ts = existing.get("last_clean_ts")
except Exception:
    pass

if orphan_fraction == 0.0:
    last_clean_ts = probe_ts  # update clean timestamp

probe_result = {
    "orphan_count": orphan_count,
    "orphan_fraction": round(orphan_fraction, 6),
    "qdrant_total": qdrant_total,
    "neo4j_total": len(neo4j_ids),
    "importance_mismatches": importance_mismatches,
    "superseded_mismatches": superseded_mismatches,
    "last_probe_ts": probe_ts,
    "last_clean_ts": last_clean_ts,
    "status": "HEALTHY" if orphan_fraction < 0.01 else "WARNING" if orphan_fraction < 0.05 else "FAILURE",
}

try:
    redis = await get_redis_store()
    # TTL: 8 days — survives one missed weekly run
    await redis.client.setex(
        "recall:health:orphan_probe",
        691200,  # 8 * 24 * 3600
        json.dumps(probe_result),
    )
except Exception as e:
    logger.warning("reconcile_probe_cache_write_failed", error=str(e))
```

Add method to `HealthComputer`:

```python
async def _orphan_sync(self) -> dict[str, Any]:
    """Read cached reconcile probe result (S5_new signal)."""
    try:
        raw = await self.redis.client.get("recall:health:orphan_probe")
        if raw:
            return json.loads(raw)
    except Exception:
        pass
    # Probe not yet run — return unknown state
    return {
        "orphan_count": None,
        "orphan_fraction": None,
        "qdrant_total": None,
        "neo4j_total": None,
        "last_probe_ts": None,
        "last_clean_ts": None,
        "status": "UNKNOWN",
    }
```

```python
@app.get("/health")
async def health_check() -> dict[str, object]:
    """Health check endpoint (always public for monitoring)."""
    checks = { ... }  # existing checks unchanged

    # S5_new: orphan sync signal
    s5 = {"status": "UNKNOWN", "orphan_fraction": None, "last_probe_ts": None}
    try:
        redis = await get_redis_store()
        probe_raw = await redis.client.get("recall:health:orphan_probe")

            probe = _json.loads(probe_raw)

                "orphan_count": probe.get("orphan_count"),

        healthy = False

    return {
        "status": "healthy" if healthy else "degraded",
        "timestamp": datetime.now(UTC).isoformat(),
        "checks": checks,
        "s5": s5,   # NEW: backward-compatible addition
```

### Scheduling: No change needed

ARQ cron in `workers/main.py` already fires `run_reconcile` every Sunday 5:30am:
```python
cron(run_reconcile, weekday=6, hour=5, minute=30)
```

This is the correct and only scheduling surface. APScheduler is installed but not wired into the app — do not use it.

### Status thresholds (aligned with Q132)

| orphan_fraction | status |
|-----------------|--------|
| < 1% (0.01) | HEALTHY |
| 1–5% (0.01–0.05) | WARNING |
| ≥ 5% (0.05+) | FAILURE |

FAILURE downgrades `/health` overall status from `healthy` to `degraded`.

### On-demand ops.py endpoint

`POST /reconcile?repair=true|false` remains unchanged as a manual repair trigger for operators. It does not write to `recall:health:orphan_probe` (that is the worker's responsibility). This separation keeps the probe cache stable — only the weekly scheduled worker updates it, preventing on-demand calls from resetting `last_clean_ts` prematurely.

### Implementation risk

- **Zero structural risk**: all changes are additive. `/health` consumers see a new `s5` key; they are not affected if they ignore it.
- **Redis dependency**: if Redis is unavailable, `_orphan_sync()` returns `UNKNOWN`, does not throw.
- **First run**: `s5.status = UNKNOWN` until the first weekly cron fires (at most 7 days). Can be short-circuited by calling `POST /reconcile?repair=false` once manually and manually writing the probe — or by triggering `run_reconcile` once via ARQ.
- **Duration tracking**: The probe does not currently track `duration_ms`. To add it, record `time.monotonic()` at start of `run_reconcile` and compute delta before writing probe. Optional for the initial implementation.
