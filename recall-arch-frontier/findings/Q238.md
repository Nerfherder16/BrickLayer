## Research Question

Design the interface between the memory system and the reasoning system (LLM) using only physical requirements. Forbidden: text, string, token, format, inject, context, prompt, prefix, message, write, output, generate, send, pass, feed, communicate, transfer, serialize, encode, decode, payload. What does the reasoning system physically need from memory, and does that exchange have to be language?

---

## What the Reasoning System Physically Needs

An LLM is a function:

```
P(next_symbol | prior_symbols) → distribution over next symbol
```

What the reasoning system needs from memory is: **a modification of its conditional distribution that increases the probability of correct outputs**.

Formally: let R be the distribution over responses given current context C. The memory system must produce a change ΔP such that:

```
H(R | C, memory_information) < H(R | C)
```

— the response distribution has lower entropy (is more certain) after receiving memory information.

**Physical requirement: the memory system must deliver a constraint on the response distribution.**

This can be done at multiple levels:

1. **Symbol sequence level**: modify C to include past observations → P(R | C + past_observations)
2. **Activation level**: modify the internal activation state directly → P(R | modified_activations)
3. **Parameter level**: modify the weights that define P → different model with knowledge embedded

The first is text injection (current approach). The second is the optimal physical interface.

---

## Why Symbol Sequence Injection Is Not the Physical Optimum

**The binding constraint of LLM inference**: context window capacity (measured in symbols consumed).

Text injection costs: `symbols_of_memory_content` context window units. For K=10 memories × 100 chars × 0.25 tokens/char ≈ 250 tokens consumed per query. This is 5-25% of a typical context window used purely for memory delivery.

**The waste**: the LLM must also attend to the reasoning task, the conversation history, and the system instructions. Memory competes for context window space with all of these. For long-running sessions or large codebases, the context window is the binding constraint — and text injection is maximally wasteful because it uses the most expensive resource (context window symbols) for memory delivery.

**The optimal physical interface**: modify the LLM's conditional distribution at **the activation level**, bypassing the symbol layer entirely.

---

## The Physical Optimum: KV-Cache Extension

Modern LLM inference caches attention key-value pairs (KV-cache). During generation, each attention head accesses KV pairs from prior positions. If we could add KV pairs representing memory content without consuming input symbol positions, the memory information would condition generation without using context window capacity.

This is **KV-cache extension**: pre-compute KV representations of memories and append them to the attention scope without consuming input positions.

```
Standard generation:
  KV-cache = KV(system_instructions, conversation_history, current_input)
  P(next | all positions)

With KV memory extension:
  KV-cache = KV(system_instructions, conversation_history, current_input)
             + KV_memory(stored_observation_1) + KV_memory(stored_observation_2) + ...
  P(next | all positions + memory positions)

  Memory positions DO NOT COUNT toward context window length.
  They are attended to but not visible as input symbols.
```

This is the mechanism used in:
- **FiD (Fusion-in-Decoder)**: retrieved passages attend directly to decoder generation
- **REALM, Atlas, RAG**: retrieval-augmented generation at the attention layer
- **Gist tokens** (Mu et al., 2023): compress long contexts into short "gist" KV vectors
- **MemGPT infinite context**: pages memory in/out at the activation level

---

## Is the Optimum Achievable with Current Recall 2.0 Architecture?

**Current constraint**: Recall 2.0 uses API-accessible inference (Ollama + qwen3:14b, Claude API). Neither exposes KV-cache extension. Symbol injection (context window prepend) is the only available mechanism through standard inference APIs.

**The path from current to optimal**:

1. **Near-term** (1-2 years): Ollama adds KV-cache injection API (experimental work in progress). When available, Recall 2.0 can deliver memories at the activation level for the local Ollama model.

2. **Current approximation**: prefix caching (Ollama and Claude both support prompt caching). Stable memories (infrastructure facts, persistent preferences) can be cached as a prefix — the KV computation is done once, not per-query. This doesn't eliminate context window cost but eliminates the compute cost of re-attending to stable memories.

3. **Structural optimization now**: since symbol injection is the only available mechanism, the priority is minimizing the symbol count per unit of information delivered. This is the **compression problem** for memory injection:
   - Current: 50-200 chars per memory in prose form
   - Optimal: entity-predicate-value assertion (Q237) in minimal form: "Recall API: port=8200" (19 chars)
   - Ratio: 3-10× improvement in context window efficiency

---

## What This Reveals About Current Memory System Design

The physical analysis reveals that **all current memory systems have converged on symbol injection because that is what the API allows**, not because it is physically optimal. The design decision is an artifact of API constraints, not a principled choice.

The physically optimal interface (KV-cache extension) would:
- Deliver memories with zero context window cost
- Allow K=100+ memories without competing with task context
- Enable memories to influence every attention head without explicit positioning

Until KV-cache APIs are available, the correct optimization target is **maximum information per symbol** in the injected memory representation — which points to Q237's assertion-level granularity and structured (entity: predicate=value) notation.

## Verdict

**BREAKTHROUGH.** The physical optimum for the memory-reasoning interface is KV-cache extension — delivering memory constraints at the activation level without consuming context window symbols. This mechanism is validated by FiD/REALM/Atlas/Gist tokens but is not achievable through current Ollama/Claude APIs. The TABOO analysis reveals that text injection is an API constraint, not a principled design choice, and that the correct interim optimization target is maximum information per symbol in the injected form. This points directly to Q237's assertion-level storage granularity (entity-predicate-value notation) as the structured representation that minimizes context window cost per unit of information delivered. The convergence of Q237 (storage granularity), Q234 (mutual information), Q235 (push-based surfacing), and Q238 (activation-level interface) reveals a coherent physical architecture for memory systems that no current production system has fully implemented.
