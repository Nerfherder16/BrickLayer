# Finding: Q033 — Absence Mapping: Query-Embedding-Only Cache (Type B)

**Question**: Has any production semantic search or AI memory system implemented a separate query-embedding cache (keyed on query text hash, independent of the full result cache) to eliminate redundant embedding calls for repeated queries?
**Question Type**: ABSENCE
**Verdict**: PROMISING
**Severity**: High
**Source field**: Database caching theory (stable-key caching), information retrieval

---

## Mechanism Under Investigation

A cache keyed on query-text hash whose value is the raw embedding vector — specifically isolated from the result cache so that it (a) persists across index changes without invalidation, and (b) eliminates the embedding-model call (e.g., Ollama's 50ms) independently of whether the full search result is cached.

## Absence Classification

**Classification**: PARTIAL ABSENCE

### Detail

Type B (query-embedding-only cache, stable, no invalidation) exists in exactly one major production framework — LangChain's `CacheBackedEmbeddings` — but with the following critical limitations that explain why the novelty score of prior questions was understated:

**LangChain `CacheBackedEmbeddings`:**
- `document_embedding_store` (required): caches document embedding vectors keyed on text hash + namespace. Type B. No invalidation needed.
- `query_embedding_store` (optional, default=`None`): if provided, caches query embedding vectors keyed on text hash + namespace. Type B. No invalidation needed.
- **Critical gap**: query caching is opt-in and off by default. The LangChain documentation actively warns against it without explaining the stability property. PR #19926 (merged Apr 2024) added caveats to the docs noting that `embed_query` is not cached — indicating this is considered a footgun rather than a recommended pattern.
- No LangChain tutorial or canonical example enables `query_embedding_store`. The opt-in friction means the vast majority of production deployments leave it disabled.

**LlamaIndex `IngestionCache`:**
- Caches node+transformation output pairs (including embeddings) during document ingestion pipelines. Type B for the ingestion path.
- Explicitly not applied at query time — `IngestionPipeline` runs once at document load, not on every search query.
- No `QueryEmbeddingCache` equivalent exists in LlamaIndex.

**All other systems:**
- Implement Type A (full result / semantic result cache), Type A+ (semantic similarity-based result cache like QVCache), or no caching.
- Zero production systems other than LangChain expose a query-embedding-only cache as a first-class API.

### Why the Gap Exists

1. **The stability property is not widely understood.** Most engineers conflate the result cache (which depends on index state) with the embedding cache (which depends only on model+text). When result caches require invalidation, engineers assume all caches tied to a query do too. LangChain's own docs don't explain why query embeddings are stable.

2. **Semantic caching collapsed the layers.** The dominant 2024-2026 optimization trend is semantic caching of LLM responses (GPTCache, Redis semantic cache, Azure Cosmos DB semantic cache). These cache at the LLM output level and include embedding generation as a component of the cache lookup — so the embedding call still happens for every query (to compute the cache key). Nobody separated "embed to make a cache key" from "embed to search."

3. **LangChain's opt-in friction.** Even though LangChain has the type B cache, the API makes it opt-out-by-default and the docs treat it as a footgun. This suppresses adoption and means it's not recognized as a pattern.

4. **Framework focus is on ingestion, not query.** Both LangChain and LlamaIndex primarily designed embedding caches for the expensive document ingestion path (many documents, slow, repeated on reload). Query embedding caching (one vector per search) was a secondary concern — even though query latency is on the critical path and document ingestion is not.

5. **Local embedding model users feel the pain most.** Systems calling OpenAI's embedding API at ~10ms may deprioritize a cache. Systems running a local model (Ollama at 50ms) feel it acutely — but Ollama-based RAG deployments are a minority of the LangChain/LlamaIndex user base. This is exactly Recall's position.

## Ideas Extracted

### Idea: query-embedding-cache-first-class

- **Novelty**: 0.60 — LangChain's `query_embedding_store` exists but is opt-in and underdocumented; no other production system implements this; the invalidation-stability insight is essentially never articulated
- **Evidence**: 0.75 — Source mechanism (deterministic embedding model + fixed text = fixed vector) is mathematically certain; confirmed by LangChain implementation; Athenic advisory confirms the two-layer architecture is correct
- **Feasibility**: 0.90 — Trivially implementable: Redis hash `query_text_hash -> embedding_vector`; zero invalidation logic needed; no coordination with index state
- **Description**: Implement a query embedding cache as a first-class, always-on component in Recall's retrieve path, keyed on `SHA256(query_text + model_name)`. The cache value is the raw embedding vector (`list[float]`). TTL is indefinite (or set to model lifetime, e.g., 90 days). No invalidation logic is needed — when the embedding model changes, the cache namespace (model name) changes automatically. This collapses the 50ms Ollama call to a ~0.5ms Redis GET for any repeated query. The cache is separate from and orthogonal to the result cache (which is invalidated on new memory stores). Expected hit rate for Recall: 30-50% of queries are repeated within a session or across sessions (same project, same person, same recurring topics).

### Idea: expose-stability-property-as-design-principle

- **Novelty**: 0.70 — The explicit articulation that "query embeddings are index-independent and never need invalidation" is absent from every framework's documentation; this is a design principle gap, not just an implementation gap
- **Evidence**: 0.80 — Confirmed by examining LangChain source (no invalidation mechanism exists for the embedding cache), and by the mathematical definition of a deterministic embedding model
- **Feasibility**: 0.95 — No implementation required; this is a documentation and design pattern contribution
- **Description**: Document the type A / type B distinction explicitly in Recall's architecture notes and in any public writing. The stability property is the key insight: result caches (type A) invalidate on index change; embedding caches (type B) invalidate only on model change. Most engineers conflate these. Making this explicit enables engineers to adopt type B caches confidently.

---

## Verdict Justification

PROMISING rather than BREAKTHROUGH because LangChain does implement the mechanism (preventing a TRUE ABSENCE verdict) — but the opt-in default, zero documentation of the stability property, and complete absence from all other major systems (LlamaIndex query path, Semantic Kernel, mem0, Zep, all vector DB clients) confirms a substantial practical gap. The novelty score for the implementation idea is 0.60 (below the 0.65 BREAKTHROUGH threshold) because the mechanism is proven in one framework. The design-principle idea scores 0.70 novelty but lower overall quality due to being non-code. Combined verdict: PROMISING.
