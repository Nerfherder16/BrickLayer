## SQL Measurement Queries

All queries target the production Recall PostgreSQL instance at `192.168.50.19:8200`. Run them in order; Query 1 is the primary calibration measurement.

```sql
-- ════════════════════════════════════════════════════════════════════
-- QUERY 1: EFFECTIVE VOCABULARY SIZE (effective_V)
-- The minimum vocabulary covering 90% of tag assignments.
-- This is the single number that determines MBC L1 hit rate.
-- ════════════════════════════════════════════════════════════════════

WITH
-- Step 1: unnest all tags into individual rows
tag_counts AS (
  SELECT
    t.tag,
    COUNT(*) AS assignment_count
  FROM memories m,
       UNNEST(m.tags) AS t(tag)
  GROUP BY t.tag
),
-- Step 2: rank tags by frequency, compute running coverage fraction
tag_ranked AS (
  SELECT
    tag,
    assignment_count,
    SUM(assignment_count) OVER ()                                    AS total_assignments,
    SUM(assignment_count) OVER (ORDER BY assignment_count DESC
                                ROWS BETWEEN UNBOUNDED PRECEDING
                                         AND CURRENT ROW)           AS cumulative_assignments,
    ROW_NUMBER() OVER (ORDER BY assignment_count DESC)               AS rank
  FROM tag_counts
),
-- Step 3: find the cutoff rank where cumulative coverage reaches 90%
coverage_cutoff AS (
  SELECT
    MIN(rank) AS effective_V_90pct
  FROM tag_ranked
  WHERE cumulative_assignments >= 0.90 * total_assignments
)
SELECT
  -- Primary result: effective_V
  cc.effective_V_90pct                                               AS effective_V,
  -- Context
  (SELECT COUNT(DISTINCT tag) FROM tag_counts)                       AS total_distinct_tags,
  (SELECT SUM(assignment_count) FROM tag_counts)                     AS total_tag_assignments,
  (SELECT COUNT(*) FROM memories)                                    AS total_memories,
  -- Interpretation helper
  CASE
    WHEN cc.effective_V_90pct <= 15  THEN 'ABOVE TARGET — P(L1) > 70% at T=3, C=5. Deploy as-is.'
    WHEN cc.effective_V_90pct <= 22  THEN 'IN TARGET — P(L1) 55-70% at T=3, C=5. Deploy as-is.'
    WHEN cc.effective_V_90pct <= 30  THEN 'BORDERLINE — P(L1) 43-54%. Add synonym normalization.'
    WHEN cc.effective_V_90pct <= 50  THEN 'NORMALIZATION REQUIRED — P(L1) 28-42%. Restrict to top-30 tags.'
    ELSE                                  'COMPACTION CRITICAL — P(L1) < 28%. Full corpus re-tagging needed.'
  END                                                                AS action_required
FROM coverage_cutoff cc;


-- ════════════════════════════════════════════════════════════════════
-- QUERY 2: TOP 30 TAGS BY FREQUENCY
-- Reveals the actual vocabulary distribution and synonym candidates.
-- ════════════════════════════════════════════════════════════════════

WITH
tag_counts AS (
  SELECT
    t.tag,
    COUNT(*)                                        AS assignment_count,
    COUNT(DISTINCT m.memory_id)                     AS distinct_memories
  FROM memories m,
       UNNEST(m.tags) AS t(tag)
  GROUP BY t.tag
),
totals AS (
  SELECT SUM(assignment_count) AS grand_total FROM tag_counts
)
SELECT
  tc.tag,
  tc.assignment_count,
  tc.distinct_memories,
  ROUND(tc.assignment_count::numeric / t.grand_total * 100, 2)       AS pct_of_all_assignments,
  ROUND(
    SUM(tc.assignment_count) OVER (ORDER BY tc.assignment_count DESC
                                   ROWS BETWEEN UNBOUNDED PRECEDING
                                            AND CURRENT ROW)::numeric
    / t.grand_total * 100, 2
  )                                                                   AS cumulative_pct,
  ROW_NUMBER() OVER (ORDER BY tc.assignment_count DESC)              AS rank
FROM tag_counts tc, totals t
ORDER BY tc.assignment_count DESC
LIMIT 30;


-- ════════════════════════════════════════════════════════════════════
-- QUERY 3: ZERO-TAG FAILURE RATE
-- Memories with empty tags[] cannot be matched by MBC Level 1.
-- This directly reduces effective L1 hit rate.
-- ════════════════════════════════════════════════════════════════════

SELECT
  COUNT(*)                                                           AS total_memories,
  COUNT(*) FILTER (WHERE tags IS NULL OR array_length(tags, 1) IS NULL
                      OR array_length(tags, 1) = 0)                  AS zero_tag_memories,
  COUNT(*) FILTER (WHERE array_length(tags, 1) = 1)                  AS single_tag_memories,
  COUNT(*) FILTER (WHERE array_length(tags, 1) = 2)                  AS two_tag_memories,
  COUNT(*) FILTER (WHERE array_length(tags, 1) >= 3)                 AS three_plus_tag_memories,
  ROUND(
    COUNT(*) FILTER (WHERE tags IS NULL OR array_length(tags, 1) IS NULL
                        OR array_length(tags, 1) = 0)::numeric
    / COUNT(*) * 100, 2
  )                                                                   AS zero_tag_pct,
  ROUND(AVG(COALESCE(array_length(tags, 1), 0))::numeric, 2)         AS avg_tags_per_memory,
  ROUND(PERCENTILE_CONT(0.50) WITHIN GROUP
        (ORDER BY COALESCE(array_length(tags, 1), 0))::numeric, 1)   AS median_tags_per_memory,
  -- By domain (reveals which hook/domain is failing to assign tags)
  -- Run separately if needed:
  -- SELECT domain, COUNT(*) FILTER (WHERE ...) / COUNT(*) AS zero_tag_rate
  -- FROM memories GROUP BY domain ORDER BY zero_tag_rate DESC
  'See domain breakdown below'                                        AS domain_note
FROM memories;

-- Domain-level zero-tag breakdown (run as separate query):
SELECT
  domain,
  COUNT(*)                                                           AS total,
  COUNT(*) FILTER (WHERE tags IS NULL OR array_length(tags, 1) IS NULL
                      OR array_length(tags, 1) = 0)                  AS zero_tag_count,
  ROUND(
    COUNT(*) FILTER (WHERE tags IS NULL OR array_length(tags, 1) IS NULL
                        OR array_length(tags, 1) = 0)::numeric
    / COUNT(*) * 100, 2
  )                                                                   AS zero_tag_pct,
  ROUND(AVG(COALESCE(array_length(tags, 1), 0))::numeric, 2)         AS avg_tags_per_memory
FROM memories
GROUP BY domain
ORDER BY zero_tag_pct DESC;


-- ════════════════════════════════════════════════════════════════════
-- QUERY 4: SYNONYM CANDIDATE DETECTION
-- Finds tag pairs that co-occur frequently enough to be synonyms.
-- Run after reading Query 2 results to confirm normalization targets.
-- ════════════════════════════════════════════════════════════════════

WITH
tag_pairs AS (
  SELECT
    LEAST(t1.tag, t2.tag)    AS tag_a,
    GREATEST(t1.tag, t2.tag) AS tag_b,
    COUNT(*)                 AS co_occurrence_count
  FROM memories m,
       UNNEST(m.tags) AS t1(tag),
       UNNEST(m.tags) AS t2(tag)
  WHERE t1.tag < t2.tag
  GROUP BY LEAST(t1.tag, t2.tag), GREATEST(t1.tag, t2.tag)
  HAVING COUNT(*) >= 3   -- minimum co-occurrence threshold
),
tag_counts AS (
  SELECT t.tag, COUNT(*) AS freq
  FROM memories m, UNNEST(m.tags) AS t(tag)
  GROUP BY t.tag
)
SELECT
  tp.tag_a,
  tc_a.freq                 AS freq_a,
  tp.tag_b,
  tc_b.freq                 AS freq_b,
  tp.co_occurrence_count,
  ROUND(tp.co_occurrence_count::numeric / LEAST(tc_a.freq, tc_b.freq) * 100, 1)
                            AS jaccard_approx_pct,
  -- High overlap = likely synonyms or parent-child relationships
  CASE
    WHEN tp.co_occurrence_count::numeric / LEAST(tc_a.freq, tc_b.freq) >= 0.85
      THEN 'LIKELY SYNONYM — merge rarer into more frequent'
    WHEN tp.co_occurrence_count::numeric / LEAST(tc_a.freq, tc_b.freq) >= 0.50
      THEN 'PARENT-CHILD — keep both or merge to parent'
    ELSE 'CO-TOPIC — keep separate'
  END                       AS relationship_type
FROM tag_pairs tp
JOIN tag_counts tc_a ON tp.tag_a = tc_a.tag
JOIN tag_counts tc_b ON tp.tag_b = tc_b.tag
WHERE tc_a.freq >= 5 OR tc_b.freq >= 5   -- only tags with meaningful frequency
ORDER BY jaccard_approx_pct DESC
LIMIT 40;


-- ════════════════════════════════════════════════════════════════════
-- QUERY 5: GIN INDEX STATUS CHECK
-- Verifies the MBC routing prerequisite index exists.
-- ════════════════════════════════════════════════════════════════════

SELECT
  indexname,
  indexdef,
  -- Index size in MB
  pg_size_pretty(pg_relation_size(indexname::regclass)) AS index_size
FROM pg_indexes
WHERE tablename = 'memories'
  AND indexdef ILIKE '%gin%'
ORDER BY indexname;

-- If no GIN index on tags exists, create it:
-- CREATE INDEX CONCURRENTLY idx_memories_tags_gin ON memories USING GIN(tags);
-- (Takes minutes on existing data; safe to run on live table)
```

## Analytical Predictions by effective_V

Formula (exact hypergeometric, from Q060):

```
P(L1 hit) = 1 - C(V-C, T) / C(V, T)

where: V = effective vocabulary size
       T = average tags per memory = 3 (observed estimate)
       C = context_tags set size = 5 (early session, conservative)
```

### How context_tags accumulation rescues higher V values (T=3)

The Q045 MBC design accumulates context_tags across the session. C grows from ~5 (cold start) to 12-15 (well-progressed session). The table below shows how session warmup compensates for larger vocabularies:

| V | C=3 (cold) | C=5 (early) | C=8 (mid) | C=12 (warm) | C=15 (full session) |
|:-:|:----------:|:-----------:|:---------:|:-----------:|:-------------------:|
| 20 | 40.4% | 60.1% | 80.7% | 95.1% | 99.1% |
| 25 | 33.0% | 50.4% | 70.4% | 87.6% | 94.8% |
| 30 | 28.0% | 43.3% | 62.1% | 79.9% | 88.8% |
| 50 | 17.3% | 27.6% | 41.4% | 57.0% | 66.6% |

**Insight**: Even at V=50 (pessimistic), a session that accumulates C=12 context_tags achieves 57% L1 hit rate — inside the target. Cold-start quality (C=3-5) is the critical failure mode for any vocabulary size above V=22.

### Zero-tag failure rate decision criteria

Zero-tag memories cannot participate in Level 1 routing at all. They degrade the effective L1 hit rate multiplicatively:

```
effective_P(L1) = (1 - zero_tag_fraction) × P(L1 | V, T, C)
```

| zero_tag_fraction | Effective P(L1) at V=20 | Status |
|:-----------------:|:-----------------------:|:-------|
| 0% | 60.1% | OK |
| 5% | 57.1% | OK (barely) |
| 10% | 54.1% | BELOW TARGET — fix tag assignment pipeline |
| 15% | 51.1% | BELOW TARGET — urgent |
| 20% | 48.1% | BELOW TARGET — urgent |
| 30% | 42.1% | CRITICAL — vocabulary compaction irrelevant until this is fixed |

**Threshold**: zero_tag_fraction > 8% makes even a perfectly-sized vocabulary (V=20) fail the 55% floor. Fix zero-tag assignment before measuring effective_V.

## Synonym Normalization Proposal

The following normalization map should be applied in the `observe-edit.js` store hook immediately before writing tags to PostgreSQL. This brings the effective_V from the "likely" range (V≈28-35) toward the "best case" range (V≈18-22).

### Implementation point in observe-edit.js

```javascript
// In the store path, after LLM tag extraction, before API call:
const NORMALIZE_MAP = { /* see table below */ };

function normalizeTags(rawTags) {
  const seen = new Set();
  const normalized = [];
  for (const tag of rawTags) {
    const canonical = NORMALIZE_MAP[tag.toLowerCase().trim()] ?? tag.toLowerCase().trim();
    if (!seen.has(canonical)) {
      seen.add(canonical);
      normalized.push(canonical);
    }
  }
  return normalized;
}
```
