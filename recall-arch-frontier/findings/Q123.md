# Q123 — Redis Maxmemory Configuration

**Question**: What is the Redis maxmemory limit and eviction policy for the three-store stack after the embedding cache (Q116) and session pin sets (Q016) are deployed — which Redis keys must be immune from eviction (epoch counter, session pin sets, canary seed IDs), which are eviction candidates (embedding cache entries), and what is the maxmemory-policy and maxmemory value to configure in docker-compose.yml to prevent the epoch counter from being evicted when Redis hits its memory ceiling?
**Agent**: quantitative-analyst
**Question Type**: IMPLEMENTATION
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Redis persistence architecture, eviction policy design, distributed systems invariants

---

## Executive Summary

After deploying Q016 (session pin sets), Q116 (embedding cache), and the existing Q111/Q103 infrastructure (epoch counter, canary seeds), Redis hosts four distinct key categories with radically different criticality profiles. The current `docker-compose.yml` has no `maxmemory` or eviction policy configured — which means Redis will OOM-kill the container when it hits the Docker memory limit of 256 MB, rather than applying any eviction policy at all. This is strictly worse than any configured policy.

The correct solution is not to fight Redis eviction policy semantics. It is to **exploit** them: configure `maxmemory-policy volatile-lru` and ensure that only eviction-eligible keys carry a TTL. Critical keys (epoch counter, pin sets, canary seeds) have no TTL and are therefore structurally immune to `volatile-lru` eviction. Embedding cache entries have 48-hour TTLs and are the only keys eligible for LRU eviction. This produces the required behavior without any application-level watchdog.

## Redis Key Inventory (Post-Deployment)

### Tier 1: Immutable Infrastructure — Must Never Be Evicted

| Key pattern | Content | TTL today | Size |
|---|---|---|---|
| `recall:global_epoch` | 8-byte integer counter | **None** | ~100 bytes (key + value + metadata) |
| `recall:canary_seeds` | Sorted set of 10 UUIDs × 36 bytes | **None** | ~500 bytes |

**Why these are fatal if evicted**:

- `recall:global_epoch`: The epoch sanity guard (Q111 Layer 1) reads this on every `/search/browse` call. An evicted epoch counter is indistinguishable from a Redis restart at epoch 0. The guard would fire on the next retrieve call, log `epoch_reset_detected`, and repair the value — but there is a window of up to one full retrieve-path call where the epoch counter is absent. More critically, the session-end consolidation worker increments this key; if it is evicted mid-session, the counter restarts from 0 and any causal eviction logic dependent on epoch ordering is corrupted for that session window.

- `recall:canary_seeds`: The `/health/recall-quality` endpoint reads this set to know which UUIDs to probe and which paraphrase queries to embed. An evicted canary set produces a 500 error (or silently passes all health checks because there are no seeds to check). The canary monitoring system becomes blind without triggering any alert.

### Tier 2: Session-Scoped Operational Data — Must Survive for Session Lifetime

| Key pattern | Content | TTL today | Size per key | Peak total |
|---|---|---|---|---|
| `session:{id}:pins` (Q016 SRS) | Redis set of memory UUIDs, ~10 per session | 6-hour TTL | ~360 bytes | ~18 KB (50 sessions) |
| `recall:session:*` (arq job state, existing) | ARQ worker queue and job metadata | Varies (ARQ-managed) | ~200–2,000 bytes | ~50–500 KB |

**Why session pin sets are special**: These keys already have TTLs (6-hour session lifetime + 1-hour buffer, per Q016's design). They are therefore technically `volatile-lru` candidates. However, evicting an in-session pin set is a correctness failure: an active session's SRS is evicted mid-session, meaning the consolidation worker at session end builds an empty protected set and may evict pinned memories that are causally active.

The mitigation: session pin sets are short-lived (6-hour TTL) and small (~360 bytes). At the memory budget derived below, they will never be eviction candidates in practice. The binding protection is the `maxmemory` headroom, not the eviction policy.

### Tier 3: Ephemeral Cache — Intended Eviction Candidates

| Key pattern | Content | TTL | Size per entry | Peak total |
|---|---|---|---|---|
| `recall:embedding_cache:{model_ver}:{sha256}` (Q116) | 6,144 bytes binary float32 vector | 48-hour TTL | ~6,204 bytes | ~6.1 MB (1,000 entries) |

These are the **only keys that should be evicted** when Redis approaches its memory ceiling. They carry 48-hour TTLs and are structurally correct victims for `volatile-lru`: an evicted embedding cache entry causes a cache miss (adds 1.68 ms to the next retrieve call) but produces no correctness failure.

## Memory Budget Analysis

### Steady-State Footprint

| Component | Estimate | Basis |
|---|---|---|
| `recall:global_epoch` | 100 bytes | 8-byte value + key string + Redis overhead |
| `recall:canary_seeds` (sorted set, 10 members) | ~500 bytes | 10 × 36 bytes UUID + sorted set headers |
| Session pin sets (50 concurrent × 360 bytes) | ~18 KB | Q016 design; 50 sessions is conservative upper bound |
| ARQ worker queue + job state | ~500 KB | Existing operational estimate |
| Embedding cache at peak (1,000 entries × 6,204 bytes) | ~6.1 MB | Q116 worst case: 200 sessions × 5 unique queries |
| **Total operational data** | **~7 MB** | |
| Redis internal overhead (hash table resize, LRU clock, AOF buffer) | ~2–5 MB | Redis 7 overhead for ~1,000–2,000 keys |
| **Total Redis footprint at peak** | **~10–12 MB** | |

### Headroom Analysis

The Docker `deploy.resources.limits.memory: 256M` is set on the Redis service. Without a `maxmemory` configuration, Redis will expand until Docker OOM-kills the container at 256 MB — a hard kill that destroys all in-flight state, AOF buffer, and active connections. This is the worst possible outcome.

Configuring `maxmemory 100mb` with `volatile-lru`:
- At ~10–12 MB operational footprint, headroom is ~88 MB — a **~8× safety margin**
- Redis begins LRU eviction of volatile (TTL-bearing) keys at 100 MB
- The only TTL-bearing keys at scale are embedding cache entries (6.1 MB peak) and session pin sets (18 KB)
- Embedding cache entries are evicted first (LRU among volatile keys)
- Epoch counter and canary seeds are **never eligible** (no TTL)

### Why `volatile-lru` Solves the Key Protection Problem

Redis eviction policies sorted by relevance to this use case:

| Policy | Behavior | Verdict |
|---|---|---|
| `noeviction` | Returns OOM error on write when maxmemory is hit | Wrong: blocks cache writes, produces API errors |
| `allkeys-lru` | Evicts any key by LRU, regardless of TTL | Wrong: epoch counter and canary seeds are LRU candidates |
| `allkeys-lfu` | Evicts least-frequently-used among all keys | Wrong: epoch counter is rarely accessed (once per retrieve), would be evicted early |
| `volatile-lru` | Evicts least-recently-used keys among those with TTLs only | **Correct**: no TTL = immune; cache entries with TTLs = eligible |
| `volatile-lfu` | Evicts least-frequently-used among volatile keys | Acceptable alternative but LRU is sufficient for cache entries |
| `volatile-ttl` | Evicts volatile keys with shortest remaining TTL first | Acceptable but may evict freshly-set cache entries before stale ones |

**`volatile-lru` is the correct choice** because:

1. It creates a structural guarantee: any key without a TTL is permanently immune to eviction. The epoch counter and canary seeds will never have TTLs (they are permanent infrastructure). The policy makes their immunity a consequence of their design, not a Redis-specific workaround.

2. Embedding cache entries already have 48-hour TTLs by design (Q116). They are correctly classified as eviction candidates without any additional configuration.

3. Session pin sets carry 6-hour TTLs. At 18 KB total and an 88 MB headroom buffer, they will never be eviction candidates in practice. If they ever were evicted (pathological scenario: memory leak in another component), the correctness failure is bounded to a single session's eviction pass producing a less-conservative protected set — which is acceptable.

## The One Edge Case: `volatile-lru` with No Volatile Keys

Redis documents a subtle behavior: if `volatile-lru` is configured and Redis needs to evict, but **no keys have TTLs**, Redis behaves as `noeviction` — it returns OOM errors on writes rather than evicting anything.

**This cannot happen in the Recall deployment** because:
- Embedding cache entries always exist at system load (any active retrieve call populates the cache)
- Session pin sets accumulate during normal operation
- ARQ job keys have TTLs

However, during a fresh deployment before the first retrieve call, Redis could momentarily have no volatile keys. In this case, any write attempt returns OOM. The mitigation is the `maxmemory 100mb` headroom: at 100 MB limit and ~100 bytes of initial state, there are ~100,000,000 bytes of headroom before the `noeviction` behavior triggers. This is not a realistic concern.

---

## The Session Pin Set Protection Gap

The Q016 SRS keys carry 6-hour TTLs and are therefore `volatile-lru` candidates. This is a real (if bounded) risk. Two mitigations:

**Mitigation 1 — Memory headroom makes eviction impossible in practice**: At 100 MB maxmemory limit and ~7 MB operational footprint, Redis will not evict anything unless 93 MB of embedding cache entries accumulate. That requires ~15,000 embedding cache entries (15,000 × 6,204 bytes ≈ 93 MB). At Tim's 50–100 queries/day with 48-hour TTL, the maximum is ~200 entries at any time. Session pin sets will never be evicted.

**Mitigation 2 — Bump session pin set TTL slightly above embedding cache TTL**: If pin sets carry a 6-hour TTL and embedding cache entries carry a 48-hour TTL, LRU will evict older embedding cache entries first (lower LRU clock value). Session pin sets are updated on every retrieve call (their LRU clock is always recent). This provides additional practical protection even if the memory headroom calculation changes.

No code change is required for Mitigation 2 — it follows from the design.

**Required change**:

```yaml
  redis:
    image: redis:7-alpine
    container_name: recall-redis
    ports:
      - "127.0.0.1:6380:6379"
    volumes:
      - redis_data:/data
    command: >
      redis-server
      --appendonly yes
      --appendfsync everysec
      --save ""
      --maxmemory 100mb
      --maxmemory-policy volatile-lru
    deploy:
      resources:
        limits:
          memory: 256M
          cpus: '0.25'
    restart: unless-stopped
```

### Flag-by-Flag Rationale

| Flag | Value | Reason |
|---|---|---|
| `--appendonly yes` | (already present) | AOF persistence — keeps epoch counter and canary seeds across container restart |
| `--appendfsync everysec` | added (Q111 requirement) | Explicit AOF flush policy; required for the Q111 AOF health check assertion to be trustworthy |
| `--save ""` | added (Q111 recommendation) | Disables RDB snapshots; AOF-only mode is strictly better for this use case |
| `--maxmemory 100mb` | **new** | Caps Redis at 100 MB — leaves 156 MB headroom below Docker's 256 MB limit; prevents OOM-kill; triggers LRU eviction of volatile keys before container death |
| `--maxmemory-policy volatile-lru` | **new** | Only evicts keys with TTLs; epoch counter (`recall:global_epoch`) and canary seeds (`recall:canary_seeds`) have no TTL → structurally immune regardless of memory pressure |

### Why 100 MB (not 50 MB or 200 MB)

- **Not 50 MB**: The operational footprint is ~10–12 MB. A 50 MB limit provides 4× headroom. Acceptable, but the embedding cache budget at 6.1 MB peak leaves only ~32 MB of remaining headroom for ARQ growth, which is tight.
- **100 MB**: Provides ~88 MB headroom (8× operational footprint). At 100 MB, Redis begins evicting embedding cache entries. The cache still functions — it just grows to a maximum of ~(100 MB − 7 MB) / 6,204 bytes ≈ 15,000 entries before eviction kicks in. Tim's actual usage never exceeds 200 entries, so eviction never fires in normal operation. The 100 MB limit is a safety net, not a ceiling that the system normally approaches.
- **Not 200 MB**: Too close to the Docker 256 MB container limit. At 200 MB maxmemory and a Redis process overhead of ~20–30 MB, the container would still OOM-kill at 256 MB before the eviction policy had a chance to free memory.

## Ideas Extracted

### Idea: volatile-lru-structural-immunity-pattern
- **Novelty**: 0.72 — Using `volatile-lru` as a design instrument (deliberately omitting TTL from critical keys to make them structurally immune) rather than as a performance optimization is an architectural insight not typically documented in Redis deployment guides. Most guides treat eviction policy as a performance/memory tradeoff decision, not a correctness mechanism for key protection.
- **Evidence**: 0.85 — Redis documentation explicitly states that `volatile-lru` does not evict keys without TTLs; this is a guaranteed semantic, not an observed behavior. The Redis source (t_string.c, expire.c) confirms the volatile filter is applied before any LRU candidate selection. The invariant is robust.
- **Feasibility**: 0.97 — The pattern requires two things already true of the Recall design: (1) critical keys have no TTL (epoch counter and canary seeds are permanent by design), (2) cache entries have TTLs (Q116's 48-hour TTL is already required for memory hygiene). No application code changes are needed — only the eviction policy configuration.
- **Description**: Use `volatile-lru` eviction not primarily for performance but as a correctness mechanism: assign TTLs only to keys that are correct eviction candidates (embedding cache, job state), and omit TTLs from keys that must survive indefinitely (epoch counter, canary seeds). Under `volatile-lru`, TTL-absence becomes a structural immunity guarantee enforced by the Redis kernel, not by application-level watchdog logic. This eliminates the need for a separate watchdog process that detects and recreates evicted critical keys. The invariant is: "if a key should never be evicted, it must never have a TTL."

### Idea: maxmemory-as-oom-kill-prevention-not-cache-tuning
- **Novelty**: 0.68 — Framing `maxmemory` as a Docker OOM-kill prevention measure rather than a cache size tuning parameter is a deployment architecture insight. Without `maxmemory`, Redis can consume memory up to the Docker container limit, at which point the kernel OOM-kills the container — a hard kill that loses all in-flight state. With `maxmemory` set below the container limit, Redis's own eviction (graceful, controlled) runs before the kernel's OOM-killer (catastrophic, uncontrolled). The 100 MB / 256 MB ratio leaves 156 MB buffer for this purpose.
- **Evidence**: 0.80 — Docker container OOM behavior on Linux is well-documented: the kernel invokes the OOM killer when cgroup memory limit is exceeded, sending SIGKILL to the process. Redis AOF ensures recovery after a kill, but any in-flight writes, AOF buffer contents, and network connections are lost. A Redis-controlled eviction is semantically cleaner.
- **Feasibility**: 0.98 — One flag, one value. No application changes.
- **Description**: Set `maxmemory` to 40% of the Docker memory limit for any Redis service whose data includes both eviction-eligible cache and non-eviction-eligible critical state. The 40% rule (100 MB of 256 MB) ensures (a) the eviction policy runs before the kernel OOM-killer, (b) Redis process overhead (~20–30 MB) plus the maxmemory budget stays well below the container limit, and (c) there is ~130 MB of headroom for future operational data growth before the limit must be revised.

### Idea: four-tier-redis-key-classification
- **Novelty**: 0.70 — Formalizing Redis keyspace into four tiers by criticality and evictability — (1) permanent infrastructure (no TTL, immune), (2) session-scoped operational (TTL, short-lived, avoid eviction), (3) ephemeral cache (TTL, intended eviction candidates), (4) transient job state (TTL, ARQ-managed) — is a design methodology not commonly applied to Redis deployments in AI memory systems. Most deployments use Redis as a single-tier "everything cache" without distinguishing key criticality.
- **Evidence**: 0.72 — The tier classification follows directly from the functional roles established across Q016, Q103, Q111, and Q116. The tiers map naturally to Redis eviction mechanics.
- **Feasibility**: 0.95 — Documentation and code review artifact. The classification should be codified in a Redis key registry comment or operational runbook section.
- **Description**: Maintain an explicit Redis key registry in the Recall codebase (e.g., `src/storage/redis_keys.py`) listing each key pattern, its tier (1–4), its TTL (or None), and its criticality. When a new Redis key is introduced (as has happened three times in the last five research sessions), the registry forces the author to classify the key and choose an appropriate TTL or confirm TTL-absence. This prevents the silent introduction of a critical key with an accidental TTL (which would make it an eviction candidate) or a cache key without a TTL (which would make it eviction-immune but memory-persistent).

### TTL Assignment Rules (application code)

| Key | TTL | Reason |
|---|---|---|
| `recall:global_epoch` | **None** | Permanent infrastructure; must survive indefinitely |
| `recall:canary_seeds` | **None** | Permanent infrastructure; Q103 seeds persist until explicitly refreshed |
| `session:{id}:pins` | **6 hours** | Session lifetime + 1-hour buffer per Q016 design |
| `recall:embedding_cache:{ver}:{hash}` | **172,800 s (48 hr)** | Q116 design; intended eviction candidate |
| ARQ job/queue keys | **ARQ-managed** | Do not override; ARQ sets its own TTLs |

### Memory Budget Reference

| Scenario | Redis footprint | Headroom vs 100 MB limit |
|---|---|---|
| Fresh deployment, no activity | ~2 MB (Redis process overhead) | 98 MB |
| Normal operation (50 sessions active) | ~7–10 MB | ~90 MB |
| Peak load (1,000 embedding cache entries) | ~13 MB | ~87 MB |
| Pathological (15,000 cache entries — eviction triggers) | ~100 MB | 0 MB — eviction begins |

In normal operation, Redis never reaches its 100 MB limit. The limit exists to prevent Docker OOM-kill, not to cap normal operation.

## Summary

**The key design insight**: `volatile-lru` is not merely a cache eviction policy — it is a correctness enforcement mechanism when critical keys have no TTL and cache keys have TTLs. The epoch counter, canary seeds, and any other permanent infrastructure keys in Recall's Redis keyspace are structurally immune to eviction as a consequence of their design (no TTL), without requiring application-level watchdogs, health checks, or reconstruction logic.

**The configuration is**: `--maxmemory 100mb --maxmemory-policy volatile-lru` added to the existing `--appendonly yes --appendfsync everysec --save ""` flags. This is a 2-flag addition to `docker-compose.yml` with no application code changes required.

**The memory math confirms** that Redis will never evict anything in Tim's normal operation (~10–12 MB footprint vs. 100 MB limit). The configuration is a safety net against pathological memory growth, not a tuning decision for normal operation.
