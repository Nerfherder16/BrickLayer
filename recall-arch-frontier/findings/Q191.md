# Finding: Q191 — Per-User Hopfield Isolation: Weight-Matrix vs Query-Level

**Question**: Should Recall 2.0's per-user memory be isolated at the weight-matrix level (separate Hopfield instance per user) or at the query level (shared index with user_id namespace on every query)?
**Agent**: quantitative-analyst
**Question Type**: [ADVERSARIAL]
**Verdict**: PROMISING
**Severity**: Medium
**Source field**: Adversarial pair analysis — first principles, no external search. Thread A: file-per-user LMDB. Thread B: shared LMDB with user_id namespace.

---

## Evidence

### Thread A Position: Separate Hopfield Weight Matrix Per User

**Core claim**: Each user gets their own CO_RETRIEVED weight store (separate LMDB file). No user_id in key schema. Reads: open user's file, query by `{mem_a}:{mem_b}`. Writes: route to user's file.

**What Thread A assumes**:
- Each user's retrieval patterns are meaningfully different (different topic domains, different workflows)
- Cross-user weight contamination would cause measurable retrieval degradation
- The per-user LMDB file is small enough to manage (≤50 users for homelab)
- File-per-user GDPR deletion is a feature: `rm recall-data/hopfield/{user_uuid}.lmdb`

**What Thread A exposes**:
1. **File descriptor pressure at SaaS scale**: 10K users → 10K open LMDB file handles. Linux default ulimit is 1024 fd per process. Manageable at homelab (≤50 users), requires reconfiguration at SaaS scale.
2. **Cold-start isolation problem**: New user has zero CO_RETRIEVED edges. No bootstrapped graph, no gravity signal until organic usage accumulates over 30+ sessions. Thread A provides no mechanism to pre-seed from similar users.
3. **Operational complexity grows with user count**: Backup, migration, health checks must enumerate per-user files. No SQL JOIN across users.
4. **Memory overhead**: 50 users × ~20MB LMDB per user (1.24M bootstrap edges) = 1GB LMDB data. Manageable but not O(1) with user count.

**What Thread A gets unambiguously right**:
- Mathematically impossible for user A's CO_RETRIEVED weights to contaminate user B's retrieval gravity — separate files, separate key spaces, separate processes of reading
- Simplest code: no user_id filtering in co_retrieval_gravity query; just open user's LMDB
- Matches intuition: one memory system per person

---

### Thread B Position: Shared Index with user_id Namespace

**Core claim**: Single LMDB file for all users. Key schema: `{user_id}:{mem_a}:{mem_b} → weight`. All gravity queries scan prefix `{user_id}:`. All flush worker writes include user_id.

**What Thread B assumes**:
- LMDB prefix scans are efficient for per-user queries (they are — cursor seeks to `{user_id}:` and scans forward)
- The shared LMDB remains performant at 10K users × 1.24M edges each = 12.4B potential edge entries (12.4B × 16 bytes = ~200GB — this does NOT scale)
- Cross-user weight interference is theoretical, not practical at homelab user counts

**What Thread B exposes**:
1. **The schema discipline requirement**: Correctness of Thread B depends entirely on user_id being in EVERY CO_RETRIEVED edge write. A single flush worker bug that omits user_id on an edge would silently contaminate all users' gravity scores for that memory pair. Thread A cannot have this bug (file routing is hard to get wrong).
2. **Hot tenant problem**: One active user generates write bursts to the shared LMDB. LMDB's single-writer model serializes all writes — a high-activity user blocks all other users' weight updates during the flush.
3. **Scan performance degrades with user count**: Prefix scan for `{user_id}:` at 10K users with 1.24M edges each is a cursor scan across up to 1.24M entries — same as Thread A per-user file. No advantage.
4. **GDPR deletion complexity**: Delete all edges for user X requires a cursor scan + delete for all `{user_id}:` prefix entries. Not O(1) — O(user_edges). Thread A is O(1): delete file.

**What Thread B gets right**:
- File descriptor usage is O(1) regardless of user count (one LMDB environment)
- Operational simplicity at moderate user count (one file to backup, one file to health-check)
- No per-user file management, no startup enumeration of user files

---

### The Critical Analysis: Both Positions Converge on the Same Invariant

**The Q188 finding framed this incorrectly.** Q188 concluded that CO_RETRIEVED weight matrices "cannot be namespace-filtered." This is only true if user_id is omitted from the edge schema. With user_id in every edge, both Thread A and Thread B are correct:

- Thread A: `file/{user_id}.lmdb` key `{mem_a}:{mem_b}` — user isolation via file routing
- Thread B: `shared.lmdb` key `{user_id}:{mem_a}:{mem_b}` — user isolation via key prefix

The mathematical isolation guarantee is identical. The difference is **physical organization, not semantic correctness**.

The real invariant: **user_id MUST appear in every CO_RETRIEVED edge, whether in the filename (Thread A) or the key (Thread B). Omitting user_id in either model produces cross-user contamination.**

### Synthesis: Scale-Dependent Correct Answer

| Tier | Users | Correct Choice | Reason |
|---|---|---|---|
| Solo homelab | 1 | Either (trivial) | No multi-user concern |
| Homelab team | 2-50 | Thread A (file-per-user) | O(n_files) manageable; simplest code; no schema discipline risk |
| Small SaaS | 50-10K | Thread B (shared + prefix) | File descriptor limits; O(1) operational overhead |
| Large SaaS | 10K+ | Thread B + Citus sharding | Neither LMDB model scales; need distributed KV |

**Key insight**: The transition from Thread A to Thread B happens at ~50-100 users, driven by file descriptor limits and operational overhead — not by correctness concerns (both are correct). A well-designed system uses Thread A by default (simpler, safer) with a configuration switch to Thread B for SaaS deployment.

### The Cold-Start Problem Is Symmetric

Both threads have the cold-start problem: a new user has zero organic CO_RETRIEVED edges. The bootstrap job (Q170) mitigates this for both — K-means bootstraps intra-cluster edges at w=0.1 for the new user's memory corpus, giving the graph structure from day one. Thread A bootstraps into the user's LMDB file; Thread B bootstraps into the shared LMDB with the user's key prefix. Cold-start behavior is identical.

## Ideas Extracted

### Idea: scale-tiered-co-retrieved-isolation-thread-a-to-b-migration
- **Novelty**: 0.75 — The scale-dependent transition (Thread A at ≤50 users, Thread B at SaaS scale) is a standard operational architecture pattern. The novelty is in the specific threshold calculation for Recall's LMDB deployment and the migration path between the two models.
- **Evidence**: 0.85 — Thread A correctness at small scale is analytically confirmed. Thread B performance at large scale is confirmed by LMDB's cursor-scan efficiency for prefix queries. The file descriptor limit calculation is exact: Linux default 1024 fd/process, Thread A requires 1 fd per user, effective limit is ~1000 concurrent users.
- **Feasibility**: 0.85 — Implement Thread A as default. The LMDB key schema in Thread A (`{mem_a}:{mem_b}`) and Thread B (`{user_id}:{mem_a}:{mem_b}`) differ by a prefix — migration is a key-rename pass with a single cursor scan. Config: `storage.isolation = "file_per_user" | "shared_lmdb"`. The flush worker and gravity query have two implementations selected by config.
- **Description**: Default to Thread A (file-per-user LMDB) for homelab deployment — simpler, safer, zero schema discipline risk. When Recall 2.0's user count exceeds 50, switch to Thread B (shared LMDB with user_id prefix) via a migration job that reads each user's LMDB file, rewrites all keys with `{user_id}:` prefix, and merges into a single shared LMDB. Config flag enables Thread B mode. The flush worker and gravity query use the same interface (`get_weight(user_id, mem_a, mem_b)`) implemented differently in each mode — Thread A opens user's file, Thread B reads shared file with prefix key.

### Idea: co-retrieved-schema-user-id-mandatory-invariant
- **Novelty**: 0.70 — This is a schema discipline finding derived from the adversarial analysis. The novelty is in formalizing the invariant: CO_RETRIEVED edges must include user_id regardless of isolation model, and the consequence of omitting it is cross-user contamination that is silent and hard to debug.
- **Evidence**: 0.95 — Analytically certain: if `MERGE (m1)-[r:CO_RETRIEVED]->(m2)` in the flush worker has no user_id property, all users share the same edge weight for (m1, m2). Thread A's file-per-user routing makes this impossible by construction (there's no cross-user edge); Thread B's namespace model makes this a schema discipline risk.
- **Feasibility**: 0.99 — Add a type-level guarantee in the Rust flush worker: `CoRetrievedEdge { user_id: UserId, mem_a: MemoryId, mem_b: MemoryId, weight_delta: f32 }` — the `user_id` field is non-optional. The compiler enforces the invariant. No runtime check needed.
- **Description**: In Recall 2.0's Rust flush worker, the `CoRetrievedEdge` struct must include `user_id: UserId` as a non-optional field. All flush worker code paths that write CO_RETRIEVED edges compile-time fail if user_id is absent. This makes the isolation invariant a type guarantee rather than a schema convention. For Thread A mode: `user_id` is used to select the LMDB file (not written to the key). For Thread B mode: `user_id` is prepended to the key. In both modes, the type system enforces that every edge write includes a user identity.

### Idea: weight-matrix-isolation-not-needed-for-correctness-only-for-simplicity
- **Novelty**: 0.70 — This finding corrects Q188's conclusion. Q188 stated weight matrix isolation requires file-per-user and cannot be solved by namespace filtering. The adversarial analysis shows this is wrong: with user_id in every edge, shared-LMDB namespace isolation is semantically correct. The simpler/safer model depends on scale, not on fundamental correctness concerns.
- **Evidence**: 0.90 — Analytically derived: `w(user_A, mem1, mem2)` and `w(user_B, mem1, mem2)` are independent values when user_id is in the key. Reading `{user_A}:{mem1}:{mem2}` cannot return `{user_B}:{mem1}:{mem2}` from a LMDB cursor. The isolation is complete.
- **Feasibility**: 0.99 — No action required beyond noting the Q188 correction. The architecture decision should be made on operational grounds (scale, fd limits, GDPR simplicity), not on a mistaken belief that namespace filtering is fundamentally insufficient.
- **Description**: Correct Q188's finding: namespace filtering (Thread B) provides complete isolation for CO_RETRIEVED weights when user_id is included in every edge key. The weight contamination risk is a schema bug (missing user_id), not an architectural limitation. Both Thread A (file-per-user) and Thread B (shared + prefix) are architecturally sound. Choose Thread A for homelab (≤50 users) for operational simplicity; Thread B for SaaS (>50 users) for fd efficiency.

## Analytical Summary

The adversarial analysis reveals that the Thread A vs Thread B choice is about **operational complexity and scale**, not about fundamental correctness. Both models provide correct CO_RETRIEVED weight isolation when user_id is included in every edge.

Thread A wins at homelab scale: simpler code (no user_id in key), file-level GDPR deletion, impossible to have schema discipline bugs. Thread B wins at SaaS scale: O(1) file descriptors, no per-user file management, one backup target.

**The invariant that both must satisfy**: user_id appears in every CO_RETRIEVED edge, either as file routing (Thread A) or as key prefix (Thread B). This invariant should be enforced at the type level in the Rust flush worker.

**Q188 correction**: The finding that "weight matrix isolation cannot be solved by namespace filtering" was overstated. Namespace filtering IS sufficient when the schema includes user_id. The real Q188 finding is: Recall 1.0 did NOT include user_id in CO_RETRIEVED edges (single-user design), and adding it is required for Recall 2.0. The isolation MODEL (file vs shared) is a secondary operational concern.
