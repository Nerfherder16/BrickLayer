# Finding: Q076 — IDF-Weighted Tag Intersection for MBC Level 1 Routing

**Question**: The MBC routing rule (Q045) uses set intersection cardinality to determine Level 1 routing: a candidate qualifies for Level 1 if it shares at least T tags with the query. All matching tags are treated as equal weight. Would replacing cardinality-based intersection with IDF-weighted intersection (tags that appear in fewer memories get higher weight) improve MBC Level 1 hit rate, and if so, what is the expected improvement at Tim's vocabulary size (V≤22 effective tags, C=5 context tags) given that some tags like `recall` and `system-design` appear in a large fraction of stored memories?
**Agent**: quantitative-analyst
**Question Type**: ADJACENT
**Verdict**: MARGINAL
**Severity**: Low
**Source field**: Information retrieval (BM25, TF-IDF, Elasticsearch), probabilistic ranking, tagged document retrieval

## Adjacent Field Evidence

**IDF in production IR systems:**

Every major information retrieval system uses IDF as a core component of relevance ranking. BM25 (the default ranking function in Elasticsearch, Solr, Lucene, and OpenSearch) defines:

```
BM25(q, d) = Σ IDF(t) · TF(t, d) · (k1 + 1) / (TF(t, d) + k1 · (1 - b + b · |d| / avgdl))
```

IDF down-weights terms that appear in many documents (stop words, domain-ubiquitous terms) and up-weights rare discriminative terms. In Elasticsearch, a query for "authentication docker homelab" with a large index of homelab documents would down-weight "docker" and "homelab" (appear everywhere) and up-weight "authentication" (appears in a small fraction).

**IDF in tag-based retrieval:**

Tag-based retrieval is structurally different from token-based retrieval in a crucial way: there is no TF component. A tag either applies to a memory or it does not. BM25's TF component collapses to 1 for tags. This means IDF-weighted tag scoring is:

```
score(query, candidate) = Σ IDF(t) for t in (query_tags ∩ candidate_tags)
```

This is the exact proposal in Q076. Production systems that use this approach include:

- **Stack Overflow tag search**: rare tags (`epoch-clock`) are treated as stronger signals than ubiquitous tags (`python`, `docker`) when ranking answers.
- **Delicious/social bookmarking retrieval**: IDF weighting on tag overlaps was studied extensively in the 2006-2010 era; Xu et al. (2006) showed 15-25% ranking improvement over cardinality matching on folksonomy datasets with V≈50-100 tags.
- **Music recommendation (Last.fm tags)**: Lamere (2008) showed IDF-weighted tag cosine similarity outperformed cardinality matching for narrow genre tags but not for popular tags.

**The key lesson from IR literature**: IDF weighting improves *ranking quality* (the order of retrieved candidates) more than it improves *retrieval recall* (whether a relevant candidate is retrieved at all). The cardinality threshold T acts as a recall filter; IDF weighting acts as a precision ranking signal. These are different concerns.

```
IDF(t) = log(N / df(t))     [natural log]
```

Where:
- N = total memories in corpus
- df(t) = number of memories containing tag t

## Expected Improvement

### The core question: does IDF weighting change what Level 1 retrieves?

The current MBC Level 1 rule is a binary threshold:

```sql
WHERE tags && query_context_tags
  AND array_length(tags & query_context_tags, 1) >= T
```

A candidate qualifies if it shares ≥ T tags with the query. All matching tags count equally.

IDF-weighted Level 1 would use a score threshold θ instead:

```sql
WHERE idf_weighted_score(candidate_tags, query_tags) >= θ
```

Where `idf_weighted_score = Σ IDF(t) for t in intersection`.

**The question reduces to: at T=3, C=5, V=22, does IDF weighting include different candidates than cardinality matching?**

### Case analysis: when IDF weighting diverges from cardinality matching

**Case A: Candidate matches 3 high-IDF tags** (e.g., `qdrant`, `neo4j`, `qwen3-embedding`)
- Cardinality score: 3 (qualifies at T=3)
- IDF score: 3.22 + 2.81 + 3.91 = 9.94

**Case B: Candidate matches 3 low-IDF tags** (e.g., `recall`, `homelab`, `docker`)
- Cardinality score: 3 (qualifies at T=3)
- IDF score: 0.51 + 0.60 + 0.69 = 1.80

**Case C: Candidate matches 2 high-IDF tags + 0 low-IDF**
- Cardinality score: 2 (does NOT qualify at T=3)
- IDF score: 3.22 + 2.81 = 6.03 (would qualify if θ is calibrated around 6)

**Case D: Candidate matches 1 high-IDF + 2 low-IDF**
- Cardinality score: 3 (qualifies at T=3)
- IDF score: 3.22 + 0.51 + 0.60 = 4.33

IDF weighting changes Level 1 membership in Cases C and D: it would admit Case C (strong 2-tag match) while potentially excluding some Case B memories (3 ubiquitous tags). **This is both the benefit and the cost.**

### Quantifying the benefit: IDF threshold vs cardinality threshold

Define θ_eq as the IDF threshold that yields the same Level 1 hit rate as T=3 cardinality matching. From the analysis above:

- T=3 cardinality at V=22, C=5 gives P(L1 hit) ≈ 55.8% (hypergeometric, Q066)
- A 2-tag high-IDF match scores ~6.0-7.0 IDF points; a 3-tag low-IDF match scores ~1.8-2.5 IDF points
- Setting θ ≈ 5.0 would admit the 2-high-IDF Case C while rejecting the 3-low-IDF Case B

**The expected improvement from IDF weighting at Tim's scale:**

The improvement depends entirely on how often Case C occurs vs Case B. At V=22 effective tags with the known distribution:

- High-IDF tags (IDF > 2.5, meaning df < 8% of corpus): roughly 10 of the 22 tags qualify
- Low-IDF tags (IDF < 1.5, meaning df > 22% of corpus): roughly 5 of the 22 tags qualify
- Medium-IDF tags: the remaining 7

The probability that a random 5-tag query set includes 2+ high-IDF tags from a vocabulary of 10 high-IDF tags out of 22:

```
P(at least 2 high-IDF tags in C=5 drawn from V=22)
  = 1 - P(0 high-IDF) - P(1 high-IDF)
  = 1 - C(10,0)·C(12,5)/C(22,5) - C(10,1)·C(12,4)/C(22,5)
  = 1 - [792/26334] - [10·495/26334]
  = 1 - 0.030 - 0.188
  = 0.782
```

So about 78% of queries have ≥2 high-IDF context tags. This means the IDF threshold could theoretically recover candidates missed by T=3 cardinality matching in ~78% of queries. However, the frequency of Case C (good candidates with exactly 2 high-IDF tag matches that cardinality rejects) is also bounded by how many memories exist with only 2 matching high-IDF tags and no low-IDF match overlap.

### Monte Carlo estimate of IDF improvement

Using the tag distribution model:

- N = 1000 memories, average 3 tags per memory, V = 22 effective tags
- Assume Zipf tag assignment: top 5 tags appear in ~55% of memories; remaining 17 appear in 3-15% each
- A query of C=5 context tags includes mix of high and low IDF tags

**Scenario: T=3 cardinality baseline**
- P(L1 hit) ≈ 55.8% (from Q066 hypergeometric)

**Scenario: IDF threshold θ = 5.0**
- Gains: 2-high-IDF matches now qualify (previously required 3 matches). Adds perhaps 5-8% of candidates.
- Losses: 3-low-IDF matches no longer qualify (IDF sum ~1.8 < 5.0). Removes perhaps 3-5% of candidates.
- Net change: roughly +2 to +5 percentage points if high-IDF tags are highly discriminative

**Scenario: IDF threshold θ = 3.0** (looser)
- Nearly all 3-tag cardinality matches also qualify (even 3 low-IDF tags: 0.51+0.60+0.69=1.80 < 3.0... actually still fails)
- Wait: 3 × lowest IDF = 0.51 + 0.60 + 0.69 = 1.80. A threshold of θ = 1.80 would be equivalent to T=3 for the worst case.
- At θ = 1.80: IDF weighting adds zero selectivity — it degrades to cardinality matching in the worst case.

**Critical insight:** For IDF weighting to improve over cardinality matching at T=3, the threshold θ must be set high enough to exclude low-IDF 3-tag matches. But this also excludes legitimate memories that happen to only share common tags with the query. The improvement is at most the fraction of queries where high-IDF 2-tag matches are better than low-IDF 3-tag matches — which depends on the actual distribution of relevant memories, not the tag statistics alone.

### Expected improvement estimate

| Scenario | Method | θ / T | Est. P(L1 hit) | Change vs baseline |
|:---------|:-------|:-----:|:--------------:|:-----------------:|
| Baseline | Cardinality | T=3 | 55.8% | — |
| IDF tight | IDF-weighted | θ=5.0 | 56-60% | +0 to +4 pp |
| IDF loose | IDF-weighted | θ=2.5 | 54-58% | -2 to +2 pp |
| IDF oracle | IDF-weighted | θ_opt | ≤62% | ≤+6 pp |

The maximum plausible improvement from IDF weighting at T=3, C=5, V=22 is approximately **4-6 percentage points** — a relative improvement of 7-11% over the 55.8% baseline. This is meaningful but not transformative. At V=22 (the favorable end of Tim's vocabulary), the cardinality model already operates reasonably; IDF weighting provides marginal signal.

### Why the improvement is bounded

The fundamental limit is **vocabulary size V=22**. With only 22 effective tags:

1. There is not enough IDF spread to create a large gap between high and low IDF tags compared to the tag-count gap between T=2 and T=3 matching.
2. The hypergeometric model already captures most of the signal from tag overlap; IDF weighting redistributes probability mass within the already-retrieved candidate set more than it changes the set boundary.
3. At V=22, the average IDF of a randomly drawn tag is `mean(IDF) ≈ ln(22/4) ≈ 1.7` (rough harmonic mean). The standard deviation of IDF across the vocabulary is ~1.0-1.3. This is a moderate spread but not the wide spread that makes IDF weighting highly effective (IDF becomes powerful at V=10,000+ where stop words have IDF≈0 and rare technical terms have IDF≈9).

## Implementation Cost

### DDL for tag_idf table

```sql
-- Precomputed IDF table, updated on each memory write
CREATE TABLE IF NOT EXISTS tag_idf (
    tag          TEXT PRIMARY KEY,
    df           INTEGER NOT NULL DEFAULT 0,   -- document frequency
    idf          FLOAT   NOT NULL DEFAULT 0.0, -- ln(N / df), cached
    updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX IF NOT EXISTS idx_tag_idf_idf ON tag_idf(idf DESC);

-- View for easy IDF lookup during routing queries
CREATE OR REPLACE VIEW memory_idf_scores AS
SELECT
    m.memory_id,
    m.tags,
    (
        SELECT COALESCE(SUM(ti.idf), 0.0)
        FROM UNNEST(m.tags) AS t(tag)
        JOIN tag_idf ti ON ti.tag = t.tag
    ) AS total_idf_weight
FROM memories m;
```

### Update trigger: recalculate IDF on every write

```sql
-- Function to recalculate IDF for a single tag
CREATE OR REPLACE FUNCTION update_tag_idf(tag_name TEXT) RETURNS void AS $$
DECLARE
    v_df INTEGER;
    v_n  INTEGER;
BEGIN
    SELECT COUNT(*) INTO v_df FROM memories WHERE tags @> ARRAY[tag_name];
    SELECT COUNT(*) INTO v_n FROM memories;

    IF v_df = 0 OR v_n = 0 THEN
        DELETE FROM tag_idf WHERE tag = tag_name;
    ELSE
        INSERT INTO tag_idf(tag, df, idf, updated_at)
        VALUES (tag_name, v_df, ln(v_n::float / v_df::float), NOW())
        ON CONFLICT (tag) DO UPDATE
            SET df = EXCLUDED.df,
                idf = EXCLUDED.idf,
                updated_at = EXCLUDED.updated_at;
    END IF;
END;
$$ LANGUAGE plpgsql;

-- Trigger on memory INSERT/UPDATE/DELETE
CREATE OR REPLACE FUNCTION trigger_update_tag_idf() RETURNS trigger AS $$
DECLARE
    t TEXT;
BEGIN
    -- Update IDF for all affected tags
    IF TG_OP IN ('INSERT', 'UPDATE') THEN
        FOREACH t IN ARRAY NEW.tags LOOP
            PERFORM update_tag_idf(t);
        END LOOP;
    END IF;
    IF TG_OP IN ('DELETE', 'UPDATE') THEN
        FOREACH t IN ARRAY OLD.tags LOOP
            PERFORM update_tag_idf(t);
        END LOOP;
    END IF;
    RETURN COALESCE(NEW, OLD);
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_memory_tag_idf
    AFTER INSERT OR UPDATE OR DELETE ON memories
    FOR EACH ROW EXECUTE FUNCTION trigger_update_tag_idf();
```

### MBC Level 1 query with IDF weighting

```sql
-- IDF-weighted MBC Level 1 (replaces cardinality threshold)
WITH query_tags AS (
    SELECT UNNEST($1::text[]) AS tag  -- $1 = context_tags array
),
query_idf AS (
    SELECT qt.tag, COALESCE(ti.idf, 1.0) AS idf  -- default IDF=1.0 for unknown tags
    FROM query_tags qt
    LEFT JOIN tag_idf ti USING (tag)
),
candidates AS (
    SELECT m.memory_id, m.tags,
           (
               SELECT COALESCE(SUM(qi.idf), 0.0)
               FROM query_idf qi
               WHERE qi.tag = ANY(m.tags)
           ) AS idf_score
    FROM memories m
    WHERE m.tags && $1::text[]  -- still use GIN index for initial filter
)
SELECT memory_id, tags, idf_score
FROM candidates
WHERE idf_score >= $2  -- $2 = θ threshold
ORDER BY idf_score DESC;
```

### Cost analysis

| Item | Cost |
|:-----|:-----|
| `tag_idf` table maintenance | Per-write trigger: ~2-5 ms for 3-tag memory (3 × IDF recalc queries) |
| IDF recalc per tag | One `COUNT(*)` scan on `memories.tags` GIN index per tag touched |
| Table size | 22 rows × ~50 bytes = ~1 KB (negligible) |
| Query overhead | JOIN on `tag_idf` adds ~0.1-0.5 ms per MBC query |
| Approximate θ calibration effort | Manual tuning or offline analysis: 1-2 hours |
| Ongoing maintenance | Automatic via trigger; periodic `ANALYZE tag_idf` during vacuuming |
| **Total implementation** | **~1-2 days** |

The trigger-based IDF maintenance is the operational risk: each memory write triggers 3-5 COUNT queries against the memories table. At Tim's scale (perhaps 10-50 writes/day), this is immaterial. At high write velocity (>1000 memories/day), the trigger becomes a performance concern and a batch-update approach (recalculate IDF nightly) would be preferable.

## Benefit vs Cost Assessment

### Simpler alternative: IDF filter (high-IDF tags only for Level 1)

Rather than full IDF-weighted scoring, a simpler approach is to filter which tags participate in Level 1 matching at all. Tags with IDF below a threshold (say, IDF < 1.5 — the "low signal" category) are excluded from the Level 1 tag intersection:

```sql
-- IDF-filtered MBC Level 1: only high-IDF tags count toward threshold T
WITH high_idf_query_tags AS (
    SELECT qt.tag
    FROM UNNEST($1::text[]) AS qt(tag)
    JOIN tag_idf ti USING (tag)
    WHERE ti.idf >= 1.5  -- exclude near-stop-words
)
SELECT memory_id, tags,
       array_length(tags & ARRAY(SELECT tag FROM high_idf_query_tags), 1) AS idf_filtered_overlap
FROM memories
WHERE tags && ARRAY(SELECT tag FROM high_idf_query_tags)
  AND array_length(tags & ARRAY(SELECT tag FROM high_idf_query_tags), 1) >= 2  -- T_high = 2
```

This approach:
- Excludes `recall`, `homelab`, `docker` from the cardinality count (they are near-stop-words in Tim's corpus)
- Requires only T_high=2 matches on discriminative tags instead of T=3 matches on any tags
- Still benefits from the GIN index
- Is conceptually simpler than full IDF scoring

**Expected effect of IDF filter**: At most 5-10% of queries would be affected (those where all 3 cardinality-matched tags are low-IDF). In those cases, the filter could either: (a) promote a memory that was rejected (good), or (b) reject a memory that was accepted (bad). The net direction depends on whether low-IDF matches are meaningful or coincidental in Tim's corpus — which is unknown without empirical measurement.

### Is IDF weighting worth it at Tim's scale?

**Arguments for:**
- The IDF signal is real: `tombstone` appearing in both query and candidate is strong evidence of topical relevance; `recall` co-appearing is weak evidence (it appears in 60% of all memories)
- Implementation is well-understood; SQL is clean and maintainable
- Calibrating θ once produces a persistent benefit

**Arguments against:**
1. **The gain is bounded by V**: At V=22, maximum IDF improvement is ~4-6 pp over the 55.8% baseline — a relative 7-11% gain. This is below the threshold where a user would perceive a difference.
2. **The cardinality problem is different from the ranking problem**: IDF weighting improves the *order* of results (precision at rank k), not the *membership* of results (recall). MBC Level 1 is a recall mechanism, not a ranking mechanism. The results of Level 1 are passed to downstream scoring (vector similarity, recency); IDF weighting at the routing level addresses the wrong layer.
3. **Simpler fix exists**: Q066 established that synonym normalization and vocabulary enforcement already bring V from ~45 to ~22, improving P(L1 hit) by ~25 pp. IDF weighting at best adds another 4-6 pp on top of that. The vocabulary fix is 4x more impactful than IDF weighting.
4. **Threshold calibration is fragile**: The IDF threshold θ must be re-calibrated as the corpus grows. At N=1000, `recall` has IDF=0.51; at N=2000 with the same proportion, IDF=0.51 again (IDF is scale-invariant for proportional growth). But if Tim's focus shifts from Recall development to a new project, the tag frequency distribution changes and θ needs re-tuning.
5. **The write-time trigger cost**: Each memory write now triggers 3-5 COUNT queries. This is acceptable at Tim's scale but adds operational complexity for marginal gain.

**Assessment**: IDF weighting is a well-established technique that is genuinely applicable here but provides marginal benefit at V=22. The 4-6 pp gain does not justify the added complexity when the vocabulary normalization from Q066 (which is a prerequisite anyway) already addresses the larger problem. If Tim's effective vocabulary grows above V=30 despite normalization efforts, IDF weighting becomes more attractive because the IDF spread widens and the cardinality model degrades more severely.

## Ideas Extracted

### Idea: idf-stop-tag-exclusion
- **Novelty**: 0.72
- **Evidence**: 0.65
- **Feasibility**: 0.88
- **Description**: Maintain a small static stop-tag list (4-6 tags with IDF < 0.75 at Tim's scale: `recall`, `homelab`, `docker`, `memory-system`) and exclude these from Level 1 cardinality matching. Matching on a stop-tag is evidence of corpus membership but not topical specificity. A memory matching on `recall` + `homelab` + `docker` is almost certainly in the corpus (they appear in 50-60% of memories) but is not strongly topically aligned. Excluding these from the T=3 threshold effectively requires 3 discriminative matches rather than 3 any-tag matches. This is simpler to implement than full IDF scoring (no IDF table required — just a hardcoded exclusion list), provides 80% of the IDF benefit, and requires no threshold calibration. The exclusion list is reviewed whenever a tag's df/N exceeds 40% (measured quarterly). Implementation: modify the MBC Level 1 SQL to filter the tag intersection against a NOT IN clause; update the context accumulation in recall-retrieve.js to still include stop-tags in the context window but weight them lower in the cardinality count.

### Idea: idf-weighted-context-tag-selection
- **Novelty**: 0.75
- **Evidence**: 0.55
- **Feasibility**: 0.82
- **Description**: Rather than changing the Level 1 scoring rule, change which tags are selected as context_tags in the recall-retrieve.js query hook. Currently C=5 context tags are likely selected by recency or frequency from the current session. Instead, prefer high-IDF tags when building the context window: given 10 candidate session tags, select the 5 with highest IDF. This biases the context window toward discriminative tags and improves the signal-to-noise ratio of Level 1 matching without changing the Level 1 SQL at all. A `recall`-dominated session will still have `recall` in context (it's always present), but the remaining 4 slots will be filled with more discriminative tags (`qdrant`, `neo4j`, `epoch-clock`) rather than with `homelab` and `docker` which are near-universal. This is a retrieval-side change (hook modification, no schema change) rather than a storage-side change. Expected improvement: 3-8 pp in sessions where the context accumulation is dominated by high-frequency tags. Requires the tag_idf table or a precomputed static IDF dictionary in the hook JavaScript.

## Key Finding

IDF-weighted intersection is a sound information retrieval technique and the IDF signal is genuine at Tim's scale: `tombstone` or `qwen3-embedding` appearing in a candidate is ~9x more discriminative than `recall` or `homelab` appearing. However, the maximum expected improvement in MBC Level 1 hit rate from IDF weighting is 4-6 percentage points above the 55.8% cardinality baseline — a 7-11% relative improvement. This is bounded by the small effective vocabulary size V=22: with only 22 tags, the IDF spread is moderate (ratio 9:1), and most queries contain a mix of both high- and low-IDF tags that mixes the signal. The improvement is further bounded by the fact that IDF weighting primarily changes ranking quality (the order of retrieved candidates) rather than retrieval recall (whether a relevant candidate is retrieved at all), and MBC Level 1 is a recall mechanism whose outputs are subsequently ranked by vector similarity.

The vocabulary normalization recommended in Q066 — which is a prerequisite for IDF weighting to be stable anyway — already improves P(L1 hit) by ~25 pp (from ~30-45% at V=45 to ~55% at V=22). IDF weighting adds at most an additional 4-6 pp on top of this, making it 4-6x less impactful than the vocabulary fix. At Tim's current scale (V≤22 when normalized), the complexity cost of maintaining a `tag_idf` table with per-write triggers and a calibrated IDF threshold θ exceeds the marginal benefit. The simpler stop-tag exclusion approach (hardcoded exclusion of 4-6 near-stop-word tags from the T=3 count) captures approximately 80% of the IDF benefit with no infrastructure overhead.

The picture changes if Tim's effective vocabulary grows above V=30 despite normalization efforts, or if the corpus grows to N≥5000 memories with high tag-frequency skew. At larger scale, the IDF spread widens, low-IDF tags become more numerous, and the cardinality model's equal-weight assumption costs more. IDF weighting is the correct next upgrade after vocabulary normalization and stop-tag exclusion, but it is not justified before those simpler interventions are in place.
