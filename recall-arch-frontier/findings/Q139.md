# Q139 — ef_search Adequacy at 22K Vectors

**Question**: What is the theoretical maximum corpus size at which ef_search=64 in Qdrant's HNSW configuration maintains ANN recall@1 ≥ 0.95 — and has the 22,349-vector corpus already exceeded this ceiling, requiring an ef_search increase to preserve retrieval precision?
**Question Type**: PHYSICS
**Verdict**: INCREMENTAL
**Severity**: Low-Medium
**Source field**: HNSW approximation theory (Malkov & Yashunin 2018), ann-benchmarks empirical recall curves, information retrieval theory

---

## Physics Ceiling: HNSW Recall@1 vs. ef_search at N=22,349

### Operation definition

- **Input**: Query vector (1536-dim float32, qwen3-embedding:0.6b); corpus of N=22,349 stored vectors in Qdrant's HNSW index with M=16, ef_construction=100
- **Output**: Top-k approximate nearest neighbors by cosine similarity
- **Guarantees required**: R@1 > 0.95 (the true nearest neighbor is returned at rank 1 in at least 95% of queries)

---

### Theoretical Minimum

| Resource | Minimum | Calculation |
|---|---|---|
| Exact NN comparisons for brute-force | N × d = 22,349 × 1,536 = 34.3M FLOPs | Every stored vector must be compared to query |
| HNSW comparisons at ef=64 | ~64 × log(N) × d ≈ 64 × 10.0 × 1,536 = 984K FLOPs | Greedy beam search visits O(ef × log N) nodes |
| HNSW approximation ratio | 34.3M / 984K = 34.8× fewer operations | The cost of approximation error |
| Minimum possible miss probability | 0 (exact search) | Only achievable at ef → ∞ or brute-force |

HNSW's fundamental tradeoff: each ef_search step saved from brute-force introduces a nonzero probability of the true NN falling outside the explored subgraph. The miss probability is bounded below zero only by exact search.

---

### HNSW Recall Degradation Model

The miss probability (P_miss = 1 − R@1) for HNSW scales with corpus size and ef_search according to the greedy search dynamics derived in Malkov & Yashunin (2018):

**Proportionality relationship** (calibrated from ann-benchmarks empirical data on high-dim semantic embeddings, M=16):

```
P_miss(ef, N) ∝ log(N) / ef
```

The intuition: HNSW's hierarchical structure creates an effective "entry distance" to the true NN that grows as log(N) (the number of graph layers grows as log(N)), while ef_search is the exploration budget that counters this. When log(N)/ef grows, so does the probability of the greedy search terminating before finding the true NN.

**Calibration from Q118 anchor points:**

Q118 established empirically (from ann-benchmarks style data, M=16, semantic embeddings):
- N=10,000, ef=64: R@1 ≈ 0.95–0.97 (taking midpoint: R@1 = 0.96, P_miss = 0.04)
- N=5,000, ef=64: R@1 ≈ 0.98–0.99 (P_miss ≈ 0.015)

Using the calibrated proportionality to extrapolate to N=22,349:

```
P_miss(ef=64, N=22,349) = P_miss(ef=64, N=10,000) × [log(22,349) / log(10,000)]
                         = 0.04 × [10.015 / 9.210]
                         = 0.04 × 1.087
                         = 0.0435

R@1(ef=64, N=22,349) ≈ 1 − 0.0435 ≈ 0.957
```

**Sensitivity to calibration anchor:**

| Q118 anchor assumed | P_miss at N=10K | Extrapolated R@1 at N=22,349 |
|---|---|---|
| Optimistic (R@1=0.97) | 0.03 | 0.967 |
| Midpoint (R@1=0.96) | 0.04 | **0.957** (central estimate) |
| Pessimistic (R@1=0.95) | 0.05 | 0.946 |

**Central conclusion**: R@1 at N=22,349 with ef=64 is approximately **0.946–0.967**, with central estimate **0.957**.

The 0.95 threshold is within measurement uncertainty. The system is at the edge of the envelope, not clearly inside or outside it. Whether ef=64 is technically adequate depends on which end of Q118's empirically-derived range is correct for Tim's specific data distribution.

### Gap Analysis

**Physics gaps** (unavoidable):

- **ANN fundamental tradeoff**: Any sub-linear index trades recall for speed. At N=22,349 with 34.3M FLOPs for exact search, some miss rate is the price of the 500ms SLA. This cannot be engineered away without accepting full scan latency (~80–150ms for 34.3M float32 ops on modern hardware — actually feasible given the SLA, but not the current architectural choice).

- **Graph navigability at high dimension**: In 1536-dim space, the "effective" intrinsic dimensionality of semantic embeddings is ~50–200. The HNSW navigability degrades gracefully with intrinsic (not ambient) dimension, which is why performance is better than naive 1536-dim would suggest. This is a favorable physics property for Tim's use case.

**Architecture gaps** (design choices):

- **ef_search=64 chosen at N≈360**: The ef_search value was calibrated when the corpus was ~360 vectors (Q100 finding). The corpus has grown 62× to 22,349 vectors without a corresponding ef_search adjustment. The value was appropriate then; its adequacy now depends on the calibration range.

- **Static ef_search**: ef_search is a fixed Qdrant collection parameter. It does not adapt as the corpus grows. A corpus that grows from 360 to 22,349 vectors (and continues growing) warrants periodic recalibration. The current architecture has no mechanism for this.

**Implementation gaps** (code quality):

- **No ef_search recalibration trigger**: There is no hook, health signal, or monitoring that detects when corpus growth warrants an ef_search review. The corpus has grown 62× with no ef_search review mechanism.

### Required ef_search to Restore R@1 > 0.95 with Margin

**To match the performance of ef=64 at N=10,000 (i.e., restore to the Q100 design intent):**

```
ef_required = ef_current × [log(N_actual) / log(N_calibrated)]
            = 64 × [log(22,349) / log(10,000)]
            = 64 × [10.015 / 9.210]
            = 64 × 1.087
            = 69.6  →  round to 70
```

ef_search=70 technically restores equivalence with the Q100 design intent. However, since the design intent itself was expressed as "safe until N≈5,000–10,000" (Q118), using N=10,000 as the calibration point already implies some margin was included.

**To achieve R@1 ≥ 0.97 (comfortable margin above 0.95, allowing for future growth):**

Using P_miss_target = 0.03 (R@1 = 0.97):
```
ef_required = ef_current × [P_miss(ef=64, N=22,349) / P_miss_target]
            = 64 × [0.0435 / 0.030]
            = 64 × 1.45
            = 92.8  →  round to ef_search=96 or 100
```

**To maintain R@1 ≥ 0.95 with safety margin through 2× corpus growth (N≈45,000):**

```
P_miss(ef=64, N=45,000) = 0.04 × [log(45,000) / log(10,000)]
                         = 0.04 × [10.714 / 9.210]
                         = 0.04 × 1.163
                         = 0.0465  → R@1 ≈ 0.954 (marginal at 45K)

ef_required for R@1=0.97 at N=45,000:
  = 64 × [0.0465 / 0.030] = 64 × 1.55 = 99.2  →  ef_search=100
```

**Recommendation table:**

| Scenario | Recommended ef_search | Resulting R@1 (est.) | Latency overhead |
|---|---|---|---|
| Minimum fix for N=22,349 | 70 | ~0.960 | +9% vs current |
| Comfortable margin, N=22,349 | 96 | ~0.970 | +50% |
| Future-proof through N=45,000 | 100 | ~0.970 at 45K | +56% |
| Conservative with 2× safety buffer | 128 | ~0.978 at 22K; ~0.974 at 45K | +100% |

**ef_search=128 is the practical recommendation**: it is a standard Qdrant configuration value, doubles the exploration budget (which halves P_miss by the proportionality model), and adds an estimated 10–20ms to ANN query latency at N=22K — well within the 500ms SLA. It also provides headroom through N≈80,000–100,000 before another recalibration is needed.

## Qualifying Ideas

### ef-search-auto-scaling
- (N=0.72, E=0.85, F=0.92)
- Add a corpus-size milestone trigger to the Recall health system that fires when N crosses predefined thresholds (10K, 25K, 50K, 100K) and checks whether the current ef_search still meets the R@1 ≥ 0.95 target using the calibrated log-N degradation model. At each milestone, if the predicted P_miss exceeds 0.05, emit a health warning and recommend the next ef_search value. The model (ef_required = ef_current × log(N_actual)/log(N_calibrated)) is closed-form and adds <1ms to health check latency. No production systems implement this pattern for personal memory stores; it is novel in that context.

### ef-search-vs-latency-pareto-dashboard
- (N=0.68, E=0.78, F=0.88)
- Expose a `/diagnostics/hnsw-pareto` endpoint that returns the R@1 vs. latency tradeoff curve for the current corpus size across ef_search values (32, 64, 96, 128, 192, 256). Computed analytically using the calibrated model — no actual benchmark queries needed. Allows Tim to select the operating point interactively when the corpus grows. The endpoint outputs: `{ef_search, predicted_r1, latency_overhead_ms}` for each candidate value, and marks the current setting and the recommended minimum.

## Recommendations

1. **Update ef_search to 128 immediately**: This is a single Qdrant collection update call (`PUT /collections/recall` with `hnsw_config: {ef: 128}`). No re-indexing, no data migration. Estimated R@1 improvement: +0.021 (from 0.957 to 0.978 at N=22,349). This future-proofs through N≈80,000.

2. **Add a corpus-growth ef_search check to `/health/recall-quality`**: When the health check runs and the current corpus size has crossed a new decade milestone (10K, 25K, 50K, 100K), compute the predicted P_miss and emit a `hnsw_ef_warning` if it exceeds 0.05. This is a ~10-line addition to the health endpoint.

3. **Confirm ef_search is a query-time parameter in this Qdrant version**: In Qdrant, `hnsw_config.ef` can be set at collection creation and overridden per-query via the search request's `params.hnsw_ef`. The collection-level setting is the default; the per-query override is available. Verify which path the Recall API currently uses — if it uses per-query ef, the fix is in the FastAPI search handler, not the collection config.

4. **Note**: The ef_search calibration question is secondary to the canary displacement issue (Q127). The 7/10 HIGH RISK canary seeds are a retrieval quality failure that ef_search tuning does not address — those failures are caused by semantic corpus crowding (organic memory density), not HNSW approximation error. ef_search adjustment improves approximate nearest-neighbor accuracy; sentinel-token hardening (Q127) addresses the semantic displacement problem. Both are necessary; neither substitutes for the other.
