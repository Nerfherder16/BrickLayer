## Research Question

Design a retrieval system for heterogeneous memory content — code snippets, image descriptions, structured JSON, free text — using only physical requirements. Forbidden: multi-modal, cross-modal, embedding, vector, fusion, late fusion, early fusion, representation learning. Start from: what physical property lets you compare things that are structurally different?

---

## The Physical Property

**The key physical property that any two content items share regardless of content type:**

> **Information overlap** — how much of the information in document X is also present in document Y?

This is formally grounded in **Kolmogorov complexity** and **Shannon mutual information**. Two items are "similar" to the extent they contain the same information. The operative physical question: can compressing X and Y together produce a smaller result than compressing them separately?

**Normalized Compression Distance (NCD)**:
```
NCD(x,y) = [C(xy) - min(C(x), C(y))] / max(C(x), C(y))
```

Where C(x) = compressed length of x using any compressor (gzip, zstd, lzma).
- NCD = 0: x and y are identical
- NCD = 1: x and y share no information
- NCD = 0.3: 30% of the content is non-shared

**Why this is content-type agnostic:** gzip does not know or care whether it is compressing Python code, a JSON object, or English prose. It operates on byte sequences. The shared information in "port 8200" appears the same whether the document is `{"port": 8200}`, `app.listen(8200)`, or "the service runs on port 8200."

NCD requires compressing query+each_document at query time: O(N) operations, each taking ~1ms for 1KB documents. At N=22K: 22 seconds per query. Unusable.

## The Indexable Mechanism: Character-Level N-gram Jaccard + MinHash LSH

**Physical grounding**: n-gram Jaccard similarity is an approximation to NCD. Shared n-grams = shared information fragments. Character-level n-grams (not word-level) are the natural primitive for heterogeneous content because:

- Words are a linguistic construct; characters are physical
- JSON keys, function names, identifiers, and prose words are all character sequences
- A character n-gram of length k=4 from `"connection refused"` and from `error_message = "connection refused"` will share most n-grams

**The algorithm:**

```
Index time (per memory):
  1. Content → character sequence (already true: all Recall memories are text strings)
  2. Extract all character n-grams of length k=4 (shingle set S)
     "port 8200" → {"port", "ort ", "rt 8", "t 82", " 820", "8200"}
  3. MinHash: compute b×r hash functions on S → b-band signature
  4. Store signature in LSH bucket structure

Query time:
  1. Query → character n-grams → MinHash signature
  2. LSH bucket lookup: O(1) approximate nearest neighbors
  3. Verify: compute exact Jaccard on top candidates
  4. Return ranked by Jaccard similarity
```

**The physical justification for k=4**: k=4 character n-grams are the empirical optimum for language-agnostic text similarity (used in spam detection, plagiarism detection, biological sequence comparison). At k=4, the shingle set is large enough to be discriminative but small enough to be sensitive to short shared patterns like "8200" or "null".

## Why Heterogeneity Dissolves

The TABOO question assumes heterogeneous content requires special handling. The physical analysis reveals this assumption is wrong:

**Recall 2.0 stores all memories as text strings, regardless of origin.**

| Content type | Stored form | Physical retrieval |
|---|---|---|
| Code snippet | `"def handle_error(e): log.error(str(e))"` | Character n-grams |
| Image description | `"Screenshot showing error: connection refused on port 8200"` | Character n-grams |
| JSON payload | `'{"host": "100.70.195.84", "port": 8200}'` | Character n-grams |
| Free text | `"The Recall service is at 100.70.195.84:8200"` | Character n-grams |

A query for "port 8200" has high Jaccard similarity to ALL FOUR of these, despite their different structures. The character sequence "8200" appears in each. The physical retrieval mechanism needs no special handling for content type because **the memory system's universal representation is the text string**.

The heterogeneity is at the source level. It is resolved at storage time by text serialization. Physical retrieval operates on the serialized form.

## What "Relevant" Means Physically

The TABOO question asks: what does "relevant" mean when the query is text and the memory is code?

**Physical definition of relevance**: a memory is relevant to a query if knowing the memory would reduce the query's uncertainty. Operationally: the memory and query share information.

Information sharing is approximated by character n-gram overlap:
- High overlap → high probability the memory contains something the query asks about
- The mechanism doesn't need to "understand" either document
- It doesn't need to know that `connection_refused` (code) and "connection refused" (prose) are related — the character n-grams `"conn"`, `"onne"`, `"nnec"`, `"nect"`, `"ecti"`, `"ctio"`, `"tion"`, `"on r"`, `"n re"` etc. appear in both

## Physical Retrieval vs. Embedding-Based Retrieval

| Query type | Physical (n-gram Jaccard) | Semantic (embedding) |
|---|---|---|
| Exact identifiers (port 8200, IP address) | **Excellent** — direct character match | Poor — near-identical embeddings |
| Code pattern search (`handle_error`) | **Excellent** — exact token match | Moderate |
| JSON key lookup (`"host"`) | **Excellent** | Poor |
| Semantic paraphrase ("the service went down" vs "connection refused") | Poor — few shared n-grams | **Excellent** |
| Conceptual queries ("memory persistence strategy") | Poor | **Excellent** |

Physical retrieval is **exactly complementary** to embedding-based retrieval. The cases where physical retrieval wins are the cases where embedding-based retrieval fails (Q231 finding: identifier-dense homelab memories).

## Implementation for Recall 2.0

**Option A: Character n-gram BM25 (Tantivy with character tokenizer)**

Tantivy supports custom tokenizers including character n-gram tokenizers. Configuration:
```rust
let ngram_tokenizer = NgramTokenizer::new(3, 4, false);  // 3-4 char n-grams
schema_builder.add_text_field("content_ngram", STORED | ngram_tokenizer);
```

This gives BM25-weighted character n-gram retrieval. Storage: ~300KB for 22K memories (slightly larger than word-level BM25). Latency: <5ms.

**Option B: MinHash LSH in LMDB**

Pure MinHash without Tantivy. Simpler, lower storage overhead, but no BM25 weighting. For k=4 n-grams, b=20 bands, r=5 rows: 100 hash values per memory × 22K memories × 8 bytes = 17.6MB additional storage. Query time: O(b) = O(20) bucket lookups.

**Recommended**: Option A (Tantivy) since Q231 already recommends adding Tantivy for BM25. The character n-gram tokenizer is a configuration option on the same Tantivy instance. Single index, two tokenizers (word-level + character n-gram), merged via RRF.

## Verdict

**BREAKTHROUGH.** The TABOO analysis reveals that the heterogeneous retrieval problem dissolves when approached from physical requirements: all memories in Recall 2.0 are stored as text strings, making character n-gram similarity the universal retrieval primitive. Character-level n-gram Jaccard similarity approximates Normalized Compression Distance (information overlap), which is the formally grounded content-type agnostic similarity measure. Physical retrieval is precisely complementary to embedding-based retrieval — it excels on identifier-dense queries where semantic retrieval fails, confirming the Q231 hybrid retrieval direction from a completely different starting point. Implementation cost: add a character n-gram tokenizer to the Tantivy instance already planned for Phase 4.5.
