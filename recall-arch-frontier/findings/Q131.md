The embedding cache (Q116) has **zero entries** — it has not been deployed. Five key categories are present in Redis that were not modeled in Q116 or Q123, the largest being 1,365 `recall:markov:trans:*` keys forming a file-transition graph.

### Key structural observations

**1. Markov transition graph (1,365 keys, 191 KB)**

The largest key category by count. Keys follow the pattern `recall:markov:trans:{filepath}` — e.g., `recall:markov:trans:adbp/questions.md`, `recall:markov:trans:agents/hypothesis-generator.md`. These are hash keys averaging 144 bytes each. They represent a file-to-file transition probability graph used by a prefetch/prediction subsystem (`recall:markov:prefetch:*`). This subsystem does not appear in the Q116 or Q123 memory models. No TTL is set on these keys — they accumulate permanently. At 144 bytes each, 1,365 keys = 191 KB today; at the current Recall growth rate, this grows proportionally to Tim's file-editing activity.

**2. Write guard (1 key, 196 KB, 12.8-day TTL)**

`recall:write_guard:injection_embeddings_v1` is a single string key of 165,274 bytes (165 KB raw value, 196 KB with Redis overhead). Its 12.8-day TTL (1,110,141 seconds) classifies it as a `volatile-lru` candidate. The key appears to be a serialized set of injection embedding fingerprints used to deduplicate observed file edits before storing memories. At 165 KB, it is the second-largest individual key in the keyspace.

**3. Retrieval context cache (123 keys, 19.2 KB, 1.7-hour TTL)**

Keys follow `recall:retrieval_ctx:{UUID}`. Each is 160 bytes. TTLs cluster around 6,000 seconds (~1.7 hours). This is a per-retrieve-call context cache — likely storing the result set from a recent retrieval to avoid duplicate computation. These are correctly volatile (`volatile-lru` eviction candidates). At 123 active contexts × 160 bytes, the footprint is negligible.

**4. Absent infrastructure keys**

`recall:global_epoch` has `TTL = -2` (key does not exist). `recall:canary_seeds` is also absent. These are the two Tier 1 infrastructure keys Q111 and Q103 depend on. Their absence means:
- The epoch sanity guard (Q111) has never been initialized on this deployment, or was deleted.
- The canary health check (Q103) cannot run — there are no seed UUIDs to probe.
- Q126's `--appendfsync everysec` and `--save ""` flags are absent from docker-compose.yml, confirming Q123 recommendations were not applied.

## Critical Gaps

### Gap 1: maxmemory=0 and noeviction (OOM-kill risk)

The live docker-compose.yml (`command: redis-server --appendonly yes`) has no `--maxmemory` or `--maxmemory-policy` flags. This was confirmed by `CONFIG GET maxmemory` returning `0`. Redis will consume memory unchecked. At current growth rates this is not an imminent risk (2.76 MB vs 256 MB Docker limit), but:

- The Markov transition graph grows without bound (no TTL, no size cap).
- The write guard key is 196 KB and rotates on a 12.8-day TTL cycle. Its size may grow with corpus growth.
- No eviction safety net exists if a new subsystem introduces a memory leak.

### Gap 2: recall:global_epoch is absent

`TTL recall:global_epoch` returns `-2` (key does not exist). The Q111 epoch sanity guard depends on this key. If the epoch counter has never been initialized, the guard will fire a false `epoch_reset_detected` on the first retrieve call that checks it, triggering a repair write. The Q103 canary system also cannot function without `recall:canary_seeds`.

### Gap 3: Markov graph has no memory ceiling

The 1,365 `recall:markov:trans:*` keys have no TTL and no documented size cap. At 144 bytes per key, the current cost is 191 KB — trivial. However, this graph grows with the number of distinct file paths Tim edits. At 10× file diversity (13,650 keys), this reaches 1.9 MB — still small, but the absence of any bounding mechanism is a latent risk for deployments with high file diversity.

## Qualifying Ideas

### markov-graph-redis-memory-audit
- (N=0.67, E=0.90, F=0.95)
- The `recall:markov:trans:*` category (1,365 keys, no TTL) represents an unmodeled, unbounded subsystem in Redis. A periodic audit that counts these keys and emits a health signal when the count exceeds a configurable threshold (e.g., 10,000 keys = 1.4 MB) would close the memory ceiling gap. Implementation: add `recall:markov:trans count` to the `/health` endpoint response alongside `used_memory_human`. No code refactor needed — a single `DBSIZE` filtered by prefix.

### write-guard-size-tracking
- (N=0.68, E=0.88, F=0.92)
- The `recall:write_guard:injection_embeddings_v1` key is 165 KB and growing with corpus size (it appears to store embedding fingerprints for all injected content). At 22,349 vectors it is 165 KB; the size scales with corpus unless the key is periodically pruned. The 12.8-day TTL provides a natural ceiling (the key rotates), but if the corpus doubles before rotation, the key may reach 300–400 KB. Adding `MEMORY USAGE recall:write_guard:*` to the health check output makes this growth visible before it becomes a problem.

### redis-config-deployment-urgency
- (N=0.62, E=0.97, F=0.99)
- The Q123 `--maxmemory 100mb --maxmemory-policy volatile-lru --appendfsync everysec --save ""` flags are the highest-ROI single-line change available to the Recall deployment. Confirmed absent. Applying these flags closes the OOM-kill risk, restores `volatile-lru` structural immunity for future critical keys, and enables proper AOF flush policy. Risk of applying: zero (requires `docker compose restart redis`; AOF replays all persistent keys; 2.76 MB footprint means eviction will never fire). The `recall:global_epoch` and `recall:canary_seeds` initialization should be paired with this change.

## Recommendations

1. **Apply Q123 docker-compose.yml change immediately** — add `--maxmemory 100mb --maxmemory-policy volatile-lru --appendfsync everysec --save ""` to the Redis `command:` block. This is the only change needed to close the OOM-kill risk. The 2.76 MB footprint means eviction will never trigger in practice; the `maxmemory` cap is a safety net only.

2. **Initialize recall:global_epoch and recall:canary_seeds** — both keys are absent. The epoch counter should be set to its correct value (or 0 if this is effectively a fresh deployment). The canary seeds should be populated with the Q103 UUIDs and hardened content (Q127 specification).

3. **Document the Markov transition graph in the Redis key registry (Q124)** — the `recall:markov:trans:*` category is the largest by key count (1,365 keys) and is not in the Q123 model. It should be classified as Tier 3 (unbounded operational data, no TTL) and given a soft ceiling in the health check.

4. **Deploy the Q116 embedding cache** — the cache has zero entries and zero latency savings today. Every retrieve call pays 1.68 ms for Ollama embedding. At Tim's usage rate, deploying the cache saves ~1 ms per retrieve call at 65% hit rate. The footprint cost is ~485 KB at steady state — negligible.

5. **Do not set TTL on recall:global_epoch or recall:canary_seeds** — confirm these are initialized without TTL so `volatile-lru` structural immunity applies as soon as Q123 config is deployed.
