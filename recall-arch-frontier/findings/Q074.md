# Finding: Q074 — PG-Restore Direction Creates Symmetric Blind Spot in Q067 Epoch Guard; Bidirectional Guard Required

**Question**: The epoch continuity guard (Q067) detects the case where `pg_max > redis_epoch` (Redis was reset while PG retained data). But the guard runs only at session start. What happens in the opposite failure direction: `redis_epoch >> pg_max` (PostgreSQL was restored from a backup taken before the current epoch value)? In this case all memories appear stale (`last_access_epoch` far below current epoch), the LLPC working-set filter rejects every candidate, and retrieval returns empty results. Is this scenario detectable by the same guard, what is the correct recovery action, and how long would the system silently return empty results before the symptom became apparent to Tim?
**Agent**: quantitative-analyst
**Question Type**: ADVERSARIAL
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: distributed systems (epoch continuity protocols, backup-restore consistency), database operations (point-in-time recovery, CasaOS self-hosted infrastructure)

## Failure Mode Analysis

### The failure direction

PostgreSQL is restored from a backup taken at epoch 100. Redis has continued running and is currently at epoch 500. The backup may have been taken for any reason: routine snapshot, disaster recovery test, corrupt PG data directory, accidental volume rollback on CasaOS.

After the restore:
- `pg_max = SELECT MAX(last_access_epoch) FROM memories = 100`
- `redis_epoch = GET recall:global_epoch = 500`

The LLPC working-set filter at session start evaluates:

```
WHERE last_access_epoch >= (current_epoch - tau)
WHERE last_access_epoch >= (500 - 20)
WHERE last_access_epoch >= 480
```

Every memory in the restored database has `last_access_epoch <= 100`. The threshold is 480. Zero rows pass the filter. Every LLPC routing query returns an empty result set.

### Concrete scale example

Suppose Tim had been running Recall for 10 months before the backup was taken (epoch ~100), and has run 400 more sessions since the backup snapshot (epoch 500). All memories from those 400 sessions are simply gone from PostgreSQL — the backup predates them. Memories that existed at backup time have `last_access_epoch` in [0, 100], far below the 480 threshold.

The effect is identical to a complete memory erasure: Tim asks a question, Recall finds nothing, no memories are injected. The hooks appear to run (no errors), the API returns a valid empty response, and Claude operates without any memory context for every session.

### What PG data is affected

The backup-restore failure is two-layered:

1. **Missing memories**: Any memory created after the backup date is permanently lost (PG rows do not exist).
2. **Existing memories invisible**: Memories that did exist at backup time are present in PG but cannot pass the epoch filter. These memories are retrievable — they are not gone — but the filter excludes them silently.

Layer 2 (the invisible-but-present memories) is the recoverable component. Layer 1 (the lost memories) requires a decision about which backup to restore from. Q074 focuses on layer 2: the epoch-filter invisibility of memories that are structurally intact in PostgreSQL.

## Detection Analysis

### Does the Q067 guard detect this direction?

No. The Q067 guard checks:

```python
if pg_max > redis_epoch:
    # Redis reset detected — restore Redis to pg_max + 1
```

In the PG-restore direction, `pg_max (100) < redis_epoch (500)`. The condition `pg_max > redis_epoch` is `100 > 500`, which evaluates to **False**. The Q067 guard fires on one inequality direction only. The PG-restore failure produces the opposite inequality and passes through the guard silently.

This is a symmetric blind spot in the guard design. Q067 was designed to detect Redis falling behind PG (Redis reset). It has no detection path for PG falling behind Redis (PG restore).

### Why the normal inequality assumption breaks down

Under normal operation, Q067's assumption is:
- Redis epoch always leads PG max by a small amount (1-2 sessions worth), because `INCR recall:global_epoch` fires at session end before the `last_access_epoch` UPDATE completes.
- Therefore, `redis_epoch >= pg_max` is always true in steady state, and `pg_max > redis_epoch` is an unambiguous anomaly.

After a PG restore from backup:
- `redis_epoch >= pg_max` is STILL true (500 >= 100).
- The Q067 guard sees this as the normal steady-state condition.
- The guard concludes "no anomaly detected" and returns `redis_epoch = 500`.
- All LLPC routing queries use epoch 500 as the current epoch.
- All memories in the restored PG database have epoch <= 100.
- `100 >= 480` is false. Zero memories qualify.

### What guard can detect this?

The PG-restore direction requires a different detection predicate. The relevant signal is not `pg_max vs redis_epoch` but rather the **epoch gap magnitude**:

```
epoch_gap = redis_epoch - pg_max
```

In steady state, this gap should be small: 0 to a few sessions (the amount of time Redis epoch has advanced since the last PG write). A gap of 400 epochs (redis=500, pg_max=100) is not a steady-state value — it is a signal that either PG has been restored from an old backup or PG has received no writes for 400 sessions (implying all recent sessions failed to persist, itself a critical anomaly).

A bidirectional guard adds a second threshold:

```python
EPOCH_GAP_ALARM_THRESHOLD = 50  # Sessions — configurable

async def check_epoch_continuity_bidirectional(redis_client, pg_pool) -> tuple[int, str]:
    redis_epoch, pg_max = await asyncio.gather(
        redis_client.get("recall:global_epoch"),
        pg_pool.fetchval("SELECT MAX(last_access_epoch) FROM memories")
    )
    redis_epoch = int(redis_epoch or 0)
    pg_max = int(pg_max or 0)

    if pg_max > redis_epoch:
        # Direction 1: Redis reset (Q067 case)
        # Redis epoch is below the highest epoch stored in PG
        restored_epoch = pg_max + 1
        await redis_client.set("recall:global_epoch", restored_epoch)
        return restored_epoch, "REDIS_RESET_RECOVERED"

    epoch_gap = redis_epoch - pg_max
    if epoch_gap > EPOCH_GAP_ALARM_THRESHOLD:
        # Direction 2: PG restore from old backup (Q074 case)
        # Redis epoch has advanced far beyond the highest epoch in PG
        # Cannot auto-recover: the correct action depends on operational intent
        return redis_epoch, f"PG_RESTORE_SUSPECTED:gap={epoch_gap}"

    return redis_epoch, "OK"
```

The gap threshold of 50 is calibrated against tau=20: if `redis_epoch - pg_max > tau`, then no memory in the database can satisfy the working-set filter, making the threshold operationally meaningful. The exact value should be configurable; `tau + some_safety_margin` is the natural anchor.

## Recovery Procedure

The Q067 (Redis reset) recovery is simple and automatic: set Redis to `pg_max + 1`. The PG-restore recovery is fundamentally different and requires an operator decision.

### Why auto-recovery is incorrect for the PG-restore direction

In the Redis reset case, PostgreSQL is the ground truth. PG contains the correct epoch history; Redis has lost its counter. Setting Redis to `pg_max + 1` is safe and unambiguous.

In the PG-restore case, Redis is the ground truth for the current epoch. PostgreSQL is behind because it was deliberately restored from an earlier state. The operator must decide:

**Option A — Epoch retraction (accept the backup, roll back Redis):**
- Set `recall:global_epoch` in Redis to `pg_max + 1` (e.g., 101).
- This acknowledges that the 400 sessions since the backup are lost (they cannot be recovered without the lost PG data).
- Memories present in the backup are immediately retrievable (last_access_epoch <= 100, threshold = 101-20 = 81, memories accessed in epochs 81-100 qualify).
- The epoch counter restarts from 101 and continues forward.
- Cost: 400 sessions of memory are permanently lost; epoch continuity is restored.

**Option B — Epoch preservation (reject the restore, find a better backup):**
- Do not touch Redis.
- Restore PG from a more recent backup (one taken closer to epoch 500).
- If no recent backup exists, consider whether the CasaOS backup schedule needs adjustment.

**Option C — Emergency bypass (retrieve without working-set filter):**
- Temporarily set tau to a very large value (e.g., 10,000) to allow all memories to pass the filter.
- This acts as a full-corpus retrieval mode — every memory in the backup is eligible.
- Useful for immediate operational continuity while the correct recovery option is evaluated.
- Revert tau to 20 after permanent resolution.

### Step-by-step recovery (Option A — most common CasaOS scenario)

1. **Detect the condition** (automated, at session start): Guard returns `PG_RESTORE_SUSPECTED:gap=400`. Log this; alert Tim via Discord or stdout.
2. **Pause Recall writes**: Temporarily disable `observe-edit.js` and `recall-session-summary.js` to prevent new writes to the restored PG instance from accumulating while the situation is assessed.
3. **Assess backup freshness**: Check the backup date. If the backup is recent (< 1 week), data loss is bounded. If the backup is months old, evaluate whether a newer backup exists elsewhere (CasaOS snapshots, secondary volume).
4. **Execute epoch retraction**: `redis-cli SET recall:global_epoch $(psql -t -c "SELECT MAX(last_access_epoch) + 1 FROM memories")`
5. **Verify retrieval resumes**: Start a new session. The guard should now see `redis_epoch = 101`, `pg_max = 100`, gap = 1. Status: OK. LLPC routing with tau=20 admits memories from epochs 81-100.
6. **Re-enable writes**: Restore `observe-edit.js` and `recall-session-summary.js` to active state.
7. **Document the data loss window**: The memories from epochs 101-500 (the ~400 sessions since the backup) are permanently lost. Write a note to `project-brief.md` marking the recovery date and the epoch gap.

## Timing Analysis

### How long before Tim notices?

The failure produces **complete retrieval silence**: every session returns zero memories. This is immediately noticeable in quality terms (Claude has no memory of anything) but not immediately noticeable as a system error (no error messages, no hook failures, no API 500s).

**Timeline to detection:**

| Time | What Tim observes |
|------|-------------------|
| Session N+1 (immediate) | Claude has no memory context. Seems like a fresh session. Tim may attribute this to context loss or Recall being slow. |
| Session N+3 to N+5 | Tim notices that Claude does not recall facts from recent sessions that were definitely stored. The behavior is consistently empty, not intermittent. |
| Day 2-3 | Tim asks Recall directly: "What do you know about X?" — recall_search returns nothing. He investigates. |
| Day 3-7 | Tim checks CasaOS Docker logs, finds no errors. He runs `curl http://192.168.50.19:8200/health` — returns 200. He runs a manual search — returns empty. He checks the PostgreSQL table and finds memories present but notices the epoch values are all in [0, 100] range while Redis reports 500. |

**Realistic detection window: 1-7 days**, depending on how much cognitive load Tim places on memory continuity in any given session. Sessions where he explicitly asks Claude about past work are more likely to trigger recognition.

### Why the window is longer than the Q067 (Redis reset) window

In the Q067 failure (Redis reset), all memories flood into every prompt — Tim notices immediately that irrelevant memories are being surfaced. The signal is noisy and visible. In the Q074 failure (PG restore), the opposite occurs: silence. Silence is harder to notice than noise, especially in an AI assistant context where "Claude didn't use memory this session" is a plausible false negative on any individual session.

At 5 sessions per day, Tim's expected detection window is approximately:
- Best case: Day 1 (if he explicitly tests recall after a known-memorable session).
- Expected case: Day 2-5 (3-5 sessions of confusion before investigation).
- Worst case: 1+ weeks if Tim is working on new unfamiliar topics where he would not expect memory continuity.

During this entire window, the system silently returns empty results. The session summary hook continues running and attempting to write new memories — those new writes succeed and accumulate correctly in the restored PG (with new epoch values starting from wherever Redis is). This means a self-healing dynamic does NOT exist: unlike the Q067 case where natural recovery occurs after tau sessions, the Q074 case has no natural recovery. The epoch gap between Redis (500) and the highest PG epoch (100 + sessions since restore) grows every session, keeping all backup-era memories permanently invisible.

### The self-healing absence (critical asymmetry)

Q067's Redis reset failure has a natural recovery path: after 867 sessions (173 days), the false positive resolves as Redis epoch grows past the old PG epoch values. The system is wrong for 173 days but eventually self-corrects.

Q074's PG restore failure has **no natural recovery**. Redis epoch continues advancing (501, 502, 503...) while PG's backup-era memories remain frozen at epochs <= 100. The working-set window `[redis_epoch - 20, redis_epoch]` moves forward every session, leaving the backup-era memories permanently behind. This is a monotonically worsening condition, not a self-healing one.

After 100 more sessions (redis_epoch = 600), the gap is 500. After 1000 more sessions (redis_epoch = 1500), the gap is 1400. The backup-era memories are never retrievable without operator intervention.

## Ideas Extracted

### Idea: bidirectional-epoch-continuity-guard
- **Novelty**: 0.78 — Q067's unidirectional guard (`pg_max > redis_epoch`) is a known distributed systems pattern (Lamport timestamp recovery after counter reset). The extension to detect the opposite direction (`redis_epoch >> pg_max` as a backup-restore signal) via an epoch gap threshold is not observed in current AI memory systems. The specific coupling of the gap threshold to tau (the working-set window) — using `gap > tau` as the operational alarm boundary rather than an arbitrary constant — is a novel calibration that ties the guard's sensitivity directly to the system's functional behavior.
- **Evidence**: 0.72 — Epoch gap monitoring is used in distributed databases to detect replica lag (e.g., PostgreSQL `pg_stat_replication.write_lag`, MySQL `SHOW SLAVE STATUS`) and in message queue systems to detect consumer lag (Kafka consumer group lag monitoring). The backup-restore detection via epoch gap is structurally analogous to replica lag detection, with the backup snapshot taking the role of the lagging replica. The tau-calibrated threshold is derived from first principles from the working-set filter definition and is independently verifiable.
- **Feasibility**: 0.90 — The bidirectional guard is a 15-line extension of the existing `check_epoch_continuity()` function from Q067. The gap threshold is a single configurable constant. The guard already runs at session start and issues both the Redis GET and PG MAX queries; adding the gap comparison costs zero additional I/O. The result is a tuple `(epoch, status)` rather than a single integer, requiring the caller to handle the new status codes — a minor interface change. The auto-recovery for Direction 1 (Redis reset) remains unchanged; Direction 2 (PG restore) logs an alarm and allows the system to continue with degraded behavior rather than attempting incorrect auto-recovery.
- **Description**: Extend the Q067 epoch continuity guard from a unidirectional check (`pg_max > redis_epoch`) to a bidirectional check. Direction 1: Redis reset — unchanged from Q067, auto-recovers by setting Redis to `pg_max + 1`. Direction 2: PG restore — detected when `redis_epoch - pg_max > EPOCH_GAP_ALARM_THRESHOLD` (where threshold = tau + safety margin, default 50). Direction 2 cannot auto-recover safely; instead, the guard emits a structured alarm (`PG_RESTORE_SUSPECTED:gap=N`) and returns the current Redis epoch unchanged. The alarm triggers operator notification (Discord webhook, log warning, or a FastAPI `/health` endpoint that returns degraded status). Without this extension, the PG-restore direction passes the existing guard silently and causes indefinite retrieval silence with no natural recovery.

## Key Finding

The Q067 epoch continuity guard has a symmetric blind spot: it detects only the Redis-reset direction (`pg_max > redis_epoch`) and is completely blind to the PG-restore direction (`redis_epoch >> pg_max`). When PostgreSQL is restored from a backup taken at epoch 100 while Redis continues running at epoch 500, the guard evaluates `100 > 500 = False`, concludes the system is healthy, and returns the current Redis epoch (500) to all callers. The LLPC working-set filter then demands `last_access_epoch >= 480`, and every memory in the restored database (all with `last_access_epoch <= 100`) is silently rejected. Retrieval returns empty results for every session with no error, no log warning, and no natural recovery path.

This is strictly worse than the Q067 failure mode in one critical dimension: the Q067 false-positive resolves naturally after tau sessions as the Redis epoch grows past the old PG values (173 days worst case). The Q074 false-negative has no natural recovery — the epoch gap grows every session (redis_epoch advances while pg_max is frozen at 100), keeping backup-era memories permanently invisible. Tim's expected detection window is 2-7 days based on observational patterns (Claude appearing to have forgotten everything, confirm by manual Recall search returning empty), followed by an investigation path that is non-obvious (the API returns 200 OK, the Docker containers are healthy, the PG table has data in it — only inspection of the epoch values reveals the gap).

The correct fix is a bidirectional guard that additionally checks `redis_epoch - pg_max > EPOCH_GAP_THRESHOLD` (where threshold is calibrated to tau, e.g., tau + 30 = 50 sessions). Unlike the Redis-reset direction, the PG-restore direction cannot auto-recover safely — the guard must emit an alarm and require operator intervention to choose between epoch retraction (accept the data loss, roll Redis back to `pg_max + 1`) or finding a more recent backup. Q069's deployment sequence requires an additional post-restore runbook step documenting this decision tree, as no such contingency currently exists in the operational documentation.
