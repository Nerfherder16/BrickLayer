## Component 1: Cypher Change — LIMIT 50 as Default

### Current Cypher (from Q083 baseline)

```cypher
MATCH (seed:Memory {id: $seed_id})-[:CO_RETRIEVED]-(neighbor:Memory)
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)
RETURN neighbor.id LIMIT 20
```

### Updated Cypher — K=50 default

```cypher
-- K=50 is the RANKING limit (how many candidates are returned to Python for downstream ranking).
-- This is NOT the traversal limit. Neo4j reads all fanout edges regardless of LIMIT.
-- Per Q083: K=50 is safe at p95 traversal cost (<5ms) for non-hub nodes through Month 24.
-- Hub node protection is provided by the asyncio.wait_for timeout, not by reducing LIMIT.
MATCH (seed:Memory {id: $seed_id})-[:CO_RETRIEVED]-(neighbor:Memory)
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)
RETURN neighbor.id, neighbor.last_access_epoch
ORDER BY neighbor.last_access_epoch DESC
LIMIT 50
```

### Why LIMIT 50, not LIMIT 20

Q083's BREAKTHROUGH finding distinguishes two separate K caps:

| Cap | What it protects | Bound | Mechanism |
|-----|-----------------|-------|-----------|
| K_fanout (traversal budget) | Neo4j read time | ≤200 edges before 10ms graph sub-budget exceeded | Graph structure — cannot be changed by LIMIT |
| K_limit (ranking budget) | Python downstream processing | 20–50 is appropriate | LIMIT clause in Cypher |

`LIMIT 20` was derived for worst-case hub-node traversal (Month 24, hub fanout 500–1500). But the traversal cost is determined by the node's fanout, not by LIMIT. The LIMIT only controls how many *surviving* candidates are returned to Python after the epoch filter. Raising LIMIT from 20 to 50 does not change Neo4j traversal cost at all — it expands the downstream ranking candidate pool at zero traversal cost for non-hub nodes, and at zero *additional* traversal cost for hub nodes (they are still reading all F relationships regardless). The asyncio timeout provides hub-node protection.

## Component 2: asyncio.wait_for Wrapper

### Design constraints

- FastAPI runs on Python asyncio (uvicorn event loop)
- The Neo4j `AsyncGraphDatabase` driver returns coroutines from `session.run()` and `result.data()`
- `asyncio.wait_for(coro, timeout)` cancels the coroutine if it does not complete within `timeout` seconds and raises `asyncio.TimeoutError`
- The 8ms timeout target leaves 2ms margin below the 10ms Neo4j sub-budget at p95

### Full implementation

```python
import asyncio
import logging
import time
from typing import Optional
from neo4j import AsyncGraphDatabase

logger = logging.getLogger(__name__)

# ──────────────────────────────────────────────────────────────────────────────
# Constants
# ──────────────────────────────────────────────────────────────────────────────

NEO4J_TIMEOUT_MS = 8          # Circuit-breaker trigger threshold (milliseconds)
NEO4J_TIMEOUT_S  = NEO4J_TIMEOUT_MS / 1000.0  # asyncio.wait_for takes seconds
K_DEFAULT        = 50         # Ranking K: default, safe for non-hub nodes
K_FALLBACK       = 20         # Ranking K: fallback when circuit breaker is open
CB_OPEN_AFTER    = 3          # Consecutive timeouts before circuit breaker opens
CB_CLOSE_AFTER   = 10         # Successful calls before resetting consecutive counter
TAU_DEFAULT      = 20         # Working-set window (sessions)


# ──────────────────────────────────────────────────────────────────────────────
# Cypher templates
# ──────────────────────────────────────────────────────────────────────────────

_CYPHER_BFS = """
MATCH (seed:Memory {{id: $seed_id}})-[:CO_RETRIEVED]-(neighbor:Memory)
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)
RETURN neighbor.id, neighbor.last_access_epoch
ORDER BY neighbor.last_access_epoch DESC
LIMIT {k}
"""

CYPHER_BFS_K50 = _CYPHER_BFS.format(k=K_DEFAULT)   # Pre-formatted at import time
CYPHER_BFS_K20 = _CYPHER_BFS.format(k=K_FALLBACK)  # Pre-formatted at import time


# ──────────────────────────────────────────────────────────────────────────────
# Circuit breaker state
# ──────────────────────────────────────────────────────────────────────────────

class _CircuitBreakerState:
    """
    Session-scoped circuit breaker for Neo4j K=50 BFS calls.
    Thread-safe for a single asyncio event loop (no lock needed — all access
    is from the same event loop via await).
    """

    def __init__(self):
        self.consecutive_timeouts: int = 0   # Resets on any success
        self.is_open: bool = False            # True = use K=20 permanently this session
        self.total_timeouts: int = 0          # Lifetime counter (for observability)
        self.total_calls: int = 0

    def record_timeout(self) -> None:
        self.total_timeouts += 1
        self.consecutive_timeouts += 1
        if self.consecutive_timeouts >= CB_OPEN_AFTER:
            if not self.is_open:
                logger.warning(
                    "neo4j_circuit_breaker_open: %d consecutive timeouts on K=%d BFS; "
                    "switching to K=%d for remainder of session",
                    self.consecutive_timeouts, K_DEFAULT, K_FALLBACK,
                )
            self.is_open = True

    def record_success(self) -> None:
        self.total_calls += 1
        # Do NOT reset is_open during a session — only reset consecutive_timeouts
        # so that a single success does not re-enable K=50 prematurely.
        # The circuit breaker stays open for the session boundary.
        if self.consecutive_timeouts > 0:
            self.consecutive_timeouts = 0

    def effective_k(self) -> int:
        return K_FALLBACK if self.is_open else K_DEFAULT

    def effective_cypher(self) -> str:
        return CYPHER_BFS_K20 if self.is_open else CYPHER_BFS_K50


# Module-level state (one instance per worker process, survives request boundaries)
_cb_state = _CircuitBreakerState()


def reset_circuit_breaker() -> None:
    """
    Call at session boundary (e.g., at the start of each Claude Code session hook).
    Resets consecutive timeout counter and closes the circuit breaker.
    """
    global _cb_state
    _cb_state = _CircuitBreakerState()
    logger.info("neo4j_circuit_breaker_reset: new session boundary")


# ──────────────────────────────────────────────────────────────────────────────
# Core retrieval function
# ──────────────────────────────────────────────────────────────────────────────

async def _fetch_bfs_neighbors(
    driver: AsyncGraphDatabase,
    seed_id: str,
    current_epoch: int,
    tau: int,
    cypher: str,
    k: int,
) -> list[dict]:
    """
    Execute one BFS query. Returns list of {id, last_access_epoch} dicts.
    Raises asyncio.TimeoutError if the driver does not respond within NEO4J_TIMEOUT_S.
    """
    async with driver.session() as session:
        result = await session.run(
            cypher,
            seed_id=seed_id,
            current_epoch=current_epoch,
            tau=tau,
        )
        # .data() fetches all rows — we wrap the entire driver round-trip in wait_for
        records = await result.data()
    return records


async def get_bfs_neighbors(
    driver: AsyncGraphDatabase,
    seed_id: str,
    current_epoch: int,
    tau: int = TAU_DEFAULT,
) -> tuple[list[dict], int, bool]:
    """
    Retrieve BFS neighbors for a seed node, applying the adaptive K cap.

    Returns:
        (neighbors, k_used, timed_out)
        - neighbors: list of {id, last_access_epoch} dicts (len <= k_used)
        - k_used: actual K limit applied (50 or 20)
        - timed_out: True if the K=50 call timed out and K=20 fallback was used

    Latency contract:
        - p50: 3-5ms (non-hub nodes, cache-warm NVMe)
        - p95: ≤8ms (non-hub nodes); ≤NEO4J_TIMEOUT_MS before fallback fires
        - fallback path (K=20): re-executes Cypher with K=20 (see Q090 Rollback section)
    """
    cypher = _cb_state.effective_cypher()
    k = _cb_state.effective_k()

    t0 = time.monotonic()

    try:
        neighbors = await asyncio.wait_for(
            _fetch_bfs_neighbors(driver, seed_id, current_epoch, tau, cypher, k),
            timeout=NEO4J_TIMEOUT_S,
        )
        elapsed_ms = (time.monotonic() - t0) * 1000
        _cb_state.record_success()
        logger.debug(
            "neo4j_bfs_ok seed=%s k=%d elapsed_ms=%.1f results=%d",
            seed_id, k, elapsed_ms, len(neighbors),
        )
        return neighbors, k, False

    except asyncio.TimeoutError:
        elapsed_ms = (time.monotonic() - t0) * 1000
        _cb_state.record_timeout()
        logger.warning(
            "neo4j_bfs_timeout seed=%s k=%d elapsed_ms=%.1f consecutive=%d is_open=%s",
            seed_id, k, elapsed_ms,
            _cb_state.consecutive_timeouts,
            _cb_state.is_open,
        )

        # ── Fallback path: re-execute with K=20 ──────────────────────────────
        # The timed-out coroutine has been cancelled by asyncio.wait_for.
        # We cannot truncate its partial result — the connection is in an
        # indeterminate state and the coroutine was cancelled at the Python level.
        # We must issue a fresh Cypher query with LIMIT 20. See Q090 Rollback section.
        try:
            neighbors = await asyncio.wait_for(
                _fetch_bfs_neighbors(
                    driver, seed_id, current_epoch, tau,
                    CYPHER_BFS_K20, K_FALLBACK,
                ),
                timeout=NEO4J_TIMEOUT_S,  # Give K=20 its own 8ms budget
            )
            logger.info(
                "neo4j_bfs_fallback_ok seed=%s k=20 results=%d",
                seed_id, len(neighbors),
            )
            return neighbors, K_FALLBACK, True
        except asyncio.TimeoutError:
            # Hub node so severe that even K=20 times out (fanout >> 500, cold cache)
            logger.error(
                "neo4j_bfs_fallback_timeout seed=%s — hub node likely; returning empty",
                seed_id,
            )
            return [], K_FALLBACK, True
```

### Key asyncio.wait_for details

1. **What `asyncio.wait_for` does on timeout**: It cancels the wrapped coroutine by injecting a `CancelledError` into it at the next `await` point. The Neo4j driver's internal socket read is interrupted. The connection is returned to the driver's pool in a closed state (the driver handles pool eviction on cancelled connections). There is no partial result returned to the caller.

2. **Why the timeout is 8ms, not 10ms**: The 10ms sub-budget is the Neo4j allocation in the 50ms LLPC total. The asyncio.wait_for overhead (coroutine scheduling, event loop tick) adds ~0.1–0.5ms of Python-side latency around the actual Bolt call. Targeting 8ms leaves 2ms margin for asyncio overhead and ensures the total Neo4j slot stays within 10ms.

3. **Driver session lifecycle**: The `async with driver.session()` context manager is inside the coroutine that `wait_for` cancels. On cancellation, `__aexit__` is called via the finally clause of the context manager, which closes the session. The Neo4j async driver is designed to handle session cancellation gracefully.

4. **No partial result truncation possible**: Once `asyncio.wait_for` fires, the coroutine is cancelled and no result is available. There is no way to retrieve the rows that Neo4j had already sent over the wire before the timeout. The fallback must be a fresh Cypher query. See Component 5 for the full rollback path analysis.

## Component 3: Circuit Breaker State Machine

### State diagram

```
                              ┌─────────────────────────────────────────┐
                              │           CLOSED (K=50 active)          │
                              │  consecutive_timeouts < CB_OPEN_AFTER   │
                              └─────────────────────────────────────────┘
                                     │                        ▲
              consecutive_timeouts   │                        │ session boundary
              >= CB_OPEN_AFTER (3)   │                        │ reset_circuit_breaker()
                                     ▼                        │
                              ┌─────────────────────────────────────────┐
                              │            OPEN (K=20 active)           │
                              │  is_open = True                         │
                              │  All BFS calls use K=20 for this session│
                              └─────────────────────────────────────────┘
```

### State transitions

| Event | From state | To state | Action |
|-------|-----------|---------|--------|
| BFS call succeeds | CLOSED | CLOSED | `consecutive_timeouts = 0` |
| BFS timeout | CLOSED | CLOSED (if < 3 consecutive) | `consecutive_timeouts += 1` |
| BFS timeout (3rd consecutive) | CLOSED | OPEN | Log warning; `is_open = True` |
| Any BFS call | OPEN | OPEN | Always use K=20; `consecutive_timeouts = 0` (success) |
| Session boundary | OPEN | CLOSED | `reset_circuit_breaker()` called by hook |

### Session boundary semantics

The circuit breaker stays open for the **entire session** once opened. "Session" corresponds to one Claude Code tool-call context — roughly the duration of one `DISABLE_OMC=1 claude --dangerously-skip-permissions` invocation. The `recall-session-summary.js` Stop hook (which runs at session end) should call the Recall API `POST /api/recall/reset-circuit-breaker` endpoint before the session summary is stored. The next session starts with a fresh closed circuit breaker.

This design is conservative: once three consecutive hub-node timeouts are observed, that seed node's graph neighborhood is adversarial for the remainder of the session. Reopening within the session creates thrashing (K=50 attempt → timeout → fallback → repeat). The session boundary gives the graph traversal sub-system a clean slate, and the hub node's fanout does not change intra-session.

### Why N=3 consecutive timeouts (not cumulative)

Cumulative timeouts over a long session would incorrectly open the circuit breaker after a mix of hub and non-hub seeds. If the session traverses 50 seed nodes, and 3 of them are hub nodes (6%), cumulative counting would open the breaker after 3 timeouts even if they are spread across hours of session time. Consecutive counting requires 3 timeouts in a row — only possible if the same hub node (or a cluster of hub nodes) is being repeatedly seeded, which indicates a genuine pathological access pattern.

## Component 4: Hub Node Detection Without a Registry

### The hub node problem

Q078 established that hub nodes (500–1,500 fanout by Month 24) cause traversal latency spikes. Q083 confirmed K=50 is safe for non-hub nodes (p95 traversal < 5ms) but can spike to 15–48ms for hub nodes at cold cache.

The circuit breaker protects against hub nodes without requiring a separate hub-node registry. The mechanism is behavioral: hub nodes are detected at runtime by the latency of their BFS queries exceeding the 8ms asyncio.wait_for threshold.

### Why no registry is needed

A hub-node registry would require:
1. A separate pass over the Neo4j fanout distribution (Q080's `MATCH (n:Memory) RETURN size((n)-[:CO_RETRIEVED]->()) AS fanout`)
2. A Redis or PostgreSQL table storing `{node_id: is_hub_flag}`, updated as fanout grows
3. The retrieval path checking the registry before deciding which K to use

This is three additional components for a problem that the circuit breaker solves dynamically. The circuit breaker observes the actual latency outcome — which is exactly what matters — without needing to predict it from fanout counts.

### How circuit breaker handles hub nodes

| Scenario | What happens |
|---------|-------------|
| Non-hub seed (fanout ≤ 100) | K=50 BFS completes in 3-5ms, well under 8ms; circuit breaker remains closed; K=50 used |
| Mild hub seed (fanout 100–300) | K=50 BFS takes 5–8ms; may occasionally hit the 8ms threshold at cold cache; 1-2 timeouts before returning to success; consecutive counter resets on success |
| Severe hub seed (fanout 500+) | K=50 BFS repeatedly exceeds 8ms; after 3 consecutive timeouts, circuit opens; K=20 used for remainder of session |
| Severe hub seed, K=20 fallback | K=20 BFS takes 3-5ms even for hub nodes at cold cache (20 × 0.05ms per-neighbor = 1ms + overhead ≈ 3ms total) |

### Cold-cache protection

The circuit breaker also protects against cold-cache scenarios (Neo4j page cache evicted after idle period on CasaOS). Cold-cache traversal costs are 5–10× higher than cache-warm. At cold cache:
- Non-hub nodes (fanout 50): 50 × 0.1ms = 5ms + overhead ≈ 6ms — under the 8ms threshold, succeeds
- Mild hub nodes (fanout 100): 100 × 0.1ms = 10ms — exceeds threshold, falls back to K=20
- K=20 at cold cache: 20 × 0.1ms = 2ms — safely within 8ms threshold

The 8ms threshold was chosen to be forgiving at cache-warm (non-hub nodes at 3–5ms have 3ms headroom) while still catching cold-cache hub nodes (which spike to 10–50ms).

## Component 5: Rollback Path Analysis

### Question: can the already-returned result set be truncated instead of re-executing?

**No.** The answer depends on understanding what `asyncio.wait_for` does to the Neo4j driver coroutine.

When `asyncio.wait_for` fires:
1. It injects `asyncio.CancelledError` into the wrapped coroutine at the next `await` point
2. The coroutine is the `_fetch_bfs_neighbors` function, which is awaiting `result.data()`
3. `result.data()` is awaiting the Neo4j Bolt protocol to finish streaming result rows over TCP
4. The cancellation interrupts the Bolt protocol read in mid-stream
5. The Neo4j driver's session `__aexit__` closes the connection; the driver discards the partial read buffer
6. No rows are returned to `get_bfs_neighbors` — the `asyncio.TimeoutError` is raised immediately

**There are no partially-returned rows to truncate.** The `asyncio.TimeoutError` gives the caller nothing. The only option is a fresh Cypher query with `LIMIT 20`.

### Why re-executing K=20 is not expensive

The K=20 fallback query re-traverses the same graph neighborhood. This appears wasteful (traversal is done twice), but:

1. **Neo4j page cache warming**: The first K=50 traversal (which timed out) spent 8+ms reading relationship records and property chains into the Neo4j page cache. The second K=20 traversal of the same neighborhood finds those records already in the page cache. Cold-cache traversal cost on the second pass is dramatically lower — the cache-warm K=20 traversal completes in 1–3ms.

2. **Total time for timeout + fallback**: 8ms (timeout) + 1-3ms (cache-warm K=20) = 9–11ms. This is within the 10ms Neo4j sub-budget at p95 with slight overage, but the LLPC budget has margin elsewhere (see Component 6).

3. **Frequency of double-execution**: The circuit breaker opens after 3 consecutive timeouts. Assuming 50 BFS calls per session and 2% hub-node incidence (1 hub node), at most 1 double-execution per session occurs before the breaker opens and all subsequent calls use K=20 directly.

### Alternative: result-set truncation via LIMIT in Python

If the Bolt result were streamed row-by-row (rather than fetched entirely via `result.data()`), it would be possible to cancel the coroutine after the first 20 rows are received, avoiding re-execution. This requires changing the driver call pattern:

```python
# Streaming approach (alternative — not recommended)
async with driver.session() as session:
    result = await session.run(cypher_k50, ...)
    neighbors = []
    async for record in result:  # row-by-row iteration
        neighbors.append(record.data())
        if len(neighbors) >= K_FALLBACK:
            break  # Stop after 20 rows — cancel the rest
```

**Why this is NOT recommended:**

1. It introduces a protocol mismatch: the Bolt server continues streaming rows (it sent LIMIT 50 to Neo4j and Neo4j is producing up to 50 rows) while the Python client breaks out of the iteration. The driver must then close the result cursor mid-stream, which sends a `RESET` message over the Bolt connection and adds ~0.5–1ms of protocol overhead.

2. It does not solve the timeout case — we still need `asyncio.wait_for` around the streaming path for hub nodes with very high property-read latency. The streaming approach only helps if the first 20 rows arrive before the timeout, which is not guaranteed for high-fanout nodes (Neo4j reads all F property chains before returning any rows, because the `ORDER BY last_access_epoch DESC` requires a full sort).

3. The `ORDER BY last_access_epoch DESC` clause means Neo4j must read all surviving neighbors before sorting and returning any — the sort is not streaming. The first row is not returned until Neo4j has read all F property chains, sorted them, and begun streaming. For a hub node with F=500 surviving neighbors, the first row arrives after 500 property reads — at the same time all 500 would have arrived without the streaming optimization.

**Conclusion: re-execute with LIMIT 20 is the correct fallback path.** Do not attempt streaming truncation.

## Component 6: Latency Accounting — 50ms p95 SLA Verification

### Full LLPC retrieval path at K=50 default (no timeout)

| Component | p50 latency | p95 latency | Notes |
|-----------|------------|------------|-------|
| Redis epoch read | ~0.5ms | ~1ms | Local LAN, Redis on CasaOS 192.168.50.19 |
| Qdrant ANN search | ~3–5ms | ~5–8ms | Vector similarity over 5K–50K embeddings |
| Neo4j BFS K=50 | ~3–5ms | ~5–8ms | Non-hub nodes, cache-warm NVMe; p95 from Q083 analysis |
| PG metadata join | ~1–2ms | ~2–3ms | Batch SELECT on neighbor IDs returned from Neo4j |
| Python processing | ~1–2ms | ~2–3ms | Filtering, ranking, context assembly |
| **Total** | **~9–14ms** | **~15–23ms** | Well within 50ms SLA |

### Full LLPC path — timeout fallback path (K=50 times out, K=20 used)

| Component | p95 latency | Notes |
|-----------|------------|-------|
| Redis epoch read | ~1ms | |
| Qdrant ANN search | ~8ms | p95 upper bound |
| Neo4j K=50 attempt (times out) | 8ms | asyncio.wait_for fires |
| Neo4j K=20 re-execute | ~3ms | Cache-warm from first attempt; K=20 at cache-warm is ~1-3ms |
| PG metadata join (20 results) | ~2ms | Fewer results → faster join |
| Python processing | ~2ms | |
| **Total (fallback path)** | **~24ms** | Still within 50ms SLA with 26ms headroom |

### Budget headroom analysis

```
Normal path (K=50):
  p95 = 23ms  →  50ms SLA has 27ms headroom (54% unused)

Timeout fallback path:
  p95 = 24ms  →  50ms SLA has 26ms headroom (52% unused)

Worst case (Neo4j cold cache, Qdrant p99, K=50 timeout + K=20 fallback):
  Redis: 2ms + Qdrant: 12ms + Neo4j timeout: 8ms + Neo4j K=20 fallback: 5ms + PG: 4ms + Python: 3ms
  = 34ms  →  16ms headroom (32% unused)
```

The 50ms p95 SLA is not threatened by K=50 adoption. Even the adversarial cold-cache scenario with Qdrant at p99 latency leaves 16ms of headroom. The 8ms asyncio.wait_for threshold is the correct operating point: it catches hub-node spikes before they consume more than 8ms of the Neo4j slot, allowing the fallback path to stay within the overall budget.

### Why 8ms was chosen, not a different threshold

| Threshold | At p95 for non-hub nodes | Effect |
|-----------|--------------------------|--------|
| 5ms | Triggers on many non-hub nodes at cold cache (~6ms cold cache p95) | Too aggressive — causes unnecessary fallbacks |
| 8ms | Catches hub nodes (>10ms at cache-warm); passes non-hub nodes at cold cache (~6ms p95) | Correct operating point |
| 10ms | Passes cold-cache hub nodes with fanout 100–200 (10ms = exact budget ceiling) | Too permissive — allows budget exhaustion before fallback |
| 12ms | Hub nodes exhaust the Neo4j sub-budget before fallback fires | Incorrect — violates the 10ms sub-budget |

8ms is the sweet spot: below the cold-cache non-hub ceiling (6ms), well above the cache-warm non-hub ceiling (5ms), and safely below the 10ms sub-budget. The 2ms margin (8ms threshold vs 10ms sub-budget) absorbs asyncio scheduling jitter.

## Ideas Extracted

### Idea: asyncio-wait-for-as-hub-node-detector
- **Novelty**: 0.72 — Using `asyncio.wait_for` as an implicit hub-node detection mechanism (rather than a defensive timeout for rare errors) is a reframing that does not appear in reviewed AI memory system literature. The standard use of asyncio timeouts is as a fault-tolerance primitive (protect against hung connections, network failures). The adaptive-K use case treats the timeout as a **latency signal** that classifies node types at runtime: nodes whose BFS traversal exceeds 8ms are hub nodes by operational definition, regardless of their fanout count. This eliminates the need for a hub-node registry (which would require a separate measurement pass and staleness management) by making hub-node classification implicit in the retrieval path itself.
- **Evidence**: 0.70 — The behavioral equivalence of "hub node" and "node whose K=50 BFS exceeds 8ms" follows from Q083's traversal cost model: K=50 BFS costs 0.05ms/neighbor × F neighbors + 0.1ms fixed = 8ms at F=(8-0.1)/0.05=158 neighbors. Any node with fanout above ~160 at cache-warm NVMe will trigger the 8ms threshold. This aligns with Q078's "top 1-5% of nodes by fanout" hub definition. The implicit classification is imperfect (cold-cache non-hub nodes may false-trigger) but the circuit breaker's consecutive-timeout requirement (N=3) prevents false-positives from sparse cold-cache events.
- **Feasibility**: 0.92 — `asyncio.wait_for` is a standard library primitive; the Neo4j `AsyncGraphDatabase` driver supports coroutine cancellation. The circuit breaker is a 40-line class. The only non-trivial aspect is ensuring the driver handles coroutine cancellation gracefully (the driver's session context manager closes the connection on `CancelledError` via its `__aexit__` path).
- **Description**: Reframe the `asyncio.wait_for` timeout not as fault protection but as implicit hub-node classification. The 8ms threshold defines the boundary between "non-hub node" (BFS completes in time → K=50 used) and "hub node" (BFS exceeds threshold → K=20 used). This eliminates the hub-node registry entirely: the registry is the runtime behavior of the retrieval system itself. The circuit breaker converts repeated hub-node detections into a session-level K policy switch, which is more appropriate than per-call switching (hub nodes do not change their fanout within a session).

### Idea: neo4j-page-cache-warming-as-fallback-acceleration
- **Novelty**: 0.68 — The observation that a timed-out K=50 BFS warms the Neo4j page cache for the subsequent K=20 fallback query — converting what appears to be a pure cost (double traversal) into a partially-amortized cost (the second traversal benefits from the first's cache loading) — is an emergent property of the implementation that is not explicitly designed but is architecturally beneficial. The cold-cache traversal cost for a hub node with fanout 500 is ~50ms (500 × 0.1ms), but after the first traversal's 8ms of cache loading, the subsequent K=20 traversal finds many of those 500 relationship records already in the Neo4j page cache and completes in 2–5ms rather than 50ms. This converts the worst-case scenario (hub node at cold cache: 8ms timeout + 50ms fallback = 58ms, exceeds LLPC budget) into an acceptable scenario (8ms timeout + 2–5ms cache-warm fallback = 10–13ms).
- **Evidence**: 0.62 — The page-cache warming effect is well-established in database systems literature (the "buffer pool reuse" effect in DBMS sequential scan patterns). For Neo4j specifically, the page cache is an LRU-eviction cache over the relationship store and property store files. The first 8ms of a hub-node BFS loads the leading portion of the relationship chain (the most-recently-reinforced edges, since Neo4j's relationship chain is append-ordered by creation time) into the cache. The K=20 fallback query traverses the same chain in the same order, finding those entries cached. The quantitative estimate (2–5ms for cache-warm K=20 fallback) follows from Q083's 0.05ms/neighbor cache-warm cost × 20 neighbors = 1ms + 0.1ms overhead ≈ 1.1ms — well within the 8ms second timeout budget.
- **Feasibility**: 0.88 — This is an emergent property of the implementation, not a feature requiring additional code. The page-cache warming effect is automatic. No code changes are needed beyond the double-execution fallback already specified in Component 5.
- **Description**: Document the cache-warming effect in the implementation as a design property, not an accident. The implementation comment should note: "The timed-out K=50 query warms the Neo4j page cache with the most-recently-reinforced edges of the hub node's relationship chain. The K=20 fallback benefits from this warming, completing in 2–5ms rather than the 50ms cold-cache worst case. This means the double-execution fallback path has total latency of 8ms + 2–5ms = 10–13ms — within the Neo4j sub-budget — rather than 8ms + 50ms = 58ms." This converts the apparent cost of double execution into a documented benefit.

### Idea: k-cap-policy-as-session-level-not-request-level
- **Novelty**: 0.70 — Standard circuit breaker implementations (Martin Fowler's formulation, Netflix Hystrix) operate at the *request type* level: the circuit breaker trips when failure rate for a given API endpoint exceeds a threshold, protecting all callers from that endpoint for a half-open/open period measured in seconds. The session-scoped circuit breaker proposed here operates at the *calling session* level: the K policy switch applies to the Claude Code session (hours), not to the Neo4j endpoint (seconds). This is appropriate because hub-node access patterns are session-correlated: if Tim is working on his homelab infrastructure (which seeds the Homelab config hub node repeatedly), hub-node access will persist for the entire session. A request-level circuit breaker with a 30-second recovery window would thrash (open → wait 30s → half-open → K=50 → timeout immediately → open again). The session-scoped design avoids this by committing to K=20 for the entire session once the pattern is detected.
- **Evidence**: 0.65 — The session-correlated nature of hub-node access follows from the causal graph's semantics: hub nodes are core infrastructure memories that appear in sessions where Tim works on those topics. If Tim spends a 2-hour session on Homelab networking, the Homelab IP-table memory will be seeded many times. A request-level circuit breaker would oscillate between K=50 and K=20 throughout that session. A session-scoped breaker commits to K=20 for the session duration, eliminating oscillation. The session boundary reset ensures K=50 is retried in the next session (in case the hub node's fanout has not grown further).
- **Feasibility**: 0.90 — The session-scoped circuit breaker is implemented as a module-level singleton reset by the Stop hook. The reset API endpoint (`POST /api/recall/reset-circuit-breaker`) is a 5-line FastAPI endpoint. The session boundary is already a defined event in the Recall system (the Stop hook fires at every session end).
- **Description**: Implement the circuit breaker at session scope, not request scope. Once opened (3 consecutive timeouts), the K=20 policy persists for the remainder of the Claude Code session. Reset occurs at session end via the Stop hook calling `POST /api/recall/reset-circuit-breaker`. This eliminates oscillation in hub-node-heavy sessions (common when Tim works on frequently-retrieved infrastructure topics) and reduces diagnostic complexity: the K policy is stable for the entire session, making latency profiles predictable and log analysis straightforward.

## Key Finding

The adaptive K cap is implementable as a three-component system: (1) a Cypher LIMIT change from 20 to 50 with a comment distinguishing ranking K from traversal K, (2) an `asyncio.wait_for(coro, timeout=0.008)` wrapper around the Neo4j Bolt call with a K=20 re-execution fallback, and (3) a session-scoped circuit breaker that permanently switches to K=20 after 3 consecutive 8ms timeouts.

The most important implementation detail is the rollback path: `asyncio.wait_for` cancels the coroutine entirely on timeout — no partial result is available. The fallback must re-execute the Cypher query with `LIMIT 20`. This re-execution is not expensive in the adversarial (hub node) case because the timed-out first query warmed the Neo4j page cache, reducing the K=20 fallback to 2–5ms. Total timeout-fallback path latency: 8ms + 3ms = 11ms, within the 10ms Neo4j sub-budget with marginal overage absorbed by the LLPC path's 27ms headroom.

The latency accounting confirms the 50ms p95 SLA is not threatened. At K=50 default, p95 LLPC latency is 15–23ms. At the timeout-fallback path, p95 is 24ms. Even the cold-cache worst case (Qdrant p99 + Neo4j cold + hub node + fallback) reaches 34ms, leaving 16ms headroom. The 8ms asyncio threshold is correctly calibrated: it is above the cold-cache non-hub ceiling (6ms) and below the cache-warm hub node floor (10ms+), providing clean separation between hub and non-hub nodes in practice.

The circuit breaker eliminates the need for a hub-node registry by making hub classification implicit in retrieval latency: any node whose K=50 BFS exceeds 8ms is operationally a hub node. Three consecutive timeouts commit the session to K=20, avoiding oscillation in hub-heavy sessions while retrying K=50 at each session boundary.
