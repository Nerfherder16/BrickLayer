# Finding: Q021 — Per-Memory vs. Cluster/Topic Importance: Which Model Is Architecturally Correct?

**Question**: Is per-memory importance scoring the right model, or should importance be a property of memory clusters/topics rather than individual memories?
**Question Type**: ADVERSARIAL
**Verdict**: BREAKTHROUGH
**Severity**: High

---

## Adversarial Analysis: Per-memory vs. cluster importance as the locus of scoring

---

### Thread A: Defense — Per-memory importance is correct because retrieval is per-memory

**Core claim**: Retrieval returns individual memories, not clusters; importance must therefore be a property of individual memories to correctly rank the candidates returned by ANN search.

**Evidence**:

**A1 — The retrieval path is item-level, not cluster-level.** Qdrant returns a ranked list of individual memory vectors with cosine similarities. The current scoring formula `final_score = cosine_similarity × importance_score` operates at the item level. Replacing `importance_score` with a cluster-level score requires either: (a) a pre-retrieval cluster lookup for every Qdrant candidate (adding a Neo4j round-trip in the hot path), or (b) a two-phase retrieve (get candidates from Qdrant → look up cluster scores → re-rank). Both add latency to the retrieve path. Per-memory importance is zero-overhead at query time because the score is a column on the memory row already fetched.

**A2 — Within-topic importance varies and must be preserved.** The claim that "all Docker memories should be elevated when working on Docker" flattens meaningful within-topic distinctions. Consider a cluster containing: "Docker port 8200 is used by Recall" (high importance — needed constantly), "Docker daemon crashed on 2024-01-15 and required restart" (low importance — one-time event), and "Docker uses overlay2 filesystem by default" (medium importance — general knowledge). A cluster-level score is a weighted average of these that masks the distinction. Per-memory scoring preserves it. The retrieval path must return the right five memories from potentially hundreds in a cluster — cluster-level scoring destroys the within-cluster discriminability needed to do this.

**A3 — Access history is genuinely per-memory.** If a user retrieves "Recall port is 8200" 40 times and "Docker overlay2 filesystem" zero times, the retrieval signal attached to these two memories is fundamentally different. This information cannot be represented at the cluster level without losing precision. A cluster-level score would be a pooling of these access counts — which is correct for eviction decisions (the cluster is active) but wrong for retrieval ranking (we need the specific memory with 40 retrievals, not the averaged cluster).

**A4 — Q015's lazy geometric decay is already correct.** The `score_new = score_old × λ^(Δt/τ) + δ` rule proposed in Q015 makes the per-memory score dynamic — it decays when a memory is not accessed and increments when it is. This already provides a form of contextual adaptation: memories from a domain that's currently active accumulate retrievals and therefore accumulate importance. The cluster elevation effect described by Thread B happens organically through the retrieval feedback loop: when the user works on Docker, Docker memories are retrieved, those specific memories get δ bumps, and they rise to the top of ranking for future Docker queries. No explicit cluster-level model is needed.

**Best case scenario**: Per-memory importance with lazy geometric decay produces a self-calibrating importance surface where the memories most relevant to the current work context naturally rise to the top through retrieval feedback, without requiring any cluster-level bookkeeping. The score distribution at any point in time is an accurate empirical map of what the user has found useful, at the individual memory level.

---

### Thread B: Attack — Per-memory importance is wrong because importance is fundamentally contextual

**Core claim**: Importance is not an intrinsic property of a memory but a relational property of the pair (memory, current-work-context); per-memory scoring assigns importance in a context vacuum, producing a static approximation of what should be a dynamic, context-sensitive signal.

**Evidence**:

**B1 — The retrieval formula bakes in a false assumption.** The formula `final_score = cosine_similarity × importance_score` treats importance as a context-independent multiplier. But Q015 established (from decision-theoretic first principles) that `I(M) = P(task | M) − P(task | ¬M)` — importance is always relative to a task T. When you multiply cosine_similarity by a context-free importance score, you are assuming that the importance of M is the same regardless of what task T is currently being executed. This is false. A memory about "Redis TTL configuration" has high importance when T = "debug Redis timeout" and near-zero importance when T = "design a new dashboard". A per-memory score cannot represent this variation.

**B2 — Access history is a lagged proxy, not the actual signal.** Thread A argues that per-memory retrieval counts capture contextual relevance organically. But retrieval counts are a historical average, not a current-context signal. They conflate: (a) memories that are universally useful (retrieved in many different task contexts), (b) memories that happened to be retrieved many times in one past context that is no longer active, and (c) memories that were retrieved recently in the current context. A per-memory score cannot distinguish these three cases. A memory retrieved 50 times during an intensive Recall development period and zero times in the six months since has a high importance score that is a lie about its current relevance.

**B3 — The cluster-level effect is real and unrepresented.** When working on Docker, all Docker-related memories become more jointly relevant — not because any individual one was retrieved, but because the current task context activates the Docker topic. A retrieval path that ranks individual memories by `cosine_similarity × importance_score` will promote memories with high historical importance scores regardless of the current topic. It will deprioritize memories in a dormant cluster with low historical importance scores even if those memories are exactly what's needed for the current Docker task. The cosine_similarity term partially compensates for this (Docker queries will cosine-match Docker memories), but the importance multiplier fights against the similarity signal for any memory that hasn't been recently retrieved.

**B4 — The Q015 lazy decay only works after retrieval — it doesn't elevate pre-retrieval.** Thread A argues that Docker memories rise through retrieval feedback when the user works on Docker. But this requires the memories to be retrieved first. For a user working on Docker for the first time, or returning to Docker after a 3-month absence, the relevant Docker memories have decayed importance scores (no recent retrievals). They will compete poorly against memories from more recently active domains that have accumulated δ bumps. The per-memory decay loop systematically disadvantages topic re-entries and cold starts.

**B5 — Adjacent fields with cluster/topic-level importance models.** Information retrieval query expansion (Rocchio, 1971): when a query vector is expanded toward a relevant cluster centroid, all cluster members receive a boost in the next retrieval — this is implicit cluster-level scoring. Contextual bandits in recommendation systems: the "context-arm" mapping elevates all items in a topic cluster when the context matches that cluster, rather than scoring items individually. Neural IR models (DPR, ColBERT): the document encoder captures cluster-level semantic coherence that the per-document importance score cannot. In each case, the field moved away from per-item static scores toward context-aware, cluster-coherent ranking.

**Failure mode**: A user who intensively used Recall for homelab work for 6 months, then pivots to a software project, finds that homelab memories with high accumulated importance scores crowd out lower-importance project memories in retrieval results — even when the current query is purely about the software project. The cosine_similarity term partially compensates, but the importance multiplier systematically biases results toward the historically-dominant topic cluster, not the currently-active one. This failure worsens as the memory store grows: more historical bias accumulates, making the importance signal increasingly misleading as a proxy for current relevance.

### Synthesis

**What Thread A assumes that Thread B exposes**:

Thread A assumes that **the retrieval path needs importance to discriminate within the result set returned by ANN search**. This is correct. But Thread A also implicitly assumes that the importance score used in `cosine_similarity × importance` should be a **static, context-free** scalar. This is the hidden assumption Thread B exposes: the formula needs an importance term, but that importance term does not have to be a per-memory context-free score — it could be a **contextually-conditioned score** that is computed at query time from both the memory's history and the current query context.

Thread B assumes that **cluster-level importance should replace per-memory importance**. This overstates the case. Retrieval IS per-memory; you cannot return "the Docker cluster" — you return five specific memories. What Thread B is actually pointing at is that the importance term in the retrieval formula should incorporate a topic-activation signal that Thread A's per-memory score cannot represent. The right target is not to replace per-memory scoring but to inject a context-conditioned boost into the per-memory score at query time.

**When Thread A is right**:
- Within-cluster discrimination: when ranking five Docker memories against each other, per-memory importance is the correct tiebreaker (it captures which specific memories have been most useful, not just which topic is active).
- After a topic is warm: once Docker queries have been running for a session, the Docker memories accumulate δ bumps from Q015's lazy decay and naturally rise in the per-memory ranking. At this point, the per-memory score accurately reflects the current context.
- For retrieval, not eviction: when evicting memories, the cluster is the right unit (Q016's finding). When retrieving memories, the individual memory is the right unit. These are separate operations with different correct granularities.

**When Thread B is right**:
- Topic re-entry and cold start: the first query in a new topic-context cannot rely on the retrieval feedback loop because the loop hasn't started. Per-memory scores at this point reflect historical usage, not current relevance.
- Cross-topic interference: when historical importance from a dominant past topic interferes with a current weak topic, per-memory scoring systematically surfaces the wrong memories.
- The importance multiplier fights the similarity signal: a high-similarity, low-historical-importance memory from a currently-active cluster loses to a lower-similarity, high-historical-importance memory from a dormant cluster. The formula's multiplicative structure makes this failure worse as importance-score variance grows.

**The architecture Thread B would accept**:

The fix is not to replace per-memory importance with cluster importance. It is to make the importance term in the retrieval formula **bimodal**: a per-memory base score (from Q015's lazy decay) plus a **query-time topic-activation boost** derived from the current session's retrieval history.

Concretely:

```
final_score(M, q) = cosine_similarity(M, q)
                  × (base_importance(M) + α × topic_activation(topic(M), session))
```

Where:
- `base_importance(M)` = Q015's lazy-decayed per-memory score (static per-memory, computed lazily)
- `topic_activation(topic(M), session)` = the sum of `base_importance` of all memories in M's topic cluster that have been retrieved in the current session
- `α` = a small scaling factor (e.g., 0.2) so the boost is additive, not multiplicative-dominant

**How this works operationally**:
1. The user's first query in a new Docker-focused session retrieves a Docker memory via cosine_similarity (the similarity term still works correctly even for cold-start). That retrieval adds the memory to the session retrieval set (already implemented for Q016's causal protection).
2. For subsequent queries in the same session, `topic_activation(Docker, session)` is non-zero because a Docker cluster member is in the session retrieval set. All Docker memories receive an additive boost proportional to the session's engagement with the Docker topic cluster.
3. Within the Docker cluster, per-memory `base_importance` still discriminates — the memory retrieved 40 times ranks above the memory retrieved twice, preserving Thread A's within-cluster discrimination.
4. After the session ends, the topic-activation boost evaporates (it's session-scoped, stored in Redis with the session retrieval set). The per-memory base importance scores carry the persistent historical signal.

**Why this is the synthesis neither thread could see alone**:

Thread A couldn't see it because Thread A was defending per-memory importance as the complete and sufficient signal. Thread B couldn't see it because Thread B was arguing for cluster-level replacement, which breaks the per-memory retrieval interface.

The synthesis reveals that the retrieval formula needs **two distinct importance components with different scopes**: a persistent per-memory historical score (what Q015 already proposes) and an ephemeral session-scoped topic-activation multiplier (what Q016's session retrieval set already partially implements, but only for eviction, not retrieval). The session retrieval set is already being maintained for causal protection — the additional cost of converting it into a topic-activation signal for the retrieval scoring formula is near-zero. The infrastructure exists; only the retrieval formula needs to change.

**The hidden architectural fact both threads missed**: The session retrieval set (Redis, already implemented by Q016) already contains exactly the information needed to compute topic_activation. When memory M is retrieved, its Neo4j cluster membership is known (or can be inferred from entity tags). The cluster members of every retrieved memory constitute an "active topic set" for the session. The retrieval formula can query this set at query time to apply the boost. No new data structures are needed — only a new connection between two systems that already exist independently.

## Ideas Extracted

### Idea: session-topic-activation-boost

- **Novelty**: 0.78 — Injecting a session-scoped topic-activation signal as an additive term in the retrieval ranking formula, derived from the current session's retrieval history, is not described in any production AI memory system. The closest analogue is query expansion in IR (Rocchio algorithm, 1971), but Rocchio expands the query vector, not the document importance scores. The specific mechanism — using the session retrieval set's cluster membership to compute a per-topic activation scalar applied at query time — is novel.
- **Evidence**: 0.65 — The concept of topic-activation elevation is implicit in contextual bandit models (UCB with context features) and information retrieval query expansion (pseudo-relevance feedback). The specific formulation for an AI memory system is untested but logically derived. Q016's session retrieval set (Redis) provides the necessary infrastructure already. The additive structure of the formula (`base + α × activation`) is conservative and reversible — if α = 0, the formula degenerates to pure per-memory importance, making empirical calibration straightforward.
- **Feasibility**: 0.82 — The session retrieval set already exists in Redis (Q016 implementation). The additional steps: (1) when a memory is added to the session retrieval set, also store its cluster ID (Neo4j entity tag or cluster label already in the Qdrant payload) in a separate Redis hash `session:{id}:active_topics → {cluster_id: sum_of_base_importance}`; (2) in the retrieve endpoint's ranking step, look up each candidate's cluster in `session:{id}:active_topics` and compute `topic_activation = session_active_topics.get(cluster_id, 0.0)`; (3) apply `final_score = cosine_similarity × (base_importance + 0.2 × topic_activation)`. Redis lookup is O(1); the formula modification is arithmetic. Implementation time: 3-5 days beyond Q016.
- **Description**: Extend the session retrieval set (Redis) to also maintain an `active_topics` hash mapping cluster IDs to their aggregate session importance. Every time a memory is retrieved, increment its cluster's session activation score by the memory's base_importance. In the retrieve ranking formula, add an additive topic-activation term: `final_score = cosine_similarity × (base_importance + 0.2 × active_topics.get(cluster_id, 0.0))`. This causes the retrieval formula to progressively favor the topic cluster that the current session is actively working in, without discarding the per-memory discrimination needed to rank within the cluster. The boost is fully ephemeral — it evaporates at session end when the Redis key expires, leaving only the persistent per-memory scores to carry cross-session historical signal. Cold-start sessions receive no boost (α × 0 = 0); warm sessions receive progressively larger boosts as more cluster members are retrieved.

### Idea: dual-importance-scope-architecture

- **Novelty**: 0.72 — Explicitly separating importance into two scopes with different lifetimes — a persistent per-memory historical score and an ephemeral session-scoped topic-activation score — and combining them additively in the retrieval formula is not described in any known AI memory system. The architectural pattern of layering a fast-changing ephemeral signal over a slow-changing persistent signal is analogous to the L1/L2 cache hierarchy (fast small L1 over slow large L2) but applied to scoring rather than storage.
- **Evidence**: 0.60 — Multi-scale signal combination is standard in ensemble recommendation systems: a long-term user preference model (slow, stable) combined with a session-level engagement signal (fast, volatile) is the standard architecture for e-commerce personalization (YouTube, Netflix, Spotify). The specific application to a personal AI memory scoring formula is new but follows a well-established principle: combine signals at different timescales to capture both persistent and contextual relevance.
- **Feasibility**: 0.80 — Both components are already partially implemented: base_importance exists as the PostgreSQL importance score (to be improved by Q015's lazy decay), and the session retrieval set exists in Redis (Q016). The architecture formalizes their combination rather than adding new systems. The key implementation task is ensuring that the retrieve endpoint's ranking query can fetch both the per-memory base_importance from PostgreSQL and the cluster's session activation from Redis in a single round-trip. The Qdrant payload already stores cluster/entity tags, so no additional Neo4j query is needed in the hot path for cluster lookup. Implementation time: 1 week for the architecture formalization and retrieve endpoint modification.
- **Description**: Codify the dual-scope importance architecture as an explicit design decision in Recall's retrieval system. Persistent scope: `base_importance(M)` in PostgreSQL, updated by Q015's lazy geometric decay on each retrieval, carrying cross-session historical signal with τ = 14-day decay half-life. Ephemeral scope: `topic_activation(cluster(M), session)` in Redis, updated by each retrieval event in the current session, cleared at session end (Redis TTL). Retrieval formula: `final_score = cosine_similarity × (base_importance + α × topic_activation)` where α is tunable (start at 0.2). This architecture correctly answers the Q021 question: importance is both per-memory (persistent scope captures individual memory history) and topic-cluster-level (ephemeral scope captures current-session topic engagement). The two scopes are not in conflict — they operate at different timescales and serve different functions.

### Idea: topic-cluster-cold-start-prior

- **Novelty**: 0.68 — When the session-topic-activation boost is zero (session start, new topic), per-memory importance is the only signal, and historically-dominant topic clusters will outrank currently-relevant but historically-dormant clusters. A cold-start correction — applying a small prior boost to clusters whose entity-tag similarity to the current query is high, even before any retrieval has confirmed topic relevance — addresses the first-query gap.
- **Evidence**: 0.55 — Cold-start handling is a classic problem in recommendation systems. The standard approach is to use content-based features (item attributes) as a prior when collaborative-filtering signals are absent. The specific mechanism here — using entity-tag similarity between the query and cluster entity tags as a pre-retrieval cluster prior — is analogous to content-based cold-start but applied to memory clusters rather than items. Untested in AI memory systems but logically well-grounded.
- **Feasibility**: 0.60 — Requires extracting entity tags from the current query (a lightweight NER step on the query text, using the same entity extractor already deployed for Recall's observe-edit.js hook) and matching them against cluster entity tags stored in the Qdrant payload. The cluster entity tags are already stored (Neo4j uses them for SAME_FACT_FAMILY edges). The cold-start prior is `cluster_prior(M) = entity_overlap(query_entities, cluster_entities(M))`, a Jaccard coefficient between two small sets — O(|entities|) per candidate. Applied only when `topic_activation(cluster, session) == 0` (session cold for this cluster). Implementation time: 2-3 weeks (entity extraction from query text is the main addition; cluster entity tags already exist).
- **Description**: When the session has no prior retrievals for a cluster (topic_activation = 0), apply a small prior boost based on entity-tag overlap between the current query and the cluster's entity tags. Extract entities from the query string (same NER pipeline as observe-edit.js uses on file edits). Compute `cluster_prior = |query_entities ∩ cluster_entities| / |query_entities ∪ cluster_entities|` (Jaccard). Apply `final_score = cosine_similarity × (base_importance + β × cluster_prior)` where β is a small coefficient (0.1) that makes the cold-start prior weaker than a confirmed topic-activation signal (α = 0.2). This ensures that the first query in a new topic-context receives some elevation for semantically-matching clusters, bridging the gap between cold start and the first retrieval that fires the Q015 δ bump and the topic-activation boost.

### Idea: importance-scope-separation-eviction-vs-retrieval

- **Novelty**: 0.70 — The synthesis reveals a critical architectural fact that neither the Defender nor the Attacker in Q021 articulated: cluster-level importance is the correct granularity for eviction decisions (Q016's finding) while per-memory importance is the correct granularity for retrieval ranking. These are two separate operations that both use "importance" but need it at different scopes. Making this explicit — defining importance as a dual-scope concept with retrieval scope (per-memory) and eviction scope (per-cluster) — is a cleaner architecture than allowing importance to conflate both purposes.
- **Evidence**: 0.72 — Q016 explicitly established that the eviction unit is the cluster (not the individual memory): "if M is a cluster member: evaluate the cluster's eviction priority as the minimum eviction priority across all cluster members." This already implies that eviction uses cluster-level importance aggregation. Q015 established that retrieval uses per-memory importance. These two findings are in tension only if "importance" is treated as a single scalar — the tension dissolves if importance is explicitly dual-scope. The architectural pattern of using different granularities for different operations (page-level eviction, row-level retrieval in database buffer pools) is standard systems design.
- **Feasibility**: 0.85 — No new data structures required. Implementation: (1) define cluster_eviction_score as `MIN(base_importance) over cluster members` (already implied by Q016, can be computed at eviction time from the PostgreSQL retrieval-weighted score); (2) define retrieval_importance as `base_importance(M) + α × topic_activation(cluster, session)` (the dual-scope formula above); (3) document the distinction explicitly in code comments so future maintainers understand why two different importance computations exist. The PostgreSQL schema already has per-memory importance scores; the cluster minimum is a GROUP BY aggregation at eviction time. No new columns needed.
- **Description**: Formalize importance as explicitly dual-scope in Recall's architecture. Retrieval scope: `retrieval_importance(M) = base_importance(M) + α × topic_activation(cluster(M), session)` — per-memory historical score plus ephemeral session-topic boost. Eviction scope: `eviction_importance(cluster) = MIN(base_importance(M) for M in cluster)` — cluster-level minimum, used to decide whether an entire cluster should be evicted. These two computations answer different questions: retrieval importance answers "which individual memory should surface for this query?" (per-memory, contextually boosted); eviction importance answers "is this cluster cold enough to evict as a whole?" (cluster-level, historical, conservative). Separating them resolves the Q021 debate: both threads are right about their respective operations. Cluster-level importance is correct for eviction. Per-memory contextually-boosted importance is correct for retrieval.

## Suggested Follow-ups

- **[SIMULATION] Q022**: Model the session-topic-activation boost effect quantitatively. If α = 0.2 and a session retrieves 5 Docker memories with average base_importance = 0.4, the topic_activation = 5 × 0.4 = 2.0, and the boost = 0.2 × 2.0 = 0.4 — larger than the base_importance itself. Does this cause the boost to dominate the ranking, potentially crowding out semantically-relevant memories from other clusters? Derive the stability condition: the maximum α such that topic_activation cannot override the cosine_similarity signal in the worst case.

- **[IMPLEMENTATION] Q023**: The topic-activation boost requires storing cluster IDs in the Qdrant payload and in Redis during retrieval. Currently, Recall's Neo4j stores entity relationships but the retrieve endpoint does not query Neo4j in the hot path. Design the minimum-latency architecture for cluster ID lookup during retrieval: (a) embed cluster ID in Qdrant vector payload at store time (O(0) lookup at retrieve time, payload returned with ANN results), (b) query Neo4j synchronously (adds ~5ms), or (c) maintain a Redis cluster-membership cache. Evaluate each option's latency and consistency tradeoffs.

- **[ADVERSARIAL] Q024**: Does the dual-scope architecture create a new failure mode — "session capture" where an early retrieval in the wrong cluster permanently biases the session's topic-activation boost toward that cluster, even if the user pivots to a different topic mid-session? If the session-topic boost is additive and never decrements, a strong early cluster activation cannot be "un-activated." Design a session-topic-activation decay mechanism: should topic activation decay within a session (decay with each query that returns zero co-cluster results), or should it be reset on session-context pivot detection?

- **[ADJACENT] Q025**: The dual-scope importance architecture (persistent per-memory + ephemeral per-cluster) resembles the two-timescale learning in biological neural networks: slow synaptic weight updates (LTP/LTD, occurring over hours) combined with fast working memory buffers (hippocampal pattern completion, occurring within seconds). Does this analogy suggest additional signals? Specifically: does hippocampal-to-cortical consolidation (slow wave sleep replay) suggest that the session-topic-activation boost should be "consolidated" into per-memory base_importance scores during the session-end consolidation job — not discarded entirely, but partially written back as a small δ to the base scores of cluster members that were active during the session?
