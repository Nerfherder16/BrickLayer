## API Endpoint Design

All endpoints are added to the Recall FastAPI service at `192.168.50.19:8200`. Each is grouped under the `/api/tags` router prefix. The design follows Tim's existing FastAPI patterns: Pydantic models for request/response, dependency-injected database sessions, structured error responses.

```python
# router: app/routers/tags.py
# Mount with: app.include_router(tags_router, prefix="/api/tags", tags=["Tag Governance"])

from fastapi import APIRouter, Depends, HTTPException, BackgroundTasks
from pydantic import BaseModel, Field
from typing import Optional
from datetime import datetime

router = APIRouter()


# ─── Request / Response models ───────────────────────────────────────

class SynonymCreate(BaseModel):
    raw_form:  str  = Field(..., min_length=1, max_length=100)
    canonical: str  = Field(..., min_length=1, max_length=100)
    notes:     Optional[str] = None
    confirmed: bool = True


class SynonymResponse(BaseModel):
    raw_form:  str
    canonical: str
    added_at:  datetime
    added_by:  str
    confirmed: bool
    notes:     Optional[str]


class CanonicalCreate(BaseModel):
    tag:         str  = Field(..., min_length=1, max_length=100)
    description: str  = Field(default='')


class MigrationResult(BaseModel):
    memories_updated: int
    elapsed_ms:       float
    synonyms_applied: list[str]


class RollbackResult(BaseModel):
    synonym_removed:  str
    memories_reverted: int
    elapsed_ms:        float


# ─── GET /api/tags/synonyms ──────────────────────────────────────────
# Returns the full normalization map as {raw_form: canonical}.
# This is the endpoint the hook polls every 60s. Only confirmed=TRUE
# entries are returned — unconfirmed (auto-detected) entries do not
# affect normalization until Tim reviews and confirms them.

@router.get("/synonyms")
async def get_synonyms(db=Depends(get_db)) -> dict[str, str]:
    rows = await db.fetch(
        "SELECT raw_form, canonical FROM tag_synonyms WHERE confirmed = TRUE"
    )
    return {row["raw_form"]: row["canonical"] for row in rows}


# ─── GET /api/tags/canonical ─────────────────────────────────────────
# Returns the ordered canonical tag list for LLM prompt construction.
# Hook caches this separately with the same 60s TTL.

@router.get("/canonical")
async def get_canonical_tags(db=Depends(get_db)) -> list[str]:
    rows = await db.fetch(
        "SELECT tag FROM tag_canonical WHERE active = TRUE ORDER BY tag"
    )
    return [row["tag"] for row in rows]


# ─── POST /api/tags/synonyms ─────────────────────────────────────────
# Adds a new synonym entry. Validates that canonical exists in
# tag_canonical. If confirmed=False, entry is stored but excluded
# from the hook's normalization map until manually confirmed.
# IMPORTANT: Does NOT automatically trigger migration. Tim must call
# POST /api/tags/migrate-synonyms separately (or use dashboard button).
# This is intentional: decouple synonym addition from retroactive write
# so Tim can review before committing the corpus update.

@router.post("/synonyms", response_model=SynonymResponse, status_code=201)
async def add_synonym(body: SynonymCreate, db=Depends(get_db)) -> SynonymResponse:
    # Validate canonical exists
    canonical_row = await db.fetchrow(
        "SELECT tag FROM tag_canonical WHERE tag = $1 AND active = TRUE",
        body.canonical
    )
    if not canonical_row:
        raise HTTPException(
            status_code=422,
            detail=f"Canonical tag '{body.canonical}' not found in tag_canonical. "
                   f"Add it first via POST /api/tags/canonical."
        )
    # Upsert: if the raw_form already maps elsewhere, update it
    row = await db.fetchrow(
        """
        INSERT INTO tag_synonyms (raw_form, canonical, added_by, confirmed, notes)
        VALUES ($1, $2, 'human', $3, $4)
        ON CONFLICT (raw_form) DO UPDATE
            SET canonical  = EXCLUDED.canonical,
                confirmed  = EXCLUDED.confirmed,
                notes      = EXCLUDED.notes,
                added_at   = NOW(),
                added_by   = 'human'
        RETURNING *
        """,
        body.raw_form, body.canonical, body.confirmed, body.notes
    )
    return SynonymResponse(**dict(row))


# ─── POST /api/tags/canonical ────────────────────────────────────────
# Adds a new canonical tag. On success, both nodes pick up the new
# tag in the LLM prompt menu within 60s (next cache TTL expiry).

@router.post("/canonical", status_code=201)
async def add_canonical_tag(body: CanonicalCreate, db=Depends(get_db)) -> dict:
    await db.execute(
        """
        INSERT INTO tag_canonical (tag, description, added_by)
        VALUES ($1, $2, 'human')
        ON CONFLICT (tag) DO UPDATE
            SET description = EXCLUDED.description,
                active      = TRUE
        """,
        body.tag, body.description
    )
    return {"tag": body.tag, "status": "created"}


# ─── POST /api/tags/migrate-synonyms ────────────────────────────────
# RETROACTIVE MIGRATION: Updates tags[] in the memories table so that
# every raw synonym form stored in tags[] is replaced with its canonical.
# Runs inside a single PostgreSQL transaction (atomic — no partial state).
# tags_raw is NOT modified (it is the immutable audit trail).
#
# This is the "dashboard button" operation. Expected runtime: < 500ms
# at 2,000 memories. Returns {memories_updated, elapsed_ms}.
#
# Tim calls this after adding new synonyms. The hook cache means new
# memories are already being written with canonical tags; this call
# fixes the historical memories from the transition window.

@router.post("/migrate-synonyms", response_model=MigrationResult)
async def migrate_synonyms(db=Depends(get_db)) -> MigrationResult:
    import time
    start = time.monotonic()

    # Fetch all confirmed synonyms (only confirmed entries are migrated)
    synonyms = await db.fetch(
        "SELECT raw_form, canonical FROM tag_synonyms WHERE confirmed = TRUE"
    )
    if not synonyms:
        return MigrationResult(memories_updated=0, elapsed_ms=0.0, synonyms_applied=[])

    raw_forms = [r["raw_form"] for r in synonyms]

    # Single-transaction UPDATE using a CASE expression over the synonym map.
    # The WHERE clause uses the GIN index on tags[] for the initial candidate set.
    # The ARRAY_AGG(DISTINCT ...) deduplicates in case a memory had both the
    # raw form and canonical form already (e.g., both 'postgres' and 'postgresql').
    result = await db.fetchrow(
        """
        WITH synonym_map(raw_form, canonical) AS (
            SELECT raw_form, canonical
            FROM tag_synonyms
            WHERE confirmed = TRUE
        ),
        updated AS (
            UPDATE memories m
            SET tags = (
                SELECT ARRAY_AGG(DISTINCT COALESCE(sm.canonical, t.tag))
                FROM UNNEST(m.tags) AS t(tag)
                LEFT JOIN synonym_map sm ON sm.raw_form = t.tag
            )
            WHERE m.tags && $1::text[]
            RETURNING 1
        )
        SELECT COUNT(*) AS n FROM updated
        """,
        raw_forms
    )

    elapsed_ms = (time.monotonic() - start) * 1000
    return MigrationResult(
        memories_updated=result["n"],
        elapsed_ms=round(elapsed_ms, 1),
        synonyms_applied=raw_forms
    )


# ─── POST /api/tags/rollback-synonym ────────────────────────────────
# ROLLBACK: Removes an incorrect synonym and re-derives tags[] from
# tags_raw for all affected memories. This is the Q078 procedure.
#
# Flow:
#   1. Remove the synonym entry from tag_synonyms
#   2. Find all memories where tags_raw contains the now-removed raw_form
#   3. Re-run normalization from tags_raw using the current (corrected) map
#   4. Update tags[] with the re-derived canonical set
#
# The cache on both hooks will expire within 60s, but for incorrect mappings
# that need immediate correction, call POST /api/tags/cache-invalidate after.

@router.post("/rollback-synonym", response_model=RollbackResult)
async def rollback_synonym(raw_form: str, db=Depends(get_db)) -> RollbackResult:
    import time
    start = time.monotonic()

    async with db.transaction():
        # Step 1: Remove the synonym
        deleted = await db.fetchrow(
            "DELETE FROM tag_synonyms WHERE raw_form = $1 RETURNING raw_form",
            raw_form
        )
        if not deleted:
            raise HTTPException(status_code=404, detail=f"Synonym '{raw_form}' not found")

        # Step 2: Re-derive tags[] from tags_raw for affected memories
        # Only memories that have tags_raw populated can be reprocessed.
        # Memories without tags_raw (pre-dual-column) are left unchanged.
        result = await db.fetchrow(
            """
            WITH current_map(raw_form, canonical) AS (
                SELECT raw_form, canonical FROM tag_synonyms WHERE confirmed = TRUE
            ),
            updated AS (
                UPDATE memories m
                SET tags = (
                    SELECT ARRAY_AGG(DISTINCT COALESCE(cm.canonical, t.tag))
                    FROM UNNEST(m.tags_raw) AS t(tag)
                    LEFT JOIN current_map cm ON cm.raw_form = t.tag
                )
                WHERE m.tags_raw IS NOT NULL
                  AND m.tags_raw && ARRAY[$1]::text[]
                RETURNING 1
            )
            SELECT COUNT(*) AS n FROM updated
            """,
            raw_form
        )

    elapsed_ms = (time.monotonic() - start) * 1000
    return RollbackResult(
        synonym_removed=raw_form,
        memories_reverted=result["n"],
        elapsed_ms=round(elapsed_ms, 1)
    )


# ─── POST /api/tags/cache-invalidate ────────────────────────────────
# Bumps the cache generation counter stored in PostgreSQL.
# Hooks check this counter on each poll; if it has advanced past their
# cached generation, they invalidate immediately rather than waiting for TTL.
# See Hook Cache Implementation section for the hook-side check.

@router.post("/cache-invalidate")
async def invalidate_cache(db=Depends(get_db)) -> dict:
    await db.execute(
        """
        INSERT INTO tag_cache_generation (id, generation, updated_at)
        VALUES (1, 1, NOW())
        ON CONFLICT (id) DO UPDATE
            SET generation = tag_cache_generation.generation + 1,
                updated_at = NOW()
        """
    )
    return {"status": "invalidated"}
```

## Hook Cache Implementation

The following is the complete 60-second TTL cache implementation for `observe-edit.js`. Two caches are maintained in parallel: the normalization map (`raw_form → canonical`) and the canonical tag list (for LLM prompt construction). Both share the same TTL and the same generation-check mechanism.

```javascript
// ════════════════════════════════════════════════════════════════════
// TAG GOVERNANCE CACHE — observe-edit.js
// Module-level state. Persists for the lifetime of the hook process
// (one hook invocation = one process in Claude Code's current model,
// but the module is re-used if the process is kept alive).
// ════════════════════════════════════════════════════════════════════

const RECALL_API   = 'http://192.168.50.19:8200';
const CACHE_TTL_MS = 60_000;  // 60 seconds

// Normalization map: { raw_form: canonical_tag, ... }
let _normMap        = null;
let _normMapExpiry  = 0;
let _normMapGen     = -1;  // last known cache generation from server

// Canonical tag list: string[] — used in LLM prompt construction
let _canonicalList       = null;
let _canonicalListExpiry = 0;

// Cache generation: lightweight check to detect server-side invalidation
// before TTL expiry (used after rollback operations)
let _genCheckExpiry = 0;
const GEN_CHECK_INTERVAL_MS = 5_000;  // check generation every 5s (cheap read)


// ─── getNormalizationMap ──────────────────────────────────────────
// Returns the confirmed normalization map as a plain object.
// Uses module-level cache with 60s TTL. On cache miss, fetches from
// the Recall API. Both casaclaude and proxyclaude hit the same endpoint.
//
// Generation-check: every 5 seconds, the hook fetches the current
// generation counter from PostgreSQL via a lightweight endpoint. If the
// server generation has advanced (e.g., after a rollback), the cache is
// immediately invalidated regardless of TTL. This bounds incorrect-mapping
// exposure to 5 seconds rather than 60 seconds when a rollback is triggered.

async function getNormalizationMap() {
  const now = Date.now();

  // 1. Check generation (cheap, every 5s)
  if (now >= _genCheckExpiry) {
    try {
      const genResp = await fetch(`${RECALL_API}/api/tags/cache-generation`, {
        signal: AbortSignal.timeout(1000)  // 1s timeout — fail fast
      });
      if (genResp.ok) {
        const { generation } = await genResp.json();
        if (generation !== _normMapGen) {
          // Server-side invalidation detected: bust cache immediately
          _normMap       = null;
          _normMapExpiry = 0;
          _normMapGen    = generation;
        }
      }
    } catch (_e) {
      // Generation check failure is non-fatal: fall through to TTL logic.
      // The hook continues with the cached map (stale by at most 60s).
    }
    _genCheckExpiry = now + GEN_CHECK_INTERVAL_MS;
  }

  // 2. Return cached map if still valid
  if (_normMap && now < _normMapExpiry) {
    return _normMap;
  }

  // 3. Cache miss (TTL expired or invalidated): fetch from API
  try {
    const resp = await fetch(`${RECALL_API}/api/tags/synonyms`, {
      signal: AbortSignal.timeout(3000)  // 3s timeout
    });
    if (!resp.ok) throw new Error(`HTTP ${resp.status}`);
    _normMap       = await resp.json();  // { raw_form: canonical, ... }
    _normMapExpiry = now + CACHE_TTL_MS;
  } catch (err) {
    // Network failure: return stale cache if available, else empty map.
    // Failing open (empty map) means tags pass through unnormalized,
    // which is preferable to dropping the memory entirely.
    if (_normMap) {
      console.warn('[observe-edit] tag map fetch failed, using stale cache:', err.message);
      _normMapExpiry = now + CACHE_TTL_MS;  // extend TTL to avoid hammering API
      return _normMap;
    }
    console.warn('[observe-edit] tag map fetch failed, no cache available — tags unnormalized');
    return {};
  }

  return _normMap;
}


// ─── getCanonicalTagList ──────────────────────────────────────────
// Returns the ordered canonical tag list for LLM prompt construction.
// Same 60s TTL pattern as getNormalizationMap.

async function getCanonicalTagList() {
  const now = Date.now();
  if (_canonicalList && now < _canonicalListExpiry) {
    return _canonicalList;
  }
  try {
    const resp = await fetch(`${RECALL_API}/api/tags/canonical`, {
      signal: AbortSignal.timeout(3000)
    });
    if (!resp.ok) throw new Error(`HTTP ${resp.status}`);
    _canonicalList       = await resp.json();  // string[]
    _canonicalListExpiry = now + CACHE_TTL_MS;
  } catch (err) {
    if (_canonicalList) {
      console.warn('[observe-edit] canonical list fetch failed, using stale cache');
      _canonicalListExpiry = now + CACHE_TTL_MS;
      return _canonicalList;
    }
    // Hardcoded fallback: the Q066 canonical set. Prevents total LLM prompt
    // degradation if the API is unreachable at startup.
    console.warn('[observe-edit] canonical list fetch failed, using hardcoded fallback');
    return [
      'casaos','docker','fastapi','homeassistant','homelab','jellyfin',
      'networking','neo4j','ollama','postgresql','proxmox','python',
      'qdrant','react','recall','redis','ssh','typescript',
      'automation','embedding','gpu','general'
    ];
  }
  return _canonicalList;
}


// ─── normalizeTags ────────────────────────────────────────────────
// Applies the normalization map to an array of raw LLM-extracted tags.
// Returns deduplicated canonical forms.

async function normalizeTags(rawTags) {
  const normMap = await getNormalizationMap();
  const seen    = new Set();
  const result  = [];
  for (const tag of rawTags) {
    const t          = tag.toLowerCase().trim();
    const canonical  = normMap[t] ?? t;
    if (canonical && !seen.has(canonical)) {
      seen.add(canonical);
      result.push(canonical);
    }
  }
  return result;
}


// ─── buildTagExtractionPrompt ─────────────────────────────────────
// Constructs the LLM tag extraction prompt dynamically from the
// canonical tag list. Called once per store event (prompt is cheap;
// the LLM call is the expensive part).

async function buildTagExtractionPrompt(memoryContent) {
  const tags = await getCanonicalTagList();
  const menu  = tags.filter(t => t !== 'general').join(', ');
  return `Extract 2-4 tags from the following memory content.
Use ONLY tags from this canonical list: ${menu}, general.
If the content clearly relates to multiple domains, include up to 4 tags.
If no specific domain applies, use 'general'.
Return tags as a JSON array of strings. Example: ["docker", "casaos"]

Memory content:
${memoryContent}`;
}
```

## Drift Detection Job

The drift detection job runs daily via cron (or the Recall background task runner if one exists). It finds raw tags that are accumulating in the corpus without a synonym mapping, then stores a governance alert as a Recall memory so it surfaces organically during Tim's next session.

### SQL query

```sql
-- ════════════════════════════════════════════════════════════════════
-- DRIFT DETECTION: unmapped raw tags appearing >= 3 times
-- Source: memories.tags (current normalized column)
--   In the dual-column design, tags[] contains canonical or near-canonical
--   forms for post-migration memories. For pre-migration memories in the
--   transition window, tags[] may still contain raw forms.
--   tags_raw[] contains ALL raw forms for ALL memories (once back-populated).
--
-- PRIMARY QUERY: scan tags_raw for unmapped forms
--   tags_raw is the audit column that always holds original LLM output.
--   This catches unmapped forms even for memories where tags[] has already
--   been migrated to canonical.
-- ════════════════════════════════════════════════════════════════════

WITH
-- Step 1: unnest tags_raw across all memories
raw_tag_counts AS (
    SELECT
        t.tag,
        COUNT(*) AS frequency
    FROM memories m,
         UNNEST(m.tags_raw) AS t(tag)
    WHERE m.tags_raw IS NOT NULL
    GROUP BY t.tag
),
-- Step 2: filter to tags that are neither a canonical tag nor a known synonym
unmapped AS (
    SELECT
        rtc.tag,
        rtc.frequency
    FROM raw_tag_counts rtc
    WHERE rtc.tag NOT IN (SELECT tag      FROM tag_canonical WHERE active = TRUE)
      AND rtc.tag NOT IN (SELECT raw_form FROM tag_synonyms  WHERE confirmed = TRUE)
      AND rtc.frequency >= 3
),
-- Step 3: exclude tags already in the drift log as pending
new_drift AS (
    SELECT u.tag, u.frequency
    FROM unmapped u
    WHERE u.tag NOT IN (
        SELECT raw_tag FROM tag_drift_log WHERE status = 'pending'
    )
)
SELECT tag, frequency
FROM new_drift
ORDER BY frequency DESC;
```

### Python job implementation

```python
# app/jobs/drift_detection.py
# Schedule: daily at 03:00 (low-traffic window)
# Invocation: cron entry or APScheduler job in the Recall FastAPI startup

import json
import logging
from datetime import datetime, timezone

log = logging.getLogger(__name__)

RECALL_API_INTERNAL = "http://localhost:8200"  # internal call, no network hop


async def run_drift_detection(db):
    """
    Daily job: detect unmapped raw tags, log to tag_drift_log, and
    store a Recall memory in the recall-governance domain so Tim's
    next session surfaces the governance alert via MBC routing.
    """
    drift_sql = """
    WITH
    raw_tag_counts AS (
        SELECT t.tag, COUNT(*) AS frequency
        FROM memories m, UNNEST(m.tags_raw) AS t(tag)
        WHERE m.tags_raw IS NOT NULL
        GROUP BY t.tag
    ),
    unmapped AS (
        SELECT rtc.tag, rtc.frequency
        FROM raw_tag_counts rtc
        WHERE rtc.tag NOT IN (SELECT tag      FROM tag_canonical WHERE active = TRUE)
          AND rtc.tag NOT IN (SELECT raw_form FROM tag_synonyms  WHERE confirmed = TRUE)
          AND rtc.frequency >= 3
    ),
    new_drift AS (
        SELECT u.tag, u.frequency
        FROM unmapped u
        WHERE u.tag NOT IN (
            SELECT raw_tag FROM tag_drift_log WHERE status = 'pending'
        )
    )
    SELECT tag, frequency FROM new_drift ORDER BY frequency DESC
    """

    rows = await db.fetch(drift_sql)

    if not rows:
        log.info("[drift-detection] No new unmapped tags found.")
        return

    # Insert into audit log
    for row in rows:
        await db.execute(
            """
            INSERT INTO tag_drift_log (raw_tag, frequency, status)
            VALUES ($1, $2, 'pending')
            ON CONFLICT DO NOTHING
            """,
            row["tag"], row["frequency"]
        )

    # Compose the governance alert memory
    tag_list = "\n".join(
        f"  - '{r['tag']}' ({r['frequency']} occurrences)"
        for r in rows
    )
    total_unmapped = len(rows)
    alert_content = f"""[TAG GOVERNANCE ALERT] Drift detection job found {total_unmapped} unmapped raw tag(s) appearing >=3 times in the corpus (detected {datetime.now(timezone.utc).strftime('%Y-%m-%d')}):

{tag_list}

ACTION REQUIRED: Review these tags and either:
  (a) Add as synonym of an existing canonical tag via dashboard Tag Vocabulary panel
      → POST /api/tags/synonyms  {{raw_form, canonical}}
  (b) Add as a new canonical tag if it represents a genuinely new domain
      → POST /api/tags/canonical {{tag, description}}
  (c) Dismiss if the tag is a one-off artifact with no semantic value

After mapping, run: POST /api/tags/migrate-synonyms (dashboard button)
to apply the new mappings retroactively to existing memories.

MBC IMPACT: Unmapped tags in tags[] degrade Level 1 routing for sessions
where these terms appear in context. Severity is proportional to frequency."""

    # Store the alert as a Recall memory in the recall-governance domain
    # Tags: ['recall', 'vocabulary-drift'] ensure it surfaces via MBC Level 1
    # when Tim is in a recall-related session, and via Level 2 (domain) for
    # recall-governance domain.
    alert_memory = {
        "content":  alert_content,
        "domain":   "recall-governance",
        "tags":     ["recall", "vocabulary-drift"],
        "tags_raw": ["recall", "vocabulary-drift"],
        "source":   "drift-detection-job",
        "metadata": {
            "unmapped_count":  total_unmapped,
            "detected_at":     datetime.now(timezone.utc).isoformat(),
            "top_tag":         rows[0]["tag"],
            "top_frequency":   rows[0]["frequency"]
        }
    }

    # Direct DB insert (avoid HTTP call from within the same process)
    await db.execute(
        """
        INSERT INTO memories (content, domain, tags, tags_raw, source, metadata, created_at)
        VALUES ($1, $2, $3, $4, $5, $6, NOW())
        """,
        alert_memory["content"],
        alert_memory["domain"],
        alert_memory["tags"],
        alert_memory["tags_raw"],
        alert_memory["source"],
        json.dumps(alert_memory["metadata"])
    )

    log.info(
        "[drift-detection] Stored governance alert memory. "
        "Unmapped tags: %d. Top: '%s' (%d occurrences).",
        total_unmapped, rows[0]["tag"], rows[0]["frequency"]
    )
```

## GIN Index Behavior

This section addresses the critical question: does a GIN index on `tags[]` (canonical forms) coexist correctly with the dual-column design (`tags` + `tags_raw`), and can it break Level 1 MBC routing during the transition window?

### GIN index scope and column binding

A PostgreSQL GIN index is bound to a specific column at creation time:

```sql
CREATE INDEX idx_memories_tags_gin ON memories USING GIN(tags);
```

This index indexes **only** the `tags` column. It has zero knowledge of `tags_raw`. The `tags_raw` column's contents are invisible to this index. Adding `tags_raw` to the schema does not affect the GIN index in any way — no rebuild, no invalidation, no performance change.

**Verdict**: GIN on `tags` and `tags_raw` (array column with no index) coexist without any interaction. No index conflict. No routing interference.

### What the GIN index does and does not index

| Content | In `tags[]` | In `tags_raw[]` | Indexed by GIN |
|:--------|:-----------:|:---------------:|:--------------:|
| Canonical forms (post-migration) | YES | NO | YES |
| Raw forms (pre-migration or transition window) | YES (until migrated) | YES | YES (for `tags[]` content only) |
| Raw forms in `tags_raw` only | NO | YES | NO |

The GIN index is on `tags[]`. If `tags[]` contains raw forms (transition window), the GIN index correctly returns those memories for queries that include the matching raw form in the WHERE predicate. But MBC Level 1 routing queries use canonical forms:

```sql
WHERE m.tags && ARRAY['docker', 'casaos']::text[]
```

If a memory was written with `tags = ['dockerfile', 'casaos']` (transition window, not yet migrated), the GIN lookup for `'docker'` returns no match because `'dockerfile' != 'docker'`. The GIN index is functioning correctly — it is correctly reporting that the canonical form `docker` is not in that memory's `tags[]` array. The routing miss is not an index bug; it is the correct behavior of an un-migrated memory.

### The transition window routing gap — is it a BREAKTHROUGH?

The gap is this: during the transition window (Phase 1 and Phase 2 from Q075), memories with raw tags in `tags[]` are correctly excluded from Level 1 routing because the GIN index faithfully reflects what is stored. The index does not lie. The problem is that the stored data is not yet canonical.

**This confirms the Q075 BREAKTHROUGH finding**: the routing degradation during the transition window is a data correctness problem, not an indexing problem. The GIN index behaves exactly as designed. Running `POST /api/tags/migrate-synonyms` immediately after adding new synonyms (via the dashboard button) is the only correct resolution.

### Does GIN on `tags` need a companion index on `tags_raw`?

No GIN index on `tags_raw` is warranted. Here is why:

1. `tags_raw` is never used in MBC routing WHERE predicates. It is an audit column.
2. `tags_raw` is only accessed in the rollback endpoint (`WHERE tags_raw && ARRAY[$1]`), which is an infrequent administrative operation. A sequential scan on 2,000-10,000 rows for this case takes < 10ms — acceptable for an admin operation.
3. A GIN index on `tags_raw` would consume extra disk space (~same size as the `tags` GIN index, roughly 200KB-2MB at Tim's scale) with no query performance benefit for the primary read path.
4. The correct solution to routing misses is migration, not indexing `tags_raw`.

**If a GIN index on `tags_raw` were added**, it would not break anything. It would simply be unused overhead. The MBC routing query will always target `tags[]` because that is the canonical column.

### Partial index consideration for transition window

One refinement worth noting: during a prolonged transition window, it is possible to create a temporary diagnostic index to measure how many memories have un-migrated raw forms:

```sql
-- Diagnostic only — not for production use
-- Returns count of memories where tags[] contains any known raw synonym
SELECT COUNT(*) AS un_migrated
FROM memories
WHERE tags && (SELECT ARRAY_AGG(raw_form) FROM tag_synonyms WHERE confirmed = TRUE);
```

This query uses the existing GIN index on `tags[]` and runs in O(log N + K) time. It quantifies the size of the transition window residue without requiring a schema change.

### Summary of GIN coexistence verdict

| Concern | Verdict |
|:--------|:-------:|
| GIN on `tags` broken by adding `tags_raw` column | No impact — GIN is column-scoped |
| GIN on `tags` indexes raw forms during transition window | Correctly indexes whatever is in `tags[]` — raw or canonical |
| Raw forms in `tags_raw` incorrectly matched by Level 1 routing | Not possible — routing only queries `tags[]` |
| GIN index needed on `tags_raw` | Not warranted |
| Routing miss for un-migrated memories | Correct GIN behavior; fix by running migrate-synonyms |
| **Critical gap**: GIN cannot compensate for un-migrated data | **CONFIRMED — this is the BREAKTHROUGH** |

The BREAKTHROUGH classification is validated: the GIN index cannot distinguish between "this memory has been migrated to canonical tags" and "this memory still carries raw forms." During the transition window, Level 1 routing silently misses memories that are conceptually relevant but stored with non-canonical tags. The only defense is a fast, safe retroactive migration (< 500ms) triggered immediately after each synonym addition.

## Ideas Extracted

### Idea: generation-counter-cache-invalidation-for-incorrect-synonym-rollback
- **Novelty**: 0.76
- **Evidence**: 0.65
- **Description**: Adding a `tag_cache_generation` singleton table (PostgreSQL, single row) alongside the 60s TTL cache changes the failure mode for incorrect synonym mappings from "60 seconds of wrong canonicalization" to "5 seconds." The hook polls the generation counter every 5 seconds (a ~0.1ms PostgreSQL read vs the 2-5ms full synonym map fetch). On generation advance — triggered by `POST /api/tags/cache-invalidate` or `POST /api/tags/rollback-synonym` — the hook busts its local map cache immediately. This solves the Q078 rollback latency problem (identified as a follow-up in Q075) without requiring push notifications or websockets. The generation counter pattern is a standard technique in CDN cache invalidation applied here to a process-local in-memory cache in a Node.js hook.

### Idea: confirmed-flag-on-tag-synonyms-for-auto-detected-entries
- **Novelty**: 0.68
- **Evidence**: 0.72
- **Description**: The `confirmed` boolean column on `tag_synonyms` (default TRUE for human-added, FALSE for auto-detected by drift detection) creates a two-tier vocabulary governance model. Auto-detected synonyms are stored in the database (and shown in the dashboard) but excluded from the normalization map until Tim reviews them. This prevents the drift detection job from autonomously modifying the normalization map (which would affect all future memory writes without human approval) while still making the detected candidates immediately visible. The pattern mirrors pull request review gates in code governance: the change is proposed and visible, but does not take effect until explicitly approved.

### Idea: tags-raw-as-reprocessing-substrate-for-normalization-correction
- **Novelty**: 0.70
- **Evidence**: 0.68
- **Description**: The `tags_raw` column enables a complete normalization reprocessing pipeline without re-invoking the LLM. When a synonym mapping is corrected (e.g., `ha` was incorrectly mapped to `homeassistant` but some occurrences meant high-availability), the rollback endpoint re-derives `tags[]` from `tags_raw[]` using the corrected map. This is the database analog of storing source code (tags_raw = source) alongside compiled output (tags = binary) — when the compiler (normalization map) is updated, the source can be recompiled without re-running the original program (LLM inference). At Tim's scale, this reprocessing costs < 100ms for a full corpus re-derivation from tags_raw.

## Key Finding

The implementation is complete and internally consistent, but the analysis reveals a critical BREAKTHROUGH gap that was partially identified in Q075 and is now fully specified: **the GIN index on `tags[]` cannot distinguish migrated from un-migrated memories, and will silently return zero Level 1 hits for any memory whose `tags[]` array still contains raw synonym forms during the transition window.** This is not an index defect — it is correct behavior. The GIN index faithfully reports what is in `tags[]`. The routing failure is a data state problem: raw forms in `tags[]` are semantically equivalent to canonical forms but lexically different, and the GIN `&&` operator performs exact array element matching with no synonym awareness.

The governance architecture resolves this correctly: `POST /api/tags/migrate-synonyms` runs in < 500ms for Tim's 2,000-memory corpus and must be called immediately after every synonym addition. The dashboard button makes this a single-click operation. The dual-column design (`tags_raw` + `tags`) ensures the migration is always reversible — incorrect mappings can be rolled back by re-deriving `tags[]` from the immutable `tags_raw[]` without re-invoking the LLM.

Three implementation details required specification that Q075 left implicit: (1) the `confirmed` flag on `tag_synonyms` is essential to prevent the daily drift detection job from autonomously altering the normalization map — auto-detected entries must be staged for human review before activation; (2) the generation-counter cache invalidation pattern (`tag_cache_generation` table + 5s poll) reduces incorrect-mapping exposure from 60 seconds to 5 seconds after a rollback; (3) `tags_raw` must be present and back-populated before the drift detection job runs, because the job queries `tags_raw` (not `tags`) to find unmapped forms — scanning `tags[]` after migration would find only canonical forms and return zero results, making the detection job useless.

The implementation is BREAKTHROUGH rather than PROMISING because deploying the dual-column schema without the `tags_raw` back-population step, or deploying the drift detection job before `tags_raw` is populated, would silently produce a governance system that appears functional but generates no alerts — a false sense of safety during the period of highest vocabulary drift risk.
