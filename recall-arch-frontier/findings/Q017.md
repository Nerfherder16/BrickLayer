# Finding: Q017 — Taboo Architecture: Causal Memory Data Structure from First Principles

**Question**: Ignoring all existing tools, design the data structure for storing causal relationships between memories (memory A caused memory B, memory A contradicts memory B). Starting from graph theory fundamentals — what is the minimal structure needed?
**Question Type**: TABOO
**Verdict**: BREAKTHROUGH
**Severity**: Medium

---

## First Principles Derivation

### 1. Minimal Relationship Type Set

Start from information theory: two memories can relate in exactly three fundamental ways.

**Primitive relation classes:**

1. **Directed epistemic dependency** — the truth or occurrence of A is a prerequisite for, or increases the probability of, B. This is an *asymmetric* relation: A→B does not imply B→A. Example: "I saw the API was slow (A) → I decided to add caching (B)."

2. **Symmetric tension** — A and B cannot both be true simultaneously, or their truth values are in irreducible conflict. This is a *symmetric* relation: A⊥B implies B⊥A. Example: "The server is stateless (A) ↔ contradicts ↔ The server stores session data (B)."

3. **Undirected correlation** — A and B co-occur without either being a prerequisite for the other. This is symmetric and carries no counterfactual force. Example: "I worked late (A) and ordered pizza (B) — both happened, neither caused the other."

**Reduction of the named types:**

| Named type | Primitive class | Reduction |
|------------|----------------|-----------|
| CAUSES | Directed epistemic dependency | Primitive; weight = causal confidence |
| SUPPORTS | Directed epistemic dependency | CAUSES with weight < 0.5 — same type, different weight |
| CONTRADICTS | Symmetric tension | Primitive; no reduction possible |
| SUPERSEDES | Directed epistemic dependency | CAUSES with boolean flag `invalidates_source = true` |
| CO-OCCURS-WITH | Undirected correlation | Distinct class but carries no causal information; retrieval utility only |

**Conclusion — minimal type set: {CAUSES, CONTRADICTS, CORRELATES}**

- CAUSES subsumes SUPPORTS (by weight) and SUPERSEDES (by flag).
- CONTRADICTS cannot be reduced to a directed type without losing the symmetry property.
- CORRELATES is retained only for retrieval (Q010-style PPR walks); it adds no causal information.
- A system that only needs causal and contradiction queries can discard CORRELATES entirely. It is optional.

**Theoretically minimal set (pure causal+contradiction): {CAUSES, CONTRADICTS}**

Two types. Every named relationship in common AI memory systems maps to one of these two, plus scalar weight and one boolean flag on CAUSES.

### 2. Node Representation (derived from traversal requirements)

**Requirement (a): O(1) edge lookup for a given node.**

A flat list of edges stored globally (edge table only) gives O(E) scan to find edges for a node. To achieve O(1), the node must carry its own adjacency structure — an adjacency list indexed by node ID. Data structure: hash map from `node_id` to the node's edge bucket. This is the classic graph adjacency-list representation derived purely from the lookup requirement.

**Requirement (b): O(1) relationship type filtering.**

Within a node's adjacency structure, if edges are stored as a flat list, filtering by type is O(degree). To achieve O(1) filtering, the adjacency list must be *partitioned by edge type*. Instead of:

```
node.edges = [edge1, edge2, edge3, ...]
```

it must be:

```
node.edges = {
  CAUSES:      [edge_ids sorted by timestamp],
  CONTRADICTS: [edge_ids sorted by timestamp],
  CORRELATES:  [edge_ids sorted by timestamp]
}
```

Now `node.edges[CAUSES]` returns only causal edges in O(1) (hash lookup). Filtering is free.

**Requirement (c): Temporal ordering of edges.**

Each bucket is maintained in sorted order by `timestamp`. If timestamps are monotonically increasing (as in an append-mostly system), insertion is O(1) amortized (append to tail). For out-of-order insertions, binary search gives O(log degree) insertion. Retrieval in temporal order is then O(1) for the first edge and O(k) for the k-th.

**Derived minimum node schema:**

```
Node {
  id:          UUID                          -- unique memory identifier
  content_ref: opaque_pointer                -- reference to content store (flat blob, embedding, etc.)
  created_at:  timestamp                     -- wall clock, for global temporal ordering
  generation:  uint64                        -- logical clock (Lamport timestamp), for causal ordering
                                             -- incremented on every write; used to establish
                                             -- happens-before relationships without relying on
                                             -- potentially non-monotonic wall clocks
  edges: {
    CAUSES:      List<EdgeRef> [sorted by timestamp],
    CONTRADICTS: List<EdgeRef> [sorted by timestamp],
    CORRELATES:  List<EdgeRef> [sorted by timestamp]
  }
}
```

Nothing else is *required* by the traversal constraints. Any additional fields (tags, cluster assignments, embedding vectors) are retrieval aids, not structural requirements.

### 3. Edge Representation (derived)

**Baseline fields (necessary and sufficient for correct traversal):**

- `source_id`: the originating node
- `target_id`: the destination node
- `type`: one of {CAUSES, CONTRADICTS, CORRELATES}
- `timestamp`: when this relationship was asserted
- `asserter`: who/what asserted this edge (human, model, inference rule) — necessary for auditing and for resolving conflicting edge assertions

**For causal strength estimation (derived from the causal chain problem):**

The query "why did I decide X?" requires ordering candidate antecedents by causal strength. Three additional fields are necessary:

- `weight` (float, 0–1): probability that the causal claim is correct. A weight of 1.0 means "I directly observed this cause-effect relationship." A weight of 0.3 means "I inferred this relationship from co-occurrence." This is the edge's epistemic confidence.

- `latency` (float, default 1.0): the traversal cost of crossing this edge. Strong, direct causal edges have latency 1.0. Weak, inferred edges have higher latency (e.g., 3.0), making them appear farther in shortest-path traversal. This encodes causal strength as *graph distance*, which allows standard shortest-path algorithms to naturally rank antecedents by causal proximity without special-casing.

- `evidence_ids` (List<UUID>): which memories or observations support this edge assertion. This enables the system to re-evaluate edge weights as new evidence arrives, and to explain *why* the system believes A caused B (not just that it does).

**For CAUSES edges only:**

- `invalidates_source` (boolean, default false): true when B supersedes A — A is now known to be incorrect or outdated. When set, traversal algorithms can automatically down-rank or exclude the source node when answering questions about current state.

**Derived minimum edge schema:**

```
Edge {
  id:               UUID
  source_id:        UUID
  target_id:        UUID
  type:             {CAUSES, CONTRADICTS, CORRELATES}
  timestamp:        timestamp
  asserter:         string
  weight:           float [0.0, 1.0]
  latency:          float [1.0, ∞)        -- only meaningful for CAUSES
  evidence_ids:     List<UUID>
  invalidates_source: boolean             -- only meaningful for CAUSES
}
```

Total: 9 fields. This is the minimum for causal chain strength estimation. Removing any field either breaks correctness (id, source, target, type, timestamp) or forces the traversal algorithm to use a cruder approximation (weight, latency, evidence_ids, invalidates_source).

### 4. Traversal Algorithm (derived)

**Problem statement:** Given an anchor node Q (a memory corresponding to a query or decision), produce the set of causally related memories ordered by causal distance from Q.

**Structural insight:** The `latency` field on CAUSES edges means the graph has non-uniform edge costs. "Causal distance" from Q to any node V is therefore the *minimum sum of latencies* along any CAUSES-typed path from Q to V. This is exactly the problem solved by single-source shortest path on a weighted directed graph — derived purely from the structure of the data, not from any named algorithm.

**Type filtering as a first-class operation:** Before expanding any edge, filter the adjacency structure by type. For "why did I decide X?", expand only CAUSES edges. For "what is consistent with X?", exclude CONTRADICTS edges. This filtering is O(1) per node due to the partitioned adjacency structure derived in Section 2.

**Bounding for the 50ms constraint:**

A full shortest-path traversal over a large graph is O(V log V + E). Under the 50ms constraint this is infeasible for large V. Two bounds are necessary:

1. **Depth bound D**: Human decision chains rarely exceed 5–7 causal hops. Bounding at D=5 limits the traversal frontier to at most fan-out^5 nodes in the worst case.

2. **Width bound K**: At each hop, retain only the K edges with the highest weight (most confident causal links). K=10 is sufficient for most use cases.

The result is a *bounded weighted expansion*: at each frontier node, retrieve its CAUSES bucket (O(1)), sort by weight descending (O(degree log degree), but degree is bounded by K so this is O(K log K) = constant), expand the top K, accumulate in a priority queue ordered by cumulative latency.

**Properties of the resulting algorithm:**
- Produces memories ordered by causal proximity to Q (minimum cumulative latency)
- Complexity: O(K^D * K log K) = O(K^(D+1) * log K) — for K=10, D=5 this is ~10^6 operations, well within 50ms on modern hardware
- Naturally prunes weak causal chains (high latency paths are de-prioritized)
- Produces an *explanation chain*: the path from Q to any returned node is the causal narrative

### 5. Contradiction Detection Algorithm (derived)

**Problem statement:** Given a new memory N, find all existing memories that contradict N. Full store scan is O(M) where M is total memory count — unacceptable as M grows.

**Key structural observation:** Memories that can contradict each other are semantically proximate — they make claims about the same entities, states, or propositions. A memory about "the server's stateless design" cannot contradict a memory about "what Tim had for lunch." Contradiction requires semantic overlap.

**Derived approach — cluster-partitioned CONTRADICTS index:**

The graph can maintain a secondary index: a map from *semantic cluster* to the set of node IDs in that cluster. The cluster assignment can be coarse (10–100 clusters based on topic or entity) — it only needs to be accurate enough to exclude obviously irrelevant memories, not to find all contradictions itself.

When memory N arrives:
1. Assign N to a cluster C in O(log C) time (C << M).
2. Retrieve all memories in cluster C: O(|C|) where |C| << M.
3. For each memory in C, check whether a CONTRADICTS edge already exists (O(1) via the partitioned adjacency structure). If not, evaluate the pair for contradiction.
4. The pairwise check within the cluster is O(|C|^2) in the worst case, but |C| is bounded by the cluster size, typically 10–100 memories per cluster for a well-partitioned clustering.

**Complexity:** O(log C + |C|^2) vs O(M) for full scan. For C=50 clusters and |C|=100 memories per cluster, this is O(log 50 + 10,000) = O(10,006) vs O(M) where M could be millions.

**What makes this sub-O(M):** The CONTRADICTS subgraph is locally dense. Memories that contradict each other cluster together in semantic space. The cluster index exploits this structural property to limit the search space. The graph does not make this faster by itself — it is the *combination* of the cluster index and the CONTRADICTS adjacency structure that achieves sub-O(M).

**Important limitation:** If the cluster assignment is wrong (N assigned to wrong cluster), genuine contradictions will be missed. The cluster assignment must be conservative — when in doubt, assign to multiple clusters. This trades recall for speed.

### 6. What the Flat-Blob Model Misses

**The query that requires full-store scan without the causal graph:**

> "What sequence of observations and decisions led to my current belief about X?"

With the causal graph: traverse CAUSES edges from the node representing "current belief about X" backward (reversing edge direction), following minimum-latency paths to depth D. Returns an ordered causal chain in O(K^D) time. The path itself *is* the explanation.

With the flat-blob model: embed the query "what caused belief X?", scan the entire store for semantically similar memories, rank by similarity score. This retrieves memories that *mention* the same concepts as X, but has no way to distinguish:
- Memories that caused X (causal antecedents)
- Memories that happened to discuss the same topic without causal connection
- Memories that were caused *by* X (causal descendants)
- Memories that contradict X

The similarity search retrieves all four categories indiscriminately. The causal graph distinguishes them by edge type and direction in O(1) per edge.

**The directional gap is the fundamental miss.** Embedding similarity is undirected — it measures topical proximity, not epistemic dependency. The causal graph encodes *why* memories are related, not just *that* they are related. This distinction is irreducible: no amount of semantic similarity search can recover causal direction without the graph, because two memories can be semantically identical while one causes the other or they merely co-occurred.

## Ideas Extracted (using real Recall stack)

### Idea: typed-adjacency-neo4j
- **Novelty**: 0.7 — Neo4j supports relationship types natively, but the partitioned-by-type adjacency structure with logical clocks on memory nodes is not standard practice in AI memory systems
- **Evidence**: 0.9 — Graph adjacency lists with type partitioning are a textbook data structure (Cormen et al.); Neo4j relationship types implement this directly
- **Feasibility**: 0.9 — 1–2 week implementation; Neo4j schema already deployed in Recall; requires adding `generation` counter to memory nodes and enforcing {CAUSES, CONTRADICTS, CORRELATES} as the only edge types
- **Description**: Restructure Recall's Neo4j schema to use exactly three relationship types (CAUSES, CONTRADICTS, CORRELATES) with `weight`, `latency`, `asserter`, and `evidence_ids` properties on each edge. Add a `generation` (Lamport timestamp) counter to memory nodes, incremented on every write. This replaces the current untyped or ad-hoc relationship model with a minimal provably-correct causal graph schema. The schema can be migrated from the existing Neo4j instance without data loss by mapping existing relationships to the three types.

### Idea: latency-weighted-causal-walk
- **Novelty**: 0.8 — Encoding causal confidence as graph edge latency (rather than a separate score) so that standard shortest-path traversal naturally ranks antecedents by causal proximity is not documented in any AI memory system surveyed
- **Evidence**: 0.7 — Weighted shortest-path on typed subgraphs is standard graph theory; the specific application to causal memory chain retrieval is engineering extrapolation
- **Feasibility**: 0.8 — 2–3 weeks; requires Neo4j `latency` property on CAUSES edges and a Cypher query using `apoc.path.expandConfig` with cost accumulation, or a custom traversal via the Neo4j Bolt driver; directly enables the Q010 PPR-retrieval finding
- **Description**: Store causal confidence as `latency` on CAUSES edges (high-confidence = latency 1.0, inferred = latency 3.0–5.0). Implement the "why did I decide X?" query as a bounded weighted expansion from the anchor node: follow only CAUSES edges, accumulate latency, return the top-K memories at minimum cumulative latency. In Neo4j this is a parameterized Cypher query with `maxDepth: 5` and `relationshipFilter: 'CAUSES>'`. The path from anchor to each returned node is the explanation chain, surfaced directly to the user. This is the causal graph foundation that Q010's PPR retrieval requires.

### Idea: cluster-contradiction-index
- **Novelty**: 0.75 — Combining semantic cluster assignment with a CONTRADICTS adjacency index for sub-O(M) contradiction detection is not implemented in any known AI memory system
- **Evidence**: 0.6 — The underlying principle (partition-then-check) is used in database query optimization and computational geometry; the specific application to memory contradiction detection is derived, not validated
- **Feasibility**: 0.7 — 3–6 weeks; requires: (1) a clustering step on Qdrant vectors (K-means or similar over existing embeddings, run offline), (2) a `cluster_id` field on Neo4j memory nodes, (3) a CONTRADICTS edge creation step in the Recall ingestion pipeline that checks same-cluster nodes using the Qdrant embedding store for semantic comparison, then writes CONTRADICTS edges to Neo4j
- **Description**: Assign each memory to one of 50–100 semantic clusters based on its Qdrant embedding. When a new memory N arrives, retrieve only the ~100 memories in N's cluster from Neo4j, compare N against each for contradiction (using the Qdrant embedding and a threshold on angular distance in the direction of negation), and write CONTRADICTS edges for confirmed contradictions. This is O(log C + |C|^2) per ingestion vs O(M) for a full scan. Contradiction detection then becomes a graph query: "find memories with a CONTRADICTS edge to N," answered in O(1) by Neo4j's adjacency index.

### Idea: invalidates-source-provenance-chain
- **Novelty**: 0.65 — The `invalidates_source` flag on CAUSES edges to represent supersession, with automatic exclusion of invalidated nodes from current-state queries, is not standard in AI memory systems
- **Evidence**: 0.8 — Event sourcing and CQRS patterns use "tombstone" records for exactly this purpose; this is a direct application to the memory graph
- **Feasibility**: 0.85 — 1–2 weeks; add a boolean `invalidates_source` property to CAUSES edges in Neo4j; update Recall's retrieval queries to filter `WHERE NOT (m)-[:CAUSES {invalidates_source: true}]->()` to exclude superseded memories from default retrieval; superseded memories remain in the graph for historical traversal
- **Description**: When Recall stores a memory B that explicitly updates or corrects memory A (e.g., "I was wrong about X; actually Y"), create a CAUSES edge from A to B with `invalidates_source: true`. Neo4j queries for "current state" filter out invalidated nodes. The full provenance chain (A→B) remains traversable for "how did my belief evolve?" queries. This is the minimal representation of supersession without requiring a separate edge type, consistent with the derived minimal type set {CAUSES, CONTRADICTS, CORRELATES}.

### Idea: generation-clock-causal-ordering
- **Novelty**: 0.6 — Lamport logical clocks on memory nodes to establish happens-before ordering independent of wall-clock timestamps is well-established in distributed systems but unused in AI memory systems
- **Evidence**: 0.95 — Lamport (1978) is peer-reviewed foundational work; happens-before ordering is provably correct for establishing causal precedence in distributed systems
- **Feasibility**: 0.9 — 1 week; add a `generation` integer counter to Neo4j memory nodes, initialized to 0 and incremented atomically on each write; when creating a CAUSES edge from A to B, assert `B.generation > A.generation`; use `generation` as the primary sort key for temporal ordering within causal chains
- **Description**: Wall-clock timestamps in Recall are unreliable for causal ordering because memories can be ingested out of order (a session summary written hours after the session, a backdated note). The Lamport generation counter provides a strictly monotonic causal order: if A.generation < B.generation, A happened-before B in the system's causal history, regardless of wall-clock time. This is added as a single integer field on Neo4j memory nodes and an assertion check on CAUSES edge creation. It costs nothing at query time and makes causal chain traversal order provably correct.
