## Risk Profile Classification

### Q144 — ef_search=128 (ATOMIC, zero risk)

- **Change type**: Single PATCH call to Qdrant REST API. No code. No restart.
- **Blast radius**: Affects only Qdrant ANN search performance. ef=128 increases candidate set from 64 to 128 nodes during HNSW traversal. Recall improves (more candidates considered). Latency increase: +2–5ms per query (negligible at current load).
- **Reversibility**: Instant. `PATCH /collections/recall_memories {"hnsw_config": {"ef": 64}}` reverts in under 1 second.
- **Failure mode**: None credible. The PATCH either succeeds (result: true) or fails (response shows error, collection unchanged). No partial state.
- **Monitoring window needed**: No. Verify with one GET call: `result.config.hnsw_config.ef == 128`.
- **Classification**: ATOMIC — deploy-and-confirm in under 60 seconds.

### Q150 — Reconcile probe (ATOMIC, additive only)

- **Change type**: Four additive file edits. No existing behavior changes. New Redis key written by worker. New `s5` field in /health response.
- **Blast radius**: /health response gains a new `s5` key. Existing consumers that parse checks{} dict are unaffected — `s5` is a new top-level key. `s5.status = UNKNOWN` immediately post-deploy is correct and expected behavior.
- **Reversibility**: Revert the four file edits and redeploy. Redis key `recall:health:orphan_probe` can be left in place — if the worker code is reverted, nothing writes to it and the /health endpoint no longer reads it. Clean.
- **Failure mode**: If Redis is unavailable at the time /health is called, the `_orphan_sync()` method returns UNKNOWN without throwing (per spec). /health continues to serve. Zero crash risk.
- **Monitoring window needed**: No for the code change itself. Note: `s5` will show UNKNOWN for up to 7 days until the next Sunday 5:30am ARQ cron fires. This is expected — not a defect.
- **Classification**: ATOMIC — deploy, confirm /health includes `s5` key, done.

### Q148 — Oracle health endpoint (STAGED, seed dependency)

- **Change type**: New endpoint in health_dashboard.py. New Redis registry key. Oracle seed population via POST /memory/store × 5.
- **Blast radius**: Additive endpoint. No existing endpoint behavior changes. No risk to existing functionality.
- **Reversibility**: Remove the endpoint function, redeploy. Oracle seeds in Qdrant/Neo4j/Redis persist but are harmless (high-importance memories with is_oracle=true that get injected into sessions — desirable even if the health endpoint is removed).
- **Failure mode if seeds not written**: Endpoint returns `{"status": "uninitialized", "overall_pass": false}` — correct behavior, not an error. Monitoring systems should not alarm on UNINITIALIZED during the seeding window.
- **Staging requirement**: Two-step deployment:
  1. Deploy code (endpoint available, returns UNINITIALIZED)
  2. Write 5 oracle seeds via recall API, capturing UUIDs
  3. Write oracle registry to Redis key `recall:oracle:registry`
  4. Verify endpoint transitions from UNINITIALIZED → HEALTHY
- **Monitoring window**: 15–30 minutes between code deploy and seed verification. Not a risk window — a completion window.
- **Classification**: STAGED (two-step), but low risk at each step.

### Q156 — co_retrievals migration (STAGED, connection-race risk)

- **Change type**: Additive SQL executed at connect() time in postgres_store.py. Creates new table `co_retrieval_outbox` plus 3 indexes. UPSERT writes flow after deploy.
- **Blast radius**: The migration itself is idempotent (CREATE TABLE IF NOT EXISTS + CREATE INDEX IF NOT EXISTS pattern implied by "idempotent" in the spec). Risk is the connection-race: if an ARQ worker holds an open PostgreSQL connection and begins the hourly consolidation task at the exact moment the new code calls `_MIGRATE_CO_RETRIEVALS_OUTBOX` at connect(), there is potential for a transaction conflict or lock contention on the new table creation.
- **Reversibility**:
  - If migration has not run: revert code, redeploy. No database state to clean up.
  - If migration has run: revert code, redeploy. The `co_retrieval_outbox` table remains in PostgreSQL but no code writes to it. Harmless. Optional cleanup: `DROP TABLE co_retrieval_outbox;` from psql, but not required for service health.
  - UPSERT writes that flowed during the brief deployment window are orphaned if code is reverted — acceptable data loss given the table is a new outbox, not primary storage.
- **Failure mode**: If the migration SQL fails at connect() (e.g., lock timeout), the application fails to start. This is the primary risk: a failed migration blocks the service from coming up.
- **Mitigation**: Deploy during the safe window — after hourly consolidation completes and before the next one starts. Hourly consolidation runs on the ARQ worker; identify its schedule and deploy in the ~50-minute window between runs.
- **Monitoring window needed**: Yes — 5 minutes post-deploy to confirm:
  1. Application started successfully (no connect() failure in logs)
  2. PostgreSQL table exists: `\dt co_retrieval_outbox`
  3. UPSERT writes are flowing (check postgres audit log or new table row count)
- **Classification**: STAGED — deploy in low-traffic window with 5-minute post-deploy observation.

## Correct Deployment Sequence

### Recommended Order: Q144 → Q150 → Q156 → Q148

**Rationale for ordering:**

**Step 1: Q144 (ef_search=128)** — First, because it is the only change with zero blast radius to the application code. It is a pure infrastructure tweak. Deploying it first establishes a baseline for "everything is healthy" before any code changes land. If something else breaks later, ef_search is not a suspect.

**Step 2: Q150 (reconcile probe)** — Second, because it is additive code with no failure mode. Deploying it early means the S5 monitoring signal will begin accumulating. The 7-day UNKNOWN window starts here — the sooner Q150 is deployed, the sooner the first Sunday cron makes S5 meaningful. Deploying Q150 before Q156 also means /health is instrumented before the riskier database migration, giving better observability during Q156.

**Step 3: Q156 (co_retrievals migration)** — Third, because it carries the only real deployment risk (connection-race at connect() time). Timing constraint: deploy between hourly ARQ consolidation runs. With Q150 already in place, /health is actively monitored during Q156 deploy. Post-deploy monitoring window: 5 minutes.

**Step 4: Q148 (oracle health endpoint + seed population)** — Last, because it requires the most human interaction (manual seed writing, UUID capture, registry write). This is not a risk-based ordering — oracle seeding takes attention and should not be rushed. Placing it last allows Q144/Q150/Q156 to land cleanly before any interactive seed process begins.

### Deployment schedule (wall-clock example)

```
T+00:00  Q144: SSH to CasaOS, PATCH Qdrant, verify ef==128.
         Total time: ~3 minutes including verification.

T+10:00  Q150: git pull on CasaOS, docker compose restart recall-api.
         Verify /health includes "s5" key with status "UNKNOWN".
         Total time: ~5 minutes.

T+20:00  Q156: Confirm hourly ARQ cron last ran (check ARQ logs).
         Wait if consolidation is in progress. Typical run: <5 min.
         git pull (already done at Q150 step if same deploy), docker compose restart recall-worker.
         Recall API already running new code from Q150 deploy.
         Watch logs for connect() migration success.
         Verify co_retrieval_outbox table exists in PostgreSQL.
         Monitoring window: 5 minutes.
         Total time: ~10 minutes.

T+40:00  Q148: Write 5 oracle seeds via curl, capture UUIDs, write registry to Redis.
         Call /admin/health/recall-quality, confirm transition to HEALTHY.
         Total time: ~20 minutes.

Total wall-clock: ~1 hour, safe to execute in a single maintenance window.
```

**Note on combined vs split deployments**: Q144 can be deployed at any time independently — it requires no code change and no service restart. If the goal is to minimize blast radius in a single window, Q144 can be done opportunistically before the other three. The three code changes (Q150, Q156, Q148) benefit from being in the same deploy event because they share a git pull + docker compose restart cycle.

## Rollback Procedures

### Q144 rollback (instant)
```bash
# From CasaOS host:
curl -X PATCH 'http://localhost:6333/collections/recall_memories' \
  -H 'Content-Type: application/json' \
  -d '{"hnsw_config": {"ef": 64}}'
# Verify:
curl -s 'http://localhost:6333/collections/recall_memories' | python3 -c \
  "import json,sys; d=json.load(sys.stdin); print(d['result']['config']['hnsw_config']['ef'])"
```
Time to revert: 30 seconds.

### Q150 rollback (revert 4 files, redeploy)
```bash
# Revert the 4 file changes (workers/reconcile.py, core/health.py x2, api/main.py)
git revert <commit-hash>
docker compose restart recall-api recall-worker
# Verify /health no longer contains "s5" key
curl http://192.168.50.19:8200/health | python3 -m json.tool
```
Time to revert: 5 minutes. Redis key `recall:health:orphan_probe` left in place — harmless.

### Q156 rollback (two cases)

**Case A — Migration did not run** (connect() failed, service did not start):
```bash
git revert <commit-hash>
docker compose restart recall-api recall-worker
# Service restores to previous version, no DB cleanup needed
```
Time to revert: 5 minutes.

**Case B — Migration ran successfully, then regression found**:
```bash
git revert <commit-hash>
docker compose restart recall-api recall-worker
# co_retrieval_outbox table remains in PostgreSQL — do NOT drop unless manually desired
# UPSERT writes during the window are orphaned — acceptable
# Optional cleanup (NOT required for service health):
# psql -U recall_user -d recall_db -c "DROP TABLE IF EXISTS co_retrieval_outbox;"
```
Time to revert: 5 minutes (plus optional psql cleanup ~1 minute).

**Critical Q156 rollback constraint**: If the connect() migration fails and the service does not start, do not retry the deploy while the ARQ consolidation worker is active. Wait for the worker's hourly cron to complete, then retry or revert.

### Q148 rollback (two steps)

**Step A — Remove endpoint code**:
```bash
git revert <commit-hash>
docker compose restart recall-api
# /admin/health/recall-quality returns 404
```

**Step B — Oracle seeds** (optional, seeds are beneficial even without the endpoint):
Seeds stored in Qdrant/Neo4j/PostgreSQL are high-importance semantic memories (is_oracle=true). They will surface in normal retrieval and are beneficial. There is no functional reason to delete them on rollback. If the Redis registry key needs clearing: `redis-cli DEL recall:oracle:registry`.

Time to revert: 5 minutes. Seed deletion optional.

## Qualifying Ideas

- **combined-restart-for-Q150-Q156**: (N=0.65, E=0.85, F=0.95) — Q150 and Q156 both require `docker compose restart recall-api recall-worker`. If Q156 timing constraint is satisfied (post-consolidation window), deploy both code changes in a single git pull + restart rather than two separate restart events. This reduces total service downtime and simplifies the change record. Q144 still precedes as a separate operation (no restart). Q148 still follows as a separate step (interactive seeding).

- **manual-reconcile-trigger-post-Q150**: (N=0.55, E=0.80, F=0.98) — After deploying Q150, immediately trigger `run_reconcile` via ARQ to populate `recall:health:orphan_probe` rather than waiting up to 7 days for the next Sunday cron. Command: invoke the ARQ task directly via Python or Redis queue. This makes S5 meaningful immediately rather than showing UNKNOWN for a week.

- **pre-deploy-snapshot-via-health-snapshot-endpoint**: (N=0.70, E=0.85, F=0.95) — Before any of the four changes, call `POST /admin/health/snapshot-baseline` (already exists per health_dashboard.py). This captures current metrics as a pre-deploy baseline. If post-deploy metrics diverge, the snapshot provides a before/after comparison for triage. Takes 30 seconds. Zero risk. Should be part of any multi-change deployment checklist for this system.
