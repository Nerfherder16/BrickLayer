# Finding: Q078 — Causal Graph Fanout Physics and Epoch Filter Gap

**Question**: The Neo4j causal graph (Q057) stores directed edges representing causal relationships between memories (memory A was relevant during the same session as memory B). At Tim's session density and the co-retrieval rate implied by MBC Level 1 routing, what is the expected edge count growth rate per month, what is the maximum fanout (edges per node) before Neo4j graph traversal latency exceeds the 50ms LLPC retrieval budget, and does the epoch-clock working-set filter (`last_access_epoch >= current_epoch - tau`) need to be applied to causal graph traversal as well as vector search to prevent the graph from expanding retrieval sets beyond the working-set boundary?
**Agent**: quantitative-analyst
**Question Type**: PHYSICS
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Neo4j traversal performance benchmarks, graph database internals (BFS/DFS execution plans, relationship index structures), causal graph co-retrieval combinatorics

## Operation Definition

This finding measures three quantities for the Recall causal graph stored in Neo4j:

1. **Edge growth rate**: How many new directed co-retrieval edges are created per month at Tim's usage pattern (~10 retrieve calls/session, ~5 sessions/day), accounting for 80% duplication (edges that already exist are not re-created, only weight-incremented).

2. **Maximum safe fanout**: The maximum number of outgoing edges a node can have before a 1-hop BFS from that node exceeds the 50ms LLPC retrieval latency budget. Neo4j traversal is fast but not free; at high fanout the neighbor-set reading itself can saturate the budget.

3. **Epoch filter gap**: Whether the working-set boundary enforced by the epoch-clock filter (`last_access_epoch >= current_epoch - tau`) in PostgreSQL/Qdrant vector search is also applied during Neo4j causal graph traversal — and if not, whether graph traversal can silently expand the retrieval set to include stale memories that would be filtered out by the epoch gate on the direct-retrieval path.

### Neo4j 1-hop BFS traversal floor

Neo4j stores relationships in a doubly-linked list per node (the "relationship chain"), indexed by relationship type in Neo4j 4.x+ via the relationship type index. A 1-hop BFS from a seed node with fanout F requires:

1. One relationship type index lookup: ~0.1ms (fixed, fanout-independent)
2. Reading F relationship records from the chain: ~0.01ms per record (NVMe SSD, relationship record = 34 bytes, cache-hot assumed for active nodes)
3. Loading F neighbor node header records for deduplication: ~0.02ms per neighbor (node record = 15 bytes, first-time load)

**Minimum traversal cost for 1-hop BFS at fanout F (cache-warm):**

| Fanout F | Relationship reads | Neighbor loads | Total estimated | Within 50ms budget? |
|----------|--------------------|----------------|-----------------|---------------------|
| 10 | 0.1ms | 0.2ms | ~0.5ms | Yes (10x headroom) |
| 50 | 0.5ms | 1.0ms | ~2ms | Yes (25x headroom) |
| 200 | 2ms | 4ms | ~7ms | Yes (7x headroom) |
| 500 | 5ms | 10ms | ~16ms | Yes (3x headroom) |
| 1,000 | 10ms | 20ms | ~32ms | Marginal (1.6x) |
| 2,000 | 20ms | 40ms | ~62ms | **EXCEEDS budget** |
| 5,000 | 50ms | 100ms | ~153ms | **3x over budget** |

The above assumes NVMe SSD, relationship records cache-warm (active nodes' chains in Neo4j page cache), and no lock contention. In the adversarial case (cold cache, spinning disk, or high concurrency), the constants grow 5–10x, moving the budget wall to ~F=200–400.

**LLPC latency budget breakdown**: The total LLPC retrieval budget is 50ms. Of this, the Q057 finding implies graph traversal is a single component alongside PostgreSQL MBC routing and Qdrant vector search. A realistic allocation for graph traversal alone is 10–15ms, not the full 50ms. At 10ms budget for graph traversal:

| Fanout F | Estimated traversal cost | Within 10ms graph budget? |
|----------|--------------------------|--------------------------|
| 50 | ~2ms | Yes |
| 200 | ~7ms | Yes |
| 400 | ~14ms | **EXCEEDS 10ms sub-budget** |
| 500 | ~16ms | Exceeds |

**Conservative safe fanout ceiling: ~300 edges per node** (allows 10ms sub-budget with 30% margin at NVMe cache-warm; collapses to ~100 on spinning disk or cold cache).

## Fanout Latency Analysis

### At what fanout does 1-hop BFS exceed 50ms?

Using the conservative per-record costs from the Theoretical Minimum section:

```
Traversal cost = 0.1ms (index lookup) + F × 0.01ms (relationship reads) + F × 0.02ms (neighbor loads)
               = 0.1 + F × 0.03ms

Setting = 50ms: F × 0.03 = 49.9 → F ≈ 1,663
```

**At NVMe cache-warm, 1-hop BFS exceeds the full 50ms LLPC budget at approximately F = 1,600 edges.**

However, this is not the relevant threshold because:

1. **The graph sub-budget is 10–15ms**, not the full 50ms. At 10ms: F × 0.03 = 9.9 → F ≈ 330.
2. **Cold-cache penalty**: If Neo4j's page cache has been evicted (after idle period, or under memory pressure on CasaOS), relationship record reads cost 0.1–0.5ms each (SSD random read latency). At 0.1ms/record: budget exhausted at F ≈ 90.
3. **2-hop BFS amplification**: If traversal is 2 hops deep and the average fanout is F, the number of records read is O(F²). At F=50: 2,500 records, ~75ms at cache-warm — already exceeding the full 50ms budget.

**Operational fanout limits:**

| Cache state | Max safe fanout (10ms sub-budget) | Max safe fanout (50ms full budget) |
|-------------|----------------------------------|-----------------------------------|
| NVMe cache-warm | ~330 | ~1,600 |
| NVMe cache-cold | ~90 | ~450 |
| Spinning disk | ~15 | ~75 |

**CasaOS context**: Tim's NAS-grade CasaOS deployment likely uses NVMe or SATA SSD storage for Docker volumes, but the Neo4j container is not co-located with high-memory hardware (the RTX 3090 machine is separate at 192.168.50.62). At the Docker host (192.168.50.19), page cache eviction is likely under concurrent load. **Conservative operational ceiling: 200 edges per node before any risk of latency spikes.**

### 2-hop BFS: the hidden multiplier

Q057 established that 1-hop BFS from K_llpc=2 seed nodes is the intended traversal depth. But if K_llpc seed nodes are themselves neighbors of hub nodes, and the Cypher query uses `MATCH (seed)-[:CO_RETRIEVED*1..2]->(neighbor)`, the traversal visits up to:

```
2 seed nodes × F_avg neighbors × F_avg second-hop neighbors
= 2 × F × F = 2F²
```

At F=50: 5,000 records = ~150ms. This is 3× over the full LLPC budget. **2-hop traversal from hub nodes is latency-unsafe above F ≈ 20–30 average fanout.**

### The 20-core-memory scenario

If Tim's 20 most-retrieved memories (Homelab IP table, System-Recall architecture, ADBP constants, etc.) are co-retrieved in every session:

**Graph structure after 30 days:**
- 20 nodes, fully connected (K₂₀ clique)
- 20 × 19 / 2 = 190 undirected edges (380 directed edges)

**Traversal behavior:**
- 1-hop BFS from any of the 20 hub nodes returns all 19 neighbors: safe (~0.6ms)
- The 19 neighbors are also hub nodes; if 2-hop BFS is triggered, each of their 19 neighbors must be visited: 19 × 19 = 361 additional node visits in the second hop

- Query: "What is Tim's Ollama model?"
- LLPC seeds: 2 memories (Ollama config, Homelab networking)

**The clique creates a query-agnostic retrieval set expansion**: any query that seeds from a hub node pulls in all other hub nodes regardless of topical relevance. The causal graph edges encode temporal co-occurrence, not semantic relevance. This is the core adversarial pathology.

**Scale projection**: At Month 24, with ~10,800 nodes and a 100-node hub set (top 1%), each hub node may have fanout 500–1,500. A single-hop BFS from a hub seed returns 500–1,500 candidates for downstream ranking — a 50–150× expansion of the seed set. At K_mbc=10 (the MBC result target), the graph traversal would return 50–150× more candidates than needed, forcing a re-ranking step that was not in the original LLPC design budget.

## Epoch Filter on Graph Traversal

### The gap: epoch filter is applied to vector search but NOT to graph traversal

Q062 established the epoch-clock design: memories with `last_access_epoch < current_epoch - tau` are outside the working-set and excluded from LLPC/MBC routing queries via the PostgreSQL predicate `last_access_epoch >= (current_epoch - tau)` with tau=20 by default.

Q071 confirmed that the health signal set does not include a causal graph traversal health signal — there is no S5 for "graph traversal epoch compliance."

**The epoch filter gap**: The Neo4j Cypher query for Level 1.5 traversal (Q057) is:

```cypher
MATCH (seed:Memory {id: $seed_id})-[:CO_RETRIEVED]->(neighbor:Memory)
WHERE neighbor.id IN $llpc_memory_ids OR neighbor.relevance_score > $threshold
RETURN neighbor.id
```

This query has no epoch predicate. It returns neighbors regardless of when they were last accessed. A memory that was last accessed 100 sessions ago — deep outside the tau=20 working-set window, completely invisible to PostgreSQL and Qdrant retrieval — is fully reachable via Neo4j graph traversal if it was historically co-retrieved with a currently-active seed.

**Concrete failure mode:**

1. Memory M was frequently co-retrieved with Tim's core Homelab config 80 sessions ago. It has `last_access_epoch = current_epoch - 80`, well outside the tau=20 window.
2. M has edges to 15 currently-active hub nodes.
3. Current session seeds from a hub node H.
4. 1-hop graph traversal returns M as a Level 1.5 candidate.
5. M is passed to the LLPC context window — but it is a stale memory, possibly superseded, reflecting the system state 80 sessions in the past.
6. The epoch filter that protects vector search and MBC routing does NOT fire for M because M arrived via graph traversal, bypassing the PostgreSQL working-set predicate.

**This is a silent staleness injection**: stale memories enter the context window through a path that bypasses the working-set boundary designed to prevent exactly this.

### Severity assessment

The severity depends on graph density and edge age distribution:

- **Month 1–3 (sparse graph, fresh edges)**: Most edges are recent; stale-node traversal is uncommon. Low severity.
- **Month 6–12 (maturing graph, hub nodes accumulating)**: Hub nodes have edges to memories from all periods of their history. A hub node accessed today has edges to memories from 6–12 months ago. Graph traversal routinely retrieves outside the working-set window. **Medium-high severity.**
- **Month 24+ (dense graph)**: Hub nodes have hundreds of edges spanning 2+ years of sessions. The majority of 1-hop neighbors may be outside the working-set window. Graph traversal without epoch filtering becomes a primary staleness injection vector. **High severity.**

### Should the epoch filter be applied to graph traversal?

**Yes, the epoch filter must be applied to causal graph traversal.** The epoch-clock working-set boundary is a semantic invariant of the entire Recall retrieval system, not just an optimization for PostgreSQL query cost. Its purpose is to prevent stale memories from entering the context window. Any retrieval path that bypasses the epoch gate violates this invariant.

**Implementation options:**

**Option A: Epoch predicate in Cypher (preferred)**

Requires propagating `last_access_epoch` as a node property in Neo4j (in addition to storing it in PostgreSQL). This creates a dual-write: when a memory is accessed and `last_access_epoch` is updated in PostgreSQL, the same value must be written to the Neo4j node property.

```cypher
MATCH (seed:Memory {id: $seed_id})-[:CO_RETRIEVED]->(neighbor:Memory)
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)
RETURN neighbor.id, neighbor.last_access_epoch
```

Cost: one additional integer property per Neo4j node, one additional property write per memory access. At K_accessed=10/session and 5 sessions/day, this is 50 Neo4j property writes/day — negligible.

Benefit: staleness filtering happens inside the graph engine before returning candidates. No post-processing in Python. Epoch-filtered 1-hop BFS returns only working-set neighbors, naturally capping the fanout to the subset accessed within tau=20 sessions.

**Option B: Post-filter in Python**

Graph traversal returns all neighbors, Python filters them against the PostgreSQL epoch predicate before passing to LLPC.

```python
neighbors = neo4j.run_cypher(seed_cypher)
active_neighbors = [n for n in neighbors if n["last_access_epoch"] >= current_epoch - tau]
```

Cost: requires a PostgreSQL lookup per neighbor (or a cached epoch map in Redis) to retrieve `last_access_epoch` for each neighbor ID. At fanout F=50, this is 50 PostgreSQL lookups or 50 Redis GETs — adds 5–50ms depending on batch efficiency. Not suitable for the 50ms LLPC budget without batching.

**Option B (batched):**
```python
neighbor_ids = neo4j.run_cypher(seed_cypher)
epoch_map = await pg.fetch(
    "SELECT id, last_access_epoch FROM memories WHERE id = ANY($1)",
    neighbor_ids
)
active_neighbors = [n for n in epoch_map if n["last_access_epoch"] >= current_epoch - tau]
```

One PostgreSQL batch query regardless of F. Adds ~2ms for the batch lookup. Acceptable but requires a round-trip to PostgreSQL after Neo4j traversal.

**Option C: Edge age filtering (alternative)**

Rather than filtering by node epoch, filter by edge age: only traverse edges created within the last tau sessions.

```cypher
MATCH (seed:Memory {id: $seed_id})-[r:CO_RETRIEVED]->(neighbor:Memory)
WHERE r.last_reinforced_epoch >= ($current_epoch - $tau)
RETURN neighbor.id
```

Semantics differ from Option A: this filters on when the causal relationship was last active, not when the neighbor memory was last accessed. A memory that was co-retrieved with the seed 50 sessions ago but not recently would be excluded. This is a more conservative filter (excludes stale causal relationships, not just stale neighbors) and may be the more semantically correct boundary.

### Tradeoffs summary

| Option | Epoch adherence | Latency impact | Implementation complexity | Recommended? |
|--------|----------------|----------------|--------------------------|-------------|
| A: Cypher epoch predicate (node property) | Full — filters inside graph engine | Minimal (adds one WHERE clause to Cypher) | Medium (dual-write to Neo4j) | **Yes — preferred** |
| B: Python post-filter with batch PG lookup | Full — all candidates checked | +2ms for batch lookup | Low (one query after traversal) | **Yes — acceptable fallback if dual-write is deferred** |
| C: Edge age filter (relationship property) | Partial — filters stale edges, not stale nodes | Minimal | Medium (requires edge property write on each co-retrieval) | Yes — as complement to A |
| None (current state) | None — working-set boundary violated | Zero | Zero | **No — current state is incorrect** |

### Fanout cap recommendation

Regardless of epoch filtering, a maximum fanout cap (max K neighbors returned per BFS hop) is needed to bound worst-case traversal latency:

**Recommended: K_graph = 20 neighbors maximum per BFS hop**

At K_graph=20 per hop from K_llpc=2 seeds: maximum 40 graph-traversal candidates entering LLPC ranking. Combined with epoch filtering, this eliminates both the latency risk from hub-node traversal and the retrieval set inflation from the dense-subgraph adversarial case.

Implementation: `LIMIT 20` in the Cypher ORDER BY clause, ordering by edge weight (co-occurrence count) descending to prefer the most-reinforced causal relationships:

```cypher
MATCH (seed:Memory {id: $seed_id})-[r:CO_RETRIEVED]->(neighbor:Memory)
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)
RETURN neighbor.id, r.co_occurrence_count
ORDER BY r.co_occurrence_count DESC
LIMIT 20
```

## Ideas Extracted

### Idea: causal-graph-epoch-filter-dual-write
- **Novelty**: 0.78 — The insight that a working-set boundary applied to one retrieval path (vector search via PostgreSQL) but not to an alternative path (graph traversal via Neo4j) creates a semantic inconsistency — a staleness injection vector — is not explicitly present in any reviewed AI memory system design. Graphiti applies graph proximity as a reranking signal without a temporal working-set concept. MemGPT/Letta applies LLM-managed eviction without epoch-clock structures. The specific failure mode (stale memory enters context via graph traversal, bypassing epoch gate on the vector/SQL path) requires the multi-path architecture of Recall to manifest.
- **Evidence**: 0.72 — The epoch-clock working-set boundary is established in Q062 (BREAKTHROUGH verdict) and its health monitoring is defined in Q071. The absence of an epoch predicate in the Q057 Cypher template is observable in that finding's code snippets. The consequence (stale node retrieval) follows directly from graph traversal semantics: Neo4j has no knowledge of the LLPC epoch invariant unless that predicate is explicitly embedded in the Cypher query or applied in a post-processing step. The dual-write overhead cost (50 Neo4j property writes/day) is computable from Tim's session density.
- **Feasibility**: 0.88 — Requires adding `last_access_epoch` as a Neo4j node property and writing it at memory access time alongside the PostgreSQL update. The Cypher query in Q057 gains one WHERE clause. The existing `recall-retrieve.js` hook already writes to PostgreSQL at access time; adding a Neo4j property update is a 5-line change. The Redis epoch cache (already required by Q062) provides the `current_epoch` value for both the PostgreSQL predicate and the Cypher predicate without an additional lookup.
- **Description**: Add `last_access_epoch` as a property on Neo4j Memory nodes, written at the same time as the PostgreSQL `last_access_epoch` column update (at each memory access event). Amend all Cypher traversal queries to include `WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)` — the same tau=20 working-set window used by PostgreSQL LLPC/MBC routing. This closes the epoch filter gap and ensures the working-set boundary is enforced uniformly across all retrieval paths.

### Idea: graph-traversal-fanout-cap-k20
- **Novelty**: 0.68 — Applying a hard retrieval cardinality cap to graph BFS results before passing to a downstream ranker is a standard technique in graph-augmented RAG systems (Microsoft GraphRAG uses top-K community summary selection; Graphiti uses a configurable result count). The novel element here is grounding the specific cap value (K=20) in the LLPC budget arithmetic: K_graph × K_llpc seeds × per-record traversal cost must fit within the 10ms graph sub-budget, and K_graph × K_llpc must not overwhelm the downstream ranking step. The combined derivation (latency + ranking capacity) to arrive at K=20 is not found in the reviewed literature.
- **Evidence**: 0.65 — The traversal cost model (0.03ms per record at NVMe cache-warm) is derived from Neo4j relationship record sizes and NVMe access latency benchmarks. The 10ms graph sub-budget is a budget allocation inference (total LLPC budget = 50ms; three concurrent components: MBC SQL, Qdrant vector search, graph traversal; graph gets ~10–15ms). The K=20 cap is consistent with Neo4j best practices for bounded traversal in production graph queries (Neo4j documentation recommends LIMIT clauses on all production traversal queries to prevent accidental full-graph scans).
- **Feasibility**: 0.95 — A `LIMIT 20` clause with `ORDER BY r.co_occurrence_count DESC` is a single-line change to every Cypher traversal query. No schema changes. No infrastructure changes. Eliminates the worst-case hub-node latency spike and the dense-subgraph retrieval set inflation pathology.
- **Description**: Add `LIMIT 20` (with ORDER BY co_occurrence_count DESC) to all causal graph BFS Cypher queries. This caps the graph traversal output at 20 candidates per seed node, regardless of the seed node's actual fanout. Combined with K_llpc=2 seed nodes, the maximum graph-traversal contribution to the LLPC candidate set is 40 memories — bounded, predictable, and consistent with the 10ms graph sub-budget even at worst-case hub-node fanout.

### Idea: graph-edge-age-filter-as-staleness-proxy
- **Novelty**: 0.72 — Filtering causal edges by the epoch of their last reinforcement (rather than filtering neighbor nodes by their last access epoch) captures a different and complementary staleness signal: a causal relationship between two memories may be stale even if both memories individually remain within the working-set window. For example, memory A (Ollama config) and memory B (ADBP simulation params) may both be accessed occasionally, but their co-retrieval relationship was formed 60 sessions ago when Tim was using both simultaneously — the relationship is now stale even if both nodes are individually active. Node-level epoch filtering (Option A) cannot catch this; edge-level epoch filtering (Option C) can.
- **Evidence**: 0.65 — The distinction between node freshness and relationship freshness is a property of temporal knowledge graphs. Graphiti explicitly manages relationship timestamps and uses edge timestamps in its retrieval pipeline (documented in Graphiti's temporal consistency section). The Recall causal graph already stores `co_occurrence_count` per edge (per Q057's schema); adding `last_reinforced_epoch` is a natural extension. The semantic motivation (stale co-retrieval relationships should not elevate stale relevance signals) is logically sound.
- **Feasibility**: 0.82 — Requires adding a `last_reinforced_epoch` property to `CO_RETRIEVED` relationships in Neo4j, written when the edge is created or weight-incremented. The existing edge-creation logic (at session end, when co-retrieval pairs are identified) already has the current epoch in scope. Cypher query amendment: one additional WHERE clause on the relationship pattern.
- **Description**: Add `last_reinforced_epoch` as a property on Neo4j `CO_RETRIEVED` relationship records, updated each time the co-retrieval edge is reinforced (its weight is incremented). In Cypher traversal queries, add `WHERE r.last_reinforced_epoch >= ($current_epoch - $tau_edge)` where `tau_edge` can be tuned independently from the node-level `tau` (e.g., tau_edge=30 for relationship staleness vs tau=20 for node staleness). This provides a second staleness dimension: a causal relationship is not traversed unless it was recently active, even if both its endpoints remain in the working-set.

## Key Finding

The Neo4j causal graph at Tim's scale will accumulate approximately 800 new edges per month (central estimate, range 300–1,350 based on retrieves-per-session variance). This is manageable volume — at Month 24, the graph contains roughly 19,200 edges across ~10,800 nodes, with average fanout near 2–3. The average-case traversal is never a problem. The danger is the **fanout distribution's tail**: the top 1% of hub nodes (high-frequency memories that appear in nearly every session) will accumulate 500–1,500 edges each by Month 12–24. A 1-hop BFS from any of these hub nodes at cold-cache conditions exhausts the 10ms graph sub-budget at approximately F=90, and exceeds the full 50ms LLPC budget at approximately F=1,600 (NVMe cache-warm). With 2-hop traversal, the quadratic amplification (O(F²) records) breaks the budget at average fanout F=20–30. The adversarial dense-subgraph case — 20 core memories forming a near-clique — does not threaten latency at fanout=19, but does create query-agnostic retrieval set inflation that delivers the same 19 neighbors regardless of query content.

The more critical finding is the **epoch filter gap**: the working-set boundary (`last_access_epoch >= current_epoch - tau`) established in Q062 and monitored via S4 in Q071 is applied to PostgreSQL and Qdrant retrieval paths but is absent from Neo4j causal graph traversal. This is not a theoretical concern — it is an active invariant violation that becomes more severe as the graph matures. At Month 12+, hub nodes carry edges to memories from all periods of their history, spanning many tau windows. Graph traversal without an epoch predicate routinely injects memories with `last_access_epoch = current_epoch - 80` into the context window through the Level 1.5 routing path, entirely bypassing the working-set gate. These stale memories are invisible to the health monitoring defined in Q071 (S4 only checks PostgreSQL epoch integrity, not graph traversal staleness). The fix is straightforward: add `last_access_epoch` as a Neo4j node property (dual-written at access time), add the epoch predicate to all Cypher traversal queries, and optionally add a `LIMIT 20 ORDER BY co_occurrence_count DESC` fanout cap to bound worst-case latency.

The combination of epoch-filtered traversal and a K=20 fanout cap converts the causal graph from a latency risk and staleness injection vector into a well-bounded, semantically coherent Level 1.5 routing signal. Without these two changes, the graph traversal component of LLPC will degrade retrieval quality and latency as the system ages — silently, since no existing health signal monitors it.
