# Finding: Q065 — Re-encounter Multiplier Operand: write_importance vs. prior_importance

**Question**: The re-encounter multiplier (Q049) applies a boost when a tombstoned memory is re-encountered — but which importance value should the multiplier act on: the write_importance of the newly arriving memory text (reflecting current context relevance), or the prior_importance stored in the tombstone (reflecting historical importance at eviction time)? Four candidate formulas: (a) `new_importance = write_importance × (1.0 + 0.15 × min(tombstone_age_sessions, 10))` — current signal; (b) `new_importance = prior_importance × (1.0 + 0.15 × min(tombstone_age_sessions, 10))` — historical signal; (c) `new_importance = max(write_importance, prior_importance) × (1.0 + 0.15 × min(tombstone_age_sessions, 10))` — conservative ceiling; (d) `new_importance = (α×write_importance + (1-α)×prior_importance) × multiplier` — weighted blend.
**Question Type**: ADVERSARIAL
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: LLPC admission gate semantics, write-time importance calibration, tombstone re-encounter mechanism (Q049, Q060)

---

## Thread A: Use write_importance — formula (a)

**Core claim**: The LLPC admission gate is a real-time relevance filter, and `importance` in that gate must reflect current-session relevance; a memory that re-arrives with low write_importance genuinely is less relevant now, regardless of its past.

### Argument

The LLPC gate's function is to decide whether a memory deserves a slot in the context window *at this moment*. The context window is a scarce resource — it must contain what is most relevant to the current prompt, not what was most important at some past moment. If the gate uses prior_importance (a signal from a potentially distant session), it is making admission decisions based on stale evidence.

Consider the counter-case for prior_importance: Tim stored "CasaOS uses eth0" at importance=0.80 during initial setup. Three months later, the infrastructure has been reconfigured — but a new observation about the same topic key arrives with write_importance=0.20, because the new observation is routine ("checking interface, confirmed ens18"). If formula (b) applies prior_importance=0.80, the re-encountered memory is stored at `0.80 × multiplier` — a very high importance — despite the current observation being routine and low-signal. The LLPC gate would then inject this memory into every related context window, because its importance score is anchored to a three-month-old peak, not to current relevance.

The admission gate should answer: "Does this memory matter right now?" The write_importance of the newly arriving text is the best available signal for that question. The extractor (observe-edit.js or the session summary hook) assigns write_importance based on the content's novelty, specificity, and relevance to the current session's activity. That score reflects the current context.

Formula (a) then makes sense as a corrective: the re-encounter multiplier does not change the nature of the signal — it amplifies write_importance to account for the fact that the topic survived long enough to recur, which is weak evidence of importance beyond the current session. The multiplier is a small persistent-recurrence bonus layered on top of a fundamentally current signal.

**Evidence**: Database systems that implement importance-weighted caching (e.g., scan-resistant LRU variants, PostgreSQL's clock-sweep buffer replacement) universally use the current access signal — the frequency and recency of the current request — not a stored historical peak. A cache entry's historical importance is considered only through its accumulated access-frequency counter, not as a frozen prior that overrides the current observation. The write_importance plays the role of the "current access signal" in the Recall admission gate.

**Best case scenario**: Formula (a) correctly lets the write_importance extractor determine whether a re-encountered memory is worth storing, then applies a small bonus for survivorship. A memory that recurs but is genuinely low-relevance in the current context (write_importance=0.15) is not artificially elevated. The gate remains sensitive to current context.

## Thread B: Use prior_importance or max() — formula (b) or (c)

**Core claim**: The tombstone mechanism's entire purpose is to rescue memories that were evicted not because they are unimportant but because write-time importance scoring systematically underestimates confirmed/expected facts; using write_importance as the base value defeats this purpose entirely.

### Argument

Q049 was motivated by a specific failure mode: a memory scored 0.35 at write time and evicted, where the correct importance was 0.55+. The write-time score was low because the extractor assigns low scores to confirmed, expected, already-known facts — which is the correct behavior for a novelty-weighted scorer, but wrong for long-term retention decisions. The tombstone mechanism exists precisely to correct this error.

Now consider the re-encounter scenario under formula (a): "CasaOS network interface is ens18" is written with write_importance=0.32 (confirmed fact, low novelty in the session) and evicted at importance=0.32 < 0.40. Five sessions later, it is re-encountered. The new observation "checking ens18 interface" arrives with write_importance=0.28 — again a confirmed fact, again low novelty, again below threshold.

Under formula (a): `new_importance = 0.28 × (1.0 + 0.15 × 5) = 0.28 × 1.75 = 0.49`. This survives the 0.40 threshold — but only barely, and only because the tombstone_age happened to be 5 sessions. If the re-encounter happens at session 2 (multiplier = 1.30), `new_importance = 0.28 × 1.30 = 0.36` — still below threshold. The memory is re-evicted. The exact scenario Q049 was designed to prevent has occurred.

The deeper problem: formula (a) multiplies a systematically-underestimated signal by a correction factor and hopes the corrected result crosses the threshold. But the correction factor (max 2.5× at age 10) cannot reliably compensate when the base write_importance is low enough. A memory with write_importance=0.20 reaches `0.20 × 2.5 = 0.50` only at maximum age. If it recurs at age 4 (multiplier=1.60), it reaches 0.32 — evicted again. Formula (a) only rescues memories where write_importance is already close to the threshold, not the deep underestimates that motivated Q049.

The tombstone carries prior_importance for a reason: it is evidence that the memory was once scored high enough to warrant storage (and only evicted later when the session-scope window-drop applied). Prior_importance represents the peak valuation of this memory — the best estimate of its importance before it aged out. Using prior_importance as the base for the multiplier ensures the re-encounter signal is applied to the best historical estimate, not the current-session (likely still-underestimated) write score.

Formula (c) — `max(write_importance, prior_importance)` — is even safer: if write_importance happens to be higher than prior_importance (the re-encountered fact is newly relevant in a different way), the current signal wins. If prior_importance is higher (the fact is being confirmed at lower novelty, as expected for recurring infrastructure facts), the historical peak wins. This is strictly better than either (a) or (b) alone, because it uses whichever signal is stronger.

**Evidence**: This is directly analogous to DNS negative caching (RFC 2308) combined with TTL refresh: when a DNS entry is evicted from the cache and re-queried, the resolver does not re-score the entry from the wire response alone — it considers the historical TTL as well, and uses `max(cached_TTL, new_TTL)` in some implementations (strict RFC-compliant resolvers use the new TTL, but many production resolvers use the cached value when higher, to prevent eviction of high-TTL entries by low-TTL updates). The historical signal is retained precisely because the current-query signal is insufficient to correctly value entries with long-lived operational importance.

More directly: the Q060 simulation confirmed that recurrence probability is nearly flat across importance bands (1.74-1.80%), meaning that low write_importance memories recur at essentially the same rate as higher write_importance ones. If recurrence rate does not correlate with importance, then write_importance at re-encounter time cannot be used to distinguish "genuinely unimportant re-encounter" from "important fact, again scored low because it is expected." Prior_importance, at least, carries the memory's best historical self-assessment.

**Failure mode**: Formula (a) perpetuates the write-time importance underestimation problem it was designed to correct. A memory that is repeatedly scored low at write time (because it is an expected, confirmed fact) and repeatedly evicted creates an importance-death spiral: each re-encounter produces a low write_importance base, and even with the multiplier, the result stays below threshold at early ages. The memory never accumulates importance above threshold unless it survives to a late tombstone age — which requires not being evicted in the intervening sessions, which requires crossing the threshold, which it cannot do. The spiral is self-locking.

## Synthesis

### What Thread A assumes that Thread B exposes

Thread A assumes that `importance` in the LLPC gate is a signal about current relevance — the answer to "does this memory belong in the context window now?" Under this assumption, write_importance (the current session's assessment) is the correct input, and the multiplier is a small bonus.

Thread B exposes that this assumption is wrong for the specific class of memories the tombstone mechanism was designed to rescue. For infrastructure-configuration facts, write_importance is structurally underestimated because the extractor assigns low scores to confirmed/expected content. The write_importance signal is not noisy — it is systematically biased downward for exactly the memories that need upward correction. Thread A's correction factor cannot compensate for a systematic bias when the correction is multiplicative on a biased base.

**The hidden assumption Thread A makes**: That write_importance at re-encounter time is a fair, unbiased estimator of current relevance. It is not — for confirmed infrastructure facts, write_importance at re-encounter is almost always low, regardless of the fact's operational importance. The extractor is designed to score novelty, not operational criticality.

**The hidden assumption Thread B makes**: That prior_importance is a meaningful upper bound — i.e., the memory was once scored high enough to be worth storing. This is not guaranteed. A memory written at importance=0.65 and later evicted via session-scope window-drop (which evicts below 0.40) presents a contradiction: if prior_importance=0.65 was above threshold, why was it evicted? Session-scope window-drop operates on session-scoped memories, which may have been stored at 0.65 but aged or been explicitly demoted before the window-drop ran. The tombstone's prior_importance field must store the importance *at eviction time*, not at write time — these can differ.

**When Thread A is right**: When the write_importance extractor is well-calibrated — i.e., when the current-session assignment of write_importance is a fair signal of relevance. This is true for general-purpose memories (notes, workflow observations, session summaries) that are not systematically underscored. For these memories, formula (a) correctly avoids inflating unimportant re-encounters.

**When Thread B is right**: When write_importance is systematically biased — specifically for infrastructure-configuration facts tagged with casaos, homelab, networking, docker. For these, Q060 confirmed that recurrence probability is nearly uncorrelated with importance band (1.74-1.80% across all bands), meaning write_importance at re-encounter cannot distinguish important from unimportant. Prior_importance or max() is necessary to break the re-eviction spiral.

### The architecture both threads are circling but neither states directly

The real question is: **what does `importance` represent in the LLPC admission gate?**

It represents one thing in the current design — a single float that encodes "how important is this memory for future retrieval" — but it is being asked to do two different jobs:

1. **Admission gate score**: "Should this memory be stored?" (determined at write time, biased by novelty)
2. **Retrieval ranking score**: "How likely is this memory to be useful?" (requires a different signal — retrieval frequency, recency, domain relevance)

These two jobs require different signals. The LLPC admission gate conflates them. Write_importance is a tolerable signal for job 1 (admission) but a poor signal for job 2 (ranking), because confirmed/expected facts are low-novelty (low write_importance) but high-retrieval-probability.

The tombstone re-encounter formula is asking: given this conflation, which single number should represent importance after a re-encounter? Neither write_importance alone nor prior_importance alone is correct, because neither cleanly answers either job alone.

### Chosen formula: (c) — max(write_importance, prior_importance) × multiplier

**Derivation from LLPC purpose:**

The LLPC admission gate's purpose is to prevent low-value memories from consuming context window slots. A memory that fails admission is excluded from retrieval ranking entirely — it consumes zero context. The gate therefore needs to be conservative: it should admit memories when there is evidence they belong, and reject when there is evidence they do not.

At re-encounter time, there are two independent pieces of evidence:
- `write_importance`: the current session's assessment of relevance
- `prior_importance`: the memory's best historical self-assessment (its peak score before eviction)

Both signals are informative. Neither is redundant. `write_importance` is informative about the current session; `prior_importance` is informative about the memory's historical relevance ceiling. The conservative admission decision is: admit if *either* signal (independently, before the multiplier) suggests the memory is worth keeping. This is exactly `max(write_importance, prior_importance)`.

The multiplier then scales the stronger of the two signals:

```
new_importance = max(write_importance, prior_importance)
                 × (1.0 + 0.15 × min(tombstone_age_sessions, 10))
```

This formula:

1. **Rescues deep underestimates** (Thread B's concern): A memory with write_importance=0.28 and prior_importance=0.55 produces `max(0.28, 0.55) = 0.55`. At tombstone_age=1, `new_importance = 0.55 × 1.15 = 0.63`. Well above the 0.40 threshold. The fact is rescued even at early re-encounter.

2. **Does not inflate genuinely low-relevance re-encounters** (Thread A's concern): A memory with write_importance=0.15 and prior_importance=0.20 (both legitimately low) produces `max(0.15, 0.20) = 0.20`. At tombstone_age=10, `new_importance = 0.20 × 2.5 = 0.50`. This does breach the threshold at maximum multiplier — which is acceptable, because a memory that persists for 10 sessions before re-encountering is genuinely rare (Q060: median recurrence probability is 0.35%, so 10-session-old tombstones represent memories that took a long time to recur, which is itself evidence of some persistence).

3. **Handles the write_importance > prior_importance case cleanly**: If the current observation assigns write_importance=0.60 but prior_importance=0.35 (the memory was originally low-importance, confirmed at higher importance now), formula (c) uses the current, higher signal: `0.60 × multiplier`. No distortion.

4. **Is unambiguous about what prior_importance means**: It must store the importance at eviction time, not at write time. The tombstone schema must capture `eviction_importance` (the score that was compared against the threshold), not the original write-time score. This distinction matters when memories are updated between write and eviction.

**The formula that should not be used:**

Formula (d) — the weighted blend — adds a free parameter α with no principled derivation. What should α be? 0.5 is arbitrary. 0.7 is arbitrary. The blend cannot be calibrated without empirical data on when write_importance is trustworthy vs. biased — which requires the very measurement Q060 recommends but has not yet produced. Formula (c) avoids this parameter entirely by using the maximum: it is conservative in the correct direction (admit when either signal is strong) without introducing a tunable hyperparameter.

**Pre-deployment lock-in decision**: Before tombstone deployment, the system must define `prior_importance` in the tombstone schema as the importance value *at eviction time* (the value that was compared against the LLPC threshold), not the value at write time. This is the operand for formula (c)'s prior_importance term. The tombstone schema field should be named `eviction_importance` to prevent this confusion.

## Ideas Extracted

### Idea: max-operand-re-encounter-formula
- **Novelty**: 0.72
- **Evidence**: 0.65
- **Feasibility**: 0.90
- **Description**: The re-encounter multiplier uses `max(write_importance, prior_importance)` as its base rather than either signal alone. This is derived from the LLPC gate's purpose: admit when either independent signal (current or historical) indicates the memory belongs. The formula `new_importance = max(write_importance, eviction_importance) × (1.0 + 0.15 × min(tombstone_age_sessions, 10))` rescues deep write-time underestimates (the primary motivation for tombstoning) without introducing a tunable blend parameter. Implementation: the tombstone schema stores `eviction_importance` (the score at eviction time) alongside `write_importance` (the original score). The Phase 1 scope discriminant computes max() at re-encounter time and applies the multiplier. No additional data collection is required beyond what the tombstone already stores per Q049's design.

### Idea: eviction-importance-vs-write-importance-schema-distinction
- **Novelty**: 0.68
- **Evidence**: 0.70
- **Feasibility**: 0.95
- **Description**: The tombstone schema must distinguish `eviction_importance` (the importance score at the moment of eviction — the value that was compared against the LLPC threshold) from `write_importance` (the score assigned when the memory was first created). These can differ: a memory written at importance=0.65 may have its importance recalibrated downward (via realized-access decay or manual correction) before being evicted at 0.35. Using `write_importance` as the tombstone's historical anchor would give the wrong prior. Storing `eviction_importance` as a distinct field prevents this error and makes the tombstone's semantic meaning unambiguous: "this memory was evicted when it had this score." Schema change: add `eviction_importance FLOAT NOT NULL` to the tombstone Redis hash structure, populated by the window-drop handler at eviction time from the memory's current importance value in PostgreSQL.

## Key Finding

The re-encounter multiplier should operate on `max(write_importance, eviction_importance)`, not either signal alone. Write_importance alone perpetuates the systematic underestimation of confirmed/expected infrastructure facts — the core failure mode Q049 was designed to rescue — because the importance extractor consistently assigns low scores to novelty-low (but operationally critical) confirmed facts. Prior importance alone risks anchoring to a historical peak that may no longer reflect current relevance. The max() formulation is conservative in the correct direction: it admits when either independent signal suggests the memory belongs, without introducing a tunable blend parameter that would require empirical calibration before deployment. The tombstone schema must store `eviction_importance` (the score at eviction time) as a distinct field from write-time importance to make this formula unambiguous.

## Suggested Follow-ups

- **[PHYSICS] Q066**: Measure the distribution of `(write_importance, eviction_importance)` pairs in Tim's actual session data — specifically, how often does `write_importance > eviction_importance` (importance grew between write and eviction) vs. `eviction_importance > write_importance` (importance decayed between write and eviction)? If the dominant case is decay (write high, evict low), then formula (b) using prior_importance would systematically anchor to an obsolete peak. If the dominant case is stability (write ≈ evict), the distinction between (b) and (c) is moot and formula (c)'s max() simplifies to either signal. This measurement requires the retrieve_events audit log to have timestamps that allow computing importance trajectory per memory_id.

- **[ADVERSARIAL] Q067**: Challenge formula (c)'s max() behavior specifically for the case where `prior_importance` is an outlier peak — a memory scored 0.85 during initial setup (high novelty) that decays to 0.22 before eviction. Formula (c) uses `max(write_importance=0.18, eviction_importance=0.22) = 0.22 × multiplier`, which is appropriate. But if the tombstone mistakenly stores `write_importance=0.85` instead of `eviction_importance=0.22`, formula (c) incorrectly produces `max(0.18, 0.85) = 0.85 × multiplier` — permanently elevating a memory that the system had legitimately demoted. This is the schema-confusion failure mode. Q067 should verify that the tombstone implementation correctly captures eviction-time importance, not write-time importance, and design a validation test that can detect the confusion in a deployed system.
