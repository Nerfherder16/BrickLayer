# Finding: Q212 — Working Memory Pools in Production Agent Frameworks

**Question**: Do LangChain, AutoGen, CrewAI, Semantic Kernel, or any other production agent framework implement agent-specific working memory pools that are distinct from long-term persistent memory? Search for: separate memory classes, task-scoped memory contexts, ephemeral vs. persistent memory distinctions, working set APIs.
**Agent**: quantitative-analyst
**Question Type**: [ABSENCE]
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: 7-framework survey (LangChain/LangGraph, AutoGen/AG2, CrewAI, Semantic Kernel, MemGPT/Letta, Haystack, LlamaIndex); official documentation sources verified 2025-2026; Q208 web-search protocol; Q211 physical requirements as evaluation criteria.

---

## Survey Methodology

Q211 derived 5 physical properties that distinguish a true working memory tier from long-term memory:

- **P1**: Bypasses embedding at write time (keyword/sparse extraction instead of dense float32 vectors)
- **P2**: Task-scoped (scope boundary = task completion event, not session/thread/agent)
- **P3**: Linear scan retrieval (not vector similarity / HNSW) — valid because bounded cardinality (N≤200)
- **P4**: RAM-only, no disk durability (data loss on restart acceptable)
- **P5**: Automatic promotion mechanism to long-term store at task end

Each framework's memory tier(s) were evaluated against these 5 properties. Sources: official documentation for each framework as of 2025-2026.

## Framework Survey Results

### LangChain / LangGraph

**Relevant tiers**: `MemorySaver` (short-term checkpointer), `InMemoryStore` (optional vector search)

| Property | `MemorySaver` | Notes |
|---|---|---|
| P1: No embed on write | YES | Raw graph state, no embedding |
| P2: Task-scoped | NO | Thread-scoped (`thread_id` key); no task lifecycle hook |
| P3: Linear scan | YES | Key-value lookup by thread ID |
| P4: RAM-only | YES | Lost on restart |
| P5: Promotion mechanism | NO | No built-in flush from `MemorySaver` to persistent store |

**Verdict**: Meets P1+P3+P4, fails P2+P5. Thread ≠ task. No task completion event in the memory API.

### AutoGen / AG2

| Property | `ListMemory` | Notes |
|---|---|---|
| P1: No embed on write | YES | Stores raw `MemoryContent` objects in a Python list |
| P2: Task-scoped | NO | Session-scoped; no task lifecycle boundary |

### CrewAI

**Relevant tiers**: `ShortTermMemory` (ChromaDB RAG), `LongTermMemory` (SQLite). Note: 2025-2026 docs show a unified LanceDB-based memory system superseding these legacy classes.

| Property | `ShortTermMemory` | Notes |
|---|---|---|
| P1: No embed on write | NO | Embeds to ChromaDB via configured embedder (OpenAI default) |
| P2: Task-scoped | PARTIAL | Scoped to `crew.kickoff()` — closest to task scope in any surveyed system |

| P4: RAM-only | NO | ChromaDB writes to `chroma.sqlite3` on disk; no in-memory-only option |
| P5: Promotion mechanism | NO | No automatic flush from ShortTermMemory to LongTermMemory at kickoff end |

**Verdict**: CrewAI's `kickoff()`-scoped memory is the only mechanism in the survey that approximates task scope (P2 partial). However, it fails P1, P4, and P5, and uses the expensive embedding path at write time — the exact constraint Q211 showed is incompatible with high-frequency working memory writes.

### Semantic Kernel

**Relevant tiers**: `ChatHistory` (conversation list), `WhiteboardProvider` (LLM-extracted context), `VolatileVectorStore`/`InMemory` (RAM vector store)

| Property | `ChatHistory` | `WhiteboardProvider` | `VolatileVectorStore` |
|---|---|---|---|
| P1: No embed on write | YES | YES (LLM-based) | NO (caller provides vectors) |
| P2: Task-scoped | NO (conversation) | NO (thread-scoped) | NO |
| P3: Linear scan | YES | YES (LLM selection) | YES (no index — brute-force) |
| P4: RAM-only | YES | YES | YES |
| P5: Promotion mechanism | NO | NO | NO |

**Note**: Semantic Kernel's `VolatileVectorStore` is notable — it stores embedding vectors but has no ANN index (connector spec explicitly marks `Supported index types: N/A`). Retrieval is brute-force linear scan over stored vectors. This is architecturally closest to Q211's "linear scan over bounded corpus" but still requires pre-computed dense vectors (not keyword extraction).

**Verdict**: Meets P1+P3+P4 on `ChatHistory`, but no framework memory tier in SK is task-scoped or has promotion.

**Status**: `WhiteboardProvider` and `Mem0Provider` are marked **experimental** as of mid-2025.

### MemGPT / Letta

**Relevant tiers**: Core memory (in-context), Recall memory (conversation history, SQLite), Archival memory (LanceDB vector store)

| Property | Core Memory | Recall Memory | Notes |
|---|---|---|---|
| P1: No embed on write | YES | YES | Core = text in system prompt; Recall = raw messages in SQLite |

**Verdict**: Letta has the most sophisticated tiered architecture in the survey (3 tiers with LLM-driven promotion), but is agent-scoped (permanent) rather than task-scoped. Recall memory's keyword search most closely matches the working memory retrieval semantics from Q211.

### LlamaIndex

**Relevant tiers**: Short-term FIFO queue (in `Memory` class), `VectorMemoryBlock` (long-term component)

| Property | Short-term FIFO | Notes |
|---|---|---|
| P1: No embed on write | YES | Raw `ChatMessage` objects; no embedding |
| P2: Task-scoped | NO | Session-scoped via `session_id`; flush trigger = token count, not task end |
| P3: Linear scan | YES | Messages injected in order; no similarity ranking in short-term |
| P4: RAM-only | YES | In-process FIFO queue |
| P5: Promotion mechanism | YES (token-based) | Automatic flush to `VectorMemoryBlock` when `token_flush_size` exceeded |

**Key observation**: LlamaIndex is the **only framework in the survey with an automatic promotion mechanism** (short-term FIFO → long-term `VectorMemoryBlock`). However, the trigger is a **token count threshold** (default 3000 tokens), not a task completion event. There is no concept of "task" as a scope boundary in the memory API.

**Verdict**: Closest overall to the Q211 design — meets P1+P3+P4+partial-P5, fails P2 (task scope). The promotion mechanism exists but fires on the wrong trigger.

## Critical Absence: Task Scope Boundary

The most structurally significant finding is not any single missing property — it is the **absence of the task as a memory scope boundary** across all 7 frameworks.

Every framework scopes memory to one of:
- **Session/thread** (LangGraph, AutoGen, Haystack, LlamaIndex)
- **Conversation turn** (Semantic Kernel `ChatHistory`)
- **Run/kickoff** (CrewAI — closest to task, but one level above)
- **Agent lifetime** (Letta — permanent)

None defines a scope boundary at the level of a discrete agent task — the unit of work that Q211 identified as the natural boundary for working memory. In Claude Code's architecture, the `Stop` hook fires when a task completes: this is the task-end event that triggers promotion in Q211's design.

**Why this boundary is absent**: The session-scoped designs predate multi-agent orchestration. A single-agent LLM assistant has one meaningful scope: the conversation. Multi-agent orchestration introduces sub-task scope as a distinct level below session and above turn. None of the surveyed frameworks have updated their memory architecture for this scope.

## LlamaIndex: The Closest Production Mechanism

LlamaIndex's `token_flush_size`-triggered promotion is the closest production implementation:

```python
# LlamaIndex Memory class (token-flush promotion)
memory = Memory.from_defaults(
    session_id="session_1",
    token_limit=40000,
    token_flush_size=3000,  # flush to long-term when short-term exceeds 3000 tokens
)
```

**Structural parallel to Q211 design**:
- Short-term FIFO → Recall 2.0 working memory tier
- `token_flush_size` trigger → Stop hook (task end) trigger
- `VectorMemoryBlock` → LMDB+HNSW long-term store

**The one-parameter gap**: Replacing `token_flush_size` (proxy trigger) with a task-completion event is architecturally trivial — it is a hook wiring change, not a redesign. In Claude Code: the `Stop` hook already fires at task end. The gap between LlamaIndex's mechanism and Q211's design is not architectural — it is the choice of trigger.

## Analytical Summary

**The working/long-term memory distinction is absent from all 7 production frameworks.** No framework implements all 5 physical properties identified in Q211 as necessary for a task-scoped working memory tier.

**The most significant absent property is P2 (task scope).** Every framework scopes memory to session, thread, or agent lifetime — not to discrete task completion. This is the deepest structural absence: the other 4 properties have partial implementations across the survey, but P2 has only one partial match (CrewAI `kickoff()`-scoped) that still embeds on write.

**LlamaIndex is the closest production mechanism.** It has automatic promotion (P5), no-embed short-term tier (P1), linear scan (P3), and RAM-only short-term (P4). The single missing property is task scope (P2) — the flush trigger is token count rather than task completion. This is a trigger semantics gap, not an architectural gap.

**The absence confirms Q211's design as genuinely novel.** The complete 5-property combination does not exist in production. Recall 2.0's working memory tier would be the first production implementation to wire all 5 properties together with a task-completion event as the scope boundary.
