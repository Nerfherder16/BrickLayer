## Adversarial Analysis: Sentinel Hardening vs. Importance Score Suppression

---

### Thread A: Defense — Sentinel hardening restores rank-1 for the health check

**Core claim**: The composite score inversion identified in Q127 does not affect the `/health/recall-quality` endpoint because that endpoint queries Qdrant directly via raw ANN, bypassing the importance-weighted composite scoring entirely. Sentinel-token hardening restores the raw cosine similarity gap to >0.10, which is sufficient to guarantee rank-1 in the health check's actual scoring path.

**Evidence**:

Q103's `_run_single_canary_query()` implementation calls `await qdrant.search(collection_name="memories", query_vector=vector, limit=10, with_payload=False)`. This is a direct Qdrant ANN call. Qdrant's `search()` endpoint returns points ranked by raw cosine similarity of the query vector against stored vectors, unmodified by any application-layer importance score. The `importance` field stored as a Qdrant payload is not a query-time scoring parameter in this call — it exists as metadata for the Recall API's retrieve endpoint to apply post-ANN reranking.

Canary seeds are deliberately inserted with `"importance": 0.0` (Q103, line 449 verbatim) to suppress them from surfacing in real user retrieval queries. This is the correct design: canaries should be invisible to Tim's actual prompts. The health check queries Qdrant directly, so the same `importance: 0.0` that hides canaries from users is irrelevant to whether the health check passes.

Sentinel-token content hardening increases raw cosine similarity of the canary content relative to its paraphrase query from the thin gap range (0.0059 for CANARY_004, 0.0016 for CANARY_005) to an estimated 0.10-0.18 range, based on the natural sentinel analogue in CANARY_006 (gap = 0.2157). At that raw cosine gap, CANARY_004 and CANARY_005 will be rank-1 in the Qdrant ANN search — which is exactly what `_run_single_canary_query()` evaluates.

**Best case scenario**: Sentinel-hardened CANARY_004 and CANARY_005 are inserted. The health check runs and calls `qdrant.search()` with the paraphrase queries. Both seeds return at rank 1 with raw cosine similarity 0.75+, gap 0.10+ over the next-best organic memory. The health check reports 10/10 pass. The composite score inversion that Q127 documented is real but is a property of the *retrieval* path, not the *health check* path. Both paths exist correctly in parallel.

**Core claim**: Sentinel hardening is sufficient because the health check path is immune to importance score suppression by architectural design.

---

### Thread B: Attack — Importance score inversion is structural and the health check design is unsound

**Core claim**: The health check's reliance on raw ANN ranking creates a divergence between what the health check reports and what users actually experience during retrieval. Even if the health check passes 10/10 after sentinel hardening, the retrieval quality for the canary topic regions is degraded — organic memories with high importance scores are crowding out the semantically relevant results in real sessions. The canary is testing the wrong thing.

**Evidence**:

The purpose of `/health/recall-quality` as defined in Q103 is to verify that "semantic retrieval is functioning correctly" — that the system can find a known memory when queried for it. But users query via the Recall API `/search/query` endpoint, which applies composite scoring. A health check that passes on raw ANN but would fail on composite scoring is not measuring retrieval quality as experienced by the user.

Organic memories in the `recall` domain have average importance 0.52 (the highest domain average, Q127). With the composite formula `score = similarity × importance + graph_boost`, an organic memory with similarity 0.6485 (matching CANARY_004's raw rank-1) and importance 0.52 scores `0.6485 × 0.52 = 0.337`. A canary with importance 0.0 and similarity 0.75 (post-hardening estimate) scores `0.75 × 0.0 = 0.0`. The canary is rank-last in composite scoring regardless of sentinel hardening.

This means: after sentinel hardening, the health check says "healthy" but Tim's actual queries for "what importance score does postgres use as threshold" return organics with high importance scores rather than the canary memory covering that exact fact. The sentinel hardening creates a false signal of health. The composite score inversion is not a measurement artifact — it is a structural consequence of the `importance: 0.0` design choice interacting with the composite retrieval path.

Furthermore, the canary seeds have zero co-retrieval history — they are never retrieved during real sessions because they are invisible (importance: 0.0). Co_retrieval reinforcement in the Neo4j graph layer is therefore zero for all canary seeds. Any organic memory that is repeatedly retrieved alongside similar queries accumulates graph boost signals. The gap between canary and organic composite scores widens permanently over time.

**Failure mode**: The health check becomes a false-positive oracle. It reports "healthy" with 10/10 pass rate while the actual retrieval path for the same query topic is returning high-importance organic memories, some of which may be stale, incorrect, or less specific than the canonical canary content. Users experience degraded retrieval quality in exactly the regions the health check was designed to monitor. The divergence grows with corpus maturity.

### Synthesis

**What Thread A assumes that Thread B exposes**:

Thread A assumes the health check's purpose is to verify raw ANN integrity — that Qdrant can find a specific vector given a semantically related query. This is a correct and achievable goal, and sentinel hardening fully addresses it.

Thread B exposes that the health check was *described* in Q103 as verifying "semantic retrieval quality" — the quality as experienced by users. These are not the same thing. The hidden assumption in the health check design is that raw ANN rank-1 is a proxy for composite-scored retrieval quality. Q127's composite score inversion measurements prove this assumption fails for high-importance-domain seeds.

Both threads are right about their respective targets. Thread A is right about what the current implementation measures. Thread B is right about the gap between what it measures and what users experience.

**When Thread A is right**:

Thread A is fully correct about the health check's operational behavior. Sentinel hardening restores rank-1 in the raw ANN path. The health check will pass 10/10 after hardening. For the narrow goal of detecting infrastructure failures — Qdrant unreachable, HNSW graph disconnected, canary point deleted, embedding service down — the raw ANN health check is sound. These failure modes produce rank failure regardless of importance scores.

Thread A is also right that `importance: 0.0` for canaries is the *correct* design for keeping canaries invisible to real retrieval. Raising canary importance would pollute Tim's actual memory results with system probes — an unacceptable side effect.

**When Thread B is right**:

Thread B is correct that the health check does not measure retrieval quality in the composite-scored path. For CANARY_004 and CANARY_005 specifically, where the topic regions (PostgreSQL importance thresholds, Redis epoch key) are actively used in Tim's sessions, the composite-scored retrieve endpoint returns different rank orderings than the raw ANN health check. If the goal is to monitor whether Tim can retrieve the correct answer about these topics, the health check does not test that goal.

Thread B is also correct about the long-term trajectory: canary composite scores are permanently pinned near 0.0 (importance: 0.0 is static), while organic memory composite scores grow via repeated retrieval and co-retrieval graph reinforcement. The divergence between the two scoring paths widens monotonically.

**The architecture Thread B would accept**:

The correct design separates two distinct health signals that the current implementation conflates:

**Signal 1 — Infrastructure integrity** (raw ANN path, current implementation, works correctly after sentinel hardening):
- Calls `qdrant.search()` directly with canary paraphrase queries
- Pass condition: canary UUID at rank 1 in raw cosine similarity
- Detects: Qdrant down, HNSW disconnected, canary deleted, embedding service failure
- Canary importance: 0.0 (correct — canaries must be invisible to user retrieval)
- This signal is healthy after sentinel hardening. No change needed to the mechanism.

**Signal 2 — Retrieval quality in the composite-scored path** (new signal, not yet implemented):
- Calls the Recall API `/search/query` endpoint with probe queries (not canary UUIDs — different mechanism)
- Uses "oracle memories": high-importance organic memories with known ground-truth content, stored at importance ≥ 0.8, covering the same topic regions as canaries
- Pass condition: the oracle memory UUID appears at rank 1 in composite-scored results for its topic query
- Detects: importance score drift, composite scoring misconfiguration, topic crowding by low-quality high-importance memories
- Oracle memories are *not* canaries — they are real high-importance memories about known facts, created during system setup and verified periodically. They accumulate co-retrieval signal naturally.

The health endpoint response should aggregate both signals:

```json
{
  "status": "healthy",
  "infrastructure_check": { "pass_count": 10, "fail_count": 0, "pass_rate": 1.0 },
  "retrieval_quality_check": { "pass_count": 8, "fail_count": 2, "pass_rate": 0.8, "seeds": [...] },
  "overall_pass": true
}
```

A system can be infrastructure-healthy but retrieval-quality-degraded. The two checks diagnose distinct failure classes with distinct remediation paths.

[FINDING] The composite score inversion for CANARY_004 and CANARY_005 is **not a threat to the health check** — `_run_single_canary_query()` bypasses importance weighting entirely. Sentinel hardening is sufficient to restore raw ANN rank-1. The health check will function correctly after Q127's recommended hardening.

[FINDING] The composite score inversion **is a real gap in health observability**. The health check does not test retrieval quality as experienced by users in the composite-scored path. This is a design scope limitation, not a design defect — but it means passing health check scores do not rule out composite-path retrieval degradation.

[FINDING] The correct architectural response is not to redesign the canary health check — it is to add a second, orthogonal health signal using oracle memories that test the composite-scored retrieval path. These are structurally different probes with different importance score requirements and different failure modes.

[STAT:n] CANARY_004 composite score gap = -0.0006 (inverted); CANARY_005 = -0.0075 (inverted). Both irrelevant to health check scoring path.
[STAT:effect_size] Canary importance = 0.0 in health check path score: `0.0 × similarity = 0.0`. Qdrant search() returns raw cosine similarity rankings, not composite scores. The formula `similarity × importance` is applied only in the Recall API retrieve layer.

## Qualifying Ideas

### dual-path-health-architecture
- (N=0.82, E=0.85, F=0.78)
- The `/health/recall-quality` endpoint should be explicitly decomposed into two named sub-checks: `infrastructure_check` (raw ANN via direct Qdrant search, canary importance=0.0, tests vector store integrity) and `retrieval_quality_check` (composite-scored path via Recall API, oracle memories with importance≥0.8, tests end-to-end ranking). Each check returns its own pass_count/pass_rate. The two checks diagnose orthogonal failure classes: one catches infrastructure failures, the other catches scoring-path degradation. Currently only the infrastructure check exists. The retrieval_quality_check is the missing half.

### oracle-memory-probe-design
- (N=0.78, E=0.72, F=0.82)
- A distinct class of health probe — "oracle memories" — should complement canary seeds. Oracle memories are real, high-importance (importance≥0.8), human-verified memories about known facts, inserted during system setup and assigned the `is_oracle: true` payload flag. Unlike canaries (importance=0.0), oracle memories accumulate co-retrieval signal naturally. The health check queries the Recall API retrieve endpoint with oracle topic queries and verifies oracle UUIDs appear at composite rank 1. Oracle displacement (a high-importance organic memory outranks the oracle in composite scoring) indicates genuine retrieval quality degradation in the topic region. Oracle memories serve the goal Thread B identified; canary seeds serve the goal Thread A identified.

### importance-score-path-annotation-in-health-response
- (N=0.68, E=0.88, F=0.95)
- A low-cost immediate improvement: the `/health/recall-quality` response should include a `scoring_path: "raw_ann"` annotation per query result, making explicit that pass/fail reflects raw ANN rank, not composite-scored rank. Add a `composite_score_gap` field (currently tracked in Q127 but not surfaced in the health endpoint) computed post-hoc: re-query the Recall API retrieve endpoint for each seed's query text, extract the composite score of the canary UUID (will be 0.0 × importance = 0.0 if importance=0.0), and report it. This surfaces the Thread B gap without requiring oracle memory infrastructure.

## Recommendations

1. **Proceed with sentinel hardening for all 7 HIGH RISK seeds** (Q127 recommendation stands). The composite score inversion does not affect the health check path. Hardening restores raw ANN rank-1 for CANARY_004 and CANARY_005. This is the correct immediate action.

2. **Do not raise canary importance scores** to fix the composite score inversion. Raising importance above 0.0 would cause canary content to surface in Tim's real retrieval sessions — defeating the design goal of invisible probes. The `importance: 0.0` design is correct and intentional.

3. **Clarify the health check's scope in code comments and documentation**: `_run_single_canary_query()` tests raw ANN retrieval integrity, not composite-scored retrieval quality. Add a docstring: "This function tests Qdrant ANN search correctness via direct client call. It does not test the composite scoring path used by the Recall API retrieve endpoint."

4. **Design and implement oracle memory probes** as a second health signal (dual-path-health-architecture idea above). Oracle memories are the correct mechanism for monitoring composite-scored retrieval quality. This is a follow-on implementation task, not a prerequisite for the Saturday deployment.

5. **Q127's recommendation #2** ("Add raw cosine similarity tracking to `/health/recall-quality`") remains valid — it adds gap monitoring to the existing infrastructure check. However, re-frame it as monitoring the infrastructure check's margin-of-safety (similarity gap), not as a fix for composite score inversion.

The structural argument holds regardless: importance=0.0 canaries will always score below positive-importance organics in any composite formula that includes importance as a positive additive or multiplicative term.
