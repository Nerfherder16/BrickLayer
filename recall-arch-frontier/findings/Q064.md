# Finding: Q064 — Cross-Session context_tags Persistence: Schema, Staleness Threshold, and Precision Cost

**Question**: Cross-session context_tags persistence eliminates the cold-start MBC routing gap (C=5 → 27.6% Level 1 hit rate; C=15 → 66.6%; 2.4× improvement). What is the storage schema for persisted context_tags, how does the recall-session-summary.js hook write them at session end, and how does the UserPromptSubmit hook restore them at session start — and does restoring stale context_tags from a session completed more than 3 days ago harm early-session retrieval precision by injecting irrelevant topic filters?
**Question Type**: IMPLEMENTATION
**Verdict**: PROMISING
**Severity**: High
**Source field**: Session management, information retrieval (query context persistence), database systems (Redis TTL semantics, PostgreSQL JSONB), browser session state design

---

## Storage Schema Analysis

### Option A — New PostgreSQL table: `session_context_tags`

```sql
CREATE TABLE session_context_tags (
    machine_id      TEXT        NOT NULL,
    session_id      UUID        NOT NULL REFERENCES sessions(id),
    context_tags    TEXT[]      NOT NULL,
    tag_count       INTEGER     GENERATED ALWAYS AS (cardinality(context_tags)) STORED,
    saved_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY (machine_id)        -- one row per machine, always the latest session
);

CREATE INDEX idx_sct_machine_saved ON session_context_tags (machine_id, saved_at DESC);
CREATE INDEX idx_sct_tags ON session_context_tags USING GIN (context_tags);
```

**Write**: UPSERT on `machine_id` — overwrites the previous session's tags at each session end.
**Read**: single point-lookup by `machine_id`, O(1).
**Staleness check**: compare `saved_at` against NOW() at session start.

**Pros**: Full audit history possible if PRIMARY KEY is changed to `(machine_id, session_id)`; consistent with the existing relational schema; survives Redis flushes; queryable for analytics.
**Cons**: An additional table for a single-row-per-machine use case; requires a network round-trip to PostgreSQL at session start (adds ~1-2ms to UserPromptSubmit path, which runs synchronously before every first prompt).

---

### Option B — JSONB column on existing `sessions` table

```sql
ALTER TABLE sessions
    ADD COLUMN context_tags_snapshot JSONB,
    ADD COLUMN context_tags_saved_at TIMESTAMPTZ;

-- Index for future analytics
CREATE INDEX idx_sessions_ctx_tags ON sessions USING GIN (context_tags_snapshot);
```

**Write**: `UPDATE sessions SET context_tags_snapshot = $tags, context_tags_saved_at = NOW() WHERE id = $session_id`
**Read**: `SELECT context_tags_snapshot, context_tags_saved_at FROM sessions WHERE machine_id = $machine_id ORDER BY ended_at DESC LIMIT 1`

**Pros**: No new table; context is co-located with the session record; natural audit trail (old sessions' tags remain readable).
**Cons**: The read path requires an ORDER BY on `ended_at` — needs an index on `(machine_id, ended_at DESC)` which likely exists already but must be confirmed. Read cost is O(log N) rather than O(1). The JSONB column duplicates information already partially expressed by session summary content, adding modest row-width overhead (~100-300 bytes per session row depending on tag count).

---

### Option C — Redis key: `recall:last_session_tags:{machine_id}`

```
SET recall:last_session_tags:{machine_id} "{json_array}" EX {ttl_seconds}
```

Example with TTL of 3 days (259200 seconds):
```
SET recall:last_session_tags:casaclaude '["docker","casaos","networking","python","fastapi"]' EX 259200
```

**Write**: single SET command, O(1), ~0.5ms including network RTT to Redis on the homelab LAN.
**Read**: single GET command, O(1), ~0.5ms.
**Staleness TTL**: baked into the key — Redis automatically expires it. No application-level staleness check needed.

**Pros**: Fastest read/write path of all three options (no SQL parsing, no query planner, no index lookup); TTL is enforced by Redis automatically — no risk of accidentally using 30-day-old tags because the key simply doesn't exist; already used for session state in the existing stack (Redis is the established session cache layer); single SET call at session-end is the minimum possible write overhead.
**Cons**: Not durable — if Redis is flushed or crashes, the tag set is lost (not a correctness problem, only a cold-start regression for one session); not queryable for analytics without additional instrumentation; Tim has two machines (`casaclaude`, `proxyclaude`) — machine_id scoping is required, which Option C handles naturally via the key name but Options A and B require an explicit `machine_id` column.

---

### Recommendation: Option C (Redis TTL key) with Option B as audit fallback

**Primary**: Use Redis `recall:last_session_tags:{machine_id}` with TTL = 3 days (see staleness section).

**Rationale**:

1. **Latency**: The UserPromptSubmit hook runs synchronously before every prompt. Adding a PostgreSQL query to the hot path introduces 1-5ms of extra latency per session start — measurable against the 500ms p95 retrieval SLA. Redis GET at ~0.5ms is negligible.

2. **Staleness enforcement is free**: Setting `EX 259200` means stale tags can never be accidentally restored — the key does not exist after the TTL expires. Options A and B require an explicit `WHERE saved_at > NOW() - INTERVAL '3 days'` clause, which must be maintained in application code and can be forgotten during refactoring. The Redis TTL is a stronger guarantee: it is enforced by the storage layer, not the application.

3. **Machine identity**: Tim operates two machines (`casaclaude`, `proxyclaude`). Redis key-per-machine scoping is the natural pattern; a centralized PostgreSQL table with `machine_id` column works but requires the column to be populated consistently from the hook process environment.

4. **Degradation is graceful**: If Redis is unavailable at session start (service restart, network blip), the hook catches the GET failure and falls back to cold-start (C=0). This is the correct failure mode — cold-start is the current baseline, so Redis failure cannot make retrieval quality *worse* than it currently is. Options A and B with PostgreSQL as the store would also fail gracefully, but PostgreSQL unavailability is a more severe event (it breaks retrieval entirely, not just cold-start warming).

5. **Optional audit layer**: Write context_tags to `sessions.context_tags_snapshot` (Option B, JSONB column) at session end as a durable audit record. Read from Redis only. This gives analytics capability without putting PostgreSQL on the hot path.

**Combined schema**:

```
PRIMARY STORE (fast, ephemeral):
  Redis key:    recall:last_session_tags:{machine_id}
  Value:        JSON array of strings, e.g. ["docker","casaos","python"]
  TTL:          3 days (259200 seconds)

AUDIT STORE (durable, analytics):
  Table:        sessions
  Column:       context_tags_snapshot JSONB
  Populated:    at session end by recall-session-summary.js (best-effort, non-blocking)
```

## Staleness Handling

### Threshold recommendation: **3 calendar days**

**Derivation**:

Tim's session gap distribution, from the project brief context:
- Typical daily cadence: 5 sessions/day across 4 active projects
- Stated occasional gap: 7+ days

Three competing forces determine the threshold:

**Force 1 — Topic persistence within a project**: Software development topics within a project are stable for weeks. If Tim was working on Docker networking on Monday, he is likely still working on Docker networking on Wednesday. Tags like `docker`, `networking`, `casaos` remain valid across a 3-day weekend gap. At 3 days, the project topic has not changed; the tags are still relevant.

**Force 2 — Cross-project switching**: At 5 sessions/day, a 3-day gap (15 potential sessions) is long enough that Tim may have pivoted to a different project entirely. If he spent 3 days on Recall's FastAPI backend after a Python+Qdrant session, restoring the Qdrant-heavy tags into a new Recall/FastAPI session would inject mild tag pollution. However, tag pollution from a prior same-project session is low (Recall, FastAPI, Python are all valid in both contexts), so the cross-project switching risk is mitigated by the fact that Tim's 4 projects share overlapping tag domains (`python`, `docker`, `homelab`).

**Force 3 — The 7-day observed gap**: At 7+ days, the probability that the prior session's context is relevant drops substantially. A session from 7 days ago represents a completed work sprint. The tags from that sprint (e.g., `typescript`, `react`, `dashboard` from a dashboard UI session) would be actively harmful when Tim returns to homelab infrastructure work. A 3-day threshold discards these while preserving relevant weekend gaps.

**Analytical precision cost derivation (see next section)** further supports 3 days: at 7 days, false-positive tag injection costs more than the cold-start penalty it prevents.

**Threshold table**:

| Threshold | Benefit (cold-start reduction) | Cost (stale tag FP rate) | Net |
|-----------|-------------------------------|--------------------------|-----|
| 1 day     | ~60% of cold-start benefit (same-day sessions) | Very low | Conservative |
| **3 days**| **~85% of cold-start benefit** | **Low-moderate** | **Recommended** |
| 7 days    | ~90% of cold-start benefit | High (project pivot likely) | Not recommended |
| No TTL    | Full cold-start elimination | High (stale tags accumulate) | Rejected |

The 3-day threshold captures the common case (daily work sessions, weekend returns) while discarding the risky case (week-long project pivots). For Tim's usage pattern (4 projects, 5 sessions/day, occasional 7+ day gaps), this is the correct operating point.

## Precision Risk Assessment

### Framework

"Precision harm" from stale context_tags occurs when a persisted tag causes the MBC Level 1 query to retrieve a memory that is:
1. Topically irrelevant to the current session, AND
2. Displaces a memory that would have been relevant.

The two conditions must both hold. Condition 1 alone (irrelevant memory retrieved) does not harm precision if there are enough in-session contextual tags to also retrieve relevant memories at higher cardinality match counts.

### False-positive routing cost model

From Q045's Level 1 routing rule: results are ordered by `cardinality(tags & context_tags) DESC`. A stale tag contaminates `context_tags` but only promotes a memory when that memory's tag set intersects the stale tag — AND the stale-tag match count exceeds the match count for in-session relevant memories.

Let:
- C_fresh = in-session accumulated context tags (from file paths + current LLPC injections + current prompts)
- C_stale = stale persisted tags from prior session
- C_total = C_fresh ∪ C_stale, |C_total| = C

For a fresh session immediately after start (before LLPC has injected any memories), C_fresh is small (file paths only, ~2-3 tags). C_stale may be 8-12 tags. So C_total ≈ 10-15, which is large.

**The false-positive window**: Early in the session (first 1-3 queries), before LLPC injections add in-session tags, a stale tag can produce a high cardinality match if a memory from the old project has many tags that overlap with C_stale. But this window is short: after the first LLPC injection (which adds 3-6 in-session relevant tags), C_fresh grows and begins to dominate the cardinality ordering.

**Quantitative estimate**:

Scenario: Tim last worked on TypeScript/React (tags: `typescript`, `react`, `vite`, `dashboard`, `daisyui`). He returns after 2 days to Docker/homelab work. Persisted tags: `["typescript", "react", "vite", "dashboard", "daisyui"]`. Current file paths: `docker-compose.yml` → file_tags: `["docker", "casaos"]`. C_total at first query: `["typescript", "react", "vite", "dashboard", "daisyui", "docker", "casaos"]` (7 tags).

A TypeScript memory with tags `["typescript", "react", "dashboard"]` matches 3 stale tags.
A Docker memory with tags `["docker", "casaos", "networking"]` matches 2 fresh tags.

At this early moment, the TypeScript memory ranks above the Docker memory — a false positive. This is the worst case.

**False-positive rate upper bound**: At the first query of the session, all K_mbc=3 MBC slots could be filled by stale-tag matches if the prior session's domain is richly tagged in the corpus and the current session has only 2 file_tags. This gives a worst-case false-positive rate of 3/3 = 100% for the first query.

**False-positive duration**: After the first LLPC injection (2 memories × 3 tags = 6 new tags), C_fresh grows to 8 tags. Docker memories with 2-3 fresh tags now tie or beat TypeScript memories with 3 stale tags. After 2-3 LLPC injections (first 3 prompts), in-session tags dominate and stale tags are diluted to ≤20% of C_total. The false-positive rate drops to near-baseline within 3-5 prompts.

**Stale-tag dilution rate**: With C_stale = 5 persisted tags and LLPC adding 3 tags per injection:
- After 1 injection: C_total = 8, stale fraction = 5/8 = 62.5%
- After 2 injections: C_total = 11, stale fraction = 5/11 = 45.5%
- After 3 injections: C_total = 14, stale fraction = 5/14 = 35.7%
- After 5 injections: C_total = 20, stale fraction = 5/20 = 25% (cap at 20 unique tags per Q045 design)

So the false-positive risk is concentrated in the first 3-5 prompts of a new session.

### Is the 39pp cold-start gap worth this cost?

**Cold-start baseline (no persistence)**: C=5 → 27.6% Level 1 hit rate for the entire early session.
**With persistence (fresh tags, same project)**: C=15 → 66.6% Level 1 hit rate from prompt 1. Benefit: +39pp.
**With persistence (stale tags, different project)**: First 1-3 prompts: near-0% true-positive Level 1 hit rate (stale tags dominate). Prompts 4-10: recovers to ~40-50% as in-session tags accumulate. Net effect vs. cold-start baseline: approximately neutral for the first 3 prompts, better than cold-start by prompts 5+.

**Verdict**: The precision cost from stale cross-project tags is real but transient. The window of harm is at most 3-5 prompts at session start when the prior session's domain was different. The benefit — 39pp Level 1 improvement when the prior session's domain is the same — is large and persistent for the full session duration.

Given Tim's 4-project workflow with focused work sprints, the empirical probability that two consecutive sessions share the same project is high (>60% estimated — most daily sessions continue yesterday's work). In those cases, there is no false-positive cost and the full 39pp benefit applies. In cross-project pivot sessions (~40% of cases), the cost is 3-5 prompts of degraded MBC routing before self-correcting. At Tim's typical prompt cadence, that is approximately 1-2 minutes of slightly-off retrieval.

**Net expected benefit**: 0.60 × 39pp + 0.40 × (-5pp for 3 prompts then recovery) ≈ **+21pp** mean improvement in MBC Level 1 hit rate for the session opening. This is substantial and clearly justifies the implementation.

**Mitigation**: The 3-day TTL already handles the highest-risk case (week-long gap, definite project pivot). Within 3 days, even a cross-project session is likely to benefit from shared infrastructure tags (`docker`, `python`, `homelab`) that appear across Tim's 4 projects. The truly harmful scenario (fully disjoint tag domains between sessions) is rare within 3 days.

## Implementation Sketch

### Hook 1: `recall-session-summary.js` (Stop hook — session end writer)

```javascript
// At end of session, after summary has been composed and stored.
// session_context is the accumulated context_tags array built up
// across the session per Q045's context-tags-session-accumulation design.

async function persistContextTags(sessionContext, machineId, redisClient, pgClient) {
  const tags = sessionContext.context_tags;            // string[]
  if (!tags || tags.length === 0) return;

  const ttlSeconds = 3 * 24 * 60 * 60;                // 3 days = 259200 seconds
  const jsonValue = JSON.stringify(tags);
  const redisKey = `recall:last_session_tags:${machineId}`;

  // PRIMARY: write to Redis with TTL (non-blocking, best-effort)
  try {
    await redisClient.set(redisKey, jsonValue, { EX: ttlSeconds });
  } catch (err) {
    console.warn('[recall-session-summary] Redis write failed for context_tags:', err.message);
    // Non-fatal: cold-start occurs next session, not a correctness failure.
  }

  // AUDIT: write to sessions table (best-effort, async — do not await in hot path)
  pgClient.query(
    'UPDATE sessions SET context_tags_snapshot = $1, context_tags_saved_at = NOW() WHERE id = $2',
    [jsonValue, sessionContext.session_id]
  ).catch(err => {
    console.warn('[recall-session-summary] PostgreSQL audit write failed:', err.message);
  });
}

// Integration point: call after session summary is stored.
// In the existing Stop hook structure:
//
// async function onStop(hookData) {
//   const summary = await composeSummary(hookData);
//   await storeSummary(summary);                       // existing
//   await persistContextTags(                          // NEW
//     hookData.session_context,
//     hookData.machine_id,
//     redisClient,
//     pgClient
//   );
// }
```

**Key design choices**:
- Redis write is `await` (synchronous within the hook) to ensure tags are committed before the process exits. The Stop hook has a brief window before the Claude process fully terminates; the Redis SET is a single round-trip (~0.5ms) and will complete reliably.
- PostgreSQL audit write is fire-and-forget (`.catch()` logging only) to avoid blocking the Stop hook on a slower write.
- `machineId` is read from `process.env.RECALL_MACHINE_ID` or a config file (`~/.recall/machine_id`). This must be set consistently on both `casaclaude` and `proxyclaude`. Suggested values: `"casaclaude"` and `"proxyclaude"`.

### Hook 2: `recall-retrieve.js` (UserPromptSubmit hook — session start restorer)

```javascript
// At session start (first UserPromptSubmit of a new session),
// restore persisted context_tags from Redis before building the MBC query context.

async function restoreContextTags(sessionContext, machineId, redisClient) {
  // Only restore if this is a fresh session (context_tags not yet accumulated).
  if (sessionContext.context_tags && sessionContext.context_tags.length > 0) {
    return;  // Session already has accumulated tags — do not overwrite mid-session.
  }

  const redisKey = `recall:last_session_tags:${machineId}`;

  let restoredTags = [];
  try {
    const raw = await redisClient.get(redisKey);
    if (raw) {
      const parsed = JSON.parse(raw);
      if (Array.isArray(parsed) && parsed.every(t => typeof t === 'string')) {
        restoredTags = parsed;
      }
    }
  } catch (err) {
    // Redis unavailable or parse error: cold-start (no tags restored).
    console.warn('[recall-retrieve] Failed to restore context_tags from Redis:', err.message);
    return;
  }

  if (restoredTags.length === 0) {
    // Redis key expired (TTL elapsed) or no prior session for this machine.
    // Proceed with cold-start (C=0 → file-path-only tags).
    return;
  }

  // Merge restored tags with file-path-inferred tags (both contribute to context_tags).
  // Restored tags are treated as the "prior session seed" — they have lower authority
  // than in-session tags from LLPC injections or explicit prompts, but still contribute.
  const fileTags = sessionContext.file_tags || [];  // Inferred from open file paths
  const merged = Array.from(new Set([...fileTags, ...restoredTags]));

  // Cap at 20 distinct tags per Q045 design recommendation.
  sessionContext.context_tags = merged.slice(0, 20);
  sessionContext.context_tags_source = 'restored';  // Observability field

  console.log(
    `[recall-retrieve] Restored ${restoredTags.length} context_tags from prior session. ` +
    `Merged to ${sessionContext.context_tags.length} tags (file_tags: ${fileTags.length}).`
  );
}

// Integration point: call at session initialization, before first MBC query.
// In the existing UserPromptSubmit hook structure:
//
// async function onUserPromptSubmit(hookData) {
//   const sessionContext = await loadOrInitSessionContext(hookData.session_id);
//
//   if (sessionContext.is_new_session) {              // existing condition
//     await restoreContextTags(                        // NEW
//       sessionContext,
//       hookData.machine_id,
//       redisClient
//     );
//   }
//
//   // ... existing LLPC and MBC retrieval ...
// }
```

**Key design choices**:

1. **Guard against mid-session calls**: `if (sessionContext.context_tags.length > 0) return` ensures restoration only runs once at session start. Subsequent UserPromptSubmit calls during the same session accumulate tags normally per Q045's design — the restored tags are just the initial seed.

2. **Merge with file_tags, not replace**: File-path-inferred tags represent the current session's structural context (what files are open NOW). They have higher authority than the persisted tags from a prior session. The merge puts current file_tags first in the array, which means they are "fresher" in any tie-breaking that references insertion order.

3. **Cap at 20 tags**: Per Q045's design, to prevent context_tags bloat in long sessions. With 5-12 persisted tags and 2-3 file_tags, the merged set is typically 7-15, well within the cap.

4. **Observability field**: `context_tags_source = 'restored'` on the session context allows the dashboard to display whether a session started warm (restored) or cold. Aggregate over sessions: fraction of sessions that started warm vs. cold by TTL expiry. This directly measures the 3-day TTL's effectiveness.

5. **TTL enforcement is storage-layer**: Because the Redis key has `EX 259200`, application code never needs to check the age of the persisted tags. If the key is present, it is ≤3 days old by construction. No `saved_at` comparison needed in the restore path.

## Ideas Extracted

### Idea: redis-ttl-session-tag-persistence
- **Novelty**: 0.75 — No described AI memory or coding assistant system persists and restores session routing context (topic tags) across session boundaries using a TTL-bounded cache key. Browser session restoration (localStorage, IndexedDB) restores UI state; IDE workspace restore saves open files and cursor positions; but restoration of *retrieval routing filters* as a warm-start mechanism for memory retrieval is not a described production pattern. The TTL as the staleness enforcement mechanism (rather than application-layer age checks) is an underused pattern that eliminates a class of application bugs.
- **Evidence**: 0.72 — Web browser session persistence (session cookies, localStorage) is a production-validated pattern for restoring context across process restarts. Redis TTL-based session storage is standard in web application session management (Django, Rails, Express session middleware all support Redis as the session store with TTL-enforced expiry). The specific application to retrieval context tags is engineering application of established patterns.
- **Feasibility**: 0.92 — Implementation requires: (1) two Redis SET/GET calls (one per hook), (2) a `machine_id` environment variable on each host, (3) an optional JSONB column addition on the sessions table for audit. Redis is already deployed in the stack. The hook modifications are ~30-40 lines each. No new infrastructure. Estimated implementation: 2-4 days including testing on both machines.
- **Description**: At session end (Stop hook), serialize the accumulated `context_tags` array to a Redis key `recall:last_session_tags:{machine_id}` with a 3-day TTL. At session start (UserPromptSubmit hook, first call only), GET this key and merge the restored tags with current file-path-inferred tags (cap at 20 distinct tags). If the key is absent (TTL expired, no prior session, or Redis unavailable), fall through to cold-start behavior. The TTL enforces staleness without application-layer age checks. The merged tag set seeds the MBC Level 1 query context, giving early-session queries the benefit of C≈15 rather than C≈2-3.

### Idea: stale-tag-dilution-rate-as-session-health-metric
- **Novelty**: 0.70 — Tracking the ratio of restored (prior-session) tags to in-session (current) tags as a real-time metric — and using the dilution rate (how fast in-session tags displace stale tags in the context_tags set) as a proxy for how quickly the session "found its focus" — is not a described monitoring pattern in AI memory or retrieval systems.
- **Evidence**: 0.55 — Browser performance monitoring tracks Time-to-Interactive as a proxy for page "readiness." Network routing protocols track convergence time (how quickly routing tables stabilize after a topology change). The concept of measuring how quickly a system transitions from an initial warm-start estimate to a stable operational state is established in control systems theory (settling time). The application to memory routing tag dilution is engineering analogy.
- **Feasibility**: 0.88 — Implementation: log `context_tags_source` per retrieve event (already proposed as an observability field). At each retrieve event, count `len(restored_tags ∩ context_tags) / len(context_tags)` — the "stale fraction." When this drops below 30%, the session has self-corrected from any stale-tag contamination. This metric is computable from the session state without additional storage. Dashboard display: a per-session "warm-up curve" showing stale fraction vs. prompt count. Aggregate across sessions: the typical warm-up length (prompts until stale fraction < 30%) measures whether the 3-day TTL is correctly tuned.
- **Description**: After each LLPC injection or prompt-domain extraction, compute `stale_fraction = len(persisted_tags ∩ current_context_tags) / len(current_context_tags)`. Log this per retrieve event. Define session "self-correction" as the first retrieve event where `stale_fraction < 0.30`. Aggregate this as "warm-up depth" — the number of prompts until the restored session context is diluted to below 30% stale. If mean warm-up depth > 8 prompts, the TTL may be too long (stale tags are too dominant for too long) or the persisted tag count is too high. If mean warm-up depth < 2 prompts, the TTL could be extended (tags remain relevant longer than 3 days). This turns the TTL threshold into a data-driven calibration target rather than a fixed constant.

### Idea: per-project-tag-partition-for-cross-machine-context
- **Novelty**: 0.78 — Tim operates two machines (`casaclaude`, `proxyclaude`). Context tags persisted on `casaclaude` are not automatically available on `proxyclaude`. A per-project tag partition — where context tags are keyed by `(project_name, machine_id)` rather than just `machine_id` — allows the `proxyclaude` session to restore context from the last `casaclaude` session on the same project, enabling cross-machine warm-start. This is architecturally equivalent to browser sync (Chrome's session sync across devices) but applied to retrieval routing context rather than UI state.
- **Evidence**: 0.58 — Browser sync (Chrome, Firefox) persists session state (open tabs, bookmarks, history) across devices by keying state to the user account rather than the device. Cloud IDE session persistence (Gitpod, GitHub Codespaces) restores workspace context across browser sessions and machines. These are production-validated patterns for cross-device context transfer. The specific application to retrieval routing tag sync is engineering application.
- **Feasibility**: 0.72 — Requires: (1) the Recall API to accept a `project_name` parameter (already inferable from session_summary's domain field), (2) Redis key structure `recall:last_session_tags:{project_name}:{machine_id}` with a shared Redis instance (both `casaclaude` and `proxyclaude` already connect to the same Redis on 192.168.50.19), (3) a fallback read: first try `recall:last_session_tags:{project_name}:{machine_id}`, then `recall:last_session_tags:{machine_id}`. If the project_name key is present and recent, it provides cross-machine warm-start. This requires the hooks to know which project they are working in — inferable from the git repo's remote URL or directory name. Implementation: 1-2 additional weeks on top of the base persistence implementation.
- **Description**: Extend the Redis key scheme to `recall:last_session_tags:{project_name}:{machine_id}`, where `project_name` is derived from the git repo name (e.g., `system-recall`, `family-hub`). At session start, restore from the project-specific key first. If absent, fall back to the machine-level key. At session end, write to both keys (project-level for cross-machine sharing, machine-level for within-machine persistence). Since both machines connect to the same Redis instance on the homelab, tags written by `casaclaude` during a Recall project session are immediately visible to `proxyclaude` when it starts a Recall session. Cross-machine warm-start delivers the same 39pp Level 1 improvement even when Tim switches machines mid-sprint.

## Key Finding

Cross-session context_tags persistence is best implemented as a Redis TTL key (`recall:last_session_tags:{machine_id}`, TTL=3 days) written by the Stop hook and read once per session start by the UserPromptSubmit hook, with an optional JSONB audit column on the sessions table for durability. The 3-day TTL is the correct operating threshold: it covers same-project continuation sessions (delivering the full 39pp Level 1 improvement) while automatically discarding stale tags from project pivot gaps longer than 72 hours. The precision cost of stale cross-project tags is real but strictly bounded — concentrated in the first 3-5 prompts of a session, self-correcting as LLPC injections accumulate in-session tags, and estimated at a worst-case average precision loss of ~5pp over those early prompts versus an expected gain of +21pp across the full session. The implementation requires approximately 60-80 lines of hook code across two files, no new infrastructure, and produces an immediately measurable improvement in early-session MBC routing quality.
