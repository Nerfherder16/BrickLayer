# Finding: Q009 — Absence Mapping: Implicit Retrieval Feedback

**Question**: Has any AI memory system implemented retrieval feedback — where which retrieved memory was actually used in the response is fed back to update importance/retrieval weighting?
**Question Type**: ABSENCE
**Verdict**: PROMISING
**Severity**: High
**Source field**: Recommender systems (implicit feedback / collaborative filtering), information retrieval (relevance feedback), online learning

---

## Mechanism Under Investigation

An AI memory system that: (1) retrieves a set of memory candidates and injects them into LLM context, (2) detects — WITHOUT explicit user annotation — which of those memories actually appeared in or influenced the generated response, and (3) uses that detection signal to update each memory's importance score or retrieval weight, so that memories which are consistently used surface higher in future retrievals and memories that are consistently ignored surface lower.

The key discriminant is IMPLICIT detection: the signal comes from observing the output (response text, citations, model behavior), not from asking a human "was this useful?"

## Absence Classification

**Classification**: PARTIAL ABSENCE

**Explanation**: The mechanism has been partially approached in the RAG literature but from the wrong angle and never in production AI memory systems. The specific gap — detecting which injected memories were cited in response text, then updating importance scores — does not exist in any production or research system found.

What DOES exist (and why it is distinct):

1. **mem0's feedback mechanism**: Pure explicit signal. A developer/user calls `POST /feedback` with a memory ID and a POSITIVE/NEGATIVE rating. No connection to whether the memory appeared in any response. No automatic detection.

2. **Qdrant's Relevance Feedback Query (Feb 2026)**: A Rocchio-style mechanism that accepts positive and negative example vectors and adjusts the query vector toward positive examples and away from negative ones. Requires explicit human-labeled examples. Updates the QUERY direction, not stored document weights. Does not persist any signal across future queries.

3. **GAM-RAG (Mar 2026, arXiv:2603.01783)**: The closest published work. Accumulates retrieval experience: when a retrieval episode is "successful" (the model generates a correct answer), the contributing sentences get memory weight boosts via a Kalman-inspired update rule. The feedback signal is answer correctness — which requires ground truth labels to measure. This is NOT the implicit citation-detection mechanism; it is correctness-from-ground-truth feedback. No production deployment found; code is anonymized for review.

4. **DMA (Nov 2025, arXiv:2511.04880)**: Deployed at China Mobile. Updates a learned ranker using human feedback at document, list, and response level. The response-level signal is human preference over response pairs (thumbs up/down). Not implicit citation detection.

5. **FeedbackRAG (Nov 2025)**: The "implicit" signals are user sentiment derived from follow-up chat messages — detecting frustration vs. satisfaction in natural language. Not citation detection.

**Why the specific implicit signal gap exists**:

The mechanism requires solving two subproblems that nobody has combined:
- **Citation detection**: Which retrieved memory UUIDs had their content appear (verbatim or paraphrased) in the generated response? This is a non-trivial string/semantic matching problem on potentially paraphrased text.
- **Importance score update**: A persistent weight update for each memory object that survives across sessions and influences future ANN retrieval.

The barrier is architectural: production AI memory systems (mem0, Zep, Letta) are designed as storage + retrieval layers, not as learning systems. They have no training loop or feedback sink. The feedback infrastructure simply does not exist to receive an implicit signal and route it to a weight update. The RAG systems that DO implement feedback (DMA, FeedbackRAG) operate on document collections that are NOT user-specific episodic memories — they update ranker models, not individual document weights.

The recommender system analogy (implicit feedback from user behavior) is well-understood but has never been ported to AI memory. The closest behavioral analog would be: a streaming service that buries recommendations you skip over and surfaces recommendations you watch — but applied to memories instead of movies. No AI memory system implements this.

## Evidence

### System: mem0
Documented feedback API accepts `positive`, `negative`, `very_negative` ratings per memory ID. The docs state feedback "improves accuracy of memories and search results" and that "Mem0 continuously learns from this feedback" but the mechanism is opaque and entirely explicit. No evidence of implicit usage detection. The open-source codebase has no retrieval-feedback pathway.

### System: Zep / Graphiti
Architecture is graph-based (entities and relationships extracted from conversation). Memory updates happen when new conversation turns are ingested — new edges and nodes are added, existing ones may be superseded. There is no mechanism for observing which graph nodes were cited in a response and boosting their weight. The graph is updated forward (new input), not backward (from output observation).

### System: MemGPT / Letta
Memory tools (`core_memory_replace`, `archival_memory_insert`, `archival_memory_search`) are invoked by the LLM explicitly. The LLM decides what to write back. This is explicit LLM self-editing, not implicit retrieval feedback. Letta's Context Repositories (Feb 2026) add git-backed versioning but no retrieval feedback loop. No mechanism exists to observe which retrieved archival memories influenced a response.

### Paper: GAM-RAG (arXiv:2603.01783, submitted Mar 2026)
The most structurally similar published work. Builds a hierarchical index of sentence-level memories. After inference, sentences that contributed to a correct answer receive memory weight updates via a Kalman gain rule: weight_new = weight_old + K * (correctness_signal - weight_old), where K is an uncertainty-adjusted gain factor. Key distinctions from the target mechanism:
- Feedback signal is answer correctness (requires ground truth), not citation detection from response text.
- Designed for QA over document corpora, not episodic user memory.
- No production deployment; code anonymized for peer review.
- Does not update importances for user-specific memory objects.

### Paper: DMA — Dynamic Memory Alignment (arXiv:2511.04880, deployed Nov 2025)
Industrial RAG system deployed at China Mobile. Multi-granularity feedback loop: document-level (was this document relevant?), list-level (was the ranking order good?), response-level (was the response preferred?). All three levels require human feedback collected during interactions. The response-level signal is an RLHF-style preference between response pairs. This is the most sophisticated deployed feedback system found — but it is explicit human preference feedback, not implicit citation detection, and it updates a learned reranker model, not individual document importance scores.

### Vector DB: Qdrant Relevance Feedback (released Feb 2026)
Qdrant introduced a native Relevance Feedback Query interface in Feb 2026. Users specify positive and negative example vectors; Qdrant applies a weighted combination (Rocchio formula) to adjust the query vector. This modifies the search direction at query time. It does NOT update stored vector weights or persist any signal. Each query is independent. Requires explicit positive/negative example designation. Related `qdrant/relevance-feedback` GitHub repo (5 stars, Feb 2026) provides tooling to tune the formula parameters — still fully explicit.

### Recommender systems (mature prior art)
Collaborative filtering with implicit feedback (Hu et al., 2008; BPR, 2009; neural CF, 2017) is standard in production recommender systems. The pattern: user action (click, watch, purchase) = implicit positive signal; absence of action = implicit negative signal. Weights are updated by ALS or gradient descent. The critical difference from the target mechanism: in recommender systems, the "item" and the "interaction signal" are structurally identical across all users, enabling matrix factorization. In AI memory, each memory is unique to one user and the signal source (response text citation) is heterogeneous and requires semantic matching to detect. The port is conceptually clean but technically non-trivial.

## Ideas Extracted

### Idea: implicit-retrieval-feedback

- **Novelty**: 0.82 — The specific combination (citation detection from response text + importance score update for episodic user memories) is absent from all production systems and all research papers found. GAM-RAG is the closest prior work but uses a different signal (answer correctness, not citation detection) and targets a different domain (document QA, not personal episodic memory).
- **Evidence**: 0.70 — The sub-mechanisms are individually validated: citation detection is implemented in CiteFix (ACL 2025 industry track) and FineRef (arXiv:2602.18437); Kalman-inspired importance updates are validated in GAM-RAG; implicit feedback loops for ranking are mature in recommender systems. The synthesis for episodic AI memory is novel but not speculative.
- **Feasibility**: 0.78 — Recall's hook architecture makes this uniquely buildable without infrastructure changes. The three components map cleanly: (1) `recall-retrieve.js` already tracks which memory UUIDs are injected, (2) a new `recall-feedback.js` hook on the Stop event can compare injected UUIDs against response text, (3) a `POST /memories/{id}/feedback` endpoint updates the importance score.
- **Description**: After the LLM response is generated, `recall-feedback.js` runs a lightweight citation check: for each memory UUID that was injected in this session, check whether any content from that memory's text appears verbatim or near-verbatim in the response (fuzzy string match with ~0.85 threshold). Memories whose content appears in the response get `importance += 0.1 * (1 - current_importance)` (bounded update toward 1.0). Memories retrieved but not cited get a mild decay: `importance *= 0.98`. Over many sessions, consistently-cited memories float to higher importance scores and appear more reliably in future retrievals; consistently-ignored memories decay. The update formula should be conservative (small delta) to avoid feedback instability. As an alternative to fuzzy string match, the response and each candidate memory chunk can be co-embedded and cosine similarity threshold used as the citation signal — this handles paraphrasing but adds an Ollama round trip.

### Idea: retrieval-access-count-decay

- **Novelty**: 0.45 — Retrieval count tracking alone (not citation detection) is a simpler variant. This likely exists in some systems but was not found in any of the major AI memory frameworks. Not the full implicit feedback loop — just increments a counter on retrieval and uses count as a tie-breaker in ranking.
- **Evidence**: 0.60 — The concept is trivially implementable and well-understood; the gap is simply that no AI memory system exposes it.
- **Feasibility**: 0.92 — Trivial: add `access_count` field to Recall memory metadata, increment on retrieval, use as a secondary ranking signal.
- **Description**: A prerequisite / simpler version of implicit-retrieval-feedback. Does not require citation detection but provides a weaker signal (retrieval ≠ use). Worth implementing as a baseline before the full citation-detection loop.

## Suggested Follow-ups

- **Q015 (TABOO)**: Design an automatic importance scoring system from first principles — now informed by the concrete absence in this finding and the GAM-RAG Kalman-gain update rule, which is a validated reference design for the update mechanism.
- **Implement the simpler `retrieval-access-count-decay` first**: Gives a useful signal immediately at near-zero cost. Serves as an A/B baseline against the full citation-detection loop.
- **Evaluate fuzzy-match vs. co-embedding for citation detection**: The string-match approach is cheap (no Ollama call) but misses paraphrasing. The embedding approach handles paraphrasing but adds 50–200ms latency on the Stop hook. A hybrid (string match first, embedding fallback for near-misses) may be optimal.
- **Check GAM-RAG code release**: The code is anonymized at https://anonymous.4open.science/r/GAM_RAG-2EF6 — if the review completes and code is released under a permissive license, the Kalman gain update rule is directly adaptable for Recall's importance update logic.
- **Assess feedback stability**: A feedback loop that updates importance scores creates a potential runaway reinforcement problem (one memory keeps getting cited, becomes dominant, crowds out other memories). The GAM-RAG Kalman uncertainty term is designed to prevent this. Any implementation for Recall should include a similar stability bound.
