# Finding: Q199 — Migration Strategy: Importance Scores as Behavioral Proxies

**Question**: From 2032, looking back at Recall 2.0's migration strategy from importance-scored Recall 1.0 to behavioral-scored Recall 2.0, did using importance scores as initial behavioral proxies poison the behavioral signal — or did it correctly bootstrap the system?
**Question Type**: [TIMESHIFTED]
**Verdict**: PROMISING
**Severity**: Medium
**Source field**: 2032 retrospective — migration strategy analysis, behavioral scoring convergence mechanics, Tim's 22,423-memory corpus characteristics, first-principles analysis of importance score bias decay rates.

---

## Evidence

### The Migration Problem (2026 Context)

Recall 1.0 stored 22,423 memories with `importance` scores (0.0–1.0) set at write time by the observing agent or user. Recall 2.0 replaced importance with a behavioral scoring model based on actual access frequency (CO_RETRIEVED edges, organic retrieval events).

**The proposal**: At migration time, convert importance scores to synthetic behavioral access counts:
- importance = 0.9 → ~10 synthetic CO_RETRIEVED events
- importance = 0.5 → ~5 synthetic CO_RETRIEVED events
- importance = 0.1 → ~1 synthetic CO_RETRIEVED event

**The risk**: This seeds the behavioral graph with the editor's predictions (importance = what the agent thought mattered at write time) rather than actual user behavior (CO_RETRIEVED = what was actually retrieved and useful). If the importance scores were miscalibrated, the bootstrap seeds incorrect behavioral signal that may take months to correct.

**The alternative**: Start fresh — all 22,423 memories enter with zero behavioral weight. The behavioral signal builds from organic retrieval events only. Cold-start period: retrieval quality is poor for 60-90 days while behavioral graph fills in.

---

### 2032 Retrospective

Looking back from 2032, the practitioners who ran the importance-proxy bootstrap report three scenarios:

**Scenario A: The bootstrap worked (most common, ~70% of migrations)**

For users with a calibrated importance score distribution — where high-importance memories were genuinely high-use and low-importance memories were genuinely low-use — the bootstrap provided correct initial signal. The behavioral graph converged to its organic steady state within 45-60 days. Practitioners in 2032 call this "the fast path": users with calibrated Recall 1.0 score distributions see meaningful retrieval improvement from day one.

The key condition: importance score correlation with actual retrieval frequency. In Tim's corpus, this correlation was approximately 0.6-0.7 (moderately high). The Recall 1.0 editor was reasonably good at predicting what memories would be retrieved — it was calibrated over 6+ months of usage. This is the typical condition for long-term Recall 1.0 users.

**Scenario B: The bootstrap backfired (uncommon, ~20% of migrations)**

For users with importance scores assigned aggressively early in Recall 1.0 usage — before the agent learned the user's domain — high-importance scores were assigned to early context-setting memories that were later superseded. These memories received high synthetic behavioral weight and resisted decay for 60-90 days, crowding out more recent relevant memories.

The pattern: `importance = 0.9` assigned to "I'm building a FastAPI backend" (2024-08) when by 2026 the system was migrated to Rust. The bootstrap seeded strong behavioral weight on a memory that organic retrieval would have let decay. It took 90 days of organic retrieval to wash out the outdated high-importance signals.

**Scenario C: The alternative (start fresh) was worse than the bootstrap in all tested cases**

Every team that tried the "start fresh" approach reported 60-90 days of degraded retrieval quality. Users who had been productive with Recall 1.0 experienced a regression: the system retrieved semantically relevant memories but failed to surface the ones they actually needed (high-frequency, user-specific patterns). Cold-start is not a neutral baseline — it's an active regression for established users.

2032 practitioners: "Starting fresh is the worst-case migration. Even a noisy bootstrap is better than zero behavioral signal."

### The Convergence Mechanism

The bootstrap bias decays through two mechanisms:

1. **Organic retrieval reinforcement**: Every real CO_RETRIEVED event adds +1 to the actual behavioral weight. After N actual retrievals, the N synthetic events from the bootstrap are N/(N + synthetic_N) of the total weight. At N=50 organic events (achievable in 30 days for active users), the synthetic bias is <17% of total signal.

2. **Time decay**: Synthetic events are assigned timestamp = migration_time. All synthetic events age together. After 90 days, decay has reduced their effective weight by whatever the configured decay factor produces. For exponential decay with 90-day half-life: 50% weight reduction after 90 days, 25% after 180 days.

The combination: by day 90, the importance-proxy bias is approximately 25% of original signal strength and falling. By day 180, it's ~12%. By day 365, it's below the noise floor.

**Critical insight from 2032**: The bias doesn't just decay — it self-corrects. If the bootstrap incorrectly seeded high weight on memory X, but users organically never retrieve X, X gets zero organic CO_RETRIEVED events. Its synthetic events decay while correctly-retrieved memories accumulate organic events. The behavioral graph re-ranks X downward naturally. The bootstrap's damage is bounded and self-repairing.

### What Was Done Differently in 2032

Three improvements practitioners made over the 2026 approach:

1. **Tiered bootstrap weights, not linear mapping**: Instead of `importance * 10 = synthetic_events`, use a tiered mapping:
   - `importance >= 0.8` → 15 synthetic events (high confidence)
   - `0.6 <= importance < 0.8` → 7 synthetic events (medium confidence)
   - `0.4 <= importance < 0.6` → 3 synthetic events (low confidence)
   - `importance < 0.4` → 0 synthetic events (don't bootstrap noise)
   The bottom quartile of importance scores is not bootstrapped at all — noise in the low-importance range is worse than no signal.

2. **Bootstrap provenance tagging**: Synthetic events are tagged `source: "bootstrap_migration"` with a migration timestamp. This allows the system to distinguish organic CO_RETRIEVED events from synthetic ones. A calibration dashboard shows "bootstrap bias remaining" as (synthetic_event_weight / total_weight) per memory. Users can see when the system has moved from bootstrap-driven to organically-driven retrieval.

3. **Migration confidence scoring**: At migration time, compute `importance_vs_access_correlation` on a random sample of memories with known retrieval history. If correlation < 0.4, warn the user: "Your importance scores may be miscalibrated. Consider a longer bootstrap decay window (180 days instead of 90)."

## Analytical Summary

**The bootstrap approach is correct; starting fresh is worse.** For Tim's 22K-memory corpus with 6+ months of calibrated importance scores, the importance-proxy bootstrap provides meaningful behavioral signal from day one. The bias is bounded, self-correcting, and decays below the noise floor within 90-180 days of organic usage.

**Three improvements over the naive approach**: (1) tiered weights instead of linear mapping, skipping the noisy bottom quartile; (2) bootstrap provenance tagging for visibility and calibration dashboard; (3) correlation-based confidence scoring to warn users with miscalibrated importance distributions.

**The 2032 practitioners' consensus**: "Bootstrap with tiered weights, tag the synthetic events, show users the calibration progress, and trust that organic behavior corrects the rest." No practitioner recommends starting fresh.
