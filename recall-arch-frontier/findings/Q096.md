# Finding: Q096 — CO_RETRIEVED Type-Partitioning Migration: Cypher, Runtime, Rollback, and Background Reclassification Design

**Question**: The Q091 finding established that CO_RETRIEVED_FRESH / CO_RETRIEVED_STALE type partitioning is the ONLY architectural path to a true pre-traversal edge filter in Neo4j Community Edition. Design and specify the one-time migration Cypher that reclassifies all ~800 existing CO_RETRIEVED edges to either CO_RETRIEVED_FRESH (last_reinforced_epoch >= current_epoch - tau_edge) or CO_RETRIEVED_STALE (below threshold) — including the exact Cypher statement, the estimated runtime on Tim's CasaOS Neo4j deployment, the rollback strategy if the migration fails midway, and the ongoing background reclassification job (~45 edges/session per Q091) that maintains type accuracy after the initial migration.
**Question Type**: IMPLEMENTATION
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Neo4j Community Edition schema migration, relationship type transitions, Cypher batch processing, temporal partitioning, graph database operational patterns

---

## Constraints and Starting Conditions

Before specifying any Cypher, the key constraints must be enumerated precisely:

1. **Neo4j Community Edition** — no APOC library guaranteed. All Cypher must work without `apoc.refactor.*`, `apoc.periodic.iterate`, or any APOC procedure. Standard Cypher only.
2. **Relationship type is immutable in Neo4j** — there is no `SET TYPE` or `CHANGE TYPE` operation. The only way to reclassify a relationship is CREATE new + DELETE old, preserving all properties manually.
3. **~800 CO_RETRIEVED edges** — at Tim's current graph density. This is a bounded, modest migration; it is not a million-edge operation.
4. **Current epoch from Redis** — the classification threshold is `last_reinforced_epoch >= current_epoch - tau_edge`. The migration script must receive the current_epoch at migration time as a parameter; it cannot query Redis from Cypher.
5. **tau_edge = 20** (from Q091; note Q091 used tau=20 for node staleness and tau_edge=30 for edge staleness in the cost model, but the project brief's core tau=20. The migration should parameterize this. We use `$tau_edge` throughout and specify 20 as the concrete default).
6. **No in-flight writes during migration** — the migration is a one-time operation. The Recall API should be stopped or the Neo4j instance made read-only for write operations during migration. This is achievable on CasaOS by stopping the Recall Docker container temporarily.
7. **CO_RETRIEVED edges have properties**: `last_reinforced_epoch` (integer), `co_retrieval_count` (integer), `strength` (float). Possibly also an `id` property if used as a unique key. All properties must be transferred to the new relationship.

## Section 1: The One-Time Migration Cypher

### Why APOC Is Not Available

On CasaOS (Docker, Neo4j Community Edition), APOC is not bundled. It must be manually added as a plugin JAR and configured in `neo4j.conf` with `dbms.security.procedures.allowlist=apoc.*`. Unless Tim has explicitly done this, APOC is absent. The migration must not depend on it.

Even if APOC were available, `apoc.refactor.setRelationshipType` does not exist in APOC — APOC can clone properties and merge nodes but cannot change relationship type in-place for the same architectural reason (Neo4j's storage layer: relationship type ID is baked into the relationship record and cannot be mutated after creation). `apoc.refactor.from` does support relationship type change as CREATE+DELETE internally, but given uncertainty about APOC availability, the manual Cypher approach is canonical.

### Migration Cypher — Manual CREATE + DELETE (No APOC)

The migration is split into two passes:
- **Pass 1**: Classify and create `CO_RETRIEVED_FRESH` relationships for fresh edges
- **Pass 2**: Classify and create `CO_RETRIEVED_STALE` relationships for stale edges
- **Pass 3**: DELETE all original `CO_RETRIEVED` relationships (after both FRESH and STALE have been created)

**Executing deletions only after all new relationships are created** is the safety invariant: if the migration is interrupted after Pass 1 or 2 but before Pass 3, the original `CO_RETRIEVED` relationships are still intact and the system can roll back cleanly by deleting the partially created FRESH/STALE relationships.

#### Pass 1 — Create CO_RETRIEVED_FRESH for fresh edges

The current_epoch value must be supplied as a parameter from Python/the migration script (it is read from Redis before the Cypher runs):

```cypher
// Pass 1: Create CO_RETRIEVED_FRESH for all edges where last_reinforced_epoch >= current_epoch - tau_edge
// Parameters: $current_epoch (integer from Redis), $tau_edge (integer, default 20)
// This pass is SAFE to run multiple times (MERGE prevents duplicate creation)

MATCH (a:Memory)-[r:CO_RETRIEVED]->(b:Memory)
WHERE r.last_reinforced_epoch >= ($current_epoch - $tau_edge)
MERGE (a)-[r_fresh:CO_RETRIEVED_FRESH {
    _migrated_from_id: id(r)
}]->(b)
ON CREATE SET
    r_fresh.last_reinforced_epoch = r.last_reinforced_epoch,
    r_fresh.co_retrieval_count    = r.co_retrieval_count,
    r_fresh.strength              = r.strength,
    r_fresh.migrated_at_epoch     = $current_epoch
RETURN count(r_fresh) AS fresh_created
```

**Note on `_migrated_from_id`**: Neo4j's internal relationship ID (`id(r)`) is used as a fingerprint to make this pass idempotent — `MERGE` on `_migrated_from_id` prevents creating duplicate FRESH edges if Pass 1 is re-run after partial failure. This field should be removed from production edges after migration is confirmed (see cleanup step).

**Limitation**: `MERGE` on a relationship requires specifying BOTH endpoints and the property map used as the identity key. The `_migrated_from_id` is stored as a relationship property in the MERGE clause, which makes the MERGE unique per source relationship. This is slightly expensive (property lookup on every MERGE) but correct for ~800 edges.

**Alternative idempotent approach** (if `_migrated_from_id` feels awkward):

```cypher
// Simpler: use CREATE with a pre-check. Run in a transaction; Neo4j rolls back on failure.
MATCH (a:Memory)-[r:CO_RETRIEVED]->(b:Memory)
WHERE r.last_reinforced_epoch >= ($current_epoch - $tau_edge)
  AND NOT (a)-[:CO_RETRIEVED_FRESH]->(b)
CREATE (a)-[r_fresh:CO_RETRIEVED_FRESH]->(b)
SET
    r_fresh.last_reinforced_epoch = r.last_reinforced_epoch,
    r_fresh.co_retrieval_count    = r.co_retrieval_count,
    r_fresh.strength              = r.strength,
    r_fresh.migrated_at_epoch     = $current_epoch
RETURN count(r_fresh) AS fresh_created
```

The `NOT (a)-[:CO_RETRIEVED_FRESH]->(b)` guard makes this idempotent for the simple case where at most one CO_RETRIEVED edge exists between any two Memory nodes (the typical case for a co-retrieval graph). If multiple parallel CO_RETRIEVED edges between the same pair can exist (unlikely but theoretically possible), use the `_migrated_from_id` approach above.

#### Pass 2 — Create CO_RETRIEVED_STALE for stale edges

```cypher
// Pass 2: Create CO_RETRIEVED_STALE for all edges where last_reinforced_epoch < current_epoch - tau_edge
// Parameters: $current_epoch (integer from Redis), $tau_edge (integer, default 20)

MATCH (a:Memory)-[r:CO_RETRIEVED]->(b:Memory)
WHERE r.last_reinforced_epoch < ($current_epoch - $tau_edge)
  AND NOT (a)-[:CO_RETRIEVED_STALE]->(b)
CREATE (a)-[r_stale:CO_RETRIEVED_STALE]->(b)
SET
    r_stale.last_reinforced_epoch = r.last_reinforced_epoch,
    r_stale.co_retrieval_count    = r.co_retrieval_count,
    r_stale.strength              = r.strength,
    r_stale.migrated_at_epoch     = $current_epoch
RETURN count(r_stale) AS stale_created
```

#### Verification query — run before Pass 3

Before deleting the original CO_RETRIEVED edges, confirm that all 800 edges have a corresponding FRESH or STALE counterpart:

```cypher
// Verification: count CO_RETRIEVED that have NO corresponding FRESH or STALE
// This should return 0 before Pass 3 is executed.
MATCH (a:Memory)-[r:CO_RETRIEVED]->(b:Memory)
WHERE NOT (a)-[:CO_RETRIEVED_FRESH]->(b)
  AND NOT (a)-[:CO_RETRIEVED_STALE]->(b)
RETURN count(r) AS unmigrated_count
```

If `unmigrated_count > 0`, do not proceed to Pass 3. Investigate the unmigrated edges (there should be none given that Passes 1 and 2 are exhaustive, but floating-point epoch comparisons or null `last_reinforced_epoch` values can cause gaps — see the null guard below).

**Null guard for `last_reinforced_epoch`**: Some early CO_RETRIEVED edges may have been created before the epoch property was added (if the epoch property was introduced after the initial schema). These edges would fail both the fresh and stale predicates:

```cypher
// Pass 2b: Handle edges with NULL last_reinforced_epoch — classify as STALE
MATCH (a:Memory)-[r:CO_RETRIEVED]->(b:Memory)
WHERE r.last_reinforced_epoch IS NULL
  AND NOT (a)-[:CO_RETRIEVED_STALE]->(b)
CREATE (a)-[r_stale:CO_RETRIEVED_STALE]->(b)
SET
    r_stale.last_reinforced_epoch = 0,
    r_stale.co_retrieval_count    = coalesce(r.co_retrieval_count, 0),
    r_stale.strength              = coalesce(r.strength, 0.0),
    r_stale.migrated_at_epoch     = $current_epoch,
    r_stale.migration_note        = "null_epoch_defaulted_to_stale"
RETURN count(r_stale) AS null_epoch_stale_created
```

#### Pass 3 — DELETE original CO_RETRIEVED edges

Only run after verification query returns 0:

```cypher
// Pass 3: Delete all original CO_RETRIEVED relationships
// ONLY RUN AFTER verification query confirms 0 unmigrated edges.
// This is the POINT OF NO RETURN — once Pass 3 is committed, rollback requires
// deleting the new FRESH/STALE edges and restoring from graph backup.

MATCH ()-[r:CO_RETRIEVED]->()
DELETE r
RETURN count(*) AS deleted_count
```

**Important**: On Neo4j Community Edition, Cypher DELETE within a single transaction is ACID — if the statement fails mid-execution (e.g., OOM), the transaction rolls back and no CO_RETRIEVED relationships are deleted. The entire `MATCH ()-[r:CO_RETRIEVED]->() DELETE r` statement is atomic for ~800 relationships; this is well within Neo4j's default transaction memory limits.

#### Post-migration cleanup — remove `_migrated_from_id` property (if used)

```cypher
// Cleanup: Remove the migration tracking property from all FRESH and STALE edges
MATCH ()-[r:CO_RETRIEVED_FRESH]->()
WHERE r._migrated_from_id IS NOT NULL
REMOVE r._migrated_from_id
RETURN count(r) AS cleaned

UNION ALL

MATCH ()-[r:CO_RETRIEVED_STALE]->()
WHERE r._migrated_from_id IS NOT NULL
REMOVE r._migrated_from_id
RETURN count(r) AS cleaned
```

## Section 2: Python Migration Script (Orchestrates All Passes)

The migration is orchestrated from Python to:
1. Read current_epoch from Redis before migration
2. Run passes sequentially with verification between passes
3. Log each pass result
4. Abort if verification fails

```python
#!/usr/bin/env python3
"""
migrate_co_retrieved_types.py

One-time migration: reclassify CO_RETRIEVED edges to CO_RETRIEVED_FRESH/CO_RETRIEVED_STALE.

Usage:
    python migrate_co_retrieved_types.py [--dry-run] [--tau-edge 20]

Pre-requisites:
    - Recall API must be stopped (no concurrent writes)
    - Redis must be reachable at 192.168.50.19:6379
    - Neo4j must be reachable at bolt://192.168.50.19:7687

Safety:
    - Passes 1 and 2 are idempotent (safe to re-run after failure)
    - Pass 3 (DELETE) is only executed after verified 0 unmigrated edges
    - --dry-run skips Pass 3 and reports counts without deleting
"""

import sys
import argparse
import redis
from neo4j import GraphDatabase

NEO4J_URI = "bolt://192.168.50.19:7687"
NEO4J_USER = "neo4j"
NEO4J_PASS = "your_password_here"  # use env var in production

REDIS_HOST = "192.168.50.19"
REDIS_PORT = 6379
REDIS_EPOCH_KEY = "recall:global_epoch"

DEFAULT_TAU_EDGE = 20


def get_current_epoch() -> int:
    r = redis.Redis(host=REDIS_HOST, port=REDIS_PORT, decode_responses=True)
    val = r.get(REDIS_EPOCH_KEY)
    if val is None:
        raise RuntimeError(f"Redis key '{REDIS_EPOCH_KEY}' not found. Is Redis running?")
    epoch = int(val)
    print(f"[MIGRATION] Current epoch from Redis: {epoch}")
    return epoch


def run_migration(dry_run: bool = False, tau_edge: int = DEFAULT_TAU_EDGE):
    current_epoch = get_current_epoch()
    params = {"current_epoch": current_epoch, "tau_edge": tau_edge}

    driver = GraphDatabase.driver(NEO4J_URI, auth=(NEO4J_USER, NEO4J_PASS))
    with driver.session() as session:

        # --- Count baseline ---
        result = session.run("MATCH ()-[r:CO_RETRIEVED]->() RETURN count(r) AS total")
        total = result.single()["total"]
        print(f"[MIGRATION] Baseline: {total} CO_RETRIEVED edges to migrate")

        if total == 0:
            print("[MIGRATION] No CO_RETRIEVED edges found. Migration not required.")
            return

        # --- Pass 1: Create CO_RETRIEVED_FRESH ---
        print(f"[MIGRATION] Pass 1: Creating CO_RETRIEVED_FRESH (epoch >= {current_epoch - tau_edge})...")
        result = session.run("""
            MATCH (a:Memory)-[r:CO_RETRIEVED]->(b:Memory)
            WHERE r.last_reinforced_epoch >= ($current_epoch - $tau_edge)
              AND NOT (a)-[:CO_RETRIEVED_FRESH]->(b)
            CREATE (a)-[r_fresh:CO_RETRIEVED_FRESH]->(b)
            SET r_fresh.last_reinforced_epoch = r.last_reinforced_epoch,
                r_fresh.co_retrieval_count    = r.co_retrieval_count,
                r_fresh.strength              = r.strength,
                r_fresh.migrated_at_epoch     = $current_epoch
            RETURN count(r_fresh) AS fresh_created
        """, **params)
        fresh_count = result.single()["fresh_created"]
        print(f"[MIGRATION] Pass 1 complete: {fresh_count} CO_RETRIEVED_FRESH created")

        # --- Pass 2: Create CO_RETRIEVED_STALE (explicit threshold) ---
        print(f"[MIGRATION] Pass 2: Creating CO_RETRIEVED_STALE (epoch < {current_epoch - tau_edge})...")
        result = session.run("""
            MATCH (a:Memory)-[r:CO_RETRIEVED]->(b:Memory)
            WHERE r.last_reinforced_epoch < ($current_epoch - $tau_edge)
              AND NOT (a)-[:CO_RETRIEVED_STALE]->(b)
            CREATE (a)-[r_stale:CO_RETRIEVED_STALE]->(b)
            SET r_stale.last_reinforced_epoch = r.last_reinforced_epoch,
                r_stale.co_retrieval_count    = r.co_retrieval_count,
                r_stale.strength              = r.strength,
                r_stale.migrated_at_epoch     = $current_epoch
            RETURN count(r_stale) AS stale_created
        """, **params)
        stale_count = result.single()["stale_created"]
        print(f"[MIGRATION] Pass 2 complete: {stale_count} CO_RETRIEVED_STALE created")

        # --- Pass 2b: Handle NULL last_reinforced_epoch edges ---
        result = session.run("""
            MATCH (a:Memory)-[r:CO_RETRIEVED]->(b:Memory)
            WHERE r.last_reinforced_epoch IS NULL
              AND NOT (a)-[:CO_RETRIEVED_STALE]->(b)
            CREATE (a)-[r_stale:CO_RETRIEVED_STALE]->(b)
            SET r_stale.last_reinforced_epoch = 0,
                r_stale.co_retrieval_count    = coalesce(r.co_retrieval_count, 0),
                r_stale.strength              = coalesce(r.strength, 0.0),
                r_stale.migrated_at_epoch     = $current_epoch,
                r_stale.migration_note        = "null_epoch_defaulted_to_stale"
            RETURN count(r_stale) AS null_stale_created
        """, **params)
        null_stale_count = result.single()["null_stale_created"]
        if null_stale_count > 0:
            print(f"[MIGRATION] Pass 2b: {null_stale_count} NULL-epoch edges defaulted to STALE")

        migrated_total = fresh_count + stale_count + null_stale_count
        print(f"[MIGRATION] Total migrated: {migrated_total} / {total}")

        # --- Verification ---
        result = session.run("""
            MATCH (a:Memory)-[r:CO_RETRIEVED]->(b:Memory)
            WHERE NOT (a)-[:CO_RETRIEVED_FRESH]->(b)
              AND NOT (a)-[:CO_RETRIEVED_STALE]->(b)
            RETURN count(r) AS unmigrated_count
        """)
        unmigrated = result.single()["unmigrated_count"]
        print(f"[MIGRATION] Verification: {unmigrated} unmigrated CO_RETRIEVED edges remaining")

        if unmigrated > 0:
            print(f"[MIGRATION] ERROR: {unmigrated} edges were not migrated. ABORTING before DELETE.")
            print("[MIGRATION] The original CO_RETRIEVED edges are still intact.")
            print("[MIGRATION] Investigate the unmigrated edges before re-running.")
            sys.exit(1)

        # --- Pass 3: DELETE original CO_RETRIEVED edges ---
        if dry_run:
            print(f"[MIGRATION] DRY RUN: Would delete {total} CO_RETRIEVED edges. Skipping.")
            print("[MIGRATION] Re-run without --dry-run to complete the migration.")
            return

        print(f"[MIGRATION] Pass 3: Deleting {total} original CO_RETRIEVED edges...")
        result = session.run("""
            MATCH ()-[r:CO_RETRIEVED]->()
            DELETE r
            RETURN count(*) AS deleted_count
        """)
        deleted = result.single()["deleted_count"]
        print(f"[MIGRATION] Pass 3 complete: {deleted} CO_RETRIEVED edges deleted")
        print(f"[MIGRATION] MIGRATION COMPLETE. Graph now has:")
        print(f"  CO_RETRIEVED_FRESH: {fresh_count}")
        print(f"  CO_RETRIEVED_STALE: {stale_count + null_stale_count}")
        print(f"  CO_RETRIEVED: 0 (all migrated)")

    driver.close()


if __name__ == "__main__":
    parser = argparse.ArgumentParser()
    parser.add_argument("--dry-run", action="store_true")
    parser.add_argument("--tau-edge", type=int, default=DEFAULT_TAU_EDGE)
    args = parser.parse_args()
    run_migration(dry_run=args.dry_run, tau_edge=args.tau_edge)
```

## Section 3: Migration Runtime Estimate on CasaOS

### CasaOS Neo4j deployment profile

Based on Tim's homelab:
- **Host**: 192.168.50.19 (CasaOS, Docker)
- **Neo4j container**: Community Edition, typical Docker resource limits (2–4 vCPUs, 2–4 GB RAM allocated to Neo4j via `NEO4J_server_memory_heap_initial__size` / `max`)
- **Storage**: NVMe SSD (or equivalent) on the CasaOS host — random read latency ~0.1–0.2ms per page
- **No enterprise features**: Community Edition means single-writer transactions only, no parallel query execution across cores for a single query

### Per-operation cost model

| Operation | Cost per relationship | Count | Total |
|-----------|----------------------|-------|-------|
| MATCH relationship record (CO_RETRIEVED chain read) | ~0.02ms | 800 | 16ms |
| Read edge properties (3 properties per relationship) | ~0.03ms | 800 | 24ms |
| CREATE new relationship record (WAL write + rel store update) | ~0.5–1.5ms | 800 | 400–1,200ms |
| SET properties on new relationship (3 props) | ~0.1ms | 800 | 80ms |
| DELETE original relationship (WAL write + rel store update) | ~0.5–1.0ms | 800 | 400–800ms |
| Transaction commit (WAL flush to disk) | ~5–20ms | 3 transactions (P1, P2, P3) | 15–60ms |
| Verification MATCH | ~0.02ms × 800 full scan | 1 | ~16ms |

**Estimated total runtime: 900ms – 2,200ms (approximately 1–2 seconds for 800 edges)**

This is comfortably within a one-time maintenance window. The bottleneck is WAL write latency for 1,600 relationship operations (800 CREATE + 800 DELETE) × disk write cost.

### Batch size consideration

For 800 edges, batching is not required. All 800 edges fit in a single Neo4j transaction without hitting the default transaction memory limit (`dbms.memory.transaction.global_max_size`, default 256MB). A CREATE of 800 relationships uses approximately 800 × 200 bytes = 160KB of transaction state — negligible.

If the graph were significantly larger (>10,000 edges), the migration should be batched in groups of 1,000 using `LIMIT`:

```cypher
// Batch version for large graphs (not needed for ~800 edges)
// Run in a loop until result.migrated = 0
MATCH (a:Memory)-[r:CO_RETRIEVED]->(b:Memory)
WHERE r.last_reinforced_epoch >= ($current_epoch - $tau_edge)
  AND NOT (a)-[:CO_RETRIEVED_FRESH]->(b)
WITH a, r, b LIMIT 1000
CREATE (a)-[r_fresh:CO_RETRIEVED_FRESH]->(b)
SET r_fresh.last_reinforced_epoch = r.last_reinforced_epoch,
    r_fresh.co_retrieval_count    = r.co_retrieval_count,
    r_fresh.strength              = r.strength,
    r_fresh.migrated_at_epoch     = $current_epoch
RETURN count(r_fresh) AS migrated
```

For Tim's current scale (800 edges), the single-transaction approach is appropriate and simpler to verify.

## Section 4: Rollback Strategy

### Threat model for migration failure

The migration has four failure modes, at different stages:

| Failure point | System state | Recovery action |
|--------------|--------------|-----------------|
| Before Pass 1 starts | Unchanged — no new rels | Nothing to do. Re-run from start. |
| During Pass 1 (FRESH creation), before commit | Transaction rolled back by Neo4j — 0 FRESH rels created | Re-run Pass 1. |
| After Pass 1 commit, before Pass 2 | FRESH rels exist, no STALE rels, all CO_RETRIEVED intact | Re-run Pass 2 (idempotent due to NOT guard). |
| After Pass 2 commit, before Pass 3 | Both FRESH and STALE rels exist, all CO_RETRIEVED intact | Proceed to verification and Pass 3. Safe state — CO_RETRIEVED still exists. |
| During Pass 3 (DELETE), before commit | Transaction rolled back — CO_RETRIEVED still intact | Re-run Pass 3 after re-verifying 0 unmigrated count. |
| After Pass 3 commit | CO_RETRIEVED deleted. FRESH and STALE are the only relationship types. | **This is the only non-recoverable state without a backup.** |

**Critical insight**: The migration is designed so that the original CO_RETRIEVED relationships remain intact until the very end (Pass 3). Passes 1 and 2 are CREATE-only operations on new relationship types. They do not modify or delete any existing data. The database is therefore in a valid, rollback-able state at every point until Pass 3 commits.

### Detecting mid-migration state

If the migration process is interrupted (Docker container restart, network timeout, etc.), run the state detector:

```cypher
// Migration state detector — run after any interruption
MATCH ()-[r_orig:CO_RETRIEVED]->() WITH count(r_orig) AS orig_count
MATCH ()-[r_fresh:CO_RETRIEVED_FRESH]->() WITH count(r_fresh) AS fresh_count, orig_count
MATCH ()-[r_stale:CO_RETRIEVED_STALE]->() WITH count(r_stale) AS stale_count, fresh_count, orig_count
RETURN orig_count, fresh_count, stale_count,
       CASE
           WHEN orig_count > 0 AND fresh_count = 0 AND stale_count = 0 THEN "PRE_MIGRATION"
           WHEN orig_count > 0 AND fresh_count > 0 AND stale_count = 0 THEN "PASS1_COMPLETE_PASS2_PENDING"
           WHEN orig_count > 0 AND fresh_count > 0 AND stale_count > 0 THEN "BOTH_PASSES_COMPLETE_READY_FOR_DELETE"
           WHEN orig_count = 0 AND (fresh_count > 0 OR stale_count > 0) THEN "MIGRATION_COMPLETE"
           WHEN orig_count > 0 AND fresh_count = 0 AND stale_count > 0 THEN "PASS2_ONLY_ANOMALY"
           ELSE "UNKNOWN"
       END AS migration_state
```

Resume instructions based on detected state:
- `PRE_MIGRATION` → Run Pass 1 from scratch
- `PASS1_COMPLETE_PASS2_PENDING` → Run Pass 2 (Pass 1 is idempotent if re-run)
- `BOTH_PASSES_COMPLETE_READY_FOR_DELETE` → Run verification, then Pass 3
- `MIGRATION_COMPLETE` → Nothing to do
- `PASS2_ONLY_ANOMALY` → Anomalous state (Pass 2 ran without Pass 1); run Pass 1 then verify

### Rollback if Pass 3 succeeds but something is wrong

If Pass 3 completes but the system behaves unexpectedly (e.g., a Recall API bug surfaces with the new relationship types), the emergency rollback is:

```cypher
// EMERGENCY ROLLBACK: Re-create CO_RETRIEVED from FRESH edges
MATCH (a:Memory)-[r:CO_RETRIEVED_FRESH]->(b:Memory)
  AND NOT (a)-[:CO_RETRIEVED]->(b)
CREATE (a)-[r_orig:CO_RETRIEVED]->(b)
SET r_orig.last_reinforced_epoch = r.last_reinforced_epoch,
    r_orig.co_retrieval_count    = r.co_retrieval_count,
    r_orig.strength              = r.strength
RETURN count(r_orig) AS restored_fresh

UNION ALL

MATCH (a:Memory)-[r:CO_RETRIEVED_STALE]->(b:Memory)
WHERE NOT (a)-[:CO_RETRIEVED]->(b)
CREATE (a)-[r_orig:CO_RETRIEVED]->(b)
SET r_orig.last_reinforced_epoch = r.last_reinforced_epoch,
    r_orig.co_retrieval_count    = r.co_retrieval_count,
    r_orig.strength              = r.strength
RETURN count(r_orig) AS restored_stale
```

Then delete FRESH and STALE after verification that CO_RETRIEVED count matches the original total. This emergency rollback path is the reason `migrated_at_epoch` is stored on FRESH/STALE edges — it allows auditing which edges were present at migration time.

### Best practice: Neo4j backup before migration

Before running the migration, create a Neo4j database dump:

```bash
# On CasaOS host, stop Neo4j writes (or stop the container briefly)
docker exec -it neo4j neo4j-admin database dump neo4j --to-path=/backups/pre-migration-$(date +%Y%m%d).dump

# Verify dump exists
```

## Section 5: Ongoing Background Reclassification Job

### The reclassification problem

After the initial migration, new co-retrieval events are written as `CO_RETRIEVED_FRESH`. Over time, as epochs advance, some `CO_RETRIEVED_FRESH` edges will age past the tau_edge window (their `last_reinforced_epoch < current_epoch - tau_edge`) and should be reclassified to `CO_RETRIEVED_STALE`. Without this reclassification, the FRESH chain accumulates stale edges, partially defeating the purpose of the type partition.

From Q091: approximately 45 edges age out of the FRESH window per session at Tim's usage rate. These are the edges that were reinforced tau_edge sessions ago and have not been re-reinforced since.

### Trigger point

The reclassification job runs at the end of each session, as part of the existing session-end hook (`recall-session-summary.js`). The trigger is: after the epoch is incremented (`INCR recall:global_epoch`), the job identifies FRESH edges whose `last_reinforced_epoch < new_epoch - tau_edge` and reclassifies them.

This is the natural trigger because:
1. The epoch has just advanced, creating new staleness
2. Session end is already a write-heavy moment (session summary storage), so adding a reclassification batch does not change the critical-path latency profile
3. It bounds the maximum age of a stale edge in the FRESH chain to at most one epoch (one session) of drift

### Reclassification Cypher

```cypher
// Background reclassification job — runs at session end after epoch increment
// Parameters: $current_epoch (new epoch from Redis after INCR), $tau_edge (default 20)
// Reclassifies CO_RETRIEVED_FRESH edges that have aged past the tau_edge window.

// Step 1: Create CO_RETRIEVED_STALE for aged-out FRESH edges (only if no STALE already exists)
MATCH (a:Memory)-[r_fresh:CO_RETRIEVED_FRESH]->(b:Memory)
WHERE r_fresh.last_reinforced_epoch < ($current_epoch - $tau_edge)
  AND NOT (a)-[:CO_RETRIEVED_STALE]->(b)
CREATE (a)-[r_stale:CO_RETRIEVED_STALE]->(b)
SET r_stale.last_reinforced_epoch = r_fresh.last_reinforced_epoch,
    r_stale.co_retrieval_count    = r_fresh.co_retrieval_count,
    r_stale.strength              = r_fresh.strength,
    r_stale.reclassified_at_epoch = $current_epoch
RETURN count(r_stale) AS reclassified_to_stale
```

```cypher
// Step 2: Delete the aged-out FRESH edges (after Step 1 commits)
MATCH (a:Memory)-[r_fresh:CO_RETRIEVED_FRESH]->(b:Memory)
WHERE r_fresh.last_reinforced_epoch < ($current_epoch - $tau_edge)
  AND (a)-[:CO_RETRIEVED_STALE]->(b)
DELETE r_fresh
RETURN count(*) AS fresh_deleted
```

**Atomicity note**: Step 1 and Step 2 are separate transactions. The two-step approach (CREATE then DELETE) matches the migration pattern and provides the same safety property: if Step 1 commits but Step 2 fails, the edge exists in both FRESH and STALE temporarily — an acceptable transient state that is cleaned up on the next reclassification run. Step 2's `AND (a)-[:CO_RETRIEVED_STALE]->(b)` guard ensures it only deletes FRESH edges that have a confirmed STALE counterpart.

**Reverse reclassification (STALE → FRESH)**: If a STALE edge is re-reinforced (the same memory pair co-occurs in a new session), the new co-retrieval event handler should:
1. DELETE the `CO_RETRIEVED_STALE` edge between these two nodes
2. CREATE a new `CO_RETRIEVED_FRESH` edge with the updated `last_reinforced_epoch`
3. Set `co_retrieval_count` = old STALE count + 1, `strength` recalculated

This is the write path for reinforced pairs, distinct from the background reclassification job.

### Expected reclassification volume per session

From Q091's analysis:
- ~45 co-retrieval events per session → 45 new/reinforced CO_RETRIEVED_FRESH edges per session
- At tau_edge=20, the batch of FRESH edges written 20 sessions ago ages out
- Assuming approximately uniform co-retrieval rates, ~45 edges age out per session
- At 1.5ms per edge (Step 1 CREATE + Step 2 DELETE), total reclassification cost: ~45 × 1.5ms = **~67ms per session end**

This is within the background processing budget for a session-end hook that is already writing session summaries, embeddings, and facts.

### Monitoring: leakage detection

The "leakage rate" metric (from Q091 follow-up Q093) is the count of FRESH edges that have aged past tau_edge but not yet been reclassified. At steady state (reclassification runs every session), this count is bounded by the number of edges that aged out in the current session:

```cypher
// Leakage detector: FRESH edges that should be STALE
// Should return 0 if reclassification ran this session, or ~45 if one session was missed
MATCH ()-[r:CO_RETRIEVED_FRESH]->()
WHERE r.last_reinforced_epoch < ($current_epoch - $tau_edge)
RETURN count(r) AS fresh_leakage_count
```

Healthy value: 0 immediately after reclassification, rising to ~45 by the next session end.
Alert threshold: >200 (indicates ~4+ sessions of missed reclassification).

## Section 6: Updated BFS Query Using CO_RETRIEVED_FRESH Only

This is the 15× speedup path established in Q091, now expressed as the production query after migration:

```cypher
// Production BFS query after type-partitioning migration
// Traverses ONLY CO_RETRIEVED_FRESH edges (the fresh sub-chain)
// The tau_edge predicate is now implicit in the relationship type — no WHERE clause needed for edge staleness
// Only the node epoch predicate remains as a post-filter (neighbor.last_access_epoch)

MATCH (seed:Memory {id: $seed_id})-[r:CO_RETRIEVED_FRESH]-(neighbor:Memory)
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)
  AND neighbor.last_access_epoch <= $current_epoch
RETURN
    neighbor.id                AS neighbor_id,
    r.co_retrieval_count       AS edge_weight,
    r.strength                 AS edge_strength,
    neighbor.last_access_epoch AS neighbor_epoch
ORDER BY r.co_retrieval_count DESC
LIMIT 50
```

**What changed vs Q091's current query**:
- `[r:CO_RETRIEVED]` → `[r:CO_RETRIEVED_FRESH]`: traverses only the fresh sub-chain
- `WHERE r.last_reinforced_epoch >= ($current_epoch - $tau_edge)` → **removed**: this predicate is now encoded structurally in the relationship type
- `WHERE neighbor.last_access_epoch >= ...` → **retained**: this node predicate cannot be removed without a corresponding node type partitioning scheme (which Q091 did not address)
- The Q091 WITH-clause staging optimization can optionally still be applied to improve node property load behavior:

```cypher
// Optional: WITH-clause staged version (from Q091 Option 3) applied to FRESH-only traversal
// Further reduces node property load cost for FRESH edges that have stale neighbor nodes
MATCH (seed:Memory {id: $seed_id})-[r:CO_RETRIEVED_FRESH]-(neighbor:Memory)
WITH r, neighbor
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)
  AND neighbor.last_access_epoch <= $current_epoch
RETURN
    neighbor.id                AS neighbor_id,
    r.co_retrieval_count       AS edge_weight,
    r.strength                 AS edge_strength,
    neighbor.last_access_epoch AS neighbor_epoch
ORDER BY r.co_retrieval_count DESC
LIMIT 50
```

**Cost model for the updated query at Month 12, 500-edge hub**:

| Resource | Old (CO_RETRIEVED + post-filter) | New (CO_RETRIEVED_FRESH only) | Improvement |
|----------|----------------------------------|-------------------------------|-------------|
| Relationship records read | F_total = 500 | F_fresh = 50 | **10× reduction** |
| Edge property reads (r.last_reinforced_epoch) | 500 (wasted on stale edges) | 0 (no edge epoch predicate) | **Eliminated** |
| Node property reads (last_access_epoch) | 500 (all edges) | 50 (only FRESH edges) | **10× reduction** |
| Total traversal cost | ~25ms | ~1.6ms | **~15.6× speedup** |

This matches Q091's theoretical ceiling analysis exactly: O(F_fresh) traversal instead of O(F_total).

## Section 7: Integration with Recall Write Path

The migration is not complete without updating the Recall API write path to use the new relationship types. Three write-path operations require changes:

### 7a. New co-retrieval event (first time pair co-occurs)

Before migration:
```cypher
MERGE (a:Memory {id: $id_a})-[r:CO_RETRIEVED]->(b:Memory {id: $id_b})
ON CREATE SET r.last_reinforced_epoch = $current_epoch, r.co_retrieval_count = 1, r.strength = $initial_strength
ON MATCH SET r.last_reinforced_epoch = $current_epoch, r.co_retrieval_count = r.co_retrieval_count + 1
```

After migration:
```cypher
// New co-retrieval: always create as FRESH
MERGE (a:Memory {id: $id_a})-[r:CO_RETRIEVED_FRESH]->(b:Memory {id: $id_b})
ON CREATE SET r.last_reinforced_epoch = $current_epoch, r.co_retrieval_count = 1, r.strength = $initial_strength
ON MATCH SET r.last_reinforced_epoch = $current_epoch, r.co_retrieval_count = r.co_retrieval_count + 1
// Also: if STALE exists for this pair, delete it (the pair has been re-activated)
WITH a, b
OPTIONAL MATCH (a)-[r_stale:CO_RETRIEVED_STALE]->(b)
DELETE r_stale
```

### 7b. Health probe queries (Q092 S5 signal)

The S5 compliance probe in Q092 uses `CO_RETRIEVED` in its BFS. After migration, it must be updated to:
```cypher
MATCH (seed:Memory)-[r:CO_RETRIEVED_FRESH]->(neighbor:Memory)
WHERE seed.id IN $seed_ids ...
```

(And optionally also probe `CO_RETRIEVED_STALE` for a separate staleness coverage metric.)

### 7c. Any diagnostic queries in the dashboard or reporting layer

Any query referencing `CO_RETRIEVED` must be updated to reference `CO_RETRIEVED_FRESH`, `CO_RETRIEVED_STALE`, or both (`[r:CO_RETRIEVED_FRESH|CO_RETRIEVED_STALE]` in Cypher union syntax).

## Key Finding

The CO_RETRIEVED → CO_RETRIEVED_FRESH/CO_RETRIEVED_STALE migration is fully specifiable without APOC and without enterprise Neo4j features. The canonical approach on Tim's CasaOS Community Edition deployment is a two-pass CREATE-before-DELETE migration orchestrated from Python:

1. **Pass 1**: Create CO_RETRIEVED_FRESH for edges with `last_reinforced_epoch >= current_epoch - tau_edge` (idempotent `NOT (a)-[:CO_RETRIEVED_FRESH]->(b)` guard)
2. **Pass 2**: Create CO_RETRIEVED_STALE for edges below threshold (same guard), plus Pass 2b for NULL-epoch edges
3. **Verification**: Confirm 0 unmigrated CO_RETRIEVED edges before proceeding
4. **Pass 3**: DELETE all original CO_RETRIEVED edges (point of no return)

**Runtime**: ~1–2 seconds for 800 edges on CasaOS (WAL-write bound). No batching required at current scale.

**Rollback safety**: The system is in a fully rollback-able state until Pass 3 commits. The original CO_RETRIEVED edges are never modified before deletion. A Neo4j dump before migration provides an additional safety net. Emergency rollback from FRESH/STALE back to CO_RETRIEVED is possible post-migration by inverting the CREATE/DELETE pattern.

**Background reclassification**: Runs at session-end (after epoch INCR), reclassifying ~45 FRESH→STALE edges per session at ~67ms total cost. The session-end trigger bounds FRESH chain drift to one epoch. The leakage detector (`count CO_RETRIEVED_FRESH WHERE last_reinforced_epoch < current_epoch - tau_edge`) is a natural health signal for this job.

**Updated BFS query**: `MATCH (seed:Memory {id: $seed_id})-[r:CO_RETRIEVED_FRESH]-(neighbor:Memory) WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)` — the edge epoch predicate is completely eliminated (encoded structurally in the relationship type). This achieves the Q091 theoretical ceiling: ~15× speedup at Month 12, 500-edge hub nodes (25ms → 1.6ms).

The three qualifying ideas (N≥0.65, E≥0.40):
- `co-retrieved-type-migration-idempotent-two-phase`: N=0.68, E=0.78, F=0.92
- `background-reclassification-session-end-bounded-drift`: N=0.72, E=0.74, F=0.85
- `fresh-only-bfs-zero-edge-predicate-overhead`: N=0.82, E=0.75, F=0.88 ← strongest idea
