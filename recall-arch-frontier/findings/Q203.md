# Finding: Q203 — Vector Space vs Causal Graph as Retrieval Substrate

**Question**: Is the fundamental assumption that AI memory retrieval requires a metric vector space correct — or is the optimal substrate for a coding assistant's memory a causal dependency graph where retrieval is graph traversal, not similarity search?
**Question Type**: [ADVERSARIAL]
**Verdict**: PROMISING
**Severity**: High
**Source field**: Adversarial pair analysis — information retrieval theory, knowledge graph traversal, Recall 1.0's underutilized Neo4j, code dependency graphs (call graphs, import graphs, AST relationships), first-principles analysis of query type distribution in Claude Code sessions.

---

## Evidence

### Thread A: Vector Space is Correct

**Position**: Semantic similarity in a metric vector space is the correct retrieval substrate. Natural language expresses meaning as geometry. Related memories are close in embedding space regardless of their causal relationship. Even memories about causal relationships ("I changed X because Y failed") are correctly retrieved by semantic similarity to queries containing X or Y. The graph layer adds complexity without retrieval precision benefit — as evidenced by Recall 1.0's underutilized Neo4j.

**Thread A's best argument**: The vast majority of Claude Code queries are semantic, not causal. "What were the authentication requirements I discussed?" is a semantic query. "What library do I use for JWT?" is a semantic query. "What was my preferred approach for error handling?" is a semantic query. For all of these, the vector space retrieval is precisely correct: find the memories most semantically similar to the question.

The underutilized Neo4j in Recall 1.0 is evidence for Thread A: when given both options, users' actual query patterns revealed that semantic similarity was sufficient 95%+ of the time. The causal graph was a design investment with minimal retrieval return.

**Thread A's assumed failure mode**: There exist important query types where vector space fails.

**What Thread B exposes**: Thread A's evidence (Neo4j underutilization) is confounded by implementation failure — the CO_RETRIEVED graph was confirmed absent by Q166 (the `CO_RETRIEVED` relationship type was never written to Neo4j). The causal graph was never actually tested in production Recall 1.0. Neo4j underutilization is evidence that the graph was never connected, not that the graph substrate is wrong.

---

### Thread B: Causal Graph is the Correct Substrate

**Position**: A coding assistant's memories are almost entirely causal and procedural: "Service A calls Service B," "Migration M broke feature F," "Pattern P is deprecated because of issue I." These relations are edges in a dependency graph, not points in a continuous metric space. The correct retrieval operation is: given memory M, find all memories that caused, depended on, or were caused by M.

**Thread B's best argument**: Consider the query sequence in a typical Claude Code debugging session:
1. User reports: "File upload is broken"
2. Relevant memories should surface: "We changed the Nginx proxy config last week"
3. Then: "The Nginx change broke multipart form handling"
4. Then: "The original reason we changed Nginx was the SSL cert renewal"

This is a causal chain. Vector similarity to "file upload broken" will surface memories about file upload — but may miss the Nginx config change because "Nginx proxy config" is not semantically close to "file upload." A causal traversal from the upload service → Nginx dependency → config change → cert renewal follows the actual chain. Graph traversal retrieves the entire causal context; vector search finds semantic neighbors but misses causal predecessors.

Thread B's evidence: In code, the most valuable retrieval is often "what was the upstream cause of this behavior?" or "what would be downstream affected by this change?" These are graph operations on a dependency structure, not similarity operations on a metric space.

**What Thread A exposes about Thread B**: Building and maintaining a causal dependency graph requires that every memory write explicitly records causal relationships. Who labels the edges? The user would need to explicitly state "this memory caused that one" — burdensome. An LLM would need to infer causal relationships at consolidation time — expensive and unreliable. The graph only helps if it is populated with correct causal edges. Recall 1.0's Neo4j was populated with `RELATED_TO` edges (weak semantic, not causal) because causal edge extraction was never solved. A causal graph with wrong edges is worse than no graph.

---

### Synthesis: Hybrid Retrieval with Query-Type Classification

Both Thread A and Thread B are wrong in their exclusive claims. The correct architecture recognizes that Claude Code session queries fall into two distinct types:

**Type 1 (80-85% of queries): Semantic/associative queries**
- "What was the authentication approach?"
- "What library did I use for X?"
- "What was my preferred pattern for Y?"
- Optimal substrate: vector space cosine similarity + behavioral CO_RETRIEVED graph
- These queries are well-served by existing retrieval design

**Type 2 (15-20% of queries): Causal/dependency queries**
- "What changed upstream when this broke?"
- "What else depends on this service?"
- "Why did we choose this approach over the alternatives?"
- Optimal substrate: causal dependency graph traversal
- These queries are NOT well-served by vector search — the causal predecessor may have low semantic similarity to the symptom

**The hybrid architecture**: At query time, classify the query into Type 1 or Type 2 before routing to the retrieval substrate.

Type 1 → HNSW cosine search + CO_RETRIEVED graph scoring (existing Recall 2.0 design)
Type 2 → Causal graph traversal starting from the most semantically similar memory

**Query classification signal**: Claude Code hook context provides implicit classification. A query arriving at PostToolUse (after a tool failed or produced unexpected output) is more likely a causal query (what caused this?). A query at UserPromptSubmit at session start is more likely a semantic query (what do I know about X?). Hook context can route without explicit LLM classification.

**The causal graph population problem**: Thread A correctly identifies that causal edge extraction is the blocker. The solution: populate the causal graph from observable structure, not LLM inference.
- Import/require relationships from file contents → `IMPORTS` edges (deterministic)
- Call graph from AST analysis → `CALLS` edges (deterministic)
- Session-temporal proximity → `PRECEDES` edges (from the Q198 relation taxonomy)
- CO_RETRIEVED graph → `CO_OCCURRED` edges (from behavioral scoring)

The causal graph is populated from observable structure, not from LLM inference about causality. This solves Thread B's edge quality problem while retaining causal traversal capability.

**Evidence from knowledge graph research**: The "question type routing" pattern is well-established in enterprise knowledge graph systems. IBM's Watson Knowledge Graph (2018) routes semantic queries to vector search and relational queries to SPARQL traversal. The routing classifier adds ~2ms latency — acceptable for a Claude Code hook.

## Analytical Summary

**Neither pure vector space nor pure causal graph is optimal.** Vector space handles 80-85% of Claude Code queries (semantic/associative). Causal graph handles 15-20% (dependency/cause chains) that vector space cannot reach because causal predecessors may have low semantic similarity to the symptom.

**The synthesis**: Hook-context-based routing (semantic vs causal query type) + separate substrates + observable-structure causal graph population (imports, session ordering, CO_RETRIEVED). No LLM edge extraction required.

**Recall 1.0's Neo4j "failure" was an implementation failure**, not a substrate failure. CO_RETRIEVED was never written. The causal graph was never connected to retrieval. Q166's absence proof is the definitive evidence. Recall 2.0's CO_RETRIEVED implementation closes this gap.

**The causal graph is a v2.1 feature.** Query type routing and observable-structure graph population require the Q198 relation taxonomy implementation. The MVP (Q200's 35-day plan) includes CO_OCCURRED edges from CO_RETRIEVED behavioral scoring — this seeds the causal graph. Full causal traversal with import graphs and call graphs is v2.1 scope.
