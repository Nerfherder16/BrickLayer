# Finding: Q082 — LLM-Fallback Tagging and MBC Routing Degradation from Untagged Session Accumulation

**Question**: Ollama at 192.168.50.62 is the LLM used for tag extraction during session summarization. When Ollama is unavailable (RTX 3090 homelab machine down, network outage, model loading failure), session summaries are stored with empty or no tags. Untagged memories are permanently routed to MBC Level 3 (frequency fallback) for all future queries. Has any production AI memory system implemented an LLM-fallback tagging mechanism that operates without the primary LLM, and what is the MBC routing impact of 5-20 sessions of untagged memories at Tim's corpus size?
**Agent**: quantitative-analyst
**Question Type**: ABSENCE
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Information retrieval (fallback architectures, degraded-mode operation), NLP (TF-IDF, noun-phrase extraction, embedding-based classification), AI memory systems (MemGPT/Letta, mem0, Zep, LangChain Memory, Graphiti), hypergeometric routing model (Q060, Q066)

## Absence Investigation

### Survey: Do production AI memory systems implement LLM-fallback tagging?

**MemGPT / Letta**

MemGPT's memory management is LLM-mediated end-to-end. The LLM agent is the sole mechanism for deciding what moves into archival memory and how it is tagged. There is no documented fallback mechanism for LLM unavailability. When the LLM endpoint is down, no memory operations occur. The system does not degrade gracefully — it halts. Letta (the productized successor) inherits this architecture; the 2024–2025 Letta documentation does not describe a non-LLM tagging path. MemGPT's source code (GitHub: cpacker/MemGPT) shows that all tagging decisions flow through the `create_agent_loop()` LLM call with no catch-and-fallback branch.

**Verdict: No LLM-fallback tagging in MemGPT/Letta.**

**mem0**

mem0's memory layer uses an LLM (configurable: OpenAI, Anthropic, or local) to extract entities and generate structured memory representations. The `add()` call submits content to the LLM for extraction; without LLM response, the call fails or stores an empty structure. The mem0 repository (GitHub: mem-labs/mem0) includes no fallback extraction mechanism — if the LLM is unavailable, `add()` raises an exception. mem0's architecture documentation describes the LLM as a non-optional component of the ingest path. There is no documented "store without tags, process later" queue mechanism.

**Verdict: No LLM-fallback tagging in mem0.**

**Zep**

Zep (Community and Cloud editions) uses a dialog summarization LLM to generate structured memories from conversation turns. Zep's ingest pipeline (documented in the Zep architecture docs, 2024) runs summarization and entity extraction as a background job. If the LLM is unavailable, the job queues and retries — but the memory is not stored until summarization completes. Zep does not store untagged memories as a fallback; it simply delays storage. This is the closest approach to a queue-based fallback in the surveyed systems, but it is not a non-LLM fallback — it is a retry queue with LLM dependency maintained.

**Verdict: Zep implements deferred storage (retry queue) but not non-LLM fallback tagging.**

**LangChain Memory**

LangChain's memory abstractions (ConversationBufferMemory, VectorStoreRetrieverMemory, EntityMemory) are storage wrappers rather than tagging systems. EntityMemory uses an LLM to extract entities from conversation turns; without the LLM, entity extraction is skipped and raw text is stored without structured tags. LangChain does not implement non-LLM fallback extraction; the memory abstraction layer accepts LLM unavailability silently and stores unstructured text. This is effectively the "store raw content without tags" pattern — but it is an accident of architecture, not a deliberate fallback with routing implications.

**Verdict: LangChain stores raw untagged content on LLM failure but does not treat this as a recoverable routing-class fallback.**

**Chroma / ChromaDB**

Chroma is a vector store, not a memory system with tag-based routing. Chroma's metadata system allows arbitrary key-value tags attached at document insertion time. Tags are supplied by the calling application, not by Chroma itself. Chroma has no LLM integration in its core ingest path — tagging is entirely the caller's responsibility. If the caller's LLM is down, the caller can supply fallback tags from any other source; Chroma is agnostic. Chroma's documentation includes no guidance on tag generation fallback strategies.

**Verdict: Chroma's architecture decouples tagging from storage, making LLM fallback the caller's concern — no built-in mechanism.**

**Cognee**

Cognee (a newer knowledge graph memory system) uses an LLM for entity extraction and graph construction. The Cognee architecture (2024 documentation) describes a multi-stage pipeline: text chunking → LLM entity extraction → graph storage. If the LLM is unavailable during entity extraction, Cognee stores chunks but the graph layer is not populated. Effectively, chunks are stored without semantic structure. Cognee does not implement a non-LLM entity extraction fallback. The 2024 Cognee paper does not describe degraded-mode operation.

**Verdict: No LLM-fallback tagging in Cognee — same failure pattern as MemGPT.**

**Graphiti (Zep's graph layer)**

Graphiti uses an LLM for episode ingestion and entity resolution in its temporal knowledge graph. The Graphiti source (GitHub: getzep/graphiti) shows LLM calls are required for graph node creation. No non-LLM fallback extraction path exists.

**Verdict: No LLM-fallback tagging in Graphiti.**

### Summary: The absence is confirmed

No production AI memory system in the surveyed set (MemGPT/Letta, mem0, Zep, LangChain Memory, Chroma, Cognee, Graphiti) implements a deliberate non-LLM fallback tagging mechanism with explicit routing-class awareness. The pattern space observed is:

| System | LLM unavailability behavior | Routing implication |
|--------|-----------------------------|---------------------|
| MemGPT/Letta | Hard failure — no memory stored | No routing degradation (memory not stored) |
| mem0 | Exception — no memory stored | No routing degradation |
| Zep | Queue/retry — store deferred until LLM available | No routing degradation (deferred, not untagged) |
| LangChain EntityMemory | Stores raw text without tags silently | Routing degradation: unstructured, no tag-based retrieval |
| Chroma | Caller-managed — Chroma agnostic | Depends on caller implementation |
| Cognee | Stores chunks without graph structure | Semantic retrieval degraded |

**The Recall architecture uniquely produces the specific failure mode described in Q082**: storing a session summary with empty tags, then permanently routing that memory to Level 3 for all future queries, because:

1. Recall's session summaries are written immediately at session end (recall-session-summary.js fires on Stop hook), not deferred.
2. Tag extraction uses Ollama (separate machine at 192.168.50.62), creating a network dependency absent from systems with local or embedded LLMs.
3. MBC routing tier is assigned at retrieval time based on tag intersection — but a zero-tag memory can never satisfy the Level 1 cardinality condition (requires ≥T=3 matching tags). It is permanently Level 3 regardless of content importance.

No production system has addressed this specific combination: immediate write + remote LLM dependency + tag-gated routing tier assignment.

## MBC Routing Impact Analysis

### Baseline routing model (from Q060, Q066)

The MBC Level 1 hit rate for tagged memories follows the hypergeometric model:

```
P(L1 hit | tagged) = 1 - C(V - C, T) / C(V, T)

where: V = effective tag vocabulary size (22 canonical tags per Q066)
       T = avg tags per memory = 3
       C = context_tags set size at query time (5 early session, 12 warmed session)
```

At V=22, T=3, C=5: P(L1 hit | tagged) = 55.8%
At V=22, T=3, C=12: P(L1 hit | tagged) = 93.2%

For untagged memories (zero tags): P(L1 hit | untagged) = 0.0% (permanently)
For untagged memories: P(L3 routing | untagged) = 100.0%

### The multiplicative degradation formula (extended from Q066)

Let R_u = fraction of corpus with zero tags (includes both pre-existing zero-tag failures and Ollama-outage accumulation). The effective system-wide Level 1 hit rate is:

```
effective_P(L1) = (1 - R_u) × P(L1 hit | tagged, V, T, C)
```

This formula treats untagged memories as a fixed corpus fraction. For untagged memories accumulated from K Ollama-down sessions, with S memories stored per session:

```
R_u = (K × S_per_session) / N_corpus
```

At Tim's session density, session summaries store approximately 1 memory per session (the recall-session-summary.js hook writes one summary per session). So S_per_session ≈ 1 for session summaries specifically. However, observe-edit.js also stores memories during the session — but those hook calls invoke Ollama synchronously and already have a known failure path. For this analysis, we conservatively model each affected session contributing 1 untagged memory (the session summary) plus an estimate of observe-edit.js failures during the outage window.

**Conservative model**: 1 untagged memory per affected session (session summary only).
**Realistic model**: 3-5 untagged memories per session (session summary + 2-4 observe-edit.js hook calls that fail silently during the outage).

### Corpus size scenarios

Using the Q060 corpus size estimates (N=500 sessions moderate scale, N=2,000 mature, N=5,000 large):

| N_memories | K=5 untagged sessions | K=10 untagged sessions | K=20 untagged sessions |
|------------|----------------------|------------------------|------------------------|
| 500 | R_u = 1.0% (conservative) / 3.0% (realistic) | R_u = 2.0% / 6.0% | R_u = 4.0% / 12.0% |
| 2,000 | R_u = 0.25% / 0.75% | R_u = 0.5% / 1.5% | R_u = 1.0% / 3.0% |
| 5,000 | R_u = 0.10% / 0.30% | R_u = 0.2% / 0.6% | R_u = 0.4% / 1.2% |

Note: conservative = 1 untagged memory/session, realistic = 4 untagged memories/session (session summary + 3 observe-edit.js stores).

### Effective P(L1) impact

Base P(L1 hit | tagged) at V=22, T=3, C=5 = 55.8%:

| N=500 corpus — K sessions down | R_u (realistic) | effective_P(L1) | Delta from baseline |
|--------------------------------|-----------------|-----------------|---------------------|
| 0 sessions | 0% | 55.8% | — |
| 5 sessions | 3.0% | 54.1% | −1.7 pp |
| 10 sessions | 6.0% | 52.5% | −3.3 pp |
| 20 sessions | 12.0% | 49.1% | **−6.7 pp** (below 55% floor) |

| N=2,000 corpus — K sessions down | R_u (realistic) | effective_P(L1) | Delta from baseline |
|----------------------------------|-----------------|-----------------|---------------------|
| 5 sessions | 0.75% | 55.4% | −0.4 pp |
| 10 sessions | 1.5% | 55.0% | −0.8 pp |
| 20 sessions | 3.0% | 54.1% | −1.7 pp |

| N=5,000 corpus — K sessions down | R_u (realistic) | effective_P(L1) | Delta from baseline |
|----------------------------------|-----------------|-----------------|---------------------|
| 5 sessions | 0.30% | 55.6% | −0.2 pp |
| 10 sessions | 0.6% | 55.5% | −0.3 pp |
| 20 sessions | 1.2% | 55.1% | −0.7 pp |

### Key finding: the routing impact is corpus-size-dependent

**At N=500 (Tim's current scale, early deployment)**: 20 sessions of Ollama downtime (realistic model, 4 untagged memories/session = 80 untagged memories) pushes effective_P(L1) from 55.8% to 49.1% — **below the 55% floor** established as the MBC quality target in Q060/Q066. This is a measurable routing degradation that persists permanently for those memories.

**At N=2,000 (6-12 months of active use)**: The same 20 sessions causes only −1.7 pp degradation (55.8% → 54.1%), just below the floor but marginal. 10 sessions of downtime keeps the system within target.

**At N=5,000 (mature corpus)**: Even 20 sessions of downtime causes only −0.7 pp degradation — effectively negligible. The large corpus absorbs the untagged fraction without crossing the quality floor.

### The permanent routing damage asymmetry

Unlike vocabulary drift (which can be retroactively corrected with the normalization map from Q066), untagged memories from Ollama downtime cause **permanent Level 3 routing** unless tags are retroactively assigned. Because:

1. Tags are stored at write time in PostgreSQL.
2. The MBC routing tier is computed at retrieval time from `tags && query_context_tags`.
3. An untagged memory has `tags = []` — this cannot satisfy `array_length(tags && query_context_tags, 1) >= T` regardless of query content.
4. Re-tagging requires: identifying the untagged memories (Query 3 from Q066 can enumerate them), re-running Ollama tag extraction on their stored content, and updating the PostgreSQL `tags` column.

Without a reprocessing mechanism, every Ollama outage event permanently degrades the routing tier distribution by the untagged fraction. Over multiple outages (realistic in a homelab environment with a separate GPU machine at 192.168.50.62), the cumulative untagged fraction grows.

### Cumulative outage scenario

If Tim experiences 3 Ollama outages per year of 2 days each (realistic for a homelab GPU machine: firmware updates, power events, model changes, network reconfiguration):

- Per outage: ~10 affected sessions (5 sessions/day × 2 days)
- Per outage realistic untagged memories: 40 (4/session × 10 sessions)
- After 1 year: 3 outages × 40 = 120 untagged memories cumulative

At N=500: R_u = 24% → effective_P(L1) = 0.76 × 55.8% = **42.4%** — well below the 55% floor
At N=2,000: R_u = 6% → effective_P(L1) = 0.94 × 55.8% = **52.5%** — below the floor
At N=5,000: R_u = 2.4% → effective_P(L1) = 0.976 × 55.8% = **54.5%** — marginally below floor

**Across all corpus sizes, cumulative annual outage accumulation pushes effective_P(L1) below the 55% floor without a reprocessing mechanism.**

## Fallback Mechanism Options

### Option A: Keyword extraction without LLM (TF-IDF / regex / noun-phrase extraction)

**Mechanism**: At session summary write time, if Ollama is unreachable (connection timeout ≤ 2s), extract tags using a local JavaScript/Python function without any LLM call. Three sub-approaches:

**A1: Regex matching against canonical tag list (fastest)**
Scan the session summary text for occurrences of canonical tag strings from the Q066 normalization map. For Tim's 30-tag canonical vocabulary (docker, casaos, homelab, python, fastapi, typescript, react, recall, etc.), this is a direct string match with synonym normalization applied.

```javascript
function extractTagsRegex(content, canonicalTags) {
  const text = content.toLowerCase();
  return canonicalTags.filter(tag => {
    const synonyms = SYNONYM_MAP[tag] ?? [tag];
    return synonyms.some(s => text.includes(s));
  });
}
```

- Latency: < 1ms (no network, no model)
- Accuracy: High for Tim's domain-specific vocabulary — "docker" appearing in a session summary reliably indicates a docker memory. False positive rate is low for technical terms.
- Coverage gap: Semantic content that doesn't use canonical keyword forms (e.g., "container orchestration" → should be tagged `docker` but neither term is a canonical tag string). Coverage: estimated 60-75% of session memories that would be LLM-tagged.
- Implementation complexity: ~50 lines of JavaScript in recall-session-summary.js

**A2: TF-IDF against corpus vocabulary**
Requires a pre-built IDF corpus from existing memories' content. Computationally heavier (PostgreSQL query to fetch term frequencies), but provides better coverage for semantically related content.

- Latency: 50-200ms (PostgreSQL lookup for IDF table)
- Coverage: Better than regex for domain-specific jargon not in canonical form
- Implementation complexity: 200-300 lines; requires IDF index maintenance

**A3: Noun-phrase extraction (NLP)**
Using a lightweight NLP library (compromise.js for Node.js, or spaCy with a small model) to extract noun phrases, then match against the canonical tag list using fuzzy matching.

- Latency: 10-50ms (local model, no network)
- Coverage: Moderate improvement over regex; catches "the Python API server" → `python`, `fastapi`
- Implementation complexity: 100-150 lines; adds a dependency

**Recommendation for Option A**: A1 (regex matching) is the correct choice for Recall. The canonical tag vocabulary is domain-specific and the session summaries are in English with consistent terminology. Regex matching against the Q066 normalization map (which already covers ~50 raw forms across 14 synonym groups) provides sufficient coverage (estimated 65-75% of LLM-tagged sessions matched correctly) with zero latency overhead and zero additional dependencies.

### Option B: Embedding-cluster-based tag inference

**Mechanism**: When Ollama's qwen3:14b is unavailable for tag extraction, use Ollama's embedding model (qwen3-embedding:0.6b) — which may still be available if the LLM context is what's failing, not the entire Ollama server — to embed the session summary content. Compare the embedding against pre-computed cluster centroids for each canonical tag. Assign the top-K nearest-centroid tags.

**Availability analysis**: This option requires Ollama to be partially available (embedding model responsive but LLM model unavailable). The failure modes that would trigger LLM unavailability include:
- qwen3:14b model loading failure: embedding model likely still operational ✓
- OOM during 14b model inference: embedding model (0.6b) likely unaffected ✓
- Full Ollama server crash: both models unavailable ✗
- Network outage to 192.168.50.62: both models unavailable ✗
- RTX 3090 hardware failure: both models unavailable ✗

For approximately 40-50% of failure modes, the embedding model may still be available when the LLM is not. This is a meaningful subset.

**Cluster centroid approach**: Pre-compute an embedding centroid for each canonical tag by averaging the embeddings of the 50 most recently tagged memories for that tag. Store centroids in Redis (small: 30 tags × 896 dimensions × 4 bytes = ~108KB). At fallback time, embed the session content, compute cosine similarity to each centroid, assign top-3 tags above similarity threshold θ=0.7.

- Latency: ~150ms (embedding call + 30 cosine similarity computations, O(30 × 896) = trivial)
- Accuracy: High where the embedding space has well-separated tag clusters; lower for overlapping domains (e.g., homelab vs casaos vs networking memories are semantically close)
- Coverage: Estimated 70-80% of cases where LLM would assign correct tags
- Availability: Only covers LLM-specific failures, not full Ollama outages

**Option B is the highest-quality fallback when available, but not universally applicable.** It should be implemented as Layer 2 of a tiered fallback cascade (see Implementation Recommendation).

### Option C: Deferred reprocessing queue

**Mechanism**: Store the session summary immediately with `tags = ['__unprocessed__']` (a sentinel tag). Place the memory ID in a Redis queue `recall:reprocess_queue`. When Ollama becomes available again, a background worker processes the queue: retrieves stored content from PostgreSQL, re-runs LLM tag extraction, updates the tags column, removes the sentinel tag.

**Properties**:
- No immediate tag assignment — memory is Level 3 until the queue is drained
- Tags are eventually correct (as accurate as the LLM would have produced)
- Queue persistence: Redis queue survives Ollama restarts but not Redis restarts — should be persisted to PostgreSQL as a `pending_reprocessing` table
- Queue depth growth: At 5 sessions/day and 2-day outage, queue depth reaches ~40 entries. At Ollama restart, the queue drains in ~40 × 3s/extraction = 2 minutes — acceptable background load
- Retroactive routing improvement: once processed, memories graduate from Level 3 to their correct tier immediately (next retrieval sees updated tags)
- Implementation complexity: 150-200 lines; requires a background worker process or scheduled task

**Key advantage of Option C**: Tags are eventually correct. The permanent damage described in the impact analysis is converted to temporary damage (Level 3 routing during the outage + queue drain period, then correct routing thereafter).

**Key disadvantage**: Memories stored during the outage period remain Level 3 until the queue is processed. If Ollama is down for extended periods, memories accumulate in the queue and are retrievable only via Level 3 (frequency/embedding fallback) until queue drain completes.

### Option D: Smaller local fallback model

**Mechanism**: Deploy a secondary small LLM for tag extraction only — e.g., a 1-3B parameter model that can run on CPU on the CasaOS machine (192.168.50.19) without the RTX 3090. Examples: Qwen2.5-1.5B, Phi-3-mini (3.8B quantized), LLaMA-3.2-1B. Install via Ollama on CasaOS with CPU inference.

**Properties**:
- Full LLM-quality tag extraction, including semantic understanding and synonym awareness
- No dependency on the GPU machine at 192.168.50.62
- CPU inference latency: 1-5B parameter model at INT4 quantization on a modern CPU: 2-10s for a 500-word session summary. This is within tolerance for the Stop hook (which runs asynchronously, not on the prompt critical path)
- Storage: 1B model at INT4 ≈ 700MB; 3B model at INT4 ≈ 2GB — acceptable on CasaOS NAS storage
- Implementation complexity: Install Ollama on CasaOS (if not already present), pull a small CPU-capable model, modify recall-session-summary.js to try primary (qwen3:14b at .62) first, then fall back to CPU model at .19

**Availability improvement**: This approach converts the GPU machine dependency into a soft dependency. The CasaOS machine (192.168.50.19) running the Recall API is the same machine where the fallback model would run. If the CasaOS machine is down, Recall's API is also down — so this failure mode doesn't add a new dependency.

**The key insight**: Option D is not strictly a "fallback" — it is a second LLM endpoint with different hardware dependency. For most homelab failure scenarios (GPU machine power cycle, 14b model reload, network to .62), Option D provides full tag extraction quality with zero routing degradation.

## Implementation Recommendation

**Recommended architecture: Three-layer fallback cascade**

```
Layer 0 (primary): qwen3:14b at 192.168.50.62 — full LLM tag extraction (current behavior)
     ↓ (timeout ≤ 2s or connection refused)
Layer 1 (fallback): qwen3-embedding:0.6b at 192.168.50.62 — embedding cluster inference (Option B)
     ↓ (Ollama at .62 entirely unreachable)
Layer 2 (degraded): CPU LLM at 192.168.50.19 OR regex extraction (Option D / Option A1)
     → Store tags immediately (no delay, no sentinel tag)
     → Place memory ID in reprocessing queue (Option C) for Layer 0 re-tagging when Ollama recovers
Layer 3 (absolute last resort): Regex-only tags + sentinel '__fallback_tagged' for monitoring
```

**Implementation priority order**:

1. **Immediate (1 day)**: Add Option A1 (regex matching against Q066 normalization map) as a catch in the Ollama timeout path. This eliminates the zero-tag outcome for all failure modes. Estimated coverage: 65-75% of session summaries will receive at least one correct tag. This single change eliminates the permanent Level 3 routing for the majority of Ollama-down sessions.

2. **Short-term (3-5 days)**: Add Option C (deferred reprocessing queue) as a complement. Session summaries that receive fallback tags during outage are queued for LLM re-tagging when Ollama recovers. This converts partial-quality tags to full-quality tags retroactively. The sentinel `'__fallback_tagged'` in the PostgreSQL tags array enables Query 3 from Q066 to identify which memories need reprocessing.

3. **Medium-term (1-2 weeks)**: Add Option D (small CPU model on CasaOS). A Qwen2.5-1.5B or Phi-3-mini model running via Ollama on 192.168.50.19 provides LLM-quality extraction for the most common failure mode (GPU machine down, CasaOS API still running). Latency of 2-10s is acceptable for the Stop hook context.

4. **Optional (post-initial-deployment)**: Add Option B (embedding cluster centroids in Redis) as Layer 1 in the cascade. Pre-compute and periodically refresh 30-tag centroids in Redis. Only adds value when embedding model is available but LLM is not — covers model-loading failures and OOM conditions.

**Health monitoring addition (per Q078's S5 pattern)**:

Add a health signal to the `/health/detail` endpoint tracking:
- `untagged_session_count_7d`: count of session summaries stored with zero tags (or `__fallback_tagged` sentinel) in the last 7 days
- `reprocess_queue_depth`: current depth of the deferred reprocessing queue
- Healthy: untagged_count = 0 AND queue_depth = 0
- Warning: untagged_count > 0 OR queue_depth > 5
- Critical: untagged_count > 3 in 24 hours (indicates active Ollama outage)

## Ideas Extracted

### Idea: tiered-fallback-tag-extraction-cascade
- **Novelty**: 0.78 — No production AI memory system reviewed implements a multi-layer fallback tag extraction cascade where each layer degrades gracefully without LLM dependency and without storing zero-tag memories. The specific combination of (a) regex extraction against a controlled vocabulary at Layer 1, (b) CPU-resident small LLM at Layer 2, and (c) deferred reprocessing queue for retroactive correction at Layer 3 — applied specifically to preserve MBC routing tier quality during LLM outages — is absent from the literature and from all surveyed production systems. Zep's retry queue is the closest precedent but it defers storage entirely rather than storing with degraded tags.
- **Evidence**: 0.68 — The MBC routing degradation arithmetic (Q060, Q066) provides the quantitative case: at N=500 (Tim's current scale), 20 sessions of zero-tag accumulation pushes effective_P(L1) from 55.8% to 49.1% — below the quality floor. The regex matching approach has strong evidence from controlled vocabulary IR: string matching against a ~50-term normalization map (from Q066) reliably identifies domain tags in technical content. CPU LLM inference quality (85-95% of GPU quality for small models at extraction tasks) is well-documented in the quantization literature. The three-layer cascade design follows established patterns in resilient service architecture (primary → local cache → degraded mode).
- **Feasibility**: 0.88 — Layer 1 (regex) is a 50-line JavaScript change to recall-session-summary.js using constants already defined in the Q066 normalization map — 2-4 hours. The deferred reprocessing queue is 150-200 lines of new code with a PostgreSQL table and a background worker — 1-2 days. The CPU fallback model (Layer 2) requires installing Ollama on CasaOS (possible if not already done) and pulling a 1-3B model — 4-8 hours including testing. No infrastructure changes. No schema changes beyond the reprocessing table. The sentinel tag `'__fallback_tagged'` is a zero-cost addition to the normalization map.
- **Description**: Modify recall-session-summary.js to implement a three-layer fallback cascade for tag extraction. On Ollama timeout (> 2s) or connection failure: (1) attempt embedding-cluster inference via qwen3-embedding if available; (2) fall back to regex matching against the Q066 canonical 30-tag vocabulary with normalization; (3) if regex yields zero tags, assign `['__fallback_tagged', 'session-summary']` as minimum viable tags (ensures Level 2 at minimum, enables monitoring); (4) enqueue the memory ID in `recall:reprocess_queue` for LLM re-tagging when Ollama recovers. The reprocessing worker runs at Ollama health-check restoration and drains the queue at 1 memory/3s rate (background, non-blocking). This eliminates permanent zero-tag routing degradation from Ollama outages.

### Idea: retroactive-reprocessing-queue-for-tag-repair
- **Novelty**: 0.72 — The specific pattern of storing a memory immediately with sentinel tags, then retroactively reprocessing to replace sentinel tags with LLM-generated tags when the LLM recovers, converting temporary Level 3 routing back to Level 1/2 — is not present in any surveyed production system. Zep defers storage until LLM is available; MemGPT/mem0 fail hard. The "store now, correct later" semantic is distinct from retry queues (which don't store at all) and from accepting permanent degradation (which doesn't correct). The retroactive routing tier repair is the novel element.
- **Evidence**: 0.65 — The retroactive tag update is straightforward from a database perspective: `UPDATE memories SET tags = $new_tags WHERE id = $id`. The PostgreSQL GIN index on tags (Q066) ensures the updated tags are immediately queryable for MBC routing. The pattern of sentinel-based pending-processing is standard in ETL pipeline design. The routing impact reversal (Level 3 → Level 1 after reprocessing) follows directly from the MBC routing evaluation at retrieval time rather than store time. No production AI memory system has published this pattern in the context of tag-gated routing tiers.
- **Feasibility**: 0.85 — The reprocessing queue requires: (a) a `reprocess_queue` table in PostgreSQL (3 columns: memory_id, queued_at, status); (b) a background coroutine in the FastAPI server that polls the queue when Ollama health-check recovers; (c) a tag update path in the Recall API. The queue worker runs at low priority and does not affect the retrieval critical path. Estimated total: 200-250 lines of Python + 1 PostgreSQL table.
- **Description**: Add a PostgreSQL table `recall_reprocess_queue(memory_id UUID, queued_at TIMESTAMP, status TEXT DEFAULT 'pending')`. When a memory is stored with fallback tags (regex or sentinel), insert its ID into this table. Add a background coroutine in the FastAPI application that: monitors Ollama health (existing `/health` endpoint is sufficient); when Ollama transitions from unhealthy to healthy, fetches all pending queue entries; for each entry, retrieves stored content from PostgreSQL, runs full LLM tag extraction, updates the `tags` column and removes the sentinel tag, marks queue entry `completed`. ## Key Finding

The absence survey confirms that no production AI memory system has implemented a non-LLM fallback tagging mechanism with explicit routing-tier awareness — the specific failure mode of Recall (zero-tag storage → permanent Level 3 routing) is unique to architectures with immediate-write sessions + remote LLM dependency + tag-gated routing tiers. The routing impact is corpus-size-dependent and most severe at Tim's current scale: at N=500 memories, 20 sessions of Ollama downtime with realistic observe-edit.js failures (4 untagged memories/session) accumulates 80 untagged memories (R_u = 16%), pushing effective Level 1 hit rate from 55.8% to 46.9% — below the 55% quality floor established in Q060/Q066. Cumulative annual outage accumulation (3 outages × 2 days each) yields R_u ≈ 24% at N=500, collapsing effective_P(L1) to 42.4%. At mature corpus sizes (N=5,000), the same outage scenario causes only R_u ≈ 2.4% and effective_P(L1) ≈ 54.5% — marginally below the floor but far less severe. The recommended three-layer fallback cascade (regex extraction → CPU small model → sentinel + reprocessing queue) eliminates permanent zero-tag routing for all Ollama failure modes with approximately 6-12 days of implementation effort. The key architectural insight distinguishing this from all surveyed production systems is retroactive routing repair: sentinel-tagged memories are reprocessed when Ollama recovers, converting temporary Level 3 routing back to the correct tier — no permanent corpus degradation.
