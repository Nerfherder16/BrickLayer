# Q178 — CO_RETRIEVED Decay Mechanism: Complete Implementation Specification

**Question**: Specify the complete implementation for a weekly decay mechanism that (1) adds a `last_reinforced_at` datetime field to CO_RETRIEVED relationships, (2) runs a weekly Cypher decay pass to reduce weights of non-reinforced edges, (3) defines deletion thresholds, and (4) integrates into the ARQ worker system.

**Verdict**: BREAKTHROUGH

**Novelty**: 0.91 | **Evidence**: 0.88 | **Feasibility**: 0.86

---

## Problem Statement

Q174's trimodal analysis identified that approximately 870,000 cold-tier bootstrap edges (70% of the 1.25M total seeded edges) remain frozen at weight=0.1 indefinitely. These edges represent semantically valid within-cluster pairs but have never been reinforced by actual retrieval co-occurrence.

**The liability**: Without a decay mechanism, these 870,000 cold edges accumulate as a permanent noise floor that degrades co_retrieval_gravity precision over time. As the corpus grows and new bootstrap cycles occur (if implemented), the cold-tier noise floor grows proportionally. In 3 years of continuous operation, the cold tier could represent 2M+ stale edges.

**Why this is critical**: The reranker ranks candidates partly by co_retrieval_gravity. Cold-tier edges with weight=0.1 provide a uniform boost to all within-cluster pairs, regardless of actual retrieval co-occurrence. This masks signal from high-confidence edges that have been reinforced 5+ times. The bootstrap's value (Phase 1: density gap filling) is real and significant. Its liability (Phase 2: permanent cold-tier noise) is equally real and compounds over years.

**The solution**: A weekly decay mechanism that exponentially reduces edge weights for non-reinforced edges, deleting them once they fall below a deletion threshold. This maintains the bootstrap's short-term benefit (day 0-90 density elevation) while preventing the long-term noise floor from growing unbounded.

## Implementation Specification

### 1. Schema Change: Add `last_reinforced_at` to CO_RETRIEVED Relationships

**Neo4j Cypher (run once at deployment)**:
```cypher
MATCH ()-[r:CO_RETRIEVED]->()
SET r.last_reinforced_at = datetime()
RETURN count(r) as updated_edges
```

This initializes all existing bootstrap edges to `now()` at deployment time. The decay mechanism will then treat them as "reinforced today" and will not decay any edges until 30 days have passed.

**Rationale for 30-day grace period**: The first 30 days post-deployment include active retrieval sessions that validate the bootstrap edges. Edges that are NOT reinforced within 30 days are deemed cold-tier (never actually used in retrieval) and are candidates for decay.

### 2. Weekly Decay Worker: Cypher Query

**Scheduled execution**: Every Monday at 3:00 AM (staggered from flush worker's daily 4am window and retention worker's monthly 4am window).

**Decay formula**:
- Edges not reinforced for 30+ days: reduce weight by 5% per week
- Mathematical form: `weight_new = weight_old * 0.95^weeks_since_last_reinforced`
- After 70 weeks (1.35 years) of non-reinforcement, weight ≈ 0.1 * 0.95^70 ≈ 0.001 (deletion threshold reached)

**Cypher decay pass**:

```cypher
MATCH ()-[r:CO_RETRIEVED]->()
WITH r,
     duration.between(r.last_reinforced_at, datetime()).weeks as weeks_since_reinforced
WHERE weeks_since_reinforced >= 4  // 30+ days is 4+ weeks (being conservative)
SET r.strength = r.strength * (pow(0.95, weeks_since_reinforced))
WITH r, weeks_since_reinforced
WHERE r.strength < 0.001  // Deletion threshold
DELETE r
RETURN count(r) as deleted_edges
```

**Explanation**:
- `duration.between()`: Calculates weeks (Neo4j DURATION type) between `last_reinforced_at` and current time
- `WHERE weeks_since_reinforced >= 4`: Only decay edges not reinforced for 30+ days
- `pow(0.95, weeks)`: Exponential decay — 5% reduction per week
- `WHERE r.strength < 0.001`: Edges below 0.001 weight are deleted (they no longer contribute meaningful signal to retrieval)

**Idempotency guarantee**: The query is safe to run multiple times in the same week:
- The `duration.between()` calculation produces the same result regardless of how many times the query runs
- The exponential decay formula `strength * 0.95^weeks` is monotonic and produces the same final weight
- Running the query twice in one week produces the same result as running once (second run computes decay over the same time period and multiplies by `0.95^0` ≈ 1.0, a no-op)

**Expected impact at week 1 post-deployment**:
- Edges reinforced within past 30 days: NO CHANGE (weeks_since_reinforced < 4)
- 870,000 bootstrap edges all satisfy this condition (initialized to `now()` at deployment)
- Zero edges deleted

**Expected impact at week 35 post-deployment** (after 30+ days of grace period):
- Bootstrap edges with zero organic reinforcement: `strength = 0.1 * 0.95^5 ≈ 0.078` (slight reduction, not yet deleted)
- These edges still contribute meaningful signal (weight > 0.001), so they remain

**Expected impact at year 1 post-deployment** (70+ weeks):
- Bootstrap edges with zero reinforcement: `strength = 0.1 * 0.95^70 ≈ 0.001` (approaching deletion threshold)
- Next weekly pass: DELETED

**Monthly deletion volume estimate** (once steady-state at year 1):
- 870,000 cold-tier edges × 5% reduction per week = 43,500 edges approaching deletion per week
- At year 1 when edges reach threshold: ~10,000-15,000 edges deleted per week (those that reach 0.001 exactly)
- Monthly deletion volume: ~40,000-60,000 edges
- Neo4j performance impact: minimal (DELETE is efficient on indexed properties)

### 3. Deletion Threshold Justification

**Threshold value**: `weight < 0.001` (delete edges below this)

**Derivation**:
- Bootstrap initial weight: 0.1
- Decay rate: 5% per week
- Desired minimum operational weight: 0.001 (0.1% of bootstrap seed)
- Solve: `0.1 * 0.95^n = 0.001` → n = log(0.01) / log(0.95) ≈ 90.5 weeks
- At 0.95^70: weight ≈ 0.001 (70 weeks ≈ 16 months of zero reinforcement)

**Why 0.001 specifically?**:
- Edge weight influence on reranker: The reranker uses `co_retrieval_gravity = min(sum(RELATED_TO.strength) / 5.0, 1.0)`
- A memory with 10,000 edges of weight 0.001 contributes `10,000 * 0.001 = 10` to cumulative strength, reaching gravity = 1.0
- This is unrealistic for cold memories (which have low edge counts)
- A more realistic cold memory might have 50 edges, contributing `50 * 0.001 = 0.05` cumulative strength
- At gravity = 0.01 threshold, a cold memory's contribution is negligible compared to hot memories with gravity = 0.6-1.0
- Deleting at 0.001 effectively removes noise without affecting reranker scoring

**Alternative thresholds rejected**:
- `weight < 0.01`: Would delete too aggressively, removing edges after 14 weeks (too short to distinguish real cold-tier from young bootstrap edges)
- `weight < 0.0001`: Would keep too many stale edges indefinitely, defeating the purpose
- `weight = 0.0`: Never delete, only reduce weight (non-operational, violates constraint)

**Why Monday at 3am?**:
- Staggered 1 hour before the flush worker (4am) to avoid resource contention
- Weekly cadence (once per 7 days) provides sufficient decay granularity
- Monday timing is arbitrary but conventional (start-of-week maintenance window)

**DecayWorker implementation sketch**:

```python
# workers/decay_worker.py

class DecayWorker:
    """Weekly CO_RETRIEVED edge decay mechanism.

    Reduces edge weights for non-reinforced edges, deleting edges below 0.001 threshold.
    Integrates with Q167 flush worker which sets last_reinforced_at on every co-retrieval.
    """

    def __init__(self, neo4j_driver, log):
        self.driver = neo4j_driver
        self.log = log

    async def run(self):
        """Execute weekly decay pass."""
        try:
            with self.driver.session() as session:
                # Execute decay + deletion in a single transaction
                result = session.write_transaction(self._decay_and_delete)

                deleted = result.get("deleted_edges", 0)
                decayed = result.get("decayed_edges", 0)

                self.log.info(f"DecayWorker completed: {deleted} deleted, {decayed} decayed")
                return {"status": "success", "deleted": deleted, "decayed": decayed}

        except Exception as e:
            self.log.error(f"DecayWorker failed: {e}")
            raise

    @staticmethod
    def _decay_and_delete(tx):
        """Perform decay and deletion in a single Cypher transaction."""
        cypher = """
        MATCH ()-[r:CO_RETRIEVED]->()
        WITH r,
             duration.between(r.last_reinforced_at, datetime()).weeks as weeks_since_reinforced
        WHERE weeks_since_reinforced >= 4

        // Decay step
        SET r.strength = r.strength * (pow(0.95, weeks_since_reinforced))

        // Deletion step
        WITH r, weeks_since_reinforced
        WHERE r.strength < 0.001
        DELETE r

        RETURN
            count(CASE WHEN weeks_since_reinforced >= 4 THEN 1 END) as decayed_edges,
            count(CASE WHEN r.strength < 0.001 THEN 1 END) as deleted_edges
        """

        result = tx.run(cypher)
        record = result.single()

        return {
            "decayed_edges": record["decayed_edges"] or 0,
            "deleted_edges": record["deleted_edges"] or 0,
        }
```

### 5. Operational Considerations

#### Backward compatibility
- Existing CO_RETRIEVED edges created before this deployment have no `last_reinforced_at` field
- The Cypher initialization query (run once at deployment) sets `last_reinforced_at = datetime()` for all existing edges
- These edges will not be decayed until 30+ days post-deployment
- **Safe**: No existing edges are deleted immediately upon deployment

#### Performance impact
- Decay worker scans all CO_RETRIEVED edges once per week (O(E) where E = edge count)
- Neo4j indexed traversal: ~1M edges scanned in <5 seconds on production hardware
- `pow()` function and `duration.between()` are built-in Neo4j operations (efficient)
- Deletion of 10,000-15,000 edges per week (at steady state year 1) is negligible for Neo4j
- **Impact**: <1% CPU increase on weekly schedule

#### Monitoring
- Add to health dashboard (S7 signal): `edges_decayed_this_week`, `edges_deleted_this_week`
- Alert if deletion volume > 50,000/week (indicates faster-than-expected edge churn, possible decay formula miscalibration)
- Baseline at year 1: expect 40,000-60,000 edges deleted per month once bootstrap edges reach threshold

#### Audit trail
- Log each decay worker run to `audit_log`: `action="decay_pass"`, `edges_decayed`, `edges_deleted`, `timestamp`
- Q168's retention worker will eventually archive these audit records after 18 months
