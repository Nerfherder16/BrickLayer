## Critical Pre-Condition: feedback_summary Must Come First

`get_feedback_counts_by_memory()` (line 867) does a full-table scan of all `audit_log` rows with `action = 'feedback'` — no time filter, no LIMIT:

```sql
SELECT memory_id,
       count(*) FILTER (WHERE details->>'useful' = 'true') AS positive,
       count(*) FILTER (WHERE details->>'useful' = 'false') AS negative
FROM audit_log
WHERE action = 'feedback'
GROUP BY memory_id
```

There are at least 6 other queries that also scan `WHERE action = 'feedback'` on `audit_log` (lines 668, 714, 738, 768, 807, 877). **If feedback rows older than 90 days are deleted without pre-aggregation, those queries will return wrong counts permanently.** A `feedback_summary` materialized table is mandatory before deleting.

### Step 2: Worker — `src/workers/audit_log_retention.py`

Full file, modeled exactly on `gc.py` / `hygiene.py` conventions:

```python
"""
Audit log retention worker — bounds audit_log at ~200,000 rows.

Runs daily at 6:15am (staggered after GC at 5:00am).

Protocol (order is mandatory):
  1. Pre-aggregate ALL feedback rows into feedback_summary (upsert, no time filter).
     This preserves get_feedback_counts_by_memory() correctness after old rows are deleted.
  2. DELETE from audit_log WHERE timestamp < now() - 90 days.
  3. DELETE from injection_log WHERE timestamp < now() - 90 days.
  4. DELETE from metrics_snapshot WHERE timestamp < now() - 90 days.

At 600 rows/day ingest, 90-day window = 54,000 steady-state rows — well below 200,000.
First run will delete ~220,000 rows from current 400,377-row table (200k surplus).
Batched DELETE used on first run to avoid lock contention; batch_size=10,000.
"""

from datetime import UTC, datetime, timedelta

import structlog

from src.storage import get_postgres_store

logger = structlog.get_logger()

RETENTION_DAYS = 90
DELETE_BATCH_SIZE = 10_000


async def run_audit_log_retention(ctx: dict) -> dict:
    """Delete audit_log rows older than 90 days, after aggregating feedback."""
    logger.info("audit_log_retention_starting")

    pg = await get_postgres_store()
    cutoff = datetime.now(UTC) - timedelta(days=RETENTION_DAYS)

    stats: dict[str, int] = {}

    # ------------------------------------------------------------------
    # Step 1: Pre-aggregate feedback rows into feedback_summary.
    # Must run BEFORE deleting any rows so counts stay correct.
    # Upsert: add incremental counts for rows outside the retention window.
    # ------------------------------------------------------------------
    try:
        agg_result = await pg.pool.fetchrow(
            """
            INSERT INTO feedback_summary (memory_id, positive, negative, last_synced)
            SELECT
                memory_id,
                COUNT(*) FILTER (WHERE details->>'useful' = 'true'),
                COUNT(*) FILTER (WHERE details->>'useful' = 'false'),
                now()
            FROM audit_log
            WHERE action = 'feedback'
              AND timestamp < $1
            GROUP BY memory_id
            ON CONFLICT (memory_id) DO UPDATE
                SET positive    = feedback_summary.positive + EXCLUDED.positive,
                    negative    = feedback_summary.negative + EXCLUDED.negative,
                    last_synced = now()
            """,
            cutoff,
        )
        logger.info("audit_log_retention_feedback_aggregated")
    except Exception as e:
        logger.error("audit_log_retention_feedback_agg_failed", error=str(e))
        # Do not proceed — deleting without aggregation corrupts feedback counts.
        return {"error": str(e), "aborted": True}

    # ------------------------------------------------------------------
    # Step 2: Batched DELETE from audit_log older than cutoff.
    # Batching prevents single long-held lock on first run (~220k rows).
    # ------------------------------------------------------------------
    audit_deleted = 0
    try:
        while True:
            result = await pg.pool.fetchrow(
                """
                WITH batch AS (
                    SELECT id FROM audit_log
                    WHERE timestamp < $1
                    LIMIT $2
                )
                DELETE FROM audit_log
                WHERE id IN (SELECT id FROM batch)
                RETURNING count(*) AS n
                """,
                cutoff,
                DELETE_BATCH_SIZE,
            )
            n = result["n"] if result else 0
            audit_deleted += n
            if n < DELETE_BATCH_SIZE:
                break
        stats["audit_log_deleted"] = audit_deleted
        logger.info("audit_log_retention_audit_log_done", deleted=audit_deleted)
    except Exception as e:
        logger.error("audit_log_retention_delete_failed", error=str(e))
        stats["audit_log_error"] = str(e)

    # ------------------------------------------------------------------
    # Step 3: injection_log — same 90-day cutoff, single DELETE (smaller table).
    # ------------------------------------------------------------------
    try:
        result = await pg.pool.fetchrow(
            "DELETE FROM injection_log WHERE timestamp < $1 RETURNING count(*) AS n",
            cutoff,
        )
        n = result["n"] if result else 0
        stats["injection_log_deleted"] = n
        logger.info("audit_log_retention_injection_log_done", deleted=n)
    except Exception as e:
        logger.error("audit_log_retention_injection_log_failed", error=str(e))
        stats["injection_log_error"] = str(e)

    # ------------------------------------------------------------------
    # Step 4: metrics_snapshot — same 90-day cutoff.
    # ------------------------------------------------------------------
    try:
        result = await pg.pool.fetchrow(
            "DELETE FROM metrics_snapshot WHERE timestamp < $1 RETURNING count(*) AS n",
            cutoff,
        )
        n = result["n"] if result else 0
        stats["metrics_snapshot_deleted"] = n
        logger.info("audit_log_retention_metrics_snapshot_done", deleted=n)
    except Exception as e:
        logger.error("audit_log_retention_metrics_snapshot_failed", error=str(e))
        stats["metrics_snapshot_error"] = str(e)

    # ------------------------------------------------------------------
    # Audit the retention run itself.
    # ------------------------------------------------------------------
    try:
        await pg.log_audit(
            action="audit_log_retention_run",
            memory_id="system",
            actor="audit_log_retention",
            details=stats,
        )
    except Exception as e:
        logger.warning("audit_log_retention_self_audit_failed", error=str(e))

    logger.info("audit_log_retention_complete", **stats)
    return stats
```

## First-Run Risk: ~220,000-Row DELETE

At current table size of 400,377 rows, a single-transaction DELETE would:
- Hold an exclusive lock on the table for 30–60 seconds (estimated)
- Block concurrent `log_audit()` writes from GC, hygiene, and all other workers
- Risk statement timeout if `job_timeout = timedelta(minutes=30)` in `WorkerSettings` is hit

**Mitigation already included**: the worker uses a batched CTE loop with `LIMIT 10,000`. Each batch acquires and releases the lock independently. At 220,000 surplus rows / 10,000 per batch = 22 iterations, each taking ~50ms = ~1.1 seconds total. No operational risk.

The `RETURNING count(*) AS n` pattern on a CTE DELETE requires PostgreSQL 14+. Confirm version with `SELECT version();` if uncertain — the alternative is `GET DIAGNOSTICS n = ROW_COUNT` inside a PL/pgSQL block, which the migration-style constant pattern already uses.

## Key Findings

1. **`feedback_summary` is mandatory before DELETE** — `get_feedback_counts_by_memory()` (line 867) and 6 other queries do unbounded full-table scans on `WHERE action = 'feedback'` with no time window. Deleting without pre-aggregating produces permanently wrong feedback counts fed into the reranker trainer.

2. **Exact DELETE predicate**: `WHERE timestamp < now() - INTERVAL '90 days'` — column name is `timestamp` (TIMESTAMPTZ), confirmed at line 38 of postgres_store.py. No alias or alternative name exists.

3. **BRIN replaces B-tree on `idx_audit_log_timestamp`** — the existing B-tree index (line 47) should be dropped and replaced with `USING BRIN (timestamp)`. For a monotonic-append table with 400k+ rows, BRIN is appropriate and dramatically smaller. The migration handles this idempotently via `DROP INDEX IF EXISTS` + `CREATE INDEX IF NOT EXISTS`.

4. **Batched DELETE is required for first run** — 220,000 surplus rows in a single transaction would lock the table for 30–60s. The 10,000-row CTE batch loop completes in ~22 iterations (~1 second total) with no operational impact.

5. **ARQ cron slot**: 6:15am is the only open daily slot in `main.py`. Existing slots: 3:00, 3:15, 3:30, 4:00, 4:30, 5:00, 5:30. GC is at 5:00am. 6:15am provides 75-minute clearance after GC.

6. **No `get_positive_feedback_counts` function exists** — the actual function name is `get_feedback_counts_by_memory()`. The `feedback_summary` upsert key must be `memory_id`, not `user_id`. Upsert is an INSERT ... ON CONFLICT DO UPDATE to handle incremental runs correctly.

7. **Steady-state math**: 600 rows/day × 90 days = 54,000 rows ceiling — 3.7x safety margin under the 200,000 target. Even at 3x growth (1,800 rows/day), the ceiling stays under 200,000.
