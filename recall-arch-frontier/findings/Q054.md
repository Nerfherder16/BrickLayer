# Finding: Q054 — Intersection Failure: asyncio Phase Gap, Tombstone Window, and distinct_sessions_accessed Mis-count

**Question**: When the two-phase asyncio session-end pipeline (Q051), the tombstoned registry entry (Q049), and the distinct-sessions-admission gate (Q044) are deployed together, what new failure mode emerges at their intersection — specifically, can a tombstoned registry entry cause distinct_sessions_accessed to mis-count and incorrectly promote a burst-accessed memory to LLPC?
**Question Type**: ADVERSARIAL
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Distributed systems — linearizability, multi-version concurrency control, epoch-based reclamation, snapshot isolation anomalies

---

## The Failure Scenario Under Analysis

The three mechanisms interact across a single session-end hook invocation:

1. **Op1** (Phase 1, asyncio): Deletes the PostgreSQL row for memory UUID-A (low-importance session memory) and removes its vector from Qdrant. Returns `deleted_ids = [UUID-A]`.

2. **Op4** (Phase 2, asyncio): Reads `deleted_ids` from Op1's output. Transitions the scope registry entry for `casaos-network-interface@casaclaude` from `{uuid: UUID-A, status: LIVE}` to `{uuid: null, status: TOMBSTONED, prior_importance: 0.35, content_hash: H}`.

3. **Q044 session-end Op2 extension**: Also in Phase 2, runs `distinct_sessions_accessed += 1` for all memory IDs accessed during the session.

The question is whether a crash or exception between Phase 1 and Phase 2, or a race condition within Phase 2, can leave the registry in a state where `distinct_sessions_accessed` is incremented for a UUID that has already been deleted from PostgreSQL — and whether this can trigger incorrect LLPC promotion.

There is a second, subtler question embedded here: **can a LIVE registry entry (tombstone not yet applied) be read by the LLPC admission query during the Phase 1→Phase 2 gap, causing a memory that no longer exists to influence LLPC routing?**

## Attacker Thread

### The asyncio phase gap recreates the dangling pointer problem Q049 was designed to solve

**Core claim**: The asyncio two-phase pipeline introduces an observable window — between Op1 completing and Op4 executing — during which the scope registry holds a LIVE entry pointing to a UUID that no longer exists in PostgreSQL or Qdrant. Q049's tombstone mechanism only runs in Phase 2. Anything that reads the registry during Phase 1→Phase 2 gap sees a dangling pointer indistinguishable from a healthy LIVE entry.

**Evidence and mechanism**:

The Q051 implementation is explicit: `asyncio.gather()` fires Op1, Op3, Op5, Op6 concurrently. Phase 1 is considered complete when `asyncio.gather()` returns. Op4 fires after. This is not a single atomic transaction — it is two sequential asyncio barrier points separated by Python event loop scheduling overhead, network round-trips, and any async I/O latency in Phase 2's preamble.

The window is not just theoretical. Consider the failure modes:

**Failure Mode A: Process crash between phases.**
The session-end hook is a JavaScript process that calls a FastAPI backend. If the process is killed (OOM, SIGKILL, Claude Code restart) after Phase 1's `asyncio.gather()` returns but before Phase 2 begins, Op4 never executes. UUID-A is gone from PostgreSQL, but the registry entry remains `{uuid: UUID-A, status: LIVE}`. This is the exact dangling pointer Q049 identified as a critical defect. Q049 proposed tombstoning as the fix — but Q051's two-phase asyncio design defers tombstoning to Phase 2, preserving the crash window.

**Failure Mode B: Op4 throws an exception within Phase 2.**
Q051's Phase 2 runs `asyncio.gather(op4_scope_registry_cleanup(deleted_ids), op2_working_set_counter_update(...), return_exceptions=True)`. If Op4 raises an exception (Redis connection lost, pipeline error), `return_exceptions=True` captures it but does not retry or alert. The tombstone transition silently fails. UUID-A is deleted. Registry entry remains LIVE and dangling.

**Failure Mode C: distinct_sessions_accessed incremented for a deleted UUID.**
The Q044 session-end hook extension increments `distinct_sessions_accessed` for all memory IDs accessed during the session. Op2 (working-set update) runs in Phase 2 in parallel with Op4. Consider this ordering within Phase 2:

```
Phase 2 fires:
  Op4: (starting) — reads UUID-A from reverse index, issues HDEL via pipeline
  Op2: (starting) — executes UPDATE memories SET distinct_sessions_accessed += 1
                     WHERE id = ANY($1::uuid[]) (the accessed IDs list includes UUID-A
                     because it was accessed earlier in the session)
  Op4: (Redis pipeline executes) — HDEL scope_registry casaos-network-interface@casaclaude
  Op2: (PostgreSQL UPDATE returns 0 rows affected) — UUID-A is already deleted by Op1
```

This is the key mis-count question. Op2's UPDATE attempts to increment `distinct_sessions_accessed` for UUID-A — a memory that Op1 already deleted in Phase 1. PostgreSQL silently returns 0 rows updated (UPDATE on a non-existent row is a no-op, not an error). The increment never happens. But this means: `distinct_sessions_accessed` is NOT mis-incremented. **The ghost access is silently dropped.** This is a different failure mode than incorrect LLPC promotion — it is incorrect distinct-session accounting via under-counting on the deletion path.

**Failure Mode D: The LLPC admission query fires during the Phase 1→Phase 2 gap.**
The LLPC routing query (`WHERE distinct_sessions_accessed >= 3 AND sessions_since_last_access <= 10`) reads the PostgreSQL `memories` table. Op1 deleted UUID-A from PostgreSQL in Phase 1. If a new session begins during the Phase 1→Phase 2 gap (unlikely given wall-clock: ~200ms, but possible under concurrent session load), the LLPC routing query sees the registry entry as LIVE but cannot return UUID-A as a candidate (it no longer exists in PostgreSQL). The registry entry is invisible to the PostgreSQL-based LLPC query because the LLPC query doesn't read the Redis registry — it reads PostgreSQL directly. This failure mode is therefore **inert for LLPC promotion**: the LLPC routing query is unaffected by the dangling registry entry because the two systems are not joined in the admission path.

**Failure Mode E: Re-encounter scoring during the dangling window.**
If a new memory about `casaos-network-interface@casaclaude` arrives during the Phase 1→Phase 2 gap (Op1 complete, Op4 pending), the Phase 1 scope discriminant (Q026) reads the registry and finds `{uuid: UUID-A, status: LIVE}`. It attempts to resolve UUID-A in PostgreSQL — and gets a 404. The scope discriminant falls through to Phase 2 (cosine comparison). No re-encounter boost is applied because the tombstone does not yet exist. The new memory is stored as a fresh first occurrence. The re-encounter signal is permanently lost — the tombstone will be applied by Op4 after Phase 2, but by then the new memory has already been stored and the Phase 1 path has already been traversed.

**Failure mode summary**: The asyncio phase gap produces three distinct failure modes, not one:
1. Process-crash dangling pointer (Q049's original problem, not fully solved)
2. Silent drop of `distinct_sessions_accessed` increment for accessed memories deleted in the same hook invocation
3. Lost re-encounter boost for new memories arriving during the Phase 1→Phase 2 window (rare but non-zero probability under concurrent sessions)

None of these cause LLPC over-promotion (the original question's concern). The actual failure modes are subtler: LLPC under-promotion (the accessed memory that is deleted can't earn its increment) and lost tombstone signal (the new memory doesn't get its re-encounter boost).

**Failure mode**: The two-phase design is not wrong about correctness for the common case, but it is wrong about crash safety. A crash between Phase 1 and Phase 2 produces a dangling LIVE registry entry that Q049's tombstone was specifically designed to prevent. The tombstone arrives too late because it lives in Phase 2, while the deletion occurs in Phase 1.

## Defender Thread

### The two-phase design is safe because the crash window is no worse than sequential execution, and the distinct_sessions_accessed gate is unaffected by the phase gap

**Core claim**: The asyncio two-phase design does not introduce a new failure mode compared to a synchronous sequential implementation — the crash window between Op1 and Op4 exists in any implementation where deletion and registry cleanup are not wrapped in a distributed transaction. The distinct_sessions_accessed gate is structurally immune to the phase gap because it reads PostgreSQL, not the Redis registry.

**Evidence and mechanism**:

**Argument 1: Sequential execution has the same crash window.**
Consider a naive synchronous implementation:

```python
# Sequential (no asyncio):
deleted_ids = await op1_window_drop()          # Step 1: delete memories
await op4_registry_cleanup(deleted_ids)        # Step 2: tombstone registry
await op2_working_set_update(accessed_ids)     # Step 3: update counters
```

A crash after Step 1 but before Step 2 produces identical state to a crash in the asyncio design between Phase 1 and Phase 2. The asyncio design does not widen the crash window — it parallelizes Op3, Op5, and Op6 alongside Op1, but the Op1→Op4 sequential dependency is preserved. Phase 2 fires only after Op1 completes. The dangling pointer window is the time between `pg.execute(DELETE)` and the first Redis HDEL in Op4. This window is the same in both sequential and asyncio implementations.

The Attacker's claim that asyncio "introduces" the dangling pointer window is false. The window exists in any implementation that does not use distributed transactions. Q049 identified this as a known limitation and proposed tombstoning as the fix. The asyncio design does not make the window larger; it merely makes it more visible because the phases are explicitly named.

**Argument 2: The LLPC admission gate (Q044) is immune to the registry state.**
The LLPC routing query in Q044's architecture reads PostgreSQL's `memories` table directly:

```sql
SELECT id, content, importance
FROM memories
WHERE distinct_sessions_accessed >= 3
  AND sessions_since_last_access <= 10
ORDER BY importance DESC
LIMIT 2
```

This query does not join with the Redis scope registry. A dangling registry entry — LIVE or TOMBSTONED — has no effect on which memories are returned by the LLPC routing query. The admission gate is defined by PostgreSQL columns (`distinct_sessions_accessed`, `sessions_since_last_access`). The Redis registry is only consulted during the write path (Phase 1 scope discriminant for incoming memories). The read path (LLPC routing) is entirely PostgreSQL-based. Therefore, the phase gap produces zero LLPC mis-promotion.

This is the most important claim: **the specific concern raised in Q054's question — "can a tombstoned registry entry cause distinct_sessions_accessed to mis-count and incorrectly promote a burst-accessed memory to LLPC?" — is architecturally impossible because the LLPC admission pathway and the registry pathway are disjoint in their data sources.**

**Argument 3: Op2's UPDATE on a deleted UUID is a safe no-op.**
When Op1 deletes UUID-A in Phase 1, and Op2 attempts `UPDATE memories SET distinct_sessions_accessed += 1 WHERE id = ANY($1::uuid[])` in Phase 2 with UUID-A in the array, PostgreSQL returns 0 rows updated. This is not a corruption event — it is a safe no-op. The deleted memory simply doesn't receive a counter increment because it no longer exists. There is no phantom row created, no counter overflow, no incorrect LLPC promotion. The only consequence is that UUID-A's final `distinct_sessions_accessed` value (at the moment of deletion) was lower than it would have been if Op2 had run before Op1. This matters only if UUID-A should have been preserved rather than deleted — but Op1 only deletes memories below the importance threshold, and session-scoped low-importance memories that are deleted should not be candidates for LLPC promotion regardless.

**Argument 4: The tombstone transition timing is not critical for the re-encounter use case.**
Q049's tombstone mechanism is designed to preserve re-encounter signals across sessions. The primary concern is that a deleted memory's topic key vanishes from the registry, causing subsequent sessions to treat re-encounters as first occurrences. The asyncio Phase 1→Phase 2 gap is measured in milliseconds. The re-encounter scenario Q049 is worried about occurs sessions later (days or weeks). A millisecond window where the registry entry is LIVE pointing to a deleted UUID has no practical impact on the re-encounter signal, because no new session starts within milliseconds of the session-end hook completing. Even in the crash scenario (Phase 2 never executes), the dangling LIVE entry is only visible during subsequent writes. The Phase 1 scope discriminant for an incoming memory will attempt to resolve UUID-A in PostgreSQL, get a 404, and fall through to Phase 2 (cosine). The re-encounter boost is lost — but this is the same failure mode Q049 identified for the synchronous cascade-delete, not a new failure introduced by asyncio.

**Best case scenario**: The two-phase asyncio design is deployed. Op1 runs in ~15-40ms. Phase 2 fires immediately after. The registry tombstone transition (Op4) completes within 5-20ms after Phase 2 begins. Total dangling pointer window: ~20-60ms. During this 20-60ms, no new session starts (Tim's sessions are not sub-minute). The tombstone is in place before the session-end hook is considered complete. LLPC routing in the next session sees only PostgreSQL data, which was correctly cleaned up by Op1. No LLPC mis-promotion. No `distinct_sessions_accessed` corruption. The system works correctly.

## Synthesis

### What Thread A assumes that Thread B exposes

Thread A assumes that the LLPC admission gate and the Redis registry are architecturally disjoint — that a dangling registry entry cannot affect LLPC routing because LLPC reads PostgreSQL, not Redis. This assumption is correct for the current design, but it is an assumption about implementation consistency that Thread B exposes as invisible to the architecture itself.

Thread B surfaces three real failure modes that Thread A acknowledges but minimizes:

1. **The crash-between-phases dangling pointer**: Thread A correctly notes the crash window is identical in sequential and asyncio designs, but Thread B correctly notes that Q049's proposed tombstone fix does not eliminate this window — it merely changes the tombstone's arrival timing. In a crash, no tombstone arrives at all.

2. **The silent drop of distinct_sessions_accessed for deleted memories**: Thread A dismisses this as a safe no-op (Op2 updates a non-existent row, 0 rows affected). Thread B identifies this as an under-counting event: a memory that was accessed (and thus earned a session contribution toward LLPC promotion) during the same session that deleted it cannot log that final session access. For a memory that had `distinct_sessions_accessed = 2` and was accessed in the session that deleted it, the final value is 2 rather than 3 — it never reaches the LLPC threshold. This under-counting is correct behavior (the memory is being deleted), but it is invisible to the system.

3. **The re-encounter boost timing gap**: Thread A correctly notes the re-encounter scenario occurs across sessions (days apart), making a millisecond window irrelevant. But Thread B correctly notes the crash scenario: if the session-end hook crashes after Phase 1, no tombstone is ever written, and the next session's Phase 1 scope discriminant encounters a LIVE entry pointing to a deleted UUID. The re-encounter boost is permanently lost, identical to the Q049 cascade-delete failure.

**The hidden assumption**: Thread A assumes that Phase 2 always executes when Phase 1 succeeds. This is true under normal operation (asyncio guarantees sequential phase execution). But it is false under process crash, and the Op1→Op4 sequencing means that the most critical step (tombstone transition, which is the fix for Q049's critical defect) is placed in Phase 2, making it crash-unsafe by design.

**The key finding neither thread names explicitly**: The question asks whether a tombstoned registry entry can cause `distinct_sessions_accessed` to mis-count and incorrectly promote a burst-accessed memory to LLPC. The answer is **no — but for a reason that reveals a deeper structural issue**.

The LLPC promotion mis-count cannot happen because:
- The LLPC admission query reads PostgreSQL, not Redis
- A memory that is deleted by Op1 no longer exists in PostgreSQL and cannot be promoted by the LLPC routing query
- A tombstoned registry entry has no UUID and makes no contribution to LLPC routing

But this architectural separation is itself the structural insight: the registry (Redis) and the admission counter (PostgreSQL) are two separate state machines that must both be correct for the system to behave correctly. The asyncio design correctly sequences their updates (Op1 → Op4). But **Phase 2 atomicity is not guaranteed** — Op4 and Op2 run concurrently within Phase 2, and either can fail independently. If Op4 fails (Redis error) and Op2 succeeds, the registry is dangling and the counter is updated. If Op2 fails (PostgreSQL error) and Op4 succeeds, the tombstone is in place but the working-set counter is stale.

The synthesis reveals: **the two-phase asyncio design correctly identifies Op1→Op4 as the hard dependency, but it does not address the within-Phase-2 partial failure scenario, and it does not address the Phase-1-to-Phase-2 crash window that leaves the tombstone unapplied.**

### When Thread A is right

- Normal operation: Phase 2 always fires after Phase 1. Op4 completes within 5-20ms. The dangling pointer window is sub-100ms and falls between sessions. No LLPC mis-promotion because the LLPC routing query is PostgreSQL-only.
- The specific question posed (can tombstone cause LLPC mis-count): definitively no. The registry and LLPC admission are in separate data stores with no join in the read path.
- Within-session correctness: Op2's no-op on deleted UUIDs is safe. No counter corruption.

### When Thread B is right

- Crash between Phase 1 and Phase 2: tombstone never written, dangling LIVE entry persists indefinitely, re-encounter boost lost on next access. Identical to Q049's original critical defect despite tombstone proposal.
- Op4 exception within Phase 2: tombstone write fails silently (return_exceptions=True captures but does not retry). System appears healthy but registry has dangling LIVE entry.
- The two-phase design places the fix (tombstone) after the deletion (Op1), not before or atomic with it — violating the metadata-first delete ordering principle identified in Q049's Idea: `metadata-first-delete-ordering-for-multi-system-gc`.

### The architecture Thread B would accept

**Pre-Phase-1 tombstone staging: write the tombstone before the deletion, not after.**

The Q049 synthesis already identified the correct fix in its "metadata-first-delete-ordering" idea: tombstone the registry entry first (Redis, fast), then delete from PostgreSQL and Qdrant (slower, can fail). This maps directly to a change in the asyncio execution plan:

**Current design (problematic):**
```
Phase 1: Op1 (DELETE PostgreSQL + Qdrant) [returns deleted_ids]
Phase 2: Op4 (TOMBSTONE registry using deleted_ids)
```

**Fixed design (metadata-first):**
```
Phase 0 (new): SELECT IDs to be deleted (read-only query, produces candidate_ids)
               TOMBSTONE registry for candidate_ids (Redis HSET, fast, before any deletion)
Phase 1:       Op1 (DELETE PostgreSQL + Qdrant using candidate_ids) [no return needed]
               Op3, Op5, Op6 (unchanged, parallel)
Phase 2:       Op2 (working-set counter update, unchanged)
               [Op4 is eliminated — tombstone was applied in Phase 0]
```

This design:
1. Eliminates the crash window: if the process crashes after Phase 0 but before Op1, the tombstone is written but the memory still exists in PostgreSQL. The tombstone is stale (points to a live memory) but the Phase 1 scope discriminant resolves UUID-A correctly (it still exists). The system is in a slightly inconsistent state (TOMBSTONED entry with a valid UUID) but not in a broken state. The memory survives until the next session's window-drop attempts to delete it again — at which point Phase 0 writes a new tombstone (idempotent) and Phase 1 deletes it successfully.

2. Preserves the re-encounter boost: if a new memory arrives for `casaos-network-interface@casaclaude` after Phase 0 but before Op1, the Phase 1 scope discriminant finds a TOMBSTONED entry. It applies the re-encounter boost and stores the new memory. Then Op1 fires and finds no memory matching the filter (the new memory has higher importance than the threshold, or the original UUID-A is now superseded). The re-encounter signal is preserved.

3. Reduces Op4 to a no-op: since tombstoning happens in Phase 0 before deletion, Op4 can be removed from Phase 2, simplifying the dependency graph. The `deleted_ids` return from Op1 is no longer needed by Op4, removing the only hard sequencing constraint in Q051's design.

**Revised asyncio execution plan:**

```python
async def session_end_hook_executor(session_id, accessed_ids, newly_archived_ids, summary_text):

    # ─── Phase 0: Identify candidates and tombstone registry (metadata-first) ───
    # Read-only: identify which memories will be dropped
    candidate_ids = await identify_window_drop_candidates()
    # Write tombstones BEFORE deleting the memories
    await op0_tombstone_registry_entries(candidate_ids)

    # ─── Phase 1: All independent operations in parallel ────────────────────────
    # Op1 now uses pre-computed candidate_ids — no return value needed
    await asyncio.gather(
        op1_session_scope_window_drop(candidate_ids),  # uses pre-computed IDs
        op3_cohesion_cache_refresh(newly_archived_ids),
        op5_l0_dedup_flush(session_id),
        op6_session_summary_store(summary_text),
        return_exceptions=True,
    )

    # ─── Phase 2: Working-set update only (Op4 eliminated) ──────────────────────
    await op2_working_set_counter_update(session_id, accessed_ids)
```

**Crash safety analysis of the revised design:**
- Crash after Phase 0, before Phase 1: tombstones written, memories still alive. Stale tombstones pointing to live UUIDs. System can resolve the UUID — no dangling pointer. Next session's window-drop re-identifies and re-tombstones (idempotent HSET).
- Crash during Phase 1 (after partial deletes): some memories deleted, some not. Tombstones are written for all candidates. Some tombstones may point to memories that were not yet deleted (in-flight partial failure). A background reconciliation job (Q049's suggestion) can clean these up. The key property: **no tombstone will point to a deleted UUID — the tombstone is always written before the deletion.**
- Crash after Phase 1, before Phase 2: Op2 (working-set update) did not run. Working-set counters are stale for this session. Op2's idempotency guard (Q051) can detect this on the next re-fire.

**Op0 implementation:**

```python
async def op0_tombstone_registry_entries(candidate_ids: list[str]) -> None:
    """
    Phase 0: Tombstone scope registry entries for memories about to be deleted.
    Runs BEFORE Op1 (metadata-first delete ordering, Q049).
    """
    if not candidate_ids:
        return
    # Use reverse index (recall:uuid_to_scope:{uuid}) for O(N_candidates) lookup
    pipe = redis_client.pipeline()
    for uid in candidate_ids:
        pipe.get(f"recall:uuid_to_scope:{uid}")
    scope_keys = await pipe.execute()

    tombstone_pipe = redis_client.pipeline()
    for uid, scope_key in zip(candidate_ids, scope_keys):
        if scope_key:
            # Transition from LIVE to PRE_DELETE tombstone
            tombstone_pipe.hset(
                "scope_registry",
                scope_key,
                json.dumps({
                    "uuid": None,
                    "status": "TOMBSTONED",
                    "prior_uuid": uid,           # retained for reconciliation
                    "tombstone_phase": "pre_delete",
                    "tombstoned_at": current_session_id,
                    "eviction_reason": "importance_below_threshold",
                })
            )
    await tombstone_pipe.execute()
```

**What this fixes vs. the Q051 design:**
- Eliminates the Phase 1→Phase 2 crash window for tombstone writes
- Removes Op4 from Phase 2 (eliminates the only hard sequencing dependency in Q051)
- Enables the revised Phase 1 to run Op1+Op3+Op5+Op6 concurrently with no Phase 2 dependency on Op1's output
- Preserves re-encounter signals even in crash scenarios
- Adds Phase 0 latency: one PostgreSQL SELECT (same as Op1's current SELECT-then-DELETE, already present) + N_candidates Redis HSET calls via pipeline. At N_candidates=10: ~5ms. Net latency impact: negligible.

### Genuine uncertainty

The Phase 0 tombstone approach adds a SELECT before deletion. Op1's current implementation already does a SELECT (to get IDs before the DELETE). The Phase 0 approach can reuse this SELECT. But there is a race: between the Phase 0 SELECT (identifying candidates) and Op1's DELETE, another process could access UUID-A (updating `sessions_since_last_access = 0` and rescuing it from window-drop). The Phase 0 tombstone would then be written for a memory that Op1's re-evaluated DELETE does not actually delete (because its importance was boosted by the access, or its sessions_since_last_access reset). The tombstone would be stale pointing to a live memory.

This race is extremely unlikely in Tim's single-user system but is not zero. The fix: Op1's DELETE should use the candidate_ids from Phase 0 (a point-in-time snapshot) rather than re-evaluating the WHERE clause. If Op1 deletes using the exact IDs computed in Phase 0, the race is closed.

## Ideas Extracted

### Idea: phase-zero-metadata-first-tombstone

- **Novelty**: 0.78
- **Evidence**: 0.82
- **Feasibility**: 0.85
- **Description**: Insert a Phase 0 step before the asyncio Phase 1 execution in the session-end hook. Phase 0 performs the SELECT to identify window-drop candidates and immediately writes TOMBSTONED registry entries for all candidates via a Redis pipeline — before any deletion occurs. Op1 then uses the pre-computed candidate IDs list for its DELETE, rather than re-evaluating the WHERE clause. This enforces metadata-first delete ordering (Q049's Idea: metadata-first-delete-ordering-for-multi-system-gc) at the asyncio architecture level. Crash between Phase 0 and Phase 1 produces stale tombstones pointing to live memories — safe because UUID resolves correctly. Crash during Phase 1 produces tombstones for some deleted memories and some live ones — a background reconciliation job can clean these. No crash produces tombstones pointing to deleted UUIDs with no tombstone coverage. Phase 0 adds ~5ms at N_candidates=10. Net effect: Op4 is eliminated from Phase 2, removing the only hard Op1→Op4 sequencing constraint from Q051. Phase 1 and Phase 2 become structurally simpler.

### Idea: candidate-snapshot-isolation-for-window-drop

- **Novelty**: 0.73
- **Evidence**: 0.76
- **Feasibility**: 0.88
- **Description**: Instead of Op1 re-evaluating `WHERE scope='session' AND importance < threshold AND sessions_since_creation > 0`, Op1 should accept the pre-computed `candidate_ids` list from Phase 0 and issue `DELETE WHERE id = ANY($1::uuid[])`. This snapshot-isolation pattern closes the race between Phase 0 identification and Op1 deletion: a memory that is accessed between Phase 0 and Op1 (which would reset `sessions_since_last_access = 0` and potentially change its window-drop eligibility) is still deleted using the Phase 0 snapshot. The snapshot is the "committed view" of which memories should be dropped; subsequent accesses during the hook window are not retroactively applied. This is analogous to MVCC snapshot isolation in PostgreSQL: the transaction sees a consistent view of the data at transaction start, not a mixture of before and after concurrent modifications. The tombstone in Phase 0 already committed the "this memory is being deleted" decision. Op1's DELETE finalizes it.

### Idea: phase-two-partial-failure-audit-log

- **Novelty**: 0.67
- **Evidence**: 0.71
- **Feasibility**: 0.92
- **Description**: When `return_exceptions=True` in asyncio.gather() captures an exception from Op4 or Op2, write a structured audit record to a Redis list `recall:hook:partial_failures:{session_id}` containing the operation name, exception type, and the input data (deleted_ids for Op4, accessed_ids for Op2). A background reconciliation job reads this list and retries the failed operations. For Op4: re-run the tombstone transition for the listed UUIDs. For Op2: re-run the working-set counter update using the stored accessed_ids. This converts the current silent-failure behavior (exceptions captured and discarded) into a recoverable failure with explicit retry semantics. The audit record acts as a write-ahead log for Phase 2 operations, enabling crash recovery without distributed transactions. TTL on the audit record: 24 hours (same as Op2's idempotency guard).

## Key Finding

The specific question — can a tombstoned registry entry cause `distinct_sessions_accessed` to mis-count and incorrectly promote a burst-accessed memory to LLPC — has a definitive answer: **no, because the LLPC admission query reads PostgreSQL and does not consult the Redis scope registry**. The two data paths are architecturally disjoint.

However, the adversarial analysis surfaces a more important finding: the asyncio two-phase design (Q051) violates Q049's metadata-first delete ordering principle. The tombstone transition (Op4) runs in Phase 2, after the deletion (Op1) runs in Phase 1. A process crash between phases, or a Redis failure within Phase 2, leaves the registry in a LIVE-pointing-to-deleted state — the exact dangling pointer Q049's tombstone was designed to prevent.

The correct fix is not to move Op4 earlier in Phase 2 — it is to move tombstone writing to a new Phase 0, before any deletion occurs. Phase 0 writes tombstones for all candidates using the Redis pipeline (fast, ~5ms), then Phase 1 deletes using the pre-computed candidate list (snapshot isolation). This eliminates the dangling pointer window entirely, removes Op4 from Phase 2, and simplifies the asyncio dependency graph. The only new risk — stale tombstones for memories that survive between Phase 0 and Phase 1 — is safe because the UUID is still resolvable in PostgreSQL and is handled by the next session's window-drop.

**Verdict: BREAKTHROUGH** — the intersection analysis reveals that Q051's design is unsafe for Q049's tombstone mechanism because it places the metadata fix after the data deletion, violating the metadata-first ordering principle Q049 identified as critical. The fix requires a structural change to the asyncio pipeline (add Phase 0), not just a bug fix within Phase 2.
