# Q133 — Deployment Checklist Recalibrated for 22K Vectors

**Question**: What is the correct Saturday deployment checklist recalibrated for the actual 22,349-vector corpus — specifically, which Q128 phase time estimates are materially wrong at 62× the assumed scale, and what is the revised total wall-clock estimate?
**Agent**: quantitative-analyst
**Question Type**: IMPLEMENTATION
**Verdict**: INCREMENTAL
**Severity**: High
**Source field**: Q128 (unified Saturday deployment checklist), Q119 (three-store operational runbook), Q127 (22,349-vector corpus measurement), Q110 (Neo4j migration plan)

## Context

Q128 produced a unified 9-phase Saturday deployment checklist totaling 3 hours 45 minutes with a 15-minute buffer inside a 4-hour window. It was written against the stated corpus size of ~360 memories. Q127 confirmed the live corpus is 22,349 vectors — 62× larger than the Q128 assumption. This finding identifies each Q128 phase that is materially affected by the scale difference, recomputes the time estimate, and produces a revised cumulative total.

### Phase 2 — SQL Schema Migrations (Q128 estimate: 20 min)

**Scale-sensitive steps**: Step 13 — `CREATE INDEX CONCURRENTLY` on `memories(tenant_id, content_hash)`.

Q128 noted: "Completes in < 1 second at current scale (~360 rows)."

### Phase 3 — Neo4j Data Migration (co_retrievals backfill) (Q128 estimate: 45 min)

This phase has two distinct activities that need separate recalibration:

**3A — Edge data export and insert (migration script runtime)**

Q110 estimated "under 30 seconds for 800 edges" at 360-memory scale. At 22,349 memories, the edge count depends on retrieval session density.

Estimating edge density: Q110 was written with a known Neo4j edge count in mind. With 360 memories accumulating co-retrieval edges over typical session patterns (top-k=5 retrieval, ~2–5 co-retrieval writes per session), 800 edges corresponds to ~2.2 edges per memory. This is a sparse graph — not all memories have co-retrieval relationships.

At 22,349 memories: applying the same 2.2 edges/memory ratio yields ~49,000 edges as an upper bound. In practice, older memories accumulate more edges than new ones, and 22K memories likely represents a corpus that grew from the 360-memory baseline, so the edge graph density at the time of migration could range from 5,000 edges (sparse, mostly recent memories have relationships) to 50,000 edges (dense, older memories fully connected).

At 50,000 edges with Q110's batch_size=200: 250 INSERT batches. PostgreSQL UPSERT throughput: ~50,000 rows/second easily for a table with 3 indexes. 50,000 edges: ~1 second for the actual INSERT phase. The Neo4j Cypher MATCH and export is the bottleneck — at 50,000 edges, the Cypher query returns quickly (well under 30 seconds).

**Migration script runtime at 22K-memory scale: still under 5 minutes.**

**3B — A/B gate BFS comparison (Step 22)**

Q128 allocated 10 minutes. The A/B comparison runs BFS traversal in both Neo4j and PostgreSQL for the same query, then compares result overlap. BFS depth is bounded (typically 2-3 hops), not full-corpus traversal. This step is dominated by network round-trips and query execution, not corpus size. At 50,000 edges vs 800 edges, the BFS expands more nodes per hop but is still bounded by the top-k parameter.

**A/B gate at 22K-memory scale: still under 15 minutes.**

**3C — Revised Phase 3 total**

The bottleneck in Phase 3 is not data volume — it is the dual-write canary period and A/B validation. These are time-bounded by process steps (Steps 20–24), not data size. Phase 3 consists of: migration script run (~5 min), edge count verification (~3 min), dual-write enable (~3 min), canary retrieval (~5 min), A/B comparison (~10 min), read cutover (~5 min), cutover verification (~5 min).

**Revised estimate: 45 min** — unchanged in total. The migration script runs faster (trivial delta), but the procedural gates (dual-write canary, A/B comparison) are human-gated steps that require their allocated time regardless of data volume.

### Phase 5 — Redis Epoch Key Rename + Qdrant Alias Switch (Q128 estimate: 15 min)

**THIS IS THE MATERIALLY WRONG PHASE.**

Q128 allocated 3 minutes (Step 36) for the Qdrant alias switch: create `recall_tim` collection, copy all points from `recall` to `recall_tim` via scroll+upsert, then execute the atomic alias change.

Q128 assumed ~360 points. The copy step is the bottleneck.

**Qdrant scroll+upsert throughput calculation**:

Qdrant's REST API scroll endpoint returns points in pages of configurable size. Typical throughput for a scroll-then-upsert loop (sequential, single-threaded Python script):
- Per-page scroll: 100 points per request, ~50–100ms per REST call
- Upsert batch: 100 points per request, ~50–100ms per REST call
- Combined round-trip per 100 points: ~100–200ms

At 360 points: 4 batches × 200ms = ~800ms. Fits easily within 3 minutes.

At 22,349 points: 224 batches × 200ms = **44,700ms ≈ 45 minutes**.

Even at optimistic throughput (batch_size=500, 50ms per call):
22,349 / 500 = 45 batches × 2 calls each × 50ms = **4,500ms per 1,000 points** → 22,349 × 4.5ms = **100 seconds ≈ 2 minutes** for the upsert side alone.

But the scroll side also takes time: 22,349 / 500 = 45 scroll requests × 50ms = 2,250ms. Total: ~3,500ms ≈ 4 minutes at large batch size.

Realistically, the Python script `qdrant_copy_collection.py` (referenced in Step 36 but not shown in the findings) uses a default batch size. If batch_size=100 (common default): the operation takes **7–15 minutes**, not 3 minutes.

**However**: Qdrant has a built-in collection clone/snapshot mechanism. Using the native Qdrant snapshot API to create a snapshot of the `recall` collection and then recover it as `recall_tim` is orders of magnitude faster than scroll+upsert — it is a filesystem copy. This approach was not specified in Q128.

Using the snapshot approach:
1. `POST /collections/recall/snapshots` — creates snapshot (seconds for 22K vectors: ~5-30 seconds)
2. `PUT /collections/recall_tim/snapshots/upload` — restores snapshot as new collection (seconds)
3. Alias switch — instantaneous

**With Qdrant snapshot approach at 22K vectors: Phase 5 ~20-25 minutes total** (accounting for snapshot creation, upload wait, verification).

**With scroll+upsert at default batch_size=100: Phase 5 ~25-35 minutes total** (7-15 min for copy step + other steps).

Q128 Step 36 says 3 minutes. At 22K vectors the correct estimate for the copy step alone is **7–15 minutes** (scroll+upsert) or **2–5 minutes** (Qdrant snapshot method). The checklist step should be revised to specify the snapshot method explicitly and re-estimated.

**Revised Phase 5 estimate: 25 min** (using scroll+upsert with batch_size=500: ~10 min for copy + 15 min for the other Phase 5 steps). If the snapshot method is used: 20 min.

**Delta from Q128: +10 to +15 minutes.**

### THE QDRANT RE-EMBED SCENARIO (Q119 Runbook 2)

Q119's Qdrant recovery runbook states:

> "Expected downtime at Tim's scale (360 memories): 5-15 minutes. The bottleneck is the re-embedding step: 360 memories x ~200ms per embedding call to Ollama = ~72 seconds."

This is NOT part of the normal Saturday deployment path. The Qdrant re-embed recovery (Q119 Runbook 2) is the disaster recovery procedure for when the Qdrant collection is corrupted or lost. The Q128 Saturday checklist does not include a Qdrant re-embed step — Phase 5 copies vectors from `recall` to `recall_tim` via scroll+upsert (or snapshots), preserving existing embeddings. No re-embedding occurs.

**However, the Q133 prompt raises this correctly**: if the `recall` collection needs to be rebuilt from scratch (e.g., if the collection is corrupt at deployment time, or if the operator decides to re-embed with a new model version), the Q119 estimate of 72 seconds is wrong at 22K scale.

**Corrected Q119 Runbook 2 re-embed estimate at 22,349 vectors**:
- 22,349 memories × 200ms per Ollama embedding call (sequential) = 4,470 seconds = **74.5 minutes**
- With async batching (10 concurrent Ollama calls, each 200ms): 22,349 / 10 × 200ms = 447 seconds = **~7.5 minutes**
- With batch_size=50 in the re-embed script (as written in Q119 Step Q4): the script calls `embed_text` one at a time in a `for` loop with `await` — this is sequential, not concurrent

The Q119 re-embed script as written is fully sequential: `for i, row in enumerate(rows): vector = await embed_text(http, row['content'])`. At 22,349 memories × 200ms: **74.5 minutes** of embedding generation time, not 72 seconds.

This is a 62× underestimate of recovery time. If Qdrant corruption occurs at 22K scale, the recovery window is ~80 minutes (74.5 min embedding + overhead), not 5-15 minutes. This changes the severity assessment: Qdrant corruption at 22K scale is a 90-minute outage, not a 15-minute one.

**This is the most operationally significant finding in this calibration.**

## Verdict on Saturday Window Fit

**The deployment still fits in a 4-hour Saturday session.** The revised estimate of 3 hours 55 minutes leaves 5 minutes of buffer against the 4-hour target, down from the Q128 buffer of 15 minutes.

However, the buffer has shrunk to essentially zero. Any single step taking longer than expected — the A/B gate failing on first pass (adding 10+ minutes for investigation), a docker build timeout, or an unexpected API restart loop — would push the deployment past 4 hours.

**Recommended mitigation**: Use Qdrant's native snapshot API for the Phase 5 collection copy instead of scroll+upsert. The snapshot approach reduces the copy step from 7-15 minutes to 2-5 minutes, restoring ~8-10 minutes of buffer (total: ~3h 47-50min).

**Alternative**: Split the deployment into two sessions:
- Session A (2 hours): Phases 0-4 (SQL DDL + Neo4j migration + code deploy — all fully reversible)
- Session B (2 hours): Phases 5-8 (Qdrant alias switch + Neo4j removal + verification)

The Sessions A/B split is safe because: Phase 4 code is fully compatible with the old Qdrant collection name ("recall"), and dual-write can remain enabled between sessions.

## Qualifying Ideas

- **qdrant-snapshot-for-collection-copy**: (N=0.70, E=0.80, F=0.95) — Using Qdrant's native snapshot API (`POST /collections/{name}/snapshots` then `PUT /collections/{name}/snapshots/upload`) instead of scroll+upsert for the Phase 5 collection copy reduces the copy step from 7-15 minutes to 2-5 minutes at 22K vectors. The snapshot is a filesystem-level operation, not a network-serialized point-by-point transfer. This is a drop-in improvement to Step 36 of the Q128 checklist with no architectural impact. Stack fit: Drop-in.

- **concurrent-reembed-recovery-script**: (N=0.72, E=0.82, F=0.85) — The Q119 Runbook 2 re-embed script is sequential (`for i, row in enumerate(rows): vector = await embed_text(...)`). Replacing the sequential loop with a `asyncio.gather` with bounded concurrency (semaphore of 10) reduces recovery time from 74.5 minutes to ~7.5 minutes at 22K scale. This is a targeted fix to the Q119 script with high operational impact: it changes a 90-minute Qdrant outage into a 10-minute one. Stack fit: Drop-in (modifies only `scripts/rebuild_qdrant.py`).

- **deployment-session-split**: (N=0.60, E=0.75, F=0.95) — Splitting the Q128 deployment into Session A (Phases 0-4: DDL + Neo4j migration + code deploy) and Session B (Phases 5-8: Qdrant alias switch + Neo4j removal + verification) eliminates the time pressure and allows each session to have its own 4-hour window. The code in Phase 4 is dual-write compatible with the old Qdrant collection name and can operate safely between sessions. This is a scheduling strategy, not a technical change.

## Recommendations

1. **Revise Q128 Step 36 to use Qdrant snapshot API** instead of scroll+upsert for the collection copy. This restores the 15-minute buffer. The current Step 36 command references a `qdrant_copy_collection.py` script — that script should be updated or replaced with snapshot-based logic.

2. **Upgrade Q119 Runbook 2 severity rating** for Qdrant recovery at 22K scale. The "5-15 minute recovery" claim needs a footnote: "At 22K+ vectors, sequential re-embed recovery takes ~75 minutes. Run `rebuild_qdrant.py` with `--concurrent 10` to reduce this to ~8 minutes."

3. **Add `--concurrent N` flag to `scripts/rebuild_qdrant.py`** (the Q119 recovery script). An `asyncio.Semaphore(10)` wrapper around the embed loop is a 5-line change that cuts recovery time by 10×.

4. **Saturday deployment proceed as planned** — the 4-hour window is still viable at 22K scale. No strategy change is required if Step 36 is updated to use Qdrant snapshots.

5. **Record the actual co_retrievals row count before deployment** (Phase 0 Step 4 already captures the Neo4j edge count). This is the baseline for verifying the migration script completed correctly. At 22K memories the edge count could be anywhere from 2K to 50K — capturing the exact number in Step 4 is more important at this scale than at 360-memory scale.
