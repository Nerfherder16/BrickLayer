# Finding: Q068 — Pending Queue Implementation: Schema, Retry Logic, and Duplicate Resolution

**Question**: The write-tier partition policy (Q056) requires approximately 40 lines of JavaScript split across two hook files: observe-edit.js writes fail clean (no local buffer), and recall-session-summary.js failures write to a local pending queue at `~/.claude/recall-pending/session-{timestamp}.json` with retry at next session start. Three unresolved implementation questions: (1) JSON schema for pending files, (2) duplicate submission handling if the hook crashes after API success but before file deletion, (3) maximum pending file age discard threshold.
**Question Type**: IMPLEMENTATION
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Distributed systems — idempotent retry design (Kafka KIP-98 producer semantics, Stripe idempotency keys, AWS SQS at-least-once delivery); write-ahead log recovery (PostgreSQL WAL, SQLite WAL-mode); Firebase offline queue semantics

## Context: What the Production Hooks Show

Reading the production `recall-session-summary.js` and `recall-retrieve.js` reveals the exact API surface and code conventions the implementation must match:

- API endpoint for store: `POST ${RECALL_HOST}/memory/store`
- Auth header: `Authorization: Bearer ${RECALL_API_KEY}`
- POST body fields: `content`, `domain`, `source`, `memory_type`, `tags`, `importance`
- Hook environment: Node.js CommonJS (`require`), no ESM, no npm packages beyond Node builtins
- Fetch API is available natively (Node 18+)
- File system: `fs` builtins — `writeFileSync`, `readFileSync`, `mkdirSync`, `existsSync`, `unlinkSync`, `readdirSync`
- Existing `CACHE_DIR`: `~/.cache/recall/` — the pending queue must use a sibling directory to avoid collision
- The summary hook already has `mkdirSync`, `appendFileSync` available in scope
- `AbortSignal.timeout(ms)` is the standard pattern for fetch timeouts in these hooks
- The hook always exits 0 — never blocks

The pending directory must be `~/.claude/recall-pending/` (as specified in Q056) rather than `~/.cache/recall/` to separate durable pending writes from ephemeral cache artifacts.

### Field Definitions

| Field | Type | Required | Purpose |
|-------|------|----------|---------|
| `schema_version` | integer | yes | Forward compatibility; current value: 1 |
| `pending_id` | string | yes | Stable identifier for this pending file; equals the filename stem; used for logging |
| `created_at` | ISO-8601 string | yes | Wall-clock time the pending file was written; used for age-based discard |
| `expires_at` | ISO-8601 string | yes | Pre-computed expiry = `created_at` + discard threshold; checked at retry time without re-computing |
| `machine_id` | string | yes | Hostname from `os.hostname()` or `MACHINE_ID` env var; disambiguates multi-machine pending files if the pending directory is ever shared |
| `source_session_id` | string | yes | Claude Code session ID from the Stop hook input; used for logging and dedup audit trail |
| `submitted_at` | ISO-8601 string or null | yes | Null until submission succeeds; set to submission timestamp on success; checked by retry logic to skip already-submitted files |
| `recall_uuid` | string or null | yes | UUID returned by the Recall API on successful store; null until submission; stored for dedup audit trail (see Duplicate Submission section) |
| `writes` | array | yes | Ordered list of API write operations to replay; each element is one `POST /memory/store` call |
| `writes[].write_id` | string | yes | Stable identifier for this write within the file; `{pending_id}-{index}` |
| `writes[].endpoint` | string | yes | API path; currently always `/memory/store`; included for forward compatibility if other endpoints need buffering |
| `writes[].body` | object | yes | Complete POST body to replay verbatim; reconstructs the exact request without any re-computation |

### Why Write Order Is Preserved in a Single File

A single pending file per session (not one file per write) is correct for two reasons:

1. **Atomicity of the session summary**: The episodic summary and the extracted key decisions (semantic memories) from a single session are semantically coupled. Submitting the summary without the decisions, or the decisions in an arbitrary order, is worse than submitting them all together or not at all. A single file ensures all writes from a session are replayed as a unit.

2. **Simplified retry logic**: The retry loop iterates files, not individual writes. It can mark the entire file as submitted once all writes in it succeed. This avoids partial-submission state.

## Pending Queue Write Implementation

```javascript
// recall-session-summary.js — pending queue fallback
// Add these constants at the top of the file alongside existing constants

const { hostname } = require("os");

const PENDING_DIR = require("path").join(
  process.env.HOME || process.env.USERPROFILE || "/tmp",
  ".claude",
  "recall-pending"
);
const PENDING_MAX_AGE_DAYS = 7;
const MACHINE_ID = process.env.MACHINE_ID || hostname();

/**
 * Write a session's failed writes to the local pending queue.
 *
 * Called when any /memory/store attempt fails due to network error or HTTP 5xx.
 * Each session produces exactly one pending file containing all writes for that session.
 * The file is named session-{ISO-timestamp}.json using the session's created_at time.
 *
 * @param {string} sessionId - Claude Code session ID from Stop hook input
 * @param {Array<{endpoint: string, body: object}>} writes - All writes to buffer
 * @param {Function} debug - The existing debug() logger from the hook
 */
function writePendingQueue(sessionId, writes, debug) {
  if (!writes || writes.length === 0) return;

  try {
    require("fs").mkdirSync(PENDING_DIR, { recursive: true });
  } catch (err) {
    debug(`pending: failed to create pending dir: ${err.message}`);
    return;
  }

  const now = new Date().toISOString();
  const pendingId = `session-${now}`;
  const expiresAt = new Date(
    Date.now() + PENDING_MAX_AGE_DAYS * 24 * 60 * 60 * 1000
  ).toISOString();

  const pendingFile = {
    schema_version: 1,
    pending_id: pendingId,
    created_at: now,
    expires_at: expiresAt,
    machine_id: MACHINE_ID,
    source_session_id: sessionId || "",
    submitted_at: null,
    recall_uuid: null,
    writes: writes.map((w, i) => ({
      write_id: `${pendingId}-${i}`,
      endpoint: w.endpoint,
      body: w.body,
    })),
  };

  const filePath = require("path").join(PENDING_DIR, `${pendingId}.json`);
  try {
    require("fs").writeFileSync(filePath, JSON.stringify(pendingFile, null, 2));
    debug(`pending: wrote ${writes.length} writes to ${filePath}`);
  } catch (err) {
    debug(`pending: failed to write file: ${err.message}`);
  }
}
```

### Integration Point in recall-session-summary.js main()

## Retry Logic Implementation

```javascript
// recall-retrieve.js — retry pending queue on session start
// Add this function and call it at the top of main() before the search query,
// guarded by isFirstPrompt() so it only runs once per session.

const PENDING_DIR = require("path").join(
  process.env.HOME || process.env.USERPROFILE || "/tmp",
  ".claude",
  "recall-pending"
);
const PENDING_MAX_AGE_DAYS = 7;

/**
 * Retry pending session writes on session start.
 *
 * Called once per session (when isFirstPrompt() returns true).
 * Scans PENDING_DIR for .json files. For each:
 *   - Skip if already submitted (submitted_at is set).
 *   - Discard if expired (now > expires_at). Log and delete.
 *   - Otherwise attempt to submit all writes in the file.
 *   - On full success: set submitted_at, write back, then delete the file.
 *   - On partial failure: leave the file for the next session.
 *
 * The check for submitted_at handles the crash-after-submit race condition:
 * if the hook submitted successfully but crashed before deleting the file,
 * the next session sees submitted_at is set and skips resubmission.
 * No UUID comparison is needed.
 *
 * @param {object} headers - Fetch headers (auth, content-type)
 */
async function retryPendingQueue(headers) {
  const fs = require("fs");
  const path = require("path");

  let files;
  try {
    if (!fs.existsSync(PENDING_DIR)) return;
    files = fs.readdirSync(PENDING_DIR).filter((f) => f.endsWith(".json"));
  } catch {
    return;
  }

  if (files.length === 0) return;

  for (const filename of files) {
    const filePath = path.join(PENDING_DIR, filename);
    let pending;
    try {
      pending = JSON.parse(fs.readFileSync(filePath, "utf8"));
    } catch {
      // Corrupt file — delete and move on
      try { fs.unlinkSync(filePath); } catch {}
      continue;
    }

    // Skip already-submitted files (crash-after-submit safety)
    if (pending.submitted_at) {
      try { fs.unlinkSync(filePath); } catch {}
      continue;
    }

    // Discard expired files
    const expiresAt = pending.expires_at
      ? new Date(pending.expires_at).getTime()
      : new Date(pending.created_at).getTime() + PENDING_MAX_AGE_DAYS * 24 * 60 * 60 * 1000;

    if (Date.now() > expiresAt) {
      try { fs.unlinkSync(filePath); } catch {}
      continue;
    }

    // Attempt all writes in this file
    const writes = pending.writes || [];
    let allSucceeded = true;
    for (const w of writes) {
      try {
        const resp = await fetch(`${RECALL_HOST}${w.endpoint}`, {
          method: "POST",
          headers,
          body: JSON.stringify(w.body),
          signal: AbortSignal.timeout(4000),
        });
        if (!resp.ok) {
          allSucceeded = false;
          break; // Leave file for next session; don't partial-submit
        }
      } catch {
        allSucceeded = false;
        break;
      }
    }

    if (allSucceeded) {
      // Mark submitted in-file first (crash safety), then delete
      try {
        pending.submitted_at = new Date().toISOString();
        fs.writeFileSync(filePath, JSON.stringify(pending));
        fs.unlinkSync(filePath);
      } catch {
        // If we can't delete, submitted_at is set — next session will clean up
      }
    }
    // If not allSucceeded, leave the file untouched for next session
  }
}
```

### Integration Point in recall-retrieve.js main()

Insert the retry call immediately after the first-prompt check, before the search query is built. It runs fire-and-forget on session start so it does not delay memory retrieval:

```javascript
// In main(), after: const firstPrompt = isFirstPrompt(injectedFile);
if (firstPrompt) {
  // Retry any pending session writes from previous sessions
  retryPendingQueue(headers).catch(() => {});
}
```

The `catch(() => {})` ensures a failing retry never blocks retrieval. The retry is fire-and-forget — it starts before the browse query and finishes whenever the network responds, without blocking the user's prompt.

## Duplicate Submission Resolution

**Decision: Rely on `submitted_at` flag in the pending file, not on G3 deduplication and not on stored UUID.**

### The Race Condition

The race exists when the sequence is:
1. Hook submits write to API → API returns 200 with UUID
2. Hook prepares to mark `submitted_at` and delete the file
3. Hook crashes (process killed, machine suspend, hard timeout)
4. On next session start, the file still exists with `submitted_at: null`
5. Retry logic submits the same content again → duplicate entry in Recall

### Why G3 Deduplication Is Insufficient

G3 is a 72-hour TTL deduplication gate. The race condition produces a duplicate submission within the same retry session — typically seconds or minutes after the crash. G3 would catch this IF the content is byte-identical to a previously-stored memory AND the dedup key is based on content hash. But G3's design as documented in Q056 is described as a general TTL gate, not a content-hash dedup gate. Relying on G3 means relying on an implementation detail that is not verified and is subject to change.

### Why Storing the UUID Is Unnecessary

Storing the UUID returned by the API would let retry logic skip resubmission: "if recall_uuid is set, the write succeeded." This works but requires a two-phase update pattern:

1. Write file with `submitted_at: null, recall_uuid: null`
2. Submit to API
3. Update file with `submitted_at: now, recall_uuid: "uuid-from-response"`
4. Delete file

The race now moves to step 3-4: if the hook crashes between receiving the UUID and writing it back to the file, the file still has no UUID and gets resubmitted. Storing the UUID does not eliminate the race — it only shifts it.

### The Correct Solution: Two-Phase Write of submitted_at

The minimal correct solution is:

1. Submit to API
2. **Write `submitted_at: now` to the pending file** (durable record that submission succeeded)
3. Delete the pending file

If the hook crashes between step 2 and step 3, the file still exists on disk but has `submitted_at` set. The retry logic checks `submitted_at` first and skips files where it is non-null. The duplicate is prevented without requiring UUID comparison.

If the hook crashes between step 1 and step 2 (after the API returns 200 but before writing `submitted_at`), a duplicate submission occurs. This is the residual race. Its probability is:

- Time window: the write of 80 bytes to disk via `writeFileSync` (synchronous, < 1ms)
- Trigger: a hard process kill (SIGKILL, machine power loss, Claude Code timeout) in that sub-millisecond window

This is not zero but is so small that it is not worth the complexity of a two-phase commit protocol. Session summaries are episodic memories — a duplicate session summary does not corrupt the system; it creates a slightly redundant entry with slightly elevated retrieval probability. The consequence is noise, not data loss or causal violation.

**The practical dedup strategy for the residual race**: G3's 72-hour TTL gate handles the residual case adequately, because a crash-during-write retry occurs within seconds of the original submission, well within the 72-hour window. The content is byte-identical (same JSON body), so if G3 is a content-hash gate, it blocks the duplicate. If G3 is not a content-hash gate, the duplicate is accepted — a cosmetically imperfect but functionally harmless outcome.

**Final answer**: Use `submitted_at` flag in the pending file as the primary dedup mechanism. Do not store the UUID (unnecessary complexity). Rely on G3 as a secondary safety net for the residual sub-millisecond crash race. Accept that a rare duplicate session summary entry is a cosmetically imperfect but operationally harmless outcome.

## Discard Threshold

**Threshold: 7 calendar days.**

### Justification

The question asks whether the tau=20 sessions ≈ 4 calendar days argument is sufficient to set a shorter threshold.

**The tau=20 argument for a 4-day threshold**: Session summaries older than tau=20 sessions contribute nothing to LLPC (Last-Prompt-Causal) routing because the Markov chain's influence on routing decays to near-zero beyond 20 steps. Therefore, buffering summaries older than 4 days serves only to fill the vector store with stale context, not to improve retrieval quality.

This argument is correct for retrieval quality but misses the correct objective for the pending queue. The pending queue is not for retrieval quality — it is for data preservation. The question is not "will this memory improve retrieval?" but "should this memory exist in the system at all?"

A session summary is an episodic memory: it records what happened in a specific session. Even if it is beyond the LLPC routing horizon, it remains a factual record of session activity that may be relevant to:
- Timeline queries (`show me what I was working on last week`)
- Profile synthesis (the LLM synthesizes user profile facts from session summaries regardless of age)
- Debugging retrieval gaps (identifying which sessions were not summarized)

The 4-day threshold discards session summaries that are still within the user's recent memory and still within the typical project iteration cycle (most projects have debug/fix/iterate cycles of 1-2 weeks). Discarding a 5-day-old session summary that documented a critical architectural decision because "it's past tau" is the wrong trade-off.

**Why 7 days is the correct threshold**:

1. **Network partition duration**: A 7-day offline period is the practical ceiling for a homelab user (travel, hardware failure, extended CasaOS outage). If casaclaude is offline for 8+ days, it is not a transient outage — the user is aware and the session summaries from that period are recoverable by other means (the Claude transcript files are still on disk). Summaries older than 7 days are those from a genuinely extended outage, not a transient connectivity blip.

2. **Disk space**: Each pending file contains at most 6 writes (1 episodic summary + 5 semantic decisions) with average content length ~500 characters each. A 7-day accumulation at 5 sessions/day = 35 sessions = 35 pending files ≈ 35 × 3KB = ~105KB. This is negligible.

3. **Project-brief.md requirement**: "The system must degrade gracefully when any single backing service is down." A 4-day threshold means a 5-day outage permanently loses 1 day of sessions. A 7-day threshold means all sessions from a typical outage (up to 7 days) are recovered. The 7-day threshold more faithfully implements graceful degradation.

4. **LLPC routing vs. data preservation are different objectives**: The 4-day tau argument applies to retrieval ranking (how much weight to give a session in LLPC routing). It does not apply to whether the memory should exist. A 7-day-old session summary that is past the LLPC horizon still contributes to profile synthesis and timeline queries. Discarding it at 4 days destroys data that serves purposes other than LLPC routing.

**Implementation**: The `expires_at` field is pre-computed at write time as `created_at + 7 days`. The retry logic compares `Date.now() > expires_at` — no calendar arithmetic needed at retry time.

## Ideas Extracted

### Idea: submitted-at-flag-as-crash-safe-dedup
- **Novelty**: 0.72
- **Evidence**: 0.80
- **Feasibility**: 0.97
- **Description**: The crash-after-submit race in pending queue retry is resolved by a two-phase file update: (1) submit to API, (2) write `submitted_at` to the pending file synchronously before deleting it. On the next retry, files with `submitted_at` set are skipped and deleted. This is the minimal correct solution to the race without requiring UUID storage or content-hash dedup. The pattern is identical to PostgreSQL's WAL commit record: the commit is durable when the LSN is written to disk, not when the in-memory state is updated. No distributed systems protocol or UUID round-trip required. Implementation cost: 2 lines added to the success path in retryPendingQueue.

### Idea: single-file-per-session-pending-queue
- **Novelty**: 0.68
- **Evidence**: 0.75
- **Feasibility**: 0.95
- **Description**: Grouping all writes from a single session into one pending file (rather than one file per write) ensures semantic atomicity: the episodic summary and its extracted key decisions are either all submitted or none are. This avoids partial-submission states where the summary exists in Recall but the decisions do not, or vice versa. The single-file model also simplifies retry logic: iterate files, not writes. Each pending file is a self-contained replay unit. The trade-off is that if the API accepts the first N writes but fails on write N+1, all writes are left pending for next session — the session submits N duplicates on the next successful retry. Given the 7-day threshold and the `submitted_at` dedup, this produces at most 1 duplicate per write across the entire retry history.

### Idea: data-preservation-vs-retrieval-quality-threshold-split
- **Novelty**: 0.78
- **Evidence**: 0.65
- **Feasibility**: 0.95
- **Description**: The pending queue discard threshold (7 days) and the LLPC routing horizon (tau=20 sessions ≈ 4 days) serve different objectives and must not be conflated. The routing horizon determines how much causal weight to assign a session in real-time retrieval ranking. The discard threshold determines when to give up trying to persist a session summary at all. Setting the discard threshold to match the routing horizon destroys data that serves profile synthesis and timeline queries, which operate independently of LLPC routing. AI memory systems that implement pending queues (none reviewed) would face this same distinction: data preservation SLAs and retrieval relevance windows are different parameters that should be tuned independently.

## Key Finding

The crash-after-submit duplicate submission problem is resolved by a two-phase `submitted_at` flag written to the pending file before deletion — eliminating the need for UUID storage or G3 content-hash dedup for anything beyond the residual sub-millisecond crash window. The 7-day discard threshold is correct because the pending queue's objective is data preservation (not retrieval quality), and LLPC routing horizon tau is the wrong parameter to use for discard — those are orthogonal concerns. The complete implementation is 44 lines of JavaScript across `recall-session-summary.js` (25 lines: `writePendingQueue` function + integration) and `recall-retrieve.js` (24 lines: `retryPendingQueue` function + call site), with zero changes to the Recall API, Redis, or PostgreSQL schema.

## Suggested Follow-ups

- **[PHYSICS] Q069**: The retry loop in `retryPendingQueue` breaks on first write failure and leaves the entire pending file for the next session. This means if CasaOS has a 6-day outage and reconnects with 30 pending files, all 30 attempt submission at session start, serialized. Estimate the wall-clock cost: 30 files × average 3 writes × 3000ms timeout = up to 270 seconds worst-case. Is this acceptable at session start, or should the retry be bounded (max N files per session, remainder deferred)?

- **[ADVERSARIAL] Q070**: The single-file-per-session model submits all writes as a sequential loop, breaking on first failure. If the episodic summary succeeds but the first semantic decision write fails (e.g., the API rate-limits after the first write), the file is left unpending even though the summary was stored. On next retry, the summary is re-submitted as a duplicate. Does this violate the no-duplicate guarantee, or is a duplicate session summary harmless enough that the simplicity of the break-on-first-failure model is worth it?
