# Finding: Q067 — Epoch Reset Creates 173-Day Silent False-Positive Window; Option (a) Recovers in 0.25ms

**Question**: The epoch-clock (Q062) depends on Redis storing `recall:global_epoch`. If Redis resets (FLUSHALL, container restart without persistence), the counter goes to 0. All existing memories have `last_access_epoch` values from the prior epoch (e.g., 847). The LLPC routing query `last_access_epoch >= (current_epoch - tau)` becomes `last_access_epoch >= (0 - 20) = -20` — all stored epoch values are far above -20, so ALL memories appear to be within the recency window. This is a silent false-positive: LLPC injects stale memories for every query until tau sessions rebuild the epoch. What is the minimum detection mechanism? What is the theoretical minimum cost of recovery? Does `SELECT MAX(last_access_epoch)` require a full table scan or can it use a B-tree index? What is the latency at N=50,000?
**Question Type**: PHYSICS
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: database systems (PostgreSQL B-tree index internals), distributed systems (epoch continuity protocols, vector clock reconciliation)

---

## Failure Mode Analysis

### The failure direction

After Redis resets, `recall:global_epoch` returns to 0 (or to 1 after the first INCR). Existing memories in PostgreSQL retain their prior `last_access_epoch` values — for example, 847 if the epoch ran for 847 sessions before the reset.

The LLPC recency predicate evaluates as:

```
last_access_epoch >= (current_epoch - tau)
847             >= (0 - 20)
847             >= -20
TRUE
```

Every memory in the corpus passes this test. The working-set window filter — the entire purpose of the epoch clock — admits all N memories as "current" for every query. This is not a crash or an error message. It is a silent behavioral regression: LLPC injects the full corpus into every prompt until natural recovery occurs.

### Natural recovery timeline (without intervention)

Left uncorrected, the false positive persists until the epoch counter grows large enough that high-valued `last_access_epoch` rows fall outside the tau window:

```
last_access_epoch >= (current_epoch - tau)
847              >= (session_N - 20)
session_N        <= 867  (when this condition finally becomes false)
```

At 5 sessions per day, natural recovery takes **867 sessions = 173 calendar days**. During those 173 days, LLPC loses all working-set discrimination — it becomes equivalent to injecting the entire memory corpus for every prompt. The system degrades silently from targeted retrieval to full-corpus injection.

### Why this is worse than Redis unavailability (Q062 Failure Mode 2)

Q062 Failure Mode 2 addressed Redis being down — `INCR recall:global_epoch` raises a connection error, which is catchable and routes to the PostgreSQL sequence fallback. An epoch reset is fundamentally different: Redis is up, responds normally, and returns 0 (or 1). There is no error to catch. The application operates normally from Redis's perspective. The false positive is invisible at the connection layer.

## Theoretical Minimum

### Minimum information required for detection

Detecting an epoch reset requires answering one binary question: "Does the current Redis epoch represent a value smaller than the maximum epoch ever written to PostgreSQL memories?"

This requires:
1. One read from Redis: `GET recall:global_epoch` → current epoch value
2. One read from PostgreSQL: `SELECT MAX(last_access_epoch) FROM memories` → highest epoch ever stored

These are the two minimum necessary reads. No fewer reads can answer the question — neither source alone is sufficient.

### Does MAX(last_access_epoch) require a full table scan?

No. PostgreSQL can serve `SELECT MAX(last_access_epoch)` via a **backward index scan** on a B-tree index in O(log N) time.

A B-tree index stores values in sorted order. The maximum value is always at the rightmost leaf node. PostgreSQL's query optimizer recognizes `MAX(indexed_column)` and rewrites it as a single-row backward index scan: it walks from the root node to the rightmost leaf, reading one page at each level, then returns the last key on that leaf — without touching any other rows.

**B-tree traversal for N=50,000:**

```
B-tree fanout: ~340 keys per 8KB page (integer key, pointer pair)
Height: ceil(log_340(50,000)) = 2 levels (root → internal → leaf)
Pages read: 3 (root + one internal + rightmost leaf)
```

This is a constant-time operation — 3 page reads regardless of whether N is 50,000 or 5,000,000.

### Theoretical minimum cost table

| Operation | Floor | Calculation | Gap type |
|---|---|---|---|
| Redis GET | 0.05ms | LAN RTT physics floor (1GbE, ~100μs) | Physics |
| PG B-tree root→leaf (N=50K) | 0.05ms | 3 pages × buffer-pool hit (~16μs/page); disk floor is 3 × ~0.1ms NVMe | Physics (cached) / Architecture (cold) |
| Integer comparison MAX > epoch | ~0ns | 1 CPU cycle, ~0.3ns at 3GHz | Physics (negligible) |
| Redis SET (recovery, conditional) | 0.05ms | Same LAN RTT as GET | Physics |
| **Total (concurrent GET+MAX, no reset)** | **0.10ms** | max(Redis GET, PG MAX) if issued in parallel | Physics |
| **Total (sequential, no reset)** | **0.15ms** | Redis GET + PG MAX (hot) | Architecture gap |
| **Total (sequential, reset detected)** | **0.25ms** | Redis GET + PG MAX (hot) + Redis SET | Architecture gap |
| **Total (cold cache, reset detected)** | **12.2ms** | Redis GET(0.1ms) + PG MAX cold (12ms: 3 NVMe reads × 4ms) + Redis SET(0.1ms) | Architecture (solvable by warming) |

**At N=50,000 with a hot buffer pool (the normal steady-state case), the full detection and recovery operation costs 0.25ms.** This fits within the session-start overhead budget with room to spare.

### Full table scan baseline (no index)

For comparison: without an index, `SELECT MAX(last_access_epoch)` degrades to a sequential scan.

```
N=50,000 rows × 100 bytes/row ≈ 618 pages × 8KB = ~5MB
Sequential NVMe read at 3GB/s: ~1.7ms
PostgreSQL executor overhead: 50,000 rows × 0.3μs/row ≈ 15ms
Total full scan: ~16-17ms
```

The B-tree index delivers a **100-340x speedup** for the MAX query (0.05ms hot vs 15-17ms full scan). The index is not optional — it is what makes option (a) practical at session-start frequency.

The B-tree index on `last_access_epoch` is already recommended by Q062 for the LLPC routing query `WHERE last_access_epoch >= (:current_epoch - tau)`. The MAX detection check is a **free rider** on infrastructure that should already exist.

## Recommended Path

### Option (a): PostgreSQL MAX + Redis comparison at session start

```python
async def check_epoch_continuity(redis_client, pg_pool) -> int:
    """
    Called once per session start. Returns the validated current epoch.
    Cost: 0.15ms (no reset) or 0.25ms (reset detected and recovered).
    """
    redis_epoch, pg_max = await asyncio.gather(
        redis_client.get("recall:global_epoch"),
        pg_pool.fetchval("SELECT MAX(last_access_epoch) FROM memories")
    )
    redis_epoch = int(redis_epoch or 0)
    pg_max = int(pg_max or 0)

    if pg_max > redis_epoch:
        # Reset detected: Redis epoch is behind the highest epoch stored in PG.
        # Restore continuity by advancing Redis to pg_max + 1.
        restored_epoch = pg_max + 1
        await redis_client.set("recall:global_epoch", restored_epoch)
        return restored_epoch

    return redis_epoch
```

**Why option (a) is correct:**

- PostgreSQL is the ground truth for epoch history. `last_access_epoch` values are durable (WAL-persisted) and survive Redis restarts.
- The comparison `pg_max > redis_epoch` is an unambiguous reset signal. Under normal operation, `redis_epoch >= pg_max` always holds (the Redis epoch is incremented at session end, then `last_access_epoch` is written; Redis epoch always leads or equals PG max).
- The two reads can be issued concurrently via `asyncio.gather` — the total wall-clock is `max(Redis GET, PG MAX)` ≈ 0.10ms rather than the sequential sum of 0.15ms.
- Recovery is a single Redis SET that restores monotonicity. No PostgreSQL writes required for recovery.
- The check is idempotent: if called multiple times concurrently at session start (e.g., burst of sessions), each call reads the same pg_max and sets the same recovery value. A Redis `SET NX` (set if not exists) is not needed because the recovery value is deterministic from pg_max.

**Edge case: first boot (no memories stored yet):**

`SELECT MAX(last_access_epoch)` returns NULL when the table is empty. `int(None or 0) = 0`. Redis GET on a non-existent key returns None → 0. Both sides are 0, `0 > 0` is False — no recovery triggered. Correct.

### Why option (b) fails under FLUSHALL

`recall:epoch_watermark` is stored in Redis. A `FLUSHALL` removes both `recall:global_epoch` and `recall:epoch_watermark` simultaneously. After FLUSHALL, both keys are absent. The watermark approach cannot distinguish:
- First boot (watermark never existed)
- Post-FLUSHALL (watermark was deleted along with the epoch)
- Container restart without persistence (same as FLUSHALL)

Detecting watermark absence alone cannot trigger correct recovery — it does not know what epoch value to restore to. Option (b) is incomplete unless paired with the same PG MAX query that makes option (a) complete on its own.

Option (b) can serve as a **fast path** (one Redis GET instead of one Redis GET + one PG query) for detecting resets in the common case where the watermark is present (no reset occurred). But the recovery path always requires PG MAX. The combined approach (b)+(a-recovery) adds complexity with minimal latency benefit over option (a) alone: the PG MAX query is already 0.05ms on a hot buffer pool.

### Why option (c) adds write-path overhead for read-path safety

A PostgreSQL trigger on `memories` updates a `recall_metadata` table on every write. At 5-50 memory writes per session, this adds 5-50 trigger executions per session, each costing ~0.01ms. Total overhead: 0.05-0.5ms of added write latency per session — a non-trivial cost imposed on the normal path to guard against a rare failure mode. Option (a) places the cost at session start (where it is paid once) rather than at every write.

A PostgreSQL sequence as Redis fallback (mentioned in Q062 Failure Mode 2) is complementary — it handles Redis being down, but that is a different failure mode from Redis resetting with wrong values.

## Gap Analysis

### Physics gaps (unavoidable)

- **LAN RTT (0.05-0.1ms)**: The Redis GET and SET each require a LAN round trip. This is bounded by the speed of light over copper/fiber for the CasaOS LAN segment. Cannot be eliminated without moving Redis to the same process (which would eliminate persistence, defeating its purpose).
- **B-tree traversal I/O (0.05ms cached, 12ms cold)**: Three page reads are the minimum to find the rightmost leaf of a height-2 B-tree. The cached case is near-optimal; the cold case is bounded by NVMe random read latency (~4ms/page). Cannot be reduced without pre-warming the index pages (an architecture choice, not a physics limit).
- **One comparison operation**: The comparison `pg_max > redis_epoch` is a single ALU instruction. Physics floor.

### Architecture gaps (design choices that can be improved)

- **No continuity check in current design (the primary gap)**: The epoch-clock as designed in Q062 has no detection mechanism. The architecture gap is that `recall:global_epoch` is treated as an append-only counter without any reconciliation against its durable record in PostgreSQL. This gap costs nothing when Redis is healthy and 173 calendar days of degradation when Redis resets. The fix (option a) closes this gap in ~20 lines of code and 0.25ms of session-start latency.
- **Sequential vs. concurrent reads**: Issuing Redis GET and PG MAX sequentially adds 0.05ms compared to concurrent `asyncio.gather`. This is an implementation gap, easily closed.
- **Cold buffer pool (12ms cold path)**: If the `last_access_epoch` index is evicted from PostgreSQL's shared_buffers between sessions, the MAX query costs 12ms instead of 0.05ms. The index pages total ~3 × 8KB = 24KB. PostgreSQL's shared_buffers on a CasaOS deployment with 8-16GB RAM is typically 2-4GB — a 24KB index fits trivially. The cold case only occurs after extended idle periods (hours) where the buffer pool has been reclaimed. This is manageable by ensuring the index exists and is appropriately sized.

### Implementation gaps (code quality)

- **Missing `asyncio.gather` for concurrent reads**: A naive sequential implementation doubles the latency of the common path (no reset). One-line fix.
- **Missing NULL handling for empty table**: `SELECT MAX(...) FROM empty_table` returns NULL, not 0. Requires `int(pg_max or 0)` guard.

## Ideas Extracted

### Idea: epoch-continuity-guard-via-pg-max-watermark

- **Novelty**: 0.82 — No production AI memory system reviewed (MemGPT, Letta, mem0, Zep) implements epoch continuity reconciliation between a Redis counter and a PostgreSQL durable record. The pattern is common in distributed systems (vector clock reconciliation, Lamport timestamp recovery) but has not been observed applied to AI memory system epoch clocks. The specific combination — Redis as the fast-path epoch store, PostgreSQL MAX as the durable watermark, and a session-start reconciliation pass — is novel in this application domain.
- **Evidence**: 0.78 — Lamport clock reconciliation after node failure is a well-established distributed systems technique (Lamport 1978, "Time, Clocks, and the Ordering of Events in a Distributed System"). The specific recovery pattern (restore monotonically increasing counter to MAX(observed_value) + 1 after reset) is production-proven in distributed database systems (CockroachDB HLC recovery, TiKV timestamp oracle recovery). PostgreSQL B-tree MAX() optimization is documented behavior (backward index scan) present in all PostgreSQL versions since 7.4. The latency estimates are computable from first principles with high confidence.
- **Feasibility**: 0.92 — The implementation is 20 lines of Python using existing asyncio, Redis, and PostgreSQL clients already in the Recall stack. No new infrastructure. The B-tree index on `last_access_epoch` is already recommended by Q062 for the LLPC routing query — this idea requires no additional schema changes beyond what Q062 already specifies. Risk: the check must be placed at session start (before any LLPC query fires) and must be awaited before the epoch value is used. A single decorator or middleware call is sufficient.
- **Description**: At session start, before any LLPC routing query executes, issue `GET recall:global_epoch` and `SELECT MAX(last_access_epoch) FROM memories` concurrently. If `pg_max > redis_epoch`, a reset has occurred. Recover by setting `recall:global_epoch = pg_max + 1` in Redis. The check costs 0.10ms (concurrent) to 0.15ms (sequential) when no reset has occurred, and 0.25ms when recovery is needed. This completely eliminates the 173-day silent false-positive window at a cost of one additional asyncio operation per session start.

### Idea: pg-max-as-epoch-oracle-fallback

- **Novelty**: 0.72 — Using PostgreSQL MAX(last_access_epoch) as an oracle for Redis epoch recovery is not a pattern observed in current AI memory systems. However, the concept of a SQL aggregate serving as a durable checkpoint for a volatile cache counter is used in job queue systems (e.g., using MAX(sequence_id) from the jobs table to restore a Redis sequence after restart). The application to epoch-clock recovery is a natural extension.
- **Evidence**: 0.70 — Job queue systems (Sidekiq, Celery, BullMQ) use database MAX() queries as recovery anchors after queue broker failures. The pattern is practitioner-validated but not formally studied in the context of AI memory systems. The cost arithmetic is computed from first principles with known PostgreSQL behavior.
- **Feasibility**: 0.95 — A zero-infrastructure-change implementation: one SQL query added to session initialization. The B-tree index required is identical to the index already needed for the LLPC routing query. No new dependencies.
- **Description**: Designate `SELECT MAX(last_access_epoch) FROM memories` as the authoritative recovery oracle for `recall:global_epoch` after any Redis state loss. The MAX query costs 0.05ms (hot) against an index that already exists for routing purposes. This makes PostgreSQL the durable ground truth for the epoch counter, reducing Redis from a required dependency for epoch continuity to an optional performance cache. The epoch-clock design becomes: Redis for fast reads in steady state, PostgreSQL for recovery after any Redis state disruption.

## Key Finding

A Redis `FLUSHALL` or container restart without persistence silently corrupts the epoch-clock for up to 173 calendar days — all memories appear "current" because prior `last_access_epoch` values (e.g., 847) satisfy `847 >= (0 - 20)` after the reset. Option (a) closes this gap completely: at session start, issue `GET recall:global_epoch` and `SELECT MAX(last_access_epoch)` concurrently (0.10ms total on a hot buffer pool); if MAX exceeds the Redis epoch, restore Redis to MAX+1 in a single SET (0.25ms total for the full detection-and-recovery path). The `SELECT MAX()` query does not require a full table scan — a B-tree index on `last_access_epoch` (already required for LLPC routing) serves it in 3 page reads at O(log N) cost, independent of corpus size. The fix is 20 lines of Python with no new infrastructure.
