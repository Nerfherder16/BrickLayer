## Research Question

What is a memory system that never persists anything? Design from physical requirements only.

**Forbidden terms**: store, save, persist, write, database, file, disk, log, index, cache, retain, keep, remember, record, append, backup, archive, commit.

---

## Physical Requirement for Memory

A past observation O_t produces correct future behavior B_{t+k} that would not have occurred without O_t. For this to be possible, some **physical state** must exist at time t+k that was **caused by** O_t.

This is the only physical requirement: a causal path from past observation to future behavior. The question is what physical substrates can carry that causal path.

---

## Three Physical Mechanisms

### Mechanism 1: In-Weights Residue (Function Modification)

The observation modifies the forward-pass function itself. No external artifact is created. The knowledge is embedded in the weights.

Biological analog: long-term potentiation (LTP) — synaptic weights strengthen via Hebbian plasticity after co-activation. The "memory" is the modified connection strength, not an external encoding.

For an LLM: LoRA fine-tuning on the observation. The observation is "in" the LoRA adapter weights. The system produces correct behavior in all future sessions without any retrieval step, without any injection, without any query. The KV-cache does not need to carry it. The context window does not consume space for it.

**Properties:**
- Retrieval latency: zero (weights are always active)
- Context consumption: zero
- Scope: all future sessions without exception
- Update cost: high (gradient descent per observation)
- Precision: approximate (weights encode distributions, not facts)
- Reversibility: none (weight modification cannot be undone without re-training)

### Mechanism 2: Activation Residue (KV-Cache State)

The KV-cache of the current context is the physical consequence of having processed past tokens. This is not a retrieval step — it is the natural result of running the forward pass on the input sequence. The "memory" is the attention state itself.

This was identified in Q238 as the physical optimum for memory injection. Here the deeper point: for observations within the current context window, no external system is needed at all. The observation is physically present as activation state.

**Limitation**: activation state is lost at context reset. This is why Mechanism 1 (in-weights) is needed for cross-session memory.

### Mechanism 3: Causal World-State (Environmental Residue)

If an observation caused a physical change in the world — code was modified, a configuration was set, infrastructure was deployed — the world itself carries the memory. The agent re-observes the world state rather than re-reading an internal encoding.

This is the Gibsonian affordance view: the environment is an extension of the cognitive system. An agent that can observe its environment can "remember" any observation that changed the environment, because the changed state is the memory.

For Recall 2.0's use case: if Tim observed "Ollama is at 192.168.50.62:11434" and then modified a configuration to use that address, the configuration carries the observation. No external memory system needed — the world is the memory.

**Limitation**: not all observations change the world. Pure observational facts (what was discussed, what was learned conceptually) leave no causal world trace.

---

## The Physical Structure of Storage as a Proxy

External text-based retrieval (Recall 2.0) is a **re-injection mechanism**: it converts Mechanism 3 world-state back into Mechanism 2 activation state by reading the encoded text into the context window.

The sequence: O_t → text encoding (forbidden) → retrieval at t+k → text injection → KV-cache residue → correct behavior.

Removing the "forbidden" middle step reveals the physical requirement more clearly: **any causal path from O_t to physical state at t+k enables correct behavior**. Text storage is one such path. In-weights modification is another. World-state causation is a third.

## Complementarity for Recall 2.0

| Tier | Mechanism | Update cost | Retrieval cost | Context cost | Session scope |
|---|---|---|---|---|---|
| Tier 1 | External retrieval (current Recall) | Low | Low | High | Current only |
| Tier 2 | In-weights LoRA (not yet built) | High | Zero | Zero | All sessions |
| Tier 0 | Activation residue (KV-cache) | Zero | Zero | Medium | Current context |
| Tier 3 | World-state causation (environment) | Zero | Zero | Zero | All sessions |

The implication: Recall 2.0 covers Tier 1. Tier 2 (in-weights) would provide "ambient knowledge" that works even when the retrieval hook fails, the service is down, or K=5 retrievals miss the relevant fact. Tier 0 is already operational via the context window. Tier 3 is already operational via the environment.

**Tier 2 implementation path**: periodic LoRA fine-tuning on session summaries from Q226. Each session summary becomes a training example. After N sessions, the LoRA adapter "knows" Tim's infrastructure without any retrieval. Cost: training compute. Benefit: zero-latency ambient knowledge that survives all retrieval failures.

## The Deepest Insight

The taboo removal reveals the hidden assumption in every retrieval-based memory system: **that the primary physical mechanism is Tier 1 (external retrieval)**. But this is the highest-latency, highest-context-cost mechanism. The reason it dominates is: (a) in-weights modification is expensive per-update and irreversible, (b) activation state resets at context boundaries, (c) not all observations leave world-state traces.

For Recall 2.0, the correct physical architecture is **all four tiers operating simultaneously**:
- Tier 0: KV-cache (free for in-context observations)
- Tier 1: External retrieval (fast, exact, current sessions)
- Tier 2: In-weights LoRA (periodic, approximate, universal scope)
- Tier 3: World-state observation (for infrastructure/configuration facts)

Current Recall 2.0 implements only Tier 1. Adding Tier 2 would eliminate the class of failures where K=5 retrievals miss a critical fact that has been observed many times across many sessions — the fact would be encoded in weights and activate automatically.

## Verdict

**BREAKTHROUGH.** Removing the storage assumption reveals three physical mechanisms for memory: (1) in-weights residue (weight modification — zero retrieval latency, universal scope, high update cost), (2) activation residue (KV-cache — zero cost within context), (3) causal world-state (environment carries the memory). External text retrieval (Recall 2.0) is a re-injection mechanism that converts world-state back into activation state. The taboo removal shows Recall operates at Tier 1 only; Tier 2 (LoRA fine-tuning on session summaries) would provide ambient knowledge that survives all retrieval failures with zero latency penalty. The physical requirement for memory is any causal path from past observation to future behavior — not specifically a retrieval-query-inject sequence.
