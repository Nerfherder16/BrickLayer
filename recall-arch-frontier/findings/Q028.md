# Finding: Q028 — Taboo Architecture: Three-Layer Integrated Eviction Policy

**Question**: Design the eviction policy for a three-layer memory architecture (ARC hot cache, SWR consolidation buffer, competitive-displacement cold store) from first principles, ignoring Recall's existing TTL and importance score mechanisms.
**Question Type**: TABOO
**Verdict**: BREAKTHROUGH
**Severity**: High

---

## Taboo Design: Integrated Three-Layer Memory Eviction

### Physical Requirements

State the problem in physical terms:

- **Tier 1**: ~10MB RAM, ~100–500 memory objects, sub-millisecond access. Hash-table keyed by memory ID. Access path: synchronous, on the hot retrieval path. Mutation rate: high — every query causes reads; every session causes writes.
- **Tier 2**: ~1GB SSD/disk, ~100K memory objects, 1–10ms access. Requires I/O for cold reads. This is the primary indexed semantic store. Mutation rate: moderate — session-end consolidation, background merges.
- **Tier 3**: Unbounded capacity, 10–100ms access, batch-optimized. Not suitable for real-time retrieval. Mutation rate: low — archival writes, rare reads triggered only by deep historical queries.

**The design goal**: Given memories entering at Tier 1 and a finite capacity at each tier, derive the promotion, demotion, and eviction rules such that the system maximizes the probability that the next query retrieves the right memory from the fastest available tier.

This is equivalent to: given a reference string of future queries (unknown), minimize the expected access cost across the three tiers.

## First Principles Derivation

### 1. Tier 1 Eviction Policy (derived)

**What ARC is actually solving:**

ARC solves the recency/frequency tradeoff under an unknown workload. The four lists (T1, T2, B1, B2) are not arbitrary — they are the minimum state required to answer two questions simultaneously:

1. "Has this memory been accessed recently?" (T1/B1 signal)
2. "Has this memory been accessed repeatedly?" (T2/B2 signal)

The adaptive parameter `p` adjusts which signal dominates based on which type of eviction error the system is currently making. When B1 gets ghost hits (we evicted recency-items prematurely), increase T1's target. When B2 gets ghost hits (we evicted frequency-items prematurely), decrease T1's target.

**For semantic memories, not disk pages, what changes?**

In a disk page cache, "a page" is a fixed-size unit with no semantic content — all pages are equal except for their access history. In a memory store, memories have heterogeneous semantic weight. But the eviction decision should still be access-pattern-driven, not semantic-weight-driven, because: (a) semantic weight is assessed at write time under high uncertainty; (b) actual access frequency is a realized signal, not a prediction.

The correct mapping is therefore: use ARC exactly as designed, with memory IDs as the cache keys, ignoring semantic content. The exception is ties: when two memories have equal standing in ARC's ordering (same T1/T2 position), prefer to evict the one with lower write-time importance. But this is a tiebreaker, not the primary sort key.

**Derived promotion trigger from Tier 2 → Tier 1:**

A memory should be promoted from Tier 2 into Tier 1 when a query returns it as a result. The trigger is retrieval, not write. This means:

- Every `recall_retrieve` call that touches a Tier 2 memory promotes that memory to Tier 1 (or refreshes its position in T1/T2 if already present).
- Promotion into Tier 1 costs one eviction from Tier 1 (the ARC eviction rule selects the victim from T1-LRU or T2-LRU depending on whether |T1| ≥ p).
- Promotion does not move the memory out of Tier 2 — the memory now lives in both tiers simultaneously. Tier 1 is a cache over Tier 2, not a transfer. Demotion from Tier 1 means the memory's hot-cached representation is dropped; the canonical record remains in Tier 2.

**Derived demotion trigger from Tier 1 → Tier 2:**

Demotion is triggered by the ARC eviction rule firing. Specifically:

- If |T1| ≥ max(1, p): evict the LRU tail of T1. That memory's in-memory representation is released. The key is moved to the B1 ghost list.
- Else: evict the LRU tail of T2. That memory's key is moved to B2.

The evicted memory's canonical record is NOT deleted from Tier 2. Demotion is purely a cache eviction — the semantic store is not modified.

**Derived ghost list semantics for memory (not disk pages):**

In database ARC, a ghost hit on B1 means "we evicted this page from the recency list and it was requested again — we need more recency capacity." For semantic memories, the semantics are richer:

- **B1 ghost hit** (memory was in T1, evicted to B1, now queried again): this memory was accessed in recent sessions but fell out of the hot cache. The correct interpretation: this memory has recency value we underweighted. Signal: increase `p` (expand T1's target size), AND re-promote the memory to T2 (not T1's MRU as standard ARC would, because the memory may have moderate long-term relevance rather than session-hot relevance).
- **B2 ghost hit** (memory was in T2, evicted to B2, now queried again): this memory was a repeating cross-session memory that we dropped under capacity pressure. Signal: decrease `p` (expand T2's target within Tier 1), AND re-promote the memory to T2's MRU (standard ARC behavior — frequency-promoted position).

The ghost list TTL for memories should be bounded by session cycle, not arbitrary time: a ghost entry is valid for as long as it takes to complete one "work cycle" — typically 2–7 days for a project-focused AI memory system. After that boundary, a ghost hit is no longer informative (the access pattern may have changed).

**Full Tier 1 eviction policy:**

```
State: T1 (recency list), T2 (frequency list), B1 (recency ghost), B2 (frequency ghost), p, c = Tier 1 capacity

On query returning memory M:
  If M in T1: move M to MRU of T2 (frequency promotion)
  If M in T2: move M to MRU of T2 (frequency refresh)
  If M in B1: increase p, insert M into T2 MRU, evict from T2-LRU to B2
  If M in B2: decrease p, insert M into T2 MRU, evict from T1-LRU to B1
  If M not in any list: insert M into T1 MRU; if |T1|+|T2| = c, run eviction rule

Eviction rule (when Tier 1 at capacity):
  If |T1| >= max(1, p): demote T1-LRU to B1
  Else: demote T2-LRU to B2
```

### 2. Tier 1 → Tier 2 Handoff Semantics

**What must be preserved on demotion from Tier 1:**

When a memory is evicted from Tier 1's data set (its in-memory representation is dropped), the following must survive in the ghost list or the Tier 2 record:

1. **The memory ID** (UUID): Required for ghost list operation. Without this, a future access cannot be recognized as a ghost hit. Cost: ~16 bytes per ghost entry. With 500 ghost entries, total ghost overhead is ~8KB — negligible.

2. **The list assignment** (B1 or B2): The ghost list membership encodes whether the memory was a recency-pattern or frequency-pattern memory. This is the signal that drives `p` adjustment.

3. **The ghost entry creation time**: Required to expire ghost entries after the session-cycle boundary. A single timestamp per ghost entry. Cost: ~8 bytes.

4. **The session count at eviction** (optional but recommended): Rather than a wall-clock timestamp, tracking how many distinct sessions have elapsed since the ghost entry was created is more robust. Session count is a logical clock that is not affected by varying session length. A ghost entry is expired when `current_session_count - eviction_session_count > K` where K is the ghost TTL in sessions (recommended K = 3–5 sessions).

**What can be discarded:**

1. **The embedding vector**: The canonical embedding lives in Tier 2. The Tier 1 hot cache may hold a copy for fast retrieval, but on demotion, the embedding copy is dropped. Tier 2 retains the canonical vector.

2. **The full memory text**: The raw text lives in Tier 2. Tier 1 may cache it for context assembly, but on demotion, the text is dropped.

3. **The write-time importance score**: The canonical importance score lives in Tier 2's metadata store. Tier 1 need not retain it — the ARC policy operates on access history, not importance scores.

4. **The session context at storage time**: This is stored in Tier 2 (the session summary association). Tier 1 never needs it.

**Minimum representation for correct future ARC decisions:**

```
GhostEntry {
  memory_id: UUID        # 16 bytes
  ghost_list: B1 | B2   # 1 byte
  eviction_session: u32  # 4 bytes  (logical session clock)
  eviction_rank: float   # 4 bytes  (position in the evicted list, for tie-breaking on re-promotion)
}
```

Total: ~25 bytes per ghost entry. For 1000 ghost entries (2× the active cache size, as recommended by ARC): ~25KB. This is the complete state needed to resume correct ARC adaptation after a ghost hit.

**The invariant on handoff:**

Demotion from Tier 1 must be invisible to Tier 2. Tier 2's record of the memory is not modified when Tier 1 evicts the memory. The Tier 1 cache is a pure read-through cache over Tier 2 — demotion only affects Tier 1's state, never Tier 2's state. This preserves the independence of the tiers and means that Tier 2 can be reasoned about separately from Tier 1's eviction dynamics.

### 3. Tier 2 → Tier 3 Consolidation Trigger (derived)

**Frame the problem as a signal detection problem:**

The consolidation pass must move memories from Tier 2 (fast, query-serving) to Tier 3 (slow, archival) at the right time. Too early: useful memories become inaccessible for real-time retrieval. Too late: Tier 2 fills with low-value memories that degrade retrieval quality by adding noise to query results.

There are three candidate trigger types:

**Option A — Time-based trigger:**
Run consolidation on a fixed schedule (e.g., every 24 hours). Problem: session boundaries do not align with clock cycles. A session that runs 2 hours and generates 80 memories should trigger consolidation sooner than a session that generates 3 memories over 8 hours. Time-based triggers are insensitive to workload variance.

**Option B — Capacity-based trigger:**
Run consolidation when Tier 2 reaches a fill threshold (e.g., 80% of 100K capacity). Problem: capacity-based triggers are reactive, not anticipatory. By the time 80% capacity is reached, the quality of retrieval has already degraded (signal-to-noise in query results decreases as store size grows beyond the optimal operating range). Capacity-based triggers also produce bursty consolidation that competes with retrieval I/O.

**Option C — Event-based trigger:**
Run consolidation at a specific semantic event. The Q002 finding (SWR consolidation) identified session-end as the correct trigger — it maps to the biological "sleep onset" boundary where online encoding stops and offline consolidation begins.

**Derived answer: session-end is the correct primary trigger, with a capacity guard as a secondary trigger.**

The reasoning:

1. Session-end is the natural boundary between "memories that are likely to be needed again in this session" (high Tier 2 priority) and "memories that may be needed in a future session" (consolidation candidates). Within a session, no memory should be consolidated — the access pattern is live and consolidation would destroy relevance.

2. Session-end is already implemented in Recall's hook system (`recall-session-summary.js` fires on Stop). The consolidation trigger is a free attachment to an existing event.

3. The secondary capacity guard prevents runaway growth if the session-end trigger is unreliable (e.g., process kill without clean shutdown): when Tier 2 reaches 85% capacity, trigger an emergency consolidation pass even without a session-end event.

**Derived unit of consolidation: semantic clusters, not individual memories.**

Individual memory consolidation (move M from Tier 2 to Tier 3) is the wrong unit for two reasons:

1. A memory's value is partly relational — it is relevant because it connects to other memories (same session, same topic, same project). Moving a single memory to Tier 3 while leaving its neighbors in Tier 2 fragments the semantic neighborhood and makes the moved memory harder to retrieve even if it is directly queried.

2. The correct question is not "is this individual memory worth keeping?" but "is this cluster of related memories still worth indexing in the fast store?" A cluster of 20 memories about a project that ended 6 months ago may all be safely archived together, even if no individual memory would score below the archive threshold on its own.

**Derived consolidation procedure:**

```
On session-end event:
  1. Identify session window: all memories stored in the last session (tracked by session_id).
  2. Cluster session memories by semantic neighborhood: group memories with inter-memory
     angular distance < 0.20 in embedding space into semantic clusters.
  3. For each cluster, compute the cluster's composite access signal:
     - total retrieval count within the session
     - last retrieval time (how recently was anything in this cluster retrieved?)
     - cluster age (sessions elapsed since the cluster's oldest member was stored)
  4. Apply the consolidation decision:
     - cluster_score = retrieval_count / (1 + sessions_since_last_access)
     - If cluster_score < consolidation_threshold: move entire cluster to Tier 3.
     - If cluster_score >= threshold: leave cluster in Tier 2.
  5. For promoted clusters (staying in Tier 2), update importance scores
     based on retrieval count (access-confirmation effect).
  6. For demoted clusters (moving to Tier 3), write a cluster manifest to Tier 2:
     a lightweight record containing cluster centroid, member IDs, and the
     cluster's composite metadata. The manifest enables cluster-level retrieval
     without loading individual Tier 3 members.
```

The cluster manifest is the critical artifact: it keeps Tier 3 content discoverable without loading the full Tier 3 records. A query that matches the cluster centroid sufficiently well triggers a Tier 3 load (a deliberate, on-demand re-promotion event).

### 4. Tier 3 Competitive Displacement Policy (derived)

**Why write-time importance is the wrong quality signal for Tier 3:**

Q003 (germinal center finding) identified competitive displacement as the correct eviction mechanism for a bounded cold store. The Q003 implementation proposed comparing new memory importance against existing pool importance at write time. But in Tier 3, this is wrong because:

Write-time importance is a prediction about future relevance at the moment of initial storage. By the time a memory reaches Tier 3, it has survived session-end consolidation and been assessed as a candidate for archival. The write-time importance score is now stale — it was computed before the memory accumulated any retrieval history.

**The correct quality signal for Tier 3: realized access contribution.**

Define the quality of a Tier 3 memory as:

```
tier3_quality(M) = retrieval_count(M) * recency_weight(M) * cluster_cohesion(M)

where:
  retrieval_count(M)   = number of times M was retrieved across all sessions since storage
  recency_weight(M)    = exponential decay since last retrieval: e^(-lambda * sessions_since_last_access)
  cluster_cohesion(M)  = fraction of M's semantic neighbors also in Tier 3
                         (isolated memories have lower quality than well-connected cluster members)
```

This is a realized signal, not a predicted one. A memory that was predicted to be high-importance at write time but was never actually retrieved has `retrieval_count = 0` and `tier3_quality ≈ 0`. A memory that was predicted to be low-importance but was retrieved 3 times has higher `tier3_quality`.

The decay factor `lambda` calibrates how quickly non-retrieved memories lose their survival advantage. Recommended `lambda = 0.3` per session boundary — after 3 sessions without retrieval, `recency_weight ≈ 0.4`. After 7 sessions, `recency_weight ≈ 0.12`. This is the "soft expiry" for Tier 3 under capacity pressure.

**The cluster_cohesion term:**

This is the key insight not present in Q003's initial formulation. In Tier 3, memories that are members of dense semantic clusters are more defensible than isolated memories, because:

1. A cluster query (where the user asks about a topic that maps to a Tier 3 cluster) benefits all cluster members simultaneously — retrieving one benefits the whole group.
2. An isolated memory in Tier 3 with low cluster_cohesion is effectively orphaned: it cannot be retrieved via cluster-level queries, and its direct-match probability in any given query is lower.

**Derived competitive displacement rule:**

When a new memory arrives at Tier 3 (from Tier 2 consolidation) and Tier 3 is at capacity:

```
candidate_for_eviction = argmin over all Tier 3 memories of tier3_quality(M)
new_memory_quality = tier3_quality(new_memory)  # computed with retrieval_count from session history

If new_memory_quality > candidate_for_eviction quality:
  evict candidate_for_eviction
  insert new_memory
Else:
  new_memory stays in Tier 2 (downgrade: do not archive)
  OR
  trigger Tier 2 cluster merge (absorb new_memory into its nearest Tier 2 cluster to lower pressure)
```

The "downgrade" path (new memory stays in Tier 2 when it can't displace Tier 3 residents) is important: it prevents Tier 3 from becoming a quality-inversion store where low-quality memories accumulate simply because they arrived before high-quality newcomers.

**The relative pressure mechanism (from Q003's germinal center analogy):**

The competitive threshold is not absolute — it is relative to the current pool's quality distribution. If Tier 3 is filled with memories from the first month of the system's operation (all low-quality, all old), a new memory of any quality displaces the weakest member. If Tier 3 has been running for 2 years and contains high-quality cross-session memories, only new memories with strong realized quality can displace residents.

This is the bone-marrow niche mechanism: the carrying capacity of Tier 3 creates selection pressure that automatically elevates pool quality over time.

### 5. Cross-Tier Re-Access Rule (derived)

**The scan pollution problem:**

Suppose a broad historical query touches 50 Tier 3 memories in a single session (e.g., "what do I know about Docker?"). Without a re-insertion rule, all 50 memories would be promoted to Tier 2, potentially flooding Tier 2 with memories that are only relevant to this one broad scan and will not be accessed again.

This is the "sequential scan" problem that ARC solves in the disk cache domain: a sequential scan that accesses many pages exactly once should not displace frequently-used pages from T2.

**Derive the re-insertion rule from the same principles:**

The key insight is: a single retrieval event does not establish frequency. The re-insertion criterion should require evidence of sustained relevance, not one-time access. Concretely:

**Rule 1 — Single Tier 3 retrieval: do NOT promote to Tier 2.**

The memory remains in Tier 3. The retrieval increments `retrieval_count` in the quality score. The memory is surfaced for the current query but is not physically moved.

**Rule 2 — Two or more Tier 3 retrievals in the same session: promote to Tier 2.**

Two retrievals within a session means the memory was useful twice, which exceeds the "scan artifact" probability. This maps to ARC's T1→T2 promotion rule (second access → frequency promotion). The memory is physically moved from Tier 3 to Tier 2 (not copied — Tier 3 record is deleted, Tier 2 record is created).

**Rule 3 — Two or more Tier 3 retrievals across different sessions: promote to Tier 2 with higher priority.**

Cross-session re-access is stronger evidence than within-session re-access. A memory retrieved in session N and session N+2 has demonstrated persistence of relevance. Promote with a head-of-queue insertion in Tier 2 (MRU of T2 in ARC terms, not MRU of T1 — it goes directly to the frequency list because it has proven frequency).

**Rule 4 — Tier 3 cluster promoted: promote the cluster centroid only, not all members.**

When a query matches a Tier 3 cluster (via the cluster manifest in Tier 2), load the cluster members on demand for this query. Only the cluster centroid (the single best-representative member) is promoted to Tier 2 as a standing record. The other cluster members remain in Tier 3. The centroid serves as a "stub" — if queried again, the centroid's Tier 2 presence triggers the full cluster load from Tier 3.

This is analogous to a cache line in CPU architecture: a cache miss brings in the requested word plus its adjacent words in the same cache line (spatial locality), but only the accessed cache line is promoted, not the entire memory region. The cluster centroid is the cache line; the cluster members are the adjacent words that are loaded for this access but not permanently promoted.

**Summary of re-insertion rules:**

```
First Tier 3 retrieval in a query:
  → Return memory, increment retrieval_count, remain in Tier 3

Second Tier 3 retrieval in same session:
  → Promote to Tier 2 (T1 MRU, recency insertion)

Second Tier 3 retrieval in different session from first:
  → Promote to Tier 2 (T2 MRU, frequency insertion — skip the recency list)

Cluster-level Tier 3 match:
  → Load cluster members for this query, promote centroid to Tier 2 (T1 MRU),
     leave cluster members in Tier 3
```

### 6. Cross-Tier Interface (derived)

**The minimum interface between Tier 1 and Tier 2:**

Tier 1 is a pure read-through cache over Tier 2. The interface must support:

```
interface Tier1Cache {
  // Called by retrieval path on every query result
  access(memory_id: UUID) -> CachePosition  // returns: T1_HIT | T2_HIT | B1_GHOST | B2_GHOST | MISS

  // Called after Tier 2 loads a memory for a Tier 1 miss
  load(memory_id: UUID, data: MemoryPayload) -> EvictionCandidate | None

  // Called to retrieve cached data (returns null if not in Tier 1)
  get(memory_id: UUID) -> MemoryPayload | None

  // Called to query current ARC state for diagnostics
  state() -> ArcState  // p, |T1|, |T2|, |B1|, |B2|
}

where EvictionCandidate = { memory_id: UUID, list: T1 | T2 }
// The eviction candidate must be reported to Tier 2 so Tier 2 knows not to expect fast access for this memory
```

Tier 2 does not call into Tier 1. Data flows one direction: Tier 2 → Tier 1 on promotion. Tier 1 reports evictions to Tier 2 as a notification (so Tier 2 can update its metadata — e.g., clearing a "cached" flag — but Tier 2 does not modify its canonical record).

**The minimum interface between Tier 2 and Tier 3:**

Tier 2 is the primary store; Tier 3 is the archive. The interface must support:

```
interface Tier2Store {
  // Primary query path: returns memories ranked by relevance
  query(embedding: Vector, top_k: int, filters: Metadata) -> RankedMemoryList

  // Consolidation path: called at session-end
  get_consolidation_candidates(session_id: UUID) -> MemoryClusterList

  // Tier 3 promotion path: called by consolidation pass when cluster scores below threshold
  archive_cluster(cluster: MemoryCluster) -> ClusterManifest
  // Returns a cluster manifest (centroid + member IDs) stored in Tier 2 as a lightweight stub

  // Tier 3 re-access path: called when a Tier 3 memory qualifies for promotion
  restore_from_archive(memory_id: UUID, data: MemoryPayload) -> void
}

interface Tier3Archive {
  // Write path: called by Tier 2 consolidation pass
  store_cluster(cluster: MemoryCluster) -> void

  // Read path: called on-demand when a Tier 2 cluster manifest is matched
  load_cluster(cluster_id: UUID) -> MemoryList

  // Quality assessment: called by Tier 3 competitive displacement
  get_quality_scores(limit: int, order: ASC) -> [(UUID, float)]
  // Returns bottom-N memories by tier3_quality, for displacement candidates

  // Eviction: called when a new cluster arrives and Tier 3 is at capacity
  evict(memory_id: UUID) -> void
}
```

**The critical design constraint: Tier 1 must not know about Tier 3.**

Tier 1 is a cache over Tier 2. Tier 1's access/load/get interface operates against Tier 2 only. If a Tier 3 memory is promoted to Tier 2 (via re-access Rule 2 or Rule 3), it becomes a Tier 2 memory from Tier 1's perspective — no special handling. This preserves clean layer boundaries and prevents complexity leakage across three tiers.

**The handshake protocol at query time:**

```
Query arrives:
  1. Tier 1 lookup (hash table, O(1))
     → Hit: return cached data, update ARC position
     → Miss: fall through
  2. Tier 2 query (indexed search, 1-10ms)
     → For each result: call Tier 1.access(id) to update ARC state
     → Load result into Tier 1 via Tier 1.load(id, data)
     → Check for Tier 3 cluster manifest matches
  3. If Tier 3 cluster manifest matched:
     → Load cluster from Tier 3 on demand (10-100ms, async)
     → Apply re-access rules (promote centroid to Tier 2 if warranted)
     → Return cluster members as supplementary results
```

## Ideas Extracted (using real Recall stack)

### Idea: arc-ghost-session-clock

- **Novelty**: 0.82 — ARC's ghost lists are well-known in database systems (ZFS, Q001 finding), but calibrating ghost list TTL using a logical session clock (sessions elapsed since eviction) rather than wall-clock time is not implemented in any known system — database or AI memory. Session count as a logical clock for ghost expiry is novel.
- **Evidence**: 0.75 — ARC's ghost list mechanism is peer-reviewed and production-validated (Megiddo & Modha 2003, ZFS). The session-clock adaptation for memory systems is engineering extrapolation from the established mechanism, not independently validated. Evidence score reflects source field strength.
- **Feasibility**: 0.80 — Recall already has session tracking (session-start/stop hooks). Ghost list entries are UUIDs with a session counter. The session counter increments on each `session-start` hook event. Ghost entries are expired when `current_session_count - eviction_session_count > 5`. Implementation: ghost list as a Python dict in the FastAPI process (in-memory, ~25KB for 1000 entries), session counter stored in Redis (INCR on session-start). Total implementation: 1-2 weeks, no new infrastructure.
- **Description**: Implement ARC's ghost lists (B1, B2) in Recall's FastAPI process using memory UUIDs as keys and session-count timestamps as the ghost TTL. When a retrieval call touches a UUID in B1 or B2, adjust ARC's `p` parameter and re-promote the memory into the appropriate ARC list. Ghost entries expire after 5 sessions (configurable). This gives Recall's hot cache the self-tuning recency/frequency balance property, adapting to whether the current workload is session-local (high recency demand) or cross-session (high frequency demand), without any manual configuration.

### Idea: cluster-centroid-manifest-tier3

- **Novelty**: 0.88 — Storing a lightweight cluster centroid manifest in the fast tier (Tier 2/Qdrant) as a stub for a full Tier 3 cluster is not implemented in any known AI memory system. The pattern is analogous to a DNS pointer record or a filesystem inode that references a large data block — a lightweight index entry that enables discovery without loading the full data. This specific pattern for multi-tier memory architecture is novel.
- **Evidence**: 0.70 — The cluster-centroid-as-stub pattern derives from several engineering precedents: DNS indirect records (pointer records for large DNS payloads), database clustered indexes where a non-leaf index node holds only the partition key, and CPU cache line partial loads. None of these are directly from AI memory research; evidence is engineering analogy strength.
- **Feasibility**: 0.65 — Requires: (1) a cluster manifest record type in Qdrant — a special vector entry tagged as `type=cluster_manifest`, containing the centroid embedding, cluster_id, member count, and a reference list of Tier 3 member IDs; (2) a Tier 3 storage location (this is the archive tier — implementable as a separate Qdrant collection with lower index priority, or as a PostgreSQL JSONB table for batch access); (3) cluster detection at consolidation time (angular distance threshold grouping over session memories). Implementation: 3-4 weeks, requires new Qdrant collection and consolidation job changes.
- **Description**: At session-end consolidation, memories scoring below the archive threshold are grouped into semantic clusters by angular distance (threshold: 0.20). Each cluster is archived to Tier 3 (a low-priority Qdrant collection or PostgreSQL JSONB table). A cluster manifest vector — the centroid of the cluster's embeddings — is stored in the primary Tier 2 Qdrant collection with `type=cluster_manifest` metadata and a list of member IDs. When a query matches a cluster manifest (high similarity to the centroid), Recall loads the cluster members from Tier 3 on demand (async, 50-100ms). Only the cluster centroid is promoted to Tier 2 after two accesses; cluster members remain in Tier 3 unless individually accessed multiple times. This makes Tier 3 memories discoverable without loading them into real-time retrieval.

### Idea: realized-quality-tier3-displacement

- **Novelty**: 0.85 — Q003 identified competitive displacement as the correct Tier 3 eviction mechanism, but used write-time importance as the quality signal. Using realized access contribution (retrieval count × recency decay × cluster cohesion) as the quality signal — a signal that can only be computed after the memory has been in the archive — is novel even relative to Q003's finding. No production system computes a quality score from realized access history for archive-tier eviction.
- **Evidence**: 0.65 — The realized quality formula derives from first principles in information theory (a memory's value is proportional to its mutual information with future queries, estimated by past retrieval frequency) and the Q003 immunological mechanism (competitive niche displacement). The cluster cohesion term is engineering reasoning about relational value in retrieval contexts. No direct peer-reviewed validation of this specific formula.
- **Feasibility**: 0.60 — Requires: (1) retrieval count tracking per memory in PostgreSQL (already partially present as `retrieval_count` in Recall's schema); (2) sessions-since-last-access counter (computable from session log and memory metadata); (3) cluster cohesion computation at displacement time (query Tier 3 for the eviction candidate's nearest neighbors — one indexed lookup); (4) competitive displacement check on each new Tier 3 write. Main complexity: cluster cohesion query adds latency to the Tier 3 write path. Mitigatable by caching cohesion scores with a 24-hour TTL. Implementation: 3-5 weeks.
- **Description**: Recall's Tier 3 eviction policy replaces write-time importance with a realized quality score: `quality = retrieval_count × e^(-0.3 × sessions_since_last_access) × cluster_cohesion`. This score is computed at eviction decision time (when Tier 3 is at capacity and a new archive write arrives). The memory with the minimum realized quality is displaced. A memory with retrieval_count = 0 and sessions_since_last_access = 10 has quality ≈ 0.05 regardless of its original importance score — it is the correct eviction candidate. A memory with retrieval_count = 5 and recent access is strongly protected. Cluster cohesion (fraction of the memory's angular neighbors also in Tier 3) rewards cluster members and penalizes isolated memories. Implemented as a PostgreSQL query on the memories table with indexed retrieval_count and last_accessed_session fields.

### Idea: session-boundary-consolidation-unit

- **Novelty**: 0.78 — Triggering Tier 2 → Tier 3 consolidation at session-end (rather than on a timer or capacity threshold) and consolidating at the cluster level (not individual memory level) is a novel combination. The session-end trigger alone is the Q002 SWR finding; the cluster-as-unit aspect is the addition from this derivation. Together they form a concrete architectural decision not implemented in any known system.
- **Evidence**: 0.72 — Session-end as the consolidation trigger maps to the biological SWR replay window (Q002, validated evidence). Cluster-as-unit consolidation is engineering reasoning about relational value in retrieval networks (analogous to how operating systems evict dirty page clusters rather than individual pages in certain VM implementations). Combined evidence is moderate.
- **Feasibility**: 0.75 — Recall already has `recall-session-summary.js` firing at session-end via the Stop hook. The consolidation job attaches to this hook as a `POST /consolidate` FastAPI call. The consolidation algorithm: (1) pull session memories from PostgreSQL by session_id; (2) compute pairwise angular distances in Qdrant embedding space to identify clusters (use DBSCAN-style threshold grouping — no need for exact ANN, just group memories within angular distance 0.20); (3) score each cluster; (4) archive below-threshold clusters to the Tier 3 collection with cluster manifest creation. Implementation: 3-4 weeks (builds on Q002's `swr-offline-consolidation-batch` idea, adding cluster semantics and Tier 3 manifest creation).
- **Description**: Recall's session-end hook triggers a consolidation job that groups the session's memories into semantic clusters (angular distance < 0.20), scores each cluster by `retrieval_count / (1 + sessions_since_formation)`, and archives below-threshold clusters (score < 0.15) to the Tier 3 collection as a unit. A cluster manifest is written to the primary Qdrant collection for each archived cluster. Above-threshold clusters remain in Tier 2 with updated importance scores. This replaces the current individual-memory consolidation approach with a cluster-aware approach that preserves relational context during archival.

### Idea: two-retrieval-promotion-gate

- **Novelty**: 0.80 — The specific rule "do not promote a Tier 3 memory to Tier 2 on first retrieval; require two retrievals (same or different sessions) to trigger promotion" is a direct application of LRU-K (K=2) scan resistance logic to the cross-tier promotion problem. This prevents scan pollution in the deep retrieval path. Not implemented in any known AI memory or retrieval system in this exact form.
- **Evidence**: 0.82 — LRU-K's scan resistance property (O'Neil, O'Neil, Weikum 1993) is peer-reviewed and production-validated in IBM DB2. The K=2 threshold (second access triggers frequency promotion) is the canonical recommendation from the original paper and directly applicable to the tier promotion problem. The translation from page cache to memory tier promotion is straightforward.
- **Feasibility**: 0.85 — Requires: (1) a "Tier 3 access counter" per memory stored in PostgreSQL — a single integer column `tier3_access_count` that increments on each Tier 3 retrieval; (2) a promotion threshold check in the retrieval path: if `tier3_access_count >= 2`, trigger Tier 2 re-insertion; (3) a session-boundary check: if the two accesses span different sessions, promote to T2 MRU (frequency list); if same session, promote to T1 (recency list). Implementation: 1-2 weeks, minimal schema change and retrieval path modification.
- **Description**: Recall's cross-tier promotion rule requires two Tier 3 retrievals before a memory is promoted to Tier 2. On first Tier 3 retrieval: increment `tier3_access_count` in PostgreSQL, return the memory for this query, do not physically move it. On second retrieval: promote to Tier 2. If both retrievals are within the same session, insert into Tier 2 at recency position (T1 MRU in ARC). If retrievals span different sessions, insert into Tier 2 at frequency position (T2 MRU in ARC, bypassing the recency probationary period). This prevents a single broad historical query from flooding Tier 2 with 50 Tier 3 memories that will not be accessed again — only memories with demonstrated repeated relevance earn Tier 2 residency.

## Suggested Follow-ups

- **[PHYSICS] Q029**: What is the latency overhead of the cluster cohesion computation in the realized-quality-tier3-displacement policy? At Tier 3 size of 500K memories, the cohesion query (find angular neighbors of an eviction candidate) must complete within the consolidation job's acceptable window. Measure: (a) Qdrant search latency at 500K vectors, low-priority collection; (b) total consolidation job latency budget per session-end event. Falsifiable: if cohesion query exceeds 500ms, the displacement policy must precompute and cache cohesion scores.

- **[ADVERSARIAL] Q030**: The derived cross-tier interface forbids Tier 1 from knowing about Tier 3. An adversarial challenge: is this tier isolation constraint correct? A query routing layer that knows which tier a memory lives in could optimize retrieval order (check Tier 1, then Tier 2 manifest, then skip Tier 3 unless explicitly requested). The adversarial position: the clean abstraction is actually a performance liability — the retrieval path should be aware of tier membership and make explicit routing decisions. Defender: tier isolation enables independent scaling of each tier. This tradeoff should be explicitly evaluated.

- **[ABSENCE] Q031**: Does any production retrieval-augmented system implement a cluster manifest pattern (lightweight index stub referencing a full data cluster in a slower store)? Check: hierarchical navigable small world (for its layer structure), document store systems with tiered storage (Elasticsearch ILM policies), and columnar databases with partition pruning. The cluster-centroid-manifest-tier3 idea may have a close prior in these systems.

- **[TIMESHIFTED] Q032**: The two-retrieval-promotion-gate uses K=2 because LRU-K at K=2 is the canonical recommendation. In 2032, is K=2 still the right threshold, or has session data revealed a different K? Specifically: what fraction of Tier 3 retrievals are "scan artifacts" (never retrieved again after the promoting query) vs. "genuine re-accesses" (retrieved again within 5 sessions)? If the artifact fraction is > 80%, K=2 is too low (should be K=3 or K=4). If the artifact fraction is < 20%, K=1 is sufficient (LRU, standard behavior). The 2026 implementation should log tier3_access_count distributions to enable this calibration.
