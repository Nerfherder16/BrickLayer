# Finding: Q069 — Deployment Sequencing of Three BREAKTHROUGH Implementations Relative to Q059 Schema Migration

**Question**: Three Wave 7 BREAKTHROUGH findings require implementation in the same codebase. The Q059 schema migration defines 5 steps. What is the correct sequencing of all three BREAKTHROUGHs relative to the 5 migration steps, and do any of the three BREAKTHROUGHs require schema changes not covered by Q059's migration plan?
**Question Type**: CONVERGENCE
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Distributed systems deployment, schema migration safety, rolling deploy sequencing

---

## Dependency Analysis

### Q054 (Phase 0 metadata-first tombstone)

**Schema requirements**: Q054 proposes inserting a Phase 0 step into the session-end hook that writes tombstones to Redis before Op1 deletes from PostgreSQL and Qdrant. The tombstone operation writes structured values to the existing `scope_registry` Redis hash. The reverse UUID index (`recall:uuid_to_scope:{uuid}` as individual Redis keys, or `uuid_scope_index` as a PostgreSQL table per Q059 Step 3) must be queryable to identify which scope keys correspond to the candidate_ids identified in Phase 0.

Q054's Phase 0 requires:
1. The reverse UUID index (Redis individual keys `recall:uuid_to_scope:{uuid}` as described in Q054's Op0 implementation, or the PostgreSQL `uuid_scope_index` table from Q059 Step 3). Both are reverse-lookup structures; Q054 uses Redis keys and Q059 Step 3 uses a PostgreSQL table. These are equivalent; Q059 Step 3 is the migration plan's mechanism.
2. The tombstone value format (Q059 Step 4): the tombstone write in Phase 0 uses the same `TOMBSTONED:...` structured string as the tombstone writer in Q059 Step 4.3.
3. No new PostgreSQL columns beyond what Q059 already defines.

**Direct schema gaps introduced by Q054**: none. Q054 requires the reverse UUID index (Q059 Step 3) and the tombstone writer format (Q059 Step 4.3), both already in Q059's plan. Q054 does NOT require any new PostgreSQL columns.

**Dependency on other BREAKTHROUGHs**: Q054 is independent of Q062. It references Op4 elimination from Phase 2, which is a code-only change in the session-end hook. It does not depend on Q056.

### Q056 (write-tier partition policy — local pending queue)

**Schema requirements**: Q056 adds a local filesystem pending queue at `~/.claude/recall-pending/session-{ISO-timestamp}.json`. This is entirely a hook-file change (JavaScript). No PostgreSQL schema changes. No Redis schema changes. No FastAPI changes. No new infrastructure.

Q056's implementation touches:
1. `recall-session-summary.js` (Stop hook): adds a two-phase write with local file fallback.
2. `recall-retrieve.js` (UserPromptSubmit hook): adds a pre-flight check for pending files and retry logic.

**Direct schema gaps introduced by Q056**: none. The pending queue is a local filesystem directory. Q059's migration plan does not cover it, but Q059 does not need to — there is no database schema involved.

**Dependency on other BREAKTHROUGHs**: Q056 is fully independent of Q054 and Q062. It does not touch PostgreSQL or Redis schema. It can be deployed before, after, or concurrently with any migration step.

### Q062 (epoch-clock — replaces Op2 O(N) bulk UPDATE)

**Schema requirements**: Q062 requires exactly one schema change:
- `ALTER TABLE memories ADD COLUMN last_access_epoch INTEGER NOT NULL DEFAULT 0`

This is precisely Q059 Step 2. Q062 also requires a B-tree index on `last_access_epoch`:
- `CREATE INDEX memories_last_access_epoch_idx ON memories (last_access_epoch ASC)`

Q059 Step 2 defines the column addition but does not explicitly mention the index. This is a gap: Q059 Step 2 is necessary but not sufficient for Q062. The index is required for the routing query `WHERE last_access_epoch >= (:current_epoch - tau)` to use a B-tree range scan rather than a sequential scan. Without the index, the epoch predicate works correctly but at O(N) scan cost instead of O(log N + K) index cost.

Q062 also requires:
- Redis key `recall:global_epoch` (auto-created on first `INCR`, no explicit migration step needed).
- A one-time data migration to backfill existing rows: `UPDATE memories SET last_access_epoch = :current_epoch - sessions_since_last_access`. Q059 Step 2 describes this backfill.

**Direct schema gaps introduced by Q062**: the B-tree index on `last_access_epoch` is not explicitly defined in Q059 Step 2. Every other Q062 requirement is covered.

**Dependency on other BREAKTHROUGHs**: Q062 is independent of Q054 and Q056. The epoch INCR at session end is a new Op2 behavior that sits alongside the tombstone Phase 0 (Q054) without conflict. The two session-end changes are in different phases and different systems (Redis epoch counter vs. Redis scope registry).

## Schema Gap Analysis

### Does Q059's 5-step migration plan cover all three BREAKTHROUGHs?

**Q054 (Phase 0 tombstone)**: Fully covered. Requires Q059 Step 3 (reverse UUID index) and Step 4 (tombstone value format). No new schema steps needed.

**Q056 (write-tier partition policy)**: Not covered by Q059 — but Q059 does not need to cover it. Q056 is a hook-only change with no database schema. Q059's scope is database schema. The gap is a classification gap, not a coverage gap.

**Q062 (epoch-clock)**: Partially covered. Q059 Step 2 adds the `last_access_epoch` column and defines the backfill. Q059 does NOT define the B-tree index on `last_access_epoch`. This is a genuine omission.

**Hypothesis verdict**: The hypothesis — "Q059's migration plan is complete and no additional steps are needed for the three BREAKTHROUGHs" — is **PARTIALLY REFUTED** on one narrow point.

Q059 is complete for Q054 and Q056. Q059 is incomplete for Q062: the `CREATE INDEX memories_last_access_epoch_idx ON memories (last_access_epoch ASC)` step is missing from the migration plan. Without this index, Q062's routing queries degrade from O(log N + K) to O(N) sequential scans — which is the same cost as the old sessions_since_last_access system. The epoch-clock provides correct results without the index, but not the 5-200x speedup that is the entire justification for the BREAKTHROUGH rating.

The missing index step should be appended to Q059's Step 2 or inserted as a new Step 2b. It is a single DDL command, does not require a maintenance window (PostgreSQL CREATE INDEX CONCURRENTLY can run online), and does not affect running queries during construction.

**Additional migration step required:**

```sql
-- Step 2b (missing from Q059): B-tree index for epoch routing queries
CREATE INDEX CONCURRENTLY memories_last_access_epoch_idx
  ON memories (last_access_epoch ASC);
```

This step must run after Q059 Step 2 (the column must exist before the index can be created) and before Q062 app code is deployed to use the epoch routing query in production (otherwise the routing query runs without index benefit during the deployment window).

## Verified Hypothesis

**The hypothesis states**: Q054 requires no new schema; Q056 requires no schema; Q062 requires `last_access_epoch` column (Q059 Step 2). Therefore Q059's migration plan is complete and no additional steps are needed.

**Verdict: PARTIALLY REFUTED.**

- Q054 requires no new schema beyond Q059 Steps 3 and 4. **Confirmed.**
- Q056 requires no schema at all. **Confirmed.**
- Q062 requires `last_access_epoch` column (Q059 Step 2). **Confirmed, but incomplete.**

The gap: Q062 requires a B-tree index on `last_access_epoch` (the `CREATE INDEX CONCURRENTLY memories_last_access_epoch_idx` step). Q059 Step 2 defines the column but does not define the index. Without the index, Q062 is functionally correct but delivers no speedup — the routing query performs a sequential scan at O(N) cost, identical to the old system's behavior. The BREAKTHROUGH designation for Q062 is predicated on the 5-200x speedup, which is only achieved with the index.

This is a documentation gap in Q059, not a design flaw. The fix is a single DDL line added to Step 2 (Step 2b above). The index can be created online with `CONCURRENTLY` and does not require a maintenance window.

No other schema gaps exist. Q059 is otherwise complete for the three BREAKTHROUGHs.

## Code-Before-Schema Safety Summary

| BREAKTHROUGH | Code-before-schema result |
|-------------|--------------------------|
| Q054 (Phase 0 tombstone writer) | INCORRECT BEHAVIOR if deployed before Q059 Step 7 (tombstone reader update). Old instances receive unparseable Redis values → store errors or garbled UUIDs. Mitigation: deploy Step 7 first, verify full rollout, then deploy Q054/Step 8. |
| Q056 (local pending queue) | SAFE. No schema dependency. Deploy at any time. |
| Q062 (epoch-clock) | DEGRADED ONLY (not incorrect) if deployed before Step 2b index. Routing queries work correctly, just at O(N) cost instead of O(log N). Step 9 before Steps 2/2b produces column-not-found errors — do not deploy Step 9 before Steps 2 and 2b. |

The only genuinely dangerous sequencing violation is deploying Q054's tombstone writer (Step 8) before the tombstone reader update (Step 7) is confirmed on all instances. This is the sole CRITICAL ordering constraint in the deployment sequence.

### Idea: breakthrough-convergence-deployment-graph
- **Novelty**: 0.72
- **Evidence**: 0.80
- **Feasibility**: 0.95
- **Description**: When multiple BREAKTHROUGH findings target the same codebase and share schema dependencies, the deployment risk concentrates at a single point: the tombstone reader-before-writer sequencing constraint (Q054/Q059 Step 4). All other steps in the 10-step sequence are either safe code-before-schema (degraded performance, correct results) or fully independent (Q056). A deployment graph that marks only Step 7→Step 8 as a hard sequencing gate — with full rollout verification between them — reduces the coordination burden for the entire 10-step deployment to a single synchronization point. This pattern generalizes: multi-BREAKTHROUGH deployments should identify their single critical ordering gate and treat all other steps as order-flexible.

### Idea: index-completeness-audit-for-schema-migration-plans
- **Novelty**: 0.68
- **Evidence**: 0.82
- **Feasibility**: 0.97
- **Description**: Schema migration plans that add new columns for routing queries frequently omit the index creation step, because the column and the index are conceptually separate DDL commands. The Q059 plan defines the `last_access_epoch` column but omits the B-tree index — a gap that makes Q062 correct-but-slow rather than correct-and-fast. A lightweight audit rule for migration plans: for every new column that appears in a WHERE clause of a routing query, verify the migration plan includes a corresponding `CREATE INDEX` step. This rule catches the Q059 gap mechanically. For Recall specifically, the audit applies to `last_access_epoch`, `distinct_sessions_accessed` (if it ever appears in a range query), and any future epoch-derived columns.

## Key Finding

The hypothesis that Q059's migration plan is complete for all three BREAKTHROUGHs is correct in substance but incorrect on one specific point: Q059 Step 2 adds the `last_access_epoch` column required by Q062 but omits the `CREATE INDEX CONCURRENTLY memories_last_access_epoch_idx` step that enables the B-tree range scan. Without this index, Q062 delivers correct epoch semantics but no performance improvement — nullifying the BREAKTHROUGH's core value proposition. The fix is a single DDL line (Step 2b) that can be added to Q059's migration plan or executed as a follow-on step.

The 10-step deployment sequence has exactly one CRITICAL ordering gate: Step 7 (tombstone reader update, full rollout confirmed) must precede Step 8 (tombstone writer, Q054 Phase 0). All other steps are either code-before-schema safe (degraded performance, correct results) or fully independent of schema (Q056 local pending queue, which can be deployed at any point in the sequence). The three BREAKTHROUGHs do not conflict with each other's deployment: Q054 and Q062 both modify the session-end hook but in different phases (Phase 0 tombstone vs. Phase 2 epoch INCR), and Q056 modifies different hooks entirely.
