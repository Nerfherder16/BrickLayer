# Finding: Q117 — S4+S5 Co-Alert Diagnostic Matrix

**Question**: What is the correct health signal matrix entry for the co-alert pattern S4=CRITICAL + S5=HEALTHY — specifically, does this co-alert unambiguously indicate Redis restart with data loss (as Q106 claims) in all cases, or can it arise from other failure modes (e.g., PostgreSQL epoch column anomaly, dual-machine clock skew, or a PG vacuum that updates `last_access_epoch` in batch), and should the Q084 correlation matrix be updated with a new row for this pattern?
**Question Type**: CALIBRATION
**Verdict**: INCREMENTAL
**Severity**: Medium
**Source field**: distributed systems fault isolation, Redis persistence failure modes, PostgreSQL maintenance operations, dual-machine write skew, epoch-clock invariant analysis

---

## Context

Q106 ends with this table row and follow-up:

> S4=CRITICAL, S5=HEALTHY is the expected signal pattern for Redis restart with data loss. Update Q084's correlation matrix to include this specific co-alert pattern.

Q084 established that:
- S4 fires when `redis_epoch < pg_max` by more than tau=20 (retrograde gap — Redis fell behind PG)
- In Direction 1 (Redis reset to epoch=0), S5 evaluates the Cypher predicate `WHERE neighbor.last_access_epoch >= (0 - 20) = -20` — trivially true for all nodes — so S5 reports 100% compliance and stays HEALTHY
- S4+S5 co-alert (both non-healthy) was claimed to indicate the PG-restore direction (Direction 2) specifically

Q106's claim is more precise: **S4=CRITICAL, S5=HEALTHY is the fingerprint for Redis restart with data loss.** This calibration finding evaluates whether that fingerprint is truly unambiguous — and specifically whether the three failure modes named in the question (PG epoch column anomaly, dual-machine clock skew, PG vacuum batch update) produce the same pattern.

---

## Signal Definitions (Disambiguation)

Before evaluating alternative failure modes, the correct S4 trigger condition must be established. Q106 and Q084 both use the epoch-gap definition:

**S4**: fires when `redis_epoch < pg_max` (retrograde direction) with gap > tau=20. This is NOT a PG-to-Neo4j orphan count check — that is a separate neo4j_orphan_count probe from Q081/Q092.

**S5**: fires when > 20% of 1-hop BFS results have `last_access_epoch < current_epoch - tau` over 7-day window. With `current_epoch = 0`, the threshold collapses to -20, and all BFS candidates trivially pass — S5 reports 100% compliance.

This distinction is load-bearing. Several proposed failure modes only appear to produce S4=CRITICAL if S4 is misread as a PG-to-Neo4j consistency signal.

---

## Evidence

### The Core Mechanism (Q106 Is Correct for the Primary Case)

The logic behind Q106's claim is sound for the primary failure mode:

1. Redis restarts without AOF persistence → `recall:global_epoch` disappears
2. Client-side GET returns nil → coerced to 0 in Python
3. Working-set filter threshold: `0 - 20 = -20`; every stored `last_access_epoch >= 0 >= -20`, so all PG memories pass
4. S4 fires CRITICAL: `redis_epoch (0) << pg_max (850)`, gap=850 >> tau=20
5. S5 Cypher predicate evaluates against `current_epoch - tau = 0 - 20 = -20`; every graph neighbor passes; S5 = HEALTHY

This mechanism is logically correct. The S5=HEALTHY behavior during Redis epoch=0 is a direct mathematical consequence of S5 using the **current (reset) epoch** as its reference, not a stored pre-failure reference.

---

### Failure Mode Inventory

The following failure modes are evaluated against the correct S4 definition to determine whether each independently produces exactly S4=CRITICAL + S5=HEALTHY.

---

#### FM-1: Explicit Redis Key Deletion or Zeroing (Not a Full Restart)

**Mechanism**: `redis-cli DEL recall:global_epoch`, `redis-cli SET recall:global_epoch 0`, `redis-cli FLUSHDB`. The epoch key is absent or zero without a Redis process restart.

**Signal behavior**:
- S4: `redis_epoch = 0`, `pg_max = 850`, gap = 850 >> tau. S4 fires CRITICAL.
- S5: With `current_epoch = 0`, Cypher threshold = -20; all Neo4j nodes pass. S5 = HEALTHY.

**Verdict**: FM-1 produces **identical signal behavior** to a Redis restart. The S4=CRITICAL + S5=HEALTHY fingerprint does not distinguish "accidental key deletion" from "Redis restart without AOF." The recovery action is identical for both (set Redis epoch to pg_max + 1), so this ambiguity does not affect remediation. It matters only for post-incident analysis (AOF enforcement vs access control).

**Impact on Q106's unambiguity claim**: Weak. The fingerprint correctly narrows to "Redis epoch was zero or absent" — an accurate description of both FM-1 and Redis restart. The distinction is sub-classification within the same failure family, not a separate root cause requiring different treatment.

---

#### FM-2: Partial AOF Recovery (Epoch Restored to Non-Zero Historical Value)

**Mechanism**: Redis restarts, AOF is enabled but the log is corrupted or truncated. Redis replays the AOF to epoch 200 (out of 850) and stops. `recall:global_epoch` is restored to 200, not 850. PG still holds memories with `last_access_epoch` up to 850.

**Signal behavior**:
- S4: `redis_epoch = 200`, `pg_max = 850`, gap = 650 >> tau=20. S4 fires CRITICAL. Same retrograde direction as full restart, lower magnitude.
- S5: With `current_epoch = 200`, Cypher threshold = `200 - 20 = 180`. Neo4j nodes with `last_access_epoch >= 180` pass. In a mature graph where most BFS neighbors are recently accessed, the majority will have `last_access_epoch >> 180` and pass. S5 likely stays HEALTHY unless the graph is specifically anchored to memories in the epoch 0-180 range.

**Verdict**: FM-2 produces S4=CRITICAL + S5=HEALTHY (or possibly S5=WARNING at the boundary). The critical difference from FM-1: `redis_epoch = 200`, not 0. The recovery action is identical (set Redis epoch to pg_max + 1 = 851), and the outcome is safe. This sub-case is benign with respect to Q106's claim — the standard remediation still applies.

**Impact on Q106's unambiguity claim**: Negligible. FM-2 is the same recovery path as Redis restart; it is a severity-attenuated version of the same failure.

---

#### FM-3: PostgreSQL `last_access_epoch` Column Inflated by Bulk Write or Buggy Hook

**Mechanism**: A maintenance operation or buggy hook writes a future epoch to `last_access_epoch`. Examples: a migration script runs `UPDATE memories SET last_access_epoch = 9999 WHERE importance > 0.5`; or a hook bug writes `last_access_epoch = redis_epoch + 1000` instead of `redis_epoch`. PostgreSQL `pg_max` becomes 9999 while Redis remains at 850.

**Signal behavior**:
- S4: `redis_epoch = 850`, `pg_max = 9999`, gap = 9149 >> tau. S4 fires CRITICAL in the **same retrograde direction** (redis_epoch < pg_max) as Redis restart. Same alarm code.
- S5: With `current_epoch = 850`, Cypher threshold = `850 - 20 = 830`. The Neo4j nodes still hold their original `last_access_epoch` values (the bulk UPDATE touched PG but not Neo4j). Most BFS neighbors with recent access (epoch >= 830) pass. S5 = HEALTHY.

**Verdict**: FM-3 produces S4=CRITICAL + S5=HEALTHY. However, this failure mode requires the **OPPOSITE recovery action** from Redis restart:

| Observable | Redis Restart (FM-1) | PG Epoch Inflation (FM-3) |
|---|---|---|
| `redis_epoch` raw value | 0 (or 1-5 in early recovery) | 850 (correct, large) |
| `pg_max` raw value | 850 (correct) | 9999 (inflated) |
| Gap direction | redis_epoch < pg_max | redis_epoch < pg_max |
| Gap magnitude | 850 | 9149 |
| S4 alarm code | REDIS_RESET | REDIS_RESET (same code) |
| S5 | HEALTHY | HEALTHY |
| **Correct repair** | **SET redis_epoch = pg_max + 1 = 851** | **DO NOT advance Redis; rollback PG inflation** |

Applying the Q106/Q077 standard remediation (SET redis_epoch = pg_max + 1 = 10000) to FM-3 would catastrophically advance the working-set window to epoch 10000, making every stored memory appear stale (all `last_access_epoch <= 850` would fail the filter `>= 10000 - 20 = 9980`). Essentially the entire memory corpus would vanish from retrieval.

**Discriminant**: The absolute value of `redis_epoch` separates FM-3 from FM-1/restart at the raw-value level. Redis restart always produces `redis_epoch` near zero (0 to tau=20). PG epoch inflation leaves `redis_epoch` at its correct large value. The S4 signal fires identically in both cases (same alarm code, same direction), but the discriminant is `redis_epoch` absolute value — not the gap magnitude.

**Impact on Q106's unambiguity claim**: HIGH. FM-3 is a genuine ambiguity gap. The S4=CRITICAL + S5=HEALTHY fingerprint does not distinguish FM-3 from FM-1/restart without inspecting `redis_epoch` absolute value. The recovery actions are opposite. Q106's "unambiguous" characterization must be qualified.

---

#### FM-4: Standard Dual-Machine Clock Skew (casaclaude + proxyclaude)

**Mechanism**: Q079 and Q084 note that dual-machine deployments both increment `recall:global_epoch` via Redis INCR at session end. Under normal operation the S4 gap should not exceed ~50. A typical skew: proxyclaude has a network partition to Redis for several sessions, then reconnects; `pg_max` may advance slightly ahead of `redis_epoch` due to write ordering.

**Signal behavior analysis**:
- Standard dual-machine skew produces gaps of 2-10 epochs, well below tau=20. S4 does NOT fire.
- Even at the extreme (30+ sessions with Redis unreachable to one machine), the gap reaches 30-50 and may trigger S4 WARNING/CRITICAL. But in this scenario `redis_epoch` is a large healthy number (e.g., 880), and `pg_max` is slightly larger (e.g., 900-920). The gap is retrograde but small.
- S5: With `redis_epoch = 880`, threshold = 860. Recent BFS neighbors pass. S5 = HEALTHY.

**Does this produce S4=CRITICAL + S5=HEALTHY?** Only at the boundary. At gap=50, it triggers S4 at the alarm threshold, but `redis_epoch` is large (not near zero). This is the FM-3-adjacent scenario (large redis_epoch, retrograde gap). The recovery for this scenario is NOT the standard epoch advance — it is investigating why one machine's PG writes are using a different epoch source.

**Verdict**: Standard clock skew does NOT produce S4=CRITICAL (it stays below the threshold). Extreme clock skew produces a borderline S4 alarm with large `redis_epoch` (not near zero), which falls into the FM-3 discriminant class. The `redis_epoch` absolute value discriminant correctly separates extreme clock skew from Redis restart.

**Impact on Q106's unambiguity claim**: Moderate. Extreme dual-machine clock skew is one path into the FM-3-class (large redis_epoch, retrograde gap) that Q106's characterization does not distinguish from Redis restart.

---

#### FM-5: PG VACUUM / AUTOVACUUM Impact on `last_access_epoch`

**Mechanism**: The Q117 question specifically raises "a PG vacuum that updates `last_access_epoch` in batch." Standard PG VACUUM, AUTOVACUUM, VACUUM ANALYZE, and CLUSTER operations do NOT modify row values — they reclaim dead tuple space and update statistics. No standard PostgreSQL maintenance operation alters integer column values.

**Can AUTOVACUUM trigger FM-3?** Only if a custom trigger is defined on the memories table that fires during vacuum. Standard PostgreSQL vacuum does not execute DML triggers during heap cleanup. In the absence of non-standard trigger configurations, PG vacuum cannot inflate `last_access_epoch`.

**Verdict**: PG vacuum does not produce S4=CRITICAL + S5=HEALTHY under standard configuration. This specific concern from the question framing is not a realistic failure mode for a normal PostgreSQL deployment. The question was correct to raise it as a candidate, but it does not survive scrutiny.

---

#### FM-6: Redis Key TTL Expiry

**Mechanism**: If `recall:global_epoch` had a TTL set accidentally (via `EXPIRE` or `SET ... EX ...`), the key could expire silently without any Redis restart. The key disappears; GET returns nil → 0; INCR recreates it at 1.

**Signal behavior**: Identical to FM-1 (key deletion): `redis_epoch = 0`, `pg_max = 850`, S4=CRITICAL, S5=HEALTHY. Recovery is the same.

**Verdict**: FM-6 is another path to the identical signal state with the identical recovery action. It is a sub-variant within the Redis epoch loss family (FM-1/FM-5/FM-6 are operationally the same: "epoch key absent or zero").

---

#### FM-7: PG Restore from Old Backup (Direction 2 — The Inverse)

Q084 analyzed this extensively. PG restore from old backup causes Direction 2 (`redis_epoch >> pg_max`), triggering S4 via the bidirectional guard as `PG_RESTORE_SUSPECTED`. In this scenario, S5 also fires (Neo4j retains post-backup epoch values pointing to memories absent from the restored PG, failing the BFS compliance check). FM-7 produces **S4=CRITICAL + S5=WARNING/CRITICAL**.

**Verdict**: FM-7 does NOT produce S4=CRITICAL + S5=HEALTHY. This is consistent with Q084's finding that S4+S5 co-alert (both non-healthy) indicates PG-restore direction. FM-7 does not confound the pattern under investigation.

|---|---|---|---|---|
| FM-1: Redis key deletion / FLUSHDB / TTL expiry | CRITICAL | HEALTHY | Yes | Yes — identical |
| FM-2: Partial AOF recovery (epoch=200, pg_max=850) | CRITICAL | HEALTHY (likely) | Yes | Yes — identical |

| FM-7: PG restore from old backup | CRITICAL (Direction 2) | WARNING/CRITICAL | No — S5 also alerts | — |

### The Critical Ambiguity: FM-3 and the Discriminant

FM-3 is the only alternative failure mode that:
1. Produces S4=CRITICAL + S5=HEALTHY (identical fingerprint)
2. Requires a different recovery action (correcting PG, not advancing Redis)
3. Would be made **worse** by applying the standard Q106/Q077 remediation

The discriminant separating FM-3 from FM-1/restart is the absolute value of `redis_epoch`:
- `redis_epoch` near zero (0 to tau=20) → Redis restart / key loss → safe to auto-repair
- `redis_epoch` large (>> tau) with elevated `pg_max` → PG inflation or extreme clock skew → investigation required before any repair

This discriminant is not captured by the current S4 alarm code (`REDIS_RESET` fires identically regardless of `redis_epoch` absolute value). It is not a new signal — it is a sub-classification within the existing S4 data, requiring only a branch on `redis_epoch` value at alarm time.

## Revised Q084 Correlation Matrix Entry for S4+S5

The Q084 matrix should gain two new sub-rows for the S4=CRITICAL + S5=HEALTHY pattern (previously it only listed S4+S5 as a co-alert in the Direction 2 / PG-restore scenario):

| Pattern | Implied Root Cause | Unambiguous? | Automated Recovery | False Positive Rate |
|---|---|---|---|---|
| S4=CRIT + S5=HEALTHY + `redis_epoch` ≈ 0 (0–tau) | Redis restart, key deletion, TTL expiry, FLUSHDB (FM-1/FM-5/FM-6) | **Yes** | SET `recall:global_epoch` = pg_max + 1. Safe. | < 1% |
| S4=CRIT + S5=HEALTHY + `redis_epoch` large (>> tau) | PG epoch column inflation (buggy hook, migration) or extreme dual-machine clock skew (FM-3/FM-4) | **No** — investigation required | None until PG epoch history reviewed. **Do NOT advance Redis.** | ~5% for wrong subtype classification |
| S4=CRIT (Direction 1) + S5=HEALTHY + S3=CRIT | CasaOS restart with Redis data loss and pending queue (FM-8 / Q084 Rule 2) | **Yes** — three-signal pattern | SET epoch, then force-replay S3 queue (Q084 Rule 2, sequenced) | ~5% |
| S4=CRIT (Direction 2, S4b) + S5=WARN/CRIT | PG restore from old backup + Neo4j inconsistency (Q084 Direction 2) | Mostly yes | Operator decision (Options A/B/C, Q074) | 15–30% for Option A automation |

## Correction to Q106

Q106 states: "S4=CRITICAL, S5=HEALTHY is the expected signal pattern for Redis restart with data loss."

This should be revised to:

> S4=CRITICAL with `redis_epoch` near zero (0 to tau=20) AND S5=HEALTHY is the unambiguous fingerprint for Redis epoch loss (restart without AOF, key deletion, TTL expiry). S4=CRITICAL with `redis_epoch` large AND S5=HEALTHY is an ambiguous pattern that may indicate PG epoch column inflation from a bulk write or buggy hook, and **must NOT trigger the standard auto-repair without first checking `redis_epoch` absolute value**. The discriminant is `redis_epoch` absolute value, not gap magnitude alone.

Q106's core finding (the mechanism, the ghost resurrection analysis, the monotonic guard recommendation) remains fully valid. This calibration adds one operational safeguard: the auto-repair path must be gated on `redis_epoch` near-zero, not on gap direction alone.

## Ideas Extracted

### Idea: redis-epoch-absolute-value-discriminant
- **Novelty**: 0.72 — Q106 and Q084 characterize S4 (Direction 1) by the *gap* between `redis_epoch` and `pg_max` but do not use the absolute value of `redis_epoch` as a discriminant. The observation that `redis_epoch` near zero is diagnostic for restart-with-full-loss while a large `redis_epoch` with elevated `pg_max` indicates PG inflation is a new diagnostic dimension within the existing S4 alarm. It requires no new signal — only a sub-classification within the existing S4 CRITICAL alarm — but it separates two scenarios that require opposite recovery actions.
- **Evidence**: 0.72 — The argument is analytical: FM-3 (PG epoch inflation) necessarily leaves `redis_epoch` at its pre-inflation value (a large number), while Redis restart necessarily sets `redis_epoch` to 0 or near-zero. This distinction is clean at the extremes and ambiguous only for FM-2 (partial AOF recovery), where `redis_epoch` may be non-zero but historical — yet FM-2's recovery is still identical to FM-1 (safe advance to pg_max+1). At Tim's scale (850 sessions, ~5/day), the zero-restart zone is `redis_epoch in [0, tau=20]` and the large-redis zone is `redis_epoch >> tau`. No observed operational steady-state places a healthy system in `redis_epoch in [0, 20]` after the first 20 sessions.
- **Feasibility**: 0.88 — The discriminant is a single conditional on `redis_epoch` raw value, evaluated at the moment S4 fires. The correlation layer in `/health/detail` already reads both `redis_epoch` and `pg_max` to compute the gap. Adding `if redis_epoch <= tau: "REDIS_RESET_ZERO_CLASS" else: "REDIS_RESET_LARGE_CLASS"` is a 3-line addition to the S4 alarm classification. Alarm codes can be `REDIS_RESET_ZERO` (safe to auto-repair) and `REDIS_EPOCH_RETROGRADE_LARGE` (requires investigation; suppress auto-repair).
- **Description**: Add a sub-classification to the S4 CRITICAL alarm in the Direction 1 path (redis_epoch < pg_max). When S4 fires in Direction 1: if `redis_epoch <= tau` (currently 20), classify as `REDIS_RESET_ZERO` and permit auto-repair by setting Redis epoch to pg_max+1; if `redis_epoch > tau`, classify as `REDIS_EPOCH_RETROGRADE_LARGE`, suppress auto-repair, and emit a structured alarm requesting PG epoch distribution investigation. This 3-line discriminant separates two recovery paths that Q106 and Q084 currently conflate under a single "Redis reset" pattern.

### Idea: pg-epoch-inflation-sentinel-check
- **Novelty**: 0.68 — S4 fires on redis/pg divergence but does not validate whether `pg_max` itself is anomalously large relative to the system's age and session density. A sentinel that checks `pg_max > redis_epoch + EPOCH_INFLATION_CEILING` (where CEILING is much larger than EPOCH_GAP_ALARM_THRESHOLD — e.g., 500) would detect FM-3 class failures as a distinct alarm independent of S4's direction logic. At Tim's usage density (~5 sessions/day), `pg_max` should never advance more than 200 epochs ahead of `redis_epoch` under any legitimate operational scenario.
- **Evidence**: 0.65 — At Tim's scale, `pg_max` advancing 500+ epochs ahead of `redis_epoch` requires either a multi-month network partition (implausible) or an FM-3 inflation event (a single bulk write). The check `SELECT MAX(last_access_epoch) FROM memories WHERE last_access_epoch > $redis_epoch + 500` is a direct observable that has zero false positives at normal operation. The threshold 500 can be tuned based on observed session density.
- **Feasibility**: 0.82 — A single SQL query added to the S4 evaluation path, running as a supplementary check when S4 fires in Direction 1. If the query returns count > 0, escalate the alarm to `PG_EPOCH_INFLATION_SUSPECTED` with a structured message requesting investigation of recent PG writes, migrations, and hook deployments. The query is O(1) on a GIN-indexed integer column. Cost: negligible (S4 fires rarely).
- **Description**: When S4 fires in Direction 1, run a supplementary PG query: `SELECT COUNT(*) FROM memories WHERE last_access_epoch > $redis_epoch + EPOCH_INFLATION_CEILING` (default 500). If count > 0, escalate S4 alarm to `PG_EPOCH_INFLATION_SUSPECTED` and suppress the standard auto-repair. Include in the alarm payload: count of affected memories, their approximate epoch range, and a recommended inspection query (`SELECT id, last_access_epoch, created_at FROM memories ORDER BY last_access_epoch DESC LIMIT 20`). This prevents the standard Redis epoch advance from being applied to a PG inflation scenario and protects the working-set filter from epoch-advance catastrophe.

### Idea: s4-subtype-diagnostic-in-health-api-response
- **Novelty**: 0.66 — When S4 fires today, the correlation layer emits the gap value and direction but not the sub-classification or the `redis_epoch` absolute value. Including `redis_epoch_absolute`, `pg_max_epoch`, and the `inferred_subtype` field in the S4 alarm section of `/health/detail` converts the raw-metric response into an actionable diagnostic. This pattern (enriched alarm payloads with inferred context) is common in cloud monitoring (CloudWatch Insights, Datadog anomaly detection) but is not present in the current Recall health API design.
- **Evidence**: 0.65 — The sub-classification is a direct function of observable values already computed by the S4 check. Emitting them explicitly in the API response costs nothing and enables the Discord notification layer (Q098) to dispatch the correct operator action without requiring manual Redis inspection. The `inferred_subtype` can be one of: `REDIS_RESET_ZERO` (auto-repairable), `REDIS_EPOCH_RETROGRADE_LARGE` (investigate PG), `CASAOS_RESTART_WITH_DATA_LOSS` (S3 co-alert detected).
- **Feasibility**: 0.82 — Adds a structured `alarm_context` field to the S4 section of `/health/detail` response: `{ "redis_epoch": 0, "pg_max": 850, "gap": 850, "gap_direction": "retrograde", "inferred_subtype": "REDIS_RESET_ZERO", "auto_repair_safe": true, "recommended_action": "SET recall:global_epoch = 851" }`. No schema changes. No infrastructure changes. The existing S4 computation already has all these values; they just need to be serialized into the response.
- **Description**: Enrich the S4 alarm section of `/health/detail` with a structured `alarm_context` object containing `redis_epoch`, `pg_max`, `inferred_subtype`, and `auto_repair_safe`. The subtype is determined by: (1) Is S3 also CRITICAL? → `CASAOS_RESTART_WITH_DATA_LOSS`. (2) Is `redis_epoch <= tau`? → `REDIS_RESET_ZERO` (auto-repair safe). (3) Otherwise → `REDIS_EPOCH_RETROGRADE_LARGE` (auto-repair NOT safe). The Discord notification layer and operator runbook both key off `inferred_subtype` to route to the correct remediation path. This converts a "red light with a number" alert into a "red light with a diagnosis and a command" alert.

## Verdict Rationale

**INCREMENTAL** (not BREAKTHROUGH): Q106's claim that S4=CRITICAL + S5=HEALTHY indicates Redis restart is correct for the dominant failure mode. This calibration confirms the mechanism is sound and identifies one genuine ambiguity gap (FM-3: PG epoch inflation produces the same fingerprint but requires the opposite recovery action). PG vacuum and standard clock skew do not produce the pattern. The resolution — discriminate on `redis_epoch` absolute value — is a 3-line conditional, not an architecture change. The Q084 matrix should gain two sub-rows (zero-class and large-class), the Q106 auto-repair path should be gated on the zero-class discriminant, and the system's operational correctness is preserved. This is a precision improvement to an existing finding, not a new insight category.

When S4 fires Direction 1: if `redis_epoch <= tau`, emit `REDIS_RESET_ZERO` (auto-repair safe); if `redis_epoch > tau` AND `pg_max > redis_epoch + 500`, emit `PG_EPOCH_INFLATION_SUSPECTED` (suppress auto-repair). - **[CALIBRATION] Q119**: What is the practical likelihood of FM-3 (PG epoch inflation) at Tim's deployment? Does any current hook, maintenance script, or migration tool write `last_access_epoch` values derived from a clock source rather than the Redis INCR path? If all epoch writes pass through Redis INCR, FM-3 requires a bug introduced during migration or an explicit malicious write — placing its probability well below 1% per year. If a clock-based epoch path exists, FM-3 is an active risk requiring the Q118 discriminant guard.

- **[DESIGN] Q120**: Should the Recall API enforce `last_access_epoch` write validation — rejecting any PG write where the new value exceeds `recall:global_epoch + EPOCH_WRITE_CEILING` (e.g., ceiling = 10)? This would make FM-3 impossible at the API layer by preventing hook bugs or migration scripts from writing future epochs into the memories table. Cost: one Redis GET per memory write (already done for the INCR operation, so effectively free).
