# Finding: Q044 — Burst-Promotion Failure in the `access_count >= 3` LLPC Admission Threshold

**Question**: Is the `access_count >= 3` LLPC admission threshold in Q042's working-set routing rule the right calibration, or does it prematurely lock in a stale working set for sessions with bursty access patterns?
**Agent**: quantitative-analyst
**Question Type**: ADVERSARIAL
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Database buffer management (2Q algorithm), OS virtual memory (working set model), CPU branch predictor design

---

## Adversarial Analysis

### Thread A: Defense — `access_count >= 3` is correct because the recency gate compensates

**Core claim**: Three accesses is the minimum frequency that distinguishes genuine multi-session relevance from one-off retrieval artifacts, and when combined with the recency gate, the threshold is self-correcting against the bursty-session failure mode.

**Evidence**:

The 2Q paper (Johnson & Shasha, VLDB 1994) chose a threshold of 2 (Am admission after second access) for general buffer management. Q042 raised this to 3 for good reason: the 2Q paper was designed for uniform access streams (OLTP financial traces). A personal coding assistant has far higher single-session access bursts — a developer debugging a problem may hit the same memory 5-10 times in one session purely because they are reading and re-reading it as they work. The 2Q threshold of 2 would be insufficient to filter these intra-session duplicates; 3 is already conservative.

The attacker's failure scenario contains a partial self-correction: if a Docker networking memory is accessed 3 times in one session and `sessions_since_last_access` resets to 0, the recency gate gives it a maximum of 10 more sessions of LLPC eligibility. If the project is genuinely over, the memory ages out automatically. Thread A claims this is sufficient.

The 2Q paper's empirical result supports threshold flexibility: Johnson & Shasha showed that performance was stable from threshold=2 through threshold=5 on mixed workloads, with the two-queue structure itself providing the benefit rather than the specific count. Tim's session patterns are mixed workloads (different projects, different tools) — a count in the range 3-5 should be robust.

**Best case scenario**: A memory that represents stable background knowledge (CasaOS SSH credentials, Docker compose patterns, Recall API endpoint) accumulates `access_count >= 3` across distinct work sessions, correctly earns LLPC admission, and stays in LLPC for the duration of that project phase. The recency gate demotes it automatically when topic focus shifts.

---

### Thread B: Attack — `access_count >= 3` fails because it conflates intra-session burst with inter-session persistence

**Core claim**: The threshold conflates intra-session repetition with inter-session persistence, and the 2Q algorithm the design derives from explicitly requires promotion to occur across a temporal boundary — a condition the SQL rule does not enforce.

**Evidence**:

The 2Q paper's Am admission mechanism is not "access K times total" — it is "access again after surviving A1's eviction timeout." The key structural feature is the A1 queue: an item enters A1 on first access and is **only promoted to Am if it is accessed again after a sufficient temporal gap for A1 turnover**. The mechanism is about crossing a time boundary between access events, not about raw access count.

The SQL rule `access_count >= 3` has no temporal structure on the access events themselves. A memory accessed 3 times within 60 seconds in one session satisfies `access_count >= 3` identically to a memory accessed once each week for 3 weeks. These represent categorically different evidence strengths for LLPC admission.

**Concrete failure mode**: During a 2-hour Docker debugging session, Tim retrieves a "Docker bridge networking" memory at t=0min, t=25min, t=55min. `access_count` hits 3. `sessions_since_last_access` resets to 0. The LLPC routing query now injects this memory into every subsequent session for up to 10 sessions — even when Tim is working on an unrelated Python project. The Docker memory earns LLPC via same-session burst, not multi-session persistence.

**The recency gate does not fully compensate for two reasons**:

First, `sessions_since_last_access` resets to 0 on any access, including incidental glances. A memory that earns burst-promotion and is then incidentally accessed (Tim briefly opening a related file) every 5-8 sessions will perpetually satisfy `sessions_since_last_access <= 10` despite representing stale knowledge. The recency gate requires 10 consecutive sessions without any access — not 10 sessions without deep use.

Second, `access_count` is monotonically non-decreasing. Once a memory crosses the threshold during any burst, it never un-crosses it. The threshold asymmetry means burst-promoted memories cannot be demoted by frequency signal alone; only the recency gate can remove them, and the recency gate can be reset by marginal access events.

**Adjacent field confirmation**: In CPU branch predictor design, the 2-bit saturating counter requires two mispredictions to change prediction state — one misprediction is noise, two is a signal. Yeh & Patt (1992, "Two-Level Adaptive Training Branch Prediction") specifically designed two-level history to avoid count-conflation failure in bursty access patterns (loops that run many times in one execution context). Branch predictors using raw count thresholds without inter-event temporal structure had measurably worse performance on bursty workloads. The same structural argument applies here.

Denning's working set model (1968) is also unambiguous: W(t, τ) is the set of pages referenced in the time interval (t−τ, t]. A page referenced 3 times at t=0 and never again has `access_count=3` but is **outside the working set** for all t > τ. The working set model has no "count it 3 times and it's permanently hot" behavior — membership is re-evaluated at every time step. The raw `access_count >= 3` threshold does not implement Denning's model; it implements a permanent ratchet that the recency gate imperfectly compensates for.

**Failure mode**: 3,000 tokens of wasted context budget across 10 sessions (2 days at Tim's session density) from a stale working set locked in by one bursty sprint.

## Synthesis

### What Thread A assumes that Thread B exposes

Thread A assumes the recency gate (`sessions_since_last_access <= 10`) provides sufficient temporal structure to correct for same-session burst promotion. Thread B exposes why this assumption is brittle: the recency gate only evicts memories that go un-accessed for 10+ consecutive sessions. For any memory that earns burst-promotion and then receives even one incidental access every 5-8 sessions, the recency gate never fires.

Thread A also assumes that the 2Q algorithm's temporal boundary is reproduced by the recency gate. This is incorrect. 2Q's temporal boundary operates on the qualifying access events themselves (were the re-accesses separated from the first access by a temporal gap?). The recency gate operates on the most recent access relative to the present. A bursty memory satisfies the recency gate but fails the 2Q temporal condition. These are different temporal predicates.

### When Thread A is right

When Tim's session access patterns are relatively uniform — roughly one access per session spread across multiple work sessions on different days. In this regime, `access_count >= 3` reliably signals multi-session persistence. Memories about stable infrastructure (SSH credentials, service endpoints, persistent tool configurations) accumulate access_count via uniform patterns and correctly earn LLPC admission.

### When Thread B is right

When Tim's access patterns are bursty — concentrated in intensive work sprints followed by topic shifts. In this regime, `access_count >= 3` promotes memories based on sprint depth, not breadth of relevance across sessions. The attacker is correct that any memory earning burst-promotion becomes nearly permanent: `access_count` never decreases, and `sessions_since_last_access` resets on any marginal touch.

### The architecture Thread B would accept

The fix is to track `distinct_sessions_accessed` (count of distinct sessions in which the memory was retrieved at least once) separately from `access_count` (total raw retrieve events). Replace the LLPC admission criterion:

```sql
-- Current (vulnerable to burst-promotion):
WHERE access_count >= 3
  AND sessions_since_last_access <= 10

-- Fixed (implements 2Q temporal boundary in session units):
WHERE distinct_sessions_accessed >= 3
  AND sessions_since_last_access <= 10
```

`distinct_sessions_accessed` is incremented at most once per session, regardless of within-session access frequency. Tim accessing a Docker memory 5 times in one session increments `distinct_sessions_accessed` by 1, not 5. LLPC admission requires 3 distinct temporal events (sessions), not 3 raw accesses — directly implementing the 2Q A1/Am temporal boundary requirement.

`access_count` is retained as a within-session depth signal for MBC ordering by access velocity (`access_count / sessions_since_creation`). The two counters serve different purposes: `distinct_sessions_accessed` measures breadth of relevance across sessions (LLPC gate); `access_count` measures total depth of engagement (MBC velocity ranking).

**Session-end hook change**: In the existing `session-boundary-working-set-update` mechanism, the hook already collects all memory IDs retrieved during the session. Extend it to:
1. Increment `distinct_sessions_accessed` by 1 for each memory accessed at least once in this session (regardless of within-session frequency)
2. Increment `access_count` by the actual retrieve count per memory (already tracked)
3. Reset `sessions_since_last_access = 0` for accessed memories
4. Increment `sessions_since_last_access` by 1 for all non-accessed memories

This is the complete fix. Schema change: one integer column. Hook change: one additional bulk UPDATE. LLPC routing SQL: replace one column name.

## Ideas Extracted

### Idea: distinct-sessions-admission-gate
- **Novelty**: 0.82 — No AI memory system distinguishes same-session repetition from cross-session recurrence as an LLPC admission criterion. The closest analogue is 2Q's temporal boundary requirement, not implemented in any known personal AI memory system with this semantic structure.
- **Evidence**: 0.85 — The 2Q algorithm (Johnson & Shasha, VLDB 1994) requires temporal separation between qualifying accesses via A1 queue turnover. Denning's working set model (1968) defines membership on a time-window basis, not raw count. Yeh & Patt (1992) two-level branch predictor design identifies count-conflation failure specifically for bursty patterns. Three independent adjacent-field sources converge on the requirement for temporal separation between qualifying events.
- **Feasibility**: 0.90 — One new integer column (`distinct_sessions_accessed INT DEFAULT 0`). One additional bulk UPDATE in the session-end hook (already firing, already collecting accessed memory IDs). One column name change in the LLPC routing SQL. `access_count` is retained for MBC velocity ordering. No new infrastructure. Estimated: 1-2 days.
- **Description**: Add `distinct_sessions_accessed` to the memories table, incremented at most once per session per memory by the session-end hook. Replace `access_count >= 3` with `distinct_sessions_accessed >= 3` in the LLPC admission query. A memory accessed 5 times in one debugging sprint earns `distinct_sessions_accessed = 1` — it must prove relevance across 2 more independent sessions before LLPC admission. `access_count` remains available for MBC access-velocity ordering.

### Idea: session-burst-detection-penalty
- **Novelty**: 0.75 — Using the ratio `access_count / distinct_sessions_accessed` as a burstiness signal to modify LLPC ranking priority is not implemented in any AI memory system. Cache admission systems do not generally compute intra-period access density as a ranking penalty.
- **Evidence**: 0.65 — Burstiness as a scan-vs-steady-state discriminant is used in network traffic analysis (Poisson vs. bursty Pareto traffic models) and CPU prefetcher design (stride vs. irregular pattern detection). The specific application to memory retrieval admission ranking is engineering extension of established burstiness concepts from adjacent fields.
- **Feasibility**: 0.80 — Computable from `access_count / GREATEST(distinct_sessions_accessed, 1)` once `distinct-sessions-admission-gate` is implemented. No additional schema. Apply as a ranking modifier in the LLPC ORDER BY: memories with high burstiness (ratio > 3) rank lower than memories with the same `distinct_sessions_accessed` but lower burstiness.
- **Description**: Compute a burstiness ratio per memory: `access_count / GREATEST(distinct_sessions_accessed, 1)`. Ratio near 1.0 means roughly one access per session (steady background knowledge). Ratio > 3 means the memory is accessed in concentrated sprints. Apply as a ranking penalty in the LLPC ORDER BY: prefer low-burstiness memories for the K_llpc=2 slots. A ratio-weighted sort selects the "antibody titer" memories — consistently relevant across sessions — over "sprint memories" that were deeply used once. This ensures the scarce K_llpc=2 slots hold steady knowledge, not project-specific debugging artifacts from past sprints.

### Idea: two-phase-llpc-probation
- **Novelty**: 0.78 — A probationary LLPC tier (memories that qualify by frequency but are served in LLPC slots only after a temporal observation period) is not implemented in any AI memory system. The concept maps directly to 2Q's A1 queue applied as a third tier between MBC and permanent LLPC admission.
- **Evidence**: 0.80 — The 2Q A1/Am two-queue structure is the direct source. The A1 queue is explicitly a probationary tier: items survive A1 pressure or are evicted, and only promotion after surviving distinguishes the hot set. Johnson & Shasha (1994) show that the probationary structure — not the specific count threshold — is the primary mechanism eliminating scan pollution. Applying the same structural principle to LLPC admission provides the same guarantee.
- **Feasibility**: 0.65 — Requires a new state field (`llpc_status ENUM('mbc','probation','admitted')`), a probation session counter, and session-end logic to advance or reset probation state. More complex than the two-column approach but provides stronger guarantees. Estimated: 3-5 days including state machine changes in the session-end hook.
- **Description**: Three-state LLPC admission process. (1) MBC: default, on-demand retrieval only. (2) Probationary LLPC: memory qualifies by `distinct_sessions_accessed >= 3`, begins appearing in LLPC slots for a 3-session trial period. If accessed during probation, advance to Admitted. If not accessed during the 3-session probation window, demote back to MBC. (3) Admitted LLPC: full membership, auto-injected every session. The probationary phase implements 2Q's A1 temporal requirement: a memory must demonstrate continued relevance during a post-qualification observation window, not just at the moment it hits the count threshold. Burst-promoted memories that are not accessed in the week following their sprint fail probation automatically.
