# Finding: Q055 — Empirical Calibration of K_llpc, K_mbc, and the distinct_sessions_accessed Threshold for Tim's Session Topology

**Question**: What are the empirically correct values for K_llpc, K_mbc, and the distinct_sessions_accessed >= N threshold given Tim's actual session density and project topology — and how sensitive is retrieval quality to miscalibration of each parameter?
**Agent**: quantitative-analyst
**Question Type**: CALIBRATION
**Verdict**: PROMISING
**Severity**: High
**Source field**: 2Q buffer management (Johnson & Shasha, VLDB 1994), working-set model (Denning 1968, 1980), information-theoretic optimal split derivation

## Analysis

### Part 1: Session Topology and the meaning of "distinct sessions"

Tim runs 5 sessions/day across 4 active projects with the following estimated distribution:

| Project | Session share | Sessions/day |
|---|---|---|
| System-Recall (primary) | 50% | 2.5 |
| FamilyHub | 20% | 1.0 |
| Media-Stack | 15% | 0.75 |
| Relay-AI | 15% | 0.75 |

P(any two consecutive sessions belong to the same project) = sum(f²) = 0.50² + 0.20² + 0.15² + 0.15² = **0.335** (33.5%). Two-thirds of session transitions cross project boundaries.

**Calendar sessions vs. project-scoped sessions:**

For a *project-specific* memory (e.g., a System-Recall architecture detail), both interpretations are equivalent. That memory is only retrieved during System-Recall sessions, so `distinct_sessions_accessed` increments only during SR sessions regardless of whether counting is global or project-scoped.

For a *cross-project* memory (e.g., SSH credentials, Docker networking patterns used across all 4 projects), global counting reaches DSA>=3 in 3/5 = **0.60 days**, while project-scoped counting (counting only System-Recall sessions) reaches DSA>=3 in 3/2.5 = **1.20 days** — exactly 2x slower.

**The critical interpretive question:** The current design from Q044 uses a global `distinct_sessions_accessed` counter. This is correct for project-specific memories but undercounts temporal breadth for cross-project memories — a cross-project memory accessed 3 times in 3 different FamilyHub sessions earns LLPC admission despite never being demonstrated as relevant to System-Recall work. The `distinct_sessions_accessed_within_project` alternative is addressed in Part 4.

### Part 2: Optimal K_llpc and K_mbc via 2Q Am/A1 structure

The 2Q algorithm (Johnson & Shasha, VLDB 1994) designates the **Am queue** as the protected set of items that have survived A1 and been re-accessed. Empirically, Am occupies **70–85% of the total buffer** in steady-state for mixed OLTP workloads. This is the design-grounded split ratio for K_llpc/K_mbc.

Applied to K_total = 5:

```
K_llpc (Am fraction): floor(0.70 × 5) = 3  to  floor(0.85 × 5) = 4
K_mbc  (A1/residual): K_total - K_llpc = 1  to  2
```

**Current configuration K_llpc=2, K_mbc=3 inverts the 2Q Am/A1 ratio.** It allocates 40% of retrieval budget to the auto-inject tier and 60% to the on-demand tier. The 2Q model predicts the Am tier should hold 70–85%, not 40%. This means the current LLPC tier is undersized relative to the theoretical optimum.

**Token budget impact of correcting the split:**

| K_llpc | K_mbc | LLPC tokens | MBC tokens | Total tokens | LLPC % of 19k budget |
|---|---|---|---|---|---|
| 2 (current) | 3 | 300 | 450 | 750 | 1.6% |
| **3 (recommended)** | **2** | **450** | **300** | **750** | **2.4%** |
| 4 | 1 | 600 | 150 | 750 | 3.2% |

All three splits consume identical total tokens (750). Moving to K_llpc=3 costs 150 additional auto-injected tokens (0.8% of budget) while delivering one more guaranteed-relevant memory per session.

**Working-set size corroboration:**

A hot System-Recall memory is accessed with probability 0.105 per random session (50% chance it is an SR session × 20% chance that memory is retrieved within SR sessions + 50% × 1% cross-project incidental). Expected DSA inside the tau=10 window = 0.105 × 10 = **1.05**. A hot SR memory reaches DSA>=3 in 28.6 sessions on average (~5.7 calendar days). This is slow enough that K_llpc=3 (admitting memories once they earn DSA>=3) correctly represents only the most persistently relevant set — the expected working set size at any moment is small.

### Part 3: Sensitivity Analysis

**Corpus model:** 3,000 memories after 30 days of operation (20 memories/session × 5 sessions/day × 30 days).

**DSA accumulated per memory category after 30 days:**

| Memory category | n (est.) | Corpus % | DSA accrual rate (/day) | DSA after 30d |
|---|---|---|---|---|
| Always-on infrastructure (SSH, API endpoints) | 60 | 2% | 0.80 | 24.0 |
| Active project contexts (SR architecture, configs) | 450 | 15% | 0.45 | 13.5 |
| Secondary project contexts (FamilyHub, Media) | 600 | 20% | 0.20 | 6.0 |
| Historical/lookup (completed tasks) | 1,050 | 35% | 0.08 | 2.4 |
| One-time/never retrieved | 840 | 28% | 0.00 | 0.0 |

**LLPC eligibility pool by threshold N:**

| N | Memories meeting DSA>=N | Corpus % | Categories admitted |
|---|---|---|---|
| 2 | 2,160 | 72% | All except one-time. Includes historical/lookup. |
| **3 (current)** | **1,110** | **37%** | **Always-on + Active + Secondary** |
| 4 | 1,110 | 37% | Same as N=3 (historical stops at DSA=2.4, below 3 or 4) |

**N=2 miscalibration (lowering threshold by 1):**
- Admits 600 additional secondary-project-context memories (20% of corpus) into the LLPC eligibility pool
- These memories have DSA=6.0 — well above N=2 but correctly excluded at N=3 for now
- These would have earned N=3 eligibility in 5.0 more days under natural accrual
- Net effect: LLPC pool bloats by 600 candidates; K_llpc=3 cap means actual auto-injected memories stay at 3, but the ranking competition changes — higher-accrual secondary memories may displace active-project-context memories in limited LLPC slots
- **Verdict: N=2 is a meaningful false-positive risk. Severity: MEDIUM.**

**N=4 miscalibration (raising threshold by 1):**
- No change in eligibility vs N=3 at 30d (historical tops out at DSA=2.4, already excluded)
- Active project context memories are delayed by (4−3)/0.45 = **2.2 additional days** before LLPC admission
- During that window, 450 memories (15% of corpus) serve only as MBC candidates — partial degradation
- After 30d of operation, N=4 and N=3 admit identical sets (both exclude historical/lookup)
- **Verdict: N=4 is a minor false-negative during early life of memories. Severity: LOW.**

**Recency window tau sensitivity:**

| tau (sessions) | tau (calendar days) | P(hot SR memory inside window) |
|---|---|---|
| 5 | 1.0 | 0.426 |
| 7 | 1.4 | 0.540 |
| **10 (current)** | **2.0** | **0.670** |
| 14 | 2.8 | 0.788 |
| **20 (recommended)** | **4.0** | **0.891** |

At tau=10 (2 calendar days), a hot SR memory has only a **67% chance** of being inside the recency window in any given session. This is because at 5 sessions/day, the working set window expires before a typical weekend gap (Tim may not use the system Saturday–Sunday). Extending tau to 20 sessions (4 calendar days) raises coverage to **89%** — a 22-percentage-point gain.

### Part 4: Project-aware threshold analysis

**Proposal:** Replace `distinct_sessions_accessed >= 3` (global) with `distinct_sessions_accessed_within_project >= 2`.

**Timing comparison:**

| Memory type | Global DSA>=3 (days to admission) | Project-scoped DSA>=2 (days) | Ratio |
|---|---|---|---|
| Always-on infrastructure | 3.75 | 5.00 | 1.33x SLOWER |
| Active project contexts | 6.67 | 8.89 | 1.33x SLOWER |
| Secondary project contexts | 15.0 | 20.0 | 1.33x SLOWER |

The model reveals that project-scoped DSA>=2 is **uniformly 1.33x slower** than global DSA>=3. This is because project-scoped counting restricts accrual to 50% of sessions (the System-Recall project fraction), requiring N=2 project-sessions to equal N=3 global sessions in absolute time only when project fraction = 2/3 — which is not Tim's topology.

**The trade-off is semantic precision vs. admission speed:**
- Global DSA>=3 admits faster but accepts cross-project incidental accesses as evidence
- Project-scoped DSA>=2 admits 1.33x slower but each qualifying access is within-project, providing stronger evidence of project-relevant utility

**When project-scoped is strictly superior:**
A System-Recall memory accessed once during a FamilyHub debugging session (incidental cross-project lookup) earns global DSA+=1 but project-scoped DSA+=0 for the FamilyHub project. This is the correct behavior — the memory should not earn LLPC credit for FamilyHub contexts where it happened to be retrieved incidentally.

**When global is sufficient:**
For Tim's single-machine, single-user setup (no multi-project LLPC confusion), the global counter is a valid approximation if domains are cleanly separated. The Recall system already uses domain tagging, and the LLPC routing SQL already filters by session context if the query is domain-scoped. The project-scoped counter adds precision at the cost of a more complex schema (requiring a `project_id` foreign key on each `distinct_sessions_accessed` increment).

**Conclusion on Part 4:** Project-scoped DSA>=2 is architecturally cleaner but slower to admit. For Tim's 4-project topology, the simplest correct approach is to keep global DSA>=3 and add a domain-scoping clause to the LLPC query itself, ensuring LLPC candidates are selected from the current session's domain first. This achieves the same semantic isolation without a new counter column.

## Recommendations

**Recommended parameter set (in priority order):**

**1. K_llpc = 3, K_mbc = 2** (change from 2/3 to 3/2)
- Grounds K_llpc in the 2Q Am fraction (70–85% of K_total=5)
- Token cost increase: 150 tokens/session (0.8% of budget) — negligible
- Expected benefit: 1 additional guaranteed-relevant memory per session without query overhead
- Implementation: change two integer constants in the routing query LIMIT clauses

**2. tau = 20 sessions** (change from 10 to 20)
- Extends the recency window from 2 calendar days to 4 calendar days
- Retains 89% of hot SR memories inside the window vs. 67% at tau=10
- Critical for weekend resilience — Tim's sessions likely have 2-day gaps
- Implementation: change one constant in the LLPC WHERE clause

**3. DSA_N = 3** (confirm current value)
- N=3 is correctly calibrated. Do not change to N=2 (pool bloat) or N=4 (admission delay)
- The global counter is correct for Tim's single-machine, domain-separated usage

**4. Project-scoped DSA>=2 as optional future enhancement**
- Implement as a domain-scoped LLPC query filter rather than a separate counter column
- Add `AND domain = :current_session_domain` to the LLPC routing SQL
- This achieves semantic isolation without schema additions
- Defer until after K_llpc and tau corrections are deployed

**Corrected LLPC routing query (incorporating all recommendations):**

```sql
-- LLPC track: auto-inject K_llpc=3 memories per session
SELECT memory_id
FROM memories
WHERE distinct_sessions_accessed >= 3
  AND sessions_since_last_access <= 20          -- tau=20 (was 10)
  AND domain = :current_session_domain          -- optional: project-scoped isolation
ORDER BY (access_count * 10 / GREATEST(sessions_since_creation, 1)) DESC
LIMIT 3;                                        -- K_llpc=3 (was 2)
```

```sql
-- MBC track: on-demand pool K_mbc=2
SELECT memory_id
FROM memories
WHERE NOT (distinct_sessions_accessed >= 3 AND sessions_since_last_access <= 20)
  AND access_count >= 1
ORDER BY access_count DESC, sessions_since_last_access ASC
LIMIT 2;                                        -- K_mbc=2 (was 3)
```

## Ideas Extracted

### Idea: tau-session-density-correction
- **Novelty**: 0.72
- **Evidence**: 0.82
- **Feasibility**: 0.95
- **Description**: The recency window tau should be expressed as `tau_calendar_days × sessions_per_day` rather than as a fixed session count. For Tim at 5 sessions/day, tau=10 sessions = 2 calendar days — too short for weekend resilience. The correct derivation is: choose tau_calendar = 4 days (covers Mon-Thu to Mon, or Fri to Wed), then tau_sessions = 4 × 5 = 20. This parameterizes the working-set window in human-meaningful calendar units and automatically adjusts if session density changes. A configuration variable `tau_calendar_days = 4` with runtime conversion `tau_sessions = tau_calendar_days × rolling_avg_sessions_per_day` makes the recency gate session-density-agnostic. Grounded in Denning (1980) working-set stability analysis, which expresses τ in real time units (seconds), not process ticks.

### Idea: k-split-2q-anchored-derivation
- **Novelty**: 0.68
- **Evidence**: 0.85
- **Feasibility**: 0.98
- **Description**: Derive K_llpc and K_mbc directly from the 2Q Am/A1 ratio rather than setting them by intuition. The rule is: K_llpc = floor(Am_fraction × K_total), K_mbc = K_total − K_llpc, where Am_fraction ∈ [0.70, 0.85] from Johnson & Shasha (1994). For K_total=5 this gives K_llpc ∈ {3, 4}. Choose K_llpc=3 (conservative lower bound) to preserve at least 2 MBC slots for on-demand coverage. This derivation is grounded in empirical buffer management data and produces a parameter value rather than an arbitrary choice. Applicable whenever K_total changes: if K_total grows to 8 (via context window expansion), K_llpc = floor(0.70×8) = 5, K_mbc = 3.

### Idea: domain-scoped-llpc-filter
- **Novelty**: 0.70
- **Evidence**: 0.65
- **Feasibility**: 0.90
- **Description**: Add `AND domain = :current_session_domain` to the LLPC routing SQL rather than introducing a new `distinct_sessions_accessed_within_project` counter column. This achieves project-scoped semantic isolation at zero schema cost: memories compete for LLPC slots only within their own domain, preventing a System-Recall memory from being auto-injected during a FamilyHub session. Requires that the session hook passes the current session's inferred domain (already available in the domain tagging system). The filter is optional and can be disabled for cross-domain LLPC (e.g., SSH credentials that are legitimately cross-domain). Implementation: one SQL WHERE clause addition, one parameter binding.

## Key Finding

The current parameter set (K_llpc=2, K_mbc=3, DSA>=3, tau=10) has two confirmed miscalibrations and one confirmed correct value. The K_llpc/K_mbc split inverts the 2Q Am/A1 ratio — the auto-inject tier should hold 70–85% of K_total, not 40%. The recency window tau=10 sessions equals only 2 calendar days at Tim's 5-session/day density, insufficient to survive a normal weekend gap. The DSA threshold N=3 is correctly calibrated: N=2 causes 20% corpus bloat in the LLPC eligibility pool, while N=4 imposes a 2.2-day unnecessary admission delay. The highest-priority fix is tau: increasing from 10 to 20 sessions raises hot-memory working-set retention from 67% to 89% per session (+22 percentage points). The second priority is K_llpc: changing from 2 to 3 adds one guaranteed-relevant auto-injected memory at a cost of 0.8% of context budget.
