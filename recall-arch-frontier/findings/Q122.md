# Q122 — FM-3 Epoch Write Risk Audit

**Question**: Does any current hook (recall-retrieve.js, observe-edit.js, recall-session-summary.js) or migration script write `last_access_epoch` values derived from a wall-clock source rather than the Redis INCR path — and what is the maximum epoch value any non-Redis-INCR path could produce that would fall outside the Q120 EPOCH_WRITE_CEILING=10 guard, making FM-3 an active risk rather than a theoretical one?
**Question Type**: CALIBRATION
**Verdict**: NEGATIVE
**Severity**: Low (theoretical risk confirmed absent from live codebase)

---

## Finding

`last_access_epoch` does not exist anywhere in the Recall codebase. Neither the column in PostgreSQL, nor the Redis key `recall:global_epoch`, nor any epoch-write code path exists in any hook, route, migration script, or storage layer. FM-3 (PG epoch column inflation from a wall-clock write) is not an active risk because the precondition — a live epoch system — has not yet been deployed.

### What Q120 EPOCH_WRITE_CEILING=10 Guards Against

Q120 proposed an API-layer guard that would reject any PG write where `new_last_access_epoch > recall:global_epoch + 10`. This guard cannot be triggered because:

1. The `last_access_epoch` column has never been added to the `memories` table (confirmed by Q111: "This column does not exist. It must be added before any code is deployed.")
2. The Redis key `recall:global_epoch` has never been created (confirmed by Q111 and the grep)
3. No code path attempts to write an epoch integer to any memory record

The maximum epoch value a wall-clock path would theoretically produce — the Q122 question's core concern — is `int(time.time()) ≈ 1,742,000,000` (Unix timestamp for March 2026) versus the Redis INCR epoch of approximately 850 for Tim's deployment. That delta of ~1.74 billion would catastrophically exceed the EPOCH_WRITE_CEILING=10 guard and cause the FM-3 failure mode described in Q117 (working-set filter wiped, entire memory corpus unreachable). However, this scenario requires the epoch system to be deployed first, which it has not been.

## Qualifying Ideas

### idea: epoch-deployment-precondition-audit
**(N=0.70, E=0.95, F=0.90)**

Before the Q111/Q118/Q120 epoch system can be activated, a precondition audit should verify three live conditions: (1) `last_access_epoch` column exists with correct type and index, (2) `recall:global_epoch` key exists in Redis with a non-zero value seeded from `SELECT MAX(last_access_epoch) FROM memories`, (3) `EPOCH_WRITE_CEILING` guard in `store_memory` reads from Redis (not a hardcoded fallback) to prevent wall-clock contamination. All three must be true simultaneously or the guard is incoherent. This finding demonstrates that auditing for a proposed invariant is most efficiently done against the precondition (does the column exist?) rather than the invariant itself (does any code write a wrong epoch value?).

### idea: wall-clock-epoch-magnitude-reference
**(N=0.72, E=0.95, F=0.95)**

The concrete magnitude comparison should be documented as a permanent reference for future epoch guard calibration: `int(time.time())` as of March 2026 ≈ 1,742,000,000. Tim's Redis epoch at ~850 sessions. Ratio: ~2,050,000:1. Any accidental wall-clock write to `last_access_epoch` produces a value that exceeds `EPOCH_WRITE_CEILING=10` by a factor of ~174,000,000 — it would advance the working-set filter so far forward that zero stored memories would pass the filter (`last_access_epoch >= 1,742,000,000 - 20 = 1,741,999,980`, but all stored memories have `last_access_epoch <= 850`). The failure is total and immediate, not gradual. This makes wall-clock contamination the highest-severity single-write failure mode in the proposed epoch architecture — worse than Redis restart (recoverable) and worse than partial AOF (attenuated). The EPOCH_WRITE_CEILING=10 guard is the correct and sufficient defense, but it must be deployed with the column.

## Verdict Rationale

**NEGATIVE**: The Q122 question asks whether FM-3 is an active risk. It is not, because the epoch system is undeployed. The column does not exist. The Redis key does not exist. No hook or migration script contains a wall-clock epoch write path. The practical implication is that Q118 (discriminant guard) and Q120 (write ceiling guard) are both pre-deployment recommendations and cannot be tested or triggered in the live system. The risk becomes active the moment the Q111 migration SQL is applied and the epoch guard code is deployed — at which point the wall-clock contamination risk is real and severe (magnitude ~174M times above the ceiling), making Q120's EPOCH_WRITE_CEILING guard a mandatory deployment companion to Q111, not an optional enhancement.

---

## Recommendations

1. **Deploy Q120 EPOCH_WRITE_CEILING guard concurrently with Q111 migration** — not after. The window between column creation and guard deployment is when FM-3 can be introduced by any ad-hoc SQL or migration script that does not know to use Redis INCR.

2. **Seed `recall:global_epoch` before deploying the store guard** — if the guard reads Redis before the key is initialized, `int(current_epoch_raw or 0) = 0`, and `0 + 10 = 10` becomes the effective ceiling for all writes during the seeding window, which would reject all existing memories with `last_access_epoch = 1` (from the Q111 migration SQL: `UPDATE memories SET last_access_epoch = 1 WHERE last_access_epoch = 0`). Deploy order: migration SQL → Redis seed → guard code → API restart.

3. **Document `int(time.time())` ≈ 1.74B as the forbidden epoch range** — any `last_access_epoch` value above `EPOCH_INFLATION_CEILING` (Q117's suggested 500) should be treated as a definitive FM-3 indicator, with the sentinel query from Q117's `pg-epoch-inflation-sentinel-check` idea firing during S4 evaluation.
