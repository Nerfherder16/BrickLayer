# Finding: Q032 — Physics Ceiling: Maximum Store Throughput Under Concurrent Write Load

**Question**: What is the theoretical maximum throughput (memories per second) for Recall's store operation under concurrent write load from two machines, and which component saturates first?
**Question Type**: PHYSICS
**Verdict**: BREAKTHROUGH
**Severity**: High

---

## Operation Definition

- **Input**: Memory text blob (~50 tokens typical for observe-edit.js output), domain string, tags, importance float
- **Output**: Stored memory with UUID, confirmed across Qdrant + Neo4j; PostgreSQL audit record fire-and-forget
- **Guarantees required**: Qdrant write confirmed (vector stored), Neo4j node created (compensating delete on failure), PostgreSQL audit log best-effort; contradiction detection synchronous before response returns

---

## Step 1: Define the Critical Path

Reading `src/api/routes/memory.py` reveals the **synchronous blocking chain** that every store request must traverse before a response is returned:

```
1.  Write guard Stage 1 (regex)                   ~0 ms     pure CPU
2.  Qdrant content-hash dedup check               ~2-5 ms   Qdrant RTT (local)
3.  Embedding via Ollama (single embed call)       ~20-80 ms GPU inference + 2x LAN hop
4.  Write guard Stage 2 (Redis cosine similarity) ~1-2 ms   Redis RTT (local)
5.  Qdrant semantic dedup search (limit=1)         ~2-5 ms   Qdrant ANN search
6.  Qdrant upsert (store vector)                  ~2-5 ms   Qdrant write
7.  Neo4j MERGE node creation                     ~5-15 ms  Neo4j write + transaction
8.  Contradiction detect (Qdrant search ×2)       ~10-20 ms 2 × ANN search (limit=5 each)
9.  Redis working memory update (if session)       ~1-2 ms   Redis RTT
10. PostgreSQL audit log                           ~2-5 ms   asyncpg write
```

**Critical correction from code**: The contradiction detector (step 8) is **synchronous** on the critical path, not a background task. The comment in memory.py line 371-373 is explicit: *"synchronous so SUPERSEDES edges are committed before the store endpoint returns."* This adds a second Qdrant ANN search pass (10 results across 2 domains) to every store.

## Step 2: Theoretical Minimum

### 2a. GPU Embedding Floor

**Model**: qwen3-embedding:0.6b
- Architecture: ~600M parameter transformer, embedding-only (encoder)
- Output: 1024 dimensions (per config.py; original question stated 384 — **actual is 1024**)
- VRAM footprint: ~1.2 GB at fp16 (600M × 2 bytes)

**RTX 3090 specs**:
- FP16 tensor core throughput: 35.6 TFLOPS
- Memory bandwidth: 936 GB/s
- VRAM: 24 GB

**FLOP requirement for a single embedding (batch=1, 50 tokens)**:

A 600M parameter encoder processes 50 tokens. At batch=1, each attention head and FFN layer runs once per token. Rough estimate:

- FLOPs per token ≈ 2 × (number of parameters) for transformer forward pass = 2 × 600M = 1.2 GFLOP per token
- For 50 tokens: 50 × 1.2 GFLOP = 60 GFLOP per embedding

At 35,600 GFLOP/s (RTX 3090 FP16):
- **Minimum compute time** = 60 GFLOP / 35,600 GFLOP/s = **1.7 ms**

This is the physics floor — the minimum possible time to run the transformer forward pass at batch=1, 50 tokens, on this GPU.

**Practical latency at batch=1**: 20–40 ms is typical for 600M models via Ollama. The gap from 1.7 ms to ~30 ms is:
- Ollama HTTP overhead: ~3-5 ms
- CUDA kernel launch overhead: ~2-3 ms
- Tokenization: ~1 ms
- Model weight streaming from VRAM to SM cache at batch=1 (memory-bound, not compute-bound at small batch): ~15-20 ms

**Key insight**: At batch=1, this workload is **memory-bandwidth-bound, not compute-bound**. The model weights (1.2 GB at fp16) must be streamed from VRAM to SMs for each forward pass. At 936 GB/s: 1.2 GB / 936 GB/s = **1.3 ms** weight-streaming floor. At batch=1, we use this bandwidth once per embedding; at batch=N, we amortize the weight streaming across N embeddings.

**Batch scaling**:

| Batch Size | Weight Stream (amortized) | Compute | Total Theoretical | Practical | Embeddings/sec |
|---|---|---|---|---|---|
| 1 | 1.3 ms | 1.7 ms | ~3 ms | ~30 ms | ~33 |
| 4 | 0.3 ms | 6.8 ms | ~7 ms | ~45 ms | ~89 |
| 8 | 0.16 ms | 13.6 ms | ~14 ms | ~70 ms | ~114 |
| 16 | 0.08 ms | 27 ms | ~27 ms | ~130 ms | ~123 |

At batch=8, the model becomes compute-bound rather than memory-bound. **Peak useful batch is approximately 8**, yielding ~100-120 embeddings/second theoretical.

However: Recall sends **one embedding request per store** (single `embed()` call, not `embed_batch()`). The `embed_batch()` method exists but is never called from the store path.

### 2b. Network Latency Floor (CasaOS → Ollama)

Ollama runs on a separate host: 192.168.50.62. The embedding request must cross the LAN twice (request + response).

- LAN round-trip: ~0.2-0.5 ms (1 GbE switched network, same subnet)
- HTTP/1.1 keepalive is configured (`keepalive_expiry=30s`) — connection reuse eliminates TCP handshake after first request
- Request payload: ~50 tokens = ~200-300 bytes JSON
- Response payload: 1024 floats × 4 bytes = 4096 bytes JSON (as text: ~6-8 KB)

**Network floor**: 0.4–1 ms round-trip. This is negligible compared to GPU inference time.

**Connection pool constraint**: `httpx.Limits(max_connections=5, max_keepalive_connections=2)`. Under concurrent load from two machines, both hitting the same FastAPI process, Ollama requests will queue when all 5 connections are busy. At 30 ms per embedding, 5 connections can sustain **5 / 0.030 = ~167 embeddings/sec** before queuing (but Ollama itself is single-threaded at batch=1, so the real limit is lower).

### 2c. Ollama Concurrency Model

Ollama processes one request at a time per model (serialized GPU execution). If two concurrent store requests arrive simultaneously (from casaclaude and proxyclaude), their embedding calls queue behind each other at the Ollama layer.

**Effective serialization**: 2 concurrent machines × 1 embedding each = embedding requests queued 2-deep at Ollama. Each waits for the other. Throughput is still ~33 embeddings/sec (the GPU processes them one by one), but each individual request sees ~60 ms latency instead of ~30 ms.

### 2d. Write-Target Throughput Ceilings

**Qdrant** (local, same host as FastAPI):
- Qdrant uses WAL + memtable + segment flush architecture
- Single upsert of 1024-dim float32 vector = 4096 bytes vector + ~500 bytes payload = ~5 KB per point
- Qdrant's async gRPC/REST API handles concurrent upserts; internal write path serializes per-segment
- Documented write throughput (local, single collection): 1,000-10,000 upserts/sec for small payloads
- **Qdrant is not the bottleneck at this load**

**Neo4j** (Docker container, same CasaOS host):
- Each store fires a `MERGE (m:Memory {id: $id}) SET ...` transaction
- Neo4j Community Edition: single-writer, MVCC for reads
- Bolt connection pool overhead: ~2-3 ms per transaction round-trip (local Docker)
- Neo4j write throughput: ~200-1,000 simple MERGE transactions/sec on modern hardware
- **Neo4j is not the bottleneck at this load**

**PostgreSQL** (Docker container, same CasaOS host):
- Audit log write is fire-and-forget (`try/except` around `pg.log_audit(...)`)
- Simple INSERT into audit_log; asyncpg connection pool
- PostgreSQL INSERT throughput: 5,000-50,000 simple rows/sec
- **PostgreSQL is not the bottleneck**

**Redis** (Docker container, same CasaOS host):
- Working memory set add: O(1), ~0.1-0.5 ms
- **Redis is not the bottleneck**

## Step 3: Current Implementation Cost

### Single store latency (one machine, no contention):

| Step | Cost | Type |
|---|---|---|
| Write guard Stage 1 (regex) | ~0.5 ms | Implementation |
| Qdrant content-hash dedup | ~3 ms | Architecture (extra Qdrant RTT) |
| Network RTT to Ollama | ~0.5 ms | Physics (LAN) |
| Ollama embedding (batch=1) | ~25-40 ms | Physics gap + architecture gap |
| Write guard Stage 2 (Redis cosine) | ~1 ms | Architecture |
| Qdrant semantic dedup search | ~3 ms | Architecture |
| Qdrant upsert | ~3 ms | Physics |
| Neo4j MERGE | ~8 ms | Physics + architecture |
| **Contradiction detect (Qdrant ×2)** | **~12 ms** | **Architecture gap** |
| Redis working memory | ~1 ms | Physics |
| PostgreSQL audit log | ~3 ms | Architecture (fire-and-forget, but still awaited before return) |

**Total actual latency**: ~60–75 ms per store (single machine, no queue)
**Theoretical minimum**: ~3 ms embedding + ~3 ms Qdrant write + ~8 ms Neo4j = **~14 ms**
**Gap ratio**: 60–75 ms / 14 ms = **4.3–5.4×**

### Two-machine concurrent load:

**Observed pattern**: Both machines run `observe-edit.js` on every file save. At active coding pace: 5–10 saves/minute per machine = 10–20 saves/minute combined = **0.17–0.33 stores/second total**.

This is far below any saturation point. The system is **comfortably idle** at real-world two-machine load.

**Burst scenario** (e.g., both machines process a long session summary simultaneously):
- 2 concurrent embedding requests → Ollama serializes → second request waits ~30 ms for first to complete
- Total latency for the second machine: ~90–110 ms (vs ~60 ms with no contention)
- This is still well within acceptable bounds

**Saturation point**: Ollama GPU saturates when concurrent embedding requests × per-embedding time exceeds the serialization capacity. At 30 ms per embedding, Ollama can sustain ~33 new stores/second before queue depth grows unboundedly. Two machines at 0.17–0.33 stores/sec represents **0.5–1.0% of GPU capacity**.

## Step 4: Gap Analysis

### Physics gaps (unavoidable):

- **GPU inference time** (~1.3–3 ms floor): Minimum time to move model weights and compute forward pass. Reducible by batching but not eliminable.
- **LAN RTT to Ollama** (~0.5 ms): Speed-of-light bound for two hosts on the same switch. Not worth optimizing.
- **Qdrant write latency** (~2-3 ms): Storage durability requires fsync or WAL commit. Not eliminable.
- **Neo4j transaction latency** (~5-8 ms): Bolt protocol + ACID transaction overhead. Compressible to ~3 ms with connection pooling but not zero.

### Architecture gaps (design choices):

1. **Serial embedding without batching** (~25 ms vs ~3 ms if batched): Every store calls `embed()` one at a time. `embed_batch()` exists in the service but is never called from the store path. If stores were queued and batched, a 100 ms window could accumulate 3–4 stores from two machines and embed them in a single GPU call at ~35 ms total instead of 4 × 25 ms = 100 ms. **Gap: ~3.3× throughput improvement possible.**

2. **Contradiction detection is synchronous** (~12 ms on critical path): The contradiction detector runs a Qdrant ANN search (limit=5 per domain, 2 domains = 10 results) before the response is returned. This was an intentional MemBench design choice (avoids retrieval races), but it adds ~12-20 ms to every store. Moving this to background with an eventual-consistency model would cut store latency by ~20%. **Gap: ~20% latency reduction possible.**

3. **Three sequential Qdrant round-trips before write**: content-hash dedup (step 2) → semantic dedup search (step 5) → contradiction search ×2 (step 8) = 4 Qdrant queries before any data is written. These are all reads against the same index and could be parallelized with `asyncio.gather`. Currently they are sequential `await` chains. **Gap: ~3-5 ms savings from parallelization.**

4. **Connection pool depth to Ollama** (max_connections=5): Under sustained concurrent load, the pool limits potential parallelism. This is correctly sized for current load (2 machines) but would need expansion for 10+ machines.

5. **No request coalescing at FastAPI layer**: When two concurrent stores arrive within milliseconds (e.g., both machines save at similar times), the system processes them independently with no opportunity to share the Ollama round-trip. A coalescing window of 50–100 ms would allow batch embedding without changing the hook or MCP client code.

### Implementation gaps (code quality):

- **`asyncio.gather` not used for parallel Qdrant reads**: Steps 2, 5, and the contradiction detect searches are all sequential. A `gather()` of dedup-hash + semantic-dedup would save ~3 ms.
- **PostgreSQL audit log**: Despite "fire-and-forget" comments, the code `await`s the call inside a try/except before returning — it's not truly async. This adds ~3 ms to the critical path unnecessarily. (Moving it to `background_tasks.add_task()` would save this.)

## Ideas Extracted

### Idea 1: Speculative Batch Embedding with Coalescing Window

**Slug**: `coalescing-embed-window`

**Description**: Introduce a coalescing buffer at the FastAPI embed call site. Stores arriving within a configurable window (50 ms default) are held and their texts are sent to Ollama as a single `embed_batch()` call. The batch response is demuxed back to waiting coroutines. This is invisible to callers — each `await embed(text)` still returns a single vector, but the GPU sees batch=N. At 2 machines × 10 saves/minute, average batch size ≈ 1 (no benefit). At 10+ machines or document ingestion bursts, batch size ≈ 4-8 (3-8× throughput improvement).

The coalescing pattern is standard in database connection pools (pgBouncer transaction mode) and distributed tracing systems (batched span export). Applied to GPU inference, it's the same as what vLLM's continuous batching does — accumulate requests, schedule as a unit.

**Evidence from adjacent field**: NVIDIA Triton Inference Server implements this as "dynamic batching" with configurable `max_queue_delay_microseconds`. Papers on inference serving (Orca, vLLM, AlpaServe) show 3-10× throughput improvement from dynamic batching over batch=1 for transformer models. This is well-established, not novel.

- **Novelty**: 0.55 — Applying continuous batching to a hook-driven memory store path is non-obvious. The insight that Recall's async Python event loop already has a natural coalescing point (the embed service singleton) where multiple concurrent coroutines compete for the same resource is concrete. However, the underlying technique (dynamic batching) is established.
- **Evidence**: 0.85 — vLLM, Triton, ORCA all demonstrate this pattern at production scale.
- **Feasibility**: 0.90 — Pure Python asyncio implementation in `EmbeddingService`. No infrastructure changes. ~50 lines.
- **Quality score**: 0.55 × 0.40 + 0.85 × 0.35 + 0.90 × 0.25 = 0.22 + 0.30 + 0.225 = **0.745** → BREAKTHROUGH

The gap this addresses: 7.6× throughput gap between batch=1 (current) and batch=8 (theoretical). At current load the gap doesn't matter, but it becomes relevant during document ingestion or if the system is extended to more machines.

### Idea 2: Contradiction Detection Deferred with Read-Time Resolution

**Slug**: `lazy-contradiction-resolve`

**Description**: Move contradiction detection entirely to background (already the design intent per the docstring at the top of `contradiction_detector.py`: *"Runs as a background task"*), but add a **read-time consistency guard** in the retrieval path that suppresses results flagged as `contradiction_pending=true` until the background check completes (typically <200 ms). The store endpoint returns immediately (~50 ms instead of ~65 ms). The MemBench race concern is addressed by the read-time guard rather than by blocking the write.

This is the same pattern as "read-your-writes consistency with eventual convergence" in distributed databases: writes are acknowledged fast, reads apply a staleness filter until background propagation completes.

- **Novelty**: 0.60 — The specific combination of (a) optimistic write acknowledgment + (b) pending-flag suppression at read time for a contradiction system is not described in production AI memory papers. The MemBench concern is real and the solution is non-trivial.
- **Evidence**: 0.70 — DynamoDB conditional writes, Cassandra LWT, PostgreSQL deferred constraints are all analogs. Well-understood in DB literature.
- **Feasibility**: 0.80 — Requires adding `contradiction_pending` field to Qdrant payload, filtering it in search, and a background task that removes the flag when done. ~100 lines of change.
- **Quality score**: 0.60 × 0.40 + 0.70 × 0.35 + 0.80 × 0.25 = 0.24 + 0.245 + 0.20 = **0.685** → BREAKTHROUGH

### Idea 3: Parallel Read Fusion for Dedup + Contradiction

**Slug**: `parallel-read-fusion`

**Description**: The current store path fires 4 sequential Qdrant queries: (1) content-hash dedup, (2) semantic dedup search, (3) contradiction domain search, (4) contradiction cross-domain search. Queries 1-4 are all reads against the same Qdrant collection and can be launched concurrently with `asyncio.gather`. The only dependency is that the embedding (step 3) must complete before any of them. After gathering, results are processed together.

Expected savings: 3 sequential reads (each ~3 ms) collapse to 1 concurrent read batch = ~3 ms instead of ~9 ms for the read phase. Total store latency reduction: ~6 ms (~9% of total).

- **Novelty**: 0.30 — Standard async IO pattern. Not novel.
- **Evidence**: 0.95 — asyncio.gather is the canonical tool.
- **Feasibility**: 0.95 — Trivial refactor.
- **Quality score**: 0.30 × 0.40 + 0.95 × 0.35 + 0.95 × 0.25 = 0.12 + 0.3325 + 0.2375 = **0.69** → BREAKTHROUGH (borderline; primarily an implementation cleanup, not architecture)

This idea scores in BREAKTHROUGH territory only because feasibility and evidence are extremely high; novelty is low. Flag as an implementation quick-win, not a research finding.

## Primary Answer

**Which component saturates first?** At real-world two-machine load: **nothing saturates**. The system has ~100× headroom on all components. If load scaled to ~33 stores/second, **Ollama's GPU serialization** would be the first bottleneck — it processes one embedding at a time and cannot saturate the RTX 3090's theoretical throughput because batch=1 is memory-bandwidth-bound rather than compute-bound.

**Is batched embedding the right optimization?** Yes — but only if load increases beyond 5-10 concurrent writers. At 2 machines and hook-driven saves, batching provides no benefit since arrivals rarely overlap. The correct trigger for implementing `coalescing-embed-window` is: any use case involving batch document ingestion (the `/ingest` endpoint already exists) or expanding to 5+ concurrent machines.

**Gap ratio summary**:
- Embedding: 10× worse than batch=8 theoretical (30 ms vs 3 ms), but irrelevant at current load
- Total store path: 4.6× worse than theoretical minimum
- System headroom: ~100× before saturation (GPU is the first to saturate)
