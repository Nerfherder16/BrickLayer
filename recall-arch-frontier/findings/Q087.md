# Finding: Q087 — LLPC Token Injection Budget Model and Context Overflow Threshold

**Question**: With tau=20 working-set sessions, K_mbc=10 memories from MBC routing, and K=20 graph neighbors from Neo4j BFS, what is the expected token injection into Claude's context window per LLPC retrieval call — and at what session memory density does the system risk exceeding the Claude Sonnet 4.6 context window limit?
**Question Type**: PHYSICS
**Verdict**: BREAKTHROUGH
**Severity**: Medium
**Source field**: Information retrieval (bounded-K retrieval theory), tokenization (GPT-family token/word ratio), context window allocation (LLM prompt engineering), working-set theory (Denning 1968)

---

## Physics Ceiling: LLPC Token Injection

### Operation Definition

- **Input**: A Claude Code session start event. The recall-retrieve.js hook fires, queries the Recall API at 192.168.50.19:8200, and receives a candidate memory list.
- **Output**: A `<system-reminder>` block injected into Claude's context window containing the retrieved memory texts.
- **Guarantees required**: Every retrieved memory must be injected verbatim (no truncation, no summarization). The system-reminder block must be syntactically valid. Ordering is by relevance score descending.
- **Parameters**: tau=20 sessions, K_mbc=10 (MBC tag routing), K_graph=20 (Neo4j BFS, or K_graph=50 per Q083 relaxation), TOKENS_PER_MEMORY=150 (100-word average = 130 tokens + 20-token structured header).

---

### Theoretical Minimum

The minimum token injection is bounded by the information-theoretic requirement for retrieval signal: at least one memory must be injected to provide any retrieval benefit. At K=1, injection = 150 tokens. However, Q042 established that K_llpc=2 is the practical minimum for the working-set always-inject tier (two memories at 300 tokens). The minimum for meaningful MBC routing signal is K=3-5 (450-750 tokens), consistent with Recall's historical default.

| Resource | Minimum | Calculation |
|---|---|---|
| Token injection floor (1 memory) | 150 tokens | 100-word body × 1.3 tokens/word + 20-token structured header |
| Token injection floor (working-set minimum K=2) | 300 tokens | K_llpc=2 × 150 tokens/memory |
| Token injection floor (practical retrieval K=5) | 750 tokens | K=5 × 150 tokens/memory; Recall's prior default |
| Bandwidth floor (inject bytes) | ~600 bytes | 300 tokens × ~2 bytes/token (UTF-8 ASCII) |
| Fraction of 200K context window (K=2) | 0.15% | 300 / 200,000 |
| Fraction of 200K context window (K=5) | 0.38% | 750 / 200,000 |

The theoretical minimum injection is not constrained by the context window — it is constrained by retrieval utility. One memory at 150 tokens is a negligible context cost but a non-zero retrieval benefit.

### Token Model: Memory Size

**Body tokens**: A 100-word average memory (session summary or extracted fact) tokenizes at approximately 130 tokens using GPT-family tokenization (1.3 tokens/word, consistent across Anthropic and OpenAI models for English prose with light markup).

**Header tokens**: Each memory in the `<system-reminder>` block includes structured metadata:
- UUID: ~6 tokens
- Timestamp (ISO-8601): ~8 tokens
- Domain tag: ~3 tokens
- Tag list (2-4 tags): ~5 tokens

Total header: ~22 tokens. Rounding: **20 tokens per header**.

**Per-memory token cost**: 130 (body) + 20 (header) = **150 tokens per memory**.

**Token range**: 100-word memories are the average. The distribution is right-skewed:
- p10: 50 words = 85 tokens (short extracted facts)
- p50: 100 words = 150 tokens (typical session summary paragraph)
- p90: 300 words = 410 tokens (long session summaries with code snippets)
- p99: 500 words = 670 tokens (detailed multi-project summaries)

For injections at K=28 post-dedup, the p99 memory length produces:
28 × 670 = **18,760 tokens** — still within a 20K-token memory budget.

### Working-Set Filter: How Many Memories Survive tau=20?

The working-set filter (`last_access_epoch >= current_epoch - tau`) retains only memories accessed at least once in the last 20 sessions. For a corpus of C memories with 10 retrievals per session:

```
P(not accessed in tau=20 sessions) = (1 - 10/C)^20

Active fraction = 1 - P(not accessed)
```

| Corpus C | Active fraction | Active count |
|---|---|---|
| 100 | 0.878 | 88 |
| 500 | 0.332 | 166 |
| 1,000 | 0.183 | 183 |
| 2,000 | 0.095 | 191 |
| 5,000 | 0.039 | 196 |
| 10,000 | 0.020 | 198 |

**Critical result**: The working-set active count rapidly saturates to approximately 200 memories regardless of corpus size for C ≥ 500. This is because the access model (10 retrievals/session × 20 sessions = 200 unique memory-slots) acts as an asymptotic ceiling on the number of distinct memories that can be accessed in the tau window. At C=2000, only 9.5% of the corpus is active; at C=5000, only 3.9%. The vast majority of the corpus is cold at any moment.

**Implication**: The working-set filter does not gate how many candidates survive for injection — the K parameters gate it. At C ≥ 200, the number of active memories (≥166) always exceeds K_mbc + K_graph (30-60). The filter never becomes the binding constraint at Tim's scale.

### Candidate Count: K-Bounded Ceiling

The LLPC retrieval pipeline combines two candidate sources:

1. **MBC routing** (K_mbc=10): tag intersection query returns top-10 memories from the working-set.
2. **Neo4j BFS** (K_graph=20 or 50): causal graph traversal returns K neighbors of LLPC-seeded memories.

These two sets are deduplicated by UUID before injection. Overlap occurs when a memory is both a tag-intersection hit AND a causal neighbor of current LLPC memories — this describes "hub" memories: frequently co-retrieved with the current working set AND tagged with current context tags.

**Overlap model**: Hub memories represent the top ~5-10% of the active corpus by access frequency. At K_mbc=10 and K_graph=20, the expected overlap is 2-3 memories (20-25% of min(K_mbc, K_graph)). This estimate is consistent with information retrieval literature on top-K set intersection rates between independent rankers sharing the same corpus (Kendall tau ≈ 0.6 implies ~25% overlap in top-K sets).

| Scenario | Pre-dedup count | Overlap | Post-dedup count | Token injection |
|---|---|---|---|---|
| K_graph=20, overlap=0 (zero overlap) | 30 | 0 | 30 | 4,500 tokens |
| K_graph=20, overlap=2 (expected) | 30 | 2 | 28 | 4,200 tokens |
| K_graph=20, overlap=5 (high overlap) | 30 | 5 | 25 | 3,750 tokens |
| K_graph=50, overlap=2 (expected) | 60 | 2 | 58 | 8,700 tokens |
| K_graph=50, overlap=5 (high overlap) | 60 | 5 | 55 | 8,250 tokens |

|---|---|---|---|---|---|

**Key observation**: Injection size is **corpus-size invariant** above C ≈ 200. The injection distribution does not change between C=500 and C=5,000 because the K parameters are the binding gate, not the working-set size. This is a fundamental structural property of the K-bounded retrieval design.

### Gap Analysis

**Theoretical minimum (practical, K=5)**: 750 tokens
**Actual p50 injection (K=20 graph)**: 3,570 tokens
**Gap ratio**: 3,570 / 750 = **4.8×**

**Theoretical minimum (absolute, K=1)**: 150 tokens
**Actual p50 injection**: 3,570 tokens
**Gap ratio**: 3,570 / 150 = **23.8×**

#### Physics Gaps (unavoidable)

- **Memory body size**: 130 tokens/memory for a 100-word body is determined by the tokenization model and the information density of natural-language memory descriptions. Compressing memories below 50 words (65 tokens) risks losing critical context. The floor is approximately 65-100 tokens/memory for useful memories.

- **Structured header overhead**: 20 tokens per header (UUID, timestamp, domain, tags) is the minimum metadata required for the system to parse the injection block and display it in the Recall dashboard. This cannot be reduced below ~10 tokens without losing traceability.

- **K floor for retrieval utility**: K=1 provides negligible retrieval signal. K=3-5 is the empirically established minimum for useful retrieval (Q042). The minimum injection for useful retrieval is 3 × 150 = 450 tokens.

#### Architecture Gaps (design choices)

- **Full verbatim injection**: The current system injects complete memory text for every retrieved memory. Alternatives: (1) inject a 20-token pointer/headline only, fetching the full text on demand if Claude references it, or (2) inject a ranked summary that compresses N memories into a single structured block. Gap: ~3,570 - 450 = 3,120 tokens at p50.

- **Flat injection ordering**: All K_post_dedup memories are injected at the same weight. An adaptive budget would inject the top-3 at full length and truncate memories 4-28 to headlines only (50 tokens each). Gap: (28 × 150) - (3 × 150 + 25 × 50) = 4,200 - 1,700 = 2,500 tokens saved.

- **No injection deduplication across session-reminder block**: The `<system-reminder>` injection has no mechanism to detect when a memory was already injected in the prior session and is still in the conversation window (for resumed sessions). Re-injecting memories already in context wastes tokens proportional to session continuity rate.

- **K parameters are static**: K_mbc=10 and K_graph=20 (or 50) are hardcoded. The optimal K depends on the current conversation depth, available context window, and memory size distribution. A dynamic K that adapts to remaining context budget would eliminate the risk of injection exceeding budget under pathological memory sizes (500-word memories).

#### Implementation Gaps (code quality)

- **No token-count check before injection**: The recall-retrieve.js hook does not measure the total token count of the assembled injection before inserting it into the system-reminder. If a session accumulates long memories (p99 = 670 tokens/memory), the K=50 path injects 58 × 670 = 38,860 tokens — approaching 20% of the context window.

- **Header format is verbose**: The current header format (based on inspect of session summary outputs) likely includes full ISO-8601 timestamps and complete UUIDs. A compact format (epoch integer + 6-char UUID prefix) reduces header overhead from 20 to ~10 tokens/memory.

### Overflow Threshold Analysis

**Context budget assumptions**:

The Claude Sonnet 4.6 200K context window is consumed by:
- CLAUDE.md global instructions + rules files: ~8,500 tokens (measured from file sizes in this session)
- Tool definitions (available deferred tools, MCP tools): ~2,000 tokens
- Memory injection (LLPC system-reminder): variable
- Conversation history + in-session work: grows throughout session
- **Fixed overhead (non-memory)**: ~10,500 tokens
- **Available for memory + conversation**: ~189,500 tokens

A 10K-token memory budget (injected once at session start) = 5.3% of the available window — this leaves 179,500 tokens for conversation, tool calls, and tool responses, which is sufficient for a full multi-hour session.

**Conservative budget: 10,000 tokens = 67 memories at 150 tokens each.**

At the current K parameters (K_post_dedup ≈ 28), the injection is 4,200 tokens — **42% of the conservative budget, with 58% headroom**.

At Q083's K=50 relaxation (K_post_dedup ≈ 58), the injection is 8,700 tokens — **87% of the conservative budget, with 13% headroom**.

**Overflow risk exists only under pathological memory sizes**:

| Memory length | K=20 path injection | K=50 path injection | Exceeds 10K budget? |
|---|---|---|---|
| 50 words (p10) | 1,540 tokens | 3,190 tokens | No |
| 100 words (p50, baseline) | 4,200 tokens | 8,700 tokens | No (K=50 marginal) |
| 200 words (p80) | 7,980 tokens | 16,530 tokens | K=50 yes |
| 300 words (p90) | 11,340 tokens | 23,490 tokens | Both paths yes |
| 500 words (p99) | 18,760 tokens | 38,860 tokens | Both paths; K=50 approaches 20% of full window |

**The overflow risk is triggered by memory length, not corpus size.** At Tim's current corpus (~500 memories, average 100-word memories), neither K=20 nor K=50 overflows a 10K-token budget. The risk materializes at p90+ memory lengths (300+ words) with K=50.

**Corpus-size invariance**: Injection size does not grow with C once C > 200. The K-bounded ceiling is a fixed architecture property. A corpus of 50,000 memories injects the same token count as a corpus of 500 memories — the working-set filter removes cold memories and the K gate limits the returned set.

### K=50 Impact at Tim's Current Scale (C≈500)

From Q083, K can safely be raised from 20 to 50 for the ranking LIMIT in the Cypher query (the traversal cost is unchanged — Neo4j reads all F relationship chains regardless of LIMIT). The token budget impact:

| Path | Post-dedup memories | Injection tokens | % of 200K window | % of 10K budget |
|---|---|---|---|---|
| K=20 (current) | 27.5 | 4,125 tokens | 2.06% | 41.3% |
| K=50 (Q083 relaxed) | 57.5 | 8,625 tokens | 4.31% | 86.3% |
| Delta | +30.0 | +4,500 tokens | +2.25% | +45.0% |

**At C≈500 and 100-word average memories**: K=50 is safe. The 8,625-token injection is within the 10K budget (86% utilization). There is no overflow risk.

**The concern at K=50** is not the context window — it is the **budget headroom shrinking to 13%** (1,375 tokens). If Tim's average memory length grows from 100 words toward 150-200 words (as session summaries become more detailed with project maturity), K=50 would push the injection toward or past 10K tokens. This is not a current emergency but is a forward-looking constraint.

**Recommendation**: K=50 is safe at current scale and memory lengths. Add a token-count guard in recall-retrieve.js that measures the injection size before inserting it, and falls back to K=20 candidates if the total would exceed a configurable threshold (default: 8,000 tokens). This guard costs one `str.length` check and eliminates the memory-length pathology.

### Deduplication Analysis

**MBC and Neo4j BFS return overlapping UUIDs when**:
1. A memory is both a tag-intersection hit (MBC) AND a causal neighbor of a current LLPC-seeded memory (BFS).
2. This condition is satisfied by "hub" memories: high `access_count`, frequently co-retrieved, AND matching current context tags.

**Expected overlap rate calculation**:

At tau=20 with K_mbc=10 and K_graph=20:
- MBC returns 10 memories ranked by tag-intersection score from the active corpus.
- BFS returns 20 causal neighbors ranked by `co_occurrence_count` from LLPC seed nodes.
- Overlap = memories appearing in both sets.

In information retrieval, the expected intersection size between two top-K ranked lists from independent rankers on the same corpus scales as:
```
E[|A ∩ B|] ≈ min(K_A, K_B) × (min(K_A, K_B) / N_active)
            = 10 × (10 / 166) = 0.60 memories (independence model)
```

The independence model underestimates overlap because MBC and BFS are NOT independent — both favor high-access-frequency memories. Applying a coupling factor of 3-5× for access-frequency correlation:

```
E[|A ∩ B|] ≈ 0.60 × 4 = 2.4 memories
```

This aligns with the 2-3 overlap estimate used in the token budget calculations above.

**Effective post-dedup candidate count**:

| K_graph | Pre-dedup | Expected overlap | Post-dedup (expected) | Post-dedup (worst-case no overlap) |
|---|---|---|---|---|
| 20 | 30 | 2.4 | 27.6 ≈ 28 | 30 |
| 50 | 60 | 2.4 | 57.6 ≈ 58 | 60 |

At K_graph=50, the overlap does not increase significantly because the additional 30 graph neighbors (positions 21-50) are lower-frequency causal neighbors less likely to appear in MBC's tag-intersection top-10.

## Key Finding

The LLPC injection is **bounded by the K parameters, not by corpus size**. Once C exceeds approximately 200 memories (which Tim's deployment already has), the working-set filter is irrelevant to injection size — there are always more active memories than K allows. The injection ceiling is:

- **K=20 path**: 28 post-dedup memories × 150 tokens = **4,200 tokens** = 2.1% of the 200K context window
- **K=50 path**: 58 post-dedup memories × 150 tokens = **8,700 tokens** = 4.35% of the 200K context window

At Tim's current scale (C≈500, 100-word average memories), **neither K=20 nor K=50 overflows any reasonable memory budget**. The 200K context window is not at risk. The 10K-token conservative budget has 58% headroom at K=20 and 13% headroom at K=50.

The overflow risk is **memory-length-driven**, not corpus-size-driven or K-driven within the ranges analyzed. A session that stores 300-word summaries (p90) with K=50 injection would push the injection to 23,490 tokens — exceeding both the 10K conservative budget and likely crowding conversation history. This scenario requires a token-count guard, not a K reduction.

**Verdict rationale (BREAKTHROUGH)**: The corpus-size invariance of injection size is a non-obvious structural property of the K-bounded retrieval design. The intuition that "more memories = more injection" is false above C≈200. The binding constraint is always K, never the working-set filter at Tim's scale. This changes the correct framing for future architecture decisions: injection growth is controlled entirely by K parameter changes, making K a first-class configuration dial rather than a performance optimization. The finding also reveals that K=50 is safe at current scale (no overflow), resolving the uncertainty from Q083's K relaxation suggestion.

## Ideas Extracted

### Idea: token-count-guard-before-injection
- **Novelty**: 0.70 — Measuring the total byte/token count of the assembled memory injection before inserting it into the system-reminder block, and truncating or falling back to a smaller K if the budget would be exceeded, is standard practice in production RAG systems (LangChain's `max_tokens_limit` parameter, LlamaIndex's `TokenCountingHandler`) but has not been applied to the Recall hook architecture. The novelty is in applying this mechanism to the Claude Code hook injection path specifically.
- **Evidence**: 0.82 — Production RAG systems universally implement token-count guards to prevent context overflow. LangChain's `ContextualCompressionRetriever`, LlamaIndex's `SentenceEmbeddingOptimizer`, and Anthropic's Claude documentation all emphasize token budget management as a first-class concern. The specific implementation (pre-injection length check in recall-retrieve.js with fallback to K=20 from K=50) is straightforward.
- **Feasibility**: 0.95 — Implementation is a single conditional in recall-retrieve.js: count the total character length of the assembled injection block (÷4 for approximate token count), compare to a configurable threshold (default 8,000 tokens × 4 chars ≈ 32,000 chars), and if exceeded, re-run retrieval with K_graph=20 instead of K_graph=50. No schema changes required. The character-counting approach avoids a full tokenizer dependency. Estimated implementation: 2 hours.
- **Description**: In recall-retrieve.js, after assembling the injection block from retrieved memories, measure `injection_block.length / 4` as an approximate token count. If this exceeds a configurable `MAX_INJECTION_TOKENS` threshold (default: 8,000 tokens), truncate the injection to the top-`floor(MAX_INJECTION_TOKENS / estimated_tokens_per_memory)` memories ranked by relevance score. Log the truncation event for monitoring (feeds into S1 health signal). This guard eliminates the memory-length pathology (300-500 word summaries at K=50) without requiring a static K reduction.

### Idea: tiered-injection-headlines-then-full
- **Novelty**: 0.75 — Injecting a short headline (25-token summary of the memory) for all K retrieved memories, then injecting full text only for the top-3 highest-ranked memories, is analogous to the "progressive disclosure" pattern in search engines (snippet + full document on click). Applied to memory injection, this reduces the p50 injection from 4,200 tokens to approximately 3 × 150 + 25 × 25 = 450 + 625 = 1,075 tokens — a 75% reduction while retaining the full text for the most relevant memories. No reviewed AI memory system implements this tiered injection pattern.
- **Evidence**: 0.68 — Progressive disclosure in retrieval is well-established in web search (BM25 snippet generation, Elasticsearch highlight API) and in RAG systems (LlamaIndex's `compact_and_refine` response synthesis mode, which iteratively processes document chunks). The specific application to the Recall hook injection path — injecting headlines for low-ranked memories and full text for high-ranked ones — is a direct engineering application of this pattern but has not appeared in published AI memory system work.
- **Feasibility**: 0.80 — Requires the Recall API to return both a `headline` field (first 25 tokens of the memory body, already available as a substring) and the full `content` field for each memory. The recall-retrieve.js hook assembles the injection block with full text for positions 1-3 and headline-only for positions 4-K. The Recall API endpoint `/recall/retrieve` already returns full content; adding `headline` is a 1-line Python substring operation in the retrieve endpoint. Estimated implementation: 4-6 hours.
- **Description**: Modify the LLPC injection block format to use two tiers: (1) **Full injection** for the top-3 memories by relevance score: complete body text at 150 tokens each = 450 tokens. (2) **Headline injection** for memories 4-K: first 25 tokens of body + 10-token header = 35 tokens each. At K_post_dedup=28: 3 × 150 + 25 × 35 = 450 + 875 = 1,325 tokens total — a 68% reduction from the current 4,200-token p95 injection. The headline format: `[RECALL] {domain} | {primary_tag}: {first_sentence}`. This is sufficient for Claude to recognize if the memory is relevant to the current prompt and optionally ask for the full text.

### Idea: k-adaptive-to-remaining-context-budget
- **Novelty**: 0.72 — Dynamically setting K at session start based on the estimated remaining context budget (context window minus already-consumed tokens from system prompt, tool definitions, and conversation history) rather than using a static K is not documented in reviewed AI memory systems. The pattern is established in LLM inference serving (dynamic context length adjustment, speculative decoding token budget) but has not been applied to retrieval K selection in the Recall hook architecture.
- **Evidence**: 0.65 — Adaptive token budget allocation is a documented challenge in production LLM systems. Anthropic's prompt caching, OpenAI's structured output token limiting, and LangChain's `max_token` parameters all address this concern. The specific mechanism — reading the current conversation token count from the Claude API context at hook fire time and computing K as `floor(remaining_budget / TOKENS_PER_MEMORY)` — requires the hook to have access to the current token count, which is available via the Claude Code hook event payload (the `conversation_length` or equivalent field).
- **Feasibility**: 0.65 — Depends on whether the `UserPromptSubmit` hook payload includes a token count estimate for the current conversation context. If available, K adaptation is a 5-line calculation. If not available, the hook can estimate from the character length of the `last_n_turns` field (÷4 for tokens). This approach introduces session-to-session variability in injection size, which complicates the S1 IPS health signal (Q071) — the exploration oracle must normalize P@3 by K, not assume fixed K. Medium implementation complexity.
- **Description**: At `UserPromptSubmit` hook fire time, estimate the current conversation token consumption from available payload fields. Compute `remaining_budget = max_injection_budget - current_tokens`. Set `K_graph = floor((remaining_budget - K_mbc × TOKENS_PER_MEMORY) / TOKENS_PER_MEMORY)`, clamped to [10, 50]. This allocates the memory budget adaptively: early in a session with a fresh context, K=50 injects the full Q083 relaxed set; late in a long session with a deep context, K falls to 10-15 to preserve conversation space. The IPS health signal must be updated to normalize P@3 by session K to avoid a spurious S1 degradation signal during long sessions.
