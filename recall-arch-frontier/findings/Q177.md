# Q177 — audit_log DELETE Lock Contention Adversarial Analysis

**Question**: If the audit_log retention worker (Q168) deletes 200,000 rows on its first run and the `get_positive_feedback_counts()` query runs concurrently — does the DELETE acquire a table-level lock that blocks the concurrent SELECT, and if so, does the 500ms retrieval SLA break during the 4am retention window?
**Agent**: quantitative-analyst
**Verdict**: CONFIRMED_SAFE

---

## Thread A (Optimistic): No SLA Impact

**Core claim**: A 200,000-row DELETE on audit_log cannot block concurrent SELECTs on the same table due to PostgreSQL's MVCC model, and the function named in the question is not called during memory retrieval at all — it is a health dashboard query with a 10-minute cache.

**Evidence**:

PostgreSQL's MVCC (Multi-Version Concurrency Control) model provides the fundamental guarantee: readers never block writers and writers never block readers. The lock compatibility matrix is definitive:

| Lock Mode | Acquired by | Compatible with AccessShareLock (SELECT)? |
|---|---|---|
| AccessShareLock | SELECT | Yes — with itself |
| RowShareLock | SELECT FOR UPDATE | Yes |
| RowExclusiveLock | INSERT, UPDATE, DELETE | **Yes** |
| ShareUpdateExclusiveLock | VACUUM, ANALYZE | Yes |
| ShareLock | CREATE INDEX | No |
| ExclusiveLock | REFRESH MATERIALIZED VIEW | No |
| AccessExclusiveLock | DROP TABLE, ALTER TABLE | No |

A `DELETE FROM audit_log WHERE timestamp < $1` acquires a `RowExclusiveLock`. A `SELECT ... FROM audit_log` acquires an `AccessShareLock`. These two lock modes are **explicitly compatible** in PostgreSQL's lock compatibility table. The DELETE does not block the SELECT. Full stop.

The second part of the argument is stronger: the function referenced in the question, `get_feedback_counts_by_memory()`, is not called during memory retrieval. Code verification (see below) confirms it is called only from `compute_memory_quality()`, which is the `/health/memory-quality` dashboard endpoint. That endpoint itself has a 10-minute Redis TTL cache (`MEMORY_QUALITY_CACHE_TTL = 600`). At 4am, with no active users, the cache is almost certainly warm from the last dashboard access, meaning the underlying SQL query may not even execute at all.

**Best case scenario**: The retention DELETE runs at 4am, completes in under 10 seconds, generates ~70MB of WAL writes, triggers autovacuum on the dead tuple heap, and no SELECT against audit_log experiences any blocking whatsoever. The memory retrieval SLA is completely unaffected because the retrieval pipeline never touches audit_log at query time.

## Thread B (Pessimistic): I/O Saturation Risk

**Core claim**: Even without lock blocking, a 200,000-row single-transaction DELETE generates sufficient I/O pressure to transiently degrade query latency on a CasaOS system with shared storage, and autovacuum compounds this pressure after the DELETE commits.

**Evidence**:

**WAL amplification calculation**:

The `audit_log` schema:
```sql
CREATE TABLE audit_log (
    id          BIGSERIAL PRIMARY KEY,  -- 8 bytes
    timestamp   TIMESTAMPTZ NOT NULL,   -- 8 bytes
    action      TEXT NOT NULL,          -- variable, ~10-20 bytes avg
    memory_id   TEXT NOT NULL,          -- ~36 bytes (UUID)
    actor       TEXT,                   -- ~10 bytes
    details     JSONB,                  -- variable, ~100-500 bytes (the largest field)
    session_id  TEXT,                   -- ~36 bytes
    user_id     INTEGER                 -- 4 bytes
);
```

Average row size: approximately 250-400 bytes of heap data. Each row deletion generates:
- A heap tuple visibility update (setting `t_xmax`) in the heap page
- A WAL record for that heap update: ~50-100 bytes of WAL overhead per row

For 200,000 rows: 200,000 × 75 bytes = ~15 MB of WAL writes (not 70 MB — the 70 MB estimate in the brief overestimates by assuming full row size per WAL record; WAL records for DELETE are header-only plus row TID, not full row content). However, the heap pages themselves are read from disk during the sequential index scan and written back with updated `t_xmax` bits. If the 400K-row table's heap is larger than the shared_buffers pool (default 128MB in a Docker container, but CasaOS may have a constrained allocation), the DELETE will generate significant buffer cache thrashing.

**Autovacuum timing**: Autovacuum fires *after* the DELETE transaction commits, not during. The `autovacuum_vacuum_threshold` default is 50 rows, and `autovacuum_vacuum_scale_factor` default is 0.2. After 200,000 dead tuples are created, autovacuum triggers almost immediately post-commit. On a CasaOS system (eMMC or SATA SSD, not NVMe), autovacuum processing 200,000 dead tuples across the audit_log heap means: sequential read of all heap pages containing dead tuples + write of updated visibility maps + FSM (free space map) updates. This is a sustained I/O operation that may take 30-120 seconds on slow storage.

**The shared I/O bus risk**: CasaOS runs all Docker services on the same host. PostgreSQL, Qdrant, Redis, Neo4j, and Ollama all share the same storage I/O bus. A sustained autovacuum pass on 200,000 dead tuples reads and writes the audit_log heap pages. If other queries (Qdrant vector search, Redis operations) compete for the same I/O bus during this window, tail latencies increase. On NVMe, this is negligible. On eMMC (found in some NAS/mini-PC hardware), 100 MB/s sequential I/O can become a bottleneck.

**Failure mode**: Retrieval latency at P99 increases from ~200ms to 400-600ms for 30-120 seconds during the autovacuum window immediately following the DELETE. This is transient and bounded but could technically break the 500ms SLA at P99 — not at median — on slow storage hardware.

## Code Verification

The question asks about `get_positive_feedback_counts()` — this exact function name does not exist in the codebase. The closest function is `get_feedback_counts_by_memory()` in `src/storage/postgres_store.py` (line 867):

```python
async def get_feedback_counts_by_memory(self) -> dict[str, dict[str, int]]:
    """Get positive/negative feedback counts per memory from audit_log."""
    rows = await self.pool.fetch(
        """
        SELECT memory_id,
               count(*) FILTER (WHERE details->>'useful' = 'true') AS positive,
               count(*) FILTER (WHERE details->>'useful' = 'false') AS negative
        FROM audit_log
        WHERE action = 'feedback'
        GROUP BY memory_id
        """
    )
    return {
        r["memory_id"]: {
            "positive": r["positive"] or 0,
            "negative": r["negative"] or 0,
        }
        for r in rows
    }
```

## Lock Matrix Analysis

**PostgreSQL lock compatibility: DELETE vs SELECT on audit_log**

| Operation | Lock Acquired | Conflicts with AccessShareLock? |
|---|---|---|
| `DELETE FROM audit_log WHERE ...` | RowExclusiveLock | **No** — compatible |
| `SELECT ... FROM audit_log` | AccessShareLock | No — compatible with all row-level locks |
| `VACUUM` (autovacuum) | ShareUpdateExclusiveLock | No — compatible with AccessShareLock |
| `CREATE INDEX CONCURRENTLY` | ShareUpdateExclusiveLock | No |
| `ALTER TABLE` | AccessExclusiveLock | **Yes — blocks everything** |

Conclusion: No lock contention scenario exists between a DELETE and a concurrent SELECT in PostgreSQL's MVCC model. The question's premise ("does the DELETE acquire a table-level lock") is incorrect — PostgreSQL DELETE acquires row-exclusive locks at the table level, which are explicitly designed to be compatible with readers.

**The only locking scenario that would block SELECTs would be a `LOCK TABLE audit_log IN ACCESS EXCLUSIVE MODE` statement — which the proposed retention worker does not issue.**

## Synthesis

**What Thread A assumes that Thread B exposes**:

Thread A assumes the host storage is fast enough that a sustained autovacuum pass on 200,000 dead tuples is invisible to concurrent queries. This is valid on NVMe but not guaranteed on eMMC storage (found in some Intel NUC and mini-PC CasaOS deployments). The shared I/O bus is a hidden assumption that Thread A treats as a constant rather than a variable.

**When Thread A is right**:
- When `get_feedback_counts_by_memory()` is the function being analyzed: always. It runs behind a 10-minute Redis cache and is only called from the health dashboard, not from the retrieval pipeline. The SLA question is moot for this specific function.
- When the host has NVMe storage or sufficient shared_buffers to hold the audit_log heap in RAM: the I/O pressure from DELETE + autovacuum is too small to matter.
- At 4am with low or zero active sessions: even if I/O is briefly saturated, no retrieval requests are in flight.

**When Thread B is right**:
- When the real concern is P99 latency on *any* audit_log SELECT (not just `get_feedback_counts_by_memory()`): other health dashboard queries that scan audit_log (`get_feedback_stats`, `get_action_counts`, `get_noisy_memories`) could experience elevated latency during the autovacuum window.
- When CasaOS runs on eMMC storage: the autovacuum I/O burst on 200,000 dead tuples (reading and rewriting heap pages) competes with Qdrant's mmap I/O and Redis AOF writes on a slow shared bus.
- When autovacuum is configured with default `autovacuum_vacuum_cost_delay = 2ms` (its throttling parameter): this actually *helps* Thread B's case — default autovacuum is already throttled to avoid saturation, meaning the 200,000-tuple cleanup is spread over many minutes rather than a single burst.

**The architecture Thread B would accept**:

A **batched DELETE with explicit inter-batch sleep** reduces the peak WAL write rate and gives the autovacuum process time to catch up between batches. Instead of committing 200,000 deletions in one transaction:

```python
async def run_audit_log_retention(pool, cutoff_days=90, batch_size=10_000, sleep_ms=100):
    cutoff = datetime.now(UTC) - timedelta(days=cutoff_days)
    total_deleted = 0
    while True:
        result = await pool.execute(
            """
            DELETE FROM audit_log
            WHERE id IN (
                SELECT id FROM audit_log
                WHERE timestamp < $1
                LIMIT $2
            )
            """,
            cutoff, batch_size
        )
        rows_deleted = int(result.split()[-1])
        total_deleted += rows_deleted
        if rows_deleted < batch_size:
            break  # Done
        await asyncio.sleep(sleep_ms / 1000)
    return total_deleted
```

This approach:
1. Commits 10,000 rows at a time → WAL pressure is ~150KB per batch rather than ~15MB total in a burst
2. The 100ms sleep between batches allows autovacuum to reclaim dead tuples incrementally, keeping the heap's dead-tuple ratio below autovacuum's trigger threshold per batch
3. Spreads the 200,000-row deletion over ~20 batches × 100ms = ~2 seconds minimum spread time (plus actual delete execution time), trading a 10-second burst for a 30-60 second graceful drain
4. Each individual transaction is small enough that WAL checkpointing catches up between batches

**The third option neither thread considered**:

The real architectural fix is not to optimize the DELETE — it is to move `get_feedback_counts_by_memory()` to use its result from the existing Redis cache everywhere, and ensure no health dashboard query is called at 4am without caching. The health dashboard's `/health/memory-quality` endpoint already caches for 10 minutes. The SLA question dissolves entirely: if no health dashboard request arrives at 4am (likely — it's 4am and Tim is asleep), the cache is served from the last warm result. No SQL hits audit_log at all during the retention window.

## Recommendation

**Primary: Batched DELETE is the safer implementation.**

Despite the locking analysis confirming zero lock contention risk, the I/O burst case from Thread B is real on slower storage. The batched DELETE pattern costs nothing in correctness and provides meaningful insurance against autovacuum I/O pressure on eMMC/SATA-class storage.

**Recommended retention worker spec**:

```python
# In src/workers/hygiene.py or a new src/workers/audit_retention.py
# Scheduled via APScheduler: monthly at 04:00

async def run_audit_log_retention(
    pool: asyncpg.Pool,
    cutoff_days: int = 90,
    batch_size: int = 10_000,
    sleep_between_batches_ms: int = 150,
) -> dict:
    """
    Delete audit_log rows older than cutoff_days in batches.
    Batching prevents WAL burst and allows autovacuum to keep pace.
    """
    cutoff = datetime.now(UTC) - timedelta(days=cutoff_days)
    total_deleted = 0
    batches = 0

    while True:
        result = await pool.execute(
            """
            DELETE FROM audit_log
            WHERE id IN (
                SELECT id FROM audit_log
                WHERE timestamp < $1
                ORDER BY id
                LIMIT $2
            )
            """,
            cutoff,
            batch_size,
        )
        rows_deleted = int(result.split()[-1])
        total_deleted += rows_deleted
        batches += 1

        if rows_deleted < batch_size:
            break

        await asyncio.sleep(sleep_between_batches_ms / 1000.0)

    await pool.execute("ANALYZE audit_log")  # Update statistics post-deletion
    return {"total_deleted": total_deleted, "batches": batches, "cutoff": cutoff.isoformat()}
```

**Additional safeguard**: The existing BTREE index on `timestamp` (from Q161's recommendation to replace with BRIN) should be done *before* the first retention run. The DELETE subquery uses `ORDER BY id` (primary key) with a `timestamp < $1` filter — this benefits from the timestamp index to identify candidate rows efficiently before deleting by PK.

**On the SLA question directly**: The 500ms retrieval SLA does not depend on `get_feedback_counts_by_memory()`. The retrieval pipeline (`src/core/retrieval.py`) does not query audit_log. The question's premise is based on an incorrect assumption about which path calls this function. The actual call site (health dashboard, Redis-cached) makes the SLA question moot: at 4am, the cache is warm from the last dashboard view or the endpoint is never called, so the underlying SQL does not execute.

## Key Findings

1. **The function `get_positive_feedback_counts()` does not exist in the codebase.** The closest match is `get_feedback_counts_by_memory()` in `postgres_store.py` line 867, which queries `audit_log WHERE action = 'feedback'` with a full table GROUP BY.

2. **`get_feedback_counts_by_memory()` is NOT called from the retrieval pipeline.** It is called only from `compute_memory_quality()` in `src/core/health.py`, which is invoked by the `/health/memory-quality` dashboard endpoint — a completely separate code path from memory retrieval.

3. **The `/health/memory-quality` endpoint has a 10-minute Redis cache** (`MEMORY_QUALITY_CACHE_TTL = 600` in `health_dashboard.py`). At 4am, this cache is likely warm or the endpoint is simply never called. The underlying SQL does not execute during the retention window.

4. **PostgreSQL lock compatibility confirms zero blocking risk.** `DELETE` acquires `RowExclusiveLock`; `SELECT` acquires `AccessShareLock`. These are explicitly compatible — no blocking can occur regardless of table size or transaction duration.

5. **The I/O saturation risk is real but bounded.** A 200,000-row DELETE generates ~15 MB of WAL writes (not 70 MB as initially estimated — WAL records for DELETE store row TID and header, not full row content). Autovacuum fires post-commit and is the larger I/O consumer, spread over 30-120 seconds depending on storage speed.

6. **Batched DELETE (10,000 rows, 150ms sleep) is the correct implementation** regardless of whether locking is a concern. It limits peak WAL write rate per batch to ~150KB, allows autovacuum to reclaim dead tuples incrementally, and does not meaningfully increase total execution time for a monthly 4am job.

7. **The real architectural protection is the Redis cache.** Since all audit_log health queries are cached, even if the retention DELETE + autovacuum temporarily elevated audit_log query latency, no SELECT against audit_log would execute at 4am unless the cache was cold — an unlikely scenario for a health dashboard that was last viewed during business hours.
