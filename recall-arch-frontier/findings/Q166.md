## Minimum Bridge Implementation

### 1. Cypher MERGE query (bidirectional, weight-accumulating)

```cypher
UNWIND $pairs AS pair
MATCH (a:Memory {id: pair.uuid_a}), (b:Memory {id: pair.uuid_b})
MERGE (a)-[r:CO_RETRIEVED]-(b)
ON CREATE SET
    r.weight = 1,
    r.created_at = $now
ON MATCH SET
    r.weight = r.weight + 1,
    r.last_reinforced = $now
```

Note: The undirected `-(b)` (not `->`) matches the symmetric semantics of co-retrieval — A and B were retrieved together regardless of direction. The `weight` field is a raw integer count, distinct from the `strength` float (0-1) used on RELATED_TO edges, to avoid semantic collision.

### 2. Outbox SELECT query

```sql
SELECT id, memory_uuid_a, memory_uuid_b
FROM co_retrievals_outbox
WHERE processed_at IS NULL
LIMIT 1000
FOR UPDATE SKIP LOCKED
```

The `FOR UPDATE SKIP LOCKED` is required to make concurrent worker runs safe — each run locks only the rows it will process and skips rows another worker has locked.

### 3. Minimal Python function signature

```python
async def flush_co_retrievals_to_neo4j(ctx: dict) -> None:
    """
    Drain co_retrievals_outbox rows into Neo4j CO_RETRIEVED edges.

    Reads up to 1000 unprocessed rows from co_retrievals_outbox,
    fires a batched UNWIND MERGE against Neo4j, then marks rows
    as processed via UPDATE processed_at = now().
    """
```

Full minimal implementation:

```python
# src/workers/flush_co_retrievals.py

from datetime import UTC, datetime
import structlog
from src.storage import get_neo4j_store, get_postgres_store

logger = structlog.get_logger()
_BATCH_LIMIT = 1000


async def flush_co_retrievals_to_neo4j(ctx: dict) -> None:
    pg = await get_postgres_store()
    neo4j = await get_neo4j_store()
    now = datetime.now(UTC).isoformat()

    async with pg.pool.acquire() as conn:
        async with conn.transaction():
            rows = await conn.fetch(
                """
                SELECT id, memory_uuid_a, memory_uuid_b
                FROM co_retrievals_outbox
                WHERE processed_at IS NULL
                LIMIT $1
                FOR UPDATE SKIP LOCKED
                """,
                _BATCH_LIMIT,
            )
            if not rows:
                return

            pairs = [
                {"uuid_a": r["memory_uuid_a"], "uuid_b": r["memory_uuid_b"]}
                for r in rows
            ]
            ids = [r["id"] for r in rows]

            async with neo4j.driver.session() as session:
                await session.run(
                    """
                    UNWIND $pairs AS pair
                    MATCH (a:Memory {id: pair.uuid_a}), (b:Memory {id: pair.uuid_b})
                    MERGE (a)-[r:CO_RETRIEVED]-(b)
                    ON CREATE SET r.weight = 1, r.created_at = $now
                    ON MATCH SET r.weight = r.weight + 1, r.last_reinforced = $now
                    """,
                    pairs=pairs,
                    now=now,
                )

            await conn.execute(
                "UPDATE co_retrievals_outbox SET processed_at = now() WHERE id = ANY($1::bigint[])",
                ids,
            )

    logger.info("co_retrievals_flushed", count=len(ids))
```

## Key Findings

1. **Zero production code paths write CO_RETRIEVED edges to Neo4j.** The string `CO_RETRIEVED` does not appear in any Python file in the Recall codebase (confirmed by exhaustive grep across `src/`, including workers, storage, core, and API layers).

2. **The term `co_retrieval` appears only 5 times in the entire Python codebase**, all in `health.py` or its test. All 5 occurrences are a local Python variable that proxies RELATED_TO edge strength — semantically distinct from the missing CO_RETRIEVED edge type.

3. **The `co_retrievals_outbox` table is never read.** The string appears nowhere in any `.py` file. The PostgreSQL outbox accumulates rows with no consumer.

4. **`RelationshipType` enum does not define CO_RETRIEVED.** The 12 defined types (RELATED_TO, CAUSED_BY, SUPERSEDES, etc.) do not include CO_RETRIEVED. Any production write via `create_relationship()` would require enum registration.

5. **Neo4j already has a `_safe_rel_type()` validator** (lines 129-134 in `neo4j_store.py`) that would accept `CO_RETRIEVED` without modification — it simply validates `^[A-Z_][A-Z0-9_]*$` regex.

6. **The `strengthen_relationships_batch()` method is the correct pattern to extend.** The UNWIND + MERGE pattern at lines 221-240 is directly reusable for CO_RETRIEVED edges, substituting `weight` accumulation for `strength` float capping.

7. **Q163's TIMESHIFTED projection is empirically confirmed correct.** The prior finding that "no promotion pathway exists from the PostgreSQL outbox to Neo4j CO_RETRIEVED edges" holds against exhaustive grep with zero counter-evidence.

8. **The health.py `co_retrieval_gravity` metric currently measures RELATED_TO strength, not actual co-retrieval signal.** This is a latent semantic bug — the metric is named for one concept but measured from another. Bridging the outbox would require a second patch in `health.py` to read CO_RETRIEVED edges.
