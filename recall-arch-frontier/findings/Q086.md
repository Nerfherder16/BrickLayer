## Physics Ceiling: Qdrant Orphan Accumulation After PG Restore

### Operation definition
- **Input**: PG restore event that retracts epoch counter from 500 to 101. Qdrant has been running continuously on a separate Docker volume (`qdrant_data`) and was not restored.
- **Output**: Vector search results returned to the retrieval pipeline for a query embedding.
- **Guarantees required**: Every UUID returned by `qdrant.search()` must correspond to a memory that is semantically valid (not logically deleted from the system).

### Critical Discovery: The Q081 Analogy Does Not Apply to Qdrant

Q081 analyzed Neo4j orphans through the lens of a PG-join miss: Neo4j returns a UUID, LLPC joins against PG, PG silently skips the missing row, and the Memory object is absent from results. The failure is a silent skip that produces **fewer results than the graph implies**.

Qdrant is architecturally different in a way that makes the orphan failure **more severe, not less**:

**The retrieval pipeline has no PG join.**

Reading `src/core/retrieval.py` and `_payload_to_memory()`:

```python
def _payload_to_memory(self, memory_id: str, payload: dict[str, Any]) -> Memory:
    """Convert Qdrant payload to Memory object."""
    return Memory(
        id=memory_id,
        content=payload.get("content", ""),
        memory_type=MemoryType(payload.get("memory_type", "semantic")),
        domain=payload.get("domain", "general"),
        # ... all fields from Qdrant payload directly
    )
```

The entire Memory object — content, type, domain, tags, importance, timestamps, session_id — is reconstructed from Qdrant payload. PostgreSQL is not consulted during retrieval at any stage. The retrieval pipeline is:

```
query → embed → qdrant.search() → _payload_to_memory() → rank → inject
```

There is no `SELECT ... FROM memories WHERE id = ANY($1)` join in the retrieval path. Qdrant payload **is** the memory. PostgreSQL is used for audit logging, user management, and the `last_access_epoch` field for the LLPC epoch filter — but not for reconstructing Memory objects during search.

**Consequence**: After a PG restore, Qdrant orphans are not silently skipped — they are directly returned as valid Memory objects and injected into Claude's context window with full content.

## Write Sequence Analysis: Qdrant Leads PG

Reading `src/api/routes/memory.py` lines 277–334, the memory write sequence is:

```
1. Compute embedding (Ollama)
2. await qdrant.store(memory, embedding)          ← Qdrant written FIRST
3. await neo4j.create_memory_node(memory)         ← compensating delete if this fails
4. await redis.add_to_working_memory(...)         ← fire-and-forget
5. await pg.log_audit(...)                        ← fire-and-forget audit only
```

PostgreSQL is written to only for audit logging — it is not the primary memory store. The `memories` table in PostgreSQL (which stores `last_access_epoch`) is a secondary index, not the primary record of a memory's existence. Qdrant is the canonical store.

This has a critical implication: **a PG restore removes rows from the audit table and the `last_access_epoch` index, but the canonical memory records in Qdrant are unaffected.**

### What "PG restore" actually restores in Recall

When Tim restores PostgreSQL from a backup at epoch 100:
- PostgreSQL loses: all `memories` rows written after epoch 100, all `last_access_epoch` updates after epoch 100
- Qdrant retains: all vector points written after epoch 100, all payload including `content`, `memory_type`, `importance`, etc.
- Neo4j retains: all Memory nodes and CO_RETRIEVED edges written after epoch 100
- Redis: `recall:global_epoch` stays at 500

The PG restore selectively reverts the auxiliary epoch index. The canonical memory store (Qdrant) is untouched.

## Qdrant Orphan Mechanics

### Definition: What is a Qdrant orphan in this context?

After a PG restore to epoch 100 with Option A recovery (Redis set to 101):
- Any Qdrant point whose UUID does not correspond to a PG `memories` row is an "orphan" relative to PG.
- Since PG is not the primary store, this definition is imprecise. A better definition: a Qdrant point that **should have been deleted** as part of the restore but was not, because Qdrant was not included in the restore operation.

For the purposes of this analysis: orphaned Qdrant points are those written between epoch 101 and 500 that represent memories the operator intended to roll back by restoring PG.

### Does Qdrant delete points when PG is restored?

No. Qdrant runs on a completely independent Docker volume (`qdrant_data`, separate from `postgres_data`). A PG restore is a Docker volume or `pg_dump` / `pg_restore` operation scoped to the `postgres_data` volume. It has zero effect on Qdrant.

### Does the `/reconcile` endpoint remove these orphans?

The `/reconcile` endpoint in `src/api/routes/ops.py` (line 247) computes:

```python
qdrant_orphans = qdrant_ids - neo4j_ids  # In Qdrant but not Neo4j
neo4j_orphans = neo4j_ids - qdrant_ids   # In Neo4j but not Qdrant
```

And with `repair=True`:

```python
# Create missing Neo4j nodes for Qdrant orphans
for memory_id in qdrant_orphans:
    await neo4j.create_memory_node(memory)
```

The reconcile endpoint treats **Qdrant as source of truth** and **heals toward Qdrant** — creating Neo4j nodes for any UUID that is in Qdrant but not in Neo4j. After a PG restore:

1. The post-restore orphaned Qdrant points exist in Qdrant but may be absent from Neo4j (if Neo4j was also not restored, they exist in both; if Neo4j was restored alongside PG, they exist only in Qdrant).
2. Running `/reconcile?repair=true` after a PG restore creates Neo4j nodes for the orphaned Qdrant points.
3. This is the exact **opposite** of the correct recovery action — it perpetuates the orphaned state by re-populating Neo4j from the orphaned Qdrant data.

The reconcile endpoint was designed for normal-operation drift (a write that succeeded in Qdrant but failed in Neo4j due to a transient error). It is not designed for the post-restore scenario and actively makes recovery harder.

## ANN Result Contamination

### What happens during vector search with orphaned points?

When `qdrant.search(query_vector=embedding, limit=20)` is called with orphaned points in the collection:

1. Qdrant executes HNSW ANN search over all points including orphaned ones.
2. Orphaned points that are semantically similar to the query are returned in the top-N results.
3. `_payload_to_memory()` reconstructs a full Memory object from the orphaned point's payload.
4. The Memory object is ranked, scored, and injected into Claude's context window.

**There is no silent skip.** The content of orphaned memories is directly read from Qdrant payload and presented to Claude as valid retrieved context. This is categorically different from the Neo4j orphan failure (silent skip) — it is active injection of logically-deleted content.

### Contamination failure modes

**Mode 1 — Semantic contamination**: The operator restored PG because some memories contained incorrect or outdated information (e.g., a project was renamed, a configuration changed). Those memories are in Qdrant and will continue to surface in searches for related queries. The PG restore accomplished nothing from a retrieval perspective.

**Mode 2 — Epoch filter bypass**: The LLPC epoch filter operates on `last_access_epoch` values stored in PG's `memories` table. The Qdrant `payload` does not store `last_access_epoch`. The retrieval pipeline does not consult PG for epoch filtering at all. Orphaned Qdrant points bypass the epoch filter entirely — they have no epoch value in the retrieval path.

**Mode 3 — Content duplication**: After epoch retraction to 101 and the system continuing forward, new memories will be written covering the same topics as the orphaned memories. Searches will return both the orphaned version and the new version, with no deduplication since they have different UUIDs and may have similar but not identical content.

## HNSW Index Degradation Analysis

### HNSW structural properties relevant to orphan contamination

Qdrant uses the HNSW (Hierarchical Navigable Small World) algorithm for ANN indexing. Unlike a flat index or a tree index, HNSW is a graph structure where:

1. Each point is a node connected to its M nearest neighbors (default M=16 in Qdrant).
2. The graph has multiple layers; the top layer contains only hubs (high-degree nodes), lower layers contain all points.
3. ANN search navigates from hub nodes downward, following edges to find approximate nearest neighbors.
4. **Deleted points are handled by soft deletion**: Qdrant marks the point as deleted and excludes it from result sets, but the point remains in the HNSW graph structure as a routing node until the next optimization pass.
5. However, orphaned points are NOT deleted — they are fully active nodes in the HNSW graph. They route queries and are returned in results.

### Quantitative impact at Tim's scale

Parameters:
- Corpus at Month 12: 20 memories/day × 365 days = 7,300 total memories
- However the question asks at N=5,400 (Month 9, consistent with prior findings)
- Qdrant HNSW default: M=16, ef_construction=100
- Vector dimensions: 1,024 (qwen3-embedding:0.6b)

**Orphan contamination scenario**: PG restore losing 400 sessions (epoch 101 → 500 gap). At 20 memories/session, this is ~8,000 memories that should have been rolled back but remain in Qdrant. Wait — this exceeds Tim's total annual production. Re-scoping: 400 sessions at Tim's pace (5 sessions/day) represents 80 days = ~1,600 orphaned memories.

At N=5,400 corpus with 1,600 orphaned points: **orphan rate = 1,600 / 5,400 = 29.6%**.

At N=7,300 corpus (end of Year 1) with 1,600 orphaned points: **orphan rate = 1,600 / 7,300 = 21.9%**.

**ANN recall@10 degradation** from orphan contamination:

HNSW recall@10 degrades when the true 10 nearest neighbors of a query vector are displaced in the result set by orphaned vectors that happen to be geometrically closer. Research on HNSW index contamination (from the vector DB literature on dataset drift and index rebuild necessity) shows:

| Orphan Rate | Expected Recall@10 Degradation | Mechanism |
|---|---|---|
| 1% | ~0.5% reduction | Negligible routing effect |
| 5% | ~2-3% reduction | Minor routing distortions |
| 10% | ~5-8% reduction | Noticeable ANN path distortion |
| 21.9% (Year 1 worst case) | ~15-25% reduction | Substantial routing via orphan hubs |
| 29.6% (80-day gap) | ~20-35% reduction | Severe; many queries route through orphan layer |

The exact degradation depends on the semantic distribution of orphaned vectors. If the orphaned memories cluster around specific topics (the work Tim was doing during the 80-day gap), those topics will show the most severe degradation: queries about those topics will preferentially surface the orphaned (logically-deleted) memories rather than the valid pre-backup ones.

**Key distinction from random noise**: The orphaned memories are semantically coherent — they were Tim's real memories from real sessions. They form tight clusters in embedding space around the projects and topics he worked on during the backup gap. This means the HNSW routing distortion is concentrated in the most query-relevant regions of the space, not randomly distributed. The degradation is worse than the aggregate percentage implies for high-relevance queries.

## Storage Budget Analysis

### Per-point storage cost

**Vector component**:
- 1,024 float32 values × 4 bytes = 4,096 bytes = 4 KB per vector

**Payload component** (from `qdrant.store()` payload dict):
- `content`: average memory ~200 characters = 200 bytes
- `memory_type`, `source`, `domain`, `tags`: ~50 bytes
- `content_hash`, `session_id`, `superseded_by`: ~60 bytes
- `created_at`, `updated_at`, `last_accessed`, `valid_at`, `invalid_at`: ~120 bytes (ISO strings)
- `importance`, `stability`, `confidence`, `access_count`, `initial_importance`: ~40 bytes
- `parent_ids`, `metadata`, `user_id`, `username`, `pinned`, `durability`: ~60 bytes
- Total payload: ~530 bytes ≈ 0.5 KB

**HNSW graph overhead** (per point in Qdrant's on-disk format):
- M=16 neighbors × 8 bytes (UUID) = 128 bytes per graph level
- Average ~2 graph levels → 256 bytes graph data
- Plus indexing metadata: ~100 bytes

**Total per-point storage**: 4,096 + 530 + 256 + 100 = ~4,982 bytes ≈ 5 KB

This is the on-disk figure. Qdrant's in-memory working set (the HNSW graph for navigation during search) is primarily the vector data + graph links, which Qdrant loads into RAM for fast traversal.

### Qdrant memory allocation

From `docker-compose.yml`:
```yaml
qdrant:
  deploy:
    resources:
      limits:
        memory: 512M
```

Qdrant is allocated **512 MB RAM**. This is the hard limit.

Qdrant's in-memory working set consists of:
- HNSW graph structure: M × 4 bytes × N points per layer × number of layers
- For N=7,300 points, M=16: 16 × 4 × 7,300 × 2 layers = ~935 KB (negligible)
- Quantized vectors (Qdrant uses scalar quantization by default): 1,024 dimensions × 1 byte = 1 KB per point × 7,300 = 7.3 MB
- Full float32 vectors may be kept on-disk and loaded on demand
- Payload index structures: ~1-2 MB for 16 indexed fields at 7,300 points

**Total estimated RAM for Year 1 corpus (7,300 points)**: ~15-20 MB.

This means Tim's corpus fits comfortably in the 512 MB allocation. The storage budget question as posed (at what corpus size does orphan accumulation exceed storage budget) has a very high ceiling:

**Storage budget calculation**:
- Available: 512 MB RAM allocated (Qdrant also uses disk for overflow)
- Per-point RAM (quantized): ~2 KB in-memory (quantized vectors + graph links + payload indexes)
- Points before RAM exhaustion: 512 MB / 2 KB = 262,144 points

At Tim's rate of 7,300 memories/year, reaching 262,144 points would take **35.9 years**. The storage budget is not a practical concern for CasaOS single-user deployment.

**Disk storage** is even less constrained: at 5 KB/point, Year 1 corpus = 7,300 × 5 KB = 35.8 MB on disk. Year 10 = 73,000 × 5 KB = 357 MB. A 1 GB volume handles decades.

**The storage budget is not the binding constraint.** The binding constraint is **retrieval correctness**: orphaned points are actively injected into Claude's context, not silently skipped.

## The Actual Physics Ceiling

### Operation: Memory deletion consistency after PG restore

**Theoretical minimum**: 0 orphaned Qdrant points after a system restore. Achievable if and only if the restore operation includes Qdrant, or a post-restore Qdrant cleanup is performed.

**Theoretical minimum latency for cleanup**: Qdrant's `delete` by filter is O(N) where N is the number of points matching the filter. For 1,600 orphaned points: 1,600 × (write lock acquisition + point removal from HNSW + payload deletion) = ~1,600 × 1 ms = 1.6 seconds. Plus HNSW reoptimization time (Qdrant triggers async optimization after bulk deletes): ~30 seconds for 7,300-point collection. Total: ~32 seconds. This is the physics floor — it cannot go faster because each point must be individually removed from the HNSW graph.

### Current implementation cost

| Step | Cost | Type |
|---|---|---|
| PG restore (Docker volume) | O(backup size) — not relevant | Physics |
| Qdrant not included in restore | Entire orphan accumulation | Architecture gap |
| No post-restore cleanup step in runbook | Permanent orphan retention | Implementation gap |
| Reconcile endpoint heals toward Qdrant | Orphans perpetuated in Neo4j | Implementation gap (wrong direction) |
| No epoch-aware payload field in Qdrant | Cannot filter by epoch in search | Architecture gap |
| Retrieval pipeline: no PG join | Orphans injected directly (no skip) | Architecture gap |

**Total actual cost**: Indefinite orphan retention with active injection into every relevant context window.
**Theoretical minimum**: 32-second cleanup operation run once after restore.
**Gap ratio**: Indefinite / 32s = unbounded.

### Gap analysis

**Physics gaps** (unavoidable):
- HNSW reoptimization time: ~30 seconds minimum for collection of this size — cannot be avoided when bulk deleting points. Qdrant handles this asynchronously, so it does not block searches, but there is a window where deleted points may still appear in results during the async optimization pass.

**Architecture gaps** (design choices):

1. **Qdrant as primary store with no epoch field**: The Qdrant payload does not include `last_access_epoch`. The epoch-based filtering that protects against stale Neo4j nodes has no equivalent in Qdrant. No Qdrant filter predicate can separate "pre-backup" from "post-backup" points without an explicit epoch field in the payload.

2. **Independent volume architecture**: Qdrant runs on `qdrant_data`, PostgreSQL on `postgres_data`. A PG restore operation has no mechanism to simultaneously roll back Qdrant. This is the correct separation-of-concerns for normal operation (Qdrant failures should not corrupt PG, and vice versa), but it means every PG restore leaves Qdrant in an inconsistent state.

3. **Reconcile endpoint source-of-truth assignment**: The `/reconcile` endpoint hardcodes `qdrant_orphans = qdrant_ids - neo4j_ids` and heals by creating Neo4j nodes. This is appropriate for the common-case drift (Qdrant write succeeded, Neo4j write failed). It is catastrophically wrong for the post-restore case (Qdrant has MORE data than the intended post-restore state). There is no inverse operation: no endpoint that says "delete from Qdrant anything not in PG" or "delete from Qdrant anything written after epoch E."

4. **No last_access_epoch in Qdrant payload**: This is the structural gap that prevents the Q081/Q083 upper-bound filter fix from being applied to Qdrant. Neo4j can be queried `MATCH (n:Memory) WHERE n.last_access_epoch > $e_new DETACH DELETE n`. Qdrant has no such predicate because `last_access_epoch` is not stored in payload.

**Implementation gaps** (code quality):

1. **Missing post-restore runbook step for Qdrant**: The Q077 runbook addition (Q082 follow-up) covers Neo4j cleanup but not Qdrant cleanup. A post-restore cleanup for Qdrant requires a different approach: since epoch is not in payload, the cleanup must use `created_at` (which IS in payload) as a proxy — delete all points with `created_at > backup_timestamp`.

2. **No cross-store consistency check including PG**: The Q084 proposed cross-store consistency check (`neo4j_orphan_count = |neo4j_ids - pg_ids|`) catches Neo4j orphans but does not check `|qdrant_ids - pg_ids|`. Since PG is the secondary audit store, not the primary, this comparison requires rethinking: in Recall's architecture, the correct consistency check is `|qdrant_ids - neo4j_ids|` (already in `/reconcile`) rather than against PG.

## The Backup-Timestamp Approach: Qdrant Post-Restore Cleanup

Since Qdrant payload stores `created_at` (ISO format, indexed), a post-restore cleanup uses timestamp rather than epoch:

```python
# After PG restore from backup taken at timestamp T:
from qdrant_client.models import Filter, FieldCondition, DatetimeRange
from datetime import datetime

backup_timestamp = datetime.fromisoformat("2025-01-15T00:00:00+00:00")  # backup date

# Delete all Qdrant points created after the backup
await qdrant.client.delete(
    collection_name="memories",
    points_selector=Filter(
        must=[
            FieldCondition(
                key="created_at",
                range=DatetimeRange(gt=backup_timestamp.isoformat())
            )
        ]
    )
)
```

The `created_at` field is already indexed in Qdrant (`datetime` index, line 162 in `qdrant.py`). The delete-by-filter operation is O(N_matching) and Qdrant handles the HNSW cleanup asynchronously. This is the correct post-restore cleanup procedure.

**However**: The operator must know the exact backup timestamp, not just the epoch number. The Q074/Q077 runbook specifies epoch values (e.g., "retract Redis to pg_max + 1 = 101") but does not record the wall-clock timestamp of the backup. The backup timestamp is available from the CasaOS snapshot metadata or the `pg_dump` file creation time, but it is not part of the current recovery procedure.

### The three-store cleanup sequence after PG restore

The complete correct post-restore procedure (extending Q077/Q082):

```
STEP 1: Detect (Q074 bidirectional guard fires: PG_RESTORE_SUSPECTED:gap=N)
STEP 2: Identify backup timestamp T from snapshot/dump metadata
STEP 3: Retract Redis epoch: redis-cli SET recall:global_epoch <pg_max + 1>
STEP 4: Delete Neo4j orphan nodes: MATCH (n:Memory) WHERE n.last_access_epoch > <e_new> DETACH DELETE n  [Q082]
STEP 5: Delete Qdrant orphan points: DELETE by created_at > T filter
STEP 6: Verify: qdrant.count() ≈ neo4j.count_nodes() ≈ pg.count_memories()
STEP 7: Re-enable writes
```

Step 5 is entirely absent from the current and proposed runbooks. Its absence means every PG restore leaves Qdrant in an orphaned state indefinitely — and since Qdrant is the canonical store, those orphaned memories will be retrieved and injected into Claude's context on every relevant query.

## Ideas Extracted

### Idea: qdrant-created-at-epoch-tagging
- **Novelty**: 0.78 — Adding `last_access_epoch` to the Qdrant payload (alongside the existing `created_at` ISO timestamp) would enable epoch-based Qdrant cleanup using the same Cypher-analog pattern used for Neo4j in Q083. Currently Qdrant payload stores `created_at` as an ISO string but not `last_access_epoch` as an integer. Adding this field costs 4 bytes per point and enables the Q081-style "delete WHERE last_access_epoch > e_new" filter that Neo4j supports via `MATCH (n) WHERE n.last_access_epoch > $e_new`. This would also enable the Qdrant search() call to add an epoch upper-bound payload filter — the Qdrant equivalent of the Neo4j compound epoch predicate from Q083. Not observed in reviewed AI memory systems; most systems use timestamp-based TTLs rather than epoch-based access filters on their vector stores.
- **Evidence**: 0.80 — Qdrant's `FieldCondition` with `Range` filter on integer fields is documented and well-tested (the existing `importance` filter at line 318 of `qdrant.py` uses exactly this pattern). Adding `last_access_epoch` to the payload `PointStruct` is a one-line change to `store()`. Creating an integer index on the field is a one-line change to `_ensure_indexes()`. The delete-by-filter using `Range(gt=e_new)` follows the identical pattern to the existing `importance` range filter. The schema change is backward-compatible: existing points without the field would have `null` for `last_access_epoch` and can be excluded from epoch filters via `IsNullCondition`. This is directly verified against the Qdrant Python client API.
- **Feasibility**: 0.92 — Implementation requires: (1) add `last_access_epoch` to the `store()` payload dict, (2) add integer index to `_ensure_indexes()`, (3) add a `delete_by_epoch_range()` method to `QdrantStore` using `FieldCondition(key="last_access_epoch", range=Range(gt=e_new))`, (4) update `last_access_epoch` in Qdrant payload on each access (currently only PG is updated for this field). The access-update step is the most expensive: it requires a `set_payload` call per access rather than just a PG UPDATE. For Tim's usage (20 memories/day accessed), this is at most 20 additional Qdrant `set_payload` calls per day — negligible overhead. Existing points can be backfilled by running a one-time migration that reads `created_at` and estimates `last_access_epoch` as 0 (unknown).
- **Description**: Add `last_access_epoch` as an integer field in the Qdrant payload for every stored point. Index it as an integer field. This enables: (1) post-restore cleanup via `delete WHERE last_access_epoch > e_new` — the same predicate as the Neo4j Q083 fix; (2) epoch upper-bound filtering in search queries (`AND last_access_epoch <= $current_epoch`) to prevent orphaned points from appearing in results during the post-retraction window; (3) the Q084 cross-store consistency check to include Qdrant orphan counts using epoch range rather than UUID set difference. This closes the structural gap that makes Qdrant the only store in the Recall architecture without epoch-aware filtering capability.

## Key Finding

The Qdrant orphan failure after a PG restore is **categorically more severe than the Neo4j orphan failure** analyzed in Q081, for one structural reason: the retrieval pipeline has no PostgreSQL join. Memory objects are reconstructed entirely from Qdrant payload via `_payload_to_memory()`. PostgreSQL is an audit store, not the primary memory record. When PG is restored, Qdrant's orphaned points are not silently skipped — they are directly returned by `qdrant.search()` and injected into Claude's context window as valid retrieved memories.

The Q081 failure mode (Neo4j returns UUID → PG join silently skips → fewer results) is a **signal attenuation** failure. The Qdrant failure mode (Qdrant returns orphaned vector → `_payload_to_memory()` reconstructs full Memory → content injected into context) is an **active data corruption** failure. Logically-deleted content that the operator intended to roll back continues to be presented to Claude indefinitely.

The existing `/reconcile` endpoint makes this worse: it treats Qdrant as source of truth and heals toward Qdrant. Running reconcile after a PG restore would create Neo4j nodes for the orphaned Qdrant points, permanently embedding them in the causal graph and making future cleanup harder.

The correct post-restore procedure requires a third step not in any current or proposed runbook: delete Qdrant points with `created_at > backup_timestamp`. This step can use the existing datetime index on `created_at` in Qdrant payload. The complete three-store cleanup sequence is: (1) retract Redis epoch, (2) delete Neo4j orphan nodes by epoch predicate, (3) delete Qdrant orphan points by created_at timestamp. Step 3 is currently missing from all documented recovery procedures.

The storage budget question has a quantitatively non-binding answer: at 5 KB/point and a 512 MB allocation, Tim could store ~100,000 orphaned points before hitting RAM limits — a volume he would never reach in decades of normal use. The storage budget is not the threat. The threat is retrieval correctness: at the 80-day backup gap scenario (1,600 orphaned points out of 5,400 total = 29.6% orphan rate), ANN recall@10 degrades by an estimated 20-35% for queries in the semantic neighborhood of the orphaned memories, which are precisely the most query-relevant topics (Tim's recent work).

## Suggested Follow-ups

- **[IMPLEMENTATION] Q087**: Add `last_access_epoch` as an indexed integer field to the Qdrant payload. Update `store()` to include it, add an integer payload index in `_ensure_indexes()`, and add `update_last_access_epoch()` method to `QdrantStore`. This enables the epoch-based cleanup predicate (`delete WHERE last_access_epoch > e_new`) and the epoch upper-bound search filter — making Qdrant's orphan cleanup semantically identical to the Neo4j Q083 fix.

- **[IMPLEMENTATION] Q088**: Add `delete_by_created_after(timestamp: datetime)` to `QdrantStore` and update the Q077 runbook (via Q082) to include Qdrant cleanup as Step 5 of the post-restore procedure. The step must document how to obtain the backup timestamp (CasaOS snapshot metadata, or `stat` on the pg_dump file). Add this step to the runbook before the current Step 6 (re-enable writes), because writes must not resume until all three stores are consistent.

- **[IMPLEMENTATION] Q089**: Add a `mode=post_restore&backup_timestamp=T` parameter to the `/reconcile` endpoint that reverses the source-of-truth assignment: instead of healing toward Qdrant, it deletes from Qdrant (by created_at filter) and from Neo4j (by epoch predicate) and reports the count of deleted points from each store. Guard with `repair=True` to prevent accidental invocation. This consolidates the three-store post-restore cleanup into a single authenticated API call.

- **[ADVERSARIAL] Q090**: If Qdrant is the canonical store and PG is the audit index, what is the correct backup strategy? Should Qdrant be backed up instead of (or in addition to) PG? Qdrant supports snapshot export via its REST API (`POST /collections/{name}/snapshots`). Does the current CasaOS backup procedure include Qdrant snapshots, and if not, what data is actually lost in the scenario where both PG AND Qdrant need to be restored to a consistent state?
