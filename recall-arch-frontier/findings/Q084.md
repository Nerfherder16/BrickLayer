# Finding: Q084 — Cross-Signal Correlation Rules for S1–S5 Health Dashboard

**Question**: Design the minimal cross-signal correlation rules for all pairwise combinations of health signals S1-S5 (adding S5 for graph traversal epoch compliance from Q078 to the existing S1-S4 from Q071). Determine: which co-alert patterns are unambiguous root-cause indicators (e.g., S3+S4 simultaneously = CasaOS unavailability), whether the S4+S5 co-alert (epoch integrity + graph epoch compliance both failing) can automatically trigger Option A recovery (retract Redis epoch to pg_max+1) rather than requiring operator intervention, and what the false positive rate is for each automated recovery trigger.
**Question Type**: CONVERGENCE
**Verdict**: PROMISING
**Severity**: High
**Source field**: distributed systems observability, causal inference in alert correlation, SRE runbook design, epoch-clock integrity protocols

## Individual Signal Definitions (S1–S5)

| Signal | What it measures | Alert threshold | Failure direction |
|--------|-----------------|-----------------|-------------------|
| **S1**: IPS P@3 delta | LLPC retrieval quality — does IPS reranking beat raw-count ordering? | < 0.00 delta for 3+ consecutive sessions; critical < −0.10 for 7 days | Gradual degradation; no sudden onset |
| **S2**: MBC Level 1 fraction | Tag-vocabulary routing health — are session summaries tagging correctly? | Level 1 fraction < 40% over 7-day window | Gradual drift OR sudden drop if CasaOS unreachable |
| **S3**: Pending queue depth | Write-tier partition — are session summaries reaching CasaOS? | Count > 5 files in `~/.claude/recall-pending/` | Sudden onset when CasaOS unreachable; gradual if hook retry broken |
| **S4**: Epoch integrity delta | Redis vs PostgreSQL epoch consistency — has the epoch clock been disrupted? | `MAX(last_access_epoch)` vs `recall:global_epoch` gap > tau=20 in the **retrograde** direction (`redis_epoch < pg_max`) | Sudden onset; binary (either inverted or not) |
| **S5** (from Q078): Graph traversal epoch compliance | Neo4j BFS staleness — are graph-traversal candidates respecting the working-set boundary? | > 20% of 1-hop BFS results have `last_access_epoch < current_epoch - tau` over 7-day window | Gradual worsening as graph matures; OR sudden onset after PG restore |

**Key asymmetry**: S4 and S5 are epoch-domain signals but measure different disruption directions:
- S4 fires when `redis_epoch < pg_max` — Redis fell behind PostgreSQL (Redis reset, Direction 1 from Q074).
- A complementary S4b condition (`redis_epoch >> pg_max`, gap > EPOCH_GAP_ALARM_THRESHOLD) detects the PG-restore direction (Q074 Direction 2), but Q074 specifically defines this as an *alarm* distinct from the standard S4 epoch-inversion signal.
- S5 fires independently of the Redis/PG epoch relationship — it measures whether the Neo4j traversal query respects the current `current_epoch - tau` window.

## S4+S5 Auto-Recovery Analysis

### When S4 and S5 co-alert: two mechanistically distinct scenarios

**Scenario A: Redis reset (Direction 1 — `redis_epoch < pg_max`)**

In the Q067/Q074 Direction 1 failure, Redis was reset and the epoch counter fell to 0 while PostgreSQL retained the correct epoch history. The LLPC working-set filter now admits everything (filter evaluates false for every memory, since `last_access_epoch >= 0 - 20` is always true when redis_epoch=0). Meanwhile, Neo4j traversal, which caches `last_access_epoch` as a node property (per Q078 Option A), still holds the *pre-reset* epoch values on nodes.

Effect on S5: With `redis_epoch = 0`, the Cypher epoch predicate `WHERE neighbor.last_access_epoch >= ($current_epoch - $tau) = (0 - 20) = -20` is always true. S5 does NOT fire in this scenario because the graph epoch predicate uses the *current (reset) epoch* as its reference — all nodes pass the filter trivially. **S4+S5 co-alert does NOT occur in the Direction 1 (Redis reset) scenario.**

**Scenario B: PG restore from old backup (Direction 2 — `redis_epoch >> pg_max`, gap > EPOCH_GAP_ALARM_THRESHOLD)**

This is Q074's undetected failure mode. PostgreSQL was restored from a backup at epoch 100 while Redis continues at epoch 500. Now:
- S4 fires via the bidirectional guard (gap = 400 > EPOCH_GAP_ALARM_THRESHOLD=50): `PG_RESTORE_SUSPECTED`
- Neo4j node properties still hold `last_access_epoch` values from before the restore — these are the *correct* values from the live graph (Neo4j was not restored). Some nodes have `last_access_epoch` in the range 101–500 (sessions since the backup). The Cypher epoch predicate `WHERE neighbor.last_access_epoch >= (500 - 20) = 480` would exclude neighbors with epoch < 480. But the backup-era PG memories (epoch <= 100) are the ones the system is now trying to serve. Graph traversal returns neighbors from the most recent sessions (epochs 480–500), pointing to memories that no longer exist in the restored PostgreSQL database (they were created after the backup). **S5 fires** because traversal returns nodes whose `last_access_epoch` values indicate recent activity but whose PG records are absent (or more precisely, because a significant fraction of BFS results point to post-backup memories invisible to PG).

**Scenario C: Graph maturity without epoch disruption (S5 alone)**

S5 can fire without S4 in the steady-state scenario where the causal graph has matured and hub nodes accumulate edges to memories outside the working-set window. In this case, S4 is healthy (Redis and PG are consistent), and S5 fires independently.

### Can S4+S5 co-alert automatically trigger Option A recovery?

**Option A recovery** from Q074 is: set `recall:global_epoch` in Redis to `pg_max + 1`. This acknowledges the backup, rolls back Redis to the PG level, and restores epoch consistency at the cost of losing the sessions since the backup.

**The critical question is: is S4+S5 co-alert in Scenario B (PG restore) unambiguous enough to trigger Option A automatically?**

**Answer: No — not safely. Verdict: PROMISING (not BREAKTHROUGH).**

Reasons:

1. **S4 in Direction 2 (PG restore) already requires operator decision by Q074 design.** The Q074 bidirectional guard explicitly returns `PG_RESTORE_SUSPECTED:gap=N` and requires the operator to choose between Option A (accept data loss, retract epoch), Option B (find a better backup), or Option C (temporary tau override). Auto-executing Option A means automatically accepting data loss proportional to the epoch gap (400 sessions = potentially months of memories permanently destroyed). This is a destructive, irreversible action.

2. **The false positive rate for S4+S5 co-alert in the PG-restore direction is non-trivial.** See false positive analysis below.

3. **Scenario A (Redis reset) does NOT produce S4+S5 co-alert** as shown above, so S4+S5 co-alert *does* specifically indicate the PG-restore direction. This is a useful diagnostic signal. But "indicates PG restore" is not sufficient to auto-recover — the operator must still decide whether Option A (epoch retraction) or Option B (better backup) is correct.

4. **However: S4+S5 co-alert CAN automatically narrow the triage.** The diagnostic value is real: when S4 fires with a large gap (PG_RESTORE_SUSPECTED) AND S5 fires simultaneously, the combined signal is strong evidence that the Neo4j graph is inconsistent with the current PostgreSQL state — the graph has edges pointing to post-backup memories that PG no longer contains. This combination rules out the Redis-reset direction and the slow-graph-maturity direction, pointing specifically to PG restore as the root cause.

### False Positive Rate for Each Automated Recovery Trigger

**S4 Direction 1 (Redis reset) automated recovery — currently implemented in Q067:**

The recovery action is: `SET recall:global_epoch = pg_max + 1`.

False positive conditions:
- `redis_epoch < pg_max` under normal operation is never expected (Redis epoch leads PG epoch by 0–2 sessions). The only legitimate scenario where `redis_epoch < pg_max` is Redis data loss (FLUSHALL, restart without persistence). There is no normal-operation path that produces this inequality.
- With dual-machine operation (casaclaude + proxyclaude), both machines increment `recall:global_epoch` at session end. If one machine's session-end hook fires and writes `last_access_epoch = 500` to PG while the other machine's Redis is at epoch 499 (slightly behind due to a race condition), a transient `pg_max > redis_epoch` could occur for a few milliseconds. This resolves naturally and would not persist to the next session start.

**Estimated false positive rate: < 1% of session starts.** The inequality `pg_max > redis_epoch` is structurally anomalous in steady state. The Direction 1 automated recovery is safe to keep automated.

**S4 Direction 2 (PG restore) + S5 co-alert automated recovery — NOT currently implemented:**

If one were to implement auto-recovery (Option A, retract Redis epoch):

False positive conditions:
1. **Legitimate large epoch gap without a PG restore**: Q076's follow-up question explicitly raised this concern. With dual-machine operation and one machine's PG write path degraded for 30+ sessions, `redis_epoch - pg_max` could legitimately reach 30–50. At EPOCH_GAP_ALARM_THRESHOLD=50, this sits at the alarm boundary. If S5 also fires coincidentally (graph maturity, Month 12+), a co-alert occurs without a PG restore. Auto-executing Option A in this case would retract the Redis epoch to `pg_max + 1`, causing the working-set filter to suddenly start matching only the last tau=20 memories from the degraded PG state — likely causing retrieval quality collapse.

2. **S5 chronic background state**: At Month 12+, S5 can be in a persistent warning/critical state due to graph maturity (hub nodes with stale edges) without any epoch disruption. If S4 fires for any reason (even a genuine Redis reset, Direction 1) while S5 is in this chronic state, the co-alert triggers despite the root cause being Direction 1 (not Direction 2). Option A would be the wrong recovery for a Direction 1 event.

3. **Temporal coincidence during planned maintenance**: If the operator manually adjusts `recall:global_epoch` during migration work, S4 may fire briefly. If S5 is simultaneously in warning state, co-alert occurs without a genuine PG restore.

**Estimated false positive rate for S4+S5 co-alert auto-triggering Option A**: 15–30%. This is unacceptably high for an irreversible, data-destructive operation (Option A permanently discards post-backup memories).

**Contrast with S2+S3 auto-recovery (force-replay)**: The S2+S3 CasaOS connectivity pattern has near-zero false positive rate for its recovery action (force-replay pending queue) because the recovery is safe and idempotent — replaying an already-submitted summary produces a duplicate that the API can deduplicate. The recovery is not destructive.

**S3+S4 co-alert recovery (apply S4 fix then force-replay S3)**: Estimated false positive rate < 2%. The recovery is correctly sequenced (epoch first, replay second) and the S4 Direction 1 fix is safe. The S3 force-replay is also safe (idempotent).

## Recommended Automated Recovery Rules

### Rule 1: S2+S3 co-alert → CasaOS connectivity check + force-replay (AUTOMATED)

```python
if s2.status in ("warning", "critical") and s3.status in ("warning", "critical"):
    root_cause = "CASAOS_UNAVAILABILITY"
    action = "check_casaos_health(); force_replay_pending_queue()"
    automated = True  # Force-replay is safe and idempotent
```

**Rationale**: Unambiguous root cause (Q077). Recovery is safe (replay cannot corrupt data). False positive rate: < 5% (only fails if S2 degrades from vocabulary drift at the same time S3 accumulates from a separate hook failure — independent concurrent failures are rare).

---

### Rule 2: S3+S4 co-alert → Apply S4 recovery first, then force-replay S3 (AUTOMATED, sequenced)

```python
if s3.status in ("warning", "critical") and s4.status == "critical":
    root_cause = "CASAOS_RESTART_WITH_REDIS_DATA_LOSS"
    action_1 = "redis_set_global_epoch(pg_max + 1)"  # S4 fix first
    action_2 = "force_replay_pending_queue()"         # S3 fix second, after epoch is restored
    automated = True  # Both actions are safe in sequence
```

**Rationale**: CasaOS restart with Redis data loss is the canonical shared cause. S4 Direction 1 fix is proven safe (Q067). S3 replay after epoch restoration ensures replayed writes use the correct epoch. False positive rate: ~5% (Redis reset without a write-tier blockage, or vice versa, due to independent failure paths).

### Rule 3: S1+S4 co-alert → Apply S4 recovery, monitor S1 for recovery (SEMI-AUTOMATED)

```python
if s1.status in ("warning", "critical") and s4.status == "critical":

    action_2 = "monitor_s1_for_7_days"               # S1 watchful waiting
    automated_s4 = True
    automated_s1 = False  # If S1 does not recover after 7 days, refit propensity model
```

### Rule 4: S4+S5 co-alert → Emit diagnostic alarm, require operator decision (NOT AUTOMATED)

```python
if s4.status in ("warning", "critical") and s5.status in ("warning", "critical"):
    # Determine which S4 direction is firing
    if epoch_gap > EPOCH_GAP_ALARM_THRESHOLD:
        root_cause = "PG_RESTORE_SUSPECTED_WITH_GRAPH_INCONSISTENCY"
        alarm_message = (
            f"S4 PG-restore direction detected (gap={epoch_gap}). "
            f"S5 also alerting: Neo4j graph inconsistent with PG state. "
            f"This combination strongly indicates PostgreSQL was restored from backup "
            f"while Neo4j was not. Options: "
            f"(A) Retract Redis epoch to pg_max+1 (accept data loss of {epoch_gap} sessions). "
            f"(B) Restore PG from a more recent backup. "
            f"(C) Set tau to a large value for immediate operational continuity. "
            f"Human decision required before any automated action."
        )
        automated = False  # False positive rate 15-30%; action is irreversible
    else:
        # S4 Direction 1 (Redis reset) — S5 is in chronic background state
        action_1 = "redis_set_global_epoch(pg_max + 1)"  # Safe S4 recovery
        root_cause = "REDIS_RESET_WITH_CHRONIC_GRAPH_STALENESS"
        automated_s4 = True
        note_s5 = "S5 is background graph maturity condition; implement Q078 Cypher predicate"
```

**Rationale for withholding automation on Option A**: The false positive rate of 15–30% combined with the irreversible nature of Option A (permanent memory loss proportional to the epoch gap) makes automated execution unacceptably risky. The alarm message must be structured enough to enable rapid operator decision, but the decision itself requires human intent.

## Ideas Extracted

### Idea: s4-direction-discriminant-via-s5-co-alert
- **Novelty**: 0.78 — The insight that S4+S5 co-alert *disambiguates* the S4 failure direction (Redis reset vs PG restore) is not present in Q074 or Q071. Q074 defines two alarm codes (`REDIS_RESET_RECOVERED` and `PG_RESTORE_SUSPECTED`) but provides no cross-signal discriminant for distinguishing them at runtime without inspecting raw epoch values. The observation that Redis reset (Direction 1) does NOT produce S5 co-alert (because the Cypher epoch predicate uses `current_epoch - tau = 0 - 20 = -20`, admitting all nodes) while PG restore (Direction 2) DOES produce S5 co-alert (because the graph retains post-backup epoch values pointing to missing PG records) is a novel diagnostic insight derived from the joint signal behavior.
- **Evidence**: 0.68 — Derived from first principles: in Direction 1 (Redis reset), the Cypher predicate threshold collapses to near-zero and S5 trivially passes; in Direction 2 (PG restore), the Cypher predicate uses the Redis epoch (still high), and many graph neighbors have `last_access_epoch` values in the post-backup range, producing S5 failures for queries whose neighbors point to missing PG records. The asymmetry is logically sound given Q078's Cypher epoch predicate design.
- **Feasibility**: 0.85 — This is a diagnostic rule implemented in the `/health/detail` correlation layer, not a code change to any storage component. The rule requires S4 direction information (already available from the bidirectional guard's alarm code) and S5 status (new signal from Q078). Cross-referencing them in the correlation layer is a 10-line conditional.
- **Description**: Add an S4 direction discriminant to the `/health/detail` correlation layer: when S4 fires as `PG_RESTORE_SUSPECTED` (gap > EPOCH_GAP_ALARM_THRESHOLD), check S5 status. If S5 is also alerting, escalate the diagnosis to `PG_RESTORE_CONFIRMED_BY_GRAPH_STATE` — Neo4j's retained post-backup epoch values provide independent evidence of the restore. This combination is more specific than S4 alone (which can misfire at the gap threshold boundary) and narrows the triage to the PG restore scenario with high confidence. The co-alert does not enable automated Option A recovery, but it reduces the operator's decision time by providing convergent evidence from two independent subsystems.

### Idea: correlation-layer-in-health-detail-endpoint
- **Novelty**: 0.72 — The Q071 `/health/detail` endpoint reports signals independently with no cross-signal interpretation. Adding a `correlations` field to the response that maps co-alert patterns to implied root causes is a structural extension of the health endpoint design. This pattern (symptom correlation in health API responses) is used in cloud monitoring systems (AWS CloudWatch Insights, Datadog anomaly correlation) but is not documented in AI memory system health monitoring. The minimal implementation (a rule table applied to the current signal statuses) requires no new infrastructure.
- **Evidence**: 0.72 — The pairwise co-alert rules derived in this finding are evidence-backed (Q077 for S2+S3; Q074 for S4+S5 PG direction; Q071 for S3+S4 CasaOS restart). ## Key Finding

Ten pairwise co-alert rules cover all combinations of S1–S5. Of these ten, only three have unambiguous root-cause mappings sufficient to justify automated recovery:

1. **S2+S3** = CasaOS unavailability (force-replay safe, false positive rate ~5%)
2. **S3+S4** = CasaOS restart with Redis data loss (epoch fix then force-replay, ~5% false positive)
3. **S1+S4** = epoch reset cascading to quality degradation (S4 auto-fixed; S1 watchful waiting, ~2% false positive)

The S4+S5 co-alert is the most *diagnostically informative* co-alert in the system — it discriminates between the two S4 failure directions (Redis reset vs PG restore) in a way that S4 alone cannot, because the Neo4j graph's retained post-backup epoch values provide independent evidence of a PG restore. However, this discrimination does NOT enable automated Option A recovery. The false positive rate (15–30%) is too high, and Option A is irreversible (permanently discards post-backup session memories proportional to the epoch gap). The correct response to S4+S5 co-alert in the PG-restore direction is a high-confidence structured alarm that provides the operator with the three recovery options (A/B/C from Q074) and the convergent evidence from both signals — reducing decision time but requiring human intent.

The S5 signal, when alerting in isolation (S4 healthy), is a chronic background condition caused by graph maturity: hub nodes accumulating edges to stale neighbors as the system ages past Month 6–12. This is the only co-alert pattern that has no recoverable current state — it requires a code change (implementing the Cypher epoch predicate from Q078 Option A) rather than a configuration or operational action. S5 in isolation should trigger a severity-escalating reminder that the Q078 implementation is overdue, not an alarm that implies imminent data loss.

The boundary between automated and operator-required recovery maps cleanly onto the boundary between reversible and irreversible actions: all three safely automated co-alert recoveries (S2+S3, S3+S4, S1+S4) are idempotent or easily reversible. The one that cannot be automated (S4+S5 → Option A) is permanently destructive. This is the correct design principle for automated recovery in epoch-clock systems.
