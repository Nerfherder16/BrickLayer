### File 2: `src/workers/flush_co_retrievals.py` — New worker module (complete file)

Create this file at `src/workers/flush_co_retrievals.py`:

```python
"""
flush_co_retrievals worker — promote co_retrievals_outbox rows to Neo4j CO_RETRIEVED edges.

The co_retrievals_outbox table (created by Q156 migration) accumulates empirical
co-occurrence pairs written by store_co_retrievals() at retrieval time. This worker
drains the outbox every 15 minutes by MERGEing CO_RETRIEVED edges into Neo4j, then
stamping processed_at on each successfully promoted row.

Design choices:
- FOR UPDATE SKIP LOCKED: safe for concurrent workers (none planned, defensive)
- UNWIND batch MERGE: single round-trip to Neo4j per flush cycle
- Bidirectional MERGE: CO_RETRIEVED is symmetric — co-occurrence is undirected
- ON MATCH weight accumulation: repeated co-retrievals strengthen the edge (cap 1.0)
- Dead-letter on attempt_count >= 3: prevents poison-pill rows from blocking the queue
- processed_at IS NULL filter: idempotent — re-runs skip already-promoted rows
"""

import structlog

from src.storage import get_neo4j_store, get_postgres_store

logger = structlog.get_logger()

_BATCH_SIZE = 500  # rows per flush cycle


async def flush_co_retrievals_to_neo4j(ctx: dict) -> dict:
    """
    Drain co_retrievals_outbox into Neo4j CO_RETRIEVED edges.

    ARQ cron function — called every 15 minutes by WorkerSettings.
    Returns stats dict for logging.
    """
    logger.info("flush_co_retrievals_starting")

    pg = await get_postgres_store()
    neo4j = await get_neo4j_store()

    # --- Step 1: Select pending outbox rows with advisory lock ---
    # FOR UPDATE SKIP LOCKED prevents double-processing if two workers ever run
    # concurrently. attempt_count < 3 skips rows that have repeatedly failed.
    rows = await pg.pool.fetch(
        """
        SELECT id, from_memory_id, to_memory_id, tenant_id, epoch
        FROM co_retrievals_outbox
        WHERE processed_at IS NULL
          AND attempt_count < 3
        ORDER BY created_at ASC
        LIMIT $1
        FOR UPDATE SKIP LOCKED
        """,
        _BATCH_SIZE,
    )

    if not rows:
        logger.info("flush_co_retrievals_nothing_to_flush")
        return {"promoted": 0, "dead_lettered": 0, "batch_size": 0}

    logger.info("flush_co_retrievals_batch", count=len(rows))

    # --- Step 2: Build pairs list for Neo4j UNWIND MERGE ---
    # Each pair is bidirectional: (A→B) and (B→A) both get the edge.
    # weight_increment is fixed at 0.05 per co-occurrence (matches strengthen_relationship).
    pairs = [
        {
            "source_id": str(row["from_memory_id"]),
            "target_id": str(row["to_memory_id"]),
            "increment": 0.05,
        }
        for row in rows
    ]

    # --- Step 3: UNWIND MERGE CO_RETRIEVED edges into Neo4j ---
    # Uses the same UNWIND pattern as strengthen_relationships_batch() but with
    # CO_RETRIEVED relationship type instead of RELATED_TO.
    # Bidirectional MERGE: MERGE (a)-[r:CO_RETRIEVED]-(b) covers both directions.
    # ON CREATE: initial strength = 0.5 + increment (0.55)
    # ON MATCH: accumulate strength, cap at 1.0
    promoted_ids = []
    failed_ids = []

    try:
        async with neo4j.driver.session() as session:
            await session.run(
                """
                UNWIND $pairs AS pair
                MATCH (a:Memory {id: pair.source_id}), (b:Memory {id: pair.target_id})
                MERGE (a)-[r:CO_RETRIEVED]-(b)
                ON CREATE SET r.strength = 0.5 + pair.increment,
                              r.created_at = datetime(),
                              r.updated_at = datetime()
                ON MATCH SET r.strength = CASE
                    WHEN r.strength + pair.increment > 1.0 THEN 1.0
                    ELSE r.strength + pair.increment
                END,
                r.updated_at = datetime()
                """,
                pairs=pairs,
            )
        promoted_ids = [row["id"] for row in rows]
    except Exception as e:
        logger.error("flush_co_retrievals_neo4j_error", error=str(e))
        # On Neo4j failure: increment attempt_count on all rows; don't mark processed.
        # After 3 failures, rows are dead-lettered by the next cycle's attempt_count < 3 filter.
        failed_ids = [row["id"] for row in rows]

    # --- Step 4: Stamp processed_at on successfully promoted rows ---
    if promoted_ids:
        await pg.pool.execute(
            """
            UPDATE co_retrievals_outbox
            SET processed_at = now()
            WHERE id = ANY($1::bigint[])
            """,
            promoted_ids,
        )
        logger.info("flush_co_retrievals_promoted", count=len(promoted_ids))

    # --- Step 5: Increment attempt_count on failed rows ---
    dead_lettered = 0
    if failed_ids:
        await pg.pool.execute(
            """
            UPDATE co_retrievals_outbox
            SET attempt_count = attempt_count + 1,
                last_attempt_at = now()
            WHERE id = ANY($1::bigint[])
            """,
            failed_ids,
        )
        # Move rows that have now hit attempt_count = 3 to dead-letter table
        dead_rows = await pg.pool.fetch(
            """
            DELETE FROM co_retrievals_outbox
            WHERE id = ANY($1::bigint[]) AND attempt_count >= 3
            RETURNING id, from_memory_id, to_memory_id, tenant_id, epoch, created_at,
                      'max_attempts_exceeded' AS final_error
            """,
            failed_ids,
        )
        if dead_rows:
            await pg.pool.executemany(
                """
                INSERT INTO co_retrievals_dead_letter
                    (outbox_id, from_memory_id, to_memory_id, tenant_id, epoch,
                     created_at, final_error)
                VALUES ($1, $2, $3, $4, $5, $6, $7)
                """,
                [
                    (
                        r["id"],
                        r["from_memory_id"],
                        r["to_memory_id"],
                        r["tenant_id"],
                        r["epoch"],
                        r["created_at"],
                        r["final_error"],
                    )
                    for r in dead_rows
                ],
            )
            dead_lettered = len(dead_rows)
            logger.warning("flush_co_retrievals_dead_lettered", count=dead_lettered)

    stats = {
        "promoted": len(promoted_ids),
        "dead_lettered": dead_lettered,
        "batch_size": len(rows),
    }
    logger.info("flush_co_retrievals_complete", **stats)
    return stats
```

## Idempotency Guarantees

**FOR UPDATE SKIP LOCKED** (Step 1 of worker):
- If two ARQ workers somehow run simultaneously (possible during deployment with `max_jobs=10`), only one acquires the lock on each row. The second skips those rows entirely rather than blocking.
- This prevents double-promotion: a row promoted by worker A is invisible to worker B's concurrent SELECT because it is locked.

**processed_at IS NULL filter** (Step 1 of worker):
- After worker A stamps `processed_at = now()` on a row, subsequent SELECT queries skip it forever.
- If the worker crashes between Step 3 (Neo4j MERGE) and Step 4 (processed_at UPDATE), the row remains unprocessed. On the next cycle, Neo4j's MERGE is idempotent (ON MATCH just increments strength slightly more than intended — acceptable). The row is re-processed at most once more per cycle until processed_at is stamped.

**Neo4j MERGE idempotency**:
- `MERGE (a)-[r:CO_RETRIEVED]-(b)` creates the edge if absent, updates strength if present.
- Re-running the same pair twice increments strength by 0.05 each time (capped at 1.0). This is the intended semantics: repeated co-occurrence strengthens the edge.

**attempt_count guard**:
- Rows with `attempt_count >= 3` are excluded by the SELECT filter and moved to `co_retrievals_dead_letter`. This prevents Neo4j connection failures (e.g., during maintenance windows) from creating an infinite retry loop that fills the outbox.

## Neo4j Index Prerequisite

The CO_RETRIEVED MERGE `MATCH (a:Memory {id: pair.source_id}), (b:Memory {id: pair.target_id})` requires a lookup by `Memory.id`. This is covered by the **UNIQUE CONSTRAINT** created in `Neo4jStore.connect()` at lines 40–45:

```python
await session.run(
    """
    CREATE CONSTRAINT memory_id IF NOT EXISTS
    FOR (m:Memory) REQUIRE m.id IS UNIQUE
    """
)
```

Neo4j automatically creates a B-tree index to back uniqueness constraints. The `MATCH (m:Memory {id: $id})` lookup is therefore O(log N), not O(N). No additional `CREATE INDEX` is needed. The constraint was present before any co_retrievals work began; it is not a new prerequisite.

**Cross-check**: `strengthen_relationships_batch()` at lines 221–240 uses the identical `MATCH (a:Memory {id: pair.source_id}), (b:Memory {id: pair.target_id})` pattern without issue — confirming the index is sufficient.

4. **pool.fetch() / pool.execute() are called directly on pg.pool** — not via `pool.acquire()` context manager. This matches the dominant pattern in `postgres_store.py` (lines 386, 338, 464, 795, etc.). `pool.acquire()` is only used in `connect()` (line 304) for the DDL migration block, not in query methods.

5. **CO_RETRIEVED enum value must be added before the flush worker is deployed** — if the health.py filter references `RelationshipType.CO_RETRIEVED` before the enum exists, it will raise `AttributeError`. The models.py change is a prerequisite. The health.py change can use the string literal `"CO_RETRIEVED"` temporarily, but the enum is the clean path.

6. **health.py filter currently hard-codes `"RELATED_TO"`** (line 123) — CO_RETRIEVED edges produced by the flush worker are completely invisible to the force profile dashboard until the set `{"RELATED_TO", "CO_RETRIEVED"}` change at line 123 is applied. This is the observable production effect referenced in Q163: Q156's table migration produces no visible dashboard effect until both the flush worker AND the health.py filter are deployed.

7. **Bidirectional MERGE semantics**: `MERGE (a)-[r:CO_RETRIEVED]-(b)` without arrow direction matches or creates an undirected edge. Neo4j stores it as directed internally (A→B) but the pattern matches regardless of direction. This is the correct semantics for co-retrieval (symmetric: if A was co-retrieved with B, B was also co-retrieved with A).

8. **The complete deployment sequence** (respecting Q156's constraint that tables must exist before the flush worker is wired in): (1) apply Q156 migration to postgres_store.py + restart, (2) add CO_RETRIEVED to models.py, (3) create flush_co_retrievals.py, (4) update main.py (import + functions + cron), (5) update health.py filter, (6) restart worker service.

### Idea: co-retrieved-gravity-blind-until-health-filter-patched

- Feasibility: 0.97
- Description: The CO_RETRIEVED edges written by the flush worker are structurally valid in Neo4j from the first flush cycle, but the health dashboard's `co_retrieval_gravity` score remains 0.0 for any memory whose graph connections are exclusively CO_RETRIEVED edges (not RELATED_TO). ### Idea: outbox-batch-size-determines-latency-floor
- Novelty: 0.68
- Evidence: 0.88
- Feasibility: 0.99
- Description: The `_BATCH_SIZE = 500` constant sets the maximum rows processed per 15-minute cycle. At the current retrieval rate (8–12 retrievals/session, ~2–3 co-occurrence pairs per retrieval), a typical session generates 16–36 outbox rows. The 500-row batch size means the outbox drains completely every cycle under normal load. If retrieval volume spikes (e.g., during a high-throughput session with 100+ retrievals), the 15-minute cycle may not drain the full outbox in one pass, but the next cycle catches up. The batch size does not need tuning unless the outbox consistently shows `pending > 500` after a flush cycle.

### Idea: neo4j-session-per-cycle-not-from-ctx
- Novelty: 0.65
- Evidence: 0.91
- Feasibility: 0.98
- Description: The flush worker calls `get_neo4j_store()` to obtain the singleton, then opens a fresh `async with neo4j.driver.session()` for the UNWIND MERGE. This is preferable to using `ctx["neo4j"]` (the session stored at worker startup) because the ctx session may have accumulated state from prior workers in the same process. The `get_neo4j_store()` singleton pattern is the same pattern used by `run_gc`, `run_reconcile`, and all other workers that need Neo4j — none use `ctx["neo4j"]` directly. The ctx dict stores the store reference, not an open session; `ctx["neo4j"]` is the same `Neo4jStore` object returned by `get_neo4j_store()`.
