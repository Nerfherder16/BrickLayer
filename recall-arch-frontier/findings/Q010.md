# Finding: Q010 — Graph-Walk Retrieval as Primary Mechanism

**Question**: Has any production memory system implemented graph-walk retrieval as a primary retrieval mechanism (not a post-filtering step), where entity relationships guide traversal from a query anchor to related memories?
**Question Type**: ABSENCE
**Verdict**: PROMISING
**Severity**: Medium
**Source field**: Knowledge graph traversal algorithms (PageRank, BFS/DFS spreading activation, beam search on KGs)

## Mechanism Under Investigation

A retrieval system where the graph walk IS the ranking function — the query is used to find an anchor node via entity extraction or exact match, and then graph traversal (BFS, PPR, spreading activation) determines which memories are returned, with vector similarity as a tiebreaker or secondary signal rather than the primary path.

## Absence Classification

**Classification**: PARTIAL ABSENCE

**Explanation**: Graph-walk-primary retrieval EXISTS in research (HippoRAG, ToG, PersonalAI/AriGraph, MixPR) but NO production AI memory system ships it as the primary retrieval path. Every production system — Graphiti, GraphRAG, Letta, mem0 — uses vector/BM25 as the primary path, with graph traversal as a post-filter, reranker, or expansion step.

The hypothesis was mostly correct but slightly too strong: it is not that pure graph-walk retrieval "doesn't exist" — it exists in multiple research codebases. The gap is specifically between research implementations and production memory systems. The mechanism works and outperforms vector-only retrieval by up to 20% on multi-hop QA (HippoRAG), yet no production memory system has shipped it as the primary path.

### Why does this gap exist?

**Reason 1 — Entity extraction is brittle in practice.** Graph-walk retrieval requires identifying the anchor node reliably. If entity extraction fails (the query mentions "that Redis thing from last week" rather than "Redis"), the walk starts from nowhere. Vector search degrades gracefully; graph walk fails hard.

**Reason 2 — Graph structure quality dependence.** PPR-as-primary assumes the graph has high-quality, densely connected edges. If memory is sparse or entity linking is noisy, the walk retrieves structurally connected but semantically unrelated memories. Production systems see highly variable graph density.

**Reason 3 — Vector search is a solved problem.** HNSW at production scale is 1-10ms with well-understood quality. Graph walk algorithms (especially PPR) require tuning the teleportation parameter, handling disconnected components, and managing cold-start for new entities. The operational overhead is real.

**Reason 4 — Research benchmarks don't reflect memory workloads.** HippoRAG and ToG are benchmarked on multi-hop QA (MuSiQue, 2WikiMultiHop). Production memory queries are mostly single-hop ("what did I say about X?") where vector search is already near-optimal. The graph-walk advantage only materializes at multi-hop depth — which current memory systems rarely need.

**Reason 5 — Mental model mismatch.** Memory system designers think in terms of "find similar text." Graph traversal requires thinking in terms of "find connected entities." The two mental models produce different storage structures at index time — graph-walk retrieval requires entity-centric graphs, while vector retrieval just requires embeddings.

## Ideas Extracted

### Idea: ppr-retrieval-for-causal-chains
- **Novelty**: 0.75 — PPR-as-primary exists in HippoRAG (research), but applying it to a memory system's entity graph (not a document KG) and specifically for causal chain queries is novel at production level.
- **Evidence**: 0.80 — HippoRAG NeurIPS 2024 demonstrates 20% improvement over vector-only on multi-hop QA. PersonalAI AriGraph demonstrates A*/BFS-primary retrieval works for personalized agent memory.
- **Feasibility**: 0.55 — Recall already has Neo4j with RELATED_TO, CAUSED_BY, DEPENDS_ON edges. Entity extraction exists (entity_extractor.py). The missing pieces are: (1) entity-to-node resolution at query time, (2) PPR implementation on the existing graph, (3) hybrid ranking that uses PPR score as primary and cosine as tiebreaker. Moderate implementation effort.
- **Description**: At query time, extract named entities from the query text (reuse existing entity_extractor). Resolve each entity to a node in Neo4j. Seed a Personalized PageRank computation from those nodes, propagating activation through RELATED_TO/CAUSED_BY edges with the existing stability multipliers (already in _graph_expand). Use PPR scores as the primary retrieval ranking for queries classified as "causal" or "multi-hop" by query_intent. Use cosine similarity as tiebreaker within PPR score bands. This would be most valuable for queries like "why did X happen?" or "what led to Y?" where causal chains are load-bearing.

### Idea: hybrid-walk-with-intent-routing
- **Novelty**: 0.70 — Intent-gated switching between vector-primary and graph-walk-primary retrieval modes is not implemented in any production system found.
- **Evidence**: 0.65 — ragwalla.com article notes that vector-grounded graph retrieval is dominant because "user queries are fuzzy natural language." But Recall's QueryIntent classifier already detects query types. Routing causal/relational queries to graph-walk-primary while keeping factual/lookup queries on vector-primary would combine the advantages of both.
- **Feasibility**: 0.70 — Recall already has query intent classification (query_intent.py). The routing logic would be a small addition. The graph-walk engine already exists in _graph_expand; it would need to be promoted to primary for specific intent classes rather than used as a post-filter.
- **Description**: Extend QueryIntent to classify queries as CAUSAL (why/how/what-caused), RELATIONAL (what connects X to Y), or FACTUAL (what is X). For CAUSAL and RELATIONAL intents: execute graph walk as primary (from entity anchors), use cosine only to break ties within PPR bands. For FACTUAL: keep current vector-primary pipeline. This is a low-risk change because the existing pipeline remains the fallback, and the routing condition is easily tunable.

### Idea: cold-walk-for-novel-sessions
- **Novelty**: 0.60 — Modest novelty. Using graph walk specifically as the primary mechanism for novel queries (no strong vector match found) is a fallback mode rather than a primary mode.
- **Evidence**: 0.50 — Recall already caps graph-only results at 0.08. The idea is to flip this: when vector search returns no results above a threshold, switch to graph-walk primary seeded from entity anchors rather than returning empty.
- **Feasibility**: 0.80 — Small code change. Current code (line 423) caps graph results. A pre-check — "if vector results are empty or below threshold, run entity-anchored graph walk as primary" — is a tractable addition.
- **Description**: Add a stage 2.5: if all vector results are below 0.40 similarity, extract entities from the query, find corresponding Neo4j nodes, run BFS/spreading activation from those nodes as the primary retrieval path. This converts the current "graph never wins" architecture to "graph is the fallback primary when vector fails." Most valuable for queries about relationships between known entities where the relationship itself has no strong embedding match.

## Suggested Follow-ups

- **Implement HippoRAG PPR pipeline against Recall's entity graph**: The entity graph (entity_store.py, HAS_FACT edges) is structurally similar to the KG HippoRAG uses. A prototype could test whether PPR on Recall's existing graph improves multi-hop retrieval quality.
- **Benchmark single-hop vs multi-hop query distribution in Recall**: The graph-walk advantage only materializes at multi-hop depth. Measure what fraction of actual Recall queries require multi-hop reasoning. If >20%, the investment in graph-walk-primary is justified; if <5%, the current architecture is already near-optimal.
- **Evaluate MixPR (Zyphra) against Recall's memory store**: MixPR uses TF-IDF sparse embeddings + PageRank, runs on CPU, and achieves SOTA on long-context benchmarks. Could it replace or augment Qdrant ANN for certain query types?
- **Q019 connection**: The adversarial question "flat text blobs vs structured knowledge graphs from the start" is directly load-bearing here — graph-walk-primary retrieval only achieves its full potential if memories are stored as entity-centric structures at index time, not as text blobs with post-hoc entity extraction.
