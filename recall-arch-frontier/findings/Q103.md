## Part 1: bulk_delete_and_optimize() — Complete Function Specification

### Role in the sequence

`bulk_delete_and_optimize()` is called **after** `/reconcile?mode=post_restore&repair=true` completes. The reconcile endpoint handles the deletion itself; this function's responsibility is to:
1. Force an optimize pass on the now-dirty Qdrant collection (ghost nodes remain after soft-delete)
2. Poll until Qdrant reports the optimizer is idle
3. Verify the optimized collection has the expected point count

The function does NOT perform the deletion. It is a post-reconcile recovery gate.

### Function Signature

```python
async def bulk_delete_and_optimize(
    qdrant_client: AsyncQdrantClient,
    collection_name: str,
    expected_count_after_delete: int,
    *,
    optimize_timeout_s: float = 30.0,
    poll_interval_s: float = 0.5,
    count_tolerance: int = 2,
) -> BulkOptimizeResult:
    """
    Force Qdrant HNSW optimization after bulk deletion and verify the result.

    Called AFTER /reconcile?mode=post_restore has deleted orphaned points.
    Ensures the HNSW index is rebuilt on the surviving clean corpus before
    ANN searches resume.

    Args:
        qdrant_client: Async Qdrant Python client.
        collection_name: Collection to optimize (typically "memories").
        expected_count_after_delete: The surviving point count reported by
            /reconcile (neo4j_deleted_count and qdrant_deleted_count give
            the deletion total; caller computes: pre_delete_count - qdrant_deleted_count).
        optimize_timeout_s: Maximum seconds to wait for optimizer_status.status == "ok".
            Default 30s — well above the 2s wall-clock for 500 points, providing a 15×
            safety margin. Raise to 120s for large collections (>5,000 points).
        poll_interval_s: Seconds between optimizer_status polls. Default 0.5s.
        count_tolerance: Acceptable absolute difference between expected_count_after_delete
            and the actual Qdrant count after optimization. Default 2 (allows for ±1 timing
            race on boundary-epoch inserts).

    Returns:
        BulkOptimizeResult with fields:
            success: bool
            optimize_elapsed_ms: int
            final_count: int
            expected_count: int
            count_ok: bool
            failure_step: str | None  ("optimize_trigger", "poll_timeout", "count_mismatch", None)
            failure_detail: str | None
    """
```

### Complete Implementation Logic

```python
import asyncio
import time
from qdrant_client import AsyncQdrantClient

async def bulk_delete_and_optimize(
    qdrant_client: AsyncQdrantClient,
    collection_name: str,
    expected_count_after_delete: int,
    *,
    optimize_timeout_s: float = 30.0,
    poll_interval_s: float = 0.5,
    count_tolerance: int = 2,
) -> BulkOptimizeResult:

    start_ms = int(time.monotonic() * 1000)

    # ── STEP A: Trigger optimize_collection ──────────────────────────────────
    # Timeout budget: 5s for the API call itself (should return immediately —
    # the call schedules optimization, it does not block on completion).
    try:
        async with asyncio.timeout(5.0):
            await qdrant_client.update_collection(
                collection_name=collection_name,
                optimizer_config=OptimizersConfigDiff(
                    # Force vacuum by temporarily lowering the deleted_threshold
                    # below the current deleted fraction. This triggers an
                    # immediate optimization pass.
                    deleted_threshold=0.0,
                ),
            )
    except asyncio.TimeoutError:
        return BulkOptimizeResult(
            success=False,
            optimize_elapsed_ms=int(time.monotonic() * 1000) - start_ms,
            final_count=-1,
            expected_count=expected_count_after_delete,
            count_ok=False,
            failure_step="optimize_trigger",
            failure_detail=(
                "update_collection() did not return within 5s — "
                "Qdrant may be under heavy load or unreachable."
            ),
        )
    except Exception as exc:
        return BulkOptimizeResult(
            success=False,
            optimize_elapsed_ms=int(time.monotonic() * 1000) - start_ms,
            final_count=-1,
            expected_count=expected_count_after_delete,
            count_ok=False,
            failure_step="optimize_trigger",
            failure_detail=f"update_collection() raised: {exc!r}",
        )

    # ── STEP B: Poll optimizer_status until "ok" ─────────────────────────────
    # Timeout budget: optimize_timeout_s (default 30s).
    # At 500 surviving points: wall-clock ~500ms–2s. 30s provides 15× margin.
    # At 5,000 points: wall-clock ~15–30s. Caller should pass optimize_timeout_s=120.
    poll_deadline = time.monotonic() + optimize_timeout_s
    optimizer_ok = False

    while time.monotonic() < poll_deadline:
        try:
            async with asyncio.timeout(3.0):
                collection_info = await qdrant_client.get_collection(collection_name)
        except (asyncio.TimeoutError, Exception):
            # Transient poll failure — keep trying until deadline
            await asyncio.sleep(poll_interval_s)
            continue

        optimizer_status = collection_info.optimizer_status
        # optimizer_status.status is a string: "ok", "optimizing", "error"
        if hasattr(optimizer_status, "status"):
            status_str = str(optimizer_status.status).lower()
        elif hasattr(optimizer_status, "ok"):
            # Some Qdrant client versions expose a boolean .ok
            status_str = "ok" if optimizer_status.ok else "optimizing"
        else:
            status_str = str(optimizer_status).lower()

        if status_str == "ok":
            optimizer_ok = True
            break

        await asyncio.sleep(poll_interval_s)

    if not optimizer_ok:
        elapsed_ms = int(time.monotonic() * 1000) - start_ms
        return BulkOptimizeResult(
            success=False,
            optimize_elapsed_ms=elapsed_ms,
            final_count=-1,
            expected_count=expected_count_after_delete,
            count_ok=False,
            failure_step="poll_timeout",
            failure_detail=(
                f"optimizer_status did not reach 'ok' within {optimize_timeout_s}s. "
                "The collection may still be optimizing. "
                "Do NOT re-enable writes or run canary tests yet. "
                "Re-run bulk_delete_and_optimize() or wait and retry manually."
            ),
        )

    # ── STEP C: Count verification ───────────────────────────────────────────
    # Timeout budget: 3s for the count call.
    try:
        async with asyncio.timeout(3.0):
            count_result = await qdrant_client.count(
                collection_name=collection_name,
                exact=True,
            )
        final_count = count_result.count
    except Exception as exc:
        elapsed_ms = int(time.monotonic() * 1000) - start_ms
        return BulkOptimizeResult(
            success=False,
            optimize_elapsed_ms=elapsed_ms,
            final_count=-1,
            expected_count=expected_count_after_delete,
            count_ok=False,
            failure_step="count_verify",
            failure_detail=f"count() raised after successful optimize: {exc!r}",
        )

    elapsed_ms = int(time.monotonic() * 1000) - start_ms
    count_ok = abs(final_count - expected_count_after_delete) <= count_tolerance

    if not count_ok:
        return BulkOptimizeResult(
            success=False,
            optimize_elapsed_ms=elapsed_ms,
            final_count=final_count,
            expected_count=expected_count_after_delete,
            count_ok=False,
            failure_step="count_mismatch",
            failure_detail=(
                f"Expected {expected_count_after_delete} ± {count_tolerance} points "
                f"after optimize, got {final_count}. "
                "Either the reconcile deletion count was wrong, or a concurrent write "
                "added/removed points during the optimize window. "
                "Inspect manually before re-enabling writes."
            ),
        )

    # All steps passed
    return BulkOptimizeResult(
        success=True,
        optimize_elapsed_ms=elapsed_ms,
        final_count=final_count,
        expected_count=expected_count_after_delete,
        count_ok=True,
        failure_step=None,
        failure_detail=None,
    )
```

### Timeout Budget Per Step Summary

| Step | Operation | Timeout | Failure Mode |
|------|-----------|---------|--------------|
| A | `update_collection()` trigger call | 5s | `optimize_trigger` — Qdrant unreachable or overloaded |
| B | Poll `optimizer_status` until `"ok"` | 30s (default) | `poll_timeout` — optimizer still running at deadline |
| C | `count(exact=True)` | 3s | `count_verify` — transient error after successful optimize |
| C (check) | Compare count to expected | instant | `count_mismatch` — deletion count wrong or concurrent write |

**Total worst-case wall-clock**: 5s + 30s + 3s = 38s. In practice at 500 points: 5s + 2s + 0.5s = 7.5s.

## Part 2: Failure Mode Handling Per Step

### Step A failure: `optimize_trigger`

**What happened**: The `update_collection()` API call to Qdrant did not return within 5 seconds, or returned an exception.

**Cause analysis**:
- Qdrant Docker container is unresponsive (OOM, crash, resource contention)
- Network timeout within Docker bridge
- Qdrant is in a recovery state from prior error

**Correct response**:
- Return `BulkOptimizeResult(success=False, failure_step="optimize_trigger")`
- Do NOT re-enable writes
- Do NOT proceed to canary test
- Operator action: check `docker compose logs qdrant`, restart Qdrant if needed, then re-run
- The deletion (performed by `/reconcile`) is already committed — Qdrant has soft-deleted the orphan points. The HNSW graph is in the degraded state (R@10 ≈ 0.73). Retrying `bulk_delete_and_optimize()` is safe (idempotent — the update_collection trigger is safe to re-issue regardless of prior state).

### Step B failure: `poll_timeout`

**What happened**: The `update_collection()` trigger succeeded but `optimizer_status` did not reach `"ok"` within `optimize_timeout_s`.

**Cause analysis**:
- Very large collection (>5,000 points) — caller passed the default 30s timeout which is too short
- Qdrant is under concurrent write pressure (new writes pause the optimizer)
- Qdrant optimizer encountered an internal error (check `optimizer_status.error` if present)

**Correct response**:
- Return `BulkOptimizeResult(success=False, failure_step="poll_timeout")`
- The optimizer is still running in the background — do NOT treat this as "optimization failed"
- Do NOT re-enable writes
- Options for caller:
  1. **Wait and poll manually**: Poll `GET /collections/{name}` every 5s until `optimizer_status.status == "ok"`, then proceed to count verify and canary
  2. **Re-run with longer timeout**: Call `bulk_delete_and_optimize(..., optimize_timeout_s=120.0)` — the `update_collection()` trigger is idempotent
  3. **Declare degrade and proceed with caution**: If the timeout was hit at 30s and Qdrant is known to still be running (not crashed), accept that optimization will complete soon. Log a warning, proceed to canary test — if canary passes (the optimizer may have just completed), allow write re-enablement. If canary fails, halt.

**Critical constraint**: A `poll_timeout` failure does NOT mean the HNSW graph is broken. It means the rebuild window exceeded the configured budget. The graph WILL complete rebuilding; the question is when.

### Step C failure: `count_verify` (exception)

**What happened**: The Qdrant `count()` call raised an exception after the optimizer completed.

**Cause analysis**:
- Transient network hiccup between optimizer completion and count call
- Qdrant is briefly busy with segment swap finalization

**Correct response**:
- Retry once after 2 seconds (single retry, not a loop)
- If retry succeeds: proceed with count comparison
- If retry fails: return failure, do NOT re-enable writes, operator must manually verify

### Step C failure: `count_mismatch`

**What happened**: The optimizer completed and the count call succeeded, but the actual point count differs from `expected_count_after_delete` by more than `count_tolerance`.

**Cause analysis** and appropriate response:

| Scenario | Count direction | Correct response |
|----------|----------------|------------------|
| `final_count` < `expected` by >2 | Over-deletion: more points were deleted than the reconcile report indicated | **HALT immediately** — possible Qdrant snapshot rollback candidate. Check if `recall:pre_cleanup_qdrant_snapshot` exists in Redis. Do NOT re-enable writes. |
| `final_count` > `expected` by >2 | Under-deletion: some orphan points were not removed, OR concurrent writes added points | Investigate: run `GET /reconcile?mode=post_restore&repair=false` to see if orphan count is now 0. If non-zero orphans remain, re-run reconcile + optimize. If zero orphans but count high, a concurrent write happened — safe to proceed. |
| Count matches within tolerance | As expected | `count_ok = True`, proceed to canary test |

## Part 3: Canary Seed Storage Format and Update Mechanism

### Storage Location and Format

Canary seeds are stored in Redis as a JSON-encoded list under the key `recall:canary_seeds`:

```python
# Redis key
CANARY_SEEDS_KEY = "recall:canary_seeds"

# Python type
from typing import TypedDict

class CanarySeed(TypedDict):
    seed_id: str           # Stable UUID, NOT the Qdrant point UUID (which changes on re-ingest)
    memory_id: str         # The Qdrant point UUID for this canary memory
    query_text: str        # The query string that should return this memory as top-1
    expected_uuid: str     # Same as memory_id — the Qdrant UUID we expect at rank 1
    content_snippet: str   # First 80 chars of memory content (for human audit of canary health)
    created_at: str        # ISO-8601 when this seed was created/refreshed
    collection_version: int  # Increments each time canary seeds are refreshed

# Example
canary_seeds: list[CanarySeed] = [
    {
        "seed_id": "canary-001",
        "memory_id": "3f2a1b4c-...",
        "query_text": "embedding model qwen3 dimensions",
        "expected_uuid": "3f2a1b4c-...",
        "content_snippet": "CANARY_001: the embedding model is qwen3-embedding:0.6b with 1536 dimensions",
        "created_at": "2026-03-16T00:00:00Z",
        "collection_version": 1,
    },
    # ... 9 more seeds
]
```

**Why separate `seed_id` from `memory_id`**: The `seed_id` is a stable human identifier for the canary slot (canary-001 through canary-010). The `memory_id` is the Qdrant UUID, which can change if the canary memory is deleted and re-ingested. `collection_version` lets the canary test code detect when a seed was last refreshed.

### When Canary Seeds Are Created

**Initial creation**: At Recall system initialization (`POST /admin/init-canaries` or called from startup if not present):

```python
async def init_canary_seeds(
    qdrant_client: AsyncQdrantClient,
    embedding_service: EmbeddingService,
    redis_client: Redis,
    collection_name: str = "memories",
    collection_name_domain: str = "system_canary",

    seeds = []
    for (seed_id, content), (_, query) in zip(CANARY_TEMPLATES, CANARY_QUERIES):

        point_uuid = str(uuid.uuid4())
        await qdrant_client.upsert(
            collection_name=collection_name,
            points=[
                PointStruct(
                    id=point_uuid,
                    vector=vector,
                    payload={
                        "content": content,
                        "domain": collection_name_domain,
                        "tags": ["canary"],
                        "importance": 0.0,  # Low importance — won't surface in real retrieval
                        "created_at": datetime.utcnow().isoformat(),
                        "is_canary": True,
                    },
                )
            ],
        )
        seeds.append(CanarySeed(
            seed_id=seed_id,
            memory_id=point_uuid,
            query_text=query,
            expected_uuid=point_uuid,
            content_snippet=content[:80],
            created_at=datetime.utcnow().isoformat(),
            collection_version=1,
        ))

    await redis_client.set(CANARY_SEEDS_KEY, json.dumps(seeds))
    return len(seeds)
```

### When Canary Seeds Are Refreshed

Canary seeds must be refreshed in exactly two scenarios:

1. **After a `/reconcile?mode=post_restore` cleanup that deleted canary points**: The canary memories are stored in Qdrant with `domain="system_canary"`. The reconcile endpoint's `created_at > backup_timestamp` filter will delete canary points that were created after the backup timestamp (i.e., any canary whose `created_at` is in the orphan window). The post-restore sequence must detect this and re-initialize canaries before running the canary test.

   **Detection**: In `bulk_delete_and_optimize()`, after the count verify step, compare the final count against the expected count. If `final_count` is within tolerance but the `/health/recall-quality` canary test returns fewer than 10 seeds found, the canary seeds were deleted by the reconcile and must be re-initialized.

   **Correct handling**: Run `init_canary_seeds()` after `bulk_delete_and_optimize()` completes and before calling `GET /health/recall-quality`. The `init_canary_seeds()` call will detect missing canary UUIDs and re-insert them.

2. **After a full Qdrant collection recreation**: If `recreate_collection()` is called (e.g., schema migration, manual reset), all canary UUIDs are invalidated. The Redis `recall:canary_seeds` key retains stale UUIDs pointing to non-existent Qdrant points. `init_canary_seeds()` handles this: it checks for missing UUIDs and refreshes.

**Canary seeds are NOT refreshed**:
- After normal memory deletions (canary domain is `system_canary`, separate from user memories)
- After `optimize_collection` (optimization does not delete points, only rebuilds the index)
- On a routine schedule (seeds are stable; their UUIDs are long-lived)

## Part 4: optimize_collection Polling Loop with Timeout and Backoff

### The poll loop (full specification)

```python
async def _poll_optimizer_until_ok(
    qdrant_client: AsyncQdrantClient,
    collection_name: str,
    timeout_s: float,
    poll_interval_s: float,
) -> tuple[bool, str]:
    """
    Poll GET /collections/{name} until optimizer_status.status == "ok".

    Returns:
        (True, "") on success.
        (False, reason_string) on timeout or error.
    """
    deadline = time.monotonic() + timeout_s
    consecutive_errors = 0
    MAX_CONSECUTIVE_ERRORS = 5

    while time.monotonic() < deadline:
        try:
            async with asyncio.timeout(3.0):
                info = await qdrant_client.get_collection(collection_name)
            consecutive_errors = 0  # Reset on success
        except asyncio.TimeoutError:
            consecutive_errors += 1
            if consecutive_errors >= MAX_CONSECUTIVE_ERRORS:
                return False, f"get_collection() timed out {MAX_CONSECUTIVE_ERRORS} times consecutively"
            await asyncio.sleep(poll_interval_s)
            continue
        except Exception as exc:
            consecutive_errors += 1
            if consecutive_errors >= MAX_CONSECUTIVE_ERRORS:
                return False, f"get_collection() raised {MAX_CONSECUTIVE_ERRORS} consecutive errors: {exc!r}"
            await asyncio.sleep(poll_interval_s)
            continue

        # Parse optimizer_status — Qdrant client versions vary in how they expose this
        status = _extract_optimizer_status(info)

        if status == "ok":
            return True, ""
        elif status == "error":
            # Qdrant optimizer encountered an internal error — not a timeout, a real failure
            error_detail = getattr(info.optimizer_status, "error", "no error detail available")
            return False, f"Qdrant optimizer entered error state: {error_detail}"
        # status == "optimizing" or unknown — keep polling
        await asyncio.sleep(poll_interval_s)

    return False, f"Optimizer did not reach 'ok' within {timeout_s}s"


def _extract_optimizer_status(collection_info) -> str:
    """Normalize optimizer_status across Qdrant client versions."""
    opt = collection_info.optimizer_status
    if opt is None:
        return "ok"  # No optimizer info means no optimization pending
    if hasattr(opt, "status"):
        return str(opt.status).lower()
    if hasattr(opt, "ok") and opt.ok is True:
        return "ok"
    if hasattr(opt, "ok") and opt.ok is False:
        return "optimizing"
    # Fallback: treat the object as a string
    s = str(opt).lower()
    if "ok" in s:
        return "ok"
    return "optimizing"
```

### Backoff strategy

The default `poll_interval_s=0.5s` is fixed-interval (no exponential backoff). This is intentional:

- **Why not exponential backoff**: The optimizer runs for a predictable wall-clock time (~500ms–2s at 500 points). Exponential backoff would delay detection of completion. The goal is to detect completion as soon as it happens.
- **Why 0.5s**: Half-second polls give 2 checks per second during the 500ms–2s optimization window at 500 points — detecting completion within 0.5–1.0s of actual completion. This is well within the 38s total budget.
- **For large collections (>5,000 points, optimize ~30s+)**: The caller can pass `poll_interval_s=2.0` to reduce polling frequency without meaningfully delaying detection.

### Timeout budget rationale

| Collection size | Estimated optimize time | Recommended timeout |
|----------------|------------------------|-------------------|
| ≤500 points | 500ms–2s | 30s (default, 15× margin) |
| 500–2,000 points | 2s–8s | 60s |
| 2,000–5,000 points | 8s–20s | 90s |
| >5,000 points | 20s–60s | 120s |

The default 30s is correct for Tim's current deployment (≤500 points). As the corpus grows, callers should pass a larger `optimize_timeout_s`.

## Part 5: What Happens If optimize_collection Exceeds the Timeout Budget

### The three options evaluated

**Option 1: Declare failure and halt** (recommended for the post-restore sequence)

Return `BulkOptimizeResult(success=False, failure_step="poll_timeout")`. The post-restore sequence returns an error to the operator. Qdrant's background optimizer continues running. The operator must:
1. Wait for optimizer to complete (check `GET /collections/{name}` manually)
2. Then call `GET /health/recall-quality` to verify R@10 is restored
3. Then re-enable writes manually

**Rationale**: The post-restore sequence is already blocking writes (503 gate is up). A poll_timeout does NOT mean the optimizer failed — it means the recovery is taking longer than expected. Halting here is safe: writes remain blocked (preventing new data from using a degraded index), and the operator receives a clear status message.

**Option 2: Retry with longer timeout**

The caller can immediately re-call `bulk_delete_and_optimize(..., optimize_timeout_s=120.0)`. The `update_collection()` trigger is idempotent. If the optimizer is still running, the new poll loop simply continues monitoring it.

**This is the correct recovery path for a poll_timeout failure.**

**Option 3: Degrade gracefully and re-enable writes** (NOT recommended for post-restore)

Accept the degraded HNSW state and re-enable writes, with R@10 ≈ 0.73 until the optimizer completes naturally. This is the default behavior if no explicit optimize is forced — but the Q100 physics analysis establishes clearly that R@10=0.73 means 2–3 of 10 retrieved memories are wrong or missing, which means Claude is receiving stale/deleted content in its context.

**Do not choose Option 3 for the post-restore scenario.** The optimizer completes in seconds to minutes at Tim's scale; there is no business justification for accepting a corrupted context window while waiting.

### Implementation Logic

```python
@router.get("/health/recall-quality")
async def recall_quality_health(
    min_pass_rate: float = Query(default=0.8, ge=0.0, le=1.0),
    timeout_per_query_ms: int = Query(default=300, ge=50, le=2000),
    redis: Redis = Depends(get_redis),
    qdrant: AsyncQdrantClient = Depends(get_qdrant_client),
    embedding_service: EmbeddingService = Depends(get_embedding_service),
) -> RecallQualityResponse:

    start_ms = int(time.monotonic() * 1000)

    # Load canary seeds from Redis
    seeds_raw = await redis.get(CANARY_SEEDS_KEY)
    if not seeds_raw:
        return RecallQualityResponse(
            status="error",
            seeds_found=0,
            seeds_expected=10,
            pass_count=0,
            fail_count=0,
            pass_rate=0.0,
            min_pass_rate=min_pass_rate,
            elapsed_ms=int(time.monotonic() * 1000) - start_ms,
            error="canary seeds not initialized; run POST /admin/init-canaries",
            query_results=[],
        )

    seeds: list[CanarySeed] = json.loads(seeds_raw)
    seeds_found = len(seeds)

    query_results = []
    pass_count = 0

    for seed in seeds:
        query_start = time.monotonic()
        result = await _run_single_canary_query(
            qdrant=qdrant,
            embedding_service=embedding_service,
            seed=seed,
            timeout_ms=timeout_per_query_ms,
        )
        result["elapsed_ms"] = int((time.monotonic() - query_start) * 1000)
        query_results.append(result)
        if result["passed"]:
            pass_count += 1

    fail_count = seeds_found - pass_count
    pass_rate = pass_count / seeds_found if seeds_found > 0 else 0.0
    elapsed_ms = int(time.monotonic() * 1000) - start_ms

    # Status determination
    if seeds_found < 10:
        status = "degraded"  # Some seeds are missing from Redis — needs re-init
    elif pass_rate >= min_pass_rate:
        status = "healthy"
    elif pass_rate >= 0.5:
        status = "degraded"  # Partial recall degradation
    else:
        status = "critical"  # Severe HNSW degradation — do not re-enable writes

    return RecallQualityResponse(
        status=status,
        seeds_found=seeds_found,
        seeds_expected=10,
        pass_count=pass_count,
        fail_count=fail_count,
        pass_rate=round(pass_rate, 3),
        min_pass_rate=min_pass_rate,
        elapsed_ms=elapsed_ms,
        error=None,
        query_results=query_results,
    )


async def _run_single_canary_query(
    qdrant: AsyncQdrantClient,
    embedding_service: EmbeddingService,
    seed: CanarySeed,
    timeout_ms: int,
) -> dict:
    try:
        async with asyncio.timeout(timeout_ms / 1000):
            vector = await embedding_service.embed(seed["query_text"])
            results = await qdrant.search(
                collection_name="memories",
                query_vector=vector,
                limit=10,  # Top-10 — the expected UUID should appear at rank 1
                with_payload=False,
            )
    except asyncio.TimeoutError:
        return {
            "seed_id": seed["seed_id"],
            "passed": False,
            "rank": None,
            "top1_uuid": None,
            "expected_uuid": seed["expected_uuid"],
            "reason": f"query timed out after {timeout_ms}ms",
        }
    except Exception as exc:
        return {
            "seed_id": seed["seed_id"],
            "passed": False,
            "rank": None,
            "top1_uuid": None,
            "expected_uuid": seed["expected_uuid"],
            "reason": f"query error: {exc!r}",
        }

    result_ids = [str(r.id) for r in results]
    expected = seed["expected_uuid"]

    if not result_ids:
        return {
            "seed_id": seed["seed_id"],
            "passed": False,
            "rank": None,
            "top1_uuid": None,
            "expected_uuid": expected,
            "reason": "empty result set",
        }

    top1_uuid = result_ids[0]
    passed = (top1_uuid == expected)

    # Also check if expected UUID appears anywhere in top-10 (not just rank 1)
    rank = (result_ids.index(expected) + 1) if expected in result_ids else None

    return {
        "seed_id": seed["seed_id"],
        "passed": passed,
        "rank": rank,
        "top1_uuid": top1_uuid,
        "expected_uuid": expected,
        "reason": None if passed else (
            f"expected UUID not at rank 1 (rank={rank})" if rank else
            "expected UUID not in top-10 results"
        ),
    }
```

## Part 7: Integration with the Q098 Restore Sequence

The Q098 runbook established an 8-step post-restore sequence. `bulk_delete_and_optimize()` and `/health/recall-quality` integrate as explicit replacements for Steps 5–7:

### Updated Sequence (incorporating Q100 and this finding)

```
STEP 1: Detect restore event
  - Q074 bidirectional guard fires: PG_RESTORE_SUSPECTED:gap=N
  - Identify backup_timestamp T from pg_dump file mtime or CasaOS snapshot metadata
  - Halt writes (return 503 from API)

STEP 2: Retract Redis epoch
  - redis-cli SET recall:global_epoch <pg_max_after_restore + 1>

STEP 3: Take Qdrant pre-cleanup snapshot (Q098 pre-safeguard)
  - POST /collections/memories/snapshots
  - Store snapshot name in Redis: recall:pre_cleanup_qdrant_snapshot (24h TTL)
  - This converts Qdrant deletion from irreversible to recoverable-in-30s

STEP 4: Execute three-store cleanup via /reconcile
  - POST /reconcile?mode=post_restore&backup_timestamp=T&pg_max_epoch=E&repair=false
    → verify counts look correct (dry-run)
  - POST /reconcile?mode=post_restore&backup_timestamp=T&pg_max_epoch=E&repair=true
    → execute cleanup; record {neo4j_deleted_count, qdrant_deleted_count} for Step 5
  *** /reconcile handles Neo4j DETACH DELETE and Qdrant DatetimeRange delete ***
  *** bulk_delete_and_optimize() is called AFTER this step completes ***

STEP 5 [NEW — replaces old "Force optimize_collection" step]:
  Call bulk_delete_and_optimize():
    expected_count = pre_restore_qdrant_count - qdrant_deleted_count
    result = await bulk_delete_and_optimize(
        qdrant_client=qdrant,
        collection_name="memories",
        expected_count_after_delete=expected_count,
        optimize_timeout_s=30.0,   # 30s for ≤500 points
    )
    if not result.success:
        → Log failure_step and failure_detail
        → If failure_step == "poll_timeout": wait and retry with timeout_s=120
        → If failure_step == "count_mismatch" and final_count < expected: HALT,
           consider snapshot rollback (recall:pre_cleanup_qdrant_snapshot in Redis)
        → Do NOT proceed to Step 6 until bulk_delete_and_optimize() returns success=True

STEP 6 [NEW — replaces old "count verify" step]:
  Refresh canary seeds if needed:
    - Check if canary seeds exist: GET /health/recall-quality
    - If seeds_found < 10 (some deleted by reconcile): POST /admin/init-canaries
    - After init: wait 2s for canary embeddings to be indexed

STEP 7 [NEW — replaces old "canary recall test" step]:
  Run GET /health/recall-quality
    - If status == "healthy" (pass_rate >= 0.8): proceed to Step 8
    - If status == "degraded": wait 5s, retry once
    - If retry still "degraded" or "critical": HALT, do not re-enable writes
      Log: "HNSW rebuild did not restore recall quality — R@10 estimate: {pass_rate}"
      Operator must investigate Qdrant logs before manual write re-enablement

STEP 8: Re-enable writes
  - Return 200 from API (lift 503 gate)
  - DEL recall:retrieval_mode (re-enable Neo4j traversal if vector-only mode was set)
  - Log: "post-restore sequence complete, N_qdrant=X, N_neo4j=Y, N_pg=Z,
          optimize_ms=W, canary_pass={pass_count}/10"
```

### Position of bulk_delete_and_optimize() in the sequence

```
/reconcile (deletion) → bulk_delete_and_optimize() → canary_seed_refresh? → /health/recall-quality → re-enable writes
```

The function is a **blocking gate** between deletion and write re-enablement. Writes must not resume until:
1. `bulk_delete_and_optimize()` returns `success=True`
2. `/health/recall-quality` returns `status="healthy"`

## Ideas Extracted

### Idea: typed-post-delete-optimize-gate

- **Novelty**: 0.75 — A typed, async `bulk_delete_and_optimize()` function with per-step timeout budgets, explicit failure_step categorization, and count verification as a post-optimize integrity check is not present in any reviewed AI memory system recovery implementation. The specific pattern of (a) triggering optimize via `update_collection(optimizer_config.deleted_threshold=0.0)` to force immediate vacuum, (b) polling `optimizer_status` with a deadline, and (c) verifying count within tolerance before declaring success converts a non-deterministic background process into an explicitly-gated recovery operation. The failure_step enum (`optimize_trigger`, `poll_timeout`, `count_mismatch`) provides structured error handling that is absent from Qdrant's own documentation for post-bulk-delete recovery.
- **Evidence**: 0.72 — The technical basis is well-established: Q100's physics analysis proves optimize_collection is mandatory after δ≥0.15 deletion; Qdrant's Python client exposes `update_collection()` and `get_collection()` with `optimizer_status`; the poll-until-stable pattern is standard operational practice for async operations in distributed systems (analogous to waiting for Elasticsearch index refresh or PostgreSQL REINDEX completion). The timeout budget (30s default, 15× margin at 500 points) is derived from Q100's measured 500ms–2s wall-clock for this scale.
- **Feasibility**: 0.88 — Implementation: ~80 lines of Python using the existing `AsyncQdrantClient`. No new infrastructure. The `BulkOptimizeResult` dataclass and `_poll_optimizer_until_ok()` helper are self-contained. Integration into the post-restore sequence is a 5-line caller in the `/reconcile` endpoint handler.

### Idea: canary-seed-stable-uuid-registry

- **Novelty**: 0.78 — Storing 10 synthetic canary memories in Qdrant at system initialization, with their UUIDs registered in Redis, provides a persistent ANN recall quality probe that survives restarts, schema migrations, and routine operations. The distinction between a stable `seed_id` (human identifier) and a mutable `memory_id` (Qdrant UUID, can change on re-ingest) prevents stale UUID references after canary re-initialization. The `collection_version` field enables canary health auditing across re-initialization events. The `is_canary: True` payload flag allows easy exclusion from production retrieval without filtering the entire system_canary domain.
- **Evidence**: 0.68 — Canary query sets for IR system regression testing are established practice at Elasticsearch/Solr deployments. The specific design (10 queries, expected top-1 UUID, paraphrase queries not keyword copies) follows information retrieval evaluation methodology. The semantic distinctiveness requirement (disjoint topic areas per canary, prefix `CANARY_XXX:`) is a standard technique for ensuring canary queries do not produce near-ties with real corpus content.
- **Feasibility**: 0.85 — Implementation: `init_canary_seeds()` is ~50 lines using the existing embedding service and Qdrant upsert. Redis JSON storage of the seed registry is 5 lines. The idempotent check (verify existing UUIDs, refresh only missing) prevents duplicate canary insertion on repeated calls. The endpoint `POST /admin/init-canaries` is a 10-line wrapper. Total: ~65 lines.

### Idea: three-status-health-endpoint-with-per-query-detail

- **Novelty**: 0.72 — A `/health/recall-quality` endpoint that returns per-query pass/fail detail (including the actual top-1 UUID returned vs. the expected UUID, and the rank of the expected UUID in top-10 results) provides diagnostic depth not present in any reviewed vector store health endpoint. Standard vector DB health endpoints report latency and availability — not retrieval quality. The three-status model (`healthy` / `degraded` / `critical`) with configurable `min_pass_rate` threshold and per-query `rank` reporting enables both automated gating (post-restore sequence) and operator diagnostics (which specific canary queries are failing and why).
- **Evidence**: 0.65 — The endpoint design follows from the Q100 finding that R@10=0.73 at δ=0.28 means ~2–3 of 10 expected top-1 results are incorrect. A 10-query canary test with a pass threshold of 8/10 detects this degradation reliably (if R@10=0.73, expected canary pass rate ≈ 0.73, which is below the 0.80 threshold). The per-query rank detail (expected UUID appears at rank 3 instead of rank 1) is more informative than a binary pass/fail — it distinguishes "HNSW graph is degraded but memory exists" from "HNSW graph lost the memory entirely."
- **Feasibility**: 0.85 — Implementation: ~80 lines of Python. Uses the existing embedding service and Qdrant search path. No new infrastructure. The `_run_single_canary_query()` helper is 30 lines. Integration as a FastAPI route is standard. The endpoint is callable as a one-shot health check during recovery or as a recurring health probe.
