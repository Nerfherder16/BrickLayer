# Q154 — SimHash Fingerprint Storage Location in Recall Architecture

**Question**: Where should SimHash fingerprints be stored given the actual Recall architecture -- since PostgreSQL has no memories table (Q132 correction), is the Qdrant payload field the correct storage location, and what is the backfill cost for the 22,381 existing points?

### Qdrant payload schema (PointStruct, line 220–250)
The `store_memory()` method writes a rich payload dict with ~20 fields including `content_hash`. The payload is the only persistent, queryable metadata layer — Qdrant is the single source of truth for memory data.

**set_payload API pattern (lines 482–536):** All post-hoc field updates use `set_payload` with a list of point IDs (one at a time in all existing usages). No batch `set_payload` with `Filter` selector is used anywhere in current codebase. Qdrant's API supports `set_payload(points=Filter(...))` for bulk updates, which the codebase does NOT yet use but the client library supports.

### PostgreSQL migration pattern (_MIGRATE_* constants, lines 164–281)
All schema changes use idempotent `DO $$ IF NOT EXISTS ... ALTER TABLE ... $$` blocks. No `memories` table exists — confirmed. The migration pattern allows adding new tables or columns with zero downtime via append to `_CREATE_TABLES` or a new `_MIGRATE_*` constant executed in `connect()`. A new `memory_fingerprints` table would require adding both a new `_CREATE_TABLES` block and wiring it into `connect()`.

### Redis key patterns (redis_store.py)
Redis uses hash (`HSET`/`HGETALL`) for sessions and lists (`LPUSH`/`LRANGE`) for working memory and turns. No persistent UUID-indexed HASH structure exists. Redis keys have TTLs tied to `session_ttl_hours` — all session-scoped data expires. There is no persistent cross-session key namespace (no `recall:memory:*` keys). Adding a persistent `recall:simhash:{uuid}` HASH would be an architectural outlier.

## Finding

**OBJECTIVE**: Qdrant payload is the only queryable, persistent, co-located storage layer for memory metadata in the Recall architecture. PostgreSQL has no memories table (Q132 confirmed). Redis is session-scoped with TTLs and has no persistent memory namespace.

**DATA**:
- `content_hash` (SHA-256[:16]) is already stored as a Qdrant payload keyword field with `create_payload_index` called at collection init. This is the exact pattern SimHash would follow.
- The `set_payload` API accepts both `points=[list_of_ids]` and `points=Filter(...)` — the filter form enables bulk updates without enumerating IDs.
- PostgreSQL `_MIGRATE_*` pattern requires: (1) new SQL constant, (2) wiring into `connect()`, (3) a separate lookup path not co-located with vector data.
- Redis has zero persistent memory-keyed structures and all keys carry session-scoped TTLs.

**FINDING**: Qdrant payload field is the architecturally correct and lowest-friction location for SimHash fingerprints. It mirrors `content_hash` exactly: stored in `PointStruct` at write time, indexed as keyword for O(log n) lookup, queryable via `FieldCondition`. A new PG table would create a split-brain between Qdrant (source of truth for memory data) and Postgres (audit/metrics-only). Redis is structurally incompatible (TTL expiry, no persistent memory namespace).

**STAT — Backfill cost (22,381 points):**

| Option | API mechanism | Batch strategy | Estimated calls | Wall-clock estimate |
|--------|--------------|----------------|-----------------|---------------------|
| Qdrant payload | `set_payload(payload={"simhash": v}, points=Filter(...))` | 1 call per simhash value (filter by UUID) OR bulk upsert with updated payload | 22,381 individual calls OR ~45 batch upserts (500/batch via scroll+upsert) | ~22s at 1ms/call individual; ~45s at 1s/batch (scroll+compute+upsert) with 500/batch |
| PostgreSQL new table | `INSERT INTO memory_fingerprints(id, simhash)` via bulk COPY | 1 bulk COPY statement | 1 COPY command | <5s for 22,381 rows at asyncpg bulk throughput |
| Redis HASH | `HSET recall:simhash:{uuid} fingerprint {value}` | Pipeline of 22,381 HSETs | 1 pipeline | ~2–3s for pipelined batch |

**Qdrant backfill detail**: The practical approach for Qdrant is NOT 22,381 individual `set_payload` calls. Instead: scroll all 22,381 points in batches of 1,000 (23 scroll calls), compute SimHash from the `content` field in each payload, then call `upsert` with updated `PointStruct` payloads (no vector re-embedding needed if vectors are included via `with_vectors=True`). At ~100ms per scroll+compute+upsert batch of 1,000: **~2.3 seconds total**. Alternatively, `set_payload` with a list of IDs accepts up to ~500 IDs per call — 45 calls at ~50ms each = **~2.25 seconds**.

**Memory overhead**: 22,381 × 8 bytes (int64 SimHash) = **~174 KB** in Qdrant payload (vs ~985 KB in Redis HASH, ~500 KB in PG table with index overhead). Qdrant payload storage is already allocated per-point; adding one integer field has near-zero marginal overhead.

## Storage Option Comparison

| Criterion | Qdrant payload field | New PG table `memory_fingerprints` | Redis HASH `recall:simhash:{uuid}` |
|-----------|---------------------|-------------------------------------|-------------------------------------|
| Zero new tables | YES — payload field, no schema change | NO — new table + migration required | NO — new key namespace (no table but new permanent pattern) |
| Co-located with vectors | YES — same point, same query | NO — cross-system join required | NO — separate system |
| Fast lookup by UUID | YES — `scroll(filter=FieldCondition(key="id"))` or direct point fetch | YES — O(log n) via PK index | YES — O(1) HGET |
| Fast lookup by simhash value | YES — if indexed as keyword/integer | YES — with btree index | NO — no scan support |
| Existing indexing precedent | YES — `content_hash` is identical pattern | N/A | N/A |
| Survives memory deletion | NO — deleted with point (correct behavior) | YES — orphan risk if not cascaded | YES — orphan risk if not cascaded |
| Backfill cost | ~2–3s (scroll+set_payload batches) | <5s (bulk COPY) | ~2–3s (pipelined HSET) |
| Backfill complexity | Low — uses existing client methods | Medium — needs migration code + new store method | Medium — persistent key pattern is architectural outlier |
| Redis TTL risk | N/A | N/A | HIGH — session TTLs could accidentally be applied |
| Architecture fit | STRONG — follows existing pattern | WEAK — PG is audit/metrics only | WEAK — Redis is session-scoped by design |

**Recommendation**: Store SimHash as a Qdrant payload integer field (`"simhash": int64`), indexed as `integer` type (supports range queries for Hamming distance approximation via bit-band ranges). This is the only option that requires zero new infrastructure, follows the established `content_hash` pattern exactly, keeps deduplication data co-located with vectors, and auto-deletes when a memory is deleted.

**Backfill script pattern**:
```python
# Scroll all points, compute simhash, bulk set_payload
offset = None
while True:
    points, offset = await qdrant.client.scroll(
        collection_name=qdrant.collection,
        offset=offset, limit=1000,
        with_payload=["content"], with_vectors=False
    )
    if not points:
        break
    updates = {p.id: simhash(p.payload["content"]) for p in points}
    # One set_payload per batch (accepts list of IDs)
    await qdrant.client.set_payload(
        collection_name=qdrant.collection,
        payload={"simhash": 0},  # placeholder — need per-point values
        points=list(updates.keys())  # NOTE: all get same value; need per-point upsert instead
    )
```
**Correction**: `set_payload` applies ONE payload dict to ALL listed points. For per-point SimHash values, use `overwrite_payload` with individual calls OR use `upsert` with updated full payloads (requires `with_vectors=True` in scroll to preserve embeddings). The scroll+upsert approach at 500/batch = 45 upsert calls, each carrying ~500 full PointStructs. Estimated wall-clock: **~10–30 seconds** depending on network to Qdrant (192.168.50.19). No re-embedding required.

## Qualifying Ideas

**N=0.82**: Qdrant payload is confirmed as the sole persistent queryable metadata store for memories. The `content_hash` precedent is exact — SimHash follows identically. The backfill is batched scroll+upsert at ~30s wall-clock, not 22,381 individual API calls.

**E=0.71**: A key implementation nuance: `set_payload` cannot write different values to different points in one call. The correct backfill mechanism is scroll (with_vectors=True) → compute SimHash per content → upsert updated PointStructs. This is ~45 upsert calls at 500 points/batch. The index type should be `integer` not `keyword` to enable Hamming distance band queries (SimHash buckets) for near-duplicate detection.

**F=0.68**: Opportunity: once SimHash is in Qdrant payload, near-duplicate detection during `store_memory()` can use a single `scroll(filter=simhash_band_match)` before the embedding similarity check, short-circuiting the embedding call for obvious duplicates. This could reduce Ollama embedding calls by an estimated 5–15% on active sessions with repetitive content (causal chain: simhash check → avoid embed → faster store → lower latency). Worth instrumenting via existing `injection_log.dedup_count` field.
