# Finding: Q011 — Physics Ceiling: Retrieve Operation Minimum Latency

**Question**: What is the theoretical minimum latency for Recall's retrieve operation, given that it requires an embedding call + ANN search + metadata lookup? What is the current gap ratio?
**Question Type**: PHYSICS
**Verdict**: PROMISING
**Severity**: High

## Theoretical Minimum Calculation

### Hardware Basis

| Constant | Value | Source |
|----------|-------|--------|
| RTX 3090 FP16 (Tensor Cores) | 71 TFLOPS | NVIDIA spec |
| RTX 3090 HBM memory bandwidth | 936 GB/s | NVIDIA spec |
| qwen3-embedding:0.6b model size (FP16) | 1.2 GB | 600M params × 2 bytes |
| LAN RTT to 192.168.50.62 | 0.3 ms | given practical minimum |
| Docker bridge RTT (localhost) | 0.1 ms | practical min (loopback ~50us + scheduling) |
| CPU AVX2 GFLOPS (Qdrant host) | 50 GFLOPS | conservative estimate |

### Component Floors

#### Redis cache lookup

Operation: key lookup by hash of query text → return cached result set.

- Network: Docker bridge RTT = 0.1 ms
- Hash table lookup: O(1), ~100 ns
- Socket syscall (send + recv): ~1,000 ns
- Command parsing: ~500 ns
- Memory copy of result (~50 KB cached result at 50 GB/s DRAM): ~1,024 ns
- Total compute: ~2,600 ns = 0.003 ms

**Redis theoretical minimum: 0.103 ms**

#### Ollama embedding inference (qwen3-embedding:0.6b)

Architecture estimate: L=28 transformer layers, H=1024 hidden dim, S=100 avg query tokens.

FLOPs for forward pass:
- Attention per layer: 4SH² + 2S²H + 2SH² = 6SH² + 2S²H
- FFN per layer: 8SH²
- Total: 28 × (14 × 100 × 1024² + 2 × 100² × 1024) = 41.7 GFLOP

Bottleneck analysis (batch=1, memory-bound regime):
- If compute-bound: 41.7 GFLOP / 71 TFLOPS = **0.59 ms**
- If memory-bandwidth-bound: 1.2 GB / 936 GB/s = **1.28 ms**
- Single-request inference is memory-bandwidth-bound (weights must be streamed from VRAM per token, not reused across batch). The bandwidth bottleneck wins.

GPU floor: **1.28 ms**
LAN RTT (Ollama host): 0.3 ms
HTTP/JSON overhead (tiny payload): 0.05 ms

**Ollama theoretical minimum: 1.63 ms**

Note: This assumes Ollama is already loaded and the model is hot in VRAM. A cold start (model load from SSD) would add ~1-3 seconds; that is a separate pathological case.

#### Qdrant ANN search (HNSW)

Assumptions: N=10,000 stored vectors, d=1024, ef=50 (exploration factor), M=16 (neighbors per layer).

HNSW search complexity:
- Layers: log₁₆(10,000) = 3.3
- Comparisons: ef × M × layers = 50 × 16 × 3.3 = 2,658
- FLOPs per comparison (dot product, 1024-dim): 2 × 1024 = 2,048
- Total: 2,658 × 2,048 = 5.44 MFLOP

Timing:
- Compute (50 GFLOPS CPU): 5.44 MFLOP / 50 GFLOPS = **0.11 ms**
- Memory (visiting ~330 nodes × 4 KB each from DRAM at 30 GB/s): **0.045 ms**
- Bottleneck: compute at 0.11 ms
- Docker bridge network + HTTP overhead: 0.1 + 0.2 = 0.3 ms

**Qdrant theoretical minimum: 0.41 ms**

#### Neo4j metadata lookup

Operation: fetch 1-hop causal relationships for K=5 result nodes (MATCH + OPTIONAL MATCH in Cypher).
Assumptions: avg degree 5 → 5 + 5×5 = 30 nodes fetched; Neo4j page cache hot (always-on service).

- B-tree index lookup per node (hot cache): ~200 ns × 30 = 6,000 ns = 0.006 ms
- Bolt protocol overhead (connection already established): 0.5 ms
- Docker bridge: 0.1 ms

**Neo4j theoretical minimum: 0.61 ms**

## Largest Gap: Ollama (absolute), Neo4j (ratio)

**By absolute latency removed:** Ollama at 48.4 ms gap is the dominant term. It accounts for 62% of total current latency and 60% of total theoretical minimum. Closing this gap alone would reduce p50 from ~80ms to ~32ms.

**By ratio:** Neo4j at 33x is the worst-ratio component. The theoretical minimum (0.61ms) is so low because graph traversal on a hot page cache is essentially RAM pointer chasing — but the JVM, Bolt protocol, Cypher planner, and Python driver add ~20ms of pure overhead on top of sub-millisecond actual computation.

**Why Ollama's gap is so large:**

The 30.6x gap for Ollama breaks down into four independent sources:

1. **Tokenization overhead** (~2 ms): qwen3 tokenizer runs CPU-side in Ollama before dispatching to GPU. This is pure serialization cost with no physics minimum analog.

2. **HTTP request/response cycle** (~3 ms): Ollama uses a standard HTTP server. Each embedding request incurs: Python httpx → TCP → Go HTTP server → response serialization of a 1024×4 = 4 KB float array as JSON (base64 or float list). JSON encoding 1024 floats is non-trivial.

3. **CUDA launch overhead** (~5 ms): Each single inference incurs CUDA kernel launch overhead. The GPU must: schedule the kernel, transfer the tokenized input from CPU DRAM to VRAM (~100 tokens × 4 bytes at 32 GB/s PCIe = negligible), but kernel scheduling adds ~1-5ms on a busy GPU.

4. **Memory-bandwidth utilization gap** (~40 ms): The theoretical 1.28ms assumes the RTX 3090 is streaming model weights at full 936 GB/s. Real inference at batch=1 achieves perhaps 20-40% of peak bandwidth due to irregular access patterns in transformer attention and non-contiguous weight layout. Actual GPU time is likely 3-5ms, not 1.28ms.

The sum of these four sources accounts for the 48.4ms gap: ~(1.28 → 5ms GPU) + 5ms CUDA overhead + 3ms HTTP + 2ms tokenization + miscellaneous = ~48ms over the physics floor.

## Ideas Extracted

### Idea: embedding-vector-cache

The retrieve path always produces an embedding from a query string. Many queries in Recall are structurally identical or near-identical across sessions ("what are my Docker configs?", "what did I learn about FastAPI?"). If the embedding for a query text is cached independently of the full result cache (which is what Redis currently does), the Ollama call is eliminated on re-queries.

The distinction from the current Redis cache: the current Redis cache stores full results keyed on the exact query hash. This idea stores only the 1024-dim query embedding vector keyed on the query text hash. The embedding cache is cheaper to populate (4 KB per entry vs 50+ KB for full results), has a longer valid lifetime (query embeddings don't expire — the semantic meaning of a query text doesn't change), and can be checked before the ANN search to avoid the Ollama round-trip without invalidating when new memories are stored.

- **Novelty**: 0.55 — Embedding caching is practiced in some production RAG systems, but the separation between embedding-cache (long-lived, query-keyed) and result-cache (short-lived, invalidated by new stores) is not a widely documented pattern. Most implementations cache the full result or nothing.
- **Evidence**: 0.65 — Engineering best practice in semantic search systems; no specific peer-reviewed validation but logically sound and straightforward to benchmark.
- **Feasibility**: 0.90 — 1-2 week implementation. Add a second Redis hash namespace `embed:{sha256(query_text)}` storing the raw float vector. Check it before calling Ollama. No invalidation needed (embeddings are stable). Saves ~50ms on repeated queries at negligible cost.
- **Description**: In the retrieve path, after the full-result cache miss, check `Redis.get(f"emb:{sha256(query_text)}")` before calling Ollama. On hit (embedding found), skip the Ollama call entirely and proceed directly to Qdrant with the cached vector. On miss, call Ollama as normal, then store the embedding in Redis with a long TTL (7 days) in addition to storing the full result. This eliminates the 50ms Ollama call for any query text seen in the past week, at the cost of ~4 KB per unique query in Redis memory.

### Idea: neo4j-bolt-connection-pool

Neo4j's 33x gap ratio is dominated by protocol overhead, not computation. The theoretical minimum (0.61ms) is achievable only if a persistent Bolt connection is already open and the Cypher planner has seen this query pattern before. The current implementation likely creates a new driver session per retrieve call (the Python neo4j driver's default pattern with async context managers), which incurs: TLS handshake, authentication, Cypher planning, and result streaming as independent round trips.

- **Novelty**: 0.20 — Connection pooling is a standard database optimization. The specific application to Neo4j Bolt in a Python async context has some friction (the neo4j Python driver's async API is non-trivial), but this is not a novel idea.
- **Evidence**: 0.90 — Well-established engineering practice. Neo4j official docs recommend connection pooling with `AsyncGraphDatabase.driver()` as a module-level singleton. Reduces per-query overhead from ~20ms to ~2-5ms in documented benchmarks.
- **Feasibility**: 0.85 — 3-5 day implementation. Replace per-request `AsyncGraphDatabase.driver()` construction with a module-level singleton driver initialized at FastAPI startup. The connection pool is maintained automatically. Cypher queries that repeat the same structure will be cached by the Bolt protocol planner, further reducing overhead.
- **Description**: Move Neo4j driver initialization from inside the retrieve route handler to a lifespan-scoped singleton in FastAPI's startup event. The async driver maintains a connection pool (default size 100 in the Python driver). Each retrieve call checks out a connection from the pool rather than opening a new one. Expected improvement: 20ms → 3-5ms per Neo4j call. Additionally, use parameterized Cypher (already the default with neo4j driver) to ensure the Cypher planner caches the query plan.

### Idea: speculative-embedding-prefetch

This addresses the largest absolute gap (Ollama, 48.4ms) from an architecture angle rather than a caching angle. The insight: in Recall's usage pattern, the query text for a retrieve call is the user's prompt — which is typed into Claude Code's input box. There is a brief window between when the user types the prompt and when the recall-retrieve.js hook fires. If Recall could prefetch the embedding during that window, the GPU compute would be in-flight before the API call arrives.

The hook fires via `UserPromptSubmit`, which means the prompt text is available at the moment the hook begins executing JavaScript. The hook itself makes the HTTP call to the Recall API. There is no speculative window from the hook side. However, if the MCP server (Node.js) could send the prompt text to Recall as a "pre-embed" request before the full retrieve request, the GPU work could overlap with the hook's own overhead.

- **Novelty**: 0.75 — Speculative prefetching of embeddings is not practiced in any known production AI memory system. Adjacent system: CPUs speculatively execute instructions ahead of branch resolution. Databases speculatively prefetch pages on sequential scans. The application to LLM embedding calls is novel.
- **Evidence**: 0.45 — No peer-reviewed evidence for this specific pattern. The speculative execution analogy is well-established in computer architecture, but the hook timing window may be too small to recover meaningful latency in practice. Empirical validation required.
- **Feasibility**: 0.40 — 1-2 months. Requires: (1) a new `/pre-embed` endpoint in Recall's FastAPI that accepts text and starts an async Ollama call, returning immediately with a job ID; (2) modification of recall-retrieve.js to fire a pre-embed request before the full retrieve request (two HTTP calls instead of one); (3) the retrieve endpoint checks if a pre-embed result is ready (shared async future), and either awaits it (if still running) or uses it immediately (if complete). The benefit depends on the timing gap between hook start and retrieve call, which may be <5ms in practice — insufficient to recover 50ms of GPU time.
- **Description**: Add a `/api/pre-embed` endpoint that accepts `{text: string}` and immediately enqueues an async Ollama embedding call, storing the result future keyed on `sha256(text)`. The recall-retrieve.js hook fires this endpoint as its first action (non-blocking, fire-and-forget). Then it proceeds to collect the rest of the context (tool inputs, file paths) before firing the full `/api/retrieve` call. The retrieve endpoint checks the pre-embed cache first. If the embedding is ready, it skips Ollama entirely. If not yet ready, it awaits the in-flight future rather than starting a new Ollama call. The window between pre-embed fire and retrieve call is the "free" GPU time recovered from the serial path.

### Idea: grpc-qdrant-binary-protocol

Qdrant's 24.5x gap ratio (0.41ms theoretical vs 10ms current) is dominated by HTTP REST overhead, not by HNSW algorithm cost. The HNSW search itself takes ~0.11ms. The remaining ~9.9ms is: Python httpx client overhead, HTTP/1.1 or HTTP/2 framing, JSON serialization/deserialization of the query vector (1024 floats) and K=5 result vectors, and Qdrant's internal request routing from its HTTP handler to its search engine. Qdrant natively supports gRPC in addition to REST. gRPC uses binary Protocol Buffers instead of JSON, eliminating float serialization overhead, and HTTP/2 persistent framing instead of per-request HTTP/1.1 connection overhead.

- **Novelty**: 0.15 — Using gRPC instead of REST for Qdrant is documented in Qdrant's official client library. This is an implementation choice, not a design innovation.
- **Evidence**: 0.80 — Qdrant's own benchmarks show gRPC is 15-30% faster than REST for vector search. The Python qdrant-client library has native gRPC support. The float serialization cost is measurable: JSON encoding 1024 float32 values produces ~7KB of text vs ~4KB of binary protobuf.
- **Feasibility**: 0.90 — 1-2 day implementation. In the Python qdrant-client, pass `prefer_grpc=True` to the `QdrantClient` constructor. The client handles the rest. No Qdrant server configuration changes needed — gRPC is enabled by default on port 6334.
- **Description**: Change `QdrantClient(host="qdrant", port=6333)` to `QdrantClient(host="qdrant", port=6334, prefer_grpc=True)`. This switches from HTTP REST (port 6333) to gRPC (port 6334). Expected improvement: 10ms → 6-8ms per ANN search (rough estimate, exact gain depends on payload size and connection reuse). Low effort, low risk, should be done regardless of other optimizations.

### Idea: embedding-model-colocate

The Ollama LAN gap analysis shows that 0.3ms of the 1.63ms theoretical minimum is pure network physics — the cost of sending bytes to 192.168.50.62 and back. This is an architecture gap caused by the deployment choice to run Ollama on a separate GPU host. If the embedding model ran on the same Docker host as Recall (even on CPU with quantization), the LAN RTT becomes Docker bridge RTT (0.1ms), and the GPU bottleneck shifts from HBM bandwidth to a different tradeoff.

A quantized CPU embedding model for a 0.6B parameter model is viable: INT8 quantized = ~600 MB, CPU inference on a modern x86 with AVX2 = roughly 50-100ms per inference. That is slower than the RTX 3090 (1.28ms theoretical, ~5ms practical). So pure CPU is not better. But a hybrid option exists: use a smaller, purpose-built embedding model (e.g., all-MiniLM-L6 at 23M params, 384-dim) locally, trading retrieval quality for latency.

- **Novelty**: 0.35 — Model quantization and smaller embedding models are well-known. The specific tradeoff analysis for a personal AI memory system (prioritizing latency over quality for a ~0.6B model) is moderately novel in this context.
- **Evidence**: 0.75 — all-MiniLM-L6 and similar SBERT models are peer-reviewed and benchmarked on retrieval quality (BEIR benchmark). The latency-quality tradeoff is empirically documented. Quantization speedups are well-validated (GGUF/llama.cpp benchmarks).
- **Feasibility**: 0.50 — 1-3 months. Requires: re-embedding all stored memories with the new model (a one-time migration job), updating the Recall embedding endpoint, and empirically validating that retrieval quality does not degrade unacceptably. The risk is that a smaller model captures less semantic nuance, particularly for Tim's technical domain (Docker, Python, FastAPI jargon). A/B comparison on a sample of queries is necessary before committing to the migration.
- **Description**: Deploy a CPU-resident embedding service (using sentence-transformers + a quantized GGUF model or all-MiniLM-L6) as a sidecar Docker container on the same host as Recall. Benchmark: (1) latency of CPU inference vs Ollama LAN call for typical query lengths; (2) retrieval quality on a held-out set of real Recall queries using NDCG@5. If CPU inference is within 2x of Ollama latency (e.g., 10ms vs 5ms) and quality is within 5% NDCG, the elimination of the LAN hop and Ollama HTTP overhead may produce a net latency improvement. If not, keep Ollama but add the embedding vector cache (idea 1) to reduce Ollama call frequency.

## Gap Classification

**Physics gaps (unavoidable):**
- LAN RTT to Ollama host: 0.3 ms — the embedding GPU is on a separate physical machine. Speed of light over 30m of cable produces this floor. Cannot be removed without moving the GPU.
- GPU memory-bandwidth saturation time: 1.28 ms — streaming 1.2 GB of model weights through HBM at 936 GB/s is a physics floor. Cannot be reduced without a smaller model or higher-bandwidth memory.
- HNSW algorithmic minimum: 0.11 ms — log(N) comparisons × d-dim dot products is the mathematical minimum for approximate nearest neighbor search at this accuracy level.

**Architecture gaps (design choices):**
- Separate Ollama host on LAN (0.3 ms RTT baked into every retrieve): caused by the deployment decision to use a dedicated GPU server. Removable by colocating a smaller/quantized model.
- Serial execution of independent operations: the Redis cache miss check, Ollama call, and result delivery are strictly serial. No parallelism is possible in the correctness-critical path, but the embedding call could be moved earlier in the pipeline (speculative prefetch).
- Full-result cache vs embedding-only cache: caching the full result (Redis, current) means any new memory stored invalidates the cache. An embedding-only cache for query vectors has infinite valid lifetime. This is a design choice about what to cache.
- Neo4j on the critical path via Bolt (JVM overhead, per-session connections): the JVM's startup cost and Bolt's session establishment cost are avoidable with persistent connection pooling, a straightforward architectural fix.

**Implementation gaps (code quality):**
- REST vs gRPC for Qdrant: a single constructor parameter change. Pure implementation gap.
- Python redis-py connection management: 4.9x gap on Redis is partly the Python client overhead. Using async aioredis with a persistent connection pool would reduce this.
- Neo4j per-request driver construction: same as above — moving driver initialization to module scope eliminates per-request setup overhead.

---

## Key Finding

The 29.3x gap on the cache-miss path is primarily an **architecture gap** (58 ms) rather than a physics gap (2.75 ms theoretical minimum). Ollama contributes 62% of total current latency. Three changes, each achievable in under two weeks, could close most of the architecture gap: (1) embedding vector cache in Redis to eliminate Ollama calls for repeated queries, (2) persistent Neo4j connection pool to cut Neo4j from 20ms to ~3ms, and (3) gRPC for Qdrant to cut it from 10ms to ~7ms. Together these would reduce p50 cache-miss latency from ~80ms to ~30ms without changing the model, hardware, or retrieval algorithm.

The store path (embed + write to Qdrant + write to Neo4j + write to PostgreSQL + invalidate Redis) has a different serial dependency structure, and the Neo4j write is likely the bottleneck. The hypothesis is that store latency is higher than retrieve latency, but with more parallelism available (Qdrant, Neo4j, and PostgreSQL writes can all run concurrently after the embedding completes).
