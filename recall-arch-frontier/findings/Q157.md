# Q157 — Double-Fault Orphan Survival Through Reconciliation

**Question**: Does the double-fault scenario (Neo4j write fails AND the compensating Qdrant delete also fails) produce a ghost memory that persists through the weekly reconcile scan — specifically, does the current reconcile.py detect and repair the Qdrant-without-Neo4j orphan class, or does this class survive reconciliation indefinitely?

**Verdict**: BREAKTHROUGH

---

## Code Trace

### reconcile.py — What Happens to qdrant_orphans

```python
# Line 29
qdrant_orphans = qdrant_ids - neo4j_ids

# Lines 83–99
for memory_id in qdrant_orphans:
```

**Critical finding**: The worker (`reconcile.py`) does NOT delete qdrant_orphans. It CREATES a new Neo4j node for them, treating Qdrant as the source of truth (stated in the module docstring: "Qdrant is source of truth. Orphaned Neo4j nodes are deleted; Qdrant nodes missing from Neo4j are recreated.").

## Thread A Analysis (Optimistic) — Partially Correct, Wrong Conclusion

Thread A is correct that reconcile detects qdrant_orphans. It is wrong about what it does with them.

**What Thread A predicted**: ghost orphans are deleted at reconcile time.
**What actually happens**: ghost orphans are PROMOTED. The reconcile system reads the Qdrant payload and writes a fresh Neo4j node from it, treating the ghost as a valid memory that just lost its graph counterpart.

The double-fault orphan does not get deleted. It gets permanently installed into Neo4j.

---

## Thread B Analysis (Pessimistic) — Correct Direction, Understated

Thread B asked whether reconcile actually deletes qdrant_orphans. The answer is no. But the outcome is worse than "does nothing" — reconcile actively heals the wrong way by creating the missing Neo4j node, converting the transient ghost into a fully normalized, permanent memory in both stores.

---

## Failure Mode Taxonomy

### The Double-Fault Lifecycle

1. **T=0**: POST /memory/store writes Qdrant (success). Writes Neo4j (fails). Compensating qdrant.delete() also fails. Result: ghost Qdrant point, no Neo4j node.

2. **T=0 to T+7 days**: Ghost Qdrant point participates in ANN similarity search. Every vector query potentially returns this ghost in results. No Neo4j metadata to back it up (no graph traversal, no relationship context, no importance weighting from Neo4j). But the raw content is in the Qdrant payload — it surfaces in search hits.

3. **T+7 days (Sunday 5:30am reconcile)**: Worker detects the ghost as a qdrant_orphan. Rather than flagging it for deletion or quarantine, it reads the Qdrant payload fields (content, memory_type, source, domain, tags, importance) and calls `neo4j.create_memory_node(memory)`. The ghost is now fully normalized. It has passed the point of no return.

4. **Post-reconcile**: The double-fault orphan is now indistinguishable from any legitimately stored memory. It has both a Qdrant vector point AND a Neo4j node. Future reconcile runs will see it in `common_ids`, not `qdrant_orphans`.

---

## The ANN Injection Window

The ghost Qdrant point participates in similarity search for up to 7 days before reconcile runs. During this window:

- The ghost has a valid vector embedding (written at T=0 before the Neo4j failure)
- The Qdrant payload has the content fields intact (content, memory_type, domain, tags)
- ANN queries will return it in top-K results based purely on vector similarity
- The retrieval hook (`recall-retrieve.js`) uses these results to inject context into Claude prompts

This is a confirmed ANN injection surface: an unvalidated, possibly partial-write memory that escaped commit semantics is actively participating in context retrieval for up to 7 days.

---

## Key Asymmetry: The "Source of Truth" Decision

The module docstring explicitly states the design intent:

```
Qdrant is source of truth. Orphaned Neo4j nodes are deleted;
Qdrant nodes missing from Neo4j are recreated.
```

This design decision is the root cause of the vulnerability class. It is correct for the normal case (Qdrant is the primary store, Neo4j is derived metadata). It is incorrect for the double-fault case (a Qdrant point that exists ONLY because the compensating delete failed is not a valid source of truth — it is an uncommitted write artifact).

The reconcile system has no mechanism to distinguish:
- A valid Qdrant point whose Neo4j node was accidentally deleted (should be recreated)
- A double-fault ghost whose Neo4j write never succeeded (should be deleted)

Both look identical: Qdrant has the point, Neo4j does not.

---

## Neo4j Orphan Handling Contrast

For the symmetric case (Neo4j has the node, Qdrant does not), the reconcile system DOES delete:

```python
# Lines 72–78
for memory_id in neo4j_orphans:
    try:
        await neo4j.delete_memory(memory_id)
```

Neo4j orphans are deleted. Qdrant orphans are promoted. The asymmetry is intentional (Qdrant-as-truth) but creates the vulnerability class described here.

---

## Is 7 Days Too Long?

For the ANN injection risk: yes. A ghost memory from a double-fault write participates in vector similarity search for up to 168 hours. Depending on content, this could:

- Surface in unrelated queries via semantic similarity
- Inject incorrect or partial context into Claude sessions
- Persist permanently post-reconcile with no audit trail indicating its anomalous origin

The weekly cadence is appropriate for the intended use case (eventual consistency for legitimate data). It is too slow for quarantining uncommitted write artifacts.

---

## Missing Mechanisms

1. **No write-ahead log or pending-write registry**: There is no record that the double-fault occurred, so reconcile cannot distinguish it from a valid Qdrant point.

2. **No quarantine vs. promote distinction**: Qdrant orphans are always promoted. There is no quarantine path.

3. **No creation-timestamp check**: If reconcile compared the Qdrant point's creation time to the Neo4j write pattern, very recent orphans (< 1 hour old) might be candidates for quarantine rather than promotion.

4. **No payload validation on reconcile**: reconcile.py reconstructs a Memory object from Qdrant payload fields without verifying that the content makes semantic sense or matches any session record.

---

## Summary

**The double-fault orphan does NOT survive reconciliation as a ghost — it survives as a legitimate, fully-normalized memory.** Reconciliation detects the orphan correctly but repairs it in the wrong direction: instead of deleting the uncommitted ghost, it creates the missing Neo4j node and installs the memory permanently. After reconcile, the ghost is promoted to a first-class memory indistinguishable from any intentionally stored one.

Thread A is wrong about the mechanism (deletion vs. promotion). Thread B is correct that reconcile does not delete qdrant_orphans, but understated the outcome — the result is worse than survival, it is permanent installation.

The ANN injection window is up to 7 days, during which the ghost participates in real retrieval. Post-reconcile, the ghost is permanent. The system has no mechanism to distinguish a double-fault artifact from a valid orphaned memory, because Qdrant-as-source-of-truth is a blanket policy with no transactional semantics.

---

## Verdict: BREAKTHROUGH

This is a structural finding, not an edge case. The "Qdrant is source of truth" design principle, combined with the absence of any write-pending registry, guarantees that double-fault orphans are promoted rather than quarantined. Any double-fault event creates a permanently installed, unaudited memory. The existing weekly reconcile does not mitigate this — it converts the transient ghost into a permanent fixture.
