## Deletion Path Analysis

Reading `src/api/routes/memory.py` lines 952–976 (`delete_memory` endpoint) and lines 1398–1411 (batch delete):

```
delete_memory() execution order:
  1. await qdrant.delete(memory_id)              ← FIRST
  2. await neo4j.delete_memory(memory_id)        ← SECOND
  3. await qdrant.delete_facts_for_memory(...)   ← OPTIONAL (try/except skips on error)
```

**The deletion path DOES propagate to Qdrant.** Qdrant is deleted first, Neo4j second.

### Failure Mode Asymmetry

The Qdrant-first ordering has a structural consequence for orphan accumulation:

| Failure scenario | Result | Orphan type | ANN injection risk |
|---|---|---|---|
| `qdrant.delete()` fails | Exception propagates; deletion request fails entirely | No orphan created | None |
| `qdrant.delete()` succeeds, `neo4j.delete_memory()` fails | Qdrant point deleted, Neo4j node remains | **Neo4j orphan** | None (silently skipped in BFS) |
| Both succeed | Clean deletion | None | None |

**Key conclusion**: The Qdrant-first ordering structurally favors Neo4j orphans over Qdrant orphans in the transient-failure case. This is the better failure mode: Neo4j orphans are silently skipped in BFS (Q081), while Qdrant orphans are actively injected into context (Q086). The current implementation makes the less-dangerous failure the more likely one.

### No Compensating Delete for Neo4j Failure

If `neo4j.delete_memory()` fails after `qdrant.delete()` succeeds, there is no compensating rollback — the Qdrant delete is not reversed. This permanently creates a Neo4j orphan for that UUID. Under normal operation, Neo4j orphans would accumulate at the rate of Neo4j write failures. At the current measurement (22,381 Qdrant = 22,381 Neo4j), the accumulation rate has been effectively zero — no Neo4j write failures have left orphans.

## The Q125 Specification Correction

Q125 assumed a `memories` table in PostgreSQL modeled on prior findings (Q086 described PG as "the secondary audit store" but the Q115/Q125 analysis incorrectly assumed PG retained a `memories` row per vector). The actual architecture is:

| Store | Role | Memory records? |
|---|---|---|
| Qdrant | Canonical memory store — all Memory objects reconstructed from payload | YES — 22,381 points with full payload |
| Neo4j | Graph store — CO_RETRIEVED edges, memory nodes | YES — 22,381 Memory nodes |
| PostgreSQL | Audit/ops store only | NO — no `memories` table |

This means the "S4+S5_new Direction 2" diagnostic (Q125, line 164: "PG restore deletes memories from PG that still exist in Qdrant") is also incorrect. A PG restore does not delete memory records because PG never stored them. PG restore would only revert audit_log rows, session_archive rows, and metrics. The dangerous restore scenario is a **Qdrant volume revert or Neo4j volume revert**, not a PG restore.

### Revised S4+S5_new Diagnostic Matrix

| Pattern | Correct three-store interpretation |
|---|---|
| S4=CRIT + S5_new=HEALTHY | Epoch integrity violation (Redis-PG audit timing) without store-level orphans |
| S4=CRIT (epoch jump) + S5_new=CRIT | Qdrant or Neo4j selective restore without corresponding cleanup in the other store |
| S5_new=CRIT alone | Steady-state orphan accumulation: `neo4j.delete_memory()` repeatedly failing after `qdrant.delete()` succeeds |

## Health Endpoint Gap

The `/health` endpoint exposes `qdrant (N memories)` and `neo4j (N nodes)` counts in human-readable form but does not compute or surface the S5_new orphan fraction. To implement S5_new as a health signal:

```python
# Fast approximation (add to health_check())
qdrant_n = await qdrant.count()
neo4j_n  = (await neo4j.get_statistics())['memories']
orphan_approx = max(0, qdrant_n - neo4j_n)
orphan_fraction = orphan_approx / qdrant_n if qdrant_n else 0.0

checks['s5_orphan_fraction'] = {
    'value': round(orphan_fraction, 4),
    'status': 'HEALTHY' if orphan_fraction < 0.01
              else 'WARNING' if orphan_fraction < 0.05
              else 'CRITICAL'
}
```

The precise UUID-level scan (scroll Qdrant IDs, diff against Neo4j IDs) is implementable via the existing `/reconcile` endpoint at cost ~3–8 seconds for the 22,381-point corpus, making it suitable for scheduled weekly probes but not synchronous health checks.

[LIMITATION] The fast approximation (count comparison) cannot detect equal-count UUID mismatch: a scenario where N Qdrant points were deleted and N different points were created without corresponding Neo4j updates would show count=0 but have 2N orphans. A UUID-level set-difference scan via `/reconcile` is required for ground truth. This scan was not executed due to its 3–8 second wall-clock cost and non-availability of a direct Qdrant connection (port 6333 refused externally; only the Recall API proxy is accessible).

[LIMITATION] The Qdrant `count()` method reads `points_count` from `get_collection()` info, which may reflect the collection's indexed count rather than the precise live count during active writes. [LIMITATION] The Q125 correlation matrix rows for S4+S5_new were written under the incorrect assumption that PG stores memory rows. All five S4+S5_new combination rows in Q125 require re-evaluation under the corrected Qdrant-Neo4j definition.

## Qualifying Ideas

[N>=0.65, E>=0.40]

- **s5-spec-correction-neo4j-denominator**: (N=0.82, E=0.90, F=0.95) — Q125 specified S5_new as `qdrant_count - pg_count`, but PostgreSQL has no `memories` table in this deployment. The correct denominator is Neo4j. The `/reconcile` endpoint already implements `qdrant_ids - neo4j_ids` as the orphan set — S5_new must be redefined to use this comparison. This correction changes the recovery action too: PG-authoritative reconciliation (Q125 Recommendation 2) is wrong; the correct recovery is Neo4j-authoritative reconciliation (delete Qdrant points not in Neo4j, or create Neo4j nodes for Qdrant orphans — direction depends on which failure occurred). Implement by updating the Q125 specification and the planned S5_new health probe.

- **qdrant-first-deletion-ordering-as-safety-property**: (N=0.75, E=0.88, F=0.92) — The `delete_memory()` function deletes Qdrant first, then Neo4j. This ordering is a latent safety property: in the transient-failure case (one store delete fails), the Qdrant-first ordering structurally produces Neo4j orphans (silently skipped in BFS, Q081) rather than Qdrant orphans (actively injected into context, Q086). This is the safer orphan type. The ordering is not documented as a deliberate choice in the codebase — it is an accidental safety property that should be documented and preserved. A future refactor that reverses the order (Neo4j first, Qdrant second) would convert the failure mode from silent-skip to active-injection without any visible API change.

- **pg-is-not-a-memory-store-correction**: (N=0.78, E=0.90, F=0.98) — Multiple prior findings (Q084, Q086, Q115, Q125) contain references to "PG memories table", "PG restore deletes memory rows", and "pg_count of memories". These are all incorrect for the actual deployed system. PostgreSQL in Recall stores only: audit_log, session_archive, metrics_snapshot, injection_log, profile_proposals, tuning_ledger, tuning_baselines, prompt_metrics, membench_runs. The dangerous restore scenario for memory consistency is not "PG restore" but "Qdrant volume revert" or "Neo4j volume revert without the other". The Q074 bidirectional restore guard and the Q077 recovery runbook are both targeted at the wrong store. The epoch (Redis-PG epoch comparison) is about operational timing, not memory record integrity.

- **s5-health-endpoint-gap**: (N=0.68, E=0.88, F=0.90) — The `/health` endpoint exposes Qdrant and Neo4j counts as human-readable strings (`"ok (22381 memories)"`) but does not compute the S5_new orphan fraction or surface it as a structured field. Adding `s5_orphan_fraction: {value, status}` to the health response requires 4 lines of code (subtract counts, divide, threshold-classify) and zero new network calls since both counts are already fetched for the health check. This is the lowest-cost possible health signal addition — it requires no new queries, no schema changes, and no new dependencies.

## Recommendations

1. **[SPECIFICATION] Update Q125 S5_new formula to use Neo4j as denominator** — High priority, low effort. Replace all references to `pg_count` in the S5_new probe with Neo4j node count. Update the correlation matrix rows for S4+S5_new: the dangerous restore scenario is Qdrant or Neo4j volume revert, not PG restore. Update Q125 Recommendation 2 (PG-authoritative reconciliation) to reflect that the correct recovery direction depends on which store is authoritative (Qdrant is canonical for Memory objects; Neo4j for graph relationships).

2. **[IMPLEMENTATION] Add S5_new orphan fraction to `/health` response** — Medium priority, trivial effort. Add 4 lines to `health_check()`: compute `max(0, qdrant_count - neo4j_count) / qdrant_count`, threshold-classify, add to `checks` dict as a structured field. The dashboard and MCP tools can then surface this as a health signal without any additional calls.

3. **[DOCUMENTATION] Document the Qdrant-first deletion ordering as a safety invariant** — Medium priority, low effort. Add a comment in `delete_memory()` stating that Qdrant is deleted first intentionally, because Qdrant orphans (active injection) are more dangerous than Neo4j orphans (silent BFS skip). This prevents a future refactor from reversing the order without understanding the failure-mode implications.

4. **[CALIBRATION] Run UUID-level set-difference scan via `/reconcile?repair=false`** — Low priority (given current HEALTHY count parity), medium effort. The `/reconcile` endpoint already scrolls all Qdrant IDs and all Neo4j IDs and returns the symmetric difference. Running it with `repair=false` gives the precise orphan count without any mutations. If the result returns non-zero `qdrant_orphans` or `neo4j_orphans`, follow up with Q135 (timing analysis for the precise scan at 22,381-vector scale).

5. **[ARCHITECTURE] Add `memories` table to PostgreSQL for cross-store consistency** — Low priority given current architecture works, high architectural value. Adding a lightweight PG `memories (id UUID PRIMARY KEY, created_at TIMESTAMPTZ, tenant_id TEXT)` table — written at store time, deleted at delete time — would enable the three-way consistency check originally envisioned in Q125 and Q086. This would make PG the tiebreaker in disagreements between Qdrant and Neo4j. Current cost: 0 PG rows for memories because the table doesn't exist. Implementation cost: one migration + two lines in `store()` and `delete_memory()`.
