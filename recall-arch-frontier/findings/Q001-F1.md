# Finding: Q001-F1 — Adjacent Field: LSM-Tree Compaction for Recall Cold Store

**Question**: What mechanism does a log-structured merge tree (LevelDB/RocksDB) use to decide when to compact (merge + discard superseded data), and how would it apply to Recall's unbounded cold store?
**Question Type**: ADJACENT
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Log-structured merge trees (LevelDB, RocksDB, Cassandra TWCS)

---

## Mechanism Deep-Dive

### Background: Why LSM Trees Need Compaction

LSM trees decouple writes from in-place updates. Every write — including updates and deletes — appends a new record. Superseded values and tombstones (delete markers) accumulate until a compaction pass explicitly merges SSTables, removes outdated versions, and eliminates tombstones. Without compaction, the store grows unboundedly and read latency degrades because every read must scan multiple SSTables for the latest version of each key.

This is structurally identical to Recall's problem: every session stores new embeddings, some of which are updated or redundant versions of facts already in Qdrant. Without a compaction analog, Qdrant accumulates superseded memory embeddings indefinitely.

---

### Mechanism 1: Level-Based Compaction (RocksDB Leveled Compaction)

**How it works — the trigger:**

RocksDB maintains a compaction score for every level:
- L0: `score = max(file_count / level0_file_num_compaction_trigger, total_L0_size / max_bytes_for_level_base)`. Default trigger is 4 files in L0.
- L1 through Lmax: `score = level_size_bytes / target_level_size_bytes`.

The level with the highest score above 1.0 is selected for compaction. Only one level compacts at a time (though RocksDB supports subcompaction parallelism within a level). Target sizes grow by a factor of `max_bytes_for_level_multiplier` (default: 10) per level: L1=256MB, L2=2560MB, L3=25.6GB, etc.

**How files are selected:**

Within the chosen level, file selection follows a compaction priority policy. The default (`kOldestSmallestSeqFirst`) selects the file covering the oldest sequence numbers — the file that has accumulated the most un-compacted new writes on top of it. An alternative (`kOldestLargestSeqFirst`) selects the file most likely to have dense tombstones. Once a file is selected from Lk, all overlapping files in Lk+1 are pulled into the compaction.

**The merge operation:**

Files from Lk and overlapping files from Lk+1 are k-way merge sorted. For each key, only the latest version is retained (all superseded versions are discarded). Tombstones are dropped if no earlier version of the key exists at lower levels (i.e., the tombstone has propagated all the way down). The output is one or more new SSTables written to Lk+1.

**Write amplification:** A record written to L0 may be rewritten O(log N / log fanout) times as it flows down levels. With fanout=10 and 5 levels, write amplification is roughly 10-50x. This is the cost of maintaining the sorted, non-overlapping invariant within each level.

**Space amplification:** Because each level has non-overlapping key ranges, at most one live version of each key exists per level (plus one at most from higher levels). Space amplification is approximately 1.1x (10% overhead from data being compacted downward at any moment).

**Tombstone TTL and periodic compaction:**

RocksDB addresses tombstones that never reach the bottom level due to sparse write patterns via two mechanisms:
- `ttl`: files older than the configured TTL are force-scheduled for compaction.
- `periodic_compaction_seconds`: files older than N seconds are eligible for compaction even if their level's score is below 1.0. Default: 30 days.

These ensure that tombstones eventually propagate and deleted data is truly removed.

---

### Mechanism 2: Size-Tiered Compaction Strategy (STCS)

**How it works:**

STCS groups SSTables into buckets by size. A bucket contains all SSTables whose size falls within a configurable ratio (default: 0.5–2.0x of a "mean" size for that tier). When any bucket accumulates `min_threshold` SSTables (default: 4), all SSTables in that bucket are merged into a single larger SSTable, which then falls into a higher-size bucket.

This is "write-efficient" compaction: data is only merged when there are multiple SSTables of similar size, avoiding the repeated small-into-large writes of leveled compaction.

**Write amplification:** Lower than leveled — typically 3-10x. A record is merged only when it lands in a bucket with enough peers.

**Space amplification:** Higher than leveled — O(T) where T is the fanout. Because SSTables at different size tiers may contain overlapping key ranges, multiple versions of the same key can exist across tiers simultaneously. In Scylla testing with T=4, 1.2 GB of data occupied 9.3 GB on disk during peak compaction.

**Failure mode:** Updates or deletes in a write-heavy workload cause space amplification to grow until the next compaction cycle merges the tiers. For AI memory systems with frequent "update this fact" operations, STCS can hold 5-8x the actual live data on disk.

---

### Mechanism 3: FIFO Compaction

**How it works:**

FIFO compaction does not merge SSTables. It simply drops the oldest L0 SSTable when the total database size exceeds a configured limit. This is "cache-like" compaction: it enforces a bounded store at the cost of discarding old data entirely, without deduplication or merging.

**Use case:** Time-series data where old data has no value. RocksDB uses it for metrics stores and ephemeral caches.

**Write amplification:** ~1x — no rewrites, just appends and drops.

**Space amplification:** Bounded by `max_table_files_size` configuration.

**Limitation:** FIFO has no deduplication semantics. It cannot distinguish between "this is a superseded version of an important memory" and "this is a unique memory from 6 months ago." It discards by age alone.

---

### Mechanism 4: Cassandra TWCS (Time Window Compaction Strategy)

**How it works:**

TWCS divides time into fixed-width windows (e.g., 1 hour, 1 day). Each SSTable is assigned to a window based on the maximum timestamp of its records. Within a window, SSTables accumulate via STCS rules (compact when 4+ SSTables of similar size exist). When a time window closes (transitions from current to expired), a major compaction merges all its SSTables into a single SSTable.

SSTables from different time windows are never compacted together. This is the defining constraint.

**Tombstone and TTL mechanics:**

TWCS is most powerful when all data in a window has a uniform TTL. When an entire SSTable's worth of data expires simultaneously, the SSTable is dropped without requiring a compaction pass — zero write amplification for deletion. This is analogous to dropping an entire memory segment rather than hunting through it for individual deleted items.

**Failure modes:**
1. Non-TTL deletes create tombstones in one window that can never be merged with the deleted data in another window — tombstones persist indefinitely.
2. Out-of-order writes (repairs, hints) contaminate newer windows with old-timestamp data, delaying expiry.
3. Updates to the same key across windows cause multiple versions to live in separate windows permanently, preventing deduplication.

**Write amplification:** Very low — approximately 1x within a window, with one major compaction per window on close.

**Space amplification:** Low once windows expire. But can be high transiently when multiple windows contain overlapping key versions.

---

### Mechanism 5: Tombstone Mechanics (Cross-Strategy)

In all LSM strategies, a delete generates a tombstone record — a key with a special "deleted" marker. The tombstone must propagate all the way to the lowest level (Lmax in leveled, or the largest size tier in STCS) before the key's data can actually be freed. The tombstone cannot be removed at an intermediate level because it might hide an older version of the same key at a lower level.

The k-way merge sort during compaction detects superseded values by key ordering: when multiple records share the same key, only the one with the highest sequence number survives. Tombstones survive until they are the only record for a key and they have propagated past all levels that could hold older versions.

## Application to Recall

### The Structural Analogy

Qdrant is Recall's LSM-equivalent cold store. Each `recall_store` call writes a new embedding. If the user writes "I prefer Python 3.12" and later writes "I now use Python 3.13", both records exist in Qdrant as separate vectors. The newer fact does not supersede the older one — Qdrant has no compaction analog that would merge these.

The result: Qdrant accumulates:
1. Redundant memories (same fact restated across sessions)
2. Superseded memories (old versions of facts that have been updated)
3. Stale memories (things that were true but no longer are, with no "tombstone" to mark them)
4. Noise (low-importance single-session memories that were never worth long-term storage)

Qdrant does have a Merge Optimizer that consolidates small segments into larger ones (analogous to STCS bucket consolidation) and a Vacuum Optimizer that removes soft-deleted vectors when deleted vectors exceed 20% of a segment. But these operate at the storage layer — they do not reason about semantic content. Two vectors representing "Python 3.12" and "Python 3.13" are not deduplicated by Qdrant's optimizer because they are different vectors with different payloads.

The missing layer is semantic compaction: a process that identifies semantically near-duplicate or superseded memory embeddings and merges, updates, or tombstones them.

---

### Mapping Each LSM Mechanism to Recall

**Level-Based Compaction → Tiered Semantic Compaction:**

L0→L1→L2 maps naturally to Recall's memory lifecycle:
- L0: current session buffer (Redis hot cache, last N=100 memories)
- L1: recent project context (memories from last 30 days, ~1,000 records)
- L2: long-term cold store (everything older, unbounded)

A compaction score for each tier: `session_memories_count / session_threshold` triggers L0→L1 compaction (session-end consolidation job from Q007). L1 size exceeding a configurable threshold triggers L1→L2 compaction. The key insight from leveled compaction: the trigger is score-based, not time-based. This produces adaptive scheduling — if a heavy session writes 200 memories, compaction fires immediately; a light session with 5 memories defers compaction until the buffer accumulates.

**STCS Bucket Triggering → Clustering-Based Batch Deduplication:**

STCS's insight — wait until you have N similar-sized units before compacting them — maps to batch deduplication in Recall. Rather than checking each new memory individually against all existing Qdrant records (expensive), accumulate new memories in a staging area and run a batch clustering pass when the batch reaches size N. Within each cluster, keep the most recent or highest-importance record and tombstone the others.

This is more efficient than the one-at-a-time cosine similarity check implied by the Q007 session-end-consolidation-job idea. A STCS-style batch approach enables HNSW approximate nearest-neighbor search across the whole batch simultaneously, reducing the number of Qdrant queries from O(N) per session to O(1) batch queries per compaction cycle.

**FIFO Compaction → Age-Based Cold Store Pruning:**

For memories below an importance threshold and older than a configurable age, FIFO semantics are appropriate: drop the oldest low-importance memories when the store exceeds a soft size limit. This is a simpler and more defensible policy than LLM-driven importance judgment: if a memory has never been retrieved in 180 days and its importance score is below a threshold, drop it.

This does not require understanding the content — it only requires access count and age metadata, both of which Recall can track.

**TWCS Window Boundary Drop → Session-Window-Based Expiry:**

TWCS's most powerful feature is dropping entire SSTable files when all their data expires. The Recall analog: assign each memory a session_id and a TTL category (session-scoped, project-scoped, permanent). Session-scoped memories (things relevant only to a single coding session) get a 7-day TTL. At the session window boundary, drop all session-scoped memories as a batch — zero embedding-by-embedding overhead, just a single Qdrant collection filter-delete.

This is significantly simpler than importance-decay-based eviction and has zero false-positive risk (you explicitly tagged the memory as session-scoped at write time).

**Tombstone Propagation → Contradiction Marking:**

LSM tombstones solve the "later information supersedes earlier information" problem. Recall needs a version of this: when a new memory explicitly contradicts an old one ("I now use Python 3.13" contradicts "I use Python 3.12"), the old memory should be tombstoned. The mechanism: during semantic compaction, when cosine similarity between a new memory and an existing memory exceeds 0.92 (near-duplicate), check whether the new memory's content can be classified as "update" vs. "addition." If update, tombstone the old record (mark it `superseded=True` in Qdrant payload) and increment the new memory's importance score. Tombstoned memories are excluded from retrieval but retained for 30 days before physical deletion (in case the tombstone was erroneous).

## Ideas Extracted

### Idea: tiered-compaction-score-trigger

- **Novelty**: 0.90 — No production AI memory system implements a score-based compaction trigger (level_size / target_size > 1.0 analog). All known systems use either time-based triggers (run every N hours), event-based triggers (on session end), or no trigger at all. The score-based model is adaptive and self-regulating without requiring calendar scheduling.
- **Evidence**: 0.95 — Level-based compaction with score-based triggers is the default compaction strategy in RocksDB (deployed at Facebook, LinkedIn, Dropbox, Uber, Netflix) and is validated in production at petabyte scale. The CIDR 2017 paper "Optimizing Space Amplification in RocksDB" (Dong et al.) documents the score formula and its behavior under production workloads. The formula is simple, deterministic, and well-understood.
- **Feasibility**: 0.80 — Implementation requires: (1) a `compaction_score` function that divides current Qdrant collection size by a tier target, (2) a background scheduler (APScheduler already likely in Recall's FastAPI) that checks the score, (3) the actual compaction logic (batch semantic deduplication). The score computation is a single Qdrant `count()` call. The trigger threshold is a configuration value. Estimated 1-2 weeks for the trigger scaffolding; the compaction logic is the bulk of the work.
- **Description**: Add a `CompactionScheduler` class to Recall's FastAPI server. On each session-end hook, compute `score = qdrant.count(collection) / target_size` for each logical tier (session buffer, project context, cold store). If score > 1.0 for any tier, enqueue a compaction job for that tier. The session buffer target might be 500 memories (fire after every session that grows beyond this), the project context target 5,000 memories, and the cold store target 50,000 memories. This decouples the compaction trigger from calendar time — heavy-use sessions trigger more compaction, quiet periods defer it. The score can also incorporate a "staleness" component: `score = (collection_size / target) + (days_since_last_compaction / compaction_interval)`. When the staleness component alone exceeds 1.0, periodic compaction fires regardless of size.

### Idea: stcs-batch-semantic-deduplication

- **Novelty**: 0.85 — The specific pattern of accumulating new memories in a staging batch and running a single clustering pass (rather than per-memory similarity checks) is absent from production AI memory systems. mem0 does per-write similarity checks (synchronous, one at a time). AWS AgentCore runs LLM-driven consolidation. The STCS insight — batch when you have N similar-sized inputs — has not been applied to semantic deduplication.
- **Evidence**: 0.80 — STCS is production-validated in Apache Cassandra (the default strategy until TWCS), ScyllaDB, and was the original Cassandra compaction strategy. The batch efficiency insight is well-established: k-way merge of N SSTables is more efficient than N sequential one-at-a-time merges. For semantic deduplication, the equivalent is that a single HNSW approximate nearest-neighbor search across a batch of N new memories is substantially cheaper than N individual cosine similarity searches against the full Qdrant index.
- **Feasibility**: 0.75 — Requires: (1) a Redis staging queue (not the hot cache — a separate list of recently written memory IDs awaiting compaction), (2) a batch HNSW query against Qdrant once the queue reaches size N (e.g., N=50 or N=100), (3) a clustering step on the results (group memories whose nearest neighbor cosine distance is below 0.12), (4) for each cluster, merge into a single canonical record. The main implementation challenge is defining the merge semantics: which record wins, how to combine metadata (tags, importance scores, session IDs). Estimated 3-4 weeks including the merge algebra.
- **Description**: Add a `StagingQueue` (Redis list, key `recall:staging:pending`) to which `recall_store` appends each new memory ID immediately after writing to Qdrant. A background job checks the queue size; when it reaches `batch_size=100` memories, it runs: (1) fetch all 100 memory embeddings from Qdrant, (2) compute pairwise HNSW search for each to find near-duplicates within cosine distance 0.10, (3) build connected components (groups of near-duplicate memories), (4) for each component with >1 member, merge them: keep the most recently written record's text, union all tags, sum importance scores, delete duplicates with `superseded=True` tombstone. This processes 100 memories in roughly 2-3 Qdrant API calls rather than 100 individual calls. The batch size of 100 is calibrated to typical session output — a busy session writes ~50-100 memories, making the batch align naturally with session boundaries.

### Idea: ttl-category-session-window-drop

- **Novelty**: 0.80 — The TWCS insight of assigning memories to TTL categories at write time and dropping entire categories as a batch is absent from production AI memory systems. Most systems either apply uniform TTL to all memories (missing the category distinction) or apply no TTL at all (Recall's current state). The specific pattern of dropping an entire logical window (session-scoped memories from session X) as a single bulk operation rather than individual TTL checks is novel in AI memory systems.
- **Evidence**: 0.85 — TWCS is production-validated in Apache Cassandra and ScyllaDB specifically for time-series workloads. The core mechanism (bulk SSTable drop on window expiry) is deployed at scale across thousands of Cassandra clusters for IoT time-series data. The window-based drop is the most efficient delete mechanism in any LSM system — it requires no compaction pass, no tombstone propagation, just a file delete.
- **Feasibility**: 0.90 — This is the most feasible idea in this finding. Implementation requires: (1) adding a `memory_scope` payload field to each Qdrant record at write time: values are `session`, `project`, or `permanent`, (2) adding a `session_id` field for session-scoped memories, (3) a lightweight cleanup job that runs `qdrant.delete(filter={"must": [{"key": "memory_scope", "match": {"value": "session"}}, {"key": "created_at", "range": {"lt": cutoff_timestamp}}]})`. No semantic reasoning required. No embedding comparisons. A single Qdrant filter-delete. Estimated 1-2 days of implementation. The main design decision is what qualifies as session-scoped vs. project-scoped at write time — this can be a simple heuristic (memories tagged with only session-specific entities are session-scoped; memories mentioning project names, technology preferences, or reusable facts are project-scoped).
- **Description**: Extend `recall_store` to classify each new memory into one of three scopes at write time: `session` (useful only for the current conversation, e.g., "user is currently debugging a specific error"), `project` (useful across sessions within a project, e.g., "this project uses FastAPI"), or `permanent` (fundamental preferences and facts, e.g., "user prefers Python"). Add `memory_scope` and `session_id` as Qdrant payload fields. On session end, the existing `recall-session-summary.js` hook triggers a cleanup job that deletes all `memory_scope=session` records from the previous session using a Qdrant filter delete (not individual record deletes). After 30 days, delete all `memory_scope=project` records not accessed in 30 days via a scheduled maintenance job. Permanent memories are never auto-deleted. This mirrors TWCS's window-boundary drop: session window closes → entire session's ephemeral memories are dropped as a batch. No cosine similarity needed. No LLM calls needed.

### Idea: semantic-tombstone-supersession

- **Novelty**: 0.85 — LSM tombstone semantics applied to AI memory supersession is novel. No production AI memory system implements a formal tombstone mechanism where an updated memory explicitly invalidates its predecessor in the store. mem0 updates memories at write-time but does not retain a tombstone for the old version — the old version is overwritten. Zep/Graphiti retains temporal edges but does not mark old nodes as tombstoned/superseded in the retrieval path. The pattern of: new memory detected as update → old memory marked `superseded=True` → old memory excluded from retrieval → old memory physically deleted after a grace period, is not implemented anywhere.
- **Evidence**: 0.90 — LSM tombstone mechanics are extremely well-validated across every production LSM implementation (LevelDB, RocksDB, Cassandra, ScyllaDB, HBase). The tombstone pattern has been in production for 15+ years. The semantic extension (detecting content-level supersession rather than exact key match) requires adding a cosine similarity check, but the tombstone pattern itself is fully proven.
- **Feasibility**: 0.65 — Requires: (1) a supersession detector: during compaction, for each new memory, query Qdrant for existing memories with cosine distance < 0.08 and overlapping tags, (2) a content classifier: determine if the new memory is an "update" (same topic, different value) or an "addition" (new information on same topic), (3) if update: mark old memory `superseded=True` in Qdrant payload, log the supersession event in PostgreSQL, (4) a delayed physical delete: cron job removes `superseded=True` records older than 30 days. The content classifier in step 2 is the main complexity — it either requires an LLM call or a heuristic based on embedding distance and syntactic analysis. A conservative threshold (cosine distance < 0.05, meaning essentially the same sentence rephrased) allows a non-LLM implementation. Estimated 4-6 weeks for a robust implementation.
- **Description**: During the STCS batch deduplication pass (see stcs-batch-semantic-deduplication), add a supersession check: for each cluster of near-duplicate memories, compare their content timestamps. If a newer memory in the cluster uses language indicating replacement ("now", "currently", "changed to", "updated"), mark the older cluster members `superseded=True`. Superseded memories are immediately excluded from retrieval (Qdrant filter on `must_not: [{key: "superseded", match: {value: true}}]`). They are retained for 30 days as an audit trail — if the supersession was erroneous, it can be reversed by clearing the flag. After 30 days, a batch delete removes them permanently. This implements the LSM tombstone guarantee: no memory consumes retrieval resources once superseded, but the data is not immediately destroyed (allowing rollback). Unlike LLM-driven conflict resolution (mem0's approach), this is a payload flag set by a deterministic rule — fast, auditable, and reversible.

### Idea: dynamic-level-size-target

- **Novelty**: 0.75 — RocksDB's dynamic level size feature (the level target grows proportionally as data accumulates, rather than using fixed targets) is not applied to AI memory tier sizing. Most AI memory systems use fixed limits for each tier. The dynamic model means the cold store tier naturally expands to accommodate growing user histories without manual reconfiguration.
- **Evidence**: 0.80 — RocksDB's dynamic level size compaction (documented in the RocksDB blog post "Dynamic Level Size for Level-Based Compaction," 2015) is a production feature used at Facebook and LinkedIn scale. The key benefit: prevents the "last level too small" problem where L1 repeatedly overflows into L2 even when the total store could accommodate L1 data without compaction. This is directly analogous to Recall's situation: the "project context" tier may hold 500 memories for a small project but 5,000 for a large one — a fixed tier target wastes compaction cycles on the small project and triggers too-frequent compaction on the large one.
- **Feasibility**: 0.80 — Implementation is a one-line formula: `tier_target = base_target * (total_memories / base_memories)^(tier_level / max_levels)`. This replaces fixed per-tier targets with dynamically computed ones. The formula is evaluated once per session end (when compaction scores are checked). No additional infrastructure required. Estimated 2-4 hours of implementation once the compaction score trigger (tiered-compaction-score-trigger) is in place.
- **Description**: Replace fixed tier targets in the CompactionScheduler with dynamically computed targets. Formula: `tier_target(k) = base_size * multiplier^k` where `multiplier` is computed from `total_store_size / base_total_target`. As Recall's total memory store grows from 1,000 to 10,000 to 100,000 memories, each tier's target grows proportionally — the session buffer stays at ~5% of total, the project context stays at ~10%, and the cold store expands to fill the remainder. This means the compaction frequency automatically adjusts to workload: a user with 100 memories compacts rarely; a user with 100,000 memories compacts more often but with larger batches. No manual tuning of tier boundaries is required.

## Qdrant's Existing Compaction vs. What Is Missing

Qdrant's Merge Optimizer (consolidates small segments) and Vacuum Optimizer (removes soft-deleted vectors above 20% threshold) operate at the storage layer. They are the analogs of LSM's physical compaction — merging SSTable files and reclaiming space from deleted records.

What is entirely absent from Qdrant (and from any AI memory system) is the semantic compaction layer:
- No detection of semantically near-duplicate embeddings across segments
- No supersession detection (newer version of same fact invalidates older version)
- No TTL category differentiation (session vs. project vs. permanent)
- No score-based compaction trigger (the Merge Optimizer fires on segment count, not on semantic redundancy)

The ideas in this finding implement the semantic compaction layer on top of Qdrant's existing storage compaction. They are not in conflict — Qdrant's optimizers handle storage-level efficiency; Recall's semantic compaction handles content-level efficiency.
