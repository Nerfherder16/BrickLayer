**Question**: Does the C+ chunking pipeline latency hold inside Claude Code's actual hook execution chain?
**Agent**: quantitative-analyst
**Question Type**: PHYSICS
**Verdict**: PROMISING
**Severity**: Medium
**Source field**: Benchmark — Node.js process spawn overhead on Tim's casaclaude (Windows 11, N=50 spawn iterations); analysis of the live observe-edit.js hook source at C:/Users/trg16/Dev/Recall/hooks/observe-edit.js

## Summary

The 2ms claim from Q266 applies to the **C+ chunking algorithm only** (Q267 confirmed: p95=0.611ms). The actual full hook path takes **~220ms p95**, dominated by Node.js process spawn overhead (~117ms) and a mandatory `setTimeout(100ms)` sleep in the current hook architecture. Adding C+ chunking adds 0.611ms — a **0.3% increase** over the existing baseline. The C+ pipeline adds negligible overhead to the real hook path. The hook latency finding is: the current observe-edit.js hook already waits ~220ms per Write/Edit event; Strategy D does not change this.

---

## Hook Execution Model

The current `observe-edit.js` is a **per-invocation Node.js subprocess**:
- Claude Code spawns a new Node.js process for every PostToolUse hook invocation
- Hook reads tool context from stdin, processes it, fires async fetch to Recall API, then exits
- No long-lived process: every Write/Edit spawns a fresh Node.js cold start

### Source Analysis (observe-edit.js lines 128-146)

Key pattern in the hook:

```
fetch(RECALL_HOST/observe/file-change, { signal: AbortSignal.timeout(5000) })
  .catch(err => debug(...));  // fire-and-forget — not awaited

setTimeout(() => process.exit(0), 100);  // hardcoded 100ms sleep
```

Critical findings from source:
1. Recall API call is fire-and-forget — not awaited
2. `setTimeout(100ms)` is hardcoded — hook always runs at least 100ms
3. The 100ms sleep allows HTTP request dispatch before process exit (without it, the event loop terminates and the fetch never sends)
4. Recall API store (embedding + HNSW write, ~500ms) does NOT add to the hook sync path

## Full Hook Latency Benchmark

| Stage | Latency p95 | Notes |
|-------|-------------|-------|
| Node.js process spawn | 117ms | Measured N=50 on Windows 11; includes OS process creation + V8 init |
| Stdin read | ~2ms | Small JSON payload |
| **C+ chunking (Q267)** | **0.611ms** | Boundary detection + chunk split + entity regex |
| HTTP fetch dispatch | ~5ms | TCP connect + HTTP send to LAN (192.168.50.19) |
| setTimeout(100ms) floor | 100ms | Mandatory; concurrent with fetch I/O |
| **Total hook p95** | **~220ms** | Spawn + stdin + chunking + sleep |

**C+ overhead as fraction of total: 0.611ms / 220ms = 0.28% — negligible.**

## Implications for Strategy D

The 2ms budget from Q266 was the chunking algorithm time. The actual hook invocation is ~220ms — current baseline, unchanged by Strategy D.

| Metric | Value |
|--------|-------|
| Current hook baseline | ~220ms p95 |
| C+ chunking additional overhead | +0.611ms |
| Total with Strategy D | ~221ms |

This is acceptable. PostToolUse hooks fire after the tool result is shown; Claude can begin its next response while the hook runs.

### Optimization: reduce setTimeout floor

The 100ms sleep is for LAN HTTP dispatch safety margin. With LAN TCP taking <10ms, a 25ms sleep provides sufficient margin:
- Current: ~220ms total
- With setTimeout(25ms): ~145ms total (34% reduction)
- One-line change in observe-edit.js

## Verdict

**PROMISING.** C+ chunking adds 0.611ms to an existing ~220ms hook invocation — negligible. The 2ms budget from Q266 is confirmed for the algorithm. Total hook overhead is ~220ms, dominated by Windows Node.js spawn (~117ms) and the setTimeout(100ms) sleep floor. Strategy D does not worsen this. The immediately actionable optimization is reducing setTimeout from 100ms to 25ms (save 75ms per invocation). Both this optimization and a persistent daemon architecture are independent of the Strategy D extraction design choice.
