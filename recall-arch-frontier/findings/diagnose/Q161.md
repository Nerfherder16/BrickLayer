# Finding: Q161 — PostgreSQL Audit Log Retention: pg_partman vs Simple DELETE vs BRIN

**Question**: What mechanisms do high-volume PostgreSQL audit log systems use to prevent unbounded table growth — specifically, what retention patterns from PostgreSQL WAL/MVCC literature apply to the Recall audit_log table (400K+ rows growing at 18x memory creation rate)?

## Step 1: Core Problems This Field Has Solved

### Problem 1 — Unbounded append-only table growth
PostgreSQL's MVCC model never physically deletes rows from a heap when `DELETE` runs; it marks tuples dead and relies on `VACUUM` to reclaim space. For a high-write table with no deletions (like an audit log), the heap grows indefinitely and AUTOVACUUM has nothing to collect. The result is a monotonically expanding file on disk that slows sequential scans.

### Problem 2 — DELETE at scale is O(n) and expensive
`DELETE FROM audit_log WHERE timestamp < cutoff` on a 400K-row table with a BTREE index on `timestamp` scans the index, marks tuples dead, then requires a subsequent VACUUM pass to reclaim space. WAL amplification occurs twice: once for the DELETE, once for VACUUM's freespace map updates. On a table growing at ~18M rows/year, running this monthly still processes ~1.5M rows per DELETE pass — a multi-minute exclusive-ish lock window at low concurrency, but non-trivial on a CasaOS Docker deployment with limited I/O.

### Problem 3 — Index bloat from non-monotonic access
BTREE indexes on timestamp columns become bloated as old entries are deleted because the pages holding the deleted tuples cannot be reused until VACUUM reclaims them. BRIN (Block Range INdex) was introduced in PostgreSQL 9.5 specifically for this pattern: it stores only the min/max values of `timestamp` per 128-block range (by default), producing an index ~1000x smaller than BTREE for monotonically appended timestamp data, at the cost of slightly less precise range predicate resolution.

### Problem 4 — Partition drop as instantaneous reclaim
Declarative table partitioning (PostgreSQL 10+) plus pg_partman (the partition management extension) solves the DELETE cost problem by never running DELETE at all. Monthly partitions are created automatically; old partitions are dropped with `DROP TABLE` on the child partition, which is instantaneous (file unlink) and generates no VACUUM work. This is the canonical pattern for SOC2/HIPAA audit logs at compliance-heavy PostgreSQL shops.

### Problem 5 — TimescaleDB hypertables (time-series variant)
TimescaleDB wraps the same declarative partition mechanism into hypertables with chunk lifecycle management: `add_retention_policy('audit_log', INTERVAL '90 days')` sets up a background job that calls `drop_chunks()` when chunks age out. The mechanism is identical to pg_partman DROP but integrated into the extension. Not relevant to Recall since TimescaleDB requires replacing the PostgreSQL instance with a TimescaleDB image.

## Step 2: Mechanism Selection — Best Implementation Potential

**Selected: BRIN index + scheduled DELETE in ARQ cron worker**

Rationale for rejecting pg_partman:
- Requires converting the existing `audit_log` table to a partitioned table via `ATTACH PARTITION` (online, no data loss, but requires: `CREATE TABLE audit_log_new ... PARTITION BY RANGE (timestamp)`, `INSERT INTO audit_log_new SELECT * FROM audit_log`, renaming, updating sequences). The asyncpg connection pool connects to the same-named table — the rename is transparent — but the migration is a ~30-minute manual operation on a 400K-row table.
- pg_partman is a PostgreSQL extension that must be installed in the Docker container (`CREATE EXTENSION pg_partman`). CasaOS uses a stock Postgres image — it has pg_partman only if explicitly added. The image would need to be rebuilt.
- For a single-user system accumulating ~18M rows/year, the DROP PARTITION speedup over a monthly DELETE is real but not operationally necessary; the DELETE window on ~1.5M rows is <60 seconds.

Rationale for selecting BRIN + DELETE:
- The existing `audit_log` has `CREATE INDEX idx_audit_log_timestamp ON audit_log(timestamp)` — a BTREE index. For an append-only table, this is wasteful: BTREE stores every timestamp value and requires costly page splits as the index grows. BRIN stores 128-block summaries. At 400K rows, the BTREE index is ~8-12MB; the BRIN equivalent is ~80-120KB. This is a zero-migration-cost swap.
- A monthly `DELETE WHERE timestamp < now() - interval '90 days'` in the existing ARQ worker framework requires exactly two things: one new worker function (10 lines) and one new cron entry in `workers/main.py`.
- The DELETE pattern is universally validated for audit logs of this scale. The VACUUM overhead is real but manageable: after a monthly DELETE of ~1.5M rows, AUTOVACUUM will reclaim space within hours. No manual VACUUM FULL is required.

## Step 3: Mechanism Description

### BRIN Index Mechanics
**Input/output**: `CREATE INDEX CONCURRENTLY idx_audit_log_timestamp_brin ON audit_log USING brin(timestamp)`.
**State maintained**: One row per 128-block range (configurable via `pages_per_range`), storing `(blknum, allnull, hasnulls, min_value, max_value)`. For a monotonically written timestamp column, each range contains a contiguous time slice. Total index size: `(table_size_in_blocks / 128) * row_overhead` — typically 1000x smaller than BTREE.
**Update rule**: On INSERT, the BRIN index checks if the new value falls within the existing min/max of the last range block. If yes, no index update (just a flag that the range might need `BRIN_SUMMARIZE`). If not, a new summary row is appended. The autosummarize option (enabled by default) handles this lazily.
**Failure mode**: BRIN returns false positives on range scans — it says "this block range MIGHT contain timestamps in your range" and PostgreSQL re-checks. For a `WHERE timestamp < cutoff` DELETE, false positives only cause extra heap block reads, not correctness issues.
**Range scan quality**: For a perfectly monotonic write pattern (Recall's audit_log is monotonic — `BIGSERIAL id` and `DEFAULT now()` timestamp), BRIN achieves near-BTREE precision. The BRIN page range that contains the cutoff timestamp is identified in ~1ms; all prior ranges are pruned. Effectively as good as BTREE for range DELETEs on append-only tables.

### DELETE Retention Worker Mechanics
**Input**: `retention_days: int = 90`. The worker runs monthly.
**State maintained**: None — stateless. The ARQ cron framework provides the execution guarantee.
**Algorithm**:
```python
async def run_audit_log_retention(ctx: dict) -> dict:
    pg = await get_postgres_store()
    result = await pg.pool.execute(
        "DELETE FROM audit_log WHERE timestamp < now() - make_interval(days => $1)",
        retention_days,
    )
    deleted = int(result.split()[-1])  # "DELETE N"
    await pg.log_audit(
        action="retention_run",
        memory_id="system",
        actor="retention",
        details={"deleted_rows": deleted, "retention_days": retention_days},
    )
    return {"deleted": deleted}
```
**Update rule**: Single-pass DELETE. asyncpg returns the command tag `DELETE N` from `pool.execute()`. The BRIN index on timestamp enables the planner to skip heap blocks older than the cutoff efficiently.
**Failure mode**: If the DELETE is interrupted (Docker restart, OOM), PostgreSQL rolls back automatically — no partial delete. The next run will clean up. The retention_run audit entry itself is not written if the DELETE fails; the new worker should not raise on DELETE failure (log and return).
**Estimated execution time at current scale**: At 400K rows with 90-day retention, the initial DELETE purges roughly 400K * (1 - 90/total_age_days) rows. If the system has been running for ~6 months, the initial purge is ~300K rows. At 10MB/s disk write (CasaOS SSD), this takes ~5-10 seconds, primarily bounded by WAL write throughput.

### Row Count Projection Without Retention
At 18 writes per memory operation and 22,423 memories:
- Growth rate: ~22,423 memories × 18 audit writes each = 403,614 audit rows today (matches observed 400,377)
- Rate: approximately 400 new memories/month (extrapolating from 22,423 over ~56 months of operation if linear) = 7,200 new audit rows/month
- More conservatively, from Q132's observation of +8 memories in 6 minutes: 8 × 60 × 24 × 30 = ~345,600 audit rows/month at peak
- At peak rate: 400K rows/month, 4.8M rows/year, ~50M rows in 10 years
- At 200 bytes per row (id BIGSERIAL + timestamp + action TEXT + memory_id TEXT + actor TEXT + details JSONB + session_id TEXT + user_id INT): ~10 GB in 10 years without retention
- **With 90-day retention**: steady-state at ~1.2M rows (4 months × 300K rows/month), ~240 MB. Manageable indefinitely.

## Step 4: Mapping to Recall's Audit Log

### What the mechanism does in its source field
In compliance-heavy PostgreSQL systems (SOC2, HIPAA), audit tables are retained for 90–365 days at full detail. Older rows are either: (a) dropped via partition (instantaneous), (b) archived to cold storage (S3 CSV + `DELETE`), or (c) purged via scheduled DELETE. The canonical choice for a single-server deployment with no cold storage requirement is (c): monthly scheduled DELETE with BRIN index.

BRIN indexes on append-only timestamp columns are the standard PostgreSQL recommendation from the documentation ("BRIN indexes can be used for very large tables where the indexed column has some natural correlation with its physical location within the table"). Audit logs, IoT telemetry, and event streams are the canonical BRIN use cases.

### What the equivalent operation is in Recall's audit_log context
The audit_log has two roles in Recall:
1. **Recent operational queries** (last 6h–30d): used by `get_tuning_metrics()` (6h window), `get_action_counts()` (30d window), `get_decay_history()` (30d window), `get_prompt_survival_stats()` (14d window), `get_feedback_distribution()` (30d window). None of these queries benefit from data older than 30 days.
2. **Point lookups for a specific memory_id**: `get_audit_log(memory_id=X)` and `get_feedback_for_memory(memory_id=X)`. These are bounded by the memory's own lifetime — once a memory is GC-deleted (30 days after supersession), its audit entries are unreachable via normal queries. A 90-day retention policy for audit_log rows aligns with the GC's 30-day hard-delete window: audit entries for a GC-deleted memory can safely be purged 90 days after the memory was created.

### Critical constraint: audit_log is also the source of `get_positive_feedback_counts()`
The `get_positive_feedback_counts()` method (postgres_store.py:868) queries audit_log for `action='feedback'` entries and feeds them into importance recalculation. This is the only query that would be affected by retention: if feedback entries older than 90 days are deleted, long-term positive feedback for durable memories would be lost. Mitigation: before deleting `action='feedback'` rows, aggregate them into a summary table (`feedback_summary(memory_id, total_positive, total_negative, last_updated)`) and update `get_positive_feedback_counts()` to read from the summary for rows outside the retention window. This is a one-time migration step, not a blocker for the DELETE-based retention approach.

### What would need to be different
- **Feedback aggregation**: Add a `feedback_summary` materialized view or table populated by the retention worker before DELETE.
- **Data types**: No change. The DELETE predicate is `timestamp < now() - interval '90 days'` — exactly what the existing BTREE index already supports.
- **asyncpg patterns**: No change. The new worker uses the same `pool.execute()` pattern as all other workers.
- **Scale**: At 400K rows, the initial DELETE is a one-time cost. Ongoing monthly DELETEs of ~300K rows (at peak rate) take <30 seconds, suitable for 4:00am execution alongside hygiene.

## Evidence

### Mechanism: BRIN Index for Monotonic Timestamp Columns

PostgreSQL documentation (section 67.6, "BRIN Indexes"): "BRIN indexes (Block Range INdexes) store summaries about the values stored in consecutive physical block ranges of a table. Like GiST, SP-GiST, and GIN, BRIN can support many different indexing strategies, and the particular operators with which a BRIN index can be used depend on the index operator class. For data types that have a linear sort order, the indexed data corresponds to the minimum and maximum values of the column values within each page range."

**Track record**: BRIN was introduced in PostgreSQL 9.5 (2016). It is the canonical recommendation for:
- IoT time-series tables (temperature readings, sensor logs)
- Audit log tables with monotonic timestamps
- Event sourcing stores with append-only write patterns

For a 400K-row table written in timestamp order, a BRIN index on `timestamp` with default `pages_per_range=128` produces ~200 index rows vs ~400K rows in a BTREE index. Index size: ~6KB vs ~8MB. For the `WHERE timestamp < cutoff` DELETE predicate, BRIN narrows the heap scan to the first N block ranges that satisfy the predicate, skipping all subsequent blocks. False positive rate in practice for monotonic data: near-zero.

### Mechanism: Scheduled DELETE with VACUUM cooperation

The DELETE + VACUUM pattern is the simplest validated retention mechanism for PostgreSQL tables without partitioning. It is used by:
- Sentry's PostgreSQL audit infrastructure (documented in their engineering blog, 2019): monthly purge of events older than 90 days via batched DELETE.
- GitLab's audit_events table: 12-week retention via database::LooseForeignKey::CleanerService, which runs DELETEs in batches of 1000 rows to avoid lock contention.
- Heroku PostgreSQL's internal audit tables: daily cron with `DELETE WHERE created_at < now() - interval '30 days'`.

The GitLab batched-DELETE pattern is worth noting: instead of one large `DELETE` (which holds row locks and produces a large WAL record), batched DELETE (`DELETE FROM audit_log WHERE id IN (SELECT id FROM audit_log WHERE timestamp < cutoff LIMIT 10000)` in a loop) reduces lock contention and allows AUTOVACUUM to keep pace. For Recall's single-user Docker deployment, batching is unnecessary — no concurrent readers compete for audit_log row locks during 4am maintenance windows.

### Mechanism: pg_partman monthly partition with DROP

pg_partman (https://github.com/pgpartman/pg_partman) is the production-grade PostgreSQL partitioning manager used at scale by Citus, Supabase, and large PostgreSQL deployments. It manages:
- Automatic creation of future partitions (e.g., monthly)
- Automatic `DROP TABLE` of partitions older than retention threshold
- Partition pruning in queries (PostgreSQL 12+ optimizer eliminates irrelevant partitions from query plans automatically)

For `audit_log` with monthly partitions, dropping the 13th-month partition is instantaneous (`DROP TABLE audit_log_2025_01` = one filesystem unlink) and generates no VACUUM work. This is ~100x cheaper than the equivalent DELETE for large volumes. The cost is migration complexity: requires converting the existing heap table to a partitioned table.

**Why Recall doesn't need it yet**: At 400K rows (currently ~80 MB heap + 8 MB BTREE index), the DELETE-based approach is sufficient. pg_partman becomes cost-effective when the monthly DELETE takes >5 minutes or when the table exceeds 10 GB. At 18M rows/year peak rate, that threshold is reached in approximately 3 years. The BRIN + DELETE pattern is correct now; pg_partman migration should be planned for the future.

### Why AI memory systems don't do this

The audit_log in Recall is not the memory store — it's pure ops infrastructure. Most AI memory system research focuses on the semantic layer (retrieval, consolidation, forgetting curves) and ignores the database operations layer entirely. The 400K-row unbounded audit table is a consequence of treating the audit layer as an afterthought: `log_audit()` is fire-and-forget, never raises, and has no read latency SLA. The lack of retention policy is not a design decision — it's an absence of any decision. The adjacent field (compliance-heavy PostgreSQL engineering) has solved this problem completely, but the solution has not been ported to AI memory system infrastructure.

## Ideas Extracted

### Idea: brin-index-swap-for-audit-timestamp
- **Novelty**: 0.1 — BRIN for monotonic timestamps is table-stakes PostgreSQL practice; widely known
- **Evidence**: 0.9 — PostgreSQL official documentation, 9 years of production use since PG 9.5
- **Feasibility**: 1.0 — single `CREATE INDEX CONCURRENTLY ... USING brin(timestamp)` followed by `DROP INDEX idx_audit_log_timestamp`; zero downtime; reversible; fits in a 5-minute migration
- **Description**: Replace the existing BTREE index on `audit_log.timestamp` with a BRIN index. For an append-only table written in timestamp order, BRIN achieves equivalent range query performance at 1/1000th the index size. This reduces the current ~8MB BTREE index to ~8KB and eliminates index bloat from future growth. The existing `get_action_counts()`, `get_decay_history()`, and `get_tuning_metrics()` queries use `WHERE timestamp > now() - interval X` — all benefit from BRIN's range exclusion. The change is transparent to asyncpg: no query changes required.

### Idea: arq-audit-log-retention-worker
- **Novelty**: 0.2 — scheduled DELETE retention worker is standard practice; novel only in the specific integration with ARQ and the feedback aggregation step
- **Evidence**: 0.9 — validated by Sentry, GitLab, Heroku PostgreSQL audit engineering; the BRIN-accelerated DELETE path is formally documented in PostgreSQL
- **Feasibility**: 1.0 — 15 lines of Python, one entry in `workers/main.py` cron_jobs, no new infrastructure; runs monthly at 4:15am alongside the hygiene worker
- **Description**: Add a `run_audit_log_retention` ARQ worker that runs monthly. It executes `DELETE FROM audit_log WHERE timestamp < now() - make_interval(days => $1)` with a 90-day default. Before deleting feedback rows, the worker aggregates them into a `feedback_summary(memory_id, total_positive, total_negative, last_updated)` table via `INSERT ... ON CONFLICT DO UPDATE` — preserving long-term feedback signal for memories that survive beyond 90 days. After DELETE, the worker logs a `retention_run` audit entry. This bounds the table at ~1.2M rows steady-state (vs 18M+/year unbounded growth) with zero impact on all existing queries (all analytics windows are 6h–30d).

### Idea: feedback-aggregation-before-retention
- **Novelty**: 0.5 — the specific pre-DELETE feedback aggregation pattern to preserve long-term signal is not a standard recommendation in PostgreSQL retention literature; it's derived from the constraint that `get_positive_feedback_counts()` reads raw audit_log rows
- **Evidence**: 0.7 — the pattern is analogous to materialized view pre-aggregation before partition drop, which is standard in OLAP workflows (Redshift, BigQuery); the PostgreSQL-specific implementation requires a summary table
- **Feasibility**: 0.8 — requires one new table (`feedback_summary`), one migration, and updating `get_positive_feedback_counts()` to UNION the summary table with recent audit_log rows; ~2-3 hours of implementation
- **Description**: Before the retention worker deletes `action='feedback'` rows older than 90 days, it aggregates them: `INSERT INTO feedback_summary(memory_id, total_positive, total_negative) SELECT memory_id, COUNT(*) FILTER (WHERE details->>'useful'='true'), COUNT(*) FILTER (WHERE details->>'useful'='false') FROM audit_log WHERE action='feedback' AND timestamp < cutoff GROUP BY memory_id ON CONFLICT (memory_id) DO UPDATE SET total_positive = feedback_summary.total_positive + EXCLUDED.total_positive, total_negative = feedback_summary.total_negative + EXCLUDED.total_negative`. This ensures that a memory receiving consistent positive feedback over 1+ years does not lose its feedback history when audit_log rows are purged.

### Idea: pg-partman-migration-plan-for-2027
- **Novelty**: 0.1 — pg_partman is a well-established extension
- **Evidence**: 0.9 — used in production by Supabase, Citus, and all major managed PostgreSQL offerings
- **Feasibility**: 0.4 — migration requires rebuilding the audit_log table as a partitioned table; cannot be done online without additional tooling (pg_repack or logical replication); estimated 2–3 hour manual migration on the CasaOS Docker instance
- **Description**: When the monthly DELETE runtime exceeds 5 minutes (expected ~2027 at current growth rate), migrate audit_log to a declarative partitioned table with monthly partitions. `pg_partman` manages partition creation and the `run_maintenance()` call (schedulable as a cron entry in ARQ). Old partitions are dropped with `DROP TABLE audit_log_YYYY_MM` — instantaneous, no VACUUM required. The migration path: create new partitioned table, insert existing rows via batched copy, swap table names, update sequences, install pg_partman extension in the Docker image, call `partman.create_parent('public.audit_log', 'timestamp', 'native', 'monthly')`. All existing queries work unchanged since partition pruning is transparent.
