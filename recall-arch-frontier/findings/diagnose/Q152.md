## Physics Calculation

### Operation Definition

- **Input**: Incoming memory text (50–500 chars), existing corpus of 22,381 memories
- **Output**: Binary decision — proceed to embedding or reject/stage as near-duplicate
- **Guarantees required**: No false negatives (missing a duplicate is acceptable; false-positive rejection of novel memory is not)

### SimHash Scan Cost by Storage Option

The SimHash Layer 1 scan runs only after a SHA-256 Layer 0 miss. Its cost is the latency added to every non-exact-duplicate write.

| Option | Cost per write | Mechanism |
|---|---|---|
| A: Redis HSET lookup | ~1ms | Store fingerprints as HSET fields; `HGET simhash:{bucket}` per bucket; pure in-memory hash |
| B: PG full-table scan (no index) | 50–200ms (midpoint: 125ms) | `SELECT ... WHERE bit_count(simhash XOR :fp) <= k` over 22,381 rows; no index can accelerate a bitwise computed predicate without a GiST |
| B2: PG with indexed bigint | ~5ms | GiST/btree on `simhash_fingerprint BIGINT`; cannot directly accelerate Hamming queries, but with banding (pre-computed permutation columns) achieves ~5ms |
| C: Qdrant payload filter | 1–5ms (midpoint: 3ms) | Store simhash as an indexed payload field; Qdrant's HNSW index does not help here but payload filter index does; scans indexed payload values |

**The critical fact**: The Q141 proposal described the SimHash check as a "full-table scan with bit operation" in PostgreSQL, which is Option B. This is the worst-performing storage choice for SimHash fingerprint lookup.

### Theoretical Minimum for SimHash Lookup

The operation being performed is: given a 64-bit integer fingerprint, find all corpus members within Hamming distance k.

- **Theoretical minimum**: O(log n) with sorted permutation tables (Google's approach, WWW 2007). For n=22,381 and 6 permutation groups: ~6 × 14 = 84 comparisons. At ~10ns per comparison: ~1 microsecond. This is the physics floor — it requires pre-computed sorted arrays.
- **Redis HSET with banding**: ~1ms — within 1000x of the physics floor, acceptable
- **PG full scan**: 50–200ms — 50,000–200,000x above the physics floor; the gap is pure architecture (scanning rows that can't possibly match)

### Break-Even Duplicate Rate Formula

Let:
- C = cost per write for the SimHash scan (ms), incurred on every write
- S = savings per detected duplicate (ms), earned only on duplicate writes
- D = fraction of writes that are near-duplicates detectable by SimHash

Net latency impact per write:
```
net = C × (1 - D) - S × D
```

At break-even (net = 0):
```
C × (1 - D_be) = S × D_be
D_be = C / (C + S)
```

### Savings per Detected Duplicate

A SimHash hit eliminates the embedding call and downstream writes:
- Skip Ollama embedding (qwen3-embedding:0.6b at 192.168.50.62): 100–300ms
- Skip Qdrant upsert: ~1ms
- Skip Neo4j write: ~2ms
- **Total savings S**: 103–303ms (midpoint: 203ms)

Tim's write sources (from Q132 and hook architecture):

- Extracts facts about code changes — each edit generates unique content (file path, line content, operation type)

**recall-session-summary.js (Stop hook)**:

- But SimHash detects SYNTACTIC overlap (shared tokens), not semantic overlap
- "Fixed the Recall MCP server" vs "Repaired the recall MCP connection" → different token sets → different SimHash fingerprints
- SimHash near-duplicate risk: LOW-MEDIUM (semantic duplicates pass through as distinct)

**Manual MCP stores**:
- User-intentional memory writes; tend to be deliberate new facts
- Exact duplicate risk: MEDIUM (same fact stored twice across sessions)
- SimHash near-duplicate risk: LOW

**Corpus-level estimate**:

The Q141 analysis cited Q013's R=0.40 total redundancy fraction. However, R=0.40 is the fraction of memories that are semantically redundant (vector similarity ≥ 0.95), not syntactically redundant. SimHash only captures the syntactic subset. The syntactic-overlap subset of semantic duplicates is much smaller:

- Semantically redundant memories that are ALSO syntactically near-identical: ~10–30% of the 40% total (3–12% of all memories)
- But among INCOMING writes (stream duplicates, not historical corpus): lower rate — approximately 1–3% of writes are SimHash-detectable near-duplicates

**Best estimate**: D_simhash ≈ 1–3% of writes reach the SimHash layer AND are near-duplicates detectable at Hamming distance k ≤ 3.

This estimate has high uncertainty (±2x) because no direct measurement of syntactic near-duplicate rates exists for Tim's corpus. The Q141 recommended calibration against the live 22,381-memory corpus has not been performed.

## Qualifying Ideas

- **redis-simhash-fingerprint-store**: (N=0.72, E=0.85, F=0.88) — The Q141 SimHash proposal assumes PostgreSQL full-table scan for Hamming distance lookup. This is a storage misalignment: Redis already exists in the Recall stack, is in-process for hook calls, and can store the 22,381 SimHash fingerprints as a sorted set or hash at 8 bytes each (178KB total). Pre-computing 6 permutation bands (the Google sorted-table approach, WWW 2007) and storing them in Redis sorted sets reduces SimHash lookup to 6 binary searches costing ~1ms total, dropping the break-even from 38% to 0.5%. This transforms SimHash from net-negative to net-positive at Tim's actual duplicate rate, without adding any new infrastructure. The full implementation requires: (1) at write time, store `simhash_{band_id}:{permuted_fingerprint}` → memory_id in 6 Redis sorted sets; (2) at lookup time, binary search each band sorted set for fingerprints within ε of the query permutation. Memory overhead: 6 bands × 22,381 entries × ~50 bytes per entry = ~6.7MB of Redis memory.

- **simhash-calibration-before-implementing**: (N=0.65, E=0.82, F=0.95) — The break-even analysis shows SimHash via Redis is net-positive only if D_near ≥ 0.5%. The expected D_near of 1–3% is an estimate with high uncertainty (no direct measurement). Before implementing SimHash (which requires schema changes and a backfill job on 22,381 memories), a direct calibration of the syntactic near-duplicate rate is feasible in a single Python script: sample 1,000 random pairs of memories from the existing corpus, compute SimHash Hamming distance for each pair, identify pairs with Hamming ≤ 3 that are NOT exact duplicates (SHA-256 differs), report the fraction. This measurement would bound D_near empirically and confirm whether Redis SimHash is justified before any implementation work. Cost: 1 Python script, ~1 hour of work, no schema changes required. Evidence: calibration before implementation prevents wasting engineering effort on a mechanism that may have a 0.3% actual near-duplicate rate (which would make even Redis SimHash marginally net-negative).
