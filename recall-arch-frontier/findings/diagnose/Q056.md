# Finding: Q056 — Partition Behavior of the Redis INCR Write Path and the Correct Fallback

**Question**: What are the failure modes of the Redis INCR HLC replacement (Q048) under a network partition between casaclaude and the CasaOS Redis instance — and does a partition-tolerant fallback exist that preserves LWW correctness without introducing causal ordering violations?
**Agent**: quantitative-analyst
**Question Type**: ADVERSARIAL
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Distributed systems — partition tolerance, CAP theorem (Brewer 2000; Gilbert and Lynch 2002 SIGACT), write-ahead logging and crash recovery (PostgreSQL WAL, SQLite WAL-mode), retry semantics in distributed systems

## Adversarial Analysis: Partition Failure Modes of Redis INCR and the Correct Fallback

---

### Thread A: Defense — Redis INCR is correct and the right partition behavior is to fail writes cleanly

**Core claim**: Redis INCR is co-located with the API server on CasaOS, so Redis unavailability is identical to API unavailability — both fail at the same network boundary — making partition-tolerance of the counter a non-problem. The correct behavior when casaclaude loses connectivity to CasaOS is for writes to fail, and the Stop hook should buffer the session summary locally for retry. No local counter fallback is needed.

**Evidence**:

Q048 explicitly resolved this: "If CasaOS is unreachable, casaclaude cannot write — this is acceptable by design." The reasoning is sound: Recall's write path is `casaclaude hook → FastAPI (CasaOS) → Redis (CasaOS) → PostgreSQL (CasaOS) → Qdrant (CasaOS) → Ollama (192.168.50.62)`. Every component except Ollama is on CasaOS. If the network between casaclaude and CasaOS fails, the FastAPI endpoint is unreachable before Redis is ever contacted. The partition cannot selectively kill Redis while leaving FastAPI alive — they share a network address.

This is the fundamental property of co-location: it collapses the failure domain. In the CAP theorem framing (Brewer 2000, Gilbert and Lynch 2002), Recall has chosen Consistency over Availability for the write path, and that choice is coherent given the personal, single-API-server architecture. There is no ambiguity about which partition side "wins" — the API server is the single writer, so whichever side can reach it can write. There is no split-brain.

The Stop hook already buffers session summaries locally (the `recall-session-summary.js` hook fires on session end). If CasaOS is unreachable, the summary cannot be stored immediately. A simple retry-on-reconnect mechanism — the hook writes the summary to a local queue file and retries on the next session start — handles this without any changes to the Redis counter or the write path. This pattern is identical to how SQLite WAL mode handles write failures: the write is not lost, it is deferred.

The Dynamo paper (DeCandia et al., SOSP 2007) distinguishes between systems that require "always writable" (shopping carts, IoT sensors) and systems that can accept "write when connected." Recall is unambiguously the latter. A personal memory system where writes occasionally fail during network outages is correct behavior. The alternative — local writes during partition — creates a more dangerous failure: silent causal violations that are hard to detect and harder to repair.

**Best case scenario**: casaclaude loses connectivity to CasaOS for 30 minutes during a session. The Stop hook attempts to store the session summary, fails with a connection error, writes the summary to `~/.claude/recall-pending/session-{timestamp}.json`, and logs "Recall write deferred: CasaOS unreachable." On the next session start, the retrieve hook checks for pending writes and retries them before fetching memories. The user sees no interruption in session continuity — the memory appears on the next session, one step later than normal. No data is lost, no causal ordering violations occur, and no complex fallback protocol is needed.

---

### Thread B: Attack — Redis INCR creates a write bottleneck that fails in ways the defender has not examined, and the local-buffer fallback problem is solvable

**Core claim**: The defender's "co-location means single failure domain" argument is correct for full CasaOS outages but ignores the more common failure mode — intermittent latency spikes, partial connectivity, and session-end timing — where a local buffer fallback would preserve writes that the current design silently drops. Furthermore, a local counter fallback does NOT necessarily create sequence number conflicts with the global Redis counter if the counter namespace is partitioned by machine ID.

**Evidence**:

The hidden failure mode is not full partition — it is the timing gap between observe-edit.js fires (during the session, while CasaOS may be reachable) and recall-session-summary.js fires (at session end, potentially after a reachability change). In Recall's hook architecture:

- `observe-edit.js` fires during active editing — typically when casaclaude is actively connected to CasaOS
- `recall-session-summary.js` fires at session end — but sessions can end abruptly (Claude Code crash, machine suspend, network transition)

The session summary is the highest-value write (it contains the synthesized understanding of the entire session). If the Stop hook fires during a 30-second window when CasaOS is unreachable — WiFi handoff, momentary router hiccup, CasaOS Docker restart — the summary is lost unless there is a local buffer. This is not a hypothetical: the project-brief.md notes "The system must degrade gracefully when any single backing service is down." A silently dropped session summary does not meet "degrade gracefully."

The defender's proposed retry mechanism (writing to `~/.claude/recall-pending/`) is itself a local buffer. The attacker's position is not that a local buffer is wrong — it is that the defender's proposed buffer implementation requires thought about sequencing.

Now, the critical question: does a local buffer create sequence number conflicts?

The defender assumes the answer is yes. The attacker argues this is wrong. Here is the mechanism for a conflict-free local buffer using machine-partitioned sequence numbers:

Instead of a single global counter `recall:write_seq` that increments for all writes from all machines, use machine-local counters:
- `recall:write_seq:casaclaude` — incremented only by casaclaude
- `recall:write_seq:proxyclaude` — incremented only by proxyclaude

Each memory record stores `(machine_id, local_seq)` as a vector pair. This is exactly the vector clock structure from Q038, but used as a sequence number rather than a conflict detection mechanism. During partition, casaclaude cannot increment `recall:write_seq:casaclaude` on Redis — but it CAN maintain a local sequence counter in a file or SQLite database. When reconnection occurs, casaclaude replays its locally buffered writes to the API with locally-assigned sequence numbers. There is NO conflict with proxyclaude's sequence numbers because the namespaces are disjoint. There is NO conflict with the Redis counter because casaclaude's replayed writes get fresh Redis-assigned sequence numbers on submission — the local sequence numbers are only used for ordering within the buffer, not as the authoritative global sequence.

The key insight the defender misses: the local buffer does not need to use the global Redis sequence number. It needs to use a locally-monotonic ordering for the queued writes. The global sequence number is assigned by the API at submission time — which happens when connectivity is restored. The local buffer just needs to preserve write order, not global causal position.

Analog: Apache Kafka's producer exactly-once semantics (KIP-98, 2017) uses a producer-local sequence number per partition — separate from the globally-ordered Kafka offset — to detect and deduplicate retried writes. The producer assigns local sequence numbers, and Kafka assigns authoritative offsets at commit time. Recall can use the same pattern.

The failure mode of the defender's approach: casaclaude is in a session summarizing three hours of work on a critical project. The Stop hook fires. CasaOS happens to be in a 45-second Docker restart cycle. The session summary is lost. The user has no way to know, because the hook fires silently and exits. Three hours of memory formation — including architecture decisions, code patterns learned, error resolutions — disappears. The hook logs an error, but without a retry mechanism, the data is gone. The defender says this is "acceptable by design" — but the project-brief.md says the system must degrade gracefully, and silent data loss is not graceful degradation.

**Failure mode**: High-value session summaries silently dropped during the 30-60 second window when CasaOS is unavailable at session end. No user notification, no retry, no recovery. The loss compounds over time — sessions that weren't summarized create gaps in the memory graph that affect retrieval quality for weeks.

### Synthesis

**What Thread A assumes that Thread B exposes**:

Thread A assumes that "Redis unavailability = API unavailability = write failure is visible and expected." This is true for the steady-state full-partition case but false for the timing-gap failure case. The Stop hook fires at a moment the writer (casaclaude) does not control — session end may coincide with a 30-second CasaOS unavailability window. The assumption that "the user expects writes to fail during outages" ignores that session summaries are the most important writes in the system, and they fire exactly once, at the moment of maximum vulnerability (session end), with no retry by default.

More precisely: Thread A assumes the cost of write failure is uniform across write types. It is not. A failed `observe-edit.js` write loses one fact extraction — low cost, because the session summary will capture it anyway. A failed `recall-session-summary.js` write loses the synthesized understanding of the entire session — high cost, because nothing else captures it. The partition behavior is different for different write types.

**When Thread A is right**:

- For `observe-edit.js` writes: write failure during partition is acceptable. These writes are high-frequency, low-value-per-write, and the session summary will capture the important content anyway. The "fail cleanly" behavior is correct.
- For full CasaOS outages (minutes to hours): a local buffer cannot meaningfully help because the user is unlikely to be in an active session. Write failure is correct.
- For the counter-conflict question specifically: Thread A is completely right that a shared Redis INCR counter does not create split-brain conflicts when Redis is the single sequencing authority, because there is no alternative authority generating conflicting sequence numbers. The defender correctly identifies that the conflict concern only arises if a local counter is used.

**When Thread B is right**:

- For `recall-session-summary.js` writes: the session summary should be locally buffered with retry, because it is a once-per-session write that fires at session end — the most vulnerable moment. Write failure at this point is not "acceptable" in any meaningful sense of graceful degradation.
- For the 30-60 second intermittent unavailability case: this is the realistic failure mode, not full partition. A retry mechanism with a local pending queue is the correct engineering response.
- For the counter-conflict analysis: Thread B correctly identifies that a local buffer does NOT require a local counter to conflict with the global Redis counter, as long as the local buffer uses locally-ordered sequencing and the global sequence is assigned at submission time.

**The architecture Thread B would accept**:

The correct partition behavior is asymmetric by write type:

**Tier 1 — `observe-edit.js` (fact extraction writes)**: Fail cleanly. No local buffer needed. The session summary will capture the important content. Concretely: the hook makes one attempt, logs failure, exits. No retry.

**Tier 2 — `recall-session-summary.js` (session summary writes)**: Local buffer with retry. The hook writes the session summary to a local pending file at `~/.claude/recall-pending/session-{timestamp}.json` if the API is unreachable. On the next session start, the `recall-retrieve.js` hook checks for pending writes and submits them before fetching memories. The pending file uses a locally-monotonic sequence within the session (irrelevant for ordering purposes since each session summary is independent). The API assigns the authoritative `write_seq` at submission time. No counter conflicts. No causal violations.

The key architectural rule: **global sequence numbers are assigned at write submission to the API, not at write creation on the hook**. Local buffers use local-only ordering (a timestamp or a local file sequence) purely to preserve write order within the buffer. The global causal order is established by the API when the buffered write is replayed.

This architecture satisfies both threads:
- Thread A's concern (no counter conflicts) is preserved because the global Redis counter is only incremented at API submission, never locally.
- Thread B's concern (session summaries not lost) is addressed by the local pending queue.
- The CAP theorem tension is resolved by accepting unavailability for low-value writes and buffering for high-value writes — a tiered availability design.

The analogy from distributed systems: this is identical to the client-side offline write queue in Firebase (Google, 2016). Firebase clients buffer writes locally when offline, using local monotonic ordering within the queue. When the connection is restored, Firebase replays the queue, and the server assigns authoritative document timestamps. The local ordering is discarded after replay. Firebase explicitly documents that the local ordering is provisional and the server ordering is authoritative.

## Ideas Extracted

### Idea: write-tier-partition-policy
- **Novelty**: 0.78
- **Evidence**: 0.72
- **Feasibility**: 0.90
- **Description**: Recall's partition behavior is differentiated by write tier. Tier 1 writes (observe-edit.js, fact extractions) fail cleanly during partition — they are low-value-per-write and high-frequency, and their content is captured by the session summary. Tier 2 writes (recall-session-summary.js, session summaries) are locally buffered at `~/.claude/recall-pending/` with automatic retry on next session start. This asymmetric policy matches the value distribution of Recall's write types: session summaries are once-per-session, high-value, irreplaceable; fact extractions are many-per-session, individually low-value, recapturable. The same asymmetry is used by Firebase Realtime Database, which distinguishes between "critical offline writes" (buffered to IndexedDB) and "streaming updates" (dropped on disconnect). Implementation cost: 30 lines of JavaScript in the Stop hook, 10 lines of retry logic in the retrieve hook, zero changes to the API or Redis.

### Idea: api-assigned-global-sequence-for-buffered-replays
- **Novelty**: 0.72
- **Evidence**: 0.80
- **Feasibility**: 0.88
- **Description**: Local buffers in Recall's pending queue use locally-monotonic ordering (file timestamps or an incrementing local counter in the pending file) only to preserve intra-buffer write order. The authoritative global sequence number (`write_seq` from Redis INCR) is assigned by the FastAPI server at the moment the buffered write is replayed to the API. This means the local buffer never generates sequence numbers that could conflict with the global Redis counter. The pattern is identical to Apache Kafka's KIP-98 producer sequence numbers: the producer assigns local sequence numbers per partition (for exactly-once detection), but Kafka assigns the authoritative offsets at commit. The local numbers are internal to the producer and never become the authoritative ordering. Applying this to Recall: the hook's pending queue can use `created_at` timestamps for local ordering without touching the global write_seq counter. No split-brain. No causal violations.

### Idea: session-summary-local-pending-queue
- **Novelty**: 0.68
- **Evidence**: 0.75
- **Feasibility**: 0.92
- **Description**: The `recall-session-summary.js` Stop hook gains a two-phase write: (1) attempt to POST to the Recall API; (2) on failure (network error or HTTP 5xx), write the session summary JSON to `~/.claude/recall-pending/session-{ISO-timestamp}.json` and exit with log message "Session summary deferred: stored locally for retry." The `recall-retrieve.js` UserPromptSubmit hook gains a pre-flight check: before querying memories, check if `~/.claude/recall-pending/` contains any files; if yes, attempt to submit each one to the API; on success, delete the file; on failure, leave the file for the next attempt. This implements durable write semantics for session summaries with no changes to the API, no changes to Redis, and no counter conflicts. The pending queue is a simple directory of JSON files — no database, no schema, no migration. The retry is best-effort: if CasaOS is still unreachable at the next session start, the pending file stays. Eventually it is submitted. The user never loses a session summary to a transient outage.

### Idea: partition-exposes-write-value-heterogeneity
- **Novelty**: 0.80
- **Evidence**: 0.65
- **Feasibility**: 0.95
- **Description**: The adversarial analysis reveals a design gap: Recall's write path treats all writes as equivalent in value and urgency, but they are not. Session summaries are irreplaceable once-per-session high-value writes. Fact extractions are replaceable many-per-session low-value writes. This value heterogeneity is invisible until partition stress-tests the system. The correct response is not to make all writes partition-tolerant (expensive, complex, overkill for fact extractions) or to make all writes fail on partition (acceptable for fact extractions, unacceptable for session summaries). The response is to assign each write type an explicit durability tier and implement the appropriate partition behavior for each tier. This is a design principle with general applicability to AI memory systems: as write types proliferate (fact extractions, session summaries, profile updates, graph edges), each type should declare its durability requirement, and the write path should route accordingly. No reviewed AI memory system (mem0, Zep, MemGPT, LangMem) implements tiered write durability. They treat all writes equivalently.

## Key Finding

The Redis INCR HLC replacement (Q048) has one genuine vulnerability under partition: session summary writes may be lost when CasaOS is transiently unreachable at session end. This is not a flaw in the Redis INCR counter design itself — the counter conflict concern raised by the Attacker is resolved by the principle that global sequence numbers are assigned at API submission time, not at buffer creation time. Local pending queues do not generate global sequence numbers and therefore cannot conflict with the Redis counter.

The correct partition behavior is asymmetric by write type:
- `observe-edit.js` writes: fail cleanly (no local buffer, no retry)
- `recall-session-summary.js` writes: buffer locally with retry on next session start

The implementation is 40 lines of JavaScript across two hook files, zero API changes, zero schema changes, zero new infrastructure. It directly satisfies the project-brief.md requirement that "the system must degrade gracefully when any single backing service is down."

The deeper finding: partition stress-testing reveals that Recall's write path implicitly assumes write-value homogeneity. It does not. This assumption becomes load-bearing under failure, and failure exposes it. The fix (tiered durability) is simple; the insight (write types have different durability requirements) is the transferable result.

## Suggested Follow-ups

- **[IMPLEMENTATION] Q057**: What is the exact implementation of the session-summary local pending queue in `recall-session-summary.js`? Specifically: what JSON schema should the pending file use, how should the retry logic in `recall-retrieve.js` handle partial submission failures (the API accepts the summary but the hook crashes before deleting the pending file, causing a duplicate submission on the next session), and should the pending queue have a maximum age after which stale pending files are discarded rather than submitted?

- **[PHYSICS] Q058**: How often does the session-summary-loss failure mode actually occur in Recall's homelab? Estimate: if casaclaude runs 3 sessions/day and CasaOS has 99% uptime (3.65 days/year downtime), and session end is uniformly distributed across 24 hours, the probability that a session end coincides with a CasaOS unavailability window of 30 seconds is approximately: `P = (downtime_seconds / year) × (session_end_rate / second)`. With 3.65 days = 315,360 seconds of downtime and 3 session ends/day = 0.0000347 session ends/second, `P ≈ 315,360 × 0.0000347 ≈ 11 lost session summaries per year`. At 99.9% uptime (8.76 hours downtime), P ≈ 1 lost summary per year. Whether this warrants the pending queue implementation is an engineering judgment call, not a pure correctness question.

- **[ADVERSARIAL] Q059**: Does the local pending queue for session summaries create a new failure mode — stale deferred summaries submitted out of order? If casaclaude is offline for 3 days, accumulates 9 pending session summaries, and then reconnects, those 9 summaries are submitted in a burst. Their `write_seq` values will be assigned sequentially at submission time, but their `created_at` timestamps will reflect the original session end times (3 days ago). Does this create any anomalies in the timeline query or the session-relative rehydrate query? Specifically: are there queries that use `write_seq` for ordering (causal) rather than `created_at` (temporal)? If temporal and causal ordering diverge for deferred writes, which ordering is correct for Recall's retrieval semantics?

- **[CONVERGENCE] Q060**: Q048 concluded that Redis INCR is a viable simplification over HLC for the conflict resolution role. Q056 concludes that the partition behavior of Redis INCR is acceptable with a local pending queue for session summaries. These two findings together define a complete write path design.
