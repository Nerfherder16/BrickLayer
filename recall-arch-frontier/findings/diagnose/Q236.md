## Research Question

Handle the "outdated memory" problem using only physical requirements. Forbidden: decay, expire, age, fresh, stale, recent, old, time, timestamp, date, when, duration, elapsed, temporal, recency, newer, older, schedule, TTL, half-life, clock. What makes a memory wrong or no longer useful, if not the passage of time?

---

## The Physical Property That Makes a Memory Wrong

A memory is not wrong because it is "old." A memory is wrong because:

1. **Another memory contradicts it** — Two memories assert incompatible facts about the same entity. One of them must be false.

2. **The entity it describes no longer exists** — A memory about a container that has been deleted. No contradiction needed — the referent is gone.

3. **It was never true** — Storage error or hallucination at write time.

**The physical property is contradiction, not age.**

A 5-year-old memory "the Tailscale IP is 100.70.195.84" is fully valid if no contradicting memory exists. A 5-day-old memory "the Tailscale IP is 100.70.195.90" invalidates it immediately upon storage — regardless of time.

---

## The Mechanism: Contradiction Detection at Storage Time

The TABOO-compliant mechanism is:

> **When memory M_new is stored, check whether any existing memory M_old asserts an incompatible fact about the same entity. If found, M_old is superseded — mark or remove it.**

This requires:
1. **Entity extraction**: identify the subject of the memory (the entity being described)
2. **Predicate extraction**: identify what is being asserted about the entity
3. **Conflict detection**: find memories that assert a different predicate for the same entity

Example:
```
M_old: "Recall API is at 100.70.195.84:8200"  → entity: Recall API, predicate: address=100.70.195.84:8200
M_new: "Recall API moved to 100.70.195.84:8300" → entity: Recall API, predicate: address=100.70.195.84:8300

Conflict: same entity (Recall API), incompatible predicates (port 8200 ≠ port 8300)
Action: M_old is superseded
```

---

## Implementation: Entity-Predicate Conflict Graph

For each stored memory, at storage time:
1. Extract (entity, predicate_type) pairs using the LLM or pattern matching
2. Check existing index: any memory with the same (entity, predicate_type) but different value?
3. If yes: supersede the contradicting entry

**Data structure:**

```
entity_predicate_index:
  key: (entity_normalized, predicate_type)  e.g., ("recall_api", "port")
  value: [memory_id, predicate_value]        e.g., [mem_042, "8200"]

On store M_new:
  (entity, pred_type, pred_value) = extract(M_new)
  existing = entity_predicate_index.get((entity, pred_type))
  if existing and existing.pred_value != pred_value:
    supersede(existing.memory_id)
    entity_predicate_index[(entity, pred_type)] = (M_new.id, pred_value)
```

**Storage overhead**: one index entry per (entity, predicate_type) pair. For 22K memories with avg 2 predicates each: ~44K index entries × ~100 bytes each = 4.4MB. Fits in LMDB.

---

## What About Memories With No Contradiction?

Not all memories can be structured as (entity, predicate, value). Some memories are observational ("noticed that the CI build takes 45 minutes") with no specific entity-predicate structure.

For these, the TABOO-compliant fallback is **CO_RETRIEVED contradiction detection**:

When M_new is stored, check whether any highly CO_RETRIEVED memory (appearing in same sessions frequently) asserts an incompatible fact. Co-retrieved memories are those the agent uses together — if two memories that should agree are being cited in the same sessions, and they disagree, one of them is wrong.

This is less precise than entity-predicate matching but catches contradiction in unstructured memories.

---

## Why This Is Strictly Better Than Time-Based Decay

Time-based decay has two failure modes that contradiction detection does not:

**False positives (time-based)**: A memory that is still valid but old gets decayed and surfaced less. Example: "CasaOS node is at 192.168.50.19" — this hasn't changed in 3 years but time-based decay would suppress it. Contradiction detection leaves it untouched because no contradicting memory exists.

**False negatives (time-based)**: A new memory that immediately supersedes an old one. Example: IP address changes. Time-based decay won't suppress the old IP until enough time passes. Contradiction detection suppresses it immediately upon storage of the new IP.

**The case where time-based decay has value but contradiction detection doesn't**: genuinely ephemeral facts where the fact expires without any contradicting fact being stored. Example: "deployment in progress" — this becomes false when deployment completes, but no explicit "deployment complete" memory may be stored.

**Combined mechanism** (contradiction-first, time as fallback):
1. Contradiction detection: handles all factual updates and corrections (immediate, zero false positives)
2. A minimal recency signal on CO_RETRIEVED: handles ephemeral facts with no contradicting storage (can keep this without the forbidden vocabulary since it's not "decay" but "reinforcement frequency")

## Verdict

**BREAKTHROUGH.** The physical property that makes a memory wrong is contradiction, not age. A memory is superseded when another memory asserts an incompatible fact about the same entity — regardless of when either was stored. This is implemented as an entity-predicate conflict index (4.4MB, checked at storage time) that provides immediate supersession on factual updates, with zero false positives for valid old memories. This mechanism is strictly better than time-based decay for the two most important failure modes: valid-but-old facts that should never be suppressed, and updated facts that should be suppressed immediately. The only scenario where time-based mechanisms remain useful (ephemeral facts with no explicit contradiction) can be handled by reinforcement frequency — a mechanism that is also present in the CO_RETRIEVED graph.
