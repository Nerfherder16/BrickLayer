# Finding: Q031 — IDF Retrieval-Frequency Penalty vs. Implicit Citation Feedback: Cross-Purposes or Coherent Composition?

**Question**: Does the retrieval-frequency IDF penalty (Q013) work at cross-purposes with the implicit citation feedback loop (Q009), or do they compose into a coherent importance signal?
**Agent**: quantitative-analyst
**Question Type**: ADVERSARIAL
**Verdict**: BREAKTHROUGH
**Severity**: Critical

---

## Signals Under Analysis

| Signal | Source | Formula / Mechanism | Acts On |
|--------|--------|---------------------|---------|
| IDF penalty | Q013 | `adjusted_score = cosine × importance × log(N / retrieval_count)` | Retrieval ranking (per-query score modifier) |
| Citation feedback | Q009 | `importance += 0.1 × (1 - importance)` on cited memories; `importance *= 0.98` on uncited | Stored importance field (persistent, cross-session) |

Both signals update or read from the same `importance` field in PostgreSQL. The conflict arises when a memory is ubiquitous (high retrieval_count) AND consistently cited (high importance from citation feedback). The IDF term pushes its retrieval rank down; the citation term pushes its stored importance up. At steady state, one signal neutralizes the other unless they are allowed to act on separate quantities.

---

## Adversarial Analysis

---

### Thread A: Defense — The IDF penalty is correct to dominate

**Core claim**: Retrieval frequency is a proxy for omnipresence in context, not for utility. A memory that appears in 95% of queries has stopped being a discriminative retrieval signal — it is part of the invariant background. Citation feedback is inflated by injection frequency (you can only cite what was injected), so high citation rates on ubiquitous memories are a tautological artifact, not evidence of genuine value. The IDF penalty correctly suppresses these memories so session-specific signals can surface.

**Evidence**:

1. **The injection-bias problem in citation feedback**: Q009's citation detection checks whether injected memory content appears in the response. If a memory is injected in 95% of sessions and the LLM writes responses that naturally include the fact (e.g., "on CasaOS at 192.168.50.19"), the citation rate will approach the injection rate. This is not the LLM finding the memory useful — it is the LLM regurgitating injected context. Citation feedback cannot distinguish "this memory caused the LLM to say something it would not have otherwise said" from "this memory was injected and appeared in the response because the response described the same topic the memory describes." The IDF penalty exists precisely to break this tautological reinforcement loop.

2. **IDF's information-theoretic grounding**: The IDF term `log(N / retrieval_count)` is not heuristic — it is the inverse of the probability that a randomly selected session retrieves this memory. A memory with retrieval_count = N has zero discrimination power: it tells you nothing about the current session that differs from every other session. Shannon's mutual information framing: `I(memory; session) = 0` when `P(memory | session) = P(memory)`. The IDF penalty is the correct operationalization of this: zero discriminative value should produce zero contribution to adjusted score.

3. **Recommender systems analogy (co-occurrence bias)**: In implicit feedback collaborative filtering, interaction counts from popular items are well-known to bias models toward ubiquitous items (the "popularity bias" problem). The standard correction in RecSys is to downweight item-user interactions by item popularity — precisely the IDF structure. Papers from Steck (2011), Abdollahpouri et al. (2019), and Boratto et al. (2021) on popularity debiasing all confirm that naive implicit feedback loops amplify popular items at the expense of niche but genuinely relevant items. Citation feedback without IDF correction will produce the same pathology in Recall.

4. **The correct treatment of truly omnipresent facts**: A memory that is genuinely needed in every session is not a retrieval problem — it is a context problem. It should move to the always-injected hot tier (Q003's LLPC analog), not remain in the retrieval pool. Once it is promoted to the hot tier (automatically injected without querying), it no longer affects retrieval ranking at all. The IDF penalty is the mechanism that signals "this memory belongs in hot tier, not in the retrieval pool" — its adjusted score converges to 0 and the hot-tier promotion logic should trigger.

**Best case scenario**: IDF penalty cleanly separates the retrieval pool into session-specific discriminative memories (high IDF score, injected when relevant) and invariant background facts (low IDF score, promoted to hot tier or dropped from retrieval). Citation feedback acts within each tier — boosting importance among the session-specific memories that are genuinely used, without polluting the IDF signal with tautological injection counts.

---

### Thread B: Attack — Citation feedback is more reliable; IDF mistakes value for ubiquity

**Core claim**: The IDF penalty conflates two distinct causes of high retrieval frequency: (1) a memory that is retrieved because it is semantically close to many queries (could be valuable or redundant), and (2) a memory that is retrieved because it contains universally relevant facts (is definitionally valuable). Citation feedback distinguishes between these cases because it measures whether the LLM actually used the injected memory to produce its response — a much stronger utility signal than mere retrieval frequency. The IDF penalty is a blunt instrument that suppresses the second case wrongly.

**Evidence**:

1. **The causal direction problem in IDF**: `log(N / retrieval_count)` penalizes memories that are retrieved frequently. But high retrieval frequency has two distinct causes: (a) the memory is semantically close to many query vectors (proximity-based retrieval artifact — this is what IDF is trying to penalize), and (b) the memory contains information that is genuinely relevant to many tasks (value-based retrieval — this is NOT what IDF should penalize). The IDF formula cannot distinguish these cases. A fact like "CasaOS is at 192.168.50.19" is retrieved in 95% of infrastructure-related sessions because it IS relevant to 95% of infrastructure-related sessions — not because it is noise. Suppressing it from retrieval leaves every infrastructure query missing the primary connection fact.

2. **Citation feedback provides causal evidence that IDF cannot**: When the LLM generates a response and includes the CasaOS IP address, and the citation check detects this, the evidence is: the injected memory contained information the LLM chose to use. This is a weak but real causal signal — the LLM did not have this fact in its base knowledge (local IP addresses are not in training data) and used it from context injection. This is categorically different from the LLM citing a retrieved memory about a fact it already knew from training (where citation rate would be high but injection-causal). For Recall's use case (personal homelab infrastructure, project-specific facts), the LLM's base knowledge is sparse and injection-causality is high. IDF's frequency penalty discards this.

3. **Adjacent field failure: TF-IDF degradation in specialized corpora**: In information retrieval, TF-IDF is known to perform poorly on specialized, narrow corpora where "common" terms are domain-specific jargon that carries high discriminative value within the domain. In a legal corpus, "habeas corpus" appears in many documents — IDF would penalize it — but it is a highly discriminative term for a legal practitioner. The exact same failure mode applies to Recall: in a homelab corpus, "CasaOS" and "192.168.50.19" appear in many memories but they are highly discriminative for a homelab user who uses multiple machines and IP addresses. IDF's assumption of "frequent = low-value" breaks down in personalized narrow-domain corpora.

4. **The IDF signal is confounded by query distribution, not just memory value**: If the user asks 90% infrastructure-related questions, infrastructure memories will have high retrieval counts regardless of their importance. The IDF penalty would suppress these memories specifically because the user uses them — punishing the user's actual workflow. A user who asks exclusively about CasaOS would have every CasaOS memory suppressed by IDF. The penalty is a function of the user's query distribution, not the memory's intrinsic value. Citation feedback does not have this confound: a memory that is cited in 90% of injections is cited because it is useful to 90% of tasks, which IS evidence of value.

5. **The separation-of-concerns failure**: Q013's IDF formula applies the penalty at retrieval time as a score modifier. Q009's citation feedback updates the stored importance score persistently. These operate on different timescales and different quantities — but they interact through the importance field. After many sessions: citation feedback drives importance toward 1.0 for frequently-cited memories; IDF's log(N/retrieval_count) term shrinks toward 0 as retrieval_count grows. Adjusted_score = cosine × importance × log(N/retrieval_count). As importance → 1.0 and log(N/retrieval_count) → 0, adjusted_score → 0 regardless of how valuable the memory is. The two signals do not compose coherently — they cancel each other into silence for precisely the memories that are most consistently useful.

**Failure mode**: The system converges to a state where the most-used memories have high importance (from citation feedback) but near-zero adjusted scores (from IDF penalty), effectively disappearing from retrieval. They are not in the hot tier (no hot-tier promotion logic exists yet in Recall). They are not retrieved by semantic query (IDF suppresses them). They are invisible. This is the worst possible outcome: the system has identified its most valuable memories and then hidden them.

### Synthesis

**What Thread A assumes that Thread B exposes**:

Thread A assumes that retrieval frequency is a reliable proxy for "injected regardless of query relevance" — i.e., that high-frequency memories are being pulled by semantic proximity inflation rather than genuine relevance. Thread B exposes this assumption as false: in a narrow-domain personal corpus (homelab infrastructure), high retrieval frequency CAN be high-frequency genuine relevance. The hidden assumption is that the memory store is a broad corpus with many topics, where Zipf's law ensures that frequent items are generically common rather than domain-specifically valuable. In a personal assistant corpus, the "document collection" is not a broad corpus — it is a narrow-domain, user-specific fact set where every memory is already domain-filtered. The IDF assumption from text retrieval does not transfer cleanly.

Thread A also assumes the hot tier exists and will catch the memories that IDF suppresses. It does not exist in Recall yet. The proposal to "promote to hot tier when IDF converges to zero" is a sound architectural idea but is NOT implemented. In the absence of a hot tier, IDF suppression is simply erasure.

**When Thread A is right**:

1. When the memory store contains high redundancy (R = 0.40–0.50 as measured in Q013): multiple paraphrase-duplicates of the same fact will all have high retrieval counts, and IDF correctly suppresses all but the best one. Here IDF is doing deduplication work.

2. When retrieval frequency is driven by query-space density rather than memory value: a memory about a topic the user asks about constantly (e.g., "how to use git") may have high retrieval counts because queries are dense in that area, not because the memory is irreplaceable. IDF correctly penalizes generically-applicable memories whose content is available from LLM base knowledge.

3. When the hot tier exists and has a promotion trigger that fires before IDF has fully suppressed the memory. In that architecture, IDF acts as a "detector" (this memory is always relevant, promote it) rather than a "suppressor" (hide this memory). Thread A's defense is only coherent under this condition.

4. After semantic deduplication has been applied (Q013's prerequisite): with redundancy eliminated, the surviving high-frequency memories are genuinely unique cross-session facts, and IDF correctly signals "always-relevant" which should trigger hot-tier promotion.

**When Thread B is right**:

1. When the hot tier does not exist: without a promotion path, IDF suppression is permanent erasure of the system's best memories.

2. For Recall's current narrow-domain personal corpus: infrastructure IPs, project-specific constants, and homelab topology facts are high-frequency because they ARE genuinely relevant to most tasks the user performs — not because they are redundant noise. Citation feedback correctly identifies these as high-value.

3. When the store has been deduplicated: post-deduplication, the remaining high-frequency memories are by definition unique facts (no paraphrase copies survived). Their high frequency is signal, not noise. IDF's penalty is then actively harmful.

4. When the user's query distribution is domain-concentrated: a user who exclusively asks about CasaOS, Qdrant, and Docker will have all their most important memories suppressed by IDF because those topics saturate the query space.

**The architecture Thread B would accept**:

The conflict is resolvable because the two signals are measuring orthogonal properties that have been incorrectly mapped to the same quantity:

- **Citation feedback** measures: utility (was this memory causally involved in producing a good response?)
- **IDF penalty** measures: discrimination (does this memory distinguish the current session from other sessions?)

These are genuinely different properties. A memory can be high-utility AND low-discrimination (the universal facts). A memory can be low-utility AND high-discrimination (rarely-injected memories about niche topics that turn out to be unhelpful). The error is requiring them to compose through a single adjusted_score formula.

**The correct composition rule is a two-axis scoring model**:

```
axis_1: importance (pure citation feedback signal — measures utility)
axis_2: discrimination (IDF signal — measures session-specificity)
```

Memory lifecycle by quadrant:

| Importance | Discrimination | Action |
|---|---|---|
| High | High | Standard retrieval — these are session-specific, genuinely useful memories. Surface them. |
| High | Low | Promote to hot tier — these are universal facts that should be always-injected, not retrieved. Remove from retrieval pool. |
| Low | High | Candidate for decay — retrieved often but never cited. These are semantic false positives. Decay importance. |
| Low | Low | Short-lived tier — rarely retrieved, rarely useful. Expire after 30 days. |

The formula becomes a routing rule, not a combined score:

```
if importance > UTIL_THRESHOLD and discrimination < DISC_THRESHOLD:
    promote_to_hot_tier(memory)        # High importance, low IDF = always-inject
elif importance > UTIL_THRESHOLD and discrimination >= DISC_THRESHOLD:
    standard_retrieval_pool(memory)    # High importance, discriminative = normal retrieval
elif importance <= UTIL_THRESHOLD and discrimination >= DISC_THRESHOLD:
    decay_importance(memory, rate=0.02)  # Retrieved but unused = weaken
else:
    schedule_expiry(memory, days=30)   # Low both = allow to die
```

This resolves the cancellation problem: citation feedback drives importance up for genuinely useful memories, IDF discrimination drives routing decisions (hot tier vs. retrieval pool) rather than fighting importance for control of adjusted_score. The IDF term never suppresses a high-importance memory — it routes it to a different tier where it is always available.

**Implementation delta from current system**:

1. Separate `importance` (citation utility signal) from `discrimination_score` (IDF signal). Store both in PostgreSQL.
2. Update `discrimination_score = log(N / retrieval_count)` after each retrieval. This is a pure metadata update with no Qdrant changes.
3. Add hot-tier promotion logic: if `importance > 0.75 and discrimination_score < 0.5`, mark the memory as `tier=hot` and inject it automatically in every session. Remove from ANN retrieval pool.
4. Retrieval ranking formula: `adjusted_score = cosine × importance` (no IDF in the formula — IDF is used for routing, not ranking within the retrieval pool).
5. The hot tier is naturally size-bounded by the context window: if more than K memories are in hot tier, promote only the top-K by importance and demote the rest back to the retrieval pool. This implements Q003's LLPC niche competition at the hot-tier level.

**The key insight neither thread saw alone**:

The IDF penalty should not be a multiplicative term in the retrieval ranking formula. It should be a threshold detector that fires a tier-promotion action. When IDF detects ubiquity (discrimination_score converges below a threshold), the correct response is promotion, not suppression. The signal means "this memory has graduated from session-specific retrieval to universal context injection" — not "hide this memory because it is noise." Mixing the routing decision into the scoring formula is the architectural error that creates the cross-purposes problem.

## Ideas Extracted

### Idea: two-axis-importance-discrimination-model
- **Novelty**: 0.78 — Separating utility (citation feedback) from discrimination (IDF) into independent axes with quadrant-based routing is not implemented in any known AI memory system. Information retrieval systems use TF-IDF as a combined scoring function; recommender systems use implicit feedback for a single relevance score. The deliberate separation into two axes with different downstream actions (routing vs. ranking) is novel in this domain.
- **Evidence**: 0.62 — The orthogonality claim (utility ≠ discrimination) is supported by the IDF failure-on-narrow-corpora literature and by the recommender systems popularity-debiasing literature (Abdollahpouri et al. 2019). The specific implementation as a routing model rather than a combined score is a design proposal without direct empirical validation; evidence score reflects the well-validated underpinnings of each individual axis.
- **Feasibility**: 0.80 — Requires adding a `discrimination_score` field to PostgreSQL (trivial), updating it on each retrieval event (already happening implicitly via retrieval_count), and adding hot-tier promotion logic (1–2 weeks). Does not require changing Qdrant schema or the embedding model. The hot-tier injection can use the existing recall-retrieve.js hook with a priority-fetch prefix. Total implementation: 2–3 weeks.
- **Description**: Add a second axis to Recall's memory scoring: `discrimination_score = log(N / retrieval_count)`, stored in PostgreSQL and updated on each retrieval. The `importance` field continues to be driven exclusively by citation feedback (Q009). Retrieval ranking uses `cosine × importance` (no IDF multiplicative term). Hot-tier promotion is triggered when `importance > 0.75 AND discrimination_score < 0.5`: these memories are auto-injected into every session and removed from the ANN retrieval pool. The hot tier is capped at K=10 memories (context window budget); when the cap is reached, the lowest-importance hot-tier memory is demoted back to the retrieval pool. Result: citation feedback and IDF no longer fight over adjusted_score. IDF acts as a detector and router; citation feedback acts as a quality ranker within each tier.

### Idea: hot-tier-context-window-budget
- **Novelty**: 0.65 — Auto-injection of a small fixed set of always-relevant memories (the "hot tier") is implicit in Q003's LLPC analog and in Q013's IDF proposal, but has not been explicitly designed with a context-window-budget cap and a competitive displacement mechanism. The combination of (injection cap) + (importance-ranked eviction when cap exceeded) is the key design that makes the hot tier stable rather than growing unboundedly.
- **Evidence**: 0.70 — Q003's plasma cell niche competition (Radbruch et al. 2006) provides the biological validation that a bounded competitive pool with quality-based displacement is stable and self-optimizing. The engineering translation is direct.
- **Feasibility**: 0.85 — Trivially implementable as a PostgreSQL query: SELECT top-K memories WHERE tier=hot ORDER BY importance DESC LIMIT K. The niche cap (K=10) keeps the hot tier's context window cost bounded. Update: when a new memory qualifies for hot tier and the cap is full, compare its importance to the current minimum in hot tier and displace if higher. Implementation: 1 week.
- **Description**: The hot tier is a fixed-size pool (K=8–12 memories, configurable) of always-injected facts. Each session starts by loading the hot tier from PostgreSQL and prepending it to the context window before the semantic retrieval results. Hot-tier memories are excluded from ANN search (they are already injected). The pool is bounded: when a memory qualifies for hot-tier promotion (high importance + low discrimination) and the cap is full, the lowest-importance current hot-tier member is demoted to the standard retrieval pool. This implements Q003's niche competition in a single concrete mechanism.

### Idea: idf-as-routing-signal-not-ranking-signal
- **Novelty**: 0.72 — The specific architectural decision to use IDF as a binary routing trigger (promote to hot tier) rather than as a continuous multiplicative score modifier is novel. Standard IR uses IDF in the ranking formula; this proposal removes IDF from the ranking formula entirely and uses it only as a promotion threshold detector. The conceptual reframing — from "lower the score of ubiquitous memories" to "graduate ubiquitous memories to a different tier" — has no precedent in the reviewed literature.
- **Evidence**: 0.55 — The argument is primarily structural/logical rather than empirically validated. The failure mode of multiplicative IDF in narrow-domain corpora is documented in the IR literature (domain-specific jargon suppression). The routing interpretation is a design proposal derived from that failure analysis.
- **Feasibility**: 0.90 — Removing IDF from the ranking formula is a trivial change (delete the `× log(N/retrieval_count)` term from adjusted_score). Adding a promotion trigger that checks the IDF threshold is a lightweight PostgreSQL query added to the post-retrieval hook. Implementation: < 1 week.
- **Description**: Remove `log(N / retrieval_count)` from Recall's retrieval ranking formula. Instead, run a nightly background job that queries PostgreSQL for memories where `retrieval_count / N > 0.5` (retrieved in more than half of recent sessions) AND `importance > 0.7` (consistently cited as useful) and promotes them to the hot tier. Retrieval ranking reverts to `cosine × importance` — simpler, no IDF cancellation. The IDF signal is computed and stored but used only for routing decisions, not ranking. This removes the signal conflict at its root.

## Verdict Summary

The two signals are measuring orthogonal properties (utility vs. session-discrimination) and should NOT be composed multiplicatively into a single adjusted_score. The current proposed composition creates a cancellation pathology: the system's most valuable memories accumulate high importance (from citation feedback) while simultaneously accumulating near-zero discrimination (from IDF), producing adjusted_score near zero. This is the worst-case architectural error: detecting high-value memories and then silencing them.

The correct architecture separates the signals into distinct functions:
- Citation feedback (importance) = ranking within the retrieval pool
- IDF (discrimination_score) = routing between retrieval pool and hot tier

Neither signal dominates. They operate in different phases on different quantities. The Defender is right that IDF correctly identifies ubiquitous memories as needing different treatment. The Attacker is right that the treatment should be promotion, not suppression. Both threads converge on the same conclusion from opposite directions: ubiquitous high-citation memories belong in the always-injected hot tier, not in the ANN retrieval pool.

4. **[ADVERSARIAL] Q035**: Is citation detection from response text a reliable enough signal to trust for importance updates, or is it so noisy (the LLM cites injected context tautologically) that citation feedback is net-harmful? 5. **[TIMESHIFTED]**: The two-axis model proposed here assumes citation detection is the utility signal. In 2028, utility may be better measured by outcome feedback (did the session produce a working result?) rather than by syntactic citation detection. Deploying IDF without a hot-tier promotion path causes permanent erasure of the system's most valuable memories.
