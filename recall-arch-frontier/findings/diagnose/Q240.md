## Research Question

Design mandatory memory permanence using only physical requirements. Forbidden: legal, comply, regulation, audit, retain, hold, mandate, policy, rule, enforce, immutable, lock, prevent, deletion, HIPAA, SEC, GDPR, compliance, requirement, obligation, law. What physical property guarantees a fact cannot be lost? What makes destruction physically impossible rather than policy-prohibited?

---

## The Physical Property: Replication Depth

A fact cannot be lost if **the cost of destroying it exceeds the resources of any single actor who might want to destroy it**.

This is not a policy. It is a physical constraint on information destruction: destruction requires writing zeros (or overwriting) to every copy of the fact. If the copies are distributed beyond the reach of any single actor, destruction is physically impossible — not policy-forbidden.

**The physical mechanism is replication to a depth that exceeds destruction capacity.**

---

## What Makes Information Indestructible

**Case 1: Single copy in mutable storage** (current Recall 2.0):
- Destruction cost: O(1) — delete the LMDB entry
- Physically destroyable by: anyone with write access to the LMDB file

**Case 2: Replicated to N independent nodes**:
- Destruction cost: O(N) coordinated operations on N different systems
- Physically destroyable by: any actor who controls all N nodes

**Case 3: Append-only log with external replication**:
- Writes are appended; no delete primitive exists at the storage layer
- Destruction requires physically destroying the storage medium
- Replicated across media: destruction requires destroying N physical media simultaneously

**Case 4: Content-addressed immutable storage** (IPFS / git / blockchain):
- Each entry is addressed by its own hash: CID = SHA256(content)
- "Deleting" CID x requires overwriting the hash — which changes the address
- The original content at the original address physically cannot be changed (it would produce a different hash → different address)
- Destruction: must ensure no node retains a copy at that CID

**Case 5: Cryptographic commitment**:
- Publish hash(observation) to an external timestamping service or public ledger at storage time
- The hash cannot be changed. The content can be destroyed locally, but the commitment proves it existed
- This is the minimum-cost physical permanence mechanism

---

## The Cheapest Physical Permanence Mechanism

**Cryptographic commitment to external hash ledger:**

At storage time, for each observation:
```
commitment = SHA256(observation_text + timestamp + user_id)
publish commitment to external service (Trusted Timestamp Authority, IPFS, or public blockchain)
```

Cost: negligible (one SHA256 + one API call per observation)
Storage locally: any mutable format (can be deleted)
Physical guarantee: the hash exists externally and proves the observation existed at that time

This is the physical equivalent of "notarization" — not that the fact is preserved, but that its existence at a specific point in time is permanently committed.

For genuine non-destructibility (not just proof-of-existence): replicate the actual content to IPFS (CID-addressed). The content cannot be changed or deleted without changing its address.

---

## Append-Only Log as Intermediate Mechanism

Between "single mutable copy" and "replicated to N external nodes," the intermediate mechanism is an **append-only log**:

```
LMDB is NOT append-only — it supports delete
Log structure (separate from LMDB): append-only WAL where delete is not a primitive

On store: append observation to log
On "delete": append a DELETION_MARKER to log (never remove original)
On read: play back log to current state
```

The observation is never physically removed from the log — only logically superseded by a deletion marker. Physical destruction requires destroying the log file.

With log replication (rsync to second host): physical destruction requires destroying both copies.

This is the Q226 session WAL extended to support permanence. The session WAL already uses this pattern for session state — extending it to individual observations provides the physical permanence mechanism.

---

## Is Physical Permanence Strictly Better Than Policy-Enforced Permanence?

**Where physical permanence wins**: when the actor enforcing policy might themselves want to destroy the information (compromised admin, malicious insider, ransomware). Policy says "don't delete" — physical permanence makes deletion physically impossible regardless of who is enforcing the policy.

**Where policy suffices**: when the threat model is accidental deletion or unauthorized access, not malicious insider with root access to storage.

**For Recall 2.0**: the primary threat model is accidental deletion and human error, not malicious insiders. Policy-enforced permanence (LMDB write-protected mode, regular rsync backup) is sufficient. Physical permanence (IPFS content addressing) is a Phase 7+ feature for medical/trading deployments where the threat model includes malicious insiders.

**The cost model**:
- Append-only log: zero additional cost (WAL already exists in Q226)
- IPFS CID addressing: ~100ms per observation at storage time (CID computation + pin to IPFS daemon)
- External hash commitment: negligible (SHA256 + API call)
- Full N-node replication: proportional to replication factor × storage cost

## Verdict

**BREAKTHROUGH.** The physical mechanism for mandatory memory permanence is replication depth: information is indestructible when the cost of destroying it (coordinated writes to N copies) exceeds the resources of any single actor. The cheapest implementation is the append-only log (delete = append marker, never physical removal), which is an extension of the Q226 session WAL already planned for Recall 2.0. For stronger guarantees: IPFS content-addressing (CID = SHA256(content), deletion requires changing the address) or cryptographic commitment to an external ledger at storage time. The analysis reveals that policy-enforced permanence (LMDB write-protection, rsync backup) is sufficient for Recall 2.0's current threat model (accidental deletion), while physical permanence (IPFS, N-node replication) is reserved for medical/trading deployments where the threat model includes malicious insiders.
