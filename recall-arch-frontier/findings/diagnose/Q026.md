# Finding: Q026 — Adversarial: Cosine Distance vs Scope Chain as Store Discriminant

**Question**: Is cosine-distance-vs-existing the correct discriminant for the store-or-merge decision, or should the discriminant be identifier-existence in the scope chain?
**Agent**: quantitative-analyst
**Question Type**: ADVERSARIAL
**Verdict**: BREAKTHROUGH
**Severity**: High

---

## Defender Thread: Cosine Distance Is Correct

### Steel-man for cosine distance

The core problem the store discriminant must solve is: "does this incoming text represent a fact already encoded in the memory store, or does it represent new information?" This is a semantic equivalence problem, not an identity problem. Cosine distance over dense embeddings is the right signal because:

**1. Semantic equivalence is form-independent.**

The same fact can be stored under dozens of different surface forms across different sessions. "Redis is running on the homelab server at port 6379" and "homelab Redis instance, port 6379, running" are semantically equivalent but share no common identifier. A scope-chain discriminant cannot detect this duplication — neither fact has a canonical name that appears in both. Cosine distance catches it because the embedding space maps semantic content, not surface form.

**2. Cross-session deduplication is structurally impossible without a semantic signal.**

The scope chain model is session-local. When the casaclaude session ends and a new session begins, the scope chain is rebuilt. A scope-chain-only discriminant has no mechanism to detect that "redis_host = 192.168.50.19" stored in session:abc123 is the same fact as "redis_host = 192.168.50.19" stored in session:xyz789. Cosine distance compares against the full vector store across all sessions and all scopes — it is scope-agnostic by design, which is its strength for deduplication.

**3. The formal property that makes it right: translation invariance.**

Cosine distance is invariant to surface transformations that preserve meaning (paraphrase, rephrase, reorder). For the store problem, translation invariance is exactly what you want: if you have already stored a fact, you do not want to store it again just because the incoming text uses different words. The embedding model is the compression function that reduces surface variation to a semantic point. Cosine distance then measures distance in that compressed semantic space.

**4. Named-fact aliasing is a real failure mode of identifier-based systems.**

Consider: a user refers to the same infrastructure host as "the CasaOS box", "192.168.50.19", "casaclaude server", and "homelab primary" in different sessions. A scope-chain-identifier system would create four separate bindings (different identifiers, same referent). Cosine distance would cluster all four near the same semantic point and correctly identify them as referring to the same entity. Identifier-based systems produce orphaned duplicates for every alias.

**Core claim**: Cosine distance is the correct primary discriminant because the store-or-merge decision is fundamentally about semantic equivalence, and semantic equivalence is a meaning-space question that requires an embedding-space metric.

**Evidence**: DG pattern separation (Leutgeb et al. 2007) uses an orthogonalized similarity measure, not an identity check. The biological mechanism's input is the similarity of distributed representations, not the existence of a named symbol. Dense retrieval literature (Karpukhin et al. 2020, Dense Passage Retrieval) confirms cosine distance in embedding space as the canonical measure for semantic equivalence in retrieval systems.

**Best case scenario**: A well-calibrated cosine threshold (tuned per domain, potentially dynamic per Q002's ACh model) catches 85-90% of duplicates regardless of surface form variation, supports cross-session deduplication without session-scope bookkeeping, and handles aliasing that identifier systems cannot. The casaclaude/proxyclaude contamination problem is solved by adding scope metadata as a filter *after* the cosine comparison, not by replacing it.

## Attacker Thread: Scope Chain Is Correct

### Steel-man for scope-chain identifier-existence

The cosine distance approach solves the wrong problem. It answers "is this text semantically similar to existing text?" when the actual question the store operation must answer is "is there already an authoritative binding for this fact in the current context?" These are not the same question, and conflating them produces a discriminant that is both wrong and unsafe.

**1. Cosine distance is uninvertible for negations and opposites.**

"Redis is healthy" and "Redis is broken" have high cosine similarity. The embedding model encodes them as nearby points because they share domain vocabulary (Redis, health status) and sentence structure. The cosine discriminant would classify them as near-duplicates and attempt to merge or suppress one. But they are not duplicates — they are contradictory facts about the same entity at different times. The scope chain discriminant handles this correctly: both "redis_status = healthy" and "redis_status = broken" have the same identifier (`redis_status`), and the scope chain correctly routes the inner-scope binding to shadow the outer-scope binding without merging.

**2. Cosine distance is uninvertible for facts that are structurally identical but context-separated.**

"Redis host is 192.168.50.19" (casaclaude infrastructure) and "Redis host is 192.168.50.62" (proxyclaude infrastructure) have extremely high cosine similarity — same subject, same predicate template, only the IP differs. An embedding model will place these within 0.85-0.95 cosine similarity because the sentence structures are nearly identical. The cosine discriminant with a 0.90 threshold would merge them. This is a catastrophic failure: two distinct facts about two distinct machines are collapsed into one, destroying machine-level isolation.

The scope-chain discriminant cannot make this mistake: "redis_host on casaclaude" and "redis_host on proxyclaude" are different scope-qualified identifiers. They never share a binding. Merge is structurally impossible.

**3. The formal property that makes it right: referential opacity.**

The scope chain discriminant is referentially opaque — it asks "is there a binding for this name in this context?" The answer is a boolean based on identity, not a float based on similarity. Identity-based decisions are safe for facts about named entities: either the entity exists in the current scope or it does not. Similarity-based decisions are unsafe because "similar" is not "same." Two facts can be arbitrarily similar in embedding space while referring to different real-world entities (different machines, different time periods, different causal contexts).

**4. The store decision is not about novelty — it is about assignment.**

The cosine distance framing treats the store decision as "is this new information?" (novelty detection). But for structured facts — host configurations, project states, task assignments, machine-specific metadata — the correct framing is "is there already an assignment for this variable in this scope?" This is the compiler's `define-vs-set!` distinction from Q005. The store operation is an assignment operation, and assignments are identity-based, not similarity-based. You do not ask "is this assignment semantically similar to an existing assignment?" — you ask "does a binding for this name already exist in this scope?"

**Core claim**: Scope-chain identifier-existence is the correct primary discriminant because the store operation for structured facts is an assignment operation, and assignments are resolved by name identity in scope, not by semantic similarity to existing content.

**Evidence**: Every compiler (GCC, LLVM, V8, CPython) uses identifier-existence in the scope chain as the discriminant for define-vs-set. This is not a design choice — it is the formal semantics of assignment in typed lambda calculus. The correctness of this approach is proven in every formal semantics text since Stoy (1977). The casaclaude/proxyclaude contamination failure is a direct consequence of using a similarity-based discriminant for an identity-based decision.

**Failure mode**: Cosine distance with a 0.90 threshold, applied to the casaclaude/proxyclaude problem, merges infrastructure facts that differ only in the IP address suffix. The resulting merged memory contains contradictory information and is worse than storing two separate memories. No threshold calibration can fix this — the problem is structural.

## Falsifying Cases

### Worst case against cosine distance:

**The infrastructure mirror scenario.**

Tim's homelab runs identical stacks on casaclaude (192.168.50.19) and proxyclaude. Both machines have Redis. Both sessions store the memory: "Redis is running at [IP]:6379". The two memories differ only in the IP address — cosine similarity between them is approximately 0.92-0.95 (confirmed by the structural near-identity of the sentences). The cosine discriminant at threshold 0.90 merges them into a single memory. The merged memory inherits one of the two IPs (last-write-wins, or concatenation creates a contradictory entry). Any subsequent retrieval for "where is Redis?" returns the merged fact, which is wrong for whichever machine did not "win" the merge.

This is not a contrived edge case — it is Tim's described production failure. The specific technical structure: two facts that are semantically near-identical (same infrastructure fact about same service) but referentially distinct (different machines, mutually exclusive facts) will always be within cosine merging distance. No threshold can separate them because their embedding distance reflects their semantic similarity, and their semantic similarity is genuinely high.

**Generalization**: any two facts of the form "X on system-A has value V1" and "X on system-A has value V2" where V1 and V2 are similar types (two IPs, two port numbers, two hostnames) will be within merge distance. This includes: different versions of the same software on different machines, different configurations of the same service, different status readings taken at different times.

### Worst case against scope chain:

**The unnamed duplicate scenario.**

Tim stores a discovery during a debugging session: "discovered that increasing Redis maxmemory-policy to allkeys-lru fixes the eviction problem." This memory has no canonical identifier — it is not "redis_config" or "eviction_fix", it is a freeform insight. The same discovery is made in a second session (after the first session's scope expires and the memory was not promoted to project scope): "Redis eviction issues are caused by maxmemory-policy being set to noeviction — changing to allkeys-lru resolves it." The scope-chain discriminant cannot detect that this is a duplicate because there is no named identifier to look up. The two memories are stored separately. The memory store accumulates semantically identical insights as separate records, creating noise.

**Generalization**: scope-chain identifier-existence fails completely for unstructured content — debugging insights, architectural observations, lessons learned, project-scoped discoveries that do not have canonical names. The identifier must exist and be consistent across sessions for the scope chain to detect duplicates. For all memories that exist as free-form text without a stable identifier, scope chains provide zero deduplication.

**Second failure mode**: canonical identifier drift. Tim refers to the Redis host as "casaclaude Redis", "homelab Redis", "System-Recall Redis", and "the 8200 host's Redis" in different sessions. Each is a different identifier. The scope chain discriminant creates four bindings for the same entity, all coexisting in different scopes without any merge signal. Cosine distance would cluster them correctly.

## Synthesis

### What Thread A assumes that Thread B exposes:

Thread A (cosine) assumes that the primary risk is duplicate creation from semantic paraphrase — the "same fact, different words" problem. Thread B exposes that this assumption is only valid for unstructured, freeform content. For structured facts (host configs, status readings, configuration values), the primary risk is not paraphrase duplication — it is referential confusion, where two semantically similar sentences refer to two distinct real-world entities. Cosine distance cannot distinguish "semantically similar because same fact" from "semantically similar because same sentence template applied to different referent."

Thread B (scope chain) assumes that facts have stable canonical identifiers that are consistent across sessions. Thread A exposes that this assumption fails for a large class of real memories: freeform insights, discovered behaviors, emergent properties of systems — content that is stored as natural language without a pre-assigned name.

**The hidden assumption cosine makes**: that semantic proximity implies referential identity. This is false for facts with high structural similarity but different referents (the infrastructure mirror case).

**The hidden assumption scope chain makes**: that facts have stable, session-consistent identifiers. This is false for free-form memories and for aliased entities.

### When Thread A is right:

- Content is unstructured free-form text (insights, discoveries, debugging observations)
- The deduplication problem is paraphrase variation ("Redis needs allkeys-lru" vs. "set maxmemory-policy to allkeys-lru for Redis")
- Cross-session deduplication is needed (same fact stored in different sessions)
- Entities do not have stable canonical identifiers

### When Thread B is right:

- Content is structured fact about a named, scoped entity ("redis_host on casaclaude", "project.status", "machine.redis_port")
- The scope defines the referent (different scopes = different machines = different referents)
- The risk is merging facts that are semantically similar but referentially distinct
- The facts have stable canonical identifiers that persist across sessions

### The architecture Thread B would accept:

A two-phase store discriminant that gates on scope-qualified identifier-existence first, then falls through to cosine distance for unmatched content:

**Phase 1 — Scope-qualified identifier lookup (O(1), runs first):**

Before any embedding comparison, check whether the incoming memory can be assigned a scope-qualified identifier. If the content carries structural metadata (machine, session, project) AND a recognizable topic key (the "name" of the fact), construct the scope-qualified identifier `topic_key@scope_level`. Look up this identifier in the scope chain registry (Redis hash, O(1)).

- If found at the current scope level: this is a `set!` operation — update the existing binding. No cosine comparison needed.
- If found at an outer scope level only: this is potentially a `define` operation (local override). Ask: is the new value contradicting the outer binding? If yes, create a local scope-level shadow (do not merge). If same value, skip storage (already encoded).
- If not found at any scope level: fall through to Phase 2.

**Phase 2 — Cosine distance comparison (runs only when Phase 1 misses):**

For content that has no scope-qualified identifier match (free-form text, insights, unnamed observations), run the cosine comparison against the existing vector store. Use the dynamic threshold from Q002's ACh model. This is the current behavior, now restricted to content that cannot be handled by Phase 1.

**The key rule**: Phase 1 operates on identity; Phase 2 operates on similarity. Phase 1 runs first. Phase 1's binary decision (binding exists or not) is authoritative and never overridden by Phase 2's similarity score.

**What both approaches miss — the third signal:**

Both cosine distance and scope-chain identifier-existence are static signals. They measure the current state of the store against the incoming content. Neither captures the **temporal axis**: whether an incoming fact is a more recent update to a previously stored fact, or a conflicting historical artifact that should be invalidated.

The missing signal is **causal ordering** (logical clocks or timestamps with provenance). Consider:

- Session 1 (casaclaude, Monday): "Redis is healthy, at 192.168.50.19"
- Session 2 (casaclaude, Tuesday): "Redis is DOWN, connection refused"
- Session 3 (casaclaude, Wednesday): "Redis is healthy again after restart"

The scope-chain discriminant sees three bindings for `redis_status@casaclaude` across three sessions. Without temporal ordering, it cannot determine which is the current ground truth. Cosine distance sees three semantically similar memories and conflates them.

A Lamport-clock or vector-clock approach would assign a causal order to these facts, enabling the store to mark the older facts as superseded rather than merging or shadowing them. The correct behavior is not "merge Monday's and Wednesday's facts" (cosine) and not "shadow Monday's with Wednesday's" (scope chain alone) — it is "Wednesday's fact supersedes Monday's and Tuesday's facts in the `redis_status@casaclaude` binding."

This temporal provenance signal is the third discriminant that both existing approaches are missing. It is most closely analogous to the hippocampal reconsolidation mechanism from Q002: retrieved memories that are replayed with new information are updated *with a causal stamp* indicating which version superseded which.

## Ideas Extracted

### Idea: two-phase-store-discriminant
- **Novelty**: 0.85 — No production AI memory system runs a scope-qualified identifier lookup as a Phase 1 gate before falling through to semantic similarity comparison. All current systems (mem0, Zep, MemGPT, Recall) run a single-phase cosine search. The specific two-phase architecture (identity-check first, similarity-check second) with different semantics per phase (set!/define for Phase 1, pattern-separation for Phase 2) is absent from all known systems.
- **Evidence**: 0.80 — The precedence of identifier-existence over similarity in name resolution is formally established by the environment model of computation (SICP Ch. 3, 40+ years of compiler implementation). The DG pattern separation mechanism (Q002) provides the biological analogue for the fallthrough Phase 2. The combination of both into a two-phase gate is novel engineering, not a direct implementation of either prior.
- **Feasibility**: 0.65 — Phase 1 requires: (1) a scope-qualified identifier registry in Redis (hash map: `{topic_key}@{scope}` → `{memory_id}`); (2) a topic-key extractor that attempts to assign a canonical name to incoming content (can be a simple NER pass using Ollama, or a regex over structured facts); (3) a scope-chain lookup that walks from innermost to outermost. Phase 2 is the current cosine comparison, unchanged. The main complexity is the topic-key extractor — structured facts (configs, status readings) yield to extraction readily; free-form text may not yield any key and always falls through. Implementation: 3-5 weeks. Risk: the topic-key extractor quality determines what fraction of stores hit Phase 1 vs. Phase 2; poor extraction means Phase 1 rarely fires.
- **Description**: Recall's store endpoint runs two sequential discriminant checks. First: attempt to extract a scope-qualified topic key from the incoming content (e.g., from structured metadata, or via a lightweight Ollama extraction call: "what named entity does this memory describe?"). If a key is found, look it up in the scope-qualified registry. A hit means the incoming content updates an existing binding (set! semantics: update in place, timestamp the update, do not create a new vector). A miss falls through to the second phase: embed the content and run cosine comparison against the Qdrant store. Phase 2 uses the current merge/extend/create three-way decision from Q002. This two-phase architecture gives scope-chain precision for structured facts and cosine-distance coverage for free-form content, with neither phase overriding the other.

### Idea: scope-qualified-identity-registry
- **Novelty**: 0.80 — A dedicated key-value registry that maps `{topic_key}@{scope_chain_level}` to memory IDs is not present in any AI memory system. Recall's current data model uses Qdrant vectors + PostgreSQL rows with metadata but has no identifier registry. The registry enables O(1) identity lookup as a pre-filter to semantic search.
- **Evidence**: 0.90 — Scope-qualified identifier registries are standard in compiler symbol tables (Aho et al., Dragon Book), module systems (Python's `sys.modules`, Node.js module cache), and database engines (PostgreSQL system catalog). The pattern is among the most validated in CS. The novelty is the application to AI memory, not the underlying data structure.
- **Feasibility**: 0.80 — Redis hash maps are already in Recall's stack. The registry is a Redis hash at key `recall:identifiers:{machine_id}:{project}` with fields `{topic_key}` → `{memory_uuid}`. Topic key extraction for structured facts (machine configs, project state) can be done with a simple pattern: `{subject}_{attribute}` extracted from the memory's tags or domain. For example, a memory tagged `["redis", "casaclaude", "host"]` yields topic key `redis_host@casaclaude`. The Redis HGET lookup adds < 1ms to the store path. Writing the registry entry (HSET) after a successful store adds another < 1ms. Total overhead for Phase 1 when it fires: < 2ms. Implementation: 1-2 weeks, standalone from Phase 2 changes.
- **Description**: At store time, Recall constructs a topic key from the incoming memory's tags and domain: `{primary_tag}_{secondary_tag}@{machine_id}` (e.g., `redis_host@casaclaude`, `project_status@system-recall`, `ollama_model@casaclaude`). This key is looked up in a Redis hash registry. If the key exists, the stored memory UUID is returned and the incoming content is used to update the existing memory's text field in PostgreSQL and re-embed in Qdrant — no new record is created. If the key does not exist, a new memory is created and the key is registered. This eliminates the need for cosine comparison for any content that carries structured tag metadata. The registry is scoped per machine and per project, preventing cross-machine identifier collisions (casaclaude's `redis_host` and proxyclaude's `redis_host` are different keys because the machine_id differs).

### Idea: cosine-plus-scope-veto
- **Novelty**: 0.75 — Using scope metadata as a veto over cosine-triggered merges (rather than as a replacement discriminant) is a more conservative architecture that preserves the existing cosine pipeline while adding scope-safety. The specific pattern — "cosine says merge, but scope says no, so block the merge" — is absent from production systems but requires minimal changes to implement.
- **Evidence**: 0.75 — The formal basis is the referential opacity argument: two memory objects that live at different scope levels are by definition distinct referents and must not be merged, regardless of semantic content similarity. This is a logical constraint, not a heuristic. The implementation is analogous to constraint-based optimization where hard constraints (scope isolation) take precedence over soft objectives (deduplication efficiency).
- **Feasibility**: 0.90 — This is the smallest possible change to Recall's current architecture. The existing cosine comparison in the store path already identifies near-duplicates. Add one check after the cosine match: if the candidate near-duplicate lives at a different machine_id scope than the incoming memory, block the merge and create a new memory instead. This is a single conditional added to the merge logic. Machine_id is already stored as a memory metadata field (or can be added trivially). Implementation: 2-3 days.
- **Description**: In Recall's current store path, after identifying a cosine match (similarity > 0.90), add a scope-isolation check: if the matched memory's `machine_id` differs from the incoming memory's `machine_id`, the merge is vetoed — the incoming memory is stored as a new record even though it is semantically near-identical to the matched memory. This prevents the casaclaude/proxyclaude contamination immediately, with minimal architecture change. The veto can be extended to other scope dimensions: session_id veto (session-local memories never merge with memories from other sessions), project_id veto (project-specific memories never merge with memories from other projects). This is a partial implementation of the full scope-chain model that can ship in days rather than weeks and eliminates the most severe failure modes.

### Idea: causal-supersession-timestamps
- **Novelty**: 0.88 — Tracking causal ordering of memory updates (which fact superseded which) using logical clocks or causal timestamps is not implemented in any production AI memory system. All current systems either merge on similarity or keep all versions separately. The concept of marking earlier facts as "superseded" by later facts about the same entity is absent. Closest analogue: event sourcing in databases, but not applied to AI memory.
- **Evidence**: 0.65 — Lamport clocks (Lamport 1978, "Time, Clocks, and the Ordering of Events in a Distributed System," CACM) provide the formal foundation. Vector clocks (Fidge 1988, Mattern 1988) extend this to distributed systems. Both are extensively validated in distributed systems literature. The application to AI memory store decisions is engineering extrapolation, not direct validation.
- **Feasibility**: 0.60 — Each memory in PostgreSQL gains a `causal_chain` JSON column: `{"version": N, "supersedes": [memory_uuid_1, memory_uuid_2], "superseded_by": null}`. When a new fact updates an existing binding (via Phase 1 identity match or Phase 2 cosine match), the old memory's `superseded_by` is set to the new memory's UUID. The old memory remains in the store (for historical queries) but is filtered from default retrieval (marked inactive). Retrieval queries add `WHERE superseded_by IS NULL` to return only the current version. Implementation: 2-3 weeks. Risk: the causal chain logic is complex for concurrent sessions (casaclaude and proxyclaude simultaneously updating the same fact) — requires a distributed lock or conflict resolution strategy.
- **Description**: Recall's store operation, after determining a match (via Phase 1 identity or Phase 2 cosine), creates a causal link between the incoming (new) memory and the existing (old) memory. The existing memory is marked `superseded_by = new_memory_uuid` and excluded from default retrieval. The new memory records `supersedes = [old_memory_uuid]`. A timeline query (`recall_timeline`) can traverse the causal chain to reconstruct the history of how a fact evolved. This solves the temporal axis problem that both cosine and scope-chain discriminants miss: "Redis is DOWN on Tuesday" correctly supersedes "Redis is healthy on Monday" as a causal update to the same binding, rather than being stored as a separate memory or merged with it. The supersession relation makes the memory store a directed acyclic graph of fact evolution rather than a flat collection of similarity-clustered blobs.

### Idea: content-type-router
- **Novelty**: 0.70 — Routing store decisions through different discriminants based on content type (structured fact vs. free-form insight) is a simple dispatch pattern used in type systems (static vs. dynamic dispatch) but not applied to AI memory store operations. The novelty is the application context, not the mechanism.
- **Evidence**: 0.80 — The argument that different content types require different discriminants is logically sound: structured facts have identifiers (use scope chain), free-form insights do not (use cosine). This is the same argument that justifies static vs. dynamic dispatch in compilers (Cardelli 1985, "On understanding types, data abstraction, and polymorphism"). The content-type distinction as a routing criterion is validated in database design (schema-on-read vs. schema-on-write) and API design (typed vs. untyped endpoints).
- **Feasibility**: 0.85 — A simple heuristic classifies incoming content: if the store call includes at least 2 structured tags (machine_id, domain, project) AND the content is < 200 tokens, classify as "structured fact" → route to Phase 1 (scope-chain discriminant). Otherwise, classify as "free-form insight" → route to Phase 2 (cosine discriminant). No ML classifier needed; the heuristic uses existing metadata fields. Implementation: 1 week. The content_type label can also be stored in PostgreSQL for future analysis of which type of content drives which retrieval patterns.
- **Description**: Recall's store endpoint inspects incoming metadata: if `machine_id` is set AND the content has ≥ 2 domain tags AND content length < 200 tokens, classify the memory as a "structured fact" and route it through the scope-qualified identity registry (Phase 1, identity-based discriminant). If any of these conditions fail, classify as "free-form insight" and route through the cosine comparison (Phase 2, similarity-based discriminant). Structured facts get define/set! semantics with scope isolation; free-form insights get pattern-separation semantics with dynamic thresholds. The routing decision is logged in PostgreSQL alongside the memory, enabling future analysis: "what fraction of session memories are structured facts vs. insights? which type is retrieved more often?"
