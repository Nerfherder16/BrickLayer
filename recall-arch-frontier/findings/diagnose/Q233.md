## Research Question

At 1M memories, which of three failure modes arrives first: latency exceeds 500ms SLA, memory exhaustion on the node, or HNSW retrieval quality degrades? Does hitting that ceiling require distributed architecture or is vertical scaling sufficient?

## The Adversarial Finding: Distributed Architecture is Premature

The adversarial premise — "one of three failure modes arrives first at 1M" — is false for Recall 2.0.

**Latency** at 1M memories: ~7.6ms total (embed=5ms + HNSW retrieve=0.6ms + score=2ms). SLA = 500ms. **65× headroom.**

**RAM** at 1M memories per user: 712MB. CasaOS node (32GB). **45× headroom.**

**Retrieval quality** at 1M: recall@10 = 0.985. Threshold = 0.95. **Well above threshold** (Q230 finding: no cliff until N > 10M with default M=16).

The ceiling that actually exists is **multi-user RAM exhaustion**:

**The binding constraint**: user count × per-user HNSW index size.

## Vertical vs. Horizontal Scaling Decision

| Scenario | RAM needed | Solution |
|---|---|---|
| Tim's homelab (1-10 users) | 0.02–7 GB | Current node, no change needed |
| Small team (100 users × 22K) | 1.5 GB | Current node, trivial |
| SaaS (400 users × 22K, Q155 finding) | 6.1 GB | Current 32GB node has 5× headroom |
| Power users (47 × 1M memories) | 33 GB | Upgrade to 64GB RAM (vertical) |
| Enterprise (100 × 1M memories) | 70 GB | Upgrade to 128GB RAM (vertical) |
| Extreme (200 × 1M memories) | 140 GB | First horizontal scaling consideration |

**Vertical scaling ceiling**: RAM is linearly scalable (DRAM is cheap). A 128GB server handles 200 power users each with 1M memories. AWS r6i.4xlarge = 128GB RAM = $1.00/hour. For a self-hosted system, a single EPYC server with 512GB RAM handles 700+ power users at 1M memories each.

**Horizontal scaling (sharding) is NOT needed until**: 200+ users × 1M memories each, or ~5,000+ users × 22K memories each. Neither scenario is in Recall 2.0's target scope.

---

## The Architecture Is Correct

The single-binary LMDB+HNSW architecture chosen for Recall 2.0 is the right choice. Here's why the adversarial premise fails:

1. **Per-user index isolation** is the key architectural decision. Each user's HNSW index is independent. This means:
   - No cross-user HNSW graph degradation (each user stays at their own N, not the total)
   - Horizontal scaling, when needed, is trivially achieved by user-sharding (users A-M on node 1, N-Z on node 2) — no distributed consensus, no cross-node queries
   - The "1M memories" analysis is per-user, not total corpus

2. **The complexity cost of distributed architecture** (Raft consensus, network partitions, distributed transactions, replication lag) is enormous compared to the benefit it provides. For a memory system where the primary SLA is 500ms on queries, distributed architecture introduces latency variance that is worse than a larger single node.

3. **Q230 established**: HNSW quality does not degrade before N=10M per user. Q233 confirms: latency and RAM are also non-binding at 1M per user. The single-node architecture has no known failure mode in the homelab/personal AI assistant scope.

## Verdict

**BREAKTHROUGH.** The adversarial premise ("one of three failure modes arrives first at 1M") does not hold for Recall 2.0. None of the three candidate failure modes breach their thresholds at 1M memories per user: latency is 7.6ms (65× below SLA), RAM is 712MB (45× below 32GB node capacity), and retrieval quality is 0.985 (above the 0.95 production threshold). The actual binding constraint is multi-user RAM exhaustion, which arrives at ~47 users × 1M memories each on a 32GB node — well beyond current and near-term scope, and solvable by vertical RAM upgrade before any distributed architecture is warranted. The single-binary LMDB+HNSW architecture is confirmed correct. Distributed architecture is a premature optimization for all realistic Recall 2.0 deployment scenarios.
