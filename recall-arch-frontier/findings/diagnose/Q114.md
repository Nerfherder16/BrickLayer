# Finding: Q114 — S7+S5 Silent Co-Failure: Data Integrity vs Retrieval Quality

**Question**: Does a silent reclassification job failure (Q104) cause S5 to report healthier than it actually is — specifically, if CO_RETRIEVED_FRESH edges that should be STALE remain FRESH and inflate the BFS traversal result set, does S5's compliance check measure the wrong denominator (total BFS results including stale-classified-FRESH edges) and thereby undercount the true orphan fraction?

**Severity**: High

## Adversarial Analysis: S5 denominator inflation under reclassification job failure

---

### Thread A: S5 Is Blind to Reclassification Failure

**Core claim**: S5's HEALTHY reading is structurally accurate about data integrity (no orphans, no PG join misses) but structurally silent about retrieval quality degradation (stale-but-real memories inflating the result set). S5 reports correctly for what it measures — but what it measures is the wrong question during a reclassification failure.

**Evidence and mechanism**:

S5 probes whether BFS-returned neighbor UUIDs exist in PostgreSQL. The probe runs a Cypher BFS with the node-level epoch predicate:

```cypher
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)
  AND neighbor.last_access_epoch <= $current_epoch
```

Under reclassification job failure at 30 missed sessions (from Q104), the CO_RETRIEVED_FRESH chain contains approximately 1,350 extra stale-classified-FRESH edges on top of the expected 1,350 correct FRESH edges. Each of those stale-FRESH edges points to a real Memory node. That node:

1. Exists in Neo4j (was never tombstoned or deleted)
2. Exists in PostgreSQL (the memory row is intact)
3. Has a valid Qdrant vector (was never evicted)
4. Passes the node-epoch predicate IF its `last_access_epoch` falls within [current_epoch - tau, current_epoch]

On point 4: during Phase 1 of the reclassification failure (0 to tau sessions of silence, i.e., 0–20 sessions), the stale FRESH edge endpoints DO pass the node-epoch predicate. They were reinforced within the last tau sessions in many cases — the edge is stale in the tau_edge (30-session) window sense, but the endpoint node is still fresh in the tau (20-session) window sense for edges reinforced in sessions 20–30 ago.

For those nodes, S5's PG join returns a found row. compliance_ratio stays at 1.0. S5 signals HEALTHY.

Meanwhile, from Q104's accumulation table at 30 missed sessions: precision has degraded to ~0.27. The BFS is returning 2× the intended number of neighbors (1,350 correct FRESH + 1,350 stale-FRESH), and roughly half are temporally irrelevant to the current context. The result set injected into Claude's context window includes associations from 30–60 sessions ago. S5 sees no orphans, measures a perfect compliance ratio, and emits no alarm.

**The denominator problem is real but not exactly as framed**: S5's denominator (total BFS results) IS inflated by stale-FRESH-edge endpoints in Phase 1. But the inflation does not reduce the compliance ratio — it does not change the numerator/denominator ratio because ALL the extra nodes pass the PG join. The denominator grows; the numerator grows by the same amount. The ratio stays 1.0. S5 is not "measuring the wrong denominator" in the sense that it computes a misleading ratio — it is measuring the right ratio for the wrong question. The ratio is a perfect 1.0: every result that BFS returns does exist in the stores. The problem is that the result set itself is contaminated with temporally stale entries, and S5 has no mechanism to detect that.

**Best case scenario**: The reclassification job resumes after 5 sessions. S5 was HEALTHY throughout. No alarm fired. S7 (drift_fraction) caught the problem at session 5 (drift_fraction > 0.15 threshold from Q104). The S5/S7 division of labor worked as designed — S7 owns this failure mode, S5 was never the right sensor.

**Worst case scenario**: The reclassification job fails for 60+ sessions. S5 is HEALTHY throughout. S7 fires early (session 5) but the operator does not understand why S7 is alerting while S5 is HEALTHY — the conceptual framework for reading both signals together is absent from documentation. The operator concludes "S5 says the data is fine, S7 must be a false positive" and suppresses the alarm. Retrieval quality degrades to precision ~0.16. The system continues surfacing memories from months ago as if they are current context.

### Thread B: S5 Self-Corrects via Node Epoch Filter

**Core claim**: S5 is not perfectly blind — it has a partial, time-delayed self-correction mechanism via the node-epoch filter that bounds the Phase 1 inflation and terminates it in Phase 2. The "S5 denominator inflation" has a natural ceiling, and the inflated denominator is populated entirely by valid data, which is exactly what S5 is designed to confirm.

**Evidence and mechanism**:

The BFS node-level predicate `WHERE neighbor.last_access_epoch >= ($current_epoch - tau)` provides a second independent freshness gate beyond the edge type (FRESH vs STALE). A stale-classified FRESH edge leads to a real Memory node, but that node must itself have been accessed within the last tau=20 sessions to enter the BFS result set.

**The timing argument**: A CO_RETRIEVED_FRESH edge is reinforced when memories A and B are co-retrieved together. At the time of reinforcement, both A and B have `last_access_epoch` equal (approximately) to the reinforcement epoch. The edge type becomes stale at tau_edge=30 sessions after reinforcement. At that point, the edge endpoint (memory B) also has a `last_access_epoch` approximately equal to the reinforcement epoch. The node-epoch predicate evaluates as:

```
B.last_access_epoch >= (current_epoch - tau)
= reinforcement_epoch >= (current_epoch - 20)
= current_epoch - reinforcement_epoch <= 20
```

If the edge has been silent (not re-reinforced) since the reinforcement event, and more than tau=20 sessions have passed since that event, memory B's `last_access_epoch` is older than current_epoch - 20. The node-epoch predicate REJECTS it. The stale FRESH edge's endpoint becomes invisible to BFS.

This means:
- Edges reinforced 21–30 sessions ago: stale at the edge level (tau_edge=30), but the endpoint node is also old enough to be rejected by the node predicate (tau=20). No inflation in BFS results.
- Edges reinforced 0–20 sessions ago: within the node-epoch window; they would be FRESH edges anyway (not yet stale under tau_edge=30). No inflation.
- The only window where stale FRESH edges ACTUALLY inflate the denominator: edges reinforced exactly 21–30 sessions ago, which are stale at the edge level but their endpoints still pass the node predicate.

Wait — this requires re-examination. tau_edge=30 means an edge becomes stale after 30 sessions. tau=20 means a node is filtered after 20 sessions of no access. For an edge reinforced at epoch E, the edge becomes stale at epoch E+30. The endpoint node is filtered at epoch E+20 (if not re-accessed). So:

- At epoch E+20 to E+30: the edge is still FRESH (hasn't hit tau_edge=30 yet), and the node is being filtered (has hit tau=20). But this edge was correctly classified FRESH — it is not a source of inflation.
- At epoch E+30+: the edge is stale-classified-FRESH (if the job failed). The node's last_access_epoch is at E (oldest case). The node is filtered at E+20, which means at epoch E+30 the node was already filtered for 10 sessions. The stale FRESH edge's endpoint is REJECTED by the node predicate.

**Thread B's key insight is therefore structural**: When tau < tau_edge (here tau=20 < tau_edge=30), every stale FRESH edge (reinforced more than tau_edge sessions ago) has an endpoint that has already passed the node expiry window (tau sessions ago). Because tau=20 < tau_edge=30, by the time an edge becomes stale (30 sessions after reinforcement), the endpoint node has already aged out of the BFS node filter (20 sessions after reinforcement + 10 more = 30 total, or exactly at the stale boundary). In the tau < tau_edge case, the node filter automatically catches the stale FRESH edges before they can inflate the BFS result set.

**The boundary case**: If tau = tau_edge (both=20), then edges become stale exactly as nodes age out. No inflation. If tau > tau_edge (node window longer than edge window), some stale FRESH edges lead to nodes that still pass the node filter. Inflation occurs. If tau < tau_edge (current design: tau=20 < tau_edge=30), every stale FRESH edge's endpoint is guaranteed to have aged out of the node filter before the edge classification job even needed to reclassify it.

**Thread B's conclusion**: The architecture with tau < tau_edge has a structural self-correction property: the node-epoch filter backstop prevents stale FRESH edges from inflating the BFS denominator. S5 is not blind to reclassification failure — it is insulated from it by the tau/tau_edge co-design. The S5 denominator inflation question in the problem framing assumes tau >= tau_edge, which is not the current design.

**Failure mode**: Thread B is correct only under the specific condition tau < tau_edge AND nodes are not re-accessed after the initial co-retrieval event. If a node is re-accessed (its `last_access_epoch` is updated by a solo retrieval unrelated to the co-retrieval pattern), its node-epoch window resets. The stale FRESH edge endpoint is then "fresh" again at the node level and re-enters the BFS. Thread B's self-correction fails for frequently re-accessed "hub" nodes — nodes Tim returns to regularly regardless of context.

### Synthesis

**What Thread A assumes that Thread B exposes**:

Thread A assumes that stale FRESH edges uniformly inflate the BFS denominator during reclassification failure. Thread B exposes that this is only true when tau >= tau_edge — which is NOT the current design. Under tau=20 < tau_edge=30, there is a 10-session "safety gap": by the time an edge becomes stale (at tau_edge=30 sessions), its endpoint node has already aged past the node filter (at tau=20 sessions), making it invisible to BFS regardless of edge classification. The structural inflation that Thread A describes is partially neutralized by the tau/tau_edge design ratio — but only for nodes that have not been re-accessed since the co-retrieval event.

**What Thread B assumes that Thread A exposes**:

Thread B assumes node access patterns are uncorrelated with co-retrieval patterns — that a node last accessed in a co-retrieval event will not be re-accessed individually before the edge ages out. This assumption fails for hub nodes: memory nodes Tim returns to repeatedly (key infrastructure facts, persistent project conventions, frequently-referenced concepts). These nodes have high `last_access_epoch` values independent of co-retrieval, so they never age out of the node filter. Stale FRESH edges pointing to hub nodes DO inflate the BFS denominator even under tau < tau_edge. The higher a node's access frequency, the more likely it is to remain in the BFS working set regardless of edge staleness, and the more damage the reclassification failure does to retrieval precision for that specific node type.

**When Thread A is right**:
- When any stale FRESH edge endpoints are hub nodes (high individual access frequency, last_access_epoch never falls below current_epoch - tau)
- When tau >= tau_edge (node filter window equals or exceeds edge window — the safety gap disappears)
- When the reclassification job has been silent for more than tau sessions (Phase 1: 0–20 sessions), where the timing geometry still allows some stale edges with fresh-enough endpoint nodes
- When Tim pivots to a new project domain, making old co-retrieval associations genuinely irrelevant rather than merely stale

**When Thread B is right**:
- For non-hub nodes (nodes Tim accesses primarily through co-retrieval, not independently): the tau < tau_edge safety gap holds structurally
- For the majority of Memory nodes at Tim's density, where most nodes are accessed in bursts then quiescent: the self-correction mechanism operates as described
- For Phase 2 of the reclassification failure (> tau_edge sessions of silence): the node filter has fully engaged for all non-hub endpoints, BFS results stabilize, and S5's denominator inflation is bounded

**Is the S5 HEALTHY reading accurate or misleading?**

The S5 HEALTHY reading is **accurately describing data integrity** and **silently misrepresenting retrieval quality**. These are two genuinely different properties:

- **Data integrity** (S5's actual domain): "Are the memories returned by BFS real, uncorrupted, cross-store consistent?" Yes. Every stale-FRESH-edge endpoint exists in Neo4j, PostgreSQL, and Qdrant. No orphans. No corrupted rows. No fake UUIDs. S5 = HEALTHY is correct.
- **Retrieval quality** (S7's domain): "Are the memories returned by BFS temporally relevant to the current context?" Unknown from S5. S5 was never designed to measure this. S7 (drift_fraction from Q104) is the correct signal for this question.

The problem is not that S5 is wrong — it is that S5 and S7 measure orthogonal properties of the retrieval system, and when both signals are HEALTHY (or when S5 is HEALTHY and S7 is absent from the dashboard), the operator has no single indicator that retrieval precision has degraded. The health signal matrix is incomplete for the reclassification failure mode.

**Should the S7 threshold be tightened from 0.15 to 0.05?**

Yes, with caveats. The Q104 threshold of 0.15 (approximately 5 missed sessions) was set to balance alarm frequency against false positives. However, given that S5 is silent throughout the reclassification failure and the operator may not understand the S5/S7 conceptual distinction, tightening S7 to 0.05 (approximately 1–2 missed sessions) provides earlier detection while the precision degradation is minimal (precision ~0.95 at 2 missed sessions). The false positive risk for S7 at 0.05 is low: in normal operation with the job running correctly, drift_fraction = 0 immediately after each reclassification run. A threshold of 0.05 fires only when >67 stale FRESH edges accumulate — equivalent to 1.5 missed job cycles. This is not a false positive; it is a real signal.

Recommended S7 thresholds: WARNING at 0.05 (early warning, 1–2 missed sessions), CRITICAL at 0.15 (confirmed degradation, 3–4 missed sessions). This creates a 3-session detection window before Phase 1 precision degradation exceeds ~0.92.

**The conceptual distinction that must be documented**:

The health signal matrix requires explicit documentation of what each signal measures and what it explicitly does NOT measure:

| Signal | Measures | Does NOT measure |
|--------|----------|-----------------|
| S5 (graph_traversal_compliance) | Data integrity: BFS results exist in all three stores (Neo4j, PG, Qdrant) | Temporal relevance: whether BFS results are from the correct epoch window |
| S7 (drift_fraction) | Retrieval quality: fraction of FRESH edges that are misclassified (stale but labeled FRESH) | Data integrity: whether the memories themselves are real and uncorrupted |

The silent co-failure mode: **S5 = HEALTHY AND S7 = WARNING does NOT mean "the system is mostly fine with a minor quality issue." It means "the data is completely intact but what is being retrieved is increasingly stale." This distinction is operationally critical — the remediation is completely different (reclassification job restart vs orphan cleanup).**

**The architecture Thread B would accept**:

A health signal documentation layer that explicitly defines signal orthogonality. Concretely:

1. **S7 threshold tightening**: WARNING at drift_fraction > 0.05, CRITICAL at drift_fraction > 0.15. This gives a 3-session detection window before precision falls below 0.92.

2. **S5+S7 co-alert rule in the correlation layer**: When S5 = HEALTHY AND S7 = WARNING/CRITICAL, emit a new co-alert:
   ```
   alarm_code: "RETRIEVAL_QUALITY_DEGRADED_DATA_INTACT"
   interpretation: "Data integrity confirmed (S5 healthy). Retrieval precision degrading
   from reclassification job failure (S7 alerting). Stale-classified FRESH edges are
   inflating the BFS working set with temporally irrelevant associations. Impact: retrieval
   precision approximately [computed from drift_fraction]. Remediation: restart the
   session-end reclassification hook immediately."
   automated: True  # job restart is safe; no data loss risk
   ```

3. **Signal orthogonality documentation** in the `/health/detail` API response: each signal includes a `measures` field (one sentence) and a `does_not_measure` field (one sentence), so operators reading the raw API response understand the scope boundary of each signal.

4. **Composite retrieval health score**: A derived scalar combining S5 compliance_ratio (weighted 0.5) and S7 drift_fraction inverted (weighted 0.5) into a single "retrieval_health" metric. This prevents the operator from reading S5 = 1.0 and concluding retrieval is perfect when S7 = 0.27.

### Idea: signal-orthogonality-documentation-in-health-api

- **Description**: Each signal in the `/health/detail` API response should include a `measures` field (one sentence describing what the signal confirms) and a `does_not_measure` field (one sentence describing the adjacent failure mode that this signal cannot detect). No new queries required.

### Idea: tau-tau-edge-safety-gap-design-principle
- **Novelty**: 0.73
- **Evidence**: 0.69
- **Feasibility**: 0.97
- **Description**: The architectural property that tau < tau_edge creates a structural safety gap where stale FRESH edges (aged past tau_edge) have endpoints that have also aged past the node filter (tau). For non-hub nodes, this means reclassification job failures produce zero BFS denominator inflation — the node-epoch filter catches what the edge-classification job missed. This is an emergent defense-in-depth property of the dual-predicate architecture. Formalize it as a design principle: tau_edge should always exceed tau by at least the maximum tolerable reclassification job silence (in sessions). Current values (tau=20, tau_edge=30) provide a 10-session safety gap. This principle should be documented in the architecture spec and enforced as a configuration invariant (reject configurations where tau_edge <= tau).

### Idea: s7-threshold-tightening-to-early-warning

- **Evidence**: 0.74
- **Feasibility**: 0.96
- **Description**: Tighten the S7 drift_fraction WARNING threshold from 0.15 to 0.05 (approximately 1–2 missed reclassification sessions). At drift_fraction = 0.05, retrieval precision is still ~0.95 — well above the perceptible degradation threshold. But this gives a 3-session detection window before precision falls below 0.90, and it compensates for S5's silence during reclassification failures. The false positive rate at 0.05 is near-zero in normal operation (drift_fraction = 0 immediately after each reclassification run; 0.05 requires ~67 stale FRESH edges, equivalent to 1.5 missed cycles).
