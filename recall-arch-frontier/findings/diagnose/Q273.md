## Summary

**The async LLM queue saturation scenario does not materialize in practice.** At qwen3:14b throughput (~26 calls/minute on RTX 3090), the 50-call session budget drains in under 2 minutes — fast enough to process all qualifying events within any normal session. Even in the worst case (150 Bash events, 52 qualify — 2 over budget), only 2 events fall back to C+. Critically: even at **0% LLM extraction**, MRR = 88% — exactly at the Q266 threshold, not below it. Queue saturation cannot push MRR below the acceptable range.

## MRR Impact Analysis

**Q266 MRR model:**
- Strategy A (full LLM): 1.00 (baseline)
- Strategy D (hybrid): 0.94 (6% below A)
- Strategy C+ (no LLM): 0.88 (12% below A, 10% threshold)

The 6% MRR gap between C+ and D comes from async LLM extraction for ~25% of qualifying Bash events. If the queue saturates and only X% of qualifying events receive LLM extraction:

| LLM extraction rate | MRR estimate | Status |
|--------------------|-------------|--------|
| 100% (full) | 94.0% | Strategy D target |
| 80% (mild saturation) | 92.8% | Well above threshold |
| 60% | 91.6% | Above threshold |
| 40% | 90.4% | Above threshold |
| 20% | 89.2% | Above threshold |
| **0% (no LLM at all)** | **88.0%** | **Exactly at threshold** |

**The floor is 88% MRR** — which is the C+ baseline. Queue saturation cannot push MRR below the C+ level, because falling back to C+ IS the fallback. Even total queue failure (0% LLM extraction) degrades Strategy D to Strategy C+, which is still within the Q266 acceptance criterion (88% = exactly at 10% gap threshold, not below).

## What Happens If Ollama Is Unavailable

If qwen3:14b (192.168.50.62) is offline during a session:
- Async LLM queue fires requests that fail with connection error
- Memories are stored with `description: null` (C+ path)
- No session-blocking behavior (fire-and-forget pattern from Q270)
- MRR falls to 88% — still within acceptable range
- Recovery: when Ollama comes back online, descriptions can optionally be filled retroactively

**Graceful degradation is built into the architecture.** The fire-and-forget queue means Ollama unavailability is invisible to the user.
