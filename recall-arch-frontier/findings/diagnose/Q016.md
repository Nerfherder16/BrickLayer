# Finding: Q016 — Taboo Architecture: Forgetting System from First Principles

**Question**: Q016 — Ignoring all existing tools, design a forgetting system for a text memory store. It must decide what to evict when storage is constrained. It must not evict memories that are still causally relevant to active work. What is the eviction algorithm?
**Agent**: quantitative-analyst
**Question Type**: TABOO
**Verdict**: BREAKTHROUGH
**Severity**: High

## First Principles Derivation

### 1. The Causal Relevance Invariant

**Formal definition**: Memory M is causally relevant to task T if and only if there exists a path from some fact F contained in M to the output of T through T's inference graph — meaning F influences a decision, recommendation, or action that T produces. Removing M changes T's output.

**The proxy detection problem**: We cannot execute T to verify this. We need a conservative approximation derivable without running the task.

**Analogy from compiler theory — liveness analysis**: In data-flow analysis, a variable V is "live" at program point P if there exists a path through the control-flow graph from P to a subsequent use of V. The compiler approximates liveness statically — it cannot run the program; it traces reference chains. This approximation is always conservative (it may declare a variable live when it is actually dead, but never the reverse).

**Derived proxy for causal relevance**: Memory M is *approximately* causally relevant to the active work context if:
1. M was retrieved during the current session window (a retrieve event exists for M in the session retrieval set), OR
2. M belongs to a fact cluster (a set of semantically related memories) where at least one cluster member was retrieved during the current session.

Condition 1 is direct evidence: T already used M, so M is in T's inference graph.
Condition 2 is structural evidence: if M's cluster-mates are active, M is likely reachable via the same inference path.

**Conservative approximation direction**: This over-protects (some protected memories may no longer be needed). Over-protection is the correct failure mode — a spurious retention wastes storage; a spurious eviction corrupts an active task's context.

**The formal invariant**: The eviction algorithm must maintain the property that no memory M in the current-session retrieval set, and no memory M whose cluster contains a member in the current-session retrieval set, is included in the eviction candidate pool. This invariant must hold at every point during the eviction pass.

### 2. The Eviction Priority Function (derived)

**Starting requirement**: Evict what is least likely to be causally relevant to future work.

**What signals predict future causal relevance?**

From information theory, the expected future causal relevance of M to a future task T' is proportional to the probability P(T' references a fact in M). This probability is estimated by three observable signals:

- **Retrieval frequency** (`retrieval_count`): the total number of times M has been retrieved across all historical sessions. High-frequency retrieval is evidence that M participates in many task contexts — it is broadly relevant. Low or zero retrieval frequency is evidence that M has never been needed.

- **Temporal recency of last retrieval** (`time_since_last_retrieval`): a memory retrieved 10 minutes ago is far more predictive of current task relevance than one retrieved 6 months ago. This is the temporal locality property well-established in cache replacement theory — access patterns exhibit temporal clustering.

- **Cluster activity** (`cluster_active`): if M belongs to a fact cluster where any member was retrieved in the last session, M's relevance probability is elevated. Causal relevance propagates through the cluster because the task that retrieved a cluster member is likely to need other cluster members.

**Derived eviction priority function** (higher score = higher eviction priority = evict these first):

```
eviction_priority(M) = base_staleness(M) × isolation_factor(M)

base_staleness(M) = (1 / log(retrieval_count + 2))
                    × exp(-time_since_last_retrieval / τ_half_life)

isolation_factor(M) = 0.05  if cluster_active(M) this session
                    = 0.30  if cluster_active(M) in last 7 days (not current session)
                    = 1.00  if M is isolated (no cluster) or cluster dormant > 7 days
```

Where `τ_half_life` is an empirically tuned decay constant (recommended starting value: 14 days for technical system memories; 3 days for session-specific memories).

**Properties the function satisfies by construction**:
- Memories with retrieval_count = 0 and recent storage receive maximum base staleness: they have never been used and are the least likely to be causally relevant.
- Memories retrieved many times recently are nearly impossible to evict: retrieval_count drives the log term up, and the recency term approaches 1.0.
- Cluster membership dramatically suppresses the priority: a cluster member that was accessed this session has priority 0.05 × base_staleness — essentially immune to eviction.

### 3. The Causal Protection Mechanism (derived)

**The protection problem**: At eviction time, we do not know which task T' will run next. We must protect memories that T' might need without knowing what T' is.

**Analogy from OS design — page pinning**: When the OS initiates a DMA (direct memory access) operation to a physical memory page, it "pins" that page — marks it as non-evictable by the virtual memory pager — for the duration of the DMA operation. The pin is released when the operation completes. This provides guaranteed safety without knowing what other processes need the page.

**Derived mechanism — session retrieval set (SRS)**:

The SRS is a set of memory IDs populated by every retrieve operation during the current session. It is stored in the ephemeral session layer (Redis in Recall's stack) and cleared at session end.

At eviction candidate selection time:
1. Read the SRS from Redis: `srs = redis.smembers(f"session:{session_id}:retrieved")`
2. For each candidate memory M: if M's ID is in the SRS → exclude from candidate pool (pinned).
3. Cluster extension: for each cluster C where any member's ID is in the SRS → exclude all members of C from the candidate pool.

**The cluster extension** requires that clusters are pre-computed and stored (see Idea: `causal-cluster-index`). Clusters can be maintained as edges in the graph layer (Neo4j in Recall's stack) with edge type `SAME_FACT_FAMILY`. Traversal at eviction time: for each SRS member, BFS one hop along `SAME_FACT_FAMILY` edges to find all cluster members, add to protected set.

**Why this is sufficient and conservative**: The SRS captures all memories that this session's tasks have already demanded. Any memory in the SRS was proven causally relevant to this session (definition: it was retrieved, meaning some task used it). Cluster extension is the conservative over-approximation — it protects memories that share a fact family with proven-relevant memories, on the theory that if one fact in a cluster is needed, the others may be needed too.

### 4. The Eviction Trigger Condition (derived)

**The trigger problem**: We cannot run eviction on every store operation (O(N) cost per write). We cannot run it never (store fills). We need the correct intermediate.

**From storage capacity physics**: Let C be the capacity limit (maximum number of memories in the long-lived store) and N be the current count. The store is under pressure when N approaches C.

**Analogy from log-structured storage compaction**: LSM-tree compaction is triggered when the ratio of stale data (deleted or superseded entries) to total data in a level exceeds a threshold — not on every write. This amortizes the O(N) cost across many writes and ensures compaction runs before the level overflows.

**Derived trigger conditions** (any one is sufficient to fire eviction):

- **Capacity trigger**: `N > 0.85 × C` — triggered when the store is 85% full. This gives a 15% buffer before the hard limit, ensuring eviction completes before the store is actually full.
- **Burst trigger**: more than K new memories stored in the current session (e.g., K=100). A write burst may indicate a large ingestion event that will push N over threshold; a lightweight "zero-retrieval stale eviction" pass runs immediately for memories with retrieval_count=0 and age > 30 days.
- **Session-end trigger**: always runs the eviction candidate assessment at session end (the consolidation job from Q007). This is the primary eviction path — it runs in the background, does not block any live operation, and has access to the full session context for the causal protection decision.

**What not to trigger on**: individual store operations. The store endpoint must remain O(1). Eviction is deferred work, always background, always amortized across a session boundary.

---

### 5. The Eviction Unit (derived)

**The unit question**: Should the eviction algorithm evict individual memories or fact clusters?

**Analogy from database buffer pool management**: A database does not evict individual rows from the buffer pool — it evicts pages (typically 4KB or 8KB blocks). Rows within the same page have spatial locality: they were stored together because they are related. Evicting a page that contains one cold row but also one hot row is an error — the hot row becomes unavailable and must be fetched from disk on the next access.

**The same logic applies to fact clusters**: A fact cluster (e.g., "Redis configuration cluster": port, TTL, connection pool size, cluster mode flag) has semantic locality — its members are co-retrieved whenever a Redis-related task is active. Evicting one member while retaining others creates a "partial cluster" — a degraded state where the memory store presents an incomplete picture of a technical system.

A partial cluster is worse than either:
- A complete cluster (all facts available, full context for reasoning), or
- An absent cluster (no facts available, the agent knows to ask the user for the information rather than inferring from incomplete data).

**Derived eviction unit rule**:

- If M is isolated (no cluster membership): evict M alone when its eviction priority exceeds the eviction threshold.
- If M is a cluster member: evaluate the cluster's eviction priority as the minimum eviction priority across all cluster members (the "coldest" member sets the floor). Evict the entire cluster if and only if the cluster priority exceeds the threshold AND no cluster member is in the current-session SRS. If any member is SRS-protected, the entire cluster is protected.

**The minimum-priority rule for clusters** is correct because: if the cluster contains even one warm member (retrieved recently, high eviction priority resistance), the cluster as a unit should not be evicted — the warm member signals that the cluster's topic is still active.

### 6. What Importance-Score Eviction Gets Wrong

**The violated invariant**: Causal relevance is a relational property of the pair (M, T), not an intrinsic property of M alone.

Importance-score eviction assigns a scalar value to M at write time and uses that scalar to determine eviction eligibility. This assumes that `importance(M)` is well-defined without reference to T — that some memories are intrinsically important and others are intrinsically unimportant, regardless of the current task.

This assumption is false. Consider two memories:

- Memory A: "Redis is running on port 6379 with a 48-hour TTL." Assigned importance score: 0.3 (a mundane operational detail at write time).
- Memory B: "The system architecture uses microservices with event-driven communication." Assigned importance score: 0.9 (a high-level design decision, seems very important at write time).

**Importance-score eviction would evict A and retain B.** But when the user is actively debugging a Redis connection timeout — task T = "fix the Redis connection issue" — removing A changes the output of T (the agent would not know to check port 6379 or the TTL configuration). B is entirely irrelevant to T. The eviction algorithm made the worst possible choice.

**The root cause**: importance scores measure "how significant did this fact seem when written?" — an intrinsic, time-of-write property. Causal relevance measures "does this fact change the output of the current task?" — a relational, time-of-use property. These are orthogonal. The signals that determine write-time importance (semantic richness, stated significance, association with high-stakes outcomes) have no predictable relationship to the task-specific causal relevance of the memory at eviction time.

**The correct signal hierarchy for eviction**:
1. Causal protection (is M in the SRS or a protected cluster?) — binary, not a score; protects regardless of importance.
2. Retrieval history (how often and how recently was M retrieved across tasks?) — a relational signal accumulated over time.
3. Cluster cohesion (is M's cluster active?) — a relational signal derived from the graph structure.
4. Importance score — last resort tiebreaker among memories that are equally cold by signals 1-3.

Importance score is demoted from primary signal to tiebreaker. This is the correct role: it contains weak evidence about future relevance (things deemed important at write time are slightly more likely to be needed again), but it is dominated by actual retrieval evidence when that evidence is available.

## Ideas Extracted (using real Recall stack components)

### Idea: session-retrieval-set-pin
- **Novelty**: 0.85 — No known AI memory system maintains a session-scoped retrieval set as an explicit eviction exclusion list. Recall's existing infrastructure (Redis session storage) makes this immediately implementable, but the specific mechanism — a pinning set that protects memories from eviction during their session of use, analogous to OS page pinning — is absent from all production systems examined in Q007.
- **Evidence**: 0.90 — OS page pinning for DMA operations is a standard, well-understood mechanism in every major OS kernel (Linux `mlock()`, Windows `VirtualLock()`). The translation to memory store eviction is direct: pins are held for the duration of an operation and released on completion. The mechanism is not speculative — it is engineering best practice in storage systems.
- **Feasibility**: 0.95 — Recall already uses Redis for session state. Implementation: on every successful `recall_retrieve` call, add the returned memory ID(s) to a Redis set keyed by session ID: `SADD session:{session_id}:pinned {memory_id}`. Set TTL on the key to session lifetime + 1 hour buffer. In the consolidation job (already established in Q007's `session-end-consolidation-job`), before building the eviction candidate list, read the pinned set and exclude all pinned IDs. The entire implementation is approximately 20 lines of Python in the existing consolidation worker, plus one Redis `SADD` call in the retrieve endpoint. Implementation time: 2-3 days.
- **Description**: Add a session retrieval set (SRS) to Recall's Redis layer. Every time `recall_retrieve` returns a memory, the memory's ID is added to `session:{session_id}:pinned` with a 6-hour TTL. The consolidation worker at session end reads this set before running any eviction: all pinned IDs and their cluster-mates (reachable via `SAME_FACT_FAMILY` Neo4j edges) are excluded from the eviction candidate pool. This directly implements the causal protection invariant: memories proven to be in the current session's inference graph are unconditionally protected from eviction during that session. Pins are released automatically when the Redis TTL expires at session end.

### Idea: causal-cluster-index
- **Novelty**: 0.80 — Clustering memories by semantic relatedness for retrieval purposes is known. The novel element is using the cluster as the unit of causal protection and eviction — specifically, the rule that the entire cluster is protected if any member is in the SRS, and the entire cluster is evicted as a unit only when all members are cold. This "cluster-granularity eviction" has no known production implementation in AI memory systems.
- **Evidence**: 0.85 — Database buffer pool page-level eviction (rather than row-level) is a textbook mechanism (Database System Concepts, Silberschatz et al.) with decades of production validation. The spatial locality argument — that related items are co-accessed and should be co-evicted — is well-established. The translation to semantic fact clusters in a memory store follows directly from the same locality principle.
- **Feasibility**: 0.70 — Recall already has Neo4j for relationship storage. Implementation requires: (1) at store time, when a memory M is created, check its semantic neighbors (query for memories within a similarity threshold); if neighbors exist and share common entity references (same service name, same project, same configuration domain), create `SAME_FACT_FAMILY` edges between M and its cluster-mates in Neo4j. (2) At eviction time, the consolidation worker uses the cluster index to implement cluster-granularity eviction: BFS one hop from each candidate to find cluster members, evaluate minimum eviction priority across the cluster, protect or evict the cluster as a unit. Main complexity: defining "same fact family" — entity extraction from memory text to identify shared domain references (e.g., "Redis" as a shared entity). A lightweight named-entity approach using the Qdrant payload's existing tag fields is sufficient to start. Implementation time: 2-3 weeks.
- **Description**: Build a causal cluster index in Neo4j by adding `SAME_FACT_FAMILY` edges between memories that share domain entities (service names, project identifiers, configuration keys) and fall within a semantic similarity threshold. At store time: extract entity tags from the incoming memory text (already present in Recall's payload schema), query for existing memories with matching entity tags, and create bidirectional `SAME_FACT_FAMILY` edges. At eviction time: the eviction worker performs a one-hop BFS in Neo4j from each eviction candidate to discover its cluster, evaluates the cluster's collective eviction priority (minimum across all members), and makes a cluster-level eviction decision. This ensures that fact families are kept whole or evicted whole — never fragmented into partial clusters that present incomplete context to the agent.

### Idea: retrieval-weighted-eviction-score
- **Novelty**: 0.75 — Combining retrieval frequency and temporal recency as the primary eviction signal (rather than importance score) is a principled departure from current practice. The specific formulation — log(retrieval_count + 2) × exp(-time_since_last_retrieval / τ) — treating the memory store as analogous to a CPU cache with a time-decaying access weight — is not documented in any production AI memory system. LFU (Least Frequently Used) and LRU (Least Recently Used) caching policies each capture one signal alone; this derivation combines both into a unified priority.
- **Evidence**: 0.80 — LFU and LRU cache replacement policies are textbook computer science with extensive theoretical analysis (King 1971, Sleator & Tarjan 1985). The LRFU (Least Recently/Frequently Used) combination policy was proposed by Lee et al. (2001) in "LRFU: A Spectrum of Policies that Subsumes the Least Recently Used and Least Frequently Used Policies" (IEEE Transactions on Computers) and shown to outperform either alone on mixed workloads. The application to semantic memory store eviction is a direct analogy.
- **Feasibility**: 0.85 — Recall already stores importance scores in the PostgreSQL `memories` table. Adding `retrieval_count` (integer, default 0, incremented on each retrieve) and `last_retrieved_at` (timestamp, updated on each retrieve) requires a schema migration: two new columns. The eviction priority function runs as a PostgreSQL query sorting by the derived score. The consolidation worker selects the bottom-N candidates by eviction priority, applies the SRS exclusion, then deletes them from Qdrant and PostgreSQL. Implementation time: 1 week (schema migration + updated consolidation worker query).
- **Description**: Add `retrieval_count` and `last_retrieved_at` columns to Recall's PostgreSQL `memories` table. The retrieve endpoint increments `retrieval_count` and updates `last_retrieved_at` on every successful retrieval. The consolidation worker computes eviction priority as `(1 / LOG(retrieval_count + 2)) * EXP(-EXTRACT(EPOCH FROM (NOW() - last_retrieved_at)) / 1209600.0)` (τ = 14 days in seconds), orders all non-pinned, non-cluster-protected candidates by this score descending, and evicts the top-N until the store drops below 75% capacity. Importance score is used only as a final tiebreaker when two memories have identical retrieval-weighted scores. This demotes importance score from primary eviction signal to last resort, replacing it with empirical retrieval evidence that directly approximates causal relevance.

### Idea: capacity-threshold-eviction-trigger
- **Novelty**: 0.65 — Capacity-threshold eviction triggers are standard in many systems. The novel element here is the specific combination: (a) using a ratio trigger (85% full) rather than an absolute count, (b) coupling the trigger exclusively to the session-end job rather than the store hot path, and (c) the "burst mode" lightweight pre-eviction for high write-velocity sessions. This specific design for an AI memory store is not documented elsewhere, though the individual components (ratio triggers, background eviction) are standard engineering.
- **Evidence**: 0.90 — Capacity-ratio triggers are used in every major storage system: LSM-tree compaction triggers (RocksDB triggers at 1.1× level size limit), Linux kernel page reclaim (triggers at watermark thresholds, not absolute counts), Redis `maxmemory-policy` (triggers when configured maxmemory is reached). The ratio-based trigger is universally preferred over absolute-count triggers because it scales with configured capacity. This is mature engineering practice.
- **Feasibility**: 0.95 — The trigger condition is a single COUNT query against Qdrant or PostgreSQL. Implementation: at the start of the session-end consolidation job, check `SELECT COUNT(*) FROM memories WHERE tier='long_lived'`. If count > `CAPACITY_LIMIT * 0.85`, run the eviction pass. The capacity limit is a configuration parameter in Recall's settings. The burst trigger adds a lightweight check at store time: if `session_store_count > 100` (tracked in Redis with `INCR`), run a quick eviction pass targeting only zero-retrieval memories older than 30 days. Implementation time: 2-3 days (one configuration parameter, two COUNT queries, existing consolidation worker handles the rest).
- **Description**: Implement capacity-ratio eviction triggering in Recall's consolidation worker. Configure `MEMORY_CAPACITY_LIMIT` as a deployment parameter (recommended initial value: 100,000 memories for Recall's current hardware). The session-end consolidation job checks `COUNT(memories)` against `CAPACITY_LIMIT * 0.85`. If below threshold, skip eviction (no-op). If above threshold, run the full eviction pass (SRS exclusion + cluster protection + retrieval-weighted scoring). Additionally, instrument the store endpoint with a Redis counter `INCR session:{session_id}:store_count`; if this exceeds 100 within a session, trigger a lightweight pre-eviction targeting only zero-retrieval + old memories, to prevent write bursts from filling the store before the session-end job runs. These two triggers cover both the steady-state case (gradual fill) and the burst case (large ingestion events).

### Idea: importance-score-demotion
- **Novelty**: 0.70 — Demoting importance score from primary eviction signal to tiebreaker, and articulating the formal reason (importance is an intrinsic property; causal relevance is relational), is a conceptual contribution. No production system has explicitly documented this architectural choice or its justification. The implementation change is small; the conceptual shift is significant.
- **Evidence**: 0.75 — The distinction between intrinsic and relational properties is standard in database theory (functional dependencies are relational; column values are intrinsic). The argument that write-time importance does not predict retrieval-time relevance is well-supported by the general observation that access patterns in document stores are workload-dependent and not predictable from document content alone. Cache replacement policy literature (King 1971; Lee et al. 2001) consistently shows that recency/frequency signals outperform content-based signals for cache eviction when the workload has temporal locality.
- **Feasibility**: 0.90 — The demotion is a change to the consolidation worker's eviction sort order: ORDER BY retrieval_weighted_score DESC, importance_score ASC (importance used as secondary sort key only). No schema changes required. The importance score column remains in PostgreSQL and continues to be written at store time — it is just no longer the primary eviction criterion. Implementation time: 1 day (one line change in the consolidation worker's eviction query).
- **Description**: Change Recall's consolidation worker to use retrieval-weighted score as the primary eviction criterion and importance score as a secondary tiebreaker only. In the eviction candidate query: `ORDER BY retrieval_weighted_score DESC, importance_score ASC LIMIT {eviction_count}`. Document the reason in a code comment: "Importance score measures write-time salience (intrinsic property). Causal relevance to active tasks is determined by retrieval history (relational property). Retrieval history is a superior eviction signal because it reflects actual task demand, not predicted importance." This makes the architectural decision explicit and auditable, preventing future maintainers from reverting to importance-score-primary eviction without understanding the tradeoff.

## Summary: The Complete Eviction Algorithm

**Pseudocode for the session-end consolidation + eviction pass**:

```
function eviction_pass(session_id):
    # Step 1: Build the protected set (causal protection invariant)
    pinned_ids = redis.smembers(f"session:{session_id}:pinned")
    cluster_protected_ids = neo4j.bfs_one_hop(pinned_ids, edge_type="SAME_FACT_FAMILY")
    protected = pinned_ids ∪ cluster_protected_ids

    # Step 2: Check trigger condition
    total_count = postgres.count("SELECT COUNT(*) FROM memories WHERE tier='long_lived'")
    if total_count <= CAPACITY_LIMIT * 0.85:
        return  # No eviction needed

    eviction_target = total_count - (CAPACITY_LIMIT * 0.75)

    # Step 3: Score candidates (retrieval-weighted, cluster-granularity)
    candidates = postgres.query("""
        SELECT m.id, m.cluster_id,
               (1.0 / LOG(m.retrieval_count + 2))
               * EXP(-EXTRACT(EPOCH FROM (NOW() - m.last_retrieved_at)) / 1209600.0)
               AS eviction_priority
        FROM memories m
        WHERE m.id NOT IN %(protected)s
          AND m.tier = 'long_lived'
        ORDER BY eviction_priority DESC
    """, protected=protected)

    # Step 4: Cluster-granularity eviction
    evicted_count = 0
    evicted_clusters = set()
    for candidate in candidates:
        if evicted_count >= eviction_target:
            break
        cluster = get_cluster(candidate.cluster_id)
        if cluster is None:
            # Isolated memory: evict individually
            evict(candidate.id)
            evicted_count += 1
        else:
            # Cluster: evict only if entire cluster is cold and unprotected
            if all(m.id not in protected for m in cluster):
                if candidate.cluster_id not in evicted_clusters:
                    for m in cluster:
                        evict(m.id)
                        evicted_count += 1
                    evicted_clusters.add(candidate.cluster_id)

    # Step 5: Cleanup
    redis.delete(f"session:{session_id}:pinned")
```

This algorithm satisfies all five derived properties:
1. Causal protection invariant: protected set is computed before any eviction begins.
2. Correct priority function: retrieval-weighted score, not importance score.
3. Conservative approximation: cluster protection extends to all cluster members.
4. Correct trigger: capacity-ratio based, deferred to session end.
5. Correct eviction unit: cluster-granularity for clustered memories, individual for isolated memories.

## Suggested Follow-ups

- **[IMPLEMENTATION] Q017**: What is the correct semantic threshold for `SAME_FACT_FAMILY` edge creation? The cluster index depends on a similarity threshold for linking memories — too tight and the cluster index is sparse (no protection), too loose and the entire store is one cluster (everything protected, no eviction possible). Derive the threshold from the retrieval pattern data already in Recall's PostgreSQL log.

- **[ADVERSARIAL] Q018**: Does cluster-granularity eviction introduce a new failure mode — "immortal clusters" where a single high-retrieval member protects an entire cluster of stale low-retrieval members indefinitely? Formalize the condition under which a cluster becomes immortal and derive a remedy (e.g., partial cluster eviction for members that have not been retrieved even when their cluster-mates are active).

- **[PHYSICS] Q019**: What is the retrieval SNR degradation curve as a function of store size N? The trigger condition assumes SNR degrades as N grows, but the rate of degradation determines the optimal capacity limit C. Measure: at N = 1K, 10K, 50K, 100K memories in Qdrant, measure the precision@3 of the retrieve endpoint against a held-out ground truth set. This empirically determines the capacity limit where retrieval quality begins to degrade — setting C from measurement rather than from an arbitrary default.

- **[ADJACENT] Q020**: Does the Q002 hippocampal SWR replay mechanism suggest a "relevance replay" pass — not just eviction, but active re-testing of candidate memories against recent task contexts to determine causal relevance before evicting? Rather than approximating causal relevance via retrieval history, the system could explicitly probe: "if I removed M, would the recent session's queries return different top-3 results?" This is more expensive but would replace the approximation with a direct causal test.
