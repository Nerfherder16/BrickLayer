# Finding: Q020 — Adversarial: Retrieval-on-Demand vs. Live Session State for Memory Architecture

**Question**: Is a retrieval-on-demand model (query → retrieve → inject) the right architecture for session memory, or should context be continuously maintained in a live session state?
**Agent**: quantitative-analyst
**Question Type**: ADVERSARIAL
**Verdict**: BREAKTHROUGH
**Severity**: High

---

## Adversarial Analysis: Retrieval-on-Demand vs. Live Session State

---

### Thread A: Defense — Retrieval-on-demand is correct because it is the only architecture that scales, degrades gracefully, and matches the substrate

**Core claim**: Retrieval-on-demand is the correct architecture for Recall specifically because the LLM substrate (Claude) is stateless by design, the memory store is unbounded in principle, and the 500ms latency budget forces a single synchronous query — conditions that uniquely favor pull-based retrieval over push-based session accumulation.

**Evidence**:

**The substrate argument.** Claude has no persistent state between prompts. Each prompt is a fresh inference over a fixed context window. There is no "live session state" in any technical sense at the LLM layer — there is only the context window at inference time. A push-based architecture that continuously maintains session context would need to maintain it *somewhere outside the LLM*, deliver it into the context window on every prompt, and do so within the 500ms budget. This is architecturally equivalent to retrieval-on-demand with a different selection policy. The question is not "retrieval vs. no-retrieval" — it is "what is the selection algorithm for context injection?"

**The scaling argument.** A live session state approach — whether implemented as a growing summary, a context window carry-forward, or a Redis-cached session object — must be bounded. If the session runs for 6 hours across 200 prompts, the live state either truncates (losing early context), compresses (requiring LLM calls for summarization), or bloats the prompt exponentially. Retrieval-on-demand is scale-invariant: the retrieved set is always K=5 regardless of session length, store size, or elapsed time.

**The cold-start / hot-start equivalence argument.** The project brief documents a "cold start problem" as a known weakness. But warm-up for live session state has exactly the same problem: the *first* prompt of a session has no session state either way. The question is which architecture handles subsequent prompts better. For a working memory model to outperform retrieval-on-demand, it must accumulate state *faster* than retrieval-on-demand surfaces it — which requires that within-session context is more valuable than cross-session historical memory. This is true for some tasks (ongoing debugging) but not for others (infrastructure lookup, preference recall).

**Production validation.** Every deployed AI memory system — mem0, Letta/MemGPT, Zep/Graphiti, Microsoft's Mnemis — uses retrieval-on-demand as the primary mechanism. Letta's closest analog to "live session state" is its in-context working memory (a fixed-size scratchpad that can be read/written during a session), but even Letta's archival memory uses retrieval-on-demand via embedding similarity search. The convergence of every production architecture on pull-based retrieval is evidence that the alternatives have real practical costs.

**The graceful degradation argument.** Retrieval-on-demand degrades gracefully when the Recall API is slow or unavailable: the hook returns empty context, and Claude operates without memory (reduced quality, not broken). A live session state model that accumulates context in Redis or in-process state creates a new failure dependency: if the session state store is unavailable or corrupted, the entire session context is lost. Graceful degradation (project brief invariant: "must degrade gracefully when any single backing service is down") favors pull-over-push.

**Best case scenario**: Retrieval-on-demand with K=5, a well-maintained store, and good semantic coverage retrieves the correct 5 infrastructure facts, preferences, and project state memories before each prompt. The LLM sees exactly the facts it needs for this prompt — no noise, no stale state, no growing context overhead. Each prompt is independently optimal.

### Thread B: Attack — Retrieval-on-demand is wrong for session memory because it conflates session memory with long-term memory

**Core claim**: Retrieval-on-demand is the wrong architecture for the within-session memory problem specifically, because it uses a global vector store optimized for cross-session retrieval to handle a local, temporally-ordered, causally-connected memory problem that vector similarity cannot represent — and by doing so, discards the highest-signal memory (what happened 3 prompts ago) in favor of the highest-cosine-similarity memory (what happened 3 months ago).

**Evidence**:

**The working memory / long-term memory distinction.** Human memory architecture has two distinct subsystems that serve different functions: working memory (7±2 items, high bandwidth, short retention, direct access) and long-term memory (unbounded, low bandwidth, accessed via cue-based retrieval). AI cognitive architectures in academic literature (Laird 1987 SOAR, ACT-R, CogArch) and modern AI agent systems have inherited this distinction. Letta's architecture explicitly implements it: in-context core memory (working memory equivalent) versus archival memory (long-term memory, vector retrieval). Recall collapses both functions into a single retrieval-on-demand mechanism — using the long-term memory architecture for both problems.

**The causal continuity problem.** Within a session, prompts are causally connected. Prompt 3 says "now try the alternative approach." Without working memory carrying "what was the original approach from prompt 1?", the retrieval-on-demand system must somehow re-derive that context by querying the vector store with prompt 3's text — which may or may not surface the relevant prior context. Vector similarity between "try the alternative approach" and "the original approach was X" is low if the two are semantically distant. The causally critical context (what came 3 turns earlier) is invisible to the retrieval system because causal adjacency is not encoded in vector similarity.

**Empirical evidence from conversational AI.** The conversation AI literature demonstrates the working memory failure mode concretely. MultiDoc QA, LoCoMo benchmark (long-context conversational memory, ENGRAM paper arXiv:2511.12960), and SSDML all show that models without explicit session state carry-forward fail on multi-turn dependency resolution — questions where the answer depends on earlier turns in the same session. The failure is systematic and predictable: retrieval-on-demand cannot surface "what was said 4 turns ago" reliably because recency is not a primary retrieval signal (Q006 confirms: no production system implements temporal decay in the ANN traversal itself).

**The K=5 tax.** Retrieval-on-demand injects K=5 memories from the global store into every prompt. For a prompt asking about a task underway since the start of the session, 3 of those 5 slots are "wasted" on global long-term context that the LLM already re-derives from the conversation history in its context window. The within-session context that is *not* in the context window (summaries of earlier turns that have scrolled off) is the high-value material — but the retrieval-on-demand system competes for the same K=5 slots with the global store, which is larger and older and thus statistically more likely to surface globally-relevant but session-stale material.

**The session summary loss.** Recall stores session summaries at session-end via `recall-session-summary.js`. This means within-session state is only ever available to *future* sessions via retrieval, not to the *current* session in progress. The most recent session summary in the store is always from the previous session — not the current one. This creates a permanent one-session lag: the system can remember what happened last session, but never what happened earlier in *this* session (beyond what's in the current context window). Live session state would close this gap entirely.

**The temporal signal starvation.** Q006 establishes that every production AI memory system fails to incorporate temporal recency into its ANN retrieval. Q018 confirms that cosine similarity has zero temporal signal. The combined result: retrieval-on-demand applied to within-session memory preferentially surfaces *old* globally-important memories over *recent* session-specific ones, precisely backwards from what working memory should do.

**Failure mode**: In a 4-hour debugging session, Recall's retrieval-on-demand surfaces the same 5 infrastructure memories on every prompt (CasaOS IP, Redis config, Ollama host — globally important, high-importance score, consistently high cosine similarity to any technical query). The session-specific context — "we already ruled out the network timeout hypothesis in prompt 12, the issue is in the Redis eviction policy" — lives only in the scrolling context window. When that context scrolls off, it is gone. The retrieval system cannot surface it because it was never stored as a discrete memory.

### Synthesis

**What Thread A assumes that Thread B exposes**:

Thread A assumes that "session memory" and "long-term memory" are the same retrieval problem — that the mechanism that works for "remember what CasaOS's IP is across sessions" also works for "remember what we decided 10 turns ago in this session." Thread B exposes that these are categorically different memory problems with different time scales, different access patterns, and different relevance criteria. Thread A is correct that retrieval-on-demand is the right architecture for *long-term* cross-session memory. Thread B is correct that retrieval-on-demand is the wrong architecture for *within-session* working memory. The current Recall architecture uses one mechanism for both problems.

Thread A also assumes the relevant alternative to retrieval-on-demand is "continuously maintained live state instead of retrieval" — a wholesale replacement. Thread B does not actually argue for replacing retrieval-on-demand; it argues for adding a second memory tier that handles within-session state. The strawman is architectural extremism: "retrieval everywhere" vs. "live state everywhere." The correct architecture is a two-tier system.

**The hidden assumption both threads share**: That memory architecture is a binary choice between retrieval-on-demand and live session state. This is false. The correct framing is that these serve different memory functions and should coexist, not compete.

**When Thread A is right**:
- Cross-session retrieval: user asks about infrastructure they set up last month; global vector store with K=5 is the correct mechanism
- Store size exceeds what can fit in a context window: selection (retrieval) is necessary; live accumulation cannot work at 100k+ memories
- The prompt is a factual lookup with no causal dependency on earlier prompts in the session
- The session is short (< 5 prompts): retrieval-on-demand adequately covers working memory because the full session fits in the context window anyway

**When Thread B is right**:
- Multi-turn sessions with causal dependencies between turns (debugging chains, iterative development, ongoing planning)
- The context window scrolls past the point where earlier turns are available
- Prompts refer to earlier decisions, ruled-out hypotheses, or accumulated findings in the current session
- The session is long enough (> 10 prompts at substantial context) that early turns fall out of the active context window

**The third option — what Thread B would accept as actually fixing the problem it raised**:

A two-tier memory architecture with explicit separation of working memory (within-session) and long-term memory (cross-session):

**Tier 1: Live Session State (Working Memory)**
- A session-scoped accumulator stored in Redis as a structured JSON object
- Updated after every prompt by a lightweight hook: extract 1-2 key decisions/findings from the current turn, append to the session state object
- Session state is retrieved in full on every prompt (not via similarity search — it is always small)
- Maximum size: 2,000 tokens (approximately 10-15 turns of extracted summaries)
- Eviction: oldest entries drop off when the 2,000 token limit is reached (FIFO with recency preservation)
- At session-end: the accumulated session state becomes the session summary stored to the global long-term store

**Tier 2: Retrieval-on-Demand (Long-Term Memory)**
- Current Recall mechanism, unchanged
- K=5 retrieved from global Qdrant store by semantic similarity
- Continues to handle cross-session context, infrastructure facts, preferences, patterns

**Context injection at prompt time (combined)**:
```
[Session Working Memory — Tier 1]
  Turn 3: Ruled out network timeout hypothesis (Redis responded normally in test)
  Turn 7: Found eviction policy set to noeviction — investigating memory consumption
  Turn 9: Memory usage at 4.2GB; threshold is 4GB; this is the root cause

[Long-Term Memory — Tier 2]
  Redis host: 192.168.50.62, port 6379
  CasaOS Redis container: redis-recall
  Redis eviction policy last set: allkeys-lru (from 3 months ago — may be stale)
  ...
```

The combined context gives the LLM both the active working context (Tier 1) and the persistent background facts (Tier 2) — precisely the two-tier structure that human working memory and long-term memory provide.

**Why this is architecturally sound**: Tier 1 solves exactly the failure mode Thread B identifies (causal continuity, within-session context that scrolls off) without changing Tier 2 (which Thread A correctly defends). The Redis infrastructure is already in the Recall stack — Tier 1 is a write to a session-scoped Redis key, not a new infrastructure component. The cost is one lightweight extraction operation per turn (a few sentences summarizing the current turn's decisions), not a full LLM call.

**Why neither thread could see this alone**: Thread A couldn't see it because Thread A was defending retrieval-on-demand and had no reason to split the problem into two tiers. Thread B couldn't see it because Thread B was attacking retrieval-on-demand and framing live session state as its replacement rather than its complement. Only by running both simultaneously does it become visible that the argument is about scope (within-session vs. cross-session) not mechanism (retrieval vs. accumulation).

## Ideas Extracted

### Idea: two-tier-session-plus-longterm-memory
- **Novelty**: 0.75 — Two-tier memory architectures exist in AI cognitive architectures (Letta's in-context vs. archival, MemGPT's similar split). The specific formulation for Recall — a Redis-accumulated within-session working memory tier operating alongside the existing Qdrant retrieval tier, with extraction done by a hook on every prompt (not by a full LLM call), and with the session state converting directly into the session summary at session-end — is not implemented in any production system at this level of hook integration and cost constraint.
- **Evidence**: 0.72 — Letta's architecture demonstrates two-tier memory is superior to single-tier for long sessions (production system). The Park et al. 2023 Generative Agents paper implements a "stream of consciousness" working memory buffer alongside long-term retrieval (research validation). LoCoMo benchmark results show systematic failures in single-tier retrieval for multi-turn causal dependency resolution (arXiv:2511.12960). Redis is already in Recall's stack with session-scoped keys used in Q016's SRS mechanism.
- **Feasibility**: 0.82 — Implementation path: (1) new hook `session-state-update.js` fires on PostToolUse or UserPromptSubmit (after, not before), extracts 1-2 sentences from Claude's response summarizing decisions made, appends to Redis key `session:{id}:working_memory` as a timestamped array; (2) modify `recall-retrieve.js` to also read `session:{id}:working_memory` from Redis and prepend it to the injected context before the Qdrant results; (3) modify `recall-session-summary.js` to include the accumulated working memory in the session summary stored at session-end. No new infrastructure. No schema changes. Estimated: 2-3 days implementation.
- **Description**: Add a second memory tier to Recall that captures within-session working memory via a lightweight Redis accumulator. Unlike the existing retrieval tier (which queries Qdrant globally), the working memory tier is always-available (no similarity search, no latency beyond a Redis GET), always-relevant (session-scoped, so every entry was created in the current session), and causally ordered (FIFO array preserves the sequence of decisions). The combined context injection becomes: `[Session Working Memory: N turns of extracted decisions] + [Long-Term Memory: K=5 Qdrant results]`. Total context overhead increases by roughly 200-400 tokens per prompt for a session with 5-10 active decisions tracked — well within Claude's context budget. This directly fixes the one-session-lag problem (session summaries only available to future sessions) and the causal continuity failure (decisions from turn 3 still available at turn 30).

### Idea: extraction-quality-determines-tier1-value
- **Novelty**: 0.68 — The specific problem of per-turn extraction quality as the load-bearing variable in a working memory tier is not discussed in any system reviewed. It is framed as an obvious implementation detail, but it is actually the primary failure mode of the two-tier design.
- **Evidence**: 0.65 — A working memory tier is only as good as its extraction step. If the extraction hook produces verbose, low-signal summaries ("the user asked about Redis and Claude responded"), the 2,000-token working memory budget fills quickly with noise. If extraction produces high-signal, decision-specific summaries ("ruled out network timeout; investigating memory eviction policy"), the tier is extremely valuable. The LangChain TimeWeightedVectorStoreRetriever's known failure mode is analogous: the retrieval signal is only as good as the stored documents — garbage in, garbage out. The same principle applies to working memory extraction.
- **Feasibility**: 0.70 — Two implementation approaches: (a) LLM-based extraction: a fast qwen3:14b call with a prompt "Summarize the key decision or finding from this assistant turn in one sentence." Cost: ~300ms per turn, which is acceptable if run asynchronously post-turn. (b) Rule-based extraction: detect sentences containing action verbs + negation patterns ("ruled out", "found", "decided", "is the cause") — no LLM call, ~1ms, but lower quality. Recommend starting with rule-based extraction to validate the tier architecture, then upgrading to LLM extraction once the value is demonstrated.
- **Description**: The session working memory tier's extraction step should be a first-class design concern, not an implementation detail. Define a strict extraction schema: `{"turn": N, "timestamp": ISO8601, "type": "finding|decision|ruling-out|state-change", "content": "one sentence"}`. Enforce the one-sentence constraint. Log extraction quality via a simple heuristic (sentences containing "because", "therefore", "found", "ruled out", "confirmed", "changed" score higher than sentences without). Measure working memory tier hit rate: what fraction of Claude's turns reference content that appeared in working memory vs. content that appeared only in Qdrant results? High working memory hit rate validates the tier; low hit rate indicates extraction quality problems.

### Idea: session-state-cold-start-bridge
- **Novelty**: 0.70 — The specific problem of bridging the gap between "session ended last time" and "session starts fresh this time" using a mechanism that is faster and more causally accurate than Qdrant retrieval has not been directly addressed in any finding.
- **Evidence**: 0.58 — The project brief explicitly lists "cold start problem — new sessions start with zero context about what was recently relevant" as a known weakness. The `rehydrate` operation (session ID → retrieves the memory set relevant at the end of the last session) exists in the Recall API but is not invoked by any hook. The rehydrate mechanism is an underutilized bridge between session state and cold start. If the two-tier working memory tier is implemented, the final state of Tier 1 at session-end becomes a structured, high-quality seed for the next session's context — far better than the generic session summary currently stored.
- **Feasibility**: 0.75 — The `rehydrate` endpoint already exists. The change needed: (1) at session-start, call `rehydrate` with the last session ID (stored in a lightweight `~/.recall/last_session_id` file or Redis key) and inject the rehydrated working memory state as initial Tier 1 context; (2) combine this with the existing Qdrant K=5 retrieval for cross-session long-term facts. The session-start context then has: `[Last Session Working Memory: final state of previous session's Tier 1] + [Long-Term Memory: K=5 Qdrant results]`. This eliminates cold start for recent sessions while maintaining graceful degradation (if no last session ID exists, Tier 1 starts empty as today).
- **Description**: Wire the existing `rehydrate` endpoint into session start. The `recall-retrieve.js` hook (fires on UserPromptSubmit) should check for a last session ID and call `rehydrate` on the first prompt of a new session. The rehydrated state is a structured summary of the last session's working memory — not the full Qdrant retrieval results, but the accumulated decision log from Tier 1. This gives the new session "where we left off" context in a form that is causally ordered (not just cosine-similar), factual (decisions, not paraphrases), and compact (bounded by the working memory tier's 2,000-token limit). The cold start problem becomes a "warm start" where the prior session's final state is the initial state of the new session's working memory.

### Idea: working-memory-ttl-as-task-boundary
- **Novelty**: 0.72 — Using the TTL of the session working memory tier as an implicit task boundary signal — automatically expiring working memory when the task changes, even within a session — is not found in any reviewed system. Letta's core memory is persistent (manually reset); MemGPT's memory is session-scoped. Neither uses task-change detection to reset working memory.
- **Evidence**: 0.52 — Task boundaries within a session are a real phenomenon: a session that starts debugging Redis at prompt 1-15, pivots to implementing a new feature at prompt 16-30, then returns to debugging at prompt 31-40 would benefit from task-scoped working memory rather than session-scoped working memory. The debugging working memory from prompts 1-15 is irrelevant to the feature work at prompts 16-30 (and potentially misleading). TTL-based expiry with task-change detection would automatically clear the stale working memory at the task boundary. Evidence that task changes are detectable: query intent classification already exists in Recall (query_intent.py); a topic model over consecutive prompts can signal topic shift.
- **Description**: Implement working memory as task-scoped rather than session-scoped. Define a "task" as a sequence of causally connected prompts on the same topic (detected by topic similarity between consecutive prompts). When consecutive prompts are topic-dissimilar (cosine similarity between prompt embeddings falls below 0.70), mark the current working memory as "previous task" and start a new working memory tier for the new task. Keep the previous task's working memory as an additional, lower-priority context source labeled `[Previous Task Context]`. This means within a session with multiple task pivots, the context injection has: `[Current Task Working Memory]` + `[Previous Task Context (summarized)]` + `[Long-Term Memory: K=5]`. The task boundary detection reuses the existing embedding infrastructure and requires no new models.

## Falsifying Conditions

### When the two-tier architecture is not worth implementing:

1. **Sessions are always short** (< 5 prompts, always fit in context window): working memory tier adds no value because earlier turns are always available in the context window anyway. For Tim's usage pattern (often multi-hour debugging sessions), this condition is unlikely to be met in practice.

2. **Extraction quality is too low**: if the extraction hook produces low-signal summaries, the working memory tier fills with noise rather than decisions, consuming context budget without value. Validate extraction quality before committing to LLM-based extraction.

3. **Tasks are independent and single-turn**: if each prompt is a fresh factual lookup with no causal dependency on prior prompts, the retrieval-on-demand tier is sufficient and working memory adds no value.

4. **The 500ms latency budget cannot accommodate the additional Redis GET**: in practice, Redis GET is sub-millisecond on a local network. This condition will not be met in Recall's deployment (Redis at 192.168.50.19, same LAN as the Recall API).

### When the defender collapses:

- Sessions exceed ~10 prompts where early context scrolls off the context window (likely in any coding session > 30 minutes)
- Multi-turn debugging where hypotheses are formed, tested, and ruled out across many turns
- The same session includes reference to prior decisions ("go back to the approach we discussed earlier")

### When the attacker collapses:

- Every prompt is independent (factual lookups, one-shot tasks)
- Session length always fits in the context window
- The LLM substrate gains persistent state (if Claude eventually supports session state natively, the external accumulation becomes redundant)

## Suggested Follow-ups

- **[PHYSICS] Measure the within-session context loss rate**: In a 4-hour session, at what prompt number does Claude's context window stop containing the first prompt's content? For Claude's 200k token context window and typical prompt/response sizes of ~2,000 tokens per turn, this is approximately turn 100. For Claude's practical working context (where earlier content still influences output reliably), the effective depth is likely closer to turn 30-50. This measurement directly determines how urgently the working memory tier is needed.

- **[ADVERSARIAL] Q021 follow-up**: The working memory tier's Tier 1 accumulator would naturally surface information at the cluster level (decisions about Redis, decisions about the current feature) rather than at the individual memory level. This connects to Q021 (is importance scoring right at the individual memory level, or should it be cluster-level?). The two-tier finding suggests cluster-level importance is the right model for Tier 1 (working memory is topic-coherent by construction), while individual-level importance remains correct for Tier 2 (cross-session retrieval).

- **[ABSENCE] Are there production systems that implement a Redis-backed within-session working memory tier alongside a vector retrieval long-term tier, with a hook-based extraction step?** Letta's architecture is the closest analog (in-context scratchpad + archival retrieval), but Letta's scratchpad is manually managed by the LLM, not automatically populated by hooks. The specific hook-automated, rule-extracted, bounded-size Redis tier is likely absent from all production systems.

- **[ADJACENT] Cognitive science: what is the relationship between the two-tier architecture here and the psychological construct of "priming"?** In human cognition, recently activated concepts are primed — they are more easily accessed in subsequent retrieval. The session working memory tier is essentially a priming mechanism: it pre-loads the concepts that have been active this session so the LLM encounters them before the global long-term memory results. Does the cognitive science literature on priming suggest an optimal ordering for how the two tiers should be presented in the injected context?

- **[TABOO] Q020-extension — Design a working memory system for an LLM-based AI assistant without using Redis, without using a vector store, and without using any database.** Start from: what is the minimum physical requirement for working memory in a stateless LLM substrate? Derive the mechanism from the substrate constraints.
