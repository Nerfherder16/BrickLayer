## Answers to the Three Sub-Questions

### (a) What happens when `recall:canary_seeds` is empty or absent?

Nothing happens. The endpoint that would read this key does not exist. There is no code path in the deployed Recall service that reads `recall:canary_seeds`, runs paraphrase queries against Qdrant, or evaluates canary pass/fail status.

The Q103 specification described the endpoint as it *should* be built, not as it *is* built. This is the root cause of the false-positive: the monitoring system was designed but never implemented. Absence of the endpoint means the system silently omits canary-based health assessment entirely, not that it returns HEALTHY — it returns no result because no request can be made.

### (b) Does the endpoint return HEALTHY, DEGRADED, or an error?

None of the above. `GET /health/recall-quality` returns **HTTP 404 Not Found** from the FastAPI router. The live system has no route registered at that path. An operator monitoring system that calls this endpoint would receive a 404 error, which is either:
- **Silently swallowed** if the monitoring client treats 404 as "endpoint not yet available" (common in service health polling)
- **Misclassified as HEALTHY** if the monitoring client only fails on 5xx responses
- **Correctly flagged** only if the monitoring client explicitly checks for 404 as a missing-implementation sentinel

In practice, since the Recall dashboard UI queries the deployed endpoints, and `/health/recall-quality` is never called from `health_dashboard.py` or any other UI component, the 404 is never surfaced to the operator. The effective behavior is: **the check is never run, and the operator has no signal that it is missing**.

### (c) Should 0 seeds = DEGRADED?

Yes, but the correct framing is sharper than the question implies. The three distinct failure states must be distinguished:

| State | Redis `recall:canary_seeds` | Correct status |
|---|---|---|
| Seeds inserted, 8+/10 pass | Key exists, seeds present | `HEALTHY` |
| Seeds inserted, <8/10 pass | Key exists, seeds present | `DEGRADED` |
| Seeds inserted, 0/10 pass (all displaced) | Key exists, seeds present but displaced | `DEGRADED` |
| Seeds never inserted (key absent or empty) | Key absent or empty list | **`UNINITIALIZED`** — distinct from DEGRADED |
| Endpoint does not exist | N/A | **`MISSING`** — not a runtime state, a deployment gap |

The Q103 specification collapses `UNINITIALIZED` into `DEGRADED` ("0 seeds registered, 0 tests run, 0 failures" → treat as DEGRADED). This is correct reasoning but incomplete: `UNINITIALIZED` should be a distinct named status so operators can distinguish "the monitoring system has never been bootstrapped" from "the monitoring system ran and found failures." Both are operational problems, but they require different remediation actions (seed insertion vs. hardening/re-seeding).

**The correct behavior when `recall:canary_seeds` is absent or empty**:
```
{
  "status": "UNINITIALIZED",
  "seeds_registered": 0,
  "tests_run": 0,
  "pass_rate": null,
  "message": "No canary seeds registered. Run seed insertion before relying on this health check.",
  "action_required": "POST /health/recall-quality/seed"
}
```
Returning `HEALTHY` for 0 seeds is the false-positive the question anticipates — and it would be the exact defect if the endpoint were implemented naively as `pass_rate = 0/0 = undefined → default to HEALTHY`. This should be an explicit guard in any implementation.

## Qualifying Ideas

### endpoint-nonexistence-vs-false-positive
- (N=0.85, E=0.90, F=0.95) — The Q138 question assumed the endpoint exists and asked what it returns. The actual finding is more severe: the endpoint does not exist at all. This is a category shift from "broken health check" to "missing health check." The risk profile is different — a broken check gives misleading signal; a missing check gives no signal. In a system where HNSW degradation is silent and gradual, no signal is operationally equivalent to perpetual HEALTHY.

### retrieval-quality-proxy-gap
- (N=0.78, E=0.85, F=0.88) — The existing `/health/retrieval-quality` endpoint measures injection frequency as a proxy for retrieval quality. This proxy is blind to HNSW degradation because: (1) degraded graphs still return k results — just wrong ones; (2) injection frequency depends on session activity, not vector-space accuracy; (3) the endpoint does not compare returned results against ground-truth expected matches. Adding a "daily injection count > 0" check to monitoring would give a false sense of coverage. The canary-based endpoint is the only mechanism that directly tests the HNSW graph's ability to find known-correct answers.

### uninitialized-vs-degraded-status-distinction
- (N=0.72, E=0.80, F=0.85) — Health check implementations that treat zero-denominator states as passing are a recurring SRE anti-pattern. The fix is a named `UNINITIALIZED` status distinct from `HEALTHY` and `DEGRADED`. This distinction matters operationally: `DEGRADED` triggers immediate incident response; `UNINITIALIZED` triggers a one-time setup task (seed insertion). Collapsing them forces operators to treat every fresh deployment as a degradation incident. The Q103 spec should be amended to include this three-way status in the HTTP response schema.

## Recommendations

1. **Implement `/health/recall-quality` as specified in Q103** — the endpoint does not exist. The Q103 specification is complete. The implementation gap is the highest-priority action item from this finding. Until this endpoint exists, there is no direct signal for HNSW retrieval accuracy degradation.

2. **Add 0-seeds guard before implementing** — the endpoint must return `status: UNINITIALIZED` (HTTP 200 with body flag, not HTTP 500 or HTTP 200 with `HEALTHY`) when `recall:canary_seeds` is absent or empty. Do not divide by zero and default to HEALTHY.

3. **Do not mistake `/health/retrieval-quality` for canary coverage** — the existing endpoint measures injection frequency, not ANN accuracy. It should remain as a separate signal (session activity health) but must not be treated as a substitute for canary-based recall@k measurement.

4. **Insert sentinel-hardened canary seeds as the first deployment action** — per Q127, 7 of 10 seeds require immediate sentinel-token hardening before insertion. The seeds should be inserted immediately after the endpoint is implemented, not before (inserting seeds with no endpoint to monitor them provides no value and wastes importance budget on non-user memories).

5. **Add monitoring client 404-awareness** — any automated monitoring that calls `/health/recall-quality` should treat HTTP 404 as `MISSING` (critical alert), not as a transient error. This prevents the missing-endpoint failure mode from being silently ignored by polling infrastructure.
