# Finding: Q075 — Tag Normalization Map Governance: Storage, Evolution, Migration, and Transition Risk

**Question**: The tag normalization map (Q066) defined a synonym group structure (~50 raw forms mapping to ~22 canonical tags) and a LLM prompt with canonical menu. But the normalization map is a static artifact. What is the complete governance model for evolving this map over time: where is it stored as the single source of truth (in-code constant, PostgreSQL table, Redis hash, or flat file), how does a new synonym get added when Tim coins a new project-specific term, how does the change propagate to memories already stored with the old raw tag, and what is the risk of a partially-migrated tag vocabulary producing MBC routing degradation during the transition window?
**Agent**: quantitative-analyst
**Question Type**: IMPLEMENTATION
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Database governance (schema migration, controlled vocabulary management), distributed systems (multi-node config sync), information retrieval (vocabulary drift)

## Storage Substrate Analysis

The normalization map must be evaluated against four candidate substrates: in-code constant, PostgreSQL table, Redis hash, and flat file. Each has a distinct failure mode relative to the casaclaude + proxyclaude dual-node requirement.

### In-code constant (current design from Q066)

The Q066 finding embeds the map as a JavaScript object constant in `observe-edit.js`. Both casaclaude and proxyclaude run independent hook processes reading their own local copies of the hook file.

**Critical failure mode**: When Tim coins a new project-specific term (e.g., `familyhub`, `sadie`, `relay-ai`), the map must be updated in the hook file on *both* machines. In practice, the hook directory is git-tracked at `~/.claude/` or the project directory. If the file is pushed to a shared git remote and both machines pull, synchronization is automatic — but only if both machines pull promptly. A window exists between push and pull where casaclaude and proxyclaude apply different normalization rules. New memories written during this window carry different canonical tags for the same raw input.

**Additional failure mode**: Adding a new entry requires a code change (git commit, push, pull on both nodes), which is a higher barrier than the natural workflow of noticing a synonym during a session. Tim is unlikely to context-switch to maintaining hook code mid-session. Synonyms accumulate as discovered items and get added in batches — meaning drift can persist for days or weeks before a correction push.

**Verdict on in-code constant**: Appropriate for the initial map of well-established synonyms. Inadequate as the long-term governance substrate when Tim is actively coining new project terms across multiple machines.

### PostgreSQL table (single source of truth candidate)

A `tag_synonyms` table in the Recall PostgreSQL instance at `192.168.50.19:8200`:

```sql
CREATE TABLE tag_synonyms (
  raw_form    TEXT PRIMARY KEY,
  canonical   TEXT NOT NULL REFERENCES tag_canonical(tag),
  added_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  added_by    TEXT NOT NULL DEFAULT 'human',   -- 'human' | 'auto-detected'
  notes       TEXT
);

CREATE TABLE tag_canonical (
  tag         TEXT PRIMARY KEY,
  description TEXT,
  added_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

Both casaclaude and proxyclaude call the same Recall API endpoint (`192.168.50.19:8200`). If the store hook fetches the normalization map from the API before each write, both nodes always use the current map. This eliminates the split-brain window entirely.

**Latency cost**: A network round-trip to PostgreSQL adds ~2-5ms on LAN for each store event. The observe-edit.js hook already makes an API call to store the memory. Adding a pre-store map fetch doubles the API calls unless the map is cached client-side with a TTL.

**Recommended caching pattern**: The hook caches the normalization map in process memory with a 60-second TTL. On startup, it fetches the current map. On cache miss (TTL expired), it re-fetches. This means both nodes are within 60 seconds of each other after any map update — a transition window of at most 60 seconds, down from the git-based window of hours-to-days.

**Verdict on PostgreSQL table**: Best long-term substrate. Single source of truth. Both nodes converge within one cache TTL. Adds a CRUD API surface for map management (can be exposed in the Recall dashboard). Supports audit trail via `added_at` and `added_by` columns.

### Redis hash

A Redis hash `recall:tag_synonyms` at the same CasaOS host maps raw → canonical as key-value pairs.

**Advantage over PostgreSQL**: Redis GET on a hash is ~0.1ms vs ~2-5ms for a PostgreSQL query. For a map of ~50-200 entries, a single HGETALL is ~0.5ms and returns the full map for client-side caching.

**Disadvantage**: Redis is a cache layer, not a system of record. The Recall stack uses Redis as a hot cache already (per Tim's stack notes). If Redis is flushed or restarted, the normalization map is lost. Redis does not natively support the audit columns (`added_at`, `added_by`) needed for governance. Redis would require a write-through policy where PostgreSQL remains the source of record and Redis is populated on startup and on each map update.

**Verdict on Redis hash**: Appropriate as a *read cache* for the PostgreSQL table, not as the primary substrate. The governance model should be PostgreSQL-primary, Redis-accelerated.

### Flat file (`tag-normalization.json`)

A JSON file committed to the Recall git repo (e.g., `config/tag-normalization.json`) that both hooks load at startup.

**Advantage**: Human-readable, diff-friendly in git history, no API dependency for reads. Tim can edit directly.

**Disadvantage**: Same synchronization problem as the in-code constant — both machines must pull the updated file. Additionally, flat files provide no transactional guarantee: if the file is partially written when the hook reads it (unlikely on a modern OS with atomic renames, but possible), the hook receives a malformed map.

**Verdict on flat file**: Better than in-code constant (separate from code, easier to diff and review) but still subject to the git-pull synchronization lag. Acceptable for an interim improvement over the current design if the PostgreSQL table migration is deferred.

### Recommendation summary

| Substrate | Sync lag | Audit trail | Dashboard-editable | Failure on restart |
|:----------|:--------:|:-----------:|:-----------------:|:------------------:|
| In-code constant | git push/pull (hours–days) | git log | No | N/A |
| Flat file (git) | git push/pull (hours–days) | git log | No | N/A |
| Redis hash | N/A (primary) or ~instant (cache) | No | With custom code | Yes (data loss) |
| **PostgreSQL table** | **~60s (cache TTL)** | **Yes (native)** | **Yes** | **No (durable)** |

PostgreSQL table is the correct long-term substrate. Flat file is acceptable as an interim step if the PostgreSQL migration is deferred.

## Change Propagation Procedure

When Tim coins a new project-specific term (e.g., starting a new project `relay-ai` with tags like `relay`, `elevenlabs`, `twilio`), the propagation procedure is:

### Step 1: Detect the new synonym (observation phase)

The Recall dashboard Query 2 (from Q066 — top 30 tags by frequency) surfaces new tags after a few sessions. Tim notices `relay` (15 occurrences), `relay-ai` (8 occurrences), and `relay_ai` (3 occurrences) all referring to the same project. The synonym group is identified.

Alternatively, the observe-edit.js hook can auto-detect potential synonyms: if a new raw tag has frequency >= 3 and edit-distance <= 2 from an existing canonical tag, flag it in a `potential_synonyms` log table for Tim's review.

### Step 2: Add the canonical tag (if new domain)

If `relay-ai` is a genuinely new canonical tag not covered by existing ones:

```sql
INSERT INTO tag_canonical (tag, description)
VALUES ('relay-ai', 'Relay AI Receptionist project — ElevenLabs + Twilio phone AI');
```

If the new term is a synonym of an existing canonical tag, skip this step.

### Step 3: Add synonym entries

```sql
INSERT INTO tag_synonyms (raw_form, canonical, added_by, notes) VALUES
  ('relay',      'relay-ai', 'human', 'short form of project name'),
  ('relay_ai',   'relay-ai', 'human', 'underscore variant'),
  ('elevenlabs', 'relay-ai', 'human', 'primary vendor for relay-ai voice synthesis'),
  ('twilio',     'relay-ai', 'human', 'telephony provider for relay-ai');
```

Cost: 1-2 minutes of SQL or dashboard interaction.

### Step 4: Update the LLM prompt canonical menu

The canonical tag menu in the observe-edit.js LLM extraction prompt must also include the new tag. With the PostgreSQL governance model, the hook fetches the canonical list from `SELECT tag FROM tag_canonical ORDER BY tag` on startup and cache-miss. No manual prompt update is needed — the prompt is dynamically constructed from the canonical table.

This is a critical design point: if the LLM prompt is hardcoded (current Q066 design), adding a canonical tag requires a code change in addition to the database insert. The prompt should be dynamically composed from the canonical table to keep governance in one place.

### Step 5: Propagate to existing memories (retroactive migration)

See the Retroactive Migration Analysis section below.

### Step 6: Both nodes pick up the change

casaclaude and proxyclaude both re-fetch the normalization map on their next cache-miss cycle (within 60 seconds if using PostgreSQL with 60s TTL). From that point, all new memories are written with the canonical tag.

**Total elapsed time from decision to both nodes updated**: ~2 minutes (SQL insert + cache TTL expiry). Compare to the git-based approach: git commit, push, pull on both machines — realistically 10-30 minutes if Tim is attentive, hours-to-days if not.

## Retroactive Migration Analysis

At 2,000 stored memories, the Q066 corpus normalization SQL UPDATE is the migration mechanism. The question is whether to run it eagerly (immediately after adding new synonyms) or lazily (normalize on read, leaving stored tags as-is).

### Option A: Eager migration (UPDATE on store)

After adding new synonym entries, immediately run:

```sql
UPDATE memories
SET tags = (
  SELECT ARRAY_AGG(DISTINCT
    COALESCE(s.canonical, t.tag)
  )
  FROM UNNEST(tags) AS t(tag)
  LEFT JOIN tag_synonyms s ON s.raw_form = t.tag
)
WHERE tags && (SELECT ARRAY_AGG(raw_form) FROM tag_synonyms WHERE canonical = 'relay-ai');
```

**Cost at 2,000 memories**: The WHERE clause uses a GIN index on `tags[]`. The index lookup for 4 new synonym forms will return O(10-50) matching memories — the typical fraction of a 2,000-memory corpus that contain project-specific tags for a single project. The UPDATE affects at most ~100 rows. Execution time: < 100ms. This is effectively free at Tim's scale.

**For a full corpus re-normalization** (all synonym groups, as in Q066): The WHERE clause with 35+ raw forms will match a larger fraction of the corpus. At 2,000 memories with ~3 tags each = ~6,000 tag assignments. Assuming 30% contain at least one raw form (600 memories affected): execution time < 500ms. Still effectively instant.

**Risk of eager migration**: If a synonym mapping is incorrect (e.g., mapping `ha` to `homeassistant` when a memory actually refers to "high availability"), the migration irreversibly overwrites the original tag. Mitigation: keep original tags in a `tags_raw` column (append-only, never normalized) for auditability.

### Option B: Lazy migration (normalize on read)

Instead of updating stored tags, the MBC routing query joins against the synonym table at read time:

```sql
-- Normalize context_tags at query time
WITH normalized_ctx AS (
  SELECT COALESCE(s.canonical, c.tag) AS tag
  FROM UNNEST(ARRAY[:context_tags]::text[]) AS c(tag)
  LEFT JOIN tag_synonyms s ON s.raw_form = c.tag
),
-- Normalize each memory's tags at read time
normalized_memories AS (
  SELECT
    m.memory_id,
    ARRAY_AGG(DISTINCT COALESCE(s.canonical, t.tag)) AS canonical_tags
  FROM memories m,
       UNNEST(m.tags) AS t(tag)
  LEFT JOIN tag_synonyms s ON s.raw_form = t.tag
  GROUP BY m.memory_id
)
...
```

**Cost of lazy migration**: The join against `tag_synonyms` on every MBC query adds a per-row unnest + join. At 2,000 memories and K=3 MBC results passing the GIN filter, this might involve joining 50-200 candidate rows. Cost: ~5-10ms additional per query. Acceptable.

**Critical flaw of lazy migration**: The GIN index on `tags[]` stores the raw forms. A query for context_tag `relay-ai` will NOT match memories stored with raw tag `relay` unless the WHERE predicate also includes the raw synonyms. Lazy normalization of the query side alone is insufficient — the index lookup operates on stored values, not on the normalized forms. To make lazy migration work correctly, the WHERE predicate must be:

```sql
WHERE m.tags && (
  SELECT ARRAY_AGG(raw_form) FROM tag_synonyms WHERE canonical = ANY(normalized_ctx_array)
  UNION ALL
  SELECT ARRAY_AGG(canonical) FROM tag_synonyms WHERE canonical = ANY(normalized_ctx_array)
)
```

This is significantly more complex and adds a subquery that cannot be fully GIN-indexed. The GIN index efficiency degrades proportionally to the synonym fan-out. For a synonym group with 5 raw forms, the GIN lookup must check 5 distinct tag values.

**Verdict**: Eager migration (Option A) is strongly preferred at Tim's scale. The cost is negligible (< 500ms for full corpus), the GIN index works on canonical forms after migration, and the MBC query remains simple. Lazy migration is architecturally correct but imposes compounding query complexity and defeats the GIN index efficiency that Level 1 routing depends on.

### Recommended migration design

1. **Maintain `tags_raw` column** (append-only): Store the original LLM-extracted tags before normalization. Used for auditing incorrect mappings and re-running normalization if a mapping is corrected.

2. **Run eager migration immediately** after adding new synonyms: The cost at 2,000 memories is ~100ms. Wrap in a transaction with rollback capability.

3. **Normalize at write time** (existing Q066 design): The hook normalizes before writing to PostgreSQL. This ensures `tags[]` always contains canonical forms.

4. **Verify post-migration**: Re-run Q066 Query 1 (effective_V SQL) after each migration batch to confirm effective_V did not increase (which would indicate a new synonym entered without being mapped).

## Transition Window Risk

The transition window is the period between when a new synonym starts appearing in the corpus (new memories being written with the raw form) and when the normalization map is updated and the retroactive migration is complete.

### Window phases and their MBC impact

**Phase 1: New project coined, no map update yet**

Tim starts a new project and the observe-edit.js hook sees new tags (`relay`, `elevenlabs`, `twilio`). These are written to the corpus as raw forms. The LLM extraction prompt does not include them in the canonical menu, so the LLM may generate them inconsistently — sometimes `relay`, sometimes `relay-ai`, sometimes `relay_receptionist`.

During this phase, the MBC Level 1 routing for relay-ai sessions must match on raw tags that are inconsistent. A query context_tag set of `{relay-ai, twilio}` will not match memories stored with tag `relay` — zero intersection. The memories fall to Level 2 (domain match) or Level 3 (frequency fallback).

MBC degradation: Level 1 hit rate for relay-ai sessions drops toward 0% for the new project's memories. Level 2 catches them if `memory_creation_domain` matches, but Level 2 retrieval quality is ~35-50% overlap with optimal results (from Q045 analysis).

**Phase 2: Map updated, migration pending**

Tim inserts the new synonyms into `tag_synonyms`. Both hooks pick up the new map within 60 seconds (PostgreSQL + 60s TTL design). New memories are now written with canonical tags.

But the ~50-100 memories written during Phase 1 still carry raw tags. A query context with `relay-ai` does not intersect `relay` (stored raw). The GIN index still returns no Level 1 matches for those memories.

Duration of Phase 2: The time between Tim noticing the synonym accumulation and running the retroactive UPDATE. With the dashboard surfacing top-tag frequencies, this is detectable after ~5-10 sessions in the new domain.

**Phase 3: Migration running**

The retroactive UPDATE takes < 500ms at 2,000 memories. Effectively instantaneous. PostgreSQL executes it as a single transaction. No read-time inconsistency (transaction isolation).

**Phase 4: Post-migration — fully normalized**

Level 1 routing resumes full effectiveness for relay-ai sessions.

### Quantifying the MBC degradation at Tim's scale

Assumptions:
- Tim's corpus: 2,000 memories total
- New project memories during transition: ~50 (approximately 10 sessions × 5 memories/session)
- Fraction of relay-ai sessions in total session volume: ~20% (active project phase)
- Base Level 1 hit rate for relay-ai sessions without degradation: ~60% (V=22, T=3, C=5 from Q066)
- Level 1 hit rate for relay-ai sessions with raw-tag pollution: ~15% (context_tags include canonical forms; stored memories carry raw forms — few intersections)

**Degraded MBC quality during transition**:
- Level 1 drops from ~60% to ~15% for relay-ai sessions
- Level 2 picks up ~35% (domain match fallback)
- Net effective retrieval quality: ~50% of optimal

**Transition window duration**: With the PostgreSQL + 60s TTL design, Phase 2 is ~60 seconds. Phase 1 is the critical unknown — it depends on how quickly Tim detects vocabulary drift. Without automated detection, Phase 1 can span the entire active project period (weeks).

**The critical gap**: There is no automatic trigger that alerts Tim when a new raw tag is accumulating without a canonical mapping. The Q066 synonym detection SQL (Query 4) must be run manually or on a schedule. Without this automation, Phase 1 is unbounded.

**Quantitative MBC routing degradation summary**:

| Transition phase | Duration (PostgreSQL model) | Level 1 hit rate (relay-ai sessions) | Net quality vs. optimal |
|:-----------------|:---------------------------:|:------------------------------------:|:-----------------------:|
| Phase 1: undetected drift | Days to weeks | ~15% | ~50% |
| Phase 2: map updated, migration pending | ~60 seconds | ~15% (old memories) | ~50% |
| Phase 3: migration running | < 1 second | Transitional | — |
| Phase 4: fully normalized | — | ~60% | 100% |

The dominant risk is Phase 1: silent vocabulary drift with no automated detection. This is the governance gap that makes this finding BREAKTHROUGH rather than merely PROMISING.

## Implementation Recommendation

### Recommended architecture: PostgreSQL primary + Redis cache + automated drift detection

**1. PostgreSQL as source of truth**

Deploy the `tag_canonical` and `tag_synonyms` tables. Expose a REST endpoint in the Recall API:

```
GET  /api/tags/synonyms         → returns full map as JSON object {raw_form: canonical}
GET  /api/tags/canonical        → returns canonical tag list as string[]
POST /api/tags/synonyms         → add new synonym entry (dashboard UI)
POST /api/tags/canonical        → add new canonical tag
```

The Recall dashboard already exists (React + Vite at port 8200). Add a "Tag Vocabulary" panel that displays the canonical list, synonym groups, and a form for adding new entries. This eliminates the SQL-in-terminal workflow for Tim.

**2. Hook-side caching**

In observe-edit.js, cache the normalization map in a module-level variable:

```javascript
let _normMapCache = null;
let _normMapExpiry = 0;
const NORM_MAP_TTL_MS = 60_000;  // 60 seconds

async function getNormalizationMap() {
  const now = Date.now();
  if (_normMapCache && now < _normMapExpiry) return _normMapCache;
  const resp = await fetch('http://192.168.50.19:8200/api/tags/synonyms');
  _normMapCache = await resp.json();
  _normMapExpiry = now + NORM_MAP_TTL_MS;
  return _normMapCache;
}
```

Both casaclaude and proxyclaude call the same endpoint. Both converge on the same map within 60 seconds of any update. No git operations required.

**3. Dynamic LLM prompt construction**

The canonical tag menu in the extraction prompt is fetched from the API alongside the synonym map:

```javascript
async function buildTagPrompt(content) {
  const canonicalTags = await getCanonicalTags(); // cached same way
  return `Extract 2-4 tags from this memory. Use ONLY tags from this list: ${canonicalTags.join(', ')}. ...`;
}
```

When Tim adds a new canonical tag (e.g., `relay-ai`) via the dashboard, the LLM prompt automatically includes it on the next cache cycle. No code changes.

**4. Automated drift detection**

Add a scheduled job (cron or the Recall background task runner) that runs every 24 hours:

```sql
-- Find raw tags appearing >= 3 times with no synonym mapping
SELECT t.tag, COUNT(*) AS freq
FROM memories m, UNNEST(m.tags) AS t(tag)
WHERE t.tag NOT IN (SELECT raw_form FROM tag_synonyms)
  AND t.tag NOT IN (SELECT tag FROM tag_canonical)
GROUP BY t.tag
HAVING COUNT(*) >= 3
ORDER BY freq DESC;
```

If this query returns any rows, create a Recall memory with domain `recall-governance` and tags `['recall', 'vocabulary-drift']` containing the unmapped tags and their frequencies. This memory surfaces during Tim's next Recall session via MBC routing (domain match on `recall`), prompting him to add the mappings.

This closes the Phase 1 detection gap. The transition window shrinks from "weeks of silent drift" to "24 hours maximum before an alert surfaces."

**5. Retroactive migration as a Recall API endpoint**

Expose:
```
POST /api/tags/migrate-synonyms
```

This runs the full corpus normalization UPDATE within a PostgreSQL transaction. Callable from the dashboard "Tag Vocabulary" panel with a single button. Tim does not need to write SQL. The endpoint returns `{memories_updated: N, elapsed_ms: M}`.

**6. Preserve `tags_raw` for auditability**

Add a `tags_raw TEXT[]` column to the `memories` table. The store hook writes the LLM-extracted raw tags to `tags_raw` before normalization, and the normalized canonical tags to `tags`. This allows:
- Auditing incorrect synonym mappings
- Re-running normalization if a mapping is corrected without re-invoking the LLM
- Debugging effective_V drift by comparing `tags_raw` vocabulary size vs. `tags` vocabulary size

Migration: `ALTER TABLE memories ADD COLUMN tags_raw TEXT[];` — non-blocking in PostgreSQL. Back-populate by copying current `tags` to `tags_raw` for existing rows (these are already normalized from the Q066 migration).

## Ideas Extracted

### Idea: vocabulary-drift-detection-as-recall-memory
- **Novelty**: 0.78
- **Evidence**: 0.72
- **Feasibility**: 0.88
- **Description**: A scheduled SQL job (daily) detects unmapped raw tags appearing >= 3 times in the corpus and stores the finding as a Recall memory in the `recall-governance` domain. This memory surfaces organically during Tim's next Recall session via MBC routing (domain match). The governance alert is delivered through the same retrieval channel that governs all other operational context — no separate monitoring infrastructure needed. The key novelty is using Recall's own retrieval pipeline as the delivery mechanism for vocabulary governance alerts, creating a self-monitoring property where the memory system monitors its own tag health.

### Idea: tags-raw-plus-tags-canonical-dual-column-design
- **Novelty**: 0.70
- **Evidence**: 0.65
- **Feasibility**: 0.92
- **Description**: Store two tag arrays per memory: `tags_raw` (LLM output before normalization, append-only, never updated) and `tags` (canonical forms, the normalization target, updated by migration). The GIN index operates on `tags` for query performance. `tags_raw` provides an audit trail and enables normalization correction without re-invoking the LLM. This is the database equivalent of storing both the source and compiled forms — the source is immutable (raw LLM output), the compiled form is the optimization target (canonical tags). Reprocessing the source is always possible if the compiler (normalization map) changes.

### Idea: postgresql-backed-dynamic-llm-prompt-construction
- **Novelty**: 0.73
- **Evidence**: 0.68
- **Feasibility**: 0.85
- **Description**: The LLM extraction prompt's canonical tag menu is constructed dynamically from `SELECT tag FROM tag_canonical ORDER BY tag` rather than hardcoded in the hook. This means adding a new project domain via the dashboard automatically propagates to the extraction prompt within one cache TTL cycle — no code changes, no git operations, no coordinated deploys. The hook's system prompt becomes a function of the database state rather than a code artifact. This reduces the governance surface from two synchronized code files (one per machine) to a single database table. Novel in the context of AI memory systems; similar in principle to data-driven configuration in enterprise systems (e.g., feature flags stored in databases rather than code).

## Key Finding

The normalization map governance reveals a BREAKTHROUGH gap: the current in-code constant design creates a split-brain risk between casaclaude and proxyclaude with a synchronization lag measured in hours-to-days, and — more critically — no automated mechanism to detect when Tim coins a new project-specific term that begins accumulating as an unmapped raw tag in the corpus. This silent vocabulary drift produces a Phase 1 transition window of unbounded duration where MBC Level 1 hit rate for new-project sessions drops from ~60% to ~15%, with Level 2 fallback recovering only ~35-50% of optimal quality. The degradation is invisible without active monitoring because Level 2 still returns results — just less relevant ones.

The correct governance architecture is PostgreSQL as the single source of truth (`tag_canonical` + `tag_synonyms` tables), both hooks fetching via a 60-second TTL cache from the shared Recall API, and a daily SQL job that detects unmapped raw tags and surfaces alerts as Recall memories in the `recall-governance` domain. This reduces the transition window Phase 2 from hours-to-days to ~60 seconds, and Phase 1 from unbounded to 24 hours maximum. At 2,000 memories, eager retroactive migration costs < 500ms and is exposed as a single dashboard button. The GIN index operates on canonical forms after migration, preserving the O(log N + K) Level 1 query efficiency that MBC routing depends on.

The dual-column design (`tags_raw` + `tags`) is the correct auditability pattern: raw LLM output is immutable and reprocessable; canonical tags are the optimization target updated by governance operations. This directly parallels the source/compiled form distinction in build systems and enables correction of incorrect synonym mappings without re-invoking the LLM. Tim adding a new canonical tag for an emerging project takes ~2 minutes via the dashboard and propagates to both nodes and the LLM extraction prompt automatically, compared to ~30 minutes minimum for a git-based update across two machines.
