# Finding: Q043 — Cluster Cohesion Is Free at 500K; Precomputed TTL Opens Architecture Gap

**Question**: What is the maximum latency overhead of the cluster-cohesion computation in Q028's realized-quality-tier3-displacement policy at Tier 3 scale of 500K memories, and at what scale does it require restructuring from on-demand to precomputed?
**Agent**: quantitative-analyst
**Question Type**: PHYSICS
**Verdict**: PROMISING
**Severity**: Medium
**Source field**: Vector database benchmarks (Qdrant), HNSW algorithm complexity analysis, memory bandwidth modeling

---

## Operation Definition

- **Input**: A single eviction candidate memory in Tier 3 (its ID + stored embedding). Tier 3 collection size N = 500,000 vectors at 768 dimensions (qwen3-embedding:0.6b output).
- **Output**: A scalar `cluster_cohesion` score in [0, 1] — the mean pairwise cosine similarity among the candidate's nearest neighbors in Tier 3.
- **Guarantees required**: Score must be computed before the Tier 3 displacement decision completes. The displacement decision fires at session-end, so it is not on the real-time retrieval path. The 500ms p95 retrieval SLA does not apply. The relevant budget is session-end consolidation latency, where a soft ceiling of 60 seconds is used as the restructuring trigger.

---

## Theoretical Minimum

| Resource | Minimum | Calculation |
|---|---|---|
| Latency (HNSW search) | 0.34 ms | Lower bound: 3,359 RAM accesses × 100ns each. This assumes every HNSW traversal hits main memory with no cache reuse — a pessimistic floor. Actual HNSW has partial locality within graph layers, so real floor is lower. |
| Compute (HNSW search) | 0.52 ms | 5.16M FLOPs at 10 GFLOPS sustained (memory-bound, not compute-bound — RAM dominates). |
| Compute (pairwise cosine, 10 members) | 6.9 us | 45 pairs × 768-dim dot product × 2 FLOPs = 69,120 FLOPs at 10 GFLOPS. |
| Network (Docker local) | 0.1 ms | Speed-of-light floor for loopback: sub-millisecond. Docker bridge adds overhead above this. |
| Bandwidth | 30 KB | 10 neighbor vectors × 3,072 bytes (768-dim float32) transferred from Qdrant to caller. |

**Compute floor (pairwise cosine only)**: 6.9 microseconds — this is the irreducible minimum for the arithmetic itself, assuming vectors already in cache.

**Theoretical search floor**: ~0.34 ms — bounded by RAM latency for the 3,359 HNSW distance computations.

## Current Implementation Cost

HNSW parameters: ef_search=16, m=16, N=500,000, dims=768.

Qdrant stores normalized vectors for cosine collections, so each distance computation reduces to a dot product. With `with_vector=True` in the search call, neighbor vectors are returned in the same response — no second round-trip required.

| Step | Cost (p50) | Cost (p99) | Type |
|---|---|---|---|
| HNSW search (3,359 distance evals) | 1.5 ms | 4.0 ms | Physics (RAM-bound graph traversal) |
| Pairwise cosine, 10 members, 45 pairs | 0.007 ms | 0.007 ms | Physics (compute, negligible) |
| Docker bridge network RTT | 0.3 ms | 1.0 ms | Architecture (process boundary) |
| Mean cohesion computation | < 0.001 ms | < 0.001 ms | Implementation (trivial) |
| **Total per cohesion query** | **1.81 ms** | **5.01 ms** | |

**Basis for HNSW estimates**: Qdrant publishes benchmarks showing ~2–5ms at 1M vectors, 768 dims, ef=16 on commodity hardware. At 500K vectors the index is shallower (log₂(500K) = 18.9 layers vs log₂(1M) = 19.9 layers) and the 1.43 GB index fits comfortably in 16–32 GB RAM with better cache utilization. Extrapolated range: 1–3ms p50, 3–5ms p99.

**HNSW complexity derivation**:
```
distance_computations = log(N) × ef_search × m
                      = log(500,000) × 16 × 16
                      = 13.12 × 256
                      = 3,359 computations

FLOPs = 3,359 × 768 × 2 (MACs) = 5.16M FLOPs
RAM accesses = 3,359 × 100ns = 0.34ms (memory-bandwidth floor)
```

**Pairwise cosine computation**:
```
pairs = C(10, 2) = 45
FLOPs = 45 × 768 × 2 = 69,120 FLOPs
time  = 69,120 / (10 × 10^9) = 6.9 microseconds
```

## Crossover Analysis

**Question**: At what Tier 3 size does on-demand cohesion computation × N_archive exceed 60 seconds?

**At N_tier3 = 500K, ef=16** (the target scale):
```
p99 per cohesion query = 5.01 ms
N_archive to hit 60s  = 60,000 ms / 5.01 ms = 11,983 memories/session
```

11,983 archived memories per session is unreachable for a personal AI memory system. A productive session generates tens to low-hundreds of memories. The 60-second threshold is not reachable at 500K scale regardless of session size.

**Generalized crossover across scales** (N_archive = 50, worst case):

| N_tier3 | p99 per query | Batch total (N=50) |
|---|---|---|
| 100,000 | 4.5 ms | 226 ms (0.2s) |
| 500,000 | 5.0 ms | 250 ms (0.3s) |
| 1,000,000 | 5.2 ms | 261 ms (0.3s) |
| 5,000,000 | 5.7 ms | 285 ms (0.3s) |
| 10,000,000 | 5.9 ms | 296 ms (0.3s) |

The batch total is nearly flat with respect to N_tier3. This is because HNSW latency grows as O(log N) — a 20× increase in collection size (500K → 10M) adds only 18% to per-query latency. The 60-second ceiling is never approached.

**Algebraic crossover for N_archive=50**:
```
solve: 50 × (4.0 × log(N) / log(500K) + 1.007) = 60,000
→ log(N) = (1199/4.0) × log(500K) = 3,932
→ N = e^3932 ≈ 10^1,708
```

This is not a physical quantity. The on-demand policy has no crossover threshold under realistic operational conditions. The 60-second budget is never binding.

**True restructuring driver is different**: the 60-second rule was the wrong framing. The actual reason to precompute cohesion scores is not latency but **consistency** — if the eviction policy is called thousands of times during a nightly consolidation sweep rather than just at session-end, on-demand cohesion queries would compete with live Qdrant retrieval queries, causing tail latency spikes on the retrieval path. The restructuring trigger is query contention, not absolute latency.

## Precomputed Alternative

**Nightly batch: recompute all 500K cohesion scores from scratch**

Cost per memory = HNSW search (1.5ms p50) + pairwise compute (0.007ms) + PostgreSQL UPDATE (0.1ms) = 1.61 ms

```
Sequential: 500,000 × 1.61ms = 805s = 13.4 minutes
4 threads + Qdrant batch search (5× amortization of RTT overhead): ~0.5 minutes
```

Fits comfortably within a 24-hour TTL window. The nightly batch completes in under 1 minute with parallel Qdrant batch search.

**Incremental update at session-end** (preferred path):

At session-end, only the newly archived memories and their existing Tier 3 neighbors have stale cohesion scores. Neighbor sets are small (cluster size ≈ 10). For N_archive = 50 new memories, at most 50 × 10 = 500 existing memories need cohesion recomputation.

```
500 affected memories × 1.61ms = 805ms = 0.8 seconds
```

0.8 seconds is acceptable at session-end (not on the retrieval path). This makes full nightly recomputation unnecessary — incremental updates at session-end keep scores fresh with a worst-case lag of one session boundary.

**Staleness impact**: A 24-hour stale cohesion score affects eviction decisions when Tier 3 is at capacity. For a personal memory system, Tier 3 capacity pressure is rare enough that stale scores have negligible quality impact. Incremental session-end updates provide sufficient freshness.

## Gap Analysis

**Physics gaps (unavoidable)**:
- HNSW graph traversal memory latency: 0.34ms floor from ~3,359 random RAM accesses. Cannot be eliminated without changing the index structure (e.g., switching to IVF which trades recall for sequential access, but recall degradation is unacceptable for cohesion scoring).
- Pairwise cosine FLOPs: 69,120 FLOPs is the irreducible arithmetic minimum for 10-member cluster cosine matrix. Not a practical concern at 6.9 microseconds.

**Architecture gaps (design choices)**:
- **Process boundary (Docker bridge)**: 0.3–1.0ms RTT per cohesion query is pure overhead from the FastAPI→Qdrant network hop. This is an architecture gap: if cohesion were computed inside the Qdrant process (via a custom scoring plugin), the RTT would drop to near-zero. However, Qdrant does not expose custom scoring hooks, so this gap is currently fixed by the stack choice.
- **On-demand vs precomputed**: Cohesion is recomputed from scratch on every eviction decision. The precomputed alternative caches cohesion in PostgreSQL and updates incrementally at session-end. This eliminates all Qdrant queries from the displacement hot path, reducing eviction decision cost from ~5ms to a single PostgreSQL read (~0.1ms). This is the primary exploitable architecture gap.

**Implementation gaps**:
- None significant at this scale. The pairwise computation is trivially fast and requires no optimization.

## Ideas Extracted

### Idea: incremental-cohesion-cache-session-boundary

- **Novelty**: 0.80 — Precomputed cluster cohesion scores with session-boundary incremental refresh is not implemented in any known production AI memory system. The pattern is common in recommendation systems (precomputed item-item similarity matrices refreshed on schedule), but the specific application to archive-tier eviction scoring with session-event-driven incremental refresh is novel. Existing vector DB eviction policies (Qdrant, Weaviate, Milvus) use no cluster-awareness at all.
- **Evidence**: 0.85 — The calculation is grounded in published Qdrant benchmarks (1M vectors, 768 dims, ef=16 → 2–5ms). The HNSW complexity O(log N × ef × m) is peer-reviewed (Malkov & Yashunin 2018). Pairwise cosine FLOPs are exact. The incremental update cost derivation (session-end: ~800ms for 500 affected memories) is back-of-envelope but conservative. Evidence quality is high because all inputs are from established benchmarks.
- **Feasibility**: 0.85 — Requires: (1) a `cohesion_score FLOAT` column in PostgreSQL on the memories table (schema migration, trivial); (2) a session-end hook that computes cohesion for newly archived memories and their neighbors (attaches to the existing `recall-session-summary.js` Stop hook, which already fires `POST /consolidate`); (3) modification to the Tier 3 displacement check to read `cohesion_score` from PostgreSQL rather than querying Qdrant on-demand. No new infrastructure. No new services. Implementation: 1–2 weeks.
- **Description**: Cache each Tier 3 memory's `cluster_cohesion` score in a PostgreSQL column, computed and refreshed at session-end rather than on-demand at eviction time. At session-end, the consolidation job identifies the N_archive newly archived memories and their Tier 3 neighbors (≤ 500 vectors total), runs one batched Qdrant search to compute their cohesion scores, and writes results to PostgreSQL. The Tier 3 displacement check then reads `cohesion_score` from a single indexed PostgreSQL row (~0.1ms) rather than issuing a Qdrant search (~5ms p99). This reduces eviction decision latency by 50× and eliminates Qdrant query contention during any future bulk eviction sweeps. The precomputed score is never more than one session boundary stale — acceptable for an archive-tier eviction policy where the quality signal changes slowly.

---

## Key Finding

**The cohesion computation is not a performance problem at 500K scale.** At p99, one cohesion query costs 5ms. A full session-end batch of 50 archived memories costs 250ms — well within any reasonable consolidation budget. The 60-second restructuring threshold is not reachable under realistic Recall workloads; the crossover requires 10^1,708 Tier 3 memories, which is not a physical quantity.

**The correct reason to precompute** is not latency but eviction path simplicity and query contention avoidance. The precomputed alternative costs ~800ms at session-end for incremental refresh and eliminates Qdrant queries entirely from the eviction decision path. This is worthwhile even though on-demand is fast enough — it decouples the eviction policy's correctness from Qdrant availability and keeps the displacement hot path on PostgreSQL only.

**Implementation recommendation**: implement the `incremental-cohesion-cache-session-boundary` idea from day one. The overhead of on-demand cohesion queries at 500K is benign, but precomputed scores are architecturally cleaner, require no new infrastructure, and future-proof against higher-frequency eviction patterns. The session-end incremental refresh adds ~0.8 seconds to consolidation — below any perceptible threshold.
