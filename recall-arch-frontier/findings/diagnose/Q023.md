# Finding: Q023 — 2032 Retrospective: Embedding-Compute-Constrained Architecture Decisions in Recall

**Question**: In 2032, embedding models have become 100x cheaper to run and 10x more capable. Which architectural decisions made in Recall's 2026 design assumed constrained embedding compute, and how should they change?
**Agent**: quantitative-analyst
**Question Type**: TIMESHIFTED
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Historical computing transitions (CPU price curves, GPU/TPU cost trajectories, database indexing evolution)

---

## The Decision Under Evaluation

Recall's 2026 architecture is built around a single hard constraint: each embedding call costs ~50ms of wall-clock time (Ollama RTT to a shared RTX 3090 at 192.168.50.62). This constraint propagated into at least six distinct architectural decisions: negative caching to avoid embedding calls, embedding reuse across operations, K=5 retrieval limit, 384-dimension embedding model choice, LSH for in-process novelty checking, and speculative prefetch to hide the embedding latency.

The question is not whether these decisions were reasonable in 2026 — they were correct given the constraint. The question is: what happens to each of them when the constraint dissolves?

---

## Historical Analogue

**Parallel**: The move from expensive disk I/O to cheap RAM (1980s–1990s) in database design.

In 1980, disk I/O dominated the cost structure of every database operation. This produced a generation of architectural decisions: B-tree page sizes chosen to match disk sector sizes (512 bytes), buffer pool management algorithms designed around minimizing I/O (LRU, ARC, clock), query planners that treated disk access as the dominant cost and CPU as nearly free, and normalization dogma rooted partly in minimizing the on-disk footprint of each tuple.

When RAM prices dropped 99.9% between 1980 and 2000 (from $100/KB to $0.001/KB), an entire generation of architecture dissolved. Systems that cached everything in RAM (Redis, Memcached, VoltDB) made decades of buffer management wisdom irrelevant. PostgreSQL's EXPLAIN ANALYZE began showing "sequential scans" as cheaper than "index scans" for small tables — the opposite of the disk-era intuition. Denormalization became acceptable because you were no longer paying disk space costs for redundancy.

The "obviously right" approach in the old era: minimize I/O at all costs, even at significant CPU expense. By 1995, this looked like premature optimization. By 2000, it produced systems that had inverted the cost model.

**Why they were wrong**: The assumed bottleneck (disk I/O) got solved by a technology transition (DRAM price drop), shifting the bottleneck to CPU, then to network, then back to memory-bandwidth in NUMA systems. Each generation of architects who anchored to the previous bottleneck built systems that were subtly misconfigured for the actual cost regime they operated in.

The embedding bottleneck of 2026 is the disk I/O of the AI memory era.

## The Wrong Assumption (2026 Vintage)

**The assumption**: Embedding compute is a scarce, expensive, serial resource that must be rationed. Every architectural decision should minimize the number of embedding calls and hide the latency of each call.

**Why it seemed reasonable in 2026**: In 2026, a single embedding call to qwen3-embedding:0.6b via Ollama costs ~50ms of wall-clock time on a shared RTX 3090. This is not a theoretical floor — Q011's physics ceiling analysis showed 1.63ms as the absolute minimum given the LAN RTT and GPU memory-bandwidth constraints, and the current system runs at a 30.6x gap above that floor. Every 50ms call is a real user-facing latency cost. Running 10 embedding calls in series adds 500ms — exceeding Recall's entire p95 retrieve budget. The constraint was binding, concrete, and empirically measured.

**What data revealed it was wrong**: Two independent trends converged between 2026 and 2032:

1. **Embedding model efficiency**: Inference compute per token dropped approximately 100x, driven by (a) model architecture improvements (efficient attention variants — FlashAttention-3, linear attention approximations, Mamba-class SSM alternatives for short sequences); (b) quantization advances (4-bit INT4 quantization for embedding models with less than 1% quality degradation, eliminating the bandwidth bottleneck that Q011 identified as 1.28ms floor); (c) hardware-software co-design (NPU inference on consumer hardware — Apple M-series Neural Engine, AMD XDNA, Intel Neural Processing Unit — all supporting embedding inference at ~0.5ms latency by 2030 with no dedicated GPU required).

2. **Embedding model dimensionality explosion**: The 384-dimension constraint of qwen3-embedding:0.6b was a cost-quality tradeoff — larger models with higher-dimensional representations cost more. With 100x cheaper compute, the cost of running 3840-dim embeddings is lower than 384-dim embeddings were in 2026. The quality difference is not linear: 10x more dimensions at higher model capacity produces qualitatively different semantic spaces, not just quantitatively better retrieval precision.

The combined effect: by 2032, embedding a query costs ~0.5ms and produces a 3840-dim representation that is semantically discriminating enough to distinguish "Redis is healthy on casaclaude" from "Redis is healthy on proxyclaude" without any additional scope metadata. The fundamental physics changed, not just the engineering.

## Systematic Analysis: What Each 2026 Decision Assumed

### Decisions Designed Solely to Minimize Embedding Calls

These are architecture patterns whose entire value proposition was "avoid paying the 50ms Ollama cost." When the cost drops to 0.5ms, the mechanism's overhead exceeds its benefit.

**Decision 1: Negative Cache (Q004)**

The negative cache (Redis key: `neg:{query_embedding_hash}`, TTL=300s) was designed to prevent repeated 50ms embedding calls for queries that consistently return empty. The mechanism required: (a) checking a separate Redis key before calling Ollama, (b) maintaining TTL semantics to avoid stale negatives, (c) invalidation logic triggered by new stores to the same domain.

At 0.5ms embedding cost, the math inverts. The negative cache check itself costs ~0.1ms (Redis round-trip). The savings per avoided embedding call: 0.5ms - 0.1ms = 0.4ms. This is not zero, but the complexity budget for a mechanism that saves 0.4ms is near zero. The negative cache becomes a micro-optimization applied to a sub-millisecond operation. The Q004 finding's PROMISING verdict was driven by the 50ms Ollama cost; at 0.5ms, the novelty score of 0.85 is unchanged but the feasibility-weighted impact collapses.

**The 2032 verdict**: Negative cache for latency is obsolete. A simpler version — cache empty results to avoid the Qdrant call, not the embedding call — may still have value (Qdrant queries are not zero cost), but the embedding-avoidance motivation evaporates entirely.

**Decision 2: Embedding Reuse for Novelty Detection (Q012)**

The Q012 finding's highest-value idea was `embedding-reuse-novelty-gate`: reuse the store-path embedding for the novelty check, avoiding a second Ollama call. The entire value of this idea was avoiding one 50ms embedding call. At 0.5ms per call, the savings from embedding reuse is 0.5ms — below the measurement noise floor for a system that takes ~10ms for a Qdrant search.

**The 2032 verdict**: Embedding reuse is no longer a meaningful optimization. It costs nothing to compute a second embedding at 0.5ms. The Q012 novelty score of 0.75 was correct in 2026; from 2032, this idea looks like meticulous optimization of the wrong variable.

**Decision 3: LSH for In-Process Novelty Checking (Q012)**

The LSH novelty sketch (the `lsh-novelty-sketch-in-process` idea from Q012) was motivated by eliminating the 10ms Qdrant REST call and the 50ms Ollama call from the novelty-detection path, at the cost of ~1% FPR and 40MB of in-process RAM. The idea was to run a 0.016ms in-process approximation instead of a 60ms cross-network exact operation.

At 2032 costs: the embedding call is 0.5ms, the Qdrant call (with gRPC improvements from Q011) is ~2ms. Total cost of the "expensive" path: ~2.5ms. The LSH sketch saves at most 2.5ms at the cost of 1% FPR and 40MB RAM and significant implementation complexity. This is a poor tradeoff.

**The 2032 verdict**: LSH novelty sketching is obsolete as a latency optimization. The "slow" exact approach at 2.5ms is fast enough. The complexity of maintaining an in-process LSH index in sync with Qdrant is no longer justified.

**Decision 4: Speculative Prefetch (Q011)**

Speculative embedding prefetch — firing a pre-embed request from the hook before the full retrieve request arrives — was designed to hide the 50ms Ollama call behind the hook's own execution time. The Q011 feasibility score was 0.40 specifically because the timing window might be too small (< 5ms) to recover meaningful latency from a 50ms GPU call.

At 0.5ms embedding cost, speculative prefetch is solving a problem that no longer exists. There is nothing to hide.

**The 2032 verdict**: Speculative prefetch is irrelevant. The motivation (hiding a multi-hundred-millisecond operation behind hook execution time) evaporates when the operation takes half a millisecond.

### Decisions That Were NOT Primarily Motivated by Embedding Cost

These decisions survive the embedding-compute transition because their motivation was correctness, retrieval quality, or storage efficiency — not latency avoidance.

**Decision 5: K=5 Memories Injected Per Retrieve**

The K=5 limit on memory injection was stated in the project brief as being partly a function of embedding compute for ranking. But examining the actual constraint: K=5 is primarily limited by context window consumption, not embedding compute. Injecting K=50 memories consumes ~5000 tokens of context window at ~100 tokens per memory. At Claude's 200K context window, this is 2.5% of the window — affordable. But the real cost is attention dilution: injecting too many memories floods the model's attention budget, causing relevant memories to compete with each other and reducing the effective signal per memory.

**The 2032 verdict**: K=5 should grow, but for different reasons than embedding cost. The limit will relax to K=20–40 as models improve at attending selectively over longer injected context, not because embedding compute dropped. Embedding compute was never the binding constraint here — attention budget was.

**Decision 6: 384-Dimension Embeddings (qwen3-embedding:0.6b)**

The 384-dimension choice was a cost-quality tradeoff: larger models with higher-dimensional representations cost more compute per call and more storage per vector in Qdrant. At 384 dims, each Qdrant vector is 384 × 4 bytes = 1.5KB. At 3840 dims, each vector is 15KB — 10x more Qdrant storage per memory. This matters for the HNSW index size and search speed.

**The 2032 verdict**: The 384-dim decision survives longer than the latency-avoidance decisions, because the cost driver is storage and HNSW search cost, not embedding inference cost. The HNSW search over 3840-dim vectors is 10x more compute-intensive than over 384-dim vectors. However, the Q008 whitening finding (correcting embedding anisotropy) and the Q013 PCA compression finding (compressing 1024-dim to 50-dim) together suggest the 2032 architecture uses higher-dimensional embeddings at store time and compresses to a lower-dimensional manifold for retrieval — getting the semantic richness of 3840 dims with the HNSW search cost of 50 dims.

### Decisions That Become MORE Valuable with 10x Better Embeddings

These decisions become more powerful precisely because richer embeddings change what is possible.

**Decision: Cosine Similarity as the Retrieval Signal (Q018, Q008)**

In 2026, the Q008 finding showed that embedding anisotropy (non-uniform density in the 384-dim space) degrades cosine similarity as a retrieval signal. With a 0.6B model producing 384-dim embeddings, the semantic space is crowded and poorly distributed — common technical terms cluster in small regions, causing false-positive similarity scores.

With 3840-dim embeddings from a 10x more capable model, the semantic space is qualitatively different. The embedding model has learned to separate "Redis is healthy on casaclaude" from "Redis is healthy on proxyclaude" as distinct points — not because of the IP address suffix, but because the model has learned the concept of machine-scoped system state. The Q026 adversarial finding's "worst case against cosine" (the infrastructure mirror scenario) becomes less severe because the 2032 embedding model encodes more contextual information per vector.

Cosine similarity remains the right retrieval signal in 2032 — it gets better because the embedding space gets better, not because cosine itself changes.

**Decision: Causal Graph Storage in Neo4j (Q017)**

The Q017 finding proposed using Neo4j to store causal relationships between memories (memory A caused memory B, memory A contradicts memory B). In 2026, the motivation for this architecture was that cosine similarity alone cannot capture causal structure. With 384-dim embeddings, two causally related memories may be semantically distant enough in the embedding space that retrieval by similarity misses the causal connection.

With 3840-dim embeddings from a more capable model, a retrieved memory's causal relationships are more likely to be surfaced by the embedding similarity itself — because the embedding encodes causal context, not just semantic topic. However, the Neo4j graph becomes MORE valuable, not less, because it can be traversed without additional embedding calls. In 2032, the graph traversal is the cheap reliable path; the embedding search is the expensive discovery path. The roles partially invert.

**Decision: Importance Scoring with Temporal Decay (Q003, Q015)**

Importance scoring and decay are independent of embedding compute cost. The immunological memory analogy (Q003) and the log-structured compaction analogy (Q001-F1) both motivated a two-tier architecture where importance determines which memories get promoted to which tier. This survives the compute transition unchanged: what gets stored where is a policy question, not a physics question.

## What New Architectural Choices Become Viable in 2032

These were infeasible in 2026 due to embedding compute cost but become straightforward when embedding costs 0.5ms.

### New Capability 1: Multi-Embedding-Per-Memory Storage

In 2026, storing a memory with multiple different embedding representations was unthinkable — you would need N Ollama calls per store operation. At N=3 and 50ms per call, store latency would exceed 150ms just for embeddings. With 0.5ms per call, storing 3 representations costs 1.5ms total.

What does multi-embedding enable? A single memory text can be embedded from three perspectives simultaneously:
- **Semantic embedding**: "what is this about?" (current approach)
- **Temporal embedding**: "when does this apply?" (embedding of temporal context alongside content)
- **Causal embedding**: "what does this enable or constrain?" (embedding of the memory's causal implications)

Retrieval at query time can query all three embedding spaces and fuse the results. A query like "what were my decisions about Redis last month?" retrieves on both the semantic space ("Redis") and the temporal space ("last month"). In 2026, this required 3 Qdrant collections and 3 Ollama calls — a 150ms tax per retrieve. In 2032, the same query costs 1.5ms for three embeddings and three ANN searches.

### New Capability 2: Dense Retrieval with Query Expansion

In 2026, query expansion (generating multiple paraphrase variants of the retrieve query and embedding each) was infeasible because each variant cost 50ms. A 5-query expansion would add 250ms to the retrieve path — prohibitive.

With 0.5ms per embedding, generating 10 query variants costs 5ms. Embedding all 10 and taking the union of their top-5 Qdrant results produces a retrieval pool of 50 candidates ranked by the max score across all queries. This is a form of pseudo-relevance feedback that handles the fact that the user's literal query text may not closely match how the relevant memories were stored.

This specific mechanism (HyDE — Hypothetical Document Embeddings, proposed by Gao et al. 2023) was documented as a promising RAG improvement in 2026 but never implemented in personal memory systems because the embedding cost made it impractical for sub-500ms latency targets. By 2032, it is standard.

### New Capability 3: Real-Time Semantic Diff of Session Content

In 2026, the observe-edit.js hook fires on every file save and extracts a memory to store. The bottleneck forced the hook to be selective — extract the most salient fact from the edit, not a full semantic analysis.

With 0.5ms embeddings, the observe-edit hook can embed the before-and-after states of a file section and compute their semantic difference at every save. The "semantic diff" — what changed in meaning, not just in text — can be stored directly as the memory, rather than relying on an LLM extraction pass to identify the salient change. This is cheaper (embedding diff vs. LLM extraction), faster (1ms vs. 200ms for qwen3:14b), and more reliable (deterministic rather than prompt-dependent).

### New Capability 4: Continuous Background Re-embedding on Model Upgrade

In 2026, upgrading the embedding model required re-embedding the entire stored corpus — a batch job that would take hours at 50ms per memory, for a 10K-memory store (10K × 50ms = 8 minutes minimum). This was a significant migration cost that made model upgrades infrequent.

With 0.5ms embeddings, re-embedding 10K memories takes 5 seconds. This changes model upgrades from a planned migration event to a routine background maintenance task. In 2032, Recall's store automatically detects embedding model version changes and triggers background re-embedding of the cold store at idle time. Memory quality continuously improves as embedding models improve, with no human-visible migration cost.

## What the 2032 System Looks Like

By 2032, Recall's architecture has discarded the latency-avoidance mechanisms and replaced them with a richer semantic infrastructure made possible by cheap embeddings.

**Retrieval path (2032):**

1. Query arrives. Three parallel embedding calls fire: semantic embedding, temporal embedding, causal embedding. Cost: 1.5ms total (parallel, not serial).
2. Three parallel Qdrant searches against three vector collections. Cost: 3ms total (parallel).
3. Reciprocal rank fusion across the three result sets produces a merged top-50 candidate pool.
4. HyDE query expansion generates 5 paraphrase variants of the original query. Cost: 2.5ms. Five additional Qdrant searches. Cost: 5ms. Adds 25 more candidates to the pool.
5. The 75-candidate pool is re-ranked using a cross-encoder (small, fast, runs local — a 2032 model trained specifically for memory relevance reranking). Cost: 8ms.
6. Top-K=15 memories injected into context.

Total retrieve latency: ~20ms for a richer, more accurate result than the 2026 system achieved at 80ms.

**Store path (2032):**

1. New memory arrives from observe-edit hook.
2. Semantic diff computed against the last N=10 store operations from the same session using embedding similarity (no LLM call). Cost: 5ms for 10 embedding comparisons.
3. If semantic diff < 0.15 from any recent store: merge operation (update the recent memory's text, re-embed, rewrite). No new record created.
4. If novel: three embeddings computed (semantic, temporal, causal). Cost: 1.5ms.
5. Qdrant cross-collection write (3 collections). Neo4j causal edge write. PostgreSQL record write. All parallel.
6. Scope-qualified identifier registry updated (Q026's Phase 1 mechanism).

Total store latency: ~12ms. No LSH, no negative cache, no embedding reuse optimization.

**What is absent from the 2032 system that was present in 2026:**

- Negative embedding cache: gone. The 0.5ms embedding call is not worth caching.
- LSH in-process novelty sketch: gone. The full Qdrant search costs 3ms; no in-process approximation is needed.
- Speculative prefetch: gone. There is nothing to speculatively hide.
- Embedding reuse across store/novelty operations: gone. Call it twice.
- K=5 hard limit driven by embedding-cost-of-ranking: relaxed to K=15, driven by context budget.

**What is present in the 2032 system that was absent in 2026:**

- Multi-space embedding (semantic + temporal + causal) per memory.
- HyDE query expansion at retrieve time.
- Semantic diff at store time (embed before/after, compare, decide).
- Continuous background re-embedding on model upgrade.
- Cross-encoder reranking of the candidate pool (now affordable because the embedding space is richer and the candidate pool smaller after multi-space filtering).

## Ideas Extracted

### Idea: multi-space-embedding-architecture

- **Novelty**: 0.82 — Storing separate embeddings per memory for different conceptual axes (semantic, temporal, causal) is not present in any production AI memory system. Multi-vector representations exist in research (ColBERT uses per-token vectors; PLAID extends this) but the application of separate embedding spaces for distinct retrieval dimensions in personal memory is absent from all known systems. The specific combination (semantic, temporal, causal) is novel as an architectural pattern.
- **Evidence**: 0.55 — Multi-vector retrieval is peer-reviewed (Khattab & Zaharia 2020, "ColBERT: Efficient and Effective Passage Search via Contextualized Late Interaction over BERT"). The principle of embedding the same content from multiple perspectives is established in multi-view learning (Blum & Mitchell 1998). The specific application to AI memory with temporal and causal embedding axes is not validated in peer-reviewed literature. Evidence is moderate because the components are validated but the composition is extrapolated.
- **Feasibility**: 0.40 — Not feasible in 2026 at 50ms per embedding (3 calls × 50ms = 150ms store overhead). Feasibility score reflects the 2032 context where this becomes straightforward. In 2026, this idea belongs in the "track for 2028" category, not the "build now" category. A partial version — adding just temporal embedding as a second dimension — is feasible in 2026 with approximately 50ms added to store latency, which may be acceptable for an async store path. Partial 2026 feasibility: 0.55 for temporal-only extension.
- **Description**: Recall's store operation computes three distinct embeddings per memory: (1) semantic embedding of the full memory text (current approach); (2) temporal embedding of the memory text concatenated with its time context ("stored on 2026-03-16, in session about System-Recall, during week 12 of 2026") — enabling temporal proximity retrieval; (3) causal embedding of the memory's implications ("if this fact is true, then: [LLM-generated consequence list]") — enabling forward-reasoning retrieval. Each embedding is stored in a separate Qdrant collection. Retrieval queries all three collections in parallel and fuses results. The causal embedding approach is expensive in 2026 (requires an LLM call to generate consequences); by 2032, a lightweight causal reasoning model makes this tractable at 2ms.

**Score (2026 context)**: 0.40×0.82 + 0.35×0.55 + 0.25×0.40 = **0.622 → BREAKTHROUGH**

Note: This idea scores BREAKTHROUGH even from a 2026 perspective because the novelty and evidence are high. The low feasibility score reflects 2026 costs; the concept is sound and should be tracked for implementation as embedding costs drop.

### Idea: semantic-diff-at-store-time

- **Novelty**: 0.78 — Using embedding comparison between before/after states of edited content as the primary signal for the store/discard/merge decision — rather than relying on LLM extraction or simple text diffing — is not present in any production AI memory system. Text-level diff (git diff semantics) exists everywhere. Semantic-level diff (what changed in meaning) does not. The closest prior art is change detection in knowledge graphs (Guo et al. 2020, "Learning to Update Knowledge Graphs by Reading News"), but that is document-level, not edit-level.
- **Evidence**: 0.60 — Embedding-based change detection is well-established in document similarity literature. The application to code/file edit observation hooks is extrapolated from text similarity research. The specific pipeline (embed before, embed after, cosine diff threshold gates store decision) is engineering extrapolation with no direct validation. Evidence score reflects solid foundations with moderate application-specific evidence.
- **Feasibility**: 0.65 — Requires the observe-edit hook to capture both pre-edit and post-edit content, which is a change from the current hook that sees only the post-edit state. The before-state can be captured from git (the previous version of the file, available via `git show HEAD:{filename}`). Then two embedding calls compute the semantic before/after vectors. If cosine(before, after) > 0.90, the edit did not change meaning significantly — skip storage. If cosine(before, after) < 0.50, the edit significantly changed meaning — store the semantic diff as the memory. In 2026, two embedding calls = 100ms added to every hook execution. This makes the hook visibly slow. Mitigation: run the semantic diff asynchronously after a short delay, discarding if the result is "store nothing." Feasibility is moderate because the hook architecture requires modification and the latency impact is real in 2026.
- **Description**: Recall's observe-edit hook, on each file save, computes: (1) git diff of the changed region; (2) git show HEAD:{file} to get the pre-edit content of the changed region; (3) embed both pre-edit and post-edit content; (4) compute cosine distance. If distance < 0.10 (edit was syntactic/formatting, not semantic): skip storage. If distance between 0.10 and 0.40 (small semantic change): store a delta memory ("changed approach to X from Y to Z"). If distance > 0.40 (major semantic change): store a full memory replacing or superseding the pre-edit memory. This eliminates the need for LLM-based extraction of "what changed" and reduces false-positive stores (trivial edits generating noise memories). In 2032, the three-call pipeline runs in 1.5ms total; in 2026, it runs asynchronously at 100ms without blocking the edit hook's acknowledgment to Claude Code.

**Score (2026 context)**: 0.40×0.78 + 0.35×0.60 + 0.25×0.65 = **0.685 → BREAKTHROUGH**

### Idea: hyde-query-expansion-at-retrieve

- **Novelty**: 0.72 — HyDE (Hypothetical Document Embeddings, Gao et al. 2023) is a published RAG technique. Its application to personal AI memory retrieve operations is not present in production systems. The novelty is the application context, not the technique itself. In the RAG literature, HyDE is applied to document search; applying it to ephemeral personal memory with session-scoped stores is an adaptation.
- **Evidence**: 0.75 — Gao et al. (2023) "Precise Zero-Shot Dense Retrieval without Relevance Labels" demonstrates HyDE improving retrieval precision on BEIR benchmark. The technique is peer-reviewed. The application to personal memory retrieve is engineering extrapolation from a validated research finding. Evidence is strong for the underlying mechanism.
- **Feasibility**: 0.30 — In 2026, HyDE requires generating hypothetical documents via an LLM call (200–500ms for qwen3:14b) and then embedding each variant (50ms × N variants). For N=5, total overhead is 450–700ms — exceeding Recall's entire retrieve budget. HyDE is infeasible in 2026 without a dedicated fast inference path. In 2032, the LLM generation becomes the bottleneck, not embedding; but by 2032 small fast models handle this in ~20ms.
- **Description**: Recall's retrieve endpoint, instead of embedding the user's literal query, generates N=5 hypothetical memories via a small fast language model: "If a memory system contained the answer to this query, what text would that memory contain?" Each hypothetical memory is embedded (5 × 0.5ms = 2.5ms in 2032). The union of top-K results from each hypothetical embedding's Qdrant search produces a candidate pool. This handles the vocabulary mismatch between how users formulate queries and how memories were stored. A user asking "what did I decide about the Redis config?" retrieves memories that say "changed Redis maxmemory-policy to allkeys-lru" even though the word "decide" does not appear in the memory. The hypothetical document generation bridges the lexical gap.

**Score (2026 context)**: 0.40×0.72 + 0.35×0.75 + 0.25×0.30 = **0.626 → BREAKTHROUGH**

Note: Verdict is BREAKTHROUGH on novelty and evidence despite low feasibility. This idea is correct in principle and should be queued for implementation when embedding costs make it viable — approximately 2028 based on the hardware trajectory visible in 2026.

### Idea: continuous-background-reembedding

- **Novelty**: 0.68 — Continuous background re-embedding of the cold store on model upgrade is not present in any production AI memory system. Database systems re-index on schema changes (PostgreSQL REINDEX CONCURRENTLY), but the analogue for vector embeddings — automatically re-embedding all stored memories when the embedding model changes — is absent from all known systems. Pinecone and Weaviate treat re-embedding as a manual migration task.
- **Evidence**: 0.72 — Concurrent re-indexing in databases (PostgreSQL REINDEX CONCURRENTLY, MySQL ALTER TABLE ALGORITHM=INPLACE) provides the architectural precedent. The technical feasibility is well-established: read old embeddings, compute new embeddings, write new embeddings, swap the index. The automation of this process for AI memory on model version change is extrapolated from database practice. Evidence is strong for the mechanism, moderate for the specific application.
- **Feasibility**: 0.55 — In 2026, re-embedding 10K memories at 50ms each takes 8 minutes of serial compute time, during which the embedding model is unavailable for real-time queries (if they share the same GPU). Background re-embedding requires either a dedicated second GPU or a careful interleaving of real-time queries and batch re-embedding — technically feasible but operationally complex. In 2032 at 0.5ms per embedding, re-embedding 10K memories takes 5 seconds. The 2026 feasibility is moderate because it is achievable but requires careful scheduling.
- **Description**: Recall maintains an `embedding_model_version` field in PostgreSQL alongside each memory's UUID. When the embedding model changes (qwen3-embedding:0.6b → qwen3-embedding:1.2b → a 2028 successor model), Recall's background maintenance job detects the version mismatch, fetches memories in batches of 100, re-embeds each batch using the new model, updates the Qdrant collection, and updates the `embedding_model_version` field. Queries that hit memories with the old embedding version are re-embedded on-the-fly at retrieve time (with a small latency penalty) until the background job catches up. This converts model upgrades from a planned downtime event to a transparent rolling migration. In 2026, this should run during idle periods (session-end, overnight) to avoid competing with real-time embed requests on the shared RTX 3090.

**Score (2026 context)**: 0.40×0.68 + 0.35×0.72 + 0.25×0.55 = **0.657 → BREAKTHROUGH**

## Ideas That Look Wrong From 2032

**The negative embedding cache (Q004, high 2026 novelty score = 0.85)**

In 2026, the Q004 negative cache was scored as the highest-novelty latency optimization in the entire system (0.85). From 2032, it looks like the most expensive solution to a problem that solved itself. The engineering complexity required to maintain correct negative cache invalidation on store operations, tune the TTL to balance false-positive empty returns vs. stale negative entries, and extend the cache key space to cover near-duplicate queries — all of this complexity was premature optimization of a 50ms operation that became a 0.5ms operation by routine hardware progress.

The 2026 engineer who implemented the negative cache did the right thing given their constraints. The 2032 engineer reading that code sees a sophisticated solution to a problem that no longer exists and must spend time removing it rather than adding features.

**The LSH novelty sketch (Q012, novelty score = 0.68)**

The LSH in-process novelty sketch was the most technically elegant solution in the Q012 finding. From 2032, it is also the most over-engineered. Maintaining a synchronized in-process HNSW or LSH index alongside the Qdrant collection — handling race conditions when the FastAPI process restarts, handling index drift when Qdrant receives direct writes, managing the 40MB in-process RAM budget — all of this complexity served a single goal: avoiding a 10ms Qdrant call that Q011's gRPC improvement would have reduced to 2ms anyway.

The lesson from the LSH sketch: when the bottleneck is moving due to an external trend (hardware cost), invest in removing the architectural coupling to the bottleneck rather than building increasingly clever approximations of the bottleneck operation.

**The embedding-model-colocate idea (Q011, feasibility score = 0.50)**

The Q011 finding proposed deploying a smaller, faster embedding model on the Recall host CPU (e.g., all-MiniLM-L6 at 23M params) to eliminate the LAN RTT to the GPU host. This was a quality-latency tradeoff: the smaller model embeds faster but retrieves worse. From 2032, this idea looks like it was solving the wrong problem. By 2030, the efficient model running locally on the Recall host is not a 23M-param degraded model — it is a 600M-param model running on the host CPU's NPU at 0.5ms. The LAN RTT problem dissolved not because the architecture changed but because the hardware improved to meet the constraint on the compute side. The quality-latency tradeoff the Q011 idea accepted was unnecessary.
