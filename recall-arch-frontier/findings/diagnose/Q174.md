## 2032 Retrospective Analysis

### The Decision Under Evaluation

Deploying SimHash Layer 1 dedup (Q155/Q176) and the co_retrieval bootstrap clustering (K=200, seeding 1.25M CO_RETRIEVED edges at weight=0.1) simultaneously on the same day, and asking whether these two deployments interact in ways that affect the 30-day health dashboard trajectory.

The question is whether the two systems are neutral to each other (they address different failure modes — SimHash prevents duplicate writes, bootstrap seeds graph density), or whether they are in tension (SimHash removes near-duplicate memories that the bootstrap may have co-seeded as related).

### Historical Analogue

**Parallel**: The simultaneous deployment of database deduplication and index pre-population in early enterprise data warehouse migrations (1995-2003).

When data warehouse teams discovered that their source systems had accumulated years of duplicate records, they faced a dilemma: deduplicate first (shrinking the record set) or build the relationship indexes first (populating the graph structure). Teams that built the indexes before deduplication discovered that a measurable fraction of their index entries pointed to records that were subsequently tombstoned, creating orphaned index entries that degraded query precision.

**The "obviously right" approach in hindsight**: Deduplicate first, then build secondary indexes. The actual order deployed — bootstrap then observe decay — turned out to be acceptable precisely because the near-duplicate rate was low enough (1-3%) that orphaned bootstrap edges were a small fraction of total seeded edges, and those edges decayed naturally through the weight mechanism without manual remediation.

## The Wrong Assumption (2026 Vintage)

**The assumption**: The bootstrap's 1.25M seeded edges are durable value that compounds over time with organic reinforcement. Every seeded edge either gets reinforced (becoming organic signal) or stays dormant (contributing nothing but harmless noise).

**Why it seemed reasonable in 2026**: Q163 and Q172 both framed bootstrap as a pure positive — cross the density threshold immediately, let organic accumulation then refine. The interaction with SimHash dedup (which retroactively removes some of the memories the bootstrap treated as distinct nodes) was not modeled.

**What data revealed it was wrong**: The bootstrap is constructed from memories that exist at deployment time, many of which are near-duplicates of each other (since SimHash wasn't running before). When SimHash is deployed simultaneously, new near-duplicates of corpus members are redirected to existing memories rather than creating new nodes. This does not remove existing nodes — SimHash dedup only prevents new duplicate writes, it does not tombstone pre-existing duplicates. The assumption was wrong in a subtle way: the wrong assumption was not about bootstrap edge durability but about what SimHash does to the existing corpus.

**The correct understanding**: SimHash retroactively removes zero memories from the corpus at deployment time — it only prevents future near-duplicate writes. Bootstrap edges seeded from the existing 22,423-node corpus therefore point to nodes that remain stable. The interaction between SimHash and bootstrap is weaker than assumed, in both directions.

### Part (a): Bootstrap Edge Survival After 30 Days

**The SimHash non-interaction result**:

SimHash, as specified in Q155 and Q176, operates exclusively on the write path: `store_memory()`. It intercepts new writes and redirects near-duplicates to existing memories. It does not scan or modify the existing corpus. At day 0 deployment, all 22,423 nodes exist as distinct Qdrant points regardless of whether they are near-duplicates of each other.

Therefore: bootstrap edges seeded on day 0 connect exactly these 22,423 nodes. SimHash deployment on the same day does not tombstone any of them.

**What fraction of bootstrap edges survive 30 days?**

The question is whether any bootstrapped memories are removed from the corpus in the 30-day window through mechanisms other than SimHash (explicit user deletion, TTL expiry, importance decay to zero). From Q143, the corpus has no active forgetting mechanism — importance scores are set at write time and never updated, and there is no eviction policy. The system grows indefinitely.

**Result**: Approximately 99.5-99.9% of bootstrap edges survive 30 days intact. The surviving fraction is not limited by SimHash but by:
- Explicit user deletions: estimated near-zero (no observed pattern in Tim's usage)
- Superseded memories: the `superseded_by` field tombstones some memories when contradiction detection fires; estimated <0.5% of corpus per 30 days

**What fraction of bootstrap edges point to memories that would have been deduped if SimHash had been running earlier?**

This is the more interesting sub-question. If SimHash had been running for the past 6+ months before deployment, some fraction of the existing 22,423 nodes would not exist — they would have been redirected to earlier near-duplicates. How many nodes is this?

Near-duplicate write rate: 1-3% of writes. At Tim's write rate (Q143: ~50 writes/session × 3 sessions/day = ~150 writes/day), the 6-month pre-deployment period accumulated approximately:
- 150 writes/day × 180 days = 27,000 writes
- Near-duplicate fraction: 1-3% → 270 to 810 writes that SimHash would have blocked
- These blocked writes would have pointed to existing memories rather than creating new nodes
- So the pre-deployment corpus is 270-810 nodes larger than it would have been under SimHash

Bootstrap edges touching these "phantom" nodes (nodes that would not exist under SimHash) as a fraction of all bootstrap edges:
- Phantom nodes: 270-810 out of 22,423 = 1.2% to 3.6% of corpus nodes
- In a random bootstrap graph, each node participates in proportional edges
- Bootstrap edges touching at least one phantom node: approximately 2 × 3.6% - 3.6%^2 ≈ 7% (upper bound, uniform assumption)
- But phantom nodes are by definition near-duplicates of other nodes, meaning they cluster in the same semantic neighborhoods as their originals
- Bootstrap edges between a phantom node and its near-duplicate original are semantically valid (they should be related) and represent no quality loss
- Bootstrap edges from phantom nodes to semantically distant nodes are the actual noise: approximately 1-2% of total bootstrap edges

**Summary for part (a)**:
- Bootstrap edges surviving 30 days: ~99.5%
- Bootstrap edges pointing to "would-have-been-deduped" memories: ~1.2-3.6% of nodes
- Bootstrap edges that are semantically meaningless noise due to phantom nodes: ~1-2% of 1.25M = 12,500-25,000 edges
- This is negligible against the 1,245,766 total seeded edges

### Part (b): co_retrieval_gravity Dashboard Metric — Bootstrap vs. No-Bootstrap Control

**The Scenario A caveat**: Q163 established that co_retrieval_gravity reads from Neo4j RELATED_TO edge strength, not from the PostgreSQL co_retrievals table. The following analysis assumes the full implementation is deployed (bootstrap writes directly to Neo4j CO_RETRIEVED edges; organic reinforcement increments RELATED_TO strength by +0.1 per co-retrieval event). Without Neo4j propagation, both systems produce co_retrieval_gravity = 0.0 regardless.

**Bootstrap-seeded system at day 30**:

At day 0, 1,245,766 CO_RETRIEVED edges are seeded at weight = 0.1 each.

After 30 days of organic accumulation:
- Organic pairs per day: ~900 (K=15 live capture)
- Organic increments over 30 days: 900 × 30 = 27,000 increments
- Unique organic edges over 30 days: ~3,600 (from Q163's birthday problem model with 3× clustering correction — actual new unique edges from the 4,050 raw draws)
- Of those 3,600 new edges, approximately 2,800 will be within bootstrap clusters (high overlap probability given that bootstrap used K=200 semantic clusters, and live retrieval draws from similar semantic regions)
- Those 2,800 edges receive a +0.1 increment on top of their 0.1 bootstrap seed: weight now 0.2
- The remaining ~800 organic edges are either between clusters (cross-cluster retrieval) or between nodes in the active pool that bootstrap connected at lower frequency

**co_retrieval_gravity formula** (from Q163, health.py lines 118-128):
```
gravity = min(sum(RELATED_TO.strength) / 5.0, 1.0)
```

A memory needs cumulative CO_RETRIEVED strength across all its edges to reach 5.0 for gravity = 1.0.

At day 30 in the bootstrap-seeded system:
- Hub memories (top 10% by degree): ~50-80 CO_RETRIEVED edges × 0.1 base = 5.0-8.0 cumulative strength → gravity = 1.0
- Active memories (top 30%): ~15-30 edges × 0.1 base = 1.5-3.0 cumulative strength → gravity = 0.30-0.60
- Cold memories (bottom 70%): 0-5 edges × 0.1 = 0.0-0.5 → gravity ≈ 0.0-0.10
- Corpus-average gravity: approximately 0.18-0.28

**No-bootstrap control at day 30**:

Starting from zero edges, 30 days of organic accumulation:
- 3,600 unique edges with weight 0.1 (single organic hit)
- 131 edges with weight 0.2 (two organic hits) — from Q163's birthday collision model
- ~10 edges with weight ≥ 0.5 (five or more organic hits)
- Hub memories: maximum cumulative strength ≈ 3.0-4.0 (they participate in many of the 3,600 organic edges at weight 0.1) → gravity = 0.60-0.80
- Active memories: cumulative strength 0.1-0.5 → gravity = 0.02-0.10
- Cold memories (97%+ of corpus): zero organic edges → gravity = 0.0
- Corpus-average gravity: approximately 0.003-0.008

**Measurable difference?**

Bootstrap-seeded: corpus-average gravity ~0.18-0.28
No-bootstrap control: corpus-average gravity ~0.003-0.008

The difference is 20-90x in corpus-average gravity at day 30. This is not a statistical edge case — it is a categorical difference. The bootstrap-seeded health dashboard shows a rich bimodal distribution (hub cluster near 1.0, active pool in 0.3-0.6 range, cold majority near 0.0), while the no-bootstrap control shows an approximately uniform distribution at near-zero with a thin right tail (a few hub memories reaching 0.6-0.8 through organic accumulation alone).

The force profile visual in the health dashboard would be immediately distinguishable to a human observer without any statistics:
- Bootstrap: bimodal (cold spike at 0.0 and second peak at 0.3-0.6)
- No-bootstrap: monotonically decreasing from 0.0 (almost entirely cold)

**Statistical test at day 30**: A Mann-Whitney U test on per-memory gravity scores from the two conditions would have power >0.99 to detect the difference at any sample size above 100 memories. The effect size (Cohen's d) is approximately 2.5-4.0, which is extremely large. No sophisticated statistics required — visual inspection of the dashboard suffices.

### Part (c): Bootstrap Decay Rate

**The core question**: How long does the initial weight=0.1 signal remain distinguishable from organic accumulation?

At 600 retrievals per day with K=15 live capture, how many organic hits does each bootstrap edge receive per day?

Per-edge organic hit rate:
- Organic pairs generated per day: K×(K-1)/2 × retrieve_calls/day = 105 × 600/3 retrievals × ...

Let me be more precise using Q163's numbers:
- Retrieve calls per day: sessions (3) × calls per session (5) × K=15 → 3 × 5 × 105 = 1,575 new pair-opportunities per day
- But many pairs repeat; unique new pairs per day ≈ 900 (high scenario from Q163 corrected for K=15)
- Total bootstrap edges: 1,245,766
- Expected organic hits per bootstrap edge per day: 900 / 1,245,766 = 0.000723 hits per edge per day

At this rate:
- Days for average bootstrap edge to receive 1 organic hit: 1 / 0.000723 ≈ 1,383 days (3.8 years)
- Days for average bootstrap edge to receive 2 organic hits (weight 0.3 total): 2,766 days
- Days for average bootstrap edge to double its weight: 1,383 days

**But this is the wrong frame.** Bootstrap edges are not equal. They follow a power-law degree distribution in the K=200 cluster graph: hub memories in dense semantic neighborhoods have many bootstrap edges, while peripheral memories have few. The organic accumulation also follows a similar topology — hub memories get more organic hits.

The correct question is: **for edges that organic accumulation actually touches, how quickly does the organic signal dominate the bootstrap signal?**

For an edge that receives organic reinforcement:
- Bootstrap seed: 0.1 weight
- Each organic hit adds: 0.1 weight
- After 1 organic hit: weight = 0.2 (50% organic, 50% bootstrap)
- After 4 organic hits: weight = 0.5 (80% organic, 20% bootstrap)
- After 9 organic hits: weight = 1.0 (90% organic, 10% bootstrap)

For hub-hub edges (between the top-100 most retrieved memories), expected organic hits per day:
- These edges are drawn from a 100-node active subgraph
- 100-choose-2 = 4,950 possible hub-hub pairs
- At 900 unique pairs/day with 3× clustering, hub-hub pairs dominate: ~30% of daily unique pairs are hub-hub
- Hub-hub organic hits per day: ~270 across 4,950 edges = 0.055 hits/edge/day
- Days for hub-hub edges to achieve weight=0.5 (4 organic hits): 4/0.055 = 73 days
- Days for hub-hub edges to achieve weight=1.0 (9 organic hits): 9/0.055 = 164 days

**Bootstrap decay timeline for edges that organic accumulation reaches:**

| Edge tier | Organic hits/day | Days to organic dominance (>80%) | Days to bootstrap-indistinguishable (>90%) |
|---|---|---|---|
| Hub-hub (top-100 nodes) | 0.055 | ~73 days | ~164 days |
| Active-active (top-30% nodes) | 0.004 | ~1,000 days | ~2,250 days |
| Cold edges (bottom-70% nodes) | ~0.0001 | Never in system lifetime | Never |

**The bootstrap decay pattern is trimodal:**

1. **Hot tier (hub-hub edges, ~5% of all bootstrap edges)**: Organic accumulation dominates by day 90. By day 180 these edges are indistinguishable from purely organic edges. The bootstrap seed accelerated their trajectory by approximately 1-2 organic hits worth of weight — meaningful but modest.

2. **Warm tier (active-active edges, ~25% of all bootstrap edges)**: Organic accumulation never catches up to fully dominate within 3 years at current session rates. The bootstrap seed remains a measurable component of edge weight throughout the system's operational life at current usage intensity.

3. **Cold tier (never-reinforced bootstrap edges, ~70% of all bootstrap edges)**: These edges receive zero organic hits. They sit permanently at weight=0.1 unless a decay mechanism is implemented. From a retrieval perspective, they function as low-confidence priors: if two cold memories are retrieved together, the bootstrap's 0.1 weight provides a tiny boost that would not otherwise exist. Without decay, they accumulate as permanent noise.

**Decay rate summary**: The term "bootstrap decay" is misleading. There is no decay mechanism in the 2026 architecture (Q143: no forgetting mechanism). Bootstrap edges do not decay — they are either reinforced by organic accumulation (increasing their weight, converging to organic signal), or they remain frozen at weight=0.1 indefinitely. The 70% cold tier is permanent background at weight=0.1.

Whether this is a problem depends on the reranker's sensitivity to low-weight edges. If the reranker applies a minimum weight gate (Q163's `weight-threshold-5-as-noise-filter` idea: only use edges with weight ≥ 5 organic hits worth, i.e., weight ≥ 0.5 after bootstrap seed), the cold tier bootstrap edges are entirely invisible to retrieval and cause no harm. If the reranker uses all edge weights without a threshold, the cold tier adds a uniform low-intensity noise floor to retrieval — semantically valid (they ARE within-cluster pairs) but undifferentiated from truly related memories.

### Part (d): One-Time Cost With Indefinite Payoff?

**The two-phase bootstrap value model:**

**Phase 1 (Days 0-90): The density gap fills**

Without bootstrap, the co_retrieval graph at day 30 has ~3,600 edges — approximately 0.3% of the 1.25M seeded by bootstrap. The health dashboard shows near-zero co_retrieval_gravity for 97%+ of memories. The reranker has essentially no graph signal to work with.

With bootstrap, day 0 immediately crosses the density threshold identified in Q163 (~177,600 edges needed for measurable ordering improvement in the active pool). The 1.25M seeded edges represent 7× this threshold. Retrieval quality improvement is immediate and measurable from day 1, not from day 465-1,450 (the organic accumulation timeline from Q163).

This Phase 1 payoff is unambiguously real. The bootstrap collapses 1.5-4 years of organic accumulation into a one-time 7-42 minute job (Q172's nominal-to-best-case range).

**Phase 2 (Days 90+): Signal quality divergence**

After 90 days, the organic accumulation has reinforced the high-value bootstrap edges (hub-hub tier). These edges now have weights of 0.2-1.0 and are grounded in actual retrieval patterns. The cold-tier bootstrap edges (70% of seeded edges) remain at 0.1 — semantically valid but unvalidated.

The question from 2032 is: does the unvalidated cold tier create retrieval quality problems over time?

**From 2032 retrospective**: The unvalidated cold tier was found to create a specific failure mode called "semantic neighborhood bleed." In a K=200 cluster graph, two memories that are semantically adjacent but actually unrelated in practice (e.g., two separate infrastructure concerns that happen to use similar vocabulary) accumulate 0.1 weight edges that, without decay, persist indefinitely. When either of those memories is retrieved, the other gets a small co_retrieval boost — not large enough to flip its rank position (0.1 weight is below the ordering flip threshold without organic reinforcement), but large enough to affect the gravity score displayed in the health dashboard.

The practical consequence in 2032 production data: approximately 15-20% of cold-tier bootstrap edges in the first-year corpus pointed to memory pairs that turned out to be thematically unrelated when evaluated by humans (they shared cluster membership via embedding proximity but were not conceptually linked). These edges never received organic reinforcement, confirming their inutility. Without decay, they remained in the graph permanently.

**The crossover point — when does organic accumulation alone achieve equivalent graph density without bootstrap?**

From Q163's density model:
- Target: 1,245,766 edges (the bootstrap threshold)
- Organic accumulation rate: ~900 unique pairs/day with clustering saturation
- Saturation-adjusted accumulation: 80% new in months 1-3, then declining
- Effective rate: year 1 ≈ 250,000 new unique edges; year 2 ≈ 150,000; year 3 ≈ 100,000
- Cumulative organic edges by year: Y1: 250K, Y2: 400K, Y3: 500K, Y4: 600K

The bootstrap seeded 1.25M edges. Organic accumulation alone reaches 1.25M edges somewhere around year 5-6 at current session rates — if the active pool expands with corpus growth, possibly never (because each new memory dilutes the density).

**The crossover is not a sharp threshold.** The relevant question is not "when does organic accumulation match bootstrap quantity" but "when does organic accumulation reach the active-pool density threshold for measurable retrieval quality." That threshold (2.22% active pool density from Q163) is reached organically at ~860 days (mid scenario). Bootstrap crosses it immediately.

**Verdict on indefinite payoff**: Bootstrap provides unambiguous value in the first 18-24 months. After 24 months, organic accumulation has reinforced enough hub-tier edges that the bootstrap seed is no longer the primary contributor to retrieval quality for high-frequency memory pairs. For low-frequency pairs (the cold tier), bootstrap remains the only signal indefinitely — whether that signal is valuable or noise depends on whether a decay mechanism is ever deployed.

Bootstrap is best characterized as: a one-time cost that buys 18-24 months of retrieval quality that organic accumulation cannot otherwise deliver, while permanently seeding a cold-tier noise floor that requires a decay mechanism to clean up.

## Bootstrap Decay Analysis

### The Trimodal Edge Lifecycle

The 1,245,766 seeded edges split into three functional tiers based on post-deployment behavior:

**Tier 1 — Reinforced edges (approximately 5-8% of bootstrap edges, ~62,000-100,000 edges)**
These are edges between hub memories (top-100-200 most retrieved nodes). Organic accumulation reinforces them within 30-90 days. By day 90, their weight is 0.2-0.5 from organic hits stacking on the 0.1 bootstrap seed. By day 180, the bootstrap seed contribution is less than 50% of edge weight; by day 365, less than 20%. These edges are functionally organic within the first year.

**Tier 2 — Partially reinforced edges (approximately 20-25% of bootstrap edges, ~250,000-310,000 edges)**
These are edges involving at least one active-pool memory (top 30%). They receive organic hits at a rate of 0.004/day. Over 3 years, they accumulate 4-5 organic hits on average. The bootstrap seed remains a measurable component (~20-30%) of their final weight. These edges are permanently "hybrid" — partially organic, partially bootstrap.

**Tier 3 — Unreinforced edges (approximately 70% of bootstrap edges, ~870,000 edges)**
These connect cold memories that are never organically retrieved. They sit at weight=0.1 indefinitely. They are semantically valid within their K=200 clusters but untested by actual retrieval patterns. Without a decay mechanism, they accumulate as permanent background at the minimum weight level. With a decay mechanism (e.g., ×0.95/week on edges with no organic hits in 30 days), they would fade to effectively zero (weight < 0.01) within approximately 70 weeks.

### Decay Rate Table (with hypothetical 0.95/week decay on unreinforced edges)

| Weeks | Tier 3 weight | Tier 1 weight (reinforced hub-hub, 0.055 hits/day) |
|---|---|---|
| 0 | 0.1 (bootstrap seed) | 0.1 (bootstrap seed) |
| 4 | 0.081 | 0.254 (0.1 + 3.3 organic hits × 0.1) |
| 13 (day 91) | 0.051 | 0.1 + 9.1 × 0.1 = 1.01 |
| 26 (day 182) | 0.026 | 0.1 + 18.2 × 0.1 = 1.92 |
| 52 (day 364) | 0.007 | saturated (weight ≥ 5.0 → gravity = 1.0) |
| 70 | 0.002 (below noise) | (irrelevant — saturated) |

If decay is implemented, Tier 3 edges effectively disappear within 70 weeks (~16 months), leaving a clean graph of only organically-validated edges plus the surviving Tier 1 and 2 hybrid edges.

### Simhash — Bootstrap Interaction: The Narrow Edge Case

The one genuine interaction between SimHash and bootstrap occurs when:
1. A memory M exists in the corpus at day 0 (bootstrap seeded M as a node)
2. After day 0, a new write arrives that is a near-duplicate of M
3. SimHash correctly redirects the new write to M (no new node created)
4. The bootstrap edge (M, M') connects M to some memory M'

This scenario does not degrade M's bootstrap edges — M still exists, its edges still point to valid nodes. The near-duplicate write, instead of creating a new node that would have accumulated its own bootstrap edges (had SimHash not been running), is merged into M. This is the correct behavior. The bootstrap edge set is undisturbed.

The only loss is marginal: had SimHash not been running, the new node would have accumulated its own organic retrieval signal that would have been partially distinct from M's signal. SimHash collapses that potential distinction. This is not a degradation — it is the intended behavior of deduplication.

**SimHash and bootstrap are orthogonal.** They do not interfere with each other in any meaningful way at the 1-3% near-duplicate rate.

## Key Findings

### (a) Bootstrap Edge Survival

At day 30, approximately 99.5-99.9% of bootstrap edges survive intact. SimHash dedup does not affect existing corpus nodes at deployment time — it only prevents future near-duplicate writes. The ~1-2% of bootstrap edges that are "semantically questionable" (pointing between pre-existing near-duplicate corpus members) are valid edges from Neo4j's perspective and cause no correctability issue.

### (b) Dashboard Metric Measurability

The co_retrieval_gravity distribution at day 30 is categorically different between bootstrap-seeded and no-bootstrap systems:
- Bootstrap-seeded: corpus-average gravity ~0.18-0.28, bimodal distribution
- No-bootstrap control: corpus-average gravity ~0.003-0.008, near-zero monotonic distribution
- Effect size (Cohen's d): approximately 2.5-4.0 (extremely large)
- Detection: immediate visual inspection without statistics

The bootstrap effect on the health dashboard is not subtle. Any operator who views the dashboard on day 1 will see the difference. This distinguishability persists for at least 18-24 months.

### (c) Bootstrap Decay

"Decay" is a misnomer in the 2026 architecture — there is no weight decay mechanism. The correct framing is "bootstrap signal dilution":
- Tier 1 (hub-hub, 5-8% of edges): organically dominated by day 90-180
- Tier 2 (active-active, 20-25% of edges): permanently hybrid; organic signal ~20-50% after 3 years
- Tier 3 (cold-cold, ~70% of edges): permanently frozen at weight=0.1 unless decay is deployed

Without decay, bootstrap signal is never fully erased — it is at minimum a permanent 0.1-weight floor on all seeded edges. Whether that floor is beneficial (prior for retrieval) or harmful (noise for precision) depends on whether the reranker applies a minimum weight gate.

The time for bootstrap-seeded hub edges to become "indistinguishable from organic" is approximately 90-180 days at Tim's session rates.

### (d) Payoff Duration

Bootstrap provides:
- Immediate value: retrieval quality improvement from day 1 (crossing the density threshold 1.5-4 years before organic accumulation would)
- Primary payoff window: months 1-18 (the period where organic accumulation would otherwise be below the measurable threshold)
- Secondary benefit: permanent cold-tier prior at weight=0.1 for all within-cluster pairs (useful if reranker applies a weight gate; noise if it doesn't)
- Long-term liability: the Tier 3 cold edges never self-clean without a decay mechanism; they represent ~870,000 edges of permanently frozen background signal

**Crossover point**: Organic accumulation achieves equivalent active-pool graph density to bootstrap at approximately year 5-6 (if no decay, no corpus growth dilution) or never (if corpus grows faster than retrieval patterns can cover). Bootstrap is not a 30-90 day advantage — it is a multi-year structural advantage for single-user systems at 22K+ corpus scale.

## Ideas

### Idea: decay-gate-for-cold-bootstrap-edges
- **Novelty**: 0.74
- **Evidence**: 0.80
- **Feasibility**: 0.85
- **Description**: Implement a lightweight weekly decay job that multiplies CO_RETRIEVED edge weight by 0.95 for any edge that has received zero organic increments in the past 30 days. This collapses the Tier 3 cold bootstrap edges to negligible weight (~0.007 after 70 weeks) while preserving Tier 1 and Tier 2 edges that have received organic reinforcement. The job requires only: `MATCH (a)-[r:CO_RETRIEVED]->(b) WHERE r.last_reinforced_at < (now - 30 days) SET r.weight = r.weight * 0.95`. The `last_reinforced_at` field must be added to CO_RETRIEVED relationships at bootstrap write time. Without this field, decay cannot distinguish bootstrap-only edges from organic edges. This is a one-line addition to the bootstrap write job and the organic reinforcement path.

### Idea: staged-bootstrap-k-refinement
- **Novelty**: 0.68
- **Evidence**: 0.76
- **Feasibility**: 0.82
- **Description**: Rather than seeding all 1.25M K=200 bootstrap edges at equal weight=0.1, apply a two-tier initial weight: within-cluster edges for the top-K most recently-accessed memories (estimated top-2,000 nodes) seed at weight=0.2, while the remaining cold-pool nodes seed at weight=0.05. This pre-differentiates the active pool from the cold pool at bootstrap time, giving the reranker a higher-confidence prior for the memories most likely to be retrieved. The active-pool weight=0.2 edges are above the "two organic hits" equivalence threshold, making them immediately useful for retrieval ordering; the cold-pool weight=0.05 edges are below the minimum meaningful threshold and effectively defer activation to organic reinforcement. This approach requires knowing which nodes are "recently accessed" at bootstrap time, which is available from the PostgreSQL access_count field.

### Idea: simhash-bootstrap-retroactive-dedup-sweep
- **Novelty**: 0.70
- **Evidence**: 0.72
- **Feasibility**: 0.78
- **Description**: On the day SimHash is deployed, run a one-time sweep of the existing corpus to identify near-duplicate pairs (using the same 8-band SimHash lookup on all existing memories) and merge them before the bootstrap job runs. This shrinks the corpus from 22,423 to approximately 21,800-22,200 nodes (removing 220-620 near-duplicates at 1-3% rate) before bootstrap clustering begins. The result: the K=200 bootstrap cluster graph does not include any near-duplicate pairs as distinct nodes; all seeded edges are between genuinely distinct memories. This eliminates the ~1-2% of bootstrap edges that are "phantom" (pointing between pre-existing near-duplicates). The sweep takes approximately 22,423 × 2.5ms (SimHash band query time) = 56 seconds of sequential queries, or 10-15 seconds with parallel workers. Recommended order: retroactive sweep → bootstrap job → SimHash write-path activation.

### Ideas That Look Wrong From 2032

**Treating bootstrap as "30-90 day" value only**: The question framing implied bootstrap was relevant only in the short term. In practice, the Tier 2 hybrid edges (20-25% of bootstrap) remain the dominant signal source for active-pool pairs for 2-3 years at Tim's session rates. Bootstrap is not a short-term accelerant — it is a long-term structural feature of the graph.

**Expecting bootstrap-seeded edges to decay naturally without a decay mechanism**: The 2026 architecture has no forgetting mechanism. Tier 3 bootstrap edges are permanent absent a deliberate implementation choice. Building the bootstrap job without simultaneously planning the decay mechanism is structurally incomplete — the bootstrap creates a long-term liability that grows over years as the cold tier accumulates.

**The "organic accumulation as the true signal" narrative**: From 2032, the clean separation between "organic = trustworthy" and "bootstrap = synthetic noise" was found to be wrong for the Tier 2 edges. Memory pairs that cluster together in K=200 embedding space and that also organically co-occur in retrieval are reinforced by both mechanisms. These edges are doubly validated — the bootstrap predicts co-relevance from content, and the organic signal confirms it from actual usage. In 2032, the hybrid Tier 2 edges were found to be the highest-precision co_retrieval edges in the graph, outperforming purely organic edges (which include session-clustering noise at low weight) and purely bootstrap edges (which lack usage validation). The clean "organic good, bootstrap noise" framing missed this.
