# Finding: Q089 — Normal-Operation Orphan Accumulation: Does the Tombstone Reaper Clean Neo4j?

**Question**: In the absence of any PostgreSQL restore events, do Neo4j orphan nodes accumulate from normal tombstone hard-deletes — and what is the steady-state orphan count after 12 months of normal operation at Tim's usage density?
**Agent**: quantitative-analyst
**Question Type**: ADVERSARIAL
**Verdict**: PROMISING
**Severity**: Medium
**Source field**: Distributed systems (multi-store consistency, garbage collection symmetry), database operations (cross-store deletion discipline), AI memory systems (tombstone lifecycle management)

## Adversarial Analysis

### Thread A: Defense — The Reaper Cleans Both Stores Symmetrically

**Core claim**: The tombstone reaper that hard-deletes from PostgreSQL also deletes the corresponding Neo4j node via a symmetric write path, so steady-state orphan count from normal TTL expiry is near zero.

**Evidence**: In a well-engineered dual-store system, write operations are typically symmetric. The Recall write path (per Q057) stores Memory nodes in Neo4j AND rows in PostgreSQL at the same time — this dual-write discipline at store time suggests dual-delete discipline at reap time. Additionally, the reaper is purpose-built code written by the system designers who knew the architecture; it is not a post-hoc runbook like the Q081 restore procedure. Runbooks miss steps because they are written by operators under time pressure; reaper code is written by engineers who have the schema and code in front of them.

**Best case**: The reaper runs `DETACH DELETE` in Neo4j immediately after the PostgreSQL DELETE. Steady-state orphan count = 0. Q089 is MARGINAL — normal operation produces no orphan accumulation beyond the PG-restore path identified in Q081.

### Thread B: Attack — The Reaper Is PostgreSQL-Only; Orphans Accumulate at 730/Year

**Core claim**: The tombstone reaper is a scheduled Python job that operates on the PostgreSQL `memories` table. It has no corresponding Neo4j cleanup, because the reaper's job description is "delete expired tombstones from the database" — and "the database" in operational contexts refers to the relational store. Neo4j is a secondary graph store consulted at retrieval time, not part of the standard delete path.

**Evidence 1 — The Q081 analogy is the strongest available proxy**: Q081 BREAKTHROUGH established that the human-written Q077 runbook omitted the Neo4j cleanup step for the PG-restore path. The reasoning Q081 used applies directly here: if human experts writing a deliberate recovery procedure forgot the Neo4j step, automated code written to clean up PostgreSQL tombstones is at least as likely to omit it — possibly more so, because the reaper's conceptual scope is explicitly "expired tombstone rows in PostgreSQL." The reaper was designed to solve the PostgreSQL problem (unbounded row accumulation from tombstoned memories). Engineers cleaning up "the database" default to the relational primary.

**Evidence 2 — Architecture asymmetry exposes a cognitive gap**: Q057 describes two distinct write paths into Neo4j: (1) the store-time path (Memory node creation, which has a symmetric PostgreSQL write), and (2) the retrieval-time path (CO_RETRIEVED edge updates, which has NO PostgreSQL symmetric). The existence of a retrieval-time-only Neo4j write path means Neo4j already contains a category of data (CO_RETRIEVED edges) that has no PostgreSQL counterpart. If the reaper developer was testing the delete path, they would test "does the PG row disappear?" — and it does. "Does the Neo4j node disappear?" is a separate check requiring understanding of the full multi-store architecture. A plausible reaper implementation:

```python
# Plausible reaper logic (inferred from Q070's dual-bound tombstone schema)
expired = await pg.fetch("""
    SELECT id FROM memories
    WHERE status = 'tombstoned'
      AND (current_epoch - created_epoch >= session_ttl
           OR NOW() - created_ts >= interval '90 days')
""")
for row in expired:
    await pg.execute("DELETE FROM memories WHERE id = $1", row['id'])
    # <--- Neo4j DETACH DELETE must be explicitly coded here.
    # If omitted: the Memory node persists in Neo4j indefinitely.
```

Neo4j cleanup is a cross-service operation requiring a separate Bolt driver call. It does not happen implicitly when PostgreSQL deletes a row. It must be explicitly coded. If the developer treated Neo4j as a "best-effort cache" rather than a store requiring lifecycle management, the Neo4j call simply does not appear.

**Evidence 3 — Q070's tombstone TTL mechanics confirm the reaper's scope**: The tombstone hard-delete fires when BOTH bounds are met: 50-session ceiling AND 90-day calendar floor. The reaper is a FastAPI background task or cron job that scans the PostgreSQL `memories` table for expired tombstone rows. Its output is a set of PostgreSQL row deletions. The reaper is defined by its input (PG query) and output (PG deletes). Neo4j cleanup is a side effect that must be deliberately added.

## Synthesis

### What Thread A Assumes That Thread B Exposes

Thread A assumes the delete path received the same deliberate dual-store design as the write path. Thread B exposes that this symmetry is not architecturally enforced — it is a developer convention that must be explicitly followed for every deletion code path. The Q081 finding that the HUMAN-AUTHORED RUNBOOK missed the Neo4j cleanup step is the strongest proxy available: runbooks and reapers share the same cognitive failure mode (the author is thinking about the primary store and does not loop back to the secondary store). The write path is the critical path (retrievals depend on it); the delete path is a maintenance path that receives less review and less testing.

### When Thread A Is Right

If the Recall FastAPI backend wraps all deletion operations in a shared cleanup function that explicitly calls both `DELETE FROM memories WHERE id = $1` AND `MATCH (n:Memory {id: $id}) DETACH DELETE n`, the reaper inherits correct behavior automatically. If the system was architected with a "dual-store client" abstraction layer, the delete path is symmetric by construction. Alternatively, if Q081's findings were incorporated into the architecture before the reaper was written — prompting explicit dual-store deletion discipline — Thread A is correct and Q089 is MARGINAL.

### When Thread B Is Right

If the reaper is a standalone scheduled job written to solve the PostgreSQL storage growth problem (the most likely motivation), and the developer treated Neo4j as a retrieval cache rather than a store requiring lifecycle management, Thread B is correct. In homelab self-hosted systems, Neo4j cleanup is frequently an afterthought. The 9% steady-state orphan fraction at Month 12 is the expected outcome.

### The Critical Interaction With Q078's Epoch Filter

The relationship between Q089's orphan path and Q078's Cypher epoch filter is the most important architectural finding in this analysis.

By definition, any memory that reaches tombstone hard-delete has gone without re-encounter for at least 50 sessions (the session-count TTL ceiling from Q070). This means every orphaned Neo4j node from normal TTL expiry has `last_access_epoch` that is at least 50 sessions older than the current epoch. With tau=20, the epoch filter predicate `WHERE neighbor.last_access_epoch >= (current_epoch - tau=20)` would reject these orphans — because their last access epoch is at least 50 sessions behind, well outside the tau=20 working-set window.

**This produces a critical asymmetry between the two orphan paths**:

| Orphan source | `last_access_epoch` relative to `current_epoch` | Epoch filter (Q078) behavior | BFS contamination |
|--------------|--------------------------------------------------|------------------------------|-------------------|
| **Q081 path** (PG restore) | ABOVE current_epoch (pre-retraction values are numerically higher than retracted epoch) | Filter PASSES them (they appear fresh) | **Yes — contamination occurs** |
| **Q089 path** (TTL expiry) | BELOW current_epoch by ≥50 sessions | Filter REJECTS them (they appear stale) | **No — filter quarantines them** |

The epoch filter, once implemented per Q078, functions as an automatic quarantine for the Q089 orphan path. Nodes orphaned via normal TTL expiry are self-excluded from BFS results because their staleness is encoded in their epoch value. The BFS contamination from TTL-expiry orphans is self-healing once Q078 is implemented.

However, the filter does NOT prevent Neo4j graph topology corruption:
- Orphaned nodes continue to consume storage in the Neo4j node store.
- Orphaned nodes inflate node counts, making consistency checks against PostgreSQL show growing discrepancy.
- Orphaned nodes' CO_RETRIEVED edges may retain pointers to still-valid nodes, creating asymmetric relationships (valid node → orphan → filtered at retrieval, but the valid node's degree count still includes the orphaned neighbor).
- The Q081 cross-store consistency check (`neo4j_orphan_count`) would correctly identify and count Q089 orphans, indistinguishable from Q081 PG-restore orphans.

### The Third Option: What Thread B Would Accept

Thread B would accept two complementary mechanisms that do not require certainty about whether the reaper is correct:

1. **The Q081 cross-store consistency check** (`neo4j_orphan_count`, running every 6 hours) detects orphan accumulation regardless of source. If the reaper correctly cleans Neo4j, `neo4j_orphan_count` remains near zero through Month 12. If the reaper omits Neo4j cleanup, `neo4j_orphan_count` rises at ~2/day and reaches ~180 by Month 3 — observable long before Month 12's 730-orphan state.

2. **A tombstone-reaper deletion audit log** that records `(memory_id, deleted_from_pg_at, neo4j_delete_status)` for every hard-delete event. This provides per-deletion confirmation that both stores were cleaned, without requiring operators to infer correctness from aggregate counts.

### Genuine Uncertainty

The critical unknown is whether the Recall FastAPI tombstone reaper explicitly calls `MATCH (n:Memory {id: $id}) DETACH DELETE n` after the PostgreSQL DELETE. This is a code-level fact not determinable from architecture findings alone. Q057's Cypher templates describe traversal queries, not deletion paths. No prior finding has traced the reaper execution path into Neo4j.

The adversarial pair has strong structural grounds (Q081 analogy, write-path vs. delete-path engineering attention asymmetry) to assign > 60% probability to Thread B's scenario — but cannot rule out that the reaper was written correctly. The practical consequence of this uncertainty is that the Q081 cross-store consistency check should be implemented unconditionally: if the reaper is correct, the check confirms it with zero cost. If the reaper is missing the Neo4j step, the check surfaces it within 3 days of deployment (time for ~6 orphans to accumulate above the detection threshold of > 10).

## Key Finding

Q089 identifies a **distinct normal-operation orphan path** that is independent of the Q081 PG-restore failure mode. Whether this path actually produces orphans depends on whether the tombstone reaper was written with dual-store deletion discipline. The structural evidence (Q081's runbook gap as a proxy for reaper code quality, the PostgreSQL-centric framing of the reaper's design goal) suggests > 60% probability that the reaper omits Neo4j cleanup.

The most important architectural finding in Q089 is the interaction with Q078's epoch filter: **orphans from normal TTL expiry are self-quarantined by the epoch filter once Q078 is implemented**, because these nodes' `last_access_epoch` values are at least 50 sessions behind the current epoch by definition. The Q089 orphan path therefore does NOT degrade S5 compliance if Q078 is already implemented — contrasting with the Q081 PG-restore orphan path, which produces nodes with epoch values ABOVE the current epoch that PASS the epoch filter.

This means:
1. Q078 implementation closes the Q089 BFS contamination risk as a side effect.
2. The Q081 cross-store consistency check closes the Q089 graph hygiene risk.
3. Neither fix requires knowledge of whether the reaper was written correctly.
4. Implementing both Q078 and the Q081 cross-store check makes the system robust to both orphan paths without requiring archaeology of the reaper code.

The steady-state orphan count at Month 12 under the worst case (reaper omits Neo4j cleanup) is **730 orphaned Neo4j nodes** — a 9% node-corpus fraction that degrades graph hygiene but (with Q078 implemented) does not degrade BFS retrieval quality.

## Ideas Extracted

### Idea: tombstone-reaper-dual-store-delete-audit

- **Novelty**: 0.78 — The specific pattern of instrumenting a tombstone reaper with a per-deletion audit log that records `neo4j_delete_status` as a confirmation field — and treating a NULL or failed status as a health alert — is not described in any reviewed AI memory system. The Q081 cross-store consistency check catches orphans in aggregate; the audit log catches them at the individual deletion event, enabling root-cause isolation (was the Neo4j call attempted and failed, or never attempted?). This converts a silent architectural gap into a verifiable deletion contract.
- **Evidence**: 0.62 — The structural argument is validated by the Q081 finding: the human-authored runbook missed the Neo4j cleanup step, establishing that deletion paths in multi-store systems are gap-prone. The audit log pattern is standard in financial transaction systems (dual-ledger audit trails, debit-credit symmetry checks) but is not documented in AI memory system implementations. The evidence is analogical rather than direct measurement of reaper code behavior.
- **Feasibility**: 0.87 — Implementation: add a `deletion_audit` table to PostgreSQL (`memory_id UUID, deleted_from_pg_at TIMESTAMPTZ, neo4j_delete_status TEXT, neo4j_deleted_at TIMESTAMPTZ`). The tombstone reaper writes an audit row after each PostgreSQL DELETE, then attempts the Neo4j DETACH DELETE and updates the `neo4j_delete_status` field. The `/health/detail` endpoint queries `SELECT COUNT(*) FROM deletion_audit WHERE neo4j_delete_status IS NULL AND deleted_from_pg_at < NOW() - INTERVAL '1 hour'` and surfaces non-zero counts as a `reaper_neo4j_gap_count` warning. Total implementation: ~100 lines of Python + 1 PostgreSQL table. The audit table is append-only and can be pruned on a 30-day rolling window.
- **Description**: Instrument the tombstone reaper to record a deletion audit entry for every hard-delete. Each entry captures: `memory_id`, `deleted_from_pg_at` (timestamp of PostgreSQL DELETE), `neo4j_delete_status` (NULL if not attempted, 'success' if DETACH DELETE completed, 'failed' with error if attempted but errored). A FastAPI background task that runs hourly queries for audit entries with NULL `neo4j_delete_status` older than 1 hour — these are confirmed reaper gaps (PostgreSQL deleted, Neo4j not cleaned). The count is surfaced as `reaper_neo4j_gap_count` in the `/health/detail` response. If the reaper correctly cleans both stores, every entry shows `neo4j_delete_status = 'success'` and the health signal stays at zero. If the reaper omits Neo4j cleanup, the signal climbs at the hard-delete rate (~2/day at Tim's usage density), reaching 6 within 3 days — above any reasonable alert threshold — and identifying the gap before significant orphan accumulation occurs.

### Idea: epoch-filter-as-ttl-orphan-quarantine

- **Novelty**: 0.71 — The insight that the Q078 Cypher epoch filter (`WHERE neighbor.last_access_epoch >= (current_epoch - tau)`) automatically quarantines TTL-expiry orphans — because nodes evicted after 50+ sessions have `last_access_epoch` at least 50 epochs behind the current epoch, far outside the tau=20 window — is a non-obvious consequence of the tombstone dual-bound TTL design from Q070. This interplay between Q070's session-count TTL ceiling, Q078's epoch filter, and Q089's orphan path creates a self-sealing property: implementing Q078 eliminates BFS contamination from both graph maturity AND normal-operation orphans simultaneously, without requiring the reaper to be fixed. No prior finding drew this three-way connection.
- **Evidence**: 0.65 — The claim follows deductively from: (1) Q070's dual-bound TTL requires a tombstoned memory to go at least 50 sessions without re-encounter before hard-delete; (2) Q078's epoch filter uses tau=20 to reject neighbors with `last_access_epoch < current_epoch - 20`; (3) therefore, any node eligible for hard-delete has `last_access_epoch` at least 50 sessions behind, which is at least 30 sessions outside the tau=20 window. The deduction is logically tight given the two established parameter values. The contrast with the Q081 PG-restore orphan path (where nodes have epoch values ABOVE the current epoch, specifically defeating the epoch lower-bound filter) reinforces the finding by showing the asymmetry is not accidental.
- **Feasibility**: 0.90 — This is an analytical finding, not an implementation task. Its operational implication is: implement Q078 before Month 6 to simultaneously close three S5 degradation mechanisms (graph maturity, PG-restore orphans from Q081, and TTL-expiry orphans from Q089). No additional implementation beyond Q078 is required to protect BFS retrieval quality from Q089 orphans. Graph hygiene (orphan node accumulation in Neo4j storage) still requires the reaper fix or the cross-store consistency check + manual pruning, but retrieval quality is protected by Q078 alone.
- **Description**: The Q078 Cypher epoch filter functions as an orphan quarantine for the normal-operation (TTL-expiry) orphan path as a side effect of its primary design purpose (excluding stale memories from BFS results). Any Memory node that was evicted through the tombstone lifecycle had `last_access_epoch` that was already at least 50 sessions behind the current epoch at the time of hard-delete. After hard-delete, the node persists in Neo4j with this stale epoch value. The epoch filter at tau=20 rejects any neighbor with epoch < current_epoch - 20, which definitionally includes all TTL-expiry orphans (they are at least 50 - 20 = 30 sessions outside the filter window). Implementing Q078 therefore eliminates BFS retrieval contamination from Q089 orphans without requiring any reaper code changes. The residual risk is Neo4j storage growth and node-count inflation from accumulated ghost nodes — addressed by the Q081 cross-store consistency check and periodic Neo4j pruning via the Cypher tombstone-cleanup query (`MATCH (n:Memory) WHERE n.last_access_epoch < (current_epoch - 50) DETACH DELETE n`), which itself becomes a safe batch cleanup operation once Q078 protects retrieval.
