# Finding: Q106 — Redis Epoch Reset: Ghost Memory Resurrection

**Question**: What happens when Redis restarts and the epoch clock resets to zero — specifically, does a Redis restart cause the working-set filter (`last_access_epoch >= (current_epoch - tau)`) to treat all previously-accessed memories as outside the working set (epoch = 0 makes current_epoch - tau negative or zero, admitting everything), and what is the attack surface if an adversary (or misconfiguration) could force repeated Redis restarts to bypass LLPC routing thresholds?
**Agent**: quantitative-analyst
**Question Type**: ADVERSARIAL
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Redis persistence configuration, Python signed integer arithmetic, epoch-based working-set filters, multi-store coherence, health signal blind spots

---

## Thread A (Pessimist): Redis Epoch Reset Is a Critical Vulnerability with Immediate Token Budget Consequences

**Core claim**: A Redis restart without AOF persistence causes `recall:global_epoch` to reset to 0, which — due to Python's arbitrary-precision signed integer arithmetic — makes `last_access_epoch >= (0 - 20) = last_access_epoch >= -20` trivially true for ALL stored memories, producing instant ghost resurrection: every memory in PostgreSQL passes the working-set filter simultaneously.

**Evidence and mechanism**:

The working-set filter in the FastAPI retrieve endpoint evaluates:

```python
threshold = current_epoch - tau   # tau = 20
# If current_epoch = 0: threshold = 0 - 20 = -20
# Python integers are arbitrary precision — no underflow, no wrap-around
# All stored memories have last_access_epoch >= 0
# Therefore: last_access_epoch >= -20 is True for EVERY memory in PG
```

Python integers have no concept of underflow. Unlike C's `int32_t` (which would wrap to a large positive number and admit nothing), Python's integer subtraction returns exactly -20. Every stored `last_access_epoch` is a non-negative integer (it starts at 0 when the memory is first accessed and can only increase via `INCR`). So `last_access_epoch >= -20` is unconditionally true for the entire PostgreSQL memories table.

The ghost resurrection mechanics at Redis restart:

| Redis State Before Restart | After Restart (No AOF) | Working-Set Filter Behavior |
|---|---|---|
| `recall:global_epoch = 850` | `recall:global_epoch` key ABSENT | First `GET` returns nil; `INCR` would create key at 1; but a `GET` before any `INCR` returns 0 (nil coerced to 0 in most Redis client integer reads) |
| 850 memories have `last_access_epoch` in [830, 850] | Epoch reads as 0 | Threshold = 0 - 20 = -20; ALL memories pass |

**Concrete failure scenario**: Tim's CasaOS restarts Redis without AOF (the default Redis Docker image ships with AOF disabled). The session-start hook fires. The retrieve endpoint reads `current_epoch = 0`. The working-set filter computes `threshold = -20`. The Qdrant ANN query returns K=50 candidate memories. The PostgreSQL working-set filter admits all 50 (every `last_access_epoch >= 0 >= -20`). The LLPC routing check runs on the entire candidate set: any memory that qualifies as LLPC (access_count >= 3 AND last_access_epoch <= tau from the WRONG perspective) may be injected as full-text.

**LLPC injection budget explosion**: Q097 established that f_LLPC is the primary token budget risk. Under normal operation, the LLPC pool is bounded by the working-set window (tau=20 epochs). Under ghost resurrection (epoch=0), the effective working-set window becomes infinite — all memories in the corpus are "current." If the LLPC pool contains K_llpc memories and all pass the filter, the token injection budget spikes to K_llpc × 350 tokens (full-text injection for each). At Month 9 (hub node emergence threshold, Q094), Tim's corpus contains approximately 4,050 nodes. If even 5% qualify as LLPC by access count, that is ~200 LLPC memories × 350 tokens = 70,000 tokens injected — exceeding any practical context window.

**The no-persistence attack surface**: The Redis Docker image on CasaOS may have AOF disabled by default (Redis's own default is AOF off, RDB on with 15-minute save intervals). A CasaOS host restart, a `docker compose down && docker compose up`, or a Redis OOM event all cause `recall:global_epoch` to vanish from memory. If RDB persistence is also unconfigured or the data directory is lost (e.g., after a Docker volume wipe for debugging), the key starts at 0. This is not an exotic attack — it is the default behavior of a Redis instance that was deployed without explicit persistence configuration.

**Repeated restart attack surface**: An adversary (or a misconfiguration loop) that can force repeated Redis restarts exploits the resurrection window between restart and the first `INCR` that advances the epoch beyond 0. Each restart resets the window, creating repeated ~20-session ghost resurrection events. During each window:
- Every memory in PG passes the working-set filter
- LLPC memories receive full-text injection regardless of recency
- f_LLPC = 1.0 for the LLPC pool until the epoch advances past tau=20
- The 20-session (at 5 sessions/day = 4 hours) window is long enough to cause sustained token budget overruns in a normal homelab usage pattern

**No existing health signal detects this**: The S4 epoch integrity signal (Q071) monitors `redis_epoch < pg_max` — a discrepancy between the two stores. But after a Redis restart from zero, the S4 check would read `redis_epoch = 0` and `pg_max = 850`. This IS a detectable discrepancy: `redis_epoch (0) < pg_max (850)`. So S4 SHOULD alert. But whether the alert fires depends on whether S4 is checking BEFORE the first session attempts to use the epoch. If the session-start hook runs the retrieve endpoint before the S4 health check runs, the ghost resurrection has already begun.

**Best case scenario**: AOF is enabled with `appendfsync everysec`. On restart, Redis replays the AOF log and restores `recall:global_epoch` to within 1 epoch of its pre-restart value. The vulnerability window is negligible (one epoch = one session). Q081 established this as the standard recovery path.

**Failure mode**: AOF disabled (default), RDB disabled or volume wiped, or malicious `FLUSHDB`/`DEL recall:global_epoch` command. Ghost resurrection begins immediately on the next session. f_LLPC spikes to 1.0. Token budget explodes. S4 may alert, but only if the health check runs before the retrieve path and the operator has defined a remediation procedure.

## Thread B (Optimist): The Epoch Reset Is Self-Healing Within 20 Sessions — Manageable with One Guard

**Core claim**: The ghost resurrection vulnerability is real but time-bounded: after the Redis restart, epoch advances by 1 per session (via `INCR`). The working-set filter becomes accurate again after exactly tau=20 sessions. The vulnerability window for Tim's usage (5 sessions/day) is approximately 4 hours — a narrow window with a clear mitigation.

**Evidence and mechanism**:

The self-healing dynamics:

| Sessions After Redis Restart | current_epoch | Threshold (tau=20) | Memories That Pass Filter |
|---|---|---|---|
| 0 (restart) | 0 | -20 | All (last_access_epoch >= -20) |
| 1 | 1 | -19 | All (last_access_epoch >= -19) |
| 5 | 5 | -15 | All (last_access_epoch >= -15) |
| 20 | 20 | 0 | Only memories accessed in last 20 sessions |
| 21 | 21 | 1 | Only memories with last_access_epoch >= 1 |
| 850+20 | 870 | 850 | Normal steady-state working-set window |

The filter is self-correcting. After exactly tau=20 sessions, it behaves identically to the pre-restart state (from the filter's perspective — it cannot distinguish epoch 20 post-restart from epoch 20 at system bootstrap). Memories that had `last_access_epoch = 700` (stale, from 150 sessions ago) are filtered out once current_epoch reaches 720 (700 + 20). The filter heals completely after tau sessions regardless of the pre-restart epoch value.

**PostgreSQL is the durable anchor**: PostgreSQL stores `last_access_epoch` per memory and retains its values across Redis restarts. The epoch values in PG are accurate records of when each memory was last accessed. After the self-healing period (20 sessions), the PG values become the correct filter input again. Redis's role is only to provide the current epoch counter — the per-memory values that determine working-set membership are stored in PG, not Redis.

**S4 will alert**: After a Redis restart (epoch resets to 0) but with PG containing `MAX(last_access_epoch) = 850`, the S4 signal reads: `redis_epoch (0)` vs `pg_max (850)`. This is a massive discrepancy (850 vs threshold of ~50). S4 will fire as CRITICAL immediately. The existing remediation procedure (Q074, Q077) applies: set Redis to `pg_max + 1 = 851`. If the operator follows the runbook, the epoch is restored before any session has a chance to use the corrupt epoch=0 value for retrieval.

**The window requires concurrent failures**: For ghost resurrection to cause material token budget damage, three things must be true simultaneously:
1. Redis restart occurs (AOF disabled or volume wiped)
2. The retrieve endpoint is called BEFORE the S4 health check fires the alert
3. The operator does not apply the Q077 remediation within the window

For Tim's deployment, where CasaOS is on a stable LAN and Redis is not directly internet-exposed, the probability of a malicious forced restart is low. The more realistic risk is accidental misconfiguration (AOF off by default in some Docker images).

**Monotonic epoch guard is the complete fix**: A single guard eliminates the vulnerability entirely. At session-start, before executing the working-set filter, the retrieve endpoint queries `MAX(last_access_epoch)` from PostgreSQL:

```python
pg_max = await pg_pool.fetchval("SELECT MAX(last_access_epoch) FROM memories")
if current_epoch < pg_max - EPOCH_TOLERANCE:
    # Redis epoch is behind PG — Redis restart or PG advance
    # Option: raise an error, or auto-repair by setting current_epoch = pg_max + 1
    raise EpochIntegrityError(f"Redis epoch {current_epoch} < PG max {pg_max}")
```

This turns a silent ghost resurrection into an immediate error or auto-repair. Cost: one PG SELECT per session start (< 1ms on a GIN-indexed integer column at Tim's scale).

**Best case scenario**: AOF is enabled, epoch persists across restart, S4 fires on any discrepancy, and the monotonic guard provides a second layer of protection. Ghost resurrection never occurs because no combination of single failures reaches the retrieve path with epoch=0.

## Synthesis

**What Thread A assumes that Thread B exposes**:

Thread A assumes that ghost resurrection will go undetected long enough to cause material damage. Thread B exposes the hidden assumption: the S4 signal already detects the epoch discrepancy (redis_epoch=0 vs pg_max=850 is an 850-epoch gap, far above any reasonable alarm threshold). The real vulnerability is not that the system cannot detect the failure — it is that the detection mechanism (S4) and the remediation path (Q077 runbook) may run AFTER the first retrieve call has already used epoch=0. The race condition between health-check and retrieve is the actual threat surface.

Thread B assumes that the 20-session self-healing window is "manageable." Thread A exposes the compounding consequence: in those 20 sessions, every LLPC memory is injected at full-text tier. At Month 9+ (Q094: hub nodes emerge, f_LLPC rising), the token budget consequence is not linear — it is catastrophic for single sessions.

**The compound truth**: Ghost resurrection is a real, exploitable vulnerability with both an accidental path (AOF misconfiguration, default Docker images) and a structural path (the retrieve endpoint does not validate epoch sanity before executing the filter). The 20-session window is not acceptable under Tim's usage density because 20 sessions at 5/day = 4 hours of potential context window explosions. The S4 signal detects the discrepancy but the existing remediation is manual (operator sets Redis epoch). The monotonic epoch guard turns this from a manual-remediation scenario into an automatic self-repair.

**When Thread A is right**: AOF is disabled (default Redis Docker), Redis volume is wiped (common during debugging), or a malicious operator has access to `redis-cli FLUSHDB`. In these scenarios, ghost resurrection begins on the next session and persists for 20 sessions without self-repair. S4 fires but remediation is manual. If the operator does not act within one session, every subsequent session in the window receives a full-corpus working-set injection.

**When Thread B is right**: AOF is enabled with `appendfsync everysec`, the system is running in a stable homelab environment, and S4 is monitored with automated remediation. In this scenario, epoch loss is bounded to < 1 session per restart, S4 fires with enough lead time, and the self-healing dynamics kick in within 20 sessions even without manual intervention.

**The architecture Thread B would accept**:

The minimal mitigation that resolves Thread A's objection while preserving Thread B's correctness argument is a two-layer defense:

**Layer 1 — Retrieve-path epoch sanity guard** (eliminates the race condition):

```python
# In FastAPI retrieve endpoint, BEFORE executing working-set filter:
redis_epoch = await redis_client.get("recall:global_epoch") or 0
pg_max_epoch = await pg_pool.fetchval("SELECT MAX(last_access_epoch) FROM memories")

if redis_epoch < pg_max_epoch - EPOCH_RESET_TOLERANCE:
    # Redis is behind PG — restart without persistence detected
    # Auto-repair: reset Redis epoch to pg_max + 1
    await redis_client.set("recall:global_epoch", pg_max_epoch + 1)
    redis_epoch = pg_max_epoch + 1
    logger.warning(f"Epoch reset detected: Redis was {redis_epoch_raw}, PG max is {pg_max_epoch}. Auto-repaired.")
```

`EPOCH_RESET_TOLERANCE` should be set to the maximum legitimate epoch divergence between Redis and PG, which Q079 established as ~50 for dual-machine deployments. Any gap > 50 is a restart event, not normal divergence.

**Layer 2 — Redis AOF enforcement** (eliminates the persistence gap):

```yaml
# docker-compose.yml for Redis service on CasaOS:
redis:
  image: redis:7-alpine
  command: redis-server --appendonly yes --appendfsync everysec --save ""
  # --save "" disables RDB snapshots (AOF is sufficient and lower-latency)
  volumes:
    - recall_redis_data:/data
```

This configuration must be explicit in the Recall deployment documentation and `docker-compose.yml`. The default Redis Docker image starts with `appendonly no`. Without an explicit override, the vulnerability exists by default.

**Answers to the six required questions**:

**(a) Redis key and value type**: `recall:global_epoch` is a Redis string key storing an integer value, manipulated exclusively via `INCR` semantics. `INCR` atomically reads, increments, and writes the integer. If the key does not exist, `INCR` creates it with value 1 (not 0). A raw `GET` on a non-existent key returns nil; client-side coercion to integer produces 0. This means the dangerous epoch=0 state occurs during the window between key absence and the first `INCR` call — specifically, if the retrieve endpoint reads the epoch via `GET` (returning nil→0) rather than via `INCR`.

**(b) Python signed arithmetic**: Python integers are arbitrary precision. There is no underflow, no wrap-around, no undefined behavior. `0 - 20 = -20` exactly. `last_access_epoch >= -20` is unconditionally true for all memories with non-negative stored epoch values (which is all of them — epoch values start at 0 and can only increase). Ghost resurrection admits the ENTIRE PostgreSQL memories table to the working-set filter. This is worse than the C integer case: in C, `int32_t` underflow would wrap to a large positive number (0 - 20 = 4294967276 as unsigned, or -20 as signed), and the signed comparison `last_access_epoch >= -20` would still be true for all non-negative epoch values. Python does not change the outcome — both admit all memories.

**(c) Required Redis persistence configuration**: `appendonly yes` with `appendfsync everysec` is required. This limits epoch loss to at most 1 second (1 epoch) on restart. RDB-only persistence (the Redis default) saves every 15 minutes, meaning up to ~300-900 sessions could be lost in an outage scenario (depending on Tim's session density and the timing of the last RDB save). No persistence at all means full epoch loss on every restart. The deployment must explicitly set AOF in the `docker-compose.yml` command override — relying on defaults creates the vulnerability silently.

**(d) Monotonic epoch guard — necessary or overkill**: NECESSARY. The guard converts the detection mechanism from "external health check notices discrepancy" (which creates a race condition with the retrieve path) to "retrieve path validates its own preconditions" (which eliminates the race). The S4 signal is a monitoring layer, not an enforcement layer. The retrieve endpoint can and should enforce the invariant it depends on. The guard adds one PG read per session start (~1ms at Tim's scale) and provides complete protection against the race condition. It is the retrieve endpoint's equivalent of an assertion check — exactly the pattern Q081 identified as missing from the Neo4j traversal path.

**(e) S4 and S5 signal behavior during ghost resurrection**: S4 spikes immediately and loudly. After a Redis restart (epoch=0, pg_max=850), S4 reads: `redis_epoch=0`, `pg_max_epoch=850`, gap=850. The alarm threshold (Q079: ~50 for dual-machine deployments) is exceeded by 17×. S4 fires CRITICAL. S5 (graph traversal epoch compliance) is orthogonal to the epoch reset scenario — S5 monitors whether Neo4j traversal respects the epoch window, not whether Redis and PG are coherent. During ghost resurrection (epoch=0), S5 would see the working-set threshold collapse to -20, meaning essentially all traversed Neo4j nodes pass the filter. Whether S5 detects this depends on whether S5's "compliance check" measures the fraction of traversed nodes within the normal epoch window: if S5 measures `fraction with last_access_epoch >= current_epoch - tau`, and tau=20, epoch=0, threshold=-20, then ALL nodes pass, S5 fraction = 1.0, and S5 reports **healthy** (full compliance). S5 is silent during ghost resurrection. S4 fires. The misleading combination: S4=CRITICAL, S5=HEALTHY. This matches the S3+S4 co-alert pattern from Q084's correlation layer (CasaOS restart with Redis data loss), not the S4+S5 pattern (PG restore direction). The operator following Q084's correlation rules should correctly diagnose "Redis restart with data loss" and apply the Q077 remediation.

**(f) Silent ghost resurrection attack surface**: YES, a misconfiguration can silently enable ghost resurrection without any health signal firing — under one specific condition. If the retrieve endpoint is called in the same application startup sequence BEFORE the health check background task runs its first S4 evaluation, the ghost resurrection has already begun without any alert. In a FastAPI application that starts background tasks (including health checks) after the HTTP server begins accepting requests, there is a startup race: the first incoming session-start hook call may arrive and execute the retrieve endpoint before the health monitor has completed its first cycle. During this window, `recall:global_epoch = 0` (Redis just restarted), the filter admits all memories, and S4 has not yet fired. The duration of this window is approximately the health check startup delay — which, if the health monitor polls every 60 seconds and starts polling 5 seconds after FastAPI startup, creates a 5–55 second window where ghost resurrection is invisible. For an automated system that processes sessions continuously, 55 seconds may contain one or more retrieve calls. The monotonic epoch guard in the retrieve path (Layer 1 mitigation) is the only mechanism that closes this window completely, because it validates the precondition synchronously within each retrieve call, regardless of whether the health monitor has run.

## Ideas Extracted

### Idea: retrieve-path-epoch-sanity-guard
- **Novelty**: 0.78 — The pattern of validating store preconditions synchronously within the retrieve path (rather than relying on external monitoring to detect and remediate violations before retrieval executes) is documented in database transaction isolation literature but has not been applied to AI memory system epoch integrity. The specific combination of Redis epoch + PG max comparison as an inline precondition check — using PG as the durable epoch anchor and Redis as the fast-path counter — resolves a class of race conditions that monitoring-only approaches cannot address. No reviewed AI memory system (MemGPT, mem0, Zep, LangChain) maintains an epoch counter, let alone validates its integrity within the retrieval hot path.
- **Evidence**: 0.82 — The race condition between health-check and retrieve-path is a well-known class of TOCTOU (time-of-check, time-of-use) problem in distributed systems. The S4 signal fires correctly on a large epoch gap, but fires asynchronously. The retrieve path executes synchronously on each request. Without an inline guard, the S4 detection-remediation cycle cannot prevent the first N retrieval calls (where N is the number of calls that execute before the operator applies the runbook). The PG `MAX(last_access_epoch)` query is O(1) on a GIN-indexed integer column and adds < 1ms to session start. The EPOCH_RESET_TOLERANCE=50 boundary (from Q079) correctly separates legitimate dual-machine divergence from restart events.
- **Feasibility**: 0.92 — Implementation is 8 lines of Python at the top of the retrieve endpoint. No schema changes. No new infrastructure. The `pg_max_epoch` query reuses the existing asyncpg pool. The auto-repair action (set Redis epoch to pg_max + 1) mirrors the existing Q077 runbook manual step but executes it automatically. The guard can be feature-flagged with a `EPOCH_GUARD_ENABLED` env var for deployments where the PG query overhead is a concern (though at Tim's scale it is negligible). The only operational risk is a false-positive auto-repair if the PG epoch legitimately advances ahead of Redis due to dual-machine write skew — but Q079's analysis shows this gap never exceeds 50 under normal operation, making EPOCH_RESET_TOLERANCE=50 a safe boundary.
- **Description**: Add a synchronous epoch sanity guard to the FastAPI retrieve endpoint, executing before the working-set filter. The guard reads `recall:global_epoch` from Redis and `MAX(last_access_epoch)` from PostgreSQL. If the gap exceeds EPOCH_RESET_TOLERANCE (default 50), it auto-repairs by setting Redis epoch to `pg_max + 1`, logs a warning, and continues with the corrected epoch. This eliminates the race condition between S4 detection and retrieve-path execution, converting ghost resurrection from a silent data corruption event into a logged, auto-repaired one-session anomaly. The guard is the retrieve endpoint's equivalent of a transaction isolation check — it enforces the precondition that the epoch counter is consistent, rather than relying on external monitoring to enforce it after the fact.

### Idea: redis-aof-enforcement-as-deployment-invariant
- **Novelty**: 0.67 — The observation that Redis AOF configuration must be treated as a deployment invariant (not a tuning parameter) for any system whose correctness depends on Redis counter persistence is not novel in the Redis operations literature. However, its application specifically to AI memory systems that use Redis epoch counters for working-set filtering — where the consequence of missing AOF is ghost resurrection rather than mere data delay — adds a consequence severity argument that is novel to this domain. The formalization as a deployment invariant (must appear in docker-compose.yml command override, not as documentation advice) distinguishes this from generic "enable persistence" recommendations.
- **Evidence**: 0.78 — The Redis default configuration ships with AOF disabled and RDB every 900 seconds (15 minutes). Common Docker Redis deployments omit explicit persistence configuration (this is observable from Recall's current docker-compose.yml if it does not include `--appendonly yes`). Q081's synthesis established that Q081's compound failure was undocumented in the Q077 runbook — the AOF configuration gap is an equivalent undocumented operational prerequisite. The `appendfsync everysec` variant limits epoch loss to < 1 second (< 1 epoch), which is below any meaningful impact threshold. The `--save ""` addition (disabling RDB when AOF is active) prevents the RDB save interval from providing false reassurance about persistence status.
- **Feasibility**: 0.95 — A two-line change to the docker-compose.yml `command` override. No code changes. No schema changes. Backward-compatible with existing deployments (enabling AOF on an existing Redis instance simply begins recording the AOF log from the current state; existing data in memory is persisted on the next `BGREWRITEAOF`). The change can be deployed with a `docker compose restart redis` with no application downtime.
- **Description**: Enforce Redis AOF persistence as a documented deployment invariant for Recall. Specifically: (1) add `--appendonly yes --appendfsync everysec --save ""` to the Redis service command in docker-compose.yml; (2) add a deployment verification step to the Q077 runbook that checks `redis-cli CONFIG GET appendonly` returns `yes`; (3) add an S4-adjacent startup check that reads the Redis `CONFIG GET appendonly` value and emits a WARNING if AOF is disabled, before any session processing begins. This converts the AOF misconfiguration from a silent vulnerability to a detected and flagged deployment error.

### Idea: epoch-zero-bootstrap-trap-detection
- **Novelty**: 0.71 — The specific failure mode where Redis `GET recall:global_epoch` returns nil (key absent post-restart), client-side integer coercion produces 0, and the retrieve path computes threshold = -20 admitting all memories — is a distinct failure path from the general "epoch gap" that S4 monitors. The epoch=0 state can be distinguished from epoch=1 (normal bootstrap): on a fresh deployment, epoch=0 is expected; after 850 sessions, epoch=0 is catastrophic. The combination of `current_epoch == 0` AND `pg_max > tau` is an unambiguous indicator of restart-with-data-loss. A dedicated zero-epoch guard that fires when `current_epoch == 0 AND pg_max > tau` provides an alarm that is higher-precision than the general epoch-gap S4 signal (which also fires on normal forward divergence).
- **Evidence**: 0.73 — The epoch=0 case is analytically distinguishable from all valid operational states. At system bootstrap (Day 0), both `current_epoch = 0` and `pg_max = 0` (no memories yet). At any point after 20 sessions, `pg_max >= 20`. Therefore `current_epoch == 0 AND pg_max >= 20` is a zero-false-positive indicator of Redis restart with data loss. No normal operational state can produce this combination. The S4 signal (epoch gap > threshold) would also fire for this case (gap = 0 vs 850 >> 50), but S4's implementation uses a threshold that was calibrated for dual-machine forward divergence — a different failure mode. A dedicated zero-epoch guard requires zero tuning and has zero false positive rate in post-bootstrap operation.
- **Feasibility**: 0.88 — Implementation: `if current_epoch == 0 and await pg_pool.fetchval("SELECT MAX(last_access_epoch) FROM memories") > tau: trigger_alert("epoch_zero_restart_detected")`. This is a 3-line addition to the retrieve path epoch sanity guard (Idea 1). It fires an immediate alert with the highest possible severity — epoch=0 post-bootstrap is an unambiguous emergency. The alert can trigger an automatic epoch repair (set Redis to pg_max + 1) without operator intervention, since the false positive rate is effectively zero.
- **Description**: Add a dedicated epoch-zero bootstrap trap to the retrieve-path epoch guard. If `current_epoch == 0` (Redis key absent or explicitly zero) AND `MAX(last_access_epoch) > tau` (PostgreSQL has post-bootstrap epoch values), fire an EPOCH_ZERO_RESTART alert with Critical severity. This alert bypasses the configurable EPOCH_RESET_TOLERANCE threshold (which was designed for dual-machine forward divergence, not zero-restart events) and triggers immediate auto-repair. The trap is analytically zero-false-positive after the first tau sessions and catches the most dangerous form of ghost resurrection (full-corpus working-set admission) with zero ambiguity.
