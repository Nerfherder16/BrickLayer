# Finding: Q006 — Absence Mapping: Temporal-Weighted ANN Retrieval in Production AI Memory

**Question**: Has any production AI memory system implemented retrieval weighted by both semantic similarity AND recency decay (temporal-weighted ANN)? What is the evidence for its presence or absence?
**Agent**: quantitative-analyst
**Question Type**: ABSENCE
**Verdict**: PROMISING
**Severity**: High
**Source field**: Production AI memory systems / information retrieval

---

## Mechanism Under Investigation

A retrieval system where the final ranking score at ANN search time is a direct function of BOTH cosine similarity between embeddings AND a recency decay factor — such that `score = f(cosine_similarity, age)` — meaning time is a multiplicative or additive factor in the score that drives which vectors are returned, not applied as a post-retrieval re-rank or as a metadata filter.

---

## Evidence

### System: LangChain `TimeWeightedVectorStoreRetriever`

**Verdict**: PARTIAL — closest production implementation, but the blend happens POST-ANN, not IN-ANN.

Source: `libs/langchain/langchain_classic/retrievers/time_weighted_retriever.py` (verified via GitHub API, March 2026).

**The formula** (from source code and documented in LangChain docs):

```
score = (1.0 - decay_rate) ^ hours_since_last_access + vector_relevance
```

Where `decay_rate` defaults to 0.01, and `hours_since_last_access` counts hours since the document was last retrieved (not created — this is last-access time, implementing a Spotlight-style recency-of-use model).

**Architecture of the two-step process**:

1. Step 1 — ANN search: `vectorstore.similarity_search_with_relevance_scores(query, k=100)` — runs standard cosine similarity ANN, returns top-100 candidates.
2. Step 2 — Post-ANN re-scoring: For each of the 100 candidates, computes `(1.0 - decay_rate)^hours_passed + vector_relevance`, then returns the top-k by combined score.

The ANN index itself (FAISS, Chroma, etc.) is still pure cosine similarity. Time is not a factor during the ANN traversal — it is applied as a post-retrieval re-rank on the candidate set. This is **post-ANN re-ranking**, not temporal-weighted ANN.

**Production status**: Shipped in LangChain since ~2023. Used in the Generative Agents (Park et al. 2023) simulation as the reference implementation. Classified as `langchain_classic` in the current codebase, suggesting it is legacy/community-maintained rather than a flagship feature.

**Critical flaw in their design**: `hours_passed` is time-since-last-access, not time-since-creation. Frequently accessed old memories stay perpetually "fresh." This conflates recency with access frequency — a deliberate design choice that mirrors the Spotlight memory model but is semantically different from age-based decay.

### System: Generative Agents (Park et al. 2023) — arXiv:2304.03442

**Verdict**: RESEARCH — paper describes the mechanism; implementation delegates to LangChain's retriever.

The paper describes a three-component retrieval score:

```
retrieval_score = α_recency × recency + α_importance × importance + α_relevance × relevance
```

Where recency uses exponential decay over time, importance is an LLM-assigned scalar, and relevance is cosine similarity. This is the most cited description of temporal-weighted retrieval in the AI memory literature.

**However**: The reference implementation uses `TimeWeightedVectorStoreRetriever`, which does post-ANN re-ranking (not true in-ANN temporal weighting). The paper's formula is a conceptual specification; the code is a post-retrieval approximation of it.

**Production status**: Research simulation (not a shipped product). The agents paper is primarily an HCI/UIST paper, not a systems paper.

### System: mem0 (getmem0.com)

**Verdict**: ABSENT

mem0's vector store implementations (verified via `mem0ai/mem0` GitHub, `/mem0/vector_stores/`) delegate all retrieval to the underlying vector DB (Qdrant, Chroma, Pinecone, etc.) using standard cosine similarity search. No temporal decay factor is applied at retrieval time. mem0 does maintain `created_at` metadata but uses it only for filtering, not scoring.

The reranker feature (shipped in v1.0.0) uses LLM-based reranking for semantic relevance; no temporal component is documented or found in source.

---

### System: Letta / MemGPT (letta-ai/letta)

**Verdict**: ABSENT — temporal retrieval exists as a separate mode, not blended.

Verified via `letta/helpers/tpuf_client.py` (GitHub API, March 2026). Letta's Turbopuffer-backed retrieval supports four explicit search modes:

- `"vector"` — pure ANN: `rank_by: ("vector", "ANN", query_embedding)`
- `"fts"` — BM25 full-text: `rank_by: ("text", "BM25", query_text)`
- `"hybrid"` — weighted combination of vector + BM25 scores (configurable `vector_weight` / `fts_weight`)
- `"timestamp"` — recency sort: `rank_by: ("created_at", "desc")`

Time-based retrieval is a separate mode (`"timestamp"`), not blended with vector similarity. The hybrid mode blends vector + BM25 but not vector + recency. There is no mode that computes `cosine_similarity × f(age)`.

**Root cause**: Letta's architecture treats memory as a paging system (hot context vs. cold archival), not as a scored retrieval problem. Recency is handled by context management (what's in the active window), not retrieval scoring.

### System: Zep / Graphiti (getzep/graphiti)

**Verdict**: ABSENT at ANN score level — temporal awareness exists at graph structure level.

Verified via `graphiti_core/search/search_utils.py` (GitHub API, March 2026). Graphiti's search pipeline uses:

- `calculate_cosine_similarity()` — standard cosine; no temporal factor
- Full-text BM25 search for fact retrieval
- Graph traversal for entity relationships
- `DEFAULT_MIN_SCORE = 0.6` threshold for cosine filtering

Graphiti stores temporal metadata (`valid_at`, `expired_at`, `created_at`) on edges, enabling temporal knowledge graph features (tracking relationship validity over time). But at retrieval time, edges are scored by cosine similarity only. Time is not a factor in the similarity score.

A GitHub issue (#1300, opened March 2026) explicitly requests "temporal decay algorithms for retention management" — confirming this is not yet implemented in Graphiti.

The Zep technical paper (arXiv:2501.13956, Jan 2025) describes Graphiti as a "temporally-aware knowledge graph" but this refers to tracking knowledge evolution over time (knowing that fact X was true from date A to date B), not to decaying retrieval scores by age.

### System: Qdrant

**Verdict**: ABSENT

Qdrant's production search API supports: cosine/dot/euclidean/manhattan similarity metrics, payload filtering, score thresholding, and the ACORN filtered-ANN algorithm. There is no first-class temporal-weighted scoring mode. Custom scoring functions are not supported in the ANN traversal layer. Temporal weighting would require post-retrieval application-level re-ranking.

### System: Weaviate

**Verdict**: ABSENT

Weaviate supports vector search (near-vector, near-text, near-object), hybrid BM25+vector search, and metadata filtering. No temporal decay blended into ANN scoring. Temporal filtering by date range is supported as a WHERE clause, not as a scoring dimension.

---

### System: Pinecone

**Verdict**: ABSENT

Pinecone supports sparse-dense hybrid search, metadata filtering, and namespace partitioning. No temporal-weighted ANN scoring. Metadata-based recency filtering is possible but not scoring.

---

### System: Hindsight (hindsight.vectorize.io)

**Verdict**: PARTIAL — temporal signals treated as parallel retrieval leg, not blended into ANN score.

Hindsight (blog post, March 12 2026) implements "time-aware spreading activation" for memory graphs. The mechanism:

1. Vector ANN search with a semantic threshold filter
2. Temporal proximity scoring computed separately from ANN results
3. Graph traversal walking causal/temporal links
4. Late fusion of all signals

This is architecturally a parallel multi-signal system where temporal scores and vector scores are fused after each leg completes independently. Time is not blended into the ANN traversal itself. Notably, Hindsight explicitly frames this as: "Most retrieval systems treat time as a filter. We treat it as a dimension to traverse." — confirming the observation that no system treats time as an ANN-score factor.

---

### System: Microsoft GraphRAG / Mnemis

**Verdict**: ABSENT for temporal decay; PARTIAL for graph-structured retrieval.

Microsoft GraphRAG (global/local community graph retrieval) does not implement temporal decay in retrieval scoring. A Microsoft Research paper, Mnemis (arXiv:2602.15313, Jan 2026), implements dual-route retrieval (similarity search + hierarchical graph traversal) for LLM memory, but the similarity route is standard cosine ANN — no temporal weighting.

### Academic Literature

**RecallM** (arXiv:2307.02738, Kynoch et al. 2024): Proposes temporal understanding for LLM memory via belief updating in a knowledge graph, not via decay-weighted ANN retrieval. The temporal mechanism is about tracking which version of a fact is current, not about scoring by age.

**ENGRAM** (arXiv:2511.12960): State-of-the-art LoCoMo benchmark system (2025). Uses top-k dense neighbors per memory type with simple set-union merging. No temporal decay in scoring. Achieves SOTA without temporal weighting.

**SSGM** (arXiv:2603.11768, March 2026): Governance framework for evolving memory. Focuses on consistency verification, not retrieval scoring.

**T-GRAG** (Temporal GraphRAG, 2025): Proposes a temporal knowledge graph framework with time-stamped graph structures and temporal query decomposition. Retrieval is via graph traversal, not decay-weighted ANN.

**Adaptive Focus Memory** (arXiv:2511.12712): Features "temporal decay" alongside semantic relevance for assigning fidelity levels in multi-turn dialogue — research prototype, no production implementation.

**Classification**: PARTIAL ABSENCE

**Confirmed absent from** (production, temporal weighting at ANN score level):
- mem0
- Letta / MemGPT
- Zep / Graphiti
- Qdrant (vector DB)
- Weaviate (vector DB)
- Pinecone (vector DB)
- Microsoft GraphRAG / Mnemis

**Partially implemented (post-ANN re-ranking, not in-ANN)**:

**Research only (no production implementation of the combined score)**:
- Generative Agents (Park et al. 2023) — specifies the formula; delegates to LangChain's post-ANN approximation
- T-GRAG, RecallM, Adaptive Focus Memory — research prototypes

**The precise gap**: No production system implements temporal decay as a first-class scoring factor INSIDE the ANN traversal (i.e., the ANN graph does not penalize older vectors during traversal). All implementations that combine time and similarity do so as post-ANN re-ranking, which means the ANN candidate set is still biased toward high-cosine matches — old but highly similar memories can still dominate the top-100 candidate pool before re-ranking applies.

## Explanation for the Absence

Three compounding reasons:

**1. ANN algorithm incompatibility.** HNSW, IVF, and other ANN algorithms are designed to find nearest neighbors in a fixed metric space. Injecting a time-dependent score into the HNSW graph traversal would require re-visiting and re-scoring neighbors as the clock ticks, invalidating precomputed distances. The index would need to be rebuilt or queried with a time-parameterized metric, which is not how any standard ANN library (FAISS, hnswlib, nmslib, ScaNN) works. This is a genuine algorithmic barrier.

**2. Post-ANN re-ranking is "good enough" in practice.** If you retrieve top-100 by cosine and then re-rank by `cosine + decay`, the result is usually acceptable for conversation-length memory (tens to hundreds of memories). The failure mode (old but cosine-similar memory beats a newer, slightly less similar one) only becomes visible at scale (thousands of memories) where the top-100 candidate pool may not include recent memories with slightly lower cosine scores.

**3. The LangChain implementation anchored the ecosystem.** Because LangChain shipped a working "time-weighted retriever" in 2023 (inspired by the Generative Agents paper), practitioners accepted post-ANN re-ranking as the standard approach. The limitation (that ANN candidate selection is still pure cosine) is not documented as a known defect — it's treated as the intended design.

## Ideas Extracted

### Idea: in-ann-temporal-decay

- **Novelty**: 0.82 — LangChain does post-ANN re-ranking; no system does true in-ANN temporal weighting
- **Evidence**: 0.75 — LangChain source code confirms post-ANN design; Letta and Graphiti source confirm absence; ANN algorithm incompatibility is real but not insurmountable
- **Feasibility**: 0.55 — Requires either (a) custom ANN traversal with time-parameterized scoring, (b) vector augmentation (append time-encoded dimensions to embedding), or (c) scored ANN with payload-boosted filtering (Qdrant's `filter` + custom `payload_boost` — not yet shipped)
- **Description**: Implement temporal decay as a factor in the ANN candidate selection step itself, not as a post-retrieval re-rank. The practical path in Qdrant or Weaviate: encode recency as additional vector dimensions appended to the embedding (e.g., a learned or hand-crafted temporal encoding), so that the L2/cosine metric naturally penalizes older memories. Alternative: use Qdrant's formula scoring (when available) or implement a custom HNSW with a composite score. For Recall specifically: at retrieval time, retrieve top-k×5 candidates by cosine, then apply `(1.0 - λ)^days_since_creation` before final ranking — this is LangChain's approach but with age-based (not last-access) decay. The LangChain formula should be adapted for Recall with `created_at` not `last_accessed_at`.

### Idea: dual-decay-parameters

- **Novelty**: 0.70 — LangChain uses a single decay_rate; no system uses separate decay curves for different memory types
- **Evidence**: 0.65 — Immunological memory research (Q003 finding) describes two-tier decay; neuroscience literature distinguishes episodic vs. semantic memory decay rates
- **Feasibility**: 0.75 — Low implementation cost; requires only per-memory-type decay_rate metadata
- **Description**: Apply different decay rates to different memory categories. Episodic memories (specific events, conversations) decay fast — `λ_episodic = 0.05` (half-life ~14 days). Semantic memories (learned facts, preferences, patterns) decay slowly — `λ_semantic = 0.002` (half-life ~1 year). Procedural memories (workflows, toolchain knowledge) do not decay. This maps directly to the immune memory model from Q003: effector-like episodic memories have short lifespans; memory B-cell-like semantic memories persist indefinitely under normal conditions.

### Idea: last-access-vs-creation-decay

- **Novelty**: 0.55 — LangChain conflates these; the distinction has not been made explicit in any production system
- **Evidence**: 0.80 — LangChain source code explicitly uses `last_accessed_at`; the semantic difference is clear
- **Feasibility**: 0.90 — Trivial to implement in Recall; requires storing both timestamps
- **Description**: LangChain's decay is based on `last_accessed_at`, which rewards recently retrieved memories (Spotlight model). Age-based decay should use `created_at`, which penalizes old memories regardless of access frequency. For Recall, use `created_at`-based decay as the default and track `last_accessed_at` as a separate "importance boost" factor. The combined formula: `score = cosine_similarity + α × (1-λ)^days_since_creation + β × (1-μ)^days_since_last_access`, where α and β are tunable weights. This separates two distinct signals: how old is this memory, and how recently was it relevant?

## Suggested Follow-ups

1. **Implement LangChain's formula in Recall** as a starting point: retrieve top-k×5 by cosine, re-rank by `cosine_similarity + (1.0 - decay_rate)^days_since_creation`. Use `created_at` not `last_accessed_at`. Measure whether retrieval precision improves on session replay tests.

2. **Test whether the top-100 candidate pool problem is real for Recall's scale.** At Recall's current memory count (likely <10k entries), the post-ANN re-ranking approximation is probably fine. The failure mode matters at 100k+ memories where a recent memory with cosine=0.72 might not make the top-100 candidate pool, losing to older memories with cosine=0.74-0.80.

3. **Explore vector augmentation for in-ANN temporal weighting.** Append a time-encoding dimension to the embedding vector (e.g., sinusoidal encoding of `exp(-λ × age_in_days)` normalized to the same scale as embedding dimensions). This bakes temporal decay into the metric space itself, making it a first-class ANN factor.

4. **Q007 follow-up**: The Letta architecture (separate `"timestamp"` retrieval mode) suggests a clean two-mode design that is one step away from a blended score — implementing the blend at the Letta layer could be done as a PR.

5. **Check Chroma's scoring API** — Chroma's `where` filtering and `include` fields may support payload-based score modification not covered in this search.
