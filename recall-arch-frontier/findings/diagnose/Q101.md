## Evidence

### Module 1: TTL Reaper Race Window

The S5 probe races against TTL hard-deletes if a PG row is deleted between the Neo4j BFS read (Phase 2) and the PG batch join (Phase 3). The BFS latency from Q092 is 2–15ms.

**Rate of TTL hard-deletes**: 5/day = 5.79×10⁻⁵ per second.

**Probability that a TTL delete fires during the probe window** (2–15ms):

| BFS window | P(delete fires) | P(hits sampled node) | P(race FP per probe) |
|-----------|----------------|---------------------|---------------------|
| 2ms (best) | 1.16×10⁻⁷ | 0.0111 | 1.29×10⁻⁹ |
| 15ms (worst) | 8.68×10⁻⁷ | 0.0111 | 9.65×10⁻⁹ |

**Critical insight**: TTL-expiring nodes have NOT been accessed recently. Their `last_access_epoch` is old (≫ tau epochs ago). The BFS epoch predicate `WHERE last_access_epoch >= (current_epoch - tau)` filters them OUT before the PG join. The race window scenario above is the residual from the tiny fraction of TTL expirations that happen to fall on recently-accessed nodes — which is near zero at Tim's TTL=90 days and tau=20 epochs.

**FP rate contribution from TTL reaper: ~1×10⁻⁸ per probe (negligible).**

### Module 2: Tombstone Lag (Dominant False-Positive Source)

A tombstoned memory (PG soft-deleted, Neo4j node awaiting DETACH DELETE) lives in Neo4j with its last_access_epoch set to the epoch at deletion time. It passes the BFS epoch filter if deleted fewer than tau=20 epochs (4 days) ago.

**Rate**: 5% eviction rate × 50 memories/day = 2.5 tombstones/day.

**Mean tombstones in tau window at steady state**: 2.5/day × 4 days = 10 nodes.

**Tombstone fraction of population**: 10 / 4,500 = 0.0022.

**Expected tombstone misses per probe**: 0.0022 × 50 = 0.111.

**P(WARNING triggered by tombstone lag | no real orphans)**:
Using Binomial(50, 0.0022) ≥ 3 misses threshold:

```
P(≥ 3 misses out of 50) = 1.99×10⁻⁴ per probe
```

**Over 30 days (8,640 probes)**:
- Expected false WARNINGs: 1.72
- P(at least one false WARNING in 30 days): **82.1%**

**P(CRITICAL triggered by tombstone lag)**:
```
P(≥ 8 misses out of 50) = 2.94×10⁻¹³ per probe
Expected false CRITICALs per 30 days: 2.54×10⁻⁹ (effectively zero)
```

**Key finding**: S5 WARNING fires spuriously ~1.72 times per 30 days from tombstone lag alone at 288 probes/day. S5 CRITICAL never fires spuriously.

The dual-write pattern (Neo4j node written first, PG row written second) creates a 2–500ms window where a node exists in Neo4j but not yet in PG. At Tim's density (5 sessions/day × 10 memories/session = 50 writes/day):

| Lag | Expected in-flight nodes at probe | P(in-flight hits BFS sample) |

P(WARNING from dual-write race, worst case 500ms): **6.51×10⁻¹³ per probe** — negligible relative to tombstone lag.

### Module 6: Redis Key Patterns and Neo4j Traversal Anomalies

**Redis key patterns that trigger false positives**:

1. `recall:tombstone:{uuid}` — tombstone markers. The corresponding Neo4j node survives until the next cleanup sweep (up to 4 days / tau=20 epochs). If the tombstone marker is present but Neo4j cleanup hasn't run, the BFS returns the node → PG join fails. This is the **dominant false-positive pattern**.

2. `recall:session:{session_id}:*` — session processing keys. An in-flight session-end hook that has deleted PG rows but not yet run Neo4j cleanup creates a transient inconsistency window. At 5 sessions/day and 2–500ms write lag: P ≈ 1×10⁻⁸ per probe (negligible).

3. TTL expiry on `recall:memory:{uuid}` — the key expires → PG hard-delete races probe. Residual probability: 1×10⁻⁸ per probe (epoch filter eliminates the vast majority).

**Neo4j traversal anomalies that trigger false positives**:

1. **Missing tombstone filter in BFS Cypher (code defect in Q092 implementation)**: The current S5 Cypher query (`WHERE last_access_epoch >= ... AND last_access_epoch <= ...`) does NOT filter nodes with `is_tombstoned = true` or `deleted_at IS NOT NULL`. If Neo4j uses a tombstone property to mark pending-delete nodes, these pass the epoch predicate and reach the PG join as misses. The fix is one additional WHERE clause: `AND n.is_tombstoned IS NULL` (or `AND n.deleted_at IS NULL`).

2. **Hub node accumulation of stale edges**: As the graph matures (Month 6–12+), hub nodes accumulate CO_RETRIEVED edges to low-epoch neighbors. The epoch predicate filters these, but if a hub node's seed position in the BFS returns mostly stale neighbors that fail the epoch filter, the BFS returns a small sample (< 5 neighbors), triggering `insufficient_data` rather than a false alarm. This is benign.

### Module 7: Minimum Observation Period (SPRT Analysis)

Using the Sequential Probability Ratio Test (SPRT) with alpha=0.01 (1% false alarm) and 1-beta=0.80 (80% detection power):

| Failure scenario | N_orphans | P(WARNING per probe) | SPRT min probes | Observation time |
|-----------------|-----------|---------------------|----------------|-----------------|
| Acute (PG restore) | 100 | 9.98×10⁻² | 6 | **0.5 hours** |
| Moderate | 50 | 1.82×10⁻² | 49 | 4.1 hours |
| Chronic drift | 25 | 2.77×10⁻³ | 676 | **56 hours (2.4 days)** |

For a **gradual degradation trend** (1 orphan/day accumulation, reaching 30 orphans after 30 days):
- P(single probe WARNING at day 30): 0.0046
- Expected alerts per day at day 30: 1.32 probes/day fire WARNING
- This is detectable as a trend within ~2.4 days of sustained observation, but cannot be distinguished from noise on any single day.

**For the Q081 acute failure (N=100+ orphans)**: the probe detects it within 0.5 hours (6 probes). **No 30-day observation window is required for acute failures.**

**For chronic drift (N < 25 orphans)**: minimum observation period is **2–3 days** to achieve SPRT power of 80%.

**Conclusion**: The 30-day observation window question frames the problem incorrectly for S5. CRITICAL status requires zero observation period (single-probe confidence interval is negligible). WARNING from true orphan injection (N ≥ 100) requires <1 hour. The 30-day window is relevant only for tracking a very slow trend (<10 orphans/week accumulation rate).

### Module 8: Hysteresis Rules to Reduce False-Positive Rate Below 1% per 30 Days

**Current configuration (every 5 min, no hysteresis)**:
- Expected false WARNINGs in 30 days: 1.72
- P(any false WARNING in 30 days): **82%** — fails the 1% target

**Option A: Probe frequency reduction only**:
- Need to drop from 288 probes/day to ~6/day (every 4 hours) to get P < 7%
- Still doesn't reach 1% target without hysteresis

**Option B: Hysteresis only (N consecutive alerts required)**:

| N consecutive | P(FP per probe) | E[FA/30d] | P(any FA/30d) | Detection delay |
|--------------|----------------|-----------|---------------|----------------|
| 1 | 1.99×10⁻⁴ | 1.72 | 82% | 0 min |
| **2** | **3.96×10⁻⁸** | **0.0003** | **0.03%** | **5 min** |
| 3 | 7.88×10⁻¹² | ~0 | ~0 | 10 min |

**N=2 consecutive alerts (5-min probe interval) achieves 0.03% false alarm rate in 30 days — well below 1% target. Detection delay is only 5 minutes (one extra probe).**

**Recommended configuration**:
```
Probe interval:    every 15–30 minutes (reduces unnecessary Neo4j load)
Hysteresis (WARNING): RAISE alarm on 2 consecutive WARNING probes
Hysteresis (CRITICAL): RAISE alarm on SINGLE CRITICAL probe (no hysteresis needed)
Alarm clearing:   CLEAR after 5 consecutive HEALTHY probes
```

**At 30-min interval, N=2 consecutive**:
- P(any false WARNING in 30 days): **0.006%** — 170× below target
- Detection at N=100 orphans: ~50 hours expected (100 probe-pairs × 30 min)
- Detection at CRITICAL (N >> 100): single probe, <30 minutes

**The asymmetry is intentional**: CRITICAL alarms require only one probe because the false CRITICAL rate (2.94×10⁻¹³) is negligible even over years of operation. WARNING alarms require two consecutive probes because the tombstone-lag noise floor makes single-probe WARNINGs unreliable.

S5 CRITICAL (compliance_ratio < 0.85, ≥8 misses in 50) false fires at 2.94×10⁻¹³ per probe. Over 30 days (8,640 probes) the expected count is 2.54×10⁻⁹. S5 CRITICAL can be treated as a zero-false-positive signal: any observed CRITICAL reading represents a genuine acute failure.

[STAT:effect_size] Expected false CRITICAL events in 30 days: 2.54×10⁻⁹

## [FINDING 2] S5 WARNING fires spuriously ~1.7 times per 30 days under current config

At 288 probes/day with no hysteresis, tombstone lag (the dominant noise source) produces P(false WARNING) = 1.99×10⁻⁴ per probe, yielding 1.72 expected false WARNINGs per 30 days and an 82% probability of at least one spurious WARNING in any 30-day window.

## [FINDING 4] S5 WARNING is reliable (SNR > 10) when N ≥ 25 true orphans exist

The signal-to-noise crossover occurs at 25 orphans (0.6% of the 4,500-node graph). Below 25 orphans, tombtone lag noise dominates. Above 25, detection power rises steeply: at N=100 orphans P(single-probe WARNING) = 9.98%, at N=450 orphans P = 88.8%.

## [FINDING 5] Minimum observation period: 0.5 hours for acute failure, 2.4 days for chronic drift

SPRT analysis (80% power, 1% false alarm): acute PG-restore failures (N≥100 orphans) are detectable in 6 probes (0.5 hours at 5-min intervals). Chronic slow-drift scenarios (N=25 orphans, ~5 orphans/week accumulation) require 676 probes (56 hours, 2.4 days) to distinguish from tombstone-lag noise with 80% power. The "30-day observation window" is not operationally justified for S5 — acute detection is sub-hour, and chronic drift requires days, not weeks.

[STAT:effect_size] Acute vs chronic detection time ratio: 56h / 0.5h = 112×
[STAT:n] SPRT with alpha=0.01, 1-beta=0.80; H0: p_noise=1.99×10⁻⁴; H1: p_signal at each N

## [FINDING 6] Dominant false-positive trigger is tombstone lag, not TTL reaper or dual-write race

The tombstone lag mechanism (Neo4j nodes for recently-deleted memories survive within the tau epoch window until cleanup runs) accounts for >99.9% of S5 false-positive probability. TTL reaper races contribute ~1×10⁻⁸ per probe. Dual-write races contribute ~6×10⁻¹³ per probe. The tombstone lag is addressable by: (a) adding `AND n.is_tombstoned IS NULL` to the S5 BFS Cypher — a one-line fix that eliminates the dominant FP source entirely.

[STAT:effect_size] Tombstone lag / all-other-sources ratio: 1.99×10⁻⁴ / (1×10⁻⁸) = 19,900×
[STAT:p_value] FP rate after tombstone Cypher fix: ~0 per probe (residual: 1×10⁻⁸)

## [LIMITATION]

1. **Tombstone rate assumed at 5%/day eviction**: this figure is from Q092 estimation, not empirical measurement. If eviction rate is higher (10–20%), the tombstone-lag FP rate scales proportionally — e.g., at 10% eviction, P(false WARNING/probe) ≈ 3.8×10⁻⁴, doubling the 30-day expected false WARNINGs to ~3.3.

2. **Uniform BFS sampling assumed**: the model assumes random node sampling for BFS seeds. In practice, seeds are weighted by `last_access_epoch` (most recent nodes preferred). Tombstoned nodes that are recently accessed are more likely to be sampled than a uniform draw would suggest, which could increase the actual false-positive rate above the model estimate.

3. **Neo4j tombstone filter not confirmed present**: the analysis assumes tombstoned nodes carry a `is_tombstoned` or `deleted_at` property. If the production implementation uses a different tombstone pattern (e.g., removing edges before removing nodes), the Cypher fix would differ. The one-line fix should be validated against the actual schema.

4. **Tau=20 assumed fixed**: if tau changes (e.g., tuned to 10 or 30), the tombstone window scales accordingly. tau=10 halves the FP rate; tau=30 increases it 50%.

## Ideas Extracted

### Idea: tombstone-cypher-filter-eliminates-s5-noise-floor
- **Novelty**: 0.78 — The identification that the S5 false-positive rate is dominated by a single, fixable code omission (missing tombstone filter in BFS Cypher) rather than an irreducible noise floor from system-level race conditions is a structural insight. The current Q092 Cypher uses epoch bounds as the only filter. Adding `AND n.is_tombstoned IS NULL` to the BFS WHERE clause eliminates the dominant noise source at zero architectural cost. No production AI memory health monitoring system in the survey has been documented as implementing probe-side tombstone exclusion in graph traversals.
- **Evidence**: 0.82 — The tombstone lag mechanism is deterministic: tombstoned nodes within the tau epoch window pass the epoch predicate, enter the PG join, and fail because PG rows are already deleted. This is provable from first principles. The quantification (P=1.99×10⁻⁴ per probe, 82% chance of false alarm in 30 days) is derived from Binomial(50, 0.0022) with well-characterized inputs (2.5 tombstones/day, tau=4 days, 4,500 nodes). The fix (single WHERE clause) is standard Cypher practice.
- **Feasibility**: 0.97 — One Cypher clause addition to the S5 probe in `health_signals.py`. No schema change. No new property required if tombstoned nodes already carry a `deleted_at` or `is_tombstoned` flag. If not, add the flag at tombstone creation time (2-line change to the delete handler).
- **Description**: Add `AND n.is_tombstoned IS NULL` (or `AND n.deleted_at IS NULL`) to the BFS WHERE clause in the S5 probe Cypher. This prevents tombstoned nodes from entering the PG join entirely. After this fix, the S5 false-positive rate drops from 1.99×10⁻⁴ per probe to ~1×10⁻⁸ per probe — reducing expected false WARNINGs per 30 days from 1.72 to 0.00009. This single-line fix transforms S5 from an alert that fires spuriously every month into one that fires spuriously at most once per 30 years.

### Idea: asymmetric-hysteresis-by-severity-level
- **Novelty**: 0.71 — Applying different hysteresis rules to WARNING vs CRITICAL status based on their statistically different false-positive rates is a novel threshold design pattern for distributed health monitors. The standard practice is uniform hysteresis (N consecutive alerts for all severity levels). The mathematical justification here is structural: CRITICAL has a 3×10⁷-year mean time between false alarms (no hysteresis needed), while WARNING has an 8.7-day mean time between false alarms (hysteresis required). Applying the same N=2 hysteresis to CRITICAL adds unnecessary detection latency to the system's most urgent signal.
- **Evidence**: 0.79 — The asymmetry follows directly from the threshold structure: CRITICAL requires ≥8/50 misses (p=2.94×10⁻¹³), WARNING requires ≥3/50 misses (p=1.99×10⁻⁴). The ratio is 6.8×10⁸. This 9-order-of-magnitude difference in false-positive rate fully justifies different hysteresis policies. Statistical power is maintained because CRITICAL alarms also have much higher detection probability (P=81% at N=900 orphans vs 9.98% for WARNING at N=100).
- **Feasibility**: 0.94 — Implementation is a two-branch conditional in the alarm state machine: `if compliance_ratio < CRITICAL_THRESHOLD: raise_alarm_immediately()` vs `if compliance_ratio < WARNING_THRESHOLD: increment_consecutive_count(); if consecutive >= 2: raise_alarm()`. No new infrastructure. Compatible with the existing Q092 FastAPI endpoint structure.
- **Description**: Implement a split hysteresis policy in the S5 state machine: CRITICAL alarms are raised immediately on the first probe (single-event confirmation, ~0 false alarm rate). WARNING alarms require N=2 consecutive probes before raising (0.03% false alarm rate in 30 days vs 82% with single-probe). The alarm clears after M=5 consecutive HEALTHY probes (providing hysteresis on the clearing edge to prevent alert flapping). The justification is quantitative: the 9-order-of-magnitude difference in per-probe false alarm rates between WARNING and CRITICAL makes uniform hysteresis a suboptimal design.

### Idea: sprt-cusum-trend-detector-for-s5
- **Novelty**: 0.73 — Applying Sequential Probability Ratio Testing (SPRT) or CUSUM as a continuous drift detector on top of the S5 compliance_ratio signal is not present in any reviewed AI memory system health monitoring implementation. The use case is chronic drift detection: when compliance_ratio slowly degrades (1 orphan/week accumulation), no single probe crosses the WARNING threshold, but the cumulative deviation from the noise floor is statistically significant. SPRT detects this within 2.4 days vs the 30-day observation window the original question assumes.
- **Evidence**: 0.75 — SPRT and CUSUM are well-established in statistical process control (SPC) with extensive peer-reviewed literature. The specific application to health probe data (Bernoulli observations with a known noise floor) is a textbook use case. The SPRT minimum sample sizes computed above (N=6 probes for acute, N=676 probes for chronic) are derived from Wald's sequential analysis equations with documented convergence guarantees.
- **Feasibility**: 0.72 — SPRT on Bernoulli observations requires maintaining a running log-likelihood ratio — approximately 10 lines of Python. The state (current S_n, hypothesis decision boundary) is stored in Redis or the health endpoint's in-process state. The CUSUM variant requires a single float (cumulative sum) per probe. No external libraries required. The implementation fits within the existing `health_signals.py` module.
- **Description**: Add a CUSUM or SPRT drift layer above the S5 threshold logic. The cumulative sum S_n = Σ(X_i - p_noise) accumulates over probes; when S_n exceeds a detection threshold h, a `CHRONIC_DRIFT` alarm is raised. Under H0 (no orphans), E[S_n] ≈ 0. Under H1 (N=25 orphans), E[S_n] drifts upward at rate (p_signal - p_noise) per probe and crosses h in ~676 probes (56 hours). This provides a third alarm category below WARNING: `TREND_DETECTED` — the system is degrading but not yet at threshold, and the trend will reach WARNING in an estimated time horizon.
