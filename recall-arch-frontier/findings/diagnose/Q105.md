## Synthesis

### Pre-Reading: What Makes This Restore Different

A CasaOS snapshot restore that rolls back PostgreSQL is NOT a clean system restore. Qdrant runs on a separate Docker volume and is not rolled back. Neo4j retains all graph nodes written after the backup point. Redis retains the epoch counter at its current value. The result: the canonical memory store (Qdrant) contains logically-deleted content that will be injected directly into Claude's context on every relevant query — silently, with no error, and with no health signal to detect it.

The standard Q077 runbook addresses epoch integrity (Redis vs PG) but does NOT address the three-store consistency problem. This runbook closes all three gaps:
1. Neo4j orphan nodes that pass the epoch filter (Q081)
2. Qdrant orphan vectors that are returned directly as valid memories (Q086)
3. HNSW graph degradation after bulk deletion that requires explicit optimization (Q100)

**Qdrant is the canonical store.** PG is the audit index. Orphaned Qdrant vectors are not silently skipped — they are reconstructed into full Memory objects and injected into Claude. This is an active data corruption failure, not a silent skip.

## Part 2: Recovery Steps

### GO/NO-GO Notation

Each step includes a gate:
- **AUTO**: Executes automatically (or can be run without additional confirmation because it is idempotent/non-destructive/reversible).
- **GO/NO-GO**: Requires the operator to review the output of the prior step and explicitly decide to proceed. These steps are irreversible.

### Step 1 — Activate Vector-Only Retrieval Mode (Circuit Breaker)

**Gate**: AUTO — safe, reversible (2-hour TTL)
**Wall-clock time**: < 5 seconds

```bash
redis-cli -h 192.168.50.19 SET recall:retrieval_mode vector_only EX 7200
```

**Expected output**: `OK`

**Why this first**: Any Claude session that starts during the recovery window will use Qdrant-only retrieval. Neo4j graph traversal (the source of orphan injection) is suspended. This stops active data corruption immediately while the rest of the recovery proceeds. The 2-hour TTL ensures retrieval is not permanently degraded if Step 10 is delayed.

**Failure signal**: `redis-cli` command times out or returns ERROR. Check Redis container status:
```bash
ssh nerfherder@192.168.50.19 'docker ps | grep redis'
```

**Go/no-go gate**: PASS automatically. No orphan injection is possible while this flag is set.

### Step 2 — Take Pre-Cleanup Qdrant Snapshot (Rollback Checkpoint)

**Gate**: AUTO — must complete before any destructive action
**Wall-clock time**: 2–10 seconds (at ~5,400 points and 35 MB on NVMe)

```bash
# Via Qdrant REST API
curl -s -X POST http://192.168.50.19:6333/collections/memories/snapshots \
  | python3 -m json.tool
```

**Expected output**: JSON with `"result": {"name": "memories-YYYYMMDD-HHMMSS.snapshot", ...}`

Record the snapshot name. Store it:
```bash
# Store snapshot name for rollback reference
SNAPSHOT_NAME=$(curl -s -X POST http://192.168.50.19:6333/collections/memories/snapshots \
  | python3 -c "import json,sys; print(json.load(sys.stdin)['result']['name'])")
echo "Rollback snapshot: $SNAPSHOT_NAME"
redis-cli -h 192.168.50.19 SET recall:pre_cleanup_qdrant_snapshot "$SNAPSHOT_NAME" EX 86400
```

**Why before deletion**: Qdrant is the canonical store. If the Qdrant cleanup in Step 5 over-deletes (wrong backup_timestamp), the only recovery path is snapshot restore. Without this snapshot, Qdrant over-deletion has no programmatic recovery. Taking the snapshot converts the highest-risk action in this runbook from "irreversible" to "recoverable in ~30 seconds."

**Failure signal**: API returns 404 (collection not found) or 500. Check Qdrant container:
```bash
ssh nerfherder@192.168.50.19 'docker ps | grep qdrant'
```

**Go/no-go gate**: Snapshot name must be recorded. Do not proceed to Step 3 without confirming the snapshot was created.

**Go/no-go gate**: `GET recall:global_epoch` must return `E+1` before proceeding. If it returns a value higher than `E+1`, another process may be incrementing the epoch — halt and investigate before continuing.

### Step 4 — /reconcile Dry Run (Count Verification Gate)

**Gate**: GO/NO-GO — operator must review counts before destruction
**Wall-clock time**: 5–30 seconds (depends on Neo4j query time for batch count)

```bash
# Dry run — repair=false by default, shows what WOULD be deleted
curl -s "http://192.168.50.19:8200/reconcile?mode=post_restore&backup_timestamp=T&pg_max_epoch=E&repair=false" \

```

Substitute actual T (ISO-8601 backup timestamp) and E (pg_max_epoch from PC-3).

**Expected output**:
```json
{
  "mode": "post_restore",
  "dry_run": true,
  "neo4j_would_delete": 847,
  "qdrant_would_delete": 861,
  "duration_ms": 4200
}
```

**Go/no-go decision criteria**:

- `neo4j_would_delete > 0` AND `qdrant_would_delete > 0`: counts are non-zero, confirming the predicates are matching orphaned records. Proceed.
- Both counts are 0: the predicates did not match anything. Either backup_timestamp T is incorrect (too recent), or pg_max_epoch E is wrong. STOP. Re-verify T and E from PC-2 and PC-3 before proceeding.
- `|neo4j_would_delete - qdrant_would_delete| / max(neo4j, qdrant) > 0.10`: counts differ by more than 10%. This indicates a pre-existing secondary inconsistency (prior incomplete cleanup, or write failures during the backup gap). Investigate the discrepancy before running repair=true.
- Counts are in the expected range for the backup gap size: for an 80-day gap at 5 sessions/day and 20 memories/session, expect ~1,600 orphaned records. Large deviations are a signal to double-check T.

**Failure signal**: API returns 404 (endpoint not yet deployed) or 422 (parameter validation error — check ISO-8601 format for backup_timestamp). If mode=post_restore is not yet implemented, fall back to the manual three-step procedure in Appendix A.

**Go/no-go gate**: Operator must see plausible non-zero counts with < 10% divergence between stores. Do not proceed to Step 5 without explicit decision.

### Step 5 — Execute /reconcile (Neo4j + Qdrant Cleanup)

**Gate**: GO/NO-GO — IRREVERSIBLE. Neo4j DETACH DELETE and Qdrant vector deletion cannot be undone without the snapshot from Step 2.
**Wall-clock time**: 5–60 seconds (depends on orphan count; at 1,600 orphans across both stores, expect ~15–30 seconds)

```bash
# Execute cleanup — repair=true
curl -s -X POST "http://192.168.50.19:8200/reconcile?mode=post_restore&backup_timestamp=T&pg_max_epoch=E&repair=true" \
  | python3 -m json.tool
```

**Expected output**:
```json
{
  "mode": "post_restore",
  "repair": true,
  "dry_run": false,
  "neo4j_deleted_count": 847,
  "qdrant_deleted_count": 861,
  "duration_ms": 18400,
  "status": "complete",
  "warnings": []
}
```

**Interpreting warnings**:
- `"status": "partial_failure", "failed_step": "qdrant"`: Neo4j cleanup succeeded, Qdrant failed. The system is now in asymmetric state (Neo4j clean, Qdrant still has orphans). Re-run the full endpoint — it is idempotent. Neo4j will skip (0 deleted), Qdrant will retry. Do not proceed to Step 6 until `status: "complete"`.
- `"status": "partial_failure", "failed_step": "neo4j"`: Neo4j failed. Nothing was mutated. Check Neo4j container health and retry the full step.
- Warnings about count discrepancy > 10%: proceed, but log for investigation after recovery is complete.

**Idempotency**: The endpoint is safe to call twice with the same parameters. A second call returns `{neo4j_deleted_count: 0, qdrant_deleted_count: 0}` without error. Retry freely until `status: "complete"`.

**Failure signal**: API returns 500 or connection timeout. Check both Neo4j and Qdrant containers:
```bash
ssh nerfherder@192.168.50.19 'docker ps | grep -E "neo4j|qdrant"'
```

**Go/no-go gate**: `status` must be `"complete"` with non-zero deleted counts matching the dry-run estimates from Step 4. Do not proceed to Step 6 if status is `"partial_failure"`.

### Step 6 — Force Qdrant HNSW Optimization

**Gate**: AUTO — mandatory; do not skip even if Qdrant's automatic optimizer is pending
**Wall-clock time**: 500 ms – 2 seconds (at 360 surviving points on CasaOS NVMe; longer for larger corpora)

At Tim's steady-state 360-point corpus, orphan deletion of ~140 points produces δ = 0.28 deletion fraction. Without explicit optimization:
- ANN recall@10 is estimated at 0.73 (model-derived in Q100)
- ~2–3 of 10 retrieved memories will be wrong or missing
- The automatic Qdrant optimizer (fires at δ > 0.20) may not have triggered yet and has non-deterministic timing even when triggered

```bash
# Force optimize_collection via Qdrant REST API
curl -s -X POST "http://192.168.50.19:6333/collections/memories/index" \
  | python3 -m json.tool

# Poll for optimizer completion
for i in $(seq 1 20); do
  STATUS=$(curl -s "http://192.168.50.19:6333/collections/memories" \
    | python3 -c "import json,sys; r=json.load(sys.stdin); print(r['result']['optimizer_status']['status'])")
  echo "[$i] Optimizer status: $STATUS"
  if [ "$STATUS" = "ok" ]; then
    echo "Optimization complete"
    break
  fi
  sleep 1
done
```

**Expected output**: After polling, `optimizer_status.status == "ok"`.

**Wall-clock by corpus size**:
- 360 surviving points (Tim's 80-day gap scenario): 500 ms – 2 s
- 500 points (full steady-state): 1 – 3 s
- 7,300 points (Year 1 maximum): 30 – 60 s (Q086 estimate)

**Why this cannot be skipped**: The Qdrant delete operation marks points as "soft deleted" — they remain as ghost routing nodes in the HNSW graph. LLPC calls before optimization traverses through these ghost nodes, wasting traversal budget and degrading ANN recall. Forcing optimization converts the non-deterministic background rebuild into a deterministic, audited step.

**Failure signal**: Optimizer status stays `"running"` for more than 60 seconds. Check Qdrant container logs:
```bash
ssh nerfherder@192.168.50.19 'docker logs --tail=50 recall_qdrant_1'
```

**Go/no-go gate**: `optimizer_status.status == "ok"` must be confirmed before proceeding to Step 7.

## Part 5: Partial Failure Decision Trees

### Scenario A: Neo4j reconcile fails during Step 5

**Observed**: Step 5 returns `"status": "partial_failure", "failed_step": "neo4j", "neo4j_deleted_count": 0, "qdrant_deleted_count": 0`

This means the Neo4j step failed before any mutations occurred. Nothing was changed.

**Decision tree**:
```
Neo4j step failed before any mutations
│
├── Check Neo4j container status
│   ssh nerfherder@192.168.50.19 'docker ps | grep neo4j'
│   docker logs recall_neo4j_1 --tail=50
│
├── If Neo4j container is down:
│   Restart: ssh nerfherder@192.168.50.19 'docker compose restart neo4j'
│   Wait 30s for Neo4j startup
│   Re-run Step 5 with identical parameters (idempotent)
│
├── If Neo4j is up but query times out:
│   The batch DETACH DELETE may be too large for a single transaction.
│   Fall back to manual batch deletion (Appendix A, Step N4-manual):
│     cypher-shell -a bolt://192.168.50.19:7687 -u neo4j -p password \
│       "MATCH (n:Memory) WHERE n.last_access_epoch > $E WITH n LIMIT 500 DETACH DELETE n RETURN count(*)"
│   Repeat until the query returns 0 deleted.
│   Then re-run Step 5 with repair=true (Qdrant step will run; Neo4j step will return 0 idempotently).
│
└── If Neo4j returns authentication error:
    Check Neo4j credentials in the Recall API configuration.
    The endpoint requires Neo4j write credentials, not read-only.
```

**Impact of this failure**: No data was mutated. Qdrant orphans remain. Neo4j orphans remain. Vector-only mode from Step 1 is still protecting against injection. Safe to retry without additional risk.

**Recovery timeline impact**: +5–15 minutes depending on root cause.

## Part 6: Rollback Procedure (If Cleanup Was Wrong)

### Detecting over-deletion

Run within 1 session after cleanup:
```bash
# Compare Qdrant count to expected post-cleanup count
curl -s http://192.168.50.19:6333/collections/memories \
  | python3 -c "import json,sys; print('Qdrant:', json.load(sys.stdin)['result']['points_count'])"
psql -h 192.168.50.19 -U recall_user -d recall_db -tAc "SELECT COUNT(*) FROM memories;"
```

If Qdrant count is significantly lower than PG count (> 15% divergence), over-deletion may have occurred.

Additional signal: Tim notices Claude does not recall things it should — this is the most direct detection signal.

### Qdrant rollback from snapshot

```bash
# Get snapshot name (stored in Step 2)
SNAPSHOT_NAME=$(redis-cli -h 192.168.50.19 GET recall:pre_cleanup_qdrant_snapshot)
echo "Restoring from: $SNAPSHOT_NAME"

# Restore via Qdrant REST API
curl -s -X POST "http://192.168.50.19:6333/collections/memories/snapshots/$SNAPSHOT_NAME/recover" \
  | python3 -m json.tool
```

After Qdrant restore, re-run standard reconcile (NOT post_restore mode) to rebuild Neo4j from Qdrant state:
```bash
curl -s -X POST "http://192.168.50.19:8200/reconcile?repair=true" | python3 -m json.tool
```

**Limitation**: CO_RETRIEVED edges between memories are not stored in Qdrant payload. After Neo4j re-populate from Qdrant, the causal graph has nodes but no edges. Edges reconstruct naturally over subsequent sessions.

### Idea: three-gate-recovery-sequence

- **Novelty**: 0.75 — The identification of three mandatory gates in the post-restore sequence — dry-run count verification (Step 4), canary ANN recall test (Step 7), and health signal green (Step 9) — as a pattern for multi-store recovery validation is novel. ### Idea: vector-only-mode-as-pre-recovery-safety-gate

- **Novelty**: 0.78 — Making the vector-only retrieval mode activation the FIRST step in the recovery sequence (before any diagnosis, before any cleanup) is a specific design choice that differs from typical runbooks where the safety gate comes after diagnosis. The rationale: the cost of a false positive (activating vector-only mode when no PG restore occurred) is a 2-hour TTL degradation of graph-assisted retrieval — low cost. The cost of NOT activating it immediately (orphan injection continues during the 8–12 minute recovery window) is active data corruption for every Claude session started during that window. This asymmetry justifies front-loading the safety action.
- **Evidence**: 0.74 — Q086 established that orphaned Qdrant vectors are injected directly into Claude's context (no PG join, no silent skip). Q098 established the automated circuit-breaker pattern. The 2-hour TTL from Q098 ensures the gate auto-expires if the operator completes recovery in under 2 hours (which the 8–12 minute target guarantees). The asymmetry argument (low cost of false positive, high cost of missed orphan injection) is structurally sound given that retrieval remains functional in vector-only mode — just without graph-assisted reranking.
- **Feasibility**: 0.92 — One Redis SET command. Already specified in Q098. Requires LLPC to check `recall:retrieval_mode` at session start and conditionally skip Neo4j traversal (~5 lines of code). The 2-hour TTL is a safety net for the case where Step 10 is delayed or forgotten.
- **Description**: Activate `recall:retrieval_mode = vector_only` with a 2-hour TTL as the first action in the post-restore runbook, before any diagnosis, before any cleanup. This immediately stops orphan injection into Claude's context for any sessions started during the recovery window. The gate is reversible (Step 10 clears it), has a safety TTL (auto-expires if recovery completes normally), and incurs only a graceful degradation cost (graph-assisted reranking suspended, core Qdrant retrieval intact). Front-loading this action converts the recovery window from "actively corrupting" to "safely degraded."
