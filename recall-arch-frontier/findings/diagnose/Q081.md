# Finding: Q081 — Compound Failure: PG Restore + Epoch Filter Gap Creates Unhandled Neo4j-to-PG Join Misses

**Question**: When both the Q074 failure (PG restore from backup) and Q078's epoch filter gap occur simultaneously, a compound failure emerges: after epoch retraction (Option A recovery — Redis set to pg_max+1=101), Neo4j retains nodes and CO_RETRIEVED edges that reference UUID rows deleted by the PG restore. The Q078 fix (Cypher epoch filter) prevents retrieval of stale nodes, but does NOT clean orphaned Neo4j nodes. When LLPC tries to join the Neo4j graph results against PostgreSQL (to fetch memory content), it will attempt to fetch rows with UUIDs that no longer exist in the restored PG. What is the behavior when this join fails (NULL result, error, silent skip?), is this an unhandled exception path, and does Q077's runbook need a post-restore Neo4j reconciliation step?

**Verdict**: BREAKTHROUGH
**Severity**: High

## Compound Failure Analysis

### State after PG restore + Option A epoch retraction

Starting conditions after the Q074 scenario plays out and Option A recovery is applied:

| Store | State |
|-------|-------|
| PostgreSQL | Restored to backup state; contains memories with `last_access_epoch` in [0, 100] only |
| Redis | Set to `recall:global_epoch = 101` (pg_max + 1 per Option A) |
| Neo4j | Unmodified — contains ALL nodes and CO_RETRIEVED edges that existed before the restore, including nodes for memories created between epoch 101 and 500 |

The critical asymmetry: the PG restore is a point-in-time operation that selectively reverts one store. Neo4j and Redis are not restored. Neo4j retains nodes for ~400 sessions worth of memories that no longer exist as rows in PostgreSQL.

### Step-by-step compound failure sequence

**Step 1 — Epoch retraction applied.**
Redis is set to 101. The Q074 recovery procedure in Q077's runbook (S4 critical branch) covers the Redis SET command. The runbook stops here — it does not mention Neo4j.

**Step 2 — LLPC session starts. Epoch filter evaluated.**
Current epoch = 101. The Q078 epoch filter (if implemented as the Cypher predicate `WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)`) evaluates as: `last_access_epoch >= (101 - 20) = 81`.

**Step 3 — Neo4j traversal returns orphaned nodes.**
Neo4j nodes for memories created between epoch 101 and 500 have `last_access_epoch` values in [101, 500]. These values are all >= 81. The epoch filter PASSES them. This is the core of the problem established in the question context: the epoch filter is designed to exclude stale (old) memories but not future-relative (post-restore) memories. After epoch retraction, memories from epochs 101–500 are not "stale" from the filter's perspective — they have high epoch values — but they are physically absent from PostgreSQL.

Specifically:
- Nodes with `last_access_epoch` in [81, 100]: exist in PG (within backup). Filter passes them. PG join succeeds. No problem.
- Nodes with `last_access_epoch` in [101, 500]: exist in Neo4j only. Filter passes them (>= 81). PG join fails — rows do not exist in the restored database.

**Step 4 — LLPC attempts PostgreSQL join for graph-traversal candidates.**
The LLPC Python code (implied by Q057's architecture) performs a query of the form:

```python
neighbor_ids = neo4j.run_cypher(seed_cypher)  # returns UUIDs including orphaned ones
rows = await pg_pool.fetch(
    "SELECT id, content, importance_score, last_access_epoch FROM memories WHERE id = ANY($1)",
    neighbor_ids
)
```

For orphaned UUIDs (epochs 101–500), the `WHERE id = ANY($1)` returns no matching row. These UUIDs are silently absent from the result set.

**Step 5 — Join miss handling determines failure mode.**
What happens next depends on how the Python LLPC code processes the result. This is the critical branch point analyzed in the next section.

**Step 6 — Context window receives partial or incorrect results.**
Even in the best case (silent skip), the LLPC candidate set is drawn from a graph that references a partially nonexistent database. The causal graph is structurally invalid: edges point from valid nodes to ghost nodes. The retrieved context is whatever survives the join — missing the orphaned memories, and potentially biased toward the pre-backup memory set in ways that do not reflect actual session co-retrieval patterns from the backup era.

## PostgreSQL Join Behavior for Missing UUIDs

### The join semantics

A PostgreSQL `WHERE id = ANY($1)` with a list of UUIDs simply omits rows for UUIDs that do not exist. It does not raise an error. It does not return a NULL placeholder row. It silently returns fewer rows than were requested.

This is standard SQL behavior: the result set contains exactly the rows that match the predicate. Missing rows generate no error, no warning, and no indication that they were requested.

### How Python async pg libraries handle this

The two libraries most likely in use for a FastAPI + asyncpg stack:

**asyncpg** (`pg_pool.fetch()`): Returns a list of `Record` objects for rows that exist. If 10 UUIDs are passed and 4 do not exist in the restored PG, `fetch()` returns 6 records. No exception is raised. No indication that 4 UUIDs were requested and not returned. The caller receives fewer results than requested and has no way to know which UUIDs were missing unless it explicitly computes the difference between `neighbor_ids` and `{row['id'] for row in rows}`.

**psycopg2 / psycopg3**: Same behavior. The `IN` or `= ANY` predicate is a filter; missing rows are silently excluded.

### Is this an unhandled exception path?

The missing-UUID join does NOT raise an exception under normal library behavior. This means:

1. **No crash**: The LLPC code does not fail. It continues with a reduced candidate set.
2. **No alert**: No exception reaches the FastAPI error handler. No log entry of class ERROR or CRITICAL is generated for the missing UUIDs.
3. **No health signal**: None of the four health signals in Q071 (S1–S4) monitors for the case where Neo4j-to-PG join resolution rate falls below 100%. S4 monitors epoch inversion (redis_epoch < pg_max), not the post-restore epoch retraction state.
4. **Silent data corruption of context window**: The LLPC candidate set is silently smaller than the graph suggests. If 15 graph-traversal candidates are returned by Neo4j and 6 are orphaned, LLPC ranks and returns memories from the 9 valid ones — but the ranking is distorted because co-retrieval relationships that informed the graph topology no longer have their referenced nodes present.

The failure is **a silent skip producing undetected context window corruption**. This is the most dangerous class of failure: it does not surface as an error, it does not trigger any existing health monitor, and it produces results that appear valid but are incomplete in a structurally meaningful way.

### Exception path if code does explicit UUID validation

If the LLPC code explicitly validates that all requested UUIDs were returned — a defensive pattern — the behavior changes:

```python
returned_ids = {row['id'] for row in rows}
missing_ids = set(neighbor_ids) - returned_ids
if missing_ids:
    logger.warning(f"Neo4j graph references {len(missing_ids)} UUIDs missing from PG: {missing_ids}")
```

This would surface the failure as a WARNING log entry but still not an exception. The correct defensive addition is to also flag this condition to the health system. Without this code, the failure is completely invisible.

### The asymmetry with normal missing-UUID cases

In normal operation, a UUID might be missing from PostgreSQL for legitimate reasons: a memory was tombstoned and the PG row was hard-deleted after the tombstone TTL expired. In that case, the Neo4j node would also eventually be pruned (if a cleanup job exists) or would be filtered by its own epoch value (if tombstone epoch is old enough). The PG restore scenario is different: the missing UUIDs have high, recent-looking epoch values that pass the epoch filter, and there is no tombstone record to explain their absence. The join miss is anomalous, not a routine pruning artifact.

## Epoch Filter Interaction

### Does Q078's Cypher epoch filter protect against orphaned Neo4j nodes after PG restore?

No. The Q078 epoch filter is designed to exclude stale memories — those whose `last_access_epoch` is more than tau sessions behind the current epoch. After epoch retraction to 101, the filter threshold is `last_access_epoch >= 81`.

Orphaned nodes have `last_access_epoch` values in [101, 500]. These values are:
- All greater than 81 (the filter threshold)
- Greater than the current epoch (101) — they are from the "future" relative to the post-retraction epoch counter

The epoch filter correctly removes memories accessed at epochs [0, 80] (too old relative to the new epoch=101 current). It provides zero protection against memories accessed at epochs [101, 500] (above the current epoch, but not filtered because the filter is a lower bound, not an upper bound).

### The filter's conceptual mismatch

The Q062 epoch filter was designed for a single monotonic failure mode: Redis reset causes epoch to jump backward, making old memories appear current. The filter solution is a lower bound: reject memories below current_epoch - tau.

The PG restore + epoch retraction produces a novel failure mode: epoch is deliberately set backward (101 from 500), and Neo4j retains nodes from the future (101–500). The filter has no upper bound to catch this. No existing filter predicate in the Recall architecture targets `last_access_epoch > current_epoch` as a validity condition, because under normal operation that condition is impossible — memories cannot be accessed in future epochs.

### The correct epoch predicate after retraction

The full valid epoch range after Option A recovery (Redis set to 101, PG has epochs 0–100) is:

```
[0, current_epoch] = [0, 101]
```

Nodes with `last_access_epoch > current_epoch` (i.e., > 101) are from the pre-retraction epoch space and should be rejected. The correct compound Cypher predicate is:

```cypher
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)
  AND neighbor.last_access_epoch <= $current_epoch
```

The upper bound `<= $current_epoch` is redundant during normal operation (where epochs are monotonically increasing and no node can have a future epoch). But it is essential correctness protection during and after an epoch retraction event. Without it, the epoch filter is not a valid range check — it is only a lower bound, and the post-retraction state creates a class of nodes that violate the epoch space but pass the lower bound.

### Duration of the vulnerability window

Even with the upper-bound predicate added to Cypher queries, the orphaned nodes in Neo4j remain indefinitely until explicit cleanup. Every session during the post-retraction period (epoch 101, 102, 103, ...) advances the current_epoch. As current_epoch rises, the upper bound `<= current_epoch` gradually includes more and more orphaned nodes:

- At current_epoch = 101: nodes with last_access_epoch in [81, 101] pass. Orphaned nodes (102–500) are blocked by the upper bound.
- At current_epoch = 150: nodes with last_access_epoch in [130, 150] pass (working-set). Orphaned nodes (151–500) still blocked. But orphaned nodes at [130–150] would now pass if they fall in the window — except new sessions generate epochs 101, 102, etc., so no new orphaned nodes exist in [130–150] unless Neo4j's epoch values were set by a system that advanced separately.

Actually: after epoch retraction to 101, new session summaries will write new PG rows and update Neo4j node properties with epoch values 102, 103, etc. These new nodes ARE in PG (they are written post-retraction) and are valid. The orphaned nodes remain at their pre-retraction epoch values (101–500) permanently — their `last_access_epoch` in Neo4j is never updated because no session references them (they don't exist in PG and LLPC never retrieves them). The upper bound predicate would block them permanently as long as current_epoch < their last_access_epoch.

This means: if orphaned nodes have `last_access_epoch` in [101, 500], and the new epoch counter advances from 101 to 500 over time (i.e., ~400 more sessions — roughly the same number that were lost in the backup gap), they will be re-admitted through the upper bound one-by-one as current_epoch grows. Each orphaned node becomes "active" again from the epoch filter's perspective when current_epoch catches up to its stored epoch. At that point, the PG join will miss again.

**The upper-bound predicate is a short-term mitigation, not a permanent fix.** The permanent fix requires Neo4j orphan cleanup.

## Recovery Procedure for Neo4j Orphans

### Identifying orphaned Neo4j nodes

After PG restore + epoch retraction to epoch E_new (101), all Neo4j nodes with `last_access_epoch > E_new` reference memories that do not exist in the restored PostgreSQL. The identification query:

```cypher
MATCH (n:Memory)
WHERE n.last_access_epoch > $e_new
RETURN n.id, n.last_access_epoch
ORDER BY n.last_access_epoch ASC
```

This returns all orphaned node UUIDs with their epoch values. The result set can be cross-validated against PostgreSQL:

```python
# Get all Neo4j node IDs with last_access_epoch > e_new
orphan_candidates = neo4j.run(
    "MATCH (n:Memory) WHERE n.last_access_epoch > $e RETURN n.id",
    e=e_new
)
candidate_ids = [r['n.id'] for r in orphan_candidates]

# Check which actually exist in PG (should be none, but validate)
existing = await pg_pool.fetch(
    "SELECT id FROM memories WHERE id = ANY($1)",
    candidate_ids
)
existing_ids = {row['id'] for row in existing}
confirmed_orphans = set(candidate_ids) - existing_ids
```

In the pure Q074 scenario (PG fully restored, no partial writes after restore), `confirmed_orphans` should equal `candidate_ids` — every Neo4j node with epoch > e_new is an orphan.

### Deleting orphaned nodes and edges

```cypher
MATCH (n:Memory)
WHERE n.last_access_epoch > $e_new
DETACH DELETE n
```

`DETACH DELETE` removes both the node and all its incident CO_RETRIEVED edges. This is the correct operation because:
1. The node does not exist in PG — it cannot be retrieved and should not influence graph traversal.
2. Its edges are semantically invalid — they represent co-retrieval relationships that involved a memory that no longer exists.
3. Leaving ghost edges in place biases the causal graph toward pre-restore session patterns, which may not be valid given the data loss.

### Batch operation for large orphan counts

If the epoch gap was large (e.g., 400 sessions, potentially thousands of orphaned nodes), a single `DETACH DELETE` across all orphans may be a heavy write operation for Neo4j. The recommended approach is batch deletion:

```cypher
MATCH (n:Memory)
WHERE n.last_access_epoch > $e_new
WITH n LIMIT 500
DETACH DELETE n
RETURN count(*) AS deleted
```

Run this in a loop until it returns 0. At ~800 new nodes/month (Q078's edge growth analysis implies comparable node growth), a 400-session gap may yield ~600–800 orphaned nodes — manageable in a single pass for Tim's deployment scale. For larger deployments, batch iteration is essential.

### Timing: when to run the cleanup

The Neo4j cleanup should run immediately after Option A epoch retraction, before any new LLPC sessions query the causal graph. If sessions run between the PG restore and the Neo4j cleanup, those sessions may receive orphaned node references in their retrieval results (silent skip behavior as described above). The cleanup is not write-blocking to PostgreSQL or Redis — it is a standalone Neo4j operation.

### Why the runbook gap is high severity

Without the reconciliation step, Tim can correctly follow the Q077 and Q074 recovery procedures (set Redis to pg_max + 1, verify retrieval resumes) and believe the system is fully recovered — while Neo4j silently contains hundreds of orphaned nodes that:

1. Pass the epoch filter (last_access_epoch > current_epoch is blocked, but as current_epoch advances they will be re-admitted one batch at a time)
2. Cause silent PG join misses in every session where an orphaned node is returned by causal graph traversal
3. Bias causal graph topology toward co-retrieval patterns from the lost-memory era, inflating the apparent connectivity of nodes that were frequently co-retrieved with now-missing memories
4. Are completely invisible to all four health signals in Q071

The system appears healthy by every observable metric but is producing silently corrupted retrieval results. Tim would have no way to detect this without explicitly querying Neo4j for nodes with `last_access_epoch > current_epoch` — a diagnostic step that does not appear anywhere in the current documentation.

## Ideas Extracted

### Idea: post-restore-neo4j-orphan-reconciliation-step
- **Novelty**: 0.82 — The specific failure mode where a PG point-in-time restore creates orphaned nodes in a co-located graph store (Neo4j) that then pass an epoch lower-bound filter due to their high epoch values is not documented in reviewed AI memory system literature. Graphiti and MemGPT do not expose this failure because they use single-store or tightly-coupled store architectures. The Recall multi-store design (PG + Neo4j + Redis + Qdrant) makes this cross-store consistency gap structurally possible and the combination of epoch retraction + high-epoch orphan filter bypass is a novel compound failure path. The reconciliation procedure (delete Neo4j nodes with `last_access_epoch > new_current_epoch` after retraction) and the upper-bound Cypher predicate are direct consequences of the failure analysis.
- **Evidence**: 0.78 — The compound failure is constructed from three independently verified facts: (1) Q074 established that PG restore + epoch retraction leaves Redis/Neo4j out of sync with PG; (2) Q078 established that Neo4j Cypher traversal has no epoch filter (gap confirmed in Q057 Cypher template); (3) PostgreSQL `ANY()` join semantics definitively return no error for missing rows — this is documented SQL behavior. The upper-bound filter logic (`last_access_epoch <= current_epoch`) follows from the epoch semantics of the retraction scenario. The gradual re-admission of orphaned nodes as current_epoch advances is a logical consequence of the epoch filter being a lower bound without an upper bound.
- **Feasibility**: 0.90 — The reconciliation Cypher (`DETACH DELETE WHERE last_access_epoch > e_new`) is a standard Neo4j operation. The batch LIMIT pattern is documented Neo4j practice for large deletes. The operational window (run immediately after Option A, before new sessions) is well-defined. The upper-bound Cypher predicate addition (`AND neighbor.last_access_epoch <= $current_epoch`) is a one-line change per traversal query. The runbook addition is a documentation change that references the existing Option A procedure and appends the Neo4j cleanup step.
- **Description**: Add a mandatory post-restore Neo4j orphan reconciliation step to the Q077 operator runbook, triggered whenever Option A epoch retraction is applied. The step identifies all Neo4j Memory nodes with `last_access_epoch > new_current_epoch` (the epoch value after retraction) and batch-deletes them with `DETACH DELETE` (removing both nodes and their incident CO_RETRIEVED edges). Additionally, add an upper-bound epoch predicate to all Cypher traversal queries (`AND neighbor.last_access_epoch <= $current_epoch`) to prevent orphaned nodes from being re-admitted as the epoch counter advances past their stored values. This step is absent from the current Q077 runbook and its omission produces silent PG join misses that are undetectable by any existing health signal.

### Idea: cross-store-consistency-check-endpoint
- **Novelty**: 0.75 — The `/epoch-status` endpoint proposed in Q074 monitors the Redis vs PG epoch gap but does not monitor Neo4j vs PG consistency. A complementary cross-store consistency check that compares the set of Neo4j node UUIDs against the set of PG memory UUIDs — reporting the count of orphaned Neo4j nodes (present in Neo4j, absent from PG) and the count of PG-only memories (present in PG, absent from Neo4j) — would make the compound failure immediately visible. This extends the Q074 epoch-gap health signal idea to cover the multi-store consistency dimension that epoch monitoring alone cannot detect.
- **Evidence**: 0.70 — The orphan count is computable as: `|{n.id for n in neo4j.run("MATCH (n:Memory) RETURN n.id")} - {row.id for row in pg.fetch("SELECT id FROM memories")}|`. At Tim's scale (~5,400 Neo4j nodes at Month 12, ~5,400 PG rows), this set difference computation is feasible as a periodic health check (not per-request). Running it every 6 hours via a FastAPI background task or cron would detect orphaned nodes within hours of a PG restore event. The false-positive rate during normal operation is near zero: the only legitimate source of orphaned Neo4j nodes (PG rows hard-deleted after tombstone TTL expiry) should be accompanied by a corresponding Neo4j node deletion in the tombstone cleanup code — a gap that itself warrants investigation.
- **Feasibility**: 0.82 — Implementation: a FastAPI background task that runs every 6 hours, computes the UUID set difference between Neo4j and PG, and stores the orphan count in the health state alongside the existing epoch gap. The `/health/detail` endpoint would expose `neo4j_orphan_count: 0` as a new field under the epoch-related signals. Alert threshold: `neo4j_orphan_count > 10` triggers a warning (small orphan counts from tombstone lag are acceptable). This adds one new background task and one new health field — low implementation burden relative to the visibility it provides.
- **Description**: Add a periodic cross-store consistency check (every 6 hours) that computes the difference between Neo4j Memory node UUIDs and PostgreSQL memory row UUIDs. Store the result as `neo4j_orphan_count` (in Neo4j but not PG) and `pg_only_count` (in PG but not Neo4j) in the health state. Expose these via `/health/detail` alongside the existing S4 epoch integrity signal. A `neo4j_orphan_count > 10` warning would have immediately surfaced the Q081 compound failure after a PG restore, enabling detection without requiring Tim to know the specific Cypher diagnostic query.

## Key Finding

The compound failure created by simultaneous Q074 (PG restore) and Q078 (missing epoch filter on Neo4j traversal) is more severe than either failure in isolation. After applying the standard Option A recovery (Redis epoch retracted to 101), the Q078 epoch filter does not protect against orphaned Neo4j nodes — it fails in the opposite direction: nodes with `last_access_epoch` in [101, 500] are from the pre-retraction epoch space and pass the lower-bound filter (`>= 81`) while having no corresponding row in the restored PostgreSQL. The LLPC join against PostgreSQL for these UUIDs produces silent row omissions with no exception, no error log, and no health signal alert. The system returns corrupted retrieval results — fewer candidates than the graph topology implies, with the surviving candidates drawn from a causally inconsistent graph — while reporting healthy status across all four Q071 health signals.

The Q077 runbook is missing a mandatory Neo4j orphan reconciliation step for the PG restore recovery path. The current runbook covers: (1) detecting the epoch gap, (2) applying Option A Redis retraction, (3) verifying S4 returns healthy. Step (4) — deleting all Neo4j nodes with `last_access_epoch > new_current_epoch` and their incident edges — does not appear anywhere in the documented recovery procedure. Without this step, a complete and correct execution of the Q077 runbook leaves the system in a silently corrupted state that degrades causal graph retrieval indefinitely.

An additional structural defect: the Q078 epoch filter is a lower bound only. It was designed for the normal-operation failure mode (epoch reset, stale memories surface). After epoch retraction, the orphaned nodes are not stale — they have higher-than-current epoch values — and they pass the lower bound. Adding an upper bound (`AND neighbor.last_access_epoch <= $current_epoch`) to all Cypher traversal queries would block these nodes during the post-retraction window, but provides only temporary protection: as the new epoch counter advances (102, 103, ...) it will eventually catch up to the stored epoch values of the orphaned nodes, re-admitting them one cohort at a time until Neo4j cleanup is performed. The upper-bound predicate is a mitigation; the reconciliation deletion is the fix.
