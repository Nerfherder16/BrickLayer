## S5 Definition: Which Measurement Option Is Correct

Three candidate measurement approaches were proposed in the question framing:

- **Option A**: Sample N random seed nodes, run BFS with epoch predicate, count how many returned neighbors produce PG join misses (UUID in Neo4j, UUID not in PG). Compliance ratio = (returned neighbors - miss count) / returned neighbors.
- **Option B**: Sample N random Memory nodes in Neo4j, check if their UUIDs exist in PG; report orphan ratio.
- **Option C**: Run the full cross-store UUID set-difference from Q081 and use orphan count as S5 value.

**S5 must be Option A.**

The rationale is operational specificity. The signal name is "graph traversal epoch compliance," not "graph orphan count." S5's purpose, as established in Q084, is to detect when the LLPC retrieval path — specifically the BFS-based Level 1.5 routing path through Neo4j — is returning candidates that do not exist in PostgreSQL. This is the path that produces silent context-window corruption (Q081). Option A is the only measurement that exercises this exact path: it seeds from real graph nodes, runs the BFS with the epoch predicate as Q078 Option A specifies, and then performs the same PG join that LLPC performs. The fraction of BFS results that fail the PG join is the compliance ratio. This directly measures retrieval-path integrity.

Option B is simpler but wrong for S5. Sampling random Memory nodes and checking their PG existence measures orphan prevalence across the entire node population, not across the traversal path. Orphaned nodes that are never returned by any realistic BFS (because they have no active edges to seeded nodes) would inflate the violation count without affecting any real retrieval. Option B is better suited to the Q081 background consistency check (`neo4j_orphan_count`) — a different signal.

Option C is the full 6-hour cross-store consistency check proposed in Q081. It is correct for detecting the *existence* of orphans but is too expensive (full UUID set enumeration across both stores) for a per-request or per-session health probe. It belongs as a scheduled background task populating `neo4j_orphan_count`, not as S5.

**The distinction between S5 and `neo4j_orphan_count` is therefore:**

| Signal | What it measures | Scope | Frequency | Mechanism |
|--------|-----------------|-------|-----------|-----------|
| `s5_graph_compliance` | Fraction of BFS-returned candidates that exist in PG — the actual traversal path quality | Traversal path only | Per health probe | Neo4j BFS sample + PG batch ANY() |
| `neo4j_orphan_count` | Total orphaned Neo4j nodes (in Neo4j but absent from PG) | Entire node population | Every 6 hours | Full UUID set-difference (Option C) |

These are complementary. `neo4j_orphan_count` tells you how many orphans exist. S5 tells you how many are entering the retrieval path right now.

## Cypher Probe Query Design

The probe must run in <100ms. The design parameters are:

- **K_seeds = 5** randomly sampled Memory nodes as BFS seeds
- **K_neighbors = 10** neighbors returned per seed (matching the Q078 recommendation of `LIMIT 20` for 2 seeds; at 5 seeds with 10 each the total candidate set is 50, which is representative)
- The epoch predicate must be included (`WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)`) to measure compliance with the working-set boundary, not just orphan existence
- The probe must also include the Q081 upper-bound predicate (`AND neighbor.last_access_epoch <= $current_epoch`) so that the probe accurately reflects what the BFS would return after the full Q078+Q081 fix is applied

### Phase 1: Sample seeds (Cypher)

```cypher
MATCH (n:Memory)
WHERE n.last_access_epoch IS NOT NULL
WITH n ORDER BY rand() LIMIT 5
RETURN n.id AS seed_id
```

Estimated latency: 3–8ms (random sampling with index on `last_access_epoch`; Neo4j cannot use a property index for ORDER BY rand() but the scan is over the ~5,400 node set at Month 12, which is fast).

### Phase 2: BFS with epoch predicate per seed (Cypher, run once for all seeds)

```cypher
MATCH (seed:Memory)-[r:CO_RETRIEVED]->(neighbor:Memory)
WHERE seed.id IN $seed_ids
  AND neighbor.last_access_epoch >= ($current_epoch - $tau)
  AND neighbor.last_access_epoch <= $current_epoch
WITH neighbor, r.co_occurrence_count AS weight
ORDER BY weight DESC
WITH seed_id, collect(neighbor.id)[0..10] AS top_neighbors
UNWIND top_neighbors AS neighbor_id
RETURN neighbor_id
```

In practice this is cleaner as a single-pass query that collects results across all seeds without a per-seed loop:

```cypher
MATCH (seed:Memory)-[r:CO_RETRIEVED]->(neighbor:Memory)
WHERE seed.id IN $seed_ids
  AND neighbor.last_access_epoch >= ($current_epoch - $tau)
  AND neighbor.last_access_epoch <= $current_epoch
WITH seed, neighbor, r.co_occurrence_count AS weight
ORDER BY seed.id, weight DESC
WITH seed, collect(neighbor.id)[0..10] AS top_neighbors
UNWIND top_neighbors AS nid
RETURN DISTINCT nid AS neighbor_id
```

Estimated latency at Month 12 (9,600 edges, 5,400 nodes, 5 seeds, average fanout ~3.6): accessing all edges for 5 seed nodes = 5 × 3.6 ≈ 18 relationship reads, then epoch filtering. Cache-warm: ~2ms. Cold cache worst case: ~15ms. With the `LIMIT 10` per seed implicit in the collect slice, total returned candidate count: 0–50 UUIDs.

**Edge case**: If the randomly sampled seeds have no edges (early deployment, Month 1), the BFS returns 0 neighbors. Handle with a minimum-sample guard (described in the Python implementation below).

### Phase 3: PG batch join (Python + asyncpg)

```python
rows = await pg_conn.fetch(
    "SELECT id FROM memories WHERE id = ANY($1::uuid[])",
    neighbor_ids
)
found_ids = {row['id'] for row in rows}
miss_count = len(neighbor_ids) - len(found_ids)
compliance_ratio = (len(neighbor_ids) - miss_count) / len(neighbor_ids)
```

Estimated latency: 1–3ms for a batch of 0–50 UUIDs using `ANY($1::uuid[])`. Total probe latency: 5–25ms, well within 100ms.

## Threshold Calibration

### Steady-state analysis (no failures)

At steady state with no orphaned nodes, compliance_ratio = 1.0. The BFS returns only nodes whose UUIDs exist in PG because all Memory nodes in Neo4j are written alongside their PG counterparts (dual-write pattern from Q078 Option A).

The only legitimate source of sub-1.0 compliance in steady state is tombstoned memories: PG rows may be hard-deleted after tombstone TTL expiry while the Neo4j node persists (if the tombstone cleanup job does not also clean Neo4j). Q089 was tasked with quantifying this rate. In the absence of Q089 data, a conservative estimate: if tombstone hard-deletes run monthly and Tim accumulates ~50 memories/day, roughly 1,500 memories/month are created and a small fraction (5–10%, per typical memory system eviction rates) are tombstoned, yielding ~75–150 hard deletes/month. At 5,400 active nodes, this is a 1.4–2.8% orphan rate from tombstone lag alone. However, tombstone-orphaned Neo4j nodes accumulate a low `last_access_epoch` and will be filtered out by the epoch predicate (`>= current_epoch - tau`) in the BFS. Tombstone orphans are effectively invisible to S5 because they are stale. **Steady-state compliance from tombstone orphans: ~1.0 (tombstone orphans filtered by epoch predicate before PG join).**

### PG restore scenario (Q081 failure mode)

After PG restore from backup at epoch 100 with Redis continuing at epoch 500:
- Current epoch: 500 (Redis)
- Epoch predicate: `last_access_epoch >= 480` (tau=20)
- Orphaned nodes: those with `last_access_epoch` in [101, 500] — approximately 400 sessions worth of nodes
- BFS seeds are randomly sampled; at Month 12 with 5,400 nodes, ~7.4% are from the backup era (epoch ≤ 100) and ~92.6% are orphans (epoch 101–500)
- Probability that a randomly sampled seed has edges to orphaned neighbors: high (the most recent 400 sessions dominate the edge density, as edges are reinforced continuously)
- Expected fraction of BFS neighbors that are orphans: proportional to the fraction of recent nodes that are orphaned, which is approximately (epoch_gap / tau_range) within the working-set window

More precisely: the epoch predicate filters for neighbors with `last_access_epoch >= 480`. All nodes in [481, 500] pass the filter. Of these, those with `last_access_epoch` in [481, 500] that were created between the backup and the present are orphans — i.e., all of them, since the backup was at epoch 100. Every node in the working-set window [480, 500] is an orphan in the Q081 scenario. The compliance_ratio collapses toward 0.0 for seeds that have any edges within the working-set.

**In practice at Q081 scale (400-epoch gap)**: Within the tau=20 working-set, 20 epochs of orphaned nodes exist. Seeds with connections to any of these yield 100% miss rate from the PG join. Seeds with no connections in [480, 500] (old seeds from the backup era) yield 0% miss rate. The weighted average compliance_ratio across 5 random seeds depends on the ratio of old-to-new edges.

At Month 12 with 9,600 edges spread over 500 epochs (approximately 19.2 edges/epoch), the working-set window [480–500] contains ~19.2 × 20 = 384 edges. These all point to orphaned nodes. Seeds from the backup era (epoch ≤ 100) are in the minority (~100/500 = 20% of nodes). A random seed draw of 5 nodes will include on average 1 old seed and 4 orphaned seeds. The 4 orphaned seeds have no edges after epoch filtering (they themselves pass as seeds, but their neighbors in the working-set window are all orphans — PG join misses). **Expected compliance_ratio in Q081 scenario: approximately 0.20–0.40**, depending on exact graph structure.

However, the earlier framing in the question posited a compliance_ratio drop to 0.70–0.85 after a 400-epoch PG restore. This was assuming a smaller orphan fraction (perhaps an earlier deployment point with fewer orphaned working-set nodes). The actual range will vary. The thresholds below are calibrated to detect both moderate and severe orphan conditions.

### Threshold values

| Condition | compliance_ratio | Status |
|-----------|-----------------|--------|
| Ideal (no orphans, no failures) | 1.0 | HEALTHY |
| Tombstone lag only (orphans epoch-filtered out of BFS) | ~1.0 | HEALTHY |
| Minor orphan contamination (< 5% of BFS results are misses) | ≥ 0.95 | HEALTHY |
| Moderate orphan contamination (5–15% miss rate) — early PG restore detection or partial graph corruption | [0.85, 0.95) | WARNING |
| Severe orphan contamination (> 15% miss rate) — confirmed PG restore or large-scale Neo4j inconsistency | < 0.85 | CRITICAL |

**Rationale for 0.85 CRITICAL threshold**: In steady-state operation, the BFS epoch predicate filters tombstone orphans, keeping compliance near 1.0. A compliance_ratio below 0.85 requires a structural mismatch between Neo4j and PG that cannot be explained by routine tombstone lag — specifically, it requires recently-accessed nodes (within the tau=20 window) whose PG rows are absent. This is the Q081 fingerprint: nodes that pass the epoch predicate (recent activity) but fail the PG join (PG restore deleted them). The 0.85 threshold tolerates up to 15 misses per 100 BFS results before alarm, providing headroom for minor inconsistencies from timing windows (e.g., a Neo4j node written 2ms before its PG row during a dual-write).

**Rationale for 0.95 WARNING threshold**: A compliance_ratio between 0.85 and 0.95 suggests mild orphan contamination: either a small PG restore event (< 50 epochs of data loss) or an early-stage accumulation from tombstone cleanup gaps. It warrants investigation but does not indicate an active retrieval path failure.

**Insufficient sample guard**: If the BFS returns fewer than 5 total neighbors across all seeds (sparse graph, early deployment, or all seeds are isolated nodes), S5 should return `status: "insufficient_data"` rather than computing a potentially misleading ratio from 2–3 samples.

## FastAPI Implementation

### S5 probe function

```python
# health_signals.py — S5
# Measures graph traversal epoch compliance: what fraction of BFS-returned neighbors
# exist in PostgreSQL (i.e., are not orphaned by a PG restore event).
#
# Distinct from neo4j_orphan_count (Q081 Option C):
#   - neo4j_orphan_count: full UUID set-difference, runs every 6h, measures ALL orphans
#   - s5_graph_compliance: BFS sample probe, runs on health check, measures TRAVERSAL-PATH orphans

import asyncpg
from neo4j import AsyncSession as Neo4jSession

MINIMUM_NEIGHBOR_SAMPLE = 5  # require at least 5 BFS results to compute a meaningful ratio
S5_SEED_COUNT = 5             # random seed nodes for BFS
S5_NEIGHBORS_PER_SEED = 10    # max neighbors collected per seed (LIMIT in Cypher slice)


async def compute_graph_traversal_compliance(
    neo4j_session: Neo4jSession,
    pg_conn: asyncpg.Connection,
    current_epoch: int,
    tau: int = 20,
) -> dict:
    """
    S5: Graph Traversal Epoch Compliance

    Samples K random seed nodes, runs BFS with epoch predicate, and checks what
    fraction of returned neighbors exist in PostgreSQL. A compliance_ratio below
    1.0 indicates orphaned Neo4j nodes are entering the retrieval path — the
    fingerprint of the Q081 compound failure (PG restore without Neo4j cleanup).

    Compliance ratio:
        1.0 = all BFS-returned neighbors found in PG  (healthy)
        0.85–0.95 = 5–15% miss rate                  (warning)
        < 0.85 = > 15% miss rate                     (critical)
    """

    # Phase 1: Sample K random seed nodes with at least one epoch property set
    seed_result = await neo4j_session.run(
        """
        MATCH (n:Memory)
        WHERE n.last_access_epoch IS NOT NULL
        WITH n ORDER BY rand() LIMIT $k
        RETURN n.id AS seed_id
        """,
        k=S5_SEED_COUNT,
    )
    seed_records = await seed_result.data()
    seed_ids = [r["seed_id"] for r in seed_records]

    if not seed_ids:
        return {
            "signal": "graph_traversal_compliance",
            "status": "insufficient_data",
            "compliance_ratio": None,
            "neighbor_count": 0,
            "miss_count": None,
            "note": "No Memory nodes with last_access_epoch found in Neo4j",
        }

    # Phase 2: BFS with compound epoch predicate (Q078 lower bound + Q081 upper bound)
    # ORDER BY co_occurrence_count DESC ensures the most-reinforced edges are sampled
    # collect()[0..10] applies the per-seed neighbor cap
    bfs_result = await neo4j_session.run(
        """
        MATCH (seed:Memory)-[r:CO_RETRIEVED]->(neighbor:Memory)
        WHERE seed.id IN $seed_ids
          AND neighbor.last_access_epoch >= ($current_epoch - $tau)
          AND neighbor.last_access_epoch <= $current_epoch
        WITH seed, neighbor, r.co_occurrence_count AS weight
        ORDER BY seed.id, weight DESC
        WITH seed, collect(neighbor.id)[0..$max_per_seed] AS top_neighbors
        UNWIND top_neighbors AS nid
        RETURN DISTINCT nid AS neighbor_id
        """,
        seed_ids=seed_ids,
        current_epoch=current_epoch,
        tau=tau,
        max_per_seed=S5_NEIGHBORS_PER_SEED,
    )
    bfs_records = await bfs_result.data()
    neighbor_ids = [r["neighbor_id"] for r in bfs_records]

    if len(neighbor_ids) < MINIMUM_NEIGHBOR_SAMPLE:
        return {
            "signal": "graph_traversal_compliance",
            "status": "insufficient_data",
            "compliance_ratio": None,
            "neighbor_count": len(neighbor_ids),
            "miss_count": None,
            "seed_count": len(seed_ids),
            "note": (
                f"BFS returned only {len(neighbor_ids)} neighbors "
                f"(minimum {MINIMUM_NEIGHBOR_SAMPLE} required). "
                "Graph may be sparse — normal at early deployment stages."
            ),
        }

    # Phase 3: PG batch join — check which neighbor UUIDs exist in PostgreSQL
    pg_rows = await pg_conn.fetch(
        "SELECT id FROM memories WHERE id = ANY($1::uuid[])",
        neighbor_ids,
    )
    found_ids = {str(row["id"]) for row in pg_rows}
    missing_ids = [nid for nid in neighbor_ids if nid not in found_ids]
    miss_count = len(missing_ids)
    total = len(neighbor_ids)
    compliance_ratio = (total - miss_count) / total

    # Threshold evaluation
    if compliance_ratio < 0.85:
        status = "critical"
        interpretation = (
            f"CRITICAL: {miss_count}/{total} BFS-returned neighbors "
            f"({100 * miss_count / total:.1f}%) are absent from PostgreSQL. "
            f"This indicates orphaned Neo4j nodes in the active working-set window "
            f"(last_access_epoch in [{current_epoch - tau}, {current_epoch}]). "
            f"Most likely cause: PostgreSQL was restored from backup without "
            f"corresponding Neo4j orphan cleanup (Q081 compound failure). "
            f"Required action: run Neo4j orphan reconciliation "
            f"(MATCH (n:Memory) WHERE n.last_access_epoch > <pg_max> "
            f"WITH n LIMIT 500 DETACH DELETE n) before retrieval quality is further corrupted."
        )
    elif compliance_ratio < 0.95:
        status = "warning"
        interpretation = (
            f"WARNING: {miss_count}/{total} BFS-returned neighbors "
            f"({100 * miss_count / total:.1f}%) are absent from PostgreSQL. "
            f"Mild orphan contamination detected. Possible causes: "
            f"(a) small PG restore event with partial epoch gap; "
            f"(b) tombstone cleanup gap where PG rows were hard-deleted "
            f"but Neo4j nodes were not pruned. "
            f"Monitor over next 3 sessions; escalate to CRITICAL if ratio drops below 0.85."
        )
    else:
        status = "healthy"
        interpretation = (
            f"Graph traversal epoch compliance nominal: "
            f"{total - miss_count}/{total} BFS neighbors confirmed in PostgreSQL "
            f"(compliance_ratio={compliance_ratio:.3f}). "
            f"No orphaned nodes detected in active working-set traversal path."
        )

    return {
        "signal": "graph_traversal_compliance",
        "status": status,
        "compliance_ratio": round(compliance_ratio, 4),
        "neighbor_count": total,
        "miss_count": miss_count,
        "seed_count": len(seed_ids),
        "current_epoch": current_epoch,
        "epoch_window": f"[{current_epoch - tau}, {current_epoch}]",
        "threshold_warning": 0.95,
        "threshold_critical": 0.85,
        "interpretation": interpretation,
        "missing_sample": missing_ids[:5] if missing_ids else [],  # first 5 for debug
    }
```

### Response schema extension

The `/health/detail` response gains a fifth signal key under `signals`:

```python
# In recall_api/routers/health.py — updated /health/detail endpoint

@router.get("/detail")
async def health_detail(
    pg=Depends(get_pg_pool),
    redis=Depends(get_redis_client),
    neo4j=Depends(get_neo4j_session),   # new dependency
):
    async with pg.acquire() as conn:
        # Fetch current epoch for S4 and S5
        global_epoch_raw = await redis.get("recall:global_epoch")
        current_epoch = int(global_epoch_raw) if global_epoch_raw else 0

        signals = {
            "s1_ips_p3_delta":        await compute_ips_p3_delta(conn),
            "s2_mbc_level_dist":      await compute_mbc_level_distribution(conn),
            "s3_pending_queue":       compute_pending_queue_depth(),
            "s4_epoch_integrity":     await compute_epoch_integrity(redis, conn),
            "s5_graph_compliance":    await compute_graph_traversal_compliance(
                                          neo4j, conn, current_epoch, tau=20
                                      ),
        }

    # Compute individual signal statuses
    s4 = signals["s4_epoch_integrity"]
    s5 = signals["s5_graph_compliance"]

    overall = (
        "critical" if any(
            s.get("status") == "critical" for s in signals.values()
            if s.get("status") not in ("insufficient_data",)
        ) else
        "warning" if any(
            s.get("status") == "warning" for s in signals.values()
            if s.get("status") not in ("insufficient_data",)
        ) else
        "healthy"
    )

    # Correlation layer — apply Q084 co-alert rules
    correlations = compute_correlations(signals)

    return {
        "overall": overall,
        "signals": signals,
        "correlations": correlations,
    }
```

### S4+S5 co-alert rule in the correlation layer

The correlation layer evaluates the 10 pairwise co-alert rules from Q084. The S4+S5 rule is the most diagnostically complex because the correct response depends on the S4 alarm direction (Redis-reset vs PG-restore). The implementation below integrates with the S4 signal structure established in Q071, which exposes `epoch_gap` and an implied alarm direction:

```python
# health_signals.py — correlation layer

EPOCH_GAP_ALARM_THRESHOLD = 50  # from Q074

def compute_correlations(signals: dict) -> list[dict]:
    """
    Apply Q084 pairwise co-alert rules. Returns a list of active correlation
    objects, each describing: which signals are co-alerting, the implied root
    cause, confidence level, recommended first action, and whether the action
    is automated or requires operator decision.
    """
    correlations = []

    s1 = signals.get("s1_ips_p3_delta", {})
    s2 = signals.get("s2_mbc_level_dist", {})
    s3 = signals.get("s3_pending_queue", {})
    s4 = signals.get("s4_epoch_integrity", {})
    s5 = signals.get("s5_graph_compliance", {})

    def alerting(sig):
        return sig.get("status") in ("warning", "critical")

    # --- S2 + S3: CasaOS unavailability (most unambiguous pattern, Q077) ---
    if alerting(s2) and alerting(s3):
        correlations.append({
            "signals": ["s2_mbc_level_dist", "s3_pending_queue"],
            "implied_root_cause": "CASAOS_UNAVAILABILITY",
            "confidence": "unambiguous",
            "recommended_action": (
                "Check CasaOS health: curl http://192.168.50.19:8200/health — "
                "if reachable, force-replay: POST /recall/replay-pending — "
                "if unreachable, restart Recall Docker container."
            ),
            "automated": True,
        })

    # --- S3 + S4: CasaOS restart with Redis data loss ---
    if alerting(s3) and s4.get("status") == "critical":
        epoch_gap = s4.get("epoch_gap", 0)
        if epoch_gap < 0:  # Direction 1: redis_epoch < pg_max (Redis reset)
            correlations.append({
                "signals": ["s3_pending_queue", "s4_epoch_integrity"],
                "implied_root_cause": "CASAOS_RESTART_WITH_REDIS_DATA_LOSS",
                "confidence": "likely",
                "recommended_action": (
                    f"Step 1 (epoch fix): {s4.get('recovery_action', 'redis-cli SET recall:global_epoch <pg_max+1>')} — "
                    "Step 2 (after epoch restored): POST /recall/replay-pending"
                ),
                "automated": True,
                "note": "Apply epoch fix BEFORE replay to ensure replayed writes use correct epoch.",
            })

    # --- S1 + S4: Epoch reset cascading to quality degradation ---
    if alerting(s1) and alerting(s4):
        correlations.append({
            "signals": ["s1_ips_p3_delta", "s4_epoch_integrity"],
            "implied_root_cause": "EPOCH_RESET_WITH_QUALITY_DEGRADATION",
            "confidence": "likely",
            "recommended_action": (
                f"Apply S4 recovery first: {s4.get('recovery_action', 'see S4 signal details')} — "
                "then monitor S1 for natural recovery over next 7 days. "
                "If S1 does not recover after S4 is resolved, refit propensity model."
            ),
            "automated": "s4_only",
        })

    # --- S4 + S5: The Q084 diagnostic co-alert (most complex rule) ---
    if alerting(s4) and alerting(s5) and s5.get("status") != "insufficient_data":
        epoch_gap = s4.get("epoch_gap", 0)  # positive = redis ahead of pg (PG restore direction)
        # Determine S4 direction from the gap sign and magnitude
        # epoch_gap = global_epoch - max_db_epoch
        # Negative gap = Direction 1 (Redis reset: redis < pg_max)
        # Large positive gap > EPOCH_GAP_ALARM_THRESHOLD = Direction 2 (PG restore: redis >> pg_max)

        if epoch_gap is not None and epoch_gap > EPOCH_GAP_ALARM_THRESHOLD:
            # Direction 2: PG restore suspected — S5 provides confirmatory evidence
            # This is the "PG_RESTORE_CONFIRMED_BY_GRAPH_STATE" escalation from Q084
            implied_sessions_lost = epoch_gap
            correlations.append({
                "signals": ["s4_epoch_integrity", "s5_graph_compliance"],
                "implied_root_cause": "PG_RESTORE_CONFIRMED_BY_GRAPH_STATE",
                "confidence": "high",
                "alarm_code": "PG_RESTORE_CONFIRMED_BY_GRAPH_STATE",
                "recommended_action": (
                    f"CONVERGENT EVIDENCE: S4 detects epoch gap={epoch_gap} "
                    f"(~{implied_sessions_lost} sessions of data loss if Option A is applied). "
                    f"S5 independently confirms: {s5.get('miss_count', '?')} of "
                    f"{s5.get('neighbor_count', '?')} BFS-returned neighbors are absent from PG — "
                    f"Neo4j graph has edges pointing to post-backup memories that PG no longer contains. "
                    f"This combination rules out Redis-reset direction and confirms PG restore. "
                    f"Recovery options (operator decision required — Option A is irreversible): "
                    f"(A) Retract Redis epoch to pg_max+1 (accept ~{implied_sessions_lost} session memory loss) "
                    f"    then run Neo4j orphan cleanup: "
                    f"    MATCH (n:Memory) WHERE n.last_access_epoch > <pg_max> "
                    f"    WITH n LIMIT 500 DETACH DELETE n [repeat until 0 deleted]. "
                    f"(B) Restore PG from a more recent backup if available. "
                    f"(C) Temporarily set tau to {epoch_gap + 20} to re-include backup-era memories "
                    f"    while root cause is investigated. "
                    f"Do NOT auto-execute Option A — false positive rate is 15-30% for automated trigger."
                ),
                "automated": False,
                "false_positive_rate": "15-30%",
                "note": (
                    "S5 co-alert disambiguates S4 direction: in the Redis-reset scenario, "
                    "the Cypher epoch predicate collapses to near-zero threshold admitting all nodes, "
                    "so S5 does NOT fire. S5 firing alongside S4 with large positive gap "
                    "is specific to the PG-restore scenario (Q084 direction discriminant)."
                ),
            })
        else:
            # Direction 1 (Redis reset, epoch_gap < 0) OR small positive gap:
            # S5 is in a chronic background state (graph maturity, Q078 Month 12+ scenario).
            # Apply S4 fix (safe); log S5 as queued for Q078 Cypher predicate implementation.
            correlations.append({
                "signals": ["s4_epoch_integrity", "s5_graph_compliance"],
                "implied_root_cause": "REDIS_RESET_WITH_CHRONIC_GRAPH_STALENESS",
                "confidence": "likely",
                "recommended_action": (
                    f"S4 automated recovery: {s4.get('recovery_action', 'see S4 signal details')} — "
                    "S5 is in a chronic background condition (graph maturity — stale hub-node edges). "
                    "S5 does not require immediate recovery but the Q078 Cypher epoch predicate "
                    "should be implemented to eliminate the staleness injection path."
                ),
                "automated": "s4_only",
            })

    # --- S5 alone: chronic graph maturity (no S4 co-alert) ---
    if alerting(s5) and not alerting(s4) and s5.get("status") != "insufficient_data":
        correlations.append({
            "signals": ["s5_graph_compliance"],
            "implied_root_cause": "GRAPH_EPOCH_FILTER_GAP",
            "confidence": "likely",
            "recommended_action": (
                "Graph traversal is returning neighbors that fail PG join without any epoch disruption. "
                "Most likely cause: graph has matured (Month 6–12+) and hub nodes have accumulated "
                "edges to memories outside the working-set window, but the Q078 epoch predicate "
                "is not yet applied to Cypher queries (OR tombstone cleanup is not removing Neo4j nodes). "
                "Required code change: add "
                "WHERE neighbor.last_access_epoch >= ($current_epoch - $tau) "
                "AND neighbor.last_access_epoch <= $current_epoch "
                "to all CO_RETRIEVED traversal Cypher queries."
            ),
            "automated": False,
        })

    return correlations
```

## Integration with Q084 S4+S5 Co-Alert Rule

The critical insight from Q084 (reproduced here for completeness) is that the S4+S5 co-alert direction discriminant works because of an asymmetry in epoch predicate behavior:

**In the Redis-reset direction (S4 Direction 1, `redis_epoch < pg_max`)**: After Redis is reset to 0, the BFS epoch predicate evaluates as `neighbor.last_access_epoch >= (0 - 20) = -20`. This is universally true — all nodes pass. The BFS returns nodes at random, not biased toward orphans, and PG join succeeds for all of them (the memories still exist in PG; Redis reset does not delete PG rows). **S5 does not fire.** Therefore S4+S5 co-alert does NOT occur in the Redis-reset scenario.

**In the PG-restore direction (S4 Direction 2, `redis_epoch >> pg_max`, gap > EPOCH_GAP_ALARM_THRESHOLD)**: Redis continues at epoch 500; PG is restored to epoch 100. The BFS epoch predicate evaluates as `neighbor.last_access_epoch >= 480`. All nodes in the working-set window [480–500] pass. These nodes were created between the backup (epoch 100) and the present (epoch 500) — they are all post-backup nodes, and they all fail the PG join (their PG rows were deleted by the restore). **S5 fires.** The compliance_ratio collapses. S4+S5 co-alert occurs, and the S5 miss_count provides a direct measurement of the orphan density in the active traversal path.

This asymmetry makes the S4+S5 co-alert a *direction discriminant* for the S4 signal — something S4 alone cannot provide without inspecting the raw epoch values. The `alarm_code: "PG_RESTORE_CONFIRMED_BY_GRAPH_STATE"` in the correlation layer encodes this: two independent subsystems (Redis epoch clock and Neo4j graph topology) have converged on the same diagnosis, reducing the probability of a false alarm compared to either signal alone.

**False positive scenarios for S4+S5 co-alert in the PG-restore direction** (from Q084):
1. Legitimate large epoch gap (> 50) from dual-machine operation without PG restore, simultaneous with chronic S5 graph maturity: ~15–30% false positive rate — too high for automated Option A recovery.
2. S5 in chronic warning state (Month 12+) co-alerting with a transient S4 Direction 1 event: the code above handles this correctly by checking the gap sign/magnitude to discriminate directions.

The `automated: False` flag on the PG_RESTORE_CONFIRMED_BY_GRAPH_STATE correlation is therefore mandatory. The correlation provides diagnostic convergence — not an automated recovery trigger.

### Idea: s5-traversal-path-compliance-vs-population-orphan-count
- **Novelty**: 0.76 — The distinction between measuring orphan prevalence across the full node population (Option C, Q081's background check) versus measuring orphan prevalence specifically within the BFS traversal path (Option A, S5) is a novel refinement in the health monitoring design. Population-level orphan counts can be non-zero without affecting retrieval quality (orphans with low `last_access_epoch` are filtered out before PG join; they exist in Neo4j but never enter the retrieval path). S5's path-specific measurement answers the operationally relevant question: "are orphans currently corrupting retrieved context?" rather than "do orphans exist somewhere in the graph?" This framing — instrumenting the actual retrieval path rather than the storage state — is the correct observability principle for a retrieval system and is not present in reviewed AI memory system health monitoring designs.
- **Evidence**: 0.74 — The argument is grounded in the epoch predicate's filtering behavior: tombstone orphans have low `last_access_epoch` (filtered before PG join) while PG-restore orphans have high `last_access_epoch` (pass the filter and reach the PG join). This asymmetry means population-level orphan count is dominated by tombstone orphans (benign, invisible to retrieval) while path-level compliance ratio is sensitive specifically to PG-restore orphans (dangerous, enter retrieval path). The two measurements are complementary but S5 is the operationally critical one. The asymmetry follows directly from the epoch predicate lower-bound design established in Q078 and Q081.
- **Feasibility**: 0.88 — The BFS probe runs in 5–25ms. The PG batch join is a single `ANY($1::uuid[])` query. The entire S5 probe is approximately 80 lines of Python. The only new dependency is an async Neo4j session in the health endpoint — likely already available if Q078 is implemented.
- **Description**: S5 measures graph traversal epoch compliance by sampling the actual BFS path: seed K=5 random nodes, run BFS with the compound epoch predicate (lower + upper bound from Q081), collect up to 50 neighbor UUIDs, batch-join against PG, compute compliance_ratio = (found / total). This measures exactly what LLPC experiences at retrieval time — not what exists in the graph in abstract. Compliance_ratio below 0.95 triggers WARNING; below 0.85 triggers CRITICAL. The probe runs in <100ms and detects PG-restore orphan contamination within the first health check after the failure occurs.

## Key Finding

S5 should be implemented as Option A (BFS probe with PG join miss counting) rather than Option B (raw orphan sampling) or Option C (full set-difference). The distinction matters operationally: tombstone orphans have low epoch values and are filtered out of the BFS before the PG join, making them invisible to S5 and correctly leaving the compliance_ratio at ~1.0 in steady state. PG-restore orphans have high epoch values, pass the BFS epoch predicate, and fail the PG join — driving compliance_ratio below the warning threshold. S5 is therefore specifically sensitive to the Q081 compound failure while being insensitive to routine tombstone lag. This selectivity is the core design property that makes S5 useful in the correlation layer: an S5 alert means post-restore orphans are actively entering the retrieval path, not merely that some orphans exist somewhere in the graph.

The implementation is approximately 215 new lines of Python across the probe function and the correlation layer, with zero new infrastructure. The S4+S5 co-alert correlation rule emits `PG_RESTORE_CONFIRMED_BY_GRAPH_STATE` when both signals are alerting and the S4 epoch gap exceeds EPOCH_GAP_ALARM_THRESHOLD — providing convergent evidence from two independent subsystems (Redis epoch clock and Neo4j graph topology) that a PG restore occurred. The `automated: False` flag is mandatory: the false positive rate (15–30%) and the irreversibility of Option A recovery (permanent session memory loss proportional to the epoch gap) make automated execution unsafe. The correlation layer converts the health endpoint from a raw signal reporter into a diagnostic assistant that narrows operator decision time from open-ended investigation to option selection.
