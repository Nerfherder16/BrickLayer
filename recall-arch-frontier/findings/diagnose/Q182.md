## Evidence

### Feedback Summary Lifecycle Post-Wave 19 Deployment

Q168 specified an `audit_log` retention worker that runs daily at **6:15am**. This worker does two operations:

1. Pre-aggregates all feedback actions from `audit_log` into `feedback_summary` (memory_id PK, positive COUNT, negative COUNT, last_synced TIMESTAMP)
2. Deletes old audit_log rows (>30 days) via batched CTE loop

**Timeline after Wave 19 deployment (assume 12:00am UTC):**
- **T=0:00** (deployment): `feedback_summary` table exists but is **empty** (no aggregation has happened yet). This is correct.
- **T=0:00 to T=6:15am** (18-hour grace period): `feedback_summary` remains empty. Application behavior is correct (reranker doesn't use feedback signal until first sync). **No alert should fire.**
- **T=6:15am**: First ARQ worker run. `feedback_summary` is populated with aggregated counts from `audit_log`. `last_synced` is SET to now().
- **T=6:15am onward**: `feedback_summary` contains data. If `last_synced < now - 2 days`, the worker has not run and counts are stale. **Alert should fire.**

### The Problem with Naive Thresholds

**Approach 1: `last_synced < now - 2 days`**
- ✓ Detects actual staleness after 24-48 hours of missed runs
- ✗ **False positive**: Immediately after deployment (T=0:00), `last_synced` is NULL or very recent. On first check 30 minutes after deployment, the condition `NULL < now - 2 days` evaluates to NULL (unknown), which most SQL engines treat as FALSE. This is safe. **However**, if the first run is delayed (due to ARQ queue backlog, worker crash, etc.), `last_synced` could remain NULL until T=8am. Any check after T=8:02am will see `NULL < 8am - 2 days` and alert incorrectly.

**Approach 2: `feedback_summary IS EMPTY AND deployed_at > now - 24 hours`**
- ✓ Allows 24-hour grace period for first sync
- ✓ Correctly allows empty state immediately post-deployment
- ✗ **True negative on staleness**: If the worker runs successfully at T=6:15am, `feedback_summary` is NO LONGER empty. Even if the worker then crashes and doesn't run again, staleness is detected by `last_synced < now - 2 days`, not by the empty check.
- ✗ **After 24 hours, still doesn't fire**: If deployed_at is 12:00am and now is 1:00pm (13 hours), the condition `deployed_at > now - 24 hours` is still true (12:00am > 1:00pm - 24 hours = 1:00pm previous day). The `IS EMPTY` check would fire a false alert if there was a deployment failure and the worker never ran. This is correct for detection but only 13 hours in, not the intended 24-hour grace.

**Approach 3: Composite with NULL handling**
- ✓ Handles both empty+new and populated+stale states
- ✓ 24-hour grace period for first sync
- ✓ Detects both missing sync (NULL with age > 24h) and stale sync (last_synced < now - 2 days)

## S6 Definition (Recommended)

### Health Signal: `feedback_summary_sync_health`

**Alert Condition**:
```sql
-- Fire alert if ANY of:
-- 1. feedback_summary has data but staleness exceeds 2 days
-- 2. feedback_summary is empty AND deployment is >24 hours old (worker never ran)

WITH deployment_age AS (
  SELECT EXTRACT(EPOCH FROM (NOW() - deployment_time))/3600 as hours_since_deploy
  FROM deployment_metadata
  ORDER BY deployment_time DESC
  LIMIT 1
),
summary_status AS (
  SELECT
    COUNT(*) as row_count,
    MAX(last_synced) as last_sync_time,
    EXTRACT(EPOCH FROM (NOW() - MAX(last_synced)))/3600 as hours_since_sync
  FROM feedback_summary
)
SELECT
  CASE
    -- Grace period: empty and <24 hours old
    WHEN summary_status.row_count = 0 AND deployment_age.hours_since_deploy < 24
      THEN 'OK' -- Expected empty state, first sync not yet run

    -- Stale data: has rows but last sync >48 hours ago
    WHEN summary_status.row_count > 0 AND summary_status.hours_since_sync > 48
      THEN 'STALE' -- Worker missed runs, counts are stale

    -- Never synced: empty and >24 hours post-deployment
    WHEN summary_status.row_count = 0 AND deployment_age.hours_since_deploy >= 24
      THEN 'NEVER_SYNCED' -- Worker should have run by now, did not

    -- Normal: has recent data
    ELSE 'HEALTHY'
  END as s6_status;
```

**Alert Thresholds**:
| Status | Alert? | Severity |
|--------|--------|----------|
| OK | No | — |
| HEALTHY | No | — |
| STALE | Yes | **High** — reranker is using outdated feedback counts, may rank incorrectly |
| NEVER_SYNCED | Yes | **Critical** — worker has never run post-deployment, indicates ARQ failure or cron misconfiguration |

### Implementation in health.py

Add to health.py's `_compute_graph_health()` or in a new function `_check_feedback_summary_health()`:

```python
async def _check_feedback_summary_health(pg_store) -> HealthSignal:
    """
    Check feedback_summary staleness.

    Returns HealthSignal with:
      status: "ok" | "degraded" | "unhealthy"
      message: human-readable explanation
      last_checked: ISO-8601 timestamp
    """
    try:
        # Get deployment age (assume from PostgreSQL or config)
        deployment_age_hours = await pg_store.get_deployment_age_hours()

        # Check feedback_summary
        summary_check = await pg_store.execute("""
            SELECT
              COUNT(*) as row_count,
              EXTRACT(EPOCH FROM (NOW() - MAX(last_synced)))/3600 as hours_since_sync
            FROM feedback_summary
        """)

        row_count = summary_check[0]["row_count"] if summary_check else 0
        hours_since_sync = summary_check[0]["hours_since_sync"] if summary_check and summary_check[0]["hours_since_sync"] else None

        # Apply logic
        if row_count == 0 and deployment_age_hours < 24:
            # Grace period: empty is expected
            return HealthSignal(
                status="ok",
                message="feedback_summary empty (expected within 24h post-deployment)",
                last_checked=datetime.utcnow(),
                component="s6_feedback_sync"
            )

        elif row_count > 0 and hours_since_sync is not None and hours_since_sync > 48:
            # Stale
            return HealthSignal(
                status="degraded",
                message=f"feedback_summary stale: last sync {hours_since_sync:.1f}h ago (>48h threshold)",
                last_checked=datetime.utcnow(),
                component="s6_feedback_sync"
            )

        elif row_count == 0 and deployment_age_hours >= 24:
            # Never synced post-deployment
            return HealthSignal(
                status="unhealthy",
                message=f"feedback_summary never synced: empty {deployment_age_hours:.1f}h post-deployment (worker may have crashed)",
                last_checked=datetime.utcnow(),
                component="s6_feedback_sync"
            )

        else:
            # Normal
            return HealthSignal(
                status="ok",
                message=f"feedback_summary healthy: synced {hours_since_sync:.1f}h ago",
                last_checked=datetime.utcnow(),
                component="s6_feedback_sync"
            )

    except Exception as e:
        # Database error = degraded (cannot verify health)
        return HealthSignal(
            status="degraded",
            message=f"Cannot check feedback_summary health: {str(e)}",
            last_checked=datetime.utcnow(),
            component="s6_feedback_sync"
        )
```

## Ideas Extracted

### Idea 1: Deployment Age Tracking for Grace Periods

- **Novelty**: 0.52 — Grace periods in system health monitoring are standard (deployment-aware alerting is common in observability)
- **Evidence**: 0.91 — Widely used in production systems (e.g., Prometheus, DataDog, CloudWatch all support deployment-aware alerting rules)
- **Feasibility**: 0.98 — Requires storing deployment timestamp in PostgreSQL (one row in a metadata table) or reading from environment variable at startup
- **Description**: Use deployment age (now - deployment_timestamp) to suppress false alerts during initialization windows. This pattern applies to any background job: allow N hours post-deployment for first run before alerting on missing execution. For feedback_summary specifically, 24 hours provides sufficient buffer for the 6:15am cron to run (even if delayed).

### Idea 2: Composite Health Condition (Status Enumeration)

- **Novelty**: 0.68 — Three-state alerting (ok/degraded/critical) is standard; explicitly modeling empty+young vs populated+stale is novel to Recall
- **Evidence**: 0.88 — Grounded in observability best practices (PagerDuty, AlertManager, Datadog all use composite conditions)
- **Feasibility**: 0.94 — Requires one SQL query with CASE logic, can run in <10ms on small feedback_summary table (max 22,423 rows)
- **Description**: Define S6 as a state machine with three outputs: OK (empty within grace period, or recent sync), DEGRADED (stale by 48h), UNHEALTHY (never synced post-deployment). This eliminates ambiguity and allows the dashboard to display precise failure modes (stale data vs. worker crash vs. initialization).

### Idea 3: Two-Threshold Design (Grace Period + Staleness Horizon)

- **Novelty**: 0.74 — Dual-threshold alert design (allow time for initialization, then alert on staleness) is novel to Recall's health signals
- **Evidence**: 0.89 — Supported by observability literature (e.g., "deployment-aware alerting" in Google SRE Book)
- **Feasibility**: 0.95 — Requires storing only two values: deployment_time and staleness_threshold (2 days). No new infrastructure.
- **Description**: Implement two independent thresholds: (1) grace period = 24 hours (suppress alerts immediately post-deployment), (2) staleness limit = 48 hours (alert if last sync older than this). Both must pass for alert to fire. This pattern generalizes: any time-sensitive health check should have a grace period parameter.
