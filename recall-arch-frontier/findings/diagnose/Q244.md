## Research Question

How do Kafka Streams and Apache Flink implement tumbling window state stores? What happens when a window close event is missed? Does Flink's event-time watermark mechanism map to Recall's session-boundary compaction, and does it suggest a more robust trigger mechanism?

---

## How Stream Processing Handles Window Finalization

### Kafka Streams Tumbling Window

Kafka Streams uses **RocksDB-backed state stores** for windowed aggregation:

1. **Window boundary detection**: Based on wall-clock time or event timestamps. Tumbling window closes when `current_time > window_end`.
2. **Late event handling**: Default grace period of 0ms. Late events after window close are discarded (or sent to dead letter topic with DLQ enabled).
3. **Exactly-once semantics**: Kafka transactions ensure the aggregation result is committed atomically with the window close marker in the changelog topic.
4. **Missed window close**: If the processor crashes between the window close time and the commit, the window re-processes on restart (at-least-once default). With EOS enabled: exactly-once replay from the last committed offset.

**Key property**: the window close is driven by **time passing** in the event stream, not by an explicit "close" command. This is crucial: the window closes whether or not a specific event arrives.

### Apache Flink Event-Time Watermarks

Flink separates event-time (timestamp in the data) from processing-time (wall clock). Windows close based on event-time watermarks:

```
Watermark W = max(event_timestamps_seen) - allowed_lateness

When W >= window_end: window fires (closes)
```

**Late event handling**: Events with timestamp < W are "late." Flink keeps the window state open for `allowed_lateness` time after the watermark passes. Late events update the window result and trigger a delta output.

**The key mechanism**: the watermark advances automatically as events with higher timestamps arrive. The window close is not a "command" — it is an emergent property of the event stream. Even if no explicit "close" event arrives, when enough events with post-window timestamps are seen, the window fires.

## Mapping to Recall 2.0 Session-Boundary Compaction

| Stream processing concept | Recall 2.0 analog |
|---|---|
| Tumbling window | Session (one Claude Code session = one window) |
| Window close event | Stop hook (ClaudeCodeStop) |
| Event-time watermark | Next session's UserPromptSubmit timestamp |
| Window state store | Session WAL (Q226) |
| Aggregation result | Session summary + compaction output |
| Missed window close | Stop hook failure (Claude crash, force-kill) |

**The critical insight from Flink**: the window close mechanism should not be a single event (Stop hook). It should be a **watermark-style trigger**: when the next session's UserPromptSubmit arrives, the previous session's window automatically fires if the Stop hook hasn't already fired.

**Current Recall 2.0 vulnerability**: if the Stop hook fails (Claude process killed, network error, hook timeout), the session WAL is never processed. The session's memories are stored individually but the session-level summary, compaction, and entity-predicate updates never run.

**Flink-equivalent fix**: at UserPromptSubmit, check whether the previous session's compaction has run. If not (session WAL exists but no session summary for that session_id), run the compaction now. This is the **watermark-delayed finalization** pattern: the window closes at the next session start if it didn't close at session end.

## The Exactly-Once Guarantee Analog

Kafka's EOS for window finalization: commit the aggregation result atomically with the window close marker.

For Recall 2.0:
- **Window close marker**: a LMDB record `session_compacted: {session_id} → {timestamp}`
- **Atomic commit**: mark session_compacted only after all compaction operations succeed
- **On replay**: if session_compacted doesn't exist for a past session, replay the compaction from the session WAL

This is the **idempotent compaction replay** pattern: the compaction can be run multiple times on the same session WAL and produce the same result (because the session WAL is an immutable log).

The session WAL (Q226) provides this naturally: append-only log, fixed events per session, deterministic compaction output = exactly-once equivalent without distributed consensus.

## The Late Event Problem for Recall 2.0

Flink's `allowed_lateness` has a direct analog: memories stored after the session technically ended (asynchronous storage from a previous hook that completed late).

**Current behavior**: if a PostToolUse hook fires after the Stop hook has already run, the memory is stored with a new session_id (from the late hook's context) — effectively in a new session window, not the original session.

**Correct behavior** (Flink-equivalent): allow a grace period (e.g., 60 seconds after Stop hook fires) during which memories with the previous session_id are still attributed to that session and included in the compaction.

**Implementation**: session WAL tracks `session_close_time`. Any write within 60 seconds of `session_close_time` with `session_id == previous_session_id` is included in the session's compaction run.

## Verdict

**BREAKTHROUGH.** The Flink watermark-delayed finalization pattern directly solves Recall 2.0's Stop hook single-point-of-failure problem. The fix: at UserPromptSubmit, if the previous session's compaction has not run (no session_compacted marker in LMDB), run it now from the session WAL. The session WAL's append-only nature makes compaction replay idempotent — the Kafka EOS equivalent without distributed consensus. Additionally, a 60-second grace period for late PostToolUse writes (the Flink `allowed_lateness` analog) prevents late hook completions from orphaning memories into phantom sessions. Both mechanisms are buildable with zero additional infrastructure beyond the session WAL (Q226) and one LMDB marker per session.
