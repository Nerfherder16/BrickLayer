## Measurement

Current state of `delete_memory()` in `src/api/routes/memory.py` (lines 952–987):

```python
@router.delete("/{memory_id}")
async def delete_memory(memory_id: str, user: User | None = Depends(require_auth)):
    """Delete a memory."""
    try:
        qdrant = await get_qdrant_store()
        neo4j = await get_neo4j_store()

        # Multi-tenancy: non-admin users can only delete their own memories
        if user and user.id > 0:
            result = await qdrant.get(memory_id)
            if result:
                _, payload = result
                mem_owner = payload.get("user_id")
                if mem_owner is not None and mem_owner != user.id:
                    raise HTTPException(status_code=404, detail="Memory not found")

        # Delete from both stores + facts sub-embeddings
        await qdrant.delete(memory_id)
        await neo4j.delete_memory(memory_id)
        try:
            await qdrant.delete_facts_for_memory(memory_id)
        except Exception as facts_err:
            logger.debug("delete_facts_skipped", memory_id=memory_id, error=str(facts_err))
```

No comment mentions ordering, safety, orphan behavior, or the asymmetric consequence of partial failure.

## Finding

[OBJECTIVE] The delete_memory() API endpoint deletes Qdrant before Neo4j (line 969 precedes line 970) with no explanatory comment. The only inline comment at that block is "Delete from both stores + facts sub-embeddings" — which describes scope but says nothing about sequence or why sequence matters.

[DATA] Three deletion sites exist in the codebase, each with different ordering and different documentation quality:

1. **delete_memory() in memory.py** (the API endpoint): Qdrant first, then Neo4j. Zero ordering comment.
2. **gc.py** (the GC worker): Neo4j first, then Qdrant. Explicitly documented in the module docstring — "Order: Neo4j DETACH DELETE first, then Qdrant delete. Neo4j failure skips the entry (orphaned Qdrant point will be caught by next reconcile). Qdrant failure is logged and counted as skipped." Reasoning given: preserve reconcile coverage.
3. **reconcile.py** (the weekly repair worker): Qdrant declared as source of truth in module docstring — "Qdrant is source of truth. Orphaned Neo4j nodes are deleted; Qdrant nodes missing from Neo4j are recreated."

[FINDING] The safety invariant identified in Q132 — that Qdrant-first deletion is less harmful because Qdrant orphans get injected into context (surfacing the problem) while Neo4j orphans are silently skipped in BFS graph traversal — is real and verifiable. However, it is entirely undocumented in the API endpoint where it lives. Worse: gc.py deliberately uses the *reverse* ordering (Neo4j first) with explicit documented rationale, creating the appearance that the API endpoint's ordering is arbitrary rather than intentional. A developer reading both files would have no basis to determine which order is "correct" for the API path. The gap between the two orderings is not acknowledged anywhere.

## Qualifying Ideas

- **add-ordering-invariant-comment**: (N=0.72, E=0.85, F=0.99) — Add a three-line comment block immediately before `await qdrant.delete(memory_id)` in delete_memory() stating the ordering rationale: Qdrant must be deleted first because a partial failure leaving a Qdrant orphan is detectable (the memory surfaces in retrieval but has no graph context, creating a visible anomaly) whereas a partial failure leaving a Neo4j orphan is silent (BFS traversal skips unknown nodes, the memory simply disappears from graph results with no error). The comment should explicitly contrast with gc.py's reverse order: gc.py deletes Neo4j first because its failure mode is covered by the weekly reconcile worker that treats Qdrant as source of truth. The API path has no such backstop. Implementation time: under 5 minutes. No code change — comment only. The documentation pattern itself (module-level docstring explaining deletion order) already exists in gc.py and can be copied verbatim as a template.
