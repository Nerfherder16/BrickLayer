## Source Audit

### 1. Store Endpoint Schema (`src/api/routes/memory.py`)

**Request model** (`StoreMemoryRequest`):
```
content:      str   (required, 1–50000 chars)
memory_type:  str   (default: "semantic")
source:       str   (default: "user")
domain:       str   (default: "general")
tags:         list  (default: [])
importance:   float (default: 0.5, range 0.0–1.0)
confidence:   float (default: 0.8)
durability:   str | None  ("ephemeral" | "durable" | "permanent")
session_id:   str | None
metadata:     dict  (default: {})
```

**Response model** (`StoreMemoryResponse`):
```json
{
  "id": "<uuid>",
  "content_hash": "<sha256-hex>",
  "created": true,
  "message": "Memory stored successfully",
  "durability": null,
  "initial_importance": 0.95
}
```

Duplicate content returns `"created": false` and `"message": "Duplicate -- identical content already stored"`. Near-duplicates (cosine similarity above `semantic_dedup_threshold`) also return `"created": false`. Oracle memories must be semantically distinct enough to avoid dedup collapse.

### 2. Redis Architecture (`src/storage/redis_store.py`)

The `RedisStore` class exposes no method for writing arbitrary keys — it only manages session hashes, working memory lists, turn storage, pending signals, and the active sessions set. **There is no Recall API endpoint that allows setting arbitrary Redis keys.** The admin routes in `src/api/routes/admin.py` cover only: `/admin/consolidate`, `/admin/decay`, `/admin/ollama`, `/admin/users`.

**Conclusion**: The oracle registry key `recall:oracle:registry` must be written directly via `docker exec` + `redis-cli` on the host running the containers. The Redis container is named `recall-redis` (confirmed in `docker-compose.yml`).

## Failure Modes and Mitigations

| Symptom | Cause | Fix |
|---------|-------|-----|
| `"created": false` on store | Semantic dedup collision between two oracle memories | Rephrase the colliding oracle — ensure > ~10% semantic distance |
| `GET /memory/<UUID>` returns 404 | Qdrant write failed silently | Re-run the store curl; check Qdrant container health |
| Redis `GET recall:oracle:registry` returns nil | docker exec failed or wrong container name | Verify `docker ps` shows `recall-redis`; re-run SET |
| Health endpoint stays UNINITIALIZED after registry write | Health check not yet implemented at this endpoint | The endpoint implementation in Q148 must be deployed first |
| Health endpoint returns DEGRADED | Oracles retrievable but composite score below threshold | Check Ollama at 192.168.50.62:11434; re-run recall-retrieve retrieval smoke test |

## Key Findings

1. **No admin API for arbitrary Redis writes** — The Recall API has no endpoint to write arbitrary Redis keys. The `recall:oracle:registry` must be seeded via `docker exec recall-redis redis-cli SET`. This is a one-time bootstrap operation and is appropriate.

2. **Semantic dedup is a real risk** — The store endpoint rejects near-duplicates above `semantic_dedup_threshold`. The five oracle memories above are topically distinct (infrastructure, embeddings, hooks, scoring, identity) to avoid collapse.

3. **`durability: "permanent"` is essential** — Oracle memories must not be eligible for decay. The decay worker respects the durability field; permanent memories are exempt.

4. **Registry has no TTL** — Writing with plain `SET` (no `EXPIRE`) ensures the registry survives Redis restarts (with appendonly persistence enabled per docker-compose).

5. **End-to-end seeding is fully curl-scriptable** — The entire Phase 1–3 sequence can be wrapped in a `seed_oracles.sh` script that extracts UUIDs with `jq` and constructs the registry SET command automatically, enabling idempotent re-seeding.
