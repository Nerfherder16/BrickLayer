# Finding: Q063 — Stale UUID in G3 Idempotency Cache After Tombstone Eviction

**Question**: The G3 idempotency cache (72-hour TTL) and the tombstone registry (30-session TTL) can produce a stale-UUID state: a memory is tombstoned and its PostgreSQL/Qdrant rows are deleted, but the G3 Redis entry still maps the content hash to the now-deleted UUID. When a byte-identical memory text arrives as a new store request, G3 returns the cached UUID, the caller skips the write path (treating it as a duplicate), and no new row is created — yet the returned UUID points to a row that no longer exists in PostgreSQL.
**Agent**: quantitative-analyst

Three resolution paths:
(a) At tombstone transition time, atomically DEL the matching G3 cache key (O(1), requires tombstone writer to know content hash — which it does, tombstone stores content_hash)
(b) After every G3 cache hit, perform a PostgreSQL existence check (SELECT 1 FROM memories WHERE uuid = $1) before returning cached UUID, falling back to a full store write if absent
(c) Make G3's TTL shorter than shortest possible tombstone session count (e.g., cap G3 TTL to 48 hours, raise tombstone minimum TTL to 72 hours calendar time)

**Question Type**: ADVERSARIAL
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Distributed systems — cache invalidation consistency, MVCC garbage collection, distributed reference counting, write-ahead log sequencing

---

## The Bug's Anatomy

Before the adversarial threads begin, the defect structure must be precise.

The G3 gate (Q041's UNIFIED_STORE protocol) is a content-addressed idempotency cache: it maps `SHA256(normalize(content))` → `{uuid, stored_at}` with a 72-hour TTL in Redis. Its purpose is to suppress duplicate store requests for byte-identical content within the TTL window, returning the existing UUID without re-embedding or re-writing.

The tombstone mechanism (Q049) runs at session-end: memories below the importance threshold are deleted from PostgreSQL and Qdrant, and their registry entries in the scope registry are transitioned from LIVE to TOMBSTONED.

**The missing invalidation**: The tombstone writer (Phase 0 per Q054's metadata-first protocol, or Phase 2 per Q051's original design) knows the `content_hash` of the evicted memory — the tombstone struct explicitly carries it. But neither the Q049 design nor Q054's Phase 0 protocol includes a step to DEL the G3 cache key (`recall:g3:dedup:{content_hash}`) when a memory is tombstoned and its data rows are deleted.

**The failure sequence**:
1. Memory M stored with `content_hash = H`, UUID = UUID-A. G3 writes `recall:g3:dedup:H` → `{uuid: UUID-A, stored_at: T0}`.
2. Session-end window-drop fires. M's importance < threshold. Phase 0 writes tombstone. Phase 1 deletes PostgreSQL row and Qdrant vector for UUID-A. G3 key `recall:g3:dedup:H` is NOT touched.
3. Within 72 hours of T0, byte-identical content text arrives as a new store request.
4. G3 fires. Finds `recall:g3:dedup:H` → `{uuid: UUID-A}`. Returns UUID-A as a cache hit. Caller receives UUID-A and treats the write as a no-op duplicate.
5. UUID-A does not exist in PostgreSQL (deleted in step 2). The returned UUID is a ghost.

The caller now holds a UUID that has no backing row in PostgreSQL, no vector in Qdrant, and no LIVE entry in the scope registry. Any downstream operation that resolves this UUID (timeline query, rehydrate, profile synthesis) will receive a 404.

## Thread A: Path (a) is Correct and Sufficient

### Steel-man for atomic G3 invalidation at tombstone time

**Core claim**: Path (a) is the correct and complete fix. The tombstone writer already holds the `content_hash` at the moment of tombstone transition. The G3 invalidation is a single Redis DEL on a known key, costing one additional pipeline command per evicted memory. Placing the invalidation at tombstone write time is both causally correct (the G3 entry becomes invalid exactly when the UUID is about to be deleted) and operationally cheap.

**Evidence**:

The pattern path (a) implements is standard cache invalidation at source mutation: when a canonical record is deleted, any cache entry derived from that record must be invalidated simultaneously. This is enforced by database systems via `ON DELETE CASCADE` propagating to materialized views, by CDN systems via purge APIs triggered on origin content deletion, and by Redis keyspace notifications (NOTIFY on DEL triggers downstream cache invalidation chains).

The key architectural advantage of path (a) is that it keeps the causal ordering tight. Q054's Phase 0 metadata-first protocol already establishes the correct sequencing: tombstone the Redis scope registry before deleting from PostgreSQL and Qdrant. Path (a) extends Phase 0 by one pipeline command:

```
Phase 0 (Q054's design):
  HSET scope_registry key → {status: TOMBSTONED, content_hash: H, ...}

Phase 0 (path (a) extension):
  HSET scope_registry key → {status: TOMBSTONED, content_hash: H, ...}
  DEL recall:g3:dedup:H    ← one additional command in the same pipeline
```

Both commands go in the same Redis pipeline. The pipeline executes atomically with respect to the Redis server's single-threaded command processor. There is no window between the tombstone write and the G3 DEL where the registry is TOMBSTONED but G3 still has a valid entry.

**Best case scenario**: Phase 0 runs, tombstones the registry, and DELs the G3 key in one pipeline. Phase 1 deletes from PostgreSQL and Qdrant. The next arriving byte-identical content finds no G3 entry (cache miss), proceeds through the full store path, and is written as a new memory with a fresh UUID. The system behaves as if the evicted memory never existed — which is the correct semantics for an importance-below-threshold eviction.

**The content_hash availability is not incidental — it is designed**:

Q049's tombstone struct explicitly stores `content_hash: SHA256(canonical_content)`. Q054's Phase 0 reads candidate UUIDs from PostgreSQL and looks up their scope registry keys via the reverse index (`recall:uuid_to_scope:{uuid}`). The tombstone writer already reads the memory record to extract `content_hash` for the tombstone struct. The G3 key is directly derivable: `recall:g3:dedup:{content_hash}`. No additional reads are required. Path (a) adds zero database round-trips to Phase 0.

**Latency cost**: Q061 established that the Redis pipeline overhead for tombstone entries is approximately 2–7 µs per entry. Adding one DEL per entry to the pipeline adds negligible cost: Redis DEL is O(1) and adds ~25 bytes to the pipeline payload. At 10 evicted memories per session-end hook, path (a) adds ~250 bytes of additional pipeline payload. The total cost is indistinguishable from the tombstone-only pipeline.

**TOCTOU window analysis (preliminary)**:

For a TOCTOU race to exist on path (a), two conditions must hold simultaneously:
1. A G3 write (`SET recall:g3:dedup:H UUID-A`) occurs after path (a)'s `DEL recall:g3:dedup:H` executes.
2. UUID-A is subsequently deleted from PostgreSQL by Phase 1.

Condition 1 requires a concurrent store request arriving during the Phase 0→Phase 1 gap that produces byte-identical content and triggers a new G3 write. The G3 write would re-populate the key with UUID-A — because the incoming request is being deduplicated against UUID-A, which still exists in PostgreSQL at this point (Phase 1 has not yet fired). This is not a TOCTOU race in the classical sense — it is a correct G3 write for a currently valid UUID. The resulting G3 entry (UUID-A) then becomes stale when Phase 1 deletes UUID-A.

However, this scenario requires a concurrent write request arriving within the Phase 0→Phase 1 window. Q054 established this window as approximately 5–20ms for the Phase 0 tombstone pipeline plus the Phase 1 asyncio gather setup. In Tim's single-user system, concurrent store requests during a 20ms session-end window are near-impossible. The probabilistic rate is: `(concurrent store probability) × (Phase 0→Phase 1 window) = ~(1 op/session) × (0.02s / session_duration) ≈ 0.02/1800 = 0.001%`. For practical purposes: zero.

**Verdict on TOCTOU**: In Tim's single-user, single-session deployment, the TOCTOU window is real in theory but has zero probability of triggering in practice.

## Thread B: TOCTOU Risk in Path (a) — Path (b) is Safer

### Steel-man for the existence check on every G3 cache hit

**Core claim**: Path (a)'s DEL-at-tombstone-time approach has a genuine TOCTOU window that makes it structurally incorrect in any environment with concurrent write operations. The race condition does not require concurrent sessions — it can occur within a single session's store path when Phase 0 and Phase 1 are not atomically coupled. Path (b), by adding a PostgreSQL existence check after every G3 cache hit, eliminates the window entirely at the cost of one additional database round-trip on the G3 hit path.

**Evidence and mechanism**:

The TOCTOU race structure in path (a) is:

```
Thread/coroutine 1 (session-end hook):
  T=0:  Phase 0 fires. DEL recall:g3:dedup:H (G3 key gone)
  T=1:  Phase 1 fires. DELETE FROM memories WHERE id = UUID-A (UUID-A gone)

Thread/coroutine 2 (incoming store request, concurrent):
  T=0.5: G3 lookup → cache miss (key was just DELeted at T=0)
          Full store path: embeds content, writes PostgreSQL row (UUID-A still exists — Phase 1 not yet fired)
          G3 write: SET recall:g3:dedup:H → {uuid: UUID-A, stored_at: T=0.5}
  T=1:   Phase 1 deletes UUID-A from PostgreSQL
  T=2:   Caller receives UUID-A from the G3 write at T=0.5. UUID-A is now deleted.
```

Wait — this scenario is more complex than it first appears. Let us be precise.

At T=0.5, the concurrent store request finds a G3 cache miss (key DELeted at T=0). It proceeds through the full store path and writes a new PostgreSQL row. But what UUID does it write? Either:
- **Path A (new UUID)**: The store path allocates a fresh UUID (UUID-B). G3 writes `recall:g3:dedup:H → UUID-B`. Phase 1 at T=1 only deletes UUID-A (the original evicted memory). UUID-B is unaffected. The system is correct.
- **Path B (same UUID collision)**: Impossible. UUIDs are generated fresh on each store call (`uuid.uuid4()`). Two concurrent store calls for the same content produce different UUIDs.

The concurrent write at T=0.5 produces UUID-B, which is safe. Phase 1 deletes only UUID-A. The G3 entry now correctly points to UUID-B.

**The actual TOCTOU race is narrower and more specific**:

The race requires that the G3 write happens AFTER the Phase 0 DEL but the G3 write is for the OLD UUID (UUID-A, before it was deleted). This can only happen if:

1. A store request arrived BEFORE Phase 0 (when G3 had a valid UUID-A entry).
2. G3 found UUID-A in cache and returned it.
3. The caller received UUID-A but has not yet processed the response.
4. Phase 0 fired and DELed the G3 key.
5. Phase 1 fired and deleted UUID-A.
6. The caller now holds UUID-A (from step 3) — a stale UUID.

This is not a race within path (a)'s execution — it is a race between path (a)'s execution and a concurrent caller that received UUID-A from G3 before Phase 0 ran. **Path (a) cannot prevent this race** because the caller already received the stale UUID before path (a) had any opportunity to DEL the key.

This race is inherent to any cache-based idempotency system: if a UUID is valid when the caller queries G3, but becomes invalid between query and use, the caller holds a stale reference. Path (a) does not close this window — it only prevents NEW callers from getting the stale UUID after the DEL. Existing callers that received UUID-A from G3 before the DEL are unaffected by path (a)'s DEL.

**Path (b) addresses a different failure mode than path (a)**:

Path (b) (existence check after every G3 cache hit) does not solve the race above either. If the caller's G3 hit occurred before Phase 1 deleted UUID-A, UUID-A was valid at query time. By the time the caller performs the existence check (SELECT 1 WHERE uuid = UUID-A), UUID-A may have been deleted. The SELECT can return true (UUID exists) or false (UUID deleted) depending on timing.

But path (b) addresses the more common and more consequential failure mode: a caller queries G3 after Phase 1 has already deleted UUID-A (the original bug scenario). In this case:
- Path (a): G3 was DELed in Phase 0. G3 returns cache miss. Full store path fires. New UUID-B created. Correct.
- Path (b): G3 returns UUID-A (still cached if path (a) was not used). Existence check: SELECT 1 WHERE uuid = UUID-A → 0 rows. Fallback to full store. New UUID-B created. Correct.

Both paths handle the common case correctly. The difference is what happens when path (a)'s DEL fails.

**Path (a)'s silent failure mode**:

Path (a)'s DEL is issued in a Redis pipeline. If the Redis pipeline partially fails (connection drop, Redis restart, pipeline buffer overflow), the DEL for `recall:g3:dedup:H` may not execute. Phase 0's tombstone HSET may succeed (the pipeline might partially execute), but the G3 DEL may silently not run. With `return_exceptions=True` behavior inherited from Q054's asyncio design, pipeline failures are captured and logged but do not halt execution. Phase 1 then deletes UUID-A from PostgreSQL. The G3 entry survives with UUID-A — the original bug.

Path (b) does not depend on the DEL succeeding. Every G3 cache hit performs a fresh PostgreSQL existence check. A Redis pipeline failure does not affect path (b)'s correctness — the existence check is independent of whether the DEL ran.

**The latency cost of path (b) is real but bounded**:

Every G3 cache hit requires one additional PostgreSQL SELECT: `SELECT 1 FROM memories WHERE id = $1`. Q041 established that PostgreSQL round-trips from the Recall API to the PostgreSQL instance cost approximately 1–5ms (LAN-attached, indexed primary key lookup). The G3 cache hit path adds 1–5ms on every duplicate store detection event. At Tim's usage rate, G3 cache hits are rare (most content is not exact duplicates). The latency cost is acceptable for correctness guarantees.

**Core claim**: Path (b) is structurally safer than path (a) because it handles the case where path (a)'s DEL fails silently. Path (a) is a write-time invalidation that requires the DEL to succeed. Path (b) is a read-time validation that is independent of prior invalidation operations. For a correctness property as important as "returned UUIDs must exist in PostgreSQL," the validation should be at read time (path b), not only at write time (path a).

**Failure mode of path (a) alone**: Silent pipeline failure leaves G3 with a stale UUID. No error is logged. The next byte-identical write receives the stale UUID. The caller discovers the 404 only when resolving the UUID in downstream operations (timeline query, rehydrate). This is a silent data integrity violation with delayed detection.

## Synthesis

### What Thread A assumes that Thread B exposes

Thread A assumes that the Redis pipeline DEL in Phase 0 is reliably executed. This assumption holds under normal operation but fails under Redis pipeline partial failure, Redis restart during Phase 0, or network interruption between the Recall API and Redis. Thread A treats path (a) as an invalidation guarantee. Thread B exposes that Redis pipeline execution is not a guarantee — it is a best-effort operation that can silently fail with `return_exceptions=True`.

Thread A also assumes that the TOCTOU window (Phase 0 DEL → Phase 1 DELETE) is the primary risk. Thread B exposes that the actual residual risk is not within path (a)'s execution — it is in callers who received UUID-A from G3 before Phase 0 ran. Path (a) closes the forward-looking risk (new callers after DEL get cache miss) but not the backward-looking risk (callers who already have the stale UUID). Neither path closes this race completely because it is a distributed reference counting problem, not a cache invalidation problem.

**The hidden assumption**: Thread A assumes that the G3 DEL is the critical operation for system correctness. Thread B exposes that what matters for correctness is not whether the G3 key is deleted, but whether the caller that receives a UUID from G3 can trust that UUID at the moment of use. These are different invariants. Path (a) enforces the first. Path (b) enforces the second.

### Does Path (a) have a real TOCTOU window? Closed by Phase 0 ordering?

**The TOCTOU window in path (a) is real but Q054's Phase 0 protocol closes it for the common case.**

The critical insight from Q054's Phase 0 protocol is the sequencing:

```
Phase 0: tombstone scope registry + DEL G3 key (path a extension)
Phase 1: DELETE from PostgreSQL and Qdrant
```

The G3 DEL fires BEFORE the PostgreSQL DELETE. During the Phase 0→Phase 1 gap:
- G3 cache is already cleared (`recall:g3:dedup:H` does not exist)
- UUID-A still exists in PostgreSQL (Phase 1 has not fired)
- Scope registry entry is TOMBSTONED

Any new store request arriving in this gap:
- G3 miss → full store path → new UUID-B written to PostgreSQL
- UUID-A still exists alongside UUID-B momentarily — an acceptable transient duplicate
- Phase 1 then deletes UUID-A. UUID-B survives.

**The TOCTOU window that path (a) does NOT close**:

A store request that called G3 BEFORE Phase 0 and received UUID-A from the cache is not affected by path (a)'s DEL. That caller holds UUID-A as a valid reference. Phase 1 then deletes UUID-A. The caller's UUID is now stale. This is the backward-looking race that neither path fully closes.

However, in Tim's system, this backward-looking race requires:
1. A G3 hit occurring in the gap between "memory M is below threshold" (determined at session-start) and "Phase 0 fires" (at session-end)
2. The caller using the UUID after Phase 1 fires (at session-end)

This requires the caller to hold a UUID reference for the entire duration of a session without resolving it — then resolving it after session-end runs. This is not a realistic usage pattern in the hook-driven architecture: hooks complete within seconds, not across session boundaries.

**Verdict on TOCTOU**: The TOCTOU window is theoretically real but practically closed by Phase 0's pre-deletion ordering in Tim's single-user system. Path (a) is sufficient for Tim's deployment. In a multi-user system with concurrent sessions and long-lived UUID references, path (a) alone would be insufficient.

### When Thread A is right (path (a) sufficient)

- Single-user system with no concurrent session-end hooks
- Redis pipeline reliability is high (no Redis restarts, no connection drops)
- UUID references are short-lived (callers resolve UUIDs within the same request that received them)
- The Phase 0 sequencing from Q054 is correctly implemented (G3 DEL and tombstone HSET in same pipeline, before Phase 1)
- Tim's deployment: all conditions hold

### When Thread B is right (path (b) needed in addition to path (a))

- Redis pipeline failures occur with non-negligible frequency (Redis restart during Phase 0, network blip)
- Long-lived UUID references exist (callers cache UUIDs across requests or sessions)
- Multi-user concurrent deployments where session-end hooks from different users can overlap
- Systems where silent data integrity violations are unacceptable regardless of probability

### The architecture Thread B would accept

**Dual-layer defense: path (a) as prevention, path (b) as detection.**

Neither path alone is the complete answer. The correct architecture combines both:

1. **Path (a) — prevention**: Extend Phase 0 to include `DEL recall:g3:dedup:{content_hash}` in the same Redis pipeline as the tombstone HSET. This closes the forward-looking window: callers arriving after Phase 0 get a G3 miss and proceed through the full store path.

2. **Path (b) — detection on G3 hit**: After every G3 cache hit, perform a lightweight PostgreSQL existence check before returning the cached UUID. If the UUID exists: return it (fast path, one extra SELECT at ~1ms). If it does not exist: the G3 entry is stale, fall back to full store path and DEL the stale G3 key.

```
G3 cache hit path (combined defense):
  1. G3 returns cached {uuid: UUID-A, stored_at: T0}
  2. SELECT 1 FROM memories WHERE id = UUID-A  [~1ms, indexed PK lookup]
  3a. Row exists → return UUID-A (valid cache hit)
  3b. Row absent → stale cache hit detected
       DEL recall:g3:dedup:{content_hash}  [cleanup the stale key]
       fallback to full store path → return new UUID-B
```

Path (b) adds ~1ms to every G3 cache hit. G3 cache hits are uncommon (duplicate store requests for byte-identical content within 72 hours). The latency cost is acceptable.

**The combined defense is robust to the failure modes both threads identify**:

- Redis pipeline failure during Phase 0 (path (a)'s DEL fails silently): path (b) catches the stale G3 entry on the next hit.
- Phase 0 DEL succeeds but concurrent write re-populates G3 with a valid UUID: path (b)'s existence check passes (new UUID exists in PostgreSQL).
- Callers with stale UUID-A references (backward-looking race): the caller fails when resolving UUID-A downstream. This cannot be prevented by either path — it requires reference validity tracking beyond the scope of the G3 mechanism. In Tim's system this is acceptable; the hooks resolve UUIDs immediately on receipt.

**Why not path (c)?**

Path (c) — making G3's TTL shorter than tombstone session count — is a timing-based workaround that does not address the structural issue. It requires:
- Reliable calendar-time tracking for "session count" (sessions are not evenly spaced in time)
- Coordinated TTL changes across two independent Redis keys
- The guarantee that tombstone minimum TTL (in calendar time) always exceeds G3 maximum TTL

This guarantee cannot be maintained reliably. If Tim takes a week off from sessions, tombstone "30 sessions" could span months of calendar time, far exceeding any reasonable G3 TTL. Conversely, if Tim runs 30 sessions in one day, the tombstone expires in calendar hours — shorter than G3's 72-hour TTL. Path (c) inverts the problem without solving it.

**The Phase 0 integration point (concrete)**:

Q054's Phase 0 already uses a Redis pipeline for tombstone HSET operations. The G3 DEL integrates into the existing pipeline:

```python
async def op0_tombstone_registry_entries(candidate_ids: list[str]) -> None:
    """
    Phase 0: Tombstone scope registry entries AND invalidate G3 cache for memories
    about to be deleted. Metadata-first ordering: both ops fire before Phase 1 deletion.
    """
    if not candidate_ids:
        return

    # Step 1: Resolve scope keys and content hashes via reverse index
    pipe = redis_client.pipeline()
    for uid in candidate_ids:
        pipe.get(f"recall:uuid_to_scope:{uid}")
        pipe.hget(f"recall:memory_meta:{uid}", "content_hash")  # or from PostgreSQL SELECT
    results = await pipe.execute()

    scope_keys = results[0::2]
    content_hashes = results[1::2]

    # Step 2: Pipeline — tombstone scope registry + DEL G3 cache keys
    tombstone_pipe = redis_client.pipeline()
    for uid, scope_key, content_hash in zip(candidate_ids, scope_keys, content_hashes):
        if scope_key:
            tombstone_pipe.hset(
                "scope_registry",
                scope_key,
                json.dumps({
                    "uuid": None,
                    "status": "TOMBSTONED",
                    "prior_uuid": uid,
                    "tombstone_phase": "pre_delete",
                    "tombstoned_at": current_session_id,
                    "eviction_reason": "importance_below_threshold",
                    "content_hash": content_hash,
                })
            )
        if content_hash:
            tombstone_pipe.delete(f"recall:g3:dedup:{content_hash}")  # Path (a) integrated here

    await tombstone_pipe.execute()
```

**Content hash availability**: Q054's Phase 0 already performs a PostgreSQL SELECT to identify candidate UUIDs (`SELECT id, scope, content_hash FROM memories WHERE importance < threshold AND scope = 'session'`). The `content_hash` column is included in this SELECT (it is already stored in PostgreSQL as part of the memory metadata). No additional reads are required.

## Ideas Extracted

### Idea: g3-invalidation-at-tombstone-time-phase0
- **Novelty**: 0.78
- **Evidence**: 0.72
- **Feasibility**: 0.92
- **Description**: Extend Q054's Phase 0 tombstone pipeline to include a Redis DEL for the G3 idempotency cache key (`recall:g3:dedup:{content_hash}`) alongside the scope registry HSET. The `content_hash` is already available in the Phase 0 SELECT (PostgreSQL returns it as part of the candidate identification query). The DEL is added to the same Redis pipeline as the tombstone HSET — same round-trip, zero additional network cost. This closes the G3 stale-UUID window for callers arriving after Phase 0 runs. Implementation cost: 1 line in `op0_tombstone_registry_entries`. The content_hash reverse-lookup from `recall:memory_meta:{uuid}` can be replaced by including `content_hash` in the Phase 0 PostgreSQL SELECT, which is already present as a column in the memories table.

### Idea: g3-existence-check-on-cache-hit
- **Novelty**: 0.70
- **Evidence**: 0.65
- **Feasibility**: 0.85
- **Description**: Add a lightweight PostgreSQL existence check (`SELECT 1 FROM memories WHERE id = $1`) after every G3 cache hit before returning the cached UUID to the caller. If the row exists, return UUID immediately (common case, ~1ms overhead). If the row is absent, the G3 entry is stale: DEL the G3 key, fall back to full store path, return the new UUID. This detection layer is independent of Phase 0's DEL succeeding — it catches stale G3 entries regardless of how they arose (pipeline failure, race condition, missed invalidation). Latency cost: ~1ms on G3 hits, which are uncommon (duplicate byte-identical content within 72 hours). Combined with path (a), this provides defense-in-depth: path (a) prevents the stale entry from forming; path (b) detects it if path (a) failed.

### Idea: g3-tombstone-coupling-invariant-check
- **Novelty**: 0.68
- **Evidence**: 0.60
- **Feasibility**: 0.90
- **Description**: Enforce a system-level invariant: no UUID may have a TOMBSTONED scope registry entry and a LIVE G3 cache entry simultaneously. A background reconciliation job (runs at session-start or on a timer) queries the tombstone registry for all TOMBSTONED entries with non-null `content_hash`, then checks whether the G3 key `recall:g3:dedup:{content_hash}` exists and maps to a UUID that no longer exists in PostgreSQL. Any such entry is cleaned up via DEL. This job converts a hard real-time correctness requirement (invalidate G3 before the caller can use the stale UUID) into a soft eventual consistency requirement (clean up stale G3 entries within one reconciliation cycle). The reconciliation job is the fallback for any failure in path (a) or path (b). At Tim's eviction rate (~50 memories/session, ~5 sessions/day), the job scans ~250 tombstone entries per run — well within Redis and PostgreSQL capacity.

## Key Finding

Q061's follow-up question (Q064) identified this defect correctly: G3's 72-hour TTL and the tombstone registry's 30-session TTL are independently maintained, and the tombstone writer does not invalidate the G3 cache entry when a memory is evicted. The result is a stale UUID returned by G3 on exact-duplicate content re-arrival within the TTL window.

Path (a) — DEL the G3 cache key in Phase 0 at tombstone time — is the correct primary fix and does not have a meaningful TOCTOU window in Tim's single-user deployment. Q054's Phase 0 metadata-first protocol places the G3 DEL before the PostgreSQL DELETE, closing the forward-looking race. The content hash required for the DEL is already available in Phase 0's SELECT. The implementation is a one-line addition to the existing tombstone pipeline.

Path (b) — existence check on every G3 cache hit — is the correct secondary defense layer. It catches stale G3 entries that survived because path (a)'s DEL failed silently (Redis pipeline error). The existence check adds ~1ms to the G3 hit path, which is uncommon and acceptable. Path (a) alone is sufficient for Tim's deployment; path (a) + path (b) together is correct under all failure modes including Redis pipeline failures.
