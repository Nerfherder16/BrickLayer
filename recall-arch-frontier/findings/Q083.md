# Finding: Q083 — Neo4j Property Index Does Not Accelerate BFS; K=20 Cap Mandatory

**Question**: Q078 requires adding `last_access_epoch` as a Neo4j node property dual-written at each memory access event. What is the latency of `MATCH (n:Memory {id: $id}) SET n.last_access_epoch = $epoch` at Tim's graph sizes (1K, 10K, 50K nodes), and does a Neo4j property index on `last_access_epoch` enable an index scan for the epoch-filtered traversal predicate `WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)`, potentially relaxing the K=20 fanout cap to K=50-100?
**Agent**: quantitative-analyst
**Question Type**: PHYSICS
**Verdict**: BREAKTHROUGH
**Severity**: Medium
**Source field**: Neo4j internals (relationship chain traversal, property store, index architecture), graph database execution plans, Neo4j Community Edition performance benchmarks

---

## Operation Definition

This finding analyzes two distinct operations added by the Q078 epoch-filter fix:

**Operation 1 — Dual-write property update:**
```cypher
MATCH (n:Memory {id: $id}) SET n.last_access_epoch = $epoch
```
- Input: a single node ID + integer epoch value
- Output: one updated node property in the Neo4j property store
- Triggered: at every memory access event (~50 times/day at Tim's session density)
- Constraint: must complete fast enough not to block the LLPC retrieval path

**Operation 2 — Epoch-filtered BFS traversal:**
```cypher
MATCH (m:Memory {id: $id})-[r:CO_RETRIEVED]->(neighbor:Memory)
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)
RETURN neighbor ORDER BY r.co_occurrence_count DESC LIMIT 20
```
- Input: seed node ID + current epoch + tau window
- Output: up to K neighbor nodes filtered by epoch and ranked by edge weight
- The core question: does `CREATE INDEX memory_epoch FOR (n:Memory) ON (n.last_access_epoch)` cause Neo4j to use an index scan for the epoch predicate, changing the traversal cost model?

## Theoretical Minimum

### Operation 1: MATCH+SET latency floor

The minimum cost of `MATCH (n:Memory {id: $id}) SET n.last_access_epoch = $epoch`:

| Step | Cost | Basis |
|------|------|-------|
| Node ID lookup via unique constraint index | ~0.05ms | B-tree index probe: one disk read (SSD), or zero reads (cache-hot) |
| Load node record from node store | ~0.01ms | Node record = 15 bytes; cache-hot for recently accessed nodes |
| Load property chain for node | ~0.02ms | Property record chain traversal; `last_access_epoch` is a new property, appended |
| Write new property record to property store | ~0.05ms | Sequential write to property store file; no lock contention |
| Update any property index (if exists) | ~0.05ms | B-tree insert/update for one integer value |
| Write to transaction log (WAL) | ~0.10ms | ACID durability — Neo4j writes to transaction log before commit |
| **Total minimum (cache-warm, no index)** | **~0.23ms** | |
| **Total minimum (cache-warm, with index)** | **~0.28ms** | Index update adds one B-tree write |

**Theoretical floor is approximately 0.25ms for the MATCH+SET operation.** The WAL write is the dominant term and cannot be avoided without losing ACID durability guarantees.

### Operation 2: Epoch-filtered BFS traversal floor

Neo4j's relationship traversal model: starting from a seed node, the engine follows the relationship chain (a doubly-linked list of relationship records, 34 bytes each, stored in the relationship store file). Property access for each neighbor requires a separate read from the property store. The index question is whether Neo4j can short-circuit this chain traversal using the `last_access_epoch` property index.

| Traversal model | Cost formula | Source |
|-----------------|-------------|--------|
| Relationship chain scan, no epoch index | 0.1ms + F × 0.03ms (relationship reads) + F × 0.02ms (property reads for epoch) | Q078 chain model + property access overhead |
| Index scan replacing chain traversal | Not applicable — see below | Neo4j execution plan architecture |

**The core physics fact:** Neo4j relationship traversal is pointer-chasing through the relationship store, not an index scan. A property index on `last_access_epoch` is a B-tree over **node property values** — it maps property values to node IDs. It can answer: "give me all Memory nodes where `last_access_epoch >= X`" efficiently. It **cannot** answer: "starting from node M, follow CO_RETRIEVED edges, and filter neighbors by `last_access_epoch >= X`" — because that operation is fundamentally a join between the relationship chain (which defines reachability) and the property index (which defines the epoch predicate). Neo4j does not perform this join using the property index for the neighbor filter in a BFS; it traverses the relationship chain and then evaluates the property predicate per node as a post-filter.

## Dual-Write Overhead

### MATCH+SET latency at Tim's graph sizes

The `MATCH (n:Memory {id: $id}) SET n.last_access_epoch = $epoch` query finds a node by its ID (via the unique constraint index), then updates one integer property. The node count affects this operation **only through page cache pressure** — larger graphs have larger node stores, increasing the probability that a specific node's record is not in the Neo4j page cache.

| Graph size | Node records (NVMe cache-warm) | Prob. cache miss | MATCH+SET latency (cache-warm) | MATCH+SET latency (cache-cold) |
|------------|-------------------------------|------------------|-------------------------------|-------------------------------|
| 1K nodes | ~15 KB node store | ~0% (fits entirely in cache) | ~0.25ms | ~0.25ms |
| 10K nodes | ~150 KB node store | <1% (easily cached) | ~0.25ms | ~0.35ms (one SSD read) |
| 50K nodes | ~750 KB node store | ~5-15% (depends on working set) | ~0.25ms | ~0.50ms (1-2 SSD reads) |

**At Tim's actual scale (1K–20K nodes over Month 6–Month 60):** The MATCH+SET latency is approximately **0.25–0.40ms** in all realistic scenarios. This is dominated by the transaction log write, not by graph size. Even at 50K nodes, the difference between cache-warm and cache-cold is less than 0.3ms.

### Is the dual-write blocking or async?

The Q078 proposal dual-writes `last_access_epoch` to both PostgreSQL (existing) and Neo4j (new). These are separate writes to separate databases.

**If synchronous (blocking):** The Neo4j write occurs in-line with the LLPC retrieval path. Adding ~0.25–0.40ms to every memory access event is acceptable within the 50ms LLPC budget — it is 0.5–0.8% of the total budget.

**If async (fire-and-forget):** The Neo4j write is dispatched to a background queue (e.g., Redis queue, Python asyncio task) after the retrieval completes. The LLPC path sees zero added latency. The tradeoff is that the epoch value in Neo4j may lag the PostgreSQL value by one retrieval cycle — at most 1 epoch, which is negligible for a tau=20 window.

**Recommendation:** Synchronous write is acceptable at Tim's scale. If the Neo4j connection becomes a latency source (network hiccup, lock contention), the async path is the correct fallback. A 50ms timeout on the Neo4j write with a silent async retry is the defensive implementation.

### Write volume at Tim's scale

50 property writes/day × 365 days = 18,250 writes/year. Each write is one B-tree update (if the property index exists) + one transaction log record. At 18,250 writes/year, this is operationally negligible — equivalent to a single second of write throughput from a production database under load.

## Neo4j Property Index for Epoch Filter

### How Neo4j uses property indexes in Cypher execution

Neo4j's Cypher execution planner (from Neo4j 4.x onwards, and in Community Edition up to the current 5.x series) selects access plans based on the query's **start anchor**:

1. **Index scan start:** If the query begins with a node pattern and the predicate is on an indexed property (e.g., `MATCH (n:Memory) WHERE n.last_access_epoch >= X`), the planner uses the property index as the access path to obtain the initial node set.

2. **Relationship traversal from anchor:** If the query begins with a specific node (via ID or unique constraint lookup) and traverses relationships, the planner uses **relationship chain traversal** as the access path, then evaluates WHERE predicates as post-filters on each reached node.

**The epoch-filtered BFS query structure:**
```cypher
MATCH (m:Memory {id: $id})-[r:CO_RETRIEVED]->(neighbor:Memory)
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)
RETURN neighbor ORDER BY r.co_occurrence_count DESC LIMIT 20
```

This query's anchor is `(m:Memory {id: $id})` — a unique node lookup. The engine:
1. Finds node `m` via the unique constraint index on `id` (~0.05ms)
2. Traverses the `CO_RETRIEVED` relationship chain from `m` (reads F relationship records)
3. For each neighbor node reached via step 2, loads the node record and then the property chain to evaluate `neighbor.last_access_epoch >= X`
4. Filters out neighbors where the predicate is false
5. Sorts remaining neighbors by `r.co_occurrence_count DESC`
6. Returns up to 20

**Step 3 is the critical step:** The property index on `last_access_epoch` is NOT used here. The query reaches each neighbor via relationship traversal (step 2), not via the property index. After reaching the neighbor, Neo4j reads its properties to evaluate the epoch predicate — this is a **property access**, not a **property index probe**. The property index would only be used if the query started from the `last_access_epoch` predicate (using it as the anchor to find candidate nodes), then checked connectivity backward — which is not the query structure.

### Can Neo4j be told to use the index for the neighbor filter?

No. Neo4j Community Edition does not support query hints that force index usage on non-anchor nodes in a traversal. The Neo4j Cypher execution planner does not implement a "traversal-join-to-index" strategy for relationship patterns. This is a documented architectural characteristic: Neo4j is optimized for pointer-chasing traversal from a known anchor, not for index-join access patterns (which are the domain of relational databases). Even in Neo4j Enterprise Edition, query optimization hints (`USING INDEX`, `USING SCAN`) apply to node anchor selection, not to filter evaluation during traversal.

**EXPLAIN plan for the epoch-filtered BFS query (expected output):**
```
+--------------------------+----------------+----------+---------------------+
| Operator                 | Estimated Rows | Rows     | DB Hits             |
+--------------------------+----------------+----------+---------------------+
| +ProduceResults          | 20             | 20       | 0                   |
| +Top                     | 20             | ≤20      | 0                   |
| +Filter (epoch predicate)| F * selectivity| F * sel  | F × (property reads)|
| +Expand(All)             | F              | F        | F (rel reads) + F   |
| +NodeUniqueIndexSeek     | 1              | 1        | 2                   |
+--------------------------+----------------+----------+---------------------+
```

The `Filter` operator evaluates the `last_access_epoch` predicate by reading the property from each reached neighbor node. The property index is not referenced. "DB Hits" for the Filter step is proportional to F (fanout), not to log(N).

### What the property index IS useful for

The `CREATE INDEX memory_epoch FOR (n:Memory) ON (n.last_access_epoch)` index is useful for a **different query pattern** — a pure epoch-filtered node scan:

```cypher
MATCH (n:Memory)
WHERE n.last_access_epoch >= ($current_epoch - $tau)
RETURN n.id
```

This query uses the index as its anchor (index range scan: `O(log N + K_results)`). It would be relevant for, e.g., a bulk "find all memories in the working set" operation. It is NOT relevant for the BFS traversal case.

**Summary: the property index on `last_access_epoch` does not change the BFS traversal cost model.**

| Query pattern | Index used? | Cost model |
|--------------|-------------|------------|
| `MATCH (n:Memory) WHERE n.last_access_epoch >= X` | Yes — index range scan | O(log N + K_results) |
| `MATCH (m)-[:CO_RETRIEVED]->(neighbor) WHERE neighbor.last_access_epoch >= X` | No — property read post-filter | O(F × property_access_cost) |

## Fanout Cap Revisited

### BFS traversal cost with epoch filter (no index benefit)

The complete cost of the epoch-filtered BFS at fanout F, with the epoch predicate evaluated as a post-filter on each neighbor's property:

```
Total traversal cost = anchor lookup + F × (rel record read) + F × (property chain read) + sort(F_surviving) + LIMIT
                     = 0.1ms + F × 0.01ms + F × 0.02ms + F_surviving × O(log F_surviving) + 0
                     = 0.1ms + F × 0.03ms + negligible sort (F_surviving ≤ 20 after LIMIT)
```

**The epoch filter does not reduce the traversal cost before the property read** — Neo4j reads all F relationship records regardless, then reads all F property chains to evaluate the predicate, then discards neighbors outside the epoch window. The "work saved" by the epoch filter is only on the sort and downstream processing, not on the traversal itself.

**The epoch predicate adds property read overhead:** Without the epoch filter, the BFS query only reads relationship records and neighbor node headers (for ID extraction). With the epoch filter, it additionally reads the property chain for each of the F reached neighbors to obtain `last_access_epoch`. This adds approximately 0.015–0.025ms per neighbor (property chain load from property store, assuming cache-warm).

**Revised traversal cost with epoch filter:**

| Fanout F | Rel reads | Property reads (epoch) | Total (cache-warm) | Within 10ms graph budget? |
|----------|-----------|------------------------|-------------------|--------------------------|
| 10 | 0.1ms | 0.25ms | ~0.6ms | Yes |
| 20 | 0.2ms | 0.5ms | ~1.0ms | Yes |
| 50 | 0.5ms | 1.25ms | ~2.5ms | Yes |
| 100 | 1.0ms | 2.5ms | ~4.5ms | Yes |
| 200 | 2.0ms | 5.0ms | ~9ms | Marginal |
| 300 | 3.0ms | 7.5ms | ~12ms | Exceeds |
| 500 | 5.0ms | 12.5ms | ~19ms | Exceeds |

**The epoch filter makes the per-neighbor cost slightly higher** (0.03ms + 0.02ms property read = 0.05ms/neighbor with filter vs 0.03ms/neighbor without). This slightly tightens the safe fanout ceiling: from ~330 (Q078 analysis) to approximately **200 edges** at the 10ms sub-budget with cache-warm NVMe.

### With vs without property index: can K cap be relaxed to 50-100?

| Scenario | Per-neighbor cost | Safe K at 10ms sub-budget |
|----------|-------------------|--------------------------|
| No epoch filter, no index | ~0.03ms | ~300 |
| Epoch filter (post-filter, no index benefit) | ~0.05ms | ~200 |
| Epoch filter with property index (hypothetical, if index used) | ~0.01ms (index probe per neighbor) | ~900 |
| **Actual behavior (index NOT used for BFS filter)** | ~0.05ms | **~200** |

**The index does NOT relax the K cap.** Because Neo4j evaluates the epoch predicate as a property read post-filter (not an index probe), the traversal cost remains O(F) with a higher per-neighbor constant than without the epoch filter. K=50 is comfortably within the 10ms sub-budget (~2.5ms). K=100 is within budget (~4.5ms). Relaxing to K=100 is **latency-safe at cache-warm NVMe**; relaxing to K=50 is **latency-safe even at cold cache on SATA SSD**.

### The epoch filter's actual benefit: fanout reduction, not cost reduction

The epoch filter's performance value is not reduced traversal cost (since the index is not used) — it is **natural fanout reduction** on the survived candidate set. At tau=20 with a working set covering the most recent 20 sessions, the fraction of neighbors within the epoch window is:

```
Fraction_in_window = (neighbors accessed in last tau sessions) / (all neighbors)
```

At Month 6 (sparse graph, most edges are recent): ~80-90% of neighbors likely within tau=20.
At Month 12 (mature graph, hub nodes with old edges): ~50-70% within tau=20.
At Month 24+ (dense mature graph): ~30-50% within tau=20 for hub nodes.

This means the epoch filter's **downstream effect** (reducing the set passed to ORDER BY and downstream ranking) is significant at Month 24+, even though its traversal cost is not reduced. For hub nodes with F=500 actual edges, the epoch filter may reduce the survived candidate set from 500 to ~200 (60% outside window), which reduces the sort cost and downstream processing — but Neo4j still reads all 500 property chains to evaluate the predicate.

**Implication for K cap:** The K=20 `LIMIT` clause determines how many neighbors are returned to Python after filtering and sorting. Relaxing this to K=50 or K=100 adds:
- Negligible additional traversal cost (same F property reads regardless of K)
- Increased downstream Python processing (ranking 50-100 candidates vs 20)
- Potential LLPC context window quality improvement (broader causal graph signal)

The K cap in the `LIMIT` clause is **not the same** as the K cap that protects against hub-node latency. The latency protection is provided by the property-read ceiling (~200 neighbors before 10ms is exceeded). The `LIMIT` only reduces the Python-side processing, not the Neo4j traversal cost. Therefore:

**K=20 LIMIT is conservative and can safely be relaxed to K=50 without any latency concern, and to K=100 for nodes with fanout ≤ 200.** The 10ms sub-budget is not the binding constraint at K=20–100 for any realistic graph at Tim's scale.

### Safe K values for Tim's graph trajectory

| Time | Node count | Typical hub fanout | Safe K (10ms budget, cache-warm) | Safe K (10ms budget, cache-cold SATA SSD) |
|------|------------|-------------------|----------------------------------|------------------------------------------|
| Month 6 | ~5K | 50-150 | 50-100 | 20-50 |
| Month 12 | ~10K | 150-400 | 50-100 | 20-40 |
| Month 24 | ~20K | 300-800 | 50 (hub nodes), 100 (median nodes) | 20 (hub nodes), 50 (median nodes) |
| Month 60 | ~50K | 500-2000 | 20-50 | 10-20 |

**Recommended K: 50 as the default, dropping to 20 via runtime circuit breaker if the Neo4j query plan detects high fanout.** The Q078 K=20 recommendation was derived for the worst-case hub-node fanout at Month 24. K=50 is safe for all non-hub nodes at all time horizons, and for hub nodes through Month 12.

## Ideas Extracted

### Idea: epoch-filter-as-fanout-reducer-not-cost-reducer
- **Novelty**: 0.68 — The distinction between "the index reduces traversal cost" (false) and "the epoch filter reduces the downstream candidate set, which reduces ranking cost" (true) is a precise framing that does not appear explicitly in Neo4j documentation or graph database literature on temporal filtering. Most discussions of property filtering in graph traversal assume the filter reduces work; for BFS from a known anchor, it only reduces post-traversal work. The practical implication (the K cap protects against traversal cost, the epoch filter protects against ranking cost) changes the design logic for where to place the cap.
- **Evidence**: 0.72 — Neo4j's execution plan architecture (relationship chain traversal as the primary BFS mechanism, property index as an anchor-only access path) is documented in the Neo4j 5.x query planning documentation and in the Neo4j Graph Algorithms book (Needham & Hodler, 2019). The absence of a "traversal-join-to-index" plan in Neo4j Community Edition is confirmed by the Neo4j query plan reference. The cost model (O(F) property reads, not O(log N)) follows directly from this architecture.
- **Feasibility**: 0.95 — No implementation change required; this is an analytical finding about the existing system. The implication (K limit can be relaxed based on downstream processing budget, not traversal budget alone) is a parameter change.
- **Description**: Reframe the K cap as two separate limits with different rationales: (1) a **traversal K** that limits the number of relationship records read (protects Neo4j traversal latency — ceiling ~200 at 10ms for NVMe cache-warm), and (2) a **ranking K** that limits the candidates passed to downstream reranking (protects Python processing time and context window quality — set to 20-50 depending on available downstream budget). The `LIMIT 20` in the Cypher query conflates these two concerns. The traversal is bounded by fanout (which is determined by the graph structure, not the LIMIT), while the ranking is bounded by the LIMIT. Separating the constraints allows K to be raised from 20 to 50 on the ranking dimension without exceeding the traversal budget.

### Idea: neo4j-epoch-predicate-as-working-set-cache-invalidator
- **Novelty**: 0.71 — Using the epoch-filtered BFS traversal result set (the set of neighbors that survive the `WHERE neighbor.last_access_epoch >= (current_epoch - tau)` filter) as a working-set cache validity signal in Redis is not found in reviewed AI memory system literature. The idea is: if a seed node returns zero epoch-surviving neighbors (all its causal links point to stale memories), this is a signal that the node is a "detached hub" — it was once highly connected to the working set but its neighborhood has aged out. Detached hubs should be deprioritized in MBC Level 2 routing even if they remain individually within the working-set window.
- **Evidence**: 0.55 — Graph neighborhood epoch decay as a routing signal is a novel composition. Individual node epoch tracking (tau-based working-set window) is established in Q062. Graph traversal epoch filtering is established in Q078. The composition — using the size of the epoch-surviving neighborhood as a secondary relevance signal — is novel. The closest adjacent concept is "graph centrality decay" in temporal knowledge graphs (Trivedi et al., DyRep, 2019), but that work uses continuous-time event embeddings rather than discrete epoch-clock working sets.
- **Feasibility**: 0.72 — Requires tracking the fraction of epoch-surviving neighbors per seed node and storing this in Redis as a secondary routing weight. The traversal already computes this (F_total vs F_surviving). Adding `COUNT(*)` before the LIMIT clause to capture F_surviving, and storing this ratio in Redis, adds ~0.5ms and one Redis write per traversal. The routing weight update requires modifying MBC Level 2 candidate scoring, which is a medium-complexity change.
- **Description**: After each epoch-filtered BFS traversal, compute the epoch-survival ratio for the seed node: `epoch_survival_ratio = F_surviving / F_total`. Nodes with low epoch-survival ratios (most of their causal neighborhood is outside the working-set window) are "detached hubs" whose causal graph signal is stale. Store this ratio in Redis with TTL=1 day, and incorporate it as a downweight in MBC Level 2 routing scores: `adjusted_score = base_score × (0.5 + 0.5 × epoch_survival_ratio)`. A node with 100% epoch survival (all neighbors fresh) gets full base score; a detached hub with 10% epoch survival gets 55% of its base score. This converts the epoch filter from a binary neighbor gate into a continuous hub-freshness signal.

## Key Finding

The `CREATE INDEX memory_epoch FOR (n:Memory) ON (n.last_access_epoch)` index does NOT accelerate the epoch-filtered BFS traversal. Neo4j's Cypher execution planner uses the property index only as an anchor access path — when a query starts from a property predicate to find candidate nodes. In the BFS query structure (anchor on a specific node by ID, traverse relationships, filter neighbors by property), the planner traverses the relationship chain and evaluates the epoch predicate by reading each neighbor's property chain as a post-filter. This is O(F) property reads, not O(log N). The index is irrelevant to the traversal cost.

**Consequence 1 — The K=20 cap is a traversal budget cap, not a ranking cap.** At Tim's graph scale (1K–50K nodes, hub fanout 50–2000), the binding constraint is the number of relationship records and property chains read, not the number of results returned. The K=20 LIMIT in the Cypher query conflates traversal cost and ranking cost. K=50 is latency-safe for all non-hub nodes at all time horizons (cache-warm NVMe: ~2.5ms). K=100 is safe through Month 12 for all nodes. The K=20 recommendation from Q078 is overly conservative for the average case; it correctly protects against Month 24+ hub nodes in cold-cache conditions. A runtime-adaptive cap (K=50 default, fallback to K=20 via timeout circuit breaker) is more appropriate than a static K=20.

**Consequence 2 — The dual-write overhead is negligible.** The `MATCH (n:Memory {id: $id}) SET n.last_access_epoch = $epoch` operation costs approximately 0.25–0.40ms at all realistic graph sizes (1K–50K nodes). This is dominated by the Neo4j transaction log write (ACID durability) and is insensitive to graph size within Tim's scale. At 50 writes/day, this operation contributes 12.5–20ms/day of total Neo4j write time — operationally invisible. The write is safe to execute synchronously within the LLPC retrieval path (adds <0.5ms, within the 50ms budget with 100× headroom), or async if a defensive timeout is desired.

**Consequence 3 — The epoch filter's value is post-traversal, not pre-traversal.** Adding the epoch predicate to the Cypher query does not reduce the number of relationship records traversed (Neo4j reads all F chains regardless), but it does reduce the candidate set returned to Python for downstream ranking and context window inclusion. For hub nodes at Month 24 (fanout 500–1,500, epoch-survival ratio 30–50%), the filter reduces the Python-side candidate set from 500–1,500 to 150–750 — still above the K=20 LIMIT, so the LIMIT is the actual gate. The filter's most important function remains semantic correctness (preventing stale memories from entering the context via graph traversal), not performance acceleration.

**Verdict rationale (BREAKTHROUGH):** The question anticipated that the property index might enable an index scan for the BFS neighbor filter, potentially relaxing K from 20 to 50-100. The finding is that no such index acceleration is possible in Neo4j's architecture — the K=20 cap is mandatory for the traversal budget, not optionally relaxable by adding an index. This is a hard architectural boundary of the Neo4j query planner that cannot be engineered away within the Community Edition. However, K can be relaxed at the ranking/LIMIT level (not the traversal level) from 20 to 50, since the traversal budget is not the binding constraint for the average node. The K cap's rationale must be re-specified: protect against traversal cost (requires K_fanout ≤ 200 for 10ms budget) separately from protecting against downstream ranking cost (K_limit = 20–50 is appropriate).
