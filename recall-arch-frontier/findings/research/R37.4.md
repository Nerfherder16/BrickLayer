## 2. Build Order with Rationale

### Component 1 (first): Component A — C+ structural chunking for Write/Edit

**Rationale:**

This is the highest-leverage change available. The current observer.py calls the LLM for every observation (verified in source: `observe.py` line 81 delegates to `observer.py`, which calls `llm.generate()` at line 88 for ALL events). For Write/Edit events, this is Strategy A (full LLM extraction). The cosine dilution finding from Q262 shows that the current blob storage approach drops cosine similarity to 0.35-0.42 for targeted queries — below the Qdrant 0.65 threshold. Chunking is the fix. It does not require any Recall API changes: the hook can send multiple chunks as multiple sequential fire-and-forget POSTs to the existing `/observe/file-change` endpoint. The API already accepts them. The C+ pipeline runs in 0.611ms p95 in Node.js (Q267) — fully within the existing hook budget.

**Minimum hours to implement:** 2-3 hours (pure JavaScript, no infrastructure changes).

**MRR impact delivered by this component alone:** ~88% of baseline (C+ path) vs. current unknown baseline that is presumably below 88% due to dilution failures.

### Component 2 (second): Component B — Entity extraction per chunk

**Rationale:**

Once chunks are produced by A, adding entity extraction is a 30-line addition to the same loop. The 8 regex patterns from Q266 are already fully specified. Entity extraction populates the `entities` field in the metadata contract, which feeds BM25 indexing on the Recall API side. This component cannot meaningfully precede A because extracting entities from a 10,000-token blob produces spurious and uncontextualized entity lists (e.g., an IP address extracted from a file blob is associated with the entire file, not the specific function that configured it). After A chunks the content, the same loop runs entity extraction per chunk, binding entities to their correct semantic scope. Implementation is co-located with A — a natural second pass in the same code.

**Minimum hours to implement:** 1 hour (in the same PR as A, or immediately after).

**MRR impact:** closes another 2-3 points toward the 94% Strategy D target by enabling BM25 exact-match for identifier queries.

### Component 3 (third): Component C — Bash schema detection

**Rationale:**

Component C is independent of A and B and operates on a different branch of the hook (the Bash event path). The current hook already distinguishes between Write/Edit and Bash events — the branching logic is partially present (the hook reads `toolName` from stdin). Component C adds the routing logic: check `cmd_prefix` against STRUCTURED_PREFIXES, route to schema parser or to the async LLM queue decision. This component by itself prevents LLM calls for structured Bash output (docker ps, git status, npm install) — the most frequent Bash events in a development session. Each structured schema parser is 10-30 lines. The top 5 schemas (docker ps, git log, git status, npm/pip install, pytest/cargo test) cover the majority of qualifying Bash events.

**Minimum hours to implement:** 3-4 hours (schema parsers add up, but each is simple).

**MRR impact:** Eliminates unnecessary LLM calls for ~60-70% of Bash events while improving entity extraction quality for those events (deterministic schema parsers extract structured fields, not LLM paraphrases).

### Component 4 (fourth): Component D — Async Bash LLM extraction queue

**Rationale:**

Component D builds on C's routing decision: unstructured Bash output with >200 tokens is queued to a new `/observe/bash-llm-extract` endpoint on the Recall API. The hook fires this as a second fire-and-forget call alongside the initial storage call. The Recall API endpoint queues the LLM extraction and returns 202 immediately. The session budget counter (discussed in section 5) is the primary design question for this component. Component D is last because: (1) it requires C's routing gate, (2) it requires a new Recall API endpoint that does not currently exist (all other components can reuse the existing `/observe/file-change` endpoint), and (3) its MRR contribution is the smallest marginal gain (from 88% C+ to 94% Strategy D, a 6-point lift that only applies to the ~25% of Bash events that are unstructured and >200 tokens).

**Minimum hours to implement:** 4-5 hours (includes the new Recall API endpoint, async LLM call, Redis counter, and graceful fallback on Ollama unavailability).

## 4. Async Bash Queue Implementation Pattern

### The design question

The async Bash LLM queue in Component D requires a 50-call session budget counter. The hook is per-invocation (new Node.js process per PostToolUse event). There is no shared process state between hook invocations. The question is where the counter lives.

### Option analysis

**Option (a): Local session file (~/.recall/session-budget.json)**

A JSON file written and read on each hook invocation. The hook reads the current count, increments, and writes back. This creates a race condition: if two hook invocations run in parallel (which can happen when Claude executes multiple tool calls rapidly), both can read count=3, both increment to 4, and both write back — losing one increment. File locking in Node.js requires the `lockfile` module or a manual lock-file pattern (`session-budget.lock`), adding complexity. On Windows (Tim's machine), file locking is advisory and not guaranteed. Additionally, the session boundary must be detected (when does the file reset?) — requiring either a session ID comparison on every read or a separate reset mechanism tied to the session-start hook. This approach is fragile.

**Option (b): Recall API session state**

The hook already passes `session_id` in the request body. A counter field in the Recall API's session state (PostgreSQL or Redis) could track budget consumption. The hook would need to know the count before deciding whether to send the LLM request — but the hook is fire-and-forget. For the budget check to work, the hook would need to query the API for the current count, then conditionally fire the LLM request. This adds a synchronous round-trip (or a best-effort fire with server-side gating) and requires the hook to await a response before deciding. It converts a fire-and-forget architecture into a request-response architecture, violating the core hook design principle (exit quickly, never block Claude). It also means the counter check requires Recall to be online — counter unavailability would mean no budget enforcement.

**Option (c): Redis (already in the Recall stack on port 6380)**

The Recall API already uses Redis for session state, Markov chain transitions, and pre-fetch cache. Adding a budget counter is a `INCR recall:bash-budget:{session_id}` operation on the API side, not the hook side. The correct implementation is: the hook fires the `/observe/bash-llm-extract` endpoint (fire-and-forget), and the Recall API endpoint checks the Redis counter before queuing the LLM call. If `INCR recall:bash-budget:{session_id}` returns a value > 50, the endpoint skips the LLM call and stores with `description: null`. The hook never needs to know the current count — the server enforces the budget. The key expires automatically when the session ends (set TTL = 8 hours on first INCR). This is the fire-and-forget pattern preserved end-to-end.

### Recommendation: Option (c) — Redis, server-side enforcement

**The correct pattern is:**

```
Hook (fire-and-forget):
  POST /observe/bash-llm-extract
    { session_id, command, output, ... }

Recall API endpoint (/observe/bash-llm-extract):
  count = INCR recall:bash-budget:{session_id}
  if count == 1: EXPIRE recall:bash-budget:{session_id} 28800  // 8h TTL
  if count > 50:
    store memory with description=null (C+ path)
    return 200 {"status": "budget_exceeded", "description": null}
  else:
    enqueue LLM extraction task (background)
    return 202 {"status": "queued"}
```

This design:
- Preserves fire-and-forget (hook exits immediately after POST)
- Makes budget enforcement atomic (Redis INCR is atomic — no race conditions)
- Requires no changes to the hook beyond firing the new endpoint URL
- Gracefully degrades to C+ when budget is exceeded
- Session boundary is implicit: session_id changes per session; old keys expire after 8h

## 5. Session Budget Persistence: Final Recommendation

**Answer: Option (c) — Redis with server-side INCR enforcement.**

Detailed rationale from the three-option comparison above. The key constraint is the per-invocation spawn model: the hook has no shared in-process state and cannot reliably use filesystem locking on Windows. The budget counter must live in a service that (a) is already running in the Recall stack, (b) provides atomic increment, and (c) supports TTL-based expiry for automatic session boundary detection.

Redis satisfies all three. It is already deployed at 192.168.50.19:6380 (same host as the Recall API). The INCR command is O(1) and atomic. TTL handles session expiry without any reset hook.

The budget enforcement lives entirely on the Recall API server side. The hook implementation for Component D is minimal: one additional fire-and-forget fetch call to `/observe/bash-llm-extract` for qualifying Bash events. The hook does not check, increment, or persist the counter.

## 6. Critical Path Summary

**What must be done first, before anything else:**

**Step 1: Implement Component A (C+ chunking) in observe-edit.js.**

This is the gate. Until Write/Edit events are chunked, all observations stored by the hook are blobs. The cosine dilution failure identified in Q262 (cosine 0.35-0.42 vs. 0.65 threshold) means targeted queries currently fail to retrieve the correct memory even when the content is in the corpus. Every hour of operation under the current blob model accumulates diluted memories that the eventual C+ implementation cannot retroactively fix (existing blobs stay as blobs; new writes get chunks). Starting today minimizes this accumulation.

**Step 2: Add entity extraction (Component B) to the same PR.**

Co-location with A is the natural implementation path. The chunk loop is already iterating over chunks; adding entity regex extraction per chunk is 20-30 additional lines in the same loop. Separating this into a later commit saves no effort and delays BM25 indexing.

**Step 3: Implement Bash schema detection (Component C) as a second PR.**

After A+B are shipped and validated, C can be built independently. It requires no Recall API changes and no Redis changes. It is pure hook-side routing logic.

**Step 4: Implement async Bash LLM queue (Component D) as a third PR.**

This requires a new Recall API endpoint (`/observe/bash-llm-extract`), Redis counter logic, and LLM queue configuration. It is the only component that touches the Recall API. It is correctly scoped as its own PR.

## Grounding Notes

The following facts from source analysis are load-bearing for this specification:

1. **observe-edit.js currently sends a single blob** (lines 106-118). The `MAX_CONTENT_SIZE = 10000` guard means files over 10,000 chars are truncated, not chunked. This is the primary dilution failure mode.

2. **The `/observe/file-change` endpoint already accepts `content` as a string field** (observe.py line 25, `FileChangeObservation.content`). Multiple per-chunk POSTs to this endpoint require no API changes.

3. **observer.py currently calls LLM for ALL observations** (line 84: `llm = await get_llm()` followed by `response = await llm.generate(...)`). After Commit 1, the LLM is called once per chunk — which is actually worse (more LLM calls per Write event). The full Strategy D benefit for observer.py (replacing LLM with C+ entity extraction) requires modifying observer.py to detect chunk-format observations and skip the LLM call. This is a Phase 2 change to the server side. Commit 1 (hook-only) improves storage granularity but temporarily increases LLM call volume.

4. **The setTimeout floor is already 25ms** (line 147 of current hook, confirmed by Q270 finding that this was already optimized). The 25ms budget is sufficient for multiple fire-and-forget fetches since they all dispatch concurrently before the timeout fires.

5. **Session ID is already passed in the hook** (line 120: `const sessionId = parsed.session_id || ""`). The async Bash LLM queue endpoint can receive `session_id` directly from the existing body structure.
