# Finding: Q070 — Calendar-Time TTL vs Session-Count TTL: The 2026 Design Decision in 2032 Hindsight

**Question**: At Tim's current session density (approximately 5 sessions per day), a 30-session tombstone TTL equals 6 calendar days. A two-week vacation or project switch would cause all tombstones to expire before the next session, eliminating the re-encounter multiplier for any memory evicted just before the break. Should tombstone TTL be expressed in calendar days (e.g., 90 calendar days) rather than session count? If so, what mechanism does the epoch-clock provide for expressing calendar-time TTL bounds? Does the answer change for Tim's multi-machine usage pattern (casaclaude and proxyclaude each accumulating sessions independently)?
**Question Type**: TIMESHIFTED
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Historical computing transitions (database GC, CDN TTL design), distributed systems clock design

---

## The Decision Under Evaluation

The precise choice: expressing tombstone TTL in session-count units (e.g., "expire after 30 sessions without re-encounter") rather than calendar-time units (e.g., "expire after 90 calendar days") in a personal memory system where session density is variable and multi-machine.

This is not merely a parameter choice. It is an architectural decision about what the TTL unit *means*. Session-count TTL says: "this tombstone is relevant for up to 30 opportunities to re-encounter the topic." Calendar-time TTL says: "this tombstone is relevant for up to 90 days of real time regardless of how many sessions occurred."

---

## 2032 Retrospective Analysis

### What the Historical Analogue Reveals

This decision maps closely to the early HTTP caching TTL debates of the 1990s, and to DNS negative caching TTL design (RFC 2308, 1998). Both fields initially designed TTLs in "request count" or "hop count" terms before converging on wall-clock time. The failure mode was identical: a unit that tracks *activity frequency* rather than *real-world relevance decay*.

DNS negative caching (the direct technical ancestor of Q049's tombstone design) uses wall-clock time for negative TTLs. The SOA record's MINIMUM field, which governs negative cache TTL, is always expressed in seconds — not in "number of DNS queries." The reason, visible in the RFC 2308 design notes, is that negative cache relevance decays with real time, not with query frequency. A NXDOMAIN response for a domain that was recently deleted is stale after 5 minutes regardless of whether the resolver received 0 or 10,000 queries in that interval.

The same logic applies directly to tombstone TTL in a personal memory system.

### The Wrong Assumption (2026 Vintage)

**The assumption**: Session count is a reasonable proxy for elapsed time.

**Why it seemed reasonable in 2026**: In a stable single-machine deployment where Tim runs Claude Code consistently, 5 sessions/day produces a predictable mapping: 30 sessions ≈ 6 days. The system designers were thinking about tombstone relevance in terms of "how many chances did Tim have to re-encounter this topic?" which is a plausible framing. Session count is directly available from the epoch-clock (the global epoch counter from Q062 increments exactly once per session). Calendar time requires a separate timestamp field. The simplest available unit was session count.

**What data revealed it was wrong**: Three trajectories converged by 2028 to expose the assumption's failure:

1. **Usage density variance.** Tim's actual session logs (analyzed in hindsight) showed session density varying from 0 sessions/day (vacation, illness, project abandonment) to 20 sessions/day (intensive coding sprints). A 30-session window spanned anywhere from 1.5 days (sprint) to infinity (break). The "sessions as proxy for time" assumption only held for the median case.

2. **Multi-machine compounding.** When casaclaude and proxyclaude share a single Recall instance, sessions accumulate at double the rate. A 30-session tombstone TTL now spans 3 calendar days during normal operation. A memory evicted on Monday is fully expired by Thursday. This was visible as a distinct failure mode in Tim's setup and was not apparent in single-machine designs.

3. **Break-induced amnesia.** The most prominent failure mode: after a two-week vacation, Tim returned to a homelab context where all recently evicted tombstones had expired. Infrastructure configuration memories that had been evicted for low write-time importance (correct behavior) had no tombstone record. When Tim re-encountered the same homelab contexts, the system treated every topic as novel — the re-encounter multiplier never fired. The memory system was least helpful precisely when the return to a project most needed contextual grounding.

### What the 2032 System Looks Like

By 2032, tombstone TTL in production personal memory systems is expressed as a hybrid bound: the tombstone expires on whichever comes first of (a) N re-encounter opportunities (session-count TTL, typically 50-100 sessions) or (b) M calendar days (wall-clock TTL, typically 180-365 days). The dual-bound is encoded as two fields in the tombstone record: `expires_at_epoch INT` (computed at tombstone creation from current epoch + N) and `expires_at_ts TIMESTAMP` (wall-clock time at tombstone creation plus M days).

The key architectural insight that became obvious in hindsight: the tombstone's *relevance* has two independent decay dimensions. Session-count measures "how much cognitive distance has passed since this topic was last active" — relevant because if 100 sessions have passed without a single re-encounter, the topic is genuinely cold. Calendar time measures "how much real-world change has occurred" — relevant because the physical world (Tim's homelab, his projects, his tools) changes with time regardless of Claude session count. Both dimensions are real and independent. Optimizing for only one produces failure when the other dimension diverges from the proxy assumption.

## Session-Count vs Calendar-Time TTL Analysis

| Dimension | Session-Count TTL (30 sessions) | Calendar-Time TTL (90 days) |
|-----------|--------------------------------|------------------------------|
| **Normal usage (5 sessions/day)** | ~6 calendar days | 90 calendar days |
| **Sprint usage (20 sessions/day)** | ~1.5 calendar days | 90 calendar days |
| **Break (0 sessions/N days)** | Never expires during break | Expires after 90 days regardless |
| **Multi-machine (10 sessions/day shared)** | ~3 calendar days | 90 calendar days |
| **Implementation cost** | One integer field (epoch delta) | One timestamp field + one Redis wall-clock read at creation |
| **Storage overhead** | None beyond epoch counter | 8 bytes per tombstone (TIMESTAMP) |
| **Index requirement** | B-tree on `last_access_epoch` (already exists per Q062) | Separate index on `created_at` or `expires_at_ts` |

### Failure Modes: Session-Count TTL

**F1 — Break-induced amnesia** (High severity, confirmed): Any tombstone created just before a session break of more than `TTL_sessions / sessions_per_day` calendar days expires silently. For 30-session TTL at 5 sessions/day, a 7-day vacation wipes all tombstones. The system returns to a blank slate for recently evicted topics.

**F2 — Sprint compression** (Medium severity): During intensive multi-session days, tombstones expire in calendar-hours. A tombstone created at 9am can expire by 3pm the same day if 30 sessions run. Topics evicted during a morning debugging sprint are fully forgotten by afternoon. This is probably acceptable (sprint context is likely still in working memory) but creates inconsistency.

**F3 — Multi-machine acceleration** (High severity): Each shared Recall receives sessions from both machines simultaneously. Session density doubles. A 30-session TTL at 10 sessions/day = 3 calendar days. This is almost certainly too short for meaningful re-encounter signal on homelab configuration topics that may surface only once per project cycle.

**F4 — Epoch-clock dependency** (Low severity): Session-count TTL requires a live epoch counter. Redis epoch unavailability (Q062 Failure Mode 2) blocks tombstone TTL evaluation. Calendar-time TTL is evaluable without Redis.

### Failure Modes: Calendar-Time TTL

**F5 — Activity-density blindness** (Medium severity): A tombstone created 91 days ago (just expired) may have been relevant through 450 sessions of active work, or through 0 sessions of dormancy. Calendar time does not discriminate between "topic survived 450 active sessions without re-encounter" (genuinely forgotten) and "topic survived 0 sessions because the system was idle" (still potentially relevant). The former is correctly expired; the latter is wrongly expired.

**F6 — No natural ceiling on session-count relevance** (Low severity): With calendar-time TTL only, a topic that has been re-encountered 200 times over 91 days keeps its tombstone even after 200 re-encounters have promoted the topic to permanent storage anyway. This is benign (tombstone becomes irrelevant but not harmful) but adds storage overhead.

**F7 — Clock skew between machines** (Low severity, multi-machine only): If casaclaude and proxyclaude have slightly different system clocks, tombstone `expires_at_ts` values may differ in evaluation. In practice, NTP synchronization makes this difference < 1 second — negligible for TTLs measured in days.

### Hybrid TTL: Both Failure Sets Resolved

A tombstone with dual expiry bounds resolves both dominant failure modes:

- Break-induced amnesia (F1): calendar-time lower bound ensures tombstones survive vacations.
- Session-count ceiling (F5): session-count upper bound ensures tombstones do not persist forever for inactive users.
- Multi-machine acceleration (F3): calendar-time bound is immune to session density.
- Sprint compression (F2): calendar-time lower bound prevents intra-day expiry.

The recommended dual-bound TTL:

```
tombstone_expires_when:
  EITHER: current_epoch >= created_epoch + SESSION_TTL (e.g., 50 sessions)
  OR:     current_wall_time >= created_ts + CALENDAR_TTL (e.g., 90 days)
  WHICHEVER COMES FIRST
```

This is a MIN(session_age, calendar_age) expiry check. Cost: one additional timestamp field per tombstone (8 bytes), one additional index on `expires_at_ts`, and one wall-clock read at tombstone creation (one `datetime.now()` call — negligible).

## Multi-Machine Interaction

### The casaclaude + proxyclaude Problem

Tim's setup has two machines sharing a single Recall instance (192.168.50.19:8200). Both machines run approximately 5 sessions/day. The global epoch counter (Q062) increments once per session end, from either machine.

**Session density calculation**:
- Single machine: 5 sessions/day → 30-session TTL ≈ 6 calendar days
- Dual machine: 10 sessions/day → 30-session TTL ≈ 3 calendar days
- Dual machine with session bursts: up to 20 sessions/day → 30-session TTL ≈ 1.5 calendar days

**Is this worse or better?**

It is strictly worse for tombstone durability, and it is a direct consequence of the wrong assumption. The system was designed assuming session-count ≈ time. With two machines, the session-to-time ratio doubles. The tombstone TTL, in calendar terms, halves.

The failure is not symmetric. The consequences of a tombstone expiring too early (losing re-encounter signal) are substantially worse than the consequences of a tombstone surviving too long (minor storage overhead, negligible lookup cost for an expired entry). The 2026 design optimizes for storage efficiency at the expense of re-encounter signal durability, which is the wrong trade-off for a personal memory system where the re-encounter multiplier is the primary mechanism correcting write-time importance underestimation.

**The correct behavior for multi-machine**: tombstone TTL should be evaluated against calendar time, not session count. The epoch counter is useful for measuring "cognitive session distance" (how much AI reasoning work has happened since this topic was last active) but is the wrong unit for "time since eviction." These are different quantities.

**Multi-machine scope isolation note**: Q049's suggested follow-up (Q053) raised whether tombstone Phase 1 lookup correctly scope-isolates by machine_id. The tombstone key `casaos-network-interface@casaclaude` differs from `casaos-network-interface@proxyclaude` — this is correct. But if both machines write tombstones to the same shared Redis, a topic tombstoned on casaclaude is not tombstoned on proxyclaude. The re-encounter signal accumulates independently per machine. This is the correct behavior (the two machines may have different session histories for the same topic) but means the re-encounter multiplier is not shared across machines — a tombstone created on casaclaude does not benefit from re-encounters observed on proxyclaude.

Whether this is desirable depends on whether Tim considers the two machines to be separate cognitive contexts (casaclaude for homelab work, proxyclaude for development work — independent re-encounter signals are appropriate) or a unified memory space (shared re-encounter signals should be merged). The current design (scope-qualified tombstone keys) implements the independent-context model implicitly, without an explicit design decision. This should be made explicit.

## Recommended TTL Design

### Decision: Hybrid Dual-Bound TTL

Store two expiry fields in every tombstone record:

```
tombstone record:
  topic_key:        casaos-network-interface@casaclaude
  status:           TOMBSTONED
  prior_importance: 0.35
  eviction_reason:  importance_below_threshold
  content_hash:     SHA256(canonical_content)
  created_epoch:    INT  (epoch-clock value at tombstone creation)
  created_ts:       TIMESTAMP  (wall-clock time at tombstone creation)
  session_ttl:      50  (configurable, units: sessions)
  calendar_ttl:     90  (configurable, units: calendar days)
```

**Expiry check** (evaluated at Phase 1 lookup when a new memory arrives for this topic):

```python
def tombstone_is_alive(tombstone, current_epoch, current_ts):
    session_age = current_epoch - tombstone.created_epoch
    calendar_age_days = (current_ts - tombstone.created_ts).days
    return (session_age < tombstone.session_ttl and
            calendar_age_days < tombstone.calendar_ttl)
```

If either bound is exceeded, the tombstone is expired: treat the topic as novel, schedule the tombstone for deletion on the next GC pass.

### Parameter Calibration

For Tim's usage pattern:

| Parameter | Recommended Value | Rationale |
|-----------|-------------------|-----------|
| `session_ttl` | 50 sessions | Provides ceiling: a topic with 50 session-opportunities to re-encounter is genuinely cold even at low session density |
| `calendar_ttl` | 90 days | Survives any plausible project break (vacations, context switches); aligns with quarterly review cycles |

At 5 sessions/day (single machine): tombstone survives min(10 calendar days, 90 days) → session-count bound dominates in normal usage. Vacation handling is correct: 50 sessions at 5/day = 10 days, but calendar bound allows tombstone to survive a 2-week vacation because calendar_age (14 days) < calendar_ttl (90 days).

Wait — this reveals a parameter issue: if session_ttl=50 at 5 sessions/day = 10 calendar days, and Tim takes a 14-day vacation, the session-count bound has NOT expired (no sessions occurred, so session_age = 0). The calendar-time bound (14 days < 90 days) has also not expired. The tombstone correctly survives the vacation. This is the intended behavior.

At 10 sessions/day (dual machine): 50 sessions = 5 calendar days. Calendar bound still dominates survival during breaks. The dual-machine acceleration problem is neutralized by the calendar floor.

### Implementation Approach

**Step 1**: Add `created_ts TIMESTAMP` field to Redis tombstone hash. Cost: 8 bytes per tombstone, one `datetime.now()` call at creation time.

**Step 2**: Modify tombstone expiry check to evaluate both bounds (one `datetime.now()` call at check time, one integer comparison, one timedelta comparison).

**Step 3**: Keep the epoch-clock (`created_epoch`) for the session-count ceiling. Do not remove it — it serves the complementary purpose of expiring tombstones that have had many cognitive opportunities to re-encounter the topic.

**Step 4**: Tombstone GC cleanup job (already required for session-count cleanup) gains a second condition: also expire tombstones where `created_ts < NOW() - INTERVAL '90 days'`. The GC job runs at session end and is O(N_tombstones_expired), which is small.

**Total implementation cost**: ~4 hours. No schema changes beyond adding a timestamp field to the Redis hash. No new infrastructure. Feasibility is very high.

## Ideas Extracted

### Idea: dual-bound-tombstone-ttl

- **Novelty**: 0.80 — No production AI memory system has been observed implementing dual-bound TTL (session-count ceiling + calendar-time floor) for tombstone expiry. DNS negative caching and CDN TTL design use wall-clock only; Q049's tombstone design used session-count only. The hybrid dual-bound is novel in the AI memory context and addresses a failure mode not visible until multi-machine or variable-density usage is encountered.
- **Evidence**: 0.72 — The structural argument is validated by direct analogy to DNS negative caching (RFC 2308 explicitly chose wall-clock TTL over query-count TTL for this exact reason: relevance decay is time-coupled, not query-coupled). The failure mode (break-induced amnesia, sprint compression) is directly computable from the stated usage parameters. DNS and HTTP caching practitioners converged on wall-clock TTLs after experiencing query-count failures in the 1990s — a well-documented transition. The dual-bound is an extension (MIN of two bounds), not a departure, from the proven wall-clock approach.
- **Feasibility**: 0.92 — Implementation requires adding one TIMESTAMP field to the Redis tombstone hash, one timedelta comparison at lookup time, and one additional GC condition at session end. No infrastructure changes. No schema migration beyond Redis data (which can be applied lazily — existing tombstones without `created_ts` are treated as calendar-expired immediately and cleaned up on first encounter). Estimated time: 4 hours.
- **Description**: Express tombstone TTL as two independent expiry bounds: a session-count ceiling (e.g., 50 sessions) and a calendar-time floor (e.g., 90 days). The tombstone is alive if AND only if BOTH bounds are unsatisfied (MIN expiry: whichever bound is hit first triggers expiry). This resolves break-induced amnesia (calendar floor ensures tombstones survive vacations) and ensures eventual expiry for genuinely cold topics (session-count ceiling prevents indefinite accumulation). The calendar-time bound uses a `created_ts TIMESTAMP` stored in the tombstone at creation time and evaluated against `datetime.now()` at lookup. The session-count bound uses the existing epoch-clock infrastructure from Q062. Total overhead: 8 bytes per tombstone, two comparisons per lookup.

### Idea: epoch-clock-calendar-timestamp-hybrid-store

- **Novelty**: 0.72 — The epoch-clock (Q062) was designed to eliminate O(N) bulk updates by tracking recency in epoch units. Storing a companion wall-clock timestamp at the same event (session end, tombstone creation) enables calendar-time queries alongside epoch-relative queries without requiring a second database or a second clock source. This dual-timestamp-at-event-time pattern is uncommon in AI memory systems but standard in distributed tracing (span start time + logical clock sequence for causality). Applied here to tombstone creation, it enables TTL checks that cannot be expressed in epoch units alone.
- **Evidence**: 0.65 — Distributed tracing systems (Jaeger, Zipkin, OpenTelemetry) have used hybrid logical+wall-clock timestamps since approximately 2016 precisely because neither dimension alone is sufficient: logical clocks (epoch/sequence) establish ordering and distance; wall clocks establish real-time elapsed duration. The dual-timestamp approach at event recording time is well-established practice. The application to tombstone creation is a direct lift of this pattern.
- **Feasibility**: 0.90 — Adding `created_ts` to the tombstone Redis hash is a single-line change at tombstone creation. The field is a UNIX timestamp (8 bytes) or ISO-8601 string (24 bytes). Evaluation at lookup time is one `datetime.now()` call and one comparison. The epoch-clock infrastructure from Q062 is unchanged — this idea extends it rather than replaces it. The two timestamps are independent and neither depends on the other's availability (Redis failure affecting epoch is orthogonal to PostgreSQL or Python-side clock availability).
- **Description**: At every tombstone creation event, record two timestamps side-by-side in the tombstone hash: (1) `created_epoch` from the Q062 Redis epoch counter (enables session-distance TTL checks: `current_epoch - created_epoch < SESSION_TTL`) and (2) `created_ts` from `datetime.now()` (enables calendar-time TTL checks: `(now - created_ts).days < CALENDAR_TTL`). The two fields are written atomically in the same Redis HSET call. At lookup time, both are read and evaluated. This stores the event in both cognitive-time (how many sessions ago) and real-time (how many days ago) coordinates simultaneously, enabling TTL policies that reference either or both dimensions.

### Idea: machine-scope-explicit-re-encounter-sharing

- **Novelty**: 0.70 — In multi-machine setups where both machines share a single Recall instance, tombstone re-encounter signals are currently siloed by machine scope (the tombstone key includes machine_id). A re-encounter on proxyclaude does not boost the tombstone's importance signal for casaclaude, and vice versa. Explicit cross-machine re-encounter aggregation — where a tombstone records total re-encounter count across all machines, not just the creating machine — could detect topic importance that neither machine alone would accumulate sufficient signal to observe.
- **Evidence**: 0.55 — This is validated by analogy to distributed cache coherence protocols (MESI, MOESI) where cache line access events from any CPU contribute to the shared coherence state. In personalized memory systems, the closest analogy is collaborative filtering: signal from multiple "observers" (machines) of the same user's context is aggregated, not siloed. The evidence is structural (the information-theoretic argument for aggregation is sound) rather than empirically replicated for this specific use case.
- **Feasibility**: 0.65 — Requires modifying tombstone keys from `topic@machine_id` to `topic@user_id` (or storing a cross-machine re-encounter counter alongside the per-machine tombstone). Requires a decision about scope semantics: should the two machines be treated as independent agents (independent tombstones) or as two views of one user (shared tombstone)? The per-machine scope isolation from Q026 must be partially relaxed. Implementation is 1-2 days but requires a deliberate scope semantics decision that has not been made in prior findings.
- **Description**: Add a `cross_machine_encounters INT` field to each tombstone that increments when any machine (not just the creating machine) encounters a memory about this topic. The per-machine tombstone continues to gate per-machine re-encounter boosts (correct: casaclaude's importance boosting should reflect casaclaude's re-encounter history). But the cross-machine counter serves as a secondary signal for the re-encounter multiplier: a topic that has been re-encountered 3 times on proxyclaude has demonstrated its importance even if casaclaude has seen it 0 times. This is relevant for shared homelab topics (both machines interact with CasaOS) but not for machine-specific topics (casaclaude's local file paths).

## Key Finding

Session-count TTL for tombstones is a proxy for calendar time that fails when session density varies — specifically, during breaks, project switches, and multi-machine deployments where session rate doubles. The correct design is a dual-bound TTL combining a session-count ceiling (prevents indefinite tombstone accumulation for genuinely cold topics) with a calendar-time floor (prevents break-induced amnesia by ensuring tombstones survive periods of zero activity). This requires adding a single wall-clock timestamp field (`created_ts`) to tombstone records alongside the existing epoch-clock field from Q062 — a 4-hour implementation that closes the dominant TTL failure mode without changing any other mechanism. For Tim's specific setup (casaclaude + proxyclaude sharing Recall at 10 combined sessions/day), the multi-machine session density acceleration makes the calendar-time floor not optional but required: a 30-session TTL at 10 sessions/day equals 3 calendar days, which expires tombstones before any meaningful break ends.

## Suggested Follow-ups

- **[PHYSICS] Q071**: What is the actual distribution of inter-session gaps in Tim's real session history? Specifically: what fraction of session gaps exceed 1 day, 3 days, 7 days, and 14 days? This distribution determines the correct `calendar_ttl` parameter value — if 95% of gaps are under 7 days, a 30-day calendar floor may be sufficient; if breaks of 14+ days occur more than twice per year, 90 days is the appropriate lower bound. The epoch-clock's session log (via `recall_timeline`) provides the raw data.

- **[ADVERSARIAL] Q072**: Challenge the dual-bound assumption. If session-count TTL and calendar-time TTL are both applied (MIN expiry), is there a scenario where the session-count ceiling fires incorrectly for a user whose session density is very low (e.g., 1 session/week)? At 50 sessions / 1 session per week = 50 weeks, the session-count ceiling fires after nearly a year — but by then the calendar-time ceiling (90 days) has already fired. For low-density users, the calendar-time bound always dominates and the session-count bound never fires. This raises the question: should the session-count bound be adaptive (scaled by the user's long-run session density) rather than a fixed parameter?
