# Finding: Q036 — Cross-Instance Write Deduplication via Producer Idempotency Keys

**Question**: What mechanism does a streaming database (Kafka producer idempotency, Flink exactly-once checkpoints) use to deduplicate writes across multiple concurrent producers, and how would it apply to Recall's multi-machine write conflict problem?
**Agent**: quantitative-analyst
**Question Type**: ADJACENT
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Streaming database systems — Apache Kafka KIP-98 idempotent producer protocol, Apache Flink TwoPhaseCommitSinkFunction, Stripe idempotency key pattern

---

## Evidence

### Step 1: The source problem

Kafka, Flink, and payment APIs each face the same fundamental problem: a write operation may be issued more than once by a client (due to retry, crash, or duplicate triggering), and the server must accept exactly one copy. The problem is complicated when multiple distinct clients (producers, machines, instances) can each independently trigger the same logical write.

This is Recall's exact problem: casaclaude and proxyclaude both observe a file edit, both call `/memories/store`, and both produce a semantically-identical memory within a 2–10 second window.

---

### Mechanism 1: Kafka Producer Idempotency (KIP-98)

**How it works in Kafka:**

When `enable.idempotence=true`, the Kafka broker assigns each producer a unique **Producer ID (PID)** during the `InitPidRequest` handshake. The producer also increments a monotonic **sequence number** per (PID, TopicPartition) pair for each message batch.

The broker maintains in memory the mapping:

```
(PID, topic, partition) → (epoch, sequence_highwater)
```

Acceptance rule: the broker accepts a message batch if and only if:
```
incoming_sequence == sequence_highwater + 1
AND
incoming_epoch >= stored_epoch
```

If `incoming_sequence <= sequence_highwater`, the batch is a duplicate — it is silently dropped. If `incoming_sequence > sequence_highwater + 1`, data loss is assumed and an `OutOfOrderSequenceException` is thrown.

**Epoch fencing (zombie producer prevention):**

When a producer with the same `TransactionalId` reconnects (e.g., after a crash and restart), the transaction coordinator **bumps the epoch** and returns the same PID with epoch+1. This fences all prior instances: any message from the old producer epoch is rejected with `ProducerFencedException`. The new producer instance takes over cleanly.

**Minimum state the broker must maintain:**

Per active producer per partition: one integer (sequence_highwater) + one integer (epoch). Approximately 16 bytes per (PID, partition) pair. This state is persisted to the replicated log so a new leader after failover continues deduplication without gaps.

**Deduplication window:**

The sequence number is strictly monotonic and persisted. There is no fixed time window — deduplication is permanent for the lifetime of the PID (default lease: 24 hours of inactivity before the PID can be reused). Within a PID's lifetime, every duplicate is detected regardless of timing.

**What Kafka does NOT solve:**

Kafka's idempotency is per-producer, not per-content. If two different producers (different PIDs) send identical payloads, Kafka writes both — it has no concept of content identity. This is a critical distinction: Kafka deduplicates *retries from the same logical producer*, not *independent concurrent producers writing the same data*.

This is precisely the gap: casaclaude and proxyclaude are two different PID producers. Kafka-style sequence numbers alone would not deduplicate them.

### Mechanism 2: Flink Exactly-Once Two-Phase Commit

**How it works:**

Flink injects **checkpoint barriers** into the data stream. When a barrier reaches a sink operator, the sink must:

1. **Pre-commit phase**: Flush all buffered writes to an external transaction (e.g., open a Kafka transaction). Checkpoint the transaction handle to Flink state backend.
2. **Commit phase**: After the JobManager confirms all operators have completed pre-commit, all sinks commit their external transactions simultaneously.

Failure handling: if a sink crashes between pre-commit and commit, Flink restores the sink's state to the pre-committed transaction handle and retries the commit. Commits must therefore be idempotent — committing the same transaction twice must be safe.

**What this mechanism actually solves:**

Flink 2PC prevents duplicate records from a single logical stream being written twice due to exactly-once replay after failure. It does NOT address two independent parallel Flink operators writing the same logical content — if two source operators both read and process an identical event, Flink will write it twice without additional content-awareness.

**Mapping to Recall:** Flink 2PC is not directly applicable. It solves single-stream replay duplication, not multi-instance concurrent write duplication. Recall's problem is closer to the two-independent-producer problem that Flink also does not solve.

### Mechanism 3: Stripe / AWS Idempotency Key Pattern

**How it works:**

The client generates a **content-derived idempotency key** before issuing a write. This key is sent as a header (`Idempotency-Key: <uuid>` or `Idempotency-Key: <hash_of_content>`). The server:

1. On first receipt: acquires a database lock on the key (using `SERIALIZABLE` isolation or `SELECT FOR UPDATE`). Executes the write. Stores `(key, request_params_hash, response_code, response_body)` in an `idempotency_keys` table.
2. On repeat receipt: finds the existing record, compares `request_params_hash` (rejects if params differ — different operation using same key is an error), and returns the stored response. Does not re-execute the write.
3. Concurrent duplicate receipt: if two requests arrive simultaneously with the same key, one acquires the lock and executes; the other waits and then reads the stored response.

**Minimum backend state:**

```sql
CREATE TABLE idempotency_keys (
    idempotency_key  TEXT        PRIMARY KEY,
    created_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
    locked_at        TIMESTAMPTZ,           -- NULL after completion
    request_hash     TEXT        NOT NULL,  -- SHA-256 of (endpoint + body)
    response_code    INT,
    response_body    JSONB,
    ttl              INTERVAL    DEFAULT '72 hours'
);
```

Row size: approximately 200–400 bytes per unique write operation. TTL cleanup removes rows after 72 hours. In-flight requests set `locked_at` to current timestamp; any concurrent duplicate request that arrives while `locked_at` is set receives a `409 Conflict` response and should retry after the lock expires.

**The critical insight:** The idempotency key can be content-derived. If casaclaude and proxyclaude both generate their idempotency key as `SHA-256(endpoint + normalized_memory_content)`, they will produce the **same key** for the same logical memory. The server therefore sees two requests with identical keys — the second is a duplicate and is dropped regardless of which machine sent it.

This is the mechanism that Kafka does not provide: content-addressed idempotency.

### Mechanism 4: CRDTs — What They Solve vs. What Recall Needs

CRDTs (Conflict-free Replicated Data Types) solve a different problem: *convergence under concurrent writes*. A CRDT guarantees that if two replicas diverge due to concurrent writes, they will eventually merge to an identical state. The merge function must be commutative, associative, and idempotent.

For state-based CRDTs (CvRDTs), the merge function's idempotence property incidentally handles duplicate writes: merging the same update twice produces the same result as merging once.

**However:**

CRDTs do not prevent duplicate writes — they tolerate them. The store still receives two copies of the memory; the CRDT merge simply collapses them without error. For Recall's use case (vector database + graph), a CRDT does not prevent Qdrant from receiving two insert operations, which would create two distinct records at slightly different embedding positions (deterministic embedding models produce identical vectors, but the Qdrant record IDs would differ).

**Verdict on CRDTs:** Wrong tool for this problem. CRDTs solve eventual consistency in distributed replicas. Recall's problem is preventing duplicate *insertions* into a single-master store, which requires deduplication at write time, not convergence semantics.

The Q013 Bloom filter operates within a single machine's memory and does not coordinate across machines. No reviewed system implements content-derived idempotency keys that would coordinate across independent agent instances.

**Why this gap exists:**

All reviewed AI memory systems assume a single logical writer (one agent, one session). The multi-instance pattern (two Claude Code instances on different hardware writing to the same memory backend) is rare enough that it has not been treated as a design constraint. The missing insight is that the fix costs almost nothing: a 30-line middleware function that computes a content hash and checks it before insert.

## Mechanism Transfer: Content-Derived Idempotency Keys for Recall

### The minimum mechanism

The Stripe idempotency key pattern, adapted for content deduplication across multiple Recall writers:

**Key generation (client-side, in the hook):**

```
idempotency_key = SHA-256(
    "recall_memory_store" +
    normalize(content)         # lowercase, strip whitespace
    + agent_session_id         # optional: scope by agent if needed
)
```

The `agent_session_id` is intentionally OMITTED if the goal is cross-machine deduplication of identical content. With only `content` in the hash, casaclaude and proxyclaude generate the same key for the same memory.

**Server-side (Recall FastAPI layer):**

```
POST /memories/store
Headers: X-Idempotency-Key: <sha256_hex>

1. SELECT * FROM idempotency_keys WHERE key = $1 FOR UPDATE
2. If found AND locked_at IS NULL (completed):
      → return cached 200 response, skip insert
3. If found AND locked_at IS NOT NULL (in-flight):
      → return 409, client retries after lock_ttl (5 seconds)
4. If not found:
      → INSERT (key, now(), request_hash, NULL, NULL)
      → Execute memory write (Qdrant + Neo4j + PostgreSQL)
      → UPDATE row: locked_at=NULL, response_code=200, response_body=...
      → Commit
```

**State required:** One PostgreSQL table row per unique memory written. At 300 bytes/row and 1,000 writes/day, the table holds 72-hour history at 22 MB — trivially small. A nightly reaper deletes rows older than 72 hours.

**Deduplication window:** 72 hours. Any duplicate from casaclaude or proxyclaude within 72 hours of the first write is dropped. This covers the entire race window (seconds to hours).

**Failure mode:** If the Recall server crashes between the INSERT and the UPDATE (steps 4a and 4c), the row remains with `locked_at` set. On restart, a background job clears locks older than 30 seconds and re-tries or marks them failed. This is identical to Stripe's lock recovery strategy.

### What this is NOT

This is not Kafka's PID+sequence mechanism (that mechanism is per-producer monotonic ordering, not content-addressed). It is not Flink 2PC (that addresses single-stream replay). It is specifically the Stripe idempotency key pattern applied with content-derived keys, which transforms a payment-API safety mechanism into a cross-instance write deduplication mechanism.

### Interaction with existing Recall components

The Q013 per-machine Bloom filter and the content-derived idempotency key serve different layers:

| Layer | Mechanism | Where it fires | Catch rate |
|-------|-----------|---------------|-----------|
| Machine-local | Q013 Bloom filter | Hook, before HTTP call | ~95% of same-machine retries |
| Network boundary | Content-derived idempotency key | FastAPI middleware, before DB write | 100% of cross-machine duplicates within 72h |
| Storage layer | Qdrant cosine similarity check (Q013) | Qdrant search before insert | Near-duplicate semantic variants |

These three layers are complementary. The idempotency key catches the exact-duplicate race (byte-identical content, two machines). The Bloom filter prevents the HTTP call entirely for same-machine retries. The Qdrant cosine check catches semantically-equivalent but not byte-identical memories.

## Ideas Extracted

### Idea: recall-content-derived-idempotency-keys
- **Novelty**: 0.90 — No reviewed AI memory system implements cross-instance write deduplication via content-derived idempotency keys. Stripe/AWS use idempotency keys for payment safety, not for multi-agent memory coordination. The combination of content-hashing + idempotency key table for AI memory is not found in mem0, Zep, MemGPT, LangMem, AWS AgentCore, or Cognee.
- **Evidence**: 0.85 — The Stripe idempotency key pattern is production-proven at scale (hundreds of millions of API calls/day). The PostgreSQL `SELECT FOR UPDATE` locking mechanism is well-established. Content-derived SHA-256 keys for API deduplication are widely deployed (AWS SNS content-based deduplication uses this exact approach). The mechanism requires no new theory — only a new application domain.
- **Feasibility**: 0.90 — Implementation requires: (1) one new PostgreSQL table (idempotency_keys), (2) one middleware function in Recall's FastAPI layer (~40 lines), (3) hook modification to compute and send the idempotency key header (~10 lines in recall-retrieve.js and observe-edit.js). No new infrastructure. Estimated time: 3–5 days including testing. The lock expiry and reaper job are optional initially — even without them, the mechanism eliminates the race window for the common case.
- **Description**: Add a content-derived idempotency key to every write request Recall's hooks generate. The key is `SHA-256(normalize(memory_content))`, computed in the hook before the HTTP call. The Recall FastAPI layer checks this key against a `idempotency_keys` PostgreSQL table before executing any write. If the key is found (written within the last 72 hours), the request returns the cached response without touching Qdrant, Neo4j, or the memories table. This eliminates cross-machine duplicate writes from casaclaude and proxyclaude without requiring any coordination protocol between the machines — each machine independently generates the same key for the same content, and the server handles the collision. The Bloom filter (Q013) continues to operate as the first line of defense for same-machine retries; the idempotency key table is the second line that coordinates across machines.

### Idea: recall-epoch-fencing-for-stale-hooks
- **Novelty**: 0.75 — Epoch fencing from Kafka's transactional producer model has not been applied to AI memory hook coordination. The problem it solves for Recall is different from the cross-machine dedup case: it addresses the "zombie hook" scenario where a hook from a crashed session continues sending stale memories.
- **Evidence**: 0.80 — Kafka's epoch fencing is production-proven in Apache Kafka since version 0.11.0 (2017) and is deployed at massive scale. The mechanism is peer-reviewed in KIP-98 and validated in Confluent's production infrastructure.
- **Feasibility**: 0.65 — Requires session-level producer registration: each Claude Code session registers with Recall on startup (issuing a session_id + epoch), and all writes from that session carry the epoch. Recall stores the current epoch per logical producer identity (casaclaude, proxyclaude). Writes with stale epochs are rejected. The additional complexity is: what is the "logical producer identity"? It must be stable across session restarts on the same machine (hostname or machine ID), not the session ID. Implementation: 2–3 weeks including session registration flow.
- **Description**: Recall adopts Kafka's epoch fencing pattern for hook writers. Each machine (casaclaude, proxyclaude) registers a stable producer identity (machine hostname hash) with Recall at session start. Recall returns a session epoch. All subsequent write requests from that machine must carry the current epoch. If a stale hook (from a crashed or restarted Claude Code session) attempts to write with an old epoch, Recall rejects the write with a `ProducerFencedException`-equivalent error (HTTP 409 with `X-Fencing: epoch_stale`). The new session (new epoch) writes without conflict. This prevents the scenario where a zombie hook from a crashed session continues writing memories after a new session has taken over. This is complementary to content-derived idempotency keys: the idempotency key prevents content duplicates; epoch fencing prevents writes from stale producers.

### Idea: recall-write-ahead-log-for-cross-machine-ordering
- **Novelty**: 0.70 — Write-ahead log (WAL) patterns from database systems have not been applied to AI memory write ordering across multiple instances. The concept is well-known in databases but its application to multi-agent memory coordination is not found in reviewed systems.
- **Evidence**: 0.70 — WAL is a cornerstone of database crash recovery (PostgreSQL WAL, MySQL binlog, SQLite WAL mode). The mechanism is peer-reviewed and deployed in virtually every relational database system.
- **Feasibility**: 0.45 — Recall already uses PostgreSQL, which has a WAL internally. However, exposing a WAL-based ordering guarantee to external hook writers requires a new API layer: hooks write to a WAL endpoint first, receive a sequence number, then Recall's internal consumer processes the WAL and executes writes in order. This is a significant architectural change (essentially implementing a mini Kafka within Recall). Estimated time: 6–10 weeks.
- **Description**: Recall exposes a write-ahead log endpoint: hooks write to `POST /wal/append` and receive a sequence number. The WAL stores (sequence_number, machine_id, timestamp, memory_content_hash, full_payload). A background consumer reads the WAL in sequence order and processes each entry. Deduplication is trivial: the consumer compares each WAL entry's content hash against a processed-hashes set. If the hash already appears in the set (written by either machine), the entry is acknowledged but not executed. Sequence numbers allow ordering guarantees: memories from the same machine are written in the order they were observed. Cross-machine ordering is best-effort (WAL ordering by arrival time). This is the most powerful but most complex of the three mechanisms — appropriate only if Recall scales to more than 2 machines or requires strict write ordering.

## Suggested Follow-ups

1. **Prototype recall-content-derived-idempotency-keys as a 3-day spike** [ABSENCE]: This is the highest-priority follow-up. The mechanism is completely specified above. Implement the `idempotency_keys` PostgreSQL table, the FastAPI middleware, and the hook modification. Test by deliberately sending the same memory from two simulated machines within a 1-second window. Measure: (a) does only one DB write occur? (b) does the second request receive the cached response? (c) what is the latency overhead of the idempotency key lookup (should be under 2ms with a PK index)? If this passes, it is production-ready.

2. **Measure actual duplicate rate in live Recall** [PHYSICS]: Before implementing, quantify the problem. Query the Recall PostgreSQL memories table for pairs of memories with (a) identical content hashes, (b) created_at within 60 seconds of each other, (c) different source annotations (casaclaude vs. proxyclaude). If the duplicate rate is under 0.1% of writes, the priority is lower. If it is 5%+ (expected for simultaneous session starts and similar file edits), the idempotency key is high priority.

3. **Evaluate whether content normalization needs semantic normalization** [ADJACENT]: SHA-256 of the raw text handles byte-identical duplicates. But casaclaude might write "CasaOS runs on 192.168.50.19" and proxyclaude might write "The CasaOS host is at 192.168.50.19" — semantically identical, byte-different. This is NOT caught by content hash. The Q013 Qdrant cosine similarity check handles this case. The question is whether the combined coverage of (content hash for exact duplicates) + (Qdrant cosine for near-duplicates) leaves a meaningful gap in the middle. This requires empirical measurement on actual Recall duplicate pairs.

4. **Investigate whether Kafka's PID approach suggests a machine-registration protocol for Recall** [ADJACENT]: Kafka's PID assignment (each producer registers once and receives a persistent ID) is more durable than ad-hoc hostname hashing. Recall could maintain a `producers` table (machine_id, last_seen, current_epoch) and issue PIDs to hook clients on first contact. This enables the epoch fencing idea (recall-epoch-fencing-for-stale-hooks) and also provides a clean audit trail of which machine wrote which memory. Hypothesis: a machine registration protocol adds 1 day of implementation overhead and enables both content deduplication and epoch fencing as a combined solution.

5. **Check whether mem0's cloud product handles multi-device writes** [ABSENCE]: mem0 explicitly markets itself as a persistent memory layer across multiple AI agents and sessions. Its cloud product (mem0.ai) almost certainly handles concurrent writes from multiple devices. The question is whether it uses content-derived idempotency keys, last-write-wins, or CRDT-style merge. If mem0 has solved this problem, the novelty score for recall-content-derived-idempotency-keys drops from 0.90 to approximately 0.50. This should be verified before implementation to ensure the idea's novelty claim holds.
