## Day 1 Verification Checklist

Execute these steps in order after completing the Q069 10-step deployment sequence. Each step names the mechanism being verified, the command to run, and the expected output that confirms correctness. If any step fails, halt and consult the triage tree before proceeding.

### Step 1 — Confirm PostgreSQL Schema (Q062, Q058, Q059)

Verify all required columns exist before any application traffic runs.

```bash
psql -h 192.168.50.19 -U recall_user -d recall_db -c "
SELECT column_name, data_type, column_default
FROM information_schema.columns
WHERE table_name = 'memories'
  AND column_name IN ('last_access_epoch', 'distinct_sessions_accessed')
ORDER BY column_name;
"
```

Expected output: two rows returned — `distinct_sessions_accessed` (integer, default 0) and `last_access_epoch` (integer, default 0).

If either column is missing, the Q059 migration did not complete. Stop. Re-run the schema migration steps before proceeding.

### Step 2 — Confirm B-tree Index Exists (Q062, Q067)

The epoch routing query and MAX() recovery check both require this index. Without it, Q062 delivers correct results at O(N) scan cost — the BREAKTHROUGH speedup does not materialize and S4 detection takes 15-17ms instead of 0.25ms.

```bash
psql -h 192.168.50.19 -U recall_user -d recall_db -c "
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'memories'
  AND indexname = 'memories_last_access_epoch_idx';
"
```

Expected output: one row with `indexname = memories_last_access_epoch_idx` and a definition containing `USING btree (last_access_epoch)`.

If missing, run:
```sql
CREATE INDEX CONCURRENTLY memories_last_access_epoch_idx
  ON memories (last_access_epoch ASC);
```

This is Step 2b from Q069 — it runs online without a maintenance window.

### Step 3 — Confirm Epoch Clock Initialized and Consistent (Q062, Q067)

```bash
# Read Redis epoch counter
redis-cli -h 192.168.50.19 GET recall:global_epoch

# Read max stored epoch from PostgreSQL
psql -h 192.168.50.19 -U recall_user -d recall_db -c \
  "SELECT MAX(last_access_epoch) AS pg_max_epoch FROM memories;"
```

Expected: `recall:global_epoch` is a non-null integer. On a fresh deployment with no existing memories it will be 0 or 1. The Redis value must be >= the PostgreSQL MAX value (redis_epoch >= pg_max_epoch). If redis_epoch < pg_max_epoch, an epoch inversion exists — run the S4 recovery command:

```bash
redis-cli -h 192.168.50.19 SET recall:global_epoch $(
  psql -h 192.168.50.19 -U recall_user -d recall_db -tAc \
    "SELECT MAX(last_access_epoch) + 1 FROM memories;"
)
```

Inspect the TOMBSTONED value format. It must include `content_hash` (for deduplication) and a TTL-compatible timestamp. To verify that Redis TTL is being set on tombstone-bearing scope keys:

Expected: a positive integer (seconds remaining). If -1 (no TTL) or -2 (key missing), the dual-bound TTL mechanism from Q070 is not writing the expiry. This is a correctness defect — tombstoned memories will never be fully pruned from the registry.

### Step 11 — Confirm Re-encounter Multiplier Is Active (Q065)


## 30-Day Steady-State Checks

Run these checks on a rolling monthly basis. Recommended: schedule a daily cron that hits `/health/detail` and alerts on `overall != "healthy"`, plus a monthly manual review of the metrics below.

---

### Monthly Check 1 — S1: IPS P@3 Delta (Q058)

```bash
curl -s "http://192.168.50.19:8200/health/detail" \
  | python3 -c "
import json, sys
r = json.load(sys.stdin)
s1 = r['signals']['s1_ips_p3_delta']
print(f'Status: {s1[\"status\"]}')
print(f'P@3 delta (IPS vs raw-count): {s1.get(\"delta\")}')
print(f'Exploration sessions evaluated: {s1.get(\"exploration_sessions_evaluated\")}')
"
```

Thresholds:
- **Healthy**: delta >= 0.00 (IPS ranking meets or beats raw-count ordering)
- **Warning**: delta < 0.00 for 3 consecutive days (IPS underperforming raw counts — review propensity model refit schedule)
- **Critical**: delta < -0.10 sustained 7 days (systematic regression — disable IPS reranking, fall back to raw-count ordering)

At 30 days, also verify exploration session count is accumulating. If `exploration_sessions_evaluated` is still < 5 at 30 days, the epsilon-greedy exploration is misconfigured or the `is_exploration` logging is broken.

### Monthly Check 4 — S4: Epoch Clock Integrity (Q062, Q067)

```bash
REDIS_EPOCH=$(redis-cli -h 192.168.50.19 GET recall:global_epoch)
PG_MAX=$(psql -h 192.168.50.19 -U recall_user -d recall_db -tAc \
  "SELECT MAX(last_access_epoch) FROM memories;")

echo "Redis epoch:  $REDIS_EPOCH"
echo "PG max epoch: $PG_MAX"

if [ "$REDIS_EPOCH" -ge "$PG_MAX" ] 2>/dev/null; then
  echo "STATUS: HEALTHY (redis >= pg_max)"
else
  echo "STATUS: CRITICAL — epoch inversion detected"
  echo "RECOVERY: redis-cli -h 192.168.50.19 SET recall:global_epoch $((PG_MAX + 1))"
fi
```

Thresholds:
- **Healthy**: redis_epoch >= pg_max_epoch
- **Warning**: epoch gap > 1000 (Redis epoch far ahead of any stored memory — possible data migration artifact or stale corpus)
- **Critical**: redis_epoch < pg_max_epoch OR redis_epoch = 0 with pg_max > 10

At 30 days, also verify epoch is increasing monotonically:

```bash
# Check epoch has advanced since installation
redis-cli -h 192.168.50.19 GET recall:global_epoch
# Should be significantly above 0 after 30 days of use (1 INCR per session)
```

If the epoch has not advanced from its installation value after 30 days of reported use, the session-end hook is not firing the INCR. This means last_access_epoch values in PostgreSQL are stagnating and the recency filter will eventually exclude all memories.

### Monthly Check 6 — Re-encounter Multiplier Drift (Q065)

```bash
psql -h 192.168.50.19 -U recall_user -d recall_db -c "
SELECT
  CASE
    WHEN distinct_sessions_accessed = 0 THEN '0 sessions (new)'
    WHEN distinct_sessions_accessed BETWEEN 1 AND 3 THEN '1-3 sessions'
    WHEN distinct_sessions_accessed BETWEEN 4 AND 10 THEN '4-10 sessions'
    ELSE '10+ sessions (max multiplier)'
  END AS age_bucket,
  COUNT(*) AS memory_count,
  ROUND(AVG(importance_score), 3) AS avg_importance
FROM memories
GROUP BY 1
ORDER BY 1;
"
```

Expected: avg_importance increases monotonically across age buckets, reflecting the `(1 + 0.15 × min(age, 10))` multiplier. If avg_importance is flat across buckets, the re-encounter formula is not being applied. This is a silent quality defect — older, repeatedly accessed memories do not surface above new ones.

## Triage Decision Tree

When the `/health/detail` endpoint returns a non-healthy status, use this decision tree to identify root cause and apply the correct fix. Each branch is self-contained.

---

### S1 Alert: IPS P@3 Delta Below Threshold

```
S1 status = WARNING (delta < 0.00 for 3 consecutive days)
│
├─ Check: is exploration_sessions_evaluated >= 5?
│   NO → Data is insufficient for evaluation.
│         The is_exploration flag may not be logging.
│         Run: grep -c "is_exploration" ~/.claude/hooks/recall-retrieve.js
│         If 0: the exploration logging is absent from the hook. Redeploy.
│         If >= 1: the flag is present but no exploration sessions have run.
│                  Check epsilon value in IPS config — it may be 0 (no exploration).
│
└─ YES (sufficient data, delta is genuinely negative)
    │
    ├─ Check: has propensity model been refit in the last 30 days?
    │   NO → Refit the propensity model against the rolling 200-session window.
    │         The IPS weights are stale and no longer reflect current session patterns.
    │
    └─ YES (model recently refit, delta still negative)
        │
        ├─ Check: has tag vocabulary or session structure changed recently?
        │   YES → The propensity model was trained on a different tag distribution.
        │          Refit again on the most recent 200 sessions only.
        │
        └─ NO → Escalate to critical path below.

S1 status = CRITICAL (delta < -0.10 sustained 7 days)
│
├─ Immediate action: disable IPS reranking, fall back to raw-count ordering.
│   This prevents the degraded IPS weights from actively harming retrieval.
│
├─ Investigate: check retrieve_events for data integrity.
│   psql -c "SELECT COUNT(*), MIN(event_timestamp), MAX(event_timestamp)
│             FROM retrieve_events WHERE is_exploration = TRUE;"
│   If count is very low (< 50 total), the exploration corpus is too small
│   for reliable IPS estimation. The delta is noise, not signal.
│   Reduce the warning threshold or require a minimum of 100 exploration events.
│
└─ If exploration corpus is adequate: the IPS propensity model has a systematic
   bias. Inspect the model coefficients for features with extreme weights.
   Consider resetting propensity to uniform (all exploration slots equally likely)
   and retraining from scratch.
```

## Key Finding

This convergence finding reveals an interdependency structure across the nine validated mechanisms that was not previously documented as a unified system property. Three mechanisms — Q062 (epoch routing), Q067 (epoch reset detection), and Q071 S4 (epoch health signal) — share a single enabling resource: the B-tree index on `last_access_epoch`. The omission of this index (Step 2b in Q069, itself a documented gap in Q059) silently degrades all three simultaneously without any error message. This cross-mechanism dependency creates a new correctness guarantee: the index is not merely a performance optimization but a correctness prerequisite for the epoch subsystem's health monitoring to function within its specified latency budget. The Day 1 verification checklist surfaces this as a single observable test (Step 2), converting a multi-mechanism architectural dependency into a one-line operator check.

The triage decision tree exposes a second previously undocumented interdependency: S2 (MBC level distribution) and S3 (pending queue) share a common failure mode pathway through CasaOS unavailability. When CasaOS is unreachable, S3 goes critical (pending queue accumulates) AND S2 goes stale (no new MBC data to evaluate). A system where both S2 and S3 alert simultaneously should be triaged as a CasaOS connectivity problem rather than two independent failures. The runbook encodes this logic in the S3 critical branch, but the operator should recognize the S2+S3 co-alert pattern as a single root cause.

The 30-day steady-state checks identify monotonic Level 3 fraction growth as the highest-value leading indicator across all nine mechanisms. A rising Level 3 fraction is detectable weeks before any user-visible retrieval quality decline, it implicates a specific correctable cause (tag vocabulary drift), and it is computable from data that must already be present for MBC routing to function. Unlike S1 (which requires 7+ days of exploration sessions to be statistically meaningful) and S4 (which is a binary healthy/critical signal with no gradual degradation), S2 provides a continuous quality gradient that enables proactive intervention. Operators should treat S2 as the primary 30-day leading indicator for Recall health.
