# Q119 — Three-Store Operational Runbook

## Verdict: BREAKTHROUGH
## Quality Score: N=[0.78] E=[0.82] F=[0.90]

## Finding

The Q077 and Q105 runbooks were written for the five-store architecture (PostgreSQL + Neo4j + Qdrant + Redis + Ollama). After the Q115 BREAKTHROUGH confirming the three-store collapse (PostgreSQL + Qdrant + Redis) and Q110's Neo4j elimination migration, every reference to Neo4j — `DETACH DELETE`, S5 health signal, Neo4j orphan count, `cypher-shell`, bolt:// connections, vector-only circuit breaker mode — is obsolete. The three-store architecture has three distinct failure modes with fundamentally different severity profiles, recovery sequences, and downtime characteristics. This finding provides the complete operational runbook for each.

**Critical architectural change**: PostgreSQL is now the sole authoritative content store. Before Neo4j elimination, Qdrant was described as the "canonical store" (Q105: "Qdrant is the canonical store. PG is the audit index."). After Q115, this inverts: **PostgreSQL holds canonical content**; Qdrant holds derived vectors that can be fully reconstructed from PG content via re-embedding. This changes the severity ranking: PG failure is now the most severe (authoritative data at risk), while Qdrant failure is recoverable (vectors can be regenerated).

**Health signals after collapse**: S1, S2, S3, S4, S6 — five signals, no S5 (eliminated per Q115). The go/no-go confirmation gate at each runbook step verifies all applicable signals return to GREEN.

## Runbook 1: Redis Key Loss (Epoch Counter Absent or Zero)

### Failure Mode

The Redis key `recall:tim:global_epoch` (post Q112 multi-tenant rename) is absent, zero, or at a stale historical value. Causes: Redis container restart without AOF persistence, accidental `FLUSHDB`, key TTL expiry (FM-6 from Q117), or AOF file corruption.

**Severity**: Medium. Redis holds only the epoch clock in the three-store architecture (Q115: "Redis keyspace simplifies to epoch counters only"). No memory content is lost. The failure causes the working-set filter to evaluate TRUE for all memories (epoch 0 - tau = negative threshold), injecting the full corpus into every prompt until repaired.

**Expected downtime at Tim's scale (360 memories)**: < 2 minutes total.

### Step R1 — Confirm Failure State

**Gate**: AUTO (non-destructive diagnostic)
**Time**: < 10 seconds

```bash
# Check Redis epoch value
docker exec recall-redis redis-cli GET recall:tim:global_epoch
# Expected if failure: (nil) or "0"

# Check PG max epoch for comparison
docker exec recall-postgres psql -U recall -d recall -tAc \
  "SELECT MAX(last_access_epoch) FROM memories;"
# Expected: a positive integer (e.g., 847)

# Check AOF status
docker exec recall-redis redis-cli CONFIG GET appendonly
# Expected: "appendonly" "yes" — if "no", AOF was not enforced (root cause)
```

**Go/no-go**: If `redis_epoch` is nil or 0 AND `pg_max` is a positive integer >> 0, this is confirmed Redis key loss. Proceed. If `redis_epoch` is a large number but less than `pg_max`, apply the Q117 discriminant: check `redis_epoch` absolute value. If `redis_epoch > tau (20)`, this may be PG epoch inflation (FM-3) — do NOT apply the standard repair. Investigate PG writes first.

### Step R2 — Repair Redis Epoch

**Gate**: AUTO (safe — sets Redis to pg_max + 1, which is the correct monotonic value)
**Time**: < 5 seconds

```bash

  "SELECT COALESCE(MAX(last_access_epoch), 0) FROM memories;")

# Set Redis epoch to pg_max + 1
NEW_EPOCH=$((PG_MAX + 1))
docker exec recall-redis redis-cli SET recall:tim:global_epoch $NEW_EPOCH
echo "Redis epoch repaired to: $NEW_EPOCH"
```

**Go/no-go**: Verify the SET succeeded:

```bash

### Step Q4 — Re-embed All Memories from PostgreSQL (THE NOVEL STEP)

**Gate**: GO/NO-GO (this is the most time-consuming step; operator should confirm PG content is intact before proceeding)
**Time**: 1-5 minutes at Tim's scale (360 memories)

This is the recovery path that is unique to the three-store architecture. PostgreSQL is authoritative — every memory's content text lives in the `memories.content` column. Re-embedding iterates over all PG rows, calls Ollama's embedding API, and bulk-upserts the resulting vectors into the freshly created Qdrant collection.

```

**Verify Ollama embedding service is reachable**:

**Execute re-embedding** (Python script — run inside recall-api container or as standalone):

```bash
docker exec recall-api python3 -c "
import asyncio
import httpx
from qdrant_client import QdrantClient
from qdrant_client.models import PointStruct
import asyncpg
import uuid
import json
import time

OLLAMA_URL = 'http://192.168.50.62:11434'
EMBEDDING_MODEL = 'qwen3-embedding:0.6b'
QDRANT_HOST = 'qdrant'
PG_DSN = 'postgresql://recall:recallmemory@postgres:5432/recall'
COLLECTION = 'recall_tim'
BATCH_SIZE = 50

async def embed_text(client, text):
    resp = await client.post(f'{OLLAMA_URL}/api/embed', json={
        'model': EMBEDDING_MODEL,
        'input': text
    })
    data = resp.json()
    return data['embeddings'][0]

async def main():
    # Connect to PG
    pool = await asyncpg.create_pool(PG_DSN, min_size=1, max_size=2)

    # Get all memories
    rows = await pool.fetch(
        'SELECT id, content, content_hash, importance_score, domain, tags, created_at '
        'FROM memories WHERE tenant_id = \$1 ORDER BY created_at',
        'tim'
    )
    print(f'Found {len(rows)} memories to re-embed')

    # Connect to Qdrant
    qclient = QdrantClient(host=QDRANT_HOST, port=6333)

    # Embed and upsert in batches
    async with httpx.AsyncClient(timeout=30.0) as http:
        points = []
        start = time.time()
        for i, row in enumerate(rows):
            vector = await embed_text(http, row['content'])
            point = PointStruct(
                id=str(row['id']),
                vector=vector,
                payload={
                    'content_hash': row['content_hash'],
                    'importance_score': float(row['importance_score'] or 0),
                    'domain': row['domain'] or '',
                    'tags': row['tags'] or [],
                    'created_at': row['created_at'].isoformat() if row['created_at'] else '',
                }
            )
            points.append(point)

            if len(points) >= BATCH_SIZE or i == len(rows) - 1:
                qclient.upsert(collection_name=COLLECTION, points=points)
                elapsed = time.time() - start
                rate = (i + 1) / elapsed if elapsed > 0 else 0
                print(f'  Upserted batch: {i+1}/{len(rows)} ({rate:.1f} memories/sec)')
                points = []

        elapsed = time.time() - start
        print(f'Re-embedding complete: {len(rows)} memories in {elapsed:.1f}s')

    # Verify count
    info = qclient.get_collection(COLLECTION)
    print(f'Qdrant points after re-embed: {info.points_count}')

    await pool.close()

asyncio.run(main())
"
```

## Runbook 3: PostgreSQL Failover or Data Loss

### Failure Mode

PostgreSQL is the authoritative content store in the three-store architecture. PG failure scenarios: container crash with corrupted WAL, volume loss, accidental `DROP TABLE`, or snapshot restore to an old backup. This is the **most severe** failure mode because PG holds:

1. `memories` table — all canonical content, importance scores, tags, domains, timestamps, epoch values
2. `co_retrievals` table — the entire co-retrieval graph (replaces Neo4j per Q108/Q115)
3. `canary_displacement_log` — canary test audit trail

**Severity**: CRITICAL. If PG data is lost without backup, memory content is unrecoverable. Qdrant holds vectors (not content text) and Redis holds only epoch counters. Neither can reconstruct PG content.

**Expected downtime at Tim's scale**: 2-30 minutes depending on scenario:
- Container crash (volume intact): 1-2 minutes (restart)
- PG restore from backup: 5-15 minutes (restore + consistency repair)
- Volume loss without backup: **UNRECOVERABLE** for content — vectors in Qdrant become orphans

#### Step P-B4 — Identify and Delete Qdrant Orphans (Dry Run)

**Gate**: GO/NO-GO (operator reviews counts before destruction)
**Time**: 10-30 seconds

Qdrant orphans are vectors whose `content_hash` does not exist in PostgreSQL. Since PG is authoritative, any Qdrant point not backed by a PG row is an orphan.

```bash

```

**Go/no-go**: If estimated orphans = 0, no Qdrant cleanup needed — skip to Step P-B6. If orphans > 0, proceed to deletion. If orphan count seems unreasonably high (> 50% of Qdrant points), double-check the backup timestamp and PG row count.

#### Step P-B5 — Delete Qdrant Orphans

**Gate**: GO/NO-GO (IRREVERSIBLE without Step P-B2 snapshot)
**Time**: 5-60 seconds

```bash
# Delete Qdrant points that have no matching PG row
# This requires scrolling Qdrant and checking each point's content_hash against PG
docker exec recall-api python3 -c "
import asyncio
from qdrant_client import QdrantClient
import asyncpg

async def main():
    pool = await asyncpg.create_pool('postgresql://recall:recallmemory@postgres:5432/recall')

    # Get all PG content hashes
    pg_hashes = set()
    rows = await pool.fetch('SELECT content_hash FROM memories WHERE tenant_id = \$1', 'tim')
    for r in rows:
        pg_hashes.add(r['content_hash'])
    print(f'PG hashes: {len(pg_hashes)}')

    # Scroll Qdrant and find orphans
    qclient = QdrantClient(host='qdrant', port=6333)
    orphan_ids = []
    offset = None
    while True:
        points, offset = qclient.scroll('recall_tim', limit=100, offset=offset, with_payload=True)
        if not points:
            break
        for p in points:
            ch = p.payload.get('content_hash', '')
            if ch not in pg_hashes:
                orphan_ids.append(p.id)
        if offset is None:
            break

    print(f'Qdrant orphans found: {len(orphan_ids)}')

    if orphan_ids:
        # Delete in batches of 100
        for i in range(0, len(orphan_ids), 100):
            batch = orphan_ids[i:i+100]
            qclient.delete('recall_tim', points_selector=batch)
            print(f'  Deleted batch {i//100 + 1}: {len(batch)} points')

    # Final count
    info = qclient.get_collection('recall_tim')
    print(f'Qdrant points after cleanup: {info.points_count}')

    await pool.close()

asyncio.run(main())
"
```

**Go/no-go**: Qdrant point count must now match PG memory count (within ±1).

#### Step P-C2 — Partial Recovery from Qdrant Metadata

**Gate**: GO/NO-GO (operator accepts partial recovery — content text is lost)
**Time**: 5-15 minutes

Qdrant payloads contain `content_hash`, `importance_score`, `domain`, `tags`, and `created_at` — but NOT the full content text. A partial recovery can reconstruct the PG metadata from Qdrant payloads, but the actual memory content is **irrecoverably lost** unless the original session summaries or hook outputs exist elsewhere.

```bash
# Extract Qdrant payloads for metadata reconstruction
docker exec recall-api python3 -c "
from qdrant_client import QdrantClient
import json

qclient = QdrantClient(host='qdrant', port=6333)
offset = None
memories = []
while True:
    points, offset = qclient.scroll('recall_tim', limit=100, offset=offset, with_payload=True)
    if not points:
        break
    for p in points:
        memories.append({
            'id': str(p.id),
            'payload': p.payload
        })
    if offset is None:
        break

print(f'Recovered {len(memories)} memory metadata records from Qdrant')
print('WARNING: Content text is NOT recoverable from Qdrant vectors.')
print('Only metadata (hash, importance, domain, tags, timestamps) is available.')

with open('/tmp/qdrant_recovery.json', 'w') as f:
    json.dump(memories, f, indent=2)
print('Metadata saved to /tmp/qdrant_recovery.json')
"
```

**Recommendation**: At Tim's scale (360 memories, single user), the fastest recovery path is to start fresh. The co_retrievals graph and importance scores will rebuild naturally over subsequent sessions. The epoch counter can be reset to 1. The critical learning: **implement regular PG backups** (see Recommendations).

## Qualifying Ideas

- **three-store-runbook-with-pg-as-authority**: (N=0.78, E=0.82, F=0.90) — The inversion of canonical authority from Qdrant (Q105: "Qdrant is the canonical store") to PostgreSQL (post-Q115: PG holds all content, Qdrant is derived) fundamentally changes the operational runbook structure. Qdrant failure drops from catastrophic to recoverable; PG failure escalates from manageable to critical. This authority inversion is the most operationally significant consequence of the three-store collapse that was not explicitly stated in Q115's architecture diagram.

- **qdrant-re-embed-from-pg-recovery**: (N=0.82, E=0.72, F=0.85) — The full Qdrant re-embedding recovery path (iterate PG content rows, call Ollama embedding API in batch, bulk upsert to fresh collection) is a novel operational procedure enabled by the three-store collapse. In the five-store architecture, Qdrant was canonical and PG was the audit index — Qdrant loss was catastrophic. Now that PG is authoritative, Qdrant becomes a materialized view that can be fully rebuilt in ~72 seconds at Tim's scale (360 memories x 200ms/embedding). This converts Qdrant from "irreplaceable data store" to "rebuildable cache."

- **pg-backup-as-critical-path-gate**: (N=0.70, E=0.80, F=0.95) — With PostgreSQL as the sole authoritative store, PG backup becomes the single most important operational practice. A daily `pg_dump` to a separate volume (or off-host location) is the difference between "all failure modes are recoverable" and "PG volume loss is catastrophic." The backup script is trivial: `docker exec recall-postgres pg_dump -U recall -d recall > /path/to/backup/recall_$(date +%Y%m%d).sql`. Schedule via cron. Retention: 7 daily + 4 weekly.

- **neo4j-orphan-elimination-from-runbooks**: (N=0.68, E=0.90, F=0.98) — Every Neo4j-specific operational step from Q077 and Q105 — `DETACH DELETE`, S5 health signal monitoring, Neo4j orphan count checks, vector-only circuit breaker mode, bolt:// connectivity verification, JVM heap tuning — is eliminated from the operational surface. The three-store runbook has zero Neo4j references. This reduces the operator's cognitive load and eliminates an entire class of failure modes (Neo4j JVM OOM, Cypher query timeout, bolt connection pool exhaustion) from the incident response playbook.

## Recommendations

1. **Implement automated PG backups immediately.** This is the single highest-priority operational task after the three-store collapse. Without PG backups, Runbook 3C (volume loss) has no recovery path.

   ```bash
   # Add to CasaOS crontab (run daily at 3 AM)
   0 3 * * * docker exec recall-postgres pg_dump -U recall -d recall | gzip > /mnt/backup/recall_$(date +\%Y\%m\%d).sql.gz
   # Retention: keep 7 daily + 4 weekly
   0 4 * * 0 find /mnt/backup/ -name "recall_*.sql.gz" -mtime +30 -delete
   ```

Include a `--dry-run` flag that counts PG rows and estimates embedding time without making API calls.
