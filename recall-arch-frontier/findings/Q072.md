# Finding: Q072 — Tombstone Registry Memory Ceiling and LRU Eviction Defeat of Dual-Bound TTL

**Question**: The dual-bound tombstone TTL (Q070) stores `created_epoch` and `created_ts` as fields in each Redis tombstone hash. At Tim's observed eviction rate, how large does the tombstone registry grow in Redis memory over 90 calendar days, and does the answer change if lazy-expiry (Redis only evicts a key when it is accessed) is used instead of active-expiry (background scan)? Is there a tombstone count ceiling above which Redis memory pressure triggers LRU eviction of non-expired tombstones, defeating the dual-bound TTL guarantee?
**Question Type**: ADVERSARIAL
**Verdict**: PROMISING
**Severity**: Medium
**Source field**: Redis internals (expiry mechanics, memory accounting, maxmemory policies), distributed systems (TTL guarantee durability)

---

## Analysis

### Part 1: Tombstone Registry Size at 90 Days

#### Tombstone accumulation rate

Tim's usage parameters:
- Sessions/day: 5 (normal, single machine) to 10 (high activity, both machines)
- Evictions/session: 5–15 memories at steady state; use 10 as the midpoint
- TTL per tombstone: min(50 sessions, 90 calendar days) — dual-bound from Q070

At 90 days, tombstones created in days 1–90 are still alive (within the calendar TTL). The session-count bound (50 sessions) fires first for most tombstones at Tim's session density:

| Scenario | Sessions/day | Evictions/session | Tombstones created over 90 days | Session-count TTL (50 sessions) expires after |
|---|---|---|---|---|
| Normal (single machine) | 5 | 10 | 5 × 90 × 10 = **4,500** | 50/5 = **10 calendar days** |
| High activity (dual machine) | 10 | 15 | 10 × 90 × 15 = **13,500** | 50/10 = **5 calendar days** |
| Burst (20 sessions/day) | 20 | 15 | 20 × 90 × 15 = **27,000** | 50/20 = **2.5 calendar days** |

Because the session-count TTL (50 sessions) fires at 5–10 calendar days under Tim's usage, tombstones are **not** accumulating for 90 days. They expire after 5–10 days in practice. The live tombstone set at any given moment is bounded by the evictions that occurred in the last 5–10 calendar days:

| Scenario | Live tombstones at steady state |
|---|---|
| Normal | 5 sessions/day × 10 days × 10 evictions = **500 live tombstones** |
| High activity | 10 sessions/day × 5 days × 15 evictions = **750 live tombstones** |

The total over the full 90-day window (counting all tombstones created, including expired ones still in Redis until active expiry cleans them) caps at the figures above (4,500 / 13,500) only if Redis expiry were disabled entirely. Under active expiry (`hz=10`), expired tombstones are collected within seconds to minutes of TTL expiry. The live set stays small.

#### Per-tombstone Redis memory cost

The tombstone key is `recall:tombstone:{content_hash}` where `{content_hash}` is a 64-byte hex SHA-256 string. Key string length: 17 (`recall:tombstone:`) + 64 = **81 bytes**.

Each tombstone is a Redis hash with 4 fields:
| Field | Content | Bytes |
|---|---|---|
| `content_hash` | 64-char hex SHA-256 | 64 |
| `created_epoch` | integer (encoded as string in Redis) | ~4–8 |
| `created_ts` | ISO-8601 string `2026-03-16T14:23:01.123456` | ~26 |
| `reason` | e.g., `importance_below_threshold` | ~30 |

Raw field data: ~128 bytes.

Redis hash encoding: With `content_hash` as a 64-byte value, the hash exceeds the default `hash-max-ziplist-value 64` threshold (64 bytes is at the boundary; the field value is exactly 64 bytes, which may or may not trip the threshold depending on Redis version and configuration). Assuming **listpack/ziplist encoding** (all values ≤ 64 bytes):
- Ziplist overhead per entry: ~11 bytes (prevlen + encoding + length)
- 4 fields × (key + value + overhead): (7 + 64 + 11) + (13 + 8 + 11) + (11 + 26 + 11) + (6 + 30 + 11) ≈ ~209 bytes for the ziplist body
- Ziplist header: ~10 bytes
- Redis hash robj: ~16 bytes
- Total hash value object: ~235 bytes

Redis key overhead:
- SDS string for the key: 81 bytes data + 8 bytes SDS header = 89 bytes
- Redis dict entry (key → value pointer): ~32 bytes
- robj wrapper for key: ~16 bytes
- Total key overhead: ~137 bytes

**jemalloc allocation rounding**: Redis uses jemalloc which allocates in power-of-2 or size-class-aligned blocks. The hash object (~235 bytes) rounds up to 256 bytes; the key SDS (~89 bytes) rounds up to 96 bytes.

**Total per tombstone**: 256 (hash value) + 96 (key SDS) + 16 (key robj) + 32 (dict entry) = **~400 bytes per tombstone**.

This aligns well with the question's stated ~200 bytes payload: the overhead factor is approximately 2×, which is standard for Redis hash objects at this size.

#### Memory totals

Using 400 bytes per tombstone:

| Scenario | Live tombstones (steady state) | Redis memory | Max tombstones if expiry disabled (90 days) | Memory if expiry disabled |
|---|---|---|---|---|
| Normal | 500 | **~200 KB** | 4,500 | **~1.8 MB** |
| High activity | 750 | **~300 KB** | 13,500 | **~5.4 MB** |
| Burst | 1,000 | **~400 KB** | 27,000 | **~10.8 MB** |

**Conclusion**: The tombstone registry is negligibly small — well under 1 MB in normal operation, under 11 MB even in the pathological no-expiry case. On Tim's homelab Redis instance with no `maxmemory` set, this produces zero memory pressure.

### Part 2: Lazy Expiry vs Active Expiry — Does the Size Change?

#### How Redis expiry works

Redis manages TTL expiry through two mechanisms that can operate independently or together:

**Active expiry** (background scan): Every `1000/hz` milliseconds (100ms at `hz=10`), Redis samples up to 20 random keys from the keyspace that have TTLs set, expires those that are past their deadline, and loops until fewer than 25% of the sampled keys were expired (otherwise it scans more). At `hz=10`, this runs 10 times per second, expiring up to ~200 keys per second across the entire keyspace. For Tim's tombstone keyspace of 500–1,000 live keys, all expired tombstones are statistically cleaned within seconds of TTL fire.

**Lazy expiry** (`lazyfree-lazy-expire=yes`): Redis only deletes an expired key when it is next accessed. If a tombstone is never re-accessed (never re-encountered), it remains in memory indefinitely after TTL, occupying memory until either active expiry finds it or it is accessed.

#### Impact of lazy expiry on tombstone registry size

With **lazy expiry ONLY** (active expiry disabled, which is not possible with standard Redis — `hz` cannot be set to 0 — but approximates very-low-hz behavior):

Tombstones accumulate past their TTL until one of:
1. Active expiry background scan eventually reaches them (probabilistic, at `hz=10` this is reliable within minutes for small keyspaces)
2. A Phase 1 lookup accesses the key (triggers lazy deletion at that moment)

In practice, **`lazyfree-lazy-expire` only changes the deletion moment, not the expiry check**. When Redis evaluates an expired key for any reason (access or background scan), it marks it for deletion. The `lazyfree-lazy-expire=yes` setting means the actual memory reclaim is deferred to a background thread rather than blocking the main thread — it does not mean expired keys are kept alive until accessed. The key still expires (appears deleted to callers) at TTL time; the memory reclaim is what's deferred.

**The question's framing** — "lazy-expiry: Redis only evicts a key when it is accessed" — describes the behavior of keys where TTL checking is deferred until access. This is actually the behavior when active expiry is the ONLY mechanism and is scaled back (high `hz` load, many keys). In a small keyspace (500 tombstones) with `hz=10`, active expiry is highly effective and effectively continuous.

**Practical difference**: For Tim's tombstone count (~500–1,000 live keys), the difference between lazy and active expiry is milliseconds to seconds on expiry timing. Memory impact is negligible either way. The tombstone keyspace is small enough that Redis's probabilistic active expiry scans it completely several times per second.

**Worst case for lazy expiry** (no active expiry at all, tombstones only expired on access): Tombstones for topics that are never re-encountered persist until Redis restarts or an explicit SCAN+DEL cleanup job runs. At Tim's eviction rate, over 90 days without any active expiry, the registry could accumulate the full 90-day corpus (4,500–13,500 keys, ~1.8–5.4 MB). Still negligible memory.

**Conclusion**: The answer barely changes between lazy and active expiry for this keyspace size. The maximum difference is ~5.4 MB of additional memory (the full 90-day corpus versus the steady-state 300 KB). Neither figure creates meaningful memory pressure.

### Part 3: LRU Eviction and the Dual-Bound TTL Guarantee

#### The threat model

The dual-bound TTL guarantee (Q070) states: a tombstone survives until EITHER 50 sessions OR 90 calendar days have passed, whichever comes first. This guarantee requires that the tombstone key exists in Redis at the moment a Phase 1 lookup occurs. If Redis evicts the tombstone key before its TTL fires — via LRU eviction driven by `maxmemory` pressure — the tombstone appears absent, the re-encounter multiplier does not fire, and the topic is incorrectly treated as novel. This defeats the dual-bound TTL guarantee silently.

#### Redis `maxmemory` on Tim's homelab

Tim's Redis deployment has **`maxmemory` unset** (confirmed in question context). With `maxmemory=0` (unset), Redis uses **no eviction policy** — it grows unbounded and never evicts keys for memory reasons. LRU eviction is impossible with `maxmemory` unset.

**Current state: the LRU defeat scenario cannot occur** because Redis has no eviction policy active.

#### When LRU eviction WOULD defeat the TTL guarantee

If Tim (or a future deployer) sets `maxmemory` and uses an eviction policy, the risk depends on the policy chosen:

| `maxmemory-policy` | LRU defeat of tombstone TTL? | Explanation |
|---|---|---|
| `noeviction` | No | Redis returns OOM errors instead of evicting. Tombstones safe. |
| `allkeys-lru` | **Yes — high risk** | Redis evicts any key by LRU access order regardless of TTL. Tombstones that haven't been accessed recently (most tombstones, since re-encounter is rare) are prime eviction candidates. |
| `volatile-lru` | **Yes — moderate risk** | Redis evicts only keys WITH TTLs, sorted by LRU. All tombstones have TTLs — they are ALL in the eviction pool. A tombstone created 9 days ago (near its 10-day session-count TTL expiry) has not been accessed in 9 days and is highly likely to be evicted before it expires naturally. |
| `allkeys-random` | Yes — low-probability | Random eviction; tombstones have the same probability as any other key. |
| `volatile-random` | Yes — low-probability | Random eviction among TTL-bearing keys; tombstones are in the pool. |
| `volatile-ttl` | **Yes — maximum risk** | Redis evicts keys with the soonest TTL first. Tombstones near their natural expiry are evicted BEFORE they expire. A tombstone 1 day from its TTL firing is a prime eviction target. |
| `allkeys-lfu` | Yes — high risk | LFU eviction: tombstones are accessed at most once (at creation) and once at Phase 1 lookup. Access frequency = 1-2 total. Very likely to be LFU-evicted. |

#### The tombstone count ceiling that triggers LRU pressure

With `maxmemory` set, LRU eviction begins when total Redis memory approaches the `maxmemory` limit. The tombstone registry competes with:
- G3 idempotency cache (Q063): ~72-hour TTL entries, potentially hundreds of keys
- Scope registry: N_memories × ~200 bytes ≈ small
- Epoch counter (Q062): trivial
- Vector search cache (if any): potentially large

For a homelab Redis instance with `maxmemory=100MB` (a common configuration), and the tombstone registry at ~300 KB steady state, the tombstone registry would need to coexist with ~99.7 MB of other data before any eviction pressure touches tombstones. At Tim's scale, this is not a realistic threshold.

**The actual ceiling**: LRU tombstone eviction becomes possible only if `maxmemory` is set to a value smaller than the total Redis memory usage. For Tim's deployment, this is not currently a risk. For a future production deployment serving many users:
- 1,000 users × Tim's eviction rate = 500,000 live tombstones × 400 bytes = **200 MB**
- A `maxmemory=100MB` configuration would trigger constant LRU eviction of tombstones
- `volatile-lru` or `volatile-ttl` would systematically eliminate older tombstones — exactly the ones at highest risk of defeating the guarantee (they are closest to their TTL but haven't fired yet)

#### Redis configuration to prevent LRU defeat of tombstone TTL

Three approaches, in order of preference:

**Option 1: Dedicated Redis instance for tombstones with `maxmemory-policy noeviction`**

Separate the tombstone registry from all other Redis data (G3 cache, epoch counter, scope registry). Run a dedicated Redis instance on a separate port configured with:
```
maxmemory 50mb
maxmemory-policy noeviction
```
With `noeviction`, Redis returns an OOM error when memory is exhausted rather than silently evicting keys. The application can detect and alert on OOM errors; it cannot detect silent LRU eviction of tombstones. `noeviction` converts a silent correctness failure into a loud operational failure — the correct trade-off for a correctness-critical keyspace.

At Tim's scale, 50 MB is ample for the tombstone registry (max ~11 MB even under worst-case assumptions). The application catches Redis OOM on tombstone write and falls back to writing the tombstone to PostgreSQL as a backup store.

**Option 2: Dedicated Redis DB with `maxmemory-policy noeviction` for the entire instance**

If a separate instance is operationally burdensome, use `SELECT 1` (a separate DB within the same Redis instance) for tombstones. Set `maxmemory-policy noeviction` for the entire Redis instance. Downside: `noeviction` applies to ALL databases including the G3 cache, which might benefit from LRU eviction when memory pressure occurs. Tradeoff: accept potential G3 OOM errors rather than silent tombstone eviction.

**Option 3: Application-layer TTL enforcement as primary, Redis as secondary**

If `maxmemory` must be set with an eviction policy (e.g., for the G3 cache), decouple tombstone durability from Redis. At Phase 1 lookup:
1. Check Redis tombstone key (fast path)
2. If Redis key absent (expired OR LRU-evicted), check PostgreSQL tombstone table as fallback
3. Write tombstones to BOTH Redis (fast lookup) AND PostgreSQL (durable backup)

This makes the dual-bound TTL guarantee independent of Redis's eviction behavior. The guarantee is enforced by PostgreSQL (which has no LRU eviction) and Redis serves as a cache for fast lookup. Redis LRU eviction of tombstone keys degrades lookup performance (misses go to PostgreSQL) but does not defeat the TTL guarantee.

#### `hz` setting analysis

The default `hz=10` is sufficient for Tim's tombstone keyspace. Redis expires ~200 keys/second via active expiry. With 500–1,000 live tombstones and a natural expiry rate of ~100 tombstones/day (10 evictions/session × 10 sessions/day), the active expiry system processes expired tombstones roughly 1,000× faster than they expire. Increasing `hz` to 100 would reduce expiry latency from ~seconds to ~100ms — meaningless for TTLs measured in days. There is no benefit to changing `hz` for the tombstone use case.

## Ideas Extracted

### Idea: tombstone-noeviction-dedicated-redis
- **Novelty**: 0.68 — Separating a correctness-critical keyspace (tombstones) from an evictable keyspace (G3 cache) into a dedicated Redis instance with `noeviction` policy is not novel in distributed systems (it is standard practice to separate durable and evictable caches), but it has not been explicitly applied to AI memory system tombstone registries in prior findings.
- **Evidence**: 0.72 — The correctness argument is mathematically tight: `volatile-lru` or `allkeys-lru` policies can and will evict tombstone keys before TTL under memory pressure. The risk is not hypothetical — it is a direct consequence of how LRU eviction works in Redis. The mitigation (noeviction policy) is the only policy that provides a hard guarantee against silent key removal. PostgreSQL-backed tombstone fallback provides defense-in-depth and is a well-established pattern (cache + durable store layering).
- **Feasibility**: 0.88 — Adding a dedicated Redis instance on Tim's homelab (CasaOS) is a one-line Docker Compose service addition. Configuring `maxmemory-policy noeviction` is a single redis.conf line. Modifying the tombstone write path to also write to PostgreSQL is a ~2-hour implementation. The combined change converts tombstone correctness from a Redis-configuration-dependent property into a system property that survives Redis misconfiguration.
- **Description**: Run the tombstone registry on a dedicated Redis instance configured with `maxmemory-policy noeviction` and a generous `maxmemory` ceiling (50MB). Optionally write each tombstone to a PostgreSQL `tombstone_registry` table as a durable backup. Phase 1 lookup checks Redis first (fast); on Redis miss (expired or evicted), falls back to PostgreSQL. This makes the dual-bound TTL guarantee independent of Redis eviction policy and Redis availability, at the cost of one additional write per tombstone creation and one PostgreSQL lookup per Redis miss.

### Idea: tombstone-lru-defeat-detection-canary
- **Novelty**: 0.70 — Using canary keys to detect silent LRU eviction of a correctness-critical keyspace is a novel application of a known technique (canary key injection for Redis health monitoring). In standard Redis monitoring, LRU eviction is tracked via `evicted_keys` in `INFO stats`, but this does not distinguish which keyspace was evicted. A tombstone-specific canary detects eviction within the tombstone keyspace specifically.
- **Evidence**: 0.58 — The canary approach is structurally sound: a long-TTL key in the tombstone keyspace, written at system startup, is a valid sentinel for detecting LRU eviction of the keyspace. If the canary disappears before its TTL, LRU eviction is actively affecting tombstones. The eviction detection is correct in principle. Evidence is structural rather than empirically validated for this specific use case.
- **Feasibility**: 0.82 — At session start, write a canary key `recall:tombstone:__canary__` with a 7-day TTL and a known value. At each Phase 1 lookup, check whether the canary still exists. If absent and fewer than 7 days have elapsed (checked via the canary's creation timestamp stored in a companion PostgreSQL row), LRU eviction is confirmed and an alert is raised. The canary check adds one Redis EXISTS call per session, negligible latency. Implementation: ~1 hour.
- **Description**: Inject a sentinel tombstone key `recall:tombstone:__canary__` with a 7-day TTL at Recall startup. Store the canary's creation timestamp in PostgreSQL. At each Phase 1 lookup (session start), verify the canary exists. If it is absent before 7 days have elapsed, Redis has silently evicted keys from the tombstone keyspace via LRU pressure. Log a CRITICAL alert and fall back to PostgreSQL-backed tombstone lookup until the condition resolves. This converts silent LRU defeat of the TTL guarantee into a detectable, alertable condition.

## Key Finding

The tombstone registry is a **non-problem for memory** at Tim's scale. At steady state (session-count TTL dominating at 5–10 calendar days), only 500–1,000 tombstones are live at any moment, consuming ~200–400 KB of Redis memory — less than 1% of a typical homelab Redis instance. Even if active expiry were completely disabled and all 90 days of tombstones accumulated (4,500–13,500 keys), the registry would peak at 1.8–5.4 MB. Lazy expiry (deferred memory reclaim) adds at most a few megabytes to the worst-case figure; it does not change the order of magnitude. The tombstone registry will never trigger memory pressure on Tim's homelab.

The LRU eviction threat is **real but currently dormant**. With `maxmemory` unset (Tim's current configuration), Redis uses no eviction policy and cannot LRU-evict tombstones. The threat becomes real if and when a `maxmemory` limit is set — especially with `volatile-lru` or `volatile-ttl` policies, which would preferentially evict tombstones (they all have TTLs and are rarely accessed). The worst policy is `volatile-ttl`: it evicts keys with the soonest TTL first, meaning tombstones closest to their natural expiry are evicted just before the dual-bound guarantee fires — producing a silent correctness failure that is indistinguishable from a normal cache miss. This failure mode is invisible without monitoring: the tombstone disappears from Redis, Phase 1 treats the topic as novel, and no error is logged.

The correct defense is two-layered: (1) configure `maxmemory-policy noeviction` for the Redis instance or DB containing tombstones, converting silent eviction into a loud OOM error; and (2) write tombstones to PostgreSQL as a durable backup and fall back to PostgreSQL on Redis miss, making the dual-bound TTL guarantee independent of Redis's eviction configuration. Option (2) is the more robust design and aligns with the existing PostgreSQL infrastructure used throughout the Recall system. The combination of `noeviction` policy and PostgreSQL fallback means the tombstone TTL guarantee is upheld even under Redis misconfiguration, Redis restart, or Redis memory exhaustion.
