# Q172 — K-means Bootstrap Clustering: Physics Ceiling

**Question**: What is the theoretical minimum time to complete a one-time K=200 K-means clustering pass on 22,423 float32 vectors of dimension 1024 on the CasaOS N100 CPU — and at what K value does the within-cluster pair count exceed Neo4j single-transaction write capacity, requiring batched MERGE commits?
**Question Type**: PHYSICS
**Verdict**: PHYSICS_CONFIRMED
**Severity**: Medium
**Source field**: Computer architecture (roofline model), graph database benchmarks, Q163 bootstrap idea
**Quality Score**: N=0.68 E=0.90 F=0.92

---

## Operation Definition

**Task A — K-means clustering:**
- Input: 22,423 float32 vectors of dimension 1024 = 91.8 MB corpus
- Output: K=200 cluster assignments + centroid positions
- Constraints: Euclidean distance metric, 10 iterations (convergence criterion), correctness of assignment

**Task B — Neo4j within-cluster MERGE writes:**
- Input: K=200 cluster assignments, 1,245,766 within-cluster pairs
- Output: CO_RETRIEVED relationship records in Neo4j
- Constraints: ACID semantics per transaction, index lookup per node pair, no OOM in heap

---

## Operation Profile

### K-means FLOP Budget

sklearn (and equivalent implementations) compute pairwise distances via the BLAS identity:

```
||a - b||^2 = ||a||^2 + ||b||^2 - 2*(a . b)
```

The dominant term is the matrix multiply `(N, D) @ (D, K)` — a SGEMM call.

| Step | Formula | FLOPs per iteration |
|---|---|---|
| SGEMM: distance matrix | 2 * N * D * K | 9,184,537,600 (9.18 GFLOP) |
| Norm precompute | 2 * (N + K) * D | 46,333,440 (0.046 GFLOP) |
| Centroid update | (N + K) * D | 23,166,720 (0.023 GFLOP) |
| **Total per iteration** | | **9.254 GFLOP** |
| **Total for 10 iterations** | | **92.54 GFLOP** |

The SGEMM term is 99.4% of total FLOPs. All other costs are noise.

### Memory Footprint

| Object | Size | Cache fate |
|---|---|---|
| All 22,423 vectors | 91.8 MB | DRAM — does not fit in 6 MB L3 |
| K=200 centroids | 0.82 MB | L3 cached — fits comfortably |
| Distance matrix (N, K) scratch | 17.5 MB | DRAM |
| Cluster assignment array (N, 1) | 0.09 MB | L1/L2 cached |

## Physics Ceiling Calculations

### N100 Hardware Specification

| Resource | Value | Notes |
|---|---|---|
| Cores | 4 | Alder Lake-N E-cores |
| Boost clock | 3.4 GHz | Sustained boost, all-core |
| SIMD width | 128-bit (SSE4.2) | 4 float32 per register |
| SIMD ops/cycle | 1 practical (2 theoretical) | No FMA3, no AVX, no AVX2 |
| Peak SP GFLOPS (theoretical) | 4 × 3.4 × 4 × 2 = **108.8 GFLOPS** | Two SIMD ports |
| Peak SP GFLOPS (practical) | 4 × 3.4 × 4 × 1 = **54.4 GFLOPS** | Dependent chain reality |
| Memory bandwidth | **34 GB/s** | LPDDR5-4800, conservative |
| L3 cache | 6 MB | Shared across 4 cores |

The N100 Alder Lake-N has SSE4.2 but no AVX/AVX2/FMA3. This is the critical hardware constraint: FMA halves FLOP count by combining multiply and add into one instruction. Without FMA, the effective throughput for dense linear algebra is roughly half of FMA-capable hardware at the same clock and SIMD width.

### Roofline Analysis

```
Arithmetic intensity = FLOPs / bytes_from_DRAM
                     = 9,184,537,600 / 91,800,832
                     = 100.0 FLOP/byte

Ridge point (compute/memory crossover) = peak_GFLOPS / mem_bandwidth
                                       = 54.4 GFLOPS / 34 GB/s
                                       = 1.6 FLOP/byte
```

**K-means at 100.0 FLOP/byte is 62.5x above the ridge point. It is deeply compute-bound, not memory-bound.** The memory bandwidth floor is irrelevant: streaming 91.8 MB at 34 GB/s takes only 2.7 ms per iteration. The compute ceiling is what matters.

### Theoretical Minimum Per Iteration

| Bound | Time | Bottleneck |
|---|---|---|
| Memory bound (stream 91.8 MB at 34 GB/s) | 2.7 ms | Not the bottleneck |
| Compute bound (theoretical 108.8 GFLOPS) | **84.4 ms** | Physics minimum |
| Compute bound (practical 54.4 GFLOPS) | 168.8 ms | Achievable with perfect scheduling |

**Physics minimum: 84.4 ms per iteration = 0.844 seconds for 10 iterations.**

This assumes 100% utilization of both SIMD ALU ports on all 4 cores simultaneously — an upper bound that no real software achieves. It is the hard ceiling set by the transistor count and clock speed.

### Practical Time Estimates

OpenBLAS SGEMM on SSE4.2-only hardware (no AVX) typically achieves 40–65% of theoretical peak for non-square matrix shapes. The (22423, 1024) × (1024, 200) shape is rectangular with a tall first dimension, which is favorable for cache blocking but not square-optimal.

| Efficiency assumption | Per-iteration time | 10 iterations |
|---|---|---|
| 60% of practical peak (best: well-tuned OpenBLAS) | 284 ms | **2.8 s** |
| 40% of practical peak (nominal: sklearn defaults) | 425 ms | **4.3 s** |
| 25% of practical peak (worst: conda/pip numpy, thermal throttle) | 678 ms | **6.8 s** |

**Expected time for the clustering step alone: 4–5 seconds on the N100.**

The Q163 estimate of "~8-12 minutes on commodity hardware" was off by roughly 100x. That estimate confused the K-means clustering cost with the Neo4j write cost that follows it — a category error between FLOP-bound and I/O-bound workloads.

### K Threshold Derivation

**For pairs per cluster to exceed 10,000 (conservative Neo4j transaction limit):**

```
n*(n-1)/2 > 10,000
n^2 - n - 20,000 > 0
n > (1 + sqrt(1 + 4*20,000)) / 2 = 141.9
```

Average cluster size exceeds 141.9 when K < N/141.9 = 22,423/141.9 = **158.0**

**At K < 158, individual clusters on average exceed 10,000 pairs per cluster, requiring multiple batch commits per cluster even at the 10K batch limit.**

**For pairs per cluster to exceed 5,000 (stricter limit for constrained heap environments):**

```
n > (1 + sqrt(1 + 4*10,000)) / 2 = 100.5
K < 22,423/100.5 = 223.1
```

**At K < 223, individual clusters on average exceed 5,000 pairs per cluster.**

### K=200 Specific Analysis

```
Avg cluster size:  22,423 / 200 = 112.1 memories
Pairs per cluster: 112.1 × 111.1 / 2 = 6,229 pairs
Total pairs:       6,229 × 200 = 1,245,766 pairs

At 5K batch limit: 2 batches per cluster, 250 total batch commits
At 10K batch limit: 1 batch per cluster, 125 total batch commits
```

K=200 sits comfortably below the 10K threshold (6,229 < 10,000) but marginally above the 5K threshold (6,229 > 5,000). A 10K batch limit handles K=200 in a single commit per cluster. A 5K limit requires 2 commits per cluster.

## Gap Analysis: Current vs Theoretical Minimum

### K-means Clustering

| Measure | Value |
|---|---|
| Total FLOP budget (10 iterations) | 92.54 GFLOP |
| Physics minimum (108.8 GFLOPS, 100% utilization) | 0.844 s |
| Best-case practical (OpenBLAS, 60% efficiency) | 2.8 s |
| Nominal practical (sklearn defaults, 40% efficiency) | 4.3 s |
| Gap ratio (nominal / physics minimum) | 5.1x |

**Physics gaps (unavoidable):**
- No FMA on N100: doubles the instruction count vs AVX2+FMA hardware. This costs a factor of ~2x vs a Zen4 or Ice Lake CPU at the same clock and core count. Cannot be engineered around without different hardware.
- Memory bandwidth irrelevant: at 100 FLOP/byte, the compute-to-bandwidth ratio is so high that even halving memory bandwidth would have no effect on total time. The operation is fundamentally FLOP-bound.

**Architecture gaps (design choices, addressable):**
- Single-threaded BLAS: OpenBLAS will use all 4 cores for the SGEMM call automatically if linked correctly. If sklearn is installed without a threaded BLAS (e.g., using the numpy shipped with some conda environments), the operation becomes single-threaded and 4x slower (~17 seconds for 10 iterations). Verifiable with `numpy.show_config()`.
- Python overhead per iteration: argmin across (22423, 200) array, centroid recompute in Python loop. This adds ~3 ms per iteration — negligible at this scale.

**Implementation gaps (code quality, addressable):**
- Using `sklearn.cluster.KMeans` with `algorithm='lloyd'` and `n_init=1` is the correct path — it uses BLAS SGEMM internally. Using `algorithm='elkan'` or `faiss.Kmeans` would change the FLOP structure but not improve the physics minimum on this hardware.

### Neo4j MERGE Writes

| Measure | Value |
|---|---|
| Total pairs to write (K=200) | 1,245,766 |
| Physics minimum (100K IOPS NVMe, 4 IOPS/MERGE) | 49.8 s |
| Best-case practical (10K batch, 4 workers, 5K ops/s/worker) | 62 s |
| Nominal practical (5K batch, single thread, 3K ops/s) | 415 s (6.9 min) |
| Single MERGE/tx (no batching) | 2,491 s (41.5 min) |
| Gap ratio (no-batching / physics minimum) | 50x |

**Physics gaps (unavoidable):**
- I/O depth: each MERGE requires index lookups. Even with 100% IOPS utilization and perfect index cache warmth, 4,983,063 IOPS at 100K IOPS/s = 49.8 seconds irreducible floor.
- Transaction overhead: each transaction commit requires a write-ahead log flush to durable storage. This is a durability constraint, not an optimization opportunity.

**Architecture gaps (the dominant factor):**
- Transaction granularity: without batching, each MERGE is its own transaction (1-5 ms latency each). 1,245,766 sequential single-MERGE transactions = 20-100 minutes. Batching 10,000 MERGEs per transaction amortizes the per-transaction commit overhead across 10,000 operations, achieving ~5,000 ops/s vs ~500 ops/s without batching — a 10x throughput gain from a single architectural choice.
- Cluster-serial vs parallel writes: writing one cluster at a time through a single Neo4j connection caps throughput to one transaction at a time. Four parallel workers writing to disjoint cluster sets (no overlapping node IDs within a batch window) would give 4x throughput and bring the nominal time to ~1.7 minutes.

### Total Bootstrap Job

| Scenario | Clustering | Neo4j Writes | Total |
|---|---|---|---|
| Physics minimum | 0.84 s | 49.8 s | **50.7 s** |
| Best case (BLAS + batched + 4 workers) | 2.8 s | 62 s | **65 s** |
| Nominal (sklearn + 5K batch + single thread) | 4.3 s | 415 s | **420 s (7.0 min)** |
| Worst case (no batching, single thread) | 6.8 s | 2,491 s | **2,498 s (41.6 min)** |
| **Gap ratio (nominal / physics min)** | | | **8.3x** |

**The clustering step is not the bottleneck.** At K=200 and N=22,423, clustering takes 4–5 seconds. The Neo4j write step dominates at 7–42 minutes depending on batching strategy. Q163's "8-12 minutes" estimate applied the right order of magnitude for the write step but attributed it to the wrong component.

## Recommended K and Batch Size

**Recommended K: 200**

Justification:
- 6,229 pairs per cluster fits within a 10K Neo4j batch limit with one commit per cluster
- 200 clusters of ~112 memories provides semantically meaningful groupings at qwen3-embedding:0.6b embedding resolution (at K=500, clusters of ~45 memories become too fine-grained for co-retrieval pre-seeding)
- Total write volume (1.24M pairs) is tractable in 7-10 minutes with 5K batching, or under 2 minutes with 4 parallel workers

**Recommended batch size: 10,000 MERGEs per transaction**

Justification:
- At K=200, one batch per cluster with zero cluster-spanning commits
- Transaction overhead (125 commits × 10 ms) = 1.25 seconds total overhead — negligible
- Keeps below Neo4j Community Edition OOM risk: 10,000 in-flight objects at ~500 bytes each = 5 MB heap per active transaction, well within a 512 MB heap allocation

**Do not use K < 158 without multi-batch-per-cluster handling.** At K=157, average cluster size crosses 142 memories and each cluster generates 10,128 pairs — exceeding the 10K batch boundary. Below K=100, clusters generate 25,000+ pairs each and require 3+ commits per cluster with careful SKIP/LIMIT pagination logic.

## Key Findings

1. **The N100 K-means physics minimum is 0.844 seconds for 10 iterations, not 8-12 minutes.** The correct expected practical time is 4–5 seconds for the clustering step alone. Q163's "8-12 minutes" estimate was 100-150x too high and incorrectly attributed write-step time to compute-step time.

2. **K-means on this workload is compute-bound, not memory-bound.** Arithmetic intensity = 100 FLOP/byte; roofline ridge point = 1.6 FLOP/byte. The operation sits 62x above the ridge. Memory bandwidth is not a bottleneck and halving it would have no measurable effect on wall-clock time.

3. **The N100's absence of AVX/FMA3 costs approximately 2x vs AVX2+FMA hardware** at the same clock speed. This is a fixed physics gap — it cannot be addressed in software. Moving the job to the RTX 3090 (Ollama host, 192.168.50.62) would provide CUDA GEMM acceleration and reduce the clustering step to under 0.5 seconds.

4. **The K threshold for single-transaction-per-cluster safety is K >= 158** (using a 10K batch limit) **or K >= 224** (using a conservative 5K batch limit). At K=200, each cluster produces 6,229 pairs — one 10K batch per cluster, 125 total batch commits.

5. **The Neo4j write step dominates total job time by 100:1 at minimum.** Clustering at 4 seconds is negligible. Neo4j MERGE throughput without batching (500 ops/s) produces a 41-minute write time. With 10K batching and 4 parallel workers, this collapses to under 2 minutes. The architecture gap — transaction granularity — is the entire optimization opportunity.

6. **At K=200, the bootstrap write generates 1,245,766 CO_RETRIEVED pairs.** This is 4.96x the active-pool density threshold identified in Q163 (177,600 edges needed for measurable retrieval ordering improvement). The bootstrap immediately crosses the threshold that organic accumulation would otherwise take 2+ years to reach.

7. **The 8-12 minute total job estimate from Q163 is achievable at the nominal operating point** (5K batching, single thread) — it just applies to the Neo4j write step, not the clustering step. The decomposition matters for optimization: clustering is already near its physics floor; Neo4j writes have an 8-50x architecture gap that batching and parallelism can close.

## Qualifying Ideas

### kmeans-on-rtx3090-not-n100
- **Novelty**: 0.62
- **Evidence**: 0.88
- **Feasibility**: 0.95
- **Description**: The bootstrap clustering job should run on the RTX 3090 (Ollama host, 192.168.50.62) rather than on the CasaOS N100. cuBLAS SGEMM on a 3090 achieves ~35,580 GFLOPS (FP32) — approximately 327x the N100's theoretical peak. The same 92.54 GFLOP K-means workload completes in under 3 milliseconds on GPU vs 0.844 seconds on N100. Total job time shifts from compute-bound to network-transfer-bound: streaming 91.8 MB of embeddings over LAN (1 Gbps = ~0.73 seconds) exceeds the compute time, making the 3090 path network-latency-dominated rather than FLOP-dominated. Net result: GPU path is ~300x faster for the clustering step but the Neo4j write step (still on CasaOS) dominates regardless of where clustering runs. The GPU path is useful for K > 500 or N > 100K where clustering time becomes material.

### neo4j-parallel-cluster-writers
- **Novelty**: 0.65
- **Evidence**: 0.82
- **Feasibility**: 0.90
- **Description**: The bootstrap write step can be parallelized across 4 workers with zero lock contention by partitioning clusters across workers (worker 1 handles clusters 1-50, worker 2 handles 51-100, etc.). Within-cluster pairs write to disjoint node pairs, so no relationship-level conflicts exist. Neo4j Community Edition supports concurrent write transactions; the only shared resource is the transaction log. Four workers at 5,000 ops/s each = 20,000 ops/s effective throughput, reducing the 6.9-minute nominal case to 1.7 minutes. Implementation: a Python multiprocessing.Pool with 4 workers, each holding its own Neo4j driver session, iterating over its assigned cluster range. This is a 10-line architectural change that captures 4x of the 8.3x gap between nominal and physics minimum.

### two-tier-bootstrap-K-selection
- **Novelty**: 0.70
- **Evidence**: 0.78
- **Feasibility**: 0.88
- **Description**: The optimal K for bootstrap initialization differs from the optimal K for operational use. Bootstrap should use K=200-300 (larger clusters, coarser semantic groupings, higher initial edge density) while operational co-retrieval capture uses K=10-20 (fine-grained, per-query reinforcement). These two K values are independent: bootstrap creates the initial graph topology, and live reinforcement subsequently reshapes it by strengthening edges that are actually co-retrieved and allowing unreinforced edges to decay. Setting K=200 for bootstrap and K=15 for live capture is not a contradiction — it is a two-timescale design. The bootstrap topology provides the cold-start density; live capture provides the personalized refinement. This decomposition resolves the tension in Q163 between "K=200 for bootstrap coverage" and "K=10 for retrieval granularity."
