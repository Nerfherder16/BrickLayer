# Finding: Q045 — MBC On-Demand Retrieval via Compiler Scope Chain and DNS Zone Delegation

**Question**: Design the MBC on-demand retrieval path using ONLY structural session features — which files are open, which domains appeared in recent prompts, which tags appeared in the last 3 retrieved memories — without query embeddings, cosine similarity, or importance scores.
**Question Type**: TABOO
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Compiler design (lexical scoping, symbol table resolution), DNS protocol (zone delegation, authoritative record matching)

---

## Taboo Design: MBC On-Demand Retrieval Without Query Embeddings

### Physical requirements

- **Store**: N memories in a relational table. Each memory has: `tags[]` (string array), `access_count` (integer), `sessions_since_last_access` (integer), `memory_creation_domain` (string). No float fields, no vector columns, no LLM-assigned scores.
- **Session context**: At query time, the session state provides: `last_retrieved_tags[]` (tags from the 3 most recently LLPC-injected memories), `file_paths_open[]` (open editor files from Claude Code context), `recent_domains[]` (domain strings extracted from the last 3 user prompts by string matching against a known domain vocabulary).
- **Retrieve**: Return exactly K_mbc=3 memories per query event. Must be disjoint from the LLPC set (memories with `access_count >= 3 AND sessions_since_last_access <= 10` are LLPC territory; MBC excludes them).
- **Latency**: < 200ms p95. The query arrives after a user prompt; the MBC fetch must complete before response generation begins. This requires an index-friendly predicate (GIN index on `tags[]`), not a full-table scan.
- **Routing rule constraint**: Expressible as a SQL WHERE clause using only `ANY`, `@>`, `&&` string/array operators and integer comparisons. No float arithmetic. No subqueries that touch all N rows without index support.
- **Falsifiability**: The rule must produce a deterministic set that can be compared retrospectively against embedding-based retrieval on the same queries, using the audit log to reconstruct session context at query time.

---

### Analogous mechanism found in: Compiler design + DNS protocol

**Mechanism 1 — Lexical scoping in compiler symbol table resolution**

When a compiler resolves a symbol reference, it does not score the reference against all names in the symbol table and return the closest match. Instead, it walks a deterministic scope chain. The current lexical scope is searched first (local variables), then the enclosing scope (function parameters), then the module scope (file-level declarations), then the global scope (imported names). At each level, the match predicate is boolean: the name either exists in this scope or it does not. No partial-match score is computed. No floating-point distance is involved.

The scope chain is defined structurally. The file currently being compiled determines the module scope. The function containing the reference determines the local scope. A symbol from `docker_utils.py` is in scope for code in the same file; it is not in scope for unrelated files unless explicitly imported. Relevance is determined by structural membership, not semantic distance.

This mechanism maps directly onto MBC retrieval. The "symbol reference" is the current session query. The "scope chain" is a priority-ordered set of structural matching rules:

- **Innermost scope**: memories whose tags intersect with tags already proven relevant this session (tags from recently LLPC-injected memories — these have already passed the frequency gate and are the session's "local variables")
- **Enclosing scope**: memories whose tags intersect with domains explicitly mentioned in recent prompts (these are the "function parameters" — declared by the user's own words)
- **Module scope**: memories whose `memory_creation_domain` matches a domain in `recent_domains[]` even if tag intersection fails (these are "module-level declarations" — same broad domain but potentially different sub-topic)
- **Global scope**: frequency-ordered fallback for all non-LLPC memories (the "global namespace")

**Mechanism 2 — DNS zone delegation and authoritative record matching**

DNS resolves a hostname by walking a delegation chain of zones. To resolve `api.docker.internal`, the resolver queries the root zone (`.`), which delegates to `.internal`, which delegates to `docker.internal`, which returns the A record for `api.docker.internal`. Each zone is authoritative for a string-prefix-defined set of names. The match predicate at each level is string equality or string prefix containment — no semantic similarity.

DNS also defines **zone cuts**: the boundary between what one zone answers and what it delegates. A memory tagged `["docker", "networking"]` is in the `docker` zone and the `networking` zone. A query from a session where `docker` is in `recent_domains[]` should resolve to memories in the `docker` zone first, regardless of how many tags match — the zone cut defines which records are candidates before ordering begins.

DNS negative caching (NXDOMAIN TTL) maps onto the MBC's structural exclusion of LLPC memories: once a memory is confirmed to be in the LLPC tier (frequency-proven), it is "authoritatively answered" by the LLPC track and the MBC resolver does not re-query it. The exclusion is structural (NOT WHERE clause), not computed per-query.

**Why these two mechanisms together solve the structural routing problem:**

The compiler scope chain provides the **priority ordering** (most specific structural match first, most general last) and the **fallback chain** (try innermost scope before widening). The DNS zone delegation provides the **membership predicate** (is this memory in the active zone for this session?) expressed as string equality on tag values. Together they define a multi-level routing rule where every predicate is a boolean string operation, every ordering criterion is an integer comparison, and no floating-point distance is computed at any level.

The critical insight from lexical scoping: the correctness of symbol resolution does not require semantic understanding of what the symbol means. A compiler resolves `docker_network_create()` to the right function not because it understands networking but because the structural scope chain places the definition in the accessible scope. Similarly, MBC retrieval can surface the right memory not because it understands the query but because the tag intersection places the memory in the accessible scope of the session.

### Derived architecture

#### Session context extraction (pre-query, O(1) amortized)

Before any retrieval query, derive the session's structural fingerprint from three sources:

**Step A — File path tag extraction**: Map `file_paths_open[]` to tags via a static lookup table of path patterns. Examples:
- Path contains `docker-compose`, `Dockerfile`, `.docker/` → tag `docker`
- Path contains `.py`, `fastapi/`, `uvicorn`, `pydantic` → tags `python`, `fastapi`
- Path contains `package.json`, `.tsx`, `.ts`, `vite.config` → tags `typescript`, `react`
- Path contains `casaos`, `192.168.50` → tags `casaos`, `homelab`
- Path contains `docker-compose.yml` in a directory named after a project → also tag the project name

This lookup executes at session start once, not per query. Cost: O(F × P) where F = open files (~20) and P = pattern table size (~50). Sub-millisecond.

**Step B — Session tag accumulation**: The LLPC track already injects K_llpc=2 memories at session start. Their tags are available in session state. Take the union of all tag arrays from injected LLPC memories. This is the "innermost scope" — tags whose relevance has been structurally confirmed by the frequency gate.

**Step C — Prompt domain extraction**: The MBC query fires after a user prompt. Extract domain strings from the prompt text by matching against a domain vocabulary (a static list of ~100 technology/domain strings: `docker`, `python`, `fastapi`, `casaos`, `homelab`, `networking`, `typescript`, etc.). Store the last 3 extracted domain sets in session state as `recent_domains[]`.

The merged context tag set:

```
context_tags = UNIQUE(file_tags[] UNION last_retrieved_tags[] UNION recent_domains[])
```

This is computed once per query event (not once per session), incorporating the most recent prompt's domain. The session_tags component (Steps A and B) is cached and updated only when files open or LLPC injections occur.

#### The three-level MBC routing rule (SQL)

```sql
-- LEVEL 1: Tag scope match (compiler innermost scope)
-- Memories whose tags intersect with the session's context tag set
-- Ordered by intersection cardinality (most tags matched first), then access_count
-- Excludes LLPC track members

WITH context_tags AS (
  SELECT ARRAY[:context_tags]::text[] AS ctx
),
level1 AS (
  SELECT
    m.memory_id,
    cardinality(m.tags & (SELECT ctx FROM context_tags)) AS match_count
  FROM memories m
  WHERE m.tags && (SELECT ctx FROM context_tags)
    AND NOT (m.access_count >= 3 AND m.sessions_since_last_access <= 10)
  ORDER BY match_count DESC, m.access_count DESC
  LIMIT 3
),

-- LEVEL 2: Domain zone fallback (DNS zone delegation)
-- Memories whose creation domain matches any domain in recent_domains[]
-- Used only if Level 1 returns fewer than K_mbc=3 results
level2 AS (
  SELECT m.memory_id
  FROM memories m
  WHERE m.memory_creation_domain = ANY(ARRAY[:recent_domains]::text[])
    AND NOT (m.access_count >= 3 AND m.sessions_since_last_access <= 10)
    AND m.memory_id NOT IN (SELECT memory_id FROM level1)
  ORDER BY m.access_count DESC, m.sessions_since_last_access ASC
  LIMIT GREATEST(0, 3 - (SELECT COUNT(*)::int FROM level1))
),

-- LEVEL 3: Frequency floor fallback (global scope)
-- Fill remaining slots with highest-velocity non-LLPC memories
-- Velocity = access_count * 100 / sessions_since_last_access (integer, no floats)
level3 AS (
  SELECT m.memory_id
  FROM memories m
  WHERE NOT (m.access_count >= 3 AND m.sessions_since_last_access <= 10)
    AND m.access_count >= 1
    AND m.memory_id NOT IN (SELECT memory_id FROM level1)
    AND m.memory_id NOT IN (SELECT memory_id FROM level2)
  ORDER BY (m.access_count * 100 / GREATEST(m.sessions_since_last_access, 1)) DESC
  LIMIT GREATEST(0, 3 - (SELECT COUNT(*)::int FROM level1)
                      - (SELECT COUNT(*)::int FROM level2))
)

SELECT memory_id, 1 AS priority_level FROM level1
UNION ALL
SELECT memory_id, 2 AS priority_level FROM level2
UNION ALL
SELECT memory_id, 3 AS priority_level FROM level3;
```

The `priority_level` column is returned to the caller for observability — it records which routing level supplied each result, enabling the retrospective falsifiability comparison.

#### Constraint derivation for the WHERE predicates

**Level 1 — tag intersection predicate `m.tags && ctx`**:

The `&&` operator on PostgreSQL array columns is supported by a GIN index. `CREATE INDEX CONCURRENTLY idx_memories_tags ON memories USING GIN(tags)`. The query plan for Level 1 becomes: GIN index lookup on context_tags → candidate set of O(K) memories sharing at least one tag → sort candidate set by match_count. For a corpus of N=10,000 memories with a tag vocabulary of 200 distinct values and average 3 tags per memory, a context_tags set of 5 values will match approximately 5 × (10,000 / 200 × 3) = 750 memories at the 3-tags-per-memory distribution. The GIN index reduces the scan from 10,000 rows to ~750 rows for this example.

The ordering by `match_count DESC` implements the compiler scope chain priority: a memory matching 3 of the 5 context tags is more "in scope" than one matching only 1. This is the structural analog of a symbol being defined in the local scope (3-tag match = closer lexical scope) vs. being imported from a distant module (1-tag match = wider scope).

**Level 2 — domain zone predicate `m.memory_creation_domain = ANY(recent_domains)`**:

`memory_creation_domain` is a single string field. The predicate is an equality check against a small array (3 elements). A B-tree index on `memory_creation_domain` makes this O(log N × 3). Level 2 fires only when Level 1 has fewer than 3 results — in sessions with well-tagged memories, Level 2 is rarely invoked. In sessions early in a new domain (few memories yet carry the new domain's tags), Level 2 provides the DNS-style fallback: "no tag-level records, but the zone delegation says this domain is relevant."

**Level 3 — frequency floor `access_count * 100 / GREATEST(sessions_since_last_access, 1)`**:

Integer arithmetic only. The multiply-by-100 preserves two significant figures without floating point. This is the access velocity metric from Q042. Level 3 fires only when Levels 1 and 2 together return fewer than 3 results — typically only for new users with sparse corpora or new topic domains where neither tags nor domain strings match. It ensures K_mbc=3 is always satisfied.

#### Worked example with full trace

**Session state**: Files open include `docker-compose.yml` and `casaos/appdata/` paths. Last LLPC-injected memories had tags `["casaos", "networking"]` and `["docker", "ports"]`. Last 3 prompts mentioned `docker`, `networking`, `bridge`.

**context_tags derivation**:
- file_tags: `docker`, `casaos`
- last_retrieved_tags: `casaos`, `networking`, `docker`, `ports`
- recent_domains: `docker`, `networking`, `bridge`
- merged, deduplicated: `{docker, casaos, networking, ports, bridge}`

**Level 1 candidates** (memories whose tags intersect `{docker, casaos, networking, ports, bridge}`):
- Memory A: tags `["docker", "networking", "firewall"]` → match_count=2, access_count=7 → selected first
- Memory B: tags `["docker", "volumes"]` → match_count=1, access_count=12 → selected second
- Memory C: tags `["casaos", "ports", "jellyfin"]` → match_count=2, access_count=3 → but access_count=3 AND sessions_since_last_access=5 ≤ 10 → **excluded** (this memory is in the LLPC track)
- Memory D: tags `["networking", "vlan"]` → match_count=1, access_count=5 → selected third

Level 1 returns 3 results. Levels 2 and 3 are not invoked.

**MBC result set**: Memory A (priority_level=1, match_count=2), Memory B (priority_level=1, match_count=1, tiebroken by access_count=12), Memory D (priority_level=1, match_count=1, tiebroken by access_count=5).

#### Falsifiability test design

The retrospective comparison against embedding-based retrieval is constructed as follows:

1. **Extract test cases from audit log**: For each retrieve event in Recall's audit log (PostgreSQL table `retrieve_events`), reconstruct the session context at that moment: which memories were in the LLPC set, which files were open (from Claude Code session metadata if logged), which domains appeared in the prior 3 prompts (from session_summary records).

2. **Apply the structural routing rule**: For each test case, run the Level 1 → Level 2 → Level 3 query against the memory corpus state at that time. Produce a set of 3 memory IDs with their priority_level assignments.

3. **Apply the embedding-based baseline**: Run the current production retrieval (embedding distance ranked top-3) for the same query text against the same corpus state.

4. **Measure overlap**: Count how many of the structural MBC set's 3 memory IDs appear in the embedding-based top-3.

5. **Stratify by priority level**: Report overlap separately for cases where Level 1 filled all 3 slots vs. cases requiring Level 2 or Level 3 fallback.

**Expected outcomes from the structural analysis**:

- Level 1 fills all 3 slots: expected 55-70% overlap with embedding-based top-3. The mechanism: in a focused coding session, the context_tags set directly reflects the semantic domain of the session. Tags were assigned at memory write time to capture the domain. The embedding-based retrieval is responding to the same domain signal, just through a different representation. When both systems operate on the same domain, they should converge on similar memories.

- Level 2 required: expected 35-50% overlap (lower, because domain-only matching misses sub-topic specificity that tag intersection would capture).

- Level 3 required: expected 20-35% overlap (frequency ordering has no domain signal at all; overlap is incidental).

**The falsifiable prediction**: The structural routing rule achieves ≥ 55% overlap with embedding-based retrieval in sessions where Level 1 supplies all 3 MBC slots. If the retrospective test shows < 40% overlap for Level 1-only sessions, the tag vocabulary is too coarse (many semantically distinct memories share the same broad tags) and the tag assignment protocol at write time needs refinement, not the routing rule.

## Ideas Extracted

### Idea: compiler-scope-chain-mbc-routing
- **Novelty**: 0.90 — No production AI memory system uses a multi-level scope chain inspired by compiler lexical scoping to route on-demand memory retrieval. Existing on-demand retrieval systems either use embedding distance (which is forbidden in this design constraint) or simple keyword matching (which is a degraded single-level version of this idea without the priority chain or the fallback structure). The specific three-level design — tag intersection → domain zone match → frequency floor — mirrors the compiler scope chain structure and has not appeared in AI memory literature.
- **Evidence**: 0.85 — Lexical scoping and symbol table resolution are foundational compiler theory, described in every compiler textbook (Aho, Lam, Sethi, Ullman "Compilers: Principles, Techniques, and Tools", 2nd ed., 2006, Chapter 8). The scope chain priority ordering (innermost first, outermost last) is implemented in every production programming language. DNS zone delegation is an IETF standard (RFC 1034, RFC 1035, 1987, Mockapetris) with 40 years of production deployment. Both mechanisms are well-validated for structural, non-semantic name resolution. The application to memory routing is engineering application of established mechanisms.
- **Feasibility**: 0.85 — Implementation requires: (1) a GIN index on the `tags[]` column (one SQL statement, no schema change to memory rows), (2) a static file-path-to-tag mapping table (~50 entries, maintained as a JSON config file), (3) session state extension to accumulate `last_retrieved_tags[]` and `recent_domains[]` (already partially tracked in session summary hooks), (4) the three-level SQL query shown above. No new infrastructure. No new services. The GIN index creation is concurrent-safe and takes minutes on an existing table. Estimated implementation: 1-2 weeks including the session state instrumentation.
- **Description**: At query time, derive a `context_tags[]` set from the union of: tags from LLPC-injected memories this session, tags inferred from open file paths via a static lookup table, and domain strings extracted from the last 3 user prompts. Run a three-level routing rule: Level 1 selects memories whose tags intersect context_tags (ordered by intersection cardinality), Level 2 falls back to memories whose `memory_creation_domain` matches `recent_domains[]`, Level 3 falls back to frequency-ordered non-LLPC memories. The rule is a deterministic SQL query using only `&&`, `ANY`, and integer comparisons. Returns K_mbc=3 results. GIN index on `tags[]` makes Level 1 O(log N + K) rather than O(N). Each result carries a `priority_level` field (1/2/3) for observability.

### Idea: tag-intersection-cardinality-ordering
- **Novelty**: 0.80 — Within Level 1 results, ordering by the cardinality of the tag intersection (how many tags match, not just whether any match) is a structural priority signal that directly implements compiler scope distance. A memory sharing 3 of the session's 5 context tags is structurally "closer" to the session than one sharing only 1. This ordering criterion requires no float arithmetic — it is `array_length(m.tags & ctx_tags, 1)`, an integer count. Existing systems that use tag matching typically treat it as a binary filter (match or no match) rather than using intersection cardinality as an ordering key.
- **Evidence**: 0.75 — Intersection cardinality as a ranking signal is established in information retrieval as term frequency (TF) — a document with more matching query terms ranks higher than one with fewer. The difference here is that both the query (context_tags) and the index (memory tags) are controlled vocabularies, so the cardinality measure is more precise than unstructured TF. The mechanism is also analogous to compiler scope depth ordering: a symbol found at depth 1 in the scope chain is ranked before one found at depth 3. Well-established in compiler theory; the specific application to memory tag ordering is engineering application.
- **Feasibility**: 0.90 — The PostgreSQL `&` operator on arrays returns the intersection array. `cardinality()` returns its length as an integer. `array_length(m.tags & ctx_tags, 1)` is a native PostgreSQL expression, no extension needed. The GIN index on `tags[]` already supports the `&&` filter; the cardinality computation runs on the filtered candidate set only (not all N memories). Schema change: none beyond the GIN index.
- **Description**: Within the Level 1 MBC result set (memories passing the `tags && context_tags` filter), order by `cardinality(tags & context_tags) DESC` before ordering by `access_count DESC`. A memory tagged `["docker", "networking", "firewall"]` in a session with context_tags `["docker", "networking", "bridge"]` scores intersection cardinality 2 and ranks above a memory tagged `["docker", "volumes"]` (cardinality 1) even if the latter has higher `access_count`. This implements the compiler scope-distance principle: structural proximity to the current session's topic focus determines priority, with access frequency as a tiebreaker among equally-proximate memories. The ordering is a single integer column in the ORDER BY clause.

### Idea: file-path-tag-inference-static-map
- **Novelty**: 0.75 — The static file-path-to-tag mapping table (path pattern → technology tag) as a first-class component of the MBC routing pipeline is not described in AI memory systems. Existing systems that use file context (e.g., IDE plugins that augment code completion with recent file history) do not maintain a named-tag vocabulary derived from file paths, and do not use that vocabulary as a retrieval filter. The mechanism is analogous to compiler build system "include path" rules — the path tells you what namespace the file belongs to without parsing the file's contents.
- **Evidence**: 0.70 — Static path-pattern-to-namespace mapping is used extensively in build systems (Make, CMake, Gradle — path patterns determine which build rules apply), IDE language servers (which language server to activate for a `.py` vs. `.ts` file is determined by file extension pattern matching), and web server routing (nginx `location` blocks are path-pattern-to-handler mappings). The specific application to memory retrieval context extraction is engineering application of a well-understood pattern-matching mechanism.
- **Feasibility**: 0.95 — The static lookup table is a JSON config file with ~50 entries. The matching logic is 10-20 lines of string operations. The result is a `file_tags[]` array that is cached per session and updated only when files open or close. No new infrastructure, no database changes, no external services. The config file can be updated by the user or auto-extended when new technology domains are encountered. Implementation: 1-3 days.
- **Description**: Maintain a static JSON config file (`file-tag-map.json`) that maps file path patterns to technology domain tags. Patterns are matched using simple string containment or suffix matching: `"docker-compose"` → `"docker"`, `".tsx"` → `["typescript", "react"]`, `"fastapi/"` → `["python", "fastapi"]`. At session start, apply this map to `file_paths_open[]` to derive `file_tags[]`. Merge `file_tags[]` into `context_tags[]` before any MBC query. The map is a plain configuration file, not a learned model. It can be extended by pattern: when the user works on a new technology domain, add one line to the map. The tags derived from file paths are structurally independent of what the user says in prompts — they represent the "ambient context" of the development environment rather than the explicitly stated intent.

### Idea: priority-level-observability-field
- **Novelty**: 0.72 — Returning the routing level (1, 2, or 3) that supplied each MBC result as a first-class field in the retrieval response is an observability mechanism not present in any described AI memory system. It creates a natural audit trail for measuring routing quality: if Level 3 (frequency fallback) is supplying most results, the tag vocabulary is too sparse or misaligned with session context, indicating a need for better tag assignment at write time. This field transforms the routing rule from a black box into a self-diagnosing system.
- **Evidence**: 0.65 — Priority-level annotation is standard in DNS responses (the answer section indicates whether the record came from authoritative vs. cached vs. recursive resolution). Network routing protocols return path attributes that indicate which routing policy (OSPF, BGP, static) contributed each route. In compiler diagnostics, the source of a symbol resolution (local, module, imported) is tracked for error messages and refactoring tools. The application to memory retrieval observability is direct engineering application of diagnostic-layer thinking from network and compiler systems.
- **Feasibility**: 0.95 — The `priority_level` field is already in the SQL query shown above. No additional computation. The retrieve endpoint returns this field alongside `memory_id` and the memory content. The dashboard can display it as a label on each retrieved memory: "Matched by tag intersection (level 1)" vs. "Matched by domain (level 2)" vs. "Frequency fallback (level 3)". Aggregated over sessions, a histogram of priority_level distribution reveals tag vocabulary health. Implementation: 0 days for the field itself (it is already in the query); 1-2 days for dashboard display.
- **Description**: Each memory returned by the MBC routing query carries a `priority_level` integer (1, 2, or 3) indicating which routing level selected it. Level 1 means the memory's tags intersected the session's context_tags set; Level 2 means the memory's creation domain matched a recent prompt domain; Level 3 means neither matched and the memory was selected by frequency ordering alone. This field is logged per retrieve event. Aggregate metrics over N sessions: fraction of MBC results from Level 1 (target: > 60%), from Level 2 (target: 20-30%), from Level 3 (target: < 20%). If Level 3 fraction rises above 30% over a rolling 7-day window, the system flags: "tag vocabulary coverage degraded — recent sessions are hitting frequency fallback frequently." This turns a retrieval quality concern into a measurable, actionable signal without any LLM involvement.

### Idea: context-tags-session-accumulation
- **Novelty**: 0.78 — The session-scoped accumulation of context_tags (growing the set as LLPC memories are injected and as prompts arrive, rather than recomputing from scratch per query) is a stateful context-building mechanism analogous to the compiler's incremental symbol table construction during parsing. As the compiler parses through a file, it adds symbols to the table; later references can resolve to earlier definitions. As the session progresses, the context_tags set grows to encode the session's accumulated topic history, making later MBC queries in the same session structurally more informed than the first query.
- **Evidence**: 0.70 — Incremental symbol table construction is foundational compiler theory (Aho et al., 2006, Chapter 2: each declaration adds a symbol; each scope entry/exit updates the chain). DNS resolvers maintain a session-scoped cache that accumulates resolved records during a connection. The TCP/IP ESTABLISHED state's routing cache accumulates next-hop decisions. All are examples of session-scoped state accumulation that makes later operations in the same session cheaper and more accurate than earlier ones. The application to memory retrieval context is engineering application of incremental state accumulation.
- **Feasibility**: 0.85 — The session state object already exists (used by the LLPC track for `sessions_since_last_access` updates and by the session-end hook). Extend the session state to include a `context_tags` array initialized at session start from file_tags, updated when LLPC memories are injected (add their tags to context_tags), updated when a prompt arrives (add domains extracted from the prompt to context_tags). The context_tags set only grows within a session; it is discarded at session end. Maximum size is bounded by the number of LLPC injections (2 memories × avg 3 tags = 6 tags) plus prompts (bounded by session length × tags per prompt). Cap at 20 distinct tags to prevent context_tags bloat in very long sessions.
- **Description**: Initialize `context_tags = file_tags[]` at session start. After each LLPC injection, extend: `context_tags = UNIQUE(context_tags UNION injected_memory.tags)`. After each user prompt, extend: `context_tags = UNIQUE(context_tags UNION extracted_domains(prompt))`. Use this accumulated set for each MBC query during the session. A session that starts working on Docker networking and then shifts to a FastAPI debugging problem will have context_tags that include both `docker`, `networking` (from early LLPC injections) and `python`, `fastapi` (from later prompt domains). This multi-domain context_tags set enables MBC Level 1 to match memories relevant to either the session's start context or its current focus, without discarding the accumulated history.
