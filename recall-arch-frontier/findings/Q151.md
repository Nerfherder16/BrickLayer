## Finding

**OBJECTIVE**: Determine if `session_archive` has a retention policy and whether a PG restore would create ghost references.

**DATA**:
1. Schema has no TTL column, no foreign keys to Qdrant/Neo4j memory IDs, no cascade rules.
2. Zero DELETE/TRUNCATE statements exist for `session_archive` in the entire codebase.
3. No scheduled worker targets `session_archive` for cleanup.
4. `session_archive` stores aggregate session metadata (counts of memories created/retrieved, not memory IDs themselves). It does not store pointers to individual memory IDs in Qdrant or Neo4j.
5. `audit_log.session_id` is a TEXT field with no FK constraint back to `session_archive.session_id`.

**FINDING**: `session_archive` grows unboundedly. There is no retention policy, TTL, or cleanup job. However, the severity is low because:

(a) **No ghost references to memories**: `session_archive` stores only aggregate counters (`memories_created`, `memories_retrieved`, `signals_detected`, `turns_count`) — not individual memory IDs. A PG restore would not create dangling foreign key references to Qdrant/Neo4j because no such references exist in the schema. The table is a metrics/ops log, not a join table.

(b) **Storage growth is manageable**: At ~1 row per session and small row size (~300 bytes per row including overhead), even 10,000 sessions = ~3 MB. Not a practical concern on any reasonable timeline.

(c) **The real unbounded growth risk is `audit_log`**: At 400,377 rows (already known from Q132) and ~18 writes per memory operation, `audit_log` grows 18x faster than memories and has no retention policy either. This is the more significant unbounded table.

**STAT**: `session_archive` row count is not exposed via public API. Estimated count: 700-3,000 rows based on project age and usage patterns. Exact count requires direct DB access or authenticated `/stats` call.

## Qualifying Ideas

**Idea 1 (N=0.72, E=0.55)**: Add a retention policy to `session_archive` — e.g., keep only sessions from the last 365 days. This is low-risk since the table is pure ops metadata with no foreign key consumers. A `DELETE FROM session_archive WHERE archived_at < now() - interval '365 days'` run monthly would bound the table. However, given the low growth rate (~1 row/session), this is cosmetic rather than necessary.

**Idea 2 (N=0.68, E=0.40)**: Expose `session_archive` count via the `/health` endpoint alongside `audit_log`. Currently only audit entries are counted. Adding session count would confirm the actual row count without requiring auth and improve operational visibility.

**Idea 3 (N=0.65, E=0.48)**: The deeper risk is `audit_log` unbounded growth at 400,377 rows and growing at 18x the memory creation rate (~18 audit rows per memory event). A retention policy for `audit_log` (e.g., keep 90 days of detail, archive older rows to a summary table) would have meaningfully more impact than any `session_archive` policy. This is the actionable finding from this investigation.
