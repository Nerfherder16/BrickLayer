# Finding: Q037 — Cluster-Manifest Stub: False Absence (Partial Production Match)

**Question**: Has any production retrieval or storage system implemented a lightweight cluster-manifest stub — a centroid vector plus member-ID list stored in a fast tier to represent a full cluster in a slow tier — as its primary cross-tier discoverability mechanism?
**Agent**: quantitative-analyst
**Question Type**: ABSENCE
**Verdict**: PROMISING
**Severity**: High
**Source field**: distributed storage systems, vector databases

## Mechanism Under Investigation

A cluster-manifest stub is: a centroid embedding PLUS a member-ID list stored together in a fast/warm tier, such that a query can match the centroid above a threshold to identify the cluster, then use the member-ID list to selectively lazy-load only the relevant records from a slow/cold tier — without loading any slow-tier data during the discovery phase.

The critical element is the **member-ID list in the fast tier**. The stub must answer two questions from fast storage alone: (1) is this cluster relevant to the query, and (2) which specific records would we fetch if we do load?

## Evidence

### Falsifying Condition 1: HNSW Upper Layers

HNSW upper layers store **graph nodes that happen to be sampled representatives**, not semantic centroids of clusters. The upper layers are a navigational shortcut, not an index of cluster membership. There is no posting list of member IDs at any HNSW layer. Qdrant's 1.16 release introduced "inline storage" that places vector data inside HNSW nodes on disk, and allows HNSW itself to be stored on disk — but this is a monolithic index that does not separate centroid-fast / member-slow. **Not a match.**

### Falsifying Condition 2: Elasticsearch ILM (Hot/Warm/Cold/Frozen)

ILM moves entire indices across tiers based on age/size policies. The hot tier has no centroid stub representing a frozen-tier index. Discoverability of a frozen index goes through cluster metadata and shard routing tables — not through a semantic embedding lookup. The tier transition is index-level, not cluster-level, and carries no embedding-based routing signal. **Not a match.**

### Falsifying Condition 3: Parquet / Delta Lake Partition Statistics

Partition zone maps store min/max column values for predicate pushdown. These are scalar range statistics, not vector embeddings. There is no production system (as of March 2026) that stores centroid embeddings in a Parquet/Delta Lake manifest for semantic routing. The closest adjacent work is geospatial Parquet (MBRs for spatial skip), but this is geometric bounding, not embedding-space centroid routing. **Not a match.**

### Falsifying Condition 4: FAISS IVF Coarse Quantizer

FAISS IVF is the closest structural analogue. The coarse quantizer holds centroid embeddings in memory, and posting lists (containing member IDs and compressed vectors) live on disk via `IndexIVFOnDisk`. During search, the query is matched to the nearest k centroids in memory, and then the corresponding disk posting lists are memory-mapped and scanned.

This is a **partial match**. What FAISS IVF stores in memory: centroid embeddings only. What lives on disk: member IDs + vector codes in posting lists. The fast-tier index does NOT hold a member-ID list — only the centroid vector. The posting list (member IDs) is a disk-side structure.

The Q037 mechanism differs in one key way: the stub in the fast tier would include the member-ID list so that the caller can reason about "which specific records would I pull" before committing to the disk fetch. FAISS IVF cannot do this — the fast tier is purely a routing gate, not a manifest. **Structural ancestor, not a match for the full pattern.**

### Falsifying Condition 4b: SPANN (Production at Microsoft Bing)

SPANN (NeurIPS 2021, deployed to Microsoft Bing at hundreds-of-billions scale) implements the closest known production approximation:

- **Fast tier (SPTAG in-memory index)**: centroid embeddings for all posting lists, indexed with a space-partition tree + neighborhood graph.
- **Slow tier (SSD)**: posting lists containing the actual member vectors and their IDs.
- **Mechanism**: query hits the in-memory centroid index, retrieves the k nearest centroids, then fetches only those posting lists from SSD.

SPANN stores **centroid vectors** in the fast tier — not member-ID lists. The fast tier answers "which posting list?" but cannot answer "which specific member IDs are in that posting list?" without a disk read. The member-ID list is fully disk-resident.

**Verdict on SPANN vs Q037**: SPANN implements centroid-in-fast-tier routing (the discovery half of the pattern), but does NOT store a member-ID manifest in the fast tier (the lightweight-stub half). SPANN's fast tier is purely a directional pointer, not a queryable list of members. **Partial match — the routing mechanism exists in production, but the manifest element does not.**

### Falsifying Condition 4c: Vespa Hybrid HNSW-IF

Vespa's billion-scale HNSW-IF system uses a two-cluster deployment:
- **Graph cluster (high-memory)**: 20% of vectors designated as centroids, stored in-memory HNSW.
- **IF cluster (low-memory SSD)**: remaining 80% of vectors in inverted index posting lists, with centroid-ID → vector-ID mappings on disk.

Vespa stores centroid embeddings in memory and non-centroid posting lists on disk. The inverted index maps centroid IDs to lists of non-centroid IDs — but this mapping is disk-side, not in the fast tier. The fast tier holds only the HNSW graph of centroids. **Same pattern as SPANN — partial match, no fast-tier member-ID manifest.**

### Falsifying Condition 5: AI Memory Systems (MemGPT, mem0, Zep, LangChain, Chroma, Weaviate)

None of these systems implement cluster-centroid-based cross-tier routing. They use flat vector similarity search (cosine/dot product) against all stored memories, sometimes with metadata filters. Zep uses a temporal knowledge graph with three hierarchical subgraphs and a node-distance reranker, but this is graph-distance reranking within a unified store — not centroid-based cross-tier discoverability. Chroma, Weaviate, and LangChain Memory do not implement any form of cluster-manifest tiering. **Not a match.**

## Absence Classification

**Classification**: PARTIAL ABSENCE

**Explanation**: The routing half of the Q037 mechanism (centroid embeddings in a fast tier that direct queries to a slow-tier posting list) is implemented in production at scale — SPANN at Microsoft Bing, Vespa HNSW-IF, and FAISS IVF on-disk all do this. What does NOT exist in any production system found is the second half: **storing the member-ID list alongside the centroid in the fast tier as a lightweight manifest stub**. Every production system pushes the member-ID list to the slow tier. The fast tier is a pure routing gate (pointer), not a manifest (queryable inventory). This distinction is meaningful for Recall's use case: the manifest enables decisions like "skip loading this cluster because I see its member IDs and none are relevant" — a pre-fetch filter that current production systems cannot do.

**Why the gap exists**: The member-ID list in the fast tier adds memory cost that current billion-scale systems cannot afford (20% centroids in memory is already the limit in Vespa's deployment). At billion scale, adding even a 32-byte manifest stub per member would double fast-tier memory. For AI memory systems at Recall's scale (thousands to low millions of memories), this tradeoff is entirely different: a manifest stub of 8–16 bytes per memory ID is trivially affordable in Redis, and it unlocks pre-fetch filtering before any slow-tier I/O.

## Ideas Extracted

### Idea: recall-cluster-manifest-stub
- **Novelty**: 0.82 — The routing half (centroid-in-fast-tier) exists in production. The manifest half (member-ID list in fast tier alongside centroid) does not appear in any production system found. The combined pattern as a lightweight discoverability stub is novel.
- **Evidence**: 0.72 — SPANN (NeurIPS 2021, Microsoft Bing production) proves the centroid-routing architecture is sound at scale. The manifest extension is a straightforward structural addition, not a speculative mechanism. The information-theoretic case for pre-fetch filtering via ID manifest is clear.
- **Feasibility**: 0.78 — Recall already has Redis (fast tier) and Qdrant (slow tier). Implementation: (1) on memory write, compute k-means centroid for the cluster (or assign to nearest existing centroid), (2) store {centroid_vector, [member_ids]} as a Redis hash keyed by cluster_id, (3) on query, embed query, scan Redis centroid keys with cosine similarity above threshold, (4) collect member IDs from matching stubs, (5) fetch only those specific vectors from Qdrant by ID. Estimated 2–3 weeks of engineering, no new infrastructure.
- **Description**: Redis stores a small set of cluster centroids (one per ~50 memories, ~200 clusters for 10K memories). Each cluster key holds the centroid embedding (float32 array, ~3KB for 768-dim) plus a list of member memory-IDs (~400 bytes for 50 IDs at 8 bytes each). At query time, Redis scans ~200 centroids (fast — all in memory), returns matching cluster IDs and their member lists, then Qdrant does a targeted `get_by_ids` on only those members. This bypasses Qdrant's HNSW search entirely for cluster-routed retrieval, and pre-filters at the ID level before any vector I/O. The stub can also answer "how many memories are in this cluster" for cluster-size-weighted scoring without a Qdrant call.

### Idea: recall-spann-routing-port
- **Novelty**: 0.65 — Directly porting the SPANN routing pattern (centroid-in-fast-tier as a pointer to slow-tier data) to Recall's two-tier architecture. Lower novelty because SPANN is published production work.
- **Evidence**: 0.80 — SPANN is peer-reviewed (NeurIPS 2021) and deployed at production scale. High confidence the routing mechanism works.
- **Feasibility**: 0.72 — Simpler than the full manifest stub — only centroid vectors in Redis, no member-ID lists. On query match, issue a Qdrant filter query for `cluster_id == X`. Relies on Qdrant payload filtering rather than ID-level lookup. ~1–2 weeks.
- **Description**: Redis holds centroid embeddings only (no member IDs). Query matches centroid above threshold, then Qdrant runs `filter: {must: [{key: "cluster_id", match: {value: X}}]}` to retrieve members. Simpler than the manifest stub, but incurs Qdrant filter scan cost rather than O(1) ID lookup. Lower pre-fetch filtering capability — cannot reject a cluster by member identity before the Qdrant call.

## Suggested Follow-ups

- **Q038**: What is the optimal cluster granularity for Recall's manifest stub — how many memories per cluster balances fast-tier memory cost against Qdrant round-trip savings? (Simulate at N=1K, 10K, 100K memories)
- **Q039**: Does Qdrant's `get_by_ids` batch call have meaningfully lower latency than an HNSW search with the equivalent recall? If the manifest routes to the right IDs, is the ID-fetch actually faster end-to-end than a standard similarity search?
- **Q040**: Can the centroid be computed incrementally (online k-means) as memories are added, or does Recall need offline recomputation? What is the centroid drift rate as new memories arrive?
- **Q041**: Should the cluster manifest in Redis use approximate centroid matching (scan all centroids) or a secondary index structure (HNSW over centroids in Redis)? At what N does a flat centroid scan become the bottleneck?
- **Q042**: SPANN uses hierarchical balanced clustering to equalize posting list lengths. Does Recall need this, or is naive k-means sufficient given the much smaller scale and the ability to rebalance offline?
