# Finding: Q107 — Full LLPC Retrieval Pipeline Roofline

**Question**: What is the theoretical minimum latency roofline for a full LLPC memory retrieval pipeline on Tim's hardware — covering all five phases (PostgreSQL tag lookup, Redis epoch filter, Qdrant ANN search, Neo4j BFS traversal, token injection assembly) with their arithmetic intensities, memory bandwidths, and parallelization opportunities — and what is the maximum retrieval throughput (requests/second) before any single phase becomes the bottleneck?
**Question Type**: PHYSICS
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Roofline model (Williams et al. 2009), memory hierarchy latency analysis, pipeline critical path theory, GPU embedding inference (Q095), Neo4j BFS cost model (Q091), tiered injection token budget (Q097)

### Per-Phase Analysis

#### Phase 1 — PostgreSQL Tag Lookup

**Operation**: SELECT memories WHERE tags match session context → returns content_hashes; GIN index on tags, ~10k rows.

**Data volume**:
- Heap: 10,000 rows × ~200 bytes = ~2.0 MB
- GIN index: 10,000 rows × 3 tags × 20 bytes = ~600 KB
- Query touches: GIN B-tree probe (3 levels = 3 × 8 KB pages) + heap fetch (~50 pages for 100–500 matching rows) = 53 pages total cold

**Arithmetic intensity**: ~0.01 FLOPs/byte — no computation, pure buffer read. Deep in memory-bandwidth territory (ridge point = 3.6 FLOPs/byte).

**Theoretical minimum** (warm, data in PostgreSQL shared_buffers / OS page cache):
```
Minimum bytes: GIN posting list (100 TIDs × 6 bytes) + heap tuples (100 rows × 200 bytes) = 20.6 KB
Time at DDR4 bandwidth (40 GB/s): 20,600 / 40e9 × 1000 = 0.0005 ms
```

**Implementation cost (P50 warm)**:
- GIN B-tree probe (3 pages in shared_buffers): 3 × 60 ns = 0.18 µs
- Heap page fetches (50 pages): 50 × 5 × 60 ns = 15 µs
- Query parse, plan, socket, result serialize overhead: ~0.30 ms
- **P50 warm: ~0.32 ms**
- P50 cold (NVMe): 53 pages × 0.05 ms + 0.30 ms = **2.95 ms**

| Resource | Minimum | Calculation |
|---|---|---|
| Latency (warm) | 0.0005 ms | 20.6 KB / 40 GB/s |
| Latency (cold) | 2.65 ms | 53 pages × 0.05 ms/page |
| Bandwidth consumed | 20.6 KB | posting list + matching heap tuples |
| Compute | ~0 FLOPS | comparison, no arithmetic |

**Gap ratio (warm)**: 0.32 ms / 0.0005 ms = **640×** — gap is entirely software overhead (query parsing, PostgreSQL executor, socket IPC).

#### Phase 2 — Redis Epoch Filter

**Operation**: GET recall:global_epoch (1 Redis call); Python compares last_access_epoch per candidate memory against `current_epoch - tau`; runs in-process after PG returns results. No per-memory Redis call.

**Data volume**:
- 1 Redis GET: 1 key × ~8 bytes
- Python comparisons: ~100 candidate memories × 8 bytes = 800 bytes

**Arithmetic intensity**: ~0.01 FLOPs/byte (integer subtraction and comparison). Effectively zero compute.

**Theoretical minimum**:
```
Python integer comparisons: 100 × 5 ns = 0.5 µs
```

**Implementation cost (P50)**:
- Redis GET (localhost Docker): ~100 µs
- Python loop over 100 candidates: ~0.5 µs
- **P50: ~0.10 ms** (dominated by Redis TCP socket latency, not computation)

| Resource | Minimum | Calculation |
|---|---|---|
| Latency | 0.0005 ms | 100 × 5 ns CPU |
| Bandwidth consumed | 800 bytes | 100 epoch values |
| Compute | ~200 integer ops | subtract + compare per candidate |

**Gap ratio**: 0.10 ms / 0.0005 ms = **200×** — gap is pure Docker network socket roundtrip overhead. This phase could be eliminated entirely (see Ideas).

#### Phase 3 — Qdrant ANN Search

**Operation**: query_points with 1536-dim float32 session embedding, K=38, in-memory HNSW index, 360 total points.

**Data volume**:
```
Total vector store: 360 × 1536 × 4 bytes = 2.21 MB
HNSW ef=100: visits 100 candidate nodes = 27.8% of corpus
Bytes read for distance computations: 100 × 1536 × 4 = 600 KB
```

**Arithmetic intensity**:
```
FLOPs per comparison: 1536 × 2 (multiply + add) = 3,072 FLOPs
Bytes per comparison: 1536 × 4 = 6,144 bytes
Arithmetic intensity: 3,072 / 6,144 = 0.50 FLOPs/byte
N100 ridge point: 3.6 FLOPs/byte → MEMORY-BANDWIDTH BOUND
```

**Theoretical minimum** (HNSW ef=100, warm in-memory):
```
Compute time: 307,200 FLOPs / 109e9 FLOPS = 2.8 µs
Memory BW time: 600 KB / 30 GB/s = 20.5 µs
Bottleneck: memory bandwidth → 20.5 µs = 0.021 ms
```

Note: At N=360, brute-force scan (2.21 MB / 30 GB/s = 73.7 µs = 0.074 ms) is only ~3.6× the HNSW minimum. HNSW provides negligible algorithmic benefit at this corpus size — the graph navigation overhead exceeds the savings.

**Implementation cost (P50)**:
- Qdrant in-memory HNSW search at 360 points is dominated by software overhead: Python → gRPC/HTTP socket → Qdrant process → Rust HNSW → result serialization
- **P50: ~1.0 ms** (Qdrant published benchmarks at tiny scale: 0.5–2 ms for sub-1000 point collections)

| Resource | Minimum | Calculation |
|---|---|---|
| Latency | 0.021 ms | 600 KB / 30 GB/s |
| Bandwidth consumed | 600 KB | 100 visited vectors × 6 KB each |
| Compute | 0.307 MFLOPs | 100 × 1536 × 2 |

**Gap ratio**: 1.0 ms / 0.021 ms = **~49×** — gap is IPC overhead (Python → Qdrant process), not data movement.

**Important note on embedding precondition**: Qdrant search requires the session context embedding vector. This embedding is generated by qwen3-embedding:0.6b on the Ollama machine (192.168.50.62 over LAN):

```
Embedding computation:
  Model size: 0.6B params × 2 bytes (FP16) = 1.2 GB
  Bandwidth time: 1.2 GB / 936 GB/s = 1.28 ms
  Prefill (500 tokens): 2 × 0.6B × 500 FLOPs / 71.16 TFLOPS = 8.4 ms
  Bottleneck: prefill is compute-bound, dominates → ~8.4 ms compute-bound minimum
  Actual P50 (with LAN RTT + overhead): ~1.68 ms total (short prompts are BW-bound)

LAN network transfer of result vector:
  6,144 bytes / 125 MB/s = 49 µs ≪ RTT
  LAN RTT: ~0.3 ms
```

**If embedding must be generated on the critical path, Qdrant cannot start until embedding completes.**
Embedding P50 estimate: **1.68 ms** (bandwidth-bound for short prompt; compute would dominate at >200 output tokens).
Embed + Qdrant chain: 1.68 ms + 1.0 ms = **2.68 ms**.

#### Phase 4 — Neo4j BFS Traversal

**Operation**: MATCH (seed)-[:CO_RETRIEVED_FRESH*1..2]->(neighbor) WHERE seed.content_hash IN {K=38 results}; batch UNWIND. Warm cache = 1.1 ms (Q091).

**Data volume**:
```
Graph: ~800 edges × ~200 bytes = 160 KB (easily fits in 2 GB Neo4j page cache)
BFS from K=38 seeds, F_fresh avg ~2 edges/seed (CO_RETRIEVED_FRESH):
  Edge records: 38 × 2 × 34 bytes = 2,584 bytes
  Node records: (38 + 76) neighbors × 200 bytes = 22,800 bytes
  Total minimum read: ~25 KB
```

**Arithmetic intensity**: ~0.05 FLOPs/byte (property comparison for epoch predicates). Memory-bandwidth bound at all scales.

**Theoretical minimum** (warm JVM heap):
```
25 KB / 20 GB/s (JVM heap bandwidth) = 1.3 µs
```

**Implementation cost (P50)**:
- Q091 established 1.1 ms for a single-seed warm BFS
- K=38 seeds via batch UNWIND (one Cypher query): estimated **~2.2 ms** (2.0× single-seed, not 38×, because UNWIND amortizes Cypher parsing/planning overhead)
- Cold (page cache miss, NVMe read): **50 ms** (Q091)

| Resource | Minimum | Calculation |
|---|---|---|
| Latency (warm) | 0.0013 ms | 25 KB / 20 GB/s |
| Latency (cold) | 8–50 ms | NVMe random reads for uncached pages |
| Bandwidth consumed | 25 KB | edge + node records for fresh subgraph only |
| Compute | ~1,000 integer ops | epoch predicate evaluation |

**Gap ratio (warm)**: 2.2 ms / 0.0013 ms = **~1,700×** — dominated by JVM overhead (GC pauses, Cypher query compilation, Bolt protocol serialization, Java object graph traversal). Architecture gap from Q091: CO_RETRIEVED_FRESH type partitioning reduces this to ~1.6 ms per Q091's speedup analysis.

#### Phase 5 — Token Injection Assembly

**Operation**: Python: sort K=38 results by score, apply tiered injection (top-3 full text ~350 tokens, 35 headlines ~35 tokens), format into Claude context string. Output ~2,030 tokens (from Q097: K=38 tiered = 2,030 tokens).

**Data volume**:
```
Input: 38 memories × ~200 bytes metadata = 7.6 KB
String assembly: 3 × 350 × 4 bytes (full text) + 35 × 35 × 4 bytes (headlines) = 4.1 KB + 4.9 KB = 9.0 KB
Sort: K=38 × log2(38) = ~199 comparisons (float64 scores)
```

**Arithmetic intensity**: ~0.02 FLOPs/byte. Memory-bandwidth bound.

**Theoretical minimum**:
```
9.0 KB / 30 GB/s = 0.3 µs
```

**Implementation cost (P50)**:
- Sort (38 elements): negligible (<1 µs)
- Python f-string formatting (38 iterations): ~38 × 100 ns = 3.8 µs
- Python interpreter overhead: ~0.1 ms
- **P50: ~0.10 ms**

| Resource | Minimum | Calculation |
|---|---|---|
| Latency | 0.0003 ms | 9.0 KB / 30 GB/s |
| Bandwidth consumed | 9.0 KB | input metadata + output string |
| Compute | ~400 float comparisons | sort + score comparisons |

**Gap ratio**: 0.10 ms / 0.0003 ms = **333×** — pure Python interpreter overhead.

### Critical Path Analysis

The pipeline has two parallelizable branches and one mandatory sequential chain:

```
t=0: [Start]
     │
     ├──── Branch A: PG tag lookup (0.32ms) ──→ Redis epoch filter (0.10ms) ──→ done at t=0.42ms
     │
     └──── Branch B: Embedding (1.68ms) ──→ Qdrant ANN (1.00ms) ──→ done at t=2.68ms

t=2.68ms: [Qdrant done — has K=38 seeds]
           │
           └──→ Neo4j BFS (2.20ms) ──→ done at t=4.88ms
                │
                └──→ Token Assembly (0.10ms) ──→ DONE at t=4.99ms
```

**Branch A (PG + Redis) finishes at 0.42 ms. Branch B (Embed + Qdrant) finishes at 2.68 ms. Branch B is the critical path bottleneck.**

Branch A results (filtered candidate content_hashes) are ready more than 2 ms before Neo4j BFS starts. PG results can be passed to the BFS query as an additional filter without cost — they are not on the critical path.

**P50 critical path latencies (warm)**:

| Scenario | Total | Critical path |
|---|---|---|
| Embedding pre-computed (session start cache hit) | 3.30 ms | Qdrant (1.00ms) → Neo4j (2.20ms) → Asm (0.10ms) |
| Embedding required (typical) | **4.99 ms** | **Embed (1.68ms) → Qdrant (1.00ms) → Neo4j (2.20ms) → Asm (0.10ms)** |
| Cold Neo4j (first session after restart) | 52.79 ms | Embed (1.68ms) → Qdrant (1.00ms) → Neo4j cold (50ms) → Asm (0.10ms) |
| Theoretical minimum (no software overhead) | 0.022 ms | Qdrant bandwidth (0.021ms) → Neo4j (0.001ms) |

**The 4.99 ms P50 is dominated by two phases: embedding computation (1.68 ms, physics-bound) and Neo4j BFS (2.20 ms, architecture-bound JVM overhead).**

### Bottleneck Analysis

#### At Tim's load (5 sessions/day)

Average load: 5 / 86,400 = 0.000058 req/s.

| Phase | P50 Latency | Max Throughput | Utilization at Tim Load |
|---|---|---|---|
| P1 PG (warm) | 0.32 ms | 3,173 req/s | 0.002% |
| P2 Redis | 0.10 ms | 9,950 req/s | 0.0006% |
| P3 Qdrant | 1.00 ms | 1,000 req/s | 0.006% |
| P4 Neo4j (warm) | 2.20 ms | 455 req/s | 0.013% |
| P5 Assembly | 0.10 ms | 9,606 req/s | 0.0006% |
| Embed + Qdrant | 2.68 ms | 373 req/s | 0.016% |

**First bottleneck phase: Embed + Qdrant chain at 373 req/s.**

Tim's load (0.000058 req/s) is 6,442,830× below the bottleneck. No phase is anywhere close to saturation. The system would need to sustain ~1,000 concurrent sessions simultaneously to approach Neo4j saturation.

**Burst mode** (single user typing fast, 1 request/minute = 0.017 req/s): Still 22,000× below the bottleneck. Saturation is not a real concern at any plausible single-user load.

#### At scale (hypothetical SaaS deployment)

The first phase to saturate under increasing concurrent load:

1. **Neo4j BFS (455 req/s)** — bottleneck before Qdrant because Neo4j is single-threaded for Cypher execution in Community Edition
2. **Embed+Qdrant chain (373 req/s)** — if embedding is required per-request; GPU-bound at 936 GB/s bandwidth divided among concurrent requests
3. **Qdrant (1,000 req/s)** — if embedding is cached/pre-computed

Neo4j Community Edition's single-threaded write lock makes it the saturation point at scale, not the vector search.

### Roofline Model

All five phases are **memory-bandwidth bound** on the N100 CPU. None are compute-bound.

```
N100 Roofline:
  Peak compute: 109 GFLOPS
  Peak memory bandwidth: 30 GB/s
  Ridge point: 109 / 30 = 3.6 FLOPs/byte

Phase AIs (all far below ridge point):
  P1 PG:    0.01 FLOPs/byte  [↓ 360× below ridge]
  P2 Redis: 0.01 FLOPs/byte  [↓ 360× below ridge]
  P3 Qdrant: 0.50 FLOPs/byte [↓ 7.2× below ridge]
  P4 Neo4j: 0.05 FLOPs/byte  [↓ 72× below ridge]
  P5 Asm:   0.02 FLOPs/byte  [↓ 180× below ridge]

RTX 3090 (Embedding):
  Ridge point: 71,160 GFLOPS / 936 GB/s = 76 FLOPs/byte
  Embedding at 500 tokens prefill: ~2 × 0.6B × 500 / (0.6B × 2 bytes) = 500 FLOPs/byte
  → Prefill is compute-bound on RTX 3090 for long contexts, bandwidth-bound for short (<200 tokens)
  → At 500 tokens: approaches ridge; actual 1.28 ms bandwidth estimate is near-floor
```

**The pipeline has zero compute-bound phases on the N100.** Adding CPU cores or improving N100 compute does not help. The only hardware improvement that would reduce latency is:
1. Faster storage (NVMe → RAM for PG cold start)
2. Faster RAM bandwidth (DDR5 host → ~50% reduction in Qdrant and Neo4j theoretical floors)
3. Co-located embedding model (eliminate LAN RTT from critical path)

### Gap Analysis

#### Physics gaps (unavoidable given current topology)

- **Embedding LAN roundtrip (0.3 ms RTT)**: embedding model lives on 192.168.50.62; physics of 1Gbps LAN RTT cannot be reduced without co-locating the model on CasaOS or caching embeddings.
- **Embedding bandwidth time (1.28 ms)**: qwen3-embedding:0.6b model weights (1.2 GB) must traverse VRAM bandwidth once per inference. Cannot be reduced without a smaller model.
- **Neo4j JVM cold start (50 ms)**: page cache miss + NVMe read for 160 KB graph that is not yet cached. Cannot be eliminated without persistent warm cache guarantee.

#### Architecture gaps (design choices — removable)

- **Redis IPC overhead (0.10 ms, 200× above physics floor)**: Redis epoch filter runs as a separate process. If the global epoch counter were maintained as a Python variable (updated at session start from one Redis GET, then used in-memory for all comparisons), this phase drops to 0.0005 ms. The Redis call is architecture overhead, not physics overhead.

- **Qdrant IPC overhead (~49× above physics floor)**: Qdrant runs as a separate Docker process. At 360 points, the physics of distance computation is 20 µs; IPC adds 980 µs. Options: (a) use a Python-native vector library (faiss, annoy, sklearn NearestNeighbors) in-process — zero IPC — or (b) accept the overhead given Qdrant's production benefits at larger scale.

- **Neo4j JVM overhead (~1,700× above physics floor for warm case, ~1× above for architecture floor)**: The 2.2 ms warm BFS cost is dominated by Cypher query compilation, JVM object graph traversal, and Bolt protocol serialization. The CO_RETRIEVED_FRESH type-partitioning from Q091 reduces this to ~1.6 ms by shrinking the traversal set. Further reduction requires either (a) eliminating Neo4j in favor of a PostgreSQL self-join (pgvector-as-graph, identified in synthesis as a priority question), or (b) pre-compiling the BFS Cypher as a stored procedure.

- **Embedding on critical path**: The embedding computation (1.68 ms) blocks Qdrant from starting. If session context embeddings were computed eagerly (at session start before the retrieval hook fires) and cached in Redis for the session duration, this phase moves off the critical path. The embedding would overlap with the human typing their first prompt.

#### Implementation gaps (code quality)

- **Python interpreter for Assembly (0.10 ms, 333× above physics floor)**: Pure interpreter overhead for formatting 38 memory records. Could be reduced 10× with a C extension or pre-compiled Jinja2 templates, but at 0.10 ms this is noise compared to the other phases.

- **Sequential Redis call for epoch**: One Docker socket roundtrip adds 100 µs where 0.5 µs would suffice. Moving epoch to a Python module-level cached variable is a one-line change.

## Ideas Extracted

### Idea: eager-session-context-embedding-cache
- **Novelty**: 0.72
- **Evidence**: 0.68
- **Feasibility**: 0.88
- **Description**: Compute the session context embedding eagerly at the start of each Claude Code session — before the UserPromptSubmit hook fires and before any retrieval is needed — and cache the resulting 1536-dim float32 vector in Redis with a session-scoped key (TTL = session duration). The recall-retrieve.js hook reads the cached vector instead of generating a new one. This removes embedding computation from the retrieval critical path entirely, replacing 1.68 ms (physics-bound LAN + bandwidth) with a Redis GET of 6,144 bytes (~0.10 ms). The embedding generation overlaps with session initialization (which has human think time), not with retrieval latency. Implementation: add a session-start hook that calls qwen3-embedding:0.6b with the initial session context (project name + recent file paths, ~50 tokens), stores the vector in Redis at `recall:session:{id}:embedding`. The recall-retrieve hook reads this key instead of calling the embedding API. If the key is missing (fallback), generate inline as today. Estimated critical path reduction: 4.99 ms → 3.30 ms (1.69 ms saved, 34% reduction).

### Idea: in-process-epoch-counter-eliminates-redis-ipc
- **Novelty**: 0.68
- **Evidence**: 0.82
- **Feasibility**: 0.95
- **Description**: Replace the per-retrieval Redis GET for the global epoch counter with a module-level Python integer variable, initialized at session start with one Redis GET and updated in-memory thereafter. The current architecture pays a 100 µs Docker socket roundtrip for a single integer read that is stable for the entire session. The Redis call is not required for correctness — the epoch value only changes at session end when Op2 increments it, which is after retrieval completes. A module-level `CURRENT_EPOCH = None` initialized at session start to `redis.get("recall:global_epoch")` eliminates the IPC cost entirely. Phase 2 drops from 0.10 ms to ~0.001 ms. This is a one-line initialization change in the recall-retrieve hook. The gap between physics floor (0.0005 ms) and current (0.10 ms) is 200× and is entirely caused by this avoidable IPC call. Note: this is safe because retrieval always happens within a session, and the epoch does not change mid-session.

### Idea: pgvector-self-join-eliminates-neo4j-jvm-overhead
- **Novelty**: 0.78
- **Evidence**: 0.65
- **Feasibility**: 0.60
- **Description**: Replace the Neo4j BFS traversal (2.20 ms warm, driven by JVM overhead) with a PostgreSQL self-join on a `co_retrievals(memory_uuid_a, memory_uuid_b, co_occurrence_count, last_reinforced_epoch)` table. With a GIN index on `(memory_uuid_a)` and K=38 seeds, the query is `SELECT DISTINCT r.memory_uuid_b FROM co_retrievals r WHERE r.memory_uuid_a = ANY($seeds) AND r.last_reinforced_epoch >= $epoch_floor ORDER BY r.co_occurrence_count DESC LIMIT 100`. This runs in PostgreSQL on CasaOS — no LAN hop, no JVM, no Bolt serialization. The data fits in shared_buffers after first query (800 rows × ~100 bytes = 80 KB). Estimated warm latency: 0.50–0.80 ms (PG query on 800-row table with index). This would reduce the critical path from 3.30 ms (no embed) to ~1.70 ms and eliminate all three Neo4j failure modes identified in Waves 8–10 (hub fanout, epoch filter gap, orphan compound failure). The consolidation from 4-store to 3-store architecture also eliminates the multi-store coherence problem class. Feasibility is 0.60 because the causal graph structure (2-hop BFS) requires a self-join that may be less natural in SQL than in Cypher, and migration risk from the existing Neo4j graph is non-trivial.

### Idea: in-process-qdrant-replacement-at-360-points
- **Novelty**: 0.70
- **Evidence**: 0.75
- **Feasibility**: 0.85
- **Description**: At 360 points, the Qdrant process boundary costs 49× the physics floor (1.0 ms vs 0.021 ms). For a corpus that fits in 2.21 MB of RAM, a Python-native HNSW implementation (hnswlib, ~10 lines to load and query) or even a brute-force cosine scan (numpy dot product: 360 × 1536 float32 = 0.074 ms bandwidth-bound on N100) eliminates all IPC overhead. The in-process approach achieves 0.074–0.21 ms latency for the ANN search itself (brute force to HNSW), down from 1.0 ms. This saves ~0.8–0.9 ms on the critical path. The trade-off is losing Qdrant's persistence (write path), filtering, and payload support — but these can remain in Qdrant as the write-side store while the read path uses an in-process numpy/hnswlib mirror loaded at session start. The mirror is 2.21 MB loaded once into the Python process on startup. Novelty score accounts for the fact that in-process ANN is well-known; the specific insight is that Qdrant's IPC overhead is 49× the physics floor at this scale, making the replacement clearly worthwhile by the arithmetic.

### Idea: retrieval-pipeline-roofline-as-slo-instrument
- **Novelty**: 0.73
- **Evidence**: 0.70
- **Feasibility**: 0.90
- **Description**: The roofline analysis establishes physics floors for each phase that can be used as SLO lower bounds in the Q071 health dashboard. Each phase has a computed minimum latency below which something is wrong (e.g., PG below 0.001 ms means the query is bypassing the database; Neo4j below 0.01 ms warm means the query is returning empty results, not real data). Add per-phase latency tracking to the health dashboard with three-band alerting: GREEN (within 10× of physics floor), YELLOW (10×–100×), RED (above 100× of physics floor or above P99 threshold). The Neo4j warm BFS at 2.20 ms is currently 1,700× above the physics floor but within expected range for JVM overhead — its health threshold is 20 ms warm, not the physics floor. This turns the Q107 roofline into a living operational instrument rather than a one-time analysis. Each phase threshold is derived from first principles, not empirical percentiles, making it robust to regime changes (model upgrades, corpus growth, hardware changes).

## Summary

**Theoretical minimum critical path (pre-computed embedding)**: 0.022 ms (22 µs)
**Theoretical minimum critical path (with embedding)**: 1.30 ms
**P50 warm (typical, embedding needed)**: 4.99 ms
**P50 cold (Neo4j cold start)**: 52.79 ms
**Gap from minimum to actual (typical)**: 3.8× — surprisingly tight because embedding physics is the dominant floor

**All five phases are memory-bandwidth bound.** None are compute-bound on the N100 CPU. The N100 compute capacity (109 GFLOPS) exceeds the computational requirements of all five phases combined by at least 3 orders of magnitude.

**Bottleneck at Tim's load**: No phase is close to saturation. The Embed+Qdrant chain (373 req/s limit) is the first to saturate under load, but Tim's 5 sessions/day is 6.4 million times below that threshold.

**Dominant latency contributors** (P50 warm):
1. Neo4j BFS: 2.20 ms — architecture gap (JVM overhead, 1,700× above physics floor)
2. Embedding: 1.68 ms — physics gap (bandwidth-bound on RTX 3090 + LAN RTT)
3. Qdrant: 1.00 ms — architecture gap (IPC overhead, 49× above physics floor)

**The largest addressable gap is Neo4j (2.20 ms → potentially 0.5–1.6 ms** with pgvector consolidation or CO_RETRIEVED_FRESH partitioning from Q091). The second-largest is moving embedding off the critical path via eager caching (saves 1.68 ms). Together these two changes could reduce P50 from 4.99 ms to approximately 1.5–2.0 ms — a 2.5–3.3× improvement that is realizable without hardware changes.
