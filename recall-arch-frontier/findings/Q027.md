# Finding: Q027 — Physics Ceiling: Retrieve Latency with Negative Cache

**Question**: What is the theoretical minimum latency for Recall's retrieve operation assuming a 40% negative cache hit rate, and what is the current gap ratio?
**Question Type**: PHYSICS
**Verdict**: BREAKTHROUGH
**Severity**: High

---

## Latency Model

### Assumptions

| Parameter | Value | Basis |
|-----------|-------|-------|
| Negative cache hit rate | 40% | Repeated queries on unstored topics (new projects, unfamiliar services, out-of-scope topics) within and across sessions |
| Positive cache hit rate | 35% | Redis hot-result cache from Q004; established prior finding |
| Full cache miss rate | 25% | Queries on stored topics with expired or uncached embeddings |
| Negative cache hit latency | 0.5 ms | Redis key lookup (Python redis-py client overhead on Docker bridge); same as current positive cache hit |
| Positive cache hit latency | 0.5 ms | Q011 baseline |
| Current miss-path latency | 80.5 ms | Q011 baseline: Redis(0.5) + Ollama(50) + Qdrant(10) + Neo4j(20) |
| Theoretical-minimum miss-path latency | 2.75 ms | Q011: physics floor for all components at hardware limits |
| Miss-path latency distribution | Log-normal, sigma_ln = 0.30 | 30% coefficient of variation; typical for network-bound inference calls with LAN jitter and Ollama batch-1 variance |

The log-normal assumption for miss latency is conservative (30% CV). Ollama's GPU inference time at batch=1 can spike 2-3x from mean during memory pressure or CUDA scheduling contention; the model captures this tail without overstating it.

**p-percentile calculation method**: With (p_neg + p_pos) = fast fraction of calls, the p95 of the mixture distribution falls at quantile q = (0.95 - fast_fraction) / miss_fraction of the miss-path distribution. This is exact for a two-component mixture where the fast component has negligible variance relative to the miss component (0.5ms << 80.5ms satisfies this).

### Latency Distribution Under Each Scenario

| Scenario | p50 | p95 | Mean |
|----------|-----|-----|------|
| Current (no caches) | 80.5 ms | 126.1 ms | 80.5 ms |
| Negative cache only (40% fast, 60% miss) | 80.5 ms | 116.5 ms | 48.5 ms |
| Negative + positive cache (75% fast, 25% miss) | 0.5 ms | 99.1 ms | 20.5 ms |
| All caches + theoretical-min miss path | 0.5 ms | 3.4 ms | 1.1 ms |

**Derivation detail for each p95:**

Scenario 1 (baseline): p95 is the 95th percentile of a log-normal with mean=80.5ms, sigma_ln=0.30.
- mu_ln = ln(80.5) - 0.5(0.3)^2 = 4.388 - 0.045 = 4.343
- p95 = exp(4.343 + 1.645 × 0.30) = exp(4.837) = 126.1 ms

Scenario 2 (negative cache only): 40% at 0.5ms, 60% miss. p95 is at the 91.7th percentile of the miss distribution (z = 1.383).
- p95_miss(0.917) = exp(4.343 + 1.383 × 0.30) = exp(4.757) = 116.5 ms
- p50 is unchanged: 50th percentile falls in the 60% miss bucket (50% > 40%).

Scenario 3 (negative + positive): 75% at 0.5ms, 25% miss. p95 is at the 80th percentile of miss distribution (z = 0.842).
- p95_miss(0.80) = exp(4.343 + 0.842 × 0.30) = exp(4.595) = 99.1 ms
- p50 moves to 0.5ms: 50th percentile is in the 75% fast bucket.

Scenario 4 (all caches + theory min): 75% at 0.5ms, 25% at 2.75ms theoretical min. p95 at 80th percentile of theory-min miss distribution.
- mu_ln_theory = ln(2.75) - 0.5(0.3)^2 = 1.012 - 0.045 = 0.967
- p95 = exp(0.967 + 0.842 × 0.30) = exp(1.220) = 3.4 ms

### Does Negative Cache Alone Close the Gap?

**No — and the reason is structurally important.**

Negative cache alone (Scenario 2) improves the mean by 1.7x (80.5ms to 48.5ms) but leaves p50 and p95 virtually unchanged (p50: 80.5ms, p95: 116.5ms vs 126.1ms). The 500ms SLA is trivially met by all scenarios. The relevant question is whether sub-50ms or sub-100ms targets are achievable:

- **p50 < 50ms**: Negative cache alone does NOT achieve this. The 50th percentile sits in the 60% miss bucket and remains at 80.5ms. Only when the combined fast fraction exceeds 50% does p50 improve — which requires adding the positive cache (75% combined fast fraction), at which point p50 drops to 0.5ms.
- **p95 < 100ms**: Negative cache alone does NOT achieve this (116.5ms). Negative + positive cache together barely achieves it (99.1ms — marginal at the current miss-path latency). The theoretical-minimum miss path collapses p95 to 3.4ms.

The critical structural insight: **the p50 breakpoint is at 50% fast fraction; the p95 breakpoint requires the miss path itself to improve.** Neither threshold is achievable from negative caching alone. Both require the positive result cache (to tip p50 into the fast bucket) plus, for p95 < 50ms, meaningful improvement to the miss-path latency.

### Required Miss-Path Improvement

With both caches deployed (75% fast, 25% miss), p95 falls at the 80th percentile of the miss distribution. To achieve p95 < 100ms:

- Required miss-path mean: 81.3 ms (current is 80.5 ms — barely satisfies this)
- This means: the current implementation, with both caches, achieves p95 just under 100ms by a razor-thin margin (99.1ms). Any Ollama latency spike breaks it.

To achieve p95 < 50ms with both caches deployed:

- Required miss-path mean: 40.6 ms (a 2.0x reduction from current 80.5ms)
- Achievable by: eliminating Ollama on cache-miss queries (embedding vector cache from Q011, which stores query embeddings independently of full results), reducing the miss path to Qdrant(10ms) + Neo4j(20ms) + Redis(0.5ms) = 30.5ms. That satisfies 40.6ms.
- With embedding vector cache: Ollama is eliminated for any query seen in the past week. Miss path becomes: Redis emb-check (0.5ms, miss) + Redis full-check (0.5ms, miss) + emb-cache-check (0.5ms, miss) + Qdrant (10ms) + Neo4j (20ms) = ~31.5ms. p95 = lognormal(31.5, 0.30)[80th] = 38.4ms. Achieves sub-50ms p95.

The theoretical minimum (2.75ms miss path) achieves p95 < 4ms — a 30x improvement over the current no-cache p95.

**Summary of what each mechanism independently contributes to p95:**

| Mechanism | p95 | Improvement vs baseline |
|-----------|-----|------------------------|
| No caches (baseline) | 126.1 ms | 1.0x |
| + Negative cache | 116.5 ms | 1.1x |
| + Negative + positive cache | 99.1 ms | 1.3x |
| + Embedding vector cache (Ollama eliminated on repeat queries) | ~38 ms | 3.3x |
| + All caches + theoretical-min miss | 3.4 ms | 37x |

The embedding vector cache (eliminating the Ollama call for any re-queried text) contributes the largest single p95 improvement — more than negative and positive caches combined.

## Ideas Extracted

### Idea: two-cache-minimum-deployment

Deploy negative cache and positive result cache together as a paired unit, never separately. The analysis proves that neither cache alone crosses the p50 < 50ms threshold — only the combination (75% fast fraction) tips p50 into the sub-millisecond bucket. A deployment that adds only the negative cache (the simpler, lower-effort implementation) achieves a 1.7x mean improvement but leaves p50 and p95 essentially unchanged from the user's perspective. The correct minimum viable deployment is both caches simultaneously.

- **Novelty**: 0.60 — The insight that two caches must be deployed together to cross a threshold boundary (rather than additively improving performance) is not explicitly documented in any AI memory system design guide. Cache deployment is typically treated as incremental. The threshold effect — where the p50 breakpoint is mathematically at 50% fast fraction — makes the pair non-separable for p50 improvement.
- **Evidence**: 0.75 — The math is exact given the distributional assumptions. The log-normal miss distribution is a well-validated model for network-bound inference latency. The threshold behavior is not speculative.
- **Feasibility**: 0.90 — Both caches are Redis-based with 1-2 week implementation each. The paired deployment recommendation adds no engineering overhead — it is an ordering and scope decision at the project planning level.
- **Description**: When scoping the caching work for Recall, implement negative cache (key: `neg:{sha256(query_text)}`, value: "empty", TTL: 300s, invalidated on store to matching domain) and positive result cache together in a single sprint. Sequence: (1) add negative cache check before Ollama call in the retrieve path; (2) confirm positive result cache is writing on every non-empty Qdrant result; (3) deploy both; (4) measure p50 and p95 before and after. The combined 75% fast fraction is the first configuration that produces a user-perceivable p50 improvement. Deploying them separately produces no perceptible p50 change until both are live.

### Idea: embedding-vector-cache-as-p95-gate

The Q011 embedding-vector-cache idea (store query embedding separately, never re-call Ollama for a previously seen query text) is the single mechanism that drives p95 below 50ms when both caches are already deployed. Without it, p95 hovers at ~99ms even with both caches live. With it, the miss path drops from 80.5ms to ~31ms (Qdrant + Neo4j only), and p95 falls to ~38ms. This reframes the embedding-vector-cache not as a latency optimization but as the gating mechanism for p95 < 50ms.

- **Novelty**: 0.65 — The Q011 finding established this idea at novelty 0.55 (embedding caching is practiced, separation of embedding-cache from result-cache is less documented). The new contribution here is the quantitative framing: embedding-vector-cache is not merely an optimization, it is the p95 threshold gate. This framing — identifying which specific mechanism is the p95 critical path — is novel in the context of AI memory system design.
- **Evidence**: 0.75 — Q011 established engineering basis (0.65). This finding adds the mathematical proof that the miss-path must drop below 40.6ms for p95 < 50ms with both caches, and that eliminating Ollama achieves 31ms — the only mechanism from the existing idea backlog that hits this target without hardware changes.
- **Feasibility**: 0.90 — Q011 established 1-2 week implementation. Mechanism: check `Redis.get(f"emb:{sha256(query_text)}")` before calling Ollama; cache is populated on every Ollama call with 7-day TTL. No invalidation required (query text embeddings are stable). This is the single highest-leverage implementation task for achieving p95 < 50ms given both caches already deployed.
- **Description**: After deploying the two-cache pair (negative + positive), the miss path still triggers Ollama for any query text not in the full result cache. Add a third Redis namespace `emb:{sha256(query_text)}` that stores the raw 4KB float vector from every Ollama call, with 7-day TTL. Before calling Ollama in the miss path: check this namespace. On hit: skip Ollama entirely, proceed directly to Qdrant with the cached vector. On miss: call Ollama and write both the embedding cache and the result cache on return. This reduces the miss path from (Redis 0.5 + Ollama 50 + Qdrant 10 + Neo4j 20) to (Redis 0.5 + Redis 0.5 + Qdrant 10 + Neo4j 20) = 31ms for repeat query texts — crossing the p95 < 50ms threshold.

### Idea: negative-cache-similarity-matching

The current negative cache design uses exact query hash matching: `neg:{sha256(query_text)}`. This only catches byte-identical repeat queries. A large fraction of negative cache value comes from semantically similar queries that are not identical: "what are my Docker ports?", "list my Docker port mappings", "show Docker container ports" are semantically equivalent negative results if no Docker port memories are stored. Similarity-based negative cache lookup — using a locality-sensitive hash (LSH) or embedding cosine distance threshold — would increase the effective negative cache hit rate from 40% (exact match) to potentially 60-70%.

- **Novelty**: 0.80 — Similarity-based negative caching is not implemented in any known production AI memory system. DNS negative caching is exact-match only (QNAME + QTYPE). The extension to semantic similarity is a non-obvious application of LSH to cache key lookup. Adjacent precedent: Bloom filters for approximate membership testing; product quantization for approximate vector lookup. The specific application to cache existence queries in AI memory is novel.
- **Evidence**: 0.55 — LSH and approximate nearest-neighbor methods are peer-reviewed and production-validated at scale (Faiss, ANNOY). The specific application to negative cache key matching in a personal AI memory system has not been benchmarked. The 60-70% hit rate estimate is speculative and requires empirical validation on real Recall query logs.
- **Feasibility**: 0.60 — 1-3 months. Two sub-approaches: (A) Store the query embedding (not just the hash) in the negative cache entry; on a new query, compute its embedding, then run a low-ef Qdrant ANN search against the negative cache namespace to find similar past-empty queries. Cost: one Qdrant query (10ms) to check the negative cache — defeating part of the latency benefit. (B) Use a lightweight LSH (e.g., SimHash on the query token fingerprint) without a full embedding: fast but lower recall. Approach B is more tractable: SimHash of query tokens gives a 64-bit integer; queries with Hamming distance < 4 bits are likely semantically similar. Check `Redis.get(f"neg_lsh:{simhash(query_tokens)}")` in O(1) before calling Ollama.
- **Description**: Extend the negative cache key from `neg:{sha256(exact_text)}` to include SimHash bucket keys. When storing a negative result, compute `simhash(tokenize(query_text))` and store under multiple LSH bucket keys (using multi-probe LSH with 3-4 probes). On a new query, check all LSH bucket keys before calling Ollama. A hit in any bucket returns empty without the Ollama call. Threshold tuning: set Hamming distance threshold conservatively (3 bits out of 64) to keep false-negative rate below 1%. Expected improvement: effective negative cache hit rate increases from 40% (exact) to 55-65% (LSH), reducing mean latency by an additional 12-20ms at the cost of 3-4 SimHash computations per query (negligible, <0.1ms in Python).

### Idea: negative-cache-domain-partitioned-ttl

The 300-second TTL on negative cache entries is uniform across all query domains. But the invalidation semantics differ sharply by domain: a negative cache entry for "Docker port mappings" should be invalidated the moment any Docker-related memory is stored, while a negative cache entry for "Relay AI receptionist Twilio" (a paused project) could safely remain cached for days. Partitioning the negative cache TTL by memory domain — with short TTL for active project domains and long TTL for inactive/out-of-scope domains — increases the practical hit rate without increasing the staleness risk that a new memory fails to surface.

- **Novelty**: 0.65 — Domain-partitioned cache TTL is practiced in CDN systems (different TTL for static assets vs. API responses vs. HTML). The application to negative cache entries in an AI memory system, keyed on query domain inference rather than URL path, is novel. No AI memory system tracks domain-specific negative TTLs.
- **Evidence**: 0.70 — CDN TTL partitioning is well-documented engineering practice with clear production results. The extension to AI memory domains is a logical analog with the same reasoning: staleness risk is domain-specific, not uniform. The implementation requires domain inference on the query text, which Recall already performs (domain tagging on store operations).
- **Feasibility**: 0.75 — 2-4 weeks. Recall's store path already tags memories with a domain (session, project, user-profile, etc.). The retrieve path can infer the likely domain of a query using a lightweight keyword or embedding classifier. Store negative cache entries under `neg:{domain}:{query_hash}` with domain-specific TTLs: active project domain (e.g., "recall"): 60s; inactive project domain: 3600s; out-of-scope domain: 86400s. On a new store operation to domain X, issue a Redis `DEL` pattern delete for `neg:{domain_X}:*` to invalidate all negative entries in that domain. This keeps the negative cache effective even as new memories are stored.
- **Description**: At Recall's retrieve endpoint, after computing the query embedding, classify the query into a domain bucket (lightweight: check for project keywords, fallback to "general"). Look up `neg:{domain}:{sha256(query_text)}` with a domain-appropriate TTL. On store, invalidate `neg:{stored_domain}:*` via Redis key scan or by maintaining a domain-indexed set of negative cache keys. TTL table: `session_domain=60s`, `active_project=120s`, `paused_project=3600s`, `out_of_scope=86400s`. This replaces the uniform 300s TTL with a semantics-aware TTL, increasing the effective hit rate for stable negative domains (Twilio, out-of-scope topics) while keeping invalidation fast for active project domains.

### Idea: miss-path-parallelization-after-embed

With both caches deployed, the 25% miss path still runs serially: Ollama (50ms) -> Qdrant (10ms) -> Neo4j (20ms) = 80ms. Neo4j and Qdrant are independent once the embedding is available — Neo4j could be pre-warmed with likely graph contexts while Qdrant is running, or both could execute concurrently after the embedding completes. This parallelization was identified as an architecture gap in Q011 but not scored as a standalone idea. With the miss path now dominating p95 (it is the entire p95 tail), parallelizing Neo4j and Qdrant after embedding completion reduces the miss path from 80.5ms to 50ms + max(10ms, 20ms) = 70ms, a 14% improvement on the miss path and a meaningful reduction in p95 tail.

- **Novelty**: 0.40 — Parallel database queries are a standard pattern. asyncio.gather() for concurrent Qdrant and Neo4j calls is a known technique. The specific application to this retrieval path is not novel. Scoring lower because this is standard async programming rather than a new mechanism.
- **Evidence**: 0.85 — asyncio.gather() for concurrent I/O-bound calls is well-validated in Python. Qdrant and Neo4j calls are entirely independent once the embedding vector is available (Neo4j query uses stored node IDs, not the embedding directly). The independence is exact, not approximate.
- **Feasibility**: 0.90 — 2-3 days. Change the retrieve endpoint from sequential `await qdrant.search()` + `await neo4j.query()` to `results, graph = await asyncio.gather(qdrant.search(embedding), neo4j.query(seed_ids))`. Requires restructuring so that the Neo4j seed IDs come from a preliminary ANN result or are pre-seeded from the query domain — if Neo4j requires Qdrant result IDs as input, true parallelism is not possible. If Neo4j is queried for domain context independently, parallelism is exact. Verify the call graph before implementing.
- **Description**: In Recall's retrieve route handler, after obtaining the query embedding (Ollama call), launch both `qdrant_client.search(collection, embedding, k=5)` and `neo4j_session.run(domain_context_query, domain=inferred_domain)` concurrently with `asyncio.gather()`. The Qdrant call returns top-k memory IDs. The Neo4j call returns domain graph context (related concepts, causal chains for the inferred query domain). Merge the results in the application layer. This reduces the post-embedding latency from 10+20=30ms sequential to max(10,20)=20ms parallel — a 10ms reduction on the miss path, bringing it from 80.5ms to 70.5ms, and p95 from 99ms to 87ms.

Request coalescing would ensure only one Ollama call is issued for concurrent identical queries within a 100ms window.
