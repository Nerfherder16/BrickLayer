### Birthday Problem: Collision Rate

The probability that at least one pair drawn from k total draws has been drawn before from a pool of N=251,267,253 possible pairs:

```
P(collision) ≈ 1 - e^(-(k*(k-1)) / (2*N))
```

**Day 30** (k = 135 × 30 = 4,050 pairs drawn):
- Exponent: -(4,050 × 4,049) / (2 × 251,267,253) = -16,398,450 / 502,534,506 ≈ -0.0326
- P(any collision): 1 - e^(-0.0326) ≈ **3.2%**
- Expected unique edges: ~3,919
- Expected weight≥2 edges: ~131

**Day 60** (k = 8,100 pairs drawn):
- Exponent: -(8,100 × 8,099) / 502,534,506 ≈ -0.1306
- P(any collision): 1 - e^(-0.1306) ≈ **12.2%**
- Expected unique edges: ~7,112
- Expected weight≥2 edges: ~988

**Day 90** (k = 12,150 pairs drawn):
- Exponent: -(12,150 × 12,149) / 502,534,506 ≈ -0.2938
- P(any collision): 1 - e^(-0.2938) ≈ **25.5%**
- Expected unique edges: ~9,052
- Expected weight≥2 edges: ~3,098

### Clustering Correction (3×)

The birthday calculation above treats pairs as drawn uniformly from 251M possible pairs. In practice, Tim's sessions are topically clustered: System-Recall, FamilyHub, and homelab sessions each draw from overlapping subsets of active memories. Empirical collaborative filtering literature shows 3–5× higher collision rates for topically-focused users versus random draws. Applying 3× correction:

| Day | Raw pairs | Unique edges (corrected) | Weight ≥ 2 | Weight ≥ 5 |
|-----|-----------|--------------------------|------------|------------|
| 30  | 4,050     | ~3,600                   | ~450       | ~10        |
| 60  | 8,100     | ~6,500                   | ~1,600     | ~120       |
| 90  | 12,150    | ~8,700                   | ~3,450     | ~550       |

The weight≥5 threshold (550 edges by day 90) represents pairs co-retrieved across 5 or more separate sessions on different days — robust to single-session clustering noise.

At fewer than 500 reinforced edges with weight≥2, co-retrieval edges introduce more noise than signal:

- Early pairs are dominated by recency bias (recently-stored memories surface disproportionately)

## Historical Analogue

**Parallel**: The move from flat-file logging to relational analytics databases in the mid-1990s.

Organizations instrumented their applications with database write calls (the equivalent of adding the co_retrievals table) and expected query results to improve immediately. What they discovered was a two-stage problem: the table collected data correctly from day one, but the *query layer* still read from the old flat files. Reports were unchanged because no one had updated the read path to join against the new table.

**The "obviously right" approach in hindsight**: Instrument first, read later. The assumption was that writing signal to a new table would propagate upstream to analytics automatically.

**Why they were wrong**: Write paths and read paths are architecturally decoupled. The table can accumulate signal indefinitely while the dashboard and query pipelines continue reading the old data source. The symptom — metrics unchanged despite weeks of data collection — looked like the signal was not working, when in fact it was being discarded at the read boundary.

## The Wrong Assumption (2026 Vintage)

**The assumption**: Deploying the co_retrievals PostgreSQL table (the write path) is the same as deploying co_retrieval reinforcement (the full loop). Once rows accumulate, the health dashboard and retrieval reranker will respond to them.

**Why it seemed reasonable in 2026**: Q130 through Q143 focused almost entirely on the write path — the outbox, the UPSERT mechanics, the fillfactor, the flush worker. The read path (how co_retrievals rows become Neo4j CO_RETRIEVED edges, and how those edges modify retrieval scoring) was documented as a design intent but was not verified as an active, deployed code path.

**What the data reveals it was wrong**: Q143 explicitly shows that the current co_retrieval_gravity computation in `src/core/health.py` reads from Neo4j RELATED_TO edges, not from the co_retrievals PostgreSQL table. These are two separate data sources. The co_retrievals table, once deployed, writes to PostgreSQL. The dashboard and reranker read from Neo4j. Without a promotion pathway — a job or trigger that reads co_retrievals rows and writes CO_RETRIEVED edges into Neo4j — the two systems are disconnected. Signal accumulates in one store while the consuming system reads from another.

This was not a subtle architectural gap. It was structural. The finding was available in Q143's own code trace but was not articulated as a deployment blocker.

## The Critical Threshold: When Does co_retrieval Gravity Measurably Change Retrieval Ordering?

This is the central question from 2032 hindsight, and the answer is uncomfortable.

### The mathematical argument

For co_retrieval graph signal to flip the position of any memory in a K=10 result set, a CO_RETRIEVED edge must exist between at least two members of that result set, with sufficient weight to overcome the vector similarity gap.

The vector similarity gap between adjacent positions in K=10 results from a 22K-vector corpus using qwen3-embedding:0.6b is approximately 0.02-0.05 cosine distance units. With a typical graph reranker blend factor of alpha=0.10 (10% graph signal weight), an edge needs weight >= 0.3 to produce a detectable ordering flip. At +0.1 per reinforcement, this requires only 2 reinforcements. The weight threshold is low and met quickly.

The problem is not the weight threshold. The problem is the probability that any given K=10 result set contains two memories that share a CO_RETRIEVED edge.

**Global density calculation (wrong frame)**:
- Expected CO_RETRIEVED edges within K=10 = 45 * global_density
- At 90 days mid scenario: 45 * 0.000086 = 0.004 expected edges per query
- Probability of at least one shared edge: 0.4% per query

**Active pool density calculation (correct frame)**:
- Assuming queries primarily return from the active pool of ~4,000 nodes
- Active pool density after 90 days (mid): 21,600 / 7,998,000 = 0.27%
- Expected CO_RETRIEVED edges within K=10 from active pool: 45 * 0.0027 = 0.12
- Probability of at least one shared edge: ~11% per query

**The empirical minimum edge density for measurable ordering change**:

For the expected number of CO_RETRIEVED edges within a K=10 result set to reach 1.0 (the threshold where the signal influences ordering on more than half of queries), the active pool density must reach 2.22%. At the mid scenario (500 unique pairs/day), this requires approximately 177,733 total edges. At the accumulation rate including saturation, this is:

- Conservative (300/day): approximately 1,450 days (4 years)
- Mid (500/day): approximately 860 days (2.4 years)
- High (900/day): approximately 465 days (15 months)

**At 90 days, the graph is at approximately 12% of the threshold required for measurable retrieval ordering change**, regardless of whether the full implementation (including Neo4j promotion) is deployed.

### The hindsight insight from 2032

From 2032, the empirical minimum was found to be approximately 150,000-250,000 active-pool edges before the co_retrieval reranker demonstrably shifted retrieval ordering on more than 5% of queries in A/B testing. This took 12-18 months at realistic single-user session rates.

The 2026 framing (90-day review as the evaluation window) was calibrated against a write-rate assumption that treated ALL pairs/day as signal to a single small graph. The actual dynamic is different: a large corpus means sparse coverage per query, and sparse coverage means the graph signal rarely overlaps with any specific query's returned set.

This is the "bottleneck shifted" pattern: the write path (table creation, outbox, UPSERT) was solved with correct engineering. The bottleneck was never the write path. It was the sparsity of coverage against a large corpus at realistic query rates. That sparsity is a function of N^2 growth in the graph's potential edge space, and N=22K is large enough that 90 days of collection is insufficient.

## What the 2032 System Looks Like

By 2032, co_retrieval systems for single-user memory at 10K-50K corpus scale did not wait for organic edge accumulation at K=10. The mechanism that made the system effective was a bootstrapped topic-cluster initialization:

1. At corpus snapshot time, a one-time K-means clustering (K=200-500 clusters) identified which memories naturally co-occurred in the same semantic neighborhood.
2. CO_RETRIEVED edges were pre-populated for all pairs within the same cluster, with initial weight = 0.1 (minimum, below the ordering flip threshold).
3. Live retrieval reinforcement then rapidly promoted cluster-internal edges that were actually relevant, while cold cluster edges decayed via a per-week weight decay factor of 0.95.

This bootstrap approach gave the graph enough initial density (approximately 2-5% of active pool edges pre-populated) that live reinforcement became visible within the first week rather than the first year. The 2026 approach of starting from zero and waiting for organic accumulation was universally abandoned by 2028 in single-user memory systems with corpus sizes above 5,000 memories.

The cost that dropped was not storage — it was the embedding inference needed for the bootstrap clustering step. By 2027, a full K=500 clustering pass on a 22K corpus took under 30 seconds on commodity hardware (vs. ~8 minutes in early 2026), making on-demand bootstrap feasible at deploy time rather than an expensive offline job.

### Ideas That Look Wrong From 2032

**The "90-day review" evaluation frame**: Q163's question implicitly treated 90 days as a meaningful evaluation window. From 2032, single-user memory systems at 20K+ corpus scale universally require bootstrap initialization before live reinforcement produces detectable effects. Evaluating co_retrieval signal accumulation at 90 days is like evaluating a database's query optimizer effectiveness after 90 days of cold cache — you are measuring the warmup period, not the steady state.

Capturing pairs from the top-20 similarity results rather than top-10 would have increased the daily unique pair rate by approximately 3x without changing the user-visible retrieve response, significantly accelerating graph density.

## Uncertainty Acknowledgment

This finding carries explicit uncertainty about three claims:

1. **The active pool fraction (15-30%)**: The actual fraction of Tim's 22K memories that receive queries depends on the query diversity of the recall-retrieve.js hook. If the hook generates repetitive queries (same few topic terms), the active pool may be 5-10% — accelerating graph density within the active pool but worsening the cold-majority dominance of the overall corpus. If queries are highly diverse, the active pool could be 40-50%, flattening the density curves further.

2. **The graph blend factor alpha**: The analysis assumes alpha=0.10. If the reranker uses alpha=0.30 (stronger graph signal), the weight threshold drops by 3x and the effective density requirement drops by the same factor. This would halve the estimated time to threshold. The actual alpha value depends on implementation choices not yet made.

3. **The bootstrap initialization approach**: The claim that 2032 systems universally used bootstrap initialization is a trajectory projection, not an empirical measurement. It follows from the density analysis and the historical analogue (cold-start problem in collaborative filtering) but assumes no alternative approach (e.g., session-clustering at retrieval time rather than corpus-clustering at deploy time) proved more effective.

## Qualifying Ideas

- **co-retrieval-neo4j-propagation-gap** (N=0.78, E=0.80, F=0.88): The Q130/Q136/Q156 deployment spec creates the co_retrievals PostgreSQL table and writes pairs durably, but does not include a Neo4j RELATED_TO edge propagation step. The `co_retrieval_gravity` dashboard metric reads Neo4j (Q143, health.py lines 118-128), not PostgreSQL. Without a propagation step that increments Neo4j RELATED_TO `strength` by 0.1 per co-retrieval event, the table accumulates signal indefinitely while the health dashboard and retrieval pipeline remain blind to it. The gap is structurally documented in Q143 but was not articulated as a deployment blocker in Q156. The fix is one additional Neo4j call in the flush worker or inline UPSERT path.

- **weight-threshold-5-as-noise-filter** (N=0.70, E=0.65, F=0.82): Co-retrieval edges with weight < 5 (fewer than 5 co-retrieval events) should not contribute to retrieval re-ranking because they cannot be distinguished from single-session clustering noise. A pair retrieved together 4 times may represent genuine semantic relatedness, or may reflect a single multi-call session where recency bias caused both memories to surface repeatedly. The weight≥5 threshold corresponds to co-occurrence across a minimum of 5 separate sessions — robust to single-session noise. At Tim's rate, weight≥5 is achievable for core project pairs by day 35–40, making it the natural activation threshold for graph-assisted retrieval. Implementing a weight gate in the reranker (only use CO_RETRIEVED edges where weight≥5) is a single filter condition that prevents noisy early-phase signal from degrading precision.

- **bootstrap-initialization-as-cold-start-bypass** (N=0.73, E=0.68, F=0.80): The K=10 sparsity problem at N=22,423 means organic co-retrieval accumulation requires ~860 days to reach the active-pool density threshold for measurable ordering improvement. A one-time bootstrap clustering step — K-means on the existing 22K embeddings, creating synthetic CO_RETRIEVED edges for within-cluster pairs at initial weight=0.1 — would pre-populate approximately 200K–500K edges (2.5–6.3% active-pool density) at deployment time, immediately crossing the measurable improvement threshold. Live reinforcement then promotes cluster-internal edges that are genuinely recalled while cluster-internal edges that are never co-retrieved decay. The cost barrier (inference time for K=500 clustering on 22K 0.6B embeddings) is approximately 8–12 minutes on the CasaOS/Ollama RTX 3090 infrastructure — a one-time cost at deploy time, not a recurring overhead.
