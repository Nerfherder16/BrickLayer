# Q120 — FM-3 Epoch Write Ceiling Guard

**Question**: What is the correct last_access_epoch write validation guard that makes FM-3 (PG epoch column inflation) structurally impossible at the API layer — specifically, should the FastAPI store endpoint reject any epoch write where the new value exceeds recall:global_epoch + EPOCH_WRITE_CEILING, what is the exact code insertion point in src/api/routes/, and what HTTP response is returned when a write is rejected?
**Question Type**: IMPLEMENTATION
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: API validation design, defensive programming, database integrity, epoch-clock invariant enforcement

---

## Executive Summary

FM-3 (PostgreSQL `last_access_epoch` column inflated by a buggy hook or migration script writing a wall-clock timestamp like `int(time.time()) ≈ 1,710,000,000` instead of the Redis INCR epoch ≈ 850) is identified in Q117 as the one failure mode that produces an identical S4=CRITICAL + S5=HEALTHY signal to Redis restart but requires the **opposite recovery action**. Q117 shows the discriminant (check `redis_epoch` absolute value after S4 fires) but that is a reactive diagnosis. Q120 closes the gap preventively: a write-path guard at the `POST /memory/store` endpoint that rejects any `last_access_epoch` write exceeding `current_redis_epoch + EPOCH_WRITE_CEILING` makes FM-3 structurally impossible before any row reaches PostgreSQL.

**Critical pre-condition established by source inspection**: Neither `recall:global_epoch` nor `last_access_epoch` exists in the current codebase. The guard is a greenfield addition to a proposed epoch system. PostgreSQL in the current deployment stores only audit_log, session_archive, metrics_snapshot, and users tables — the `memories` table lives in Qdrant. This finding specifies the guard as part of the same deployment package as Q111's schema migration, not as a patch to an existing system.

## Source-Level Findings

### Current store endpoint anatomy (src/api/routes/memory.py:104–432)

The `POST /memory/store` endpoint (`store_memory`) is the sole public write path for individual memories. Its validation structure, in order:

1. **Write Guard Stage 1** (lines 120–133): synchronous regex/heuristic check via `_write_guard.check()` — rejects on pattern match with `HTTPException(422)`.
2. **Durability resolve** (line 136): `Durability(request.durability)` — string enum coercion.
3. **Domain normalize** (line 139): `normalize_domain(request.domain)`.
4. **Dedup check** (lines 175–191): Qdrant content-hash lookup — returns early with `created=False` on duplicate.
5. **Embedding** (lines 194–195): `embedding_service.embed()` — Ollama call on the critical path.
6. **Write Guard Stage 2** (lines 197–229): embedding similarity against cached injection pattern — `HTTPException(422)` on block.
7. **Semantic dedup** (lines 231–275): cosine similarity threshold check — returns early on near-duplicate.
8. **Qdrant write** (line 278): `qdrant.store(memory, embedding)`.
9. **Neo4j write** (lines 281–287): `neo4j.create_memory_node(memory)` with compensating Qdrant delete on failure.

**Key observation**: The `StoreMemoryRequest` model (lines 42–54) has no `last_access_epoch` field today. The epoch guard cannot validate a field that does not exist in the request schema. The guard must be implemented as part of the same epoch-system deployment that adds:
- The `last_access_epoch` column to the memories schema (Q111 migration)
- The `recall:global_epoch` Redis key (populated by session-end INCR)
- The `last_access_epoch` field to `StoreMemoryRequest`

### Batch store path (src/api/routes/memory.py:1196–1370)

`POST /memory/batch/store` (`batch_store_memories`) processes up to 50 items per request. It runs Write Guard Stage 1 per item but does NOT run the embedding guard or semantic dedup. Any epoch ceiling guard added to the single-store path must also be added here, as the batch path is an independent write surface.

### Import path (src/api/routes/ops.py:92–end)

`POST /ops/import` (`import_memories`) reads a JSONL file and directly hydrates `Memory` objects from JSON. This path bypasses both write guard stages entirely. The `Memory` object constructor does not validate epoch values. This path is the highest-risk surface for FM-3 — a migration script that imports a JSONL where `last_access_epoch` was set from `int(time.time())` would write inflated epoch values past every guard. The `--force-epoch` bypass (proposed in Q120 question framing) applies here.

## The Guard Design

### (a) Exact validation location

The epoch ceiling guard inserts at **one location** in the primary store path and **one location** in the batch store path.

**Primary path insertion point**:
- **File**: `src/api/routes/memory.py`
- **Function**: `store_memory`
- **Line**: After Write Guard Stage 1 (after line 133), before durability resolve (line 136)
- **Rationale**: Inserting after the write guard ensures poisoning/injection content is rejected first (cheaper check). Inserting before embedding means a ceiling violation never reaches the Ollama network call — cost savings on every rejection. The epoch check is O(1) (one Redis GET against a hot key) vs embedding which is O(embed_dim) over network.

The guard is a **hard reject** (raises `HTTPException`) unlike the retrieve-path guard (Q111) which is non-fatal. On the write path, a ceiling violation is an anomalous input that must never reach storage. The write-path guard is fatal by design.

**Batch path insertion point**:
- **File**: `src/api/routes/memory.py`
- **Function**: `batch_store_memories`
- **Line**: After Write Guard Stage 1 per item (after line 1234), before `Memory()` constructor call (line 1242)
- **Behavior**: A ceiling violation on a batch item produces a per-item error result (same pattern as the write guard rejection at lines 1225–1234), not a request-level 422. The batch continues processing remaining items. The rejected item's `BatchStoreResult` message field contains the structured error.

**Import path behavior** (addressed in part e below).

### (b) Redis GET on critical path vs. cached epoch

**The Redis GET is on the critical path, but it is effectively free in this deployment.**

The `recall:global_epoch` key is a Redis integer — a single-field GET on a hot key with no serialization cost. Redis GET on a local Docker network returns in < 0.5ms. The Ollama embedding call (already on the critical path) takes 20–150ms. The epoch GET adds < 1% latency overhead.

However, a clean implementation uses a **request-scoped cached value**, not a per-store Redis GET, for the batch path:

- **Single store**: one Redis GET per request. Acceptable — the request already does multiple Redis operations (working memory, session counter).
- **Batch store** (up to 50 items): fetch `recall:global_epoch` **once before the item loop**, cache as `current_epoch_ceiling = fetched_epoch + EPOCH_WRITE_CEILING`, then validate each item against the cached value. This avoids 50 Redis GETs for a 50-item batch.

The cached value introduces a race window: if `recall:global_epoch` increments between the batch GET and the last item write, the ceiling check uses a slightly stale epoch. The ceiling of 10 absorbs this: the maximum legitimate INCR during a single batch request is 0 (session-end hooks fire after the store, not during). The cache is safe.

**Session state is not used**: The current `StoreMemoryRequest` carries a `session_id` but the epoch is not tracked in session state. A session-state cached epoch would require the session to be active and have a current epoch value — not guaranteed. The direct Redis GET is simpler and correct.

### (c) HTTP response code and body for ceiling violations

**Primary store path**: `422 Unprocessable Entity` with structured error body.

This is consistent with the existing write guard rejections at lines 124–127 and 213–216, which both use `HTTPException(status_code=422, detail=...)`. The detail string for epoch violations follows the write guard pattern but carries structured fields for diagnostics.

```python
raise HTTPException(
    status_code=422,
    detail={
        "error": "epoch_write_ceiling_violation",
        "message": "Rejected: last_access_epoch exceeds current epoch ceiling",
        "submitted_epoch": submitted_epoch,
        "current_redis_epoch": current_epoch,
        "ceiling": current_epoch + EPOCH_WRITE_CEILING,
        "EPOCH_WRITE_CEILING": EPOCH_WRITE_CEILING,
    },
)
```

**Why 422 and not 400?**:
- 400 (Bad Request) means malformed syntax.
- 422 (Unprocessable Entity) means syntactically valid but semantically rejected — the correct code for a value that passes type validation but fails a domain invariant. The write guard already uses 422 for the same reason.

**Why structured dict over string**:
- The existing write guard uses a plain string: `detail=f"Memory rejected by write guard: {guard_result.reason}"`. A structured dict enables the MCP layer and any monitoring consumers to parse the rejection reason programmatically without string parsing.
- FastAPI serializes a dict `detail` as `{"detail": {...}}` in the JSON response body — valid per RFC 7807 problem semantics.

**Batch store path**: no 422 at the request level. The item is added to the `results` list with `created=False`, `message="epoch_ceiling_violation: submitted={submitted_epoch}, ceiling={ceiling}"`, and `errors` counter incremented. The response is still 200 with per-item status. This matches the existing write guard rejection pattern in the batch path (lines 1225–1234).

**Logging**: The guard emits a `logger.warning("epoch_write_ceiling_violation", ...)` in addition to returning the 422. This creates an observable signal for monitoring. A violation during normal operation (not a migration run) indicates a live hook bug.

### Full code specification for primary store path

**Guard insertion in `store_memory`** (after line 133, before line 136):

```python
        # --- Epoch write ceiling guard (Q120 / FM-3 prevention) ---
        # Rejects any write where last_access_epoch exceeds the current Redis
        # epoch + EPOCH_WRITE_CEILING. Makes FM-3 (PG epoch column inflation
        # from a buggy hook writing wall-clock int(time.time())) structurally
        # impossible at the API layer.
        # EPOCH_WRITE_CEILING = 10: allows for in-flight session bursts but
        # rejects wall-clock values (~1,710,000,000) with zero false positives.
        EPOCH_WRITE_CEILING = 10
        if request.last_access_epoch is not None:
            try:
                _redis_epoch_store = await get_redis_store()
                assert _redis_epoch_store.client is not None
                _raw_epoch = await _redis_epoch_store.client.get("recall:global_epoch")
                _current_epoch = int(_raw_epoch or 0)
                _submitted_epoch = request.last_access_epoch
                if _submitted_epoch > _current_epoch + EPOCH_WRITE_CEILING:
                    logger.warning(
                        "epoch_write_ceiling_violation",
                        submitted_epoch=_submitted_epoch,
                        current_redis_epoch=_current_epoch,
                        ceiling=_current_epoch + EPOCH_WRITE_CEILING,
                        EPOCH_WRITE_CEILING=EPOCH_WRITE_CEILING,
                    )
                    raise HTTPException(
                        status_code=422,
                        detail={
                            "error": "epoch_write_ceiling_violation",
                            "message": "Rejected: last_access_epoch exceeds current epoch ceiling",
                            "submitted_epoch": _submitted_epoch,
                            "current_redis_epoch": _current_epoch,
                            "ceiling": _current_epoch + EPOCH_WRITE_CEILING,
                            "EPOCH_WRITE_CEILING": EPOCH_WRITE_CEILING,
                        },
                    )
            except HTTPException:
                raise  # ceiling violations must surface as-is
            except Exception as _epoch_guard_write_err:
                # Guard is non-fatal on Redis connectivity failure.
                # Log and continue — never block a store on guard infrastructure failure.
                logger.warning(
                    "epoch_write_guard_error", error=str(_epoch_guard_write_err)
                )
        # --- End epoch write ceiling guard ---
```

**Why the guard is non-fatal on Redis failure**: If `recall:global_epoch` is unreachable (Redis down), the guard cannot validate — but the store should still proceed. A Redis-down state is handled by the health system, not by blocking writes. The guard is a preventive measure against buggy clients, not a Redis availability gate.

**Why `HTTPException` is re-raised explicitly**: The outer `except Exception as e:` at line 430 would convert an unhandled HTTPException to a 500 — losing the 422 semantics. The existing write guard uses the same pattern at lines 426–427. The re-raise is mandatory.

### (e) Guard scope: public endpoint vs. migration paths

**Three distinct write surfaces exist**:

| Path | Guard behavior |
|------|---------------|
| `POST /memory/store` (single) | Guard active. `HTTPException(422)` on ceiling violation. |
| `POST /memory/batch/store` (up to 50) | Guard active per item. Per-item error result on violation. |
| `POST /ops/import` (JSONL migration) | Guard bypassed with `force_epoch=True` query parameter. |

**The `--force-epoch` bypass for import**:

The import path (`import_memories` in `src/api/routes/ops.py`) reads Memory objects from JSONL. During a legitimate migration, the imported JSONL may contain `last_access_epoch` values that are historically correct relative to the epoch system at export time but appear out-of-range if the receiving system's epoch counter has been reset or is at a different value. Blocking migration imports on epoch validation would make backup restoration impossible.

The bypass is implemented as a query parameter `force_epoch: bool = Query(default=False)`:

```python
@router.post("/import")
async def import_memories(
    file: UploadFile,
    conflict: str = Query(default="skip", pattern="^(skip|overwrite)$"),
    regenerate_embeddings: bool = Query(default=False),
    force_epoch: bool = Query(default=False),  # ADD: bypass epoch guard for migration
):
```

When `force_epoch=True`, the epoch ceiling guard is skipped for import items. The import path already bypasses write guard stages 1 and 2 — this is consistent with the existing design philosophy that the import endpoint is an operator-level tool (not a hook endpoint) and trusts its input.

**Security consideration for `force_epoch`**: The import endpoint should require admin-level authentication (or at minimum the API key). The current `import_memories` does not call `require_auth`. This is a pre-existing security gap, not introduced by Q120. When the epoch guard is added to the import path, it is appropriate to simultaneously add `user: User | None = Depends(require_auth)` to the import endpoint signature. Force-epoch bypass without authentication would allow any API client to write arbitrary epoch values by using the import endpoint, negating the guard entirely.

**Admin routes (`src/api/routes/admin.py`)**: `migrate_durability`, `rehabilitate_importance`, `remigrate_domains` are admin routes that do not write `last_access_epoch`. They are not write surfaces for FM-3 and require no guard.

## EPOCH_WRITE_CEILING Value Analysis

The proposed ceiling of 10 is correct for Tim's deployment.

**Why 10 is the right ceiling**:

At Tim's usage density (~5 sessions/day), the maximum number of legitimate INCRs between a hook writing `last_access_epoch` and the Redis GET in the guard is bounded by concurrent sessions. The hook architecture is sequential (one session at a time per machine, two machines max). Maximum legitimate burst: casaclaude ends a session (INCR to 852) while proxyclaude also ends a session (INCR to 853) and the next hook fires before the Redis GET returns — net delta of 3. A ceiling of 10 provides 3x headroom over the worst-case legitimate burst.

The FM-3 scenario always produces values 6+ orders of magnitude above the current epoch (wall-clock ≈ 1,710,000,000 vs epoch ≈ 850). The ceiling of 10 has zero overlap with FM-3 values. There is no tradeoff — the ceiling is both tight enough to catch FM-3 and loose enough to never false-positive in normal operation.

**Why not ceiling=0 or ceiling=1**: A ceiling of 0 would reject writes during any concurrent session burst. A ceiling of 1 would false-positive on the two-machine scenario. Ceiling = 10 is the minimum safe value with margin.

## Qualifying Ideas

- **epoch-write-guard-pre-embedding-placement**: (N=0.72, E=0.78, F=0.95) — Inserting the epoch ceiling check after Write Guard Stage 1 but before embedding is the correct latency optimization: the epoch GET is O(1) Redis on local network (< 0.5ms), while embedding is a 20–150ms Ollama network call. Ordering validation checks by cost (cheapest first) is a defensive programming principle that reduces latency on the rejection path. A ceiling violation that would be caught after 150ms of embedding work is instead caught in < 1ms. This "fail-fast cheap gate" ordering applies to all multi-stage validation pipelines, not just epoch guards.

- **guard-fatal-on-write-nonfatal-on-read**: (N=0.68, E=0.82, F=0.98) — The Q111 retrieve-path epoch guard is explicitly non-fatal (wrapped in `try/except`, logs warning, continues). The Q120 write-path epoch guard is fatal (raises `HTTPException(422)`). The asymmetry is principled: a retrieval error degrades quality but the user still gets results; a write error must reject the write because storing an inflated epoch cannot be reversed without manual intervention. This read-nonfatal/write-fatal asymmetry is a general invariant for data integrity guards in hybrid read-write APIs.

- **batch-ceiling-cached-epoch-pattern**: (N=0.65, E=0.75, F=0.98) — Fetching `recall:global_epoch` once before the item loop in `batch_store_memories` and caching as `current_ceiling` avoids N Redis GETs for an N-item batch. The cached value is safe because the maximum legitimate epoch advance during a single HTTP request is 0 (session-end INCRs happen after store returns, not during). This "read-once, validate-many" pattern applies to any per-item validation that depends on a shared counter — rate limit windows, sequence numbers, version fences.

- **force-epoch-flag-requires-auth-coupling**: (N=0.70, E=0.72, F=0.88) — Introducing a `force_epoch=True` bypass on the import endpoint simultaneously exposes the need to add `require_auth` to that endpoint. The bypass creates a new attack surface (any unauthenticated caller could write arbitrary epochs). The correct implementation couples the bypass to admin authentication. This is a general principle: any guard bypass flag should be treated as a privilege escalation point and requires stronger auth than the base endpoint, not weaker.

## Recommendations

1. **Add `last_access_epoch: int | None` to `StoreMemoryRequest`** with `ge=0` constraint. Default `None` — existing callers that don't send the field are unaffected. The guard only activates when the field is explicitly present.

2. **Insert the epoch ceiling guard** in `store_memory` at `src/api/routes/memory.py` after line 133 (after Write Guard Stage 1), before line 136 (durability resolve). Use `EPOCH_WRITE_CEILING = 10`. Guard is fatal on ceiling violation (422), non-fatal on Redis connectivity failure.

3. **Mirror the guard in `batch_store_memories`** — fetch `recall:global_epoch` once before the item loop, validate each item's `last_access_epoch` against the cached ceiling, produce per-item error result on violation.

4. **Add `force_epoch: bool = Query(default=False)` to `import_memories`** in `src/api/routes/ops.py`. Simultaneously add `user: User | None = Depends(require_auth)` to that endpoint — the bypass creates a new privilege escalation surface that requires authentication coverage.

5. **Deploy in sequence** (same deployment gate as Q111): SQL migration first (adds `last_access_epoch` column to `memories` — see Q111) → then code deploy (guard + schema change). Running the guard code before the column exists is harmless (the guard reads Redis, not PG), but the store endpoint writing `last_access_epoch` to a PG column that doesn't exist would fail. The deployment order constraint is on the PG write side, not the guard side.

6. **Implement the 9-case unit test suite** in `tests/api/test_epoch_write_guard.py` with mocked Redis. The critical test cases are Test 3 (one-above-ceiling rejected), Test 4 (wall-clock rejected), Test 6 (Redis-down non-fatal), and Test 8 (batch per-item behavior).

## Verdict Rationale

**BREAKTHROUGH**: The guard closes a structural gap in the epoch integrity system that Q117 identifies but cannot close reactively. Q117's discriminant (check `redis_epoch` absolute value after S4 fires) catches FM-3 after the inflated value is already in PostgreSQL — at that point recovery requires manual PG rollback and working-set filter repair. The Q120 guard prevents FM-3 from ever entering the store: the rejection happens in the HTTP layer before any PG write. The combination of (a) correct insertion point (pre-embedding, post-write-guard Stage 1), (b) non-fatal-on-Redis-down behavior, (c) fatal-on-violation behavior, (d) migration bypass via `force_epoch` flag, and (e) ceiling=10 calibration for Tim's deployment density constitutes a complete, immediately deployable specification that fully implements the FM-3 prevention layer. The guard is novel in the context of this epoch system (no existing analog in the codebase), has strong analytical evidence, and is implementable in approximately 30 lines of code across two functions.
