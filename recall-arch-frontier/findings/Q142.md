## Current Hook Analysis

### What the hook does (v2.9.4)

`recall-session-summary.js` is a Stop hook that fires when a Claude Code session ends. Its pipeline:

1. Reads stdin JSON: `{ transcript_path, cwd, session_id }`
2. Parses the session transcript (last 200 lines of JSONL from `transcript_path`)
3. Extracts user messages and all messages
4. If fewer than 2 user messages: exits immediately (no-op for trivial sessions)
5. **In parallel**:
   - Submits feedback on injected memories (`POST /memory/feedback`)
   - Builds an LLM summary via Ollama qwen3:14b (4s timeout), fallback to regex-built summary
6. **Stores the episodic summary**: `POST /memory/store` with `memory_type: "episodic"`, `tags: ["session-summary", projectName]`
7. Fire-and-forget: extracts key decisions via Ollama (15s timeout), stores each as `memory_type: "semantic"`
8. Fire-and-forget: posts `POST /observe/markov-session-end` to finalize Markov chain session transitions
9. Hard cap: force-exits after 8s to never block Claude Code's stop sequence

### What data a session summary contains

The payload sent to `POST /memory/store`:

```json
{
  "content": "<2-3 sentence LLM summary or regex fallback>",
  "domain": "<mapped from cwd: 'development' | 'ai-ml' | 'infrastructure' | 'general'>",
  "source": "system",
  "memory_type": "episodic",
  "tags": ["session-summary", "<projectName>"],
  "importance": 0.5
}
```

Key fields required for replay:
- `content` (the summary text, ≤2000 chars)
- `domain`
- `tags`
- `importance`
- `cwd` (for domain re-derivation on retry; store for auditability)
- `session_id` (for correlation; not sent to API but useful for deduplication)
- `created_at` (original session-end timestamp; API assigns `write_seq` at submission time per Q056)

### Current failure mode

When `POST /memory/store` raises a network error or timeout (line 401: `catch (err) { debug(...) }`), the summary is silently dropped. The debug log at `~/.cache/recall/session-summary-debug.log` records the failure but there is no retry path. The session summary is permanently lost.

The `extractKeyDecisions` call (lines 407–428) and the `markov-session-end` call (lines 431–438) are fire-and-forget `.catch(() => {})` — these are already designed to fail silently. Only the episodic summary (lines 385–404) is the critical Tier A write.

## Implementation Spec

### Why SQLite over JSON files

Q056 recommended `~/.claude/recall-pending/session-{timestamp}.json` files. This finding upgrades that recommendation to SQLite for three reasons:

1. **Atomic write**: SQLite `INSERT` is atomic. A JSON file write can leave a partially-written file if the hook is killed mid-write (hard cap at 8s, machine suspend). SQLite WAL mode guarantees the row is either fully written or not at all.
2. **Single flush lock**: Multiple Claude Code instances running simultaneously (Tim's casaclaude + proxyclaude pattern) each try to flush pending summaries. SQLite's file-level locking prevents duplicate submissions without additional coordination.
3. **TTL/cleanup**: SQLite makes it trivial to implement a `delivered_at IS NOT NULL` cleanup and a `created_at < now()-30d` stale-entry purge. JSON files require directory listing + mtime parsing.

The implementation cost is identical: Node.js has `better-sqlite3` (synchronous) available, or the native `node:sqlite` module in Node 22+. The hook already uses synchronous `readFileSync` / `appendFileSync` — a synchronous SQLite write fits the existing pattern.

### SQLite Schema

Database location: `~/.cache/recall/pending.db`

(Same `CACHE_DIR` used by the existing debug log at line 22: `join(HOME, ".cache", "recall")`)

```sql
CREATE TABLE IF NOT EXISTS pending_summaries (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id  TEXT    NOT NULL,
  cwd         TEXT    NOT NULL,
  content     TEXT    NOT NULL,
  domain      TEXT    NOT NULL DEFAULT 'general',
  tags        TEXT    NOT NULL DEFAULT '["session-summary"]',  -- JSON array serialized as TEXT
  importance  REAL    NOT NULL DEFAULT 0.5,
  created_at  TEXT    NOT NULL,  -- ISO-8601 timestamp of original session end
  attempts    INTEGER NOT NULL DEFAULT 0,
  last_attempt_at TEXT,          -- ISO-8601; NULL if never attempted
  delivered_at    TEXT,          -- ISO-8601; NULL until successfully stored to Recall API
  error_hint      TEXT           -- last error message, for debug
);

CREATE INDEX IF NOT EXISTS idx_pending_summaries_delivered
  ON pending_summaries (delivered_at)
  WHERE delivered_at IS NULL;
```

Columns:

| Column | Type | Purpose |
|--------|------|---------|
| `id` | INTEGER PK AUTOINCREMENT | Row identity; used as stable reference during flush |
| `session_id` | TEXT | Original session_id from hook stdin; used to detect duplicate flush attempts from the same session |
| `cwd` | TEXT | Working directory at session end; re-derives `domain` if missing |
| `content` | TEXT | The summary text (≤2000 chars) |
| `domain` | TEXT | Recall domain classification |
| `tags` | TEXT | JSON-serialized array (e.g., `["session-summary","recall-arch-frontier"]`) |
| `importance` | REAL | 0.4 (fallback) or 0.5 (LLM summary); stored as recorded |
| `created_at` | TEXT | ISO-8601 timestamp of original session end; sent as metadata; API assigns authoritative `write_seq` at replay time |
| `attempts` | INTEGER | Retry attempt count; for observability; no hard retry cap (session summaries are Tier A — retry indefinitely) |
| `last_attempt_at` | TEXT | ISO-8601 timestamp of most recent flush attempt |
| `delivered_at` | TEXT | ISO-8601 timestamp when `POST /memory/store` returned HTTP 2xx; NULL until delivered |
| `error_hint` | TEXT | Last error message from failed attempt; debug only |

**No hard retry cap** — unlike the co_retrievals dead-letter queue (Q130, 5 attempts max), session summaries are Tier A must-not-lose. They should retry indefinitely until delivered or manually purged. The `attempts` counter is for observability, not for expiry.

**Stale purge** — entries where `created_at < now() - 90 days AND delivered_at IS NOT NULL` are safe to delete. Entries where `delivered_at IS NULL AND created_at < now() - 90 days` should be logged as a warning but not auto-deleted (user should decide).

### Write Path

The write to SQLite happens **before** the network call to `POST /memory/store`. This is the critical ordering: the summary must be durably recorded locally before any attempt to deliver it remotely. If the hook crashes or times out after the SQLite write but before the network call, the summary is preserved and will be flushed on the next session start.

Pseudocode (inserted after line 382, replacing the existing `if (finalSummary)` block at lines 385–404):

```
function openDb():
  mkdirSync(CACHE_DIR, { recursive: true })
  db = new Database(join(CACHE_DIR, "pending.db"))
  db.exec(CREATE TABLE IF NOT EXISTS pending_summaries ...)
  db.exec(CREATE INDEX IF NOT EXISTS ...)
  return db

function writePendingEntry(db, sessionId, cwd, content, domain, tags, importance):
  // Check for existing undelivered entry from the same session (idempotency guard)
  existing = db.prepare(
    "SELECT id FROM pending_summaries WHERE session_id = ? AND delivered_at IS NULL"
  ).get(sessionId)
  if existing: return existing.id  // already queued; do not insert duplicate

  stmt = db.prepare("""
    INSERT INTO pending_summaries
      (session_id, cwd, content, domain, tags, importance, created_at)
    VALUES (?, ?, ?, ?, ?, ?, ?)
  """)
  result = stmt.run(sessionId, cwd, content, domain, JSON.stringify(tags), importance, new Date().toISOString())
  return result.lastInsertRowid

function attemptDelivery(db, rowId, headers, RECALL_HOST):
  row = db.prepare("SELECT * FROM pending_summaries WHERE id = ?").get(rowId)
  if not row or row.delivered_at: return true  // already delivered

  db.prepare("""
    UPDATE pending_summaries SET attempts = attempts + 1, last_attempt_at = ?
    WHERE id = ?
  """).run(new Date().toISOString(), rowId)

  try:
    resp = await fetch(`${RECALL_HOST}/memory/store`, {
      method: "POST",
      headers,
      body: JSON.stringify({
        content:     row.content,
        domain:      row.domain,
        source:      "system",
        memory_type: "episodic",
        tags:        JSON.parse(row.tags),
        importance:  row.importance,
      }),
      signal: AbortSignal.timeout(3000),
    })
    if resp.ok:
      db.prepare("UPDATE pending_summaries SET delivered_at = ? WHERE id = ?")
        .run(new Date().toISOString(), rowId)
      debug(`summary delivered (row ${rowId}): ${resp.status}`)
      return true
    else:
      db.prepare("UPDATE pending_summaries SET error_hint = ? WHERE id = ?")
        .run(`HTTP ${resp.status}`, rowId)
      debug(`summary delivery failed (row ${rowId}): HTTP ${resp.status}`)
      return false
  catch err:
    db.prepare("UPDATE pending_summaries SET error_hint = ? WHERE id = ?")
      .run(err.message.slice(0, 200), rowId)
    debug(`summary delivery error (row ${rowId}): ${err.message}`)
    return false

// In main(), replacing the existing summary store block:
if (finalSummary):
  try:
    db = openDb()
    rowId = writePendingEntry(db, sessionId, cwd, finalSummary, domain,
                              ["session-summary", projectName], importance)
    debug(`queued summary row ${rowId}`)
    await attemptDelivery(db, rowId, headers, RECALL_HOST)
  catch sqliteErr:
    // SQLite unavailable — fall back to direct network call (see Failure Mode section)
    debug(`SQLite unavailable: ${sqliteErr.message} — attempting direct store`)
    try:
      resp = await fetch(`${RECALL_HOST}/memory/store`, { ... existing payload ... })
      debug(`direct summary stored: ${resp.status}`)
    catch netErr:
      debug(`direct store also failed: ${netErr.message} — summary lost`)
```

**Key ordering guarantee**: The SQLite `INSERT` completes before the `fetch()` is issued. If the hook is killed by the 8s hard cap after the INSERT but before the fetch returns, the entry remains in the database with `delivered_at = NULL` and `attempts = 0`. It will be flushed at the next session start.

### Flush Path

The flush runs at the **start** of the next hook invocation. It must run before the session's own summary is queued, so that pending summaries from prior sessions are submitted first (preserving temporal order in Recall's memory timeline).

The flush belongs in `recall-session-summary.js` itself at the start of `main()`, not in `recall-retrieve.js`. Rationale: the Stop hook already has the network headers, RECALL_HOST, and auth key configured. Adding flush logic to `recall-retrieve.js` (the UserPromptSubmit hook) would require duplicating the auth setup and could add latency to session start. The Stop hook runs at session end — a moment when a few hundred milliseconds of flush work has no user-visible impact.

Pseudocode (inserted near the start of `main()`, after `parsed` is available but before `userMessages` is extracted):

```
async function flushPendingEntries(db, headers, RECALL_HOST):
  pending = db.prepare("""
    SELECT id FROM pending_summaries
    WHERE delivered_at IS NULL
    ORDER BY created_at ASC
  """).all()

  debug(`flush: ${pending.length} pending entries`)
  let flushed = 0
  for (const { id } of pending):
    // Respect the 8s hard cap — don't spend more than 2s per entry
    delivered = await attemptDelivery(db, id, headers, RECALL_HOST)
    if delivered: flushed++
    else: break  // If Recall API is still unreachable, stop trying (avoid timeout burn)

  debug(`flush: delivered ${flushed}/${pending.length}`)
  return flushed

// In main(), after headers are constructed (after line 364):
try:
  db = openDb()
  await flushPendingEntries(db, headers, RECALL_HOST)
catch sqliteErr:
  debug(`flush: SQLite unavailable: ${sqliteErr.message}`)
```

**Early-exit on first delivery failure**: If the Recall API is still unreachable (the condition that caused the original deferred write), the first `attemptDelivery` call will fail. There is no point trying the remaining entries. `break` on first failure and let the next session handle them.

**Temporal ordering**: Pending entries are flushed in `created_at ASC` order. This means older sessions are submitted before newer ones, preserving the temporal order in Recall's memory timeline. The API assigns authoritative `write_seq` at submission time (Q056 principle: global sequence numbers assigned at API ingestion, not at buffer creation).

**Cleanup**: After flushing, run periodic cleanup of old delivered entries:

```
db.prepare("""
  DELETE FROM pending_summaries
  WHERE delivered_at IS NOT NULL
    AND delivered_at < datetime('now', '-30 days')
""").run()
```

Run this once per session start (before the flush), not on every hook invocation. The `delivered_at IS NOT NULL` guard ensures only successfully-delivered entries are purged.

### Failure Mode: SQLite Unavailable

SQLite can be unavailable in three scenarios:

**Scenario 1: `~/.cache/recall/` does not exist and `mkdirSync` fails** (permissions, disk full)
- Catch the `mkdirSync` or `Database()` constructor error in `openDb()`
- Fall back to the **direct network call** (the current behavior — no SQLite queue)
- Log: `"SQLite unavailable: ${err.message} — attempting direct store (no durability guarantee)"`
- If the direct network call also fails, the summary is lost — same as current behavior
- This is the correct graceful degradation: SQLite unavailability reduces durability to current levels, not below

**Scenario 2: SQLite database is locked** (two Claude Code instances stopping simultaneously)
- `better-sqlite3` uses synchronous blocking writes; the second write will block until the first releases the lock (default timeout: ~5 seconds)
- The 8s hard cap is above this; the second instance will succeed
- If the lock wait exceeds the hard cap budget: catch `SQLITE_BUSY` error, fall back to direct network call

**Scenario 3: SQLite database is corrupted** (disk error during write)
- SQLite WAL mode (`PRAGMA journal_mode=WAL`) limits corruption to the most recent uncommitted transaction
- On open, better-sqlite3 will throw if the database is corrupted
- Catch in `openDb()`, attempt to delete and recreate the database file, retry once
- If recreation fails: fall back to direct network call

**Graceful degradation summary**:

```
SQLite available AND Recall API available → normal delivery, delivered_at set
SQLite available AND Recall API unavailable → entry queued, retried next session
SQLite unavailable AND Recall API available → direct delivery (no durability queue)
SQLite unavailable AND Recall API unavailable → summary lost (same as current behavior)
```

The SQLite queue converts the "API unavailable" failure from a data loss event to a deferred write. It does not protect against simultaneous SQLite + API failure, which is the same loss condition as today.

### ~11 Losses/Year Calculation Verification

**From Q056 (inline calculation)**:

```
P(loss) = downtime_seconds_per_year × session_end_rate_per_second
        = (0.01 × 365 × 24 × 3600) × (3 / (24 × 3600))
        = 315,360 s/yr × 0.0000347 session-ends/s
        ≈ 10.94 ≈ 11 lost summaries/year
```

**Verification with explicit assumptions**:

- CasaOS uptime: 99% → downtime fraction = 0.01
- Downtime per year: `0.01 × 365 × 24 × 3600 = 315,360 seconds/year`
- Session rate: 3 sessions/day (from Q056; Tim's stated usage pattern)
- Session-end rate: `3 / 86,400 = 3.47 × 10⁻⁵ session-ends/second`
- Session-end distribution: assumed uniform over 24h (conservative; in practice sessions cluster during waking hours, which may or may not correlate with CasaOS maintenance windows)

**Calculation**:

```
E[lost summaries/year] = downtime_per_year_s × session_end_rate_per_s
                       = 315,360 × 3.47 × 10⁻⁵
                       = 10.94 ≈ 11
```

**Sensitivity analysis**:

| CasaOS Uptime | Sessions/Day | Losses/Year |
|---------------|-------------|-------------|
| 99.0% | 3 | **10.9** (base case) |
| 99.9% | 3 | 1.1 |
| 99.0% | 5 | 18.2 |
| 99.5% | 3 | 5.5 |
| 99.0% | 2 | 7.3 |

**Distribution of downtime events**: CasaOS runs on Docker. The 315,360 s/year downtime is not uniformly distributed — it likely concentrates in brief restart events (30–120s each) rather than extended outages. If downtime events are 60s each: `315,360 / 60 ≈ 5,256 restart events/year ≈ 14.4/day`. Each restart has a probability of `3.47 × 10⁻⁵ × 60 = 0.0021` of catching a session end. Over 5,256 events: `5,256 × 0.0021 = 11` — same result, consistent with the continuous-time calculation.

**Conclusion**: The ~11 losses/year estimate is mathematically correct given the stated assumptions (99% uptime, 3 sessions/day). It is an expected value; actual losses follow a Poisson distribution with λ=10.94 (variance = 10.94, standard deviation = 3.3). In a given year, 95% probability of 5–18 losses.

## Finding

[OBJECTIVE] The W3 session summary write (recall-session-summary.js Stop hook) currently has no local durability buffer. The hook attempts one network call to `POST /memory/store`; if it fails, the session summary is permanently lost with no retry path and no user notification. [DATA] The hook fires at session end, calls `RECALL_HOST/memory/store` with `memory_type: "episodic"` payload (content ≤2000 chars, domain, tags, importance=0.5), and has an 8-second hard cap before force-exit. Failure is logged to `~/.cache/recall/session-summary-debug.log` but not retried. [FINDING] The complete SQLite-backed pending queue can be implemented in approximately 80 lines of JavaScript added to the existing hook, with zero API changes, zero schema changes on the Recall server, and zero new processes. The SQLite database lives at `~/.cache/recall/pending.db` alongside the existing debug log. The write path queues the summary before the network call (write-then-send, not send-then-queue). The flush path runs at the start of the next Stop hook invocation, before the current session's summary is queued, preserving temporal ordering. SQLite unavailability degrades gracefully to current behavior (direct network call). [STAT:n=10.94] Expected session summary losses per year at 99% CasaOS uptime with 3 sessions/day. [STAT:effect_size=1.0] Implementing the SQLite queue reduces expected annual loss from 10.94 to 0 for all transient API failures (network blips, CasaOS restarts ≤ hours); the only remaining loss mode requires simultaneous SQLite failure + API unavailability, which is negligible.

## Qualifying Ideas

[N≥0.65, E≥0.40]

- **sqlite-queue-for-stop-hook-writes**: (N=0.72, E=0.85, F=0.99) — SQLite as a local durability layer for webhook-style fire-once hook writes. The pattern (queue-before-send, flush-on-next-invocation) is well-established in event sourcing and outbox literature (Hohpe & Woolf Transactional Outbox, 2003; Firebase offline writes, 2016), but applying it specifically to Claude Code Stop hooks is novel. No known production AI coding tool implements SQLite-backed durable hook writes. Implementation: ~80 lines of JS in an existing hook file, uses synchronous `better-sqlite3` or Node 22 native `node:sqlite`, WAL mode for crash safety. The write path guarantees summary durability even if the 8s hard cap fires between the SQLite INSERT and the fetch() call. Score: (0.72×0.40) + (0.85×0.35) + (0.99×0.25) = 0.288 + 0.298 + 0.248 = **0.834**

- **write-then-send-not-send-then-write**: (N=0.68, E=0.90, F=0.99) — The critical ordering principle: the durable record must be written before the delivery attempt, never after. "Send then queue on failure" has a TOCTOU window: if the process is killed between send and queue, the data is lost even though delivery succeeded (idempotency concern) or not (data loss). "Write then send, mark delivered on success" eliminates this window. This is the same principle as the PostgreSQL transactional outbox pattern (Q130): the outbox entry is written in the same transaction as the application state change, before any external call. Applying this to a JavaScript hook context (no transactions, but atomic SQLite INSERT) yields the same safety guarantee. Score: (0.68×0.40) + (0.90×0.35) + (0.99×0.25) = 0.272 + 0.315 + 0.248 = **0.835**

- **no-retry-cap-for-tier-a-writes**: (N=0.65, E=0.80, F=0.99) — The co_retrievals dead-letter queue (Q130) uses a 5-attempt cap, after which entries move to `co_retrievals_dead_letter`. This is appropriate for Tier B writes (eventually-consistent reinforcement signals). Session summaries are Tier A (must-not-lose, irreplaceable). A hard retry cap on a Tier A write converts deferred failure into guaranteed loss after N attempts — exactly wrong. The correct design for Tier A pending entries is indefinite retry (attempts counter for observability, no expiry) combined with a stale-entry warning threshold (log a warning if `created_at < now() - 30 days AND delivered_at IS NULL`). This preserves the "never lose" guarantee while surfacing persistently-stuck entries for human inspection. Score: (0.65×0.40) + (0.80×0.35) + (0.99×0.25) = 0.260 + 0.280 + 0.248 = **0.788**

- **session-summary-deduplication-via-session-id**: (N=0.70, E=0.75, F=0.99) — The pending_summaries table uses `session_id` as a deduplication key: `INSERT` is skipped if an undelivered row for the same session_id already exists. This handles the case where the Stop hook fires multiple times for the same session (edge case: Claude Code retry on non-zero exit, or manual re-invocation). Without deduplication, each hook invocation would insert a new pending row, and the same summary would be delivered multiple times to Recall. Since session summaries are episodic memories indexed by content embedding, duplicate submissions create near-duplicate vectors in Qdrant. The `session_id` guard prevents this. Score: (0.70×0.40) + (0.75×0.35) + (0.99×0.25) = 0.280 + 0.263 + 0.248 = **0.791**
