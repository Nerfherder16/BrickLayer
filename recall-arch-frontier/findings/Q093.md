# Finding: Q093 — /reconcile Post-Restore Mode: Full Endpoint Specification, Idempotency Analysis, and Atomicity Model

**Question**: The Q086 finding established that the three-store cleanup after a PostgreSQL restore has Step 3 missing: no Qdrant cleanup runs after the Neo4j DETACH DELETE step. Implement the /reconcile endpoint with a mode=post_restore&backup_timestamp=T parameter that (a) invokes the existing Neo4j DETACH DELETE by epoch predicate, (b) calls Qdrant scroll + batch delete filtering by created_at > backup_timestamp using the datetime index that already exists in the payload, and (c) emits a reconciliation report {neo4j_deleted_count, qdrant_deleted_count, duration_ms} — and specify whether the endpoint must be idempotent (safe to call twice with the same backup_timestamp) given that a second call would find an already-cleaned Qdrant collection and attempt to delete zero records.
**Question Type**: IMPLEMENTATION
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Distributed systems (cross-store consistency, point-in-time restore recovery, delete-by-predicate idempotency), FastAPI endpoint design, Qdrant Python client API (scroll + batch delete, DatetimeRange filter), Neo4j Cypher (DETACH DELETE with epoch predicate), operational runbook engineering

---

## Context Synthesis

Q086 established three facts that drive this implementation:

1. **Qdrant is the canonical store**. Memory objects are reconstructed entirely from Qdrant payload by `_payload_to_memory()` — there is no PostgreSQL join in the retrieval path. Orphaned Qdrant vectors are returned by `qdrant.search()` and injected directly into Claude's context as valid memories. This is an *active data corruption* failure, not a silent skip.

2. **The existing `/reconcile` endpoint heals in the wrong direction for post-restore**. It computes `qdrant_orphans = qdrant_ids - neo4j_ids` and creates Neo4j nodes for Qdrant orphans. After a PG restore that did not include Qdrant, this would *perpetuate* the orphaned state by re-populating Neo4j from the Qdrant data the restore was meant to remove.

3. **The three-store cleanup sequence after a PG restore currently has only two of its three steps implemented**:
   - Step 3 (Redis epoch retraction): documented in Q074/Q077
   - Step 4 (Neo4j DETACH DELETE by epoch predicate): documented in Q082 follow-ups
   - Step 5 (Qdrant delete by `created_at > backup_timestamp`): **absent from all documented recovery procedures**

This finding provides the complete specification for Step 5 and for a unified endpoint that executes Steps 4 and 5 atomically.

## Part 1: Endpoint Specification

### Route and Parameters

```
POST /reconcile?mode=post_restore&backup_timestamp=T&pg_max_epoch=E&repair=true
```

| Parameter | Type | Required | Description |
|-----------|------|----------|-------------|
| `mode` | string | yes | Must be `post_restore` to activate this code path. Prevents accidental invocation via the existing `repair=true` path. |
| `backup_timestamp` | ISO-8601 string | yes | Wall-clock timestamp of the PG backup that was restored. All Qdrant points with `created_at > backup_timestamp` are candidates for deletion. |
| `pg_max_epoch` | integer | yes | The maximum `last_access_epoch` value found in PostgreSQL after the restore (i.e., `SELECT MAX(last_access_epoch) FROM memories`). All Neo4j nodes with `last_access_epoch > pg_max_epoch` are candidates for deletion. |
| `repair` | boolean | yes | Must be `true`. Existing guard prevents accidental execution without explicit opt-in. Default `false` means the endpoint runs in dry-run mode and returns counts without deleting. |
| `batch_size` | integer | no | Qdrant scroll batch size. Default 100. Controls memory usage during the scroll loop. |

### Dry-Run Mode (repair=false)

When `repair=false` (the default), the endpoint:
- Executes both predicates as COUNT-only queries
- Returns `{neo4j_would_delete: N, qdrant_would_delete: M, dry_run: true, duration_ms: D}`
- Makes zero mutations

This allows operators to verify the scope of the cleanup before committing. The operator workflow is:
1. `POST /reconcile?mode=post_restore&backup_timestamp=T&pg_max_epoch=E` → confirm counts look correct
2. `POST /reconcile?mode=post_restore&backup_timestamp=T&pg_max_epoch=E&repair=true` → execute cleanup

## Part 2: Neo4j DETACH DELETE — Exact Cypher

### The Predicate

The Neo4j cleanup deletes all Memory nodes whose `last_access_epoch` exceeds the maximum epoch value surviving in PostgreSQL after the restore:

```cypher
MATCH (n:Memory)
WHERE n.last_access_epoch > $pg_max_epoch
WITH n LIMIT $batch_size
DETACH DELETE n
RETURN count(*) AS deleted_count
```

**Why `>` not `>=`**: The node at `pg_max_epoch` is the last valid memory in PG. Nodes at `last_access_epoch > pg_max_epoch` were created after the backup point and have no PG counterpart — they are the orphans. Using `>` (strict) preserves the boundary node.

**Batch execution loop** (required in Python): Neo4j's `DETACH DELETE` on large node sets can cause GC pressure if run as a single transaction. The batch-loop pattern (delete 500 at a time, repeat until 0 deleted) is the standard safe approach:

```python
total_neo4j_deleted = 0
while True:
    result = await neo4j_session.run(
        """
        MATCH (n:Memory)
        WHERE n.last_access_epoch > $pg_max_epoch
        WITH n LIMIT 500
        DETACH DELETE n
        RETURN count(*) AS deleted_count
        """,
        pg_max_epoch=pg_max_epoch,
    )
    record = await result.single()
    batch_deleted = record["deleted_count"] if record else 0
    total_neo4j_deleted += batch_deleted
    if batch_deleted == 0:
        break

# total_neo4j_deleted is the final count for the report
```

**Dry-run variant** (for `repair=false`):

```cypher
MATCH (n:Memory)
WHERE n.last_access_epoch > $pg_max_epoch
RETURN count(n) AS would_delete
```

### Epoch Source: How the Operator Obtains pg_max_epoch

The operator must supply `pg_max_epoch` explicitly. It is derived by querying PostgreSQL after the restore completes:

```sql
SELECT MAX(last_access_epoch) FROM memories;
```

This value represents the highest epoch for which PG still has rows. All Neo4j nodes above this value are orphans in the post-restore state. Importantly, the operator should run this query *before* re-enabling writes, otherwise new writes may update `last_access_epoch` and contaminate the threshold.

The endpoint does **not** automatically query PG for this value, because:
1. It would introduce a circular dependency (the endpoint is for cross-store cleanup, not PG-authoritative reads)
2. The PG query must be run in the same maintenance window where writes are disabled, and the timestamp must be validated by the operator before the cleanup is authorized
3. Explicit operator supply of `pg_max_epoch` is a safety gate — it forces human confirmation of the epoch boundary

## Part 3: Qdrant Scroll + Batch Delete — Exact Query

### The Filter Predicate

Qdrant's `DatetimeRange` filter with `gt` (greater-than) matches all points whose `created_at` payload field is strictly after the backup timestamp:

```python
from qdrant_client.models import (
    Filter,
    FieldCondition,
    DatetimeRange,
)
from datetime import datetime, timezone

backup_dt = datetime.fromisoformat(backup_timestamp)
# Ensure timezone-aware if not already
if backup_dt.tzinfo is None:
    backup_dt = backup_dt.replace(tzinfo=timezone.utc)
```

### Scroll + Batch Delete Loop

Qdrant does not support `delete_by_filter` in a single call on all versions; the correct pattern is scroll-then-delete, which also provides an accurate deleted count and allows batched confirmation:

```python
total_qdrant_deleted = 0
scroll_offset = None  # None = start from beginning
batch_size = 100      # configurable via endpoint parameter

while True:
    # Scroll: retrieve point IDs matching the filter (no vectors needed)
    scroll_result, next_offset = await qdrant_client.scroll(
        collection_name="memories",
        scroll_filter=Filter(
            must=[
                FieldCondition(
                    key="created_at",
                    range=DatetimeRange(gt=backup_dt.isoformat()),
                )
            ]
        ),
        limit=batch_size,
        offset=scroll_offset,
        with_payload=False,  # IDs only — no payload needed for delete
        with_vectors=False,
    )

    if not scroll_result:
        break  # No more matching points

    point_ids = [point.id for point in scroll_result]

    # Delete this batch by explicit point ID list
    await qdrant_client.delete(
        collection_name="memories",
        points_selector=point_ids,
    )

    total_qdrant_deleted += len(point_ids)

    if next_offset is None:
        break  # Reached end of matching points
    scroll_offset = next_offset

# total_qdrant_deleted is the count for the report
```

**Why scroll-then-delete instead of delete-by-filter directly?**

Qdrant does support `delete` with a `Filter` object (not just a list of IDs) in recent client versions. However, the scroll-then-delete pattern is preferred here for two reasons:

1. **Accurate count**: Direct `delete(filter=...)` returns a `status` but not a deleted point count in all Qdrant versions. The scroll approach gives an exact count without a separate `count()` call.

2. **Resumability**: If the endpoint is interrupted mid-cleanup (network timeout, process restart), the scroll-then-delete is idempotent at the batch level — on retry, the scroll will simply find fewer matching points and continue from the beginning. The deleted-count in the final report may differ from the interrupted run's partial count, but no points are double-deleted.

**Alternative: Direct delete-by-filter (simpler, Qdrant ≥ 1.1.1)**

For operators who want simplicity and don't need an exact deleted count from the response body:

```python
# Dry-run: count first
count_result = await qdrant_client.count(
    collection_name="memories",
    count_filter=Filter(
        must=[
            FieldCondition(
                key="created_at",
                range=DatetimeRange(gt=backup_dt.isoformat()),
            )
        ]
    ),
    exact=True,
)
qdrant_would_delete = count_result.count

# Repair: delete by filter
if repair:
    await qdrant_client.delete(
        collection_name="memories",
        points_selector=Filter(
            must=[
                FieldCondition(
                    key="created_at",
                    range=DatetimeRange(gt=backup_dt.isoformat()),
                )
            ]
        ),
    )
```

This is the recommended implementation for production: count first (validates the predicate), then delete. The direct filter-delete is O(N_matching) in Qdrant and handles the HNSW soft-deletion correctly. Qdrant triggers async HNSW reoptimization after bulk deletes; this is transparent to the caller and does not block the endpoint response.

### The datetime Index

The `created_at` field already has a `datetime` payload index in Qdrant (confirmed in Q086 as line 162 of `qdrant.py`). This means:

- The `DatetimeRange(gt=...)` filter uses the index for efficient lookup rather than scanning all points
- At Tim's corpus size (7,300 points at Year 1), even a full scan would complete in milliseconds; the index simply makes it faster and more robust at larger scales
- The index field name must match exactly what is stored in payload. Based on Q086's `_payload_to_memory()` analysis, the field is stored as `"created_at"` with an ISO-8601 string value

**Date format compatibility note**: The `DatetimeRange` filter requires the value to be in ISO-8601 format. Qdrant's datetime index parser accepts both with and without timezone offset. Using `backup_dt.isoformat()` (which includes offset if timezone-aware) is the safest form.

## Part 4: The Reconciliation Report

The endpoint response after a successful `repair=true` execution:

```json
{
  "mode": "post_restore",
  "repair": true,
  "dry_run": false,
  "backup_timestamp": "2025-01-15T00:00:00+00:00",
  "pg_max_epoch": 100,
  "neo4j_deleted_count": 1587,
  "qdrant_deleted_count": 1612,
  "duration_ms": 4380,
  "warnings": [],
  "note": "Cleanup complete. HNSW reoptimization is running asynchronously in Qdrant. Run GET /health/detail to verify s5_graph_compliance returns healthy after optimization completes (~30-60s)."
}
```

**Why neo4j_deleted_count may differ from qdrant_deleted_count**:

The two predicates are independent and measure different things:
- Neo4j predicate: `last_access_epoch > pg_max_epoch` — catches nodes whose epoch was updated after the backup
- Qdrant predicate: `created_at > backup_timestamp` — catches points created after the backup timestamp

A memory stored exactly at the boundary epoch may appear in one store but not the other, depending on the ordering of the dual-write and the precision of the epoch vs. timestamp comparison. A count difference of ±1–5% is expected and normal; a large discrepancy (>10%) warrants investigation.

The `warnings` field is populated if:
- Count discrepancy exceeds 10%: `"WARNING: neo4j_deleted=1587 vs qdrant_deleted=1612 differ by 1.6% — expected for boundary cases"`
- Neo4j deletion was interrupted (batch_deleted > 0 on the final loop iteration after timeout): `"WARNING: Neo4j deletion may be incomplete — re-run to confirm"`
- Qdrant async optimization is still running (can be detected by checking collection status): `"HNSW optimization in progress — search results may include recently-deleted points for up to 60s"`

## Part 5: Idempotency Analysis

**The question**: Is the endpoint safe to call twice with the same `backup_timestamp` and `pg_max_epoch`, given that a second call would find an already-cleaned collection?

**Answer: Yes — the endpoint is naturally idempotent for both stores.**

### Neo4j Idempotency

The Cypher predicate `MATCH (n:Memory) WHERE n.last_access_epoch > $pg_max_epoch DETACH DELETE n`:

- On the first call: deletes all N orphan nodes
- On the second call: the `MATCH` finds zero nodes matching the predicate (all orphans were deleted), the loop body executes zero iterations, `total_neo4j_deleted = 0`
- No errors, no state corruption, correct report: `neo4j_deleted_count: 0`

DETACH DELETE on an empty match is a no-op in Cypher. This is the standard idempotency property of predicate-based deletion.

### Qdrant Idempotency

The `DatetimeRange(gt=backup_dt.isoformat())` filter:

- On the first call: the scroll finds N matching points and deletes them
- On the second call: the scroll finds zero matching points (all were deleted in the first call), the loop exits immediately, `total_qdrant_deleted = 0`
- No errors, no state corruption, correct report: `qdrant_deleted_count: 0`

Attempting to delete a non-existent point ID in Qdrant returns a successful response without error — Qdrant treats delete-of-nonexistent as a no-op. For the filter-based delete pattern, there is nothing to match, so nothing is attempted.

### Idempotency Conclusion

**The endpoint is fully idempotent.** A second call with the same `backup_timestamp` and `pg_max_epoch` returns `{neo4j_deleted_count: 0, qdrant_deleted_count: 0, duration_ms: ~50}` without error. This is the correct and expected behavior.

**Practical implication**: The operator can safely run the cleanup endpoint multiple times without risk of over-deletion. This is critical for operational confidence — the operator does not need to track "have I already run this?" before executing the cleanup.

**One caveat**: Idempotency applies only to repeated calls with identical parameters. A second call with a *different* `backup_timestamp` (e.g., an earlier timestamp) would match and delete more records than the first call. This is not a bug — it is correct behavior (the operator is specifying a different cleanup boundary) — but it must be documented to prevent confusion.

## Part 6: Atomicity and Rollback Model

### The Partial-Failure Scenario

The two cleanup operations are independent: Neo4j cleanup (Cypher DETACH DELETE) and Qdrant cleanup (scroll+delete). They run sequentially in the implementation. The failure scenario is:

```
Step 4: Neo4j DETACH DELETE → SUCCESS (1,587 nodes deleted)
Step 5: Qdrant delete → FAILURE (network timeout, Qdrant OOM, etc.)
```

The system is now in an asymmetric state: Neo4j has been cleaned (fewer nodes), but Qdrant retains its orphaned points. Since Qdrant is the canonical store, this state is *worse* than the pre-cleanup state in one important way: Neo4j no longer has graph nodes for those memories, but Qdrant still has vectors for them. A search query will now return the orphaned Qdrant points, reconstruct Memory objects via `_payload_to_memory()`, but have no corresponding Neo4j nodes for causal graph traversal.

### Why Rollback Is Not the Correct Model

**Rolling back Neo4j (un-deleting the 1,587 nodes) is not feasible** for three reasons:

1. Neo4j's `DETACH DELETE` is permanent and irreversible in Community Edition. There is no `UNDELETE` or undo log exposed to the application layer.

2. Even if rollback were possible, the goal is to clean Neo4j (correct behavior). Undoing the cleanup returns the system to a state with both Neo4j and Qdrant orphans — the pre-cleanup state.

3. The correct recovery from the partial-failure scenario is to **retry the Qdrant step**, not to undo the Neo4j step.

### The Correct Atomicity Model: Retry-to-Completion, Not Rollback

The endpoint is designed for retry-to-completion semantics:

1. **Neo4j is cleaned first**: If this fails, nothing has been mutated and the operator can retry the full endpoint.

2. **Qdrant is cleaned second**: If this fails, Neo4j is already clean. The operator can retry the endpoint — on retry, the Neo4j step produces `neo4j_deleted_count: 0` (idempotent) and the Qdrant step retries from scratch (idempotent — the scroll finds the same orphaned points, which still exist).

3. **The endpoint response includes a completion status**: `"status": "complete"` when both steps succeed, `"status": "partial_failure"` with `"failed_step": "qdrant"` when Neo4j succeeds but Qdrant fails. The partial-failure response includes the neo4j_deleted_count (already committed) and an explicit instruction: `"Qdrant cleanup did not complete. Re-run the endpoint — Neo4j cleanup is idempotent and will skip already-deleted nodes."

### The Asymmetric Intermediate State

During the window between Neo4j cleanup (Step 4) and Qdrant cleanup (Step 5), the system is in an asymmetric state:

- Qdrant: has orphaned points (vectors and payload present)
- Neo4j: no orphan nodes (already cleaned)

Searches against Qdrant will return orphaned points, but graph traversal from those points will find no Neo4j nodes (the nodes were deleted in Step 4). The `_payload_to_memory()` function still reconstructs the Memory object from Qdrant payload alone — so the orphaned content is still injected into Claude's context, but without causal graph context (no CO_RETRIEVED edges, since the node was deleted).

This intermediate state is slightly better than the pre-cleanup state (causal graph contamination is removed) but still produces Qdrant-sourced content injection. It is a transient state during the ~5-second Qdrant cleanup execution and should not persist beyond a retry.

**Implementation guidance**: Run Step 4 (Neo4j) and Step 5 (Qdrant) within a single request handler with a single unified try/except. Log the Neo4j count before starting Qdrant. If Qdrant fails, return the partial-failure response immediately — do not attempt to roll back Neo4j.

```python
@router.post("/reconcile")
async def reconcile(
    mode: str = Query(...),
    backup_timestamp: str = Query(...),
    pg_max_epoch: int = Query(...),
    repair: bool = Query(default=False),
    batch_size: int = Query(default=100),
    neo4j_session: Neo4jSession = Depends(get_neo4j_session),
    qdrant: QdrantStore = Depends(get_qdrant),
):
    if mode != "post_restore":
        # Route to existing reconcile logic
        return await _standard_reconcile(repair=repair)

    start_ms = time.monotonic() * 1000

    # --- Step 4: Neo4j cleanup ---
    neo4j_deleted = 0
    if repair:
        neo4j_deleted = await _neo4j_detach_delete_orphans(
            neo4j_session, pg_max_epoch, batch_size=500
        )
    else:
        neo4j_deleted = await _neo4j_count_orphans(neo4j_session, pg_max_epoch)

    # --- Step 5: Qdrant cleanup ---
    qdrant_deleted = 0
    qdrant_status = "success"
    try:
        backup_dt = _parse_backup_timestamp(backup_timestamp)
        if repair:
            qdrant_deleted = await _qdrant_delete_after_timestamp(
                qdrant, backup_dt, batch_size=batch_size
            )
        else:
            qdrant_deleted = await _qdrant_count_after_timestamp(qdrant, backup_dt)
    except Exception as exc:
        qdrant_status = f"failed: {exc}"

    duration_ms = round(time.monotonic() * 1000 - start_ms)

    overall_status = "complete" if qdrant_status == "success" else "partial_failure"

    return {
        "mode": "post_restore",
        "repair": repair,
        "dry_run": not repair,
        "backup_timestamp": backup_timestamp,
        "pg_max_epoch": pg_max_epoch,
        "neo4j_deleted_count": neo4j_deleted,
        "qdrant_deleted_count": qdrant_deleted,
        "duration_ms": duration_ms,
        "status": overall_status,
        "qdrant_step": qdrant_status,
        "warnings": _generate_warnings(neo4j_deleted, qdrant_deleted, qdrant_status),
    }
```

## Part 7: Integration with the Three-Store Cleanup Sequence

The complete post-restore recovery sequence with this endpoint:

```
STEP 1: Detect (S4 fires: epoch_gap > 50; S4+S5 co-alert emits PG_RESTORE_CONFIRMED_BY_GRAPH_STATE)
STEP 2: Disable writes (set maintenance mode flag, or just coordinate manually)
STEP 3: Identify backup_timestamp T and pg_max_epoch E:
         - T: stat the pg_dump file mtime, or read CasaOS snapshot metadata
         - E: SELECT MAX(last_access_epoch) FROM memories;
STEP 3: Retract Redis epoch: redis-cli SET recall:global_epoch <E+1>
STEP 4+5 (new unified step): POST /reconcile?mode=post_restore&backup_timestamp=T&pg_max_epoch=E
         → dry-run first (repair=false) to verify counts
         → then repair=true to execute
STEP 6: Verify: GET /health/detail → s5_graph_compliance should show status: healthy
STEP 7: Wait ~60s for Qdrant HNSW async reoptimization to complete
STEP 8: Re-enable writes
```

The endpoint consolidates what would otherwise be two separate operations (Neo4j Cypher, Qdrant Python) into a single authenticated API call. This reduces operator error from the current three-separate-manual-steps approach and makes the cleanup auditable (the endpoint logs the counts and duration).

### Idea: reconcile-post-restore-mode-unified-endpoint
- **Novelty**: 0.70 — A `/reconcile` endpoint that flips its source-of-truth assignment based on a `mode` parameter — defaulting to "heal toward Qdrant" (correct for transient write failures) but switching to "delete from both stores by temporal predicate" (correct for post-restore) when `mode=post_restore` is specified — is not documented in any reviewed AI memory system. The asymmetry between the two modes (one repairs store divergence, one executes a temporal rollback of recent writes) within the same endpoint design makes the mode parameter load-bearing for correctness, not just for routing. The implementation pattern (backup_timestamp as Qdrant filter, pg_max_epoch as Neo4j filter, dry_run guard, idempotent re-runnable semantics) is a novel operational pattern for multi-store AI memory system recovery.
- **Evidence**: 0.72 — The Qdrant `DatetimeRange(gt=...)` filter on the `created_at` field is documented API behavior verified in Q086 against the existing `created_at` datetime index in `qdrant.py`. The Neo4j `MATCH WHERE last_access_epoch > N DETACH DELETE` batch-loop pattern is the standard Neo4j large-node-deletion pattern and is structurally identical to the pattern already discussed in Q082 follow-ups. The idempotency analysis (zero-match predicate → zero-deletion → no error) follows from the semantics of both databases. The partial-failure → retry semantics are established distributed systems practice for non-transactional cleanup operations. The `repair=false` dry-run guard is already present in the existing `/reconcile` endpoint.
- **Feasibility**: 0.85 — Implementation requires approximately 120 lines of Python in `src/api/routes/ops.py`: one new query parameter (`mode`), one new required parameter (`pg_max_epoch`), one refactored required parameter (`backup_timestamp`, which Q086 already proposed), two new private functions (`_neo4j_detach_delete_orphans`, `_qdrant_delete_after_timestamp`), and an updated response schema. The existing route handler structure and guard pattern (`repair=True` check) are reused. The Qdrant `DatetimeRange` filter follows the identical pattern to the existing `delete_user_data()` method (verified in Q086 at line 1215). The Neo4j batch-loop delete follows standard Neo4j patterns. No new infrastructure, no schema changes, no new dependencies (assuming the Neo4j async driver and Qdrant client are already dependencies).
- **Description**: Extend the existing `/reconcile` endpoint with a `mode=post_restore` branch that accepts `backup_timestamp` (ISO-8601) and `pg_max_epoch` (integer). In this mode, the endpoint abandons the existing Qdrant-vs-Neo4j UUID diff approach and instead executes two independent temporal predicates: (1) `MATCH (n:Memory) WHERE n.last_access_epoch > pg_max_epoch DETACH DELETE n` in Neo4j (batch loop, 500/iteration); (2) scroll + delete all Qdrant points with `created_at > backup_timestamp` using the existing datetime index. A `repair=false` dry-run mode counts without deleting. The response reports `{neo4j_deleted_count, qdrant_deleted_count, duration_ms, status}`. The endpoint is fully idempotent: a second call with the same parameters finds zero matching records in both stores and returns zero counts without error. Partial failure (Neo4j succeeds, Qdrant fails) returns `status: "partial_failure"` with explicit retry instruction — rollback of the Neo4j step is not performed because the cleanup was correct and Qdrant cleanup can be retried independently.

### Idea: retry-to-completion-atomicity-for-non-transactional-cleanup
- **Novelty**: 0.67 — The explicit design choice to use "retry-to-completion" semantics (rather than 2PC or rollback) for multi-store cleanup operations, justified by the idempotency of each individual cleanup predicate, is a specific pattern for non-transactional distributed systems that is not documented in AI memory system literature. The reasoning chain (idempotent predicates → retry-safe → partial failure is recoverable by re-running the full endpoint → no rollback mechanism needed) is a useful design principle that applies to any multi-store cleanup operation where each store supports predicate-based deletion. The specific framing — "the asymmetric intermediate state is tolerable because it is transient and self-correcting via retry" — is the key design insight that distinguishes retry-to-completion from saga-pattern rollback.
- **Evidence**: 0.65 — The idempotency of predicate-based deletion (zero-match → zero-deletion → no error) is a verified property of both Neo4j Cypher (DETACH DELETE on empty MATCH) and Qdrant (delete-by-filter returns success for zero matches, delete-by-ID of non-existent point returns success). The characterization of the intermediate asymmetric state (Neo4j clean, Qdrant still has orphans) as "slightly better than pre-cleanup" (causal graph contamination removed, but Qdrant content injection continues) is correct given Q086's finding that `_payload_to_memory()` does not consult Neo4j for Memory object construction. The retry-to-completion pattern is the standard approach in distributed systems where 2PC is unavailable (different database vendors, no shared transaction coordinator).
- **Feasibility**: 0.92 — No additional implementation required beyond the endpoint itself. The `status: "partial_failure"` response field and the explicit retry instruction in the response body implement the atomicity model at zero additional code cost. The operator guidance ("re-run the endpoint — Neo4j cleanup is idempotent") is the entire recovery procedure. No saga orchestration, no compensating transactions, no undo log.
- **Description**: Document the atomicity model of the post-restore reconcile endpoint explicitly: it uses retry-to-completion semantics rather than 2PC or rollback. If the Qdrant step fails after the Neo4j step succeeds, the response includes `status: "partial_failure"` and the instruction to re-run with the same parameters. On re-run, Neo4j cleanup idempotently skips (zero matching nodes) and Qdrant cleanup retries from scratch (orphaned points still present). This model is correct because each cleanup predicate is idempotent at the database level and the intermediate asymmetric state (Neo4j clean, Qdrant dirty) is tolerable for the duration of a single retry attempt. The model eliminates the need for application-level undo logic in an environment where cross-store transaction rollback is architecturally unavailable.

### Idea: dry-run-count-before-commit-as-restore-verification-gate

## Key Finding

The `/reconcile?mode=post_restore` endpoint is fully specifiable and implementable in approximately 120 lines of Python using existing Qdrant and Neo4j client APIs. The two deletion predicates are orthogonal, temporally grounded, and naturally idempotent. The endpoint is **safe to call twice** with identical parameters: a second call finds zero matching records in both stores and returns `{neo4j_deleted_count: 0, qdrant_deleted_count: 0}` without error. The atomicity model is retry-to-completion rather than rollback: if the Qdrant step fails after the Neo4j step succeeds, the operator re-runs the full endpoint; Neo4j idempotently skips and Qdrant retries from scratch. No cross-store transaction coordinator or compensating transaction is required.

The critical operational insight from this analysis is the **dry-run count comparison**: running with `repair=false` first and verifying that `neo4j_would_delete ≈ qdrant_would_delete` serves as a cross-store sanity check before the destructive cleanup commits. A count discrepancy of >10% indicates a secondary inconsistency (prior incomplete cleanup, or write failures during the backup gap) that should be investigated before the destructive `repair=true` execution.

The endpoint completes the three-store cleanup sequence that Q086 identified as missing Step 5 from all documented recovery procedures. With this endpoint, the full post-restore recovery becomes a single authenticated API call (after the operator has already retracted Redis epoch to `pg_max+1`), reducing operator error and making the cleanup auditable through the structured response body.

No rollback mechanism is needed because each predicate-based deletion leaves zero matching records for a retry, and the partial-failure recovery path (Qdrant step fails after Neo4j succeeds) is resolved by re-running the full endpoint rather than undoing the Neo4j step.
