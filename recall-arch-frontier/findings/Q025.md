## Methodology

**Total findings evaluated**: 43 (Q001–Q043, excluding Q025)
**Total IDEAS in simulate.py evaluated**: 135

**Ranking formula**: adjusted_score = (novelty × 0.4 + evidence × 0.35 + feasibility × 0.25) × stack_multiplier

Stack compatibility multipliers:
- Drop-in (no interface changes): 1.2
- Extension (adds new capability): 1.0
- Replacement (replaces existing component): 0.7
- Infrastructure-new (requires new infrastructure): 0.5

**Feasibility filter**: F ≥ 0.65 (43 ideas pass; remainder are deferred or dropped)

**Architectural leverage factor** applied qualitatively to break ties: ideas that unblock downstream work or fix structural weaknesses in the current stack are promoted. Ideas that are self-contained optimizations with no dependencies are deprioritized relative to ideas that are preconditions for other ideas.

## Idea Clusters

### Cluster 1: Write Integrity (Cross-Machine Deduplication)

What unifies these: casaclaude and proxyclaude write to the same shared backend. Without coordination, the same memory is stored multiple times, or conflicting versions of the same fact coexist. These ideas collectively ensure exactly-one-write semantics across both machines.

Ideas: `recall-content-derived-idempotency-keys` (Q036), `unified-three-gate-store-protocol` (Q041), `scope-qualified-identity-registry` (Q026), `cosine-plus-scope-veto` (Q026), `lww-register-with-hybrid-logical-clock` (Q039), `recall-vector-clock-conflict-detection` (Q038), `two-phase-store-discriminant` (Q026), `session-scoped-staging-queue-partition` (Q035)

Compounding: `recall-content-derived-idempotency-keys` must come first — it is the G3 gate that the unified protocol (Q041) wraps. The scope-qualified identity registry (G1 gate) should follow within days, so that the unified three-gate protocol can be assembled. Vector clock conflict detection (Q038) extends the registry with causal ordering once the registry itself exists.

### Cluster 2: Retrieval Signal Quality

What unifies these: the retrieval path currently uses cosine similarity as a mono-signal. These ideas add a suite of richer, more discriminating signals that collectively close the 29.3× performance gap to physics ceiling (Q011) and the 92% paraphrase contamination problem (Q013).

Ideas: `mmr-diversity-reranker` (Q018), `session-retrieval-set-pin` (Q016), `query-embedding-cache-first-class` (Q033), `embedding-vector-cache` (Q011), `miss-path-parallelization-after-embed` (Q027), `recall-fused-whitened-pca-transform` (Q034), `four-factor-memory-score` (Q018), `implicit-retrieval-feedback` (Q009), `ips-corrected-citation-importance` (Q040)

Compounding: query-embedding-cache (Q033) and miss-path-parallelization (Q027) should come first because they directly cut the dominant 50ms Ollama cost without requiring any schema changes. MMR (Q018) comes next to fix paraphrase contamination. The fused whitened-PCA transform (Q034) requires offline computation of the corpus covariance matrix and must come after at least 1,000 memories exist in the store. IPS-corrected citation importance (Q040) requires at least 200 sessions of logged data before the propensity model is stable enough to trust.

### Cluster 3: Memory Lifecycle / Tiered Eviction

What unifies these: memories currently accumulate without a principled eviction or archival policy. These ideas implement the full lifecycle from hot working memory through warm long-term memory to cold archive, with cluster-aware eviction signals.

Ideas: `working-set-llpc-routing` (Q042), `session-scope-window-drop` (Q024), `lazy-decayed-importance-in-postgres` (Q015), `session-retrieval-set-pin` (Q016), `capacity-threshold-eviction-trigger` (Q016), `arc-ghost-session-clock` (Q028), `two-tier-expiry-slpc-llpc` (Q003), `hot-tier-by-effective-score-not-static-importance` (Q029), `incremental-cohesion-cache-session-boundary` (Q043), `cluster-centroid-manifest-tier3` (Q028)

Compounding: `working-set-llpc-routing` (Q042) and `session-scope-window-drop` (Q024) are foundational — they establish the basic session lifecycle and TTL expiry that all tiering logic depends on. `lazy-decayed-importance-in-postgres` (Q015) must be deployed before `hot-tier-by-effective-score-not-static-importance` (Q029), since the latter's routing logic depends on effective (decayed) importance scores rather than static ones. `arc-ghost-session-clock` (Q028) requires the hot tier to exist before it can implement ghost list promotion between tiers.

### Cluster 4: Store Discriminant / Identity

What unifies these: the current store path does not distinguish between creating a new fact and updating an existing one. These ideas implement scope-qualified identity so that set! (update) semantics coexist with define (create) semantics, preventing the gradual accumulation of redundant near-duplicate memories about the same named facts.

Ideas: `scope-qualified-identity-registry` (Q026), `cosine-plus-scope-veto` (Q026), `two-phase-store-discriminant` (Q026), `causal-supersession-timestamps` (Q026), `content-type-router` (Q026), `recall-semantic-deduplication-at-write` (Q013), `mmr-diversity-reranker` (Q018)

Compounding: cosine-plus-scope-veto is the fastest path (2–3 days) and should ship first. The full two-phase store discriminant with scope registry requires the registry infrastructure. The causal-supersession timestamps depend on the HLC mechanism from Q039.

### Cluster 5: Causal Memory Structure

What unifies these: Neo4j is in the stack but underused for causal memory relationships. These ideas build out the causal graph as a first-class retrieval signal.

Ideas: `typed-adjacency-neo4j` (Q017), `latency-weighted-causal-walk` (Q017), `ppr-retrieval-for-causal-chains` (Q010), `generation-clock-causal-ordering` (Q017), `invalidates-source-provenance-chain` (Q017)

Compounding: `typed-adjacency-neo4j` must be built first — it is the schema that the walk and PPR algorithms require. The causal graph is a Month 3 build because it requires the store discriminant cluster to be stable first (you need stable identity to know which nodes to connect causally).

### Cluster 6: Importance Signal Architecture

What unifies these: Recall's current importance scoring is a single float that conflates multiple distinct signals. These ideas separate and improve the individual components.

Ideas: `dual-importance-scope-architecture` (Q021), `idf-as-routing-signal-not-ranking-signal` (Q031), `hot-tier-context-window-budget` (Q031), `two-axis-importance-discrimination-model` (Q031), `session-topic-activation-boost` (Q021), `epsilon-greedy-memory-exploration` (Q040), `co-retrieval-graph-edge-importance` (Q015)

Compounding: `idf-as-routing-signal-not-ranking-signal` (Q031) requires the hot tier to exist before deployment (per the explicit Q031 sequencing warning). `hot-tier-context-window-budget` builds on both the hot tier and the IDF signal. `dual-importance-scope-architecture` is a prerequisite for the session-topic-activation-boost.

### Cluster 7: Embedding Quality

What unifies these: the raw embedding space from qwen3-embedding:0.6b is not optimally structured for Recall's specific memory distribution. These ideas transform or augment the embedding space to improve retrieval precision.

Ideas: `recall-fused-whitened-pca-transform` (Q034), `recall-threshold-recalibration-protocol` (Q034), `whitened-embedding-store` (Q014), `bilinear-query-chunk-projection` (Q014), `co-occurrence-graph-reranker` (Q014)

Compounding: the fused whitened-PCA transform (Q034) supersedes the standalone whitening (Q008) and standalone PCA ideas. The threshold recalibration protocol (Q034) must accompany the transform deployment or cosine thresholds will reject valid memories (threshold shifts from ~0.92 to ~0.82–0.88 after whitening). The bilinear projection and co-occurrence reranker from Q014 are Month 3 extensions of this cluster.

### Cluster 8: Session Architecture

What unifies these: the retrieve path currently delivers one flat pool of memories. These ideas implement two-tier session architecture (LLPC always-inject + MBC on-demand), reducing cold-start latency and ensuring durable context is always present.

Ideas: `working-set-llpc-routing` (Q042), `two-tier-session-plus-longterm-memory` (Q020), `session-state-cold-start-bridge` (Q020), `session-scope-window-drop` (Q024), `hot-tier-context-window-budget` (Q031)

Compounding: `working-set-llpc-routing` is the foundational LLPC router and should ship in Month 1. The full `two-tier-session-plus-longterm-memory` architecture extends it with a proper warm-tier MBC pool. Session-state cold-start bridge is a Month 2 enhancement.

## Architectural Leverage Analysis

### Highest-Leverage Ideas (unblock the most downstream work)

**`recall-content-derived-idempotency-keys` (Q036)**
This is the single highest-leverage idea in the entire research corpus. It fixes the casaclaude/proxyclaude duplicate write problem immediately (3–5 days), and its SHA-256 idempotency key schema becomes the G3 gate in the unified three-gate protocol (Q041). Without this, the unified protocol cannot be assembled. It also resolves the race condition identified between Q007's session consolidation and Q001-F1's compaction pass (the two-level dedup pipeline in Q035 depends on it). Every subsequent write-path improvement is cleaner once this is in place.

**`scope-qualified-identity-registry` (Q026)**
Second highest leverage. Establishes the binding infrastructure (`topic_key@machine_id → UUID`) that the unified three-gate protocol uses as G1, that the HLC-LWW registry (Q039) stamps, and that vector clock conflict detection (Q038) requires to identify which writes target the same binding. The cosine-plus-scope-veto (2–3 days) is the fast path to get partial benefit while the full registry is built.

**`working-set-llpc-routing` (Q042)**
Establishes the LLPC/MBC session architecture that Q031's IDF routing and hot-tier context-window budget depend on. The 5-line SQL query is deployable in days. Once the LLPC tier exists, it unlocks the IDF signal (Q031 explicitly requires the hot tier to exist first), the context window budget allocation, and the two-tier session plus long-term memory architecture (Q020). The access_count and sessions_since_last_access columns it requires also enable Q016's session-retrieval-set-pin and Q029's hot-tier-by-effective-score routing.

**`lazy-decayed-importance-in-postgres` (Q015)**
Implements exponential importance decay as a SQL expression (importance × exp(−λ × age_days)) without background jobs. Once deployed, Q029's hot-tier routing by effective (decayed) score becomes correct, Q031's IDF signal becomes better calibrated (decayed importance prevents stale-but-once-popular memories from dominating the hot tier), and Q016's retrieval-weighted eviction score uses effective importance as its input.

**`session-scope-window-drop` (Q024)**
The primary garbage collection mechanism for the system. TTL tags written at store time, single filter-delete at session end. The 2032 retrospective (Q024) identifies this as the winning consolidation strategy by a large margin. Establishes the session boundary event as the natural unit of memory lifecycle management, enabling the session-end consolidation job (Q007) and the three-distinct-session-end-questions framework (Q035).

## Tier 1: Build Now (Week 1–4)

**Objective**: Fix the structural weaknesses that cause incorrect behavior at the write path and retrieval path. These are drops-in or fast extensions with no hard predecessors.

### Week 1

**Day 1–2: `mmr-diversity-reranker` (Q018)**
Source: Q018. F=0.90, N=0.88, E=0.85.
The 92% paraphrase contamination rate (Q013) means the current retrieval results are dominated by near-duplicate memories. MMR is a 1–2 day drop-in into the retrieve path (post-ANN re-ranking step) that reduces this contamination to negligible levels. No schema changes. No new infrastructure. Lambda parameter λ=0.5 is the correct starting point; Tim can tune later. This ships before anything else because it has zero dependencies and immediately improves every retrieval result.

Implementation: in the retrieve endpoint, after Qdrant returns top-K=15 candidates, apply MMR re-ranking with λ=0.5 to return K=5. Python: `scipy.spatial.distance.cosine` for pairwise similarity among candidates.

**Day 2–3: `query-embedding-cache-first-class` (Q033)**
Source: Q033. F=0.90, N=0.60, E=0.75.
The retrieve path calls Ollama at 50ms per query for embedding. A Redis hash keyed on `SHA256(query_text + model_name)` eliminates this call on cache hits. Expected hit rate 30–50% within sessions and across sessions for recurring topics. Zero invalidation logic required — when the model changes, the cache namespace (model_name suffix) changes automatically.

Implementation: Redis hash `recall:qembed:{sha256}` → `json(embedding_vector)`. TTL: 90 days or model lifetime. One Redis GET before the Ollama call; one SET after, if miss.

**Day 3–5: `recall-content-derived-idempotency-keys` (Q036)**
Source: Q036. F=0.90, N=0.90, E=0.88. Highest-leverage idea in the corpus.
Create PostgreSQL table `idempotency_keys(key VARCHAR PRIMARY KEY, locked_at TIMESTAMP, created_at TIMESTAMP, response_body JSONB)`. In the store endpoint: `SHA256(normalize(content))` as key; `SELECT FOR UPDATE` before any write. If found within 72h, return cached response. Eliminates duplicate writes from both casaclaude and proxyclaude for byte-identical content.

**Day 5–7: `session-scope-window-drop` (Q024)**
Source: Q024. F=0.92, N=0.88, E=0.85.
Tag each stored memory with `session_id` at write time. At session end (Stop hook), execute `DELETE FROM memories WHERE session_id = :sid AND importance < 0.3`. This removes low-importance ephemeral memories that were only useful within the session they were created. Eliminates accumulation of session-local noise.

### Week 2

**Day 8–10: `cosine-plus-scope-veto` (Q026)**
Source: Q026. F=0.90, N=0.85, E=0.82.
Before inserting a new memory, compute cosine similarity against the embedding of the existing memory at the same scope key (if one exists). If cosine > 0.90, veto the insert and treat it as an update to the existing memory. This is the fast path to scope-qualified deduplication without the full registry infrastructure.

**Day 10–13: `miss-path-parallelization-after-embed` (Q027)**
Source: Q027. F=0.90, N=0.80, E=0.82.
On an embedding cache miss, the current path is: embed → search Qdrant → return. With parallelization: embed AND simultaneously query Redis for any active session-pinned memories → wait for both → merge results. The Qdrant search is initiated in parallel with embedding completion (using asyncio.gather). Reduces p95 latency on cache misses by ~20ms on the retrieval path.

**Day 13–15: `working-set-llpc-routing` (Q042)**
Source: Q042. F=0.95, N=0.85, E=0.90. Deployable in 1–3 days.
Add two integer columns to memories: `access_count INT DEFAULT 0`, `sessions_since_last_access INT DEFAULT 0`. Index: `(access_count, sessions_since_last_access)`. LLPC routing SQL: `SELECT memory_id FROM memories WHERE access_count >= 3 AND sessions_since_last_access <= 10 ORDER BY (access_count * 10 / GREATEST(sessions_since_creation, 1)) DESC LIMIT 2`. Increment `access_count` on every retrieval. Implement `session-boundary-working-set-update` at session end (bulk UPDATE for session_recency counters). The MBC pool is the remaining K=3 from the existing ANN retrieval.

### Week 3

**Day 15–18: `lazy-decayed-importance-in-postgres` (Q015)**
Source: Q015. F=0.90, N=0.82, E=0.78.
Replace the static importance column with a computed expression: `importance_effective = importance_base × exp(−0.05 × age_days)`. Implement as a PostgreSQL generated column or a view. No background job required. The retrieve and eviction paths use `importance_effective` from this point forward. Threshold: memories with `importance_effective < 0.1` are candidates for eviction.

**Day 18–20: `session-retrieval-set-pin` (Q016)**
Source: Q016. F=0.95, N=0.80, E=0.82.
Track which memory IDs have been retrieved in the current session. Store in Redis: `recall:session:{session_id}:pinned` (a SET). On eviction decisions during the session, exclude pinned memories from the candidate pool. At session end, clear the pin set. Prevents evicting memories that are actively in use.

**Day 20–22: `capacity-threshold-eviction-trigger` (Q016)**
Source: Q016. F=0.95, N=0.80, E=0.78.
Add a count check at session end: if total memories > CAPACITY_THRESHOLD (configurable, default 5,000), trigger the eviction pass. Eviction candidates: memories with the lowest `importance_effective` score that are not pinned and not in the LLPC window. Removes the N lowest-scoring until count ≤ CAPACITY_THRESHOLD. This is the minimal viable eviction gate.

### Week 4

**Day 22–25: `scope-qualified-identity-registry` (Q026)**
Source: Q026. F=0.80, N=0.82, E=0.80.
Redis hash `recall:scope_registry` with compound value `{uuid}:{hlc_ms}`. Keys: `{topic_key}@{machine_id}`. On store, check registry before writing. On hit: update existing memory (set! semantics). On miss: insert new memory (define semantics) and register the binding. This is the G1 gate of the unified protocol.

**Day 25–28: `session-scoped-staging-queue-partition` (Q035)**
Source: Q035. F=0.92, N=0.88, E=0.85.
Separate the write pipeline into two phases: L0 (within-session dedup via Redis set of content hashes for the current session) and L1 (cross-session dedup via the G3 idempotency key table). L0 catches burst duplicates from hooks firing rapidly on the same content within a session (sub-second window). L1 catches cross-session and cross-machine duplicates. The session-end hook clears the L0 set.

## Tier 2: Build Next (Week 5–8)

**Objective**: Extend the clean foundation from Tier 1 with the more complex mechanisms that require it to be stable first.

**Week 5: `embedding-vector-cache` (Q011 / Q027)**
Full embedding vector cache (distinct from the query embedding cache built in Week 1 — this is the store-side cache for memories that have already been embedded). Stored embeddings should be cached in Redis so re-retrieval of the same memory does not require a Qdrant round-trip. Key: `recall:embed:{memory_uuid}`. Value: serialized float32 vector.

**Week 5: `unified-three-gate-store-protocol` (Q041)**
Assemble the three gates built in Weeks 1–4 into a unified pre-check pipeline. G3 (idempotency keys from Q036) + G1 (scope registry from Q026) run concurrently via `asyncio.gather()`. G2 (HLC timestamp finalization, local operation) finalizes after G1 returns. Ollama embedding starts only after all three gates pass. Total pre-check overhead: ~2ms. Adds the Redis SETNX fast path as the G3 first-level gate (0.5ms) before the PostgreSQL SELECT FOR UPDATE (2ms) for the common duplicate-hit case.

**Week 6: `implicit-retrieval-feedback` (Q009)**
Implement citation detection in the session summary hook: after each session, compare the session response text against injected memory content using fuzzy string matching (SequenceMatcher, threshold=0.6). Increment `access_count` (already tracking from Q042's LLPC routing) for each cited memory. Begin logging `retrieval_rank` and `cosine_similarity` per memory per session for the future IPS correction (Q040).

**Week 6: `prediction-error-novelty-gate` (Q002)**
Before calling Qdrant for novelty detection at write time, compute cosine similarity between the new embedding and a cached centroid of recently stored memories (stored in Redis, updated incrementally). If cosine > 0.88 (configurable), flag the write as likely-redundant and route it through the scope veto before proceeding to Qdrant. Reduces Qdrant write load by ~30% for high-redundancy write sessions.

**Week 7: `recall-fused-whitened-pca-transform` (Q034)**
Offline: compute the corpus covariance matrix from all stored embeddings. Derive transform T = Λ₅₀^(−1/2) V₅₀^T (50 principal components, whitened). Store T as a 50×384 float32 matrix in PostgreSQL or a static file. Apply at retrieve time: `q_transformed = T @ q_raw`. The transformed space has unit variance across all dimensions, eliminating the anisotropy in qwen3-embedding:0.6b's output that causes hub memories to dominate retrieval.

Must ship alongside `recall-threshold-recalibration-protocol`: after whitening, run a calibration pass on the last 100 sessions to determine the correct cosine threshold (expected: 0.82–0.88 vs current ~0.92).

**Week 7: `two-tier-expiry-slpc-llpc` (Q003)**
Implement the immunological two-tier expiry: SLPC memories (short-lived plasma cells analog) have TTL = 30 days and decay rapidly after session recency drops. LLPC memories (long-lived analog, promoted from SLPC after 3+ cross-session citations) have TTL = 365 days and decay slowly. The working-set-llpc-routing from Week 2 provides the citation frequency signal for promotion decisions.

**Week 8: `idf-as-routing-signal-not-ranking-signal` (Q031)**
With the hot tier now established (LLPC working set from Week 2, two-tier expiry from Week 7), add IDF weighting as a routing pre-filter. High-IDF memories (rare in the corpus) are routed to the MBC on-demand pool; low-IDF memories (ubiquitous, information-poor) are deprioritized in routing. IDF is NOT used as a ranking signal within the retrieved set — only as a gate for hot-tier membership. Compute IDF offline weekly from the PostgreSQL memory table.

**Week 8: `hot-tier-by-effective-score-not-static-importance` (Q029)**
With `lazy-decayed-importance-in-postgres` from Week 3 now stable and generating effective scores, switch the hot-tier routing criterion from static importance to effective (decayed) importance. Memories with high static importance but low effective importance (stale-but-once-popular) are demoted from the hot tier. The 2Q access frequency gate from Q042 remains in place — a memory must satisfy both `access_count >= 3 AND sessions_since_last_access <= 10 AND importance_effective >= 0.3` to qualify for LLPC.

## Tier 3: Build After (Week 9–12)

**Objective**: Complete the architectural vision with the advanced mechanisms that require the Tier 1 and Tier 2 foundations to be stable and generating data.

**Week 9: `arc-ghost-session-clock` (Q028)**
Three-tier eviction with ghost lists: ARC algorithm for the hot tier, session-clock for the warm tier, cluster-centroid manifest for cold/archive tier. The ghost list (recently evicted items, metadata only) enables re-promotion without re-embedding if a memory is needed again shortly after eviction. Requires the hot tier (Week 2) and two-tier expiry (Week 7) to be stable.

**Week 9: `dual-importance-scope-architecture` (Q021)**
Separate the importance signal into two axes: session-level activation boost (how important is this memory for the current session's topic domain?) and long-term baseline importance (how universally cited is this memory across all sessions?). The session-level axis uses the cosine similarity between the session's working query centroid and the memory's embedding. The long-term axis is the IDF-calibrated citation rate from Q031.

**Week 10: `typed-adjacency-neo4j` (Q017)**
Implement the causal memory graph in Neo4j with typed edge labels: `SUPERSEDES`, `REFERENCES`, `CONTRADICTS`, `DEPENDS_ON`. Each stored memory inspects its predecessor chain (via scope registry) and creates the appropriate Neo4j edge. The generation-clock-causal-ordering mechanism assigns monotonically increasing HLC timestamps to nodes so causally earlier memories can be identified during graph walks.

**Week 10: `lww-register-with-hybrid-logical-clock` (Q039)**
Extend the scope registry (from Week 4) with HLC stamping. Change Redis hash values from `{uuid}` to `{uuid}:{hlc_ms}`. Implement the Redis Lua compare-and-swap script for atomic LWW-with-HLC updates. The hook clients (observe-edit.js) gain a 30-line HLC implementation. This replaces non-deterministic Redis HSET overwrite with deterministic causal last-write-wins.

**Week 10–11: `recall-vector-clock-conflict-detection` (Q038)**
Add two BIGINT columns to PostgreSQL memories table: `vc_casaclaude`, `vc_proxyclaude`. Maintain authoritative vector clock in Redis hash `recall:vector_clock`. On each write: HINCRBY local counter, HGET peer counter, store both in memory record. When the unified three-gate protocol detects two writes targeting the same scope binding (G1 hit from both machines within a window), compare vector clocks to confirm concurrent write, then apply HLC tiebreaker. Archive the losing write for audit.

**Week 11–12: `ppr-retrieval-for-causal-chains` (Q010)**
With the Neo4j causal graph populated (from Week 10), add Personalized PageRank as a retrieval signal. For a given query memory, PPR on the causal graph surfaces memories that are causally connected to high-scoring nodes — even if they are not cosine-similar to the query. Bridges the semantic gap for memories that are causally related but use different vocabulary.

**Week 12: `ips-corrected-citation-importance` (Q040)**
After 200+ sessions of logged citation data (from the implicit retrieval feedback deployed in Week 6), fit the propensity model: `P(cited | rank, cosine_sim)` via logistic regression on historical data. Switch the importance update formula from naive citation boosting to IPS-corrected: `delta = 0.1 × (citation / propensity_estimate) × (1 - importance)`. This eliminates the Matthew effect where rank-1 memories accumulate disproportionate importance purely from positional advantage.

**Week 12: `incremental-cohesion-cache-session-boundary` (Q043)**
Cache `cluster_cohesion FLOAT` per memory in PostgreSQL. At session end, run the cohesion computation (Qdrant HNSW search for 10 nearest neighbors, pairwise cosine, mean) for newly archived memories and their neighbors (~500 vectors, ~800ms). The Tier 3 eviction displacement check reads from PostgreSQL (0.1ms) rather than querying Qdrant on demand (5ms). Decouples the eviction policy from Qdrant availability.

**Month 3 (extended): `epsilon-greedy-memory-exploration` (Q040)**
With the IPS infrastructure in place, activate epsilon-greedy exploration: with probability 0.05, substitute the K-th retrieved memory with a randomly selected medium-importance memory. Log `is_exploration=True`. This generates counterfactual retrieval data for chronically under-exposed memories, closing the feedback loop for memories that are never in the top-K.

## Moats (Ideas Worth Protecting)

### `recall-content-derived-idempotency-keys` (Q036)
- **Why it's a moat**: SHA-256 on normalized content as the idempotency key for a multi-machine AI memory system is not implemented in any reviewed production AI memory system (mem0, Zep, MemGPT, LangMem, AWS AgentCore). The mechanism is simple, but only Recall's architecture (two concurrent writing machines) creates the incentive to build it. Once deployed and generating 6+ months of data on duplicate rates and content normalization edge cases, the system accumulates hard-won knowledge about exactly where normalization must be tuned (Unicode, whitespace, capitalization, paraphrase boundaries). Replicating this would require the same multi-machine setup plus months of production operation.
- **Window**: 12–18 months before a cloud-hosted competitor with multi-device support discovers the same pattern (they will, eventually).
- **Build sequence**: Ship first, week 1.

### `working-set-llpc-routing` (Q042)
- **Why it's a moat**: Applying Denning's working set model plus 2Q admission policy to AI memory session routing — in pure SQL over integer columns, without importance scores or embeddings — is not described in any reviewed system. The 5-line SQL query is deceptively simple but encodes a deep insight: memories that are inside the working set window AND have crossed the frequency promotion threshold represent the durable context for Tim's current project phase. This cannot be purchased or copied by a cloud provider without also knowing Tim's session patterns. The threshold calibration (τ=10 sessions, access_count ≥ 3) will need tuning from real data, and the tuned values are themselves moat-building.
- **Window**: The mechanism is derivable from first principles; a sophisticated competitor could build it within 6 months once they encounter the problem. But the calibrated values and the session-specific tuning require Tim's actual data.
- **Build sequence**: Week 2 of Month 1.

### `unified-three-gate-store-protocol` (Q041)
- **Why it's a moat**: The composition of G1 (scope-qualified identity), G2 (HLC causal timestamp), and G3 (content-hash idempotency) into a single concurrent pre-check pipeline before every memory write is not found in any reviewed AI memory system. Each gate independently addresses a correctness failure mode that is invisible until you have two concurrent writers. Building all three requires the multi-machine architecture that creates the necessity for them. The protocol design itself is publishable; the implementation on Recall's specific stack with Recall's specific normalization choices is the moat.
- **Window**: 18–24 months. This requires the multi-machine write problem to be salient, and most AI memory systems are either single-user or cloud-hosted with database-level concurrency control.
- **Build sequence**: Weeks 4–6, after G1 and G3 gates are individually deployed and stable.

### `lww-register-with-hybrid-logical-clock` (Q039)
- **Why it's a moat**: No production AI memory system (verified across mem0, Zep, MemGPT, LangMem, AWS AgentCore, Cognee) uses vector clocks or HLC to detect or resolve concurrent writes across multiple writer instances. Distributed systems theory has had this answer since 1988 (Fidge/Mattern vector clocks) and 2014 (Kulkarni HLC). The engineering transfer to AI memory has not occurred. Once deployed and battle-tested with casaclaude/proxyclaude, the implementation is a genuine technical reference that other multi-machine memory systems would need to replicate from scratch.
- **Window**: 2–3 years. This requires the same motivation (multi-machine concurrent writes) plus familiarity with distributed systems literature that most AI memory developers lack.
- **Build sequence**: Week 10, after the scope-qualified identity registry is stable (Week 4).

### `ips-corrected-citation-importance` (Q040)
- **Why it's a moat**: IPS (Inverse Propensity Scoring) for debiasing implicit retrieval feedback has not been applied to AI memory importance scoring in any reviewed system. The mechanism requires a minimum dataset of ~200 sessions to fit the propensity model — which means early adopters who collect this data first have a 12–18 month head start before the propensity model is reliable. The model parameters are inherently personalized (Tim's query distribution, his session topics, his citation patterns) and cannot be transferred to another user without retraining. The moat compounds over time: as more sessions accumulate, the propensity model improves, which improves importance scores, which improves retrieval, which generates better citation signals.
- **Window**: Permanent — the personalized propensity model is a data moat that cannot be replicated without Tim's specific session history.
- **Build sequence**: Week 12, after 200+ sessions of logged data from implicit-retrieval-feedback (deployed Week 6).

## Compounding Sequences

### Sequence 1: Write Integrity Chain
`recall-content-derived-idempotency-keys` → `scope-qualified-identity-registry` → `unified-three-gate-store-protocol` → `lww-register-with-hybrid-logical-clock` → `recall-vector-clock-conflict-detection`

Step 1 (Q036): Each write gets a SHA-256 content fingerprint. Byte-identical cross-machine duplicates suppressed in 3–5 days.

Step 2 (Q026): Each write checks scope registry. Named-fact updates route to set! semantics; new facts route to define semantics. Identity-level deduplication achieved.

Step 3 (Q041): The three gates run concurrently pre-embedding. The store path is now formally correct: no duplicate inserts, no stale bindings, causal timestamps on every write. Total pre-check overhead: ~2ms vs. 50ms Ollama dominance.

Step 4 (Q039): The scope registry gets HLC-stamped LWW semantics. Concurrent writes from both machines converge deterministically to the causally later value.

Step 5 (Q038): Vector clock columns detect concurrent writes with mathematical precision (64 bits, 0.20% overhead). Conflicts are logged and resolved by HLC tiebreaker.

Each step makes the next one more valuable. Step 1 alone eliminates the most common failure mode. Step 5 handles the edge cases that remain.

### Sequence 2: Hot Tier Activation Chain
`working-set-llpc-routing` → `lazy-decayed-importance-in-postgres` → `hot-tier-by-effective-score` → `idf-as-routing-signal` → `hot-tier-context-window-budget`

Step 1 (Q042): The LLPC tier exists. K=2 always-inject memories are selected by frequency×recency. The access_count and sessions_since_last_access columns are now maintained.

Step 2 (Q015): Importance decay is live. Every memory has an effective importance score that decreases with age. The LLPC routing adds `AND importance_effective >= 0.3` to its WHERE clause.

Step 3 (Q029): The hot tier routing switches from static importance to effective importance. Stale-but-once-popular memories fall out of the hot tier automatically. The LLPC tier now reflects current relevance, not historical relevance.

Step 4 (Q031): IDF signal added as a routing pre-filter. Low-IDF memories (information-poor, common phrases) are demoted from hot tier. The Q031 explicit prerequisite (hot tier must exist first) is now satisfied.

Step 5 (Q031): Context window budget allocation governs how many tokens the hot tier consumes. The budget prevents LLPC expansion from consuming the context window that MBC on-demand retrieval needs.

This chain produces a fully calibrated, automatically-maintained hot tier by the end of Month 2. No manual tuning required after initial threshold setting.

### Sequence 3: Retrieval Quality Chain
`mmr-diversity-reranker` → `query-embedding-cache-first-class` → `recall-fused-whitened-pca-transform` + `recall-threshold-recalibration-protocol` → `implicit-retrieval-feedback` → `ips-corrected-citation-importance`

Step 1 (Q018, Day 1–2): 92% paraphrase contamination drops to negligible. Retrieval results are now diverse.

Step 2 (Q033, Day 2–3): The 50ms Ollama call on cache hits drops to ~0.5ms Redis GET. p95 retrieval latency decreases significantly for recurring queries.

Step 3 (Q034, Week 7): The embedding space is whitened and projected. Cosine similarity is now calibrated. The MMR diversity filter from Step 1 is more precise in the whitened space. Thresholds are recalibrated.

Step 4 (Q009, Week 6): Each session's citation data is logged. The feedback loop begins accumulating data. `access_count` updates (already active from Q042) now get citation-boosted updates.

Step 5 (Q040, Week 12+): After 200 sessions, the IPS propensity model is fitted. The Matthew effect is suppressed. Importance scores reflect genuine value, not retrieval rank position.

This chain takes retrieval from a raw cosine search with paraphrase contamination to a calibrated, debiased, diversity-aware system.

### Sequence 4: Memory Lifecycle Chain
`session-scope-window-drop` → `capacity-threshold-eviction-trigger` → `session-retrieval-set-pin` → `two-tier-expiry-slpc-llpc` → `arc-ghost-session-clock`

Step 2 (Q016, Week 3): When corpus exceeds CAPACITY_THRESHOLD, eviction fires automatically. The eviction candidate set excludes pinned memories.

Step 3 (Q016, Week 3): Pinning ensures in-session memories cannot be evicted mid-session. The eviction pass only touches truly cold memories.

Step 4 (Q003, Week 7): Two-tier expiry with SLPC/LLPC promotion. The expiry TTLs are now differentiated: session-scoped memories have a short TTL, LLPC-promoted memories have a long TTL.

Step 5 (Q028, Week 9): ARC ghost lists enable re-promotion of recently evicted memories. Three-tier architecture (hot/warm/cold) with cluster-centroid manifests for the cold tier.

## Ideas Extracted

The convergence analysis finds the following cross-cutting themes that no single finding could reveal:

### Theme 1: The 50ms Ollama Cost Is the True Bottleneck

Q011's physics ceiling analysis identified the 29.3× gap to physics ceiling. Q027's negative cache analysis confirmed the embedding call dominates at 50ms. Q033's query embedding cache shows a 30–50% hit rate is achievable. Q041's unified protocol explicitly designs the pre-check overhead (2ms) to be negligible against the 50ms embedding. Q023's 2032 retrospective predicts this bottleneck dissolves when embedding cost collapses — but for the next 3–5 years, every retrieval improvement must either reduce embedding frequency or hide it behind parallelism.

**Implication for build order**: the query-embedding-cache (Q033) and miss-path-parallelization (Q027) ship in Week 1 precisely because they address the dominant cost without any dependencies.

### Theme 2: Session Boundaries Are the Natural Unit of Memory Management

Q024, Q035, Q007, Q016, Q042, Q020 all converge on the session boundary as the correct event for lifecycle management. Session-end is when: TTL-expired memories are deleted (Q024), L0 dedup sets are flushed (Q035), consolidation runs (Q007), pinned memories are released (Q016), working-set counters are updated (Q042), and session-state summaries are stored (Q020). The stop hook is already the correct integration point; it simply needs to be extended systematically.

**Implication**: every lifecycle mechanism should hook into session-end, not into a background daemon. This keeps the system stateless between sessions and avoids the complexity of background process coordination.

### Theme 3: The Write Path Has Three Distinct Failure Modes That Require Three Distinct Fixes

Q036 (byte-identical cross-machine duplicates), Q026 (same-named-fact with updated content treated as new memory), and Q013 (near-duplicate paraphrases from the same session) are three different failure modes that look similar but require different mechanisms. G3 (SHA-256 idempotency) fixes the first. G1 (scope-qualified identity) fixes the second. Cosine-plus-scope-veto (the Phase 2 check from Q026) fixes the third. Trying to solve all three with a single cosine threshold (the current approach) fails all three.

**Implication**: the unified three-gate protocol (Q041) is a convergence result, not an overengineered solution. It addresses three genuinely distinct failure modes in the minimum unified architecture.

### Theme 4: Importance Score Is a Conflation of Three Independent Signals

Q021, Q031, Q015, Q040 all identify that Recall's single `importance FLOAT` conflates: long-term citation rate (how often is this memory cited across all sessions?), session-level activation (how relevant is this memory to the current session's topic domain?), and recency decay (how much should this memory's weight decrease as it ages?). These three signals have different update frequencies (long-term: weekly; session-level: per session; decay: continuous) and different use cases (eviction, hot-tier routing, context-window budgeting). A single float cannot correctly serve all three.

**Implication**: the `dual-importance-scope-architecture` (Q021) and IPS correction (Q040) are not competing approaches — they address different axes of the same conflation problem. The Month 3 architecture decomposes the importance score into its constituent signals.

## What NOT to Build (and Why)

### `multi-space-embedding-architecture` (Q023)
Skip. The 2032 retrospective identifies this as premature: building multi-space embedding requires significant infrastructure and produces marginal benefit when the primary cost is Ollama latency (50ms), not retrieval precision. The 2032 prediction is that embedding cost collapse makes this approach valuable — but at current hardware costs, the complexity is not justified. Build the fused whitened-PCA transform (Q034) instead, which achieves similar dimensionality benefits at a fraction of the complexity.

### `hyde-query-expansion-at-retrieve` (Q023)
Skip. F=0.30. Requires an LLM call (qwen3:14b at 194,586× slower than physics ceiling, per Q012) for every query expansion. The physics ceiling analysis (Q012) shows LLM-in-the-loop retrieval is fundamentally incompatible with the 500ms p95 SLA. The Q023 retrospective confirms this approach becomes viable only in the 2032 era of sub-millisecond embeddings.

### `lsh-novelty-sketch-in-process` (Q012)
Skip. Q023 explicitly identifies this as one of the mechanisms that becomes obsolete at 0.5ms embedding cost. The LSH sketch complexity (tuning false positive/negative rates) is not justified when the correct solution (query-embedding-cache + cosine gate) achieves the same novelty detection at lower complexity.

### `bilinear-query-chunk-projection` (Q014)
Defer to Month 4+. F=0.40. Requires training a bilinear projection matrix — a supervised learning step that requires labeled data (query × memory relevance pairs) that Recall does not yet have. Build the simpler whitening transform (Q034) first. Return to bilinear projection only if the whitened space still shows systematic retrieval failures after several months of operation.

### `continuous-background-reembedding` (Q023)
Skip. F=0.55. Running continuous background re-embedding as the model improves imposes a sustained GPU compute cost (Ollama server at 192.168.50.62) without clear retrieval quality payoff at current memory corpus size. At 10K–100K memories, a manual re-embedding on model upgrade (once every 6–12 months) is preferable to a continuous background job that competes with live retrieval.

### `stcs-batch-semantic-deduplication` (Q001-F1)
Deprioritize. The Q034 fused whitened-PCA transform effectively addresses the redundancy that semantic deduplication targets, but through the embedding space rather than through an offline batch pass. Implementing a separate batch deduplication pass adds operational complexity (scheduling, idempotency) that is not justified when whitening + MMR + scope-veto already handle the problem.

### `recall-cluster-manifest-stub` (Q037)
Defer to Month 6+. F=0.78. The cluster-manifest stub in Redis is an architecturally sound idea (SPANN routing pattern extended with a member-ID list), but it requires a stable corpus and cluster assignment to be meaningful. At Recall's current scale (thousands of memories) and with HNSW already performing well, the benefit does not justify the 2–3 weeks of implementation in the 90-day window. Build this if corpus grows beyond 50K memories and HNSW latency becomes a concern.

### `four-factor-memory-score` (Q018)
Defer the full implementation. The individual factors (cosine similarity, importance, recency, novelty) are all being addressed by individual targeted ideas in this roadmap. The unified four-factor scorer requires all four inputs to be calibrated before combining them, and the combination introduces a new hyperparameter (factor weights) that needs tuning. Build the individual factors first; combine them in Month 4+ once each is calibrated.

## Synthesis Statement

The 43 findings of this research loop reveal that System-Recall is not one system facing one problem, but three overlapping systems each facing a distinct class of failure: a write path system with no cross-machine coordination (producing silent duplicates and stale bindings), a retrieval path system with no diversity or calibration (producing paraphrase-contaminated results dominated by a cosine monoculture), and a memory lifecycle system with no principled eviction or importance decay (producing a corpus that accumulates noise indefinitely). The apparent complexity of the 135-idea IDEAS dict dissolves when these three failure classes are identified, because most ideas cluster into solutions for one of these three problems. The compounding sequences discovered here — write integrity chain, hot tier activation chain, retrieval quality chain, memory lifecycle chain — show that the ideas do not compose in an arbitrary order: there is one correct sequence where each mechanism's value is amplified by the mechanisms that preceded it. The most important insight that no single finding could show is this: the 50ms Ollama embedding call is the universal bottleneck that every high-priority idea either directly addresses (embedding cache, parallelization) or designs around (pre-checks adding 2ms vs. 50ms, session-boundary processing adding 800ms to the non-critical path). Build the retrieval speed improvements first, then the write correctness improvements, then the advanced architecture — and in that order, every subsequent improvement ships on a foundation that is both faster and more correct than the one before it.
