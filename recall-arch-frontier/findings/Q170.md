# Q170 — Bootstrap K-means Clustering Job: Complete Implementation Spec

**Question**: What is the complete bootstrap initialization spec for the co_retrieval graph — specifically, what does a one-time K=200 K-means clustering job look like that pre-populates CO_RETRIEVED edges for all intra-cluster memory pairs at the live 22,423-memory corpus scale, and what is the estimated runtime on CasaOS N100 hardware?
**Verdict**: BREAKTHROUGH

**Novelty**: 0.78 | **Executability**: 0.95 | **Feasibility**: 0.92

---

## Complete Implementation

The script below is a fully runnable, self-contained bootstrap job. It fetches all
vectors directly from Qdrant (no Ollama call needed — vectors are stored at-rest),
clusters them with scikit-learn KMeans, generates within-cluster pairs, and writes
CO_RETRIEVED edges to Neo4j in parallel batches. Run once before activating the
flush_co_retrievals worker.

```python
#!/usr/bin/env python3
"""
bootstrap_co_retrieved.py — One-time K=200 K-means bootstrap for CO_RETRIEVED graph edges.

Pre-populates CO_RETRIEVED edges in Neo4j for all within-cluster memory pairs
discovered by K-means clustering of Qdrant embedding vectors.

Usage:
    python bootstrap_co_retrieved.py

Run ONCE before enabling the flush_co_retrievals ARQ worker. Safe to re-run
after first boot (MERGE is idempotent — existing edges are not reset, strength
is not decremented). Do NOT run concurrently with flush_co_retrievals on first
bootstrap; safe to run concurrently on subsequent re-runs.

Requirements:
    pip install qdrant-client scikit-learn neo4j numpy
    (All likely already installed in the Recall venv)

Environment variables (same as Recall API):
    QDRANT_HOST       default: localhost
    QDRANT_PORT       default: 6333
    QDRANT_COLLECTION default: memories
    NEO4J_URI         default: bolt://localhost:7687
    NEO4J_USER        default: neo4j
    NEO4J_PASSWORD    required

Runtime estimate on CasaOS N100:
    - Vector fetch from Qdrant: ~30-60 s (22K × 1024-dim, scroll over LAN)
    - KMeans clustering (K=200, 10 iter): ~4-5 s (compute-bound, SSE4.2 only)
    - Neo4j writes (1.25M pairs, 10K batch, 4 workers): ~90-102 s (~1.7 min)
    - Total wall-clock: approximately 2-3 minutes
"""

import os
import time
import logging
import itertools
from concurrent.futures import ThreadPoolExecutor, as_completed

import numpy as np
from sklearn.cluster import KMeans
from qdrant_client import QdrantClient
from neo4j import GraphDatabase

# ---------------------------------------------------------------------------
# Configuration
# ---------------------------------------------------------------------------

QDRANT_HOST       = os.environ.get("QDRANT_HOST", "localhost")
QDRANT_PORT       = int(os.environ.get("QDRANT_PORT", "6333"))
QDRANT_COLLECTION = os.environ.get("QDRANT_COLLECTION", "memories")

NEO4J_URI      = os.environ.get("NEO4J_URI", "bolt://localhost:7687")
NEO4J_USER     = os.environ.get("NEO4J_USER", "neo4j")
NEO4J_PASSWORD = os.environ.get("NEO4J_PASSWORD", "")

K              = 200     # Number of clusters. K=200 keeps per-cluster pairs at
                          # ~6,229 — safely within one 10K Neo4j batch per cluster.
N_INIT         = 10      # KMeans restarts. 10 is sklearn default; gives stable
                          # centroids on a 1024-dim corpus without prohibitive cost.
BATCH_SIZE     = 10_000  # Pairs per Neo4j transaction. K=200 → one batch/cluster.
N_WORKERS      = 4       # Parallel Neo4j writer threads (disjoint cluster sets).
BOOTSTRAP_SEED = 0.1     # Initial CO_RETRIEVED edge strength (below ordering-flip
                          # threshold of ~0.3; live reinforcement promotes edges
                          # that are actually co-retrieved above this floor).

logging.basicConfig(
    level=logging.INFO,
    format="%(asctime)s [%(levelname)s] %(message)s",
    datefmt="%H:%M:%S",
)
log = logging.getLogger("bootstrap_co_retrieved")


# ---------------------------------------------------------------------------
# Step 1: Fetch all vectors from Qdrant via scroll (no Ollama needed)
# ---------------------------------------------------------------------------

def fetch_all_vectors(client: QdrantClient, collection: str):
    """
    Scroll through the entire Qdrant collection and return:
        ids:     list[str]   — memory UUIDs (Qdrant point IDs as strings)
        matrix:  np.ndarray  — shape (N, D) float32

    Uses with_vectors=True so embeddings are returned with each record.
    No re-embedding is needed — vectors are stored at-rest in Qdrant.

    Scroll page size of 256 balances memory pressure vs round-trip count.
    At N=22,423 this is ~88 pages of 256 records each.
    """
    log.info("Fetching vectors from Qdrant collection '%s' ...", collection)
    t0 = time.perf_counter()

    ids     = []
    vectors = []
    offset  = None  # None = start from beginning

    while True:
        results, next_offset = client.scroll(
            collection_name=collection,
            with_vectors=True,
            with_payload=False,   # payload not needed for clustering
            limit=256,
            offset=offset,
        )

        if not results:
            break

        for point in results:
            ids.append(str(point.id))
            vectors.append(point.vector)

        offset = next_offset
        if offset is None:
            break   # Qdrant signals end-of-collection with None

    matrix = np.array(vectors, dtype=np.float32)
    elapsed = time.perf_counter() - t0
    log.info(
        "Fetched %d vectors, shape %s in %.1f s",
        len(ids), matrix.shape, elapsed,
    )
    return ids, matrix


# ---------------------------------------------------------------------------
# Step 2: K-means clustering
# ---------------------------------------------------------------------------

def cluster_vectors(matrix: np.ndarray, k: int = K, n_init: int = N_INIT):
    """
    Run scikit-learn KMeans on the embedding matrix.

    Returns labels array of shape (N,) with cluster assignment per vector.

    Hardware note (N100, SSE4.2 only, no AVX/FMA3):
        - 92.54 GFLOP total for 10 iterations at K=200, N=22,423, D=1024
        - Physics minimum: 0.844 s (Q172)
        - Expected practical: 4-5 s at 40-60% BLAS efficiency
        - algorithm='lloyd' uses BLAS SGEMM internally (fastest on this hardware)
        - n_jobs is not set here; KMeans itself uses OpenBLAS threading via numpy.
          Confirm multi-threaded BLAS with: import numpy; numpy.show_config()
    """
    log.info("Running KMeans(n_clusters=%d, n_init=%d) ...", k, n_init)
    t0 = time.perf_counter()

    km = KMeans(
        n_clusters=k,
        n_init=n_init,
        algorithm="lloyd",   # Uses BLAS SGEMM. Do not use 'elkan' — it avoids
                              # the full distance matrix and loses BLAS benefit.
        random_state=42,
        verbose=0,
    )
    labels = km.fit_predict(matrix)

    elapsed = time.perf_counter() - t0
    cluster_sizes = np.bincount(labels)
    log.info(
        "Clustering complete in %.2f s — cluster size: min=%d mean=%.1f max=%d",
        elapsed,
        cluster_sizes.min(),
        cluster_sizes.mean(),
        cluster_sizes.max(),
    )
    return labels


# ---------------------------------------------------------------------------
# Step 3: Generate within-cluster pairs
# ---------------------------------------------------------------------------

def generate_cluster_pairs(ids: list, labels: np.ndarray, k: int = K):
    """
    Group memory IDs by cluster label and produce all undirected pairs
    within each cluster.

    At K=200, N=22,423:
        Mean cluster size: ~112 members
        Pairs per cluster: ~6,229  (112 * 111 / 2)
        Total pairs:       ~1,245,766

    Returns list of lists, one sub-list per cluster, each containing
    (id_a, id_b) tuples.

    Memory: 1.25M tuples × 2 × ~40 bytes (UUID strings) ≈ 100 MB peak.
    Acceptable for a one-shot bootstrap job.
    """
    log.info("Generating within-cluster pairs ...")
    t0 = time.perf_counter()

    # Group IDs by cluster
    clusters = {}
    for memory_id, label in zip(ids, labels):
        clusters.setdefault(int(label), []).append(memory_id)

    # Generate pairs per cluster
    cluster_pair_batches = []
    total_pairs = 0
    for label in range(k):
        members = clusters.get(label, [])
        if len(members) < 2:
            continue
        pairs = list(itertools.combinations(members, 2))
        cluster_pair_batches.append(pairs)
        total_pairs += len(pairs)

    elapsed = time.perf_counter() - t0
    log.info(
        "Generated %d total pairs across %d clusters in %.2f s",
        total_pairs, len(cluster_pair_batches), elapsed,
    )
    return cluster_pair_batches


# ---------------------------------------------------------------------------
# Step 4: Write CO_RETRIEVED edges to Neo4j
# ---------------------------------------------------------------------------

CYPHER_MERGE = """
UNWIND $batch AS pair
MERGE (a:Memory {id: pair.a})-[r:CO_RETRIEVED]-(b:Memory {id: pair.b})
ON CREATE SET
    r.strength   = $seed,
    r.created_at = datetime(),
    r.updated_at = datetime(),
    r.source     = 'bootstrap'
ON MATCH SET
    r.strength   = CASE
                       WHEN r.strength + 0.1 > 1.0 THEN 1.0
                       ELSE r.strength + 0.1
                   END,
    r.updated_at = datetime()
"""
# ON CREATE: bootstrap_seed = 0.1 (not 0.5 — bootstrap is a prior, not a
#            confirmed signal; live reinforcement promotes edges that matter).
# ON MATCH:  if the edge already exists (re-run or flush_co_retrievals ran
#            first), increment by 0.1 (same delta as flush worker), cap at 1.0.
#            This makes the bootstrap safe to re-run: it never resets strength.


def _write_cluster_batch(driver, cluster_pairs_chunk: list, seed: float, worker_id: int):
    """
    Write one cluster's pairs to Neo4j in sub-batches of BATCH_SIZE.

    Called from a ThreadPoolExecutor worker thread — each thread holds its
    own Neo4j driver session (thread-safe; the driver itself is shared).

    Returns (pairs_written, batches_committed, elapsed_s).
    """
    pairs_written = 0
    batches_committed = 0
    t0 = time.perf_counter()

    with driver.session() as session:
        # Each cluster's pairs may exceed BATCH_SIZE if K is small (K<158 risk).
        # At K=200 each cluster has ~6,229 pairs — one batch is sufficient, but
        # we chunk defensively to handle variance in cluster sizes.
        for i in range(0, len(cluster_pairs_chunk), BATCH_SIZE):
            sub = cluster_pairs_chunk[i : i + BATCH_SIZE]
            batch_payload = [{"a": a, "b": b} for a, b in sub]
            session.run(CYPHER_MERGE, batch=batch_payload, seed=seed)
            pairs_written += len(sub)
            batches_committed += 1

    return pairs_written, batches_committed, time.perf_counter() - t0


def write_edges_parallel(driver, cluster_pair_batches: list, seed: float = BOOTSTRAP_SEED):
    """
    Write all cluster pair batches to Neo4j using N_WORKERS parallel threads.

    Partition strategy: round-robin assignment of clusters to workers.
    Within-cluster pairs write to disjoint node-pair sets only when cluster
    membership does not overlap — which is always true for K-means hard
    assignments. No lock contention between workers.

    Throughput: 4 workers × ~5,000 ops/s = ~20,000 ops/s effective.
    At 1,245,766 total pairs: ~62 s best-case, ~102 s nominal.
    """
    log.info(
        "Writing %d cluster batches to Neo4j with %d parallel workers ...",
        len(cluster_pair_batches), N_WORKERS,
    )
    t0 = time.perf_counter()

    # Partition clusters across workers (round-robin keeps load balanced)
    worker_assignments = [[] for _ in range(N_WORKERS)]
    for i, cluster_pairs in enumerate(cluster_pair_batches):
        worker_assignments[i % N_WORKERS].append(cluster_pairs)

    total_pairs_written = 0
    total_batches = 0

    with ThreadPoolExecutor(max_workers=N_WORKERS) as pool:
        futures = {}
        for worker_id, assigned_clusters in enumerate(worker_assignments):
            # Flatten all clusters assigned to this worker into one iterable
            # so the worker processes them sequentially within its thread.
            # Each cluster is still written as a separate transaction (clean
            # boundary for retry / progress tracking).
            future = pool.submit(
                _write_worker_clusters,
                driver,
                assigned_clusters,
                seed,
                worker_id,
            )
            futures[future] = worker_id

        for future in as_completed(futures):
            worker_id = futures[future]
            try:
                pairs, batches, elapsed = future.result()
                total_pairs_written += pairs
                total_batches += batches
                log.info(
                    "Worker %d done: %d pairs, %d batches in %.1f s",
                    worker_id, pairs, batches, elapsed,
                )
            except Exception as exc:
                log.error("Worker %d failed: %s", worker_id, exc, exc_info=True)
                raise

    elapsed = time.perf_counter() - t0
    log.info(
        "Neo4j write complete: %d pairs, %d transactions in %.1f s (%.0f pairs/s)",
        total_pairs_written,
        total_batches,
        elapsed,
        total_pairs_written / elapsed if elapsed > 0 else 0,
    )
    return total_pairs_written


def _write_worker_clusters(driver, assigned_clusters: list, seed: float, worker_id: int):
    """
    Process all clusters assigned to a single worker thread sequentially.
    Returns aggregated (pairs_written, batches_committed, elapsed_s).
    """
    total_pairs = 0
    total_batches = 0
    t0 = time.perf_counter()

    with driver.session() as session:
        for cluster_pairs in assigned_clusters:
            for i in range(0, len(cluster_pairs), BATCH_SIZE):
                sub = cluster_pairs[i : i + BATCH_SIZE]
                batch_payload = [{"a": a, "b": b} for a, b in sub]
                session.run(CYPHER_MERGE, batch=batch_payload, seed=seed)
                total_pairs += len(sub)
                total_batches += 1

    return total_pairs, total_batches, time.perf_counter() - t0


# ---------------------------------------------------------------------------
# Step 5: Verify — count edges written
# ---------------------------------------------------------------------------

def verify_edge_count(driver):
    """Query Neo4j for total CO_RETRIEVED edge count post-bootstrap."""
    with driver.session() as session:
        result = session.run(
            "MATCH ()-[r:CO_RETRIEVED]-() RETURN count(r) AS n"
        )
        record = result.single()
        return record["n"] if record else 0


# ---------------------------------------------------------------------------
# Main
# ---------------------------------------------------------------------------

def main():
    t_job_start = time.perf_counter()
    log.info("=== Bootstrap CO_RETRIEVED graph (K=%d) ===", K)

    # Validate config
    if not NEO4J_PASSWORD:
        raise RuntimeError(
            "NEO4J_PASSWORD environment variable is required. "
            "Set it before running this script."
        )

    # Clients
    qdrant = QdrantClient(host=QDRANT_HOST, port=QDRANT_PORT, timeout=60)
    driver = GraphDatabase.driver(
        NEO4J_URI,
        auth=(NEO4J_USER, NEO4J_PASSWORD),
        max_connection_pool_size=N_WORKERS + 2,  # one per worker + headroom
    )

    try:
        # 1. Fetch vectors
        ids, matrix = fetch_all_vectors(qdrant, QDRANT_COLLECTION)
        n_memories = len(ids)
        log.info("Corpus size: %d memories, embedding dim: %d", n_memories, matrix.shape[1])

        if n_memories < K * 2:
            raise ValueError(
                f"Corpus ({n_memories} memories) is too small for K={K} clusters. "
                f"Reduce K to at most {n_memories // 2}."
            )

        # 2. Cluster
        labels = cluster_vectors(matrix, k=K, n_init=N_INIT)

        # 3. Generate pairs
        cluster_pair_batches = generate_cluster_pairs(ids, labels, k=K)

        # 4. Write to Neo4j
        total_written = write_edges_parallel(driver, cluster_pair_batches, seed=BOOTSTRAP_SEED)

        # 5. Verify
        edge_count = verify_edge_count(driver)
        log.info(
            "Verification: Neo4j reports %d CO_RETRIEVED edges "
            "(directed count, ~2x undirected pairs written=%d)",
            edge_count, total_written,
        )

        elapsed_total = time.perf_counter() - t_job_start
        log.info(
            "=== Bootstrap complete in %.1f s (%.1f min) — %d edges written ===",
            elapsed_total, elapsed_total / 60, total_written,
        )

    finally:
        driver.close()
        qdrant.close()


if __name__ == "__main__":
    main()
```

## Runtime Analysis

### Component breakdown on CasaOS N100

| Phase | Operation | Time estimate | Notes |
|---|---|---|---|
| Vector fetch | Qdrant scroll, 22,423 × 1024-dim | 30–60 s | Network + disk I/O; ~91.8 MB at 100 MB/s LAN effective |
| Clustering | KMeans K=200, n_init=10, 10 iter | 4–5 s | Compute-bound (Q172): 92.54 GFLOP at 40–60% SSE4.2 BLAS efficiency |
| Pair generation | itertools.combinations, 1.25M pairs | ~1 s | CPU + memory; pure Python but bounded |
| Neo4j writes | 1.25M pairs, 10K batch, 4 workers | 90–102 s | I/O-bound; 4 × ~5K ops/s effective |
| **Total** | | **~2–3 min** | Dominated by Neo4j write phase |

### Physics derivation (from Q172)

- **Clustering physics floor**: 0.844 s (100% N100 SIMD utilization, 10 iterations).
  Practical: 4–5 s at 40–60% OpenBLAS SSE4.2 efficiency. The N100 has no AVX/FMA3,
  costing ~2x vs AVX2+FMA hardware at the same clock. This cannot be optimized in software.

- **Neo4j write physics floor**: 49.8 s (100K IOPS NVMe, 4 IOPS/MERGE, 1.25M pairs).
  Nominal with 10K batching + 4 workers: ~90–102 s. Without batching (single MERGE/tx):
  ~41 minutes — a 25x architecture gap closed entirely by batch sizing.

- **K=200 batch safety**: 6,229 pairs/cluster < 10,000 batch limit → one commit per cluster,
  125 total commits. No cluster-spanning commit logic needed. (Q172 threshold: K ≥ 158 for
  safety at 10K limit.)

### Expected edge count

```
N = 22,423 memories
K = 200 clusters
Mean cluster size = 22,423 / 200 = 112.1 members
Pairs per cluster = 112.1 × 111.1 / 2 = 6,229
Total pairs = 6,229 × 200 = 1,245,766

Neo4j MATCH ()-[r:CO_RETRIEVED]-() returns ~2,491,532
(Neo4j counts each direction of an undirected MERGE separately)
```

This is 7.02x the Q163 active-pool density threshold (177,600 edges) — the bootstrap
immediately crosses the threshold that organic accumulation at K=10 retrieval would
take 2+ years to reach.

## Key Findings

1. **The complete bootstrap job runs in approximately 2–3 minutes on the N100**, not
   8–12 minutes. The 8–12 minute figure from Q163 was the correct order of magnitude for
   the Neo4j write step only; Q172 corrected the attribution. Clustering itself takes 4–5 s
   and is not the bottleneck.

2. **No Ollama call is required.** Vectors are fetched directly from Qdrant with
   `with_vectors=True`. The 91.8 MB corpus is streamed via scroll in ~30–60 s over LAN.
   This makes the job fully self-contained and runnable without GPU server availability.

3. **The bootstrap seed strength is 0.1, not 0.5.** The Q167 flush worker uses 0.5 as the
   ON CREATE default for live co-retrievals (empirically confirmed pairs). Bootstrap edges
   are structural priors, not confirmed signal — 0.1 places them below the ~0.3 ordering-flip
   threshold. Live reinforcement then rapidly promotes edges that are genuinely co-retrieved
   above this floor while unpromoted cluster edges remain dormant.

4. **Four parallel Neo4j writer threads capture a 4x throughput gain** with zero lock
   contention. K-means hard assignment means cluster membership is disjoint — workers write
   to non-overlapping node-pair sets. Implementation cost: `ThreadPoolExecutor(max_workers=4)`
   wrapping a per-thread `driver.session()`. This is the dominant optimization available
   (collapses 6.9-minute single-threaded nominal to ~1.7 minutes).

5. **Concurrency with flush_co_retrievals**: The bootstrap must complete before the ARQ
   worker is activated on first run. On re-runs, concurrent execution is safe —
   `MERGE ... ON MATCH SET r.strength = CASE ...` is idempotent (it increments strength,
   never resets it). The flush worker and the bootstrap job never conflict at the transaction
   level because MERGE semantics handle race conditions correctly.

6. **The Cypher pattern from Q167 is used with one modification**: the ON CREATE seed
   is 0.1 (not 0.5 as in the flush worker). The ON MATCH accumulation uses +0.1 delta
   (matching the flush worker's increment, capped at 1.0). The undirected MERGE pattern
   `MERGE (a)-[r:CO_RETRIEVED]-(b)` is preserved — CO_RETRIEVED is symmetric.

7. **After bootstrap, the graph immediately exceeds the measurable improvement threshold.**
   Q163 established 177,600 active-pool edges as the density floor for detectable retrieval
   ordering change on ≥5% of queries. Bootstrap writes 1,245,766 edges — 7x the threshold —
   making the first week of live reinforcement immediately meaningful rather than a warmup
   period measured in months.

8. **The script is safe to re-run.** MERGE idempotency means re-running after the
   flush_co_retrievals worker has been active only increments existing edge strengths by 0.1
   per cluster pair. This is acceptable behavior and does not corrupt the graph state.

## Interaction with flush_co_retrievals Worker (Q167)

```
Deployment sequence (respecting all prerequisites):

  1. Apply Q156 migration     → creates co_retrievals_outbox table in PostgreSQL
  2. Add CO_RETRIEVED to      → models.py RelationshipType enum (Q167 File 1)
     RelationshipType enum
  3. Create flush worker      → src/workers/flush_co_retrievals.py (Q167 File 2)
  4. Register in main.py      → import + functions list + cron (Q167 File 3)
  5. Update health.py filter  → add "CO_RETRIEVED" to gravity set (Q167 File 4)
  6. *** RUN BOOTSTRAP ***    → python bootstrap_co_retrieved.py (~2-3 min)
  7. Restart worker service   → ARQ picks up flush_co_retrievals cron (15-min cadence)

Steps 1–5 can be done while the system is live. Step 6 is the bootstrap job.
Step 7 activates the live reinforcement loop. Steps 6 and 7 must not overlap
on first deployment; they are safe to overlap on subsequent re-runs.
```

The bootstrap and the flush worker are complementary, not competing:
- **Bootstrap** (one-time): seeds the graph with structural priors from corpus geometry.
  Provides the initial density (~7x threshold) that organic K=10 accumulation cannot
  reach within 90 days.
- **Flush worker** (recurring every 15 min): reinforces edges that are actually co-retrieved
  at runtime. Promotes structurally-seeded edges that are genuinely relevant; ignores
  cluster-internal pairs that are never co-retrieved in practice.

After 30–60 days of live reinforcement, the graph reflects both corpus structure and
actual usage patterns — the bimodal force profile described in Q163 (Scenario B).

## Ideas

- **IDEA bootstrap-dry-run-mode**: Add a `--dry-run` flag that fetches vectors, runs
  clustering, computes pair counts, and prints the runtime/edge estimate without writing to
  Neo4j. Useful for validating Qdrant connectivity and estimating actual cluster size
  variance before committing the write job.

- **IDEA incremental-recluster**: Schedule a quarterly re-clustering run that fetches only
  new memories added since the last bootstrap (using Qdrant's `created_at` filter), assigns
  them to existing centroids (no full KMeans re-fit), and writes CO_RETRIEVED edges for
  new intra-cluster pairs only. This keeps the bootstrap current without a full 1.25M-edge
  rewrite as the corpus grows past 22K.

- **IDEA cluster-label-as-neo4j-property**: Write the cluster label (0–199) as a property
  on each Memory node during bootstrap (`SET m.cluster_label = pair.cluster`). This enables
  Cypher queries that retrieve all members of a given semantic cluster directly, without
  traversing CO_RETRIEVED edges — useful for debug inspection and for the dashboard's
  force profile view.

- **IDEA gpu-offload-for-large-corpus**: At N > 100K memories, the clustering step
  becomes material (Q172: ~45 s at N=100K on N100). At that scale, offload KMeans to the
  RTX 3090 (Ollama host, 192.168.50.62) via cuML or faiss-gpu. The network transfer cost
  (~0.7 s at 1 Gbps for 91.8 MB) is already below the N100 clustering time at current
  corpus scale, making the GPU path network-latency-dominated rather than FLOP-dominated.
  Not needed at 22K; relevant at 100K+.

- **IDEA strength-decay-for-cold-bootstrap-edges**: After 90 days, apply a one-time decay
  pass (`SET r.strength = r.strength * 0.5 WHERE r.source = 'bootstrap' AND r.updated_at
  < datetime() - duration({days: 90})`). This down-weights cluster edges that were never
  reinforced by live retrieval — structural priors that turned out not to reflect actual
  usage — preventing them from permanently occupying retrieval bandwidth with stale signal.
