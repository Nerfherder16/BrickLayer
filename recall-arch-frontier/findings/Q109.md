## Architecture Analysis

### Memory Isolation Threat Model

The failure mode to prevent is **cross-tenant memory injection**: Sadie retrieves a Tim memory about a Claude Code debugging session and injects "FastAPI router refactor" into her response to a family recipe question. Or Tim's LLPC retrieval pulls a Sadie-generated family context ("Emma prefers no onions") into a coding session where it has zero relevance, consuming injection budget and displacing technically useful memories.

This is not a theoretical concern. If both tenants share a single Qdrant collection with undifferentiated vectors, the ANN graph traversal has no structural barrier against cross-tenant retrieval. The HNSW graph builds connections based on embedding cosine similarity. Sadie's family memories ("birthday party planning") and Tim's technical memories ("Qdrant HNSW parameter tuning") occupy different regions of embedding space — but edge cases exist: generic memories like "set a reminder" or "check the status" will embed near both tenants' query distributions. A `tenant_id` payload filter on a shared HNSW graph does not prevent those neighbor links from forming in the first place; it only hides them at query time. The ANN graph structure is permanently contaminated even when the query filter is applied.

The secondary threat is **epoch clock collision**: if Tim and Sadie share a single Redis epoch counter, Tim's session at 10am and Sadie's voice interaction at 10:03am both INCR the same counter. Tim's CO_RETRIEVED edges and Sadie's edges share an epoch axis, making the tau and tau_edge windows meaningless as independent per-tenant controls. A session dense with Sadie's family interactions would advance the epoch counter, making Tim's memories appear staler than they are relative to Tim's own usage pattern.

The tertiary threat is **causal graph cross-contamination in Neo4j**: if Tim and Sadie share a single Neo4j graph database with no tenant partition, a CO_RETRIEVED_FRESH edge could form between Tim's technical memory node and Sadie's family memory node if — hypothetically — the same content was retrieved in both contexts. More practically, the BFS traversal from a Tim seed node would traverse the entire graph topology including Sadie's subgraph unless a `WHERE n.tenant_id = 'tim'` predicate is applied at every hop. The cost analysis from Q091 establishes that property filters on BFS traversals are post-filter operations that still traverse the full chain. At multi-tenant scale, this means Tim's BFS traverses the combined Tim+Sadie graph and discards Sadie's nodes after reading them — wasting traversal time proportional to Sadie's graph size.

### Per-Store Multi-Tenant Design

#### 1. Qdrant — SEPARATE COLLECTIONS REQUIRED (not payload filter)

**Recommendation**: `recall_tim` and `recall_sadie` as distinct collections.

**Why payload filter is insufficient**: Qdrant's HNSW graph is built globally across all vectors in a collection. When two tenants' vectors occupy the same collection, the HNSW construction algorithm creates neighbor links between any two vectors within cosine proximity regardless of their `tenant_id` payload. A query with `filter: {must: [{key: "tenant_id", match: {value: "tim"}}]}` is applied as a post-ANN filter: Qdrant first traverses the HNSW graph to find the approximate k-nearest neighbors in the full embedding space, then discards any results where `tenant_id != "tim"`. Crucially, the traversal path through the HNSW graph can pass through Sadie's vectors as intermediate bridge nodes — the ANN traversal score is influenced by the geometric layout of the combined embedding space, including Sadie's vectors, even though they are filtered from the final result. This is the ANN leakage problem: the presence of Sadie's voice memory embeddings changes which of Tim's memories appear as ANN neighbors of any given query, even after filtering.

At Tim's current scale (360 points), this contamination is minor. But Sadie's voice memory distribution will be distinct: short transcribed utterances, named-entity-dense (family member names, food preferences, calendar events), lower semantic complexity. These embeddings cluster differently from Tim's technical memories. The HNSW graph will attempt to bridge these clusters via low-probability neighbor links, which degrade retrieval precision for both tenants.

Collection-level isolation provides **absolute ANN isolation**: the HNSW graph for `recall_tim` is built exclusively from Tim's vectors. The graph for `recall_sadie` is built exclusively from Sadie's vectors. There is no mechanism by which a Tim query can traverse a Sadie vector or vice versa.

**Schema change**: Create `recall_sadie` collection with identical parameters to `recall_tim` (vector size, distance metric, HNSW m/ef_construction). All existing data in `recall` migrates to `recall_tim`. No changes to point payload schema required (tenant_id field unnecessary per collection).

**Operational note**: `recall_tim` starts at ~360 points. `recall_sadie` starts empty. Qdrant's memory per collection at ~360 points is approximately 2–5 MB (vector store + HNSW graph at default m=16 parameters). Two collections at current Tim scale add ~5–10 MB total. Negligible on CasaOS.

#### 2. PostgreSQL — TENANT_ID COLUMN (Option A, not separate schemas)

**Recommendation**: Add `tenant_id VARCHAR(32) NOT NULL DEFAULT 'tim'` to the `memories` table. Add composite index on `(tenant_id, content_hash)`. All queries gain `WHERE tenant_id = :tenant_id`.

**Why not separate schemas**: PostgreSQL schemas (`tim.memories`, `sadie.memories`) require connection-pool routing per tenant, schema-qualified table names throughout the ORM, and separate migration tracks. This is operational overhead disproportionate to a two-tenant homelab deployment. The tenant_id column approach is standard practice for low-tenant-count multi-tenancy and requires only query-layer isolation (enforced by the API) rather than schema-layer isolation.

**Why not separate databases**: Completely disproportionate. Separate PostgreSQL database instances require separate connection pools, separate pg_wal streams, separate backup procedures. Two-tenant system with 32 bytes of additional storage per row does not warrant this.

**Isolation guarantee**: PostgreSQL tenant_id isolation is **query-enforced** (not structural). The FastAPI API layer is the isolation boundary: every database query must include `WHERE tenant_id = :tenant_id`. A bug in the API layer that omits this predicate would allow cross-tenant reads. This is the correct tradeoff for a homelab system: the isolation overhead matches the actual security risk (this is not a SaaS product where tenants are adversarial strangers; this is a home network where Sadie and Tim simply need separate memory namespaces).

**Migration path**: `ALTER TABLE memories ADD COLUMN tenant_id VARCHAR(32) NOT NULL DEFAULT 'tim';` — existing 360 rows receive `tenant_id = 'tim'` automatically via the DEFAULT. Add composite index: `CREATE INDEX idx_memories_tenant_hash ON memories (tenant_id, content_hash);`. Zero data loss, zero downtime migration.

**Schema change summary**:
```sql
-- Migration
ALTER TABLE memories
  ADD COLUMN tenant_id VARCHAR(32) NOT NULL DEFAULT 'tim';

CREATE INDEX idx_memories_tenant_hash
  ON memories (tenant_id, content_hash);

-- All queries gain this predicate
WHERE tenant_id = :tenant_id
-- (e.g., 'tim' or 'sadie')
```

All tags, audit logs, and last_access_epoch entries follow the same column addition. If there are separate tables for tags or sessions, apply the same `tenant_id` column to each.

#### 3. Neo4j — RELATIONSHIP TYPE PREFIXING (not separate instances)

**Recommendation**: Prefix all tenant-scoped relationship types: `TIM_CO_RETRIEVED_FRESH`, `TIM_CO_RETRIEVED_STALE`, `SADIE_CO_RETRIEVED_FRESH`, `SADIE_CO_RETRIEVED_STALE`. Add `tenant_id` property to all Memory nodes.

**Why not separate Neo4j instances**: Neo4j Community Edition runs as a single Docker container. Running two Neo4j instances on CasaOS requires two containers with separate port mappings, separate data volumes, separate JVM heap allocations (Neo4j's default heap is 512MB–1GB), and separate backup procedures. Memory cost: approximately 1–2 GB additional RAM on a system where RAM is a constrained resource. For two tenants whose graph datasets are small (Tim: months of sessions; Sadie: voice interactions that will be sparser), this is disproportionate overhead.

**Why relationship type prefixing instead of property filtering**: Q091 established that Neo4j property filters on BFS traversals are post-filter operations — they traverse the full relationship chain and discard non-matching records. A `WHERE n.tenant_id = 'tim'` predicate on a BFS traversal reads all CO_RETRIEVED_FRESH edges (including Sadie's) and discards Sadie's nodes after loading them. Neo4j organizes relationship chains **per relationship type**: a node's `TIM_CO_RETRIEVED_FRESH` chain contains only Tim-tenant edges. BFS traversal over `TIM_CO_RETRIEVED_FRESH` reads only Tim's fresh edges at the structural level — no Sadie edges are loaded, discarded, or traversed. This matches the type-partitioning insight from Q091 and extends it to serve as the tenant isolation mechanism simultaneously.

**Node isolation**: Add `tenant_id` property to all Memory nodes. This allows cross-tenant queries (e.g., admin/health queries that span tenants) without the relationship type prefix, and provides a secondary isolation check. For all user-facing retrieval queries, the relationship type prefix is the primary isolation guarantee (structural), and `WHERE n.tenant_id = :tenant_id` is a defensive secondary predicate.

**Migration path**: Existing nodes get `SET n.tenant_id = 'tim'`. Existing `CO_RETRIEVED_FRESH` edges are renamed to `TIM_CO_RETRIEVED_FRESH` via a migration Cypher batch. New Sadie edges are written as `SADIE_CO_RETRIEVED_FRESH` from first write.

```cypher
-- Migration: tag existing nodes
MATCH (n:Memory)
SET n.tenant_id = 'tim';

-- Migration: reclassify existing relationship types
-- (Neo4j does not support RENAME relationship type directly;
--  must use MERGE+DELETE pattern from Q091's reclassification job)
MATCH (a:Memory)-[r:CO_RETRIEVED_FRESH]->(b:Memory)
MERGE (a)-[r2:TIM_CO_RETRIEVED_FRESH]->(b)
SET r2 = properties(r)
DELETE r;

MATCH (a:Memory)-[r:CO_RETRIEVED_STALE]->(b:Memory)
MERGE (a)-[r2:TIM_CO_RETRIEVED_STALE]->(b)
SET r2 = properties(r)
DELETE r;
```

At Tim's current scale (~360 Memory nodes, estimated 500–1,000 CO_RETRIEVED edges), this migration runs in under 30 seconds.

**BFS query per tenant** (Tim example):
```cypher
MATCH (seed:Memory {id: $seed_id, tenant_id: 'tim'})
      -[r:TIM_CO_RETRIEVED_FRESH]-(neighbor:Memory)
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau_tim)
RETURN neighbor.id LIMIT $k_tim
```

Sadie's BFS uses `SADIE_CO_RETRIEVED_FRESH` and `$tau_sadie`, `$k_sadie`. The relationship type prefix guarantees structural isolation at the graph traversal level.

**Reclassification job**: Both tenants get independent reclassification jobs. The session-end hook for Tim's Claude Code sessions runs the `TIM_CO_RETRIEVED_FRESH → TIM_CO_RETRIEVED_STALE` reclassification. Sadie's Family Hub session handler runs the `SADIE_CO_RETRIEVED_FRESH → SADIE_CO_RETRIEVED_STALE` reclassification. They share the same Neo4j instance but operate on non-overlapping relationship type chains.

#### 4. Redis — KEY PREFIX PER TENANT (trivial)

**Recommendation**: Change `recall:global_epoch` to `recall:tim:global_epoch` and create `recall:sadie:global_epoch` independently.

**Why**: Epoch clocks must be independent. If Tim and Sadie share an epoch counter, every Sadie voice interaction advances Tim's epoch clock, making Tim's edges appear staler than they are relative to Tim's own retrieval patterns. The tau windows become meaningless without independent epoch references.

**Schema change**: Two Redis keys:
- `recall:tim:global_epoch` — INCR'd at each Tim session end (currently `recall:global_epoch`)
- `recall:sadie:global_epoch` — INCR'd at each Sadie voice interaction cycle (new)

**Migration**: Rename the existing key: `RENAME recall:global_epoch recall:tim:global_epoch`. Initialize Sadie's: `SET recall:sadie:global_epoch 0`. Zero data loss; the current epoch value carries over for Tim.

**Isolation guarantee**: Structural. Redis keyspace isolation by key name provides absolute separation between two epoch counters. There is no mechanism for Tim's epoch to influence Sadie's epoch or vice versa.

**Optional keyspace isolation**: Redis supports logical keyspace separation via database numbers (`SELECT 0` for Tim, `SELECT 1` for Sadie) or Redis 7.0+ key prefix access control via ACLs (`~recall:tim:*` for one client, `~recall:sadie:*` for another). For a two-tenant homelab system, the key prefix convention is sufficient without these additional layers.

### API Changes

The FastAPI API at `:8200` requires the following additions:

**New parameter on all endpoints**: `tenant_id: str = Query(default="tim", pattern="^[a-z_]{1,32}$")`.

Endpoints requiring the parameter:
- `POST /store` — routes write to `recall_{tenant_id}` Qdrant collection, `memories` table with `tenant_id`, writes to `recall:{tenant_id}:global_epoch`
- `GET /retrieve` — queries `recall_{tenant_id}` Qdrant collection, WHERE tenant_id in Postgres, BFS on `{TENANT_ID}_CO_RETRIEVED_FRESH` in Neo4j
- `POST /reconcile` — scoped to tenant (orphan cleanup runs per-tenant collection only)
- `GET /health` — add `tenant_id` scope; health metrics are per-tenant (drift_fraction for `TIM_CO_RETRIEVED_FRESH` vs `SADIE_CO_RETRIEVED_FRESH`)

**New per-tenant configuration endpoint** (recommended):
```
GET /config/{tenant_id}
→ { "tau": int, "tau_edge": int, "k": int, "token_budget": int }
```

**Authentication note**: For a homelab system, `tenant_id` as a query parameter with no authentication is acceptable — Tim's Claude Code hooks send `?tenant_id=tim`, Sadie's Family Hub sends `?tenant_id=sadie`. Adding a simple API key per tenant (Bearer token or header) is recommended but not required for initial deployment.

**Backward compatibility**: The `default="tim"` on all endpoints means Tim's existing Claude Code hooks require zero changes. The hooks currently call the API without a `tenant_id` parameter; they will continue to work and implicitly operate on the Tim tenant. Sadie's Family Hub adds `?tenant_id=sadie` to all calls.

### Tenant Configuration

Tim and Sadie require distinctly different parameter profiles driven by their interaction modalities:

| Parameter | Tim (Claude Code sessions) | Sadie (Voice interactions) |
|---|---|---|
| `tau` (node staleness window) | 20 sessions | 10 interaction cycles |
| `tau_edge` (edge staleness window) | 30 sessions | 15 interaction cycles |
| `K` (BFS neighbor limit) | 38 (tiered injection) | 5–8 (compact voice context) |
| `token_budget` | ~4,000 tokens | <200 tokens |
| Epoch unit | Per Claude Code session | Per voice interaction cycle |
| Retrieval style | Tiered hierarchical injection | Single compact passage |
| Memory content type | Technical/procedural | Relational/contextual/schedule |
| Primary embedding queries | Code patterns, error messages, architecture | Named entities, preferences, calendar |

**Rationale for Sadie's tighter tau**: Sadie's voice interactions are time-sensitive. A family preference expressed 2 weeks ago ("Emma is vegetarian now") is highly relevant. A preference expressed 6 months ago that was never reinforced may be stale or overridden. The 10-cycle tau aligns with a roughly 2-week active window at typical Sadie interaction density.

**Rationale for Sadie's K=5–8**: Voice responses must be immediate and concise. Injecting 38 graph neighbors into a voice interaction context would dominate the response generation latency and produce verbose, incoherent responses. Sadie needs the top-5 most strongly reinforced recent associations, not a deep causal graph traversal.

**Configuration storage**: Per-tenant config can be stored as a PostgreSQL table (`tenant_configs`) or as a FastAPI startup config file. A simple YAML config at API startup is sufficient for two tenants:

```yaml
tenants:
  tim:
    tau: 20
    tau_edge: 30
    k: 38
    token_budget: 4000
    qdrant_collection: recall_tim
    neo4j_rel_prefix: TIM
    redis_epoch_key: recall:tim:global_epoch
  sadie:
    tau: 10
    tau_edge: 15
    k: 6
    token_budget: 180
    qdrant_collection: recall_sadie
    neo4j_rel_prefix: SADIE
    redis_epoch_key: recall:sadie:global_epoch
```

**Neo4j — dual tenant relationship type chains**:

- After: `TIM_CO_RETRIEVED_FRESH/STALE` + `SADIE_CO_RETRIEVED_FRESH/STALE` chains
- Sadie's voice interaction graph will be smaller than Tim's (fewer reinforcement cycles, smaller per-session edge count). Estimated Sadie graph at 6 months: ~200–400 CO_RETRIEVED edges (Tim has ~800 after the same period)
- Additional Neo4j storage: ~200–400 relationship records × 34 bytes = ~8–14 KB on disk; negligible JVM heap impact

- **Verdict**: Negligible. The JVM heap is dominated by the buffer pool, not by the relationship record count at this scale. No additional container needed.

**PostgreSQL — tenant_id column**:
- Additional storage: 32 bytes × (Tim's row count + Sadie's row count) ≈ 32 × 1,200 rows = ~38 KB

- **Verdict**: Negligible.

**Redis — two epoch keys**:
- Two 64-bit integer keys: 128 bytes total
- **Verdict**: Rounding error.

**Summary**: The dual-tenant architecture adds approximately 3–10 MB of total RAM across all stores, with zero CPU overhead at Tim's scale. The dominant operational cost is the one-time migration effort (schema changes, data relabeling), not ongoing resource consumption.

### Migration Path

The migration must be performed while the system is idle (Tim not actively running Claude Code, Sadie not actively interacting). Estimated total migration time: under 5 minutes.

**Step 1: Redis key rename** (instant, no data loss)
```bash
redis-cli RENAME recall:global_epoch recall:tim:global_epoch
redis-cli SET recall:sadie:global_epoch 0
```

**Step 2: PostgreSQL column addition** (seconds for <5,000 rows)
```sql
ALTER TABLE memories
  ADD COLUMN tenant_id VARCHAR(32) NOT NULL DEFAULT 'tim';
CREATE INDEX idx_memories_tenant_hash ON memories (tenant_id, content_hash);
-- Verify: all rows now have tenant_id = 'tim'
SELECT tenant_id, count(*) FROM memories GROUP BY tenant_id;
```
Apply the same column addition to any sibling tables (tags, sessions, audit_log).

**Step 3: Qdrant collection rename/copy** (seconds at 360 points)
- Create `recall_tim` collection with identical configuration as current `recall`
- Upload all existing vectors from `recall` into `recall_tim` (API scroll + upload batch)
- Create empty `recall_sadie` collection
- **Do not delete `recall` until integration tests pass** — keep it as a read-only backup for one week
```python
# Pseudocode — use Qdrant Python client
scroll_result = qdrant.scroll("recall", limit=1000)
qdrant.upsert("recall_tim", points=scroll_result.points)
```

**Step 4: Neo4j relationship type migration** (20–60 seconds at 500–1,000 edges)
```cypher
-- Tag all existing nodes as Tim's tenant
MATCH (n:Memory)
SET n.tenant_id = 'tim';

-- Migrate CO_RETRIEVED_FRESH → TIM_CO_RETRIEVED_FRESH
MATCH (a:Memory)-[r:CO_RETRIEVED_FRESH]->(b:Memory)
MERGE (a)-[r2:TIM_CO_RETRIEVED_FRESH]->(b)
SET r2 = properties(r);
-- Verify count matches, then:
MATCH ()-[r:CO_RETRIEVED_FRESH]->() DELETE r;

-- Migrate CO_RETRIEVED_STALE → TIM_CO_RETRIEVED_STALE
MATCH (a:Memory)-[r:CO_RETRIEVED_STALE]->(b:Memory)
MERGE (a)-[r2:TIM_CO_RETRIEVED_STALE]->(b)
SET r2 = properties(r);
MATCH ()-[r:CO_RETRIEVED_STALE]->() DELETE r;

-- If CO_RETRIEVED (pre-Q091 partition) still exists:
MATCH (a:Memory)-[r:CO_RETRIEVED]->(b:Memory)
MERGE (a)-[r2:TIM_CO_RETRIEVED_FRESH]->(b)
SET r2 = properties(r);
MATCH ()-[r:CO_RETRIEVED]->() DELETE r;
```

**Step 5: API configuration update**
- Add tenant config YAML
- Add `tenant_id` parameter to all endpoints
- Update all queries to route to tenant-specific stores
- Run integration test: Tim's existing Claude Code hook calls `/retrieve?tenant_id=tim` (or no parameter, using default) and receives correct memories

**Step 6: Family Hub integration**
- Update Sadie's Family Hub API calls to include `?tenant_id=sadie`
- First Sadie `/store` call creates initial Sadie memories; Qdrant collection `recall_sadie` receives first vectors

**Rollback**: If migration produces errors, Step 1–2 are immediately reversible (rename Redis key back, DROP COLUMN on PostgreSQL). Steps 3–4 should be executed with pre-migration snapshots (Qdrant collection backup via `/snapshot` API; Neo4j data directory copy). The one-week retention of the original `recall` collection provides an ANN rollback path.

## Ideas Extracted

### Idea: relationship-type-as-tenant-partition
- **Novelty**: 0.78 — Using Neo4j relationship type prefixes (`TIM_CO_RETRIEVED_FRESH`, `SADIE_CO_RETRIEVED_FRESH`) as the primary tenant isolation mechanism in a multi-tenant graph database — simultaneously serving as tenant isolation AND temporal freshness partitioning (from Q091) — is not documented in Neo4j multi-tenancy literature. Community Edition's single-database constraint forces this approach; the insight is that the relationship type namespace already provides structural per-chain isolation, meaning the tenant and freshness dimensions can be encoded together in a single type prefix without performance penalty. This double-duty use of relationship types is architecturally novel.
- **Evidence**: 0.75 — Follows directly from Q091's finding that Neo4j organizes relationship chains per type. The structural isolation guarantee (BFS over `TIM_CO_RETRIEVED_FRESH` reads zero Sadie edges) is a direct consequence of Neo4j's storage architecture, not an assumption. The four-type schema (`TIM_CO_RETRIEVED_FRESH`, `TIM_CO_RETRIEVED_STALE`, `SADIE_CO_RETRIEVED_FRESH`, `SADIE_CO_RETRIEVED_STALE`) is derivable from the cross-product of {tenant} × {freshness state}. No empirical validation required — the isolation is structural.
- **Feasibility**: 0.72 — Requires a one-time migration Cypher batch (Step 4 above), updates to all Cypher queries referencing `CO_RETRIEVED_FRESH/STALE`, and updates to the reclassification job to use tenant-prefixed relationship types. The migration is well-specified and runs in under 60 seconds at Tim's current scale. The main risk is query surface completeness: any Cypher query that still references the old `CO_RETRIEVED_FRESH` type after migration will return zero results (not wrong results — a detectable failure mode). The drift_fraction health metric from Q104 can be adapted to monitor per-tenant: `TIM` and `SADIE` drift fractions independently.
- **Description**: Replace the two-type CO_RETRIEVED partition with a four-type schema: `TIM_CO_RETRIEVED_FRESH`, `TIM_CO_RETRIEVED_STALE`, `SADIE_CO_RETRIEVED_FRESH`, `SADIE_CO_RETRIEVED_STALE`. Neo4j's per-type relationship chain organization provides structural tenant isolation at zero marginal cost: BFS traversal over `TIM_CO_RETRIEVED_FRESH` reads only Tim's fresh edges. This eliminates the need for `WHERE n.tenant_id = :tenant` post-filter predicates on the BFS path, achieving both tenant isolation and freshness partitioning at the structural level. The reclassification job runs independently per tenant, with independent tau_edge parameters (Tim: 30, Sadie: 15).

### Idea: separate-qdrant-collections-as-hnsw-isolation
- **Novelty**: 0.70 — The argument that a shared HNSW graph with payload filtering provides only query-time tenant isolation while collection-level separation provides graph-structure-time tenant isolation — specifically that cross-tenant vectors influence each other's ANN neighborhoods even under filtering — is a concrete articulation of ANN leakage that is not commonly discussed in multi-tenant vector database design literature. Most multi-tenant vector DB guides (Pinecone, Weaviate, Qdrant documentation) recommend payload filtering for "soft" multi-tenancy and separate collections for "hard" multi-tenancy, but do not explain the HNSW graph structure mechanism that makes the distinction non-obvious.
- **Evidence**: 0.72 — The ANN leakage mechanism is derivable from HNSW algorithm internals (Malkov & Yashunin 2018): neighbor links are constructed greedily during insertion, selecting the ef_construction nearest neighbors regardless of payload. A payload filter at query time removes these neighbors from results but does not undo the graph structure. The claim that Sadie's embeddings would influence Tim's ANN neighborhood is most credible for semantically adjacent memories (both involve domestic task management, general questions, reminders). For maximally distinct tenants (e.g., Sadie's family conversations vs. a financial trading system's technical memories), the contamination would be near-zero. Tim's case is intermediate — some semantic overlap exists.
- **Feasibility**: 0.90 — Creating a second Qdrant collection is a single API call. At Tim's scale (~360 points), the operational overhead is negligible (3–5 MB additional RAM). The migration from a single `recall` collection to `recall_tim` is a scroll + upsert batch operation taking under 30 seconds. No Qdrant configuration changes required; the client simply routes writes and reads to the tenant-appropriate collection name.
- **Description**: Create separate Qdrant collections `recall_tim` and `recall_sadie` rather than using a single collection with `tenant_id` payload filtering. Each collection has its own HNSW graph built exclusively from that tenant's vectors. ANN traversal for Tim queries never touches Sadie's embedding space, and vice versa. The performance profile of each tenant's retrieval is independent — Sadie's potentially noisier voice transcription embeddings do not degrade the quality of Tim's technical memory ANN graph. At 360 Tim points + 0 Sadie points (initial state), the RAM cost is 3–5 MB additional. At estimated steady-state (500 Tim points + 1,000 Sadie points after 6 months), the cost remains under 15 MB total for both collections.

### Idea: independent-epoch-clocks-per-tenant
- **Novelty**: 0.65 — The framing of epoch clocks as tenant-scoped resources — where sharing an epoch clock causes tau-window distortion by advancing one tenant's temporal reference frame via the other tenant's activity density — is not explicitly stated in multi-tenant memory system literature. Most discussions of epoch-based temporal decay treat the epoch counter as a global system clock. The insight that the epoch is actually a session-density normalization device (it advances once per session, not once per calendar unit) means that two tenants with different session densities must have independent epochs or the tau window becomes asymmetric: high-density tenant (Sadie, many short voice interactions) advances the clock faster, making the low-density tenant's memories appear staler than the low-density tenant's own usage pattern would imply.
- **Evidence**: 0.68 — The tau-distortion argument follows from the epoch counter semantics established in earlier project findings: `global_epoch` is INCR'd once per Claude Code session (or once per interaction cycle for Sadie). If the counter is shared, a day with 10 Sadie voice interactions and 1 Tim session advances the epoch by 11 — but Tim's memories should age at the rate of Tim's 1 session, not Sadie's 10. With a shared counter, Tim's tau window of 20 epochs would correspond to 20 combined events (Sadie + Tim), which might cover only 2 days of real usage rather than the intended ~20 Tim sessions (approximately 20 days at Tim's current density). This is a concrete, quantifiable distortion.
- **Feasibility**: 0.95 — A Redis key rename + new key creation (2 commands, instant, zero data loss). One configuration update to route epoch reads and writes to the tenant-appropriate key. This is the lowest-effort change in the entire multi-tenant migration — it requires no schema migration, no data transformation, and no query rewrites beyond replacing the hardcoded key name with a tenant-parameterized key name.
- **Description**: Replace the single `recall:global_epoch` Redis key with two tenant-scoped keys: `recall:tim:global_epoch` and `recall:sadie:global_epoch`. Each epoch clock advances only when its tenant has an interaction. Tim's tau window of 20 epochs corresponds to approximately 20 Tim Claude Code sessions, regardless of how many Sadie voice interactions occur. Sadie's tau window of 10 epochs corresponds to 10 Sadie interaction cycles. Without independent epoch clocks, the tau windows become meaningless as independent per-tenant controls — high-density tenant activity would erode the low-density tenant's memory relevance windows at an accelerating rate. Migration: 2 Redis commands in under 1 second.

### Idea: default-tenant-backward-compatibility
- **Novelty**: 0.65 — Adding `tenant_id: str = Query(default="tim")` to FastAPI endpoints — maintaining full backward compatibility with Tim's existing hooks (which send no tenant_id parameter) while enabling Sadie's hooks to opt into the new tenant namespace by sending `?tenant_id=sadie` — is a clean zero-disruption deployment pattern. The default-to-tim approach means the multi-tenant migration produces zero behavior change for Tim's existing infrastructure before Sadie is connected. This is a deployment pattern observation rather than a novel algorithm, but its application to AI memory multi-tenancy is worth formalizing.
- **Evidence**: 0.72 — FastAPI's `Query(default=...)` behavior is documented and reliable. The pattern avoids a cutover event: Tim's hooks don't need to be updated to include `?tenant_id=tim` before Sadie goes live. The multi-tenant API is deployed; Sadie's Family Hub is connected; Tim's hooks remain unchanged; both work correctly. The only risk is if a future third tenant (e.g., a family member's Claude instance) accidentally uses the default and routes to Tim's namespace — but this is caught immediately by the `WHERE tenant_id = 'tim'` constraint returning only Tim's memories to the wrong caller.
- **Feasibility**: 0.97 — Single FastAPI parameter change per endpoint. No hook changes required on Tim's side. Immediately deployable as part of the multi-tenant migration.
- **Description**: All multi-tenant API endpoints use `tenant_id: str = Query(default="tim")`. This provides backward compatibility for Tim's existing Claude Code hooks (which currently send no `tenant_id` parameter) while enabling new tenants to self-identify via query parameter. The migration can be deployed without any changes to Tim's hook scripts: the system becomes multi-tenant-capable before Sadie's integration is complete. Once Sadie's Family Hub is updated to send `?tenant_id=sadie`, both tenants operate in isolation with no further API changes needed.

## Key Finding

The minimal architectural change for absolute memory isolation between Tim and Sadie requires changes to all five stores, but all changes are lightweight on Tim's CasaOS hardware:

1. **Qdrant**: Separate collections (`recall_tim`, `recall_sadie`) — required for ANN graph isolation. Payload filtering on a shared collection is insufficient because cross-tenant vectors influence HNSW neighbor construction at insert time. RAM overhead: <10 MB.

2. **PostgreSQL**: Add `tenant_id` column with DEFAULT 'tim' — query-enforced isolation via the API layer. All existing Tim rows migrate automatically. Zero data loss, seconds to execute.

3. **Neo4j**: Relationship type prefixing (`TIM_CO_RETRIEVED_FRESH`, `SADIE_CO_RETRIEVED_FRESH`) — structural isolation at the graph traversal level, simultaneously serving as the freshness partition from Q091. No second Neo4j instance required. Migration runs in under 60 seconds.

4. **Redis**: Independent epoch keys per tenant — essential for tau-window correctness. Without independent clocks, Sadie's interaction density would advance Tim's temporal reference frame, making Tim's memories appear staler than his own usage pattern warrants.

5. **FastAPI**: `tenant_id` parameter with `default="tim"` on all endpoints — backward compatible, zero changes required to Tim's existing hooks.

The isolation guarantee hierarchy is: **Qdrant (structural, graph-level)** > **Neo4j relationship types (structural, traversal-level)** > **Redis key prefix (structural, key-level)** > **PostgreSQL tenant_id (query-enforced, API-boundary)**. The weakest link is PostgreSQL, where isolation depends on all API queries including the `WHERE tenant_id = :tenant_id` predicate. A defensive measure: add a PostgreSQL row-level security policy on `memories` if security requirements grow (not needed for current homelab threat model).

Total operational overhead on CasaOS: approximately 5–15 MB additional RAM across all stores at Sadie's estimated 6-month steady state. Zero additional Docker containers required.
