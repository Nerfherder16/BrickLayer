# Finding: Q061 — Tombstone Overhead on the G1 Gate: Theoretical Minimum and Bottleneck Threshold

**Question**: What is the minimum latency overhead added by the tombstoned registry entry mechanism (Q049) to the Phase 1 scope discriminant (G1 gate in Q041's three-gate protocol) — specifically, what is the cost of reading a tombstone entry from Redis, computing the content hash comparison, and applying the re-encounter importance multiplier — and at what registry size does the tombstone lookup become the G1 gate's bottleneck?
**Question Type**: PHYSICS
**Verdict**: PROMISING
**Severity**: Low
**Source field**: Redis internals, SHA256 hardware benchmarks, network serialization theory

---

## Operation Definition

### G1 Gate (baseline — LIVE entry)

- **Input**: `scope_key = "{topic_key}@{machine_id}"` (string, ~30 bytes typical); Redis hash `recall:scope:{topic_key}@{machine_id}`
- **Output**: UUID string (~36 bytes) — the live memory binding
- **Guarantees required**: O(1) lookup; must complete within G1's budget; result must be consistent with the live PostgreSQL row

### G1 Gate with Tombstone (extended path — TOMBSTONED entry)

- **Input**: Same scope key; Redis HGET now returns a compound struct: `status` (8 bytes) + `content_hash` (64 bytes, SHA256 hex) + `prior_importance` (8 bytes float) + `evicted_at` (8 bytes epoch) = ~100 bytes
- **Output**: `(TOMBSTONED, content_hash, prior_importance, evicted_at)` tuple; then: SHA256 of incoming memory canonical text (~800 bytes = ~200 tokens normalized); then: `new_importance = write_importance × (1.0 + 0.15 × min(age_sessions, 10))` — float arithmetic
- **Guarantees required**: Re-encounter multiplier must be applied before the importance threshold check; tombstone must not produce a dangling UUID (Q049 design); result must distinguish LIVE vs. TOMBSTONED vs. ABSENT

## Theoretical Minimum

| Resource | Minimum | Calculation |
|---|---|---|
| Network latency (G1 HGET, any path) | 0.05ms | Docker bridge RTT on same host: ~50 µs per TCP round-trip. Physics floor for localhost socket: 2–5 µs. Using Docker-bridge LAN as lower bound. |
| Payload serialization: +64 bytes (100 vs 36 bytes) | 0.000512 µs | 64 bytes × 8 bits / 1,000,000,000 bits/s (1 Gbps) = 512 ns ≈ 0.0005 µs. At 10 Gbps: 0.00005 µs. Negligible at any realistic link speed. |
| SHA256 on 800 bytes (hardware floor, SHA-NI) | 0.27 µs | SHA-NI processes ~3 GB/s. 800 bytes / 3,000,000,000 bytes/s = 0.267 µs. 800 bytes = 13 × 64-byte compression blocks, ~4 cycles/byte on SHA-NI, 3200 cycles at 3 GHz = 1.07 µs (conservative). Range: 0.27–1.1 µs. |
| SHA256 on 800 bytes (software fallback, no SHA-NI) | 1.6 µs | 800 bytes at ~500 MB/s software rate = 1.6 µs. Upper bound without hardware acceleration. |
| Re-encounter multiplier (integer arithmetic) | ~1 ns | `1.0 + 0.15 × min(age, 10)` = 1 multiply + 1 add + 1 compare. At 3 GHz, ~3 CPU cycles = 1 ns. |
| Minimum total tombstone added cost | ~1–2 µs | Payload diff (negligible) + SHA256 (~1 µs) + multiplier (~1 ns) = ~1 µs above the base HGET round-trip. |

**Key result**: The theoretical minimum added latency from the tombstone code path above a bare HGET is dominated entirely by the SHA256 computation on the incoming memory text, at approximately **1–2 microseconds**. This is 250–500x smaller than the base G1 HGET round-trip on a LAN-attached Redis instance (0.5ms = 500 µs).

## Data Structure Comparison: HASH vs. JSON vs. MessagePack

All three options require exactly **one HGET round-trip** — the network latency is identical. The differences are in payload size, client-side deserialization cost, and operational convenience.

| Criterion | Redis HASH (HMGET) | JSON string (1 HGET) | MessagePack string (1 HGET) |
|---|---|---|---|
| Round-trips required | 1 (HMGET 4 fields) | 1 | 1 |
| Payload size on wire | ~100 bytes (binary fields) | ~145 bytes (JSON overhead) | ~95 bytes |
| Network time difference vs HASH | +45 bytes → +0.00036 µs | baseline | -5 bytes → negligible |
| Client deserialization | None (fields direct) | Python `json.loads()`: 1–5 µs | `msgpack.unpackb()`: 0.5–2 µs |
| Schema evolution (add a field) | ALTER HMGET call, no migration | Update JSON key, backward-compatible | Requires coordinated update |
| Redis memory per tombstone entry | 4 fields × ~25 bytes avg = ~100 bytes | 1 field × 145 bytes = ~145 bytes | 1 field × 95 bytes = ~95 bytes |
| Operational debuggability | `HGETALL` readable | `GET` readable (plain JSON) | `GET` returns binary (opaque) |
| Partial-field read (status only) | `HGET status` — 1 HGET, ~25 bytes | Must deserialize entire blob | Must deserialize entire blob |

**Verdict on data structure choice:**

Redis HASH with HMGET is the preferred structure for three reasons:
1. It allows a fast-path status check: `HGET scope_key status` returns 8 bytes (LIVE / TOMBSTONED / absent) before committing to the full struct fetch. This adds a second round-trip on the TOMBSTONED path but eliminates struct deserialization for LIVE entries entirely.
2. Individual fields are readable in `redis-cli` without tooling, aiding debugging.
3. Field addition (e.g., adding `eviction_reason`) requires no migration of existing entries — absent fields return nil.

The performance difference between the three options is negligible (all below 5 µs, against a 500 µs HGET baseline). Choose based on operational convenience, not performance.

**One optimization available for the HASH approach**: Use a two-step read — `HGET status` first, then `HMGET content_hash prior_importance evicted_at` only on TOMBSTONED result. This costs a second round-trip (0.5ms) on the TOMBSTONED path but returns only 8 bytes (status) on LIVE entries, reducing the common-case HGET response from 100 bytes to 8 bytes. However, this is a pure implementation tradeoff: it adds 0.5ms latency on the TOMBSTONED path to save ~0.0001ms of deserialization on the LIVE path. The two-step approach is worse in practice. Single HMGET of all fields in one round-trip is the correct implementation.

## Bottleneck Threshold Analysis

**The question**: At what registry size does the tombstone lookup become the G1 gate's bottleneck?

**Answer**: Never, at any registry size achievable by this system, for the following reasons.

Redis HGET is O(1) regardless of hash size. Redis hash lookups use an internal hashtable with O(1) average-case lookup. The lookup time does not increase with the number of entries in the registry hash.

The only registry-size-dependent failure modes are:

| Failure mode | Threshold | Tim's scale | Time to reach |
|---|---|---|---|
| Redis maxmemory eviction (LRU) | Depends on maxmemory setting. At 256MB default and 200 bytes/tombstone: 1,280,000 entries | ~50,000 evictions/year (Q049) | ~25 years |
| Redis single-threaded command saturation | >100,000 commands/second sustained | Recall is low-throughput (hook-driven, ~1–10 ops/sec) | Never at Tim's scale |
| Hash-to-ziplist promotion overhead | Redis promotes hashes from ziplist encoding to hashtable at >128 fields (default hash-max-ziplist-entries). The scope registry is one hash per topic_key@machine_id, not one giant hash. Each entry is a separate hash. No size-dependent encoding change applies. | N/A | N/A |

**Practical bottleneck conclusion**: The G1 gate bottleneck is the LAN TCP round-trip (0.3–0.5ms physics floor on external Redis), not the tombstone payload size or registry cardinality. The tombstone mechanism adds ~2–7 µs of overhead that is constant with registry size and constant with tombstone count. No registry size causes it to become a bottleneck relative to the base HGET.

**The only regime where tombstone lookup could become disproportionate**: If Redis were co-located on localhost (0.05ms HGET), the SHA256 computation on a large incoming memory (e.g., 10,000-token content = 40KB normalized text) would take ~13 µs (hardware SHA-NI) to ~80 µs (software). At 80 µs SHA256 vs 50 µs HGET, the compute step would dominate. But Q049's design normalizes content before hashing, and 200-token (~800 byte) content is the typical target — well within the sub-2-µs SHA256 range even on localhost-Redis deployments.

## Gap Analysis

### Physics gaps (unavoidable)

- **LAN TCP round-trip for Redis HGET**: 0.3–0.5ms — Redis is on 192.168.50.19 (a separate physical host). The speed-of-light minimum for a 1m Ethernet cable is 3.3 nanoseconds; the practical floor for a TCP round-trip on a 1 Gbps switched LAN is ~50–100 µs. The 0.5ms gap above the 0.05ms floor is partly physics (TCP handshake phases, interrupt coalescing, NIC buffers) and partly architecture.
- **SHA256 hardware floor**: ~1 µs for 800-byte input. Cannot be reduced without hardware cryptographic acceleration beyond SHA-NI, which already achieves this floor. Not a practical concern at this scale.

### Architecture gaps (design choices)

- **External Redis over LAN (0.45ms excess above localhost floor)**: Redis on 192.168.50.19 adds 0.45ms above the 0.05ms localhost floor. If Redis were co-located in the same Docker network as the FastAPI server (same host, bridge network), G1 would cost ~0.05ms instead of 0.5ms — a 10× improvement. This is an architecture decision: Redis is a shared infrastructure component on CasaOS, not embedded with the API server. Removing this gap requires co-location (or Unix socket) of Redis with the Recall API container. The tradeoff is operational complexity vs. latency.
- **Two-struct-field reads when status-only suffices for LIVE path**: The current design (implied by Q049) reads the full struct on every G1 lookup. A status-first read (`HGET status` → only on TOMBSTONED fetch full struct) could halve the response payload for LIVE entries. Architecture gap: the protocol does not distinguish fast-path (LIVE, common case) from slow-path (TOMBSTONED, less common). However, as calculated above, this adds a second round-trip on the TOMBSTONED path, making it strictly worse. The gap exists but the optimization is not worth taking.
- **SHA256 redundancy: G3 already computes SHA256 of the same content**: Q041's G3 gate computes `SHA256(normalize(content))` as the idempotency key. The tombstone path in G1 computes `SHA256(normalize(incoming_content))` to compare against the stored `content_hash` in the tombstone. These are the same computation on the same input. If G3 runs before G1 (Q041's concurrent protocol), the SHA256 is already available in the G3 result by the time G1 fires. The tombstone's SHA256 comparison can reuse the pre-computed hash rather than recomputing it. This eliminates the ~1–2 µs SHA256 cost entirely.

### Implementation gaps (code quality)

- **SHA256 recomputation when G3 result is available**: If G3 and G1 run concurrently (Q041's asyncio.gather), G3's SHA256 is computed first and its result is available before G1 processes the TOMBSTONED status. An implementation that passes `g3_content_hash` into the tombstone comparison step eliminates the redundant SHA256. This is a one-line change: pass `content_hash` as a parameter to the G1 decision function rather than recomputing it. Savings: ~1–2 µs.
- **Tombstone struct parsing**: If the tombstone value is stored as a JSON string (not HASH fields), deserializing it via `json.loads()` on every HGET adds 1–5 µs. Using Redis HASH fields with HMGET eliminates JSON parsing entirely. This is an implementation choice, not an architecture one.

## Ideas Extracted

### Idea: g3-sha256-passthrough-to-tombstone-comparison
- **Novelty**: 0.72
- **Evidence**: 0.68 — Hash reuse between pipeline stages is a standard optimization in streaming data systems (e.g., Kafka Streams materializing intermediate results, Apache Flink's operator chaining). The specific application — passing G3's SHA256 result into G1's tombstone comparison to eliminate redundant computation — is a trivial implementation optimization. Its novelty lies in the explicit recognition that Q041's concurrent G3+G1 protocol already computes the hash the tombstone needs, and the design should make this explicit in the function signature rather than allowing recomputation.
- **Feasibility**: 0.95 — One-line change: the `unified_store` function passes `content_hash` computed in Step 0 into the G1 tombstone path. No new infrastructure. No schema changes. Zero risk.
- **Description**: In Q041's UNIFIED_STORE protocol, `content_hash = SHA256(normalized)` is computed at Step 0 (before G3 and G1 run concurrently). The G1 tombstone comparison needs SHA256 of the incoming content to compare against the tombstone's stored hash. Since the Step 0 hash is already computed, the tombstone comparison function should accept `content_hash` as a parameter. This eliminates the ~1–2 µs recomputation that would otherwise occur when a TOMBSTONED registry entry is found. The saving is small in absolute terms (1–2 µs vs 500 µs HGET) but represents a design hygiene principle: any value computed once in a pipeline should be threaded through subsequent stages rather than recomputed.

### Idea: redis-colocated-with-recall-api-for-sub-100us-g1
- **Novelty**: 0.68
- **Evidence**: 0.82 — Redis co-location with the application server is a standard deployment pattern for latency-sensitive lookup paths. Every major Redis deployment guide (Redis Labs docs, Antirez's original blog posts, AWS ElastiCache placement guides) recommends co-location when sub-millisecond latency is required. The specific application — moving the Recall scope registry to a Redis instance running in the same Docker network as the FastAPI API server — follows directly from this literature. The novelty is low because the pattern is well-known; the evidence is high for the same reason.
- **Feasibility**: 0.70 — Requires either (a) running a dedicated Redis container in the Recall Docker Compose stack alongside the API server, or (b) moving to Unix domain socket communication if Redis and API are on the same host. Option (a) introduces a second Redis instance (one for Recall scope registry on localhost, one for the shared CasaOS Redis). Operational complexity increases: two Redis instances to monitor, back up, and maintain. Option (b) is simpler if the Recall API and Redis are already on the same CasaOS host, but requires the API container to have access to the Redis Unix socket path via a Docker volume mount. Either option reduces G1 from 0.5ms to ~0.05ms — a 10× latency improvement on the gate.
- **Description**: The G1 gate's 0.5ms cost is 99.6% TCP round-trip overhead and 0.4% useful work (payload serialization + parsing). Co-locating a Redis instance with the Recall API server in the same Docker Compose service group (same bridge network, loopback addresses) reduces G1 to ~50 µs — the practical floor for a Docker bridge round-trip. This makes the G1 gate cost 0.05ms vs. Q041's budget of 5ms: a 100× margin vs. the current 10× margin. The impact is larger on the tombstone path because the tombstone adds 2–7 µs of local compute; at 0.05ms HGET baseline, the tombstone overhead is 4–14% of the total gate cost (vs. 0.4–1.4% at 0.5ms baseline). Co-location matters more when the physics floor is lower.

### Idea: tombstone-status-field-as-hget-fast-path-sentinel
- **Novelty**: 0.65
- **Evidence**: 0.55 — Using a sentinel value in a frequently-read field to enable early-exit without fetching the full record is a standard database optimization (covering indexes, partial column reads). The novelty here is applying this to a Redis HASH tombstone design: structuring the HASH so the `status` field is the first field checked, and if `status == "LIVE"`, no further fields are needed. Redis HGET of a single field returns only that field's value (8 bytes) regardless of how large the overall hash is. A two-phase read (HGET status → HMGET remaining fields only if TOMBSTONED) eliminates struct deserialization on the common LIVE path, at the cost of a second round-trip on the uncommon TOMBSTONED path.
- **Feasibility**: 0.60 — Implementation is straightforward: replace single HMGET with conditional two-step (HGET status; if TOMBSTONED, HMGET content_hash prior_importance evicted_at). The cost is a second LAN round-trip on the TOMBSTONED path: 0.5ms additional latency when the tombstone IS hit. The benefit is saving 1–5 µs of JSON/struct deserialization on LIVE hits. Given that LIVE hits are the common case and TOMBSTONED hits are rare (only when a topic has been evicted and re-encountered), this tradeoff is negative: the optimization costs more than it saves. Feasibility score is marked down because implementing it would make the TOMBSTONED path 0.5ms slower for no meaningful gain on the LIVE path. This idea is documented for completeness but should not be implemented.
- **Description**: Structure the Redis HASH tombstone so that a single `HGET status` call (returning 8 bytes: "LIVE" or "TOMBSTONED") serves as a sentinel for the full-struct fetch. On LIVE, the G1 gate completes with a second HGET for the UUID (or store UUID as the LIVE status value itself: `status == UUID` implies LIVE). On TOMBSTONED, fetch remaining fields. This eliminates deserialization on LIVE entries at the cost of a second round-trip on TOMBSTONED entries. Net effect: negative for typical workloads where TOMBSTONED hits are rare. Not recommended. Documented as an architecture anti-pattern: two-step reads with unequal cost profiles are only beneficial when the slow path is the common case.

## Key Finding

The tombstoned registry entry mechanism (Q049) adds **effectively zero latency** to the G1 gate in Q041's three-gate protocol. The theoretical minimum overhead is ~1–2 microseconds (SHA256 on 800-byte input with hardware acceleration), and the current implementation cost is ~2–7 microseconds (including struct deserialization). Against a 500-microsecond HGET baseline driven by LAN TCP round-trip physics, the tombstone overhead is **0.4–1.4% of G1 gate cost** and **0.14% of Q041's 5ms G1 budget**.

The G1 gate never becomes a bottleneck due to registry size. Redis HGET is O(1). At Tim's eviction rate (~50,000/year), the registry reaches Redis maxmemory limits in approximately 25 years. No registry cardinality threshold exists at which tombstone lookup latency increases.

The only material optimization identified is a design hygiene fix: pass the `content_hash` already computed at Step 0 of Q041's UNIFIED_STORE protocol into the G1 tombstone comparison, eliminating a redundant SHA256 computation. This saves ~1–2 µs and is a one-line change to the function signature.

The largest latency gap in the G1 gate (0.45ms excess above physics floor) comes from the architectural choice to host Redis on a separate physical host (192.168.50.19) rather than co-locating it with the Recall API server. This is not caused by the tombstone mechanism — it exists equally in the LIVE path. Co-locating Redis with the API server would reduce the G1 gate from 0.5ms to ~0.05ms, a 10× improvement independent of the tombstone design.

**Verdict is PROMISING rather than BREAKTHROUGH** because: (1) the tombstone overhead is confirmed negligible, which validates the Q049 design choice; (2) the SHA256 passthrough optimization is a genuine small improvement; and (3) the Redis co-location idea, though not novel in isolation, has a meaningful latency impact that compounds with the three-gate protocol's overall budget.

## Suggested Follow-ups

- **[IMPLEMENTATION] Q062**: Implement the `g3-sha256-passthrough-to-tombstone-comparison` optimization in the UNIFIED_STORE protocol: pass `content_hash` from Step 0 into the G1 tombstone comparison function. Verify in code that the same `SHA256(normalize(content))` value used for G3 idempotency is reused for the tombstone content-hash comparison, eliminating recomputation. Measure actual savings on Recall's FastAPI stack.

- **[PHYSICS] Q063**: What is the actual Redis round-trip time from the Recall API container to the Redis instance at 192.168.50.19? Q041 assumed 0.5ms. Measure `redis-cli --latency -h 192.168.50.19` from the Recall API container to get the empirical p50/p99 distribution. If p99 exceeds 1ms (due to Docker bridge NAT, network switch queuing, or Redis keyspace notifications overhead), the G1 budget consumption is higher than Q041 estimated, and the co-location case becomes stronger.

- **[CONVERGENCE] Q064**: The tombstone struct carries `content_hash` (SHA256 of the evicted memory's canonical text). Q041's G3 gate also carries `content_hash` (SHA256 of the incoming write). When a tombstoned topic is re-encountered with byte-identical content (same exact text), G3 would fire (content duplicate found) before G1 processes the tombstone. Is this interaction correct? If G3 fires on an exact re-encounter of an evicted memory, it returns the cached response (cached UUID-A) — but UUID-A no longer exists in PostgreSQL or Qdrant (it was evicted). This is a latent bug: G3's cached response contains a stale UUID that the system will fail to resolve. Define the correct interaction between G3 (72-hour idempotency window) and tombstoned entries (30-session TTL), especially when the two TTLs overlap.

- **[ADVERSARIAL] Q065**: The re-encounter multiplier formula `1.0 + 0.15 × min(age_sessions, 10)` is evaluated at G1 gate time (before the importance threshold check). This means the multiplier is applied to `write_importance` of the incoming new memory, not to the `prior_importance` stored in the tombstone. Is this the correct variable to boost? If the new memory arrives with `write_importance = 0.20` (very low) but the tombstone's `prior_importance = 0.38` (close to threshold), the multiplier on the new write gives `0.20 × 1.75 = 0.35` — still below the 0.40 threshold, and the new memory is evicted again. But if the multiplier were applied to the average of new and prior importances: `avg(0.20, 0.38) × 1.75 = 0.51` — above threshold. Which input variable to the multiplier produces the correct behavior for the re-encounter promotion goal?
