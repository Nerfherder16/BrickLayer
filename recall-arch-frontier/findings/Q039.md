# Finding: Q039 — CRDT Semantics for Recall's Scope-Qualified Identity Registry

**Question**: What mechanism do CRDTs use to merge concurrent writes without coordination, and how would it apply to Recall's scope-qualified identity registry when casaclaude and proxyclaude update the same binding simultaneously?
**Question Type**: ADJACENT
**Verdict**: PROMISING
**Severity**: High
**Source field**: Distributed systems — CRDT theory (Shapiro et al. 2011, "A Comprehensive Study of Convergent and Commutative Replicated Data Types"), production deployments in Riak KV, Redis CRDT extensions (RedisJSON / CRDB in Redis Enterprise), Azure Cosmos DB conflict resolution, Cassandra LWW columns, Automerge/Yjs collaborative editing

---

## Evidence

### Step 1: The Problem Being Solved

Recall maintains a Redis hash that functions as a **scope-qualified identity registry**: a mutable binding from `{topic_key}@{machine_id}` to a memory UUID. The use case for a concurrent conflict is:

1. casaclaude observes a file edit referencing "the CasaOS host address."
2. proxyclaude, within the same ~100ms window, observes a related file edit on its side.
3. Both independently compute the same binding key (e.g., `casaos_host@casaclaude`) and attempt to `HSET` it to a new UUID.

Redis's single-threaded command execution serializes the two `HSET` operations — but which one wins is non-deterministic (arrival-order dependent). The business question is: which UUID should the registry hold after both writes complete, and should both machines agree on that value afterward?

This is distinct from the write-deduplication problem (Q036). Q036's problem was preventing duplicate insertions into the memory store. Q039's problem is: given that two different UUIDs are legitimately competing for the same binding slot, what convergence semantics are correct, and how does Recall implement them durably?

### Step 2: What CRDTs Are — the Mechanism Precisely

**Foundational paper**: Shapiro, Preguiça, Baquero, Zawirski, "A Comprehensive Study of Convergent and Commutative Replicated Data Types," INRIA Technical Report RR-7506, 2011. Deposited in peer review as "Conflict-Free Replicated Data Types," SSS 2011. This is the canonical reference — the definitions below are from this work.

A CRDT is a data structure with two properties:

1. **State-based (CvRDT)**: Each replica holds a full state. A merge function `merge(s1, s2)` is defined that is commutative, associative, and idempotent. Replicas gossip full states; any merge order produces the same result.
2. **Operation-based (CmRDT)**: Replicas exchange operations rather than states. Operations are commutative when delivered concurrently. The delivery layer must guarantee causal ordering and exactly-once delivery per operation.

The key mathematical property: the merge function implements a **join-semilattice** — it maps two states to their least upper bound in a partial order. The partial order defines which state is "more advanced." Merging never loses information relative to that order.

**Production deployments (not academic proposals):**

| System | CRDT type used | Context |
|--------|---------------|---------|
| Riak KV (Basho, 2012–present) | LWW-Register, MV-Register, PN-Counter, OR-Set | Distributed key-value store, millions of production deployments |
| Redis Enterprise CRDB (2017–present) | LWW-Register for string keys, OR-Set for set types, PN-Counter for counters | Multi-region active-active Redis clusters |
| Azure Cosmos DB (2017–present) | LWW with configurable conflict resolver, custom merge procedures | Global distributed document store |
| Cassandra (DataStax, 2015–present) | LWW-columns with server-side timestamp | Wide-column store, petabyte-scale deployments |
| Apple CloudKit | Operational CRDTs for notes, contacts | 850M device sync at scale |
| Automerge / Yjs (2018–present) | RGA for text, LWW-Map for objects | Collaborative document editing, used in production by Notion-adjacent tools |

### Step 3: CRDT Variant Analysis for Recall's Registry

#### Variant A: LWW-Register (Last-Write-Wins Element Register)

**Mechanism:**

Each register value is paired with a timestamp `(value, timestamp)`. The partial order is: `(v1, t1) ≤ (v2, t2)` iff `t1 ≤ t2`. The merge function is:

```
merge((v1, t1), (v2, t2)) = (v1, t1) if t1 > t2
                           = (v2, t2) if t2 > t1
                           = deterministic_tiebreak(v1, v2) if t1 == t2
```

The tiebreak when `t1 == t2` is typically lexicographic comparison of the value (or UUID) so all replicas agree.

**Production implementations:**
- Cassandra LWW columns: `WRITETIME()` metadata, server-side comparison. Deployed at petabyte scale.
- Redis Enterprise CRDB: LWW resolution for conflicting `SET` operations on the same key, using millisecond-precision wall-clock time from the originating shard.
- Cosmos DB: Default conflict resolution mode is LWW on the `_ts` field (server-assigned timestamp).

**Applied to Recall's registry:**

The binding `casaos_host@casaclaude → UUID-A` from casaclaude and `casaos_host@casaclaude → UUID-B` from proxyclaude arrive at Redis within 100ms. With LWW:

- Redis stores whichever `HSET` arrives last (this is LWW-by-default in single-node Redis — the later write overwrites the earlier one).
- The surviving UUID is whichever machine had the higher wall-clock time at write time.
- If timestamps are embedded in the UUID (UUIDv7 contains a millisecond timestamp), the tiebreak is built in.

**The clock skew problem for Recall:**

casaclaude and proxyclaude are separate LAN machines (not co-located). Clock skew on a LAN with NTP is typically 1–10ms but can spike to 100ms during NTP corrections. The Recall write window is also ~100ms. This creates a non-trivial probability that `t_casaclaude > t_proxyclaude` when the "later" write was actually the stale one.

**Specific failure mode:** proxyclaude writes a newer, more authoritative UUID (from a session that just extracted a refined fact). casaclaude's hook, running 3ms later on wall clock (but holding a stale value), overwrites it. The registry now points to the older, less accurate memory. LWW conflates "later wall clock" with "more authoritative" — these are not the same under clock skew.

**Verdict on LWW for Recall's registry:** Technically correct implementation of LWW semantics, but the wrong semantics for this use case. "Latest authoritative value should win" does not map to "latest wall-clock timestamp wins" when clock skew is non-negligible relative to the conflict window.

**Mitigation if LWW is used:** Hybrid Logical Clocks (HLC) — used in CockroachDB and YugabyteDB — track both physical time and logical causality. An HLC advances monotonically and never goes backward, even after NTP corrections. HLC timestamps would make LWW robust to clock skew. Adding HLC to Recall would require embedding a HLC counter in the hook client (a ~30-line library) and storing `(UUID, hlc_timestamp)` as the registry value instead of bare UUID. This narrows the failure window from 100ms to near zero, at the cost of slightly more complex value encoding.

#### Variant B: MV-Register (Multi-Value Register)

**Mechanism:**

Each value is tagged with a vector clock `(value, VectorClock)`. The partial order is causal: `(v1, vc1) ≤ (v2, vc2)` iff `vc1` causally precedes `vc2`. The merge function:

```
merge(S1, S2) = {
    all (v, vc) in S1 ∪ S2 such that no (v', vc') in S1 ∪ S2 has vc' > vc
}
```

i.e., the merge result contains ALL values whose vector clocks are not dominated by any other value. If casaclaude and proxyclaude write concurrently (neither's vc dominates the other's), BOTH values survive in the register. The application layer must detect the concurrent state (multiple values) and resolve it.

**Production implementations:**
- Riak KV "siblings": when `allow_mult=true`, Riak returns all concurrent sibling values to the client on read. The client application must implement a resolver (typically compare content and pick one, or merge) and write back the resolved value with a new VectorClock. This is well-documented in Riak's client libraries.
- Amazon's Dynamo (2007, the original Dynamo paper by DeCandia et al.) uses vector-clock-based versioning with application-level conflict resolution for shopping cart merges.

**Applied to Recall's registry:**

When both machines write concurrently to the same binding key, the registry would hold two UUIDs: `[UUID-A (casaclaude), UUID-B (proxyclaude)]`. On the next read, Recall's API layer receives both values and must decide:

1. Query both UUIDs from the memory store.
2. Compare their content (or timestamps, or importance scores).
3. Pick one (or merge their content into a new memory).
4. Write the resolved UUID back with a new vector clock.

**Why this is too complex for Recall's binding semantics:**

Recall's registry is not a collaborative document — it is a binding table. The value at `casaos_host@casaclaude` should be a single UUID (the most recent authoritative memory about CasaOS). The MV-Register's multi-value state is appropriate when both concurrent values carry independent, non-dominatable information (as in shopping cart merges or collaborative text). For a binding that means "the current best memory for this topic," concurrent sibling values are a complexity cost without a clear resolution strategy.

The MV-Register would require Recall to implement a conflict resolver that reads and compares memory content from Qdrant/PostgreSQL on every conflicted read. This makes every read potentially involve a secondary memory lookup — violating the < 500ms p95 retrieval SLA in the worst case.

**Verdict on MV-Register for Recall's registry:** Semantically correct for preserving concurrent writes, but application-level resolution overhead is too high for a binding registry. Appropriate only if the binding itself should accumulate history rather than converge to a single authoritative value.

#### Variant C: 2P-Set (Two-Phase Set)

**Mechanism:**

A 2P-Set maintains two grow-only sets: `A` (add-set) and `R` (remove-set, tombstones). An element `e` is in the set iff `e ∈ A AND e ∉ R`. Removes are permanent: once `e ∈ R`, it can never be re-added.

```
merge(2P1, 2P2) = (A1 ∪ A2, R1 ∪ R2)
```

**Production implementations:** 2P-Set is a foundational CRDT building block, used internally in Riak and Redis CRDB, though end-users typically work with OR-Sets (see below). It appears in Shapiro et al. as a primitive.

**Applied to Recall's registry:**

The 2P-Set is designed for set membership (is topic X bound?), not for mutable registers (what UUID is topic X bound to?). Recall's registry has mutable values: a binding for `casaos_host@casaclaude` might change its UUID over time as the associated memory is refined. 2P-Set's permanent tombstones mean that once a UUID is removed from the registry, it can never be re-bound — which breaks the lifecycle where bindings evolve as memories are updated.

Specifically: if casaclaude stores an improved memory for CasaOS and writes a new UUID to the registry, the old UUID binding needs to be superseded (not permanently tombstoned). Under 2P-Set semantics, the old UUID would be in `R` (tombstoned) and the new UUID would be in `A`. This works for a single update. But if proxyclaude then produces another improved memory with UUID-C and tries to update the binding, it would add UUID-C to `A` and tombstone UUID-B in `R`. The 2P-Set would then contain `{UUID-A (tombstoned), UUID-B (tombstoned), UUID-C}` — the set grows with each supersession, never truly replacing the previous binding.

**Verdict on 2P-Set:** Wrong data type for the registry use case. Tombstone permanence is fundamentally incompatible with a mutable binding that should hold exactly one current value. The 2P-Set accumulates historical UUIDs that are no longer valid bindings — registry bloat with no mechanism for true replacement.

#### Variant D: OR-Set (Observed-Remove Set)

**Mechanism:**

Each add operation generates a unique tag `t` for the element: `add(e) → (e, t)`. The set contains elements `e` for which at least one unretracted tag exists. Remove requires specifying the exact tags to retract: `remove(e, tags)`. Concurrent add and remove of the same element are safe: the add generates a new tag that the concurrent remove has not seen, so the element survives.

```
merge(OR1, OR2) = {(e, t) : t ∈ tags(OR1) ∪ tags(OR2) AND t ∉ retracted(OR1) ∪ retracted(OR2)}
```

**Production implementations:**
- Riak KV with `datatype=set`: Riak's distributed set type uses OR-Set semantics. Adding to a set that is simultaneously being removed from on another replica results in the add winning (the add's tag is unretracted). This is extensively deployed.
- Redis Enterprise CRDB set type: Uses a similar observed-remove mechanism for the Redis `SADD`/`SREM` commands on cross-region replicated keys.

**Applied to Recall's registry:**

OR-Set semantics apply cleanly to set membership but again face the mutable register problem. The registry value is not set membership (is this topic bound?) but a mutable value (what UUID is this topic bound to?). An OR-Set can answer "is casaos_host@casaclaude in the registry?" but cannot hold a single authoritative UUID as the current binding value.

One could model the registry as an OR-Set of `(topic_key, UUID)` pairs, where each binding update is an add of the new pair and a remove of all prior tags for the same topic_key. This is more complex than needed: the binding is not a set — it is a register. Forcing OR-Set semantics onto a register problem adds the tag management overhead without a clear benefit over a properly implemented LWW-Register with Hybrid Logical Clocks.

**Verdict on OR-Set:** Better than 2P-Set (removes are not permanent), but OR-Set semantics are designed for set membership, not mutable register values. The tag management overhead to simulate a mutable register is a round-peg-square-hole fit.

#### Variant E: RGA (Replicated Growable Array)

**Mechanism:**

RGA is a CRDT for ordered sequences (lists, text). Each element is identified by a unique `(session_id, sequence_number)` pair, called an `s4vector`. Insertion at position P generates a new element with an s4vector that deterministically orders it relative to all concurrent insertions. Deletion is tombstoning (element stays in the array, visibility is false). The merge of two RGA states is a total order over all elements consistent with all local ordering decisions.

**Production implementations:**
- Google Docs (uses an OT variant, not RGA, but functionally similar for text)
- Figma's multiplayer editing
- Automerge (2019–present): RGA underlies the `List` and `Text` types
- Yjs (2018–present): Uses YATA (Yet Another Transformation Approach), similar to RGA

**Applied to Recall's registry:**

RGA is for ordered sequences. Recall's scope registry is a key-value map, not a sequence. The binding key `{topic_key}@{machine_id}` is a hash key, not a position in an array. RGA does not apply to key-value registries.

The only plausible stretch: if Recall wanted to track the ordered history of UUID bindings for a topic (UUID-A superseded by UUID-B superseded by UUID-C, in causal order), RGA could represent this as an ordered sequence of (UUID, timestamp) pairs. But this is overengineering a binding registry into a changelog — the registry already has this through the PostgreSQL memories table's `created_at` timestamps.

**Verdict on RGA:** Inapplicable to the registry use case. Designed for ordered text/list sequences, not mutable key-value bindings.

### Step 4: The Correct CRDT for Recall's Registry — LWW-Register with Hybrid Logical Clock

After eliminating 2P-Set (permanent tombstones), OR-Set (set-membership semantics mismatch), MV-Register (application-level resolution overhead), and RGA (sequence type), LWW-Register is the correct CRDT type for the registry. Its failure mode — clock skew between LAN machines — is solvable with Hybrid Logical Clocks.

**Hybrid Logical Clocks (HLC)** — Kulkarni, Demirbas, Madappa, Avva, Leone, "Logical Physical Clocks and Consistent Snapshots in Globally Distributed Databases," OPODIS 2014. Deployed in production in: CockroachDB (2016–present), YugabyteDB (2018–present), TiDB (2019–present).

HLC maintains two components: `(physical_time, logical_counter)`. Rules:

```
On send/local event:
    hlc.l = max(hlc.l, physical_clock())
    hlc.c = (hlc.l == old_hlc.l) ? hlc.c + 1 : 0
    emit (hlc.l, hlc.c)

On receive(m):
    hlc.l = max(hlc.l, m.hlc.l, physical_clock())
    if hlc.l == m.hlc.l == old_hlc.l:
        hlc.c = max(hlc.c, m.hlc.c) + 1
    elif hlc.l == m.hlc.l:
        hlc.c = m.hlc.c + 1
    elif hlc.l == old_hlc.l:
        hlc.c = hlc.c + 1
    else:
        hlc.c = 0
```

The HLC never goes backward, advances monotonically, and reflects causal precedence: if event A causally precedes event B, `hlc(A) < hlc(B)`. This makes HLC timestamps safe for LWW even under NTP-induced clock skew — the logical counter component breaks ties that wall-clock comparisons would get wrong.

**For Recall's registry specifically:**

- Each hook client (the observe-edit.js hook) maintains an HLC, initialized to `(physical_clock(), 0)`.
- Every write to the registry sends `(UUID, hlc_timestamp)` as the value instead of bare UUID.
- Redis stores the full `(UUID, hlc_timestamp)` pair.
- On conflict (two concurrent writes), the value with the higher HLC wins.
- Because HLC is causally consistent, the "later" write (in causal order) always has a higher HLC regardless of clock skew.

**What state the mechanism maintains:**

```
Redis hash: {
    "{topic_key}@{machine_id}": "{uuid}:{hlc_l}:{hlc_c}"
}

Hook client (in-memory per process):
    hlc_l: integer (milliseconds since epoch, physical component)
    hlc_c: integer (logical counter, advances when physical is tied)
```

**The update rule:**

```
On write to registry:
    advance_hlc(physical_clock())
    HSET registry_key f"{topic_key}@{machine_id}" f"{new_uuid}:{hlc.l}:{hlc.c}"

On read from registry:
    value = HGET registry_key f"{topic_key}@{machine_id}"
    uuid, hlc_l, hlc_c = parse(value)
    update_hlc_from_received(hlc_l, hlc_c)  # ensures causal consistency
    return uuid
```

**The merge rule (implemented by the Recall API, not Redis):**

Redis executes HSET commands serially — the second write overwrites the first regardless of timestamp. To implement LWW-with-HLC at the Recall layer, the write must be conditional:

```
On write request to registry (Recall FastAPI):
    current = HGET registry_key topic_binding
    if current is not None:
        current_hlc = parse_hlc(current)
        if incoming_hlc <= current_hlc:
            return "stale write, rejected"
    HSET registry_key topic_binding f"{uuid}:{incoming_hlc.l}:{incoming_hlc.c}"
```

This requires a Redis Lua script (for atomicity) or a Redis transaction (`WATCH`/`MULTI`/`EXEC`) to prevent a TOCTOU race between the `HGET` and `HSET`. Redis Lua scripts execute atomically in the single-threaded engine — this is the correct implementation path.

**Production precedent:** Redis Enterprise CRDB uses precisely this mechanism for LWW resolution on conflicting string writes across regions. The implementation is a compare-and-swap using a server-side Lua script. The HLC extension is what CockroachDB adds to make the timestamps causally consistent rather than wall-clock-dependent.

### Step 5: Why AI Memory Systems Don't Do This

The scope-qualified identity registry is a Recall-specific design — most AI memory systems do not maintain a separate binding registry at all. mem0, Zep, and LangMem treat memory storage as append-only (or LLM-mediated update), never exposing a low-level binding table to concurrent writers. The multi-machine write coordination problem is unique to Recall's architecture where two independent hook clients (casaclaude, proxyclaude) write to a shared Redis hash.

No reviewed AI memory system implements HLC-stamped LWW for a binding registry because no reviewed system exposes a binding registry at all. The problem is invisible until you have two machines writing to the same identity namespace.

## Ideas Extracted

### Idea: lww-register-with-hybrid-logical-clock
- **Novelty**: 0.95 — No AI memory system implements HLC-stamped LWW for a shared binding registry. The combination of (CRDT LWW-Register) + (Hybrid Logical Clock for clock-skew-safe timestamps) + (Redis Lua script for atomic compare-and-swap) applied to an identity binding registry is not found in mem0, Zep, MemGPT, LangMem, AWS AgentCore, or any reviewed system. The individual components (HLC in CockroachDB, LWW in Cassandra, Lua CAS in Redis) are all production-proven; their composition for AI memory binding coordination is novel.
- **Evidence**: 0.85 — LWW-Register semantics: peer-reviewed in Shapiro et al. 2011 (SSS 2011) and deployed at petabyte scale in Cassandra and Riak. Hybrid Logical Clocks: peer-reviewed in Kulkarni et al. 2014 (OPODIS 2014) and deployed in production in CockroachDB, YugabyteDB, TiDB (collectively millions of production deployments). Redis Lua atomicity for compare-and-swap: Redis documentation and well-established production pattern. All three components are independently validated; their composition is mechanically straightforward.
- **Feasibility**: 0.80 — Implementation requires: (1) a 30-line HLC implementation in the observe-edit.js hook (pure JavaScript, no new dependencies), (2) a Redis Lua script (~20 lines) implementing the compare-and-swap LWW merge, (3) a change to the registry value format from bare UUID to `{uuid}:{hlc_l}:{hlc_c}`. No new infrastructure. The Recall FastAPI layer would expose a single new endpoint (`PUT /registry/binding`) that executes the Lua script. Estimated time: 1–2 weeks including testing. The main complexity is the HLC implementation and the edge-case handling when the Redis instance is unavailable (fallback: accept the write, resolve on next read).
- **Description**: Recall's scope-qualified identity registry changes from a bare UUID hash to an HLC-stamped LWW-Register. Each hook client maintains a lightweight HLC counter (initialized from wall clock, advanced logically on each write). Every registry write sends `{uuid}:{hlc_l}:{hlc_c}` as the value. The Recall FastAPI registry endpoint executes a Redis Lua script that reads the current binding's HLC, compares it to the incoming HLC, and only updates if the incoming HLC is strictly greater. When casaclaude and proxyclaude write concurrently, the one with the causally later HLC wins — and because HLC reflects causal ordering rather than wall-clock ordering, this is correct even under 100ms NTP-induced clock skew. The loser's write is rejected with a 409 response carrying the current binding's HLC, allowing the losing client to re-evaluate whether its value is genuinely newer in causal terms. This replaces the current non-deterministic last-write-wins with deterministic causal last-write-wins.

### Idea: mv-register-for-binding-history-accumulation
- **Novelty**: 0.80 — Applying MV-Register semantics to accumulate binding history (rather than converge to a single value) has not been implemented in any AI memory system. Riak's "siblings" feature uses this for shopping carts; applying it to memory binding history to enable "show me all UUIDs that have been bound to this topic" is a novel application.
- **Evidence**: 0.75 — MV-Register (Riak siblings) is production-deployed and peer-reviewed in Shapiro et al. 2011. The application to memory binding history is a design proposal; the convergence properties are directly inherited from the validated mechanism.
- **Feasibility**: 0.55 — Requires changing the registry value from a single UUID to a set of `{uuid, vector_clock}` pairs, implementing a vector clock library in the hook clients, and implementing an application-level conflict resolver in Recall's API that fetches and compares multiple UUIDs when concurrent writes are detected. This is more complex than the HLC-LWW approach. Estimated time: 3–6 weeks. The MV-Register approach is only worth this complexity if binding history is a desired product feature (e.g., "show me what the registry said about CasaOS three weeks ago"). If the registry is purely a current-binding store, HLC-LWW dominates.
- **Description**: Recall's registry optionally stores the full causal history of binding updates as an MV-Register (vector-clock-tagged set of UUIDs). On concurrent writes from casaclaude and proxyclaude, both UUIDs are preserved as siblings. On the next read, the Recall API detects the sibling state, fetches both memories from PostgreSQL, compares their creation timestamps and importance scores, picks the more authoritative one, and writes back the resolved single value with a merged vector clock. This provides temporal binding history (useful for debugging "why did the registry change?") and safe concurrent write semantics, at the cost of read-time resolution overhead. Appropriate if Recall's registry needs to function as an audit log in addition to a current-binding store.

### Idea: or-set-for-multi-binding-per-topic
- **Novelty**: 0.75 — Using OR-Set semantics to allow a topic to have multiple simultaneous active bindings (one per machine's perspective) has not been implemented in any reviewed AI memory system. The use case is: casaclaude and proxyclaude may each have locally-authoritative memories about the same topic that differ, and both are valid from their respective machine contexts.
- **Evidence**: 0.70 — OR-Set is production-deployed in Riak KV and Redis Enterprise CRDB. The semantics are peer-reviewed in Shapiro et al. 2011. The application to multi-machine memory binding is a design proposal inheriting the validated OR-Set properties.
- **Feasibility**: 0.60 — Requires redesigning the registry from `{topic_key}@{machine_id} → UUID` to `{topic_key} → OR-Set<(machine_id, UUID, tag)>`. The binding lookup changes: instead of "what UUID is this topic bound to?", it becomes "what UUIDs are bound to this topic from each machine?" Each machine's binding survives independently unless explicitly retracted. This is a schema change that touches the registry design fundamentally. Estimated time: 4–6 weeks.
- **Description**: The registry adopts OR-Set semantics: each `(machine_id, UUID)` pair gets a unique tag on insert. A remove operation specifies the tag to retract. Concurrent adds from casaclaude and proxyclaude both survive: the registry holds `{(casaclaude, UUID-A, tag-1), (proxyclaude, UUID-B, tag-2)}`. Retrieval returns the most recent binding for the requesting machine's context (or the union if both are needed). This is appropriate when the two machines should maintain independent views of their locally-authoritative bindings rather than converging to a single global binding. Tradeoff: more complex retrieval (must filter by machine context) but eliminates all conflict resolution logic — concurrent writes simply coexist.

## CRDT Variant Decision Matrix for Recall's Registry

| CRDT Variant | Correctness for Recall | Clock-skew safety | Implementation complexity | Verdict |
|---|---|---|---|---|
| LWW-Register (bare wall clock) | Partial — "latest" can be wrong under skew | No — 100ms skew within conflict window | Low | Wrong semantics |
| LWW-Register + HLC | Correct — "causally later" wins | Yes — HLC is monotonic and causally consistent | Medium | **Recommended** |
| MV-Register (siblings) | Correct — preserves all concurrent values | Yes — vector clocks, no clock dependency | High — app-level resolver required | Use only if history needed |
| 2P-Set | Wrong — permanent tombstones break mutable binding | N/A | Low | Inapplicable |
| OR-Set | Partial — set semantics, not register semantics | Yes | Medium-High | Use only if multi-binding desired |
| RGA | Wrong — sequence type, not key-value | N/A | High | Inapplicable |

The recommended path is **LWW-Register with Hybrid Logical Clock**, implemented as a Redis Lua compare-and-swap script called from the Recall FastAPI layer.
