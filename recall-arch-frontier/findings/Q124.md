# Q124 — Tombstone Regression Test Suite

**Question**: What is the tombstone regression test suite specification for the three query paths identified as MUST FIX in Q113 — covering the exact pytest test cases for (1) BFS neighbor exclusion of tombstoned nodes, (2) seed sampling exclusion of tombstoned nodes, and (3) reconcile inclusion of tombstoned nodes as legitimate orphans — such that any future refactor that accidentally removes or misplaces a tombstone guard fails the test suite immediately?
**Question Type**: IMPLEMENTATION
**Verdict**: INCREMENTAL
**Severity**: High
**Source field**: test-driven development, regression suite design, database query correctness verification, tombstone lifecycle invariants

---

## Architecture Translation: Cypher → PostgreSQL

Q113 identified three tombstone-sensitive query paths in the proposed Neo4j/Cypher architecture. Q115 (three-store collapse BREAKTHROUGH) eliminated Neo4j entirely and replaced all Cypher with PostgreSQL CTEs. This finding targets the PostgreSQL equivalents of those three paths.

The mapping is:

| Q113 Cypher Path | PostgreSQL Equivalent | Tombstone Rule |
|---|---|---|
| BFS neighbor query (`MATCH (n:Memory)-[:CO_RETRIEVED_FRESH]-(neighbor) WHERE NOT neighbor.tombstoned`) | `depth1`/`depth2` CTEs in the BFS query — JOIN against `memories m` with `AND m.is_deleted = FALSE` | MUST EXCLUDE tombstoned neighbors |
| Seed sampling for S5 compliance scan (`MATCH (n:Memory) WHERE n.is_tombstoned IS NULL`) | Seed selection: `SELECT id FROM memories WHERE ... AND is_deleted = FALSE` | MUST EXCLUDE tombstoned seeds |
| Reconcile orphan scan (`MATCH (m:Memory) RETURN m.id`) | `SELECT id FROM memories` — full table scan for set-difference against Qdrant IDs | MUST INCLUDE tombstoned nodes |

**Note on S5**: Q115 established that S5 is eliminated as a health signal after the three-store collapse — the PostgreSQL CTE enforces epoch compliance structurally. However, the tombstone guard on the seed sampling path survives in the BFS retrieve query itself (the CTE seed selection clause). The seed sampling test below targets the production BFS CTE, not a defunct S5 probe.

**Tombstone representation in PostgreSQL**: In the `memories` table, tombstoned memories are represented as soft-deleted rows with `is_deleted = TRUE` (or equivalently, `deleted_at IS NOT NULL`). The Q115 schema uses `ON DELETE CASCADE` on `co_retrievals`, meaning that once a memory row is hard-deleted from `memories`, its edges are automatically removed. The tombstone state (soft-deleted but not yet hard-deleted) is the interval between `is_deleted = TRUE` being set and the GC sweep that runs `DELETE FROM memories WHERE is_deleted = TRUE AND deleted_at < NOW() - INTERVAL '4 days'`.

### Test Path 1 — BFS Neighbor Exclusion

**Invariant**: The depth-1 BFS CTE must never return a memory whose `is_deleted = TRUE`. The tombstone guard belongs in the JOIN condition against the `memories` table inside the depth-1 (and depth-2) CTE.

**SQL under test** (production BFS depth-1 CTE):

```sql
-- CORRECT — tombstone guard present
WITH depth1 AS (
    SELECT cr.to_memory_id AS memory_id,
           cr.weight,
           1 AS depth
    FROM   co_retrievals cr
    JOIN   memories m ON cr.to_memory_id = m.id
    WHERE  cr.from_memory_id = ANY(:seed_ids)
      AND  cr.relationship_type = 'FRESH'
      AND  cr.tenant_id = 'tim'
      AND  m.is_deleted = FALSE           -- tombstone guard
    LIMIT  50
)
SELECT memory_id FROM depth1;

-- BROKEN — tombstone guard absent (regression target)
WITH depth1 AS (
    SELECT cr.to_memory_id AS memory_id
    FROM   co_retrievals cr
    WHERE  cr.from_memory_id = ANY(:seed_ids)
      AND  cr.relationship_type = 'FRESH'
      AND  cr.tenant_id = 'tim'

)
SELECT memory_id FROM depth1;
```

### Test Path 2 — Seed Selection Exclusion

**Invariant**: The seed selection query that feeds the BFS must not select a tombstoned memory as a seed node. This is the path where Q101's fix was incomplete — the fix was applied to neighbors but not to seeds.

**SQL under test** (production BFS seed selection):

```sql
-- CORRECT — tombstone guard present
SELECT id
FROM   memories
WHERE  tenant_id = 'tim'
  AND  last_access_epoch IS NOT NULL
  AND  is_deleted = FALSE             -- tombstone guard
ORDER  BY last_access_epoch DESC
LIMIT  5;

-- BROKEN — tombstone guard absent (regression target)
SELECT id
FROM   memories
WHERE  tenant_id = 'tim'
  AND  last_access_epoch IS NOT NULL
ORDER  BY last_access_epoch DESC
```

```python
# tests/recall/test_tombstone_seed_selection.py
import pytest

class TestSeedSelectionExcludesTombstones:
    """
    Regression suite for: BFS seed selection must not include tombstoned memories.
    Invariant: WHERE is_deleted = FALSE must be present in the seed query.

    Background (Q113): Q101's fix applied the tombstone guard to neighbors but not
    to seeds. A tombstoned seed causes all its neighbors to be traversed, many of
    which may also be recently deleted. This cascades into BFS result contamination
    even if the neighbor guard is correctly present.
    """

    async def test_live_memory_is_selected_as_seed(self, pg):
        """Seed query returns a live memory when it is the most recent."""
        live_id = await insert_memory(pg, content="live seed candidate", epoch=100)

        rows = await pg.fetch(
            """
            SELECT id FROM memories
            WHERE  tenant_id = 'tim'
              AND  last_access_epoch IS NOT NULL
              AND  is_deleted = FALSE
            ORDER  BY last_access_epoch DESC
            LIMIT  5
            """,
        )
        returned_ids = {row["id"] for row in rows}
        assert live_id in returned_ids, "Live memory must be selectable as a seed"

    async def test_tombstoned_memory_is_not_selected_as_seed(self, pg):
        """Seed query must not return a tombstoned memory even if it has the highest epoch."""
        # Insert a live memory with lower epoch so tombstone would win without the guard
        await insert_memory(pg, content="live low epoch", epoch=50)
        tombstone_id = await insert_memory(
            pg, content="tombstoned high epoch", epoch=999, is_deleted=True
        )

        rows = await pg.fetch(
            """
            SELECT id FROM memories
            WHERE  tenant_id = 'tim'
              AND  last_access_epoch IS NOT NULL
              AND  is_deleted = FALSE
            ORDER  BY last_access_epoch DESC
            LIMIT  5
            """,
        )
        returned_ids = {row["id"] for row in rows}
        assert tombstone_id not in returned_ids, (
            "Tombstoned memory must not be selected as a BFS seed, even when it has "
            "the highest last_access_epoch. If this fails, the tombstone guard "
            "(AND is_deleted = FALSE) has been removed from the seed selection query."
        )

    async def test_seed_query_returns_live_when_tombstone_would_otherwise_rank_first(self, pg):
        """
        With tombstone guard: live memory at epoch=50 is selected over tombstone at epoch=999.
        Without tombstone guard: tombstone at epoch=999 would rank first and be selected.
        This test distinguishes the guarded from the unguarded query.
        """
        live_id = await insert_memory(pg, content="live memory", epoch=50)
        tombstone_id = await insert_memory(
            pg, content="tombstoned memory", epoch=999, is_deleted=True
        )

        rows = await pg.fetch(
            """
            SELECT id FROM memories
            WHERE  tenant_id = 'tim'
              AND  last_access_epoch IS NOT NULL
              AND  is_deleted = FALSE
            ORDER  BY last_access_epoch DESC
            LIMIT  5
            """,
        )
        returned_ids = {row["id"] for row in rows}
        assert live_id in returned_ids
        assert tombstone_id not in returned_ids
        assert len(returned_ids) == 1, (
            "Only the live memory should be returned as a seed candidate"
        )

    async def test_seed_limit_not_consumed_by_tombstones(self, pg):
        """
        With LIMIT 5 seeds: tombstoned memories must not consume seed slots,
        leaving live memories unselected.
        """
        # 5 tombstoned nodes with very high epochs
        tombstone_ids = set()
        for i in range(5):
            tid = await insert_memory(
                pg, content=f"tombstone {i}", epoch=1000 + i, is_deleted=True
            )
            tombstone_ids.add(tid)

        # 5 live nodes with lower epochs — should fill all 5 seed slots
        live_ids = set()
        for i in range(5):
            lid = await insert_memory(pg, content=f"live {i}", epoch=100 + i)
            live_ids.add(lid)

        rows = await pg.fetch(
            """
            SELECT id FROM memories
            WHERE  tenant_id = 'tim'
              AND  last_access_epoch IS NOT NULL
              AND  is_deleted = FALSE
            ORDER  BY last_access_epoch DESC
            LIMIT  5
            """,
        )
        returned_ids = {row["id"] for row in rows}
        assert returned_ids.issubset(live_ids), (
            "All returned seeds must be live memories. "
            "Tombstoned nodes must not consume seed slots from the LIMIT."
        )
        assert len(returned_ids) == 5, "All 5 seed slots should be filled by live memories"
```

### Test Path 3 — Reconcile Inclusion of Tombstoned Nodes

**Invariant**: The reconcile orphan scan MUST return tombstoned memories in its PostgreSQL ID set. A tombstoned memory has `is_deleted = TRUE` in PostgreSQL but its Qdrant point has been hard-deleted. The reconcile set-difference (`pg_ids - qdrant_ids`) must therefore classify it as a PostgreSQL orphan — a legitimate cleanup target. If `AND is_deleted = FALSE` is mistakenly added to the reconcile query, tombstoned nodes are silently excluded and persist in PostgreSQL indefinitely (ghost rows that GC never catches if GC itself fails).

**SQL under test** (production reconcile ID scan):

```sql
-- CORRECT — no tombstone guard; tombstoned nodes appear in result (they are legitimate orphans)
SELECT id FROM memories WHERE tenant_id = 'tim';

-- BROKEN — tombstone guard present (regression target: this breaks cleanup correctness)
SELECT id FROM memories WHERE tenant_id = 'tim' AND is_deleted = FALSE;
```

```python
# tests/recall/test_tombstone_reconcile_inclusion.py
import pytest

class TestReconcileIncludesTombstonedNodes:
    """
    Regression suite for: reconcile orphan scan MUST include tombstoned memories.

    Background (Q113): Tombstoned nodes have is_deleted=TRUE in PostgreSQL but
    their Qdrant points are already hard-deleted. The reconcile set-difference
    (pg_ids - qdrant_ids) identifies them as PostgreSQL orphans — which is correct.
    They ARE orphans that need cleanup. Adding a tombstone exclusion guard here
    breaks this invariant: tombstoned nodes would be silently excluded and persist
    in PostgreSQL as ghost rows that GC never catches if GC has a failure.

    The reconcile is the last-resort safety net. It must be maximally inclusive.
    """

    async def test_reconcile_returns_live_memory_id(self, pg):
        """Reconcile ID scan returns live memory IDs."""
        live_id = await insert_memory(pg, content="live memory", epoch=100)

        rows = await pg.fetch(
            "SELECT id FROM memories WHERE tenant_id = 'tim'"
        )
        returned_ids = {row["id"] for row in rows}
        assert live_id in returned_ids, "Live memory must appear in reconcile ID scan"

    async def test_reconcile_returns_tombstoned_memory_id(self, pg):
        """
        Reconcile ID scan must return tombstoned memory IDs.
        Tombstoned nodes are legitimate orphans: present in PG, absent from Qdrant.
        If this test fails, a tombstone guard has been mistakenly added to the
        reconcile query, breaking the cleanup safety net.
        """
        tombstone_id = await insert_memory(
            pg, content="tombstoned memory", epoch=99, is_deleted=True
        )

        rows = await pg.fetch(
            "SELECT id FROM memories WHERE tenant_id = 'tim'"
        )
        returned_ids = {row["id"] for row in rows}
        assert tombstone_id in returned_ids, (
            "Tombstoned memory must appear in the reconcile ID scan. "
            "Tombstoned nodes are legitimate PostgreSQL orphans (PG row exists, "
            "Qdrant point is deleted). If this fails, a tombstone guard has been "
            "added to the reconcile query — this breaks the cleanup safety net and "
            "causes tombstoned nodes to persist in PostgreSQL indefinitely."
        )

    async def test_reconcile_returns_both_live_and_tombstoned(self, pg):
        """Reconcile scan returns all memory IDs regardless of tombstone status."""
        live_id = await insert_memory(pg, content="live memory", epoch=100)
        tombstone_id = await insert_memory(
            pg, content="tombstoned memory", epoch=99, is_deleted=True
        )

        rows = await pg.fetch(
            "SELECT id FROM memories WHERE tenant_id = 'tim'"
        )
        returned_ids = {row["id"] for row in rows}
        assert live_id in returned_ids
        assert tombstone_id in returned_ids
        assert len(returned_ids) == 2, (
            "Both live and tombstoned memories must be returned by the reconcile scan"
        )

    async def test_guard_absence_is_detectable_by_count(self, pg):
        """
        A tombstone guard on the reconcile query reduces its row count below the total
        memory count. This test detects the guard by comparing counts.
        """
        live_id = await insert_memory(pg, content="live memory", epoch=100)
        tombstone_id = await insert_memory(
            pg, content="tombstoned memory", epoch=99, is_deleted=True
        )

        total_count = await pg.fetchval("SELECT COUNT(*) FROM memories WHERE tenant_id = 'tim'")
        reconcile_count = await pg.fetchval(
            "SELECT COUNT(*) FROM memories WHERE tenant_id = 'tim'"
            # Production query — no tombstone guard
        )

        assert reconcile_count == total_count, (
            "Reconcile ID scan count must equal total memory count. "
            "A discrepancy indicates a tombstone guard has been added to the "
            "reconcile query, excluding some memories from the orphan scan."
        )
```

### Test Path 4 — Parametric Contamination Sweep

This test parametrizes across all non-reconcile queries that accept a memory ID set as output, asserting none of them can return a tombstoned ID. It catches the case where a new query is added that looks like BFS or seed selection but lacks the tombstone guard.

```python
# tests/recall/test_tombstone_contamination_parametric.py
import pytest

# Registry of all non-reconcile query functions that must exclude tombstones.
# Each entry: (test_id, sql_template, description)
# $1 = tenant_id  (for queries that filter by tenant)
RETRIEVAL_QUERIES = [
    pytest.param(
        # BFS depth-1 neighbor query
        """
        WITH depth1 AS (
            SELECT cr.to_memory_id AS memory_id
            FROM   co_retrievals cr
            JOIN   memories m ON cr.to_memory_id = m.id
            WHERE  cr.from_memory_id = ANY(:seed_ids)
              AND  cr.relationship_type = 'FRESH'
              AND  cr.tenant_id = $1
              AND  m.is_deleted = FALSE
        )
        SELECT memory_id AS id FROM depth1
        """,
        "bfs_depth1_neighbor",
        id="bfs_depth1_neighbor",
    ),
    pytest.param(
        # Seed selection query
        """
        SELECT id
        FROM   memories
        WHERE  tenant_id = $1
          AND  last_access_epoch IS NOT NULL
          AND  is_deleted = FALSE
        ORDER  BY last_access_epoch DESC
        LIMIT  5
        """,
        "seed_selection",
        id="seed_selection",
    ),
    pytest.param(
        # S6 hub degree query
        """
        SELECT from_memory_id AS id
        FROM (
            SELECT cr.from_memory_id, COUNT(*) AS degree
            FROM   co_retrievals cr
            JOIN   memories m ON cr.from_memory_id = m.id
            WHERE  cr.relationship_type = 'FRESH'
              AND  cr.tenant_id = $1
              AND  m.is_deleted = FALSE
            GROUP  BY cr.from_memory_id
        ) t
        WHERE degree > 0
        """,
        "s6_hub_degree_scan",
        id="s6_hub_degree_scan",
    ),
]


class TestTombstoneContaminationParametric:
    """
    Parametric sweep: every non-reconcile retrieval query must be free of tombstoned IDs.
    Add new queries to RETRIEVAL_QUERIES whenever a new retrieval path is introduced.
    """

    @pytest.mark.parametrize("sql_template,query_name", RETRIEVAL_QUERIES)
    async def test_no_tombstoned_id_in_retrieval_result(
        self, pg, sql_template: str, query_name: str
    ):
        """
        Retrieval query '{query_name}' must not return any tombstoned memory ID.
        Setup: insert one live memory, one tombstoned memory, one edge between them.
        Assertion: tombstoned ID absent from query output.
        """
        live_id = await insert_memory(pg, content=f"live for {query_name}", epoch=100)
        tombstone_id = await insert_memory(
            pg, content=f"tombstone for {query_name}", epoch=99, is_deleted=True
        )
        # Edge from live → tombstone (tests neighbor exclusion)
        await insert_edge(pg, from_id=live_id, to_id=tombstone_id)
        # Edge from tombstone → live (tests seed-as-source exclusion for hub scan)
        await insert_edge(pg, from_id=tombstone_id, to_id=live_id)

        # Substitute :seed_ids with the live_id for BFS queries; $1 = tenant
        final_sql = sql_template.replace("= ANY(:seed_ids)", f"= ANY(ARRAY['{live_id}'::uuid])")

        rows = await pg.fetch(final_sql, "tim")
        returned_ids = {row["id"] for row in rows}

        assert tombstone_id not in returned_ids, (
            f"Query '{query_name}' returned a tombstoned memory ID ({tombstone_id}). "
            f"The tombstone guard (AND is_deleted = FALSE or equivalent) is missing "
            f"from this query path. Add it to the WHERE clause before merging."
        )
```

## Qualifying Ideas

### sql-tombstone-guard-as-view
- (N=0.71, E=0.65, F=0.88)
- Define a PostgreSQL VIEW `live_memories AS SELECT * FROM memories WHERE is_deleted = FALSE`. All retrieval queries JOIN against `live_memories` instead of `memories`. Reconcile queries JOIN against `memories` directly. This makes the tombstone rule structural: it is impossible to write a BFS query that forgets the guard if the query uses `live_memories`. The view cannot be dropped without breaking every retrieval test. The reconcile is the only query that explicitly uses the base table, making its intent (must include tombstones) visible in the query name.

### tombstone-guard-ast-grep-linter
- (N=0.74, E=0.58, F=0.72)
- Use `ast-grep` (or a simpler regex lint rule in CI) to detect SQL string literals in Python source that reference the `memories` table in a FROM or JOIN clause without also containing `is_deleted` or `live_memories`. Flag any match that is not in a file path matching `*reconcile*` or `*gc*`. This catches the tombstone guard omission at code-review time, before tests run. The linter runs in under 1 second. False positives (queries that legitimately omit the guard) are whitelisted by filename pattern.

### separate-deleted-table-partition
- (N=0.68, E=0.52, F=0.61)
- Move tombstoned rows to a `memories_deleted` table instead of keeping `is_deleted = TRUE` on the live table. All retrieval queries SELECT FROM `memories` (which contains only live rows by schema invariant). Reconcile queries SELECT FROM `memories UNION ALL memories_deleted`. This eliminates the possibility of forgetting the tombstone guard on retrieval queries — there are no tombstoned rows to leak. The cost is an additional table and a more complex reconcile. At Tim's scale (< 2,000 live memories) the operational overhead is low.

## Recommendations

1. Implement the four test files above (`test_tombstone_bfs_neighbor.py`, `test_tombstone_seed_selection.py`, `test_tombstone_reconcile_inclusion.py`, `test_tombstone_contamination_parametric.py`) before the three-store architecture migration goes live.

2. Add `pytest -m tombstone` as a required CI step that blocks deployment on failure. The suite runs in under 5 seconds.

3. When the BFS depth-2 CTE is implemented, duplicate the depth-1 neighbor fixture in `test_tombstone_bfs_neighbor.py` with a seed → intermediate → tombstoned-target edge chain and assert the tombstoned target is absent from the final result.

4. Register every new retrieval query in `RETRIEVAL_QUERIES` at the time of implementation, not as a follow-up task. The parametric test runs it automatically on every merge.

5. The `live_memories` view approach (qualifying idea above) is the most robust long-term solution: it makes the tombstone exclusion structural at the schema level rather than relying on developers to remember a WHERE clause. Implement it when the PostgreSQL schema is first deployed; retrofitting it onto an existing schema requires a view-creation migration only (no data movement).
