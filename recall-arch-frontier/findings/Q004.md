# Finding: Q004 — DNS Resolver TTL Mechanisms Applied to Recall's Latency and Cold-Start Problems

**Question**: What mechanism does a DNS resolver use for TTL-based cache expiry and prefetch-on-near-expiry, and how would it apply to Recall's retrieval latency problem?
**Question Type**: ADJACENT
**Verdict**: PROMISING
**Severity**: High
**Source field**: DNS resolver caching (BIND9, Unbound, RFC 8767, RFC 2308)

---

## Evidence

### Mechanism 1: Prefetch-on-Near-Expiry (BIND9 `prefetch`)

**How it works in BIND9** (documented in ISC KB, widely deployed since BIND 9.10.0):

BIND9's `prefetch` option takes two integer parameters: `expiry-ttl-seconds` (trigger) and `threshold-ttl-seconds` (eligibility).

Decision logic at query time:
1. A client query arrives for a cached record.
2. BIND checks the record's remaining TTL.
3. If remaining_TTL <= trigger (default: **2 seconds**), AND the record's original TTL was > eligibility (default: **9 seconds**), then:
   - Return the cached record immediately to the client (no latency penalty).
   - Dispatch a background fetch to the authoritative server in parallel.
4. The background fetch result updates the cache before the current copy expires.
5. If remaining_TTL > trigger: return cached record, no background fetch.

Constraint: `threshold-ttl-seconds` must be at least 6 seconds greater than `expiry-ttl-seconds`. This prevents prefetching very short-lived records that will expire before the background fetch could complete anyway.

The trigger is absolute (seconds remaining), not percentage-based. Records with original TTL < eligibility are excluded — there is no point prefetching a 5-second TTL record because the fetch latency is comparable to the TTL itself.

**Input**: query for a cached record
**Output**: immediate response + optional background fetch dispatch
**State maintained**: per-record original TTL, per-record remaining TTL, background fetch in-flight flag
**Update rule**: if remaining_TTL <= trigger AND original_TTL > eligibility → fire background fetch; replace cache entry on fetch completion
**Failure modes**: background fetch fails → record expires and next query suffers full resolution latency; trigger value too high → excessive background fetches; trigger value too low → records expire before fetch completes

### Mechanism 2: Stale-While-Revalidate (RFC 8767, Unbound `serve-expired`, BIND9 `stale-cache-enable`)

**Two distinct algorithms exist — they solve different problems:**

**Unbound original serve-expired** (since Unbound 1.6.0, `serve-expired: yes`, `serve-expired-client-timeout: 0`):
1. Client query arrives.
2. Before attempting network resolution, check cache for any record matching the query — including expired records.
3. If an expired record is found: return it **immediately** to the client (cache-hit speed).
4. Continue resolution in background regardless.
5. Background resolution result updates the cache.
6. The expired record is served with `serve-expired-reply-ttl` (recommended: 30 seconds, previously 0) so the client knows to re-query soon.
7. `serve-expired-ttl` caps how stale a record may be served (RFC 8767 recommends 86400–259200 seconds, i.e., 1–3 days).

**RFC 8767 behavior** (Unbound `serve-expired-client-timeout > 0`, BIND9 `stale-answer-client-timeout`):
1. Client query arrives.
2. Attempt normal resolution (network call).
3. If resolution completes within `serve-expired-client-timeout` (e.g., 1800ms): return fresh answer.
4. If timeout expires before resolution completes: pause, check cache for expired record.
5. If stale record exists: return it (latency bounded by client-timeout, not network RTT).
6. Resume resolution in background; update cache on success.
7. This is "stale as fallback" rather than "stale as first response."

**BIND9 serve-stale decision tree** (`stale-cache-enable: yes`, `stale-refresh-time: 30s` default):
- Cache hit, TTL > 0 → serve immediately (normal path)
- Cache hit, TTL = 0, within stale-refresh-time window → serve stale immediately (no wait)
- Cache hit, TTL = 0, stale-refresh-time expired → attempt refresh; if refresh fails → serve stale
- No cache entry → full resolution; if resolution fails → SERVFAIL
- `stale-answer-ttl` (30 seconds, RFC 8767 recommended): the TTL returned with stale answers to signal short re-query window

**Input**: client DNS query
**Output**: cached record (possibly expired) + background resolution
**State maintained**: cache entry including post-expiry flag, stale-refresh-time window per record, in-flight fetch flag
**Update rule**: serve stale immediately or after timeout; update on background fetch success
**Failure modes**: staleness window too long → clients receive dangerously stale data; staleness window too short → no benefit over standard expiry; background fetch storms if many records expire simultaneously

### Mechanism 3: Negative Caching (RFC 2308)

**How it works** (mandatory since BIND 8.2, Unbound enforces by default):

When a resolver receives NXDOMAIN (name does not exist) or NODATA (name exists, type does not exist), it caches the *absence* of data.

TTL algorithm for negative cache entries:
- Negative TTL = min(SOA MINIMUM field, SOA record TTL)
- RFC 2308 recommends 1–3 hours; values over 24 hours are explicitly discouraged
- SERVFAIL responses may be cached but MUST NOT exceed 5 minutes
- The negative TTL decrements in real time identically to a positive TTL
- NXDOMAIN is cached on (QNAME, QCLASS) — domain does not exist at all
- NODATA is cached on (QNAME, QTYPE, QCLASS) — domain exists but lacks that record type; these are distinct cache keys

**Input**: NXDOMAIN or NODATA DNS response
**Output**: cached negative entry with computed TTL
**State maintained**: negative cache entry with (QNAME, QTYPE, QCLASS) key, TTL
**Update rule**: TTL decrements; on expiry, negative cache entry is discarded, next query retries
**Failure modes**: overly long negative TTL traps clients even after the name is provisioned; SOA MINIMUM misconfiguration cascades to all resolvers

### Mechanism 4: TTL Clamping (Resolver-side normalization)

**How it works** (Unbound `cache-max-ttl`, `cache-min-ttl`; BIND9 `max-cache-ttl`, `min-cache-ttl`):

Regardless of what the authoritative server advertises, resolvers clamp TTL to a locally configured range:
- `cache-max-ttl` (Unbound default: 86400 seconds) — if authoritative server says TTL = 604800 (1 week), resolver stores it with TTL = 86400
- `cache-min-ttl` (Unbound default: 0) — if authoritative server says TTL = 0, resolver may cache it for a minimum duration
- `cache-max-negative-ttl` (Unbound default: 3600) — caps negative response TTL

**Interaction with prefetch**: TTL clamping happens before prefetch threshold evaluation. If a record arrives with TTL = 604800 but cache-max-ttl = 86400, the prefetch trigger (2 seconds) is evaluated against the *clamped* remaining TTL, not the advertised value. This prevents prefetch storms on records with very long original TTLs.

### Track Record in Source Field

All four mechanisms are production-deployed and battle-tested at internet scale:

- **BIND9 prefetch**: Available since BIND 9.10.0 (2014), deployed on millions of resolvers worldwide. ISC documents it as a standard performance feature. The 2-second trigger and 9-second eligibility defaults emerged from operational experience across diverse network conditions.
- **Unbound serve-expired**: Available since Unbound 1.6.0 (2017), RFC 8767 compliance added in 1.10.0–1.11.0 (2020). NLnet Labs documents serve-expired as the primary mechanism for maintaining cache-hit ratio under authoritative server failures.
- **RFC 2308 negative caching**: Mandatory since 1998 (BIND 8.2+). Eliminates the NXDOMAIN thundering herd — without negative caching, every client would hammer authoritative servers with repeated failing queries.
- **TTL clamping**: Universal across all production resolvers. Operational necessity — authoritative servers sometimes advertise TTLs of 0 (effectively disabling caching) or weeks (creating stale data problems).

Quantified results: DNS prefetch eliminates resolution latency spikes at TTL boundaries. Without prefetch, a popular record expiring causes a latency cliff — the first query after expiry pays full resolution latency (50–500ms on internet resolvers). With prefetch, no client ever sees that latency cliff for frequently-queried records.

---

### Why AI Memory Systems Don't Do This Yet

The mapping is non-obvious because DNS TTLs are explicit (set by the zone operator), whereas memory "freshness" in AI systems is implicit (no expiry field, no TTL metadata). The DNS prefetch mechanism requires a TTL to exist — AI memory systems currently have no equivalent. The idea of assigning decay-based TTLs to memories and using them to drive proactive background refresh is not present in any production AI memory system (MemGPT, mem0, Zep, LangChain memory). Those systems are purely reactive — retrieval happens on demand, with no prefetch or freshness tracking.

The second barrier is that DNS resolvers know *which records to prefetch* because a client query triggered them. AI memory systems face the harder problem of predicting which memories will be queried next *without* an incoming query. The DNS prefetch mechanism sidesteps this by being query-triggered, not predictive.

## Ideas Extracted

### Idea: query-triggered-background-refresh

- **Novelty**: 0.75 — no production AI memory system implements query-triggered background re-embedding or freshness refresh. Redis TTL expiry exists but there is no mechanism to proactively refresh a memory before it expires from hot cache.
- **Evidence**: 0.90 — BIND9 prefetch is deployed at internet scale since 2014, with documented latency elimination at TTL boundaries. The mechanism is proven at billions of queries/day.
- **Feasibility**: 0.80 — implementable in 1–2 weeks against Recall's existing stack. When a memory is retrieved from Redis hot cache and its TTL is below a threshold (e.g., 20% of configured TTL remaining), dispatch a background task to re-embed + re-score the memory and refresh its cache entry. No new infrastructure required — Redis TTL is already tracked, FastAPI background tasks exist.
- **Description**: When `recall-retrieve.js` pulls a memory from Redis and Redis reports the key's remaining TTL is below a threshold (configurable, e.g., 60 seconds on a 300-second cache TTL), Recall's API dispatches an async background task that re-fetches the memory from PostgreSQL, re-scores its importance, and writes it back to Redis with a fresh TTL. The client receives the cached result immediately (no latency penalty). This eliminates cache-miss spikes at session boundaries where a memory's Redis entry expires mid-session.

### Idea: stale-serve-with-bounded-latency

- **Novelty**: 0.80 — no AI memory system implements stale-serve with a client timeout fallback. Systems either return from cache or block on retrieval; none return stale results while revalidating.
- **Evidence**: 0.85 — RFC 8767 and Unbound serve-expired are production-validated. The ISC and NLnet Labs both implemented this independently and converged on similar designs, which is strong evidence the pattern is correct.
- **Feasibility**: 0.70 — requires a two-phase retrieve path in Recall's FastAPI layer. Phase 1: within a latency budget (e.g., 200ms), attempt fresh retrieval from Qdrant. Phase 2: if 200ms elapses before Qdrant responds (e.g., Ollama RTT spike), return the last known result for this session from Redis regardless of its TTL, with a metadata flag indicating staleness. Background task continues and updates Redis when fresh result arrives.
- **Description**: Recall's retrieve endpoint sets an internal timeout of 200ms. If the Ollama embedding + Qdrant ANN search complete within 200ms, fresh results are returned normally. If the 200ms budget is exhausted (common when Ollama server at 192.168.50.62 is under load), Recall immediately returns whatever is in Redis for this session key — even if stale — with a `stale: true` metadata flag. The hook injects this context anyway, but the model sees the staleness signal. A background coroutine continues the full retrieval and writes the result to Redis. The next prompt sees fresh results. This converts a p99 latency spike from 800ms (Ollama timeout) to a guaranteed 200ms with a one-prompt staleness cost.

### Idea: negative-memory-cache

- **Novelty**: 0.85 — no AI memory system caches the *absence* of relevant memories. Every query goes through the full retrieval pipeline even when the system has recently determined there is nothing relevant on this topic.
- **Evidence**: 0.80 — RFC 2308 negative caching has been mandatory since 1998 and eliminates a well-understood thundering herd pattern. The analogy is precise: repeated queries for a topic with no stored memories are equivalent to repeated NXDOMAIN queries.
- **Feasibility**: 0.85 — Redis already exists in the stack. When Qdrant returns zero results above the similarity threshold for a query, store a negative cache entry in Redis: key = `neg:{query_embedding_hash}`, TTL = 300 seconds. On future queries within embedding cosine distance < 0.1 of a cached negative key, return empty immediately without hitting Qdrant or Ollama. Cache eviction handles the case where a new memory is stored on the topic — the negative entry expires or is explicitly invalidated on store operations matching the same domain/tags.
- **Description**: After a retrieve operation returns no results above threshold, Recall writes a negative cache entry to Redis storing the query embedding (or a locality-sensitive hash of it) with a 5-minute TTL. Subsequent retrieve calls check for a matching negative cache entry before invoking Ollama for embedding. If a negative cache hit occurs, the retrieve returns empty immediately — saving the full Ollama RTT (50–150ms) + Qdrant query. The negative entry is invalidated on any store operation to the same domain, mirroring how DNS negative entries are invalidated when the zone is updated.

### Idea: ttl-clamping-for-importance-normalization

- **Novelty**: 0.70 — importance score normalization at write time exists in some systems, but using TTL-clamping semantics (ignoring what the writer claims, enforce local policy) to bound cache residence time regardless of importance score is not standard.
- **Evidence**: 0.70 — TTL clamping is universal in DNS resolvers precisely because authoritative servers cannot be trusted to set sensible TTLs. The parallel: memory importance scores set at write time are equally untrustworthy over long timescales — a memory marked importance=1.0 six months ago may be irrelevant now.
- **Feasibility**: 0.75 — implementable against existing Redis TTL and PostgreSQL schema. Enforce a cache-max-ttl per domain (e.g., session memories: 1 hour max, project memories: 24 hours max, user-profile memories: 7 days max) regardless of the importance score. High importance only prevents *eviction from the cold store*, not from setting unreasonably long cache residence.
- **Description**: Recall's Redis cache TTL is currently set based on importance score (higher importance = longer TTL). This is analogous to trusting the authoritative server's TTL blindly. Instead, enforce a maximum TTL per memory domain: session-scoped memories cap at 1 hour in Redis regardless of importance, project-scoped cap at 24 hours, global/profile memories cap at 7 days. Importance score only affects ranking within those bounds, not the cap. This prevents stale session memories from polluting the hot cache long after a project context has changed.
