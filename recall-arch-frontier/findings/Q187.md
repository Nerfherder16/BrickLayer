## Evidence

### Benchmark Setup

Python simulation using project parameters: N=22,423, K=200, cluster_size=112, k=5 retrievals/session, gamma=0.95, delta=0.05, bootstrap_weight=0.1, noise_floor=0.005 (1/K from Q183).

### Key Metrics

**Cluster structure:**
- Intra-cluster pairs per cluster: 6,216
- Total intra-cluster pairs (bootstrap edges): 1,243,200
- Total cross-cluster pairs: 250,141,053
- Ratio cross:intra = 201:1

**Per-session dynamics:**

| Pair type | E[pairs per session] | Total pairs | E[credits per specific pair per session] |
|---|---|---|---|
| Intra-cluster | 0.0495 | 1,243,200 | 3.98e-8 |
| Cross-cluster | 9.9505 | 250,141,053 | 3.98e-8 |

**Counterintuitive result**: Cross-cluster pairs are credited 201× more *in aggregate* per session (9.95 vs 0.05), but because there are 201× more of them, *each specific pair* receives the same expected credit per session as each specific intra-cluster pair (3.98e-8).

### Formation Probability Under Uniform Retrieval

Expected weight for any specific cross-cluster pair after 1000 sessions (no decay):
```
E[credits] = 1000 × (9.9505 / 250,141,053) = 3.98e-5
E[weight] = 0.05 × 3.98e-5 = 1.99e-6
```

**This is 2,500× below the noise floor of 0.005.** Under uniform retrieval, cross-cluster edges *never* form meaningfully. Expected sessions to first credit on a specific pair: 25,138,554 (~200 years at 7 sessions/week).

### Formation Under Topic-Concentrated Retrieval

For Tim's typical "focused session" where k=5 retrievals span 2 topic clusters (3+2 split):
- Cross pairs per session: 3×2 = 6
- Steady-state weight (at daily sessions): w_ss = delta × 6 / (1 - gamma) = 0.05 × 6 / 0.05 = 6.0
- Sessions to match bootstrap weight (0.1): 0.10 / (6 × 0.05) = 0 (immediately exceeded)

**Cross-cluster edges can reach the same weight magnitude as intra-cluster edges when sessions concentrate on cross-cluster topics.** A pair retrieved together 6 times per session settles at w=6.0 — 6× the organic intra-cluster steady state.

### The Bootstrap Asymmetry

Intra-cluster edges are bootstrapped at w=0.1 = 20× the noise floor. Cross-cluster edges start at w=0.

This asymmetry has a Bayesian interpretation:
- The bootstrap initialization encodes the K-means prior: memories in the same cluster are semantically similar and should be retrieved together
- Cross-cluster edges must earn their weight through actual retrieval co-occurrence — they carry no prior
- A cross-cluster edge at w=0.05 carries *more information* than an intra-cluster edge at w=0.05 decayed from 0.1, because the cross-cluster edge represents genuine usage, not just initialization

### Decay Schedule for Cross-Cluster Edges

Cross-cluster edges that form via concentrated retrieval (e.g., starting at w=0.5) follow the *identical* decay schedule as intra-cluster edges:

| Week | w(t) | Status |
|------|------|--------|
| 0 | 0.500 | alive |
| 10 | 0.299 | alive |
| 35 | 0.083 | alive |
| 58 | 0.026 | alive (above noise floor 0.005) |
| 90 | 0.005 | at noise floor |

Same gamma=0.95, same noise floor, same deletion threshold. No architectural distinction required.

### Does Cross-Cluster Need a Separate Decay Policy?

**No.** Three findings converge on this:

1. Under uniform retrieval, cross-cluster edges never reach the noise floor anyway — they're architecturally invisible
2. Under concentrated retrieval, cross-cluster edges are governed by the same Q186 stability analysis (spectral radius = gamma = 0.95)
3. Once formed, cross-cluster edges decay on the same gamma=0.95 schedule and are correctly gated by the Q183 noise floor threshold (0.005)

The Q178/Q183 framework handles all edge types uniformly. Adding a cross-cluster exception would add complexity with no benefit.

## Ideas Extracted

### Idea: bootstrap-initialization-as-bayesian-graph-prior
- **Novelty**: 0.80 — The K-means bootstrap initializing intra-cluster edges at w=0.1 has never been explicitly framed as a Bayesian prior in the reviewed literature. MemGPT, mem0, and Zep have no graph bootstrapping mechanism. Formalizing this as "bootstrap = prior encoding cluster structure" opens a design space: should the prior weight be proportional to intra-cluster cosine similarity rather than a flat 0.1?
- **Evidence**: 0.85 — The analysis shows intra-cluster edges have a 20× prior advantage (w=0.1 vs w=0 for cross-cluster). This is the intended design effect of Q170's bootstrap job. The Bayesian framing is mathematically valid.
- **Feasibility**: 0.90 — The design implication is that bootstrap weight should be calibrated to the mean intra-cluster cosine similarity (Q183-F1 suggested this). For well-separated K=200 clusters in 384-dim semantic space, typical intra-cluster cosine mean is 0.3-0.7. Setting bootstrap_weight = mean_intra_cosine would make the prior scale with actual cluster cohesion rather than an arbitrary constant.
- **Description**: Reframe Q170's bootstrap weight (currently 0.1) as a semantic prior proportional to the cluster's mean intra-cluster cosine similarity. For tight clusters (mean_cos=0.6), bootstrap_weight=0.6. For loose clusters (mean_cos=0.2), bootstrap_weight=0.2. This makes the graph initialization self-calibrating: tightly-clustered semantic groups start with stronger co_retrieval priors, loosely-clustered groups start with weaker ones. One change to the bootstrap job: weight = cluster_mean_cosine for each cluster rather than a global constant.

### Idea: cross-cluster-edge-as-cross-domain-semantic-bridge-signal
- **Novelty**: 0.85 — Cross-cluster CO_RETRIEVED edges that clear the noise floor are rarer and more information-dense than intra-cluster edges. No reviewed system distinguishes the semantic value of intra-cluster vs cross-cluster co_retrieval. A cross-cluster edge at w=0.05 required far more retrieval co-occurrence to form than an intra-cluster edge at w=0.05 (which started at 0.1 and decayed). The cross-cluster edge therefore carries a stronger usage signal.
- **Evidence**: 0.80 — Analysis confirms cross-cluster edges under uniform retrieval have expected weight 1.99e-6 after 1000 sessions — orders of magnitude below noise. Any cross-cluster edge above noise floor (w > 0.005) necessarily formed through concentrated cross-domain retrieval, making it a genuine semantic bridge.
- **Feasibility**: 0.80 — Add a property `is_cross_cluster: bool` to CO_RETRIEVED relationships in the bootstrap job (intra-cluster edges get False, organic cross-cluster edges get True via comparison of cluster IDs in the flush worker). In the co_retrieval_gravity Cypher query, apply a multiplier: `gravity += weight * (CASE WHEN r.is_cross_cluster THEN 1.5 ELSE 1.0 END)`. Cross-cluster edges become higher-signal retrieval hints. No schema migration required beyond a boolean property.
- **Description**: Mark CO_RETRIEVED edges as `is_cross_cluster: true` when the two memories belong to different K-means clusters (checked in the flush worker via a lookup table: memory_id → cluster_id). In the co_retrieval_gravity computation, weight cross-cluster edges 1.5× because they represent genuine cross-domain semantic bridges formed through actual usage rather than initialization. This implements a principled distinction: bootstrap edges encode cluster prior, cross-cluster organic edges encode usage-derived cross-domain similarity.

### Idea: no-separate-cross-cluster-decay-policy-unified-physics
- **Novelty**: 0.65 — This is primarily a confirmatory finding. The Q178/Q183 framework applies uniformly. The novelty is documenting the formal proof that no exception is needed, which simplifies the architecture specification.
- **Evidence**: 0.95 — Direct analysis of decay schedule (identical gamma=0.95 governs all edge types) and noise floor (Q183's 0.005 threshold gates both intra and cross-cluster edges equally).
- **Feasibility**: 0.99 — No action needed. The existing architecture is correct as designed.
- **Description**: The CO_RETRIEVED decay architecture (gamma=0.95 weekly decay, deletion at w<0.005) handles all edge types uniformly with no special cases for cross-cluster edges. This is a formal confirmation that Q178's spec is complete: the physics generalize correctly. Document this explicitly in the bootstrap job spec (Q170) and decay worker spec (Q178) to prevent future implementers from adding unnecessary cross-cluster exceptions.

## Analytical Summary

Cross-cluster CO_RETRIEVED edges are governed by a bifurcated formation model:

**Regime 1 (uniform retrieval)**: Cross-cluster edges are architecturally invisible. Expected weight after 1000 sessions = 1.99e-6 (2,500× below noise floor). The 201:1 ratio of cross-pair to intra-pair count perfectly cancels the 201:1 ratio of cross-pair organic credit per session, leaving per-specific-pair formation rates identical. But since cross-cluster edges start at w=0 and intra-cluster edges start at w=0.1, intra-cluster edges will always dominate in this regime.

**Regime 2 (concentrated retrieval)**: Cross-cluster edges form at the same rate as intra-cluster edges when Tim's sessions span topic clusters. They can reach w_ss = 6.0 (concentrated daily sessions, 2-cluster overlap). These edges represent genuine cross-domain semantic relationships and are actually more information-dense than equivalent-weight intra-cluster edges (higher evidence content per unit weight).

The bootstrap asymmetry (intra-cluster starts at 0.1 = 20× noise floor; cross-cluster starts at 0) is the architectural expression of the K-means semantic prior. Cross-cluster edges must prove themselves through usage. This is a design virtue, not a limitation.

**No separate decay policy is needed.** The Q178/Q183 framework is complete and correct for both edge types. The most valuable architectural refinement available is calibrating the bootstrap prior weight to cluster cohesion (mean intra-cluster cosine similarity) rather than a flat constant.
