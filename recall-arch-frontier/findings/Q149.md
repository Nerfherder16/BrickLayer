**Key architectural annotations from code comments:**

- Neo4j failure at step 2 triggers a **compensating delete** on Qdrant (`await qdrant.delete(memory.id)`, line 286), then re-raises. This is the only compensating action in the write path.
- Redis failure at step 3 is logged as a warning and **does not fail the store** (code comment: "Redis failure must not fail the store", line 289).
- PostgreSQL is **audit-only** — write failure is caught, logged as warning, and swallowed. PG holds no primary data about memory existence.
- Background tasks (auto_link, fact_extractor, state_fact_extractor, contradiction_detector, causal_extractor, mosaic linkers) are queued after all synchronous writes complete. They operate on already-committed Qdrant+Neo4j data.

**Comparison to delete path ordering (from Q132):**

- Store: also Qdrant-first (`qdrant.store()` line 278, then `neo4j.create_memory_node()` line 283)
- Both paths agree on Qdrant-before-Neo4j ordering. The delete path has no compensating action for a Neo4j failure mid-delete.

## Finding

**[OBJECTIVE]** The POST /memory/store write path in `memory.py` follows a fixed ordering: Qdrant (step 1), Neo4j (step 2), Redis working-memory (step 3, optional), Redis session counter (step 4, optional), PostgreSQL audit (step 5, fire-and-forget). PostgreSQL holds no primary memory data; it is audit-log only.

**[DATA]** The single compensating action in the entire write path is at line 286: if Neo4j write fails, a `qdrant.delete(memory.id)` is awaited synchronously before the exception propagates. This is a correct and deliberate design (confirmed by the inline comment "compensating delete on failure"). The delete path (Q132) also writes Qdrant-first, confirming the system's architectural preference is Qdrant as the primary store with Neo4j as secondary.

**[FINDING]** The system produces **no stable Qdrant orphans under single-fault conditions**. Qdrant failure → clean abort (nothing written). Neo4j failure → compensating Qdrant delete fires. Redis and PG failures are swallowed and do not produce retrieval-visible anomalies. However, a **double-fault scenario** (Neo4j fails AND the compensating `qdrant.delete()` also fails) produces the most dangerous orphan class: a Qdrant vector point with no corresponding Neo4j node. This orphan is injected into ANN search results (will appear in retrieved context), has no graph traversal path (RELATES_TO/SUPERSEDES edges are unreachable), and is not auto-recoverable. There is no outbox, no saga log, and no periodic reconciler to detect this state.

**[STAT:n]** 5-store write sequence with 1 compensating action covering 1 of 4 possible inter-store failure points.

**[STAT:effect_size]** Under normal single-fault conditions, orphan probability ≈ 0. Under double-fault (Neo4j fail + compensating-delete fail, estimated P ≈ 0.001–0.01 per store call depending on infrastructure stability), a permanently undetectable Qdrant orphan is created that injects stale/ghost context into every future retrieval.

## Qualifying Ideas

- **reconciler-background-task**: (N=0.68, E=0.82, F=0.90) — A periodic background task (or FastAPI lifespan task) that queries all Qdrant point IDs for a time window and cross-checks against Neo4j node existence. Any Qdrant ID without a matching Neo4j node is flagged as an orphan and either deleted or re-materialized. Feasible using existing `get_qdrant_store()` and `get_neo4j_store()` singletons. Eliminates the double-fault orphan class without requiring outbox infrastructure.

- **write-ahead-log-in-redis**: (N=0.75, E=0.65, F=0.72) — Before writing to Qdrant, write the memory ID + content_hash + timestamp to a Redis PENDING set. Remove from PENDING only after both Qdrant and Neo4j succeed. A background sweeper attempts re-finalization of stale PENDING entries. Converts the current at-most-once write into at-least-once with idempotent compensating logic. Requires Redis to be available at store time, which introduces a new failure mode (Redis unavailable blocks entire store), but Redis is already queried at line 182 before the write sequence.

- **neo4j-last ordering**: (N=0.66, E=0.78, F=0.95) — Invert the write order to Qdrant-last instead of Neo4j-last. If Neo4j write succeeds but Qdrant fails, the orphan is a Neo4j node without a Qdrant vector — this is detectable (no ANN results, query returns nothing) and the failure mode is silent omission rather than ghost injection. The current Qdrant-first ordering creates the more dangerous orphan class (ghost in context) in the double-fault scenario, while Neo4j-first ordering would create a silent-miss orphan class instead. Low implementation cost: swap lines 278 and 283, update compensating action to delete the Neo4j node on Qdrant failure.
