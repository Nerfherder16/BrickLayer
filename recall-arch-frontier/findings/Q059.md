|---------------|-------------------|

1. PostgreSQL `memories` table: two new integer columns
2. PostgreSQL new table: `uuid_scope_index` for O(N_deleted) Op4 lookups

## Dependency Analysis

Before ordering the migration steps, identify which application behaviors depend on which fields being present.

### Dependency D1: `distinct_sessions_accessed` is independent

The LLPC admission gate (`distinct_sessions_accessed >= 3`) is a read-only query on this field. If the field does not exist, the query fails with a column-not-found error. If the field exists with DEFAULT 0, the gate never admits any memory (0 < 3), which is conservative and correct — no memory gets promoted prematurely. No other field or table depends on `distinct_sessions_accessed` existing first.

**App code that writes to this field**: the session-end Op2 (working-set counter update), which already fires. Op2 gains one additional bulk UPDATE: `UPDATE memories SET distinct_sessions_accessed = distinct_sessions_accessed + 1 WHERE id = ANY(:session_accessed_ids_set_once_per_session)`.

**Deployment order for D1**: schema migration (ALTER TABLE) must precede app code that reads or writes the field. App code that only reads and the field is absent will error. Safe window is zero — the column must exist before any app code referencing it is deployed.

### Dependency D2: `last_access_epoch` depends on the global epoch counter existing in Redis

The global epoch clock (`recall:global_epoch` in Redis) is a prerequisite for `last_access_epoch` to carry meaningful values. The epoch field in PostgreSQL stores the value of `recall:global_epoch` at last access. If the Redis counter does not exist yet, `INCR recall:global_epoch` initializes it to 1 automatically (Redis INCR on a missing key returns 1). So there is no explicit prerequisite to create the Redis key — the first session-end hook will create it.

However, the PostgreSQL column `last_access_epoch` must exist before the app code writes `current_epoch` to it on memory access. The LLPC routing query that uses `(:current_epoch - last_access_epoch) <= tau` must not fire until both the column exists AND the Redis epoch counter is initialized (otherwise `current_epoch` is undefined).

**Dependency order**: `last_access_epoch` column in PostgreSQL → app code reads `recall:global_epoch` from Redis at session end → app code writes epoch to `last_access_epoch` on retrieval.

**Transition risk**: existing rows will have `last_access_epoch = 0` and `recall:global_epoch` will start at 1 after first session end. For existing memories, `current_epoch - 0 = current_epoch`. If tau = 10 and current_epoch reaches 11, all existing memories with `last_access_epoch = 0` (never updated) will appear as stale and be excluded from LLPC. This is correct behavior — a memory never accessed after the epoch migration is correctly treated as stale. But the window between migration and first access of existing memories means any important LLPC-admitted memories will temporarily lose LLPC status until they are accessed once and their `last_access_epoch` is updated. This is a one-time impact at migration time, not an ongoing failure mode.

**Backfill option**: to avoid the LLPC eviction window, backfill `last_access_epoch = 0` for all existing rows (already the DEFAULT), then set `recall:global_epoch = 0` so that `current_epoch - 0 = 0`, which is always within tau. This means the epoch starts at 0, and the first session end sets it to 1. Existing memories start at `last_access_epoch = 0`, so `current_epoch - 0 = 1 <= tau` — no false eviction. This is the correct initial state.

### Dependency D3: `uuid_scope_index` must be backfilled before Op4 relies on it

Op4 (scope registry cleanup) uses the reverse index to find which scope registry entries to delete. If the reverse index table is empty (newly created), Op4 falls back to the O(N_registry) HSCAN approach described in Q051 — which is still correct, just slower. The reverse index is a performance optimization, not a correctness requirement.

**Deployment order for D3**: The `uuid_scope_index` table can be created at any time. Backfilling it with existing uuid→scope_key pairs should happen before Op4 is deployed to rely on it. However, because Op4 has a safe fallback (HSCAN when the reverse index misses), backfill can be done lazily — new memory stores populate the reverse index going forward, and Op4 falls back to HSCAN for any UUID not yet in the index. The lazy approach is safe.

**Correctness during partial backfill**: Op4 looks up each deleted UUID in `uuid_scope_index`. If the UUID is not there (not yet backfilled), Op4 skips it (no scope cleanup for that memory). This is not a correctness error — it is the same behavior as the pre-Op4 system (no scope cleanup at all). The tombstone schema change from Q049 handles the more dangerous dangling pointer case. The reverse index is a latency optimization for Op4.

### Dependency D4: tombstone schema depends on the tombstone state machine code

The Redis tombstone encoding (structured colon-delimited string) is written by the session-scope-window-drop code (Op1 modified for tombstone behavior). Existing LIVE entries in the Redis scope registry remain unchanged until their memory is deleted. When Op1 runs in tombstone mode, it writes tombstoned entries with the new format. Old entries (LIVE, pre-migration) remain as `{uuid}:{hlc}` format.

**Read path during transition**: the Phase 1 scope discriminant (G1 check in Q041) reads the registry value. During the transition period, some values will be `{uuid}:{hlc}` (LIVE, old format) and some will be `TOMBSTONED:...` (new format). The application must parse both formats. The detection is straightforward: if the value starts with `TOMBSTONED:`, it is a tombstone; otherwise it is a LIVE entry.

**Risk**: if new app code is deployed that writes tombstones but old app code is still running (rolling deployment), reads by old app code of tombstoned entries will fail to parse — old code expects `{uuid}:{hlc}` and will receive `TOMBSTONED:0.35:...`. Old code parsing `TOMBSTONED:0.35:sha256:...` as a UUID:HLC pair will get a garbled UUID. This is a genuine incorrect behavior window.

**Mitigation**: make the LIVE format explicitly tagged too: `LIVE:{uuid}:{hlc}`. Both old and new formats then have a prefix sentinel. Old app code, if it encounters `LIVE:{uuid}:{hlc}`, can parse correctly by stripping the prefix. However, this requires old code to be updated first to handle the new prefix — which means deploying the parser update before deploying the tombstone writer. This is a two-deploy sequence.

Alternatively: make tombstone detection unambiguous by choosing a UUID-incompatible prefix. UUIDs are hex characters and hyphens. The string `TOMBSTONED` does not match UUID format. Old code that splits on `:` and tries `UUID(parts[0])` will raise a ValueError on `TOMBSTONED`, which can be caught and treated as a LIVE miss (fallback to Phase 2 cosine). This graceful degradation avoids the garbled UUID problem without requiring the old code to know about tombstones.

**Recommended approach**: add a `try/except` around the UUID parse in the existing G1 registry reader. If the parse fails (because the prefix is `TOMBSTONED`), treat as a registry miss (LIVE behavior). This single defensive change makes the old code tombstone-safe without requiring it to handle tombstone semantics.

## Migration Sequence

The safe ordering of all migration steps, with backfill queries where needed.

### Step 1: Add `distinct_sessions_accessed` to PostgreSQL `memories`

**Classification**: Drop-in. No existing interface changes.

**SQL migration**:
```sql
ALTER TABLE memories
  ADD COLUMN distinct_sessions_accessed INTEGER NOT NULL DEFAULT 0;
```

**Backfill**: The DEFAULT 0 is the correct backfill value. No historical data can be recovered for this field — existing memories have an unknown number of distinct sessions they were accessed in. Setting 0 means all existing memories must be re-accessed 3 times (in 3 distinct sessions) before LLPC admission. This is conservative and correct: it prevents historical memories from being erroneously admitted based on phantom session counts.

**App code change required before or after**: the LLPC routing query must be updated to use `distinct_sessions_accessed >= 3` instead of `access_count >= 3`. This change is safe to deploy after the column exists. During the window between column creation and query update, no behavioral change occurs (the old query still works; the new column exists but is not yet used).

**Deploy order**: schema first, then app code. Schema with DEFAULT 0 is backward-compatible with old app code (old code ignores the new column entirely).

**Incorrect behavior window**: none. Old app code runs correctly against the new schema. New app code requires the column to exist before deployment.

### Step 2: Add `last_access_epoch` to PostgreSQL `memories`

**Classification**: Extension. New capability alongside existing `sessions_since_last_access`.

**SQL migration**:
```sql
ALTER TABLE memories
  ADD COLUMN last_access_epoch INTEGER NOT NULL DEFAULT 0;
```

**Backfill**: Set all existing rows to `last_access_epoch = 0`. This is already handled by `DEFAULT 0`. The Redis global epoch starts implicitly at 0 (first `INCR` returns 1). Initial state: all memories have `last_access_epoch = 0`, `recall:global_epoch` does not yet exist (returns 0 via Redis GET before first INCR).

**LLPC epoch query backfill consideration**: the epoch-based LLPC window check is `(current_epoch - last_access_epoch) <= tau`. At migration, `current_epoch = 0` (epoch not yet initialized) and `last_access_epoch = 0` for all rows: `0 - 0 = 0 <= tau` — all memories appear current. After the first session end, `current_epoch = 1` and accessed memories get `last_access_epoch = 1`; non-accessed memories remain at 0 (`1 - 0 = 1 <= tau` — still within window). After 11 sessions, non-accessed memories at `last_access_epoch = 0` have `11 - 0 = 11 > 10 = tau` — correctly excluded. No special backfill beyond DEFAULT 0 is needed.

**`sessions_since_last_access` column**: this column can be retained in parallel during the transition. The LLPC query is updated to use `last_access_epoch` instead, but the old column is not dropped immediately. Drop it in a follow-on migration after the epoch system is confirmed stable (recommend: 2-4 weeks after go-live).

**Deploy order**: schema first, then app code that writes `last_access_epoch` on retrieval, then app code that reads it in LLPC queries, then increment epoch in session-end hook. This order ensures the column exists before any write, and reads happen after writes have had a chance to populate the field.

**Incorrect behavior window**: none for correctness. Minor behavioral change: during the epoch rollout window (between column creation and first session end with epoch INCR), all memories appear current (epoch difference = 0). This is conservative — no memories are falsely excluded from LLPC. Once epoch increments start, the recency signal accumulates correctly.

### Step 3: Create `uuid_scope_index` table

**Classification**: Extension. New table, no existing tables or queries modified.

**SQL migration**:
```sql
CREATE TABLE uuid_scope_index (
    uuid UUID PRIMARY KEY,
    scope_key TEXT NOT NULL,
    created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW()
);

CREATE INDEX idx_uuid_scope_index_scope_key ON uuid_scope_index(scope_key);
```

**Backfill query**: populate from the existing scope registry in Redis. The Redis scope registry is the authoritative source for live UUID→scope_key mappings. Because Redis does not have a native SQL-queryable export, the backfill runs as a migration script:

```python
# Backfill script: read all scope registry entries from Redis and write to PostgreSQL
async def backfill_uuid_scope_index():
    cursor = 0
    records = []
    while True:
        cursor, entries = await redis.hscan("scope_registry", cursor=cursor, count=100)
        for scope_key, value in entries.items():
            # Parse LIVE value format: "{uuid}:{hlc}" or "LIVE:{uuid}:{hlc}"
            parts = value.split(":")
            if parts[0] == "TOMBSTONED":
                continue  # Tombstoned entries have no live UUID — skip
            uuid_str = parts[0] if parts[0] != "LIVE" else parts[1]
            try:
                uuid.UUID(uuid_str)  # Validate format
            except ValueError:
                continue
            records.append((uuid_str, scope_key))
        if cursor == 0:
            break  # HSCAN complete
    if records:
        await pg.executemany(
            "INSERT INTO uuid_scope_index (uuid, scope_key) VALUES ($1, $2) "
            "ON CONFLICT (uuid) DO NOTHING",
            records
        )
```

**Backfill correctness**: the backfill is safe to run with the system live. New memory stores that happen during backfill will write to the reverse index going forward (once the app code change is deployed). Memories stored before the app code change but after the backfill begins are the only gap — they are handled by Op4's HSCAN fallback until the next tombstone cleanup cycle.

**Deploy order**: create table, run backfill script, deploy app code that writes to the table on store, deploy Op4 with reverse index lookup.

**Incorrect behavior window**: none. During the backfill window, Op4 uses HSCAN fallback (slower but correct). After backfill, Op4 uses the reverse index (faster).

### Step 4: Deploy tombstone value format (Redis, application layer)

**Classification**: Extension. Redis data format change, handled at application layer with no Redis schema command.

**Migration action**: deploy updated G1 reader that handles both `TOMBSTONED:...` and `{uuid}:{hlc}` formats. Deploy this reader update BEFORE deploying the tombstone writer (Op1 modified for tombstone transitions). This ensures old LIVE values are still parsed correctly and new TOMBSTONED values are treated as registry misses by any app instances still running old write code.

**Deploy sequence within Step 4**:
1. Deploy reader update: `try/except ValueError` on UUID parse → treats unparseable value as registry miss.
2. Verify all app instances have the new reader (zero-downtime rolling deploy, wait for full rollout).
3. Deploy Op1 tombstone writer: Op1 now writes `TOMBSTONED:...` instead of `HDEL` on eviction.

**Backfill**: no backfill needed. Existing LIVE entries remain as `{uuid}:{hlc}`. They transition to tombstone format only when their underlying memory is next evicted by Op1. Until then they are correctly parsed as LIVE entries.

**Incorrect behavior window**: if Step 4.3 (tombstone writer) is deployed before Step 4.1 (reader update), old app instances will receive `TOMBSTONED:...` values from the registry and fail to parse them as UUIDs, producing a parse error. With the recommended defensive `try/except` in old code, this degrades to a registry miss (Phase 2 fallback) rather than an error. Without the defensive parse, old code raises an unhandled ValueError and the store operation fails.

**This is the only migration step with a genuine incorrect behavior window if the deploy order is violated.** The incorrect behavior is: tombstoned entries cause a garbled UUID to be used in the G1 lookup, resulting in a store operation that creates a duplicate LIVE entry for a tombstoned topic. This is a data integrity defect, not just a performance degradation. The mitigation (defensive `try/except`) reduces it to a conservative registry miss (not a data integrity issue).

### Step 5: Backfill `distinct_sessions_accessed` from access logs (optional)

**Classification**: Data quality improvement, not required for correctness.

If Recall stores access logs in PostgreSQL (an `access_log` table with `memory_id` and `session_id`), a historical backfill can compute `distinct_sessions_accessed` from past data:

```sql
UPDATE memories m
SET distinct_sessions_accessed = (
    SELECT COUNT(DISTINCT session_id)
    FROM access_log al
    WHERE al.memory_id = m.id
)
WHERE EXISTS (
    SELECT 1 FROM access_log al WHERE al.memory_id = m.id
);
```

This backfill is a large sequential scan of the access log. Run during a low-traffic window. It is safe to run with the system live — it only updates `distinct_sessions_accessed`, which is not yet used by the LLPC gate until Step 1 app code is deployed.

**If no access log table exists**: skip this step. DEFAULT 0 is safe. Memories accumulate `distinct_sessions_accessed` from the migration date forward.

## Risk Flags

### Risk 1: CRITICAL — Tombstone writer before reader update (Step 4 order violation)

**Severity**: Data integrity defect.
**Scenario**: Step 4.3 deployed before Step 4.1 across all instances in a rolling deploy. Instance A (new code) tombstones an entry. Instance B (old code) reads the tombstoned entry and fails to parse the UUID. Old code's error handling determines whether this is a crash, silent miss, or garbled UUID used in a store operation.

**If old code does NOT have defensive parse**: the ValueError propagates to the store handler, causing a 500 error for the incoming memory. The memory is lost.

**If old code HAS defensive parse** (recommended mitigation): ValueError is caught, registry lookup treated as a miss, store proceeds with create semantics (Phase 2 fallback), memory is stored as a new LIVE entry under a fresh UUID. The duplicate LIVE entry coexists with the tombstoned entry in the registry. Qdrant's cosine dedup layer (Phase 2) may catch this as a near-duplicate in the next dedup pass.

**Mitigation**: always deploy the reader update (Step 4.1) first, verify full rollout, then deploy the writer (Step 4.3). The defensive `try/except` is a belt-and-suspenders measure for incomplete rollouts.

### Risk 2: HIGH — `uuid_scope_index` backfill gap during live writes

**Severity**: Op4 performance degradation, not a correctness defect.
**Scenario**: backfill script runs while the system is live. New memories stored during backfill by old app code (before Step 3 app code is deployed) do not populate `uuid_scope_index`. When Op4 fires and deletes these memories, Op4 fails to find their reverse index entry and falls back to HSCAN. HSCAN is correct but O(N_registry).

**Window**: the gap window is bounded by the time between the backfill script completion and the app code deployment (Step 3 app code). If this window is less than 1 session, at most one session's worth of new memories are missing from the index.

**Mitigation**: run the backfill script immediately before deploying Step 3 app code (minimize the gap window). Accept that Op4 may use HSCAN for a small number of memories during the window.

### Risk 3: MEDIUM — `last_access_epoch` transition causes LLPC eviction of existing memories

**Severity**: Behavioral regression for existing LLPC members.
**Scenario**: after Step 2 migration and app code deployment, memories that were previously in LLPC (admitted via `distinct_sessions_accessed >= 3` or the old `access_count >= 3`) may not yet have `last_access_epoch` set (it defaults to 0). If `recall:global_epoch` has advanced past `tau` before these memories are first accessed under the new epoch system, they will appear stale and be excluded from LLPC.

**Window**: this window is `tau` sessions long (tau = 10 by default). At Tim's session density (~5 sessions/day), this is approximately 2 calendar days.

**Mitigation**: during the epoch migration, set `recall:global_epoch` to 0 (or do not initialize it until after first session end). All existing memories at `last_access_epoch = 0` have `current_epoch - 0 = current_epoch`. As long as `recall:global_epoch` is incremented no more than `tau` times before affected memories are accessed, no false eviction occurs. At 5 sessions/day and tau=10, Tim has a 2-day window to access previously-LLPC-admitted memories at least once before they are evicted. In practice, frequently-accessed memories will be hit within 1-2 sessions naturally.

**No mitigation needed if**: `last_access_epoch` and `sessions_since_last_access` are run in parallel during a transition period (both columns exist, old query uses `sessions_since_last_access`, new query uses `last_access_epoch`). Switch the LLPC query to use `last_access_epoch` only after `last_access_epoch` has been populated for all LLPC-admitted memories. Transition period: 2-4 weeks.

### Risk 4: LOW — Op2 double-increment during rolling deploy

**Severity**: Counter inflation, not a correctness defect.
**Scenario**: during rolling deployment of Step 1 app code (which adds the `distinct_sessions_accessed` increment to Op2), one app instance has new code and another has old code. The session-end hook fires on the new-code instance, increments `distinct_sessions_accessed` for this session. The old-code instance may also fire Op2 (working-set counter update) for the same session, but since old code does not know about `distinct_sessions_accessed`, it does not increment it. No double-increment is possible for `distinct_sessions_accessed` — only the new-code instance touches it.

**No mitigation needed**: the idempotency guard for Op2 (Q051) prevents double-increment of `sessions_since_last_access`. The same guard covers `distinct_sessions_accessed` because both are in the same Op2 function.

### Idea: backward-compatible-integer-column-migration-pattern

- **Novelty**: 0.65
- **Evidence**: 0.88
- **Feasibility**: 0.97
- **Description**: Adding integer columns with conservative defaults (0 for counters, 0 for epoch timestamps) to PostgreSQL tables in a running system is fully backward-compatible when the default value produces conservative behavior in the new application code. For Recall's LLPC gate (`distinct_sessions_accessed >= 3`), DEFAULT 0 means no existing memory is promoted prematurely — the new column is additive and correctness-preserving. The pattern generalizes: any counter column whose new-value logic is "accumulate forward from migration date" can be safely added with DEFAULT 0 and requires no backfill. Old app code ignores the column; new app code reads conservative (zero) values. The migration window is zero-risk.

### Idea: tombstone-sentinel-prefix-for-rolling-deploy-safety

- **Novelty**: 0.72
- **Evidence**: 0.75
- **Feasibility**: 0.90
- **Description**: When transitioning a Redis string value format (e.g., from `{uuid}:{hlc}` to `TOMBSTONED:{fields}`) in a rolling deployment, use a sentinel prefix that is unambiguously invalid under the old parsing scheme. For UUID-based values, `TOMBSTONED` is not a valid UUID hex string, so `UUID(parts[0])` raises ValueError. Wrapping the old-code parse in `try/except ValueError → treat as miss` makes any old-code instance tombstone-safe without requiring it to understand tombstone semantics. This allows the tombstone writer to be deployed independently of the reader update, reducing the coordination requirement from a synchronized cutover to a loose sequential deploy. The pattern applies to any Redis value format transition where the new format uses a type-incompatible prefix.

### Idea: two-phase-redis-value-migration-with-reader-first

- **Novelty**: 0.70
- **Evidence**: 0.82
- **Feasibility**: 0.88
- **Description**: For any Redis value format change that must be rolled out without downtime, deploy the reader update before the writer update. The reader update adds support for both old and new formats (backward-compatible parse). The writer update starts producing new-format values only after all instances have the new reader. This two-deploy sequence eliminates the window where new-format values are produced but not parseable by old-code readers. For Recall's tombstone migration, Phase 1 adds `try/except` on UUID parse (reader update), Phase 2 enables the tombstone writer in Op1. The pattern is directly analogous to the "expand-contract" database migration strategy (first expand the schema to support both old and new, then contract to the new-only path). Applied to Redis, it provides the same rolling deploy safety without Redis-level schema commands.

### Idea: lazy-reverse-index-backfill-with-hscan-fallback

- **Novelty**: 0.68
- **Evidence**: 0.80
- **Feasibility**: 0.93
- **Description**: When adding a reverse index table (uuid→scope_key) to optimize a previously O(N_registry) scan, implement the index consumer (Op4) with a two-path lookup: fast path (reverse index hit) and slow path (HSCAN fallback for misses). Deploy the consumer before completing the backfill. The slow path is both correct and already-tested code (pre-Op4 HSCAN behavior). The backfill then fills the index lazily — new stores populate it going forward, and a one-time backfill script fills historical data. This eliminates the need to pause the system during backfill: the fallback path handles any reverse-index misses during the backfill window. Once backfill is complete, the slow path is exercised only for the rare case of missing entries (e.g., entries created between backfill script start and new app code deploy).

### Idea: coordinated-epoch-bootstrap-for-zero-eviction-migration

- **Novelty**: 0.73
- **Evidence**: 0.78
- **Feasibility**: 0.85
- **Description**: When migrating from a per-memory counter (`sessions_since_last_access`) to a global epoch model (`last_access_epoch` + `recall:global_epoch`), initialize both the PostgreSQL column and the Redis epoch counter to 0. The recency check `current_epoch - last_access_epoch <= tau` evaluates to `0 - 0 = 0 <= tau` for all existing memories at migration time, meaning no existing memory is evicted during the transition window. The epoch clock then starts from 0 and accumulates forward: after tau sessions, memories not accessed since the migration appear stale and are correctly excluded. This bootstrap ensures the migration is loss-free: no memory that was previously LLPC-eligible loses its status due to the counter format change. The technique generalizes to any monotonic counter migration where the recency predicate uses a difference operator: setting both old and new counters to the same reference value (0) at migration time produces the conservative "everything is current" initial state.

## Key Finding

The seven fields introduced by Waves 5-6 decompose into three distinct migration risk classes:

**Class A (zero-risk, deploy anytime)**: `distinct_sessions_accessed` (DEFAULT 0, conservative admission gate) and `uuid_scope_index` table (performance optimization with fallback path). These can be deployed at any time without coordinating with app code rollouts.

**Class B (one-time transition risk, epoch model)**: `last_access_epoch` requires a coordinated bootstrap — both the PostgreSQL column and the Redis epoch counter must be initialized to 0 before the first epoch-aware session end fires. The transition window creates a 2-day period where previously-LLPC-admitted memories may appear stale if not accessed. Running `sessions_since_last_access` and `last_access_epoch` in parallel for 2-4 weeks eliminates this risk.

**Class C (rolling deploy order dependency, tombstone fields)**: the Redis tombstone value format is the only field group with a genuine incorrect behavior window. If the tombstone writer is deployed before the reader update, old-code instances produce data integrity defects (garbled UUID in store operations). The mitigation is a strict two-deploy sequence: reader update first, verified full rollout, then writer. The defensive `try/except ValueError` in the reader is the belt-and-suspenders measure that degrades garbled-UUID situations to a conservative registry miss rather than a store error.

The safe overall migration sequence is: Step 1 (distinct_sessions_accessed) → Step 3 (uuid_scope_index, with backfill) → Step 2 (last_access_epoch, parallel with old counter for 2-4 weeks) → Step 4.1 (tombstone reader update, verified full rollout) → Step 4.3 (tombstone writer). The reverse UUID index backfill (Step 3) can run live with Op4's HSCAN fallback active.

- **[IMPLEMENTATION] Q062**: The tombstone value format is a colon-delimited string (`TOMBSTONED:0.35:sha256hex:1700000000:42`).
