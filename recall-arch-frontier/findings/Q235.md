## Research Question

Design a memory system where memories surface themselves without being explicitly requested. Forbidden: query, search, request, prompt, ask, lookup, fetch, retrieve, find, pull, call, invoke, trigger, explicit, on-demand, request-response. What physical mechanism causes involuntary recall in human memory, and can it be engineered?

---

## The Physical Mechanism in Human Memory

In human memory, involuntary recall occurs via **spreading activation**:

1. A concept becomes active in working memory (you see "coffee" on a sign)
2. Activation spreads along associative pathways to connected concepts
3. Connected concepts that reach activation threshold surface in consciousness
4. This happens without deliberate intent — the person did not "ask" for anything

The physical substrate: Hebbian learning produces weighted connections between neurons. When one fires, it increases the probability of firing in connected neurons. Strongly connected memories (co-activated frequently) spread activation farther.

**The physical mechanism is not pull — it is resonance.** A memory surfaces when the current active state resonates with the memory's pattern, causing the memory to exceed activation threshold without deliberate initiation.

## The Engineered Equivalent

The Recall 2.0 system already has the necessary substrate: the **CO_RETRIEVED graph** (edges between memories frequently co-activated together).

**The physical mechanism for push-based surfacing:**

> When memory M_current is activated (stored or reinforced), propagate activation along CO_RETRIEVED edges. Any memory M_connected that accumulates activation above threshold surfaces automatically — without any external initiation.

This is **activation spreading on the CO_RETRIEVED graph**.

```
Store event (PostToolUse):
  memory M_new is stored
  → Look up CO_RETRIEVED neighbors of M_new
  → For each neighbor M_neighbor:
      activation[M_neighbor] += w_edge × recency_of_M_new
  → Any M_neighbor with activation > threshold: inject into next context

Decay rule:
  activation[M] *= 0.9 per session (leaky integrator)
```

This is a **leaky integrator over the co-activation graph** — the standard neural model for spreading activation.

## When Push Beats Pull

The TABOO analysis reveals a structural gap in all current memory systems:

**Pull-based systems fail at**: memories that are contextually needed but whose content is not predictable from the current query token sequence. Example:

- Agent is editing a Docker config file
- The relevant memory is: "CasaOS uses bridge networking, containers cannot reach each other via localhost — use the service name"
- The current state does NOT contain the word "localhost" or "bridge networking"
- A pull-based system will not surface this memory because there are no query tokens to match
- A push-based system surfaces it because: editing Docker configs frequently co-activates with networking memories in past sessions → the CO_RETRIEVED graph has a strong edge → activation spreads

**Push wins** when: the needed memory is at the end of an associative chain from the current active memory, but not directly lexically connected to the current state.

**Push fails** when: the co-activation graph has spurious edges (two memories happened to be co-activated but are not causally connected). This is the same failure mode as human involuntary recall (priming false memories).

---

## Does Push Outperform Pull for the Cases That Matter?

| Scenario | Pull | Push |
|---|---|---|
| Agent edits Docker config, needs networking memory | Only if query contains right tokens | Spreads via CO_RETRIEVED from Docker memories |
| Agent asks about port 8200 | Strong (exact token match) | Redundant (pull already works) |
| Agent is in a new session with no context | No history to activate | Activates from recent session memories |
| Cross-domain inference (networking → security) | Only if semantically close | Spreads via associative chain |
| Outdated memory (wrong IP address) | Surfaces equally with correct one | May surface stale memory via old edges |

**Push-based surfacing is complementary to pull-based, not a replacement.** The cases where push wins (associative chain activation without lexical connection) are exactly the cases that require implicit contextual knowledge — the "unknown unknowns" of memory retrieval.

## Implementation for Recall 2.0

**Current state**: CO_RETRIEVED graph exists (edges between memories frequently co-retrieved). This is the substrate.

**What is needed**:

1. **Activation state** (per-session, in Redis or LMDB):
   - When a memory is stored or reinforced: add it to the active set
   - Activation decays over time (leaky integrator, half-life = 1 session)

2. **Spreading step** (at PostToolUse, not at UserPromptSubmit):
   - When memory M_new is stored: propagate activation 1 hop on CO_RETRIEVED graph
   - Neighbors with activation > 0.5 are marked as "primed"

3. **Injection** (at UserPromptSubmit):
   - Include primed memories in the K=10 context alongside pull results
   - Primed memories get a separate context annotation: "[associated]" vs "[retrieved]"
   - Deduplication: if a primed memory is also a top pull result, show it once

4. **Storage cost**: activation state per memory × 22K memories = 22K × 8 bytes = 176KB. Negligible.

**Phase**: Phase 5 (after consolidation). The CO_RETRIEVED graph must be well-populated for spreading activation to be accurate. Currently meaningful, more valuable as the graph matures.

## Verdict

**BREAKTHROUGH.** The physical mechanism for involuntary memory surfacing is spreading activation on the associative graph — exactly the CO_RETRIEVED graph already in Recall 2.0. The implementation is a leaky integrator over CO_RETRIEVED edges, activated when memories are stored (not when a query arrives). Push-based surfacing fills the structural gap in all current memory systems: the "unknown unknown" scenario where contextually needed memories have no lexical connection to the current query. The implementation cost is low (176KB activation state, 1-hop spreading at PostToolUse). The CO_RETRIEVED graph, which was originally designed for consolidation decisions, is revealed as the substrate for push-based associative recall — a dual use that justifies the graph's maintenance cost even more strongly.
