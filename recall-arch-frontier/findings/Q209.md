# Finding: Q209 — Concurrent Write Conflict Resolution in Multi-Agent Memory

**Question**: When 5 agents write memories simultaneously about the same topic, what happens to behavioral scoring, co-retrieval edges, and importance floors? Is a single-writer authoritative model better, and what coordination protocol does it require?
**Question Type**: [ADVERSARIAL]
**Verdict**: PROMISING
**Severity**: High
**Source field**: LMDB MVCC semantics (single-writer, MVCC reads); distributed systems theory (Lamport clocks, CRDT semantics, last-write-wins vs. merge); concurrent write analysis against Recall 2.0's LMDB+HNSW architecture; Q041 prior finding (Lamport Clock 64-bit per record); behavioral scoring inflation analysis; SimHash dedup behavior under parallel writes.

**Corpus anchor**: Live Recall confirmed prior Q041 finding: "Lamport Clock requires 64 bits per memory record for conflict resolution" — the multi-machine write conflict problem was previously analyzed. Q209 extends this to the specifically agentic failure mode: same-host concurrent writes from multiple agents within a single task.

---

## Evidence

### LMDB Concurrency Model (Baseline Constraint)

LMDB provides:
- **MVCC reads**: Unlimited concurrent readers, zero blocking
- **Single-writer**: Exactly one write transaction at a time; concurrent write attempts serialize (queue, not reject)
- **Durable transactions**: Write is atomic, crash-safe, no partial writes

The immediate implication: **LMDB's single-writer property eliminates write corruption races by construction.** Two agents cannot simultaneously corrupt the same key because one transaction blocks until the other commits. This eliminates Failure Modes 3 and 4 from the hypothesis (no phantom edge creation from race conditions, no corruption from lost increment).

What LMDB does NOT prevent: **semantic corruption** — the ordering of serialized writes can still produce incorrect behavioral outcomes even though each individual write is technically correct.

---

### Failure Mode Analysis

#### Failure Mode 1: CO_RETRIEVED Score Inflation (SILENT)

**Scenario**: 5 agents, all sub-tasks of one user task, each independently call `recall_search` for their context window. All 5 queries happen to be similar enough that they each retrieve memories {A, B, C}. Each agent's retrieval event creates CO_RETRIEVED increments: edge(A-B) += 1, edge(A-C) += 1, edge(B-C) += 1. Five agents × 3 edges = 15 total increments for what was effectively 1 user interaction.

**Is the corruption silent?**: YES. The LMDB write log shows 15 valid CO_RETRIEVED increments. No error. No flag. The behavioral scores for A, B, C are now 5× inflated relative to their true user-facing retrieval frequency.

**How bad is the inflation?**: The score inflation is proportional to agent_count. Relative ordering of memory pairs is preserved (if A-B is truly more co-retrieved than D-E, the inflation multiplies both equally). The inflation only distorts retrieval when comparing agentic co-retrievals against human-interactive co-retrievals — if user sessions generate ~5 CO_RETRIEVED events per session and agent swarms generate 25 per swarm, the behavioral graph drifts toward "memories that multi-agent tasks use" rather than "memories most useful to the human user."

**Severity**: Medium. Distorts behavioral scoring over time for users who run large agent swarms. Silent — no observability without metrics on co-retrieval event source.

**Mitigation**: Tag CO_RETRIEVED events with source: `{source: "agent", agent_id: "...", task_id: "..."}`. Normalize by unique task_id per time window rather than raw event count. "5 agents on 1 task retrieved A+B together" = 1 normalized event, not 5.

#### Failure Mode 2: Duplicate Memory Creation (DETECTABLE)

**Scenario**: Agent 1 and Agent 2 both process the same context (e.g., reading the same error log) and independently write memories. Agent 1 writes "Service X throws NullPointerException at startup when config Y is missing." Agent 2 writes "NullPointerException in Service X during init — config Y absent." These are the same fact with different surface forms.

**What happens**:
- SHA-256 L0 dedup: MISS (different content, different hash)
- SimHash L1 dedup: likely HIT at threshold 6 (near-duplicate phrasing, same key terms)
- Result: SimHash catches the duplicate. One memory is written; the other is rejected as a near-duplicate.

**Is the detection reliable?**: For same-session parallel writes where content is derived from the same source, SimHash threshold 6 catches ~95% of agent-derived duplicates (Q171 SimHash corpus analysis). For different-session writes derived from different observations of the same real-world fact, SimHash may miss if the surface form diverges (SimHash is form-sensitive, not meaning-sensitive at L1).

**Severity**: Low for same-session writes (SimHash catches them). Medium for cross-session writes from the same agent revisiting the same fact.

**Mitigation**: Existing SimHash L1 dedup handles this. No additional architecture needed for MVP.

#### Failure Mode 3: LMDB Write Queue Latency Under High Concurrency (DETECTABLE via metrics)

**Scenario**: 5 agents × 41ms embedding time (fastembed-rs INT8) = agents complete embeddings at staggered times within a ~50ms window. Each reaches the LMDB write gate. LMDB serializes them.

**Worst case**: If all 5 embeddings complete simultaneously (SIMD parallelism), all 5 write transactions queue. Total write time: 5 × LMDB write latency. From Q197, LMDB write is <1ms per transaction. Total: <5ms extra latency distributed across the 5 agents.

**In practice**: Agents are not perfectly synchronized. Embedding latency variance (~5-10ms) naturally staggers arrivals. Queue depth rarely exceeds 2 at any instant.

**Severity**: Low. 5ms extra latency on write path for 5 concurrent agents is negligible against 41ms embedding time. At 20+ concurrent agents: potentially 20ms extra latency (still <200ms budget).

**Mitigation**: No mitigation needed for 1-20 agent concurrency. For swarms >50 agents: separate LMDB write buffer (queue writes to in-memory buffer, flush in batches), accepting eventual consistency on behavioral scores.

#### Failure Mode 4: CO_RETRIEVED Edge Contention (NOT A FAILURE)

**Scenario**: Agents 1 and 2 simultaneously complete retrieval and both want to increment edge(A, B).

**What actually happens**: LMDB single-writer serialization. Agent 1's transaction commits edge(A,B) = 1. Agent 2's transaction reads edge(A,B) = 1, increments to 2. Both commits succeed. Edge(A,B) = 2. Correct.

**Verdict**: LMDB's single-writer property eliminates this failure mode entirely. No phantom edges, no lost increments, no races. This is the "it just works" result.

#### Failure Mode 5: HNSW Index Inconsistency During Concurrent Inserts (DETECTABLE)

**Scenario**: Agent 1 inserts memory M1 into HNSW. Agent 2 simultaneously runs a query. In instant-distance (Recall 2.0's HNSW crate), the index is held in a `RwLock`. Writers acquire exclusive access; readers share access.

**What happens**: Agent 2's query either sees the pre-M1 index state or blocks until M1 insertion completes (depending on RwLock acquisition order). No partial state is possible — RwLock ensures consistent read or consistent write.

**Severity**: Low. Minor query latency spike during parallel insert. Index consistency is maintained.

### Coordination Model Comparison

| Model | Isolation Guarantee | Overhead | Failure Mode Addressed |
|---|---|---|---|
| LMDB single-writer (current design) | Write serialization only | Zero (built-in) | FM3, FM4 |
| Single-writer authoritative (topic ownership) | One agent writes per topic | High coordination overhead | FM1, FM2 |
| CRDT merge (G-Counter for CO_RETRIEVED) | Eventual consistency, no conflicts | Medium (merge protocol) | FM3, FM4 (already solved) |
| Source-tagged events + task normalization | Semantic accuracy | Low (tag + normalize) | FM1 (the actual problem) |

**Recommended model**: LMDB single-writer (current design) + source-tagged CO_RETRIEVED events + task_id normalization.

**Why single-writer authoritative is overkill**: LMDB already enforces write serialization. The only unaddressed failure mode is FM1 (semantic inflation), which doesn't require write serialization — it requires event attribution. A single-writer authoritative model would add coordination overhead (topic lease acquisition, lease expiration, conflict retry) for a problem that source tagging solves at zero write-path cost.

**Why CRDT merge is unnecessary**: LMDB serialization + SimHash dedup already solve the technical write conflict problem. The semantic problem (FM1) is not a merge conflict — it's a counting normalization problem. CRDT adds complexity without addressing the actual gap.

### Behavioral Score Drift Estimation Under 5-Agent Load

**Setup**: Single user with 22K memory corpus, running a 5-agent orchestrated task that retrieves similar context for each agent.

**Expected CO_RETRIEVED events per task**:
- Per-agent: ~10 memories retrieved per query × 5 queries per task = 50 retrievals per agent
- C(50, 2) = 1,225 potential co-retrieval pairs per agent
- 5 agents × 1,225 = 6,125 events in one task (without deduplication)
- With 70% overlap across agents (same context): ~1,837 unique pairs, each inflated ~3.3×

**Baseline (single user, interactive session)**: ~20 memories retrieved per session × 3 sessions = 60 retrievals. C(20,2) = 190 pairs per session.

**Drift ratio**: One 5-agent task generates 6,125 CO_RETRIEVED events vs. 190 per interactive session. A user running one agent swarm per week would accumulate 30× more agentic CO_RETRIEVED events than interactive events after 1 month. The behavioral graph would converge on "what the agent found useful" not "what the user found useful."

**Mitigation math**: Task_id normalization reduces 6,125 events to ~1,837 unique pairs (same as 1 retrieval event, not 5). This preserves the relative-ordering property while preventing absolute inflation. Implementation: `INCR co_retrieved:{task_id}:{min(A,B)}:{max(A,B)}` — one unique key per task + pair. Final score update: for each unique pair, increment the persistent CO_RETRIEVED edge by 1 (not by the per-agent count).

## Ideas Extracted

### Idea: task-id-normalized-co-retrieved-prevents-agent-swarm-score-inflation
- **Novelty**: 0.75 — The specific design of task-id normalized CO_RETRIEVED scoring (counting unique tasks that co-retrieved a pair, not raw event count) is not documented in any surveyed AI memory system. It directly addresses the agentic failure mode that single-user systems cannot encounter. The normalization prevents behavioral graph drift toward "what agents find useful" vs "what users find useful."
- **Evidence**: 0.80 — Score inflation analysis: without normalization, 5 agents generate 6,125 events vs. 190 for an interactive session (32× inflation ratio). With task_id normalization: 5 agents on 1 task = 1 task-normalized event (same weight as 1 interactive session). The math is derivable from corpus parameters and retrieval density assumptions. No existing system implements this — all surveyed CO_RETRIEVED designs (Q167, Q170) assume single-agent retrieval patterns.
- **Feasibility**: 0.85 — Implementation: at CO_RETRIEVED write time, include task_id in the event: `co_retrieved_events table: (memory_a_id, memory_b_id, task_id, agent_id, timestamp)`. Batch processing: GROUP BY (memory_a, memory_b, task_id) → COUNT DISTINCT task_ids → update behavioral edge. The task_id is available via hook context (session_id serves as task_id proxy in Claude Code). LMDB key: `co_ret:{task_id}:{min(a,b)}:{max(a,b)}` → bool (seen/not seen), separate from the persistent edge score. At session end: flush task_id bloom → update persistent CO_RETRIEVED edges once per task. Feasibility: high, implementation in Phase 3 of Q200's plan.
- **Description**: Without task_id normalization, a 5-agent swarm running one task generates 32× more CO_RETRIEVED events than an interactive session, causing the behavioral graph to drift toward "what multi-agent tasks use" rather than "what users find useful." Fix: normalize CO_RETRIEVED by task (count unique tasks that co-retrieved a pair, not raw events). Implementation: at write time, record (memory_a, memory_b, task_id) in a per-session bloom/set; at session end, for each unique pair seen, increment the persistent CO_RETRIEVED edge by 1 (regardless of how many agents retrieved them). Result: 5 agents on 1 task = same behavioral signal weight as 1 interactive user session. Prevents behavioral graph drift in agentic deployments.

### Idea: lmdb-single-writer-eliminates-write-corruption-races-by-construction
- **Novelty**: 0.50 — The finding that LMDB's single-writer MVCC property prevents write corruption races is well-established in LMDB documentation. The value for Recall 2.0 is the explicit confirmation: the feared concurrent write corruption modes (phantom edges, lost CO_RETRIEVED increments) are impossible in LMDB without any additional synchronization.
- **Evidence**: 0.90 — LMDB specification: exactly one write transaction at a time, all others queue. MVCC reads proceed without blocking writers. Atomicity: a write is either complete (committed) or not started (queued) — no partial writes. This property was used in Q197's write-path analysis and is confirmed in LMDB's documented behavior.
- **Feasibility**: 0.99 — This is a property of the chosen storage engine. No implementation required. Implication for Q209: the write corruption failure modes feared in the question are prevented by the architecture. The real problem (semantic inflation from FM1) requires task normalization, not additional write coordination.
- **Description**: LMDB's single-writer MVCC property prevents all technical write corruption races in concurrent multi-agent scenarios: concurrent CO_RETRIEVED increment attempts serialize (no lost increments), HNSW inserts use RwLock (no partial index states), SimHash dedup catches near-duplicate writes. The only unaddressed failure mode is semantic inflation (FM1) — CO_RETRIEVED counts inflated by agent count rather than user-facing utility. FM1 is a normalization problem, not a synchronization problem. Architecture decision validated: no additional write coordination needed beyond LMDB's built-in serialization.

- **Novelty**: 0.65 — Storing agent_id in the memory write payload (alongside content, embedding, hook_type) enables downstream source attribution for both behavioral scoring normalization and provenance analysis. ## Analytical Summary

**The feared write corruption modes are prevented by LMDB's architecture.** Concurrent writes queue — no races, no lost increments, no phantom edges. HNSW inserts use RwLock. SimHash dedup catches duplicate content from parallel agents.

**The real failure mode is semantic inflation of behavioral scores.** Without task normalization, a 5-agent swarm generates 32× more CO_RETRIEVED events than an interactive session, drifting the behavioral graph toward "what agents use" rather than "what users find useful." This is silent — no error, no flag.

**The fix is task_id normalization, not write coordination.** Count unique tasks that co-retrieved a pair, not raw events. Implementation: per-session bloom filter of (memory_a, memory_b, task_id) tuples → flush once per session to persistent CO_RETRIEVED edges. Cost: one bloom filter per session, flushed at session end. No change to write path latency.

**Phase 3 of Q200's plan (CO_RETRIEVED + behavioral scoring)** should incorporate task_id normalization from day one. Retrofitting normalization after deployment would require recomputing all historical CO_RETRIEVED edges.
