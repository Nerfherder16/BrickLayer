### The S5 Vacancy Problem

After Q115's three-store collapse, the original S5 signal (Neo4j graph traversal epoch compliance) is vacuously true. The PostgreSQL CTE structurally enforces epoch compliance via `AND m.last_access_epoch >= :epoch_threshold` in the depth-1 and depth-2 CTEs (Q115, lines 249-264). There is no runtime mechanism by which stale-epoch memories can enter the BFS result set — the query planner rejects them before materialization. S5 as defined in Q084 (line 19: "> 20% of 1-hop BFS results have `last_access_epoch < current_epoch - tau` over 7-day window") cannot fire in the three-store architecture.

This leaves the health signal set at {S1, S2, S3, S4, S6, S7} — six signals, but with a gap in the five-slot framework that Q084's correlation matrix (10 pairwise rules), Q117's diagnostic trees (S4+S5 sub-classification), and Q114's orthogonality documentation all reference. The question is whether to (a) leave S5 vacant and renumber, (b) redefine S5 to monitor a new property, or (c) add S8 for the new property while tombstoning S5.

### Decision: Redefine S5 — Do Not Tombstone or Renumber

**Option A (tombstone S5, renumber)**: Breaks every reference in Q084 (correlation matrix uses S1-S5 index pairs), Q117 (S4+S5 diagnostic sub-rows), Q114 (S5/S7 orthogonality documentation). The renumbering cost is high: Q084 alone has 10 pairwise entries referencing signal numbers. All downstream findings, runbooks, and dashboard implementations would need updating. This is pure churn with no architectural benefit.

**Option B (leave S5 vacant)**: Preserves backward compatibility of references but leaves a permanent gap in the signal set. Dashboard displays a grayed-out S5 slot. Operators seeing "S5: N/A" will wonder whether it's a monitoring failure rather than a structural elimination. The vacant slot invites confusion without providing information.

**Option C (redefine S5_new)**: Preserves the five-signal framework, reuses the slot for the most dangerous unmonitored failure mode, and all Q084/Q117 references to "S5" are updated to mean the new definition. Q084 correlation matrix rows involving S5 (S1+S5, S2+S5, S3+S5, S4+S5, S5 alone — 5 rows) must be re-evaluated under the new definition, but this is a bounded one-time cost.

**Option D (add S8, tombstone S5)**: Inflates the signal count to 8 while leaving a gap at 5. The pairwise correlation matrix grows from C(7,2)=21 to C(8,2)=28. Seven new co-alert rules must be analyzed. This adds complexity without architectural justification.

**Decision: Option C.** The cost of re-evaluating 5 correlation matrix rows is far lower than the cost of maintaining a vacant slot or inflating the signal count. The three-store stack needs exactly the coverage that a well-chosen S5_new provides.

### Candidate Failure Modes for S5_new

The three candidates named in the question context:

#### Candidate 1: co_retrievals table write failures (silent best-effort writes)

**Mechanism**: The reinforcement UPSERT (Q115, lines 272-280) writes to `co_retrievals` at retrieval time. If this write fails silently (connection timeout, constraint violation, disk full), the co-retrieval graph becomes incomplete. Over time, the BFS CTE returns fewer neighbors than it should, degrading retrieval quality by missing legitimate co-occurrence patterns.

**Severity assessment**: Moderate. The write is a single SQL UPSERT to a table the API server already holds a connection to (same PostgreSQL instance as `memories`). A failure here almost certainly indicates a broader PG issue that S4 (epoch integrity, which also writes to PG) or S3 (pending queue, indicating CasaOS connectivity) would catch. The failure mode is real but overlaps heavily with existing signals. Furthermore, the UPSERT is idempotent — a missed write for a co-retrieval pair that was already recorded has zero effect, and a missed write for a new pair means one missing edge out of ~45 per session. The single-edge miss does not materially degrade BFS quality.

**Monitoring difficulty**: Counting "expected writes vs actual writes" requires instrumenting the write path with a separate counter. This is not a health probe that can be evaluated from data state alone — it requires process monitoring. S5_new should be a state-based signal (queryable from store state), not a process-based signal (requiring event stream instrumentation).

**Verdict on Candidate 1**: Not suitable for S5_new. Low independent information content (overlaps S3/S4), process-based rather than state-based, and low severity per-event.

#### Candidate 2: Qdrant collection drift from PostgreSQL (vectors exist in Qdrant with no PG counterpart after TTL expiry or deletion)

**Mechanism**: When a memory is deleted from PostgreSQL (TTL expiry, manual cleanup, deduplication), the corresponding Qdrant vector point may not be deleted if the deletion path fails to propagate. Q086 (lines 82-117) extensively documents the Qdrant orphan problem: Qdrant points whose UUIDs do not match any PG `memories` row are "orphans" that bypass epoch filtering (Qdrant payload has no `last_access_epoch`), contaminate ANN search results, and are directly injected into Claude's context as valid memories. Q086 line 132: "There is no silent skip. The content of orphaned memories is directly read from Qdrant payload and presented to Claude as valid retrieved context."

**Severity assessment**: HIGH. This is the most dangerous unmonitored failure mode in the three-store architecture for three reasons:

1. **Active injection, not silent skip**: Unlike Neo4j orphans (which were silently skipped in BFS), Qdrant orphans are returned by ANN search and reconstructed into Memory objects via `_payload_to_memory()`. They are actively surfaced to Claude.

2. **Epoch filter bypass**: The LLPC epoch filter operates on `last_access_epoch` stored in PG's `memories` table. Qdrant search does not consult PG for epoch filtering. Orphaned Qdrant points have no epoch gate — they bypass the working-set boundary entirely. This is the exact property S5_old was designed to detect in the graph layer, now manifesting in the vector layer.

3. **Gradual accumulation without alarm**: Every memory deletion that fails to propagate to Qdrant creates one orphan. At Tim's rate of ~20 memories/day with eventual TTL expiry, orphan accumulation is slow but unbounded. There is no existing signal that detects this drift. S6 monitors hub degree (co_retrievals fanout), not Qdrant-PG consistency. S4 monitors Redis-PG epoch consistency. No signal monitors Qdrant-PG point-set consistency.

**Monitoring feasibility**: A periodic probe can compute `orphan_count = qdrant_point_count - pg_memory_count` as a fast approximation, or run a precise join: scroll Qdrant point IDs in batches, check each against PG `memories.id`. At Tim's scale (< 10,000 points at Month 12), a full scroll takes < 500ms. This is a state-based signal derivable from existing store state.

**Verdict on Candidate 2**: STRONG candidate for S5_new. High severity, zero overlap with existing signals, state-based measurement, and fills the exact gap (cross-store consistency) that S5_old covered in the Neo4j layer.

#### Candidate 3: Embedding model version skew between stored vectors and current model

**Mechanism**: If the Ollama embedding model (qwen3-embedding:0.6b at 192.168.50.62:11434) is upgraded or changed, new memories are embedded with the new model while old memories retain embeddings from the previous model. Cosine similarity between vectors from different model versions is meaningless — the embedding spaces are not aligned. ANN search returns results ranked by a metric that no longer measures semantic similarity. Q023 (line 164) documents background re-embedding as the solution. Q116 (line 178) notes the embedding cache should be invalidated on model version change.

**Severity assessment**: CATASTROPHIC when it occurs, but extremely rare. An embedding model change is a deliberate operator action (Tim manually pulls a new model on the Ollama server). It is not a failure mode that accumulates gradually or occurs from component drift. The detection is trivial: compare the model name/version used for the current embedding request against the model name/version stored with existing memories. If they differ, every stored vector is invalid.

**Monitoring feasibility**: Store the model version string in a PG column or Qdrant payload field. At health check time, compare the current Ollama model version against the stored version. If they differ and no re-embedding job is running, alarm.

**Verdict on Candidate 3**: Real risk but wrong granularity for a health signal. Model version skew is a binary event (either all vectors match the model or none do), not a gradual degradation. It is better modeled as a startup validation check or a write-path assertion than as a periodic health signal. A health signal that is either 0% or 100% has no gradient — no WARNING state, no early detection. It fires at the moment the model changes and requires immediate full-corpus re-embedding. This is an operational gate, not a health signal.

### S5_new Specification: Qdrant-PostgreSQL Point-Set Consistency

**Definition**: S5_new measures the fraction of Qdrant vector points that have no corresponding row in PostgreSQL's `memories` table (orphan fraction).

**Signal name**: `qdrant_pg_orphan_fraction`

**Measurement query** (run as periodic health probe, recommended: every 10 minutes or at session-start):

```sql
-- Step 1: Get total Qdrant point count per tenant collection
-- (via qdrant-client: client.get_collection("recall_tim").points_count)
-- Returns: qdrant_count

-- Step 2: Get PG memory count per tenant
SELECT COUNT(*) AS pg_count
FROM memories
WHERE tenant_id = 'tim';

-- Step 3: Compute orphan estimate
-- orphan_estimate = max(0, qdrant_count - pg_count)
-- orphan_fraction = orphan_estimate / qdrant_count
```

For precise measurement (run weekly or on WARNING trigger):

```python
# Scroll all Qdrant point IDs
qdrant_ids = set()
offset = None
while True:
    result = client.scroll("recall_tim", limit=1000, offset=offset, with_payload=False)
    points, next_offset = result
    qdrant_ids.update(p.id for p in points)
    if next_offset is None:
        break
    offset = next_offset

# Check against PG
pg_ids = set(row['id'] for row in await db.fetch(
    "SELECT id FROM memories WHERE tenant_id = 'tim'"
))

orphans = qdrant_ids - pg_ids
orphan_fraction = len(orphans) / len(qdrant_ids) if qdrant_ids else 0.0
```

**Thresholds**:

| Level | orphan_fraction | Approximate orphan count at N=5,400 | Action |
|---|---|---|---|
| HEALTHY | < 0.01 (1%) | < 54 | Normal; minor drift from transient write failures |
| WARNING | 0.01 – 0.05 (1-5%) | 54 – 270 | Investigate: check if TTL deletion is propagating to Qdrant; run precise orphan scan |
| CRITICAL | > 0.05 (5%) | > 270 | ANN contamination material (Q086: 5% orphan rate = 2-3% recall@10 degradation); trigger `/reconcile` with PG-authoritative direction |

**Why these thresholds**: Q086 (lines 172-179) quantifies HNSW recall@10 degradation by orphan rate: 1% orphans = ~0.5% degradation (negligible), 5% = ~2-3% (noticeable), 10% = ~5-8% (material). The WARNING threshold at 1% provides early detection before ANN quality is affected. The CRITICAL threshold at 5% fires before the degradation becomes perceptible to the user.

**Recovery action**: At CRITICAL, run a PG-authoritative reconciliation that **deletes Qdrant points not found in PG** (the inverse of the current `/reconcile` endpoint, which heals toward Qdrant). Q086 line 115 documents that the current reconcile endpoint heals in the wrong direction for this scenario. S5_new's recovery requires implementing a `reconcile_pg_authoritative` endpoint.

### Whether a New S8 Signal Is Warranted

The three candidates evaluated above produce one S5_new (Qdrant-PG consistency) and two rejected candidates (co_retrievals write failures, embedding model version skew). Neither rejected candidate warrants an S8 signal:

- **co_retrievals write failures**: Overlaps S3/S4, process-based not state-based, low per-event severity.
- **Embedding model version skew**: Binary event, not gradual degradation. Better modeled as a startup assertion (`assert current_model_version == stored_model_version`) than as a health signal. An S8 signal for this would be either 0.0 (healthy) or 1.0 (catastrophic) with no gradient — this is an operational gate, not a monitoring signal.

**Recommendation: No S8.** The revised signal set {S1, S2, S3, S4, S5_new, S6, S7} covers:
- Retrieval quality (S1: IPS P@3 delta)
- Routing quality (S2: MBC Level 1 fraction)
- Write-tier availability (S3: pending queue depth)
- Epoch-clock consistency (S4: Redis vs PG epoch integrity)
- **Cross-store consistency (S5_new: Qdrant-PG orphan fraction)** — NEW
- Hub degree health (S6: max fanout in co_retrievals)
- Reclassification job health (S7: drift_fraction)

The seven signals partition the failure space into three domains:
1. **Retrieval quality** (S1, S7): are the right memories being surfaced?
2. **Data integrity** (S4, S5_new, S6): are the stores consistent and structurally sound?
3. **Operational health** (S2, S3): are the write and routing pathways functioning?

S5_new fills the cross-store consistency role that S5_old held (Neo4j-PG consistency via BFS epoch compliance), adapted for the three-store architecture (Qdrant-PG consistency via orphan fraction).

## Recommendations

1. **[IMPLEMENTATION] Redefine S5 to Qdrant-PG orphan fraction** — Medium effort, high impact. Implement the fast approximation probe (`qdrant_count - pg_count`) for session-start health checks and the precise scroll-and-join probe for weekly scans or WARNING-triggered deep checks. Add thresholds (HEALTHY <1%, WARNING 1-5%, CRITICAL >5%) to the `/health/detail` endpoint. Update the dashboard S5 display.

2. **[IMPLEMENTATION] Build PG-authoritative reconciliation endpoint** — Medium effort, high impact. This is the recovery action for S5_new CRITICAL. The endpoint scrolls Qdrant point IDs, checks each against PG `memories.id`, and deletes orphans. Must be idempotent and safe to auto-trigger when S5_new fires alone (S4 healthy). 3. **[DOCUMENTATION] Update Q084 correlation matrix** — Low effort, medium impact. Replace the 5 S5_old pairwise rows with the S5_new evaluations from this finding. The S4+S5_new Direction 2 co-alert is the most important update: it now provides a cleaner PG-restore discriminant than the original.

If the current deletion path does not propagate to Qdrant at all (deletion not implemented), S5_new may already be in WARNING or CRITICAL at steady state. 5. **[IMPLEMENTATION] Add embedding model version gate** — Low effort, medium impact. Store model version in PG metadata table. Assert at startup. Include version in Qdrant payload for future per-point version tracking.
