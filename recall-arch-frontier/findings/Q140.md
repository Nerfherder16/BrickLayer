# Q140 — Unified Write Durability Tier Model
**Question**: Does the WAL-derived outbox durability tier (Q129) combined with Q056's write-tier partition policy define a complete and correct durability classification for all Recall write types — and what does each write type (co_retrievals UPSERT, memory store, session summary, observe-edit fact extraction, epoch INCR) map to in the unified tier model?
**Agent**: quantitative-analyst
**Question Type**: CONVERGENCE
**Verdict**: BREAKTHROUGH
**Severity**: High

---

## Convergence Scope

This finding synthesizes four prior findings:

- **Q056**: Write-tier partition policy — asymmetric durability by write type; session summaries buffered locally with retry; observe-edit fact extractions fail cleanly
- **Q129**: WAL-derived outbox pattern for co_retrievals UPSERTs — transactional outbox in PostgreSQL eliminates silent signal loss on transient PG failures
- **Q130**: Outbox implementation spec — exact DDL, background flush worker, dead-letter table, S8 health gauge
- **Q119**: Three-store runbook — PG as canonical authority; Qdrant as rebuildable derived store; Redis as epoch-only ephemeral

These four findings were produced in different question types (adversarial, adjacent-field, implementation, operational) and never formally unified. Q140 is the synthesis.

## Step 1: Complete Enumeration of Recall Write Types

The Recall system produces writes across eight distinct categories:

| # | Write Type | Trigger | Destination(s) |
|---|-----------|---------|---------------|
| W1 | Memory store | `/recall/store` endpoint via hook or API call | PG `memories` + Qdrant collection + Redis epoch INCR |
| W2 | co_retrievals UPSERT | Retrieve call returns K results | PG `co_retrievals` table (batch UPSERT) |
| W3 | Session summary | `recall-session-summary.js` Stop hook | PG `memories` (domain=session_summary) via `/recall/store` |
| W4 | Observe-edit fact extraction | `observe-edit.js` PostToolUse hook | PG `memories` via `/recall/store` |
| W5 | Epoch INCR | Every store or retrieve call that mutates state | Redis `recall:tim:global_epoch` |
| W6 | Canary seed insertion | Startup + post-restore reconciliation | PG `canary_seeds` table + Qdrant (canary points) + Redis `recall:canary_seeds` |
| W7 | Health signal writes | `/health/detail` computation; S1-S4, S6, S8 signal persistence | In-memory only (computed on-demand); S4 epoch value read from Redis and PG |
| W8 | Outbox flush (retry) | Background `_co_retrievals_flush_worker` every 5 seconds | PG `co_retrievals` from PG `co_retrievals_outbox` |

Note: W7 (health signal writes) is a degenerate category — health signals are computed on-demand from existing store state, not independently written. They are included for completeness and immediately classified as ephemeral (no durability requirement).

## Step 2: Durability Classification Framework

Three durability tiers, derived from Q056's write-tier partition policy and Q129's outbox analysis:

**Tier A — Must-Not-Lose**: The write is irreplaceable. Loss creates a permanent gap in the memory record. Recovery requires either re-creating the lost information from external sources (difficult or impossible) or accepting a permanent knowledge gap. Examples: session summaries, memory store records, co_retrievals graph edges.

**Tier B — Eventually-Consistent / Best-Effort**: The write reinforces an existing signal. Loss degrades quality but does not create a correctness gap. The system remains functionally correct; only the signal fidelity is reduced. The write may be deferred, batched, or retried without semantic harm. Examples: co_retrievals UPSERTs (each reinforcement is one increment among many), epoch INCRs (recoverable from PG max).

**Tier C — Ephemeral / Reconstructible**: The write either has no persistence requirement (health signals, computed on-demand) or is fully reconstructible from authoritative Tier A stores. Loss is tolerable because recovery is mechanical. Examples: Qdrant vector points (rebuildable from PG content via re-embedding), Redis epoch (resettable from PG MAX(last_access_epoch)), canary Redis set (resettable from PG canary_seeds table).

## Step 3: Unified Write Durability Matrix

### W1 — Memory Store (PG + Qdrant + Redis)

**Tier A (PG INSERT into `memories`)**: The canonical content record. If lost without backup, the memory content is unrecoverable (Runbook 3C in Q119: "content unrecoverable"). This is the most critical write in the system.

**Current mechanism**: Synchronous parallel write — PG INSERT, Qdrant UPSERT, Redis INCR all fired in the same request. FastAPI returns 200 only after all three complete (or logs partial failure via S4 epoch divergence).

**Gap analysis**:
- PG INSERT has no outbox protection. If the PG transaction fails after Qdrant UPSERT and Redis INCR succeed, the epoch counter advances but no memory row is written. S4 detects this (epoch gap > 1 on consecutive writes), but there is no automatic recovery — the orphaned Qdrant vector and the incremented epoch are never matched to a PG row.
- The outbox pattern from Q129/Q130 was designed for co_retrievals, not for the primary memory store write. Q130 explicitly states: "Do not add the outbox to the memory store path (`/store` endpoint)." The reasoning: S4 epoch-integrity monitoring is the existing durability signal for memory store writes.
- **Gap status**: PARTIAL. S4 detects the failure condition (epoch divergence) but does not auto-recover. A failed memory store leaves a detectable signal (S4 CRITICAL) but no automatic retry path. At Recall's write rate (1-5 stores/session), a permanently lost memory write would show up as S4 CRITICAL on next session — detectable but not self-healing.

**Tier B (Qdrant UPSERT)**: Derived from PG content, fully reconstructible via Q119's re-embedding recovery path (Step Q4). Classified as Tier C within the memory store write.

**Tier C (Redis epoch INCR)**: Ephemeral — recoverable from PG MAX(last_access_epoch). Classified as Tier C.

---

### W2 — co_retrievals UPSERT

**Tier**: Between B and A. Individual UPSERTs are best-effort (each is one increment in a monotonically increasing count), but cumulative loss degrades graph signal quality significantly over time. Q129 classifies this as warranting outbox protection for steady-state failures.

**Current mechanism (post-Q130)**: Transactional outbox in PG — INSERT outbox entry + UPSERT co_retrievals + mark outbox processed in a single transaction. Background flush worker retries within 5 seconds. Dead-letter after 5 failed attempts. S8 health gauge monitors dead-letter count.

**Gap status**: CLOSED (post-Q130 implementation). If Q130 is not yet implemented, the gap is HIGH — approximately 3.65 failed co_retrieval writes per year at 99.9% PG uptime (Q129 Q131 estimate), each silently lost with no detection mechanism. Post-Q130: gap is closed; S8 detects any dead-letter accumulation.

**Idempotency**: At-least-once semantics (double-flush increments count by 2 instead of 1). For reinforcement signals, this is correct behavior — slight over-reinforcement, not a correctness failure. Q130 notes exactly-once requires an `outbox_id` column in `co_retrievals`, which was assessed as not warranted at Recall's write rate.

---

### W3 — Session Summary

**Tier A**: Session summaries are once-per-session, synthesized, irreplaceable. Q056 Thread B argument: "Three hours of memory formation... disappears." They flow through the standard `/recall/store` endpoint, so they receive the same PG+Qdrant+Redis treatment as W1.

**Current mechanism**: `recall-session-summary.js` Stop hook calls the Recall API. On API failure: Q056 identifies the gap (no local buffer), recommends writing to `~/.claude/recall-pending/session-{timestamp}.json` for retry on next session start.

**Gap status**:
- If Q056's local pending queue is implemented: **CLOSED**. Session summaries survive transient CasaOS outages; retry fires on next session start.
- If not implemented (current state per project-brief.md): **HIGH RISK**. Probability of loss ≈ 11 lost session summaries per year at 99% CasaOS uptime (Q058 estimate). This is the highest unaddressed durability risk in the system (see Step 5).
- The session summary write uses the same W1 path once it reaches the API — so the W1 gap (no outbox for PG INSERT) applies here too, though S4 detection still applies.

---

### W4 — Observe-Edit Fact Extraction

**Tier B → Tier C**: Q056 explicitly classifies this as "fail cleanly — no local buffer needed." Reasoning: observe-edit writes are high-frequency, low-value-per-write, and the session summary will capture the important content. Individual fact extractions are best-effort signals.

**Current mechanism**: `observe-edit.js` PostToolUse hook fires once per file edit, calls Recall API. On failure: hook logs error, exits. No retry. No local buffer.

**Gap status**: **CLOSED by design decision**. Q056 confirms this is the correct behavior. The absence of retry is intentional, not a gap. The residual risk (fact from the last edit before session end not captured if the session summary also fails) is mitigated by the session summary being a synthesized view of the entire session.

**Caveat**: Observe-edit writes use the W1 PG store path. So they are subject to the same W1 gap (no outbox for PG INSERT). But since the semantic tier is B/C (reconstructible from session summary), this is acceptable.

---

### W5 — Epoch INCR

**Tier C**: The Redis epoch counter is explicitly classified as recoverable in Q119 Runbook 1. Recovery: `SET recall:tim:global_epoch $(MAX(last_access_epoch) + 1)`. Expected downtime: < 2 minutes.

**Current mechanism**: Redis `INCR recall:tim:global_epoch`. Durability: Redis AOF persistence (Q111, Q119 Step R3). If AOF is not enabled (default Redis config without the `--appendonly yes` flag), the epoch key is lost on container restart.

**Gap status**:
- If Redis AOF is enabled: **CLOSED**. Epoch survives container restart.
- If Redis AOF is not enabled: **LOW RISK** (epoch is recoverable from PG MAX within 2 minutes, S4 detects the gap). The Q112 RENAME window is a one-time migration artifact, not an ongoing gap (Q129 confirms this).

---

### W6 — Canary Seed Insertion

**Tier**: Split across stores.

**PG `canary_seeds` table**: Tier A. The authoritative canary registry. If PG is available, canary seeds survive all failure modes short of PG volume loss.

**Qdrant canary points**: Tier C. Reconstructible from PG `canary_seeds` via re-embedding. Canary points marked with `is_canary: true` payload flag (Q128/canary-payload-flag-deletion-exclusion) are excluded from prune/TTL operations.

**Redis `recall:canary_seeds` set**: Tier C. Cache of canary UUIDs for fast lookup. Reconstructible from PG `canary_seeds` table. AOF persistence (Q111) doubles as canary Redis cache persistence, but the authoritative source is PG (canary-redis-persistence-guarantee).

**Current mechanism**: Canary seeds inserted at startup and post-restore. PG is authoritative. Redis serves as cache. Qdrant holds canary vectors.

**Gap status**: **PARTIALLY CLOSED**. The gap is that the Redis canary set may be lost on Redis restart without AOF (same as W5). If AOF is enabled and PG is the authoritative source: closed. If AOF is disabled: the canary system detects its own gap (absence of the canary key triggers the `tombstone-lru-defeat-detection-canary` check).

---

### W7 — Health Signal Writes

**Tier C / Not applicable**: Health signals (S1-S6, S8) are computed on-demand from existing store state. They are not independently persisted. S4 reads from Redis and PG; S8 reads from `co_retrievals_dead_letter`. No write durability is required because the signals are derived, not primary data.

**Gap status**: **N/A**. Health signals have no write durability gap by design — they are computed, not stored.

---

### W8 — Outbox Flush (Retry Writes)

**Tier A mechanism for Tier B data**: The outbox flush worker performs the same co_retrievals UPSERT as W2, but as a retry path. The flush worker itself is not a distinct write type — it is the recovery mechanism for failed W2 writes.

**Gap status**: **CLOSED** (post-Q130). The background asyncio task runs every 5 seconds, retries unprocessed outbox entries, moves to dead-letter after 5 failures, and logs via S8. The task survives FastAPI restart because the outbox lives in PG (inherits PG WAL durability).

## Step 3 Answer: Does Q129 + Q056 Cover All Write Types?

Q129's outbox pattern covers **W2 (co_retrievals)** and **W8 (outbox flush)** completely.

Q056's write-tier partition policy covers **W3 (session summary)** and **W4 (observe-edit)** at the conceptual level, but the implementation gap for W3 (the local pending queue) remains open unless Q056's recommendation has been implemented.

**Write types with no durability classification in Q129 or Q056**:
- **W1 (memory store PG INSERT)**: Neither Q129 nor Q056 addresses the failure mode where the PG INSERT itself fails after Qdrant UPSERT and Redis INCR succeed. Q129 explicitly excludes the memory store path. S4 detects the symptom (epoch divergence) but provides no retry path. This is a gap in the unified tier model.
- **W6 (canary seed insertion)**: Not classified in either finding. Canary writes are covered by later findings (Q119, canary-redis-persistence-guarantee, canary-payload-flag-deletion-exclusion) but not as part of the formal tier model.

**Conclusion**: Q129 + Q056 together define a **partial** unified durability tier model. They cover co_retrievals and session summary write types but leave the primary memory store write (W1) and canary writes (W6) without explicit tier classification or outbox protection.

## Step 5: Highest Unaddressed Durability Risk

**W3 — Session Summary (if Q056 local pending queue not implemented)**

The session summary is:
1. **Once-per-session** — no second chance to capture the session's synthesized knowledge
2. **Fires at session end** — the exact moment when CasaOS is most likely to be temporarily unreachable (Docker restart, system update, network transition coinciding with Claude Code session end)
3. **Irreplaceable** — the LLM synthesis of the session is not derivable from the raw observe-edit fact extractions (the summary contains higher-level abstractions)
4. **Not protected by any existing mechanism** — no outbox (Q130 explicitly scoped to co_retrievals), no local pending queue (Q056 recommended but did not implement), no retry in the Stop hook

Estimated loss rate without Q056 implementation: ~11 session summaries per year at 99% CasaOS uptime; ~1 per year at 99.9% uptime.

**Second-highest: W1 — Memory Store PG INSERT with no auto-retry**

The PG INSERT for any memory (including session summaries when they reach the API) has S4 detection but no auto-retry. If the PG transaction fails transiently after Qdrant UPSERT succeeds, the memory is lost. S4 will report CRITICAL on next session, but recovery requires manual intervention (resubmit the memory content — which the user must remember or reconstruct).

At Recall's write rate (3-15 stores/day) and typical PG uptime (99.9%), this occurs roughly 1-5 times per year. Not catastrophic, but the lack of outbox protection for the primary store path is an architectural gap, especially because the co_retrievals write (lower tier, W2) has outbox protection while the memory store write (higher tier, W1) does not.

## Step 6: The Classification Gap — W1 vs W2 Durability Inversion

A notable finding from this convergence: **the durability tier assigned to W2 (co_retrievals) is now stronger than the effective durability of W1 (memory store).**

W2 has: transactional outbox + background flush + dead-letter + S8 health gauge.
W1 has: synchronous write + S4 epoch detection + no auto-retry.

This is a **durability inversion**. A reinforcement signal (W2, eventually-consistent by design) has stronger write protection than a content record (W1, must-not-lose). The Q129 outbox was designed for co_retrievals because that was the specific failure mode analyzed (epoch INCR + PG UPSERT coordination). But the same outbox pattern would be more impactful if applied to the primary memory store path.

**Why this inversion occurred**: Q129 was scoped narrowly to the co_retrievals coordination problem (dual-write between Redis INCR and PG UPSERT). The memory store write was not analyzed as a failure mode because S4 epoch-integrity monitoring was presumed sufficient. S4 detects the symptom but does not heal it — the inversion is a gap in the original scope.

## Recommendations

**Priority 1 — Implement Q056 session summary local pending queue (W3)**

The highest-risk unaddressed gap. Estimated implementation: 40 lines of JavaScript in `recall-session-summary.js` and `recall-retrieve.js`. Write the session summary to `~/.claude/recall-pending/session-{ISO-timestamp}.json` on API failure. Retry on next session start.

**Priority 2 — Verify Q130 outbox is deployed (W2)**

If the co_retrievals outbox from Q130 is not yet deployed, this is a high-priority implementation. The tables (`co_retrievals_outbox`, `co_retrievals_dead_letter`), the flush worker, and the S8 health gauge should be added per Q130's exact DDL and Python spec.

**Priority 3 — PG backup as W1 recovery path**

The memory store PG INSERT has no outbox, only S4 detection. PG backup (daily `pg_dump`) is the sole recovery mechanism for both failed writes and PG volume loss. Q119 Recommendation 1 (daily pg_dump via cron) is the critical implementation. Without it, W1a and W6a losses are potentially unrecoverable.

**Priority 4 — Enforce Redis AOF for W5 and W6c**

AOF enablement (`--appendonly yes --appendfsync everysec`) closes the epoch counter and canary Redis cache gaps. Q111 and Q119 Step R3 both specify this. If not implemented, Redis restart causes a 2-minute S4 CRITICAL episode (recoverable) but not data loss.

**Do not implement** — outbox for W4 (observe-edit fact extractions). Q056 correctly classifies these as Tier C / fail-clean. Adding outbox protection to observe-edit writes would invert the design intent: these are deliberately ephemeral signals, and protecting them with outbox machinery adds complexity for writes that are correctly expendable.

**Do not implement** — PostgreSQL 2PC for W1 or W2. Q129 established that 2PC is overkill for Recall's write coordination problem. The epoch INCR and memory store writes are not in a failure domain that requires distributed transaction coordination.

## Synthesis Statement

Q129 (co_retrievals outbox) and Q056 (write-tier partition policy) together establish the foundational architecture for Recall's write durability model, but they leave a critical gap: the session summary write (W3) — the highest-tier, most irreplaceable write in the system — has no mechanical protection. The durability model as of Q130 has an inverted protection hierarchy: a reinforcement signal (co_retrievals, W2) is durably outbox-protected while a content record (memory store PG INSERT, W1) and an irreplaceable synthesis (session summary, W3) have weaker or no mechanical protection. The unified tier model produced here reveals that fixing this inversion requires two targeted implementations: Q056's 40-line JavaScript pending queue for session summaries (highest impact, lowest effort), and a PG backup policy (the sole recovery path for W1 failures). The co_retrievals outbox (Q130) was the right starting point because it addressed a detected failure mode (epoch coordination); the session summary gap is higher risk but was not stress-tested by any prior finding because it requires a partition scenario (network drop at session end) that does not occur during normal operation.
