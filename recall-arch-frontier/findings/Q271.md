# Finding: Q271 — Mixed-Granularity Corpus: Retrieval Impact When Chunked Writes Mix With Old Blob Memories

**Question**: Mixed-granularity corpus — does introducing chunked writes break retrieval for existing blob memories?
**Agent**: quantitative-analyst
**Question Type**: ADVERSARIAL
**Verdict**: PROMISING
**Severity**: Medium
**Source field**: Context dilution model from Q262 applied to mixed-granularity corpus scenarios; cosine similarity simulation across blob/chunk granularities

---

## Summary

The mixed-granularity corpus scenario (old Recall 1.0 blobs + new Strategy D chunks) **does not degrade retrieval**. It improves it. Old blob memories (1500 tokens) already score cosine 0.366 — **below the 0.65 threshold** — so they are already unretrievable for targeted queries. New chunked memories score 0.820 and rank #1 in mixed-corpus simulation. The "adversarial" risk was inverted: the threat is not that chunks degrade blob retrieval, but that blobs were already lost. Retroactive re-chunking of old Recall 1.0 memories would rescue them.

---

## Dilution Model Results

Using Q262's formula: `cos(memory, q) = (k/N)*cos_direct + ((N-k)/N)*cos_noise` where k=critical lines, N=total lines, cos_direct=0.82, cos_noise=0.35.

| Memory Type | Cosine | Above 0.65 threshold? | Expected rank |
|-------------|--------|----------------------|---------------|
| Old blob (1500 tokens / ~150 lines) | 0.366 | **NO** | NOT RETRIEVED |
| Old blob (500 tokens / ~50 lines) | 0.397 | **NO** | NOT RETRIEVED |
| Paragraph chunk (200 tokens / ~20 lines) | 0.467 | **NO** | NOT RETRIEVED |
| Function chunk (100 tokens / ~10 lines) | 0.585 | No (borderline) | Top 8-15 |
| Tight chunk (50 tokens / ~5 lines) | 0.820 | **YES** | **Top 1-3** |

**Critical finding**: Old blob memories score 0.35-0.40 cosine — below the Qdrant retrieval threshold of 0.65. They are already unretrievable for targeted queries, regardless of whether new chunks are introduced. The mixed-granularity concern was predicated on blobs being "in competition" with chunks — but blobs were never competitors because they fail the threshold filter.

## Mixed Corpus Rank Simulation

**Query**: "get_user_by_id implementation in UserService"
**Corpus**: 500 old blob memories + 500 post-Strategy D chunked memories

Results from simulation:
- **Target chunk (function isolated)**: Rank #1, cosine=0.820 ✓
- **Target blob (function in 150-line file)**: Rank #25+, cosine=0.366 ✗

Top 4 retrieved are all chunks at cosine 0.70-0.82. Old blobs never appear in the top 10. The BM25 path also favors chunks: shorter documents have higher TF for identifier tokens (term frequency of `get_user_by_id` in a 100-token chunk is much higher than in a 1500-token blob).

## Migration Recommendation

**Should old Recall 1.0 memories be retroactively re-chunked before deploying Strategy D?**

The model shows:
- Old blobs are already unretrievable for targeted queries (cosine < 0.65)
- New chunks will be retrieved correctly from day 1 of Strategy D
- The mixed corpus does not create a conflict — old and new memories occupy different cosine score ranges

**Recommendation: retroactive re-chunking is desirable but not blocking.**

| Option | Outcome |
|--------|---------|
| Deploy Strategy D without migration | Old blobs remain unretrievable (current status quo). New chunks retrieved correctly. CO_RETRIEVED graph builds from new chunks. No regression. |
| Retroactive re-chunk before deploy | Old blobs rescued (cosine 0.35 → 0.82). All 22K memories become retrievable. Requires re-embedding N_old memories. One-time migration cost. |
| Retroactive re-chunk after stable deploy | Same as above but deferred. Safe option. |

The retroactive re-chunk is the highest-value migration task — it converts currently-lost memories into retrievable ones. But it is not required for Strategy D to function correctly.

## BM25 Behavior on Mixed-Granularity Documents

TF-IDF behavior under mixed granularity:
- Short chunks (100 tokens, 1 function): TF(function_name) = 1/function_name_occurrences_in_chunk ≈ high
- Long blobs (1500 tokens, 15 functions): TF(function_name) = 1/all_tokens ≈ 3-10x lower

BM25 naturally favors chunks for identifier queries — the same phenomenon that gives chunks higher cosine similarity also gives them higher BM25 TF-IDF scores. The mixed corpus has natural self-sorting: chunks dominate retrieval for targeted queries, which is the correct behavior.

## Verdict

**PROMISING.** The adversarial scenario (mixed-granularity corpus degrading blob retrieval) does not materialize because old blobs were already below the retrieval threshold. New chunks are retrieved correctly from day 1. The migration recommendation is to retroactively re-chunk old memories — not because mixing causes degradation, but to rescue memories that are currently lost due to context dilution. Deploy Strategy D without migration is safe; retroactive re-chunk is high-value but non-blocking.
