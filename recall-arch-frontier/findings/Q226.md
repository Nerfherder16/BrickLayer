## Research Question

How does git implement diff and history? Is there a direct analog for tracking "what did the agent learn this session" — a memory diff — that doesn't require storing full corpus snapshots?

---

## Git's Core Data Model

Git does not store diffs. It stores **content-addressed snapshots** of complete file states. The diff is computed on demand from two snapshots.

**Git's three-layer model:**
1. **Blob**: raw file content, addressed by SHA-1(content). Identical content = identical SHA = stored once.
2. **Tree**: directory structure (name → blob SHA), addressed by SHA-1(tree). Reuses subtrees across commits.
3. **Commit**: points to a root tree + parent commit(s) + metadata. Addressed by SHA-1(commit).

**Delta compression** (pack files): Git periodically packs objects, storing delta diffs between similar blobs to reduce storage. This is an optimization, not the primary model. The conceptual model is always full snapshots.

**Why full snapshots work**: identical content deduplication via SHA means storing "no change" files costs exactly the size of a SHA pointer (~20 bytes). A commit touching 1 file in a 10,000-file repo stores 10,000 - 1 pointers (20 bytes each) + 1 new blob. Storage is proportional to change, not corpus size.

**Diff computation**: `git diff commit1 commit2` = compare the tree of commit1 to the tree of commit2, find objects that differ. This is O(changed files), not O(total files).

## The Memory Diff Model

Apply git's model to Recall 2.0 session-end diffs:

**Memory object storage** (existing): LMDB key-value (memory_id → content + metadata) + HNSW (memory_id → vector). Content-addressed by memory_id (already unique UUID).

**Session diff = three-event types:**

1. **NEW** — memories stored this session (not present in previous session-start snapshot)
2. **UPDATED** — existing memories with modified importance scores, CO_RETRIEVED score, or recency timestamp
3. **DELETED** — memories soft-deleted this session (if any)

**Implementation without full corpus snapshot:**

Track changes during session by appending to a session write-ahead log (WAL):
```
session_start: timestamp=T0, session_id=sid_abc
STORE: memory_id=mem_001, text="...", score=0.4
UPDATE: memory_id=mem_045, importance: 0.6→0.8 (CO_RETRIEVED event)
UPDATE: memory_id=mem_012, co_retrieved: 0.2→0.35
session_end: timestamp=T1, memories_new=1, memories_updated=2
```

At session end, the WAL IS the diff. No snapshot comparison needed.

**Storage cost**: Each WAL entry ≈ 64-128 bytes (IDs + delta values). A session with 5 stores + 3 importance updates = ~640 bytes. 1,000 sessions = 640KB. Negligible.

## Implementation Sketch for Recall 2.0

**Append to existing LMDB schema:**
- Key: `session_wal:{session_id}:{sequence}` → WAL entry (64-128 bytes per event)
- Key: `session_diff:{session_id}` → materialized diff JSON at session end
- Key: `session_index:{date}` → list of session_ids for timeline queries

**Hook integration:**
- `observe-edit.js` (PostToolUse): appends STORE events to session WAL
- `recall-session-summary.js` (Stop): materializes the diff, stores `session_diff:{session_id}`, clears WAL

**Dashboard integration:**
- "Session History" tab: `git log` equivalent — list of sessions with one-line summaries
- Click into a session: shows full diff (NEW/UPDATED/DELETED)
- "Memory Attribution" (future): `git blame` equivalent — which session stored this memory

**Storage cost estimate (22K memories user, 1 year, 2 sessions/day):**
- WAL entries per session: ~10-20 events × 128 bytes = ~2KB per session
- Materialized diffs: ~5KB per session
- 730 sessions × 7KB = 5.1MB per year — negligible

## Verdict

**BREAKTHROUGH.** Git's content-addressed snapshot model maps cleanly to session-scoped memory diffs. The key insight is that a diff doesn't require corpus snapshots — a session WAL (append-only log of store/update/delete events during the session) IS the diff. Storage cost is trivial (~5MB/year). The materialized session_diff JSON is both a technical primitive (provenance, audit trail, Q219 regulatory compliance) and a product feature (users can see what the AI learned each session in the dashboard). This is absent from all production memory systems and directly addresses the "black box" criticism of AI memory systems.
