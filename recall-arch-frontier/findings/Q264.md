## Research Question

What extraction strategies do production code RAG systems use, and what do their retrieval benchmarks show? Specifically: does LLM-based code summarization before embedding outperform direct function-level embedding, and does BM25 hybrid retrieval matter for code content?

---

## Motivation: Mapping to the observe-edit Problem

Recall 2.0's `observe-edit` hook fires on Write/Edit tool use. The extracted content is source code, config files, and CLAUDE.md entries — not prose. This places Recall squarely in the code RAG domain. The question is whether the extraction and retrieval strategies used in production code search systems should inform the Recall chunking and hybrid-search design.

| System | Chunking Strategy | Embedding | Retrieval | Benchmark |
|--------|-------------------|-----------|-----------|-----------|
| **cAST** (CMU / Augment Code, May 2025) arXiv:2506.15655 | AST-based recursive splitting at function/class boundaries; sibling merging | Code embedding model | Dense vector | RepoEval Recall@5, SWE-bench Pass@1 |

| **HEF — Hierarchical Embedding Fusion** (arXiv:2603.06593, Feb 2026) | Two-level: fine-grained chunks + compressed hierarchy | Dense (two-stage) | Offline cache → online pseudo-tokens | RepoBench + RepoEval |

## Benchmark Comparison: Chunking Strategies vs. Retrieval Metrics

| Comparison | Metric | Delta | Source |
|------------|--------|-------|--------|
| AST-based chunking vs. line-based chunking | Recall@5 on RepoEval | **+4.3 points** | cAST (arXiv:2506.15655) |
| AST-based chunking vs. line-based chunking | Pass@1 on SWE-bench | **+2.67 points** | cAST (arXiv:2506.15655) |
| Hybrid BM25+dense+reranking vs. dense-only | NDCG | **+26–31%** | arXiv:2402.03367 and related |
| Hybrid BM25+dense+reranking vs. dense-only | RAG accuracy (production case) | **+48%** | Hybrid E.V.A. cascading retrieval |
| In-context code + API docs vs. similar snippet retrieval | Code generation quality | Similar snippets can **degrade** results | CodeRAG-Bench (arXiv:2406.14497) |
| Hierarchical embedding fusion vs. single-level dense | RepoBench + RepoEval | Reduces inference cost, preserves repository-level info | HEF (arXiv:2603.06593) |

**Critical finding from CodeRAG-Bench**: retrieved similar code snippets can _degrade_ model performance relative to in-context code or API documentation, due to noise introduced by inexact matches. This means retrieval quality is not neutral — a bad chunk can actively hurt the downstream task.

## LLM Extraction vs. Direct Embedding: What the Evidence Says

### The Question

Should observe-edit extract a natural-language summary of each code chunk (via LLM call) before embedding, rather than embedding the raw code directly?

### Evidence

**No published benchmark shows LLM summarization of code before embedding outperforms direct function-level embedding for code retrieval tasks.**

| Approach | Evidence | Assessment |
|----------|----------|------------|
| Direct function-level embedding | cAST +4.3 Recall@5; supermemoryai/code-chunk; wide production use | Strong positive evidence |
| Code → NL summary → embed summary | arXiv:2503.20589 (March 2025): API information > similar snippets > in-context code — but this is about retrieval source type, not summarization pre-processing | No direct evidence for this pattern outperforming direct embedding |
| LLaVACode compressed representations (arXiv:2510.19644) | Visual-style code compression for generation tasks | Not applicable to retrieval |

The closest relevant finding is arXiv:2503.20589, which shows that _what you retrieve_ matters (API docs > similar code snippets), but this does not test LLM summarization as a preprocessing step before embedding.

**Verdict on LLM extraction for observe-edit**: No production evidence justifies the added latency and API cost of LLM-based summarization before embedding. Function-level structural chunking with direct embedding is the production consensus, and LLM summarization introduces an additional failure mode (hallucinated or lossy summaries) with no measured benefit for retrieval recall.

## BM25 Hybrid Impact for Code Content

BM25 has a structural advantage for code that does not apply to prose:

| Code artifact | BM25 strength | Dense embedding strength |
|---------------|---------------|--------------------------|
| Function names (`get_user_by_id`) | High — exact token match | Moderate — semantic but not exact |
| Error codes (`ECONNREFUSED`, `404`) | High — BM25 scores exact strings well | Low — rare tokens have noisy embeddings |
| Version strings (`3.2.1`, `v4.0`) | High | Low |
| Import identifiers (`from qdrant_client import`) | High | Moderate |
| Semantic intent ("database connection retry logic") | Low | High |

For Recall specifically, stored memories frequently contain Docker container names, port numbers, IP addresses, service identifiers, and error messages — all high-value BM25 targets that dense embeddings handle poorly. The +26–31% NDCG improvement and +48% RAG accuracy gains from hybrid retrieval are directly applicable to observe-edit memory content.

**Mapping to Recall 2.0**: Recall already uses Tantivy (BM25) + HNSW (dense) with hybrid fusion. This is the correct architecture. The code RAG evidence validates this design choice specifically for observe-edit content, not just prose memories.

## Consensus Minimum Viable Chunk Size for Code

| Chunk size | Problem | Source |
|------------|---------|--------|
| ~50 tokens | Loses surrounding context; function signature without body | Production consensus |
| 100–400 tokens | Sweet spot: function-level coverage with imports/scope context | cAST, supermemoryai/code-chunk, production deployments |
| ~1000+ tokens | Context dilution; multiple functions mixed; retrieval precision drops | Production consensus |

**Why function-level (100–400 tokens) is the sweet spot**:
1. Functions are the natural unit of code meaning — they have a coherent purpose
2. AST boundaries align with semantic boundaries, preventing mid-logic splits
3. Including the scope chain (imports + enclosing class) provides disambiguation without dilution
4. This size range fits comfortably in embedding model context windows with room for the query

**For observe-edit in Recall**: edited files often contain multiple functions or config blocks. The correct strategy is to split at function/class/top-level-key boundaries rather than using fixed-size line counts. A config file should be split at top-level key boundaries; a Python file at function/class boundaries; a CLAUDE.md at `##` section boundaries.

## Hierarchical Embedding (HEF): Future Direction

The February 2026 HEF paper (arXiv:2603.06593) proposes a two-stage approach: fine-grained chunks are compressed into a reusable dense hierarchy cached offline, then mapped to pseudo-tokens online. This allows repository-level context without full-document retrieval latency.

For Recall 2.0, this maps to the difference between:
- **Current**: individual memory embeddings retrieved independently
- **HEF-inspired**: a compressed representation of a session or file could act as a "parent node" that activates related fine-grained memories

This is a Phase 3 direction — not immediately implementable — but it aligns with the spreading activation / CO_RETRIEVED graph direction established in Q245 and Q256.

## Ideas

**IDEA 1** — `ast-structural-chunking-observe-edit-function-class-boundaries-100-400-tokens-no-llm-extraction` (0.80, 0.85, 0.90): For observe-edit, split extracted content at structural boundaries (function/class for code, `##` sections for CLAUDE.md, top-level keys for config) rather than fixed line counts. Target 100–400 token chunks. Include enclosing scope (imports, class header) in each chunk. No LLM extraction pass needed. Evidence: cAST +4.3 Recall@5 over line-based; supermemoryai/code-chunk production deployment.

**IDEA 2** — `bm25-hybrid-validated-for-code-identifiers-error-codes-ips-ports-direct-evidence` (0.85, 0.80, 0.95): Tantivy BM25 + HNSW hybrid already in Recall 2.0 is specifically validated for code content (identifiers, version strings, IPs, error codes). No architecture change needed; evidence confirms the existing design is correct. The +48% RAG accuracy gain from hybrid retrieval is especially applicable to observe-edit memories (homelab config, Docker commands, service identifiers).

**IDEA 3** — `llm-summarization-pre-embedding-no-evidence-skip-it` (0.70, 0.80, 0.95): LLM-based code summarization before embedding has no published benchmark support for code retrieval tasks. Adding an LLM extraction pass to observe-edit would increase latency 10–50× with no measured recall benefit and introduces lossy-summary failure modes. Decision: keep direct embedding of structural chunks.

**IDEA 4** — `hef-inspired-session-level-parent-node-phase3-future` (0.55, 0.40, 0.50): HEF (arXiv:2603.06593) compresses chunk hierarchies into reusable dense vectors. A Recall analog: compress each session's memories into a session-level parent node in the CO_RETRIEVED graph. Long-range: activating a session node could activate its constituent memories. Low priority; requires CO_RETRIEVED graph to be operational first.

## Verdict

**PROMISING.** Production code RAG systems converge on AST/structural chunking at function/class boundaries (100–400 tokens) with BM25 hybrid retrieval. Benchmarks show +4.3 Recall@5 for AST chunking over line-based (cAST) and +26–48% retrieval improvement from BM25+dense hybrid (multiple production cases). No evidence supports LLM summarization before embedding for code retrieval — direct structural chunking with embedding is the production standard. For observe-edit, this translates to: split at function/class/section boundaries, embed directly, rely on Tantivy+HNSW hybrid already in Recall 2.0. No LLM extraction layer is justified. The HEF hierarchical embedding approach is a Phase 3 future direction aligning with the CO_RETRIEVED graph architecture.
