# Finding: Q115 — Full Stack Collapse: Three-Store Architecture

**Question**: Now that Q108 eliminates Neo4j, can the Recall stack collapse from five stores (PostgreSQL + Neo4j + Qdrant + Redis + Ollama) to three (PostgreSQL + Qdrant + Redis) — specifically, is there any remaining function currently served by Neo4j that the `co_retrievals` CTE architecture in PostgreSQL does not cover, and what is the health signal impact of removing Neo4j?
**Question Type**: ARCHITECTURE
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: database consolidation, operational simplicity, homelab resource optimization

---

## Neo4j Function Audit

The following table enumerates every function Neo4j currently performs in the Recall stack and evaluates whether PostgreSQL covers it after the Q108 co_retrievals migration.

| Neo4j Function | Cypher Form | PostgreSQL Equivalent | Covered? | Notes |
|---|---|---|---|---|
| CO_RETRIEVED BFS traversal (FRESH) | `MATCH (s)-[:CO_RETRIEVED_FRESH*1..2]->(n)` | Depth-2 CTE self-join on `co_retrievals WHERE relationship_type='FRESH'` (Q108) | **YES — fully** | Q108 BREAKTHROUGH confirmed; partial index is structural analog to per-type chain |
| FRESH→STALE type migration | CREATE new rel + DELETE old rel (Q096 two-phase) | `UPDATE co_retrievals SET relationship_type='STALE' WHERE ...` (Q108) | **YES — improved** | PostgreSQL UPDATE is atomic; eliminates Q099 race condition entirely |
| Hub-degree scan for S6 | `MATCH (n)-[r:CO_RETRIEVED_FRESH]->() RETURN count(r) AS degree` | `SELECT from_memory_id, count(*) AS degree FROM co_retrievals WHERE relationship_type='FRESH' GROUP BY from_memory_id ORDER BY degree DESC LIMIT 20` | **YES — trivial** | Standard GROUP BY; O(N) table scan, same cost as Neo4j's O(N) node scan |
| S6 power-law histogram | `MATCH (n)-[r:CO_RETRIEVED_FRESH]->() WITH ... RETURN bucket, count(*)` (Q102 S6-HISTOGRAM) | `SELECT degree, count(*) FROM (SELECT from_memory_id, count(*) AS degree FROM co_retrievals WHERE relationship_type='FRESH' GROUP BY from_memory_id) t GROUP BY degree ORDER BY degree` | **YES — trivial** | Direct SQL translation of S6-HISTOGRAM Cypher query |
| Content_hash deduplication lookup | `MATCH (n:Memory {content_hash: $hash}) RETURN n` | `SELECT id FROM memories WHERE content_hash = $hash` | **YES — already in PG** | PG has always owned the content_hash; Neo4j never held the authoritative copy |
| Epoch-based node filtering | `WHERE n.last_access_epoch >= current_epoch - tau` | `WHERE last_access_epoch >= :threshold` on `memories` table | **YES — already in PG** | `last_access_epoch` column already on `memories` table |
| /reconcile orphan scan | `MATCH (n:Memory) WHERE NOT EXISTS {MATCH (:PG {hash: n.content_hash})}` | `SELECT * FROM qdrant_vectors WHERE content_hash NOT IN (SELECT content_hash FROM memories)` | **YES — already in PG** | Q086/Q093 established PG-join approach; Neo4j never owned this query |
| FRESH/STALE edge reinforcement on co-retrieval | `MERGE (a)-[:CO_RETRIEVED_FRESH]-(b) SET r.epoch=...` | `INSERT INTO co_retrievals ... ON CONFLICT DO UPDATE SET relationship_type='FRESH', ...` (Q108 UPSERT) | **YES — improved** | UPSERT is atomic; no MERGE+property-copy pattern |
| Multi-tenant structural isolation | `TIM_CO_RETRIEVED_FRESH` vs `SADIE_CO_RETRIEVED_FRESH` relationship type prefixing (Q109) | `WHERE tenant_id='tim' AND relationship_type='FRESH'` with per-tenant partial index | **YES — equivalent** | Partial index `WHERE relationship_type='FRESH' AND tenant_id='tim'` provides structural pre-filter at the index level, not a post-filter predicate |
| CAUSES/CONTRADICTS causal edges (Q017) | `MATCH (a)-[:CAUSES]->(b)`, `MATCH (a)-[:CONTRADICTS]-(b)` | Table `causal_edges(from_id, to_id, edge_type, weight, ...)` with index on `(from_id, edge_type)` | **YES — if needed** | **Q017 causal edges were never implemented beyond proposal** — see Section 1 below |

### Section 1: Causal Edge Status Determination

Q017 proposed a {CAUSES, CONTRADICTS, CORRELATES} edge schema as a first-principles derivation. It was marked BREAKTHROUGH. However, reviewing all subsequent findings (Q018–Q114), **no finding implements causal edges in the live system**. Q018–Q021 were proposed follow-ups from Q017 that were never filed as findings in this project. The Recall stack as built uses only CO_RETRIEVED_FRESH and CO_RETRIEVED_STALE relationship types in Neo4j — no CAUSES or CONTRADICTS edges exist.

**Conclusion on causal edges**: The CAUSES/CONTRADICTS types are an unimplemented design proposal. They do not constitute a Neo4j function the current system depends on. If causal edges are added in a future phase, PostgreSQL can host them via a `causal_edges` table with a `(from_memory_id, edge_type)` index — the weighted shortest-path traversal from Q017 maps to a recursive CTE or a bounded expansion loop in Python application code.

**Verdict on audit**: There is no remaining function that Neo4j provides which PostgreSQL cannot cover at Tim's graph scale through Month 36. The three conditions under which Neo4j becomes strictly necessary (depth > 3 BFS, > 50,000 edges, path-tracking queries) do not apply to the Recall architecture as designed. Q108 confirmed this at the physics level.

## Revised S6 Definition (PostgreSQL-based)

The original S6 signal (Q094, Q102) was defined as max degree in the CO_RETRIEVED_FRESH graph, measured via the S6-GAUGE Cypher query against Neo4j. With Neo4j eliminated, S6 becomes a set of PostgreSQL queries against the `co_retrievals` table.

### S6-GAUGE (replaces Cypher S6-GAUGE — run as Prometheus cron)

```sql
SELECT MAX(degree) AS max_fanout
FROM (
    SELECT from_memory_id, COUNT(*) AS degree
    FROM   co_retrievals
    WHERE  relationship_type = 'FRESH'
    GROUP  BY from_memory_id
) t;
```

Uses `idx_co_ret_fresh` partial index (Q108); reads only FRESH rows. Expected runtime: < 5ms at Month 36 scale (~22,000 edges, ~7,000 FRESH). This is **faster** than the Neo4j S6-GAUGE at equivalent scale because the partial index scan has lower per-record cost than Neo4j's chain-follow (Q107: Neo4j 2.20ms at 1,700x above physics floor; PostgreSQL CTE ~0.60ms).

### S6-THRESHOLDS (replaces Cypher S6-THRESHOLDS)

```sql
SELECT
    COUNT(*)                                                    AS nodes_with_edges,
    COUNT(*) FILTER (WHERE degree > 20)                         AS nodes_minor_hub,
    COUNT(*) FILTER (WHERE degree > 50)                         AS nodes_moderate_hub,
    COUNT(*) FILTER (WHERE degree > 100)                        AS nodes_s6_warning,
    COUNT(*) FILTER (WHERE degree > 158)                        AS nodes_timeout_risk,
    COUNT(*) FILTER (WHERE degree > 200)                        AS nodes_major_hub,
    MAX(degree)                                                 AS max_fanout,
    AVG(degree)                                                 AS mean_fanout
FROM (
    SELECT from_memory_id, COUNT(*) AS degree
    FROM   co_retrievals
    WHERE  relationship_type = 'FRESH'
    GROUP  BY from_memory_id
) t;
```

### S6-TOPHUBS (replaces Cypher S6-TOPHUBS — run on WARNING trigger)

```sql
SELECT
    cr.from_memory_id,
    LEFT(m.content, 80)  AS preview,
    COUNT(*)             AS degree
FROM   co_retrievals cr
JOIN   memories m ON cr.from_memory_id = m.id
WHERE  cr.relationship_type = 'FRESH'
GROUP  BY cr.from_memory_id, m.content
HAVING COUNT(*) > 20
ORDER  BY degree DESC
LIMIT  20;
```

### S6-HISTOGRAM (replaces Cypher S6-HISTOGRAM)

```sql
SELECT
    CASE
        WHEN degree BETWEEN 1  AND 5   THEN '01-05'
        WHEN degree BETWEEN 6  AND 20  THEN '06-20'
        WHEN degree BETWEEN 21 AND 50  THEN '21-50'
        WHEN degree BETWEEN 51 AND 100 THEN '51-100'
        WHEN degree BETWEEN 101 AND 158 THEN '101-158'
        WHEN degree BETWEEN 159 AND 200 THEN '159-200'
        WHEN degree BETWEEN 201 AND 500 THEN '201-500'
        ELSE '500+'
    END                 AS bucket,
    COUNT(*)            AS node_count,
    MAX(degree)         AS max_in_bucket
FROM (
    SELECT from_memory_id, COUNT(*) AS degree
    FROM   co_retrievals
    WHERE  relationship_type = 'FRESH'
    GROUP  BY from_memory_id
) t
WHERE degree > 0
GROUP BY bucket
ORDER BY bucket;
```

### Revised S6 Alert Thresholds

The alert ladder from Q102 is preserved unchanged — only the query backend changes from Neo4j to PostgreSQL:

| Level | max_fanout | Expected Month | Action |
|---|---|---|---|
| INFO | > 20 | Month 1.8 | Log first minor hub; no action needed |
| WATCH | > 50 | Month 3.3 | Enable weekly S6-HISTOGRAM scan |
| WARNING | > 100 | Month 5.2 | Add `LIMIT 50` inside depth-1 CTE level (hub guard from Q108) |
| CRITICAL | > 158 | Month 7.1 | Hub guard mandatory; depth-2 queries will fan out unacceptably without it |

**Key improvement from eliminating Neo4j**: The Q094 circuit breaker (asyncio.wait_for 8ms timeout, 3-consecutive-timeout Markov state machine, K=20 fallback, P(CB_open) = 1.9% at Month 12) is **entirely eliminated**. PostgreSQL CTEs with `LIMIT 50` inside the depth-1 CTE block provide deterministic hub protection that cannot fail: the planner applies the LIMIT during the index scan and never materializes more than 50 rows per seed. There is no timeout race, no Markov chain state, no degraded-K fallback path. Hub protection becomes structural rather than reactive.

### Store Responsibilities After Collapse

| Store | Primary Role | Secondary Role |
|---|---|---|
| **PostgreSQL** | Authoritative memory store (content, metadata, tags) | Graph store (co_retrievals table + partial indexes); health signal source (S6 queries) |
| **Qdrant** | ANN vector search per tenant (HNSW graph) | No secondary role; single-purpose |
| **Redis** | Per-tenant epoch clock (INCR, GET) | Pending queue coordination (S3 signal) |

**What Redis no longer needs to do**: Redis previously cached BFS traversal results and Neo4j connection pooling state. With co_retrievals in PostgreSQL, PostgreSQL's own `shared_buffers` serves as the warm cache for the co_retrievals partial index — no separate Redis BFS cache is required. This simplifies the Redis keyspace to epoch counters only.

## Multi-Tenant co_retrievals Schema

This is the final unified schema incorporating Q108 (CTE architecture) and Q109 (multi-tenant design), with Neo4j eliminated:

```sql
-- Primary memory store (unchanged from Q109 migration)
ALTER TABLE memories
    ADD COLUMN tenant_id VARCHAR(32) NOT NULL DEFAULT 'tim';

CREATE INDEX idx_memories_tenant_hash
    ON memories (tenant_id, content_hash);

-- Co-retrieval graph (replaces Neo4j entirely)
CREATE TABLE co_retrievals (
    from_memory_id      UUID NOT NULL REFERENCES memories(id) ON DELETE CASCADE,
    to_memory_id        UUID NOT NULL REFERENCES memories(id) ON DELETE CASCADE,
    tenant_id           VARCHAR(32) NOT NULL DEFAULT 'tim',
    relationship_type   VARCHAR(8)  NOT NULL CHECK (relationship_type IN ('FRESH','STALE')),
    last_reinforced_epoch INTEGER   NOT NULL,
    weight              FLOAT       NOT NULL DEFAULT 1.0,
    co_retrieval_count  INTEGER     NOT NULL DEFAULT 1,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    PRIMARY KEY (tenant_id, from_memory_id, to_memory_id)
);

-- Structural pre-filter: FRESH rows only per tenant
-- (analog to Neo4j TIM_CO_RETRIEVED_FRESH type chain from Q109)
CREATE INDEX CONCURRENTLY idx_co_ret_tim_fresh
    ON co_retrievals (from_memory_id)
    WHERE relationship_type = 'FRESH' AND tenant_id = 'tim';

CREATE INDEX CONCURRENTLY idx_co_ret_sadie_fresh
    ON co_retrievals (from_memory_id)
    WHERE relationship_type = 'FRESH' AND tenant_id = 'sadie';

-- Reverse direction for hub detection and S6 queries
CREATE INDEX CONCURRENTLY idx_co_ret_tim_fresh_to
    ON co_retrievals (to_memory_id)
    WHERE relationship_type = 'FRESH' AND tenant_id = 'tim';

-- STALE index for background reclassification job
CREATE INDEX CONCURRENTLY idx_co_ret_stale_epoch
    ON co_retrievals (last_reinforced_epoch)
    WHERE relationship_type = 'FRESH';  -- only FRESH rows eligible for STALE promotion

-- Depth-2 BFS query for Tim (production pattern)
WITH seeds AS (
    SELECT id FROM memories
    WHERE content_hash = ANY(:seed_hashes)
      AND tenant_id = 'tim'
),
depth1 AS (
    SELECT cr.to_memory_id AS memory_id,
           cr.weight,
           1 AS depth
    FROM   co_retrievals cr
    JOIN   seeds s ON cr.from_memory_id = s.id
    WHERE  cr.relationship_type = 'FRESH'
      AND  cr.tenant_id = 'tim'
    LIMIT  50   -- hub guard: replaces Q094 circuit breaker
),
depth2 AS (
    SELECT cr.to_memory_id AS memory_id,
           cr.weight * d1.weight AS weight,
           2 AS depth
    FROM   co_retrievals cr
    JOIN   depth1 d1 ON cr.from_memory_id = d1.memory_id
    WHERE  cr.relationship_type = 'FRESH'
      AND  cr.tenant_id = 'tim'
      AND  cr.to_memory_id NOT IN (SELECT id FROM seeds)
    LIMIT  50   -- hub guard at depth-2
)
SELECT memory_id, MAX(weight) AS weight, MIN(depth) AS depth
FROM   (SELECT * FROM depth1 UNION ALL SELECT * FROM depth2) combined
GROUP  BY memory_id
ORDER  BY weight DESC
LIMIT  :k;  -- k=38 for Tim, k=6 for Sadie

-- Reinforcement UPSERT (replaces Neo4j MERGE + Q096 + Q099)
INSERT INTO co_retrievals
    (from_memory_id, to_memory_id, tenant_id, relationship_type,
     last_reinforced_epoch, weight, co_retrieval_count)
VALUES (:from_id, :to_id, :tenant_id, 'FRESH', :epoch, 1.0, 1)
ON CONFLICT (tenant_id, from_memory_id, to_memory_id) DO UPDATE
    SET relationship_type    = 'FRESH',
        last_reinforced_epoch = EXCLUDED.last_reinforced_epoch,
        weight               = co_retrievals.weight + 0.1,
        co_retrieval_count   = co_retrievals.co_retrieval_count + 1;

-- Background STALE promotion (replaces Q096 reclassification job)
-- Run at session end, same trigger as epoch INCR
UPDATE co_retrievals
SET    relationship_type = 'STALE'
WHERE  relationship_type = 'FRESH'
  AND  tenant_id = :tenant_id
  AND  last_reinforced_epoch < (:current_epoch - :tau_edge);
```

### Schema Design Decisions

**PRIMARY KEY (tenant_id, from_memory_id, to_memory_id)**: Tenant-first ordering means all co-retrieval rows for 'tim' are physically adjacent on the heap. The BFS query for Tim reads only Tim's rows; Sadie's rows are in a different heap region. This is the PostgreSQL equivalent of Neo4j's separate relationship type chains per tenant (Q109 relationship-type-as-tenant-partition insight).

**ON DELETE CASCADE**: When a memory is deleted from the `memories` table, all its co_retrievals edges are automatically removed. Neo4j required explicit DELETE of relationship nodes before node deletion. This is a correctness improvement.

**co_retrieval_count**: Added as a first-class column. Neo4j had no clean mechanism to increment a property during MERGE without the READ_COMMITTED race condition analyzed in Q099. PostgreSQL's ON CONFLICT DO UPDATE is atomic — `co_retrieval_count + 1` is a single database operation.

## Operational Savings: RAM Freed by Removing Neo4j

| Resource | Before (five stores) | After (three stores) | Delta |
|---|---|---|---|
| Neo4j JVM heap | 4 GB (Community Edition default minimum for production stability) | 0 | **−4 GB freed** |
| Neo4j off-heap (page cache) | ~1–2 GB (Neo4j recommends matching page cache to store size; at ~50 MB store, JVM off-heap allocation is ~512 MB minimum) | 0 | **−0.5–2 GB freed** |
| Neo4j Docker container overhead | ~200–500 MB (JVM startup, thread stacks, class data) | 0 | **−200–500 MB freed** |
| co_retrievals table (PostgreSQL shared_buffers) | Already allocated; absorbs <1 MB for 22,000 edges at Month 36 | <1 MB | Negligible addition |
| **Total RAM freed** | — | — | **~4.7–6.5 GB freed on CasaOS** |

**Context**: CasaOS host has 32 GB DDR4. The Neo4j JVM heap consumes 12.5–20% of total CasaOS RAM for a workload that Q107 characterized as 1,700x above the physics floor (Neo4j BFS at 2.20ms vs 0.0013ms theoretical minimum). Freeing 4.7–6.5 GB is a material improvement that can be reallocated to:

- Qdrant shared memory for larger HNSW graphs (currently Qdrant uses < 50 MB for Tim's scale)
- Family Hub Docker services (speaker-id :10400, stt-whisper-http :10300)
- PostgreSQL shared_buffers increase (from default 128MB to 512MB or 1GB)
- OS file cache for Qdrant's on-disk segment files

**Docker compose simplification**: Removing the Neo4j service block eliminates one container (with its health check, volume mount, port binding, and environment variables for heap sizing). The `docker compose up` cold-start time for the Recall stack decreases by the Neo4j JVM warm-up time (typically 15–30 seconds for Community Edition).

## Health Signal Impact: S1–S6 After Neo4j Elimination

| Signal | Current source | After collapse | Impact |
|---|---|---|---|
| S1 (IPS P@3 delta) | PostgreSQL + Python | Unchanged | None |
| S2 (MBC Level 1 fraction) | Tag routing logic | Unchanged | None |
| S3 (Pending queue depth) | Filesystem + CasaOS | Unchanged | None |
| S4 (Epoch integrity) | Redis vs PostgreSQL | Unchanged | None |
| S5 (Graph traversal epoch compliance) | Neo4j BFS staleness | **Eliminated as a signal** | S5 no longer needed: PostgreSQL CTE is constrained by epoch in the WHERE clause on `memories`, not as a post-BFS filter. The epoch predicate is applied at query time structurally; there is no mechanism for stale-epoch memories to enter the result set. S5 becomes vacuously true. |
| S6 (Hub degree / max fanout) | Neo4j Cypher query | PostgreSQL SQL (revised above) | **Improved**: SQL S6-GAUGE is ~3.7x faster than Neo4j equivalent (0.60ms vs 2.20ms per Q107); deterministic hub guard via CTE LIMIT replaces probabilistic circuit breaker |

**S5 elimination rationale**: S5 was defined (Q084) as "fraction of 1-hop BFS results that have `last_access_epoch < current_epoch - tau`." In the Neo4j architecture, this could occur when the epoch predicate was missing from the Cypher query or when the Neo4j page cache held stale relationship records. In the PostgreSQL architecture, the BFS query explicitly JOINs `memories m ON cr.to_memory_id = m.id` — adding `AND m.last_access_epoch >= :epoch_threshold` to the depth-1 and depth-2 CTEs makes epoch compliance a hard constraint enforced by the planner. There is no separate monitoring needed for a constraint that the query planner enforces structurally.

**Net health signal set after collapse: S1, S2, S3, S4, S6** — five signals, down from six. All five are now sourced from PostgreSQL, Redis, or filesystem with no Neo4j dependency.

## Ideas Extracted

### Idea: three-store-stack-collapse
- **Novelty**: 0.82
- **Evidence**: 0.78
- **Feasibility**: 0.90
- **Description**: The complete elimination of Neo4j from the Recall stack — reducing from five stores (PostgreSQL + Neo4j + Qdrant + Redis + Ollama-as-embedding-service) to three (PostgreSQL + Qdrant + Redis) — is achievable with zero functional regression at Tim's graph scale through Month 36. The collapse is enabled by three Q108 insights working together: (1) PostgreSQL partial index is a structural analog to Neo4j's per-type relationship chain, (2) CTE LIMIT is a deterministic replacement for the Q094 circuit breaker Markov state machine, and (3) atomic UPSERT replaces the Q096/Q099 CREATE+DELETE race condition pattern. The resulting system has fewer moving parts, 4.7–6.5 GB freed RAM, one fewer Docker container, and no JVM startup delay. Implement via the Q108 migration path (2–4 hours) + the Q109 multi-tenant co_retrievals schema above.

### Idea: s5-structural-elimination
- **Novelty**: 0.72
- **Evidence**: 0.70
- **Feasibility**: 0.98
- **Description**: Health signal S5 (graph traversal epoch compliance) was required because Neo4j BFS had no mechanism to enforce epoch predicates structurally — the Cypher query could omit the WHERE clause, or a Neo4j cache could serve a stale relationship chain. By migrating to PostgreSQL CTEs, epoch compliance becomes a hard constraint in the WHERE clause of the CTE query itself (`AND m.last_access_epoch >= :epoch_threshold`). The query planner enforces it; no separate health signal is needed. S5 is eliminated from the monitoring stack. This is a case where architectural simplification makes a monitoring signal vacuously true — a stronger property than a signal that monitors a risk that still exists.

### Idea: pg-primary-key-tenant-ordering
- **Novelty**: 0.70
- **Evidence**: 0.65
- **Feasibility**: 0.92
- **Description**: Ordering the co_retrievals PRIMARY KEY as `(tenant_id, from_memory_id, to_memory_id)` — with tenant_id first — causes PostgreSQL's heap to store all Tim rows and all Sadie rows in separate physical regions of the table. This is the relational equivalent of Neo4j's separate relationship type chains per tenant (Q109): Tim's BFS query scans only Tim's physical heap region; Sadie's rows are never loaded into the I/O path. The effect is structural tenant isolation at the storage engine level, not just at the query predicate level. At two tenants and <50,000 total edges, the effect is modest (< 1ms difference); at higher tenant counts or larger graphs, it becomes significant. The design decision costs nothing and provides structural isolation as a free benefit of correct PK ordering.

### Idea: circuit-breaker-free-hub-protection
- **Novelty**: 0.75
- **Evidence**: 0.72
- **Feasibility**: 0.90
- **Description**: The Q094 hub node circuit breaker — a Markov chain state machine with 3-consecutive-timeout trigger, session-scoped OPEN state, asyncio.wait_for(8ms), and K=20 fallback — exists because Neo4j BFS has no built-in row cardinality cap. PostgreSQL CTE `LIMIT 50` inside the depth-1 block provides deterministic hub protection without any of this complexity: the planner applies the limit during the index scan and the query never materializes more than 50 × K_seeds rows at depth-1. Hub protection is structural and infallible rather than probabilistic and reactive. The 1.9% circuit-breaker open probability at Month 12 (Q094) drops to exactly 0% — the CTE LIMIT cannot fail to fire. This simplifies the application layer by eliminating approximately 80 lines of circuit breaker Python code and the associated asyncio timeout machinery.

### Idea: neo4j-jvm-ram-dividend
- **Novelty**: 0.68
- **Evidence**: 0.82
- **Feasibility**: 0.95
- **Description**: Reallocating the 4.7–6.5 GB freed by Neo4j elimination to PostgreSQL shared_buffers is the highest-value use of the dividend. Increasing shared_buffers from default 128MB to 1GB on the CasaOS N100 host would allow the entire `co_retrievals` table (< 3 MB at Month 36) and the `memories` table (< 5 MB at 10,000 rows) to remain permanently hot in PostgreSQL buffer cache, eliminating all cold-path NVMe accesses for the LLPC retrieval pipeline. Q107 identified the cold-path PG cost as 2.95ms vs 0.32ms warm. With 1GB shared_buffers, the PG phase of LLPC retrieval effectively becomes always-warm, reducing p95 LLPC latency by approximately 2.6ms. The RAM cost is 1GB of the 4.7–6.5GB dividend — leaving 3.7–5.5GB for other CasaOS services.
