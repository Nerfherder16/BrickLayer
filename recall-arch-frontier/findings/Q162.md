## Absence Classification

**Classification**: TRUE ABSENCE

Every audit_log reference in the production codebase falls into exactly one of three categories:

1. **INSERT** — `pg.log_audit()` is called by gc.py, hygiene.py, decay.py, reconcile.py, postgres_store.py, and the API layer. Every worker appends to the table; none remove from it.
2. **SELECT / aggregate** — postgres_store.py contains 20+ SELECT queries against audit_log for analytics, health metrics, dashboard data, feedback collection, and ML training input. These read the full history without any time filter constraint.
3. **COUNT** — API routes count rows for health checks and admin views.

No code path issues `DELETE`, `TRUNCATE`, `DROP PARTITION`, or any time-bounded purge against audit_log. The table has no PostgreSQL row TTL, no pg_cron job, no trigger-based expiry, and no application-level retention scheduler.

**Explanation**: The gap exists because audit_log was designed as an analytics and ML training source. The `reranker_trainer.py` and `signal_classifier_trainer.py` both draw training data from audit_log (`action="feedback"`, `action="signal-created"`), creating an implicit assumption that the full history must be preserved. No one introduced an explicit retention policy because the table's dual role — operational audit trail AND ML training corpus — made a blanket delete feel risky without a separate archive path. The result is indefinite accumulation with no exit.

## Minimum Viable Retention Policy

### Why 90 days is the correct horizon

- All meaningful debugging needs (session replay, decay analysis, GC audit trails) require at most 30 days of history.
- ML training data (feedback signals, signal-classifier labels) benefits from longer windows but the training jobs (`run_ml_retrain`, weekly on Sunday 4am) only fetch `limit=10000` rows — they sample, not scan the full history. Ninety days provides 198,000 rows as training candidates, well above any practical training corpus size.
- 90 days covers all current analytics queries (health metrics, dashboard 7/30-day windows, decay run history).

### Implementation

**SQL statement** (single atomic batch, safe on large tables):
```sql
DELETE FROM audit_log
WHERE timestamp < NOW() - INTERVAL '90 days';
```

Note: the timestamp column is named `timestamp` (not `created_at`) per the schema definition at `postgres_store.py` line 38. The existing B-tree index `idx_audit_log_timestamp` will serve this query efficiently once the table is at steady state. On the initial run (deleting ~200K rows), a sequential scan is likely faster than an index scan — PostgreSQL's planner will choose correctly.

**BRIN index recommendation**: The existing `idx_audit_log_timestamp` is a B-tree index. For a table that only ever appends in timestamp order, a BRIN index is 100-300x smaller and equally fast for range predicates. The migration to add it:
```sql
DROP INDEX IF EXISTS idx_audit_log_timestamp;
CREATE INDEX idx_audit_log_timestamp_brin ON audit_log USING BRIN (timestamp);
```
This should be run after the initial cleanup, not before (less overhead to rebuild on a smaller table).

**Worker integration** — wire into the existing daily 4am hygiene cron at `src/workers/hygiene.py`:

Add to `run_hygiene()` after the memory archive loop, before the final audit log entry:
```python
# Retention: purge audit_log entries older than 90 days
try:
    deleted = await pg.pool.execute(
        "DELETE FROM audit_log WHERE timestamp < NOW() - INTERVAL '90 days'"
    )
    logger.info("audit_log_retention_applied", deleted=deleted)
except Exception as e:
    logger.warning("audit_log_retention_failed", error=str(e))
```

Alternatively, add a standalone `run_audit_log_retention()` function to `gc.py` (which already handles other table-level cleanup operations) and register it as a separate cron at `5:15am` in `workers/main.py` (staggered 15 minutes after the existing GC run at 5:00am).

**Protective caveat for ML training**: The `reranker_trainer` and `signal_classifier_trainer` both use `limit=10000` when querying feedback from audit_log. Under a 90-day policy, ~198,000 rows remain as the steady-state training pool. This is larger than the current fetch limit, so no ML training quality regression is expected.

## Ideas Extracted

### Idea: audit-log-retention-gc-extension
- **Novelty**: 0.2 — straightforward time-bounded DELETE, no novel mechanism
- **Evidence**: 0.95 — absence confirmed by exhaustive grep across all Python files and schema
- **Feasibility**: 0.95 — single SQL statement, hookable into existing gc.py or hygiene.py with ~10 lines of code
- **Description**: Add a `DELETE FROM audit_log WHERE timestamp < NOW() - INTERVAL '90 days'` step to `gc.py` or `hygiene.py`. Register as a cron at 5:15am in `workers/main.py`. Add BRIN index on `timestamp` post-cleanup. Eliminates unbounded table growth and halves current table size on first run.

### Idea: tiered-audit-log-archival
- **Novelty**: 0.5 — separates the "hot" operational audit trail from the "cold" ML training corpus
- **Evidence**: 0.8 — the dual-use tension (audit trail vs. ML training source) is the root cause of why no one added a retention policy
- **Feasibility**: 0.7 — requires an `audit_log_archive` table and a move-then-delete operation instead of a simple delete
- **Description**: Create an `audit_log_archive` table (same schema) with PostgreSQL table partitioning by month. Move rows older than 90 days from `audit_log` to the archive before deletion, keeping the ML training corpus intact as a cold store. The reranker and signal classifier trainers would query both tables via a UNION view. This resolves the tension that likely caused the retention policy gap in the first place.
