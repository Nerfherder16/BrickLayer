## Evidence

### How Production Distributed DBs Handle Concurrent Writes

#### Cassandra: Last-Write-Wins (LWW) with Wall-Clock Timestamps

Cassandra uses wall-clock timestamps to determine the "winner" of concurrent writes to the same key. The write with the higher timestamp wins. LWW requires NTP clock synchronization across nodes — if clocks drift, the wrong write "wins."

**Properties**:
- No coordination required at write time (no locks, no consensus)
- O(1) conflict resolution (compare timestamps)
- Failure mode: clock skew causes incorrect winner selection; silent data loss
- Cassandra's solution: client-provided timestamps (the writer sets the time)

**Applicability to multi-agent memory**: LWW works for immutable memory content (once written, content doesn't change). For CO_RETRIEVED counters, LWW is catastrophic — Agent 1 writes count=5 at T1, Agent 2 writes count=3 at T2 > T1 (Agent 2's slower clock). LWW picks count=3 — loses Agent 1's increments silently.

**Verdict**: LWW is unsuitable for CO_RETRIEVED counters. Suitable for immutable memory content where "latest observed fact wins."

#### CockroachDB: Serializable Snapshot Isolation (SSI)

CockroachDB provides full ACID transactions across distributed nodes using Percolator-style distributed transactions with Raft consensus per range. Serializable isolation means concurrent conflicting writes are detected and one is aborted with retry.

**Properties**:
- Full serializability: no dirty reads, no lost updates, no write skew
- Conflict detection: write-write conflicts abort the slower transaction
- Overhead: Raft consensus adds 1-10ms per write (network round-trip dependent)
- Retry logic required: conflicted transactions must retry

**Applicability to multi-agent memory**: Too heavy for Recall 2.0's write path (41ms already; adding 10ms Raft would exceed budget). Valuable pattern: the **optimistic retry loop** (write, detect conflict, retry) is applicable without full distributed consensus.

**Verdict**: Overkill for single-host LMDB. The retry pattern is the transferable concept.

#### DynamoDB: Conditional Writes + Optimistic Locking

DynamoDB provides conditional writes: `PutItem(condition: attribute_not_exists(id) OR version = :expected_version)`. If the condition fails, write is rejected. Client implements optimistic locking: read current version → modify → write with `version = old_version + 1` → if rejected, re-read and retry.

**Properties**:
- No coordination at the DB layer (stateless check at write time)
- O(1) conflict detection per write attempt
- Retry adds one extra read + write per conflict
- Works for counter types using `ADD count 1` (DynamoDB atomic increment)

**Applicability to multi-agent memory**: DynamoDB's `ADD count 1` is the exact semantic needed for CO_RETRIEVED edge increments. This is equivalent to LMDB's DUPSORT-based append or Redis INCR — atomic counter increment without read-modify-write race.

**Verdict**: Atomic increment semantics are directly applicable. DynamoDB's `ADD` maps to LMDB's approach of treating the CO_RETRIEVED edge as an atomic counter.

### CRDT Literature: G-Counter and PN-Counter

**G-Counter (Grow-only Counter)**:
- Each node maintains a vector of counts: `[count_node1, count_node2, ...]`
- Increment: `counts[this_node] += 1`
- Merge: `max(counts_A[i], counts_B[i])` for each i
- Total: `sum(counts[i])`
- Eventual consistency: any merge ordering produces the same total

**PN-Counter (Positive-Negative Counter)**:
- Two G-Counters: one for increments (P), one for decrements (N)
- Total: `sum(P) - sum(N)`

**Applicability to CO_RETRIEVED edges**:
A G-Counter per (memory_A, memory_B) pair would track co-retrieval counts correctly across multiple agents and machines. Merge is commutative and associative — any order of agent write merges produces the same final count. No coordinator needed.

**The Q209 task_id normalization and G-Counter**:
These are complementary. G-Counter solves "how to merge counts from N agents without losing increments." Task_id normalization solves "how to count unique tasks, not raw events." A task-aware G-Counter would track `{task_id: count_by_agent}` per pair — merge = union of task_ids seen × max count per task. The total is the number of unique tasks that co-retrieved the pair, regardless of how many agents within each task triggered the increment.

**Verdict**: G-Counter semantics are directly applicable to CO_RETRIEVED edge updates. Combined with task_id normalization from Q209, a task-aware G-Counter perfectly models the desired behavioral signal.

### Minimum Coordination Overhead Analysis

| Model | Coordination Cost per Write | Consistency Guarantee | Failure Mode |
|---|---|---|---|
| LWW (Cassandra) | Zero (timestamps only) | Eventual (clock-dependent) | Silent data loss on skew |
| Serializable (CockroachDB) | 1-10ms (Raft round-trip) | Full ACID | Retry on conflict |
| Optimistic lock (DynamoDB) | 0 usually, 1 RTT on conflict | Linearizable per key | Retry on conflict |
| CRDT G-Counter | Zero (merge on read) | Eventual (merge semantics) | None (merge is idempotent) |
| LMDB single-writer (current) | 0 (local queue) | Sequential (ordered writes) | Queue latency at high concurrency |

**Minimum coordination overhead for eventual consistency**: **CRDT G-Counter** — zero coordination cost per write, zero coordination cost at merge. Writes always succeed. Merges are always correct. For a single-host LMDB deployment, CRDT merge is unnecessary (LMDB serialization already provides correct counts). For multi-machine Recall (future SaaS, multi-node), G-Counter becomes the coordination-free solution.

### Recommended Model for Multi-Agent Memory Writes

**For Recall 2.0 (single-host LMDB)**:
Current LMDB single-writer serialization is correct and sufficient. No additional coordination needed. Add task_id normalization (Q209) to prevent semantic inflation. No CRDT overhead required.

**For Recall 2.0 multi-node future (SaaS with 2+ Recall instances)**:
G-Counter per CO_RETRIEVED pair, implemented as: each Recall instance tracks its own increment counts; merge happens at read time or periodic sync. Specific data structure: `HashMap<(MemId, MemId), HashMap<NodeId, u64>>` — merge by taking max per NodeId. This is a standard 2D G-Counter.

**For immutable memory content (not counters)**:
LWW with causal timestamps (Lamport clocks from Q041, not wall-clock) is correct. Lamport clocks are already designed for this: increment on every write, `max(local, received) + 1` on receive. This was the Q041 recommendation and remains correct.

**For deduplication across nodes**:
SHA-256 L0 dedup + SimHash L1 dedup (Q197) already handle this. Content-addressed dedup doesn't need distributed consensus — it's stateless (compute hash, check existence, reject if found).

## Ideas Extracted

### Idea: g-counter-per-co-retrieved-pair-for-multi-node-recall
- **Novelty**: 0.65 — Applying CRDT G-Counter semantics to CO_RETRIEVED edge counting is a direct and natural extension of the existing design. The novelty is the specific data structure for multi-node Recall: `HashMap<(MemId, MemId), HashMap<NodeId, u64>>` as the distributed CO_RETRIEVED store. This is not a new concept in distributed systems but is novel in its application to AI memory behavioral scoring.
- **Evidence**: 0.85 — G-Counter CRDT is proven in production: Riak, Redis CRDT, Cassandra counter columns. Merge semantics: commutative, associative, idempotent — any order of merges produces the same result. For CO_RETRIEVED: merge(node1_counts, node2_counts) = max-merge per node entry. Total = sum across all node entries per pair. Correct regardless of network partition, message reordering, or node failure.
- **Feasibility**: 0.70 — For current single-host Recall: not needed (LMDB serialization is sufficient, simpler). For future multi-node SaaS: medium implementation cost. G-Counter per pair stored in LMDB as a CBOR-serialized map. Merge on read (any node can merge and return the aggregate). This becomes relevant when Recall has 2+ instances writing the same user's behavioral graph — which is the SaaS multi-region case.
- **Description**: For multi-node Recall deployments (SaaS multi-region or dual-write architectures), CO_RETRIEVED edge counters should use CRDT G-Counter semantics: `HashMap<(MemId, MemId), HashMap<NodeId, u64>>`. Each Recall instance increments only its own NodeId bucket. Merge = max-merge per NodeId (standard G-Counter). Total = sum across buckets. Combined with Q209's task_id normalization: the NodeId-bucketed counter tracks task-normalized increments (each entry: NodeId → count of unique tasks that co-retrieved the pair on that node). Merge produces globally correct unique-task count. Zero coordination cost at write time; consistency achieved at merge (read-time or periodic sync).

### Idea: lamport-clocks-for-immutable-memory-content-lww-with-causal-ordering
- **Novelty**: 0.50 — Q041 already recommended Lamport clocks for multi-machine write conflict resolution. This finding confirms and extends that recommendation: Lamport clocks are the correct timestamp mechanism for LWW on immutable memory content across distributed Recall instances. Wall-clock LWW (Cassandra default) is incorrect due to clock skew risk.
- **Evidence**: 0.90 — Lamport clock guarantee: if event A causally precedes event B, then timestamp(A) < timestamp(B). For memory content: if Agent 1 writes "Service X uses auth Y" at Lamport time 5 and Agent 2 overwrites with "Service X uses auth Z" at Lamport time 3 (due to clock skew), Lamport LWW correctly rejects the overwrite. Wall-clock LWW would accept it. The 64-bit record overhead (Q041) is the space cost.
- **Feasibility**: 0.80 — Implementation: one `lamport_ts: u64` field per memory record (Q041 design). Already in scope for Recall 2.0's multi-machine write conflict design. No change to single-host LMDB behavior (Lamport clocks degenerate to monotonic counters in single-node case).
- **Description**: For LWW conflict resolution on immutable memory content in multi-node Recall: use Lamport clocks (causal ordering) not wall-clock timestamps (clock skew vulnerable). Q041 confirmed 64 bits per record. Lamport LWW guarantees: the causally later write wins, independent of actual wall-clock time. In the single-host LMDB case: Lamport clocks degenerate to a monotonic counter (increment on every write), which is already LMDB's write sequence. No code change for single-host; the architecture is multi-node ready when the Lamport counter field is added.

## Analytical Summary

**The G-Counter CRDT is the correct data structure for CO_RETRIEVED edge counters in multi-node Recall.** It provides zero-coordination-cost writes with correct eventual consistency. For the current single-host LMDB deployment, this is unnecessary complexity — LMDB serialization already provides sequential correctness.

**The coordination model recommendation for Recall 2.0 phases**:
- MVP (single host): LMDB serialization + task_id normalization (Q209) — no CRDT overhead
- v2.1 (SaaS multi-instance): G-Counter per CO_RETRIEVED pair, Lamport LWW for content
- v3.0 (multi-region): Full G-Counter merge with geo-distributed nodes

**DynamoDB's `ADD count 1` pattern** (atomic increment without read-modify-write) is the most directly transferable concept. LMDB's write transaction provides equivalent atomic increment semantics for the single-host case.
