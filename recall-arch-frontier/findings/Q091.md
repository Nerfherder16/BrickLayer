# Finding: Q091 — Neo4j Relationship Property Index Cannot Pre-Filter BFS Edges; Materialized Fresh-Edge Subgraph Is the Only True Pre-Traversal Architecture

**Question**: Does creating a Neo4j relationship property index on `r.last_reinforced_epoch` for the `CO_RETRIEVED` relationship type enable pre-traversal edge filtering — reducing effective BFS fanout before node evaluation — and what is the theoretical speedup compared to the post-filter architecture confirmed in Q083?
**Question Type**: PHYSICS
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Neo4j 5.x relationship index architecture, graph database execution planning, BFS traversal cost models, temporal graph filtering

---

## Operation Definition

**Operation: Epoch-filtered BFS traversal from a seed Memory node**

```cypher
MATCH (seed:Memory {id: $seed_id})-[r:CO_RETRIEVED]-(neighbor:Memory)
WHERE r.last_reinforced_epoch >= ($current_epoch - $tau_edge)
  AND neighbor.last_access_epoch >= ($current_epoch - $tau)
RETURN neighbor.id LIMIT 50
```

- **Input**: One seed Memory node ID (via unique constraint), current epoch integer, tau_edge=30 (edge staleness window), tau=20 (node staleness window)
- **Output**: Up to 50 neighbor Memory node IDs whose causal link was reinforced within the last tau_edge epochs AND whose node was accessed within the last tau epochs
- **Guarantees required**: Correctness (only return neighbors satisfying both predicates), latency bound (within 10ms graph sub-budget at Tim's scale)
- **The specific question**: Does `CREATE INDEX FOR ()-[r:CO_RETRIEVED]-() ON (r.last_reinforced_epoch)` cause the Neo4j query planner to skip stale edges before following them to neighbor nodes — reducing the number of relationship records read from F_total to F_fresh — or does it still traverse all F_total edges and evaluate the predicate as a post-filter on each?

## Physics Ceiling: Pre-Traversal Edge Filtering

### Theoretical minimum: if true pre-filter existed

If Neo4j could use a relationship property index to pre-filter edges before following them, the traversal cost model would shift from:

```
O(F_total): read all F relationship records, filter on r.last_reinforced_epoch, discard stale ones
```

to:

```
O(F_fresh × log F_total): index probe to identify fresh edges, follow only those edges to neighbor nodes
```

At Month 12 hub nodes (F_total = 500–1,500, F_fresh = ~10% with tau_edge=30):

| Parameter | Full-fanout traversal | Hypothetical pre-filter |
|-----------|-----------------------|------------------------|
| Relationship records read | F_total = 500–1,500 | F_fresh = 50–150 |
| Property reads (r.last_reinforced_epoch) | F_total | 0 (already indexed) |
| Neighbor node loads | F_total | F_fresh |
| Traversal cost at 0.03ms/record | 15–45ms | 1.5–4.5ms |
| Index lookup overhead | 0 | ~0.1ms fixed |
| **Total cost** | **15–45ms** | **~1.6–4.6ms** |
| **Theoretical speedup** | 1× (baseline) | **~10×** |

This 10× speedup is the theoretical physics ceiling if a relationship property index could pre-filter edges. The question is whether Neo4j's execution model actually achieves this.

## Neo4j Relationship Property Index Architecture

### What Neo4j relationship property indexes actually do

Neo4j 5.x introduced relationship property indexes via:

```cypher
CREATE INDEX rel_epoch_idx FOR ()-[r:CO_RETRIEVED]-() ON (r.last_reinforced_epoch)
```

This creates a B-tree (or range) index over the `last_reinforced_epoch` property values across all CO_RETRIEVED relationships in the graph. The index structure maps: `last_reinforced_epoch value → relationship record IDs`.

**The critical architectural constraint**: This index is an **anchor index**, not a traversal filter. It enables an access pattern where the query engine can start from a relationship property value range to find candidate relationships — but this is an alternative entry point to the graph, not a filter applied during traversal from a node anchor.

**Two distinct execution plan architectures:**

**Plan A — Relationship anchor (index usable):**
```
Starting point: r.last_reinforced_epoch >= (current_epoch - tau_edge)
Access path: RelationshipIndexSeek → follow endpoints → filter node predicates
Use case: "Find all CO_RETRIEVED relationships reinforced in the last 30 epochs"
```

**Plan B — Node anchor with relationship traversal (index NOT usable for pre-filter):**
```
Starting point: seed node {id: $seed_id}
Access path: NodeUniqueIndexSeek → Expand(All) CO_RETRIEVED chains → Filter on r.last_reinforced_epoch
Use case: "Starting from this specific node, find neighbors via CO_RETRIEVED, filtered by edge epoch"
```

The BFS pattern used by Recall is **Plan B**. The seed anchor is a specific node (unique ID lookup). From that anchor, the engine must follow the CO_RETRIEVED relationship chain. The relationship chain is a linked-list data structure in Neo4j's relationship store — the engine traverses it record by record. It cannot consult the relationship property index during this traversal because the index is organized by property value, not by position in any particular node's relationship chain.

### Why the index cannot filter during chain traversal

Neo4j's relationship store layout (per Neo4j internals documentation):

```
Relationship record (34 bytes):
  - Relationship ID (8 bytes)
  - Start node pointer (8 bytes)
  - End node pointer (8 bytes)
  - Relationship type (4 bytes)
  - Next relationship in start-node chain (pointer)
  - Next relationship in end-node chain (pointer)
  - Property pointer (to property store chain)
```

The relationship chain from a node is a linked list traversed pointer-to-pointer. To evaluate `r.last_reinforced_epoch >= X` during this traversal, the engine must:

1. Load the relationship record (pointer dereference into relationship store)
2. Follow the property pointer to the property store chain
3. Scan the property chain to find `last_reinforced_epoch`
4. Evaluate the predicate

The property index is organized as: `{value → [rel_record_id_1, rel_record_id_2, ...]}`. To use the index here, the engine would need to:

1. Query the index for `last_reinforced_epoch >= (current_epoch - tau_edge)` → get a list of relationship IDs
2. Intersect that list with the relationship IDs in the seed node's CO_RETRIEVED chain

This intersection is a set join between two collections: the index result set (all fresh CO_RETRIEVED relationships, potentially millions across the entire graph) and the seed node's relationship chain (F_total relationships). Neo4j's execution planner does not implement this join strategy for BFS traversals anchored on a specific node. The intersection would require either:
- A hash join (materialize both sides → expensive for large graphs)
- Sorting both sides and merging (requires full chain traversal anyway)

Neither approach provides a speedup over simply traversing the chain and checking the property directly per record. This is why the planner does not use the relationship property index in Plan B queries — it is not architecturally advantageous.

### Confirming via EXPLAIN plan

The expected execution plan for the BFS query with `r.last_reinforced_epoch` filter:

```
+--------------------------------+------------------+----------+
| Operator                       | Estimated Rows   | DB Hits  |
+--------------------------------+------------------+----------+
| +ProduceResults                | 50               | 0        |
| +Limit                         | 50               | 0        |
| +Filter (node epoch predicate) | F*sel_node       | F × prop_reads |
| +Filter (edge epoch predicate) | F_total          | F_total × prop_reads |
| +Expand(All) CO_RETRIEVED      | F_total          | F_total (rel reads) |
| +NodeUniqueIndexSeek (id)      | 1                | 2        |
+--------------------------------+------------------+----------+
```

The `Filter (edge epoch predicate)` operator appears AFTER `Expand(All)` — it is a post-traversal filter applied to each relationship record after it has been read. The relationship property index (`rel_epoch_idx`) does NOT appear in this plan. The `Expand(All)` reads all F_total relationship records from the chain; the `Filter` then evaluates `r.last_reinforced_epoch` by loading each relationship's property chain.

**This is the same architecture as Q083's finding for node property indexes**: the index is irrelevant to BFS traversal from a specific node anchor. Q083 established this for node property indexes; Q091 extends the finding to relationship property indexes. The result is identical: the index cannot pre-filter during BFS traversal.

## Cost Analysis: Current Post-Filter Architecture

### Step-by-step traversal cost with both predicates

Starting from the Q083 cost model and extending for the edge predicate:

```
Total cost = anchor lookup
           + F_total × (relationship record load)
           + F_total × (edge property chain load for r.last_reinforced_epoch)
           + F_fresh × (node property chain load for neighbor.last_access_epoch)
           + sort(F_passing_both_filters)
           + LIMIT overhead
```

Where:
- F_total = total CO_RETRIEVED edges on the seed node
- F_fresh = edges passing the edge epoch filter (r.last_reinforced_epoch >= current_epoch - tau_edge)
- F_passing = edges also passing the node epoch filter (neighbor.last_access_epoch >= current_epoch - tau)

**Important**: The two filter predicates are applied sequentially by the planner, not simultaneously. The planner evaluates the edge predicate first (property read on the relationship record), then the node predicate (property read on the neighbor node) — but it must load the relationship record regardless before it can read the edge property. There is no short-circuit that avoids loading a relationship record.

| Hub node scenario | F_total | F_fresh (tau_edge=30, ~10% fresh) | F_passing (tau=20, ~8% pass both) | Traversal cost | Within 10ms? |
|-------------------|---------|-----------------------------------|-----------------------------------|----------------|--------------|
| Month 6, median hub | 50 | 5 | 4 | 0.1ms + 50×0.05ms = 2.6ms | Yes |
| Month 12, median hub | 150 | 15 | 12 | 0.1ms + 150×0.05ms = 7.6ms | Yes (marginal) |
| Month 12, top-1% hub | 500 | 50 | 40 | 0.1ms + 500×0.05ms = 25ms | **Exceeds** |
| Month 24, top-1% hub | 1,500 | 150 | 120 | 0.1ms + 1500×0.05ms = 75ms | **Far exceeds** |

Per-edge cost breakdown (0.05ms/edge with both predicates):
- Relationship record load: 0.01ms
- Edge property chain load (last_reinforced_epoch): 0.02ms
- Node property chain load (last_access_epoch, loaded only for F_total edges): 0.02ms

**Note**: The node property load is NOT short-circuited after the edge filter in typical Neo4j execution — the Filter operators are chained, so all F_total edges flow through the first filter (edge epoch) before reaching the second filter (node epoch). The node property load occurs for every edge, not just for the F_fresh edges that pass the edge filter. This is a subtle but important cost: loading neighbor.last_access_epoch for edges that are already disqualified by the edge epoch predicate.

## Gap Analysis

### Theoretical minimum vs actual

| Resource | Theoretical minimum (pre-filter) | Actual cost (post-filter) | Gap |
|----------|----------------------------------|---------------------------|-----|
| Relationship records read | F_fresh (index-selected) | F_total | F_total / F_fresh = ~10× |
| Edge property reads | 0 (index provides value) | F_total | F_total reads wasted |
| Node property reads | F_fresh | F_total (all edges, before node filter) | F_total / F_fresh = ~10× |
| Latency (Month 12, 500-edge hub) | ~1.6ms | ~25ms | ~15× |
| Latency (Month 24, 1500-edge hub) | ~4.6ms | ~75ms | ~16× |

**Gap ratio: approximately 15× for hub nodes at Month 12, growing as graph matures.**

### Physics gaps (unavoidable)

**Chain traversal as the BFS mechanism**: Neo4j's relationship chain linked-list structure means that finding a specific node's neighbors requires sequential traversal of the chain. There is no B-tree over "which relationships belong to node X" — the chain pointer is the only access path. This is a fundamental architectural choice in Neo4j's storage engine, not a software bug. Alternative graph databases (e.g., Amazon Neptune using OpenCypher, or TigerGraph using adjacency list indexes) have different storage layouts, but within Neo4j Community Edition, chain traversal is irreducible.

**Property store access per predicate evaluation**: Each property predicate requires loading the property chain for that record (relationship or node). This is one NVMe read per property chain access (cache-cold) or near-zero (cache-warm). At cache-warm, this is the dominant physics constraint — the property store reads cannot be eliminated without changing the storage format.

### Architecture gaps (design choices)

**Gap 1 — Full relationship chain traversal before any epoch filtering**: The engine reads all F_total relationship records from the chain before evaluating any predicate. This is an architecture gap: the chain contains both fresh and stale edges, but there is no structural way to skip stale edges without reading them first.

- What causes it: Neo4j's relationship chain is sorted by insertion order, not by relationship property values. Stale edges (old `last_reinforced_epoch`) are interleaved with fresh edges throughout the chain.
- What would remove it: If the relationship chain were sorted by `last_reinforced_epoch` (descending), the traversal could stop early once it reaches edges with `last_reinforced_epoch < (current_epoch - tau_edge)`. This is the "skip-list / sorted chain" architecture gap.

**Gap 2 — Node property loaded for all edges, not just epoch-passing edges**: The planner loads `neighbor.last_access_epoch` for every relationship record, not only the ones passing the edge epoch filter. This wastes ~90% of node property reads at tau_edge=30 (10% edge survival rate).

- What causes it: Neo4j's query planner does not reorder filters based on selectivity in BFS traversal chains. The `Filter` operators apply to every record flowing through the pipeline, regardless of whether the previous filter has already been applied to reduce the set.
- What would remove it: Short-circuit evaluation where the node property is only loaded after the edge epoch filter passes. This is a planner optimization gap, not a storage architecture gap.

**Gap 3 — Absence of a materialized fresh-edge subgraph**: No pre-computed structure exists that contains only the fresh CO_RETRIEVED edges (those with `last_reinforced_epoch >= current_epoch - tau_edge`). Such a structure would make BFS traversal operate only on fresh edges, achieving the theoretical minimum.

- What causes it: The current architecture stores all CO_RETRIEVED edges in a single relationship type, with staleness determined at query time. There is no proactive partitioning by freshness.
- What would remove it: Maintaining two relationship types — `CO_RETRIEVED_FRESH` and `CO_RETRIEVED_STALE` — with periodic reclassification as edges age out of the tau_edge window. Traversal would only follow `CO_RETRIEVED_FRESH` edges, achieving O(F_fresh) chain reads.

### Implementation gaps (code quality)

**Gap 4 — No LIMIT applied before property reads**: The current query applies LIMIT 50 after both filters, but the planner still reads all F_total edges' properties before applying LIMIT. Adding an intermediate cap (e.g., reading only the top K edges by `r.co_occurrence_count DESC` before evaluating epoch predicates) requires Cypher query restructuring, not just a configuration change.

## Architectural Path to Pre-Traversal Edge Filtering

### Option 1: Relationship type partitioning (materialized fresh-edge subgraph)

Maintain two CO_RETRIEVED relationship subtypes:

```
CO_RETRIEVED_FRESH: r.last_reinforced_epoch >= (current_epoch - tau_edge)
CO_RETRIEVED_STALE: r.last_reinforced_epoch < (current_epoch - tau_edge)
```

BFS traversal only follows `CO_RETRIEVED_FRESH` edges:

```cypher
MATCH (seed:Memory {id: $seed_id})-[r:CO_RETRIEVED_FRESH]-(neighbor:Memory)
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)
RETURN neighbor.id LIMIT 50
```

Neo4j organizes relationships by type — each node has separate chains per relationship type. Traversing `CO_RETRIEVED_FRESH` only reads the fresh chain, which has F_fresh records. This achieves the theoretical minimum traversal cost.

**Maintenance overhead**: At each session end, when co-retrieval pairs are reinforced:
1. For each reinforced pair: if `CO_RETRIEVED_STALE` exists between them, delete it and create `CO_RETRIEVED_FRESH`
2. Periodic background job: for each `CO_RETRIEVED_FRESH` edge where `last_reinforced_epoch < (current_epoch - tau_edge)`, reclassify to `CO_RETRIEVED_STALE`

The reclassification job runs at the epoch boundary. With tau_edge=30 and ~45 new co-retrieval events/session, the job reclassifies approximately 45 edges per epoch (those that were fresh 30 sessions ago and have not been reinforced since). At 800 edges/month total, this is a modest maintenance load.

**Cost of reclassification**: For each edge to reclassify, the Cypher is:
```cypher
MATCH (a:Memory)-[r:CO_RETRIEVED_FRESH {id: $edge_id}]-(b:Memory)
WHERE r.last_reinforced_epoch < ($current_epoch - $tau_edge)
MERGE (a)-[r2:CO_RETRIEVED_STALE]-(b)
SET r2 = properties(r)
DELETE r
```

This is one relationship delete + one relationship create per stale edge. Neo4j relationship creates/deletes cost approximately 0.5–2ms each (WAL write + relationship store update). At 45 reclassifications per epoch-boundary event, total cost is ~22–90ms per reclassification run. At one run per session (assuming epoch boundaries are per-session), this is acceptable within the background processing budget.

**Traversal speedup with type partitioning**:

| Scenario | F_fresh (CO_RETRIEVED_FRESH chain length) | Traversal cost | Speedup vs full chain |
|----------|------------------------------------------|----------------|----------------------|
| Month 12, 500-edge hub, tau_edge=30 | 50 | 0.1ms + 50×0.03ms = 1.6ms | **15.6× speedup** |
| Month 24, 1500-edge hub, tau_edge=30 | 150 | 0.1ms + 150×0.03ms = 4.6ms | **16.3× speedup** |

This matches the theoretical physics ceiling within ~10%.

### Option 2: Sorted relationship chain (requires Neo4j Enterprise or custom storage)

If the CO_RETRIEVED relationship chain were sorted by `last_reinforced_epoch` descending, the BFS traversal could stop early:

```
Read relationship records from chain (sorted by last_reinforced_epoch DESC)
Stop when r.last_reinforced_epoch < (current_epoch - tau_edge)
```

This would also achieve O(F_fresh) reads with O(F_total) worst case only when all edges are fresh. However, Neo4j Community Edition does not support sorted relationship chains — chain order is insertion order, not property-sorted. This option requires a custom storage engine modification or a different graph database.

### Option 3: Cypher WITH clause to stage filtering

A Cypher query rewrite that limits edge traversal before loading node properties:

```cypher
MATCH (seed:Memory {id: $seed_id})-[r:CO_RETRIEVED]-(neighbor:Memory)
WHERE r.last_reinforced_epoch >= ($current_epoch - $tau_edge)
WITH r, neighbor
WHERE neighbor.last_access_epoch >= ($current_epoch - $tau)
RETURN neighbor.id LIMIT 50
```

**Does this reduce cost?** The `WITH r, neighbor` clause between the two WHERE conditions introduces a pipelining boundary that may allow the planner to evaluate the edge predicate before loading the node predicate — but this depends on the Neo4j version's cost-based optimizer behavior. In Neo4j 5.x, the optimizer may or may not reorder filters across a `WITH` boundary. The relationship chain still reads all F_total records. The gain is potentially short-circuiting node property reads for edges that fail the edge epoch filter (loading `neighbor.last_access_epoch` only for the ~10% of edges that pass). This reduces the node property read overhead by ~90% at tau_edge=30, but does NOT reduce relationship chain traversal.

**Estimated savings from Option 3 (property read reduction only)**:

| Fanout | Without clause staging | With clause staging | Savings |
|--------|------------------------|--------------------|----|
| 500 (hub) | 500 × 0.05ms = 25ms | 500 × 0.01ms (rel) + 500 × 0.02ms (edge prop) + 50 × 0.02ms (node prop) = 16ms | ~36% |
| 1500 (hub) | 1500 × 0.05ms = 75ms | 1500 × 0.03ms + 150 × 0.02ms = 48ms | ~36% |

Option 3 saves approximately 35–40% of traversal cost but does not achieve the 10–15× theoretical speedup. It is an implementation optimization, not an architecture change.

## Ideas Extracted

### Idea: co-retrieved-fresh-stale-type-partitioning
- **Novelty**: 0.82 — Using relationship type as a freshness partition key in Neo4j — maintaining `CO_RETRIEVED_FRESH` and `CO_RETRIEVED_STALE` as distinct types, reclassified at epoch boundaries — is not found in reviewed AI memory system literature. Graphiti uses edge timestamps but traverses all edges and filters in Python post-retrieval. MemGPT's graph layer does not perform temporal edge filtering. The key insight is that Neo4j organizes relationship chains per-type, so type-based partitioning achieves structural pre-filtering that property indexes cannot. The reclassification is a novel use of relationship type as a semantic freshness label — using schema evolution (type transitions) as a temporal filter mechanism rather than query predicates.
- **Evidence**: 0.72 — Neo4j's per-type relationship chain organization is documented in the Neo4j 5.x storage engine internals. The consequence (traversing `CO_RETRIEVED_FRESH` only reads the fresh sub-chain) follows directly from this architecture. Temporal partitioning of relationship types is used in some Neo4j use cases (e.g., event sourcing patterns that partition by time window), but not in the specific context of AI memory system BFS-with-epoch-filter. The reclassification overhead at Tim's scale (~45 edges/session, ~22–90ms) is estimated from Neo4j WAL write benchmarks.
- **Feasibility**: 0.65 — Requires schema migration (creating the `CO_RETRIEVED_STALE` relationship type alongside `CO_RETRIEVED_FRESH`), a background reclassification job (runs at session end), and updates to all Cypher queries that currently reference `CO_RETRIEVED`. The migration is non-trivial: existing CO_RETRIEVED edges must be audited and reclassified based on their current `last_reinforced_epoch`. The background job must be fault-tolerant (if it fails, edges accumulate in the FRESH chain without expiration). On the upside, no new infrastructure is required — it is pure Neo4j schema and Cypher changes.
- **Description**: Replace the single `CO_RETRIEVED` relationship type with two types: `CO_RETRIEVED_FRESH` (edges with `last_reinforced_epoch >= current_epoch - tau_edge`) and `CO_RETRIEVED_STALE` (older edges). All new co-retrieval edges are written as `CO_RETRIEVED_FRESH`. At each session end, a background Cypher batch reclassifies edges whose `last_reinforced_epoch < (current_epoch - tau_edge)` from FRESH to STALE. The traversal Cypher only follows `CO_RETRIEVED_FRESH`, achieving O(F_fresh) chain traversal. Estimated speedup at Month 12 hub nodes: 15× vs the current dual-predicate post-filter. This converts the tau_edge predicate from a post-filter property read into a structural pre-filter — the only path to pre-traversal edge filtering within Neo4j Community Edition.

### Idea: cypher-with-clause-edge-predicate-staging
- **Novelty**: 0.68 — Staging the edge epoch predicate before the node epoch predicate using a Cypher `WITH` boundary to enable potential short-circuit node property loading is a query rewrite technique. The novelty is in applying it specifically to the Recall dual-predicate BFS pattern, where the edge filter's selectivity (~10% pass rate at tau_edge=30) makes it a highly effective first-stage filter before the more expensive node property load. This is a known Cypher optimization technique but its application to the specific Recall epoch-filter pattern is not documented.
- **Evidence**: 0.62 — Cypher query planning behavior around `WITH` clause pipelining varies between Neo4j versions. Neo4j 5.x's cost-based optimizer documentation notes that `WITH` introduces a result materialization boundary that can affect filter ordering. Empirical verification on the live instance at 192.168.50.19 is required to confirm whether the optimizer short-circuits node property loads after the edge filter stage. The theoretical savings (~36%) are computable if the optimizer behaves as expected.
- **Feasibility**: 0.90 — A single Cypher query rewrite with no schema changes, no background jobs, and no migration. The change is: add `WITH r, neighbor` between the edge filter and the node filter. Risk: optimizer may not respect the intended filter ordering, producing no benefit. Can be verified with EXPLAIN/PROFILE on the live instance.
- **Description**: Rewrite the dual-predicate BFS query to stage the edge epoch filter before the node epoch filter using a Cypher `WITH` clause, exploiting the ~10× selectivity advantage of the edge filter (only 10% of edges pass at tau_edge=30) to potentially short-circuit the more expensive node property load for the 90% of edges that fail. Expected savings: 35–40% of traversal cost with zero infrastructure changes. This is the minimal-effort optimization path, distinct from the type-partitioning approach (which achieves 15× but requires schema migration).

### Idea: epoch-window-driven-K-cap-relaxation
- **Novelty**: 0.72 — Q083 established that K=50 is latency-safe for the node-only filter at average fanout. Q091's finding that the dual-predicate filter (edge + node) increases per-edge cost by ~67% (from 0.03ms to 0.05ms) changes the safe K cap. However, the dual filter also reduces the effective fanout that reaches downstream ranking: at 10% edge survival rate (tau_edge=30) and 80% node survival rate (tau=20), only ~8% of edges produce qualifying neighbors. This means the K=50 LIMIT is hit far less frequently — for most hub nodes, F_passing < 50, making the LIMIT ineffective as a fanout cap. The real protection comes from F_fresh, not from LIMIT. Re-deriving the K cap as a function of F_fresh (not F_total) is a novel framing that allows K to be raised to 50–100 for the fresh-only traversal path without latency risk.
- **Evidence**: 0.65 — Follows directly from the cost model established in Q083 and extended in Q091. At tau_edge=30 and F_total=500, F_fresh=50. The LIMIT 50 is therefore effectively unlimited for most queries (F_passing ≈ F_fresh × node_survival ≈ 40 neighbors pass both filters). The K cap that was designed to protect against hub nodes (K=20 from Q078, relaxed to K=50 from Q083) is now protecting against a smaller set than assumed — because the edge filter, even as a post-filter, structurally reduces the candidate set before the LIMIT.
- **Feasibility**: 0.85 — Parameter change only. The K LIMIT in the Cypher query is updated from 50 to 100, with the understanding that the effective fanout reaching LIMIT is F_fresh × node_survival ≈ 8% of F_total. Even at Month 24 with F_total=1,500, only ~120 neighbors pass both filters — and K=100 returns the top 100 by co_occurrence_count, still bounded. The latency risk from raising K is lower than Q083 assessed because the dual-predicate filter already reduces the traversal candidate set.
- **Description**: Re-derive the K LIMIT value as a function of the dual-predicate survival rate (edge survival × node survival ≈ 8% at tau_edge=30, tau=20). For F_total hub nodes, the effective fanout through both filters is 0.08 × F_total. At Month 24 with F_total=1,500, this gives ~120 qualifying neighbors — so K=100 is a real constraint, not a vacuous one. For Month 12 hubs with F_total=500, F_passing ≈ 40, meaning K=50 is never hit (the LIMIT is not the binding constraint). The correct K is determined by the downstream ranking budget (how many candidates the Python reranker can process) rather than the traversal budget (which is bounded by the dual filter). Raise K to 100 as the default, with a timeout circuit breaker for Month 24+ hub nodes where F_passing approaches K.

## Key Finding

The Neo4j relationship property index on `r.last_reinforced_epoch` for the `CO_RETRIEVED` relationship type **cannot pre-filter BFS edges**. The finding mirrors Q083's conclusion for node property indexes: the index functions as an anchor access path only, usable when the query starts from a relationship property predicate (e.g., "find all CO_RETRIEVED edges reinforced in the last 30 epochs, regardless of which nodes they connect"). When the query anchor is a specific Memory node (BFS from `seed.id`), the execution planner uses relationship chain traversal — a pointer-chasing walk through the linked-list relationship store — and evaluates `r.last_reinforced_epoch >= X` as a post-traversal property read for each of the F_total edges in the chain. The index is structurally incompatible with BFS chain traversal because it would require a set intersection between the index result set (all fresh CO_RETRIEVED edges in the graph) and the seed node's chain (its local neighborhood) — an operation Neo4j's query planner does not implement.

**The theoretical speedup if pre-filtering were possible is ~10–15×** (F_total / F_fresh ≈ 10 at tau_edge=30, dropping traversal from 25ms to 1.6ms for a Month 12 500-edge hub node). The relationship property index delivers **0× speedup** for the BFS pattern.

**The only architectural path to true pre-traversal edge filtering within Neo4j Community Edition is relationship type partitioning**: maintaining `CO_RETRIEVED_FRESH` and `CO_RETRIEVED_STALE` as distinct relationship types, reclassified at session end when edges age past the tau_edge window. Neo4j organizes relationship chains per-type, so traversing `CO_RETRIEVED_FRESH` only reads the fresh sub-chain, achieving O(F_fresh) traversal — the theoretical minimum. This requires a schema migration and a background reclassification job, but no new infrastructure.

A lighter-weight partial improvement (Cypher WITH-clause staging, estimated 35–40% cost reduction) is available with a single query rewrite and no schema changes. It does not achieve the theoretical minimum but reduces the node property read waste for the 90% of edges disqualified by the edge epoch filter.

**The dual-predicate filter (edge + node) adds cost vs Q083's node-only baseline** (~25ms vs ~15ms for F=500 hub, at the post-filter level) while providing additional semantic correctness (filtering stale causal relationships, not just stale nodes). If type partitioning is not implemented, the Cypher WITH-clause staging should be applied immediately as a code-level optimization before the edge epoch predicate is deployed in production — otherwise the edge predicate, intended as a performance improvement via staleness reduction, actually increases traversal cost at the implementation level without any structural benefit.
