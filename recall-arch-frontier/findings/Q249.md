## Research Question

How does Git's fast-forward vs. three-way merge algorithm map to Recall 2.0's casaclaude/proxyclaude multi-device write coordination? Does this eliminate the need for distributed consensus (CRDT, Raft)?

---

## Git's Merge Algorithm

### Fast-Forward Detection

If branch B's commit history is a superset of branch A's (A is a direct ancestor of B), no conflicts are possible. Git simply advances the pointer. This is the `git pull` success case: one machine was behind, no concurrent writes.

### Three-Way Merge

When branches A and B have diverged from a common ancestor C (the Lowest Common Ancestor / LCA):

```
C (merge base / LCA)
├── A: changes to files X, Y
└── B: changes to files X, Z
```

**Resolution rules:**
- Only A changed X → take A's version
- Only B changed Z → take B's version
- Both changed X at the same line → CONFLICT (needs resolution)
- Both changed different parts of X → auto-merge (take both changes)

The LCA is the most recent commit that is an ancestor of both branches.

## Mapping to Recall 2.0 Multi-Device WAL Merge

| Git concept | Recall 2.0 analog |
|---|---|
| Commit | Session WAL record (memory write event with timestamp) |
| Branch | Machine identity (casaclaude / proxyclaude) |
| File content | Memory text for entity-predicate pair |
| LCA / merge base | Last sync point (last event both machines have seen) |
| Fast-forward merge | All new memories on one machine only since last sync |
| Three-way merge | Both machines wrote memories since last sync |
| Auto-merge (different files) | Different entity-predicates on each machine → take both |
| Conflict (same file, same line) | Same entity-predicate, different values → LWW resolution |

### Fast-Forward Case (Expected ~95%)

Tim uses one machine at a time. proxyclaude accumulates memories while casaclaude is idle. At sync:

```
sync_base = last_synced_event_timestamp
casaclaude_new = WAL.events_after(sync_base)  # empty or minimal
proxyclaude_new = WAL.events_after(sync_base)  # M1, M2, M3...

// Fast-forward: apply proxyclaude_new to casaclaude
```

This is the 95% case. No conflict detection needed.

### Three-Way Merge Case (Rare, ~5%)

Both machines wrote memories between sync points:

```
sync_base: "Ollama runs on 192.168.50.62:11434"

casaclaude wrote: "Ollama model: qwen3:14b"  (timestamp: T+100s)
proxyclaude wrote: "Ollama model: qwen3:32b" (timestamp: T+50s)
```

Same entity (Ollama), same predicate (model), different values. **Conflict.**

**Resolution rule**: Last-Write-Wins (LWW) by event timestamp. T+100s > T+50s → casaclaude's version wins.

This is the Q236 entity-predicate conflict index in the merge context: same entity-predicate pair from both WALs → keep the one with the higher timestamp.

## The Key Physical Insight

Git's three-way merge is **textual** (line-level comparison of file contents). Recall 2.0's merge is **semantic** (entity-predicate comparison of structured triples).

Semantic merge is strictly easier than textual merge:
- No "hunk" detection needed
- No diff algorithm needed
- Conflict resolution is deterministic (LWW by timestamp)
- Auto-merge coverage: ~99% of pairs (different entity-predicates rarely conflict)

The WAL event timestamp (already required by Q226 for session boundary detection) serves as the "vector clock" for LWW. No additional timestamp infrastructure is needed.

## Full Merge Algorithm for Recall 2.0

```
1. Identify sync_base: most recent event_id both machines share
2. Extract delta_A = casaclaude.WAL.events_after(sync_base)
3. Extract delta_B = proxyclaude.WAL.events_after(sync_base)

4. IF delta_A is empty: fast-forward apply delta_B → done
5. IF delta_B is empty: fast-forward apply delta_A → done
6. ELSE (three-way merge):
   a. Merge = delta_A ∪ delta_B (set union by event_id — no duplicates)
   b. Sort by event_timestamp ASC
   c. Apply events in order, letting Q236 entity-predicate index handle conflicts:
      - For each event, extract entity-predicate pairs
      - If conflict index shows existing entry with same entity-predicate:
        - Keep newer timestamp (LWW)
        - Mark older as superseded
   d. Update sync_base to max(event_timestamp) in Merge
```

**Complexity**: O(n log n) sort + O(n) conflict scan using existing LMDB entity-predicate index. No distributed consensus. No leader election. No quorum.

## Does This Eliminate CRDT/Raft?

**Yes, for the homelab use case.**

CRDTs are needed when concurrent writes must be merged without ordering guarantees (network partition, no reliable clocks). Raft is needed when a quorum must agree on write order before it is applied.

Neither applies to Recall 2.0:
- casaclaude and proxyclaude communicate via Tailscale (reliable, ordered)
- Timestamps come from hook context (wall clock per machine, sufficient for LWW)
- Sync happens at session boundaries, not real-time (no network partition window)
- LWW is a valid CRDT when timestamps are monotonic — and session WAL timestamps are monotonic per machine

The Recall 2.0 WAL merge is equivalent to a **monotonic LWW CRDT** at the entity-predicate level: the "value" for any entity-predicate pair converges to the version with the highest timestamp across all replicas, regardless of sync order.

## Sync Trigger Point

The natural sync point: beginning of UserPromptSubmit hook. Before injecting memories, check whether the remote WAL has new events. If yes, fast-forward merge (most common), then proceed with retrieval.

This is the same trigger as Q244's watermark-delayed compaction: UserPromptSubmit is the convergence point for all session-boundary operations.

The LCA is the last sync point; the auto-merge rule covers ~99% of cases (different entity-predicates don't conflict); the conflict rule is deterministic LWW by event timestamp (Q236 entity-predicate index). The WAL event timestamp (already required by Q226) serves as the vector clock. No CRDT library, no Raft, no distributed consensus needed.
