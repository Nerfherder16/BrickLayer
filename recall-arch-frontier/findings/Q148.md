## Composite Scoring Path Analysis

### Exact formula from retrieval.py source (verified)

The composite score is computed at Stage 2 (vector search) and applied throughout the pipeline:

```
score = similarity × max(importance, 0.15)
```

Floor: `max(importance, 0.15)` — importance is floored at 0.15, never zero in composite scoring. This is the primary vector search scoring formula (line 222 and line 618 of retrieval.py).

**Stage modifiers applied after base score:**

| Stage | Modifier | Condition |
|-------|----------|-----------|
| Domain affinity boost | `score *= 1.15` | Memory domain matches query domain_hint |
| Working memory boost | `score *= 1.5` | Memory ID is in active session working memory |
| Current file boost | `score *= 1.3` | Memory domain matches current file path component |
| Task tag overlap | `score *= 1 + (overlap × 0.2)` | Memory tags overlap query task keywords |
| Recency factor | `score *= 1/(1 + hours_old × 0.01)` | Legacy path only |
| Stability factor | `score *= 0.5 + stability × 0.5` | Legacy path |
| Confidence factor | `score *= 0.7 + confidence × 0.3` | Legacy path |

**Score floor before returning results**: `min_combined_score = 0.15` — results below 0.15 are dropped from output.

### Oracle memory math

A well-formed oracle memory with `importance=0.95` and query similarity `sim=0.75` produces:
```
base_score = 0.75 × max(0.95, 0.15) = 0.75 × 0.95 = 0.7125
```

A competing organic memory with `importance=0.55` and similarity `sim=0.80` (slightly better raw ANN) produces:
```
base_score = 0.80 × max(0.55, 0.15) = 0.80 × 0.55 = 0.44
```

Oracle wins composite scoring by 0.27 margin despite lower raw similarity. At `importance=0.80`:
```
oracle = 0.75 × 0.80 = 0.600
organic = 0.80 × 0.55 = 0.440 (organic would need sim=0.75 AND importance=0.80 to tie)
```

**Critical insight**: An oracle with `importance>=0.80` will dominate composite scoring for its topic region unless a competing organic memory has *both* higher similarity AND importance >=0.80. This is the reliable threshold — at importance=0.80, the oracle is compositionally dominant over typical organic memories (importance ~0.35-0.55 distribution).

### Statistical Power Analysis

**Minimum viable corpus**: 5 oracle memories.

Rationale: A health check using ≥5 probes provides:
- Rank check: each oracle either appears in top-3 or does not (binary per probe)
- Pass rate: N_pass / N_total — with N=5, meaningful thresholds are 5/5 (100%), 4/5 (80%), 3/5 (60%)
- False alarm rate: at `importance=0.95`, probability of a lower-importance organic memory outranking the oracle requires that organic to have `similarity >= oracle_similarity × (0.95/organic_importance)`. For typical organics at importance 0.50 this requires them to have 1.9× higher similarity — extremely unlikely for well-chosen oracle topics.

**Recommended corpus**: 5 founding oracles plus 1-2 oracles per new major domain added. The founding 5 cover: identity, homelab infra, Ollama infra, Recall stack, personal preferences.

**Oracle health signal reliability**: With 5 oracles, a single DEGRADED oracle (rank 4-10) indicates localized composite scoring pressure in that domain. Two or more DEGRADED oracles indicate systemic importance distribution drift.

## /health/recall-quality Endpoint Spec

### Route location

Add to `C:\Users\trg16\Dev\Recall\src\api\routes\health_dashboard.py` as:
```
GET /admin/health/recall-quality
```
Rate limit: `10/minute`. Cache TTL: 60 seconds (shorter than other health endpoints — this check reflects real-time state).

### Three States

| State | Condition | `status` field | `overall_pass` |
|-------|-----------|---------------|---------------|
| **UNINITIALIZED** | No oracle memories found (oracle_count = 0) | `"uninitialized"` | `false` |
| **HEALTHY** | ≥80% of oracles appear at composite rank 1-3 (e.g., 4/5 or 5/5) | `"healthy"` | `true` |
| **DEGRADED** | Any oracle found but <80% pass rate OR any oracle is rank 4-10 | `"degraded"` | `false` |

**UNINITIALIZED is distinct from DEGRADED**: `0/0` oracles is not a health pass — it means the oracle corpus has not been seeded. This prevents a cold-start false-healthy signal.

**Rank threshold justification**:
- Rank 1-3 (HEALTHY): Oracle is in the active injection window. The hook injects top-3 to top-10 results. Rank 1-3 guarantees oracle content reaches the LLM.
- Rank 4-10 (DEGRADED): Oracle is findable but below primary injection window. Semantic retrieval works but competitive pressure is compressing oracle rank.
- Rank 11+ (FAILED sub-state within DEGRADED): Oracle is effectively displaced. Flag this separately in the per-oracle detail for diagnostics.

**Pass rate threshold**: 80% (4/5 with 5 oracles). This tolerates one oracle being temporarily compressed by a session-specific domain boost (e.g., working memory from current session boosting a competing organic). Two or more degraded oracles = systemic issue.

### Implementation Code (Python pseudocode for health_dashboard.py)

```python
ORACLE_QUALITY_CACHE_KEY = "recall:health:recall_quality"
ORACLE_QUALITY_CACHE_TTL = 60  # 1 minute — shorter than other health checks

# Oracle registry: stored in Redis at recall:oracle:registry
# Format: list of {oracle_id, memory_uuid, probe_query, domain}
ORACLE_REGISTRY_KEY = "recall:oracle:registry"


@router.get("/health/recall-quality")
@limiter.limit("10/minute")
async def get_recall_quality(request: Request):
    """
    Oracle-based retrieval quality check for the composite-scored path.

    Tests whether high-importance oracle memories rank in top-3 of composite
    scoring for their canonical probe queries. Distinct from /health/retrieval-quality
    (which reports injection_log statistics) and the canary infrastructure check
    (which tests raw ANN via direct Qdrant search).

    Three states:
    - uninitialized: no oracle memories seeded (oracle_count=0)
    - healthy: >=80% of oracles rank in composite top-3
    - degraded: oracles found but <80% pass rate or any oracle rank 4-10
    """
    import time
    from datetime import UTC, datetime

    try:
        redis = await get_redis_store()

        cached = await redis.client.get(ORACLE_QUALITY_CACHE_KEY)
        if cached:
            return json.loads(cached)

        t_start = time.monotonic()

        # Step 1: Load oracle registry from Redis
        registry_raw = await redis.client.get(ORACLE_REGISTRY_KEY)
        if not registry_raw:
            return {
                "status": "uninitialized",
                "overall_pass": False,
                "oracle_count": 0,
                "pass_count": 0,
                "pass_rate": 0.0,
                "latency_ms": round((time.monotonic() - t_start) * 1000),
                "checked_at": datetime.now(UTC).isoformat(),
                "oracles": [],
                "message": "No oracle memories registered. Run oracle seed population protocol.",
            }

        oracle_registry = json.loads(registry_raw)
        oracle_count = len(oracle_registry)

        # Step 2: For each oracle, run composite-scored query via retrieval pipeline
        from src.core import MemoryQuery
        from src.core.retrieval import get_retrieval_pipeline

        pipeline = await get_retrieval_pipeline()
        oracle_results = []
        pass_count = 0

        for oracle in oracle_registry:
            oracle_uuid = oracle["memory_uuid"]
            probe_query = oracle["probe_query"]

            query = MemoryQuery(
                text=probe_query,
                limit=10,
                # No session_id — avoids working memory boosts contaminating the check
                # No domain filter — oracle must win in open competition
            )

            try:
                results = await pipeline.retrieve(query)

                rank = None
                composite_score = None
                similarity = None
                importance = None

                for i, r in enumerate(results):
                    if r.memory.id == oracle_uuid:
                        rank = i + 1  # 1-indexed
                        composite_score = round(r.score, 4)
                        similarity = round(r.similarity, 4)
                        importance = r.memory.importance
                        break

                oracle_pass = rank is not None and rank <= 3
                if oracle_pass:
                    pass_count += 1

                oracle_results.append({
                    "oracle_id": oracle["oracle_id"],
                    "memory_uuid": oracle_uuid,
                    "probe_query": probe_query,
                    "rank": rank,  # None if not in top-10
                    "composite_score": composite_score,
                    "similarity": similarity,
                    "importance": importance,
                    "status": "pass" if oracle_pass else ("degraded" if rank else "displaced"),
                })

            except Exception as e:
                oracle_results.append({
                    "oracle_id": oracle["oracle_id"],
                    "memory_uuid": oracle_uuid,
                    "probe_query": probe_query,
                    "rank": None,
                    "status": "error",
                    "error": str(e)[:120],
                })

        pass_rate = pass_count / oracle_count if oracle_count > 0 else 0.0
        is_healthy = pass_rate >= 0.80
        status = "healthy" if is_healthy else "degraded"

        result = {
            "status": status,
            "overall_pass": is_healthy,
            "oracle_count": oracle_count,
            "pass_count": pass_count,
            "pass_rate": round(pass_rate, 3),
            "latency_ms": round((time.monotonic() - t_start) * 1000),
            "checked_at": datetime.now(UTC).isoformat(),
            "oracles": oracle_results,
        }

        await redis.client.setex(
            ORACLE_QUALITY_CACHE_KEY,
            ORACLE_QUALITY_CACHE_TTL,
            json.dumps(result, default=str),
        )

        return result

    except Exception as e:
        logger.error("recall_quality_check_error", error=str(e))
        raise HTTPException(status_code=500, detail="Internal server error")
```

### SLA Budget

**Total latency budget: 3,000 ms** for the full endpoint response.

**Per-oracle query latency breakdown**:
- Embedding generation (Ollama qwen3-embedding:0.6b): ~80-150 ms per query
- Qdrant ANN search (top-10): ~5-15 ms
- Neo4j graph expansion (Stage 3): ~20-80 ms
- Context filter + ranking (Stage 4/5): <5 ms

**Per-oracle total**: ~120-250 ms. With 5 oracles queried **sequentially**: 600-1,250 ms.
With 5 oracles queried **concurrently** (`asyncio.gather`): ~200-300 ms (bounded by Ollama batch).

**Recommendation**: Query all 5 oracles concurrently using `asyncio.gather`. Keep oracle count ≤10 to stay within the 3,000 ms budget. Cache the result for 60 seconds — this check does not need real-time granularity and the embedding service load is non-trivial.

**Non-blocking from /health**: The root `/health` endpoint (main.py line 275) does NOT call this oracle check. The oracle check is an admin-only endpoint under `/admin/health/recall-quality`. It should not block the public `/health` liveness probe. If Tim wants oracle status in the root `/health` response, it should be fetched from cache only — never compute inline.

[DATA] Composite scoring formula verified from source: `score = similarity × max(importance, 0.15)`. Floor is 0.15 (not 0.0). At `importance=0.95`, an oracle with `similarity=0.75` scores `0.7125` — dominating any organic memory with `importance≤0.55` regardless of raw similarity (organic needs `sim > 1.29` to tie, which is impossible for cosine similarity). [FINDING] Five oracle memories with `importance=0.95`, `durability=permanent`, and `is_oracle=true` metadata flag are sufficient to provide reliable composite-path health signal. Query oracle probes concurrently with `asyncio.gather` and cache for 60 seconds to stay within 3,000 ms SLA budget.

[STAT:n] 5 oracle memories × 5 distinct topic domains × importance=0.95 = composite score floor of 0.665 (at sim=0.70), exceeding the 0.44 typical organic ceiling (sim=0.80 × importance=0.55).

## Qualifying Ideas

- **oracle-registry-in-redis**: (N=0.90, E=0.85, F=0.88) — Store the oracle registry (oracle_id → memory_uuid + probe_query mapping) as a JSON blob at Redis key `recall:oracle:registry`. This makes oracle IDs available to the health check without a semantic lookup or database scan. The populate-oracles script writes to this key after successful `POST /memory/store` calls. The health check reads it first and returns UNINITIALIZED immediately if missing.

- **asyncio-gather-oracle-queries**: (N=0.88, E=0.82, F=0.90) — Query all 5 oracle probe queries concurrently using `asyncio.gather(*[check_oracle(o) for o in registry])`. Each query hits the full retrieval pipeline independently. Concurrent execution reduces 5-oracle latency from ~1,250 ms sequential to ~300 ms parallel (bounded by Ollama embedding service). With 10 oracles, parallel is mandatory to stay within 3,000 ms SLA.

- **oracle-displacement-alert-field**: (N=0.82, E=0.78, F=0.85) — Add a `displaced_by` field to per-oracle results when the oracle is not in top-10. Run a second query to identify what memory IS at rank-1 for the probe query: return its UUID, content snippet (first 100 chars), importance score, and domain. This surfaces the specific competitor that displaced the oracle — actionable diagnostic for understanding which organic memory is creating composite pressure in that topic region.

- **oracle-rank-trend-tracking**: (N=0.70, E=0.65, F=0.72) — Log each oracle health check result to PostgreSQL `metrics_snapshot` table (already exists per health_dashboard.py line 130-133). Include oracle_id, rank, composite_score, and pass_rate per check. This builds a trend surface: if oracle OM-002 (CasaOS IP) goes from rank-1 → rank-2 → rank-3 over 30 days, the trend is visible before the oracle falls below the HEALTHY threshold. Enables early warning before state transition to DEGRADED.
