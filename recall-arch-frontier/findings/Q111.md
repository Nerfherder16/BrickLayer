### Step 1: Choose EPOCH_RESET_TOLERANCE

**Tim's topology**: Two Claude Code machines (casaclaude + proxyclaude) but BOTH point to the same Recall API at 192.168.50.19:8200. There is ONE Redis instance, ONE PostgreSQL instance, ONE epoch counter.

**Analysis of legitimate Redis-vs-PG divergence in a single-instance deployment**:

| Scenario | Max divergence |
|----------|---------------|
| Normal operation | 0 (Redis and PG updated atomically in same session-end hook) |
| Session-end hook: Redis INCR succeeds, PG UPDATE fails (rollback) | 1 epoch |
| Session-end hook: PG UPDATE succeeds, then Redis crashes before INCR | 1 epoch (PG ahead of Redis, not behind — guard doesn't fire) |
| Redis container restart (AOF replay lag) | 0 (AOF replays all INCRs) |
| Redis container restart WITHOUT AOF | Could lose many epochs — this is the failure mode the guard catches |
| Two concurrent sessions both complete at exactly the same instant | 2 epochs at most |

**Conclusion**: For a single Recall instance, the maximum LEGITIMATE gap between Redis and PG is 2 epochs under any non-failure scenario. A gap of 5 is unambiguously a Redis reset event.

```python
EPOCH_RESET_TOLERANCE = 5  # Single-instance deployment (not 50 — that was for dual-instance)
```

The Q079 value of 50 was derived for two independent Recall instances where natural clock skew between casaclaude and proxyclaude could produce legitimate divergence. With a shared instance, 50 would allow nearly a full restart event to go undetected.

**After** (guard inserted as first block inside `try:`):

```python
@router.post("/browse", response_model=BrowseResponse)
@limiter.limit("300/minute")
async def browse_memories(
    request: Request, body: SearchRequest, user: User | None = Depends(require_auth)
) -> BrowseResponse:
    """
    Token-efficient memory search.
    ...
    """
    try:
        # --- Epoch sanity guard (Q106 Layer 1) ---
        # Detects Redis epoch reset by comparing Redis counter to PG max.
        # Must run BEFORE any epoch-dependent logic in this request.
        EPOCH_RESET_TOLERANCE = 5  # Single-instance: legitimate max divergence is ~2
        try:
            from src.storage import get_postgres_store, get_redis_store as _get_redis_epoch

            _pg_epoch = await get_postgres_store()
            _redis_epoch = await _get_redis_epoch()
            assert _pg_epoch.pool is not None
            assert _redis_epoch.client is not None
            pg_max_epoch = await _pg_epoch.pool.fetchval(
                "SELECT MAX(last_access_epoch) FROM memories"
            )
            current_epoch_raw = await _redis_epoch.client.get("recall:global_epoch")
            current_epoch = int(current_epoch_raw or 0)
            if current_epoch < (pg_max_epoch or 0) - EPOCH_RESET_TOLERANCE:
                logger.warning(
                    "epoch_reset_detected",
                    redis_epoch=current_epoch,
                    pg_max_epoch=pg_max_epoch,
                    delta=(pg_max_epoch or 0) - current_epoch,
                )
                repaired_epoch = (pg_max_epoch or 0) + 1
                await _redis_epoch.client.set("recall:global_epoch", repaired_epoch)
        except Exception as _epoch_guard_err:
            # Guard is non-fatal — log and continue. Never block retrieval.
            logger.warning("epoch_guard_error", error=str(_epoch_guard_err))
        # --- End epoch sanity guard ---

        from src.core.domains import normalize_domain

        # Normalize domains to match how they're stored
        normalized_domains = [normalize_domain(d) for d in body.domains] if body.domains else None
        # domain_hint → soft boost (not a hard Qdrant filter)
        normalized_hints = [normalize_domain(body.domain_hint)] if body.domain_hint else None

        query = MemoryQuery(
            ...
        )
```

**Key decisions in this diff**:
- The guard is wrapped in its own `try/except` — a PG connectivity error must never block a retrieve call. Guard failures are logged as warnings, not errors.
- Uses `_pg_epoch.pool.fetchval` (consistent with the existing pattern in `health_check()` at `main.py:311`).
- Import is inline (consistent with the file's existing pattern of inline imports throughout).
- The `EPOCH_RESET_TOLERANCE` constant is defined locally. If it needs to be shared with `search_memories`, move it to module scope at line ~20 of the file.

**Current** (`docker-compose.yml` line 74):

```yaml

```

**Required change**:

```yaml
  redis:
    image: redis:7-alpine
    container_name: recall-redis
    ports:
      - "127.0.0.1:6380:6379"
    volumes:
      - redis_data:/data
    command: redis-server --appendonly yes --appendfsync everysec --save ""
```

**What each flag does**:

| Flag | Effect |
|------|--------|
| `--appendonly yes` | Enables AOF persistence (already present) |
| `--appendfsync everysec` | Flushes AOF buffer to disk every 1 second. Without this, Redis uses its compiled-in default (which happens to also be `everysec`) but the explicit flag is required for the health-check assertion to be trustworthy — you cannot `CONFIG GET appendfsync` reliably without it being set explicitly on some Redis builds |
| `--save ""` | Disables RDB snapshot persistence. AOF-only mode. RDB snapshots on a single-machine homelab deployment add I/O overhead with no benefit when AOF is active. If the container crashes, AOF replay recovers to within 1 second; RDB would only recover to the last checkpoint (every 60s at default). AOF is strictly better here. **RDB can be safely removed.** |

**AOF file location**: `/data/appendonly.aof` inside the container, backed by the `redis_data` Docker volume. This volume persists across container restarts.

**Apply the change**:
```bash

docker compose up -d redis
# Redis will restart and AOF will remain intact from the volume
```

### Step 5: AOF Health Check in FastAPI Lifespan

**Where to place it**: Inside the existing `lifespan` context manager in `src/api/main.py`, after `redis = await get_redis_store()` succeeds (approximately line 74). Place it in the startup block, before `yield`.

**Why lifespan, not health probe**: The health probe runs on every poll (Uptime Kuma, monitoring). AOF status doesn't change during runtime — it only matters at startup. A lifespan check fires once and logs a persistent warning that will appear in `docker compose logs recall-api`. The health probe check would generate log noise every 30 seconds.

**Code insertion** (after line 74 in `main.py`):

```python
        redis = await get_redis_store()
        postgres = await get_postgres_store()

        # AOF health check — warn if epoch reset vulnerability is active
        try:
            assert redis.client is not None
            aof_config = await redis.client.config_get("appendonly")
            if aof_config.get("appendonly") != "yes":
                logger.warning(
                    "redis_aof_disabled",
                    message="Redis AOF is disabled — epoch reset vulnerability active. "
                            "Add --appendonly yes to docker-compose.yml redis command.",
                )
            else:
                logger.info("redis_aof_enabled", appendfsync="everysec")
        except Exception as _aof_err:
            logger.warning("redis_aof_check_failed", error=str(_aof_err))
```

This check is non-fatal. If Redis is misconfigured, the API still starts, but the warning will be visible in logs.

### Step 8: False Positive Risk Assessment

With `EPOCH_RESET_TOLERANCE = 5`, false positives require Redis to legitimately fall 6+ epochs behind PG. In a single-instance deployment this cannot happen in normal operation. The only edge case:

- **API restart during active session**: A session's PG update commits, Redis INCR is in-flight when API restarts. On restart, Redis is 1 behind. Guard threshold is 5. Safe margin: 4x.
- **Database connection timeout during PG write**: PG epoch advances, Redis doesn't. Same 1-epoch divergence. Safe.

The guard is false-positive-safe for `TOLERANCE >= 3` in single-instance deployment. `5` provides a 2.5x safety margin over the theoretical maximum legitimate divergence.

## Ideas Extracted

### Idea: epoch-guard-non-fatal-wrapper-pattern
- **Novelty**: 0.70
- **Evidence**: 0.85
- **Feasibility**: 0.95
- **Description**: Wrapping infrastructure health guards in their own `try/except` with `logger.warning` (not `logger.error`) ensures that defensive instrumentation never degrades the primary operation. The guard is there to catch failures, not create new ones. Applied consistently, this pattern means every observability/guard layer in a hot path is non-fatal by construction. This is a distinct design principle from the guard logic itself: "guards must never guard-fail."

### Idea: tolerance-calibration-by-topology
- **Novelty**: 0.75
- **Evidence**: 0.80
- **Feasibility**: 0.95
- **Description**: EPOCH_RESET_TOLERANCE is not a fixed constant — it is a function of deployment topology. Single-instance: max legitimate divergence = 2, tolerance = 5. Dual-instance with shared storage: tolerance = 5. Dual-instance with separate storage (Q079 case): tolerance = 50. Making this a named, documented constant with the derivation rationale in a comment prevents the value from being cargo-culted to the wrong topology, which would either cause false positives (too low) or miss real resets (too high).

### Idea: aof-health-check-at-lifespan-not-probe
- **Novelty**: 0.68
- **Evidence**: 0.75
- **Feasibility**: 0.95
- **Description**: Infrastructure config checks (AOF enabled, connection pool size, index presence) belong in the startup lifespan, not the health probe endpoint. Health probes are polled continuously; lifespan fires once. Putting config assertions in lifespan keeps the health probe fast and clean while ensuring config drift is caught at startup with a clear log message. This is a general architectural principle for any FastAPI service with backing store dependencies.

### Idea: greenfield-epoch-requires-migration-gate
- **Novelty**: 0.72
- **Evidence**: 0.90
- **Feasibility**: 0.90
- **Description**: When a proposed guard reads a column (`last_access_epoch`) that does not yet exist in the schema, deploying the guard code before the migration produces a `ProgrammingError` on every retrieve call. The correct deployment gate is: migration applied and verified → then code deployed. For future epoch-style guards, the code should include an explicit fallback: `pg_max_epoch = await pool.fetchval("SELECT MAX(last_access_epoch) FROM memories") or 0` where the `or 0` handles both NULL (no rows) and column-not-found (via the outer try/except). The outer try/except in the guard already handles the latter case non-fatally.
