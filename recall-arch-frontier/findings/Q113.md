## Architecture Note: Actual vs. Proposed System

Before threading the adversarial analysis, a critical clarification from code inspection:

The `CO_RETRIEVED_FRESH`, `last_access_epoch`, and `is_tombstoned` properties do **not yet exist** in the deployed Recall codebase (`C:/Users/trg16/Dev/Recall/src`). The epoch-based architecture and tombstone property are forward-looking designs established by the autoresearch. The current production system uses:

- **Graph edges**: `CO_RETRIEVED`, `RELATED_TO`, `CONTRADICTS`, `SUPERSEDES` (no FRESH/STALE partitioning)
- **Deletion model**: `superseded_by` property + 30-day GC hard-delete (gc.py), not a tombstone flag
- **Reconcile**: Qdrant-vs-Neo4j UUID set-difference (`get_all_memory_ids()` — `MATCH (m:Memory) RETURN m.id`)
- **Health queries**: High-gravity node detection, no BFS epoch probe in production

Therefore this audit is prospective: it analyzes which of the *proposed* query paths (specified across Q091, Q092, Q093, Q094, Q102, Q104) will require tombstone exclusion guards when the epoch-based architecture is implemented. Q101's `AND n.is_tombstoned IS NULL` fix was applied only to the S5 BFS probe Cypher. Five additional query paths have not been audited.

## Thread A: The Tombstone Leak is Systemic — Every Cypher Path Needs the Guard

**Core claim**: Every Cypher query in the proposed epoch-based architecture that traverses or counts Memory nodes was designed with epoch-window semantics as the only exclusion mechanism. The tombstone predicate (`AND n.is_tombstoned IS NULL`) is not present in any query except the patched S5 probe. Because tombstoned nodes within the tau epoch window pass the epoch predicate by design (they were recently accessed before deletion), they will inject false results into every unguarded path.

**Evidence**:

1. **Production BFS retrieve query** (Q078 Option A, Q091 pattern): The documented BFS Cypher is:
   ```cypher
   MATCH (seed:Memory)-[r:CO_RETRIEVED_FRESH]->(neighbor:Memory)
   WHERE seed.id IN $seed_ids
     AND neighbor.last_access_epoch >= ($current_epoch - $tau)
     AND neighbor.last_access_epoch <= $current_epoch
   ```
   No `AND neighbor.is_tombstoned IS NULL` clause is documented anywhere in Q078, Q079, Q080, Q091, or Q096. This is the path that actually serves LLPC context — the highest-impact path in the system.

2. **S6 hub-degree scan** (Q102 Section 5): The S6-THRESHOLDS query is:
   ```cypher
   MATCH (n:Memory)
   WITH size((n)-[:CO_RETRIEVED]->()) AS fanout
   RETURN count(CASE WHEN fanout > 20 THEN 1 END) AS nodes_minor_hub ...
   ```
   No tombstone exclusion. A tombstoned hub node with degree D would be counted in the `nodes_minor_hub` bucket and could trigger the S6 WARNING threshold at max_fanout=100 prematurely.

3. **Reclassification job** (Q096/Q104): The reclassification Cypher finds FRESH edges to demote:
   ```cypher
   MATCH ()-[r:CO_RETRIEVED_FRESH]->()
   RETURN count(r) AS fresh_total,
          count(CASE WHEN r.last_reinforced_epoch < ($current_epoch - $tau_edge) THEN 1 END) AS stale_classified_as_fresh
   ```
   No tombstone exclusion on the source or target node. The reclassification job wastes a write cycle demoting FRESH→STALE on edges whose endpoint nodes will be deleted in the next GC sweep.

4. **Drift fraction health metric** (Q104 synthesis): The same counting query above computes `drift_fraction`. Tombstoned nodes' edges inflate `fresh_total` and `stale_classified_as_fresh`, making the drift fraction alarm noisy.

5. **Full graph reconcile** (Q093 MATCH pattern): The existing production reconcile (`ops.py`) runs `MATCH (m:Memory) RETURN m.id` — a full-graph ID scan. Tombstoned nodes would appear in this scan. However, Q093's `mode=post_restore` Cypher uses `last_access_epoch > pg_max_epoch` as its predicate, meaning tombstoned nodes with recent epochs WOULD be included in the deletion scope — which is correct behavior for post-restore cleanup.

**Best case scenario**: The tombstone fix is applied to all five paths simultaneously, reducing false-positive pressure across all health signals and improving production BFS precision by 19,900x as established in Q101.

## Thread B: Most Paths Are Self-Correcting or Correctly Include Tombstoned Nodes

**Core claim**: The tombstone exclusion guard is only operationally necessary for the production BFS retrieve path. The other four paths are either correct-by-design (they should include tombstoned nodes), self-limiting (tombstone duration is bounded by the cleanup sweep), or have negligible impact at Tim's corpus size.

**Evidence**:

1. **Production BFS retrieve query**: Thread B concedes this is the highest-risk path. However, at Tim's steady state (10 tombstones within the tau=4-day window, 4,500 nodes), a tombstoned node has a 10/4,500 = 0.22% probability of being a BFS neighbor. In top-K=10 retrieval, the expected number of tombstoned results per query is 0.022. This manifests as roughly one wrong result every 45 LLPC calls — annoying but not catastrophic.

2. **Reconcile orphan scan**: The Q093 production reconcile (`MATCH (m:Memory) RETURN m.id` in `neo4j_store.get_all_memory_ids()`) should INCLUDE tombstoned nodes. The reconcile computes the symmetric difference between Qdrant IDs and Neo4j IDs. A tombstoned node has had its Qdrant point hard-deleted (by GC or by the delete handler) but retains its Neo4j node until the next GC sweep. Including it in the Neo4j ID set correctly identifies it as a Neo4j orphan (in Neo4j, absent from Qdrant) — which is the legitimate orphan category that reconcile should surface and repair. Excluding tombstoned nodes from the reconcile would cause them to be missed by cleanup, making the fix WRONG for this path.

3. **S6 hub-degree scan**: Tombstoned hub nodes have degree D ≥ 0 in CO_RETRIEVED. However, the critical insight: once a memory is tombstoned, it receives zero new reinforcements. Its `last_reinforced_epoch` stops advancing. Over time, all edges from a tombstoned node are reclassified to STALE by the reclassification job — so the tombstoned node's CO_RETRIEVED_FRESH degree drops to 0. The S6 scan counts ALL CO_RETRIEVED edges (not just FRESH), so it still counts the tombstoned node's historical STALE edges. But the hub-degree threshold (max_fanout > 100 for WARNING) is calibrated for Month 5+ behavior; a single tombstoned node with D < 30 (the typical degree at tombstone time, since hubs don't form until Month 7+) cannot trigger the WARNING threshold alone.

4. **Reclassification job**: Processing a FRESH→STALE reclassification on an edge pointing to a tombstoned node does exactly one write to Neo4j — setting `r.edge_type = 'STALE'` on an edge that will be hard-deleted in the next GC sweep. The waste is: one unnecessary Cypher SET per stale FRESH edge per tombstoned node. At 2.5 tombstones/day and 45 edges/session, the maximum waste is ~112 unnecessary SET operations total before GC cleans the node. This is wasted work, not incorrect work.

5. **Q104 drift fraction metric**: The drift fraction denominator is `total_fresh_edges`. With 10 tombstoned nodes in the tau window, each contributing ~1-5 FRESH edges that haven't yet been reclassified, the false inflation is ~10–50 edges against a denominator of ~1,350. The bias is 0.7–3.7%. The drift_fraction alarm threshold is 0.15 (15%). A 3.7% bias cannot trigger a false alarm at this threshold.

**Failure mode**: The only path that genuinely produces observable user-facing errors without the tombstone guard is the production BFS retrieve query. The other paths either produce negligible metric bias (S6, reclassification) or are correct-by-design (reconcile).

## Synthesis

**What Thread A assumes that Thread B exposes**:

Thread A assumes all paths are equally impacted by tombstone lag. Thread B exposes a critical asymmetry: the correct behavior of tombstone exclusion depends on the *purpose* of each query. Queries that serve retrieval (BFS) must exclude tombstoned nodes. Queries that serve cleanup (reconcile) must include tombstoned nodes. Queries that serve monitoring (S6, reclassification health) have bounded tolerance for tombstone-induced noise. The tombstone predicate is not a universal fix to add to every Cypher — it is a path-specific decision.

**When Thread A is right**:
- Production BFS retrieve: tombstones within the tau window contaminate LLPC context — Thread A is correct that this needs the fix immediately
- S6 hub-degree scan at late deployment (Month 7+): if the graph has large hub nodes (degree > 158) and 5+ tombstones accumulate at once, tombstone inflation can contribute meaningful noise to the degree count — Thread A is correct to worry
- Corpus growth scenarios: at 10,000+ nodes with higher tombstone rates, the 0.22% contamination probability doubles or triples — Thread A is correct that the risk scales

**When Thread B is right**:
- Reconcile orphan scan: tombstoned nodes are correct-by-design to include — Thread B is correct that adding the tombstone exclusion guard here would break the cleanup logic
- Reclassification job at current scale: 112 wasted SET operations at 2.5 tombstones/day is negligible — Thread B is correct that this is not operationally harmful
- S6 hub-degree at Month 0–6: no hubs exist, so tombstone degree inflation cannot trigger false alarms — Thread B is correct for the current operational phase

## Query-by-Query Audit: Fix / No-Fix Recommendation

### Path 1: Production BFS Retrieve Query (LLPC /retrieve endpoint)

**Status**: UNGUARDED — needs immediate fix

**Current Cypher** (as specified in Q078/Q091):
```cypher
MATCH (seed:Memory)-[r:CO_RETRIEVED_FRESH]->(neighbor:Memory)
WHERE seed.id IN $seed_ids
  AND neighbor.last_access_epoch >= ($current_epoch - $tau)
  AND neighbor.last_access_epoch <= $current_epoch
```

**False-positive impact**: At Tim's steady state (10 tombstones in tau=4-day window, 4,500 nodes), tombstoned neighbor probability = 10/4,500 = 0.22% per BFS candidate. With K=50 candidates per probe, expected tombstoned results per retrieve call = 0.11. Over 5 sessions/day at 3 LLPC calls/session: ~1.65 contaminated LLPC contexts per day. Each contamination injects a deleted memory into Claude's active context.

**Fix**:
```cypher
MATCH (seed:Memory)-[r:CO_RETRIEVED_FRESH]->(neighbor:Memory)
WHERE seed.id IN $seed_ids
  AND neighbor.last_access_epoch >= ($current_epoch - $tau)
  AND neighbor.last_access_epoch <= $current_epoch
  AND neighbor.is_tombstoned IS NULL
```

**Verdict**: MUST FIX — highest priority, user-facing context corruption.

### Path 2: /reconcile Neo4j Orphan Scan

**Status**: CORRECTLY EXCLUDES TOMBSTONE GUARD — no fix needed

**Current Cypher** (production `ops.py` → `neo4j_store.get_all_memory_ids()`):
```cypher
MATCH (m:Memory) RETURN m.id AS id
```

**Correct behavior analysis**: The reconcile computes `neo4j_ids - qdrant_ids` to find Neo4j orphans. A tombstoned node has had its Qdrant point deleted but retains its Neo4j node until the next GC sweep. Including it in the Neo4j scan correctly marks it as a neo4j_orphan — which signals to the reconcile that this node should be cleaned. If the tombstone guard `AND m.is_tombstoned IS NULL` were added here, tombstoned nodes would be silently excluded from reconcile output, causing them to persist indefinitely in Neo4j (no GC would catch them if the GC mechanism fails).

**Verdict**: DO NOT ADD TOMBSTONE GUARD HERE — would break cleanup correctness. Tombstoned nodes appearing in `neo4j_orphans` output is intentional and correct. The reconcile is the catch-all safety net for failed or missed tombstone cleanup.

**Note on Q093 post_restore mode**: The `MATCH (n:Memory) WHERE n.last_access_epoch > $pg_max_epoch` predicate used in Q093's post-restore cleanup correctly includes tombstoned nodes within the orphan range — their deletion is desired in a post-restore scenario.

### Path 3: S6 Hub-Degree Scan (Q102 Cypher)

**Status**: UNGUARDED — low impact now, fix before Month 7

**Current Cypher** (Q102 S6-THRESHOLDS):
```cypher
MATCH (n:Memory)
WITH size((n)-[:CO_RETRIEVED]->()) AS fanout
RETURN count(CASE WHEN fanout > 20 THEN 1 END) AS nodes_minor_hub,
       count(CASE WHEN fanout > 100 THEN 1 END) AS nodes_warning_threshold,
       max(fanout) AS max_fanout
```

**False-positive impact**: A tombstoned Memory node retains all its CO_RETRIEVED edges in Neo4j until the GC sweep DETACH DELETEs it. At tombstone time, the node's degree D reflects its historical reinforcement count. At Tim's Month 0–6 (max_fanout < 124), no individual node exceeds the S6 WARNING threshold of 100. A tombstoned node with degree 50–80 would appear in `nodes_minor_hub` but not trigger any alert. At Month 7+ (max_fanout approaching 158), a high-degree tombstoned hub would inflate `nodes_timeout_risk_warm` count, potentially causing premature S6 CRITICAL firing.

**Inflation model**: At steady state (2.5 tombstones/day, GC sweep every 30 days), expected tombstoned nodes in Neo4j = 2.5/day × 30 days = 75 nodes. Of these 75, a fraction carry significant degree from their pre-tombstone lifetime. If 5% of tombstoned nodes had degree > 50 at tombstone time, that adds 4 nodes to the `nodes_moderate_hub` count — less than 2% error at Month 6 node counts (~274 nodes > 20).

**Fix** (add to all S6 queries):
```cypher
MATCH (n:Memory)
WHERE n.is_tombstoned IS NULL
WITH size((n)-[:CO_RETRIEVED]->()) AS fanout
```

**Verdict**: ADD TOMBSTONE GUARD — low urgency before Month 7, required before Month 7 to prevent false S6 CRITICAL alarms.

### Path 4: Reclassification Job (Q096/Q104 Cypher)

**Status**: UNGUARDED — wasted work, not harmful

**Current Cypher** (Q096 specification):
```cypher
MATCH ()-[r:CO_RETRIEVED_FRESH]->()
RETURN count(r) AS fresh_total,
       count(CASE WHEN r.last_reinforced_epoch < ($current_epoch - $tau_edge) THEN 1 END) AS stale_classified_as_fresh
```

**Reclassification write query** (the SET operation that runs for each stale edge):
```cypher
MATCH ()-[r:CO_RETRIEVED_FRESH]->()
WHERE r.last_reinforced_epoch < ($current_epoch - $tau_edge)
SET r.edge_type = 'STALE'
```

**False-positive impact**: When a tombstoned node has FRESH edges to non-tombstoned neighbors, those edges are still traversed by the reclassification job. The reclassification correctly ages them to STALE (the last_reinforced_epoch is genuinely old). This is not incorrect — it is wasted work (the edge will be deleted when GC runs DETACH DELETE on the tombstoned source node). The wasted SET operations: 2.5 tombstones/day × ~10 FRESH edges each = 25 unnecessary Cypher SETs per day. This is computationally negligible.

The drift_fraction metric inflation: with 10 tombstoned nodes contributing ~50 FRESH edges to the denominator of 1,350, the bias is 50/1,350 = 3.7%. The drift_fraction alarm fires at 0.15 (15%). This bias cannot trigger a false alarm.

**Verdict**: OPTIONAL GUARD — adding `WHERE startNode(r).is_tombstoned IS NULL AND endNode(r).is_tombstoned IS NULL` would eliminate wasted reclassification cycles and remove the 3.7% metric bias, but neither impact is operationally significant at Tim's scale. Implement as a quality improvement with the other fixes, not as a blocker.

### Path 5: Seed Sampling in S5 and BFS (Phase 1 Seed Query)

**Status**: PARTIALLY GUARDED — seeds also need the tombstone guard

**Current Cypher** (Q092 S5 probe Phase 1):
```cypher
MATCH (n:Memory)
WHERE n.last_access_epoch IS NOT NULL
WITH n ORDER BY rand() LIMIT 5
RETURN n.id AS seed_id
```

**Note**: Q101's fix was applied to the *neighbor* path in Phase 2, not to the *seed* selection in Phase 1. A tombstoned node can be selected as a seed. If a tombstoned seed is selected:
- Phase 2 BFS from that seed traverses its CO_RETRIEVED_FRESH edges
- Those neighbor UUIDs are returned and joined against PG
- The PG join will fail for any neighbor UUIDs whose PG rows were recently deleted

This creates a subtle second contamination vector: not just stale neighbors but seeds that were themselves deleted. The compliance_ratio in S5 is directly harmed because a tombstoned seed's entire neighborhood is likely to produce misses (the seed's most recent co-retrievals were with memories that may also have been part of the same deletion batch).

**Fix**:
```cypher
MATCH (n:Memory)
WHERE n.last_access_epoch IS NOT NULL
  AND n.is_tombstoned IS NULL
WITH n ORDER BY rand() LIMIT 5
RETURN n.id AS seed_id
```

**Verdict**: MUST FIX — same severity as Path 1. A tombstoned seed can produce a cascade of PG join misses that inflates the S5 miss_count and triggers spurious WARNING alerts. Q101's fix was incomplete; it guarded neighbors but not seeds.

### Path 6: Full-Graph Node Count Queries (Health Metrics)

**Status**: UNGUARDED — minor metric inflation

**Pattern** (multiple queries in Q102 and Q104):
```cypher
MATCH (n:Memory)
RETURN count(n) AS total_nodes
```

**Impact**: At steady state (75 tombstoned nodes in Neo4j), total_nodes is inflated by 75/4,500 = 1.7%. Health metrics that divide by node count (e.g., orphan ratio, mean degree) inherit this denominator error. The bias is negligible for threshold-based alarms.

**Verdict**: OPTIONAL GUARD — add for metric accuracy but not urgent.

## Ideas Extracted

### Idea: tombstone-guard-completeness-checklist
- **Novelty**: 0.72
- **Evidence**: 0.78
- **Feasibility**: 0.96
- **Description**: The tombstone guard (`AND n.is_tombstoned IS NULL`) must be applied to both sides of every BFS traversal — not just the neighbor clause, but also the seed selection clause. Q101's fix was applied to one of two required locations. A completeness checklist for the epoch-based architecture: (1) seed sampling WHERE clause, (2) neighbor filtering WHERE clause, (3) hub-degree scan WHERE clause. The reconcile orphan scan is the deliberate exception — tombstoned nodes must appear in neo4j_orphans output because they are legitimate cleanup targets. The rule: "add the guard to every query that serves retrieval or monitoring; omit it from every query that serves cleanup."

### Idea: two-tier-tombstone-semantics
- **Novelty**: 0.76
- **Evidence**: 0.71
- **Feasibility**: 0.90
- **Description**: The tombstone exclusion guard has opposite correct behavior depending on query purpose. Queries serving retrieval/monitoring should exclude tombstoned nodes (they are noise). Queries serving cleanup/reconcile should include tombstoned nodes (they are signal — the legitimate orphan population). This two-tier semantic can be encoded as a query-decoration rule: every Cypher query tagged `purpose=retrieval` or `purpose=health_probe` automatically inherits `AND n.is_tombstoned IS NULL` from a shared Cypher fragment; every query tagged `purpose=cleanup` omits the guard. Implemented as a Python constant or Cypher template string, this prevents the Q101-style partial-fix failure mode (guard applied to one clause, missing from another).

### Idea: seed-contamination-cascade-amplifier
- **Novelty**: 0.74
- **Evidence**: 0.73
- **Feasibility**: 0.95
- **Description**: A tombstoned seed node amplifies tombstone contamination: it contributes not one but all K of its neighbors to the BFS result, and those neighbors are disproportionately likely to also be recently tombstoned (memories deleted together tend to be thematically related — the same project, session, or domain sweep). The expected S5 miss_count from a single tombstoned seed (versus a single tombstoned neighbor) is K_neighbors × P(each neighbor is also tombstoned) = 10 × (marginal conditional probability). If tombstone events are clustered (deletion of a project removes 5–10 related memories at once), the conditional probability of neighbor tombstoning given seed tombstoning could be 20–50%, not the unconditional 0.22%. A single clustered tombstone event could produce 2–5 PG join misses through the seed contamination cascade, temporarily pushing S5 past the WARNING threshold. This is a non-linear amplification that Q101's analysis did not account for, and it is why seed-level tombstone exclusion is equally important as neighbor-level exclusion.

### Idea: reconcile-as-tombstone-safety-net
- **Novelty**: 0.68
- **Evidence**: 0.77
- **Feasibility**: 0.94
- **Description**: The /reconcile endpoint's full-graph ID scan is the correct-by-design tombstone safety net. By including tombstoned nodes in the neo4j_orphans output, reconcile catches any tombstoned node that the GC sweep missed, was skipped due to error, or had its GC timer reset by an edge write. The tombstone guard should never be added to reconcile, because doing so would convert the safety net from a catch-all to a catch-most — degrading cleanup reliability. The design principle: the reconcile scan is the last line of defense and must be maximally inclusive. The BFS and health scans are first-line retrieval paths and must be maximally exclusive of deleted content.

## Recommended Fix Priority

1. **Immediate (before epoch architecture goes live)**: Add `AND n.is_tombstoned IS NULL` to both Phase 1 (seed selection) and Phase 2 (neighbor selection) of the production BFS retrieve query.
2. **Immediate**: Add `AND n.is_tombstoned IS NULL` to the S5 probe Phase 1 seed sampling query — Q101's fix was incomplete.
3. **Before Month 7**: Add `WHERE n.is_tombstoned IS NULL` to all S6 hub-degree scan queries (S6-GAUGE, S6-HISTOGRAM, S6-THRESHOLDS, S6-TOPHUBS).
4. **Quality improvement**: Add source/target tombstone exclusion to the reclassification counting and SET queries; add tombstone exclusion to full-graph node count queries.
5. **Never add**: Tombstone guard to the /reconcile full-graph orphan scan (`get_all_memory_ids()`).
