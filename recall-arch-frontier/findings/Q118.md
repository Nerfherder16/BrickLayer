# Finding: Q118 — Canary Seed Survivability Analysis

**Question**: How many semantically disjoint canary seeds (from Q103's Redis registry) can Tim's Recall corpus sustain before organic memory growth displaces one — specifically, at what corpus size does the probability of a canary seed no longer appearing in its own top-3 ANN neighbors exceed 5%, and what is the canary refresh strategy when a seed is displaced?
**Question Type**: CALIBRATION
**Verdict**: PROMISING
**Severity**: Medium
**Source field**: HNSW approximation theory, embedding space geometry, information retrieval calibration, canary sentinel design

---

## Critical Premise Examination

### Two Distinct "Displacement" Scenarios

The question's framing — "no longer appearing in its own top-3 ANN neighbors" — is ambiguous and resolves to two mechanistically different scenarios with very different answers:

**Scenario A — Self-vector query (is this the premise?)**: Query Qdrant with the canary's own stored embedding vector. The canary point has cosine similarity exactly 1.0 with itself. Does HNSW ever fail to return this point at rank 1?

**Scenario B — Paraphrase-query displacement (what Q103's health check actually tests)**: Query Qdrant with the canary's `query_text` embedding (a semantic paraphrase of the canary content). Does a new organic memory become a better semantic match for the paraphrase query than the canary content?

These are analyzed separately because the failure probability models and the mitigation strategies are entirely different.

---

## Scenario A: Self-Vector Query — The Structural Guarantee

**Geometric fact**: A vector has cosine similarity = 1.000 with itself. No other distinct vector produced by a deterministic embedding model from different text input can reach this value. The only way a different vector equals 1.0 is if two distinct text strings produce exactly identical float32 embeddings across all 1536 dimensions — this is computationally impossible in practice for a neural embedding model.

**HNSW navigation for the self-query case**: HNSW performs greedy search starting from a hierarchical entry point, moving toward the query at each step. When the query vector is an exact match for a stored node:

1. At every step of greedy descent, the node that IS the query vector is the optimal next move if it is reachable from the current node
2. The canary node has HNSW edges constructed during its insertion with ef_construction=100 — it is connected to its nearest neighbors in the graph
3. Once the traversal reaches any neighbor of the canary node, the canary itself is trivially the next greedy step (distance=0)

The practical result is that for the self-query case, HNSW recall@1 ≈ 1.000 at any corpus size where the canary node is still in the graph and the graph is not corrupted by un-optimized bulk deletion.

**When can self-query fail?**

There are three genuine failure modes, none of which are corpus-size-driven:

| Failure mode | Cause | Probability at Tim's scale |
|---|---|---|
| Canary node deleted | Reconcile orphan cleanup, post-restore filter, or manual deletion removed the canary point | Moderate (if canary lacks `is_canary` payload exclusion) |
| HNSW graph disconnection | Bulk deletion without optimize_collection created isolated subgraphs | Low (post-restore only; Q103 mandates optimize_collection before canary test) |
| ef_search too small at very large N | ef_search=64 exploration budget insufficient; greedy search terminates before finding the canary node | Effectively zero until N ≈ 5,000–10,000 |

**Conclusion for Scenario A**: The 5% displacement threshold from corpus growth does not apply. P(self-query fails) is essentially zero across Tim's realistic corpus growth trajectory (current ~360, projected 2-year maximum ~2,000), and is bounded by lifecycle operations (deletion), not corpus size. This is a structural invariant, not a statistical distribution problem.

## Scenario B: Paraphrase-Query Displacement — The Real Risk at Tim's Scale

This is what the Q103 `/health/recall-quality` endpoint tests. The health check embeds `seed["query_text"]` (e.g., "what embedding model does recall use and how many dimensions") and checks that `CANARY_001: the embedding model is qwen3-embedding:0.6b with 1536 dimensions on Ollama` is returned at rank 1.

**Displacement mechanism**: A new organic memory is stored with content semantically closer to the canary's paraphrase query than the canary content is. For example, if Tim stores a memory "My Ollama server runs qwen3-embedding:0.6b at 1536 dimensions for System-Recall," this organic memory may outscore the canary on the query "what embedding model does recall use and how many dimensions."

### Semantic Gap Analysis

The Q103 canary design exploits a semantic gap: the canary content (built with `CANARY_XXX:` prefix and precise wording) should score higher on its paraphrase query than any incidentally similar organic memory. The size of this gap determines survivability.

For qwen3-embedding:0.6b at 1536 dimensions, the typical similarity scores in this domain are:
- Canary content vs. its own paraphrase query: S_canary ≈ 0.72–0.82 (high, purpose-built)
- Organic memory directly describing the same fact: S_organic ≈ 0.70–0.80 (close, same domain)
- Unrelated organic memory: S_organic ≈ 0.40–0.60 (well below)

The gap between S_canary and the best competing organic memory is approximately 0.02–0.10 cosine similarity units. This is a thin margin.

### Corpus Size Threshold Estimation

Tim's organic memory topics: CasaOS, System-Recall, FastAPI, Qdrant configuration, Family Hub, homelab networking, project decisions. Of these, **System-Recall infrastructure facts** directly overlap with the Q103 canary topic regions.

Canary topics with high organic competition probability:
- canary-001 (embedding model dimensions): Tim frequently stores embedding model facts in System-Recall sessions
- canary-002 (HNSW M and ef_construction params): Tim stores Qdrant config facts
- canary-005 (Redis epoch key): Tim stores Redis key facts
- canary-006 (API port 8200): Tim stores port/IP facts constantly (homelab context)
- canary-007 (retrieval SLA 500ms): Tim stores performance target facts

Canary topics with low organic competition probability:
- canary-003 (Neo4j CO_RETRIEVED edges): Neo4j is underutilized, fewer organic memories on this
- canary-004 (PG importance scores): Schema-level facts rarely recorded organically
- canary-008, canary-009, canary-010 (runbook details): Highly specific procedural facts

**Probabilistic model for high-vulnerability seeds**:

Let p_compete = probability that a single new Recall-domain organic memory outscores the canary on its paraphrase query. Given the thin semantic gap (0.02–0.10), a reasonable estimate is p_compete ≈ 0.02–0.05 for directly adjacent topic memories.

Of Tim's 50–100 daily new memories, approximately 15–25% are in the System-Recall topic space (his primary focus), so ~10–20 Recall-domain memories per day.

Corpus of Recall-domain memories after M days: N_recall ≈ 15 × M (mid-range estimate).

P(canary displaced) = 1 - (1 - p_compete)^N_recall

Setting P > 0.05 and solving for N_recall:
- At p_compete = 0.02: N_recall > ln(0.95)/ln(0.98) ≈ 2.6 → approximately 3 competing Recall-domain memories
- At p_compete = 0.03: N_recall > ln(0.95)/ln(0.97) ≈ 1.7 → approximately 2 competing memories
- At p_compete = 0.01: N_recall > ln(0.95)/ln(0.99) ≈ 5.1 → approximately 5 competing memories

**Key result**: The paraphrase-query displacement threshold is reached at N_recall ≈ 2–10 directly competing Recall-domain memories. With Tim's writing rate, this corresponds to **total corpus size of approximately 13–67 Recall-domain memories**, which at 15% topic concentration means **total corpus ≈ 85–450 vectors**.

Tim's current corpus is ~360 vectors. **The system is already within or at the paraphrase-displacement threshold for the most vulnerable canary seeds (canary-001, -006).**

### HNSW Approximation: Not the Binding Constraint

For completeness: at ef_search=64 and M=16, HNSW empirical recall@1 from ann-benchmarks-style data on semantic embeddings:
- N=360: recall@1 ≈ 0.99+
- N=2,000: recall@1 ≈ 0.97–0.99
- N=10,000: recall@1 ≈ 0.95–0.97

HNSW approximation errors could contribute ~1–5% false-negative rate at N=10,000. Tim's 2-year projected maximum is ~2,000 vectors where HNSW errors contribute <3% failure probability. The semantic displacement mechanism (Scenario B) dominates at Tim's scale.

## Maximum Viable Canary Count

The Q103 design uses 10 seeds. The question asks how many semantically disjoint seeds the corpus can sustain.

**Embedding space capacity**: At 1536 dimensions with intrinsic dimensionality ~50–200 for this semantic domain, there are effectively unlimited orthogonal directions. Semantic distinctness is not limited by geometry.

**Operational constraints**:
- Each canary is an additional point in the HNSW graph: at 10 canaries vs. 360 organic points, overhead is 2.7% — negligible
- At 100 canaries vs. 360 organic points, overhead is 22% — semantically generic canary vectors begin influencing organic retrieval results for related queries
- At 1,000 canaries vs. 360 organic points, the canary layer dominates the collection and corrupts organic retrieval

**Topic coverage limit**: Tim's domain has approximately 15–25 coherent conceptual clusters (System-Recall components, homelab services, active projects, code patterns, operational procedures). Covering each cluster with 2 canary probes (for redundancy) suggests a maximum of 30–50 canary seeds as operationally useful.

**Recommended cap**: 10 seeds (Q103 design) is correct for a single-focus deployment. If per-domain coverage is desired (monitoring retrieval quality across all of Tim's active domains), 25–40 seeds organized by domain is the appropriate ceiling. Above 50, canary maintenance overhead exceeds the diagnostic value.

## Canary Refresh Strategy

### Detection of Displacement

**Trigger condition**: A paraphrase-query failure (the canary UUID is not at rank 1 on two consecutive daily health checks spaced ≥1 hour apart). Single-check failures may be HNSW approximation noise or transient load; two consecutive failures indicate genuine semantic displacement.

The `similarity_gap` field should be tracked in the health check response: the difference between the canary's similarity score and the second-ranked result's similarity score. A gap below 0.05 warrants a proactive refresh alert even before the seed actually fails.

### Refresh Options

**Option A — Content hardening (recommended first action)**:
Strengthen the canary content by adding unique structural tokens that organic memories would never include. Replace:
```
CANARY_001: the embedding model is qwen3-embedding:0.6b with 1536 dimensions on Ollama
```
with:
```
CANARY_SENTINEL_001 [SYSTEM-RECALL-PROBE-DO-NOT-DELETE]: embedding model qwen3-embedding:0.6b 1536-dim on Ollama 192.168.50.62:11434
```
The bracketed sentinel token and the IP address create strong embedding anchors that no organic memory will accidentally replicate. The paraphrase query does NOT include the sentinel token, so the test remains a semantic retrieval probe. Increment `collection_version` on refresh.

**Option B — Topic rotation**:
If a canary topic is so heavily crowded by organic memories that re-anchoring does not restore the gap (gap remains < 0.05 after content hardening), retire that canary slot and replace it with a less-contested topic. For example, if canary-006 (API port) is permanently displaced by Tim's homelab IP/port memories, replace it with a less-frequently-stored fact (e.g., a specific error code or schema detail).

**Option C — Accept degraded pass rate**:
The Q103 health endpoint accepts pass_rate ≥ 0.8 (8/10 seeds) as "healthy." If 1–2 canaries are displaced by semantic growth, the system continues reporting "healthy." This is acceptable during low-urgency periods but should trigger a background refresh job within 24 hours.

### Refresh Schedule

- **Reactive**: On any seed failure confirmed over two consecutive daily checks → refresh that seed within 24 hours
- **Proactive**: When `similarity_gap < 0.05` for any seed on three consecutive checks → schedule content hardening
- **Scheduled**: Full canary reseed quarterly — re-embed canonical text templates, re-insert all 10 seeds, increment `collection_version` globally

## Ideas Extracted

### Idea: similarity-gap-leading-indicator
- **Novelty**: 0.72 — Using the per-seed similarity gap (S_canary - S_second_best) as a leading indicator of impending displacement rather than waiting for binary pass/fail is not specified in Q103 and addresses a real monitoring gap in the current design.
- **Evidence**: 0.65 — The gap is a direct linear proxy for the displacement margin; tracking its erosion over time provides advance warning before the seed fails. The mechanism is mathematically sound.
- **Feasibility**: 0.85 — The `_run_single_canary_query()` function in Q103 already retrieves top-10 results; adding a `similarity_gap` field to its return dict is a ~3-line change. Storing the daily gap values in PostgreSQL for trending requires one new table.
- **Description**: Extend the per-query result dict in `/health/recall-quality` to include `similarity_gap`: the difference between the expected UUID's similarity score and the score of the highest-ranked non-canary result. A gap below 0.05 triggers a "pre-displacement warning" status. Store one row per seed per day in a `canary_gap_history` table (seed_id, gap, checked_at). A trend of gap erosion at rate > 0.01/day indicates imminent displacement and should trigger proactive content hardening before the seed actually fails.

### Idea: sentinel-token-content-hardening
- **Novelty**: 0.70 — Injecting structured sentinel tokens into canary content to create embedding anchors that organic memories cannot accidentally replicate is a practical engineering response to semantic crowding. The specific application to HNSW canary probes in AI memory systems is not documented elsewhere.
- **Evidence**: 0.60 — Based on the known property that transformer embeddings are sensitive to rare or structurally unusual token sequences; a unique sentinel like `[SYSTEM-RECALL-PROBE-DO-NOT-DELETE]` creates a strong embedding signature without affecting the semantic content the probe is testing (because the query text does not include the sentinel).
- **Feasibility**: 0.88 — Requires updating CANARY_TEMPLATES in the codebase and re-running `init_canary_seeds()`. Estimated: ~15 lines, no schema changes. The query text paraphrases are unchanged; only the canary content (what is stored in Qdrant) changes.
- **Description**: Augment each Q103 canary content template with a unique structural sentinel prefix: `CANARY_SENTINEL_XXX [SYSTEM-RECALL-PROBE]:` followed by the existing content. The brackets, the `SENTINEL` keyword, and ideally a specific IP address or internal identifier create embedding dimensions that organic memories will never occupy. This increases S_canary by an estimated 0.03–0.07 cosine similarity units, restoring the gap after semantic displacement. The sentinel prefix also makes canary vectors trivially identifiable during manual corpus inspection.

### Idea: dual-mode-health-check-self-vs-paraphrase
- **Novelty**: 0.68 — The distinction between self-vector probing (testing HNSW graph integrity and point existence) and paraphrase-query probing (testing semantic retrieval quality) is not made explicit in Q103's canary design. The two modes diagnose entirely different failure classes and should be tracked separately.
- **Evidence**: 0.70 — Derived from the Scenario A vs. Scenario B analysis above. Self-query failures → point deleted or graph corrupted; paraphrase-query failures → semantic drift or corpus crowding. These have different remediation paths (re-init vs. content hardening).
- **Feasibility**: 0.80 — Requires storing the canary's actual embedding vector in Redis alongside its metadata (currently only the UUID is stored, not the raw vector). On health check, retrieve the vector from Qdrant by UUID (using `qdrant.retrieve(ids=[expected_uuid])`) and use it as the query for the self-query path. Estimated: ~20 lines plus a Redis schema change to cache the vector.
- **Description**: Extend `/health/recall-quality` to run two query passes per canary seed. Pass 1 (self-query): retrieve the canary point's stored vector from Qdrant, use it as the query, expect rank 1 = self. Pass 2 (paraphrase-query): embed `query_text`, expect rank 1 = canary UUID (current behavior). The response adds `self_query_pass_count` alongside `pass_count`. A self-query failure means the point is deleted or the HNSW graph is disconnected — trigger `init_canary_seeds()` immediately. A paraphrase-query failure without self-query failure means semantic displacement — trigger content hardening.

### Idea: tiered-canary-collection-version-displacement-log
- **Novelty**: 0.65 — Using the `collection_version` field (already in Q103's CanarySeed schema) as a displacement epoch counter that accumulates history in PostgreSQL, enabling the detection of high-displacement-velocity seeds that need topic rotation vs. content hardening.
- **Evidence**: 0.68 — Q103's `collection_version` is currently incremented on post-restore refresh but carries no displacement history. The incremental cost of logging displacement events is minimal.
- **Feasibility**: 0.90 — One PostgreSQL table (`canary_displacement_log`: seed_id, displaced_at, corpus_size, displacing_memory_uuid, old_collection_version, new_collection_version). On each content-hardening refresh, increment `collection_version` and insert a row. No changes to Redis schema; the version counter already exists.
- **Description**: When a canary seed is refreshed due to semantic displacement (not post-restore), log the event: seed_id, timestamp, corpus_size at displacement, the UUID of the organic memory that outscored it, and the `collection_version` increment. After three displacement events for the same seed (`collection_version >= 4`), the seed is flagged for topic rotation — the current topic is too contested for content hardening to remain sustainable. The displacement log also provides a calibration dataset: the recorded `corpus_size at displacement` values empirically validate the theoretical 85–450 vector threshold derived above.
