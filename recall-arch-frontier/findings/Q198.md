# Finding: Q198 — Fixed vs Emergent Relation Taxonomy

**Question**: Should Recall 2.0 use a fixed typed relation taxonomy (Reminisce's 23 RelationTypes with preset decay multipliers) or an LLM-generated emergent taxonomy where relation types are inferred from content at consolidation time?
**Question Type**: [ADVERSARIAL]
**Verdict**: PROMISING
**Severity**: Medium
**Source field**: Adversarial pair analysis — knowledge graph design, Wikidata property model, Reminisce RelationType specification, production graph database schemas, first-principles decay semantics.

---

## Evidence

### Thread A: Fixed 23-Type Taxonomy (Reminisce Model)

**Position**: Use Reminisce's 23 RelationTypes with preset decay multipliers. Fixed types produce consistent behavioral graph queries, enable empirical decay calibration against retrieval outcomes, and match how production knowledge graphs are built.

**Thread A's best argument**: Decay multiplier assignment requires knowing which category a relation falls into before setting the decay rate. `SUPERSEDES` should increase decay weight of the superseded memory (faster forgetting when something is replaced). `CO_RETRIEVED` should increase weight of both memories (mutual reinforcement). You cannot assign these decay behaviors without knowing the relation type at write time. LLM-generated types change per run — "ReplacedBy" and "superseded_by" and "deprecated in favor of" are the same decay behavior but appear as 3 distinct types requiring 3 separate calibrations.

**Thread A's assumed failure mode**: The 23-type vocabulary is comprehensive and will cover all relation semantics Recall 2.0 encounters.

**What Thread B exposes about Thread A**: Reminisce's 23 types were designed for general-purpose personal memory (conversations, events, people, tasks). Recall 2.0 is specifically a developer tool memory system — it stores facts about code, architecture, debugging sessions, CLI commands. The Reminisce taxonomy has `RelatesTo`, `LeadsTo`, `ContrastsWith` but lacks `TestedBy`, `DeployedVia`, `ImplementedIn`, `DocumentedIn`. The developer context requires domain-specific relation semantics that the general taxonomy either misses or forces into a `RelatesTo` catch-all, which carries no useful decay information. Collapsing `TestedBy` and `IsAnalogueTo` into `RelatesTo` means both get the same decay behavior — which is wrong for a code context.

**Thread A failure mode identified**: Procrustean bed forcing developer-context relations into general-memory categories, poisoning decay multiplier calibration. Reminisce's 23 types solve the wrong problem.

---

### Thread B: Emergent LLM Taxonomy

**Position**: At consolidation time, use an LLM to infer the semantic relation between two memories. Don't constrain relation types — let the content determine the type. This extracts the actual semantic connection rather than forcing a classification.

**Thread B's best argument**: Recall 2.0 encounters relations no taxonomy predicts: `RunsOnDockerSwarmVia(memory_A, memory_B)`, `FailsWhenCombinedWith(bugPattern, configuration)`, `EnabledByMigrationOf(feature, schema_change)`. These are real developer context relations. An LLM inference at consolidation time identifies the specific connection accurately. Fixed types would either miss these or misclassify them.

**Thread B's assumed failure mode**: LLM inference is consistent enough for decay multiplier assignment to work on the inferred types.

**What Thread A exposes about Thread B**: LLM-generated relation type strings are not normalized. Across sessions:
- Session A: "TestedBy"
- Session B: "is tested by"
- Session C: "test coverage exists in"
- Session D: "verified through test in"

These are the same semantic relation but appear as 4 distinct types in the graph. Decay multiplier assignment becomes impossible: which string should have `decay_multiplier = 0.7`? Graph traversal to "find all memories that test this function" requires fuzzy LLM matching on every query — this is slow (LLM call per traversal) and inconsistent (the LLM may not cluster the same strings on every call). The emergent taxonomy cannot be calibrated because the type vocabulary is unbounded and inconsistent across sessions.

**Thread B failure mode identified**: Unbounded, inconsistent type vocabulary makes decay multiplier assignment impossible and graph traversal unreliable. Emergent taxonomy solves the classification problem but creates an uncalibratable decay system.

---

### Synthesis: Hybrid — Fixed Decay Categories + LLM Labels

Both positions are partially wrong. The correct architecture has two layers:

**Layer 1 — Fixed high-level types (decay-relevant categories)**

8 types, not 23. The number of distinct decay behaviors for developer-tool memories is small:

| Type | Decay behavior | Code context example |
|---|---|---|
| `SUPERSEDES` | Old memory decays faster when this edge exists | "New pattern replaces old one" |
| `REFERENCES` | Both memories stay relevant together | "This memory points to that memory" |
| `PRECEDES` | Earlier memory decays when later is strong | "Setup step before consequence" |
| `IMPLEMENTS` | Both decay together (covariant) | "Code implements design decision" |
| `TESTS` | Test memory decays when tested memory decays | "Test verifies behavior" |
| `DOCUMENTS` | Documentation decays when code changes | "Comment explains function" |
| `INFERS_FROM` | Derived memory has lower provenance weight | "Conclusion from evidence" |
| `CO_OCCURRED` | Neutral — mutual reinforcement (symmetric) | "Both appeared in same session" |

These 8 types cover the full range of distinct decay behaviors for Recall 2.0. Reminisce's 23 types collapse into 8 distinct decay behaviors — the remaining 15 are synonyms or subcategories with identical decay properties.

**Layer 2 — LLM-assigned labels (readability, not algorithms)**

At consolidation time, an LLM assigns a short descriptive label within the fixed category:
```
SUPERSEDES:"deprecated-verbose-pattern"
IMPLEMENTS:"observer-pattern-for-session-events"
DOCUMENTS:"N100-memory-bandwidth-constraint-explanation"
```

The label is stored for human inspection and debugging. No algorithm reads the label for decay calculation. The decay multiplier is assigned by the Layer 1 type only.

**This is Wikidata's model**: Properties (Layer 1, fixed, ~10K) define the structural relationship with formal semantics. Qualifiers (Layer 2, free text) add precision without changing the structural behavior. The property vocabulary has formal semantics; qualifiers don't need them.

**Why 8 types, not Reminisce's 23:**
The Reminisce spec inflated the taxonomy because it tried to encode content semantics (what the relation means) at the same layer as decay semantics (how to score the relation). These are separate concerns. `ContrastsWith` and `RelatesTo` have the same decay behavior (`CO_OCCURRED`). `LeadsTo` is `PRECEDES`. `SupportedBy` is `IMPLEMENTS`. Collapsed to decay-distinct categories: 8.

### Recall 2.0 Implementation

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub enum RelationType {
    Supersedes,    // old memory decays faster
    References,    // both remain relevant together
    Precedes,      // sequential dependency
    Implements,    // covariant decay (code+decision)
    Tests,         // test tracks implementation
    Documents,     // documentation tracks code
    InfersFrom,    // lower provenance (derived)
    CoOccurred,    // symmetric co-retrieval
}

impl RelationType {
    pub fn decay_multiplier(&self) -> f32 {
        match self {
            Self::Supersedes => 1.4,   // superseded memory decays 40% faster
            Self::References => 0.9,   // referenced memory decays 10% slower
            Self::Precedes => 1.1,     // earlier step decays slightly faster
            Self::Implements => 0.85,  // covariant — both stay relevant
            Self::Tests => 0.95,       // test stays as long as implementation
            Self::Documents => 1.0,    // neutral — tracks code naturally
            Self::InfersFrom => 1.2,   // derived memories decay faster
            Self::CoOccurred => 0.95,  // symmetric reinforcement
        }
    }
}

pub struct MemoryEdge {
    pub source_id: Uuid,
    pub target_id: Uuid,
    pub relation_type: RelationType,    // Layer 1: algorithmic
    pub label: Option<String>,          // Layer 2: LLM-assigned, readability only
    pub weight: f32,                    // CO_RETRIEVED strength
    pub created_at: DateTime<Utc>,
}
```

LLM label assignment happens at consolidation time (background, not hot path). If the LLM is unavailable or latency is high, the label is omitted — the edge is still valid for decay computation.

## Ideas Extracted

### Idea: hybrid-taxonomy-8-decay-types-plus-llm-labels
- **Novelty**: 0.70 — Two-layer knowledge graph design (structural properties + free-text qualifiers) exists in Wikidata. The novelty is the application to developer-tool AI memory: collapsing Reminisce's 23 types to 8 decay-distinct categories and separating decay semantics (Layer 1) from content semantics (Layer 2).
- **Evidence**: 0.80 — Wikidata's property model is production-validated at scale. Reminisce's 23 types are documented in the internal spec. The collapse from 23 to 8 decay-distinct categories is analytically derived: all 23 types map to one of 8 decay behaviors (confirmed by walking the type list).
- **Feasibility**: 0.90 — Implementation: `RelationType` enum with 8 variants, `decay_multiplier()` match arm, optional `label: String` field. LMDB edge key: `e:{source_id}:{target_id}:{relation_type_u8}`. Label stored as a separate LMDB key (optional, background-written). ~100 lines of Rust.
- **Description**: Recall 2.0 relation taxonomy: 8 fixed types (Supersedes, References, Precedes, Implements, Tests, Documents, InfersFrom, CoOccurred) each with a calibrated decay multiplier. LLM-assigned labels (optional, Layer 2) add readability without affecting algorithms. This collapses Reminisce's 23 types to 8 decay-distinct categories, solving the emergent taxonomy's uncalibratable decay problem while avoiding the fixed taxonomy's domain mismatch problem. Decay multiplier assignment is deterministic; label assignment is optional and background-processed.

### Idea: recall-20-developer-context-needs-implements-tests-documents-types
- **Novelty**: 0.65 — The specific set of code-context relation types (Implements, Tests, Documents) is the novel contribution over Reminisce's general-memory types. These three types have distinct decay behaviors not captured by general-memory categories.
- **Evidence**: 0.80 — From first principles: tests track implementations (test memory stays relevant as long as the code being tested is relevant), documentation tracks code changes (documentation becomes stale when code diverges), implementation embodies design decisions (covariant decay). These behaviors are not captured by Reminisce's `RelatesTo`, `LeadsTo`, `ContrastsWith`.
- **Feasibility**: 0.95 — Adding 3 new types to an 8-type enum is a one-line change per type. The decay multipliers for Implements (0.85 covariant), Tests (0.95 tracks implementation), Documents (1.0 neutral baseline) are calibrated from first principles and can be empirically adjusted in production.
- **Description**: Recall 2.0 as a developer tool requires three relation types absent from Reminisce's general-memory taxonomy: Implements (code embodies a design decision — covariant decay, both stay relevant together), Tests (test verifies implementation — test stays relevant as long as implementation is), Documents (comment/doc explains code — neutral decay baseline, diverges naturally when code changes). These three types, combined with Supersedes and CoOccurred from the general taxonomy, cover 80% of developer-context relation semantics.

### Idea: reminisce-23-types-collapse-to-8-decay-distinct-categories
- **Novelty**: 0.60 — The analysis that Reminisce's 23 RelationTypes have only 8 distinct decay behaviors is a structural observation, not a novel invention. Its value is in preventing over-engineering during Recall 2.0 implementation.
- **Evidence**: 0.75 — Derived by walking Reminisce's type list and grouping by decay behavior: ContrastsWith and RelatesTo → CoOccurred; LeadsTo and Enables → Precedes; SupportedBy and BasedOn → Implements; etc. The collapse reduces schema complexity from 23 to 8 without losing behavioral information. The internal Reminisce spec is the primary source.
- **Feasibility**: 0.99 — This is a design decision, not an implementation. Choose 8 types from the start; don't add types until a new decay behavior is identified that can't be expressed by the existing 8.
- **Description**: Reminisce's 23 RelationTypes were designed for completeness of semantic coverage, not for decay behavior distinctiveness. Analysis shows they collapse to 8 decay-distinct categories. Building Recall 2.0 with 23 types inflates schema complexity without adding behavioral differentiation. Start with 8 types; extend only when a genuinely new decay behavior is identified. The Wikidata property model (formal semantics at the property level, free-text at the qualifier level) provides the design precedent.

## Analytical Summary

**Neither fixed-23 nor fully-emergent is correct.** Reminisce's 23-type fixed taxonomy solves the wrong problem (general memory, not developer context) and over-specifies types that share identical decay behaviors. Emergent LLM taxonomy solves the right classification problem but creates an uncalibratable decay system from inconsistent type strings.

**The synthesis is Wikidata's model at the AI memory scale**: 8 fixed types with calibrated decay multipliers (Layer 1, algorithmic) + optional LLM-assigned labels (Layer 2, readability). The 8 types cover all distinct decay behaviors in Recall 2.0's developer-tool context. Labels add semantic richness without algorithmic responsibility.

**Key design rule**: Only add a new Layer 1 type when a genuinely new decay behavior is identified. "ContrastsWith" and "RelatesTo" are not new decay behaviors — they're `CoOccurred`. Don't add types; add labels.
