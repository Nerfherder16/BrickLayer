## Evidence

### Mechanism 1: ARC — Adaptive Replacement Cache

**Origin**: Nimrod Megiddo and Dharmendra S. Modha, "ARC: A Self-Tuning, Low Overhead Replacement Cache," USENIX FAST 2003. Implemented in IBM's storage products, ZFS (FreeBSD/Solaris/Linux ZFS), and studied extensively in database buffer management.

**Precise description of the data structure:**

ARC maintains four lists, not two. Each list holds cache keys (not the data itself — the data occupies a single fixed-size pool):

- **T1**: Pages seen exactly once recently (recency list). These are pages that entered via a cache miss and have not been re-referenced.
- **T2**: Pages seen at least twice (frequency list). A page moves from T1 to T2 on its second access.
- **B1**: Ghost list for pages evicted from T1. Holds the key only — no data. Acts as a "recently evicted, was-recency-only" register.
- **B2**: Ghost list for pages evicted from T2. Holds the key only. Acts as a "recently evicted, was-frequently-accessed" register.

**The adaptive parameter `p`:**

`p` is an integer in `[0, c]` where `c` is total cache capacity. It represents the target size of T1. The algorithm adapts `p` based on which ghost list a miss hits:

- Cache miss and the page key is found in **B1** (we evicted a recency-only page and it's being requested again): `p = min(p + max(1, |B2|/|B1|), c)` — increase T1's target. We should have kept recency pages longer.
- Cache miss and the page key is found in **B2** (we evicted a frequently-accessed page and it's being requested again): `p = max(p - max(1, |B1|/|B2|), 0)` — decrease T1's target. We should have kept frequency pages longer.
- Cache miss with no ghost entry: `p` unchanged.

**Eviction rule when cache is full:**

The algorithm decides which list to evict from based on the current sizes relative to `p`:

```
if |T1| >= max(1, p):
    evict LRU end of T1, move key to B1
else:
    evict LRU end of T2, move key to B2
```

**Key property**: ARC is scan-resistant. A sequential scan (accessing N distinct pages once each) fills T1 but does not pollute T2, because none of the scan pages are accessed twice. When T1 grows due to scan, `p` adapts downward if B2 hits occur, preserving the frequently-accessed pages in T2.

**Input/output specification:**
- Input: page request (page ID)
- Output: cache hit or miss; on miss, which page to evict (if full)
- State maintained: T1 (doubly-linked list + hash map), T2, B1, B2, scalar `p`, total cache size `c`
- Update rule on access: if hit in T1 → move to MRU end of T2; if hit in T2 → move to MRU end of T2; if miss with ghost hit → adjust `p`, move page into T2 (it was seen before); if miss with no ghost → insert at MRU end of T1

**Failure modes**: B1/B2 ghost lists themselves consume memory proportional to 2c entries; if cache is very small, ghost overhead matters. ARC also requires patent licensing (IBM Patent US6996676) — ZFS uses it under specific licensing; this is why PostgreSQL uses Clock-Sweep instead.

### Mechanism 2: LRU-K

**Origin**: Elizabeth J. O'Neil, Patrick E. O'Neil, Gerhard Weikum, "The LRU-K Page Replacement Algorithm for Database Disk Buffering," ACM SIGMOD 1993. Implemented in IBM DB2's buffer pool manager.

**Precise description:**

LRU-K generalizes LRU by using the K-th most recent access time as the eviction key, rather than the most recent access time.

Each cached page maintains a fixed-length array of the last K access timestamps: `hist[1..K]` where `hist[1]` is the most recent and `hist[K]` is the K-th most recent.

**The eviction key** for page p is:
- If page p has been accessed fewer than K times: `BACKWARD_K_DISTANCE(p) = ∞` (treated as if infinitely old)
- If page p has been accessed at least K times: `BACKWARD_K_DISTANCE(p) = current_time - hist[K](p)`

On cache full, evict the page with maximum BACKWARD_K_DISTANCE. Pages with fewer than K accesses are evicted before any page with K or more accesses.

**K=1 is standard LRU** (distance = time since last access). **K=2** is the standard recommendation from the paper: it separates pages accessed at least twice from pages accessed only once. K=2 eliminates the "Belady's anomaly" exposure to sequential scans because a sequential scan page never reaches K=2 accesses before being evicted.

**Input/output**: same as standard cache replacement. State: per-page history array of K timestamps. Update: on access, shift timestamps: `hist[2..K] = hist[1..K-1]`, `hist[1] = current_time`. Eviction: scan for max BACKWARD_K_DISTANCE.

**Failure modes**: Full scan of all pages to find maximum BACKWARD_K_DISTANCE is O(n). In practice, maintained with a priority queue. K history storage costs K timestamps per cached page — manageable but non-trivial for very large caches.

### Mechanism 3: CLOCK-Pro

**Origin**: Song Jiang, Feng Chen, Xiaodong Zhang, "CLOCK-Pro: An Effective Improvement of the CLOCK Replacement," USENIX ATC 2005. Implemented in Linux kernel page cache (partially), and studied in relation to ARC.

**Precise description:**

CLOCK-Pro extends the basic CLOCK algorithm (a low-overhead approximation of LRU that uses a circular buffer with a single reference bit) to handle both recency and frequency, while maintaining the O(1) amortized complexity of CLOCK.

CLOCK-Pro maintains three "hands" scanning a circular list of pages:

- **Hand-hot**: Points into the "hot" region (frequently accessed pages). When it passes a hot page with reference bit set, it clears the bit. When it passes a hot page with reference bit unset, it demotes the page to cold.
- **Hand-cold**: Points into the "cold" region (recently but not frequently accessed). When it passes a cold page with reference bit set (accessed since last scan), it promotes it to hot. When it passes a cold page with reference bit unset, it evicts it.
- **Hand-test**: Manages ghost entries (like ARC's B1/B2 but unified). Tracks cold pages recently evicted. If a cold page is in the test period (ghost) and gets re-requested, it gets promoted directly to hot. This is the frequency signal.

Pages start as cold. The hot/cold ratio is dynamically adjusted based on the proportion of ghost hits.

**Key property**: O(1) amortized per operation (like CLOCK), no patent encumbrance (unlike ARC). Approximates ARC quality while being implementable without patent licensing.

### Track record in source field

**ARC**: Deployed in ZFS (in production on millions of FreeBSD, TrueNAS, and Linux ZFS systems since 2006). IBM's storage systems use it. Megiddo & Modha (2003) showed ARC performs within 5% of optimal (OPT) on database workloads and outperforms LRU by 30-50% on mixed workloads with both recency and frequency patterns. ZFS ARC manages pools measured in terabytes of production data.

**LRU-K**: IBM DB2 buffer pool uses a variant. The O'Neil et al. (1993) paper showed K=2 reduced I/O by 33% vs. LRU on TPC-B benchmark. ORCA (Oracle's buffer advisory) uses frequency-weighted variants.

**CLOCK-Pro**: Jiang et al. (2005) showed it matches ARC performance on standard database traces (OLTP, TPC-C, Financial) with constant time overhead vs. ARC's variable overhead on large ghost lists.

**Scale validated**: ZFS ARC runs with cache sizes from 512MB to 1TB in production. PostgreSQL's Clock-Sweep (a simpler CLOCK variant) manages buffer pools of hundreds of GB in enterprise deployments.

### Why AI memory systems don't do this yet

Three reasons:

1. **The problem is not perceived as a cache problem.** AI memory systems (MemGPT, mem0, Zep, Recall) think in terms of "storage" and "retrieval" — not cache eviction. The framing is "what do I store?" not "what do I evict from a bounded hot set?" Since most current systems let the store grow indefinitely, the eviction decision never becomes urgent enough to engineer carefully.

2. **The access pattern signal isn't captured.** Standard TTL-based caching (including Recall's current Redis layer) doesn't track access counts per memory. Without K-th access timestamps or ghost hit tracking, the inputs required by ARC/LRU-K/CLOCK-Pro simply don't exist. Building these inputs requires instrumenting the retrieval path to record every cache hit.

3. **The ghost list concept is non-obvious outside database systems.** The key insight of ARC — that evicted items' keys should be retained to detect when eviction was premature — requires ghost lists. This "remember what you forgot" pattern is counterintuitive and absent from AI memory system designs.

## Ideas Extracted

### Idea: arc-hot-cache

- **Novelty**: 0.85 — No production AI memory system (MemGPT, mem0, Zep, LangChain Memory, Recall) implements ARC or any frequency-aware eviction for its in-memory memory cache. Redis's built-in eviction policies include LRU and LFU approximations (allkeys-lru, allkeys-lfu) but these are applied uniformly across the entire keyspace — not as a two-tier T1/T2 structure with ghost list adaptation. The ARC structure with ghost-list-driven `p` adaptation is unimplemented.
- **Evidence**: 0.90 — Megiddo & Modha (FAST 2003) is peer-reviewed, widely cited (1000+ citations), and implemented in production ZFS on millions of systems. ARC's superiority over LRU on mixed recency+frequency workloads is a settled empirical result.
- **Feasibility**: 0.70 — Implementable in Python on top of Recall's current stack in 2-4 weeks. Redis does not natively support ARC eviction, so this would run as a Python-side data structure that sits in front of Redis: the ARC structure manages which memory IDs are "hot" (kept in Redis), and the eviction logic is executed in the FastAPI server rather than Redis. Ghost lists store only memory IDs (UUIDs), not embeddings — low memory overhead. The main implementation challenge is making the access-tracking path non-blocking.
- **Description**: Replace Recall's current TTL-only Redis eviction with an ARC-managed hot cache. The FastAPI server maintains T1/T2/B1/B2 lists (in-process, using Python dicts and deques) keyed by memory UUID. On each `recall_retrieve` call, for each returned memory: if it is in T1, move it to T2 (frequency promotion); if it is already in T2, move to T2's MRU end. When the hot cache is full and a new memory needs to be loaded from Qdrant into Redis, the ARC eviction rule selects which memory to drop. Ghost list hits adjust `p`, making the cache automatically balance toward recency-heavy workloads (fast-moving sessions) or frequency-heavy workloads (repeated project context). This replaces the current "everything expires after N minutes" model with a model where frequently-retrieved memories stay warm indefinitely and scan-resistant eviction protects high-value long-term memories from being displaced by one-off retrieval bursts.

### Idea: lru-k-scan-resistance

- **Novelty**: 0.80 — No AI memory system implements K-th access time tracking. The specific pattern of recording the last K access timestamps per memory and using the K-th oldest as the eviction key is absent from all known production AI memory systems.
- **Evidence**: 0.85 — O'Neil, O'Neil, Weikum (SIGMOD 1993) is a foundational database systems paper, peer-reviewed, with production deployment in IBM DB2. The 33% I/O reduction result on TPC-B is a concrete, replicated finding.
- **Feasibility**: 0.80 — Simpler to implement than full ARC. Requires storing only K timestamps per cached memory (K=2 is the standard recommendation). The eviction key computation is O(1) per memory (just read hist[K]). Priority queue maintenance is straightforward in Python. Can be implemented as a thin wrapper around Redis: the LRU-K priority queue lives in the FastAPI process; Redis holds the actual memory embeddings/content. Implementation is 1-2 weeks for K=2.
- **Description**: Add a per-memory access history to the Recall retrieval path. Each time a memory is returned by `recall_retrieve`, append the current timestamp to a fixed-size ring buffer (size K=2) stored in PostgreSQL alongside the memory's metadata. The hot cache eviction priority becomes: memories accessed only once are evicted before any memory accessed twice or more, regardless of recency. This makes Recall's hot cache scan-resistant: a broad "what do I know about Docker?" query that retrieves 20 memories once each does not displace the 5 memories that get retrieved in every session (project context, user preferences). The second-access threshold is the key discriminator — it separates genuinely useful memories from accidental retrievals.

### Idea: ghost-list-adaptive-ttl

- **Novelty**: 0.90 — The specific mechanism of maintaining ghost lists (keys of evicted memories, with no associated data) to detect premature eviction and adaptively extend cache lifetime is not present in any known AI memory system. This is the most structurally novel aspect of ARC relative to what current AI memory systems do.
- **Evidence**: 0.80 — The ghost list mechanism is the core contribution of ARC (Megiddo & Modha 2003) and is validated both theoretically (ARC is online-optimal relative to a class of algorithms) and empirically across ZFS production deployments.
- **Feasibility**: 0.65 — Ghost lists for Recall require: (1) a separate Redis key namespace or PostgreSQL table for evicted-but-recently-evicted memory IDs; (2) instrumentation on the retrieval path to detect ghost hits (a requested memory was recently in hot cache but evicted); (3) TTL management on ghost entries themselves (they should expire after a session boundary or a configurable time window). The complication is that Recall's memories don't have fixed "pages" — a ghost hit means "I evicted this memory UUID recently, and now it's being retrieved again." The ghost list TTL needs to be calibrated to something like 24-48 hours (one work session cycle). Implementation is 2-3 weeks including the adaptive TTL mechanism.
- **Description**: After evicting a memory from Redis hot cache, retain its UUID in a separate "ghost" set for 48 hours. On each retrieval, check if any returned memory UUID is in the ghost set. A ghost hit means: "we recently evicted this memory but the user is retrieving it again — we evicted it too early." Each ghost hit triggers an upward adjustment of the hot cache's TTL or size for memories in that domain/tag group. Conversely, if a memory ages out of hot cache and is never retrieved again (no ghost hit within 48 hours), that's evidence the TTL was appropriate or too long. This self-calibrating TTL replaces the current manually-configured Redis TTL with one that adapts to actual retrieval patterns per project context.

### Idea: clock-pro-hot-cold-memory-tiers

- **Novelty**: 0.75 — CLOCK-Pro's specific hot/cold/test-period three-hand structure has not been applied to AI memory systems. However, the general concept of memory tiers (hot/cold) is more commonly understood in the AI memory community than ARC's ghost lists, reducing novelty somewhat.
- **Evidence**: 0.80 — Jiang, Chen, Zhang (USENIX ATC 2005) is peer-reviewed. CLOCK-Pro matches ARC performance on standard database traces with O(1) amortized overhead. Linux kernel page cache incorporates related ideas.
- **Feasibility**: 0.75 — CLOCK-Pro's primary advantage over ARC is O(1) amortized eviction (versus ARC's occasional O(n) ghost list scan). For Recall's expected hot cache size (hundreds to low thousands of memories), this advantage is negligible. CLOCK-Pro is somewhat simpler to implement than ARC if starting from scratch, but ARC's conceptual mapping to Recall's needs is cleaner (T1=recency, T2=frequency maps directly to session vs. cross-session memories). Score this slightly below ARC for Recall specifically.
- **Description**: Structure Recall's hot cache as two circular lists: cold (memories accessed once in this session) and hot (memories accessed multiple times, likely to be needed again). A "test period" tracks recently evicted cold memories. Cold memories accessed during their test period (within 24 hours of eviction) are promoted directly to hot. The clock hands advance on each eviction decision, clearing reference bits and demoting hot pages that haven't been accessed since the last sweep. This produces the same recency/frequency balance as ARC without the patent encumbrance. For Recall's self-hosted deployment context, patent concerns are irrelevant, making ARC the preferred implementation — but CLOCK-Pro is the correct choice if integrating into an open-source distribution.
