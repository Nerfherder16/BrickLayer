# Finding: Q200 — Ranked Build Order for Recall 2.0

**Question**: Given all prior findings on Recall 1.0's implementation gaps AND the commercial scope reframe, produce a ranked build order for Recall 2.0's components — what gets built first to deliver fastest time-to-useful for a single self-hosted user?
**Question Type**: [CONVERGENCE]
**Verdict**: BREAKTHROUGH
**Severity**: Critical
**Source field**: Synthesis of Q001-Q199 — 199 questions across 24 waves covering substrate architecture, write integrity, behavioral scoring, deduplication, source provenance, operational health, migration strategy, and commercial design.

---

## Evidence

### The Core Constraint

**Build for Tim's 22,423-memory corpus on Day 1.** Not for a blank-slate user. The constraint is:
1. The three existing hooks (recall-retrieve.js, observe-edit.js, recall-session-summary.js) must work without modification
2. Recall 1.0's memories must migrate with zero retrieval quality regression from day one
3. Must improve measurably on Recall 1.0 retrieval quality by day 14
4. Single developer (Tim), homelab hardware (N100), no operational overhead beyond initial deployment

All 23 waves of prior findings converge on one fact: **the correct build order minimizes time to behavioral scoring**. Every other feature — health monitoring, relation taxonomy, multi-device relay, memory limits — can wait. Behavioral scoring (CO_RETRIEVED graph) is what makes Recall 2.0 meaningfully better than Recall 1.0, and it requires the embedded inference substrate to be in place first.

### Ranked Build Order

#### Phase 1: Foundation (Days 1-5) — "Make It Work"

**P1-A: Single-Binary Rust Service**
- LMDB key-value store (memories + embeddings)
- fastembed-rs INT8 embedded inference (BGE-small-en-v1.5)
- HNSW index (instant-distance crate, D=1024, ef=200, M=16)
- Basic REST API: POST /memory (store), GET /search (retrieve)
- Recall 1.0 failure it fixes: Ollama dependency (network latency, Docker dependency, offline failure)
- Write-path latency: 41ms INT8 (confirmed by Q197) — 3.8x faster than Ollama

**P1-B: Hook Compatibility Layer**
- /memory endpoint accepts the existing observe-edit.js and recall-session-summary.js payload format
- /search endpoint returns the same JSON structure as Recall 1.0
- No hook modifications required
- Recall 1.0 failure it fixes: brittle multi-service integration; drops from 5 services (PG+Redis+Neo4j+Qdrant+Ollama) to 1

**P1-C: Corpus Migration Tool**
- Read Recall 1.0 memories from PostgreSQL
- Generate embeddings via fastembed-rs
- Build HNSW index from migrated embeddings
- Tiered bootstrap: importance>=0.8→15 synthetic CO_RETRIEVED, >=0.6→7, >=0.4→3, <0.4→skip (Q199)
- Tag synthetic events with source=BootstrapMigration
- Recall 1.0 failure it fixes: cold-start regression (confirmed by Q199 retrospective: starting fresh = 90-day regression)
- Runtime: ~5 minutes for 22K memories on N100 (Q172 physics ceiling)

**Phase 1 exit condition**: Tim's three hooks work. 22,423 memories are searchable. Retrieval quality matches Recall 1.0 on day one (bootstrap seeded).

---

#### Phase 2: Deduplication (Days 5-10) — "Stop Accumulating Garbage"

**P2-A: SHA-256 Exact Dedup (L0)**
- Content hash on normalize(content)
- LMDB key: `h:{sha256_hex}` → memory_uuid
- INSERT: check L0 before write; on collision update metadata (CRDT merge: max(importance), union(tags))
- Recall 1.0 failure it fixes: exact duplicate storage (same memory stored N times from multiple sessions)

**P2-B: SimHash Near-Duplicate Dedup (L1)**
- 64 random projections of the embedding vector → 64-bit SimHash fingerprint
- Band partitioning: 8 bands × 8 bits each (confirmed optimal by Q180: H≤6 threshold, FP rate 1.5%)
- LMDB band table: `band:{band_id}:{hash_bits}` → [memory_uuid, ...]
- L1 fires before fastembed-rs embedding call (saves 40ms per near-duplicate write)
- Recall 1.0 failure it fixes: 66x memory redundancy confirmed by Q013; 1,592 false alarms/write at H≤8 reduced to ~335 at H≤6

**Phase 2 exit condition**: No near-duplicate memories accumulate in the live corpus. SimHash pre-screens at H≤6. Write path: L0 (exact) → L1 (SimHash) → L2 (HNSW ANN at cosine>0.92) → store.

---

#### Phase 3: Behavioral Scoring (Days 10-21) — "Make It Smart"

This is the phase that makes Recall 2.0 fundamentally better than Recall 1.0.

**P3-A: CO_RETRIEVED Event Collection**
- Every /search response includes retrieved memory UUIDs
- Store a CO_RETRIEVED event batch: (memory_a_id, memory_b_id, session_id, timestamp) for all pairs in each retrieved set
- mpsc channel buffer (lock-free, ~50ns per event — confirmed Q197)
- Flush worker: runs every 15 minutes, writes LMDB edge key: `co:{uuid_a}:{uuid_b}` → weight (f32)
- Weight update: w = w + learning_rate × (1 - w) — bounded increment
- Recall 1.0 failure it fixes: importance scores as retrieval weights (Q016 BREAKTHROUGH: importance is a write-time prior, provably wrong as retrieval proxy)

**P3-B: Retrieval Scoring Formula**
- Replace cosine-only search with three-signal formula:
  ```
  score = (cosine × 0.6) + (co_retrieved_gravity × 0.2) + (importance × 0.2)
  ```
  where `co_retrieved_gravity` = mean CO_RETRIEVED weight of memory's top-K graph neighbors
- The importance component retains backward compatibility with bootstrapped scores
- Q195's provenance multiplier added here: `final_score = score × provenance.trust_multiplier()`
- Recall 1.0 failure it fixes: flat cosine retrieval misses memories that are contextually relevant to the current session but not semantically similar to the query

**P3-C: K-means CO_RETRIEVED Bootstrap**
- One-time background job after migration
- K=200 K-means clustering on the 22K embedding vectors
- Generate 1.25M initial CO_RETRIEVED edges at weight=0.1
- Provides 7x graph density above the retrieval threshold (Q174: organic accumulation alone requires 5-6 years)
- Runtime: 4-5s clustering + 7-42 minutes LMDB write (batch size dependent)
- Recall 1.0 failure it fixes: zero behavioral graph on day 1 (the complete absence confirmed by Q166)

**Phase 3 exit condition**: Retrieval quality measurably better than Recall 1.0 within 14 days. CO_RETRIEVED edges accumulate organically. The behavioral graph is seeded and growing.

---

#### Phase 4: Source Trust Provenance (Days 21-28) — "Trust What Users Said"

**P4-A: SourceProvenance Enum**
- 5 tiers: UserDirect(0.9), VerifiedSystem(0.8), ToolOutput(0.5), AgentGenerated(0.4), Derived(0.3)
- Field: `provenance: SourceProvenance` on every memory
- Hook-type mapping: UserPromptSubmit→UserDirect, PostToolUse:Write→ToolOutput, PostToolUse:Bash→VerifiedSystem, Stop→AgentGenerated, direct API→UserDirect
- trust_multiplier() × base retrieval score
- Legacy memories: default provenance = UserDirect (conservative, appropriate for high-confidence older memories)
- Recall 1.0 failure it fixes: hallucination amplification (Q195 BREAKTHROUGH — no production system defends against agent-inferred facts stored at full retrieval weight)

**Phase 4 exit condition**: Agent-generated memories surface at 55-67% less retrieval prominence than user-stated facts. Source provenance visible in the dashboard.

---

#### Phase 5: Operational Layer (Days 28-35) — "Know When It's Working"

**P5-A: Health Signals**
- S1: Top-K consistency ratio (HNSW query stability indicator)
- S2: CO_RETRIEVED edge density per user (behavioral graph health)
- S3: Bootstrap bias remaining: synthetic_weight / total_weight (provenance tracker)
- S4: Write-path latency p95 (embedding inference health)
- Recall 1.0 failure it fixes: no retrieval quality health signals (Q071 — infrastructure vs retrieval quality distinction)

**P5-B: Basic Dashboard**
- Memory count, recent writes, search query history
- Bootstrap calibration progress indicator (Q193: cold-start visibility reduces early churn)
- Source provenance breakdown (% UserDirect vs AgentGenerated)

**Phase 5 exit condition**: Tim can see that the system is working. Bootstrap progress is visible. No silent failures.

### Minimum Viable Recall 2.0 Definition

Recall 2.0 is viable for launch when ALL of the following are true:

1. **Hook compatibility**: recall-retrieve.js, observe-edit.js, recall-session-summary.js work without modification
2. **Embedded inference**: No Ollama dependency; fastembed-rs INT8 provides sub-50ms write path
3. **Dedup active**: L0 (SHA-256) + L1 (SimHash H≤6) prevent garbage accumulation from day one
4. **Behavioral graph seeded**: K-means bootstrap (K=200) provides initial CO_RETRIEVED graph; flush worker accumulates organic events
5. **Scoring formula active**: cosine×0.6 + CO_RETRIEVED×0.2 + importance×0.2 × provenance.trust_multiplier()
6. **Migration complete**: Recall 1.0 corpus imported with tiered bootstrap (importance≥0.4 seeded)
7. **Source provenance tracked**: New writes tagged by hook type; hallucination amplification defense active

That is the MVP. **35 days of focused single-developer work.**

## Ideas Extracted

### Idea: recall-20-mvp-7-conditions-35-day-build
- **Novelty**: 0.65 — The specific synthesis of 199 questions into a 35-day build plan with 7 launch conditions is new. The novelty is not in any individual finding but in the integration: this is the first time the 199-question research body has been translated into a concrete, ordered, time-bounded implementation plan.
- **Evidence**: 0.90 — Every component in the build plan is individually validated by prior findings: write-path latency (Q197), SimHash threshold (Q180), CO_RETRIEVED bootstrap (Q170-Q174), source provenance (Q195), migration strategy (Q199), hook compatibility (Q190). The 35-day estimate is derived from component complexity assessments across the wave analysis.
- **Feasibility**: 0.85 — Single developer (Tim), homelab hardware. Each phase has confirmed physics (write-path latency, corpus migration time, K-means bootstrap runtime). The plan has no speculative components — every part has been validated in prior waves.
- **Description**: Recall 2.0 MVP requires 5 build phases over 35 days: Phase 1 (Days 1-5): single-binary Rust service + hook compatibility + migration tool. Phase 2 (Days 5-10): SHA-256 exact dedup + SimHash near-duplicate dedup. Phase 3 (Days 10-21): CO_RETRIEVED event collection + behavioral scoring formula + K-means bootstrap. Phase 4 (Days 21-28): source trust provenance (BREAKTHROUGH defense). Phase 5 (Days 28-35): health signals + dashboard. 7 launch conditions define MVP. Components deferred to v2.1: relation taxonomy, relay, memory limits, decay mechanism.

### Idea: behavioral-scoring-is-the-differentiating-phase-phases-1-2-are-table-stakes
- **Novelty**: 0.60 — The finding that Phases 1 and 2 (substrate + dedup) are "table stakes" and Phase 3 (behavioral scoring) is the differentiating feature clarifies build priority. Phases 1-2 are necessary but not sufficient for Recall 2.0 to be meaningfully better than Recall 1.0.
- **Evidence**: 0.85 — Phase 1 replaces Ollama with embedded inference (3.8x faster, confirmed Q197) — improvement but not differentiation. Phase 2 eliminates redundancy (66x reduction, confirmed Q013) — hygiene but not differentiation. Phase 3 replaces importance scores with organic behavioral data (Q016 BREAKTHROUGH) — this is the feature that makes Recall 2.0 qualitatively different: "the system learns what you actually need vs what the agent thought you needed at write time."
- **Feasibility**: 0.90 — CO_RETRIEVED event collection, flush worker, K-means bootstrap, and scoring formula are individually specced (Q166-Q174). Combined implementation: ~800 lines of Rust. Timeline: 11 days of Phase 3 is the correct allocation — it's the most complex and the most valuable phase.
- **Description**: Phases 1-2 of Recall 2.0 make the system faster and cleaner than Recall 1.0 but do not change the fundamental retrieval model. Phase 3 (behavioral scoring via CO_RETRIEVED) is the architectural differentiation: it replaces write-time agent predictions (importance scores) with organic user behavior as the retrieval signal. The 11-day Phase 3 allocation (vs 5 days for Phase 1 and 5 days for Phase 2) reflects this priority correctly. Do not skip or defer Phase 3 to reach "MVP faster" — without behavioral scoring, Recall 2.0 is Recall 1.0 with a faster embedding service.

### Idea: source-trust-provenance-is-phase-4-not-optional-in-mvp
- **Novelty**: 0.70 — The classification of source trust provenance as Phase 4 (before operational layer, after behavioral scoring) reflects a judgment call about MVP scope. The novelty: making provenance a launch condition rather than a v2.1 feature.
- **Evidence**: 0.85 — Q195's hallucination amplification finding is BREAKTHROUGH severity: without provenance weighting, agent-generated content stored by observe-edit hook accumulates behavioral weight equal to user-stated facts, compounding hallucinations across sessions. The observe-edit hook fires on every PostToolUse:Write event — Claude writes code constantly, and observe-edit extracts facts from that code. Without provenance weighting, Recall 2.0 would amplify errors at the frequency of Claude's code writes. This is not a v2.1 concern.
- **Feasibility**: 0.90 — Provenance implementation: ~220 lines Rust (schema field + enum + scoring multiplier + hook-side reporting). 7 days of Phase 4 is correct. The hook-type → provenance mapping is automatic (zero user burden, zero LLM calls, zero latency).
- **Description**: Source trust provenance (Q195 BREAKTHROUGH) must be in the Recall 2.0 MVP, not deferred to v2.1. Rationale: the observe-edit hook fires on every Claude file write — without provenance weighting, every PostToolUse:Write event potentially stores agent-generated content at full retrieval weight. In an active Claude Code session, this means multiple agent-inference extractions per hour accumulating at the same retrieval prominence as user-stated facts. The hallucination amplification problem compounds with usage. Launching without provenance weighting and adding it in v2.1 means the behavioral graph is seeded with unweighted agent-generated content that takes 90+ days to correct. Include it in Phase 4 (7 days) to prevent the v2.1 correction cost.

## Analytical Summary

**The ranked build order is:**
1. Foundation: single binary + hook compatibility + migration (Days 1-5)
2. Deduplication: L0 SHA-256 + L1 SimHash (Days 5-10)
3. Behavioral scoring: CO_RETRIEVED collection + formula + bootstrap (Days 10-21)
4. Source trust provenance: hook-type mapping + retrieval multiplier (Days 21-28)
5. Operational layer: health signals + dashboard (Days 28-35)

**Minimum viable launch conditions** (7 required): hook compatibility, embedded inference, dedup active, behavioral graph seeded, scoring formula active, migration complete, provenance tracked.

**Total MVP estimate**: 35 days of focused single-developer Rust implementation.

**The sequence is not negotiable**: Phase 3 (behavioral scoring) requires Phase 2 (clean dedup) which requires Phase 1 (substrate). Phase 4 (provenance) seeds new writes correctly; applying it before the behavioral graph is seeded means provenance is clean from graph population. Phase 5 (health) observes the live system after all components are active.

**The key insight from 199 questions**: Recall 2.0's competitive advantage is not faster embeddings or smaller storage. It is the behavioral scoring model — organic retrieval patterns determining what surfaces, not write-time agent predictions. Everything else is enabling infrastructure.
