# Q171 — Oracle Redis Cold-Start Adversarial Analysis

**Question**: If the oracle query vectors are pre-stored in Redis at seed time (Q165's primary fix) but Redis is flushed or restarted before the oracle health check fires, does the health check degrade gracefully to the live-embed fallback path — or does it return a false UNHEALTHY verdict that triggers an automated recovery action before the pre-stored vectors are repopulated?

**Question type**: ADVERSARIAL
**Priority**: High
**Verdict**: BREAKTHROUGH

---

## Adversarial Analysis: Oracle Cold-Start After Redis Flush

---

### Thread A: Defense — The health check degrades gracefully

**Core claim**: Q165's proposed implementation already includes a nil-guard before using the pre-stored query vector, and the fallback is live embedding — not UNHEALTHY. A clean Redis restart does not corrupt the health signal, it merely returns the check to its pre-Q165 baseline behavior (slower, but not wrong).

**Evidence**:

1. **Q165's pseudocode is nil-guarded by design.** The exact code block proposed in Q165 (Finding section, "Health check modification") reads:
   ```python
   oracle_embedding_raw = await redis.client.get(
       f"recall:oracle:query_embedding:{oracle['oracle_id']}"
   )
   if oracle_embedding_raw:
       # Fast path: ANN search with pre-computed vector — no Ollama call
       query = MemoryQuery(embedding=json.loads(oracle_embedding_raw), limit=10)
   else:
       # Fallback: live embed (cold path, first seed or Redis eviction)
       query = MemoryQuery(text=oracle["probe_query"], limit=10)
   ```
   The `else` branch is explicitly documented as handling "first seed or Redis eviction." The author of Q165 anticipated this scenario.

2. **Redis restart behavior is bounded by AOF persistence.** The docker-compose.yml configures Redis with `command: redis-server --appendonly yes`. AOF (Append-Only File) persistence means every write operation is journaled to disk (`/data/appendonly.aof` inside the container, backed by the `redis_data` named volume). On container restart — whether via `docker compose restart`, CasaOS container manager, or OOM kill — Redis replays the AOF log and all keys are restored. The registry key `recall:oracle:registry` and query embedding keys `recall:oracle:query_embedding:*` survive restarts because they were written without TTL and AOF is enabled.

3. **No automated recovery action is wired.** The health check as specified in Q148 returns a status object (`{"status": "HEALTHY"/"DEGRADED"/"UNINITIALIZED", ...}`). No scheduler, ARQ worker, or webhook fires on UNHEALTHY status. The Recall codebase has no watchdog that monitors `/admin/health/recall-quality` and auto-triggers seed repopulation. UNHEALTHY is a dashboard metric, not an operational trigger.

4. **Redis eviction policy defaults to `noeviction`.** The docker-compose.yml sets no `maxmemory` limit and no `maxmemory-policy` directive. Redis 7's default when `maxmemory` is unset is `noeviction` — it refuses new writes rather than evicting existing keys. The oracle query embedding vectors (5 × 1024 float32 = ~20 KB total) cannot be silently evicted under memory pressure when the policy is `noeviction`.

**Best case scenario**: Redis restarts, AOF replays, all oracle keys are present within ~1 second, the next health check fires 60 seconds later and reads the pre-stored vectors correctly. The restart is a non-event operationally.

---

### Thread B: Attack — Redis flush creates a false UNHEALTHY cascade

**Core claim**: The nil-guard in Q165's pseudocode is a proposal, not deployed code. When the oracle health endpoint is implemented without this guard — a realistic error of omission in a first-pass implementation — the check receives a nil registry or nil embedding vectors, passes empty/nil UUIDs into Qdrant queries, gets zero results back, scores 0/5 oracles retrieved, and emits UNHEALTHY when the correct verdict is UNINITIALIZED.

**Evidence**:

1. **The oracle health check does not exist yet.** Searching the entire `src/` directory for "oracle", "recall-quality", "UNHEALTHY", and "UNINITIALIZED" returns zero matches. The endpoint is a spec (Q148) and a fix proposal (Q165) — not deployed code. Any implementation will be written by a human or agent from these specs. The nil-guard exists in pseudocode, not in a test suite or type system. Implementation errors of omission are the most common class of spec-to-code divergence.

2. **The oracle registry cold-start path has two independent nil cases.** Even if the query embedding nil-guard is implemented correctly, the oracle registry itself (`recall:oracle:registry`) must also be nil-guarded. If `redis.client.get("recall:oracle:registry")` returns nil, the health check must decide: iterate over an empty list (returning `oracles_retrieved=0` → UNHEALTHY) or return UNINITIALIZED. The Q148 spec defines UNINITIALIZED as the correct response when "oracle registry does not exist in Redis." But if the implementation decodes nil as an empty list and scores it as `0/5`, it emits UNHEALTHY — a false verdict.

3. **AOF persistence is not a guarantee against all flush scenarios.** AOF protects against clean container restart. It does not protect against:
   - `docker compose down -v` (destroys named volumes including `redis_data`)
   - CasaOS storage migration or volume backup/restore to a different location
   - Redis container rebuilt from scratch (new volume, no historical AOF)
   - Manual `redis-cli FLUSHALL` (synchronous, flushes AOF-tracked keys immediately)
   - AOF file corruption (Redis detects and stops, requiring manual `redis-check-aof --fix`)
   The Q171 question specifically asks about "Redis is flushed" — FLUSHALL or volume recreation both bypass AOF. These are realistic homelab scenarios.

4. **Partial eviction under AOF + maxmemory is possible at deployment time.** At the time Q165's fix is implemented, the operator may configure `maxmemory` to limit Redis memory consumption (a reasonable production practice). If `maxmemory` is set without explicitly setting `maxmemory-policy`, the default in some Redis 7 distributions is `noeviction`, but CasaOS Redis images may be configured differently. More critically: if an operator later runs `CONFIG SET maxmemory-policy allkeys-lru` for other reasons (e.g., to handle session working memory pressure), the oracle query vectors become eligible for silent LRU eviction despite having no TTL. This creates a partial degradation: 2/5 oracle vectors evicted, health check returns DEGRADED with misleading scores, 3 oracles pass but 2 hit the fallback embed path and may timeout under GPU contention.

5. **UNHEALTHY status, if emitted falsely, may trigger human panic responses.** Even without automated recovery, a false UNHEALTHY on a health endpoint causes operators to initiate manual recovery procedures: inspecting logs, running seed scripts, restarting services. If the operator runs the seed script while oracles already exist in Qdrant, the dedup logic returns `"created": false` for all 5 oracles (content hash collision), the operator receives ambiguous feedback, and the correct fix (re-writing Redis keys) is not obvious from the seed script output alone.

**Failure mode**: FLUSHALL or volume wipe → registry nil → implementation without nil-guard → zero-score health check → UNHEALTHY emitted → operator runs seed script → Qdrant returns "duplicate" for all 5 memories (they survive in Qdrant, only Redis was flushed) → operator confused → manual redis-cli SET required but not in runbook → 15-30 minute outage of health signal.

## Synthesis

**What Thread A assumes that Thread B exposes**:

Thread A assumes that Q165's nil-guard will be implemented correctly in the first pass of the oracle health endpoint. This is an assumption about implementation fidelity, not a code guarantee. Thread B exposes that the nil-guard is in pseudocode only, and that two separate nil cases exist (registry nil and per-oracle embedding nil) — both must be guarded independently. Thread A also assumes that AOF persistence covers all flush scenarios; Thread B correctly identifies that FLUSHALL and volume recreation bypass AOF.

**When Thread A is right**:
- Clean container restarts (the most common operational scenario on CasaOS)
- When Q165's nil-guard is implemented as written (both registry nil and per-embedding nil paths)
- When no `maxmemory-policy` is set (current docker-compose configuration, `noeviction` default)
- When the health check has no automated recovery trigger (confirmed: none exists)

**When Thread B is right**:
- When Redis is explicitly flushed (`redis-cli FLUSHALL`) or volumes are recreated
- When the oracle health endpoint is implemented without the registry nil-guard (treats nil as empty list → UNHEALTHY)
- If an operator adds `maxmemory` configuration without explicitly pinning the oracle keys
- The "Qdrant alive but Redis flushed" scenario creates a genuinely ambiguous state: oracles exist in durable storage (Qdrant) but their Redis-resident routing keys are gone — the health check cannot reach them without re-seeding Redis

**The third option neither thread saw alone**:

Both threads treat Redis persistence as a binary (present or absent). The real issue is that the oracle system has two distinct durability tiers: oracle *memory content* lives in Qdrant (durable, volume-backed, not affected by Redis flush) and oracle *routing metadata* lives in Redis (semi-durable, AOF-protected but vulnerable to explicit flush). Q165's fix adds a third item to the Redis tier: oracle *query embeddings*. The architecture now has three stores for oracle-related data, with the weakest link (Redis) holding the keys that unlock the other two.

The synthesis reveals: the correct fix is not just a nil-guard (Thread A's implied solution) — it is to treat a nil oracle registry as a signal to attempt *auto-recovery from Qdrant* rather than returning UNINITIALIZED or UNHEALTHY. If `recall:oracle:registry` is nil but Qdrant contains memories tagged `["oracle", "health-check"]`, the health check can reconstruct the registry from Qdrant's tag filter, re-write Redis, and proceed — making the cold-start invisible to the operator.

## Required Fix

The oracle health check implementation must handle three distinct nil scenarios explicitly:

**Scenario 1 — Registry nil, oracle memories present in Qdrant (Redis flush without volume wipe)**

```python
# Auto-recovery path
registry_raw = await redis.client.get("recall:oracle:registry")
if registry_raw is None:
    # Attempt Qdrant-based recovery: find all oracle-tagged memories
    oracle_memories = await qdrant.scroll(
        scroll_filter={"must": [{"key": "tags", "match": {"any": ["oracle", "health-check"]}}]},
        limit=10,
    )
    if oracle_memories:
        recovered_ids = [str(m.id) for m in oracle_memories]
        await redis.client.set("recall:oracle:registry", json.dumps(recovered_ids))
        registry = recovered_ids
        logger.warning("oracle_registry_recovered_from_qdrant", count=len(recovered_ids))
    else:
        return {"status": "UNINITIALIZED", "reason": "no oracle memories found in Qdrant"}
```

**Scenario 2 — Per-oracle embedding nil (Q165 nil-guard, already specified)**

```python
embedding_raw = await redis.client.get(f"recall:oracle:query_embedding:{oracle_id}")
if embedding_raw:
    query = MemoryQuery(embedding=json.loads(embedding_raw), limit=10)
else:
    # Fallback: live embed (slower, GPU-contention-sensitive, but correct)
    query = MemoryQuery(text=oracle["probe_query"], limit=10)
```

**Scenario 3 — Eviction protection for oracle embedding keys**

After writing oracle query embeddings, immediately pin them against LRU eviction:
```bash
# After SET recall:oracle:query_embedding:{id} <vector>
redis-cli OBJECT PERSIST recall:oracle:query_embedding:{id}  # noop but documents intent
# Actual protection: set maxmemory-policy noeviction or use OBJECT ENCODING to verify
```

The docker-compose already defaults to `noeviction` (no `maxmemory` set), so this is currently a non-issue. The risk surfaces if the operator adds `maxmemory` later. A defensive comment in the seed script and a warning in the runbook is sufficient: "If you add maxmemory to Redis config, set maxmemory-policy to noeviction or volatile-lru, and ensure oracle keys have no TTL."

**The seed script must distinguish two FLUSHALL recovery states:**
- State A: Qdrant intact, Redis flushed → auto-recovery from Qdrant (new behavior above)
- State B: Qdrant also lost → full re-seed required (existing seed_oracles.sh flow)

## Key Findings

1. **The oracle health endpoint does not exist in the codebase.** Both Q148 (spec) and Q165 (optimization) are unimplemented proposals. The adversarial analysis is about implementation design, not deployed behavior. This must be stated clearly in any implementation ticket.

2. **Redis AOF protects against the most common scenario (container restart) but not explicit flush.** `docker-compose.yml` has `--appendonly yes` with a named volume — this covers CasaOS container manager restarts, host reboots, and OOM kills. It does not cover `FLUSHALL` or `docker compose down -v`.

3. **The nil-guard for oracle registry is a two-case problem, not one.** Q165's pseudocode guards against nil per-oracle embedding. It does not guard against nil oracle registry. Both guards are required, and the registry nil case is the more dangerous one because it can masquerade as UNHEALTHY (0/5 oracles retrieved) rather than UNINITIALIZED.

4. **Redis default eviction policy (`noeviction`) protects oracle keys.** With no `maxmemory` configured, Redis cannot silently evict the oracle query embedding vectors. Thread B's LRU eviction scenario requires an operator to have explicitly set `maxmemory` — which they have not done. This is the clearest win for Thread A.

5. **No automated recovery action is wired to UNHEALTHY.** Confirmed by full source search. A false UNHEALTHY causes operator confusion and manual intervention, not a cascade failure. This bounds the blast radius.

6. **The architecture synthesizes a self-healing recovery path.** If `recall:oracle:registry` is nil but oracle memories are tagged in Qdrant, the health check can auto-reconstruct the registry via a Qdrant scroll filter on `tags: ["oracle", "health-check"]`. This eliminates the operator manual step (redis-cli SET) for the Redis-flush-without-Qdrant-wipe scenario — which is the most likely real-world case.

7. **The Qdrant oracle memories are durable regardless of Redis state.** Oracle memories are stored with `durability: "permanent"` in Qdrant (volume-backed, no TTL, decay-exempt). Qdrant is the reliable tier. Redis is the fast-lookup tier. The architecture should exploit this asymmetry: Qdrant is truth, Redis is cache — and the health check should treat them accordingly.
