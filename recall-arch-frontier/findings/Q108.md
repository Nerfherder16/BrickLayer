# Finding: Q108 — pgvector as Neo4j Replacement: CO_RETRIEVED CTE Architecture

**Question**: Is pgvector (PostgreSQL vector extension) a viable replacement for Neo4j in the Recall architecture — specifically, can the CO_RETRIEVED graph traversal be replicated as a `co_retrievals` CTE self-join in PostgreSQL, what is the latency comparison at Tim's graph size (5,000 edges at Month 24), and under what conditions (graph size, traversal depth, edge type diversity) does Neo4j become strictly necessary?
**Question Type**: ARCHITECTURE
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: PostgreSQL query planning, B-tree partial index cost model, graph traversal complexity theory, Q091/Q094/Q096/Q099 prior findings, CTE materialization semantics

## Architecture Analysis

### Current Neo4j Capability

Neo4j provides BFS traversal via pointer-chasing through per-type relationship chains. The breakthrough of Q091 was that Neo4j organizes relationship records in **separate linked lists per relationship type** — traversing `CO_RETRIEVED_FRESH` only reads the FRESH sub-chain, achieving structural pre-filtering that a property index cannot. This is the property that makes Neo4j feel indispensable for the Recall use case.

What Neo4j provides that is genuinely hard to replicate:
1. **Pointer-chain traversal**: each relationship record has a `next` pointer to the next relationship in the same node's chain. BFS is a linked-list walk with no query planner overhead.
2. **Per-type chain isolation**: `CO_RETRIEVED_FRESH` and `CO_RETRIEVED_STALE` live in separate chains. Following only the FRESH chain physically skips STALE records without a filter predicate.
3. **Native depth-N BFS**: Cypher `*1..N` applies the same pointer-walk recursively without the query planner materializing intermediate result sets.

What Neo4j does NOT provide at Community Edition:
- `SET TYPE` (must CREATE+DELETE — Q096)
- APOC procedures for batch operations
- Multi-statement transactions across type transitions
- SERIALIZABLE isolation (READ_COMMITTED only — Q099)

### PostgreSQL CTE Equivalent

**Schema:**

```sql
CREATE TABLE co_retrievals (
    from_memory_id  UUID NOT NULL REFERENCES memories(id),
    to_memory_id    UUID NOT NULL REFERENCES memories(id),
    relationship_type VARCHAR(8) NOT NULL CHECK (relationship_type IN ('FRESH','STALE')),
    last_reinforced_epoch INTEGER NOT NULL,
    weight          FLOAT DEFAULT 1.0,
    created_at      TIMESTAMPTZ DEFAULT now(),
    PRIMARY KEY (from_memory_id, to_memory_id)
);

-- Partial index: only FRESH rows, clustered on from_memory_id
CREATE INDEX idx_co_ret_fresh ON co_retrievals(from_memory_id)
    WHERE relationship_type = 'FRESH';

-- Supporting index for reverse-direction traversal and hub detection
CREATE INDEX idx_co_ret_to_fresh ON co_retrievals(to_memory_id)
    WHERE relationship_type = 'FRESH';
```

The partial index on `relationship_type = 'FRESH'` is the PostgreSQL structural analog to Neo4j's per-type relationship chain: only FRESH rows are indexed, so the index scan physically skips STALE rows — it never reads them. This is a genuine pre-filter, equivalent in effect to the CO_RETRIEVED_FRESH type partition.

**Depth-1 BFS (K seeds):**

```sql
WITH seeds AS (
    SELECT id FROM memories WHERE content_hash = ANY(:seed_hashes)
),
depth1 AS (
    SELECT cr.to_memory_id  AS memory_id,
           cr.weight,
           1               AS depth
    FROM   co_retrievals cr
    JOIN   seeds s ON cr.from_memory_id = s.id
    WHERE  cr.relationship_type = 'FRESH'
)
SELECT memory_id, weight, depth FROM depth1
ORDER BY weight DESC
LIMIT 50;
```

Planner uses `idx_co_ret_fresh` (partial index), reads only FRESH rows for those from_memory_ids. Equivalent to `MATCH (s)-[:CO_RETRIEVED_FRESH]->(n)` in Cypher.

**Depth-2 BFS:**

```sql
WITH seeds AS (
    SELECT id FROM memories WHERE content_hash = ANY(:seed_hashes)
),
depth1 AS (
    SELECT cr.to_memory_id AS memory_id, cr.weight, 1 AS depth
    FROM   co_retrievals cr
    JOIN   seeds s ON cr.from_memory_id = s.id
    WHERE  cr.relationship_type = 'FRESH'
),
depth2 AS (
    SELECT cr.to_memory_id AS memory_id,
           cr.weight * d1.weight AS weight,
           2 AS depth
    FROM   co_retrievals cr
    JOIN   depth1 d1 ON cr.from_memory_id = d1.memory_id
    WHERE  cr.relationship_type = 'FRESH'
      AND  cr.to_memory_id NOT IN (SELECT id FROM seeds)  -- no backtrack to seeds
)
SELECT memory_id, MAX(weight) AS weight, MIN(depth) AS depth
FROM   (SELECT * FROM depth1 UNION ALL SELECT * FROM depth2) combined
GROUP  BY memory_id
ORDER  BY weight DESC
LIMIT  50;
```

**Depth-3 BFS:**

```sql
WITH seeds AS (
    SELECT id FROM memories WHERE content_hash = ANY(:seed_hashes)
),
depth1 AS (
    SELECT cr.to_memory_id AS memory_id, cr.weight, 1 AS depth
    FROM   co_retrievals cr
    JOIN   seeds s ON cr.from_memory_id = s.id
    WHERE  cr.relationship_type = 'FRESH'
),
depth2 AS (
    SELECT cr.to_memory_id AS memory_id, cr.weight * d1.weight AS weight, 2 AS depth
    FROM   co_retrievals cr
    JOIN   depth1 d1 ON cr.from_memory_id = d1.memory_id
    WHERE  cr.relationship_type = 'FRESH'
      AND  cr.to_memory_id NOT IN (SELECT id FROM seeds)
),
depth3 AS (
    SELECT cr.to_memory_id AS memory_id, cr.weight * d2.weight AS weight, 3 AS depth
    FROM   co_retrievals cr
    JOIN   depth2 d2 ON cr.from_memory_id = d2.memory_id
    WHERE  cr.relationship_type = 'FRESH'
      AND  cr.to_memory_id NOT IN (SELECT id FROM seeds)
)
SELECT memory_id, MAX(weight) AS weight, MIN(depth) AS depth
FROM   (SELECT * FROM depth1
        UNION ALL SELECT * FROM depth2
        UNION ALL SELECT * FROM depth3) combined
GROUP  BY memory_id
ORDER  BY weight DESC
LIMIT  50;
```

**Hub protection LIMIT (analogous to Q094 circuit breaker):**

```sql
WITH seeds AS (...),
depth1 AS (
    SELECT cr.to_memory_id, cr.weight
    FROM   co_retrievals cr
    JOIN   seeds s ON cr.from_memory_id = s.id
    WHERE  cr.relationship_type = 'FRESH'
    LIMIT  50   -- hard cap per seed: hub guard, replaces K=20 fallback
),
...
```

Adding `LIMIT 50` inside each CTE level caps fan-out per seed, preventing hub explosions. This is the SQL equivalent of the K=20 circuit breaker fallback from Q094.

**Type management — FRESH to STALE promotion:**

The relationship_type column replaces the Neo4j FRESH/STALE relationship type partition. The background reclassification job (Q096) becomes a single SQL UPDATE:

```sql
UPDATE co_retrievals
SET    relationship_type = 'STALE'
WHERE  relationship_type = 'FRESH'
  AND  last_reinforced_epoch < (:current_epoch - :tau_edge);
```

Unlike Neo4j's CREATE+DELETE workaround, this is an atomic in-place UPDATE with no property-copying race condition (Q099). PostgreSQL MVCC handles concurrent reads correctly: a SELECT on the partial FRESH index sees a consistent snapshot; an UPDATE commits atomically; no two-phase CREATE+DELETE window exists.

**Race condition resolution**: The compare-and-swap WHERE clause fix from Q099 becomes unnecessary. PostgreSQL UPDATE is atomic; there is no window between reading the epoch value and writing the new type.

### Latency Comparison

Cost model basis:
- PostgreSQL partial index scan: 0.05ms per index B-tree traverse + 0.01ms per heap row (NVMe, shared_buffers warm, small table)
- PostgreSQL connection + parse + plan overhead: 0.2ms (pgBouncer or persistent connection reduces this to ~0.05ms)
- Neo4j warm BFS: 1.1ms (Q091, single hop, mean node)
- Neo4j CO_RETRIEVED_FRESH chain: 0.03ms per relationship record in chain (Q091)
- K_seeds = 3 (typical Recall query)

**At 800 edges / Month 12 (current), FRESH fraction ~60%, mean fresh fanout F=2.4:**

| Depth | PostgreSQL CTE | Neo4j (FRESH chain) | Winner |
|-------|---------------|---------------------|--------|
| depth-1 | 0.42ms (7 rows) | 0.32ms | Neo4j (1.3x) |
| depth-2 | 1.20ms (24 rows) | 0.83ms | Neo4j (1.4x) |
| depth-3 | 2.57ms (66 rows) | ~4ms (estimated) | PG |

**At 5,000 edges / Month 24, FRESH fraction ~30%, mean fresh fanout F=1.5:**

| Depth | PostgreSQL CTE | Neo4j (FRESH chain) | Winner |
|-------|---------------|---------------------|--------|
| depth-1 | 0.40ms (4 rows) | 0.24ms | Neo4j (1.7x) |
| depth-2 | 0.79ms (11 rows) | 0.44ms | Neo4j (1.8x) |
| depth-3 | 1.26ms (21 rows) | ~1ms (estimated) | Comparable |

**Hub node scenario — Month 12 (F_hub=50 FRESH edges):**

| Depth | PostgreSQL CTE | Neo4j (FRESH chain) | Winner |
|-------|---------------|---------------------|--------|
| depth-1 | 1.85ms (150 rows) | 1.60ms | Neo4j (1.2x) |
| depth-2 | 92.45ms (7,650 rows) | 229.60ms (estimated) | **PG (2.5x)** |

**Hub node scenario — Month 24 (F_hub=150 FRESH edges):**

| Depth | PostgreSQL CTE | Neo4j (FRESH chain) | Winner |
|-------|---------------|---------------------|--------|
| depth-1 | 4.85ms (450 rows) | 4.60ms | Comparable |
| depth-2 | 726.95ms (67,950 rows) | 2,038.60ms (estimated) | **PG (2.8x)** |

**Key finding on hub depth-2**: PostgreSQL CTE wins at hub nodes for depth-2 because Neo4j's depth-2 BFS must traverse *every depth-1 neighbor's* FRESH chain — the total relationship records read grows as F^2 in both models. PostgreSQL's per-row cost (0.01ms heap read) is lower than Neo4j's per-relationship cost (0.03ms chain follow + property read), so PostgreSQL's F^2 scaling is faster in absolute terms at the relevant fanouts.

**Both systems exhibit O(F^2) complexity at depth-2**. The key insight is that neither has an algorithmic advantage at depth-2; the winner is determined purely by per-record access cost, and PostgreSQL's B-tree + heap model is cheaper per row than Neo4j's linked-list chain follow + property store read for the `co_retrievals` table size range.

**Summary**: PostgreSQL is within 1.3-1.8x of Neo4j for mean-node depth-1 and depth-2. PostgreSQL is 2.5-2.8x faster than Neo4j for hub-node depth-2. Both are well within the 10ms graph sub-budget at Tim's graph size through Month 24.

### Break-Even Analysis

**Depth-2 CTE break-even by fresh fanout (F_fresh per node):**

The PostgreSQL CTE is faster than Neo4j for depth-2 when F_fresh >= 5 (at K_seeds=3). Below F=5, Neo4j's lower overhead wins. At Tim's current graph (F_fresh=2.4), Neo4j is marginally faster. At Month 24 (F_fresh=1.5 mean), Neo4j is faster for mean nodes but PostgreSQL wins at hub nodes.

| F_fresh | Neo4j d2 | PG CTE d2 | PG/Neo4j | Regime |
|---------|----------|-----------|----------|--------|
| 1 | 0.28ms | 0.58ms | 2.05x | Neo4j wins |
| 3 | 1.18ms | 1.51ms | 1.28x | Neo4j wins |
| 5 | 2.80ms | 2.67ms | 0.96x | **Break-even** |
| 10 | 10.0ms | 6.65ms | 0.67x | PG wins |
| 50 | 229.6ms | 92.4ms | 0.40x | PG wins (2.5x) |
| 150 | 2,038ms | 727ms | 0.36x | PG wins (2.8x) |

**Break-even for depth-2: F_fresh ≈ 5 edges per node.**

Tim's graph sits at F_fresh=2.4 (Month 12), meaning Neo4j is currently faster for mean nodes by ~1.4x. However this difference is ~0.4ms absolute and both are well under the 10ms budget. At hub nodes (F=50+), PostgreSQL wins decisively.

**Depth-3 break-even: neither system is acceptable.** At F_fresh=10 and depth-3, both PostgreSQL (estimated 100ms+) and Neo4j (estimated 400ms+) exceed the latency budget. Depth-3 BFS requires explicit row-count limiting at each CTE level (PG) or a LIMIT clause per hop (Cypher) regardless of store. Depth-3 BFS is not viable without per-level LIMIT in either system.

**When Neo4j becomes strictly necessary:**

Neo4j provides no latency advantage at Tim's graph size (Month 12-24, depth-1 or depth-2 with LIMIT). Neo4j would become strictly necessary only under conditions that do not apply to Recall:

1. **Depth > 3 without per-level capping**: PostgreSQL CTEs materialize each level in full. Depth-5+ BFS without aggressive LIMIT becomes a memory exhaustion risk. Neo4j's pointer-walk naturally prunes visited nodes with the `MATCH ... *1..N` visited-node tracking. For Recall, depth > 2 is not used — Q094's K=50 design caps at depth-2.

2. **Graph size > ~50,000 edges**: At large graphs where the co_retrievals table no longer fits in PostgreSQL shared_buffers, heap access becomes random I/O. Neo4j's relationship store, also heap-backed, suffers similarly, but graph databases have more sophisticated cache strategies for relationship chains. At Tim's Month 24 projection of 5,000 edges, the co_retrievals table is approximately 500KB — trivially in-memory at all times.

3. **Multi-hop path queries with path tracking**: If Recall ever needs to return the actual path between two memory nodes (not just the neighbor set), PostgreSQL CTEs cannot track path provenance efficiently. Neo4j's `shortestPath()` and `allShortestPaths()` are natively supported. Recall's current use case is neighbor-set retrieval, not path queries — so this does not apply.

4. **Graph algorithms (PageRank, community detection, centrality)**: Neo4j GDS library provides these. Without APOC or GDS, Neo4j Community Edition does not provide them either. PostgreSQL + Apache AGE or pgRouting can approximate some of these, but for the Recall use case these algorithms are not needed.

**Conclusion**: Neo4j is NOT strictly necessary at Tim's graph size through Month 24 for the depth-1/depth-2 BFS retrieval pattern. The conditions under which Neo4j becomes necessary (depth > 3 queries, > 50,000 edges, path tracking, graph algorithms) are not present in the Recall architecture as designed.

### Migration Path

The migration preserves all semantic content from the existing Neo4j graph and eliminates all Neo4j-specific complexity accumulated across Q091-Q099.

**Step 1 — Export from Neo4j (Cypher):**

```cypher
// Export FRESH edges
MATCH (a:Memory)-[r:CO_RETRIEVED_FRESH]-(b:Memory)
RETURN a.id AS from_id, b.id AS to_id,
       r.last_reinforced_epoch AS epoch,
       coalesce(r.weight, 1.0) AS weight,
       'FRESH' AS rel_type;

// Export STALE edges
MATCH (a:Memory)-[r:CO_RETRIEVED_STALE]-(b:Memory)
RETURN a.id AS from_id, b.id AS to_id,
       r.last_reinforced_epoch AS epoch,
       coalesce(r.weight, 1.0) AS weight,
       'STALE' AS rel_type;
```

Write both result sets to `/tmp/co_retrievals.csv`.

**Step 2 — Load into PostgreSQL:**

```sql
COPY co_retrievals (from_memory_id, to_memory_id, last_reinforced_epoch, weight, relationship_type)
FROM '/tmp/co_retrievals.csv'
WITH (FORMAT CSV, HEADER);

ANALYZE co_retrievals;
```

At 800 edges, this COPY completes in under 1 second.

**Step 3 — Validate:**

```sql
SELECT relationship_type, COUNT(*) FROM co_retrievals GROUP BY 1;
-- Should match Neo4j counts

SELECT COUNT(*) FROM co_retrievals cr
LEFT JOIN memories m ON cr.from_memory_id = m.id
WHERE m.id IS NULL;
-- Should be 0 (no orphaned edges)
```

**Step 4 — Update application code:**

Replace all Cypher BFS calls in `recall_api.py` (or equivalent) with the depth-2 CTE SQL above. The session-end reinforcement hook (which currently does `MERGE (a)-[:CO_RETRIEVED_FRESH]-(b) SET ...`) becomes:

```sql
INSERT INTO co_retrievals (from_memory_id, to_memory_id, last_reinforced_epoch, weight, relationship_type)
VALUES (:a, :b, :epoch, 1.0, 'FRESH')
ON CONFLICT (from_memory_id, to_memory_id) DO UPDATE
    SET last_reinforced_epoch = EXCLUDED.last_reinforced_epoch,
        weight = co_retrievals.weight + 0.1,  -- weight increment on reinforce
        relationship_type = 'FRESH';  -- promote to FRESH on any reinforcement
```

The UPSERT handles both new edges and reinforcement of existing STALE edges atomically. The CREATE+DELETE pattern from Q096 is eliminated entirely.

**Step 5 — Schedule background type maintenance:**

Replace the Neo4j reclassification job (Q096) with:

```sql
UPDATE co_retrievals
SET    relationship_type = 'STALE'
WHERE  relationship_type = 'FRESH'
  AND  last_reinforced_epoch < (:current_epoch - :tau_edge);
```

Run at session end, same trigger as the Neo4j reclassification. No timing race because PostgreSQL UPDATE is atomic (no CREATE + DELETE window).

**Step 6 — Remove Neo4j from docker-compose.yml:**

```yaml
# Remove this service block from CasaOS docker-compose.yml:
# neo4j:
#   image: neo4j:5
#   ...
```

Archive the Neo4j data volume for 30 days before deletion.

**Estimated migration time**: 2-4 hours of implementation, < 1 second of data migration for 800 edges.

## Ideas Extracted

### Idea: postgres-co-retrievals-neo4j-replacement
- **Novelty**: 0.75
- **Evidence**: 0.72
- **Feasibility**: 0.90
- **Description**: Replace Neo4j with a `co_retrievals` table in the existing PostgreSQL instance. Use a partial index on `relationship_type = 'FRESH'` as the structural analog to Neo4j's per-type relationship chain — the index physically excludes STALE rows so the index scan never reads them (true pre-filter, not a predicate evaluated post-read). Use a depth-2 CTE self-join for BFS. Replace the Neo4j CREATE+DELETE type migration (Q096) with an atomic SQL UPDATE. Replace the READ_COMMITTED race fix (Q099) with a simple UPSERT. Replace the circuit breaker K logic (Q094) with a per-CTE-level LIMIT clause. Removes Neo4j from the stack entirely, reducing from 5 stores to 4.

### Idea: pg-partial-index-as-structural-prefilter
- **Novelty**: 0.80
- **Evidence**: 0.70
- **Feasibility**: 0.92
- **Description**: A PostgreSQL partial index WHERE relationship_type = 'FRESH' is architecturally equivalent to Neo4j's per-type relationship chain for the Recall BFS use case. Both provide structural pre-filtering: Neo4j's chain contains only FRESH-type relationships, so traversal never reads STALE records; the partial index contains only FRESH rows, so the index scan never reads STALE rows. The key distinction from a full index with a predicate filter: a partial index does not store STALE rows at all, so there are no STALE rows to evaluate. This is the same O(F_fresh) characteristic that Q091 identified as Neo4j's core architectural advantage. Applies the same principle to any relational table where a type/status column partitions rows into hot and cold sets.

### Idea: upsert-replaces-create-delete-type-migration
- **Novelty**: 0.72
- **Evidence**: 0.85
- **Feasibility**: 0.95
- **Description**: The entire Q096 two-phase migration (CREATE new type + DELETE old type) and Q099 race condition fix (compare-and-swap WHERE clause) are artifacts of Neo4j Community Edition's immutable relationship type constraint. PostgreSQL's ON CONFLICT DO UPDATE (upsert) atomically handles both new edge creation and existing edge reinforcement in a single statement — including type promotion from STALE to FRESH on reinforcement. The background reclassification (STALE promotion) becomes a single SQL UPDATE with no two-step pattern. This eliminates approximately 80 lines of Cypher and Python migration logic accumulated across Q096 and Q099.

### Idea: pg-hub-limit-replaces-circuit-breaker
- **Novelty**: 0.68
- **Evidence**: 0.75
- **Feasibility**: 0.93
- **Description**: The hub node circuit breaker (Q094, K=50 warmup / K=20 fallback, asyncio.wait_for(8ms), Markov chain state machine) exists because Neo4j BFS has no built-in row cap — it traverses all F_fresh edges and returns. A PostgreSQL CTE with `LIMIT 50` inside the depth-1 CTE block provides deterministic hub protection: the query planner applies the limit during the index scan, never materializing more than 50 rows per seed. This is not a timeout-based circuit breaker but a structural cardinality cap. The Markov chain complexity (3-consecutive-timeout state machine, P(CB_open) = 1.9% at Month 12) becomes unnecessary. Hub protection degrades gracefully: at F_fresh=50+, the LIMIT 50 activates and returns the 50 highest-weight fresh neighbors ordered by weight DESC.

### Idea: cte-depth-limit-hard-boundary
- **Novelty**: 0.65
- **Evidence**: 0.82
- **Feasibility**: 0.88
- **Description**: PostgreSQL CTEs materialize each level fully before the next join. At depth-3 with F_fresh=10, this touches K*F^3 = 3,000 rows — manageable. At depth-3 with F_fresh=50, it touches 375,000 rows — 3-4 seconds at heap read cost. The correct architecture is: cap BFS at depth-2 for production queries (consistent with Q094 design), reserve depth-3 only for offline analysis with explicit LIMIT at each CTE level. PostgreSQL 14+ supports LIMIT inside recursive CTEs, but standard non-recursive CTEs (the depth-2 self-join pattern) require explicit LIMIT inside each CTE block to avoid fan-out explosion. Document depth-2 as the hard production boundary, depth-3 as offline-only.
