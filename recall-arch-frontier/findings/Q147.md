## Measurement

### content_hash derivation — `src/core/embeddings.py` lines 264–267

```python
def content_hash(text: str) -> str:
    """Generate a hash for deduplication."""
    normalized = " ".join(text.lower().split())
    return hashlib.sha256(normalized.encode()).hexdigest()[:16]
```

The hash is derived **exclusively from memory text content**: lowercased, whitespace-normalized, then SHA-256. No timestamp, no source machine, no session ID. The function is called as `content_hash(request.content)` and `content_hash(item.content)` in the write path — the hash is content-only.

### Qdrant lookup — `src/storage/qdrant.py` lines 462–480

```python
async def find_by_content_hash(self, hash_value: str, user_id: int | None = None) -> str | None:
    """Find a memory by content_hash. Returns memory_id or None."""
    conditions = [
        FieldCondition(
            key="content_hash",
            match=MatchValue(value=hash_value),
        ),
    ] + self._user_conditions(user_id)
    results, _ = await self.client.scroll(
        collection_name=self.collection,
        scroll_filter=Filter(must=conditions),
        ...
    )
```

The lookup queries Qdrant's payload index on `content_hash`. No HTTP header or PostgreSQL idempotency_keys table exists — the mechanism lives entirely inside the write handler via Qdrant payload scroll.

## Finding

[OBJECTIVE] Q036's finding was framed around streaming database theory (Kafka KIP-98, Stripe idempotency keys) and concluded those mechanisms did not exist in Recall, describing the content-derived idempotency key as a future implementation gap. This calibration directly tests that claim by reading the actual source.

[DATA] The Recall write path at `POST /memory/store` computes `SHA-256(lowercase(whitespace_normalized(content)))[:16]` from the memory text before calling Ollama. It queries Qdrant's `content_hash` payload field; if an identical hash is found, the write returns early with `created=False` and the message `"Duplicate -- identical content already stored"`. The Ollama embedding call is bypassed entirely in this code path (the early return at line 179–191 precedes `embedding_service.embed()` at line ~200). The same dedup check is present in the batch store path. There is no `idempotency_keys` PostgreSQL table, no `X-Idempotency-Key` HTTP header handling, and no epoch fencing — the Q036 proposals remain unimplemented as HTTP-layer mechanisms.

[FINDING] The Q141 SHA-256 Layer 0 write-time exact deduplication is **already implemented** in the current Recall codebase. The content hash is: (1) content-derived only — not metadata-based; (2) computed before Ollama embedding is called; (3) checked against persistent storage (Qdrant payload index) so it works across sessions, machines, and time. This closes the exact-dedup gap that Q141 identified as a 100–300ms savings opportunity. Q036's analysis correctly identified the need for content-derived deduplication, but it described the mechanism as absent when it was already present under a different architectural shape (Qdrant payload index rather than PostgreSQL idempotency_keys table).

[STAT:1] content_hash = SHA-256(normalized_text)[:16] — 16 hex characters = 64-bit collision space. For a store with 10,000 memories, the birthday collision probability is (10,000²) / (2 × 2^64) ≈ 2.7 × 10⁻⁹ — negligible. The truncation to 16 hex chars introduces no practical collision risk at Recall's current scale.

[STAT:2] The Qdrant `scroll` query used for `find_by_content_hash` is a payload filter scan, not a vector ANN search. Performance depends on Qdrant's payload index efficiency. If the `content_hash` field has a payload index configured, this is O(log n) and fast (< 5ms). If not indexed, it degrades to O(n) full scan — potentially slower than the 100–300ms Ollama embedding call for very large collections.

## Qualifying Ideas

- **verify-content-hash-qdrant-payload-index**: (N=0.70, E=0.85, F=0.90) — The dedup check uses `qdrant.client.scroll()` with a payload filter on `content_hash`. If Qdrant's `content_hash` payload field is not indexed, this is a full collection scan on every write — O(n) cost that grows with collection size. At 10,000+ memories, an unindexed scan could cost 50–200ms, eliminating the embedding-skip savings. Verifying that a Qdrant payload index exists on `content_hash` (check `src/storage/qdrant.py` collection creation code or the Qdrant collection config at 192.168.50.19) is a low-effort, high-value validation step before claiming the Layer 0 dedup delivers its promised latency savings.

- **content-hash-cross-machine-coverage-confirmed**: (N=0.65, E=0.90, F=0.95) — The hash is derived purely from content (not session ID, source machine, or timestamp), which means casaclaude and proxyclaude will generate the same hash for the same memory text and the second write will be rejected. This is the cross-machine deduplication Q036 described as absent. The mechanism is already present. The open question is whether the Qdrant `scroll` query's latency is low enough that the early-return path is genuinely faster than the full write path including Ollama — worth a direct timing measurement on the live system to confirm the 100–300ms claim from Q141.
