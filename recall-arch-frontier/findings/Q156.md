    from_memory_id      UUID        NOT NULL,

## Notes on the Q115 DDL vs. Q136 DDL

Q115 specified `REFERENCES memories(id) ON DELETE CASCADE` foreign keys on `from_memory_id` and `to_memory_id`. Q136 omitted these FK constraints from `_MIGRATE_CO_RETRIEVALS_OUTBOX`. The migration constant above follows Q136's pattern (no FK constraints) for three reasons:

1. The `memories` table is in Qdrant, not PostgreSQL — there is no `memories` table in `postgres_store.py`'s `_CREATE_TABLES` block to reference.
2. FK constraints would require the UUIDs to exist in a PG `memories` table first, which they do not (Qdrant is the authoritative store).
3. Q130's UPSERT pattern stores UUIDs directly as `UUID NOT NULL` without FK validation, consistent with the fire-and-forget philosophy.

The `PRIMARY KEY (tenant_id, from_memory_id, to_memory_id)` provides the ON CONFLICT target for the UPSERT path.

- **fillfactor-70-is-step-4-not-step-1**: (N=0.68, E=0.92, F=0.98) — The fillfactor=70 setting cannot be embedded in `CREATE TABLE IF NOT EXISTS` DDL portably for idempotent migrations (`WITH (fillfactor=70)` can be added to CREATE TABLE but `IF NOT EXISTS` ignores storage parameters if the table already exists). The correct deployment sequence is: (1) create table via migration, (2) apply `ALTER TABLE co_retrievals SET (fillfactor = 70)` as a separate one-time DDL after creation. Q137 confirmed this is load-bearing (not cosmetic): without fillfactor=70, every UPSERT ON CONFLICT generates a dead tuple because the page is 100% full and HOT chains cannot form. The separation of table creation from storage parameter tuning is intentional and correct.

- **co-retrievals-outbox-tables-needed-before-flush-worker**: (N=0.72, E=0.94, F=0.97) — The `_co_retrievals_flush_worker` from Q130/Q136 polls `co_retrievals_outbox WHERE processed_at IS NULL`. If the outbox table does not exist at startup, the flush worker's first query will raise `asyncpg.exceptions.UndefinedTableError`, which the worker's outer `except Exception` will catch and log, then continue looping every 5 seconds producing 8,640 errors/day. The migration must be applied before the flush worker is wired into `main.py`. The correct deployment order: (1) apply migration (Step 1–2), (2) restart service (Step 3), (3) apply fillfactor (Step 4), (4) add flush worker to main.py in a subsequent deploy.
