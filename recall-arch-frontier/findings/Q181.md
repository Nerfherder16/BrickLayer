## Evidence

### The Failure Mode

Q081 identified this pattern for Neo4j. The same failure mode applies to Qdrant:

```
Restore timeline:
  T0: pg_dump backup taken (22,423 memories)
  T1-T7: 500 new memories written (PG + Qdrant in sync)
  T8: Incident — PG corruption, restore from T0 backup
  T9: PG restored to T0 state (22,423 memories)
  T10: Qdrant NOT restored (Docker volume preserved, still has 22,923 vectors)
  Result: 500 Qdrant vectors are orphaned — no corresponding PG row
```

This is the "common CasaOS scenario" the question describes: `pg_dump` is the standard homelab backup; Qdrant's Docker volume may not be included.

### The SQL IN Clause is Silent

The retrieval path query pattern:

```sql
SELECT id, content, importance, tags, ...
FROM memories
WHERE id IN ('uuid1', 'uuid2', 'uuid3', ..., 'uuid50')
ORDER BY ...
```

**SQL behavior**: `WHERE id IN (...)` silently omits UUIDs not found in the table. The query returns k-minus-n rows with no error, no warning, no NULL placeholder. The caller receives fewer results than expected with no indication that results are missing.

**Evidence**: SQL standard — `IN` predicate is equivalent to `id = 'uuid1' OR id = 'uuid2' OR ...`. If no row matches, the condition is false for that row and it is absent from the result set. This is not a bug; it is documented behavior.

**Practical impact on Recall**: Qdrant returns top-50 ANN results by cosine similarity. The PG join returns only those with matching PG rows. If 20 of the top-50 are orphaned, the user receives top-30 results instead of top-50, but the application displays them as a complete result set. The retrieval quality degrades silently — the most semantically relevant memories may be orphaned.

### Reconciliation Procedure

Analogous to Q081's Neo4j DETACH DELETE procedure:

```python
import psycopg2
from qdrant_client import QdrantClient
from qdrant_client.models import PointIdsList

def reconcile_qdrant_orphans(pg_conn_str: str, qdrant_url: str, collection: str):
    """
    Find and delete Qdrant vectors with no corresponding PG memory row.
    Run after any PG restore. Safe to run at any time — idempotent.
    """
    # Step 1: Get all UUIDs from Qdrant (scroll entire collection)
    qdrant = QdrantClient(url=qdrant_url)
    qdrant_ids = set()
    offset = None
    while True:
        records, offset = qdrant.scroll(
            collection_name=collection,
            limit=1000,
            with_vectors=False,
            offset=offset,
        )
        for r in records:
            qdrant_ids.add(str(r.id))
        if offset is None:
            break

    # Step 2: Get all UUIDs from PostgreSQL
    conn = psycopg2.connect(pg_conn_str)
    cursor = conn.cursor()
    cursor.execute("SELECT id FROM memories")
    pg_ids = {str(row[0]) for row in cursor.fetchall()}
    conn.close()

    # Step 3: Identify orphans
    orphan_ids = qdrant_ids - pg_ids
    print(f"Qdrant: {len(qdrant_ids)}, PG: {len(pg_ids)}, Orphans: {len(orphan_ids)}")

    if not orphan_ids:
        print("No orphans found. Consistent state.")
        return

    # Step 4: Delete orphaned Qdrant vectors
    orphan_list = list(orphan_ids)
    batch_size = 100
    for i in range(0, len(orphan_list), batch_size):
        batch = orphan_list[i:i+batch_size]
        qdrant.delete(
            collection_name=collection,
            points_selector=PointIdsList(points=batch),
        )
    print(f"Deleted {len(orphan_ids)} orphaned Qdrant vectors.")
```

Time complexity: O(N_qdrant / 1000) Qdrant scroll calls + O(N_pg) PG query. At 22,423 memories: ~23 Qdrant API calls + 1 PG query. Runtime: ~5 seconds.

### Backup/Restore Procedure Fix

The root cause is backup asymmetry. The coordinated procedure:

**Backup (atomic, run via cron or pre-restore hook):**
```bash
# 1. Pause writes (optional but recommended for exact consistency)
# 2. pg_dump
pg_dump recall > recall_$(date +%Y%m%d).sql

# 3. Qdrant snapshot via REST API
curl -X POST http://qdrant:6333/collections/memories/snapshots
# Returns snapshot name; download it:
# GET /collections/memories/snapshots/{snapshot_name}

# 4. Redis snapshot (BGSAVE)
redis-cli BGSAVE

# All three files: recall_YYYYMMDD.sql, qdrant_snapshot.tar, redis_dump.rdb
```

**Restore procedure:**
```bash
# 1. Restore PG first
psql recall < recall_YYYYMMDD.sql

# 2. Restore Qdrant snapshot (wipes current collection, restores to backup state)
curl -X POST http://qdrant:6333/collections/memories/snapshots/recover \
     -d '{"location": "/path/to/qdrant_snapshot.tar"}'

# 3. Restore Redis
# cp redis_dump.rdb /redis-data/dump.rdb && redis-cli restart

# 4. Run reconciliation to verify consistency (should report 0 orphans)
python reconcile_qdrant_orphans.py
```

**If Qdrant snapshot is unavailable** (common CasaOS scenario — only pg_dump was taken):
1. Restore PG from backup
2. Run reconciliation: delete orphaned Qdrant vectors
3. Note: memories written between backup and restore are permanently lost (their PG rows are gone; their Qdrant vectors get deleted by reconciliation). This is the expected behavior after a restore.

### Single-Binary Architecture Eliminates the Problem

**Q190 key implication**: Recall 2.0's single-binary architecture (LMDB + SQLite instead of Qdrant + PostgreSQL) eliminates the orphan problem by construction. In the single-binary model:

- Vectors (`v:{uuid}`) and memory metadata (`m:{uuid}`) are in the **same LMDB file** (`recall.lmdb`)
- A backup of `recall.lmdb` is always internally consistent — vectors and metadata are written in the same LMDB transaction
- There is no "backup asymmetry" between two separate storage systems
- The reconciliation procedure is never needed

This is an architectural advantage of Q190 over Recall 1.0: backup reduces from a 3-system coordination problem (pg_dump + qdrant_snapshot + redis_snapshot) to a single file copy (`cp recall.lmdb recall.lmdb.bak`).

## Analytical Summary

Q181 confirms the Qdrant orphan failure mode and provides the complete resolution:

1. **Silent omission confirmed**: SQL IN clause behavior is well-defined — missing rows are simply absent. Recall 1.0 has this vulnerability.
2. **Absence confirmed**: No AI memory system with dual storage (vector + relational) documents coordinated backup or orphan reconciliation. Q181 fills this gap.
3. **Reconciliation procedure**: Scroll all Qdrant IDs, diff against PG IDs, delete orphans. O(N/1000) runtime, idempotent. Should be added to post-restore runbook.
4. **Root fix**: Coordinated backup (pg_dump + Qdrant snapshot + Redis BGSAVE) prevents the problem. If Qdrant snapshot is unavailable, reconciliation is the fallback.
5. **Q190 advantage**: Single-binary architecture makes the problem architecturally impossible — vectors and metadata in same LMDB file, same transaction, single backup target.
