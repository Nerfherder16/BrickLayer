# Finding: Q057 — Graph-Distance Level 1.5: Causal Edge Proximity as MBC Routing Signal

**Question**: Can the compiler scope-chain MBC routing rule (Q045) be extended with Neo4j graph-distance signals — specifically, using causal edge proximity between memories as an additional routing priority level between Level 1 (tag intersection) and Level 2 (domain zone match) — without requiring embeddings?
**Question Type**: ADJACENT
**Verdict**: PROMISING
**Severity**: Medium
**Source field**: Graph database retrieval literature (Neo4j GraphRAG, Graphiti, CausalRAG); graph traversal performance benchmarking

---

## Mechanism Under Investigation

Q045 defined a three-level SQL-only MBC routing rule:
- Level 1: tag intersection with context_tags (GIN-indexed, O(log N + K))
- Level 2: domain zone match (B-tree indexed, fires when Level 1 < K_mbc)
- Level 3: frequency floor fallback

The proposed Level 1.5 extension: if a memory has a causal edge distance of exactly 1 hop from any LLPC-injected memory in the current session, elevate it as a Level 1.5 candidate — above domain zone match but below tag intersection. The structural analog is the compiler's parent scope: a parent-scope symbol is one scope level removed from the local scope and should be tried before the module scope (Level 2 in Q045's mapping).

The causal edges already exist in Neo4j. They encode supersession relationships (memory B invalidated memory A via a `set!` operation, per Q026's scope-qualified identity registry). The question is whether 1-hop traversal from K_llpc=2 seed nodes is fast enough and distinct enough from Level 1 tag intersection to be worth inserting as a routing tier.

## Evidence from Adjacent Fields

### Mechanism 1: Graphiti Node Distance Reranking

Graphiti (Zep, getzep/graphiti — 23,786 GitHub stars as of March 2026, production deployment at Zep.ai) implements exactly the graph-distance-as-routing-signal pattern in production. Its retrieval API accepts a `focal_node_uuid` parameter that activates "Node Distance Reranking": results from a hybrid semantic+BM25 search are re-weighted by their graph proximity to the focal node.

From Zep's official documentation (help.getzep.com/graphiti/working-with-data/searching):

> "Node Distance Reranking is particularly useful for entity-specific queries, providing more contextually relevant results. It weights facts by their closeness to the focal node, emphasizing information directly related to the entity of interest."

The concrete example in Graphiti's docs demonstrates the discriminating power: a hybrid search for "Can Jane wear Allbirds Wool Runners?" returns three generic results about the product. The same query with `focal_node_uuid=jane_node_uuid` returns Jane-specific results including her wool allergy — a causally linked fact that tag/keyword matching alone fails to surface because the allergy fact shares few keywords with the query.

Graphiti uses semantic embeddings for its base retrieval; the graph distance reranking is a second signal layered on top. The proposal for Recall's Level 1.5 differs in that it applies graph distance as a routing tier without embeddings — using it as a structural inclusion criterion rather than a reranking weight.

### Mechanism 2: CausalRAG (ACL 2025)

Wang et al., "CausalRAG: Integrating Causal Graphs into Retrieval-Augmented Generation," ACL 2025 (Case Western Reserve University) demonstrates that tracing causal relationships during retrieval preserves contextual continuity and improves precision over both standard RAG and graph-based RAG approaches on knowledge-intensive tasks. CausalRAG constructs causal graphs from the corpus and uses causal path tracing to identify retrieval candidates.

The mechanism is closer to what Q057 proposes than Graphiti is: CausalRAG uses the causal graph structure as a primary retrieval selection criterion, not merely a reranking weight. Their evaluation shows "superiority across multiple metrics" over vector-only and graph-based RAG approaches. Specific numbers are not extractable from the abstract, but the ACL 2025 acceptance indicates peer-reviewed validation of the core claim.

### Mechanism 3: Hybrid Retrieval — Vectors + Graphs Production Pattern

The Cognilium AI hybrid search benchmark (March 2025, publicly documented) reports that combining vector search, keyword search, and graph-based retrieval with Reciprocal Rank Fusion outperforms any single method by 23% on their evaluation set. The graph component specifically captures "dependencies, hierarchies, and causal chains" that vector and keyword search miss.

AWS/Lettria's GraphRAG deployment (documented December 2024, AWS Machine Learning Blog) reports up to 35% improvement in answer precision versus vector-only retrieval when graph structure is incorporated. The mechanism: "the graph's ability to model complex relationships and dependencies between data points, providing a more nuanced and contextually accurate foundation."

The consistent pattern across all three evidence sources: graph proximity signals add a distinct retrieval dimension that tag/keyword signals do not fully cover, because two memories can share zero tags while being causally connected — and causal connection is a stronger relevance signal than domain co-occurrence for the class of "superseded fact" queries.

## Cost Analysis

### Cypher Query for Level 1.5

The proposed Level 1.5 query is:

```cypher
MATCH (seed:Memory)-[:SUPERSEDES|:CAUSES|:UPDATES]->(neighbor:Memory)
WHERE seed.memory_id IN $llpc_seed_ids
  AND NOT (neighbor.access_count >= 3 AND neighbor.sessions_since_last_access <= 10)
  AND NOT neighbor.memory_id IN $level1_ids
RETURN neighbor.memory_id
LIMIT 10
```

With K_llpc=2 seed nodes and average fanout=3, this query returns at most 6 candidates (2 seeds × 3 neighbors) before LIMIT filtering.

### Neo4j's Index-Free Adjacency Model

Neo4j's storage model stores adjacency lists directly on each node as part of the node record — relationship traversal does not require a global index lookup. From Neo4j's own documentation and the "This Is How Neo4j Actually Traverses Graphs" technical explainer (Medium, June 2025): "each node stores pointers to its relationship records. To find neighbors, Neo4j just follows pointers — O(degree) at most." This is what Neo4j calls "index-free adjacency."

For a 1-hop traversal from 2 seed nodes with average fanout=3:
- Node lookup by memory_id: O(1) with a property index on memory_id (standard best practice)
- Relationship traversal from each seed: O(degree) = O(3) — pointer following, not a scan
- Total nodes touched: 2 seeds + 6 neighbors = 8 nodes
- Graph layer: this does not touch N=50,000 nodes; it touches exactly degree(seed_1) + degree(seed_2) nodes

The FalkorDB vs. Neo4j benchmark (December 2024) shows Neo4j multi-hop aggregate queries at aggregate workload p99 approaching multi-second latency under high concurrency load — but this is for aggregation-heavy graph analytics workloads, not simple 1-hop neighbor lookups. The Samyama benchmark (March 2026) demonstrates 115K Cypher queries/sec at 1M nodes for read queries on commodity hardware, confirming that simple traversals are sub-millisecond at rest.

The Graphiti Reddit thread (February 2026) reports a user experiencing 50-second retrieval on a local Neo4j instance — but the root cause diagnosis in responses points to missing vector indexes causing full scans, not traversal cost. With proper memory_id indexing, the 1-hop lookup for 2 seeds with fanout=3 should complete in 1-3ms on a local instance at N=50,000.

**Estimated latency budget for Level 1.5 at Recall's scale:**
- Neo4j query: 1-3ms (single hop, 2 seeds, indexed by memory_id)
- Network overhead (localhost): negligible (Neo4j bolt protocol, same host)
- Result hydration (6 memory_ids): < 1ms
- **Total Level 1.5 overhead: 2-5ms**

The MBC budget is 200ms p95 (from Q045). Level 1 SQL runs in roughly 5-20ms on a GIN-indexed corpus of 50,000. Level 1.5 at 2-5ms fits within budget.

However, this estimate carries a caveat: the Graphiti user report (r/Rag, February 2026) indicates that local Neo4j without proper indexing configuration can degrade catastrophically. The 2-5ms estimate assumes: (a) a B-tree or range index on `memory_id`, (b) the relationship type `SUPERSEDES` is the only relationship type traversed (no wildcard relationship patterns that force full relationship scans), and (c) the Neo4j instance is warmed (page cache populated).

## Ideas Extracted

### Idea: graph-distance-level-1.5

- **Novelty**: 0.75 — Graphiti (production) uses graph proximity as a reranking weight applied after semantic retrieval. CausalRAG (ACL 2025) uses causal edges as retrieval candidates. Neither applies graph proximity as a discrete routing tier in a purely structural (no-embedding) pipeline of the type Q045 designed. The specific insertion of a graph-distance tier between tag intersection (L1) and domain zone match (L2), as a discrete priority level in a SQL+Cypher hybrid routing pipeline with no embedding involvement, is not described in any known system.
- **Evidence**: 0.65 — The Graphiti production evidence confirms that graph proximity adds a retrieval dimension orthogonal to keyword/tag matching (the Jane/wool allergy example is a clean demonstration). CausalRAG's ACL 2025 acceptance validates the causal edge retrieval concept. The 23% and 35% improvement figures (Cognilium, AWS/Lettria) are for hybrid pipelines that include graph structure alongside semantic search — the graph-only signal improvement is not isolated. No paper directly benchmarks "1-hop graph proximity from seed memories" as a standalone routing signal against "tag intersection only" in an AI memory context. Evidence is indirect.
- **Feasibility**: 0.70 — Requires: (a) a Cypher query from 2 seed node IDs with 1-hop traversal (10-15 lines, straightforward); (b) an index on `memory_id` in Neo4j (already recommended practice); (c) session state to carry the 2 LLPC memory IDs as seed parameters; (d) orchestration logic to merge Level 1 exclusions into the Level 1.5 query to prevent overlap. The main risk is Neo4j cold-start latency on a self-hosted single-machine instance (Recall's deployment). Level 1.5 adds a cross-service call (PostgreSQL → Neo4j) that Level 1 and Level 2 avoid. If the MBC routing budget is already tight, this cross-service call may not be worth it.
- **Description**: After Level 1 (tag intersection) fills its K_mbc slots, before invoking Level 2 (domain zone match), execute a 1-hop Cypher traversal from the session's K_llpc=2 LLPC-injected memories to retrieve their direct causal neighbors in Neo4j. A causal neighbor is a memory connected by a SUPERSEDES, CAUSES, or UPDATES edge — the relationships already stored by Recall's Q026 scope-qualified identity registry. Any causal neighbor not already selected by Level 1 and not in the LLPC tier is elevated to Level 1.5. This surfaces memories that are causally linked to the session's most-proven memories but happen to share no tags with the current context_tags set. The Jane/wool-allergy pattern: LLPC has "Jane bought Allbirds Wool Runners" injected; Level 1.5 surfaces "Jane has a wool allergy" because it is a 1-hop causal neighbor even though "wool allergy" shares no tags with the current context_tags of `["allbirds", "purchase", "runners"]`.

### Idea: causal-neighbor-as-tag-audit

- **Novelty**: 0.72 — Using the discrepancy between Level 1 results and Level 1.5 results as an audit signal — specifically, when Level 1.5 consistently surfaces memories that Level 1 would have missed, it indicates that the tag vocabulary is failing to encode causal relationships (related memories lack shared tags) — is a diagnostic mechanism not described anywhere. It turns the Level 1.5 tier from a pure retrieval mechanism into a feedback loop for tag quality.
- **Evidence**: 0.55 — The priority_level observability field from Q045 already provides the template for this kind of diagnostic. The specific cross-tier comparison (Level 1 vs. Level 1.5 overlap rate as a tag-vocabulary health metric) is a straightforward engineering extension of Q045's observability design, not a validated mechanism from the source field.
- **Feasibility**: 0.80 — Implementation is a logging extension to the existing priority_level audit trail. When Level 1.5 selects a memory, log: `{session_id, memory_id, level: "1.5", seed_memory_id, edge_type}`. Aggregate: for each memory selected at Level 1.5, check whether it would have been selected by Level 1 if the tag `{edge_type}` had been included in context_tags. If Level 1.5 selections have tags that cluster around values not in the current tag vocabulary, that cluster is a candidate for tag expansion. Implementation: zero new infrastructure, just a log query aggregated weekly.
- **Description**: Track Level 1.5 selections separately in the audit log. Periodically aggregate: "which tags appear frequently on Level 1.5 memories that were NOT in the context_tags set that session?" These tags are candidates for addition to the static file-path-to-tag inference table (Q045's `file-tag-map.json`) or the domain vocabulary for prompt-domain extraction. If Level 1.5 repeatedly selects memories tagged `["credential", "ssh"]` that Level 1 misses, it suggests the session's file-path inference table lacks patterns for SSH config files. This converts graph-proximity signal into vocabulary improvement signal, making the structural routing pipeline self-improving over time without any LLM calls.

## Key Finding

**The Level 1.5 extension is structurally sound and latency-feasible, but its incremental value over tag intersection is conditional on tag vocabulary coverage.**

The core question from the research brief — "is Level 1.5 worthwhile given that tag intersection already captures many of the same causal relationships?" — has a qualified answer:

**When tag vocabulary is dense and consistent:** Level 1.5 provides minimal incremental value. If memory B supersedes memory A and both were stored with consistent tags reflecting the same domain (e.g., both tagged `["redis", "casaclaude", "host"]`), then Level 1's tag intersection already surfaces B when A is in the LLPC set (they share all tags). Level 1.5 is redundant.

**When tag vocabulary is sparse or inconsistent:** Level 1.5 provides real value. The Jane/wool-allergy pattern above is a clean example: the causal relationship is semantically strong but tag-disjoint. In Recall's actual use case, this would manifest as: LLPC injects memory "Redis on casaclaude is healthy" (tags: `["redis", "casaclaude"]`); Level 1.5 would surface "Redis port 6379 is bound to loopback only after casaos update" (tags: `["casaos", "networking", "port"]`) — a causally linked fact with zero tag overlap.

**The tag vocabulary coverage determines the answer.** This is falsifiable: run the Level 1.5 query for K sessions and measure the fraction of Level 1.5 selections that would have been Level 1 selections if perfect tags were assigned. If > 70% of Level 1.5 selections are tag-disjoint from Level 1 context, Level 1.5 is capturing real signal. If < 30% are tag-disjoint, tag intersection is already covering the same ground and Level 1.5 is redundant.

**The cross-service call adds architectural complexity.** Levels 1, 2, and 3 in Q045's design are all PostgreSQL queries — single-service, single round-trip. Level 1.5 adds a Neo4j round-trip. On Recall's self-hosted stack (both PostgreSQL and Neo4j on CasaOS at 192.168.50.19), this is a localhost call, but it adds connection overhead and a second query planning cycle. The question is whether 2-5ms of added latency at Level 1.5 is worth the marginal retrieval improvement. Given that Level 1.5 only fires when Level 1 has fewer than K_mbc=3 results (the slot-filling model from Q045), the actual average latency addition is: fraction_of_sessions_with_L1_deficit × 3ms. If Level 1 fills all 3 slots in 80% of sessions (the expected case in well-tagged corpora), the average Level 1.5 overhead is 0.6ms — negligible.

**Verdict: PROMISING with one prerequisite.** The Level 1.5 extension is worth implementing after the tag vocabulary coverage measurement confirms tag-disjoint causal relationships exist at meaningful frequency (> 20% of causal neighbor pairs in the corpus). If the measurement shows tags already encode causal proximity adequately, Level 1.5 can be skipped. The measurement itself (a Cypher query that reports tag overlap fraction across all SUPERSEDES edges) is a one-time diagnostic that takes minutes to run and directly answers whether the extension is warranted.

## Suggested Follow-ups

- **[SIMULATION] Q058**: Measure tag overlap across all SUPERSEDES edges in Recall's current Neo4j graph. For each edge (A)-[:SUPERSEDES]->(B), compute `cardinality(A.tags & B.tags)`. Report the distribution: what fraction of causal pairs have zero tag overlap, 1-tag overlap, 2+ tag overlap? This is the prerequisite measurement that determines whether Level 1.5 captures non-redundant signal. If median overlap is 2+ tags, Level 1.5 is redundant with Level 1. If median overlap is 0-1 tags, Level 1.5 captures distinct signal and warrants implementation. [SIMULATION]

- **[PHYSICS] Q059**: What is the actual p95 latency of the Level 1.5 Cypher query on Recall's live Neo4j instance (localhost, 192.168.50.19)? Profile: `MATCH (seed:Memory)-[:SUPERSEDES]->(n:Memory) WHERE seed.memory_id IN $ids RETURN n.memory_id LIMIT 10` with K=2 seed IDs, warmed and cold cache. Compare against the 2-5ms estimate from the index-free adjacency model. If p95 > 15ms on the live instance, the cross-service overhead exceeds the latency budget for a tier that fires in the minority of sessions. [PHYSICS]

- **[ADJACENT] Q060**: Graphiti's Node Distance Reranking uses graph proximity as a continuous weight applied after semantic retrieval. Recall's Level 1.5 uses it as a discrete inclusion tier applied before any retrieval. What is the retrieval quality difference between these two architectures — continuous reranking vs. discrete tier gating — for the same graph structure? In the compiler analogy: does scope depth produce better results as a hard ordering rule (compiler semantics) or as a soft weight (fuzzy matching)? The information retrieval literature on hard vs. soft retrieval constraints may have evidence. [ADJACENT]

- **[ADVERSARIAL] Q061**: Challenge the causal edge quality assumption. Level 1.5's value depends on Neo4j's SUPERSEDES edges accurately encoding semantic causality. In practice, the edges are written by Q026's scope-qualified identity registry at store time — they reflect structural supersession (new binding for same topic key), not semantic causality. A memory about "Redis config change" and a memory about "Redis is now slower" may have a genuine causal relationship but no SUPERSEDES edge (they were stored in different topic keys). What fraction of semantically causal memory pairs in Recall's corpus actually have a SUPERSEDES edge? If edge coverage is below 40% of true causal relationships, Level 1.5 is providing noisy partial signal, not reliable causal proximity. [ADVERSARIAL]
