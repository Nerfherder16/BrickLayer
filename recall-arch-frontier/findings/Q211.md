## Physical Requirements Derivation

### Step 1: What Function Does Working Memory Serve?

An agent executing a task must hold observations, deductions, and partial results that are relevant to the current computation. These facts are:

1. **Temporally local**: All accesses occur within a bounded time horizon (minutes to hours)
2. **High write frequency**: The agent writes new observations continuously during execution (100-1000x per task vs. 1-10x for long-term)
3. **High read frequency**: Accessed multiple times before the task ends
4. **Bounded cardinality**: The task's working set is bounded by the agent's output scope (not all 22K long-term memories — only facts relevant to the current goal)
5. **Zero post-task value**: Once the task completes, most working-memory facts are either promoted (have long-term value) or discarded (task-specific and non-generalizable)

### Step 2: Physical Constraints from Requirements

**From requirement 1 (temporal locality)**:
The access pattern is temporally clustered — all accesses within a task window. This matches the physics of **RAM** (random access, low latency, not durable across power loss) better than **disk** (durable, sequential-optimized, higher per-operation latency).

**From requirement 2 (high write frequency)**:
Long-term write cost: 41ms (embedding) + 1ms (LMDB) = 42ms per write. At 100 writes per task, this is 4,200ms = 4.2 seconds blocked in embedding. This is physically incompatible with a responsive working memory. The write cost budget for working memory: 42ms / 100 = **0.42ms per write maximum**.

**Conclusion from write frequency**: Working memory writes **cannot use float32 embedding** (41ms). The write step must be ≤0.42ms. This means either: (a) sparse/binary encoding at <1ms, or (b) raw text keyword extraction at <0.1ms.

**From requirement 3 (high read frequency)**:
Reads must be fast enough to not block the agent's reasoning. Budget: <0.1ms per read (10× faster than HNSW's 0.215ms, justified by the small working set).

**From requirement 4 (bounded cardinality)**:
Estimated working set per task: 50-200 facts. At full float32 embedding: 200 × 4KB = 800KB — fits in L3 cache (6MB). At sparse binary: 200 × 128B = 25KB — fits in L2 cache (512KB). The bounded cardinality unlocks **linear scan** as a viable retrieval strategy: 200 entries × (scan cost per entry) < 0.1ms.

**From requirement 5 (zero post-task value)**:
No durability requirement. No fsync. No write-ahead log. Pure RAM writes. Data loss on process restart is acceptable — the task can be restarted.

### Step 3: Physical Data Structure Derivation

The physical requirements constrain the design to:

**A bounded ordered activation sequence in RAM:**
- Ordered by: (activation_score × recency_weight) — most relevant and most recent facts at the front
- Bounded by: N_max entries (200 is adequate for task scope)
- Storage: pure RAM (no disk persistence)
- Write: keyword extraction from raw text (O(1) tokenization, ~0.1ms) + scored insertion (O(log N) into sorted list, negligible at N≤200)
- Retrieval: linear scan over N≤200 entries (800KB L3-resident at float32; 25KB L2-resident at binary)

**Retrieval latency at N=200**:
From Q205's linear scan formula: 200 × 4KB / 38.4 GB/s = 0.021ms for float32. For sparse binary at 200 × 128B: 0.0007ms. Both are 10-300× faster than HNSW long-term retrieval.

**Write path at N=200**:
Keyword extraction: ~0.1ms. Sorted insertion: O(log 200) ≈ 8 comparisons ≈ negligible. Total: ~0.1ms per write. **413× faster than long-term write path (42ms).** Satisfies the 0.42ms budget.

### Step 4: Physical Contrast with Long-Term Memory

| Property | Working Memory | Long-Term Memory |
|---|---|---|
| Cardinality | N ≤ 200 | N = 22,423+ |
| Storage medium | RAM (L2-L3 resident) | Disk (LMDB, DRAM paged) |
| Write cost | ~0.1ms (keyword extraction) | ~42ms (float32 embedding) |
| Retrieval cost | ~0.021ms (linear scan) | ~0.215ms (HNSW) |
| Retrieval semantics | Exact keyword match OR linear scan | Approximate semantic cosine |
| Durability | None (RAM only) | Full (LMDB fsync on commit) |
| Post-task retention | Selective promotion (~10% entries) | Permanent (decay only) |
| Access pattern | Write-heavy during task, zero after | Read-heavy across sessions |

**The key physical incompatibility**: Long-term memory's 41ms write cost (embedding) is incompatible with working memory's 100× higher write frequency. A single store cannot serve both workloads — write latency for working memory requires bypassing the embedding step, while long-term memory's retrieval quality requires float32 embeddings.

### Step 5: Promotion Mechanism (Working → Long-Term)

At task completion, entries in the working sequence are scored for promotion:
- High activation_score AND multiple reads during task → promote to long-term (run full embedding, write to LMDB+HNSW)
- Single-use entries (write once, never re-read) → discard
- Expected promotion rate: ~10% of working memory entries become long-term memories

**Physical reason for deferred embedding**: The embedding step (41ms) is a batch operation on the working memory, not a per-write operation. At 200 entries with 10% promotion, the batch embedding cost is: 20 entries × 41ms = 820ms at task end. This is acceptable as a background operation that doesn't block the agent's next task start.

## Ideas Extracted

### Idea: working-memory-tier-keyword-indexed-ram-resident-bounded-list
- **Novelty**: 0.70 — A physically-derived specification for a working memory tier (distinct from long-term memory) that bypasses embedding for writes and uses linear scan for retrieval is not implemented in any surveyed AI memory system. The physical derivation (write frequency → no embedding → keyword extraction; bounded cardinality → linear scan; no durability → RAM only) produces a clear architectural specification that differs from long-term memory at the physics level.
- **Evidence**: 0.80 — Write budget derivation: 100 writes/task × 42ms/write = 4.2s blocked in embedding (infeasible). At 0.1ms keyword extraction: 100 writes/task = 10ms total (feasible). Retrieval: 200 entries × 4KB = 800KB L3-resident = 0.021ms linear scan (10× faster than HNSW). Promotion cost: 20 entries × 41ms = 820ms batch at task end (acceptable async). All derivations from hardware specs validated in Q197/Q205.
- **Feasibility**: 0.75 — Implementation: an in-memory `Vec<(f32, u64, String, Vec<String>)>` sorted by score×recency (activation_score, timestamp, content, keywords). Write: extract keywords, score = 1.0, push to vec. Read: scan for keyword overlap, return top-K by score. Promote: at task end, sort by score, embed top-20%, write to LMDB. No external dependency — pure Rust in-process. Add to Recall 2.0 as an optional working memory endpoint (`POST /working-memory`, `GET /working-memory/search`). v2.1 scope.
- **Description**: Physical requirements analysis reveals working memory and long-term memory have incompatible write frequency requirements: long-term write (42ms) cannot support working memory's 100×/task write frequency without 4.2s of embedding overhead per task. Physical solution: working memory as a bounded (N≤200), RAM-resident, keyword-indexed ordered list. Write: keyword extraction (~0.1ms). Retrieval: linear scan of L2-resident corpus (~0.021ms, 10× faster than HNSW). No durability (RAM only). Promotion: batch-embed top 10% of entries at task completion → write to long-term store. The same LMDB+HNSW store cannot serve both tiers optimally — write-speed and retrieval-quality requirements are in direct physical tension.

### Idea: deferred-batch-embedding-for-working-memory-promotion
- **Novelty**: 0.65 — Deferring the expensive embedding step to task-completion time (batch-embedding only entries selected for long-term promotion) is not implemented in any surveyed system. All current systems embed at write time. Deferred embedding separates the "agent decides this is worth remembering" signal (fast, during task) from "embed for semantic retrieval" (slow, after task). The combined working memory → promotion gate → long-term memory write path reduces per-task embedding cost from 100×41ms to ~20×41ms (5× reduction) by embedding only the promoted fraction.
- **Evidence**: 0.75 — Derivation: 100 working memory writes at 0.1ms = 10ms total write cost during task. 200 entries at task end, 10% promotion = 20 embeddings × 41ms = 820ms background batch. Total embedding cost per task: 820ms (background) vs. 4,200ms (all-writes-embedded, blocking). Reduction: 5.1× lower embedding latency, 99% lower blocking latency during task execution. The 10% promotion rate is estimated from typical session analysis (most agent observations are task-specific, not cross-session relevant).
- **Feasibility**: 0.80 — Implementation: add a `working_memory_flush` hook that fires at session/task end. Hook calls the promotion filter (score threshold + minimum_read_count) on the working memory Vec, batches the selected entries to the fastembed-rs embedding pipeline, then writes to LMDB+HNSW. No change to the long-term write path. Prerequisite: working memory endpoint implementation (Idea 1). Phase 3 scope (CO_RETRIEVED sprint, same task as working memory write promotion).
- **Description**: Deferred batch embedding separates "decide to promote" (during task, fast) from "embed for retrieval" (end of task, slow batch). During a task, the agent writes 100 working memory entries at 0.1ms/write. At task end, the top-10% highest-score entries (20 entries) are batch-embedded at 41ms each = 820ms total. This runs in the background while the next task starts. Result: the agent's task execution has ZERO embedding-blocked write latency; the embedding pipeline runs asynchronously at task boundaries. 5× reduction in per-task embedding compute (only embed what matters, not everything observed).

## Analytical Summary

**Working memory and long-term memory have physically incompatible write frequency requirements.** Long-term write (42ms, embedding-dominated) cannot serve working memory's 100×/task write frequency. The same LMDB+HNSW store cannot optimally serve both tiers.

**The physical solution**: Working memory as bounded RAM-resident keyword-indexed list (write 0.1ms, retrieval 0.021ms linear scan, no durability). Long-term as LMDB+HNSW (write 42ms, retrieval 0.215ms HNSW, full durability). Promotion: batch-embed top-10% at task end.

**The absence hypothesis for Q212**: If no production agent framework implements this distinction, it is likely because: (1) LLM context window serves as implicit working memory (the agent "remembers" within-context without explicit working memory store); (2) the write frequency constraint only becomes critical at 50+ writes per task — below that, embedding latency is acceptable. The constraint becomes relevant when agents are running large-scope tasks with hundreds of intermediate observations.
