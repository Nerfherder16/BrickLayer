# Finding: Q051 — Session-End Hook Sequencing: Dependency Graph and asyncio Execution Plan

**Question**: Given that the session-end Stop hook now needs to execute six distinct operations (session-scope-window-drop, working-set counter update, cohesion cache refresh, scope registry cleanup, L0 dedup set flush, and session summary store), what is the correct sequencing and parallelization strategy, and which operations have ordering constraints that prevent parallelization?

**Question Type**: CONVERGENCE
**Verdict**: BREAKTHROUGH
**Severity**: Critical
**Source field**: Distributed systems protocol design — LSM compaction sequencing (Q035), OS working set update (Q042), vector database cohesion physics (Q043), idempotency key management (Q036, Q041)

## Operation Inventory

Before building the dependency graph, each operation must be stated in precise terms: what it reads, what it writes, and what it leaves as output for downstream operations.

**Op1 — session-scope-window-drop**
```
READ:  memories WHERE scope='session' AND importance < threshold AND sessions_since_creation > 0
WRITE: DELETE those rows from PostgreSQL; remove corresponding vectors from Qdrant
OUTPUT: set of deleted_memory_ids
```
Source: Q024 — the TTL-category filter-delete. One SQL DELETE with a filter condition.
Timing profile: O(N_deleted) deletes, but expressed as a single filter statement — fast.

**Op2 — working-set counter update**
```
READ:  accessed_this_session (set of memory IDs retrieved at least once during the session)
WRITE: UPDATE memories SET sessions_since_last_access = 0 WHERE id = ANY(accessed_ids)
       UPDATE memories SET sessions_since_last_access = sessions_since_last_access + 1
              WHERE id != ALL(accessed_ids)
OUTPUT: none (pure state mutation)
```
Source: Q042 — the Denning working-set epoch increment. Two SQL UPDATE statements.
The second UPDATE is O(N_total_memories) — it touches every row not in the accessed set.
At N=10,000 memories: 10K integer increments, a single sequential scan with no index benefit.

**Op3 — cohesion cache refresh**
```
READ:  newly archived memories (those promoted to Tier 3 in this session) + their Qdrant neighbors
WRITE: UPDATE memories SET cohesion_score = X for affected records in PostgreSQL
OUTPUT: none (updates cached scores for use by eviction policy)
```
Source: Q043 — the incremental-cohesion-cache-session-boundary idea. Requires knowing which memories were archived this session — specifically the output of the archival step (Q028 logic). At N_archive=50: ~800ms at p99, dominated by 500 Qdrant HNSW searches.

**Op4 — scope registry cleanup**
```
READ:  deleted_memory_ids (output of Op1)
WRITE: DELETE from Redis scope_registry any HSET entries whose value UUID is in deleted_memory_ids
OUTPUT: none
```
Hard dependency on Op1: cannot execute until Op1 produces deleted_memory_ids.
Timing: one Redis HSCAN + conditional HDELs. Very fast (sub-millisecond per deletion).

**Op5 — L0 dedup set flush**
```
READ:  Redis key recall:staging:{session_id} (session-scoped staging queue from Q035)
WRITE: promote entries to recall:staging:global (global L1 accumulation queue)
       DELETE recall:staging:{session_id} key
       (then optionally trigger L1 score check: LLEN recall:staging:global / target > 1.0)
OUTPUT: count of promoted entries; updated global queue length
```
Source: Q035 — session-scoped-staging-queue-partition. The L0 deduplication pass (batch HNSW over session memories, merge near-duplicates) runs as part of this step before promotion. Output feeds no downstream operation in this hook (the L1 score check can be deferred or run here).
Timing: batch HNSW query over N_session_memories + PostgreSQL merges. For N=50: ~250ms.

**Op6 — session summary store**
```
READ:  session content (summary text generated by the hook)
WRITE: POST /recall/store → triggers the three-gate store protocol (G1+G2+G3 pre-check, Ollama embed, Qdrant upsert, PostgreSQL insert)
OUTPUT: memory_uuid of stored summary
```
Source: Q041 — the unified three-gate store protocol. Cost dominated by Ollama embedding (~50ms). Independent of all other operations.

## Hard Ordering Constraints

### Constraint A: Op4 must run AFTER Op1

Op4 (scope registry cleanup) reads the set of deleted UUIDs produced by Op1. There is no way to determine which Redis scope registry entries are dangling until Op1 identifies which memories were deleted. This is a strict producer-consumer dependency.

**Sequencing enforced**: `Op1 → Op4`

### Constraint B: Op3 must run AFTER the archival step (which is outside these 6 ops)

Op3 (cohesion cache refresh) requires knowing which memories were archived to Tier 3 in the current session. This archival step corresponds to Q028's Tier 2→3 consolidation, which fires at session end. If the archival step is not included in these six operations but happens prior to the hook (or is step 0), then Op3 must be ordered after it.

Within the six operations listed, Op3 has no dependency on any of Ops 1, 2, 4, 5, or 6. It is independent of all of them except the archival step (pre-condition, not in the set of six).

**Sequencing enforced**: `[Tier2→3 archival] → Op3`
**Within the six**: Op3 is fully independent of all others.

### Constraint C: Op1 must run BEFORE Op2 (soft ordering — correctness reason)

Op2 updates `sessions_since_last_access` for ALL memories. If Op1 deletes memories first, Op2's bulk UPDATE will not waste cycles on rows that no longer exist. This is not a hard correctness constraint — Op2 would succeed either way — but running Op1 first reduces the set size Op2 must touch.

More importantly: the deleted memories from Op1 should not receive a counter increment in Op2, because they are being deleted. Running Op2 after Op1 means the deleted rows are gone from PostgreSQL and the UPDATE naturally skips them. Running Op2 before Op1 means Op2 increments counters on rows that are about to be deleted — wasted I/O.

**Sequencing recommended**: `Op1 → Op2` (not strictly required for correctness, but avoids wasted I/O)

### Constraint D: No constraint between Op5 and any other operation

Op5 (L0 dedup flush) operates entirely on the Redis session-staging key and the Qdrant/PostgreSQL records written during the session. It does not touch the scope registry (Op4's domain), the working-set counters (Op2's domain), the cohesion scores (Op3's domain), or the session summary write path (Op6's domain). The L0 batch HNSW query in Op5 reads session-scoped new memories from Qdrant — these are different from the session-scoped window-drop memories in Op1 (Op1 drops old low-importance session memories; Op5 processes newly written session memories for deduplication).

**Op5 is independent of all others.**

### Constraint E: No constraint between Op6 and any other operation

Op6 writes the session summary to Recall via the standard store endpoint. It does not read any output from Ops 1–5, and Ops 1–5 do not read Op6's output. It is fully independent.

**Op6 is independent of all others.**

## Dependency Graph

```
                   [Tier2→3 archival — precondition, runs before hook]
                          │
                          ▼
  ┌─────────────────────────────────────────────────────┐
  │  SESSION-END HOOK FIRES                             │
  └─────────────────────────────────────────────────────┘
           │               │          │          │
           ▼               ▼          ▼          ▼
          Op1             Op5        Op6        Op3
   (window-drop)    (L0 dedup     (summary   (cohesion
                       flush)      store)     refresh)
           │
           ▼
    ──── Op1 completes, yields deleted_memory_ids ────
           │               │
           ▼               ▼
          Op4             Op2
   (scope registry  (working-set
      cleanup)      counter update)
```

**Parallel groups:**
- Group A (fire immediately on hook entry): Op1, Op5, Op6, Op3
- Group B (fire after Op1 completes): Op4, Op2

Op4 and Op2 are also mutually independent and can run in parallel within Group B.

## Optimal asyncio Execution Plan

```python
import asyncio

async def session_end_hook(
    session_id: str,
    accessed_memory_ids: list[str],  # IDs retrieved during session
    newly_archived_ids: list[str],   # IDs promoted to Tier 3 (from archival precondition)
    session_summary_text: str,
) -> dict:
    """
    Session-end hook: six operations in two-phase asyncio execution.
    Phase 1: Four independent operations in parallel.
    Phase 2: Two operations dependent on Phase 1's Op1 result, run in parallel.
    """

    # ─── Phase 1: All independent operations in parallel ───────────────────────
    # Op1, Op3, Op5, Op6 share no data dependencies.
    # asyncio.gather() fires all four concurrently.
    # Op1 returns deleted_memory_ids needed by Phase 2.
    (
        deleted_ids,   # Op1 result
        _,             # Op3 result (void)
        _,             # Op5 result (void)
        _,             # Op6 result (summary UUID — optional use)
    ) = await asyncio.gather(
        op1_session_scope_window_drop(),
        op3_cohesion_cache_refresh(newly_archived_ids),
        op5_l0_dedup_flush(session_id),
        op6_session_summary_store(session_summary_text),
        return_exceptions=True,  # isolate failures — one op's failure must not cancel others
    )

    # ─── Phase 2: Op4 and Op2 depend on Op1 completing ─────────────────────────
    # Both are independent of each other — run in parallel.
    await asyncio.gather(
        op4_scope_registry_cleanup(deleted_ids),
        op2_working_set_counter_update(accessed_memory_ids),
        return_exceptions=True,
    )
```

### Why `return_exceptions=True` is mandatory

Without `return_exceptions=True`, a single failure in any Phase 1 task cancels all sibling tasks. If Op5 (Qdrant batch query) times out, Op6 (summary store) and Op3 (cohesion refresh) would be cancelled. This is exactly the failure mode where hook unreliability causes silent data loss. With `return_exceptions=True`, each task's exception is captured as a return value. The Phase 2 launch checks whether `deleted_ids` is an exception (Op1 failure) before passing it to Op4.

### Handling Op1 failure before Phase 2

```python
    # Check Op1 result before Phase 2
    if isinstance(deleted_ids, Exception):
        # Op1 failed — skip Op4 (cannot clean registry without deleted IDs)
        # Still run Op2 (working-set update is independent)
        await op2_working_set_counter_update(accessed_memory_ids)
        log.error("Op1 failed, scope registry cleanup skipped", exc=deleted_ids)
    else:
        await asyncio.gather(
            op4_scope_registry_cleanup(deleted_ids),
            op2_working_set_counter_update(accessed_memory_ids),
            return_exceptions=True,
        )
```

## Wall-Clock Time Estimate at N=50 New Memories per Session

### Phase 1 latency (dominated by the slowest parallel task)

| Op | Component | p50 | p99 | Notes |
|----|-----------|-----|-----|-------|
| Op1 | PostgreSQL filter-DELETE + Qdrant delete by filter | 15ms | 40ms | One SQL DELETE, one Qdrant filter delete |
| Op3 | 500 Qdrant HNSW searches (50 new memories × 10 neighbors) | 800ms | 1200ms | Q043: ~1.6ms per cohesion query, 500 affected memories |
| Op5 | Batch HNSW over 50 session memories + merge writes | 250ms | 500ms | Q043: N=50 batch = 250ms p99 |
| Op6 | Ollama embed (50ms) + Qdrant upsert (5ms) + PostgreSQL (2ms) + G1/G3 pre-checks (2ms) | 60ms | 90ms | Q041: ~65ms total store path |

**Phase 1 wall-clock = max(Op1, Op3, Op5, Op6) = max(40ms, 1200ms, 500ms, 90ms) = ~1200ms (Op3 dominates)**

Op3 is the critical path. At p99, cohesion refresh for 500 affected memories takes 1.2 seconds.

### Phase 2 latency (dominated by the slower parallel task)

| Op | Component | p50 | p99 | Notes |
|----|-----------|-----|-----|-------|
| Op4 | Redis HSCAN + conditional HDELs for deleted_ids | 5ms | 20ms | Fast for typical deletion count |
| Op2 | Two PostgreSQL UPDATEs across N_total_memories | 50ms | 200ms | Second UPDATE is O(N_total) — Q042: 50K updates/day trivial but still sequential scan |

**Phase 2 wall-clock = max(Op4, Op2) = max(20ms, 200ms) = ~200ms (Op2 dominates)**

At N_total_memories=10,000, the second UPDATE in Op2 (incrementing all non-accessed rows) is a sequential scan touching ~9,950 rows. At 20µs per row update: 199ms. This is the longest Phase 2 task.

### Total hook latency estimate at N=50

| Phase | Wall-clock (p50) | Wall-clock (p99) |
|-------|-----------------|-----------------|
| Phase 1 | ~850ms | ~1200ms |
| Phase 2 | ~50ms | ~200ms |
| **Total** | **~900ms** | **~1400ms** |

This fits within a 2-second SLA budget (the Q024 session-boundary compaction benchmark uses a 2-second ceiling). The critical path is Op3 (cohesion refresh). The session summary (Op6) at 60–90ms is fast enough that it does not delay Phase 2 startup — Phase 2 waits only for Op1, which completes in <40ms p99.

### Optimization: Op3 is an optional Phase 2 upgrade path

Op3 at ~800ms p50 is the session-end hook's dominant cost. It is also the least time-sensitive: cohesion scores are used only by the Tier 3 eviction policy, which is rarely triggered. If the 2-second SLA becomes constrained, Op3 can be demoted to an asynchronous background task (fire-and-forget via a Redis-backed task queue) with a 1-session staleness tolerance. Q043 explicitly notes that one session of cohesion staleness is acceptable for the eviction use case.

**Revised execution plan with Op3 as background task:**

```python
    (deleted_ids, _, _) = await asyncio.gather(
        op1_session_scope_window_drop(),
        op5_l0_dedup_flush(session_id),
        op6_session_summary_store(session_summary_text),
        return_exceptions=True,
    )

    # Op3 deferred: enqueue for background worker rather than blocking hook
    await redis.lpush("recall:background:cohesion_refresh",
                      json.dumps(newly_archived_ids))

    # Phase 2 unchanged
    ...
```

With Op3 moved to background: Phase 1 drops to max(40ms, 500ms, 90ms) = ~500ms p99. Total hook wall-clock: ~700ms p99.

## Idempotency Analysis

Which operations are safe to retry if the Stop hook fires twice (e.g., if Claude Code crashes and restarts within the same session)?

| Op | Idempotent? | Reason |
|----|------------|--------|
| Op1 (window-drop) | **Yes** | A DELETE WHERE filter on already-deleted rows produces zero rows affected. No error, no duplicate deletion. |
| Op2 (working-set update) | **No** | A double-fire increments `sessions_since_last_access` twice for the same session boundary. The counter overestimates by 1. For non-accessed memories: counted as 2 sessions elapsed instead of 1. |
| Op3 (cohesion refresh) | **Yes** | A recomputed cohesion score overwrites the same PostgreSQL column. Overwriting with the same value is idempotent. Two cohesion refreshes within one session produce the same final state. |
| Op4 (scope registry cleanup) | **Yes** | Redis HDEL on already-deleted keys is a no-op. Q036 idempotency pattern applies. |
| Op5 (L0 dedup flush) | **Partially** | On second fire: `recall:staging:{session_id}` was already deleted by the first fire. LRANGE returns empty set — no re-processing, no harm. However, the promote-to-global step may have already added entries to `recall:staging:global`. Second fire: session staging key is empty, so nothing is promoted again. Safe if the session staging key was atomically deleted (GETDEL or LRANGE+DEL within a Redis transaction). |
| Op6 (session summary store) | **Yes** | The three-gate store protocol (Q041) uses content-derived idempotency keys. The session summary content is deterministic per session. A second `/recall/store` call with the same content hits the G3 gate and returns the cached response without writing again. |

### Op2 requires deduplication guard

Op2 is the only non-idempotent operation. The fix: record in the session state (Redis key `recall:session:{session_id}:working_set_updated`) that Op2 has executed. On second fire, check this key before running Op2.

```python
async def op2_working_set_counter_update(session_id, accessed_ids):
    guard_key = f"recall:session:{session_id}:working_set_updated"
    already_ran = await redis.get(guard_key)
    if already_ran:
        return  # Idempotency guard: skip on re-fire
    # ... execute both UPDATEs ...
    await redis.set(guard_key, "1", ex=86400)  # TTL: 24 hours
```

This makes Op2 idempotent within a 24-hour window — sufficient to cover all plausible double-fire scenarios.

## Scope Registry Cleanup: Implementation Detail

Op4 requires scanning the Redis scope registry for entries whose UUID value matches a deleted memory ID. The Q041 Redis schema stores values as `{uuid}:{hlc}` — so Op4 must parse the compound value to extract the UUID before comparison.

The naive approach (HSCAN entire registry, parse every value) is O(registry_size). For a registry with 1,000 scope bindings and Op1 deleting 10 memories, 990 scans produce no deletions.

A more efficient approach: maintain a reverse index in Redis. At memory write time, in addition to the forward `scope_registry: topic_key@machine_id → uuid:hlc`, write a reverse entry `recall:uuid_to_scope:{uuid} → topic_key@machine_id`. Op4 then:

```
for uuid in deleted_memory_ids:
    scope_key = await redis.get(f"recall:uuid_to_scope:{uuid}")
    if scope_key:
        await redis.hdel("scope_registry", scope_key)
        await redis.delete(f"recall:uuid_to_scope:{uuid}")
```

This is O(N_deleted) instead of O(N_registry). At N_deleted=10 and N_registry=1,000, the reverse index is 100x cheaper.

## Interaction with the Three-Gate Store Protocol (Q041)

The session-end hook's six operations interact with the three-gate protocol at Op6 and Op5:

**Op6**: Calls the store endpoint directly, passing through G1 (scope identity), G2 (HLC), and G3 (idempotency). The session summary is a free-form memory — G1 will likely miss (no stable topic key for a session summary). G3 catches exact duplicate summaries (unlikely but safe). G2 assigns the HLC. No special interaction required.

**Op5**: The L0 dedup flush reads memories from `recall:staging:{session_id}` — these are memories already written through the three-gate protocol during the session. The L0 dedup pass does not re-run the gate protocol; it operates on already-committed memories in Qdrant. The dedup pass runs HNSW queries and merges at the Qdrant/PostgreSQL level directly, bypassing the store endpoint. This avoids triggering G3 (which would see the same content hash from the session and return cached responses, preventing dedup from reading the actual memory objects). The L0 dedup pass is an administrative operation, not a store operation.

## Complete Reference Implementation

```python
import asyncio
import hashlib
import json
import logging
from typing import Optional

log = logging.getLogger("recall.session_end_hook")


async def session_end_hook_executor(
    session_id: str,
    accessed_memory_ids: list[str],
    newly_archived_ids: list[str],  # from Tier2→3 archival step (precondition)
    session_summary_text: str,
    defer_cohesion_to_background: bool = False,
) -> dict[str, object]:
    """
    Optimal session-end hook: 2-phase asyncio execution.
    Phase 1: Op1 + Op3 (or background) + Op5 + Op6 in parallel.
    Phase 2: Op4 + Op2 in parallel (both gated on Op1 completion).
    Total wall-clock at N=50: ~900ms p50, ~1400ms p99 (Op3 inline).
    With Op3 deferred:       ~500ms p50,  ~700ms p99.
    """

    results = {}

    # ─── Phase 1 ────────────────────────────────────────────────────────────────
    phase1_tasks = [
        op1_session_scope_window_drop(),
        op5_l0_dedup_flush(session_id),
        op6_session_summary_store(session_summary_text),
    ]

    if not defer_cohesion_to_background:
        phase1_tasks.append(op3_cohesion_cache_refresh(newly_archived_ids))
    else:
        # Enqueue for background worker — acceptable 1-session staleness
        await redis_client.lpush(
            "recall:background:cohesion_refresh",
            json.dumps(newly_archived_ids)
        )

    phase1_results = await asyncio.gather(*phase1_tasks, return_exceptions=True)

    # Op1 result is always first in phase1_tasks
    deleted_ids_result = phase1_results[0]
    results["op1_window_drop"] = deleted_ids_result
    results["op5_l0_flush"] = phase1_results[1]
    results["op6_summary_store"] = phase1_results[2]

    # ─── Phase 2 ────────────────────────────────────────────────────────────────
    if isinstance(deleted_ids_result, Exception):
        log.error(
            "Op1 (session-scope-window-drop) failed — skipping Op4 (scope registry cleanup)",
            exc_info=deleted_ids_result
        )
        # Op2 still runs — it does not depend on Op1's output
        results["op2_working_set"] = await op2_working_set_counter_update(
            session_id, accessed_memory_ids
        )
        results["op4_scope_cleanup"] = None
    else:
        deleted_ids: list[str] = deleted_ids_result
        phase2_results = await asyncio.gather(
            op4_scope_registry_cleanup(deleted_ids),
            op2_working_set_counter_update(session_id, accessed_memory_ids),
            return_exceptions=True,
        )
        results["op4_scope_cleanup"] = phase2_results[0]
        results["op2_working_set"] = phase2_results[1]

    return results


# ─── Operation implementations ─────────────────────────────────────────────────

async def op1_session_scope_window_drop() -> list[str]:
    """DELETE session-scoped memories below importance threshold."""
    # Get IDs first (needed for Op4)
    rows = await pg.fetch(
        """
        SELECT id FROM memories
        WHERE scope = 'session'
          AND importance < $1
          AND sessions_since_creation > 0
        """,
        IMPORTANCE_THRESHOLD,
    )
    ids = [str(r["id"]) for r in rows]
    if ids:
        # Delete from PostgreSQL
        await pg.execute(
            "DELETE FROM memories WHERE id = ANY($1::uuid[])", ids
        )
        # Delete from Qdrant
        await qdrant.delete(
            collection_name="memories",
            points_selector=ids
        )
    return ids


async def op2_working_set_counter_update(
    session_id: str, accessed_ids: list[str]
) -> None:
    """Bulk working-set epoch increment (Q042 mechanism). Idempotency-guarded."""
    guard_key = f"recall:session:{session_id}:working_set_updated"
    if await redis_client.get(guard_key):
        log.info("Op2 idempotency guard hit — skipping double-fire")
        return

    # Reset counter for accessed memories
    if accessed_ids:
        await pg.execute(
            "UPDATE memories SET sessions_since_last_access = 0 WHERE id = ANY($1::uuid[])",
            accessed_ids,
        )
    # Increment counter for all others
    await pg.execute(
        "UPDATE memories SET sessions_since_last_access = sessions_since_last_access + 1"
        " WHERE id != ALL($1::uuid[])",
        accessed_ids or [],
    )
    await redis_client.set(guard_key, "1", ex=86400)


async def op3_cohesion_cache_refresh(newly_archived_ids: list[str]) -> None:
    """Incremental cohesion score update for newly archived memories (Q043)."""
    if not newly_archived_ids:
        return
    # For each archived memory, query Qdrant for neighbors and compute cohesion
    # Batch: all archived memories in one pass (500 affected total at N_archive=50)
    await update_cohesion_scores_incremental(newly_archived_ids)


async def op4_scope_registry_cleanup(deleted_ids: list[str]) -> None:
    """Remove dangling scope registry bindings for deleted memories."""
    if not deleted_ids:
        return
    # Use reverse index (recall:uuid_to_scope:{uuid}) for O(N_deleted) lookup
    pipe = redis_client.pipeline()
    for uid in deleted_ids:
        pipe.get(f"recall:uuid_to_scope:{uid}")
    scope_keys = await pipe.execute()

    del_pipe = redis_client.pipeline()
    for uid, scope_key in zip(deleted_ids, scope_keys):
        if scope_key:
            del_pipe.hdel("scope_registry", scope_key)
            del_pipe.delete(f"recall:uuid_to_scope:{uid}")
    await del_pipe.execute()


async def op5_l0_dedup_flush(session_id: str) -> None:
    """Promote session-staging queue to global L1; run L0 intra-session dedup (Q035)."""
    staging_key = f"recall:staging:{session_id}"
    # Atomic read-and-delete of session staging partition
    async with redis_client.pipeline(transaction=True) as pipe:
        pipe.lrange(staging_key, 0, -1)
        pipe.delete(staging_key)
        session_memory_ids, _ = await pipe.execute()

    if session_memory_ids:
        # L0 dedup: batch HNSW over session memories
        await run_l0_dedup_pass(session_memory_ids)
        # Promote survivors to global queue
        await redis_client.lpush("recall:staging:global", *session_memory_ids)


async def op6_session_summary_store(summary_text: str) -> Optional[str]:
    """Store session summary via the three-gate store protocol (Q041)."""
    content_hash = hashlib.sha256(
        summary_text.strip().lower().encode()
    ).hexdigest()
    response = await recall_api.post(
        "/memories/store",
        json={"content": summary_text, "scope": "session", "tags": ["session_summary"]},
        headers={"X-Idempotency-Key": content_hash},
    )
    return response.get("uuid")
```

## Ideas Extracted

### Idea: two-phase-asyncio-session-end-sequencing

- **Novelty**: 0.80 — No production AI memory system has published a dependency graph and asyncio execution plan for a multi-operation session-end hook. The specific identification that Op1→Op4 is the only hard producer-consumer dependency (enabling all others to parallelize into two phases) is an architectural finding that is not found in any reviewed system. The related Q035 finding addressed the L0/L1 ordering constraint but did not address the full six-operation set.
- **Evidence**: 0.82 — The dependency analysis rests on well-established distributed systems principles: producer-consumer ordering (standard), asyncio.gather() for independent I/O-bound tasks (Python docs, PEP 3156), return_exceptions=True for failure isolation (asyncio best practice). ### Idea: reverse-uuid-to-scope-index

- **Novelty**: 0.72 — Maintaining a reverse index in Redis (`recall:uuid_to_scope:{uuid} → scope_key`) alongside the forward scope registry enables Op4 to perform O(N_deleted) cleanup instead of O(N_registry) HSCAN. This specific optimization for AI memory scope registry cleanup is not found in any reviewed system.
- **Evidence**: 0.85 — Reverse indexes for efficient reverse lookups are a standard database design pattern (PostgreSQL functional indexes, Redis sorted set scoring). The tradeoff is documented in Redis In Action and the Redis documentation on secondary indexing. The O(N_deleted) vs. O(N_registry) improvement is straightforward.
- **Feasibility**: 0.95 — Implementation requires one additional Redis WRITE per memory store operation: `redis.set(f"recall:uuid_to_scope:{uuid}", scope_key)`. Op4 then reads the reverse index instead of scanning the registry. Implementation: half a day. The reverse index TTL should match the scope registry TTL (no expiry for permanent scope bindings; TTL for session-scoped bindings).
- **Description**: At memory write time, when the G1 scope registry is updated (`HSET scope_registry topic_key@machine_id {uuid}:{hlc}`), also write a reverse mapping: `SET recall:uuid_to_scope:{uuid} topic_key@machine_id`. At session-end Op4, look up each deleted UUID in the reverse index instead of scanning the full registry hash. Delete the registry entry and the reverse index entry atomically via a Redis pipeline. This reduces Op4's latency from O(N_scope_registry) to O(N_deleted), which at typical values (N_deleted=10, N_registry=1,000) is a 100x improvement.

### Idea: op2-session-scoped-idempotency-guard

- **Novelty**: 0.68 — Using a session-scoped Redis key as an idempotency guard for a bulk SQL UPDATE operation to prevent double-increment on hook re-fire is a simple but not widely documented pattern for session-end hook safety. The specific application to working-set counter updates is novel in the AI memory context.
- **Evidence**: 0.80 — Redis as a distributed lock/flag is production-standard. The SET ... NX EX pattern (set-if-not-exists with TTL) is described in the Redis documentation as the canonical distributed lock primitive. The idempotency concern is directly validated by Q036's analysis of multi-fire hook scenarios.
- **Feasibility**: 0.97 — One Redis SET NX EX call at the top of Op2, one SET call at the bottom. Three lines of code. No schema changes. No new infrastructure.
- **Description**: Before executing Op2's bulk UPDATEs, check `redis.get(f"recall:session:{session_id}:working_set_updated")`. If set, return immediately (guard hit — double-fire detected). After both UPDATEs complete successfully, set the guard key with a 24-hour TTL. This makes Op2 idempotent within the window of any plausible hook re-fire scenario (crash recovery, hook timeout, duplicate Stop event). The 24-hour TTL ensures the guard is cleared before the next day's sessions could be incorrectly guarded.

## Key Insight

The session-end hook's six operations resolve into exactly two dependency levels. The single hard constraint is Op1→Op4. Everything else is independent. This means the hook's wall-clock time is bounded by:

```
total = Phase1_duration + Phase2_duration
      = max(Op1, Op3, Op5, Op6) + max(Op4, Op2)
```

At N=50 new memories per session, Phase 1 is dominated by Op3 (~1200ms p99). If Op3 is moved to a background task, Phase 1 drops to ~500ms p99 and total hook time is ~700ms — well within the 2-second SLA. The choice of whether to run Op3 inline or in background is the single highest-leverage architectural decision for the hook's latency budget.
