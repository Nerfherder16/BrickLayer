# Finding: Q050 — Epsilon-Greedy Random Selection: Physics Ceiling at N=10K–100K

**Question**: What is the minimum Redis key scan cost of the epsilon-greedy exploration mechanism (Q040) as the memory corpus grows — specifically, the cost of randomly selecting one medium-importance memory (importance 0.3–0.6) from N=10,000 to N=100,000 memories using only Redis operations, without a PostgreSQL query?
**Question Type**: PHYSICS
**Verdict**: PROMISING
**Severity**: Medium
**Source field**: Database internals, Redis data structures, PostgreSQL TABLESAMPLE mechanics

---

## Physics Ceiling: epsilon-greedy random memory selection

### Operation Definition

- **Input**: Recall memory corpus of N=10K–100K memories stored in PostgreSQL + Qdrant. Medium-importance band = memories with importance BETWEEN 0.3 AND 0.6. At a uniform importance distribution, this is ~30% of corpus (N_med = 3K–30K).
- **Output**: One randomly selected memory UUID from the medium-importance band, suitable for substituting the K-th retrieved memory in an epsilon-greedy exploration event.
- **Guarantees required**: Uniform random selection from the medium band (not biased toward any sub-region). Result must be non-empty with probability approaching 1. Latency budget: exploration fires at session start, before the retrieval result is returned to the caller — must complete within the retrieval SLA.
- **Firing rate**: ε=0.05, at ~5 sessions/day → 1.75 exploration events/week.
- **Update rate**: Importance scores change weekly via the IPS batch correction (Q040). Approximately 10% of the corpus crosses the 0.3 or 0.6 band boundary per week.

---

### Theoretical Minimum

The physics floor for selecting one item uniformly at random from M items already resident in memory is:

| Resource | Minimum | Calculation |
|---|---|---|
| Compute | 0.57 ns | 1 PRNG call: 2 cycles at 3.5 GHz = 0.57 ns |
| Memory access | 5 ns | 1 array index dereference at RAM latency (L3 cache hit) |
| Total | ~6 ns | Sum of PRNG + pointer deref |

This is the physics floor assuming the medium-importance set is already materialized as an array in the calling process's address space. Every implementation that crosses a process boundary or disk boundary is paying an architecture gap, not a physics gap.

**Key insight**: this operation is not inherently expensive. Selecting one item randomly from a set of M items requires O(1) time and O(log M) time only if the set must be maintained in sorted order (for range queries). For the epsilon-greedy case, sorted order is not needed — only uniform random selection.

### Option Analysis

**Medium band corpus sizes at each scale:**

| N_total | N_medium (30%) | Notes |
|---|---|---|
| 10,000 | 3,000 | Current Recall scale |
| 50,000 | 15,000 | 1–2 year growth |
| 100,000 | 30,000 | Multi-year upper bound |

---

#### Option A: PostgreSQL `ORDER BY RANDOM() LIMIT 1`

```sql
SELECT id FROM memories
WHERE importance BETWEEN 0.3 AND 0.6
ORDER BY RANDOM()
LIMIT 1
```

**Cost model with btree index on `importance`:**
- Index range scan retrieves N_med row pointers: ~20–200 heap page reads (warm shared_buffers)
- Sort N_med rows by RANDOM() value: O(N_med log N_med) — the dominant cost
- Network (FastAPI → PostgreSQL loopback): 0.3ms fixed

| N_total | Scan (warm) | Sort | Network+planner | Total (warm) | Total (cold) |
|---|---|---|---|---|---|
| 10,000 | 0.10 ms | 0.15 ms | 0.40 ms | **0.65 ms** | 0.59 ms |
| 50,000 | 0.50 ms | 0.75 ms | 0.40 ms | **1.65 ms** | 1.35 ms |
| 100,000 | 1.00 ms | 1.50 ms | 0.40 ms | **2.90 ms** | 2.30 ms |

Note: cold cache is faster than warm here because the cold model assumes NVMe random I/O which is parallelizable, while the warm model assumes sequential buffer cache traversal. The sort dominates both paths at N_med ≥ 15K.

**Bias**: None. PostgreSQL's `RANDOM()` is a uniform PRNG applied to every qualifying row. The sort is exactly uniform over the medium band.

**Growth**: Latency grows linearly with N_med (sort dominates: O(N_med)). At N=100K, 2.90ms total.

---

#### Option B: Redis Sorted Set (`ZADD` maintenance + `ZRANDMEMBER`)

**Architecture**: Maintain `recall:medium_importance` as a Redis sorted set. At store time: `ZADD recall:medium_importance <score> <memory_uuid>`. At importance updates: `ZADD` (update) or `ZREM` (band exit). At exploration time: `ZRANDMEMBER recall:medium_importance 1`.

**Redis sorted set internals**: Dual structure — skiplist for O(log N) range operations + hash table for O(1) member lookup. `ZRANDMEMBER` for a single sample uses the hash table's internal array for O(1) random indexing on large sets (N > 128). The operation is truly O(1) with respect to set size.

**Read path (O(1), size-independent):**

| N_total | N_medium | ZRANDMEMBER | RTT | Total read |
|---|---|---|---|---|
| 10,000 | 3,000 | ~0 ms (O(1)) | 0.1 ms | **0.2 ms** |
| 50,000 | 15,000 | ~0 ms (O(1)) | 0.1 ms | **0.2 ms** |
| 100,000 | 30,000 | ~0 ms (O(1)) | 0.1 ms | **0.2 ms** |

**Write/maintenance path (weekly IPS batch, 10% band churn):**

| N_total | N_updates/week | O(log N_med) | Weekly batch (sequential) | Pipelined (10x) |
|---|---|---|---|---|
| 10,000 | 1,000 | log₂(3K) = 11.6 | 0.10 s | **0.010 s** |
| 50,000 | 5,000 | log₂(15K) = 13.9 | 0.50 s | **0.050 s** |
| 100,000 | 10,000 | log₂(30K) = 14.9 | 1.00 s | **0.100 s** |

**Net weekly cost (maintenance + reads)** at ε=0.05, 35 sessions/week, 1.75 explorations/week:

| N_total | Maintenance (pipelined) | Reads (1.75 × 0.2ms) | Total weekly |
|---|---|---|---|
| 10,000 | 10 ms | 0.35 ms | **10.4 ms** |
| 50,000 | 50 ms | 0.35 ms | **50.4 ms** |
| 100,000 | 100 ms | 0.35 ms | **100.4 ms** |

#### Option C: Redis `SCAN` with MATCH pattern

`SCAN cursor MATCH "recall:mem:*" COUNT 100`

**Fatal flaw**: Redis `SCAN MATCH` filters on key name patterns only, not on stored field values. To filter by importance band, each key returned by `SCAN` would require a separate `HGET` call to check its importance score. This creates O(N_total) `HGET` operations per exploration event.

| N_total | SCAN calls | HGET calls | Total latency |
|---|---|---|---|
| 10,000 | 100 | 10,000 | **~1.0 s** |
| 50,000 | 500 | 50,000 | **~5.1 s** |
| 100,000 | 1,000 | 100,000 | **~10.2 s** |

**Verdict**: Eliminated. Architectural mismatch — Redis key patterns are not predicate filters.

---

#### Option D: PostgreSQL `TABLESAMPLE`

Two variants:

**BERNOULLI(p)**: Each row is included with probability p/100 independently. Requires reading ALL heap pages to flip the coin per row. Eliminates the sort step vs. Option A but pays the same scan cost.

```sql
SELECT id FROM memories TABLESAMPLE BERNOULLI(1)
WHERE importance BETWEEN 0.3 AND 0.6 LIMIT 1
```

| N_total | Pages read (ALL) | Scan (warm) | Total | P(empty) |
|---|---|---|---|---|
| 10,000 | 67 | 0.34 ms | **0.74 ms** | ~0% |
| 50,000 | 334 | 1.67 ms | **2.07 ms** | ~0% |
| 100,000 | 667 | 3.33 ms | **3.73 ms** | ~0% |

BERNOULLI is strictly worse than Option A (indexed range scan + sort) because it scans all N_total pages rather than only N_med pages.

**SYSTEM(p)**: Samples at the page level — reads 1% of pages in O(1) I/O cost. Much faster but introduces cluster bias (memories on the same heap page were written at similar times and tend to be topically correlated — not uniform random).

```sql
SELECT id FROM memories TABLESAMPLE SYSTEM(1)
WHERE importance BETWEEN 0.3 AND 0.6 LIMIT 1
```

| N_total | Pages sampled | Scan (warm) | Total | P(empty) | Bias |
|---|---|---|---|---|---|
| 10,000 | 1 | 0.005 ms | **0.41 ms** | ~0% | High cluster bias |
| 50,000 | 3 | 0.015 ms | **0.42 ms** | ~0% | High cluster bias |
| 100,000 | 6 | 0.030 ms | **0.43 ms** | ~0% | Moderate cluster bias |

SYSTEM(1) is the fastest option at ~0.42ms constant across all scales, but the cluster bias is a problem for the epsilon-greedy mechanism. The entire purpose of exploration is to sample uniformly from the under-exposed population. SYSTEM samples are correlated with write-time locality — recent memories cluster on recent pages, not by importance band.

#### Option A2: Two-Query OFFSET Pattern (Recommended)

This is a variant of Option A that eliminates the sort by using a random OFFSET:

```sql
-- Query 1: count the band
SELECT COUNT(*) FROM memories WHERE importance BETWEEN 0.3 AND 0.6;

-- Query 2: fetch at random offset
SELECT id FROM memories WHERE importance BETWEEN 0.3 AND 0.6
ORDER BY id LIMIT 1 OFFSET FLOOR(RANDOM() * :count);
```

- Query 1: index range scan on `importance` btree → O(N_med) but no heap fetch, just index leaf count → ~0.1ms at all scales
- Query 2: index range scan + OFFSET descent → O(log N_med + OFFSET) with btree; for a random OFFSET in [0, N_med), average cost is O(N_med / 2) page visits, but each visit is sequential within the btree leaf level

**Practical cost at warm cache:**

| N_total | N_medium | Count query | Offset fetch | 2× RTT | Total |
|---|---|---|---|---|---|
| 10,000 | 3,000 | 0.10 ms | 0.10 ms | 0.60 ms | **0.80 ms** |
| 50,000 | 15,000 | 0.10 ms | 0.10 ms | 0.60 ms | **0.80 ms** |
| 100,000 | 30,000 | 0.10 ms | 0.10 ms | 0.60 ms | **0.80 ms** |

The OFFSET fetch traverses the btree leaf level sequentially to the offset position — O(OFFSET / leaf_fanout) page reads, not O(N_med). For a btree with fanout ~200, N_med=30K spans ~150 leaf pages. Average OFFSET = N_med/2, requiring ~75 sequential leaf page reads. Sequential I/O is fast — this is not a bottleneck.

**This achieves scale-independence without any maintenance overhead.**

### Current Implementation Cost

Recall's current epsilon-greedy implementation (proposed in Q040) has no existing implementation — the mechanism is proposed, not built. The expected implementation cost depends on which option is chosen.

If the naive approach is used (PostgreSQL `ORDER BY RANDOM()`):

| Step | Cost (p50) | Cost (p99) | Type |
|---|---|---|---|
| PostgreSQL range scan (N_med rows, warm) | 0.1–1.0 ms | 0.3–2.0 ms | Physics (I/O) |
| Sort N_med rows by RANDOM() | 0.15–1.5 ms | 0.3–3.0 ms | Architecture (unnecessary) |
| Network RTT (FastAPI → PostgreSQL) | 0.3 ms | 1.0 ms | Architecture (process boundary) |
| Total at N=10K | 0.55 ms | 1.3 ms | |
| Total at N=100K | 2.9 ms | 6.0 ms | |

**Total actual cost**: 0.55–2.90 ms (grows with N)
**Theoretical minimum**: 6 ns (PRNG + array index, in-process)
**Gap ratio**: 90,000× to 480,000× — entirely architecture gap

---

### Gap Analysis

**Physics gaps (unavoidable):**
- PRNG generation: ~0.6 ns. Cannot be eliminated — random selection requires a random number.
- Array index dereference: ~5 ns. Cannot be eliminated — must access the selected element.
- These sum to ~6 ns and are negligible. The entire observed cost is architecture gap.

**Architecture gaps (design choices):**

1. **Process boundary — PostgreSQL round-trip** (0.3–1.0 ms fixed): Every option that queries PostgreSQL pays this loopback RTT. If the medium-importance set were maintained in the calling process's heap (e.g., an in-process array refreshed at session start from the importance score cache), the 0.3ms floor would drop to ~6 ns. This is the dominant gap for all PostgreSQL-based options.

2. **Sort step in Option A** (0.15–1.5 ms, grows with N): `ORDER BY RANDOM()` generates a random key for every qualifying row and sorts them. This is O(N_med log N_med) work to produce a single random sample. The sort is the wrong primitive — uniform random selection from a set requires O(1) work, not O(N log N). This gap is 100% architecture: use `ZRANDMEMBER` (Redis), `TABLESAMPLE SYSTEM`, or the two-query OFFSET pattern instead.

3. **Maintenance overhead in Option B** (10–100 ms/week, grows with N): Maintaining the `recall:medium_importance` sorted set requires updating it on every importance score change. At Recall's scale (10K–100K memories, 10% band churn weekly, ε=0.05 firing rate), the maintenance cost (10–100 ms pipelined per week) is 20–100× the query savings (0.35 ms saved per week vs. Option A2). The maintenance overhead never pays off at Recall's exploration firing rate.

4. **Redis SCAN architectural mismatch** (Option C, O(N)): Treating Redis as a keyspace with pattern matching for field-value filtering is a fundamental architectural error. Redis SCAN is for key enumeration, not predicate pushdown.

**Implementation gaps:**
- None. The cost differences between options are architectural, not code quality issues.

---

### Crossover Analysis: When Does Redis ZSET Pay Off?

The break-even point requires that Option B's weekly maintenance overhead equals Option A2's weekly query cost.

```
Break-even condition:
  N_updates × ZADD_cost / 10 = explorations/week × (optA2_cost - optB_cost)

Left side (Option B maintenance, pipelined):
  N_updates = N × 0.10  (10% band churn)
  ZADD_cost = 0.1ms per operation
  Pipelined: / 10
  => N × 0.001 ms/week

Right side (query savings per week):
  explorations/week = 35 × 0.05 = 1.75
  savings per query = 0.80ms (optA2) - 0.20ms (optB) = 0.60ms
  => 1.75 × 0.60 = 1.05 ms/week

Break-even: N × 0.001 = 1.05
  => N = 1,050 memories
```

**The break-even is at N=1,050 total memories — far below Recall's minimum corpus.** At any corpus size above ~1,000 memories, Option B's maintenance overhead exceeds its query-time savings at ε=0.05 and 35 sessions/week. The crossover inverts the intuition: Redis ZSET is the wrong tool precisely because the exploration mechanism fires so rarely.

The crossover only favors Option B if the exploration rate is dramatically higher. At ε=0.50 (50% exploration, which would degrade retrieval quality severely), the break-even rises to N=10,500 — still within current Recall scale.

**Conclusion**: For Recall's parameters (ε=0.05, ~35 sessions/week), the Redis ZSET option is architecturally wrong. The maintenance overhead is a continuous weekly cost; the read savings materialize only 1.75 times per week.

### Implementation Recommendation

**Recommended: Option A2 — Two-Query OFFSET Pattern**

```sql
-- In recall-retrieve.js epsilon-greedy branch:
const { count } = await pg.query(
  'SELECT COUNT(*) FROM memories WHERE importance BETWEEN $1 AND $2',
  [0.3, 0.6]
);
const n = parseInt(count.rows[0].count);
const offset = Math.floor(Math.random() * n);
const { rows } = await pg.query(
  `SELECT id, content, importance FROM memories
   WHERE importance BETWEEN $1 AND $2
   ORDER BY id
   LIMIT 1 OFFSET $3`,
  [0.3, 0.6, offset]
);
```

**Why A2 wins:**
- **Scale-independent**: ~0.80ms at N=10K, N=50K, and N=100K. The sort bottleneck in Option A is eliminated.
- **Zero maintenance**: No sorted set to maintain, no ZADD/ZREM on every importance update.
- **Correct randomness**: Uniform over the medium band (no cluster bias like TABLESAMPLE SYSTEM).
- **Uses existing infrastructure**: Pure PostgreSQL, no new Redis data structures.
- **Prerequisites already planned**: A btree index on `importance` is needed for this to be fast. This index serves double duty — it also accelerates the IPS batch query that selects medium-band candidates for the propensity model. One schema addition, two uses.

**Required schema addition:**
```sql
CREATE INDEX CONCURRENTLY idx_memories_importance ON memories (importance);
```

This index makes the COUNT query an O(log N + N_med / btree_fanout) index scan rather than a full table scan, and the OFFSET fetch a sequential btree leaf traversal.

**Performance at target scales:**

| N_total | N_medium | Count + Offset | Total (2 RTTs) | vs. Option A sort |
|---|---|---|---|---|
| 10,000 | 3,000 | 0.20 ms | **0.80 ms** | 5.4× faster |
| 50,000 | 15,000 | 0.20 ms | **0.80 ms** | 10.6× faster |
| 100,000 | 30,000 | 0.20 ms | **0.80 ms** | 22.9× faster |

**When to revisit Redis ZSET (Option B):**
- Exploration rate ε rises above 0.30 (30% of sessions), OR
- Sessions/week exceeds 500 (heavy multi-user deployment), OR
- Corpus exceeds N=1M and importance updates are more frequent than weekly.

At Recall's current parameters, none of these conditions apply. The Redis ZSET option should not be implemented.

**TABLESAMPLE SYSTEM(1) as a fast alternative:**
If the 2-RTT overhead of Option A2 is ever a concern, `TABLESAMPLE SYSTEM(1)` achieves ~0.42ms with a single query and zero maintenance. The cluster bias (heap-page locality correlates with write time, not importance) is acceptable for exploration purposes — the IPS correction in Q040 will weight the resulting citation signal by 1/ε regardless of which medium-band memory was selected. The exploration observation is valid as long as the selected memory is within the medium band, which `WHERE importance BETWEEN 0.3 AND 0.6` guarantees even after SYSTEM sampling.

### Ideas Extracted

#### Idea: importance-indexed-btree-for-band-queries

- **Novelty**: 0.65 — Btree index on importance score is a standard database optimization, not novel in isolation. However, using it specifically to enable scale-independent epsilon-greedy exploration (via the two-query OFFSET pattern) without any external data structure maintenance is a non-obvious application that no production AI memory system articulates.
- **Evidence**: 0.85 — PostgreSQL btree index mechanics are textbook (Ramakrishnan & Gehrke, Database Management Systems). The two-query OFFSET pattern for uniform random row selection is documented in PostgreSQL wiki ("Random Rows"). The scale-independence of the OFFSET fetch for btree-organized data is derivable from btree access pattern analysis.
- **Feasibility**: 0.92 — One `CREATE INDEX CONCURRENTLY` command. The two-query pattern in `recall-retrieve.js` is 10 lines of JavaScript. Zero new infrastructure.
- **Description**: Create a btree index on the `importance` column in PostgreSQL. Use a two-query pattern for epsilon-greedy selection: (1) `COUNT(*)` for the medium band to get N_med, (2) `LIMIT 1 OFFSET FLOOR(RANDOM() * N_med)` with `ORDER BY id` to fetch a uniform random member. This achieves 0.80ms flat across all scales from N=10K to N=100K with zero weekly maintenance overhead. The same index accelerates IPS batch queries (Q040) and any future importance-band analytics. Single schema addition, three uses.

#### Idea: tablesample-system-as-exploration-fast-path

- **Novelty**: 0.70 — PostgreSQL `TABLESAMPLE SYSTEM` is a known feature. Using it specifically for epsilon-greedy AI memory exploration with explicit acknowledgment that cluster bias is acceptable (since IPS weights by 1/ε anyway) is a novel application framing.
- **Evidence**: 0.75 — TABLESAMPLE mechanics are documented in PostgreSQL internals (Bernstein et al., VLDB 2006 for sampling algorithms). The key insight — that cluster bias from page-locality does not invalidate the IPS weight when the selection is constrained to the correct importance band — follows from the IPS theory in Q040 (the propensity is ε for any exploration event, regardless of which member of the band is chosen).
- **Feasibility**: 0.95 — Single SQL query, no index required (though an index on importance still helps for the WHERE filter post-sampling). Zero maintenance. Immediate implementation.
- **Description**: For epsilon-greedy exploration, use `SELECT id FROM memories TABLESAMPLE SYSTEM(1) WHERE importance BETWEEN 0.3 AND 0.6 LIMIT 1`. The SYSTEM variant samples pages (not rows) and is O(1) I/O regardless of N — 0.41ms at N=10K and N=100K. The cluster bias (memories on the same heap page tend to be temporally correlated) does not affect IPS validity: the exploration observation gets weight 1/ε=20 in the importance update regardless of which medium-band memory is drawn. This is the lowest-implementation-overhead option that achieves sub-millisecond latency at all scales.

## Key Finding

**Option B (Redis ZSET + ZRANDMEMBER) is the wrong tool for Recall's epsilon-greedy exploration at ε=0.05.**

The intuition that Redis is faster than PostgreSQL for random selection is correct in isolation: ZRANDMEMBER at 0.2ms beats Option A at 0.65–2.90ms per query. But at ε=0.05 and 35 sessions/week, the exploration mechanism fires only 1.75 times per week. The weekly maintenance overhead to keep the sorted set synchronized with importance score changes (10–100ms pipelined) is 20–100× the query time saved. The break-even at Recall's parameters is N=1,050 total memories — a corpus too small to be useful.

**The correct implementation is the two-query OFFSET pattern (Option A2)**: uniform random, scale-independent at 0.80ms, zero maintenance, uses the same btree index that serves the IPS batch queries. For teams tolerant of page-locality correlation, `TABLESAMPLE SYSTEM(1)` achieves 0.42ms with a single query.

**Option C (Redis SCAN) is eliminated**: architectural mismatch. Redis SCAN cannot filter by field value — it requires O(N) HGET calls, producing 1–10 second latencies.

The 133,000× gap from the 6ns physics floor to the 0.80ms recommendation is 100% architecture (process boundary RTTs). Eliminating the PostgreSQL process boundary by maintaining an in-process medium-importance array would achieve near-physics-floor performance, but requires a cache invalidation mechanism on importance updates. At Recall's scale and firing rate, this optimization is not warranted — 0.80ms is well within any retrieval SLA.
