## Research Question

How do package managers track dependency provenance? Is there a direct analog for memory attribution — tracing which stored memory influenced which agent output?

---

## Package Manager Provenance Model

### How npm/pip/cargo track dependency provenance

**Lock files** (package-lock.json, Pipfile.lock, Cargo.lock): Every dependency pinned to exact version + integrity hash. Resolves the "which version of X did we use?" question permanently.

**Dependency tree**: `npm ls`, `pip show`, `cargo tree` — who depends on what, at what version. Transitive dependencies are visible. "Why does this package exist?" = find the direct dependency that caused it.

**Content integrity**: SHA-256/512 hash of downloaded tarball. Cargo additionally signs packages (crates.io registry). npm has `--audit` for known vulnerabilities per version.

**Provenance chain for a package**:
```
my-app
└── fastapi@0.115.0        (direct: requires[fastapi])
    └── starlette@0.41.0   (indirect: fastapi depends on starlette)
        └── anyio@4.6.0    (indirect: starlette depends on anyio)
```

**The key insight**: provenance in package management is a DAG, not a list. Each node knows its parent(s) and its children. Attribution = traverse the DAG to find the root cause.

## Memory Attribution Analog

Apply the package manager model to memory attribution in Recall 2.0:

**The question being solved**: "Why did the agent say X?" → "Which stored memory caused the agent to include claim X in its output?"

### Direct mapping:

| Package concept | Memory attribution equivalent |
|---|---|
| Package version pinned in lock file | Memory ID + content hash at retrieval time |
| Direct dependency | Memory retrieved by query (K=10 results) |
| Transitive dependency | Memory retrieved in a previous session that informed a stored memory |
| `npm ls` — dependency tree | "Memory lineage" — which memories contributed to this session's output |
| Integrity hash | Content hash of stored memory (detect tampering) |
| Dependency tree traversal | CO_RETRIEVED graph traversal (memories that were co-retrieved with the attributed memory) |
| `npm audit` (vulnerability scan) | SourceTrust audit (which memories have low source trust) |

### The Memory Provenance Graph

A memory provenance graph differs from the existing Neo4j CO_RETRIEVED graph:

**CO_RETRIEVED graph** (existing): edges = "these two memories were retrieved together." Captures which memories are typically co-retrieved. Undirected, weighted by co-retrieval frequency.

**Memory provenance graph** (proposed): edges = "this agent output Y was influenced by memory X." Directed: Memory → Agent Output. Connected via session_id and retrieval event logs.

The two graphs are complementary:
- CO_RETRIEVED: which memories are semantically related (discovery, consolidation)
- Memory provenance: which memories caused which outputs (attribution, audit, debugging)

### Data structure

```
Retrieval event (at UserPromptSubmit):
  query_hash: SHA256(query_text)
  session_id: sid_abc
  retrieved_memory_ids: [mem_001, mem_004, mem_012]
  scores: [0.94, 0.87, 0.82]
  timestamp: T

Agent output event (at Stop hook):
  session_id: sid_abc
  output_hash: SHA256(agent_response_text)
  source_memory_ids: [mem_001, mem_004]  # subset that appeared in injected context
  timestamp: T+90min
```

The edge is: `output_hash → [source_memory_ids]`.

**Query**: "Why did the agent say X?"
1. Hash X or search for it in output log
2. Find output_hash entries containing X
3. Traverse: output_hash → source_memory_ids → memory content

### How it differs from Neo4j CO_RETRIEVED graph

The CO_RETRIEVED graph already in Recall 2.0 is closer to a "co-occurrence network" than a provenance DAG. Adding the agent output event log creates a true provenance chain:

```
output: "Recall 2.0 uses LMDB+HNSW"
    ↑ (attribution edge)
memory: "Recall 2.0 uses LMDB for persistence and HNSW for similarity search"
    [mem_001, score=0.94, session_id=sid_abc]
    ↑ (storage edge)
session: sid_abc, 2026-03-17 (observe-edit hook stored this from a file edit)
```

This is a three-hop attribution chain: output → memory → session → file edit. "Why does the AI know this?" becomes a graph traversal, not a black box.

## Feasibility Assessment

**Is this buildable?**

Yes, with minor additions to existing infrastructure:

1. **Output event log**: At Stop hook, log output_hash + session_id. Storage: 64 bytes per output × 10 outputs/session × 730 sessions/year = 468KB/year. Negligible.

2. **Retrieval event log**: Already partially implemented via Q219 retrieval audit trail. Extend to include which retrieved memories actually appeared in the injected context (not just which K=10 were retrieved).

3. **Graph construction**: Build directed edges output → memory lazily (on query) or incrementally (at Stop hook). Store in LMDB alongside CO_RETRIEVED edges.

**Does it outperform CO_RETRIEVED?**

For attribution queries: yes. CO_RETRIEVED answers "what is typically retrieved together" — it cannot answer "why did the agent say X on March 17." The provenance graph answers the March 17 question directly.

For discovery and consolidation: CO_RETRIEVED is still better (broad co-occurrence patterns, not tied to specific outputs).

---

## Why This is INCREMENTAL (Not BREAKTHROUGH)

The memory provenance graph is a useful complement to CO_RETRIEVED and supports regulatory audit requirements (Q219). However:
- It requires a significant logging infrastructure investment
- The user-facing value ("why did the AI say X?") is a debugging/audit use case, not a daily-use feature
- Session WAL (Q226) + memory attribution (session_id per memory) provides 80% of the value at 5% of the cost
- The full provenance graph is a Phase 6+ feature, not Phase 4-5

The session_id-per-memory and session WAL (Q226) already give "which session stored this memory" and "what was learned per session" — the two most common attribution questions. The full output→memory→session DAG is overkill for current scope.

## Verdict

**INCREMENTAL.** The package manager provenance model maps cleanly to memory attribution, producing a directed output→memory→session DAG that is complementary to (not replacing) CO_RETRIEVED. The full provenance graph answers the "why did the agent say X?" question precisely. However, session_id per stored memory (already suggested in Q226) provides 80% of the attribution value at minimal cost. The full provenance DAG is a Phase 6 investment. Incremental because the mechanism is well-understood and the value is primarily for debugging/audit (not daily retrieval quality improvement).
