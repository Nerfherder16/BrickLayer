## Physics Ceiling: HNSW ANN Search at ef=64 vs ef=128

### Operation definition
- **Input**: 1024-dimensional float32 query vector; HNSW index over 22,423 stored vectors
- **Output**: Top-K results by Cosine similarity (Qdrant normalizes vectors at ingestion, reducing cosine to dot product)
- **Guarantees required**: ANN (approximate nearest neighbor) -- recall R@1 ≥ 0.95; no ordering or durability guarantees
- **Hardware**: Intel N100 (4 Gracemont cores, 3.4 GHz, AVX2, no AVX-512); 6 MB L3 cache; ~30 GB/s DDR4 memory bandwidth

---

### Theoretical Minimum

#### FLOP count

HNSW traversal complexity: O(ef × log₂(N) × 2d)

Where:
- N = 22,423 → log₂(22,423) = **14.45**
- d = 1024 → FLOPs per comparison = 2 × 1024 = **2,048** (dot product on normalized vectors)
- ef = number of candidates examined

| ef | Total FLOPs | Calculation |
|----|-------------|-------------|
| 64  | 1,894,343 FLOPs (~1.89 MFLOPs) | 64 × 14.45 × 2,048 |
| 128 | 3,788,686 FLOPs (~3.79 MFLOPs) | 128 × 14.45 × 2,048 |

ef=128 is exactly 2× the FLOP count of ef=64 (the ratio scales linearly with ef).

#### Compute-bounded minimum latency

N100 single-core throughput with AVX2 (no AVX-512):
- Peak theoretical: 3.4 GHz × 8 FLOPS/cycle (AVX2 FP32) = **27.2 GFLOPS**
- Realistic for HNSW (memory-latency limited): **~12 GFLOPS effective**

| ef | @ 27.2 GFLOPS peak | @ 12 GFLOPS realistic |
|----|--------------------|-----------------------|
| 64  | 69.6 µs | 157.9 µs |
| 128 | 139.3 µs | 315.7 µs |

#### Memory-bandwidth-bounded minimum

Vector size: 1024 × 4 bytes = **4,096 bytes (4 KB)** per vector.

Minimum vectors loaded (prompt formula: ef direct candidates only):
- ef=64: 64 × 4 KB = 256 KB → at 30 GB/s: **8.7 µs**
- ef=128: 128 × 4 KB = 512 KB → at 30 GB/s: **17.5 µs**

Full corpus size: 22,423 × 4 KB = **87.6 MB** — exceeds the N100's 6 MB L3 cache entirely. Vectors are not cached across queries. The pessimistic bound (ef × log₂(N) unique node accesses = ~1,850 vectors × 4 KB = 7.4 MB for ef=128) exceeds L3 but fits within RAM with ~2.9 GB/s effective bandwidth per the HNSW traversal pattern.

#### Binding constraint

The compute bound (315.7 µs at ef=128, realistic) exceeds the memory transfer floor (17.5 µs for 512 KB), making this **compute-bound in the theoretical model**. In practice, cache misses against the 87.6 MB corpus elevate the memory penalty, making the actual workload **co-bounded** between compute and memory latency.

| Resource | ef=64 minimum | ef=128 minimum |
|----------|--------------|----------------|
| Compute  | 157.9 µs (realistic GFLOPS) | 315.7 µs |
| Memory   | 8.7 µs (minimum transfer) | 17.5 µs |
| **Theoretical floor** | **~0.16 ms** | **~0.32 ms** |

### Current Implementation Cost

Qdrant's Rust HNSW engine operates well above the theoretical floor due to structural overhead:

| Step | ef=64 cost | ef=128 cost | Type |
|------|-----------|-------------|------|
| Python qdrant_client serialization (1024-dim vector → bytes) | ~0.2 ms | ~0.2 ms | implementation |
| gRPC/HTTP dispatch to Qdrant Rust engine | ~0.1–0.5 ms | ~0.1–0.5 ms | architecture |
| HNSW neighbor-list traversal (not captured in O(ef·log·d) formula; m=16 neighbor edges per node) | ~0.5–1.5 ms | ~1.0–3.0 ms | architecture |
| Cache miss penalty: 87.6 MB corpus >> 6 MB L3; cold-traversal miss rate ~60% on bottom layer | ~0.5–1.5 ms | ~1.0–2.5 ms | physics |
| Result serialization + response building | ~0.2–0.5 ms | ~0.2–0.5 ms | implementation |
| OS scheduling / context switch | ~0.05–0.1 ms | ~0.05–0.1 ms | physics |
| **Total expected operational latency (P50)** | **~2–4 ms** | **~3.5–7 ms** | — |
| **Total expected operational latency (P95)** | **~4–6 ms** | **~7–10 ms** | — |

**Theoretical minimum**: ef=64 → 0.16 ms; ef=128 → 0.32 ms
**Expected operational P50**: ef=64 → ~3 ms; ef=128 → ~5 ms
**Gap ratio**: ef=64 → ~19× above floor; ef=128 → ~16× above floor

The gap is slightly smaller for ef=128 because the fixed overhead (serialization, dispatch) amortizes over a larger compute kernel. This is expected behavior.

### Gap Analysis

**Physics gaps** (unavoidable):

- **L3 cache miss penalty**: 87.6 MB corpus >> 6 MB N100 L3. Every HNSW query touches vectors distributed across the full corpus. Cold-traversal cache miss rate ~60% on bottom layer. Estimated cost: 1–2 ms. Cannot be removed without larger L3 or moving to GPU (which has wider HBM bandwidth). Scaling the corpus to 100K vectors would increase this penalty only slightly (HNSW accesses ~ef × log(N) nodes, which grows as O(log N)).

- **OS scheduling**: Single-query Qdrant search on a 4-core N100. Query handling competes with background OS tasks. Context switch jitter: ~50–100 µs floor. Unavoidable without real-time OS.

**Architecture gaps** (design choices):

- **HNSW neighbor list overhead**: The O(ef × log(N) × d) formula undercounts actual comparisons. Qdrant's HNSW traversal visits each candidate's m=16 neighbor edges during beam search. Actual comparisons ≈ ef × m × (log₂(N) / 2) ≈ 128 × 16 × 7.2 = 14,746 comparisons (not 1,849). This 8× undercount in the formula is the primary gap between theoretical and observed. It is architectural to HNSW — not implementation noise — but it is what HNSW trades for sub-linear scaling.

- **Python binding serialization**: 1024 floats × 4 bytes = 4 KB per query vector must be marshaled from Python into the gRPC payload. Cost: ~0.2–0.5 ms. Removable by using the gRPC path natively (qdrant_client defaults to gRPC since v1.2) or by pre-serializing query vectors. Estimated savings: 0.1–0.3 ms.

- **REST vs gRPC path**: If Recall's qdrant_client is on an older version using REST, HTTP JSON overhead (~1–2 ms) replaces gRPC overhead (~0.1–0.3 ms). This is a client configuration gap. The Q144 finding confirms the codebase uses `qdrant_client` — checking the gRPC vs REST mode is worth 0.5–1.5 ms if REST is active.

**Implementation gaps** (code quality):

- **No query vector caching**: The oracle health check (Q148) runs the same 5 probe queries repeatedly (60-second cache TTL). The embedding vectors for these queries could be cached in Redis at startup, eliminating the Ollama call entirely for health checks. This is not the Qdrant gap, but it is the dominant cost in the oracle health check pipeline.

### SLA Budget Assessment: ef=128 vs Q148's 100ms Qdrant Stage Budget

Q148 specifies the oracle health check SLA structure:
- Total budget: 3,000 ms for the full 5-oracle health check
- Per oracle budget: ~600 ms sequential or ~300 ms concurrent (asyncio.gather)
- Qdrant stage budget: **≤100 ms per oracle query** (derived from pipeline stage decomposition)

**ef=128 Qdrant search time at P95: ~7–10 ms**

| ef | P50 Qdrant | P95 Qdrant | vs 100ms budget | Headroom |
|----|-----------|-----------|-----------------|----------|
| 64  | ~3 ms | ~6 ms | 94 ms surplus | 94% headroom |
| 128 | ~5 ms | ~10 ms | 90 ms surplus | 90% headroom |

**Assessment: ef=128 fits comfortably within the 100ms Qdrant stage budget.** The P95 worst case uses only 10% of the allocated budget.

The actual bottleneck in the oracle health check is **Ollama embedding generation** (~80–150 ms per probe query), not Qdrant. The ef_search upgrade from 64 to 128 adds at most ~4 ms to the Qdrant stage (P95 delta), which is invisible in the 80–150 ms Ollama baseline. The end-to-end oracle check timing is dominated by Ollama, not HNSW depth.

Even if ef were raised to 512 (4× current), the Qdrant stage would be ~20–40 ms — still within the 100 ms budget and still below the Ollama floor.

## Qualifying Ideas

- **embed-query-cache-for-health-check**: (N=0.72, E=0.78, F=0.92) — The oracle health check (Q148) embeds the same 5 probe queries on every invocation. These embeddings are deterministic (same text → same vector). Store the 5 oracle probe embeddings as pre-computed float32 arrays in Redis at startup. The health check then calls Qdrant directly with the cached vector, skipping the 80–150ms Ollama round-trip entirely. Oracle check latency drops from ~200–300ms to ~20–50ms. This is the highest-leverage optimization in the oracle health check pipeline.

- **ef-per-query-health-vs-production**: (N=0.65, E=0.70, F=0.90) — Use Qdrant's per-request `hnsw_ef` override (`SearchParams(hnsw_ef=64)`) for production recall queries where latency matters more, while the oracle health check (which is admin-only and cached for 60 seconds) uses ef=128 for higher recall fidelity. This would let the production path stay at ~3ms while health checks validate at ef=128 accuracy. Implementable via a single line change in the search params.

- **corpus-cache-warmup-on-startup**: (N=0.58, E=0.65, F=0.75) — On Qdrant container startup, issue a sweep of representative queries to warm the HNSW graph into OS page cache. The 87.6 MB corpus fits easily in the 16 GB system RAM. After warmup, repeated traversals hit RAM (via OS page cache, not L3), reducing effective memory latency from ~70ns (L3 miss → DRAM) to ~70ns (page cache hit — same). This is already the steady-state behavior after the first few queries; it eliminates cold-start latency spikes only.
