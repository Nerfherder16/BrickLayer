# Finding: Q265 — Observe-Edit Hook Information Structure Without Similarity-Based Retrieval

**Question**: Design the observe-edit hook assuming similarity-based retrieval is unavailable — what information must the hook send for retrieval to work?
**Question Type**: TABOO
**Verdict**: PROMISING
**Severity**: Medium
**Constraint**: Analysis is conducted without reference to: embedding, cosine, similarity, vector, dense, semantic. Design from first principles only.

---

## Evidence

### The Core Problem

The observe-edit hook fires on PostToolUse:Write, Edit, and Bash events. Its job is to extract information from tool output and store it as a memory unit. The design question is: what structure must that memory unit have such that a future retrieval query — asking about "the Recall API port" or "how async cascade routing works" — reliably surfaces it?

This is a retrieval design problem, not a storage design problem. The memory structure must be shaped to match the retrieval mechanisms available. Working backward from those mechanisms produces the required hook output.

---

### Section 1: Taxonomy of Retrieval Signals

#### Tier A — Lightweight-Extractable Signals (C+ / regex / AST)

These signals can be produced by the hook with no language model call. They are present in the raw text of Write/Edit/Bash output as structured or semi-structured patterns.

**1. Lexical anchors**
Exact-match tokens that a future query is likely to contain. Categories:
- *File paths*: `src/hooks/recall_retrieve.js`, `app/routers/memory.py` — producible from the tool input (the path argument to Write/Edit)
- *Function and class names*: extracted via regex from source code (`def handle_request`, `class MemoryStore`, `async function recallSearch`) — high lexical specificity
- *Import statements*: the imported library names (e.g., `fastapi`, `qdrant-client`, `neo4j`) — signal the domain and technology
- *Config keys and values*: `PORT=8200`, `host: 100.70.195.84`, `model: qwen3:14b` — high-value anchors because queries about infrastructure frequently use the exact key name
- *Numeric patterns*: port numbers, IP addresses, version strings, thresholds — regex-extractable and highly discriminating (`8200` occurs in far fewer memories than the word "port")
- *Error codes and HTTP status codes*: `404`, `422`, `500`, `ECONNREFUSED` — exact tokens in a narrow vocabulary
- *Environment variable names*: `RECALL_API_URL`, `OLLAMA_HOST` — camelCase/UPPER_SNAKE patterns are distinctive

**2. Structural signals**
Properties of the edit event itself, producible from hook metadata:
- *Tool type*: Write vs. Edit vs. Bash — each produces qualitatively different content
- *File extension*: `.py`, `.ts`, `.json`, `.yml`, `.env` — signals the likely token vocabulary of the content
- *Change type*: new file creation vs. modification (inferred from Write vs. Edit event type)
- *Line count delta*: proxy for change magnitude
- *Project path prefix*: the top-level directory component of the file path signals which project the edit belongs to

**3. Temporal and provenance signals**
- *Session ID*: links memories written in the same session — enables behavioral co-retrieval clustering
- *Timestamp*: enables recency ranking and temporal queries ("what changed this week")
- *Hook type*: Bash output has different retrieval properties than Write output (see Section 3)
- *Preceding tool chain*: which tool calls immediately preceded this write, if available from session context

**4. Explicit entity tags**
A small, deterministic set of high-value tag classes extracted by pattern matching:
- `port:<number>` — extracted when a numeric pattern in range 1024-65535 appears adjacent to words like "port", "listen", "bind", "expose"
- `host:<string>` — IP address or hostname patterns
- `version:<string>` — semver patterns (`1.4.2`, `^2.0`)
- `service:<name>` — known service name patterns (qdrant, neo4j, redis, ollama, etc.) when they appear as literal strings
- `path:<path>` — the file path itself, already a tag by construction

---

#### Tier B — LLM-Required Signals

These signals cannot be produced deterministically from the raw text. They require language-model inference to produce.

**1. Semantic paraphrase reformulation**
The critical gap in lexical retrieval: a future query uses different words than the stored content.

Canonical example: a config file contains `PORT=8200`. A future query asks "what port does the Recall API run on?" The word "port" appears in both query and memory. BM25 handles this case adequately — "port" and "PORT" are the same after normalization.

But: a function is named `handle_async_cascade_v2()`. A future query asks "how does Recall handle parallel request routing?" The words "parallel", "routing", and "handle" are absent from the function name. BM25 assigns this memory near-zero score for that query. Only an LLM can produce the paraphrase: "this function handles concurrent cascade routing and determines request dispatch order" — a natural-language description that uses the query vocabulary.

This is the one class of signal that is categorically unavailable to C+ processing and non-redundant with the Tier A signals listed above.

**2. Concept normalization**
The vocabulary gap between how a fact is stored and how it is queried:
- "remember" ≠ "persist" ≠ "store" — all can refer to the same mechanism
- "slow" ≠ "latency" ≠ "performance bottleneck" — all can describe the same observation
- "broken" ≠ "error" ≠ "regression" — all can describe the same failure

BM25 treats these as different tokens. An LLM extraction step can produce a normalized description that uses the standard vocabulary for the concept.

**3. Cross-reference bridging**
A file mentions a service name ("the recall API endpoint"), but the service's address is defined elsewhere. An LLM can recognize that this edit is *about* Recall's API and add the tag `service:recall`, which links it to memories that contain the actual address. Without LLM extraction, this bridge does not exist unless the service name appears as a literal string (Tier A handles that case, but not the pronoun/reference case).

**4. Intent capture for opaque identifiers**
Function names like `_proc_batch_v3`, variable names like `tmp_buf_k`, or Bash scripts with terse flag combinations carry no natural-language information. BM25 cannot match them to queries that use prose vocabulary. LLM extraction converts opaque identifiers to natural-language descriptions that BM25 can index.

### Section 2: Which Signals Are Load-Bearing in Recall 2.0's Retrieval Stack

Recall 2.0 has three retrieval mechanisms. Mapping each to the signal taxonomy:

**BM25 (keyword matching)**
Load-bearing signals: Tier A lexical anchors — file paths, function names, config keys, port numbers, IP addresses, library names. BM25 scores based on term frequency × inverse document frequency across the stored memory corpus. High-specificity tokens (port numbers, IP addresses, function names) produce high-confidence retrievals with only Tier A signals. Low-specificity tokens ("handles", "service", "config") produce low-confidence retrievals that require Tier B NL descriptions to disambiguate.

Verdict for BM25: Tier A is sufficient for high-specificity queries. Tier B is additive for low-specificity queries and necessary for paraphrase queries (where the query vocabulary is entirely different from the stored vocabulary).

**CO_RETRIEVED behavioral graph**
This mechanism clusters memories that co-occur in retrieval sessions: if memory A and memory B are retrieved together in 5 sessions, the graph learns they are related and boosts B when A is retrieved (and vice versa).

Load-bearing signals: session_id (to attribute co-retrievals to a session) and memory unit boundaries (to know which memories co-occurred). Both are Tier A signals — the session_id is a provenance signal available at hook time; memory unit boundaries are defined by the hook's output structure.

Verdict for CO_RETRIEVED: Tier A is fully sufficient. Behavioral clustering does not require any NL descriptions — it operates on retrieval event patterns, not memory content.

**Hypothetical rank fusion**
Any retrieval stack that combines BM25 scores with CO_RETRIEVED graph scores must fuse them. The fusion weight for BM25 depends on BM25 score quality, which depends on whether the memory contains matching tokens for the query. Tier B NL descriptions improve BM25 quality for paraphrase queries, which in turn improves fusion recall.

### Section 3: Content-Type Breakdown

#### Write / Edit on Structured Source Code (.py, .ts, .js, .tsx)

Source code is lexically rich. It contains function names, variable names, import statements, class names, config constants — all Tier A extractable.

- **BM25 coverage**: High. Queries about "how X works" where X is a module, class, or function name hit the exact tokens.
- **Paraphrase gap**: Moderate. A query about "authentication flow" when the code has `auth_middleware.py` and `validate_token()` — BM25 will match on "auth" and "token" but may miss on "flow" and "validate" depending on the query phrasing.
- **LLM necessity**: Low for identifier-heavy code. LLM adds value for: (a) complex business logic in opaque function names, (b) cross-cutting concerns that span multiple files.
- **Verdict**: C+ sufficient for ~75% of Write/Edit on source code. LLM adds value for the remaining 25% involving opaque naming or cross-reference bridging.

#### Write / Edit on Configuration and Environment Files (.json, .yml, .env, .toml)

Configuration files are Tier A gold mines. They contain explicit key-value pairs: service names, addresses, ports, versions, flags. These are the highest-value memories for infrastructure queries.

- **BM25 coverage**: Very high. "What port does Qdrant run on?" maps exactly to `QDRANT_PORT=6333`.
- **Paraphrase gap**: Low. Config queries tend to use the key name or the service name — both appear literally in the file.
- **LLM necessity**: Very low. The only case where LLM adds value is when a config value is meaningful only in context (e.g., `timeout: 30` — 30 what? An LLM can extract "Qdrant query timeout is 30 seconds").
- **Verdict**: C+ sufficient for ~90% of Write/Edit on config/env files. LLM adds marginal value only for unit-disambiguation.

#### Bash Output

Bash output is the most heterogeneous content class. It includes:
- Command output with structured patterns: `docker ps`, `git log`, `npm audit` — parseable with known schemas
- Error messages: structured (`Error: ECONNREFUSED 127.0.0.1:6333`) to unstructured (`subprocess returned exit code 1`)
- Script execution results: raw stdout from arbitrary commands
- Installation/build output: package lists, compilation warnings, stack traces

**Structured Bash output (docker ps, git status, process lists)**
C+ can extract service names, container IDs, ports, image names, git branch names, commit hashes. High Tier A coverage.

**Error messages**
Mixed. Error codes and connection strings are Tier A. The *meaning* of an error ("this means the Qdrant container isn't running") is Tier B.

**Unstructured script output**
Low Tier A coverage. A script that prints "Backup completed successfully — 14,832 vectors migrated" contains useful facts (14,832, "vectors", "backup", "migrated") that are partially Tier A, but the significance is Tier B.

**Verdict for Bash**: Tier A sufficient for ~50% (structured command output). LLM extraction provides non-redundant value for ~50% (unstructured output, error interpretation, script results with implicit significance).

### Section 4: Verdict on LLM Extraction Necessity

**The non-redundant signal that only LLM extraction can produce**: a natural-language description of what the code or output *means*, in the vocabulary that a future query is likely to use.

This signal is non-redundant because:
1. It cannot be produced by any deterministic text processing
2. It fills the vocabulary gap that BM25 cannot bridge
3. It is not subsumed by CO_RETRIEVED (which operates on retrieval patterns, not content meaning)

**Where LLM extraction is necessary**:
- Opaque function names and variable names where the identifier is not self-describing
- Bash output from unstructured scripts and error messages
- Cross-reference contexts where a file *refers to* a service/component without naming it explicitly
- Any content where the query vocabulary and the stored token vocabulary are non-overlapping

**Where LLM extraction is redundant (C+ is sufficient)**:
- Config and env files with explicit key-value pairs
- Source code with self-describing identifiers (most professional code)
- Structured Bash output (docker, git, npm with known output schemas)
- Numeric patterns (ports, IPs, versions) — Tier A captures these with higher precision than LLM

**Overall frequency estimate**:
- Write/Edit on source code: ~70% C+ sufficient, ~30% LLM adds non-redundant value
- Write/Edit on config/env: ~90% C+ sufficient, ~10% LLM adds marginal value
- Bash output: ~50% C+ sufficient, ~50% LLM adds non-redundant value
- Blended across typical observe-edit event distribution (Write/Edit ~75%, Bash ~25%): ~75% C+ sufficient, ~25% LLM extraction provides non-redundant value

**Verdict**: PROMISING. C+ lightweight extraction is sufficient as the primary mechanism. LLM extraction is conditionally necessary — required specifically for: (a) unstructured Bash output, (b) opaque identifiers, (c) paraphrase bridging. A tiered hook architecture that routes content to LLM extraction only when structural signals indicate low Tier A coverage would capture most of the LLM extraction value at a fraction of the latency and cost of running LLM extraction on every event.

## Ideas Extracted

### Idea: tiered-extraction-routing

- **Novelty**: 0.80 — No known observe-edit hook implementation conditionally routes to LLM extraction based on structural signals from the tool event. Current implementations either always call LLM (high cost, uniform quality) or never call LLM (low cost, poor coverage for unstructured content). The routing criterion — using file extension, tool type, and identifier density as a proxy for Tier A sufficiency — is a novel design.
- **Evidence**: 0.85 — The signal taxonomy analysis above is derivable from first principles and consistent with known BM25 behavior. The content-type breakdown percentages (75%/90%/50% for code/config/bash) are estimates, but the directional claim (config needs LLM least, Bash needs it most) is strongly supported by the structural argument.
- **Feasibility**: 0.90 — Implementable as a pure routing function with no new infrastructure. The routing decision requires: (1) file extension from tool metadata (O(1) lookup), (2) identifier density from a pass over the content (count regex matches for function/variable/config key patterns, divide by line count — O(n) in content length, fast), (3) tool type from hook event type. If identifier density > threshold and file extension in {.py, .ts, .js, .json, .yml, .env}, skip LLM extraction. Otherwise, call LLM extraction. Expected LLM call reduction: 65-75% of events.
- **Description**: Add a routing function to the observe-edit hook that classifies incoming tool events as "Tier A sufficient" vs. "LLM extraction needed" before deciding whether to call the LLM. Classification criteria: file extension is config/source type AND identifier density (regex matches per line for function names, config keys, numeric patterns) exceeds 0.3 patterns/line → Tier A sufficient. Tool type is Bash OR identifier density < 0.1 → LLM extraction needed. Mixed signals → LLM extraction (conservative fallback). The routing function adds ~2ms to hook latency; LLM extraction avoided for ~70% of events reduces average hook latency from ~800ms to ~240ms while preserving retrieval quality for the 30% of events where LLM extraction adds non-redundant value.

### Idea: explicit-anchor-injection

- **Novelty**: 0.75 — The specific mechanism of injecting a structured "anchor block" of high-specificity Tier A tokens alongside the NL description is not a standard pattern in memory hook designs. Standard designs produce either a flat text blob or a structured JSON object; the anchor block design is a deliberate optimization for BM25 recall that mirrors how document retrieval engineers add keyword metadata fields to search indexes.
- **Evidence**: 0.80 — BM25 retrieval quality is well-established to be highly sensitive to token presence in indexed fields. Adding a high-weight field containing exact-match tokens (file paths, port numbers, service names) is a standard technique in search engineering. Applying this to memory storage is a direct transfer.
- **Feasibility**: 0.85 — The anchor block is produced entirely by C+ processing. Format: a structured field in the stored memory containing a deduplicated list of high-specificity tokens extracted by regex: `{anchors: ["port:8200", "host:100.70.195.84", "service:qdrant", "path:app/core/config.py", "fn:get_settings"]}`. BM25 indexes this field with boosted weight. No LLM call required to produce it. The anchor block supplements rather than replaces the NL description field (which may be LLM-produced or absent depending on routing).
- **Description**: The observe-edit hook always produces a structured anchor block alongside whatever NL content it generates (LLM-produced description or raw text excerpt). The anchor block contains: all file paths from the tool call, all `PORT=`, `HOST=`, `IP=` patterns extracted by regex, all function/class names from source code (regex on `def `, `class `, `function `, `const `, `interface `), all service name matches from a small fixed vocabulary (qdrant, neo4j, redis, ollama, fastapi, etc.), all numeric patterns matching port range or IP format. BM25 is configured to weight anchor block matches 3× higher than NL description matches. This ensures that high-specificity infrastructure queries ("what port does X run on", "where is Y deployed") hit with high precision even when the NL description is absent or low-quality.

### Idea: bash-output-schema-registry

- **Novelty**: 0.70 — Schema-based parsing of known CLI tool outputs (docker, git, npm) is a standard technique in log processing and observability tooling. Applying it to a memory hook to improve Tier A extraction quality for Bash events is a direct transfer, but the specific application to a memory system hook is novel.
- **Evidence**: 0.75 — The claim that structured Bash output (docker ps, git log, npm audit) can be parsed with known schemas is well-supported — these tools have stable, documented output formats. The value claim (C+ extraction from structured Bash output is as good as LLM extraction for retrieval purposes) follows from the observation that structured output contains explicit Tier A tokens in known positions.
- **Feasibility**: 0.80 — A registry of ~10-15 common CLI output schemas (docker ps, docker logs, git status, git log, npm install, npm audit, pip install, curl responses, python test output) covers the majority of Bash events in a development workflow. Each schema is a small parser: regex + field extraction → structured anchor block. The hook checks the first line of Bash output against a command-name prefix (extracted from the Bash tool's command argument) to select the schema. Unrecognized commands fall back to generic regex extraction or LLM routing. Registry maintenance cost is low — schemas are stable and rarely change.
- **Description**: Add a schema registry to the observe-edit hook for Bash output events. The registry maps command prefixes (`docker ps`, `git log`, `npm audit`) to extraction functions that produce structured anchor blocks from known output formats. Example: `docker ps` output parser extracts container names, image names, exposed ports, and status; `git log` parser extracts commit hashes, author names, and commit messages (high-value retrieval targets); `npm audit` parser extracts package names, vulnerability severities, and CVE identifiers. When a Bash event's command matches a registry entry, the hook uses the schema parser instead of LLM extraction to produce the anchor block. This eliminates LLM extraction for the most common structured Bash events while maintaining high Tier A coverage.

## Suggested Follow-ups

- **[PHYSICS] anchor block weight calibration**: What BM25 field weight ratio between anchor block tokens and NL description tokens maximizes retrieval recall without over-indexing on exact-match queries? The tiered-extraction-routing idea assumes a 3× boost for anchor block fields, but this is uncalibrated. A falsifiable experiment: vary the weight ratio from 1× to 10× against a fixed set of held-out query pairs drawn from real Recall session logs, measure recall@5 at each weight. Expected result: optimum is between 2× and 5×.

- **[TABOO] inverse question — query-side paraphrase expansion**: This finding establishes that LLM extraction adds non-redundant value on the *store* side for ~25% of events. An inverse question: can the same paraphrase bridging be achieved on the *query* side instead? A query expansion step that generates 3-5 synonym queries per user prompt would allow C+-only storage while recovering the paraphrase recall gap at query time. Tradeoff: query-side expansion increases retrieval latency for every query vs. store-side LLM extraction increases storage latency for ~25% of events. Which is preferable depends on the relative frequency of writes vs. reads in Recall's usage pattern.

- **[ABSENCE] bash output schema coverage audit**: What fraction of Bash events in an actual Recall development session match a known CLI output schema? The bash-output-schema-registry idea's value depends on this coverage rate. Falsifiable: instrument observe-edit in the Recall development environment for 2 weeks, log the first token of every Bash command argument, compute the distribution. If the top 10 commands by frequency are all in the proposed schema registry, coverage is likely >70%.
