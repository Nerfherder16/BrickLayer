# Q126 — Write-Path Latency Roofline: Three-Store Stack After Neo4j Elimination

**Question**: What is the theoretical minimum write-path latency for the three-store stack after Neo4j elimination — covering the parallel write pipeline (PG INSERT + Qdrant upsert + Redis INCR) and the session-end co_retrievals batch UPSERT — and does the new write-path p95 allow a synchronous co_retrievals INSERT per retrieve call without violating the 500ms retrieval SLA?
**Question Type**: PHYSICS
**Verdict**: BREAKTHROUGH
**Quality Score**: N=0.72 E=0.68 F=0.90

---

## Finding

### Operation Definitions

**Write path (Op1 — memory store):**
- Input: content blob (~200 bytes), domain, tags, importance score, tenant_id; embedding vector (6,144 bytes) pre-generated at Ollama
- Output: PG row inserted (memories table), Qdrant point upserted, Redis INCR on epoch counter
- Guarantees: all three stores updated; eventual consistency acceptable (no cross-store transaction); partial failure must be detectable via S4 epoch divergence signal

**Write path (Op2 — co_retrievals reinforcement per retrieve call):**
- Input: pair of memory UUIDs (from_id, to_id), tenant_id, current epoch
- Output: single UPSERT to co_retrievals table (Q115 schema)
- Guarantees: atomic UPSERT; no read-modify-write race (PostgreSQL ON CONFLICT handles this natively)
- Placement question: hot path (synchronous within retrieve handler) vs. cold path (queued, session-end batch)

**Retrieval path SLA:** 500ms p95. This is the governing constraint that determines whether synchronous co_retrievals INSERTs are viable.

### Theoretical Minimum: Parallel Write Pipeline (PG || Qdrant || Redis)

#### PostgreSQL INSERT (memories table)

**Operation**: INSERT of ~1 row (~200 bytes payload + UUID + tags array + 6,144-byte embedding vector if stored in PG, or just metadata if embedding lives in Qdrant only).

**Physics floor:**
```
Minimum bytes written: ~400 bytes (row header + columns, no embedding)
Kernel buffer write: 400 bytes / 30 GB/s = 0.013 µs
WAL record: ~200 bytes → 200 / 30 GB/s = 0.007 µs
Socket roundtrip (loopback): ~2 × 50 µs = 0.10 ms (kernel TCP stack minimum per Docker socket)
```

**Physics minimum**: ~0.10 ms — dominated by the loopback socket roundtrip, not the data write.

**Implementation cost (P50 warm, synchronous_commit=off):**
- asyncpg connection pool (persistent connection, no TCP setup): ~0.0ms connect
- PG executor + query parse + row insert: ~0.20 ms
- WAL buffer write (synchronous_commit=off: WAL record buffered, no fsync): ~0.01 ms
- shared_buffers dirty page write (no disk I/O): ~0.01 ms
- Docker socket + asyncpg protocol overhead: ~0.10 ms
- **P50 warm: ~0.32 ms**
- **P95 warm: ~0.80 ms** (PG autovacuum contention, connection pool wait)
- **P95 cold (connection setup): ~2.0 ms** (rare with persistent pool)

Note on `synchronous_commit`: Default PostgreSQL is `synchronous_commit=on`, which means WAL records are written to disk (fsync) before returning. On an NVMe with 0.05ms random write latency, this adds ~0.05ms per INSERT. With `synchronous_commit=off`, WAL is buffered; no fsync on write path. For a self-hosted memory system where durability within a session matters but per-record ACID is not critical, `synchronous_commit=off` is the appropriate setting. At `synchronous_commit=off`, P95 warm stays at ~0.80ms.

**Gap ratio**: 0.32 ms / 0.10 ms = **3.2×** — gap is PG executor overhead (query planning, row formatting, MVCC bookkeeping). Much smaller than Q107's Phase 1 read gap (640×) because writes have lower query planning overhead than GIN index scans.

#### Qdrant Upsert (recall_tim collection)

**Operation**: Upsert 1 point — 1536-dim float32 vector (6,144 bytes) + payload (~200 bytes metadata) into in-memory HNSW graph.

**Physics floor:**
```
Vector bytes: 6,144 bytes
HNSW insert at N=360: O(log N) graph links = log2(360) ≈ 9 hops
  Each hop: read neighbor vector (6,144 bytes) + compare = 9 × 6,144 = 55,296 bytes
  At 30 GB/s: 55,296 / 30e9 × 1000 = 0.0018 ms
gRPC/REST loopback roundtrip: ~0.10 ms
```

**Physics minimum**: ~0.10 ms — dominated by loopback socket, not the HNSW graph mutation.

**Implementation cost (P50 warm):**
- qdrant-client Python → gRPC → Qdrant Rust process: ~0.10 ms IPC overhead
- HNSW graph insert at N=360 (trivial — effectively O(1) at this scale): ~0.02 ms
- Payload serialization (msgpack/JSON): ~0.01 ms
- **P50 warm: ~1.5 ms** (Qdrant upsert benchmarks at sub-1000 point collections: 0.5–3 ms; single-point upsert is dominated by IPC and HTTP/gRPC stack overhead)
- **P95 warm: ~5 ms** (gRPC connection contention, HNSW lock contention during concurrent writes)

Note: Q107 established Qdrant read P50 at ~1.0 ms. Upsert (write) is heavier than read in Qdrant due to HNSW graph mutation (link insertion and rebalancing) vs. HNSW traversal (read-only). The upsert P50 is conservatively 1.5× the read P50.

**Gap ratio**: 1.5 ms / 0.10 ms = **15×** — gap is IPC overhead (Python → Qdrant process) plus HNSW lock acquisition. Smaller than read gap (49×) because write path has less result serialization.

#### Redis INCR (epoch counter)

**Operation**: `INCR recall:tim:global_epoch` — atomic increment of a single integer key.

**Physics floor:**
```
Data: 8 bytes (int64 increment)
Redis in-memory O(1): 5 ns
Loopback socket roundtrip: ~0.10 ms
```

**Physics minimum**: ~0.10 ms — entirely socket roundtrip.

**Implementation cost (P50):**
- redis-py async client, persistent connection: ~0.01 ms connect overhead
- Redis server processing (O(1), in-memory): ~0.005 ms
- Docker socket + protocol overhead: ~0.10 ms
- **P50: ~0.12 ms**
- **P95: ~0.30 ms** (Docker bridge jitter, Redis keyspace notifications if enabled)

**Gap ratio**: 0.12 ms / 0.10 ms = **1.2×** — nearly at physics floor. Redis INCR is the closest-to-optimal operation in the entire stack.

### Parallel Write Pipeline: Combined Cost

Op1 (memory store) runs all three writes in parallel:

```
t=0: Launch PG INSERT || Qdrant UPSERT || Redis INCR
     PG INSERT completes at t≈0.32ms (P50) / t≈0.80ms (P95)
     Redis INCR completes at t≈0.12ms (P50) / t≈0.30ms (P95)
     Qdrant UPSERT completes at t≈1.50ms (P50) / t≈5.00ms (P95)

t = max(PG, Qdrant, Redis) = Qdrant dominates
```

| Scenario | PG | Redis | Qdrant | Pipeline total (parallel) |
|---|---|---|---|---|
| Theoretical minimum | 0.10 ms | 0.10 ms | 0.10 ms | **0.10 ms** |
| P50 warm | 0.32 ms | 0.12 ms | 1.50 ms | **1.50 ms** |
| P95 warm | 0.80 ms | 0.30 ms | 5.00 ms | **5.00 ms** |
| P99 (worst case) | 2.00 ms | 0.50 ms | 10.0 ms | **10.0 ms** |

**The parallel write pipeline is Qdrant-bound.** PG and Redis complete well before Qdrant. The physics minimum for the full parallel pipeline is 0.10 ms (loopback socket RTT). P50 warm is ~1.5 ms. P95 warm is ~5 ms.

**Gap ratio (P50)**: 1.50 ms / 0.10 ms = **15×** — write path is much closer to the physics floor than the retrieval path (which was 150–226× in Q107). Write paths have lower overhead because they involve fewer result rows to serialize and no query planning for index scans.

### co_retrievals UPSERT: Isolated Cost

**Operation** (Q115 schema): `INSERT INTO co_retrievals (...) ON CONFLICT (tenant_id, from_memory_id, to_memory_id) DO UPDATE SET ...`

This is a PostgreSQL write. Physics floor and implementation cost follow the same model as the PG INSERT above, with one difference: the UPSERT requires reading the existing row on conflict (to increment `weight` and `co_retrieval_count`), which adds one heap page read.

**Physics floor:**
```
Write: same as INSERT = ~0.10 ms (loopback dominates)
Conflict check: index probe on PRIMARY KEY (tenant_id, from_memory_id, to_memory_id)
  = 2–3 B-tree levels × 8 KB pages (warm in shared_buffers) = negligible
Read-modify-write (conflict path): ~10 µs additional (warm shared_buffers)
Total: ~0.10 ms — still loopback-dominated
```

**Implementation cost:**
- asyncpg UPSERT: same as INSERT + 1 index probe + 1 heap read + 1 heap write
- Warm (shared_buffers): **P50 ~0.40 ms** (slightly higher than INSERT due to conflict path read)
- **P95 warm: ~1.0 ms**
- Cold (table not in shared_buffers, first access after restart): ~3.0 ms (NVMe random read for co_retrievals heap pages)

At Month 36 scale, co_retrievals table is < 3 MB (Q115: ~22,000 edges × ~130 bytes/row). This fits entirely in PostgreSQL shared_buffers even at the default 128 MB setting. The table will always be warm after the first post-restart access. The cold-path cost is a one-time startup penalty, not a recurring concern.

### Revised Retrieval Pipeline After Neo4j Elimination

Q107 established the five-phase retrieval pipeline with Neo4j as Phase 4 (2.20 ms, architecture gap). After Neo4j elimination (Q108/Q115), Phase 4 becomes the PostgreSQL BFS CTE.

Q115 estimates the PostgreSQL S6-GAUGE query at < 5 ms for Month 36 scale. The BFS CTE itself (Q108 latency comparison table):

- **Depth-2 BFS at K=38 seeds, Month 36 (~22,000 edges, FRESH fraction ~25%)**: estimated ~0.60 ms warm (partial index scan on idx_co_ret_tim_fresh, ~5,500 FRESH rows, K=38 seeds × avg 2–3 fresh neighbors = ~100 rows traversed at depth-1, ~200 at depth-2 with LIMIT 50 hub guard).

**Revised retrieval pipeline critical path (warm, embedding pre-cached per Q116):**

### SLA Budget Analysis: Synchronous co_retrievals INSERT on the Retrieve Hot Path

**The design question from Q115**: co_retrievals reinforcement happens on each retrieve call. Should this UPSERT be:
- (A) Synchronous — awaited inside the retrieve handler before response is returned
- (B) Asynchronous — fire-and-forget task, not on the response critical path
- (C) Batched — accumulated per session, flushed at session end

**Budget calculation for option (A) — synchronous:**

```
Three-store retrieval P95 (embedding generated):
  P95 = P50 × ~2.5× (typical tail ratio for mixed IPC workloads)
  = 3.38 ms × 2.5 = ~8.5 ms

co_retrievals UPSERT P95 warm: ~1.0 ms
co_retrievals UPSERT P95 cold: ~3.0 ms (first access after restart, extremely rare)

Total retrieval + synchronous UPSERT:
  P50: 3.38 ms + 0.40 ms = 3.78 ms
  P95: ~8.5 ms + 1.0 ms = ~9.5 ms
  P99: ~15 ms + 3.0 ms = ~18 ms (extreme tail, cold path)
```

**SLA headroom:**
```
500 ms SLA - 9.5 ms P95 = 490.5 ms headroom
500 ms SLA / 9.5 ms P95 = 52.6× below SLA
```

**Verdict on synchronous co_retrievals INSERT**: Viable with enormous margin. The synchronous UPSERT adds 0.40 ms (P50) to 1.0 ms (P95) to the retrieval latency. This does not approach the 500ms SLA at any realistic operating condition. Even a 10× latency spike (PostgreSQL under heavy autovacuum contention, table lock on co_retrievals) would produce a worst-case ~10 ms addition — still 49× below the 500ms SLA.

**However, option (A) is not the optimal design for a different reason:**

The co_retrievals UPSERT adds write amplification to the retrieve handler. Each retrieve call that returns K=38 memories generates K×(K-1)/2 = 703 pair-wise co_retrieval records to update (all pairs among the K returned memories co-occurred in the same retrieval). Updating 703 rows synchronously changes the cost from 0.40 ms (single UPSERT) to potentially:

```
703 UPSERTs individually: 703 × 0.40 ms = 281 ms → VIOLATES 500ms SLA
703 UPSERTs as batch INSERT...ON CONFLICT: ~5–15 ms → SLA-safe
```

**The critical design constraint is not latency per UPSERT but cardinality of the batch.** If co_retrievals is updated for all K×(K-1)/2 pairs on each retrieval (the semantically correct approach — all K returned memories co-occurred), the batch size is 703 rows. A single batched multi-row UPSERT for 703 rows is estimated at:

```
Batch UPSERT (703 rows):
  PG parse + plan: ~0.20 ms (one-time)
  Row writes: 703 × ~0.005 ms (heap write per row, warm shared_buffers) = ~3.5 ms
  Index updates on PRIMARY KEY: 703 × ~0.002 ms = ~1.4 ms
  Total: ~5.1 ms P50
  P95: ~12 ms (PG write amplification under autovacuum pressure at large batch)
```

**Revised SLA budget with full co_retrieval batch:**

```
Retrieval P95: ~8.5 ms
co_retrievals batch UPSERT P95: ~12 ms
Total: ~20.5 ms P95

500 ms SLA / 20.5 ms = 24.4× below SLA
```

Still viable. But the design choice matters: 703-row synchronous batch UPSERT adds ~5–12 ms to the retrieve handler, consuming 2–4% of the SLA budget. This is acceptable but non-trivial.

**Option (B) — fire-and-forget async task** is the preferred design:

```python
# In retrieve handler:
asyncio.ensure_future(update_co_retrievals(returned_memory_ids, current_epoch))
# Returns immediately; co_retrievals update runs in background event loop
```

This moves the entire co_retrievals write cost off the retrieval critical path. The retrieval response returns in ~3.4 ms P50. The background task completes in ~5–12 ms concurrently. No SLA impact, no latency addition to retrieval, and no functional difference because co_retrievals is a reinforcement signal (eventual consistency is correct for this use case — a retrieval that completes without updating co_retrievals is not semantically broken, it simply doesn't reinforce the co_occurrence signal for that pair).

### Gap Analysis

**Physics gaps (unavoidable):**

- **Docker loopback socket overhead (~0.10 ms per call)**: All three stores are in separate Docker containers. Even on the same host, Docker bridge network adds ~0.10 ms per round-trip vs. ~0.005 ms for a Unix socket call to an in-process library. This is the physics floor for IPC-bound operations. Cannot be reduced without co-locating the store into the FastAPI process (changes the architecture entirely).
- **HNSW graph mutation (Qdrant upsert)**: Inserting a vector into an in-memory HNSW graph requires log(N) random memory accesses for graph link updates. At N=360, this is ~9 hops × 6,144 bytes each = ~55 KB touched. This is the true computational minimum for an HNSW insert and cannot be reduced without using a simpler (less accurate) index structure.

**Architecture gaps (design choices — removable):**

- **Qdrant IPC overhead (~15× gap, ~1.4 ms excess)**: Qdrant runs as a separate process. At N=360, the HNSW insert physics takes ~0.01 ms; IPC adds ~1.4 ms. An in-process write-through to a Python-native index (hnswlib or numpy matrix) would eliminate this gap for the write path. As established in Q107 for the read path, the Qdrant process boundary costs 15–49× the physics floor at this scale. For the write path, same architecture gap applies.
- **Batch UPSERT cardinality (703 rows per retrieve call, if synchronous)**: The semantic choice to record all K×(K-1)/2 co_retrieval pairs is correct but creates a large synchronous write batch if placed on the hot path. Option (B) fire-and-forget is a zero-cost architectural fix that removes the batch from the critical path without changing the semantics.
- **synchronous_commit=on (PG default)**: With default settings, each PG INSERT flushes WAL to NVMe, adding ~0.05–0.10 ms. Setting `synchronous_commit=off` at the session or table level eliminates this at the cost of up-to-`wal_writer_delay` (200ms default) data loss window on crash — acceptable for a personal memory system that reconstructs from session content.

**Implementation gaps:**

- **Python asyncio overhead for fire-and-forget tasks**: Using `asyncio.ensure_future()` adds ~1 µs task scheduling overhead per retrieval call. Negligible.
- **PG query plan caching**: The co_retrievals UPSERT query plan can be cached via asyncpg's `prepare()` mechanism, reducing per-call planning overhead from ~0.05ms to ~0.005ms. A one-line code change.

## Qualifying Ideas

### Idea: co-retrievals-fire-and-forget-async
- **Novelty**: 0.72
- **Evidence**: 0.75
- **Feasibility**: 0.95
- **Description**: Move the co_retrievals batch UPSERT from the retrieve handler's synchronous response path to a fire-and-forget asyncio task dispatched immediately after the retrieval results are assembled. The handler returns the K=38 memories without awaiting the co_retrievals write. The background task runs in the same asyncio event loop and completes within 5–12 ms (P95) concurrently with the client parsing the response. This eliminates 5–12 ms from the retrieval critical path — reducing P95 from ~20.5 ms to ~8.5 ms — without any change to the semantic correctness of the co_retrieval signal. The key insight: co_retrievals is a reinforcement signal for future retrieval ranking, not a guarantee required by the current retrieval response. Eventual consistency (writes complete within ~12 ms of retrieval) is the correct consistency model for this use case. Implementation is a one-line change to the retrieve handler: `asyncio.ensure_future(batch_upsert_co_retrievals(...))`.

### Idea: write-path-qdrant-ipc-bypass-at-small-scale
- **Novelty**: 0.68
- **Evidence**: 0.72
- **Feasibility**: 0.82
- **Description**: The Qdrant upsert IPC gap (15×, ~1.4 ms excess over physics minimum) can be eliminated by maintaining a Python-native in-process HNSW write buffer alongside Qdrant. On each vector store, the write goes to both Qdrant (durable, authoritative, async) and an in-process hnswlib index (ephemeral, for fast reads). This is the write-path analog of Q107's `in-process-qdrant-replacement-at-360-points` idea for the read path. The combined effect (read + write bypass) would reduce Qdrant's contribution to both the read critical path (1.0 ms → 0.074 ms) and write critical path (1.5 ms → 0.02 ms). The parallel write pipeline floor drops from 1.5 ms to 0.32 ms (PG-bound). Feasibility is 0.82 because it requires managing two write targets (Qdrant for durability, hnswlib for latency) and a startup warm-up load to populate hnswlib from Qdrant on FastAPI restart.

### Idea: synchronous-commit-off-for-co-retrievals-table
- **Novelty**: 0.70
- **Evidence**: 0.80
- **Feasibility**: 0.95
- **Description**: Apply `ALTER TABLE co_retrievals SET (fillfactor=70)` and set `synchronous_commit=off` at the session level for co_retrievals writes specifically. The co_retrievals table is a reinforcement signal — it can tolerate up to 200ms of data loss on crash (the wal_writer_delay window) because it is reconstructible from session history and the loss affects only the weight/count increment on the affected row, not the existence of the edge. Setting `synchronous_commit=off` per-transaction for co_retrievals UPSERTs reduces P95 from ~1.0 ms to ~0.60 ms (eliminates WAL fsync). The `fillfactor=70` leaves 30% of each heap page free for UPDATE-in-place on HOT updates (Heap Only Tuple), reducing write amplification when `co_retrieval_count` and `weight` columns are updated frequently on existing rows. Together these reduce co_retrievals P95 from ~1.0 ms to ~0.50 ms. Evidence score reflects well-documented PostgreSQL performance tuning for append-heavy and update-heavy tables.

## Recommendations

**1. Place co_retrievals UPSERT as a fire-and-forget async task** (option B, not A or C). The SLA budget allows either synchronous or async, but async eliminates the latency addition entirely at zero semantic cost. 703-row synchronous batch adds 5–12 ms unnecessarily.

**2. The parallel write pipeline (PG || Qdrant || Redis) has P95 = ~5 ms.** This is the correct engineering estimate for the three-store write path after Neo4j elimination. The theoretical minimum is 0.10 ms (loopback floor); actual P95 is 50× above that floor, entirely due to Qdrant IPC overhead. No optimization is required for Tim's use case (5 sessions/day).

**3. The 500ms retrieval SLA has 24–52× headroom after the three-store collapse.** The SLA is not threatened by adding synchronous co_retrievals writes to the retrieve handler. The practical constraint is not the SLA but write amplification: avoid 703 individual UPSERTs; use a single multi-row batch statement or fire-and-forget async dispatch.

**4. Session-end batch flush (option C) is the current Q115 design.** This is correct as a baseline. Upgrading to per-retrieve fire-and-forget async (option B) provides real-time reinforcement without the 5–12 ms hot-path cost. At Tim's session frequency (5/day), the difference is imperceptible in practice — but fire-and-forget is architecturally cleaner because it keeps the retrieve handler latency deterministic regardless of batch size.

**5. Qdrant remains the write-path bottleneck.** P95 Qdrant upsert (~5 ms) dominates the parallel pipeline. At Tim's write rate (1–5 stores/session), this is irrelevant. If write throughput ever becomes a concern, the in-process hnswlib write bypass (idea above) reduces the write bottleneck from 5 ms to 0.32 ms.
