### Key Architectural Constraints from Codebase Inspection

Before giving the spec, three constraints from the real files:

1. **PostgreSQL client is asyncpg** (raw, no ORM). `postgres_store.py` uses `asyncpg.create_pool()` and executes DDL via `_CREATE_TABLES` string run at startup. The outbox DDL must follow this pattern — it cannot use SQLAlchemy or Alembic syntax.

2. **Lifespan pattern is `@asynccontextmanager`** in `src/api/main.py:56`. Background tasks must be spawned as `asyncio.create_task()` inside the `yield` block and cancelled in the `finally` block. The lifespan already manages storage singletons; the outbox worker is another singleton task.

3. **Fire-and-forget style is the established convention**. Every section of `search.py` and `memory.py` wraps non-critical writes in `try/except: pass`. The outbox worker must not propagate exceptions to the caller.

### Exact DDL: co_retrievals_outbox Table

Insert this into `_CREATE_TABLES` in `src/storage/postgres_store.py` after the `membench_runs` table block (after line ~162):

```sql
CREATE TABLE IF NOT EXISTS co_retrievals_outbox (
    id              BIGSERIAL PRIMARY KEY,
    from_memory_id  UUID        NOT NULL,
    to_memory_id    UUID        NOT NULL,
    tenant_id       VARCHAR(32) NOT NULL DEFAULT 'tim',
    epoch           INTEGER,                        -- NULL if Redis INCR failed during write
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    processed_at    TIMESTAMPTZ,                    -- NULL = pending, set on success
    attempt_count   INTEGER     NOT NULL DEFAULT 0,
    last_attempt_at TIMESTAMPTZ,
    error_msg       TEXT                            -- last failure reason, for diagnostics
);
CREATE INDEX IF NOT EXISTS idx_co_outbox_pending
    ON co_retrievals_outbox (created_at)
    WHERE processed_at IS NULL;

CREATE TABLE IF NOT EXISTS co_retrievals_dead_letter (
    id              BIGSERIAL PRIMARY KEY,
    outbox_id       BIGINT      NOT NULL,
    from_memory_id  UUID        NOT NULL,
    to_memory_id    UUID        NOT NULL,
    tenant_id       VARCHAR(32) NOT NULL,
    epoch           INTEGER,
    created_at      TIMESTAMPTZ NOT NULL,
    failed_at       TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    final_error     TEXT
);
```

**Why `epoch` is nullable**: Q129 clarifies that the Redis INCR may fail during the Q112 migration window. If epoch is unknown, the outbox records the pair with `epoch = NULL`. The flush worker's UPSERT handles `NULL` via `COALESCE(epoch, last_reinforced_epoch, 0)` — it reinforces the edge without overwriting a valid epoch if one already exists.

**Why a partial index on `processed_at IS NULL`**: The flush worker polls unprocessed entries every 5 seconds. The partial index ensures the poll is an index-only scan against a near-empty set (only backlogged entries). At Recall's write rate, this set is almost always empty — the scan cost is negligible.

### Exact Python: Write Path (Atomic Outbox + co_retrievals UPSERT)

This is the function that fires after each `/browse` or `/query` call returns results. It runs as a fire-and-forget `asyncio.create_task()` in the retrieve handler.

**File**: `src/storage/postgres_store.py` — add as a new method on `PostgresStore`:

```python
async def store_co_retrievals(
    self,
    pairs: list[tuple[str, str]],  # list of (from_memory_id, to_memory_id)
    tenant_id: str,
    epoch: int | None,
) -> None:
    """
    Write co_retrieval pairs atomically: outbox entry + UPSERT in same transaction.

    If the transaction commits, both the outbox record and the co_retrievals row
    are durable. If it rolls back, neither write occurred — clean failure, no
    silent signal loss.

    The outbox entry is marked processed_at=NOW() inside this same transaction,
    so the background flush worker will find nothing to retry for successful writes.
    The outbox's value is for the failure path: if this transaction fails after
    partial execution, the flush worker sees unprocessed entries on next poll.

    NOTE: This is intentionally NOT called by the flush worker — the flush worker
    has its own retry path. This method is the hot-path write.
    """
    if not pairs or not self.pool:
        return

    async with self.pool.acquire() as conn:
        async with conn.transaction():
            for from_id, to_id in pairs:
                # Step 1: Insert outbox entry (durable intent record)
                outbox_id = await conn.fetchval(
                    """
                    INSERT INTO co_retrievals_outbox
                        (from_memory_id, to_memory_id, tenant_id, epoch)
                    VALUES ($1::uuid, $2::uuid, $3, $4)
                    RETURNING id
                    """,
                    from_id,
                    to_id,
                    tenant_id,
                    epoch,
                )

                # Step 2: Attempt the co_retrievals UPSERT
                await conn.execute(
                    """
                    INSERT INTO co_retrievals
                        (from_memory_id, to_memory_id, tenant_id,
                         relationship_type, last_reinforced_epoch,
                         weight, co_retrieval_count)
                    VALUES ($1::uuid, $2::uuid, $3, 'FRESH',
                            COALESCE($4, 0), 1.0, 1)
                    ON CONFLICT (tenant_id, from_memory_id, to_memory_id) DO UPDATE
                        SET relationship_type    = 'FRESH',
                            last_reinforced_epoch = CASE
                                WHEN $4 IS NOT NULL
                                THEN GREATEST(co_retrievals.last_reinforced_epoch, $4)
                                ELSE co_retrievals.last_reinforced_epoch
                            END,
                            weight              = co_retrievals.weight + 0.1,
                            co_retrieval_count  = co_retrievals.co_retrieval_count + 1
                    """,
                    from_id,
                    to_id,
                    tenant_id,
                    epoch,
                )

                # Step 3: Mark outbox entry processed (same transaction)
                # If we reach here, the UPSERT succeeded. Mark it done.
                # If the transaction rolls back, processed_at stays NULL
                # and the flush worker will retry.
                await conn.execute(
                    """
                    UPDATE co_retrievals_outbox
                    SET processed_at = NOW()
                    WHERE id = $1
                    """,
                    outbox_id,
                )
```

**Idempotency guarantee**: The UPSERT's `ON CONFLICT (tenant_id, from_memory_id, to_memory_id)` clause is the idempotency key. If the flush worker retries a pair that was already processed (double-flush due to a crash between the PostgreSQL COMMIT acknowledgment and the outbox DELETE), the UPSERT increments `co_retrieval_count` a second time. This is at-least-once semantics. For Recall's reinforcement signal, this is correct: a double-flush produces a count of 2 instead of 1 for that pair — a minor over-reinforcement, not a correctness failure. The signal remains monotonically increasing.

To achieve exactly-once, add `outbox_id BIGINT` to `co_retrievals` and check `WHERE NOT EXISTS (SELECT 1 FROM co_retrievals WHERE outbox_id = $outbox_id)` before the UPSERT. At Recall's write rate (3–15 pairs/day), the double-flush probability is near zero; exactly-once complexity is not warranted.

### Exact FastAPI Lifespan Background Task: 5-Second Flush Worker

**File**: `src/api/main.py` — modify the `lifespan` function.

Insert the flush worker task after `logger.info("storage_connections_established")` (after line ~82) and cancel it in the `finally` block.

```python
import asyncio
import structlog

_logger = structlog.get_logger()

MAX_RETRY_ATTEMPTS = 5
FLUSH_INTERVAL_SECONDS = 5.0


async def _co_retrievals_flush_worker(postgres: "PostgresStore") -> None:
    """
    Background task: flush pending co_retrievals_outbox entries to co_retrievals.

    Polls every 5 seconds. In steady state (no failures), the outbox is always
    empty (all entries were processed inline and marked processed_at=NOW()).
    This worker only activates when an inline write failed mid-transaction.

    Retry backoff: fixed 5-second poll. No exponential backoff because:
    - At Recall's write rate (3-15 pairs/day), there is no thundering-herd risk.
    - Exponential backoff would delay recovery for hours if PostgreSQL was down
      briefly; fixed 5-second poll recovers within one poll cycle of PG restart.
    - After MAX_RETRY_ATTEMPTS (5) failures, the entry moves to dead_letter.
      Dead-letter entries are NOT retried by this worker — they require manual
      review or a separate reconciliation process.

    Survival across FastAPI restart: YES — because the outbox lives in PostgreSQL
    (not in-process memory), unprocessed entries survive container restart. The
    flush worker reads `WHERE processed_at IS NULL` on startup and immediately
    retries any entries that were pending at shutdown.
    """
    _logger.info("co_retrievals_flush_worker_started")
    while True:
        try:
            await asyncio.sleep(FLUSH_INTERVAL_SECONDS)
            if not postgres.pool:
                continue

            # Fetch pending entries (partial index makes this fast)
            async with postgres.pool.acquire() as conn:
                pending = await conn.fetch(
                    """
                    SELECT id, from_memory_id, to_memory_id, tenant_id, epoch
                    FROM   co_retrievals_outbox
                    WHERE  processed_at IS NULL
                      AND  attempt_count < $1
                    ORDER  BY created_at
                    LIMIT  100
                    """,
                    MAX_RETRY_ATTEMPTS,
                )

            if not pending:
                continue

            _logger.warning(
                "co_retrievals_flush_worker_retrying",
                count=len(pending),
            )

            for row in pending:
                outbox_id = row["id"]
                from_id = str(row["from_memory_id"])
                to_id = str(row["to_memory_id"])
                tenant_id = row["tenant_id"]
                epoch = row["epoch"]

                try:
                    async with postgres.pool.acquire() as conn:
                        async with conn.transaction():
                            await conn.execute(
                                """
                                INSERT INTO co_retrievals
                                    (from_memory_id, to_memory_id, tenant_id,
                                     relationship_type, last_reinforced_epoch,
                                     weight, co_retrieval_count)
                                VALUES ($1::uuid, $2::uuid, $3, 'FRESH',
                                        COALESCE($4, 0), 1.0, 1)
                                ON CONFLICT (tenant_id, from_memory_id, to_memory_id)
                                DO UPDATE
                                    SET relationship_type    = 'FRESH',
                                        last_reinforced_epoch = CASE
                                            WHEN $4 IS NOT NULL
                                            THEN GREATEST(
                                                co_retrievals.last_reinforced_epoch, $4)
                                            ELSE co_retrievals.last_reinforced_epoch
                                        END,
                                        weight              = co_retrievals.weight + 0.1,
                                        co_retrieval_count  =
                                            co_retrievals.co_retrieval_count + 1
                                """,
                                from_id,
                                to_id,
                                tenant_id,
                                epoch,
                            )
                            await conn.execute(
                                """
                                UPDATE co_retrievals_outbox
                                SET processed_at = NOW(),
                                    last_attempt_at = NOW(),
                                    attempt_count = attempt_count + 1
                                WHERE id = $1
                                """,
                                outbox_id,
                            )

                except Exception as upsert_err:
                    # Increment attempt_count; move to dead_letter after MAX attempts
                    _logger.error(
                        "co_retrievals_flush_worker_retry_failed",
                        outbox_id=outbox_id,
                        error=str(upsert_err),
                    )
                    async with postgres.pool.acquire() as conn:
                        async with conn.transaction():
                            await conn.execute(
                                """
                                UPDATE co_retrievals_outbox
                                SET attempt_count   = attempt_count + 1,
                                    last_attempt_at = NOW(),
                                    error_msg       = $2
                                WHERE id = $1
                                """,
                                outbox_id,
                                str(upsert_err)[:500],
                            )
                            # Move to dead_letter after MAX_RETRY_ATTEMPTS
                            exhausted = await conn.fetchval(
                                "SELECT attempt_count FROM co_retrievals_outbox WHERE id = $1",
                                outbox_id,
                            )
                            if exhausted and exhausted >= MAX_RETRY_ATTEMPTS:
                                await conn.execute(
                                    """
                                    INSERT INTO co_retrievals_dead_letter
                                        (outbox_id, from_memory_id, to_memory_id,
                                         tenant_id, epoch, created_at, final_error)
                                    SELECT id, from_memory_id, to_memory_id,
                                           tenant_id, epoch, created_at, $2
                                    FROM   co_retrievals_outbox
                                    WHERE  id = $1
                                    """,
                                    outbox_id,
                                    str(upsert_err)[:500],
                                )
                                await conn.execute(
                                    "DELETE FROM co_retrievals_outbox WHERE id = $1",
                                    outbox_id,
                                )

        except asyncio.CancelledError:
            _logger.info("co_retrievals_flush_worker_cancelled")
            return
        except Exception as loop_err:
            # Never let the worker die from an unexpected error
            _logger.error("co_retrievals_flush_worker_error", error=str(loop_err))
```

### Survival Across FastAPI Restart: YES, Unconditionally

The outbox in PostgreSQL (not in-process memory) guarantees restart survival:

1. FastAPI shuts down. The `finally` block cancels `_flush_task` via `CancelledError`.
2. The `asyncio.CancelledError` propagates to the `await asyncio.sleep(...)` call or the `await conn.execute(...)` call, whichever is in flight.
3. If cancelled during `asyncio.sleep`: no in-flight write. All pending outbox entries remain in PostgreSQL with `processed_at IS NULL`. Zero data lost.
4. If cancelled during an in-flight `conn.transaction()`: asyncpg rolls back the transaction automatically on `CancelledError`. The outbox entry remains with `processed_at IS NULL` and `attempt_count` at its pre-cancel value. Zero data lost.
5. On next startup, the flush worker polls immediately after `FLUSH_INTERVAL_SECONDS = 5.0` seconds and retries all pending entries.

**What SQLite in-process outbox would NOT survive**: An in-memory outbox (e.g., a Python `list` or `asyncio.Queue`) is lost on container restart. A SQLite file inside the container is lost unless mounted on a Docker volume. The PostgreSQL-hosted outbox inherits PostgreSQL's durability guarantees (WAL) — it survives container restarts, OOM kills, and host reboots (as long as the PG volume persists, which it does on CasaOS).

### Retry Backoff Decision: Fixed 5-Second Poll, Not Exponential

Exponential backoff would be wrong for Recall's pattern:

| Factor | Fixed 5s | Exponential (1s → 2s → 4s … 512s) |
|---|---|---|
| Recovery after brief PG restart | ≤ 5s | Up to 512s (8+ minutes) |
| Thundering-herd risk | None (3-15 pairs/day) | None |
| Dead-letter delay | 25s (5 × 5s) | 1+2+4+8+16 = 31s (similar) |
| Complexity | Trivial | Requires backoff state per entry |

At Recall's write rate, the outbox accumulates at most ~15 entries per day during normal operation (all of which are immediately processed inline and marked `processed_at`). Backlogged entries only appear on failure events — which are rare. A fixed 5-second poll with `attempt_count < 5` limit delivers the right recovery latency at zero implementation complexity.

### Integration with Retrieve Handler

In `src/api/routes/search.py`, after the `browse_memories` handler assembles `browse_results`, add:

```python
# Fire-and-forget co_retrievals reinforcement
if browse_results and body.session_id:
    try:
        from src.storage import get_postgres_store, get_redis_store

        _pg = await get_postgres_store()
        _redis = await get_redis_store()
        _epoch = None
        try:
            _epoch = await _redis.client.incr("recall:tim:global_epoch")
        except Exception:
            pass  # epoch unknown; outbox stores NULL, flush worker handles it

        # Build all K*(K-1)/2 co-retrieval pairs
        _ids = [r.id for r in browse_results]
        _pairs = [
            (_ids[i], _ids[j])
            for i in range(len(_ids))
            for j in range(i + 1, len(_ids))
        ]
        # Cap at 100 pairs to avoid degenerate large-K batches
        _pairs = _pairs[:100]

        import asyncio as _asyncio
        _asyncio.ensure_future(_pg.store_co_retrievals(_pairs, "tim", _epoch))
    except Exception:
        pass  # never block results
```

The `ensure_future` call dispatches the write to the asyncio event loop immediately. The retrieve handler returns without awaiting it. The `store_co_retrievals` method runs its transaction in the background. If it fails (PG down, connection pool exhausted), the outbox entry remains unprocessed and the flush worker retries within 5 seconds.

## Qualifying Ideas

### Idea: outbox-restart-survives-pg-wal
- (N=0.68, E=0.85, F=0.95) — The PostgreSQL-hosted outbox inherits PostgreSQL's own WAL durability, which means the outbox survives any failure short of PG volume corruption — including FastAPI container restart, OOM kill, and host reboot. This is a stronger guarantee than a SQLite outbox (which requires a mounted volume) or an in-process queue (which is always ephemeral). The proof is structural: the outbox table is committed via asyncpg's transaction protocol; PostgreSQL's WAL flush guarantees the committed state persists. The flush worker, on restart, reads `WHERE processed_at IS NULL` and recovers all pending entries without any startup handshake protocol. Implementation cost: zero beyond the DDL.

### Idea: fixed-poll-over-exponential-backoff-at-low-rate
- (N=0.70, E=0.78, F=0.97) — For reinforcement signal writes at Recall's rate (3–15 co_retrieval events per day), a fixed 5-second poll interval is strictly superior to exponential backoff. Exponential backoff introduces recovery latency proportional to the number of attempts — at 5 attempts with base=60s, recovery is 4+ minutes after a brief PG restart. Fixed 5-second poll recovers within 5 seconds. The deadletter boundary (5 attempts × 5s = 25 seconds) is comparable to exponential (1+2+4+8+16=31s). The only advantage of exponential backoff is avoiding thundering-herd at high write rates — irrelevant at 15 events/day. Fixed poll is simpler, faster to recover, and equivalent in dead-letter timing. This decision is reproducible and can be revisited if write rates increase by 1000×.

### Idea: null-epoch-graceful-upsert
- (N=0.72, E=0.75, F=0.96) — Storing `epoch = NULL` in the outbox when the Redis INCR fails (Q112 RENAME window or any Redis unavailability) and using `COALESCE($epoch, last_reinforced_epoch, 0)` in the UPSERT preserves the co-retrieval edge record without overwriting a valid epoch with a null. The CASE expression in the UPSERT (`WHEN $4 IS NOT NULL THEN GREATEST(co_retrievals.last_reinforced_epoch, $4) ELSE co_retrievals.last_reinforced_epoch`) ensures that a null-epoch retry does not corrupt the `last_reinforced_epoch` field. This is the minimal protocol for epoch-nullable reinforcement writes: record the pair durably, reinforce the edge, and leave epoch ordering intact for any future write that does have a valid epoch.

## Recommendations

1. **Add the DDL** to `_CREATE_TABLES` in `src/storage/postgres_store.py` (two tables: `co_retrievals_outbox` and `co_retrievals_dead_letter`).

2. **Add `store_co_retrievals()`** as a method on `PostgresStore`. The method contains the three-statement atomic transaction: INSERT outbox → UPSERT co_retrievals → UPDATE outbox processed_at.

3. **Add `_co_retrievals_flush_worker()`** as a module-level async function in `src/api/main.py`. Start it via `asyncio.create_task()` inside the lifespan `yield` block. Cancel it in the `finally` block.

4. **Add S8 dead-letter gauge** to the health dashboard. Zero is nominal; any non-zero value requires investigation.

5. **Do not** add the outbox to the memory store path (`/store` endpoint). The outbox is only for co_retrievals reinforcement writes — not for the primary PG INSERT + Qdrant UPSERT + Redis INCR write pipeline, which already has S4 epoch-integrity monitoring as its durability signal.

6. **The outbox survives restart unconditionally** because it lives in PostgreSQL. No SQLite file, no mounted volume, no in-process queue is required. The flush worker recovers pending entries within 5 seconds of any restart.
