## Architecture Dependencies Map

Before the checklist, the dependency graph that determines sequencing:

```
Redis AOF config change
    └─► Redis restart (AOF active before epoch guard code reads it)
        └─► epoch guard code deploy (Q111)

SQL: ALTER TABLE memories ADD COLUMN last_access_epoch    ← Q111 pre-req
SQL: ALTER TABLE memories ADD COLUMN tenant_id            ← Q112
SQL: CREATE TABLE co_retrievals (with tenant_id)          ← Q110+Q115
SQL: CREATE INDEX CONCURRENTLY (all indexes)              ← Q112, Q115 indexes
    └─► FastAPI code deploy (new code is compatible with new schema; old code was compatible too)

Redis RENAME epoch key                                    ← Q112 Phase B
    └─► FastAPI config update + restart                  ← Q112 Phase C

Qdrant alias switch                                       ← Q112 (recall → recall_tim)

Neo4j data migration → dual-write → A/B gate → cutover   ← Q110 Phases 2-5
    └─► Neo4j container removal                          ← Q110 Phase 6

Embedding cache code deploy                               ← Q116
    └─► (no schema prereqs — Redis key namespace only)

Tombstone regression smoke test                           ← Q124 (after all code is deployed)

Health signal matrix verification                         ← Q125 (final gate: all signals GREEN)
```

**Key sequencing constraint**: All SQL DDL runs before any code is deployed. All code deploys before Neo4j is removed. AOF enforcement runs before epoch guard code. The Redis key rename is the narrowest window (1–2 second hook error exposure) and should be scheduled during a pause in Claude Code usage.

### PHASE 2 — SQL Schema Migrations (20 min)

  --pg-dsn "postgresql://recall@localhost:5432/recall" \

4. Report total edges migrated

## Dependency Ordering Rationale

The checklist implements four sequencing constraints:

**1. DDL before code** (Q112 principle): SQL schema migrations run in Phase 2 before any code is deployed in Phase 4. Old code running against new schema (extra columns) is a no-op. New code running against old schema (missing columns) fails. Therefore DDL always precedes code.

**2. AOF before epoch guard** (Q111 Step 7): The epoch guard's AOF health check fires at API startup lifespan. If AOF is not enforced before the guard code is deployed, the lifespan check would log a warning on startup, which is acceptable but misleading. More critically, if Redis restarts without AOF during the deployment window, the guard fires and auto-repairs — which is correct behavior but adds noise during deployment.

**3. Redis RENAME in Phase 5, not Phase 4** (Q112 config-last ordering): The code deployed in Phase 4 must still read the old key name (`recall:global_epoch`) at deploy time. The RENAME happens after code is deployed and running. Then the config is updated (new key name) and the API restarts. This prevents the 1-epoch-loss failure mode that config-first ordering creates.

**4. Neo4j removal after dual-write validation** (Q110 Phase ordering): The A/B gate in Step 22 is the critical dependency. Neo4j cannot be removed until the PG co_retrievals result set matches Neo4j's result set at ≥ 70% overlap. The Q110 5-phase structure is preserved intact within this checklist (Phases 3–6 in order).

## Qualifying Ideas

### epoch-guard-non-fatal-wrapper-pattern (N=0.70, E=0.85, F=0.95)
Guards that instrument the health of infrastructure (epoch reset detection, AOF presence checks) must never block the primary operation they guard. Wrapping each guard in its own `try/except` with `logger.warning` ensures that a guard failure does not create a new failure mode. This principle is implemented in the epoch guard (Q111) and should be applied to all future health-check insertions on hot paths.

### migration-before-code-deployment-ordering (N=0.68, E=0.72, F=0.95)
When DDL adds columns with safe defaults, old code running against new schema is a no-op, but new code running against old schema fails. The forced ordering constraint (DDL first, code second) is a precise, derived rule — not a convention. Applied throughout Phase 2.

### redis-rename-config-last-ordering (N=0.70, E=0.68, F=0.92)
The three-phase atomic Redis key rename (deploy code → RENAME key → update config) prevents both the config-first epoch-loss failure and the code-last prolonged error window. The only exposure is a 1–2 second window of hook errors, which is logged and non-fatal.

### tombstone-guard-completeness-checklist (N=0.72, E=0.78, F=0.96)
The tombstone guard must be applied to both sides of the BFS traversal (seed selection AND neighbor selection). Q101's partial fix applied it only to the neighbor clause; Q113 identified the seed clause as an equally critical vector. The completeness checklist: (1) seed sampling WHERE clause, (2) neighbor filtering WHERE clause, (3) all monitoring queries except reconcile.

### content-hash-embedding-cache-determinism (N=0.72, E=0.85, F=0.95)
Keying the embedding cache by SHA-256 of the normalized query string makes the cache session-independent and cross-session. Because qwen3-embedding:0.6b is deterministic, the cache is effectively a global lookup table, not a per-session scratchpad. Expected 55–65% cache hit rate yields ~1 ms average latency reduction per retrieve call.

## Recommendations

1. **Run Phase 0 the night before** — take the pg_dump and docker-compose.yml backup on Friday evening so Saturday starts from a known-good baseline.

2. **Do NOT run Neo4j removal (Step 40) before the A/B gate passes** — the ≥ 70% overlap check in Step 22 is the single most important STOP gate in the entire checklist. Everything else is reversible in under 5 minutes; premature Neo4j removal with a flawed `co_retrievals` dataset is the one scenario that creates real data risk.

3. **Q124 smoke test (Step 45) is mandatory before declaring success** — the tombstone guard is a correctness guarantee for context injection, not a performance optimization. A deploy that ships without confirming the tombstone exclusion works is actively harmful.

5. **Schedule the Redis RENAME window (Step 31)** — even though the exposure is only 1–2 seconds of logged errors, scheduling it during a natural break in Claude Code usage avoids any user-visible disruption to hook operation.

6. **Keep the Neo4j volume for 7 days before `docker volume rm neo4j_data`** — the data is safe, the container is gone. The volume is a ~50 MB recovery backstop during the confidence-building period.
