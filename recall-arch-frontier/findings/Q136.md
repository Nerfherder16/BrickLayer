### Gap 1: Migration Pattern — Q130 Missed the `_MIGRATE_*` Conventions

**Q130's recommendation**: "Insert this into `_CREATE_TABLES` in `src/storage/postgres_store.py` after the `membench_runs` table block (after line ~162)."

**Actual codebase**: The real `postgres_store.py` does NOT put new tables into `_CREATE_TABLES` for additions to a running system. The `_CREATE_TABLES` block contains only tables that were in the original schema. All later additions use separate `_MIGRATE_*` string constants that are executed in sequence in `PostgresStore.connect()` (lines 304–313).

Current migration sequence in `connect()`:
```python
await conn.execute(_CREATE_TABLES)
await conn.execute(_MIGRATE_AUDIT_USER_ID)
await conn.execute(_MIGRATE_MEMBENCH_V2)
await conn.execute(_MIGRATE_INJECTION_LOG_V2)
await conn.execute(_MIGRATE_INJECTION_LOG_V3)
await conn.execute(_MIGRATE_INJECTION_LOG_V4)
await conn.execute(_MIGRATE_INJECTION_LOG_V5)
await conn.execute(_MIGRATE_USERS_PASSWORD_HASH)
await conn.execute(_MIGRATE_USERS_EMAIL)
```

**Correction**: The outbox DDL must be a new `_MIGRATE_CO_RETRIEVALS_OUTBOX` string constant, NOT embedded in `_CREATE_TABLES`. Using `CREATE TABLE IF NOT EXISTS` makes it idempotent for re-runs.

**Exact constant to add** (after `_MIGRATE_INJECTION_LOG_V5`, before `_parse_dsn`):

```python

    id              BIGSERIAL PRIMARY KEY,

    outbox_id       BIGINT      NOT NULL,
    from_memory_id  UUID        NOT NULL,
    to_memory_id    UUID        NOT NULL,
    tenant_id       VARCHAR(32) NOT NULL,
    epoch           INTEGER,
    created_at      TIMESTAMPTZ NOT NULL,

```

**And add to `connect()` after `_MIGRATE_USERS_EMAIL`**:

### Gap 2: Pool Access Pattern — Direct Pool vs. `pool.acquire()`

**Q130's code**: Uses `async with self.pool.acquire() as conn:` throughout `store_co_retrievals()`.

**Actual codebase**: Both patterns are used in `postgres_store.py`. For single-statement calls, many methods call `self.pool.execute(...)` or `self.pool.fetch(...)` directly (e.g., `log_audit` at line 338, `get_audit_log` at line 386). For multi-statement transactions, `pool.acquire()` + `conn.transaction()` is the correct pattern.

**Verdict**: Q130's pattern is correct for the outbox write (which is a multi-statement transaction). The direct pool pattern would not support `conn.transaction()`. No change needed for the `store_co_retrievals()` method.

**One correction**: The `log_audit` method accesses `self.pool.execute(...)` without checking `if not self.pool` — it relies on the caller not invoking it before connect. The `store_co_retrievals()` method in Q130 correctly guards with `if not pairs or not self.pool: return`. This guard is correct and consistent.

### Gap 3: Lifespan Structure — `yield` Inside `try`, Not After `finally`

**Q130's recommendation**:
```python
# After: logger.info("storage_connections_established")
# Add:
_flush_task = asyncio.create_task(...)

yield  # application runs here

# In the finally block, before closing postgres:
_flush_task.cancel()
```

**Actual lifespan structure** (lines 55–127 of `main.py`):
```python
@asynccontextmanager
async def lifespan(app: FastAPI) -> AsyncIterator[None]:
    qdrant = None
    neo4j = None
    redis = None
    postgres = None

    try:
        qdrant = await get_qdrant_store()
        neo4j = await get_neo4j_store()
        redis = await get_redis_store()
        postgres = await get_postgres_store()
        # ... graphiti init ...
        logger.info("storage_connections_established")

        yield  # <-- yield is INSIDE the try block, line 84

    finally:
        logger.info("shutting_down_recall_api")
        if qdrant: await qdrant.close()
        # ... close others ...
        # Reset singleton globals
```

**Critical observation**: The `yield` is at line 84, inside the `try:` block, not after `finally`. This means `_flush_task` must be created before `yield` and cancelled in `finally`. The flush task variable must be declared at the same scope as `qdrant`, `neo4j`, etc. — before the `try` block or inside it, but accessible in `finally`.

**Correct lifespan integration** (the exact diff):

```python
# Before `try:`, add variable declaration alongside the existing None-inits:
_flush_task = None  # add alongside qdrant = None, neo4j = None, etc.

# After logger.info("storage_connections_established") and BEFORE yield:
_flush_task = asyncio.create_task(
    _co_retrievals_flush_worker(postgres),
    name="co_retrievals_flush_worker",
)

yield  # existing line 84

# In finally block, BEFORE `if qdrant: await qdrant.close()`:
if _flush_task is not None:
    _flush_task.cancel()
    try:
        await _flush_task
    except asyncio.CancelledError:
        pass
```

The `asyncio` import is already present as a transitive import but is not explicitly imported in `main.py`. **Add `import asyncio` at the top of `main.py`** (currently the file uses only `from collections.abc import AsyncIterator` for async-related imports; `asyncio` itself is not in the import list — verified by reading lines 7–14).

### Gap 4: `_co_retrievals_flush_worker` Placement — Module-Level in `main.py` is Correct but Needs Co-location with Imports

**Q130's recommendation**: Define `_co_retrievals_flush_worker` as a module-level function in `src/api/main.py`.

**Actual codebase**: `main.py` is 489 lines and handles app configuration, middleware, and route registration. It imports `postgres` store via `get_postgres_store()` from `src.storage` (line 25). Adding a 100-line background task function here is technically correct but creates a file that mixes concerns.

**Better alternative** (not a gap, an improvement): The function could live in `src/storage/postgres_store.py` as a module-level function alongside `PostgresStore`. This keeps all PostgreSQL logic in one place and avoids importing `PostgresStore` type annotation cross-module. However, `_co_retrievals_flush_worker` uses `asyncio` and `structlog` which are already imported in `postgres_store.py` — making it the natural home.

**Decision for this finding**: Recommend placing `_co_retrievals_flush_worker` in `src/storage/postgres_store.py` (not `main.py`). This requires exporting it or importing it in `main.py`. The import line in `main.py` lifespan:
```python
from src.storage.postgres_store import _co_retrievals_flush_worker
```
This is a minor change from Q130's spec but architecturally cleaner.

### Gap 5: Health Dashboard S8 Signal — Placement and Pattern

**Q130's recommendation**: Add S8 to `src/api/routes/health_dashboard.py` as a bare PostgreSQL query inside an existing endpoint.

**Actual `health_dashboard.py` structure**: The dashboard does NOT make direct `pool.acquire()` calls inline in route handlers. Instead, it delegates entirely to `create_health_computer()` (from `src.core.health`) and calls methods on the returned computer object:

```python
computer = await create_health_computer()
dashboard = await computer.compute_dashboard()
```

For routes that do need direct PG access (like `snapshot-baseline`), the pattern is:
```python
from src.storage import get_postgres_store, get_qdrant_store
pg = await get_postgres_store()
await pg.save_metrics_snapshot(...)
```

**Correct S8 placement**: S8 should NOT be a bare `async with postgres.pool.acquire() as conn:` block inline in a route. It should be a new endpoint or integrated into the existing `get_retrieval_quality` endpoint (which already calls `pg.get_retrieval_quality_stats()`). The cleanest integration is a new method on `PostgresStore` called `get_outbox_dead_letter_count()`, called from a new endpoint:

```python
# New endpoint in health_dashboard.py:
@router.get("/health/outbox-health")
@limiter.limit("10/minute")
async def get_outbox_health(request: Request):
    """
    co_retrievals outbox dead-letter count (S8 signal).
    Any non-zero value indicates reinforcement signal loss.
    """
    try:
        from src.storage import get_postgres_store
        pg = await get_postgres_store()
        count = await pg.get_outbox_dead_letter_count()
        return {
            "dead_letter_count": count,
            "status": "nominal" if count == 0 else "alert",
            "threshold_minor": 5,
            "threshold_major": 5,
        }
    except Exception as e:
        logger.error("outbox_health_error", error=str(e))
        raise HTTPException(status_code=500, detail="Internal server error")
```

**New method on `PostgresStore`** (to be added in `postgres_store.py`):
```python
async def get_outbox_dead_letter_count(self, hours: int = 24) -> int:
    """Return count of co_retrievals dead-letter entries in the last N hours."""
    if not self.pool:
        return 0
    return await self.pool.fetchval(
        """
        SELECT COUNT(*)
        FROM co_retrievals_dead_letter
        WHERE failed_at > NOW() - ($1 || ' hours')::INTERVAL
        """,
        str(hours),
    )
```

**Note on Q130's inline pattern**: The inline `async with postgres.pool.acquire() as conn:` pattern in Q130's S8 spec would fail because `health_dashboard.py` routes do not have a local `postgres` variable — they get the store via `await get_postgres_store()`. Q130's S8 code snippet would raise `NameError: name 'postgres' is not defined`. This is a concrete bug in Q130's spec.

### Gap 6: `asyncio.ensure_future` vs. `asyncio.create_task` in Search Handler

**Q130's retrieve handler integration**:
```python
import asyncio as _asyncio
_asyncio.ensure_future(_pg.store_co_retrievals(_pairs, "tim", _epoch))
```

**Actual codebase convention**: The existing codebase does not use `asyncio.ensure_future()` anywhere — it's deprecated in Python 3.10+ in favor of `asyncio.create_task()`. Checking `main.py` and `health_dashboard.py`: no usage of `ensure_future`.

**Correction**: Use `asyncio.create_task()`:
```python
import asyncio
asyncio.create_task(_pg.store_co_retrievals(_pairs, "tim", _epoch))
```

This also requires `import asyncio` at the top of `search.py` if not already present.

## Qualifying Ideas

### Idea: co-retrievals-outbox-migration-over-create-tables
- (N=0.68, E=0.90, F=0.98) — The `_MIGRATE_*` pattern used consistently in the live codebase is the correct DDL delivery mechanism for new tables in a running system. Using `_CREATE_TABLES` for post-initial-schema additions would require either: (a) dropping and recreating tables (data loss) or (b) relying on `IF NOT EXISTS` (which works but violates the codebase's established convention). The `_MIGRATE_*` + idempotent `connect()` pattern is the established mechanism: each migration runs on every startup and is idempotent via `IF NOT EXISTS` or `DO $$ IF NOT EXISTS ... $$`. Q130's spec was written against an inferred codebase; the actual `postgres_store.py` makes the migration pattern unambiguous.

### Idea: outbox-health-dedicated-endpoint-over-inline-query
- (N=0.65, E=0.88, F=0.97) — The S8 dead-letter signal deserves a dedicated `/admin/health/outbox-health` endpoint (following the pattern of `/admin/health/retrieval-quality`, `/admin/health/injection-efficiency`, `/admin/health/inter-agent`). Embedding the query inline in an existing endpoint violates the single-responsibility structure of `health_dashboard.py` and the delegation pattern (routes call methods on `PostgresStore`, not raw `pool.acquire()` blocks). The dedicated endpoint with a `get_outbox_dead_letter_count()` method on `PostgresStore` is consistent with every other PG-backed health signal in the file.

### Idea: asyncio-create-task-over-ensure-future
- (N=0.65, E=0.92, F=0.99) — `asyncio.ensure_future()` is deprecated since Python 3.10 and produces `DeprecationWarning` in Python 3.12 (which is likely the target given the `|` union type syntax used in `postgres_store.py`). `asyncio.create_task()` is the modern equivalent, provides the same semantics (schedules a coroutine on the running event loop), and accepts an optional `name` parameter useful for debugging. The correction is one word; the existing codebase uses no `ensure_future` calls anywhere, confirming `create_task` is the established convention.

## Recommendations

1. **Use `_MIGRATE_CO_RETRIEVALS_OUTBOX`** — not `_CREATE_TABLES` — for the DDL. This is the critical correction. The existing DB has all tables from `_CREATE_TABLES` already; adding new tables there has no effect on a running system.

2. **Place `_co_retrievals_flush_worker` in `postgres_store.py`** alongside other PostgreSQL logic. Import it into `main.py` lifespan with one line.

3. **Fix S8**: Create a dedicated `/admin/health/outbox-health` endpoint following the established `health_dashboard.py` delegation pattern. Q130's inline `postgres` variable would raise `NameError`.

4. **Use `asyncio.create_task()`** not `asyncio.ensure_future()`.

5. **All Q130 core logic is correct**: the atomic three-step transaction, epoch-nullable handling, fixed-poll retry, dead-letter escalation, and restart-survival argument are all valid. The corrections are integration details, not design changes.

6. **Do not add `co_retrievals` or `co_retrievals_outbox` to the primary write path (`/memory/store`)** as Q130 correctly states — the outbox is only for retrieval reinforcement.
