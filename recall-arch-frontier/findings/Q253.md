## Research Question

What does a memory system look like if the agent never queries? Design from physical requirements.

**Forbidden terms**: query, search, retrieve, find, lookup, fetch, ask, request, pull, get, seek, scan, probe, match.

---

## The Removed Assumption

Every retrieval system assumes: the agent has a need → issues a query → receives ranked results → uses the results.

This is the **pull model**. The agent initiates. The memory system responds.

Removing the query step forces the deeper question: what physical mechanism causes a memory to become available at the moment it is needed, without the agent initiating a request?

---

## Physical Analysis: What Must Be True for Memory to Be Available Without a Request

For a memory M to be available at the moment of use, the memory's representational content must be present in the agent's active context AT THAT MOMENT — without the agent asking for it.

This is the **push model**. Two physical mechanisms can achieve it:

### Mechanism 1: Preemptive Activation (Spreading Activation — Q235)

When a new memory is stored, it activates its CO_RETRIEVED neighbors. These neighbors' activation level increases — they become more likely to surface in the next response even without being explicitly requested.

The key insight: spreading activation IS retrieval without a query. The activations propagate through the CO_RETRIEVED graph continuously as memories are stored. By the time the agent needs a memory, it may already be in the activation foreground — not because the agent asked, but because storing a related memory pushed its activation up.

**This is the biological memory analog**: priming. Hearing "bread" activates "butter" before you decide to think about butter. No query — the activation propagates automatically.

Q235's spreading activation is the complete answer to "memory without a query" — it is exactly a push model.

### Mechanism 2: Predictive Injection (Anticipatory Context Loading)

Rather than waiting for a query, the memory system predicts what the agent will need and injects it preemptively.

Physical basis: the agent's current task context is itself a signal. If the agent just ran a command that modified a configuration file, the memory system knows (via PostToolUse hook) that the agent will likely need the context about that configuration next. The memory can be injected into the context window before the agent's next response — not as a response to a query, but as anticipatory context.

This extends the Q235 spreading activation mechanism: instead of activating neighbors in response to a STORED memory, activate neighbors in response to a USED tool or file.

### Mechanism 3: Context-Triggered Passive Surface

The memory system monitors the agent's current context passively and triggers memory injection when certain conditions are met — without the agent asking.

Physical implementation: a PostToolUse hook monitors which files are edited and which services are referenced. When a file matching a known pattern is touched (e.g., a `.yml` file in a service directory), the hook injects the memories associated with that service into the next response.

This is the **rule-based push**: memory surfaces when a contextual trigger fires, not when a query is issued.

## The Q235 Connection: Spreading Activation IS Query-Free Memory

Q235's spreading activation resolves the Q253 question directly:

1. Memory M1 is stored → its CO_RETRIEVED neighbors (M2, M3, M4) get activation increments
2. At the NEXT UserPromptSubmit, M2, M3, M4 have elevated activation weights
3. When the K=10 results are computed, M2, M3, M4 rank higher due to activation boost
4. They appear in the injected context — not because the agent asked for them, but because M1's storage triggered their pre-activation

The "query" still happens (embedding + HNSW + Tantivy), but the RELEVANT memories surface because of pre-activation, not because the query was well-formed.

**For the truly query-free case**: the activation level alone (without any query) could determine what surfaces. If a memory's activation exceeds a threshold, inject it unconditionally — the query is optional.

## The Push-Only Model

A complete push-only memory system:

```
PostToolUse (tool_used):
  → extract entity-predicate pairs from tool result (Q243 Tier 1)
  → for each matched entity-predicate: activate its CO_RETRIEVED neighbors
  → if any neighbor's activation > SURFACE_THRESHOLD: flag for next injection

UserPromptSubmit:
  → collect all memories with activation > SURFACE_THRESHOLD
  → inject them into context (no embedding, no ANN, no query)
  → reset activation of injected memories to baseline
```

This is a zero-query memory system. The agent never issues a request. Memories surface because past tool usage activated them.

**Limitation**: the push-only model misses memories that are not yet activated (no CO_RETRIEVED neighbors pointing to them). First-time topics have no activation pre-history. The push model is complementary to the pull model, not a replacement — it handles the "known unknowns" (Tim didn't know to ask about X, but X is relevant because he just touched Y).

## Complementarity: Push + Pull Together

| Model | Handles | Misses |
|---|---|---|
| Pull (current Recall) | Directly queried topics | Unknown unknowns |
| Push (spreading activation) | Pre-activated related topics | First-time topics |
| Push + Pull | Both | Nothing in the CO_RETRIEVED graph |

The two models together cover the full memory access spectrum. The push model handles exactly the gap that Q235 identified: memories the agent doesn't know to ask about.

## Verdict

**BREAKTHROUGH.** Q235's spreading activation IS the physical answer to "memory without retrieval": memories surface because past observations activated them, not because the agent asked for them. The push model — PostToolUse triggers activation propagation, UserPromptSubmit injects above-threshold activations — is a complete zero-query memory system for the "unknown unknown" class. Its limitation: first-time topics have no activation history → push model misses them. The pull model (Q231 hybrid retrieval) handles first-time topics. Together they cover the full spectrum. The taboo removal reveals that the query step is not physically necessary for pre-activated memories — it is an optimization for unactivated memories.
