# Q121 — Memory Deduplication Strategy

**Question**: What is the correct memory deduplication strategy when two memories have cosine similarity above 0.97 — should near-duplicate pairs be merged (one winner, loser tombstoned with content union), co-exist with a STALE co_retrievals edge connecting them, or be handled exclusively by the Q013 compaction pass, and what is the information-theoretic argument for which strategy minimizes retrieval precision loss when a future query is semantically equidistant between the two near-duplicates?
**Question Type**: ADVERSARIAL
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: information theory, RAG deduplication, database compaction design, embedding model calibration, retrieval precision analysis

---

## Architecture Anchoring

Before the adversarial threads: the relevant architecture is the three-store stack from Q115. Neo4j is eliminated. The co-retrieval graph lives in PostgreSQL's `co_retrievals` table with columns including `co_retrieval_count`, `weight`, `last_reinforced_epoch`, and `relationship_type` (FRESH/STALE). There is no separate "STALE co_retrievals edge connecting near-duplicates" in the Q115 architecture — the FRESH/STALE partition applies to reinforcement age, not to semantic similarity relationships. The deduplication question therefore reduces to: **merge with tombstone** (one record, content union, loser's `co_retrieval_count` and `weight` can be optionally folded into winner), **co-existence** (two records, both retrievable), or **deferred to compaction** (Q013 nightly consolidation pass).

---

## Thread A: Defense — Merge Is Always Correct

**Core claim**: At cosine similarity ≥ 0.97, near-duplicate memories are functionally identical for all retrieval purposes, and allowing them to co-exist is a strictly dominated strategy: it splits the relevance signal across two records, amplifies retrieval noise, and wastes context window tokens on semantically redundant content.

### Evidence

**1. The 0.97 threshold represents near-zero semantic distance.**

For sentence-level embeddings, the empirical literature on near-duplicate detection (Wang et al. 2022, "Near-Duplicate Detection for Short Texts") places the paraphrase boundary at cosine ≥ 0.92–0.95. At 0.97, two memories are in the same semantic equivalence class with high confidence: the residual 3% embedding distance corresponds to surface-level linguistic variation (word order, synonym substitution, minor expansion), not semantic divergence. The Q013 analysis established that a mature Recall store has R = 0.40–0.50 redundancy fraction, with pairs above 0.92 threshold constituting the dominant noise source.

**2. Co-existence degrades retrieval precision measurably.**

Q013's false-competitor model: with N=1,000 memories and R=0.40, `P(all top-K distinct) = (0.6)^5 = 7.8%`. Applied to a near-duplicate pair at 0.97 similarity: a future query semantically equidistant between the two near-duplicates (the exact scenario the question poses) will retrieve BOTH records in the top-K, because both are equally close to the query vector. This consumes two context slots for one unit of information. Effective K drops from 5 to 4 for this query, or worse if the redundancy clusters are deeper.

**3. The co_retrieval_count loss from merging is addressable.**

The key technical constraint posed in the question is: "merging destroys the frequency signal for the loser." This is true for a naive merge, but the information loss is bounded and recoverable. The PostgreSQL co_retrievals UPSERT pattern from Q115 already handles the analogous case: `ON CONFLICT DO UPDATE SET co_retrieval_count = co_retrievals.co_retrieval_count + 1`. A merge operation can fold the loser's co_retrieval_count into the winner's co_retrieval_count at merge time. The loser's `weight` accumulation can also be folded: `winner.weight = MAX(winner.weight, loser.weight)` or summed. This is not a loss — it is a concentration of signal into a single record.

**4. Content union is lossless for the text component.**

The merged memory's content is the union (or the longer/more specific version) of both originals. No propositions are lost. The winner inherits all tags (union), max importance score, and oldest created_at. The only genuine loss is the loser's independent provenance trail — which matters for debugging but not for retrieval quality.

**5. The "0.97 threshold may drift with model version bumps" objection is weak at this threshold.**

The concern that 0.97 today maps to 0.94 after a model bump has merit for threshold values near 0.92 (the paraphrase boundary). At 0.97, the semantic content of two memories is so similar that model-version drift would need to be catastrophically large to reclassify 0.97-similar memories as semantically distinct. The threshold is conservative. A model that maps two near-identical sentences to 0.97 cosine similarity will continue to map them above 0.92 after any reasonable model update. The stability argument applies strongest where the threshold has the most headroom above the paraphrase boundary — 0.97 has substantial headroom.

**Best case scenario**: Write-time merge at ≥ 0.97 eliminates the dominant retrieval noise source, reduces effective K pollution from 92% of queries (Q013) to near-zero for high-similarity duplicates, and produces a co_retrieval_count and weight that accurately represents the aggregate reinforcement signal of both originals.

## Thread B: Attack — Co-existence Is Always Correct

**Core claim**: At any fixed cosine threshold, the merge operation irreversibly destroys information that cannot be recovered, while co-existence with deferred compaction preserves full provenance and allows the system to correct mistaken near-duplicate classifications after the fact.

### Evidence

**1. Cosine similarity at 0.97 is NOT a reliable semantic identity signal at Recall's embedding model scale.**

Recall uses `qwen3-embedding:0.6b` — a 0.6-billion-parameter model running locally on a GPU server with network latency in the critical path (Q027 baseline: embedding latency is a known overhead). Q013 established that this model's embedding space has intrinsic dimensionality of approximately 25 dimensions out of 1,024. With 97.6% of the embedding vector being non-informative noise, the effective dimensionality contributing to cosine similarity is ~25 dimensions. A small embedding model maps many semantically distinct sentences to high cosine similarity in a compressed low-dimensional subspace. Consider two memories:

```
Memory A: "CasaOS host IP is 192.168.50.19, port 8200 for Recall"
Memory B: "Recall API is at 192.168.50.19:8200 on the CasaOS host"
```

These are paraphrases of the same fact — merge is correct. Now consider:

```
Memory C: "Qdrant runs on port 6333, accessed via the recall-qdrant container"
Memory D: "Qdrant vector store port is 6333 on the recall stack"
```

Also very high similarity — but Memory C encodes container name context while Memory D encodes stack context. At qwen3-embedding:0.6b's limited capacity, the cosine distance between C and D may easily fall at 0.97–0.98. The merge would lose the container name.

A larger model (ada-002, OpenAI; BGE-large; E5-large) would place C and D at 0.93–0.95, still near-duplicate but with more headroom. At 0.6b parameters, the embedding model compresses the semantic space aggressively, pushing many functionally distinct memories into the 0.97+ band.

**2. The co_retrieval_count loss is NOT recoverable by folding.**

Thread A's proposed fix — "fold the loser's co_retrieval_count into the winner" — conflates two distinct signals:

- `co_retrieval_count` for a pair (A, B) represents: "memory A and memory B were retrieved together N times." This is a relational signal about the joint retrieval event between A and B.
- After merging A into A' (winner), the folded `co_retrieval_count` on A' becomes: "A' was co-retrieved with its neighbors N_A + N_B times total." But N_B's contribution was co-retrievals between B and B's neighbors — which are NOT the same neighbors as A's. B's neighborhood in the co_retrievals graph is a distinct set of memories.

The folding destroys the *asymmetry* of the co_retrieval_count signal: B may have been co-retrieved with memories that A was never co-retrieved with (different context trajectories, different session patterns, different temporal clusters). After merge, those cross-neighborhood signals are lost — the winner's co_retrieval_count now incorrectly represents a count of events that involved the winner directly.

**3. Timestamped provenance is not a debugging concern — it is a retrieval quality signal.**

Each memory has a `created_at` timestamp. At retrieval time, recency weighting (a known improvement from Q013 follow-ups) would use `created_at` to boost newer memories. The loser's `created_at` is discarded in a merge (the canonical rule is "preserve oldest created_at"). If Memory B was written last week (newer, more accurate fact) and Memory A was written six months ago (older, possibly stale), a merge that preserves A's `created_at` makes the merged record appear older than it is, suppressing recency weighting that would have correctly boosted B.

**4. The Q013 compaction pass is specifically designed for this exact scenario.**

Q013's `recall-importance-weighted-consolidation` idea (N=0.75) proposes a nightly consolidation job at threshold 0.88, with cluster-size importance bonus and full audit trail. This is the correct venue for merge decisions because: (a) it operates over clusters of N memories, not just pairs, allowing optimal canonical selection across the full cluster; (b) it runs on the full store including historical memories that bypassed write-time detection; (c) it is reversible in a staging/dry-run mode; (d) it has a full log for post-hoc review of merge decisions. The compaction pass addresses all of Thread B's concerns about irreversibility and information loss — the loser is tombstoned with its full content preserved in an audit table rather than deleted.

**5. The write-time path cannot handle all near-duplicate cases.**

At write time, only the incoming memory and its top-K nearest neighbors are visible. If two memories arrive through different hooks on the same day (observe-edit vs. session-summary), one may not be present in the vector index when the other arrives. The write-time check queries Qdrant for the top-3 most similar existing memories — if the near-duplicate has not yet been indexed (Qdrant indexing is asynchronous), the write-time check misses it. The compaction pass, running nightly over the full index, catches all pairs that write-time missed.

**Failure mode**: Premature write-time merge at a fixed 0.97 threshold destroys irrecoverable information (cross-neighborhood co_retrieval signals, provenance timestamps, minor content variants that matter for the specific use case) and cannot be undone. A false merge is permanently damaging. A false non-merge (allowing co-existence) costs at most one context window slot per query — a recoverable, bounded loss.

## Synthesis

### What Thread A assumes that Thread B exposes

Thread A assumes that cosine similarity at the 0.97 threshold is a reliable proxy for semantic identity. Thread B exposes that this assumption is model-dependent: at qwen3-embedding:0.6b's limited capacity (25 effective dimensions, Q013), the 0.97 threshold may be more conservative than it appears because the model's compressed embedding space forces many semantically near-but-distinct memories into the high-similarity band. The hidden assumption is "similarity ≥ 0.97 implies semantic equivalence" — Thread B shows this is only true for embeddings with sufficient intrinsic dimensionality to preserve fine-grained semantic distinctions.

Thread A also assumes that the co_retrieval_count loss can be recovered by folding. Thread B exposes that the co_retrieval signal is relational (it encodes pairwise co-retrieval events between specific memory IDs, not aggregate retrieval counts), so a sum-fold after merge conflates two distinct provenance trails into one incorrect aggregate. This is not a marginal concern: the co_retrievals table IS the Recall graph after Q115 — it is the primary reinforcement learning signal for retrieval ranking.

### When Thread A is right

- **When the merge happens at query time (virtual merge), not at write time (physical merge)**: If the retrieval pipeline detects near-duplicate results in the top-K and collapses them before returning (returning one merged result with union content and summed retrieval signal), Thread A's precision gains are achieved without any permanent information loss. The store remains intact; the merge is ephemeral and reversible.
- **When the near-duplicate pair shares all co_retrieval neighborhoods**: If Memory A and Memory B are pure linguistic paraphrases that arose from the same event (same session, same context, same hooks), they will have nearly identical co_retrieval neighbors. Folding their co_retrieval_counts is approximately correct because their co-retrieval neighborhoods overlap at ~95%+.
- **When the compaction pass has confirmed the merge decision**: After the Q013 nightly consolidation pass has identified the pair as a cluster, the merge is now a reviewed, audit-trailed operation rather than a real-time write-time decision. Thread A's merge prescription is correct in this context.
- **When the embedding model is upgraded to a larger model (BGE-large, E5-large, or a future qwen3-embedding:2b+)**: A larger model with 50+ effective dimensions places the 0.97 threshold firmly in the semantic-identity zone. The model-drift risk is lower; merge confidence is higher.

### When Thread B is right

- **Write-time merge at a fixed threshold is wrong for qwen3-embedding:0.6b**: The small embedding model's compressed semantic space makes the 0.97 threshold less reliable than it would be for a larger model. Write-time merge is irreversible. The risk asymmetry (false merge >> false non-merge) favors caution.
- **The co_retrieval_count loss from physical merge is a genuine information-theoretic loss**: The signal is relational (pairwise event count), not additive. Folding is an approximation that is only valid when the two memories' co_retrieval neighborhoods are near-identical.
- **Deferred compaction with tombstoning is strictly better than write-time merge**: The compaction pass can compare full clusters, perform dry runs, preserve the loser's content in an audit trail, and be triggered after model version bumps with recalibrated thresholds.

### The Query-Time Equidistance Scenario: Information-Theoretic Analysis

The question poses a specific scenario: a future query is semantically equidistant between two near-duplicates. What is the precision loss under each strategy?

**Case 1: Merge (physical, write-time)**

- Store contains one memory: the merged winner, content = union(A, B), co_retrieval_count = folded_sum.
- Future query equidistant between A and B: the merged winner has embedding close to both A and B (centroid, approximately). The query returns the merged winner at rank 1. Retrieval precision = 1/1 for the relevant fact. No context window waste. Co_retrieval signal is aggregated (lossy but functional).
- **Precision at this query: 1.0 (for the merged record), but the co_retrieval neighborhood may be distorted.**

**Case 2: Co-existence (no merge)**

- Store contains both A and B. Both are in top-K because the query is equidistant.
- Two context slots consumed for one unit of information. The context slot that could carry a third distinct relevant memory is consumed by the duplicate.
- The co_retrieval graph for A and B each retains its independent, accurate provenance trail.
- **Precision at this query: 0.5 effective distinct facts per K slot consumed (for the duplicated pair). Co_retrieval graph is maximally accurate.**

**Case 3: Deferred compaction (Q013 nightly pass)**

- Before the compaction pass runs (lag: 0–24 hours): behavior is identical to Case 2.
- After the compaction pass: behavior converges to Case 1, but with a better merge decision (cluster-wide optimal canonical selection, audit trail, reversible).
- **Precision trajectory: starts at Case 2, converges to Case 1 within 24 hours. Co_retrieval signal is preserved during the window and folded at merge time with cluster-wide knowledge.**

**Information-theoretic verdict**: The compaction pass (Case 3) dominates both Case 1 and Case 2:
- It achieves Case 1's precision gains (after the pass completes).
- It avoids Case 1's irreversibility risk at write time.
- It has a bounded precision cost: one context slot wasted per near-duplicate pair, for at most 24 hours.
- It preserves the co_retrieval signal at full accuracy during the window, then folds it with cluster-wide context.

The information-theoretically optimal strategy minimizes the integral of precision loss over time. For a query that arrives BEFORE the compaction pass, Case 1 (write-time merge) has zero slot waste. But Case 1 commits a potentially incorrect merge permanently. Case 3's 24-hour slot waste is a bounded, one-time cost that buys permanent correctness.

**The exception**: write-time merge IS correct at a higher threshold — specifically, ≥ 0.99 for qwen3-embedding:0.6b. At 0.99, even the compressed embedding model reliably places only linguistic near-duplicates (same sentence, different punctuation or whitespace). The merge decision can be made with high confidence at write time, and the co_retrieval neighborhood overlap is effectively 100%.

### The Architecture Thread B Would Accept

**Layered deduplication with threshold stratification:**

1. **Layer 1 — Write-time hard duplicate guard (threshold ≥ 0.99)**: At content-hash identity or cosine ≥ 0.99, reject the incoming memory at write time and update the existing record (max importance, union tags, update timestamp). This catches byte-level and near-byte-level duplicates that are unambiguously the same memory. No co_retrieval information is lost because the two records have had zero independent retrieval history by definition (the new memory has zero co_retrieval_count at write time).

2. **Layer 2 — Write-time near-duplicate flag (threshold 0.95–0.99)**: Do not merge. Instead, insert a `near_duplicate_candidate` row in a PostgreSQL staging table: `(memory_a_id, memory_b_id, similarity_score, created_at)`. This costs one row insert, preserves full information, and feeds the compaction pass.

3. **Layer 3 — Compaction pass (Q013 nightly, threshold ≥ 0.88 for cluster detection, ≥ 0.95 for merge action)**: The compaction pass reads the `near_duplicate_candidate` staging table, evaluates clusters, selects the canonical winner (longest content, max importance, union tags), tombstones the loser, and folds the co_retrieval_count using the correct relational fold: sum the `co_retrieval_count` columns AND reroute all co_retrieval edges from the loser UUID to the winner UUID. The loser's co_retrieval edges become winner's edges via an UPDATE: `UPDATE co_retrievals SET from_memory_id = :winner_id WHERE from_memory_id = :loser_id`. This preserves the full co_retrieval graph structure — the loser's neighborhoods are not lost, they are re-attributed to the winner.

4. **Layer 4 — Query-time deduplication (retrieval result collapsing)**: If two results in the top-K have cosine similarity ≥ 0.95 to each other, collapse them before returning to the caller: return the higher-importance record, merge their content in the returned payload, and log the collapse event. This is a retrieval-time safety net that catches any pair that Layers 1–3 missed, at zero store modification cost.

**Co_retrieval edge re-attribution (the key missing piece)**: The correct merge operation for co_retrievals is not a count fold but an edge re-attribution. When loser B is tombstoned and winner A absorbs it:

```sql
-- Reroute B's outgoing edges to A (B's neighbors become A's neighbors)
INSERT INTO co_retrievals (from_memory_id, to_memory_id, tenant_id, relationship_type, last_reinforced_epoch, weight, co_retrieval_count)
SELECT :winner_id, to_memory_id, tenant_id, relationship_type, last_reinforced_epoch, weight, co_retrieval_count
FROM co_retrievals
WHERE from_memory_id = :loser_id
  AND to_memory_id != :winner_id  -- avoid self-loop
ON CONFLICT (tenant_id, from_memory_id, to_memory_id) DO UPDATE
    SET weight              = co_retrievals.weight + EXCLUDED.weight,
        co_retrieval_count  = co_retrievals.co_retrieval_count + EXCLUDED.co_retrieval_count,
        relationship_type   = CASE WHEN EXCLUDED.relationship_type = 'FRESH' THEN 'FRESH' ELSE co_retrievals.relationship_type END;

-- CASCADE DELETE will clean up B's rows when B is tombstoned (ON DELETE CASCADE from Q115 schema)
```

This preserves the topological structure of the co_retrieval graph: B's neighborhoods become A's neighborhoods. The co_retrieval_count for each specific pair (A, former_B_neighbor) correctly represents the combined co-retrieval history of both A's direct events AND B's events with that neighbor — which is semantically correct because A and B encoded the same fact.

## Ideas Extracted

### Idea: threshold-stratified-deduplication-pipeline
- **Novelty**: 0.78
- **Evidence**: 0.68
- **Feasibility**: 0.80
- **Description**: Three-layer deduplication at different cosine thresholds with different actions: (1) ≥ 0.99 write-time reject (byte-level and linguistic-identical duplicates), (2) 0.95–0.99 write-time flag into a `near_duplicate_candidate` staging table (no immediate action, feeds compaction), (3) nightly compaction pass that reads candidates, validates clusters, selects winner, tombstones loser, and re-attributes co_retrieval edges. The stratification matches threshold confidence to action irreversibility: the most destructive action (physical merge) is reserved for the highest-confidence threshold. For qwen3-embedding:0.6b with ~25 effective dimensions, the 0.99 threshold is the reliable identity boundary; the 0.95–0.99 band needs compaction-pass review.

### Idea: co-retrieval-edge-reattribution-merge
- **Novelty**: 0.82
- **Evidence**: 0.60
- **Feasibility**: 0.75
- **Description**: When two near-duplicate memories are merged, the correct operation for the co_retrievals graph is not a count fold but a full edge re-attribution: reroute all outgoing edges from the loser UUID to the winner UUID using an INSERT ... ON CONFLICT DO UPDATE that sums weights and counts. This preserves the topological structure of the co_retrieval graph — the loser's retrieval neighborhoods are not destroyed, they are attributed to the winner. The ON DELETE CASCADE foreign key from Q115 then handles cleanup automatically when the loser is tombstoned. This is the correct information-preserving merge because the co_retrieval signal is relational (pairwise events), not scalar (aggregate counts), and the re-attribution operation accurately represents what the merged record's co_retrieval history should be: all events that involved either of the two semantically identical records.

### Idea: query-time-result-deduplication-safety-net
- **Novelty**: 0.70
- **Evidence**: 0.65
- **Feasibility**: 0.90
- **Description**: Add a post-retrieval deduplication step in the FastAPI retrieve endpoint: after Qdrant returns top-K results, check all pairs for pairwise cosine similarity ≥ 0.95. Collapse any near-duplicate pair to the higher-importance member, merging content in the returned payload. This is a zero-cost safety net (no store writes, no irreversible decisions) that catches all near-duplicates regardless of which layer of the write/compaction pipeline missed them. At K=10 with R=0.40 redundancy, expected near-duplicate pairs per query: ~1.6 pairs — the check requires ~45 pairwise comparisons on 1,024-dim vectors (or 25-dim after PCA from Q013), which completes in microseconds. This layer also handles the model-version-bump scenario: if an embedding model update changes the similarity landscape, query-time deduplication degrades gracefully to slightly looser collapse rather than triggering incorrect write-time merges.

### Idea: near-duplicate-candidate-staging-table
- **Novelty**: 0.65
- **Evidence**: 0.70
- **Feasibility**: 0.92
- **Description**: A `near_duplicate_candidates` PostgreSQL table `(id, memory_a_id, memory_b_id, similarity_score, detected_at, status)` serves as the bridge between write-time near-duplicate detection and compaction-pass merge action. Write-time detection at threshold 0.95–0.99 inserts a candidate row (no store modification). The compaction pass reads candidates, validates them against the current embedding index (similarity may have changed if one memory was edited), groups them into clusters, and performs the merge-or-reject decision with a full audit trail. The `status` column (PENDING, MERGED, REJECTED) provides an observable queue that the dashboard can display, giving Tim visibility into the deduplication pipeline without any irreversible action occurring automatically.

## Recommendations

1. **Do not implement write-time physical merge at the 0.97 threshold.** The qwen3-embedding:0.6b model has ~25 effective dimensions; the 0.97 threshold is not reliably in the semantic-identity zone for this model. The merge operation is irreversible and destroys the co_retrieval graph's relational provenance. The cost of a false merge (permanent graph distortion) exceeds the cost of a false non-merge (one context slot wasted per query for ≤24 hours until compaction runs).

2. **Implement write-time hard-duplicate rejection at threshold ≥ 0.99 only.** At 0.99, even qwen3-embedding:0.6b reliably identifies near-identical sentences. Reject incoming memories at this threshold: update the existing record (max importance, union tags), skip the insert. This addresses the byte-level duplicate case with no information loss (new memory has zero co_retrieval history).

3. **Implement the `near_duplicate_candidates` staging table and feed it from write-time detection in the 0.95–0.99 band.** This creates an observable pipeline without irreversible actions. The staging table costs approximately 100 bytes per candidate row — negligible.

4. **Implement the Q013 compaction pass with co_retrieval edge re-attribution (not count fold).** The merge operation for co_retrievals must re-attribute the loser's edges to the winner via INSERT ... ON CONFLICT DO UPDATE on the `co_retrievals` table. ON DELETE CASCADE then handles the loser's cleanup atomically. This is the only merge operation that preserves the full topological information of the co_retrieval graph.

5. **Add query-time result deduplication as a retrieval-path safety net.** After Qdrant returns top-K, check pairwise similarity ≥ 0.95 among the results. Collapse near-duplicate pairs before returning to the caller. This provides immediate precision improvement without waiting for the compaction pass, at zero store modification cost.

6. **The "co-exist with a STALE co_retrievals edge" option is architecturally incoherent in the Q115 schema.** The FRESH/STALE partition in co_retrievals encodes edge age (reinforcement recency), not semantic similarity. Creating a STALE edge between near-duplicates to signal their relationship would pollute the reinforcement-age semantics of the FRESH/STALE distinction. This option should not be implemented.
