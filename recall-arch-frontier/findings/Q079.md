# Finding: Q079 — Bidirectional Epoch Continuity Guard: Dual-Machine Threshold Calibration Reveals Single-Threshold Indistinguishability; Dual-Signal Detection Required

**Question**: Implement the bidirectional epoch continuity guard identified in Q074. Specifically: (1) define the `EPOCH_GAP_ALARM_THRESHOLD` constant and calibrate it safely for dual-machine (casaclaude + proxyclaude) usage where epoch advances at 2× single-machine rate, (2) implement `check_epoch_continuity_bidirectional()` as an extension of Q067's unidirectional function, (3) specify the `/epoch-status` FastAPI endpoint response schema and implementation, (4) define how the `PG_RESTORE_SUSPECTED` alarm is delivered (Discord webhook, FastAPI degraded status, or both), and (5) determine whether the existing Q069 deployment sequence Step 9 (epoch guard) needs to be updated to use the bidirectional version.
**Question Type**: IMPLEMENTATION
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Distributed systems (epoch continuity protocols, dual-node write ordering), operational monitoring (alarm calibration, false-alarm rate)

## Threshold Calibration

### Single-machine steady-state gap (baseline)

Under single-machine operation, the write ordering per Q067 is: `INCR recall:global_epoch` fires at session end, then `UPDATE memories SET last_access_epoch = :current_epoch` commits. The gap between redis_epoch and pg_max at any point in time is the number of sessions that have INCRed Redis but not yet committed their PG UPDATE. Under reliable network conditions this is 0 or 1 (the INCR and UPDATE are in the same session-end hook call, separated by milliseconds). In steady state: **single-machine gap = 0 to 1**.

### Dual-machine steady-state gap (normal operation)

Casaclaude and proxyclaude both write to the same CasaOS PostgreSQL (192.168.50.19) and the same Redis instance. Each machine independently INCRs `recall:global_epoch` at session end and then writes `last_access_epoch` to PG. Under normal reliable operation, both machines commit their PG writes promptly.

The concurrent session scenario: casaclaude and proxyclaude each start a session simultaneously. Both INCR redis_epoch (redis_epoch advances by 2). Both write last_access_epoch to PG within the same hook call. At any point in time, at most 2 sessions (one per machine) are in the window between their Redis INCR and their PG commit. **Dual-machine steady-state gap = 0 to 2**.

### Worst-case legitimate gap: single-machine PG write degradation

This is the critical calibration scenario. Casaclaude experiences degraded PG connectivity — its Redis INCRs succeed (Redis is on a different host or the Redis connection is more resilient) but its PG UPDATE calls to CasaOS time out or fail silently. Proxyclaude runs normally (both Redis INCR and PG UPDATE succeed on every session).

For N casaclaude sessions with failed PG writes and M proxyclaude sessions running normally during the same period:
- redis_epoch increases by N + M (both machines INCR successfully)
- pg_max increases by M (only proxyclaude sessions write successfully to PG)
- **Legitimate gap = N** (the number of failed casaclaude PG writes)

At 10 sessions/day dual-machine (approximately 5 each), if casaclaude's PG writes fail for 6 hours (half a day = ~25 casaclaude sessions with failed PG writes), the legitimate gap reaches N ≈ 25. At 30 casaclaude sessions with failed PG writes, gap ≈ 30.

### The threshold indistinguishability problem

The verdict criteria state: BREAKTHROUGH if the legitimate steady-state gap is larger than tau (meaning a simple threshold cannot distinguish PG-restore from normal dual-machine lag).

| Scenario | redis_epoch | pg_max | gap | Alarm? |
|----------|------------|--------|-----|--------|
| True steady state (both machines, reliable writes) | 500 | 499 | 1 | No |
| PG-restore: backup taken at epoch 100, redis at 500 | 500 | 100 | 400 | Yes (obvious) |
| PG-restore: backup taken at epoch 470, redis at 500 | 500 | 470 | 30 | Depends on threshold |
| Single-machine PG degradation: 30 casaclaude sessions lost | 500 | 470 | 30 | Same gap as PG-restore above |

A PG restore from a backup taken 30 sessions ago produces an identical epoch gap signature to single-machine PG write degradation lasting 30 sessions. Both produce gap = 30. A simple threshold at T cannot distinguish them.

### Tau relevance

Tau = 20 sessions. If gap > tau, the LLPC working-set filter `WHERE last_access_epoch >= (redis_epoch - tau)` can no longer admit memories from before the gap. Memories written before the gap event (backup era or pre-degradation era) are invisible.

For a PG restore with gap = 30 > tau = 20: retrieval returns empty (all backup-era memories have last_access_epoch <= 470, threshold = 500-20 = 480, so epochs 471-479 are in range but those are sessions that occurred after the backup — PG does not have them because they were written after the backup snapshot. The backup-era memories at epochs <= 470 are below the 480 threshold. Zero rows qualify.)

For single-machine PG degradation with gap = 30: the 30 casaclaude sessions have `last_access_epoch` NOT written. Their memories effectively have last_access_epoch = 0 (default) or the value from the previous session. Proxyclaude's memories are written normally. Retrieval is PARTIALLY degraded, not completely empty — proxyclaude's memories from epochs 481-500 pass the filter. The casaclaude memories from those 30 sessions are invisible.

**This partial-vs-complete emptiness is the distinguishing signal** a simple threshold cannot capture. The threshold cannot distinguish the two cases by gap alone, but the retrieval emptiness pattern differs.

### Calibrated threshold recommendation

Given the indistinguishability finding, the threshold must be set to balance two competing risks:

- **Set threshold too low (e.g., T = 25)**: False alarms during legitimate single-machine PG degradation of 25+ sessions. A 6-hour PG write outage on one machine triggers a PG_RESTORE alarm. This is a false positive but also a true signal of a problem (PG writes are failing).
- **Set threshold too high (e.g., T = 100)**: PG restores with gaps between 21 and 99 sessions cause complete retrieval silence without alarming. The system operates silently broken for an extended period.

**Recommended calibration**:

```python
EPOCH_GAP_ALARM_THRESHOLD = 50
```

Rationale:
- Above tau (20): any gap > 20 means some memories are inaccessible to the working-set filter.
- Above worst-case short-burst PG degradation: 50 sessions at dual-machine rate ≈ 5 days of one machine's PG writes failing completely. A 5-day silent PG write failure is itself a P0 event that should generate its own write-failure alarms from the hook error path.
- Below catastrophic PG restore distance: restoring from a backup more than 50 sessions old (approximately 5 days at 10 sessions/day) correctly triggers the alarm.

**Critical limitation**: a PG restore from a backup taken 1-5 days ago (gap 10-49) combined with tau = 20 means retrieval may be partially or fully empty without triggering the alarm. The guard with T = 50 does NOT catch near-term PG restores with gaps between 21 and 49. This is acceptable if the operator ensures backups are taken at least every 10 sessions (1 day at current usage rate). Tighter backup cadence makes the gap from any restore smaller, ensuring the alarm threshold is hit before the retrieval window is fully dark.

**The fundamental finding**: a single threshold cannot provide both high sensitivity (low false alarm rate) and high specificity (catch all PG restores that break retrieval). The correct architectural response is dual-signal detection described in the Implementation section below.

## Implementation

### Complete `check_epoch_continuity_bidirectional()` function

```python
import asyncio
import logging
from enum import Enum
from dataclasses import dataclass
from typing import Optional
import httpx

logger = logging.getLogger(__name__)

# Calibrated for dual-machine (casaclaude + proxyclaude) usage.
# Above tau (20): any gap here means some memories are invisible to the working-set filter.
# At 10 sessions/day dual-machine, T=50 represents ~5 days of sustained single-machine PG failure.
# See Q079 for full calibration analysis.
EPOCH_GAP_ALARM_THRESHOLD: int = 50

# Tau must match the value used in the LLPC routing query.
# If tau changes, update both the routing query AND this constant.
EPOCH_TAU: int = 20


class EpochStatus(str, Enum):
    OK = "OK"
    REDIS_RESET_RECOVERED = "REDIS_RESET_RECOVERED"
    PG_RESTORE_SUSPECTED = "PG_RESTORE_SUSPECTED"
    # Gap exceeds tau but is below EPOCH_GAP_ALARM_THRESHOLD.
    # Retrieval is partially degraded; likely single-machine PG write failures.
    PARTIAL_DEGRADATION = "PARTIAL_DEGRADATION"


@dataclass
class EpochCheckResult:
    redis_epoch: int
    pg_max: int
    gap: int
    status: EpochStatus
    retrieval_healthy: bool      # True iff gap <= tau (working-set filter can admit memories)
    alarm_active: bool           # True iff gap > EPOCH_GAP_ALARM_THRESHOLD
    recovered_from: Optional[int] = None  # Set when REDIS_RESET_RECOVERED: the old redis_epoch value


async def check_epoch_continuity_bidirectional(
    redis_client,
    pg_pool,
) -> EpochCheckResult:
    """
    Bidirectional epoch continuity guard. Called once per session start.

    Direction 1 (Q067 case): pg_max > redis_epoch
        Redis was reset (FLUSHALL, container restart without persistence).
        Auto-recovers by setting redis_epoch = pg_max + 1.
        Cost: 0.25ms (hot buffer pool).

    Direction 2 (Q074 case): redis_epoch - pg_max > EPOCH_GAP_ALARM_THRESHOLD
        PostgreSQL was restored from an old backup, OR single-machine PG write path
        has been degraded for an extended period.
        Cannot auto-recover safely (requires operator decision).
        Emits PG_RESTORE_SUSPECTED alarm. Returns current redis_epoch unchanged.

    Direction 2 partial (gap > tau but <= threshold):
        Gap is large enough to hide some memories from the working-set filter,
        but below the alarm threshold. Likely transient single-machine write lag.
        Returns PARTIAL_DEGRADATION status. No alarm emitted.

    Returns:
        EpochCheckResult with validated epoch to use for this session.
    """
    redis_epoch_raw, pg_max_raw = await asyncio.gather(
        redis_client.get("recall:global_epoch"),
        pg_pool.fetchval("SELECT MAX(last_access_epoch) FROM memories"),
    )

    redis_epoch = int(redis_epoch_raw or 0)
    pg_max = int(pg_max_raw or 0)
    gap = redis_epoch - pg_max

    # Direction 1: Redis reset (Q067 case).
    # pg_max > redis_epoch means Redis epoch is behind the highest value in PG.
    # This is unambiguous: under normal operation redis_epoch >= pg_max always holds
    # (INCR fires before PG UPDATE in the write ordering).
    if pg_max > redis_epoch:
        old_epoch = redis_epoch
        restored_epoch = pg_max + 1
        await redis_client.set("recall:global_epoch", restored_epoch)
        logger.warning(
            "Epoch continuity: Redis reset detected. "
            "redis_epoch=%d pg_max=%d restored_to=%d",
            old_epoch, pg_max, restored_epoch,
        )
        return EpochCheckResult(
            redis_epoch=restored_epoch,
            pg_max=pg_max,
            gap=restored_epoch - pg_max,
            status=EpochStatus.REDIS_RESET_RECOVERED,
            retrieval_healthy=True,
            alarm_active=False,
            recovered_from=old_epoch,
        )

    # gap = redis_epoch - pg_max >= 0 from here.

    # Direction 2: PG restore suspected (Q074 case).
    # Gap exceeds alarm threshold. Cannot auto-recover.
    if gap > EPOCH_GAP_ALARM_THRESHOLD:
        logger.error(
            "Epoch continuity: PG_RESTORE_SUSPECTED. "
            "redis_epoch=%d pg_max=%d gap=%d threshold=%d tau=%d retrieval_healthy=%s",
            redis_epoch, pg_max, gap, EPOCH_GAP_ALARM_THRESHOLD, EPOCH_TAU,
            gap <= EPOCH_TAU,
        )
        return EpochCheckResult(
            redis_epoch=redis_epoch,
            pg_max=pg_max,
            gap=gap,
            status=EpochStatus.PG_RESTORE_SUSPECTED,
            retrieval_healthy=(gap <= EPOCH_TAU),
            alarm_active=True,
        )

    # Direction 2 partial: gap > tau but <= threshold.
    # Some memories from before the gap are invisible to the working-set filter.
    # This is likely transient single-machine PG write lag, not a full restore.
    if gap > EPOCH_TAU:
        logger.warning(
            "Epoch continuity: PARTIAL_DEGRADATION. "
            "redis_epoch=%d pg_max=%d gap=%d tau=%d "
            "(memories with last_access_epoch < %d are invisible)",
            redis_epoch, pg_max, gap, EPOCH_TAU, redis_epoch - EPOCH_TAU,
        )
        return EpochCheckResult(
            redis_epoch=redis_epoch,
            pg_max=pg_max,
            gap=gap,
            status=EpochStatus.PARTIAL_DEGRADATION,
            retrieval_healthy=False,
            alarm_active=False,
        )

    # Steady state: gap <= tau. Working-set filter can admit memories from [redis_epoch - tau, redis_epoch].
    return EpochCheckResult(
        redis_epoch=redis_epoch,
        pg_max=pg_max,
        gap=gap,
        status=EpochStatus.OK,
        retrieval_healthy=True,
        alarm_active=False,
    )
```

### Caller interface change from Q067

Q067's `check_epoch_continuity()` returned a single `int` (the validated epoch). The bidirectional version returns `EpochCheckResult`. The caller (session-start hook or FastAPI session initialization) must be updated:

```python
# Q067 (old):
current_epoch = await check_epoch_continuity(redis_client, pg_pool)

# Q079 (new):
epoch_result = await check_epoch_continuity_bidirectional(redis_client, pg_pool)
current_epoch = epoch_result.redis_epoch

# Store result in app state for /epoch-status endpoint:
app.state.last_epoch_check = epoch_result

# Trigger alarm delivery if needed:
if epoch_result.alarm_active:
    await deliver_pg_restore_alarm(epoch_result)
```

### Application state for caching

The guard runs once per session start (per Q067's design — memoized for the session duration). The result is stored in FastAPI application state so the `/epoch-status` endpoint can serve it without additional I/O:

```python
# In FastAPI app startup:
app.state.last_epoch_check: Optional[EpochCheckResult] = None
app.state.last_epoch_check_timestamp: Optional[float] = None
```

### Endpoint implementation

```python
import time
from fastapi import APIRouter
from datetime import datetime, timezone

router = APIRouter()


def _build_diagnosis(result: EpochCheckResult) -> str:
    """Build a human-readable diagnosis string for operator use."""
    if result.status == EpochStatus.OK:
        return (
            f"Epoch continuity healthy. Gap {result.gap} <= tau {EPOCH_TAU}. "
            f"Working-set filter admits memories from epoch "
            f"{result.redis_epoch - EPOCH_TAU} to {result.redis_epoch}."
        )
    elif result.status == EpochStatus.REDIS_RESET_RECOVERED:
        return (
            f"Redis epoch reset detected and auto-recovered. "
            f"Old redis_epoch={result.recovered_from}, "
            f"restored to pg_max+1={result.redis_epoch}. "
            f"System is now healthy. Monitor for repeat resets."
        )
    elif result.status == EpochStatus.PARTIAL_DEGRADATION:
        invisible_below = result.redis_epoch - EPOCH_TAU
        return (
            f"Gap {result.gap} > tau {EPOCH_TAU}. "
            f"Memories with last_access_epoch < {invisible_below} are invisible to retrieval. "
            f"Likely: single-machine PG write failures ({result.gap} sessions unwritten). "
            f"Check CasaOS PG write logs. Gap below alarm threshold ({EPOCH_GAP_ALARM_THRESHOLD}), "
            f"no alarm sent."
        )
    elif result.status == EpochStatus.PG_RESTORE_SUSPECTED:
        return (
            f"ALARM: Gap {result.gap} > threshold {EPOCH_GAP_ALARM_THRESHOLD}. "
            f"PostgreSQL may have been restored from a backup taken at epoch ~{result.pg_max}. "
            f"Retrieval {'is empty (gap > tau)' if not result.retrieval_healthy else 'partially degraded'}. "
            f"Operator action required. Options: "
            f"(A) redis-cli SET recall:global_epoch {result.pg_max + 1} (accept data loss, restore continuity); "
            f"(B) restore PG from a more recent backup; "
            f"(C) temporarily set tau to {result.gap + 100} for emergency bypass."
        )
    return "Unknown status."


@router.get("/epoch-status", response_model=EpochStatusResponse)
async def get_epoch_status(request: Request) -> EpochStatusResponse:
    """
    Returns the current epoch continuity state as observed by the bidirectional guard.

    The guard result is cached from the last session-start check. This endpoint
    performs zero additional I/O — it reads from app state only.

    Use this endpoint for:
    - External monitoring (uptime-kuma, cron health checks)
    - Manual diagnosis when retrieval seems empty
    - Verifying recovery after operator intervention

    retrieval_healthy=false means the LLPC working-set filter cannot admit
    all memories. The system may return empty results.
    """
    result: Optional[EpochCheckResult] = request.app.state.last_epoch_check
    checked_at_ts: Optional[float] = request.app.state.last_epoch_check_timestamp

    if result is None:
        # Guard has not yet run (no session has started since API restart).
        return EpochStatusResponse(
            redis_epoch=0,
            pg_max=0,
            gap=0,
            status="NO_DATA",
            retrieval_healthy=True,
            alarm_active=False,
            tau=EPOCH_TAU,
            epoch_gap_alarm_threshold=EPOCH_GAP_ALARM_THRESHOLD,
            checked_at=None,
            checked_at_epoch_age_seconds=None,
            recovered_from=None,
            diagnosis="No epoch check has run since API startup. Start a session to initialize.",
        )

    checked_at_dt = datetime.fromtimestamp(checked_at_ts, tz=timezone.utc) if checked_at_ts else None
    age_seconds = (time.time() - checked_at_ts) if checked_at_ts else None

    return EpochStatusResponse(
        redis_epoch=result.redis_epoch,
        pg_max=result.pg_max,
        gap=result.gap,
        status=result.status.value,
        retrieval_healthy=result.retrieval_healthy,
        alarm_active=result.alarm_active,
        tau=EPOCH_TAU,
        epoch_gap_alarm_threshold=EPOCH_GAP_ALARM_THRESHOLD,
        checked_at=checked_at_dt,
        checked_at_epoch_age_seconds=age_seconds,
        recovered_from=result.recovered_from,
        diagnosis=_build_diagnosis(result),
    )
```

## Alarm Delivery

### Decision: both Discord webhook AND FastAPI degraded status

Discord webhook and the `/epoch-status` degraded status serve different monitoring surfaces and different urgency modes. Both should be implemented. They are not mutually exclusive and add no meaningful complexity.

**Discord webhook** (immediate push notification):
- Delivers the alarm to Tim within seconds of the first session that detects the condition.
- Does not require Tim to be actively checking a dashboard or health URL.
- Already used for Recall error reporting — the webhook URL and HTTP client are already configured.
- Appropriate for P0 alarms (complete retrieval silence).

**FastAPI `/epoch-status`** (pull-based monitoring):
- Enables uptime-kuma, cron jobs, or curl health checks to detect the condition.
- Provides structured diagnostic data (gap, pg_max, options) for investigation.
- Appropriate for lower-severity conditions (PARTIAL_DEGRADATION) that do not warrant immediate Discord noise.
- The endpoint response can be embedded in CasaOS monitoring dashboards.

**Alarm routing by severity:**

| Status | Discord webhook | /epoch-status |
|--------|----------------|---------------|
| OK | No | retrieval_healthy=true |
| REDIS_RESET_RECOVERED | Yes (informational) | retrieval_healthy=true |
| PARTIAL_DEGRADATION | No (avoid noise) | retrieval_healthy=false |
| PG_RESTORE_SUSPECTED | Yes (P0 alert) | retrieval_healthy=false, alarm_active=true |

### Discord webhook implementation

```python
import httpx
import os
from datetime import datetime, timezone

DISCORD_WEBHOOK_URL = os.environ.get("RECALL_DISCORD_WEBHOOK_URL")


async def deliver_pg_restore_alarm(result: EpochCheckResult) -> None:
    """
    Sends a Discord webhook notification for PG_RESTORE_SUSPECTED and
    REDIS_RESET_RECOVERED epoch continuity events.

    Does NOT send for PARTIAL_DEGRADATION (below alarm threshold).
    Fire-and-forget: failures are logged but do not propagate.
    """
    if not DISCORD_WEBHOOK_URL:
        logger.warning(
            "RECALL_DISCORD_WEBHOOK_URL not set — epoch alarm not delivered to Discord."
        )
        return

    if result.status == EpochStatus.PG_RESTORE_SUSPECTED:
        color = 0xFF0000  # Red
        title = "RECALL ALARM: PG_RESTORE_SUSPECTED"
        description = (
            f"Epoch gap **{result.gap}** exceeds alarm threshold **{EPOCH_GAP_ALARM_THRESHOLD}**.\n"
            f"redis_epoch=**{result.redis_epoch}**, pg_max=**{result.pg_max}**\n"
            f"Retrieval is **{'EMPTY' if not result.retrieval_healthy else 'PARTIAL'}** "
            f"(gap {'>' if result.gap > EPOCH_TAU else '<='} tau={EPOCH_TAU}).\n\n"
            f"**Immediate actions:**\n"
            f"• Check: `curl http://192.168.50.19:8200/epoch-status`\n"
            f"• Option A (accept loss): `redis-cli SET recall:global_epoch {result.pg_max + 1}`\n"
            f"• Option B: Restore PG from a more recent CasaOS backup\n"
            f"• Option C (bypass): Set tau to {result.gap + 100} temporarily\n\n"
            f"See Q074 and Q079 findings for full recovery procedure."
        )
    elif result.status == EpochStatus.REDIS_RESET_RECOVERED:
        color = 0xFFAA00  # Amber
        title = "RECALL INFO: Redis Epoch Reset Auto-Recovered"
        description = (
            f"Redis epoch was reset to {result.recovered_from} (possibly FLUSHALL or container restart).\n"
            f"Auto-recovered to pg_max+1=**{result.redis_epoch}**.\n"
            f"System is **healthy**. Monitor for repeat resets — repeated resets indicate "
            f"Redis persistence is not configured."
        )
    else:
        return  # No Discord delivery for OK or PARTIAL_DEGRADATION

    payload = {
        "username": "Recall Monitor",
        "embeds": [
            {
                "title": title,
                "description": description,
                "color": color,
                "timestamp": datetime.now(tz=timezone.utc).isoformat(),
                "footer": {"text": "Recall FastAPI — 192.168.50.19:8200"},
            }
        ],
    }

    try:
        async with httpx.AsyncClient(timeout=5.0) as client:
            response = await client.post(DISCORD_WEBHOOK_URL, json=payload)
            response.raise_for_status()
            logger.info("Epoch alarm delivered to Discord. status=%s gap=%d", result.status, result.gap)
    except Exception as exc:
        # Alarm delivery failure must not crash the session-start path.
        logger.error("Failed to deliver epoch alarm to Discord: %s", exc)
```

## Deployment Sequence Update

### Does Q069 Step 9 need updating?

**Yes.** Q069 Step 9 deploys the Q062 epoch-clock, which implicitly includes the Q067 unidirectional guard (`check_epoch_continuity()`). The bidirectional guard (`check_epoch_continuity_bidirectional()`) replaces `check_epoch_continuity()` in the same call site. Step 9 must be updated to deploy the bidirectional version.

The update does not require schema changes. The bidirectional guard uses the same two queries already executed by Q067's guard (`GET recall:global_epoch` and `SELECT MAX(last_access_epoch) FROM memories`). The only changes are:

## Ideas Extracted

### Idea: dual-signal-epoch-alarm-distinguisher
- **Novelty**: 0.81 — The finding that a single gap threshold cannot distinguish PG-restore from single-machine PG write degradation under dual-machine usage is not addressed in Q074 or Q067. The proposed resolution — combining gap threshold with PG write success rate monitoring as a second signal — creates a two-dimensional detection space (gap AND write failure rate) rather than a single-dimensional threshold. This distinguishes the two failure modes: PG restore produces gap WITHOUT write failures (writes succeed, they're just writing new memories on top of the restored PG); PG write degradation produces gap WITH write failures (the INCR succeeds but the UPDATE fails). This second-signal distinction is novel in the context of AI memory system epoch monitoring.
- **Evidence**: 0.68 — The two-signal approach is analogous to distributed tracing systems that separate error rate from latency as distinct health signals. The write success/failure rate is already observable from the session-end hook error log — it is not a new metric, just a new combination. The distinction (restore produces no write errors, degradation produces write errors) follows from the operational definition of each failure mode and can be verified by inspection of the hook error log.
- **Feasibility**: 0.75 — Requires tracking PG write success/failure rate as a second application state variable. The session-end hook already logs write failures; adding a rolling counter (e.g., failed writes in the last 100 sessions) to FastAPI app state is a moderate implementation effort. The `/epoch-status` endpoint would then include `pg_write_failure_rate_recent` as an additional field, enabling automatic classification: high gap + high failure rate = PG write degradation; high gap + zero failure rate = PG restore.
- **Description**: Extend the epoch alarm with a second diagnostic signal: recent PG write failure rate. When the bidirectional guard detects a large gap, the alarm classification should be: (1) if recent PG write failures > 0 and gap is increasing with each session, classify as SINGLE_MACHINE_PG_DEGRADATION (the gap is growing because writes are failing); (2) if recent PG write failures = 0 and gap appeared suddenly (was small in the previous check), classify as PG_RESTORE_SUSPECTED (writes succeed, but pg_max jumped backward). This two-signal classification reduces false positives for the PG_RESTORE_SUSPECTED alarm and provides more actionable diagnosis — degradation requires fixing the write path, restore requires the option A/B/C decision.

### Idea: uptime-kuma-epoch-continuity-probe
- **Novelty**: 0.67 — Wiring an uptime-kuma or equivalent self-hosted monitoring probe against `/epoch-status` to check `retrieval_healthy` and `alarm_active` provides continuous epoch continuity monitoring without Tim needing to manually investigate. The pattern of using structured health endpoint responses as machine-readable monitoring targets is standard (Kubernetes readiness probes, Consul health checks), but its application to AI memory system epoch continuity monitoring is not observed in current systems.
- **Evidence**: 0.72 — Uptime-kuma is deployed on Tim's homelab (inferred from CasaOS infrastructure pattern — uptime-kuma is a common CasaOS app store install). The `/epoch-status` endpoint response is JSON with a stable schema. A custom HTTP keyword monitor in uptime-kuma checking for `"retrieval_healthy": true` in the response body provides a one-setup alarm that persists across sessions. The alternative (log file monitoring) requires additional tooling (logwatch, promtail) and is more brittle.
- **Feasibility**: 0.90 — Uptime-kuma HTTP keyword monitor: URL = `http://192.168.50.19:8200/epoch-status`, method = GET, keyword to check = `"retrieval_healthy": true`, check interval = 5 minutes. If the keyword is absent (retrieval degraded), uptime-kuma triggers its configured notification (Discord, email). This is a 2-minute configuration change with no code. The only prerequisite is that `/epoch-status` is implemented (Q079 above) and accessible from the uptime-kuma container.
- **Description**: Configure an uptime-kuma HTTP keyword monitor against `http://192.168.50.19:8200/epoch-status` that checks for `"retrieval_healthy": true`. Any epoch condition that degrades retrieval (gap > tau, regardless of whether it triggers the Discord alarm from the guard itself) will be caught by the uptime-kuma probe within 5 minutes and forwarded to Discord. This provides a redundant monitoring path that catches PARTIAL_DEGRADATION conditions that the guard intentionally does not alarm on (gap > tau but <= 50). It also catches cases where the guard ran but the Discord delivery failed.

## Key Finding

The threshold calibration for `EPOCH_GAP_ALARM_THRESHOLD` under dual-machine operation reveals a fundamental indistinguishability problem: a PG restore from a backup taken 30 sessions ago produces an identical epoch gap signature (gap = 30) to single-machine PG write degradation lasting 30 sessions. Since tau = 20 and a gap of 30 causes complete invisibility of pre-gap memories, both conditions damage retrieval quality identically — but they have different recovery procedures (epoch retraction vs. fixing the write path). A simple threshold cannot distinguish the two cases. The verdict is BREAKTHROUGH because the legitimate steady-state gap under dual-machine usage with partial PG write degradation can exceed tau (the working-set window boundary), meaning the threshold cannot simultaneously avoid false alarms from routine write failures AND catch all PG restores that break retrieval.

The recommended threshold is `EPOCH_GAP_ALARM_THRESHOLD = 50`, calibrated to represent approximately 5 days of sustained single-machine PG write failure at current usage rates (10 sessions/day). This threshold eliminates false alarms from short-duration write failures while catching large-gap PG restores (backup age > 5 days). The gap window between tau (20) and the threshold (50) — representing restores from backups 2-5 days old — is a blind spot: these restores cause partial or complete retrieval degradation without triggering the Discord alarm. The mitigation is tighter backup cadence (daily or more frequent CasaOS PG snapshots) and secondary monitoring via uptime-kuma's `/epoch-status` keyword probe, which catches `retrieval_healthy: false` regardless of whether the gap exceeds the alarm threshold.

The correct alarm delivery uses both Discord webhook (immediate push for `PG_RESTORE_SUSPECTED` and `REDIS_RESET_RECOVERED`) and the `/epoch-status` FastAPI endpoint (pull-based health monitoring for all conditions including `PARTIAL_DEGRADATION`). The two channels are complementary: Discord catches urgent P0 conditions immediately; the health endpoint enables continuous monitoring via uptime-kuma that catches the intermediate degradation states below the alarm threshold. Q069's Step 9 requires updating to deploy the bidirectional guard in place of Q067's unidirectional version, with the return type change (`int` → `EpochCheckResult`) being the only breaking interface change at the call site. A new Step 9a (uptime-kuma health probe configuration) is added to ensure ongoing monitoring coverage.
