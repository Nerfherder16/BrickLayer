# Finding: Q071 — Recall Health Dashboard: Minimum Observable Signal Set

**Question**: No question in the research program has addressed how to know when the Recall system is functioning correctly at the component level. What is the minimum set of observable signals constituting a Recall health dashboard — covering: (1) LLPC retrieval quality (P@3 from IPS exploration oracle, Q058), (2) MBC routing efficiency (Level 1/2/3 distribution, Q045), (3) session-summary pending queue depth (write-tier partition, Q056), (4) epoch-clock integrity (epoch reset detection, Q062/Q065). At what threshold does each signal indicate degradation requiring human intervention? What is the minimum infrastructure to expose these signals? Which signals can be computed from existing data without new telemetry? What is the implementation cost of each signal (lines of code)?
**Question Type**: CONVERGENCE
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: observability engineering, SRE (site reliability engineering), database internals (PostgreSQL system catalog queries), distributed systems health monitoring

---

## Health Signal Definitions

| Signal | Source | Healthy | Warning | Critical | Compute cost |
|--------|--------|---------|---------|----------|-------------|
| **S1: IPS P@3 delta** (exploration-oracle evaluation, Q058) | PostgreSQL `retrieve_events` table; `is_exploration` flag per slot | IPS P@3 ≥ raw-count P@3 (delta ≥ 0.00) over rolling 7-day window | delta < 0.00 for 3 consecutive days (IPS ranking underperforming raw counts) | delta < −0.10 sustained 7 days (systematic IPS regression) | ~30 lines Python, scheduled weekly query |
| **S2: MBC Level distribution** (routing efficiency, Q045) | PostgreSQL `retrieve_events`; `priority_level` column per MBC result | Level 1 ≥ 60% of MBC results over rolling 7-day window | Level 1 < 40% OR Level 3 > 30% (tag vocabulary coverage degrading) | Level 3 > 50% (structural routing collapsed to frequency fallback) | ~15 lines SQL, zero new telemetry if `priority_level` is logged |
| **S3: Pending queue depth** (write-tier partition policy, Q056) | Filesystem: count of `.json` files in `~/.claude/recall-pending/` | 0 pending files | 1–3 files (isolated outage, normal retry expected) | ≥ 4 files OR any file older than 48 hours (persistent CasaOS unavailability or hook retry broken) | ~5 lines shell/Python, zero new telemetry |
| **S4: Epoch-clock integrity** (epoch reset detection, Q062/Q065) | Redis: `recall:global_epoch` current value; PostgreSQL: `MAX(last_access_epoch)` across memories table | `global_epoch` ≥ `MAX(last_access_epoch)` AND `global_epoch` increasing monotonically session-to-session | `global_epoch` < `MAX(last_access_epoch)` (Redis reset, stale epoch reference in DB rows) | `global_epoch` = 0 AND `MAX(last_access_epoch)` > 10 (Redis FLUSHALL or restart without persistence; all LLPC/MBC working-set filters now evaluate false for every memory) | ~10 lines Python, one Redis GET + one PostgreSQL MAX query |

## Infrastructure Decision

**The minimum viable implementation is a single FastAPI `/health/detail` endpoint** backed by four queries, callable on-demand and by a cron job. No new infrastructure beyond what exists. No new database tables. No new services. The four signals map directly onto data structures already required by Q045, Q056, Q058, and Q062 — the only precondition is that those findings' data fields are being populated (specifically: `is_exploration`, `priority_level`, the pending queue directory, and the epoch columns).

**Option comparison:**

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| SQL query scheduled via cron (daily) | Zero new infrastructure, trivially auditable | No real-time visibility, no dashboard | Acceptable for S3 and S4 only |
| Python script (`health_check.py`) run manually | Self-contained, no service dependency | No persistent history, no alerting | Acceptable for development; not for production |
| FastAPI `/health/detail` endpoint | Callable by dashboard, monitoring tools, curl; integrates with existing Recall API server | Adds one endpoint to the API | **Recommended** |
| New Recall dashboard page (React + Vite) | Best UX, visual history | 2–4 weeks frontend work; overkill for 4 signals | Deferred to Phase 2 |

The existing Recall API server (FastAPI on CasaOS, port 8200) already has a `/health` endpoint (referenced in project-brief.md). The implementation adds a `/health/detail` route returning a JSON object with all four signals. The existing React dashboard can render this JSON in a health panel with minimal frontend changes.

## SQL/Python Snippets

### S1: IPS P@3 Delta (exploration-oracle evaluation)

```python
# health_signals.py — S1
# Requires: retrieve_events table with columns (session_id, memory_id, rank_under_ips,
#           rank_under_raw_count, was_cited, is_exploration, event_timestamp)
# is_exploration must be logged from the epsilon-greedy injection path (Week 1 requirement, Q058)

import asyncpg
from datetime import datetime, timedelta

async def compute_ips_p3_delta(pg_conn, window_days: int = 7) -> dict:
    cutoff = datetime.utcnow() - timedelta(days=window_days)
    rows = await pg_conn.fetch("""
        SELECT
            session_id,
            -- P@3 under IPS ranking: exploration event where IPS rank <= 3 AND was_cited
            AVG(CASE WHEN rank_under_ips <= 3 AND was_cited THEN 1.0 ELSE 0.0 END)
                AS session_p3_ips,
            -- P@3 under raw-count ranking: exploration event where raw rank <= 3 AND was_cited
            AVG(CASE WHEN rank_under_raw_count <= 3 AND was_cited THEN 1.0 ELSE 0.0 END)
                AS session_p3_raw
        FROM retrieve_events
        WHERE is_exploration = TRUE
          AND event_timestamp >= $1
        GROUP BY session_id
        HAVING COUNT(*) >= 3  -- minimum 3 exploration events for a session to count
    """, cutoff)

    if len(rows) < 5:
        return {"status": "insufficient_data", "exploration_sessions": len(rows),
                "minimum_required": 5, "delta": None}

    delta = sum(r["session_p3_ips"] - r["session_p3_raw"] for r in rows) / len(rows)
    status = "healthy" if delta >= 0.0 else ("warning" if delta >= -0.10 else "critical")
    return {
        "signal": "ips_p3_delta",
        "status": status,
        "delta": round(delta, 4),
        "exploration_sessions_evaluated": len(rows),
        "window_days": window_days,
        "threshold_warning": 0.00,
        "threshold_critical": -0.10,
    }
```

Implementation cost: ~30 lines Python. Zero new infrastructure. Requires `is_exploration`, `rank_under_ips`, `rank_under_raw_count` columns in `retrieve_events` — these are Week 1 prerequisites from Q058, not new additions.

### S2: MBC Level Distribution

```python
# health_signals.py — S2
# Requires: retrieve_events table with column priority_level INT (1/2/3)
# priority_level is the field returned by the Q045 three-level routing SQL

async def compute_mbc_level_distribution(pg_conn, window_days: int = 7) -> dict:
    cutoff = datetime.utcnow() - timedelta(days=window_days)
    row = await pg_conn.fetchrow("""
        SELECT
            COUNT(*)                                                   AS total_mbc_results,
            SUM(CASE WHEN priority_level = 1 THEN 1 ELSE 0 END)       AS level1_count,
            SUM(CASE WHEN priority_level = 2 THEN 1 ELSE 0 END)       AS level2_count,
            SUM(CASE WHEN priority_level = 3 THEN 1 ELSE 0 END)       AS level3_count
        FROM retrieve_events
        WHERE retrieve_type = 'mbc'
          AND event_timestamp >= $1
    """, cutoff)

    if not row or row["total_mbc_results"] == 0:
        return {"status": "no_data", "total_mbc_results": 0}

    total = row["total_mbc_results"]
    l1_pct = row["level1_count"] / total
    l3_pct = row["level3_count"] / total

    if l1_pct < 0.40 or l3_pct > 0.50:
        status = "critical"
    elif l1_pct < 0.60 or l3_pct > 0.30:
        status = "warning"
    else:
        status = "healthy"

    return {
        "signal": "mbc_level_distribution",
        "status": status,
        "level1_pct": round(l1_pct, 3),
        "level2_pct": round(row["level2_count"] / total, 3),
        "level3_pct": round(l3_pct, 3),
        "total_mbc_results": total,
        "window_days": window_days,
        "threshold_warning": {"level1_min": 0.60, "level3_max": 0.30},
        "threshold_critical": {"level1_min": 0.40, "level3_max": 0.50},
        "interpretation": (
            "Tag vocabulary coverage degraded — sessions increasingly falling back to frequency ordering"
            if status != "healthy" else "Structural routing performing within normal bounds"
        ),
    }
```

Implementation cost: ~15 lines Python + the SQL already embedded in Q045. Zero new infrastructure. Requires `priority_level` and `retrieve_type` columns in `retrieve_events` — these are native outputs of the Q045 routing query and should be logged at MBC retrieval time.

### S3: Pending Queue Depth

```python
# health_signals.py — S3
# Zero new infrastructure. Reads the filesystem directory from Q056.

import os
from pathlib import Path

def compute_pending_queue_depth(
    pending_dir: str = os.path.expanduser("~/.claude/recall-pending")
) -> dict:
    p = Path(pending_dir)
    if not p.exists():
        return {"signal": "pending_queue_depth", "status": "healthy", "depth": 0,
                "oldest_file_hours": None, "note": "pending directory does not exist"}

    files = sorted(p.glob("session-*.json"), key=lambda f: f.stat().st_mtime)
    depth = len(files)
    oldest_hours = None
    if files:
        import time
        oldest_hours = round((time.time() - files[0].stat().st_mtime) / 3600, 1)

    if depth >= 4 or (oldest_hours is not None and oldest_hours >= 48):
        status = "critical"
    elif depth >= 1:
        status = "warning"
    else:
        status = "healthy"

    return {
        "signal": "pending_queue_depth",
        "status": status,
        "depth": depth,
        "oldest_file_hours": oldest_hours,
        "threshold_warning": {"depth_min": 1},
        "threshold_critical": {"depth_min": 4, "oldest_hours_min": 48},
        "interpretation": (
            "CasaOS persistently unreachable or hook retry loop broken"
            if status == "critical" else
            "Isolated outage — retry expected on next session start"
            if status == "warning" else
            "No pending session summaries"
        ),
    }
```

Implementation cost: ~20 lines Python. Zero new infrastructure. Zero new telemetry. The pending directory is created by the Stop hook implementation from Q056.

### S4: Epoch-Clock Integrity

```python
# health_signals.py — S4
# Detects the epoch reset failure mode described in Q062/Q065.
# Redis FLUSHALL or restart without AOF/RDB persistence resets recall:global_epoch to 0.
# All memories then have last_access_epoch > global_epoch, breaking the working-set filter.

import redis.asyncio as aioredis

async def compute_epoch_integrity(
    redis_client: aioredis.Redis,
    pg_conn: asyncpg.Connection,
) -> dict:
    global_epoch = await redis_client.get("recall:global_epoch")
    global_epoch = int(global_epoch) if global_epoch is not None else None

    max_db_epoch = await pg_conn.fetchval(
        "SELECT MAX(last_access_epoch) FROM memories"
    )
    max_db_epoch = int(max_db_epoch) if max_db_epoch is not None else 0

    if global_epoch is None:
        return {
            "signal": "epoch_clock_integrity",
            "status": "critical",
            "global_epoch": None,
            "max_db_epoch": max_db_epoch,
            "interpretation": "recall:global_epoch key missing from Redis — epoch counter not initialized or Redis data lost",
        }

    epoch_gap = global_epoch - max_db_epoch  # should be >= 0

    if global_epoch == 0 and max_db_epoch > 10:
        status = "critical"
        interpretation = (
            f"Epoch reset detected: Redis epoch=0 but DB records show max epoch={max_db_epoch}. "
            f"All LLPC/MBC working-set filters will evaluate FALSE for every memory. "
            f"Recovery: SET recall:global_epoch {max_db_epoch + 1} in Redis."
        )
    elif epoch_gap < 0:
        status = "critical"
        interpretation = (
            f"Epoch inversion: Redis epoch ({global_epoch}) < max DB epoch ({max_db_epoch}). "
            f"Redis may have been reset to an earlier checkpoint. Working-set filters broken."
        )
    elif epoch_gap > 1000:
        status = "warning"
        interpretation = (
            f"Epoch gap is large ({epoch_gap}): Redis epoch is {epoch_gap} sessions ahead of the "
            f"most recently accessed memory. This may indicate the corpus has grown stale "
            f"(no memories accessed recently) or a data migration issue."
        )
    else:
        status = "healthy"
        interpretation = f"Epoch consistent: global={global_epoch}, max_db={max_db_epoch}, gap={epoch_gap}"

    return {
        "signal": "epoch_clock_integrity",
        "status": status,
        "global_epoch": global_epoch,
        "max_db_epoch": max_db_epoch,
        "epoch_gap": epoch_gap,
        "interpretation": interpretation,
        "recovery_action": (
            f"redis-cli SET recall:global_epoch {max_db_epoch + 1}"
            if status == "critical" else None
        ),
    }
```

Implementation cost: ~35 lines Python. One Redis GET + one PostgreSQL MAX query. Zero new telemetry. `last_access_epoch` column must exist (Q062 schema migration). The recovery action is printed in the response so no manual diagnosis is needed.

### FastAPI endpoint wiring

```python
# In recall_api/routers/health.py — new route, ~25 lines

from fastapi import APIRouter, Depends
from recall_api.dependencies import get_pg_pool, get_redis_client
from .health_signals import (
    compute_ips_p3_delta,
    compute_mbc_level_distribution,
    compute_pending_queue_depth,
    compute_epoch_integrity,
)

router = APIRouter(prefix="/health", tags=["health"])

@router.get("/detail")
async def health_detail(
    pg=Depends(get_pg_pool),
    redis=Depends(get_redis_client),
):
    async with pg.acquire() as conn:
        signals = {
            "s1_ips_p3_delta":        await compute_ips_p3_delta(conn),
            "s2_mbc_level_dist":      await compute_mbc_level_distribution(conn),
            "s3_pending_queue":       compute_pending_queue_depth(),
            "s4_epoch_integrity":     await compute_epoch_integrity(redis, conn),
        }
    overall = (
        "critical" if any(s.get("status") == "critical" for s in signals.values()) else
        "warning"  if any(s.get("status") == "warning"  for s in signals.values()) else
        "healthy"
    )
    return {"overall": overall, "signals": signals}
```

Total new lines of code across all four signals and the endpoint: approximately **135 lines** (30 + 15 + 20 + 35 + 25 + imports/boilerplate). This is the complete implementation. No new tables. No new services. No schema changes beyond what Q045, Q056, Q062 already require.

## Calibration: When to Alert

Each signal has a defined recovery action, not just a threshold. The health dashboard is only useful if it tells the operator what to do, not just that something is wrong.

| Signal | Warning action | Critical action |
|--------|---------------|----------------|
| S1: IPS P@3 delta | Review propensity model refit schedule (Q058 suggests rolling 200-session window); check if epsilon-greedy exploration is still active | Disable IPS reranking and fall back to raw-count ordering; investigate whether `is_exploration` logging is still functioning |
| S2: MBC Level distribution | Review tag assignment quality for recent session summaries; check if `file-tag-map.json` covers recent project domains | Reset tag vocabulary: review 10 most recent sessions where Level 3 fired, identify missing tags, add to static map |
| S3: Pending queue depth | No immediate action — monitor for natural retry on next session start | Manual: `curl -X POST http://192.168.50.19:8200/recall/replay-pending` (endpoint to force retry); check CasaOS Docker status |
| S4: Epoch integrity | Investigate epoch gap cause; run consistency check | Execute recovery command printed in the signal response: `redis-cli SET recall:global_epoch {max_db_epoch + 1}` |

## Ideas Extracted

### Idea: recall-health-detail-endpoint
- **Novelty**: 0.70 — A structured, machine-readable health endpoint that reports retrieval quality (not just service availability) is not present in any described AI memory system (mem0, Zep, MemGPT, Letta). Standard health checks verify connectivity (is Qdrant reachable? is PostgreSQL accepting connections?). This endpoint reports whether the system is functioning correctly at the semantic layer: is retrieval quality improving, is routing working structurally, are write durability guarantees holding? The distinction between infrastructure health and retrieval quality health is novel in this context.
- **Evidence**: 0.72 — Site reliability engineering practice (Beyer et al., "Site Reliability Engineering," Google, 2016) establishes the "golden signals" framework: latency, traffic, errors, saturation. The extension to AI system health — adding retrieval quality signals alongside infrastructure signals — is consistent with the SRE framework but applied to a domain (AI memory retrieval) where it has not been formally described. The specific signal set (P@3 delta, level distribution, queue depth, epoch integrity) is derived directly from the validated mechanisms in Q058, Q045, Q056, Q062.
- **Feasibility**: 0.95 — 135 lines of Python across one new router file. Integrates with the existing FastAPI server. No new services. No new databases. No new infrastructure. All data sources (PostgreSQL, Redis, filesystem) are already reachable from the API server process. Implementation: 1-3 days including tests.
- **Description**: A `/health/detail` FastAPI endpoint that computes and returns all four Recall health signals in a single JSON response. Each signal reports its current value, status (healthy/warning/critical), the threshold that triggered the status, a human-readable interpretation, and (for critical states) the exact recovery command to run. The endpoint aggregates to an `overall` status field. Callable by the existing React dashboard, by monitoring tools (Uptime Kuma, Grafana), or by a daily cron job that sends a notification if `overall != "healthy"`.

### Idea: epoch-reset-recovery-command-in-response
- **Novelty**: 0.68 — Embedding the exact recovery command in the health check response (rather than requiring the operator to know the correct Redis command) is a self-documenting health signal. The S4 epoch integrity check prints `redis-cli SET recall:global_epoch {max_db_epoch + 1}` directly in the JSON response when a reset is detected. This converts a cryptic infrastructure failure into an actionable one-line fix. The pattern is used by some database health tools (pg_activity, pgBadger) but is not present in AI memory system health monitoring.
- **Evidence**: 0.65 — The "remediation action in the alert" pattern is documented in SRE runbook design (Limoncelli, Hogan, Chalup, "The Practice of System and Network Administration," 3rd ed., 2016): alerts should include the corrective action, not just the symptom. Applied here to the specific failure mode of Redis epoch reset, which has a deterministic one-command fix.
- **Feasibility**: 0.97 — Zero additional implementation cost beyond computing `max_db_epoch` (already required for S4). The recovery string is a format operation on two integers already in scope. Adds ~3 lines to the epoch integrity function.
- **Description**: When S4 detects an epoch reset (Redis epoch < max DB epoch or Redis epoch = 0 with DB epoch > 10), include a `recovery_action` field in the signal response containing the exact Redis command to restore consistency: `redis-cli SET recall:global_epoch {max_db_epoch + 1}`. This command sets the global epoch to one greater than the highest epoch value stored in any memory row, restoring the invariant `global_epoch >= MAX(last_access_epoch)`. The operator can copy-paste this command from the dashboard response without needing to understand the epoch model internals.

### Idea: retrieval-quality-health-vs-infrastructure-health
- **Novelty**: 0.75 — The conceptual distinction between infrastructure health (is the service reachable?) and retrieval quality health (is the service returning good memories?) has not been implemented in any reviewed AI memory system. The existing `/health` endpoint in Recall checks service availability. The `/health/detail` endpoint proposed here checks whether the retrieval algorithms are functioning as designed. This distinction matters: a system can be infrastructurally healthy (all services responding) while retrieval quality has silently degraded (MBC routing collapsed to frequency fallback, IPS ranking underperforming raw counts). Infrastructure monitoring does not surface these silent degradations.
- **Evidence**: 0.70 — The application-layer health check concept is established in observability literature. Google's SRE book distinguishes between "symptom-based alerting" (the service is returning errors to users) and "cause-based alerting" (a specific component has failed). The signals defined here are symptom-based for the retrieval layer: they detect when the user experience (memory quality) has degraded, regardless of what caused it. This is the correct approach for a personal memory system where the user is the only observer of retrieval quality.
- **Feasibility**: 0.90 — The four signals defined in this finding constitute the minimum set. No additional signals are needed. The entire implementation is 135 lines. The existing React dashboard can display the `/health/detail` JSON in a dedicated panel with color-coded status indicators.
- **Description**: Recall's health monitoring must operate on two distinct layers. Layer 1 (infrastructure): all backing services reachable, latency within SLA, no error rate spikes. This is already partially implemented via the existing `/health` endpoint. Layer 2 (retrieval quality): the signals defined in S1–S4. Layer 2 catches silent degradations that Layer 1 cannot see: IPS reranking regressing below raw counts (S1), tag vocabulary coverage collapsing (S2), session summaries accumulating unsubmitted (S3), epoch-clock invariant violated (S4). A complete health monitoring strategy requires both layers. The `/health/detail` endpoint implements Layer 2.

## Key Finding

The minimum health dashboard for Recall consists of exactly four signals derivable from existing data sources (PostgreSQL, Redis, filesystem) with approximately 135 lines of new Python. Three of the four signals (S2, S3, S4) require zero new telemetry beyond what the research findings already mandate; S1 requires three new columns in `retrieve_events` that are Week 1 prerequisites of the Q058 IPS evaluation framework anyway. The most operationally urgent signal is S4 (epoch integrity), because an epoch reset causes a silent catastrophic failure — all LLPC and MBC working-set filters evaluate false for every memory — and the recovery is a single Redis command that the health check can print automatically. The most diagnostically rich signal is S2 (MBC level distribution), because a rising Level 3 fraction is the earliest detectable symptom of tag vocabulary drift, preceding any observable retrieval quality decline by days or weeks.

The critical insight across all four signals is that retrieval quality health is not the same as infrastructure health. A system can pass all standard infrastructure health checks (all services reachable, all endpoints responding 200) while silently degrading at the semantic layer. The four signals defined here are the minimum set that closes this observability gap for Recall's specific architecture.
