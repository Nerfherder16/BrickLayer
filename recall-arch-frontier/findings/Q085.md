# Finding: Q085 — 60s Cache Window Race Condition: Net Migration Progress and Coordination Protocol for Dual-Node Writes

**Question**: During the 60-second TTL cache window after `POST /api/tags/migrate-synonyms` is called (Q075's governance design), the other machine's hook continues writing raw-form tags — creating new pollution at the same rate the migration corrects old ones. What is the net migration progress if casaclaude runs the migration while proxyclaude continues writing at 10 memories/session, and what is the correct coordination protocol given that casaclaude and proxyclaude have no shared IPC channel beyond the Recall API itself?
**Agent**: quantitative-analyst
**Question Type**: ADVERSARIAL
**Verdict**: MARGINAL
**Severity**: Low
**Source field**: Distributed systems (cache invalidation, consistency windows), queueing theory (arrival rate vs. service rate)

---

## Race Condition Analysis

The adversarial claim is that during the 60-second TTL window after `POST /api/tags/migrate-synonyms` is called, the non-migrating machine (proxyclaude) continues writing raw-form tags at a rate that cancels migration progress. The claim requires scrutiny at the actual arrival rate.

### Arrival rate derivation

From the question parameters:
- Total sessions across both machines: ~5/day
- Memories per session: ~10
- Total new memories per day: ~50
- Per-machine rate (proxyclaude share): ~2-3 sessions/day = 20-30 memories/day

Converting to a per-second rate:
```
Rate = 25 memories/day ÷ 86,400 seconds/day = 0.000289 memories/second
```

### Expected writes during the 60-second window

```
E[writes in 60s] = 60s × 0.000289 mem/s = 0.0174 memories
```

Rounding generously to the upper bound (30 memories/day):
```
E[writes in 60s] = 60s × (30 / 86,400) = 0.0208 memories
```

**Expected writes during the 60-second window: < 0.03 memories (approximately 1 write every 48 minutes from proxyclaude).**

This is not a steady-state rate that fills the window — it is an extremely sparse Poisson process. The probability of even one raw-form write occurring during any given 60-second window is:

```
P(at least 1 write in 60s) = 1 - e^(-λt) = 1 - e^(-0.0208) ≈ 2.06%
```

A race condition event (one raw-form tag written during the migration window) occurs roughly once every 48 sessions, or approximately once per 10 days at Tim's session rate.

## Net Migration Progress Calculation

The migration (`POST /api/tags/migrate-synonyms`) runs in < 500ms and corrects the entire existing corpus at the moment of the call. At 2,000 stored memories with ~30% containing at least one synonym-form tag (600 memories), the migration corrects 600 memories in a single PostgreSQL transaction.

Against this, during the 60-second window, the expected number of new raw-form writes is 0.02.

**Net migration progress calculation:**

```
Memories corrected by migration:        ~600
Memories newly polluted during window:  ~0.02 (expected)

Net corrected:  599.98 (expected)
Net progress:   99.997% of theoretical maximum
```

Even in the worst-case scenario where proxyclaude happens to be in an active session during the exact 60-second window — a 2% probability event — the result is one additional raw-form memory in a corpus of 600 newly corrected memories. The migration is not meaningfully undermined.

The migration is also idempotent. Re-calling `POST /api/tags/migrate-synonyms` costs < 500ms and corrects any memories written with raw forms after the first run. A second call after the cache TTL expires on both nodes eliminates all remaining pollution at zero risk.

**The adversarial premise does not hold at Tim's session rate.** The claim of "new pollution at the same rate the migration corrects old ones" requires an arrival rate orders of magnitude higher than Tim's actual 0.00029 memories/second. To reach parity between migration and pollution, the write rate would need to be 600 corrections / 60 seconds = 10 raw-form writes per second — roughly 864,000 memories/day. Tim's system operates at 50 memories/day.

## Protocol Comparison

### Protocol (a): Accept the 60-second window

**Mechanism**: No coordination. Migration runs, both nodes converge within one TTL cycle. Rerun if needed (idempotent, < 500ms).

**Latency cost**: Zero. No additional API calls, no blocking.

**Implementation complexity**: Zero. Already implied by the Q075 design.

**Correctness guarantee**: Probabilistically correct at Tim's rate. Expected pollution during window: 0.02 memories. Expected rerun frequency: once per 10 days maximum. At Tim's scale, this is the correct operating point.

**Failure mode**: If Tim's session rate increases dramatically (e.g., both machines running automated bulk imports), the window risk grows proportionally. Not a present concern.

**Verdict**: Correct for Tim's actual usage pattern. The cost of any alternative exceeds the expected damage.

---

### Protocol (b): Push-invalidation via Redis version key

**Mechanism**: `POST /api/tags/migrate-synonyms` atomically increments a Redis key `recall:tag-map-version:N` (TTL=0, persistent). On each memory write, the hook checks the current version key and compares to its cached version. If versions differ, the hook drops its in-process cache and re-fetches immediately.

**Latency cost**: Each memory write requires an additional Redis GET before proceeding — roughly 0.3-0.5ms on LAN. At 50 memories/day, this adds < 25ms of total latency per day across both machines. Negligible in absolute terms.

However, the implementation requires:
1. A persistent Redis key managed by the Recall API (not the cache layer)
2. The hook to track its cached version number alongside the cached map
3. A Redis connection in the hook process (or a version-check endpoint on the Recall API)

**Implementation complexity**: Moderate. The hook must maintain version state and make an additional API call per write. This doubles the hook's network operations from one (store) to two (version-check + store), or three if the version-check triggers a map re-fetch (version-check + map-fetch + store).

**Correctness guarantee**: Strong. Any node that has written to its cache before the migration call will detect the version increment on its next write and re-fetch the map immediately, rather than waiting up to 60 seconds.

**Verdict**: Architecturally correct but disproportionate to the problem size at Tim's rate. The implementation cost (additional Redis key management, hook version tracking, extra network hop per write) solves a problem with expected impact of 0.02 memories per 60-second window. Protocol (b) is the right answer if Tim's system ever scales to multiple automated agents writing continuously — it is not the right answer for Tim's current setup.

**Reserved use case**: If Tim ever adds a CI/CD pipeline or automated ingestion that writes memories at high frequency, Protocol (b) should be enabled. The infrastructure is simple enough to add proactively.

---

### Protocol (c): Coordinated write-pause with 202 + token

**Mechanism**: `POST /api/tags/migrate-synonyms` returns HTTP 202 with a coordination token. All hooks check this token before each write and block for up to 60 seconds until the token clears, allowing the migration to complete before any new writes proceed.

**Latency cost**: Every memory write during a migration event is blocked for up to 60 seconds. A session where Tim calls the migration and then immediately creates a memory experiences up to 60-second write latency. This is a user-visible stall in the observe-edit.js hook, which runs synchronously in the PostToolUse hook path.

**Critical flaw**: The hooks have no shared IPC channel beyond the Recall API (stated in the question). To implement Protocol (c), each hook must poll the API for token status on every write. This requires:
1. A token issuance endpoint on the Recall API
2. A token-check endpoint (or same endpoint) callable by hooks
3. Hook-side blocking logic with a timeout loop
4. Token cleanup after migration completes or times out

**Correctness guarantee**: Theoretically strong — prevents all writes during the migration window. In practice, correctness depends on hooks reliably checking the token before every write and correctly handling the blocking/timeout case. A hook that crashes, times out, or is restarted during the pause window bypasses the coordination entirely.

**Additional failure modes**:
- If the migration call fails mid-run and the token is never cleared, all hooks block indefinitely until timeout. Requires careful token TTL management and cleanup.
- The 60-second block applies to both machines, including casaclaude (which initiated the migration). Both nodes are write-paused during the migration window.
- The observe-edit.js hook must be refactored to support async blocking, which the current hook architecture may not support (hooks are typically fire-and-forget or short-running).

**Verdict**: Architecturally over-engineered and introduces new failure modes more severe than the problem it solves. Protocol (c) adds up to 60 seconds of observable latency to memory writes, requires distributed coordination state, and has correctness gaps (hook restart, timeout handling) that the simple idempotent-migration approach does not. Protocol (c) is the wrong answer for any scale Tim is likely to reach.

## Recommendation

**Protocol (a) is correct for Tim's dual-machine setup.**

The mathematical case is closed: at 50 memories/day, the probability of even one raw-form write occurring during a 60-second migration window is 2.06%. The expected number of such writes is 0.02 — two hundredths of one memory against 600 corrected. The migration is idempotent and costs < 500ms to rerun.

The correct operational procedure is:

1. Tim calls `POST /api/tags/migrate-synonyms` from the dashboard (casaclaude).
2. The migration completes in < 500ms, correcting the existing corpus.
3. Both nodes (casaclaude and proxyclaude) will re-fetch the updated normalization map within 60 seconds on their next cache-miss cycle.
4. After 60 seconds, Tim optionally calls `POST /api/tags/migrate-synonyms` a second time to catch any memories written during the window. Given the 2% probability, this second call is almost never necessary — but costs nothing meaningful if Tim makes it a habit after any map update.

The IPC constraint (no shared channel beyond the Recall API) is not a problem because no coordination is needed. The API itself is the coordination point: both hooks read from the same endpoint, both converge within one TTL, and the migration endpoint is idempotent.

**If Protocol (b) is ever needed** (future automated ingestion scaling writes to > 100/day), implement it as an opt-in mode: the Recall API exposes a version header on `GET /api/tags/synonyms`, and hooks optionally check `If-None-Match` with the cached ETag. On migration, the ETag changes. Hooks detect the mismatch and re-fetch. This is the HTTP-native cache invalidation pattern and requires no additional Redis keys or polling — the version is piggy-backed on the existing map-fetch response.

## Ideas Extracted

### Idea: idempotent-migration-double-call-as-coordination-primitive
- **Novelty**: 0.67
- **Evidence**: 0.72
- **Feasibility**: 0.97
- **Description**: When distributed nodes share only an API endpoint and have sparse write rates, the correct coordination protocol is not push-invalidation or write-pausing but idempotent-operation repetition. Calling `POST /api/tags/migrate-synonyms` twice — once immediately and once after the TTL window — costs ~1 second total and eliminates the race condition without any additional infrastructure. This generalizes: for any API operation that is (a) idempotent, (b) fast, and (c) correcting a corpus against sparse concurrent writes, the correct consistency protocol is scheduled repetition rather than distributed locking. The double-call pattern requires zero coordination state because the operation's idempotency absorbs the inconsistency.

### Idea: etag-based-cache-invalidation-for-normalization-map
- **Novelty**: 0.68
- **Evidence**: 0.65
- **Feasibility**: 0.85
- **Description**: Rather than a Redis version key (Protocol b) or write-pause token (Protocol c), the Recall API can signal normalization map changes using standard HTTP ETag/If-None-Match semantics on the existing `GET /api/tags/synonyms` endpoint. The API returns an ETag (e.g., hash of the map content or a monotonic version integer). Hooks store the ETag alongside the cached map. On cache-miss re-fetch, the hook sends `If-None-Match: <etag>`. If the map has not changed, the API returns 304 Not Modified with no body — saving bandwidth and parse time. If changed, the API returns 200 with the new map and new ETag. On migration, the ETag changes. This makes immediate cache invalidation possible without a separate Redis key: hooks that happen to re-fetch during the migration window get the new map immediately. No additional infrastructure, no polling, no blocking — just HTTP caching done correctly. The migration event increments the ETag, which propagates naturally through the existing fetch path.

## Key Finding

The adversarial claim that the 60-second cache window causes net-negative migration progress fails at Tim's actual session rate. At 50 memories/day (0.00029 memories/second per node), the expected number of raw-form writes during any 60-second window is 0.02 — a 2% probability event delivering one write against 600 corrections. The migration corrects 99.997% of the corpus even without coordination. Protocol (a) — accepting the window with optional idempotent rerun — is correct and requires no implementation work beyond what Q075 already specifies.

Protocol (b) push-invalidation is architecturally sound but disproportionate at Tim's scale. It becomes the right answer only if automated ingestion raises write rates above ~100 memories/day. When that threshold is crossed, the cleanest implementation uses HTTP ETag semantics on the existing `GET /api/tags/synonyms` endpoint rather than a separate Redis key — no new infrastructure, no additional polling, standard HTTP cache invalidation. Protocol (c) write-pause introduces latency and failure modes more dangerous than the problem it solves and is the wrong answer at any scale for a hook-based architecture.

The broader finding is that the IPC constraint (no shared channel beyond the Recall API) is not a limitation — it is the correct architecture. The API itself is the coordination primitive. Idempotent operations plus the existing TTL convergence give a consistency model appropriate for Tim's write volume. Complex distributed coordination is only warranted when write rate × window duration exceeds a meaningful pollution threshold, which at Tim's rate requires a 500x increase in session frequency.
