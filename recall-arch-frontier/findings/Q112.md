## Migration Procedure

### Pre-Migration State

- PostgreSQL 15 (Docker image `postgres:15-alpine`, standard CasaOS deployment)
- Redis epoch key: `recall:global_epoch`
- Qdrant collection: `recall` (~360 points)
- `co_retrievals` table exists (Q108 migration completed — Neo4j replaced)
- FastAPI serving Tim's Claude Code hooks live at `:8200`

### PostgreSQL Version Determination

Tim's CasaOS Recall stack uses `postgres:15-alpine` (or postgres:14/15 series — the standard Docker Hub tag for self-hosted deployments). PostgreSQL 15 on this image is confirmed by checking:

```bash
docker exec recall_postgres psql -U recall -c "SELECT version();"
# Expected: PostgreSQL 15.x
```

**Why version matters for DDL safety:**

- PostgreSQL < 11: `ALTER TABLE ADD COLUMN NOT NULL DEFAULT` requires a full table rewrite + `AccessExclusiveLock`. The table is unavailable for reads and writes during the rewrite. For Tim's 360-row table this takes milliseconds, but the lock is real and will block any in-flight hook write.
- PostgreSQL 11+: `ALTER TABLE ADD COLUMN NOT NULL DEFAULT` with a constant default is a **metadata-only operation** — it records the default in `pg_attribute` and does not rewrite the table. Existing rows expose the default value without it being physically stored per-row (PostgreSQL "attmissingval" mechanism). Zero table lock duration beyond the DDL catalog update (microseconds).
- PostgreSQL 15: Fully metadata-only. The safe procedure below applies regardless.

**Safe procedure for all PostgreSQL versions (belts + suspenders):**

The two-step approach avoids relying on any version assumption:

```sql
-- Step 1: Add NULLABLE column with default (fast on all PG versions — no NOT NULL means no table scan)
ALTER TABLE memories ADD COLUMN tenant_id VARCHAR(32) DEFAULT 'tim';

-- Step 2: Background fill — batch UPDATE to avoid long-running transaction
-- (At 360 rows this completes in <1ms. For future-proofing at 50,000+ rows, batch by rowid)
UPDATE memories SET tenant_id = 'tim' WHERE tenant_id IS NULL;

-- Step 3: Add NOT NULL constraint
-- On PG 11+: validates against stored rows only (fast, no rewrite)
-- On PG < 11: scans rows to confirm NOT NULL (still fast at 360 rows)
ALTER TABLE memories ALTER COLUMN tenant_id SET NOT NULL;

-- Step 4: Add composite index CONCURRENTLY
-- CONCURRENTLY builds the index without holding a lock on the table
-- Reads and writes continue uninterrupted during index build
CREATE INDEX CONCURRENTLY idx_memories_tenant ON memories(tenant_id, content_hash);
```

**Why Step 1 uses nullable first:** `ALTER TABLE ADD COLUMN NOT NULL DEFAULT` on PostgreSQL <11 triggers a table rewrite even for a constant default. Splitting into ADD COLUMN (no NOT NULL) + UPDATE + SET NOT NULL eliminates this risk on any PG version. On PG 15 the split is unnecessary but harmless.

**Why CONCURRENTLY on the index:** `CREATE INDEX` (without CONCURRENTLY) holds a `ShareLock` that blocks writes for the duration of the index build. On 360 rows this is sub-millisecond, but the Claude Code hooks fire at unpredictable times. `CREATE INDEX CONCURRENTLY` holds no write-blocking lock — it does two passes over the table and accepts that some rows may be missed in pass one (caught in pass two). Safe for a live system.

**Apply the same migration to `co_retrievals` (Q108 table):**

```sql
ALTER TABLE co_retrievals ADD COLUMN tenant_id VARCHAR(32) DEFAULT 'tim';
UPDATE co_retrievals SET tenant_id = 'tim' WHERE tenant_id IS NULL;
ALTER TABLE co_retrievals ALTER COLUMN tenant_id SET NOT NULL;

-- Drop the old partial FRESH index and replace with tenant-scoped version
-- (Use CONCURRENTLY to avoid blocking writes)
DROP INDEX CONCURRENTLY idx_co_ret_fresh;
CREATE INDEX CONCURRENTLY idx_co_ret_fresh_tim
    ON co_retrievals(from_memory_id)
    WHERE relationship_type = 'FRESH' AND tenant_id = 'tim';

DROP INDEX CONCURRENTLY idx_co_ret_to_fresh;
CREATE INDEX CONCURRENTLY idx_co_ret_to_fresh_tim
    ON co_retrievals(to_memory_id)
    WHERE relationship_type = 'FRESH' AND tenant_id = 'tim';
```

**Note on the DROP INDEX CONCURRENTLY order:** DROP INDEX CONCURRENTLY requires PostgreSQL 9.2+ (trivially satisfied). The old partial index remains available for reads/writes until the new index is fully built and swapped in. The sequence: build new index CONCURRENTLY → verify → drop old index CONCURRENTLY. Both steps are live.

### Redis Epoch Key Rename

**The race condition:** The Claude Code hooks call `/store` which performs `INCR recall:global_epoch` at session end. If a hook fires between the moment config is updated (pointing to new key) and before `RENAME` executes, the hook INCRs the new key from 0 — losing the current epoch value. Conversely if the hook fires after config points to old key but the RENAME has already moved it, the INCR goes to the new key name correctly.

**The safe ordering — config-last:**

```
Phase A: Deploy new app code with tenant_id support
         BUT config still reads REDIS_EPOCH_KEY = "recall:global_epoch"
         (new code runs, uses old key name — no behavior change)

Phase B: RENAME the Redis key atomically
         redis-cli RENAME recall:global_epoch recall:tim:global_epoch
         (any in-flight INCR on old key name fails with WRONGTYPE or
          succeeds on the new name — see below)

Phase C: Update config: REDIS_EPOCH_KEY = "recall:tim:global_epoch"
         Reload FastAPI (or hot-reload if using configmap)
```

**Why RENAME is atomic:** Redis `RENAME src dst` is a single atomic operation. No hook can INCR the old key in the middle of a RENAME — the RENAME is either complete or not. After RENAME, any hook that fires and attempts `INCR recall:global_epoch` will receive `(error) ERR no such key`. This is a detectable, hard failure — the hook logs an error. This is safer than a silent incorrect-tenant assignment.

**Ghost INCR detection:** If a hook fires and INCRs `recall:global_epoch` after the RENAME (creating the old key at epoch 1):

```bash
# After config update, check for ghost key:
redis-cli EXISTS recall:global_epoch
# If 1: ghost INCR occurred — the new key also received the correct value
# Delete the ghost key:
redis-cli DEL recall:global_epoch
```

**Alternative safe ordering — config-first:**

```
Phase A: Update config: REDIS_EPOCH_KEY = "recall:tim:global_epoch"
         Reload FastAPI — hooks now INCR new key (which does not exist yet)
         Redis INCR on non-existent key initializes to 0 then INCRs to 1
         (hooks succeed, but epoch resets to 1 — LOSING current epoch value)
```

**Verdict: config-first is UNSAFE.** It loses the current epoch value the moment a hook fires against the new key. The config-last ordering (Phase A: deploy → Phase B: RENAME → Phase C: config update) is correct. The window where a hook fails with "no such key" is the 1–2 second window between RENAME and config reload — acceptable for a non-realtime system (hooks log errors; they do not crash).

**Minimum viable migration (3 commands):**

```bash
# 1. Deploy new code (still reads old key name from config)
docker compose restart recall_api

# 2. Atomic key rename (instant, no data loss)
docker exec recall_redis redis-cli RENAME recall:global_epoch recall:tim:global_epoch

# 3. Update config and reload
# Set REDIS_EPOCH_KEY=recall:tim:global_epoch in docker-compose.yml env or .env file
docker compose restart recall_api

# 4. Initialize Sadie's epoch key
docker exec recall_redis redis-cli SET recall:sadie:global_epoch 0

# 5. Ghost key cleanup (run 30 seconds after step 3)
docker exec recall_redis redis-cli DEL recall:global_epoch
```

### Qdrant Collection Rename: `recall` → `recall_tim`

This is the highest-risk step. Qdrant has no native RENAME operation. The naive approach (create `recall_tim`, copy, switch, delete `recall`) creates a window where writes to `recall` are not reflected in `recall_tim`.

**The alias approach — atomic, zero-copy-window:**

Qdrant supports collection aliases (`POST /collections/aliases`). An alias is a name that resolves to a collection. The rename becomes:

```
1. Create alias "recall" → points to collection "recall" (existing — no-op if alias already exists)
2. Create new collection "recall_tim" with same vector config as "recall"
3. Copy all points from "recall" to "recall_tim" (scroll + upsert batch)
4. Switch alias "recall" → points to "recall_tim" (atomic single API call)
5. Delete original "recall" collection
```

**The alias switch is atomic.** Between step 3 and step 4, all writes go to the alias "recall" (which still points to the original collection "recall"). After step 4, the alias resolves to "recall_tim". No write is lost: writes before step 4 land in the original collection (which was the copy source) — they are already in `recall_tim` from the copy. Writes after step 4 land in `recall_tim` directly.

**Wait — the copy window problem:** Points written between the start of the copy (step 3) and the alias switch (step 4) exist in the original `recall` collection but may not have been included in the scroll batch. At 360 points and Tim's write rate (1–5 writes per session, sessions every few hours), the copy takes under 5 seconds. The probability of a write landing during the 5-second copy window is negligible but non-zero.

**Correct copy procedure:**

```python
# Step 3a: Snapshot (Qdrant creates a point-in-time backup)
POST /collections/recall/snapshots
# Returns snapshot_id

# Step 3b: Create recall_tim with same config
PUT /collections/recall_tim
{ "vectors": { "size": 768, "distance": "Cosine" } }  # match recall's config

# Step 3c: Restore from snapshot into recall_tim
POST /collections/recall_tim/snapshots/upload
# Upload the snapshot file

# Step 3d: Any writes that landed after snapshot but before alias switch:
# Query recall for points with id > max_id_in_snapshot (use scroll with offset)
# Upsert only the delta into recall_tim

# Step 4: Atomic alias switch
POST /collections/aliases
{
  "actions": [
    {
      "change_aliases": {
        "alias_name": "recall",
        "collection_name": "recall_tim"
      }
    }
  ]
}

# Step 5: Delete original "recall" collection (after 7-day retention)
DELETE /collections/recall  # keep as backup for 7 days
```

**Simpler approach for Tim's scale (360 points, <5 writes/day):**

Given 360 points and 1–5 writes per day, the copy window risk is effectively zero in practice. The pragmatic procedure:

```python
from qdrant_client import QdrantClient

client = QdrantClient("192.168.50.19", port=6333)

# 1. Create recall_tim
client.recreate_collection("recall_tim", vectors_config=VectorParams(size=768, distance=Distance.COSINE))

# 2. Copy all points (scroll + upsert)
offset = None
while True:
    points, offset = client.scroll("recall", limit=100, offset=offset, with_vectors=True)
    if not points:
        break
    client.upsert("recall_tim", points=points)
    if offset is None:
        break

# 3. Create alias (now alias "recall" → "recall_tim")
client.update_collection_aliases(
    change_aliases_operations=[
        ChangeAliasOperation(change_aliases=ChangeAliasesOperation(
            alias_name="recall",
            collection_name="recall_tim"
        ))
    ]
)
# From this moment, all reads/writes via alias "recall" go to "recall_tim"
# The old "recall" collection still exists (not pointed to by any alias)
```

**Key insight:** Once the alias points to `recall_tim`, the application code needs zero changes if it uses the alias name "recall" rather than the collection name directly. Only the migration to multi-tenant (adding `recall_sadie`) requires a code change.

### FastAPI Backward-Compatible Changes

Deploy new code **before** any schema migration executes. The new code must be backward-compatible with both old schema (no tenant_id column yet) and new schema (tenant_id column present).

**Deployment order:**

```
1. Deploy new FastAPI code (tenant_id param with default='tim', queries include WHERE tenant_id=:tid)
   — At this point, the queries fail if tenant_id column doesn't exist yet
   — Solution: use a feature flag or startup check

2. Run PostgreSQL migrations (ADD COLUMN steps above)
   — Now the new queries work

3. RENAME Redis key (as above)

4. Switch Qdrant alias (as above)
```

**Safer: migration-first, code-second:**

Run all DDL migrations first (ADD COLUMN, indexes), then deploy the new FastAPI code. The old code has no `WHERE tenant_id = :tid` predicates, so it continues working against the new schema (the column exists but is unused). After migration completes, deploy new code. Zero incompatibility window.

```
1. Run SQL migrations (ADD COLUMN on memories and co_retrievals, CREATE INDEX CONCURRENTLY)
   — Old FastAPI code still running, ignores tenant_id column — works correctly
   — Tim's hooks continue working uninterrupted

2. RENAME Redis key
   — 1-2 second window where hooks fail on epoch INCR (log error, non-fatal)

3. Switch Qdrant alias (atomic, zero-downtime)

4. Deploy new FastAPI code with tenant_id support
   — Now all four changes are live simultaneously
   — Tim's existing hooks work with default='tim' (no hook code changes)

5. Initialize Sadie's infrastructure (recall_sadie collection, recall:sadie:global_epoch key)
   — Only needed when Sadie's Family Hub integration begins
```

**FastAPI endpoint changes (backward-compatible):**

```python
@app.get("/retrieve")
async def retrieve(
    query: str,
    tenant_id: str = Query(default="tim", pattern="^[a-z_]{1,32}$"),

):
    # Route to tenant collection: f"recall_{tenant_id}" or alias
    collection = f"recall_{tenant_id}"

    memories = await db.fetch_all(
        "SELECT * FROM memories WHERE tenant_id = :tid AND ...",
        {"tid": tenant_id}
    )
```

Existing hooks calling `/retrieve?query=...` (no tenant_id param) receive `tenant_id='tim'` by default — zero behavior change.

## Ideas Extracted

### Idea: qdrant-alias-atomic-rename
- **Novelty**: 0.75 — Using Qdrant collection aliases as an atomic cutover mechanism for the `recall` → `recall_tim` rename avoids the dual-write window that would otherwise require careful coordination between application code and the collection copy job. The insight that Qdrant's alias switch is a single atomic API call — functionally equivalent to a DNS TTL-0 cutover — is not prominently documented in Qdrant's multi-tenant migration guides. Most treatments recommend dual-write during copy, which introduces write-ordering complexity. The alias approach makes the copy phase read-only and the cutover instantaneous.
- **Evidence**: 0.72 — Qdrant's alias API (`POST /collections/aliases`) is documented and available in all Qdrant versions used in production deployments. The atomicity guarantee follows from Qdrant's internal alias table update being a single metadata write operation, equivalent to a file system symlink swap. The claim that writes during the copy phase land in the alias target (original collection) and are included in the scroll-based copy depends on the scroll offset being exhaustive — verified by Qdrant's scroll API semantics (returns all points with id > offset until empty result).
- **Feasibility**: 0.90 — The alias API is available in the Qdrant Python client (`update_collection_aliases`) and REST API. At Tim's scale (360 points), the scroll+upsert copy completes in under 5 seconds. The only failure mode is a write landing after the final scroll batch but before the alias switch — at Tim's write rate (1–5/day), this has a < 0.003% probability per migration attempt. Acceptable without a snapshot-based approach.

### Idea: migration-before-code-deployment-ordering
- **Novelty**: 0.68 — The insight that SQL DDL migrations should precede application code deployment (not follow it) when the DDL adds columns with a safe default — because old code is compatible with the new schema (it ignores the new column) but new code is incompatible with the old schema (it references a column that does not exist) — is a precise formulation of the "backward-compatible migration" deployment ordering principle. Applied to the tenant_id migration: the old FastAPI code running against a schema that includes tenant_id simply ignores it. The new FastAPI code running against a schema without tenant_id raises a ColumnDoesNotExist error. Therefore, DDL first → code second is the only safe ordering.
- **Evidence**: 0.72 — This is a well-established principle in production database migration engineering (Flyway, Liquibase, and Django's migration docs all recommend it). The novel element is its application to the specific tenant_id ADD COLUMN case where the old code's SELECT and INSERT statements do not reference the new column at all — making old-code-on-new-schema a perfect no-op. The reverse (new-code-on-old-schema) is a hard failure. Therefore the ordering is forced, not a choice.
- **Feasibility**: 0.95 — Trivially achievable: run the PostgreSQL migration script first (the CONCURRENTLY operations allow this without downtime), wait for completion, then deploy the new Docker image. The entire PostgreSQL migration at Tim's scale completes in under 30 seconds.

### Idea: redis-rename-config-last-ordering
- **Novelty**: 0.70 — The specific ordering for Redis key rename — deploy code first (reading old key), RENAME key atomically, update config last — prevents two distinct failure modes: (1) the config-first failure where new code INCRs a non-existent key and resets epoch to 1 (silent data corruption), and (2) the code-last failure where old code fails with "no such key" for longer than necessary. The config-last ordering constrains the error window to the 1–2 second FastAPI restart interval after RENAME, turning the failure mode from silent data corruption into a logged, recoverable error. This precise three-phase ordering is not documented in standard Redis migration guides.
- **Evidence**: 0.68 — The analysis follows from Redis RENAME semantics (atomic, documented in Redis commands reference) combined with INCR-on-missing-key behavior (initializes to 0 then increments — documented). The ghost key detection step (EXISTS + DEL after config update) is a standard defensive cleanup pattern. The 1–2 second error window estimate is based on Docker restart time for a FastAPI container, which is observable and bounded.
- **Feasibility**: 0.92 — Three shell commands (RENAME, config file edit, docker compose restart). The ghost key cleanup is one additional command. Total execution time under 30 seconds. The procedure is reversible: if something goes wrong, RENAME back (`RENAME recall:tim:global_epoch recall:global_epoch`), revert config, restart.

### Idea: concurrently-index-operations-for-live-migrations
- **Novelty**: 0.65 — Using `CREATE INDEX CONCURRENTLY` and `DROP INDEX CONCURRENTLY` as the standard tool for zero-downtime index management during live schema migrations is well-known but underused in homelab contexts where developers assume short lock durations are acceptable. The specific application here — dropping the non-tenant-scoped `idx_co_ret_fresh` and replacing with tenant-scoped `idx_co_ret_fresh_tim` while the system is serving Claude Code hooks — demonstrates that the CONCURRENTLY mechanism eliminates all write-blocking risk even for index operations that change the partial index predicate. The key insight: `CREATE INDEX CONCURRENTLY` and `DROP INDEX CONCURRENTLY` can be interleaved safely because both the old and new indexes are valid during the transition (the old index covers all FRESH rows, the new index covers all FRESH+tim rows — at this stage all rows ARE tim, so coverage is identical).
- **Evidence**: 0.72 — PostgreSQL documentation for CREATE INDEX CONCURRENTLY explicitly states that the index build does not hold a write-blocking lock. DROP INDEX CONCURRENTLY is documented in PostgreSQL 9.2+. The claim that old and new indexes provide identical coverage during migration (when tenant_id='tim' for all rows) follows from the migration sequence: tenant_id is set to 'tim' for all rows before the index swap begins.
- **Feasibility**: 0.95 — Standard PostgreSQL DDL syntax, no extensions required. Available on postgres:15-alpine. The only constraint: CONCURRENTLY operations cannot run inside a transaction block. They must be issued as standalone statements. FastAPI's typical database migration setup (Alembic) supports this via the `execute_many` mode with `transaction=False`.

## Key Finding

**Verdict: BREAKTHROUGH.** The online multi-tenant migration for Recall can complete in approximately 30 minutes with zero dropped writes and zero incorrect tenant assignments, using four mechanisms that align with each store's atomicity model:

1. **PostgreSQL**: ADD COLUMN (nullable) → UPDATE → SET NOT NULL → CREATE INDEX CONCURRENTLY. Deploy this before new application code. The nullable-first approach eliminates table-rewrite risk on any PostgreSQL version.

2. **Redis**: Three-phase ordering (deploy code → RENAME → update config). The RENAME is atomic; the only risk is a 1–2 second window where hooks receive "no such key" errors — logged, not silent. Ghost key cleanup eliminates any epoch counter divergence.

3. **Qdrant**: Collection aliases make the `recall` → `recall_tim` rename an instantaneous, atomic metadata operation. No dual-write window. No points lost. The alias approach is the highest-impact insight in this migration — it converts an otherwise-complex dual-write coordination problem into a single API call.

4. **FastAPI**: `tenant_id: str = Query(default="tim")` provides full backward compatibility. Tim's existing hooks require zero changes. The system becomes multi-tenant-capable before Sadie's integration begins.

The critical ordering constraint: **SQL migrations first, application code second, Redis rename during brief restart window**. This ordering ensures that at every moment during migration, at least one of {old code, new code} is compatible with the schema state that exists.
