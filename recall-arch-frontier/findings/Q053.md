# Finding: Q053 — Absence Mapping: Session-Boundary-Driven Memory Lifecycle Management

**Question**: Has any personal knowledge management or developer-tools AI assistant implemented session-boundary-driven memory lifecycle management — specifically, the pattern of tagging memories with a session_id at write time and then bulk-deleting low-importance session-scoped memories at session close — in production?
**Question Type**: ABSENCE
**Verdict**: BREAKTHROUGH
**Severity**: High
**Source field**: Database compaction (TWCS window-boundary drops), operating system memory management (page lifecycle), sleep-stage memory consolidation (neuroscience)

---

## Mechanism Under Investigation

A production memory system that (1) writes a `session_id` tag onto every stored memory at creation time, (2) applies a quality or importance threshold at session-close time, and (3) executes a bulk DELETE of all session-scoped memories below that threshold in a single database operation — as distinct from TTL-based expiry (time-driven, not session-boundary-driven), manual deletion (user-initiated), or LLM-driven consolidation (content-reasoning-based, not threshold-gated).

The three-part compound is the distinguishing criterion:
- **Part A**: session_id written at memory-creation time (scope tagging)
- **Part B**: quality/importance threshold evaluated at session-end (not continuously, not on schedule)
- **Part C**: bulk DELETE of sub-threshold session memories (one filter-delete, not per-record)

Any system missing any one of the three parts does not satisfy the mechanism.

## Absence Classification

**Classification**: TRUE ABSENCE

**Explanation**: After checking all ten specified systems plus adjacent frameworks (Cognee, LangChain, Google ADK, OpenAI Agents SDK, OpenClaw), no production system implements the three-part compound mechanism. Individual parts appear in isolation:

**Part A (session_id at write time)** is present in several systems. mem0 uses `session_id` / `run_id` for scoping. AWS Bedrock uses caller-supplied session identifiers. Zep sessions tag messages. But tagging for scoping purposes is not the same as tagging for lifecycle management — in every case found, the session_id is used to scope retrieval, not to trigger deletion at a boundary.

**Part B (quality/importance threshold at session end)** does not appear in any production system as an automatic trigger. The Letta sleeptime community forum thread describes a reference-count-based promotion rule ("referenced 3+ times → promote") as a community heuristic, not a shipped feature. OpenClaw lists importance scoring as a future enhancement explicitly. No production system evaluates a quality or importance threshold automatically when a session closes.

**Part C (bulk DELETE of sub-threshold session memories)** is the most absent of the three. Every system that supports deletion either requires user action, uses a time-based TTL (AWS: 1-365 days; GitHub Copilot: 28 days; Vertex AI: configurable TTL), or triggers an LLM-driven consolidation pass (Letta sleeptime). No system runs a filter-DELETE against the session_id + score < threshold predicate as an automatic session-close hook.

**Why this gap exists — three compounding reasons:**

1. **No session-close event in most frameworks.** Most production AI memory systems are stateless between calls. There is no session lifecycle in the OS/database sense — no moment where the system knows "this session is definitively over." The closest approximations are either caller-declared (`endSessions=true` in AWS Bedrock) or timeout-based. Without a reliable session-close event, the trigger mechanism for Part B/C has nothing to attach to.

2. **Importance scoring at write time is not standard.** Systems that do assign importance scores (mem0's LLM-driven UPDATE/ADD/DELETE decision at write time) use the score to decide whether to store, not as a lifecycle tag for later deletion. The concept of "store now, evaluate importance at session end against a threshold" requires a deferred evaluation model that no production system has implemented. The field's mental model is: evaluate now or not at all.

3. **The efficiency argument has not been made visibly.** The optimization insight — that bulk filter-DELETE on (session_id, score < threshold) is dramatically cheaper than per-record processing — is not in any public documentation reviewed. The database efficiency argument (one DELETE WHERE query vs. N individual deletes) is obvious to database engineers but has not been applied to AI memory systems in any production codebase or paper found. The session-scope window-drop from Q024's 2032 retrospective attributes this to "the TWCS window-boundary drop from Q001-F1" — a pattern from Cassandra's time-windowed compaction strategy that has not crossed over into AI memory system design.

**Stealth implementation caveat:** Proprietary systems (OpenAI's memory in ChatGPT, Anthropic's memory layer if any, Google's internal Gemini memory) may implement this internally. ChatGPT's memory now reportedly "moves less important memories to the background" (GPT-5.2 auto-prioritization), which could be a threshold-gated session operation — but the mechanism is undocumented and unverifiable from public sources. This is classified as an unverifiable uncertainty rather than a confirmed absence, but there is no public evidence it operates on session boundaries with bulk deletes.

## Ideas Extracted

### Idea: session-scope-tag-and-sweep

- **Novelty**: 0.82
- **Evidence**: 0.85
- **Feasibility**: 0.93
- **Description**: Write a `session_id` tag and an initial `importance_prior` onto every stored memory at creation time, then execute a single SQL/Qdrant filter-delete at session-close time: `DELETE WHERE session_id = X AND importance < threshold`. The initial prior is derived from a rule-based classifier (entity type, topic category, explicit vs. inferred fact) — not an LLM call. The threshold is calibrated from historical retrieval rates for that domain/topic. This is the minimum viable version of the session-scope-window-drop mechanism from Q024's 2032 retrospective. It requires: (1) adding `session_id` and `importance_prior` columns to the memory schema, (2) a session-close hook that fires the filter-delete, (3) a rule-based prior classifier (50-100 lines of code, no LLM). Expected outcome: 35-60% reduction in cold-store growth rate for users with daily sessions, because session-scoped ephemeral memories constitute approximately 40% of all stored memories by volume (from Q024). Feasibility is very high because all three components are straightforward to implement in an existing system — this is primarily a missing design decision, not a missing technology.

**Quality score**: 0.82 × 0.40 + 0.85 × 0.35 + 0.93 × 0.25 = 0.328 + 0.298 + 0.233 = **0.859** → BREAKTHROUGH

### Idea: deferred-importance-evaluation-at-session-boundary

- **Novelty**: 0.77
- **Evidence**: 0.72
- **Feasibility**: 0.85
- **Description**: Rather than evaluating memory importance at write time (the current industry default), defer importance evaluation to session close. At write time, store a `write_context` embedding (embedding of the message that triggered the memory) and a `category` tag (rule-based). At session close, compute retrieval frequency within the session: any memory retrieved at least once during the session gets promoted; unretrieved memories are evaluated against the prior only. This produces a better importance estimate than write-time scoring because it incorporates within-session evidence: a memory that the agent immediately retrieved in the same session has demonstrated utility. A memory that was stored but never retrieved in the session is a candidate for deletion if its prior is also low. The mechanism is: retrieve frequency within session + write-time prior → final session-end score → DELETE if below threshold. No LLM required. This generalizes beyond session-scope to any scope boundary (project-scope, topic-scope) where there is a definable "end" event.

**Quality score**: 0.77 × 0.40 + 0.72 × 0.35 + 0.85 × 0.25 = 0.308 + 0.252 + 0.213 = **0.773** → BREAKTHROUGH

---

### Idea: session-close-hook-as-standard-memory-API-primitive

- **Novelty**: 0.70
- **Evidence**: 0.80
- **Feasibility**: 0.88
- **Description**: The absence of session-boundary-driven lifecycle management in every production system reviewed traces to a single missing API primitive: a session-close hook. AWS Bedrock comes closest with `endSessions=true` as a caller-supplied flag on the final invocation, but this is caller-declared and not used to trigger memory operations. Google ADK has `add_session_to_memory()` as a manual call. The missing standard primitive is a callback-based session-close hook that every memory write within the session is registered against. When the hook fires (caller-declared end, idle timeout, or explicit close), the registered session-id triggers the lifecycle sweep. This is the equivalent of a database transaction commit callback — a standard primitive in database systems that is absent from every AI memory framework reviewed. Implementing this as a hook in any of the open-source frameworks (mem0, Zep, LangChain) is a 1-2 day implementation. The architectural contribution is naming and standardizing the primitive, not the implementation complexity.

**Quality score**: 0.70 × 0.40 + 0.80 × 0.35 + 0.88 × 0.25 = 0.280 + 0.280 + 0.220 = **0.780** → BREAKTHROUGH

## Suggested Follow-ups

- **[IMPLEMENTATION]**: Implement `session-scope-tag-and-sweep` in the System-Recall stack. The minimum viable version requires: (1) add `session_id VARCHAR` and `importance_prior FLOAT` to the Qdrant payload schema, (2) populate both at write time in the existing `recall-session-summary.js` hook using a rule-based prior classifier, (3) add a sweep step to `recall-session-summary.js` that calls `qdrant.delete(filter={session_id: X, importance_prior: {lt: threshold}})` before the hook exits. Estimated implementation time: 4-6 hours. This directly operationalizes the Q024 session-scope-window-drop mechanism.

- **[VALIDATION]**: Before implementing, measure the actual proportion of session-scoped vs. long-lived memories in the existing System-Recall corpus. Query Qdrant for all memories stored in the last 30 sessions and calculate what fraction were retrieved in subsequent sessions vs. never retrieved again. If the "never retrieved again" fraction exceeds 30%, the case for sweeping is validated empirically.

- **[THRESHOLD CALIBRATION]**: The deletion threshold for `importance_prior` must be calibrated to minimize false-positive deletes (deleting a memory that would have been useful later). The calibration data is available in the existing retrieval logs: memories with prior below X that were subsequently retrieved represent false positives. Run this analysis on System-Recall's existing Neo4j access log data to determine the threshold that produces false-positive rate < 5%.

- **[ABSENCE CHALLENGE]**: The one stealth-implementation risk is ChatGPT's undocumented "auto-prioritization" memory feature (GPT-5.2). Monitor OpenAI's changelog and published research for any description of session-boundary-driven memory sweeps. If found, reclassify this finding from TRUE ABSENCE to PARTIAL ABSENCE.

- **[EXTENSION]**: The session-close hook primitive (Idea 3) could be contributed as an open-source PR to mem0 or Zep. The PR adds: `MemoryClient.on_session_close(session_id, callback)` where the callback receives the list of session-tagged memories and their scores. The mem0 codebase is Python/MIT-licensed and the architecture is modular — this is a realistic open-source contribution that would also validate the mechanism in a production system other than System-Recall.

---

## Summary

The three-part compound mechanism — session_id at write time + quality threshold at session end + bulk DELETE — does not exist in any production AI memory system reviewed as of March 2026. The absence is not explained by the mechanism being technically infeasible (it is extremely feasible, estimated 4-6 hours to implement in System-Recall). The absence is explained by three converging factors: (1) most frameworks lack a reliable session-close event to attach the trigger to, (2) the field's mental model treats importance evaluation as a write-time decision rather than a deferred decision, and (3) the database efficiency argument for filter-delete over per-record processing has not been applied to AI memory systems. The mechanism is implementable today with existing infrastructure components and has no known prior art in production systems.
