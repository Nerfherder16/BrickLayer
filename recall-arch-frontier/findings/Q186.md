### Eigenvalue Analysis: CO_RETRIEVED Weight Update Matrix

**Update model specification:**

Let W be the N×N CO_RETRIEVED weight matrix (N=22,423 memories). The update rule per week is:

```
W(t+1) = gamma × W(t) + delta × C(t)
```

Where:
- gamma = 0.95 (Q178 decay factor per week)
- delta = increment per co_retrieval event (Q167 spec does not define this; assume delta=0.05 per organic co_retrieval)
- C(t) = binary co-occurrence matrix for week t (C_ij = 1 if memories i and j were co-retrieved)

**Fixed-point solution:**

At steady state: W_ss = delta × C_expected / (1 - gamma) = delta × C_expected / 0.05 = 20 × delta × C_expected

For organic usage (k=5 retrievals per session, 1 session per day, N=22,423):
- Expected organic pairs per session: k(k-1)/2 = 10
- Expected pairs per week: 10 × 7 = 70 unique pairs (0.3% of N)
- Expected organic C_ij ≈ 70/N² = 3.7e-7 per element
- Organic steady-state edge weight: W_ss_organic = 20 × 0.05 × 1 = 1.0 (for frequently co-retrieved pairs)

**Stability condition — spectral radius analysis:**

The update operator A = gamma × I (where I is the N×N identity, since each edge updates independently). The spectral radius of A is:

```
rho(A) = gamma = 0.95 < 1.0
```

**The system is globally asymptotically stable.** For ANY initial weights and ANY co-retrieval pattern (including adversarial), W(t) converges to the fixed point W_ss = 20 × delta × C_ss where C_ss is the time-averaged co-occurrence rate. No eigenvalue instability is possible.

**Critical reframe**: The feedback loop is STABLE but AMPLIFYING. The fixed-point weight for an adversarially repeated pair is:

```
w_ss(adversarial) = 20 × delta × k_per_week
```

Where k_per_week = adversarial co_retrievals per week for the target pair.

| k_per_week | w_ss (delta=0.05) | Ratio to organic (w_ss=1.0) |
|---|---|---|
| 1 (organic) | 1.0 | 1x baseline |
| 5 | 5.0 | 5x |
| 10 | 10.0 | 10x |
| 20 | 20.0 | 20x |

A pair retrieved together 10× per week stabilizes at w=10.0. This dominates co_retrieval_gravity for both memories — effectively hijacking the reranker for any query touching either memory.

### Adversarial Attack Scenarios (Tim's Single-User Context)

For a solo developer (Tim) rather than multi-user bot traffic, the relevant risk is **organic self-reinforcing loops**, not malicious adversaries:

**Scenario 1 — Topic concentration sessions**: Tim works on the same topic for 3 hours → 20+ retrievals per session → all same-topic pairs get credited → intra-topic w rises to 20× cross-topic w → the reranker preferentially surfaces topic-related memories in future sessions even for unrelated queries.

**Scenario 2 — Session clustering artifact**: In sessions where Tim asks about memory architecture, the same 5-10 memories appear together repeatedly. Within 4 weeks, their pairwise w reaches 5.0. These pairs then dominate co_retrieval_gravity in all architecture-related sessions, creating a self-reinforcing retrieval cluster that ignores new architecture memories.

**Scenario 3 — k² session amplification**: The co_retrieval pair count per session = k(k-1)/2. A session with k=10 retrievals generates 45 co_retrieval pairs. A session with k=20 generates 190 pairs — 4.2× more. High-retrieval sessions disproportionately reinforce all their pairwise edges simultaneously. If Tim's retrieval sessions have variable k, high-k sessions dominate the co_retrieval graph.

### Guards from Adjacent Field

**Guard G1 — Per-pair per-session deduplication (Temporal Bloom filter)**:
The standard click-fraud mitigation: a (A, B) pair can only be counted once per session, regardless of how many times A and B appear together. This caps the session-level amplification from k² to k(k-1)/2 (the natural maximum), and eliminates Scenario 1's per-session concentration effect.

Implementation: In the flush worker (Q167), deduplicate the co_retrieval outbox by (memory_id_a, memory_id_b, session_id) before counting — only count each pair once per session. The outbox already has session_id context from the retrieval log.

**Guard G2 — k-normalization: divide by session retrieval count**:
Weight increment per pair = delta / k where k is the session's retrieval count. This ensures high-k sessions don't disproportionately amplify co_retrieval edges. A 20-retrieval session contributes the same total weight as a 5-retrieval session, distributed across more pairs.

Implementation: In the flush worker, compute k for each session (COUNT of retrievals in the session from injection_log), then use delta/k as the increment. One additional SQL COUNT query per session batch.

**Guard G3 — Temporal velocity gate**:
Alert if any single edge w(A,B) increases by more than `max_increment_per_window` in a rolling 7-day window. Threshold: 3× the expected organic increment rate. For Tim: expected organic increment = 70 pairs/week × 0.05/pair = 3.5 total weight added across all edges per week → average per-edge = 3.5/N² ≈ negligible. A single edge gaining more than 0.5 per week signals anomalous concentration.

Implementation: Add a `weekly_increment` column to CO_RETRIEVED relationships, updated by the decay worker. Alert in health.py if max(weekly_increment) > threshold.

## Ideas Extracted

### Idea: per-pair-per-session-deduplication-guard
- **Novelty**: 0.80 — No reviewed AI memory system (MemGPT, mem0, Zep) implements co_retrieval deduplication at the session level. The temporal Bloom filter pattern from click-fraud detection is well-established but has never been applied to AI memory co_retrieval systems.
- **Evidence**: 0.90 — Temporal stateful Bloom filter achieves 99% accuracy on click-stream deduplication (IJDATS 2012). Per-session deduplication is the standard first-line guard in recommender systems against session-level amplification. The Q167 flush worker's FOR UPDATE SKIP LOCKED pattern already provides idempotency at the row level; session-level dedup adds one more GROUP BY clause.
- **Feasibility**: 0.92 — One SQL change in the flush worker: `GROUP BY memory_id_a, memory_id_b, session_id` before counting. The outbox table (from Q167 spec) stores session_id alongside each co_retrieval record. No new infrastructure.
- **Description**: In the flush worker (Q167), change the aggregation from `GROUP BY memory_id_a, memory_id_b` to `GROUP BY memory_id_a, memory_id_b, session_id HAVING COUNT(*) >= 1`. Then collapse session-level counts to unique-session counts: `COUNT(DISTINCT session_id)` as the increment multiplier. This ensures that k=20 sessions don't generate 190 pairs and k=5 sessions generate 10 — both generate the same number of unique session credits per pair. Eliminates session-concentration amplification completely.

### Idea: k-normalized-co-retrieval-weight-increment
- **Novelty**: 0.85 — Session-size normalization of co_retrieval weight increments is not documented in any reviewed AI memory system. The adjacent field analogy is Jaccard-normalized co-occurrence in recommender systems: |A∩B| / |A∪B| penalizes large itemsets that generate many spurious pairs.
- **Evidence**: 0.85 — Jaccard normalization of co-occurrence matrices is standard in collaborative filtering (Sarwar et al., 2001 item-based CF). The equivalent for session-based co_retrieval: delta_ij = 1 / (k_i + k_j - overlap_ij) normalizes for session size. A simpler approximation: delta_ij = 2 / k where k is the session retrieval count.
- **Feasibility**: 0.85 — Requires injecting session retrieval count into the flush worker batch. The injection_log table already has session_id and retrieval timestamps; a `SELECT COUNT(*) FROM injection_log WHERE session_id = X` query provides k per session. Adds one COUNT query per session batch to the flush worker (~1ms overhead). The weight increment becomes `2/k` instead of a fixed delta.
- **Description**: Modify the flush worker (Q167) to divide the weight increment by k (session retrieval count). For k=5: delta=0.40, for k=10: delta=0.20, for k=20: delta=0.10. Steady-state weight for any pair is now proportional to the fraction of sessions in which they co-occur, not the raw count. This is the correct normalization: a pair that co-occurs in 50% of Tim's sessions should have w_ss=1.0 regardless of whether those sessions had k=5 or k=20 retrievals.

### Idea: co-retrieval-velocity-gate-health-signal-s7
- **Novelty**: 0.75 — Velocity gates on weight update matrices are used in production recommender systems (Netflix, Spotify item-item similarity). The novelty is in applying it as a health signal (S7) for the Recall health dashboard. This extends the S1-S6 signal set with a data-quality signal rather than an infrastructure signal.
- **Evidence**: 0.85 — Temporal velocity gating is the standard real-time anomaly detection technique for feedback loops in recommender systems (Wang et al. TMLR 2024). A weekly maximum increment threshold above 3-sigma of the organic distribution provides a reliable early warning for loop concentration without false positives at normal usage rates.
- **Feasibility**: 0.80 — Add `max_weekly_increment` and `p99_weekly_increment` to the weekly decay job output (already computing per-edge strength updates). Expose as S7 in the health dashboard via a single MAX/P99 Cypher query on CO_RETRIEVED relationships. Alert threshold: max_weekly_increment > 5× (5 standard deviations above mean).
- **Description**: S7 (co_retrieval velocity) = the maximum single-edge weight increment in the past 7 days. Add a `last_increment` property to CO_RETRIEVED relationships in the flush worker. The weekly decay job computes MAX(last_increment) and P99(last_increment). Alert if MAX > 5× mean. This detects session-concentration loops before they corrupt the reranker, surfacing the problem when w has grown 5x above baseline rather than after it has grown to 10-20x (at which point precision is already degraded).

## Analytical Summary

The CO_RETRIEVED feedback loop has spectral radius 0.95 — it is globally stable and CANNOT diverge or oscillate regardless of co_retrieval patterns. Every initial condition converges to a finite fixed point determined by the average co_retrieval rate.

The engineering concern is not instability but **steady-state amplification**: an adversarially concentrated retrieval pattern converges to w_ss = 20 × delta × k_per_week, which can be 10-20× the organic baseline. For Tim's single-user context, the realistic risk is organic self-reinforcing topic clusters, not malicious bot traffic.

Three independent guards address this: G1 (per-pair per-session dedup) eliminates session-level amplification entirely; G2 (k-normalization) ensures session size doesn't distort pair weights; G3 (velocity gate as S7) provides early-warning before the reranker precision degrades. G1 alone is sufficient for most realistic single-user scenarios. G2+G3 add defense-in-depth for high-k sessions and long-term drift detection.
