## Deployment Analysis

### (a) Can Q167 and Q168 Deploy in the Same Restart Window as Q164's Wave 18 Changes?

**Answer: Yes, with one constraint.**

Q167 (flush_co_retrievals worker) and Q168 (audit_log retention worker) both require:
- A new worker Python file dropped into `src/workers/`
- Import and registration additions in `src/workers/main.py`
- An ARQ cron entry

Neither change alters any existing function or modifies any existing table. Both are purely additive. They can be merged into the same git commit as Q150 and Q156's code changes, included in the same `git pull`, and activated by the same `docker compose restart recall-worker` that Q156 already requires.

The one substantive question is whether Q168's migration DDL (`_MIGRATE_AUDIT_LOG_RETENTION`) — which runs at PostgresStore.connect() — creates any ordering risk relative to Q156's `_MIGRATE_CO_RETRIEVALS_OUTBOX` migration.

**Assessment**: No conflict exists. Both migrations run inside `PostgresStore.connect()` as sequential `await conn.execute()` calls. Q168's migration adds two DDL statements:
1. `CREATE TABLE IF NOT EXISTS feedback_summary` — creates a new table; no lock on audit_log
2. `DROP INDEX IF EXISTS idx_audit_log_timestamp` + `CREATE INDEX IF NOT EXISTS idx_audit_log_timestamp_brin ON audit_log USING BRIN (timestamp)` — index swap on audit_log

The critical question for (a) is whether the BRIN index creation at connect() time conflicts with in-flight audit_log writes. It does not block reads on audit_log; it takes a brief ShareLock on the table during index construction, which completes in milliseconds on PostgreSQL 14+ (the BRIN index is approximately 100x smaller than the B-tree it replaces — construction is near-instant). Concurrent `log_audit()` INSERT calls queue behind the ShareLock momentarily and then proceed. This is identical risk to Q156's migration already in Q164's runbook.

**Conclusion for (a)**: Q167 and Q168 can be deployed in the same maintenance window as Q164's Wave 18 changes. The combined deployment is: one git pull, one `docker compose restart recall-api recall-worker`. Q168's migration runs alongside Q156's migration in the same connect() sequence. No separate maintenance window is required for Wave 19.

However, there is a **code ordering dependency within the commit**: `models.py` must include the `CO_RETRIEVED = "co_retrieved"` enum addition (Q167 File 1) before the flush worker is importable. If these land in separate commits, the import at `main.py` line 28 would fail on startup until models.py is updated. Since both changes ship in the same git pull, this is automatically satisfied.

### (b) Does Q170's Bootstrap Script Require Q167 to Be Deployed First?

**Answer: No. Q170 is fully independent of Q167.**

Q170's bootstrap clustering job:
- Reads embedding vectors from Qdrant (via scroll API — no Ollama calls needed since embeddings already exist)
- Runs K-means on the 22,423 × 1024 float32 matrix (4–5 seconds on N100, per Q172)
- Writes CO_RETRIEVED edges directly to Neo4j via batched MERGE (7–10 minutes nominal, per Q172)

Q167's flush worker:
- Drains the `co_retrievals_outbox` PostgreSQL table
- Promotes co-occurrence pairs to Neo4j CO_RETRIEVED edges on a 15-minute cron cadence
- Writes edges from live retrieval co-occurrence signals, not from clustering

These two write paths are independent. Both write CO_RETRIEVED edges to Neo4j using MERGE semantics. Q170 writes edges with initial strength 0.1 (bootstrap seed weight); Q167 writes edges with initial strength 0.55 (ON CREATE: 0.5 + 0.05 increment). Neo4j MERGE is idempotent: if Q170 has already written a CO_RETRIEVED edge between nodes A and B, and Q167 subsequently flushes an outbox row for the same pair, the ON MATCH SET branch fires and increments strength. No conflict, no data loss, no ordering requirement.

**The only practical reason to prefer deploying Q167 before running Q170**: after Q170 writes 1.25M bootstrap edges, Q167 begins reinforcing the edges that correspond to actual live co-retrievals. This reinforcement starts from the bootstrap weight (0.1) rather than from zero, which is the intended design. But this does not constitute a hard deployment dependency — Q170 can run before, after, or concurrent with Q167's first cron cycle.

**Practical sequencing recommendation**: Deploy Q167 as part of the main service restart. Run Q170 as the final step of the maintenance window, after the service is confirmed healthy. This order is preferable because it ensures Q167 is registered and healthy before the bootstrap edges are written — meaning Q167's first cron at :00/:15/:30/:45 will begin reinforcing those edges immediately, rather than the edges aging with no reinforcement until Q167 is deployed.

### (c) Does Q168's feedback_summary Migration Conflict with In-Flight audit_log Queries?

**Answer: No conflict. The DDL is safe to run during an active maintenance window.**

Q168 established: `CREATE TABLE IF NOT EXISTS feedback_summary` — a new table creation with no dependency on audit_log structure. This DDL is fully safe and takes no locks on audit_log.

The BRIN index swap on audit_log requires two DDL operations:
1. `DROP INDEX IF EXISTS idx_audit_log_timestamp` — acquires an AccessExclusiveLock on the index (not the table), releases in milliseconds. Concurrent queries that were using the B-tree index for range scans will briefly stall and then proceed (PostgreSQL re-plans). Concurrent INSERT statements on audit_log are not blocked — INSERT does not use the timestamp index.
2. `CREATE INDEX IF NOT EXISTS idx_audit_log_timestamp_brin ON audit_log USING BRIN (timestamp)` — acquires a ShareLock on audit_log during creation. BRIN index creation on a 400k-row monotonic table takes approximately 50–150ms (BRIN stores only min/max per 128-page range, not a per-row entry). Concurrent reads and writes queue behind the ShareLock for this brief window.

**The critical safety fact** established in Q168: the `feedback_summary` table DDL and the BRIN index creation are both wrapped in a `DO $$ BEGIN ... END $$;` PL/pgSQL block with `IF NOT EXISTS` guards. The migration is fully idempotent. If the connect() migration fails partway through (extremely unlikely for DDL-only work), the next connect() call re-attempts and succeeds without side effects.

**No query corruption risk**: the migration does not ALTER any column on audit_log, does not drop any data, and does not modify the B-tree index on `action` or `memory_id`. The only visible effect to in-flight queries is the brief ShareLock during BRIN construction — a pause measured in milliseconds, not seconds.

## Key Findings

1. **Q167 and Q168 are safe to collapse into Q164's Step 2 restart.** Both are purely additive. Both register new workers that do not interfere with existing cron slots. Q168's BRIN index creation takes milliseconds and takes no exclusive lock on audit_log. Q156 and Q168 migrations both run idempotently in the same `connect()` sequence. Zero interaction between them.

2. **Q170 is fully independent of Q167.** Both write CO_RETRIEVED edges to Neo4j using MERGE semantics. If Q170 runs before Q167's first cron, the edges are written at bootstrap weight 0.1. When Q167 subsequently flushes outbox rows for the same pairs, ON MATCH SET increments strength to 0.15. Correct behavior. If Q167 has already run first (which it will, since it fires every 15 minutes), bootstrap edges arrive as ON MATCH updates to organically-seeded edges. Also correct. No ordering requirement.

3. **The feedback_summary migration DDL does not conflict with in-flight audit_log queries.** CREATE TABLE IF NOT EXISTS takes no lock on audit_log. The BRIN index creation takes a brief ShareLock (~100ms) on audit_log, blocking concurrent writes only for that window. No data is modified, no columns are altered, no existing queries return wrong results.

4. **The combined seven-change deployment is approximately 65–75 minutes wall-clock** — under the 3-hour constraint by a factor of 2.5x. The original Q164 estimate of ~60 minutes for four changes scales to ~75 minutes for all seven because Q170's script dominates the incremental time (7–10 minutes for Neo4j writes) and verification steps add 5–10 minutes.

5. **The collapse from two maintenance windows to one is the key operational win.** Without this analysis, Q167 and Q168 would have been treated as a second maintenance window ("Wave 19 deployment"). This finding confirms they can ride the same `docker compose restart` as Q150/Q156, reducing operational overhead, reducing the blast radius of restart count, and making the entire seven-change set a single coordinated event rather than two separate windows.

6. **Q170 must be staged last.** Not because of a technical dependency, but because: (a) the bootstrap script runs against live Neo4j and if run before Q167 is registered and healthy, any Neo4j-side issue is harder to diagnose; (b) the Q172 physics analysis confirms 7–10 minutes of Neo4j write time — running this while the service restart is still being verified would be a distraction; (c) the flush worker should be confirmed healthy (firing at its first :00/:15/:30/:45 cron after restart) before initiating a 1.25M-edge batch write.

7. **feedback_summary starts empty after deployment.** This is correct. The retention worker's first run fires at 6:15am the next morning and performs the initial upsert of all feedback rows older than 90 days into feedback_summary before deleting them. No manual intervention is needed. The table being empty until the first cron does not affect any existing endpoint — `get_feedback_counts_by_memory()` continues to query audit_log directly and returns correct results. The feedback_summary table only becomes authoritative after the first deletion run removes rows from audit_log.

- **unified-wave-commit-strategy** (N=0.68, E=0.90, F=0.97): All five code changes (Q150, Q156, Q167, Q168, Q148) should be staged in a single git commit with a structured commit message listing all five changes. This allows `git revert <single-commit-hash>` to roll back the entire code deploy atomically if a critical issue is found post-deployment, rather than requiring five separate reverting operations. Q144 and Q170 are non-git operations (Qdrant PATCH and Python script) and are excluded.

- **arq-cron-slot-audit-before-wave-19** (N=0.65, E=0.88, F=0.99): With Q167 (minute={0,15,30,45}) and Q168 (hour=6, minute=15) both added, the ARQ cron schedule grows to 16 registered functions and 10+ cron entries. Before deploying Wave 19, enumerate all existing cron slots to confirm Q168's 6:15am slot does not collide with any other daily worker. Q168 notes that 6:15am is "the only open daily slot" (existing slots: 3:00, 3:15, 3:30, 4:00, 4:30, 5:00, 5:30 per Q168's finding #5). Independently verify this before deployment.

- **bootstrap-script-dry-run-first** (N=0.60, E=0.92, F=0.99): Q170's bootstrap script should support a `--dry-run` flag that performs the Qdrant scroll and K-means clustering but does not write to Neo4j, instead printing the edge count and estimated runtime. Running dry-run first confirms Qdrant connectivity, verifies the vector count matches the expected 22,423, and gives an accurate time estimate before committing the 7–10 minute write window. Dry-run cost: ~60 seconds (scroll + clustering only). No Neo4j writes.

- **s5-manual-trigger-post-step2** (N=0.55, E=0.85, F=0.99): After Step 2 confirms Q150 deployed (s5=UNKNOWN in /health), immediately trigger the reconcile ARQ task manually via Redis queue rather than waiting up to 7 days for the next Sunday 5:30am cron.
