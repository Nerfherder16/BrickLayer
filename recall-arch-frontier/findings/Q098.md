## Part 1: The Escalation Decision Tree

### S5 State Transitions

```
S5 HEALTHY (compliance_ratio >= 0.95)
    │
    │ compliance_ratio drops to [0.85, 0.95)
    ▼
S5 WARNING
    │ monitoring cadence: probe every session-start (standard)
    │ additional: run S5 probe manually every 30 minutes
    │
    │ Three consecutive probes in WARNING (without improvement)
    │     OR single probe below 0.85
    ▼
S5 CRITICAL (compliance_ratio < 0.85)
    │
    ├── Automated actions (no human required) ────────────────────────────────┐
    │   • Fire Discord CRITICAL notification (see Part 4)                     │
    │   • Record probe details to health audit log                            │
    │   • Suppress new LLPC sessions from injecting graph traversal results   │
    │     (degrade gracefully: fall back to Qdrant-only retrieval, skip L1.5) │
    │   • Run S5 probe again in 5 minutes to confirm persistence              │
    │                                                                          │
    └── Then: check S4 correlation ───────────────────────────────────────────┘
                │
                ├── S4 also alerting AND epoch_gap > 50?
                │       → PG_RESTORE_CONFIRMED_BY_GRAPH_STATE (Q084/Q092)
                │         Automated: emit structured alarm with Option A/B/C commands
                │         Manual (go/no-go required): DETACH DELETE + Qdrant cleanup
                │
                └── S4 not alerting OR epoch_gap <= 50?
                        → GRAPH_EPOCH_FILTER_GAP or TOMBSTONE_LEAK
                          Automated: emit diagnostic alarm
                          Manual (go/no-go required): limited Neo4j tombstone cleanup only
```

### WARNING Phase Monitoring Cadence

During S5 WARNING, the system should:

1. Continue standard per-session-start S5 probe (already running)
2. Enable an additional 30-minute polling interval via background task
3. Track the trajectory: is compliance_ratio stable, improving, or declining?
4. If 3 consecutive WARNING probes show declining compliance_ratio (each below the previous), escalate to CRITICAL pre-alert even before the 0.85 threshold is crossed

The 30-minute poll during WARNING serves two purposes: early CRITICAL detection (most PG-restore events drive compliance from WARNING to CRITICAL within a single session) and trajectory data for the Discord payload (how fast is compliance degrading?).

### CRITICAL Confirmation Before Action

Before executing any cleanup, the system must confirm S5 CRITICAL is not a measurement artifact:

```python
CONFIRMATION_PROTOCOL:
  1. Re-run S5 probe with doubled sample size (K=10 seeds, K=20 neighbors per seed)
     Rationale: standard K=5/K=10 can produce high-variance results on sparse graphs
  2. If re-probe is also < 0.85: CONFIRMED CRITICAL → proceed to notification + await go/no-go
  3. If re-probe is >= 0.85: LIKELY FALSE POSITIVE → downgrade to WARNING, log, continue monitoring
     Rationale: random seed sampling can occasionally return seeds biased toward orphan-heavy subgraphs
```

## Part 2: Automated Actions (Safe Without Human Confirmation)

The following actions are idempotent, non-destructive, or reversible without data loss. They execute automatically when S5 CRITICAL is confirmed.

### Action A1: Fire Discord Notification (see Part 4)
- **Safe**: read-only operation (queries health state, sends HTTP webhook)
- **Idempotent**: duplicate notifications are annoying, not harmful
- **TTL**: set a 30-minute re-alert timer; if Tim does not respond with go/no-go, fire again

### Action A2: Downgrade graph traversal to vector-only mode
```python
# In retrieval.py — activate at S5 CRITICAL
await redis.set("recall:retrieval_mode", "vector_only", ex=7200)  # 2-hour TTL
# LLPC reads this flag at session start; if set, skips L1.5 (Neo4j traversal)
# and uses Qdrant-only retrieval
# This is safe because: it reduces retrieval quality but prevents orphan injection
# It is reversible: TTL expires automatically or operator clears the key
```
- **Safe**: falls back to still-functional retrieval path
- **Reversible**: Redis key expires in 2 hours (or operator runs `DEL recall:retrieval_mode`)
- **Effect**: stops new S5 orphan injections while Tim decides on cleanup; S5 compliance_ratio becomes irrelevant during vector-only mode since graph traversal is not running

### Action A3: Run S4 probe immediately to determine co-alert status
- **Safe**: read-only query
- **Purpose**: populates the Discord notification with the correct correlation context (PG restore vs. graph maturity)

### Action A4: Count estimated orphans (read-only assessment)
```cypher
-- Run in Neo4j: count nodes that S5 would flag, without deleting anything
MATCH (n:Memory)
WHERE n.last_access_epoch > (current_epoch - 20)   -- working-set window
  AND n.last_access_epoch <= current_epoch
WITH n.id AS nid
// Batch-sample 200 to estimate miss rate without full enumeration
RETURN nid
LIMIT 200
```
Then batch-join against PG and compute estimated orphan count:
```python
# Sample 200 candidates from working-set; extrapolate orphan count to full corpus
sample_miss_rate = miss_count / 200
total_neo4j_nodes = await neo4j.run("MATCH (n:Memory) RETURN count(n) AS c")
estimated_orphans = int(total_neo4j_nodes * sample_miss_rate)
```
- **Safe**: read-only
- **Purpose**: populates Discord notification with estimated_orphan_count and corpus_impact_pct

### Action A5: Compute Qdrant orphan estimate
```python
# Check created_at distribution in Qdrant to estimate how many points
# were created after a suspected backup timestamp
# Use the epoch_gap from S4 (if S4 is alerting) to estimate backup timestamp:
#   epoch_gap * avg_session_duration (hours) = time since backup
# If S4 is not alerting, use the estimated_orphan_count from A4 and corpus size
# to estimate Qdrant contamination rate
qdrant_count = await qdrant.client.count("memories")
estimated_qdrant_orphans = int(qdrant_count.count * sample_miss_rate)
corpus_impact_pct = round(sample_miss_rate * 100, 1)
```
- **Safe**: read-only
- **Purpose**: Discord notification field `corpus_impact_pct`

## Part 3: Actions Requiring Human Confirmation (go/no-go Required)

These actions are irreversible, have high blast-radius, or have a false positive rate above 10%. They must not execute without Tim's explicit approval.

### Action M1: Neo4j DETACH DELETE (CRITICAL — irreversible)

**Why manual**: Permanently deletes Neo4j nodes and all incident CO_RETRIEVED edges. If the epoch boundary is miscalculated (see Part 5), this deletes valid nodes. False positive rate for S5 CRITICAL triggering this at Month 12+ without a genuine PG restore: ~15-30% (Q084).

**Command** (presented in Discord notification):
```cypher
-- Step 1: Count before deletion (run first, report count)
MATCH (n:Memory)
WHERE n.last_access_epoch > $e_boundary
RETURN count(n) AS will_delete

-- Step 2: Execute in batches of 500
MATCH (n:Memory)
WHERE n.last_access_epoch > $e_boundary
WITH n LIMIT 500
DETACH DELETE n
RETURN count(*) AS deleted_batch
-- Repeat until returns 0
```

**Required inputs**: `e_boundary` = the epoch value that separates valid from orphaned nodes. In the PG restore scenario this is `pg_max_after_restore`. **Tim must confirm this value** before execution.

**Go/no-go decision criteria**:
- S4 also alerting with epoch_gap > 50 → strong indication of PG restore → lower threshold for go
- S4 not alerting → may be chronic graph maturity → higher threshold, consider Cypher predicate fix first

### Action M2: Qdrant delete by created_at (CRITICAL — irreversible, directly corrupts retrieval)

**Why manual**: Qdrant is the canonical memory store (Q086 confirmed). Deleting Qdrant points permanently removes memory content — not just routing metadata. If backup_timestamp is wrong, valid memories are destroyed with no recovery path.

**Command** (presented in Discord notification):
```python
# POST to /reconcile?mode=post_restore&backup_timestamp=T&repair=true
# OR manually via Qdrant client:
await qdrant.client.delete(
    collection_name="memories",
    points_selector=Filter(must=[
        FieldCondition(
            key="created_at",
            range=DatetimeRange(gt=backup_timestamp.isoformat())
        )
    ])
)
```

**Required inputs**: `backup_timestamp` = wall-clock time of the PG backup. Tim must supply this from CasaOS snapshot metadata or `stat` on the pg_dump file.

**Order dependency**: Action M2 must execute AFTER Action M1. If Neo4j is cleaned first, the `/reconcile` endpoint (in standard mode) would try to re-create Neo4j nodes for the now-orphaned Qdrant points. Action M2 removes the Qdrant source before any reconcile can run.

**Critical warning**: Do NOT run `/reconcile?repair=true` (standard mode) after a PG restore. Standard reconcile heals toward Qdrant (creates Neo4j nodes for Qdrant orphans) — the exact opposite of correct recovery. This is documented in Q086 as the "reconcile endpoint catastrophic misuse" failure mode.

### Action M3: Clear vector-only retrieval mode
```bash
redis-cli DEL recall:retrieval_mode
```
**Why manual**: Restoring graph traversal before cleanup is complete re-exposes the orphan injection path. Tim should execute this only after both M1 and M2 are verified complete.

## Part 5: Rollback Procedure — If DETACH DELETE or Qdrant Cleanup Was Wrong

### Detecting Over-Deletion

#### Scenario: DETACH DELETE deleted valid nodes (epoch boundary was miscalculated)

Detection signals — appear within 1-2 sessions after cleanup:

1. **S1 degrades suddenly**: IPS P@3 delta falls sharply (valid memories deleted from graph → LLPC candidate set depleted → IPS reranking has fewer candidates to work with)
2. **Session summary references "can't find memory X"**: LLPC retrieval returns unexpectedly sparse results for topics that should have strong graph connections
3. **Direct Neo4j count check**:
   ```cypher
   MATCH (n:Memory) RETURN count(n) AS node_count
   ```
   Compare against pre-cleanup count (should be logged automatically before any DETACH DELETE runs). If count dropped by more than the estimated orphan count from Action A4, over-deletion occurred.
4. **Qdrant-to-Neo4j divergence**: After cleanup, `qdrant.count() >> neo4j.count()` by more than expected. Standard reconcile would normally flag this, but check before running reconcile.

**Timing window for detection**: over-deletion is detectable within the first session after cleanup because LLPC will return sparse or absent results for previously well-covered topics.

#### Scenario: Qdrant delete-by-created_at deleted valid memories

Detection signals:

1. **Memory retrieval returns empty or unexpectedly sparse** for topics Tim was actively working on during the "backup gap" period
2. **Qdrant count drops** by more than estimated_qdrant_orphans from Action A5
3. **Tim notices missing memories**: this is the most direct and unfortunately most reliable detection — Tim observes that Claude doesn't recall things it should

**This is harder to detect programmatically than Neo4j over-deletion** because there is no PG join miss to count. Qdrant is the canonical store; after deletion there is no source of truth to compare against.

### Recovery Options After Over-Deletion

#### Neo4j Over-Deletion Recovery

Neo4j deletion does not destroy data — it only removes nodes and edges from the graph. The underlying memories still exist in Qdrant (if Qdrant cleanup was not yet run) and PostgreSQL (within the backup range). Recovery procedure:

```python
# Option R1: Re-run Qdrant-to-Neo4j reconcile (standard mode)
# This re-creates Neo4j nodes for every UUID in Qdrant that is not in Neo4j
# Use ONLY if Qdrant cleanup has NOT been run yet
POST /reconcile?repair=true

# Result: Neo4j nodes re-created from Qdrant payload
# CO_RETRIEVED edges are NOT recovered — these represent session co-retrieval patterns
# and cannot be reconstructed without replay logs
# The graph topology is lost; nodes exist but have no edges
```

**Limitation**: CO_RETRIEVED edges are not in Qdrant payload. They are reconstructed over time as Tim's sessions generate new co-retrievals. The causal graph returns to a "day 1" state for the affected nodes — connectivity recovers naturally over weeks.

**Cost**: Edge loss is significant but recoverable over time. Node existence (the critical part for avoiding PG join misses) is fully recoverable from Qdrant.

#### Qdrant Over-Deletion Recovery

**There is no programmatic recovery.** Qdrant is the canonical store. Deleted points are permanently gone.

Recovery options (in priority order):

1. **Qdrant snapshot restore** — if a Qdrant snapshot was taken before cleanup:
   ```bash
   # Qdrant snapshot restore via API (Q090 established this is a gap in current backup strategy)
   POST /collections/memories/snapshots/upload
   # Requires: pre-cleanup snapshot exists
   # Best practice: always take Qdrant snapshot BEFORE running cleanup
   ```
   This is the only full-recovery option. It is available only if the Q090 backup gap has been addressed.

2. **Accept partial loss** — the over-deleted memories are gone. Continue forward; they will be partially reconstructed through future sessions if the topics recur.

3. **Manual re-injection** — for critical memories that are known to be missing (e.g., Tim knows he had specific context about a project): manually re-ingest via the Recall store API. Labor-intensive and incomplete, but targeted.

### Pre-Cleanup Safeguards That Must Be Automated

Given the severity of Qdrant over-deletion recovery, these safeguards should execute automatically BEFORE any M1/M2 actions, as part of the go/no-go response:

```python
# Automated pre-cleanup snapshot (runs when Tim responds "go" to Discord notification)
# Before DETACH DELETE:
neo4j_backup_path = f"/tmp/neo4j_backup_{datetime.utcnow().strftime('%Y%m%d_%H%M%S')}.cypher"
# Export current Neo4j state as Cypher statements (for rollback)
await neo4j.run(f"CALL apoc.export.cypher.all('{neo4j_backup_path}', {{}})")

# Before Qdrant delete:
# Always take Qdrant snapshot first (this should be the Q090 standard)
qdrant_snapshot_result = await qdrant.client.create_snapshot("memories")
snapshot_name = qdrant_snapshot_result.result.name
await redis.set(
    "recall:pre_cleanup_qdrant_snapshot",
    snapshot_name,
    ex=86400  # 24-hour TTL
)
```

This converts Qdrant over-deletion from "no recovery" to "recoverable in ~30 seconds."

## Part 6: S4+S5 Simultaneous — PG Restore vs Different Failure Mode

### The Q084 Direction Discriminant (Applied to S5 CRITICAL)

When S4 fires simultaneously with S5 CRITICAL, the direction of the S4 signal determines the root cause:

#### Case 1: S4 fires with epoch_gap > 50 (large positive gap, PG restore direction)

```
S4: epoch_gap = 412 (redis_epoch >> pg_max)
S5: compliance_ratio = 0.71 CRITICAL

→ DIAGNOSIS: PG_RESTORE_CONFIRMED_BY_GRAPH_STATE
  - Redis epoch clock is far ahead of PG max epoch
  - Neo4j graph has edges to nodes with last_access_epoch in the gap range
  - Those nodes fail the PG join → compliance_ratio collapsed
  - This is the Q081 compound failure: PG restored from backup,
    Qdrant/Neo4j not restored, orphans entered working-set window

→ RECOVERY: Full three-store cleanup (M1 + M2) is appropriate
  - Confidence: HIGH (convergent evidence from two independent subsystems)
  - False positive rate: ~15-30% (see Q084)
  - Irreversibility: HIGH — operator confirmation still required
```

#### Case 2: S4 fires with epoch_gap <= 0 (negative gap, Redis reset direction)

```
S4: epoch_gap = -350 (redis_epoch << pg_max)
S5: compliance_ratio = 0.71 CRITICAL

→ DIAGNOSIS: INCONSISTENT — THIS IS NOT THE PG RESTORE PATTERN
  - Redis reset (Direction 1) causes epoch predicate to collapse to near-zero
  - With redis_epoch ≈ 0, the BFS predicate: last_access_epoch >= (0 - 20) = -20
  - ALL nodes pass this predicate
  - In a Redis reset scenario, S5 should NOT fire (Q084 key finding)
  - If S5 IS firing in the Redis reset direction, this is anomalous

  ANOMALY: If we observe S4 Direction 1 (Redis reset) co-alerting with S5 CRITICAL,
  one of the following is happening:
  (a) S5 is in a pre-existing chronic CRITICAL state from graph maturity (Month 12+),
      and the Redis reset is an independent concurrent event
  (b) The epoch predicate was not implemented (Q078 still pending), so the BFS
      returns all nodes regardless of epoch, and many happen to fail the PG join
  (c) There is a cascading failure where Redis reset caused a write path failure that
      also orphaned some Neo4j nodes (rare)

→ RECOVERY: Fix S4 first (automated: redis-cli SET to pg_max+1)
  - S5 chronic state: implement Q078 Cypher predicate (code change, not cleanup)
  - Do NOT run DETACH DELETE in this scenario — the S5 signal is misleading
  - S5 cleanup in Direction 1 context has a ~60-70% false positive rate
```

#### Case 3: S4 fires with small positive epoch_gap (1-50 range, ambiguous)

```
S4: epoch_gap = 35 (small positive gap, boundary of alarm threshold)
S5: compliance_ratio = 0.71 CRITICAL

→ DIAGNOSIS: AMBIGUOUS
  - Small positive gap could be legitimate dual-machine operation variance
    (casaclaude and proxyclaude writing at different rates to PG and Redis)
  - OR small PG restore (< 35 sessions lost)
  - S5 CRITICAL with only a 35-epoch gap is unusual for the PG restore scenario
    (at 35 sessions, orphan count would be small; S5 should show WARNING not CRITICAL)

→ RECOVERY: Investigate before cleanup
  - Run Q081 diagnostic: count Neo4j nodes with last_access_epoch > pg_max
    If count >> 35: indicates either a larger restore or epoch miscalculation
    If count ≈ 35: small PG restore; limited cleanup safe
  - Check dual-machine activity: was proxyclaude active during the gap?
  - CRITICAL threshold: do not run M1/M2 without confirming the epoch boundary
```

#### Summary Table: S4+S5 Co-Alert Root Cause Mapping

| S4 Direction | S4 epoch_gap | S5 CRITICAL | Diagnosis | Cleanup Appropriate? |
|---|---|---|---|---|
| Large positive | > 50 | Yes | PG_RESTORE_CONFIRMED | Yes (with backup_timestamp confirmed) |
| Large positive | > 50 | No | PG_RESTORE_SUSPECTED (not graph-confirmed) | Caution; check Neo4j orphan count first |
| Small positive | 1-50 | Yes | Ambiguous — investigate | Only after boundary confirmed |
| Negative (Redis reset) | < 0 | Yes | S5 is chronic + independent S4 event | Fix S4; defer S5 cleanup |
| None | S4 healthy | Yes | GRAPH_EPOCH_FILTER_GAP | Code fix preferred; limited tombstone cleanup safe |

## Ideas Extracted

### Idea: s5-critical-auto-graceful-degradation
- **Novelty**: 0.78 — Automatically switching the retrieval pipeline from L1.5 (Neo4j BFS) to Qdrant-only mode when S5 CRITICAL fires is a novel operational response that preserves retrieval functionality while eliminating the corruption path. Standard health monitoring systems fire alerts and wait for operator response; this design takes a safe autonomous action (degrading to a known-good retrieval path) immediately, buying time for the human decision without leaving the system in an actively corrupting state. The pattern is analogous to circuit breakers in distributed systems (Hystrix-style fallback to a degraded-but-functional state), applied to a multi-path AI retrieval architecture.
- **Evidence**: 0.72 — The vector-only retrieval path is confirmed functional (Qdrant is the canonical store per Q086; PG join is not in the retrieval path; vector search still works). The fallback is architecturally sound: L1.5 (Neo4j graph traversal) is additive to the core Qdrant retrieval path, not required for basic function. The degradation is safe because S5 CRITICAL specifically indicates the L1.5 path is corrupted; bypassing it eliminates the known corruption source while the human decision is pending.
- **Feasibility**: 0.88 — Implementation: add `recall:retrieval_mode` Redis key check at session start in LLPC (one conditional, ~5 lines). Automatic TTL (2 hours) ensures retrieval is not permanently degraded if Tim forgets to re-enable. The Redis SET command is included in the automated action set for S5 CRITICAL. No changes to the retrieval algorithm itself — just a gating flag.

## Key Finding

The S5 CRITICAL escalation runbook has a fundamental split: the S4 direction discriminant (Q084) is the single variable that determines whether S5 CRITICAL warrants a destructive three-store cleanup or a code change (Cypher predicate). Getting this wrong in the cleanup direction (running DETACH DELETE + Qdrant delete when S5 is showing chronic graph maturity, not a PG restore) permanently destroys valid memories from the canonical store (Qdrant). The runbook must encode this discriminant as the central go/no-go pivot.

**Automated actions** (safe without confirmation): S5 probe re-confirmation with doubled sample size, vector-only retrieval mode activation (circuit breaker), Discord notification, orphan count estimation, pre-cleanup Qdrant snapshot creation.

**Manual confirmation required**: DETACH DELETE (epoch boundary must be operator-confirmed), Qdrant delete-by-created_at (backup_timestamp must be operator-confirmed), re-enabling graph traversal after cleanup.

**Critical ordering constraint**: M1 (Neo4j) → M2 (Qdrant) → M3 (re-enable traversal). Running Qdrant cleanup before Neo4j cleanup creates a window where the standard `/reconcile` endpoint would incorrectly re-populate Neo4j from the already-orphaned Qdrant points. The pre-cleanup Qdrant snapshot converts the highest-risk action from irreversible to recoverable-in-30-seconds.

**Over-deletion rollback**: Neo4j over-deletion is recoverable via standard reconcile (re-creates nodes from Qdrant, loses CO_RETRIEVED edges which reconstruct over subsequent sessions). Qdrant over-deletion has no programmatic recovery without a pre-cleanup snapshot. This asymmetry makes the automated Qdrant snapshot a mandatory pre-condition for M2 execution, not an optional safeguard.

The Discord payload is designed around the under-30-second decision constraint: six fields in decision-flow order, S4 correlation as the central pivot, three clear options with copy-paste commands, direct links to inspection and action endpoints. The 30-minute TTL re-alert ensures Tim sees the notification even on multi-machine concurrent sessions.
