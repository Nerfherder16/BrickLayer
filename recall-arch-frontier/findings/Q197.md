# Finding: Q197 — Write-Path Latency Floor on N100 CPU

**Question**: What is the theoretical minimum write-path latency for a Recall 2.0 write that includes SHA-256 hash check, SimHash band lookup, semantic embedding inference, CRDT merge, Hopfield weight update, and LMDB persistence — running on N100 CPU with embedded fastembed-rs inference?
**Agent**: quantitative-analyst
**Question Type**: [PHYSICS]
**Verdict**: PROMISING
**Severity**: High
**Source field**: q197_benchmark.py — component-by-component latency model on Intel N100 (Alder Lake-N, 4-core, 3.4GHz burst, AVX2, SHA-NI).

## Evidence

### Hardware Parameters: Intel N100

- 4 cores, no HyperThreading, burst 3.4GHz
- AVX2 (256-bit SIMD): 8 FP32 operations per cycle
- SHA-NI extensions: hardware SHA-256 acceleration
- L3 cache: 6MB, LPDDR5 memory bandwidth: ~38.4 GB/s
- TDP: 6W passive cooling

### Component-by-Component Latency Floor

| Component | FP32 (ms) | INT8 (ms) | % of FP32 total |
|---|---|---|---|
| SHA-256 hash check (SHA-NI, 500B) | 0.0006 | 0.0006 | ~0% |
| SimHash compute (AVX2) + LMDB band lookup | 0.055 | 0.055 | ~0% |
| fastembed-rs ONNX embedding (BGE-small) | 80.0 | 40.0 | 99% / 98% |
| CRDT merge (hash compare + tags union) | 0.0001 | 0.0001 | ~0% |
| Hopfield mpsc channel send (lock-free) | 0.00005 | 0.00005 | ~0% |
| LMDB write + NVMe fsync (4.6KB) | 0.055 | 0.022 | ~0% |
| SQLite WAL write (metadata row) | 0.50 | 0.50 | 1% |
| HNSW insert (ef=200, M=16, D=1024) | 0.34 | 0.34 | ~0% |
| **TOTAL** | **80.95** | **40.92** | — |

### Budget Comparison

- Write-path budget (from write-path.md): **186 ms**
- FP32 total: **81 ms** = 44% of budget → **PASS**
- INT8 quantized total: **41 ms** = 22% of budget → **PASS**

### Embedding Dominance

The embedding inference component (fastembed-rs BGE-small-en-v1.5) accounts for:
- FP32: 80ms / 80.95ms = **99% of total write-path latency**
- INT8 quantized: 40ms / 40.92ms = **98% of total write-path latency**

Every other component combined contributes less than 1ms:
- SHA-256 (SHA-NI, 500B): ~0.6 microseconds
- SimHash (64 × 1024-dim AVX2 projections): ~4.8 microseconds compute + 50 microseconds LMDB
- CRDT merge (hash + tags): ~0.1 microseconds
- Hopfield mpsc send: ~50 nanoseconds (lock-free atomic CAS)
- LMDB write + NVMe fsync: ~55 microseconds

### Secondary Bottlenecks (non-embedding)

After embedding, the next largest components:
1. **SQLite WAL write**: 0.5ms — metadata row insert with WAL sync
2. **HNSW insert**: 0.34ms — ef_construction=200 graph update with RwLock acquisition
3. **LMDB fsync**: 0.055ms — NVMe small write (durability cost)

### Ollama Comparison

The fastembed-rs embedded inference provides significant advantage vs Ollama network calls:
- Ollama network embedding: ~150ms (includes HTTP round-trip + queuing)
- fastembed-rs FP32: 80ms → **1.9x speedup**
- fastembed-rs INT8 quantized: 40ms → **3.8x speedup**

### FLOP Analysis for BGE-small-en-v1.5

BGE-small: 12 transformer layers, hidden_dim=384, seq_len=256
- Estimated FLOPs: ~1.81 GFLOPS per inference
- N100 peak throughput: 218 GFLOPS (4 cores × 3.4GHz × 8 FP32/cycle × 2 FMA)
- Effective throughput: ~21.8 GFLOPS (10% efficiency, memory bandwidth bound)
- FLOP-derived estimate: 1.81 / 21.8 = **83ms** → consistent with 80ms practical estimate

The memory-bandwidth constraint (not compute) explains the 10% efficiency. N100's single-channel LPDDR5 (38.4 GB/s) cannot feed the transformer's weight matrix loads fast enough for full AVX2 utilization.

### Why HNSW Is Not the Bottleneck

The HNSW insert model:
- ef_construction=200, M=16, D=1024
- ~3.28M approximate dot product ops → 409K SIMD ops at 2 cycles each
- Compute: 0.24ms + RwLock write acquisition: 0.1ms + amortized checkpoint: 0.0006ms
- **Total: 0.34ms** — fast enough that embedding latency completely masks it

At write rates well under 100 writes/second (typical for a Claude Code session), HNSW contention is negligible.

### Hopfield Hot Path Justification

The Hopfield weight update is designed for non-blocking hot-path operation:
- Write path: mpsc channel push (~50ns, lock-free atomic CAS)
- Actual LMDB write: background worker, flushed every 15 minutes
- Background cost: ~20 edges × 0.1ms/edge = 2ms per flush (not on critical path)

This design means Hopfield adds zero measurable latency to the write path.

## Analytical Summary

**The write path is not the bottleneck.** At 41ms (INT8) to 81ms (FP32), both variants are well within the 186ms write-path budget on N100 hardware. The 3.8x speedup over Ollama network embedding validates the fastembed-rs embedded inference architecture.

**INT8 quantization is the primary optimization lever.** It halves the entire write-path latency because embedding accounts for 99% of total cost. fastembed-rs provides INT8 models by default — this is a zero-effort choice.

**All non-embedding components are negligible (<1ms combined).** SHA-256, SimHash, CRDT merge, Hopfield mpsc send, and LMDB fsync sum to under 1ms. Engineering complexity in these components is not justified by performance gain.

**Secondary engineering concerns** (if write optimization is ever needed):
1. SQLite WAL write (0.5ms) — consider WAL with `synchronous=NORMAL` instead of `FULL`
2. HNSW insert (0.34ms) — already fast; batch inserts would amortize RwLock cost
