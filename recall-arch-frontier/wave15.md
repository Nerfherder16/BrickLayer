### [ADVERSARIAL] Q121: What is the correct memory deduplication strategy when two memories have cosine similarity above 0.97 — should near-duplicate pairs be merged (one winner, loser tombstoned with content union), co-exist with a STALE co_retrievals edge connecting them, or be handled exclusively by the Q013 compaction pass, and what is the information-theoretic argument for which strategy minimizes retrieval precision loss when a future query is semantically equidistant between the two near-duplicates?
**Status**: PENDING

**Rationale**: Q013 BREAKTHROUGH (tiered-compaction-score-trigger) specified batch HNSW cosine clustering fires at the 0.92 threshold and merges clusters — but the merge decision (which memory wins, what happens to the loser content, how co_retrievals edges are updated) was never specified. Q036 content-hash idempotency keys handle exact duplicates (similarity=1.0). Three strategies exist with different information-theoretic properties: (1) MERGE: select the higher-importance memory as survivor, tombstone the other, update survivor content to be a union of both via an LLM summarization call; (2) ONE-WINS: tombstone the lower-importance memory, survivor is unchanged; (3) CO-EXIST: create a co_retrievals STALE edge between the pair, let BFS treat them as mutually reinforcing — no tombstone. The CO-EXIST strategy requires no LLM call but doubles the HNSW slot usage for the near-duplicate region. The MERGE strategy is most space-efficient but requires an LLM call on the compaction path. The finding must specify: (a) the recommended strategy for each similarity range (0.92-0.95 vs. 0.95-0.99 vs. 1.0), (b) whether the Q115 co_retrievals schema supports CO-EXIST without schema changes, (c) the content-merge algorithm if MERGE is recommended (LLM call or rule-based union), and (d) whether a query equidistant between two near-duplicates gets better retrieval precision under MERGE (one memory) vs. CO-EXIST (two memories reinforce each other via STALE edge).
