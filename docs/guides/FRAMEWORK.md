# Autoresearch Framework

A reusable autonomous research system based on the karpathy/autoresearch pattern.
Claude acts as the agent — it iterates on scenario parameters, runs simulations,
evaluates failure conditions, and loops forever until interrupted.

## Source Authority Hierarchy

Every agent reads sources in this order. Higher tiers win on conflict.
Contradictions are never silently resolved — `question-designer` writes `CONFLICTS.md`.

| Tier | Sources | Notes |
|------|---------|-------|
| 1 — Ground truth | Recall canon anchors, `project-brief.md`, `docs/*.md`, `CLAUDE.md`, `README.md` | Human-written. Treated as fact. |
| 2 — Mechanics | `constants.py`, `simulate.py` | Structural. Defines what can vary in simulation. |
| 3 — Interpretation | `findings/*.md`, analysis files | Claude-generated. Verified against Tier 1. |

**`project-brief.md` is optional.** If absent, agents derive understanding from `docs/` +
`CLAUDE.md` + `README.md` and note the absence as a warning. For complex projects
(multiple stakeholders, regulatory constraints, prior misinterpretations), always write one.

## Starting a New Project

1. **Copy the template**:
   ```bash
   cp -r template/ {project_name}/
   cd {project_name}/
   ```

2. **Write `project-brief.md`** (optional, recommended for complex projects):
   - State what the system actually does in your own words
   - List the key invariants that cannot be wrong
   - Document anything Claude has previously misunderstood about this project
   - Delete the file if the project is simple enough that `docs/` + README are sufficient

3. **Add supporting documents to `docs/`**:
   - Architecture specs, legal memos, design briefs, investor docs
   - Anything you consider authoritative that agents should read before generating questions

4. **Adapt `constants.py`** — real thresholds and seed capital (immutable during loop)

5. **Adapt `simulate.py`** — replace stub model with your actual simulation engine

6. **Invoke `question-designer`** — reads all Tier 1 sources, checks for conflicts,
   generates `questions.md`. If `CONFLICTS.md` is created, resolve before continuing.

7. **Verify baseline**: `python simulate.py` — confirm `verdict: HEALTHY`

8. **Initialize git and start the loop**:
   ```bash
   git init && git add . && git commit -m "chore: init {project_name} autoresearch"
   git checkout -b {project_name}/$(date +%b%d | tr '[:upper:]' '[:lower:]')
   claude --dangerously-skip-permissions "Read program.md and questions.md, then begin the research loop. NEVER STOP."
   ```

## How the Loop Works

The loop mirrors karpathy/autoresearch's `train.py` pattern:

```
simulate.py  ←→  train.py        (the agent edits this)
constants.py ←→  (immutable)     (the agent never touches this)
verdict      ←→  val_bpb         (the metric the agent chases)
FAILURE      ←→  improvement     (keep the commit)
HEALTHY      ←→  no improvement  (git reset --hard HEAD~1)
```

The key inversion: autoresearch optimizes *toward* a good metric (lower perplexity).
This framework optimizes *toward* finding failure — FAILURE verdicts are the "wins".

## Agent Architecture

| Agent | Domain | When to use |
|-------|--------|-------------|
| `quantitative-analyst` | D1, D5, D6 | Simulation runs, threshold mapping, parameter sweeps |
| `regulatory-researcher` | D2 | Legal classification, licensing, tax treatment |
| `competitive-analyst` | D3 | Market analogues, fee benchmarking, participation rates |
| `synthesizer` | All | End-of-session integration, best-way-forward narrative |
| `benchmark-engineer` | D1, D4, D5 | Real-system measurement harnesses (latency, throughput, precision) — use when simulate.py calls a live service rather than a pure model |
| `hypothesis-generator` | All | Generates next-wave questions from completed findings — invoke when questions.md is exhausted or a finding implies unexplored follow-on risks |
| `question-designer` | All | Generates the *initial* question bank at project setup — reads constants.py + simulate.py and produces 18-30 project-specific falsifiable questions. Run once before the loop starts. |

The main loop (Claude Code) acts as orchestrator. Agents are invoked per domain
when the question type matches the agent's specialty.

**Lifecycle**:
1. `question-designer` → generates `questions.md` at project init (run once)
2. Main loop → picks PENDING questions, invokes domain agents, logs results
3. `hypothesis-generator` → invoked when questions.md is exhausted, appends Wave N questions
4. `synthesizer` → invoked at session end before running `analyze.py`

## Agent-to-Agent Communication via Recall

Agents share working memory through Recall using `domain="{project}-autoresearch"`.
This is **optional enrichment** — the loop never depends on it. Files are the critical path.

**When to use Recall for inter-agent communication**:
- `regulatory-researcher` discovers a legal framework that `competitive-analyst` shouldn't re-research
- `synthesizer` stores its dependency map so `hypothesis-generator` gets richer context for Wave 2
- `benchmark-engineer` stores baseline measurements so future sessions compare against a persistent baseline

**Pattern** (in agent instructions, not code):
```
# Store working memory
recall_store(
    content="[intermediate finding or shared context]",
    memory_type="semantic",
    domain="{project-name}-autoresearch",
    tags=["autoresearch", "agent:{agent-name}", "type:working-memory"],
    importance=0.6,
    durability="durable",
)

# Retrieve at agent start
recall_search(
    query="[what you need to know before starting]",
    domain="{project-name}-autoresearch",
    limit=5,
)
```

**Scope rules**:
- Always `domain="{project-name}-autoresearch"` — never bleeds into personal memories
- Filter by both `domain` AND `tags` for precise agent-scoped retrieval
- If Recall is unreachable, agents proceed without it — never block the loop on a Recall call

## Recall Integration

Recall is already available as MCP tools in Claude Code (`recall_store`, `recall_search`).
No code changes to Recall are needed — use the `domain` field to scope memories per project.

**Storing a finding** (in program.md instructions, not code):
```
recall_store(
    content="[finding summary]",
    memory_type="semantic",
    domain="{project_name}-autoresearch",   # scoped — never bleeds into personal memories
    tags=["autoresearch", "finding", "severity:{Critical|High|Medium}"],
    importance=0.9,   # findings are high-importance
    durability="durable",
)
```

**Retrieving at session start**:
```
recall_search(
    query="autoresearch findings failures",
    domain="{project_name}-autoresearch",   # filters to this project only
    limit=10,
)
```

**Scope rules**:
- Always use `domain="{project_name}-autoresearch"` — the domain field is how Recall
  prevents cross-project bleed. Personal memories use `domain="user"` or no domain.
- `recall_search` filters by `domain`, so searching `domain="adbp-autoresearch"` never
  returns personal or other-project memories.
- Filter by `domain` + `tags` together for precise retrieval (e.g. `tags=["agent:regulatory-researcher"]`).
- Store findings at session end via program.md instructions; retrieve at session start.

## End-of-Session Analysis

After the research loop completes (or when interrupted):

```bash
cd {project_name}/
python analyze.py
# Opens reports/analysis_{timestamp}.pdf
```

The PDF includes:
- Executive summary (verdict counts, severity distribution)
- Failure boundary map (all WARNING/FAILURE scenarios)
- Per-domain findings
- Best way forward (synthesized recommendations)
- Raw data appendix

## Starting Claude Autonomously

```bash
# From the autosearch/{project}/ directory:
claude --dangerously-skip-permissions "Read program.md and questions.md. Begin the research loop. Pick the first PENDING question, form a hypothesis, modify simulate.py, run it, log the result, write the finding. NEVER STOP until all questions are DONE. When done, generate new questions from findings and continue."
```

The `--dangerously-skip-permissions` flag bypasses permission prompts so Claude
can run git commands, python scripts, and file edits without interruption.

**Note**: Use this only in isolated research directories where autonomous file
modification is safe. Never run it in directories with production code.
