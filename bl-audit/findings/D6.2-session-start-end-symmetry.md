# D6.2 — Session Start/End State Symmetry

**Question ID:** D6.2
**Domain:** D6 Structural Problems
**Priority:** MEDIUM
**Verdict:** FALSE_POSITIVE
**Severity:** Info

## Summary

`masonry-session-start.js` and `masonry-session-end.js` are symmetrical for their shared state. The session snapshot (`masonry-snap-{sessionId}.json` in tmpdir) is written on start and deleted on end. Session-end adds `.autopilot/session-notes.md` and `.ui/session-notes.md` as durable state — these are intentionally persistent, not cleanup-required.

## Evidence

**State written by session-start:**
- `masonry-snap-{sessionId}.json` in `os.tmpdir()` — session-level context snapshot

**State handled by session-end:**
- Line 44-46: `fs.unlinkSync(masonry-snap-{sessionId}.json)` — snap deleted ✓
- `.autopilot/session-notes.md` — written if build is in progress; this is intentional durable state, read by next session's resume logic
- `.ui/session-notes.md` — written if UI compose in progress; same rationale

**Intentional asymmetry (by design):**
- Session-start restores context from session-notes and snap; session-end creates the notes that the NEXT session-start will consume
- This is a relay pattern: start reads, end writes, next start reads — correct for multi-session continuity
- Session-notes are `.autopilot/` and `.ui/` scoped — both of which are in `.gitignore` (confirmed in D6.4)

**No orphaned state accumulation:** The only tmpdir file (snap) is cleaned up. The `.autopilot/` and `.ui/` state files are intentionally persistent.

## Impact

No bug. Session state management is correctly symmetrical for transient state and correctly asymmetrical for durable context-relay state.

## Files

- `masonry/src/hooks/masonry-session-end.js` — deletes snap, writes notes
