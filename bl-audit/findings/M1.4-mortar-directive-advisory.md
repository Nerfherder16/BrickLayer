# M1.4 — Mortar Directive Is Advisory Context, Not Automatic Agent Invocation

**Question**: When the Mortar directive IS successfully injected, does Claude Code invoke mortar.md as an agent subagent or treat it as advisory context text?

**Status**: DONE
**Verdict**: CONFIRMED
**Severity**: High
**Confidence**: 0.95

## Summary

The `[MASONRY] Route this prompt through Mortar (.claude/agents/mortar.md)` directive is advisory context text. Claude Code does not automatically spawn mortar.md as a subagent when it sees this text. The "Every request → Mortar" policy depends entirely on Claude choosing to comply with the directive — it is not technically enforced.

## Evidence

### Mechanism Analysis

`masonry-register.js` writes the directive as `additionalContext` (once M1.3 is fixed). In Claude Code's hook architecture, `additionalContext` is injected as a system-level prefix to the user's prompt. The model receives it as context text, not as a trigger that forces agent invocation.

For Claude to "route through Mortar," it would need to:
1. Read the directive in its context
2. Choose to invoke `Act as the mortar agent in .claude/agents/mortar.md`
3. Let Mortar decide routing, then dispatch to the specialist

This is a **voluntary compliance model**, not an enforcement model. Claude Code provides no mechanism to force agent invocation from a hook.

### mortar.md Invocation Pattern

mortar.md specifies how Mortar invokes other agents:
```
Act as the {agent_name} agent defined in .claude/agents/{agent_name}.md.
```

But there is no mechanism in Claude Code that forces this to happen when the directive is seen. Claude can — and often does — handle tasks directly without invoking Mortar.

### CLAUDE.md Claim vs Reality

CLAUDE.md states: "Every request → Mortar routes it first"

This is a **behavioral expectation**, not a technical constraint. The hook mechanism (masonry-register.js) was designed to reinforce this expectation via repeated context injection, but:
1. M1.1 (SyntaxError) prevented the hook from running
2. M1.3 (wrong output format) means even after the fix, the directive isn't injected
3. Even if both were fixed, Claude may ignore the advisory directive

### Routing Log Correlation

The routing_log.jsonl has 29 `start` events for agents like `karen`, `planner`, `question-designer-bl2`, `trowel`, etc. These were spawned directly (not via Mortar dispatch) because the Mortar directive was never injected (M1.1 + M1.3). The log doesn't contain any `mortar` entries — Mortar has never been invoked as an agent.

## Impact

The "Every request → Mortar" architectural principle exists only in documentation. The technical enforcement mechanism has two unfixed bugs (M1.1 now fixed; M1.3 still unfixed). Even after both are fixed, the architecture remains advisory — no hard guarantee that Mortar runs on every prompt.

## Recommendation

To make Mortar routing technically robust, consider one of:
1. Fix M1.3 and accept the advisory model — rely on Claude's compliance with the directive
2. Add a CLAUDE.md rule that explicitly states "when you see [MASONRY] directives, act on them before responding"
3. Accept that Mortar is a framework convention, not an execution engine
