# Finding: E1.8 — `signatures.py` Has 3 Dead Signature Classes; Only `ResearchAgentSig` Is Used

**Agent**: economizer
**Date**: 2026-03-22
**Severity**: Low
**Verdict**: CONFIRMED

## Summary

`masonry/src/dspy_pipeline/signatures.py` (112 lines) defines 4 DSPy Signature classes. Only `ResearchAgentSig` has any callers — it is lazy-imported by the MCP server's `masonry_optimize_agent` handler and by `run_optimization.py`. The other three classes (`DiagnoseAgentSig`, `SynthesizerSig`, `QuestionDesignerSig`) have zero callers outside their own definitions. `optimizer.py` even has a comment acknowledging `DiagnoseAgentSig` is "not currently populated." The file itself is not dead — it's in a lazy import path invoked only when optimization is manually triggered — but 75% of its class definitions are dead weight. Additionally, all callers of `signatures.py` use lazy imports (inside function bodies), which means the `import dspy` at the top of `signatures.py` only executes when the optimization endpoint fires, not at server startup.
