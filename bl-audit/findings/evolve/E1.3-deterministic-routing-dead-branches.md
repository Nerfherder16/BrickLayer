## Summary

The hypothesis that `deterministic.py` contains dead routing branches for retired agents does not hold. All 34 hardcoded keyword patterns target agents that exist in `agent_registry.yml`. The two helper functions `_read_file` and `_read_json` (lines 228–244) are both called within the file and are not dead. The file is 387 lines (29% over the 300-line limit), but the size is structural: ~170 lines are regex pattern constants forming the routing table, which is a single cohesive logical unit. One lower-severity issue was found: the registry-driven keyword routing (step 1b) only has one agent with `routing_keywords` populated, making that code path largely inactive in practice, though it is not dead.

### Workflow Token Targets (Intentional Design)

Six `target_agent` values are not agent names but workflow tokens: `build-workflow`, `fix-workflow`, `verify-workflow`, `campaign-conductor`, `ui-compose-workflow`, `ui-review-workflow`. These are returned by `_decision()` and consumed by the caller (Mortar) which dispatches them to the appropriate workflow. This is intentional routing indirection, not dead code.

### `_read_file` and `_read_json` Helper Status

Both functions are called within `route_deterministic()`:
- `_read_file` called at line 355 (autopilot mode) and line 372 (UI mode)
- `_read_json` called at line 365 (campaign state)

Neither is dead. They provide 3-error-type try/except wrappers that are cleaner than inline try/except at each call site. Not duplicated in `registry_loader.py`.

### File Size Analysis

The 387 lines break down as:
- Lines 1–23: module header, imports, one constant
- Lines 25–222: 34 regex pattern constants (~5 lines each = ~200 lines) — all active
- Lines 223–252: 3 internal helpers (`_read_file`, `_read_json`, `_decision`) — all called
- Lines 254–387: 2 functions (`_route_from_registry_keywords`, `route_deterministic`) — ~134 lines of logic

The size is driven by the routing table constants, which cannot be meaningfully condensed without sacrificing readability.

## Proposed Fix

No dead code to remove. One improvement opportunity:

**Populate `routing_keywords` in agent frontmatter** — The registry-driven keyword routing (step 1b) was designed to eventually replace many hardcoded patterns. The `economizer` agent already demonstrates the pattern. Populating `routing_keywords` for the top 10 most commonly routed agents would allow gradual removal of hardcoded patterns from `deterministic.py`, incrementally reducing file size without any structural split.

**Effort**: Low per agent
**Impact**: Medium — allows routing table to shrink organically as agents gain frontmatter keywords
