# D3.6 — Windows Path Normalization in Hooks

**Question ID:** D3.6
**Domain:** D3 Bugs & Logic Errors
**Priority:** MEDIUM
**Verdict:** FALSE_POSITIVE
**Severity:** Info

## Summary

`masonry-stop-guard.js` and `masonry-lint-check.js` both handle Windows path normalization correctly. Neither hook has cross-platform path comparison bugs. The hypothesis was unfounded.

## Evidence

**masonry-stop-guard.js** — `normalizeCwd()` function:
```javascript
function normalizeCwd(raw) {
  // Converts /c/Users/... (WSL/Git Bash POSIX) → C:\Users\... (Windows)
  return raw.replace(/^\/([a-z])\//i, (_, d) => d.toUpperCase() + ':\\')
            .replace(/\//g, '\\');
}
```
- Called on both the `cwd` from the hook input and the path stored in the activity log
- Ensures comparison uses consistent Windows-style paths
- Guards against both WSL paths and forward-slash Windows paths

**masonry-lint-check.js** — `normalizeWindowsPath()`:
```javascript
function normalizeWindowsPath(p) {
  return p.replace(/\\/g, '/');  // Converts to forward slashes for Node.js APIs
}
```
- Converts backslashes to forward slashes (the opposite direction — for Node.js path API compatibility)
- Applied consistently to all file paths before comparison

Both functions are correct for their respective use cases. No cross-platform path comparison bug exists.

## Impact

No bug. Windows path normalization is handled correctly in both hooks.

## Files

- `masonry/src/hooks/masonry-stop-guard.js` — `normalizeCwd()` correct
- `masonry/src/hooks/masonry-lint-check.js` — `normalizeWindowsPath()` correct
