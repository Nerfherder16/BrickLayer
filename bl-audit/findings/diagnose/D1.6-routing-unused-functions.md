# D1.6 — Routing Sub-Module Functions All Referenced

**Question ID:** D1.6
**Domain:** D1 Dead Code
**Priority:** MEDIUM
**Verdict:** FALSE_POSITIVE
**Severity:** Info

## Summary

All public functions in `masonry/src/routing/` are called by `router.py`, which is in turn called by `masonry/mcp_server/server.py`. No dead functions were found. The routing module is well-structured with clear caller chains.

## Evidence

**Function call chains confirmed:**

`deterministic.py`:
- `route_deterministic(request_text, project_dir, registry)` → called by `router.py`
- `_read_file()`, `_read_json()`, `_decision()` — private helpers used internally ✓

`semantic.py`:
- `route_semantic(request_text, registry)` → called by `router.py`
- Internal helper functions used within `route_semantic` ✓

`llm_router.py`:
- `route_llm(request_text, registry)` → called by `router.py`
- No other public functions defined ✓

`router.py`:
- `route(request_text, project_dir)` → called by `server.py` (masonry_route tool)
- `_make_fallback()` — private helper used within `route()` ✓
- `_load_registry()` — private helper called by `route()` ✓

`server.py` also calls `masonry_route` → `route()` directly.

## Files

- `masonry/src/routing/deterministic.py` — all functions used
- `masonry/src/routing/semantic.py` — all functions used
- `masonry/src/routing/llm_router.py` — all functions used
- `masonry/src/routing/router.py` — all functions used
- `masonry/mcp_server/server.py` — calls `route()` via masonry_route tool

## Note

While no functions are dead, D3.3 identified a separate bug: `semantic.py` reads `OLLAMA_URL` env var while `settings.json` sets `OLLAMA_HOST`, causing semantic routing to always use the hardcoded fallback URL.
