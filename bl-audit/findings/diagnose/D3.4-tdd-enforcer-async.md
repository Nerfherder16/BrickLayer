# D3.4 — TDD Enforcer Cannot Block (async:true Defeats exit(2))

**Question ID:** D3.4
**Domain:** D3 Bugs & Logic Errors
**Priority:** HIGH
**Verdict:** CONFIRMED
**Severity:** High

## Summary

`masonry-tdd-enforcer.js` is registered in `settings.json` with `async: true`, which means Claude Code NEVER waits for its exit code. The hook calls `process.exit(2)` to block file writes when no test file is found, but this exit code is silently ignored because async hooks cannot block. The TDD enforcement feature is entirely non-functional.

## Evidence

**settings.json registration:**
```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "node .../masonry-tdd-enforcer.js",
            "async": true   // ← cannot block
          }
        ]
      }
    ]
  }
}
```

**masonry-tdd-enforcer.js line ~299:**
```javascript
// In build mode, when no test file found:
process.exit(2);  // Intent: block the Write/Edit
```

**Claude Code invariant (from program.md):**
> Invariant #5: `async: true` hooks fire after the tool result is returned; exit code 2 from an async hook is silently ignored.

The tool call (Write/Edit) completes before the async hook even starts. By the time `process.exit(2)` executes, Claude Code has already accepted the write.

**Consequence:** A developer can write any implementation file without a corresponding test file in build mode, and the TDD enforcer will not block it. It may log a warning to stderr (which appears in the hook output panel) but cannot prevent the write.

## Impact

- TDD enforcement is completely non-functional in build mode
- CLAUDE.md states: `/build` and `/fix` use TDD; orchestrator validates tests pass before DONE — but the hook that enforces the initial test-first discipline doesn't work
- Tests may still be written by convention, but the mechanical enforcement is broken
- If changed to `async: false`, there's a latency concern on Windows (cold start) — but correctness must come before performance

## Files

- `~/.claude/settings.json` — `masonry-tdd-enforcer.js` registered as `async: true`
- `masonry/src/hooks/masonry-tdd-enforcer.js` — calls `process.exit(2)` expecting to block

## Recommendation

Change `masonry-tdd-enforcer.js` registration from `async: true` to `async: false` (or remove the `async` field, which defaults to synchronous). Accept the latency cost for correctness. If blocking on Windows is unacceptably slow, convert the hook to only warn (stdout message) rather than block, and rely on the orchestrator's post-task test validation step instead.
