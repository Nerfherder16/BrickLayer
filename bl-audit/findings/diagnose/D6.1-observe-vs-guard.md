# D6.1 — masonry-observe.js vs masonry-guard.js Duplication

**Question ID:** D6.1
**Domain:** D6 Structural Problems
**Priority:** MEDIUM
**Verdict:** FALSE_POSITIVE
**Severity:** Info

## Summary

`masonry-observe.js` and `masonry-guard.js` serve completely different purposes despite both firing on Write/Edit events. They do not duplicate campaign-detection logic — they handle entirely separate concerns with no copy-paste divergence risk.

## Evidence

**masonry-observe.js purpose (three responsibilities):**
1. (A) Finding detection: when a `findings/*.md` file is written, stores it to Recall + updates masonry-state.json
2. (B) Activity log: appends every Write/Edit to `masonry-activity-{sessionId}.ndjson`
3. (C) Code fact extraction: extracts Python/JS definition names from non-finding edits, stores to Recall

**masonry-guard.js purpose (one responsibility):**
- Error fingerprinting and 3-strike escalation: tracks write errors, builds fingerprints, emits warnings to stderr when the same error pattern recurs 3+ times across different files

These hooks share zero domain logic. Guard doesn't inspect findings or extract code facts. Observe doesn't do error fingerprinting. They both receive the same hook events but process orthogonal concerns.

**Campaign detection:** Neither hook uses `isResearchProject()` — they both apply universally regardless of whether a research project is active. Only `masonry-approver.js` uses `isResearchProject()` for mode switching.

## Impact

No structural problem. The two hooks are correctly separated by concern.

## Files

- `masonry/src/hooks/masonry-observe.js` — findings + activity + code facts
- `masonry/src/hooks/masonry-guard.js` — error fingerprinting + 3-strike escalation
