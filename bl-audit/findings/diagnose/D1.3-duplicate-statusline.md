# D1.3 — Duplicate Statusline Script

**Question ID:** D1.3
**Domain:** D1 Dead Code
**Priority:** MEDIUM
**Verdict:** CONFIRMED
**Severity:** Low

## Summary

There are two copies of the statusline script: `masonry/src/hooks/masonry-statusline.js` (repo copy) and `~/.claude/hud/masonry-statusline.js` (live copy registered in settings.json). The live copy has diverged — it adds an `execSync` import that the repo copy lacks. The repo copy is the stale version.

## Evidence

- `~/.claude/settings.json` `statusLine` field points to: `C:/Users/trg16/.claude/hud/masonry-statusline.js`
- The live `~/.claude/hud/masonry-statusline.js` includes: `const { execSync } = require('child_process');`
- `masonry/src/hooks/masonry-statusline.js` does NOT include this import
- The `execSync` import was added at some point to enable synchronous git/shell calls in the statusline; the repo copy was not updated

## Impact

- Low severity: the live script works correctly; the repo copy is simply outdated
- If someone deploys from the repo copy (e.g. reinstalling on a new machine), they get the older version without `execSync` — any statusline feature using `execSync` would fail silently or throw
- Creates confusion about which file is canonical

## Files

- `~/.claude/hud/masonry-statusline.js` — live version (authoritative, has execSync)
- `masonry/src/hooks/masonry-statusline.js` — repo copy (stale, missing execSync)
- `~/.claude/settings.json` — statusLine field points to the live copy

## Recommendation

Update `masonry/src/hooks/masonry-statusline.js` to match the live copy, or establish a clear policy: the `~/.claude/hud/` location is the deployment target and the repo `masonry/src/hooks/` copy is the source — add a deploy/sync step so they stay in sync.
