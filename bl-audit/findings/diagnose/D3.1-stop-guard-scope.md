# D3.1 — Stop Guard Session Scoping

**Question ID:** D3.1
**Domain:** D3 Bugs & Logic Errors
**Priority:** HIGH
**Verdict:** FALSE_POSITIVE
**Severity:** Info

## Summary

`masonry-stop-guard.js` correctly scopes its uncommitted-change detection to the current session only, using `masonry-activity-{sessionId}.ndjson` as the write log. It does NOT inspect writes from sibling sessions. The concern in the hypothesis was unfounded.

## Evidence

`masonry-stop-guard.js` key logic:
```javascript
function loadSessionWrites(sessionId, cwd) {
  const activityFile = path.join(os.tmpdir(), `masonry-activity-${sessionId}.ndjson`);
  // reads ONLY the file for this session ID
}
```

- The activity file is named with `sessionId` — each Claude Code session writes to its own file
- The hook reads `session_id` from the Stop event input and queries only that session's activity log
- `normalizeCwd()` converts `/c/Users/...` WSL paths to `C:\Users\...` Windows paths to prevent path-comparison mismatches
- Session isolation is correct by design

## Impact

No bug. The stop guard works as intended — it only blocks Stop if the current session has uncommitted writes.

## Files

- `masonry/src/hooks/masonry-stop-guard.js` — correctly scoped, no fix needed
