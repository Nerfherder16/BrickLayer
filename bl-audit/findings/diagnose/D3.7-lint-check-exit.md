# D3.7 — Lint Check Hook Exit Behavior

**Question ID:** D3.7
**Domain:** D3 Bugs & Logic Errors
**Priority:** MEDIUM
**Verdict:** FALSE_POSITIVE
**Severity:** Info

## Summary

`masonry-lint-check.js` always exits with code 0 — it never blocks a write, regardless of lint outcome. This is the correct and intentional design: the hook provides advisory linting feedback via stdout but does not block Claude Code's PostToolUse flow. No bug exists.

## Evidence

Review of `masonry-lint-check.js`:
- All code paths terminate with `process.exit(0)` or fall through to an implicit exit 0
- Lint failures (ruff errors, prettier failures, eslint errors) are reported via stdout/stderr as warnings
- The hook catches all errors in try/catch blocks to prevent unexpected crashes
- PostToolUse hooks cannot block tool calls regardless of exit code (invariant #2)

**Design rationale:** The hook is registered as PostToolUse (not PreToolUse), meaning the file has already been written by the time it runs. Blocking with exit(2) after a PostToolUse event would have no effect. The hook's value is in providing immediate linting feedback that the agent can choose to act on.

**Note:** The hook IS registered as synchronous (no `async: true`), meaning Claude Code waits for it to complete and displays its stdout output. This is correct — the user sees lint warnings inline.

## Impact

No bug. Advisory-only lint checking is the intended behavior.
