# D3.5 — Hook Exit Code Validity

**Question ID:** D3.5
**Domain:** D3 Bugs & Logic Errors
**Priority:** MEDIUM
**Verdict:** FALSE_POSITIVE
**Severity:** Info

## Summary

All hooks in `masonry/src/hooks/` use only exit code 0 (allow/continue) or exit code 2 (block). No hooks use invalid exit codes (1, 3, 127, etc.). The concern in the hypothesis was unfounded.

## Evidence

Grep for `process.exit` across all hook files:
- All hooks exit with 0 (pass through, allow, non-blocking warn) or 2 (block PreToolUse)
- `masonry-stop-guard.js`: exits 2 to block Stop; exits 0 otherwise
- `masonry-approver.js`: exits 1 for approve, 0 for not-approve (correct Claude Code convention for PreToolUse approve hooks)
- `masonry-context-safety.js`: exits 2 to block ExitPlanMode; exits 0 otherwise
- All PostToolUse hooks: exit 0 only (PostToolUse hooks cannot block regardless of exit code)
- `masonry-tdd-enforcer.js`: exits 2 (broken due to async:true — see D3.4, but exit code is valid)

Note: PreToolUse approve hooks use exit code 1 for approval (not 0 or 2), which is the correct Claude Code convention for that specific hook type. This is distinct from blocking hooks.

## Impact

No bug. Exit codes are valid throughout.

## Files

- All `masonry/src/hooks/*.js` — exit codes correct
