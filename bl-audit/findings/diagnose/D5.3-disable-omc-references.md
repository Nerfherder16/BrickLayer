# D5.3 — Active DISABLE_OMC References in Non-Historical Files

**Question ID:** D5.3
**Domain:** D5 Stale References
**Priority:** MEDIUM
**Verdict:** CONFIRMED
**Severity:** Medium

## Summary

`DISABLE_OMC=1` appears in multiple active files. The three highest-priority active references are: `recall/program.md` (instructs users to launch with DISABLE_OMC=1), `masonry/src/hooks/masonry-session-start.js` (emits DISABLE_OMC=1 in help output), and `bl/nl_entry.py` (generates code with DISABLE_OMC=1 in example launch strings). Historical documentation references are acceptable.

## Evidence

**Active references in live code/instructions (must be updated):**

1. `recall/program.md` lines 3-9:
   ```
   > SESSION ISOLATION REQUIRED — Always launch this loop with `DISABLE_OMC=1` to prevent
   > OMC hooks from intercepting agent spawns.
   > DISABLE_OMC=1 claude --dangerously-skip-permissions "..."
   ```
   This instructs users to set a dead env var. Since OMC is uninstalled, DISABLE_OMC=1 does nothing — but the instruction is misleading.

2. `masonry/src/hooks/masonry-session-start.js` line 142:
   ```javascript
   lines.push(`    $env:DISABLE_OMC=1; claude --dangerously-skip-permissions "..."`);
   ```
   The session-start hook outputs this PowerShell command to users as a help message. Users following this guidance set a dead env var.

3. `bl/nl_entry.py` line 1167:
   ```python
   "Run: DISABLE_OMC=1 claude --dangerously-skip-permissions "
   ```
   Live Python code generating a launch instruction with DISABLE_OMC=1.

4. `template/.claude/agents/karen.md` and project copies:
   ```
   how to start a research campaign with `DISABLE_OMC=1 claude --dangerously-skip-permissions`
   ```
   karen.md is an active agent — this guidance gets injected into agent prompts.

**Acceptable historical references:**
- `CHANGELOG.md` — commit messages about wiring DISABLE_OMC (historical, ok)
- `docs/masonry/MASONRY-SPEC.md` — spec explaining why OMC/DISABLE_OMC was replaced (historical context, ok)
- BL1.x/BL2.x project findings and docs archives (historical, ok)

## Impact

- Users following active instructions set a no-op env var — no operational harm, but teaches incorrect mental model
- Agents reading karen.md for guidance on how to start campaigns get outdated launch instructions
- The correct launch command is now: `claude --dangerously-skip-permissions "..."` (no DISABLE_OMC)
- masonry-session-start.js is the most visible: it displays this instruction live in the Claude Code interface

## Files

- `recall/program.md` — active DISABLE_OMC instructions
- `masonry/src/hooks/masonry-session-start.js` — line 142, emits DISABLE_OMC in help
- `bl/nl_entry.py` — line 1167, generates DISABLE_OMC launch string
- `.claude/agents/karen.md` and project copies — documents DISABLE_OMC workflow

## Recommendation

Replace all active `DISABLE_OMC=1` launch instructions with the correct modern command. The isResearchProject() auto-detection in hooks makes DISABLE_OMC unnecessary. Update masonry-session-start.js line 142, recall/program.md, bl/nl_entry.py, and karen.md.
