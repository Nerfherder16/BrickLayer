# D6.5 — BL1.x vs BL2.0 Agent Dispatch Boundary

**Question ID:** D6.5
**Domain:** D6 Structural Problems
**Priority:** MEDIUM
**Verdict:** CONFIRMED
**Severity:** Medium

## Summary

`trowel.md` explicitly uses BL2.0 agents (`hypothesis-generator-bl2`, `synthesizer-bl2`, `question-designer-bl2`) — the BL2.0 boundary is correct at the Trowel level. However, `mortar.md` does not explicitly disambiguate between BL1.x and BL2.0 agent variants, and the deterministic router has no mechanism to distinguish them. Both BL1.x and BL2.0 variant agents have `modes: []` (per D4.3), making accidental cross-routing possible when routing via Mode field.

## Evidence

**trowel.md — explicit BL2.0 agent dispatch (correct):**
```
call hypothesis-generator-bl2  (not hypothesis-generator)
call synthesizer-bl2           (not synthesizer)
invoke question-designer-bl2   (not question-designer)
```
Trowel correctly uses the BL2.0 variants for all campaign operations.

**mortar.md — no BL version disambiguation:**
Mortar's routing table does not distinguish BL1.x vs BL2.0 agents. Mortar dispatches to "Trowel" for campaigns — Trowel then handles the BL2.0 agent selection. This is the correct layered design.

**Overlapping agent names and modes:**
Both `hypothesis-generator` (BL1.x) and `hypothesis-generator-bl2` (BL2.0) have `modes: []` in the registry. Neither is reachable via Mode field routing. If someone adds `modes: [hypothesis]` to both, the deterministic router would return the first match alphabetically — potentially the wrong version.

**Ambiguity scenario:**
- An agent prompt says `**Mode**: hypothesis`
- Deterministic router calls `get_agents_for_mode(registry, "hypothesis")`
- Returns whichever agent is first alphabetically: `hypothesis-generator` (BL1.x)
- A BL2.0 campaign gets routed to the BL1.x agent

This scenario cannot currently happen (both have empty modes), but is a latent risk when D4.3 is fixed.

**The BL2.0 pipeline specifically:** Since Trowel hardcodes BL2.0 agent names in its instructions, the current operational risk is low — Trowel always calls the right agents. The risk is in non-Trowel routing paths.

## Impact

- Current operational risk: LOW (Trowel hardcodes correct agents)
- Latent risk when D4.3 is resolved: BL1.x and BL2.0 variant agents need distinct mode labels
- `synthesizer` vs `synthesizer-bl2`: BL1.x synthesizer is at `.claude/agents/synthesizer.md`, BL2.0 is `synthesizer-bl2.md` — if both get `modes: [synthesis]` they conflict

## Files

- `.claude/agents/trowel.md` — correctly hardcodes BL2.0 agents ✓
- `.claude/agents/mortar.md` — no BL version routing (delegates to Trowel, acceptable)
- `masonry/agent_registry.yml` — both BL1/BL2 variants have `modes: []`

## Recommendation

When fixing D4.3 (adding modes to agents), use distinct mode values for BL1.x vs BL2.0 variants:
- `hypothesis-generator`: `modes: [hypothesis-bl1]`
- `hypothesis-generator-bl2`: `modes: [hypothesis, hypothesis-bl2]`
- `synthesizer`: `modes: [synthesis-bl1]`
- `synthesizer-bl2`: `modes: [synthesis, synthesis-bl2]`
This prevents cross-version routing via Mode field.
