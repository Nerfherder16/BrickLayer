# D3.8 — Router Fallback Implementation

**Question ID:** D3.8
**Domain:** D3 Bugs & Logic Errors
**Priority:** MEDIUM
**Verdict:** FALSE_POSITIVE
**Severity:** Info

## Summary

`masonry/src/routing/router.py` correctly implements the fallback layer. When all three routing layers (deterministic, semantic, LLM) return None, the `_make_fallback()` function returns a valid `RoutingDecision` with `target_agent="user"`, `layer="fallback"`, `confidence=0.0`. The server.py `masonry_route` tool also wraps `route()` in a try/except, ensuring no uncaught exceptions.

## Evidence

`router.py`:
```python
def _make_fallback() -> RoutingDecision:
    return RoutingDecision(
        target_agent="user",
        layer="fallback",
        confidence=0.0,
        reason="No routing rule matched — needs human clarification",
    )

def route(request_text, project_dir):
    # Layer 1
    decision = route_deterministic(request_text, project_dir, registry)
    if decision:
        return decision
    # Layer 2
    decision = route_semantic(request_text, registry)
    if decision:
        return decision
    # Layer 3
    decision = route_llm(request_text, registry)
    if decision:
        return decision
    # Layer 4
    return _make_fallback()
```

`server.py` masonry_route handler:
```python
try:
    result = route(request_text, project_dir)
    return result.model_dump()
except Exception as e:
    return {"error": str(e), "target_agent": "user", "layer": "error"}
```

Both fallback paths correctly return a non-None result with `target_agent="user"`.

## Impact

No bug. The routing system correctly handles all failure cases without raising unhandled exceptions.
