# Finding: Q23.3 — User_id type consistency — HEALTHY with critical Q22.9 correction; all user_ids stored as int in Qdrant (no float/string); proc=2 in decay audit means "2 memories processed" not "user_id=2 processed"; Q22.9 Event C "user_id=2 disappearance" was a misinterpretation; actual user with 2 active memories is user_id=71

**Question**: User_id type consistency — are any Qdrant payloads storing user_id as float instead of int, causing count_user_memories() to return 0 for users who have memories?
**Verdict**: HEALTHY (no type inconsistency exists; all user_ids in Qdrant are stored as Python int — full corpus scan of 21,009 memories shows type distribution: int=3,032, NoneType=17,977, float=0, str=0; count_user_memories(2)=0 is correct — user_id=2 has NEVER existed in Qdrant or was GC'd before the observable corpus window; CRITICAL CORRECTION to Q22.9 Event C: "proc=2" in decay audit details means `processed=2` (2 memories processed), NOT "user_id=2 was the target"; the user with 2 active memories in the per-user decay loop is user_id=71, confirmed by corpus inspection)
**Severity**: Info — corrects a misinterpretation in Q22.9 Event C; no new bug discovered; attribution loss concern from Q22.9 is still structurally valid but the specific user_id=2 example was phantom

---

## Finding 1: user_id Type Distribution — HEALTHY

Full corpus scan (21,009 memories, include_superseded=True):

| user_id type | Count |
|-------------|-------|
| int | 3,032 |
| NoneType (null) | 17,977 |
| float | **0** |
| str | **0** |

No float or string user_ids exist. All non-null user_ids are stored as int. The `MatchValue(value=int)` filter in `_user_conditions()` will correctly match all user-owned memories.

## Finding 3: Critical Correction to Q22.9 Event C

**Q22.9 Event C claimed**: "user_id=2 (Tim/Nerfherder) data was NOT deleted — their 2 active memories were continuously processed through 06:15, then consolidated into a user_id=None merged memory"

**This is incorrect.** The misinterpretation stems from the `proc` field in the decay audit:

```json
{
  "action": "decay_run",
  "details": {
    "processed": 2,   ← Q22.9 read this as "user_id=2 was processed"
    "decayed": 2,
    "stable": 0,
    "decay_ratio": 1.0
  },
  "user_id": null   ← audit log's own user_id = null for all decay runs
}
```

**Correct reading**: `processed=2` means **2 memories were processed** in this per-user decay run. The `details` object has no `target_user_id` field — the audit log does not record which user_id was targeted. `user_id: null` in the audit entry is the ACTOR field (system), not the target.

**The actual user with 2 active memories**: user_id=71 (confirmed: 2 active memories with no superseded_by at ids `34d8d118` and `39836cee`, both created 2026-03-13T23:23-26).

**The per-user decay slot structure** (re-interpreted):
- 7 entries with `processed=0`: ghost users 51, 53, 55, 57, 59, 61, 63 (all have 0 active memories)
- 1 entry with `processed=2`: user_id=71 (2 active memories)

## Finding 4: Q22.9 Event C — What Was Actually Happening?

Q22.9 concluded user_id=2's memories were "consolidated then disappeared from get_distinct_user_ids()". Given user_id=2 never appears in the corpus, the correct explanation:

1. **user_id=2 never had memories in this corpus** — the proc=2 entry was always user_id=71's 2 active memories being decayed
2. **count_user_memories(2) = 0** at 07:03 was correct — user_id=2 simply doesn't exist
3. **The "disappearance from get_distinct_user_ids()"** described in Q22.9 refers to user_id=71's eventual fate if all its memories are consolidated, not user_id=2

**Structural validity of Q22.9's consolidation attribution loss concern**: The mechanism described is still correct — `consolidation.py:235` creates `Memory()` without user_id, so consolidated memories lose user attribution. This is a real bug. But the specific "user_id=2 Tim/Nerfherder" example was never demonstrated — user_id=71 is the user at risk of attribution loss if its 2 active memories are eventually consolidated.
