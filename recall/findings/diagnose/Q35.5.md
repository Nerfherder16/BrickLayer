### Directly Relevant Endpoints

#### POST /admin/importance/rehabilitate
**Purpose**: Boosts importance for memories over-decayed while access was prevented.

**Rules (per OpenAPI description)**:
- `access_count >= 3 OR pinned`: boost to `max(importance, initial_importance * 0.5, 0.3)`
- `durability = "durable" OR "permanent"`: boost to `max(importance, 0.2)`
- Only targets memories with `importance < 0.05` (floor level)
- Idempotent — safe to run multiple times

**Parameters**: None (no request body)

**Live test result**:
```json
{ "processed": 0, "rehabilitated": 0 }
```

**Critical finding**: Rehabilitate processed zero memories. The endpoint targets `importance < 0.05`, but all floor-clamped memories are at exactly `0.050` (the floor value), not below it. The filter condition `importance < 0.05` is exclusive — 0.050 does not satisfy `< 0.05`. This endpoint **cannot reach the floor-clamped population**.

---

#### POST /admin/importance/amnesty
**Purpose**: One-time amnesty for memories unfairly decayed while the v3.2.5 domain-filter bug prevented reinforcement.

**Rules (per OpenAPI description)**:
- Durable memories below 0.4 → boost to `max(current, initial*0.5, 0.3)`
- Memories with graph edges → additional floor of 0.25
- Pinned/permanent memories: skip
- Ephemeral memories: skip

**Parameters**: `dry_run=true/false` (default: `true`)

**Live dry-run result**:
```json
{
  "processed": 5202,
  "boosted": 4173,
  "skipped": 884,
  "dry_run": true,
  "examples": [
    { "old_importance": 0.05, "new_importance": 0.35, "durability": null, "graph_strength": 0.81 },
    { "old_importance": 0.15, "new_importance": 0.30, "durability": "durable", "graph_strength": 2.45 },
    ...
  ]
}
```

**Critical finding**: Amnesty DID find and would boost 0.05-importance memories (examples show `old_importance: 0.05 → new_importance: 0.35`). The condition is `importance < 0.4` (inclusive of floor-clamped), and memories qualify via graph edges regardless of durability. This endpoint **can address the floor-clamped crisis**.

Scope: 5,202 memories processed, 4,173 would be boosted (80% boost rate). This covers the 846 null-user_id floor-clamped casualties and also broader floor-clamped population.

---

#### POST /admin/memory/purge
**Purpose**: Delete low-importance, zero-access memories.

**Parameters**: `max_importance` (default 0.15), `min_age_days` (default 7), `dry_run` (default true)

**Assessment for mitigation**: This is the **wrong tool** for the mortality crisis. Purge deletes floor-clamped memories rather than rescuing them. It could reduce the count metric by deletion, but that is net negative — it destroys memories permanently rather than recovering their value. Do not use for this purpose.

---

#### POST /admin/decay
**Purpose**: Trigger importance decay on-demand, optionally simulating time passage.

**Assessment**: This endpoint accelerates the problem, not fixes it. Q34.4 flagged this as a risk. Do not use.

---

#### POST /admin/memory/purge-domain
**Purpose**: Delete ALL memories with exact domain match.

**Assessment**: Too destructive; deletes entire domain populations. Not applicable to this problem.
