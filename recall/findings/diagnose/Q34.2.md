# Finding: Q34.2 — Integer-0 vs Null user_id: Code Paths and Active Growth

**Question**: Which code paths produce integer-0 vs null user_id memories? Is the integer-0 pool still growing?
**Agent**: quantitative-analyst

**Verdict**: FAILURE

**Severity**: HIGH — integer-0 and null memories are semantically distinct but the distinction is accidental, not intentional. The integer-0 pool is actively growing (58 new today, 2026-03-15). The `_user_conditions(user_id=0)` contract is violated by any code path that passes `user_id=0` to `store()`, because `_user_conditions` was designed under the assumption that `user_id=0` means "IS NULL" (unowned), but the auth layer writes `user_id=0` as an explicit integer into the payload.

---

## Finding 1: Two Distinct Code Paths — Auth Resolves to user_id=0; Workers Omit user_id Entirely

### Path A — Authenticated requests using the master admin key → `user_id=0` (integer)

`src/api/auth.py:40`:
```python
return User(id=0, username="system", display_name="System", is_admin=True)
```

When any request arrives bearing the `RECALL_API_KEY` (the master admin key), `require_auth()` returns `User(id=0)`. Callers then pass `user_id=user.id` to `Memory(...)`:
- `src/api/routes/memory.py:168` → `user_id=user.id if user else None`
- `src/api/routes/memory.py:1234` → same pattern (batch store)

This writes `"user_id": 0` (integer zero) into the Qdrant payload.

**All hook-sourced memories fall here.** The hooks (`recall-session-summary.js`, `observe-edit.js`) send `Authorization: Bearer recall-admin-key-change-me` (the default master key). Every session summary, session checkpoint, observer-extracted fact, and approved signal created via the hooks is stored with `user_id=0`.

### Path B — Background workers and internal pipelines → `user_id=null` (absent from payload)

Workers that construct `Memory()` without passing `user_id=` get the Pydantic default (`user_id: int | None = None`). The `qdrant.store()` method writes `"user_id": memory.user_id` directly — if that is `None`, Qdrant stores `null` (absent field).

Confirmed null-producing call sites:

| File | Line | user_id passed? |
|------|------|-----------------|
| `src/workers/observer.py:151` | fact extraction | NO — omitted |
| `src/workers/observer.py:286` | session snapshot | NO — omitted |
| `src/workers/signals.py:259` | auto-detected signal | NO — omitted |
| `src/workers/patterns.py:241` | pattern extraction | NO — omitted |
| `src/api/routes/ingest.py:221` | approved signal | NO — omitted |

These paths run as internal async workers, not through authenticated HTTP endpoints, so there is no `User` object available to thread through.

### Path C — Consolidation inherits from cluster[0] → may be 0 or null

`src/core/consolidation.py:250`:
```python
user_id=cluster[0].user_id,
```

The merged memory inherits whatever `user_id` the first memory in the cluster carries. If the cluster was built from hook-sourced memories (user_id=0), the consolidated memory also gets user_id=0. If built from worker memories (user_id=null), it gets null.

**However**, the `_get_eligible_memories()` path at `consolidation.py:123` deserializes memories from Qdrant payload **without reading `user_id`**:
```python
memory = Memory(
    id=memory_id,
    content=payload.get("content", ""),
    memory_type=...,
    domain=...,
    importance=...,
    stability=...,
    confidence=...,
    tags=...,
    parent_ids=...,
)  # user_id NOT read from payload → defaults to None
```

This means consolidation silently loses user_id attribution: even if a memory was originally created with `user_id=0` (admin key), after it passes through `_get_eligible_memories()` it appears as `user_id=None` in the cluster. The `cluster[0].user_id` propagation at line 250 therefore consistently writes `user_id=None` for ALL consolidated memories, regardless of origin.

## Finding 2: The `_user_conditions` Contract is Broken

`src/storage/qdrant.py:106–117`:
```python
def _user_conditions(self, user_id: int | None) -> list:
    if user_id is not None and user_id > 0:
        return [FieldCondition(key="user_id", match=MatchValue(value=user_id))]
    if user_id == 0:
        return [IsNullCondition(is_null=PayloadField(key="user_id"))]  # ← queries IS NULL
    return []
```

The docstring says: `user_id=0 → only unowned memories (user_id IS NULL)`.

But the actual storage writes `"user_id": 0` (integer) for admin-key requests. So:

- **Decay worker calling `_user_conditions(user_id=0)`** will scroll for `IS NULL` memories only. It will **miss all 2,996 integer-0 memories entirely**. Those memories receive no decay.
- **Retrieval calling `_user_conditions(user_id=0)`** will similarly miss integer-0 memories when doing a system-level decay or scan.

The conceptual contract ("0 = system/unowned, queries IS NULL") and the storage reality ("admin key writes integer 0") are mismatched. The decay worker's `get_distinct_user_ids()` at `qdrant.py:1187` only returns user IDs where `uid is not None`, so `user_id=0` stored as integer **does get returned** — meaning some code paths will correctly find user_id=0 via MatchValue, while others use `_user_conditions(0)` and search IS NULL, finding nothing.

## Finding 4: No Null → 0 or 0 → Null Transform in store()

`src/storage/qdrant.py:241`:
```python
"user_id": memory.user_id,
```

The store method writes `memory.user_id` verbatim. There is no coercion. `None` → stored as `null` (absent in Qdrant). `0` → stored as integer `0`. The transform is entirely upstream, at Memory construction time.

**Verdict: FAILURE** — Four distinct defects identified:

1. **Auth writes integer-0; `_user_conditions(0)` queries IS NULL** — decay/scan for "system" memories systematically misses 2,996 integer-0 records.
2. **Five worker code paths omit `user_id`** — observer, signals, patterns, ingest, session snapshot all produce null-attributed memories regardless of call context.
3. **Consolidation drops user_id in deserialization** — `_get_eligible_memories()` rebuilds Memory objects without reading `user_id` from payload, so merged memories always get `user_id=None`.
4. **Integer-0 pool growing at ~58/day** — all from session-summary hook using the admin key; decay does not apply to them under the current `_user_conditions` contract.
