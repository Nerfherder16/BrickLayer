## Code Path

**Route definition** (`src/api/routes/admin.py`, lines 124–139):

```python
@router.post("/decay", response_model=DecayResponse)
@limiter.limit("10/minute")
async def trigger_decay(request: Request, body: DecayRequest):
    worker = DecayWorker(qdrant, neo4j)
    stats = await worker.run(hours_offset=body.simulate_hours)
    return DecayResponse(**stats)
```

**Request model** (`DecayRequest`):
```python
class DecayRequest(BaseModel):
    simulate_hours: float = Field(default=0.0, ge=0.0)
```

There is no `user_id` parameter in `DecayRequest`. The call to `worker.run()` passes only `hours_offset=body.simulate_hours`, leaving `user_id` at its default of `None`.

**Worker behavior** (`src/workers/decay.py`, lines 43–48, 73–77):

```python
async def run(self, hours_offset=0.0, feedback_stats=None, user_id=None):
    # ...
    results = await self.qdrant.scroll_all(
        include_superseded=False,
        user_id=user_id,   # None → no filter → ALL memories
    )
```

When `user_id=None`, `scroll_all` applies no user filter. The worker iterates and potentially modifies the importance value of every memory in the corpus across all users.

**Contrast with cron path** (`run_decay_all_users`, lines 278–301): The cron job iterates per user — calling `worker.run(user_id=uid)` for each distinct user, then `worker.run(user_id=0)` for unowned memories. The HTTP endpoint skips this scoping entirely.

**Audit actor assignment:** The actor field `"decay"` is hardcoded inside `_log_decay_audit()` regardless of who triggered the run. An admin-triggered call would write `actor="decay"` to the audit log, making it indistinguishable from a scheduled run.

## Authentication

The `/admin` router is registered in `src/api/main.py` with `dependencies=[Depends(require_auth)]`. Any valid API key satisfies this check — no admin-role enforcement is applied at the router level. Any authenticated user who knows the endpoint exists can trigger full-corpus decay.

A rate limit of `10/minute` is applied, which constrains throughput but does not prevent a sustained attack (600 calls/hour).

---

## Blast Radius

A single call to `POST /admin/decay` with `simulate_hours=8760` (one year) would:

1. Scroll every non-superseded memory in the corpus — currently ~6,000 records across all users.
2. Apply the decay formula with 8,760 hours of elapsed time to each memory.
3. Batch-update `importance` in both Qdrant and Neo4j for all memories that show a change > 0.001.
4. Log the summary entry with `actor="decay"`, leaving no distinguishing mark in the audit log.

Memories with `durability="permanent"` or `pinned=true` are immune. All others — including `durability="durable"` (which merely slows decay to 0.15x rather than preventing it) — are affected. At `hours_offset=8760` and `base_decay_rate * 0.15 * access_mod * feedback_mod`, durable memories would reach or approach their floor value (`decay_default_floor`, typically ~0.05).

Effective result: a single API call could reduce the vast majority of the corpus to floor-level importance, making most memories effectively invisible to retrieval (importance-weighted search would rank them below threshold).

## Why HEALTHY Was Not Assigned

The endpoint was never called by a non-cron actor (no confirmed incident), which prevents a FAILURE verdict. However, the structural risk is fully present:

- No `user_id` scoping in the HTTP path
- No admin-role check beyond a valid API key
- Audit trail is indistinguishable from legitimate cron runs
- `simulate_hours` parameter can amplify the decay by arbitrary time

---

## Verdict

**WARNING** — Endpoint has never been called by a non-cron actor in the observable window, but remains structurally capable of full-corpus, all-user importance destruction via a single authenticated HTTP call. The audit log provides no forensic differentiation between a cron run and a manual trigger.

---

## Recommended Fix

1. Add `user_id: int | None = None` to `DecayRequest`. Pass it through to `worker.run()`.
2. Require admin role on the decay endpoint specifically (not just `require_auth`).
3. Stamp the audit log with the caller identity: pass an `actor` parameter to `_log_decay_audit` so admin-triggered runs are distinguishable from cron runs.
4. Consider capping `simulate_hours` to a reasonable maximum (e.g., 168 = one week) to prevent time-amplified attacks.
