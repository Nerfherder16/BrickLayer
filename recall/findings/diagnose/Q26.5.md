## Finding 3: Re-Accumulation Timeline

After IS NULL-only deployment (Q21.5 fix):

**Immediate effect**: `get_distinct_user_ids()` scans only active memories with `superseded_by IS NULL`. The current 7 ghost users/slot would disappear immediately because those ghost user_ids come from user_id=None entries in superseded memories — and the IS NULL filter would exclude superseded memories from the scan.

**Re-accumulation mechanism**:
1. Hourly consolidation creates new merged memories with user_id=None (~796/day from consolidation events)
2. When an existing merged memory (user_id=None, source=consolidation) gets re-consolidated, it becomes superseded
3. Its user_id=None enters the superseded pool
4. `get_distinct_user_ids()` (even with IS NULL filter) scans ACTIVE memories; the re-accumulation path for ghost users specifically requires user_id=None entries in ACTIVE memories that have no actual active memories for that user_id

**Critical distinction**: Ghost users in the current get_distinct_user_ids() result come from the superseded pool (memories with superseded_by IS NOT NULL but user_id not null — wait, that's not right). Let me clarify:

Ghost users occur when:
- A user_id appears in ANY Qdrant point (active or superseded) in the current scan (no IS NULL filter)
- That user_id has ZERO active memories (all their memories are superseded)
- Decay runs for that user_id, finds 0 active memories → proc=0

With IS NULL filter deployed:
- Only user_ids from active memories are returned
- A user_id=None entry in active memories would also appear (as a "user" with id=None)
- But proc=0 entries would occur for any returned user_id that has no active memories — which would be impossible by definition if the IS NULL filter correctly restricts to active memories

**Re-accumulation estimate**: With 1,833 active merged memories with user_id=None, and IF the IS NULL filter treats user_id=None as a valid user entry (distinct from not having a user_id), then get_distinct_user_ids() would still return None as a "user_id" and run a "user" decay pass for user_id=None — which would be the system-run, the double-decay root cause.

The ghost user re-accumulation for proc=0 entries specifically requires consolidation-source memories to become orphaned (no active successors). This is currently driven by the ~796/day consolidation events creating a superseded pool.
