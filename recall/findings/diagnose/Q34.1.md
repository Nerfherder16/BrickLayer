## Analysis

### 1. Control Group Validity Failure

The null pool mean (0.2584) is **higher** than the int-0 control mean (0.1852) by +0.0732. This is the opposite of what double-decay suppression would predict. If the null pool had been suppressed by extra decay cycles, its mean should be significantly *below* the never-decayed control.

This inversion invalidates the int-0 pool as a control group for measuring importance suppression. The two populations differ in origin:

- **Int-0 memories**: Predominantly low-importance system-generated memories (Q34.5: 92% never accessed, mean 0.185 at creation). Most were written with inherently low initial importance.
- **Null user_id memories**: Mixed population including actively-created session memories with a wider initial importance range (mean 0.258, 5.7% above 0.700).

The double-decay system applied systematic per-cycle decay to the null pool, but the null pool started with higher importance values. Decay suppressed them, but not enough to drop below a lower-baseline never-decayed cohort. **Without `initial_importance` stored on each memory, the true magnitude of suppression cannot be calculated.**

### 2. Floor Casualty Count

The floor casualty analysis does not require a valid control group — it only asks: how many null pool memories are currently clamped at the decay floor (0.05)?

| Metric | Value |
|--------|------:|
| Null pool at strict floor (≤0.05) | **795** |
| Null pool near-floor (≤0.051, including float drift) | **846** |
| Int-0 pool floor rate | 19.8% |
| Null pool floor rate | 15.7% |

**795 memories in the double-decayed cohort are at the hard floor.** These are "premature casualties" in the sense that:
- They have been decayed at 2× rate per 6-hour cycle
- Under single-decay, some fraction would still be above 0.05
- Exactly how many is not computable without initial importance values

The raw count of 795 floor-clamped null memories exceeds the FAILURE threshold of 500.

**However**, the null pool floor rate (15.7%) is actually *lower* than the int-0 rate (19.8%). This means the double-decayed null pool has proportionally fewer floor-clamped memories than the never-decayed control — which further undermines the comparability assumption. The null pool's higher baseline importance is protecting it from floor-clamping even under double-decay.

### 3. Importance-Units Stolen Estimate

The question asks for: `(expected_mean_single_decay - actual_mean) × count_null_pool`

This requires `expected_mean_single_decay` — which requires either:
- The initial importance of each memory (not stored), OR
- A valid control group representing "importance under single decay" (int-0 pool is invalid per above)

**This calculation is INCONCLUSIVE.** No valid estimate of expected single-decay mean is available.

### 4. What the Data Does Show

The double-decay bug has definitively operated on 5,048 null user_id memories. The structural evidence of damage is:

- **795 memories are floor-clamped** — they cannot decay further and will remain at minimum importance until accessed or expired
- The null pool has a long left tail: P10 = 0.05 (at floor), meaning the bottom 10% are all clamped
- 1,629 memories (32.3%) are in the 0.10–0.20 range — a cluster that may have been pushed down from 0.20–0.40 range under double decay
- The median (0.1734) is lower than the mean (0.2584), confirming left-skew toward the floor

## Implications

The 795 floor-clamped null memories represent a **retrieval dead zone**: these entries will never surface in importance-weighted queries regardless of their content value. Under the double-decay bug, memories that should have taken longer to reach floor were pushed there prematurely. The fix (correct decay routing so null user_id memories decay once per cycle) will prevent further accumulation but will not restore already-clamped memories.

Remediation options:
1. **Accept the damage**: Floor-clamped memories will naturally expire when `invalid_at` is reached
2. **Bulk importance reset**: Set floor-clamped null memories back to a baseline (e.g., 0.15) pending proper decay from that point
3. **Priority repair**: Reset only floor-clamped null memories from high-value domains (infrastructure, tooling, recall)
