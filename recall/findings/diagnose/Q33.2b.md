# Finding: Q33.2b — Qdrant user_id payload audit: why IS NULL returns 5,994

**Question**: Do all 5,994 active memories have user_id=1, and why does IsNullCondition also return 5,994?
**Agent**: quantitative-analyst
**Verdict**: WARNING
**Severity**: HIGH — Root cause of double-decay fully identified; three-way population split confirmed; exact fix specified

---

## Finding 1: Memory store() — is user_id included in Qdrant payload?

`store()` in `qdrant.py` lines 220–250 explicitly includes `"user_id": memory.user_id` in the payload dict. The field is always present in newly stored memories. However, the value stored depends on the `Memory` object's `user_id` attribute — which can be `None`, `0`, or a positive integer.

Key observation: the payload also stores `"username": memory.username`. System-ingest paths set `user_id=0, username="system"`. Legacy consolidation paths that predate the user_id field produced records with `user_id=null` in Qdrant (stored as JSON null, interpreted as "missing field" by IsNullCondition).

## Finding 2: scroll_all() filter application

`scroll_all()` at lines 661–708 calls `_user_conditions(user_id)` and appends the result to `conditions`, then builds `Filter(must=conditions)`.

```python
# lines 674–682
conditions = self._user_conditions(user_id)
if not include_superseded:
    conditions.append(
        IsNullCondition(is_null=PayloadField(key="superseded_by"))
    )
scroll_filter = Filter(must=conditions) if conditions else None
```

`_user_conditions(user_id=0)` returns `[IsNullCondition(is_null=PayloadField(key="user_id"))]`.

At the Qdrant engine level, `is_null` on a field matches points where the field is **absent** OR where the field value is **JSON null**. Both `user_id: null` (Python None stored as JSON null) and memories where the field was never written match this condition.

## Finding 3: Qdrant client version + IsNullCondition API

- **Installed version**: `qdrant-client==1.17.0` (confirmed via `docker exec recall-worker pip show qdrant-client`)
- **Requirement spec**: `qdrant-client>=1.7.0` in `pyproject.toml`
- **IsNullCondition syntax**: `IsNullCondition(is_null=PayloadField(key="user_id"))` — this is the **correct syntax** for qdrant-client 1.7+. The `PayloadField` wrapper is required; bare `IsNullCondition(key="user_id")` would fail.
- **Conclusion**: The Python client IsNullCondition is constructed correctly. The issue is not a client API mismatch.

## Finding 6: Root cause determination

### Why does `run_decay_all_users` double-decay?

```python
# decay.py lines 278–301
async def run_decay_all_users(qdrant_store, neo4j_store):
    worker = DecayWorker(qdrant_store, neo4j_store)
    user_ids = await qdrant_store.get_distinct_user_ids()  # returns [0]

    for uid in user_ids:                        # uid = 0
        stats = await worker.run(user_id=uid)   # Pass 1: scroll_all(user_id=0) → 5,043 null memories

    # "system/unowned memories" explicit pass
    stats = await worker.run(user_id=0)         # Pass 2: scroll_all(user_id=0) → same 5,043 null memories
```

`get_distinct_user_ids()` (lines 1187–1212) scrolls active memories, reads `payload.get("user_id")`, and adds to set if `uid is not None`. Since `user_id=0` is `not None`, it is included in the returned list → `get_distinct_user_ids()` returns `[0]`.

So the per-user loop runs `worker.run(user_id=0)` for uid=0, then the explicit system pass at line 296 runs `worker.run(user_id=0)` again. **Every null-user_id memory is decayed twice per cycle.**

### Why does `scroll_all(user_id=0)` return ~5,043 active memories (not 975)?

`_user_conditions(user_id=0)` → `IsNullCondition(is_null=PayloadField(key="user_id"))`. This condition matches records where `user_id` is JSON null or absent — it does **not** match records where `user_id=0` (integer zero). So:

- The 975 active memories with `user_id=0` (integer) are **NOT** caught by either pass in `run_decay_all_users`.
- The 5,043 active memories with `user_id=null` are caught **twice** (both the uid=0 per-user pass and the explicit system pass).

### Three-population breakdown of active memories (6,083 total):

| Population | Count | Decay behavior |
|------------|-------|---------------|
| `user_id = null` (legacy) | 5,043 | Decayed **twice** — double-decay confirmed |
| `user_id = 0` (integer, system) | 975 | **Never decayed** — missed by both passes |
| `user_id = 1+` | ~65 | Not present (user_id=1 has 0 records) |

### The semantic mismatch:

The code treats `user_id=0` as a sentinel meaning "system/unowned memories stored with user_id=null." But the actual data has **two distinct populations**: memories stored with explicit integer `0` and memories stored with JSON `null`. The `IsNullCondition` only catches the null population, not the integer-0 population. Meanwhile `get_distinct_user_ids()` includes integer-0 in the per-user loop, so the null population gets decayed in both the per-user loop (as uid=0 → IS NULL) AND the explicit system pass (also IS NULL).

The double-decay affects the 5,043 active null-user_id memories (83% of the active corpus). The 975 integer-0 memories are never decayed (under-decay). No fix has been deployed yet.

## Actionable fix

Two changes required in `qdrant.py` and `decay.py`:

**Fix 1 — `get_distinct_user_ids()` must exclude user_id=0 from the per-user loop** (since 0 is a system sentinel, not a real user):

```python
# qdrant.py ~line 1207
if uid is not None and uid > 0:   # was: if uid is not None
    user_ids.add(int(uid))
```

**Fix 2 — `run_decay_all_users()` system pass must also cover integer-0 memories** (currently missed):

```python
# decay.py ~line 293
# Run for system/unowned memories: both user_id IS NULL and user_id=0
try:
    stats_null = await worker.run(user_id=0)    # catches null memories
    stats_zero = await worker.run(user_id=-1)   # new sentinel: catches user_id=0
    # OR: add a new _user_conditions branch for user_id=-1 → FieldCondition(match=0)
```

Simplest alternative — unify all system memories under a single sentinel at ingest time (store `user_id=null` always, never store integer 0) and fix ingest paths that write `user_id=0`. This eliminates the split population entirely.

**Immediate mitigation without code changes**: Remove the explicit `worker.run(user_id=0)` call at line 296. The per-user loop already runs it for the null population (via the uid=0 → IS NULL mapping). This stops the double-decay. The integer-0 under-decay is then a separate lower-priority fix.
