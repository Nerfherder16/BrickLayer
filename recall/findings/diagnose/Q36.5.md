## Findings

### 1. Current code vs required fix

**Current code** (`qdrant.py` line 1207):
```python
if uid is not None:
    user_ids.add(int(uid))
```

This allows `uid=0` into the set. If any memory has `user_id=0` stored explicitly (not NULL), the per-user loop will call `worker.run(user_id=0)` for it, and then the explicit system block also calls `worker.run(user_id=0)` — causing double-decay of system memories.

**Required fix**: `if uid is not None and uid > 0:`

This ensures uid=0 is excluded from the per-user set, so system memories are decayed exactly once via the explicit block.

### 2. Callers of get_distinct_user_ids

Four callers found via grep:

| File | Line | Context |
|------|------|---------|
| `src/workers/decay.py` | 281 | Primary — the double-decay site |
| `src/workers/patterns.py` | 351 | Pattern worker per-user loop |
| `src/workers/dream_consolidation.py` | 332 | Dream consolidation per-user loop |
| `src/core/consolidation.py` | 395 | Consolidation per-user loop |

All four callers use the same pattern: iterate over returned user_ids, process per user, then separately handle the system case. The fix applies only to `qdrant.py` `get_distinct_user_ids`. The change is a single function — all callers benefit identically.

**Side effect assessment for other callers**: patterns.py, dream_consolidation.py, and consolidation.py would also stop processing uid=0 in their per-user loops after the fix. If any memory currently has `user_id=0` stored (not NULL), those workers would stop processing them in the per-user pass. However, none of those workers have an explicit system block equivalent to decay.py's `worker.run(user_id=0)` — they skip system memories silently. This is an edge case but represents a pre-existing behavior gap unrelated to Fix-1's stated scope.

### 3. Post-fix behavior with no uid>0 users

**Current state**: `get_distinct_user_ids` returns the set of all non-NULL user_ids in active memories.

After Fix-1: the function returns only uid>0 values. Given the query below shows 74 memories with user_id>0, the function would return those user IDs (IDs 53, 57, 59, 61, 63, 71, etc. from sampling) rather than returning an empty list.

**Correction to the original assumption**: Fix-1 does NOT cause an empty list. The system has 74 memories with `user_id > 0` (real user IDs in range 53–71+). These users will still be processed in the per-user loop. Only uid=0 is excluded.

The explicit system block at decay.py line 296 (`worker.run(user_id=0)`) handles NULL memories (system user). This runs regardless of the per-user loop result.

- `user_id > 0`: 74 memories (FamilyHub user memories)

No memories with `user_id = 0` (explicit integer zero) are expected, but this was not directly counted. The NULL vs 0 distinction is the core of the bug: system memories use NULL, not 0.

### 5. ARQ worker hot-reload behavior

The ARQ worker runs as: `arq src.workers.main.WorkerSettings`

The `docker-compose.yml` mounts `./src:/app/src` (source code is volume-mounted), but ARQ does **not** support hot-reload. Changes to source files do not take effect until the container is restarted.

**Deployment requirement**: `docker compose restart recall-worker` after deploying the fix.

The API container (uvicorn) also runs in development mode but with `--reload` implied by the `RECALL_ENV=development` configuration — however, `get_distinct_user_ids` is only called by background workers, not the API, so only the worker restart matters.

## Verdict

**WARNING: safe but edge cases with future uid>0 users and other callers**

The fix is a low-risk single-line change. In the current system, the dominant concern (double-decay of system memories) is resolved cleanly: `user_id=0` is excluded from the per-user set, and the explicit system block handles NULL memories once.

However, two edge cases warrant documentation:

1. **Other callers** (patterns.py, dream_consolidation.py, consolidation.py) do not have explicit system-user fallback blocks. After Fix-1, any memory with `user_id=0` stored explicitly would be silently skipped by those workers. This is a pre-existing architectural gap, not introduced by Fix-1.

2. **FamilyHub users (uid > 0) now processed**: After Fix-1, the 74 FamilyHub memories (uid 53–71) will be included in the per-user decay loop where previously uid=0 may have been inadvertently included. This is the correct behavior.

**Deployment requires container restart** (`docker compose restart recall-worker`). No hot-reload available.
