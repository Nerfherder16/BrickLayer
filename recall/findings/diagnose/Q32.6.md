## Finding 1: LLM timeout implementation in llm.py

Source: `C:/Users/trg16/Dev/Recall/src/core/llm.py`

Key structural facts:

**Semaphore**: `_llm_semaphore = asyncio.Semaphore(1)` — global, enforces one concurrent Ollama call. The semaphore is acquired with `async with _llm_semaphore:` at line 93, wrapping the full `httpx.post` call through the `finally` block. The semaphore is held for exactly the duration of the HTTP request, bounded by the `timeout=` argument passed to `client.post()`.

**Timeout path** (lines 96–100):
```python
response = await self.client.post(
    f"{self.settings.ollama_host}/api/generate",
    json=body,
    timeout=timeout,  # ← explicit per-callsite value from Q31.7 fix
)
```

**Exception handling** (lines 117–125): `httpx.TimeoutException` is caught, logs `llm_timeout` via structlog, increments `recall_llm_requests_total{status="timeout"}` metric, and raises `LLMError`. The semaphore is released by the `async with` context manager immediately upon exception — no semaphore leak is possible.

**Effective max hold time**: The highest tier timeout is 90s (consolidation, patterns, cognitive_distiller, dream_consolidation, profile_drift). With Q31.7's 17/17 compliance, no callsite can hold the semaphore beyond its assigned tier ceiling. The Q15.3 worst-case of 180s (unbounded `httpx.AsyncClient` default timeout) is structurally eliminated.

**Latency metric**: `recall_llm_latency_seconds` histogram observed in the `finally` block captures actual duration regardless of success/timeout/error path. This provides the authoritative p95 latency signal.

There is no structural mechanism that could produce a >90s semaphore hold given the current codebase state confirmed by Q31.7.
