### Event B: 521 Importance Mismatches — Q21.1 Bug Confirmed in Production

**Mechanism** (from static code analysis):
1. Consolidation runs: `consolidate(user_id=None)` pairs similar memories into clusters
2. Source memories: `neo4j.mark_superseded(source_id, merged_id)` sets `importance=0.0` (Q21.1 bug)
3. Source memories: `qdrant.mark_superseded(source_id, merged_id)` sets `superseded_by` + `invalid_at` — **does NOT update importance**
4. Reconcile scan (`reconcile_stores(dry_run=True)`) compares Neo4j importance vs Qdrant importance
5. `abs(0.0 - original_importance) > 0.001` → ALWAYS TRUE → adds to `importance_mismatches` list
6. **Result**: Every supersedure event creates exactly 1 permanent importance mismatch in reconcile

**Mismatch count trajectory** (confirmed via audit):
- Total consolidate audit entries (1000 visible, from 2026-03-02 to 2026-03-15): ~1,000 consolidation events
- Each consolidation supersedes 2–5 source memories → ~2,000–5,000 supersedure events total
- Current mismatch count = 521 = count of memories currently in `superseded` state that have `importance=0.0` in Neo4j
- Reconcile dry-run: `importance_mismatches: 521, repairs_applied: 0` (dry_run=True)

**Not caused by the anomaly window specifically**: The 521 mismatches accumulated over the ENTIRE history of the system. The anomaly window (18:15–00:16) contributed only ~21 additional supersedure events (from audit log).

### Event C: User_id=2 Disappearance from Decay Loop

**Timeline of user_id=2 active memories**:
| Time | Observation | Evidence |
|------|-------------|----------|
| 2026-03-14T18:15 | 2 active memories for user_id=2 | `decay_run: proc=2` in decay audit |
| 2026-03-15T00:16 | 2 active memories still present | `decay_run: proc=2` in per-user slot |
| 2026-03-15T06:15 | 2 active memories still present | `decay_run: proc=2` in per-user slot |
| 2026-03-15T06:45 | per-user entries ABSENT | System runs return (proc=5936), per-user loop not logged |
| 2026-03-15T07:03 | `count_user_memories(2) = 0` | Export API confirms 0 memories |

**Root cause of disappearance**:

Between 06:15 and 06:45, the consolidation worker (running continuously) superseded user_id=2's 2 active memories:
1. user_id=2's memory A + memory B → `consolidate()` creates merged memory C with `user_id=None`
2. `qdrant.mark_superseded(A, C)` and `qdrant.mark_superseded(B, C)` — A and B now have `superseded_by` set in Qdrant
3. `neo4j.mark_superseded(A, C)` and `neo4j.mark_superseded(B, C)` — Neo4j nodes A and B get `importance=0.0`
4. `scroll_all(user_id=2, include_superseded=False)` → now returns 0 (A and B are superseded)
5. `get_distinct_user_ids()` → eventually returns empty for user_id=2 (when ALL memories with user_id=2 are superseded)
6. `count_user_memories(2)` = 0 — confirms A and B are the only Qdrant points with user_id=2, and both are superseded

**No data deletion occurred**: `count_user_memories` has NO superseded filter — it counts ALL points with `user_id=2`. Result = 0 means neither active nor superseded memories with `user_id=2` exist. But 2 superseded memories (A, B) should still have `user_id=2`...

**Critical unresolved detail**: `count_user_memories(2)=0` while `proc=2` decay entries persist in the audit suggests one of:
a) The two memories have `user_id=2.0` (float) in Qdrant payload, not int — `get_distinct_user_ids()` converts via `int(uid)` so 2 appears in the list, but `MatchValue(value=2)` (int exact match) returns 0 results
b) The memories were genuinely deleted (no audit entry exists for this)
c) Consolidation ran after 07:03 API check, superseding both memories (proc=2 at 06:15 was the last run before this)

The most parsimonious explanation: the consolidation ran between 06:15 and 06:45 (continuously running at 06:00–06:01), user_id=2's memories were consolidated, and `count_user_memories` returns 0 because both are superseded — BUT the `user_id` type theory (float vs int) would also explain persistent 0 across all historical API calls.

**Content is preserved**: The merged memory C (user_id=None) contains the synthesized content of A and B. No information loss, only attribution loss (C has `user_id=None` instead of `user_id=2` — confirmed by consolidation.py:235 which creates `Memory(...)` without user_id field).

## Secondary Findings

### get_distinct_user_ids() User Attribution Blindspot
When ALL of a user's memories are superseded (by system consolidation), `get_distinct_user_ids()` stops returning that user_id. The consolidation creates merged memories with `user_id=None`. This means:
- A user's data can be "ghosted" from the decay loop not by deletion but by consolidation
- The only surviving copy of their data (the merged memory) has `user_id=None`
- This is an attribution loss bug: user ownership of consolidated knowledge is not preserved

### Consolidation user_id Propagation Gap (confirmed from Q22.9 investigation)
`consolidation.py:235` creates `merged = Memory(...)` without user_id. This is the confirmed root cause of attribution loss. When system consolidation (`user_id=None` run) merges two memories from user_id=2, the merged memory gets no user attribution.

### ARQ Cron Schedule Change
Between 18:15 and 00:16, the slot structure changed:
- Pre-outage (18:15): 10 entries [system, 7×ghost, user_id=2, system] — two system runs
- Post-outage (00:16, 06:15): 8 entries [7×ghost, user_id=2] — zero system runs
- After ~06:30: 2 entries [system, system] — system runs return, per-user runs stop
This suggests the ARQ scheduler configuration may have changed during the restart, or the double-decay bug behavior changed (possibly the `run_decay_all_users()` entrypoint was restarted separately from the cron).
