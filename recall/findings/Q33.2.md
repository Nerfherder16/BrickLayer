# Finding: Q33.2 — ARQ double-enqueue: stale Redis job ruled out; source is worker restart catch-up

**Question**: Is a stale Redis job the source of the second run_decay enqueue per cycle?
**Verdict**: WARNING
**Severity**: Medium

---

## Finding 1: run_decay function body — no self-enqueue, no double worker.run()

`run_decay` in `src/workers/main.py` (lines 120–169) does exactly this:

```python
async def run_decay(ctx: dict):
    ...
    from src.workers.decay import run_decay_all_users
    all_stats = await run_decay_all_users(ctx["qdrant"], ctx["neo4j"])
    ...
```

It calls `run_decay_all_users()` exactly **once**. There is no:
- `ctx["redis"].enqueue_job("run_decay")` or any ARQ re-enqueue
- Second call to `run_decay_all_users()`
- Any other path that would produce two full-corpus scans within `run_decay`

`run_decay_all_users` in `src/workers/decay.py` (lines 278–301) iterates per-user, then adds a `user_id=0` pass (the IS NULL fix). Each `worker.run()` call writes its own `decay_run` audit entry via `_log_decay_audit`. There is no summary-level audit write — only per-`worker.run()` entries. The full-corpus entry (processed=5994) is therefore one `worker.run()` call with all memories in scope.

**Conclusion**: No self-enqueue, no internal double-call. The two full-corpus runs come from two separate `run_decay()` invocations at the cron level.

## Finding 2: on_startup and lifecycle hooks in main.py

`WorkerSettings` in `main.py` (lines 330–449):

- `on_startup = startup` — only initializes Qdrant/Neo4j/Redis connections. No `enqueue_job` calls anywhere.
- `on_shutdown = shutdown` — only closes connections.
- No `on_job_start`, `on_job_end`, `after_job_ends`, or any other lifecycle hook that touches `run_decay`.
- `cron_jobs` has **exactly one** entry for `run_decay`:
  ```python
  cron(run_decay, hour={0, 6, 12, 18}, minute=15)
  ```
  No duplicate entry. `run_decay` does not appear anywhere else in `cron_jobs`.

**Conclusion**: No on_startup enqueue, no lifecycle hook trigger. The double-fire is not from code inside the worker configuration.

Between the two full-corpus runs in each cycle, there are 7–9 additional `decay_run` entries with `processed=0` or `processed=2`. These are the per-user shard runs from `run_decay_all_users` (each `worker.run(user_id=uid)` logs its own audit entry). The pattern at 12:15 UTC confirms:

1. Run1 (id=376685, processed=5994) — first full invocation of `run_decay_all_users` completes
2. IDs 378109–378116 (8 entries, processed=0/0/0/0/0/0/0/2) — per-user shards of the **second** invocation
3. Run2 (id=378119, processed=5994) — second `run_decay_all_users` final user pass

The 19–21s gap is equal to the execution time of one full `run_decay_all_users()` pass. **The two runs are sequential, not concurrent.** A stale Redis job sitting in the queue would be picked up and executed concurrently with (or at approximately the same time as) the scheduled run — not exactly one full run-duration later.

## Finding 5: Root cause determination

**A stale Redis job is NOT the source.**

The evidence against the stale-job hypothesis:
1. **Sequential execution with gap = run duration**: If a stale job were pending in the ARQ job queue, ARQ would pick it up before or concurrently with the scheduled cron job. The 19–21s gap equals the time `run_decay_all_users` takes to process 5,994 memories — meaning the second invocation starts immediately after the first completes. This is consistent with two separate cron-triggered calls being dispatched in sequence, not a pre-queued job being consumed.

2. **Variable gaps (13s, 20s, 21s, 259s)**: A stale job would produce a fixed clock-time gap from when it was enqueued. Instead, the gap varies and correlates with corpus size and execution time, ruling out a static queued job.

3. **Off-schedule cycles at 06:45, 09:48**: These are not scheduled cron times (cron fires at 06:15, 12:15, 18:15, 00:15). Off-schedule runs strongly indicate **worker restart catch-up behavior**. ARQ's cron implementation records the last-run time in Redis. When a worker restarts and finds a missed cron window, it fires the job immediately upon startup. This produces: immediate catch-up run (e.g., 06:45) followed by the next regular cron fire (~30min later at 06:49 or the next slot). The 4m19s gap at the 06:45/06:49 cycle is consistent with two separate restart-triggered runs or a restart + scheduled fire.

**Most probable mechanism**: Two ARQ worker processes are running simultaneously (e.g., due to Docker container restart without stopping the old one, or `docker compose up --scale worker=2`). Both workers register the same cron job. When the cron window fires, both workers independently fire `run_decay` within seconds of each other. Worker 1 finishes in ~19s; Worker 2 started slightly after Worker 1 (ARQ distributes cron jobs via Redis, but with two workers both holding cron state, both can fire). This produces two sequential full-corpus runs per cycle.

**Alternative mechanism**: Single worker with ARQ catch-up. The IS NULL fix deployment on 2026-03-14T18:15 restarted the worker. ARQ fired a catch-up run immediately, then the scheduled 18:15 cron fired 13s later. Subsequent cycles show the same 2-run pattern, suggesting the cause persists beyond the initial deployment — consistent with two worker replicas remaining in production.

**Why not stale job**: The stale-job hypothesis would predict one extra run at deployment time only, not a persistent 2-per-cycle pattern across 23+ consecutive waves.

## Verdict Threshold Application

**Verdict: WARNING** — The source of the double-decay has been identified through static analysis and audit log pattern analysis: the double-enqueue is not from a stale Redis job but from two ARQ worker processes (or persistent ARQ catch-up restart behavior) causing two sequential `run_decay` invocations per cron cycle. The root cause is persistent (23+ waves) and architectural (worker replica count or restart policy), not a transient queue artifact. No fix has been deployed. Each cycle processes the full corpus twice, inflating decay by approximately 2× per 6-hour period.

**Fix required**: Verify `docker compose ps` on CasaOS to confirm whether 2 worker containers are running. If yes, scale to 1 replica. If only 1 worker, investigate ARQ cron state key collisions (ARQ stores next-run timestamps in Redis under `arq:cron:*` keys — if these were not cleared on redeploy, they may be firing at incorrect intervals).
