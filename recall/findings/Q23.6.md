# Finding: Q23.6 — Per-supersedure audit logging EXISTS; WARNING — user_id always null; 26/1000 duplicate memory_ids from double-processing

**Question**: Mark_superseded audit coverage — does consolidation emit per-supersedure audit entries, or is each consolidation event invisible except for the aggregate consolidate action?
**Verdict**: WARNING — Per-supersedure `log_audit("supersede", ...)` calls exist in consolidation.py (lines 299–305) and fire for every source memory in a cluster. 1000+ entries confirmed in audit trail spanning 2026-03-13 to 2026-03-15. However: (1) user_id is always null — Q24.5 bug (missing user_id= in Memory() constructor) propagates to the supersede audit trail, making per-user attribution impossible; (2) 26/1000 recent entries have duplicate memory_ids, indicating Q22.1 double-decay causes repeated consolidation of already-superseded memories. Chain is reconstructable (source→merged via details.superseded_by), but attribution is broken.
**Severity**: WARNING — Not FAILURE (entries exist, coverage is consistent). Not HEALTHY (user_id null, duplicate events from Q22.1 contamination).

## Finding 1: Static Code Analysis — consolidation.py

```python
# consolidation.py:287–307
# Audit log — consolidation merge + supersedes
try:
    from src.storage import get_postgres_store

    pg = await get_postgres_store()
    source_ids = [m.id for m in cluster]
    await pg.log_audit(
        "consolidate",
        merged.id,
        actor="consolidation",
        details={"source_ids": source_ids, "source_count": len(source_ids)},
    )
    for source_memory in cluster:
        await pg.log_audit(
            "supersede",
            source_memory.id,
            actor="consolidation",
            details={"superseded_by": merged.id},
        )
except Exception as audit_err:
    logger.warning("consolidation_audit_failed", error=str(audit_err))
```

Two log_audit calls per cluster:
1. **consolidate** action: one entry per cluster, memory_id = merged.id, details include source_ids list
2. **supersede** action: one entry per source memory, memory_id = source_memory.id, details = {"superseded_by": merged.id}

This is per-supersedure logging — each mark_superseded() call is paired with a log_audit("supersede", ...) call in the same block.

## Finding 4: Data Quality Gaps

### Gap 1: user_id always null
```
log_audit("supersede", source_memory.id, actor="consolidation", details={...})
```
No user_id parameter is passed. All 1000 sampled entries show `user_id: null`. Root cause: Q24.5 FAILURE — consolidation.py missing user_id propagation throughout. The supersede audit trail cannot answer: "which user's memories were consolidated?"

### Gap 2: Duplicate memory_ids (Q22.1 contamination)
```
Unique memory_ids in 1000 entries: 974
Duplicate memory_ids: 26 (2.6%)
```
26 memory_ids appear more than once in the recent 1000 entries. These are memories that were superseded, then re-processed by a later consolidation run — consistent with Q22.1 (double-decay + global processing causes already-superseded memories to be re-evaluated as cluster candidates and re-superseded). Each re-supersedure generates a new audit entry for the same memory_id.
