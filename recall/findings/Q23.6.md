**Question**: Mark_superseded audit coverage — does consolidation emit per-supersedure audit entries, or is each consolidation event invisible except for the aggregate consolidate action?
**Verdict**: HEALTHY with caveat (per-supersedure audit entries exist: action="supersede", memory_id=source_id, details.superseded_by=merged_id, actor="consolidation"; 1000 entries visible spanning 2026-03-12T14:00 to 2026-03-15T07:02 (3 days); paired with consolidate entries, full supersedure chain is reconstructable; CAVEAT: user_id=null in all supersede entries — to identify which user's memory was superseded requires joining against memory table by memory_id; this means Q22.9's reliance on reconcile mismatch count for attribution loss counting was overly indirect — supersede audit provides direct evidence)
**Severity**: Info — audit coverage is adequate; user_id omission from supersede entries is a minor observability gap

## Finding 2: Correlation with Consolidate Entries

Each consolidate audit entry (action="consolidate") corresponds to multiple supersede entries:

```
2026-03-15T07:01:52 | consolidate | source_count=3, source_ids=[10c08ff5, 8e383749, 90b7513b]
2026-03-15T07:01:52 | supersede   | memory_id=90b7513b → superseded_by=47b1d059
2026-03-15T07:01:52 | supersede   | memory_id=8e383749 → superseded_by=47b1d059
2026-03-15T07:01:52 | supersede   | memory_id=10c08ff5 → superseded_by=47b1d059
```

The consolidate entry and its N supersede entries share the same timestamp (down to milliseconds). The supersede entries' `superseded_by` values match the merged memory ID implied by the consolidate entry.

**Total coverage**: 1000 visible supersede entries over 3 days; at ~700/day rate, the visible window represents ~3 days × 700 = 2,100 supersedures with 1000 visible (API cap at 1000). Pagination would be required to see the full historical record.

---

## Finding 3: User Attribution Gap in Audit

All 1000 supersede entries have `user_id: null`. This means:
- The audit trail records WHICH memories were superseded, but not WHOSE memories
- To determine attribution loss, a join against the memory record is required:
  `audit.memory_id → memory.user_id`
- Since the source memory may later be garbage collected (DETACH DELETE), this join window may close

**Practical impact**: The supersede audit is sufficient for answering "which memory was superseded and into what", but insufficient for "which user lost attribution" without a concurrent memory lookup.

## Correction to Q22.9 Methodology

Q22.9 stated: "event reconstruction had to infer individual supersedure events from the mismatch count rather than reading them directly from the audit trail"

**Correction**: Individual supersedure events ARE directly available via `GET /admin/audit?action=supersede`. Q22.9's use of the reconcile mismatch count as a proxy was unnecessarily indirect. The supersede audit provides a direct, deterministic count of supersedure events.

**However**, Q22.9 was investigating events 8+ hours prior, and the audit API cap at 1000 entries may have obscured older entries. At 700 supersedures/day, 1000 entries spans only ~1.4 days — so entries from 13+ days prior (the full corpus history) would require higher-limit queries or database access.
