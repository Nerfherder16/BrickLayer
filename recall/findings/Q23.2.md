## Phase 4 Detail — Current State (Q22.1 Still Active)

```
2026-03-15T06:45:09 | proc=5936 dec=2258 uid=None  ← first system run
2026-03-15T06:49:28 | proc=5936 dec=1907 uid=None  ← second system run (4 min later)
```

Both entries show `proc=5936` (processing all active memories) with `uid=None` (system-level run). The 4-minute separation suggests:
1. The system was restarted around 06:30-06:40 (consistent with Q22.9 observation: "After ~06:30: system runs return")
2. On restart, the ARQ scheduler triggered an immediate decay run (06:45)
3. A second trigger fired 4 minutes later (06:49) — possibly from a duplicate cron registration or a queued-then-executed job

**Why this is double-decay:** The `last_accessed` field in Qdrant is NOT updated by decay operations — it only updates when a memory is retrieved/accessed by a user. When the second run at 06:49 executes:
- It reads each memory's current `importance` (already reduced by the 06:45 run)
- It reads each memory's `last_accessed` (unchanged — same as before 06:45)
- It computes `hours_since = now - last_accessed` — the SAME hours as the 06:45 run
- It applies `importance × (0.99)^hours_since` — decaying an already-decayed value by the same factor
- This is mechanically identical to the Q22.1 double-decay
