# Finding: Q23.5 — Consolidation worker rate and backoff — HEALTHY; worker runs on 1-hour fixed schedule (NOT continuous); processes similar pairs in 1-4 second bursts of 64-65 seconds per batch; natural termination when similar pairs exhausted; Q22.9 "runs continuously" was incorrect — 3528-3540 second inter-batch gaps confirm hourly cron schedule

**Question**: Consolidation worker rate and backoff — does the continuously-running consolidation worker have any rate limit or natural termination condition?
**Verdict**: HEALTHY (consolidation runs on a fixed **1-hour schedule**, confirmed by 3528-3540 second gaps between hourly batches; within each batch, consolidations complete in 1-4 second bursts over ~64 seconds; natural termination is evident from the highly variable per-hour count (1-28/h) — when similar pairs are exhausted, the batch terminates early; no unbounded looping; Q22.9 incorrectly described this as "continuously running" — the 06:00-06:01 window observed in Q22.9 was a single hourly batch executing; rate is 299/day = ~12.5/hour average, max observed 28 in one hour)
**Severity**: Info — corrects Q22.9 characterization; no new bug; consolidation behavior is well-controlled

---

## Key Measurement: Inter-Batch Gap

From 50 most recent consolidate audit entries, gap analysis:

```
07:00:47 to 07:01:52  ← hourly batch at 07:00 (65 seconds total, 7 consolidations)
07:00:50 to 06:01:49: **3540 second gap** = 59 minutes
06:00:45 to 06:01:49  ← hourly batch at 06:00 (64 seconds total, 12 consolidations)
06:00:45 to 05:01:57: **3528 second gap** = 58.8 minutes
05:01:55 to 05:01:57  ← hourly batch at 05:00
```

**Conclusion**: The consolidation worker runs on a **~60-minute (1-hour) cron schedule**, not continuously.

---

## Batch Processing Pattern

Within each hourly batch:
- Inter-consolidation gaps: **1-4 seconds** (rapid sequential processing)
- Total batch duration: **~60-65 seconds**
- Consolidations per batch: **1-28** (highly variable)
- Average source_count: **2.31** (range: 2-8)

The variable per-batch count is the natural termination signal: the worker scans for similar pairs until no more exceed the similarity threshold, then stops. No explicit per-batch limit is required — the similarity search naturally converges when the corpus is sufficiently consolidated.

Q22.9 stated: "consolidation runs continuously at 06:00–06:01"

**Correct interpretation**: The 06:00–06:01 window was a single **hourly batch** executing. The worker ran for 64 seconds, processed 12 consolidations, then stopped. It does not run "continuously" in any sense — it fires once per hour for ~1 minute.

**Verdict: HEALTHY** — Consolidation runs on a 1-hour cron schedule, executes in ~60 second batches of 1-28 consolidations, and naturally terminates when the similarity search finds no more pairs above threshold. The actual consolidation load (~299/day, 12.5/hour average) is predictable and bounded by the similarity pool size.
