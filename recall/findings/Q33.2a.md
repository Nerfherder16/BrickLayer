## Finding 4: True Source of Double-Decay — run_decay_all_users Logic Bug

The 12:15 UTC decay run (the most data-rich, 5994 memories) reveals the actual cause:

```
12:15:04  decay_scroll_done    count=5994   # user_id=1 (all memories)
12:15:24  decay_completed      decayed=1423 processed=5994  # user pass 1
...
[6 empty user iterations: user_ids 2-7 with 0 memories]
...
12:15:24  decay_completed      decayed=2    processed=2     # user_id=8 (2 system-owned)
12:15:26  decay_scroll_done    count=5994   # system/null pass
12:15:43  decay_completed      decayed=1315 processed=5994  # system pass 2
```

**Final log line**: `decay_complete total_decayed=2740 users_processed=9`

The root cause is in `src/workers/decay.py::run_decay_all_users()`:

```python
async def run_decay_all_users(qdrant_store, neo4j_store):
    worker = DecayWorker(qdrant_store, neo4j_store)
    user_ids = await qdrant_store.get_distinct_user_ids()

    for uid in user_ids:
        stats = await worker.run(user_id=uid)   # user_id=1 → scrolls ALL 5994 memories

    # "user_id IS NULL only — avoids double-decay" — COMMENT IS WRONG
    stats = await worker.run(user_id=0)          # user_id=0 → also scrolls ALL 5994 memories
```

**The problem**: `scroll_all(user_id=uid)` when `uid=1` (the primary user) returns ALL 5994 memories — not just memories belonging to user 1. The per-user filter is not working as intended. Then the `user_id=0` system pass also scrolls all 5994 memories and applies decay again. Result: every memory is decayed **twice per cron cycle**, producing ~2738 total decayed entries when ~1423 would be correct.

The comment in the code says "avoids double-decay of user memories" but this protection is not effective because the user_id scoping in `scroll_all` is apparently returning all memories regardless of user filter for user_id=1.

## Verdict Threshold Application

**Verdict: WARNING** — Two replicas are definitively NOT the cause (one container, no ARQ heartbeats, fixed container_name). However, the double-decay source is confirmed and identified: `run_decay_all_users` runs a full-corpus decay pass for user_id=1 AND a second full-corpus decay pass for user_id=0 (system), because `scroll_all(user_id=1)` returns all 5994 memories rather than only user-1-owned memories. This is a logic bug in either `scroll_all`'s user scoping or in how user ownership is stored.

**Actionable fix**:

1. **Immediate**: Add a guard in `run_decay_all_users` — skip the `user_id=0` system pass if all memories were already covered by user passes (check `users_processed > 0 and total from user passes == total qdrant count`).

2. **Root fix**: Audit `scroll_all(user_id=uid)` in `src/storage/qdrant_store.py` — verify the Qdrant filter for `user_id` is correctly scoping by payload field. If all memories have `user_id: 1` in payload, the system pass (`user_id=0` / NULL filter) would also return them if the NULL filter is incorrect.

3. **No docker change needed** — `docker compose scale worker=1` is already the state.
