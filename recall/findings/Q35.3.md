## Key Findings

### 1. Development domain is the primary casualty zone
229 floor-clamped development memories = 27.1% of all casualties. This is more than double any other domain. Development memories document code patterns, architectural decisions, implementation notes, and debug findings — exactly the context that makes cross-session work coherent. Their loss is high-impact.

### 2. Distribution is not random — it follows memory production rates
The ranking (development > general > frontend > api > configuration > infrastructure) matches the approximate rate at which memories are created in each domain. There is no evidence of targeted decay hitting specific domains disproportionately. All domains decay uniformly under the same time-based function; domains producing more memories accumulate more casualties.

### 3. Critical domains hold 413 casualties above floor
With 229 development + 67 infrastructure + 29 recall + 29 database + 26 security + 23 tooling + 10 devops = 413 memories at floor, the system has lost substantial structured knowledge across every operational domain. The recall domain (29 memories) being floor-clamped means memories about the memory system itself have decayed — a recursive loss that may impair the system's self-knowledge.

### 4. Growth since Q34.7: +202 casualties
Q34.7 reported 644; current count is 846. That is a 31% increase of unknown elapsed time (Q34.7 was earlier in this wave). This confirms the crisis is ongoing and accelerating.
