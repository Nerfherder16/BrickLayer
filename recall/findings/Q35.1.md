# Q35.1 — Rehabilitate-Importance Endpoint: Does It Catch Floor-Clamped Casualties?

**Status: FAILURE**

## Question

Does the `rehabilitate-importance` endpoint catch the 644 floor-clamped premature casualties (clamped at exactly 0.050)?

## Findings

### Endpoint Exists

`POST /admin/importance/rehabilitate` is defined in `src/api/routes/admin.py:1097`.

### Critical Defect: Wrong Filter Operator (Line 1123)

```python
# Only rehabilitate floor-level memories
if importance >= 0.05:
    continue
```

The filter uses `>= 0.05`, meaning it **skips** any memory with `importance == 0.05` exactly. The 644 floor-clamped casualties were clamped by the decay floor to exactly `0.050`. Every one of them passes `importance >= 0.05` as `True` and is skipped by the `continue`.

The correct filter should be `if importance > 0.05:` — i.e., only exclude memories *above* the floor. Memories at exactly the floor (0.050) are the primary targets.

### No Scope by user_id

The endpoint iterates `await qdrant.scroll_all()` with no `user_id` parameter. This is not a defect — it processes all users in one pass — but it means there is no per-user targeting capability.

### Endpoint Has Never Been Run

Audit log query for `action=rehabilitate`:
```json
{"entries":[],"count":0}
```
Zero entries. The endpoint has never been invoked in production.

### Rehabilitation Formula (Correct When Reached)

If a memory did pass the filter (importance strictly `< 0.05`), the restoration formula is sound:
- High-access (`access_count >= 3`) or pinned: `max(importance, initial_importance * 0.5, 0.3)` — restores to at least 0.3
- Durable/permanent: `max(importance, 0.2)` — restores to at least 0.2

The formula is appropriate. The defect is entirely in the filter predicate.

## Root Cause

Off-by-one operator: `>= 0.05` instead of `> 0.05`. The floor is defined as `0.05` throughout the system. Memories clamped *to* the floor are the rehabilitation targets. The current filter excludes them entirely.

## Impact

644 floor-clamped memories (confirmed in Wave 34 findings) would survive the scan entirely unprocessed. They remain stuck at importance 0.050 indefinitely unless the operator is corrected and the endpoint is run.

## Verdict

**FAILURE** — endpoint uses `>= 0.05` (misses floor at exactly 0.05) AND has never been run. Both failure conditions from the question rubric are satisfied.

## Required Fix

`src/api/routes/admin.py:1123` — change:
```python
if importance >= 0.05:
```
to:
```python
if importance > 0.05:
```
Then invoke the endpoint once to process the existing backlog.
