## Analysis

### Segmentation of the 846 Null-Pool

The amnesty endpoint's graph_strength criterion (>= 2.0) provides a floor of 0.25, but the
durable/None durability criterion is the primary eligibility gate. The 846 break down as:

**N1 — Rescued by amnesty (durable/None, importance < 0.4)**
- Estimated: 846 - N3 = 846 - 8 = **~838 memories**
- These had durability "durable" or None (defaulting to durable)
- Amnesty boosted them from 0.05 to max(current, initial×0.5, 0.3) = 0.30-0.35
- Evidence: Q36.3 direct count found only ~8 ephemeral ac<3 in null-pool
- Evidence: Post-amnesty dry_run shows only 5 residual eligible across full active pool

**N2 — Skipped by amnesty due to graph criteria but still durable**
Per source code review, there is no separate "insufficient graph strength" skip condition.
The amnesty skips only: ephemeral, pinned/permanent, or importance >= 0.4.
Graph strength affects the boost *amount* (floor of 0.25 if graph_strength >= 2.0) but
NOT eligibility. All durable/None memories below 0.4 are processed.

Correction from Q36.1 framing: the "899 skipped due to insufficient graph strength" interpretation
was incorrect. The 899 skipped in Q36.1 were: ephemeral + pinned/permanent + (importance >= 0.4 in batch).
There is no graph-strength-based skip. N2 as a distinct segment does not exist.

**N3 — Stranded (ephemeral, amnesty-ineligible)**
- Estimated: **~8 memories** (from Q36.3 direct count: ephemeral durability, access_count < 3)
- These are legitimately short-lived memories that should not be rehabilitated
- Importance at 0.05 is appropriate for ephemeral memories with low access
- Neither amnesty nor rehabilitate endpoint rescues these (by design)

### Proportional Estimate vs Direct Count

Secondary proportional estimate (for cross-check):
- Null-pool fraction of active batch: 846 / 6,110 = 13.8%
- Total skipped in Q36.1: 899 × 13.8% = ~124 null-pool skipped (N3 proportional)
- Direct count (Q36.3): ~8 null-pool ephemeral ac<3

The proportional estimate (124) is an overestimate because:
1. The null-pool (null user_id) skews toward older, pre-user-system memories
2. Older memories are disproportionately "durable" or durability=None (default)
3. Ephemeral memories are newer (assigned at creation) and more likely to have a user_id
4. The Q36.3 direct count used an actual DB query filtered to null user_id + ephemeral + ac<3

The direct count of ~8 is the more reliable figure.

### Graph Connectivity Picture

The amnesty endpoint uses Neo4j `get_bulk_edge_strengths()` to score all active memories.
For the 846 null-pool casualties:

- Well-connected (graph_strength >= 2.0): received floor of 0.25 in boost formula
- Weakly connected (graph_strength < 2.0): still eligible if durable; boost = max(current, initial×0.5, 0.3)
- Both categories boosted to ~0.30-0.35 (from initial_importance × 0.5 = 0.5 × 0.5 = 0.25, but floor of 0.3 applies)

Graph isolation did NOT create an N2 "stranded via graph" segment. The endpoint's design ensures all durable memories are rescued regardless of graph connectivity — graph strength only affects the boost ceiling, not eligibility.

### Post-Amnesty Dry_Run Confirmation

The 5 "boosted" memories in the post-amnesty dry_run have:
- importance: 0.37-0.40 (near but below 0.4 threshold)
- durability: null (defaulting to durable) or "durable"
- graph_strength: 2.46-5.65 (all well-connected)

These are NOT v3.2.5 bug victims at 0.05 — they have decayed naturally to sub-0.4 range since the
amnesty ran. They represent ongoing normal decay, not residual floor-clamping.
