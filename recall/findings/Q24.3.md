## Finding 2: create_memory_node() Code Path — No Defect

**Location**: `src/storage/neo4j_store.py:87-124`

```python
async def create_memory_node(self, memory: Memory):
    async with self.driver.session() as session:
        await session.run(
            """
            MERGE (m:Memory {id: $id})
            SET m.memory_type = $memory_type,
                m.domain = $domain,
                m.importance = $importance,    ← explicitly set from memory.importance
                ...
            """,
            id=memory.id,
            importance=memory.importance,      ← passed as parameter
            ...
        )
```

The Cypher uses `MERGE` (create-or-update) with `SET m.importance = $importance` where `$importance` comes from `memory.importance`. This cannot produce neo4j=0.0 unless `memory.importance=0.0` at creation time (which is valid by the Pydantic schema but would require an explicit caller override).

**Store path error handling** (`memory.py:278-281`):
```python
try:
    neo4j = await get_neo4j_store()
    await neo4j.create_memory_node(memory)
except Exception as neo4j_err:
    logger.error("neo4j_write_failed_compensating", id=memory.id, error=str(neo4j_err))
```
If neo4j creation fails, the exception is swallowed (logged only). The memory is stored in Qdrant with correct importance but has NO Neo4j node — appearing as a `qdrant_orphan` in reconcile. The current reconcile shows `qdrant_orphans: 0`, confirming no orphaned Qdrant points. No silent neo4j creation failures are occurring.

## Finding 3: Reconciliation of Q23.4 "Active Mismatch" Observation

**Q23.4 claimed**: "50 NEW active-memory mismatches (neo4j=0.0, qdrant=0.7+) — different bug from Q21.1"

**What actually happened**:
1. Memory `41569d71` created 2026-03-15T05:33:27 — active at Q23.4 measurement time
2. Heavy consolidation on 2026-03-15T05:01-08:02 (~100 supersedures in 3h)
3. `41569d71` was in or near a consolidation batch that ran after 05:33
4. Post-Q23.4 measurement: `41569d71` got superseded via consolidation → qdrant.mark_superseded sets qdrant.superseded_by; neo4j.mark_superseded had already set neo4j.importance=0.0
5. Now: `41569d71` is superseded in both stores, appearing as a supersede-path mismatch (not creation-path)

**Alternative explanation** (millisecond-window): consolidation.py calls qdrant.mark_superseded (line 282) THEN neo4j.mark_superseded (line 283). If neo4j was called first in a prior code version, or if there was a brief window visible in Q23.4's measurement, neo4j could have importance=0.0 before qdrant had superseded_by set, making the memory appear active in a reconcile scan. However, current consolidation.py:282-283 order shows qdrant first, making this explanation less likely.

**Conclusion**: Q23.4's observation was accurate for its measurement instant — 50 memories appeared to be active with neo4j=0.0. The underlying cause was the mark_superseded path operating on recently-created memories, NOT a separate creation-path importance bug.
