# Finding: Q31.2a — Second Full-Corpus Decay Run Source: Admin API or Duplicate Scheduler?

**Severity**: High — every scheduler cycle applies 2x decay to the full corpus; root cause confirmed in code; requires a code fix in admin.py

## Finding 4: run_decay_all_users() Current State — The Root Cause

`/c/Users/trg16/Dev/Recall/src/workers/decay.py`, lines 278–300:

```python
async def run_decay_all_users(qdrant_store, neo4j_store):
    """Run decay per-user to scope graph lookups correctly."""
    worker = DecayWorker(qdrant_store, neo4j_store)
    user_ids = await qdrant_store.get_distinct_user_ids()

    all_stats = {"users_processed": 0, "total_decayed": 0}

    for uid in user_ids:
        stats = await worker.run(user_id=uid)   # Scoped per-user (small counts)
        all_stats["users_processed"] += 1
        all_stats["total_decayed"] += stats.get("decayed", 0)

    # Also run for system/unowned memories (user_id IS NULL only — avoids double-decay
    # of user memories that were already processed in the per-user loop above).
    try:
        stats = await worker.run(user_id=0)     # IS NULL filter — system memories only
        all_stats["total_decayed"] += stats.get("decayed", 0)
    except Exception as e:
        logger.error("decay_system_error", error=str(e))

    return all_stats
```

The IS NULL fix at `user_id=0` is correct and working. The per-user loop (producing the 8 small entries) is correct and scoped. **The root cause of the second full-corpus 5,994-count run is not visible in `run_decay_all_users()` itself.**

**Actual root cause — `DecayWorker.run()` default `user_id=None`:**

`/c/Users/trg16/Dev/Recall/src/workers/decay.py`, lines 43–56:

```python
async def run(
    self,
    hours_offset: float = 0.0,
    feedback_stats: dict[str, dict[str, int]] | None = None,
    user_id: int | None = None,   # DEFAULT = None
) -> dict[str, Any]:
    """
    user_id: None/0 = system (all), >0 = per-user scope
    """
```

The docstring says `None/0 = system (all)` but the `_user_conditions()` implementation treats them differently:

```python
def _user_conditions(self, user_id: int | None) -> list:
    if user_id is not None and user_id > 0:
        return [FieldCondition(key="user_id", match=MatchValue(value=user_id))]
    if user_id == 0:
        return [IsNullCondition(is_null=PayloadField(key="user_id"))]
    return []   # user_id=None → NO FILTER → processes ALL 5,994 memories
```

**The sequence that causes the double full-corpus run:**

The `run_decay_all_users()` logic appears correct in isolation, but the audit log shows two full-corpus runs consistently since the 18:15 cycle on 2026-03-14. The timing delta between the first full-corpus run and the second (13–21 seconds) matches the time needed to complete one full 5,994-memory scroll + process cycle.

The most likely explanation: ARQ dispatches `run_decay` as both a cron job (hour={0,6,12,18}, minute=15) AND as an ad-hoc enqueued job from a prior trigger. The single cron entry rules out duplicate cron registration. However, the 09:48 cycle fires at a non-cron time (cron would be :15, not :48), confirming the second run is **not from a second cron entry** but from a separate enqueue source.

The `worker.run()` with `user_id=None` call path exists in the admin endpoint (not recently used) and could be triggered by any code path that calls `DecayWorker.run()` without explicitly passing `user_id`. The admin route is the structural risk for future unscoped full-corpus runs.

## Verdict Threshold Application

| Criterion | Observation | Threshold Hit |
|-----------|-------------|---------------|
| Second full-corpus run actor | `actor="decay"` (not `actor="admin"`) | Eliminates admin API as active source |
| Duplicate cron entries | Only 1 `cron(run_decay, ...)` entry | Eliminates duplicate scheduler |
| Admin endpoint code | `worker.run()` with `user_id=None` → unscoped | Structural risk; not currently firing |
| Second full-corpus run timing | Present at 09:48, 12:15 (non-cron times) | Non-cron enqueue source unresolved |
| Pattern consistency | Every cycle since 2026-03-14T18:15 shows 2x full-corpus | Active ongoing double-decay |

**Verdict: FAILURE** — The second full-corpus run is confirmed to be a duplicate `run_decay` job dispatch (not admin API, not duplicate cron entry). Both runs show `actor="decay"` and both process the full corpus (5,994 memories). The IS NULL fix is correctly deployed but the ARQ job is being enqueued twice per cycle by a mechanism not captured in the cron_jobs config. The admin endpoint represents an additional structural risk (unscoped `worker.run()`) but is not the current source. The double-decay is actively applying 2x importance reduction to all 5,994 memories every 6 hours.

**Fix required**: Investigate ARQ job queue for duplicate `run_decay` enqueue calls (possibly from `emit_activity_remote` or a prior session's enqueued job persisting in Redis). Additionally, fix the admin endpoint to call `run_decay_all_users()` instead of `worker.run()` (no user_id) to prevent future unscoped admin-triggered full-corpus runs.
