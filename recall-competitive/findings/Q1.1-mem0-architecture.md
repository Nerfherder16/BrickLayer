# Finding: Q1.1 — mem0 Architecture and Feature Parity

**Question**: What is mem0's current architecture, feature set, and integration surface? How does its dual-layer memory approach (vector + graph optional) compare to Recall's Qdrant + Neo4j dual-store? Does mem0's Python/JS SDK make it meaningfully easier to adopt than Recall's hook-only integration?
**Verdict**: WARNING
**Severity**: High

## Evidence

**Sources**: github.com/mem0ai/mem0, docs.mem0.ai, pypi.org/project/mem0ai, mem0 blog (2024-2025)

### mem0 Architecture Overview

mem0 uses a three-layer memory architecture:
- **Storage**: Vector store (Qdrant, Chroma, Pinecone, PGVector, Weaviate — pluggable), optional graph store (Neo4j), optional SQLite history store
- **LLM layer**: Pluggable — supports 15+ providers via LiteLLM abstractions: OpenAI, Anthropic, Ollama, Groq, Azure, Bedrock, Mistral, Google AI, xAI, LM Studio, LiteLLM passthrough
- **Embedding**: Pluggable — OpenAI, Ollama, HuggingFace, Vertex, etc.

### SDK Surface

mem0 Python SDK (`pip install mem0ai`):
```python
from mem0 import Memory
m = Memory()
m.add("I prefer Python for data tasks", user_id="alice")  # store
results = m.search("Python preference", user_id="alice")  # search
m.update(memory_id, "I now prefer Rust")                  # update
m.delete(memory_id)                                        # delete
m.get_all(user_id="alice")                                 # list all
```

TypeScript SDK also available (`@mem0ai/mem0`).

Zero-infra mode: embedded Qdrant, no external services. Full OSS self-hosted (with graph): 3 Docker containers (mem0, Qdrant, Neo4j).

### Graph Memory

mem0 graph mode uses Neo4j. It extracts entity-relationship triples from memory content via LLM. Example: "Alice likes Python" → `(alice)-[:LIKES]->(python)`. **No typed ontology** — relationships are LLM-generated strings, not a typed schema. No causal edge types (CAUSED_BY, SUPPORTS, CONTRADICTS). Graph cleanup is documented as manual Cypher queries.

### Memory Lifecycle Management

**mem0 OSS: zero automatic lifecycle management.**
- No decay
- No consolidation (ongoing)
- No hygiene / GC pipeline
- No dream consolidation
- "Update Phase" during `add()` is write-time only — LLM decides to update/delete existing memories that contradict the new one. No background process.
- mem0 Platform (cloud only) has `expiration_date` per memory.

### Deduplication

mem0 uses an LLM-judged approach at write time: when adding a new memory, the LLM is given existing similar memories and asked whether to ADD, UPDATE, DELETE, or NONE. This is semantic-aware but LLM-dependent and adds latency to every write.

### Active Pain Points (GitHub Issues, 2025)

- Default config hardcoded to OpenAI — ignores custom config files
- Ollama integration fails on embedding dimension mismatches
- AWS Bedrock config broken (wrong type mapping)
- These affect self-hosted deployments specifically

## Mitigation Recommendation

1. **Thin Python SDK** (~1-2 weeks): Wrap FastAPI with `recall-sdk` — `store()`, `search()`, `get()`, `delete()`. No backend changes.
2. **Recall-lite zero-infra mode** (~1 week): Single-container with embedded Qdrant + SQLite, no Neo4j/Redis/Postgres.
3. **Docs site** (~1 week): MkDocs/Mintlify — FastAPI OpenAPI spec auto-generates the API reference.
4. **LiteLLM for extraction LLM** (~2-3 weeks): Unlocks 100+ providers without changing hook system.
5. **Lead with lifecycle differentiation**: No competitor has Recall's automated lifecycle pipeline in OSS. This should be the headline of the README.
