# Finding: Q3.4 — Multi-User Isolation

**Question**: How do competitors handle multi-user memory isolation? What does mem0's user management look like? Does Zep have workspace/project separation? How does Recall's user_id field compare in practice? Is multi-user isolation a feature that Recall's target users (homelab, single-user) actually need?

## Evidence

### mem0: Four-Dimension Isolation with Org-Level Scoping

From docs.mem0.ai/platform/features/entity-scoped-memory (scraped 2026-03-15) and docs.mem0.ai/cookbooks/essentials/entity-partitioning-playbook:

mem0 implements four independent scoping dimensions:

| Dimension | Field | Scope | Example |
|-----------|-------|-------|---------|
| User | `user_id` | Persistent account or persona | `"customer_6412"` |
| Agent | `agent_id` | Distinct agent context | `"meal_planner"` |
| Application | `app_id` | Product surface or white-label | `"ios_retail_demo"` |
| Session | `run_id` | Short-lived conversation/thread | `"ticket-9241"` |

Implicit null scoping: passing only `{"user_id": "alice"}` automatically restricts results to records where `agent_id`, `app_id`, and `run_id` are null. This prevents accidental cross-agent leakage without requiring explicit filters on every query.

**Multi-tenant architecture** (from mem0docs.xyz/task/blog/single-user-vs-multi-tenant-ai-memory-architecture/, 2026-03-09):

mem0's docs explicitly document three isolation tiers for SaaS deployments:
1. **User-level memory** (`user_id`): personal preferences, per-user history
2. **Organizational memory** (`org_id`): company-wide context shared across all users in an organization
3. **Agent-level memory** (`agent_id`): task-specific context, isolated per workflow

This three-tier model (user/org/agent) is the same pattern used by Salesforce, HubSpot, and other enterprise SaaS platforms. It enables row-level security without database schema changes.

**Cloud platform**: mem0's hosted platform adds project-level API key scoping, org dashboards, and GDPR-compliant RTBF (Right to Be Forgotten) via the delete-by-user API.

**Self-hosted**: The same `user_id`/`agent_id`/`run_id` API works identically in the self-hosted OSS package. No infrastructure difference — isolation is enforced in the query layer via vector DB metadata filtering.

**Known issue — multi-agent bleed**: GitHub issue #3998 (openclaw/openclaw, Feb 2026): In multi-agent setups sharing one gateway user ID, memories bleed between agents. The fix (extracting agentId from session key as effective userId) is a 15-line community patch, not a first-party solution. This suggests the implicit null scoping has edge cases in practice.

### Zep: User-Centric Graph Isolation with RTBF

From help.getzep.com/users (scraped 2026-03-15):

Zep's isolation model is **user-node-based** in a knowledge graph:

- Each user has a unique `UserID` (any string: username, email, UUID)
- Each user has an associated **User Graph** — a private knowledge graph containing all that user's context across all threads
- Users can have multiple `Thread`s, but all thread data integrates into a single User Graph
- Cross-thread context integration is automatic: "insights learned in one conversation thread are automatically available in all other threads for the same user"

**RTBF implementation**: Deleting a user cascades to all threads, thread artifacts, and the entire user graph. This is a single API call — production-grade compliance support.

**No organizational-level scoping**: Zep does not expose an `org_id` or `workspace_id` concept in its current API. Each user is independent; there is no shared org-level memory tier. This is a gap vs mem0 for multi-tenant SaaS use cases.

**Cloud only**: Community Edition was archived April 2025. All current Zep usage is cloud API. Self-hosted multi-user isolation is not available outside cloud pricing.

### Letta: Agent-Level Isolation

Letta's memory model is **agent-centric**, not user-centric. Each agent has its own memory blocks (core memory, archival memory, recall memory). Multiple users interacting with a shared Letta agent share that agent's memory unless the application explicitly creates separate agents per user.

For multi-user scenarios, the recommended pattern is to provision a separate Letta agent per user. This gives complete memory isolation but multiplies infrastructure costs linearly with user count. There is no native `user_id` scoping within a single agent — the isolation boundary is the agent itself.

**Practical implication**: Letta is best suited for single-agent-per-user deployments. Multi-tenant shared-agent deployments require application-layer routing.

### Recall: user_id Field Exists; No UI or Multi-User Workflow

Recall's memory schema includes a `user_id` field on stored memories. The field is present and queryable — the API filter system accepts `user_id` as a parameter.

**What is missing**:
- No user management UI (create user, list users, delete user + cascade)
- No RTBF delete-by-user API endpoint
- No organizational or team scoping dimension
- No documented multi-user workflow or quickstart
- The dashboard does not expose user selection or per-user views
- No per-user API key support (all requests use a single shared token)

In practice, Recall operates as a single-user homelab system. The `user_id` field is a schema artifact rather than an enforced isolation boundary with full lifecycle support.

### Is Multi-User Isolation a Priority for Recall's Target Users?

Recall's target persona (per CLAUDE.md and project-brief context) is: homelab developer, single user, self-hosted, privacy-first. This persona has zero multi-user isolation requirements.

However, the market signal from community data suggests adjacent use cases that do require isolation:

1. **Family hub / household assistants** (Tim's own "Sadie" project): Two+ users (Tim, family members) with distinct memory spaces. A `user_id` per family member is needed but no tooling exists to manage it.
2. **Agent pool use cases**: A Claude Code user running multiple AI agents simultaneously (e.g., a coding agent and a research agent) needs agent-level isolation to prevent cross-context pollution. The openclaw GitHub issues #3998 and #38417 (Feb-Mar 2026) document exactly this failure mode with mem0 in multi-agent setups.
3. **Small team / pair development**: Two developers sharing a homelab instance would need per-developer memory spaces. Currently unsupported.

**The gap is not SaaS-scale multi-tenancy** — that is clearly out of scope for Recall's homelab target. The gap is **household and multi-agent isolation**, which IS within the target use case and requires only 2-5 named user profiles, not thousands.

## Mitigation Recommendation

**Priority 1 — Multi-agent isolation via agent_id field (effort: 2-3 days)**
Add `agent_id` as a first-class filter dimension alongside `user_id`. This directly addresses the multi-agent bleed problem documented in GitHub issues (openclaw #3998, openclaw #38417) and is a likely near-term need for Recall users running Claude Code agent swarms. Zero schema migration required — add the filter parameter and index it.

**Priority 2 — RTBF / delete-by-user API endpoint (effort: 1-2 days)**
Add `DELETE /api/v1/users/{user_id}` that cascades to all memories, graphs, and episodic records for that user. This is a privacy-first feature that aligns with Tim's stated values and makes Recall compliant for household multi-user use (e.g., family members can request their data be removed).

**Priority 3 — Dashboard user switcher (effort: 3-5 days)**
Add a user selector to the Recall dashboard that scopes all views (search, timeline, memory list) to the selected user_id. This makes the household use case practical without any API changes.

**Priority 4 — Do NOT build org-level scoping yet**
mem0's `org_id` tier is a SaaS feature — irrelevant for Recall's homelab target and would add complexity with zero payoff for the current user persona. Confirm this is out of scope in project-brief.md.
