### Letta Recent Investments (v0.13.0–v0.16.6, Oct 2025–Mar 2026)

Source: github.com/letta-ai/letta/releases (scraped 2026-03-15), letta.com/blog

**Blog/feature announcements:**

- **Sleep-time Compute (Apr 21, 2025):** Background agents that re-write memory state while primary agent is idle. Research paper at arXiv:2504.13171. Reduces test-time compute ~5x on reasoning tasks. Ships in production — configurable frequency. Source: letta.com/blog/sleep-time-compute
- **Skill Learning (Dec 2025):** Agents turn mistakes and feedback into reusable markdown "skill files" (cheatsheets/handbooks). Skills shared across agents in same workspace. Completely OSS. Source: Charles Packer LinkedIn (Dec 2, 2025)
- **Letta Code (Dec 16, 2025):** Memory-first coding agent. #1 model-agnostic OSS agent on TerminalBench. Uses sleep-time compute + skill learning for continual improvement. Source: letta.com/blog/letta-code
- **Conversations API (Jan 21, 2026):** Single agent can have hundreds of concurrent conversations; each forms cross-conversation memories. Enables Slack-integration institutional memory pattern. Source: letta.com/blog/conversations

**Recent release changelog (v0.13.0–v0.16.6):**
- v0.16.6 (Mar 4, 2026): Conversations API expanded (conversation_id="default"); core memory block char limit 20k→100k; context window default 32k→128k; system message compiled at conversation creation

- Implies ongoing investment in memory-block management, skill learning, sleep-time compute as the unified product strategy

**No passive capture hooks in Letta.** Memory is agent-controlled: agents call memory tools explicitly. Sleep-time compute requires explicit agent configuration. No background hook-based observation of file edits, tool calls, or IDE events.

### Industry-Wide Investment Signals (2025-2026)

From broader search results (2026-03-15):

1. **Passive capture is becoming a third-party ecosystem pattern, not a first-party feature.** Multiple OpenClaw community plugins (openclaw-mem0, openclaw-memos-lifecycle-plugin, tensakulabs/openclaw-mem0) implement auto-capture/auto-recall hooks themselves. mem0 is building this officially for OpenClaw. This is exactly Recall's hook-based value proposition applied to other agents.

2. **Memory decay/lifecycle is a recognized unsolved problem.** A DEV.to article (Mar 15, 2026) describes building Ebbinghaus forgetting-curve decay as an MCP memory server, benchmarking 34% vs 18% LoCoMo Recall@5 over mem0. This community demand validates Recall's lifecycle focus — but also shows competitors are on notice.

3. **Temporal search is becoming table stakes.** mem0 shipped temporal search filtering (Feb 28, 2026). Graphiti's dual-timestamp model has always done this. A memory system without time-aware retrieval is increasingly seen as incomplete.

4. **MCP is the integration battleground.** mem0 (OpenMemory), Graphiti (mcp-v1.0.2), Letta (Letta Code MCP) all now ship MCP servers. MCP is where capture and injection hooks live in coding agent workflows.

5. **Graph memory + vector search convergence.** Graphiti's RFC (Feb 2026) for pluggable vector store overlay signals graph-only competitors moving toward hybrid architecture. mem0 ships both vector + graph. Zep is consolidating on cloud-only.

**The second sharpest signal: Letta's sleep-time compute is a legitimate architectural innovation Recall has no equivalent to.** Background agents that re-write memory state during idle time is a new capability class — proactive memory improvement vs reactive storage.

## Mitigation Recommendation

**Priority 1 — Defend the passive capture moat before it closes (3-6 month window):**
mem0 is building OpenClaw plugins with auto-capture. Within 6 months this pattern will exist for all major agent frameworks. Recall must ship a public capture hook specification: a documented protocol for how any agent framework can integrate Recall's observation pipeline. The Claude Code hook integration is a proof-of-concept; turning it into a general spec (hook-based API, not Claude-specific) is the defensive move. Effort: Medium (2-4 weeks to spec + implement generic hook endpoint).

**Priority 2 — Ship temporal search before it becomes a disqualifier (Critical):**
Both mem0 and Graphiti have time-aware retrieval. Recall has no temporal query capability. Users increasingly expect "what did I do last week?" to work. Add time-range filters to recall_search: `since`, `before`, `date_range` parameters. Effort: Low-Medium (1-2 weeks, metadata already stored with timestamps).

**Priority 3 — Implement automatic decay scoring to seize an unclaimed differentiator:**
No major competitor has Ebbinghaus/access-frequency decay. The community is building this as third-party MCP servers (YourMemory, DEV.to Mar 15 2026). Recall should own this. A background job that re-scores memories based on access frequency + recency + importance would be a genuine competitive differentiator. Effort: Medium (2-3 weeks for scoring; 1 week for background job).

**Priority 4 — Investigate sleep-time compute analogue (Medium priority, 6-month horizon):**
Letta's sleep-time compute (background agents that re-write memory) is an architectural capability Recall has no equivalent of. A lightweight version: a background consolidation pass that runs after sessions end, merging near-duplicate memories and promoting frequently-accessed ones. This is the intersection of lifecycle management and background compute. Effort: High (6-8 weeks for robust implementation).

**Priority 5 — Hybrid retrieval is table stakes, ship it:**
mem0 shipped hybrid search Jan 2026. Graphiti RFC is in progress. Per Q5.1 finding this is already on Recall's radar. Ship BM25 + vector fusion before Q3 2026 or Recall becomes the retrieval-quality laggard. Effort: Medium (per Q5.1 analysis).
