# Finding: Q2.3 — Semantic Deduplication Best Practices

**Question**: What deduplication strategies do memory systems use? Is Recall's dual approach (content hash + cosine similarity threshold) standard? Are there better approaches — entity-centric dedup, clustering-based merge, LLM-judged duplicates? What threshold values do production systems use?
**Verdict**: HEALTHY
**Severity**: Low
**Researched**: 2026-03-15 via mcp__exa__web_search_exa, mcp__firecrawl-mcp__firecrawl_scrape (mem0 docs, NVIDIA NeMo docs, GitHub issues)

## Evidence

### Recall's Current Approach (from project-brief.md)

Recall uses a two-stage dedup pipeline at ingest time (observer.py):
1. **Content hash** — exact duplicate detection before vector search
2. **Cosine similarity threshold** — near-duplicate detection against existing Qdrant memories
3. **Hourly consolidation pass** (consolidation.py) — background merge of near-duplicates that slipped through

Threshold value is not documented in project-brief.md. This is a configuration gap (see mitigation).

### mem0 Deduplication Strategy

mem0 uses an LLM-judged approach, not a pure cosine threshold approach.

**Write-time "Update Phase"** (primary dedup mechanism):
- On each `memory.add()` call, mem0 retrieves the top-K most similar existing memories via vector search
- An LLM then classifies each (new_memory, existing_memory) pair as one of: `ADD`, `UPDATE`, `DELETE`, `NOOP`
- `NOOP` is the dedup signal — if the new content is semantically equivalent to an existing memory, it is discarded without storage
- Source: Zylos Research (2026-01-05), docs.mem0.ai/llms.txt

**Known dedup bugs in mem0** (confirmed via GitHub issues):
- Issue #1674 (Aug 2024, closed Oct 2025): "adding same memory twice deletes old memory" — the LLM's Update Phase incorrectly issued `DELETE` instead of `NOOP` for near-identical content. Root cause: when the new content scored 0.9999 similarity, the LLM reasoned the old memory was "superseded" rather than "duplicate."
- Issue #2165 (Jan 2025, closed Jan 2026): "Vector store search result should remove same records, or the memory will be deleted" — duplicate IDs returned from vector search caused the Update Phase to DELETE memories that should have been NOOPed. Fixed in PR #2201 (Feb 2025): deduplicate IDs in `retrieved_old_memory` before LLM pass.
- PR #1676 (Aug 2024, closed Jan 2025): Community PR to add configurable threshold for search — the PR added `threshold` as a configurable parameter for the vector search step that feeds the LLM Update Phase. Not merged into main; suggests mem0's threshold is hardcoded internally.
- Issue #4294 (Mar 2026, still open): Redis vector store returns threshold scores as strings, causing TypeError in threshold-based search.

**mem0 Graph-Level Dedup** (separate mechanism):
- When graph memory is enabled, entity node matching uses embedding similarity with configurable `threshold` parameter
- Default threshold: **0.7** (range 0.0–1.0)
- Threshold guidelines per docs.mem0.ai: UUIDs/IDs → 0.95–0.99, structured data → 0.85–0.9, general purpose → 0.7–0.8, natural language → 0.6–0.7
- Source: docs.mem0.ai/platform/features/graph-threshold (scraped 2026-03-15)

### Production RAG / Memory System Dedup Patterns

**SemDeDup (NVIDIA NeMo Curator / Abbas et al. 2023)**:
- Academic origin: clustering-based semantic dedup for training data at web scale
- Pipeline: embed → k-means cluster → pairwise cosine similarity within cluster → remove pairs above `eps_to_extract` threshold → keep one representative per cluster
- Typical threshold: `eps_to_extract = 0.01–0.10` (very low = strict; 0.01 means only near-identical retained)
- Production use: NVIDIA NeMo Curator; proven to remove 20–50% of web-scale data with minimal performance loss
- Applies to: bulk dataset dedup, NOT real-time memory system dedup
- Source: docs.nvidia.com/nemo-framework/user-guide/25.07/datacuration/semdedup.html (scraped 2026-03-15)

**Production RAG Dedup (Bryson 2025)**:
- Standard recommendation: MinHash/SimHash n-gram shingling for near-duplicate detection (threshold 0.8 Jaccard)
- Persist `duplicate_cluster_id` to suppress at query time
- MMR (Maximal Marginal Relevance) at retrieval time as a complementary dedup pass
- No LLM-judged dedup recommended for production pipelines — cost and latency concerns
- Source: mitchellbryson.com/articles/ai-rag-data-quality-at-scale (May 2025)

**R2R (SciPhi)**:
- Explicitly supports entity deduplication in knowledge graphs (document-level; graph-level planned)
- Uses entity embedding similarity + merging strategy
- Source: r2r-docs.sciphi.ai/documentation/deduplication (URL returned 404 on direct scrape; content surfaced via Exa)

**Zep / Graphiti**:
- No explicit dedup layer documented. Temporal invalidation (valid_at/invalid_at edges) is their approach to memory refresh — superseded facts get new edges rather than dedup
- Entity disambiguation via fuzzy name matching (e.g., "Bob" = "Robert") documented in Graphiti

**Letta**:
- No automatic dedup pipeline. Agent explicitly controls what goes into archival vs. core memory
- Duplicate prevention is agent-responsibility, not system-responsibility

### Academic Survey (2025–2026)

- **EVOLVE-MEM (Delhi Tech, 2025)**: "empirically tuned similarity thresholds" for clustering-based dedup; uses adaptive k-means across memory tiers — no specific threshold reported
- **Synapse (UGA, Jan 2026)**: Spreading activation model avoids dedup by treating near-duplicates as reinforcing nodes rather than removing them — a different architectural philosophy
- **MemR3 (Dec 2025)**: Focuses on retrieval loop quality rather than dedup; no dedup mechanism described

## Recall vs Competitor Comparison

| Dimension | Recall | mem0 | Zep | Letta | Gap Direction |
|-----------|--------|------|-----|-------|---------------|
| Exact dedup (content hash) | Yes (ingest) | Implicit via NOOP | No | No | Recall ahead |
| Near-dedup at ingest | Cosine threshold (value unknown) | LLM-judged NOOP | No | No | Recall on par / ahead |
| Background consolidation pass | Yes (hourly, consolidation.py) | No | No | No | Recall ahead |
| Graph entity dedup | Unknown | Yes (threshold=0.7, configurable) | Via Graphiti name matching | No | Recall behind on graph |

| Threshold bugs in production | Unknown | Multiple documented (issues #1674, #2165, #4294) | N/A | N/A | mem0 more fragile |

### Key observation

Recall's dual-pass approach (content hash + cosine threshold at ingest, plus background consolidation) is architecturally more complete than any single competitor. mem0's LLM-judged NOOP is semantically smarter but has documented reliability issues (false DELETEs) and adds LLM call cost per write. Recall's consolidation.py background pass is unique — no competitor has an equivalent.

The primary gap is:
1. Recall's cosine threshold value is not publicly documented — it may be poorly tuned or absent
2. Recall has no graph-level entity dedup (mem0 has configurable 0.7 default with published guidelines)
3. Recall has no LLM-judged dedup (mem0 does; it catches semantic equivalence that cosine similarity misses for paraphrased content)

## Mitigation Recommendation

**Priority: Low** — Recall's dedup pipeline is structurally sound and more comprehensive than competitors. The gaps are refinements, not architectural failures.

**Specific improvements by effort:**

1. **Document and expose the cosine threshold** (1 hour): Add `DEDUP_COSINE_THRESHOLD` to the Recall config with a documented default and the rationale for the value. mem0 publishes threshold guidelines by use case (0.6–0.7 natural language, 0.85–0.9 structured); Recall should do the same. This also enables per-domain threshold tuning.

2. **Graph entity dedup** (1–2 days): When storing memories to Neo4j, apply the same cosine threshold against existing node embeddings before creating a new node. mem0 uses threshold=0.7 as the default; Recall can adopt the same value and expose it as a configurable. This would eliminate duplicate entity nodes accumulating in the graph.

3. **LLM-judged dedup as optional enhancement** (1 week): Add an optional post-ingest LLM pass (using qwen3:14b, already deployed) that compares a new memory against the top-3 most similar existing memories and issues NOOP/CONSOLIDATE decisions. This is most valuable for paraphrased duplicates that cosine similarity misses (e.g., "Tim uses PostgreSQL" vs "Tim's DB stack is Postgres"). mem0's production experience shows this adds value but introduces correctness risks — gate it behind a config flag. The background consolidation.py pass partially covers this already.

4. **MMR at retrieval time** (2–3 days): Apply Maximal Marginal Relevance to the Qdrant top-K results before returning to the caller. This suppresses near-duplicate memories in a single retrieval response even if they exist in storage, improving retrieval quality without requiring aggressive dedup at write time. Qdrant supports this natively.

## Suggested Follow-ups

None required for this severity level. Optional follow-ups if dedup becomes a user-reported pain point:

- Audit the actual `consolidation.py` merge logic: does it correctly handle the case where two memories should be MERGED (content combined) vs. one deleted? mem0's documented bug (#1674) suggests LLM-judged merge is error-prone — validate Recall's approach is not subject to the same false-DELETE failure mode.
- Measure dedup effectiveness empirically: after 30 days of real usage, query Qdrant for the distribution of cosine similarity scores among stored memories. A bimodal distribution (cluster near 1.0 and cluster near 0.3) suggests the threshold is working. A long tail near 0.9 suggests near-duplicates are slipping through.
