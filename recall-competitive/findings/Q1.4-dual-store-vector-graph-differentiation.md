# Finding: Q1.4 — Dual-Store Vector+Graph Architecture Differentiation

**Question**: Survey the competitive landscape for AI memory systems that use both a vector store AND a knowledge graph. Who does this? How do their graph schemas compare to Recall's Neo4j model (RELATED_TO, causal edges like CAUSED_BY/SUPPORTS/CONTRADICTS/SUPERSEDES, domain partitioning)? Is the dual-store approach a genuine differentiator or becoming table stakes?

**Verdict**: WARNING
**Severity**: Medium
**Researched**: 2026-03-15 via mcp__exa__web_search_exa, mcp__firecrawl-mcp__firecrawl_scrape, mcp__recall__recall_search

### mem0 Graph Schema Analysis

**Source**: docs.mem0.ai/open-source/features/graph-memory (scraped 2026-03-15)

- Graph backend is **optional** — requires `pip install mem0ai[graph]` and a separate Neo4j/Memgraph/Neptune/Kuzu instance
- Relationship types are **LLM-inferred at write time**, not predefined typed edges
- The docs demonstrate: "Alice met Bob at GraphConf 2025" → `(Alice)-[?]->(Bob)` with relationship label derived from conversation
- No documented CAUSED_BY, SUPPORTS, CONTRADICTS, SUPERSEDES, or any causal edge taxonomy
- Graph returns results in a `relations` array alongside vector hits — **graph does not reorder vector results**
- Supports confidence threshold filtering (`threshold` config key) to suppress noisy edges
- Custom prompts can guide which relationships become edges, but schema is freeform
- Multiple graph backends supported: Neo4j, Memgraph, Amazon Neptune, Amazon Neptune DB, Kuzu
- Known production issues: GitHub issues #2070 (2024-12), #1906 (2024-09), #2434 (2025-03) — multiple users report graph edge generation failures with Ollama models and self-hosted Neo4j

**Key difference from Recall**: mem0 graph uses **opaque, LLM-generated relationship labels** (e.g., "met at", "loves", "works for"). No causal taxonomy. No domain partitioning documented.

### Zep / Graphiti Graph Schema Analysis

**Source**: arxiv.org/html/2501.13956v1, help.getzep.com/graphiti, github.com/getzep/graphiti (23.7K stars, v0.28.2, 2026-03-11)

Graphiti's graph model is significantly more structured than mem0's:

- **Three node types**: `EpisodicNode` (raw source data), `EntityNode` (extracted entities), `CommunityNode` (entity clusters)
- **Two primary edge types** (confirmed via GitHub issue #380, Apr 2025):
  - `EntityEdge` — relationships between entities; edge type is a **free-form `relation_type` string** in SCREAMING_SNAKE_CASE (e.g., `WORKS_AT`, `LIVES_IN`, `IS_FRIENDS_WITH`) derived by LLM from relationship predicate
  - Static structural edges linking episodic→entity, entity→community
- **Temporal metadata on every edge**: `valid_at`, `invalid_at` — enables point-in-time queries and automatic fact invalidation when contradictions are detected
- **Contradiction handling**: When a new fact contradicts an existing edge, Zep marks the old edge `invalid_at` timestamp rather than deleting it — **temporal invalidation, not typed CONTRADICTS edges**
- Edge `relation_type` is free-form but static in terms of graph DB indices — Graphiti maintainer confirmed in issue #380 (Apr 2025): "edge types are going to be static (we need those static values in order to properly build indices for performant queries)" — custom edge types not yet supported as of that date
- **Custom Entity Types**: supported as of 2025 (Pydantic BaseModel subclasses), but custom edge types were not yet supported as of Apr 2025
- Hybrid search: BM25 + semantic + graph traversal; sub-second latency
- As of Mar 2026: 193 releases, active development (last push 2026-03-13), 307 open issues

**Key difference from Recall**: Zep excels at **temporal fact lifecycle** (bi-temporal tracking, invalidation). It does not have explicit causal relationship types (CAUSED_BY, SUPPORTS, CONTRADICTS as named typed edges). Contradiction is handled temporally (invalidation), not as a typed edge.

### Cognee Graph Schema Analysis

**Source**: cognee.ai/blog/fundamentals/vectors-and-graphs-in-practice (Sep 2025), lancedb.com/blog/case-study-cognee (Sep 2025)

- Explicit dual-store architecture: vector + graph run **side-by-side by design** (not optional)
- Vector backends: LanceDB, Qdrant, pgvector
- Graph backends: Neo4j, Kuzu, Memgraph
- Uses GraphRAG-style pipeline: vectors for semantic candidates, graph for multi-hop relationship traversal and provenance
- Relationship schema is **domain-specific and LLM-extracted** from document content
- "Memify Pipeline" (Oct 2025): post-processing enrichment that evolves the graph incrementally
- $7.5M seed round (Pebblebed); ~12K GitHub stars (training data — verify exact count)
- No documented causal edge taxonomy; schema is inferred from document structure

---

### MAGMA Research Architecture (Academic, Jan 2026)

**Source**: arxiv.org/html/2601.03236v1

- Proposes **four orthogonal graphs** for a single memory store: semantic, temporal, causal, and entity graphs
- Causal graph explicitly tracks CAUSED_BY-style relationships
- Retrieval is policy-guided traversal across graphs based on query type
- Outperforms Zep/mem0 on LoCoMo and LongMemEval benchmarks
- **No production release** — academic paper only (University of Texas at Dallas + University of Florida)
- Directly validates Recall's architectural instinct: causal edges are valuable and orthogonal to pure semantic search

---

### Microsoft GraphRAG (Not directly comparable)

**Source**: microsoft.com/en-us/research/project/graphrag, arxiv.org/pdf/2404.16130v2

- Extracts entity knowledge graph from **static document corpora** in batch
- Uses community detection (Leiden algorithm) to generate hierarchical summaries
- Graph is used for global/local search distinctions, not agent memory lifecycle
- Does NOT handle: incremental updates, temporal fact tracking, conversation memory, or agent memory at all
- Not a memory system — is a document Q&A system
- Not a dual-store in the agent memory sense — vector search is secondary

## Is Dual-Store Still a Differentiator?

**Short answer**: Dual-store (vector + graph) was a differentiator in 2023. By early 2026 it is becoming table stakes for serious AI memory systems, but **schema expressiveness and causal edge taxonomy remain genuinely rare**.

Evidence:
1. The vectorize.io 2026 comparison lists 5+ systems with vector+graph — mem0, Zep, Cognee, Letta, Hindsight
2. Graphiti has 23.7K stars and 193 releases — strong adoption signal for graph-based memory
3. However: **none of the surveyed systems use explicit causal typed edges** (CAUSED_BY, SUPPORTS, CONTRADICTS, SUPERSEDES) as first-class schema elements
4. Zep achieves contradiction handling via **temporal invalidation** — a different model (fact becomes invalid vs. fact is contradicted)
5. MAGMA paper (Jan 2026) from academic researchers explicitly names causal graphs as a separate orthogonal dimension that current production systems lack — validates Recall's direction
6. The mem0 graph is the weakest — optional, freeform, production-buggy with Ollama/self-hosted setups

**Conclusion**: Recall's causal edge schema (CAUSED_BY, SUPPORTS, CONTRADICTS, SUPERSEDES) is a genuine and currently unique differentiator. The dual-store foundation is no longer differentiating on its own. Temporal edge tracking (what Zep does well) is a gap Recall should address.

## Mitigation Recommendation

1. **Add temporal edge tracking to Recall's Neo4j schema** (High priority, Medium effort — 1-2 weeks): Zep's bi-temporal model (`valid_at`, `invalid_at` on every edge) enables point-in-time queries and fact invalidation without deleting history. Recall currently has no equivalent. Add these fields to all edge types. This closes the primary gap Zep has over Recall in the graph dimension.

2. **Document the causal edge taxonomy publicly** (Low effort — 1 day): Recall's CAUSED_BY/SUPPORTS/CONTRADICTS/SUPERSEDES schema is a genuine differentiator but is not surfaced in any public documentation. Write a blog post or README section explaining the schema and its advantages over freeform LLM extraction. This converts a hidden strength into a marketing asset.

3. **Consider adding custom entity types via Pydantic** (Medium priority, Medium effort): Zep/Graphiti supports domain-specific entity types. Recall could add a similar extension mechanism. This would allow users to define `Project`, `Decision`, `Person` etc. as first-class graph nodes.

4. **Add graph backend flexibility** (Low priority, High effort): Supporting Kuzu (embedded, no infra) or Memgraph alongside Neo4j would reduce the ops burden of the graph stack. Kuzu in particular enables zero-infra deployment comparable to mem0's embedded approach.
