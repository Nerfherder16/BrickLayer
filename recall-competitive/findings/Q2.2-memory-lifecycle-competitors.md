# Finding: Q2.2 — Memory Lifecycle — How Competitors Handle Aging and Pruning

**Question**: What lifecycle management do mem0, Zep, Letta, and other serious memory systems implement? Do any have automatic decay, consolidation, or hygiene pipelines comparable to Recall's? Is Recall's multi-stage lifecycle (decay → consolidation → hygiene → GC → dream consolidation) genuinely differentiated or are competitors catching up?

## Evidence

### mem0 — Lifecycle Management

Source: https://docs.mem0.ai/cookbooks/essentials/memory-expiration-short-and-long-term (scraped 2026-03-15)

mem0's lifecycle management is **manual expiration only**. Key quotes:

- "By default, Mem0 memories persist forever."
- "Set `expiration_date` to make memories temporary."
- "After 7 days, the session context automatically disappears — no cron jobs, no manual cleanup. Mem0 handles expiration transparently."

The mechanism is entirely write-time TTL: the developer passes an ISO timestamp at insertion, mem0 enforces the hard expiry. There is **no automatic decay based on access patterns, importance scoring, or age**. There is **no background consolidation pipeline** — the "consolidation" in mem0's architecture (from the ECAI paper at arxiv.org/html/2504.19413v1) refers to write-time deduplication during ingestion (the "Update Phase"), not an ongoing background process.

From the mem0 research paper (Mem0: Building Production-Ready AI Agents with Scalable Long-Term Memory, ECAI 2025): "We introduce Mem0, a scalable memory-centric architecture that addresses this issue by dynamically extracting, consolidating, and retrieving salient information from ongoing conversations." The "consolidating" here refers to the extraction pipeline deciding whether to ADD, UPDATE, or DELETE a memory at write time — not a background process.

mem0's "Update Phase": For each extracted memory from a conversation, mem0 compares it to existing memories via vector search, then invokes an LLM to decide MEMORY_ACTION (ADD / UPDATE / DELETE / NONE). This is write-time conflict resolution, not lifecycle management. It does not address:
- Memories that become stale without a contradicting update
- Access-frequency-based importance decay
- Near-duplicate clustering over time across sessions
- Cross-domain relationship discovery (dream consolidation equivalent)

**Verdict on mem0 lifecycle**: Feature-poor. Expiration is developer-managed TTL. No automatic decay. No background hygiene. The GitHub issue RFC #17711 on "Memory Consolidation Hooks" in the openclaw project (a different OSS tool, 313K stars) was filed Feb 2026 and closed as "not_planned" — confirming this is a broadly recognized gap that even well-resourced projects are not yet solving.

### Zep / Graphiti — Lifecycle Management

Sources: arxiv.org/abs/2501.13956 (Zep paper, Jan 2025), blog.getzep.com/beyond-static-knowledge-graphs/, github.com/getzep/graphiti issue #1300 (Mar 2026)

Zep's approach to "lifecycle" is architectural, not procedural. Key mechanism: **dual-timestamp model**. Every fact (edge in the knowledge graph) carries:
- `valid_from`: when the fact became true
- `valid_to`: when the fact was superseded (set when a contradicting fact is ingested)

When a new fact contradicts an existing one ("Alice works at Google" → "Alice now works at Meta"), Zep automatically invalidates the old edge (`valid_to = now`) and creates a new edge. This is temporal validity management — the graph retains history while marking what is current.

This is **not decay**. It is deterministic invalidation triggered by new input. Zep does not:
- Decay memories based on access frequency
- Consolidate near-duplicate edges over time without contradicting new input
- Run background hygiene on unused/stale facts
- Have a GC or soft-delete pipeline

**Critical evidence**: GitHub issue getzep/graphiti #1300 (opened 2026-03-05): "Real-time knowledge graphs lack temporal decay algorithms for retention management." The issue was opened by an automated growth-hacking tool scanning the codebase and explicitly flagging the absence of decay as a gap. The repo has 23K stars and the issue was opened this month — confirming that as of March 2026, Graphiti/Zep **does not implement automatic temporal decay**.

Zep paper on LongMemEval: "accuracy improvements of up to 18.5% while simultaneously reducing response latency by 90%." Their advantage is temporal reasoning quality (knowing what fact was valid when), not lifecycle management.

### Letta/MemGPT — Lifecycle Management

Source: https://docs.letta.com/guides/core-concepts/memory/archival-memory/ (scraped 2026-03-15), medium.com/@piyush.jhamb4u/stateful-ai-agents-a-deep-dive-into-letta-memgpt-memory-models-a2ffc01a7ea1

Letta's memory model is agent-controlled, not system-managed. Key characteristics:

- `archival_memory_insert`: Agent decides what to store (explicit tool call)
- `archival_memory_search`: Agent decides what to retrieve (on-demand)
- "Agent-immutable — Agents cannot easily modify or delete archival memories (though developers can via SDK)"
- "Unlimited storage — No practical size limits"

There is **no automatic lifecycle management** in Letta. No decay. No consolidation. No hygiene. The agent must explicitly choose to insert or search. If the agent doesn't decide a memory is no longer relevant, it stays forever. This is fundamentally the MemGPT OS metaphor: the LLM agent manages its own "disk" (archival memory) like an OS manages disk I/O. Eviction/pruning is agent-decision, not system-automatic.

Overflow handling: Letta's context window management (the MemGPT contribution) handles the in-context "RAM" tier — when context fills, the LLM agent is prompted to decide what to move to archival memory. But archival memory itself has no automatic pruning.

### Academic and Broader Survey

Source: arxiv.org/html/2512.23343v1 (AI Meets Brain: Unified Survey on Memory Systems, Dec 2025), arxiv.org/abs/2512.13564 (Memory in the Age of AI Agents, Dec 2025)

The December 2025 survey identifies "Memory Dynamics" as a taxonomy axis including: Formation, Evolution, and Retrieval. The survey notes that "self-optimizing memory management" is an **emerging frontier** — not yet standard practice. This confirms that Recall's automated lifecycle pipeline is ahead of the curve rather than catching up to an established standard.

The RFC for Memory Consolidation Hooks (openclaw/openclaw #17711, Feb 2026) was filed and closed as "not_planned" within weeks — industry confirmation that background memory lifecycle pipelines remain an unsolved and largely unaddressed problem across the field.

## Recall vs Competitor Comparison

| Lifecycle Feature | Recall | mem0 OSS | mem0 Platform | Zep/Graphiti | Letta |
|-------------------|--------|-----------|---------------|--------------|-------|
| Automatic age-based decay | YES (daily, multi-factor) | NO | NO | NO | NO |
| Access-frequency decay | YES (access count signal) | NO | NO | NO | NO |
| Background consolidation (near-dupes) | YES (hourly) | NO (write-time only) | NO (write-time only) | NO | NO |
| Daily hygiene (soft-delete stale) | YES | NO | NO | NO | NO |
| Hard-delete GC (superseded memories) | YES (daily) | Manual TTL | Manual TTL | Yes (edge invalidation, not GC) | Manual (dev SDK) |
| Cross-domain relationship discovery | YES (dream consolidation, nightly) | NO | NO | NO | NO |
| Write-time conflict resolution | YES | YES (Update Phase LLM) | YES (Update Phase LLM) | YES (temporal edge invalidation) | Agent-controlled |
| TTL/expiration date field | NO (implicit via hygiene) | YES (developer sets) | YES (developer sets) | NO | NO |

**Summary**: Recall's lifecycle pipeline is the most comprehensive in the surveyed field. mem0 offers write-time conflict resolution plus developer-managed TTL but no autonomous background lifecycle. Zep offers temporal invalidation of superseded facts but no decay, no GC, no consolidation. Letta has no automatic lifecycle at all.

The gap is material: Recall runs at minimum 4 distinct automated background processes (decay, consolidation, hygiene, GC) plus dream consolidation. Competitors run 0–1 automated processes, and those processes are simpler (TTL expiry or point-in-time invalidation).

## Mitigation Recommendation

No immediate mitigation needed — this is a HEALTHY finding confirming Recall's advantage. Strategic recommendations:

1. **Document and market the lifecycle pipeline explicitly.** Recall's multi-stage background pipeline (decay → consolidation → hygiene → GC → dream) is genuinely unique and provides compounding quality improvement over time. This should be the leading differentiator in any positioning against mem0 or Zep. Currently this is buried in implementation details.

2. **Add a TTL/expiration_date field** as a quality-of-life feature for developers who want explicit manual control (as mem0 provides). This is not replacing the automatic pipeline but complementing it for use cases like session context that users know should expire.

3. **Monitor Zep/Graphiti for decay feature development.** Issue #1300 confirms the gap; if Graphiti ships a decay algorithm, Recall's window on this differentiator narrows. Watch the getzep/graphiti commit history.

4. **Expose lifecycle metrics in the dashboard.** Show users: memories decayed today, memories consolidated, memories GC'd. This makes the pipeline's value visible and creates trust in the system's long-term quality improvement.
