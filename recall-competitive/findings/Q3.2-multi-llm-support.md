# Finding: Q3.2 — Multi-LLM Support

**Question**: How do mem0, Zep, and Letta handle multi-LLM support? Is LLM-agnosticism a genuine feature or marketing? Does Recall's deep Claude Code hook integration represent a trade-off (depth vs breadth) or a genuine moat? What would it take for Recall to support Cursor, Windsurf, or other AI coding tools?

mem0 supports 15 named LLM provider integrations in its self-hosted open source build:

| Anthropic | Claude family |
| Ollama | Local models |

| LM Studio | Local |

| Sarvam AI | Regional |

**TypeScript limitation**: Only OpenAI, Anthropic, and Groq are supported in the TS SDK. Python SDK has full coverage.

The LiteLLM provider is the LLM-agnosticism workhorse — it accepts any OpenAI-compatible API endpoint, covering Ollama, vLLM, LM Studio, OpenRouter, and essentially every provider with an API. This is genuine breadth, not marketing.

**Assessment**: mem0's multi-LLM support is real and production-grade. Provider switching requires one config dict change. Community PRs actively maintain new providers (PR #1507 for LiteLLM/TogetherAI, 2024; API key config PR #1872, 2024).

### Zep: Cloud LLM Required (No Full Local Path)

Zep/Graphiti's extraction pipeline requires a cloud-capable LLM for temporal knowledge graph construction — entity extraction, edge classification, contradiction detection. The system defaults to OpenAI and supports Anthropic, Groq, and Google via Graphiti configuration. There is no documented Ollama integration for the core extraction pipeline. Community Edition was archived April 2025. All current Zep usage goes through the cloud API.

**Assessment**: Zep is LLM-agnostic at configuration level (OpenAI-compatible APIs accepted) but requires a capable model for graph construction. No fully local path exists.

### Letta: 10+ Providers via BYOK

From docs.letta.com/letta-code/providers/ (scraped via Exa, 2026-03-15):

Letta Cloud supports: OpenAI, Anthropic, Google Gemini, Together AI, Azure OpenAI, Claude on Amazon Bedrock, OpenRouter, Minimax, xAI, zAI. Also supports ChatGPT Pro/Plus plan OAuth ("Codex") and direct coding plan subscriptions. Self-hosted Letta accepts any OpenAI-compatible endpoint including Ollama.

Model handles use `provider/model-name` format: `openai/gpt-4o`, `anthropic/claude-sonnet-4-5-20250929`, `google/gemini-2.0-flash`, etc. Provider switching requires changing a string parameter.

**Assessment**: Letta's multi-LLM is genuine and practical. The BYOK (bring your own key) model lets users pick providers independently.

### Recall: Ollama-Only for Extraction; Claude Code-Only for Hooks

Recall's memory extraction pipeline is hardcoded to Ollama (qwen3:14b default). The hooks that trigger automatic memory storage (`UserPromptSubmit`, `PostToolUse`, `Stop`) are Claude Code lifecycle events — they are not implemented in Cursor, Windsurf, Copilot, or any other IDE/coding assistant.

**However — critical nuance — the MCP layer is fully tool-agnostic:**

From Windsurf docs (docs.windsurf.com/windsurf/cascade/mcp, scraped 2026-03-15):
> "Cascade now natively integrates with MCP... supports stdio, Streamable HTTP, and SSE transports."

From Cursor MCP guide (truefoundry.com/blog/mcp-servers-in-cursor-setup-configuration-and-security-guide, 2026-03-03):
> "They work identically to MCP servers in Claude Code and Claude Desktop since they all use the same underlying protocol."

Recall's MCP server (`recall_search`, `recall_store`, etc.) can be connected to Cursor or Windsurf today by adding it to `.cursor/mcp.json` or `~/.codeium/windsurf/mcp_config.json`. The tools will be callable from those IDEs' chat interfaces.

**What would NOT work in Cursor/Windsurf:**
- `UserPromptSubmit` hook (fires before every user message in Claude Code only)
- `PostToolUse` hook (fires after Claude Code tool execution)
- `Stop` hook (session summary trigger)

**What WOULD work:**
- Manual MCP tool calls (`recall_search`, `recall_store`) in Cursor/Windsurf chat
- Any integration that those IDEs' own hook/extension systems can invoke

### Competitor-Equivalent Hook Systems

Claude-Mem (github.com/thedotmack/claude-mem) documented its architecture on docs.claude-mem.ai/architecture/hooks (scraped via Exa, 2026-03-15). The 5-stage hook system (SessionStart, UserPromptSubmit, PostToolUse, Stop, SessionEnd) is explicitly described as Claude Code-specific. The docs note: "This two-process architecture works in both Claude Code and VS Code" — VS Code support is achieved via the extension API, which maps to equivalent VS Code extension lifecycle events.

**This means the hook architecture is portable to VS Code extensions** — including the Cursor and Windsurf VS Code extensions — if someone implements matching extension event handlers. The protocol doesn't have to be Claude Code hooks specifically; it can be adapted to any IDE with an extension API.

### Market Signal: Demand for Cross-IDE Memory is Real

Multiple community threads confirm real demand:
- r/mcp: "How to build a shared MCP memory service for AI IDEs (Copilot, Cursor, Windsurf, Claude…)?" — seeking unified MCP memory across IDEs (Oct 2025)
- Cursor Community Forum: "Recallium" (an unrelated self-hosted memory system) posted Dec 2025: "Works with Cursor, Claude Desktop/code, VS Code and any MCP-compatible tool. Store context once, access everywhere."
- r/windsurf: "Cursor to WindSurf MCP & Memory Bank" — user explicitly migrating MCP config from Cursor to Windsurf (May 2025)

**Competitive interpretation**: There is an emerging market for IDE-agnostic memory systems. Any memory product that locks to Claude Code hooks alone will miss users of Cursor (~4M users as of 2025 per public claims), Windsurf, and Copilot.

## The Trade-Off Analysis

**Recall's moat is real but narrow:**

Recall's automatic observation via Claude Code hooks (UserPromptSubmit injection + PostToolUse extraction) is genuinely unique — no competitor implements passive automatic observation at the IDE lifecycle level. mem0, Zep, and Letta all require explicit `.add()` calls from application code.

**But the moat is bounded to Claude Code users.** That is currently a small-but-growing population. If Recall's value proposition is "memory that captures everything automatically without developer effort," that proposition only delivers for Claude Code users today.

**The MCP bridge is the unlock:** Recall's MCP server means the storage and retrieval layer already works in Cursor, Windsurf, and any MCP client. The missing piece is automatic capture — which requires implementing equivalent hooks for each IDE's extension system.

## Mitigation Recommendation

**Priority 1 — Cursor/Windsurf MCP documentation (effort: 1 day)**
Document how to add Recall's MCP server to `.cursor/mcp.json` and Windsurf's `mcp_config.json`. This unlocks manual use of `recall_search` and `recall_store` in two additional major IDEs immediately, with zero code changes.

**Priority 2 — VS Code extension hook adapter (effort: 2-3 weeks)**
Implement a VS Code extension that mirrors Recall's Claude Code hooks using VS Code extension API events: `onDidOpenTextDocument`, `onDidSaveTextDocument`, chat participant API, etc. Since Cursor and Windsurf are VS Code forks, this would give Recall automatic observation in those IDEs. See claude-mem's documented pattern as reference architecture.

**Priority 3 — LiteLLM extraction provider (effort: 1-2 weeks)**
Add a configurable `extraction_provider` setting that accepts any LiteLLM-compatible endpoint instead of hardcoding Ollama. This makes Recall's extraction pipeline work with any cloud key for users who don't self-host Ollama. Config example: `extraction_provider: litellm`, `extraction_model: anthropic/claude-haiku-3-5`.

**Priority 4 — Minimal local LLM toggle (effort: 3 days)**
Allow users to switch the extraction model to any Ollama-installed model via a single config line. Currently requires code changes.
