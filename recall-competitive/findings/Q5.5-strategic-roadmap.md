## Positioning Strategy

### Who Recall Should Target
**Primary**: Privacy-first developers building on Claude Code. Self-hosters who cannot use cloud memory (legal, healthcare, finance, enterprise with data restrictions). The 44% of enterprises blocked on AI adoption by privacy concerns.

**Secondary**: Any Claude Code power user who uses Cursor or Windsurf and wants cross-tool memory continuity.

**Positioning statement** (once Phase 1-2 complete):
> "The only AI coding tool memory system where zero data leaves your machine — fully automatic, with a memory pipeline that gets smarter over time."

### What Recall Should NOT Try to Be
- **A general-purpose LLM memory library** — that's mem0's space with $24M and 50K stars. Competing head-on is wrong.
- **A multi-tenant enterprise SaaS** — that's Zep's space. Recall's strength is homelab-first.
- **An agent memory framework** — that's Letta's space. Recall's extraction-not-agent-paging approach is a feature for reliability, not a framework choice.

## The Single Most Important Action

**If only one thing can be done**: Publish `recall-sdk` to PyPI with a 10-line quickstart.

The SDK is the trust signal that: (a) the project is maintained, (b) it can be used programmatically, (c) it has a documented API. Without it, every developer who finds Recall via web search will bounce at the README. The docs site is the second action. Together, they take the project from "Tim's private homelab tool" to "serious self-hosted memory system."

The lifecycle pipeline, graph depth, and hook integration that make Recall technically superior are irrelevant if developers never get past the README.
