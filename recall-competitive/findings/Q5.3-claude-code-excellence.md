## Evidence

### What Claude Code Users Actually Need (from Q4.2 research)

From Cursor forum research (multiple high-upvote threads, Jan 2025 – Feb 2026):

The core user complaint is: "architectural decisions don't always carry forward, and I end up re-explaining or correcting things that were already agreed on."

This is NOT the same as "I need a general-purpose LLM memory." It's specifically:
1. **Architectural decision persistence** — "we use JWT RS256, not sessions" stays true across sessions
2. **Convention enforcement** — "always use snake_case for Python, camelCase for TS" doesn't require re-stating
3. **Project context restoration** — "this service is responsible for X, not Y" remembered automatically
4. **Error pattern avoidance** — "don't try to use X library, we discovered it doesn't support Y"
5. **Progress continuity** — "last session we got to step 7 of 12; continue from there"

None of these are well-served by general-purpose memory systems (mem0, Zep). Those systems optimize for: personal preferences ("user likes Python"), biographical facts ("user lives in Seattle"), and conversational recall ("user asked about auth last Tuesday").

### Where Recall Already Leads for Claude Code

1. **Hook-based passive capture** — zero friction. observe-edit.js captures facts from file edits without any user action. No competitor has this architecture. mem0's hooks require user action or explicit API calls.

2. **Session summary** — recall-session-summary.js runs on Stop hook. No competitor captures session-end state automatically.

3. **UserPromptSubmit retrieval** — memories are injected before every prompt. Context is pre-loaded, not fetched on demand. This is architecturally superior for interactive coding sessions.

4. **Domain partitioning** — memories are tagged by domain (coding, preferences, infrastructure, etc.). Claude Code context can be pre-filtered to relevant domains.

5. **Causal edge graph** — CAUSED_BY, SUPPORTS, CONTRADICTS edges. When a new memory contradicts an existing one ("we switched from PostgreSQL to SQLite for local dev"), the contradiction is captured and surfaced. No competitor can do this.

### Gaps Specific to Claude Code Users (not general memory gaps)

**Gap 1: Architecture Decision Memory Type** (no competitor has this)

Currently Recall stores all memories uniformly. Claude Code users specifically need `memory_type: "decision"` with:
- Higher base importance (doesn't decay fast)
- Permanent durability by default
- SUPERSEDES edge when a decision is reversed
- Tag auto-detection: commit messages or file changes containing "decided", "chose", "rejected", "switched from", "convention is" should auto-flag as architecture decisions

**Implementation**: 2-3 days in observe-edit.js hook. Look for decision language in diffs and commit messages → set `durability: permanent, tags: ["arch-decision"]`.

**Gap 2: Write Confirmation UX** (from Q4.3 — top abandonment cause)

No visible feedback after memories are stored. The #1 reason users abandon memory systems: "I stored something but it wasn't there later." For Claude Code, the Stop hook runs recall-session-summary.js but users see no summary.

**Implementation**: End-of-session output: "Recall stored 3 memories this session: [brief descriptions]". 1 day of work.

**Gap 3: Cross-Tool MCP Documentation** (from Q4.2)

Recall's MCP server is tool-agnostic (standard MCP protocol). It can serve Claude Code, Cursor, AND Windsurf with zero code changes. Users switching from Cursor to Windsurf would keep memory continuity automatically. But this is not documented.

**Implementation**: Write documentation for `cursor/.cursor/mcp.json` and Windsurf equivalent. 1-2 days. Zero code change.

**Gap 4: Session Context Restoration Summary** (feature gap)

When a new Claude Code session starts, the retrieve hook fires. But users don't see what memories were retrieved or what context was pre-loaded. An opt-in "Recall loaded N memories for this session" message would build trust.

**Implementation**: UserPromptSubmit hook can output a brief pre-amble. 1 day.

**Gap 5: Stale Memory Detection** (from Q4.2, Q4.3)

A memory from 6 months ago saying "use Python 3.9" is now wrong. No tool surfaces stale memories proactively. Recall could add: "Warning: 3 memories are >90 days old and haven't been accessed recently — review for staleness."

**Implementation**: Query Qdrant for memories where `last_accessed_at < 90 days ago AND access_count < 3`. Dashboard widget. 2-3 days.

### What mem0/Zep Cannot Easily Replicate

1. **Claude Code hook depth** — mem0's API is call-from-code; hooks are passive. Building a Claude Code hook that observes PostToolUse (every file edit) would require mem0 to create a hook framework from scratch. They have no roadmap for this.

2. **Observe-edit fact extraction from diffs** — extracting semantic facts from file diffs (not conversation) is architecturally unique. This is Recall's killer feature for coding use cases. mem0 only extracts from conversation messages.

3. **Full lifecycle during coding sessions** — dream consolidation (cross-domain relationship discovery) runs nightly. This means Recall gets smarter with every coding session over time. mem0's "consolidation" is single-session only.

4. **Local privacy** — for teams working on confidential code. 44% of enterprises cite privacy as top AI adoption barrier. Recall's zero-cloud-call architecture is uniquely suited to this. Documented correctly, this is a strong moat.
