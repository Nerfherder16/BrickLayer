# Finding: Q3.3 — Self-Hosting Complexity Comparison

**Question**: What is the minimum infrastructure to self-host mem0, Zep CE, and Letta? Compare to Recall's 4-service + Ollama requirement.
**Verdict**: WARNING
**Severity**: High
**Researched**: 2026-03-15 via mcp__exa__web_search_exa, mcp__firecrawl-mcp__firecrawl_scrape (dev.to/mem0, docs.letta.com, github.com/getzep/zep, github.com/getzep/graphiti)

---

## Evidence

### mem0 — two self-hosting modes

**Mode 1: Zero-infrastructure (pip-only)**
```bash
pip install mem0ai
```
`Memory()` with no config uses embedded Qdrant (file-based at `/tmp/qdrant`) + SQLite. Zero Docker. Zero infrastructure. Works offline. LLM defaults to OpenAI; swap to Ollama with a config dict.

This path is unavailable to Recall entirely. Recall has no embedded mode.

**Mode 2: Full self-hosted stack (3 Docker services)**

Source: https://dev.to/mem0/self-hosting-mem0-a-complete-docker-deployment-guide-154i (published 2026-02-23)

Services:
1. `mem0` — FastAPI REST server (custom Dockerfile wrapping `mem0/mem0-api-server:latest`)
2. `postgres` — `ankane/pgvector:v0.5.1` — vector store + relational
3. `neo4j` — `neo4j:5.26.4` with APOC plugin — graph store

```yaml
docker compose up -d --build
```

Total: **3 containers**, 1 docker-compose.yaml, 1 Dockerfile, 1 .env. First-run time: 2-5 minutes (image pulls + Neo4j initialization). Neo4j requires 90-second start period.

**Important note**: No auth by default. CORS is `allow_origins=["*"]`. Production requires a reverse proxy (Caddy/Nginx). Passwords are hardcoded in plain text without modification — requires .env file setup.

**Minimum viable path for graph-off use**: Just `postgres` + `mem0` (2 containers). Neo4j only needed for graph memory.

### Zep — community edition (now archived)

The original Zep Community Edition repo was **archived April 2, 2025** and is read-only.

The CE docker-compose.ce.yaml (last version before archival) had:
```yaml
services:
  zep:        # zepai/zep:latest — Go binary
  db:         # ankane/pgvector:v0.5.1 — PostgreSQL
  graphiti:   # zepai/graphiti — FastAPI Python service (Zep's graph engine)
```

The `graphiti` service itself requires Neo4j (either a 4th container or Neo4j cloud/desktop).

**Current self-hosting path (post-CE)**: Install Graphiti directly:
```bash
pip install graphiti-core
```
Requires: Neo4j 5.26+ (separate Docker or Neo4j Desktop), OpenAI API key.

**Verdict on Zep self-hosting**: The CE path is dead. The current path to self-hosted Zep requires understanding Graphiti (a separate project, 8.6k stars on GitHub), Neo4j setup, and manually wiring the Zep server binary. This is **more complex** than Recall's documented compose stack, not less. The official "self-hosting" story is now Zep Cloud.

Source: https://github.com/getzep/zep/blob/main/docker-compose.ce.yaml (archived), https://github.com/getzep/graphiti

### Letta — single container, simplest competitor

**Minimum viable self-host**: 1 Docker container.

```bash
docker run \
  -v ~/.letta/.persist/pgdata:/var/lib/postgresql/data \
  -p 8283:8283 \
  -e OPENAI_API_KEY="sk-..." \
  letta/letta:latest
```

That's it. The Letta Docker image bundles **PostgreSQL with pgvector** inside the container. No separate DB service. No Redis. No graph database.

Source: https://docs.letta.com/guides/docker/

**What's included**: PostgreSQL + pgvector (bundled), vector search for archival memory, REST API on port 8283, optional password protection via `SECURE=true`.

**What's not included**: Graph memory (no Neo4j — Letta uses pgvector for all memory search), no extraction LLM server (uses external API — OpenAI/Anthropic/Ollama URL passed as env var).

**Ollama support**: `OLLAMA_BASE_URL="http://host.docker.internal:11434/v1"` — just set an env var. No separate Ollama container management.

**Self-host complexity rating**: 1/10 — single docker run command, no compose file needed.

### Recall — current stack

Per project-brief and Tim's CLAUDE.md:

Services:
1. `qdrant` — vector store
2. `neo4j` — graph store
3. `postgresql` — relational store
4. `redis` — caching/queue
5. `ollama` — local LLM for extraction (qwen3:14b) + embedding (qwen3-embedding:0.6b)
6. `recall-api` — FastAPI application

**Total: 6 processes / 5+ Docker services** (Ollama may run on separate hardware at 192.168.50.62)

Hardware required: The Ollama RTX 3090 is a separate physical machine. Practical minimum for Tim's setup is two machines. For a new user self-hosting, they would need GPU hardware for Ollama OR an external LLM API key.

## Key Finding: Recall vs mem0 Service Count

Recall's full stack (6 services) is **2x more services** than mem0's full self-hosted stack (3) and **6x more** than Letta's single-container setup. The specific services driving Recall's complexity:

- **Redis**: Adds operational overhead (cache invalidation, connection pooling) not present in any competitor.
- **Qdrant + PostgreSQL separately**: mem0 uses pgvector to consolidate these. Recall uses two separate databases for vectors and relational data.
- **Separate Ollama hardware**: Requires GPU, dedicated machine, or paying for API access. mem0 and Letta accept an API key as a 1-line config change.
- **Neo4j**: Shared with mem0's full-stack path — this is not unique overhead.

---

## Verdict Calibration

This is a **WARNING not FAILURE** because:
1. Recall's 6-service stack is more capable than any single-container alternative (graph + vector + lifecycle pipeline + local LLM)
2. Self-hosting complexity correlates with capability — Letta's single container has no graph memory and no lifecycle management
3. Recall's target user (Tim, homelab operators) already runs multi-service Docker stacks and has Ollama
4. The mem0 embedded mode is the real comparison benchmark — that's where the **FAILURE** framing would apply (no equivalent minimal path exists for Recall)

The severity is High because:
- A new user evaluating Recall sees a 6-service stack before they see any benefit
- mem0's `pip install mem0ai` + 8-line demo will convert developers before they look at Recall

## Score Updates

- `self_hosting_simplicity`: **0.30** — Recall requires 5-6 services vs 0-3 for competitors at comparable capability. No embedded mode. GPU hardware dependency is a hard barrier for non-homelab users. Significant but not catastrophic given the target user profile.

---

## Mitigation Recommendation

**High impact, medium effort (1-2 weeks)**:

1. **Eliminate Redis**: Audit whether Redis is essential or an artifact of early design. If caching can move to PostgreSQL or be removed, drop one service.

2. **Add a "lite mode"**: A `docker-compose.lite.yml` that uses pgvector instead of Qdrant (dropping one service), uses an external API key instead of Ollama (dropping the GPU dependency), and skips Neo4j (dropping graph memory). This gives new users a 2-container first experience (FastAPI + PostgreSQL) at the cost of graph memory and local-only operation.

3. **Document the `OPENAI_API_KEY` path**: If Recall already supports LLM providers beyond Ollama, document it prominently. If not, add it — this removes the GPU hard requirement.

4. **One-liner bootstrap script**: `curl -sL recall.ai/install | bash` that generates the right compose file for the user's hardware. Mem0 has this for OpenMemory MCP (`curl -sL ... | bash`).
