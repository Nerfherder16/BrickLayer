### The Three-Tier Memory Architecture

Letta's "LLM as Operating System" paradigm organizes memory into three tiers:

**Tier 1: Core Memory (in-context, always present)**
- Structured as "memory blocks" — each block has a label, description, value string, and character limit
- Blocks are embedded directly into the system prompt; always in the agent's context window
- Default blocks: `persona` (agent identity) and `human` (user profile)
- Character limit recently updated from 20k to 100k per block (v0.16.6 release notes)
- Agent-editable via tool calls: `core_memory_append`, `core_memory_replace`, `memory_rethink`, `memory_apply_patch`, `memory_create`, `memory_delete`
- Source: letta.com/blog/agent-memory (July 2025), Medium deep-dive Feb 2026

**Tier 2: Recall Memory (conversational history)**
> "Recall memory preserves the complete history of interactions that can be searched and retrieved when needed, even when not in the active context window. In Letta, recall memory saves to disk automatically, while other frameworks require developers to handle persistence manually." — letta.com/blog/agent-memory

- Full message history stored automatically in PostgreSQL
- Searchable but not in-context by default — agent must retrieve explicitly
- This is auto-persisted: every message goes to recall storage without agent action
- Analogous to disk storage in OS terms

**Tier 3: Archival Memory (external knowledge store)**
- Explicitly formulated knowledge pushed by the agent via `archival_memory_insert()`
- Stored in vector DB (pgvector) or optionally a graph database
- Agent retrieves via semantic search tool call when needed
- Not automatically populated — agent must decide what to archive

**Message Buffer (working memory)**
- The most recent messages in the active conversation thread
- When it fills up, Letta uses recursive summarization (evict 70% → summarize → store summary)
- Eviction is automatic based on context window limits; summarization is also automatic

### Sleep-Time Compute (Asynchronous Memory Agents)

Introduced in April 2025. Specialized "sleep-time agents" run asynchronously to:
- Reorganize and improve memory blocks during idle periods
- Run proactive memory refinement without blocking conversation latency
- Rewrite core memory blocks based on accumulated interactions

This is opt-in and requires configuring additional agents — it is not a default background pipeline.
Source: letta.com/blog/sleep-time-compute (April 2025)

### Lifecycle Management: Agent-Controlled, Not Automatic

The core design principle: **the LLM agent decides what to remember**.

- Core memory updates: agent calls tools explicitly when it judges something worth remembering
- Archival memory inserts: agent calls `archival_memory_insert()` explicitly
- Eviction/summarization: automatic only for message buffer overflow — not for semantic memory
- No background decay, importance scoring, or automatic pruning of stored memories
- No forgetting curve — memories in archival storage persist indefinitely unless agent deletes them
- No automatic extraction from file edits or external tool outputs

Compared to Recall's hook-based pipeline: Recall's `observe-edit.js` and `recall-session-summary.js` hooks extract memories passively without any agent involvement. Letta requires the agent to be running and explicitly decide to update memory.

### Infrastructure and Self-Hosting

Self-host requires:
- **PostgreSQL with pgvector extension** (`CREATE EXTENSION IF NOT EXISTS vector;`)
- Docker Compose file provided in repo (`compose.yaml`)
- No separate Redis or Neo4j required by default
- Optional: external PostgreSQL (set `LETTA_PG_URI` env var)
- Optional: graph DB for archival memory (not included in default compose)

The default self-host is **2 services**: Letta server + PostgreSQL/pgvector.
This is simpler than Recall's 4-service stack (FastAPI + Qdrant + Neo4j + PostgreSQL + Redis).

Source: docs.letta.com/guides/docker/postgres/, AWS blog post on Aurora PostgreSQL integration (Nov 2025)

### SDK and Integration Surface

**Python SDK**: `pip install letta-client` — released April 17, 2025
**TypeScript SDK**: `npm install @letta-ai/letta-client` — released December 2024
**Agentic Learning SDK**: `pip install agentic-learning` — wraps any OpenAI-compatible client with one context manager (`with learning(agent="my_agent"):`) to make existing agents stateful

The Letta client SDK pattern:
```python
from letta_client import Letta
client = Letta(api_key=os.getenv("LETTA_API_KEY"))
agent = client.agents.create(model="openai/gpt-5.2", memory_blocks=[...])
response = client.agents.messages.create(agent_id=agent.id, input="...")
```

**Recall has no SDK.** The mcp__recall tools and REST API are the only programmatic surfaces. (confirmed from prior Q1.1 research)

### Multi-LLM Support

Recall's extraction LLM is Ollama-only (qwen3:14b). (confirmed from prior research)

### Knowledge Graph Support

Letta supports graph databases as an **optional archival memory backend**:
- No native graph schema is defined — graph integration is pluggable
- Not enabled in the default compose setup
- No named relationship types (CAUSED_BY, SUPPORTS, etc.) comparable to Recall's Neo4j schema
- Zep's Graphiti (separate project) is the primary competitor with native temporal graph memory

## Mitigation Recommendation

The critical finding from Letta analysis is that **Recall's automatic pipeline is a genuine differentiator** that Letta does not replicate. Letta's agent-controlled memory requires the model to actively manage its own memory, which is powerful for agentic use cases but inappropriate for passive background memory injection. Recall's hook-based extraction that fires on every edit and session end has no equivalent in Letta's architecture.

However, Recall is significantly behind on:

1. **SDK (Critical, ~2 weeks effort)**: A thin Python wrapper around the existing REST API is the minimum viable fix. Model after letta-client's simple CRUD pattern: `RecallClient(base_url=...).store(content=...).search(query=...)`. The agentic-learning SDK pattern (`with learning():`) is worth studying as a one-line integration model.

2. **Self-hosting simplicity (High, architectural)**: Letta runs on PostgreSQL + pgvector only — no separate vector DB service. Recall's requirement for Qdrant + Neo4j + Redis + PostgreSQL + Ollama is the largest self-hosting barrier. Consider whether a single-service mode (SQLite + embedded vector) could serve as an entry point, similar to mem0's embedded Qdrant option.

3. **LLM-agnostic extraction (High, ~1-2 months)**: Letta's model-agnostic stance means teams using non-Ollama stacks can adopt it. Recall's extraction hardcoded to Ollama/qwen3:14b prevents adoption by teams using cloud LLMs. An `EXTRACTION_PROVIDER` env var supporting OpenAI, Anthropic, and Ollama would close this gap.

Recall's advantages that should be protected and highlighted:
- Automatic passive memory extraction (no agent involvement required)
- Typed knowledge graph edges (CAUSED_BY, SUPPORTS, CONTRADICTS — Letta has none)
- Full memory lifecycle with decay, consolidation, hygiene, and garbage collection
- Native Claude Code hook depth
