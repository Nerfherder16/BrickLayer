## Evidence

### Current Status and Architecture

- **Zep Cloud** (the live product): Cloud-only managed service. Python, TypeScript, and Go SDKs. Sub-200ms context retrieval claimed. SOC 2 Type II + HIPAA BAA.
- **Zep Community Edition**: Deprecated as of April 2, 2025. Code archived in `legacy/` folder. No updates, no support. Source: https://blog.getzep.com/announcing-a-new-direction-for-zeps-open-source-strategy/
- **Graphiti** (the OSS engine): Apache 2.0, actively maintained, 4.2k stars. Powers Zep Cloud internally. Self-hostable with Neo4j or FalkorDB + Python 3.10+. This is Zep's only meaningful open-source offering. Source: https://github.com/getzep/graphiti
- **GitHub repo** (github.com/getzep/zep): Now an examples/integrations repo. Last commit Feb 14, 2026. 4.2k stars, 588 forks. Pivoted from product to examples hub. Source: https://github.com/getzep/zep

### Core Architecture: Temporal Knowledge Graph (Graphiti)

Zep's central technical innovation is **temporal fact invalidation** — a design Zep has published as an academic paper (arXiv:2501.13956, Jan 2025). Every fact stored in the graph carries `valid_at` and `invalid_at` timestamps on its edges. When new information contradicts a prior fact, the old edge is closed (given an `invalid_at` date) rather than deleted, and a new edge is created. This preserves the full history of how facts evolve.

Example from official CE announcement blog post:
> **Fact:** "Kendra loves Adidas shoes." (valid_at: 2024-08-10)
> **User update:** "I'm so angry! My Adidas shoes fell apart! Puma's are my new favorite."
> **Result:**
> - "Kendra loves Adidas shoes." (valid_at: 2024-08-10, invalid_at: 2024-09-25)
> - "Kendra's Adidas shoes fell apart." (valid_at: 2024-09-25)
> - "Kendra prefers Puma." (valid_at: 2024-09-25)

Source: https://blog.getzep.com/announcing-zep-community-edition/

This is **the most mature temporal fact lifecycle of any memory system surveyed**. It handles fact contradiction and supersession as a first-class concern — something Recall does not currently implement.

### Key Concepts (Current Zep Cloud)

From https://help.getzep.com/concepts:

| Concept | Description |
|---------|-------------|
| **Knowledge Graph** | Unified knowledge store; nodes = entities, edges = facts/relationships. Updates dynamically. |
| **Context Block** | Optimized string containing user summary + relevant facts. Includes `valid_at` / `invalid_at` metadata. Injected directly into LLM system prompt. |
| **Fact Invalidation** | When new data invalidates a prior fact, the `invalid_at` date is stored on the edge. Old facts are never deleted — they expire. |
| **Episodes** | Any data unit ingested (message, JSON payload, text block). Core billing unit. |
| **User Graph** | Per-user graph with full session history. All threads for a user feed a single user graph. |
| **Group Graphs** | Shared graphs across users for non-user-specific memory (org knowledge, shared context). |
| **Custom Entity Types** | Pydantic-like classes defining domain-specific entities (patients, products, orders). Shipped Q2 2025. |
| **Custom Edge Types** | Define custom relationship types. Experimental as of Q2 2025. |
| **Context Templates** | Customize the format of the context block returned by `memory.get()`. |

### Session Management

Zep manages **threads** (conversation sessions) as first-class objects. Key behaviors:
- All threads for a user automatically feed into that user's unified Knowledge Graph.
- `memory.add(session_id, messages)` — single call to ingest a conversation turn.
- `memory.get(session_id)` — retrieves context relevant to the current session from the full user graph.
- Sessions were previously first-class retrieval units; as of the May 2025 deprecation wave, session-level retrieval is replaced by graph-level retrieval.
- Old session-scoped APIs retired: `memory.searchSessions`, `memory.endSession`, `memory.getSummaries`, `memory.synthesizeQuestion`. These are replaced by `graph.search` and `memory.get.context`.
- Source: https://blog.getzep.com/zep-feature-retirements-may-2025/

### Retrieval Architecture (Current)

From https://help.getzep.com/retrieving-memory:

Three retrieval methods:
1. **Context Block** (automatic): Combines semantic search + full text search + breadth-first graph traversal. Uses the last 2 messages as query. Returns a formatted string ready to inject into prompts.
2. **Custom Context Templates**: Same trigger, custom format.
3. **Advanced Context Block**: Full query control — custom queries, custom re-rankers (options include `episode_mentions` and `cross_encoder`), filter by node labels or edge types, min fact rating.

The Context Block is retrieved asynchronously pre-computed — Zep claims sub-200ms assembly latency because the graph is maintained incrementally, not computed on demand.

### Infrastructure Requirements (Self-Hosted / Graphiti)

For self-hosted usage via Graphiti:
- **Neo4j 5.26+** (or FalkorDB 1.1.2+)
- **Python 3.10+**
- **OpenAI API key** by default for LLM inference and embedding
- Alternative LLM providers: Azure OpenAI, Google Gemini, Anthropic, Groq, local Ollama models
- `pip install graphiti-core`

Source: https://help.getzep.com/graphiti/graphiti/installation

This means **full Zep self-hosting no longer exists** — it was retired with CE. Self-hosted graph memory is now Graphiti-only, which lacks the session management, user graph, and context assembly layers that Zep Cloud provides.

### Memory Lifecycle: What Zep Has

**Temporal invalidation (STRONG):** Full `valid_at` / `invalid_at` tracking on every edge. Facts expire when superseded. Historical state is queryable. This is equivalent to a first-class lifecycle pipeline for fact validity.

**Memory decay (ABSENT):** No time-based importance decay. Facts do not naturally fade from relevance based on recency unless explicitly superseded by new data. If a fact was stated once and never updated, it remains at full weight indefinitely.

**Consolidation (ABSENT):** No automatic merging of similar facts into a unified entity representation. Graphiti does have "communities" (groups of related nodes) but these are a structural feature, not a consolidation pipeline.

**Hygiene / GC (ABSENT):** No garbage collection of abandoned or orphaned memories. The graph grows monotonically. Deletion is manual via API.

**Dream/compression pipeline (ABSENT):** No overnight consolidation, importance recalculation, or compression cycle.

**Fact rating (PRESENT):** Zep Cloud has a `min_fact_rating` filter — facts can be assigned a relevancy rating, and retrieval can filter below a threshold. This is a primitive form of importance scoring. Source: https://help.getzep.com/v2/sdk-reference/memory/get

## Mitigation Recommendation

### Critical Gap: Temporal Fact Invalidation (High effort, High impact)

Recall currently accumulates facts without tracking whether a prior fact has been superseded. Zep's `valid_at` / `invalid_at` pattern on graph edges is the correct design. Recommendation:

1. Add `valid_at` and `invalid_at` timestamps to the Neo4j memory edge schema.
2. During consolidation, when a new memory contradicts an existing one, close the old edge (set `invalid_at = now()`) rather than deleting it.
3. Retrieval should by default return only facts where `invalid_at IS NULL` (currently valid), with an option to query historical states.
4. The contradiction resolution hook in the consolidation pipeline is the natural place to implement this.

Effort estimate: 2-3 weeks (schema migration + consolidation logic update + retrieval filter).

### Advantage to Protect: Lifecycle Pipeline

Zep has NO decay, NO consolidation, NO hygiene pipeline, and NO GC. This is Recall's strongest competitive differentiator. The lifecycle pipeline (decay → consolidation → hygiene → dream → GC) is not something Zep is building. Protect this by:
- Documenting it prominently (currently invisible to potential users)
- Adding metrics that show memory quality improvement over time (demonstrate the value of the pipeline)

### Gap: Context Assembly

Zep's Context Block (pre-assembled, sub-200ms, formatted for injection) is a developer experience win. Recall returns raw memories; the agent must format them. Add a `/context/{session_id}` endpoint that returns a formatted context string ready for system prompt injection. This is a low-effort, high-visibility improvement.

Effort estimate: 1-2 days.

### Note on Self-Hosting

Zep CE's deprecation (April 2025) is a strategic opportunity for Recall. Developers who need a fully self-hosted memory system with graph capabilities can no longer use Zep. Graphiti is available but lacks session management, context assembly, and user management. Recall fills this gap — but only if it has documentation and an SDK to lower the adoption barrier.
