## Evidence

### What mem0's SDK Teaches Us

From Q3.1 research, mem0's SDK is the competitive benchmark:

```python
# mem0 — 5 lines to first memory stored (embedded mode, ZERO infrastructure)
from mem0 import Memory
m = Memory()
m.add("I prefer Python for data science", user_id="alice")
results = m.search("programming preferences", user_id="alice")
print(results)
```

Key mem0 SDK properties:
- `pip install mem0ai` — installable without infrastructure (embedded mode)
- 2,034,758 monthly PyPI downloads
- Methods: `add`, `search`, `get`, `get_all`, `update`, `delete`, `delete_all`, `history`, `reset`
- User/agent/session filtering as first-class parameters
- Zero infrastructure in embedded mode (SQLite + local embedding)
- 3-container infrastructure for production (Qdrant + Neo4j + PostgreSQL optional)

### What Recall's REST API Already Has (API Audit)

From the FastAPI server at 192.168.50.19:8200/docs (inferred from Q3.5 finding — FastAPI auto-generates OpenAPI schema):
- POST /memories — store a memory
- GET /memories/search — semantic search
- GET /memories/{id} — retrieve by ID
- DELETE /memories/{id} — delete
- GET /timeline — chronological browse
- GET /health — health check

The REST API exists. A Python SDK is a thin HTTP wrapper around it. This is NOT a rearchitecting task.

### Minimum Viable SDK Spec

**Package name**: `recall-sdk` (or `system-recall`)

**Installation**: `pip install recall-sdk`

**Core API** (10 methods — mirrors mem0 ergonomics):

```python
from recall import RecallClient

# Initialize — connects to self-hosted instance
client = RecallClient(
    base_url="http://192.168.50.19:8200",  # or env: RECALL_API_URL
    api_key="..."                           # or env: RECALL_API_KEY (optional, if auth enabled)
)

# Store a memory
memory = client.store(
    content="Tim prefers Python for backend APIs",
    domain="preferences",         # optional: categorize
    tags=["python", "backend"],   # optional
    importance=0.8,               # optional: 0.0-1.0
    durability="durable"          # optional: ephemeral/durable/permanent
)
# returns: Memory object with id, content, created_at, importance

# Search
results = client.search(
    query="What language does Tim prefer?",
    limit=5,
    domain="preferences"          # optional: filter by domain
)
# returns: List[Memory] with relevance scores

# Retrieve by ID
memory = client.get(memory_id="abc-123")

# Delete
client.delete(memory_id="abc-123")

# Browse timeline
timeline = client.timeline(limit=20, domain=None)

# Health check
status = client.health()  # returns {"status": "ok", "services": {...}}
```

**10-line integration example** (comparable to mem0):

```python
from recall import RecallClient

client = RecallClient(base_url="http://localhost:8200")

# Store architectural decision
client.store(
    "We use JWT RS256 with httpOnly cookies — never localStorage",
    domain="architecture",
    tags=["auth", "security"],
    durability="permanent"
)

# Retrieve at next session
memories = client.search("authentication approach")
for m in memories:
    print(f"[{m.domain}] {m.content}")
```

### Async Support

Provide both sync (`RecallClient`) and async (`AsyncRecallClient`) variants. mem0 added async in Q3 2025 — users expect this. Implementation: `httpx.AsyncClient` for async, `httpx.Client` for sync. Same interface.

### Authentication Pattern

Recall's API currently uses no authentication (private homelab). SDK should support:
1. No auth (default, for local homelab)
2. Bearer token: `RecallClient(api_key="token")` → adds `Authorization: Bearer {token}` header

### Effort Estimate

**Core SDK**: 2-3 days of work
- `recall/client.py` — sync HTTP wrapper around REST endpoints (httpx)
- `recall/async_client.py` — async version (same interface)
- `recall/models.py` — Pydantic models for Memory, SearchResult, HealthStatus
- `tests/` — basic integration tests against local instance

**PyPI packaging**: 1 day
- `pyproject.toml`, `setup.py`, GitHub Actions CI for publish-on-tag

**Documentation**: 1-2 days (linked to Q5.2/Q3.5 — should be done together with docs site)
- README with 10-line quickstart
- API reference (auto-generated from docstrings via pdoc or mkdocs)

**Total: 4-6 days (1 week)**. This is NOT a large project. mem0's SDK is 3,200 lines of Python. Recall's minimum viable SDK is ~400 lines.

### Critical Missing Piece: Embeddable Mode

mem0's biggest adoption advantage is its embedded mode (zero infrastructure). Recall cannot easily replicate this because it requires Qdrant + Neo4j + Redis + PostgreSQL + Ollama. However, a **minimal embedded mode** could be:
- SQLite for storage (replaces PostgreSQL + Redis)
- Qdrant local mode (in-process, no server needed — Qdrant supports this)
- No graph (Neo4j required for graph; skip for embedded mode)
- OpenAI or Ollama embedding

This is a 2-4 week project but would dramatically close the adoption gap for developers who don't want infrastructure.

## Mitigation Recommendation

**This is the single highest-ROI action for Recall's competitive position.**

Week 1 sprint (4-6 days total):
1. Create `recall-sdk` Python package wrapping existing REST API
2. Publish to PyPI (test.pypi.org first, then pypi.org)
3. Write 10-line quickstart as README
4. Bundle SDK launch with minimal Mintlify docs site

The docs site and SDK are mutually dependent — publishing the SDK without docs, or docs without SDK, is half-value. Do both in the same sprint.

TypeScript SDK: defer until Python SDK has adoption signal (>100 downloads/week). mem0 prioritized Python first, TS second. Same strategy makes sense for Recall.
