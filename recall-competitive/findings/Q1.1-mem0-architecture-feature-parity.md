### Architecture Overview

mem0 uses a two-phase pipeline for every memory write:

1. **Extraction Phase**: An LLM reads the most recent M conversation messages, extracts discrete facts ("Raw Memories"), and passes them to the Update Phase.
2. **Update Phase**: Evaluates incoming raw memories against existing memories — decides whether to ADD, UPDATE, or DELETE. This is the deduplication and merge logic. The LLM judges whether a new memory supersedes, contradicts, or extends an existing one.

Storage is dual-layer (vector + optional graph):
- **Vector store** (always-on): Qdrant is the recommended default for OSS; also supports pgvector, ChromaDB, Milvus, Pinecone, Weaviate, Redis, Faiss, Azure AI Search, Vertex AI, and 10+ others. OSS quickstart defaults to an embedded Qdrant at `/tmp/qdrant`.
- **Graph store** (opt-in): Neo4j, Memgraph, Amazon Neptune Analytics, Amazon Neptune DB, or Kuzu (embedded, no server required). Requires `pip install "mem0ai[graph]"`.
- **History store**: SQLite at `~/.mem0/history.db` — local audit trail of every memory operation.

Default OSS config wires:
- LLM: `gpt-4.1-nano-2025-04-14` (OpenAI)
- Embedder: `text-embedding-3-small` (OpenAI, 1536 dims)
- Vector store: Qdrant (embedded, `/tmp/qdrant`)
- History: SQLite

Source: https://docs.mem0.ai/open-source/python-quickstart, https://docs.mem0.ai/open-source/configuration

### Graph Memory Implementation

mem0's graph memory extracts entities (people, places, projects) and typed relationships from every `memory.add()` call, then mirrors them in the graph backend in parallel with the vector write.

- **Providers supported**: Neo4j (Aura or self-hosted with APOC), Memgraph (Docker), Amazon Neptune Analytics, Amazon Neptune DB, Kuzu (embedded)
- **Retrieval**: `memory.search()` runs vector similarity first (optionally reranked) and returns results. Graph runs in parallel and appends related entities in a `relations` array — it does not rerank the vector results.
- **Custom prompts**: You can constrain what becomes a node/edge via `custom_prompt` in the graph_store config.
- **Per-request toggle**: `memory.add(..., enable_graph=False)` — can disable graph writes per call.
- **Confidence threshold**: Configurable `threshold` to filter noisy edges.
- **Quoted doc text**: "Mem0 extracts entities and relationships from every memory write, stores embeddings in your vector database, and mirrors relationships in a graph backend. On retrieval, vector search narrows candidates while the graph returns related context alongside the results."

Source: https://docs.mem0.ai/open-source/features/graph-memory

### Python SDK Surface

```python
from mem0 import Memory

m = Memory()                                    # zero-config default
m.add(messages, user_id="alex")                # extract + store
m.search("query", filters={"user_id": "alex"}) # vector search
m.get_all(filters={"user_id": "alex"})         # list all memories
m.update(memory_id, data)                      # update specific memory
m.delete(memory_id)                            # delete specific memory
m.delete_all(user_id="alex")                   # wipe user memories
m.history(memory_id)                           # audit trail for a memory
```

Also available: `MemoryClient` (cloud-hosted API), `AsyncMemory` (async variant).
TypeScript/Node SDK (`mem0ai` npm package) covers same CRUD surface; `import { Memory } from "mem0ai/oss"` for self-hosted.

**Install complexity**: `pip install mem0ai` — single command. Graph extras: `pip install "mem0ai[graph]"`.

Source: https://github.com/mem0ai/mem0 README, https://docs.mem0.ai/open-source/python-quickstart

### Multi-LLM Support

Supported LLMs (Python SDK): OpenAI, Ollama, Azure OpenAI, Anthropic, Together, Groq, LiteLLM, Mistral AI, Google AI, AWS Bedrock, DeepSeek, xAI, Sarvam AI, LM Studio, Langchain.
TypeScript SDK supports: OpenAI, Anthropic, Groq only.

Note: A current open bug (Issue #4192, 2026-03-02) shows `get_default_configuration()` in config.py ignores `default_config.json` and hardcodes OpenAI defaults — meaning non-OpenAI users must always explicitly pass config. PR #4199 filed but not yet merged as of research date.

Source: https://docs.mem0.ai/components/llms/overview, https://github.com/mem0ai/mem0/issues/4192

### Embedding Model Support

Configured via `embedder.provider`. Known supported providers: OpenAI (`text-embedding-3-small` default), Ollama (`nomic-embed-text`, others), VertexAI, Azure OpenAI. Embedding dimension must be set manually in vector store config when using non-OpenAI models.

Known issue: dimension mismatch errors are a recurring pain point when switching embedding models (GitHub Issues #3439, #2758). Users report spending days debugging the `embedding_model_dims` requirement.

### Self-Hosting Infrastructure Requirements

**Minimal OSS (vector only)**:
- Python 3.10+
- `pip install mem0ai`
- Embedded Qdrant (no separate service — runs in-process, stores to `/tmp/qdrant`)
- An LLM API key (OpenAI by default; can use Ollama for fully local)

**With graph memory (self-hosted server mode)**:
Three Docker containers (per official blog post, 2026-02-23):
1. `mem0/mem0-api-server:latest` — FastAPI REST API
2. `ankane/pgvector:v0.5.1` — PostgreSQL + pgvector (replaces embedded Qdrant)
3. `neo4j:5.26.4` — graph store with APOC plugin

Total Docker images: ~500 MB. Neo4j has a 90-second startup time. No authentication or CORS protection by default — docs note a reverse proxy is required before production use.

Note: The "fully local" path (Ollama LLM + Ollama embedder + embedded Qdrant) requires zero external services — just `pip install mem0ai` and Ollama running locally.

Source: https://mem0.ai/blog/self-host-mem0-docker

### Memory Lifecycle — Deduplication and Expiration

**Deduplication**: mem0's dedup is LLM-driven, not cosine-threshold-based. The Update Phase LLM decides whether a new memory conflicts with, extends, or supersedes existing memories. This is more semantic than Recall's content-hash + cosine approach but is LLM-call-dependent (adds latency and cost per write).

**Memory expiration**: Available on the **Platform (cloud)** tier via `expiration_date` parameter on `client.add()`. The docs describe this as a Platform feature. No evidence of TTL/decay in the OSS/self-hosted path. Memory persists forever by default in OSS.

**No automatic decay pipeline**: There is no equivalent to Recall's `decay.py`, `consolidation.py`, `hygiene.py`, or `gc.py` in mem0 OSS. The graph docs mention manual Cypher cleanup (`MATCH (n) WHERE n.lastSeen < date() - duration('P90D') DETACH DELETE n`) as the recommended approach for graph node pruning — this is manual, not automated.

Source: https://docs.mem0.ai/platform/features/expiration-date, https://docs.mem0.ai/open-source/features/graph-memory

### Known User Complaints (from GitHub Issues)

1. **Default config hardcoded to OpenAI** (Issue #4192, open 2026-03-02): `get_default_configuration()` ignores custom config files, forces users to always pass explicit config.
2. **Ollama integration fragile** (Issue #2758, closed; #3439, open): Embedding dimension mismatches, model schema failures — users report "spending a week" debugging local setups.
3. **Memory entries too small** (Issue #2762): LLM fact extraction with weaker models (e.g., Ollama's mistral:instruct) produces single-sentence memories from large documents — context loss on long inputs.
4. **AWS Bedrock config broken** (Issue #4136, open 2026-02-26): `LlmFactory` maps `aws_bedrock` to `BaseLlmConfig` instead of `AWSBedrockConfig`, causing `TypeError` on any AWS-specific params.

Source: GitHub issues linked above

## Mitigation Recommendations

### 1. Minimal Python SDK (High priority, ~1-2 weeks effort)
Build a thin Python package `recall-sdk` wrapping the existing REST API. Minimum surface:

```python
from recall import Recall

r = Recall(base_url="http://192.168.50.19:8200")
r.store(content, domain, tags, importance)
r.search(query, domain, limit)
r.get(memory_id)
r.delete(memory_id)
```

This is a REST wrapper — no new backend logic. The existing FastAPI is already complete. The SDK is purely a developer convenience layer. mem0's pattern shows this is the #1 adoption driver.

### 2. Zero-config local path (High priority, ~1 week effort)
Create a `recall-lite` mode: a single Docker container (or even a single Python process) that uses embedded Qdrant + SQLite history, no Neo4j, no Redis, no Postgres. This would match mem0's zero-infrastructure entry point and dramatically lower the barrier for new users. Full mode (all 5 services) remains available for power users.

### 3. Documentation site (Medium priority, ~1 week effort)
Stand up a docs site (MkDocs, Mintlify, or Docusaurus) with: quickstart, configuration guide, REST API reference (auto-generated from FastAPI OpenAPI spec), hook integration guide, and a cookbook. mem0's docs site is the clear model for what makes a self-hosted memory system feel credible.

### 4. Multi-LLM extraction support (Medium priority, ~2-3 weeks effort)
The observer.py LLM call for fact extraction is the main bottleneck. Abstract the LLM client to support LiteLLM as the backend, which would unlock 100+ provider compatibility. This doesn't require changing the hook system — only the extraction pipeline.

### 5. Leverage lifecycle advantage in marketing (Low effort, high impact)
Recall's multi-stage lifecycle pipeline (decay + consolidation + hygiene + GC + dream consolidation) is a **genuine unique differentiator** — no competitor has anything comparable in OSS. This needs to be front-and-center in any README or docs: "The only self-hosted memory system with automatic memory lifecycle management." This is already built; it just needs to be communicated.

## Verdict Rationale

**WARNING** (not FAILURE) because:

The verdict threshold for FAILURE was: "mem0 has SDK + multi-LLM support + graph memory AND is simpler to deploy." This is true — mem0 satisfies all three conditions. However, the FAILURE threshold assumed mem0 would be "simpler to deploy" comprehensively. The picture is nuanced: mem0's minimal path (pip install, embedded Qdrant, no graph) is far simpler; but the full self-hosted path with graph (3 Docker containers) is not dramatically simpler than Recall's full stack (5 containers), and Recall's graph schema and lifecycle pipeline are meaningfully more sophisticated.

The SDK gap and documentation gap push toward WARNING rather than FAILURE because Recall's target audience (technical self-hosters, Claude Code users) can tolerate raw HTTP integration better than general developers. The lifecycle pipeline advantage is real and significant. The gap is not existential — it is bridgeable with a thin SDK wrapper and documentation site.
