## Evidence

### Starting Point: Recall Leads the Field on Lifecycle

From Q2.2: Zero competitors implement automatic lifecycle management:
- mem0: "memories persist forever" by default; only manual TTL at write time; no background process
- Zep/Graphiti: temporal invalidation (new fact overwrites old) but no access-frequency decay, no GC
- Letta: zero automatic lifecycle; agent-controlled only

Recall's pipeline (decay → consolidation → hygiene → GC → dream consolidation) is unique in production systems. The December 2025 survey confirms "self-optimizing memory management" is an emerging frontier.

This section identifies specific improvements to the pipeline — not to catch up to competitors, but to widen the lead.

### Improvement 1: Temporal Validity Windows (adopt from Graphiti) — HIGH VALUE

**What Graphiti does**: Every fact has `valid_from` and `valid_until` timestamps. When "Alice works at Google" is superseded by "Alice works at Meta", the old fact is invalidated (valid_until = now) but retained for historical queries. Full audit trail.

**What Recall does**: Uses SUPERSEDES edge in Neo4j graph. Old fact is not deleted but not explicitly invalidated. Retrieval score is reduced via importance decay. Historical reconstruction requires graph traversal — not first-class.

**Improvement**: Add `valid_from` and `valid_until` fields to the Memory schema (Qdrant payload + PostgreSQL audit). When a SUPERSEDES edge is created:
1. Set `valid_until = now` on the old memory
2. Set `valid_from = now` on the new memory
3. Old memory continues to exist in the vector store with zero importance score — queryable only with `include_historical=True`

**Value**:
- Provides full historical audit trail (currently a Graphiti advantage)
- Enables "what did I know at time T?" queries — uniquely powerful for coding sessions (debug "when did we decide to use JWT?")
- Closes the one genuine architectural gap Graphiti has over Recall

**Effort**: Medium (2-4 days)

### Improvement 2: Entity-Centric Deduplication — MEDIUM VALUE

**Current approach**: Content hash (exact) + cosine similarity threshold (near-duplicate).

**Problem**: Two memories about the same entity may not be near-duplicates semantically:
- "Tim's homelab runs on CasaOS at 192.168.50.19"
- "The CasaOS server has a Ryzen 5 3600 CPU"

Both are about the same entity (CasaOS server) but will not cluster as near-duplicates by cosine similarity alone. Over 1000 sessions, there could be 50 memories about CasaOS scattered in the vector space.

**Improvement**: Entity-centric consolidation pass:
1. During the consolidation pipeline, group memories by extracted entities (Neo4j entity nodes)
2. For entities with >10 associated memories, trigger a consolidation LLM call: "Summarize these 10 facts about CasaOS server into a coherent updated profile"
3. Replace the 10 individual memories with 1 consolidated memory; create SUPERSEDES edges to originals

**Value**: Dramatically improves retrieval quality over long sessions (100+ memories). Currently, Recall's vector space will accumulate scattered fragments about the same entity.

**Effort**: Medium-High (1 week — requires entity extraction in consolidation pipeline)

### Improvement 3: Access-Pattern-Weighted Importance Decay — MEDIUM VALUE

**Current approach**: Importance decay based on age. Access count tracked but not used as decay signal.

**Improvement**: Implement differential decay:
- High-access memories (>10 retrievals): decay rate × 0.5 (slow decay — user finds it useful)
- Zero-access memories (0 retrievals in 60 days): decay rate × 2.0 (fast decay — possibly wrong or irrelevant)
- Permanent durability: exempt from all decay (for arch-decision memories once implemented)

**Value**: System gets smarter about what's actually useful. Currently, a memory stored 90 days ago decays at the same rate regardless of whether it has been retrieved 0 or 50 times.

**Effort**: Low (1-2 days — access_count already tracked in Qdrant payload)

### Improvement 4: Contradiction Detection and Resolution — HIGH VALUE

**Current state**: Recall creates CONTRADICTS edges in Neo4j when a memory contradicts an existing one. But the resolution is implicit — higher importance wins. No LLM-based adjudication.

**Improvement**: Add explicit contradiction resolution step to the hygiene pipeline:
1. Identify memory pairs with CONTRADICTS edges
2. Query for context: when were these memories stored? which is more recent? what was the source (session vs edit)?
3. Prompt Ollama: "Given these two contradicting facts, which is more likely correct? Why?"
4. Mark the less-likely fact as `valid_until = now` and boost the more-likely fact's importance

**Value**: Recall's CONTRADICTS edge is unique in production. Making it actionable (not just recorded) would significantly improve long-term memory quality. No competitor even records contradictions.

**Effort**: Medium (3-5 days)

### Improvement 5: Dream Consolidation — Cross-Domain Insight Quality — MEDIUM VALUE

**Current state**: Dream consolidation runs nightly, discovers relationships between memories from different domains.

**Improvement**: Structured consolidation prompts rather than open-ended discovery:
1. "What patterns exist across these memories about Tim's coding habits?"
2. "What infrastructure decisions are potentially inconsistent with each other?"
3. "What preferences expressed in casual conversation might be relevant to architectural decisions?"

This moves dream consolidation from general relationship discovery to purposeful insight synthesis.

**Effort**: Low (1-2 days — prompt engineering only)
