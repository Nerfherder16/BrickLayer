# Finding: Q2.4 — Knowledge Graph Schemas

## Evidence

### mem0 Graph Memory Schema

Source: docs.mem0.ai/open-source/features/graph-memory (scraped 2026-03-15); arxiv.org/html/2504.19413v1 (Mem0 paper, May 2025); GitHub PR #2073.

mem0's graph mode uses **fully freeform, LLM-extracted relationship labels**. The extraction pipeline:

1. LLM call 1: Identify entities and their types from conversation payload (returns dict `{node: type}`)
2. LLM call 2: Establish relationships between extracted nodes (returns list of `source -- relationship -- destination` triplets)

The relationship label in step 2 is whatever string the LLM decides to emit — e.g., "met_at", "works_for", "prefers", "attended". There is no fixed ontology, no enum of allowed edge types, and no type validation. The mem0 docs confirm: "Mem0's extraction LLM identifies entities, relationships, and timestamps from the conversation payload." The docs example shows `Alice met Bob at GraphConf 2025`, producing an edge labeled by whatever verb phrase the LLM extracts.

Per the ResearchGate figure caption (figure 3 of the Mem0 paper): "The extraction phase uses LLMs to convert conversation messages into entities and relation triplets. The update phase employs conflict detection and resolution mechanisms when integrating new information into the existing knowledge graph." No typed edge schema is mentioned anywhere in public documentation.

mem0's graph backend supports Neo4j, Memgraph, AWS Neptune Analytics, AWS Neptune DB, and Kuzu — all configured identically, with no schema constraints imposed by mem0 itself.

**Key limitation**: Because edge labels are freeform strings, querying "all memories where A caused B" or "all claims that contradict memory X" is impossible without full-text scan of edge labels, which will match inconsistently ("caused", "caused_by", "led_to", "resulted_in" are all different labels). Graph traversal for causal or contradiction queries is unreliable.

The Mem0 paper (arXiv:2504.19413, May 2025) confirms graph mode yields ~2% accuracy improvement over base Mem0 on LOCOMO — significant but modest, suggesting the freeform graph adds relationship context but does not enable structured causal reasoning.

### Zep / Graphiti Schema

Sources: help.getzep.com/graphiti/core-concepts/custom-entity-and-edge-types (scraped 2026-03-15); blog.getzep.com/beyond-static-knowledge-graphs/ (scraped 2026-03-15); arxiv.org/html/2501.13956v1 (Zep paper, Jan 2025); github.com/getzep/graphiti (v0.28.2, last push 2026-03-13).

Graphiti uses a **bi-temporal edge schema** as its distinguishing innovation. Every edge carries four temporal fields:

- `created_at`: when the relation was added to the database
- `expired_at`: when the relation was invalidated in the database (nullable)
- `valid_at`: when the relation became true in the real world (LLM-extracted, optional)
- `invalid_at`: when the relation stopped being true in the real world (LLM-extracted, optional)

The **default edge type** in Graphiti is `RELATES_TO` — a generic fallback. The Zep docs state explicitly: "If an entity pair doesn't have a defined edge type mapping, Graphiti will use default relationship types and the relationship will still be captured with a generic RELATES_TO type."

Graphiti added **custom entity and edge types** (Pydantic models) as an opt-in feature. The docs show example custom edge types including `Employment`, `Investment`, `Partnership` — all domain-specific, user-defined. The built-in default types remain generic (`RELATES_TO`), and the examples shown (`works_as`, `BOUGHT`, `DIVORCED_FROM`, `MARRIED_TO`) are LLM-extracted freeform strings, exactly like mem0. The temporal fields are the layer that differentiates Graphiti from mem0, not the edge type semantics.

The Zep arXiv paper (2501.13956) describes Graphiti's three node types: **Episodic** (raw conversation episodes), **Semantic** (extracted facts/entities), and **Community** (clusters of related semantic nodes). No causal or logical relationship types are defined at the system level.

**Key distinction from Recall**: Graphiti's edge semantics answer "when was this true?" not "why is this true?" or "does this contradict that?" Temporal invalidation is automatic (LLM detects conflicts and sets `expired_at`), but the invalidation is binary (valid/invalid), not typed as CONTRADICTS, SUPERSEDES, or SUPPORTS. A user cannot query "which memories have been contradicted?" — only "which edges are currently valid?"

Graphiti reaches v0.28.2 with 23,703 stars and active development (last push 2026-03-13), confirming it is the most mature competitor graph system.

### Letta / MemGPT Archival Memory

Source: Q1.3-letta-hierarchical-memory.md (prior finding, confirmed training data).

Letta has **no graph layer**. Archival memory is flat pgvector (PostgreSQL with vector extension). There are no nodes, no edges, and no relationship types. The "archival" store is a vector similarity index. No graph schema comparison applies.

### Cognee Schema

Source: docs.cognee.ai/how-to-guides/cognee-sdk/ontology-support; cognee.ai/blog/deep-dives/ontology-ai-memory (scraped via search, 2026-03-15).

Cognee takes an **OWL ontology-grounded** approach. Users supply `.owl` files (RDF/XML) defining formal ontologies with:
- Classes (e.g., `TechnologyCompany`, `Car`)
- Object properties with domain/range constraints (e.g., `produces` with `domain=CarManufacturer`, `range=Car`)
- Named individuals

Cognee's `cognify` pipeline validates extracted entities and relations against the ontology's domain/range constraints and class hierarchy. This is more structured than mem0 or Graphiti's defaults, but the relationship types are entirely user-defined per-domain — there is no built-in causal, logical, or epistemic schema. Cognee is primarily focused on institutional/document knowledge graphs, not conversational agent memory.

### MAGMA Paper (arXiv:2601.03236, Jan 2026)

The MAGMA paper explicitly validates the multi-graph approach by identifying the **four orthogonal memory dimensions** that production systems conflate into a single store:

1. **Semantic graph** — concept similarity and meaning
2. **Temporal graph** — chronological ordering and time-based relationships
3. **Causal graph** — cause-effect relationships between events
4. **Entity graph** — named entity co-occurrence and association

MAGMA's core argument: "existing approaches largely rely on semantic similarity over monolithic memory stores, entangling temporal, causal, and entity information. This design limits interpretability and alignment between query intent and retrieved evidence." The paper demonstrates that separating causal edges from semantic edges enables structured "why" queries that vector similarity cannot answer.

MAGMA outperforms all tested agentic memory systems on LoCoMo and LongMemEval benchmarks. This is direct academic validation that typed causal edges are an unimplemented capability gap in all production competitors.

### Recall Schema

Source: simulate.py comment (current session); prior findings Q1.4 and Q2.2.

Recall uses Neo4j with the following **typed, semantic edge labels**:

- `RELATED_TO` — generic association
- `CAUSED_BY` — causal provenance
- `SUPPORTS` — epistemic support
- `CONTRADICTS` — epistemic contradiction
- `SUPERSEDES` — temporal invalidation with semantic intent preserved

Plus **domain partitioning** — memories are isolated by domain string (e.g., `recall-competitive-autoresearch`), enabling multi-tenant namespace queries within a single graph instance.

This schema enables Cypher queries that no competitor supports:
- "Find all memories that contradict memory X" → `MATCH (a)-[:CONTRADICTS]->(b) WHERE b.id = X`
- "Trace the causal chain behind memory Y" → `MATCH path = (a)-[:CAUSED_BY*]->(b) WHERE a.id = Y`
- "Which memories support or undermine claim Z?" → `MATCH (m)-[:SUPPORTS|CONTRADICTS]->(z) WHERE z.id = Z`

The `SUPERSEDES` edge provides typed temporal invalidation — semantically richer than Graphiti's binary `expired_at` flag because the superseded memory is retained in the graph with its provenance chain intact, and the query can retrieve "what replaced what" rather than just "what is currently valid."

## Score Updates

- `graph_relationship_depth`: 0.85 → **0.88**

  Reason: This finding confirms and deepens prior evidence. Recall's CAUSED_BY / SUPPORTS / CONTRADICTS / SUPERSEDES typed causal schema is present in no competitor as a built-in capability. Zep/Graphiti's custom edge type system (discovered this session) is a closer competitor than previously known — it allows user-defined edge types via Pydantic models — but the built-in default is still generic `RELATES_TO`, causal/epistemic types require user implementation, and the temporal invalidation model is fundamentally different (binary valid/invalid vs. typed provenance chain). MAGMA paper (Jan 2026) provides the strongest external validation yet that causal graph is an orthogonal unimplemented dimension. Score moves from 0.85 to 0.88 — not to 0.90+ because Graphiti's custom type system partially narrows the expressiveness gap (users who invest in defining custom ontologies can approximate typed edges), and Cognee's OWL support shows the market is moving toward structured schemas.

## Mitigation Recommendation

Recall's typed causal schema is the strongest evidence-backed architectural advantage across all researched dimensions. However, two risks should be addressed:

**Risk 1: Graphiti custom types are catching up.** Graphiti now supports user-defined Pydantic edge types with domain/range constraints (as of recent releases, v0.28.x). A motivated Graphiti user can define `Contradiction`, `Causal`, `Support` edge types manually. Recall's advantage is that these types are built-in and automatically applied — no user configuration required.

**Recommendation**: Document Recall's built-in causal schema explicitly in all user-facing materials. Add a comparison page showing that achieving equivalent expressiveness in Graphiti requires 50+ lines of custom Pydantic model definitions that the user must maintain. Frame Recall's schema as "batteries included epistemic graph" vs. competitors' "bring your own schema."

**Risk 2: MAGMA paper creates academic credibility for 4-graph architecture.** The MAGMA architecture (semantic + temporal + causal + entity as separate graphs) is more sophisticated than Recall's single Neo4j graph with typed edges. MAGMA uses policy-guided traversal across orthogonal graph views. If Recall is compared directly to MAGMA-like architectures, the single-graph approach looks simpler.

**Recommendation** (Medium effort, 2-4 weeks): Consider exposing Recall's typed edge categories as explicit query filters matching the four MAGMA dimensions — semantic (RELATED_TO, SUPPORTS), temporal (SUPERSEDES), causal (CAUSED_BY), entity. This reframes the existing schema as a unified-graph implementation of MAGMA's multi-graph concept, which is architecturally simpler without sacrificing expressiveness.

**Risk 3: Cognee's OWL ontology approach is more formally rigorous.** Domain validation, class hierarchy, and SPARQL-compatible queries are possible in Cognee with user-defined OWL files. For enterprise knowledge graph use cases, this is more audit-friendly than Recall's string-labeled edges.

**Recommendation** (Low priority for current target market): File as a backlog item to support optional OWL/SKOS ontology files for domain-specific deployments.
