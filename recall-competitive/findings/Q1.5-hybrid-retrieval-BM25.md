## Evidence

### Competitor Retrieval Architectures

**Zep** — Full hybrid BM25 + vector in production (confirmed, current docs):

> "Zep's graph search provides powerful hybrid search capabilities that combine semantic similarity with BM25 full-text search to find relevant information across your knowledge graph."
>
> Source: https://help.getzep.com/searching-the-graph (scraped 2026-03-15)

Zep's hybrid pipeline:
- Step 1: Dual-path retrieval — semantic (dense vector) + BM25 full-text in parallel
- Step 2: Reciprocal Rank Fusion (RRF) as default reranker — merges both result sets
- Additional rerankers available: MMR (diversity), cross-encoder (highest accuracy), node_distance (graph proximity), episode_mentions (frequency-weighted)
- All configurable per-request via `reranker` parameter

**mem0 OSS** — Vector search + cross-encoder reranker (NOT BM25 by default):

The mem0 OSS overview page (docs.mem0.ai/open-source/overview, scraped 2026-03-15) lists "Tune Retrieval & Rerankers — Hybrid retrieval and reranker controls" as a core self-hosting feature. The reranker docs confirm the retrieval flow is:

1. Initial vector search (candidate retrieval via cosine similarity)
2. Optional reranker pass: Cohere, Sentence Transformer (cross-encoder/ms-marco-MiniLM-L-6-v2), HuggingFace, LLM Reranker, Zero Entropy
3. Reordered results returned

mem0 does NOT use BM25 sparse vectors in its default retrieval path. Hybrid search was added only for the Azure AI Search vector store backend (PR #2408, merged March 2025). The Qdrant backend (Recall's identical backend) uses pure dense vectors + optional cross-encoder reranker.

**Zep academic paper** (arxiv.org/html/2501.13956v1, January 2025): Zep demonstrates 94.8% vs 93.4% on the Deep Memory Retrieval (DMR) benchmark vs MemGPT, attributing gains partly to its hybrid retrieval architecture via Graphiti knowledge graph engine.

**Recall** — Pure vector cosine similarity, qwen3-embedding:0.6b, no BM25, no reranker.

### The Critical Nuance: BM25 and Short Text

Qdrant (the vendor whose vector database Recall uses) published a definitive analysis of why classical BM25 is poorly suited for short document retrieval — which is exactly Recall's use case.

From https://qdrant.tech/articles/bm42/ (July 2024, current):

> "The typical length of a document in RAG is much shorter than that of web search... the document size in RAG is small and fixed. That effectively renders the term importance in the document part of the BM25 formula useless. The term frequency in the document is always 0 or 1, and the relative length of the document is always 1."

The BM25 formula has two meaningful components:
- **IDF (Inverse Document Frequency)**: Measures how rare a term is across the corpus. Remains fully valid for short documents.
- **Term frequency in document**: Measures how often a term appears within one document. Collapses to 0-or-1 for short text. Adds no signal.

**Conclusion from Qdrant**: For short-text memory (50-200 chars), traditional BM25 provides only IDF benefit — which is real but limited. Qdrant invented BM42 to address this: it replaces term-frequency weighting with transformer attention weights while retaining IDF, yielding meaningfully better short-text retrieval than BM25.

BM42 benchmark on Quora dataset (short question-deduplication, avg ~5.6 sparse elements per doc):

| Method | Recall@10 |
|--------|-----------|
| BM25 (tantivy) | 0.83 |
| BM25 (sparse vector) | 0.83 |
| BM42 | 0.85 |

These are sparse-only numbers. In hybrid combination with dense vectors, BM42 adds complementary coverage.

### Quantified Benefits of Hybrid Retrieval (General RAG Literature)

From arxiv preprint "Hybrid Dense-Sparse Retrieval for High-Recall Information Retrieval" (January 2026, ResearchGate DOI: 10.13140/RG.2.2.23909.46562):

> "Our method achieves up to 580% improvement in Recall@10 over dense-only retrieval on MS MARCO (13.9% → 80.8%)."

**Critical caveat**: MS MARCO consists of web documents (hundreds of tokens). The 580% figure does not apply to short-text memory retrieval. MS MARCO is a worst case for pure dense retrieval because dense models miss exact named entities.

From "From Retrieval to Generation" paper (arxiv 2502.20245, Feb 2025):
> "hybrid models improve nDCG@10 scores on BEIR from 43.42 (BM25) to 52.59"

This is BM25 alone → hybrid. The top dense-only models (Voyage, Cohere) already reach 52-54 nDCG@10 without BM25, meaning the hybrid gain over strong dense-only is much smaller (~0-2 nDCG points).

**BEIR 2.0 Leaderboard (2025)** (app.ailog.fr/en/blog/news/beir-benchmark-update):

| Rank | Model | nDCG@10 | Type |
|------|-------|---------|------|
| 1 | Voyage-Large-2 | 54.8% | Dense |
| 2 | Cohere Embed v4 | 53.7% | Dense |
| 6 | Qwen3-Embedding-8B | 51.5% | Dense |
| 9 | BM25 | 41.2% | Sparse |

Note: qwen3-embedding (Recall's embedder) already at 51.5% nDCG@10 on BEIR — strong dense-only performance. BM25 alone is 41.2%, showing BM25 hybrid would help primarily in recall gaps, not overall nDCG.

### Qdrant Hybrid Implementation: Effort to Add

Qdrant 1.10.0+ (July 2024) has a native Query API supporting sparse+dense hybrid in a single server-side call. Implementation involves:

1. Add sparse vector config to existing collection (or create new collection with both):
```python
client.create_collection(
    collection_name="recall-memories",
    vectors_config={"dense": models.VectorParams(size=1024, distance=models.Distance.COSINE)},
    sparse_vectors_config={
        "bm42": models.SparseVectorParams(modifier=models.Modifier.IDF)
    }
)
```

2. At query time, generate sparse embedding (FastEmbed BM42, CPU-friendly) + dense embedding, combine with RRF:
```python
client.query_points(
    collection_name="recall-memories",
    prefetch=[
        models.Prefetch(query=sparse_embedding.as_object(), using="bm42", limit=20),
        models.Prefetch(query=dense_embedding.tolist(), using="dense", limit=20),
    ],
    query=models.FusionQuery(fusion=models.Fusion.RRF),
    limit=10
)
```

BM42 model is `Qdrant/bm42-all-minilm-l6-v2-attentions`, available via `fastembed`. CPU-only, fast inference. No GPU required.

**Effort estimate**: 2-3 days engineering, no new infrastructure. Qdrant already deployed. Requires collection recreation or migration (add sparse vector field to existing collection). BM42 embedding adds ~5-15ms per query.

## The Real Gap: Reranker, Not BM25

For Recall's specific use case (50-200 char factual memory strings), the analysis leads to a counterintuitive finding:

**Traditional BM25 is low-value for short memories.** The term-frequency component collapses; only IDF adds signal. On short factual strings like "Tim prefers self-hosted infrastructure over cloud services", BM25 retrieves "Tim prefers..." when the query is "infrastructure preference" — but so does a good dense embedding.

**Where Recall's retrieval can actually fail:**
- Exact token queries: "what is Tim's CasaOS IP?" — dense may return semantically close memories about other IPs if the exact address isn't in the top match. BM42 IDF component would boost exact "192.168.50.19" matches.
- Proper nouns and technical tokens: "qwen3-embedding", "proxyclaude", "CasaOS" — dense embeddings may not distinguish these from similar terms.

**The higher-ROI improvement**: Adding a cross-encoder reranker (like mem0 offers via Sentence Transformer). This re-scores the top-K vector candidates by jointly encoding query + memory, catching ordering errors that vector cosine misses. For a 500-memory collection, reranking top-20 with MiniLM-L6-v2 adds ~5-15ms latency on CPU and meaningfully improves result ordering.

## Mitigation Recommendation

**Priority 1 (Medium effort, high ROI): Add cross-encoder reranker**
- Integrate sentence-transformers `cross-encoder/ms-marco-MiniLM-L-6-v2` as optional post-retrieval pass
- Retrieve top-20 from Qdrant, rerank to top-5, return to caller
- CPU-only, runs on existing hardware. ~5-15ms added latency per query.
- This closes the gap with mem0's reranker feature and partially closes the Zep gap.

**Priority 2 (Low-medium effort, moderate ROI for exact-token queries): Add BM42 sparse layer**
- Use FastEmbed `Qdrant/bm42-all-minilm-l6-v2-attentions` for sparse vectors
- Add sparse vector field to Recall's Qdrant collection (requires migration)
- At query time: dual-path (dense + BM42) fused with RRF
- Particularly valuable for queries containing exact technical tokens (IP addresses, service names, model names)
- Estimated 5-15% recall improvement for exact-match queries in Recall's specific corpus

**Priority 3 (Do NOT implement): Classical BM25**
- Traditional BM25 with term-frequency scoring is not worth implementing for 50-200 char memories
- BM42 is a strictly better choice for this use case
- Qdrant's own documentation explicitly states BM25 TF component "effectively renders... useless" for short RAG chunks
